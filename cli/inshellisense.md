# inshellisense — IDE 스타일 셸 자동완성

<https://github.com/microsoft/inshellisense>

HN 토론: <https://news.ycombinator.com/item?id=38167363> (388점, 148개 댓글)

## 소개

inshellisense는 Microsoft가 만든 오픈소스 도구로, 터미널 셸에 IDE 수준의 명령줄 자동완성을
제공한다.
600개 이상의 CLI 도구에 대한 자동완성 정의를 내장하고 있으며, Windows·Linux·macOS를 모두
지원한다.
bash, zsh, fish, pwsh, powershell, cmd, xonsh, nushell까지 8가지 셸과 통합된다.
최근 0.0.1 정식 릴리즈를 냈다.[^awbrg789]

설치는 npm(`npm install -g @microsoft/inshellisense` 후 `is init`)이나 Homebrew
(`brew tap microsoft/inshellisense` 후 `brew install inshellisense`) 두 경로를 지원한다.
`is` 명령으로 세션을 시작하고, Tab·화살표키·Esc로 자동완성 제안을 탐색하며, `exit`로 종료한다.
셸 설정 파일에 초기화 스크립트를 추가하면 셸을 열 때마다 자동으로 활성화할 수 있다.
설정은 `~/.inshellisenserc` 또는 `~/.config/inshellisense/rc.toml`의 TOML 파일로 이뤄지며,
키바인딩 커스터마이징, 별칭(alias) 활성화, 최대 제안 수 조절 등을 지원한다.

라이선스는 MIT다.
코드베이스는 TypeScript가 87.1%로 대부분을 차지하고, Shell 9.3%, JavaScript 1.8%로
구성돼 있으며 Node.js 런타임 위에서 동작한다.
현재 302개 커밋, 224개 포크, 1만 명이 넘는 스타를 보유하고 있다.

## 분석

### “600개 이상의 CLI 지원”이라는 숫자가 가리키는 유지보수 부담의 구조

셸 자동완성 도구의 가치는 등록된 CLI 정의의 개수에 정비례한다.
그러나 이 숫자는 동시에 이 프로젝트가 짊어져야 할 유지보수 부담의 크기이기도 하다.
각 CLI 도구는 자신만의 서브커맨드, 플래그, 인자 구조를 가지고 있고, 그 도구가 버전업되며
인터페이스가 바뀔 때마다 inshellisense 쪽의 정의도 함께 갱신돼야 한다.

이 구조는 언어 서버 프로토콜(LSP)이 에디터 생태계에서 풀어낸 문제와 정확히 같은 형태다.
LSP 이전에는 에디터마다 각 언어의 자동완성을 개별적으로 구현해야 했고, LSP 이후에는
언어 하나당 서버 하나만 구현하면 모든 에디터가 이를 재사용할 수 있게 됐다.
inshellisense가 이 방향으로 가려면 각 CLI 개발자가 스스로 자동완성 스펙을 제공하고
inshellisense가 이를 소비하는 구조가 이상적이지만, 현재는 반대로 inshellisense 쪽에서
600개 CLI 각각의 정의를 직접 유지하는 중앙집중형 구조로 보인다.
이 구조가 Microsoft라는 대형 조직의 지원 없이 장기적으로 지속 가능한지는
프로젝트의 커밋·기여자 활동을 지켜봐야 판단할 수 있는 지점이다.

### 8가지 셸 지원이 요구하는 것은 단순한 이식이 아니라 각 셸의 실행 모델 이해다

bash, zsh, fish부터 Windows의 cmd, powershell, 그리고 상대적으로 최근에 등장한 xonsh,
nushell까지 지원한다는 것은 단순히 “8개 언어로 번역”하는 문제가 아니다.
각 셸은 명령줄을 파싱하고 실행하는 방식, 프롬프트를 그리는 방식, 키 입력을 가로채는 방식이
근본적으로 다르다.
특히 nushell처럼 구조화된 데이터(structured data)를 다루는 셸과 전통적인 텍스트 스트림
기반 셸은 자동완성이 개입해야 하는 지점 자체가 다를 수 있다.

이 다양성 때문에, inshellisense가 “IDE 수준의 자동완성”을 모든 셸에서 동일한 품질로
제공하는지는 셸마다 편차가 있을 가능성이 높다.
가장 널리 쓰이는 bash·zsh에서의 지원 성숙도와, 상대적으로 신생 셸인 nushell·xonsh에서의
지원 성숙도 사이에는 실제 사용 경험의 차이가 있을 것으로 예상되며,
이 편차가 얼마나 큰지는 README만으로는 확인할 수 없다.

### inshellisense는 사실 Fig의 완성 엔진을 재사용한 결과물이다

HN 댓글 중 ilc는 코드를 직접 읽고 중요한 사실을 짚는다 — inshellisense는 Fig.io의 내부
완성 엔진을 재사용해 실제 작업을 수행하며, 이 때문에 TypeScript라는 언어 선택이
강제된 것이라는 지적이다.[^ilc]
무거운 작업은 사실상 Fig의 엔진이 담당하고, inshellisense는 그 위에 셸 통합 계층을
얹은 것에 가깝다는 뜻이다.

이 사실은 앞서 짚은 “600개 이상의 CLI 지원”이라는 숫자의 기원을 다시 생각하게 만든다.
이 정의들을 inshellisense가 처음부터 직접 구축한 것이 아니라 Fig 생태계에서 이미
축적된 자산을 가져온 것이라면, 이 프로젝트의 실제 기여는 정의 자체보다 그 정의를
다양한 셸에 통합하는 계층에 있다고 봐야 한다.
Fig의 공동창업자 mschrage가 댓글에서 “IDE 스타일 자동완성의 대안적 구현을 보는 것이
멋지다”고 반응한 것도, 이것이 Fig와 경쟁하는 프로젝트라기보다 같은 아이디어의 또 다른
구현이라는 뉘앙스를 준다.[^mschrage]

### TypeScript 구현이라는 선택 자체가 커뮤니티에서 논쟁거리가 됐다

HN 댓글의 guessmyname은 셸 자동완성처럼 매 키 입력마다 실행돼야 하는 도구를
TypeScript보다 더 효율적인 언어로 작성했어야 한다고 지적하며, 과거 bash_completion.sh를
껐던 경험 — 세션 시작 속도와 Tab 완성 지연이 300ms를 넘었던 경험 — 을 근거로 든다.[^guessmyname]
k8svet는 한 걸음 더 나아가 Go로 작성된 셸 무관 완성 도구 carapace-bin과의 비교를
제안한다.[^k8svet]

이 논쟁은 셸 자동완성이라는 카테고리에서 언어 선택이 단순한 취향 문제가 아니라는 것을
보여준다.
자동완성은 사용자가 타이핑하는 매 순간 개입해야 하므로 지연시간에 극도로 민감하고,
Node.js 런타임 기동 비용이 이 지연시간에 어떻게 기여하는지가 실사용 경험을 좌우한다.
앞서 짚은 유지보수 부담의 문제와 마찬가지로, 이것도 inshellisense가 Fig 엔진의 언어
선택을 그대로 물려받은 결과일 가능성이 높다는 점에서 ilc의 지적과 같은 뿌리를 공유한다.

## 비평

### “IDE 스타일”이라는 프레이밍은 셸의 조합적 표현력을 놓칠 위험을 안고 있다

이 도구가 표방하는 “IDE 스타일 자동완성”은 본질적으로 사전에 알려진 명령·옵션 집합에서
후보를 제시하는 방식이다.
그러나 셸의 강력함은 파이프(`|`), 리다이렉션(`>`), 명령 치환(`$(...)`), 서브셸처럼
사전에 정의된 명령 목록만으로는 예측할 수 없는 조합적 표현에서 나온다.

즉 IDE의 자동완성이 잘 작동하는 이유는 프로그래밍 언어의 문법과 API가 상대적으로
고정돼 있기 때문인데, 셸에서는 사용자가 명령들을 조합하는 방식 자체가 무한에 가깝다.
inshellisense가 개별 명령의 플래그와 서브커맨드를 잘 제안하더라도, 여러 명령을 파이프로
엮은 복잡한 한 줄짜리 셸 스크립트에서 그 조합 전체의 의도를 이해하고 제안하는 것은
전혀 다른 난이도의 문제다.
README는 이 조합적 복잡성에 어떻게 대응하는지에 대한 설명 없이, 개별 CLI 도구 단위의
자동완성만을 강조한다.

### 자동완성 정의의 정확성을 검증하는 방법이 README에 드러나지 않는다

600개 이상의 CLI 정의를 유지한다는 것은, 그 정의 중 일부가 실제 CLI의 최신 동작과
어긋날 가능성을 항상 안고 있다는 뜻이다.
어떤 CLI가 새 버전에서 플래그를 하나 추가하거나 이름을 바꿨는데 inshellisense의 정의가
갱신되지 않으면, 사용자는 실제로는 존재하지 않는 옵션을 제안받거나 존재하는 옵션을
제안받지 못하게 된다.

README에는 이런 정의의 정확성을 자동으로 검증하는 테스트 체계나, 각 CLI 버전과
정의 버전을 어떻게 동기화하는지에 대한 설명이 보이지 않는다.
자동완성 도구에서 이 문제는 사소하지 않다 — 잘못된 자동완성 제안은 아무 제안도 없는 것보다
나쁠 수 있다. 사용자가 도구를 신뢰하고 제안을 그대로 받아들였다가 존재하지 않는 플래그를
사용해 명령이 실패하는 경험은, 자동완성 기능 자체에 대한 신뢰를 깎는다.

## 인사이트

### 터미널의 자동완성은 결국 “터미널이 텍스트 인터페이스인가 구조화된 인터페이스인가”라는 더 큰 질문의 축소판이다

inshellisense가 IDE 수준의 자동완성을 셸에 가져오려는 시도 자체가, 터미널이라는 인터페이스가
오랫동안 순수한 텍스트 스트림으로 남아 있었다는 사실을 거꾸로 드러낸다.
IDE가 자동완성을 잘 제공할 수 있는 이유는 코드가 구문적으로 파싱 가능한 구조를 갖고 있기
때문인데, 전통적인 셸 명령줄은 그런 구조 없이 문자열을 그대로 프로그램에 넘기는 방식으로
작동해왔다.

nushell 같은 최근의 구조화된 셸이 등장하고, inshellisense 같은 도구가 IDE 스타일 자동완성을
가져오려는 시도가 동시에 나타나는 것은 우연이 아니다.
이는 터미널이라는 40년 넘은 인터페이스가 서서히 “구조화된 데이터를 다루는 인터페이스”
쪽으로 이동하고 있다는 더 큰 흐름의 일부로 읽을 수 있다.
자동완성이 좋아진다는 것은 표면적 편의이지만, 그 이면에는 셸이 다루는 명령과 데이터를
기계가 이해할 수 있는 형태로 구조화하려는 압력이 깔려 있다.

### AI 코딩 에이전트의 확산이 셸 자동완성 도구의 존재 이유를 재편할 수 있다

Claude Code, Cursor 같은 AI 코딩 에이전트가 터미널 명령을 직접 실행하는 빈도가 늘어나면서,
사람이 직접 셸에 명령을 타이핑하고 자동완성의 도움을 받는 상황 자체가 점점 줄어들 가능성이
있다.
에이전트는 자동완성 UI가 필요 없다 — 정확한 명령을 알고 있거나, 매뉴얼을 검색해서
바로 완전한 명령을 생성하기 때문이다.

만약 이 흐름이 계속되면, inshellisense가 쌓아온 600개 이상의 CLI 정의는 사람을 위한
자동완성 제안이 아니라, AI 에이전트가 어떤 CLI를 어떻게 사용해야 하는지 파악하기 위한
구조화된 지식 베이스로 재활용될 여지가 있다.
자동완성 정의는 본질적으로 “이 명령은 어떤 하위 명령과 옵션을 갖는가”에 대한 기계 판독
가능한 명세이므로, 사람의 타이핑을 돕는 용도에서 에이전트의 명령 생성을 검증하는 용도로
그 쓸모의 축이 옮겨갈 가능성이 있다.
이렇게 되면 이 도구의 가치는 “자동완성 UI”가 아니라 “CLI 명세 데이터베이스”에 있게 되고,
그 경우 지금의 터미널 UI 통합보다 API 형태의 데이터 접근이 더 중요한 기능으로 부상할 수 있다.

### 셸마다 다른 자동완성 경험의 편차는 결국 “가장 약한 셸의 지원 수준”이 전체 사용자 경험을 결정한다

8가지 셸을 지원한다는 홍보 문구는 있는 그대로 받아들이면 모든 셸에서 동일한 경험을
제공하는 것처럼 들리지만, 실제로는 그 반대의 동학이 작동할 가능성이 크다.
사용자는 자신이 사용하는 셸에서의 실제 경험만을 체감하기 때문에, 지원 목록에 있는
셸 개수보다 자신이 쓰는 특정 셸에서의 완성도가 만족도를 결정한다.

이는 크로스플랫폼 도구 일반이 마주하는 문제이기도 하다 — 플랫폼 목록의 길이와
사용자가 실제로 체감하는 품질은 별개의 지표이며, 오히려 지원 범위를 넓히려는 노력이
가장 널리 쓰이는 플랫폼에 대한 완성도 투자를 분산시킬 위험도 있다.
inshellisense가 8개 셸을 나열하는 것보다, 각 셸에서의 실제 완성도를 사용자 스스로
검증해보는 것이 이 도구의 실질적 가치를 판단하는 더 정확한 방법이다.
