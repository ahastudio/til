# 파비콘 생성 zsh 스크립트

<https://gist.github.com/dodok8/95329264445a9f76fa42f1fc26f5e06b>

## 소개

ImageMagick을 사용해 단일 입력 이미지에서 웹 프로젝트에 필요한 파비콘 자산 일체를 생성하는 zsh 스크립트다.
별도 GUI 도구나 외부 서비스 없이 커맨드라인 하나로 파비콘 세트를 만들 수 있다.

## 주요 기능

하나의 입력 이미지에서 다음 파일들을 생성한다.

- `apple-touch-icon.png` (180×180)
- `icon-192.png`
- `icon-512.png`
- `icon-mask.png` (마스크 가능 아이콘)
- `favicon.ico`
- SVG 입력 시 SVG 아이콘
- `manifest.webmanifest`
- HTML `<head>` 삽입용 스니펫 (`favicon.html`)

래스터(PNG, JPEG 등)와 SVG 입력을 모두 지원한다.
결과물은 `site-root/` 디렉터리에 정리되어 바로 배포에 사용할 수 있다.

## CLI

```bash
./make-favicon.zsh <input-image> [options]
```

| 옵션                   | 설명                                     | 기본값             |
| ---------------------- | ---------------------------------------- | ------------------ |
| `-o, --out <dir>`      | 출력 디렉터리                            | `./favicon-output` |
| `-n, --name <name>`    | 웹사이트 이름 (manifest 파일에 사용)     |                    |
| `-b, --background <c>` | 캔버스 배경 색상                         |                    |
| `-p, --prefix <path>`  | 자산 URL 경로 접두사                     |                    |
| `-d, --density <val>`  | SVG를 PNG로 변환할 때의 렌더 밀도        | `2048`             |

## 의존성

ImageMagick 7이 시스템 PATH에 설치되어 있어야 한다.

```bash
# macOS (Homebrew)
brew install imagemagick

# Ubuntu/Debian
sudo apt install imagemagick
```

## 분석

### 단순한 스크립트가 해결하는 반복적 마찰

웹 프로젝트를 시작할 때마다 파비콘 세트를 준비하는 것은 필요하지만 반복적이고 지루한 작업이다.
다양한 크기와 형식, manifest 파일, HTML 스니펫을 일일이 생성하는 것은 시간과 집중력을 소모한다.
이 스크립트는 이 반복 작업을 단일 커맨드로 자동화한다.

GUI 기반 파비콘 생성 서비스들이 존재하지만, 이들은 이미지를 외부 서버에 업로드해야 하고 출력 형식에 제한이 있다.
로컬에서 실행되는 스크립트는 프라이버시와 커스터마이징 측면에서 유리하다.

### ImageMagick을 선택한 이유

ImageMagick은 이미지 변환 분야의 오랜 표준 도구다.
복잡한 설치 없이 패키지 매니저로 설치 가능하고, 다양한 이미지 형식과 변환 파라미터를 지원한다.
SVG 래스터화, 멀티 크기 ICO 생성, PNG 최적화를 모두 하나의 도구로 처리할 수 있다.

단점은 ImageMagick 7이 일부 환경에서 6과 함께 설치되어 `magick` vs `convert` 명령어 충돌을 유발할 수 있다는 점이다.
스크립트가 ImageMagick 7 전용이라면 이 환경 호환성이 잠재적 마찰 포인트다.

### zsh 선택의 실용성

bash 대신 zsh를 선택한 것은 macOS 기본 셸이 zsh로 전환된 이후 macOS 개발자 대상 스크립트에서 합리적인 선택이다.
그러나 Linux 환경이나 CI/CD 파이프라인에서는 bash가 더 범용적이다.
프로젝트 자동화 파이프라인에 통합하려면 셸 호환성을 확인해야 한다.

## 비평

### 의존성이 단 하나라는 점은 장점이자 전제 조건

ImageMagick 하나만 의존한다는 것은 환경 설정이 단순하다는 장점이다.
그러나 ImageMagick 7은 일부 Linux 배포판에서 기본 패키지가 아니고, 버전 관리가 까다롭다.
CI/CD 환경에서 이 스크립트를 사용하려면 ImageMagick 7 설치가 파이프라인에 포함되어야 한다.

### 출력 파일 구조가 프레임워크마다 다른 요구 사항과 다를 수 있다

Next.js의 `app/` 디렉터리, Astro의 `public/`, Vite 기반 프로젝트의 정적 파일 위치는 프레임워크마다 다르다.
스크립트가 생성하는 `site-root/` 구조가 특정 프레임워크의 요구 사항과 일치하지 않으면 결과물을 수동으로 이동해야 한다.
`-o` 옵션으로 출력 디렉터리를 지정할 수 있지만, 프레임워크별 프리셋이 있다면 더 편리할 것이다.

## 인사이트

### CLI 도구의 가치는 기능이 아니라 마찰 제거에 있다

이 스크립트의 기능 자체는 새롭지 않다.
ImageMagick으로 파비콘을 생성하는 방법은 이미 알려져 있고, GUI 서비스도 존재한다.
가치는 “이미 알고 있는 작업을 할 때마다 검색하거나 도구를 열어야 하는 마찰”을 제거하는 데 있다.
개인 도구의 진정한 효용은 기능 우위보다 워크플로에 얼마나 자연스럽게 통합되는가에 달려 있다.

### zsh 스크립트를 gist로 공유하는 패턴은 재발견 가능성이 낮다

개인 도구를 gist로 공유하는 것은 Github 검색이나 패키지 관리자를 통해 찾기 어렵다는 단점이 있다.
Homebrew Cask, pip 패키지, npm 글로벌 패키지로 배포하면 재발견 가능성이 높아지지만 유지보수 부담이 늘어난다.
가벼운 도구는 gist + 검색 가능한 커뮤니티 공유(Hacker News, GeekNews)의 조합이 현실적인 배포 전략이다.
