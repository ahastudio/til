# CLI Printing Press: 에이전트를 위한 CLI 생성기

<https://github.com/mvanhorn/cli-printing-press>

## 소개

CLI Printing Press는 API 명세나 웹사이트, HAR 파일을 입력받아
에이전트 친화적인 CLI 도구와 MCP 서버를 자동으로 생성하는 Go 기반
프로젝트다. 핵심 목표는 단순하다. AI 에이전트가 API를 다룰 때
문서를 헤매거나 잘못된 호출로 토큰을 낭비하지 않도록, 잘 설계된
명령형 인터페이스를 미리 만들어 두는 것이다. 프로젝트는 이를
“잘 설계된 CLI는 에이전트에게 근육 기억과 같다”라는 문장으로
요약한다. 문서를 뒤질 필요도, 잘못된 길로 들어설 일도, 낭비되는
토큰도 없다는 주장이다.

도구는 Go 1.26.4 이상에서 동작하며, 생성 결과물로 Cobra 기반 CLI
(`<api>-pp-cli`)와 MCP 서버(`<api>-pp-mcp`) 두 개의 바이너리를
내놓는다. 여기에 FTS5 전문 검색과 증분 동기화를 지원하는 SQLite
데이터 레이어가 함께 붙는다. 즉 에이전트가 매번 원격 API를 직접
때리는 대신, 로컬에 미러링된 데이터를 빠르게 질의하도록 만드는
구조를 지향한다. 예제로 제시된 `linear-pp-cli`는 로컬 SQLite
미러를 통해 50ms 수준의 복합 질의를 달성한다고 한다.

흥미로운 점은 이 도구가 API 엔드포인트를 단순히 래핑하는 데
그치지 않는다는 것이다. Printing Press는 각 API의 “숨겨진 정체성”
혹은 “비자명한 통찰(Non-Obvious Insight)”을 찾으려 한다. 예를
들어 Discord를 단순한 채팅 앱이 아니라 검색 가능한 지식 베이스로
재해석하고, 그 관점에서 `stale`, `bottleneck`, `health` 같은
도메인 특화 명령을 일반 엔드포인트 래퍼와 함께 생성한다. 입력
방식은 OpenAPI 명세(`--spec`), 웹사이트 URL(브라우저 캡처로 명세
자동 추출), 그리고 DevTools에서 추출한 HAR 파일(`--har`) 세 가지를
지원한다.

전체 인터페이스는 Claude Code의 슬래시 명령을 통해 동작한다.
`/printing-press <app>`로 생성을 시작하고, `-reprint`,
`-polish`, `-publish`, `-amend` 같은 보조 명령으로 재생성, 진단,
라이브러리 발행, 개선 PR 생성을 처리한다. 라이선스는 MIT이며,
Peter Steinberger의 discrawl/gogcli, Trevin Chow의 에이전트 네이티브
CLI 원칙 등을 영향원으로 명시하고 있다.

## 사용법

```bash
# 설치
curl -fsSL https://raw.githubusercontent.com/mvanhorn/cli-printing-press/main/scripts/install.sh | bash

# Claude Code 안에서 CLI 생성
claude
/printing-press Notion
/printing-press https://postman.com/explore
```

생성 파이프라인은 6단계로 구성된다. Phase 0에서 입력 유형을
판별하고 기존 결과물을 재사용하며(1~3분), Phase 1에서 API 정체성과
경쟁 도구를 분석하는 리서치 브리프를 작성한다(5~10분). Phase 1.5의
“생태계 흡수 게이트”는 경쟁 도구의 기능을 목록화하고(5~10분),
Phase 2에서 실제 Go CLI와 MCP 서버를 생성한다(1~2분). Phase 3은
흡수한 기능과 새로운 명령을 더해 “GOAT”를 만들고(10~20분),
Phase 4의 Shipcheck가 검증 스위트를 돌린다(3~8분).

검증은 네 가지 기계적 관문으로 이뤄진다. 100점 만점의 2계층
스코어카드(인프라 + 도메인 정확성), 죽은 플래그와 잘못된 경로를
잡아내는 dogfood 테스트, 경로 존재·플래그 사용·SQLite 파이프라인·
인증 형식을 확인하는 행위 증명, 그리고 자격 증명이 주어지면 실제
API에 대해 읽기 전용으로 돌리는 라이브 스모크 테스트다.

## 분석

### 에이전트를 일급 사용자로 상정한 설계

이 프로젝트의 가장 분명한 전제는 CLI의 주 소비자가 사람이 아니라
에이전트라는 것이다. 생성된 도구는 파이프로 연결되면 별도 플래그
없이 자동으로 JSON을 출력하고, `--compact` 모드로 토큰을 60~80%
줄이며, 0/2/3/4/5/7로 분류된 타입드 종료 코드를 제공한다. 이
모든 결정의 근거가 “에이전트가 그것을 소비하기 때문”이라는 점에서,
사람 사용자를 1차 대상으로 삼던 전통적 CLI 설계와 명확히 갈라진다.

타입드 종료 코드는 특히 시사적이다. 인증 실패(4), 미발견(3),
레이트 리밋(7)을 구분해 반환한다는 것은, 에이전트가 출력 텍스트를
파싱해 추론하는 대신 종료 코드만 보고 다음 행동을 자기 교정하도록
설계했다는 뜻이다. 이는 LLM이 자연어 에러 메시지를 해석하며 토큰과
추론 단계를 소모하는 비용을 구조적으로 줄이려는 시도다.

`--dry-run`을 통한 안전한 탐색, `--no-input`·`--yes` 같은
비대화형 플래그 역시 사람이 아니라 자동화 루프를 가정한 흔적이다.
즉 Printing Press는 “에이전트가 쓰기 좋은 CLI란 무엇인가”라는
질문에 대한 하나의 의견서에 가깝다.

### 단순 래핑을 넘어선 도메인 재해석

두 번째 축은 “창의성 사다리(Creativity Ladder)”라는 개념이다.
생성된 CLI는 다섯 단계를 오른다. (1) API 래퍼 명령, (2) 출력
포맷팅, (3) SQLite 영속화와 FTS5 검색, (4) `stale`·`orphans`
같은 도메인 분석, (5) `health`·`similar` 같은 행위 통찰이다.
대부분의 생성기가 1~2단계에서 멈추는 반면, 이 도구는 3~5단계를
목표로 한다고 주장한다.

이 야심은 곧 “비자명한 통찰”과 “도메인 원형(archetype)” 분류로
이어진다. API를 프로젝트 관리, 커뮤니케이션, 결제, 인프라,
콘텐츠 등으로 자동 분류하고, 각 원형에 맞는 워크플로 명령을
템플릿에서 생성한다. 단순 엔드포인트 미러링이 아니라, API가
실제로 가능하게 하는 작업의 관점에서 명령을 설계하겠다는 것이다.

다만 이 지점이 도구의 가장 큰 야심이자 동시에 가장 검증하기 어려운
부분이기도 하다. “숨겨진 정체성”을 찾는 작업은 본질적으로 해석적
판단이며, 그 판단의 품질은 LLM의 리서치 능력에 직접 의존한다.

### 로컬 상태 레이어를 경유하는 패턴

세 번째 축은 에이전트가 원격 API를 직접 호출하는 대신, 로컬 SQLite
미러를 경유하게 만드는 아키텍처다. FTS5 전문 검색과 증분 동기화를
기본 제공한다는 것은, 단발성 API 클라이언트가 아니라 상태를 보존하는
데이터 레이어를 만들겠다는 의도다.

이 패턴은 실용적 이점이 분명하다. 레이트 리밋 회피, 오프라인 질의,
여러 엔드포인트를 가로지르는 복합 질의의 가속이 모두 로컬 미러에서
나온다. `linear-pp-cli`의 50ms 복합 질의가 그 사례다. 에이전트
입장에서는 매 호출의 지연과 실패 가능성이 줄어드는 만큼 추론 루프가
안정된다.

이는 더 넓은 흐름과도 맞닿는다. 에이전트용 도구 설계가 HTTP API
직접 호출에서, 검증 가능한 CLI와 상태 저장 레이어를 경유하는
방향으로 이동하는 경향이다. Printing Press는 이 패턴을 자동
생성으로 고정하려는 시도로 읽을 수 있다.

## 비평

### 사용 시나리오가 Claude Code에 강하게 묶여 있다

가장 먼저 짚을 점은 이 도구의 1차 인터페이스가 Claude Code의
슬래시 명령이라는 사실이다. 설치 후 핵심 워크플로는 `claude`를
실행하고 `/printing-press`를 입력하는 것으로 시작한다. 즉
“API 명세를 넣으면 CLI가 나온다”는 일반적 코드 생성 도구처럼
들리지만, 실제로는 특정 에이전트 런타임에 깊이 결합되어 있다.

이는 양날의 검이다. 한편으로 6단계 파이프라인의 리서치·분석·생성
단계 상당 부분이 LLM 추론에 의존하므로 에이전트 결합은 불가피하다.
다른 한편으로 이 결합은 결과물의 재현성과 이식성을 약화시킨다.
같은 입력이라도 모델 버전이나 프롬프트 변화에 따라 다른 CLI가
나올 수 있고, 다른 에이전트로 옮기는 일은 자명하지 않다.

`-reprint` 명령이 “최신 머신으로 재생성”을 제공한다는 점 자체가,
생성물이 시간이 지나면 노후화되는 산출물임을 인정하는 셈이다.
생성형 도구의 결과물은 손으로 작성한 코드와 달리 “고정된 자산”이
아니라 “재생성 대상”에 가깝다. 이를 자산으로 취급하고 유지보수할
주체가 누구인지는 별개의 문제로 남는다.

### 자동 생성 CLI의 신뢰 경계

검증 스위트가 네 단계로 촘촘해 보이지만, 그 검증이 보장하는 것은
대체로 “구조적 정합성”이다. 경로가 존재하는지, 플래그가 실제로
쓰이는지, 인증 형식이 맞는지, SQLite 파이프라인이 연결되는지를
확인한다. 그러나 이것은 생성된 명령이 의미적으로 올바른 작업을
하는지와는 다른 차원이다.

특히 4~5단계의 “도메인 분석”과 “행위 통찰” 명령, 예컨대
`bottleneck`이나 `health` 같은 것은 정의상 해석적이다. 무엇이
병목이고 무엇이 건강한 상태인지는 LLM이 리서치 단계에서 내린
판단에 달려 있다. dogfood 테스트는 이 명령이 죽지 않고 실행되는지는
확인하지만, 그 명령이 반환하는 분석이 실제로 타당한지는 사람이
검토하지 않으면 알기 어렵다.

여기에 더해 URL이나 HAR에서 명세를 역추출하는 입력 모드는
잠재적 위험을 안고 있다. 브라우저 캡처로 추론한 비공식 API는
문서화된 계약이 없으므로, 권한 범위·레이트 리밋·약관 위반 여부가
불투명하다. 자동 생성된 CLI가 어떤 범위의 호출을 하는지 사람이
사후에 일일이 검토해야 한다면, 절약했다는 토큰과 시간이 다른
곳에서 새어 나갈 수 있다.

### 작은 저장소의 지속 가능성

이 도구가 해결하려는 문제는 분명 실재한다. 에이전트가 API를
안정적으로 다루기 어렵다는 통증은 현재 진행형이다. 그러나
Printing Press가 제시하는 해법은 상당히 정교하고 무거운 파이프라인
이며, 이를 유지하는 부담은 결코 작지 않다.

6단계 파이프라인, 도메인 원형 템플릿, 2계층 스코어카드, 다중
입력 모드를 모두 최신 상태로 유지하려면 지속적인 손길이 필요하다.
API들은 끊임없이 변하고, 의존하는 에이전트 런타임도 빠르게
진화한다. `printingpress.dev` 카탈로그에 19개 이상의 사전 검증
CLI가 있다고 하지만, 이들이 대상 API의 변경을 따라가지 못하면
빠르게 낡은 결과물이 된다.

결국 이 프로젝트의 가치는 도구 자체보다 그 뒤를 받치는 유지보수
역량과 커뮤니티에 달려 있다. 트렌딩 순위와 별 수는 관심의
지표일 뿐, 장기 신뢰성의 보증은 아니다. “한 번 생성하면 끝”이
아니라 “계속 재생성해야 하는” 도구라면, 그 재생성 비용을 누가
어떻게 감당할지가 채택의 실질적 기준이 된다.

## 인사이트

### SDK의 시대에서 에이전트 인터페이스의 시대로

전통적으로 API 통합은 언어별 SDK를 작성·배포하는 방식이 중심이었다.
Printing Press는 이 무게 중심을, 사람과 에이전트가 함께 쓰는 CLI와
로컬 색인, 복합 명령 설계로 옮기려 한다. 이는 단순한 도구 하나의
등장이 아니라, “에이전트가 소비할 인터페이스를 어떻게 설계할
것인가”라는 새로운 설계 범주의 부상으로 읽힌다.

흥미로운 2차 효과는 인터페이스의 청중이 바뀌면 설계 우선순위
자체가 뒤집힌다는 점이다. 사람을 위한 CLI는 가독성과 도움말의
친절함을 중시하지만, 에이전트를 위한 CLI는 토큰 효율, 타입드
종료 코드, 비대화형 동작을 우선한다. 같은 “CLI”라는 단어를
쓰지만 설계 철학은 다른 종(種)에 가깝다.

이 흐름이 계속된다면, 앞으로 많은 API 제공자가 “에이전트용 CLI”를
SDK와 별도의 1급 산출물로 제공하게 될 가능성이 있다. 사람용
문서와 기계용 인터페이스가 분기하는 셈이다.

### 추상화 계층을 하나 더 쌓는 일의 양면성

Printing Press는 API와 에이전트 사이에 자동 생성된 CLI라는 계층을
하나 더 끼워 넣는다. 역사적으로 새로운 추상화 계층은 비용을
숨기는 동시에 통제력을 빼앗는다. ORM이 SQL을 가렸듯, 자동 생성
CLI는 원시 API의 세부를 가리고 에이전트의 시야를 명령형 인터페이스
안으로 좁힌다.

이는 명백한 이득과 글로싱된 트레이드오프를 함께 가진다. 이득은
에이전트가 문서를 헤매지 않고 일관된 표면을 얻는다는 것이다.
숨겨진 비용은, 생성기가 노출하지 않기로 한 엔드포인트나 파라미터는
에이전트의 세계에서 사실상 존재하지 않게 된다는 점이다. 생성
과정의 “비자명한 통찰”이 곧 무엇을 보이고 무엇을 가릴지를
결정하는 권력이 된다.

따라서 이 도구를 채택하는 조직은 편의를 얻는 대신, API에 대한
에이전트의 인식을 한 생성기의 해석에 위임하게 된다. 그 해석이
틀렸을 때 디버깅은 원시 API가 아니라 생성된 계층에서 시작해야
하므로, 추상화가 새는 순간 비용은 오히려 커질 수 있다.

### 검증의 자동화가 곧 신뢰의 자동화는 아니다

이 프로젝트가 가장 공들인 부분 중 하나는 네 단계의 기계적 검증
관문이다. 이는 생성형 도구가 직면하는 근본 문제, 즉 “기계가 만든
것을 어떻게 믿을 것인가”에 대한 정직한 응답이다. 무작정 코드를
뱉어내는 대신, 출력에 점수를 매기고 행위를 증명하려는 태도는
평가할 만하다.

그러나 자동 검증은 검증 가능한 속성만 검증한다는 한계를 본질적으로
안는다. 경로 존재와 플래그 정합성은 기계가 확인하기 쉽지만, 어떤
명령이 실제로 유용한 통찰을 주는지, 도메인 재해석이 적절한지는
기계적 점수로 환원되지 않는다. 100점짜리 스코어카드가 의미하는
바는 “구조적으로 흠이 없다”이지 “의미적으로 옳다”가 아니다.

이는 LLM이 만든 산출물 전반에 적용되는 구조적 교훈이다. 검증
스위트가 정교할수록 사람은 결과를 더 신뢰하기 쉬워지지만, 그
신뢰의 근거가 무엇을 포함하고 무엇을 빠뜨리는지를 이해하지 못하면
오히려 위험하다. 검증의 자동화는 사람의 판단을 줄여 주는 것이
아니라, 사람이 어디에 판단을 집중해야 하는지를 재배치할 뿐이다.
