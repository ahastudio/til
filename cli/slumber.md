# Slumber — 터미널 기반 HTTP/REST 클라이언트

<https://slumber.lucaspickering.me/>

<https://github.com/LucasPickering/slumber>

HN 토론: <https://news.ycombinator.com/item?id=48231938> (193점, 69개 댓글)

## 소개

Slumber는 TUI(터미널 사용자 인터페이스) 기반 HTTP 클라이언트다.
HTTP 요청을 정의하고 실행하며 공유하기 위한 도구로, TUI 외에 CLI와 Python 패키지 형태로도 사용 가능하다.
Rust로 작성됐으며 MIT 라이선스로 배포된다.

세 가지 핵심 약속을 제시한다.
영원히 무료로 사용 가능하다.
모든 설정과 데이터는 로컬에 저장되며 버전 관리 시스템에 체크인할 수 있다.
절대로 “개악(enshittification)”되지 않는다.

## 핵심 개념

작업 단위는 **컬렉션(collection)**이다.
컬렉션은 관련된 HTTP 요청의 집합이다.
각 컬렉션은 YAML 파일(`slumber.yml`)로 정의되며, 요청 템플릿인 **레시피(recipe)**들로 구성된다.

레시피는 다른 요청, 파일, 셸 명령의 결과를 참조하여 동적으로 구성될 수 있다.
프로필(profile)을 통해 환경(개발, 스테이징, 프로덕션)을 전환한다.
응답 탐색 시 JSONPath 셀렉터를 지원한다.

## 설치 및 사용법

`slumber.yml` 설정 파일을 생성한 후 `slumber` 명령으로 TUI를 실행한다.

```yaml
requests:
  get_user:
    method: GET
    url: https://api.example.com/users/{{ user_id }}
```

Insomnia 등 다른 HTTP 클라이언트에서 가져오기(import)를 지원한다.

## 분석

### TUI HTTP 클라이언트의 위치

HTTP 클라이언트 시장은 세 세대로 나눌 수 있다.
첫 번째는 cURL과 같은 순수 CLI 도구로, 스크립팅에 강하지만 인터랙티브 사용이 어렵다.
두 번째는 Postman, Insomnia 같은 GUI 도구로, 사용하기 쉽지만 팀 공유와 버전 관리가 어렵다.
세 번째가 Slumber가 위치한 TUI 도구다.
터미널에서 작동하면서 인터랙티브 UI를 제공하고, 설정 파일 기반이라 Git에 체크인할 수 있다.

Slumber의 포지셔닝은 명확하다.
터미널을 주 작업 환경으로 사용하는 개발자가 GUI 클라이언트의 마우스 중심 UX 없이 인터랙티브한 HTTP 탐색을 원할 때 사용한다.
“Postman에 너무 익숙해졌다”는 반응이 GeekNews에서 나온 것은 이 전환의 마찰을 정직하게 보여준다.

### 설정 파일 중심 설계의 트레이드오프

Slumber의 모든 요청이 YAML 파일로 정의된다는 것은 협업과 재현 가능성에서 강력한 장점이다.
같은 요청 컬렉션을 팀 전체가 공유하고 버전 관리할 수 있다.
특정 버전의 API와 인터랙션했던 요청 집합을 Git 히스토리로 추적할 수 있다.

그러나 이것은 첫 사용 허들을 높인다.
Postman처럼 URL을 입력하고 바로 요청하는 것이 아니라, 먼저 YAML 파일을 만들어야 한다.
빠른 탐색(one-off request)보다 반복 사용하는 요청 집합을 관리하는 데 더 적합하다.
이 도구가 어떤 사람에게 맞는지는 이 트레이드오프에 대한 입장에서 결정된다.

HN에서 vladde는 “GUI 기반 도구를 벗어나고 싶지만 curl만으로는 마땅한 워크플로를 못 찾겠다”고 말했다[^vladde].
이 발언은 Slumber가 채우려는 공백을 정확히 묘사하면서도, YAML 파일을 먼저 작성해야 한다는 요구가 여전히 그 공백을 완전히 메우지 못할 수 있음을 시사한다.
한편 voidUpdate는 YAML 선택 자체에 의문을 제기했다[^voidUpdate].
YAML이 JSON보다 더 복잡하고 덜 인간 친화적이며 보안 취약점도 있다는 지적이다.
설정 파일 형식으로 YAML을 선택한 것은 관성적 결정에 가까우며, 더 단순한 대안을 검토했는지는 문서에서 명확히 드러나지 않는다.

### Rust 선택의 의미

HTTP 클라이언트를 Rust로 만든다는 것은 몇 가지 설계 의도를 암시한다.
성능과 메모리 효율이 TUI 도구에서 체감 가능한 수준으로 중요하고, 단일 바이너리 배포를 통해 의존성 없는 설치 경험을 제공하며, 장기 유지보수를 위한 안전성을 중시한다는 것이다.

TUI 생태계에서 Rust는 ratatui 같은 성숙한 라이브러리를 가지고 있어 선택이 자연스럽다.
그러나 Rust 기반 도구는 컨트리뷰터 풀이 더 작다는 단점이 있다.
1,306 커밋과 1.1k 스타는 활발한 프로젝트지만, 기여자 수가 제한적일 때 유지보수 지속 가능성이 핵심 리스크다.

## 비평

### “절대로 개악되지 않는다”는 약속의 검증 불가능성

Slumber의 세 번째 약속—“절대로 enshittification되지 않겠다”—은 강력하게 들리지만 구속력이 없다.
이 약속은 법적 의무나 구조적 장치가 아니라 저자의 선의에 기반한다.
오픈소스 MIT 라이선스는 포크(fork)를 통해 프로젝트 방향에 이의를 제기할 수 있는 안전망을 제공하지만, 그것이 원 프로젝트가 방향을 바꾸지 않는다는 보장은 아니다.

이 약속이 의미 있으려면 “개악”이 무엇인지 명확히 정의하고, 그 정의를 유지하는 구조적 메커니즘이 필요하다.
약속의 내용이 명확하지 않다면, 저자의 다음 결정이 그 약속을 위반했는지 여부를 판단하기 어렵다.
“영원히 무료”라는 첫 번째 약속도 마찬가지다.
유지보수 비용이 증가하거나 저자의 상황이 바뀌면 이 약속을 유지하기 어려울 수 있다.

### TUI는 접근성 문제를 내포한다

TUI 도구는 터미널 환경에 익숙한 사용자에게 효율적이지만, 그 외 사용자에게는 높은 진입 장벽이다.
HTTP 클라이언트를 팀 전체가 공유해야 할 때, TUI를 불편하게 느끼는 팀원이 있다면 도입이 어렵다.
Postman이나 Insomnia가 시장을 지배하는 이유 중 하나는 마우스 중심 GUI가 더 넓은 사용자 기반을 가지기 때문이다.

Slumber가 “Insomnia에서 가져오기” 기능을 지원하는 것은 기존 GUI 도구 사용자의 전환을 쉽게 하려는 시도지만, 전환 후의 경험이 전환 전만큼 직관적이지 않다면 이 기능만으로는 불충분하다.
도구는 가장 저항이 큰 팀원을 기준으로 선택되는 경우가 많다.
keyle는 “Postman 컬렉션과 환경 변수를 import/export할 수 있다면 평생 고객이 되겠다”고 말했다[^keyle].
전환 마찰의 핵심이 가져오기 지원에 있다는 이 지적은 타당하지만, 정작 Postman으로의 내보내기가 되지 않으면 락인(lock-in)을 우려하는 사용자는 처음부터 이동하지 않을 것이다.

### 1.1k 스타의 의미와 한계

1.1k 스타는 인지도가 있지만 대규모 채택이 이루어지지 않은 도구임을 보여준다.
이 규모에서는 에코시스템 통합, 플러그인, 커뮤니티 지원이 제한적이다.
cURL과 Postman이 가진 광범위한 커뮤니티 리소스—StackOverflow 답변, 튜토리얼, 통합 가이드—와의 격차가 실용적 선택에서 장벽이 된다.

Slumber가 특정 사용자 집단에서 가치를 증명하고 있다면, 그 사용자들이 어떤 사람들인지가 중요하다.
오픈소스에 친숙하고 터미널 중심 워크플로를 가진 1인 또는 소규모 팀 개발자들에게는 훌륭한 선택일 수 있다.
그러나 다양한 배경을 가진 팀이 공용 도구를 선택해야 한다면, 커뮤니티 규모가 선택의 중요한 요소가 된다.

## 인사이트

### 개발 도구의 “enshittification 저항”은 비즈니스 모델에서 시작한다

Slumber가 “enshittification되지 않겠다”는 약속을 할 수 있는 것은 수익을 목표로 하지 않는 개인 프로젝트이기 때문이다.
Postman이 과거보다 무거워지고 더 많은 기능을 유료로 전환한 것은 VC 투자를 받은 후 성장 압력에 노출된 결과다.
도구가 좋을수록 더 많은 사용자가 생기고, 더 많은 사용자가 생기면 수익화 압력이 생기며, 수익화가 시작되면 초기의 단순함이 복잡성으로 대체된다.

이 역설을 피하는 방법은 두 가지다.
하나는 Slumber처럼 개인 프로젝트로 유지하는 것이고, 다른 하나는 수익 모델을 도구 자체가 아닌 다른 곳에서 찾는 것이다.
오픈소스 도구의 지속 가능한 비즈니스 모델—호스팅, 엔터프라이즈 지원, 교육—은 도구 자체를 무료로 유지하면서 성장할 수 있는 경로다.
Slumber가 장기적으로 어느 경로를 선택하는가가 “enshittification 안 함” 약속의 진짜 테스트가 될 것이다.

### YAML 기반 설정이 만드는 새로운 협업 패턴

Slumber의 YAML 기반 설정은 단순히 파일 형식의 문제가 아니다.
HTTP 요청 컬렉션을 코드처럼 다룬다는 철학이다.
이것은 인프라를 코드로 관리하는 IaC(Infrastructure as Code) 패러다임의 HTTP 클라이언트 버전이다.

이 패러다임이 팀에 정착하면 API 통합 방식이 달라진다.
새로운 API 엔드포인트를 탐색할 때 결과물이 즉시 버전 관리된다.
신규 팀원이 온보딩할 때 API 인터랙션 패턴을 코드 리뷰를 통해 전달할 수 있다.
API 변경으로 인한 요청 수정이 Git 커밋으로 추적된다.
이것은 HTTP 클라이언트를 개인 도구에서 팀 문서화 도구로 전환하는 가능성이다.

### 터미널 르네상스와 개발자 도구의 분화

최근 몇 년간 터미널 도구들이 부활하고 있다.
TUI 기반 Git 클라이언트(lazygit), 파일 관리자(yazi), 데이터베이스 클라이언트(pgcli) 등이 GUI 도구의 대안으로 인기를 얻고 있다.
Slumber는 이 터미널 르네상스의 일부다.

이 트렌드의 배경에는 VS Code 같은 통합 GUI 환경에서 벗어나 더 모듈화된 터미널 워크플로를 선호하는 개발자들의 증가가 있다.
AI 코딩 도구들이 터미널 인터페이스를 통해 작동하는 경우가 많아지면서, 터미널 중심 워크플로의 가치가 더 높아지고 있다.
이 맥락에서 Slumber 같은 TUI 도구는 AI 에이전트와 인간 개발자가 같은 인터페이스를 공유할 수 있는 미래를 위한 포지셔닝이기도 하다.

HN 토론에서는 Slumber 외에도 유사한 대안들이 언급됐다.
ananthakumaran은 Emacs 환경에서 verb 패키지를 Postman 대안으로 오래 사용해왔다고 밝혔고[^ananthakumaran], unkn0wn_root는 `.rest`/`.http` 파일 형식을 사용하는 Resterm을 제시했다[^unkn0wn_root].
TUI HTTP 클라이언트는 하나의 표준이 없는 파편화된 공간이다.
각 도구가 서로 다른 설정 형식과 인터페이스를 사용한다는 것은 이 영역이 아직 성숙하지 않았음을 보여주며, 표준화가 이루어지기 전에 어느 도구에 투자할지는 개인의 도박이기도 하다.

반면 marcuskaz는 이런 도구들이 이제 너무 쉽게 만들어지는 시대라며, 직접 만들어 쓰면 되니 굳이 공유하는 의미가 있느냐고 말했다[^marcuskaz].
이것은 도구 생태계의 역설이다.
개발 생산성이 높아질수록 개별 특화 도구가 많아지고, 그만큼 커뮤니티가 분산되며 각 도구의 생존 가능성은 낮아진다.
