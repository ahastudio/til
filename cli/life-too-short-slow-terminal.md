# 느린 터미널에 쓰기엔 인생은 너무 짧다

원문: https://mijndertstuij.nl/posts/life-is-too-short-to-write-to-slow-terminals/
HN 토론: <https://news.ycombinator.com/item?id=48424545> (14점, 6개 댓글)

## 요약

이 글은 쉘 시작 속도를 최적화하기 위한 핵심 전략으로 **덜어내기(subtraction)**를
제시한다. 저자는 약 30ms의 쉘 시작 시간을 달성했는데, 이는 무언가를 추가해서가
아니라 불필요한 프레임워크와 플러그인을 제거함으로써 이루어졌다.

핵심 기법들은 다음과 같다. oh-my-zsh나 prezto 같은 프레임워크를 쓰지 않고,
꼭 필요한 플러그인 3개만 수동으로 클론해 사용한다. `compinit`(zsh 자동완성
초기화)은 glob 한정자를 활용해 24시간마다만 완전히 실행되도록 캐시한다. nvm,
kubectl 등 시작 시 불필요한 도구들은 지연 로딩(lazy-loading)한다. Git 상태
표시를 위해 비동기 프롬프트(pure)를 사용해 프롬프트 렌더링이 쉘 시작을 블로킹
하지 않게 한다. 터미널 자체는 GPU 가속 Ghostty로 교체했다.

속도 측정 방법도 구체적으로 제시한다. 기본 측정은 `time zsh -i -c exit`이며,
100ms 미만은 수용 가능, 50ms는 우수한 수준이다. 정밀 측정을 위해서는 `hyperfine`
도구나 zsh 내장 `zprof` 프로파일러를 활용한다. 저자가 강조하는 세 가지 지연
요소는 시작 시간, 프롬프트 렌더링 시간, 입력 레이턴시다.

## 분석

### 덜어내기로서의 최적화

성능 개선을 이야기할 때 대부분의 글은 “무엇을 추가할까”에 집중한다. 더 빠른
파서, 더 효율적인 알고리즘, 더 나은 캐싱 레이어. 이 글은 반대 방향을 택한다.
이미 쓰고 있던 oh-my-zsh를 제거하고, 사용하지 않는 플러그인을 삭제하고, 시작
시에 필요 없는 도구의 초기화를 뒤로 미룬다.

이 접근법이 특히 쉘 설정에 효과적인 이유가 있다. 쉘은 매우 오래된 확장 메커니즘
(`.zshrc`에 소스하기)을 갖고 있어, 시간이 지날수록 “일단 추가”된 것들이 누적된다.
프레임워크(oh-my-zsh)는 편의성을 제공하지만 수백 개의 파일 소스와 함수 정의를
시작 시에 실행한다. 실제로 쓰는 기능이 그 중 일부라도.

### 측정 지표의 세분화가 중요하다

`time zsh -i -c exit`로 측정한 시작 시간이 전부가 아니라는 지적은 중요하다.
실제 사용자 경험에서 의미 있는 지표는:

1. 첫 프롬프트가 뜨기까지의 시간
2. 프롬프트가 뜬 후 Git 상태 등이 렌더링되기까지의 블로킹 여부
3. 타이핑 시작부터 화면에 반영되기까지의 입력 레이턴시

비동기 프롬프트가 중요한 이유가 여기 있다. Git 상태를 동기적으로 가져오면
저장소 크기에 따라 프롬프트 렌더링이 수백 ms 지연될 수 있다.

## 비평

### 프레임워크 비판은 맥락 의존적

oh-my-zsh 없이 zsh를 설정하는 것은 숙련된 사용자에게는 합리적인 선택이지만,
설정의 복잡성을 직접 관리해야 한다는 트레이드오프가 있다. 3개의 플러그인을
“수동으로 클론”해 관리하는 것은, 플러그인 관리자 없이 업데이트와 의존성을
직접 처리해야 함을 의미한다.

[GeekNews 댓글](https://news.hada.io/topic?id=30236)에서 제기된 지적처럼,
`time zsh -i -c exit`는 실제 사용 패턴과 다른 메트릭을 측정한다. 300ms 시작
시간을 감수하더라도 oh-my-zsh의 기능들을 유지하겠다는 결정도 타당하다.
Fish 쉘이 더 좋은 기본값을 제공한다는 대안도 언급되었다.

### 터미널 자체가 필요한가

이 글에 달린 가장 도발적인 반론은 zeflonex2의 댓글이다.[^zeflonex2] "터미널에 쓰기엔 인생이 너무
짧다. 터미널 도구들의 열풍을 이해 못 하겠다. 대부분의 경우 GUI 경험이 같거나
더 낫다." 이 반론은 이 글 전체의 전제를 흔든다. 쉘을 30ms로 만드는 최적화보다
근본적인 질문 — 그 30ms 쉘이 정말 필요한가 — 을 던지는 것이다.

터미널 주의자와 GUI 주의자의 간극은 좁혀지지 않는 취향의 차이이기도 하지만,
작업 유형에 따른 실질적 차이이기도 하다. 원격 서버 관리, 스크립트 자동화,
파이프라인 조합에서 터미널은 여전히 GUI가 대체하기 어렵다. 그러나 일반 개발
워크플로에서는 zeflonex2의 반론이 무게를 갖는다.

### GPU 터미널 전환의 비용

Ghostty 도입은 입력 레이턴시 개선에 기여하지만, GPU 가속 터미널은 그 자체로
새로운 의존성과 설정 복잡성을 도입한다. 미니멀리즘 철학과 다소 긴장 관계에 있다.

## 인사이트

### 도구 축적의 관성

개발자의 쉘 설정은 소프트웨어 아키텍처의 축소판이다. 처음에는 필요해서 추가한
것들이 점점 쌓이고, 나중에는 제거했을 때 어떤 문제가 생길지 몰라 그대로 두게
된다. oh-my-zsh는 처음에는 편의성을 위해 설치되지만, 시간이 지날수록 “제거하면
뭔가 안 될 것 같다”는 막연한 불안의 대상이 된다.

이 글이 유의미한 이유는 실제로 제거하고 측정했기 때문이다. 무엇이 실제로
필요한지 확인하는 과정 없이 두려움에 기반해 유지하는 것이 아니라, 데이터로
검증한다. 소프트웨어 설계에서도 동일한 원칙이 적용된다 — 레거시 코드를 두려움
때문에 건드리지 않는 것과, 실제로 이해하고 필요한 것만 남기는 것의 차이.

### 레이턴시 감각은 훈련된다

30ms와 300ms의 쉘 시작 시간 차이를 일반 사용자는 느끼지 못할 수도 있다. 그러나
터미널을 하루에 수십 번 여는 개발자에게 이 누적 지연은 인지적 마찰(cognitive
friction)을 만든다. 컴퓨터가 즉시 반응하지 않는다는 느낌, 그 미세한 지연이
쌓여 작업 흐름을 방해한다.

레이턴시에 민감한 감각을 갖는 것은 좋은 개발자 경험 설계의 출발점이다. 자신의
도구에서 불편함을 느끼고, 측정하고, 개선하는 반복이 결국 사용자에게 더 나은
제품을 만드는 감각으로 이어진다.

### 비동기화는 인지 모델을 바꾼다

동기 Git 상태 표시에서 비동기로 전환할 때, 사용자는 “프롬프트가 뜨면 아직
정보가 없을 수 있다”는 새로운 정신 모델을 받아들여야 한다. 이것은 단순한
성능 개선이 아니라, 사용자와 도구 간의 상호작용 모델 변화다.

이 트레이드오프는 소프트웨어 전반에서 반복된다. 사용자에게 즉각적인 피드백을
줄 것인가(동기), 아니면 더 정확한 최종 결과를 줄 것인가(비동기). 정답은
맥락에 따라 다르지만, 중요한 것은 이 선택이 사용자 경험에 미치는 영향을
의도적으로 설계하는 것이다.
