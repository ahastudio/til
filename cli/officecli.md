# OfficeCLI — AI 에이전트를 위한 Office 문서 편집 도구

<https://github.com/iOfficeAI/OfficeCLI>

HN 토론: <https://news.ycombinator.com/item?id=48807225> (156점, 43개 댓글)

## 소개

OfficeCLI는 Word, Excel, PowerPoint 파일을 Microsoft Office 설치 없이
읽고 편집할 수 있는 오픈소스 명령줄 도구다.
단일 바이너리로 배포되며, 라이선스는 Apache License 2.0이다.
프로젝트 스스로를 “AI 에이전트를 위한 Office 스위트”로 소개하고 있고,
실제로 MCP 서버 등록 명령을 내장해 Claude Code, Cursor, VS Code 같은
에이전트 환경에 바로 연결할 수 있게 설계되어 있다.
공식 웹사이트는 officecli.ai이며, 이 글을 쓰는 시점 기준 8.9k 스타,
642개 포크, 5,571개 커밋으로 활발히 개발되고 있다.

핵심 개념은 문서를 계층적 경로로 주소화하는 것이다.
`/slide[1]/shape[2]` 같은 경로로 요소를 지정하고,
`get`, `query`, `set`, `add`, `remove`, `move`, `swap` 같은 명령어로
문서를 조작한다.
모든 명령은 `--json` 플래그로 일관된 스키마의 출력을 반환하는데,
이는 사람이 읽기 위한 도구가 아니라 에이전트가 파싱하기 위한 도구라는
설계 의도를 분명히 보여준다.

## 주요 기능

### 3계층 아키텍처

OfficeCLI는 편집 작업을 세 단계로 나눈다.

- **L1(읽기)**: outline, 텍스트, 주석 처리된 뷰, 통계, 문제 목록, HTML,
  SVG, 스크린샷 등 문서를 사람이나 모델이 훑어보기 좋은 형태로 제공한다.
- **L2(DOM)**: 구조화된 요소 단위로 조작한다.
  `get`, `query`, `set`, `add`, `remove`, `move`, `swap` 명령이 여기에
  해당한다.
- **L3(원본 XML)**: OOXML을 XPath로 직접 건드리는 계층이다.
  `raw`, `raw-set`, `add-part`, `validate` 명령으로 L2가 다루지 못하는
  세부 구조까지 접근한다.

단순한 작업은 L1/L2로 끝내고, L2로 표현할 수 없는 예외적인 구조만 L3로
내려가는 구조다.

### 포맷별 지원 범위

Word, Excel, PowerPoint 세 포맷 모두 읽기·수정·생성을 지원한다.

| 포맷               | 주요 기능                                                                                          |
| ------------------ | --------------------------------------------------------------------------------------------------- |
| Word(.docx)        | 단락/표/스타일, 텍스트박스, 헤더·푸터, LaTeX 방정식, Mermaid 다이어그램, 각주, TOC, 추적 변경사항  |
| Excel(.xlsx)       | 350개 이상의 함수를 계산하는 포뮬라 엔진, VLOOKUP/XLOOKUP, 피벗 테이블, 조건부 서식, 스파크라인    |
| PowerPoint(.pptx)  | 애니메이션(강조 15종, 종료 16종), 모프 전환, 3D 모델(.glb), 슬라이드 줌, 테마, 커넥터              |

세 포맷 모두에 내장 HTML 렌더링 엔진이 붙어 있어서, Office 없이도
HTML/스크린샷/라이브 프리뷰(`watch` 명령)로 결과물을 바로 확인할 수 있다.

### 배치 처리와 템플릿 병합

`batch` 명령은 JSON으로 정의된 여러 작업을 한 번에 실행하고,
`merge` 명령은 `{{key}}` 형태의 자리 표시자를 문서 템플릿에 채워 넣는다.
`dump` 명령은 기존 문서를 재생 가능한 JSON으로 직렬화하는데,
이는 문서 구조를 학습하거나 그대로 재현하는 데 쓰인다.
`validate` 명령은 OpenXML 스키마 검증을 수행해서, 에이전트가 스스로
결과물을 점검하고 고치는 자가 치유 워크플로를 가능하게 한다.

## 분석

### 경로 기반 주소 지정은 “에이전트가 실수해도 복구 가능한” 설계다

OfficeCLI가 모든 요소에 `/slide[1]/shape[2]` 같은 안정된 경로를 부여한
것은 단순한 편의 기능이 아니다.
LLM 에이전트는 한 번의 호출로 완벽한 편집을 끝내지 못하고,
조회와 수정을 여러 차례 반복하며 상태를 확인해야 한다.
경로가 안정적이면 에이전트는 이전 단계에서 얻은 경로를 다음 단계에서도
재사용할 수 있고, 중간에 실패해도 같은 경로로 재시도하거나 검증할 수
있다.
반대로 경로가 매 호출마다 달라지거나 문서 내부 구현에 종속된다면,
에이전트는 매번 문서 전체를 다시 훑어야 하고 오류 복구 비용이 커진다.

이 설계는 결국 “에이전트를 신뢰하지 않는” 전제 위에 서 있다.
`validate`와 `view issues` 같은 명령을 별도로 마련한 것도 같은 맥락이다.
에이전트가 만든 결과물이 항상 옳다고 가정하지 않고,
스스로 검증하고 고칠 수 있는 도구를 붙여준 것이다.

### 렌더링 엔진 내장은 “확인 없는 편집”의 위험성을 인정한 결과다

OfficeCLI가 HTML/스크린샷/라이브 프리뷰를 핵심 기능으로 넣은 이유는
명확하다.
XML 구조를 직접 조작하는 도구는 논리적으로는 올바른 명령을 내려도
시각적으로는 텍스트가 넘치거나, 표가 깨지거나, 차트가 겹치는 결과를
낳을 수 있다.
HN 댓글에서도 여러 사용자가 PowerPoint 작업에서 시각적 피드백이
필수적이라는 점을 지적했는데, 이는 OpenXML을 다루는 도구 일반이
공유하는 근본적인 제약이다.
문서 형식 자체가 사람의 눈으로 확인하는 것을 전제로 설계되었기 때문에,
그 확인 과정을 생략한 자동화는 항상 결과물의 품질을 장담할 수 없다.

OfficeCLI가 이 문제를 해결한 것은 아니고, 렌더링 엔진을 붙여서 확인
비용을 낮췄을 뿐이다.
에이전트가 실제로 렌더링 결과를 보고 판단을 내리는 루프를 구성하지
않으면, 렌더링 엔진의 존재 자체는 안전성을 보장하지 않는다.

### MCP 서버 통합은 이 도구가 겨냥하는 사용자를 분명히 보여준다

`officecli mcp claude`, `officecli mcp cursor` 같은 명령으로 에이전트
환경에 스스로 등록되는 기능은, 이 도구가 사람의 CLI 사용 경험이 아니라
에이전트의 도구 호출 경험을 최우선으로 설계했다는 뜻이다.
바이너리를 설치하면 Claude Code, Cursor, VS Code의 설정 디렉터리를
자동으로 감지해서 스킬 파일을 심는 방식은, 사람이 문서를 짜맞추는
게 아니라 에이전트가 스스로 도구를 발견하고 사용법을 익히는 흐름을
전제한다.
이는 최근 개발 도구들이 “사람이 먼저 배우고 에이전트에게 알려주는”
방식에서 “도구가 에이전트에게 스스로를 알리는” 방식으로 옮겨가는
흐름과 맞닿아 있다.

## 비평

### 검증은 스키마 준수를 확인할 뿐, 의미론적 정확성은 보장하지 않는다

`validate` 명령이 확인하는 것은 OpenXML 스키마 준수 여부다.
문서가 스키마에 맞게 구성되어 있는지는 확인해 주지만,
그 문서가 사용자가 의도한 내용을 정확히 담고 있는지는 별개의 문제다.
예를 들어 에이전트가 수식을 잘못된 셀 범위에 적용했거나,
표의 헤더와 데이터 행이 뒤바뀌었거나, 템플릿 병합에서 잘못된 값이
들어갔더라도 스키마상으로는 완전히 유효한 문서가 만들어질 수 있다.
스키마 검증을 “자가 치유”라는 이름으로 부르는 것은 실제로 제공하는
보장 범위보다 과장된 인상을 준다.
HN에서 StahlGuo가 지적한 “검증, 인용, 속성 추적이 실제 조직에서 더
중요하다”는 비판도 같은 지점을 겨냥한다.
스키마가 유효한 문서와 내용이 정확한 문서는 전혀 다른 문제인데,
이 도구가 강조하는 검증은 전자에 머물러 있다.

### 추적 변경사항과 매크로를 다루는 자동 편집은 되돌릴 수 없는 상태 손상을 일으킬 수 있다

Word의 추적 변경사항(track changes)은 단순한 텍스트가 아니라 편집
이력과 저자 정보를 포함하는 구조화된 상태다.
에이전트가 L2나 L3 계층에서 문단을 옮기거나 텍스트를 치환할 때,
그 문단에 걸린 추적 변경 마크업이나 댓글 스레드를 온전히 유지할지,
아니면 조용히 깨뜨릴지는 README만으로는 확인하기 어렵다.
Excel의 매크로나 명명 범위, PowerPoint의 애니메이션 시퀀스도 마찬가지다.
이런 요소들은 문서를 열어 눈으로 보기 전까지는 손상 여부를 알기 어려운
“숨은 상태”이고, `view`나 `dump`가 이 숨은 상태를 얼마나 충실히
드러내는지가 실제 안전성을 좌우한다.
에이전트가 대량의 배치 작업(`batch`)을 자동으로 실행하는 워크플로에서는
이런 숨은 상태 손상이 검증 없이 누적될 위험이 특히 크다.
즉, 겉보기에는 정상적으로 렌더링되는 문서가 매크로나 추적 변경 이력
차원에서는 이미 훼손되어 있을 가능성을 이 도구의 설계만으로는 배제할
수 없다.

### 350개 함수 자동 계산이라는 주장은 Excel과의 완전한 호환을 의미하지 않는다

README는 포뮬라 엔진이 350개 이상의 함수를 계산하고 VLOOKUP, XLOOKUP,
동적 배열 함수를 지원한다고 밝히지만, Excel의 실제 계산 엔진은 수천
가지 조합의 부동소수점 처리, 순환 참조 처리, 배열 확장 규칙을 갖고
있다.
자체 구현한 포뮬라 엔진이 이 모든 경계 사례를 Excel과 동일하게
재현한다고 보장할 근거는 README에 제시되어 있지 않다.
에이전트가 이 도구로 재무 모델이나 복잡한 피벗 테이블을 생성했을 때,
실제 Excel에서 열었을 때 계산 결과가 미묘하게 달라질 위험은 사용자가
직접 검증하기 전까지 드러나지 않는다.
이는 라이선스 문제로 논쟁이 된 smalldocs 사례와 마찬가지로,
“기능 목록의 풍부함”과 “그 기능이 실제 프로덕션 환경에서 신뢰할 수
있는가”는 서로 다른 질문이라는 점을 보여준다.

rcarmo는 ECMA 376 호환성 테스트 케이스가 보이지 않는다는 점을 지적하며,
이미 1년 전에 유사한 작업을 시작했고 `python-office-mcp-server`와
`go-ooxml` 같은 선행 프로젝트가 ECMA 376 준수를 기준으로 개발됐다고
언급했다.[^rcarmo]
기능 목록이 넓다는 것과 표준 규격을 얼마나 정확하게 구현했는가는
별개의 질문이며, 후자에 대한 검증이 없다면 프로덕션 신뢰성은
주장에 머물 수밖에 없다.

## 인사이트

### “결정론적 JSON 출력”은 에이전트 시대의 도구가 사람보다 기계를 먼저 설계하는 신호다

전통적인 CLI 도구는 사람이 읽기 좋은 텍스트 출력을 기본으로 하고,
기계가 파싱하기 좋은 형식은 부가 옵션(`--json`)으로 취급해 왔다.
OfficeCLI는 모든 명령에 `--json`을 지원하되, 그 스키마의 일관성 자체를
핵심 판매 포인트로 내세운다.
이는 이 도구의 1차 사용자가 사람이 아니라 에이전트라는 사실을 설계
철학 차원에서 인정한 것이다.
앞으로 이런 도구들이 늘어나면, “사람이 읽기 좋은 출력”과 “에이전트가
파싱하기 좋은 출력”이 분리되는 것을 넘어, 아예 사람이 직접 다루지
않는 것을 전제로 설계된 CLI라는 새로운 범주가 자리 잡을 가능성이 크다.
그렇게 되면 도구의 품질을 평가하는 기준도 “사용성”에서 “에이전트가
얼마나 적은 시행착오로 원하는 결과에 도달하는가”로 옮겨갈 것이다.

### 문서 형식의 “숨은 상태”는 코드 리팩터링과 다른 종류의 회귀 위험을 만든다

소프트웨어 코드를 리팩터링할 때는 테스트 스위트로 동작의 동등성을
어느 정도 검증할 수 있다.
그러나 Office 문서의 매크로, 추적 변경 이력, 명명 범위, 조건부 서식
규칙 같은 요소는 테스트로 자동 검증하기 어려운 “숨은 상태”이며,
이 상태들은 서로 얽혀 있어서 한 부분을 편집하면 다른 부분이 조용히
깨질 수 있다.
코드 리팩터링의 회귀는 대체로 실행 시점에 드러나지만, 문서의 회귀는
사람이 열어서 재확인하기 전까지 드러나지 않는 경우가 많다.
에이전트가 문서 편집을 대량으로 자동화할수록, 이 “열어보기 전까지는
모르는” 회귀가 누적되어 결국 사람이 감사(audit)하는 비용이 자동화로
아낀 시간을 상쇄할 위험이 있다.
이는 코드 자동화에서 이미 논의된 “자율성을 높일수록 검증이 느슨해져
사고로 이어진다”는 패턴이 문서 자동화 영역에서도 똑같이 반복될 수
있음을 시사한다.

### "왜 Office인가, LaTeX이 더 나은 선택 아닌가"라는 질문은 이 도구의 전제를 건드린다

beepbooptheory는 LLM이 문서를 생성할 때 Office 형식보다 LaTeX 같은
구조화된 형식이 더 자연스러운 선택이 아닌지 물었다.[^beepbooptheory]
이 질문은 OfficeCLI가 해결하려는 문제의 전제, 즉 "기업 환경에서
결과물은 Office 형식이어야 한다"는 제약을 겨냥한다.
karado가 지적했듯 대부분의 기업 현장은 Office 스위트에 깊이 의존하고
있으며, LaTeX는 그 의존을 끊지 않는 한 대안이 될 수 없다.[^karado]
pietz는 인터랙티브·애니메이션 기능이 필요 없다면 에이전트가 HTML로
슬라이드를 만들고 PDF로 변환하는 방식이 실질적인 대안이 된다고
제안했다.[^pietz]
결국 이 질문은 "더 나은 형식"의 문제가 아니라 워크플로와 협업 대상이
무엇인가라는 맥락 문제이며, Office 형식의 필요성은 기술적 우열이 아닌
조직 관성에서 나온다.

### 라이선스 갈림은 이 생태계가 아직 표준을 정하지 못했다는 신호다

HN 댓글에서 지적된 것처럼, 유사한 프로젝트인 smalldocs는 Elastic
라이선스를, OfficeCLI는 Apache 2.0을 택했다.
이 차이는 단순한 법률적 세부사항이 아니라, “에이전트용 Office 도구”라는
새로운 카테고리가 아직 상업적 포지셔닝조차 합의되지 않은 초기 단계임을
보여준다.
오픈소스 라이선스 선택은 프로젝트가 스스로를 인프라로 보는지,
아니면 보호해야 할 상업적 자산으로 보는지를 드러내는 신호인데,
이 카테고리에서 서로 다른 답이 나온다는 것은 아직 누구도 “이 종류의
도구는 이렇게 취급되어야 한다”는 업계 관행을 만들지 못했다는 뜻이다.
Org Mode 같은 기존 텍스트 포맷을 활용하자는 lelanthran의 제안 역시,
새 포맷과 새 도구를 만드는 대신 이미 검증된 형식 위에서 에이전트
호환성을 확보하자는 대안적 접근이며, 이 갈림은 앞으로 유사한 도구들이
등장할 때마다 반복될 근본적인 설계 선택으로 남을 것이다.
