# Hexana: JetBrains의 WebAssembly/바이너리 분석 툴킷

<https://jetbrains.github.io/hexana/>

<https://github.com/JetBrains/hexana>

## 소개

Hexana는 JetBrains가 개발한 WebAssembly 및 바이너리 분석 툴킷이다.
IntelliJ 플랫폼 플러그인과 VS Code 확장이라는 두 가지 제품으로 제공되며,
둘 다 Kotlin Multiplatform으로 작성된 공용 분석 코어를 공유한다.

핵심 목적은 개발자가 WebAssembly 파일을 네이티브 바이너리와 함께
검사하고, 편집하고, 디버깅할 수 있게 하는 것이다.
여러 런타임 환경과 개발 워크플로를 통합 지원한다.

두 제품이 공통으로 갖는 기능은 다음과 같다.

- Core Wasm, Component Model, GC, SIMD, Threads 등 여러 확장 제안을 지원하는
  WASM 파서
- import/export, 함수 카탈로그, 크기 프로파일링, 데드 코드 탐지를 포함한
  바이너리 분석 기능
- 의존성 해석을 포함한 Component Model 인식
- ELF, Mach-O, PE 바이너리에 대한 실험적 지원
- Wasmtime, WAMR, GraalVM 실행 지원과 프로포절 플래그 자동 감지

JetBrains 플러그인 전용 기능은 다음과 같다.

- 편집 가능한 WAT 뷰를 포함한 멀티탭 `.wasm` 에디터
- 고급 편집을 지원하는 WIT 언어 지원
- AI 어시스턴트를 위한 MCP 서버 통합
- GraalWasm과 Chicory를 위한 Java 자동 완성
- WebAssembly 인스턴스화를 위한 JavaScript/TypeScript 타입 추론
- `.class`, `.jar`, `.jit` 파일을 위한 JVM 아티팩트 뷰어
- 교체 가능한 디스어셈블러 백엔드

VS Code 확장 전용 기능은 다음과 같다.

- Compose-for-Web 기반 커스텀 에디터
- 가상 스크롤 헥스 뷰어
- 11개의 구조 분석 탭
- 빠른 검사를 위한 더 가벼운 리소스 사용량

JetBrains 플러그인은 버전 0.13(IDE 2025.2 이상 지원), VS Code 확장은
버전 0.6.0(VS Code 1.102 이상 지원)이다.
GitHub 저장소는 이 프로젝트를 “제한된 리소스로 개발된 실험적 MVP”로
소개하며, 프로덕션 완성도보다 아키텍처 탐색을 우선시한다는 점을 명시하고
있다.
실제 사용 사례로부터의 피드백이 특히 가치 있다고 강조하며, 어떤 실험적
기능을 프로덕션 수준으로 끌어올릴지 커뮤니티 의견을 통해 정한다는 방침도
밝히고 있다.

## 분석

### 하나의 코어, 두 개의 표면

Hexana의 구조적 선택은 “Kotlin Multiplatform 분석 코어 하나에 두 개의
에디터 표면을 얹는다”는 것이다.
이 선택은 우연이 아니다.
WASM 파서, 바이너리 분석기, Component Model 해석기 같은 핵심 로직은
IDE와 무관한 순수 도메인 로직이다.
반면 에디터 UI, 완성 기능, 디버거 연동은 각 플랫폼(IntelliJ, VS Code)의
확장 API에 강하게 종속된다.
코어와 표면을 분리하면 파서나 분석기의 버그 수정, 새 WASM 프로포절 지원
같은 작업이 한 번의 구현으로 두 제품에 동시에 반영된다.

동시에 이 구조는 두 제품의 기능 격차를 설명한다.
JetBrains 플러그인이 MCP 서버 통합, JVM 아티팩트 뷰어, 타입 추론처럼
훨씬 무거운 기능을 갖는 반면 VS Code 확장은 헥스 뷰어와 구조 분석 탭
중심의 가벼운 도구로 남는다.
이는 표면 계층이 코어와 분리되어 있기 때문에 가능한 비대칭이다.
공용 코어가 있다고 해서 두 제품이 동등한 기능을 가질 필요는 없다.
오히려 각 IDE의 사용자층과 워크플로에 맞춰 표면을 다르게 설계할 수
있다는 것이 이 아키텍처의 실질적 이점이다.

### WASM 도구가 “바이너리 분석 툴킷”으로 확장된 이유

Hexana라는 이름 자체가 이미 힌트를 준다.
Hex(십육진수) 뷰어에서 출발한 도구가 WASM 전용을 넘어 ELF, Mach-O, PE
같은 네이티브 바이너리 포맷까지 실험적으로 지원하는 것은 자연스러운
확장이다.
WASM 모듈은 실제 배포 환경에서 흔히 네이티브 코드와 함께 존재한다.
Emscripten으로 컴파일된 C/C++ 코드, wasm-bindgen을 통한 Rust
바인딩처럼 하나의 애플리케이션이 WASM과 네이티브 바이너리를 넘나드는
경우가 많다.

개발자가 이런 혼합 환경을 디버깅할 때 WASM 파일만 볼 수 있는 도구는
반쪽짜리다.
Hexana가 JVM 아티팩트(`.class`, `.jar`, `.jit`)까지 뷰어를 제공하는
것도 같은 맥락이다.
JetBrains의 주력 제품이 JVM 생태계(IntelliJ IDEA)에 뿌리를 두고 있다는
점을 생각하면, GraalWasm이나 Chicory처럼 JVM 위에서 WASM을 실행하는
런타임과의 통합은 JetBrains만이 가질 수 있는 경쟁 우위다.

### “실험적 MVP”라는 선언이 하는 일

GitHub 저장소가 스스로를 “제한된 리소스로 개발된 실험적 MVP”라고
명시적으로 규정하는 것은 단순한 겸손이 아니라 전략적 포지셔닝이다.
이 선언은 사용자의 기대치를 낮추는 동시에 이슈 트래커의 성격을
규정한다.
저장소는 일반 IntelliJ 이슈나 WebAssembly 스펙 논쟁을 명시적으로
제외하고, Hexana 고유의 피드백만 받겠다고 밝힌다.
이는 큰 조직(JetBrains)이 작은 실험적 프로젝트를 운영할 때 흔히
마주치는 문제, 즉 이슈 트래커가 관련 없는 요청으로 뒤덮이는 것을
사전에 차단하려는 장치다.

동시에 “실제 사용 사례가 특히 가치 있다”는 표현은 이 프로젝트가
아직 자체 로드맵보다 사용자 피드백에 의해 방향이 결정되는 단계임을
보여준다.
버전 번호가 0.13과 0.6.0으로 1.0에 도달하지 못한 것도 이와 일치한다.
JetBrains 같은 대기업이 사이드 프로젝트 성격의 도구를 공개 저장소로
운영하며 커뮤니티 신호를 수집하는 방식은, 신규 개발자 도구 시장에서
정식 출시 전 검증 단계를 거치는 흔한 패턴이다.

## 비평

### “두 제품”이라는 표현이 가리는 유지보수 부담

공식 소개는 두 제품이 “공용 코어를 공유”한다는 점을 장점으로
내세우지만, 실제로는 IDE별 표면 UI를 각각 유지보수해야 하는 구조다.
공유 코어가 커버하는 것은 파서와 분석 로직뿐이고, 에디터 통합, 완성
기능, 디버거 연동 같은 사용자 체감 기능은 IntelliJ 플랫폼 API와 VS
Code 확장 API라는 서로 다른 두 개의 확장 생태계 위에서 별도로
구현해야 한다.
JetBrains 플러그인 쪽 기능 목록이 VS Code보다 훨씬 풍부한 이유도
이것으로 설명된다.
결국 “코어 공유”는 로직 중복을 줄여주지만, 두 플랫폼의 UI 계층을
동시에 따라잡아야 하는 부담 자체를 없애주지는 않는다.
장기적으로 한쪽 제품이 뒤처지거나 폐기될 위험은 이 구조에서 상시
존재하는 조건이다.

### 실험적 지원 범위가 실제 필요와 어긋날 가능성

ELF, Mach-O, PE에 대한 지원을 “실험적”이라고 표시한 것은 정직하지만,
이 세 포맷을 동시에 실험 범위에 포함시킨 선택 자체는 검증되지 않은
가정에 기반한다.
WASM과 네이티브 바이너리가 섞이는 실제 시나리오의 대부분은 Emscripten이
만드는 WASM+JS 조합이나 wasm-bindgen이 만드는 WASM+Rust 조합에서
발생한다.
이 경우 필요한 것은 ELF(Linux)나 Mach-O(macOS) 지원이지 세 포맷
모두는 아닐 수 있다.
세 가지 네이티브 포맷을 동시에 실험적으로 지원하겠다는 선언은 리소스가
제한된 프로젝트가 스코프를 넓게 잡는 전형적인 위험 신호다.
어떤 포맷이 실제로 쓰이는지에 대한 사용 데이터 없이 세 포맷을 병렬로
유지보수하면, 어느 것도 프로덕션 수준에 도달하지 못한 채 장기간
실험 단계에 머무를 수 있다.

### MCP 서버 통합의 성숙도가 검증되지 않았다

JetBrains 플러그인의 “AI 어시스턴트를 위한 MCP 서버 통합” 기능은
소개 문구만으로는 실제로 무엇을 할 수 있는지 알기 어렵다.
바이너리 분석 도구가 MCP를 통해 AI 어시스턴트에 제공할 수 있는 정보는
함수 카탈로그, import/export 목록, Component Model 의존성 그래프 같은
구조화된 데이터일 것으로 추정되지만, 이것이 실제로 코드 생성이나
디버깅 워크플로에 어떤 가치를 더하는지는 공개된 자료만으로 검증할 수
없다.
“AI 어시스턴트 지원”이라는 문구가 최근 개발자 도구 마케팅에서 거의
필수 항목처럼 등장하는 흐름을 고려하면, 이 기능이 실질적 워크플로
개선인지 아니면 시류에 맞춘 부가 기능인지 구분하려면 실제 사용
사례와 문서화가 더 필요하다.

## 인사이트

### 대기업의 “실험적 MVP” 공개는 채용과 커뮤니티 형성의 부산물이기도 하다

JetBrains 같은 회사가 리소스가 제한된 실험적 프로젝트를 공개
저장소로 운영하는 방식은 표면적으로는 사용자 피드백 수집이 목적이지만,
동시에 이는 회사 내부의 사이드 프로젝트를 외부화해 그 프로젝트에
참여하는 엔지니어의 가시성과 동기를 높이는 효과를 낸다.
공개된 실험적 프로젝트는 사내에서 조용히 종료되기 어렵다.
사용자와 기여자가 생기는 순간 그 프로젝트는 조직 내부의 정치적
무게를 갖게 되고, 이는 역설적으로 “실험적”이라는 딱지가 붙은
프로젝트의 생존 확률을 오히려 높이는 방향으로 작용할 수 있다.
WASM처럼 아직 표준 도구 체계가 확립되지 않은 신생 생태계에서 이런
초기 공개 전략은 향후 그 생태계의 사실상 표준 도구 자리를 선점하려는
움직임으로도 읽을 수 있다.

### WASM 개발 도구의 경쟁은 “누가 먼저 IDE에 통합하는가”로 이동하고 있다

WASM 생태계는 지금까지 CLI 도구(wasm-tools, wabt, wasmtime)와 런타임
중심으로 발전해 왔다.
Hexana는 이 흐름에서 한 단계를 더 밟는다.
CLI로 검사하던 작업을 IDE 안의 멀티탭 에디터, 구조 뷰, 디스어셈블러로
끌어들이는 것이다.
이 이동은 과거 Java나 C++ 생태계가 컴파일러와 별도 디버거 중심에서
IDE 통합 디버깅과 리팩터링 도구로 넘어간 궤적과 구조적으로
닮았다.
어떤 언어나 바이너리 포맷이든, 개발자 커뮤니티가 어느 정도
성숙해지면 다음 단계는 항상 CLI에서 IDE로의 이동이었다.
WASM이 지금 이 전환점에 있다는 신호로 Hexana의 등장을 읽을 수 있다.
문제는 이 전환에서 승자가 되는 도구가 반드시 기술적으로 가장 뛰어난
도구가 아니라, 기존 IDE 사용자 기반을 가장 크게 가진 도구라는 점이다.
JetBrains와 Microsoft(VS Code)가 이미 개발자 도구 시장을 양분하고
있는 상황에서, 신생 독립 WASM 전용 IDE가 이 자리를 놓고 경쟁하기는
갈수록 어려워진다.

### 바이너리 포맷을 넘나드는 통합 뷰어는 디버깅 경계를 흐리게 만든다

WASM, ELF, Mach-O, PE, JVM 아티팩트를 하나의 도구에서 검사할 수
있다는 것은 편의성 측면에서는 분명한 이득이지만, 동시에 디버깅 작업의
심리적 경계를 흐리게 만드는 부작용을 낳을 수 있다.
개발자가 “지금 내가 보고 있는 것이 WASM 모듈의 문제인지, 네이티브
코드의 문제인지, JVM 런타임의 문제인지”를 구분하는 인지적 노력은
원래 도구가 분리되어 있을 때 자연스럽게 강제됐다.
하나의 통합 뷰어에서 모든 포맷을 오갈 수 있게 되면 이 구분이
흐려지고, 실제로는 다른 계층에서 발생한 문제를 잘못된 계층에서
찾으려는 시행착오가 늘어날 위험이 있다.
통합 도구가 주는 편의성과, 계층 간 경계가 강제하던 사고의 명료함
사이에는 실제로 트레이드오프가 존재하며, 이는 도구가 성숙해질수록
UI 설계에서 더 신중하게 다뤄야 할 문제다.
