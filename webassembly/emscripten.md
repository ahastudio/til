# Emscripten: C/C++를 WebAssembly로 컴파일하는 툴체인

<https://emscripten.org/>

<https://github.com/emscripten-core/emscripten>

## 소개

Emscripten은 C와 C++ 코드를 WebAssembly(WASM)로 컴파일하는 오픈소스 컴파일러
툴체인이다. LLVM과 Binaryen을 핵심 컴포넌트로 사용하며, 컴파일된 결과물은 웹
브라우저, Node.js, 독립형 WASM 런타임 등 다양한 환경에서 실행된다.

핵심 진입점은 `emcc`(Emscripten Compiler Frontend)다. 이것은 `gcc`나 `clang`의
드롭인(drop-in) 대체제로 동작한다. C/C++ 코드를 컴파일할 때 기존 빌드 스크립트의
컴파일러만 `emcc`로 바꾸면 대부분의 포팅 작업이 시작된다. 최종 출력은 `.wasm`
파일과 이를 브라우저에서 로드하고 실행하는 JavaScript 글루 코드(glue code)로
구성된다.

최적화 파이프라인은 LLVM, Binaryen, Closure Compiler가 협력한다. WebAssembly는
“네이티브 코드에 근접한 속도로 실행되는 고도로 최적화 가능한 실행 형식”으로
설명된다. OpenGL과 SDL2 같은 이식성 높은 API를 웹 구현으로 제공해 그래픽 집약적
애플리케이션도 브라우저에서 실행할 수 있다.

실제 포팅 성공 사례로 Unreal Engine 4, Unity, Google Earth가 있다. C나 C++로
작성된 사실상 모든 이식성 있는 코드베이스를 웹으로 가져올 수 있다는 것이
프로젝트의 주장이다. Qt 같은 응용 프레임워크도 포함된다.

포팅에 주의가 필요한 부분도 있다. C 표준 라이브러리와 C++ 표준 라이브러리,
예외 처리는 폭넓게 지원되지만, 메인 루프 구조와 파일 처리는 브라우저 환경에
맞게 수정해야 한다. 브라우저는 동기 블로킹 루프를 허용하지 않기 때문에 이벤트
루프 기반으로 재구성이 필요하다.

## 분석

### C/C++ 생태계를 웹으로 이식하는 다리

Emscripten의 기술적 의미는 단순한 컴파일러를 넘어선다. 수십 년간 축적된 C/C++
코드베이스 — 게임 엔진, 물리 시뮬레이터, 이미지 처리 라이브러리, 암호화 구현,
과학 계산 코드 — 를 웹 플랫폼에서 재사용할 수 있게 만드는 다리다.

이 다리가 작동하는 방식은 세 층위로 나뉜다. 첫째, 컴파일러 층에서 LLVM이
C/C++를 플랫폼 독립적인 중간 표현(IR)으로 변환하고, Binaryen이 이를 최적화된
WebAssembly로 변환한다. 둘째, API 호환성 층에서 OpenGL 호출을 WebGL 호출로,
파일 시스템 접근을 가상 파일 시스템(VFS)으로 변환하는 에뮬레이션 계층이 동작한다.
셋째, 런타임 층에서 메모리 관리, 스레드 에뮬레이션, 예외 처리를 담당하는
JavaScript 글루 코드가 생성된다.

### 간접 포팅의 파급력

README에서 언급하는 “다른 언어의 런타임을 WebAssembly로 컴파일하는” 간접 포팅
전략은 Emscripten의 영향력을 C/C++ 너머로 확장한다. Pyodide(Python), Lua,
Ruby, PHP 등이 이 방식으로 브라우저에서 실행된다. C/C++로 작성된 언어 인터프리터나
런타임을 Emscripten으로 컴파일하면 해당 언어 전체가 브라우저에서 동작한다.

이 전략은 WebAssembly를 “범용 실행 환경”으로 향하게 한다. 브라우저가 단순히
JavaScript를 실행하는 것이 아니라, 적절한 런타임이 주어지면 어떤 언어로 작성된
코드도 실행할 수 있는 플랫폼이 된다.

### LLVM 생태계의 수혜자

Emscripten이 WebAssembly 타깃으로 작동하는 것은 Rust의 `wasm32-unknown-emscripten`
타깃 지원에서 보듯이 LLVM 생태계 전체에 열려 있다. C/C++만이 아니라 LLVM
프론트엔드를 가진 모든 언어가 잠재적 수혜자다. Go, Swift, Kotlin/Native 등도
다양한 경로로 WebAssembly를 타깃으로 삼는다. Emscripten은 이 생태계의 선구자이자
레퍼런스 구현이다.

## 비평

### 강점: 성숙도와 검증된 사례

GitHub 2.7만 스타, 3.5천 포크라는 수치보다 더 중요한 것은 Unity와 Google Earth
같은 대규모 상용 소프트웨어의 실제 배포 사례다. “이론적으로 가능하다”가 아니라
“실제로 배포되어 수억 명이 사용한다”는 증거다. 이 성숙도는 에지 케이스 처리,
디버깅 도구, 커뮤니티 지식 축적에서 나타난다.

### 약점: 포팅의 실제 복잡성

“드롭인 대체제”라는 표현은 다소 낙관적이다. 메인 루프 재구조화, 비동기 파일
I/O 적응, 스레딩 모델 변경은 간단한 플래그 변경이 아니라 아키텍처 수준의
수정이다. 특히 멀티스레드 코드는 SharedArrayBuffer와 Atomics에 의존하는
WebAssembly 스레드 모델로 재작성이 필요하다. 마케팅과 실제 작업량 사이의
간극이 존재한다.

### 약점: 번들 크기와 초기 로딩

WebAssembly 바이너리는 JavaScript에 비해 초기 파싱이 빠르지만, C++ 표준
라이브러리와 Emscripten 런타임이 포함되면 번들 크기가 상당히 커진다. 네이티브
JavaScript로 구현하면 수십 킬로바이트면 충분한 기능이 Emscripten 포팅 후
수 메가바이트가 되는 경우가 있다. 대형 애플리케이션에서는 이 오버헤드가 무시할
만하지만, 소규모 라이브러리 포팅에서는 비용-편익 계산을 신중히 해야 한다.

## 인사이트

### 플랫폼 전쟁의 새 전선: 런타임 포터빌리티

소프트웨어 역사에서 플랫폼 전쟁은 반복된다. Windows vs Mac, iOS vs Android,
모든 전쟁의 핵심은 “어떤 플랫폼에서 더 많은 소프트웨어가 실행되는가”다.
Emscripten은 이 전쟁에서 근본적으로 다른 전략을 구현한다. 새 플랫폼(웹)이
기존 플랫폼의 소프트웨어를 전부 흡수할 수 있다면, 플랫폼 전쟁 자체가 무의미해진다.

웹 브라우저는 이미 가장 널리 설치된 소프트웨어 플랫폼이다. Emscripten이 성숙하면
할수록, C/C++ 코드베이스를 보유한 기업들은 별도의 네이티브 앱 없이도 웹을 통해
자신의 소프트웨어를 배포할 수 있게 된다. 이것은 운영체제 벤더, 앱 스토어, 패키지
관리자의 영향력을 장기적으로 약화시키는 기술적 압력이다.

Unity와 Google Earth의 웹 배포는 이 방향의 초기 사례다. 포토샵이 웹으로 왔고,
AutoCAD도 왔다. “웹으로 올 수 없는 소프트웨어”의 목록이 Emscripten이 발전할수록
줄어들고 있다. 이 추세의 끝에 무엇이 있는지는 아직 불분명하지만, 플랫폼 패권의
지형이 변화하고 있다는 것은 분명하다.

### LLVM이 만드는 컴파일러 생태계의 중력

Emscripten이 Rust의 `wasm32-unknown-emscripten` 타깃을 자연스럽게 지원한다는
사실은 LLVM 중심의 컴파일러 생태계가 얼마나 강력한 네트워크 효과를 만드는지
보여준다. LLVM을 프론트엔드로 사용하는 언어라면 WebAssembly 타깃으로 가는 경로가
이미 열려 있다.

이것은 단순한 기술적 호환성이 아니다. 새 언어를 설계하는 사람들이 LLVM 프론트엔드
구현을 선택하면, 그 순간부터 Emscripten이 지원하는 모든 타깃에서 코드를 실행할
수 있게 된다. WebAssembly도, ARM도, x86도 “무료”로 얻는다. 이 중력은 신생
언어들을 LLVM 생태계로 끌어당긴다. 10년 뒤 어떤 언어가 살아남을지를 예측하는
변수 중 하나로 “LLVM 생태계 진입 여부”가 들어가야 할지도 모른다.

Emscripten은 이 생태계에서 WebAssembly라는 “새 타깃”을 LLVM의 일급 시민으로
만든 핵심 구현이다. Emscripten이 없었다면 WebAssembly 생태계의 성숙은 훨씬
느렸을 것이다. 이 프로젝트의 역사적 역할은 단순한 컴파일러 툴체인을 넘어, LLVM과
WebAssembly 두 생태계를 연결하는 다리를 놓은 것이다.

### 레거시 코드의 수명 연장이라는 조용한 혁명

소프트웨어 산업에서 레거시 코드는 골칫거리로 취급받는다. 유지보수 비용이 높고,
현대적 플랫폼과 통합이 어렵고, 새 인재가 기피한다. 기업들은 레거시 시스템을
현대화하기 위해 엄청난 비용을 투입한다. 종종 재작성은 실패하고, 원래 코드보다
나빠지기도 한다.

Emscripten은 이 문제에 다른 답을 제시한다. C/C++로 작성된 30년 된 코드가
지금 브라우저에서 실행될 수 있다면, 그것은 레거시가 아니라 자산이다. 수백만
줄의 검증된 수치 계산 코드, 물리 시뮬레이션, 이미지 처리 알고리즘이 재작성
없이 웹 플랫폼에서 계속 가치를 창출한다.

이 관점에서 Emscripten이 하는 일은 레거시 코드의 수명을 연장하는 것이다.
재작성의 리스크 없이, 검증된 구현을 새 플랫폼으로 가져온다. 소프트웨어 공학에서
“검증된 알고리즘의 새 구현은 종종 버그를 새로 도입한다”는 것은 잘 알려진 사실이다.
Emscripten이 가능하게 하는 “포팅 없는 포팅”은 이 리스크를 원천 차단한다.
수십 년간 수백만 명이 사용하며 버그가 제거된 코드를 그대로 가져오는 것과
“동일한 기능”을 처음부터 재구현하는 것 사이의 신뢰도 차이는, 금융 계산이나
의료 이미징 같은 정밀도가 중요한 영역에서 결정적이다.
