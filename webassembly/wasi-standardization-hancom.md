# WASI 표준화로 가능하게 되는 것들

원문: <https://tech.hancom.com/2024-12-06-wasi/>

## 요약

한컴테크의 이수동이 2024년 12월에 작성한 글로, WASI(WebAssembly System Interface)의 개념과 표준화 현황, 실습 예제를 다룬다.
WebAssembly가 브라우저 밖에서도 실행될 수 있도록 파일 시스템, 네트워킹, 시간, 난수 등 시스템 리소스에 접근하는 표준 API를 제공하는 것이 WASI의 핵심 역할이다.

WASI가 가능하게 하는 것은 세 가지다. 첫째, 크로스플랫폼 애플리케이션이다. 단일 릴리스 파일을 Wasm 런타임이 있는 모든 플랫폼에서 실행할 수 있다.
둘째, 플랫폼 간 코드 재사용이다. 클라이언트-서버, 모바일-데스크톱 등 다양한 아키텍처에서 같은 코드를 쓸 수 있다.
셋째, 단일 환경으로의 컨테이너화다. 모든 의존성을 Wasm 파일 하나로 컴파일해 컨테이너 간 오버헤드를 없앤다.

WASI는 호스트-게스트 모델로 동작한다. 호스트(Wasmtime 같은 Wasm 런타임)가 게스트 모듈을 실행하면서 기능 기반(capability-based) 보안 모델로 리소스 접근을 통제한다.
게스트 모듈은 호스트가 명시적으로 허용한 기능만 사용할 수 있다. 예를 들어 Wasmtime에서 `--mapdir` 플래그를 통해 디렉토리 접근 권한을 명시적으로 부여해야 파일 읽기·쓰기가 가능하다.

WASI의 버전은 두 가지가 있다. WASI 0.1은 WITX IDL로 정의된 초기 버전이다.
WASI 0.2는 2024년 2월 출시된 버전으로, 컴포넌트 모델과 WIT(WebAssembly Interface Type) IDL을 도입했다.
WASI 0.2에는 I/O, Clocks, Random, Filesystem, Sockets, CLI, HTTP API가 포함되며, 서로 다른 언어로 컴파일된 Wasm 컴포넌트 간의 상호 운용이 가능하다.

WASI의 장기적 방향성은 애플리케이션 개발자가 아닌 **언어 표준 라이브러리 개발자**를 위한 것이다.
개발자가 표준 라이브러리를 사용해 코드를 작성하면 Wasm으로 자연스럽게 컴파일되는 생태계를 목표로 한다.
Rocket 웹 프레임워크 예제처럼 WASI 인터페이스를 직접 쓰지 않아도 표준 Rust 코드가 Wasm으로 컴파일되는 것이 최종 지향점이다.

## 분析

### 호스트-게스트 모델이 의미하는 것

WASI의 호스트-게스트 모델은 기능 기반 보안의 구체적 구현이다.
전통적인 OS에서 프로세스는 기본적으로 파일 시스템 접근 권한을 가지고, 특정 경로를 막는 방식으로 제한한다.
WASI는 반대다. 기본적으로 아무것도 접근할 수 없고, 호스트가 명시적으로 허용한 것만 가능하다.
`wasmtime --mapdir /helloworld::.` 명령에서 `--mapdir`이 없으면 게스트는 파일 시스템 어디에도 접근할 수 없다.

이 역전된 기본값이 보안 모델의 핵심이다.
전통적 OS에서 악성 코드는 권한을 “빼앗아야” 한다. WASI에서 악성 코드는 아무것도 할 수 없다.
플러그인, 서드파티 코드, 사용자 제공 로직을 실행하는 시스템에서 이 차이는 결정적이다.

### WASI 0.1과 0.2의 아키텍처적 차이

WASI 0.1(모듈 기반)과 WASI 0.2(컴포넌트 기반)의 차이는 단순한 버전 업그레이드가 아니다.
모듈은 기본 숫자 타입만 인터페이스로 교환할 수 있어, 문자열이나 복잡한 구조체를 전달할 때 메모리 포인터를 주고받아야 했다.
컴포넌트 모델과 WIT는 언어 수준의 타입을 인터페이스로 정의해 이 한계를 해결한다.
Rust로 컴파일된 컴포넌트가 Go로 컴파일된 컴포넌트와 연동될 수 있다는 것은, WASI 0.2가 언어 경계를 넘는 소프트웨어 재사용의 기반을 만든다는 의미다.

### “라이브러리 개발자를 위한” 방향성의 전략적 의미

WASI가 “애플리케이션 개발자가 아닌 표준 라이브러리 개발자를 위한 것”이라는 선언은 채택 전략의 핵심이다.
애플리케이션 개발자에게 새로운 API를 배우게 하는 것보다, 기존 언어의 표준 라이브러리가 WASI를 지원하면 그 언어의 모든 사용자가 자동으로 WASI 혜택을 받는다.
Rocket 예제가 보여주듯, 개발자는 Rust 코드를 그대로 쓰고 컴파일 타깃만 `wasm32-wasi`로 바꾸면 된다.
이 접근은 기존 코드베이스의 전환 비용을 최소화하면서 Wasm 생태계를 확장하는 전략이다.

## 비평

### 실습 예제의 범위 한계

Hello World 수준의 파일 쓰기 예제는 WASI의 기초를 보여주지만, 실무에서 마주치는 복잡성을 다루지 않는다.
네트워킹은 기사 작성 시점 기준으로 아직 WASI 표준에 완전히 포함되지 않았다고 언급하지만, 그 영향이 무엇인지 설명하지 않는다.
HTTP 클라이언트, 소켓 통신이 필요한 애플리케이션을 WASI로 마이그레이션할 때 현실적으로 어떤 제약이 있는지는 이 글에서 파악하기 어렵다.

### WASI 0.2의 채택 현황 부재

2024년 2월 출시된 WASI 0.2의 런타임 지원 현황이 없다.
Wasmtime, Wasmer, WasmEdge가 각각 어느 수준의 WASI 0.2를 지원하는지, 컴포넌트 모델을 지원하는 언어 툴체인은 어떤 것이 있는지가 실무 채택을 결정하는 핵심 정보인데 다루지 않는다.
“WASI 0.2는 미래 기술”이라고 소개하면서도 현재 쓸 수 있는 수준이 무엇인지 명확하지 않다.

### 한컴의 컨텍스트 부재

한컴테크 블로그에 WASI 글이 실렸다는 것은 한컴이 이 기술에 실제적인 관심을 갖고 있다는 신호다.
한컴 오피스, 한글 등의 데스크톱 소프트웨어가 웹으로 이동하거나 크로스플랫폼 배포에 WASI를 활용하는 맥락이 있을 수 있다.
그러나 글은 이 배경을 전혀 드러내지 않는다. “왜 한컴이 WASI를 공부하는가”에 대한 답이 없다.

## 인사이트

### 표준화의 두 경로: 하향식과 상향식

소프트웨어 표준화에는 두 가지 경로가 있다. 하향식(top-down)은 위원회나 기업 연합이 표준을 설계하고 구현체가 따르는 방식이다. 상향식(bottom-up)은 여러 구현이 실전에서 수렴하면서 사실상의 표준(de facto standard)이 되는 방식이다.
WASI는 흥미롭게도 두 경로를 동시에 걷고 있다.
W3C Community Group이라는 공식 표준화 기구가 있으면서도, Cloudflare Workers, Fastly, Wasmtime 같은 실제 구현들이 표준 진행 방향에 압력을 행사한다.

이 이중 경로는 표준의 속도와 적실성을 동시에 확보하는 방법이기도 하다.
순수한 하향식은 완벽한 표준을 느리게 만들고, 순수한 상향식은 빠르지만 파편화된다.
WASI가 “아직 표준화 과제로 남아 있다”고 인정하면서도 실제 배포 환경에서 쓰이는 이유는 이 이중 경로 덕분이다.
현실의 요구가 표준을 당기고, 표준이 구현들을 수렴시킨다.

소프트웨어 역사에서 이런 패턴은 반복됐다. HTTP가 Tim Berners-Lee의 초기 구현에서 출발해 IETF RFC로 표준화됐듯, TCP/IP가 ARPANET 실험에서 출발해 DoD 표준을 거쳐 인터넷의 기반이 됐듯, 성공적인 표준은 실험과 공식화가 교차하며 성숙한다.
WASI 0.1에서 0.2로의 이행이 이 과정의 현재 단계다.

### 최소 권한 원칙의 기본값화

WASI의 기능 기반 보안 모델이 흥미로운 이유는 최소 권한 원칙을 “옵션”이 아닌 “기본값”으로 만들기 때문이다.
전통적 시스템에서 최소 권한은 적극적으로 설정해야 하는 것이었다. SELinux 정책을 작성하거나, seccomp 프로파일을 정의하거나, AppArmor 규칙을 추가하는 것은 전문 지식이 필요한 작업이다.
WASI에서는 아무것도 설정하지 않으면 자동으로 최소 권한이다. 권한을 추가하려면 명시적으로 선언해야 한다.

이 설계 철학의 2차 효과는 감사 가능성(auditability)이다.
`wasmtime --mapdir /helloworld::. hello.wasm` 명령 하나가 이 프로그램이 파일 시스템 어느 부분에 접근할 수 있는지를 완전히 문서화한다.
별도의 설정 파일이나 정책 명세 없이 실행 명령 자체가 보안 정책의 전체 표현이다.
소프트웨어 공급망 보안이 중요해진 현재, 의존성 패키지가 어떤 리소스에 접근하는지를 선언적으로 알 수 있는 것은 큰 가치다.

운영 관점에서도 이것은 사고 대응(incident response)을 단순화한다.
“이 Wasm 프로그램이 어떤 리소스에 접근했는가”를 조사하려면 실행 시 부여된 기능 목록을 보면 된다.
런타임에 동적으로 권한이 획득되는 전통적 시스템과 달리, WASI에서는 권한 집합이 실행 전에 고정된다.

### 언어의 장벽을 제거하는 인프라

WASI의 장기 비전, 즉 “언어와 OS의 장벽이 없는 생태계”는 소프트웨어 생산성의 구조적 문제를 겨냥한다.
현재 소프트웨어 개발의 많은 비효율은 언어 경계에서 발생한다.
최고의 암호화 라이브러리가 C로 작성됐어도 Python 개발자는 ctypes 바인딩을 만들어야 한다.
최고의 ML 모델 서빙 라이브러리가 C++로 작성됐어도 Java 서버는 JNI 통로를 거쳐야 한다.
이 변환 비용이 “언어 생태계 분열”의 실질적 대가다.

WASI 0.2의 컴포넌트 모델이 성숙하면 이 비용이 사라진다.
Rust로 구현된 최고의 암호화 컴포넌트를 Python 프로젝트에서 Wasm 컴포넌트로 직접 가져다 쓸 수 있다면, 언어 선택이 라이브러리 선택을 제한하지 않는다.
이것은 오픈소스 생태계에서 “최선의 구현”이 특정 언어에 갇히지 않고 보편적으로 사용될 수 있다는 의미다.

이 변화의 역방향 효과도 있다. 라이브러리 개발자가 특정 언어 생태계를 위해 구현체를 여러 번 만들 필요가 없어진다.
HTTP 파서, JSON 처리기, 정규표현식 엔진이 언어마다 별도로 존재하는 현재 상태는 중복 노력의 결과다.
단 하나의 최선 구현을 Wasm 컴포넌트로 만들고 모든 언어에서 사용하는 미래는, 오픈소스 유지보수 부담을 근본적으로 줄이는 방향이다.
한컴이 한글과 한컴 오피스 같은 복잡한 소프트웨어를 다양한 플랫폼에서 일관되게 제공해야 하는 현실적 과제를 생각하면, WASI가 가리키는 방향이 어떤 실용적 가치를 갖는지가 명확해진다.
