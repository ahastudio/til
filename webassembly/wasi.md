# WASI — WebAssembly System Interface

<https://wasi.dev/>

## 소개

WASI(WebAssembly System Interface)는 W3C WebAssembly 표준으로 컴파일된 소프트웨어를 위한 표준 API 명세 모음이다.
웹 브라우저, 서버, 엣지, 임베디드 기기 등 다양한 환경에서 WebAssembly 모듈이 운영체제 자원에 안전하게 접근할 수 있도록 인터페이스를 표준화한다.
W3C WebAssembly Community Group 산하 WASI Subgroup이 개발하며, 표준화 트랙 명세로 관리된다.

WASI의 핵심 가치 명제는 “HTTP 기반 마이크로서비스 같은 무겁고 불편한 중간 계층 없이, 서로 다른 언어로 작성된 소프트웨어를 직접 조합할 수 있게 한다”는 것이다.
플러그인 아키텍처, 서버리스 함수, 데이터베이스 사용자 정의 함수(UDF), 임베디드 컨트롤러, 네트워킹 사이드카 필터 등이 주요 적용 영역이다.

현재 두 개의 주요 버전이 있다.
0.1(Preview 1)은 POSIX와 CloudABI에서 영향을 받은 초기 API로 witx IDL을 사용하며, 현재도 광범위하게 배포되어 있다.
0.2(Preview 2)는 WIT(WebAssembly Interface Type) IDL로 재설계한 현재 안정 버전으로, 더 풍부한 타입 시스템, 모듈성, 가상화 가능성을 제공한다.
0.3은 현재 초안 단계로 같은 6개 핵심 API를 포함하며 범위를 확장 중이다.

지원 런타임으로는 Wasmtime, WAMR, WasmEdge, wazero, Wasmer, wasmi, wasm3, jco, pywasm 등이 있으며, 각각 다른 배포 환경에 특화되어 있다.
Node.js도 `node:wasi` 모듈로 WASI(Preview 1)를 실험적으로 지원한다.

## 분석

### 인터페이스 명세의 범위

WASI 0.2의 핵심 API는 6개 영역으로 구성된다.

| 인터페이스  | 역할                        |
| ----------- | --------------------------- |
| Clocks      | 시간 관련 기능              |
| Random      | 난수 생성                   |
| Filesystem  | 파일 시스템 접근            |
| Sockets     | 네트워크 소켓 연산          |
| CLI         | 커맨드라인 인터페이스 지원  |
| HTTP        | HTTP 프로토콜 처리          |

이 6개 인터페이스는 전통적인 POSIX가 다루는 시스템 호출 범위 중 Wasm 환경에서 가장 빈번히 필요한 부분을 추려낸 것이다.
머신 러닝(wasi-nn), 암호화, 키-값 저장소, 로깅, SQL 등 20여 개 추가 제안이 Phase 0~5의 표준화 단계에 걸쳐 진행 중이다.
이 확장 제안 목록은 WASI가 최소 공통 분모를 넘어 범용 시스템 인터페이스로 진화하려는 방향을 보여준다.

### 기능 기반(Capability-Based) 보안 모델

WASI의 설계 핵심 중 하나는 기능 기반 보안 모델이다.
전통적인 POSIX 환경에서 프로세스는 시작 시 사용자 권한을 상속받고, 파일 시스템과 네트워크에 광범위하게 접근할 수 있다.
WASI는 이를 뒤집어, Wasm 모듈이 명시적으로 부여받은 기능(capability)만 사용할 수 있도록 한다.

Node.js 구현의 `preopens` 옵션이 이 모델을 잘 보여준다.
`{ '/local': '/some/real/path' }` 형태로 가상 경로와 실제 경로를 명시적으로 매핑해야만 Wasm 모듈이 해당 디렉터리에 접근할 수 있다.
선언되지 않은 경로는 존재 자체를 알 수 없다.
이것은 최소 권한 원칙(Principle of Least Privilege)을 API 설계 수준에서 구현한 것이다.

단, Node.js 공식 문서는 현재 구현이 완전한 파일 시스템 샌드박싱을 보장하지 않으며, 신뢰되지 않는 코드 실행에 의존해서는 안 된다고 명시한다.
이론적 모델과 구현의 성숙도 사이의 간극이 아직 존재한다.

### "에뮬레이션의 에뮬레이션"에서 표준으로

WASI 이전 브라우저 밖 WebAssembly는 Emscripten의 JavaScript 글루 코드에 의존했다.
Emscripten은 원래 브라우저 제약을 위해 설계됐는데, 서버사이드 런타임들이 이 브라우저용 글루 코드를 다시 구현하는 상황이 벌어졌다.
Lin Clark(Mozilla)은 이를 "에뮬레이션의 에뮬레이션(emulation of an emulation of POSIX)"이라 불렀다.
불안정한 기반 위에서 각 런타임이 내부 구현 세부 사항을 독자적으로 재현하는 이 구조는 지속 가능한 생태계를 만들 수 없었다.

WASI는 이 에뮬레이션 계층을 제거하고 표준 인터페이스를 직접 정의함으로써 문제를 근원에서 해결한다.
Rust는 WASI 함수를 직접 타깃으로 컴파일하고, C/C++는 WASI 기본 요소 위에 libc를 구현한 wasi-sysroot를 통해 접근한다.
런타임(Wasmtime, Node.js, 브라우저 등)은 각자의 플랫폼에 맞게 WASI 구현을 제공하므로, 같은 Wasm 바이너리가 서로 다른 환경에서 실행된다.
이것이 WASI가 제공하는 이식 가능한 바이너리의 기반이다.

### Component Model과의 연계

WASI 0.2는 WebAssembly Component Model 명세와 긴밀하게 결합된다.
Component Model은 Wasm 모듈을 더 큰 단위의 컴포넌트로 조합하는 표준을 정의한다.
WASI 인터페이스는 WIT IDL로 기술되며, 컴포넌트 간 타입 안전한 상호 운용을 가능하게 한다.

이 연계의 의미는 “언어 간 경계를 넘는 함수 호출이 직렬화/역직렬화 없이 가능하다”는 것이다.
HTTP 마이크로서비스가 JSON을 통해 언어 경계를 넘는 것과 달리, Component Model 기반 Wasm은 메모리 내에서 직접 인터페이스를 연결한다.
이것이 wasi.dev가 말하는 “costly and clunky interface systems 없이 소프트웨어를 조합한다”는 가치 명제의 기술적 근거다.

## 비평

### Preview 1과 Preview 2 사이의 생태계 단절

WASI의 가장 현실적인 문제는 버전 간 호환성이다.
Preview 1은 witx IDL을 사용하고, Preview 2는 WIT IDL로 완전히 재설계됐다.
두 버전은 단순 상위호환 관계가 아니라 인터페이스 설계 철학이 다른 별개의 API 모음에 가깝다.

실제로 Preview 1 기반으로 작성된 Wasm 바이너리는 Preview 2 런타임에서 직접 실행되지 않는다.
런타임들이 각자의 속도로 Preview 2를 지원하면서, 같은 Wasm 바이너리가 어떤 런타임에서는 실행되고 어떤 런타임에서는 실패하는 상황이 발생한다.
“어디서나 실행된다”는 약속이 Java의 “Write Once, Run Anywhere”가 현실에서 마주한 것과 유사한 한계를 드러낸다.

### Node.js의 Experimental 상태 장기화

Node.js `node:wasi` 모듈은 현재 Stability: 1(Experimental)이다.
이 상태는 API가 언제든 변경될 수 있음을 의미한다.
Node.js가 웹 서버 생태계에서 갖는 영향력을 고려하면, `node:wasi`의 안정화가 서버사이드 WASI 채택에 중요한 신호가 될 것이다.
그러나 Preview 1만 지원하고 Preview 2 지원이 명확하지 않은 현재 상황은, Node.js 환경의 WASI 활용이 여전히 실험적 영역임을 의미한다.

### 보안 모델의 이론과 구현 간 간극

기능 기반 보안은 설계 원칙으로는 강력하지만, 실제 구현의 완전성은 런타임마다 다르다.
Node.js 문서가 명시적으로 경고하듯, 파일 시스템 샌드박싱은 현재 다양한 기법으로 우회 가능하다.
Wasm의 보안 가치 명제가 “신뢰되지 않는 코드를 안전하게 실행”인데, 그 전제가 아직 완전히 충족되지 않는다.
보안 모델을 신뢰하기 위해서는 런타임별 구현을 개별적으로 검증해야 하는 부담이 사용자에게 남는다.

## 인사이트

### POSIX의 계승과 단절

POSIX는 1988년 최초 버전 이후 Unix 계열 운영체제들이 따르는 표준 인터페이스가 됐다.
파일 디스크립터, 프로세스, 신호, 소켓 등의 추상화는 수십 년간 이식 가능한 소프트웨어의 공통 기반이었다.
WASI는 이 유산을 두 가지 방향에서 동시에 다룬다. 핵심 개념(파일, 소켓, CLI)을 계승하면서도, POSIX의 가장 큰 설계 약점인 “신뢰를 가정한 전역 접근”을 근본부터 바꾼다.

POSIX 환경에서 프로세스는 사용자 권한에 따라 파일 시스템 전체에 잠재적으로 접근할 수 있다.
이것은 1970년대 멀티유저 타임셰어링 시스템을 위한 합리적 설계였지만, 신뢰되지 않는 코드가 일상적으로 실행되는 현대 환경에서는 공격 표면이 된다.
WASI는 “기본적으로 아무것도 허용하지 않고, 명시적으로 부여한 것만 허용”하는 역전된 기본값을 채택함으로써 같은 추상화를 다른 보안 전제 위에 올려놓는다.

이 전환의 장기적 의미는 단순히 “더 안전한 POSIX”가 아니다.
기능 기반 보안이 OS 수준의 표준으로 자리잡으면, 소프트웨어 배포와 실행의 신뢰 모델 자체가 달라진다.
현재 컨테이너 보안이 네트워크 정책, seccomp 필터, AppArmor 프로파일의 조합으로 복잡하게 구성되는 것을, WASI는 인터페이스 명세 수준에서 단순하게 표현할 수 있는 방향을 제시한다.

### 마이크로서비스 시대 이후의 서비스 조합

마이크로서비스 아키텍처는 “서비스를 독립적으로 배포하고 HTTP로 연결한다”는 원칙으로 조직 규모의 소프트웨어 개발을 가능하게 했다.
그러나 HTTP 호출이 수반하는 네트워크 지연, 직렬화/역직렬화 비용, 분산 트레이싱의 복잡성은 “하나의 프로세스에서 함수 호출로 처리했다면 불필요했을 오버헤드”다.
마이크로서비스가 해결한 것은 기술적 문제가 아니라 조직적 문제였다. 팀 간 결합을 끊고 독립 배포를 가능하게 하는 것.

WASI가 Component Model과 함께 제시하는 그림은 이 트레이드오프를 다른 방식으로 해결하는 대안이다.
서로 다른 언어로 작성된 컴포넌트를 단일 Wasm 런타임 안에서 인터페이스 경계를 유지하며 직접 호출한다면, HTTP 오버헤드 없이 조합 가능성을 얻을 수 있다.
팀 자율성은 컴포넌트 경계와 WIT 인터페이스 명세가 보장하고, 실행 효율은 프로세스 내 직접 호출이 보장한다.

이 접근이 현실화되려면 Component Model과 언어 툴체인 지원이 성숙해야 하는데, 이는 아직 진행 중이다.
그러나 방향성 자체는 “서비스 메시의 복잡성 없이 격리와 조합을 동시에 달성한다”는 아키텍처적으로 매력적인 목표를 향한다.
마이크로서비스가 SOA(Service-Oriented Architecture)의 과잉 공학에 대한 반성으로 등장했듯, WASI 기반 컴포넌트 모델은 마이크로서비스의 네트워크 과잉에 대한 반성으로 읽힐 수 있다.

### 런타임 표준화가 언어 설계에 미치는 영향

역사적으로 새로운 런타임 표준이 등장할 때마다 언어 생태계가 재편됐다.
JVM이 등장하자 Scala, Clojure, Kotlin이 JVM을 타깃으로 설계됐고, Java 바이트코드는 언어 다양성의 공통 기반이 됐다.
LLVM이 등장하자 Clang, Rust, Swift, Julia가 LLVM IR을 통해 네이티브 코드를 생성하게 됐고, 컴파일러 백엔드를 독자적으로 구현할 필요가 없어졌다.
Wasm이 안정적인 컴파일 타깃으로 자리잡으면 비슷한 재편이 웹과 서버 환경에서 일어날 것이다.

특히 WASI가 흥미로운 점은, JVM과 달리 특정 언어의 가비지 컬렉터나 객체 모델을 가정하지 않는다는 것이다.
GC 제안이 Wasm 표준에 추가됐지만, 이는 옵션이지 전제가 아니다.
C, Rust처럼 메모리를 직접 관리하는 언어와 Go, Python처럼 GC를 사용하는 언어가 동일한 런타임에서 공존할 수 있다.
이 언어 중립성이 Wasm을 JVM보다 더 넓은 생태계의 수렴점으로 만드는 핵심 특성이다.

AI 코딩 도구와의 접점도 있다.
AI 에이전트가 임의의 코드를 생성하고 실행하는 시나리오에서, 신뢰되지 않는 생성 코드를 안전하게 실행하는 샌드박스로 WASI 런타임은 자연스러운 선택지다.
“AI가 쓴 코드를 어떻게 안전하게 실행하는가”라는 문제에 대해, Docker 컨테이너보다 가볍고 WASI보다 표준화된 답은 아직 없다.
런타임 표준화가 AI 에이전트 인프라의 보안 기반으로 진화할 가능성이 있다.
