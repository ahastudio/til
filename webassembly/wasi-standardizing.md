# WASI 표준화 — 웹 밖에서 WebAssembly를 실행하기 위한 시스템 인터페이스

원문: <https://hacks.mozilla.org/2019/03/standardizing-wasi-a-webassembly-system-interface/>
HN 토론: <https://news.ycombinator.com/item?id=19501614> (434점, 231개 댓글)

## 요약

2019년 3월 Mozilla의 Lin Clark이 발표한 이 글은 WASI(WebAssembly System Interface) 표준화 시작을 알린다.
WebAssembly는 이미 브라우저에서 성능 집약적 작업을 처리하는 데 쓰이고 있었지만, 브라우저 밖에서 실행할 때 운영체제 자원에 접근하는 표준 방법이 없었다.
파일, 네트워크, 클록, 난수 같은 기본 시스템 기능을 어떻게 Wasm 모듈에 제공할 것인가가 핵심 문제였다.

기존 해법의 구조적 결함이 WASI의 출발점이다.
브라우저 밖 Wasm 실행은 Emscripten이 제공하는 JavaScript 글루 코드에 의존했는데, 이 코드는 원래 브라우저 제약을 위해 설계된 것이었다.
서버사이드 런타임(wasmtime, Lucet)이 등장하자 이 JavaScript 함수들을 다시 네이티브로 재구현했다.
Lin Clark은 이 상황을 “에뮬레이션의 에뮬레이션(emulation of an emulation of POSIX)”이라 불렀다.
함수 이름은 `_system3(which, varargs)` 같은 불투명한 형태였고, 인수는 레지스터가 아닌 선형 메모리를 통해 전달되어 타입 안전성과 성능 모두를 희생했다.

WASI는 두 가지 설계 원칙 위에 세워진다.
첫째는 이식 가능한 바이너리(portable binary)다. 소스 코드가 아닌 컴파일된 바이너리 자체가 어느 환경에서나 실행되어야 한다.
둘째는 기능 기반(capability-based) 보안이다. POSIX처럼 사용자 권한을 상속하는 전역 접근 대신, 모듈이 호스트로부터 명시적으로 부여받은 기능(capability)만 사용할 수 있다.
이 두 원칙이 교차하는 지점에 WASI의 설계 결정들이 있다.

구체적 구현은 모듈형 설계를 따른다.
wasi-core는 파일, 네트워크, 클록, 난수 등 대부분의 애플리케이션이 필요로 하는 최소 인터페이스를 정의한다.
POSIX와 유사하지만 Wasm의 제약에 맞게 재설계됐다.
프로세스, 센서, 3D 그래픽 같은 특화 기능은 별도 제안 프로세스를 통해 확장한다.
이 모듈 분리 덕분에 임베디드 기기처럼 `fork`가 필요 없는 환경은 wasi-core만 구현하면 된다.

언어별 통합 방식도 명시한다.
Rust는 표준 라이브러리의 `open` 함수가 내부적으로 `__wasi_path_open`을 호출하도록 구현한다.
C/C++는 wasi-core 함수 위에 libc를 구현한 wasi-sysroot를 통해 접근한다.
런타임(wasmtime, Lucet, Node.js, 브라우저)은 WASI 함수를 Wasm 모듈의 import로 제공하며, 각자의 플랫폼에 맞게 구현한다.
발표 당시 Fastly, Node.js, npm이 지지를 표명했다.

## 분석

### 보호 링(Protection Ring)에서 기능 기반 보안으로

이 글의 핵심 논증은 운영체제 보안 모델의 역사적 진화를 참조한다.
전통적인 POSIX는 보호 링(protection ring) 아키텍처 위에 사용자 소유권 기반 접근 제어를 구축했다.
프로세스는 실행 사용자의 권한을 상속받고, 그 사용자가 접근할 수 있는 파일과 네트워크에 광범위하게 접근한다.
이 모델은 1970년대 신뢰할 수 있는 사용자들이 운영하는 멀티유저 시스템을 전제로 설계됐다.

현대 환경은 이 전제를 깨뜨린다.
라이브러리 의존성에서 받아오는 코드, 사용자 제공 플러그인, 서드파티 SDK는 모두 “신뢰할 수 없는 코드”다.
악의적인 라이브러리 관리자가 패키지에 파일 접근 코드를 심으면, 그 라이브러리를 사용하는 프로그램이 실행하는 사용자의 모든 파일에 접근할 수 있다.
WASI의 기능 기반 모델은 이 공격 경로를 인터페이스 수준에서 차단한다.

기능 기반 보안의 구체적 동작 방식이 이 글에서 명확히 설명된다.
런타임이 최초 파일 디스크립터를 최상위 코드에 전달하고, 코드는 이것을 다른 모듈에 제한적으로 위임할 수 있다.
경로 기반 `open(“/etc/passwd”)` 대신, 이미 허용된 디렉터리의 파일 디스크립터를 가지고 있어야만 접근이 가능하다.
최소 권한 원칙(Principle of Least Privilege)이 API 호출 방식 자체에 내재화된 구조다.

### libpreopen과 점진적 마이그레이션 전략

이 글이 제시하는 libpreopen은 기존 POSIX 코드를 점진적으로 마이그레이션하는 현실적 경로다.
`open` 같은 경로 기반 API를 직접 쓰는 기존 코드는 openat 기반의 기능 지향 API로 즉시 전환하기 어렵다.
libpreopen은 합법적으로 접근이 필요한 경로 목록을 미리 선언하면, 기존 `open` 호출이 그 경로들에 대해서만 작동하도록 제한한다.
“모든 것을 한번에 바꿔라”가 아닌 “지금 있는 코드가 작동하면서 보안을 얻어라”는 접근이다.

이 마이그레이션 경로의 존재는 WASI 설계의 실용주의를 보여준다.
이상적인 기능 기반 설계와 현실의 수십 억 줄 POSIX 코드 사이의 간극을 메우는 도구를 표준화와 함께 제시했다.
새로운 패러다임 도입에서 흔히 실패하는 지점, 즉 “기존 코드는 어떻게 하나”에 대한 답을 처음부터 포함한 것이다.

### 산업 지지의 전략적 의미

발표 시점에 Fastly, Node.js, npm이 동시에 지지를 표명한 것은 단순한 홍보 이상의 의미를 가진다.
Fastly는 엣지 컴퓨팅, Node.js는 서버사이드 JavaScript, npm은 패키지 배포라는 서로 다른 관심사를 대표한다.
이 세 영역이 공통적으로 “신뢰되지 않는 코드를 안전하게 실행”이라는 문제에 직면해 있다는 것을 보여준다.
WASI의 기능 기반 보안이 특정 플랫폼의 요구가 아니라 현대 소프트웨어 배포의 구조적 문제를 다룬다는 신호다.

## 비평

### 비동기 I/O의 부재

이 글은 비동기 I/O를 wasi-core 범위에서 명시적으로 제외하고 “나중에 해결할 문제”로 남긴다.
그러나 현대 서버 소프트웨어에서 비동기 I/O는 선택이 아닌 기본 요구사항이다.
Node.js가 이벤트 루프 기반 비동기 모델로 성공한 이유, Go가 goroutine을 도입한 이유가 여기에 있다.
wasi-core가 동기 I/O만 제공하는 상태에서는 고성능 서버 애플리케이션을 Wasm으로 구현하기 어렵다.
이 부재가 WASI 채택의 실질적 병목이 됐다는 것은 이후 wasi-io와 wasi-http 제안이 등장한 배경으로 확인된다.

dmitriid는 웹에서의 WebAssembly 자체도 아직 표준화할 것이 수백 가지 남아있는데 굳이 웹 바깥 실행부터 표준화하는 것이 맞느냐고 질문했다.[^dmitriid] 로드맵과 미래 기능 목록을 링크하며 브라우저 안에서 먼저 완성하는 것이 더 중요하지 않으냐는 입장이었다. 이는 WASI가 브라우저 내 Wasm 성숙과 병렬로 진행되는 것이 자원 분산인지 필연적인 방향인지에 대한 실질적 질문이었다.

### POSIX와의 거리

“POSIX와 유사하지만 Wasm의 제약에 맞게 재설계됐다”는 설명은 실제 호환성 수준을 명확히 하지 않는다.
wasi-core를 POSIX 대체로 사용하려면 어떤 POSIX API가 지원되고 어떤 것이 안 되는지, 그 경계가 어디인지가 중요하다.
이후 Preview 1에서 Preview 2로의 비호환 전환이 발생한 것은, 2019년 발표 시점의 API 설계가 충분히 안정적이지 않았음을 보여준다.
“POSIX와 유사하다”는 표현이 기대치를 높였지만, 실제 호환성의 한계는 도입을 시도하는 개발자들이 직접 발견해야 했다.

### 모듈 분리의 표준화 속도 문제

wasi-core 이후의 확장 모듈들이 표준화되는 속도가 불균일했다.
발표 시점에 언급된 프로세스, 센서, 3D 그래픽 확장은 2025년 현재도 표준화 초기 단계에 있다.
wasi-core만으로는 풀 수 없는 요구사항(예: 소켓 기반 네트워크 서버, 프로세스 생성)을 가진 애플리케이션은 표준화를 기다리거나 런타임별 비표준 확장에 의존해야 했다.
“필요한 것만 구현하면 된다”는 모듈 분리의 장점이 “필요한 모듈이 아직 없다”는 채택 장벽으로 뒤집혔다.

## 인사이트

### 보안의 기본값을 바꾸는 방법

소프트웨어 보안에서 “기본값을 바꾸는 것”은 “도구를 추가하는 것”보다 훨씬 어렵지만 훨씬 효과적이다.
방화벽, 안티바이러스, 샌드박스 도구들은 기존 POSIX 모델 위에 보안을 덧씌우는 방식이었다.
이 접근은 기본값이 “모든 것을 허용”이고 도구가 일부를 차단하는 구조라서, 도구가 없거나 설정이 잘못되면 취약해진다.
WASI는 이 방향을 뒤집는다. 기본값이 “아무것도 허용하지 않음”이고 명시적 부여가 접근을 열어주는 구조다.

이 전환이 소프트웨어 공급망 보안에 갖는 함의가 크다.
npm, PyPI, crates.io 패키지들이 실행될 때 어떤 시스템 자원에 접근하는지 현재는 코드를 감사하지 않는 이상 알 수 없다.
WASI 런타임에서 실행되는 패키지는 선언된 capability 외에는 접근이 불가능하다.
패키지 레지스트리가 “이 패키지가 요청하는 capability 목록”을 메타데이터로 공개하면, 사용자는 설치 전에 접근 범위를 확인할 수 있게 된다.
이것은 Android 앱 권한 모델이 모바일 생태계에서 한 것을 서버사이드와 웹 패키지 생태계에서 시도하는 것이다.

기본값의 전환이 성공하려면 마이그레이션 경로가 있어야 한다는 것을 이 글이 보여준다.
libpreopen이 존재하는 이유다. 이상적인 새 모델이 아무리 좋아도 기존 코드와의 단절이 너무 크면 채택이 일어나지 않는다.
“기존 코드가 작동하면서 새 보안 모델을 점진적으로 적용할 수 있다”는 경로를 함께 제시하는 것이 패러다임 전환 제안의 필수 요소다.

### 표준의 타이밍 — 너무 이르거나 너무 늦거나

WASI 발표는 기술 표준화의 타이밍 문제를 잘 보여주는 사례다.
2019년 시점의 WASI는 기술적으로 완성되지 않았다. 비동기 I/O, 소켓, 프로세스 같은 핵심 기능이 wasi-core에 없었고, Preview 1의 API 설계는 Preview 2에서 비호환 방식으로 교체됐다.
그러나 이 불완전한 시점에 표준화를 발표한 것이 오히려 생태계 형성에 기여했다.

너무 늦게 표준화하면 이미 런타임마다 제각각의 구현이 고착된다.
너무 이르게 표준화하면 설계가 충분히 검증되지 않아 비호환 변경이 불가피해진다.
WASI는 이르게 시작해서 비호환 변경을 겪었지만, 그 과정에서 실제 사용 경험을 반영한 더 나은 설계(Preview 2)에 도달했다.
“완벽한 설계를 기다리다 표준화하는 것”과 “불완전하지만 일찍 시작해 현장에서 개선하는 것” 사이에서, WASI는 후자를 택했다.

이 선택의 비용이 Preview 1 기반 생태계의 단절이었다.
Preview 1로 작성된 수많은 Wasm 바이너리가 Preview 2 런타임에서 실행되지 않는다.
“표준이 바뀌면 기존 코드가 깨진다”는 불신이 채택을 늦추는 요인이 됐다.
표준화 타이밍의 트레이드오프는 WASI만의 문제가 아니라 모든 신생 표준이 직면하는 구조적 딜레마다.

### “Java를 다시 발명한 것인가”라는 질문

HN에서 가장 많이 제기된 질문은 “JVM이나 CLR이 이미 이것을 하지 않느냐”였다.
gwbas1c는 “잠깐, JVM이 이것을 하기로 되어 있지 않았나? .NET CLR도 그렇고. 무엇이 다른가?”라고 물었고[^gwbas1c], naikrovek는 “솔직히 Java를 다시 발명한 것 아닌가”라고 직접적으로 물었다.[^naikrovek]
이 질문들은 중요하다. “어디서나 실행되는 바이너리”라는 약속은 반복적으로 등장해 왔고, 반복적으로 한계를 드러냈기 때문이다.

JVM과 WASI의 핵심 차이는 추상화 레벨과 설계 목적이다. JVM은 Java 언어의 런타임을 표준화했고 Java 생태계에 종속됐다. Wasm은 언어 중립적 명령어 집합을 표준화하며 어떤 언어든 컴파일 타깃으로 삼을 수 있다. 그리고 JVM은 신뢰할 수 있는 코드 실행을 가정하며 보안 모델이 나중에 덧붙여졌지만, Wasm과 WASI는 “신뢰할 수 없는 코드를 안전하게 실행”을 설계의 첫 번째 제약으로 삼았다.
steveklabnik는 “올해(2019년)는 WebAssembly에 있어 극도로 흥미로운 시기”라며 다양한 런타임이 쏟아지고 있다고 언급했다.[^steveklabnik] 이 결정들의 효과는 1년 뒤에야 보일 것이라는 신중한 전망이었다.

### 단일 바이너리가 “어디서나 실행된다”는 약속의 역사

“한 번 컴파일해서 어디서나 실행된다(Write Once, Run Anywhere)”는 약속은 Java의 JVM이 1995년에 내걸었다.
이 약속이 완전히 실현된 적은 없다. JVM 버전 불호환, 플랫폼별 네이티브 코드 의존성, 브라우저 Java 플러그인의 보안 취약점이 누적되며 약속은 좁아졌다.
WASI가 같은 약속을 다시 내걸 때, “이번엔 왜 다른가”라는 질문에 설득력 있는 답이 필요하다.

Lin Clark의 글이 제시하는 답은 추상화 레벨의 차이다.
JVM은 특정 언어(Java)의 런타임을 표준화했다. Wasm은 명령어 집합 아키텍처(ISA)를 표준화한다.
언어에 종속되지 않는 중간 계층이기 때문에, 어떤 언어든 Wasm을 타깃으로 컴파일할 수 있다.
JVM이 Java 생태계에 종속됐다면, Wasm은 언어 생태계 위의 중립 지대를 목표로 한다.

그러나 이 추상화 레벨의 차이가 “이번엔 성공할 것”을 보장하지는 않는다.
LLVM IR도 언어 중립적 중간 표현이지만, LLVM IR을 배포 단위로 쓰는 생태계는 형성되지 않았다.
Wasm이 LLVM IR과 다른 점은 브라우저 벤더들의 공동 설계와 보안 모델의 중요성이다.
LLVM IR은 컴파일러 내부 표현으로 설계됐지만, Wasm은 안전한 배포와 실행을 첫 번째 요구사항으로 설계됐다.
이 설계 목적의 차이가 “배포 단위로서의 Wasm”을 가능하게 하는 근거다.

`_jn`은 Gary Bernhardt의 강연 “The Birth & Death of JavaScript”를 연결했다.[^jn] 그 강연에서 Bernhardt는 JavaScript가 브라우저를 지배하고 나중에는 운영체제처럼 모든 것을 작동시킨다는 가상의 미래를 그렸다. `_jn`은 “그것이 실제로 일어나고 있는 것 같다”고 말했다. 단지 JavaScript가 아니라 Wasm과 WASI를 통해서.
`_bxg1`는 이것을 더 명확히 표현했다. “웹의 미래는 사용자 공간을 대체하고, 모든 프로그램이 크로스 플랫폼이 되고 샌드박스에서 실행되며, JavaScript가 '웹' 타깃 언어 집합으로 희미해지는 것”이라는 예측이 실현되는 것을 보고 있다고 했다.[^bxg1]
