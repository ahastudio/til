# WebAssembly는 컨테이너를 대체할 수 있을까

원문:

- <https://creston.blog/wasm-will-replace-containers/>
- <https://www.itworld.co.kr/article/3604881>

## 요약

WebAssembly(Wasm)가 Docker와 Kubernetes로 대표되는 컨테이너를 대체할 수 있는지에 대해 두 가지 상반된 시각이 존재한다.

Creston의 글은 강한 주장을 편다. 2030년에는 아무도 Kubernetes를 기억하지 못할 것이라고 예언한다.
그 근거는 Wasm의 진정한 이식성이다. 컨테이너는 애플리케이션 코드와 Linux 의존성을 함께 묶어 “이식 가능한” 척하지만, 실제로는 Linux 커널이 없으면 실행할 수 없다.
반면 Wasm은 V8 엔진이 있는 어디서나 실행된다. 브라우저, 서버, 엣지, 모바일, IoT 모두에서.
Cloudflare Workers처럼 같은 V8 런타임에서 여러 워커가 실행되면 마이크로서비스의 개발자 경험과 모놀리스의 런타임 성능을 동시에 얻는다는 것도 핵심 논거다.

ITWorld가 6명의 전문가를 인터뷰해 정리한 글의 결론은 대조적으로 신중하다.
Wasm과 컨테이너는 대체 관계가 아니라 보완 관계라는 것이 다수 의견이다.
Wasm이 유리한 영역은 엣지 컴퓨팅, IoT, 서버리스, 플러그인 아키텍처다.
컨테이너가 여전히 유리한 영역은 장기 실행 스테이트풀 서버 애플리케이션, 데이터베이스, Linux 생태계에 깊이 통합된 엔터프라이즈 시스템이다.
Adobe Photoshop 웹 버전, Shopify Functions, Istio가 Wasm을 이미 채택했다는 점은 현실적 도입 현황을 보여준다.

양측이 공통으로 인정하는 사실이 하나 있다. Wasm의 시스템 인터페이스 지원(파일 접근, 네트워킹, 스레딩)이 아직 성숙하지 않았다는 점이다.
WASI가 이 문제를 다루고 있지만, 컨테이너 수준의 완전한 Linux 환경을 제공하기까지는 시간이 필요하다.

## 분析

### 두 기술이 해결하는 문제의 차이

컨테이너가 해결한 문제는 “개발 환경과 프로덕션 환경의 일관성”이다.
“내 컴퓨터에서는 되는데”를 해결하기 위해 OS 수준의 격리를 제공했다.
이것은 근본적으로 배포 문제이자 팀 협업 문제다.

Wasm이 해결하는 문제는 다르다. “신뢰되지 않는 코드를 안전하게 실행하는 것”과 “이진 형식의 이식성”이다.
브라우저가 임의의 코드를 실행하면서도 사용자 시스템을 보호해야 하는 문제에서 출발했다.
해결하는 문제가 다른 두 기술을 단순 비교하면 논점이 흐려진다.
진짜 질문은 “Wasm이 컨테이너를 대체하는가”가 아니라 “Wasm이 컨테이너가 해결한 문제를 더 잘 해결하는가”다.

### 아키텍처 수준의 단위 차이

컨테이너는 OS 프로세스 수준의 격리다. 그 안에서 어떤 언어, 어떤 런타임을 쓰든 무관하다.
Wasm은 컴파일 타깃이자 실행 환경이다. 애플리케이션 코드의 컴파일 방식을 바꾼다.
이 둘은 추상화 레벨이 다르다. 결과적으로 Wasm 모듈을 컨테이너 안에서 실행하거나, 컨테이너 대신 Wasm 런타임을 Kubernetes 노드에서 실행하는 혼합 모델이 현실적이다.
Docker가 WASM 컨테이너를 지원하는 방향으로 개발 중이라는 사실은 이 혼합 모델의 현실성을 뒷받침한다.

### JVM과의 비교

Creston의 글이 JVM을 비교 대상으로 드는 것은 적절하다.
JVM도 “한 번 작성하면 어디서나 실행”을 약속했지만 브라우저에서 실행되지 못해 웹 플랫폼과 분리됐다.
Wasm은 브라우저에서 태어난 기술이다. 서버로 확장하는 방향이 JVM이 서버에서 태어나 브라우저 진출에 실패한 것과 정반대다.
이 출발점의 차이가 채택 경로를 다르게 만든다. Wasm은 이미 모든 모던 브라우저에 내장되어 있다.

## 비평

### “대체” 프레임의 문제

기술 논쟁에서 “A가 B를 대체할 것인가”라는 프레임은 단순하지만 생산적이지 않다.
NoSQL이 관계형 데이터베이스를 대체하지 않았고, 마이크로서비스가 모놀리스를 대체하지 않았으며, 서버리스가 컨테이너를 대체하지 않았다.
기술은 대체보다 분화한다. 새로운 기술이 기존 기술이 잘 못하는 영역을 가져가고, 기존 기술은 자신이 잘하는 영역에 집중한다.
“Wasm이 컨테이너를 대체한다”는 주장은 2030년 예언으로는 자극적이지만, 현실적인 기술 전략으로는 오도적이다.

### 컨테이너 복잡성 과장 문제

Creston의 글은 “Docker 배우는 것이 시간 낭비”라고 주장하지만, 이것은 특정 규모와 팀 맥락에 치우친 관점이다.
스타트업 초기 단계에서 Docker 없이 빠르게 배포하는 것이 맞을 수 있다.
그러나 팀이 성장하고 다중 서비스가 생기면 컨테이너의 재현 가능한 환경과 오케스트레이션이 주는 가치가 복잡성 비용을 상쇄한다.
복잡성은 도구의 문제가 아니라 해결하려는 문제의 본질적 복잡성이다.

### 누락된 관점: 운영 성숙도

Wasm 생태계의 현실적 약점은 성숙한 운영 도구의 부재다.
Kubernetes에는 Helm, Prometheus, Grafana, ArgoCD, Istio처럼 수년간 검증된 도구 생태계가 있다.
Wasm 런타임(Wasmtime, WasmEdge, Wasmer)은 분열되어 있고 각자 기능 수준이 다르다.
개발자 경험은 개선될 수 있지만, 운영 경험의 성숙도는 시간과 사용자 기반 없이는 얻을 수 없다.

## 인사이트

### 컨테이너의 진짜 해자는 기술이 아니라 생태계다

Docker가 2013년 등장한 이후 컨테이너가 지배적 배포 단위가 된 것은 기술적 우월성 때문만이 아니다.
Docker Hub의 이미지 레지스트리, Kubernetes의 오케스트레이션, Helm 차트, CI/CD 파이프라인 통합, 모니터링 도구들이 상호 보강하며 만들어진 생태계가 진짜 해자다.
기술을 교체하려면 기술 교체뿐 아니라 이 생태계 전체를 교체해야 한다.
Wasm이 기술적으로 우월한 부분이 있더라도, 조직의 전환 비용은 컨테이너 생태계에 오랫동안 결박된다.

이 패턴은 반복적이다. QWERTY 키보드가 기술적으로 최선이 아닌데도 대체되지 않는 것, 관계형 데이터베이스가 NoSQL 혁명 이후에도 여전히 지배적인 것, Java가 더 현대적인 언어들 속에서도 엔터프라이즈의 주력인 것은 모두 같은 이유다.
기술의 생존은 기술 자체의 품질이 아니라 그 기술을 중심으로 형성된 사람, 도구, 프로세스의 네트워크에 달려 있다.
Wasm이 컨테이너를 “대체”하려면 이 네트워크를 이기거나 우회해야 한다.

한 가지 가능한 시나리오는 새로운 워크로드가 컨테이너를 건너뛰는 것이다.
AI 추론 서빙, 엣지 함수, 플러그인 아키텍처처럼 컨테이너 생태계가 아직 깊게 침투하지 않은 영역에서 Wasm이 기본값이 되면, 기존 컨테이너 워크로드를 전환하지 않아도 영향력을 확보할 수 있다.
대체가 아닌 잠식이다.

### 엣지 컴퓨팅이 결정적 전장이다

컨테이너가 클라우드 데이터센터에 최적화된 기술이라면, Wasm은 엣지에서 진가를 발휘한다.
Cloudflare Workers, Fastly Compute@Edge, Vercel Edge Functions는 모두 Wasm 기반이다.
엣지 노드는 수백 밀리초의 콜드 스타트도 허용하기 어렵고, 메모리가 제한적이며, 보안이 중요하다.
Wasm은 마이크로초 수준의 콜드 스타트, 킬로바이트 단위의 메모리 풋프린트, 샌드박스 실행으로 이 세 조건을 충족한다.

엣지 컴퓨팅 시장이 성장할수록 Wasm의 기반도 강해진다.
IoT 기기, CDN 팝(PoP), 네트워크 장비에서 실행되는 코드가 Wasm으로 쓰이기 시작하면, Wasm 개발 도구와 라이브러리 생태계가 성숙해진다.
이 성숙도가 서버 중심 워크로드로의 역진출 기반이 된다.
역사적으로 기술의 확산은 “성능이 부족하지만 편리한” 쪽에서 시작해 성능이 개선되면서 “편리하면서도 충분히 빠른”으로 이동하는 패턴을 따른다.

### 격리의 단위가 프로세스에서 모듈로 이동한다

컨테이너가 프로세스를 격리 단위로 삼았다면, Wasm은 모듈을 격리 단위로 삼는다.
하나의 프로세스 안에서 여러 Wasm 모듈이 독립적으로 실행될 수 있다는 것은 아키텍처의 근본적 변화다.
Cloudflare Workers가 하나의 V8 런타임에서 수천 개의 격리된 Workers를 실행하는 것은 이 가능성의 현재 구현이다.

격리 단위가 작아지면 오버헤드가 줄어든다. 컨테이너 하나를 올리는 것보다 Wasm 모듈 하나를 로드하는 것이 몇 자릿수 빠르다.
이 차이는 서버리스와 마이크로서비스의 비용 구조를 바꾼다.
함수 하나를 호출할 때마다 컨테이너를 시작하는 대신 Wasm 모듈을 로드하면, 콜드 스타트 비용이 사라지고 더 세밀한 규모 조정이 가능해진다.

격리 단위의 변화는 보안 모델도 바꾼다.
컨테이너에서 격리 경계를 넘으려면 컨테이너 탈출 취약점이 필요하다.
Wasm에서는 모듈 간 격리가 언어 수준에서 강제되며, 호스트가 허용하지 않는 한 다른 모듈의 메모리에 접근할 수 없다.
이것은 마이크로서비스 아키텍처에서 서비스 간 신뢰 모델을 재설계할 기회를 만든다.
네트워크 경계 대신 코드 경계로 신뢰 범위를 정의하는 것이다.
