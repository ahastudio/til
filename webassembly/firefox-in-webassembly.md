# Firefox in WebAssembly

<https://developer.puter.com/labs/firefox-wasm/>

<https://github.com/HeyPuter/firefox-wasm>

HN 토론: <https://news.ycombinator.com/item?id=48926939> (186점, 97개 댓글)

## 소개

Puter Labs가 공개한 실험 프로젝트로, Gecko 엔진을 WebAssembly로 컴파일해
실제 Firefox UI를 브라우저 탭 안에서 그대로 실행한다.
“Show HN: Firefox in WebAssembly”라는 제목으로 소개된 이 프로젝트는
Emscripten을 이용해 C++로 작성된 Gecko 엔진을 wasm으로 옮기고,
Rust 코드는 `wasm32-unknown-emscripten` 타깃으로 함께 빌드한다.
저장소 언어 구성을 보면 JavaScript가 97.6%, C++가 1.6%, TypeScript가
0.3%로, 실제 Gecko 코드 자체보다 이를 감싸는 브라우저 내 실행 환경과
브리지 코드가 프로젝트의 대부분을 차지한다.

데모 페이지에 접속하면 약 52~54MB 크기의 Firefox 바이너리를
내려받은 뒤, 실제 xul 기반 크롬(chrome) UI를 부팅하는 과정을
스타트업 로그로 보여준다.
로그를 보면 `/opfs`에 네이티브 OPFS(Origin Private File System)
백엔드를 마운트하고, `/gre`에 GRE(Gecko Runtime Environment)
프로바이더를 마운트한 뒤, `NS_InitXPCOM`으로 XPCOM을 초기화하고
`chrome://browser/content/browser.xhtml`을 로드해 실제 Firefox
브라우저 창을 띄운다.
최종적으로 “Firefox front-end booted”라는 로그와 함께 실제 Firefox
UI가 캔버스 위에 렌더링된다.

빌드는 Linux 환경에서만 가능하며 macOS는 지원하지 않는다.
`libpulse-dev`, Node.js, `pnpm`이 필요하고, `emsdk` 6.0.1을 설치한
뒤 Rust에 wasm 타깃을 추가하고 `make` 명령으로 웹용 빌드를
수행하는 절차를 거친다.
라이선스는 Mozilla Public License 2.0이며, 이 글을 작성하는 시점
기준으로 GitHub 저장소는 스타 61개, 포크 3개, 열린 이슈 0개로
아직 작은 실험 프로젝트 단계에 머물러 있다.

## 네트워킹과 렌더링

일반적인 브라우저는 OS 소켓 API로 직접 TCP/TLS 연결을 맺지만,
브라우저 탭 안에서 실행되는 wasm 바이너리는 그런 저수준 네트워크
접근 권한이 없다.
이 프로젝트는 이 문제를 Puter가 호스팅하는 Wisp 프로토콜 기반
WebSocket 프록시 서버로 우회한다.
Gecko의 네트워크 스택이 만드는 요청을 WebSocket으로 감싸 프록시
서버에 전달하고, 프록시가 실제 목적지로 TCP 연결을 대신 맺어주는
방식이다.
스타트업 로그에 “disabled IPv6 (SOCKFS ws:// url can't encode IPv6
literals)”라는 경고가 남아 있는데, 이는 이 우회 구조 자체의 한계를
드러내는 대목이다.
SOCKFS가 목적지 주소를 WebSocket URL에 인코딩하는 방식이다 보니
IPv6 리터럴 주소를 표현할 수 없어 IPv6 자체를 비활성화한 것이다.

렌더링 쪽에서는 `GECKO_GPU=1` 옵션으로 GPU/WebRender 경로를 켜서
`WR-DIRECT: in-process WebRender transaction path`로 렌더링
결과를 캔버스에 직접 그리는 방식을 쓴다.
동시에 `GECKO_GL_PASSTHROUGH=1`, `GECKO_COARSE_CLOCK=1`처럼 wasm
환경에 맞춰 조정한 별도의 환경 변수들이 다수 존재하는데, 이는 이
포팅이 단순히 코드를 컴파일만 한 것이 아니라 브라우저 엔진 내부의
스레딩 모델, 타이머 정밀도, 렌더링 파이프라인까지 상당 부분을
wasm/브라우저 샌드박스 제약에 맞게 손을 댔다는 것을 시사한다.
실행은 `xul_init: forced non-remote tabs + extensions
(single process)`, `disabled socket/network process
(single process)` 로그에서 보듯 멀티프로세스 구조 없이 단일
프로세스로 강제된다.
데스크톱 Firefox의 핵심 안전장치인 프로세스 격리(sandboxing)를
그대로 가져올 수 없는 환경이기 때문이다.

실험적 기능으로 JavaScript를 WebAssembly로 JIT 컴파일하는 시도가
포함되어 있는데, README는 이를 “많은 웹사이트에서 아직 쓸 수 없는
수준”이라고 명시하며 `GECKO_NOWASMJIT=1` 환경 변수로 끌 수 있게
해두었다.
웹 콘텐츠 프록시와 네트워킹은 Puter.js가 제공한다.
Mozilla에서 SpiderMonkey를 다뤄본 HN 사용자 eqrion은 이 JIT의
세부 구현이 궁금하다며, SpiderMonkey 팀도 과거 wasm32 JIT
백엔드를 시도했지만 완성하지 못했던 적이 있고, 이 프로젝트에는
SpiderMonkey에 없는 WebAssembly 인터프리터 백엔드까지 포함된 것
같다고 언급했다[^eqrion].
이런 반응은 이 실험이 단순한 취미 수준을 넘어, 실제 브라우저
엔진 개발자들도 시도했다가 완성하지 못한 영역을 건드리고 있다는
것을 보여준다.

이 프로젝트가 처음은 아니다.
HN 댓글에서는 이미 여러 유사 선행 사례가 언급되었는데, WebKit
렌더링 엔진을 JavaScript로 포팅한 WebKit.js[^azakai], iOS에서
애플이 금지한 WebKit 이외 엔진을 사이드로드할 수 있게 Gecko를
포팅한 reynard-browser[^tech234a], 그리고 WebKit을 WebAssembly로
포팅해 좀 더 현대적이고 실사용 가능하다고 소개된
WebkitWasm[^ohonbob]이 그것이다.
브라우저 엔진을 다른 브라우저 위에서 재구현하려는 시도는 이미
하나의 작은 장르를 이루고 있는 셈이다.

## 분석

### wasm 포팅은 “컴파일”이 아니라 “재설계”에 가깝다

이 프로젝트가 흥미로운 지점은 Gecko를 Emscripten으로 그냥 컴파일한
것이 아니라는 데 있다.
스타트업 로그에 나열된 `unsupported syscall: __syscall_madvise`,
`unsupported syscall: __syscall_prlimit64` 같은 경고, IPv6를
통째로 비활성화한 결정, 멀티프로세스 구조를 단일 프로세스로 강제
전환한 조치는 모두 리눅스 시스템 콜과 프로세스 모델을 전제로 설계된
20년 묵은 코드베이스를 브라우저 샌드박스라는 완전히 다른 실행
환경에 맞춰 재설계했다는 증거다.
표면적으로는 “Firefox를 브라우저에서 돌린다”는 한 줄로 요약되지만,
실제로는 OS 파일 시스템을 OPFS로, TCP 소켓을 WebSocket 프록시로,
멀티프로세스 IPC를 단일 프로세스 함수 호출로 각각 치환하는 세
갈래의 별도 엔지니어링 작업이 필요했다는 뜻이다.

이런 구조는 Emscripten 기반 포팅 프로젝트에서 반복적으로 나타나는
패턴이다.
소스 코드 자체는 최소한만 수정하고, 대신 그 코드가 기대하는 OS
인터페이스(파일 시스템, 소켓, 스레드, 시그널)를 wasm 환경에서
흉내 내는 방식이다.
이 프로젝트에서 Rust 코드 비중이 C++보다 낮으면서도 별도로
언급되는 이유도 여기에 있다고 추정할 수 있다.
Gecko 코드 자체를 고치기보다 그 주위를 감싸는 브리지·에뮬레이션
레이어를 새로 쓰는 접근이 전체 작업량의 상당 부분을 차지했을
가능성이 크다.

### 데모의 “완성도”와 코드베이스의 “완성도”는 다른 층위에 있다

라이브 데모는 실제로 동작하는 Firefox UI를 보여주지만, 이것이 곧
프로젝트의 성숙도를 의미하지는 않는다.
저장소는 스타 61개, 이슈 0개인 신생 프로젝트이고, JS-to-wasm JIT은
README 스스로 “많은 사이트에서 쓸 수 없다”고 인정할 만큼 실험적
단계에 머물러 있다.
반면 데모 페이지는 실제 브라우저 창, 실제 xul 크롬, 실제 URL 바를
보여주기 때문에 사용자에게는 훨씬 완성된 제품처럼 느껴진다.

이 간극은 실행 가능한 데모가 기술적 성숙도의 대리 지표로 소비되는
현상을 보여주는 사례다.
데모가 “동작한다”는 사실은 핵심 부팅 경로가 안정화되었다는
뜻이지, 임의의 웹사이트에서 임의의 JavaScript가 안정적으로
돌아간다는 뜻은 아니다.
Show HN 같은 채널에서 화제가 되는 프로젝트일수록 이 구분이 쉽게
흐려진다.
실제로 HN 댓글 다수가 이 지점을 파고들었다.
pmarreck는 인상적이고 성능도 나쁘지 않지만, 정작 이걸 실제로
어디에 쓸 수 있는지 묻고 “WASM을 홍보하기 위한 데모”처럼
보인다고 평했다[^pmarreck].
안정성 문제도 함께 보고되었는데, brewmarche와 simonw는 각자의
환경에서 실행이 실패했다며 스타트업 로그를 첨부했고 공통적으로
`__syscall_madvise` 미지원 경고 뒤에 오류가 이어졌다고
밝혔다[^brewmarche].
모바일에서는 한계가 더 뚜렷했다.
rmac은 모바일 Chrome/Android에서 테스트한 결과 IME 입력, 복사·
붙여넣기, 터치 스크롤, AI 사이드패널이 동작하지 않았다고
보고했다[^rmac].
이런 반응들은 이 프로젝트가 부팅 시연에는 성공했지만, “실제
브라우저처럼 쓸 수 있는가”라는 기준에서는 아직 갈 길이 멀다는
것을 보여준다.

### 왜 지금, 왜 Puter인가

Puter는 브라우저 안에서 데스크톱급 OS 환경을 구현하는 것을
지향하는 플랫폼이다.
이 맥락에서 Firefox-in-WASM은 단발성 기술 시연이 아니라, Puter가
이미 구축해 둔 네트워킹 인프라(Wisp 프록시, Puter.js)를 재사용해
“브라우저 안의 OS”라는 기존 비전에 “브라우저 안의 브라우저”라는
한 조각을 더한 것으로 읽을 수 있다.
Puter Labs라는 이름 자체가 정식 제품이 아니라 실험 공간임을
명시하고 있다는 점도 이 프로젝트의 위치를 이해하는 데 중요하다.
프로덕션 품질을 약속하는 것이 아니라, 가능성을 시연하는 것이
목표라는 뜻이다.

## 비평

### IPv6 비활성화는 임시방편이 아니라 아키텍처의 구조적 한계다

README와 스타트업 로그는 IPv6 비활성화를 “SOCKFS ws:// URL이
IPv6 리터럴을 인코딩할 수 없어서”라는 구현 세부사항으로
설명하지만, 이는 사실 이 프로젝트가 선택한 네트워킹 우회 방식
전체의 구조적 한계를 드러낸다.
WebSocket 프록시 위에 소켓 계층을 얹는 SOCKFS 방식은 애초에 주소
체계를 URL 문자열로 직렬화해야 하는데, 이 직렬화 규약을 확장하지
않는 한 IPv6는 영구히 지원 대상에서 빠질 수밖에 없다.
즉 이것은 “아직 구현하지 못한 기능”이 아니라 “현재 설계로는 풀기
어려운 문제”에 가깝다.
프로젝트가 이를 사소한 제약처럼 한 줄 경고로 처리하고 넘어가는
태도는 이 한계의 무게를 축소해서 보여준다.

### 단일 프로세스 강제는 보안 모델의 후퇴를 수반한다

`disabled socket/network process`, `forced non-remote tabs`
같은 조치는 데스크톱 Firefox가 수년에 걸쳐 구축한 프로세스 격리
기반 보안 모델을 사실상 포기한 결정이다.
데스크톱 Firefox에서 콘텐츠 프로세스와 네트워크 프로세스를
분리하는 이유는 단순한 성능 최적화가 아니라, 악성 웹 콘텐츠가
브라우저 코어나 다른 탭을 침해하지 못하도록 막는 핵심 방어선이기
때문이다.
이 프로젝트는 이를 단일 프로세스로 되돌리면서도, 그 보안적 함의를
README나 데모 페이지 어디에서도 명시적으로 언급하지 않는다.
물론 wasm 자체가 브라우저 샌드박스 안에서 실행되므로 호스트
시스템 관점에서는 격리되어 있다고 볼 수 있지만, Gecko 내부에서
탭 간 격리가 사라졌다는 사실은 “실제 Firefox를 그대로 재현했다”는
프로젝트의 주장과 긴장 관계에 있다.
사용자 눈에는 익숙한 Firefox UI로 보이지만, 그 UI가 보장하던
보안 속성 중 상당 부분은 이미 빠져 있다는 뜻이다.

### 웹 콘텐츠 프록시 구조는 이 프로젝트를 사실상 “Puter 종속형”으로 만든다

모든 네트워크 트래픽이 Puter가 호스팅하는 Wisp 프록시를 거친다는
사실은 기술적 성취와는 별개로 중요한 함의를 갖는다.
이 프로젝트가 공개한 코드를 그대로 받아 직접 빌드해도, 실제로
웹을 탐색하려면 Puter의 프록시 인프라에 의존해야 한다.
오픈소스 라이선스(MPL 2.0)로 공개되어 있다는 점과, 실질적인
운영 가능성이 특정 사업자의 인프라에 묶여 있다는 점은 서로 다른
층위의 개방성이다.
코드가 열려 있다고 해서 이 프로젝트를 독립적으로 재현하거나
운영하기 쉬운 것은 아니라는 뜻이며, 이는 README에서 충분히
드러나지 않는 전제다.

이 문제는 추상적인 우려에 그치지 않는다.
HN 사용자 zerof1l은 데모를 사용해 보니 모든 네트워크 트래픽이
Puter 서버를 거쳐서, 자신의 실제 위치가 아니라 인도의 Cloudflare
IP로 표시되었다고 보고하며 Puter를 신뢰하지 않는다고
밝혔다[^zerof1l].
프로젝트 설명 중 일부가 “완전한 종단간 암호화”를 언급한 데
대해서는 bawolff가 더 근본적인 반박을 내놓았다.
서버(Puter)가 브라우저 안에서 실행되는 wasm 코드 자체를
배포·통제하는 이상, 그 서버가 코드를 바꿔치기하면 클라이언트
쪽 암호화를 무력화할 수 있으므로 진정한 의미의 종단간 암호화는
성립하지 않는다는 지적이다[^bawolff].
두 댓글은 서로 다른 각도에서 같은 결론에 도달한다.
이 프로젝트의 네트워킹 구조는 Puter라는 단일 사업자를 신뢰할
것을 전제로 하며, 그 신뢰가 깨지는 순간 오픈소스 라이선스는
사용자를 보호해 주지 못한다.

## 인사이트

### 데스크톱 애플리케이션의 wasm 포팅은 기능 이식이 아니라 “권한의 재협상”이다

Firefox-in-WASM이 보여주는 진짜 패턴은 Gecko 코드를 얼마나
고쳤는가가 아니라, OS가 원래 애플리케이션에 부여했던 권한들
(임의의 TCP 연결, 멀티프로세스 생성, 파일 시스템 직접 접근,
IPv6 소켓)을 브라우저 샌드박스라는 훨씬 제한된 권한 모델 안에서
어떻게 재협상했는가에 있다.
이 재협상은 항상 손실을 수반한다.
IPv6는 포기되고, 프로세스 격리는 단일 프로세스로 축소되고,
네트워크 접근은 특정 사업자의 프록시로 대체된다.
이는 Firefox 하나에 국한된 문제가 아니라, 데스크톱/네이티브
애플리케이션을 wasm으로 옮기는 모든 프로젝트가 마주치는 일반
패턴이다.
CAD 소프트웨어, 게임 엔진, 데이터베이스 서버를 wasm으로 포팅하는
프로젝트들도 정도만 다를 뿐 동일한 재협상 과정을 거친다.
따라서 이런 프로젝트를 평가할 때 중요한 질문은 “돌아가는가”가
아니라 “어떤 권한을 어떤 방식으로 포기했는가”여야 한다.

### 브라우저 안의 브라우저는 웹 플랫폼이 이미 범용 실행 환경이 되었다는 신호다

Firefox 전체를 다른 브라우저의 탭 안에서 돌린다는 발상 자체가
성립할 수 있다는 사실은, 웹 플랫폼이 더 이상 문서를 보여주는
용도에 머물지 않고 사실상 범용 실행 환경(general-purpose
runtime)으로 자리 잡았음을 보여주는 강력한 신호다.
2000년대의 브라우저는 OS 위에서 실행되는 하나의 애플리케이션이었지만,
지금의 브라우저는 그 자체로 OS 커널이 하던 역할(프로세스 격리
대신 wasm 샌드박스, 파일 시스템 대신 OPFS, 소켓 대신 WebSocket)을
일부 떠맡는 플랫폼이 되어 가고 있다.
Puter가 “브라우저 안의 OS”를 표방하며 이 프로젝트를 자사 랩에
포함시킨 것도 우연이 아니다.
이런 흐름이 계속되면, “브라우저에서 무엇을 돌릴 수 있는가”라는
질문 자체가 무의미해지고 대신 “어떤 성능·보안 손실을 감수할
것인가”라는 질문만 남게 된다.

### 이런 프로젝트의 확산은 “브라우저 지문 채취”와 “탐지 회피”라는 새로운 공격 표면을 연다

원문이 다루지 않는 지점이지만, Firefox UI 전체를 다른 브라우저 안에서
wasm으로 재현할 수 있다는 사실은 보안 관점에서 흥미로운 이차
효과를 만든다.
지금까지 웹사이트의 봇 탐지·핑거프린팅 로직은 브라우저 엔진의
API 표면, 렌더링 특성, 타이밍 특성이 실제 브라우저 바이너리에서
비롯된다는 전제 위에 서 있었다.
그런데 wasm으로 구현된 Gecko가 실제 Firefox와 구분하기 어려운
API·렌더링 시그니처를 내놓을 수 있다면, 크롤러나 자동화 도구가
“진짜 Firefox처럼 보이는” 실행 환경을 임의의 호스트 브라우저
안에서 원하는 대로 생성할 수 있는 길이 열린다.
스타트업 로그에 등장하는 `general.useragent.override` 조작이나
`ui.systemUsesDarkTheme` 같은 항목들은 이미 이 프로젝트가 호스트
환경의 신호를 Firefox의 신호로 어떻게 덮어쓰는지를 보여주는
단서다.
이 프로젝트 자체가 악의적 의도로 만들어진 것은 전혀 아니지만,
같은 기법이 안티봇 우회나 자동화 탐지 회피에 재사용될 가능성은
프로젝트가 성숙해질수록 함께 커질 수밖에 없는 부산물이다.

HN 댓글에서도 비슷한 우려가 다른 각도로 제기되었다.
EvanAnderson은 광고 차단을 막으려는 사이트들이 이런 기술을 이용해
난독화된 “내부 브라우저”를 페이지 안에 서빙함으로써 광고
차단기를 무력화할 수 있다고 지적하며, 접근성 문제만 해결되면
주요 사이트들이 실제로 이 방식을 채택할 것이라고
내다봤다[^EvanAnderson].
지문 채취 우회가 크롤러·자동화 도구 쪽의 공격 표면이라면, 광고
차단 무력화는 정반대로 콘텐츠 제공자가 사용자의 브라우저 확장을
무력화하는 데 같은 기법을 쓰는 경우다.
두 시나리오 모두 “브라우저 안에서 또 다른 브라우저를 완전히
통제된 형태로 재현할 수 있다”는 동일한 능력에서 갈라져 나온
서로 다른 이해관계자의 악용 가능성이라는 점에서, 이 기술이
가진 양날의 성격을 뒷받침한다.
