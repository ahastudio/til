# WebAssembly — 포터블하고 깨지지 않는 앱의 미래

원문: [Pay attention to WebAssembly | Harshal Sheth](https://harshal.sheth.io/2022/01/31/webassembly.html)

HN 토론: <https://news.ycombinator.com/item?id=30155295> (344점, 244개 댓글)

## 요약

Harshal Sheth는 WebAssembly(Wasm)를 단순한 웹 최적화 기술이 아닌, 소프트웨어 배포의 근본적 패러다임 전환으로 분석한다.
Wasm은 이름과 달리 웹에도, 어셈블리에도 국한되지 않는다. C++, Rust부터 Python, Go, Ruby까지 30개 이상의 언어를 컴파일할 수 있는 이식 가능한 바이트코드 형식이다.

저자는 Wasm의 다섯 가지 핵심 강점을 제시한다. 첫째, 이식성이다. 표준화된 바이너리 형식이 95%의 브라우저와 Wasmtime, Wasmer 같은 서버 런타임에서 실행된다.
둘째, 언어 지원의 폭이다. 시스템 언어에 그치지 않고 가비지 컬렉션 언어도 포괄한다.
셋째, 성능이다. 네이티브 코드 대비 1.45~1.55배 느리지만, 연산 집약적 워크로드에서는 JavaScript를 압도한다.
넷째, 빠른 시작이다. OS 프로세스 오버헤드를 제거해 Docker 컨테이너 대비 콜드 스타트가 10~100배 빠르다.
다섯째, 보안이다. 메모리 샌드박싱과 기능 제약으로 무거운 프로세스 격리 없이 시스템 접근을 통제한다.

실제 도입 사례도 인상적이다. Figma는 Wasm으로 로딩 시간을 3배 단축했고, 1Password는 폼 중심 사이트에서 13~39배 속도 향상을 달성했다.
Envoy 프록시, Redpanda, Open Policy Agent는 Wasm을 플러그인 아키텍처에 채택해 사용자 제공 확장 코드를 안전하게 실행한다.
Cloudflare Workers와 Fastly Compute@Edge는 Wasm 기반 서버리스 함수를 이미 운영 중이다.

저자는 장기적으로 WebAssembly Component Model이 마이크로서비스를 더 작은 독립 배포 단위로 분해하는 아키텍처 전환을 이끌 것이라고 예측한다.
현재 채택을 막는 주요 장벽으로는 오해를 부르는 마케팅, 불완전한 WASI 표준화, 언어 간 도구 미성숙, 디버깅 지원 부족을 꼽는다.
그러나 이 장벽들이 모두 활발히 해결되고 있다는 점에서 낙관적이다.

## 분석

### 다섯 가지 강점의 상호 강화

저자가 제시한 다섯 가지 강점은 독립적인 것이 아니라 서로를 강화하는 구조다.
이식성이 있기 때문에 언어 다양성이 의미를 갖는다. 어떤 언어로 써도 같은 Wasm 바이너리가 된다면, 언어 선택이 배포 방식과 분리된다.
빠른 시작이 가능하기 때문에 보안 샌드박스의 비용이 낮아진다. 격리된 실행 환경을 스핀업하는 데 마이크로초가 걸린다면, 요청마다 새 인스턴스를 만드는 것이 현실적이 된다.
이 상호 강화가 Wasm을 단순한 성능 최적화 도구가 아닌 **아키텍처 선택지**로 만든다.

### 플러그인 아키텍처에서의 Wasm

저자가 제시하는 Envoy, Redpanda, Open Policy Agent 사례는 Wasm의 비브라우저 사용에서 가장 빠르게 성숙하는 영역을 보여준다.
플러그인 아키텍처의 고전적 딜레마는 확장성과 안전성의 트레이드오프였다.
Native 플러그인은 빠르지만 호스트를 충돌시킬 수 있고, 별도 프로세스 플러그인은 안전하지만 느리다.
Wasm은 이 딜레마를 깨는 세 번째 선택지를 제공한다. 동일 프로세스 안에서 샌드박스를 강제하는 방식이다.
이것은 “플러그인 = 위험”이라는 오래된 등식을 “플러그인 = 안전하게 격리된 확장”으로 바꾼다.

### WebAssembly Component Model의 함의

저자가 미래 전망으로 꼽은 WebAssembly Component Model은 현재 Wasm의 가장 큰 한계, 즉 모듈 간 인터페이스 문제를 다룬다.
현재 Wasm 모듈은 기본 숫자 타입만 직접 교환할 수 있다. 문자열이나 복잡한 데이터 구조를 전달하려면 메모리 포인터를 주고받아야 하고, 언어마다 메모리 표현이 달라 호환성 문제가 생긴다.
Component Model은 인터페이스 타입과 WIT(Wasm Interface Type) 형식을 정의해 이 문제를 해결한다.
이것이 완성되면 Rust로 만든 Wasm 컴포넌트가 Python 컴포넌트를 호출하는 것이 자연스럽게 되고, 언어 경계를 넘나드는 마이크로서비스 분해가 가능해진다.

## 비평

### 성능 수치의 맥락 의존성

“네이티브 대비 1.45~1.55배 느리다”는 수치는 특정 벤치마크에서 나온 것이다.
이 수치가 어떤 워크로드 유형(CPU 집약적, 메모리 집약적, I/O 집약적)에서 측정됐는지에 따라 실제 적용 결과는 달라진다.
“Docker보다 콜드 스타트가 10~100배 빠르다”는 수치도 마찬가지다. 컨테이너 설정, 레이어 캐싱 상태, 네트워크 조건에 따라 범위가 크게 변한다.
구체적인 사용 사례에서 어느 쪽 숫자가 의사결정에 실질적인 영향을 줄지는 직접 측정 없이는 알 수 없다.

HN 댓글에서는 이 수치에 직접적인 반례가 제시됐다.[^maga]
선형 회귀에서 JavaScript가 Wasm보다 10배 빠르고, Levenshtein 거리 계산에서도 JavaScript가 30% 앞선다는 벤치마크가 공유됐다.
“연산 집약적 워크로드에서 JavaScript를 압도한다”는 주장이 워크로드의 성격에 따라 뒤집힐 수 있음을 보여준다.
JIT 컴파일러가 최적화된 JavaScript 엔진과 경쟁하는 상황에서 Wasm이 항상 앞선다는 가정은 검증이 필요하다.

### "왜 Wasm은 다를까" — 이전 시도들과의 비교

Java 바이트코드, SPIR-V, LLVM IR도 모두 이식 가능한 바이너리 형식이었다.[^fancyfredbot]
"잠재력이 있다"는 것과 "실제로 성공한다"는 것은 다른 문제다.
fancyfredbot은 "Java 바이트코드, SPIR, 그 밖의 것들도 잠재력이 있었는데, 왜 Wasm은 이것들이 하지 못한 것을 해낼 것인가"라고 물었다.
이 질문이 중요한 이유는, Wasm이 단순히 기술적으로 우수하다는 주장만으로는 충분하지 않다는 것을 지적하기 때문이다.

이전 시도들이 실패한 각각의 이유가 있었다. Java 플러그인은 보안 취약점의 반복으로 브라우저에서 퇴출됐다.[^SquibblesRedux]
LLVM IR은 안정적인 ABI를 보장하지 않아 배포 단위로 쓸 수 없었다.[^labrador]
Wasm이 이 실패 패턴을 피하는 구체적인 이유가 있다. 보안이 설계의 핵심 요구사항이었고, 안정적인 바이너리 형식이 표준화의 목표였으며, 브라우저 벤더들이 처음부터 공동 설계에 참여했다.
그러나 역사적 선례는 낙관론을 경계하는 근거로 남는다.

### 2022년 시점의 한계와 현재

이 글은 2022년 초에 작성됐다. 당시 “WASI 불완전”, “언어 간 도구 미성숙”, “디버깅 지원 부족”으로 꼽은 장벽들이 2025년 현재 얼마나 해소됐는지가 중요하다.
WASI preview2가 공개됐고, Component Model 구현이 진행됐으며, 언어별 지원도 성숙했다.
반면 런타임 생태계의 분열(Wasmtime, WasmEdge, Wasmer)은 여전히 해소되지 않았다.
시간이 지나면서 일부 예측은 맞고 일부는 더 복잡해졌다는 점에서, 이 글은 현재 시점 필터를 끼고 읽어야 한다.

### 서버사이드 채택의 실질적 장벽

usrbinbash는 "WebAssembly가 서버사이드 언어를 대체하지는 않을 것"이라고 주장했다.[^usrbinbash]
백엔드 인프라는 이미 Linux와 x86-64로 수렴했고, 현대 CI/CD 파이프라인에서 컴파일 시간은 문제가 되지 않는다.
"어디서나 실행된다"는 약속이 서버 세계에서는 이미 컨테이너가 충분히 해결했다는 시각이다.

hankman86은 더 구체적인 현실적 장벽을 지적했다.[^hankman86]
Wasm의 스레딩 모델이 Web Workers와 SharedArrayBuffer에 의존해 실용적으로 다루기 까다롭고, 서버리스 환경에서 컴파일 단계를 요구하는 구조가 제약이 된다는 것이다.
goombacloud는 다른 각도를 짚었다.[^goombacloud]
Linux 프로세스 모델에서 당연하게 여기던 것들, 즉 하위 프로세스 스핀업, 파이프, `strace`와 `gdb`를 통한 추적과 디버깅을 Wasm 모델에서는 잃는다.
유니커널이 기술적으로는 우수했지만 이 편의성 손실 때문에 채택이 느렸던 것과 같은 패턴이 Wasm에서도 반복될 수 있다.

### 블록체인 활용의 과도한 낙관

Polkadot, EOS, CosmWasm의 Wasm 스마트 컨트랙트 사례를 들며 블록체인 개발 민주화를 전망하지만, 이 영역은 저자의 전망보다 훨씬 복잡하게 전개됐다.
Wasm 런타임을 쓴다는 사실이 개발 복잡성을 낮추지는 않으며, 각 블록체인 생태계의 특수한 요구사항(결정론적 실행, 가스 계량, 상태 모델)은 Wasm 표준과 별개의 문제다.
“언어 중립적 스마트 컨트랙트”라는 비전은 기술적으로 가능하지만 생태계적으로 실현되기 어려운 것이 현실이다.

## 인사이트

### 소프트웨어 배포의 “마지막 마일” 문제

소프트웨어 개발에서 “어디서나 실행된다”는 약속은 여러 번 시도됐지만 완전히 실현된 적이 없다.
Java의 JVM은 “Write Once, Run Anywhere”를 약속했지만 버전 불호환, JVM 미설치 환경, 브라우저 플러그인 보안 문제로 후퇴했다.
컨테이너는 “환경 재현 가능성”을 달성했지만 Linux 커널 의존성이라는 한계가 있다.
Wasm은 이 문제에 다른 각도로 접근한다. 런타임 환경을 표준화하는 것이 아니라 **바이너리 형식 자체를 이식 가능하게** 만든다.

이 접근의 핵심 차이는 추상화 레벨이다.
JVM은 언어 런타임을 표준화했다. 그래서 Java로만 작동하거나 JVM을 타깃으로 설계된 언어만 이 혜택을 받았다.
컨테이너는 OS 인터페이스를 표준화했다. 그래서 어떤 언어든 쓸 수 있지만 Linux 의존성이 생겼다.
Wasm은 명령어 집합 아키텍처(ISA)를 표준화했다. 언어에도 OS에도 종속되지 않는 중간 계층이다.
이 중간 계층이 처음으로 “배포 단위”와 “실행 환경”을 완전히 분리하는 가능성을 열었다.

실용적 함의는 “마지막 마일”의 문제가 사라질 수 있다는 것이다.
Wasm 바이너리를 배포하면 수신 측의 런타임이 무엇이든 실행된다.
브라우저, 서버, IoT 기기, 엣지 노드가 모두 동일한 바이너리를 실행할 수 있다면, “어떤 환경에 배포할 것인가”의 결정이 개발 시점이 아닌 운영 시점으로 이동한다.
이것은 소프트웨어의 유통 방식을 근본적으로 바꾸는 변화다.

### POSIX 이후의 시스템 인터페이스

UNIX/POSIX는 1970년대에 설계된 시스템 인터페이스 표준이다.
파일, 소켓, 프로세스, 신호라는 추상화는 수십 년간 소프트웨어 개발의 공통 언어였다.
그러나 이 추상화는 단일 머신, 신뢰할 수 있는 코드, 동기적 실행을 전제로 설계됐다.
클라우드, 분산 시스템, 신뢰되지 않는 코드 실행이 일상이 된 환경에서 POSIX는 점점 맞지 않는 옷이 되고 있다.

WASI(WebAssembly System Interface)는 POSIX 이후의 시스템 인터페이스를 설계하는 시도다.
기능 기반(capability-based) 보안 모델을 채택해, 코드가 어떤 자원에 접근할 수 있는지를 인터페이스 수준에서 제어한다.
“파일을 열 수 있다”는 POSIX의 전제 대신, “이 특정 디렉토리에만 접근할 수 있다”는 명시적 권한 모델이다.
이것은 단순히 보안이 좋은 POSIX가 아니라, 최소 권한 원칙을 기본 설계 원칙으로 삼는 새로운 인터페이스다.

이 변화의 2차 효과는 소프트웨어 공급망 보안이다.
npm 패키지가 파일 시스템 전체에 접근하거나, PyPI 패키지가 네트워크 연결을 임의로 열 수 있는 것은 POSIX 모델의 필연적 결과다.
WASI 기반 패키지 생태계에서는 패키지가 선언한 권한만 실행 시점에 허용된다.
의존성 공급망 공격의 피해 범위가 패키지에 허용된 권한으로 제한된다.
이것은 소프트웨어 공급망 보안의 기술적 기반을 바꾸는 잠재력이다.

### 컴포넌트 모델이 만드는 새로운 소프트웨어 경제

WebAssembly Component Model이 완성되면, 소프트웨어 재사용의 단위가 바뀐다.
현재는 라이브러리가 같은 언어 생태계 안에서만 재사용된다. npm 패키지는 JavaScript/TypeScript 프로젝트에서만, PyPI 패키지는 Python 프로젝트에서만 쓸 수 있다.
언어 경계를 넘으려면 FFI(Foreign Function Interface)라는 복잡한 통로가 필요하고, 이 통로는 대개 C ABI를 통해야 한다.

Component Model이 언어 중립적 인터페이스를 표준화하면, Rust로 작성된 고성능 암호화 라이브러리를 Python 프로젝트에서 Wasm 컴포넌트로 직접 가져다 쓸 수 있게 된다.
재사용의 마찰이 크게 줄어들면, 각 언어 생태계가 중복해서 개발하는 라이브러리들이 통합된다.
“이 기능의 최고 구현체”가 언어마다 따로 존재하는 것이 아니라, Wasm 컴포넌트로 하나 존재하고 모든 언어에서 사용되는 구조가 가능해진다.

이것은 오픈소스 생태계의 구조를 바꾼다.
현재는 언어별로 패키지 레지스트리(npm, PyPI, crates.io, Maven)가 분리되어 있다.
언어 중립 Wasm 컴포넌트 레지스트리가 성숙하면, 기여의 단위도 언어 종속적인 라이브러리에서 언어 중립적인 컴포넌트로 이동할 수 있다.
성능이 중요하거나 보안이 검증된 핵심 구현체를 공유하면서, 언어별 래퍼만 얇게 제공하는 생태계 구조다.
Java 시대의 “한 번 작성하면 어디서나 실행”이 언어 차원에서 실패한 것을, 컴포넌트 차원에서 다시 시도하는 셈이다.
