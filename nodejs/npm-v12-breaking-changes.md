# npm v12의 breaking changes 예고

원문: <https://github.blog/changelog/2026-06-09-upcoming-breaking-changes-for-npm-v12/>

HN 토론: <https://news.ycombinator.com/item?id=48467705> (310점, 108개 댓글)

## 요약

GitHub는 2026년 6월 9일, 2026년 7월 출시 예정인 npm v12에 포함될 세 가지
주요 breaking changes를 예고했다. 세 변경 모두 “보안 관련 기본값 변경”으로
규정되며, 설치 과정에서 자동으로 실행되는 코드와 외부 의존성 해석 경로를
기본적으로 차단하는 방향이다. 핵심 메시지는 명확하다. 지금까지 npm이
암묵적으로 허용해 오던 동작들을 명시적 승인이 있을 때만 허용하도록
기본값을 뒤집겠다는 것이다.

첫 번째 변경은 `allowScripts`의 기본값이 off로 바뀌는 것이다. 이제
`npm install`은 의존성의 `preinstall`, `install`, `postinstall`, `prepare`
스크립트를 프로젝트에서 명시적으로 허용하지 않는 한 실행하지 않는다.
`node-gyp`를 사용하는 네이티브 빌드 역시 기본적으로 차단된다. 사용자는
v12로 올라가기 전에 `npm approve-scripts --allow-scripts-pending` 명령으로
신뢰하는 패키지를 미리 허용해 둘 수 있으며, 이 기능은 npm 11.16.0
이상에서 제공된다.

두 번째 변경은 `--allow-git`의 기본값이 `none`으로 바뀌는 것이다. 이로써
직접 또는 전이(transitive) Git 의존성이 기본적으로 차단된다. 이 변경은
이미 2026년 2월 18일에 한 차례 예고되었으며, Git 의존성의 `.npmrc`가
Git 실행 파일 자체를 덮어쓸 수 있는 코드 실행 취약점을 닫는 것을 목표로
한다. 이 경로는 `--ignore-scripts`를 적용해도 막히지 않던 문제였다. 이
기능은 npm 11.10.0 이상에서 제공된다.

세 번째 변경은 `--allow-remote`의 기본값이 `none`으로 바뀌는 것이다.
HTTPS tarball 같은 원격 URL 의존성이 명시적으로 허용되지 않는 한
기본적으로 차단된다. 이 기능은 npm 11.15.0 이상에서 제공된다. GitHub는
사용자에게 npm 11.16.0 이상으로 미리 올라가서 설치 경고를 확인하고,
`npm approve-scripts`로 신뢰하는 패키지를 승인한 뒤 v12로 이행할 것을
권한다. 문서로는 `npm approve-scripts`, `npm deny-scripts`,
`allow-scripts` 설정이 안내되며, 2026년 5월 22일자 “Staged publishing and
new install-time controls for npm” 게시물이 관련 자료로 연결된다.

## 분석

### 세 변경의 공통 표적은 “설치 시점 코드 실행”이다

세 가지 변경은 표면적으로 서로 다른 기능을 건드리지만, 실제로는 하나의
공격면을 겨냥한다. 바로 `npm install`이라는 명령 하나가 임의의 코드 실행과
임의의 네트워크 페치를 동시에 트리거할 수 있다는 사실이다. 라이프사이클
스크립트는 코드 실행을, Git 의존성과 원격 URL 의존성은 신뢰되지 않은
출처로부터의 코드 유입을 담당한다. npm은 이 셋을 묶어 한 번에 기본값을
뒤집음으로써, “패키지를 설치한다”는 행위와 “패키지의 코드를 실행한다”는
행위를 분리하려 한다.

이 분리는 공급망 공격(supply chain attack)의 표준 시나리오에 대한 직접적인
대응이다. 지난 몇 년간 npm 생태계를 흔든 사건 대부분은 악성 패키지가
`postinstall` 훅에서 자격 증명을 탈취하거나, 전이 의존성으로 숨어든
Git 참조가 빌드 머신에서 코드를 실행하는 형태였다. npm의 논리는,
대다수 패키지는 설치 시점에 코드를 실행할 필요가 없으므로, 실행이
필요한 소수만 명시적으로 허용하면 공격면이 극적으로 줄어든다는 것이다.

HN에서도 이 논리에 동의하는 목소리가 있었다. `woodruffw`는 사용자
대다수(어쩌면 과반)는 설치 스크립트가 필요 없으므로 기본값을 끄는
것은 순수한 보안 개선이며, 스크립트가 필요한 쪽은 명시적이고 추적·감사
가능한 결정으로 전환된다는 점을 짚었다[^woodruffw]. `grassfedgeek`는
웹 앱에서만 쓰이는 JavaScript 라이브러리에는 “첫 실행” 자체가
존재하지 않으므로, 그 부류의 패키지 전체가 이 변경으로 안전해진다고
지적했다[^grassfedgeek]. 한편 이 방향이 새롭지 않다는 냉소도 있었다.
`thatmf`는 pnpm이 18개월 전에 같은 결정을 내린 것을 npm이 뒤늦게
따라온 것이라 꼬집었고[^thatmf], `retardedsecguy`는 “npm은 이제
사실상 pnpm”이라고 일축했다[^retardedsecguy].

### 기본값을 뒤집는다는 것의 무게

이 변경의 본질은 새 기능 추가가 아니라 기본값 역전이다.
`allowScripts`, `--allow-git`, `--allow-remote`에 해당하는 동작은 이미
오래전부터 존재했고, 끄는 옵션(`--ignore-scripts` 등)도 있었다. 바뀌는
것은 단 하나, 명시적으로 끄지 않으면 켜져 있던 것을 명시적으로 켜지
않으면 꺼져 있도록 바꾸는 것이다. 이것은 보안 공학에서 “secure by
default”라 부르는 원칙의 교과서적 적용이다.

그러나 기본값 역전은 기능 추가보다 훨씬 파괴적이다. 새 플래그는 쓰는
사람만 영향을 받지만, 기본값 변경은 아무것도 바꾸지 않은 사람까지
전부 영향을 받는다. npm은 이 점을 알기에 변경을 단계적으로 풀어 왔다.
각 기능이 npm 11.10.0, 11.15.0, 11.16.0에서 미리 사용 가능하도록
배포된 뒤, 메이저 버전 경계인 v12에서 비로소 기본값이 뒤집힌다.
즉 “기능은 미리, 기본값은 나중에”라는 패턴이다.

### npm의 긴 deprecation 역사 속 위치

이 변경은 npm이 메이저 버전마다 생태계에 부담을 전가해 온 역사의
연장선에 있다. npm v7은 peerDependencies 자동 설치를 강제하면서
수많은 프로젝트의 설치를 깨뜨렸고, lockfile 포맷을 v2로 바꿔 대규모
diff를 유발했다. 그 이전에도 npm은 `package-lock.json` 도입, `npm ci`
분리 등으로 워크플로를 반복적으로 재정의해 왔다.

이번 v12 변경이 이전과 다른 점은, 동기가 편의나 성능이 아니라 보안이며,
npm이 명시적으로 “공격 경로”를 언급한다는 데 있다. peerDependencies
변경은 “이렇게 하는 게 맞다”는 규범적 주장이었지만, 이번 변경은
“이렇게 하지 않으면 위험하다”는 위협 기반 주장이다. 이 차이는 사용자가
거부하기 어렵게 만든다. 성능 트레이드오프는 협상할 수 있어도, 코드
실행 취약점은 협상 대상이 되기 어렵기 때문이다.

### npm이 전제하는 생태계의 업그레이드 내성

이 발표는 npm 생태계가 상당한 업그레이드 내성(upgrade tolerance)을
가졌다고 전제한다. GitHub는 7월 출시 전에 사용자들이 11.16.0으로
올라가 경고를 검토하고 신뢰 패키지를 승인할 것이라 가정한다. 그러나
이 가정은 npm을 능동적으로 관리하는 상위 사용자층을 모델로 한다.

현실의 npm 사용자 대다수는 CI 이미지에 고정된 Node와 번들된 npm을
쓰며, 그 버전이 언제 v12로 점프할지 능동적으로 통제하지 않는다.
Node 메이저 릴리스에 npm v12가 번들되는 순간, 발표를 읽지 않은
수많은 파이프라인이 한꺼번에 기본값 변경을 맞닥뜨린다. npm의 단계적
배포 전략은 정교하지만, 그 정교함이 닿는 대상은 발표를 읽는 사람으로
한정된다는 한계가 있다.

## 비평

### 발표는 “누가 승인을 수행하는가”라는 핵심 질문을 비껴간다

`npm approve-scripts`라는 해법은 누군가가 어떤 패키지가 스크립트를
실행해도 되는지 판단하고 승인해야 한다는 노동을 전제한다. 발표는 이
명령의 존재를 안내하지만, 그 판단을 누가, 무슨 근거로 내려야 하는지는
침묵한다. 전형적인 프로젝트는 수백 개의 전이 의존성을 가지며, 그중
어떤 것이 정당한 네이티브 빌드를 위해 스크립트가 필요하고 어떤 것이
악성인지 개발자가 일일이 가려낼 능력은 사실상 없다.

이 구조의 위험은 보안 도구가 흔히 빠지는 함정과 같다. 승인 프롬프트가
너무 자주 뜨면 사용자는 내용을 읽지 않고 전부 승인하는 습관을 들인다.
브라우저의 인증서 경고, 모바일의 권한 요청이 모두 같은 운명을 밟았다.
`--allow-scripts-pending` 같은 일괄 승인 경로가 제공되는 순간,
실질적인 보안 효과는 “검토된 허용”이 아니라 “한 번의 무비판적 허용”으로
수렴할 가능성이 크다.

발표는 마치 승인이 일회성 설정 작업인 것처럼 서술하지만, 실제로는
의존성이 갱신될 때마다 새 스크립트가 등장하고 재승인이 필요해진다.
이 반복적 마찰의 비용을 발표는 전혀 계량하지 않는다. 보안의 이득은
구체적으로 제시하면서 그 대가는 “신뢰하는 패키지를 승인하세요”라는
한 문장으로 압축해 버리는 것은 비대칭적인 서술이다.

HN의 `heldrida`는 더 근본적인 의문을 제기했다. 승인 결과로 만들어지는
allowlist가 `package.json`에 기록된다면, 결국 v12 이전의 “전부 허용”
기본값과 사실상 같은 상태로 수렴하는 것 아니냐는 것이다[^heldrida]. 다만
구조상 개선의 여지를 본 시각도 있었다. `karakanb`는 allowlist가 전역
설정이 아니라 패키지 단위 화이트리스트를 지원하는 것으로 보이며, 이는
조직 차원에서 특정 패키지에만 스크립트를 허용하는 규칙을 유지하기 쉽게
만든다고 평가했다[^karakanb].

### 차단의 비용은 직접 의존성이 아니라 전이 의존성 사용자가 짊어진다

`--allow-git`과 `--allow-remote`가 전이 의존성까지 차단한다는 점은
발표에서 한 줄로 지나가지만, 여기에 가장 큰 파급이 숨어 있다. 내가
직접 추가한 의존성이 Git이나 원격 URL을 쓰지 않더라도, 내가 의존하는
어떤 패키지의 패키지가 그렇게 한다면 내 설치는 깨진다. 그리고 그 수정
권한은 나에게 없다.

이 구조에서 비용을 짊어지는 주체와 문제를 일으킨 주체가 어긋난다.
Git 의존성을 넣은 것은 상류(upstream) 메인테이너이지만, 설치
실패를 맞닥뜨리고 우회 플래그를 찾아 헤매는 것은 아무 잘못 없는 하류
사용자다. 발표는 “명시적으로 허용하라”고 말하지만, 전이 의존성의
Git 참조를 허용한다는 것은 곧 내가 통제하지 못하는 코드 유입 경로를
스스로 다시 열어 주는 일이다. 결국 보안과 동작 가능성 사이에서
사용자가 어느 쪽도 깔끔하게 선택할 수 없는 상황이 만들어진다.

발표는 이 변경이 생태계의 상류를 압박해 Git/원격 의존성을 정식 레지스트리
패키지로 전환하게 만들 것이라고 낙관할 수도 있다. 그러나 그 전환에는
시간이 걸리며, 그 사이의 공백을 메우는 것은 하류 사용자의 우회 설정과
깨진 CI다. 메인테이너가 게으르거나 이미 활동을 멈춘 패키지라면 그
전환은 영영 일어나지 않을 수도 있다.

### 단계적 배포는 정교하지만 번들링이라는 현실을 무시한다

GitHub는 기능을 11.10.0, 11.15.0, 11.16.0에 미리 풀고 v12에서 기본값을
바꾸는 신중한 경로를 택했다. 이 설계는 발표를 읽고 능동적으로 준비하는
사용자에게는 흠잡을 데 없다. 문제는 npm이 Node.js에 번들되어 배포된다는
구조적 현실이다.

대부분의 사용자는 npm을 독립적으로 업그레이드하지 않는다. 그들은 Node
버전을 올릴 때 따라오는 npm을 받을 뿐이다. 발표가 권하는 “11.16.0으로
미리 올라가 경고를 검토하라”는 조언은, npm을 Node와 분리해 관리하는
소수에게만 실천 가능하다. 다수에게 v12는 어느 날 Node 업그레이드와
함께 예고 없이 도착하며, 그 순간 단계적 경고 기간은 의미를 잃는다.

게다가 발표는 Node.js 버전 요구사항에 대해 아무 언급이 없다. npm v12가
어떤 Node 메이저에 번들될지, 기존 LTS에 백포트될지 여부가 불명확하다.
이 정보 공백은 운영팀이 가장 알고 싶어 하는 것, 즉 “우리 파이프라인이
정확히 언제 이 변경을 맞게 되는가”에 답하지 못한다. 보안 변경의 타임라인이
정작 그것을 감당할 조직에게 흐릿하게 남아 있는 셈이다.

이 번들링 문제는 HN에서 가장 구체적인 실무 질문으로 떠올랐다.
`alexghr`는 현재 LTS Node(22, 24, 26)가 번들 npm을 v12로 갱신해 이
보안 수정을 받을 것인지 물었고, 지금은 모두 v11을 달고 온다고
지적했다[^alexghr]. 이에 `jamiem`은 메이저 npm 버전이 과거에도 LTS
중간에 올라온 전례를 들며(Node v18.19.0과 v20.10.0이 npm 9를 10으로
올린 사례) 충분히 가능한 일이라고 답했다[^jamiem]. 흥미롭게도 `b112`는
바로 이 “예고 없는 깨짐”을 오히려 긍정적으로 봤다. 새 개발자들이 그냥
설치만 했다가 까다로운 패키지에서 즉각 깨짐을 겪으면, 스크립트가 당연히
실행되리라는 기대 자체를 버리게 될 것이라는 논리다[^b112].

### “secure by default”는 책임을 사용자에게 떠넘기는 수사일 수 있다

기본값을 안전하게 바꾼다는 명분은 거부하기 어렵다. 그러나 이 수사는
종종 책임의 이전을 가린다. 이전까지 npm은 편의를 위해 위험한 기본값을
유지했고, 그 위험은 npm 생태계 전체가 공유했다. 이제 기본값을 뒤집으면
설치가 깨졌을 때의 해결 책임은 개별 사용자에게 넘어간다. 플랫폼은
“우리는 안전한 기본값을 제공했고, 그것을 켠 것은 당신”이라고 말할 수
있는 위치에 선다.

이 관점에서 보면 v12 변경은 보안 향상인 동시에 책임의 재배치다.
공급망 보안의 근본 문제는 레지스트리 차원의 패키지 검증, 서명, 출처
증명이 충분히 작동하지 않는다는 데 있다. 설치 시점 차단은 이 근본
문제를 풀지 않고, 각 사용자에게 “당신이 직접 신뢰를 판단하라”고
위임함으로써 우회한다. 진짜 해법인 레지스트리 차원의 신뢰 인프라
대신, 더 손쉬운 클라이언트 차원의 차단을 택한 것이다.

물론 클라이언트 차단도 의미 있는 방어 계층이다. 그러나 발표가 이를
마치 공급망 문제의 해결책인 양 제시한다면 그것은 과장이다. 악성
패키지가 설치 시점이 아니라 런타임에 동작하도록 설계되면 이 모든
차단을 우회한다. 스크립트 실행 차단은 한 종류의 공격을 막을 뿐,
패키지가 실행되는 한 임의 코드 실행이라는 본질적 위험은 남는다.

이 지적은 HN에서 가장 자주 반복된 반론이기도 하다. `peterkelly`는
악성코드가 설치 스크립트에서 모듈 본체로 옮겨 가 결국 똑같이 실행될
것이라 내다봤고[^peterkelly], `TZubiri`는 이 변경이 침해 시점을
“첫 설치”에서 “첫 실행”으로 옮길 뿐 아니냐고 물었다[^TZubiri]. 이에
대한 응수도 날카로웠다. `cookiengineer`는 이 변경이 “보안 개선이 아니라
책임 전가”라고 잘라 말하며, 강제로 모두가 `--enable-unsecure-feature`
류의 플래그를 켜게 만들 뿐이라고 비판했다. 그는 대안으로 어떤 스크립트와
프로그램이 실행될지 미리 출력하는 샌드박스나 시뮬레이션 실행을
제시했다[^cookiengineer].

## 인사이트

### 이 변경은 “lockfile만으로는 부족하다”는 고백이다

npm은 오랫동안 `package-lock.json`을 재현 가능한 설치의 보증으로
제시해 왔다. lockfile은 무엇이 설치되는지를 고정한다. 그러나 v12
변경은 무엇이 설치되는지를 고정하는 것만으로는 무엇이 실행되는지를
통제하지 못한다는 사실을 인정한다. 설치되는 패키지가 정확히 그것이어도,
그 패키지의 `postinstall`이 무슨 일을 하는지는 lockfile이 말해 주지
않는다.

이것은 신뢰 모델의 한 층이 더 추가됨을 의미한다. 지금까지 신뢰는
“버전 고정” 한 축으로 관리되었다면, 이제 “실행 권한”이라는 별도의
축이 생긴다. 앞으로 프로젝트는 어떤 버전을 쓰는가뿐 아니라 어떤
패키지에 스크립트 실행을 허용했는가도 별도로 기록하고 관리해야 한다.
이 승인 상태 역시 일종의 lockfile처럼 버전 관리되어야 하며, 팀 간에
공유되고 코드 리뷰의 대상이 되어야 한다.

이 2차 효과는 발표가 예상하지 못한 새로운 거버넌스 부담을 만든다.
“누가 어떤 패키지의 스크립트 실행을 승인했는가”는 이제 감사
(audit)의 대상이 된다. 보안 사고가 났을 때 “왜 그 패키지를 승인했는가”가
사후 추궁의 질문이 되며, 이는 승인 행위에 심리적 무게를 더한다.
역설적으로 이 무게가 너무 무거워지면 사람들은 판단을 회피하기 위해
일괄 승인으로 다시 도망칠 수 있다.

### Python 2에서 3으로의 이행이 보여 준 교훈이 여기에도 적용된다

기본값 역전이 생태계에 미치는 충격을 이해하려면 Python 2에서 3으로의
이행을 떠올리는 것이 유용하다. Python 3는 기술적으로 더 나은 선택이었고
보안과 유지보수 측면에서 옳았지만, 생태계 전체가 따라오는 데 10년이
넘게 걸렸다. 문제는 옳음이 아니라, 옳은 변경을 강제할 때 발생하는
전이 비용이 누구에게 어떻게 분산되는가였다.

npm v12의 전이 의존성 차단은 같은 종류의 분산 문제를 안고 있다.
한 패키지가 Git 의존성을 정식 레지스트리 패키지로 전환하기 전까지,
그 패키지에 의존하는 모든 하류가 우회 설정을 유지해야 한다. Python에서
2와 3 코드가 오래 공존하며 `six` 같은 호환 계층 라이브러리가 번성했듯이,
npm에서도 “Git 의존성을 안전하게 래핑하는” 중간 계층이나 우회
도구가 등장할 가능성이 높다.

여기서 끌어낼 수 있는 교훈은, 강제된 보안 개선은 종종 그것을 우회하는
새로운 생태계 산업을 낳는다는 것이다. 마이그레이션이 충분히 고통스러우면
사람들은 마이그레이션 대신 고통을 완화하는 도구를 만든다. v12 이후
“승인 자동화”, “신뢰 패키지 화이트리스트 공유”, “조직 차원 승인 정책”
같은 도구와 서비스가 등장한다면, 그것은 npm이 떠넘긴 판단 부담을
시장이 다시 흡수하는 과정으로 볼 수 있다.

HN의 한 갈래 토론은 이 “우회 생태계” 가능성을 정면으로 다뤘다.
`zarzavat`는 공급망 공격을 막는 더 확실한 길로 게시 사용자가 모든
릴리스를 스마트폰 앱에서 승인하게 하고 정부 발급 신분증으로 신원을
검증하게 하는 레지스트리 차원의 강제를 제안했다[^zarzavat]. 그러나
`ifwinterco`는 이런 강제가 너무 고통스럽고 세계 인구의 상당수를 배제하는
탓에, 결국 그런 통제가 없는 병렬 패키지 생태계가 생겨 모두가 그쪽으로
옮겨 갈 것이라고 반박했다[^ifwinterco]. 클라이언트 차단이든 레지스트리
강제든, 마찰이 임계점을 넘으면 우회 경로가 자생한다는 동일한 동학이
양쪽 모두에 작동하는 셈이다.

### node-gyp 차단은 네이티브 생태계에 불균형한 타격을 준다

발표는 네이티브 빌드가 기본적으로 차단된다는 사실을 한 줄로 처리하지만,
이 결정은 JavaScript 생태계의 특정 부분에 불균형하게 무겁다. `node-gyp`로
네이티브 모듈을 컴파일하는 패키지들은 본질적으로 설치 시점 스크립트
실행이 필요하다. `bcrypt`, 데이터베이스 드라이버, 이미지 처리
라이브러리처럼 광범위하게 쓰이는 패키지들이 여기 속한다.

이들에게 “스크립트 실행을 차단하고 필요하면 승인하라”는 정책은 사실상
“이 패키지를 쓰려면 매번 승인하라”는 뜻이다. 즉 네이티브 패키지를 쓰는
프로젝트는 순수 JavaScript 패키지만 쓰는 프로젝트보다 구조적으로 더
많은 마찰을 떠안는다. 이는 prebuilt 바이너리를 배포하는 방식
(`prebuild`, `node-pre-gyp`, N-API 기반 prebuilds)으로의 이동을 가속할
유인이 된다.

이 trade-off는 발표가 거의 다루지 않지만 장기적으로 중요하다. 네이티브
모듈이 점점 설치 시점 컴파일을 포기하고 prebuilt 바이너리로 옮겨 가면,
이번엔 바이너리의 출처와 무결성을 누가 검증하느냐는 새로운 신뢰 문제가
생긴다. 컴파일을 막아 코드 실행을 줄였더니, 검증되지 않은 바이너리를
내려받는 또 다른 신뢰 경로가 부상하는 것이다. 한 공격면을 닫는 것이
다른 공격면을 키우는 이 풍선 효과는, 공급망 보안이 단일 지점의 차단으로
끝나지 않는 이유를 잘 보여 준다.
