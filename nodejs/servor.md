# Servør: 의존성 없는 정적 파일 서버

<https://github.com/lukejacksonn/servor>

## 소개

Servør는 싱글 페이지 애플리케이션 개발을 위한 의존성 없는(dependency
free) 파일 서버다.
Node.js와 브라우저의 네이티브 API만 사용해 최소한의 코드로 특정 작업을
수행하는 것을 목표로 처음부터 작성됐다.
`npx servor`로 바로 실행할 수 있고, Node API로 프로그래밍 방식으로도
사용할 수 있다.

빠른 시작 예시는 GitHub 저장소를 직접 clone해서 서버를 띄우는 방식을
보여준다.

```bash
npx servor gh:lukejacksonn/perflink --browse --editor --reload
```

이 명령은 `perflink` 저장소를 clone하고, 프로젝트 루트에 서버를
띄우고, 브라우저에서 URL을 열고, 에디터에서 소스 코드를 열고, 파일이
바뀌면 브라우저를 새로고침한다.

주요 기능은 정적 콘텐츠 서빙, 파일 변경 시 브라우저 리로드, 일반
파일 형식에 대한 gzip 적용, 신뢰할 수 있는 자체 서명 인증서를 이용한
HTTPS/HTTP2 지원, 프런트엔드 라우팅을 위한 단일 파일로의 경로 리다이렉트,
HTML과 JavaScript 파일 모두를 루트 파일로 허용, 기본 포트가 사용 중일
때 자동으로 다른 포트 탐색, 슬래시로 끝나는 URL에 대한 디렉터리 목록
렌더링, 빠른 시작을 위한 브라우저 탭·코드 에디터 자동 오픈 등이다.

## CLI

```bash
npx servor <root> <fallback> <port>
```

세 위치 인자는 각각 정적 파일을 서빙할 경로(기본값 `.`), 파일이 아닌
모든 요청에 서빙할 폴백 파일(기본값 `index.html`), 서빙할 포트(기본값
`8080`)다.
`<root>`에는 `gh:<user>/<repository>` 문법으로 GitHub 저장소를 직접
지정할 수도 있다.

선택 플래그로는 서버 시작 시 브라우저를 여는 `--browse`, 파일 변경 시
브라우저를 새로고침하는 `--reload`, 생성된 인증서로 HTTPS를 시작하는
`--secure`, 서버 프로세스의 stdout 로깅을 막는 `--silent`, 루트를
`script type module` 태그로 감싸는 `--module`, 중첩된 index 파일이
있으면 그쪽으로 라우팅하는 `--static`, 코드 에디터(현재 VS Code만
지원)를 프로젝트 루트에서 여는 `--editor`가 있다.

`--secure` 플래그를 위한 인증서 생성 과정은 `openssl` 명령에 의존하며
(macOS·Linux에서만 테스트됨), 인증서 파일이 없을 때 `certify.sh`가
자동으로 로컬 인증 기관, 공개 인증서, 개인 키를 생성한다.
브라우저가 자체 서명 인증서를 신뢰하게 하려면 `sudo servor --secure`로
루트 인증서를 시스템 키체인에 추가해야 하며, 이 방식은 macOS에서만
동작한다고 명시되어 있다.

## API

Node 모듈로 요청해 프로그래밍 방식으로도 사용할 수 있다.

```javascript
const servor = require('servor');
const instance = await servor({
  root: '.',
  fallback: 'index.html',
  module: false,
  static: false,
  reload: false,
  inject: '',
  credentials: null,
  port: 8080,
});
```

`servor` 함수는 설정 객체를 받아 서버를 시작하고, `url`, `root`,
`protocol`, `port`, `ips`를 담은 설정 설명 객체를 반환한다.
`inject` 속성은 루트 문서(기본값 `index.html`)에 덧붙일 문자열을
받아 설정 주입이나 기능 확장에 쓸 수 있고, `credentials` 속성은
`cert`와 `key`를 담은 객체를 받아 유효하면 HTTPS로 서빙한다.

## 분석

### “의존성 없음”이라는 제약이 설계를 결정한다

Servør의 README는 첫 문장부터 “의존성 없는(dependency free)”을
내세운다.
이 제약은 단순한 마케팅 문구가 아니라 코드베이스 전체의 설계 방향을
결정하는 원칙이다.
Node와 브라우저의 네이티브 API만으로 HTTPS 인증서 생성, gzip 압축,
파일 변경 감지, 포트 탐색까지 구현해야 하므로, 각 기능은 별도
라이브러리가 흔히 제공하는 완결성 대신 “필요한 만큼만” 구현된
형태를 띤다.
`openssl` CLI를 셸에서 직접 호출해 인증서를 만드는 방식이 대표적이다.
Node의 `crypto` 모듈로 인증서를 만드는 라이브러리(`selfsigned` 같은)를
쓰는 대신 시스템에 이미 설치된 `openssl` 바이너리에 위임함으로써,
npm 의존성 트리에 아무것도 추가하지 않으면서 기능은 구현하는 절충을
택했다.

### GitHub 저장소를 루트로 지정하는 기능은 배포 마찰을 없앤다

`gh:<user>/<repository>` 문법으로 GitHub 저장소를 곧바로 서빙 루트로
지정할 수 있게 한 것은, 로컬 개발 서버의 전통적 전제(먼저 clone하고,
그다음 서버를 켠다)를 하나의 명령으로 압축한 설계다.
빠른 시작 예시가 `perflink`라는 다른 저장소를 대상으로 하는 것은
의도적이다.
이 기능이 지향하는 사용 시나리오는 “이 프로젝트가 어떻게 동작하는지
빨리 보고 싶다”는 탐색적 요구이지, 자신의 프로젝트를 서빙하는 일상적
개발 루프가 아니다.
`--browse --editor --reload`를 한 명령에 조합한 것도 같은 맥락으로,
설치·clone·서버 실행·브라우저 열기·에디터 열기라는 다섯 단계를 하나의
`npx` 호출로 축약한다.

### CLI와 API가 같은 설정 스키마를 공유하는 구조

CLI 위치 인자(`root`, `fallback`, `port`)와 플래그(`--reload`,
`--secure` 등)는 Node API의 설정 객체 필드(`root`, `fallback`, `port`,
`reload`, `credentials` 등)와 정확히 대응한다.
이것은 두 인터페이스가 서로 다른 파서 뒤에서 결국 하나의 내부
함수(`servor(config)`)로 수렴하는 구조라는 뜻이다.
이런 설계는 CLI 전용 로직과 API 전용 로직이 따로 분기해서 생기는
불일치 버그를 줄인다.
다만 이 대응 관계가 README에 명시적으로 표로 정리되어 있지는 않고,
두 섹션을 나란히 읽어야 유추할 수 있는 형태로만 제시된다.

## 비평

### “의존성 없음”의 정의가 시스템 의존성을 가린다

Servør는 npm 패키지 의존성이 없다는 점을 강조하지만, `--secure`
플래그의 인증서 생성은 `openssl` 명령에, 신뢰 저장소 등록은 `sudo`와
`security` 명령(macOS 전용)에 의존한다.
이것은 npm 의존성 트리에는 나타나지 않지만 실질적으로는 여전히
외부 의존성이다.
사용자가 “의존성 없음”이라는 표현만 보고 어떤 환경에서도 문제없이
동작하리라 기대한다면, `openssl`이 없는 최소 컨테이너 환경이나
Windows에서 `--secure`가 작동하지 않는 상황에 부딪힐 수 있다.
README가 “macOS와 Linux에서만 테스트됨”, “macOS에서만 테스트됨”이라고
주석을 달아두긴 했지만, 이 제약이 프로젝트 설명 최상단의 “의존성
없음” 문구와 같은 비중으로 노출되지는 않는다.

### 기본값이 대부분 비활성화되어 있다는 설명이 실제 필요를 가린다

“대부분의 기능은 기본적으로 비활성화되어 있다”는 문장은 Servør를
가볍게 보이게 하지만, 실제로 개발 서버를 쓰는 상황에서 `--reload`
없이 정적 파일만 서빙하는 경우는 드물다.
`package.json`의 예시 스크립트조차 `--reload --browse`를 기본으로
포함하고 있다는 사실이 이를 방증한다.
결국 “기본값은 최소”라는 설계 원칙과 “실제 사용에는 플래그 조합이
거의 항상 필요”하다는 현실 사이에는 간극이 있고, 이 간극은 사용자가
매번 플래그를 기억해서 조합해야 하는 부담으로 전가된다.

### 라이브 리로드 메커니즘이 문서에 설명되지 않는다

`--reload` 기능이 어떤 방식으로 파일 변경을 브라우저에 전달하는지
(WebSocket인지, Server-Sent Events인지, 폴링인지) README는 전혀
설명하지 않는다.
이것은 사용자가 이 기능을 신뢰하고 채택하기 전에 확인하고 싶어할
기본적인 동작 방식인데, 문서는 “파일이 추가·삭제·수정되면 브라우저를
새로고침한다”는 결과만 기술한다.
의존성 없는 구현이라는 프로젝트의 정체성을 감안하면, 리로드
메커니즘을 어떻게 네이티브 API만으로 구현했는지가 오히려 프로젝트의
핵심 기술적 성취일 수 있는데, 그 설명이 README에서 빠져 있다.

## 인사이트

### “제로 의존성”은 신뢰의 지표가 아니라 신뢰 비용의 이전이다

npm 생태계에서 “의존성 없음”은 흔히 안전성과 안정성의 상징으로
소비된다.
공급망 공격, `left-pad` 사태 같은 사건들이 이 인식을 강화해 왔다.
하지만 Servør 사례가 보여주듯, 의존성을 없앤다는 것은 위험을
제거하는 것이 아니라 위험의 형태를 바꾸는 것에 가깝다.
npm 패키지에 위임했다면 그 패키지의 유지보수자가 인증서 생성 로직의
버그나 보안 취약점을 책임졌겠지만, 이제 그 책임은 프로젝트 저자
본인과, `openssl`이라는 시스템 바이너리의 존재 및 버전 호환성에
넘어간다.
“의존성 없음”을 표방하는 프로젝트를 평가할 때는 “정말 아무것도
필요 없는가”가 아니라 “필요한 것이 어디로 이동했는가”를 물어야
정확한 그림을 얻을 수 있다.

### 소규모 도구의 “빠른 시작 예시”는 실제 채택 시나리오를 압축해서 보여준다

Servør의 빠른 시작 예시가 자신의 저장소가 아니라 저자의 다른
프로젝트(`perflink`)를 clone해서 서빙하는 명령이라는 점은 사소해
보이지만 의미가 있다.
이것은 이 도구가 상정하는 첫 사용자 경험이 “내 프로젝트에 개발
서버를 설치한다”가 아니라 “어떤 프로젝트든 즉시 훑어본다”는 것을
드러낸다.
CLI 도구의 문서에서 첫 예시로 무엇을 고르는지는 그 도구가 스스로를
어떤 카테고리(개발 인프라 vs 탐색 도구)로 규정하는지를 보여주는
신호다.
Servør는 결과적으로 두 카테고리에 걸쳐 있지만, 문서의 첫인상은
탐색 도구 쪽에 가깝게 설계되어 있다.

### HTTPS 로컬 개발 지원의 무게중심이 시스템 신뢰 저장소로 이동했다

브라우저가 로컬 개발 환경에서도 HTTPS를 점점 더 강하게 요구하는
추세(Service Worker, 특정 Web API가 HTTPS를 전제하는 흐름)는 Servør
같은 소규모 도구조차 인증서 관리를 핵심 기능으로 편입시키게
만들었다.
과거라면 로컬 개발에서 HTTP만으로 충분했던 상황이, 이제는 자체 서명
인증서를 만들고 그것을 시스템 키체인에 등록하는 것까지 개발 서버의
책임 범위로 확장됐다.
이 흐름을 따라가면, 브라우저 보안 정책의 강화가 브라우저 자체의
변화에 그치지 않고 그 하위에 있는 온갖 개발 도구의 기능 목록을
재편하는 2차 효과를 낳고 있다는 것을 알 수 있다.
Servør의 `certify.sh`와 `sudo servor --secure`는 이 큰 흐름이 가장
작은 단위의 도구에까지 어떻게 침투하는지 보여주는 구체적 사례다.
