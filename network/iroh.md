# iroh - 공개키 기반 P2P 네트워크 라이브러리

원문: <https://github.com/n0-computer/iroh>

## 요약

iroh는 Rust로 작성된 모듈형 네트워킹 스택이다.
핵심 슬로건은 “IP addresses break, dial keys instead” — IP 주소는 변하고 깨지므로,
공개키로 직접 다이얼하자는 것이다.
엔드포인트는 IP가 아니라 암호학적 식별자(공개키)로 표현되고, 연결은 위치와
무관하게 유지된다.

주요 기능은 세 가지로 압축된다.
첫째, NAT 홀 펀칭으로 P2P 직결을 우선 시도하고 실패하면 공개 릴레이 서버로
우회하는 하이브리드 라우팅이다.
둘째, QUIC 기반으로 인증된 암호화, 다중화된 스트림, 데이터그램, head-of-line
blocking 없는 전송을 기본값으로 제공한다.
셋째, 공개키 라우팅으로 위치 독립적이고 복원력 있는 연결을 만든다.

iroh는 단일 라이브러리가 아니라 생태계로 제공된다.
콘텐츠 주소화 blob 전송(iroh-blobs), 발행-구독 네트워크(iroh-gossip), 최종 일관성
KV 저장소(iroh-docs), 그리고 새로운 iroh-willow 프로토콜까지 상위 도구가
이미 존재한다.
클라우드 서버부터 태블릿, ESP32·라즈베리파이 같은 임베디드까지 동일한 스택이
동작하며, Rust 외 언어를 위한 FFI 바인딩을 제공한다.
라이선스는 MIT/Apache 2.0 듀얼이다.

GeekNews 본문은 iroh를 “위치와 무관하게 가장 빠른 네트워크 연결을 유지하는”
라이브러리로 요약한다.
P2P 애플리케이션, 실시간 통신, IP 기반의 한계가 마찰을 일으키는 분산 시스템이
주 대상이다.

## 분석

### IP 추상화를 통째로 걷어내는 레이어

TCP/UDP → QUIC로 이어지는 전송 계층 진화는 응용 개발자 측에서 “연결은 스트림이다”
라는 추상화를 강화해왔다.
iroh는 여기에 더해 “엔드포인트는 공개키다”라는 식별 추상화를 더한다.
결과적으로 응용 개발자는 IP·포트·NAT·DNS를 의식하지 않고도 두 프로세스를 연결하는
API만 다루면 된다.
네트워킹의 UX가 파일시스템의 “경로”처럼 단순해지는 방향이다.

### 홀 펀칭 + 릴레이의 재조립

홀 펀칭은 WebRTC 이후 10년 이상 성숙해온 기법이고, 릴레이 폴백은 TURN, STUN
서버 시대부터 익숙하다.
iroh는 이들을 재발명하지 않고 조합하지만, 공개키 기반 정체성을 그 위에 덮어
“연결 가능성의 단일 API”를 만든다.
기존 WebRTC 스택이 브라우저 중심이고 신호 서버를 별도로 요구하는 데 반해, iroh는
CLI·서버·임베디드를 모두 1급 대상으로 삼는다.

### QUIC 선택의 의미

QUIC은 구글에서 시작해 HTTP/3의 기반이 됐지만, 그 진짜 가치는 “커스텀 프로토콜의
기본값”이 될 수 있다는 점이다.
iroh가 QUIC을 채택하면서 애플리케이션은 “암호화, 다중 스트림, 재전송, 혼잡 제어”를
구현하지 않고도 얻는다.
이는 수년 전 “자체 프로토콜을 TCP 위에 짜겠다”는 선택이 왜 나쁜 아이디어였는지
드러낸다.

## 비평

### 강점: 생태계가 함께 움직인다

iroh-blobs, iroh-gossip, iroh-docs처럼 상위 레이어 프로토콜이 이미 존재한다는 점은
새 네트워킹 라이브러리에게 드문 장점이다.
대부분의 “차세대 P2P 스택”은 저수준만 깔끔하고 위가 비어 있어 채택이 멈춘다.
iroh는 저수준과 상위 프로토콜이 같이 진화하는 모델을 택했다.

### 약점: 중앙 의존의 재등장

릴레이 폴백은 “순수 P2P”라는 이상에서 한 발 물러선 타협이다.
공개 릴레이 서버를 누가 운영하고, 비용 구조가 어떻게 되며, 장애 시 네트워크
전체가 어떻게 분단되는지는 문서의 기술 설명으로는 드러나지 않는다.
릴레이는 실질적으로 중앙 의존이므로, 이 레이어의 운영 모델이 iroh 전체의
탈중앙성 주장의 한계를 결정한다.

### 누락된 논의: 공개키 교환의 부트스트랩

“IP 대신 키로 다이얼”은 우아하지만, 상대 키를 처음 어떻게 알았는가의 문제는
남는다.
결국 어떤 채널(웹, QR, 레지스트리)에 의존하며, 그 채널의 신뢰는 TLS-CA나 DNS
같은 기존 시스템에 기대게 된다.
iroh는 이 부트스트랩 문제를 해결하는 대신 그 위에서 작동한다.
“공개키 네이티브”라는 수사는 이 의존을 종종 가리는 효과가 있다.

## 인사이트

### 네트워킹의 주소 공간이 “위치”에서 “정체성”으로 이동한다

인터넷의 50년은 “어디에 있는가(IP)”가 통신의 기본 단위였다.
그러나 모바일, 멀티홈, NAT, 클라우드 마이그레이션이 이 전제를 무너뜨렸다.
같은 장치가 하루에도 수십 번 IP를 바꾸고, 같은 서비스가 수십 개 IP로 동시에 존재한다.
실제 사용자가 관심 있는 질문은 “이 주소로 보내고 싶다”가 아니라 “이 정체성에
보내고 싶다”다.

이 전환은 HTTPS 인증서, Signal의 안전 번호, SSH 키 지문으로 이미 조금씩 일어났다.
iroh는 이 흐름을 전송 계층 자체로 끌어내린다.
결과적으로 “어떤 서버에 접근한다”는 개념이 “어떤 키에 접근한다”로 바뀌면,
DNS·로드 밸런서·IP 임대 같은 산업 전체가 재구성 압력을 받는다.

역사적 비유는 URL에서 콘텐츠 주소화(IPFS, Git)로의 점진적 이동이다.
파일을 “위치”가 아니라 “내용”으로 식별하면 캐싱, 복제, 검증이 본질적으로 달라졌다.
연결을 “위치”가 아니라 “정체성”으로 식별하면 라우팅, 모빌리티, 보안이 본질적으로
달라진다.
iroh의 메시지는 단순한 라이브러리가 아니라 이 관점 전환의 실무적 매개체다.

### “P2P 네이티브” 애플리케이션이 다시 가능해지는 시점

2010년대 중반까지 P2P는 “어려운 문제”였다.
NAT 통과, 신호 서버 운영, 암호화, 스트림 다중화, 데이터 검증을 각 앱이 처음부터
풀어야 했다.
그 결과 “클라우드 + REST API” 조합이 모든 것을 삼켰다.
P2P는 BitTorrent, Signal처럼 특수한 경우에만 살아남았다.

iroh 류의 스택은 이 역사적 균형을 흔들 수 있다.
P2P의 부품이 표준 라이브러리처럼 제공되면 “이 앱은 P2P가 자연스럽다”는 판단이
다시 가능해진다.
로컬 퍼스트 앱, 엣지 협업 도구, CDN 없는 미디어 배포, 디바이스 간 직접 동기화가
클라우드 백엔드 없이 그럴듯해진다.

장기적으로 이는 클라우드 비용 구조에 완만하지만 지속적인 압력을 가한다.
“굳이 중앙 서버에 왕복할 필요가 있는가”라는 질문이 20% 의 앱에서만 제기돼도
인프라 지출은 측정 가능하게 줄어든다.
iroh는 이 질문을 기술적으로 가능한 선택지로 만드는 도구다.

### 임베디드까지 같은 스택을 쓴다는 의미

iroh가 ESP32, Raspberry Pi까지 대상으로 삼는다는 점은 단순 플랫폼 지원
체크박스가 아니다.
이는 “네트워킹 레이어의 통일”이라는 의미를 갖는다.
현재 IoT는 MQTT, CoAP, Zigbee, BLE, 사설 프로토콜이 파편화된 상태다.
각 디바이스 제조사가 클라우드 게이트웨이를 운영하고, 사용자는 그 게이트웨이에
종속된다.

iroh가 임베디드에서 동작한다면, 센서·허브·모바일·서버가 같은 공개키 그래프의
노드가 된다.
제조사 클라우드를 경유하지 않고 홈 네트워크 내부에서 직접 통신하는 것이 기본
설계가 될 수 있다.
이는 사용자 프라이버시, 제조사 종속 해제, 장치 수명 연장에 모두 유리하다.

반대 방향으로는 제조사 비즈니스 모델이 흔들린다.
IoT 하드웨어가 “클라우드 구독을 요구하지 않는 P2P 장치”로 상품화되면,
장치 판매 이후 수익이 구독으로 이어지지 않는다.
iroh 같은 스택의 확산은 기술 선택이면서 동시에 비즈니스 모델 선택을 강제한다.
누군가에게는 해방이고, 누군가에게는 위협이다.
