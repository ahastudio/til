# LLM을 활용한 MikroTik 네트워크 구성

원문: <https://blog.greg.technology/2026/07/14/llm-networking-with-mikrotik.html>

HN 토론: <https://news.ycombinator.com/item?id=48927915> (81점, 36개 댓글)

## 요약

저자 Greg Sadetsky는 네트워킹 애호가로서 수개월에 걸쳐 LLM의 도움을 받아
MikroTik 장비 기반 네트워크를 구축하고 운영한 경험을 정리했다.
단일 라우터에 무선 기능이 통합된 구성에서 라우터, 스위치,
별도 무선 액세스포인트로 분리하는 마이그레이션을 수행했고,
이와는 별개로 완전히 새로운 네트워크도 두 건 구축했다.

저자는 LLM을 “혼란스러운 힘의 배수(confusion multiplier)”로 규정한다.
LLM은 RouterOS 설정에 관한 지식을 갖고 있지만 동시에 환각을 일으키므로,
결과를 신뢰하되 반드시 검증해야 한다는 태도를 유지한다.
이를 위해 저자는 하나의 LLM에만 의존하지 않고
Antigravity, Codex, Opus, Fable 등 여러 LLM에게 같은 설정을 검토시켜
합의에 도달하는 방식을 사용했다.

기술적으로는 LLM이 장비를 다루는 채널 선택이 중요하다고 강조한다.
SSH를 통해 텍스트를 주고받게 하면 “LLM이 텍스트를 SSH로 주고받을 때
발생하는 천 번의 칼에 베이는 듯한 문제(death by a thousand cuts)”가
생기므로, 훨씬 LLM 친화적인 REST/JSON API를 사용하는 편이 낫다고
권장한다.
무선 네트워크 구성에서는 CAPsMAN이 다중 액세스포인트 설정을
크게 단순화해 주며, LLM과 함께 쓰면 구성이 매우 수월하다고
평가했다.
IP 주소가 충돌하는 상황(예: 마이그레이션 중 여러 장비가 동일한
192.168.88.x 대역을 쓰는 경우)에 대비해, 저자는 MAC 계층에서
직접 장비에 접속할 수 있는 MAC-Telnet 도구를 활용했고,
LLM(Claude로 추정)이 이 도구를 더 쓰기 쉽게 만들기 위한
작은 Homebrew formula와 CLI까지 직접 제작했다.

운영 측면에서 저자는 몇 가지 실전 팁을 제시한다.
설정 변경은 최소 단위로 나누어 진행하고 매 변경마다 테스트할 것,
변경 전후로 전체 설정을 덤프하고 버전 관리할 것,
“테스트되지 않은 백업은 0으로 가득 찬 파일일 뿐”이므로 복구 절차를
실제로 검증해 둘 것을 강조한다.
보안을 위해 SSH를 제외한 관리 채널 중 암호화되지 않은 것들,
즉 HTTP, Telnet, FTP는 비활성화할 것을 권한다.
운영 편의를 위해 NTP를 설정하고 장치와 포트에 설명적인 이름을
붙여 둘 것, 마이그레이션 전에는 SSID, 비밀번호, DHCP 예약 항목
등을 미리 기록해 둘 것도 조언한다.

## 분석

### LLM을 신뢰하되 검증하는 이유는 “지식”과 “적용”의 불일치다

저자가 LLM을 “혼란스러운 힘의 배수”로 부른 표현은 정확한 진단이다.
LLM은 RouterOS 문법, CAPsMAN 개념, MAC-Telnet의 존재 같은 지식을
방대하게 갖고 있다.
문제는 그 지식을 특정 장비, 특정 펌웨어 버전, 특정 네트워크 상태에
적용하는 순간 발생한다.
RouterOS는 버전 간 문법이 바뀌는 경우가 있고, 네트워크 상태는
텍스트로 완전히 서술하기 어려운 암묵적 맥락(어떤 포트가 이미
사용 중인지, 어떤 VLAN이 실제로 트래픽을 받는지)을 담고 있다.
LLM은 이 맥락을 추론하거나 지어내는데, 지식이 정확할수록 지어낸
부분이 그럴듯하게 섞여 들어가 발견하기 어려워진다.
이것이 “배수”라는 표현이 정확한 이유다.
지식이 없는 도구라면 사용자가 처음부터 의심하지만, 지식이 있는
도구는 사용자의 의심을 무디게 만들면서 동시에 오류를 만들어낸다.

### SSH 대신 API를 쓰라는 조언은 인터페이스 설계 문제로 귀결된다

“천 번의 칼에 베이는 문제”라는 표현은 SSH를 통한 텍스트 왕복이
왜 문제인지를 잘 짚는다.
SSH 세션에서 LLM은 프롬프트 문자열을 파싱하고, 명령의 출력
형식을 추측하고, 줄바꿈이나 페이징 처리를 오차 없이 다뤄야 한다.
이 각각의 지점에서 작은 실수가 누적되고, 실수 하나하나는
치명적이지 않지만 합쳐지면 작업 전체를 신뢰할 수 없게 만든다.
반면 REST/JSON API는 구조화된 입출력을 제공하므로 LLM이 텍스트를
해석할 필요가 없어지고, 오류가 발생해도 어느 필드에서 무엇이
잘못됐는지 명확하게 드러난다.
이것은 LLM 특유의 문제라기보다 일반적인 자동화 인터페이스 설계
원칙에 가깝다.
사람이 눈으로 읽는 터미널 출력과 기계가 파싱할 구조화된 응답은
애초에 다른 목적을 위해 설계된 것이며, LLM이 네트워크 자동화에
들어오면서 이 오래된 원칙이 다시 부각된 것으로 볼 수 있다.

### CAPsMAN과 MAC-Telnet 선택은 “실패 상황을 먼저 상정하는” 태도를 보여준다

CAPsMAN으로 다중 액세스포인트를 관리하는 것과, IP 충돌 시
MAC-Telnet으로 우회 접속하는 것은 언뜻 별개의 팁처럼 보이지만
공통된 태도를 드러낸다.
CAPsMAN은 정상 시나리오를 단순화하는 도구이고, MAC-Telnet은
비정상 시나리오(IP 계층이 망가진 상태)에 대비하는 도구다.
저자가 이 둘을 나란히 언급한 것은, 네트워크 자동화를 LLM에게
맡길 때 “정상적으로 작동할 때의 편의”뿐 아니라 “설정이 잘못되어
IP 통신 자체가 끊겼을 때 어떻게 접근할 것인가”까지 미리 준비해
두었다는 뜻이다.
LLM이 실수로 IP 설정을 잘못 적용해 네트워크가 끊기는 상황은
충분히 예견 가능한 실패 모드이며, 저자가 이를 위해 별도의
Homebrew formula와 CLI까지 만들었다는 사실은 이 위험을 추상적
경고가 아니라 실제로 겪었거나 강하게 예상했다는 신호로 읽을 수
있다.

## 비평

### 다중 LLM 합의 방식은 검증 비용을 줄이지 않고 이전할 뿐이다

저자는 Antigravity, Codex, Opus, Fable에게 같은 설정을 검토시켜
합의에 이르는 방식을 제안하지만, 이 방법이 실제로 검증 부담을
줄이는지는 의문이다.
여러 LLM이 같은 학습 데이터나 유사한 RouterOS 문서를 참조한다면,
동일한 종류의 환각을 동시에 일으킬 가능성이 있다.
특히 RouterOS 버전별 문법 차이처럼 문서화가 널리 퍼지지 않은
영역에서는, 여러 모델이 같은 구버전 문법으로 “합의”에 이르렀다고
해서 그것이 옳다는 보장이 되지 않는다.
게다가 네 개의 LLM 응답을 비교하고 불일치를 조정하는 작업 자체가
새로운 인지 부담이다.
저자는 이 조정 과정에서 어떤 기준으로 최종 결정을 내렸는지,
불일치가 발생했을 때 무엇을 했는지 전혀 설명하지 않는다.
결과적으로 이 팁은 “검증을 줄이는 방법”이 아니라 “검증의 형태를
바꾸는 방법”에 가까우며, 글은 이 차이를 구분하지 않은 채
효율화로 제시한다.

### 실패 사례의 부재는 조언의 신뢰도를 떨어뜨린다

글 전체가 “이렇게 하라”는 권고로 구성되어 있지만, 정작 저자가
구체적으로 무엇이 잘못됐고 그것을 어떻게 발견했는지는 거의
서술하지 않는다.
“LLM은 환각을 일으킨다”는 문장은 있지만 어떤 설정에서, 어떤
형태로 환각이 나타났는지는 나오지 않는다.
백업과 복구 절차를 강조하면서도 “테스트되지 않은 백업은 0으로
가득 찬 파일”이라는 경구만 남기고, 저자 본인이 실제로 복구를
시도해야 했던 사건이 있었는지는 밝히지 않는다.
실패 사례가 구체적으로 제시되지 않으면 독자는 이 조언들이 실제
사고를 겪고 얻은 교훈인지, 아니면 일반적으로 알려진 모범 사례를
LLM 맥락에 재포장한 것인지 구분할 수 없다.
네트워크 자동화처럼 실수의 대가가 큰 영역에서, 실패의 구체성이
빠진 조언은 설득력이 크게 줄어든다.

HN 댓글 중 하나는 이 공백을 메울 만한 실제 안전장치를 제시한다.
한 개발자는 에이전트가 장비를 직접 조작하게 두지 않고,
“조사는 에이전트가 직접 하도록 하되, 실제 MikroTik 설정은
에이전트가 멱등성 있는(idempotent) 스크립트를 작성하게 하고
그 스크립트를 실행하는” 방식을 쓴다고 설명했다.
그는 이 스크립트가 “의도를 담은 기억(a memory of intent)” 역할을
한다고 표현했다[^arjie].
이는 저자가 강조한 "설정 덤프"보다 한 단계 더 나아간 방법으로,
LLM이 무엇을 왜 바꾸려 했는지를 재실행 가능한 형태로 남겨
검증과 재현을 동시에 확보한다.
글이 구체적 실패 사례 대신 추상적 조언에 머문 반면, 이 댓글은
"LLM의 실수를 어떻게 구조적으로 가두는가"에 대한 실행 가능한
답을 제시한다는 점에서 원문의 공백을 정확히 짚는다.

### API 우선 원칙은 MikroTik 생태계 밖에서 같은 강도로 성립하지 않는다

SSH 대신 REST/JSON API를 쓰라는 조언은 MikroTik이 비교적 잘
정비된 API를 제공하기 때문에 성립하는 조언이다.
저자는 이것을 일반 원칙처럼 서술하지만, 많은 네트워크 장비
벤더는 여전히 CLI/SSH가 유일하거나 가장 완전한 관리 인터페이스이고,
REST API가 있어도 CLI 대비 기능이 제한적인 경우가 흔하다.
이런 환경에서는 “LLM 친화적인 채널을 쓰라”는 조언이
“애초에 그런 채널이 없다”는 현실과 부딪힌다.
글은 이 조언이 MikroTik이라는 특정 생태계의 API 완성도에
의존하고 있다는 점을 명시하지 않은 채, 마치 LLM 네트워킹
전반에 적용되는 원칙처럼 제시한다.
독자가 다른 벤더 장비에 이 조언을 그대로 옮기려 하면, 애초에
전제가 되는 API 인프라가 없어 실행 불가능한 조언이 될 수 있다.
다만 HN 댓글 중 하나는 이 비평의 전제 자체를 흔든다.
MikroTik이 최근 문서 사이트를 Atlassian Confluence 위키에서
Docusaurus로 옮겼고, 어떤 페이지든 URL 끝에 `.md`를 붙이면
마크다운으로 바로 받아올 수 있다는 것이다[^mateja].
이는 우연한 API 완성도가 아니라 벤더가 의도적으로
“LLM이 읽기 좋은 문서”를 준비했다는 신호이며, 글이 놓친
맥락을 보완해 준다.
또 다른 댓글은 RouterOS 자체의 특성이 환각 여지를 원천적으로
줄인다고 지적한다.
MikroTik은 비밀 값을 제외한 전체 설정을 하나의 텍스트 파일로
덤프할 수 있고, CLI 명령어가 버전 간 매우 안정적이어서
“네트워크 개념을 RouterOS 설정으로 옮기는 과정에 군더더기가
거의 없다(pretty fat-free)”고 평가했다[^brihass].
이 댓글에 따르면 남은 문제는 대부분 사용자가 원하는 바를
명확히 specify하지 못하는 데서 오는 것이지, 환각이 끼어들
여지 자체가 크지 않다는 것이다.
한편 다른 댓글은 버전 문법 차이를 더 구체적으로 짚는다.
RouterOS 6과 7 사이에 일부 기능의 문법이 달라지므로 사용 중인
버전을 LLM에게 명시해야 하며, "결과가 꽤 좋지만 반복 작업이
필요하다"고 조언한다[^mannyv].

## 인사이트

### LLM에게 장비 접근권을 주는 것은 IAM(접근 제어) 문제를 새로 만든다

이 글이 직접 다루지 않지만 함의하는 지점은, LLM에게 네트워크
장비의 API 접근권을 부여하는 순간 전통적인 IAM 문제가 새로운
형태로 재등장한다는 것이다.
사람 관리자는 실수를 해도 대개 “지금 무엇을 바꾸고 있는지”에
대한 맥락을 유지하지만, LLM 에이전트는 세션이 끊기거나 컨텍스트가
잘리면 이전에 무엇을 바꿨는지에 대한 기억을 잃을 수 있다.
저자가 강조한 “변경 전후 전체 설정 덤프”는 사실 LLM을 위한
감사 로그(audit log)이자 상태 복원 지점 역할을 겸한다.
이는 일반적인 IaC(Infrastructure as Code) 관행과 유사하지만,
차이는 LLM이 실행하는 변경이 사람이 리뷰하는 풀 리퀘스트 없이
곧바로 장비에 적용될 수 있다는 점이다.
결국 홈 네트워크 규모에서는 가벼워 보이는 이 워크플로가, 기업
네트워크 규모로 확장되면 “누가 무엇을 언제 바꿨는가”를 추적하는
IAM/감사 체계 없이는 감당할 수 없는 위험을 갖게 된다.

### MAC-Telnet의 재조명은 “복구 채널은 주 채널과 물리적으로 분리되어야 한다”는 오래된 원칙의 반복이다

MAC-Telnet은 IP 계층에 의존하지 않는 계층2 접속 수단이라는
점에서, 서버 운영에서 오래전부터 쓰여 온 시리얼 콘솔이나
IPMI/iLO와 같은 대역 외(out-of-band) 관리 채널과 본질적으로
같은 역할을 한다.
LLM이 IP 설정을 잘못 적용해 네트워크가 끊기는 사고는 사람이
같은 실수를 저지르는 것보다 오히려 빈도가 높을 수 있는데,
LLM은 “지금 이 명령을 실행하면 내 접속 자체가 끊길 것”이라는
직관적 위기감을 사람만큼 갖지 못하기 때문이다.
따라서 LLM에게 네트워크 관리를 맡기는 워크플로에서는, 사람이
관리할 때보다 오히려 더 적극적으로 대역 외 복구 채널을 확보해
두어야 한다는 결론이 나온다.
저자가 MAC-Telnet용 도구를 직접 만들어 둔 것은 우연한 편의
기능이 아니라, LLM 자동화가 필연적으로 요구하는 안전망을
미리 갖춘 사례로 볼 수 있다.

### “LLM 친화적 인터페이스”라는 기준은 앞으로 장비 벤더의 설계 우선순위를 바꿀 것이다

저자가 REST/JSON API를 SSH보다 선호하는 이유로 든 것은 결국
“LLM이 파싱하기 쉬운가”라는 기준이다.
이 기준은 지금까지 네트워크 장비 API 설계에서 크게 고려되지
않았던 새로운 요구사항이다.
기존에는 API의 우선순위가 사람 개발자의 편의(문서화, SDK 지원)에
맞춰져 있었지만, LLM 에이전트가 직접 장비를 조작하는 사례가
늘어나면 벤더 입장에서는 “LLM이 오류 없이 호출할 수 있는가”를
API 품질 지표에 포함해야 할 유인이 생긴다.
이는 구조화된 출력, 명확한 오류 메시지, 일관된 필드 명명 같은
요소들이 사람의 가독성을 넘어 에이전트의 신뢰성 지표로
재평가된다는 뜻이다.
MikroTik처럼 비교적 작은 벤더가 이런 흐름에서 먼저 언급된다는
점은, API 완성도가 앞으로 LLM 자동화 생태계에서 벤더 선택의
숨은 기준이 될 수 있음을 시사한다.

### 자격 증명을 외부 LLM에 넘기는 문제는 별도의 위협 모델을 요구한다

글은 LLM에게 장비 접근권을 주는 것을 편의의 문제로만 다루지만,
HN에서는 이를 신뢰의 문제로 보는 반응이 나왔다.
한 댓글은 “네트워크 설정, 키, 자격 증명 같은 것들을 외부
기관 — 그것도 데이터로 먹고사는 기관 — 에 보내는 일이 이제는
‘괜찮은’ 일로 여겨진다는 것이 흥미롭다”고 꼬집었다[^redeemer].
이에 대한 답글은 “개인 키와 설정이 LLM 제공자에게 그대로
흡수된다는 생각 때문에 이런 종류의 직접적인 작업은 꺼려진다.
로컬 LLM에 딱 맞는 용도”라고 덧붙였다[^j4k0bfr].
저자의 글은 REST/JSON API가 SSH보다 "LLM 친화적"이라는
인터페이스 관점에서만 접근을 다루지만, 그 API 호출에 실리는
비밀번호와 WiFi 키, DHCP 예약 정보가 결국 외부 LLM 제공자의
서버를 거쳐 간다는 사실은 언급하지 않는다.
클라우드 LLM에 네트워크 자격 증명을 노출하는 것과 로컬 LLM으로
제한하는 것은 완전히 다른 위협 모델을 요구하며, 글이 이
구분을 아예 다루지 않았다는 점은 가정용 네트워크라는 낮은
위험도 때문에 넘어갈 수 있는 문제가 아니라, 이 워크플로가
회사나 더 민감한 환경으로 확장될 때 반드시 짚어야 할
공백으로 남는다.

### 회의적인 독자는 이 워크플로를 새로운 패러다임이 아니라 도구 연결로 본다

모든 댓글이 우호적인 것은 아니었다.
한 댓글은 원문의 “네트워킹은 그 자체로 복잡하다”는 대목을
정면으로 반박하며, MikroTik의 어려움은 “표준적인 포인트 앤
클릭 엔지니어링 작업”이 아니라 GUI에 구현되지 않은 기능이나
버전 간 설정 해석 차이, 하드웨어 지원 기간, 전반적으로
불안정한 코드에서 온다고 지적했다.
그리고 결론적으로 “이 글은 그냥 ‘LLM을 MikroTik API에 연결해서
재미로 해봤다’는 이야기 아니냐”고 되물었다[^protocolture].
이 반응은 저자의 인사이트가 실제로는 특별한 방법론이라기보다
LLM을 기존 API에 연결한 자연스러운 결과물일 수 있다는
가능성을 제기한다.
글이 제시하는 팁들(다중 LLM 검증, API 우선, 백업 관리)이
독창적인 통찰이라기보다는 어떤 자동화 프로젝트에도 적용되는
일반적인 모범 사례를 네트워킹 맥락에 옮긴 것에 가깝다는
시각이며, 이 글의 가치를 어디에 두어야 할지에 대한 균형 잡힌
평가를 위해 함께 고려할 만하다.
