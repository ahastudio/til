# Graphify: 코드베이스 지식 그래프 도구

> Graphify · the code knowledge graph for AI coding assistants

<https://graphify.com/>

<https://github.com/Graphify-Labs/graphify>

## 소개

Graphify는 코드베이스 전체—소스 코드, 문서, SQL 스키마, 설정 파일, PDF—를 쿼리 가능한
지식 그래프(knowledge graph)로 변환하는 도구다. Claude Code, Cursor, Gemini CLI 등
AI 코딩 어시스턴트에서 `/graphify` 슬래시 커맨드(skill)로 사용하도록 설계되어 있다.

핵심 차별점은 세 가지다. 첫째, 코드 파싱이 로컬에서 완전히 실행된다. tree-sitter AST
기술을 이용해 결정론적으로 코드를 분석하며 외부 서버로 코드를 전송하지 않는다. 둘째,
벡터 임베딩 대신 실제 그래프 구조를 만든다. 의미 유사도 검색이 아니라 직접 순회
가능한 그래프를 생성한다. 셋째, 관계마다 신뢰도 태그를 붙인다. 소스에서 명시적으로
추출한 관계는 `EXTRACTED`, 추론한 관계는 `INFERRED`로 구분해 투명성을 제공한다.

## 주요 기능

**설치와 실행**

```bash
uv tool install graphifyy
graphify install
```

AI 어시스턴트에서 `/graphify .` 명령을 실행하면 세 가지 아웃풋이 생성된다.

- `graph.html` — 인터랙티브 시각화
- `GRAPH_REPORT.md` — 핵심 개념 및 추천 쿼리
- `graph.json` — 쿼리 가능한 전체 그래프

**쿼리 인터페이스**

파일을 다시 읽지 않고 그래프를 쿼리할 수 있다.

```bash
graphify query "what connects auth to database?"
graphify path "ServiceA" "ServiceB"
```

**그래프 분석 기능**

- “갓 노드(god nodes)” 탐지: 가장 많이 연결된 개념 식별
- Leiden 클러스터링을 통한 커뮤니티 감지
- 함수 호출, 임포트 등 파일 간 관계 해석
- 주석과 문서의 설계 의도를 일급 그래프 노드로 추출

**지원 범위**

36개 이상 프로그래밍 언어, 문서, PDF, 이미지, 영상/오디오 트랜스크립트를 처리한다.
Claude Code, Codex, GitHub Copilot CLI, Cursor, Gemini CLI, Aider를 포함한
20개 이상 AI 플랫폼과 통합된다.

플랫폼별 상시 활성 모드 설치도 지원한다.

```bash
graphify claude install
```

이 명령을 실행하면 AI 어시스턴트가 파일 그레핑 대신 그래프 쿼리를 기본으로 선호하게 된다.

**프라이버시**

코드 추출은 API 호출 없이 로컬에서 처리된다. 문서와 미디어 처리는 사용자가 설정한 AI
모델을 사용한다. 텔레메트리나 추적이 없으며, 선택적으로 `~/.cache/graphify-queries.log`에
쿼리 로그를 남길 수 있다.

## 분석

### 벡터 임베딩 vs. 명시적 그래프: 두 패러다임의 차이

Graphify가 내세우는 가장 큰 차별화는 RAG(Retrieval-Augmented Generation) 기반 코드 검색의
주류 방식인 벡터 임베딩을 대체한다는 주장이다. 벡터 임베딩은 의미적 유사도를 기반으로 관련
코드를 검색하는 반면, 명시적 그래프는 실제 의존 관계와 호출 경로를 구조적으로 표현한다.

두 접근의 실용적 차이는 쿼리 유형에 있다. “이 함수와 비슷한 것을 찾아줘”는 임베딩이
자연스럽다. 반면 “인증 모듈에서 데이터베이스까지 실제 호출 경로를 보여줘”나 “이 클래스를
직접·간접으로 사용하는 모든 파일을 찾아줘”는 그래프가 훨씬 정확하다. Graphify가 공략하는
것은 후자의 구조적 질문들이다.

### AI 어시스턴트의 컨텍스트 문제를 해결하는 방식

AI 코딩 어시스턴트의 근본적 병목은 컨텍스트 창 한계다. 코드베이스가 수만 개 파일로 구성될 때,
어시스턴트는 어떤 파일을 컨텍스트에 포함할지 결정해야 한다. 잘못된 선택은 정확도를 급감시킨다.

Graphify의 그래프는 이 선택 문제를 선제적으로 구조화한다. 어시스턴트가 특정 함수나 모듈에
대해 질문받으면, 파일 전체를 읽는 대신 그래프에서 관련 노드와 엣지만 조회해 최소한의 정확한
컨텍스트를 구성할 수 있다. 이는 그레핑 기반 검색과 달리 전체 의존성 맥락을 함께 제공한다.

## 비평

### 'tree-sitter 결정론성'의 실제 한계

Graphify는 tree-sitter AST를 “결정론적 파싱”의 근거로 제시한다. 이는 동적 언어의 런타임
동작을 정적 분석으로 포착할 수 없다는 한계를 유보한다. Python의 동적 임포트, JavaScript의
런타임 의존성 주입, 메타프로그래밍, 런타임 생성 클래스 같은 패턴에서는 AST 기반 파싱이
실제 의존 관계를 놓친다.

이 한계는 현대 코드베이스에서 점점 더 중요해지고 있다. 의존성 주입 프레임워크, 데코레이터
패턴, 런타임 플러그인 시스템은 정적 분석으로 추출할 수 없는 관계들을 만들어낸다. Graphify가
이를 `INFERRED` 태그로 부분적으로 처리하려 하지만, 잘못 추론된 관계가 그래프를 오염시킬 때
신뢰도 태그만으로는 충분하지 않다.

### 설계 의도를 “일급 노드”로 추출하는 주장의 신뢰성 문제

주석과 문서에서 설계 의도(design rationale)를 추출해 그래프 노드로 만든다는 기능은
매력적이지만, 실제로는 이 프로세스가 얼마나 신뢰할 수 있는지 문서에서 검증하지 않는다.
코드 주석은 outdated된 정보를 담고 있는 경우가 흔하며, 의도를 설명하는 것이 아니라 “무엇”을
하는지만 설명하는 주석도 많다.

AI로 주석에서 의도를 추출하는 과정은 AI 모델의 한계를 그대로 상속한다. 특히 도메인 지식이
없으면 주석의 맥락을 잘못 해석할 가능성이 있고, 이렇게 잘못 추출된 의도 노드가 그래프에
포함되면 쿼리 결과에 잘못된 확신을 심어줄 수 있다.

## 인사이트

### 코드 이해의 병목은 검색이 아니라 구조화에 있다

AI 코딩 어시스턴트 시대에 코드 검색은 어느 정도 해결된 문제다. grep, 시맨틱 검색, 파일
트리 탐색은 대부분의 “이 코드가 어디 있는가” 질문에 답한다. 진짜 병목은 “이 코드가 전체
시스템에서 어떤 역할을 하는가”라는 구조적 이해다.

Graphify의 포지셔닝은 이 구분을 명확히 인식하고 있다. 검색 도구가 아니라 구조화 도구로
자신을 정의함으로써, AI 어시스턴트가 파일 단위 검색을 넘어 시스템 전체의 의존성 맥락을
갖추도록 돕는다. 이 방향이 성숙하면 AI 코딩 어시스턴트의 코드베이스 이해 수준이 질적으로
달라질 수 있다. 현재 어시스턴트들이 “이 파일을 읽고 저 파일을 읽고”를 반복하는 방식은
인간이 처음 코드베이스를 파악할 때 하는 것과 동일하다. 지식 그래프는 이 과정을 한 번으로
압축한다.

### 로컬 우선(local-first) 아키텍처가 기업 도입의 열쇠다

벡터 임베딩 기반 코드 검색 도구들이 직면하는 최대 도입 장벽은 코드를 외부 서버로 전송해야
한다는 점이다. 기업 보안 정책, 지적재산 보호, 데이터 거버넌스 요구가 이 모델과 충돌한다.
Graphify의 로컬 우선 아키텍처는 이 장벽을 구조적으로 제거한다.

이는 단순한 기능적 이점이 아니라 기업 영업 전략으로서의 의미가 있다. 보안에 민감한
금융, 의료, 방산, 정부 시장에서 “코드가 로컬을 벗어나지 않는다”는 보장은 의사결정 구조를
단순화한다. GitHub Copilot의 기업 버전도 같은 이유로 프라이빗 배포 옵션을 강조한다.
Graphify가 이 시장에서 성공한다면, 기술적 우수성보다 로컬 처리라는 아키텍처적 선택이
더 큰 역할을 할 가능성이 있다.

### AI 어시스턴트의 “도구 호출 레이어”가 새로운 생태계를 만들고 있다

Graphify가 `/graphify` 슬래시 커맨드로 AI 어시스턴트에 통합되는 방식은 더 큰 추세의
일부다. AI 어시스턴트가 단일 모델에서 도구 오케스트레이터로 진화하면서, 어시스턴트의
역량은 내장된 지식이 아니라 어떤 도구를 호출하는지에 의해 결정된다.

이 생태계에서 “AI 어시스턴트의 슬래시 커맨드로 등록되는 것”이 소프트웨어 배포의 새로운
형태가 되고 있다. 과거에 “IDE 플러그인 마켓플레이스에 등록한다”는 것이 개발 도구 생태계의
진입점이었다면, 지금은 Claude Code, Cursor, Gemini CLI 등의 도구 레이어에 올라타는 것이
그 역할을 대체하고 있다. Graphify가 20개 플랫폼과의 통합을 강조하는 것도 이 흐름을
반영한다. 이 도구 레이어 생태계가 성숙하면, IDE 플러그인 시장과는 다른 구조의 경쟁이
펼쳐질 것이다.
