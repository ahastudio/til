# GigaToken: GB/s 속도의 LLM 토크나이저

<https://github.com/marcelroed/gigatoken>

HN 토론: <https://news.ycombinator.com/item?id=49010167> (604점, 119개 댓글)

## 소개

GigaToken은 언어 모델 학습에 사용되는 토크나이저를 기존 대비 최대 약 1000배 빠르게 실행하는 Rust 구현체다.
HuggingFace Tokenizers와 Tiktoken의 호환 모드를 제공해 기존 코드에서 최소한의 변경으로 사용할 수 있다.
`pip install gigatoken` 한 줄로 설치되며, GPT-2부터 Llama 4, Qwen 3, DeepSeek V3까지 현재 널리 쓰이는 거의 모든 토크나이저를 지원한다.

저자는 Marcel Røed로, GitHub 저장소의 커밋 이력에서 확인할 수 있듯 코드베이스 대부분을 AI 없이 직접 작성했다.
인용 정보에 따르면 프로젝트의 공식 명칭은 “SIMD and Cache Hierarchies for 1000x Faster Byte-Pair Encoding Tokenization on Modern CPUs”이다.

## 주요 기능

### 성능

AMD EPYC 9565(144코어)에서 GPT-2 토크나이저 기준으로 `24.53 GB/s`를 달성하며,
HuggingFace Tokenizers(24.8 MB/s) 대비 989배, Tiktoken(36.0 MB/s) 대비 681배 빠르다.
Apple M4 Max(16코어)에서는 GPT-2 기준으로 HF 대비 1,268배 빠른 `8.79 GB/s`를 기록한다.
AMD Ryzen 7 9800X3D(8코어/16스레드) 같은 소비자 등급 CPU에서도 GPT-2 기준 106배 향상된 `6.27 GB/s`가 나온다.

SentencePiece 기반(Gemma, Mistral, CodeLlama 등) 토크나이저는 속도 이득이 10~21배 수준으로 상대적으로 낮다.
이는 SentencePiece 최적화가 현재 낮은 우선순위로 처리되고 있기 때문이다.

EPYC 144코어 기준으로 Common Crawl(약 1300억 토큰 규모) 전체를 6.5시간 안에 토크나이징할 수 있다는 수치도 제시된다.

### 사용 방법

HuggingFace Tokenizers 호환 모드는 기존 객체를 `gt.Tokenizer(hf_tokenizer).as_hf()`로 감싸는 방식으로 동작한다.
Tiktoken 호환 모드는 `.as_tiktoken()`을 사용한다.
두 호환 모드 모두 출력이 기존 라이브러리와 정확히 일치하도록 검증되어 있으나,
그 정확도 보장 자체가 일부 성능 비용을 수반한다.

가장 빠른 경로는 Gigatoken 고유 API로, Rust 구현체가 파일을 직접 읽어 Python 개입을 최소화한다.

```python
import gigatoken as gt

tokenizer = gt.Tokenizer("Qwen/Qwen3-8B")
file_source = gt.TextFileSource(["owt_train.txt"], separator=b"<|endoftext|>")
tokens = tokenizer.encode_files(file_source)
```

### 기술적 구조

성능의 핵심은 세 가지다.
첫째, 거의 모든 토크나이저가 의존하는 사전 토크나이저(pretokenizer) 정규식을 SIMD 명령어로 교체해 스레드당 2 GB/s 이상을 달성했다.
둘째, 사전 토큰(pretoken) 캐시 계층을 최적화해 이미 본 단어의 인코딩 결과를 효율적으로 재사용한다.
셋째, 스레드 간 통신과 Python 인터페이스 오버헤드를 줄여 병렬화 이익을 극대화한다.

현재 알려진 한계로는 WordPiece 미지원, SentencePiece 최적화 미흡, Python ABI3 사용으로 인한 오버헤드, Windows 환경 미검증이 있다.

## 분석

### 사전 토크나이저가 병목이었다는 발견

기존 토크나이저 라이브러리들이 멀티스레드 Rust로 구현되어 있음에도 불구하고
GigaToken이 수백 배의 속도 향상을 달성할 수 있었던 것은 병목이 BPE 알고리즘 자체가 아니라
그 앞단의 사전 토크나이저(pretokenizer)에 있었기 때문이다.
사전 토크나이저는 텍스트를 단어 단위로 분할하는 과정으로, 대부분의 구현에서 정규식 엔진에 위임된다.
정규식 엔진은 범용성을 위해 설계된 코드이기 때문에, 특정 패턴에만 동작하는 특화 SIMD 구현과 비교하면 필연적으로 느리다.

이 발견은 “잘 최적화된 것처럼 보이는 Rust 코드도 알고리즘 선택 수준의 문제를 숨기고 있을 수 있다”는 점을 보여준다.
HF Tokenizers와 Tiktoken 모두 이미 병렬 Rust 코드를 사용하고 있었음에도 1000배 차이가 났다는 사실이 이 점을 명확히 드러낸다.

저자 marcelroed는 HN에서 GigaToken이 실질적으로 필요한 두 가지 시나리오를 직접 제시했다.[^marcelroed-usecase]
첫째는 사전학습 데이터 혼합 비율이나 필터링 방식을 바꿀 때마다 DCLM 같은 대규모 데이터셋 전체를 재토크나이징해야 하는 경우로, 이 작업만 며칠이 걸리기도 한다.
둘째는 추론 시 긴 시스템 프롬프트처럼 반복 등장하는 접두사를 토크나이징할 때의 첫 번째 토큰까지의 지연 시간(TTFT) 최적화다.

### 캐시 계층 설계의 난이도

저자가 “캐싱은 이 도메인에서 매우 어려운 문제”라고 밝힌 부분은 주목할 만하다.
토큰 빈도 분포는 매우 편향된 형태(long-tailed)를 띠기 때문에
캐시 크기를 작게 유지하면서도 높은 적중률을 달성하려면 정교한 캐시 교체 정책이 필요하다.
GigaToken의 성능 수치가 전체 11.9 GB 파일 기준이라는 점은 이 캐시 설계가 실제로 작동함을 의미한다.

벤치마크 설명에서 HF Tokenizers는 첫 100 MB만, Tiktoken은 첫 1 GB만 측정한 반면
GigaToken은 전체 파일을 처리한다는 점도 이와 관련된다.
파일이 길어질수록 캐시 효과가 누적되므로, 전체 파일 기준 수치는 GigaToken에게 유리하게 작용한다.

### 하드웨어 다양성에 걸친 일관성

AMD EPYC(144코어), Apple M4 Max(16코어 ARM), AMD Ryzen 7(8코어 x86) 세 플랫폼에서 모두 HF 대비 수백 배 이상의 향상을 보인다.
이는 SIMD 최적화가 AVX-512, AVX2, ARM NEON 등 여러 명령어 집합을 대상으로 구현되었기 때문이다.
단일 CPU에만 특화된 최적화였다면 이런 결과가 불가능했을 것이다.

다만 상대적 배율은 코어 수에 따라 크게 달라진다.
144코어 EPYC에서 GPT-2 기준 989배인 반면 8코어 Ryzen에서는 106배다.
이는 GigaToken의 병렬화 이익이 코어 수에 크게 의존함을 시사한다.

### AI 사용 공시의 의미

저자가 README 마지막에 AI 사용 범위를 명시적으로 공시한 것은 주목할 만한 관행이다.
핵심 알고리즘(SIMD 구현, BPE 최적화, 캐시 설계)은 직접 작성했고,
사용자 API, 호환성 확장, SIMD 전략 포팅, 리팩터링에는 AI를 활용했다고 밝힌다.
이 공시는 “AI가 코드를 짰다”와 “사람이 핵심을 짜고 AI가 보조했다”를 구분하고자 하는 저자의 의도를 담고 있다.

## 비평

### 벤치마크 비교 조건의 불균형

GigaToken은 전체 11.9 GB 파일을 처리하는 반면 HF Tokenizers는 100 MB, Tiktoken은 1 GB만 측정한다.
저자는 “캐싱을 하지 않으므로 속도가 균일하다”고 해명하지만, 이 해명은 GigaToken 자신에는 적용되지 않는다.
GigaToken은 사전 토큰 캐시를 핵심 최적화 수단으로 쓰기 때문에, 파일이 길수록 캐시 적중률이 높아지고 측정 수치가 좋아진다.
즉, GigaToken은 전체 파일에서 캐시 효과를 누리는 조건으로 측정되고, 경쟁자들은 그 효과가 발현되기 전 구간만 측정된다.
이 불균형이 1000배라는 수치의 일부를 설명할 가능성이 있다.

### 토크나이저 종류별 편차가 결론의 일반성을 제한한다

BPE 기반 토크나이저(GPT-2, Llama, Qwen 등)에서는 수백~1000배 향상이 나오지만,
SentencePiece 기반(Gemma, Mistral, CodeLlama 등)에서는 7~21배 수준에 머문다.
저자는 이를 “SentencePiece 최적화는 우선순위가 낮다”고 설명하지만,
SentencePiece는 Google 계열(Gemma, T5, mT5)과 일부 오픈소스 모델이 광범위하게 사용하는 표준이다.
GigaToken이 “거의 모든 토크나이저를 지원한다”고 주장하면서도
그 범주에서 성능이 한 자릿수에서 두 자릿수 배율로 떨어지는 토크나이저 군이 존재한다는 점은
“1000x faster”라는 타이틀이 전체 사용 사례에 걸쳐 성립하지 않음을 의미한다.

### 실제 병목이 토크나이저인 파이프라인에서만 의미가 있다

데이터 전처리 파이프라인에서 토크나이저가 실제로 병목인 경우에만 GigaToken의 속도가 체감된다.
대규모 데이터셋을 수집·정제·필터링하는 파이프라인에서 토크나이저 이전 단계(HTML 파싱, 중복 제거, 품질 필터링)가
더 느린 경우에는 토크나이저를 1000배 빠르게 해도 전체 처리 속도는 크게 달라지지 않는다.
저자가 Common Crawl 전체를 6.5시간에 처리할 수 있다는 수치를 제시하지만,
실제로 Common Crawl을 처리하는 파이프라인에서 토크나이저가 마지막 병목이 되는 조직은 많지 않을 것이다.

onlyrealcuzzo는 "토크나이저는 전체 추론 시간의 0.1% 미만"이라고 지적했다.[^onlyrealcuzzo]
이에 대해 저자는 8B Qwen3 모델 실측 데이터를 제시하며 2K~32K 토큰 입력 기준으로 TTFT가 3~10% 단축됐다고 반박했다.[^marcelroed-ttft]
두 주장 모두 맞다는 점이 핵심이다 — 배치 추론처럼 토크나이저가 한 번만 쓰이는 경우에는 영향이 미미하지만,
긴 프롬프트를 실시간으로 처리하거나 스트리밍 서비스에서 TTFT가 사용자 경험에 직결되는 경우에는 체감 가능한 차이가 난다.
어디에 병목이 있는지 측정하지 않고 일반화하면 이 도구의 적용 가능성을 잘못 판단하게 된다.

luciana1u는 "전체 런타임의 0.1%에 해당하는 부분을 1000배 빠르게 만드는 것이야말로 가장 소프트웨어 개발자다운 행동"이라고 꼬집었다.[^luciana1u]
이 비꼬는 댓글은 아이러니하게도 성능 최적화 문화의 근본적인 긴장을 드러낸다 —
측정에 기반한 최적화가 원칙이지만, 현실에서는 병목이 아닌 곳을 정교하게 최적화하는 데 엄청난 노력이 투입되기도 한다.

### Windows 미지원은 실용적 제약이다

README는 “Windows는 테스트가 많이 되지 않아 WSL을 권장한다”고 밝힌다.
이는 Windows 환경에서 데이터 처리 파이프라인을 운영하는 팀에게는 실질적 장애물이다.
GigaToken의 목표 사용자(대규모 언어 모델 사전학습 데이터를 처리하는 팀)가 주로 Linux 서버를 사용한다는 가정 하에서는 이 제약이 크지 않지만,
연구자가 Windows 개발 환경에서 사용하려는 경우나 Windows Server 기반 인프라에서는 진입 장벽이 된다.

## 인사이트

### 토크나이저 속도 향상은 데이터셋 다양성을 늘릴 수 있다

토크나이저가 느릴 때는 처리 가능한 데이터 양에 시간 예산이 제약을 건다.
그 결과 데이터 큐레이션 팀은 처리 속도를 기준으로 데이터 소스를 선택하거나,
파이프라인의 후반부인 토크나이저 단계에 맞춰 앞단의 수집·필터링 단계를 조절하게 된다.
GigaToken처럼 토크나이저가 병목에서 제거되면, 이전에는 시간 예산 때문에 포기했던 다양한 소스를 포함시킬 여지가 생긴다.
이것은 단순히 “같은 데이터를 더 빠르게 처리한다”는 수준을 넘어
“어떤 데이터를 학습 집합에 넣을지” 결정하는 기준 자체를 바꿀 수 있는 변화다.

동시에 이 변화가 반드시 긍정적이지 않을 수도 있다.
토크나이저가 느리다는 제약이 사실상 품질 필터 역할을 해 온 경우라면,
그 제약이 사라졌을 때 더 많은 저품질 데이터가 학습에 포함될 가능성도 있다.
속도 향상이 데이터 품질 관리 인프라와 함께 발전하지 않으면
오히려 모델 품질에 역효과를 낼 수 있다는 점을 간과하기 쉽다.

apollopower는 GigaToken이 "추론이 아닌 오프라인 사전학습 데이터 준비에 더 유용하다"고 정확히 지적했다.[^apollopower]
대규모 학습 데이터셋을 반복적으로 재구성할 때마다 토크나이징을 다시 해야 하는 경우,
이 속도 향상은 데이터 실험 주기를 며칠에서 시간 단위로 단축시킬 수 있다.
janalsncm은 학습 데이터를 토크나이징하는 데 10~15분을 소비한 뒤 사소한 버그로 실행이 실패했던 경험을 공유했다.[^janalsncm]
이 사례는 토크나이저 속도가 단순한 처리량 문제가 아니라 개발자의 실험 피드백 루프 속도와 직결됨을 보여준다 —
토크나이징이 빠르면 작은 배치로 먼저 검증하는 습관을 들이기도 훨씬 쉬워진다.

### SIMD 특화 구현의 부활이 LLM 인프라 전반에서 반복될 것이다

GigaToken의 접근 방식 — 정규식 엔진이라는 범용 추상화를 특화 SIMD 구현으로 교체 — 은
LLM 인프라의 다른 구성 요소에서도 반복될 수 있는 패턴이다.
데이터 파이프라인의 JSON 파싱, 텍스트 정규화, 언어 감지, 중복 제거 등
현재 범용 라이브러리에 위임된 많은 작업이 잠재적인 SIMD 특화 최적화의 후보다.
실제로 simdjson(JSON 파싱), Highway(범용 SIMD 추상화) 같은 프로젝트들이
비슷한 논리로 성능 향상을 달성해 왔다.

LLM 사전학습 규모가 커질수록 이런 인프라 수준의 최적화가 모델 품질에 미치는 간접적 영향도 커진다.
같은 컴퓨팅 예산 안에서 더 많은 토큰을 학습 데이터로 처리할 수 있다면,
이는 모델 아키텍처 개선만큼이나 실질적인 경쟁 우위가 될 수 있다.

swiftcoder는 "인퍼런스 파이프라인의 다른 부분에도 1000배 최적화 기회가 남아 있는 것은 아닐까?"라고 물었다.[^swiftcoder]
이 질문은 단순한 수사적 질문이 아니다.
토크나이저가 멀티스레드 Rust임에도 범용 정규식 엔진 때문에 1000배 느렸다면,
같은 논리가 현재 "충분히 빠르다"고 여겨지는 다른 파이프라인 구성 요소에도 적용될 수 있다.
scottcha처럼 AI 플랫폼을 운영하며 라우팅·속도 제한 결정을 위해 수신 즉시 빠른 토크나이징이 필요한 경우[^scottcha]도,
이런 숨겨진 최적화 기회를 발굴하는 것이 시스템 성능의 핵심 경쟁 요소가 된다.

### 호환 레이어 전략은 채택 속도와 성능 사이의 절충을 드러낸다

GigaToken은 HuggingFace Tokenizers 및 Tiktoken 호환 모드를 제공해 최소한의 코드 변경으로 사용할 수 있게 한다.
이 전략은 채택 문턱을 낮추는 데 효과적이지만, 동시에 “호환 모드는 기대보다 빠르지 않을 수 있다”는 단서를 달고 있다.
기존 인터페이스를 유지하면서 하위 구현만 교체하면 인터페이스 경계에서 발생하는 오버헤드를 피하기 어렵기 때문이다.

이 패턴은 새로운 고성능 구현체가 기존 생태계에 진입하는 일반적인 방식을 보여준다.
완전한 API 교체를 요구하면 채택이 느리고, 호환 레이어를 먼저 제공하면 빠르게 퍼지지만
최대 성능을 원하는 사용자는 결국 새 API로 이전해야 한다.
NumPy의 다양한 드롭인 대체재들(CuPy, JAX의 numpy 호환 레이어 등)이 걸어온 길과 같은 궤적이다.
GigaToken이 장기적으로 생태계 표준이 되려면 호환 레이어를 입구로 삼되
고유 API의 생산성 우위를 설득력 있게 보여주는 문서와 도구를 갖춰야 할 것이다.

### “AI 사용 공시”는 오픈소스 품질 신호의 새로운 차원이 될 수 있다

저자가 어떤 부분에 AI를 사용했는지 명시한 것은 오픈소스 커뮤니티에서 아직 표준화되지 않은 관행이다.
그러나 이 공시가 담고 있는 정보 — “핵심 알고리즘은 사람이 설계했고, 반복적인 확장 작업에 AI가 쓰였다” — 는
코드 신뢰도를 평가하는 데 유의미한 신호다.
AI가 생성한 코드는 버그 패턴이나 취약점 분포가 인간이 작성한 코드와 다를 수 있으며,
리뷰어나 기여자 입장에서 어느 부분을 더 꼼꼼히 살펴야 하는지 우선순위를 잡는 데 도움이 된다.

향후 오픈소스 프로젝트에서 `CONTRIBUTING.md`나 `LICENSE`처럼
AI 사용 범위를 명시하는 표준 파일 형식이 등장할 가능성이 있다.
GigaToken의 공시 방식은 아직 비형식적이지만,
이런 관행이 커뮤니티 표준으로 자리잡는다면 오픈소스 소프트웨어의 품질 검증 방식을 바꿀 수 있다.
