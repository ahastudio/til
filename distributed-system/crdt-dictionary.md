# CRDT 사전 — 충돌 없는 복제 자료구조 완전 가이드

원문: [The CRDT Dictionary: A Field Guide to Conflict-Free Replicated Data Types](https://web.archive.org/web/20260331135836/https://iankduncan.com/engineering/2025-11-27-crdt-dictionary/)

## 요약

Ian Duncan이 작성한 CRDT(Conflict-free Replicated Data Types) 가이드다.
2014년 Riak를 통해 처음 접한 CRDT를 10년 후 재방문하며 실용적 참고서로 정리했다.
개념 복습부터 프로덕션 구현 고려 사항까지 폭넓게 다룬다.

CRDT가 동작하는 수학적 토대는 세 가지 성질이다.
**교환 법칙(commutative)** — 순서가 달라도 같은 결과,
**결합 법칙(associative)** — 묶는 방식이 달라도 같은 결과,
**멱등성(idempotent)** — 중복 적용해도 같은 결과.
이 세 가지가 성립하면 복제본들은 조율 없이도 수렴(convergence)을 보장받는다.

이 성질을 구조적으로 뒷받침하는 것이 **격자(lattice)** 이론이다.
자료구조의 상태가 격자를 이루고 연산이 항상 "위쪽"으로 이동한다면,
병합은 단순히 상한(join)을 취하는 것으로 귀결된다.
어떤 설계가 CRDT로 성립하고 어떤 설계가 실패하는지를 이 격자 관점으로 설명한다.

전달 방식에 따라 **상태 기반(state-based)** 과 **연산 기반(operation-based)** 으로 나뉜다.
상태 기반은 전체 상태를 전송해 `join`으로 병합하고,
연산 기반은 교환 가능한 연산만 전송한다.
이 가이드는 명확성을 위해 주로 상태 기반 접근을 다룬다.

## 분석

### CRDT 유형 카탈로그

**카운터 계열**:

| 유형       | 설명                                                          |
| ---------- | ------------------------------------------------------------- |
| G-Counter  | 복제본별 카운트를 추적. 병합 시 각 복제본의 최댓값을 취함     |
| PN-Counter | G-Counter 두 개(증가/감소)로 음수 지원. 두 카운터의 차이가 값 |

G-Counter는 가장 단순한 CRDT다.
노드 N개가 있을 때 각 노드는 자신의 카운터 슬롯만 증가시키고,
전체 합계는 모든 슬롯의 합으로 계산한다.
병합은 슬롯별로 `max`를 취하면 끝이다 — 단조 증가이므로 격자 조건을 자연스럽게 만족한다.

**집합 계열**:

| 유형            | 설명                                                                        |
| --------------- | --------------------------------------------------------------------------- |
| G-Set           | 추가만 가능. 병합은 집합 합집합                                             |
| 2P-Set          | 추가/제거 집합을 분리해 삭제 지원. 한번 삭제된 원소는 재추가 불가           |
| LWW-Element-Set | 타임스탬프 기반으로 재추가 가능. 클럭 왜곡 시 데이터 손실 위험              |
| OR-Set          | 추가마다 고유 태그 부여. 제거는 관찰된 태그만 삭제해 "add-wins" 의미론 보장 |

2P-Set에서 원소를 제거하면 제거 집합(tombstone set)에 영구 기록된다.
같은 원소를 다시 추가하려면 제거 집합에 이미 있으므로 불가능하다.
이 제약을 없애기 위해 LWW-Element-Set은 타임스탬프로 "가장 최근 상태"를 결정하고,
OR-Set은 추가 태그(unique tag)를 도입해 "어떤 버전이 살아 있는가"를 명시적으로 추적한다.

**레지스터(단일 값) 계열**:

- **LWW-Register**: 타임스탬프와 쌍을 이루며 병합 시 가장 높은 타임스탬프 승리.
- **MV-Register**: 동시 쓰기를 값의 집합으로 보존해 애플리케이션이 충돌을 직접 해결.

MV-Register는 충돌을 "누가 이겼는가"가 아니라 "무엇이 충돌했는가"로 노출한다.
Amazon DynamoDB의 버전 벡터 기반 충돌 노출이 이 패턴을 따른다.
애플리케이션이 충돌 해결 로직을 직접 구현해야 한다는 부담이 있지만,
그 대신 정보 손실 없이 사용자에게 선택권을 줄 수 있다.

**복합 구조**:

- **OR-Map**: OR-Set 원칙을 키-값 저장소에 적용한 CRDT.
- **시퀀스 CRDT**: 순서 있는 문자열/리스트 편집을 위한 세 가지 접근.
  - **RGA(Replicated Growable Array)**: 원소 간 부모-자식 관계 활용.
  - **WOOT**: 양방향 이웃 제약 사용.
  - **Logoot/LSEQ**: 밀도 있는 순서를 이루는 위치 식별자 할당.

시퀀스 CRDT는 협업 텍스트 편집기의 핵심이다.
"같은 위치에 두 사용자가 동시에 삽입하면 어떤 순서로 놓는가"라는 문제를
각 알고리즘이 서로 다른 방식으로 해결한다.
RGA는 삽입 시점의 인과 관계를 기록해 결정론적 순서를 만들고,
Logoot/LSEQ는 위치 자체에 전역 순서를 부여해 충돌을 구조적으로 회피한다.

### 가비지 컬렉션 문제

가이드에서 가장 비중 있게 다루는 실전 문제가 메타데이터 누적이다.
"CRDT는 수렴을 위해 단조적으로 정보를 축적하지만, 프로덕션 시스템은 영원히
무한정 증가할 수 없다."

OR-Set이 원소를 삭제하면 tombstone 태그가 남는다.
PN-Counter는 복제본 수만큼의 슬롯을 유지한다.
시퀀스 CRDT는 삭제된 문자의 위치 메타데이터를 보존한다.
이 메타데이터가 쌓이면 병합 비용이 증가하고 메모리 압박이 생긴다.

Duncan이 제시하는 다섯 가지 전략:

1. **시간 기반 만료**: 임계값보다 오래된 메타데이터 폐기
2. **조율된 GC**: 합의(consensus)로 안전하게 버릴 수 있는 항목 식별
3. **버전 벡터**: 인과 이력 추적으로 정밀하게 폐기 대상 식별
4. **범위 제한 구조**: 메타데이터 한도를 두고 초과 시 LWW 의미론으로 대체
5. **체크포인팅**: 스냅샷 생성 후 이전 이력 폐기

각 전략은 안전성, 조율 오버헤드, 공간 효율 사이의 트레이드오프를 수반한다.
시간 기반 만료는 구현이 단순하지만 아직 전파되지 않은 업데이트를 폐기할 위험이 있다.
조율된 GC는 안전하지만 CRDT가 피하려는 바로 그 조율을 다시 도입한다.

### 성능 특성과 선택 기준

가이드는 공간 복잡도, 연산 비용, 병합 성능을 유형별로 비교한 표를 제공한다.
핵심 통찰: "카운터 CRDT는 연산 횟수가 아니라 복제본 수에 따라 확장된다."

G-Counter의 공간은 O(N)으로 복제본 수에 비례한다.
OR-Set은 삽입된 원소마다 태그를 유지하므로 O(원소 수 × 평균 동시 추가 횟수)다.
시퀀스 CRDT는 전체 편집 이력을 보존하므로 문서가 길고 편집이 많을수록 비용이 커진다.

LWW 기반 CRDT는 클럭 동기화가 필수다.
복제본 A의 클럭이 B보다 앞서 있으면, A의 연산이 항상 B를 이긴다.
하이브리드 논리 클럭(Hybrid Logical Clock)과 복제본 ID 타이브레이커로 대응한다.

선택 기준 결정 트리:

- 추가만 필요한가 → 카운터 또는 G-Set
- 삭제가 필요하고 재추가 불필요 → 2P-Set
- 마지막 쓰기 승리를 허용 → LWW 계열
- 동시 연산 보존이 필요 → OR 계열
- 순서 있는 편집이 필요 → 시퀀스 CRDT (RGA / Logoot)
- 충돌을 애플리케이션이 해결 → MV-Register

### 프로덕션 구현 사례

가이드가 언급하는 주요 시스템:

- **Automerge**: JSON CRDT. Rust로 구현된 코어를 WebAssembly로 컴파일해 브라우저에서도 동작한다.
  문서 전체를 CRDT로 모델링해 중첩 구조의 부분 업데이트를 지원한다.
- **Yjs**: 텍스트 편집에 최적화된 CRDT 라이브러리.
  CodeMirror, ProseMirror, Monaco Editor 등 주요 에디터와 통합이 잘 되어 있다.
  내부적으로 B-tree 구조로 위치를 색인해 대형 문서에서도 성능을 유지한다.
- **Riak**: 분산 키-값 데이터베이스.
  CRDT를 데이터 타입으로 직접 노출한 최초의 상용 분산 데이터베이스 중 하나다.
  2013년 Riak 2.0에서 G-Counter, PN-Counter, Set, Map을 내장 타입으로 제공했다.

## 비평

### 강점 — 실용적 의사결정 도구로서의 설계

이 가이드의 가장 큰 강점은 "무엇이 있는가"에서 "어떤 것을 언제 쓰는가"까지
이어지는 실용적 흐름이다.
CRDT 유형 카탈로그, 복잡도 비교표, 선택 기준 결정 트리를 한 문서에서 제공해
참고서로 기능한다.
추상적 수학 설명에서 멈추지 않고 가비지 컬렉션, 클럭 동기화 같은 프로덕션
문제까지 다룬다는 점이 이론서와 차별화된다.

특히 10년간 프로덕션 경험을 바탕으로 쓴 "재방문"이라는 구조가 신뢰를 높인다.
처음 접할 때 놓쳤던 것, 실제로 부딪힌 것을 솔직하게 기술한 글은
학술 논문이나 공식 문서에서 얻기 어려운 종류의 지식을 전달한다.

### 약점 — 연산 기반 CRDT의 상대적 빈약

상태 기반을 주로 다루겠다고 명시하지만, 실제 프로덕션 환경에서 대역폭과
지연 비용이 중요한 시스템에서는 연산 기반 접근이 더 실용적인 경우가 많다.
Yjs가 연산 기반(operation-based) 알고리즘으로 높은 성능을 달성하는 것처럼,
두 접근의 트레이드오프를 비교하는 심층 분석이 아쉽다.

상태 기반은 전체 상태를 전송하므로 네트워크 비용이 크지만 구현이 단순하다.
연산 기반은 델타만 전송하므로 효율적이지만 "모든 연산이 정확히 한 번
전달된다"는 전달 보장이 필요하다.
실제로 어떤 전달 보장을 갖춘 인프라가 필요한지,
그 인프라를 구축하는 비용은 얼마인지에 대한 논의가 빠져 있다.

### 빈 자리 — 충돌 해결 의미론의 비교

OR-Set의 "add-wins" 와 LWW의 "last-write-wins"는 단순한 구현 차이가 아니라
**의미론적 선택**이다.
어떤 애플리케이션 도메인에서 어떤 의미론이 사용자 기대와 일치하는지,
잘못된 의미론 선택이 어떤 사용자 경험 문제를 낳는지에 대한 논의가
빠져 있다.

예를 들어 협업 할 일 목록에서 두 사용자가 동시에 같은 항목을 삭제하고
한 명이 복구하면 OR-Set은 "add-wins"이므로 복구된 상태로 수렴한다.
이것이 사용자의 기대와 맞는가?
어떤 맥락에서는 맞고 어떤 맥락에서는 "내가 삭제했는데 왜 다시 생겼지?"라는
혼란이 된다.
기술 선택이 사용자 경험으로 연결되는 고리가 이 가이드에서 유일하게 약한
부분이다.

## 인사이트

### CRDT는 "조율 없는 자유"가 아니라 "제약의 이동"이다

CRDT의 핵심 약속은 "중앙 서버 없이 모든 복제본이 수렴한다"는 것이다.
이 약속을 매력적으로 느끼는 이유는 분산 시스템에서 조율(coordination)이 갖는
비용 — 지연, 가용성 감소, 복잡도 — 때문이다.
그런데 CRDT를 깊이 파면, 조율이 사라진 것이 아니라 **어디서 할 것인가가
달라진 것**임이 드러난다.

중앙 서버 조율 대신 CRDT는 두 가지 비용을 요구한다.
첫째, 메타데이터 증가라는 공간 비용 — OR-Set이 삭제 이력을 보존하고,
PN-Counter가 복제본별 카운터를 유지하는 것처럼.
둘째, 가비지 컬렉션이라는 주기적 조율 비용 — 아이러니하게도 GC를 안전하게
수행하려면 결국 어느 시점에 합의(consensus)가 필요하다.

조율을 완전히 제거하는 것은 불가능하다.
CAP 정리가 말하는 것처럼, 분산 시스템에서 분할(partition) 상황에 가용성을
유지하면 일관성을 희생해야 한다.
CRDT는 일관성을 "즉각적 일관성"에서 "최종적 일관성"으로 바꾸는 트레이드오프를
설계에 내재화한 것이다.
이 이동의 비용을 인식하지 못하면 CRDT를 도입한 후 GC 부채에 직면하게 된다.

### LWW의 "마지막이 이긴다"는 의미론은 실제로 누가 이길지 예측 불가능하다

LWW(Last-Write-Wins) 계열 CRDT는 직관적으로 보인다.
"가장 최근 쓰기가 승리한다" — 합리적이고 단순하다.
그러나 "가장 최근"을 결정하는 클럭이 분산 시스템에서 신뢰할 수 없다는 문제가
있다.

물리 클럭(physical clock)은 NTP 동기화 오차, 클럭 드리프트, 의도적 조작 때문에
복제본 간 비교 불가능한 값을 산출할 수 있다.
복제본 A의 클럭이 100ms 앞서 있으면, A의 모든 쓰기가 B의 동시 쓰기를 덮어쓴다.
사용자 입장에서는 "내가 분명히 이후에 쓴 값이 사라졌다"는 경험이 된다.

하이브리드 논리 클럭(HLC)이 이 문제를 완화하지만 완전히 해결하지는 않는다.
HLC는 물리 시간을 논리 클럭으로 보완해 인과 관계를 추적하지만,
동시 발생(concurrent) 이벤트 간의 순서는 여전히 임의로 결정된다.
더 근본적으로는, LWW가 적합한 도메인인가를 먼저 물어야 한다.
단순한 사용자 프로필 필드처럼 "최근 값이 맞다"고 할 수 있는 도메인에서는
LWW가 적절하다.
그러나 재고 수량, 계좌 잔액처럼 모든 연산이 중요한 도메인에서는 LWW가
데이터 손실을 의미론적으로 정당화하는 구조가 될 수 있다.

### 협업 편집 CRDT의 역사는 "옳은 알고리즘"보다 "충분한 알고리즘"의 승리 역사다

시퀀스 CRDT — RGA, WOOT, Logoot, LSEQ — 는 각자 다른 방식으로 "동시에 같은
위치에 삽입하면 어떻게 되는가"를 해결한다.
이 문제를 해결한 최초의 실용적 시스템은 1998년 OT(Operational Transformation)
기반의 Google Wave였다.
OT는 구현이 어렵고 버그가 많아 악명이 높았지만, 충분히 동작했다.

Yjs가 대중화된 이유는 이론적으로 가장 정교해서가 아니다.
**JavaScript 생태계에서 쉽게 쓸 수 있는 라이브러리**로 먼저 나왔기 때문이다.
Automerge는 Yjs보다 표현력이 풍부한 JSON CRDT를 제공하지만,
실제 채택 기준에서는 생태계 통합과 문서화가 알고리즘 우수성보다 더 중요하게
작용했다.

기술 선택에서 "최선의 알고리즘"이 이기는 경우는 드물다.
CRDT 영역도 예외가 아니다.
설계자들이 수년간 RGA vs WOOT의 학문적 우열을 논쟁하는 동안, 시장은
"지금 쓸 수 있는 것"으로 수렴했다.
이것은 CRDT에만 국한되지 않고, 기술 표준 전반에서 반복되는 패턴이다 —
Betamax vs VHS, TCP vs 대안들, Git vs Mercurial.

### 가비지 컬렉션 문제는 CRDT의 약점이 아니라 분산 시스템 일반의 불변성이다

CRDT 가이드들이 흔히 가비지 컬렉션을 "CRDT의 단점"으로 제시한다.
그러나 이 시각은 잘못 설정된 비교 기준에서 나온다.
중앙화된 데이터베이스에서는 쓰기 스레드가 이전 값을 즉시 덮어쓰므로 메타데이터가
축적되지 않는 것처럼 보인다.
그러나 실제로는 트랜잭션 로그, MVCC(Multi-Version Concurrency Control) 버전,
WAL(Write-Ahead Log) 등 다른 형태의 이력이 축적되고 있다.

차이는 **누가 GC를 책임지는가**다.
중앙화된 시스템에서는 DBMS가 내부적으로 처리하므로 개발자에게 노출되지 않는다.
CRDT에서는 분산된 복제본들이 각자 이력을 관리하므로 개발자가 직면하게 된다.

이 차이를 인식하면 "CRDT는 GC 문제 때문에 사용하기 어렵다"는 판단이
"중앙화된 시스템의 GC는 추상화 뒤에 숨어 있다"는 사실을 외면한 것임을 알 수
있다.
프로덕션에서 CRDT를 평가할 때는 GC 전략의 비용을 명시적으로 측정해야 한다.
동시에, 기존 시스템의 숨겨진 GC 비용 — WAL 크기, MVCC 부채, 인덱스 단편화 —
도 함께 측정해 공정한 비교를 해야 한다.

### 오프라인 우선(offline-first) 아키텍처에서 CRDT는 선택이 아니라 필수다

네트워크가 항상 연결되어 있다고 가정하는 시스템에서는 중앙 서버가 충돌을
중재한다.
그러나 오프라인 우선 애플리케이션 — 모바일 앱, 로컬 우선(local-first) 소프트웨어,
P2P 협업 도구 — 에서는 중앙 서버를 거치지 않고 복제본이 독립적으로 진행한 후
재연결 시 병합해야 한다.

이 환경에서 일반적인 낙관적 잠금(optimistic locking)이나 마지막 쓰기 승리는
사용자 데이터를 조용히 날려버린다.
CRDT는 "오프라인 중에 무엇이 일어났는지 전부 보존하고 수렴을 보장한다"는
속성이 이 시나리오와 정확히 맞아떨어진다.

Martin Kleppmann의 "local-first software" 논문이 이 방향을 체계화했고,
Ink & Switch의 프로토타입들이 실제 사용 가능성을 검증했다.
CRDT가 학술 주제에서 실용 도구로 이동한 것은 클라우드 중심 아키텍처에 대한
반성과 맞닿아 있다.
데이터가 사용자 기기에 있고 소유권이 사용자에게 있어야 한다는 철학이
CRDT 채택의 저변에 있다.
