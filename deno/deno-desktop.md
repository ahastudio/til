# Deno Desktop

<https://docs.deno.com/runtime/desktop/>

HN 토론: <https://news.ycombinator.com/item?id=48626137> (1046점, 375개 댓글)

Lobste.rs 토론: <https://lobste.rs/s/0noyze/deno_desktop_apps>

## 소개

Deno Desktop은 TypeScript/Deno 프로젝트를 자립형(self-contained) 데스크톱 애플리케이션으로 변환하는 기능이다.
단일 TypeScript 파일부터 Next.js 앱까지, 기존 Deno 프로젝트를 그대로 데스크톱 바이너리로 묶어낸다.
Deno v2.9.0에서 최초 출시됐으며 현재는 canary 빌드(`deno upgrade canary`)를 통해 테스트할 수 있다.
CLI 문법, 설정 옵션, TypeScript API는 아직 불안정 상태로 변경될 수 있다.

## 주요 기능

### CLI

```bash
deno desktop main.ts
```

위 명령 한 줄로 컴파일이 완료된다.
macOS/Linux에서는 `./main`, Windows에서는 `.\main.exe`가 생성된다.

### 백엔드 옵션

세 가지 렌더링 백엔드를 지원한다.

- **WebView** (기본값): OS 네이티브 웹뷰를 사용해 바이너리 크기를 최소화한다.
- **CEF** (Chromium Embedded Framework): Chromium을 번들링해 플랫폼 간 일관된 렌더링을 보장한다.
- **Raw**: 대안적 옵션.

백엔드는 빌드 시 로컬에서 컴파일하지 않고 필요할 때 다운로드된다.

### 프레임워크 자동 감지

다음 프레임워크를 자동 인식해 설정한다:
Next.js, Astro, Fresh, Remix, Nuxt, SvelteKit, SolidStart, TanStack Start, Vite SSR.
`--hmr` 플래그로 개발 서버의 핫 리로드(HMR)도 지원한다.

### 통신 모델

백엔드와 UI의 통신은 소켓 기반 IPC가 아닌 **인-프로세스 채널(in-process channel)**을 통한다.
호출 경계에서 값이 인코딩되고 Deno 코드와 웹뷰 사이에 크로스 프로세스 왕복이 없다.
`bindings.<name>()` 형태로 웹뷰에서 Deno 코드를 직접 호출한다.

### 핵심 API

- `Deno.serve()`: 포트를 수동으로 지정할 필요 없이 자동으로 웹뷰 주소에 바인딩된다.
- `Deno.BrowserWindow`: 윈도우 생명주기, 다중 윈도우, 이벤트 처리를 제어한다.
- `Deno.autoUpdate()`: 바이너리 diff 패치, 자동 롤백 기능을 포함한 자동 업데이트를 제공한다.

### 네이티브 기능

- **알림**: 웹 표준 `Notification` API로 OS 네이티브 알림을 발송한다.
- **다이얼로그**: 네이티브 `prompt()`, `alert()`, `confirm()` 팝업을 지원한다.
- **메뉴**: 애플리케이션 메뉴와 컨텍스트 메뉴를 제공한다.
- **트레이/독**: 시스템 상태 아이콘과 macOS Dock 통합을 지원한다.

### 크로스 컴파일과 배포

단일 머신에서 macOS, Windows, Linux 타깃 바이너리를 모두 빌드할 수 있다.
`Deno.autoUpdate()`는 `latest.json` 매니페스트 하나로 바이너리 diff 자동 업데이트와 롤백을 처리한다.

### DevTools

Deno 런타임과 웹뷰 두 곳을 동시에 붙이는 통합 DevTools를 제공한다.

## 분석

### HTTP 서버를 브릿지로 쓰는 설계의 함의

Deno Desktop의 핵심 설계는 `Deno.serve()`를 애플리케이션 진입점으로 그대로 재활용한다는 점이다.
포트 지정 없이 런타임이 자동으로 웹뷰 주소에 바인딩하는 방식은
기존 Deno 웹 서버 코드를 수정 없이 데스크톱 앱으로 전환하겠다는 의도를 드러낸다.
이는 “웹 서버를 이미 쓰고 있으면 데스크톱 앱은 그냥 된다”는 포지셔닝이다.

이 설계의 이점은 명확하다.
기존 Deno 웹 프로젝트가 별도의 아키텍처 변경 없이 데스크톱으로 이식될 수 있다.
HTTP 요청/응답 모델에 익숙한 개발자에게 진입 장벽이 낮다.
그러나 이 설계는 동시에 애플리케이션 상태 관리 패턴이 웹 서버 패러다임에 묶인다는 뜻이기도 하다.
게임이나 오디오 편집기처럼 HTTP 요청-응답 모델과 맞지 않는 데스크톱 앱 유형에는 어색할 수 있다.

### 인-프로세스 통신이 Electron과 구분하는 지점

Electron의 가장 큰 비용 중 하나는 `ipcMain`/`ipcRenderer` 구조에서 발생하는 크로스 프로세스 통신 오버헤드다.
메시지를 직렬화하고 다른 프로세스 경계를 넘어 전달하는 과정이 복잡해지면
비동기 이벤트 버스처럼 변해 디버깅이 어려워진다.

Deno Desktop의 인-프로세스 채널은 이 오버헤드를 원천적으로 제거한다.
`bindings.<name>()` 호출이 크로스 프로세스 왕복 없이 같은 프로세스 안에서 완결된다.
이는 Tauri가 `invoke()` 메커니즘으로 채택한 방향과 유사하다.
성능과 단순성 측면에서 IPC 기반 아키텍처보다 유리하지만,
두 맥락(웹뷰와 Deno 런타임)이 공유 메모리 공간에 있다는 것이 의미하는 보안 경계는 별도로 검토가 필요하다.
Lobste.rs에서 mikedorf는 "인-프로세스 바인딩이 Tauri의 IPC 방식보다 진정으로 우월하다"고 평가했다.[^mikedorf]

### WebView 기본값이 의미하는 전략적 선택

백엔드 기본값이 OS 네이티브 웹뷰라는 것은 Tauri와 같은 방향이다.
macOS의 WKWebView, Windows의 WebView2, Linux의 WebKitGTK를 각각 사용한다.
이 선택은 바이너리 크기를 극적으로 줄인다. Electron 앱이 수백 MB인 반면
WebView 기반 앱은 수십 MB 수준이 될 수 있다.

그러나 이 선택에는 숨겨진 가정이 있다.
OS 웹뷰가 충분히 현대적이고 표준을 잘 준수한다는 전제다.
Windows의 WebView2는 비교적 최신이지만, Linux의 WebKitGTK는 버전 편차가 크다.
CEF 옵션을 제공하는 것은 이 현실에 대한 우회로지만,
기본값이 아니라는 점에서 개발자가 직접 호환성 문제를 진단하고 전환해야 하는 부담이 있다.
bobajeff는 이에 대해 "브라우저 엔진을 번들링하지 않으면서 webkitgtk의 혼란도 피하고 싶다면
WebUI처럼 브라우저에서 실행하는 옵션이 있으면 좋겠다"고 제안했다.[^bobajeff]
이 제안은 WebView와 CEF 사이의 실용적 중간 지점을 찾으려는 현장 개발자의 현실적 요구를 보여준다.

### 프레임워크 자동 감지의 의존성 구조

9개 프레임워크를 자동 감지하는 기능은 인상적으로 보이지만,
이 목록이 고정되어 있다는 것이 장기적 유지보수 부담을 의미한다.
각 프레임워크의 빌드 아웃풋 구조나 설정 파일 형식이 바뀔 때마다 Deno Desktop 쪽도 갱신이 필요하다.
프레임워크 생태계는 빠르게 변하고 새로운 프레임워크도 계속 등장한다.
이 목록이 어떤 기준으로 선정됐는지, 신규 프레임워크를 추가하는 프로세스가 무엇인지는 문서에 없다.

## 비평

### 웹 기술은 UI 툴킷이 아니다

jorisw는 문서의 “웹 기술은 세상에서 가장 널리 알려진 UI 툴킷이다”라는 표현에 정면으로 이의를 제기했다.[^jorisw]
Electron 앱이 비판받는 이유가 바로 UI 툴킷이 아니기 때문이라는 것이다.
버튼을 렌더링할 수는 있지만, OS 네이티브 UX 패턴을 따르지 않으며 브라우저마다 다르게 보인다.
lillesvin도 “크로스 플랫폼은 좋아하지만, 네이티브 UX 패턴을 따르는 네이티브 UI도 정말 좋아한다”며 같은 맥락의 아쉬움을 표했다.[^lillesvin]

이 비판은 Deno Desktop의 포지셔닝에서 해소되지 않는 긴장을 드러낸다.
웹 기술 기반이라는 특성 자체가 네이티브 룩앤필과 상충하며,
이 긴장은 WebView 기반 앱이라면 Electron이든 Tauri든 Deno Desktop이든 공통으로 안고 있는 구조적 한계다.

### “자동”이 숨기는 설정 복잡도

문서는 프레임워크 자동 감지와 포트 자동 바인딩을 편의 기능으로 제시한다.
그러나 이 자동화가 어떻게 작동하는지의 세부 사항은 거의 공개되지 않는다.
Next.js 앱을 `deno desktop`으로 빌드할 때 Next.js의 빌드 아웃풋을 어떻게 패키징하는지,
서버 컴포넌트(Server Components)와 같은 기능이 데스크톱 컨텍스트에서 어떻게 처리되는지,
환경 변수와 빌드 타임 설정이 어떻게 주입되는지가 불분명하다.

프레임워크를 쓰는 순간 “그냥 된다”와 “복잡한 설정이 필요하다” 사이의 거리는
실제로 사용해보기 전까지 알 수 없다.
문서가 최소 예시(단일 TypeScript 파일)에서 바로 프레임워크 지원으로 넘어가는 구성 자체가
그 간극을 의도적으로 회피하는 것처럼 보인다.

### Deno 퍼미션 모델과 네이티브 API의 경계가 불명확하다

Deno의 핵심 가치 제안 중 하나는 명시적 퍼미션 시스템이다.
`--allow-net`, `--allow-read`, `--allow-write` 등을 통해 권한을 세밀하게 제어한다.
그런데 데스크톱 앱에서 네이티브 알림, 파일 다이얼로그, 시스템 트레이에 접근할 때
이 퍼미션 모델이 어떻게 적용되는지 문서에서는 다루지 않는다.
sheept는 이 점을 직접 지적하면서, "컴파일 시점에 부여한 퍼미션이 컴파일된 바이너리에 내장된다"는 사실이
개발자 경험 측면에서 개선이 필요하다고 제안했다.[^sheept]

데스크톱 앱의 네이티브 기능들은 본질적으로 OS 리소스에 대한 접근이다.
이것이 기존 Deno 퍼미션 시스템 위에 별도 레이어로 얹히는지,
아니면 `deno desktop`으로 빌드된 앱은 퍼미션 제한이 완화되는지 불명확하다.
이 모호함은 보안 민감 컨텍스트에서 Deno Desktop을 사용하려는 팀에게 실질적 장벽이다.

### 크로스 컴파일 주장은 “백엔드 다운로드” 의존성을 가린다

“단일 머신에서 macOS, Windows, Linux 바이너리를 모두 빌드한다”는 주장은 강력하게 들린다.
그러나 “백엔드는 필요 시 다운로드된다”는 문구가 이 주장을 상당히 약화시킨다.
즉, 크로스 컴파일 자체는 네트워크 접근이 가능한 환경에서만 완전히 작동한다.
에어갭(air-gapped) 환경이나 엄격한 CI 네트워크 정책 환경에서는 이 과정이 깨질 수 있다.

더불어, 크로스 컴파일이 되었다고 해서 플랫폼별 동작이 보장되지는 않는다.
macOS 머신에서 Windows 바이너리를 빌드할 수 있지만
Windows의 WebView2가 특정 CSS 기능을 다르게 렌더링하는 문제나
Linux에서 시스템 알림 권한 처리 방식의 차이는
실제 타깃 플랫폼에서 테스트하기 전까지 발견되지 않는다.
크로스 컴파일은 빌드의 편의를 주지만 플랫폼 호환성 보장은 별개 문제다.

### 불안정 상태에서 조기 공개된 문서의 위험

현재 Deno Desktop은 canary 빌드에서만 사용 가능하고 API가 변경될 수 있다.
그럼에도 공식 문서 사이트(`docs.deno.com`)에 풀 페이지 문서가 공개되어 있다.
이 선택은 얼리어답터를 끌어들이는 동시에 API 변경 시 혼란을 야기한다.

문서가 최신 상태를 유지하지 못하면 버전 불일치로 인한 오류가 누적되고,
커뮤니티 Q&A와 Stack Overflow 답변이 구버전 API를 기반으로 확산된다.
Deno 팀이 이 비용을 감수하고 조기 공개를 선택한 것은 커뮤니티 피드백을 우선시한 결정으로 보이지만,
Deno Desktop의 첫인상이 “문서대로 안 된다”는 경험으로 형성될 위험이 있다.
Lobste.rs에서 wink는 실제로 400줄짜리 기본 앱을 컴파일했더니 `.app`이 296MB에 달했고
`--no-check` 플래그 없이는 타입 오류가 발생했다고 보고했다.[^wink]
snej도 마케팅에서 “기본적으로 작다(small by default)”를 강조하면서
CEF 백엔드가 Electron(약 100MB)보다 오히려 더 크다(150MB)는 모순을 지적했다.[^snej]
두 사례 모두 조기 공개된 문서가 현실과 괴리될 때 첫인상이 얼마나 빠르게 손상되는지 보여준다.

## 인사이트

### Deno Desktop의 실제 경쟁 상대는 Electron이 아니라 Tauri다

Deno Desktop을 Electron의 대안으로 포지셔닝하기 쉽지만, 기술적 접근이 Tauri에 훨씬 가깝다.
OS 네이티브 웹뷰 사용, 인-프로세스 통신, 작은 바이너리 크기, Rust 기반 네이티브 바인딩 대신 Deno 런타임 사용.
이 프로필은 Tauri의 아키텍처를 Rust 대신 TypeScript/Deno 개발자를 위해 재해석한 것이다.
HN 댓글에서도 droidjj는 "Tauri와 유사한 아키텍처지만, 비즈니스 로직이 Rust 대신 TypeScript"라고 정확히 요약했다.[^droidjj]

이 경쟁 구도가 중요한 이유는 Tauri가 이미 이 시장에서 상당한 입지를 확보했기 때문이다.
Deno Desktop이 Tauri 대비 제시해야 하는 가치는 명확하다:
Rust를 배우지 않아도 네이티브 바인딩을 TypeScript로 작성할 수 있다는 것,
그리고 Deno 생태계의 보안 모델과 패키지 관리를 그대로 가져올 수 있다는 것이다.
이 가치 제안이 얼마나 설득력 있을지는 Rust 장벽이 실제로 Tauri 채택을 막고 있는지에 달려 있다.

반면 bel8은 "Tauri가 150MB 추가 번들 크기가 있다 해도 다운로드에 고작 1~10초밖에 안 걸리는데
신뢰할 수 있는 렌더링 엔진을 얻는다면 왜 Deno Desktop을 써야 하나"라고 반론을 제기했다.[^bel8]
이 시각은 CEF 옵션 없이 WebView 기본값을 선택한 Deno Desktop의 타깃 사용자층이
Tauri 전체 사용자가 아닌, 바이너리 크기를 극도로 최적화하려는 특정 그룹임을 시사한다.
Tauri에서 Elixir/BEAM을 사이드카로 번들링하는 ElixirKit 같은 사례처럼[^auraham],
Tauri는 이미 다양한 런타임과의 통합 패턴이 성숙해 있다는 점도 고려해야 한다.

### 데스크톱 앱의 “웹 앱화”가 가속되면 런타임 경쟁이 데스크톱으로 옮겨온다

Node.js 기반 Electron이 데스크톱 시장을 장악한 것은 웹 기술로 데스크톱 앱을 만들 수 있다는 증명이었다.
Deno Desktop은 이 흐름의 다음 단계다: 더 나은 런타임으로 같은 패러다임을 반복한다.
이 흐름이 지속되면 데스크톱 앱 생태계는 점점 더 런타임 선택의 함수가 된다.

이것은 Deno에게 기회이면서 동시에 압박이다.
Node.js 호환성을 확장하면서 고유 장점(보안 모델, TypeScript 퍼스트, 표준 API)을 유지해야 한다.
데스크톱 시장에서도 Node.js 생태계와의 호환성이 채택의 열쇠가 될 것이다.
`--node-compat` 없이 npm 패키지를 쓸 수 있는 Deno의 현재 상태는 이 경쟁에서 유리한 출발점이다.

### 프레임워크 자동 감지 전략은 유지보수 부채를 시간으로 미룬다

9개 프레임워크를 자동 감지하는 것은 개발자 경험을 우선시한 결정이지만,
각 프레임워크의 변경에 대응하는 유지보수 비용이 Deno 팀에 집중된다는 구조적 문제가 있다.
역사적으로 이런 “모든 것을 지원하겠다”는 접근은 핵심 몇 가지를 잘 지원하는 방향보다
유지보수 부담이 지수적으로 커지는 경향이 있다.

대안 모델이 있다. 플러그인 또는 어댑터 패턴으로 프레임워크 지원을 커뮤니티에 위임하는 방식이다.
Vite 플러그인 생태계나 Webpack 로더 모델이 이 방향의 예시다.
Deno Desktop이 안정화 단계에서 이 구조를 채택하지 않는다면
9개로 시작한 지원 목록이 확장을 거부하거나 갱신이 지연되는 병목이 될 수 있다.

### 통합 DevTools는 데스크톱 앱 디버깅 경험을 근본적으로 바꿀 수 있다

Electron 앱을 디버깅할 때의 고통 중 하나는 메인 프로세스(Node.js)와 렌더러 프로세스(Chromium) 두 곳에
각각 다른 DevTools를 붙여야 한다는 것이다.
Deno Desktop이 Deno 런타임과 웹뷰를 동시에 붙이는 통합 DevTools를 제공한다는 것은
이 고통을 직접 겨냥한 설계다.

이 기능이 실제로 얼마나 통합적인지가 채택 경험을 결정할 것이다.
단순히 두 개의 DevTools를 하나의 창에 나란히 배치하는 수준인지,
아니면 웹뷰의 네트워크 요청과 Deno 런타임의 핸들러를 연결해서 추적할 수 있는 수준인지가 다르다.
후자라면 Electron보다 디버깅 경험이 질적으로 우월해진다.
이것이 완성되면 Deno Desktop의 가장 강력한 차별점이 될 수 있다.
