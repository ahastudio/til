# GPU 커널을 실행하면 무슨 일이 일어나는가

원문: <https://fergusfinn.com/blog/what-happens-when-you-run-a-gpu-kernel/>

HN 토론: <https://news.ycombinator.com/item?id=48718863> (234점, 28개 댓글)

## 요약

Fergus Finn의 이 글은 CUDA 커널 하나가 소스 코드에서 실행 결과로 돌아오기까지 전 과정을 추적한 35분 분량의 기술 심층 분석이다.
예제는 백만 개의 부동소수점을 더하는 벡터 덧셈 커널이다.
저자는 LD_PRELOAD 인터포지션, strace, cuobjdump, GPU 쪽 메모리 peek() 기법을 조합해 NVIDIA의 비공개 드라이버 스택을 직접 역공학하며 내부 동작을 밝혀낸다.

컴파일 파이프라인은 `nvcc`가 조율하는 여러 단계로 구성된다.
CICC(LLVM 기반 컴파일러)가 CUDA 소스를 PTX(Parallel Thread Execution)로 변환한다.
PTX는 무한한 가상 레지스터를 갖는 가상 ISA(Instruction Set Architecture)로, 특정 하드웨어에 종속되지 않는다.
PTXAS는 이 PTX를 특정 GPU 아키텍처에 최적화된 SASS(Streaming Assembler) 기계어로 변환하고,
fatbinary가 SASS와 압축된 PTX를 함께 패키징해 미래 아키텍처와의 순방향 호환성을 확보한다.

커널 실행 요청은 호스트에서 GPU로 복잡한 경로를 거친다.
호스트 스텁(host stub)이 커널 인수를 특정 상수 뱅크 오프셋에 맞춰 버퍼에 패킹하고,
CUDA 런타임이 등록 테이블을 조회해 호스트 함수 포인터와 디바이스 심볼을 매핑한다.
사용자 모드 드라이버인 `libcuda.so.1`은 디바이스 파일에 900개 이상의 ioctl 호출을 통해 GPU와 통신한다.

GPU 내부에서 컴퓨트 워크 디스트리뷰터(compute work distributor)가 4,096개의 블록을 128개의 SM(Streaming Multiprocessor)에 분배한다.
각 SM은 최대 48개의 상주 워프(warp)를 관리하며, 레지스터 파일 용량과 스레드 한계에 의해 제약된다.
워프 스케줄링은 런타임 하드웨어 의존성 감지가 아니라 컴파일러가 각 명령어에 심어 놓은 제어 코드에 의존한다.
제어 코드에는 고정 지연 연산의 정적 스톨 횟수, 가변 지연 연산(로드, 특수 함수)을 위한 6개의 스코어보드 배리어, 스케줄러 힌트가 포함된다.

메모리 계층을 통한 데이터 흐름도 정밀하게 추적된다.
32개 스레드의 합쳐진 요청은 4개의 32바이트 섹터 페치로 병합되며,
SM의 L1 데이터 캐시, 분산된 L2 캐시(총 72MB), GDDR6X VRAM 순서로 접근한다.
벡터 덧셈은 12바이트를 전송할 때마다 부동소수점 연산이 하나 발생하는 극단적인 메모리 바운드 연산이며,
DRAM 최대 대역폭의 79.65%를 달성하고 전체 실행 시간은 10.78μs다.
커널 완료 시 GPU가 관리하는 완료 세마포어가 카피 엔진을 통한 DMA를 트리거하며,
결과가 L2에 캐시된 상태이므로 DRAM 왕복 없이 호스트 메모리로 복사된다.

## 분석

### PTX라는 가상 ISA가 GPU 생태계의 수직 통합을 가능하게 한다

PTX는 “무한한 가상 레지스터”를 갖는 중간 표현이다.
실제 레지스터 수 제한은 PTXAS가 대상 아키텍처에 맞춰 해결하며, SASS로의 변환 시 실제 레지스터 할당이 결정된다.
fatbinary는 현재 아키텍처의 SASS와 압축된 PTX를 함께 패키징한다.
SASS가 없는 미래 아키텍처에서는 PTX가 런타임에 JIT 컴파일돼 실행된다.

이 구조는 Java 바이트코드나 .NET IL이 CPU에서 하는 역할과 구조적으로 같다.
컴파일 시점의 하드웨어 종속성을 줄이면서 특정 플랫폼(NVIDIA)에 소프트웨어 생태계를 묶어 둔다.
CUDA 프로그램은 PTX를 통해 여러 GPU 세대에서 실행될 수 있지만,
그 PTX는 NVIDIA의 비공개 PTXAS 없이는 최적의 SASS로 변환되지 않는다.
생태계 호환성을 제공하면서도 경쟁사 하드웨어로의 이식을 어렵게 만드는 구조다.

### GPU 스케줄러가 컴파일러에 의존하는 것은 의도적인 하드웨어 단순화다

CPU는 하드웨어 아웃오브오더(OOO) 실행 엔진을 통해 런타임에 명령어 의존성을 감지한다.
Tomasulo 알고리즘 같은 복잡한 하드웨어가 실행 순서를 동적으로 재배치한다.
GPU는 다른 선택을 했다.
컴파일러(PTXAS)가 스톨 횟수와 배리어를 각 명령어에 정적으로 심고,
스케줄러는 그 정보를 그대로 따른다.

이 선택의 근거는 워프의 수에 있다.
각 SM이 48개의 워프를 관리하고, 하나의 워프가 메모리 대기 상태에 있을 때 다른 워프로 전환해 지연 시간을 감춘다.
수천 개의 스레드가 통계적으로 지연 시간을 상쇄하기 때문에, CPU처럼 하나의 실행 흐름을 위한 복잡한 하드웨어 의존성 감지가 필요 없다.
대신 하드웨어를 단순하게 유지하면서 복잡성을 컴파일러로 이전했다.

이 설계의 귀결은 중요하다.
GPU 성능은 컴파일러 품질에 극도로 민감하다.
PTXAS가 레지스터 할당이나 배리어 배치를 잘못하면 하드웨어 수준에서 이를 보정하는 메커니즘이 없다.
Triton 같은 새로운 GPU 컴파일러가 성능에 큰 영향을 미치는 이유가 여기에 있다.

### 900개의 ioctl이 보여주는 드라이버 스택의 불투명성

저자가 strace로 측정한 결과, 단순한 커널 하나를 실행하는 데 900개 이상의 ioctl이 발생했다.
`libcuda.so.1`이 디바이스 파일에 내리는 이 ioctl들은 GPU를 독립적인 운영 도메인으로 관리하는 사용자 모드 드라이버의 실제 인터페이스다.
QMD(Queue Meta Data)라는 256바이트 구조체가 그리드 기하, 레지스터 수, 인수 주소를 담아 푸시버퍼 명령 스트림을 통해 GPU에 전달되고,
메모리 맵 도어벨 레지스터를 통해 실행이 트리거된다.

이 구조는 공개된 사양이 없다.
저자는 LD_PRELOAD를 이용해 드라이버 메모리 매핑을 가로채고,
GPU 쪽에서 직접 메모리를 읽는 peek() 커널을 만들어 호스트에서 접근할 수 없는 디바이스 메모리를 열람하며,
cuobjdump로 SASS 제어 워드 인코딩을 해독했다.
이 역공학 작업의 난이도 자체가 CUDA 스택의 불투명성을 증언한다.

aberrahmane_b는 이 글에서 도어벨과 QMD 섹션이 가장 유용했다고 말한다.[^aberrahmane_b]
대부분의 CUDA 설명은 커널, 블록, 워프에서 멈추는데,
이 글의 CPU-드라이버-GPU 경로 추적이 실제로 무슨 일이 일어나는지를 비로소 연결해 준다는 것이다.
QMD 구조체가 실제로 그 연결 고리를 담당하며, 이것을 이해하면 CUDA 런타임 API의 추상화가 무엇을 감추고 있는지 보인다.

einpoklum은 이 투명성 문제에 대한 실용적인 대안을 제시한다.[^einpoklum]
CUDA 런타임 API 대신 드라이버 API를 쓰면 내부 동작이 훨씬 잘 드러난다.
커널 소스를 문자열로 받아 NVIDIA의 런타임 컴파일러(nvrtc)로 컴파일하면,
중간 과정을 직접 관찰할 수 있다.
`cuda-api-wrappers` 라이브러리는 이 드라이버 API를 모던 C++ 방식으로 감싼 형태로, 가독성과 투명성을 함께 제공한다.

### 메모리 바운드 특성이 GPU 아키텍처 선택을 설명한다

벡터 덧셈은 각 부동소수점 연산에 12바이트의 메모리 전송이 필요하다.
두 입력값을 읽고(8바이트) 결과를 쓰는(4바이트) 세 번의 메모리 접근과 한 번의 덧셈.
연산보다 데이터 이동이 압도적으로 많은 이 패턴에서 GPU는 DRAM 최대 대역폭의 79.65%를 달성했다.
컴퓨트가 아니라 메모리가 병목이다.

32개 스레드가 합쳐진 요청이 4개의 32바이트 섹터로 병합되는 코얼레스드(coalesced) 접근 패턴은 이 병목을 완화하는 핵심 최적화다.
스레드들이 연속된 메모리를 함께 접근하면 하나의 큰 요청으로 합쳐지지만,
접근 패턴이 분산되면 개별 요청으로 쪼개져 대역폭을 낭비한다.
AI 모델 학습에서 “GPU 최적화”의 상당 부분이 메모리 접근 패턴을 정렬하는 작업인 이유가 이 구조에서 나온다.

## 비평

### 벡터 덧셈 예제는 AI 워크로드의 실제 병목을 대표하지 않는다

이 글은 벡터 덧셈을 통해 GPU 커널 실행 경로를 설명한다.
벡터 덧셈이 GPU 프로그래밍의 “Hello World”인 이유는 단순하기 때문이다.
그런데 이 단순함이 글의 핵심 메시지를 왜곡할 수 있다.

벡터 덧셈은 극단적으로 메모리 바운드 연산이며,
실제 AI 워크로드의 핵심인 행렬 곱셈(GEMM), 컨볼루션, 어텐션은 이와 다른 특성을 갖는다.
고품질로 구현된 GEMM은 컴퓨트 바운드에 가까우며, 텐서 코어가 활성화되면 명령어 처리량이 메모리 대역폭보다 먼저 한계에 달한다.
“GPU 성능의 병목은 메모리”라는 결론은 이 예제에서는 맞지만, 최적화된 AI 커널에서는 이야기가 달라진다.

저자는 이 글에서 warp utilization(82.77%)과 issue rate(5.17%)의 극단적인 차이를 제시한다.
워프의 82%가 활성 상태이지만 실제 명령어 발행률은 5%에 불과하다.
메모리 대기로 인해 대부분의 시간이 낭비된다는 의미인데,
실제 성능 엔지니어가 최적화해야 할 커널은 이 두 수치를 모두 끌어올리는 것이 목표다.
예제가 그 긴장을 보여주지 못한다.

### 역공학 기법의 재현 가능성이 독자에게 불분명하다

저자가 사용한 기법들은 인상적이지만 범용적이지 않다.
LD_PRELOAD 인터포지션, GPU 쪽 peek() 커널, SASS 제어 워드 디코딩은 각각 CUDA 버전, GPU 아키텍처, 드라이버 버전에 민감하다.

saagarjha는 제어 코드 설명이 단순화됐다고 지적한다.[^saagarjha]
글은 제어 코드를 명령어 내 비트로 설명하지만, 실제로는 단순한 비트 필드가 아니라 테이블 룩업 구조다.
이 차이는 미묘해 보이지만, PTXAS가 제어 코드를 생성하는 방식을 이해하려는 독자에게는 중요한 정오표다.
글의 역공학 성과 자체가 틀린 것은 아니지만, 제어 코드의 정확한 메커니즘에 대해서는 추가 조사가 필요하다.

fooblaster는 NVIDIA의 불투명성이 절대적이지 않다는 점을 지적한다.[^fooblaster]
NVIDIA가 GitHub에 공개한 오픈 GPU 문서 저장소가 존재하며, 일부 메서드 문서와 QMD 형식이 거기에 있다.
커널 소스를 읽지 않고도 이 공식 문서에서 출발할 수 있다는 것이다.
저자가 역공학으로 밝혀낸 내용 중 일부는 공식 채널을 통해서도 접근 가능하며,
글이 이 공개 문서를 참조했는지 명시하지 않는 것은 독자가 필요 이상으로 어렵다고 느끼게 만들 수 있다.

SASS 인코딩은 NVIDIA가 아키텍처마다 변경할 수 있는 비공개 사양이다.
Ampere, Ada Lovelace, Hopper 아키텍처에서 제어 워드 인코딩이 달라질 수 있으며,
저자가 발견한 구체적인 수치(스톨 사이클 수, 배리어 구조)는 특정 아키텍처에만 적용된다.
독자가 다른 GPU에서 동일한 기법을 시도하면 다른 결과를 볼 수 있다.
글은 이 아키텍처 특정성을 명시적으로 다루지 않아, 발견 사항의 일반화 범위가 불분명하다.

### PTX 런타임 JIT 컴파일의 비용이 순방향 호환성의 대가로 명시되지 않는다

fatbinary에 PTX를 함께 패키징하는 것이 “순방향 호환성”을 제공한다는 설명은 정확하다.
그러나 이 호환성에는 비용이 있다.

새 아키텍처에서 SASS 없이 PTX만으로 실행하면 런타임 JIT 컴파일이 필요하다.
이 컴파일은 첫 실행 시 수백 밀리초에서 수 초의 지연을 유발할 수 있다.
게임의 “셰이더 컴파일 버벅임”과 구조적으로 동일한 문제다.
CUDA에서는 cubin 캐싱으로 이를 완화하지만, 캐시 미스 시 지연은 실제 운영 환경에서 문제가 된다.
글은 PTX 호환성의 이점을 설명하면서 이 지연 비용을 언급하지 않는다.

## 인사이트

### GPU 컴파일러로 복잡성이 이전된 결과, 컴파일러 연구가 하드웨어 연구만큼 중요해졌다

CPU 세계에서는 하드웨어 OOO 엔진이 소프트웨어 품질을 어느 정도 보정해준다.
나쁜 명령어 순서를 하드웨어가 런타임에 재배치할 수 있다.
GPU는 이 보정 레이어를 컴파일러로 이전했다.
PTXAS가 잘못된 레지스터 할당이나 배리어 배치 결정을 내리면 하드웨어에서 복구할 방법이 없다.

이 구조는 AI 컴파일러 연구가 왜 현재 가장 뜨거운 분야 중 하나인지 설명한다.
Triton, MLIR, XLA, TVM은 모두 GPU 하드웨어 특성을 컴파일러 레벨에서 최대한 활용하려는 시도다.
하드웨어를 더 추가하는 것보다 같은 하드웨어에서 더 많은 것을 끌어내는 컴파일러 최적화가 실용적으로 더 중요한 시점에 도달했다.
이 글이 밝히는 제어 워드 구조는 컴파일러가 정확히 무엇을 생성해야 하는지를 보여주는 명세이기도 하다.

orliesaurus는 이 맥락에서 현재 급성장 중인 커널 최적화 전문 기업들의 미래를 묻는다.[^orliesaurus]
지금은 사람이 직접 커널을 최적화하는 것이 훨씬 낫지만, AI가 이 일을 대체하거나 NVIDIA가 직접 공개 라이브러리를 낼 수 있다.
spmurrayzzz는 이 전망에 중요한 단서를 더한다.[^spmurrayzzz]
새 하드웨어 세대마다 이전 세대 모델이 보지 못한 새로운 제약과 기능이 등장한다.
Blackwell의 tcgen05가 그 예로, 이전 세대 모델이 학습하지 못한 새로운 명령어 집합이다.
커널 최적화의 자동화는 특정 아키텍처에서는 가능해도, 새 아키텍처가 출시될 때마다 기존 모델이 분포 밖(out-of-distribution)에 놓이는 문제가 반복된다.
GPU 컴파일러 연구의 필요성이 하드웨어 세대마다 리셋되는 구조다.

### CUDA 드라이버의 불투명성은 역공학으로만 드러나며, 이것이 경쟁 장벽이다

900개의 ioctl, QMD 구조체, 메모리 맵 도어벨.
이 중 어느 것도 NVIDIA가 공식적으로 문서화하지 않는다.
저자가 LD_PRELOAD, strace, GPU 쪽 peek() 커널을 동원해 수백 시간의 역공학을 수행해야 이 글이 탄생했다.

이것은 우연이 아니다.
AMD의 ROCm과 Intel의 oneAPI는 CUDA와 호환 가능한 스택을 만들려 하지만,
CUDA의 성능 이점 상당 부분은 수십 년간 최적화된 비공개 드라이버 스택에서 온다.
이 스택을 완전히 재현하려면 CUDA 스택을 완전히 이해해야 하고,
그 이해는 이 글 수준의 역공학을 GPU의 모든 기능에 대해 반복해야 한다는 의미다.
2000년대 Windows 드라이버 ABI가 Linux 호환성을 가로막았던 방식과 구조적으로 같다.
드라이버 인터페이스의 불투명성이 생태계 잠금(lock-in)의 핵심 메커니즘이다.

### 메모리 대역폭이 병목임을 이해하면 AI 하드웨어 로드맵이 다르게 보인다

벡터 덧셈의 79.65% DRAM 활용률은 연산이 아니라 데이터 이동이 병목이라는 것을 보여준다.
이것은 단순한 벤치마크 결과가 아니라 AI 워크로드의 상당 부분에 적용되는 구조적 특성이다.
임베딩 룩업, KV 캐시 접근, 배치 크기가 작은 추론은 모두 메모리 바운드 특성을 갖는다.

이 통찰은 왜 AI 하드웨어 경쟁이 FLOPS보다 메모리 대역폭 중심으로 재편됐는지 설명한다.
NVIDIA가 HBM에 집착하는 이유, HBM4가 차세대 GPU 성능 향상의 핵심 요소가 되는 이유가 이 구조에서 나온다.
Cerebras가 수십 기가바이트의 SRAM을 칩 위에 올리는 전략을 선택한 것도 동일한 병목을 다른 방식으로 우회하는 시도다.
“GPU를 더 추가하라”는 처방이 메모리 바운드 워크로드에 잘 작동하지 않는 이유도 여기 있다.
연산 유닛을 늘려도 데이터를 공급하는 메모리 대역폭이 늘지 않으면 성능은 선형으로 증가하지 않는다.

### AI 소프트웨어 스택의 추상화가 가리는 것들

CUDA, PyTorch, transformers 라이브러리는 이 글이 다루는 복잡성 전체를 숨긴다.
Python 두 줄로 GPU 벡터 덧셈을 실행할 수 있고, 그 뒤에서 무슨 일이 일어나는지 모르고도 모델을 훈련시킬 수 있다.
이 추상화는 생산성을 높이지만 동시에 이해의 착각을 만든다.

성능 문제, 메모리 부족 오류, 이상한 수치 불안정성이 발생할 때,
추상화 레이어만 이해하는 엔지니어는 이유를 찾기 어렵다.
이 글이 밝히는 레벨, 즉 warp 스케줄러, 제어 코드, 메모리 코얼레싱 수준의 이해가 없으면
PyTorch 프로파일러가 보여주는 수치의 의미를 해석하기 어렵다.
고성능 AI 시스템을 다루는 엔지니어에게 이 레벨의 이해가 실용적 가치를 갖는 이유다.
추상화는 복잡성을 제거하지 않으며, 단지 특정 시점까지 미루어 둘 뿐이다.
