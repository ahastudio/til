# Parse, Don't Validate

원문: <https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/>

HN 토론: <https://news.ycombinator.com/item?id=21476261> (642점, 230개 댓글)

## 요약

Alexis King이 2019년에 쓴 이 글은 타입 안전 코드 설계의 핵심 원칙을 세 단어로 제시한다.
“Parse, don't validate” — 검증(validate)하지 말고 파싱(parse)하라.
저자에 따르면 검증과 파싱의 차이는 “정보를 얼마나 보존하는가”에 있다.
검증은 조건을 확인하지만 그 사실을 버린다.
파싱은 같은 과정을 거치되, 확인한 사실을 더 풍부한 타입으로 반환해 하위 코드에서 재사용할 수 있게 한다.

가장 직관적인 예는 리스트의 `head` 함수다.
`head :: [a] -> Maybe a`는 빈 리스트일 때 `Nothing`을 반환하는 타입이다.
그러나 실제로 `head`를 호출하는 맥락의 대부분은 리스트가 비어 있지 않음을 이미 알고 있다.
이때 타입이 `Maybe a`이면 “절대 발생하지 않을” `Nothing` 케이스를 계속 처리해야 한다.
해법은 `head :: NonEmpty a -> a`다.
`NonEmpty a` 타입 자체가 “최소 하나의 원소가 있다”는 사실을 표현하므로,
이 타입을 사용하는 순간 실패가 불가능해진다.

저자는 실용적 지침을 네 가지로 정리한다.
첫째, 불법 상태를 표현 불가능하게 만드는 데이터 구조를 선택한다.
둘째, 검증은 시스템 경계에서 한 번만 수행하고 내부에서는 파싱된 타입만 사용한다.
셋째, 완벽한 표현이 어려울 때는 스마트 생성자(smart constructor)와 추상 데이터 타입(abstract datatype)을 활용한다.
넷째, `m ()`를 반환하는 함수를 의심한다 — 이런 함수는 검증 증거를 버리는 경우가 많다.

글은 “산탄총 파싱(shotgun parsing)”이라는 언어이론적 보안(language-theoretic security) 개념도 언급한다.
검증 로직이 코드 전체에 흩어져 있으면 잘못된 데이터가 거부되기 전에 부분적으로 처리될 위험이 있다.
이것이 보안 취약점의 원천이 된다.
파싱 우선 설계는 이 문제를 구조적으로 차단한다.

## 분석

### 이 원칙의 핵심은 증거를 버리지 않는 것이다

“검증이 아니라 파싱”의 본질은 프로그래밍의 인식론적 문제다.
코드가 무언가를 확인했을 때 그 사실을 어떻게 다음 단계로 전달하는가.
검증 접근법은 `isValid(x)`가 `true`를 반환해도 컴파일러는 그 사실을 모른다.
다음 함수에서 다시 `x`가 유효한지 확인하거나, 확인 없이 가정으로 진행하거나 둘 중 하나다.
파싱 접근법은 `parse(x) :: ValidatedX`를 반환함으로써 컴파일러가 이후 코드에서 유효성을 자동으로 보증한다.

이 차이는 표면적으로 사소해 보이지만 규모에서 결정적인 차이를 만든다.
검증 방식에서는 유효성이 “전역 불변량(global invariant)”이 되어 모든 함수가 각자 책임진다.
파싱 방식에서는 유효성이 “로컬 타입 제약”이 되어 타입 시스템이 강제한다.
전자는 개발자의 기억력과 코드 리뷰에 의존하고, 후자는 컴파일러에 의존한다.
이 논문의 관점에서 타입 시스템은 문서화 도구가 아니라 증명 도구다.

이 원칙의 언어 구현 방식은 다양하다.
jillesvangurp는 Kotlin의 스마트 캐스트가 같은 아이디어를 다른 각도에서 실현한다고 지적했다.[^jillesvangurp]
`null` 검사 블록 안에서 컴파일러는 변수 타입을 자동으로 `Whatever?`에서 `Whatever`로 좁힌다.
증거를 버리지 않는다는 원칙이 언어 수준에 내재화된 예다.
개발자가 `Maybe`를 수동으로 처리하는 부담 없이 같은 보증을 얻는다.

### “불법 상태를 표현 불가능하게” 만들기는 설계 결정의 선행화다

`NonEmpty a`를 사용한다는 것은 단순한 타입 변경이 아니다.
빈 컬렉션을 처리하는 책임을 어디에 둘지라는 설계 결정을 타입 수준으로 올리는 것이다.
`head :: [a] -> Maybe a`를 쓰는 코드는 호출 지점마다 “만약 비어 있으면”을 결정한다.
`head :: NonEmpty a -> a`를 쓰는 코드는 `NonEmpty a`를 구성하는 시점에 그 결정을 내린다.

이 “선행화” 패턴은 설계의 흐름을 바꾼다.
호출 지점에서 방어적 처리를 제거하고, 데이터 구성 지점으로 책임을 모은다.
이것이 “경계에서 한 번만 검증하라”는 지침의 타입 수준 표현이다.
경계 밖에서 들어온 데이터는 검증 후 즉시 내부 타입으로 변환되고,
그 이후로는 내부 타입만 사용하므로 방어적 코드가 필요 없다.

이 패턴은 Haskell 외부에서도 반복된다.
Rust의 `NonZeroU32`, `std::num::NonZero*` 계열, TypeScript의 branded type 패턴,
Java의 Value Object 패턴이 모두 같은 구조다.
“불법 상태를 표현 불가능하게”라는 표현은 Yaron Minsky가 OCaml 맥락에서 먼저 쓴 것으로
알려졌는데, King의 글은 이 개념에 “파싱”이라는 구체적 절차를 연결함으로써 실행 가능한 지침으로 만들었다.

HN 논의에서 Izkata는 이 접근을 “타입 헝가리안 표기법(Typed Hungarian)”으로 해석했다.[^Izkata]
앱스 헝가리안 표기법(Apps Hungarian)은 변수 이름에 `validData`, `rawData` 같은 의미론적 접두어를 붙여
잘못된 조합(`showValidData(rawData)`)이 눈에 띄게 하는 관행이었다.
이것이 단순 데이터 타입 접두어를 붙이는 시스템 헝가리안으로 변질됐고,
이제 타입 시스템 자체가 그 역할을 맡는 형태로 다시 진화했다는 해석이다.
“파싱 우선 설계”는 변수 이름으로 하던 의미론적 인코딩을 컴파일러가 강제하는 타입으로 옮긴 것이다.

### 산탄총 파싱은 보안 문제이자 설계 문제다

저자가 언급하는 “산탄총 파싱(shotgun parsing)”은 언어이론적 보안(LangSec) 연구에서 나온 개념이다.
파싱과 처리가 코드 전체에 혼재된 패턴을 가리키며, 이 패턴에서는 잘못된 입력이 부분적으로 처리된 뒤 거부된다.
“부분 처리”가 발생한다는 것은 시스템 상태가 이미 변경됐을 수 있다는 의미다.
이것이 취약점의 원천이다.

파싱 우선 설계는 이 문제를 구조적으로 해결한다.
입력이 시스템 경계에서 완전히 파싱되지 않으면 내부로 진입하지 못한다.
“경계에서 한 번, 내부에서는 절대”라는 원칙이 산탄총 파싱을 설계 수준에서 불가능하게 만든다.
이것은 단순한 코딩 습관 개선이 아니라 아키텍처적 선택이다.

## 비평

### 스마트 생성자는 표현력의 타협이 아니라 캡슐화의 포기다

저자는 완벽한 타입 표현이 어려울 때 스마트 생성자와 추상 데이터 타입을 대안으로 제안한다.
그러나 이 대안은 글이 인정한 것보다 더 큰 비용을 수반한다.
스마트 생성자 패턴은 타입의 내부 생성자를 숨기고 검증 로직이 포함된 함수를 통해서만 값을 만들게 강제한다.
이를 위해 모듈 경계와 가시성 제어가 필요하다.

모든 언어가 이 메커니즘을 지원하지는 않는다.
Haskell에서는 모듈 시스템이 이를 지원하지만,
동적 타입 언어나 모듈 경계가 느슨한 코드베이스에서는 스마트 생성자가 컨벤션에 불과하다.
컴파일러가 강제하지 않는 스마트 생성자는 검증 방식과 본질적 차이가 없다 — 둘 다 개발자가 규칙을 기억해야 한다.
저자는 이 맥락 의존성을 충분히 다루지 않는다.

흥미롭게도 저자는 HN 댓글에서 스스로 이 한계에 답했다.[^lexi-lambda]
`StringEndingInDot` 타입을 직접 구현해 보이면서, 보일러플레이트 비용이 크고 합성이 안 된다는 점을 인정했다.
이 문제에 대한 대안으로 “Ghosts of Departed Proofs” 논문을 제시했는데,
모듈 경계 없이도 타입 수준 증명을 합성 가능하게 하는 기법이다.
그러나 이 기법 역시 Haskell의 고급 타입 기능 위에서만 작동한다는 맥락 의존성은 동일하게 남는다.

### 글은 Haskell의 타입 시스템을 전제하며 다른 언어로의 적용 비용을 과소평가한다

이 글이 설득력 있게 읽히는 이유 중 하나는 Haskell이 이 패턴을 지원하도록 설계됐기 때문이다.
대수적 데이터 타입(ADT), 타입 추론, 전수 패턴 매칭(exhaustive pattern matching), 강력한 모듈 시스템 — 이 모두가 “불법 상태를 표현 불가능하게” 만드는 데 기여한다.

Java나 Python에서 같은 원칙을 적용하려면 훨씬 많은 보일러플레이트가 필요하고,
컴파일러 지원이 부분적이다.
TypeScript의 branded type이나 discriminated union은 Haskell의 ADT와 유사하지만,
타입 소거(type erasure)와 구조적 타이핑의 한계로 완전하지 않다.
글은 “이 아이디어는 언어에 독립적”이라고 암시하지만,
실제로 이 아이디어가 최대 효과를 내는 맥락은 Haskell/OCaml 계열의 ML 언어다.
다른 언어에서는 원칙을 부분적으로만 실현할 수 있으며, 그 “부분”이 어디까지인지 글은 논의하지 않는다.

이 접근성 문제는 HN 논의에서도 되풀이됐다.
iddan은 “훌륭한 글이지만 Haskell 코드를 이해할 수 없다 — TypeScript나 Rust, Swift 예시를 써줄 수 없냐?”고 직접 물었고,[^iddan]
많은 개발자가 공감했다.
z3t4는 TypeScript의 구체적 한계를 지적했다.[^z3t4]
`const head = (arr: Array<number>): number => arr[0]`는 엄격한 null 검사(strict null checks)를 켜도
빈 배열에서 `undefined`를 조용히 반환한다.
TypeScript는 구조적 타이핑 구조상 이 경우를 타입으로 잡을 방법이 없다.
이것이 글이 과소평가하는 “부분”의 구체적 사례다.

### “시스템 경계”의 정의가 불분명해 실천 지침으로서의 유효성을 제한한다

저자는 “경계에서 파싱하라”고 말하지만 무엇이 경계인지 정의하지 않는다.
마이크로서비스 아키텍처에서는 모든 서비스 호출이 경계다.
내부 모듈 사이도 “경계”로 볼 수 있다.
경계를 어디로 정의하느냐에 따라 파싱 코드의 양이 크게 달라진다.

엄격하게 적용하면 모든 외부 입력 — 함수 인자, 네트워크 응답, 파일 읽기 — 이 파싱 대상이 된다.
이것은 `NonEmpty a`나 `EmailAddress` 같은 타입을 수십, 수백 개 만들어야 할 수 있음을 의미한다.
작은 코드베이스에서는 관리 가능하지만 대규모 팀에서는 타입 폭발(type explosion) 문제가 발생한다.
이 트레이드오프를 글은 인식하지 못한다.

## 인사이트

### 이 원칙을 따르면 설계 결정이 선행화되고, 이것이 가장 어려운 변화다

“파싱 우선 설계”를 채택할 때 가장 큰 변화는 기술적인 것이 아니라 프로세스적인 것이다.
타입 중심 설계는 코드를 작성하기 전에 데이터의 형태를 정의해야 한다.
`NonEmpty a`를 사용하려면 어디서 빈 리스트 가능성을 제거할지 미리 결정해야 한다.
이것은 요구사항이 불확실한 초기 단계에서 구체적 결정을 강제한다.

이 선행화는 양날의 검이다.
잘못된 설계가 나중에 발견됐을 때 타입 변경이 연쇄적으로 전파되는 비용이 크다.
검증 방식은 로직이 분산돼 있어 변경이 국소적으로 가능하지만,
파싱 방식은 타입 변경이 컴파일 오류를 통해 모든 사용 지점에 드러난다.
이 드러남이 장점이기도 하지만 리팩터링 비용을 높인다.
이 트레이드오프는 요구사항이 안정된 도메인과 탐색적 개발 단계에서 다른 판단을 낳는다.

### 이 원칙이 가장 강력한 효과를 내는 곳은 외부 세계와의 경계, 즉 직렬화 레이어다

“파싱 우선”이 가장 높은 가치를 발휘하는 곳은 코드 내부 함수 간 인터페이스가 아니라
외부 세계와의 경계다.
JSON을 `User` 타입으로 파싱하는 코드, HTTP 요청을 `ValidatedRequest`로 변환하는 코드,
데이터베이스 쿼리 결과를 도메인 모델로 변환하는 코드가 핵심 적용 지점이다.

이 레이어에서는 잘못된 입력이 시스템으로 진입하는 것을 차단하는 효과가 명확하다.
Haskell의 `aeson`, Rust의 `serde`, Python의 `pydantic`이 모두 이 레이어를 구조화한다.
최근 LLM 응용에서 “구조화된 출력(structured output)” 트렌드도 같은 원리다.
AI 모델의 자연어 출력을 타입이 있는 데이터 구조로 파싱하는 것은
“파싱 우선”의 현대적 표현이다.
코드 내부에서 “parse, don't validate”를 완벽히 실현하는 것보다,
이 경계 레이어를 체계적으로 구성하는 것이 실용적으로 더 큰 효과를 낸다.

### 이 글은 타입 시스템을 명세 언어로 사용하는 설계 패러다임 전환의 입문이다

King의 글이 수년이 지나도 반복적으로 인용되는 이유는 설득력 있는 기술 논증 때문만이 아니다.
이 글이 암시하는 패러다임 전환이 있기 때문이다.
전통적 개발에서 타입은 런타임 표현 방식을 설명한다.
King이 제안하는 방식에서 타입은 시스템이 어떤 상태를 허용하고 허용하지 않는지를 명세한다.

이 전환의 끝에는 의존 타입(dependent type)과 정리 증명(theorem proving)이 있다.
Idris, Agda, Coq 같은 언어들은 타입 시스템이 수학적 명제를 표현할 수 있는 수준으로 이 아이디어를 밀어붙인다.
“Parse, don't validate”는 이 스펙트럼에서 비교적 실용적인 지점이다.
완전한 타입 증명을 요구하지 않고, Haskell과 같은 주류 함수형 언어에서 일상적으로 적용 가능한 수준에서
타입 시스템의 증명 능력을 활용한다.

이 관점에서 보면 글이 제안하는 것은 코딩 기술이 아니라 사고방식의 전환이다.
“이 데이터가 유효한가?”라는 질문을 “이 데이터를 표현하는 타입은 무엇인가?”로 바꾸는 것이다.
이 질문의 전환이 일어날 때 타입 시스템은 방어 도구에서 설계 도구로 성격이 바뀐다.
