# 그 앱은 웹페이지면 충분했다

원문: [Your ‘App’ Could Have Been a Webpage (so I fixed it for you…) – Dan Q](https://danq.me/2026/07/09/your-app-could-have-been-a-webpage/)

HN 토론: <https://news.ycombinator.com/item?id=48869989> (833점, 500개 댓글)

Lobste.rs 토론: <https://lobste.rs/s/vhbvwc/i_converted_android_app_webpage>

GN 토론: <https://news.hada.io/topic?id=31338>

## 요약

Dan Q는 디즈니랜드 공연 여행의 일정·숙박·교통 정보를 제공하는
Travelbound 앱을 웹페이지로 대체한 과정을 기록했다.
Android Studio의 가상 기기를 루팅하고 HTTP Toolkit으로 앱의
네트워크 트래픽을 중간에서 가로챈 결과,
앱은 `travelbound.api.vamoos.com`의 API 엔드포인트에서 JSON
데이터(여행 일정 배열, 광고 배열, 참조 이미지 배열)를 받아와
화면에 표시할 뿐이었다.

핵심 발견은 이 앱이 "이미 HTML을 만들 수 있는 데이터를 가지고
있으면서 왜 웹페이지가 아닌가"였다.
Dan Q는 Cron으로 주기적으로 실행되는 Ruby 스크립트를 작성해
같은 JSON을 가져와 비밀번호로 보호된 정적 HTML 페이지로
변환했다.
결과물은 복사·인쇄·저장·북마크·검색이 모두 가능한 평범한
웹페이지였고, 거의 모든 기기에서 별도 설치 없이 열렸다.

용량 차이는 극단적이다.
원래 앱은 43MB로 시작해 콘텐츠를 모두 내려받으면 124MB까지
불어나는데, Dan Q가 만든 웹페이지는 0.05MB(이미지를 포함해도
35MB 추가)에 불과했다.
그는 앱이 광고와 추적이라는 "안티 기능"을 위해 존재할 뿐,
사용자에게 실질적인 이점을 주지 않는다고 결론지었다.

## 분석

이 사례가 흥미로운 이유는 기술적 발견 자체보다 그 발견이
드러내는 동기의 비대칭성에 있다.
앱이 처리하는 데이터의 실체는 단순한 JSON 배열이고, 이를 HTML로
변환하는 일은 어려운 작업이 아니다.
그런데도 회사가 네이티브 앱을 고집하는 이유는 사용자 경험을
개선하기 위해서가 아니라, 웹페이지에서는 하기 어렵거나 사용자가
쉽게 차단할 수 있는 일 — 기기 식별자 수집, 백그라운드 추적,
푸시 알림을 통한 재참여 유도 — 을 네이티브 앱 안에서는 훨씬
자유롭게 할 수 있기 때문이다.

Dan Q가 API를 가로채는 데 사용한 방법도 특별한 게 아니다.
가상 기기를 루팅하고 HTTP Toolkit으로 중간자 방식 트래픽 분석을
하는 것은 앱 보안 감사나 리버스 엔지니어링에서 흔히 쓰는 표준
절차다.
이 표준 절차만으로 상용 앱의 실제 데이터 구조를 온전히 재구성할
수 있었다는 것은, 이 앱에 별다른 인증서 고정(certificate
pinning)이나 트래픽 난독화가 없었다는 뜻이기도 하다.
즉 이 앱은 기술적으로 사용자를 보호하려는 노력조차 기울이지
않은 채로, 오직 앱이라는 형식 자체가 주는 이점(추적 용이성)만을
취하고 있었던 셈이다.

HN 댓글은 이 동기의 정체를 더 구체적으로 짚었다.
billyp-rva는 "광고 차단기는 브라우저에서는 작동하지만 앱에서는
작동하지 않는다"는 한 줄로 앱 문화의 핵심 동력을 요약했고[^billyp-rva],
Zak은 한발 더 나아가 "앱 사용자가 웹 사용자보다 7배 더
수익성이 높다"는 통계를 인용하며 회사가 굳이 앱을 고집하는
이유를 뒷받침했다[^zak].
datakan도 "그들이 앱을 원하는 이유는 사용자에게 스팸을 보낼 수
있기 때문"이라며 같은 결론에 도달했다[^datakan].

## 비평

이 글의 설득력은 "43MB vs 0.05MB"라는 극단적인 수치 대비에서
나온다.
다만 이 사례가 모든 앱에 일반화되지는 않는다.
Travelbound처럼 정적 콘텐츠(일정표, PDF, 이미지)를 표시하는
용도로만 쓰이는 앱은 웹페이지로 대체하기 쉬운 극단적인 사례다.
오프라인 접근, 카메라·GPS 같은 기기 API 접근, 복잡한 로컬 상태
관리가 필요한 앱이라면 웹페이지로의 전환이 이렇게 간단하지
않다.
저자도 이 사례가 "문서형 콘텐츠"에 국한된 비판임을 암묵적으로
전제하고 있으며, 이를 일반적인 "앱 vs 웹" 논쟁으로 확장하기에는
사례가 하나뿐이라는 한계가 있다.
HN에서 frankus는 정확히 이 지점을 보완했다.
"2010년 무렵에는 앱이 아니면 안 되는 이유 목록이 꽤 길었지만
지금은 훨씬 짧아졌다"면서도, 블루투스·NFC 같은 하드웨어 접근과
백그라운드 위치 권한 유지처럼 여전히 웹으로는 대체하기 어려운
영역이 남아 있다고 정리했다[^frankus].
wsdn은 반대로 이 사례의 핵심을 간결하게 짚었다.
"일정표 PDF 하나 보여주는 데 124MB짜리 앱이 필요하다면, 그건
UX 문제가 아니라 아키텍처 문제"라는 것이다[^wsdn].

또한 이 방식은 Travelbound사의 API를 사전 동의 없이 역공학해
개인적인 대체 클라이언트를 만든 것이다.
저자 개인의 여행 정보를 자신이 다시 정리해 쓰는 용도로는
무리가 없어 보이지만, 이런 접근을 다른 상용 서비스에
일반화하면 이용 약관 위반이나 API 남용으로 이어질 수 있는
회색 지대에 놓인다는 점도 함께 짚어야 한다.
HN에서도 VladVladikoff가 정확히 이 우려를 제기하며, 네트워크
요청을 가로채 재전송하는 방식이 URL 파라미터를 변경했다는
이유만으로 기소된 과거 사례들과 법적으로 어떻게 다른지
질문했다[^vladvladikoff].
dudeWithAMood는 이미 비슷한 방식을 실제로 적용한 사례를
공유했는데, 그는 Costco 앱이 사용하는 API 요청 두 개만
그대로 활용해 더 나은 웹 버전(97cost.co)을 만들었다고
밝혔다[^dudewithamood].
이는 이 글의 접근이 일회성 사고 실험이 아니라 여러 개발자가
독립적으로 재현하고 있는 패턴임을 보여준다.

## 인사이트

### 앱의 존재 이유가 사용자 편익이 아닐 수 있다

이 사례가 남기는 가장 근본적인 질문은 "이 앱은 누구를 위해
존재하는가"다.
사용자 입장에서 이 앱은 웹페이지보다 못한 경험(더 큰 용량, 느린
로딩, 설치 필요)만 제공했다.
그런데도 앱이 존재하는 이유는 사용자 편익이 아니라 사업자의
데이터 수집·재참여 유도 목적에 있었다.
이는 "왜 이렇게 간단한 정보에 앱씩이나 필요한가"라는 흔한
불평이 단순한 개발 관성이 아니라, 의도적인 설계 선택의 결과일
수 있음을 보여준다.

### 웹은 문서형 콘텐츠에서 여전히 최선의 기본값이다

HTML과 HTTP로 전달되는 문서형 콘텐츠 — 일정표, 뉴스,
가이드북류 정보 — 에는 웹이 여전히 가장 적합한 전달 형식이다.
복사·인쇄·검색·북마크 같은 웹의 기본 기능은 브라우저가 이미
수십 년간 다듬어온 것이고, 이를 네이티브 앱에서 재구현하려면
불필요한 개발 비용이 든다.
"모바일 우선"이 곧 "앱 우선"을 의미하지 않는다는 사실을,
이 사례는 극단적이지만 명확한 형태로 보여준다.

Lobste.rs에서 FRIGN이 공유한 사례는 이 흐름이 거꾸로도
진행되고 있음을 보여준다.
그의 이동통신사는 청구서 조회와 요금제 관리가 가능한 완전한
웹 기반 관리 페이지를 제공하다가, 몇 달 전 이를 없애고 이제는
청구서를 보는 것조차 앱 설치를 강제한다고 밝혔다[^frign].
한때 웹으로 잘 되던 서비스가 다시 앱으로 뒷걸음질 치는 것은
사용성 개선이 아니라 재참여 지표와 데이터 수집 유인이
서비스 형태를 결정한다는 사실을 뒷받침한다.

### Steve Jobs의 아이러니

david_chisnall이 지적한 역사적 아이러니도 곱씹을 만하다.
아이폰 출시 초기 Steve Jobs는 네이티브 SDK 없이 "훌륭한 웹
앱만으로 충분하다"고 주장했는데, 지금은 정반대로 Apple 자신의
앱스토어 심사 절차가 너무 고통스러워진 나머지 개발자들이
웹 앱으로 돌아가고 있다는 것이다[^david_chisnall].
당시 그 주장이 진심이었는지, 아니면 마음에 드는 배포 방식이
아직 없어서 나온 임기응변이었는지는 댓글에서도 논쟁거리였지만,
결과적으로 "웹이면 충분하다"는 주장이 다시 힘을 얻는 이유가
웹 기술의 발전이 아니라 네이티브 앱 생태계의 마찰(심사, 연회비,
사이드로딩 제한) 때문이라는 점은 이 글의 주제와 정확히 맞닿아
있다.
