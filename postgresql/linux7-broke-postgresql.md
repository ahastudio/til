# Linux 7.0이 PostgreSQL을 망가뜨린 방법

원문: <https://read.thecoder.cafe/p/linux-broke-postgresql>

HN 토론: <https://news.ycombinator.com/item?id=47949585> (137점, 70개 댓글)

## 요약

2026년 4월 3일, AWS 엔지니어 Salvatore Dipietro가 심각한 성능 저하를 보고했다.
동일한 96vCPU Graviton4 하드웨어에서 PostgreSQL이 Linux 7.0으로 업그레이드 후
Linux 6.x 대비 약 절반의 처리량만 나왔다. 프로파일링 결과 CPU의 55%가 `StrategyGetBuffer` 내부의 `s_lock` 함수에서 소비되고 있었다.

근본 원인은 Linux 7.0이 현대 아키텍처에서 `PREEMPT_NONE` 스케줄링 옵션을 제거하고
`PREEMPT_LAZY`로 교체한 것이었다.
`PREEMPT_NONE`에서는 커널이 실행 중인 프로세스를 방해하지 않아 락 보유자가 임계 구역을 빠르게 완료할 수 있었다.
반면 `PREEMPT_LAZY`에서는 스케줄러가 페이지 폴트 처리 중에도 프로세스를 선점(preempt)할 수 있어
스핀락(spinlock) 보유 시간이 크게 늘어났다.

PostgreSQL은 120GB 공유 버퍼 풀을 Linux 메모리 페이지로 관리한다.
기본 4KB 페이지 기준으로 약 3,100만 개의 잠재적 페이지가 생기며,
백엔드 프로세스가 처음 접근하는 미매핑 메모리 영역에서 마이너 페이지 폴트가 발생한다.
문제는 버퍼 할당을 보호하는 스핀락을 획득한 채로 페이지 폴트가 발생하면,
커널 폴트 핸들러에 진입하는 동안 락이 유지된다는 점이다.
`PREEMPT_LAZY` 환경에서 이 폴트 중인 프로세스가 선점되면,
수백 개의 다른 백엔드들이 해제되지 않는 락을 기다리며 타이트 루프로 CPU를 소진한다.

성능 저하의 배가 효과는 수학적으로 명확하다.
선점 지연으로 락 보유 시간이 `t`만큼 늘어날 때, 96개 백엔드가 스핀한다면 낭비되는 사이클은 `t × 96`이 된다.
이 승수 효과가 측정 가능한 수준의 저하를 재앙적인 수준으로 변환시켰다.

해결책은 거대 페이지(huge pages)를 활성화하는 것이다.
2MB 거대 페이지를 사용하면 잠재적 폴트 수가 3,100만에서 약 61,000으로,
1GB 거대 페이지를 사용하면 120개로 줄어든다.
`StrategyGetBuffer`가 락을 보유한 채로 폴트를 일으키는 빈도가 극적으로 감소하면서 정상 성능이 복원된다.
PostgreSQL 커뮤니티는 Peter Zijlstra가 제안한 `rseq`(재시작 가능 시퀀스) 기반 대안을 거부했다.

## 분석

### 스핀락이 선점 가능한 커널과 상충하는 이유

사용자 공간 스핀락(userspace spinlock)은 근본적으로 “짧은 임계 구역 동안 다른 스레드가 선점되지 않을 것”이라는
암묵적 가정 하에 설계된다.
커널이 그 가정을 깨면 — 선점을 허용하면 — 스핀락의 성능 특성이 완전히 달라진다.
락 경합 시 스핀락은 뮤텍스(mutex)보다 고성능이지만,
이는 락 보유 시간이 매우 짧다는 조건에서만 성립한다.
`PREEMPT_LAZY`가 이 조건을 무너뜨리면서 PostgreSQL의 스핀락 기반 버퍼 관리자가 반응성을 잃었다.

다만 fulafel은 원문의 기술 표현을 교정한다. `PREEMPT_*` 옵션들은 커널 코드 자체의 선점 가능성을 제어하는 것이지, 유저스페이스 프로세스의 선점을 직접 제어하는 게 아니라는 것이다.[^fulafel] 유저스페이스는 Linux가 선점 멀티태스킹을 도입한 시점부터 항상 선점 가능했다. 이 사건의 메커니즘은 "커널 선점 정책 변경 → 커널 코드 경로 내에서 더 잦은 선점 허용 → 페이지 폴트 처리 중인 락 보유 프로세스가 더 오래 대기"의 구조로 이해하는 것이 더 정확하다.

### 페이지 폴트가 임계 구역에서 발생하는 구조적 취약성

이 문제의 핵심은 단순히 선점 정책 변경이 아니라, PostgreSQL이 락을 보유한 채 페이지 폴트를 일으킬 수 있는 구조에 있다.
고성능 데이터베이스가 사용자 공간 스핀락을 쓰는 이유는 커널 컨텍스트 전환 비용을 피하기 위해서다.
그런데 “커널에 덜 의존한다”는 설계가 “커널 선점 정책에 더 취약해진다”는 역설을 만들어냈다.

거대 페이지 해결책이 효과적인 이유는 폴트 자체를 줄이기 때문이다.
폴트가 일어나지 않으면 락 보유 중 선점될 일이 없다.
이것은 문제를 고치는 게 아니라 문제가 발생하는 조건을 없애는 방식이다.

ahartmetz는 이 설계 문제를 스케줄러 관점에서 포착한다. "`PREEMPT_LAZY`가 페이지 폴트에서 선점을 트리거하는 것은 나쁜 아이디어처럼 보인다. 예측하지 못하게 지연되는 바로 그 순간에 프로세스를 중단시키는 것은 좋지 않다."[^ahartmetz] 페이지 폴트는 보이지 않고 계획할 수 없는 이벤트이기 때문에, 스케줄러가 이 시점을 선점 타이밍으로 선택하는 것은 특히 위험하다. 예측 가능한 시스템 콜과 달리 페이지 폴트는 코드 경로 어디서든 발생할 수 있다.

### 커널 변경의 영향 범위 예측 실패

Linux 커널 관리자들은 `PREEMPT_NONE` 제거를 오랫동안 예고했다고 알려졌다.
하지만 PostgreSQL과 같은 고성능 사용자 공간 애플리케이션에 미칠 영향은 충분히 분석되지 않았다.
이것은 시스템 소프트웨어 생태계에서 반복되는 패턴이다.
커널 변경은 정책적으로 합리적이지만, 수십 년간 특정 커널 동작에 의존해온 애플리케이션들은
예상치 못한 경로로 영향을 받는다.

## 비평

### 강점: 성능 저하의 인과 관계를 명확하게 추적

폴트 수치, 스핀 대기 프로파일, 승수 효과까지 인과 관계를 단계별로 추적하는 설명이 명확하다.
“55% CPU가 `s_lock`에서 소비된다”는 증거에서 출발해 메커니즘을 역방향으로 완전히 추적한 것은
실무 디버깅 방법론의 좋은 예시다.

### 약점: 헤드라인이 책임 소재를 단순화한다

[GeekNews 댓글](https://news.hada.io/topic?id=29142)에서 y15un이 지적하듯,
“Linux 7.0이 PostgreSQL을 망가뜨렸다”는 프레이밍은 책임 소재를 지나치게 단순화한다.
커널 관리자들은 `PREEMPT_NONE` 제거를 오랫동안 권고했으며,
“왜 Linux 7.0에서 PostgreSQL이 느린가”가 더 정확한 표현이라는 주장이다.
이 관점에서 보면, PostgreSQL이 특정 커널 선점 동작에 암묵적으로 의존하는 설계를 유지해온 것이 근본 원인이다.

HN에서도 같은 비판이 제기됐다. MBCook은 "이것은 매우 특이한 설정에서만 발생했다. 헤드라인은 PG 전반이 망가진 것처럼 암시하지만 그렇지 않았다"고 지적했으며,[^MBCook] jeltz 역시 "헤드라인이 사실과 거리가 멀다. 잘못된 설정에서만 성능이 저하됐다"며 수정을 요청했다.[^jeltz] baq의 요약이 이 사건의 실체를 가장 날카롭게 표현한다. "120GB RAM PostgreSQL에서 `hugepages=off`로 테스트했더니 락 경합이 심각에서 재앙으로 바뀌었다. Amazon이 왜 `hugepages=off` 상태로 DB 테스트를 실행했는지 의문이다."[^baq] 이 관점에서 이 사건은 커널 버그가 아니라 비최적 설정 상태에서의 성능 회귀다.

### 약점: `rseq` 거부 결정의 장기적 함의

커뮤니티가 `rseq` 방식을 거부한 이유 중 하나는 크로스 플랫폼 이식성 문제다.
[GeekNews 댓글](https://news.hada.io/topic?id=29142)에서 domuji6이 지적하듯,
`rseq`는 Linux 특화 인터페이스여서 BSD, macOS, Solaris 등 다른 플랫폼에서 동작하지 않는다.
PostgreSQL이 여러 운영 체제를 지원하는 이상, Linux 전용 최적화를 핵심 코드에 도입하는 것은
유지보수 부담을 크게 늘린다.
그러나 Linux가 서버 배포 환경의 사실상 표준인 상황에서 이 결정이 옳은지는 논쟁의 여지가 있다.

HN에서 singron은 원문의 `rseq` 설명이 부정확하다고 교정한다.[^singron] 원문은 `rseq`를 "임계 구역 중 선점됐는지 감지하고 재시작하는" 방식으로 설명했지만, 실제 제안된 메커니즘은 재시작이 아닌 타임슬라이스 연장이다. 스핀락을 보유한 프로세스가 요청 비트를 설정하면, 커널이 해당 스레드를 선점하려 할 때 선점 대신 타임슬라이스를 한 번 연장하여 임계 구역이 완료될 기회를 준다. 이 방식은 재시도 루프보다 오버헤드가 낮지만, 여전히 Linux 특화 유저스페이스 코드를 요구한다는 점에서 이식성 문제는 동일하게 남는다.

### 구조적 딜레마: “유저스페이스를 깨지 말라” vs. “유저스페이스에서 스핀락을 쓰지 말라”

[GeekNews 댓글](https://news.hada.io/topic?id=29142)에서 cafedead가 포착한 이 긴장은
이 사건의 가장 흥미로운 측면이다.
Linus Torvalds의 “유저스페이스 ABI를 깨지 말라” 원칙과
커널 개발자들의 “유저스페이스에서 스핀락을 쓰지 말라” 권고가 충돌했다.
PostgreSQL은 커널 정책 변화 이전에는 명백히 올바르게 동작했고,
두 원칙 중 어느 쪽을 우선시하느냐에 따라 책임의 소재가 달라진다.

## 인사이트

### 성능은 스택 전체의 암묵적 계약 위에 존재한다

이 사건이 보여주는 핵심은 소프트웨어 성능이 가시적인 코드뿐 아니라
운영 체제, 커널, 하드웨어가 맺은 암묵적 계약(implicit contract)의 산물이라는 점이다.
PostgreSQL은 스핀락 코드를 변경하지 않았다. 하지만 그 코드가 의존하던 커널 동작이 바뀌었다.
“커널이 짧은 임계 구역 동안 프로세스를 선점하지 않는다”는 암묵적 가정은
어느 문서에도 명시되지 않았지만, 수십 년간 지켜졌다.

이런 암묵적 계약은 성능이 중요한 모든 시스템 소프트웨어에 존재한다.
어느 레이어의 정책 변경이 어느 레이어의 성능 가정을 파괴할지는
사전에 완벽하게 예측하기 어렵다.
이것이 대형 데이터베이스와 운영 체제 사이에 공식화된 성능 계약이 필요한 이유다.

### 거대 페이지는 임시방편이 아닌 프로덕션 필수 설정이다

많은 PostgreSQL 운영자가 거대 페이지를 선택적 최적화로 취급한다.
이번 사건은 Linux 7.0 이상 환경에서 거대 페이지가 선택이 아닌 필수가 됐음을 보여준다.
TLB 압력 감소, 페이지 폴트 감소, 메모리 관리 오버헤드 감소 등의 이점은
이전에는 “더 나은 성능을 원하면 고려하라”는 권고 수준이었지만,
이제는 “이것 없이 서비스하면 성능이 절반으로 떨어질 수 있다”는 운영 요구사항이 됐다.

거대 페이지의 트레이드오프 — 선행 할당으로 인한 메모리 낭비 가능성 — 는
120GB 공유 버퍼 풀을 운영하는 대규모 데이터베이스에서는 대부분 무시할 수 있는 수준이다.
소규모 인스턴스에서는 여전히 트레이드오프를 고려해야 하지만,
프로덕션 규모에서는 거대 페이지 활성화가 기본값이 되어야 한다.

ozgrakkurt는 더 근본적인 문제를 제기한다. "2026년에도 PostgreSQL이 단순한 설정 파라미터 하나로 1GB 거대 페이지를 할당할 수 없다는 것은 문제다."[^ozgrakkurt] 4KB 페이지의 실질적 비용은 성능 저하뿐이 아니다. 128GB RAM 서버에서 페이지당 16바이트 메타데이터만 가정해도 3,100만 페이지의 메타데이터만으로 500MB 이상이 소모된다. 데이터베이스 워크로드에서 거대 페이지는 선택 사항이 아닌 운영 기본값이어야 하지만, PostgreSQL의 설정 복잡성이 그 채택을 가로막고 있다.

### 커널 업그레이드는 데이터베이스 벤치마크를 포함해야 한다

이 사건의 실용적 교훈은 커널 버전 업그레이드가 반드시 데이터베이스 성능 회귀 테스트를 포함해야 한다는 것이다.
많은 조직이 커널 업그레이드를 OS 수준 변경으로 취급하고,
애플리케이션 레이어 성능 검증을 생략한다.
그러나 `PREEMPT` 정책처럼 시스템 동작의 근본적인 특성을 바꾸는 변경은
데이터베이스, 메시지 큐, 실시간 시스템처럼 스핀락이나 타이트 루프를 사용하는 소프트웨어에
예상치 못한 방식으로 영향을 미친다.

CI/CD 파이프라인에 커널 버전별 벤치마크 매트릭스를 포함하는 것,
그리고 새 커널 릴리즈에 대한 주요 데이터베이스 성능 회귀 자동 탐지 체계는
이런 종류의 사건을 프로덕션 배포 전에 발견할 수 있게 해준다.
