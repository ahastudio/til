# PostgreSQL 19의 프로퍼티 그래프(SQL/PGQ) 이해하기

원문: <https://neovintage.org/posts/postgres-property-graphs/>

GN 토론: <https://news.hada.io/topic?id=31383>

## 소개

PostgreSQL 19의 프로퍼티 그래프 기능(SQL/PGQ)은 기존 테이블을
정점(vertex)과 간선(edge)으로 선언하고, `MATCH`로 고정된 관계
패턴을 검색하는 SQL 표준 확장이다.
핵심은 이 기능이 데이터를 복사하거나 별도 저장소로 옮기는 게
아니라, 기존 외래키 구조 위에 "그래프로 어떻게 읽을지"를
기술하는 선언적 오버레이라는 점이다.

## 문법과 예시

프로퍼티 그래프는 다음과 같이 정의한다.

```sql
CREATE PROPERTY GRAPH f1
  VERTEX TABLES (
    drivers KEY (driver_id) LABEL driver
      PROPERTIES (driver_id, code, nationality),
    results KEY (result_id) LABEL result
      PROPERTIES (result_id, grid, points)
  )
  EDGE TABLES (
    results_driver
      SOURCE KEY (result_id) REFERENCES results
      DESTINATION KEY (driver_id) REFERENCES drivers
      LABEL of_driver
  );
```

정의된 그래프는 `GRAPH_TABLE`과 `MATCH` 절로 조회한다.

```sql
SELECT g.code, g.points
FROM GRAPH_TABLE (f1
  MATCH (d IS driver)<-[IS of_driver]-(res IS result)
  COLUMNS (d.code AS code, res.points AS points)
) AS g;
```

기존 방식이라면 다음과 같이 조인으로 같은 결과를 얻어야 한다.

```sql
SELECT drivers.code, results.points
FROM results
JOIN drivers ON results.driver_id = drivers.driver_id;
```

두 쿼리는 같은 결과를 내지만, `MATCH` 절은 "드라이버 ← 결과"라는
관계를 문장처럼 표현해 조인 조건을 명시적으로 풀어 쓰는 것보다
읽기 쉽다.

## 테이블의 이중 역할

저자가 가장 흥미롭게 짚은 부분은 하나의 테이블이 같은 그래프
안에서 정점이자 간선 역할을 동시에 할 수 있다는 점이다.
예를 들어 `results` 테이블은 그 자체로 "경기 결과"라는 사건을
나타내는 정점이면서, 동시에 별칭을 세 번 사용해 드라이버·레이스·
생산자를 잇는 간선 역할도 한다.
이렇게 하면 별도의 중간 테이블 없이도 팩트 테이블을 그래프의
허브로 삼을 수 있다.

일반적인 매핑 전략은 다음과 같다.

| 테이블 유형 | 그래프 요소 |
| --- | --- |
| 차원 테이블(drivers, races) | 정점 |
| 순수 조인 테이블(student_courses) | 간선 |
| 팩트 테이블(results) | 정점이자 간선 |

## 한계

가장 큰 제약은 가변 길이 경로를 지원하지 않는다는 점이다.
"한 번에서 세 번까지 따라가기" 같은 수량자를 쓰면
`element pattern quantifier is not supported` 오류가 난다.
모든 홉을 명시적으로 나열해야 하므로,
"X에서 Y까지의 최단 경로"나 "N홉 이내에 도달 가능한 모든 정점"
같은 미지의 구조를 찾는 질의는 할 수 없다.
PageRank나 커뮤니티 탐지 같은 그래프 알고리즘도 대상이 아니다.

정점의 `KEY`가 자동으로 `PROPERTIES`에 포함되지 않는다는 점도
실수하기 쉬운 부분이다.
`PROPERTIES` 목록을 명시하지 않으면 모든 열이 프로퍼티로
노출되어 타입 충돌이 발생할 수 있다.

## 분석

이 기능의 본질은 새로운 저장 엔진이나 인덱스 구조가 아니라,
관계형 조인으로 컴파일되는 문법적 설탕(syntactic sugar)이라는
점이다.
기존 쿼리 옵티마이저와 인덱스를 그대로 활용하면서, 고정된
관계 패턴을 조인보다 선언적으로 표현할 수 있게 해준다.
이는 "그래프 데이터베이스가 필요한가"라는 질문에 대해 중요한
답을 시사한다.
많은 조직이 그래프 형태의 질의(친구의 친구, 특정 관계를 거친
경로)를 필요로 하지만, 실제로는 정해진 모양의 패턴만 반복해서
묻는 경우가 대부분이다.
이런 경우라면 Neo4j 같은 전용 그래프 데이터베이스를 새로
도입하는 대신, 이미 쓰고 있는 PostgreSQL에 오버레이만 얹어
해결할 수 있다.

동일 테이블이 정점이자 간선이 될 수 있다는 설계도 실무적으로
의미가 크다.
많은 스키마에서 팩트 테이블(주문, 거래, 경기 결과 등)은 이미
여러 차원 테이블을 잇는 허브 역할을 하고 있다.
이 관계를 억지로 별도의 조인 테이블로 재구성하지 않고도 그래프
관점에서 표현할 수 있다는 것은, 기존 스키마를 거의 그대로
재사용하면서 그래프 질의 능력만 추가할 수 있다는 뜻이다.

## 비평

이 기능의 가장 큰 강점은 "데이터 이동 없음"이라는 원칙이다.
그래프 데이터베이스로의 마이그레이션은 데이터 복제, 동기화 문제,
운영 복잡성 증가를 동반하는 큰 결정인데, SQL/PGQ는 이 결정
자체를 미룰 수 있게 해준다.

다만 가변 길이 경로 미지원이라는 한계는 단순한 기능 부족이
아니라 이 기능의 활용 범위를 근본적으로 제한한다.
그래프 질의가 매력적인 이유의 상당 부분은 "몇 단계를 거쳐야
연결되는지 모르는 상태에서 관계를 탐색"하는 데 있다.
소셜 네트워크의 인맥 거리, 조직도의 상하 관계, 추천 시스템의
연쇄 관계 같은 문제는 정확히 이 가변 길이 탐색이 필요한
영역인데, SQL/PGQ는 이런 질의에는 여전히 재귀 CTE로 돌아가야
한다.
결국 이 기능은 "그래프처럼 보이지만 실제로는 정해진 모양의
조인을 더 읽기 쉽게 쓰는 도구"에 가깝다는 점을 사용자가
분명히 인지해야 한다.

## 인사이트

### 그래프 데이터베이스 도입 전에 확인해야 할 질문

이 글이 주는 실용적인 교훈은 명확하다.
그래프 데이터베이스를 새로 도입하기 전에, 실제로 필요한
질의가 고정된 패턴인지 아니면 가변 길이 탐색인지부터
구분해야 한다는 것이다.
"이 드라이버가 함께 경주한 모든 생산자는?"처럼 패턴이 고정된
질의라면 PostgreSQL 19의 프로퍼티 그래프만으로 충분하다.
반면 "이 사람과 몇 단계 이내로 연결된 모든 사람"처럼 깊이가
정해지지 않은 탐색이 필요하다면 여전히 전용 그래프
데이터베이스나 재귀 CTE가 필요하다.
새로운 기술을 도입하기 전에 기존 도구가 이미 문제를 풀 수
있는지 확인하는 습관은, 그래프 데이터베이스 뿐 아니라 인프라
선택 전반에 적용할 수 있는 원칙이다.

### SQL 표준의 점진적 확장이 만드는 선택지

SQL/PGQ는 SQL:2023 표준에 포함된 기능으로, PostgreSQL 19가
이를 구현한 것이다.
이는 관계형 데이터베이스가 새로운 데이터 모델(그래프, 문서,
시계열)이 유행할 때마다 그 개념을 표준 SQL 문법 안으로
흡수해온 오랜 패턴의 연장선이다.
JSON 지원, 윈도우 함수, 재귀 CTE가 그랬듯, 그래프 질의도
결국 관계형 모델 위의 문법적 계층으로 통합되고 있다.
이런 흐름은 "특정 데이터 모델에 특화된 전용 데이터베이스"와
"모든 것을 흡수하는 범용 관계형 데이터베이스" 사이의 경쟁에서,
후자가 점점 더 많은 영역을 커버하게 되는 이유를 보여준다.
