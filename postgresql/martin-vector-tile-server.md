# Martin

<https://github.com/maplibre/martin>

## 소개

Martin은 MapLibre 조직이 관리하는 오픈소스 벡터 타일 서버다.
Rust로 작성됐으며, PostGIS 데이터베이스와 MBTiles·PMTiles 파일을 실시간으로 벡터 타일로 서빙한다.
Apache-2.0 및 MIT 이중 라이선스로 제공된다.

기존 타일 서버들은 사전 생성(pre-generated) 타일을 캐시에서 제공하는 방식을 사용한다.
Martin은 요청 시점에 PostGIS에서 데이터를 직접 읽어 타일을 생성하므로, 데이터 변경이 즉시 반영된다.
사전 생성 비용 없이 실시간 지도 데이터를 서빙할 수 있다는 것이 핵심 차별점이다.

## 주요 기능

**다중 데이터 소스 통합**

PostgreSQL/PostGIS, MBTiles, PMTiles를 단일 API로 통합 서빙한다.
여러 소스를 결합하거나 개별로 노출하는 설정을 지원한다.

**정적 파일 서빙**

벡터 타일뿐 아니라 스타일 JSON, 스프라이트, 폰트 글리프를 함께 서빙한다.
별도 Nginx나 CDN 설정 없이 Martin 하나로 지도 애플리케이션에 필요한 모든 정적 자산을 제공할 수 있다.

**고성능**

Rust 기반으로 극도로 낮은 메모리 사용량과 높은 처리량을 달성한다.
저사양 서버에서도 대량 요청을 처리할 수 있다.

**MBTiles 도구**

MBTiles 파일의 생성·검사·병합 등 CLI 도구를 내장한다.

## 분석

### PostGIS를 타일 서버로 직접 활용

전통적인 지도 서버 아키텍처는 PostGIS→타일 생성 파이프라인→파일 캐시→서버의 다단계 구조를 갖는다.
Martin은 중간 단계를 제거하고 PostGIS를 직접 타일 소스로 사용한다.
이 설계는 실시간성을 얻는 대신 쿼리 성능에 의존하는 트레이드오프를 갖는다.
적절한 PostGIS 인덱스와 `ST_AsMVT` 최적화가 선행되지 않으면 성능이 크게 떨어진다.

### Rust 선택의 의미

지도 타일 서버는 요청당 계산 비용이 낮아야 하며 동시 연결을 효율적으로 처리해야 한다.
Rust의 제로 코스트 추상화와 비동기 I/O는 이 요구에 잘 맞는다.
GC 멈춤(GC pause)이 없다는 것은 타일 응답 지연의 변동성을 낮춘다는 의미이기도 하다.

### MapLibre 생태계와의 연동

Martin이 MapLibre 조직 산하에 있다는 점은 전략적으로 중요하다.
MapLibre GL JS, MapLibre Native와의 통합이 자연스럽고, Mapbox로부터 독립한 오픈소스 지도 생태계 전체의 인프라 역할을 한다.

## 비평

실시간 타일 생성은 트래픽이 급증할 때 PostGIS에 직접 부하를 준다.
캐싱 레이어 없이 운영하면 데이터베이스가 병목이 된다.
Martin은 이 문제를 해결하기 위해 CDN이나 캐시 프록시와 조합해 사용하도록 권장하지만, 그 설정은 운영자의 몫이다.
“고성능”이라는 표현은 Rust 서버 자체의 성능이지, 전체 스택 성능을 의미하지는 않는다.

PMTiles 지원이 추가됐지만, 대용량 PMTiles 파일의 부분 범위 요청 성능이 MBTiles 대비 어떤지에 대한 벤치마크가 공식 문서에 충분하지 않다.
운영 결정을 내리기 전에 자체 벤치마크가 필요하다.

## 인사이트

### 지도 인프라의 오픈소스화

Mapbox가 2020년 GL JS를 상용 라이선스로 전환한 이후 오픈소스 지도 생태계가 빠르게 성숙하고 있다.
MapLibre, Martin, Protomaps(PMTiles)가 각각 렌더러, 타일 서버, 타일 포맷을 담당하는 완전한 오픈소스 스택이 형성됐다.
이 생태계는 수년 전만 해도 Mapbox나 Google Maps 없이는 불가능했던 기능을 무료로 자기 인프라에서 구현할 수 있게 해준다.

### 실시간성과 사전 생성의 경계

지도 타일의 사전 생성 방식은 정적 데이터에 적합하다.
그러나 실시간으로 변하는 데이터(배달 위치, 교통 흐름, 센서 데이터)를 지도에 표시하려면 Martin 같은 동적 타일 서버가 필요하다.
IoT와 실시간 분석이 확산되면서 이 수요는 증가할 것이고, Martin의 포지셔닝은 그 방향과 일치한다.

### 데이터베이스가 API가 된다

Martin의 접근 방식은 PostGIS 테이블과 함수가 곧 타일 API 엔드포인트가 된다는 의미다.
별도의 타일 생성 서비스나 파이프라인 없이 데이터베이스 스키마 설계만으로 지도 API가 완성된다.
이것은 “데이터베이스가 애플리케이션 레이어”라는 PostgREST나 Supabase의 철학과 같은 방향이다.
지리 데이터를 다루는 조직에서 Martin은 단순한 타일 서버가 아니라 아키텍처 단순화 도구로 기능할 수 있다.
