# PostgreSQL 19 베타 — 전방위 업그레이드

원문: <https://www.snowflake.com/en/blog/engineering/postgresql-19-features-beta/>

HN 토론: <https://news.ycombinator.com/item?id=48733031> (203점, 119개 댓글)

## 요약

Snowflake 소프트웨어 엔지니어링 디렉터 Craig Kerstiens가 2026년 6월 18일에 작성한 글이다.
PostgreSQL 19 베타를 기준으로 주목할 기능들을 정리했으며,
단일 대형 기능이 아닌 다양한 사용자 페르소나를 아우르는 “넓이의 릴리스”라고 평가한다.

`REPACK CONCURRENTLY`는 이번 릴리스의 운영자 측면 하이라이트다.
기존에는 테이블 부풀림(bloat)을 잠금 없이 정리하려면 `pg_repack` 같은 외부 익스텐션에 의존해야 했다.
PostgreSQL 19는 이 기능을 엔진에 내장해 별도 설치 없이 사용할 수 있게 했다.

파티션 관리도 크게 개선됐다.
파티션을 병합하거나 분리하는 기능이 추가돼, 초기에 설계한 파티셔닝 전략이
시간이 지나면서 맞지 않게 됐을 때 재설계할 여지가 생긴다.

논리 복제(logical replication)는 여러 방향에서 성숙해졌다.
시퀀스 값 동기화, `ALL SEQUENCES` 발행, 테이블 선택 제외를 위한 `EXCEPT` 절,
`wal_level = replica`만으로 논리 복제를 자동 활성화하는 기능,
그리고 현재 WAL 레벨을 보고하는 `effective_wal_level`이 추가됐다.

오토베큠(autovacuum)에는 병렬 워커 지원, 유지보수 긴급도 기반 우선순위 채점 시스템,
그리고 이를 확인할 수 있는 `pg_stat_autovacuum_scores` 뷰가 도입됐다.

SQL/Property Graph Queries(SQL/PGQ)는 관계형 데이터에 그래프 스타일 쿼리를 직접 실행할 수 있게 해준다.
저자는 “관계형 모델을 버리라는 것이 아니라, 이미 가진 데이터를 다른 방식으로 쿼리하는 수단을 추가하는 것”이라고 강조한다.

개발자 편의 기능도 늘었다.
`GROUP BY ALL` 구문, 윈도우 함수의 `IGNORE NULLS` / `RESPECT NULLS` 지원,
`INSERT ... ON CONFLICT DO SELECT ... RETURNING` 형태의 개선된 업서트(upsert),
그리고 `UPDATE`와 `DELETE`에 `FOR PORTION OF`를 지원하는 템포럴 쿼리 개선이 포함됐다.

성능 측면에서는 플래너의 안티조인·세미조인 처리와 집계 전 상수 폴딩 개선,
CSV 파싱을 위한 SIMD 명령어 지원, 기수 정렬(radix sort) 개선이 이뤄졌다.

## 분석

### “넓이의 릴리스”라는 포지셔닝

저자는 PostgreSQL 19를 단일 킬러 기능이 아닌 다층 개선의 릴리스로 규정한다.
애플리케이션 개발자, 운영자, 성능을 중시하는 DBA, PostgreSQL 위에 제품을 만드는 팀 모두에게
“해당하는 것”이 있다고 서술한다.
이 포지셔닝은 단순한 홍보 문구가 아니라 PostgreSQL 개발 모델의 구조적 특성을 반영한다.

PostgreSQL은 단일 기업이 아닌 커뮤니티가 개발한다.
기능 우선순위는 개별 기여자와 후원 기업의 관심, 패치 검토 대역폭에 따라 결정된다.
따라서 특정 분기에 특정 분야에 집중하는 “테마 릴리스”가 나오기 어렵고,
자연스럽게 다방면의 점진적 개선이 동시에 쌓이는 패턴이 만들어진다.
저자가 “넓이”라고 표현한 것은 결과이지 설계 방침이 아니다.

### REPACK CONCURRENTLY의 생태계 의미

`REPACK CONCURRENTLY`는 기능 자체보다 그것이 상징하는 이행(migration)이 중요하다.
`pg_repack`은 수년 동안 PostgreSQL 운영의 필수 도구였다.
네이티브 지원이 추가된다고 해서 `pg_repack`이 즉시 사라지지는 않겠지만,
“핵심 운영 요구를 익스텐션으로 채워야 하는” 상황이 하나 줄어드는 것은 분명하다.

비슷한 사례가 여럿 있다.
파티션 테이블 관리, 논리 복제, 병렬 쿼리는 모두 처음에는 서드파티 해법이 먼저 등장했고,
나중에 엔진에 흡수됐다.
PostgreSQL은 성숙한 익스텐션을 선별해 코어로 올리는 경로를 반복적으로 밟는다.
이 경로는 기술적 검증 비용을 커뮤니티에 분산하는 효율적인 방식이기도 하다.

### SQL/PGQ: 관계형 모델 위에 그래프 올리기

SQL/PGQ는 방향성이 흥미롭다.
그래프 데이터베이스 시장에서 Neo4j, Amazon Neptune, Tigergraph가 성장하고 있는 상황에서,
PostgreSQL은 별도의 그래프 DB를 도입하지 않고도 그래프 쿼리를 실행할 경로를 제공한다.
저자가 “관계형 모델을 버리지 않아도 된다”고 강조한 것은 전환 비용을 낮추겠다는 메시지다.

ISO SQL:2023에 포함된 SQL/PGQ 표준을 PostgreSQL이 구현한다는 점도 중요하다.
표준 기반 구현은 특정 벤더 종속을 피하면서 그래프 쿼리를 작성할 수 있다는 의미다.
다만 Cypher나 Gremlin처럼 그래프 전용 언어에 익숙한 개발자에게는
SQL/PGQ 구문이 상대적으로 낯설게 느껴질 수 있다.

HN에서 mickeyp는 SQL/PGQ 구문을 “끔찍하다”고 표현하며, Neo4j를 닮은 구문을 2026년에 굳이 채택한 것을 비판했다.[^mickeyp]
이에 hobofan은 SQL/PGQ가 PostgreSQL이 독자적으로 고안한 구문이 아니라
Neo4j의 Cypher 언어에서 파생돼 ISO SQL 표준에 편입된 것임을 지적했다.[^hobofan]
구문의 낯섦은 PostgreSQL의 선택이 아니라 표준화 과정의 결과라는 반론이다.

### 논리 복제 성숙의 맥락

논리 복제 개선은 단독으로 보면 작은 변화처럼 보이지만,
PostgreSQL을 중심으로 한 데이터 파이프라인 아키텍처의 가능성을 넓힌다.
시퀀스 동기화는 그동안 논리 복제의 실질적인 제약이었다.
PK 생성에 시퀀스를 쓰는 테이블을 복제할 때 시퀀스 값이 동기화되지 않으면
페일오버 후 충돌이 발생하는 문제가 있었기 때문이다.

`wal_level = replica`만으로 논리 복제를 활성화할 수 있게 된 것은 운영 단순화의 신호다.
기존에는 `logical`로 설정해야 했고, 이는 WAL 크기 증가를 수반했다.
이 변경이 실제로 WAL 오버헤드를 줄이는지, 아니면 동작 방식만 바뀌는 것인지는
릴리스 노트 전문을 확인해야 한다.

## 비평

### 베타 시점의 “전방위 소개”는 깊이를 희생한다

저자는 다양한 기능을 한 글에 담았지만, 각 기능을 피상적으로만 소개하는 데 그친다.
`REPACK CONCURRENTLY`는 “long-standing production need”를 해결한다고 했지만,
`pg_repack`과 비교해 어떤 제약이 있는지, 대용량 테이블에서 어느 정도 성능 차이가 나는지는 없다.
SQL/PGQ는 “그래프 스타일 쿼리”라고 했지만 실제 구문 예시나 사용 제약도 없다.

이 글은 “PostgreSQL 19를 주목해야 한다”는 관심 유발이 목적이지,
엔지니어가 실제로 기능을 평가할 수 있는 기술 글이 아니다.
Snowflake 엔지니어링 블로그라는 출처를 고려하면 이는 자연스러운 선택이지만,
독자가 이 글만으로 프로덕션 도입 여부를 결정할 수 없다는 사실은 명시되지 않는다.

HN 댓글에서 mkurz는 이 글이 PostgreSQL 19의 가장 주목받는 기능 중 하나인
네이티브 애플리케이션 타임 템포럴 테이블 지원을 아예 언급하지 않았다는 점을 지적했다.[^mkurz]
SQL:2011 표준 기반의 이 기능은 커뮤니티에서 별도 스레드를 만들 만큼 관심을 받았는데,
이 글의 기능 선택 기준이 PostgreSQL 생태계 전체가 아닌 Snowflake의 관심사에 편향됐을 가능성을 보여준다.

### 오토베큠 개선의 불확실성을 건너뜀

오토베큠의 채점 시스템 도입은 중요한 변화다.
기존 오토베큠은 데드 튜플 비율이나 테이블 크기 같은 간단한 임계값 기반으로 작동했다.
우선순위 채점 시스템이 도입되면 기존 `autovacuum_*` 파라미터 튜닝 경험이
그대로 통하지 않을 수 있다.

저자는 이 변화를 “향상된 가시성”이라고만 언급하고 넘어간다.
하지만 오토베큠 동작의 변화는 예상치 못한 부작용을 낳을 수 있다.
특정 테이블의 진공이 이전보다 늦게 실행되거나,
채점 로직이 특정 워크로드 패턴에서 오진하는 시나리오는 충분히 가능하다.
베타 단계에서 이 부분을 주의 깊게 검증해야 한다는 안내가 없다.

### 기능 목록이 릴리스의 의미를 가리킨다

이 글의 구조는 “기능 A, 기능 B, 기능 C…”의 나열이다.
각 기능이 PostgreSQL의 장기 방향과 어떻게 연결되는지에 대한 통찰은 빠져 있다.
예를 들어 SQL/PGQ는 그 자체로도 흥미롭지만,
PostgreSQL이 멀티모델 데이터베이스로 나아가는 추세의 일부로 보면 더 넓은 맥락에서 이해된다.
`REPACK CONCURRENTLY`는 익스텐션 생태계를 코어로 흡수하는 패턴의 반복이며,
이 패턴이 향후 어떤 익스텐션을 흡수할지 예측할 실마리가 된다.

릴리스 하이라이트 형식의 글이 이런 깊이를 다루지 않는 것은 이해할 수 있지만,
독자 입장에서는 표면만 보고 릴리스 전체를 이해했다고 착각하기 쉽다.

HN 커뮤니티는 이 글이 다루지 않은 PostgreSQL의 오래된 숙제들도 다시 꺼냈다.
aljgz는 연결당 메모리 비용이 여전히 높고, SQL Server의 인덱스 뷰(indexed view)처럼
동기적으로 갱신되는 구체화 뷰가 없다는 점을 꼽았다.[^aljgz]
ktosobcy는 Docker 환경에서 주요 버전 간 인플레이스(in-place) 업그레이드가 불가능한 것이
가장 큰 운영 고통이라고 지목했다.[^ktosobcy]
이런 반응은 “넓이의 릴리스”가 커버하지 못한 영역이 여전히 존재한다는 것을 보여준다.

## 인사이트

### PostgreSQL의 “익스텐션 → 코어” 경로는 생태계 전략이다

`REPACK CONCURRENTLY`의 코어 편입은 단발 사건이 아니라 반복되는 패턴이다.
`pg_repack`, `pg_partman`, 논리 복제의 초기 구현체들은 모두 익스텐션으로 먼저 등장했고,
그 중 일부가 PostgreSQL 코어로 올라왔다.

이 경로는 의도된 생태계 전략으로 볼 수 있다.
익스텐션 생태계는 커뮤니티가 낮은 비용으로 다양한 아이디어를 실험하는 공간이다.
성숙한 익스텐션은 실제 사용자 기반과 검증된 API를 가지고 코어 편입 논의를 시작할 수 있다.
코어 팀 입장에서는 “사용자가 이미 쓰고 있는 것”을 표준화하는 것이
완전히 새로운 기능을 설계하는 것보다 리스크가 낮다.

이 관점에서 보면 현재 활발히 사용되는 익스텐션들 — `pg_cron`, `timescaledb`의 일부 기능,
`pgvector` 등 — 이 향후 릴리스에서 코어에 흡수될 후보가 될 수 있다.
특히 `pgvector`는 AI 워크로드와의 연계로 PostgreSQL 공식 지원 가능성이 거론되기 시작했다.

한편 ethagnawl은 `GROUP BY ALL` 같은 기능이 DuckDB에서 먼저 도입된 개념임을 지적했다.[^ethagnawl]
DuckDB의 “친화적 SQL(friendly SQL)” 방언이 여러 데이터베이스 생태계에 영향을 미치고 있으며,
이는 익스텐션 생태계뿐만 아니라 외부 DB의 혁신도 PostgreSQL 발전에 흡수되는 경로가 있음을 보여준다.

### SQL/PGQ는 그래프 DB 시장 압력의 우회 전략이다

SQL/PGQ는 기술 기능이지만, 동시에 시장 포지셔닝이기도 하다.
그래프 데이터베이스 시장은 Neo4j, AWS Neptune, ArangoDB 등의 성장으로 주목받고 있다.
많은 팀이 그래프 탐색 쿼리가 필요할 때 PostgreSQL에서 다른 DB로 이전하는 선택을 해왔다.

SQL/PGQ는 그 이전 이유를 하나 제거한다.
그래프 쿼리가 필요하다는 이유만으로 PostgreSQL을 포기하지 않아도 되는 상황을 만든다.
이 전략은 PostgreSQL이 멀티모델 DB로 진화하는 방향과 일치한다.
이미 JSON 지원(`jsonb`)으로 문서 DB 사용 사례 일부를 흡수했고,
`pgvector`로 벡터 검색을 포용하기 시작했다.
그래프 쿼리는 이 흐름의 다음 단계다.

다만 전용 그래프 DB를 완전히 대체하기는 어렵다.
그래프 특화 스토리지 엔진, 그래프 알고리즘 라이브러리, 시각화 도구 등에서
Neo4j 같은 전문 시스템과 경쟁하는 것은 SQL/PGQ의 목표가 아니다.
“그래프 DB가 필요할 만큼 복잡하지 않은 그래프 쿼리”를 PostgreSQL 안에서 해결하는 것이 현실적인 범위다.

### 오토베큠 지능화는 “보이지 않는 DBA”의 시작이다

오토베큠 개선의 방향은 단순한 임계값 튜닝에서 적응형 우선순위 결정으로 이동하고 있다.
이 변화가 작아 보이지만, 더 큰 추세의 시작점이 될 수 있다.

전통적인 PostgreSQL 튜닝은 DBA가 `autovacuum_vacuum_scale_factor`,
`autovacuum_cost_delay` 같은 파라미터를 워크로드에 맞게 수동으로 조정하는 방식이었다.
채점 시스템의 도입은 PostgreSQL이 스스로 유지보수 긴급도를 판단하는 능력을 키우는 것이다.
장기적으로 이 방향은 자기 튜닝(self-tuning) 데이터베이스로 나아가는 경로의 초기 단계와 닮아 있다.

클라우드 환경에서 이 변화는 특히 의미 있다.
Snowflake, RDS, Cloud SQL처럼 PostgreSQL을 관리형 서비스로 제공하는 사업자들은
개별 고객의 DB를 일일이 튜닝하기 어렵다.
오토베큠이 스스로 더 나은 결정을 내릴수록, 관리형 서비스의 운영 부담이 줄어들고
성능 안정성이 높아진다.
Snowflake가 이 기능을 하이라이트로 꼽은 데는 그들의 운영 맥락이 반영된 것이다.

### 파티션 유연성은 초기 설계 실수의 비용을 낮춘다

파티션 병합과 분리 기능은 기술적 편의 이상의 의미가 있다.
데이터베이스 파티셔닝은 초기 설계가 중요한 결정이다.
시간 기반으로 월별 파티션을 만들었는데 데이터가 폭발적으로 증가하면 일별로 바꿔야 하고,
반대로 너무 잘게 쪼갰다가 파티션 수가 부담이 되면 합쳐야 한다.

기존에는 이 재설계가 매우 고통스러웠다.
데이터를 새 파티션 구조로 이동하는 대규모 마이그레이션이 필요했고,
그 과정에서 다운타임이나 서비스 성능 저하가 불가피했다.
파티션 분리와 병합이 네이티브로 지원되면, 이 재설계의 비용이 크게 낮아진다.

이 기능은 “처음부터 완벽하게 설계해야 한다”는 압박을 줄인다.
초기에 합리적인 판단으로 시작하고, 실제 사용 패턴을 보면서 점진적으로 조정하는 접근이
기술적으로 더 현실적이 된다.
소프트웨어 설계의 “작은 결정, 빠른 피드백” 원칙이 데이터베이스 스키마에도 적용될 여지가 생기는 것이다.

### 컬럼형 스토리지 부재는 PostgreSQL의 구조적 한계다

HN에서 sega_sai는 15년 이상 PostgreSQL을 사용한 경험자로서,
데이터셋이 커질수록 컬럼형(columnar) 스토리지가 없는 것이 점점 더 큰 제약이 된다고 지적했다.[^sega_sai]
이에 대해 tyre는 “컬럼형 DB는 별도 범주”라며 “Apple이 세탁기를 팔길 바라는 것과 같다”고 반박했다.[^tyre]

두 반응 모두 PostgreSQL의 정체성 논쟁을 드러낸다.
PostgreSQL이 `jsonb`, `pgvector`, SQL/PGQ로 멀티모델 방향을 걷고 있다면,
컬럼형 스토리지 부재는 “분석 워크로드가 필요하면 다른 DB로 가야 한다”는 경계를 남기는 것이다.
DuckDB, ClickHouse, Redshift가 분석 영역을 차지하고 있는 상황에서,
PostgreSQL이 OLTP와 OLAP의 경계를 어디까지 허물 의지가 있는지는 중장기 방향성의 핵심 질문이다.
컬럼형 스토리지 익스텐션(Cetus, Hydra 등)이 이미 존재하지만,
이것들이 코어에 흡수되는 경로를 밟을 것인지는 아직 불분명하다.
