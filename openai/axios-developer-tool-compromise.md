# OpenAI의 axios 공급망 공격 대응: macOS 서명 파이프라인 침해

원문: <https://openai.com/index/axios-developer-tool-compromise/>

관련 문서: [axios npm 공급망 공격 분석 (공격 자체)](../security/axios-npm-compromise.md)

## 요약

2026년 3월 31일(UTC), OpenAI의 macOS 앱 서명 GitHub Actions 워크플로가
악성 axios 버전 1.14.1을 자동으로 내려받아 실행했다.
이 워크플로는 ChatGPT Desktop, Codex, Codex CLI, Atlas의
macOS 앱 서명에 사용되는 인증서와 공증(notarization) 자료에 접근 권한을 갖고 있었다.

OpenAI는 제3자 디지털 포렌식 및 인시던트 대응(DFIR) 업체를 참여시켜 조사한 결과,
사용자 데이터 접근, 시스템 또는 지적 재산 침해, 소프트웨어 변조의 증거를 발견하지 못했다.
페이로드 실행 타이밍, 인증서의 작업 주입 순서, 작업 자체의 실행 순서 등
여러 완화 요인으로 인해 서명 인증서가 실제로 탈취되지 않았을 가능성이 높다고 판단했다.

그러나 OpenAI는 "풍부한 주의(abundance of caution)" 원칙에 따라
인증서가 침해된 것으로 간주하고 폐기·교체를 단행했다.
새로운 인증서로 서명된 신규 빌드가 모든 관련 macOS 제품에 배포되었다.
2026년 5월 8일부터 이전 버전의 macOS 앱은 업데이트·지원이 중단되고
macOS의 취소된 인증서 차단 정책으로 인해 기능이 멈출 수 있다.
Apple과의 협력을 통해 이전 인증서로의 신규 공증도 차단했다.

이 사건의 근본 원인은 워크플로의 CI/CD 설정 미비였다.
문제의 액션은 특정 커밋 해시 대신 부동 태그(floating tag)를 사용했고,
신규 패키지에 대한 최소 릴리스 대기 시간(minimumReleaseAge)도 설정되지 않았다.
이로 인해 침해된 의존성이 런타임에 자동으로 내려받아 실행될 수 있었다.

## 분석

### 공급망 공격의 2차 피해 구조

이 사건은 npm 공급망 공격이 직접 피해자(npm 의존 개발자)에 그치지 않고,
그 도구를 CI/CD 파이프라인에서 사용하는 조직까지 연쇄적으로 침투하는
2차 피해 구조를 명확히 보여준다.
OpenAI는 axios의 직접 사용자가 아니라,
macOS 앱 서명 과정에서 axios를 간접 의존하는 워크플로를 운영하는 조직이었다.
공격자가 npm 생태계 하나를 오염시키면,
그 파급 효과는 개발자 로컬 환경을 넘어 소프트웨어 배포 파이프라인 전체로 확산된다.

### CI/CD 파이프라인의 구조적 취약점

```
# 취약한 설정 (floating tag 사용)
uses: some-action@v2

# 안전한 설정 (고정 커밋 해시)
uses: some-action@a1b2c3d4e5f6...
```

부동 태그 사용은 편의성을 위한 선택이지만,
태그가 가리키는 커밋이 언제든 변경·삭제될 수 있어
재현 가능한 빌드와 보안 모두를 손상시킨다.
minimumReleaseAge 설정 부재는 악성 버전이 배포된 직후 즉시 채택되는 것을 막지 못했다.
신규 패키지 버전이 검증되기 전에 자동 채택되는 구조는
공급망 공격의 '속도 이점'을 공격자에게 그대로 허용한다.

### UNC1069와 북한 위협 행위자

BleepingComputer 등의 보도에 따르면 이번 공격의 배후로
북한 연계 위협 그룹 UNC1069가 지목된다.
가짜 협업 플랫폼을 통한 소셜 엔지니어링으로 axios 메인테이너를 공략하고,
멀웨어 설치 → 자격 증명 탈취 → 하류 공급망 침해로 이어지는
체계적인 공격 체인이 확인되었다.
국가 지원 위협 행위자가 오픈소스 생태계의 메인테이너를 표적으로 삼는
전략적 공급망 공격이 실제로 실행되고 있음을 확인시켜준다.

### 코드 서명 인증서의 위험 반경

코드 서명 인증서가 탈취될 경우의 위험은 단순한 데이터 유출을 훨씬 넘어선다.
공격자가 정당한 서명 인증서로 악성 소프트웨어를 서명하면,
운영체제의 코드 서명 검증 체계 전체를 우회할 수 있다.
macOS Gatekeeper, Windows Authenticode 등
OS 보안의 핵심 메커니즘이 무력화되며,
"OpenAI가 서명한 신뢰할 수 있는 소프트웨어"처럼 보이는 악성 프로그램을
수억 명의 사용자에게 배포할 수 있게 된다.
OpenAI가 증거 부재에도 불구하고 인증서를 폐기한 결정은
이 위험을 정확히 이해한 판단이다.

## 비평

### 강점

OpenAI의 사후 공시는 투명성 측면에서 평가할 만하다.
DFIR 업체 참여, 조사 결과의 구체적 공개, 완화 요인 설명,
사용자 조치 요구(업데이트) 공지까지 표준적인 인시던트 대응 절차를 밟았다.
많은 기업이 공급망을 통한 간접 피해를 쉬쉬하거나 축소하는 경향과 비교할 때,
이 수준의 공개는 업계 기준을 높이는 데 기여한다.

### 약점

사건의 근본 원인인 CI/CD 설정 미비—부동 태그, minimumReleaseAge 부재—는
2024~2025년에 이미 업계에서 반복적으로 지적된 알려진 취약점이다.
세계 최대 AI 기업 중 하나인 OpenAI의 코드 서명 파이프라인에서
이런 기초적 설정 오류가 존재했다는 사실은 의외다.
"알고는 있었지만 적용하지 않았다"면 내부 보안 통제의 실행력 문제이고,
"알지 못했다"면 보안 인식의 문제다. 어느 쪽이든 공개 사과와 다른 수준의
내부 개선이 필요한 지점이다.

2026년 5월 8일 강제 업데이트 마감도 논란의 여지가 있다.
사건 발생(3월 31일)부터 마감(5월 8일)까지 약 5주의 시간을 주었지만,
기업 환경에서 5주 안에 macOS 앱을 대규모로 업데이트하는 것은
모바일 기기 관리(MDM) 정책, 호환성 테스트, 변경 승인 프로세스 등
현실적인 제약으로 어려울 수 있다.

인증서 탈취가 "이루어지지 않았을 가능성이 높다"는 표현도
완전한 안심을 주지 못한다. 포렌식 조사로 확인할 수 없는 경로를 통해
인증서가 유출되었을 가능성을 완전히 배제할 수 없으며,
OpenAI 스스로도 이를 인정하기에 인증서를 교체한 것이다.

## 인사이트

### 서명 파이프라인은 왕관의 보석이다

코드 서명 인증서가 들어있는 CI/CD 파이프라인은
"사용자에게 도달하는 소프트웨어를 합법적으로 보이게 만들 수 있는 열쇠"다.
이 파이프라인은 일반 개발 파이프라인보다 훨씬 강력한 격리와 통제가 필요하다.
환경 변수로 인증서를 주입하고, 최소 권한 원칙을 적용하고,
외부 의존성을 최소화하거나 사전 검증된 미러에서만 내려받는 구조가 되어야 한다.
이번 사건은 서명 파이프라인이 일반 빌드 파이프라인과 동일한 방식으로 관리되었을 때
어떤 결과가 오는지를 실증했다.

보안 설계에서 "가장 강력한 자원이 가장 강력하게 보호되어야 한다"는 원칙은
당연해 보이지만, 실제 구현에서는 자주 위반된다.
코드 서명 환경에는 외부 의존성 자동 업데이트를 비활성화하고,
사용되는 모든 외부 코드의 해시를 고정하며,
파이프라인 변경 자체를 별도의 엄격한 검토 프로세스로 통제해야 한다.
프로덕션 환경과 마찬가지로 서명 환경도 "변경 요청 → 검토 → 승인 → 적용"의
체계가 필요하다.

### 속도와 보안의 등가 교환은 더 이상 작동하지 않는다

부동 태그와 자동 의존성 업데이트는 "최신 보안 패치를 자동으로 적용"하기 위한
선의의 설계였다. CI/CD 자동화 초기에는 "빠른 패치 적용"이 보안에 도움이 된다는
논리가 설득력이 있었다.
그러나 공급망 공격이 일반화된 환경에서는 이 논리가 역전된다.
자동 업데이트는 "보안 패치를 빠르게"와 함께 "악성 버전을 빠르게"도 적용하기 때문이다.

이 역설은 "최신 버전 = 더 안전"이라는 단순한 가정이 무너졌음을 의미한다.
이제는 "검증된 버전 = 더 안전"으로 패러다임이 이동해야 한다.
커밋 해시 고정, 릴리스 대기 기간, SBOM 기반 연속 모니터링이
속도 이득을 일부 희생하더라도 도입해야 하는 기본 통제가 되었다.
Socket.dev의 minimumReleaseAge 같은 도구가 이 간극을 메우려 하지만,
이를 강제하지 않는 한 "편의"는 언제나 "보안"을 이긴다.

### 국가 지원 공격자와 오픈소스 메인테이너의 비대칭 전쟁

북한 연계 UNC1069가 메인테이너를 표적으로 삼아 가짜 협업 플랫폼으로 접근하고,
자격 증명을 탈취하여 npm 생태계를 오염시킨 이 공격 체인은
오픈소스 공급망이 국가 수준 위협의 표적이 되었음을 명확히 선언한다.
한쪽에는 국가가 지원하는 정보기관급 자원과 전문성,
다른 쪽에는 무급 또는 소액 보상으로 자유 시간에 오픈소스를 유지하는 개인 메인테이너가 있다.
이 비대칭은 소셜 엔지니어링에 대한 개인의 저항력으로 해결될 수 없다.

SolarWinds, XZ Utils 백도어, 이번 axios 사건은 모두 같은 패턴을 가리킨다:
신뢰 사슬의 상류를 장악하면 하류 전체가 오염된다.
방어는 개인의 주의력이 아닌 구조적 통제로만 가능하다.
레지스트리 수준의 행위 분석, 메인테이너 계정 이상 탐지, 패키지 서명 의무화,
독립적인 재현 빌드(reproducible builds) 검증이 레지스트리 기반시설에 내장되어야 한다.
오픈소스 생태계를 국가 기반시설로 간주하고, 그에 걸맞은 보안 투자가 이루어져야 할 시점이다.
