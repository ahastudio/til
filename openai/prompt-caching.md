# OpenAI Prompt Caching 가이드

<https://developers.openai.com/api/docs/guides/prompt-caching>

## 요약

OpenAI의 프롬프트 캐싱(Prompt Caching)은 동일한 프롬프트 접두사를 반복
사용하는 요청의 레이턴시를 최대 80%, 입력 토큰 비용을 최대 90% 절감하는
기능이다. `gpt-4o` 이상의 최신 모델에서 코드 변경 없이 자동으로 활성화된다.

동작 원리는 프롬프트 초반부(보통 첫 256 토큰)의 해시값을 기반으로 요청을
특정 서버로 라우팅하는 방식이다. 해당 서버에 동일한 접두사가 캐시되어 있으면
전체 프롬프트를 재처리하지 않고 캐시된 결과를 활용한다. 캐싱은 1024 토큰
이상의 프롬프트에서 자동으로 활성화되며, 모든 응답에는 `cached_tokens` 필드가
포함되어 캐시 적중 현황을 확인할 수 있다.

캐싱 대상은 시스템/사용자/어시스턴트 메시지 배열 전체, 사용자 메시지 내
이미지(링크 및 base64 인코딩), 도구 정의, 구조화된 출력 스키마를 포함한다.
캐시는 조직 단위로 격리되어 동일 조직 내에서만 공유된다.

보존 정책은 모델에 따라 다르다. 대부분의 모델은 비활성 후 5~10분, 최대 1시간
동안 메모리에 캐시를 유지한다. `gpt-5.5`, `gpt-5.5-pro`, `gpt-5.4`,
`gpt-5.2`는 24시간 연장 보존(Extended Retention)을 지원하며, 이 경우 GPU
로컬 스토리지에 Key/Value 텐서 표현만 저장된다. 가격은 메모리 캐싱과 동일하다.

캐시 적중률을 높이려면 프롬프트 구조에서 시스템 지침, 예시, 공통 가이드라인
같은 정적 내용을 앞에, 사용자별 가변 내용을 뒤에 배치해야 한다. 선택적
파라미터인 `prompt_cache_key`를 사용하면 라우팅에 영향을 주어 적중률을 높일 수
있으나, 동일한 접두사-키 조합은 분당 약 15회 이하로 유지해야 한다.

## 분석

### 인프라 설계의 암묵적 노출

이 문서는 캐싱이 “특정 서버로의 라우팅”을 통해 구현된다는 사실을 명시한다.
즉, 프롬프트 해시가 서버 어피니티(server affinity)의 키로 사용된다. 이는
단순한 메모리 캐시가 아니라, 요청을 적합한 서버로 보내는 스마트 로드밸런서에
가깝다. `prompt_cache_key`가 “분당 15회 이하”라는 제약을 갖는 이유도 이 때문
이다. 과도한 캐시 집중은 특정 서버에 부하를 만들고 캐시를 밀어낸다.

### 24시간 연장 보존의 의미

대부분의 캐시가 5~10분 내 만료되는 것과 달리, 최신 GPT-5 계열은 24시간
보존을 지원한다. 이는 장기 세션, 야간 배치 작업, 다음 날 동일 컨텍스트를
재사용하는 시나리오를 가능하게 한다. “Key/Value 텐서만 저장하고 고객 콘텐츠는
저장하지 않는다”는 설명은 보안 및 개인정보 보호 측면에서 중요한 아키텍처
결정이다. 실제 텍스트가 아닌 신경망 중간 계산 결과를 캐싱한다는 의미이기
때문이다.

### Anthropic Claude 캐싱 방식과의 차이

Anthropic Claude의 프롬프트 캐싱은 명시적으로 `cache_control` 마커를 삽입해
캐싱 위치를 지정하는 방식이다. OpenAI 방식은 별도 코드 변경 없이 자동으로
작동한다. 개발자 경험(DX) 측면에서는 OpenAI가 더 간단하지만, 캐싱 경계를
세밀하게 제어하려는 고급 사용자에게는 Claude의 명시적 방식이 예측 가능성이
높다. 두 접근 방식은 “편의성 vs. 제어권”의 전형적인 트레이드오프다.

## 비평

### 강점: 제로 코드 변경, 즉시 효과

“코드 변경 없이 자동으로 작동한다”는 점은 도입 장벽이 거의 없다는 뜻이다.
이미 `gpt-4o` 이상을 사용하는 모든 애플리케이션이 자동으로 혜택을 받는다.
비용 절감 90%라는 수치는 시스템 프롬프트가 긴 RAG 파이프라인, 코드 편집기,
고객 지원 봇 같은 반복적 요청이 많은 워크로드에서 즉각적이고 실질적인 효과를
낸다.

### 약점: 캐시 만료의 불확실성

“5~10분 비활성 후 만료”라는 정책은 요청 간격이 불규칙한 워크로드에서는
캐시 적중률을 예측하기 어렵게 만든다. 특히 야간이나 트래픽이 낮은 시간대에
캐시가 만료되어 첫 요청에 풀 비용이 청구될 수 있다. 연장 보존 모델(`gpt-5.5`
이상)은 이 문제를 해결하지만, 하위 모델 사용자에게는 여전히 예측 불가능한
비용 변동 요인이 된다.

### 약점: `prompt_cache_key` 제약의 불명확성

“분당 약 15회 이하”라는 제약이 왜 존재하는지, 초과하면 어떤 일이 벌어지는지에
대한 설명이 부족하다. 초과 시 에러가 발생하는지, 아니면 단순히 캐시 미적중으로
처리되는지가 불분명하다. 프로덕션 환경에서 이 파라미터를 사용하는 개발자에게
는 부족한 정보다.

## 인사이트

### 캐싱은 “비용 절감”이 아니라 “아키텍처 결정”이다

프롬프트 캐싱의 효과를 극대화하려면 정적 콘텐츠를 앞에, 동적 콘텐츠를 뒤에
배치해야 한다. 이는 단순한 최적화 팁이 아니라, 애플리케이션의 프롬프트 구조
전체를 재설계하도록 유도하는 원칙이다. 코드베이스에서 “어디까지가 불변이고
어디부터가 가변인가”를 명확히 구분하는 작업은 프롬프트 엔지니어링의 핵심
아키텍처 결정이 된다.

이 원칙은 소프트웨어 설계의 고전적 원칙인 관심사 분리(Separation of Concerns)
와 정확히 맞닿아 있다. 시스템 지침, Few-shot 예시, 도구 스키마는 캐싱 레이어에
속하고, 사용자 쿼리와 컨텍스트는 동적 레이어에 속한다. 이 구분이 명확할수록
캐싱 효율이 높아지고, 애플리케이션 구조도 명확해진다.

장기적으로 이 아키텍처 패턴은 LLM 애플리케이션의 표준 설계 패턴으로 자리
잡을 가능성이 높다. “프롬프트 캐시 레이어를 어떻게 설계할 것인가”는 데이터베이스
스키마 설계처럼, LLM 애플리케이션 아키텍처에서 초기에 결정해야 할 핵심
질문이 될 것이다.

### 인프라 경쟁의 새로운 축: 캐시 인프라

OpenAI가 서버 어피니티 기반 캐싱을, Anthropic이 명시적 캐시 마커를, Google이
자체 방식을 제공하는 것은 단순한 기능 경쟁이 아니다. 이는 “어떤 플랫폼에서
개발하면 운영 비용이 더 낮아지는가”라는 경제적 락인의 새로운 축이다.

대규모 LLM 워크로드에서 입력 토큰 비용은 총 운영비의 상당 부분을 차지한다.
캐싱 효율이 높은 플랫폼은 동일한 워크로드를 더 낮은 비용으로 처리할 수 있고,
이는 개발자와 기업의 플랫폼 선택에 실질적인 영향을 미친다. 캐시 인프라 품질이
API 가격보다 실질 비용에 더 큰 영향을 미칠 수 있다는 뜻이다.

24시간 연장 보존이 최신 모델에서만 지원된다는 점도 전략적이다. 최신 고성능
모델로의 업그레이드를 유도하는 동시에, 캐싱 인프라를 성능 프리미엄의 일부로
포지셔닝한다. 모델 업그레이드의 이유가 “더 똑똑한 모델”에서 “더 효율적인
인프라”로 확장되고 있다.

### 보안과 성능의 교차점: Key/Value 텐서 캐싱

“고객 콘텐츠가 아닌 Key/Value 텐서만 저장한다”는 설명은 표면적으로는 보안
안내처럼 보이지만, 실은 깊은 기술적 함의를 담고 있다. Transformer 아키텍처의
어텐션(attention) 연산에서 Key/Value 행렬은 중간 계산 결과물이다. 이것을
캐싱한다는 것은 모델 추론 과정의 특정 단계를 저장하고 재사용한다는 의미다.

이 방식은 원본 텍스트를 재처리하는 것보다 계산 효율이 높고, 원본 텍스트를
별도로 저장하지 않기 때문에 개인정보 보호 측면의 위험을 최소화한다. 그러나
텐서 표현 자체에서 원본 텍스트를 역추적하는 것이 이론적으로 불가능하지 않다는
점은 장기적인 보안 연구의 대상이 될 수 있다.

더 넓은 관점에서, Key/Value 캐싱은 LLM 추론 최적화의 핵심 기법 중 하나다.
이 기법이 프로덕션 API에서 공식적으로 노출되고 개발자가 의식적으로 활용하게
된다는 것은, LLM 인프라의 내부 동작 원리가 점차 애플리케이션 개발의 관심사로
흡수되고 있음을 보여준다. 개발자가 신경망 아키텍처의 세부 사항을 몰라도 그
특성을 활용하는 방향으로 추상화가 진화하고 있다.
