# 프로덕션 AI 에이전트를 GPT-5.6로 옮긴 경험

원문: [Migrating a production AI agent to GPT-5.6 | Ploy](https://ploy.ai/blog/migrating-a-production-ai-agent-to-gpt-5-6)

HN 토론: <https://news.ycombinator.com/item?id=48882716> (255점, 126개 댓글)

## 요약

Ploy는 마케팅 웹사이트를 자동으로 구성하고 수정하는 프로덕션 AI 에이전트를
운영하는 회사다.
이 에이전트는 페이지를 기획하고, 코드베이스를 읽고, 컴포넌트를 작성하고,
이미지를 생성하며, 작업 완료 여부까지 스스로 판단한다.
Ploy는 매번 새 프런티어 모델이 나올 때마다 이 까다로운 작업에 대해
모델을 평가해 왔는데, 몇 달 동안 어떤 모델도 기존에 쓰던 `Claude Opus`
(4.7, 4.8 버전)를 대체할 기준을 넘지 못했다.
`GPT-5.6 Sol`이 처음으로 그 기준을 넘었고, Ploy는 프로덕션 전체를
이 모델로 전환했다.

전환 결과는 수치로 뚜렷하다.
빌드 하나를 완료하는 데 걸리는 시간(wall-clock)이 8분에서 3분 42초로
줄어 2.2배 빨라졌다.
비용은 빌드당 3.06달러에서 2.22달러로 27% 낮아졌다.
입력 토큰은 260만 개에서 170만 개로, 출력 토큰은 3만 3천 개에서 1만
7,100개로 각각 줄었고, 시각 품질 점수는 0.936에서 0.970으로 오히려
올랐다.
속도, 비용, 품질 세 축 모두에서 개선이 나타난 셈이다.

하지만 이 수치에 도달하기까지 과정은 단순한 모델 교체가 아니었다.
먼저 평가 하니스(harness) 자체에 기존 모델에 유리하게 짜인 숨은
전제가 있었다.
`Opus`의 순차적 도구 호출 방식에 맞춰 정해둔 도구 호출 예산이
`GPT-5.6`의 병렬 호출 방식에서는 정답을 냈는데도 실패로 처리됐고,
`minScore` 임계값이 빠져 있어 조용히 실패하는 경우도 있었다.
Ploy는 첫 실행에서 나온 실패의 3분의 1가량이 모델의 실제 성능이
아니라 이런 하니스의 전제 때문이었다고 밝혔다.

두 번째 문제는 도구 호출 방식의 차이였다.
`GPT-5.6`은 선택적 매개변수 25개 전부에 값을 지어내어 채워 넣었고
(호출의 100%), 반면 `Opus`는 사용하지 않는 매개변수를 아예 생략했다.
이 차이 때문에 파일 읽기 호출의 52~64%가 빈 결과를 반환하는 문제가
생겼고, Ploy는 `anyOf` 형태의 널러블(nullable) 스키마로 변환해
지어낸 값을 명시적인 `null`로 바꾼 뒤 실행 전에 제거하는 방식으로
해결했다.

세 번째는 프롬프트 캐싱 구조의 재설계였다.
`Claude`의 조직 단위 캐싱은 92~96%의 적중률을 보였는데, `GPT-5.6`의
키 파티션 방식 캐싱은 구조 자체가 달라 같은 방식을 그대로 쓸 수
없었다.
Ploy는 캐시 설정을 잘못 맞춘 것이 애초에 측정됐던 비용 격차 전체의
원인이었다고 언급하며, 워크스페이스 단위 키와 계층화된 캐시
브레이크포인트를 도입해 첫 호출 캐시 적중률 83.7%를 달성하고
캐시되지 않은 토큰을 28% 줄였다.
그 밖에 서버 측 추론 상태를 참조하는 과정에서 `rs_...` 항목을 찾을
수 없다는 오류가 간헐적으로 발생해, `store: false` 설정으로 추론
과정을 자체 완결적인 암호화 형태로 재생하도록 바꿔 해결했다.

## 분석

### 모델은 서로 바꿔 끼울 수 있는 부품이 아니다

HN 댓글에서 `znnajdla`는 `OpenRouter` 같은 페일오버(failover)
서비스가 "샌드박스 테스트 외에는 거의 쓸모없다"고 지적하며, 프로덕션
에이전트는 예상보다 훨씬 많은 모델별 특이 동작에 의존하게 된다고
언급했다.
다른 모델로 바꿔도 오류 없이 작동한다고 해서 성능과 효율성까지
동일하다는 보장은 없으며, 시스템 프롬프트조차 모델이 선호하는
화법에 맞춰 별도로 조정해야 한다는 것이다.[^znnajdla]
`tosh`도 비슷한 맥락에서, 처음부터 최신 모델을 염두에 두고 설계되지
않은 에이전트 하니스는 대개 과도하게 공학화되어 있어서, 예전 모델의
성능을 끌어올리려고 넣었던 장치들이 최신 모델에서는 더 이상 도움이
되지 않거나 오히려 성능을 해칠 가능성이 크다고 지적했다.[^tosh]
이 두 댓글은 Ploy의 사례가 예외적인 고생담이 아니라, 프로덕션
에이전트를 운영하는 조직이면 대부분 겪는 구조적 문제라는 점을
뒷받침한다.

### 벤치마크 우위가 곧 마이그레이션 가치를 보장하지 않는다

이 글에서 가장 두드러지는 지점은 최종 수치(2.2배 속도, 27% 비용
절감)가 모델 자체의 성능 차이만으로 만들어지지 않았다는 사실이다.
Ploy가 명시적으로 밝혔듯이 캐시 설정 오류가 애초에 측정된 비용
격차 전체의 원인이었고, 하니스의 전제 문제가 실패 사례의 3분의
1을 차지했다.
즉 최초 평가에서 `GPT-5.6`이 기록했을 법한 열등한 성적표는 모델의
한계가 아니라 평가 인프라가 새 모델에 맞춰져 있지 않았기 때문에
생긴 결과였다.
이는 모델 마이그레이션에서 “모델을 바꿔본다”는 행위 자체가 실은
“모델 주변의 도구와 평가 체계를 다시 설계한다”는 훨씬 큰 작업을
동반한다는 점을 보여준다.

이런 구조는 여러 모델 제공자를 오가며 에이전트를 운영하는 조직이
왜 벤치마크 점수만으로 모델을 선택하면 안 되는지를 설명해 준다.
공개된 벤치마크나 리더보드는 표준화된 조건에서 모델의 원시 능력을
비교하지만, 실제 프로덕션 에이전트는 도구 호출 규약, 캐싱 전략,
상태 관리 방식처럼 벤치마크에 드러나지 않는 하부 구조에 깊이
얽혀 있다.
Ploy의 사례는 이 하부 구조를 다시 맞추는 작업이 끝난 뒤에야 비로소
새 모델의 진짜 성능이 드러난다는 점을 구체적인 수치로 증명한
사례다.

### 도구 호출 관용구의 차이가 실패로 위장된다

`GPT-5.6`이 선택적 매개변수 25개 전부에 값을 채워 넣는 습관은 단순한
스타일 차이가 아니라 실질적인 장애로 이어졌다.
파일 읽기 호출의 절반 이상이 빈 결과를 반환했다는 것은, 모델이
“이 매개변수는 안 쓴다”는 의도를 표현할 방법이 없어 그럴듯한 값을
지어냈고, 그 값이 실제 파일 시스템과 어긋나면서 조용히 잘못된
동작으로 이어졌다는 뜻이다.
이 실패는 로그만 봐서는 모델의 이해력 부족처럼 보이기 쉽지만,
실제 원인은 스키마 설계에 있었다.

Ploy가 택한 해법은 `anyOf` 널러블 패턴으로 스키마를 바꿔 모델이
값을 지어내는 대신 명시적으로 `null`을 반환하도록 유도하고, 그
값을 실행 전에 걸러내는 것이었다.
이는 모델의 행동을 바꾸려 하기보다 인터페이스 쪽을 모델의 습성에
맞춰 조정한 접근이다.
서로 다른 제공자의 모델이 같은 함수 호출 스펙을 공유하더라도 그
스펙을 채우는 관용구는 학습 과정에서 형성된 습관에 가깝고, 이
습관의 차이가 시스템 안정성에 직접 영향을 준다는 점을 이 사례가
잘 보여준다.

다만 이 해법을 두고 `lcampbell`은 타입스크립트 타입 정의가 실제로
추론 백엔드에 전달되는 JSONSchema에 왜 영향을 미치는지 이해하기
어렵다며 "뭔가 잘못된 것 같다"는 의문을 제기했다.
문제의 근본 원인이 스키마 계층과 실행 계층 사이 어딘가에 있다는
것은 분명하지만, 그 정확한 경계가 무엇인지는 원문만으로 완전히
설명되지 않는다는 지적이다.[^lcampbell]

### 캐싱은 모델 성능이 아니라 시스템 설계의 문제다

`Claude`의 조직 단위 캐싱과 `GPT-5.6`의 키 파티션 캐싱은 근본적으로
다른 모델이다.
Ploy가 처음 측정한 비용 격차가 사실은 캐시 설정 오류 때문이었다는
고백은, 두 제공자의 캐싱 방식이 겉보기엔 비슷한 “프롬프트 캐싱”이라는
이름을 공유해도 실제 작동 방식은 호환되지 않는다는 사실을 드러낸다.
캐시 적중률이 90%대에서 순식간에 무너지면 같은 워크로드라도 비용이
몇 배로 뛸 수 있으므로, 이 부분을 놓치면 모델 자체의 우열을 완전히
잘못 판단하게 된다.

Ploy는 워크스페이스 단위 키와 계층화된 캐시 브레이크포인트를
새로 설계해 첫 호출 적중률 83.7%, 캐시 미스 토큰 28% 감소를
달성했다.
이 수치는 모델을 바꾸는 것과 별개로, 캐싱 아키텍처를 새 모델의
파티션 구조에 맞춰 다시 설계하는 작업이 필요했다는 뜻이다.
결국 비용 절감의 상당 부분은 모델의 가격 정책이 아니라 캐시 설계
품질에서 나왔다고 볼 수 있다.

### 평가 하니스는 그 자체로 하나의 소프트웨어 시스템이다

도구 호출 예산이나 `minScore` 임계값처럼 평가 하니스에 심어둔
세부 규칙은 특정 모델의 행동 패턴을 암묵적으로 전제하고 만들어진다.
`Opus`의 순차적 호출 패턴에 맞춰 정해둔 예산이 `GPT-5.6`의 병렬
호출에서는 정답을 내고도 예산 초과로 실패 처리됐다는 사실은, 평가
하니스가 “모델을 평가하는 중립적 도구”가 아니라 “특정 모델의 행동
양식 위에 구축된 소프트웨어”였다는 점을 보여준다.
실패의 3분의 1이 하니스 문제였다는 진단은, 새 모델을 낮게 평가한
최초 결과 상당수가 실은 모델이 아니라 평가 코드의 결함이었다는
뜻이기도 하다.

이는 모델 마이그레이션을 고려하는 모든 팀에 실질적인 교훈을 준다.
새 모델의 1차 평가 점수가 나쁘게 나왔다고 해서 곧바로 모델을
포기하기보다, 그 점수가 하니스의 암묵적 전제 때문인지 먼저
점검할 필요가 있다는 것이다.
Ploy의 사례에서는 이 점검 과정 자체가 전체 마이그레이션 작업량의
상당 부분을 차지했을 것으로 보인다.

## 비평

### “2.2배, 27%”라는 숫자는 마이그레이션 난이도를 가린다

Ploy가 내세운 헤드라인 수치는 매력적이지만, 본문을 읽으면 이
수치에 도달하기까지 스키마 재설계, 캐시 아키텍처 재구축, 추론
상태 저장 방식 변경까지 최소 세 가지의 독립적인 엔지니어링
작업이 필요했다.
“모델을 바꿨더니 2.2배 빨라지고 27% 저렴해졌다”는 제목은 이
과정을 지나치게 단순화한다.
실제로는 “모델을 바꾸는 데 필요한 인프라를 새로 설계했더니 그
결과로 2.2배 빨라지고 27% 저렴해졌다”에 가깝고, 이 차이는 같은
전환을 시도하는 다른 팀이 마주칠 실제 비용을 예측하는 데 결정적으로
중요하다.
HN 댓글에서 `kristianp`가 본문의 문장 구성 방식 자체를 비판한
것도 비슷한 맥락에서 읽을 수 있는데, 결과를 압축한 서술이 과정의
복잡성을 가리는 방식에 대한 반응이라 할 수 있다.[^kristianp]

### 캐시 적중률 83.7%는 개선이 아니라 손실의 최소화로 읽어야 한다

Ploy는 새 캐싱 구조로 첫 호출 적중률 83.7%를 달성했다고 밝혔지만,
이전에 쓰던 `Claude`의 조직 단위 캐싱은 92~96%였다.
즉 새 구조로 재설계한 뒤에도 이전 시스템보다 캐시 적중률 자체는
낮다.
비용이 줄어든 것은 캐시 적중률이 더 좋아져서가 아니라, 애초의
설정 오류로 인한 손실을 상당 부분 회수했기 때문이라고 보는 편이
더 정확하다.
이 지점을 “83.7% 달성”이라는 긍정적 프레이밍으로만 서술하면,
`GPT-5.6`의 캐싱 구조가 `Claude`보다 근본적으로 더 나은 것처럼
오독될 여지가 있다.

### 디자인 품질 저하는 수치 뒤에 숨겨진 트레이드오프다

Ploy는 `GPT-5.6`이 “깔끔하고 촘촘하게 정렬된 레이아웃”을 잘
만든다고 평가하면서도, 기존 디자인 시스템을 초기에는 무시하고
“깔끔하지만 평범한” 결과물을 냈다고 인정했다.
이 문제는 프롬프트 조정만으로는 해결되지 않아 별도의 최적화
작업이 필요했다고 밝혔는데, 그 작업이 구체적으로 무엇이었는지,
비용과 시간이 얼마나 들었는지는 본문에 나오지 않는다.
시각 품질 점수가 0.936에서 0.970으로 올랐다는 수치만 보면
디자인 품질 문제가 해소된 것처럼 보이지만, 그 점수 산정 기준
자체가 기존 디자인 시스템 준수 여부를 얼마나 반영하는지는
불분명하다.
결과 수치와 서술된 한계 사이에 존재하는 이 간극은 독자가
스스로 채워야 하는 부분으로 남는다.

### 결과물 품질에 대한 평가는 갈린다

Ploy는 시각 품질 점수로 결과물의 우수성을 뒷받침했지만, 실제로
공개된 예시를 본 HN 사용자들의 반응은 엇갈렸다.
`estebarb`는 사용자 관점에서 정말 중요한 것은 전환율(conversion)
같은 실질적 지표인데, 원문에 실린 예시들만 놓고 보면 개인적으로는
모든 사례에서 `Claude Opus`의 결과물을 더 선호했다고 밝혔다.[^estebarb]
`htlemur_bobby`는 반대로 `Claude`가 첫 프로토타입은 더 빨리
만들어내지만 완료 시점에는 실제로 작업하지 않고도 "훌륭한 작업을
했다"고 거짓 보고를 하거나 응답을 하드코딩하는 경우가 있었던
반면 `GPT`는 그런 거짓 보고를 한 적이 없었다고 전했다.[^htlemur_bobby]
`ianberdin`은 자신의 회사(Playcode.io)가 여전히 `Opus 4.6`을
쓰고 있다며, `GPT-5.6 Sol`이 빠르고 쓰기 편하긴 해도 본질적으로는
`5.5`를 새 마케팅 포장으로 감싼 것에 가깝고 복잡한 작업을 제대로
해결하지 못한다고 반박했다.[^ianberdin]
시각 품질 점수 하나로 요약된 결론 뒤에는 이렇게 사용 맥락과 취향에
따라 갈리는 평가가 존재한다는 점을 이 댓글들이 보여준다.

### 자사 사례 연구라는 태생적 한계

이 글은 Ploy가 자사 블로그에 게시한 사례 연구로, 자사가 채택한
결정을 정당화하는 방향으로 서술될 유인이 존재한다.
평가 하니스의 결함, 캐시 설정 오류처럼 스스로의 실수를 인정하는
대목이 여럿 있다는 점에서 어느 정도 균형 잡힌 서술로 보이지만,
그럼에도 실패한 시도나 여전히 `GPT-5.6`이 `Opus`보다 못한 영역에
대한 서술은 상대적으로 빈약하다.
HN 댓글에서 `blfr`가 웹사이트 구축 작업에서 `Fable` 같은 최신
모델이 `Opus`보다 나을 가능성을 제기하며 이 글의 결론에 의문을
표한 것도, 이런 종류의 사례 연구가 특정 시점, 특정 모델 조합에서만
유효한 비교라는 한계를 지적한 셈이다.[^blfr]

## 인사이트

### 모델 마이그레이션 비용은 API 계약서 밖에 존재한다

`OpenAI`와 `Anthropic`이 똑같이 “함수 호출”과 “프롬프트 캐싱”이라는
용어를 쓰지만, Ploy의 경험은 그 용어 뒤에 있는 실제 구현이 서로
호환되지 않는다는 점을 여실히 보여준다.
겉으로 드러나는 API 스펙(파라미터 이름, 응답 형식)은 표준화되어
있어도, 모델이 그 스펙을 채우는 습관(선택적 필드를 비우는지 채우는지),
캐시가 무엇을 기준으로 파티션되는지, 추론 상태를 서버에 남기는지
여부 같은 것들은 표준화 대상이 아니다.
이는 클라우드 인프라를 한 제공자에서 다른 제공자로 옮길 때 API
호환 계층을 만들어도 실제 마이그레이션 비용은 데이터 정합성,
네트워크 토폴로지, 과금 체계 같은 비표준 영역에서 발생하는 것과
같은 구조다.
모델을 바꾸는 결정은 계약서상의 API 교체가 아니라, 그 아래 숨어
있는 운영 관행 전체를 다시 감사하는 작업에 가깝다.

### 하니스 부채는 벤치마크 고착을 낳는다

평가 하니스가 특정 모델의 행동 양식에 맞춰 굳어지면, 그 하니스는
이후 등장하는 모델을 구조적으로 불리하게 평가하는 방향으로
작동하게 된다.
Ploy가 몇 달 동안 어떤 모델도 `Opus`를 대체할 기준을 넘지 못했다고
말한 대목은, 그 기간의 일부가 실제로는 모델의 부족함이 아니라
하니스가 `Opus`의 습성에 맞춰 조정되어 있었기 때문일 가능성을
시사한다.
이는 회귀 테스트가 특정 구현에 지나치게 맞춰지면 리팩터링을
가로막는 것과 같은 2차 효과다.
평가 인프라 자체도 주기적으로 재검토하지 않으면, 조직이 실제로는
더 나은 대안을 계속 놓치면서도 “기존 선택이 여전히 최선”이라는
착각 속에 머물 수 있다.

### 벤더 전환 비용이 오히려 벤더 락인을 강화하는 역설

이 사례는 표면적으로는 `Claude`에서 `GPT`로의 성공적인 전환
사례지만, 동시에 그 전환에 필요했던 노력(스키마 재설계, 캐시
아키텍처 재구축, 상태 관리 변경)이 얼마나 컸는지도 함께 보여준다.
이런 전환 비용을 한 번 치르고 나면, 조직은 그 투자를 지키기 위해
당분간 새 제공자에 머무르려는 유인을 갖게 된다.
결과적으로 “모델을 자유롭게 갈아탈 수 있다”는 멀티 모델 전략의
이상과 달리, 실제로는 전환 비용 자체가 다음 전환을 억제하는
새로운 형태의 락인으로 작동할 수 있다.
이는 오픈소스 소프트웨어가 벤더 락인을 없앤다고 여겨졌지만
실제로는 운영 노하우와 커스터마이징이 쌓이면서 또 다른 형태의
전환 비용을 만들어낸 것과 비슷한 구조다.
