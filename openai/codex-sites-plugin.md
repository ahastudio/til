# OpenAI Codex Sites 플러그인: 프롬프트에서 배포까지

<https://developers.openai.com/codex/sites>

## 요약

OpenAI가 Codex에서 웹사이트를 만들어 배포하는 Sites 플러그인을 공개했다.
사용자가 프롬프트를 입력하거나 기존 프로젝트를 넣으면, 웹사이트, 웹앱, 게임을 별도의 배포 워크플로우 없이 호스팅까지 처리한다.

배포 워크플로우는 2단계로 구성된다.
먼저 저장(save)으로 검토용 버전을 만들고, 승인된 버전만 배포(deploy)하여 프로덕션에 반영한다.
모든 배포 URL은 즉시 프로덕션 준비가 된 상태이며, 저장된 버전마다 Git 커밋이 자동으로 생성된다.

지원하는 사이트 유형은 네 가지다.
콘텐츠 중심 웹사이트와 랜딩 페이지, 지속적 사용자 데이터가 있는 애플리케이션, 저장 기능이 있는 게임, 파일 업로드를 지원하는 애플리케이션이다.
스토리지 옵션은 구조화된 영속 데이터를 위한 D1 관계형 데이터베이스와 파일 및 미디어를 위한 R2 오브젝트 스토리지다.

접근 제어는 세 가지 모드를 제공한다.
소유자/관리자 전용, 워크스페이스 전체 공개, 특정 사용자 또는 그룹으로의 맞춤 설정이다.
환경 변수와 시크릿은 소스 코드에 커밋되지 않고 Sites 패널에서 관리된다.
현재 ChatGPT Business와 Enterprise 워크스페이스에서 프리뷰로 제공된다.

## 분석

### Cloudflare Workers 위의 AI 네이티브 배포

기술적으로 Sites는 Cloudflare Workers 호환 ES 모듈을 빌드해야 한다.
D1과 R2는 Cloudflare의 서버리스 스토리지 서비스다.
OpenAI가 자체 컴퓨팅 인프라를 구축하는 대신 Cloudflare 생태계 위에 올라탄 선택은 실용적이다.
엣지 배포의 복잡성을 흡수하면서도 글로벌 CDN과 서버리스 실행 환경을 즉시 얻을 수 있다.

[GeekNews 댓글](https://news.hada.io/topic?id=30128)에서 원포스터는 “Cloudflare Workers를 이용하네요... 진짜 아무 웹사이트나 다 만들 수 있을 것 같아요. 바이브 코더들한테 유용한 기능이 될 듯”이라고 평했다.
“바이브 코딩(vibe coding)” — 코드의 정확한 작동보다 느낌과 속도를 우선하는 접근 방식 — 에 최적화된 도구라는 시각이다.

### 저장/배포 분리의 의미

저장과 배포를 분리한 것은 단순한 UX 결정이 아니다.
AI가 생성한 코드는 미묘한 버그나 보안 취약점을 포함할 수 있다.
검토 없이 즉시 배포하는 것은 위험하다.
두 단계의 게이트는 “빠르게 만들고 신중하게 공개한다”는 균형을 구조적으로 강제한다.
Git 커밋 자동 추적은 버전 이력을 자연스럽게 확보해주는 부산물이다.

## 비평

### 제한된 접근성

현재 ChatGPT Business와 Enterprise 전용이라는 점은 초기 타겟이 개인 개발자나 스타트업이 아니라 기업 고객임을 명확히 한다.
비용 구조가 공개되지 않은 상태에서, 실제로 얼마나 광범위하게 사용될지는 미지수다.
Cloudflare Workers 기반이므로 Cloudflare의 무료 티어 수준을 넘어서면 비용이 발생할 것이고, OpenAI가 이를 어떻게 과금할지가 채택률을 결정할 것이다.

### 벤더 락인(Vendor Lock-in)의 위험

OpenAI의 플랫폼에서 배포한 사이트를 다른 환경으로 이전하는 것이 얼마나 자유로운지가 불명확하다.
기술적으로 Cloudflare Workers 호환 코드이므로 이식성이 있을 것 같지만, Sites 패널에서 관리하는 환경 변수와 스토리지의 마이그레이션 경로가 문서화되어 있는지가 중요한 질문이다.
빠른 프로토타이핑에는 적합하지만, 프로덕션 서비스의 장기 운영에는 신중한 평가가 필요하다.

## 인사이트

### 소프트웨어 생산 파이프라인의 압축

전통적 웹 개발 파이프라인은 길다.
아이디어 → 설계 → 개발 → 테스트 → CI/CD 설정 → 배포 → 모니터링.
각 단계마다 별도의 도구와 전문성이 필요하다.
Sites는 이 파이프라인을 극단적으로 압축한다. 프롬프트 → 저장 → 배포.

이 압축은 새로운 시장을 만든다.
지금까지 기술적 장벽 때문에 웹에 아이디어를 구현하지 못했던 사람들이 처음으로 참여할 수 있게 된다.
Yamashita가 “누구나 만들 수 있게 된다”고 말한 것이 현실이 되는 장면이다.
동시에 그 글이 경고한 것처럼, 무엇을 만들지를 아는 것의 중요성이 더 부각된다.

### AI 회사의 플랫폼 확장 전략

OpenAI는 모델 API 회사에서 플랫폼 회사로 전환하는 궤적을 보여준다.
GPT Store, Codex, Sites가 연속선상에 있다.
모델을 팔던 회사가 모델 위에 생태계를 구축하고, 개발자와 최종 사용자를 직접 연결하는 플랫폼으로 확장한다.
이 전략은 Microsoft의 GitHub Copilot, Vercel의 v0와 같은 경쟁자들과 직접 충돌한다.

AI 네이티브 플랫폼 전쟁의 핵심 질문은 누가 개발자의 첫 번째 선택이 되느냐다.
Sites처럼 배포까지 통합된 경험을 제공하면 개발자가 플랫폼을 이탈할 이유가 줄어든다.
이것이 “처음부터 끝까지” 경험을 제공하려는 경쟁의 본질이다.

### 바이브 코딩의 프로덕션화

“바이브 코딩”이라는 용어는 Andrej Karpathy가 만든 개념으로, 정확한 이해 없이 LLM이 제안하는 것을 느낌으로 받아들이는 코딩 방식을 가리킨다.
Sites는 이 방식을 프로덕션 배포까지 확장한다.
저장/배포 분리와 접근 제어가 있으므로, 완전히 무방비 상태는 아니다.
하지만 저품질 코드가 실제 서비스로 배포되는 경로가 그 어느 때보다 짧아진다.

이것은 소프트웨어 품질 문제를 넘어 보안 문제다.
AI가 생성한 코드의 SQL 인젝션, XSS, 인증 취약점이 검토 없이 인터넷에 노출될 가능성이 높아진다.
OpenAI가 배포 전 자동 보안 스캔을 제공하거나, Enterprise 고객을 위한 코드 검토 체크포인트를 강화하지 않는다면, Sites는 새로운 종류의 보안 사고의 벡터가 될 수 있다.
