# AI 시대의 SRE — Google의 신뢰성 있는 운영 설계

원문: <https://sre.google/resources/practices-and-processes/ai-engineering-reliable-operations/>

## 요약

Google이 AI 시대에 맞게 SRE(Site Reliability Engineering)를 근본적으로 재설계한
접근을 공개했다.
AI 코딩 보조가 4배 생산성 향상을 가능하게 하는 환경에서, 전통적 수동 검토
방식은 지속 불가능해졌다.
Google의 답은 자율 에이전트, 실행 가드레일(Actus), 지속적 평가 파이프라인이다.

세 개의 축으로 안전성을 정의한다.

- **투명성**: 모든 자율 결정이 관찰 가능하고 추론 과정이 기록된다.
- **실시간 리스크 평가**: 배포 상태, 에러 예산, 진행 중인 인시던트를 고려한
  맥락적 위험 평가.
- **점진적 권한 부여**: 검증된 신뢰성 데이터에 기반해 자율성 수준이 단계적으로
  높아진다.

## 분석

### 다섯 단계 자율성 수준(L0~L4)

수동에서 완전 자율까지의 성숙도 경로를 정의한다.

| 수준 | 이름 | 특징 |
| ---- | ---- | ---- |
| L0 | 수동 | 사람이 모든 모니터링, 조사, 승인, 실행 담당 |
| L1 | 보조 | 자동화가 모니터링·조사; 사람이 승인·실행 |
| L2 | 부분 | 시스템이 조사·제안; 사람이 명시적 승인 후 실행 |
| L3 | 높음 | 잘 정의된 시나리오 자율 실행; 새 상황은 사람이 처리 |
| L4 | 완전 | 인시던트 전체 생명주기 자율 관리 |

현재 AI Operator는 L2~L3 수준으로 배포됐다.
중요한 작업은 인간 검토를 요구하고, 경미한 인시던트는 자율 실행한다.

### AI Operator — 자율 초기 대응자

AI Operator는 경보를 수집하고, 유사한 과거 인시던트를 처리한 전문가의 예시를
참고해 병렬 조사를 수행한다.
근본 원인 분석, 구조화된 완화 카탈로그에서 적절한 대응 선택, 결과 모니터링을
수행한다.

근본 원인을 식별할 수 없거나 안전 경계를 벗어나는 시나리오는 즉시 에스컬레이션한다.
조사 이력이 Google 인시던트 플랫폼에 자동 이전되어 컨텍스트 손실 없이
사람에게 핸드오프된다.

### Actus — 완화 제어 플레인

에이전트에게 인프라 직접 접근을 주지 않고, Actus가 모든 프로덕션 변경의
통합 안전 게이트웨이 역할을 한다.

세 단계 아키텍처:
1. **발견·계획**: 동적으로 필터링된 완화 레지스트리 제공, 에이전트 의도를
   검증 가능한 실행 계획으로 변환
2. **동적 안전 가드레일**: 필수 드라이런, 정당성 검증, 동시 실행 점검.
   위험 증가 시 자율성 수준 자동 하향
3. **사후 모니터링**: 장기 실행 작업의 성공·실패 추적

이 분리 구조는 추론 엔진과 실행 엔진을 엄격히 분리해,
AI 모델이 진화해도 안전 경계가 유지되도록 보장한다.

### 평가 데이터 파이프라인

AI 성능의 품질은 인간 검증 예시에 대한 엄격한 테스트에서 나온다.

데이터 계층:
- **Bronze**: 자동화된 레이블러가 휴리스틱으로 생성
- **Silver**: 프로그래밍으로 생성, Gold 데이터로 캘리브레이션
- **Gold**: 인간 전문가가 검증

인시던트 해결 중에 SRE가 구조화된 완화 제안을 생성하고 수락·수정하는 것만으로
골든 레이블이 시스템에 자동으로 피드백된다.
일상적 운영 워크플로우가 AI 훈련 데이터를 지속적으로 생성하는 구조다.

### 핵심 구성 요소

**Detectr** — 지표 기반 모니터링이 놓치는 새로운 장애 모드를 탐지한다.
소셜 미디어, 지원 티켓, 포럼의 비구조화된 사용자 피드백을 Gemini로 분석해
사용자 보고 장애를 조기에 감지한다.
수백 시간의 누적 고객 피해를 조기 탐지로 줄였다.

**AI 알림 에이전트** — 경보가 도착하면 2분 이내에 모니터링 시스템, 로그,
변경 이력, 의존성 그래프를 병렬 조회해 컨텍스트를 원래 경보에 추가한다.
읽기 전용으로만 동작한다.

**Antigravity CLI** — SRE가 MCP 서버를 통해 프로덕션과 자연어로 상호작용한다.
실시간 모니터링 쿼리, 로그 분석, 인시던트 상세 조회, 정책 준수 완화 실행을
지원한다.

## 비평

### 강점 — “안전 트리펙타”의 명확한 원칙

투명성, 실시간 리스크 평가, 점진적 권한 부여라는 세 원칙은 에이전트 AI를
프로덕션에 배포하는 거버넌스 원칙으로 범용 적용 가능하다.
이 원칙들은 Google 내부에만 해당하지 않고, 어떤 조직이든 AI 자율화 로드맵을
설계할 때 참조할 수 있다.

A/B 테스트로 “AI 가설 지원이 MTTM을 10% 줄였다”, “조사 대시보드 사용 시 44%
감소”라는 구체적 수치를 제시하는 것도 강점이다.
추상적 약속이 아닌 측정된 결과다.

### 약점 — Google 규모의 편향

이 접근의 대부분은 Google 규모에서 설계됐다.
수십 개의 동시 배포, 수천 명의 SRE, 자체 평가 파이프라인 인프라를 가진 조직의
해법이다.
중소 규모 조직이 같은 구조를 도입할 때 어떤 부분이 확장 불가능한지, 어떤
단순화가 가능한지에 대한 가이드가 없다.

### 빈 자리 — 실패 모드와 회복 전략

AI Operator가 L3 수준에서 자율 실행을 잘못 수행했을 때 어떻게 되는가?
드라이런이 실제 실행과 다른 결과를 낼 때의 처리는?
실패 시나리오와 복원 전략이 충분히 다뤄지지 않는다.
성공 사례 중심으로 쓰여진 글이라, 자율화의 위험을 균형 있게 평가하기 어렵다.

## 인사이트

### SRE 역할의 “추상화 사다리” 상승은 직업 변화의 선행 지표다

“SRE가 운영자에서 AI 안전 아키텍트로 전환한다”는 명제는 소프트웨어 엔지니어링
전반에 반복될 패턴의 선행 신호다.
자동화가 저수준 작업을 흡수할 때마다, 사람의 일은 그 자동화를 설계하고
제어하는 상위 수준으로 이동했다.
어셈블리 → 고급 언어, 온프레미스 → 클라우드, 수동 배포 → CI/CD가 모두
같은 경로를 밟았다.

SRE의 변화가 다른 엔지니어링 역할보다 빠른 이유는, SRE 업무의 많은 부분이
이미 잘 정의된 절차와 반복 가능한 패턴으로 구성되어 있기 때문이다.
잘 정의된 절차는 자동화가 쉽다.
소프트웨어 엔지니어링의 창의적·탐색적 부분이 자동화되려면 더 오랜 시간이
필요하지만, SRE의 운영 부분은 지금 가장 빠르게 변하고 있다.

### 데이터 플라이휠이 AI 운영의 핵심 경쟁 우위가 된다

일상적 인시던트 처리가 자동으로 골든 레이블 데이터를 생성하는 구조는
운영 규모가 AI 평가 데이터 규모로 전환되는 플라이휠이다.
인시던트를 더 많이 처리할수록, AI Operator의 평가 데이터가 더 풍부해지고,
모델이 더 정확해지고, 더 많은 인시던트를 자율 처리한다.

이 구조를 구축한 조직은 시간이 지날수록 구축하지 않은 조직과의 격차가 커진다.
초기 투자가 어렵더라도 플라이휠을 돌리기 시작하는 것이 중요한 이유다.
Google의 공개는 이 구조가 작동함을 증명하는 동시에, 유사한 구조를 구축하려는
다른 조직들에게 설계 참고서가 된다.

### “PR 문제”의 근본적 재정의

4배 속도 향상으로 AI가 코드를 생성하면, 코드 리뷰가 PR 단위 라인 검토에서
“아키텍처·안전 제약 사전 정의”로 이동해야 한다는 통찰은 소프트웨어 개발
전반에 적용된다.

이것은 시니어 엔지니어의 역할 재정의이기도 하다.
시니어가 하는 일이 “코드를 잘 작성한다”에서 “AI가 좋은 코드를 낳는 제약과
명세를 설계한다”로 바뀐다.
이 전환이 일어나면 시니어 엔지니어의 가치는 코딩 속도가 아니라 명세 능력과
판단력에서 나온다.

코드 리뷰가 “작성된 코드를 검토”에서 “생성될 코드의 제약을 정의”로 이동하는
것은, 건축 설계와 시공의 분리와 유사하다.
설계가 먼저, 시공이 나중 — AI 에이전트가 시공자가 되는 시대의 소프트웨어
개발 모델이다.
