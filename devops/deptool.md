# 내가 원했던 배포 도구를 직접 만들다 — Deptool

원문: <https://ruuda.nl/2026/deptool>

## 요약

Ruud van Asseldonk은 Ansible 같은 기존 인프라 관리 도구에 지쳐 자신만의 배포 도구 Deptool을 직접 만들었다.
동기는 블로그를 유럽 서버로 이전하는 작업이었다.
디지털 주권을 위한 이전이 기존 Python 배포 스크립트의 한계를 노출시켰고, NixOS 같은 무거운 도구를 도입하는 대신 자신의 사용 사례에 최적화된 전문 도구를 만들기로 했다.

Deptool의 설계 목표는 다섯 가지다.
첫째, 속도 — 배포 시간은 1초 미만이어야 한다. 대서양 횡단 핑이 100ms라면 기본 지연 시간이 이를 초과할 이유가 없다.
둘째, 예측 가능성 — OpenTofu처럼 계획(plan)과 적용(apply)을 분리하여 Ansible의 예측 불가능한 연쇄 변경을 피한다.
셋째, 안전성 — systemd 유닛이 실패하면 밀리초 단위로 자동 롤백한다.
넷째, 단순성 — 파일 배포와 유닛 재시작만 한다. 템플릿 복잡성은 외부 도구에 위임한다.
다섯째, 선언성 — 제거된 파일은 명시적 정리 단계 없이 서버에서 사라진다.

아키텍처는 두 가지 핵심 메커니즘으로 구성된다.
설정은 Git 추적 디렉토리에 사전 렌더링된 형태로 저장된다. 최상위 레벨은 호스트, 그 아래는 애플리케이션이다.
파일은 `/var/lib/deptool` 아래 커밋 기반 디렉토리에 위치하고, 심링크로 추적된다.
배포는 심링크 교체로 원자적으로 이루어지며, 여러 버전이 안전하게 공존한다.

SSH를 통한 원격 실행은 명령어 이스케이핑 위험을 피하기 위해 정적 에이전트 바이너리를 SSH 전송 위의 stdin/stdout으로 통신시킨다.
에이전트 설치는 패키지 매니저나 Python 없이 표준 coreutils(`uname`, `mkdir`, `dd`, `chmod`, `sha256sum`)만 사용한다.
Flatcar Linux 호환성을 위한 의도적 선택이다.
동시성 모델은 `git push`와 유사한 낙관적 동시성(optimistic concurrency)을 사용한다. 클러스터 상태를 알고 있다고 가정하고 어긋나면 중단 후 재시도한다.

한 달간의 프로덕션 사용 후 도구가 충분히 성숙했다고 판단해 Codeberg와 GitHub에 오픈소스로 공개했다.

## 분석

### 협소한 문제 정의가 가져오는 설계 명확성

Deptool의 가장 특징적인 측면은 의도적인 범위 제한이다.
“파일 배포와 systemd 유닛 관리”만 한다.
이 좁은 정의가 나머지 설계 결정을 명확하게 만든다.
템플릿 엔진이 필요 없고, 패키지 관리가 필요 없고, 복잡한 조건부 로직이 필요 없다.

이것은 Ansible이 실패하는 지점을 역으로 분석한 결과다.
Ansible은 모든 것을 할 수 있으려 하다 보니 템플릿, 변수, 조건 분기, 역할, 컬렉션이 누적되어 복잡해진다.
Deptool은 “내가 실제로 하는 것”만 구현했다.
이것이 단순성의 원천이다.

### 원자적 배포와 심링크 패턴

커밋 해시 기반 디렉토리 구조와 심링크 교체로 원자성을 구현하는 패턴은 검증된 아이디어다.
Capistrano, Deployer, 심지어 Nix/NixOS도 같은 패턴을 사용한다.
현재 버전을 가리키는 심링크를 새 버전으로 교체하는 것은 파일시스템 수준에서 원자적이다.

Deptool에서 이 패턴이 특히 잘 작동하는 이유는 systemd 유닛 파일과 함께 사용되기 때문이다.
새 설정 파일이 배포되고 유닛이 재시작되면 성공 여부를 확인할 수 있다.
실패하면 이전 심링크로 돌아가고 유닛을 재시작하는 롤백이 자동화된다.

### 낙관적 동시성의 적절한 선택

“팀 전체 경합에는 확장성이 떨어지지만 단일 운영자 시나리오에서는 빠르다”는 트레이드오프를 명시적으로 인정하는 점이 인상적이다.
낙관적 동시성은 충돌이 드물 때 효율적이다.
단일 운영자가 서버를 관리하는 경우 경합이 거의 없으므로 최적의 선택이다.
“수백 명의 동시 배포자”나 “수천 대의 서버”에는 적합하지 않다고 명시한다.

이것은 도구가 해결하고자 하는 문제의 범위를 알고 그 범위 안에서 최적화하는 방식이다.

### 에이전트 없는 초기 설치 문제의 해결

“에이전트를 설치하려면 에이전트가 있어야 한다”는 닭-달걀 문제를 표준 coreutils만으로 해결하는 접근이 우아하다.
첫 연결 시 에이전트가 있다고 가정하고 없으면 `dd`로 바이너리를 전송하여 설치한다.
이 과정에서 패키지 매니저, Python, curl이 모두 필요 없다.

SSH를 통한 stdin으로 바이너리를 전송하고 원격에서 `sha256sum`으로 검증하는 방식은 의존성 최소화의 교과서적 구현이다.

## 비평

### 강점: 설계 동기의 투명성

이 글의 강점은 “무엇을 만들었는가”뿐 아니라 “왜 기존 도구들을 선택하지 않았는가”를 명확하게 서술한다는 점이다.
NixOS, Ansible, 수동 SSH 편집 각각에 대한 트레이드오프 분석이 포함되어 있어 독자가 자신의 상황과 비교하기 쉽다.
“이 도구가 나에게 맞는가”를 판단하는 데 필요한 맥락을 제공한다.

### 약점: 운영 경험의 깊이 부족

한 달의 프로덕션 경험은 짧다.
실제로 중요한 질문들 — 예상치 못한 실패 모드, 동시 배포 충돌의 실제 빈도, 롤백 자동화의 엣지 케이스 — 은 더 긴 운영에서 드러난다.
“개인 블로그 서버 관리”는 스케일과 복잡성 면에서 다른 컨텍스트와 다를 수 있어, 사용 경험의 일반화 가능성이 제한된다.

### 약점: 보안 모델 미서술

SSH 에이전트가 stdin/stdout으로 통신한다는 것은 알지만, 에이전트의 권한 모델이 무엇인지 — root로 실행되는가, 특정 사용자로 실행되는가, systemd 유닛 재시작 권한을 어떻게 부여하는가 — 가 글에 명확히 나오지 않는다.
배포 도구의 보안 모델은 인프라 신뢰의 핵심 부분이기 때문에 이 부분의 부재가 아쉽다.

### 보완: 기존 도구와의 상호 운용성

Deptool이 파일 배포와 유닛 관리만 하기 때문에, 실제 인프라에서는 다른 도구들과 함께 사용해야 한다.
DNS 설정, TLS 인증서, 방화벽 규칙, 사용자 계정 관리 등은 어떻게 처리하는가?
저자가 다른 도구들과 조합하는 방식에 대한 설명이 있었다면 실용성이 높아졌을 것이다.

## 인사이트

### “이것만 잘 하는” 도구의 역습

Ansible, Chef, Puppet 같은 도구들이 수년에 걸쳐 기능을 추가하면서 복잡해진 것은 개별 기능 요청에 응답한 결과다.
각 기능 추가는 합리적인 결정이었지만, 결과는 대부분의 사용 사례에 필요 이상으로 복잡한 도구가 됐다.

Unix 철학의 핵심이 여기 있다. “한 가지를 잘 하라.”
Deptool은 이 철학의 현대적 구현이다.
파일 배포와 유닛 관리만 한다. 그 외 모든 것은 다른 도구에 위임한다.
이 좁은 정의가 코드베이스를 작게 유지하고, 문서를 간결하게 만들고, 사용법을 직관적으로 만든다.

흥미로운 역학이 있다. “모든 것을 하는” 도구는 처음에는 쉬워 보이지만 경계 사례에서 무너진다.
“한 가지만 하는” 도구는 처음에는 부족해 보이지만 실제 사용 사례에서 더 안정적이다.
Deptool의 한 달 운영 경험이 좋은 이유 중 하나는 “범위 밖의 것”을 시도하려는 유혹이 없기 때문이기도 할 것이다.

### 1인 인프라와 도구 선택의 불일치

엔터프라이즈 도구들은 팀이 사용하도록 설계됐다.
RBAC, 감사 로그, 협업 워크플로우, 높은 동시성이 기본 전제다.
그러나 대부분의 개인 개발자와 소규모 팀은 이 복잡성이 필요 없다.

이 불일치가 “개인 인프라 도구” 틈새를 만든다.
Deptool은 이 틈새를 명시적으로 타겟한다.
“단일 운영자 또는 소규모 팀, 수십 대 이하의 서버”가 설계 전제다.
낙관적 동시성이 이 전제에서는 최적이고, 엔터프라이즈 전제에서는 부적합하다.

이것은 더 넓은 패턴을 가리킨다.
기술 도구는 대부분 가장 복잡한 사용 사례 — 대기업, 수천 대 서버, 수백 명의 엔지니어 — 를 기준으로 설계된다.
이 도구들을 소규모에 적용하면 복잡성 오버헤드가 실제 문제보다 크다.
소규모에 맞는 도구를 만들거나 선택하는 것이 실용적이지만, 이를 실천하는 사람은 드물다.
기존 도구를 “억지로 끼워 맞추는” 것이 “내 상황에 맞는 도구를 만들거나 찾는” 것보다 쉬워 보이기 때문이다.

### 배포 속도가 행동을 바꾼다는 관찰

저자가 언급하는 한 가지 실용적 효과가 특히 흥미롭다.
“1초 미만 배포가 서버를 직접 편집하려는 유혹을 제거한다.”

이것은 단순한 편의성 개선이 아니다. 행동 변화다.
배포가 느리면 빠른 수정을 위해 서버를 직접 편집하게 된다.
직접 편집은 버전 관리 밖에서 일어나고, 변경 이력이 없고, 다른 서버에 반영되지 않는다.
결과적으로 인프라가 드리프트(drift)된다. 서버마다 조금씩 다른 상태가 된다.

Deptool의 빠른 배포가 이 유혹을 없앤다.
“직접 편집하는 것보다 배포하는 게 더 빠르다”는 상황이 만들어지면, 올바른 방식(버전 관리를 통한 배포)이 자연스럽게 기본 행동이 된다.
이것은 도구 설계에서 중요한 원칙이다. 올바른 방법이 가장 빠른 방법이 될 때, 규율이 필요 없다.
규율로 강제하는 것보다 인센티브 구조를 바꾸는 것이 더 지속 가능한 정확한 인프라 관리를 만든다.
