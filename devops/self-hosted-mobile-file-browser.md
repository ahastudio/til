# 모바일에서 쓸 수 있는 셀프 호스팅 파일 브라우저 비교

원문: <https://medium.com/@linglijunmail/73e2ce3cb9b8>

## 요약

저자 Jason Anderson은 셀프 호스팅 파일 브라우저를 모바일 환경에서 실제로 사용하며
비교한 결과를 정리했다.
핵심 결론은 “단 하나의 도구가 모바일에서 모든 것을 잘 하지는 않는다”는 것이다.

배경은 FileBrowser의 모바일 한계에서 시작된다.
FileBrowser는 데스크톱 환경을 위해 설계된 인터페이스로, 작은 버튼과 마우스 호버에
의존하는 메뉴가 터치스크린과 근본적으로 맞지 않는다.
더 중요한 문제는 FileBrowser가 파일 관리자(file manager)로 동작하는 반면, 실제
모바일 사용 패턴은 콘텐츠 뷰어에 가깝다는 점이다.

저자의 실제 요구사항은 단순하다.
마크다운 노트와 설정 파일 탐색, 다운로드 없이 문서 미리보기, 브라우저 내 직접 동영상
재생, 터치 최적화된 반응형 인터페이스가 전부다.
파일 편집, 권한 관리, 폴더 재구성 같은 기능은 필요하지 않다.

저자는 검토 결과를 세 계층으로 분류한다.
**Filestash**는 “모바일 Dropbox”로, SFTP·WebDAV·S3 연결을 통해 즉각적인 파일
미리보기와 터치 친화적인 탐색, PWA 기능을 제공한다.
**AList**는 분산 스토리지 백엔드에 걸쳐 미디어 재생에 강점을 가진 멀티 스토리지
애그리게이터로, 큰 탐색 버튼을 제공한다.
**Dufs**는 설정과 유지보수가 필요 없는 최소 단일 바이너리 디렉터리 서버다.

Nextcloud와 Seafile은 개인 사용에 지나치게 복잡했고, Cockpit은 파일 브라우저가
아니라 서버 관리 도구였다.
VS Code Web은 폰에서 성능이 나빴고, Termius·FE File Explorer 같은 프리미엄 앱들은
핵심 기능을 유료 뒤로 숨겼다.
최종 구성은 계층적이다.
Filestash를 주요 폰 접근에, FileBrowser를 데스크톱 관리 작업에, SSH 터미널을
필요 시 보조 수단으로 활용한다.

## 분석

### 요구사항 명료화가 비교의 핵심이다

이 글의 구조적 강점은 초반에 요구사항을 “무엇이 필요한가”와 “무엇이 필요하지 않은가”로
함께 명시한다는 것이다.
“파일 편집 불필요, 권한 관리 불필요, 폴더 재구성 불필요”라는 부정적 요구사항이
Nextcloud나 Seafile을 탈락시키는 근거가 된다.
도구 선택에서 “무엇을 원하지 않는가”를 명시하는 것은 과도한 기능 집합을 가진 솔루션을
걸러내는 데 필수적이지만, 많은 비교 글에서 생략된다.

### 세 계층 분류의 논리

Filestash(최우선) → AList(미디어 중심) → Dufs(최소화)의 계층 구조는 기능성과
복잡도의 역관계를 반영한다.
Filestash는 다양한 프로토콜을 지원하고 PWA 기능까지 제공하지만 그만큼 설정이 필요하다.
AList는 미디어 재생에 특화되어 있으나 분산 스토리지라는 더 복잡한 시나리오를 가정한다.
Dufs는 단일 바이너리로 추가 설정이 없지만 기능도 최소화된다.
세 도구가 서로 경쟁하는 것이 아니라 서로 다른 상황에 최적화되어 있다는 구조적 통찰이
실용적이다.

### 스마트폰의 “피드와 검색” 패러다임

저자가 결론에서 제시하는 철학적 관찰이 가장 주목할 만하다.
스마트폰은 기본적으로 피드(feed)와 검색(search) 패러다임으로 동작하지, 계층적
디렉터리 구조로 동작하지 않는다는 것이다.
iOS Files나 Android의 파일 매니저가 존재하지만, 대부분의 사용자는 앱 중심으로
콘텐츠에 접근하며 파일 시스템 경로를 탐색하지 않는다.
이 관찰은 셀프 호스팅 파일 브라우저 설계에 직접적인 함의를 가진다.

## 비평

### 강점: 실제 사용 경험 기반의 구체성

추상적인 기능 비교표가 아니라, 저자가 실제 폰에서 각 도구를 써본 경험을 바탕으로
했다.
“버튼이 작다”, “호버 메뉴가 작동하지 않는다”처럼 구체적인 UX 문제가 명시된다.
최종 구성도 단일 솔루션이 아닌 계층적 구성을 솔직하게 공개한다.
“하나로 모든 것을 해결하려 하지 않는다”는 결론은 현실적이다.

### 한계: 보안과 네트워크 설정이 생략됐다

셀프 호스팅 파일 브라우저는 네트워크 보안이 핵심 과제다.
VPN 없이 외부 노출하는가, 리버스 프록시를 사용하는가, 인증은 어떻게 처리하는가—이
부분이 완전히 생략되어 있다.
Filestash의 SFTP/S3 연결 보안 설정이나 AList의 외부 스토리지 연결에서 크리덴셜
관리 방식도 다루지 않는다.
“개인 사용”을 전제로 쓴 글이지만, 그 전제 조건이 독자에게 안전한지 보장되지 않는다.

### 보완이 필요한 관점: 안드로이드와 iOS의 차이

모바일 경험을 “스마트폰”으로 일반화하지만, 안드로이드와 iOS는 파일 접근 모델이
다르다.
iOS의 파일 시스템 제한과 Safari의 PWA 지원 한계는 Filestash의 PWA 기능이 iOS에서
어떻게 동작하는지를 불분명하게 만든다.
안드로이드는 Custom Tab이나 WebView를 통해 PWA를 더 자연스럽게 지원한다.
운영체제별 경험 차이가 도구 선택에 영향을 줄 수 있는 요소인데, 언급이 없다.

## 인사이트

### 파일 시스템은 개발자의 추상화지, 일반 사용자의 추상화가 아니다

저자가 관찰한 “스마트폰은 피드와 검색으로 동작한다”는 명제는 훨씬 더 깊은 함의를
가진다.
파일 시스템의 계층적 디렉터리 구조는 운영체제 설계자와 개발자가 만든 추상화다.
이 추상화가 사용자 인터페이스 전면에 노출된 것은 데스크톱 컴퓨터가 특정 전문 사용자를
위한 도구였던 시대의 유산이다.
iOS가 처음 등장했을 때 파일 시스템을 앱 뒤로 완전히 숨긴 것은 이 추상화의 불필요함을
일반 사용자 레벨에서 명시적으로 선언한 사건이었다.

셀프 호스팅 파일 브라우저가 모바일에서 어색한 근본 원인은 여기에 있다.
이 도구들은 파일 시스템이라는 개발자 추상화를 웹 인터페이스로 래핑한 것이다.
그 래핑이 데스크톱 UI 패러다임으로 되어 있기 때문에, 다른 추상화로 이미 적응한
모바일 사용자에게는 이중의 인지 불일치가 생긴다.
Filestash가 “즉각적인 미리보기”와 “피드처럼 탐색”을 강조하는 것은, 이 인지 불일치를
줄이는 방향으로 설계됐기 때문이다.

### 셀프 호스팅의 역설: 복잡성을 통제하려다 복잡성을 만든다

Nextcloud와 Seafile이 “개인 사용에 지나치게 복잡하다”고 탈락한 것은 셀프 호스팅
생태계의 고질적 문제를 드러낸다.
셀프 호스팅을 선택하는 가장 큰 동기는 프라이버시와 통제권이다.
하지만 그 통제권을 얻기 위해 지불하는 비용이 복잡성이다.
복잡한 설정, 업데이트 관리, 보안 패치, 백업—이 모든 것이 개인이 직접 처리해야 한다.

Nextcloud는 단순한 파일 브라우저가 아니라 Google Workspace를 셀프 호스팅하려는
야망에서 탄생했다.
캘린더, 연락처, 사무 도구, 협업 기능—이 모든 것이 단순히 “파일을 폰에서 보고 싶다”는
요구에 과잉 응답한다.
이것은 오픈소스 프로젝트의 전형적인 피처 크리프(feature creep) 문제다.
처음에는 단순한 문제를 해결하려 시작했지만, 기여자들이 늘고 사용 사례가 다양해지면서
복잡도가 기하급수적으로 증가한다.
Dufs가 “단일 바이너리, 설정 없음”으로 주목받는 것은 이 복잡성 팽창에 대한 반동이다.

### 도구 선택의 계층화 전략: “하나로 모든 것을 해결하지 말라”

저자의 최종 구성—Filestash + FileBrowser + SSH—은 세 가지 다른 사용 맥락에 세 가지
다른 도구를 배치하는 계층화 전략이다.
이것은 단일 도구가 모든 요구를 충족해야 한다는 “실버 불릿” 사고방식의 거부다.
소프트웨어 설계 원칙 중 “단일 책임 원칙(Single Responsibility Principle)”과 유사하다.
각 도구가 하나의 역할을 잘 수행하도록 하고, 다른 역할에는 다른 도구를 쓴다.

이 전략이 가진 실용적 장점은 각 도구를 최상의 조건에서 사용할 수 있다는 것이다.
FileBrowser가 데스크톱에서 잘 동작한다는 사실은 유지하면서, 모바일에서는 모바일에
최적화된 Filestash를 쓴다.
Unix 철학—“한 가지를 잘 하는 프로그램들을 파이프로 연결하라”—이 도구 생태계 레벨에서
적용된 사례다.
하지만 이 전략의 비용도 있다.
유지보수해야 할 도구가 늘고, 각 도구의 보안 업데이트를 개별적으로 추적해야 하며,
도구 간 인증 체계가 분리될 수 있다.
계층화의 이익이 이 비용을 상회하는 시점을 판단하는 것이 셀프 호스팅의 진정한 도전이다.
