# SigNoz vs Grafana Stack — 오픈소스 관측성 플랫폼 비교

## 개요

오픈소스 관측성 플랫폼을 자체 호스팅으로 구축하려는 조직이 반드시 마주치는
두 가지 선택지가 SigNoz와 Grafana Stack이다.
SigNoz는 ClickHouse 단일 스토리지 위에 로그·메트릭·트레이스를 통합한
올인원 플랫폼이고,
Grafana Stack은 Loki(로그)·Mimir(메트릭)·Tempo(트레이스)를 Grafana UI로
묶는 조합형 구조다.
두 접근 방식은 "통합을 스토리지 레벨에서 달성할 것인가,
조회 레벨에서 달성할 것인가"라는 근본적인 설계 철학의 차이에서 출발한다.

표면적으로는 기능 비교처럼 보이지만, 실제로 이 선택은 팀이 어떤 종류의
운영 복잡도를 감수할 것인가를 결정하는 문제에 가깝다.
SigNoz를 고르면 ClickHouse 클러스터 하나를 깊이 파야 하고,
Grafana Stack을 고르면 서로 다른 스토리지 세 개를 각각 이해해야 한다.
어느 쪽도 복잡도를 없애지 않는다. 다만 복잡도의 형태가 다를 뿐이다.

## 분석

### 아키텍처: 단일 엔진 vs 신호별 전문 스토어

SigNoz는 ClickHouse 하나로 세 신호를 모두 수용한다.
로그는 `logs` 테이블, 메트릭은 `samples` 테이블, 트레이스는 `traces` 테이블에
들어가며, 세 테이블이 같은 ClickHouse 클러스터 안에 공존한다.
신호 간 상관 쿼리를 단일 쿼리 엔진 안에서 처리할 수 있고,
배포해야 하는 스토리지 컴포넌트가 하나뿐이라는 점에서 운영 표면이 좁다.

Grafana Stack은 각 신호에 최적화된 저장소를 분리한다.
Loki는 로그 인덱싱 비용을 낮추기 위해 레이블 기반 인덱스만 유지하고,
전문 검색(full-text search)은 인덱스 없이 grep 방식으로 처리한다.
Tempo는 트레이스를 오브젝트 스토리지(S3, GCS)에 Parquet flat file로 저장하며,
인덱스 없이 Trace ID 직접 조회를 기본으로 삼는다.
Mimir는 Cortex에서 파생된 장기 메트릭 저장소로, 수평 샤딩과
레플리케이션을 통해 Prometheus보다 훨씬 큰 스케일을 감당한다.
이 분리는 각 신호를 독립적으로 스케일할 수 있게 하는 대신,
신호 간 상관 분석을 위해 Grafana UI 레이어의 데이터 소스 연결에 의존한다.

### 카디널리티 처리 방식

카디널리티 문제는 두 플랫폼에서 형태가 전혀 다르게 나타난다.

Loki는 구조적으로 카디널리티에 민감하다.
레이블 집합의 고유 조합 수가 스트림 수를 결정하고,
스트림 수가 메모리 사용량과 인덱스 크기를 직접 좌우한다.
`pod`, `container`, `namespace` 수준의 레이블은 안전하지만,
`user_id`, `request_id`, `trace_id` 같은 고카디널리티 값을 레이블로 넣으면
Loki 인덱서가 폭주한다.
이 제약은 로그 수집 파이프라인 설계 단계에서 레이블 선택을 강제하고,
운영 중에 카디널리티가 예상치 않게 폭발하면 Loki 전체가 불안정해진다.

ClickHouse는 컬럼형 스토리지 특성상 고카디널리티 값을 컬럼으로 저장하는 데
훨씬 관대하다.
SigNoz는 로그의 구조화 필드를 ClickHouse 컬럼으로 자동 인덱싱하며,
고카디널리티 필드도 스캔 기반 필터로 처리할 수 있다.
단, 스캔 비용은 데이터 볼륨에 비례하므로 인덱스 없는 고카디널리티 필드 쿼리는
느려진다는 점을 인지해야 한다.

Mimir와 메트릭 카디널리티 문제는 별개다.
Prometheus 계열은 레이블 조합 수가 시계열 수를 결정하며,
Mimir도 이 특성을 그대로 이어받는다.
SigNoz 역시 메트릭을 ClickHouse에 저장하지만
내부적으로 시계열 식별자를 해시로 관리해 Prometheus 카디널리티 한계를
완전히 피해가지는 못한다.

### 설치와 운영 복잡도

SigNoz는 단일 Docker Compose 파일로 시작할 수 있어 초기 진입 장벽이 낮다.
필수 컴포넌트는 ClickHouse, OTel Collector, SigNoz 프론트엔드·백엔드 몇 개뿐이다.
운영 복잡도는 ClickHouse 클러스터 한 곳에 집중된다.
샤드·레플리카 구성, 머지 트리 튜닝, TTL 정책, 백업 전략이 단일 스토어 안에서
결정된다.

Grafana Stack은 컴포넌트마다 별도 설정과 스케일 전략이 필요하다.
Kubernetes 환경에서는 Loki Helm chart, Tempo Helm chart, Mimir Helm chart,
Grafana Helm chart를 각각 유지하고, 오브젝트 스토리지 버킷을 신호별로
구성하며, Grafana 데이터 소스 등록을 별도로 관리한다.
컴포넌트 간 버전 호환성도 따져야 한다.
Grafana Alloy(구 Grafana Agent)가 수집 계층을 통합하려 하지만,
각 스토리지 백엔드의 독립적인 운영 부담은 그대로 남는다.

반면 Grafana Stack은 컴포넌트 단위 교체가 자연스럽다.
메트릭만 Thanos로 바꾸거나, 로그를 Loki 대신 OpenSearch로 대체하거나,
트레이스를 Tempo 대신 Jaeger로 운영하는 선택적 교체가 가능하다.
SigNoz에서 ClickHouse를 다른 스토리지로 교체하는 것은 사실상 플랫폼 교체다.

### OpenTelemetry 지원

SigNoz는 OpenTelemetry를 제품 정체성의 핵심으로 삼는다.
OTel Collector를 수집 계층으로 지정하고, 자체 에이전트나 프로프라이어터리 SDK를
제공하지 않는다.
OTel 시맨틱 컨벤션이 SigNoz 스키마의 기반이므로, OTel로 계측한 데이터가
SigNoz UI에 자연스럽게 매핑된다.

Grafana는 OTel을 받아들이는 방향으로 움직이고 있으나 출발점이 달랐다.
Loki는 오랫동안 Promtail 기반 수집을 기본으로 삼아왔고,
Tempo는 OTel 외에도 Zipkin, Jaeger, OpenCensus 포맷을 병렬로 지원한다.
Mimir는 Prometheus Remote Write가 1차 수집 경로다.
이 레거시 경로들이 공존하는 구조는 기존 Zipkin·Jaeger 인프라와의 공존이
필요한 조직에는 실용적이지만,
OTel 단일 경로를 고집하는 조직에게는 설정 선택지가 오히려 혼란이 된다.

Grafana의 OTel 수용이 완전하지 않다는 점도 있다.
Loki의 구조화 메타데이터 지원, Tempo의 OTel 시맨틱 컨벤션 완전 준수,
Mimir의 OTel Metrics 지원은 모두 진행 중인 작업이고,
버전마다 지원 수준이 다르다.

### 쿼리 언어

SigNoz는 시각적 쿼리 빌더, PromQL, ClickHouse SQL을 함께 제공한다.
SQL에 익숙한 팀이라면 ClickHouse SQL로 임의의 관측성 쿼리를 표현할 수 있다.
`JOIN`, `GROUP BY`, `WINDOW FUNCTION`, 서브쿼리를 관측성 데이터에 직접 적용할 수
있다는 점은 특정 질문에 대해 강력한 표현력을 준다.

Grafana Stack은 신호별 전용 쿼리 언어를 사용한다.
로그는 LogQL, 메트릭은 PromQL, 트레이스는 TraceQL이다.
LogQL은 레이블 필터와 로그 파이프라인을 조합하는 방식으로 Loki의 스트림
모델에 맞게 설계됐고, TraceQL은 스팬 속성 기반으로 트레이스를 조회하는
전용 언어다.
각 언어가 해당 신호에 특화된 표현력을 제공하지만,
팀이 세 가지 언어를 모두 익혀야 한다는 학습 부담이 따른다.
또한 신호 간 조인이 필요한 복합 쿼리는 Grafana UI의 Explore 상관 기능에 의존하며,
프로그래밍 방식의 크로스 신호 분석은 별도 구현이 필요하다.

### 알림(Alerting)

SigNoz는 자체 알림 엔진을 내장하며, 쿼리 기반 임계치 알림을 UI에서 직접 구성한다.
알림 라우팅은 PagerDuty, Slack, OpsGenie 등과 연동되며,
ML 기반 이상 탐지는 2022년 기준 로드맵 항목으로 고정 임계치 기반이 기본이다.

Grafana는 Grafana Alerting(구 Unified Alerting)을 통해 모든 데이터 소스에서
알림을 생성할 수 있다.
Prometheus AlertManager와의 통합이 성숙하고,
Grafana OnCall을 통한 온콜 스케줄 관리, Incident 워크플로까지 연결된다.
알림 생태계의 성숙도와 커뮤니티 사례 문서 양은 Grafana가 압도적으로 많다.

### 고가용성과 재해 복구

SigNoz의 HA는 ClickHouse 클러스터의 HA와 동일하다.
ZooKeeper(또는 ClickHouse Keeper)를 이용한 레플리케이션,
샤드 분산, 분산 쿼리 라우팅을 직접 구성해야 한다.
ClickHouse HA는 잘 문서화되어 있지만, 초기 구성과 튜닝에는 ClickHouse 전문
지식이 전제된다.

Grafana Stack의 HA는 컴포넌트마다 독립적으로 설계한다.
Loki는 마이크로서비스 모드에서 각 역할(ingester, querier, compactor 등)을
수평 확장하고,
Tempo와 Mimir도 유사한 마이크로서비스 아키텍처를 가진다.
오브젝트 스토리지를 기반으로 하는 Tempo와 Mimir는 스토리지 레이어의 내구성을
클라우드 오브젝트 스토리지에 위임하므로, 단일 AZ 장애에 상대적으로 강하다.
Loki의 ingester는 상태를 갖는 컴포넌트라 HA 설계가 별도로 필요하다.

### 라이선스와 오픈 코어 구조

SigNoz는 코어를 MIT 라이선스로 공개하고 `ee/` 디렉터리에 엔터프라이즈 기능을
별도 라이선스로 분리하는 오픈 코어 구조다.
SSO, SAML, Query API 키, 세밀한 접근 제어(RBAC)가 이미 엔터프라이즈 등급으로
분류됐다.

Grafana OSS는 AGPLv3로 제공된다.
AGPLv3는 네트워크를 통해 서비스를 제공하는 경우에도 소스 공개 의무를 부과하므로,
Grafana를 SaaS로 재판매하려는 조직은 반드시 라이선스 검토가 필요하다.
Grafana의 엔터프라이즈 플러그인(엔터프라이즈 데이터 소스, Grafana OnCall,
Grafana IRM 등)은 별도 상용 라이선스다.
Grafana는 2021년 Apache 2.0에서 AGPLv3로 라이선스를 전환한 사례가 있어,
오픈 코어 경계 이동의 실증 사례를 이미 한 번 제공한 셈이다.

### 커뮤니티와 생태계

Grafana는 수백만 사용자를 보유한 성숙한 생태계를 가지고 있다.
GitHub 스타 65,000+, 커뮤니티 포럼 수십만 개의 스레드, 수백 개의 공식·
비공식 플러그인, Fortune 500 기업 사례 문서가 쌓여 있다.
Grafana Labs는 Loki, Tempo, Mimir, Pyroscope, Beyla 등을 포함하는 관측성
포트폴리오 전략을 가진 상장(IPO 전) 기업으로, 제품 연속성에 대한
신뢰 기반이 상대적으로 높다.

SigNoz는 GitHub 스타 20,000+(2024년 기준)로 빠르게 성장하고 있지만,
운영 중 예상치 못한 문제에 부딪혔을 때 참조할 수 있는 커뮤니티 지식의 양은
아직 Grafana Stack과 비교하기 어려운 수준의 격차가 있다.
Stack Overflow, Reddit, 블로그 포스트 수 모두 Grafana Stack이 압도적이다.

### 비용 모델

클라우드 서비스 비교에서 SigNoz Cloud는 메트릭 백만 샘플당 0.1달러 수준의
단가 균일화 구조를 제시하며 Datadog의 호스트 기반 과금보다 예측 가능성이 높다.
Grafana Cloud는 메트릭 1만 시계열 이상부터 과금하고, 로그와 트레이스는
데이터 볼륨 기반이다.
무료 티어 한도가 Grafana Cloud가 더 넉넉해서, 소규모 팀의 초기 탐색 비용이 낮다.

자체 호스팅에서는 두 플랫폼 모두 소프트웨어 자체 비용이 없고,
인프라 비용과 운영 인력 비용이 실질 비용이다.
Grafana Stack에서 Tempo와 Mimir가 오브젝트 스토리지를 사용하면
블록 스토리지 대비 GB당 비용이 낮아지지만, 오브젝트 스토리지 요청 수에 따른
API 비용이 예상보다 높아질 수 있다.

## 비평

### "통합"이라는 단어가 두 제품에서 완전히 다른 것을 가리킨다

이 비교에서 가장 먼저 해소해야 할 혼동이 있다.
두 제품 모두 "통합 관측성 플랫폼"을 표방하지만, "통합"의 의미가 다르다.

SigNoz의 통합은 스토리지 통합이다.
세 신호가 같은 ClickHouse 클러스터에 들어간다.
트레이스 ID로 로그를 조회하는 쿼리가 단일 ClickHouse 쿼리로 표현된다.

Grafana의 통합은 UI 통합이다.
신호는 분리된 스토어에 있지만 Grafana 대시보드에서 한 화면으로 엮인다.
트레이스에서 로그로 넘어가는 것은 Tempo가 Loki에 쿼리를 던지는 방식으로 구현되며,
두 스토어 사이의 연결 고리는 Trace ID가 로그에 레이블이나 구조화 필드로
포함되어 있어야 한다는 전제 조건이 있다.

어느 쪽이 "더 통합된" 것인지는 팀의 운영 병목이 어디에 있느냐에 따라 달라진다.
스토리지 관리 포인트를 줄이는 것이 목표라면 SigNoz가 유리하고,
쿼리 경험을 단일화하는 것이 목표라면 Grafana도 충분하다.
이 두 가지를 구분하지 않고 비교하면, 서로 다른 문제를 해결하는 도구를
같은 기준으로 평가하는 오류에 빠진다.

### Loki의 레이블 카디널리티 제한은 로그 수집 파이프라인을 설계 단계부터 강제한다

Loki의 레이블 카디널리티 제한은 기능 제약이 아니라 아키텍처 제약이다.
고카디널리티 값을 레이블로 넣으면 Loki 인덱서가 폭주하고 전체 시스템이
불안정해진다는 점은, 로그 파이프라인을 설계하는 시점에 레이블 선택을
엄격하게 결정해야 함을 뜻한다.

문제는 이 결정이 나중에 번복하기 어렵다는 데 있다.
운영 중에 새로운 레이블이 필요해지면, 기존 로그 스트림의 레이블 구조를
바꾸는 것은 데이터 재수집에 가깝다.
Loki를 처음 도입하는 팀이 이 제약을 충분히 인식하지 못한 채 레이블을
자유롭게 추가하다가 카디널리티 폭발을 겪는 사례는 커뮤니티에서 반복적으로
보고된다.

SigNoz의 ClickHouse 기반 로그 저장은 이 제약이 없다.
고카디널리티 필드도 구조화 컬럼으로 넣을 수 있고, 전문 검색도 ClickHouse의
`LIKE` 연산이나 전문 인덱스로 처리된다.
단, 인덱스 없는 필드의 대용량 스캔은 비용이 비례적으로 늘어난다.

### SigNoz의 ClickHouse SQL 유연성은 운영 복잡도를 숨긴다

ClickHouse SQL로 임의의 관측성 쿼리를 표현할 수 있다는 것은 분명한 강점이다.
하지만 이 유연성이 실제로 발동되는 순간은 대부분 SigNoz UI가 커버하지 못하는
복잡한 질문에 직면했을 때다.
그 순간 팀은 ClickHouse 스키마를 이해하고, 쿼리 최적화를 직접 판단하며,
인덱스와 파티션 구조를 고려해야 한다.

이는 관측성 플랫폼의 추상화가 무너지는 지점이다.
Loki나 Tempo는 쿼리 언어가 스토리지 모델에 맞게 설계되어 있어서, 복잡한 쿼리도
해당 언어의 관용 표현 안에서 처리된다.
SigNoz에서 복잡한 질문을 SQL로 직접 해결해야 할 때,
그것은 관측성 도구를 쓰는 것이 아니라 ClickHouse를 직접 다루는 것이다.
편의성과 제어권을 동시에 제공한다는 마케팅 문구와 달리,
깊이 파고들수록 ClickHouse 운영 역량이 전제 조건으로 드러난다.

### Grafana Stack의 조합형 강점은 동시에 조합형 책임이다

Grafana Stack은 컴포넌트를 독립적으로 교체할 수 있다는 점을 강점으로 내세운다.
하지만 이 유연성은 각 컴포넌트를 독립적으로 이해하고, 독립적으로 운영하고,
독립적으로 장애를 대응해야 한다는 뜻이기도 하다.

Loki의 레이블 카디널리티 문제, Tempo의 오브젝트 스토리지 설정,
Mimir의 샤딩 전략, Grafana 데이터 소스 연결은 각각 별도의 운영 지식 체계를 요구한다.
이 네 가지를 모두 이해하는 팀은 Grafana Stack의 유연성을 온전히 활용할 수 있다.
하지만 인프라 팀이 작거나 관측성 플랫폼 운영이 주 업무가 아닌 조직에서는,
조합의 자유도가 오히려 어디서 문제가 생겼는지 파악하기 어려운 복잡도의 원천이 된다.

컴포넌트가 분리되어 있다는 것은 장애 시 책임의 경계도 분리되어 있다는 뜻이다.
Loki ingester가 죽었을 때, Tempo compactor가 느려졌을 때,
Mimir querier가 메모리를 초과할 때, 각각 다른 진단 경로와 해결 방법이 필요하다.
이 복잡도는 기능 목록 비교에서는 보이지 않는다.

### SigNoz의 단일 스토어 구조는 장애 반경을 한 지점으로 집중시킨다

ClickHouse 단일 스토어는 운영 포인트를 줄이는 대신,
ClickHouse 클러스터 장애가 로그·메트릭·트레이스 세 신호를 동시에 잃게 만든다.
관측성 플랫폼이 전부 다운되는 순간은 정확히 대규모 인프라 장애가 발생했을 때와
겹칠 가능성이 높다.
장애 대응에 가장 필요한 순간에 가시성이 함께 사라지는 시나리오가 단일 스토어
구조에서 더 현실적이다.

Grafana Stack은 한 스토어에 문제가 생겨도 나머지 신호는 유지된다.
ClickHouse에 해당하는 Loki가 다운되어도 Mimir는 살아있어 메트릭 대시보드와
알림은 계속 작동한다.
이 격리 효과는 기능이 아니라 아키텍처 분리의 부산물이다.

### 두 제품 모두 샘플링 문제를 충분히 다루지 않는다

트레이스 볼륨이 커지면 필연적으로 샘플링이 필요해진다.
헤드 기반 샘플링은 드물게 발생하는 에러 트레이스를 놓칠 위험이 있고,
테일 기반 샘플링은 OTel Collector에 버퍼링·집계 부하를 얹는다.
SigNoz의 공식 자료는 이 문제를 거의 언급하지 않는다.
Tempo도 Tail Sampling Processor 설정을 지원하지만, 대용량 환경에서의 운영
경험 문서는 부족하다.

샘플링 전략은 관측성 플랫폼 선택만큼이나 중요한 결정인데,
두 플랫폼 모두 기능 목록과 가격 비교에 훨씬 많은 지면을 할애하고
샘플링 운영 경험은 사용자가 스스로 발견해야 하는 영역으로 남겨둔다.

### 커뮤니티 격차는 기능 격차보다 더 오래 지속된다

기능 격차는 빠른 개발 속도로 줄어들 수 있다.
SigNoz는 실제로 빠르게 기능을 추가하고 있다.
하지만 커뮤니티 지식 격차는 다르다.
Stack Overflow 답변 수, 블로그 포스트 수, 운영 사례 문서의 깊이는
단기간에 따라잡을 수 있는 종류가 아니다.

SigNoz를 프로덕션에 도입한 조직이 예상치 못한 문제에 부딪혔을 때,
Grafana Loki 사용자가 구글에서 10분 만에 찾는 해결책을
SigNoz 사용자는 직접 GitHub 이슈를 뒤지거나 슬랙 커뮤니티에서 기다려야 하는
상황이 여전히 빈번하다.
이 비용은 직접 측정되지 않지만 실제 운영 비용에 포함된다.

## 인사이트

### Grafana는 "선택지의 플랫폼"이고 SigNoz는 "선택의 제거"다

두 제품의 철학적 차이를 가장 명확하게 드러내는 표현이 이것이다.
Grafana는 기존 인프라에 이미 Prometheus, Loki, Jaeger, Elasticsearch 등이
혼재하는 조직에서 그것들을 하나의 UI 아래 묶는 플랫폼이다.
선택지를 제거하지 않는다. 오히려 모든 선택지를 수용하는 것이 강점이다.

SigNoz는 반대 방향을 택한다.
ClickHouse 하나, OTel 하나, 단일 UI 하나다.
이미 선택이 이뤄져 있고, 사용자는 그 선택에 올라탄다.
이 단순함은 "이미 통일된 환경을 구축하고 싶은" 팀에게는 강점이고,
"기존 이기종 환경에 관측성을 얹어야 하는" 팀에게는 맞지 않는 전제다.

어느 쪽 철학이 더 나은가가 아니라, 지금 조직의 상황이 어느 쪽 철학과
일치하는가를 먼저 파악하는 것이 선택의 출발점이다.

### 선택 기준은 기능 목록이 아니라 팀이 감수할 수 있는 장애의 종류다

어느 쪽이 더 나은가는 팀이 어떤 종류의 장애를 더 잘 다루는가에 달려 있다.

ClickHouse 클러스터 장애를 신속하게 진단하고 복구할 역량이 있는가?
SigNoz를 고를 조건이다.
Loki 카디널리티 폭발, Tempo compactor 지연, Mimir 샤드 불균형을 각각
독립적으로 대응할 역량이 있는가?
Grafana Stack을 고를 조건이다.

도구의 기능 목록이 아니라 팀의 장애 대응 역량 지도를 먼저 그리는 것이
선행 조건이다.
그 지도 없이 기능 비교표를 보면, 스펙에서 이기는 쪽을 고르게 된다.
하지만 스펙에서 이기는 쪽이 운영에서 이기는 쪽과 항상 같지 않다.

### 두 제품 모두 자체 호스팅 관측성의 근본 딜레마를 해결하지 않는다

자체 호스팅 관측성의 근본 딜레마는 이것이다.
관측성 플랫폼 자체도 누군가 모니터링해야 한다.
SigNoz가 다운되거나 Loki가 불안정해지면 정작 장애 대응에 필요한 가시성이
함께 사라진다.
상용 SaaS APM은 이 딜레마를 벤더의 책임으로 넘기지만,
자체 호스팅을 택하는 순간 조직은 관측 대상 시스템과 관측 시스템 두 가지의
가용성을 동시에 책임져야 한다.

SigNoz든 Grafana Stack이든 이 딜레마의 해결책이 아니라 다른 형태의 트레이드오프다.
SigNoz는 단일 스토어로 운영 포인트를 줄이는 대신 장애 반경을 집중시키고,
Grafana Stack은 컴포넌트 격리로 장애 반경을 분산하는 대신 운영 복잡도를 올린다.
어느 쪽을 선택하더라도, 관측성 플랫폼 전용 인프라를 별도 클러스터나 VM으로
분리하는 것은 선택이 아니라 전제 조건이다.

### 장기적으로 OTel 표준화가 두 플랫폼의 교체 비용을 낮춘다

OTel Collector가 수집 계층을 표준화하면, 백엔드 스토리지 교체 비용이 낮아진다.
SigNoz와 Grafana Stack 모두 OTel을 지원하는 방향으로 수렴하고 있기 때문에,
지금 어느 쪽을 선택하더라도 OTel 기반으로 계측을 구성해두면 나중에 백엔드를
교체하는 비용이 상대적으로 낮다.
이는 이 선택을 "영구적인 아키텍처 결정"이 아니라 "현재 팀 역량과 예산에 맞는
운영 가능한 선택"으로 프레이밍할 수 있게 한다.

OTel 표준화의 또 다른 함의는, 두 플랫폼의 차별화 공간이 점점 좁아진다는 것이다.
수집 계층이 표준화되면 경쟁은 스토리지 효율, 쿼리 성능, UI 경험, 운영 편의성으로
이동한다.
이 영역에서 Grafana는 성숙한 생태계를, SigNoz는 단순한 아키텍처를 앞세운다.
2–3년 후 이 격차가 어떻게 변할지는 두 팀의 실행 속도에 달려 있지만,
지금 선택이 3년 후에도 유효한 선택일 필요는 없다.
