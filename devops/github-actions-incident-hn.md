# GitHub Actions·Pages 장애 해커뉴스 토론

원문: <https://news.ycombinator.com/item?id=48278090>

## 요약

GitHub Actions와 Pages에서 동시 장애가 발생했고, 해커뉴스에 372개의 댓글이 달렸다.
스레드의 첫 번째 댓글은 장애 중 배포 파이프라인이 “계정이 정지됐습니다(Your account was suspended)”라는
메시지를 출력해 패닉이 발생했다는 사례로 시작한다.
기술적 장애 자체보다 그것이 촉발한 집단적 반응이 이 스레드의 핵심 컨텐츠다.

참여자 다수는 GitHub 의존도를 즉각 문제로 지목했다.
“단일 공급자에 모든 것을 맡기지 말라”는 조언이 반복됐고,
Forgejo, GitLab 자가 호스팅, CircleCI로의 복귀, 간단한 `make test && ./deploy.sh` 방식 등
구체적인 대안이 제시됐다.
그러나 GitHub의 네트워크 효과 — Pull Request, 오픈 소스 생태계, Copilot 통합 — 를 대체할 수단이 없다는
반론도 강하게 제기됐다.

장애 원인에 대한 추측은 AI/LLM 부하 급증으로 수렴했다.
한 참여자는 이번 주 자신의 저장소 클론 수가 3,383회 증가했는데, 약 10번의 에이전트 프롬프트에서 비롯됐을 것이라 추산했다.
이것이 시스템 전체로 확산되면 1,000배 수준의 사용량 폭발이 발생한다는 계산이 나왔다.
GitHub이 LLM 트래픽의 80~95%를 독점하는 최대 공급자이기 때문에 경쟁사 대비 불균형하게 타격받는다는 분석도 있었다.

모니터링과 상태 페이지에 대한 토론도 활발했다.
사용자들은 GitHub 상태 페이지가 공식 장애를 인정하기 15분 전에 이미 문제를 경험했다고 보고했다.
이에 대해 “500 에러 하나하나에 알람을 울려야 하는가”부터 “합성 프로브(synthetic probe)와 로그 분석을 결합한
다계층 모니터링”까지 경험적 논쟁이 이어졌다.
상태 페이지 업데이트 지연에는 SLA 페널티와 법적 함의를 고려한 임원급 승인 절차가 개입한다는 내막도 공유됐다.

## 분석

### 분산 VCS 위에 구축된 중앙화 인프라의 아이러니

스레드에서 가장 날카로운 지적은 `gopalv`의 댓글에 있다.
“GitHub이 중앙화된 건 Pull Request와 Actions가 메인 브랜치 병합의 전제조건이 됐을 때부터다.”
Git은 본질적으로 분산 버전 관리 시스템이다. 로컬 복사본을 가지면 오프라인에서도 완전한 히스토리를 유지할 수 있다.
그러나 GitHub이 협업의 중심이 되면서 코드 저장소가 아닌 코드 리뷰·CI·배포 오케스트레이션이 단일 플랫폼에 결합됐다.
분산 시스템을 기반으로 중앙화된 의존성이 쌓인 구조다.

`marricks`는 이를 유머로 포착했다.
기술 계층을 쌓다 보면 처음에 해결하려던 중앙화 문제를 그대로 재현하게 된다는 것이다.
자가 호스팅 GitLab을 도입해도, 그 GitLab이 SAML 인증, 오브젝트 스토리지, 클라우드 러너에 의존하는 순간 단일 장애점은 다시 생긴다.

### AI 에이전트가 만드는 새로운 트래픽 패턴

`ModernMech`의 관찰은 구체적이다.
10번의 에이전트 프롬프트가 저장소 3,383회 클론을 일으켰다.
에이전트는 컨텍스트 수집을 위해 저장소를 반복적으로 클론하고, 의존성을 설치하고, 테스트를 실행하며, 배포 파이프라인을 트리거한다.
인간 개발자가 하루에 한두 번 발생시키던 I/O를 에이전트는 분당 발생시킨다.

이것은 단순한 부하 증가가 아니라 트래픽 패턴의 구조적 변화다.
기존 용량 계획 모델은 인간의 행동 패턴을 기반으로 하기 때문에, 에이전트가 만드는 버스티(bursty)하고 자동화된
트래픽 패턴을 예측하지 못한다. 캐싱·레이트 리미팅·오토스케일링 전략이 모두 재검토될 필요가 있다.

### 상태 페이지의 정치경제학

사용자보다 늦게 장애를 인정하는 상태 페이지는 기술적 문제가 아니다.
`re-thc`는 핵심을 짚었다: 공식 장애 인정은 SLA 페널티를 트리거하기 때문에 법적 검토와 임원 승인이 선행된다.
`swiftcoder`는 AWS에서도 마찬가지라고 확인했다 — 임원급 승인과 커뮤니케이션 검토 없이 공개 장애 공지가 나오지 않는다.

`simonjgreen`은 한 발 더 나아가 상태 페이지는 마케팅 팀이 깊이 관여한다고 밝혔다.
장애 인정의 시점, 표현, 범위 모두가 브랜드 관리의 영역이다.
기술적으로 장애를 최소화하는 것과 법적·마케팅적으로 최적화된 시점에 장애를 인정하는 것은 다른 최적화 문제다.

## 비평

### 강점: 실무 경험에서 나온 다층적 논쟁

이 스레드의 가치는 이론이 아니라 실무 경험의 충돌에 있다.
“모든 500 에러에 알람을 울려야 한다”는 주장과 “그러면 팀이 탈진한다”는 반론이 각각 실제 사례를 근거로 전개된다.
Jane Street의 다섯 나인 초과 신뢰도 사례, 핀테크 회사의 엄격한 SLA 운영 경험, IBM Cloud의 소진된 에러 버짓 사례 등
다양한 업계 관행이 한 스레드에서 비교된다.

### 약점: 개인 프로젝트와 엔터프라이즈 컨텍스트의 혼재

`nomilk`의 “로컬에서 테스트를 직접 실행한다” 접근과 엔터프라이즈 CI/CD 논쟁이 같은 맥락에서 전개되면서
혼선이 생긴다.
개인 프로젝트에서 10년간 CI 없이 문제없었다는 경험은 수십 명의 기여자가 있는 오픈 소스 프로젝트나
규정 준수가 필요한 엔터프라이즈 환경에 적용되지 않는다.
토론의 많은 부분이 이 컨텍스트 불일치로 인해 평행선을 달린다.

### 보완: 부하 급증의 책임 귀속 문제

AI 에이전트 트래픽이 장애 원인이라는 추측이 지배적이지만, 이를 뒷받침하는 공식 데이터는 없다.
그리고 이 가설이 맞다면 책임의 구조가 복잡해진다.
GitHub Actions를 AI 에이전트 작업 루프에 사용하도록 장려한 것은 GitHub 자신이다.
Copilot, GitHub Models, 에이전트 통합 모두 GitHub이 적극적으로 밀고 있는 방향이다.
에이전트가 만드는 부하에 대한 인프라 투자를 충분히 선행하지 않은 채 에이전트 사용을 촉진한 결과를
사용자 탓으로 돌릴 수는 없다.

## 인사이트

### 분산 기술 위의 중앙화는 언제나 단일 장애점을 재생산한다

Git은 분산 설계의 모범 사례로 꼽힌다.
모든 클론이 완전한 히스토리를 가지고, 단일 서버의 장애가 작업을 멈추지 않도록 설계됐다.
그런데 2025년의 현실은 Git 저장소를 사용하는 개발팀 대다수가 GitHub 하나가 다운되면 일을 멈춘다.

이것은 GitHub의 실패가 아니라 협업 도구가 확장되는 방식의 구조적 패턴이다.
이메일은 분산 프로토콜이지만 Gmail 장애는 전 세계 커뮤니케이션을 마비시킨다.
DNS는 계층적 분산 시스템이지만 Cloudflare 장애는 인터넷의 상당 부분을 다운시킨다.
HTTP는 분산 프로토콜이지만 AWS ap-northeast-1 가용 영역 장애는 한국의 수많은 서비스를 동시에 중단시킨다.

분산 기술은 단일 장애점을 없애는 것이 아니라 단일 장애점의 위치를 올린다.
하위 계층의 분산성이 상위 계층의 중앙화를 막지 못한다.
오히려 “어차피 기반 기술이 분산되어 있으니까”라는 안도감이 상위 계층의 중앙화를 가속할 수 있다.
새로운 협업 도구나 플랫폼을 설계할 때 “기반 기술이 분산되어 있는가”보다
“이 계층에서의 중앙화가 어떤 새로운 단일 장애점을 만드는가”를 먼저 물어야 한다.

### AI 에이전트는 인프라의 전제 조건을 조용히 무효화한다

소프트웨어 인프라의 용량 계획은 인간 행동 모델에 기반한다.
하루 중 트래픽 피크 시간, 배포 빈도, 저장소 클론 패턴, CI 실행 분포 — 이 모든 것이
인간 개발자의 작업 리듬을 전제로 설계된다.
에이전트는 이 전제를 소리 없이 무효화한다.

에이전트는 쉬지 않는다. 시간대 개념이 없고, 업무 시간 트래픽 패턴이 없다.
하나의 프롬프트가 수천 번의 API 호출과 저장소 클론을 일으킨다.
에이전트 오케스트레이터는 병렬로 수십 개의 서브에이전트를 실행하고, 각각이 독립적으로 전체 CI 파이프라인을 트리거한다.
인간 팀이 하루에 10번 실행하던 배포 파이프라인을 에이전트가 시간당 10번 실행하기 시작하면,
기존에 충분했던 인프라가 갑자기 10배 부족해진다.

더 근본적인 문제는 이 변화가 점진적이지 않다는 것이다.
에이전트 도입은 팀 단위로, 제품 단위로 일어난다. 어느 날 한 팀이 에이전트 기반 개발을 도입하면
그 팀의 인프라 소비량이 하루아침에 10배 증가할 수 있다.
기존의 용량 계획 방법론 — 과거 트렌드 기반 외삽, 계절성 분석, 선형 성장 가정 — 이 모두 유효하지 않게 된다.
에이전트 시대의 인프라 설계는 평균 부하가 아니라 에이전트 부하가 최대값으로 치솟는 최악의 시나리오를
기본 전제로 삼아야 한다.

### 플랫폼 의존성의 전환 비용은 기술이 아니라 생태계에 있다

“GitHub 대신 자가 호스팅 Forgejo를 쓰면 된다”는 조언이 스레드 전반에 걸쳐 반복된다.
기술적으로는 맞는 말이다. Forgejo는 GitHub 호환 API를 제공하고, 마이그레이션 도구도 있다.

그러나 `coldpie`가 정확히 지적했듯이, GitHub은 저장소 호스팅이 아니다.
GitHub은 오픈 소스 기여자들이 Pull Request를 보내는 곳이고,
의존하는 라이브러리의 이슈 트래커가 있는 곳이며,
Copilot 코드 제안이 통합된 곳이고,
Actions 마켓플레이스에서 재사용 가능한 워크플로우를 찾는 곳이다.
자가 호스팅 Forgejo는 이 생태계를 함께 가져오지 않는다.

이것은 플랫폼 전환 비용의 구조를 보여준다.
기술적 전환 비용은 낮아지고 있다. 호환 API, 마이그레이션 도구, 오픈 소스 대안이 성숙했다.
그러나 생태계 전환 비용은 오히려 높아지고 있다.
GitHub Actions 마켓플레이스, Dependabot, GitHub Packages, Copilot 통합, Security Advisories — 이
각각의 기능이 대체 불가능한 생태계 결합력을 만든다.

플랫폼이 성숙할수록 전환 비용의 무게 중심은 기술에서 생태계로 이동한다.
이것이 “플랫폼 락인(lock-in)”이 자가 호스팅 옵션이 충분히 존재하는 오늘날에도
여전히 강력한 이유다. 잠금 효과는 기술적 장벽이 아니라 생태계 이탈 비용에서 온다.
새로운 플랫폼을 구축하거나 도입할 때, 락인을 평가하는 올바른 질문은
“대안적 기술이 있는가”가 아니라 “이 플랫폼의 고유한 생태계 가치가 무엇인가”다.
