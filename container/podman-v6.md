# Podman v6.0.0 소개

원문: <https://blog.podman.io/2026/07/introducing-podman-v6-0-0/>

HN 토론: <https://news.ycombinator.com/item?id=48762098> (505점, 202개 댓글)

## 요약

Podman 팀은 2026년 7월 2일 v6.0.0을 공식 발표했다.
GitHub 릴리스로 먼저 배포되었으며 패키지 매니저 반영은 순차적으로 이루어질 예정이다.
Mohan Boddu가 작성한 이 발표 글은 “프로젝트 핵심 인프라 현대화, 보안 강화, 사용자 경험 개선을 위한 수개월의 작업”을 요약한다.

주요 변경의 첫 번째 축은 네트워킹 스택 현대화다.
슬리르프4넷(slirp4netns)과 iptables 기반 구성에서 Netavark, Pasta, nftables로 전환을 완료했다.
여기에 더해 루트리스(rootless) 컨테이너의 오랜 한계를 해결하는 Pesto 루트리스 포트 포워딩이 실험적(experimental) 기능으로 추가됐다.
Pesto는 커스텀 네트워크 위의 루트리스 컨테이너에서 소스 IP를 보존하는 기능을 제공한다.

두 번째 축은 Podman Machine 개선이다.
macOS와 Windows에서 VM을 통해 Linux 컨테이너를 실행하는 Podman Machine이 멀티 프로바이더 경험을 더 원활하게 지원하도록 개선됐으며,
VM 환경을 직접 관리하는 `podman machine os update` 명령어가 새로 추가됐다.

세 번째 축은 Quadlet의 대규모 업데이트다.
systemd 유닛 파일로 컨테이너를 선언하는 Quadlet이 REST API 지원을 갖추었고,
연관 파일 추적(file tracking) 기능이 개선됐다.
`.volume` 단위의 기능이 확장됐으며, 배포판 패키징을 위한 추가 검색 경로(search paths)가 도입됐다.

그 밖에 멀티유저 환경 관리자를 위한 설정 파일 처리 방식이 개선됐으며,
Docker API 호환성 업데이트와 명령어 출력 형식 개선으로 Docker에서 Podman으로의 전환 경험이 향상됐다.

## 분석

### slirp4netns에서 Pasta로, iptables에서 nftables로의 전환이 갖는 의미

슬리르프4넷는 사용자 공간(userspace)에서 TCP/IP 스택을 에뮬레이션하는 방식으로 루트리스 네트워킹을 구현했다.
사용자 공간 처리는 커널 네트워킹에 비해 성능 오버헤드가 크고, 고급 네트워킹 기능 지원에 제약이 있었다.
Pasta는 같은 사용자 공간 방식을 유지하면서도 더 효율적인 구현을 제공하고,
특히 소스 IP 보존 문제에 접근한다는 점에서 직접적인 후계자다.

nftables로의 전환은 Podman만의 결정이 아니다.
Linux 커널 커뮤니티에서 iptables는 이미 레거시로 취급되기 시작했고,
Red Hat은 RHEL 9부터 iptables를 deprecated 처리했다.
배포판과 커널이 먼저 방향을 정했고, Podman은 이 흐름을 v6.0.0에서 공식화한 것이다.
이 전환은 Netavark(Rust로 작성된 CNI 대체 네트워크 드라이버)와 결합하여
Podman 네트워킹 스택 전체를 현대적인 기반 위에 올려놓는다.

Netavark와 Pasta의 조합은 루트 권한 없이 실행되는 컨테이너의 네트워킹 품질을
기존 CNI+iptables 수준에 더 가깝게 끌어올리려는 시도이기도 하다.
루트리스 컨테이너가 보안 측면에서 장점이 있음에도 기업 환경에서 채택이 더딘 이유 중 하나가
네트워킹 기능의 제약이었기 때문에, 이 전환은 단순한 유지보수 이상의 의미를 갖는다.

### Quadlet이 Docker Compose와 다른 길을 걷는 이유

Quadlet은 Podman 고유의 기능으로, systemd 유닛 파일(`.container`, `.pod`, `.volume`, `.network`)로
컨테이너를 선언하고 관리한다.
Docker Compose가 별도의 YAML DSL과 compose 데몬을 필요로 하는 것과 달리,
Quadlet은 기존 Linux 운영 체계인 systemd 내에 컨테이너 관리를 흡수한다.
systemd가 서비스 수명주기, 의존성, 재시작 정책을 관리하듯이 컨테이너도 동일하게 관리된다.

v6.0.0에서 Quadlet에 REST API 지원이 추가된 것은 Quadlet의 역할을 확장한다.
CLI와 systemd 유닛 파일만으로 관리하던 컨테이너를 프로그래매틱하게 조작할 수 있게 된다.
이는 Quadlet을 단순한 선언형 구성 도구에서 오케스트레이션 레이어로 격상시키는 포석이다.

배포판 패키징을 위한 추가 검색 경로 지원도 주목할 만한 변화다.
배포판 유지관리자(packager)가 Quadlet 유닛 파일을 패키지에 포함시켜 배포하기 쉬워진다는 것은,
컨테이너 실행이 배포판 생태계와 더 긴밀하게 통합됨을 의미한다.
Fedora, RHEL, Debian 계열에서 Quadlet 기반 컨테이너 패키징이 늘어날 가능성이 있다.

2년간 홈 서버를 Podman + Quadlet으로 운영 중인 muti는 v6.0.0 릴리스 노트에서 실용적인 내부 개선을 발견했다.[^muti]
v5.6.0에서 추가된 `podman quadlet list`는 현재 실행 중인 Quadlet과 연관 컨테이너를 한 번에 확인하는 운영 명령어다.
v5.8.0에서 추가된 `podman system migrate --migrate-db`는 BoltDB에서 SQLite로의 데이터베이스 마이그레이션을 수동으로 수행하는 도구이며,
v6.0.0에서는 이 마이그레이션이 자동화됐다.
Podman의 내부 인프라 현대화가 네트워킹 스택 전환에만 국한되지 않으며, 데이터베이스 레이어까지 포괄하는 리팩토링임을 확인해 준다.

### Podman Machine의 멀티 프로바이더 전략이 드러내는 경쟁 구도

macOS와 Windows에서 Docker Desktop이 독점적이다시피 해온 영역을 Podman Machine이 공략하고 있다.
Apple Virtualization Framework, QEMU, WSL2 등 다양한 VM 프로바이더를 지원하는 것은
특정 VM 기술에 종속되지 않겠다는 선택이다.
Docker Desktop이 자체 VM 레이어를 고집하며 macOS 네이티브 경험과 마찰을 빚어온 것과 대조적이다.

`podman machine os update` 명령어는 Podman Machine VM의 OS를 직접 업데이트할 수 있게 한다.
이전에는 VM OS를 갱신하려면 Machine을 재생성하거나 수동으로 작업해야 했다.
VM 수명주기 관리가 CLI 하나로 가능해진다는 것은
개발 환경과 CI/CD 파이프라인에서 Podman Machine 관리의 자동화를 단순화한다.

victor_vhv는 macOS에서 Podman Machine을 사용하면서 랜덤한 응답 불가 현상과 아키텍처 차이로 인한 빌드 불일치를 경험했다고 보고했다.[^victor_vhv]
동일한 워크로드에서 OrbStack은 안정적이었다는 비교는,
Podman Machine의 멀티 프로바이더 전략이 방향성은 옳지만 macOS 실사용 경험의 안정성에서 아직 격차가 있다는 것을 보여준다.
v6.0.0의 멀티 프로바이더 개선이 이 문제를 해결했는지는 발표문만으로는 확인되지 않는다.

## 비평

### v6.0.0임에도 실제 breaking change 목록이 발표문에 없다

SemVer(Semantic Versioning) 기준에서 메이저 버전 증가는 하위 호환을 깨는 변경을 포함한다.
v5에서 v6으로의 전환은 반드시 마이그레이션 안내가 필요한 변경 사항을 수반한다.
블로그 포스트는 설정 파일 변경에 대해 “별도 블로그 포스트”를 언급하지만,
어떤 API, 설정, 명령어, 동작이 변경 또는 제거됐는지 발표문 자체에 명시하지 않는다.

실무 관점에서 릴리스 발표 글에서 가장 중요한 정보는 “무엇이 새로 생겼는가”가 아니라
“기존 설정 중 무엇이 더 이상 동작하지 않는가”다.
특히 slirp4netns에서 Pasta로, iptables에서 nftables로의 전환은
기존 네트워킹 설정이나 방화벽 규칙을 사용하는 사용자에게 직접적 영향을 미칠 수 있다.
이 정보가 별도 문서로 분리됨으로써, 릴리스 발표 글은 마케팅 문서의 성격이 강해지고
실제 업그레이드를 준비하는 운영자에게 필요한 정보를 제공하지 못한다.

rsyring은 Ubuntu에서 최신 Podman을 배포판 저장소 없이 설치하는 것이 사실상 불가능하다는 문제를 수년간 도입 장벽으로 꼽았다.[^rsyring]
배포판 저장소의 Podman 버전은 항상 최신보다 뒤처지고, 최신 버전을 사용하려면 복잡한 절차가 필요하다.
v6.0.0 발표가 “배포판 패키징을 위한 추가 검색 경로”를 언급하지만, 이것은 기존 패키지를 더 잘 찾기 위한 기능이지 Ubuntu 지원을 보강하는 것이 아니다.
Docker는 공식 apt 저장소를 통해 Ubuntu에서 항상 최신 버전을 설치할 수 있다.
이 비대칭이 해소되지 않는 한, Ubuntu 기반 환경에서 Podman의 진입 장벽은 릴리스 품질과 무관하게 남는다.

### “실험적” 기능을 릴리스 발표의 주요 항목으로 내세우는 것의 부담

Pesto 루트리스 포트 포워딩은 “experimental”로 명시됐지만,
릴리스 발표 글에서 네트워킹 현대화의 핵심 기능 중 하나로 소개된다.
실험적 기능은 API가 바뀌거나 삭제될 수 있으며, 프로덕션 사용을 보장하지 않는다.

릴리스 발표에서 실험적 기능을 주요 하이라이트로 배치하면
사용자가 해당 기능의 상태를 오해할 위험이 있다.
블로그 글의 독자 중 상당수는 실험적/안정적 구분을 세부적으로 읽지 않으며,
“Podman v6에서 소스 IP 보존이 된다”고 이해할 가능성이 있다.
실험적 기능이 예상대로 동작하지 않을 때, 프로젝트에 대한 신뢰도가 타격을 받는 것은
기능 자체보다 발표 방식의 문제다.

### Docker 호환성 강조가 Podman의 독자적 가치 제안을 희석한다

Podman의 구조적 차별화는 세 가지다.
데몬리스(daemonless) 아키텍처, 루트리스 컨테이너, systemd 통합(Quadlet).
이 세 가지는 Docker가 구조적으로 제공하기 어려운 기능이다.

그런데 릴리스 발표 글에서 Docker 호환성 개선을 꾸준히 주요 항목으로 포함하면,
Podman의 포지셔닝이 “더 나은 Docker”로 고정될 위험이 있다.
“더 나은 Docker”는 Docker 생태계가 표준으로 남는 한 Podman이 항상 종속 변수임을 암묵적으로 인정하는 것이다.
실제로 Podman을 처음 도입하는 조직이 “Docker 명령어가 그대로 동작하는지”를 첫 번째 기준으로 삼는 한,
Quadlet 생태계나 루트리스 보안 모델 같은 Podman 고유의 가치는 후순위로 밀린다.
이 발표가 Docker 호환성과 Podman 고유 기능 사이의 균형을 어떻게 잡느냐는
프로젝트의 장기 정체성에 영향을 미치는 서술 전략이다.

PufPufPuf는 Podman이 Docker 호환을 표방하면서도 세부 동작에서 미묘한 차이가 있어 나중에 문제로 돌아온다고 지적했다.[^PufPufPuf]
“호환된다”고 선언된 기능에서 예상치 못한 차이가 발생하면, 단순한 버그보다 신뢰 훼손이 더 크다.
반면 cdmckay는 Docker Desktop의 메모리 문제로 Podman으로 전환했을 때 docker-compose.yml을 그대로 사용했으며 아무 변경도 필요 없었다고 보고했다.[^cdmckay]
두 경험이 공존한다는 것은 Docker 호환성의 경계가 워크로드 유형과 복잡도에 따라 달라진다는 것을 의미한다.
“호환됩니다”와 “호환되지 않을 수 있습니다”가 동시에 사실인 상황에서, 이를 단일 메시지로 전달하는 것은 본질적으로 어렵다.

## 인사이트

### 컨테이너 런타임 네트워킹 스택의 교체 주기가 커널 생태계와 동기화되고 있다

slirp4netns→Pasta, iptables→nftables 전환은 Podman이 단독으로 결정한 기술 선택이 아니다.
Linux 배포판들이 nftables로 전환하고, RHEL이 iptables를 deprecated 처리하며,
커널 커뮤니티가 nftables를 중심으로 정책을 재편하는 흐름의 연장선이다.
컨테이너 런타임이 이제는 커널과 배포판의 생태계 변화에 긴밀하게 맞물려 움직인다는 것을 보여준다.

이 동기화는 양날의 검이다.
커널 표준을 따름으로써 장기 유지보수 부담이 줄고 커뮤니티 지원을 자연스럽게 받을 수 있다.
그러나 커널 생태계가 전환하기로 결정한 시점이 곧 컨테이너 런타임도 전환해야 하는 시점이 되면서,
개별 프로젝트가 독자적인 로드맵을 가지기 어려워진다.
v6.0.0의 타이밍이 2026년인 것은 기술적 준비도(readiness)와 커뮤니티 타이밍이 모두 충족된 결과이며,
이 패턴은 향후 컨테이너 런타임의 변화 속도를 커널 생태계 기준으로 예측할 수 있게 한다.

### Quadlet의 REST API 추가는 Kubernetes 없는 컨테이너 오케스트레이션의 공백을 겨냥한다

Kubernetes는 멀티노드 클러스터 오케스트레이션의 사실상 표준이 됐지만,
단일 호스트 또는 소규모 환경에서 Kubernetes는 운영 복잡성 대비 효용이 낮다.
Docker Compose가 이 영역을 담당해왔지만, Compose는 systemd와 통합되지 않는 별도의 레이어다.

Quadlet은 systemd 유닛 파일로 컨테이너를 정의함으로써
운영자가 기존에 알고 있는 서비스 관리 패러다임 안에서 컨테이너를 다룰 수 있게 한다.
여기에 REST API 지원이 추가되면 Quadlet은 프로그래매틱 관리의 게이트웨이가 되고,
이는 Ansible, Terraform 같은 인프라 자동화 도구와의 통합 경로를 만든다.
Kubernetes 없이도 컨테이너 수명주기를 코드로 관리하는 인프라 패턴이
Quadlet을 중심으로 구체화될 수 있는 조건이 갖춰지는 것이다.

Quadlet은 Red Hat 생태계(RHEL, Fedora, CentOS Stream)와 특히 잘 맞는다.
배포판 패키징 경로가 추가된 v6.0.0 이후, RHEL 기반 서버에 컨테이너 기반 소프트웨어를
Quadlet 유닛으로 패키징하여 배포하는 방식이 증가할 것이다.
이는 RPM 패키지 → systemd 서비스로 이어지던 전통적인 Linux 소프트웨어 배포 흐름에
컨테이너 레이어가 자연스럽게 결합되는 경로다.

### 루트리스 컨테이너의 기업 도입 장벽이 낮아지는 속도가 보안 정책 변화를 앞서기 시작했다

루트리스 컨테이너는 호스트 시스템에 대한 공격 표면을 줄이고, 컨테이너 탈출 취약점의 영향을 제한한다.
그러나 기업 환경에서 루트리스 컨테이너 채택을 가로막아온 실질적 장벽이 있었다.
소스 IP 보존 불가, 일부 포트 포워딩 제약, 퍼포먼스 오버헤드가 그 목록이다.

Pesto 루트리스 포트 포워딩이 안정화되면, 소스 IP 기반 접근 제어나 보안 감사(audit) 로그를
요구하는 금융, 의료, 공공 부문 환경에서 루트리스 컨테이너가 현실적인 선택지가 된다.
이 기능이 실험적 상태를 벗어나는 시점에 기업 보안 팀이 루트리스를 의무화하는 정책을
도입하기 시작한다면, Podman은 이 시장에서 Docker에 대한 구조적 우위를 갖는다.
Docker는 데몬 기반 아키텍처 특성상 루트리스를 완전하게 지원하기 어렵기 때문이다.

이 흐름의 역설은, 루트리스 컨테이너의 기업 도입이 빨라지는 속도가
보안 정책이 이를 요구하기 시작하는 속도보다 빠를 수 있다는 것이다.
기술이 준비됐을 때 정책이 뒤따르는 경우, 정책은 기술이 만들어놓은 사실관계를 추인하는 형태가 된다.
Pesto의 안정화 타이밍이 기업 보안 표준의 다음 개정 주기와 어떻게 맞물리는지가
루트리스 컨테이너 보급의 실질적 속도를 결정할 것이다.

utopiah는 Podman에 대한 인식이 “좋은 아이디어지만 동작하지 않는다”에서 “99.99%의 사례에서 Docker와 동등하게 동작한다”로 바뀐 자신의 경험을 공유했다.[^utopiah]
남은 0.01%는 구체적인 엣지 케이스로 존재하며, podman-compose의 특정 이슈가 예시로 언급됐다.
이 궤적은 기술 성숙도의 비선형성을 보여준다.
“거의 다 된다”와 “실제로 쓸 수 있다” 사이의 간극을 메우는 마지막 국면이 가장 어렵고,
Podman이 그 국면을 통과했다는 것이 사용자층 전환의 실질적인 임계점이었다.
