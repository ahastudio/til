# Sandboxd: 자체 호스팅 개발 샌드박스 엔진

<https://sandboxd.io/>

<https://github.com/tastyeffectco/sandboxd>

AI 앱 빌더 제품이 격리된 클라우드 개발 환경과 미리보기 URL을
제공할 수 있게 해 주는 오픈 소스 백엔드 엔진이다.
Docker 한 대로 단일 머신에 배포할 수 있다(베타, MIT 라이선스,
약 657 스타, 주 언어 Go 95.7%).

## 소개

Sandboxd는 멀티 테넌트 코딩 환경의 인프라 복잡성을 대신 처리한다.
핵심 동작은 네 가지다.
첫째, 독립된 파일시스템과 메모리 제한을 가진 사설 Linux 컨테이너로
샌드박스를 격리 생성한다.
둘째, OpenCode와 Claude Code CLI를 샌드박스 안에서 실행해 프롬프트를
처리한다.
셋째, 샌드박스에서 도는 애플리케이션을 공유 가능한 미리보기 URL로
즉시 노출한다.
넷째, 유휴 샌드박스는 잠들어(stop-on-idle) 메모리를 비우고, 접근이
오면 자동으로 깨어난다(wake-on-request).

설계 철학은 “지루한(boring) 신뢰성”이다.
전체 VM이 아니라 강화된 Docker 격리를 쓰고, Kubernetes 없이 SQLite·
Docker·Traefik만으로 단일 서버에 배포한다.
이 단순화는 스타트업 단계에 적합하며, 규모가 커지면 그때 강화한다는
입장이다.
신뢰할 수 있는 코드를 전제로 하므로, 신뢰할 수 없는 코드를 돌리려면
VM 수준의 추가 격리가 필요하다고 명시한다.

## 명세

아키텍처는 네 층으로 구성된다.

| 구성 요소           | 역할                                          |
| ------------------- | --------------------------------------------- |
| Traefik             | 엣지 라우터. URL 자동 라우팅과 TLS 처리       |
| Sandboxd 컨트롤플레인 | Go 단일 바이너리. Docker CLI로 샌드박스 수명 관리 |
| SQLite              | 영속 상태 저장소. 부팅 시 Docker를 DB에 수렴 |
| Docker 컨테이너     | 격리된 샌드박스 환경. 공유 네트워크 접근       |

주요 API 엔드포인트는 다음과 같다.

| 메서드·경로                                      | 용도                          |
| ------------------------------------------------ | ----------------------------- |
| `POST /sandbox`                                  | 포트·환경변수로 샌드박스 생성 |
| `GET /sandboxes`                                 | 전체 샌드박스 목록            |
| `POST /sandbox/{id}/exec`                        | 명령 실행                     |
| `POST /v1/sandboxes/{id}/tasks`                  | 프롬프트로 코딩 에이전트 실행 |
| `GET /v1/sandboxes/{id}/tasks/{taskId}/events`   | 에이전트 진행 SSE 스트림      |
| `GET/PUT /v1/sandboxes/{id}/files`               | 워크스페이스 파일 관리        |
| `POST /v1/sandboxes/{id}/stop`                   | 중지(워크스페이스 보존)       |
| `DELETE /sandbox/{id}`                           | 컨테이너 파기, 워크스페이스 유지 |
| `POST /sandbox/{id}/purge`                       | 파기 + 워크스페이스 삭제      |

## 사용법

요구 사항은 Linux 위의 Docker Engine과 Compose 플러그인이다.

```bash
git clone https://github.com/tastyeffectco/sandboxd.git
cd sandboxd
./install.sh
curl http://127.0.0.1:9090/healthz  # "ok" 반환
```

샌드박스를 만들고 코딩 에이전트를 실행하는 흐름은 다음과 같다.

```bash
# 샌드박스 생성
curl -XPOST http://127.0.0.1:9090/sandbox \
  -H 'content-type: application/json' \
  -d '{"ports":[3000]}'

# 코딩 에이전트 실행
curl -XPOST http://127.0.0.1:9090/v1/sandboxes/{ID}/tasks \
  -H 'content-type: application/json' \
  -d '{"prompt":"create a Vite todo app on port 3000","agent":"opencode"}'
```

미리보기는 `http://s-<id>-3000.preview.localhost`로 접근한다.
공개 배포 시에는 와일드카드 DNS(`*.preview.yourdomain.com`)를 호스트로
지정하고, `traefik/traefik.yml`에서 TLS(Let's Encrypt 권장)를 켜고,
`.env`에 `PREVIEW_DOMAIN`·`PREVIEW_TLS=true` 등을 설정한다.
중요한 점은 기본값이 `SANDBOXD_API_AUTH_DISABLED=true`라는 것이다 —
프로덕션에서는 반드시 `false`로 바꾸고 `SANDBOXD_API_TOKENS`를
설정해야 한다.

## 분석

### 이 프로젝트는 “AI 앱 빌더의 백엔드”라는 새 범주를 겨냥한다

Sandboxd가 푸는 문제는 코딩 환경이 아니라 *코딩 환경을 파는 제품의
인프라*다.
README가 권장 사용처로 “AI 앱 빌더와 에이전트 플랫폼”을 첫째로 꼽고,
개인용 단일 컨테이너는 “더 간단한 대안이 있다”며 비추천하는 것이
이를 분명히 한다.
즉 이것은 최종 사용자 도구가 아니라, v0·Bolt·Replit 같은 제품을
만들려는 사람을 위한 토대다.

이 포지셔닝은 시장의 한 빈틈을 정확히 노린다.
“프롬프트로 앱을 만들고 즉시 미리보기 링크를 공유한다”는 UX는 이미
여러 상용 제품이 검증했지만, 그 뒤의 멀티 테넌트 격리·라우팅·수명
관리는 각 회사가 매번 새로 구축해야 했다.
Sandboxd는 그 반복되는 하부 작업을 오픈 소스로 표준화하려 한다.
Stripe가 결제를, Auth0가 인증을 추상화했듯, 샌드박스 인프라를
추상화하는 시도다.

### 단순함을 핵심 기능으로 내세운 설계

대부분의 멀티 테넌트 격리 솔루션이 Kubernetes·Firecracker·gVisor 같은
무거운 스택으로 향하는 데 반해, Sandboxd는 의도적으로 SQLite·Docker·
Traefik이라는 최소 구성을 택한다.
이는 단순한 기술 선택이 아니라 명시적 마케팅 포인트다 — “Kubernetes
불필요”, “단일 머신 배포”, “최소 의존성”.

이 선택의 논리는 명확하다.
초기 단계 제품에서 가장 큰 비용은 컴퓨팅이 아니라 운영 복잡성이다.
Kubernetes 클러스터를 운영할 인력이 없는 작은 팀에게, “Go 바이너리
하나 + Docker”는 진입 장벽을 극적으로 낮춘다.
stop-on-idle/wake-on-request로 서버당 샌드박스 밀도를 높여 비용까지
잡겠다는 것은, 자원이 빠듯한 스타트업의 경제학을 정조준한 것이다.

### “신뢰할 수 있는 코드” 전제가 모든 것을 떠받친다

Sandboxd의 단순함은 하나의 전제 위에 서 있다 — 돌리는 코드를 신뢰할
수 있다는 것.
README는 “신뢰할 수 없는 코드는 VM 수준 격리 없이 돌리지 말라”고
못 박는다.
전체 VM이 아닌 Docker 격리로 충분하다는 판단은, 바로 이 신뢰 전제가
성립할 때만 유효하다.

이 전제는 분석의 핵심 긴장을 만든다.
그런데 Sandboxd의 주 사용처인 “AI 앱 빌더”는 본질적으로 *신뢰할 수
없는 입력*(임의 사용자의 프롬프트로 생성된 코드)을 다룬다.
즉 추천 사용처와 안전 전제가 서로 어긋난다.
이 긴장은 비평에서 본격적으로 다룬다.

## 비평

### 추천 용도와 보안 전제가 정면으로 충돌한다

README는 두 가지를 동시에 말한다.
“AI 앱 빌더에 적합하다”와 “신뢰할 수 없는 코드는 VM 격리 없이 돌리지
말라”.
그러나 AI 앱 빌더의 정의 자체가 *모르는 사용자의 프롬프트로 생성된
임의 코드를 실행*하는 것이다.
생성된 코드가 무엇을 할지 사전에 알 수 없으므로, 이것은 신뢰할 수
없는 코드의 교과서적 사례다.

따라서 Sandboxd의 핵심 추천 시나리오에서는 Docker 격리만으로는
부족하고, 사용자는 README가 경고한 “추가 VM 격리”를 스스로 붙여야
한다.
그런데 VM 격리를 붙이는 순간, Sandboxd가 내세운 “단순함·단일 머신·
Kubernetes 불필요”라는 가치 제안의 상당 부분이 무너진다.
즉 가장 적합하다고 광고한 용도가, 가장 핵심적인 장점을 포기해야
성립하는 용도인 것이다.

이 모순은 프로젝트가 “베타”이고 “스타트업 단계에 적합”하다는 단서로
일부 변호된다.
초기에는 신뢰 경계를 느슨하게 두고 빠르게 검증한 뒤, 규모가 커지면
강화한다는 것이다.
그러나 보안은 나중에 덧붙이기 가장 어려운 속성 중 하나다.
멀티 테넌트 격리를 처음부터 약하게 설계하면, 사용자 코드가 컨테이너
탈출(container escape)로 호스트나 다른 테넌트에 도달할 위험을 “나중에”
막기란 구조적으로 힘들다.

### 기본값이 안전하지 않다는 점은 작은 문제가 아니다

`SANDBOXD_API_AUTH_DISABLED=true`가 기본값이라는 사실은 우려스럽다.
컨트롤플레인 API는 샌드박스 생성·명령 실행·파일 조작 등 사실상 모든
권한을 가진 인터페이스다.
이것이 인증 없이 노출되면, API에 닿을 수 있는 누구나 임의 컨테이너를
만들고 명령을 실행할 수 있다.

README는 프로덕션에서 인증을 켜라고 안내하지만, “안전하지 않은
기본값 + 켜라는 문서”의 조합은 역사적으로 수많은 사고의 원인이었다.
기본 바인딩이 `127.0.0.1`이라는 점이 완화 요인이긴 하나, Docker
네트워킹이나 리버스 프록시 설정 한 번의 실수로 이 보호는 쉽게
사라진다.
“안전한 기본값(secure by default)” 원칙에 비추면, 인증을 끄려면
명시적으로 꺼야 하고 기본은 켜져 있어야 한다.

특히 이 프로젝트의 대상 사용자가 “Kubernetes를 운영할 여력이 없는
작은 팀”이라는 점이 위험을 키운다.
인프라 전문성이 얕은 팀일수록 기본값을 그대로 쓸 가능성이 높고,
바로 그 팀이 안전하지 않은 기본값의 최대 피해자가 된다.
단순함을 파는 도구일수록 기본값의 안전성에 더 큰 책임이 있다.

### 단일 서버 설계의 천장이 너무 일찍 온다

Sandboxd는 “멀티 호스트 샤딩은 스케일링 단계로 미룬다”며 단일 서버
배포를 핵심으로 삼는다.
이는 초기 단순함에는 유리하지만, 성공한 제품에는 이른 천장이 된다.
AI 앱 빌더가 인기를 얻으면 동시 샌드박스 수는 빠르게 수백·수천으로
늘고, 단일 머신의 메모리와 CPU는 곧 한계에 부딪힌다.

stop-on-idle/wake-on-request가 밀도를 높여 주긴 하지만, 이것은 유휴
샌드박스가 많을 때만 효과가 있다.
동시에 활성화된 샌드박스가 많아지는 — 즉 제품이 실제로 잘 되는 —
순간, 이 최적화의 이득은 사라지고 단일 서버의 물리적 한계가 그대로
드러난다.
“잘 되면 곧 다시 만들어야 하는” 아키텍처는, 성공이 곧 마이그레이션
비용으로 돌아오는 구조다.

물론 이것은 의도된 트레이드오프다.
프로젝트는 “지금 단순함”을 “나중 확장성”보다 우선했다고 솔직히 밝힌다.
문제는 단일 서버에서 멀티 호스트로 가는 전환이, 상태 저장소(SQLite)·
라우팅·격리 모델 전반의 재설계를 요구한다는 점이다.
SQLite는 단일 노드 전제 위에 있고, 멀티 호스트로 가려면 분산 상태
관리로 갈아타야 한다.
즉 이 전환은 점진적 확장이 아니라 사실상 아키텍처 교체에 가깝다.

## 인사이트

### 에이전트 인프라의 상품화는 차별화를 UX로 밀어 올린다

Sandboxd가 “샌드박스 + 미리보기 URL + 코딩 에이전트”를 오픈 소스로
표준화하면, 이 계층은 더 이상 경쟁 우위가 되지 못한다.
지금은 v0·Bolt·Replit 같은 제품이 이 인프라를 각자 구축한 것이
해자(moat)의 일부였지만, 그것이 `./install.sh` 한 줄로 복제 가능해지면
해자는 메워진다.

이것이 글이 예상하지 못한 2차 효과다.
하부 인프라가 상품화(commoditize)되면 경쟁은 그 위층 — 생성 품질,
프롬프트 UX, 디버깅 경험, 통합 생태계 — 으로 이동한다.
역사적으로 모든 인프라 추상화가 같은 길을 걸었다.
LAMP 스택이 웹 호스팅을 상품화하자 경쟁은 애플리케이션 계층으로
올라갔고, Stripe가 결제를 상품화하자 핀테크 경쟁은 사용자 경험과
규제 대응으로 옮겨갔다.

Sandboxd 같은 도구의 진짜 영향은 따라서 “더 많은 AI 앱 빌더가
생긴다”가 아니다.
오히려 “AI 앱 빌더를 만드는 것 자체가 더 이상 특별하지 않게 된다”는
것이다.
인프라가 쉬워질수록 인프라로는 이길 수 없게 되고, 차별화의 무게중심은
기술 스택에서 제품 감각으로 이동한다.

### “신뢰할 수 있는 코드” 전제는 AI 시대에 점점 깨진다

Sandboxd의 보안 모델은 “코드를 신뢰할 수 있다”는 전제 위에 서 있다.
전통적으로 이 전제는 합리적이었다 — 코드는 고용된 개발자나 검증된
기여자가 작성했다.
그러나 AI 코드 생성은 이 전제의 기반을 조용히 무너뜨린다.
이제 코드는 익명 사용자의 프롬프트에서, 비결정적 모델을 거쳐, 아무도
한 줄도 읽지 않은 채 실행된다.

이 변화는 격리의 의미를 재정의한다.
과거에 컨테이너 격리는 주로 *사고*(버그로 인한 자원 고갈, 충돌)를
막는 것이었다.
AI 시대에 격리는 *적대적 입력*(프롬프트 인젝션으로 유도된 악성 코드
생성, 데이터 탈취 시도)을 막아야 한다.
사고를 막는 격리와 공격을 막는 격리는 요구 수준이 전혀 다르고,
Docker 격리는 전자에는 충분해도 후자에는 위태롭다.

여기서 더 큰 구조적 통찰이 나온다.
AI가 코드 생성을 민주화할수록, “누가 이 코드를 썼는가”라는 신뢰의
앵커가 사라진다.
지금까지 보안은 상당 부분 코드의 *출처*에 대한 신뢰에 의존했는데,
그 출처가 “익명 사용자 + 모델”이 되면 출처 기반 신뢰는 무용지물이
된다.
앞으로의 실행 인프라는 “신뢰할 수 있는 코드”를 전제하는 대신, 모든
코드를 잠재적 적대 입력으로 간주하는 제로 트러스트 격리로 갈 수밖에
없다.
Sandboxd의 단순한 Docker 격리는 이 흐름에서 출발점일 뿐, 종착점은
될 수 없다.

### 자체 호스팅의 부활은 AI 비용 구조가 추동한다

Sandboxd가 “자체 호스팅·벤더 종속 없음·MIT”를 전면에 내세우는 것은
시대적 신호다.
지난 10년간 흐름은 매니지드 클라우드 서비스로의 이동이었는데, AI
워크로드는 이 흐름을 부분적으로 역전시킨다.
이유는 비용이다.
코딩 에이전트는 장시간 실행되는 컴퓨팅과 대량의 토큰을 소비하므로,
사용량 기반 매니지드 서비스로 돌리면 비용이 폭발한다.

이 경제학이 자체 호스팅의 매력을 되살린다.
유휴 시 잠들고 요청 시 깨어나는 밀도 최적화로 자기 서버에서 돌리면,
매니지드 서비스의 마진을 통째로 절약할 수 있다.
글은 이것을 비용 효율로만 제시하지만, 그 이면에는 더 깊은 트레이드
오프가 있다.
자체 호스팅은 비용을 절약하는 대신 운영 책임 — 보안 패치, 격리 유지,
가용성 — 을 사용자에게 되돌린다.

이 트레이드오프는 시간이 지나며 양극화를 낳을 것이다.
운영 역량이 있는 팀은 자체 호스팅으로 비용 우위를 누리고, 그렇지
못한 팀은 비싼 매니지드 서비스에 묶인다.
Sandboxd 같은 도구는 “단순함”으로 이 격차를 좁히려 하지만, 앞서 본
보안·확장성의 천장을 고려하면 그 단순함이 운영 부담을 *제거*하는
것이 아니라 *연기*할 뿐임을 잊지 말아야 한다.
자체 호스팅의 비용 우위는 운영이 잘 굴러갈 때만 우위이고, 사고가
나는 순간 절약했던 마진보다 큰 대가로 돌아올 수 있다.
