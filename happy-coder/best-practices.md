# 베스트 프랙티스와 주의점

<https://happy.engineering/docs/guides/happy-coder-best-practices/>

## 베스트 프랙티스

### 모든 디바이스 연결

가장 큰 워크플로우 개선은 모든 디바이스와
브라우저를 같은 계정에 연결하는 것이다. 주력
노트북뿐 아니라 코드를 실행할 수 있는 모든
컴퓨터를 연결한다. 컴퓨트를 많이 연결할수록
활용도가 높아진다.

### 아이디어에서 구현까지의 마찰 제거

"아이디어 → 구현"의 경로를 메시지 확인만큼
간단하게 만드는 것이 목표다. 산책 중에 더 나은
변수명이 떠오르면 폰을 꺼내서 바로 적용한다.
작은 변경이라도 반복해서 근육 기억(muscle
memory)을 만든다.

### 기존 워크플로우의 확장

Happy Coder는 기존 워크플로우를 대체하는 것이
아니라 확장한다. 터미널에서 `claude`나 `codex`를
직접 사용하는 것도 여전히 가능하다. 모바일은
의무가 아니라 선택지를 제공한다. 책상에 있을
때는 터미널에서, 자리를 비울 때는 폰에서. 세션은
멈춘 곳에서 정확히 이어진다.

### Git Worktree 활용

Happy Coder가 Git Worktree를 직접 관리하지는
않지만, 약간의 설정으로 실험을 원활하게 만들 수
있다. 병렬 세션마다 별도 worktree를 할당하면
서로 간섭 없이 독립적으로 작업할 수 있다.

### 모바일은 모니터링 중심으로

모바일에서 복잡한 코드를 직접 편집하는 것은
비현실적이다. 모바일의 역할은 다음에 집중하는
것이 효과적이다:

- 장시간 작업의 진행 상황 확인
- 권한 요청 승인/거부
- 간단한 프롬프트로 새 작업 지시
- PR 리뷰 시작

## 주의점

### 네트워크 의존성

원격 제어의 특성상 네트워크 연결이 필수다.
오프라인 환경에서는 사용할 수 없으며, 네트워크
지연이 조작 응답성에 직접 영향을 준다.

### 릴레이 서버 의존성

종단간 암호화로 내용은 보호되지만, 모든 통신이
릴레이 서버를 경유하는 구조다. 공식 서버
(`happy-api.slopus.com`) 장애 시 원격 제어가
불가능해진다. 자체 호스팅으로 가용성을 직접
관리할 수 있지만, 운영 부담이 생긴다.

### CLI 래퍼의 한계

`claude`나 `codex` 위에 래퍼를 씌우는 구조이므로
원본 CLI의 업데이트에 따라 호환성 문제가 발생할
수 있다. 원본 CLI와 래퍼 사이의 버전 동기화에
신경 써야 한다.

### 모바일 입력의 제약

모바일에서 코드를 직접 편집하거나 복잡한 명령을
입력하는 것은 데스크톱 대비 불편하다. 모바일은
모니터링과 간단한 승인/거부 위주로 사용하는 것이
현실적이다.

### IDE 통합 미지원

VS Code 등 IDE와의 통합을 지원하지 않는다. CLI
래퍼 방식만 제공하므로, IDE 내장 터미널이 아닌
환경에서 Claude Code를 사용하는 경우(예: VS
Code의 Claude Code 확장)에는 Happy Coder를
적용할 수 없다.
