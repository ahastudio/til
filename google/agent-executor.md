# Agent Executor (AX): Google의 분산 에이전트 런타임

<https://github.com/google/ax>

## 소개

AX(Agent eXecutor)는 Google이 공개한 오픈소스 분산 에이전트 런타임이다.
신뢰성(reliability), 안전성(safety), 커스터마이징 가능성(customizability), 효율성(efficiency)을 설계 목표로 내세운다.
Apache 2.0 라이선스로 공개되었으며, Go 언어로 작성되었다.

에이전트 루프의 조율과 실행 관리를 담당하며, 로컬 또는 원격 액터와 통신한다.
MCP(Model Context Protocol)와 A2A(Agent to Agent) 프로토콜을 기본 지원해 생태계 상호운용성을 확보한다.

## 주요 기능

자동 복구(automatic recovery)와 재개(resumption)가 핵심 기능이다.
장애나 중단이 발생해도 실행 상태를 이벤트 로그 기반으로 유지하므로 중단된 지점부터 재개할 수 있다.

단일 라이터 아키텍처(single-writer architecture)로 일관된 상태 관리를 보장한다.
모든 사용자·에이전트 호출을 공통 컨트롤러가 조율하므로 완전한 감사 추적(audit trail)과 관찰 가능성(observability) 훅이 제공된다.

모델 독립성이 설계 원칙 중 하나다.
사용자가 LLM과 계획 로직을 직접 가져오면 AX는 실행, 지속성, 조율을 담당한다.
특정 LLM 제공자에 종속되지 않는다.

Kubernetes를 1급 지원한다.
Agent Substrate를 통해 프로덕션 환경의 대규모 배포에 적합하도록 설계되었다.

## 아키텍처

```text
클라이언트 → 라우터 → AX 컨트롤러
                          ├── executor
                          ├── event log
                          └── registry
                      ↕
                원격 에이전트 및 도구
                      ↕
            스킬과 내장 도구가 있는 환경
```

컨트롤러, 스킬, 도구, 에이전트가 독립적으로 실행되는 분산 구조다.
이벤트 로그가 내구성 있는 실행 상태의 원천이 된다.

## CLI

```bash
go install github.com/google/ax/cmd/ax@latest
ax --help
```

주요 명령어:

- `exec` — 새로운 실행 시작 또는 기존 대화 재개
- `serve` — gRPC 서버로 컨트롤러 시작
- `fork` — 특정 체크포인트에서 새 이벤트 로그로 분기
- `trace` — 웹 UI에서 실행 추적 시각화

## 라이선스 및 상태

Apache 2.0 라이선스. 현재 “active early development” 단계로 안정화 전 주요 변경이 있을 수 있다.

## 분석

### 에이전트 신뢰성 문제를 런타임 레이어에서 해결

대부분의 에이전트 프레임워크는 LLM 호출과 도구 실행을 어떻게 연결할지에 집중한다.
AX는 다른 문제를 다룬다. 에이전트가 실패했을 때 어떻게 복구할 것인가.
장시간 실행되는 에이전트 태스크에서 중간 실패는 불가피하다.
서버 재시작, 네트워크 오류, API 타임아웃이 발생하면 기존 프레임워크는 처음부터 다시 시작해야 한다.

이벤트 로그 기반 상태 관리는 데이터베이스 WAL(Write-Ahead Log)과 같은 원리다.
모든 상태 변경이 로그에 기록되고, 장애 발생 시 로그를 재생해 상태를 복원한다.
이것은 에이전트를 단순한 스크립트가 아닌 내구성 있는 프로세스로 만드는 설계다.

### Kubernetes 네이티브 설계의 의미

에이전트를 Kubernetes에서 운영하는 것은 아직 초기 단계다.
AX가 Kubernetes를 1급 지원으로 설계한 것은 에이전트가 클라우드 네이티브 인프라의 일급 구성요소가 될 것을 전제한 설계다.
스케일 아웃, 장애 격리, 롤링 업데이트가 에이전트에도 적용된다는 의미다.

MCP와 A2A 프로토콜 지원은 에이전트 생태계의 상호운용성을 겨냥한다.
특정 프레임워크에 종속되지 않고 다양한 에이전트와 도구를 조합할 수 있는 기반이다.

## 비평

### 강점: 감사 추적의 명시적 설계

모든 에이전트 호출에 대한 완전한 감사 추적은 엔터프라이즈 환경에서 필수적이다.
“AI가 무엇을 했는가”를 사후에 재현하고 설명할 수 있어야 하는 규제 요건이 강해지고 있다.
이것을 런타임 레이어에서 기본으로 제공하는 것은 의미 있는 설계 선택이다.

### 약점: Google 프로젝트의 지속성 우려

[GeekNews 댓글](https://news.hada.io/topic?id=30158)에서 eajrezz는 “구글이 공개한 AI 관련 프로젝트의 지속성이 우려된다. 내부에서 십수 년 쓴 것을 오픈소스화한 게 아니면 못 쓸 것 같다”고 지적했다.
Google은 다수의 오픈소스 프로젝트를 중단한 전력이 있다.
“active early development” 상태라는 공식 문구도 불확실성을 확인한다.

프로덕션 환경에서 에이전트 인프라의 핵심 런타임으로 채택하기에는
프로젝트의 장기 유지보수 가능성에 대한 판단이 필요하다.

### 보완할 시각: 비용과 복잡도 트레이드오프

Kubernetes 네이티브 설계와 분산 아키텍처는 소규모 사용 사례에는 과도한 복잡도일 수 있다.
단순한 에이전트 태스크를 위해 Kubernetes 클러스터를 운영하는 것은 비용 대비 효율이 낮다.
어떤 규모와 복잡도의 에이전트 태스크에서 AX가 적합한지에 대한 명확한 가이드가 필요하다.

## 인사이트

### 에이전트 인프라의 계층 분리

소프트웨어 시스템은 성숙할수록 관심사가 계층으로 분리된다.
초기 웹 개발에서는 비즈니스 로직과 데이터베이스 접근이 한 파일에 섞여 있었다.
성숙하면서 ORM, 미들웨어, API 게이트웨이가 각 관심사를 담당하는 계층으로 분리됐다.

에이전트 개발도 같은 방향으로 진화하고 있다.
초기에는 LLM 호출, 도구 실행, 오류 처리, 상태 관리가 한 스크립트에 섞여 있었다.
AX는 실행 신뢰성, 상태 지속성, 감사 추적을 별도 런타임 레이어로 분리하는 시도다.
에이전트 개발자는 무엇을 할지에 집중하고, 어떻게 신뢰성 있게 실행할지는 런타임이 담당한다는 분리다.

이 계층 분리가 완성되면 에이전트 개발의 추상화 수준이 높아진다.
“이 에이전트가 실패하면 어떻게 하는가”라는 질문을 개발자가 매번 답하지 않아도 된다.

### “fork” 명령이 드러내는 에이전트 디버깅의 미래

`fork` 명령 — 특정 체크포인트에서 새 이벤트 로그로 분기 — 은 표면적으로 단순하지만 깊은 함의가 있다.
이것은 에이전트 실행의 타임 트래블 디버깅을 가능하게 한다.

인간 개발자가 코드를 디버깅할 때 중단점(breakpoint)을 설정하고 상태를 검사한다.
에이전트 디버깅에서 동등한 것은 “특정 결정이 내려진 시점으로 돌아가 다른 입력을 주면 어떻게 됐을까”를 실험하는 것이다.
`fork`는 이 실험을 가능하게 한다.
에이전트 행동의 원인 분석과 대안 시나리오 탐색이 이 기능 위에서 이루어진다.

장기적으로 이것은 에이전트 테스트 방법론의 기반이 된다.
“이 에이전트가 이 상황에서 올바르게 작동했는가”를 재현 가능한 방식으로 검증하는 것이다.
