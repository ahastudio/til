# Copybara - 저장소 간 코드 이동 도구

<https://github.com/google/copybara>

HN 토론: <https://news.ycombinator.com/item?id=48740698> (290점, 57개 댓글)

## 소개

Copybara는 구글이 사내에서 사용해 온, 코드를 여러 저장소 사이에서 변환하고
이동시키는 도구다.
자바로 작성됐고 3,480개 이상의 스타를 받았다.

가장 흔한 사용 사례는 비공개(confidential) 저장소와 공개(public) 저장소를
동시에 유지하면서 서로 동기화하는 것이다.
비공개 저장소의 일부 코드를 공개 저장소로 내보내거나,
공개 저장소에 기여자가 남긴 변경 사항을 비공개 저장소로 가져오는 식이다.
비authoritative 저장소에서 발생한 변경은 authoritative 저장소로 옮겨지며,
병합 충돌은 authoritative 저장소 안에서 발생하는 일반적인 충돌과 동일한
방식으로 처리된다.

Copybara는 두 저장소 중 하나를 “authoritative”(권위 있는) 저장소로 지정하도록
요구한다.
항상 하나의 단일 진실 공급원(source of truth)이 존재하게 하기 위해서다.
다만 어느 저장소에든 기여할 수 있고, 어느 저장소에서든 릴리스를 만들 수 있다.

핵심 설계 특징은 무상태(stateless)라는 점이다.
정확히는 상태를 대상(destination) 저장소 안에, 커밋 메시지의 레이블 형태로
저장한다.
이 덕분에 여러 사용자나 서비스가 동일한 설정과 저장소를 대상으로 Copybara를
실행해도 항상 동일한 결과를 얻는다.
현재 공식 지원 저장소 타입은 Git뿐이며, Mercurial 읽기는 실험적으로
지원한다.
확장 가능한 아키텍처 덕분에 원본(origin)과 대상(destination)을
거의 모든 사용 사례에 맞게 추가할 수 있다.

설정은 Starlark(Bazel의 설정 언어와 동일한 파이썬 유사 문법)로 작성한다.
`core.workflow`가 핵심 개념으로, origin, destination, destination_files,
authoring, transformations를 지정해 하나의 마이그레이션 흐름을 정의한다.
`core.replace`, `core.move` 같은 변환 함수들을 조합해 코드를 옮기면서
동시에 경로나 참조 문자열을 바꿀 수 있다.
빌드는 Bazel을 사용하며, JDK 11(사전 빌드된 바이너리 실행에는 Java 21)이
필요하다.
주간 스냅샷 릴리스, 소스 빌드, Docker 이미지 세 가지 방식으로 설치할 수 있다.

## 분석

### “권위 있는 저장소는 하나”라는 제약이 나머지 설계를 지탱한다

Copybara의 문서는 “authoritative 저장소”라는 개념을 반복해서 강조한다.
이 제약이 없다면 두 저장소가 서로 다른 방향으로 발산했을 때
어느 쪽이 옳은지 판단할 기준이 사라진다.
authoritative 저장소를 하나로 고정함으로써, 나머지 저장소에서 발생한
변경은 항상 “가져오는(import)” 대상이지 “동등하게 병합되는” 대상이
아니게 된다.

이 제약은 자유도를 제한하는 대신 예측 가능성을 확보하는 선택이다.
비authoritative 저장소에 기여가 들어오면, 그 변경은 authoritative 저장소
기준으로 재해석되어 옮겨지고, 충돌이 나면 authoritative 저장소 내부의
일반적인 충돌과 동일하게 처리된다.
즉 Copybara는 두 저장소를 대칭적인 두 개의 소스로 다루지 않고,
하나의 소스와 그 소스의 파생 뷰(view)로 다룬다.
이 비대칭성이 상태 없는(stateless) 설계와 맞물려 여러 사용자가 같은 결과를
재현할 수 있게 만드는 근거가 된다.

실제 사용자들의 후기는 이 비대칭성이 실무에서 두 가지 서로 다른 쓰임으로
갈라진다는 점을 보여준다.
한 사용자는 양방향으로 코드를 내보내고 다시 가져오는(export-then-import)
전체 순환 워크플로는 번거로워서 쓰지 않고, 대신 특정 폴더를 원래
저장소에서 한 번만 꺼내 새 저장소로 옮기면서 히스토리(git blame 포함)를
보존하는 “일회성 내보내기” 용도로만 활용한다고 밝혔다[^klodolph].
반면 다른 사용자는 사내 정책상 외부 오픈소스 프로젝트를 빌드 산출물이
아닌 소스 코드 형태로 모노레포에 지속적으로 가져오는 데,
그리고 반대로 모노레포 안에서 개발한 코드를 지속적으로 외부에
공개하는 데 양방향으로 쓰고 있었다[^asinclair].
즉 authoritative 저장소를 하나로 강제하는 이 설계는, 일회성 이전 도구로도
지속적 동기화 파이프라인으로도 쓰일 수 있을 만큼 유연하지만,
그만큼 사용자가 자신의 사용 패턴에 맞는 워크플로 구성을 스스로
찾아내야 하는 부담도 함께 남긴다.

### 상태를 커밋 메시지 레이블에 저장하는 방식은 인프라를 최소화하려는 선택이다

일반적인 동기화 도구라면 두 저장소 사이의 마지막 동기화 지점을 별도의
데이터베이스나 상태 파일에 저장할 법하다.
Copybara는 그 대신 대상 저장소의 커밋 메시지 안에 레이블 형태로 상태를
남긴다.
이는 “상태를 어디에 둘 것인가”라는 질문에 대해, 별도 인프라를 두지 않고
버전 관리 시스템 자체를 상태 저장소로 재활용하는 답변이다.

이 선택의 이점은 명확하다.
어떤 사용자나 CI 서비스든 저장소 히스토리만 읽으면 마지막 동기화 지점을
알 수 있고, 별도 데이터베이스를 운영하거나 백업할 필요가 없다.
동시에 이 설계는 근본적인 전제를 깔고 있다.
커밋 메시지가 변경되거나 히스토리가 재작성(rebase, squash 등)되면
이 상태 추적 메커니즘 자체가 깨질 수 있다는 점이다.
Git 저장소, 특히 공개 저장소는 히스토리 재작성이 드물지 않게 일어나는
환경이므로, 이 설계가 상정하는 “히스토리는 안정적으로 유지된다”는 가정은
모든 사용 맥락에서 성립하지는 않는다.

### Starlark 채택은 구글 내부 도구 생태계의 관성을 반영한다

Copybara의 설정 언어가 Bazel과 동일한 Starlark라는 점은 우연이 아니다.
`buildozer` 관련 함수들이 레퍼런스 문서에 별도 챕터로 존재한다는 사실은,
이 도구가 애초에 Bazel 기반 빌드 시스템을 사용하는 구글 내부 코드베이스를
전제로 설계되었음을 보여준다.

이는 설정 언어를 선택할 때 “새로운 사용자에게 배우기 쉬운가”보다
“기존 내부 생태계와 얼마나 잘 맞물리는가”를 우선했다는 뜻이다.
구글 밖의 사용자 입장에서는 Starlark이 낯선 진입장벽이 될 수 있지만,
동시에 이미 Bazel을 사용하는 조직이라면 별도의 새로운 설정 형식을 배울
필요 없이 기존 지식을 그대로 재사용할 수 있다는 이점도 있다.
오픈소스로 공개된 도구가 원래 설계된 조직의 기술 스택 흔적을 그대로
드러내는 전형적인 사례다.

## 비평

### “무상태”라는 표현은 상태가 없는 것이 아니라 상태의 위치를 옮긴 것이다

README는 Copybara를 “stateless”라고 소개하지만, 곧바로 “정확히는 상태를
대상 저장소에 레이블로 저장한다”고 스스로 정정한다.
이는 마케팅적으로는 매력적인 단어 선택이지만 기술적으로는 오해를 부르기
쉬운 표현이다.
진짜 무상태 시스템이라면 각 실행이 이전 실행의 결과에 의존하지 않아야
하는데, Copybara는 명백히 이전 커밋의 레이블을 읽어 다음 동작을 결정한다.

정확한 표현은 “외부 상태 저장소가 없다”거나 “상태가 대상 저장소에
내장되어 있다”에 가깝다.
이 차이는 사소해 보이지만 실무에서는 중요하다.
사용자가 “무상태”라는 말만 믿고 대상 저장소의 히스토리를 함부로
재작성하거나 압축하면, 실제로는 이 재작성이 Copybara의 동작에
영향을 미치는 상태를 삭제하는 셈이 된다.
용어가 시스템의 실제 동작 방식을 정확히 반영하지 못하면,
사용자가 시스템의 안전한 사용 범위를 잘못 판단할 위험이 커진다.

### Git 외 저장소 지원의 실질적인 성숙도가 불분명하다

README는 “Currently, the only supported type of repository is Git”이라고
명시하면서도, 곧이어 “확장 가능한 아키텍처 덕분에 원본과 대상을 거의 모든
사용 사례에 맞게 추가할 수 있다”고 덧붙인다.
Mercurial 읽기 지원은 “실험적”이라고 밝히지만, 정확히 어떤 기능이
동작하고 어떤 기능이 동작하지 않는지에 대한 구체적인 범위는 README에
없다.

“확장 가능한 아키텍처”라는 표현은 이론적 가능성을 말하는 것이지,
실제로 커뮤니티가 다른 버전 관리 시스템에 대한 origin·destination
구현체를 만들어 냈는지는 별개의 질문이다.
Perforce, Subversion처럼 여전히 널리 쓰이는 버전 관리 시스템에 대한
지원 현황이 README 어디에도 명시되어 있지 않다는 점은,
Git 이외의 환경에서 이 도구를 채택하려는 사용자가 실제로 얼마나
성숙한 지원을 받을 수 있을지 사전에 가늠하기 어렵게 만든다.

이 공백은 실제 사용자의 경험으로도 확인된다.
한 게임 스튜디오의 개발자는 자신이 직접 작성한 Perforce 지원 패치를
막 오픈소스로 공개하려던 참에 이 HN 게시물을 접했다고 밝히며,
Copybara의 원래 용도가 구글 내부에서 Piper(Perforce의 포크)에 저장된
코드를 공개하는 것임에도 정작 Perforce 지원이 의미 있는 수준으로
존재하지 않는다는 점이 아이러니하다고 지적했다.
그는 PR을 올리기 전 커밋 히스토리에서 자신이 접근할 수 없는 Gerrit
코드 리뷰 시스템의 흔적(Change-ID 마커)을 발견하고, 이 기여가 실제로
업스트림에 반영될 수 있을지 불안했다고 덧붙였다[^dijit].
이에 대해 다른 사용자는 구글이 Piper에 대해서는 애초에 Gerrit이 아니라
Critique라는 별도의 내부 코드 리뷰 도구를 사용한다고 설명했다[^bruckie].
즉 공개 저장소에 남은 Gerrit 흔적은 이 프로젝트의 외부 기여 경로 일부가
실제로는 다른, 공개되지 않은 내부 도구와 얽혀 있음을 보여주는
단서였던 셈이다.

### 문서화 미완성 상태를 스스로 인정하면서도 학습 곡선의 대안은 외부 블로그에 의존한다

“Documentation” 섹션은 “우리는 아직 문서화 작업을 진행 중이다”라고
직접 밝힌다.
그러면서 제공하는 자료는 저장소 안의 레퍼런스 문서와 예제,
그리고 외부 블로그에 있는 한 편의 튜토리얼이다.
공식 문서가 “작업 중”이라고 선언된 상태에서, 신규 사용자의 온보딩
경험이 제3자가 작성한 블로그 글의 품질과 최신성에 좌우된다는 뜻이다.

이 상황은 구글이 사내용으로 개발한 도구를 오픈소스로 공개할 때
흔히 나타나는 패턴을 보여준다.
내부 사용자들은 사내 문서, 사내 교육, 동료의 구두 설명으로 학습 곡선을
넘어서지만, 외부 공개판에는 그런 지원 체계가 그대로 이식되지 않는다.
`core.workflow`의 파라미터 하나하나가 방대한 레퍼런스 문서(reference.md)에
나열되어 있음에도, 그 파라미터들을 실제로 어떤 순서와 조합으로
사용해야 하는지에 대한 서사형 안내(narrative guide)는 여전히 부족하다.

이 공백은 실제 도입 사례에서도 확인된다.
데이터 플랫폼 회사 Dagster는 공개용 저장소들을 더 큰 내부 모노레포 안에
두는 허브-앤-스포크(hub-and-spoke) 구조를 만들기 위해 Copybara를
도입했는데, 동작은 했지만 “많은 우회적인(hacky) 작업”이 필요했다고
공개 사례 후기에서 밝혔다[^cole]. 표준 문서만으로는 복잡한 실제
워크플로를 매끄럽게 구성하기 어렵다는 뜻이며, 이는 앞서 지적한
서사형 안내의 부재가 추상적인 문제가 아니라 실제 도입 비용으로
이어진다는 것을 보여준다.

### 구글 도구를 채택하는 것은 곧 단종 위험을 함께 채택하는 것이다

한 댓글은 단순히 저장소를 미러링하는 것이 목적이라면 Copybara처럼
무거운 도구 대신 GitLab이 기본으로 제공하는 저장소 미러링 기능을 쓰는
편이 낫다고 제안하면서, Copybara 같은 구글 도구는 “kaniko를 비롯한
수많은 다른 구글 제품·도구들처럼 구글이 아카이브하거나 중단시킬
때까지만 유효할 수 있다”고 경고했다[^the_dude].

이 지적은 이 글이 전혀 다루지 않은 차원의 위험을 짚는다.
지금까지의 분석과 비평은 Copybara의 설계와 문서 품질을 다뤘지만,
이 도구를 도입하는 조직이 실제로 감수하는 위험은 기술적 완성도보다
구글이라는 단일 벤더의 제품 유지 결정에 달려 있다는 점이다.
변환 규칙과 워크플로 정의가 `copy.bara.sky` 파일 안에 깊숙이
자리 잡을수록, 이 도구가 단종됐을 때 다른 도구로 이전하는 비용도
함께 커진다.

## 인사이트

### “코드를 두 저장소 사이에서 옮긴다”는 문제는 반세기 전부터 반복되어 온 문제다

Copybara가 다루는 문제, 즉 두 개의 프로덕션 저장소 사이에서 코드를
선택적으로 복사하고 변환하는 작업은 놀랍게도 완전히 새로운 문제가
아니다.
한 HN 사용자는 이 문제의 계보를 1974년 IBM의 COPY 유틸리티까지
거슬러 올라가 짚었다.
당시 OS/MVT와 OS/VS2 환경의 파티션 데이터 세트(partitioned data set)
사이에서 코드를 옮기는 COPY, FORMAT, LIST, MERGE 유틸리티가 이미
존재했다는 것이다[^theodphn].

이 역사적 유비가 주는 통찰은 “옛날에도 비슷한 게 있었다”는 흥미로운
사실 이상이다.
소프트웨어 개발이 여러 격리된 환경(메인프레임의 데이터 세트,
오늘날의 Git 저장소)으로 파편화되는 순간, 그 경계를 넘나드는 코드
이동 도구에 대한 수요는 구조적으로 재발생한다는 뜻이다.
격리 자체가 보안, 규정 준수, 조직 경계 등의 이유로 계속 필요한 한,
이 경계를 넘는 이동 도구도 계속 재발명될 수밖에 없다.
Copybara는 이 오래된 문제의 2020년대 버전일 뿐이며, 앞으로 새로운
저장 매체나 격리 경계가 생길 때마다 유사한 도구가 다시 등장할
가능성이 높다.

### 저장소 간 코드 이동은 업계 전반에서 독자적으로 재발명되고 있다

Copybara와 유사한 목적의 도구가 다른 조직에서도 각자 독립적으로
등장했다는 사실은 이 문제의 보편성을 보여준다.
러스트(Rust) 프로젝트는 Josh라는 도구로 여러 저장소에 걸친 커밋을
동기화하고 있고, 메타(Meta)는 한때 fbshipit이라는 오픈소스 도구로
비슷한 문제를 해결했지만 지금은 더 이상 사용하지 않는다고
저장소에 명시했다[^marksweep].

세 회사가 세 개의 서로 다른 도구를 독자적으로 만들었다는 사실은,
이 문제에 하나의 표준 해법이 자리 잡지 못했다는 뜻이기도 하다.
각 도구는 자신이 속한 조직의 버전 관리 시스템, 코드 리뷰 문화,
모노레포 정책에 맞춰 최적화되어 있어서 서로 호환되지 않는다.
fbshipit이 메타 내부에서 이미 폐기됐다는 사실은 이런 도구들의
수명 자체가 조직의 내부 정책 변화에 따라 유동적임을 보여준다.
즉 이 문제 영역에는 시장 전체가 수렴할 만한 표준 도구가 없고,
당분간 각 조직이 자체적으로 도구를 만들고 폐기하는 사이클이
반복될 가능성이 높다.

### 단일 진실 공급원 강제는 분산 버전 관리의 근본 철학과 긴장 관계에 있다

Git 같은 분산 버전 관리 시스템(DVCS)의 핵심 철학은 모든 클론이 대등하며,
어느 저장소도 본질적으로 “더 권위 있지” 않다는 것이다.
그런데 Copybara는 정확히 이 철학을 거스르는 방향으로 설계되어 있다.
authoritative 저장소를 강제로 지정하게 함으로써, 분산 시스템 위에
중앙집중적 진실 공급원이라는 개념을 다시 얹는다.

이 긴장은 우연이 아니라 실무적 필요에서 비롯된 것이다.
완전히 대등한 분산 모델은 이론적으로는 아름답지만, 법무·보안·경쟁 우위
때문에 코드의 일부를 의도적으로 비공개로 유지해야 하는 조직에게는
비현실적이다.
Copybara의 존재 자체가, 대규모 조직이 오픈소스 생태계에 참여하면서도
동시에 완전한 대칭적 개방을 원하지 않을 때 어떤 타협이 필요한지를
보여주는 사례다.
이런 긴장은 앞으로도 기업이 운영하는 오픈소스 프로젝트에서 반복적으로
나타날 구조적 문제이며, “공개 저장소=진짜 소스”라는 순진한 가정으로
오픈소스 생태계를 이해하려는 시각에 대한 반례가 된다.

### 커밋 메시지를 상태 저장 매체로 쓰는 패턴은 다른 도구에서도 재발명되고 있다

버전 관리 시스템의 메타데이터(커밋 메시지, 트레일러, 태그)를 애플리케이션
상태 저장에 활용하는 패턴은 Copybara만의 것이 아니다.
`Co-Authored-By` 트레일러, `Change-Id`(Gerrit), semantic-release의
커밋 컨벤션 파싱 등은 모두 같은 발상을 공유한다.
버전 관리 시스템이 이미 제공하는 불변성과 이력 추적 기능을 재사용해서,
별도의 상태 저장소를 새로 구축하는 비용을 회피하는 것이다.

이 패턴이 누적되면 흥미로운 결과가 생긴다.
Git 저장소 하나가 단순한 코드 이력이 아니라, 여러 자동화 도구가 동시에
참조하는 암묵적인 상태 저장소 다발이 되어 간다.
문제는 이 상태들이 서로 다른 도구에 의해 다른 형식으로, 종종 문서화되지
않은 채로 커밋 메시지에 겹쳐 쓰인다는 점이다.
한 조직이 Copybara, Gerrit, semantic-release를 동시에 쓴다면,
커밋 메시지 하나에 서로 다른 목적의 메타데이터가 뒤섞이고,
그중 하나를 잘못 건드리면 어느 자동화가 조용히 깨질지 예측하기 어려운
결합(coupling)이 만들어진다.

### 코드 이동 자동화는 결국 “신뢰 경계”를 코드로 명시하는 작업이다

Copybara가 해결하려는 문제, 즉 비공개 코드와 공개 코드를 오가며 동기화하는
작업은 표면적으로는 파일 복사와 텍스트 치환의 문제처럼 보인다.
그러나 실제로 이 도구가 인코딩하는 것은 “이 코드는 누가 볼 수 있는가”,
“이 변경은 어느 방향으로만 흘러야 하는가”라는 조직의 신뢰 경계(trust
boundary)다.
`destination_files`의 glob 패턴과 exclude 규칙, authoring 정책은
모두 이 신뢰 경계를 코드로 명시한 결과물이다.

이 관점에서 보면 Copybara의 설정 파일(`copy.bara.sky`)은 단순한
자동화 스크립트가 아니라, 조직의 정보 공개 정책을 실행 가능한 형태로
번역한 문서에 가깝다.
장기적으로 이런 종류의 “신뢰 경계 코드화”는 코드 저장소를 넘어
데이터 파이프라인, ML 모델 아티팩트, 문서 저장소 등 비공개·공개
경계가 있는 모든 자산으로 확장될 가능성이 있다.
Copybara가 지금 Git 저장소에 대해 하는 일을, 미래에는 더 넓은 범위의
디지털 자산에 대해 수행하는 범용 “신뢰 경계 게이트웨이”가 등장할
여지가 있으며, 그 경우 지금 Copybara가 마주한 문제들, 즉 상태 관리의
암묵성과 문서화 부담은 그대로, 혹은 더 큰 규모로 반복될 것이다.
