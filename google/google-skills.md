# google/skills: 구글의 Agent Skills 저장소

<https://github.com/google/skills>

## 소개

`google/skills`는 구글이 자사 제품과 클라우드 기술을 AI 에이전트에게
가르치기 위해 공개한 Agent Skills 모음 저장소다. 저장소는 “active
development” 상태임을 명시하고 있고, 라이선스는 Apache 2.0이다. 설치는
`npx skills add google/skills` 한 줄로 이루어지며, 이 명령을 실행하면 저장소
안의 개별 스킬 중 원하는 것을 골라서 설치할 수 있다. 즉 구글이 만든 것은
새로운 런타임이나 SDK가 아니라, 기존에 존재하는 Agent Skills 포맷을 따르는
콘텐츠 패키지의 컬렉션이다.

수록된 스킬은 크게 세 갈래다. 첫째는 에이전트 플랫폼 API 계열로, Gemini
API, Gemini Interactions API, Managed Agents API, Skill Registry API를
다룬다. 둘째는 구글 클라우드 서비스 기초 스킬로 AlloyDB, BigQuery, Cloud
Run, Cloud SQL, Firebase, GKE(Kubernetes Engine)의 “Basics”가 포함된다.
셋째는 레시피와 프레임워크 계열로, 구글 클라우드 온보딩, 인증
(authentication), 네트워크 관측성(network observability) 레시피와 함께
Well-Architected Framework의 여섯 기둥(보안, 신뢰성, 비용 최적화, 운영
우수성, 성능 최적화, 지속가능성)을 각각 별도 스킬로 제공한다. Flutter와
Dart 스킬은 이 저장소가 아니라 `flutter/skills`, `dart-lang/skills`라는
별도 저장소에서 관리된다.

Agent Skills 포맷 자체는 구글의 발명이 아니다. `agentskills.io` 문서는
이 포맷이 “originally developed by Anthropic, released as an open
standard”라고 명시한다. 포맷의 핵심은 단순하다. 하나의 스킬은 `SKILL.md`
파일을 담은 폴더이며, 이 파일은 최소한 `name`과 `description` 메타데이터와
에이전트가 작업을 수행하는 방법을 담은 지시문(instructions)으로 구성된다.
폴더에는 선택적으로 실행 가능한 스크립트(`scripts/`), 참조 문서
(`references/`), 템플릿이나 리소스(`assets/`)를 함께 묶을 수 있다.

이 포맷을 지원하는 클라이언트는 이미 광범위하다. Gemini CLI, Claude Code,
Claude, OpenAI Codex, Cursor, GitHub Copilot, VS Code, OpenCode, OpenHands,
Goose, Roo Code, Kiro 등 수십 개의 에이전트 제품이 같은 스킬 포맷을 읽는다.
구글이 이 저장소를 내놓은 의미는, 자사 클라우드와 Gemini에 대한 “절차적
지식(procedural knowledge)”을 이 공용 포맷에 실어 어떤 호환 에이전트에서도
로드할 수 있게 만들었다는 데 있다.

## 명세와 동작

스킬은 progressive disclosure(점진적 공개)라는 세 단계로 로드된다.
첫째 Discovery 단계에서 에이전트는 시작 시 각 스킬의 `name`과
`description`만 읽어 둔다. 언제 그 스킬이 필요할지 판단할 정도의 최소
정보다. 둘째 Activation 단계에서 작업이 어떤 스킬의 description과 맞아
떨어지면, 에이전트가 그 스킬의 `SKILL.md` 전체 지시문을 컨텍스트로
읽어들인다. 셋째 Execution 단계에서 에이전트는 지시문을 따르며 필요에
따라 번들된 코드를 실행하거나 참조 파일을 추가로 로드한다. 전체 지시문은
작업이 그것을 요구할 때만 로드되므로, 많은 스킬을 손에 쥐고 있어도 평소
컨텍스트 점유는 작게 유지된다.

구글 저장소의 실제 `SKILL.md`는 이 명세를 그대로 따른다. 예를 들어
BigQuery Basics 스킬의 frontmatter는 다음과 같다.

```yaml
---
name: bigquery-basics
description: >-
  Manages datasets, tables, and jobs in BigQuery, and integrates with BigQuery
  ML and Gemini for advanced data analytics and AI-driven insights. Use when
  you need to interact with BigQuery, run SQL queries, manage BigQuery
  resources, or leverage BigQuery's built-in ML capabilities. ...
---
```

frontmatter 아래의 본문은 평범한 마크다운 산문과 단계별 절차로 이루어진다.
BigQuery 스킬의 본문은 “BigQuery는 서버리스 AI-ready 데이터 플랫폼”이라는
한 문단의 개념 설명으로 시작한 뒤, “Setup and Basic Usage” 절에서
`gcloud services enable bigquery.googleapis.com --quiet`로 API를 활성화하고
`bq mk --dataset --location=US my_dataset`로 데이터셋을 만드는 등의 CLI
절차를 번호 매긴 목록으로 나열한다. 다시 말해 스킬의 내용물은 본질적으로
잘 정리된 명령어 레시피 문서다.

여기서 description의 작성 방식이 중요하다. BigQuery 스킬의 description은
무엇을 하는지(“Manages datasets, tables, and jobs...”)뿐 아니라 언제
써야 하는지(“Use when you need to interact with BigQuery, run SQL
queries...”)를 명시적으로 적어 둔다. progressive disclosure 구조에서
에이전트는 평소 이 description만 보고 활성화 여부를 판단하므로,
description의 트리거 문구가 사실상 스킬의 라우팅 키 역할을 한다. 스킬
저작의 난이도는 본문보다 이 description을 얼마나 정확히 쓰느냐에 달려 있는
셈이다.

## 분석

### 구글이 런타임이 아니라 콘텐츠를 출하한 선택

가장 먼저 눈에 띄는 것은 구글이 새 프로토콜이나 SDK가 아니라 콘텐츠
저장소를 내놓았다는 점이다. `google/skills`에는 실행 엔진이 없다. 스킬을
로드하고 progressive disclosure를 수행하는 주체는 Gemini CLI든 Claude
Code든 사용자가 쓰는 에이전트 클라이언트이고, 구글이 채운 것은 그
클라이언트들이 읽어들일 마크다운 문서뿐이다. 이는 구글이 포맷 전쟁의
규칙을 스스로 정하기보다, 이미 굳어진 포맷 위에 콘텐츠를 얹는 follower
포지션을 택했다는 뜻이다.

이 선택은 합리적이다. Agent Skills 포맷은 Anthropic이 만들어 오픈
스탠더드로 풀었고, `agentskills.io`의 클라이언트 쇼케이스가 보여주듯 이미
수십 개 제품이 채택했다. 구글이 여기에 자체 포맷을 새로 들이밀면 또 하나의
파편을 만들 뿐이고, 정작 자사 클라우드를 에이전트에게 가르치겠다는 목표는
오히려 멀어진다. 반대로 공용 포맷에 올라타면 Gemini CLI 사용자뿐 아니라
Claude Code나 Cursor 사용자까지 단번에 잠재 청중이 된다. 콘텐츠를 출하하는
편이 분포(distribution) 면에서 압도적으로 유리하다.

다만 이 포지션에는 비대칭이 숨어 있다. 포맷의 통제권은 Anthropic과
`agentskills`/`agentskills.io` 커뮤니티 거버넌스에 있고, 구글은 그 포맷의
한 소비자다. 구글이 자사 클라우드 온보딩을 표준화하려는 의도와, 표준 자체를
누가 쥐느냐는 별개의 문제다. 구글은 콘텐츠 레이어를 장악하는 대신 포맷
레이어의 주도권은 양보했다.

### 스킬은 MCP와 경쟁하지 않고 다른 층을 채운다

Agent Skills를 흔히 MCP(Model Context Protocol)와 묶어 비교하지만, 둘은
같은 층의 경쟁재가 아니다. MCP는 에이전트가 외부 도구·데이터 소스에
연결되는 프로토콜, 즉 실행 가능한 도구 호출의 배선(wiring)이다. 반면
Skills는 “어떻게 하는지”를 담은 절차적 지식의 패키지다. BigQuery 스킬을
보면 이 차이가 분명하다. 스킬은 BigQuery에 붙는 새 API 커넥터를 제공하지
않는다. 이미 존재하는 `gcloud`와 `bq` CLI를 언제 어떤 순서로 호출해야
하는지를 자연어로 설명할 뿐이다.

이 관점에서 보면 구글 저장소가 “Gemini API”, “Skill Registry API” 같은
API 스킬과 “BigQuery Basics” 같은 CLI 레시피를 한데 모은 것이 이해된다.
공통점은 도구가 아니라 도구를 다루는 노하우라는 점이다. 스킬은 모델의
기존 능력(쉘 실행, 파일 읽기)을 전제하고, 거기에 도메인 맥락만 주입한다.
즉 Skills의 암묵적 가정은 “에이전트는 이미 손과 발은 있고, 부족한 것은
특정 영역의 절차적 기억”이라는 것이다.

이 가정이 맞다면 Skills는 MCP의 대체재가 아니라 보완재다. MCP가 도구를
연결하고, Skills가 그 도구를 능숙하게 쓰는 법을 가르친다. 구글 저장소는
이 분업을 실물로 보여주는 사례다. 다만 둘의 경계는 생각보다 흐릿하다.
BigQuery 스킬이 `gcloud` CLI를 호출하라고 지시하는 것과, BigQuery용 MCP
서버가 같은 작업을 도구 호출로 노출하는 것 사이에서 무엇을 스킬로 두고
무엇을 도구로 둘지는 명확한 규칙이 없다.

### 능력 획득 모델에 대한 전제

Skills 포맷이 깔고 있는 더 깊은 전제는 에이전트의 능력이 “맥락 주입”으로
획득된다는 믿음이다. 파인튜닝도 아니고 새 도구 추가도 아니다. 적시에 올바른
문서를 컨텍스트에 넣으면 충분히 똑똑한 모델이 알아서 일을 해낸다는 것이다.
이것이 진실이라면 도메인 능력은 모델 가중치가 아니라 마크다운 파일에 산다.
구글이 자사 클라우드 지식을 가중치에 학습시키는 대신 `SKILL.md`로 외부화한
것은 이 전제를 그대로 받아들인 결과다.

이 전제는 progressive disclosure 설계와 맞물린다. 모든 지식을 한꺼번에
컨텍스트에 넣을 수 없으니, description으로 색인을 만들고 필요할 때만 본문을
펼친다. 사실상 에이전트의 장기 기억을 파일 시스템에 두고, 컨텍스트 윈도를
워킹 메모리로 쓰는 구조다. 이는 운영체제의 가상 메모리 페이징과 같은
발상이며, 한정된 컨텍스트라는 물리적 제약을 우회하는 영리한 방법이다.

그러나 이 전제는 모델이 description만 보고 활성화 시점을 정확히 판단한다는
것, 그리고 본문 지시문을 충실히 따른다는 것을 동시에 가정한다. 둘 다 모델
역량에 달린 확률적 행위다. 능력을 콘텐츠로 외부화한 대가로, 그 콘텐츠가
실제로 작동할지는 모델의 지시 준수 능력이라는 통제 불가능한 변수에
묶이게 된다.

## 비평

### “Basics”는 검증된 절차인가, 잘 정리된 프롬프트인가

구글 저장소의 핵심 스킬 다수는 “BigQuery Basics”, “Cloud Run Basics”처럼
입문 가이드 성격이다. 그런데 이 문서들이 기존의 구글 클라우드 공식 문서나
quickstart와 본질적으로 무엇이 다른지는 분명하지 않다. BigQuery 스킬의
본문은 `gcloud services enable`, `bq mk` 같은 명령을 번호 목록으로 나열한,
사실상 공식 문서의 압축판이다. 그렇다면 Skills는 새로운 추상이라기보다
기존 문서를 에이전트가 먹기 좋은 형태로 재포장한 것에 가깝다.

재포장 자체가 나쁜 것은 아니다. progressive disclosure에 맞춰 description을
트리거로 다듬은 것은 실제 가치다. 그러나 “skill”이라는 단어가 풍기는
뉘앙스—에이전트가 새 능력을 획득한다—와 실제 내용물—명령어 레시피
문서—사이에는 간극이 있다. 마케팅 용어로서의 “skill”과 엔지니어링 실체로서의
“마크다운 문서”를 혼동하면, 이 포맷이 실제로 보장하는 것을 과대평가하게
된다.

게다가 이 레시피들은 검증 메커니즘이 약하다. 코드라면 테스트를 돌릴 수
있지만, 자연어 절차 문서는 “이 순서대로 하면 된다”는 주장이 맞는지 자동으로
확인할 길이 마땅치 않다. 저장소가 contribution으로 “bugs or inaccuracies
in the skill Markdown files” 신고를 받는다는 점이 이를 역설적으로 보여준다.
스킬의 정확성은 결국 사람의 신고와 메인테이너의 수작업 검토에 의존한다.

클라우드 CLI와 API는 빠르게 바뀐다. `bq`의 플래그가 deprecate되거나
`gcloud` 명령 구조가 개편되면, 가중치에 학습된 지식보다 마크다운 문서가
더 빨리 낡을 수 있다. 외부화가 갱신을 쉽게 만든다는 장점은, 갱신을 게을리
하면 그만큼 빠르게 오정보 소스가 된다는 위험과 한 몸이다.

### 공용 포맷의 채택은 곧 파편화의 시작이기도 하다

`agentskills.io`는 Agent Skills를 단일 오픈 스탠더드로 그린다. 그러나
클라이언트 쇼케이스를 자세히 보면, 같은 포맷을 “지원”한다는 수십 개
제품이 저마다 자체 문서(`instructionsUrl`)를 가리킨다. Gemini CLI의 스킬
문서, Claude Code의 스킬 문서, Cursor의 스킬 문서가 따로 있다. 최소 명세
(`SKILL.md` + `name` + `description`)는 같을지 몰라도, 스킬을 어디에 두고
어떻게 활성화하며 어떤 확장 필드를 인정하는지는 클라이언트마다 갈라질 수
있다.

이는 HTML이 표준이지만 브라우저마다 렌더링이 달랐던 시기를 연상시킨다.
“표준을 지원한다”는 선언과 “동일하게 동작한다”는 보장은 다르다. 구글이
공용 포맷에 올라탄 것은 분포 측면에서 이득이지만, 그 이득은 모든 클라이언트가
스킬을 똑같이 해석한다는 전제 위에서만 온전하다. 전제가 깨지면 구글은 “어떤
에이전트에서나 작동”이 아니라 “주로 Gemini CLI에서 잘 작동, 나머지는 해보세요”
상태에 빠진다.

특히 구글 스킬이 `gcloud`와 `bq` 같은 로컬 CLI 실행을 전제한다는 점이
파편화 위험을 키운다. 쉘 실행을 허용하지 않거나 샌드박싱 정책이 다른
클라이언트에서는 같은 스킬이 그대로 동작하지 않는다. 포맷은 같아도 실행
환경의 가정이 다르면 이식성은 명목상의 것에 그친다.

### 서드파티 스킬의 신뢰와 보안은 미해결로 남아 있다

`npx skills add google/skills`라는 설치 경로는 편리하지만, 동시에 임의의
원격 마크다운—그 안에 쉘 명령이 들어 있는—을 에이전트의 실행 컨텍스트로
끌어오는 행위다. 구글 공식 저장소라면 출처 신뢰가 어느 정도 성립하지만,
같은 메커니즘이 임의의 GitHub 저장소(`npx skills add <owner>/<repo>`)에도
열려 있다면 문제는 달라진다. 스킬은 본질적으로 에이전트에게 “이렇게
행동하라”고 지시하는 문서이고, 악의적 지시문은 프롬프트 인젝션의 배달
수단이 될 수 있다.

progressive disclosure 구조는 이 위험을 미묘하게 키운다. 사용자는 평소
description만 보고 스킬을 신뢰할지 결정하지만, 실제로 실행되는 것은 활성화
이후 펼쳐지는 본문 전체다. description은 무해해 보이는데 본문이 위험한 명령을
포함하는 상황을 사용자가 사전에 알아채기 어렵다. 코드 의존성의
typosquatting이나 악성 postinstall 스크립트 문제가 스킬 생태계로 그대로
옮겨올 소지가 있다.

원문과 `agentskills.io` 문서는 이 보안 모델을 거의 다루지 않는다. 서명,
권한 범위 선언, 샌드박싱 같은 신뢰 장치가 포맷 명세의 일급 시민으로
보이지 않는다. 최소 명세가 `name`과 `description`뿐이라는 단순함은 채택을
쉽게 만든 장점이지만, 신뢰와 권한이라는 어려운 문제를 명세 바깥으로 밀어낸
결과이기도 하다.

구글 저장소만 놓고 보면 출처가 명확해 당장 큰 문제는 아니다. 그러나
구글이 이 포맷을 정당화하고 대중화할수록, 출처가 불분명한 서드파티 스킬을
설치하는 행위도 함께 정상화된다. 생태계 전체의 보안 표준이 따라잡기 전에
설치 관행이 먼저 퍼지면, 신뢰 문제는 사고가 터진 뒤에야 다뤄질 공산이 크다.

## 인사이트

### 클라우드 벤더는 이제 “에이전트를 위한 문서”를 일급 산출물로 만든다

`google/skills`의 진짜 의미는 스킬 몇 개의 내용물이 아니라, 구글이
“에이전트가 읽을 문서”를 별도 산출물로 분리해 버전 관리하기 시작했다는
사실에 있다. 지금까지 클라우드 문서는 사람이 읽는 것을 전제로 쓰였다.
이제 같은 지식이 progressive disclosure에 맞춰, description은 트리거로,
본문은 실행 절차로 재구조화된다. 독자가 사람에서 에이전트로 바뀌면 문서의
형태 자체가 바뀐다.

이것은 SEO가 검색엔진을 독자로 상정해 콘텐츠 구조를 바꿨던 것과 같은
종류의 전환이다. 당시 웹 콘텐츠는 사람과 크롤러 둘 다를 만족시키도록
재편됐다. 이제 클라우드 문서는 사람과 에이전트 둘 다를 위해 이중 포맷으로
관리될 가능성이 높다. 머지않아 “사람용 docs”와 “에이전트용 skills”가 같은
소스에서 갈라져 나오는 빌드 파이프라인이 표준이 될 것이다.

이 전환의 2차 효과는 측정과 인센티브에 있다. 사람용 문서의 가치는 조회수나
체류시간으로 측정됐지만, 스킬의 가치는 “에이전트가 이 스킬을 활성화해 작업을
성공시킨 비율”로 측정될 것이다. 그 지표가 생기면 클라우드 벤더들은 자사
서비스가 에이전트 작업에서 더 자주, 더 성공적으로 선택되도록 스킬을
최적화하기 시작한다. “에이전트 친화적 문서”가 새로운 경쟁 전장이 된다.

### 설치 가능한 능력 패키지는 패키지 레지스트리의 역사를 되풀이한다

`npx skills add google/skills`라는 명령은 npm, PyPI, Homebrew가 걸어온
길을 떠올리게 한다. 처음엔 편리한 설치 경험으로 시작해 폭발적으로 패키지가
늘고, 곧이어 버전 충돌, 의존성 지옥, 악성 패키지, 유기된 메인테이너,
typosquatting이 따라왔다. 스킬 생태계도 같은 곡선을 그릴 공산이 크다.
지금은 구글·Flutter·Dart 같은 공신력 있는 저장소가 중심이지만, 설치
경로가 임의의 owner/repo로 열려 있는 한 결국 같은 문제를 마주한다.

특히 구글 저장소가 Skill Registry API를 포함한다는 점이 의미심장하다.
스킬을 등록하고 발견하는 레지스트리가 등장한다는 것은, 패키지 생태계가
밟은 “중앙 레지스트리 → 의존성·신뢰 문제 → 거버넌스 필요”의 경로를 그대로
간다는 신호다. 레지스트리가 생기는 순간 발견성(discoverability)과 큐레이션,
그리고 누가 무엇을 신뢰할지의 문제가 전면에 떠오른다.

여기서 글이 얼버무리는 트레이드오프가 드러난다. 최소 명세(`name` +
`description`)의 단순함은 초기 채택을 폭발시키지만, 바로 그 단순함이
나중에 버전·신뢰·권한 메타데이터를 추가로 욱여넣어야 하는 부채가 된다.
npm의 `package.json`이 처음엔 단순했다가 점점 복잡해진 것처럼, `SKILL.md`
frontmatter도 시간이 지나며 무거워질 것이다. 단순함은 공짜가 아니라
미뤄진 비용이다.

### 포맷을 통제하는 자와 콘텐츠를 채우는 자는 다른 게임을 한다

가장 곱씹을 만한 구조는 Anthropic이 포맷을 만들고, 구글이 그 포맷에
콘텐츠를 채우는 분업이다. Agent Skills 포맷은 Anthropic이 만들어 오픈
스탠더드로 풀었고, 구글·OpenAI(Codex)·마이크로소프트(Copilot)까지 같은
포맷을 채택했다. 즉 경쟁 관계인 거대 기업들이 라이벌이 정의한 포맷 위에서
각자 콘텐츠와 클라이언트를 경쟁시키는 그림이다.

이것은 어도비가 PDF를, 혹은 넷스케이프 시절 누군가 포맷을 열어 표준이
되면 경쟁자들이 그 위에서 싸웠던 역사와 닮았다. 포맷을 먼저 열어 표준으로
만든 쪽은 단기적으로는 통제권을 양보한 것처럼 보이지만, 생태계의 기본
어휘를 정의했다는 장기 이점을 얻는다. progressive disclosure라는 설계
철학, `SKILL.md`라는 구조, description-as-trigger라는 패턴은 모두
Anthropic이 정한 전제이고, 구글의 스킬조차 그 전제 안에서 사고한다.

구글에게 이 분업의 위험은, 콘텐츠 레이어는 모방·복제가 쉽다는 데 있다.
구글 클라우드 스킬은 구글만 잘 쓸 수 있지만, “클라우드 온보딩 레시피”라는
형태 자체는 AWS나 Azure도 똑같이 만들 수 있다. 차별화는 콘텐츠의 질과
갱신 속도, 그리고 자사 클라이언트(Gemini CLI)와의 통합 깊이에서 나올
수밖에 없다. 포맷이 공용이 된 세계에서 진짜 해자는 “에이전트가 내 서비스를
얼마나 능숙하게, 얼마나 자주 선택하게 만드느냐”로 좁혀진다. 구글이 콘텐츠
저장소를 일찍 출하한 것은 그 좁은 해자를 선점하려는 포석으로 읽힌다.
