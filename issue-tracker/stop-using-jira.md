# Stop Using JIRA: How We Scaled to 50 Engineers Using Only a Markdown File

원문: <https://medium.com/@kp9810113/799797f7fb4b>

한국어 번역: [우리 팀이 JIRA 없애고 마크다운 파일 하나로 전환한 이유](https://yozm.wishket.com/magazine/detail/3617/)

## 핵심 요약

- 팀은 JIRA 같은 이슈 트래커 대신, 하나의 Markdown 파일을 중심으로 업무를 운영했다.
- 핵심은 도구 기능의 풍부함보다, 팀 전체가 같은 문맥을 공유하는 운영 규칙이었다.
- 작업을 작은 단위로 쪼개고, 우선순위를 명시하고, 주기적으로 청소(완료/중단/보류 정리)하면서 문서를 살아있는 상태로 유지했다.
- 엔지니어 수가 늘어나도 확장된 이유는, 고급 기능이 아니라 정보 구조의 단순함과 높은 가시성이었다.
- 즉, 확장은 툴 라이선스 업그레이드가 아니라, 팀 습관의 표준화에서 나왔다.

## 인사이트

### 1. 확장성의 병목은 도구가 아니라 합의 비용이다

많은 조직은 인원 증가 시 "더 강한 도구"를 찾는다.
하지만 실제 병목은 기능 부족보다, 상태 정의와 우선순위 해석의 불일치다.
Markdown 한 장으로도 운영이 되는 팀은,
이미 "무엇이 중요하고, 무엇이 끝났는지"에 대한 합의가 강하다.

### 2. PM 도구는 종종 복잡성을 숨기고, Markdown은 노출한다

JIRA의 워크플로우, 필드, 자동화는 강력하지만, 설계가 어설프면 오히려 복잡성을 UI 뒤에 감춘다.
반면 Markdown은 구조가 거의 없어서, 팀의 실제 혼선이 그대로 드러난다.
이 노출 자체가 운영 개선의 피드백 루프가 된다.

### 3. "기능 중심 관리"에서 "리듬 중심 운영"으로

성공 포인트는 상태값 개수가 아니라 운영 리듬이다.
(예: 짧은 주기의 리뷰, 미완료 항목 정리, 우선순위 재정렬.)
도구를 바꿔도 리듬이 없으면 다시 혼란으로 돌아간다.
따라서 도입 순서는 "리듬 설계 → 도구 선택"이 맞다.

### 4. 단순한 시스템일수록 책임이 팀에 돌아온다

Markdown 체계는 관리자 버튼으로 문제를 해결해주지 않는다.
대신 팀이 직접 문서를 유지해야 한다.
이 책임 전환은 초기에는 불편하지만,
장기적으로는 "내 일의 상태를 내가 명확히 표현"하는 강한 오너십 문화로 연결된다.

### 5. 50명까지의 성공은 "최적 도구"가 아니라 "적정 복잡도"의 승리

이 사례의 메시지는 "JIRA는 나쁘다"가 아니다.
핵심은 현재 팀 규모와 성숙도에 맞는 복잡도를 선택하라는 점이다.
복잡도가 요구사항보다 앞서가면 운영 비용이 커지고, 요구사항보다 낮으면 추적성이 무너진다.

## 실무 적용 가이드

### 바로 적용 가능한 7일 실험

1. 단일 Markdown 백로그 파일 하나를 만든다.
2. 섹션을 "Now / Next / Later / Done"으로 고정한다.
3. 모든 항목에 담당자, 기대 완료일, 다음 행동 1줄을 적는다.
4. 매일 10분, 완료/중단 항목을 정리한다.
5. 주 1회, 항목 수를 강제로 제한한다(예: Now 최대 7개).
6. 2주 후, 누락/중복/지연 원인을 회고한다.
7. 회고 결과를 바탕으로 도구 추가 여부를 결정한다.

### 도구 전환 판단 기준

아래 신호가 반복되면 JIRA/Linear 같은 도구 도입을 고려한다.

- 의존성 관계가 문서에서 자주 깨진다.
- 다수 팀 간 승인 흐름을 수동으로 추적하기 어렵다.
- 감사/리포팅 요구로 이력 자동화가 필요하다.
- 동일한 메타데이터를 반복 입력하느라 시간이 크게 든다.

## 결론

이 사례의 본질은 "JIRA를 버려라"가 아니라, "팀의 실행 규칙을 먼저 단순화하라"에 가깝다.
도구는 확장을 돕는 증폭기이지만,
혼란한 운영 모델을 스스로 치유해주지는 않는다.
