# Spanlens — LLM 호출과 에이전트 트레이스 관측

<https://www.spanlens.io/>

<https://github.com/spanlens/Spanlens>

<https://news.hada.io/topic?id=30076>

## 소개

Spanlens는 LLM API 호출을 모니터링하고, 비용을 추적하며, 에이전트 트레이스를
시각화하는 오픈소스 관측 플랫폼이다.
비용 추적과 에이전트 디버깅의 개인적 불편함에서 출발한 프로젝트로,
`baseURL` 설정 변경 하나로 모든 요청을 자동 기록한다.

## 주요 기능

### 요청 로깅과 비용 추적

모든 LLM 호출을 모델명, 토큰 수, 지연 시간, 비용과 함께 기록한다.
요청별 세부 내역, 일간 합계, 예산 알림을 제공한다.

### 에이전트 트레이스

다단계 워크플로우가 폭포수 다이어그램으로 표시되며, 크리티컬 패스,
비용 귀속, 지연 이상치를 자동으로 강조한다.
LangGraph 토폴로지를 노드 단위로 시각화해 지연이 어디서 발생하는지 보여준다.

### 품질 평가

LLM-as-judge가 모든 응답을 0~1점으로 채점한다.
새 프롬프트 버전이 실제로 성능을 개선하는지 정량 비교가 가능하다.

### A/B 테스트

고정 데이터셋을 다른 프롬프트 버전과 모델에 걸쳐 재실행하고,
Welch t-test로 통계적 유의성을 검증한다.

### 이상 감지

지연 또는 비용의 7일 기준선 대비 3σ 이탈을 자동 감지한다.

### PII 보호

SSN, 신용카드 번호 같은 패턴을 로깅 시점에 탐지한다.
API 키는 저장 전 자동 마스킹되며, 요청별로 본문 로깅을 선택 해제할 수 있다.

## 기술 통합

**지원 제공자**: OpenAI(SDK·Azure), Anthropic(SDK·Bedrock), Google(Gemini·Vertex),
Mistral

**프레임워크**: TypeScript/JavaScript (Node, Edge, Next.js), Python 3.9+,
LangChain, LlamaIndex, Vercel AI SDK, OpenTelemetry(OTLP/HTTP)

**통합 방식**:

```typescript
import { createOpenAI } from "@spanlens/sdk/openai";
const openai = createOpenAI();
```

기존 OpenAI SDK를 Spanlens SDK로 교체하는 것이 전부다.
p99 오버헤드는 3ms 미만이며, 수집은 백그라운드 워커에서 비동기 처리된다.

## 분석

### 투명한 프록시 모델의 장단점

`baseURL` 변경으로 기존 코드를 수정하지 않고 관측 계층을 추가하는 방식은
진입 장벽이 낮다.
그러나 이 구조는 Spanlens가 모든 LLM 트래픽의 중간에 위치한다는 뜻이기도 하다.
Spanlens SaaS를 쓴다면 모든 프롬프트와 응답이 Spanlens 서버를 거친다.
자체 호스팅 옵션이 존재하는 이유가 바로 이 보안 우려 때문이다.

### LangGraph 토폴로지 시각화

LLM 에이전트 디버깅의 가장 어려운 점은 여러 단계에 걸친 지연과 오류를
추적하는 것이다.
LangGraph 노드 단위 시각화는 에이전트 프레임워크와 관측 도구의 통합이
점점 중요해진다는 방향을 보여준다.
에이전트 복잡성이 증가할수록 “무슨 일이 일어났는가”를 사후에 재구성하는
능력이 프로덕션 안정성의 핵심이 된다.

### 통계적 프롬프트 A/B 테스트

Welch t-test를 적용한 A/B 테스트는 “더 좋은 프롬프트”를 인상이 아닌 데이터로
결정할 수 있게 한다.
프롬프트 엔지니어링이 예술에서 과학으로 이동하는 방향과 맞닿아 있다.

## 비평

### 강점 — 자체 호스팅으로 보안 요구 대응

기업 환경에서 LLM 호출 내용은 보안 민감 데이터다.
자체 호스팅 옵션은 이 우려를 해결한다.
MIT 라이선스 오픈소스이므로 코드 감사도 가능하다.
Langfuse, LangSmith 같은 경쟁 도구 대비 자체 호스팅 지원이 명확한 장점이다.

### 약점 — 제공자 의존적 비용 계산의 정확성

토큰 비용을 자동으로 계산하지만, 제공자가 가격을 변경하면 SDK 업데이트가
필요하다.
새 모델이 출시되거나 가격 구조가 복잡해질 때 비용 추적의 정확성이 얼마나
유지되는지가 관건이다.

### 빈 자리 — 멀티모달 추적

이미지, 오디오, 비디오를 처리하는 LLM 호출의 추적은 텍스트보다 복잡하다.
멀티모달 에이전트가 일반화되면 관측 도구도 그에 맞춰 진화해야 한다.
현재 Spanlens의 멀티모달 지원 범위가 명확하지 않다.

## 인사이트

### LLM 관측 가능성은 APM이 웹 서버에 했던 일을 반복한다

2000년대 초 웹 애플리케이션이 복잡해지면서 New Relic, Datadog 같은 APM 도구가
등장했다.
“이 요청이 왜 느린가”를 답하는 도구 없이는 프로덕션 운영이 불가능해졌기
때문이다.
LLM 에이전트 시대에 같은 일이 일어나고 있다.

에이전트가 여러 LLM 호출, 도구 사용, 외부 API를 연쇄적으로 실행하면,
“어떤 단계가 실패했는가”, “비용이 어디서 발생했는가”, “응답 품질이 어떻게
변했는가”를 추적하는 것이 운영의 핵심이 된다.
Spanlens가 하는 일은 LLM 에이전트의 APM이다.
이 범주가 독립 시장으로 성장하는 것은 LLM 에이전트의 프로덕션 채택이 확대될수록
불가피하다.

### “한 줄 통합”의 가치는 신규 도입에 있고, 한계는 깊은 관측에 있다

`baseURL` 교체로 즉시 관측을 시작하는 접근은 신규 도입에 이상적이다.
그러나 깊은 관측 — 특정 에이전트 스텝에서 어떤 도구가 호출됐는가, 메모리에
무엇이 있었는가 — 은 코드 수준 인스트루멘테이션 없이는 불가능하다.
투명한 프록시 방식과 코드 내 계측 방식의 경계는 관측 깊이의 경계다.

성숙한 LLM 팀은 결국 두 가지를 모두 필요로 한다.
초기에는 프록시 방식으로 빠르게 가시성을 확보하고, 운영이 복잡해지면
세밀한 계측을 추가한다.
Spanlens가 OpenTelemetry를 지원하는 것은 이 확장 경로를 열어두는 전략적 선택이다.

### 비용 가시성이 프롬프트 설계를 바꾼다

LLM 호출 비용을 실시간으로 추적하면 프롬프트 설계에 즉각적인 피드백이 생긴다.
“이 프롬프트가 월 얼마의 비용을 낳는가”를 알면 불필요한 컨텍스트를 줄이고,
더 저렴한 모델로 충분한 작업을 분리하는 의사결정이 가능해진다.
비용 가시성 없이는 이 최적화 루프가 동작하지 않는다.

비용 추적은 관측 가능성의 부수 기능처럼 보이지만, 실제로는 LLM 시스템의
설계를 바꾸는 피드백 메커니즘이다.
“토큰을 얼마나 쓰는가”를 인식하게 만드는 것만으로 팀의 프롬프트 설계 습관이
달라진다.
