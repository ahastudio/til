# LoRA Speedrun — LoRA 파인튜닝 속도 리더보드

<https://github.com/Saivineeth147/lora-speedrun>

HN 토론: <https://news.ycombinator.com/item?id=48974325> (145점, 10개 댓글)

## 소개

LoRA Speedrun은 LoRA(Low-Rank Adaptation) 파인튜닝 기법의 벽시계(wall-clock) 시간을 겨루는 공개 리더보드다.
[modded-nanogpt](https://github.com/KellerJordan/modded-nanogpt)에서 영감을 받아,
파인튜닝 최적화 기법을 동일한 조건에서 재현 가능하게 검증하는 것을 목표로 한다.

규칙은 단순하다.
고정된 태스크, 고정된 하드웨어(Modal L40S), 공개 리더보드.
모든 기록은 동일한 하드웨어에서 3번 재실행해 독립적으로 검증해야 한다.
Modal 무료 크레딧으로 누구나 참가하고 기록을 재검증할 수 있다.

두 개의 트랙이 있다.

| | 트랙 1 | 트랙 2 |
| ---------- | -------------------------------------------- | ------------------------------------------ |
| 기본 모델  | `Qwen/Qwen2.5-1.5B`                         | `HuggingFaceTB/SmolLM2-1.7B`              |
| 태스크     | GSM8K 수학 문제 → 정확도 ≥ 57.0%            | SQuAD v1.1 QA → EM ≥ 75.5%               |
| 하드웨어   | 1× L40S (48 GB)                              | 1× L40S (48 GB)                            |
| 제약       | 어댑터만, 훈련 가능 파라미터 ≤ 3,000만 개   | 동일                                       |

트랙 1 현재 기록은 1분 44초다.
기준선(11분 57초) 대비 86% 단축됐다.
데이터 정제(최단 4,000개 선별), 공격적 학습률 1 에포크, GPU 상주 패킹 루프, 청크 단위 CE 손실이 주요 기법이다.

## 분석

### 재현 가능한 비교가 없으면 기법 주장은 검증 불가능하다

LoRA 파인튜닝 최적화를 주장하는 논문과 도구는 많다.
DoRA, rsLoRA, Unsloth, 시퀀스 패킹, 데이터 선별 — 각각 속도 향상을 주장하지만,
다른 모델, 다른 데이터, 다른 하드웨어에서 측정됐기 때문에 실제 비교가 불가능하다.

LoRA Speedrun은 이 문제를 레퍼리 방식으로 푼다.
고정된 조건에서 동일한 하드웨어로 재실행하고, 3회 재현을 요구한다.
레퍼리가 있는 경쟁이라는 형식은 speedrun 문화에서 검증된 방법이다.
nanoGPT speedrun이 사전 훈련 옵티마이저 비교에 기여했던 방식과 같다.

### 기록 문서가 실패 사례를 포함한다

리더보드의 각 기록은 기법 설명과 함께 “거부된 변형” 섹션을 포함해야 한다.
어떤 시도가 효과 없었는지 기록하는 것이 어떤 기법이 이 조건에서 작동하지 않는지를 알려준다.

이것이 단순한 리더보드와 다른 지점이다.
1위 기록뿐 아니라 실패한 시도들이 문서화되면서, 리더보드 자체가 파인튜닝 최적화의 실험실 노트가 된다.

### 제한 조건이 창의성을 강제한다

jmward01은 자원 제한이 창의성을 강제하는 효과가 있다고 지적했다.[^jmward01]
“그냥 가중치를 키우고 데이터를 더 넣는다”는 접근은 자원 제한이 없으면 가장 쉬운 선택이다.
하드웨어와 파라미터 수를 고정하면 알고리즘적 개선이 유일한 경쟁 수단이 된다.
도시 계획에서 외곽 확장을 금지하면 도심 집약 개발이 이뤄지는 것과 같은 원리다.

## 비평

### AI로 생성된 코드베이스라는 의혹

HN에서 제기된 가장 날카로운 비판은 레포지터리가 100% AI 생성으로 보인다는 점이다.[^stephantul]
커밋 메시지, 코드 스타일, README 구조가 LLM 생성의 흔적을 보인다는 주장이다.

이것이 사실이든 아니든, 코드베이스의 신뢰성과 유지 가능성에 대한 의문은 남는다.
AI 생성 코드는 표면적으로 완성도 있어 보이지만 숨겨진 결함을 포함할 수 있고,
커뮤니티 기여를 받으려는 프로젝트에서는 코드를 이해하는 유지 관리자의 존재가 중요하다.

### 단일 태스크·단일 모델의 일반화 한계

stephantul은 단일 태스크와 단일 모델 조합에서 얻은 기록이 다른 설정으로 이전되지 않을 수 있다고 지적한다.[^stephantul]
GSM8K에서 1분 44초 기록을 세운 기법이 다른 태스크나 더 큰 모델에서 같은 속도 향상을 줄 것이라는 보장이 없다.

트랙 2를 추가해 서로 다른 모델 패밀리와 태스크 유형을 다루는 것은 이 한계에 대한 부분적인 대응이다.
그러나 두 트랙 모두 1–2B 파라미터 소형 모델에 한정돼 있다.
7B 이상의 모델, 다른 언어, 다른 도메인에서의 일반화는 별도 검증이 필요하다.

### LoRA라는 이름이 만드는 혼란

HN 스레드에서 여러 댓글이 LoRA를 무선 통신 프로토콜로 오해했다.[^pigeons][^walrus01]
“Low-Rank Adaptation”이라는 기계 학습 맥락의 LoRA가 장거리 저전력 무선 통신 프로토콜 LoRa(Long Range)와 표기가 겹친다.
프로젝트 이름이 의도한 독자 외에 불필요한 혼란을 만든다는 점은 커뮤니케이션 비용이다.

## 인사이트

### 파인튜닝 최적화의 실증적 기준이 부재했다

LLM 파인튜닝 생태계는 기법 주장의 홍수 속에서 실증적 비교 기준이 없었다.
어느 논문은 DoRA가 빠르다고 하고, 어느 블로그는 Unsloth가 10배 빠르다고 한다.
그러나 이들은 모두 서로 다른 설정에서 측정됐고, 재현 경로가 없다.

LoRA Speedrun이 이 공백을 채우려는 시도라는 점은 의미 있다.
nanoGPT speedrun이 Muon 옵티마이저의 발견을 촉발했듯,
고정된 조건의 파인튜닝 경쟁이 검증된 최적화 기법을 발굴하는 메커니즘이 될 수 있다.

### 경쟁 리더보드가 머신러닝 연구 방법론을 보완한다

전통적인 머신러닝 연구는 논문 → 리뷰 → 게재의 주기가 느리다.
소셜 코딩 플랫폼에서 공개 경쟁 리더보드 형태의 실험이 늘어나고 있다.
참가자가 기법을 공개하고, 다른 참가자가 즉시 재검증하며, 실패 사례도 누적된다.

이 형태가 peer review를 대체하지는 않는다.
그러나 “어떤 기법이 실제로 이 조건에서 얼마나 빠른가”라는 실용적 질문에
논문보다 빠르게 답을 내놓는 보완 메커니즘이 된다.
AI 도구의 발전 속도를 생각하면, 6–12개월 리뷰 주기 논문만으로는 따라가기 어려운 영역이다.
