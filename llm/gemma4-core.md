# Gemma 4: Google의 오픈 가중치 멀티모달 모델 패밀리

<https://ai.google.dev/gemma/docs/core?hl=ko>

## 소개

Gemma 4는 Google이 출시한 오픈 가중치 생성형 AI 모델 패밀리로, 텍스트·이미지·비디오·오디오
입력을 지원하고 최대 256K 토큰 컨텍스트 창을 제공한다.
Apache 2.0 라이선스로 배포되어 상업적 사용과 파인튜닝, 재배포가 허용된다.
Kaggle과 Hugging Face에서 내려받을 수 있다.

아키텍처는 세 계열로 나뉜다.
**Small(소형)** 계열은 E2B(유효 파라미터 2B)와 E4B(유효 파라미터 4B)로 구성되며,
모바일, 엣지, 브라우저 배포에 최적화됐다.
파라미터-레이어 임베딩(PLE, Parameter-Layer Embeddings) 기법을 사용해 효율을 극대화한다.
**Dense(밀집)** 계열은 31B 모델로, 서버급 성능과 로컬 실행의 균형점을 목표로 한다.
**Mixture-of-Experts** 계열은 26B A4B 모델로, 전체 26B 파라미터를 로드하되 토큰당
4B만 활성화해 높은 처리량과 고급 추론을 제공한다.

메모리 요구사항은 양자화 수준에 따라 크게 달라진다.

| 모델       | BF16 (16비트) | SFP8 (8비트) | Q4_0 (4비트) |
| ---------- | ------------- | ------------ | ------------ |
| E2B        | 9.6 GB        | 4.6 GB       | 3.2 GB       |
| E4B        | 15 GB         | 7.5 GB       | 5 GB         |
| 31B        | 58.3 GB       | 30.4 GB      | 17.4 GB      |
| 26B A4B    | 48 GB         | 25 GB        | 15.6 GB      |

주요 기능은 네 가지다.
설정 가능한 **사고 모드(Thinking Mode)**—모든 모델 변형에서 추론 작업을 위한 사고 모드를
지원한다.
**함수 호출(Function Calling)** 내장 지원으로 자율 에이전트 구현이 가능하다.
**네이티브 시스템 프롬프트** 지원으로 보다 구조적이고 제어 가능한 대화를 제공한다.
**다중 토큰 예측(MTP, Multi-Token Prediction)**—모든 Gemma 4 모델에 전용 드래프트
모델이 포함되어 추론 가속(speculative decoding)을 지원한다.
컨텍스트 창은 소형 모델이 128K 토큰, 중간 모델이 256K 토큰을 지원한다.

## 분석

### 세 아키텍처가 커버하는 배포 스펙트럼

E2B/E4B → 31B Dense → 26B MoE 세 계열은 배포 환경별 최적화 전략이 다르다.
E2B는 Q4_0 기준 3.2 GB로 스마트폰급 디바이스에서도 실행 가능하며,
E4B는 5 GB로 고급형 모바일 기기나 노트북 NPU에 적합하다.
31B Dense는 BF16 기준 58.3 GB가 필요해 고사양 워크스테이션이나 서버 환경을 전제로 한다.
26B MoE는 전체 파라미터를 메모리에 올려야 하지만 활성화 파라미터가 4B뿐이어서,
동일한 추론 비용으로 더 큰 총 용량의 지식을 활용할 수 있다.

### PLE(파라미터-레이어 임베딩)의 의미

E2B/E4B 모델이 채용한 PLE 기법은 “유효 파라미터(effective parameters)”라는 표현을
낳는다.
실제 물리적 파라미터 수보다 적은 메모리로 더 넓은 임베딩 공간을 표현할 수 있게 하는
기술이다.
이는 소형 모델의 고질적 약점인 어휘 표현력 부족을 완화하는 방향으로 작동한다.
“2B 파라미터지만 4B급 성능”이라는 마케팅 언어의 기술적 근거이기도 하다.

### 사고 모드와 함수 호출의 조합

설정 가능한 사고 모드(Thinking Mode)와 함수 호출(Function Calling)의 조합은
주목할 만한 설계 선택이다.
사고 모드는 추론 작업에서 중간 단계를 생성하는 Chain-of-Thought 계열 기법이며,
함수 호출은 외부 도구와의 연동을 가능하게 한다.
이 두 기능이 함께 제공되면 “생각하고, 도구를 쓰고, 다시 생각하는” 에이전트 루프를
소형 모델에서도 구현할 수 있다.
이것은 클라우드 전용이었던 에이전트 아키텍처가 엣지로 이동하는 흐름과 직결된다.

### 다중 토큰 예측(MTP)과 드래프트 모델

모든 Gemma 4 모델에 전용 드래프트 모델이 포함된다는 점은 추론 가속 전략을 기본으로
내장한다는 의미다.
Speculative Decoding은 작은 드래프트 모델이 여러 토큰을 먼저 예측하고, 큰 검증
모델이 이를 한 번에 수락/거절하는 방식이다.
일반적으로 동일한 품질에서 2~4배 처리량 향상이 보고된다.
드래프트 모델을 별도로 선택하지 않아도 되도록 패키지화한 것은 개발자 경험을 낮추는
설계다.

## 비평

### 강점: 오픈 가중치 + Apache 2.0의 실용성

Apache 2.0 라이선스는 오픈소스 AI 생태계에서 가장 허용적인 선택 중 하나다.
상업적 사용, 파인튜닝, 재배포 모두 허용되며, 별도 라이선스 협상이 필요 없다.
여기에 Kaggle과 Hugging Face라는 두 주요 배포 채널을 동시에 지원해 접근성을 최대화했다.
이는 Meta의 Llama 3 라이선스와 비교해도 개발자에게 더 우호적인 조건이다.

### 한계: 벤치마크와 실제 성능의 간극

문서는 각 모델의 포지셔닝(“서버급 성능”, “고급 추론”)을 설명하지만,
구체적인 벤치마크 수치를 제시하지 않는다.
E4B가 경쟁 모델 대비 어떤 태스크에서 얼마나 나은지, 26B MoE가 31B Dense에 비해
언제 더 유리한지를 판단하는 데이터가 없다.
모델 선택 가이드가 배포 환경 기준(메모리, 플랫폼)으로만 구성되어, 태스크 적합성
기준이 부족하다.

### 보완이 필요한 관점: 오디오 지원의 범위

E2B와 E4B 소형 모델에 오디오 네이티브 지원이 있다고 언급되지만, 31B와 26B MoE에
대한 오디오 지원 여부가 명확하지 않다.
“오디오 지원”의 구체적 범위—음성 인식(ASR), 음성 합성(TTS), 오디오 분류—도
문서에서 명시되지 않는다.
비디오 이해(video understanding)도 마찬가지로, 어떤 해상도와 길이까지 처리 가능한지,
프레임 샘플링 방식은 무엇인지가 불분명하다.

## 인사이트

### 엣지 AI의 임계점: 5 GB 이하 모델이 바꾸는 것

E2B의 Q4_0 기준 3.2 GB, E4B의 5 GB는 단순한 메모리 수치가 아니다.
현재 고급형 스마트폰의 RAM이 12~16 GB이고, 온디바이스 AI에 할당 가능한 메모리가
통상 4~8 GB라는 현실을 고려하면, E4B까지의 모델은 실제 모바일 배포 임계점 안에
들어온다.
이것이 의미하는 것은 “클라우드 없이 스마트폰에서 멀티모달 + 함수 호출 + 사고 모드”가
가능해진다는 것이다.

클라우드 종속성이 없는 AI는 단순히 오프라인 동작을 넘어 다른 경제적 구조를 만든다.
API 호출 비용이 없고, 데이터가 기기 밖으로 나가지 않으며, 레이턴시가 네트워크
의존적이지 않다.
의료 기록, 법률 문서, 개인 금융처럼 데이터 외부 전송이 제한된 도메인에서 엣지
배포 모델은 단순히 “더 나은 선택”이 아니라 “유일한 선택”이 된다.
Gemma 4 소형 모델이 오디오와 비디오까지 처리한다는 것은, 이 시나리오가 텍스트와
이미지를 넘어 실시간 멀티미디어 처리로 확장됨을 뜻한다.

### MoE 아키텍처의 역설: “작은 계산, 큰 지식”의 트레이드오프

26B MoE 모델은 흥미로운 역설을 내포한다.
토큰당 4B만 활성화하지만 26B 전체를 메모리에 올려야 한다.
계산 비용은 4B급이지만 메모리 비용은 26B에 가깝다는 뜻이다.
Q4_0 기준 15.6 GB를 GPU에 올려야 하는데, 이는 고급형 소비자 GPU(RTX 4090의 24 GB
VRAM)로 실행 가능한 수준이다.

이 구조는 “지식의 크기”와 “추론 비용”을 분리한다.
Dense 모델에서는 파라미터 수가 지식 용량과 추론 비용 모두를 결정하지만,
MoE는 지식 용량을 결정하는 총 파라미터와 추론 비용을 결정하는 활성화 파라미터를
독립적으로 조정할 수 있다.
이것은 LLM 스케일링의 다음 차원이다.
“더 많은 것을 알지만 덜 생각하는” 모델과 “덜 알지만 더 많이 생각하는” 모델을 선택
가능한 스펙트럼으로 만든다.
Thinking Mode가 모든 아키텍처에 공통으로 제공되는 것은, “더 많이 생각하기”를 파라미터가
아니라 추론 시간 투자로 해결하려는 전략과 맞닿아 있다.

### Apache 2.0 오픈 가중치 전략: 생태계 구축의 새로운 공식

Google이 Gemma 4를 Apache 2.0으로 배포하는 것은 기술 자체보다 생태계 전략으로서
더 중요하다.
Meta Llama의 라이선스가 사용자 수 기준 제한을 포함해 기업 사용에 마찰을 만들었던
것과 달리, Apache 2.0은 어떤 규모의 기업도 제약 없이 사용할 수 있다.

오픈 가중치 모델이 확산되면 Google에게 직접적인 API 수익은 없다.
그러나 Gemma 생태계가 확장되면 Google Cloud의 TPU/GPU 인프라 수요가 늘고,
Vertex AI에서 파인튜닝을 원하는 기업들이 늘어난다.
무엇보다 Gemma 아키텍처를 기반으로 한 파인튜닝 모델들이 생태계를 점령하면,
그 위에서 Google의 도구와 인프라가 자연스럽게 선택된다.
Android AI에 Gemma가 탑재되면 10억 대 이상의 기기에서 Google 아키텍처가 실행된다.

이것은 오픈소스로 생태계를 선점하고 클라우드/인프라/디바이스에서 수익화하는
“레이어 분리 전략”이다.
과거 Android가 모바일 OS를 오픈소스로 풀어 모바일 생태계의 표준이 된 것과
구조적으로 같다.
Gemma가 LLM 생태계에서 Android와 같은 역할을 하려는 것이라면, Apache 2.0은 그
전략의 핵심 수단이다.
