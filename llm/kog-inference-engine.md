# Kog 추론 엔진 - 표준 데이터센터 GPU에서 요청당 3,000 tokens/s

원문: <https://blog.kog.ai/real-time-llm-inference-on-standard-gpus-3-000-tokens-s-per-request/>

## 요약

파리 기반 스타트업 Kog는 자사의 Kog Inference Engine(KIE) 기술 프리뷰를
공개하면서, 표준 데이터센터 GPU에서 단일 요청당 8× AMD MI300X 기준
3,000 tokens/s, 8× NVIDIA H200 기준 2,100 tokens/s를 달성했다고 밝혔다.
모두 FP16 가중치이며 양자화, speculative decoding, pruning, early exit,
KV 캐시 압축 없이 측정되었다. 현재 프리뷰는 자사 2B 모델 Laneformer를
운영 중이며, 향후 외부 MoE 모델을 비슷한 속도로 지원할 계획을 명시한다.

기술 프리뷰가 가리키는 명제는 분명하다. “일반 GPU도 전용 추론 하드웨어
카드에 견줄 속도를 낼 수 있다 — 아키텍처/엔진/커널을 함께 설계할 때”.
일반화된 메시지는 추론 속도 부족이 GPU 하드웨어 자체가 아니라 그 위에
쌓인 추론 소프트웨어 스택의 부채에서 온다는 것이다.

단일 요청 디코드 속도가 왜 결정적인지에 대해 글은 “지능과 반복 속도의
곱”이 자율 AI 에이전트 시대의 생산성 프런티어라고 말한다. 에이전트형
SE는 inspect → plan → edit → test → revise 가 순차 의존하므로 토큰
생성 속도가 곧 한 사이클 시간이다. 50,000 토큰을 100 tokens/s로 생성하면
약 8분, 3,000 tokens/s로는 20초 미만 — 이 차이가 제품 가능성의 경계가
된다.

기술적 핵심은 단일 요청에서는 메모리 대역폭이 1차 보틀넥이라는 사실이다.
배치 1 자기회귀 디코딩은 weight를 메모리 계층을 따라 흘려보내는 “matrix
× vector” 연산이며, `tokens/s ≤ effective_memory_bandwidth / (β × active_weight_bytes + KV cache)`
의 상한에 묶인다. FP16에서 산술 강도는 약 1 FLOP/byte이고, 현대 GPU는
HBM 대역폭 1바이트당 수백 FLOP를 제공하므로, MFU(Model FLOP Utilization)이
아니라 MBU(Memory Bandwidth Utilization)가 1차 지표가 된다. 2B FP16
모델의 이론 상한은 8× H200이 약 7,700 tokens/s, 8× MI300X가 약 8,400
tokens/s. 3,000 tokens/s는 이 상한의 약 36% 활용에 해당한다.

3,000 tokens/s 달성을 위한 마이크로초 예산은 토큰당 약 333μs로 모든
레이어, LM 헤드, 샘플링을 포함한다. 표준 스택이 잃는 비용은 커널 경계
오버헤드, CPU 호스트 스케줄링, 그리드 동기화, GPU 간 collective(AllReduce),
NUMA에 가까운 unified memory 경로, 불완전한 가중치 캐싱, GEMM이 아닌
연산(softmax, normalization, routing)의 비대역폭 점유로 정리된다.
Kog는 세 층의 공동 설계로 이를 줄인다. 첫째, monokernel runtime —
토큰 생성 전체가 한 개의 영속 GPU 프로그램으로 실행되어 커널 경계,
호스트 스케줄링, CPU 측 토큰 샘플링을 임계 경로에서 제거한다. 둘째,
KCCL이라는 자체 GPU 간 collective 라이브러리로 monokernel 일정에
통합 가능한 3μs 미만의 예측 가능한 지연을 제공한다. 셋째, Laneformer
모델 아키텍처가 채택한 Delayed Tensor Parallelism(DTP)으로 텐서 병렬
디코딩의 의존 구조를 바꿔, 크로스 디바이스 통신을 임계 경로 차단이
아니라 유용한 연산과 겹쳐 실행한다. AMD MI300X에서는 8개 XCD-4개 IOD
칩렛 토폴로지에 맞춰 XCD별 배리어 지연을 측정·매핑하여 약 600ns의
안정적 배리어를 달성했다.

대형 MoE 모델 확장은 “토큰당 활성 파라미터”를 기준으로 계산된다.
8× MI300X(~33.6 TB/s)에서 대역폭 상한 추정은 Qwen3-Coder-Next(3B FP8)가
~11,200 tokens/s, DeepSeek-V4-Flash(13B MXFP4/FP8)가 ~3,560,
DeepSeek-V4-Pro(49B MXFP4/FP8)가 ~940. 현재 프리뷰의 MBU 36%를 가정한
보수적 추정은 각각 약 4,000 / 1,270 / 335 tokens/s. 차세대 GPU(Rubin,
MI450 — 2026년 하반기)는 메모리 대역폭이 약 4배 향상될 것으로 명시된다.
Laneformer 2B는 HumanEval 50%, 시퀀스 4,096(128k 개발 중), Nemotron
데이터셋 6T 토큰 학습으로 보고된다.

회사 정보로 Kog는 2023년 파리에서 École Polytechnique 출신 Gaël Delalleau가
창업했고, 5명의 PhD를 포함한 11명 팀, Varsity VC와 BPI France Deep Tech
프로그램에서 500만 달러 투자, 2025년 10월 “French Tech 2030” 정부
인증을 받았다.

## 분석

### MBU 중심성 — “FLOPS 신화”의 해체

이 글의 가장 명료한 기여는 단일 요청 추론의 본질이 산술 강도 1
FLOP/byte 영역에 있다는 점을 수식으로 명시한 것이다. 학계와 업계는
GPU 성능을 FLOPS로 비교하는 습관이 깊다. 그러나 배치 1 자기회귀
디코딩에서는 가중치를 단 한 번만 곱하기 위해 HBM에서 SRAM/레지스터로
끌어와야 하므로 메모리 대역폭이 1차 제약이고, 연산 유닛은 대부분 노는
상태다. 글은 이 사실을 부등식 `tokens/s ≤ BW/(β·W + KV)`로 명시화해서
이후 모든 최적화가 BW 또는 β를 손대는 작업임을 자연스럽게 드러낸다.

### 마이크로초 회계와 “표준 스택의 7가지 손실”

토큰당 333μs라는 예산을 명시하고, 그 안에서 어디서 시간이 새는지를
7가지로 분류한 부분은 추론 엔진 설계의 “회계 장부”다. 커널 경계,
CPU 스케줄링, 그리드 sync, AllReduce, NUMA 메모리 경로, weight reload,
non-GEMM 작업. 이 분류는 학술 논문의 “병목 분석” 챕터와 동일한 구조를
띠지만 더 운영 친화적이다. 각 항목은 곧 monokernel·KCCL·DTP가 푸는
문제와 1:1로 대응한다. 즉 “문제 7개 → 해법 3개”의 매핑이 글의 골격을
이룬다.

### 칩렛 토폴로지에 대한 미시 최적화의 의미

MI300X의 8 XCD-4 IOD 토폴로지에서 XCD별 배리어 지연을 측정하고
매핑하여 600ns 배리어를 달성한 사례는 “하드웨어 친화 최적화”의 가장
구체적 형태다. 이는 [[bijou64]]가 보여준 “데이터 레이아웃을 현대 CPU의
브랜치 예측·캐시 라인에 맞추는 설계”와 같은 정신이지만 GPU 칩렛 시대로
이식된 버전이다. GPU가 더 이상 단일 die가 아니라 칩렛 조립체가 되면서
NUMA 비슷한 토폴로지 인식이 필수가 되고 있다.

### MoE 시대의 “활성 파라미터” 패러다임

DeepSeek-V4-Pro의 1.6T 총 파라미터 vs 49B 활성 파라미터 비교는 MoE가
어떻게 추론 비용을 분리하는지를 가장 선명히 보여준다. 단일 요청 속도는
총 파라미터가 아니라 토큰당 활성 파라미터로 결정되므로, 거대한 모델도
3,000 tokens/s 영역에 들어올 수 있다. 이는 [[recent-llm-architectures]],
[[deepseek-v4]]가 다룬 MoE 흐름의 추론 측면 보완이다. 모델 크기 경쟁이
“총 파라미터”에서 “활성 파라미터”로 옮겨가는 이유는 학습뿐 아니라
추론 경제학 때문이다.

## 비평

### 강점 - 이론 상한과 실측을 한 표에 놓는 정직성

MBU 36% 가정과 그에 따른 보수적 예측을 명시하는 태도는 드물게 정직하다.
보통 추론 엔진 마케팅은 최선의 케이스 수치만 노출하지만, 이 글은
대역폭 상한, 현재 MBU, 향후 추정을 모두 한 줄에 둔다. 독자가 직접
재검증할 수 있는 형태로 데이터를 제시하는 글쓰기는 신뢰의 1차 신호다.

### 약점 - 자체 모델 Laneformer라는 통제 변수

3,000 tokens/s 수치는 “외부 표준 모델”이 아니라 자체 설계 모델
Laneformer 2B에서 측정되었다. Laneformer는 Delayed Tensor Parallelism을
가정하므로 KIE 엔진과 공진화된 아키텍처다. 일반 사용자가 가져온
Qwen, Llama, DeepSeek 모델에서 같은 속도가 나올지는 별개의 질문이다.
글은 “외부 MoE 모델을 비슷한 속도로 지원할 계획”이라고 명시하지만
실측치는 아직 없다. 따라서 “표준 GPU + 표준 모델”의 결합이 아니라
“표준 GPU + 자체 모델”의 결합이라는 단서를 강조해야 한다.

### 약점 - HumanEval 50%의 모델 능력 한계

Laneformer 2B의 능력은 HumanEval 50%로 보고된다. 이는 동급 2B 모델 중
경쟁력 있는 수준이지만, 에이전트형 SE 사용 사례에서 요구되는 능력
한계 — 도구 사용, 긴 컨텍스트 추론, 다단계 계획 — 와는 거리가 있다.
“속도 × 지능” 곱이 가치라면 한쪽 항만 큰 결과를 보여주는 형태다.
프리뷰의 의도는 인프라 데모지 모델 데모가 아니라는 점에서 합리적이지만,
독자가 두 항을 분리해서 이해하지 않으면 잘못된 기대가 생긴다.

### 약점 - 8 GPU 노드의 경제성에 대한 침묵

8× MI300X 한 노드의 자본비용은 50만 달러대, 전력은 5kW를 넘는다.
이 자원으로 “요청당 3,000 tokens/s”를 내는 것이 어떤 단가에 해당하는지
글은 침묵한다. 동시 요청 처리량(throughput per dollar), 1M 토큰당
비용 같은 운영 지표가 비교군(SaaS API, vLLM 클러스터, Cerebras WSE)
대비 어디 있는지를 알아야 진짜 ROI가 보인다. 단일 요청 속도는 매혹적이지만
경제성과 곱해져야 의미가 산다.

### 약점 - β 인자(가중치 재로드)의 정량 사례 부족

부등식 핵심 변수인 β를 어떻게 1.0에 가깝게 유지했는지는 표면 설명에
그친다. monokernel이 캐시 관리를 더 잘한다는 주장이 강하지만, MI300X의
HBM3 대역폭과 SRAM 용량을 감안한 실제 β 측정치, KV 캐시가 토큰 늘어남에
따라 증가하는 곡선, 4,096 시퀀스 끝에서의 속도 저하 같은 정량 그래프가
빠져 있다.

## 인사이트

### 추론 속도는 “지능 × 반복”이라는 새 1축으로 부상한다

LLM의 1차 시대(2020~2023)는 능력 곡선이 모든 것을 지배했다. 더 큰
모델, 더 똑똑한 답이 가치의 척도였고 속도는 사용성의 부차적 변수였다.
2026년 에이전트 시대에서 이 균형이 깨진다. 에이전트 워크플로우는
inspect → plan → edit → test → revise의 사이클이고, 각 단계가 다음
단계를 차단한다. 즉 사이클 시간이 곧 한 단위 작업 시간이고, 사이클
시간은 토큰 생성 속도가 결정한다. 100 tokens/s와 3,000 tokens/s는
같은 작업을 8분 vs 20초로 가르고, 이 차이는 “시도 횟수”에 들어간다.
시도 횟수가 10배 많아지면 발견 가능한 해법 공간도 10배 커진다.

이 함의는 모델 평가 척도에 직접 영향을 준다. HumanEval처럼 “한 번
풀이”를 측정하는 벤치마크는 에이전트 시대의 가치를 측정하지 못한다.
실제 에이전트는 “50번 풀어보고 가장 잘 작동하는 것을 고른다”이고,
이 곱은 단일 풀이 정확도 70%와 풀이당 30초인 모델보다, 단일 정확도
50%지만 풀이당 3초인 모델이 더 강할 수 있다는 결과를 낳는다. 즉
speed-accuracy tradeoff가 에이전트 평가의 1급 축으로 부상한다.

세 번째 함의는 인프라 경제성을 바꾼다. 동시 요청 처리량(throughput)에
최적화된 vLLM, TGI, TensorRT-LLM 류의 서빙 스택은 “채팅 봇 SaaS”의
경제학에 맞춰져 있었다. 한 사용자 한 요청, 그 사이에 큰 idle. 에이전트
워크플로우는 단일 사용자가 단일 세션에서 수만 토큰을 직렬로 생성한다.
이 경우 throughput보다 single-request speed가 1차 KPI다. KIE 같은
엔진이 새 카테고리(“에이전트 추론 인프라”)를 정의하기 시작하는 이유다.
이는 LLM 인프라 시장이 “채팅 인프라 + 에이전트 인프라”로 분기되는
초기 신호로 읽힌다.

### MBU가 새 MFU가 되면서 GPU 비교 방식이 바뀐다

10년간 GPU 비교의 표준은 FLOPS였다. A100 vs H100 vs B100을 비교할 때
FP16 FLOPS, FP8 FLOPS, TF32 FLOPS가 마케팅의 1면을 차지했다. 그러나
배치 1 자기회귀 추론이 메모리 대역폭에 묶이는 영역이 되면서, FLOPS는
무관해지고 HBM 대역폭이 결정 변수가 된다. H100과 H200의 차이가 FLOPS가
아니라 HBM3e 대역폭에서 갈리는 이유, MI300X가 H200을 단일 요청 추론에서
앞서는 이유 — 모두 같은 메커니즘이다.

이 변화는 GPU 시장 동력학을 바꾼다. NVIDIA의 우위는 CUDA와 FLOPS
브랜드에 강하게 묶여 있는데, 단일 요청 추론에서 메모리 대역폭이 결정
변수가 되면 그 우위가 자동으로 보존되지 않는다. AMD MI300X가 KIE 위에서
H200을 앞서는 사례는 “소프트웨어 스택 + 메모리 대역폭” 조합이 “CUDA
+ FLOPS” 조합을 넘어설 가능성을 보여준다. 이는 [[deepseek-v4]]가
지적한 “하드웨어 다양화”의 흐름과 같은 방향이다.

여기서 두 번째 함의가 도출된다. GPU 선택 기준이 워크로드별로 갈라진다.
학습은 여전히 FLOPS와 NVLink 대역폭이 결정 변수이고, 배치 추론은 가격
대비 FLOPS, 단일 요청 추론은 메모리 대역폭과 칩렛 토폴로지가 결정
변수다. 한 종류의 GPU가 모든 워크로드에 최적이라는 가정은 깨지고,
워크로드별 전용 인스턴스가 나타난다. AWS가 이미 Inferentia, Trainium,
Graviton으로 워크로드를 분리한 흐름이 GPU 영역으로 확산된다.

세 번째 함의는 칩 설계 우선순위를 바꾼다. Rubin과 MI450이 “메모리
대역폭 4배”를 목표로 한다는 사실은 차세대 데이터센터 GPU가 FLOPS보다
대역폭에 베팅한다는 의미다. HBM4의 등장, on-package SRAM 확장, 칩렛
간 인터커넥트 — 모두 “FLOPS 늘리기”에서 “데이터를 더 빨리 옮기기”로
초점이 이동한다. 이는 NVIDIA Grace Hopper의 통합 메모리 설계, Cerebras
WSE-3의 대규모 on-die SRAM, Groq LPU의 SRAM-only 설계가 가리키던
동일한 방향이다.

### Monokernel은 GPU를 “느슨한 가속기”에서 “타이트한 컴퓨터”로 바꾼다

전통적인 GPU 사용은 “CPU가 짧은 커널을 launch → GPU가 실행 → 결과를
CPU로 → 다음 커널 launch”라는 루프다. 이 모델은 GPU를 “느슨하게 호출
가능한 가속기”로 본다. monokernel은 토큰 생성 전체가 하나의 영속
프로그램으로 GPU에 상주하는 모델이다. 이는 GPU를 가속기가 아니라
“CPU 같은 본 컴퓨터”로 사용하는 패러다임 전환이다.

이 전환은 추론을 넘어 일반 GPU 컴퓨팅의 미래를 시사한다. CUDA의
초기 설계 가정은 “GPU는 사용자가 짧은 커널을 던지는 보조 프로세서”
였고, 그래픽 시대의 잔재였다. 영속 커널, GPU 위 스케줄러, GPU에서
도는 인터프리터(NVIDIA Megakernel, Triton의 persistent kernel)는 모두
이 가정을 거꾸로 뒤집는다. 영속 커널이 표준이 되면 “CPU-GPU 분리”라는
운영체제 아키텍처 자체가 다시 설계되어야 한다.

여기서 도출되는 두 번째 함의는 운영체제와 런타임 사이의 경계가
재배치된다는 것이다. 토큰 샘플링이 GPU 안에서 일어나고, KV 캐시 관리,
배치 관리, 라우팅이 GPU 위 스케줄러로 이전되면, 사실상 GPU는 미니
운영체제를 호스팅한다. 이는 Cerebras, Groq 같은 전용 추론 칩이 처음
정의한 모델이고, KIE는 같은 패러다임을 표준 GPU에 옮긴다. 즉 “전용
하드웨어 vs 범용 GPU”의 경계가 “하드웨어 차이”가 아니라 “소프트웨어
스택의 차이”로 좁아진다.

세 번째 함의는 보안과 멀티테넌시다. monokernel은 한 요청이 GPU 전체를
점유한다. 이는 단일 요청 속도에는 좋지만, 멀티테넌시에는 정반대로
작용한다. 클라우드 GPU 서빙은 한 GPU에 여러 고객을 띄우는 가정 위에
구축되어 있다. monokernel이 표준이 되면 이 가정이 깨지고, GPU 단위의
강한 격리 + 더 작은 GPU 단위(예: MIG, Multi-Instance GPU) 또는 더
탄력적인 가상화 계층이 필요해진다. 추론 인프라의 다음 세대는 “단일
요청 속도”와 “멀티테넌시 효율” 사이의 새 균형을 찾아야 한다.

### “일반 하드웨어가 못한 게 아니라 그 위 스택이 못했다”라는 진단의
일반화 가능성

글의 마지막 한 줄 — “표준 데이터센터 GPU가 이 카테고리에서 경쟁하지
못한 이유는 하드웨어가 아니라 그 위에 쌓은 소프트웨어 추론 스택의
방식 때문이다” — 은 LLM 추론을 넘어서는 일반 명제다. 같은 진단이
40년간 컴퓨터 산업의 여러 영역에서 반복되었다.

1990년대 RISC vs CISC 논쟁이 그랬다. RISC가 우월하다는 학술 합의에도
불구하고 x86이 살아남은 이유는 하드웨어 본질이 아니라 컴파일러 발전과
마이크로코드 최적화라는 소프트웨어 스택 진화였다. 2000년대 GPU 컴퓨팅
초기 시절, NVIDIA가 ATI(AMD)를 추월한 것도 칩 자체보다 CUDA라는 스택
설계가 결정했다. 2020년대 LLM 학습에서 NVIDIA가 우위를 굳힌 것 역시
하드웨어가 아니라 cuDNN, NCCL, Megatron이라는 스택의 누적이었다.

“하드웨어는 충분히 좋은데 스택이 못 따라간다”라는 진단은 곧 “스택을
다시 짜면 하드웨어 잠재력의 2-3배가 잠겨 있다”라는 시장 기회 선언이다.
KIE 같은 시도가 성공하면 다음 진단 대상은 무엇이 될까. CPU 추론
스택은 여전히 cpython의 GIL과 메모리 모델 같은 70년대 가정 위에 있다.
온디바이스 추론 스택(Android NNAPI, Apple MLX)도 같은 진단을 받을
가능성이 높다. 분산 학습 스택도 “GPU 토폴로지 인식”이 부족한 영역이
넓다.

여기서 도출되는 메타 원칙은 분명하다. 하드웨어 새 세대가 등장할
때마다 그 위 스택은 한 세대 늦게 따라온다. 그 격차의 기간에 “스택을
처음부터 다시 짜는” 팀이 큰 마진을 가져간다. KIE의 진짜 메시지는
“GPU 잠재력이 잠겨 있고, 잠근 자물쇠는 소프트웨어이며, 자물쇠를 푸는
사람이 다음 세대 인프라 회사의 주인공이 된다”이다. 이는 [[harness
engineering encyclopedia]]가 에이전트 영역에서 다룬 “하니스 엔지니어링이
새 가치 영역이 된다”라는 명제와 같은 형태의 일반 원칙이다.
