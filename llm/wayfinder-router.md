# Wayfinder Router

<https://github.com/itsthelore/wayfinder-router>

HN 토론: <https://news.ycombinator.com/item?id=48704373> (111점, 55개 댓글)

## 소개

Wayfinder Router는 로컬 모델과 호스팅 모델 사이에서 LLM 쿼리를 결정론적으로 라우팅하는 CLI 도구다.
라우팅 결정에 모델 호출 없이 프롬프트 구조를 직접 읽어 복잡도를 점수화한다.
“모든 프롬프트에 대한 빠르고 오프라인인 하드/이지 판단 — 결정론적으로 점수화, 모델 호출 없음”이 핵심 가치다.

Apache 2.0 라이선스로 GitHub에 공개됐으며
219개의 스타와 14개의 포크, 186개의 커밋, 9개의 릴리스를 가지고 있다.

## 작동 방식

프롬프트 복잡도 점수는 0.0에서 1.0 사이로 산출된다.
단어 수, 제목 수, 목록 항목, 코드 블록 같은 구조적 특성을 기반으로 한다.
어휘 기반 신호(수학 표기, 추론 키워드, 제약 언어)는 선택적으로 활성화할 수 있지만
일반화 성능이 제한적이라는 이유로 기본값은 비활성화다.

임계값(기본값 0.5)을 기준으로 낮은 점수는 로컬 모델로, 높은 점수는 호스팅 모델로 보낸다.
다중 임계값 설정으로 세 개 이상의 모델 계층을 구성할 수 있다.

OpenAI 호환 API를 통한 투명한 프록시로 동작한다.
클라이언트는 `model=“auto”`로 요청을 보내고 Wayfinder가 실제 모델을 선택한다.
모든 응답 헤더에 `x-wayfinder-router-model`과 `x-wayfinder-router-score`가 포함된다.

## 주요 기능

`wayfinder-router calibrate`로 자체 트래픽 데이터를 사용해 임계값을 최적화할 수 있다.
`wayfinder-router route prompt.txt --explain`으로 각 특성별 기여도를 확인할 수 있다.

키는 설정 파일에 저장되지 않는다.
환경 변수에서 런타임에 읽거나 1Password, Keychain, Vault, AWS Secrets Manager 등을 통해 주입한다.

Claude Code 연동을 위한 `/v1/messages` 어댑터도 포함한다.
`ANTHROPIC_BASE_URL`을 Wayfinder 게이트웨이로 설정하면
Claude Code가 로컬/클라우드 모델로 자동 라우팅된다.

## 분석

### 결정론적 라우팅의 차별점

RouteLLM이나 NotDiamond 같은 기존 라우터는 라우팅 결정 자체에 모델 호출을 사용한다.
이는 라우팅 오버헤드가 생기고 네트워크 의존성이 발생하며 비용이 추가된다.
Wayfinder는 이 구조 자체를 거부한다.
구조 분석은 서브밀리초 단위로 완료되고 오프라인에서도 동작한다.

이 접근의 강점은 투명성이다.
라우팅 결정이 왜 내려졌는지 `--explain` 플래그로 정확히 확인할 수 있다.
LLM 기반 라우터는 “왜 이 모델로 보냈는가”를 설명하기 어렵지만
Wayfinder는 각 구조적 특성의 기여도를 수치로 보여준다.

### 구조적 복잡도 vs 의미론적 복잡도

프로젝트 벤치마크 문서는 솔직하게 한계를 인정한다.
구조적 지표가 없는 의미론적 난이도 프롬프트는 감지할 수 없다.
“이 문장을 번역해줘”와 “이 미적분 증명을 설명해줘”가 구조적으로 유사하다면
같은 점수를 받고 같은 모델로 라우팅된다.

이것은 근본적 제약이다.
어휘 기반 신호를 사용하면 이 문제를 부분적으로 완화할 수 있지만
저자가 기본값으로 비활성화한 이유가 있다.
도메인과 사용 패턴에 크게 의존하는 신호여서 사전 캘리브레이션 없이 적용하면 역효과가 난다.

kstenerud가 다른 중요한 문제를 제기했다.[^kstenerud]
프롬프트 A가 로컬 모델로 가고 프롬프트 B가 클라우드 모델로 간다면
두 모델이 동일한 대화 컨텍스트를 공유하지 못한다.
이것은 단발성 쿼리에서만 실용적이고
지속적 대화나 컨텍스트가 필요한 에이전트 워크플로에서는 사용이 제한된다.
Wayfinder는 이 문제를 아직 해결하지 않는다.

### dd8601fn의 README 관찰

dd8601fn이 README가 “Claude의 목소리”로 쓰여 있다고 지적한 것은[^dd8601fn]
흥미로운 메타적 관찰이다.
“the shape of”라는 표현이 반복적으로 등장하는 것이 Claude가 생성한 텍스트임을 드러낸다.
AI로 라우팅 결정을 내리지 않겠다는 도구가
AI로 생성된 문서로 자신을 설명하는 아이러니다.

## 비평

### 컨텍스트 분리 문제는 해결되지 않은 채 남아 있다

hmokiguess가 실무 경험에서 나온 중요한 문제를 제기했다.[^hmokiguess]
라우팅으로 모델이 전환될 때 새 모델이 이전 맥락을 모르기 때문에
회귀가 발생하거나 이전에 수행한 작업을 하지 않았다고 착각한다는 것이다.
AI 프로그래밍이 일상화된 후 이런 현상이 더 자주 발생한다고 보고한다.

이것은 Wayfinder의 가장 큰 미해결 문제다.
단발성 프롬프트에서는 충분히 동작하지만
대화형 에이전트 워크플로에서는 모델 전환이 맥락 손실을 의미한다.
캐시 미스 문제(kstenerud가 언급한 “hot하지 않은 캐시”)까지 고려하면
다중 모델 라우팅의 실질적 이점이 특정 시나리오에서 사라질 수 있다.

### 캘리브레이션의 닭과 달걀 문제

Wayfinder의 캘리브레이션 기능은 자체 레이블 트래픽이 있어야 동작한다.
그런데 로컬/클라우드 라우팅을 최적화하려면 먼저 두 모델의 성능 비교 데이터가 필요하고,
이를 얻으려면 이미 두 모델을 모두 돌려봐야 한다.
새로운 사용 사례에서는 캘리브레이션 없이 기본값으로 시작해야 하고,
기본값이 사용 패턴에 맞지 않으면 의도하지 않은 라우팅이 발생한다.
충분한 데이터가 쌓일 때까지의 초기 기간이 가장 취약하다.

## 인사이트

### OS 레벨 라우팅으로의 진화 가능성

josalhor는 LLM 라우팅을 애플리케이션 레벨이 아닌 OS 레벨에서 처리해야 한다고 제안했다.[^josalhor]
모바일 데이터 인프라처럼 OS가 각 애플리케이션의 LLM 요청을 적절한 추론 서비스로 라우팅하는 모델이다.
개별 앱마다 라우팅 로직을 구현하지 않아도 되고,
인터넷 연결 없이도 로컬 추론을 사용할 수 있다는 이점이 있다.

Wayfinder 같은 HTTP 프록시 방식은 현재 인프라에서 최소한의 변경으로 작동하는 실용적 해법이다.
하지만 nok22kon이 제안한 것처럼[^nok22kon] 하네스가 라우팅 플러그인을 직접 지원하는 방향으로 진화한다면
프록시 오버헤드를 제거하면서 동일한 기능을 달성할 수 있다.
이 도구의 현재 형태는 과도기적 해법의 성격이 강하다.

### 비용 투명성이 드러내는 숨겨진 가치

요청당 어느 모델로 갔는지, 복잡도 점수가 얼마였는지 헤더로 제공하는 것은
단순한 디버깅 편의를 넘어 비용 감사를 가능하게 한다.
어떤 유형의 프롬프트가 클라우드 모델로 “업그레이드”되는지 분석하면
로컬 모델 선택 기준을 개선하거나
비용이 집중되는 사용 패턴을 파악하는 데 쓸 수 있다.
이 투명성은 LLM 기반 라우터에서는 구조적으로 제공하기 어려운 특성이다.
