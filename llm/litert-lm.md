# LiteRT-LM — Google의 엣지 디바이스용 LLM 추론 프레임워크

<https://github.com/google-ai-edge/LiteRT-LM>

## 소개

LiteRT-LM은 Google이 오픈소스로 공개한 엣지 디바이스용 고성능 LLM 추론 프레임워크다.
Apache-2.0 라이선스로 제공되며, Android, iOS, 웹, 데스크톱, 라즈베리 파이 같은 IoT 기기까지
폭넓은 플랫폼을 지원한다. GPU와 NPU 하드웨어 가속을 통해 온디바이스(on-device)에서
생성형 AI 추론을 수행하며, 이미 Chrome, Chromebook Plus, Pixel Watch 같은 Google 제품에서
프로덕션 환경으로 사용 중이다.

지원 모델은 Gemma, Llama, Phi-4, Qwen 등 주요 공개 모델을 포함하며,
최신 버전 v0.10.2부터 Gemma 4를 지원한다.
멀티모달(vision, audio 입력)과 Function Calling(에이전트 워크플로우)을 지원하는 점이 특징이다.
언어 API는 Kotlin(Android), Python, C++이 안정화되어 있으며, Swift는 개발 중이다.

## 사용법

빠른 시작은 `uv tool install litert-lm` 이후 CLI 명령으로 Hugging Face 저장소에서
모델을 직접 내려받아 실행하는 방식이다. 코드 한 줄 없이 터미널에서 LLM 추론을
엣지 디바이스에서 실행할 수 있다. 프로그래매틱 API는 라이브러리 임포트 방식으로
Android 앱, Python 스크립트, C++ 네이티브 코드에 통합 가능하다.

v0.7.0에서 NPU 가속이 추가되었고, v0.8.0에서 데스크톱 GPU 지원과 멀티모달리티가,
v0.9.0에서 Function Calling이 강화되었다. Hugging Face 통합을 통해
`litert-community` 조직에서 모델을 쉽게 받아볼 수 있다.

## 분석

### Google AI Edge 생태계에서의 포지셔닝

LiteRT-LM은 Google의 엣지 AI 전략을 이루는 레이어 중 하나다.
Google AI Edge Gallery 앱, LiteRT 런타임, Google AI Edge SDK와 함께
온디바이스 AI 스택을 형성한다. LiteRT-LM은 그 중 LLM 추론에 특화된 계층이다.

Google Play와 App Store에서 내려받을 수 있는 AI Edge Gallery 앱은
이 프레임워크의 기능을 손쉽게 체험할 수 있는 쇼케이스 역할을 한다.
단순한 데모가 아니라 실제 기기에서 모델을 실행하는 레퍼런스 앱이므로,
개발자들이 통합 방식을 직접 확인할 수 있다.

### Hugging Face 통합과 모델 배포 전략

CLI에서 `--from-huggingface-repo` 플래그로 직접 모델을 받아 실행하는 설계는
Hugging Face를 모델 배포 인프라로 인정하는 명시적 선택이다.
Google이 자체 모델 저장소를 강요하지 않고 생태계 표준을 수용한 점이 인상적이다.

`litert-community` Hugging Face 조직은 LiteRT-LM 형식으로 변환된 모델들을 호스팅한다.
`.litertlm` 확장자의 모델 파일 형식은 엣지 추론에 최적화된 독자적 포맷으로 보이며,
기존 GGUF(llama.cpp)나 ONNX 같은 형식과 경쟁 관계에 놓인다.

### Function Calling과 에이전트 워크플로우

온디바이스 Function Calling 지원은 LiteRT-LM을 단순 추론 프레임워크를 넘어
에이전트 런타임으로 발전시키는 핵심 기능이다.
클라우드 없이 기기에서 에이전트가 도구를 호출하고 결과를 처리할 수 있다면,
개인정보 보호와 오프라인 작동이 필요한 시나리오에서 강력한 대안이 된다.
Google AI Edge Gallery의 Function Calling 데모는 이 방향의 가능성을 보여준다.

## 비평

### 강점

프로덕션 검증이 강점이다.
Chrome, Chromebook Plus, Pixel Watch에서 실제 사용자에게 서비스하는 것은
기술적 신뢰성을 입증하는 가장 강한 증거다.
실험적 프레임워크가 아니라 Google이 자사 제품에 사용하는 라이브러리라는 사실이
채택 결정의 리스크를 낮춘다.

CLI 경험이 매우 낮은 진입 장벽을 만든다.
`uv tool install litert-lm` 이후 단일 명령으로 Gemma 모델을 실행하는 경험은
엣지 AI 실험을 “코드 없이” 가능하게 한다.
이는 특히 ML 배경이 없는 iOS/Android 앱 개발자들에게 유리하다.

### 약점

Swift 지원이 개발 중이라는 점은 iOS 생태계 통합에 제약이다.
Kotlin API가 안정화된 상황에서 Swift가 뒤처지면,
Android 편향적 엣지 AI 프레임워크라는 인식을 줄 수 있다.
iOS에서 에지 AI 수요가 높은 시장에서 이 격차는 채택을 저해한다.

GeekNews 댓글에서 메모리 부족과 Mac 성능 문제가 언급되었다.
“돌리고 싶어도 메모리가 없다”는 반응은 엣지 디바이스용 프레임워크임에도
실제 사용 가능한 기기 범위가 제한적임을 시사한다. E2B 모델(2B 파라미터)도
일부 기기에서는 실행 어렵다면, “엣지”의 정의가 좁아진다.

### .litertlm 포맷의 생태계 잠금

`.litertlm` 형식은 LiteRT-LM 생태계에 종속되는 포맷이다.
GGUF나 ONNX가 여러 런타임에서 사용 가능한 것과 달리,
`.litertlm` 파일은 LiteRT-LM 없이는 사용하기 어렵다.
모델 변환 과정이 있겠지만, 이 과정의 마찰이 다른 런타임과의 이동을 어렵게 만든다.
오픈소스 라이선스와 독점적 파일 포맷의 조합은 흔한 생태계 전략이다.

## 인사이트

### 클라우드 AI의 “역중앙화”와 엣지의 부상

인터넷의 역사는 중앙화와 분산의 반복이다.
메인프레임에서 PC로, 클라이언트-서버에서 웹으로, 웹에서 클라우드로.
AI가 클라우드에서 시작한 것은 자연스럽다.
GPU 클러스터와 대규모 데이터가 필요한 훈련과 추론은 중앙 인프라를 요구했다.

LiteRT-LM이 대표하는 온디바이스 추론 트렌드는 이 흐름의 다음 역전을 보여준다.
“AI의 역중앙화”다. 모델이 기기에 내려오면 세 가지가 달라진다.
첫째, 개인정보가 서버로 가지 않는다.
둘째, 인터넷 없이 작동한다.
셋째, 추론 비용이 사용자의 기기 비용에 포함된다.

이 변화는 단순한 기술 분산이 아니라 비즈니스 모델의 변화다.
API 호출당 과금 모델은 온디바이스 추론에서 작동하지 않는다.
Google이 LiteRT-LM을 오픈소스로 공개한 이유는 직접적인 수익보다
Android/iOS 생태계에서의 AI 표준 플랫폼 포지션을 선점하려는 것이다.

### Gemma 4와 엣지 AI의 품질 임계점

Gemma 4 E2B(2B 파라미터) 모델이 LiteRT-LM의 최신 하이라이트다.
2B 파라미터 모델이 엣지에서 의미 있는 결과를 낼 수 있는가는
온디바이스 AI의 실용성 전체를 좌우하는 질문이다.

GPT-4급 모델과 비교하면 2B는 작다. 그러나 “충분히 좋다(good enough)”의 기준은
작업에 따라 다르다. 간단한 Q&A, 문서 요약, 코드 완성의 일부 작업에서는
2B 모델이 사용자 경험을 만족시킬 수 있다.
Gemma 4 E2B가 이 임계점을 어디에서 통과하는지가
온디바이스 AI가 “실험”에서 “프로덕션”으로 이동하는 속도를 결정한다.

역사적 유추는 있다. 초기 스마트폰의 카메라는 DSLR과 비교할 수 없었지만,
“충분히 좋은” 카메라가 되었을 때 스마트폰이 카메라 시장을 재편했다.
온디바이스 LLM이 대부분의 일상적 AI 작업에서 “충분히 좋아지는” 시점이
클라우드 AI 의존도의 구조적 전환점이 될 것이다.

### 엣지 AI와 프라이버시의 새로운 계약

클라우드 AI 서비스의 근본적 긴장은 개인화와 프라이버시의 충돌이다.
AI가 나를 더 잘 이해할수록 내 데이터가 더 많이 서버로 간다.
이 긴장은 개인정보 보호법의 강화와 사용자 신뢰 문제로 구체화되어 왔다.

온디바이스 추론은 이 긴장을 기술적으로 해소한다.
데이터가 기기를 떠나지 않으면서 AI가 사용자를 학습할 수 있다.
의료, 법률, 금융처럼 민감한 도메인에서 온디바이스 AI의 수요는
성능보다 프라이버시 보장에서 먼저 발생할 것이다.

LiteRT-LM이 Pixel Watch에서 실행된다는 사실은 상징적이다.
손목에 차는 기기는 수면 패턴, 심박수, 이동 경로를 알고 있다.
이 데이터로 훈련된 개인화 모델이 기기에서만 실행된다면,
클라우드 중심의 헬스케어 AI와는 다른 신뢰 구조가 만들어진다.
엣지 AI의 킬러 앱은 성능이 아니라 프라이버시가 요구되는 도메인에서 먼저 나올 가능성이 높다.
