# Laguna S 2.1 — Poolside의 장기 코딩 에이전트 모델

원문: [Introducing Laguna S 2.1 — Poolside](https://poolside.ai/blog/introducing-laguna-s-2-1)

HN 토론: <https://news.ycombinator.com/item?id=48995261> (362점, 68개 댓글)

## 요약

Poolside가 Laguna S 2.1을 공개했다.
118B 파라미터의 MoE(Mixture-of-Experts) 모델로, 토큰당 8B 파라미터가 활성화된다.
1M 토큰 컨텍스트 창을 지원하고, 훈련 시작부터 출시까지 9주 미만이 걸렸다.

이 모델은 장기 코딩 작업에 초점을 맞춘다.
이전 모델들이 성공을 조기에 선언하는 경향이 있었다면,
Laguna S 2.1은 실제로 완료될 때까지 계속 작업하는 “끈질김”을 강조한다.

주요 벤치마크 성능은 다음과 같다.

| 벤치마크 | 점수 |
| -------- | ---- |
| Terminal-Bench 2.1 (thinking max) | 70.2% |
| SWE-Bench Multilingual | 78.5% |
| SWE-Bench Pro | 59.4% |
| DeepSWE v1.1 | 40.4% |
| Toolathlon Verified | 49.7% |

thinking 모드 활성화 시 성능 향상이 크다.
Terminal-Bench 2.1 기준 60.4%에서 70.2%로, DeepSWE는 16.5%에서 40.4%로 올랐다.

실제 코딩 사례도 공개됐다.
JavaScript로 HTML/CSS 렌더링 엔진을 처음부터 구현했고,
Poolside 자체 에이전트 하네스를 최적화해 5.2% 속도 향상과 71% 메모리 할당 감소를 달성했다.
Erdős 문제 #397에 대한 독립적인 증명도 내놨다.

배포 옵션은 오픈 소스와 호스팅 두 가지다.
Hugging Face에서 BF16, FP8, INT4, NVFP4 가중치를 제공하고,
GGUF와 MLX 변환도 지원한다.
vLLM, SGLang, Ollama와 호환된다.
호스팅은 OpenRouter($0.10 입력/$0.20 출력/1M 토큰), Vercel AI Gateway, Baseten을 통해 이용할 수 있다.

## 분석

### 코딩 모델 평가가 단발 답변에서 장기 과제 완주율로 이동한다

Laguna S 2.1의 마케팅에서 가장 강조하는 것은 “끈질김”이다.
이전 모델들이 장기 작업에서 중간에 포기하거나 조기에 성공을 선언하는 문제를 언급하면서,
이 모델은 실제로 완료될 때까지 계속한다고 주장한다.

이 주장이 중요한 이유는 코딩 에이전트의 실제 사용 맥락과 관련있다.
단일 파일 편집이나 함수 수정은 대부분의 모델이 잘 처리한다.
그러나 긴 리팩터링, 테스트 작성, 다단계 디버깅처럼 수십 번의 도구 호출이 필요한 작업에서는
모델이 얼마나 오래 일관된 방향을 유지하느냐가 품질을 결정한다.

Terminal-Bench 2.1이 이 능력을 측정하려고 설계된 벤치마크다.
터미널 환경에서 실제 소프트웨어 엔지니어링 과제를 주고,
완주율과 품질을 동시에 측정한다.

### thinking 모드의 효과가 모델 가치 평가를 복잡하게 만든다

thinking 모드 활성화 시 성능이 크게 오른다.
DeepSWE 기준 16.5%에서 40.4%로, 2.4배 향상이다.
이것은 긍정적 신호이지만 동시에 복잡한 함의를 갖는다.

thinking 모드는 토큰을 더 많이 소비하고 지연시간이 길어진다.
성능 수치가 thinking 활성화 기준으로 발표됐다면,
실제 운영에서 비용과 지연시간이 수치보다 훨씬 클 수 있다.
벤치마크 비교에서 thinking 모드 여부, thinking 예산이 통제됐는지 확인해야 한다.

실제로 vLLM으로 로컬에서 실행하는 사용자들 사이에서 thinking이 제대로 동작하지 않는다는 보고가 초기에 많았다.
원인은 `generation_config.json`의 `max_new_tokens` 기본값이 32k로 설정돼 있어 thinking 토큰을 잘라버리는 문제였다.[^benjiro29]
이 설정을 늘리고 thinking 플래그를 명시적으로 켜면 출력 품질이 크게 달라진다는 경험이 공유됐다.

### post-training이 실제 차별화 요인이 됐다

모델의 발전이 pre-training 데이터 확장이 아니라 post-training 개선에서 나왔다는 점이 주목할 만하다.
더 긴 롤아웃 예산, 더 높은 토큰 할당, 멀티 하네스 훈련, FP8 정밀도 RL 훈련이 핵심 변화다.

이 패턴은 더 넓은 트렌드와 일치한다.
프런티어 모델들이 사전 훈련 규모 경쟁보다 후처리 방식 개선으로 차별화를 시도하고 있다.
같은 기반 모델에서 post-training 방식에 따라 실무 성능 차이가 크게 날 수 있다는 것이,
오픈 웨이트 모델 생태계에 의미하는 바도 크다.

## 비평

### 벤치마크 결과의 신뢰도에 의문을 품을 근거가 있다

한 가지 주목할 한계가 직접 명시됐다.
“하네스 과적합(harness overfitting)” — 서드파티 툴 스키마의 작은 변화에도 어려움을 겪는다.
이것은 모델이 평가에 사용된 특정 하네스 환경에 맞게 훈련됐을 가능성을 시사한다.

SWE-Bench와 Terminal-Bench 점수가 높아도 실제 다양한 코딩 환경에서 같은 성능이 재현되지 않을 수 있다.
특히 사내 도구 스키마, 독자적인 프레임워크, 비표준 API 패턴에서 얼마나 강건한지가
벤치마크 수치만으로는 판단하기 어렵다.

실제 사용 경험도 엇갈린다.
C 코드베이스에서 다른 모델이 놓친 문제를 찾아냈지만, `memfd_create()`/`mmap`을 IPC로 사용한다는 잘못된 초기 관찰도 내놓았다는 보고가 있다.[^Lwerewolf]
반면 Mozilla AI 프로젝트에서 실제로 쓸 만한 PR을 한 번에 산출했다는 긍정적인 사례도 있다.[^river_otter]
128B 모델이 1.6T 파라미터의 DeepSeek V4를 코딩 벤치마크 대부분에서 앞선다는 점은 인상적이고,
Poolside가 자사 모델을 훨씬 큰 모델들과도 투명하게 비교하는 방식은 긍정적으로 평가받는다.[^aubanel]

### 118B MoE 모델의 실제 운영 비용

118B 파라미터 MoE 모델을 직접 실행하려면 상당한 GPU 메모리가 필요하다.
FP8, INT4 양자화를 지원하지만, 이 크기의 모델을 단일 GPU에서 운영하는 것은
대부분의 팀에서 실용적이지 않다.

호스팅 API($0.10/$0.20 per 1M tokens)를 쓰면 접근은 쉽지만,
thinking 모드를 켜고 장기 작업을 실행하면 토큰 비용이 빠르게 누적된다.
벤치마크 점수와 실제 운영 비용 사이의 트레이드오프를 명확히 하지 않는 것은 마케팅 맹점이다.

## 인사이트

### 코딩 에이전트 평가의 표준이 장기 완주율 중심으로 재편될 것이다

단발 코드 생성 벤치마크는 이미 포화 상태다.
HumanEval, MBPP 같은 고전적 벤치마크에서 대부분의 프런티어 모델이 90% 이상을 기록한다.
이 수치가 더 이상 실무 능력과 상관관계가 없어지면서,
새로운 평가 방식이 필요해졌다.

Terminal-Bench, SWE-Bench Pro 같은 장기 과제 벤치마크가 그 방향을 채우고 있다.
이 벤치마크들은 모델이 수십 단계의 작업을 얼마나 일관되게 완주하는지,
실패했을 때 얼마나 잘 복구하는지를 측정한다.

앞으로 코딩 에이전트 선택의 기준은 단발 답변 품질보다 완주율, 평균 작업 길이, 실패 복구율이 될 가능성이 높다.
이 기준에서 Laguna S 2.1은 좋은 출발점이지만, 벤치마크 결과가 실제 다양한 환경에서 재현되는지는 별도 검증이 필요하다.

### 오픈 웨이트 코딩 모델이 기업 AI 도입의 선택지를 바꾼다

Laguna S 2.1이 Hugging Face에서 가중치를 공개한다는 것은 단순한 오픈소스 정책이 아니다.
118B MoE 모델을 직접 실행할 수 있는 팀이라면, 코드가 외부로 나가지 않는 환경에서
이 수준의 코딩 에이전트를 운영할 수 있다.

규제 산업, 금융, 방산처럼 코드를 외부 API로 보낼 수 없는 환경에서
이 선택지는 실질적인 의미가 있다.
오픈 웨이트 코딩 모델의 성능이 폐쇄형 API와 비교 가능한 수준에 도달할수록,
기업 AI 도입에서 온프레미스 또는 VPC 배포 옵션의 설득력이 높아진다.

MoE 구조는 대역폭 제약 환경에서도 실용적으로 동작한다는 점이 주목받는다.
Strix Halo나 DGX Spark 같은 소비자·준전문가용 하드웨어에서도 활성 파라미터 수(8B)가 작아
추론 속도가 밀집 모델 대비 유리하다.[^SwellJoe]
한편 64GB 메모리만 가진 환경을 위한 저비트 양자화 요청도 나왔고,
커뮤니티에서 곧 GGUF 변환본이 공개됐다.[^mft_]
오픈 웨이트 모델의 성능이 이 수준에 도달하면 커뮤니티 주도의 양자화·파인튜닝 생태계가 빠르게 형성된다는 사실이 다시 한번 확인됐다.[^tosh]
