# DGX Spark 한 대에 Qwen3 모델 두 개 올리기

원문: <https://www.devashish.me/p/two-qwen3-models-on-one-dgx-spark>

## 요약

Devashish Meena가 작성한 이 글은 119.67 GiB 유니파이드 메모리를 가진 NVIDIA DGX Spark 단일 기기에서
Qwen3-Next-80B-Instruct-FP8과 Qwen3-4B-Instruct-2507 두 모델을
vLLM으로 동시에 실행(coresident)하면서 겪은 메모리 배치 수학을 실전 기록으로 남긴다.
두 모델은 LiteLLM 프록시를 통해 분산 에이전트 플릿(Hermes 기반)에 서비스된다.

핵심 발견은 `gpu_memory_utilization` 파라미터의 오해에서 비롯됐다.
이 값은 남은 여유 메모리의 비율이 아니라 **전체** GPU 메모리에 대한 비율이다.
프로세스 시작 시점의 전체 카드 용량 대비 할당 목표를 계산하므로,
다른 프로세스가 이미 메모리를 사용 중이더라도 해당 사실이 반영되지 않는다.

세 번의 시도 끝에 작동하는 최종 설정은 다음과 같다.
80B 모델: `gpu_memory_utilization: 0.80`, `max_model_len: 32768`, `max_num_seqs: 2`.
4B 모델: `gpu_memory_utilization: 0.10`, `max_model_len: 16384`, `max_num_seqs: 8`.
실제 거주(residency) 측정값은 80B가 87.8 GiB(목표 95.7 GiB보다 8 GiB 낮음),
4B가 13.8 GiB(목표 12.0 GiB보다 1.8 GiB 높음), CUDA 오버헤드 약 5 GiB로 총 약 101.6 GiB를 점유한다.

세 가지 중요한 관찰도 제시된다.
첫째, 80B의 실제 거주량이 할당 목표보다 8 GiB 낮은 쿠션이 4B의 재시작 변동성을 흡수한다.
이 쿠션이 사라지는 0.85 설정에서 공존이 실패한다.
둘째, 4B는 목표 12 GiB 대비 13.8 GiB를 점유하는데, CUDA 프레임워크 오버헤드가 모델 크기와 무관하게 고정적이다.
셋째, Qwen3-Next(Mamba 아키텍처 혼합)에서 `max_model_len`을 절반으로 줄여도 KV 요구량이 비례해 줄지 않는다.
Mamba 상태 정렬이 전통적 어텐션 메커니즘보다 KV 풀 수요를 더 많이 결정하기 때문이다.

## 분석

### 메모리 배치는 선언적 목표가 아니라 실측에 기반해야 한다

저자의 핵심 교훈은 vLLM의 `gpu_memory_utilization`이 의도 선언(intent declaration)이지
실제 배치 보장이 아니라는 것이다.
이 파라미터는 “나는 총 메모리의 X%까지 사용할 것이다”를 말하지만,
실제 모델 로딩 후 메모리 점유는 여러 요인에 의해 달라진다.
KV 캐시 사전 할당, CUDA 컨텍스트 오버헤드, 모델 아키텍처별 특성, 이전 실패한 시도의 컨텍스트 잔류.

저자가 권고하는 절차는 이 불확실성에 대한 현실적 대응이다.
큰 모델을 먼저 로드해 안정화하고, `nvidia-smi`로 실제 점유량을 측정하며,
측정값 기준으로 작은 모델의 할당을 계산하고, 두 모델이 깨끗하게 두 번 재시작되는지 확인한다.
선언한 목표가 아니라 관찰된 현실을 기반으로 설계하는 것이다.

### Mamba 아키텍처의 메모리 특성은 전통적 트랜스포머와 다르다

Qwen3-Next가 Mamba 상태 머신과 트랜스포머 어텐션을 혼합한 아키텍처라는 점이
`max_model_len` 감소 → KV 메모리 비례 감소라는 직관을 무너뜨렸다.
전통적 트랜스포머에서 KV 캐시 크기는 시퀀스 길이에 선형적으로 비례하므로,
`max_model_len`을 절반으로 줄이면 KV 메모리도 절반 가까이 줄어든다.
그러나 Mamba 구성요소의 상태 크기는 시퀀스 길이가 아니라 모델 크기와 상태 차원에 의존한다.

이 비선형성은 하이브리드 아키텍처 모델을 운영할 때 전통적인 메모리 계산식이
더 이상 신뢰할 수 없음을 의미한다.
아키텍처 논문이나 모델 카드가 메모리 특성을 명시하지 않는다면,
실측 없이는 예측이 어렵다.
신형 아키텍처를 프로덕션에 배포할 때 메모리 프로파일링을 더 일찍 수행해야 한다는 교훈이다.

### 두 모델의 공존은 경쟁이지 협상이 아니다

저자의 표현 — “공존 프로세스들은 자원 배분을 협상하지 않는다. 그들은 경쟁한다” — 은
vLLM의 메모리 관리 모델을 정확히 요약한다.
각 vLLM 인스턴스는 자신의 `gpu_memory_utilization` 목표를 독립적으로 달성하려 한다.
다른 인스턴스의 존재를 인식하지 않으므로, 두 인스턴스의 목표 합이 100%를 초과하면 충돌이 발생한다.

이 경쟁적 구조에서 쿠션(cushion)의 개념이 중요해진다.
80B 모델의 실제 점유량이 목표보다 8 GiB 낮은 것은 우연이 아니라 vLLM이 KV 캐시를 지연 할당하거나
사전 예약량이 실제 사용량보다 크기 때문이다.
이 쿠션이 4B 모델의 재시작 시 변동하는 메모리 요구량을 흡수한다.
단순히 할당 목표의 합을 100% 이하로 유지하는 것만으로는 불충분하며,
실측 쿠션을 고려한 여유가 필요하다.

## 비평

### 이 설정이 프로덕션 에이전트 플릿에 적합한지는 검증되지 않았다

저자는 이 설정을 로컬 LAN의 에이전트 플릿에 사용한다고 설명하지만,
실제 프로덕션 워크로드에서의 안정성은 이 글에서 직접 다루지 않는다.
80B 모델의 `max_num_seqs: 2`와 4B 모델의 `max_num_seqs: 8`은
동시 요청 처리 용량에 직접적인 제약이다.
에이전트 플릿이 동시에 많은 요청을 보낼 때 큐 대기 시간이 어떻게 증가하는지,
그리고 80B 모델의 2개 동시 시퀀스 제한이 실제 사용 패턴에 충분한지는 다루어지지 않는다.

DGX Spark의 119.67 GiB 메모리는 인상적이지만, 80B FP8 모델과 4B 모델의 동시 서비스는
각 요청의 지연 시간과 처리량 면에서 단일 모델 전용 설정과 어떻게 비교되는지도 불명확하다.
두 모델이 메모리를 나눠 갖는 것이 각각의 추론 성능에 미치는 영향을 측정한 결과가 없다.

### “Thinking” vs. “Instruct” 변형 선택 문제는 단순화되었다

저자는 Qwen3-Next-80B-Thinking이 도구 호출을 지원하지 않아 Instruct 변형으로 교체했다고 설명한다.
`enable_thinking: false`가 no-op이었다는 것은 모델 동작을 충분히 이해하지 않고 배포했음을 시사한다.
Thinking 모드와 Instruct 모드는 단순히 설정 플래그의 차이가 아니라
근본적으로 다른 동작 패턴을 갖는다.

Thinking 모델을 에이전트 플릿의 도구 호출에 사용하려 한 것은
모델의 설계 의도를 오해한 것이다.
도구 호출은 결정론적이고 구조화된 출력을 필요로 하는데,
내부적으로 추론 흔적(`<think>` 태그)을 생성하는 모델에서 그것을 비활성화하는 것이
신뢰할 수 있는 방법이 아님은 사전에 문서를 통해 파악 가능했다.

### 글의 가치는 실패 과정의 기록에 있지만, 그 기록이 완전하지 않다

이 글의 주요 가치는 실패한 시도들과 그 이유를 솔직하게 기록한 것이다.
그러나 각 시도에서 vLLM이 정확히 어떤 오류 메시지를 출력했는지,
KV 캐시 초기화 충돌이 어떤 로그로 나타났는지, 메모리 부족이 어떤 증상으로 나타났는지가
더 구체적으로 기록되었다면 독자에게 더 직접적으로 유용했을 것이다.
설정 파라미터와 결과만 보여주는 것은 인과관계를 충분히 설명하지 못한다.

## 인사이트

### 로컬 추론 인프라의 복잡성은 클라우드 ML 인프라의 복잡성과 수렴하고 있다

DGX Spark 한 대에서 두 모델을 공존시키는 작업이 이 정도의 메모리 배치 수학,
CUDA 컨텍스트 관리, 아키텍처별 메모리 특성 이해를 요구한다는 것은
로컬 추론 인프라가 성숙 단계에 진입하고 있음을 보여준다.
AWS나 GCP에서 대형 모델을 서비스하는 것이 복잡하듯이,
개인 또는 소규모 팀 수준의 로컬 추론도 이제 진지한 인프라 엔지니어링을 요구한다.

이 수렴은 로컬 AI 인프라를 “그냥 올리면 되는 것”으로 단순화하려는 도구들의 한계를 드러낸다.
vLLM, LiteLLM, Ollama 같은 도구들이 접근성을 높였지만,
멀티 모델 공존, 메모리 최적화, 아키텍처별 특성 대응은 여전히 수동 조율이 필요하다.
클라우드 ML 인프라에서 쌓인 지식이 로컬 추론 세팅에서도 점점 더 필요해지고 있다.

### `gpu_memory_utilization`의 오해는 선언적 설정 인터페이스의 구조적 함정이다

vLLM의 `gpu_memory_utilization` 파라미터가 “전체 메모리 대비 비율”임에도
“사용 가능한 메모리 대비 비율”로 오해되는 것은 API 설계의 의사소통 실패다.
파라미터 이름 자체가 “GPU 메모리 활용률”로 두 해석 모두 자연스럽게 연결된다.
이 오해는 저자 개인의 실수가 아니라 같은 오해를 반복하는 수많은 사용자의 경험에서 확인된다.

선언적 설정 인터페이스는 복잡한 시스템을 단순하게 조작할 수 있게 해주지만,
그 단순성 뒤에 숨은 복잡한 동작을 숨기는 부작용이 있다.
`gpu_memory_utilization: 0.80`이 실제로 의미하는 것을 이해하려면
vLLM의 내부 KV 캐시 할당 전략, CUDA 메모리 모델, 프로세스 시작 시점의 측정 방식을 알아야 한다.
이 지식을 문서화하고 오해를 방지하는 것은 도구 설계자의 책임이지만,
그 책임이 충분히 이행되지 않으면 사용자가 반복적으로 같은 실패를 경험한다.

### 작은 모델은 라우팅 계층에서 가장 가치 있다

저자가 4B 모델을 “sources” 알리어스로 검색 쿼리에, 80B를 “writer”와 “reviewer”로
복잡한 작업에 배분하는 LiteLLM 설정은 실용적인 비용 최적화다.
동일한 프록시 엔드포인트에서 작업 유형에 따라 적절한 모델 크기로 라우팅하는 것은
추론 비용을 낮추면서 품질을 유지하는 검증된 패턴이다.

4B와 80B의 공존이 이 라우팅 전략의 인프라 기반이라면,
두 모델의 메모리 배치 문제는 단순한 기술적 과제가 아니라
비용 효율적인 에이전트 인프라의 기초 문제다.
이 관점에서 저자가 공유한 실측 데이터 — 실제 거주량, 쿠션 크기, 재시작 변동성 — 는
유사한 설정을 시도하는 사람들에게 직접적으로 재사용 가능한 기준점이 된다.
