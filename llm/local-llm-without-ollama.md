# 로컬 LLM 생태계에는 Ollama가 필요하지 않다

원문: <https://sleepingrobots.com/dreams/stop-using-ollama/>

## 요약

저자 Zetaphor는 Ollama가 처음 등장했을 때 llama.cpp 위에 얹힌 유용한 래퍼였지만,
이후 일련의 의심스러운 결정들을 거치며 오픈소스 원칙에서 멀어졌다고 주장한다.
핵심 비판은 크게 다섯 가지다.

첫째, Ollama는 1년 이상 README에서 llama.cpp를 언급하지 않았고 MIT 라이선스 고지도
누락했다. 커뮤니티가 400일 넘게 라이선스 준수를 요청했음에도 응답이 없었다.
나중에 추가된 속성 표기는 최소한의 것에 그쳤다.

둘째, 2025년 중반 Ollama는 llama.cpp에서 독자 백엔드로 분기했다. 그 결과 동일 하드웨어에서
llama.cpp가 Ollama보다 1.8배 빠른 처리 속도를 기록했고(161 vs 89 토큰/초),
llama.cpp가 이미 해결한 버그들이 재현되기 시작했다.

셋째, 모델 명칭을 오도적으로 표기했다. DeepSeek-R1의 소형 증류 버전을 단순히
"DeepSeek-R1"로 표기하여 사용자가 671B 모델이 아닌 8B 모델을 받는다는 사실을 숨겼다.

넷째, 2025년 7월 GUI 앱을 라이선스 없이 비공개 저장소에서 개발해 출시했다. 몇 달 후에야
코드가 공개되었다. 다섯째, 개인정보 보호를 표방하던 도구가 2025년 말 클라우드 모델을 도입하며
제3자 공급자로 프롬프트를 라우팅하기 시작했다. CVE-2025-51471로 인증 토큰 유출 취약성까지
발견되었다.

저자가 제안하는 대안으로는 llama.cpp(OpenAI 호환 API 포함), llamafile(단일 실행파일),
LM Studio(GUI), ramalama(컨테이너 기반), LiteLLM(통합 프록시)이 있다.

## 분석

### 오픈소스 래퍼의 전형적 수명 주기

이 글이 묘사하는 Ollama의 궤적은 오픈소스 생태계에서 반복되는 패턴과 일치한다.
핵심 기술(llama.cpp)이 존재하지만 사용자 경험이 복잡할 때, 래퍼 도구가 등장해 인기를
얻는다. 이후 벤처 투자가 유입되면 투자자 기대에 맞게 제품이 변형되기 시작한다.

Ollama가 VC 투자를 받은 시점과 독자 백엔드 전환, GUI 앱의 비공개 개발, 클라우드 서비스
도입이 시간적으로 겹친다. 이는 기술적 필요에 의한 변화가 아니라 비즈니스 모델 변화를
반영한다고 볼 수 있다.

### 성능 데이터의 신뢰성

저자가 제시한 1.8배 성능 차이는 단일 벤치마크에서 나온 수치다. Ollama와 llama.cpp를
공정하게 비교하려면 동일 하드웨어, 동일 모델, 동일 컨텍스트 길이, 동일 양자화 방식을
통제해야 한다. 저자는 이러한 비교 조건을 상세히 기술하지 않았다. 다만 llama.cpp 직접
실행 대비 데몬 아키텍처 오버헤드가 존재한다는 주장은 구조적으로 타당하다.

### 오도적 모델 명칭 문제

DeepSeek-R1 증류 모델을 원본 모델명으로 표기하는 문제는 사용자 신뢰의 핵심을 건드린다.
8B 모델과 671B 모델의 성능 차이는 본질적이다. 이는 단순한 명명 관행이 아니라 사용자가
기대하는 성능과 실제 성능 사이의 근본적 괴리를 만든다.

## 비평

### 강점

저자가 제기하는 라이선스 미준수 문제는 구체적 사실에 기반한다. 400일간의 응답 거부라는
수치는 검증 가능하고, MIT 라이선스 고지 누락은 법적으로도 명확한 위반이다. 모델 명칭
오도 문제 역시 스크린샷 증거가 있어 반박하기 어렵다.

대안 도구 목록이 구체적이라는 점도 강점이다. "쓰지 마라"고만 하는 것이 아니라
llamafile, LM Studio, ramalama 같은 실용적 대안을 제시한다.

### 약점

글의 톤이 감정적이다. "Friends Don't Let Friends Use Ollama"라는 제목부터 설득보다
비난에 가깝다. 이런 톤은 Ollama 사용자들이 방어적으로 반응하게 만들어 실질적
전환을 어렵게 한다.

Ollama가 해결한 실제 문제를 충분히 인정하지 않는다. "한 줄 명령으로 모델 실행"은
기술적으로 낮은 장벽을 만들어 LLM 생태계를 대중화하는 데 기여했다. 이 공헌을
무시하면 주장의 균형이 무너진다.

1.8배 성능 차이 수치의 출처와 측정 조건이 불분명하다. 독립적 검증 없이 단일 벤치마크를
근거로 삼는 것은 논리적으로 취약하다.

## 인사이트

### 편의성은 락인의 트로이 목마다

Ollama의 성공 공식은 간단했다. "docker pull" 수준의 단순함으로 LLM을 실행하게 만드는 것.
이 편의성이 사용자를 끌어들였고, 모델이 해시된 파일명으로 저장되는 구조가 자연스럽게 락인을
만들었다. 사용자는 편의성에 매력을 느끼는 시점에 이미 이탈 비용을 지불하기 시작한다.

이 패턴은 Heroku, Docker Desktop, GitHub Actions에서도 반복된다. 오픈소스 기반 위에
편리한 레이어를 얹고, 사용자가 의존하게 된 후 점진적으로 경계를 좁히는 전략이다.
로컬 LLM 생태계에서는 이 전략이 더 빠르게 작동한다. 모델 파일이 크기 때문에 한번 저장된
위치를 바꾸는 데 드는 비용(시간, 대역폭)이 실질적 장벽이 된다.

사용자 입장에서 교훈은 도구 선택 시 "오늘의 편의성"만이 아니라 "1년 후의 이탈 비용"을
함께 계산해야 한다는 것이다.

### VC 자본과 오픈소스의 구조적 긴장

오픈소스 프로젝트가 벤처 투자를 받는 순간 이해관계자 구조가 근본적으로 바뀐다.
커뮤니티 → 투자자 순서로 우선순위가 재편된다. 이것이 나쁜 의도의 문제가 아니라 구조의 문제다.
투자자는 3~7년 내 수익 실현을 기대하며, 오픈소스 기여자들의 장기적 생태계 건강에는
직접적 이해관계가 없다.

Ollama의 클라우드 전환은 이 구조적 압력의 자연스러운 결과다. llama.cpp 래퍼만으로는
SaaS 비즈니스 모델을 만들 수 없다. 클라우드 API를 통한 프롬프트 라우팅이 수익화의
가장 직접적 경로다. 결국 "로컬 AI의 프라이버시"라는 초기 가치 제안과 정반대 방향으로
이동한다.

이 패턴을 인식하면 다음 질문이 명확해진다. 어떤 오픈소스 도구든 VC 투자를 받는 순간,
"이 도구의 장기적 방향성이 내 사용 목적과 계속 일치할 것인가"를 물어야 한다.

### llama.cpp 기여자 모델의 지속가능성

저자가 지목한 문제 중 가장 근본적인 것은 속성(attribution) 거부다. Georgi Gerganov와
450명 이상의 기여자가 만든 llama.cpp 없이는 Ollama 자체가 존재하지 않는다. 이 사실을
문서에서 지우는 것은 오픈소스 생태계의 인센티브 구조를 파괴한다.

오픈소스 기여자들이 지속적으로 작업하는 이유 중 하나는 자신의 작업이 인정받는다는 확신이다.
대형 프로젝트가 소형 프로젝트의 코드를 가져다 쓰면서 출처를 숨기면, 기여자들의 동기가
약화된다. 장기적으로 이는 오픈소스 생태계 전체의 혁신 속도를 저하시킨다.

MIT 라이선스의 유일한 조건은 저작권 고지다. 이 최소한의 요구도 지키지 않는 기업이
오픈소스 생태계에서 신뢰를 얻기 어렵다는 것은, 커뮤니티의 반응이 증명한다.
llama.cpp가 지금도 활발히 개발되는 이유는 기여자들이 자신의 작업이 존중받는다고 느끼기
때문이다.
