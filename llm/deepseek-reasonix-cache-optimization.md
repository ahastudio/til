# DeepSeek Reasonix와 에이전트 캐시 최적화

원문: <https://javaexpert.tistory.com/1767>

## 요약

“코딩으로 하루 종일 써도 0.1달러가 나온다”는 제목이 과장처럼 보이지만, 저자는 이것이 모델 선택의 문제가 아니라 프롬프트 구조 설계의 문제라고 주장한다.
핵심은 프리픽스 캐싱(prefix caching)이다.
장시간 실행되는 AI 에이전트에서 반복적으로 사용되는 컨텍스트를 캐시가 히트(hit)하도록 세션을 구조화하면 토큰 비용을 대폭 절감할 수 있다는 것이다.

저자가 제시하는 다섯 가지 설계 원칙은 다음과 같다.
첫째, 불변 접두사(stable prefix) 유지: 시스템 프롬프트, 도구 정의, 프로젝트 규칙을 세션 앞부분에 고정하고 이전·다음 요청 간 동일한 앞부분을 유지한다.
둘째, 추가만 가능한 로그 구조: 새로운 내용은 뒤에만 붙이고 기존 접두사를 절대 수정하지 않는다.
셋째, 추론(reasoning) 콘텐츠 재사용 금지: 모델의 추론 과정을 다음 요청의 입력으로 넣지 않는다. 변동성이 크고 캐시 효율을 저해한다.
넷째, 세션 분리: 플래너 모델과 실행 모델, 서로 다른 AI 도구를 같은 대화 세션에 섞지 않는다.
다섯째, 신중한 압축(compaction): 컨텍스트 윈도우 한계에 도달했을 때만 압축을 수행해 캐시 리셋을 최소화한다.

이 원칙들을 적용한 결과로 “99% 이상의 입력 캐시 히트율”이 보고된다.
저자의 결론은 “에이전트 비용 최적화는 토큰 절감이 아니라 반복되는 토큰이 실제로 캐시되도록 구조적으로 유지하는 것”이다.
더 저렴한 모델 선택보다 프롬프트 구조 설계가 비용을 더 크게 결정할 수 있다는 주장이다.

## 분석

### 프리픽스 캐싱의 작동 원리

프리픽스 캐싱은 LLM API에서 동일한 입력 토큰 시퀀스의 앞부분(prefix)이 반복될 때, 그 부분의 KV 캐시(Key-Value cache)를 재사용하는 기술이다.
트랜스포머 모델은 어텐션 계산 과정에서 각 토큰에 대한 Key와 Value를 계산한다.
동일한 접두사가 반복되면 이 계산을 다시 하지 않고 저장된 결과를 그대로 쓴다.
입력 토큰의 계산 비용이 대폭 줄어들고, 그만큼 API 요금이 낮아진다.

Anthropic의 Claude는 프롬프트 캐싱(prompt caching)이라는 이름으로 이 기능을 제공한다.
캐시된 입력 토큰의 비용은 일반 입력 토큰의 10%에 불과하다.
시스템 프롬프트, 도구 정의, 긴 문서 같은 반복 콘텐츠를 캐시하면 실질적인 비용 절감이 가능하다.

### 불변 접두사 원칙의 실천적 어려움

이론적으로 단순한 이 원칙이 실무에서 지키기 어려운 이유가 있다.
에이전트 프레임워크의 기본 동작이 접두사를 변경하는 방향으로 설계된 경우가 많다.
대화 기록을 앞부분에 삽입하거나, 시스템 프롬프트를 동적으로 업데이트하거나, 도구 정의를 컨텍스트에 따라 수정하는 패턴이 일반적이다.

저자가 “추론 콘텐츠를 다음 요청의 입력으로 넣지 않는다”고 강조하는 것도 같은 맥락이다.
Chain-of-thought 방식에서 이전 추론 결과를 다음 요청에 포함시키는 것은 자연스러운 패턴이지만, 이것이 접두사의 안정성을 파괴한다.
캐시 최적화를 의식적으로 설계하지 않으면 기본 패턴이 캐시를 망가뜨린다.

### DeepSeek Reasonix의 위치

제목에 등장하는 DeepSeek Reasonix는 DeepSeek의 추론(reasoning) 특화 모델이다.
저렴한 가격과 강력한 추론 능력으로 주목받았는데, 여기에 캐시 최적화를 더하면 비용이 극적으로 낮아진다는 것이 저자의 주장이다.
“하루 종일 써도 0.1달러”라는 숫자는 DeepSeek의 낮은 기본 요금과 99%+ 캐시 히트율의 조합에서 나온다.

이것은 저렴한 모델이 등장할 때마다 반복되는 패턴이다.
가격이 10분의 1로 낮아지면 “이제 AI를 훨씬 많이 쓸 수 있다”는 기대가 생긴다.
그러나 최적화 없이 단순히 더 많이 쓰면 비용도 비례해 증가한다.
진짜 절감은 모델 가격 인하와 사용 최적화의 조합에서 나온다.

## 비평

### 강점: 숫자로 주장을 구체화

“99% 이상의 캐시 히트율”과 “0.1달러”라는 숫자는 추상적인 “비용을 절감할 수 있다”보다 훨씬 설득력이 있다.
실제 측정 가능한 수치로 주장을 뒷받침하는 것은 실무자에게 신뢰를 준다.

### 약점: 재현 가능성의 조건 부재

“99% 캐시 히트율”이 어떤 조건에서 측정된 것인지가 불명확하다.
어떤 유형의 에이전트 작업인지, 세션 길이는 어느 정도인지, 도구 정의의 크기는 어떤지에 따라 캐시 히트율이 크게 달라진다.
주로 코드 생성 작업에서의 수치인지, 다양한 작업을 포함한 수치인지를 모르면 자신의 상황에 얼마나 적용 가능한지 판단하기 어렵다.

### 보완할 시각: 캐시 최적화와 유연성의 트레이드오프

불변 접두사를 엄격하게 유지하면 에이전트의 적응성이 줄어들 수 있다.
시스템 프롬프트를 작업 유형에 따라 동적으로 바꾸는 것이 더 나은 결과를 만드는 경우가 있다.
“캐시를 위해 접두사를 고정한다”와 “맥락에 맞게 시스템 프롬프트를 조정한다” 사이의 트레이드오프를 어떻게 다뤄야 하는지에 대한 논의가 필요하다.

## 인사이트

### AI 비용 최적화는 인프라 엔지니어링이다

“더 싼 모델을 쓰면 비용이 줄어든다”는 직관은 옳지만 불완전하다.
진짜 최적화는 모델 선택이 아닌 시스템 설계 수준에서 일어난다.

이것은 데이터베이스 최적화와 유사한 구조다.
더 빠른 서버를 사는 것보다 인덱스를 올바르게 설계하는 것이 더 큰 성능 향상을 만드는 경우가 많다.
N+1 쿼리 문제를 모르는 채로 서버 사양만 높이면 비용이 선형으로 증가한다.
AI 에이전트에서 캐시 미스(cache miss)가 반복되는 채로 더 많은 요청을 보내면 비용이 동일하게 증가한다.

프리픽스 캐싱을 의식적으로 설계하는 것은 AI 인프라 엔지니어링의 시작이다.
“어떻게 하면 반복 계산을 줄일 수 있는가”라는 질문이 서버 엔지니어링과 AI 엔지니어링에서 같은 방식으로 작동한다.
AI를 도구로 쓰는 것과 AI를 시스템으로 설계하는 것의 차이가 여기서 드러난다.

### 추론 과정을 입력으로 재사용하지 말라는 원칙의 함의

“모델의 추론 콘텐츠를 다음 요청의 입력으로 넣지 않는다”는 원칙은 캐시 효율만의 문제가 아니다.
더 깊은 질문을 내포한다. 에이전트의 “기억”을 어떻게 구조화할 것인가.

CoT(Chain-of-Thought) 방식에서 이전 추론을 다음 요청에 넣는 것은 에이전트가 과거의 사고 과정을 기억하게 하는 방법처럼 보인다.
그러나 저자는 이것이 캐시를 파괴한다고 지적한다.
해결책은 추론 과정 자체가 아닌 추론의 결론만 다음 단계로 전달하는 것이다.
“왜 이 결론에 이르렀는가”는 버리고 “이 결론이 무엇인가”만 유지하는 것이다.

이것은 인간 기억의 작동 방식과 유사하다.
우리는 모든 사고 과정을 기억하지 않는다. 결론과 그 결론에 이르게 한 핵심 사실만 기억한다.
에이전트 메모리 설계에서 “무엇을 기억할 것인가”와 “어떤 형태로 기억할 것인가”는 성능과 비용 모두에 영향을 미치는 핵심 결정이다.

### 모델 가격 경쟁이 설계 역량의 중요성을 높인다

DeepSeek Reasonix가 기존 모델 대비 낮은 가격으로 유사한 성능을 제공하는 것은 반복되는 패턴의 일부다.
AI 모델의 가격은 지속적으로 낮아지고 있고, 이 추세는 계속될 것이다.

이 추세의 역설적 효과가 있다.
모델이 저렴해질수록 “어느 모델을 쓸 것인가”의 결정보다 “어떻게 사용할 것인가”의 설계가 더 중요해진다.
가격 차이로 만들 수 있는 비용 절감의 상한이 있지만, 최적화로 만들 수 있는 절감은 구조에 따라 10배, 100배가 될 수 있다.

캐시 최적화를 아는 팀과 모르는 팀은 같은 모델을 쓰면서도 완전히 다른 비용 구조를 갖는다.
AI가 범용화될수록 “AI를 쓴다”는 사실이 경쟁 우위가 되지 않는다.
AI를 얼마나 효율적으로, 얼마나 적합한 구조로 사용하는가가 실질적인 차별화 요소가 된다.
이것은 프로그래밍 언어가 범용화된 이후 “어떤 언어를 쓰는가”보다 “얼마나 잘 설계하는가”가 중요해진 것과 같은 전환이다.
