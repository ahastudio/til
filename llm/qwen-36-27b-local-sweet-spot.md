# 로컬 개발의 스위트 스팟: Qwen 3.6 27B

원문: <https://quesma.com/blog/qwen-36-is-awesome/>

HN 토론: <https://news.ycombinator.com/item?id=48721903> (778점, 562개 댓글)

## 요약

Piotr Migdał은 Qwen 3.6 27B가 로컬 AI 모델 배포의 최적 지점이라고 주장한다.
클라우드 인프라 없이도 실용적인 범용 지능 수준의 능력을 제공하면서,
소비자 하드웨어에서 실행 가능한 최초의 모델 중 하나라는 평가다.

Qwen 3.6에는 두 가지 변형이 있다.
MoE(Mixture of Experts) 방식의 35B A3B와 밀집 아키텍처의 27B다.
저자는 속도가 느리더라도 출력 품질이 더 나은 27B를 권장한다.
MacBook Max M5(128GB RAM) 기준으로 MTP(multi-token prediction) 적용 시 32 토큰/초,
35B A3B는 105 토큰/초를 달성하며, 메모리 사용량은 42~45GB다.

실증 테스트로 저자는 양자역학 토론, 조우크 댄스와 양자 물리학에 관한 8행시 생성,
단일 프롬프트로 완성된 육각형 지뢰찾기 구현, 단 하나의 지시로 만든 랜딩 페이지를 보여준다.
설정 방법은 llama.cpp를 Q8_0 양자화와 함께 사용하는 것이며,
OpenCode와의 통합은 `~/.config/opencode/opencode.jsonc`에 설정한다.

```bash
llama-server -hf unsloth/Qwen3.6-27B-MTP-GGUF:Q8_0 \
    --spec-type draft-mtp -ngl 999 -fa on -c 65536 --port 8080
```

Artificial Analysis 벤치마크에서 Qwen 3.6 27B는 2025년 중반 프런티어 모델 수준으로 평가되며,
Gemma 4 31B를 상당히 앞선다.
저자는 로컬 실행 가능한 모델이 빠르게 발전해 결국 스마트폰에서도 프런티어급 지능을 제공하게 될 것이라고 전망하며,
지식 표현과 도구 통합의 분리(separated knowledge representation and tool integration)를 그 경로로 제시한다.

## 분석

### 27B 밀집 모델이 35B MoE보다 실용적인 이유

MoE 모델은 전체 파라미터 수보다 훨씬 적은 활성 파라미터로 추론하기 때문에 속도가 빠르다.
35B A3B의 경우 35B 파라미터 중 실제 활성화되는 것은 3B 수준이다.
그 결과 105 토큰/초라는 높은 속도를 달성하지만,
활성 파라미터가 적을수록 단일 추론 단계의 표현 용량도 제한된다.

27B 밀집 모델은 모든 파라미터가 매 추론에 참여한다.
32 토큰/초로 속도는 3분의 1 수준이지만,
복잡한 추론 체인이 필요한 코딩 작업에서 밀집 모델이 MoE보다 일관성이 높은 경우가 있다.
저자가 “출력 품질”을 기준으로 27B를 선택한 것은 이 트레이드오프를 직접 경험한 결과다.
속도-품질 트레이드오프가 실제 개발 워크플로에서 어떻게 나타나는지에 대한 실증이다.

한편, cpburns2009는[^cpburns2009] DGX Spark, Mac, AMD Ryzen AI Max 같은 통합 메모리 머신을 구매하기 전에 한 가지 사실을 알아야 한다고 경고한다.
밀집 모델은 일반적으로 이런 머신에서 느리게 실행된다는 것이다.
통합 메모리의 대역폭은 전용 GPU VRAM에 비해 제한적이며, 밀집 모델은 모든 파라미터가 추론에 참여하므로 이 병목의 영향을 더 크게 받는다.

ljosifov는[^ljosifov] 더 구체적인 수치로 이를 보완한다.
AMD 7900XTX나 NVIDIA 3090은 약 900 GB/s의 메모리 대역폭을 제공하는데, M5 Max의 약 600 GB/s보다 약 50% 높다.
Qwen 3.6 27B가 이 GPU에서 실행되면 M5 MacBook보다 빠를 수 있으며, 가격은 훨씬 저렴하다.
“Apple Silicon이 로컬 LLM의 최적 하드웨어”라는 통념이 실제로는 마케팅과 생태계 성숙도의 산물일 수 있다는 점을 시사한다.

### 로컬 AI 모델의 “스위트 스팟”이 의미하는 것

“스위트 스팟”은 능력과 하드웨어 요구사항 사이의 균형점을 의미한다.
너무 작으면 실용적이지 않고, 너무 크면 소비자 하드웨어에서 실행할 수 없다.
Qwen 3.6 27B는 Q8_0 양자화 기준 약 27GB의 VRAM/RAM을 사용하며,
MTP까지 더하면 42~45GB가 필요하다.

이 지점은 고급 개발자용 MacBook의 최대 RAM 구성에서 실행 가능한 수준이다.
M4 Max의 최대 128GB, M3 Max의 최대 96GB를 고려하면,
현재 “스위트 스팟”은 고급 노트북 하드웨어와 일치한다.
1~2년 후 동일한 모델이 더 일반적인 64GB 구성에서 실행 가능해질 것이다.
스위트 스팟은 고정된 지점이 아니라 하드웨어 발전에 따라 이동하는 개념이다.

### Artificial Analysis 벤치마크가 전달하는 신호

저자는 Artificial Analysis 점수를 통해 Qwen 3.6 27B를 2025년 중반 프런티어 모델과 동급으로 평가한다.
이것은 중요한 주장이다.
1년 전 클라우드 API로만 접근 가능했던 수준의 지능이 이제 로컬 노트북에서 실행된다는 의미이기 때문이다.

이 압축 속도는 AI 모델 능력의 민주화 속도를 반영한다.
GPT-3.5 수준이 로컬에서 실행되기까지 약 2년,
GPT-4 수준이 로컬에서 실행되기까지 또 약 1년이 걸렸다.
이 주기가 짧아지고 있다.
로컬 실행 모델이 항상 최신 프런티어보다 뒤처지지만,
그 차이가 좁아지는 속도 자체가 하나의 추세다.

## 비평

### “로컬 개발”이라는 표현이 하드웨어 현실을 감춘다

저자의 권장 설정은 MacBook Max M5 128GB RAM이다.
이것은 소비자 하드웨어이지만, 가장 고가의 노트북 구성에 해당한다.
M5 Max 128GB 구성의 시장 가격은 약 4,000달러 이상이다.

“로컬 개발에서 실행 가능하다”는 주장이 맞지만,
“대부분의 개발자에게 접근 가능하다”는 의미는 아니다.
16GB 또는 32GB RAM MacBook을 가진 개발자에게 Qwen 3.6 27B는 실행 불가능하다.
기사가 이 하드웨어 전제를 명시하지 않고 “로컬”이라는 표현을 쓰는 것은
독자에게 잘못된 기대를 줄 수 있다.
실제로 이 테스트를 재현하려면 해당 하드웨어 조건이 필수다.

bensyverson은[^bensyverson] 128GB MBP의 실제 시작 가격이 $6,699임을 지적한다.
같은 금액으로 클라우드 API 크레딧을 구매하면 상당한 기간 동안 프런티어 모델을 사용할 수 있다.
“프라이버시를 위해 이 가격을 지불할 의사가 있는 사람도 있다”는 것은 맞지만,
그것은 경제적 선택이지 “로컬 개발이 실용적”이라는 주장의 일반화 근거가 되지 않는다.

iagooar는[^iagooar] 128GB M5 MacBook Pro를 실제로 사용하는 입장에서 다른 차원의 문제를 제기한다.
Qwen 3.6 같은 무거운 모델을 지속적으로 실행하면 키보드가 뜨거워지고 팬 소음이 심해진다는 것이다.
“진지한 로컬 LLM 코딩을 위해 이 MacBook을 사지 말 것”을 조언하며,
대신 Mac Mini M4 64GB를 작업 공간과 분리된 곳에 두는 방식을 제안한다.
“로컬”이라는 단어가 노트북의 편의성을 연상시키지만, 실제 사용 경험은 그렇지 않을 수 있다.
하드웨어 비용뿐 아니라 발열 관리와 소음이라는 물리적 제약도 “로컬 개발”이라는 프레임이 감추는 현실이다.

### 단일 세션 시연이 “최적 지점” 주장을 뒷받침하기에 부족하다

저자가 제시하는 증거는 몇 가지 인상적인 시연이다.
육각형 지뢰찾기와 랜딩 페이지 생성은 흥미롭지만,
이 작업들이 Qwen 3.6 27B가 “실용적인 범용 지능”임을 증명하는 기준으로 충분한지 불분명하다.

실제 개발 작업에서의 성능은 다르다.
복잡한 멀티파일 코드베이스 탐색, 긴 컨텍스트에서의 디버깅,
도메인 특화 지식이 필요한 작업에서의 성능은 단일 프롬프트 시연과 다를 수 있다.
Gemma 4 31B와의 비교도 Artificial Analysis 점수 참조에 그친다.
이 두 모델을 동일한 작업에서 직접 비교한 결과는 없다.
“스위트 스팟”이라는 강한 주장에 비해 증거가 얇다.

onion2k는[^onion2k] 이 비판을 구체적 경험으로 뒷받침한다.
그린필드 프로젝트는 어떤 모델에게도 상대적으로 쉽다.
진짜 시험은 기존 코드베이스와 함께 작업할 수 있는지 여부다.
Qwen 3.5 기준으로 Rust+React 애플리케이션에서는 충분히 잘 작동했지만,
C# 모노리스에서는 눈에 띄게 약했고 결국 로컬 모델 대신 Claude를 선택했다.
“코드베이스의 성격이 모델 선택을 결정한다”는 이 관찰은,
단일 언어·단일 아키텍처 시연으로 범용성을 주장하는 방식의 한계를 잘 보여준다.

### 미래 전망이 현재 주장의 근거로 쓰인다

저자는 “로컬 모델이 결국 스마트폰에서 실행될 것”이라는 전망으로 글을 마무리한다.
이것은 흥미로운 예측이지만, 현재 Qwen 3.6 27B를 권장하는 근거로 사용할 수 없다.
미래의 로컬 AI 능력이 좋아진다는 것이 오늘의 Qwen 3.6 27B를 채택해야 하는 이유가 되지는 않는다.

지식 표현과 도구 통합의 분리라는 아키텍처 방향도 언급되지만,
이 아이디어가 현재 Qwen 3.6에 어떻게 적용되는지, 또는 어떤 구체적 기술을 의미하는지 설명이 없다.
예측을 현재 권고의 논거로 쓰면 주장의 구조가 약해진다.

## 인사이트

### 로컬 모델이 클라우드 API의 진지한 대안이 되는 임계점

Qwen 3.6 27B가 주목받는 이유는 단순히 좋은 모델이어서가 아니라,
로컬 실행 모델이 처음으로 클라우드 API를 실제 대체할 수 있는 임계점에 도달했기 때문이다.
이전 세대의 로컬 모델(7B, 13B)은 프롬프트 작성, 기본 요약 같은 단순 작업에는 충분했지만
복잡한 코딩이나 추론에서는 GPT-4 수준 API와 격차가 명확했다.

이 임계점 통과는 개인 개발자의 워크플로를 바꾼다.
API 비용 제한 없이 실험할 수 있고, 민감한 코드를 외부 서버로 보내지 않아도 되며,
인터넷 연결 없이도 AI 보조를 유지할 수 있다.
이 세 가지 조건이 동시에 충족되는 것이 이번이 처음에 가깝다.
클라우드 vs 로컬이라는 선택이 능력 트레이드오프가 아니라 사용 맥락의 트레이드오프로 바뀌는 시점이다.

이 임계점이 실제로 작동하고 있음을 보여주는 사례가 있다.
노르웨이 정부 기관에서 개발자용 로컬 모델을 평가하고 있는 starefossen은[^starefossen]
Qwen 3.6이 그 평가에서 좋은 결과를 보이고 있다고 밝혔다.
민감한 공공 데이터를 외부 서버로 보낼 수 없는 규제 환경에서 로컬 모델의 채택은
단순한 비용 절감이 아니라 규제 준수와 데이터 주권의 문제다.
이런 기관 수요가 로컬 모델 채택을 가속할 경우, 소비자 개발자 시장보다 훨씬 빠른 속도로
고품질 로컬 모델에 대한 투자와 생태계가 성숙할 수 있다.

### MoE 모델의 속도 우위가 로컬 환경에서 역설적으로 약점이 된다

데이터센터에서 MoE는 병렬 처리로 추론 속도를 높이는 데 유리하다.
그러나 단일 로컬 머신에서는 MoE의 활성 파라미터 절약이 가져오는 메모리 효율도 있지만,
전문가 라우팅(expert routing) 레이어가 추가 오버헤드를 만들기도 한다.

저자가 105 토큰/초의 35B A3B보다 32 토큰/초의 27B를 선택한 것은
로컬 환경에서 MoE의 이점이 덜 명확하다는 실증이다.
인터랙티브한 코딩 보조 도구에서 30~50 토큰/초는 이미 실시간 출력 속도로 충분하고,
그 이상은 사용자가 읽는 속도보다 빠르다.
속도 임계점 이상에서는 추가 속도보다 품질이 사용자 경험에 더 큰 영향을 미친다.
이 관찰은 향후 로컬 모델 선택 기준이 FLOP 효율성보다 실제 품질 지표로 이동할 것을 시사한다.

### 프런티어-로컬 모델 격차 축소가 AI 도구 생태계를 재편할 수 있다

클라우드 AI API 비즈니스의 핵심 가치 제안은 “최신 프런티어 모델에 대한 접근”이다.
그런데 프런티어와 로컬 실행 가능 모델 사이의 능력 격차가 계속 줄어든다면,
클라우드 API의 차별화 요소가 약해진다.

이것은 코딩 보조 도구 시장에 직접적인 영향을 준다.
GitHub Copilot, Cursor, Codeium 같은 도구들이 클라우드 모델에 기반하는 지금의 사업 모델은,
로컬 모델이 충분히 좋아지면 “완전 로컬” 경쟁자에게 취약해진다.
이미 Continue, Aider, OpenCode 같은 도구들이 로컬 모델과의 통합을 기본으로 설계되고 있다.
데이터 프라이버시에 민감한 기업, API 비용을 최소화하려는 개인 개발자,
인터넷 연결이 제한된 환경의 개발자가 로컬 우선 도구로 이동하는 속도가 빨라질 것이다.
