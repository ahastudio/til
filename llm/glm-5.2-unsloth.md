# GLM-5.2 × Unsloth — 744B MoE 모델 로컬 실행 가이드

<https://unsloth.ai/docs/models/glm-5.2>

HN 토론: <https://news.ycombinator.com/item?id=48636377> (608점, 295개 댓글)

## 소개

GLM-5.2는 Z.AI(Zhipu AI)의 오픈소스 플래그십 모델로,
총 744B 파라미터에 40B 활성 파라미터를 사용하는 MoE(Mixture of Experts) 아키텍처다.
1M 토큰 컨텍스트 윈도우를 지원하며,
장기 코딩, 추론, 에이전틱 작업에 최적화됐다.
Unsloth 문서는 이를 “현재까지 가장 강력한 오픈 모델”로 소개하며
Claude 4.8 Opus, GPT-5.5, Gemini 3.1 Pro와 동등한 성능을 주장한다.

세 가지 사고 모드를 지원한다.

- **non-thinking**: 빠른 응답이 필요한 일반 작업
- **High reasoning**: 중간 수준의 추론 집약적 작업
- **Max reasoning**: 최고 난이도 수학·코드 문제

주요 벤치마크 결과는 다음과 같다.

| 벤치마크        | 점수 |
| --------------- | ---- |
| SWE-Bench Pro   | 62.1 |
| AIME 2026       | 99.2 |
| HLE (Reasoning) | 40.5 |

도구 호출(tool calling)과 에이전트 기반 작업을 지원한다.

## 양자화 옵션

Unsloth는 GLM-5.2를 1비트부터 8비트까지 다양한 양자화 수준으로 제공한다.

| 양자화             | 용량 축소 | 정확도          | RAM 요구량  |
| ------------------ | --------- | --------------- | ----------- |
| 1비트 (UD-IQ1_S)   | 86% 절감  | ~76.2% top-1    | 223 GB      |
| 2비트 (UD-IQ2_M)   | 84% 절감  | ~82% top-1      | 245 GB      |
| 3비트              | —         | —               | 290–360 GB  |
| 4비트 (UD-Q4_K_XL) | —         | 무손실에 근접   | 372–475 GB  |
| 5비트 (UD-Q5_K_XL) | —         | 무손실          | 570 GB      |
| 8비트              | —         | 전체 정밀도     | 810 GB      |

권장 설정은 2비트(UD-IQ2_M)로, 균형 잡힌 정확도(~82%)와 현실적인 메모리 요구량(245GB)을 제공한다.
최소 실행 환경은 256GB 통합 메모리 Mac 또는 24GB GPU + 256GB RAM이다.
동적 KV 캐시 양자화(q4_1)를 활성화하면 컨텍스트 길이를 3.2배 늘릴 수 있다.

## 실행

두 가지 방법으로 실행할 수 있다.

**Unsloth Studio**: 웹 UI 기반 인터페이스로 GPU/RAM을 자동 감지하고
코드 실행 및 모델 훈련 기능을 내장한다.

**llama.cpp**: CLI 기반 추론 도구로 모델 양자화를 직접 지원한다.
사고 모드를 비활성화하려면 다음 옵션을 사용한다.

```bash
--chat-template-kwargs '{"enable_thinking":false}'
```

추천 생성 파라미터:

- `temperature`: 1.0
- `top_p`: 0.95
- `max context`: 1,048,576 토큰

## 분析

### 744B MoE 구조가 비용 효율적인 이유와 그 한계

GLM-5.2의 744B 전체 파라미터와 40B 활성 파라미터 간 비율은
MoE 아키텍처의 핵심 트레이드오프를 드러낸다.
추론 시 실제로 계산에 참여하는 파라미터는 40B 수준이므로
연산 비용은 40B 밀집 모델과 유사하다.
그러나 모든 전문가(expert) 가중치를 메모리에 올려야 하기 때문에
메모리 요구량은 744B에 비례한다.

이것이 2비트 양자화에서도 245GB RAM이 필요한 이유다.
같은 40B 밀집 모델의 2비트 양자화는 약 10GB면 충분하다.
MoE는 “VRAM 비용 없이 더 많은 파라미터를 쓰는” 것이 아니라
“연산 비용 없이 더 많은 파라미터를 쓰는” 것이다.
메모리 요구량은 온전히 유지된다.

### 양자화 계층이 세분화될수록 배포 결정이 복잡해진다

1비트에서 8비트까지 6단계 양자화 옵션의 존재는 사용자에게 유연성을 주는 동시에
결정 피로를 만든다.
76.2%(1비트)에서 ~82%(2비트)까지의 정확도 차이가
실제 태스크에서 어떻게 나타나는지는 양자화 기술의 top-1 정확도 수치만으로는 알 수 없다.

Unsloth가 2비트를 “최적 균형”으로 권장하는 것은 합리적이지만,
이 권장이 범용 태스크에서 유효한지, 아니면 코딩·수학처럼 특정 태스크에 최적화된 것인지는
불분명하다.
1M 컨텍스트를 실제로 활용하는 시나리오(장문서 분석, 대규모 코드베이스 처리)에서
2비트 양자화의 정밀도 손실이 어떻게 나타나는지도 문서화되지 않는다.

### 실행 가능한 하드웨어 임계점이 접근성의 실질적 장벽이다

“256GB 통합 메모리 Mac”이라는 최소 사양은 현재 Apple Silicon 생태계에서
M4 Ultra Mac Studio 또는 Mac Pro 구성에서만 달성 가능하다.
이 시스템의 가격은 6,000달러 이상이다.
“24GB GPU + 256GB RAM” 옵션은 소비자용 GPU(RTX 4090, RTX 5090)와
대용량 DDR5 RAM을 조합하는 방식인데, 이 구성도 수천 달러 규모다.

개인 연구자나 소규모 팀이 744B 모델을 로컬에서 실행하기 위한 하드웨어 비용은
클라우드 API 사용 비용과 비교해야 한다.
모델이 오픈소스라는 것이 곧 접근 가능하다는 의미가 아니다.
로컬 실행의 실질적 접근 계층은 여전히 높다.

walrus01은 문서의 “256GB RAM” 최소 사양에 회의적이다.[^walrus01]
실제 경험상 512GB가 훨씬 현실적인 최소값이라는 것이다.
segmondy는 512GB RAM과 RTX 3090 두 대로 Q4_K_XL을 실행할 때 약 6 tokens/sec를 달성했다고 보고한다.[^segmondy]
총 하드웨어 비용 $2,400으로 이 속도는 “계획 수립이나 계획이 완성된 뒤의 one-shot 프롬프팅”에 적합하다는 평가다.
skiing_crawling은 생성 속도보다 프롬프트 처리(PP, prompt processing) 속도가 진짜 병목이라고 지적한다.[^skiing_crawling]
GPU 없이 RAM 오프로딩에만 의존하는 Mac Studio 같은 구성에서는
PP 속도가 순수 GPU 구성 대비 20-50배 느려지며, 이것이 실용성을 제한한다는 것이다.

## 비평

### 벤치마크 숫자의 선택적 제시가 성능 주장을 과장한다

SWE-Bench Pro 62.1, AIME 2026 99.2, HLE 40.5라는 수치는 인상적으로 보이지만,
이 세 벤치마크가 왜 선택됐는지는 설명되지 않는다.
“Claude 4.8 Opus, GPT-5.5, Gemini 3.1 Pro와 동등”이라는 주장은
어떤 벤치마크에서, 어떤 양자화 수준에서 동등한지를 명시하지 않는다.

전체 정밀도(8비트, 810GB RAM)에서의 성능이 양자화 버전(2비트, 245GB RAM)에서 유지되는지는
별도 검증이 필요하다.
양자화는 평균 성능보다 엣지 케이스와 드문 추론 패턴에서 더 큰 손실을 보이는 경향이 있다.
AIME 99.2 같은 수치가 최대 추론 모드(Max reasoning)에서만 달성되는지,
아니면 기본 모드에서도 재현되는지가 실용적으로 더 중요하다.

CGamesPlay는 “무손실에 근접”이라는 4비트 양자화 설명에 의문을 제기한다.[^CGamesPlay]
97.5% 토큰 일치율(top-1 accuracy)은 수학이나 코딩 추론처럼 정밀도가 요구되는 태스크에서
누적 오차로 이어질 수 있다.
numlock86은 양자화된 모델이 전반적으로 “뇌 일부를 제거한(lobotomized)” 느낌을 준다고 표현하며,
벤더가 직접 제공하는 Gemma 4 같은 경우는 더 나은 성능을 보인다고 말한다.[^numlock86]
처음부터 낮은 비트를 목표로 설계된 모델이 사후 양자화 모델보다 낫다는 주장인데,
현재 이런 모델은 드물다.

### Unsloth Studio의 배포 편의성이 의존성을 숨긴다

“웹 UI와 자동 GPU/RAM 감지”는 사용성을 높이지만,
Unsloth 플랫폼에 대한 의존성도 함께 만든다.
Unsloth Studio가 클라우드 서비스인지 로컬 소프트웨어인지 문서에서 명확하지 않다.
llama.cpp 경로는 완전한 로컬 실행을 가능하게 하지만
더 높은 기술적 진입 장벽을 요구한다.

“오픈소스 모델 + 상업적 배포 플랫폼” 조합은 일반적이지만,
어느 계층이 오픈이고 어느 계층이 상용인지 명확하게 분리하는 것이
사용자가 락인(lock-in) 위험을 평가하는 데 필요하다.

### 1M 컨텍스트가 실용적으로 활용 가능한 조건을 문서가 명시하지 않는다

1,048,576 토큰 컨텍스트 윈도우는 이론적으로 강력하지만,
2비트 양자화 상태에서 실제로 이 컨텍스트를 모두 활용할 수 있는지는 다른 문제다.
동적 KV 캐시 양자화가 “3.2배 더 긴 컨텍스트를 가능하게 한다”는 표현은
기준선이 무엇인지를 숨긴다.

KV 캐시는 컨텍스트 길이에 비례해 메모리를 차지하므로,
최대 1M 컨텍스트를 사용하려면 기본 245GB에 추가 메모리가 필요하다.
문서가 “최대 컨텍스트에서의 메모리 요구량”을 제공하지 않는 것은
가장 인상적인 숫자(1M 토큰)와 가장 제약적인 숫자(RAM 요구량)를
동시에 이해하는 것을 막는다.

zozbot234는 이 문제를 구체적으로 수치화한다.[^zozbot234]
KV 캐시만으로 풀 컨텍스트에서 약 96GB가 필요하며, 이는 기본 모델 메모리 요구량에 추가된다.
결과적으로 실용적인 컨텍스트 윈도우는 약 10,000 토큰 수준에 머문다는 것이다.
1M이라는 수치와 실제 제약된 사용 가능 컨텍스트 사이의 간극은
현재 문서가 다루지 않는 가장 중요한 실용 정보다.

## 인사이트

### 744B MoE 오픈소스 모델의 출현이 “오픈”의 의미를 재정의한다

GPT-3의 175B가 “너무 커서 오픈소스가 불가능하다”는 인식을 만들었던 시절과 비교하면,
744B MoE 모델의 완전 오픈소스 공개는 극적인 변화다.
그러나 “오픈소스”가 “접근 가능”과 같지 않다는 것을 GLM-5.2가 선명하게 드러낸다.

모델 가중치가 공개돼 있어도 실행을 위해 수천 달러의 하드웨어가 필요하다면,
접근의 비대칭성은 유지된다.
이것은 오픈소스 소프트웨어가 “무료지만 설치하려면 전문 지식이 필요”하던 시절과 유사하다.
다만 AI 모델에서는 전문 지식 대신 하드웨어 자본이 장벽이다.
Unsloth의 1비트 양자화(86GB)는 이 장벽을 낮추는 시도지만,
76.2% top-1 정확도가 실용적인지는 태스크 의존적이다.

Havoc은 GLM-5.2의 타이밍이 OpenAI와 Anthropic에게 불리하다고 지적한다.[^Havoc]
“선도적 위치가 있다는 것을 보여 줄 뿐, 지속적인 해자(moat)가 있다는 것을 보여 주지 않는다”는 것이다.
이 관찰은 클로즈드 모델 기업들의 경쟁 우위가 기술력이 아니라 타이밍에 기반한다는 주장으로 읽힌다.
744B MoE 오픈소스 모델이 API 서비스와 동등한 성능을 주장할 때마다,
“왜 API를 써야 하는가”라는 질문이 더 예리해진다.

### 개인 하드웨어와 클라우드 API의 경제성 비교가 AI 접근 모델을 결정한다

256GB 통합 메모리 Mac Studio(~$6,000+)에서 GLM-5.2를 실행하는 경제성을
클라우드 API 사용과 비교하면 흥미로운 계산이 나온다.
하드웨어 비용을 3년으로 상각하면 월 약 $167이다.
이 금액으로 클라우드 API를 통해 얼마나 많은 토큰을 처리할 수 있는가?

이 비교는 사용 패턴에 따라 달라진다.
프라이버시가 중요한 데이터를 처리해야 하거나,
인터넷 연결 없이 작동해야 하거나,
대량의 토큰을 지속적으로 처리해야 하는 경우에는 로컬 실행이 경제적으로 유리해진다.
그러나 간헐적 사용자에게는 클라우드 API가 더 합리적이다.
744B MoE 오픈소스 모델의 가용성은 이 계산식에 새로운 변수를 추가한다.

### MoE 스케일링의 메모리 병목이 다음 양자화 혁신의 방향을 암시한다

현재 양자화 기술은 파라미터 저장 비트 수를 줄이는 방향으로 발전했다.
1비트 양자화(UD-IQ1_S)는 이 방향의 현재 한계에 가깝다.
그러나 MoE 모델에서 진짜 병목은 활성화되지 않는 전문가 가중치를 메모리에 유지해야 한다는 것이다.

다음 단계 혁신은 파라미터 비트 수 축소가 아니라
전문가 가중치의 선택적 메모리 로딩(on-demand expert loading)일 가능성이 있다.
특정 추론 단계에서 활성화될 전문가를 예측하여 미리 로드하거나,
사용하지 않는 전문가를 디스크에서 스트리밍하는 방식이다.
이것은 가상 메모리처럼 실제 RAM보다 큰 모델을 실행하는 방향이며,
Unsloth의 “RAM/VRAM 자동 오프로딩” 기능이 이미 이 방향을 실험하고 있다.
744B 모델을 86GB에서 실행하는 것이 가능해지는 미래는
이 캐싱 전략의 성숙도에 달려 있다.
