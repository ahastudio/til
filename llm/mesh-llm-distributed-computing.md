# Mesh LLM: distributed AI computing on iroh

원문: <https://www.iroh.computer/blog/mesh-llm>

HN 토론: <https://news.ycombinator.com/item?id=48876505> (252점, 58개 댓글)

## 요약

Mesh LLM은 여러 대의 컴퓨터에 흩어진 GPU와 메모리를 하나로 모아
OpenAI 호환 API로 노출시키는 시스템이다.
사무실의 여분 GPU, 워크스테이션, 창고에 처박아 둔 서버처럼 이미 갖고
있는 하드웨어를 묶어서 대형 언어 모델을 함께 돌리자는 것이 핵심
아이디어이며, 대형 클라우드 제공자에 의존하지 않는 것을 목표로 한다.

글은 대형 제공자를 통해 모델에 접근하는 기존 방식이 여러 종속 리스크를
만든다고 주장한다.
사용자는 모델 업데이트, 데이터 위치, 하드웨어 자체에 대한 통제권을
잃고, 비용은 사용량에 따라 예측하기 어렵게 늘어난다.
Mesh LLM은 이런 문제를 “더 많은 통제권, 더 높은 플러그 가능성, 더 낮은
비용”으로 해결하겠다고 내세운다.

기술적으로 요청은 세 가지 경로 중 하나로 처리된다.
요청을 보낸 머신의 GPU에서 직접 실행하거나, 해당 모델을 이미 돌리고
있는 다른 피어 노드로 라우팅하거나, 단일 머신이 감당하기에는 너무 큰
모델의 경우 파이프라인을 여러 머신에 나누어 분할 실행한다.
2350억 파라미터 규모의 mixture-of-experts처럼 극단적으로 큰 모델을
위해서는 “Skippy” 모드를 두어 모델을 레이어 단위로 여러 노드에 분할하고,
각 단계의 활성값(activation)이 다음 단계로 흘러가도록 함으로써 개별
머신은 감당할 수 없는 모델을 여러 대의 평범한 하드웨어가 힘을 합쳐
돌릴 수 있게 한다.

네트워킹 기반으로는 iroh라는 P2P 네트워킹 라이브러리를 사용한다.
iroh는 기기 간에 인증되고 NAT를 통과하는 QUIC 연결을 제공하며, 중앙
서버 없이 홀 펀칭(hole-punching), NAT 통과, 릴레이 폴백을 통해 직접
암호화된 연결을 맺는다.
프로토콜은 세 가지 ALPN 변형으로 나뉘는데, 메인 메시 연산(가십, 라우팅,
HTTP), 컨트롤 플레인 연산(설정, 소유권), 그리고 지연에 민감한 파이프라인
활성값 전송을 각각 처리한다.
기본 연결 내부에서는 앞에 붙는 1바이트 태그로 가십, 추론 요청, 라우팅
쿼리, 피어 생명주기 이벤트 스트림을 구분한다.

Mesh LLM은 노트북에서 돌릴 수 있는 5억 파라미터 모델부터 2350억
파라미터급 거대 모델까지 40개 이상의 모델 카탈로그를 함께 제공하며,
소프트웨어 자체의 설치 용량은 약 18MB로 가볍다.
글은 P2P 인프라, 조직 내부의 컴퓨팅 자원 풀링, 에이전트 기반 애플리케이션을
주요 사용 사례로 강조하며, 향후 iroh의 Swift SDK를 통한 모바일 앱 지원과
ACP(에이전트 표준) 호환을 계획하고 있다고 밝힌다.
이 글에는 성능 벤치마크나 정량적 지표가 전혀 등장하지 않는다.

## 분석

### “소유의 재분산”이라는 오래된 P2P 서사가 LLM 인프라에도 반복된다

Mesh LLM의 핵심 주장 — 중앙화된 제공자에 대한 종속에서 벗어나 개인이
가진 자원을 모아 서비스를 만든다 — 은 BitTorrent, IPFS, 블록체인 등
과거 P2P 기술들이 반복해서 내세워 온 서사의 최신 변주다.
차이가 있다면 이번에는 파일 공유나 화폐가 아니라 연산 자원, 그중에서도
GPU라는 물리적으로 희소한 자원을 대상으로 한다는 점이다.
GPU는 대역폭이나 저장 공간과 달리 재고가 물리적으로 제한되어 있고
가격이 급등한 자원이기 때문에, 이 재분산 서사가 이번에는 실질적인
경제적 동기(고가의 GPU를 개별적으로 놀리지 않고 공유한다)를 갖는다는
점에서 과거 P2P 프로젝트들보다 설득력의 기반이 다르다.

### 파이프라인 분할은 네트워크 지연이라는 근본적 제약과 정면으로 부딪힌다

모델을 레이어 단위로 여러 머신에 나누고 활성값을 다음 단계로 흘려보내는
“Skippy” 모드는 기술적으로는 파이프라인 병렬화(pipeline parallelism)의
표준적인 아이디어를 그대로 적용한 것이다.
하지만 데이터센터 내부의 고속 인터커넥트(NVLink, InfiniBand)를 전제로
설계된 이 기법을, 홈랩이나 사무실처럼 훨씬 느리고 지연이 들쭉날쭉한
일반 네트워크 위에 얹는다는 것은 완전히 다른 문제다.
글이 성능 벤치마크를 전혀 제시하지 않는다는 사실은, 이 아키텍처가
이론적으로는 가능해도 실제 처리량이나 응답 지연이 실용적인 수준인지에
대한 답을 아직 갖고 있지 않다는 신호로 읽을 수 있다.
HN 댓글 jmercouris는 비슷한 아이디어를 스스로 고민해 봤지만 네트워크를
통한 처리량이 지나치게 느려 대화형 사용에는 쓸 수 없었다고 밝혔다[^jmercouris].
반면 SwellJoe는 글에는 없는 성능 수치를 직접 찾아냈는데, 모델 목록에서
Qwen 235B A22B 항목이 “2개 노드에 걸쳐 초당 16토큰으로 검증됨”이라고
명시하고 있었다는 것이다.
다만 그 2개 노드의 사양이나 네트워크 연결 방식은 밝혀져 있지 않았고,
초당 16토큰은 준수한 속도이긴 하지만 대화형 사용에 완전히 편안한
수준에는 살짝 못 미친다고 평가했다[^SwellJoe].
직접 사용해 본 kennywinker는 오래된 GPU에서는 설치 가능한 llama.cpp
빌드가 작동하지 않았고, 외부 llama.cpp 서비스를 프록시하는 기능도
설정에 실패했다며 아직 다듬어지지 않은 부분이 있다고 전했다[^kennywinker].

### iroh라는 네트워킹 계층 선택이 이 프로젝트의 신뢰성을 좌우한다

Mesh LLM 자체보다 그 기반이 되는 iroh 라이브러리의 성숙도가 이 시스템
전체의 실용성을 좌우한다.
NAT 통과, 홀 펀칭, 릴레이 폴백은 P2P 네트워킹에서 가장 까다롭고 오류가
잦은 영역인데, 이 글이 iroh를 신뢰할 수 있는 기반으로 전제하고 그 위에
애플리케이션 계층(모델 라우팅, 파이프라인 분할)을 쌓아 올리는 방식은,
네트워킹 문제와 AI 추론 문제를 분리해서 다룰 수 있게 해 준다는 점에서
합리적인 설계 결정이다.
다만 이는 동시에 Mesh LLM의 신뢰성 상한선이 iroh의 신뢰성 상한선을
넘을 수 없다는 뜻이기도 하다.

## 비평

### 성능 수치의 완전한 부재는 이 글을 비전 선언문 수준에 머물게 한다

이 글은 기술 아키텍처를 상세히 설명하면서도 처리량, 지연 시간, 토큰
생성 속도 같은 정량적 지표를 단 하나도 제시하지 않는다.
분산 추론 시스템에서 가장 중요한 질문은 “정말 되는가”가 아니라 “얼마나
쓸만한 속도로 되는가”인데, 이 글은 정확히 그 질문에 침묵한다.
파이프라인 분할처럼 네트워크 지연에 극도로 민감한 기법을 핵심 기능으로
내세우면서 단 하나의 벤치마크도 제공하지 않는 것은, 이 시스템이 아직
프로덕션 수준의 성능 검증을 거치지 않았거나, 검증했지만 결과가 홍보에
유리하지 않았을 가능성 둘 중 하나를 시사한다.

### 신뢰 경계와 보안 모델에 대한 설명이 전혀 없다

여러 사람의 하드웨어를 묶어 하나의 추론 클러스터로 만든다는 것은,
서로 다른 신뢰 수준을 가진 당사자들이 프롬프트와 활성값이라는 민감한
데이터를 주고받는다는 뜻이다.
글은 “인증되고 암호화된 연결”이라는 전송 계층의 보안만 언급할 뿐,
피어가 악의적으로 잘못된 활성값을 주입하거나, 다른 사용자의 프롬프트를
가로채거나, 모델 가중치를 훔쳐 갈 수 있는 시나리오에 대한 방어를 전혀
다루지 않는다.
조직 내부의 신뢰된 장비끼리 쓰는 경우라면 이 공백이 크지 않겠지만,
글이 “P2P 인프라”를 사용 사례로 명시적으로 언급하는 순간, 낯선 피어와
연결될 가능성을 전제하는 것이므로 이 보안 공백은 글의 비전과 직접
충돌한다.
HN 댓글 darkpicnic도 정확히 이 지점을 물었다.
Mesh LLM이 노드 간 페이로드를 암호화하는지, 다른 사용자의 요청을 읽는
것이 가능한지를 직접 질문했지만[^darkpicnic], 이 글 자체에는 그 답이
없다.
이 프로젝트의 기여자 중 한 명인 i386이 스레드에 직접 참여해 대형 모델을
여러 노드로 분할하는 “Skippy” 엔진을 자신이 만들었다고 밝히며 질문에
응대했는데[^i386], 이는 이런 세부 사항에 대한 답이 공식 문서가 아니라
커뮤니티 질의응답을 통해서만 보완되고 있는 프로젝트의 현재 단계를
보여준다.

### 비용 절감 주장이 전기·유지보수 비용을 배제한 채 계산된다

글은 “더 낮은 비용”을 핵심 가치 제안으로 내세우지만, 이미 보유한
하드웨어를 “그냥 놀리지 않고 쓴다”는 관점에서만 비용을 계산하는
것으로 보인다.
실제로는 GPU를 상시 가동하는 데 드는 전기 요금, 냉각, 하드웨어 마모,
그리고 무엇보다 이 분산 시스템 자체를 운영하고 디버깅하는 데 드는
인건비까지 고려하면, 클라우드 API 호출 비용과의 실질적인 비교는 훨씬
복잡해진다.
특히 개인이나 소규모 팀이 안정적인 가동 시간을 유지하려면 결국 여러
장비를 24시간 켜 둬야 하는데, 이는 “유휴 자원 활용”이라는 원래의
전제와 어긋나는 방향으로 비용 구조를 밀어붙인다.

## 인사이트

### 분산 추론은 결국 “느슨한 결합이 가능한 워크로드”와 “긴밀한 결합이 필요한 워크로드”를 분리하는 문제로 수렴할 것이다

Mesh LLM이 제시하는 세 가지 라우팅 경로(로컬 실행, 피어 라우팅, 파이프라인
분할) 중 앞의 두 가지는 네트워크 지연에 비교적 관대하지만, 파이프라인
분할은 매 레이어마다 활성값을 주고받아야 하므로 지연에 극히
민감하다.
이는 분산 추론 시스템이 결국 워크로드를 지연 민감도에 따라 분류하고,
각 계층에 맞는 다른 배치 전략을 적용해야 한다는 방향으로 발전할 것임을
시사한다.
즉 미래의 분산 추론 도구는 “이 요청은 통째로 한 노드에서 실행 가능한가,
아니면 여러 노드에 걸쳐 쪼개야 하는가”를 자동으로 판단하고 배치를
최적화하는 스케줄러 계층을 필수적으로 갖추게 될 가능성이 높다.

### 홈랩 컴퓨팅의 부상은 로컬 LLM 생태계의 다음 단계를 예고한다

Ollama, LM Studio 같은 도구들이 “내 컴퓨터 한 대에서 모델을 돌린다”는
단계를 대중화했다면, Mesh LLM류의 프로젝트는 “내가 가진 여러 대의
컴퓨터를 묶어서 돌린다”는 다음 단계를 시도하고 있다.
이 전환은 개인 하드웨어의 총합 연산 능력이 이미 상당한 수준에 이르렀고,
동시에 클라우드 AI API 비용에 대한 피로가 누적되고 있다는 두 흐름이
교차하는 지점에서 나타난다.
장기적으로 이런 프로젝트들이 성숙하면, 가정용 네트워크와 소규모
사무실 네트워크가 사실상 미니 데이터센터로 기능하는 “홈랩
데이터센터화” 흐름이 로컬 LLM 생태계의 다음 국면으로 자리 잡을 수
있다.

### 신뢰 경계가 불분명한 P2P 추론 네트워크는 결국 평판 기반 라우팅을 필요로 하게 될 것이다

이 글이 다루지 않은 보안 공백 — 낯선 피어가 신뢰할 수 없는 활성값을
주입하거나 프롬프트를 가로챌 가능성 — 은, Mesh LLM류의 프로젝트가
성장할수록 반드시 마주하게 될 문제다.
BitTorrent 생태계가 결국 트래커의 신뢰도, 피어의 업로드 이력 같은
평판 메커니즘을 발전시켰던 것처럼, P2P LLM 추론 네트워크도 결국
“어떤 피어를 신뢰하고 어떤 요청을 그 피어에게 맡길 것인가”를 판단하는
평판 시스템이나 검증 가능한 연산(verifiable computation) 메커니즘을
필요로 하게 될 것이다.
이 문제를 먼저 해결하는 프로젝트가, 단순히 기술적으로 “작동하는” P2P
추론 네트워크를 넘어 실제로 낯선 사람들 사이에서 신뢰하고 쓸 수 있는
인프라로 자리 잡을 가능성이 높다.
실제로 whs가 공유했듯, Mesh LLM 이전에도 AI Horde 같은 분산 컴퓨팅
LLM 커뮤니티가 이미 존재해 왔고, AI Horde는 KoboldCPP의 텍스트 완성
API를 표준으로 채택해 왔다[^whs].
이런 선행 커뮤니티들이 이미 겪었을 신뢰·평판 문제의 해법을 Mesh LLM이
그대로 재발명할 필요 없이 참고할 수 있다는 점은, 이 인사이트가 순수한
예측이 아니라 이미 부분적으로 검증되고 있는 방향이라는 근거가 된다.

### 소비자 하드웨어의 총합은 “가진 사람들”의 상상보다 훨씬 불균등하게 분포한다

HN 댓글 MattPerry는 이 글의 예시 이미지(GPU 리그, 노트북, 서버, 클라우드
노드)를 보고, 자신에게는 24GB VRAM 노트북도 96GB 워크스테이션도 없다는
사실을 새삼 깨달았다고 밝혔다[^MattPerry].
이는 이 글이 암묵적으로 전제하는 “우리 모두 이미 상당한 유휴 GPU를
갖고 있다”는 가정이 실제로는 특정 계층의 개발자에게만 해당하는 이야기일
수 있음을 드러낸다.
Mesh LLM 같은 프로젝트가 진정한 의미의 “탈중앙화된 컴퓨팅 자원 재분배”를
이루려면, 하드웨어를 이미 가진 소수가 서로 자원을 공유하는 것을 넘어,
애초에 하드웨어 접근성 자체가 고르지 않다는 훨씬 근본적인 불평등
문제와 마주하게 될 것이다.
