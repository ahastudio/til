# Modular RAG: 파이프라인에서 의사결정 시스템으로

출처:
<https://twitter.com/ihtesham2005/status/2028900581084131420>

참고 논문:
[Retrieval-Augmented Generation for Large Language Models:
A Survey](https://arxiv.org/abs/2312.10997)
(Gao et al., 2024)

## 요약

RAG의 진화를 세 단계로 구분한다.
Naive RAG → Advanced RAG → Modular RAG.

핵심 통찰은 단순하다.
검색(retrieval)을 고정된 파이프라인의 “단계”로
취급하지 말고, 모델이 동적으로 내리는
“의사결정”으로 전환하라는 것이다.

Modular RAG는 실제 프로덕션 태스크에서
30~40% 정확도 향상을 보인다.
복잡한 멀티홉(multi-hop) 질문에서
Advanced RAG 대비 성능 격차가 더 벌어진다.

## 세 가지 RAG 패러다임

| 패러다임     | 구조              | 특징                   |
| ------------ | ----------------- | ---------------------- |
| Naive RAG    | 검색 → 삽입 → 생성 | 고정 파이프라인        |
| Advanced RAG | 전처리 + 후처리   | 쿼리 최적화, 리랭킹   |
| Modular RAG  | 모듈 조합         | 동적 라우팅, 자기 비판 |

### Naive RAG의 한계

검색 → 컨텍스트 삽입(stuff) → 생성.
매번 이 순서를 강제한다.

문제점:

- 정밀도 낮음: 관련 없는 청크가 섞임
- 재현율 낮음: 필요한 문서를 놓침
- 한 번의 잘못된 검색이 전체 답변을 무너뜨림

### Advanced RAG의 개선

검색 전(pre-retrieval)과
검색 후(post-retrieval) 단계를 최적화한다.

- 검색 전: 쿼리 확장(expansion),
  쿼리 재작성(rewriting)
- 검색 후: 리랭킹(reranking) 알고리즘으로
  검색 결과를 정제

파이프라인 구조 자체는 여전히 고정적이다.

### Modular RAG의 전환

고정 파이프라인을 해체하고 독립 모듈을 조합한다.

확장 모듈:

- **Search**: 다양한 소스에서 검색
- **Memory**: 이전 검색·생성 결과를 기억
- **Fusion**: 여러 쿼리 결과를 통합
- **Routing**: 질문 유형에 따라 경로 결정
- **Predict**: 검색 없이 직접 예측 시도
- **Task Adapter**: 태스크별 맞춤 조정

Naive RAG와 Advanced RAG는 Modular RAG의
특수한 경우(고정 모듈 조합)에 해당한다.

## Self-RAG: 자기 비판 메커니즘

Modular RAG의 핵심 기법 중 하나다.
모델이 스스로 출력을 비판하고,
확신도(confidence)가 낮으면 추가 컨텍스트를
검색하도록 한다.

동작 원리:

1. 검색이 필요한지 스스로 판단
2. 검색한 문서가 관련 있는지 평가
3. 생성한 답변이 문서에 근거하는지 검증
4. 확신도가 낮으면 추가 검색 수행

전문 연구자의 사고방식과 일치한다.
분석가는 모든 것을 미리 검색하지 않는다.
무엇을 가져올지, 언제 더 가져올지,
이미 충분한지를 판단한다.

## 핵심 수치

논문에서 보고된 결과:

- **30% 정확도 향상**: 제어된 노이즈를 추가해
  모델이 신호를 필터링하도록 학습시킨 결과
- 복잡한 멀티홉 질문에서 Modular 시스템이
  Advanced RAG를 상회
- 여러 소스를 종합해야 하는 태스크에서
  성능 격차가 더 확대

## Naive RAG vs Modular RAG 비유

> Naive RAG made AI retrieve like a search engine.
> Modular RAG makes it retrieve like a researcher.

검색 엔진은 쿼리를 받으면 무조건 결과를 반환한다.
연구자는 질문에 따라 전략을 바꾼다.
어떤 소스를 볼지, 몇 번 검색할지,
어떤 형식으로 가져올지를 능동적으로 결정한다.

## 관련 기법

| 기법                    | 설명                         |
| ----------------------- | ---------------------------- |
| Self-RAG                | 자기 비판 기반 적응형 검색   |
| FLARE                   | 생성 중 능동적 검색 트리거   |
| RAG-Fusion              | 다중 쿼리 결과 통합(RRF)    |
| RQ-RAG                  | 멀티홉 쿼리 분해            |
| Adaptive Retrieval      | 검색 필요성 동적 판단        |
| Knowledge Graph + RAG   | 구조화된 지식과 결합         |

## 인사이트

### 1) 검색은 “단계”가 아니라 “판단”이다

Modular RAG의 가장 중요한 전환점이다.
기존 RAG는 검색을 파이프라인의 한 단계로 고정했다.
Modular RAG는 검색 자체를 모델의 의사결정으로
전환한다.
검색할지 말지, 몇 번 할지, 어디서 할지,
어떤 형식으로 할지를 모델이 결정한다.

소프트웨어 아키텍처에서 하드코딩된 절차를
플러그인 시스템으로 전환하는 것과 같다.
if-else로 가득한 코드를 전략 패턴(Strategy
Pattern)으로 리팩터링하면 새로운 전략을
추가할 때 기존 코드를 건드리지 않아도 된다.
Modular RAG에서 새로운 검색 모듈을 추가하는 것도
동일한 원리다.

더 근본적으로는 “제어의 역전(Inversion of
Control)”이 일어난다.
Naive RAG에서는 파이프라인이 검색을 호출한다.
Modular RAG에서는 모델이 필요할 때 검색을
호출한다.
프레임워크가 코드를 호출하는 것과,
코드가 라이브러리를 호출하는 것의 차이다.

### 2) 실패 격리가 품질을 만든다

Naive RAG에서는 한 번의 잘못된 검색이
전체 답변을 오염시킨다.
검색 결과에 무관한 문서가 하나라도 섞이면
모델은 그 노이즈까지 포함해서 답변을 생성한다.
“쓰레기가 들어가면 쓰레기가 나온다(GIGO)”의
전형이다.

Modular RAG는 각 모듈이 독립적이므로
한 검색의 실패가 전체를 무너뜨리지 않는다.
Self-RAG의 자기 비판 메커니즘이 여기서 핵심이다.
검색 결과를 평가하고, 부적절하면 버리고,
필요하면 다시 검색한다.

마이크로서비스 아키텍처의 장애 격리(fault
isolation), 서킷 브레이커(circuit breaker)와
동일한 원리다.
한 서비스의 장애가 전체 시스템으로 전파되지
않도록 격벽(bulkhead)을 두는 것처럼,
한 검색의 실패가 최종 답변으로 전파되지 않도록
평가 단계를 둔다.

### 3) 노이즈가 오히려 강화 신호가 된다

제어된 노이즈를 학습에 추가하면
모델이 관련 없는 정보를 필터링하는 능력이
향상된다.
30% 정확도 향상의 핵심 메커니즘이다.

드롭아웃(dropout)이 뉴런을 랜덤하게 꺼서
과적합을 방지하듯,
데이터 증강(data augmentation)이 이미지를
회전·변형시켜 일반화 능력을 키우듯,
검색 결과에 의도적으로 노이즈를 섞으면
모델이 “진짜 관련 있는 정보”를 가려내는
변별력을 학습한다.

이것은 인간의 비판적 사고 훈련과도 유사하다.
토론에서 반론에 노출되면 자신의 논거를
더 정밀하게 다듬게 되는 것처럼,
모델도 노이즈에 노출되면 신호를 더 정확하게
식별한다.

### 4) 멀티홉 추론에서 격차가 벌어지는 이유

단순 질문(“X는 무엇인가?”)에서는
Naive RAG와 Modular RAG의 차이가 크지 않다.
한 번 검색으로 충분한 답을 얻을 수 있기
때문이다.

차이가 극적으로 벌어지는 것은
멀티홉(multi-hop) 질문이다.
“A의 저자가 졸업한 대학의 설립자는 누구인가?”
같은 질문은 여러 번의 연쇄적 검색이 필요하다.

Naive RAG는 이런 질문에 구조적으로 취약하다.
한 번 검색하고 끝이기 때문이다.
Modular RAG는 질문을 분해하고,
각 하위 질문에 대해 독립적으로 검색하고,
중간 결과를 종합해서 다음 검색을 결정한다.

이것은 프로그래밍에서 재귀(recursion)와
반복(iteration)의 차이와 비슷하다.
Naive RAG는 단일 함수 호출이다.
Modular RAG는 결과에 따라 재귀적으로
자신을 호출하는 함수다.

### 5) “언제 검색하지 않을 것인가”의 가치

Modular RAG의 Predict 모듈은 간과되기 쉽지만
실전에서 큰 가치가 있다.
모델이 이미 충분한 지식을 갖고 있는 질문에
대해 불필요한 검색을 건너뛴다.

이것이 중요한 이유는 세 가지다.

첫째, 레이턴시(latency).
검색은 외부 시스템 호출이므로 시간이 걸린다.
검색이 필요 없는 경우를 걸러내면
응답 속도가 극적으로 향상된다.

둘째, 비용.
벡터 DB 쿼리, 임베딩 생성, 리랭킹 모델 호출은
모두 비용이다.
불필요한 검색을 줄이면 인프라 비용이 줄어든다.

셋째, 정확도.
검색 결과가 항상 도움이 되는 것은 아니다.
때로는 모델이 이미 알고 있는 답을
검색 결과가 오히려 방해할 수 있다.
“검색하지 않는 판단”도 정확도 향상에 기여한다.

### 6) RAG 진화는 소프트웨어 아키텍처 진화를 반복한다

Naive RAG → Advanced RAG → Modular RAG의
진화 경로는 소프트웨어 아키텍처의 역사와
놀라울 정도로 유사하다.

- Naive RAG = 모놀리식(monolithic) 아키텍처.
  모든 것이 하나의 파이프라인에 결합되어 있다.
  단순하지만 유연성이 없다.
- Advanced RAG = 계층형(layered) 아키텍처.
  전처리·후처리 계층을 추가해 품질을 높인다.
  구조는 여전히 고정적이다.
- Modular RAG = 마이크로서비스 아키텍처.
  독립 모듈을 조합하고 라우팅한다.
  유연하지만 복잡성이 증가한다.

다음 단계도 예측할 수 있다.
이벤트 드리븐(event-driven) 아키텍처처럼
모듈 간 메시지 기반 통신이 도입되거나,
서버리스(serverless)처럼 필요할 때만
모듈이 활성화되는 방향으로 진화할 가능성이 있다.

### 7) “연구자처럼 검색한다”는 메타포의 깊이

트윗의 핵심 비유다.

> Naive RAG made AI retrieve like a search engine.
> Modular RAG makes it retrieve like a researcher.

이 비유가 단순한 수사가 아닌 이유가 있다.
실제 연구자의 정보 탐색 행동을 관찰하면
Modular RAG의 모듈과 정확히 대응된다.

- 연구자는 먼저 질문을 정제한다 → Routing
- 이미 아는 것은 검색하지 않는다 → Predict
- 여러 소스를 교차 확인한다 → Fusion
- 이전 검색 결과를 기억한다 → Memory
- 찾은 정보의 신뢰도를 평가한다 → Self-RAG
- 부족하면 검색 전략을 바꿔 재시도한다
  → Adaptive Retrieval

Modular RAG는 결국 “전문가의 인지 과정”을
시스템 아키텍처로 형식화한 것이다.
인지 과학에서 말하는 메타인지(metacognition),
즉 “자신의 사고에 대한 사고”를 AI 시스템에
구현한 셈이다.

## 적용 아이디어

- 프로덕션 RAG 시스템에 검색 필요성 판단
  모듈을 추가하는 것만으로도 불필요한 검색을
  줄이고 정확도를 높일 수 있다
- Self-RAG 패턴을 도입해 생성 결과의 확신도에
  따라 추가 검색을 트리거하는 피드백 루프를
  구현할 수 있다
- 질문 유형에 따라 다른 검색 전략을 라우팅하는
  것은 비교적 간단하면서도 효과가 크다
- 기존 Naive RAG에 한 번에 전부 전환하기보다
  Routing → Predict → Self-RAG 순으로
  모듈을 점진적으로 도입하면
  각 단계의 효과를 측정할 수 있다
- 멀티홉 질문이 많은 도메인(법률, 의료, 학술)
  에서는 쿼리 분해(query decomposition)
  모듈부터 도입하는 것이 ROI가 높다
