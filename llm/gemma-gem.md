# Gemma Gem - 브라우저에서 Gemma 4 완전 온디바이스 실행

<https://github.com/kessler/gemma-gem>

## 요약

Gemma Gem은 Google의 Gemma 4 모델을 WebGPU 기반으로 브라우저에서 완전히 로컬에서
실행하는 Chrome 확장 프로그램이다. API 키나 클라우드 서버 없이 모든 추론이 사용자
기기에서 처리된다. 외부로 전송되는 데이터가 없어 개인정보 관점에서 완전한 격리를 제공한다.

모델은 두 가지 크기를 선택할 수 있다. E2B(500MB)는 빠른 응답에 적합하고 E4B(1.5GB)는
더 높은 품질의 응답을 제공한다. 두 모델 모두 q4f16 양자화와 128K 컨텍스트를 지원한다.
WebGPU 지원 Chrome이 필요하다.

에이전트 기능이 포함되어 있다. 현재 보고 있는 웹페이지 내용을 읽고, 버튼 클릭과 폼 작성
등 페이지 상호작용을 수행하며, JavaScript 실행과 스크린샷 캡처도 가능하다. CSS 셀렉터
기반 DOM 조작을 지원한다.

아키텍처는 Offscreen Document + Service Worker + Content Script의 3계층 구조다.
Offscreen document가 모델을 호스팅하고 에이전트 루프를 실행한다. WXT(Vite 기반
Chrome 확장 프레임워크)와 @huggingface/transformers를 사용하며 TypeScript로 작성되었다.
라이선스는 Apache-2.0이다.

## 분석

### WebGPU와 브라우저 ML 추론의 성숙

WebGPU는 브라우저에서 GPU를 직접 활용할 수 있는 API로, WebGL의 후속이다. @huggingface/transformers 라이브러리가 WebGPU 백엔드를 지원하면서 브라우저에서
실용적인 LLM 추론이 가능해졌다. Gemma Gem은 이 기술 조합의 현재 한계와 가능성을
동시에 보여주는 프로젝트다.

E4B 모델 1.5GB를 브라우저에서 실행한다는 것은 초기 로드에 상당한 시간이 필요함을 의미한다.
실용적 도구로 정착하려면 모델 캐싱 전략이 중요하다. Chrome의 Cache API나 IndexedDB를
활용한 로컬 모델 캐싱이 이미 구현되어 있을 가능성이 높다.

### 3계층 아키텍처의 필연성

Chrome 확장 프로그램의 보안 모델이 이 아키텍처를 강제한다. Service Worker는 UI가 없는
백그라운드 스크립트로 네트워킹과 스토리지를 담당한다. Offscreen Document는 Service
Worker에서는 불가능한 DOM 조작과 복잡한 계산을 위한 숨겨진 페이지다. Content Script는
실제 웹페이지에 주입되어 DOM에 접근한다.

모델 호스팅을 Offscreen Document에서 처리하는 것은 메모리와 GPU 자원을 Service Worker
생명주기와 독립적으로 관리하기 위한 선택이다. Service Worker는 일정 시간 비활성 상태가
되면 종료되어 상태가 사라지지만, Offscreen Document는 더 긴 생명주기를 가질 수 있다.

## 비평

### 강점

완전한 프라이버시 격리라는 가치 제안이 명확하다. 기업 환경에서 외부 AI 서비스 사용이
제한된 상황, 또는 민감한 개인 정보를 다루는 작업에서 온디바이스 실행의 가치는 크다.

에이전트 기능의 포함이 차별점이다. 단순한 채팅 인터페이스가 아니라 페이지 상호작용까지
처리하는 것은, 브라우저 내 작업 자동화라는 실용적 사용 시나리오를 만든다.

### 약점

WebGPU 지원 브라우저 요구사항이 제한적이다. Safari는 WebGPU를 부분적으로 지원하고
Firefox는 실험적 지원 단계다. 사실상 Chrome/Chromium 전용이다.

E4B 모델 1.5GB 다운로드는 처음 사용 시 상당한 장벽이다. 대역폭이 제한된 환경이나
모바일 데이터를 사용하는 경우 실용적이지 않다.

Gemma 4 모델이 지원하는 언어 범위와 한국어 품질에 대한 정보가 없다. 로컬에서 실행되는
만큼 사용자가 직접 확인해야 한다.

## 인사이트

### 온디바이스 AI의 실용화 임계점

브라우저에서 1.5GB 모델을 실행하는 것이 가능해졌다는 사실은, 온디바이스 AI의 실용화
임계점이 빠르게 낮아지고 있음을 보여준다. 2년 전에는 모바일 기기에서 소형 모델 추론이
연구 단계였지만, 지금은 브라우저 확장 프로그램으로 구현된다.

이 추세가 계속되면 AI 서비스의 경제학이 바뀐다. 현재는 API 호출당 비용이 발생하고 서버
인프라가 필요하다. 온디바이스 추론이 충분히 실용적이 되면, 비용 구조가 초기 모델 개발
비용으로 집중되고 서빙 비용이 사용자에게 전가된다. 이것이 좋은 것인지 나쁜 것인지는
사용 맥락에 따라 다르다.

프라이버시 관점에서는 명확히 긍정적이다. 데이터가 기기를 떠나지 않으면 해킹, 유출,
규제 위반의 위험이 근본적으로 다른 양상이 된다.

### 브라우저가 AI 실행 환경이 되는 의미

웹 브라우저는 HTML/CSS/JavaScript 실행 환경으로 설계되었다. 이것이 범용 애플리케이션
플랫폼이 되었고, 이제 AI 모델 실행 환경이 되고 있다. WebGPU는 브라우저의 역할을 또 한번
확장하는 기술이다.

이 변화는 AI 기능의 배포 방식을 바꾼다. 기존에는 AI 기능을 추가하려면 서버 인프라를
갖추거나 외부 API에 의존해야 했다. WebGPU 기반 온디바이스 추론이 일반화되면, 정적
웹페이지에도 AI 기능을 내장할 수 있게 된다. CDN에서 호스팅되는 파일만으로 AI 기능이
제공된다.

이것은 AI 민주화의 다른 형태다. API 키 없이, 서버 없이, 비용 없이 AI 기능을 구현하는
방법이 생긴다. 다만 사용자 기기 성능과 초기 모델 다운로드가 새로운 제약이 된다.

### 확장 프로그램 에이전트의 가능성과 위험

Gemma Gem이 구현하는 페이지 상호작용 에이전트는, 브라우저 자동화와 AI를 결합한 새로운
패러다임의 초기 형태다. 웹페이지 내용을 이해하고 동작을 수행할 수 있는 AI가 브라우저에
내장된다는 개념이다.

이 개념이 완성되면 강력한 개인 자동화 도구가 된다. 반복적인 웹 작업, 정보 수집, 폼 작성
등을 로컬에서 처리할 수 있다. 동시에 악용 가능성도 있다. 같은 기술이 자격증명 탈취나
사용자 행동 추적에 사용될 수 있다. 온디바이스 실행은 클라우드 모니터링이 불가능해
악의적 확장 프로그램의 탐지가 더 어려워진다는 의미이기도 하다.
