# Introducing Hy3

원문: <https://hy.tencent.com/research/hy3>

## 요약

텐센트의 Hy(Hunyuan) 팀은 2026년 7월 6일, 4월 말 공개했던 Hy3 프리뷰의
정식 버전인 Hy3를 발표했다.
프리뷰 출시 이후 50개 이상의 제품에서 피드백을 수집하고 더 고품질의
데이터로 post-training을 확장한 결과, Hy3는 비슷한 크기의 모델을
능가하고 2~5배 많은 파라미터를 가진 플래그십 오픈소스 모델과도
경쟁할 수 있는 수준에 도달했다고 밝힌다.

에이전트 능력 측면에서 팀은 post-training 데이터의 품질과 다양성을
개선하고 강화학습(RL) 훈련 규모를 확대했다.
추론, 에이전트 작업, 장문 컨텍스트 작업에서 훨씬 큰 플래그십
모델들과 경쟁할 수 있는 성과를 보였다고 한다.
코딩, 사무 업무, 재무 모델링, 프론트엔드 디자인, 게임 개발 등
생산성 시나리오에서 270명의 전문가를 대상으로 실제 업무 과제를
활용한 블라인드 평가를 진행한 결과, Hy3는 4점 만점에 2.67점을
기록해 GLM-5.1의 2.51점을 앞섰다.
프론트엔드 개발, 데이터·스토리지, CI/CD 작업에서 격차가 가장
컸다.

팀은 벤치마크만으로는 모델의 실용성을 온전히 포착할 수 없다고
전제하며, 제품 피드백을 바탕으로 세 가지 문제를 개선했다고
설명한다.
첫째, 도구 호출과 출력 형식의 안정성이다.
여러 기본 신뢰성 문제를 수정해 프로덕션 수준의 도구 호출 오류
복구와 효율성을 확보했으며, CodeBuddy·Cline·KiloCode 등
서로 다른 에이전트 스캐폴딩 사이에서도 SWE-Bench Verified 정확도
편차가 4%포인트 이내로 유지된다고 밝힌다.
둘째, 지식과 환각 방지다.
“근거가 있을 때만 답하고, 근거가 없으면 그렇다고 밝히며, 출처를
혼동하거나 데이터를 조작하지 않는다”는 원칙 아래 세밀한 데이터
정제와 훈련 제약을 적용한 결과, 내부 평가에서 환각률이 12.5%에서
5.4%로, 상식 오류율이 25.4%에서 12.7%로 떨어졌다고 한다.
셋째, 복잡한 맥락 유지와 다중 턴 의도 추적이다.
SFT와 RL을 함께 최적화해 대용사 해소, 생략 복원, 다중 턴 제약
계승 같은 문제를 개선했고, 내부 종합 다중 턴 테스트에서 문제
발생률이 17.4%에서 7.9%로 낮아졌으며 MRCR 같은 장문 대화
평가에서도 개선을 보였다고 한다.

실제 제품 적용 사례로 WorkBuddy의 Context Engineering 리드는
내부 테스트에서 작업 성공률이 Hy3 프리뷰의 72%에서 Hy3의 90%로
올랐고 평균 완료 시간은 34% 줄었다고 밝힌다.
문서 처리 작업에서는 GLM-5.2 대비 토큰을 47.4% 적게 사용했고,
프레젠테이션 작성에서는 49% 적게 사용했다고 한다.
Hy3는 Apache 2.0 라이선스로 GitHub, HuggingFace, ModelScope,
AtomGit에 공개되었으며, 하드웨어·소프트웨어 공동 최적화를 통해
API 가격을 1M 토큰 기준 입력 1위안, 출력 4위안, 캐시된 입력
0.25위안으로 낮췄다.
팀은 1월 말 인프라 재구축을 시작해 4월 프리뷰, 7월 정식 버전과
다양한 제품 배포에 이르기까지 6개월 만에 엔드투엔드 모델 개발
루프를 완료했다고 밝히며, Hunyuan의 재구축과 진화는 이제
시작 단계라고 마무리한다.

## 분석

### “제품 피드백”을 벤치마크보다 앞세우는 서사 구조

이 글이 가장 먼저 밝히는 전제는 “모델의 유용성은 벤치마크만으로
완전히 포착되지 않는다”는 것이다.
이 전제는 글 전체의 구조를 결정한다.
에이전트 능력 섹션에서 벤치마크형 수치(SWE-Bench, 블라인드
평가 점수)를 제시한 뒤, 곧바로 “더 신뢰할 수 있는 제품 경험”
섹션에서 벤치마크로 잡히지 않는 세 가지 문제(도구 호출 안정성,
환각, 다중 턴 맥락 유지)를 별도로 다룬다.
이는 벤치마크 경쟁이 이미 포화 상태에 이르러, 벤치마크 점수
차이만으로는 실제 채택을 설득하기 어려워졌다는 업계 전반의
인식을 반영한다.

50개 이상의 제품에서 수집한 피드백을 근거로 내세우는 것도
같은 맥락이다.
연구소 내부 평가가 아니라 실제 프로덕션 배포에서 나온 문제를
고쳤다는 서사는, 독자에게 “이 개선은 실험실이 아니라 현장에서
검증되었다”는 신호를 준다.
WorkBuddy 사례처럼 구체적인 제품명과 실무자의 인용을 배치한 것도
추상적 벤치마크 수치보다 설득력 있는 증거로 기능한다.

### RL 확장과 데이터 품질 개선을 분리해 설명하는 이유

Hy3의 개선을 설명하며 팀은 “post-training 데이터의 품질과
다양성 개선”과 “RL 훈련 규모 확대”를 병렬로 언급한다.
이 둘을 구분해 서술하는 것은 우연이 아니다.
RL 확장만으로는 모델이 이미 가진 능력을 더 잘 끌어내는 것에
그치지만, 데이터 품질 개선은 모델이 애초에 접하는 신호 자체를
바꾼다.
환각률과 상식 오류율 감소가 “세밀한 데이터 정제와 훈련
제약”이라는 표현으로 설명되는 것도 이 구분과 일치한다.

이 구분은 최근 LLM 개발 담론에서 반복되는 논쟁, 즉 “스케일이
전부인가, 데이터 큐레이션이 핵심인가”라는 질문에 대한 이
팀의 입장을 드러낸다.
Hy3는 두 축을 모두 강화했다고 주장하지만, 구체적 개선 수치의
상당수(환각률, 상식 오류율, 다중 턴 문제 발생률)가 데이터·훈련
제약 쪽 개선에서 나온 것으로 서술된다는 점에서, 이 팀이 순수한
스케일 확장보다 데이터 정제와 세부 제약 설계에 더 큰 비중을
두고 있다는 것을 짐작할 수 있다.

### 여러 에이전트 스캐폴딩 간 일관성을 강조하는 배경

SWE-Bench Verified에서 CodeBuddy, Cline, KiloCode 세 스캐폴딩
사이의 정확도 편차가 4%포인트 이내라는 수치는 언뜻 사소해
보이지만, 실제로는 이 모델이 겨냥하는 시장의 구조를 보여준다.
오픈소스로 공개되는 모델은 특정 벤더가 만든 단일 에이전트
환경이 아니라, 서드파티가 만든 다양한 에이전트 프레임워크 위에서
동작해야 한다.
어떤 스캐폴딩에서는 잘 작동하지만 다른 스캐폴딩에서는 성능이
크게 떨어지는 모델은, 서드파티 개발자 입장에서 예측 불가능한
위험 요소가 된다.

이 수치를 명시적으로 강조한다는 것은, Hy3가 자사 제품군
(WorkBuddy, Yuanbao 등)에 최적화된 폐쇄적 모델이 아니라 오픈
생태계에서 여러 도구와 조합해 쓰일 것을 전제로 설계되었다는
의도를 보여준다.
이는 오픈소스 모델의 경쟁력이 단일 벤치마크 점수가 아니라
“어떤 환경에 배치하든 예측 가능하게 동작하는가”라는 신뢰성
축으로 옮겨가고 있다는 최근 흐름과 맞닿아 있다.

## 비평

### 자체 선정한 비교 대상만으로 우위를 주장한다

Hy3의 핵심 비교 대상은 시종일관 GLM-5.1과 GLM-5.2로 한정된다.
270명 전문가 블라인드 평가에서 GLM-5.1과, 토큰 효율 비교에서
GLM-5.2와 비교하는데, 왜 하필 GLM 계열만이 비교 대상으로
선택되었는지에 대한 설명은 없다.
“2~5배 많은 파라미터를 가진 플래그십 오픈소스 모델과도 경쟁할
수준”이라는 주장 역시 구체적으로 어떤 모델을 지칭하는지
명시하지 않은 채 appendix의 벤치마크 표로 근거를 미루고 있다.

비교 대상을 선택하는 것 자체가 결과를 어느 정도 결정한다는
점에서, 이 선택적 비교는 발표 주체가 유리한 결과가 나오는
경쟁자만 골라 제시했을 가능성을 배제하지 못한다.
GLM 계열이 실제로 가장 적절한 비교군이라 하더라도, 그 이유를
설명하지 않은 채 반복적으로 GLM만을 등장시키는 서술 방식은
독자가 이 비교의 공정성을 스스로 판단할 근거를 주지 않는다.

### 내부 평가와 내부 테스트라는 표현이 반복되지만 방법론은 공개되지 않는다

환각률 12.5%→5.4%, 상식 오류율 25.4%→12.7%, 다중 턴 문제
발생률 17.4%→7.9%, WorkBuddy의 성공률 72%→90% 등 이 글의
핵심 수치는 거의 전부 “내부 평가”나 “내부 테스트”에서 나온다.
평가 데이터셋의 구성, 평가자의 수와 자격, 평가 기준의 구체적
정의는 전혀 공개되지 않는다.
270명 전문가 블라인드 평가만이 유일하게 외부 검증에 가까운
형태로 보이지만, 이 역시 전문가의 선정 기준이나 평가 프로토콜은
appendix로 미루어져 있다.

같은 팀이 만든 모델을 같은 팀이 설계한 내부 평가로 검증하고
그 결과를 개선의 증거로 제시하는 구조는, 발표자가 스스로
유리한 결과가 나오도록 평가를 설계했을 가능성을 원천적으로
배제할 수 없다.
이는 이 발표문 특유의 문제라기보다 업계 전반의 관행이지만,
“환각률이 12.5%에서 5.4%로 줄었다”는 구체적 수치가 독자에게
주는 정밀성의 인상은, 그 수치를 낳은 방법론이 검증 불가능하다는
사실과 어긋난다.

### RL 확장의 구체적 규모나 비용이 전혀 언급되지 않는다

글은 “RL 훈련 규모를 확대했다”고 여러 차례 언급하지만, 그
확대가 구체적으로 어느 정도의 컴퓨팅 자원, 훈련 시간, 데이터
양을 의미하는지는 전혀 밝히지 않는다.
1월 말 인프라 재구축을 시작해 6개월 만에 프리뷰와 정식 버전을
모두 출시했다는 타임라인은 개발 속도의 인상을 주지만, 그
속도가 어떤 규모의 투자로 뒷받침되었는지는 알 수 없다.

이 공백은 Hy3의 성과를 다른 모델과 비교해 평가하려는 독자에게
중요한 맥락을 빼앗는다.
같은 성능 개선이라도 파라미터 수를 크게 늘려 얻은 것인지,
아니면 데이터·RL 설계의 효율성만으로 얻은 것인지에 따라
그 성과의 의미는 완전히 달라진다.
글은 “비슷한 크기의 모델을 능가한다”는 주장으로 파라미터
효율성을 암시하지만, 정작 RL 훈련에 투입된 자원 규모를 밝히지
않아 그 효율성 주장을 독자가 직접 검증할 방법이 없다.

## 인사이트

### 벤치마크에서 “제품 피드백 루프”로 무게중심이 이동하는 것은 LLM 개발의 성숙 단계 진입 신호다

Hy3 발표문이 벤치마크 수치와 나란히 “50개 이상의 제품에서 수집한
피드백”을 핵심 증거로 내세우는 방식은, LLM 산업이 초기의
벤치마크 경쟁 단계를 지나 제품화 단계로 진입하고 있다는 신호로
읽을 수 있다.
초기 LLM 경쟁은 MMLU나 HumanEval 같은 고정된 벤치마크에서 몇
포인트를 앞서는가로 승부가 갈렸다.
하지만 벤치마크가 포화되고 여러 모델이 비슷한 점수대에
몰리면서, 실제 제품에 배치했을 때 드러나는 미세한 결함(도구
호출 실패, 환각, 맥락 손실)이 오히려 채택을 좌우하는 요인이
된다.

이 전환은 소프트웨어 산업의 오래된 패턴과 유사하다.
새로운 기술 범주 초기에는 “무엇을 할 수 있는가”가 경쟁의
축이지만, 기술이 성숙하면 “얼마나 안정적으로, 예측 가능하게
작동하는가”가 경쟁의 축으로 옮겨간다.
Hy3가 프론트엔드 개발, 데이터·스토리지, CI/CD 같은 실무
카테고리에서의 우위를 강조하고, 도구 호출 안정성을 별도
섹션으로 다루는 것은 LLM 경쟁이 이미 이 성숙 단계로 넘어가고
있다는 것을 보여주는 초기 사례다.
앞으로 모델 발표문에서 벤치마크 표보다 “실제 배포 사례에서
무엇이 고쳐졌는가”가 더 큰 비중을 차지하는 경향이 강해질
가능성이 크다.

### 자국 생태계 벤치마킹은 오픈소스 LLM 경쟁이 지역별로 분화되고 있음을 시사한다

Hy3가 시종일관 GLM 계열과만 비교되고, 배포처로 GitHub와
HuggingFace뿐 아니라 ModelScope와 AtomGit이라는 중국 자체
플랫폼을 나란히 명시한다는 사실은, 오픈소스 LLM 생태계가
전 세계 단일 경쟁 구도가 아니라 지역별로 독자적인 경쟁 구도를
형성하고 있다는 것을 보여준다.
GLM(Zhipu), Hy(Tencent)는 모두 중국 기업의 모델이며, 이
발표문은 사실상 중국 오픈소스 LLM 생태계 내부의 경쟁 서사로
읽힌다.

이 지역화는 단순한 우연이 아니라 구조적 요인에서 비롯된다.
데이터 접근성, 규제 환경, API 가격 정책(위안화 표시 가격 자체가
그 신호다), 주요 고객사(WorkBuddy, Yuanbao, WeChat 등 텐센트
자체 생태계)가 모두 지역 특화되어 있기 때문에, 모델 개발팀이
가장 먼저 만족시켜야 할 대상도 자연히 같은 지역의 경쟁자와
고객이 된다.
장기적으로 이런 지역화가 심화되면, 전 세계 단일 순위표로
LLM을 비교하는 방식 자체가 점점 의미를 잃고, “어느 생태계
안에서 경쟁력이 있는가”를 따로 평가해야 하는 상황이 고착될
수 있다.

### 토큰 효율성 경쟁은 모델 품질 경쟁에서 총소유비용 경쟁으로의 전환을 예고한다

WorkBuddy가 GLM-5.2 대비 문서 처리에서 47.4%, 프레젠테이션
작성에서 49% 적은 토큰을 사용했다는 수치는, 이 글에서 가장
쉽게 지나치기 쉽지만 장기적으로 가장 중요한 신호일 수 있다.
같은 작업을 더 적은 토큰으로 정확하게 완수하는 능력은
API 가격 인하(1M 토큰당 입력 1위안)와 결합되면 총소유비용을
곱절로 낮추는 효과를 낸다.
모델이 “더 똑똑한가”보다 “같은 결과를 더 싸게 내는가”가
채택 결정의 핵심 변수가 되는 것이다.

이 전환은 클라우드 컴퓨팅 산업이 겪은 경로와 유사하다.
초기 클라우드 경쟁은 “어떤 기능을 제공하는가”에 집중되었지만,
성숙기에 접어들면서 같은 기능을 얼마나 적은 비용으로
제공하는가가 핵심 경쟁 축이 되었다.
LLM 산업에서도 모델의 원시 성능이 상향 평준화되면, 다음
경쟁의 무대는 “동일 품질을 최소 토큰으로 달성하는 엔지니어링
효율”로 옮겨갈 가능성이 크다.
이 경우 모델을 선택하는 기준은 벤치마크 순위표가 아니라 실제
워크로드에서 측정한 토큰당 비용 대비 완료율이 될 것이며,
이는 지금은 부차적으로 언급되는 수치가 향후 발표문에서
주인공 자리를 차지하게 될 것이라는 예고로 읽을 수 있다.
