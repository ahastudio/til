# GLM-5 대규모 서비스 중 발견한 레이스 컨디션 버그 수정기

원문: <https://z.ai/blog/scaling-pain>

HN 토론: <https://news.ycombinator.com/item?id=47955083> (10점, 1개 댓글)

## 요약

Z.AI 엔지니어링 팀이 GLM-5 코딩 에이전트를 대규모로 서비스하는 과정에서 발견한 두 가지 심각한 인프라 버그와 그 수정 과정을 공개했다.
고동시성(high concurrency)과 장문맥(long-context) 시나리오에서 “깨진 출력, 반복 생성, 희귀 문자 생성” 현상이 간헐적으로 발생했다.
두 버그는 모두 분산 추론 시스템의 비동기 처리에서 발생하는 고전적인 동기화 문제였다.

### 버그 1: PD 아키텍처 레이스 컨디션

PD(Prefill-Decode) 분리 아키텍처에서 중단(abort) 신호가 Prefill 컴포넌트에 도달하지 못하는 버그였다.
요청이 취소되거나 타임아웃되면 Decode 측에서 중단 신호를 보내지만, 이 신호가 Prefill에 전달되기 전에 새로운 KV 캐시가 같은 메모리 영역에 할당됐다.
그 결과 RDMA 쓰기 작업이 새로 할당된 KV 캐시를 오염시켰다.
수정 후 비정상 출력 발생률이 0.1%에서 0.03% 미만으로 감소했다.

### 버그 2: HiCache 로드-사용 위반

HiCache(계층형 KV 캐시)에서 캐시 로딩과 순방향 연산(forward computation) 사이의 동기화가 누락된 버그였다.
캐시 데이터가 완전히 로드되기 전에 연산이 시작되는 read-before-ready 패턴이 발생했다.
수정 사항은 SGLang 오픈소스 프로젝트에 PR #22811로 기여됐다.

### 성능 최적화: LayerSplit

두 버그 수정과 함께 LayerSplit 기법도 적용했다.
KV 캐시를 레이어별로 여러 GPU에 분산 배치해 메모리 압박을 해소하는 방식으로, 40K-120K 토큰 요청에서 10-132%의 처리량 향상을 달성했다.

## 분석

### 분산 추론의 고유한 동기화 문제

두 버그는 LLM 추론 인프라가 일반 분산 시스템과 다른 독특한 특성을 보여 준다.
전통적인 웹 서비스에서는 요청이 취소되면 처리를 중단하고 리소스를 해제하면 된다.
하지만 PD 분리 아키텍처에서는 Prefill과 Decode가 다른 노드에서 실행되고, 중단 신호가 RDMA를 통해 고속 네트워크를 거쳐 전파되는 동안 메모리 재할당이 발생할 수 있다.
GPU 메모리는 CPU 메모리와 달리 할당 해제 패턴이 예측 가능하지 않으며, KV 캐시는 크기가 커서 재사용 압력이 높다. 이 조합이 일반적인 분산 시스템 레시피로 해결하기 어려운 버그를 만들어 낸다.

### 0.1%가 실제로 의미하는 것

0.1%라는 비정상 출력 발생률은 작아 보이지만, 대규모 서비스 맥락에서는 다르다.
초당 1만 건의 추론 요청을 처리하면 초당 10건의 오류가 발생한다.
코딩 에이전트처럼 긴 대화 맥락이 축적되는 서비스에서 오염된 KV 캐시로 인한 오류는 이후 모든 토큰 생성에 영향을 미친다.
감지하기 어려운 간헐적 오류가 왜 위험한지, 그리고 왜 0.03% 미만으로 낮추는 것이 의미 있는 목표인지를 이 수치가 설명한다.
HN에서는 비슷한 현상을 긴 컨텍스트 환경에서 직접 경험했지만 원인을 찾지 못했다는 사례가 공유됐다.[^NyxPeng]
이 포스트가 그 원인을 설명해 줬다는 반응은, serving 인프라 버그가 표면적으로는 "모델 품질 저하"로 나타나는 진단의 어려움을 시사한다.

### HiCache와 SGLang 생태계 기여

HiCache 버그 수정이 SGLang에 PR로 기여된 사실은 LLM 추론 인프라 오픈소스 생태계의 성숙을 보여 준다.
Z.AI가 내부 수정에 그치지 않고 업스트림에 기여한 것은, 같은 인프라를 공유하는 다른 팀들도 같은 버그를 겪을 수 있음을 인식했기 때문이다.
SGLang, vLLM, TensorRT-LLM 같은 추론 엔진들이 커뮤니티 주도로 발전하는 구조에서 이런 기여가 전체 생태계의 안정성을 높인다.

## 비평

### 재현 조건의 불투명성

이 버그들이 어떤 조건에서 재현되는지 정확한 명세가 공개되지 않았다.
“고동시성과 장문맥”이라는 설명은 다른 팀이 자신의 환경에서 동일한 버그가 있는지 진단하기에 충분하지 않다.
오픈소스 기여인 SGLang PR은 코드 수준의 재현 조건을 제공하지만, 블로그 포스트는 엔지니어링 스토리 중심으로 서술되어 실무 재현성 측면에서 아쉽다.

### LayerSplit 성능 수치의 범위

10-132%라는 넓은 처리량 향상 범위는 개선 효과를 예측하기 어렵게 만든다.
어떤 조건에서 10%에 그치고 어떤 조건에서 132%가 나오는지 명확한 설명이 없다.
벤치마크 방법론과 하드웨어 설정이 공개됐다면 다른 팀의 비교 평가가 가능했을 것이다.

## 인사이트

### 추론 인프라가 다음 경쟁 전선이다

LLM 모델 자체의 품질 경쟁이 치열해지면서, 추론 효율성과 안정성이 서비스 경쟁력의 핵심 변수로 부상하고 있다.
동일한 모델 가중치를 사용하더라도 추론 인프라의 최적화 수준에 따라 단위 토큰당 비용, 지연시간, 처리량이 크게 달라진다.
Z.AI가 이 버그 수정 경험을 공개한 것은 기술력 시그널링의 의미도 있다.
모델 벤치마크 점수가 아니라 실제 서비스 환경에서 만나는 문제와 해결 과정이 인프라 신뢰성을 판단하는 근거가 된다.

### 분산 시스템의 영원한 적: 타이밍 의존성

두 버그는 모두 타이밍 의존성(timing dependency)의 변종이다.
“A가 완료되기 전에 B가 시작된다”는 패턴은 단일 프로세스에서는 순서를 강제하면 해결되지만, 네트워크를 건너는 분산 시스템에서는 순서 보장 자체가 비용이 된다.
LLM 추론처럼 레이턴시가 극도로 중요한 환경에서 “모든 것을 동기화”하면 성능이 급락한다.
버그 1의 수정이 “Prefill에 중단 신호가 도착한 것을 확인한 후에만 메모리를 재할당한다”는 명시적 동기화 추가였다면, 이 동기화가 얼마나 신중하게 설계됐는지가 핵심이다. 불필요한 동기화 포인트가 하나만 늘어도 전체 처리량에 측정 가능한 영향을 줄 수 있다.

### 오픈소스 생태계와 지식 공유의 선순환

SGLang PR 기여는 경쟁 관계에 있는 LLM 서비스 기업들이 인프라 레이어에서 협력하는 흥미로운 역학을 보여 준다.
제품 차별화는 모델, UX, 비즈니스 로직에서 이루어지고, 기반 인프라 버그는 커뮤니티가 함께 고치는 분업이 자연스럽게 형성되고 있다.
이 패턴은 리눅스 커널, 웹 브라우저 엔진에서 반복된 역사다. 경쟁사들이 공통 기반 기술에 기여하며 파이를 키우고, 그 위에서 차별화를 경쟁하는 구조가 AI 추론 인프라에서도 같은 방식으로 전개되고 있다.
