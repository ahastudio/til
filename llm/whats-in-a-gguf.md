# GGUF에는 가중치 외에 무엇이 들어 있나

원문: <https://nobodywho.ooo/posts/whats-in-a-gguf/>

HN 토론: <https://news.ycombinator.com/item?id=48138332> (191점, 55개 댓글)

## 요약

GGUF는 llama.cpp가 채택한 언어 모델 파일 포맷으로,
핵심 설계 철학은 “모델 실행에 필요한 모든 것을 단일 파일에 담는다”는 것이다.
가중치(weights)만 저장하는 단순한 바이너리가 아니라,
모델이 올바르게 작동하기 위한 부가 정보들을 함께 포함한다.

현재 GGUF에 포함되는 주요 요소는 네 가지다.
첫째, **채팅 템플릿(chat template)**이다.
Jinja2로 작성된 이 템플릿은 대화 형식을 정의하며,
도구 호출, 멀티미디어 메시지 인코딩 같은 복잡한 기능도 처리할 수 있다.
Gemma4와 LFM2처럼 모델마다 서로 다른 구조를 사용하므로,
포맷에 템플릿을 포함하는 것은 호환성 측면에서 중요한 결정이다.

둘째, **특수 토큰(special tokens)**이다.
생성을 종료시키는 `<eos>` 토큰, 도구 호출 시작/종료 토큰,
대화 턴(turn) 경계 토큰 등이 여기에 정의된다.
이 토큰들을 올바르게 처리하지 않으면 모델이 의도치 않게 작동한다.

셋째, **샘플러 구성(sampler configuration)**이다.
토큰 분포에서 다음 토큰을 선택하는 방식(temperature, top-p, top-k 등)을
권장 파라미터로 메타데이터에 저장할 수 있다.
최근 GGUF 표준에는 샘플러 체인 순서를 명시하는 기능도 추가되었다.

넷째, **메타데이터(metadata)**다.
모델 구성, 아키텍처 정보, 레이어 수 등 추론 엔진이 필요로 하는 구성 정보가 중앙집중식으로 저장된다.

그러나 현재 GGUF가 아직 담지 못하는 영역들도 있다.
도구 호출 형식이 Qwen3, Gemma4마다 제각각이어서 각 추론 엔진이 개별적으로 파싱해야 하고,
멀티모달 기능은 별도의 GGUF 파일이 필요해 단일 파일 원칙이 깨진다.
또한 모델이 이미지 처리, 사고(thinking) 블록, 도구 호출 등을 지원하는지
메타데이터에서 자동으로 감지할 표준화된 방법이 없다.

## 분석

### 단일 파일 철학의 실용적 가치

GGUF 이전의 모델 배포는 JSON 구성 파일, 토크나이저 파일, 가중치 파일이 분리되어 있어
버전 불일치 문제가 빈번했다.
단일 파일에 모든 것을 담는 GGUF는 이 문제를 구조적으로 해결한다.
채팅 템플릿을 파일 안에 포함하는 결정은 특히 중요하다.
모델과 템플릿 버전이 항상 일치하므로, 추론 엔진이 외부에서 템플릿을 관리할 필요가 없다.

GGUF 스펙 설계자인 Philpax는[^philpax-single] "단일 파일 배포는 의도적인 설계 목표였다"고
밝히며, safetensors가 당시 llama.cpp의 새로운 양자화 형식을 지원하지 못했고,
JSON 리더를 실행 엔진에 강제하지 않으려는 이유도 있었다고 설명했다.
한편 Sharlin은[^sharlin] 이미지 생성 분야에서는 단일 파일이 오래된 표준이었다며,
GGUF의 접근이 LLM에만 새로운 것임을 지적했다.

### 표준화의 미완성

도구 호출 형식의 분산은 현재 오픈소스 LLM 생태계의 파편화를 상징한다.
각 모델 패밀리가 독자적인 도구 호출 형식을 사용하는 상황에서,
이를 하드코딩으로 파싱하는 추론 엔진들은 새 모델이 등장할 때마다 업데이트가 필요하다.
GGUF가 도구 호출 형식을 표준화한다면 이 유지보수 부담을 크게 줄일 수 있다.

theapadayo는[^theapadayo] 이보다 더 근본적인 문제를 지적했다.
모델 아키텍처 자체를 GGUF 외부에서 하드코딩 없이 정의할 방법이 없다는 것이다.
그는 벤더의 day-1 검증 지원 여부가 사용자의 모델 품질 인식에 결정적 영향을 미친다고 보고,
DSL로 모델 그래프를 기술하거나 PyTorch 모듈을 GGML 연산으로 변환하는 방안을 제안했다.
monocasa도[^monocasa] 같은 맥락에서 "GGUF는 계산 그래프 자체가 아니라
아키텍처 문자열과 파라미터 메타데이터만 저장하므로,
새 아키텍처마다 소비 측 코드 변경이 불가피하다"고 짚었다.

## 비평

이 글은 GGUF의 현황을 균형 있게 서술하며,
현재 포함된 내용과 아직 부족한 부분을 함께 조망한다.
기능 플래그 부재에 대한 지적은 특히 실용적이다.
모델이 어떤 기능을 지원하는지 메타데이터에서 확인할 수 없다면,
추론 엔진은 시도와 오류 방식으로 동작해야 하거나 별도의 모델 카탈로그에 의존해야 한다.

멀티모달 모델의 프로젝션 파일 분리 문제는 단순한 불편함이 아니다.
이것은 “단일 파일로 모든 것을”이라는 핵심 원칙과 충돌하며,
멀티모달 모델이 주류가 되는 시점에서 GGUF의 설계 한계가 드러날 수 있다.

실제로 badsectoracula는[^badsectoracula] 비전 지원 모델을 내려받았을 때
GGUF 하나만으로는 실행이 안 된다는 것을 뒤늦게 알고,
“GGUF가 모든 것을 담기로 한 것 아니었나”라고 반문했다.
Philpax는[^philpax-mmproj] 프로젝션 모델이 별도 파일로 분리된 결과에 대해
아쉬움을 표명하며, 두 파일을 합치려는 시도가 있기를 바란다고 밝혔다.
한편 intothemild는[^intothemild] MTP(Multi-Token Prediction) 지원 논의 과정에서
MTP 모델을 별도 파일로 분리하자는 제안이 거부된 사례를 들며,
같은 논리로 mmproj를 포함하는 방향도 열려 있다고 전했다.

채팅 템플릿 가독성에 대해서는 amelius가[^amelius] Jinja2 기반 템플릿 예시를 보고
“XML보다 읽기 어려운 포맷”이라며 비판했다.
이에 aktuel은[^aktuel] 이 포맷이 사람이 읽으려고 설계된 것이 아니라,
인터넷에서 수집한 임의 텍스트와 혼동되지 않도록 고유성을 확보하기 위한 것이라고 반론했다.

## 인사이트

### 포맷이 생태계를 통일한다

GGUF 이전에는 GGML, GPTQ, AWQ 등 다양한 포맷이 경쟁했으며,
각 포맷마다 도구 체인이 분리되어 있었다.
GGUF의 성공은 단순한 기술적 우수성보다 llama.cpp라는 강력한 생태계 중심점이 있었기 때문이다.
이는 표준 포맷의 승리가 기술적 최적해보다 생태계 견인력에 달려 있음을 보여준다.
파일 포맷 설계의 결정들—단일 파일 원칙, 채팅 템플릿 포함, 메타데이터 구조—이
사실상 표준이 되는 과정은, 기술 표준화의 전형적인 경로를 따른다.

uyzstvqs는[^uyzstvqs] "llama.cpp, whisper.cpp, stable-diffusion.cpp 같은 프로젝트가
다양한 플랫폼과 하드웨어 백엔드에서 그냥 작동한다는 점이 GGML·GGUF가 오픈소스 ML/AI
생태계에 미친 가장 큰 기여"라고 평가했다.

### 모델 자급자족성의 의미

채팅 템플릿을 모델 파일 안에 포함하는 결정은 깊은 함의를 갖는다.
이것은 모델을 “외부 구성이 필요 없는 자급자족 단위”로 만드는 설계다.
컨테이너 이미지가 실행에 필요한 모든 것을 담아 환경 의존성을 줄인 것처럼,
GGUF는 모델 자체가 실행 컨텍스트를 정의하게 한다.
이 방향이 확장될수록 모델은 더욱 이식성이 높아지고,
추론 엔진은 모델에 대한 사전 지식 없이도 올바르게 실행할 수 있게 된다.
기능 플래그, 도구 호출 형식, think 토큰 정보가 표준화되어 포함된다면,
추론 엔진과 애플리케이션은 모델에 대해 더 적은 가정을 하고도 동작할 수 있다.

### 계산 그래프 부재가 남기는 장기 부채

GGUF가 계산 그래프를 포함하지 않는 설계는 현실적 트레이드오프였다.
Philpax는[^philpax-graph] MVP 스펙을 빠르게 내놓는 것을 우선시하면서도
계산 그래프를 포함할 여지를 의도적으로 남겨 두었다고 밝혔다.
LoganDark는[^logandark] ONNX처럼 가중치에 계산 그래프를 포함하고
공통 인터페이스를 Wayland 방식의 확장으로 다루면 트랜스포머 외에도
RWKV 같은 RNN 계열 모델을 지원할 수 있다는 아이디어를 제안했다.
다만 "계산 그래프가 파일에 고정되면 가중치 변경 없이 아키텍처를 개선할 때마다
기존 파일 변환이 필요해진다"는 우려도 함께 제기했다.

### 메타데이터 표준화가 AI 애플리케이션 개발을 바꾼다

현재 개발자들은 특정 모델의 기능을 알기 위해 문서를 읽거나
커뮤니티 정보에 의존해야 한다.
모델이 도구 호출을 지원하는지, 이미지를 처리할 수 있는지,
어떤 사고(thinking) 패턴을 사용하는지를 파일에서 직접 읽을 수 있다면,
모델 선택과 애플리케이션 구성을 자동화할 수 있다.
이것은 모델 카탈로그, 라우팅 시스템, 에이전트 프레임워크가 더 동적으로 동작할 수 있는 토대다.
GGUF의 메타데이터 표준화는 단순한 편의 기능이 아니라,
모델을 프로그래밍 가능한 단위로 만드는 인프라 레이어의 진화다.
