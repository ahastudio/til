# whichllm

<https://github.com/Andyyyy64/whichllm>

## 소개

`whichllm`은 현재 하드웨어에서 실제로 실행 가능하고 최고 성능을 내는 로컬 LLM을
찾아주는 Python CLI 도구다. 단순히 VRAM에 들어가는 모델을 나열하는 것이 아니라,
파라미터 수가 아닌 실측 벤치마크 기준으로 모델을 점수화해 순위를 매긴다.

NVIDIA, AMD, Apple Silicon, CPU 전용 하드웨어를 자동 감지하며, HuggingFace에서
모델 데이터를 실시간으로 가져온다. 설치는 `uvx whichllm` 한 줄로 가능하다.

## 주요 기능

**하드웨어 자동 감지 및 랭킹**: GPU/CPU/RAM을 감지해 실행 가능한 모델을 VRAM 적합성,
속도, 벤치마크 품질 기준으로 0~100점으로 점수화한다.

**멀티 벤치마크 병합**: LiveBench, Artificial Analysis, Aider, Arena ELO를
병합해 단일 점수를 산출한다. 증거 신뢰도(직접 측정/변형/보간/자가보고)도 함께
표시한다.

**원커맨드 워크플로**: `whichllm run “qwen 2.5 1.5b”`로 모델을 다운로드하고 대화를
시작한다.

**GPU 시뮬레이션**: `--gpu “RTX 4090”` 옵션으로 구매 전 특정 하드웨어에서의
추천 결과를 미리 확인할 수 있다.

**하드웨어 플래닝**: `whichllm plan “llama 3 70b”`로 특정 모델을 실행하기 위한
GPU 요구사항을 역산한다.

**Python 코드 생성**: `whichllm snippet “qwen 7b”`로 모델을 사용하는 즉시 실행
가능한 Python 코드를 생성한다.

## CLI

```bash
whichllm                      # 하드웨어 자동 감지 후 랭킹 출력
whichllm --gpu "RTX 4090"     # 특정 GPU 시뮬레이션
whichllm run "qwen 2.5 1.5b"  # 모델 다운로드 및 대화 시작
whichllm plan "llama 3 70b"   # 실행에 필요한 GPU 요구사항 확인
whichllm snippet "qwen 7b"    # Python 코드 스니펫 생성
```

## 랭킹 알고리즘

모델 점수는 다음 요소를 종합해 산출한다.

- 병합 벤치마크(LiveBench, Artificial Analysis, Aider, Arena ELO)
- 모델 크기(log₂ 스케일 지식 대리 지표)
- 양자화(quantization) 페널티
- 증거 신뢰도(직접/변형/보간/자가보고)
- 런타임 적합 유형(풀 GPU/부분 오프로드/CPU 전용)
- 속도 추정값(신뢰도 마커 포함)

RTX 4090 기준 상위 추천 예시: “Qwen3.6-27B · Q5_K_M · score 92.8 · ~27 t/s”

실제 사용자 환경에서의 [GeekNews 댓글](https://news.hada.io/topic?id=29613)을
보면, 상위 5개 추천 중 3개가 Qwen 계열이었다는 보고가 있다(80B, 27B, 235B
파라미터 모델). AMD RYZEN AI MAX+ 395 시스템에서도 상위 10개 목록을 잘 생성한
다는 사례도 공유됐다.

## 아키텍처

핵심 모듈(`src/whichllm/`):

- `cli.py`: Typer 기반 커맨드 인터페이스
- `hardware/`: GPU/CPU/RAM 감지 및 시뮬레이션
- `models/`: HuggingFace API 연동 및 벤치마크
- `engine/`: VRAM 계산, 호환성 검사, 성능 추정
- `output/`: Rich 테이블 포매팅 및 JSON 내보내기

Python 3.11+ 필요. NVIDIA GPU 감지는 `nvidia-ml-py`가 자동 포함된다.

## 분석

### “내 하드웨어에서 실제로 잘 돌아가는 모델”이라는 문제

로컬 LLM 선택의 어려움은 정보의 부재가 아니라 정보의 과잉에 있다. HuggingFace
에는 수십만 개의 모델이 있고, 각 모델의 성능 지표는 서로 다른 벤치마크로 측정
된다. 이 중에서 “내 GPU 8GB에서 실제로 빠르게 돌아가면서 코딩을 잘 하는 모델”을
찾는 것은 다차원 최적화 문제다.

`whichllm`이 이 문제를 어떻게 공략하는지가 핵심이다. 단순 필터링(VRAM 이하)이
아니라 합성 점수를 계산한다. VRAM 적합성, 벤치마크 품질, 양자화 페널티, 속도
추정을 하나의 숫자로 압축한다. 이 압축 과정에서 정보 손실이 발생하지만, 사용자
가 실제로 필요한 것은 “다음에 무엇을 시도해볼 것인가”라는 행동 가능한 답이다.
복잡성을 숨기고 결론을 주는 것이 이 도구의 설계 철학이다.

### 증거 신뢰도를 점수와 함께 표시하는 설계

많은 추천 시스템은 점수만 보여주고 그 점수가 얼마나 신뢰할 수 있는지는 숨긴다.
`whichllm`이 “직접 측정/변형/보간/자가보고”로 신뢰도를 구분해 함께 표시하는 것
은 드문 설계 선택이다. 이것은 사용자가 “이 점수는 실제 측정값이 아니라 추정값
이다”를 알고 결정을 내릴 수 있게 한다.

이 투명성은 단기적으로 사용자를 불안하게 만들 수 있다. 그러나 추천 결과가 기대
와 다를 때 “이 모델은 자가보고 점수였으니 실제로는 다를 수 있다”는 이해를 가능
하게 한다. 불확실성을 숨기는 것보다 드러내는 것이 장기 신뢰 구축에 유리하다.

## 비평

### 벤치마크 병합의 신뢰성 문제

LiveBench, Aider, Arena ELO는 측정 대상과 방법론이 다르다. 이것을 단일 점수로
병합할 때 어떤 가중치를 쓰는지, 가중치 선택 근거가 무엇인지가 중요하다. 현재
README에는 이 부분이 명시적으로 설명되어 있지 않다. “merged benchmarks”라는
표현만으로는 사용자가 점수의 의미를 충분히 이해하기 어렵다.

특히 코딩, 수학, 일반 지식은 서로 다른 역량이다. 코딩에 특화된 모델과 일반 지식
이 강한 모델이 같은 점수를 받을 수 있다. 저자는 `--task` 옵션으로 general,
coding, vision, math 프로파일을 지원하지만, 이 프로파일이 벤치마크 가중치를
어떻게 바꾸는지는 불투명하다.

### GPU 시뮬레이션의 실용성과 한계

구매 전 특정 GPU를 시뮬레이션할 수 있다는 기능은 매력적이다. 그러나 VRAM 계산과
속도 추정은 실제 하드웨어 환경의 복잡성을 완전히 반영하지 못한다. PCIe 대역폭,
메모리 타입(GDDR6 vs GDDR6X vs HBM), 드라이버 버전, 운영체제 오버헤드 등이
실제 성능에 영향을 준다. 시뮬레이션 결과와 실제 성능 사이에 차이가 있을 수 있고,
이 차이를 도구가 얼마나 잘 처리하는지는 검증이 필요하다.

## 인사이트

### 로컬 LLM 생태계의 선택 마비가 도구를 낳는다

클라우드 LLM 시장은 소수의 강자(OpenAI, Anthropic, Google)가 이끈다. 로컬 LLM
시장은 다르다. 오픈소스 모델만 수만 개이고, 양자화 방법마다 트레이드오프가 다르며,
하드웨어 조합에 따라 최선의 선택이 달라진다. 이 복잡성이 `whichllm` 같은 도구의
존재 이유다.

이 복잡성은 앞으로 더 커질 가능성이 높다. 모델이 늘고, 양자화 기법이 다양해지고,
새로운 아키텍처가 등장할수록, “어떤 모델을 쓸 것인가”라는 결정의 비용이 올라간다.
개인 사용자에게 이 결정을 직접 내리게 하는 것보다, 추상화 레이어를 두어 “내
환경에 맞는 최선”을 자동으로 찾는 방향이 로컬 LLM 생태계의 성장을 가속화할 것이다.

### “지금 당장 시도해볼 수 있는 것”이라는 설계 철학

`whichllm run` 명령은 추천된 모델을 즉시 다운로드해 대화를 시작한다. 이것은 사
소해 보이지만 중요한 설계 선택이다. 추천과 실행 사이의 마찰을 제거하면, 사용자가
더 많이 시도하고 더 빠르게 피드백을 얻는다. “설치 방법 찾기→다운로드→설정→실행”
이라는 여러 단계를 하나의 명령으로 압축하면 진입 장벽이 낮아진다.

이 접근은 로컬 LLM 실험을 “기술적으로 숙련된 사람만 하는 것”에서 “CLI에 익숙한
누구나 하는 것”으로 이동시킨다. 도구가 복잡성을 흡수하는 만큼 사용자 저변이 넓어진다.
