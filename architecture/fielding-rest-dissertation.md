# 필딩의 REST 논문: 아키텍처 스타일과 네트워크 기반 소프트웨어 아키텍처의 설계

<https://ics.uci.edu/~fielding/pubs/dissertation/top.htm>

## 요약

로이 필딩(Roy Thomas Fielding)은 2000년 캘리포니아대학교 어바인 캠퍼스(UCI)에서
“아키텍처 스타일과 네트워크 기반 소프트웨어 아키텍처의 설계”라는 제목의 박사 학위
논문을 발표했다. 이 논문은 소프트웨어 아키텍처를 아키텍처 스타일(architectural
style)이라는 개념적 도구로 이해하는 프레임워크를 제시하고, 그 도구를 사용해 현대
웹의 아키텍처를 설명하는 것을 핵심 목적으로 삼는다. 논문의 결과물이 표현 상태 전달
(Representational State Transfer, REST)이다.

논문의 전반부(1~3장)는 소프트웨어 아키텍처 이론의 기초를 다진다. 아키텍처 스타일을
“제약 조건의 이름 붙인 집합”으로 정의하고, 파이프-필터, 클라이언트-서버, 분산 객체
등 기존의 네트워크 기반 아키텍처 스타일들을 성능, 확장성, 수정 용이성 등의 속성에
따라 분류한다. 이 분류는 이후 REST를 파생시키는 근거가 된다.

4장은 월드와이드웹(WWW)의 도메인 요구사항을 기술한다. 낮은 진입 장벽, 확장 가능성,
분산 하이퍼미디어, 인터넷 규모(Internet-scale)라는 네 가지 요구를 제시하며, 특히
“무정부적 확장성(anarchic scalability)”과 “독립적 배포(independent deployment)”를
인터넷 환경의 핵심 특성으로 강조한다. 조직 경계를 넘나드는 수천만 개의 컴포넌트가
서로 다른 속도로 진화하면서도 상호운용성을 유지해야 한다는 조건이 REST의 설계를
이끈 실질적 동기였다.

5장이 논문의 핵심이다. 필딩은 REST를 “빈 스타일(null style)”에서 출발해 제약 조건을
하나씩 추가하는 방식으로 파생시킨다. 최종적으로 여섯 개의 제약 조건이 REST를 구성한다.
클라이언트-서버 분리(client-server), 무상태(stateless), 캐시(cache), 균일한 인터페이스
(uniform interface), 계층화된 시스템(layered system), 주문형 코드(code-on-demand).
이 중 code-on-demand만 선택적이다. 균일한 인터페이스 제약은 다시 네 가지로 세분된다:
리소스 식별(identification of resources), 표현을 통한 리소스 조작(manipulation of
resources through representations), 자기 서술적 메시지(self-descriptive messages),
하이퍼미디어를 애플리케이션 상태 엔진으로(hypermedia as the engine of application
state, HATEOAS).

6장은 REST가 실제로 HTTP/1.1과 URI 표준 설계에 어떻게 적용되었는지를 기술한다.
필딩은 HTTP 작업 그룹의 주요 저자이기도 했으며, REST는 그 표준화 과정을 안내하는
“산 증인(acid test)”이었다. 이 장은 또한 현대 웹 아키텍처에서 REST 원칙을 위반하는
사례들도 솔직하게 기술하는데, 쿠키 기반 세션 상태, HTTP를 방화벽 우회 터널로 사용하는
관행 등이 그 예다.

## 분석

### REST는 발명이 아니라 발견이다

이 논문에서 가장 자주 놓치는 맥락은 REST가 처음부터 새로운 것을 설계하려는 시도가
아니었다는 점이다. 필딩은 이미 성공한 시스템인 웹이 왜 작동하는지를 사후에 설명하려
했다. 그는 1993년부터 HTTP와 URI 표준화에 직접 참여해온 당사자로서, 자신이 만들어온
결정들을 일관된 이론적 언어로 정리하는 작업을 수행한 것이다.

이 역방향 방법론은 논문이 갖는 설명력의 원천이기도 하지만 동시에 그 한계이기도 하다.
REST의 제약 조건들은 “이래야 한다”는 처방이 아니라 “이러해서 잘 됐다”는 귀납의
산물이다. 필딩 스스로가 설계 방법론을 “반복적이고 비순차적(non-sequential, iterative)”
이라고 기술하며, 6년에 걸쳐 모델을 수정해 나갔다고 밝힌다. 논문의 우아한 서술 구조,
즉 null style에서 출발해 제약 조건을 한 겹씩 쌓아가는 방식은 실제 설계 과정이 아니라
사후 재구성이다.

### 아키텍처를 “제약 조건의 집합”으로 보는 시각

이 논문이 소프트웨어 공학에 기여한 진정한 가치는 REST 그 자체보다 아키텍처를 논증
하는 방식에 있다. 필딩은 아키텍처 스타일을 “제약 조건이 컴포넌트에 유도하는 속성의
집합”으로 정의함으로써, 설계 결정을 절충점(trade-off)의 언어로 분석할 수 있게 한다.
무상태 제약은 가시성(visibility), 신뢰성(reliability), 확장성(scalability)을 유도하지만,
동시에 반복적인 데이터 전송이라는 비효율을 낳는다. 캐시 제약은 성능을 향상시키지만
오래된 데이터 문제를 야기한다. 이 논증 방식은 단순한 “좋은/나쁜” 판단을 피하고 각
제약의 비용과 이점을 명시적으로 연결한다.

이 방법론적 기여는 REST 자체의 수명보다 길다. 아키텍처를 속성의 유도 관계로 기술하는
방식은 이후 소프트웨어 아키텍처 교육과 문서화에서 광범위하게 차용되었다.

### 인터넷 규모라는 맥락이 제약 조건을 정당화한다

REST의 제약 조건들, 특히 무상태와 균일한 인터페이스는 단순히 “좋은 설계”의 추상적
원칙이 아니다. 이것들은 조직 경계를 초월한 수억 개 컴포넌트의 독립 배포라는 극단적
조건에서 도출된 것이다. 필딩이 “무정부적 확장성”이라 부른 상황에서는 서버가 클라이언트
를 신뢰할 수 없고, 클라이언트는 서버의 내부 상태를 알 수 없으며, 둘 사이의 어느 지점
에서 임의의 프록시가 요청을 처리할 수 있다.

이 맥락 없이 REST의 제약 조건을 단일 조직 내의 서비스 간 통신에 그대로 적용하면,
비용(무상태로 인한 반복 전송, 균일한 인터페이스의 비효율)은 그대로인데 이점(규모, 독립
배포)은 거의 없는 상황이 된다. 논문이 다루는 문제 영역과 오늘날 “REST API”가 쓰이는
문제 영역 사이의 괴리가 여기서 발생한다.

### HATEOAS의 중심성과 그 저평가

필딩은 균일한 인터페이스 제약 중 HATEOAS를 REST를 다른 아키텍처 스타일과 근본적으로
구별하는 핵심 요소로 다룬다. 애플리케이션 상태가 서버에서 받은 표현(representation)
에 포함된 하이퍼미디어 링크를 통해 전이된다는 이 개념은 클라이언트가 서버의 URL 구조를
미리 알지 못해도 된다는 것을 의미한다. 이것이야말로 웹 브라우저가 작동하는 방식이며,
웹이 수십 년 동안 URL 구조를 바꾸면서도 작동한 이유다.

그러나 HATEOAS는 논문 내에서 충분히 구체화되지 않았다. 어떤 링크 관계를 어떻게 표현
해야 하는지, 클라이언트가 링크를 어떻게 따라가야 하는지에 대한 사양은 논문 범위 밖으로
남겨졌다. 이 빈자리가 이후 수십 년간의 오해와 논쟁의 온상이 되었다.

## 비평

### 논문의 서술 범위와 산업의 적용 범위가 어긋난다

이 논문은 HTTP와 URI라는 인터넷 표준의 설계를 안내하기 위해 작성되었다. 필딩의 청중은
IETF 작업 그룹이었고, 그의 설계 결정은 수억 개의 독립된 소프트웨어 컴포넌트가 상호운용
해야 하는 프로토콜 레벨의 문제를 다루고 있었다. 논문이 암묵적으로 가정하는 스케일, 신뢰
경계, 진화 주기가 모두 이 맥락을 반영한다.

그러나 논문은 이 적용 범위를 명시적으로 한정하지 않았다. “네트워크 기반 애플리케이션
소프트웨어 아키텍처”라는 논문 제목은 HTTP 표준화보다 훨씬 넓은 의미로 읽힌다. 이
모호함이 REST를 단일 조직 내의 서비스 간 통신이나 단일 팀이 관리하는 API에 무비판적으로
적용하는 산업 관행으로 이어졌다. 필딩은 이 문제를 인식했는지 몰라도, 논문은 그 경계를
명확히 그어주지 않는다.

### “리소스” 정의의 철학적 웅장함이 실용적 명확성을 잠식한다

5장의 리소스 정의는 수학적으로 정교하다. 리소스를 “시간에 따라 변화하는 멤버십 함수”로
정의함으로써 URL이 특정 문서가 아니라 개념적 대상을 가리킨다는 점을 포착한다. 예를 들어
“저자의 선호 버전”과 “학회 X에 게재된 버전”은 같은 내용을 가리키더라도 별개의 리소스다.

그러나 이 정의는 실제 API 설계에서 “무엇을 리소스로 볼 것인가”라는 질문에 대한 실용적
지침을 제공하지 않는다. 어떤 정보나 개념도 리소스가 될 수 있다는 선언은 사실상 아무것도
리소스가 되지 말아야 한다는 제약을 두지 않는다는 뜻이다. 산업에서 벌어진 “액션을 URL에
어떻게 모델링할 것인가” 논쟁(예: `/payments` vs `/transfer-money`)은 이 정의의 철학적
폭이 실용적 지침의 공백을 낳은 결과다.

### REST는 완성된 설계가 아니라 진행 중인 가설이었다

논문의 6장은 실제로 REST 원칙과 충돌하는 웹의 사례들을 다수 기술한다. 쿠키 기반 세션
은 무상태 제약을 위반한다. HTTP를 방화벽 우회 터널로 쓰는 관행은 균일한 인터페이스를
훼손한다. 필딩은 이 불일치를 “아키텍처 불일치(architectural mismatch)”라 부르며 솔직하게
인정한다.

그러나 이 인정이 논문에서 이끌어내야 할 결론은 표면적으로 보이는 것보다 더 불편하다.
REST는 이미 배포된 웹이 완벽히 구현하고 있는 아키텍처가 아니라, 웹이 지향해야 할
이상적 모델이었다. 이 간극이 존재한다는 사실은 REST가 단일하고 정합적인 아키텍처
원칙이 아니라 부분적으로만 실현된 일련의 목표라는 것을 시사한다. 후대 산업이 REST를
하나의 완성된 규범으로 취급하면서 이 미완성의 성격을 지운 것은 문제적이다.

### HATEOAS는 제안이었지 검증된 실천이 아니었다

논문이 HATEOAS를 REST의 핵심 제약으로 제시하지만, 이를 실제로 구현한 대규모 사례는
논문 작성 당시 웹 브라우저와 HTML의 경우뿐이었다. 웹 브라우저는 미리 정해진 URL 없이
서버가 제공하는 링크를 따라 탐색한다는 의미에서 HATEOAS의 모범적 구현이다. 그러나
프로그래밍 방식의 API 클라이언트에서 HATEOAS가 가져다주는 실질적 이점은 논문에서
검증되지 않았다.

이후 25년의 경험은 이 질문에 냉담한 답을 제시했다. HATEOAS를 완전히 구현한 API는
거의 존재하지 않으며, 존재하더라도 클라이언트가 동적으로 링크를 탐색하기보다는 여전히
API 문서를 읽고 URL을 하드코딩하는 패턴이 지배적이다. 논문은 HATEOAS의 이점을 인터넷
규모의 독립적 배포 맥락에서 설득력 있게 논증했지만, 그 이점이 단일 API-클라이언트 쌍에서도
성립한다는 증거는 제시하지 못했다.

## 인사이트

### 산업이 읽은 것은 논문이 아니라 논문의 그림자였다

필딩의 논문이 출판된 후 수 년 동안 이 논문을 직접 읽은 개발자는 거의 없었다. 대신
이 논문에서 파생된 블로그 게시물, 강연, 격언들이 유통되었고, 그 과정에서 논문의 핵심이
체계적으로 탈락했다. “GET/POST/PUT/DELETE로 CRUD를 표현하라”는 식의 규칙이 REST의
동의어가 되었고, HATEOAS는 “선택 사항”으로 강등되었다. 필딩은 2008년 블로그 게시물에서
“HTTP를 통해 CRUD를 하는 것은 REST가 아니다”라고 직접 반론을 썼지만, 그 글 역시 논문
만큼이나 무시되었다.

이 현상은 소프트웨어 공학에서 아이디어가 확산되는 방식에 관한 구조적 패턴을 보여준다.
복잡한 원본 개념은 수용 과정에서 단순화되고, 단순화된 버전이 제도화된 후에는 원본이
오히려 수정주의적 해석으로 보인다. “진짜 REST”를 주장하는 필딩 자신이 산업이 만들어낸
“REST”에 의해 이단 취급을 받는 역설적 구조가 성립했다. 이는 이론이 실천에 선행할 때보다
사후에 정당화될 때 훨씬 쉽게 오용된다는 점을 시사한다.

### 무상태 원칙은 애플리케이션이 아니라 프로토콜 수준에서 작동했다

REST의 무상태 제약이 달성한 확장성은 HTTP 프로토콜 수준에서의 성취다. HTTP 서버가
요청 간에 상태를 유지하지 않기 때문에 로드 밸런서가 요청을 임의의 서버에 라우팅할 수
있고, CDN이 응답을 캐시할 수 있다. 이 수준에서 무상태는 인터넷 규모의 웹을 가능하게
한 핵심 설계 결정이다.

그러나 이 원칙을 애플리케이션 레이어에 적용하면 의미가 달라진다. 실제 사용자가 상호작용
하는 애플리케이션은 상태가 있다. 쇼핑 카트, 사용자 세션, 멀티스텝 폼, 롱폴링 기반 실시간
업데이트 모두 상태를 전제한다. 필딩도 이를 알았기에 “세션 상태는 클라이언트에 유지된다”는
표현을 사용했지만, 이것은 상태를 제거한 것이 아니라 위치를 이동한 것이다. 현대의 단일
페이지 애플리케이션(SPA)이 클라이언트에 방대한 상태를 유지하는 것은 REST의 원칙을
이행한 결과이면서 동시에 그 원칙이 애플리케이션 복잡성을 해결하지 못한다는 증거이기도
하다.

### 논문의 기여는 REST가 아니라 아키텍처 추론 방법론이다

이 논문을 REST 명세서로 읽으면 절반만 이해한 것이다. 논문의 더 지속적인 기여는 1~3장에
있다. 아키텍처 스타일을 제약 조건과 속성의 관계로 분석하는 방법론은 소프트웨어 공학
분야에서 독립적인 가치를 지닌다. 특정 아키텍처 결정이 “왜” 좋은지를 그 결정이 유도하는
시스템 속성으로 설명하는 방식은, 설계 문서에서 “우리는 마이크로서비스를 사용한다”고만
적는 관행보다 훨씬 엄밀하다.

이 방법론적 기여는 특히 아키텍처 결정 기록(ADR, Architecture Decision Record) 문화와
연결된다. ADR의 핵심은 결정이 아니라 그 결정의 맥락, 고려된 대안, 그리고 수용된 절충점
을 기록하는 것이다. 필딩의 “제약 조건 → 유도되는 속성 → 발생하는 절충점”이라는 서술
구조는 ADR의 이상적인 형식과 정확히 일치한다. 논문의 3장에서 17개의 아키텍처 스타일을
다섯 가지 속성 차원으로 평가한 표는 단순한 분류가 아니라 설계 공간을 체계적으로 탐색하는
방법의 예시다.

### REST가 배제한 것이 이후 20년의 문제를 예고했다

논문은 분산 하이퍼미디어라는 특정 문제를 위한 아키텍처를 기술했다. 이 선택은 몇 가지
중요한 문제 유형을 의도적으로 범위 밖에 두었다. 실시간 양방향 통신, 트랜잭션 의미론,
복잡한 쿼리, 스트리밍 데이터가 그것이다.

흥미롭게도 이 배제된 문제들은 2000년대 이후 웹 애플리케이션의 핵심 과제가 되었다.
WebSocket은 REST의 무상태·요청-응답 모델로 다룰 수 없는 실시간 통신을 위해 등장했고,
GraphQL은 REST의 균일한 인터페이스가 클라이언트의 데이터 요구를 충족시키지 못한다는
불만에서 탄생했으며, gRPC는 REST의 대규모 텍스트 표현이 서비스 간 통신에서 불필요한
오버헤드를 낳는다는 관찰에서 비롯되었다. 이것들은 REST를 “수정”하거나 “개선”한 것이
아니라, 논문이 처음부터 풀려 하지 않았던 다른 문제를 푸는 다른 아키텍처 스타일들이다.
REST를 단일한 API 설계 표준으로 취급한 산업은 그 경계를 넘는 문제를 만날 때마다 REST를
확장하거나 우회하는 방식으로 대응했고, 결과적으로 오늘날의 웹 API 생태계는 필딩이 설계한
것과 사실상 다른 여러 아키텍처 스타일의 혼합이 되었다.
