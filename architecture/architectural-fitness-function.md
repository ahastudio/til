# 아키텍처 피트니스 함수

원문: <https://www.thoughtworks.com/radar/techniques/architectural-fitness-function>

## 요약

아키텍처 피트니스 함수(Architectural Fitness Function)는 진화 컴퓨팅에서 빌려온 개념으로, 시스템이 의도된 아키텍처 특성을 얼마나 잘 충족하는지 객관적으로 평가하는 메커니즘이다.
Neal Ford, Rebecca Parsons, Patrick Kua, Pramod Sadalage의 저서 *Building Evolutionary Architectures*에서 정의되었으며, “특정 아키텍처 특성에 대한 객관적 무결성 평가를 제공하는 모든 메커니즘”을 의미한다.

Thoughtworks는 2017년 11월 Technology Radar에서 이 기법을 처음 소개하며 “Trial” 등급을 부여했고, 2018년 5월에도 같은 평가를 유지했다.
현재는 Radar에서 제거되었는데, 이는 기술이 성숙하여 주류에 합류했다는 의미로 해석할 수 있다.

피트니스 함수의 핵심은 “더 낫다”는 것이 무엇인지를 명시적으로 정의하는 것이다.
진화 알고리즘이 피트니스 함수로 개체의 적합도를 측정하듯, 아키텍처 피트니스 함수는 시스템이 원하는 특성을 유지하는지 자동으로 지속적으로 검증한다.
이를 통해 아키텍트는 구두나 문서가 아닌 실행 가능한 검증으로 아키텍처 특성을 소통하고 보존할 수 있다.

피트니스 함수는 기존의 검증 메커니즘을 포괄한다.
단위 테스트, 통합 테스트, 성능 메트릭, 모니터링, 정적 분석 등이 모두 피트니스 함수의 형태가 될 수 있다.
예를 들어 “백엔드 패키지가 프론트엔드 패키지에 의존해서는 안 된다”는 아키텍처 규칙을 ArchUnit으로 자동화하면, 그것이 곧 피트니스 함수다.

## 분석

### 진화 아키텍처와의 관계

피트니스 함수는 진화적 아키텍처(Evolutionary Architecture)의 핵심 구성요소다.
진화적 아키텍처는 시스템이 변화하는 비즈니스 요구와 기술 환경에 적응하면서도 원하는 특성을 잃지 않도록 설계하는 접근 방식이다.
변화는 불가피하다는 전제하에, 변화를 막는 대신 변화 속에서도 아키텍처 무결성을 유지하는 메커니즘을 구축한다.

피트니스 함수가 없는 시스템에서는 아키텍처 부식(architectural erosion)이 조용히 일어난다.
처음에는 잘 설계된 레이어드 아키텍처가, 기능 추가 압박 속에서 레이어 간 경계가 무너지고, 순환 의존성이 생기고, 결합도가 올라간다.
이 변화는 어느 순간 명시적으로 결정된 것이 아니라 수백 번의 작은 타협들의 누적이다.
피트니스 함수는 이 타협의 순간에 빨간 불을 켠다.

### 분류 체계

피트니스 함수는 여러 차원으로 분류된다.

**자동화 수준**: 자동화(automated)와 수동(manual)으로 나뉜다.
자동화된 피트니스 함수는 CI 파이프라인에서 실행되어 즉각적 피드백을 제공한다.
수동 함수는 아키텍처 리뷰나 보안 감사처럼 주기적으로 사람이 수행한다.

**실행 주기**: 지속적(continual)과 주기적(temporal)으로 구분된다.
지속적 함수는 코드 변경이 있을 때마다 실행되고, 주기적 함수는 일정 시간 간격으로 실행된다.

**범위**: 원자적(atomic)과 전체적(holistic)으로 나뉜다.
원자적 함수는 단일 아키텍처 특성을 검증하고, 전체적 함수는 여러 특성의 조합을 검증한다.

### 구현 도구

구조적 피트니스 함수에는 ArchUnit(Java), NetArchTest(.NET), Archfit(PHP) 등이 사용된다.
성능 피트니스 함수에는 부하 테스트 도구, 메트릭 수집 시스템이 활용된다.
보안 피트니스 함수에는 OWASP 검사 도구, 의존성 취약점 스캐너가 포함된다.

ArchUnit을 사용한 레이어드 아키텍처 규칙 예시:

```java
noClasses()
    .that().resideInAPackage("..backend..")
    .should().dependOnClassesThat().resideInAPackage("..frontend..");
```

이 코드 한 줄이 “백엔드는 프론트엔드에 의존해서는 안 된다”는 아키텍처 원칙을 실행 가능한 테스트로 바꾼다.

### 아키텍처 거버넌스의 전환

전통적 아키텍처 거버넌스는 아키텍트가 문서를 작성하고, 코드 리뷰에서 규칙 준수를 확인하고, 때로는 위반을 사후에 발견한다.
이 접근은 느리고, 주관적이며, 확장성이 없다.
피트니스 함수는 거버넌스를 개발 루프 안으로 가져온다.
위반이 커밋 시점에 즉각 감지되므로, 개발자는 맥락이 살아있는 상태에서 피드백을 받는다.
이것은 거버넌스를 장벽에서 안전망으로 전환한다.

## 비평

### 강점: 아키텍처의 실행 가능한 명세

피트니스 함수의 가장 강력한 점은 아키텍처 의도를 실행 가능한 코드로 표현한다는 것이다.
문서는 시간이 지나면 낡지만, 실행되는 테스트는 시스템과 함께 최신 상태를 강제한다.
또한 피트니스 함수를 작성하는 과정 자체가 아키텍트와 개발자 간의 대화를 촉진한다.
“어떤 특성을 보존해야 하는가?”라는 질문은 암묵적 가정을 명시적으로 만든다.

### 한계: 측정 가능한 것만 보호한다

피트니스 함수는 객관적으로 측정 가능한 특성에만 적용된다.
코드 가독성, 설계의 우아함, 팀 간 협업 패턴 같은 것들은 피트니스 함수로 포착하기 어렵다.
또한 “무엇을 검증해야 하는가”를 결정하는 것 자체가 비자명하다.
중요하지만 측정하지 않는 특성은 여전히 아키텍처 부식에 취약하다.

### 초기 단계 오버헤드

탐색적 개발 초기에는 아키텍처 특성이 유동적이다.
이 단계에서 엄격한 피트니스 함수를 적용하면 유연성을 해치고 불필요한 리팩터링을 강제할 수 있다.
피트니스 함수는 아키텍처가 어느 정도 안정된 시스템에서 더 효과적이다.

### 런타임 동작과 AI 산출물의 한계

현재 피트니스 함수는 정적 구조와 측정 가능한 메트릭에서 잘 작동한다.
그러나 분산 시스템의 창발적 동작, AI 모델의 출력 품질, 데이터 컨트랙트의 의미론적 변화는 피트니스 함수로 검증하기 어렵다.
마이크로서비스 환경에서 서비스 간 계약의 의미론적 호환성은 전통적 피트니스 함수의 범위를 넘는다.

## 인사이트

### 아키텍처는 테스트할 수 있다

소프트웨어 테스팅에서 TDD(Test-Driven Development)가 가져온 패러다임 전환은 “테스트는 사후 검증이 아니라 설계 도구”라는 인식이었다.
피트니스 함수는 아키텍처 영역에서 같은 전환을 제안한다.
“아키텍처 원칙을 테스트로 표현하라”는 것이다.
이 전환이 중요한 이유는, 테스트로 표현된 아키텍처 원칙은 자동화되고 지속적으로 검증되며 팀 전체가 참여할 수 있기 때문이다.
아키텍처가 소수 아키텍트의 머릿속에서 나와 코드베이스에 살게 된다.
이것은 아키텍처를 민주화하는 동시에 강제하는 방법이다.

### 기술 부채의 감지기로서의 피트니스 함수

기술 부채는 대부분 의도적 타협의 누적이 아니라 무의식적 경계 침범의 축적이다.
피트니스 함수가 없는 시스템에서는 레이어 경계가 무너지는 것, 순환 의존성이 생기는 것, 모듈이 점점 비대해지는 것이 즉각 감지되지 않는다.
개발자는 자신이 경계를 침범하고 있다는 사실을 모른 채 작업하고, 위반이 인식될 때쯤이면 이미 되돌리기 어려운 상태가 된다.
피트니스 함수는 이 과정에 조기 경보 시스템을 제공한다.
코드 품질 메트릭(복잡도, 커버리지, 의존성 깊이)을 피트니스 함수로 통합하면, 기술 부채가 쌓이는 속도를 실시간으로 볼 수 있게 된다.
이것은 단순히 문제를 발견하는 것이 아니라, 부채 축적 패턴을 이해하는 것이다.

### 진화와 보존의 긴장: 피트니스 함수의 메타 문제

피트니스 함수의 아이러니는, 아키텍처가 진화할 수 있도록 하기 위해 설계된 도구가 동시에 특정 아키텍처 결정을 고착화할 수 있다는 것이다.
잘못 설계된 피트니스 함수는 낡은 아키텍처 결정을 강제하는 장벽이 된다.
예를 들어, “모든 서비스는 REST API를 통해서만 통신해야 한다”는 피트니스 함수는 이벤트 기반 아키텍처로의 전환을 가로막는다.
이 메타 문제는 피트니스 함수 자체가 진화해야 한다는 것을 의미한다.
아키텍처 특성에 대한 가정이 바뀔 때, 피트니스 함수도 함께 재검토되어야 한다.
피트니스 함수는 영원한 진리를 코드화하는 것이 아니라, 현재 시점에서 중요한 것을 자동화하는 것이다.
이 차이를 이해하는 팀만이 피트니스 함수를 제약이 아닌 도구로 사용할 수 있다.
