UI나 데이터베이스 없이도 동작하는 애플리케이션을 만들어라.
그래야 자동화 회귀 테스트를 실행할 수 있고, 데이터베이스가 다운되어도 작동하며,
사람의 개입 없이 애플리케이션끼리 연결할 수 있다.

## 패턴: 포트와 어댑터 (객체 구조적 패턴)

대안 이름: 헥사고날 아키텍처

### 의도

애플리케이션이 사용자, 프로그램, 자동화 테스트, 배치 스크립트 중 어느 것으로든 동등하게
구동될 수 있게 하고, 최종 런타임 장치 및 데이터베이스와 격리된 상태에서 개발·테스트할 수
있게 한다.

외부 세계에서 포트를 통해 이벤트가 도착하면, 기술별 어댑터가 이를 사용 가능한 프로시저
호출이나 메시지로 변환해 애플리케이션에 전달한다. 애플리케이션은 입력 장치의 성질을 전혀
알지 못한다. 애플리케이션이 무언가를 내보낼 때는 포트를 통해 어댑터로 전달하고, 어댑터는
수신 기술(사람 또는 자동화 시스템)이 필요로 하는 적절한 신호를 생성한다. 애플리케이션은
어댑터와의 의미론적으로 올바른 상호작용을 유지하면서도, 어댑터 너머에 무엇이 있는지는
전혀 알지 못한다.

### 동기

수년간 소프트웨어 애플리케이션의 고질적인 문제 중 하나는 비즈니스 로직이 사용자 인터페이스
코드에 침투하는 것이었다. 이로 인한 문제는 세 가지다. 첫째, 테스트가 필요한 로직 일부가
필드 크기나 버튼 위치 같은 자주 바뀌는 시각적 세부 사항에 종속되므로, 자동화 테스트
스위트로 시스템을 깔끔하게 테스트할 수 없다. 정확히 같은 이유로, 사람이 직접 사용하던
시스템을 배치 실행 시스템으로 전환하는 것이 불가능해진다. 마찬가지 이유로, 나중에 다른
프로그램이 이 프로그램을 구동해야 할 때 그렇게 하기 어렵거나 불가능해진다.

여러 조직에서 반복된 해결책은 아키텍처에 새 레이어를 추가하면서 “이번엔 정말로 새 레이어에
비즈니스 로직을 넣지 않겠다”고 약속하는 것이었다. 그러나 약속 위반을 감지할 메커니즘이
없었기에, 몇 년이 지나면 새 레이어도 비즈니스 로직으로 가득 차고 문제는 반복되었다.

이제 애플리케이션이 제공하는 모든 기능을 API나 함수 호출을 통해 사용할 수 있다고 상상해
보자. 이 상황에서 테스트·QA 부서는 자동화 테스트 스크립트로 애플리케이션을 실행해 새
코드가 기존 기능을 깨뜨리는지 확인할 수 있다. 비즈니스 전문가는 GUI 세부 사항이 확정되기
전에 자동화 테스트 케이스를 작성해 개발자가 일을 제대로 했는지 알 수 있다(이 테스트가
테스트 부서가 실행하는 테스트가 된다). 애플리케이션을 헤드리스 모드로 배포해 API만 노출할
수 있고, 다른 프로그램이 그 기능을 사용할 수 있다. 이는 복잡한 애플리케이션 스위트의
전체 설계를 단순화하고, 비즈니스-투-비즈니스 서비스 애플리케이션이 웹을 통해 사람의 개입
없이 서로를 사용할 수 있게 한다. 마지막으로 자동화 기능 회귀 테스트가 비즈니스 로직을
프레젠테이션 레이어 밖에 두겠다는 약속 위반을 감지한다. 조직은 로직 누출을 감지하고
수정할 수 있게 된다.

보통 애플리케이션의 “반대편”으로 간주되는 곳에도 유사한 문제가 있다. 애플리케이션 로직이
외부 데이터베이스나 서비스와 결합될 때다. 데이터베이스 서버가 다운되거나 대규모 재작업이나
교체가 이루어지면, 개발자들은 데이터베이스에 종속된 작업을 할 수 없게 된다. 이는 지연
비용과 종종 사람들 사이의 갈등을 유발한다.

두 문제가 연관되어 있다는 것은 직관적으로 명확하지 않지만, 해결책의 성격 속에 대칭성이
드러난다.

### 해결책의 성격

사용자 측 문제와 서버 측 문제는 사실 동일한 설계·프로그래밍 오류, 즉 비즈니스 로직과
외부 개체 간 상호작용의 얽힘에서 비롯된다. 활용해야 할 비대칭은 애플리케이션의 좌우가
아니라 내부와 외부 사이에 있다. 지켜야 할 규칙은 내부에 속하는 코드가 외부로 누출되어서는
안 된다는 것이다.

잠시 좌우 혹은 상하 비대칭을 무시하면, 애플리케이션은 포트를 통해 외부 에이전시와 통신한다.
“포트”라는 단어는 운영체제의 포트를 연상시키도록 의도되었다. 운영체제 포트에서는 해당
포트의 프로토콜을 따르는 어떤 장치든 연결할 수 있다. 전자 기기의 포트도 마찬가지다.
포트의 프로토콜은 두 장치 간 대화의 목적으로 정해진다. 프로토콜은 API(애플리케이션 프로그램
인터페이스) 형태를 띤다.

각 외부 장치에 대해 해당 장치가 필요로 하는 신호로 API 정의를 변환하고, 그 역도 수행하는
어댑터가 있다. GUI나 그래픽 사용자 인터페이스는 사람의 움직임을 포트의 API로 변환하는
어댑터의 예다. 같은 포트에 맞는 다른 어댑터로는 FIT나 Fitnesse 같은 자동화 테스트 하네스,
배치 드라이버, 기업 내 또는 네트워크를 통한 애플리케이션 간 통신에 필요한 코드가 있다.

애플리케이션의 다른 측면에서 애플리케이션은 외부 개체와 통신해 데이터를 얻는다. 프로토콜은
보통 데이터베이스 프로토콜이다. 애플리케이션 관점에서 데이터베이스가 SQL에서 플랫 파일이나
다른 종류의 데이터베이스로 이전되어도 API를 통한 대화는 바뀌어서는 안 된다. 같은 포트에
맞는 추가 어댑터에는 SQL 어댑터, 플랫 파일 어댑터, 그리고 무엇보다 중요하게, 메모리 내에
존재하는 “목(mock)” 데이터베이스 대체물 어댑터가 포함된다.

많은 애플리케이션은 두 개의 포트만 가진다: 사용자 측 대화와 데이터베이스 측 대화. 이는
비대칭적인 모습을 만들어내고, 애플리케이션을 1차원적인 3, 4, 5레이어 스택 아키텍처로
구성하는 것이 자연스러워 보이게 한다.

이런 다이어그램에는 두 가지 문제가 있다. 첫째이자 최악의 문제는, 사람들이 레이어 다이어그램의
“선”을 진지하게 받아들이지 않는다는 것이다. 그들은 레이어 경계를 넘어 애플리케이션 로직이
누출되게 하고, 위에서 언급한 문제들을 야기한다. 둘째, 애플리케이션에는 두 개 이상의 포트가
있을 수 있어 아키텍처가 1차원 레이어 다이어그램에 맞지 않는다.

헥사고날, 즉 포트와 어댑터 아키텍처는 이 상황의 대칭성에 주목함으로써 이 문제들을 해결한다.
내부에 애플리케이션이 있고, 외부에 있는 것들과 여러 포트를 통해 통신한다는 것이다.
애플리케이션 외부의 항목들은 대칭적으로 처리될 수 있다.

육각형은 시각적으로 다음을 부각하기 위한 것이다.
(a) 내부-외부 비대칭과 포트의 유사한 성격, 1차원 레이어드 그림과 그것이 연상시키는 모든
것에서 벗어나기 위해.
(b) 두 개, 세 개, 또는 네 개(지금까지 만난 최대치) 같은 정해진 수의 서로 다른 포트의 존재.

육각형이 육각형인 이유는 숫자 6이 중요해서가 아니라, 그림을 그리는 사람이 1차원 레이어드
다이어그램에 구애받지 않고 필요에 따라 포트와 어댑터를 삽입할 공간을 주기 위해서다.
“헥사고날 아키텍처”라는 용어는 이 시각적 효과에서 비롯된다.

“포트와 어댑터”라는 용어는 다이어그램의 각 부분의 목적을 담는다. 포트는 목적이 있는
대화를 식별한다. 어떤 포트에든 보통 여러 어댑터가 있고, 해당 포트에 연결될 수 있는 다양한
기술을 위한 것이다. 일반적으로 여기에는 전화 자동 응답기, 사람의 목소리, 터치톤 전화기,
그래픽 사용자 인터페이스, 테스트 하네스, 배치 드라이버, HTTP 인터페이스, 직접 프로그램 간
인터페이스, 목(메모리 내) 데이터베이스, 실제 데이터베이스(개발·테스트·실제 사용을 위한 각기
다른 데이터베이스일 수 있음)가 포함된다.

응용 노트에서 좌우 비대칭이 다시 언급될 것이다. 그러나 이 패턴의 주요 목적은 모든 외부
항목이 애플리케이션 관점에서 동일하다고 잠시 가정하면서 내부-외부 비대칭에 집중하는 것이다.

### 구조

그림 2는 두 개의 활성 포트와 각 포트에 여러 어댑터가 있는 애플리케이션을 보여준다.
두 포트는 애플리케이션을 제어하는 측과 데이터를 조회하는 측이다. 이 다이어그램은 애플리케이션이
자동화 시스템 수준 회귀 테스트 스위트, 사람 사용자, 원격 HTTP 애플리케이션, 또는 다른
로컬 애플리케이션 중 어느 것으로든 동등하게 구동될 수 있음을 보여준다. 데이터 측에서는
애플리케이션이 메모리 내 오라클, 즉 목(mock) 데이터베이스 대체물을 사용해 외부 데이터베이스와
분리된 상태로 실행되도록 구성될 수 있고, 또는 테스트 또는 운영 데이터베이스에 대해 실행될
수 있다. 아마도 유스케이스로 작성된 애플리케이션의 기능 명세는 외부 기술 중 어느 하나가
아닌 내부 육각형의 인터페이스에 대해 작성된다.

그림 3은 같은 애플리케이션을 3레이어 아키텍처 다이어그램에 매핑한 것이다. 다이어그램을
단순화하기 위해 각 포트에 두 개의 어댑터만 표시했다. 이 다이어그램은 여러 어댑터가 상단과
하단 레이어에 어떻게 들어맞는지, 그리고 시스템 개발 중 다양한 어댑터가 사용되는 순서를
보여주기 위한 것이다. 번호가 매겨진 화살표는 팀이 애플리케이션을 개발하고 사용할 수 있는
순서를 나타낸다.

1. FIT 테스트 하네스가 애플리케이션을 구동하고, 목(메모리 내) 데이터베이스가 실제 데이터베이스를
   대체할 때.
2. GUI를 애플리케이션에 추가하면서, 아직 목 데이터베이스로 실행할 때.
3. 통합 테스트에서 자동화 테스트 스크립트(예: Cruise Control)가 테스트 데이터가 담긴 실제
   데이터베이스에 대해 애플리케이션을 구동할 때.
4. 실제 사용에서 사람이 애플리케이션을 사용해 라이브 데이터베이스에 접근할 때.

### 샘플 코드

포트와 어댑터를 보여주는 가장 단순한 애플리케이션은 다행히도 FIT 문서에 함께 제공된다.
간단한 할인 계산 애플리케이션이다.

```text
discount(amount) = amount * rate(amount);
```

이 적용에서 금액은 사용자로부터, 비율은 데이터베이스로부터 받으므로 두 개의 포트가 생긴다.
단계적으로 구현한다.

- 테스트는 있지만 목 데이터베이스 대신 상수 비율로 시작.
- 그다음 GUI 추가.
- 그다음 실제 데이터베이스로 교체 가능한 목 데이터베이스 추가.

이 예제의 코드를 제공해 준 IHC의 Gyan Sharma에게 감사드린다.

#### 1단계: FIT + 앱 + 상수-목-데이터베이스

먼저 HTML 테이블로 테스트 케이스를 만든다(FIT 문서 참조).

```text
TestDiscounter

amount | discount()
100    | 5
200    | 10
```

열 이름이 프로그램의 클래스명과 함수명이 된다. FIT에는 이 “프로그래머식 표현”을 없애는
방법이 있지만, 이 글에서는 그냥 두는 것이 더 쉽다.

테스트 데이터가 무엇인지 알았으니, FIT와 함께 제공되는 ColumnFixture인 사용자 측
어댑터를 만든다.

```java
import fit.ColumnFixture;
public class TestDiscounter extends ColumnFixture {
    private Discounter app = new Discounter();
    public double amount;
    public double discount() {
        return app.discount(amount);
    }
}
```

이 시점의 코드는 체크인하고, Cruise Control이나 자동화 빌드 머신에 연결하고,
빌드-테스트 스위트에 포함시킬 준비가 되었다.

#### 2단계: UI + 앱 + 상수-목-데이터베이스

코드가 다소 길어서 여기에 포함시키지 않겠지만, 직접 UI를 만들어 Discounter 애플리케이션을
구동해 보라. 코드의 핵심 줄들은 다음과 같다.

```java
... Discounter app = new Discounter();
public void actionPerformed(ActionEvent event) {
    ...
    String amountStr = text1.getText();
    double amount = Double.parseDouble(amountStr);
    discount = app.discount(amount);
    text3.setText("" + discount);
    ...
}
```

이 시점에서 애플리케이션은 데모할 수 있고 회귀 테스트도 가능하다. 사용자 측 어댑터 둘 다
실행되고 있다.

#### 3단계: (FIT 또는 UI) + 앱 + 목 데이터베이스

데이터베이스 측을 위한 교체 가능한 어댑터를 만들기 위해, 리포지토리 인터페이스,
목 또는 실제 서비스 객체를 생성하는 RepositoryFactory, 데이터베이스를 위한 메모리 내 목을
만든다.

```java
public interface RateRepository {
    double getRate(double amount);
}

public class RepositoryFactory {
    public RepositoryFactory() { super(); }
    public static RateRepository getMockRateRepository() {
        return new MockRateRepository();
    }
}

public class MockRateRepository implements RateRepository {
    public double getRate(double amount) {
        if(amount <= 100) return 0.01;
        if(amount <= 1000) return 0.02;
        return 0.05;
    }
}
```

이 어댑터를 Discounter 애플리케이션에 연결하려면, 애플리케이션 자체를 업데이트해 사용할
리포지토리 어댑터를 받도록 하고, (FIT 또는 UI) 사용자 측 어댑터가 사용할 리포지토리(실제
또는 목)를 애플리케이션 생성자에 전달하도록 한다.

```java
import repository.RepositoryFactory;
import repository.RateRepository;
public class Discounter {
    private RateRepository rateRepository;
    public Discounter(RateRepository r) {
        super();
        rateRepository = r;
    }
    public double discount(double amount) {
        double rate = rateRepository.getRate(amount);
        return amount * rate;
    }
}

import app.Discounter;
import fit.ColumnFixture;
public class TestDiscounter extends ColumnFixture {
    private Discounter app =
        new Discounter(RepositoryFactory.getMockRateRepository());
    public double amount;
    public double discount() {
        return app.discount(amount);
    }
}
```

이것으로 헥사고날 아키텍처의 가장 단순한 버전 구현이 완성된다.

### 응용 노트

#### 좌우 비대칭

포트와 어댑터 패턴은 의도적으로 모든 포트가 근본적으로 유사하다고 가정하며 작성되었다.
이 가정은 아키텍처 수준에서 유용하다. 구현에서 포트와 어댑터는 두 가지 형태로 나타나는데,
곧 명확해질 이유로 주도적(primary)과 피동적(secondary)이라 부르겠다. 구동 어댑터(driving
adapters)와 피구동 어댑터(driven adapters)라고도 부를 수 있다.

주의 깊은 독자는 주어진 모든 예제에서 FIT 픽스처가 왼쪽 포트에 사용되고 목이 오른쪽에
사용된다는 것을 눈치챘을 것이다. 3레이어 아키텍처에서 FIT는 상단 레이어에, 목은 하단
레이어에 위치한다.

이는 유스케이스의 “주 행위자”와 “부 행위자” 개념과 관련이 있다. 주 행위자는 애플리케이션을
구동하는 행위자(애플리케이션을 휴면 상태에서 꺼내 광고된 기능 중 하나를 수행하도록 하는
행위자)다. 부 행위자는 애플리케이션이 구동하는 대상으로, 답변을 얻거나 단순히 알림을 보내기
위한 것이다. 주와 부의 구분은 누가 대화를 시작하거나 담당하는지에 달려 있다.

주 행위자를 대체하는 자연스러운 테스트 어댑터는 FIT다. FIT는 스크립트를 읽어 애플리케이션을
구동하도록 설계되었기 때문이다. 데이터베이스 같은 부 행위자를 대체하는 자연스러운 테스트
어댑터는 목이다. 목은 애플리케이션의 쿼리에 응답하거나 이벤트를 기록하도록 설계되었기
때문이다.

이 관찰은 시스템의 유스케이스 컨텍스트 다이어그램을 따라 주 포트와 주 어댑터는 육각형의
왼쪽(또는 상단)에, 부 포트와 부 어댑터는 오른쪽(또는 하단)에 그리도록 이끈다.

주/부 포트·어댑터와 FIT·목의 각각 구현 간의 관계는 염두에 두면 유용하지만, 포트와 어댑터
아키텍처를 단축하기 위해 사용해서는 안 된다. 포트와 어댑터 구현의 궁극적인 이점은 완전히
격리된 모드로 애플리케이션을 실행할 수 있는 능력이다.

#### 유스케이스와 애플리케이션 경계

헥사고날 아키텍처 패턴을 사용해 유스케이스를 작성하는 선호 방식을 강화하는 것이 유용하다.

일반적인 실수는 각 포트 외부에 있는 기술에 대한 상세한 지식을 담은 유스케이스를 작성하는
것이다. 이렇게 작성된 유스케이스는 길고, 읽기 어렵고, 지루하고, 취약하고, 유지보수 비용이
높다는 이유로 업계에서 당연하게도 나쁜 평판을 얻고 있다.

포트와 어댑터 아키텍처를 이해하면, 유스케이스는 일반적으로 외부 기술과 무관하게 애플리케이션이
지원하는 기능과 이벤트를 명시하는 애플리케이션 경계(내부 육각형)에서 작성되어야 함을 알 수
있다. 이렇게 작성된 유스케이스는 더 짧고, 읽기 쉽고, 유지보수 비용이 낮으며, 시간이 지나도
안정적이다.

#### 포트는 몇 개?

포트가 정확히 무엇인지 아닌지는 대체로 취향의 문제다. 극단적으로, 모든 유스케이스에 자체
포트를 줄 수 있어 많은 애플리케이션에서 수백 개의 포트가 생길 수 있다. 반대로, 모든 주
포트와 모든 부 포트를 합쳐 왼쪽 하나, 오른쪽 하나, 두 개의 포트만 가질 수도 있다.

어느 극단도 최적으로 보이지 않는다.

알려진 사용 사례에서 설명하는 기상 시스템은 네 개의 자연스러운 포트를 가진다: 기상 피드,
관리자, 알림을 받는 구독자, 구독자 데이터베이스. 커피 머신 컨트롤러는 네 개의 자연스러운
포트를 가진다: 사용자, 레시피와 가격이 담긴 데이터베이스, 디스펜서, 동전 박스. 병원 약물
시스템은 세 개를 가질 수 있다: 간호사용, 처방 데이터베이스용, 컴퓨터 제어 약물 디스펜서용.

“잘못된” 수의 포트를 선택해도 특별한 문제가 없는 것 같으므로, 이는 직관의 문제다. 내
선택은 위에서 설명하고 알려진 사용 사례에서 설명한 것처럼 두 개, 세 개, 또는 네 개의 적은
수를 선호하는 경향이 있다.

#### 알려진 사용 사례

그림 4는 네 개의 포트와 각 포트에 여러 어댑터가 있는 애플리케이션을 보여준다. 이는
국가 기상청의 지진, 토네이도, 화재, 홍수에 대한 경보를 수신하고 전화나 자동 응답기로
사람들에게 알림을 보내는 애플리케이션에서 파생된 것이다. 이 시스템을 논의할 당시, 시스템의
인터페이스는 기술로 식별·논의되었고, 목적과 연결되었다. 와이어 피드로 도착하는 트리거
데이터용 인터페이스, 자동 응답기로 보내는 알림 데이터용 인터페이스, GUI로 구현된 관리
인터페이스, 구독자 데이터를 얻는 데이터베이스 인터페이스가 있었다.

개발팀은 기상청의 HTTP 인터페이스, 구독자에 대한 이메일 인터페이스를 추가해야 했고,
다양한 고객 구매 선호도에 맞게 성장하는 애플리케이션 스위트를 묶고 풀 방법을 찾아야 했기
때문에 어려움을 겪고 있었다. 모든 조합과 순열에 대해 별도 버전을 구현·테스트·유지보수해야
하는 유지보수 및 테스트 악몽에 직면한 것 같아 두려워했다.

그들의 설계 전환은 시스템 인터페이스를 기술이 아닌 목적으로 아키텍처하고, 기술을 (모든
측면에서) 어댑터로 교체 가능하게 만드는 것이었다. 즉시 HTTP 피드와 이메일 알림을 포함할
수 있게 되었다(새 어댑터는 점선으로 표시되어 있다). 각 애플리케이션을 API를 통한 헤드리스
모드로 실행 가능하게 함으로써, 앱-투-앱 어댑터를 추가하고 필요에 따라 하위 애플리케이션을
연결해 애플리케이션 스위트를 묶고 풀 수 있게 되었다. 마지막으로, 각 애플리케이션을 테스트와
목 어댑터를 갖춘 완전히 격리된 상태로 실행 가능하게 함으로써, 독립 실행형 자동화 테스트
스크립트로 애플리케이션을 회귀 테스트할 수 있는 능력을 얻었다.

#### Mac, Windows, Google, Flickr, Web 2.0

1990년대 초, 워드프로세서 같은 맥 애플리케이션은 API로 구동 가능한 인터페이스를 갖추도록
요구되었다. 그래야 애플리케이션과 사용자가 작성한 스크립트가 애플리케이션의 모든 기능에
접근할 수 있었다. Windows 데스크톱 애플리케이션도 같은 능력을 발전시켜 왔다. 현재(2005년)
웹 애플리케이션의 추세는 API를 게시하고 다른 웹 애플리케이션이 그 API에 직접 접근하도록
하는 것이다. 따라서 Google 맵 위에 지역 범죄 데이터를 게시하거나 Flickr의 사진 보관 및
주석 기능을 포함하는 웹 애플리케이션을 만들 수 있다. 이 모든 예는 주 포트의 API를 외부에
공개하는 것에 관한 것이다. 부 포트에 대한 정보는 없다.

#### 저장 출력

C2 위키의 Willem Bogaerts가 작성한 예제:

“나는 비슷한 것을 경험했는데, 주로 내 애플리케이션 레이어가 하지 말아야 할 것들을 관리하는
전화 교환기 같은 역할을 하는 강한 경향이 있었기 때문이다. 내 애플리케이션은 출력을 생성하고,
사용자에게 보여주고, 그런 다음 그것을 저장할 수 있는 가능성도 있었다. 내 주요 문제는
항상 저장할 필요가 없다는 것이었다. 그래서 내 애플리케이션은 출력을 생성하고, 버퍼에 넣고,
사용자에게 제시해야 했다. 그런 다음 사용자가 출력을 저장하기로 결정하면, 애플리케이션이
버퍼를 검색해 실제로 저장했다. 나는 이것이 전혀 마음에 들지 않았다. 그러다 해결책을
생각해냈다: 저장 기능이 있는 프레젠테이션 컨트롤을 만드는 것이다. 이제 애플리케이션은
더 이상 출력을 다른 방향으로 채널링하지 않고, 단순히 프레젠테이션 컨트롤로 출력한다.
프레젠테이션 컨트롤이 답변을 버퍼링하고 사용자가 저장할 가능성을 준다. 전통적인 레이어드
아키텍처는 UI와 저장을 다른 것으로 강조한다. 포트와 어댑터 아키텍처는 출력을 다시 단순히
'출력'으로 줄일 수 있다.”

#### C2 위키의 익명 예제

“내가 작업한 한 프로젝트에서, 우리는 컴포넌트 스테레오 시스템의 SystemMetaphor를 사용했다.
각 컴포넌트는 특정 목적을 가진 인터페이스를 가지고 있다. 그런 다음 간단한 케이블과 어댑터를
사용해 거의 무한한 방법으로 컴포넌트를 연결할 수 있다.”

#### 분산된 대규모 팀 개발

이것은 아직 시험 사용 중이므로 패턴의 사용으로 제대로 포함시킬 수 없다. 그러나 고려해 볼
만하다. 서로 다른 위치의 팀들이 모두 헥사고날 아키텍처로 빌드하면서, 애플리케이션이나
컴포넌트를 독립 실행형 모드로 테스트할 수 있도록 FIT와 목을 사용한다. CruiseControl 빌드는
30분마다 실행되며 FIT+목 조합을 사용해 모든 애플리케이션을 실행한다. 애플리케이션 서브시스템과
데이터베이스가 완성되면, 목은 테스트 데이터베이스로 교체된다.

#### UI와 애플리케이션 로직 개발 분리

이것은 아직 초기 시험 사용 중이므로 패턴 사용으로 포함시킬 수 없다. 그러나 고려해 볼 만하다.
UI 설계가 불안정하다. 구동 기술이나 은유를 아직 결정하지 못했다. 백엔드 서비스 아키텍처도
결정되지 않았고, 실제로 앞으로 6개월 동안 여러 번 바뀔 것이다. 그럼에도 불구하고 프로젝트는
공식적으로 시작되었고 시간이 흐르고 있다. 애플리케이션 팀은 FIT 테스트와 목을 만들어
애플리케이션을 격리하고, 사용자에게 보여줄 수 있는 테스트·데모 가능한 기능을 만든다.
UI와 백엔드 서비스 결정이 최종적으로 이루어지면, 그 요소들을 애플리케이션에 추가하는 것은
“간단해야 한다”. 어떻게 되는지 지켜볼 것이다.

#### 관련 패턴

어댑터(Adapter): 디자인 패턴 책에는 일반적인 어댑터 패턴에 대한 설명이 있다. “한 클래스의
인터페이스를 클라이언트가 기대하는 다른 인터페이스로 변환한다.” 포트와 어댑터 패턴은 어댑터
패턴의 특정 사용이다.

모델-뷰-컨트롤러(MVC): MVC 패턴은 Smalltalk 프로젝트에서 1974년만큼 일찍 구현되었다.
초기에는 모델-인터랙터(Model-Interactor), 모델-뷰-프레젠터(Model-View-Presenter) 같은
많은 변형이 주어졌다. 이들 각각은 주 포트에 포트와 어댑터 아이디어를 구현하지만, 부 포트에는
아니다.

목 객체(Mock Objects)와 루프백(Loopback): “목 객체는 다른 객체의 행동을 테스트하기 위해
사용되는 '이중 에이전트'다. 첫째, 목 객체는 실제 구현의 외부 행동을 모방하는 인터페이스나
클래스의 가짜 구현으로 행동한다. 둘째, 목 객체는 다른 객체가 메서드와 어떻게 상호작용하는지
관찰하고 실제 행동을 사전 설정된 기대치와 비교한다. 불일치가 발생하면, 목 객체는 테스트를
중단하고 이상을 보고할 수 있다.” 루프백 패턴은 외부 장치에 대한 내부 대체물을 만들기 위한
명시적 패턴이다.

받침대(Pedestals): “레이어드 아키텍처 생성을 위한 패턴”에서 Barry Rubel은 포트와 어댑터와
매우 유사한 제어 소프트웨어의 대칭 축을 만드는 패턴을 설명한다. Pedestal 패턴은 시스템
내의 각 하드웨어 장치를 나타내는 객체를 구현하고 이 객체들을 제어 레이어에서 연결하도록
요구한다. Pedestal 패턴은 헥사고날 아키텍처의 양측을 설명하는 데 사용될 수 있지만, 어댑터
전반의 유사성을 아직 강조하지 않는다. 또한 기계 제어 환경을 위해 작성되어, IT 애플리케이션에
패턴을 적용하는 방법을 알기 쉽지 않다.

체크(Checks): Ward Cunningham의 사용자 입력 오류를 감지하고 처리하기 위한 패턴 언어는
내부 육각형 경계를 넘는 오류 처리에 좋다.

의존성 역전(Dependency Inversion, Dependency Injection)과 SPRING: Bob Martin의 의존성
역전 원칙(Martin Fowler가 의존성 주입이라고도 부름)은 “고수준 모듈은 저수준 모듈에 의존해서는
안 된다. 둘 다 추상화에 의존해야 한다. 추상화는 세부 사항에 의존해서는 안 된다. 세부 사항은
추상화에 의존해야 한다”고 말한다. Martin Fowler의 의존성 주입 패턴은 교체 가능한 부 행위자
어댑터를 만드는 방법을 보여준다. 코드는 이 글의 샘플 코드에서 한 것처럼 직접 입력하거나,
구성 파일을 사용하고 SPRING 프레임워크가 동등한 코드를 생성하도록 할 수 있다.

#### 감사의 말

여기서 사용된 샘플 코드를 제공해 준 Intermountain Health Care의 Gyan Sharma에게 감사드린다.
어댑터 패턴과 함께 읽었을 때 육각형이 무엇에 관한 것인지 이해하는 데 도움을 준 그녀의
책 Object Design을 통해 Rebecca Wirfs-Brock에게 감사드린다. 또한 수년간 이 패턴에 대한
의견을 제공해 준 Ward의 위키 사람들에게도 감사드린다.

## 요약

헥사고날 아키텍처(Hexagonal Architecture)는 Alistair Cockburn이 2005년에 발표한 패턴으로,
공식 명칭은 “포트와 어댑터(Ports and Adapters)”다.
핵심 의도는 애플리케이션이 사용자, 프로그램, 자동화 테스트, 배치 스크립트 중 어느 것으로든
동등하게 구동될 수 있게 하고, 최종 런타임 장치나 데이터베이스와 격리된 상태에서
개발·테스트할 수 있게 만드는 것이다.

동기는 두 가지 오래된 문제에서 출발한다.

첫째, 비즈니스 로직이 UI 코드로 스며드는 문제다.
이렇게 되면 자동화 회귀 테스트가 불가능해지고, 배치 실행으로 전환하기 어려워지며,
다른 프로그램이 해당 애플리케이션을 구동하는 것도 막힌다.
수많은 조직이 새로운 레이어를 추가해 해결하려 했지만, 메커니즘 없이는 몇 년 후
새 레이어도 비즈니스 로직으로 오염되기를 반복했다.

둘째, 반대편에서 발생하는 같은 문제다.
애플리케이션 로직이 외부 데이터베이스나 서비스와 단단히 결합되면,
데이터베이스가 다운될 때 개발 자체가 멈춰버린다.

Cockburn은 이 두 문제가 동일한 설계 오류, 즉 비즈니스 로직과 외부 개체 간의 얽힘에서
비롯된다고 지적한다.
해결 원칙은 좌우 비대칭이 아니라 내부-외부 비대칭이다.
“내부(inside)”에 속하는 코드는 절대 “외부(outside)”로 누출되어서는 안 된다.

외부 세계와의 통신은 포트(port)를 통해 이루어진다.
포트는 운영체제의 포트나 전자기기의 잭처럼, 프로토콜을 따르는 어떤 장치든 꽂을 수 있는
대화의 목적(purpose of the conversation)을 정의한다.
각 포트에는 어댑터(adapter)가 붙는다.
어댑터는 API 정의를 특정 기술의 신호로 변환하고 그 역도 수행한다.
하나의 포트에 여러 어댑터가 붙을 수 있다.
예컨대 사용자 포트 하나에 GUI 어댑터, FIT 테스트 어댑터, HTTP 어댑터, 배치 드라이버가
모두 호환된다.

육각형 모양은 6이라는 숫자가 중요해서가 아니라,
1차원 레이어드 다이어그램의 한계를 벗어나
여러 포트와 어댑터를 자유롭게 배치할 공간을 주기 위한 시각적 선택이다.

포트와 어댑터에는 비대칭이 있다.
왼쪽(또는 상단)에는 애플리케이션을 구동하는 주도 어댑터(primary/driving adapter),
오른쪽(또는 하단)에는 애플리케이션이 구동하는 피동 어댑터(secondary/driven adapter)가 위치한다.
유스케이스의 “주 행위자(primary actor)”와 “부 행위자(secondary actor)” 개념과 대응된다.
주 어댑터의 자연스러운 테스트 대체물은 FIT이고,
부 어댑터의 자연스러운 테스트 대체물은 목(mock)이다.

## 분석

## 동일한 문제의 양면 인식

Cockburn의 핵심 통찰은 “UI 문제”와 “데이터베이스 문제”가 서로 다른 문제가 아니라
동일한 구조적 실수의 양면임을 명확히 한 데 있다.
기존 아키텍처 논의는 UI→로직→DB의 수직 레이어에 집중했고,
두 경계를 서로 다른 성격의 문제로 취급했다.
이 패턴은 양쪽을 같은 방식(포트+어댑터)으로 처리함으로써 대칭적 해결책을 제시한다.

## 목적 기반 인터페이스 설계

“Known Uses” 섹션의 기상 시스템 사례는 패턴의 실전 가치를 잘 보여준다.
해당 팀은 인터페이스를 기술(technology)로 구분했다가,
이를 목적(purpose)으로 재설계하자마자 새 어댑터(HTTP 피드, 이메일 알림)를 추가하는 일이
자연스럽게 풀렸다.
이것이 포트를 정의하는 올바른 기준이다.
포트는 “무엇으로 통신하는가”가 아니라 “왜 통신하는가”로 나뉜다.

## 유스케이스 작성과의 연결

Cockburn은 유스케이스가 애플리케이션 경계(내부 육각형)에서 작성되어야 한다고 주장한다.
특정 기술을 알고 있는 유스케이스는 길고, 읽기 어렵고, 유지보수 비용이 비싸고 취약하다.
애플리케이션 경계에서 작성된 유스케이스는 짧고, 기술 변경에도 안정적이다.
이는 헥사고날 아키텍처가 단순한 기술 패턴이 아니라
분석/설계 실천 전반에 영향을 미치는 원칙임을 보여준다.

## 포트 수의 실용적 선택

Cockburn은 포트의 수를 과도하게 세분화하거나 과도하게 통합하는 양극단 모두 최적이 아님을
인정한다.
기상 시스템 4개, 커피 머신 컨트롤러 4개, 병원 약물 시스템 3개 등
자연스러운 목적의 경계를 따라 2~4개가 실용적이라는 귀납적 결론을 제시한다.
이는 패턴을 경직된 규칙이 아닌 판단의 도구로 제시하는 태도다.

## 관련 패턴과의 계보

MVC는 주 포트(primary port)에만 포트와 어댑터 아이디어를 적용한 패턴이고,
의존성 역전 원칙(DIP)은 부 어댑터를 교체 가능하게 만드는 구현 메커니즘이며,
목 객체 패턴은 메모리 내 대체물을 만드는 기법이다.
헥사고날 아키텍처는 이들을 아우르는 상위 구조 패턴으로서,
각각이 담당하는 측면을 통합하는 위치에 있다.

## 비평

## 강점: 대칭성이라는 단순한 원리

이 패턴의 가장 큰 강점은 단순한 원리 하나(내부-외부 비대칭)로
UI 결합 문제와 데이터베이스 결합 문제를 동시에 해소한다는 점이다.
레이어드 아키텍처가 “이번엔 정말로 로직을 섞지 않겠다”는 약속에 의존했다면,
헥사고날 아키텍처는 구조적으로 그 약속을 지킬 수 없게 만든다.
실행 메커니즘이 존재하는 규칙과 존재하지 않는 규칙의 차이다.

## 강점: 테스트 가능성을 일등 시민으로 격상

자동화 테스트를 애플리케이션의 주요 구동자로 명시적으로 설계에 포함시킨 것은
2005년 당시로선 선진적이었다.
FIT 어댑터를 GUI 어댑터와 동등하게 취급한다는 발상은,
테스트를 “나중에 붙이는 것”이 아닌 아키텍처 차원의 결정으로 끌어올린다.

## 약점: 포트와 어댑터 경계의 불명확성

포트가 몇 개여야 하는지, 어디서 포트를 나눠야 하는지에 대한 기준이
“자연스러운 목적의 경계”라는 직관적 판단에 의존한다.
이 논문은 구체적인 설계 기준을 제공하지 않는다.
실무에서는 “서비스마다 포트를 하나씩”처럼 과도하게 세분화하거나,
반대로 “입력 포트 하나, 출력 포트 하나”처럼 지나치게 뭉뚱그리는 경향이 생긴다.

## 약점: 주도/피동 구분이 실무에서 흐려진다

이벤트 기반 시스템이나 메시지 큐 아키텍처에서는
어떤 구성 요소가 “구동”하고 어떤 것이 “구동되는”지가 모호해진다.
소비자 큐는 애플리케이션을 구동하는가, 아니면 애플리케이션이 구동하는 리소스인가?
2005년 논문이 다루지 않은 현대적 컨텍스트에서 패턴의 적용 방식이
해석마다 달라지는 이유가 여기 있다.

## 약점: 실제 구현의 과설계 함정

이론과 실무 사이의 괴리도 반복적으로 제기되는 문제다.
HN 커뮤니티에서 jrochkind1은 “아이디어는 좋지만, 헥사고날 아키텍처를 따른다고 하는 실제
코드를 볼 때마다 과설계되고 다루기 힘든 엉망진창이었다”고 지적한다.[^jrochkind1]
패턴 자체의 문제가 아니라, 원칙을 기계적으로 적용한 결과다.

반대로 구독 서비스, 결제 처리 서비스 등을 헥사고날 아키텍처로 구현한 zenogais는
“처음에는 과설계처럼 느껴지지만, 중장기적으로 얻는 유연성이 이를 충분히 보상한다”고
반론한다.[^zenogais]
이 긴장 관계는 패턴 적용의 적절한 범위 판단이 구현 성패를 가른다는 점을 시사한다.

## 약점: “헥사고날”이라는 이름의 역설

Cockburn 본인이 육각형은 시각적 편의를 위한 선택이지 6개 포트와 무관하다고 설명하지만,
이름 자체가 6이라는 숫자를 연상시켜 끊임없이 혼란을 유발한다.
“포트와 어댑터”가 더 서술적이고 정확한 이름임을 저자 스스로 인정하면서도,
“헥사고날 아키텍처”라는 이름이 살아남아 확산된 것은
아키텍처 명명(naming)의 역설적 사례다.
시각적으로 기억하기 쉬운 이름이 더 정확한 이름을 이긴다.

jerf도 같은 문제를 지적한다.[^jerf] “아이디어는 훌륭하고 항상 활용하지만, 이름이 늘
이상하게 느껴졌다. 저자 스스로 인정하듯 임의적이다. 'port-based'나 'microkernel-like'
같은 이름이 더 낫지 않을까.” 더 나아가 그는 “archipelago programming”이라는 대안 은유를
제안하기도 했다.[^jerf-archipelago] “각자 명확한 메시지 패싱 항로로 연결된 기능의 섬들”이라는
이 은유는 포트와 어댑터의 구조를 시각적으로 포착하면서도 숫자의 함정에서 자유롭다.

## 인사이트

## 포트는 기술이 아닌 의도(intent)로 정의된다

기상 시스템 팀은 인터페이스를 “wire feed”, “answering machine”, “GUI”, “database”라는
기술 이름으로 분류했다.
Cockburn은 이를 “trigger-data port”, “notification port”, “admin port”, “subscriber data port”로
재정의하길 제안했고, 그 순간 새로운 기술(HTTP, 이메일)을 추가하는 문제가 간단해졌다.

이것이 포트 설계의 핵심 원칙이다.
기술로 포트를 나누면, 새 기술이 등장할 때마다 새 포트가 필요하고 아키텍처가 흔들린다.
목적으로 포트를 나누면, 새 기술은 기존 포트에 새 어댑터를 붙이는 일이 된다.
이 구분은 객체지향의 “무엇을 하는가”와 “어떻게 하는가” 분리 원칙의 아키텍처 수준 구현이다.

실무에서 포트 설계를 검증하는 좋은 질문은 “이 포트의 이름에서 기술 이름을 빼도 의미가 통하는가?”다.
`DatabasePort`는 기술 이름이 들어가 있으므로 잘못된 추상화다.
`RateRepositoryPort`는 목적이 드러나므로 올바른 방향이다.
`UserNotificationPort`는 기술(SMS, 이메일, 푸시) 중립적이므로 좋다.

이 원칙은 현대의 인터페이스 설계, API 게이트웨이 설계, 이벤트 스키마 명명에까지 확장된다.
기술 중립적 이름을 가진 계약(contract)은 구현 변경에 탄력적이고,
기술 종속적 이름을 가진 계약은 기술 변경 시 계약 자체를 다시 협상해야 한다.

## 격리 가능성(Isolability)이 아키텍처의 최우선 가치다

Cockburn은 “The ultimate benefit of a ports and adapters implementation is the ability
to run the application in a fully isolated mode”[^1]라고 명시한다.
이것은 단순히 테스트 편의의 문제가 아니다.
격리 가능성은 개발 팀이 외부 시스템의 가용성에 종속되지 않는다는 의미이며,
이는 팀 자율성(team autonomy)의 기술적 조건이다.

Conway's Law는 조직 구조가 시스템 설계에 반영된다고 말한다.
반대 방향의 명제도 성립한다.
격리 불가능한 시스템은 팀 간 결합을 강제한다.
A 팀이 B 팀의 데이터베이스 없이 일할 수 없다면,
B 팀의 업무 중단이 A 팀의 배포를 막는다.
포트와 어댑터는 이 결합을 구조적으로 끊는다.

“Distributed, Large-Team Development” 절에서 Cockburn이 설명하는 시나리오가 이를 증명한다.
각 팀이 FIT + 목 조합으로 독립적으로 빌드하고, 실제 컴포넌트가 완성되면 목을 교체한다.
이것은 단순한 테스트 기법이 아니라 팀 간 병렬 개발을 가능케 하는 조직 설계 도구다.
현대 마이크로서비스 아키텍처에서 계약 테스트(contract testing)나
소비자 주도 계약(consumer-driven contracts)이 주목받는 이유와 정확히 맞닿아 있다.

## 레이어드 아키텍처의 실패는 기술적 문제가 아니라 강제 메커니즘의 부재다

Cockburn은 레이어드 아키텍처가 실패하는 이유를 “약속을 지키지 않아서”가 아니라
“약속 위반을 감지할 메커니즘이 없어서”라고 진단한다.
이 진단은 소프트웨어 설계 원칙 전반에 적용되는 깊은 통찰이다.

규칙만 있고 구조가 없는 아키텍처는 시간이 지남에 따라 반드시 퇴화한다.
팀이 교체되고, 일정 압박이 생기고, 원래 설계자가 떠나면,
“이 레이어에는 비즈니스 로직을 넣지 않는다”는 약속은 점점 흐려진다.
그러나 인터페이스를 통해서만 내부에 접근할 수 있고
컴파일러가 직접 참조를 거부하는 구조라면, 약속을 지키는 비용이 거의 없다.

이것은 코드 리뷰, 린팅 규칙, 아키텍처 테스트(ArchUnit 같은 도구) 모두에 해당하는 원칙이다.
“좋은 개발자라면 이렇게 하지 않을 것”이라는 기대보다,
“잘못된 방향으로 코드를 짜면 빌드가 실패하는” 구조를 만드는 것이 훨씬 강력하다.
포트와 어댑터 패턴의 진정한 가치는 좋은 설계를 강제하는 구조적 메커니즘을 제공한다는 데 있다.

## 포트와 어댑터는 아키텍처 패턴이자 인식론적 전환이다

이 패턴을 단순히 “테스트하기 좋게 코드를 구조화하는 방법”으로 이해하면 절반만 이해한 것이다.
Cockburn이 유스케이스 작성 방식에 대해 언급하는 부분이 그 증거다.

“Use cases should generally be written at the application boundary,
to specify the functions and events supported by the application,
regardless of external technology.”[^2]

이것은 분석가와 개발자가 애플리케이션의 본질(essence)과
우연적 속성(accidental properties)을 구분하는 방식에 관한 이야기다.

유스케이스에 특정 UI 기술이나 데이터베이스 쿼리가 등장한다면,
그 유스케이스는 본질이 아닌 현재의 구현에 묶여 있는 것이다.
애플리케이션 경계에서 정의된 유스케이스는 “무엇을 해야 하는가”에 관한 것이고,
어댑터는 “그것을 현재 기술로 어떻게 구현하는가”에 관한 것이다.

이 인식론적 분리는 도메인 주도 설계(DDD)의 도메인 레이어 개념,
클린 아키텍처의 엔티티/유스케이스 구분,
함수형 프로그래밍의 순수 함수와 부작용 분리와 모두 연결된다.
서로 다른 이름으로 불리지만, 이들은 모두 동일한 근본 통찰을 다른 어휘로 표현한다.
본질적인 것과 우연적인 것을 분리하라.

## Hexagonal Architecture에 대한 흔한 오해들

### Hexagonal Architecture가 Layered Architecture보다 우월하다

Cockburn의 2005년 논문은 Layered Architecture를 나쁜 아키텍처로 규정하지 않는다.
원문 Figure 3이 이를 직접 보여준다: 같은 애플리케이션을 3레이어 아키텍처 다이어그램에 매핑한 그림이 논문 안에 있다.

Cockburn이 레이어드 다이어그램의 문제로 꼽은 것은 두 가지다.

1. “사람들이 레이어 다이어그램의 '선'을 진지하게 받아들이지 않는다.”
2. “애플리케이션에는 두 개 이상의 포트가 있을 수 있어 아키텍처가 1차원 레이어 다이어그램에 맞지 않는다.”

두 문제 모두 다이어그램의 표현(representation) 문제다.
Cockburn도 명시한다: “많은 애플리케이션은 두 개의 포트만 가진다: 사용자 측 대화와 데이터베이스 측 대화.
이는 비대칭적인 모습을 만들어내고, 애플리케이션을 1차원적인 3, 4, 5레이어 스택 아키텍처로
구성하는 것이 자연스러워 보이게 한다.” 즉 포트가 두 개뿐인 애플리케이션에서 레이어드 표현은 자연스럽다.

육각형을 택한 이유는 이렇다: “그림을 그리는 사람이 1차원 레이어드 다이어그램에 구애받지 않고
필요에 따라 포트와 어댑터를 삽입할 공간을 주기 위해서.” 표현 방식의 선택이지, 대립하는 패러다임이 아니다.

### Layered Architecture의 문제는 강제 메커니즘의 부재다

원문에 “약속 위반을 감지할 메커니즘이 없었기에”라는 표현이 있다.
이 구절을 Layered Architecture의 핵심 진단으로 읽으면,
“강제 메커니즘만 도입하면 Layered Architecture도 같은 효과를 낸다”는 결론이 나온다.
그러나 이 구절이 가리키는 것은 과거의 실패한 시도, 즉 새 레이어를 추가하면서
“이번엔 비즈니스 로직을 넣지 않겠다”고 약속하던 관행이다.

Cockburn의 핵심 진단은 따로 있다:
“사용자 측 문제와 서버 측 문제는 사실 동일한 설계·프로그래밍 오류,
즉 비즈니스 로직과 외부 개체 간 상호작용의 얽힘에서 비롯된다.”
문제는 얽힘이다.

그리고 Cockburn이 제시하는 해결책은 구조다.
포트와 어댑터 패턴에서 애플리케이션은 포트를 통해서만 외부와 통신한다.
논문 첫 문장이 목표를 명확히 한다:
“UI나 데이터베이스 없이도 동작하는 애플리케이션을 만들어라.”
자동화 테스트가 가능해지는 것은 이 구조에서 따라오는 결과이지,
그것을 가능케 하기 위해 강제 수단을 추가하는 것이 해결책이 아니다.
