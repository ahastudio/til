# 월 $20 스택으로 월매출 $10K 회사를 여러 개 운영하는 법

원문: <https://stevehanov.ca/blog/how-i-run-multiple-10k-mrr-companies-on-a-20month-tech-stack>

## 요약

Steve Hanov는 단일 VPS, Go, SQLite라는 최소 스택으로
월 $10K MRR 이상의 SaaS 제품을 여러 개 운영한다.
핵심 철학은 "극단적 비용 효율이 벤처 펀딩과 동일한 런웨이를 제공한다"는 것이다.

인프라 구성:

- **서버**: Linode/DigitalOcean 단일 VPS ($5~10/월), 1GB RAM + swapfile
- **백엔드**: Go — 정적 바이너리, `scp` 한 번으로 배포, 수만 RPS 처리 가능
- **데이터베이스**: SQLite WAL 모드 — 원격 PostgreSQL TCP보다 로컬 파일 접근이 빠름
- **AI**: 개인 RTX 3090으로 VLLM 로컬 실행, 프론티어 모델은 OpenRouter 경유
- **개발 도구**: GitHub Copilot (~$60/월)

운영 중인 제품으로 websequencediagrams.com, zwibbler.com, eh-trade.ca,
rhymebrain.com, rapt.ink 등이 있다.

## 분석

### 복잡성 제거 자체가 경쟁력

서버가 하나이므로 로그 위치를 고민할 필요가 없고,
SQLite 파일이므로 백업은 파일 복사 하나로 끝나며,
Go 바이너리이므로 의존성 지옥이 없다.

### 극단적인 비용 구조

월 $20 인프라에서 월 $10K 매출이면 인프라 비용 비율이 0.2%다.
이는 대부분의 SaaS가 매출의 10~30%를 인프라에 쓰는 것과 극적으로 대비된다.
VC 자금 없이도 거의 무한한 런웨이를 확보하는 구조다.

### Go + SQLite 조합의 기술적 근거

Go는 정적 바이너리 배포와 높은 동시성 처리가 강점이고,
SQLite WAL 모드는 NVMe에서 10만 TPS 이상을 처리할 수 있다.
로컬 SQLite가 도메인 소켓을 쓰는 PostgreSQL보다도 빠르다는 것은
네트워크 계층 자체가 오버헤드라는 사실을 상기시킨다.

## 비평

### 단일 장애점(SPOF) 리스크

단일 서버 아키텍처의 가장 큰 리스크는 단일 장애점이다.
서버 장애 시 모든 서비스가 동시에 내려가며,
하드웨어 장애 시 데이터 복구도 백업 주기에 의존한다.
"수만 RPS"를 처리할 수 있다는 것과
실제로 트래픽 급증을 감당할 수 있다는 것은 다른 문제다.

### SQLite 쓰기 동시성 제한

WAL 모드에서도 쓰기는 단일 스레드이므로,
쓰기 집약적인 워크로드에서는 병목이 될 수 있다.
읽기 중심인 SaaS에는 적합하지만 모든 서비스 유형에 일반화하기는 어렵다.

### "$20 스택"이라는 표현의 마케팅성

GitHub Copilot $60, OpenRouter 사용료, 개인 GPU 전기료와 감가상각까지 포함하면
실제 운영 비용은 상당히 달라진다.

## 인사이트

### "스케일이 불필요한 구조 설계"라는 패러다임

통상적으로 "최적화는 나중에"라고 말하지만,
Hanov의 접근은 처음부터 스케일이 필요 없는 구조를 선택한 것이다.
1GB RAM 서버로 충분한 규모의 사업만 하겠다는 의식적 선택이며,
이는 DHH의 "Majestic Monolith"이나 Pieter Levels의 단일 서버 운영과 같은 맥락이다.
기술 선택이 사업 전략을 규정하는 것이 아니라, 사업 규모가 기술 선택을 규정한다.

### 인디 해커의 기술 스택은 "마찰 최소화"로 수렴한다

Go + SQLite + 단일 VPS는 2024년 기준 "핫한" 스택이 아니다.
그러나 배포 마찰(scp 한 번), 운영 마찰(서버 하나, DB 파일 하나),
디버깅 마찰(로그가 한 곳)을 극소화한다.
성공적인 1인 SaaS 운영자들이 공통적으로 보여주는 패턴은
기술의 신규성이 아니라 운영 오버헤드의 최소화다.
Hacker News의 SQLite vs PostgreSQL 논쟁에서도
핵심은 기술 우열이 아니라 "운영 맥락에서의 적합성"이었다.

### "서버가 하나면 관측 가능성 문제가 사라진다"는 역설

분산 시스템에서 관측 가능성(observability)은 가장 비싸고 어려운 문제 중 하나다.
OpenTelemetry, Datadog, 분산 트레이싱 등의 인프라를 유지하는 비용 자체가
단일 서버 월 비용의 수십 배를 넘는다.
"로그가 어디 있는지 안다"는 한 문장이 수천 달러의 관측 도구를 대체하는 셈이다.
복잡성을 해결하는 가장 효과적인 방법은 복잡성 자체를 제거하는 것이라는 교훈이다.
