# Go Concurrent Map 벤치마크

<https://github.com/puzpuzpuz/go-concurrent-map-bench>

Go의 동시성 해시맵 구현체 5종을 비교한 벤치마크.
xsync 저자(puzpuzpuz)가 만들었지만
중립성을 지키려 노력했다고 밝혔다.

## 구현체별 내부 구조

### `sync.Map` (stdlib)

Go 1.24부터 내부가 `HashTrieMap`으로 교체되었다.
16-way 분기 해시 트라이(hash trie) 구조로,
읽기는 atomic 포인터 순회로 lock-free다.
쓰기는 노드 단위 mutex를 잡아 영향 범위가
작은 서브트리로 한정된다.
트라이는 삽입 시 지연(lazy) 성장한다.

Go 1.23 이전의 `sync.Map`은 dirty map 복사 방식이라
쓰기가 잦으면 전체 맵을 복사하는 비용이 컸다.
1.24의 HashTrieMap 전환은 근본적인 구조 변경이다.

### `xsync.Map`

캐시 라인(64바이트) 크기 버킷에 최대 5개 엔트리를 담는다.
각 버킷은 자체 mutex를 가지며, 읽기는 atomic load로
완전한 lock-free다.

조회 시 SWAR(SIMD Within A Register) 기법으로
엔트리별 메타데이터 바이트를 한 번에 비교한다.
64비트 레지스터 하나로 8바이트를 동시에 필터링하므로
키 비교 전에 대부분의 후보를 걸러낸다.

리사이징은 협력적(cooperative)이다.
한 goroutine이 리사이징을 시작하면,
이후 맵에 접근하는 모든 goroutine이 버킷 마이그레이션을
분담한다. 리사이징 중에도 읽기·쓰기가 가능하다.

### `cornelk/hashmap`

해시 테이블 인덱스와 정렬 연결 리스트를 결합한
lock-free 구조다. 삽입·수정·삭제 모두
atomic CAS(Compare-And-Swap)로 처리한다.

백그라운드 goroutine이 fill factor 50% 초과 시
테이블 리사이즈를 트리거한다.
연결 리스트가 정렬 상태를 유지하므로
동시 순회 시 일관된 순서를 보장한다.

단, 정렬 연결 리스트의 순회 비용은 O(n)이다.
데이터가 늘면 CAS 재시도 빈도와 순회 비용이
동시에 증가해 성능이 급락한다.

### `alphadose/haxmap`

Harris의 lock-free 연결 리스트 알고리즘 위에
해시 테이블 인덱스를 얹은 구조다.
해싱에는 xxHash를 사용하고,
모든 변이(mutation)는 atomic CAS로 수행한다.

삭제는 lazy 방식이다. 노드를 먼저 논리적으로
마킹한 뒤, 이후 순회 시 물리적으로 제거한다.
load factor 50% 초과 시 자동 리사이즈한다.

### `orcaman/concurrent-map`

가장 단순한 접근. 32개 고정 샤드에
일반 Go `map`을 넣고 각 샤드를 `sync.RWMutex`로
보호한다. 키를 FNV-32로 해싱해 샤드에 배정한다.

단순하고 예측 가능하지만, 32개 샤드가 상한이므로
코어 수가 32를 넘으면 확장성에 한계가 온다.
Range는 채널 기반 API로 구현되어
반복(iteration) 성능이 가장 느리다.

## 벤치마크 설계

### 워크로드

permille(천분율) 기반 랜덤 연산 선택을 사용한다.

| 워크로드       | 읽기   | 쓰기   | 삭제   |
| -------------- | ------ | ------ | ------ |
| 100% 읽기      | 100%   | 0%     | 0%     |
| 99% 읽기       | 99%    | 0.5%   | 0.5%   |
| 90% 읽기       | 90%    | 5%     | 5%     |
| 75% 읽기       | 75%    | 12.5%  | 12.5%  |
| Range + 경합   | 전체 순회 중 백그라운드 쓰기   |

100% 읽기는 warm-up 전용이다.
실제 프로덕션에서 쓰기가 0%인 맵은 거의 없으므로,
99% 읽기부터가 실질적인 비교 기준이다.

### 키 타입

- **string 키:** 긴 공통 접두사를 붙여 해싱 부하를
  의도적으로 높였다. 해시 함수의 품질 차이가
  성능에 반영되도록 설계한 것이다.
- **int 키:** 해싱 비용이 낮아 순수한 자료구조
  성능을 비교할 수 있다.

### 맵 크기와 메모리 계층

| 크기      | 메모리   | 타겟 계층 | 의미                       |
| --------- | -------- | --------- | -------------------------- |
| 100       | ~15 KB   | L1 캐시   | 캐시 미스 거의 없음        |
| 1,000     | ~150 KB  | L2 캐시   | L1 미스, L2 히트           |
| 100,000   | ~15 MB   | L3 캐시   | L2 미스, L3 히트           |
| 1,000,000 | ~150 MB  | RAM       | 캐시 미스 빈번, 메모리 접근 |

맵 크기를 CPU 캐시 계층에 매핑한 것이 이 벤치마크의
핵심 설계다. 같은 알고리즘이라도 데이터가 어느 계층에
위치하느냐에 따라 성능이 수십 배 달라진다.

### Warm-up 변형

- **WarmUp:** 벤치마크 시작 전 맵을 미리 채운다.
  정상 상태(steady state) 성능을 측정한다.
- **NoWarmUp:** 빈 맵에서 시작한다.
  초기 삽입과 리사이징 비용이 포함된다.
  100% 읽기는 빈 맵에서 의미가 없으므로 생략한다.

### 환경

- **CPU:** AMD Ryzen 9 7900 12코어(24스레드)
- **OS:** Linux (amd64)
- **Go:** go1.26.0
- **GOMAXPROCS:** 1, 4, 8, 12
- **반복:** `-benchtime 3s -count 3`

## 결과 분석

### 읽기 전용 (100% reads)

모든 구현체가 읽기에서는 lock-free 또는
읽기 최적화 경로를 제공한다.
소규모(100, 1,000)에서는 구현체 간 차이가 작다.

대규모(10만, 100만)에서 `xsync.Map`이
SWAR 메타데이터 필터링 덕분에 가장 빠르다.
`orcaman/concurrent-map`은 `RWMutex`의
읽기 락 오버헤드 때문에 읽기 전용에서도 최하위다.

### 혼합 워크로드 (99%, 90%, 75% reads)

쓰기가 섞이면 구현체 간 격차가 급격히 벌어진다.

**99% 읽기:** 쓰기가 1%뿐인데도 lock-free 구현체들의
CAS 경합이 시작된다. `xsync.Map`은 per-bucket mutex
범위가 작아 영향이 미미하다.

**90% 읽기:** `cornelk/hashmap`이 대규모에서
눈에 띄게 뒤처지기 시작한다.
정렬 연결 리스트의 CAS 재시도가
데이터 크기에 비례해 증가하기 때문이다.

**75% 읽기:** 쓰기가 25%면 경합이 본격화된다.
`xsync.Map`과 나머지의 격차가 가장 크다.
`sync.Map`은 노드 단위 mutex 덕에 2위를 유지하지만,
할당 비용(9 B/op)이 `xsync.Map`(2 B/op)의 4.5배다.

### Range (반복 + 경합)

모든 goroutine이 맵을 순회하는 동안
백그라운드 goroutine이 계속 쓰기를 수행한다.

`orcaman/concurrent-map`은 채널 기반 Range API라
순회 자체가 느리고, 채널 할당 오버헤드까지 있다.
`xsync.Map`과 `sync.Map`이 상위권이다.

### 할당량 (B/op)

WarmUp 기준, 맵 크기와 GOMAXPROCS에 무관하게 일정하다.
모든 구현체가 0 allocs/op(할당 횟수)이지만,
바이트 기준(B/op)에서는 차이가 난다.

**String 키:**

| 라이브러리               | 100% | 99%  | 90%  | 75%  |
| ------------------------ | ---- | ---- | ---- | ---- |
| `sync.Map`               | 0    | 0    | 3    | 9    |
| `xsync.Map`              | 0    | 0    | 1    | 2    |
| `cornelk/hashmap`        | 0    | 0    | 1    | 4    |
| `alphadose/haxmap`       | 0    | 0    | 1    | 3    |
| `orcaman/concurrent-map` | 0    | 0    | 0    | 0    |

**Int 키:**

| 라이브러리               | 100% | 99%  | 90%  | 75%  |
| ------------------------ | ---- | ---- | ---- | ---- |
| `sync.Map`               | 0    | 0    | 3    | 8    |
| `xsync.Map`              | 0    | 0    | 0    | 2    |
| `cornelk/hashmap`        | 0    | 0    | 1    | 4    |
| `alphadose/haxmap`       | 0    | 0    | 1    | 3    |
| `orcaman/concurrent-map` | 0    | 0    | 0    | 0    |

`orcaman/concurrent-map`가 0인 이유는 일반 Go map이
기존 키 덮어쓰기 시 할당하지 않기 때문이다.
`sync.Map`은 HashTrieMap 노드 생성 비용 때문에
쓰기 비율이 올라갈수록 할당이 가파르게 증가한다.
`xsync.Map`은 비샤드 구현체 중 할당이 가장 낮다.

## 구현체별 종합 평가

### `sync.Map` — 기본값의 재탄생

Go 1.24의 HashTrieMap 전환은 `sync.Map`의
위상을 바꿔놓았다. 이전에는 "읽기 편향 전용"이라는
딱지가 붙어 있었지만, 이제는 혼합 워크로드에서도
준수한 성능을 보인다.

그러나 쓰기 할당 비용이 모든 구현체 중 가장 높다.
75% 읽기에서 9 B/op은 `xsync.Map`의 4.5배다.
GC 압력에 민감한 환경이라면 이 차이가 누적된다.

외부 의존성 없이 합리적 성능이 필요할 때 선택한다.

### `xsync.Map` — 설계의 승리

세 가지 설계 결정이 성능을 만든다.

1. **캐시 라인 정렬 버킷:** 하나의 버킷이
   하나의 캐시 라인에 들어간다.
   CPU가 인접 엔트리를 한 번의 메모리 접근으로
   가져올 수 있다.

2. **SWAR 메타데이터:** 키 전체를 비교하기 전에
   1바이트 메타데이터로 후보를 걸러낸다.
   SIMD 명령어 없이 일반 산술 연산으로 구현해
   모든 CPU에서 동작한다.

3. **협력적 리사이징:** 리사이징 비용을 모든
   goroutine이 나눠 부담한다.
   한 goroutine에 부하가 집중되지 않으므로
   tail latency가 낮다.

거의 모든 워크로드, 모든 맵 크기, 모든 병렬 수준에서
1위다. 외부 의존성을 수용할 수 있다면 최선의 선택이다.

### `cornelk/hashmap` — 이론과 실전의 괴리

lock-free 알고리즘은 이론적으로 진행 보장(progress
guarantee)을 제공한다. 하지만 실전에서는
CAS 재시도와 연결 리스트 순회 비용이
캐시 미스와 결합해 성능을 파괴한다.

10만 이상에서 성능이 너무 떨어져
벤치마크에서 아예 제외되었다.
소규모 읽기 전용 캐시 외에는 추천하기 어렵다.

### `alphadose/haxmap` — 소규모에서만 빛남

Harris 알고리즘의 lazy 삭제는
삭제가 빈번한 워크로드에서 유리할 수 있지만,
이 벤치마크에서는 그 장점이 드러나지 않는다.

소규모 읽기 전용에서는 양호하지만,
병렬 수준이 올라가면 CAS 경합으로
스케일링이 정체된다.

### `orcaman/concurrent-map` — 단순함의 한계

가장 이해하기 쉽고, 디버깅하기 쉽고,
예측하기 쉬운 구현체다. 할당도 0이다.

그러나 32개 고정 샤드가 천장이다.
GOMAXPROCS 12에서 이미 경합이 보이며,
코어 수가 더 많은 시스템에서는 악화된다.
채널 기반 Range API는 반복 성능의 병목이다.

낮은 병렬 수준에서 단순성이 중요하다면 고려한다.

## 인사이트

### lock-free가 항상 빠른 건 아니다

`cornelk/hashmap`과 `alphadose/haxmap`은
lock-free지만 `xsync.Map`(per-bucket mutex)보다
느리다. lock-free는 진행 보장이지 성능 보장이 아니다.

핵심은 동기화 방식이 아니라 데이터 배치다.
캐시 라인에 맞춘 구조가 CAS 재시도 없는 lock-free보다
빠를 수 있다. 알고리즘의 이론적 복잡도보다
실제 하드웨어와의 궁합이 성능을 결정한다.

### 메모리 계층이 알고리즘을 이긴다

맵 크기 100(L1)에서는 구현체 간 차이가 작다.
100만(RAM)에서는 수배 이상 벌어진다.
같은 O(1) 조회라도 캐시 히트와 캐시 미스는
10~100배 차이가 난다.

`xsync.Map`이 대규모에서 압도적인 이유는
알고리즘이 뛰어나서가 아니라,
데이터 배치가 캐시에 최적화되어 있기 때문이다.
고성능 자료구조 설계에서 메모리 레이아웃은
알고리즘 선택만큼, 어쩌면 그 이상으로 중요하다.

### `sync.Map`은 이제 진지한 선택지다

Go 1.24 이전에는 "읽기 전용 아니면 쓰지 마라"가
통설이었다. HashTrieMap 전환 이후,
혼합 워크로드에서도 2~3위를 기록한다.

stdlib만으로 해결해야 하는 상황이라면
더 이상 성능 때문에 주저할 이유가 없다.
단, 쓰기가 잦고 GC 압력이 민감한 환경이라면
`xsync.Map`을 검토해야 한다.

### 쓰기 비율이 선택을 결정한다

읽기 100%면 대부분의 구현체가 비슷하다.
쓰기 1%만 섞여도 차이가 보이기 시작하고,
10%면 구도가 명확해지며,
25%면 `xsync.Map`과 나머지의 격차가 확연하다.

프로덕션 워크로드의 읽기/쓰기 비율을 먼저 측정하고,
그 비율에 맞는 벤치마크 결과로 판단해야 한다.
"읽기가 대부분"이라는 직감은 위험하다.
실측 없이 99%라고 가정하면 안 된다.

### 샤드 설계의 교훈

`orcaman/concurrent-map`의 32개 고정 샤드는
간결하지만 확장 불가능한 설계의 전형이다.
GOMAXPROCS가 샤드 수에 근접하면 경합이 시작되고,
초과하면 성능이 정체된다.

반면 `xsync.Map`의 동적 버킷 확장은
코어 수에 비례해 성능이 올라간다.
동시성 자료구조에서 고정 분할(fixed partitioning)은
하드웨어 진화에 대응하지 못하는 기술적 부채다.

## 선택 가이드

```text
외부 의존성 허용?
├─ Yes → xsync.Map
└─ No  → sync.Map (Go ≥ 1.24)

맵 크기?
├─ ≤ 1,000 → 대부분 비슷. 단순함 우선이면
│             orcaman/concurrent-map도 가능
└─ ≥ 100,000 → xsync.Map 권장.
               cornelk/hashmap 제외

쓰기 비율?
├─ < 1%  → sync.Map으로 충분
├─ 1~10% → xsync.Map이 유리하기 시작
└─ > 10% → xsync.Map 강력 권장
```
