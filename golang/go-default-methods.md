# Go의 기본 메서드 구현하기

원문: <https://mcyoung.xyz/2025/08/25/go-default-methods/>

## 요약

Go 인터페이스는 구조적 타이핑(structural typing)을 사용한다.
Java나 Rust처럼 인터페이스를 명시적으로 구현 선언하는 것이 아니라, 타입이 인터페이스의
메서드 집합을 우연히 갖추고 있으면 자동으로 만족(satisfy)한 것으로 처리된다.
이 단순함은 표면적으로는 우아하지만 실제로는 여러 실용적 문제를 낳는다.
그 중 가장 큰 문제 중 하나가 인터페이스에 새로운 메서드를 추가할 수 없다는 것이다.
기본 구현(default implementation)을 메서드에 붙일 방법이 없기 때문이다.

이를 잘 보여주는 사례가 표준 라이브러리의 `flag.Value`다.
`IsBoolFlag() bool` 같은 “선택적 메서드”가 문서에만 명시되고 인터페이스에는 포함되지
않아서, 런타임이 타입 단언(type assertion)으로 이를 확인해야 한다.
`flag.Getter`는 설계 실수로 나중에 별도 인터페이스로 추가된 사례로, `flag.Value`에
처음부터 `Get() any`가 있었다면 불필요했을 것이다.

저자는 구조체 임베딩(struct embedding)을 이용해 이 문제를 우아하게 우회하는 방법을
제안한다.
인터페이스를 만족하는 모든 타입이 `ValueDefaults` 같은 구조체를 임베드하도록 강제하면,
해당 구조체에 기본 구현을 넣어 인터페이스가 확장될 때도 기존 코드가 깨지지 않는다.
미공개(unexported) 인터페이스 메서드를 이용해 외부 패키지가 임베드 없이 인터페이스를
만족하지 못하도록 강제하는 트릭도 소개한다.

이 방법의 한계와 함께, 저자는 언어 수준의 세 가지 해결책도 탐구한다.
첫째는 인터페이스 메서드에 직접 기본 구현을 붙이는 방식이지만 구조적 타이핑의 비의도성
때문에 메서드 집합 충돌 문제가 발생한다.
둘째는 선택적 메서드(`?Baz()`) 문법으로 vtable에 nil 함수 포인터를 두는 방식이다.
셋째는 관련 인터페이스의 itab 포인터를 현재 itab에 내장하는 방식이다.
저자는 선택적 메서드가 가장 실현 가능한 방향이라고 보지만, 지금은 임베드 구조체로
버티는 수밖에 없다고 결론 짓는다.

## 분석

### 구조적 타이핑의 근본적 비의도성

이 아티클에서 반복적으로 등장하는 핵심 개념은 “비의도성(non-intentionality)”이다.
Go 인터페이스를 만족하는 것은 명시적 선언 없이 메서드 집합이 우연히 일치하는 것으로
충분하다.
개발자가 특정 인터페이스를 구현하려는 의도를 표현할 공식 경로가 없으므로, 인터페이스를
변경했을 때 누가 영향을 받는지 추적하기 어렵다.
C++ 템플릿의 암묵적 개념(concept) 문제와 유사하다.

### 임베딩 트릭의 정밀도

저자가 제안하는 트릭은 Go의 세 가지 언어 특성을 조합한다.

1. **구조체 임베딩**: 임베드된 타입의 메서드 집합을 상속한다.
2. **미공개 인터페이스 메서드**: 패키지 외부에서는 직접 구현할 수 없어, 임베드 없이
   인터페이스를 만족하는 것을 원천 차단한다.
3. **타입 별칭(type alias)**: `type valueDefaults = flag.ValueDefaults`처럼 별칭을
   만들면 임베드 필드 이름이 별칭 이름을 따르므로 내부 이름을 숨길 수 있다.

이 세 가지가 맞물리면 “임베드를 강제하는 인터페이스 + 기본 구현을 담은 구조체”라는
패턴이 완성된다.

### io 패키지 예시와 성능 함의

`io.Reader`를 `io.Seeker`로 캐스팅하는 사례는 성능 관점에서 흥미롭다.
현재 `r.(io.Seeker)` 타입 단언은 itab 캐시 해시맵 조회를 수행하지만,
기본 메서드로 구현된 `r.Seeker()`는 itab의 함수 포인터 하나를 읽는 간접 점프로
처리된다.
I/O처럼 시스템 콜이 지배적인 영역에서는 의미 없는 차이지만, 타입 단언이 핫 패스에
있는 저수준 라이브러리에서는 실질적인 차이를 만들 수 있다.

### 언어 수준 해결책의 한계 분석

저자가 탐구하는 세 가지 언어 수준 해결책은 각기 다른 이유로 완전하지 않다.

- **기본 구현 직접 부착**: 구조적 타이핑에서 기본 메서드를 타입에 주입하는 시점이
  불명확하며, 리플렉션과의 호환성 문제가 발생한다.
- **선택적 메서드(`?Baz()`)**: vtable에 nil 포인터를 두는 방식으로 실현 가능성이
  가장 높지만, 인터페이스 타입 단언 시 메서드 집합 판단이 복잡해진다.
- **관련 인터페이스 itab 내장**: ABI를 변경하므로 기존 인터페이스에 소급 적용이
  불가능하다. PGO 기반으로 우회할 수 있으나 매우 복잡하다.

## 비평

### 강점: 실용적 해법과 언어 이론의 균형

이 아티클의 장점은 실무에서 바로 쓸 수 있는 기법(임베드 트릭)과 언어 설계 수준의
이론적 탐구를 동시에 제공한다는 점이다.
`flag.Value`라는 구체적 사례에서 출발해 점점 일반화되는 구조가 논리적으로 탄탄하다.
미공개 메서드를 이용한 강제 임베딩 트릭은 문서화가 부족한 Go의 숨겨진 기능을 영리하게
활용하는 사례로 실용 가치가 높다.

### 한계: godoc 버그에 대한 과도한 의존

타입 별칭으로 임베드 필드를 숨기는 기법이 “godoc 버그”에 의존한다는 점을 저자 스스로
인정한다.
문서화되지 않은 동작을 패턴의 일부로 권장하는 것은 불안정하다.
Go 팀이 이 “버그”를 수정하면 메서드들이 공개 문서에 노출되어 혼란을 야기할 수 있다.
공개 API 설계에서 이를 의존하는 것은 기술 부채를 감수하는 선택이다.

### 보완이 필요한 관점: 기존 코드베이스에서의 마이그레이션

이 패턴의 가장 큰 실용적 한계는 저자도 언급하듯 “처음부터 설계에 포함해야” 한다는
점이다.
기존 인터페이스에 소급 적용하면 모든 구현 타입이 임베드를 추가해야 하는 대규모 마이그레이션이
필요하다.
표준 라이브러리의 `flag.Value`나 `io.Reader` 같은 인터페이스에 이를 적용하는 것이
현실적으로 가능한지, 마이그레이션 경로가 어떻게 되는지에 대한 논의가 빠져 있다.

## 인사이트

### 인터페이스 설계의 “잠금 효과”는 Go만의 문제가 아니다

Go의 인터페이스에 메서드를 추가하지 못하는 문제는 언뜻 Go 특유의 구조적 타이핑이
만들어낸 병폐처럼 보이지만, 사실 공개 API 설계 전반에서 반복되는 패턴이다.
Java 8에서 인터페이스에 `default` 메서드를 추가한 것도, Python의 `collections.abc`가
믹스인 메서드를 제공하는 것도 같은 문제의 해결책이다.
REST API에서 하위 호환성을 유지하면서 응답 필드를 추가하는 문제와도 구조적으로 동일하다.
어떤 계약(contract)이든 일단 외부에 노출되면, 그것을 변경하는 비용은 설계 시점에
지불한 비용과 비교할 수 없을 만큼 커진다.
Go가 이 문제를 특히 고통스럽게 드러내는 이유는 “명시적 구현 선언”이라는 안전망이 없기
때문이다.
Java에서 인터페이스에 `default` 메서드를 추가하면 컴파일러가 모든 구현 클래스를
추적할 수 있다.
Go에서는 그 범위를 컴파일러가 알 수 없다.
이 차이가 “설계 실수를 수정할 수 있는 능력”에서 두 언어 사이의 근본적인 격차를 만든다.

### 미공개 메서드 트릭은 타입 시스템을 사회적 계약으로 바꾼다

미공개 인터페이스 메서드로 외부 구현을 차단하는 기법은 기술적으로 영리하지만,
더 흥미로운 점은 이것이 타입 시스템을 “사회적 계약(social contract)”으로 전환한다는
사실이다.
`value()` 같은 미공개 메서드를 인터페이스에 추가하면, 외부 패키지는 `ValueDefaults`를
임베드하는 것 외에 다른 선택지가 없다.
이는 Go 언어 차원에서 지원하지 않는 “이 인터페이스를 구현하려면 내가 승인한 방식으로만
가능하다”는 의도를 강제하는 방법이다.
본질적으로 이것은 명목적 타이핑(nominal typing)을 구조적 타이핑 위에 에뮬레이션하는
것이다.
Go가 “구조적 타이핑이 더 단순하다”고 선택했지만, 실제 라이브러리 설계에서 개발자들은
명목적 타이핑의 안전망을 다시 원하게 된다.
언어 설계자의 의도와 사용자의 실제 필요 사이의 간극이 이런 트릭을 낳는다.
이는 Go만의 현상이 아니다.
덕 타이핑(duck typing)을 쓰는 Python에서도 `Protocol`과 `ABC`가 생겨난 것, TypeScript에서
`interface`와 `class`가 공존하는 것, 모두 같은 긴장의 산물이다.
구조적 타이핑의 유연함을 원하면서도, 경계가 필요할 때는 명목적 타이핑의 엄격함을 원하는
것이 인간의 자연스러운 욕구이기 때문이다.

### itab의 숨겨진 비용과 Go의 성능 설계 철학

저자가 지나가듯 언급하는 사실, 즉 Go에서 임베드 인터페이스 간의 업캐스팅조차
전체 타입 단언 메커니즘을 거친다는 점은 Go 런타임의 설계 우선순위를 드러낸다.
Java의 JVM은 인터페이스 상속 계층을 컴파일 타임에 추적해서 업캐스팅을 O(1)으로
처리한다.
Go는 구조적 타이핑의 특성상 이런 최적화를 전역적으로 적용하기 어렵다.
`interface{ Foo(); Bar() }`에서 `interface{ Foo() }`로의 캐스팅은 두 인터페이스 사이의
“하위 집합” 관계를 런타임이 동적으로 확인해야 하기 때문이다.
저자가 제안하는 `A<-` 문법은 이 문제의 정교한 해결책이다.
프로그래머가 “이 인터페이스는 저 인터페이스로 자주 캐스팅된다”는 힌트를 명시하면,
컴파일러가 itab에 해당 포인터를 미리 내장할 수 있다.
이것은 사실상 런타임 다형성의 비용을 컴파일 타임 정보로 줄이는 시도다.
PGO(Profile-Guided Optimization)가 이 힌트를 자동으로 수집할 수 있다면 더 우아한
해결책이 될 것이다.
Go가 “단순함”을 추구하면서 동시에 “성능”을 원한다는 두 목표 사이의 긴장이 이런
설계 공간을 만들어낸다.
단순한 구조적 타이핑은 런타임 동적 처리를 요구하고, 성능은 컴파일 타임 정보를 원한다.
이 모순을 해소하는 방향이 Go의 다음 진화를 결정할 것이다.
