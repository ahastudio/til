# `go:linkname`으로 보증 파기하기

원문: <https://mcyoung.xyz/2026/03/31/shame/>

## 요약

Go에는 `//go:linkname`이라는 마법 주석이 있다.
C에서 `extern void* malloc(size_t bytes);`처럼 다른 파일의 심볼을 참조하듯,
Go에서는 이 주석으로 다른 패키지의 내부 심볼에 직접 링크할 수 있다.
문제는 Go 1.23 이전까지 피해 패키지의 동의 없이도 심볼 테이블을 마음대로 뒤질 수
있었다는 점이다.

Go 1.23에서는 표준 라이브러리 패키지에 한해, 명시적으로 옵트인(opt-in)된 심볼만
외부 패키지에서 `linkname`으로 참조할 수 있도록 제한했다.
그러나 이 정리 작업 과정에서 의존자가 50개 이상인 모듈이 런타임 내부 심볼을 남용한
사례가 광범위하게 발견되었고, 생태계 파괴를 막기 위해 일부 심볼에는 `linkname`
허용 주석을 달아 사실상 영구 지원을 선언했다.

저자는 이렇게 살아남은 “나쁜” 심볼들을 정리하고 자동으로 바인딩을 생성하는
`shame`이라는 라이브러리를 만들었다.
핵심 래퍼로는 직접 힙 할당을 수행하는 `shame.Alloc`, 슬라이스 리사이징 프리미티브인
`shame.Grow`, 내부 포인터 조회를 위한 `shame.FindObject`, 리플렉션 비용 없이
동적 맵 연산을 수행하는 `MapAssign*`/`MapIndex*` 패밀리 등이 있다.

저자는 “사용하지 말라”고 단호하게 말한다.
그러나 동시에 “어차피 쓸 사람이라면 문서화된 것이 낫다”는 해악 감소(harm
reduction) 논리로 라이브러리를 공개한 이유를 정당화한다.
그리고 만약 자신이 Go 프로젝트의 입장이었다면 그냥 전부 부숴버렸을 것이라고 덧붙인다.

## 분석

### go:linkname의 구조적 위험

`//go:linkname`은 단순한 편의 기능이 아니라 Go ABI 계층의 구멍이다.
`foo_stub`이 `github.com/victim/pkg.foo`의 별칭이 되는 순간, 두 심볼 사이의 ABI
일치 여부는 컴파일러도 링커도 검증하지 않는다.
타입 시그니처가 맞아야 하지만 미공개 타입 이름을 직접 쓸 수 없어서 충분히 비슷하게만
맞추면 된다.
여기에 `//go:noescape` 주석까지 잘못 달면 이스케이프 분석이 오염되고, 이는
조용한 메모리 오염으로 이어질 수 있다.

### 생태계 볼모 잡기와 Hyrum's 법칙

저자가 지적하는 핵심은 Hyrum's Law다.
충분히 많은 사람이 어떤 구현 세부사항에 의존하기 시작하면, 그것은 사실상 공개 API가
된다.
Go 프로젝트는 50개 이상의 의존자를 가진 패키지만 조사했는데도 광범위한 남용을
발견했다.
결국 생태계 파괴를 감수하지 못해 “영구 지원”을 선언한 심볼들이 생겨났고, 이는
의도치 않은 계약이 되었다.

### shame 라이브러리의 기술적 가치

`shame.Grow`는 리플렉션으로는 불가능한 `append(xs, make([]T, n)...)` 패턴과
동등한 동작을 제공한다.
`reflect.AppendSlice`는 이 관용구에 해당하는 기능이 없기 때문이다.
또한 `MapAssign*` 계열은 맵 항목에 대한 포인터를 반환해서 복사 비용 없이 읽기-수정-쓰기(RMW)
연산을 가능하게 한다.
이런 기능들은 성능이 극도로 중요한 저수준 라이브러리에서 실질적인 차별점을 만들 수 있다.

## 비평

### 해악 감소 논리의 한계

“어차피 쓸 사람이 있으니 문서화하겠다”는 해악 감소 논리는 마약 감소 정책에서 빌려온
개념이지만, 소프트웨어 라이브러리에 그대로 적용하기엔 맥락이 다르다.
마약은 수요 자체를 제거하기 어렵지만, 내부 API 남용은 공개 API 설계를 개선함으로써
수요를 줄일 수 있다.
`shame.Grow`가 필요한 이유 중 하나는 `reflect.AppendSlice`가 특정 패턴을 지원하지
않기 때문이다.
문서화된 우회로를 제공하는 것이 단기적으로는 편리하지만, 장기적으로는 공식 API에
해당 기능을 추가하도록 압력을 가하는 것이 더 건강한 생태계를 만든다.

### Go 프로젝트의 선택에 대한 평가

저자는 Go 프로젝트가 그냥 모두를 부숴버렸어야 한다고 주장하지만, 이는 현실을 단순화한다.
Go의 강점 중 하나는 강한 하위 호환성 보장이다.
그 신뢰가 생태계 성장의 토대가 되었다.
물론 내부 API 남용에 대한 제재 수위를 높여야 한다는 지적은 타당하다.
Go 1.23의 부분적 제한은 올바른 방향이지만, “50개 이상 의존자”라는 기준이 남용 허용의
기준이 되어버린 것은 인센티브 구조를 왜곡한다.

## 인사이트

### 마법 주석은 언어 설계의 실패를 드러낸다

`//go:linkname`이 마법 주석으로 존재한다는 사실 자체가 Go 언어 설계의 흥미로운 긴장을
보여준다.
Go는 명시적이고 단순한 언어를 표방하지만, 런타임과 표준 라이브러리 내부에는 일반
개발자가 접근할 수 없는 기능들이 존재한다.
그 격차를 메우는 공식적인 방법이 없을 때, 개발자들은 비공식적인 방법을 찾는다.
C에서 `static inline` 함수가 헤더에 노출되는 것이 ABI 안정성 문제를 일으키듯,
Go에서 `//go:linkname`은 내부 구현 세부사항이 사실상 공개 계약이 되는 경로를
제공한다.
언어 설계자가 “이런 기능이 필요 없도록 설계했다”고 주장하는 것과, 실제로 그 기능이
필요 없는 것은 다른 문제다.
`shame` 라이브러리가 존재한다는 것은 수요가 있다는 증거다.

### 성능 계층과 안전 계층의 분리 문제

`shame.Alloc`, `shame.Grow`, `MapAssign*`이 매력적인 이유는 리플렉션 계층을
우회해서 성능을 얻기 때문이다.
이는 Go에서 “안전하고 느린” 공개 API와 “빠르지만 위험한” 내부 API 사이의 성능 간극이
실제로 크다는 것을 의미한다.
Rust는 이 문제를 `unsafe` 블록으로 명시적으로 처리한다.
`unsafe` 코드는 눈에 띄고, 리뷰 대상이 되고, 감사가 가능하다.
Go의 `//go:linkname`은 주석 한 줄로 비슷한 효과를 내지만 훨씬 덜 가시적이다.
성능이 중요한 시스템 언어가 고수준 안전 API와 저수준 성능 API 사이의 명확한 계약을
언어 차원에서 제공하지 않으면, 개발자들은 비공식 경로를 찾게 된다.
Go가 진정한 시스템 프로그래밍 언어로 성숙하려면 이 격차를 언어 수준에서 해결해야
한다.

### 생태계 부채와 영구 지원의 역설

“50개 이상 의존자”를 가진 패키지가 내부 API를 남용하면 사실상 영구 지원이 보장된다는
역설은 소프트웨어 생태계의 구조적 문제를 드러낸다.
이는 인기 있는 라이브러리가 미문서화된 동작에 의존할수록 그 동작이 사실상 표준이 되는
Hyrum's Law의 전형적인 사례다.
C++ 표준 위원회가 `std::string`의 SSO(Small String Optimization) 구현을 바꾸지 못하는
것처럼, Go 프로젝트도 특정 런타임 내부 구현을 사실상 바꾸지 못하는 상황이 되었다.
더 흥미로운 점은 이 메커니즘이 자기강화적(self-reinforcing)이라는 것이다.
`shame` 라이브러리가 존재함으로써 더 많은 패키지가 해당 심볼에 의존하게 되고, 그럴수록
Go 프로젝트가 해당 심볼을 제거하기 더 어려워진다.
해악 감소를 위한 문서화가 의존성을 늘려 해악을 심화시키는 아이러니다.
이 역학을 끊으려면 인센티브 구조 자체를 바꿔야 한다.
즉, 내부 API 남용이 영구 지원으로 보상받는 것이 아니라, 공식 API 설계 개선으로
이어지는 경로가 필요하다.
