# How to Build a Coding Agent

<https://github.com/ghuntley/how-to-build-a-coding-agent>

## 요약

Geoffrey Huntley가 공개한 이 워크숍 리포지토리는 코딩 에이전트(coding agent)를 단계별로 직접 구현해보는 실습 자료다.
Go 언어와 Anthropic Claude API를 사용하며, 약 300줄의 코드로 동작하는 에이전트를 6단계에 걸쳐 점진적으로 완성한다.
블로그 포스트(<https://ghuntley.com/agent/>)에서 심층 해설도 제공한다.

첫 번째 단계는 단순한 챗봇(`chat.go`)에서 출발한다.
이 단계에서는 Claude API와 연결해 사용자 입력을 받아 응답을 출력하는 기본 이벤트 루프(event loop)를 구현한다.
이후 단계마다 도구(tool)를 하나씩 추가하며 에이전트의 능력을 확장한다.

6단계 구성은 다음과 같다.
파일 읽기(`read.go`), 디렉토리 탐색(`list_files.go`), 셸 명령 실행(`bash_tool.go`), 파일 편집(`edit_tool.go`), 코드 검색(`code_search_tool.go`)이
순서대로 추가된다.
최종 에이전트는 코드 읽기, 편집, 실행, 검색을 모두 수행하는 로컬 개발 보조 도구가 된다.

핵심 아키텍처는 “추론 루프(inferencing loop)”다.
에이전트는 사용자 입력을 받아 Claude에 전달하고, Claude가 도구 사용을 요청하면 해당 도구를 실행한 뒤 결과를 다시 Claude에 보낸다.
이 순환이 Claude가 최종 텍스트 응답을 반환할 때까지 반복된다.

컨텍스트 윈도우(context window) 관리도 중요하게 다뤄진다.
Claude Sonnet의 200k 토큰 윈도우는 시스템 프롬프트와 프레임워크 할당으로 인해 실제 사용 가능한 공간이 약 176k 토큰으로 줄어든다.
저자는 “컨텍스트 윈도우에 더 많이 할당할수록 성능이 저하된다”는 점을 명시적으로 경고한다.

## 분석

### 이벤트 루프 구조

에이전트의 핵심은 `Run()` 메서드 안의 이중 루프다.
외부 루프는 사용자 입력을 기다리고, 내부 루프는 Claude가 도구 사용을 멈출 때까지 추론과 도구 실행을 반복한다.
모든 도구 결과를 한 번에 수집해 한 번의 API 호출로 전달하는 배치(batch) 방식을 사용한다.

```go
// 내부 루프: 도구 사용이 없을 때까지 반복
for {
    var toolResults []anthropic.ContentBlockParamUnion
    var hasToolUse bool

    for _, content := range message.Content {
        switch content.Type {
        case "tool_use":
            hasToolUse = true
            // 도구 실행 후 결과 수집
        }
    }

    if !hasToolUse {
        break
    }
    // 수집한 결과를 한 번에 전달
    message, err = a.runInference(ctx, conversation)
}
```

### 도구 정의 방식

Go의 제네릭(generics)과 `jsonschema` 리플렉션을 활용해 JSON 스키마를 자동 생성한다.
입력 타입을 Go 구조체로 정의하면 `GenerateSchema[T]()` 함수가 Claude API에 전달할 스키마를 생성한다.
타입 안전성과 API 스키마 정의를 한 곳에서 관리할 수 있는 깔끔한 패턴이다.

```go
type EditFileInput struct {
    Path   string `json:"path"    jsonschema_description:"The path to the file"`
    OldStr string `json:"old_str" jsonschema_description:"Text to search for"`
    NewStr string `json:"new_str" jsonschema_description:"Text to replace old_str with"`
}

var EditFileInputSchema = GenerateSchema[EditFileInput]()
```

### 편집 도구의 안전 장치

`edit_file` 도구는 `old_str`이 파일 내에서 정확히 한 번만 등장해야 편집을 허용한다.
0회면 “not found” 에러, 2회 이상이면 “must be unique” 에러를 반환한다.
LLM이 잘못된 위치를 수정하는 사고를 방지하기 위한 최소한의 안전 장치다.

### 모델 선택에 대한 입장

저자는 모델을 “고안전(high-safety)”과 “에이전틱(agentic)” 스펙트럼으로 분류한다.
Claude Sonnet은 “행동 편향(biased towards action)” 모델로 설명되며, 도구 사용에 적극적인 성향을 가진다고 본다.
단, 이 분류는 저자의 실무 경험에서 나온 주관적 관찰이며, 공식 사양이 아니다.

## 비평

### 강점

워크숍 구조가 매우 명확하다.
각 단계가 독립적인 파일로 구성돼 있어 어느 단계에서든 시작하고 비교할 수 있다.
`--verbose` 플래그로 API 호출과 도구 실행 과정을 상세히 추적할 수 있어 학습에 효과적이다.

실제로 동작하는 최소한의 코드만 포함하고 있어 핵심 개념이 노이즈 없이 드러난다.
코드베이스 전체가 단일 Go 파일들로 구성돼 있어 의존성 없이 바로 실행할 수 있다.

### 약점

보안 고려가 거의 없다.
`bash` 도구는 임의의 셸 명령을 실행하며, 샌드박스나 허용 목록(allowlist)이 없다.
학습용으로는 충분하지만, 실제 환경에 그대로 배포할 경우 심각한 보안 위험이 된다.

컨텍스트 윈도우 관리에 대한 경고는 있지만 구현은 없다.
대화가 길어질수록 토큰 비용이 선형 증가하는 문제를 해결하는 방법(요약, 슬라이딩 윈도우 등)이 코드에 포함돼 있지 않다.

에러 처리가 일관되지 않다.
`ReadFile`은 `panic`을 사용하는 반면, 다른 함수는 에러를 반환한다.
학습 자료로서 이런 불일치는 좋은 패턴으로 학습되기 어렵다.

## 인사이트

### 에이전트의 본질은 루프다

코딩 에이전트를 처음 접하면 복잡한 추론 엔진이나 계획 시스템을 상상하기 쉽다.
그러나 이 워크숍이 보여주는 핵심은 그 반대다.
에이전트는 본질적으로 하나의 루프이며, “LLM 호출 → 도구 실행 → 결과 전달”을 반복하는 단순한 구조다.

이 단순성은 심오한 함의를 가진다.
에이전트의 지능은 루프 자체에 있는 것이 아니라 LLM에 있다.
프레임워크가 복잡해지는 이유는 에이전트를 “더 똑똑하게” 만들기 위해서가 아니라,
더 안전하게, 더 효율적으로, 더 관측 가능하게 만들기 위해서다.

LangChain, LlamaIndex, CrewAI 같은 에이전트 프레임워크들이 복잡해 보이는 이유를 이 관점에서 다시 볼 수 있다.
그 복잡성의 대부분은 루프를 감싸는 안전 장치, 로깅, 재시도 로직, 컨텍스트 관리다.
핵심 루프 자체는 이 워크숍의 코드처럼 여전히 단순하다.

이 이해가 없으면 프레임워크를 블랙박스로 사용하게 된다.
반면 루프의 구조를 이해하면 프레임워크의 어떤 부분이 왜 필요한지, 어디서 문제가 생기는지 추론할 수 있다.

### 도구 설계가 에이전트 능력을 결정한다

이 워크숍에서 에이전트가 단계마다 “더 똑똑해지는” 것처럼 느껴지는 이유는 모델이 개선되기 때문이 아니다.
도구가 추가될 때마다 Claude가 할 수 있는 행동의 공간(action space)이 확장되기 때문이다.

`bash` 도구 하나가 추가되는 순간 에이전트는 코드를 실행하고, 테스트하고, 빌드하고, 배포할 수 있게 된다.
이것은 모델의 능력이 아니라 도구 설계의 결과다.

이 관점은 에이전트 개발의 전략을 바꾼다.
더 좋은 프롬프트를 작성하는 것보다 더 좋은 도구를 설계하는 것이 에이전트 능력 향상에 더 직접적인 영향을 미친다.
도구의 설명(description), 입력 스키마의 명확성, 에러 메시지의 품질이 에이전트가 도구를 올바르게 사용하는 비율을 결정한다.

`edit_file` 도구의 “must be unique” 안전 장치가 좋은 예다.
이 제약이 없으면 Claude는 애매한 문자열 패턴으로 잘못된 위치를 수정할 수 있다.
도구 설계의 세부 사항이 에이전트 신뢰성의 핵심이 된다.

역사적 유사성을 찾자면, UNIX 철학의 “작고 명확한 도구를 파이프로 연결한다”는 원칙과 닮아 있다.
에이전트 시대의 도구 설계도 같은 원칙으로 수렴하고 있다.

### 컨텍스트 윈도우는 에이전트의 메모리이자 한계다

저자가 “컨텍스트에 더 많이 할당할수록 성능이 저하된다”고 경고하는 지점은 단순한 비용 문제가 아니다.
이것은 에이전트 아키텍처의 근본적인 제약이다.

현재의 LLM은 컨텍스트 윈도우 안의 모든 토큰에 어텐션(attention)을 적용한다.
윈도우가 가득 찰수록 모델이 관련 정보를 찾기 위해 처리해야 하는 노이즈가 증가한다.
200k 토큰 윈도우가 있다고 해서 200k 토큰을 채우는 것이 최적이 아닌 이유다.

이 제약은 에이전트 설계에 깊은 영향을 미친다.
장기 작업을 수행하는 에이전트는 컨텍스트를 어떻게 관리할지 명시적으로 설계해야 한다.
요약, 외부 메모리(벡터 데이터베이스), 슬라이딩 윈도우, 계층적 에이전트(상위 에이전트가 하위 에이전트의 결과를 요약) 같은 패턴이 이 제약에 대한 응답이다.

이 워크숍은 그 문제를 의도적으로 다루지 않는다.
그것이 이 워크숍의 한계이기도 하지만, 학습 목적에서는 옳은 선택이다.
컨텍스트 관리를 배우려면 먼저 루프 구조를 이해해야 하기 때문이다.
이 워크숍을 마친 다음 단계로 컨텍스트 관리를 다루는 자료를 찾아야 하는 이유가 여기 있다.
