# kumo: Go로 작성된 경량 AWS 서비스 에뮬레이터

원문: <https://github.com/sivchari/kumo>

[GeekNews 댓글](https://news.hada.io/topic?id=28793)

## 소개

kumo는 Go로 작성된 경량 AWS 서비스 에뮬레이터다.
로컬 개발과 CI/CD 테스트를 위해 실제 AWS 인프라 없이
76개의 AWS 서비스를 `localhost:4566`에서 단일 바이너리로 실행한다.
MIT 라이선스로 공개되어 있다.

인증이 필요 없어 자동화 파이프라인에 바로 투입할 수 있고,
`KUMO_DATA_DIR`을 설정하면 JSON 파일로 상태를 영속화하여
재시작 후에도 데이터를 유지할 수 있다.
로깅은 INFO와 DEBUG 레벨의 구조화된 출력을 지원한다.

## 사용법

```bash
# Docker
docker run -p 4566:4566 ghcr.io/sivchari/kumo:latest

# 바이너리 빌드
make build
./bin/kumo

# 데이터 영속화
KUMO_DATA_DIR=./data ./bin/kumo
```

## 환경 변수

| 변수              | 기본값    | 설명                       |
| ----------------- | --------- | -------------------------- |
| `KUMO_HOST`       | `0.0.0.0` | 서버 바인드 주소           |

## 데이터 영속화 메커니즘

`KUMO_DATA_DIR`을 설정하면 서비스별로 `{dir}/{service}.json` 파일에 상태를 저장한다.
쓰기는 임시 파일 생성 후 rename하는 원자적 연산으로 데이터 손상을 방지한다.
인플라이트 SQS 메시지나 멀티파트 업로드처럼 일시적인 상태는 영속화 대상에서 제외된다.
CI/CD 환경에서는 `KUMO_DATA_DIR`을 설정하지 않으면 매 실행마다 깨끗한 인메모리 상태로 시작한다.

## 분석

### LocalStack과의 위상 차이

kumo는 LocalStack과 같은 공간을 점하지만 철학이 다르다.
LocalStack은 Python 기반의 풍부한 기능 세트와 유료 Pro 플랜,
커뮤니티 에코시스템을 갖춘 사실상의 표준이다.
kumo는 단일 Go 바이너리라는 단순성을 핵심 가치로 내세운다.
Go 프로젝트에서 AWS SDK v2를 사용하는 팀이라면
같은 언어 생태계의 도구를 의존성으로 포함하는 것이
빌드·배포 파이프라인을 단순하게 유지하는 데 유리하다.

### CI/CD 친화적 설계

인증 불필요 설계는 단순한 편의 기능이 아니다.
CI/CD 환경에서 AWS 자격 증명을 주입하고 관리하는 것은
시크릿 관리, 환경 변수 전파, 최소 권한 정책 설정 등 적지 않은 운영 부담을 낳는다.
테스트 환경에서 이 부담을 완전히 제거하면
파이프라인 설정이 단순해지고, 자격 증명 노출 위험도 사라진다.
`/kumo/` 전용 엔드포인트에서 SES v2 이메일, Pinpoint SMS를
별도 외부 서비스 없이 검증할 수 있는 점도 같은 맥락이다.

### 76개 서비스의 의미

76개라는 숫자는 인상적으로 보이지만,
각 서비스의 구현 완성도가 실제 사용 가능성을 결정한다.
LocalStack도 오랜 기간 “지원은 하지만 완전하지 않은” 서비스 목록으로 혼란을 줬다.
kumo가 커뮤니티 신뢰를 얻으려면 각 서비스의 API 호환성 범위와
알려진 미구현 기능을 명확히 문서화해야 한다.

## 비평

### 강점

단일 Go 바이너리라는 배포 단순성은 실질적인 장점이다.
Python 런타임, 가상 환경, 복잡한 의존성 트리 없이
`docker run` 한 줄 또는 바이너리 하나로 작동하는 것은
컨테이너 이미지 크기와 빌드 시간에 민감한 팀에게 설득력이 있다.
Go AWS SDK v2와의 네이티브 호환성도 Go 중심 팀에게는 생태계 일관성 측면에서 유리하다.

### 약점

신생 프로젝트인 만큼 각 서비스의 API 구현 완성도가 검증되지 않았다.
S3, SQS, DynamoDB 같은 핵심 서비스의 엣지 케이스 지원,
에러 응답 포맷의 AWS 호환성, 트랜잭션 시맨틱스 등이
프로덕션 테스트에 투입할 수 있는 수준인지는 사용자가 직접 검증해야 한다.
문서와 커뮤니티 지원도 LocalStack 수준과는 차이가 있을 것이다.

## 인사이트

### 테스트 인프라의 “언어 네이티브” 경향

kumo의 등장은 더 넓은 트렌드를 반영한다.
Go 생태계에서는 Go로 작성된 테스트 도구를,
Rust 생태계에서는 Rust로 작성된 도구를 선호하는 경향이 강해지고 있다.
이는 단순한 언어 민족주의가 아니다.
동일 언어로 작성된 도구는 빌드 시스템에 자연스럽게 통합되고,
바이너리 배포가 쉬우며, 기여 장벽이 낮고,
디버깅 시 언어 경계를 넘나들 필요가 없다.
LocalStack이 Python으로 작성되어 Go 팀에게 이질적인 것처럼,
테스트 인프라도 점차 “언어 네이티브” 솔루션으로 분화하는 흐름이 있다.

이 경향이 강해지면 “하나의 표준 AWS 에뮬레이터”라는 LocalStack의 지위가
언어별로 분산될 수 있다.
Go용 kumo, Rust용 에뮬레이터, JVM용 에뮬레이터가 각각 발전하면
API 호환성 테스트 결과의 신뢰성이 분산되는 부작용도 생긴다.
“내 언어로 테스트했더니 통과했지만 실제 AWS에서는 다르게 동작했다”는
문제가 에뮬레이터 분화로 더 복잡해질 수 있다.

### 포트 4566의 사실상 표준화

kumo가 LocalStack과 동일한 포트 4566을 기본으로 사용하는 것은 전략적 선택이다.
LocalStack 기반 코드를 kumo로 마이그레이션할 때
엔드포인트 URL만 바꾸면 되도록(혹은 아무것도 바꾸지 않아도 되도록) 설계한 것이다.
이는 새로운 에뮬레이터가 기존 생태계에 진입할 때 쓰는 전형적인 호환성 전략이다.
포트 번호가 API 인터페이스의 일부처럼 작동하는 관행은
인프라 도구 세계에서 표준 포트가 얼마나 강력한 고착 효과를 만드는지 보여준다.
