# Go is the Best Language for AI Agents

<https://getbruin.com/blog/go-is-the-best-language-for-agents/>

## 배경

오픈소스 ETL/데이터 오케스트레이션 도구
Bruin을 만드는 팀의 글이다. 저자는 PHP,
Go, JavaScript, Python을 10년 넘게
전문적으로 써왔다. 데이터 생태계는
전통적으로 Python 도구를 선호하는데,
Bruin은 의도적으로 Go를 선택했다.

이 글이 나온 타이밍이 의미심장하다.
2026년, AI 에이전트가 코드를 작성하는 것이
일상이 된 시점에서 "에이전트가 어떤 언어로
코드를 쓸 때 가장 생산적인가"라는 질문은
더 이상 호기심이 아니라 실질적
엔지니어링 의사결정이 되었다.

## 핵심 주장: 컴파일 언어의 압도적 우위

에이전트는 대량의 코드를 쏟아낸다.
겉보기엔 맞는데 실제로 돌아가는지가 문제다.
**컴파일 언어는 이 검증을 구조적으로 해결한다.**
정적 타입 시스템이 타입 오류와 인자 오류를
컴파일 시점에 잡아준다.

에이전트의 작업 루프가 달라진다:
코드 생성 → 컴파일 → 에러 → 수정 → 재컴파일.
이 피드백 루프가 빠를수록 에이전트의
생산성이 올라간다. Go의 컴파일 속도는
이 루프를 극단적으로 짧게 만든다.

저자가 강조하는 핵심 메커니즘:
에이전트가 코드를 생성하면 "보통 맞아
보이는(usually looks correct)" 코드가
나온다. 문제는 "맞아 보이는"과 "동작하는"
사이의 간극이다. 컴파일러가 이 간극을
**자동으로, 즉시, 완전하게** 식별한다.
인터프리터 언어에서는 이 간극이 런타임까지
숨어 있다가 예측 불가능한 시점에 폭발한다.

## Go vs Rust: 왜 Go인가

Rust도 컴파일 언어지만 에이전트에겐
Go가 낫다는 논거:

1. **문법이 단순하다.** Go의 키워드는
   25개뿐이다. 에이전트가 생성하는 코드가
   관용적(idiomatic) 스타일에 가깝다.
   Rust의 트레이트(trait), 라이프타임
   (lifetime), 매크로(macro) 같은 고급
   기능은 에이전트가 자주 오용한다.
2. **컴파일이 빠르다.** Rust의 느린 컴파일은
   에이전트의 시행착오 루프를 치명적으로
   늦춘다. Go는 수만 줄 프로젝트도 수 초
   안에 빌드한다. 이 차이가 에이전트의
   반복 횟수를 결정한다.
3. **학습 데이터가 많다.** GitHub에 Go 코드가
   Rust보다 훨씬 많고, 모델이 더 잘 학습했다.
   Kubernetes, Docker, Terraform 등
   인프라 도구의 참조 구현이 대부분 Go다.
4. **소유권 시스템이 없다.** Rust의 borrow
   checker는 인간도 어렵다. 에이전트가
   소유권 오류를 고치려다 코드 구조를
   뒤집는 경우가 잦다. Go의 가비지 컬렉터는
   이 복잡성을 통째로 제거한다.

## 크로스 플랫폼: 숨은 킬러 피처

에이전트가 로컬에서 생성한 코드를
원격 환경에서 검증하는 시대가 왔다.
Bruin 팀은 배경 에이전트(background
agent) 실험을 하고 있다. Cursor에서
Slack 메시지로 코드 변경을 트리거하고,
로컬 세션을 원격 세션으로 넘기면서
빌드 환경에 대한 직접적 통제를
점진적으로 놓고 있다.

이때 Go의 크로스 플랫폼 지원이 빛난다.
`GOOS=linux GOARCH=amd64 go build`
한 줄로 타깃 플랫폼 바이너리가 나온다.
**같은 코드가 Linux, Windows, macOS에서
동일하게 동작하는 바이너리를 만든다.**
CGO 의존성만 없다면 크로스 컴파일에
추가 도구가 필요 없다.

모든 환경에서의 테스트 실행이 사소해진다.
에이전트가 변경할 때마다 전 환경 테스트를
돌려서 기존 기능을 깨지 않았음을 즉시
확인할 수 있다.

Python이라면? 가상 환경 설정, 의존성 충돌,
OS별 패키지 차이, C 확장 빌드 실패.
에이전트가 코드 작성 외의 환경 삽질에
토큰을 소비한다.

## 실전 결과

저자의 핵심 문장: "에이전트가 Go로
아름다운 코드를 작성하고, 실행하고,
컴파일하고, 테스트하고, 포맷하고,
다양한 머신에서 사용할 수 있는
성능 좋은 소프트웨어를 배포한다."

Claude Code에 Go CLI를 만들라고 시키면
결과를 즐길 수 있다고 한다. Bruin 팀은
이 방식으로 생산성이 높아졌고, Go에
더 집중하기로 결정했다.

주목할 점은 저자가 "에이전트와 함께여도
여전히 Go 작업을 즐긴다"고 말한 부분이다.
에이전트가 언어의 장점을 훼손하지 않고
오히려 증폭시킨다는 주장이다.

## 인사이트

### "운 좋게 맞아떨어졌다"는 겸손한 진실

저자는 "순전히 운으로(by pure luck) Go가
사용성, 성능, 보편성의 스위트 스팟에
착지했다"고 말한다. 하지만 이건 운이 아니다.
**Go는 처음부터 "단순함"을 설계 원칙으로
선택했고, 그 단순함이 에이전트 시대에
예상치 못한 배당금을 지급하고 있는 것이다.**

Rob Pike가 "Less is exponentially more"라고
했을 때, 그는 인간 개발자의 인지 부하를
말한 거였다. 그런데 그 단순함이 LLM의
코드 생성 품질도 끌어올렸다.
설계 철학이 시대를 초월한 사례다.

여기서 더 깊이 파고들면 흥미롭다.
Go의 `gofmt`는 코드 포맷을 강제 통일한다.
이것이 LLM 학습 데이터의 노이즈를 줄인다.
같은 로직이 100가지 스타일로 존재하는
Python과 달리, Go 코드는 한 가지 형태로
수렴한다. **학습 데이터의 일관성이
생성 품질의 일관성으로 이어진다.**

### 컴파일러는 가장 싼 코드 리뷰어다

에이전트가 생성한 코드를 인간이
한 줄씩 리뷰하는 건 비현실적이다.
에이전트는 한 세션에 수백, 수천 줄의
코드를 만들어낸다. 인간 리뷰의 병목이
에이전트의 속도를 상쇄한다.

**컴파일러가 1차 필터 역할을 한다.**
타입 오류, 미사용 변수, 패키지 임포트
오류를 즉시 잡는다. Go는 여기서 더 나간다.
`go vet`은 의심스러운 구조를 경고하고,
`golangci-lint`는 수십 개 린터를 통합하고,
`go fix`는 오래된 관용구를 자동 현대화한다.
이 정적 분석 도구 체인이 에이전트의
자동 수정 루프와 완벽하게 맞물린다.

에이전트의 워크플로우가 된다:
코드 생성 → `go build` → 타입 오류 수정 →
`go vet` → 구조 경고 수정 → `golangci-lint`
→ 린트 오류 수정 → 테스트 → 완료.
**각 단계가 기계적으로 검증 가능하다.**

Python은? 런타임에 터진다.
"TypeError: unsupported operand type"를
프로덕션에서 만나게 된다. 물론 `mypy`나
`pyright`가 있지만, 이것들은 선택적이고
프로젝트마다 설정이 다르다. Go의 컴파일러는
선택이 아니라 필수다. 이 차이가 크다.

### 동시성: 원문이 가볍게 넘긴 강점

원문에서 깊게 다루지 않았지만,
Bruin이 Go를 선택한 이유 중 하나가
**"많은 것이 동시에 실행되어야 한다"**였다.
데이터 오케스트레이션 도구에서 동시성은
핵심 요구사항이다.

고루틴(goroutine)과 채널(channel)은
에이전트가 병렬 작업을 수행할 때
Python의 GIL 문제 없이 자연스럽게 동작한다.
고루틴은 OS 스레드가 아니라 Go 런타임이
관리하는 경량 스레드다. 수만 개를
동시에 띄워도 메모리 부담이 적다.

에이전트가 5개 API를 동시에 호출하고,
응답을 처리하고, 액션을 조율해야 할 때
Go의 동시성 모델이 "그냥 맞다(just right)".
`select` 문으로 여러 채널을 동시에
기다리는 패턴은 에이전트의 멀티 작업
오케스트레이션과 구조적으로 닮아 있다.

### 진짜 질문: 에이전트 시대의 언어 선택 기준

이 글이 던지는 근본적 질문은
"어떤 언어가 좋은가"가 아니다.
**"에이전트가 잘 다루는 언어가 좋은 언어"로
평가 기준 자체가 바뀌고 있다는 것이다.**

전통적 기준: 개발자 생산성, 생태계, 성능.
새로운 기준: 에이전트 생성 품질, 자동 검증
용이성, 피드백 루프 속도, 배포 단순성.

Go는 두 기준 모두에서 강하다.
Python은 전통적 기준에선 강하지만
새로운 기준에선 약점이 드러난다.
Rust는 성능에선 최강이지만 에이전트
생성 품질에서 뒤처진다.

이 프레임 전환이 맞다면 파급 효과가 크다.
신규 프로젝트의 언어 선택, 기존 프로젝트의
마이그레이션 우선순위, 개발자 교육 투자
방향이 모두 달라진다. "우리 팀이 잘 쓰는
언어"가 아니라 "에이전트가 우리 팀의
코드를 잘 쓰는 언어"가 기준이 된다.

### `gofmt`의 숨은 무기

다른 언어 커뮤니티에서 포매팅 전쟁이
벌어질 때, Go는 `gofmt`로 일찍이
종결시켰다. 이것이 에이전트 시대에
예상 못한 강점이 된다.

에이전트가 생성한 Go 코드는 `gofmt`를
통과하면 모든 Go 프로젝트의 스타일과
일치한다. 코드 리뷰에서 포맷 이슈가
사라지고, diff가 순수하게 로직 변경만
보여준다. **포맷 통일이 에이전트 생성
코드의 리뷰 비용을 구조적으로 줄인다.**

Python의 Black, JavaScript의 Prettier도
비슷한 역할을 하지만, 이것들은 커뮤니티
도구다. `gofmt`는 언어에 내장되어 있고,
사실상 100%의 Go 프로젝트가 사용한다.
채택률의 차이가 학습 데이터의 일관성
차이로 이어진다.

## 비판

### 생존자 편향: 자기 선택을 합리화하는 글

이 글의 본질을 직시해야 한다.
**Go로 제품을 만든 팀이 "Go가 최고"라고
말하는 건 자기 선택의 사후 합리화다.**
Bruin이 Python으로 만들었다면
"Python이 에이전트에 최적"이라는 글을
썼을 가능성이 높다.

체계적 벤치마크가 없다.
"에이전트가 아름다운 Go를 쓴다"는
일화적 증거(anecdotal evidence)일 뿐이다.
동일 프로젝트를 Go, Python, TypeScript,
Rust로 에이전트에 시키고 코드 품질,
버그율, 완성 시간을 비교한 데이터가
어디에도 없다. 설득력 있는 주장을 하려면
최소한 에이전트 생성 코드의 컴파일
성공률, 테스트 통과율, 수정 반복 횟수를
언어별로 정량 비교해야 한다.

### "컴파일 언어"와 "정적 타입"을 혼동

원문의 핵심 논거는 "컴파일 언어라서
에이전트가 검증하기 쉽다"는 것이다.
하지만 에이전트에게 중요한 건
**컴파일 여부가 아니라 정적 타입 분석이다.**

TypeScript는 컴파일 언어가 아니지만
`tsc --noEmit`이 타입 오류를 즉시 잡는다.
Python도 `mypy --strict`이나 `pyright`로
정적 분석을 돌릴 수 있다.
**"컴파일 = 안전"이라는 등식은
논리적 비약이다.** 정적 타입의 이점이라면
Go만의 독점적 강점이 아니다.

실제로 Claude Code, Cursor 등 주요 코딩
에이전트들이 TypeScript 프로젝트에서도
높은 생산성을 보여준다. "컴파일 언어"라는
프레임이 아니라 "빠른 정적 검증이 가능한
언어"라는 프레임이 더 정확하다. 그리고
그 프레임에서는 TypeScript도 강력한
경쟁자다.

### 학습 데이터 논거가 자기 모순

"GitHub에 Go 코드가 Rust보다 많아서
에이전트가 Go를 더 잘 쓴다"고 주장한다.
이 논리대로라면 **Python이 압도적
승자여야 한다.** GitHub에서 Python은
Go보다 몇 배나 많은 코드베이스를 가진다.
JavaScript도 마찬가지다.

학습 데이터 양이 기준이라면
Go는 Python, JavaScript에 한참 밀린다.
이 논거는 Go vs Rust에서만 성립하고,
전체 언어 비교에선 Go에 불리하게 작용한다.

더 나아가, 학습 데이터의 양보다
**질이 중요하다.** Go 코드가 `gofmt`로
일관적이라는 건 장점이지만, Python의
데이터 사이언스/ML 코드베이스는
도메인 특화 패턴이 풍부하다.
"많다"와 "에이전트가 잘 쓴다" 사이에는
저자가 제시하지 않은 여러 변수가 있다.

### Docker가 크로스 플랫폼 논거를 무력화

"단일 바이너리로 어디서든 동작"은
2015년엔 킬러 피처였다.
2026년엔 **Docker가 이 문제를 이미 풀었다.**
Python이든 Node.js든 컨테이너에 넣으면
어디서든 동일하게 동작한다.

에이전트가 배포까지 담당하는 시나리오에서도
`Dockerfile` 생성은 어떤 언어든 가능하다.
단일 바이너리의 이점은 컨테이너 이미지
크기와 콜드 스타트 속도 정도로 축소되었다.

다만 공정하게 말하면, 로컬 개발 환경에서의
크로스 플랫폼은 여전히 Go가 유리하다.
Docker 없이 `GOOS`/`GOARCH` 플래그만으로
크로스 컴파일하는 편의성은 에이전트가
로컬에서 빠르게 검증할 때 마찰을 줄인다.

### AI/ML 생태계의 부재: 치명적 약점

에이전트가 "만드는" 도구 언어로서의
Go는 강할 수 있다. 하지만 에이전트를
"구축하는" 프레임워크 생태계를 보면
**Python이 압도적이다.**

LangChain, CrewAI, AutoGen, OpenAI SDK,
Anthropic SDK — 모두 Python 퍼스트다.
Go용 에이전트 프레임워크는 이제 겨우
Google ADK for Go가 나온 수준이다.
**에이전트가 쓰는 언어와 에이전트를
만드는 언어를 구분하지 않은 것은
원문의 큰 맹점이다.**

이 구분은 중요하다.
"에이전트가 Go 코드를 잘 생성한다"와
"에이전트 시스템을 Go로 구축하기 좋다"는
완전히 다른 주장이다. 원문은 전자를
주장하면서 제목에서는 후자를 암시한다.
제목의 "for AI Agents"가 의도적으로
모호하다.

### CLI 편향: 일반화의 오류

Bruin은 CLI 도구다. CLI는 Go의 홈그라운드다.
그런데 이 경험을 "AI 에이전트 전반"으로
일반화한다. 웹 서비스, 데이터 파이프라인,
ML 시스템, 프론트엔드 — 이 영역에서
에이전트가 Go를 쓰는 게 최선인가?

웹 프론트엔드는 TypeScript/JavaScript가
유일한 선택지다. 데이터 파이프라인은
pandas, spark, dbt 등 Python 생태계가
사실상 표준이다. ML 모델 서빙도
PyTorch, TensorFlow 바인딩이 Python에
집중되어 있다.

**"Go가 CLI 에이전트에 강하다"는 참이다.
"Go가 모든 에이전트에 최고"는 비약이다.**
더 정직한 제목은 "Go is a Great Language
for CLI Tools in the Agent Era"일 것이다.

### 에러 핸들링의 이면

원문이 언급하지 않은 Go의 약점이 있다.
Go의 `if err != nil` 패턴은 예측 가능한
에러 처리를 강제한다. 이것이 인간에겐
명시적이라는 장점이지만, **에이전트에겐
반복적 보일러플레이트를 양산하는 원인**이
될 수 있다.

에이전트가 Go 코드를 생성할 때
에러 핸들링 코드가 전체 코드량의
상당 부분을 차지한다. 이 반복적 패턴이
실제 비즈니스 로직을 묻히게 만들 수 있다.
Rust의 `?` 연산자나 함수형 언어의
`Result` 모나드가 더 간결할 수 있다.

물론 이 반복성이 에이전트에겐 오히려
쉬울 수 있다. 패턴이 단순하고 반복적이니
에이전트가 실수할 여지가 적다.
양면이 있는 특성이다.

## 결론

이 글은 Go의 에이전트 시대 적합성에 대한
흥미로운 관점을 제시하지만, "best"라는
단어가 감당할 수 없는 무게를 지고 있다.

**타당한 것:** 컴파일 속도, 문법 단순성,
도구 체인 통합이 에이전트의 코드 생성
→ 검증 → 수정 루프에 유리하다는 관찰.

**과장된 것:** 하나의 CLI 도구 경험을
전체 에이전트 패러다임으로 일반화한 것.
에이전트가 "쓰는" 언어와 에이전트를
"만드는" 언어를 구분하지 않은 것.

**놓친 것:** `gofmt`의 학습 데이터 일관성
효과. 에러 핸들링 보일러플레이트의 양면성.
TypeScript라는 강력한 대안의 존재.

실용적 시사점은 분명하다.
새로운 CLI 도구나 인프라 소프트웨어를
에이전트와 함께 만든다면 Go는 탁월한
선택이다. 하지만 그것을 "Go is the Best
Language for AI Agents"라고 부르는 순간,
주장은 근거를 초과한다.
