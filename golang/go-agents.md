# Go is the Best Language for AI Agents

<https://getbruin.com/blog/go-is-the-best-language-for-agents/>

오픈소스 ETL 도구 Bruin을 만드는 팀이
"Go가 AI 에이전트 시대에 최적의 언어"라고
주장한다. PHP, Go, JavaScript, Python을
10년 넘게 써온 저자의 실전 경험에 기반한 글.

## 핵심 주장: 컴파일 언어의 압도적 우위

에이전트는 대량의 코드를 쏟아낸다.
겉보기엔 맞는데 실제로 돌아가는지가 문제다.
**컴파일 언어는 이 검증을 구조적으로 해결한다.**
정적 타입 시스템이 타입 오류와 인자 오류를
컴파일 시점에 잡아준다.

에이전트의 작업 루프가 달라진다:
코드 생성 → 컴파일 → 에러 → 수정 → 재컴파일.
이 피드백 루프가 빠를수록 에이전트의
생산성이 올라간다. Go의 컴파일 속도는
이 루프를 극단적으로 짧게 만든다.

## Go vs Rust: 왜 Go인가

Rust도 컴파일 언어지만 에이전트에겐
Go가 낫다는 논거:

1. **문법이 단순하다.** 에이전트가 생성하는
   코드가 관용적(idiomatic) 스타일에
   더 가깝다.
2. **컴파일이 빠르다.** Rust의 느린 컴파일은
   에이전트의 시행착오 루프를 치명적으로
   늦춘다.
3. **학습 데이터가 많다.** GitHub에 Go 코드가
   Rust보다 훨씬 많고, 모델이 더 잘 학습했다.
4. **소유권 시스템이 없다.** Rust의 borrow
   checker는 인간도 어렵다. 에이전트에겐
   불필요한 복잡성이다.

## 크로스 플랫폼: 숨은 킬러 피처

에이전트가 로컬에서 생성한 코드를
원격 환경에서 검증하는 시대가 왔다.
Cursor에서 Slack 메시지로 작업을 트리거하고,
로컬 세션을 원격으로 넘기는 실험을
Bruin 팀이 하고 있다.

이때 Go의 크로스 플랫폼 지원이 빛난다.
**같은 코드가 Linux, Windows, macOS에서
동일하게 동작하는 바이너리를 만든다.**
빌드 환경에 대한 통제를 에이전트에게
넘길 수 있는 이유다.

Python이라면? 가상 환경 설정, 의존성 충돌,
OS별 패키지 차이. 에이전트가 코드 작성
외의 삽질에 시간을 쓴다.

## 실전 결과

저자의 결론: 에이전트가 Go로 아름다운 코드를
작성하고, 컴파일하고, 테스트하고, 포맷하고,
다양한 머신에서 동작하는 성능 좋은 소프트웨어를
배포한다. Claude Code에 Go CLI를 만들라고
시키면 결과를 즐길 수 있다고 한다.

## 인사이트

### "운 좋게 맞아떨어졌다"는 겸손한 진실

저자는 "순전히 운으로(by pure luck) Go가
사용성, 성능, 보편성의 스위트 스팟에
착지했다"고 말한다. 하지만 이건 운이 아니다.
**Go는 처음부터 "단순함"을 설계 원칙으로
선택했고, 그 단순함이 에이전트 시대에
예상치 못한 배당금을 지급하고 있는 것이다.**

Rob Pike가 "Less is exponentially more"라고
했을 때, 그는 인간 개발자의 인지 부하를
말한 거였다. 그런데 그 단순함이 LLM의
코드 생성 품질도 끌어올렸다.
설계 철학이 시대를 초월한 사례다.

### 컴파일러는 가장 싼 코드 리뷰어다

에이전트가 생성한 코드를 인간이
한 줄씩 리뷰하는 건 비현실적이다.
**컴파일러가 1차 필터 역할을 한다.**
타입 오류, 미사용 변수, 패키지 임포트
오류를 즉시 잡는다. Go는 여기서 더 나간다.
`go vet`, `golangci-lint`, `go fix`까지
정적 분석 도구 체인이 에이전트의
자동 수정 루프와 완벽하게 맞물린다.

Python은? 런타임에 터진다.
"TypeError: unsupported operand type"를
프로덕션에서 만나게 된다.

### 동시성: 언급되지 않은 강점

원문에서 깊게 다루지 않았지만,
Bruin이 Go를 선택한 이유 중 하나가
**"많은 것이 동시에 실행되어야 한다"**였다.
고루틴(goroutine)과 채널(channel)은
에이전트가 병렬 작업을 수행할 때
Python의 GIL 문제 없이 자연스럽게 동작한다.

에이전트가 5개 API를 동시에 호출하고,
응답을 처리하고, 액션을 조율해야 할 때
Go의 동시성 모델이 "그냥 맞다(just right)".

### 진짜 질문: 에이전트 시대의 언어 선택 기준

이 글이 던지는 근본적 질문은
"어떤 언어가 좋은가"가 아니다.
**"에이전트가 잘 다루는 언어가 좋은 언어"로
평가 기준 자체가 바뀌고 있다는 것이다.**

전통적 기준: 개발자 생산성, 생태계, 성능.
새로운 기준: 에이전트 생성 품질, 자동 검증
용이성, 피드백 루프 속도, 배포 단순성.

Go는 두 기준 모두에서 강하다.
Python은 전통적 기준에선 강하지만
새로운 기준에선 약점이 드러난다.
Rust는 성능에선 최강이지만 에이전트
생성 품질에서 뒤처진다.

## 비판

### 생존자 편향: 자기 선택을 합리화하는 글

이 글의 본질을 직시해야 한다.
**Go로 제품을 만든 팀이 "Go가 최고"라고
말하는 건 자기 선택의 사후 합리화다.**
Bruin이 Python으로 만들었다면
"Python이 에이전트에 최적"이라는 글을
썼을 가능성이 높다.

체계적 벤치마크가 없다.
"에이전트가 아름다운 Go를 쓴다"는
일화적 증거(anecdotal evidence)일 뿐이다.
동일 프로젝트를 Go, Python, TypeScript,
Rust로 에이전트에 시키고 코드 품질,
버그율, 완성 시간을 비교한 데이터가
어디에도 없다.

### "컴파일 언어"와 "정적 타입"을 혼동

원문의 핵심 논거는 "컴파일 언어라서
에이전트가 검증하기 쉽다"는 것이다.
하지만 에이전트에게 중요한 건
**컴파일 여부가 아니라 정적 타입 분석이다.**

TypeScript는 컴파일 언어가 아니지만
`tsc --noEmit`이 타입 오류를 즉시 잡는다.
Python도 `mypy --strict`이나 `pyright`로
정적 분석을 돌릴 수 있다.
**"컴파일 = 안전"이라는 등식은
논리적 비약이다.** 정적 타입의 이점이라면
Go만의 독점적 강점이 아니다.

### 학습 데이터 논거가 자기 모순

"GitHub에 Go 코드가 Rust보다 많아서
에이전트가 Go를 더 잘 쓴다"고 주장한다.
이 논리대로라면 **Python이 압도적
승자여야 한다.** GitHub에서 Python은
Go보다 몇 배나 많은 코드베이스를 가진다.
JavaScript도 마찬가지다.

학습 데이터 양이 기준이라면
Go는 Python, JavaScript에 한참 밀린다.
이 논거는 Go vs Rust에서만 성립하고,
전체 언어 비교에선 Go에 불리하게 작용한다.

### Docker가 크로스 플랫폼 논거를 무력화

"단일 바이너리로 어디서든 동작"은
2015년엔 킬러 피처였다.
2026년엔 **Docker가 이 문제를 이미 풀었다.**
Python이든 Node.js든 컨테이너에 넣으면
어디서든 동일하게 동작한다.

에이전트가 배포까지 담당하는 시나리오에서도
`Dockerfile` 생성은 어떤 언어든 가능하다.
단일 바이너리의 이점은 컨테이너 이미지
크기 정도로 축소되었다.

### AI/ML 생태계의 부재: 치명적 약점

에이전트가 "만드는" 도구 언어로서의
Go는 강할 수 있다. 하지만 에이전트를
"구축하는" 프레임워크 생태계를 보면
**Python이 압도적이다.**

LangChain, CrewAI, AutoGen, OpenAI SDK,
Anthropic SDK — 모두 Python 퍼스트다.
Go용 에이전트 프레임워크는 이제 겨우
Google ADK for Go가 나온 수준이다.
**에이전트가 쓰는 언어와 에이전트를
만드는 언어를 구분하지 않은 것은
원문의 큰 맹점이다.**

### CLI 편향: 일반화의 오류

Bruin은 CLI 도구다. CLI는 Go의 홈그라운드다.
그런데 이 경험을 "AI 에이전트 전반"으로
일반화한다. 웹 서비스, 데이터 파이프라인,
ML 시스템, 프론트엔드 — 이 영역에서
에이전트가 Go를 쓰는 게 최선인가?

**"Go가 CLI 에이전트에 강하다"는 참이다.
"Go가 모든 에이전트에 최고"는 비약이다.**
