# Apache Ossie: 시맨틱 메타데이터 교환 표준

<https://github.com/apache/ossie>

## 소개

Apache Ossie는 분석, AI, BI 플랫폼 사이에서 시맨틱 메타데이터를 교환하는 방식을
표준화하려는 벤더 중립 명세(specification) 프로젝트다.
핵심 문제 의식은 하나다.
동일한 KPI나 비즈니스 지표가 dbt, Looker, Salesforce, PowerBI에서 각각 다르게 정의되며,
AI 에이전트가 이 위에서 질의를 생성할 때 불일치가 증폭된다는 것이다.

Ossie는 JSON/YAML 기반 명세 파일로 시맨틱 모델을 표현하고,
이를 각 플랫폼의 형식으로 변환하는 레퍼런스 컨버터와 검증 도구를 함께 제공한다.
dbt, GoodData, Polaris, Salesforce 형식과의 상호 변환이 현재 지원된다.
Python(74.6%)과 Java(24.7%)로 구현되어 있으며,
1,213 stars · 149 forks 규모로 성장 중이다(2026년 7월 기준).

## 분석

### 시맨틱 레이어의 불일치가 AI 에이전트 시대에 증폭되다

BI 도구 사이의 지표 정의 불일치는 오랜 문제지만,
텍스트에서 SQL을 생성하는 AI 에이전트가 보편화되면서 그 비용이 커졌다.
에이전트가 “월별 매출 증가율”을 질의할 때,
플랫폼마다 다른 정의를 가진 동일한 이름의 지표 중 어느 것을 참조하는지
결정하기 어렵다.
시맨틱 계층(semantic layer)이 없거나 일관성이 없으면,
에이전트 출력의 정확도는 SQL 생성 능력보다 데이터 정의 품질에 의해 제한된다.

### 표준 명세 접근의 구조적 이점

Ossie가 선택한 전략은 각 플랫폼을 수정하는 대신
중간 표준 형식을 만드는 것이다.
JSON/YAML 명세로 “의미의 진실(source of truth)”을 선언하고,
플랫폼별 컨버터가 이를 번역한다.
이것은 데이터베이스 세계에서 SQL이 한 역할과 유사하다.
모든 플랫폼이 자체 쿼리 언어를 가졌지만,
공통 표준이 생태계를 연결했다.
시맨틱 레이어에서도 같은 패턴이 필요하다는 것이 Ossie의 핵심 주장이다.

### Apache 재단 거버넌스의 역할

Ossie가 독립 오픈소스 프로젝트가 아니라 Apache 재단을 선택한 것은
벤더 중립성과 거버넌스 신뢰를 확보하기 위한 전략적 판단이다.
BI 생태계에서 표준 경쟁은 기술보다 정치에 가깝다.
Salesforce, Microsoft, dbt Labs 같은 플랫폼이 각자의 시맨틱 형식을 가지고 있는데,
하나의 벤더가 만든 표준에 다른 벤더가 채택할 이유가 없다.
Apache라는 중립 공간이 이 장벽을 낮춘다.

## 비평

### 거버넌스 없는 표준은 지표 불일치를 이전하는 것에 불과하다

Ossie가 제공하는 것은 형식(format)이지 거버넌스(governance)가 아니다.
모든 팀이 같은 형식의 파일을 만들더라도,
“매출”의 정의를 누가 소유하고, 변경 시 누가 승인하고, 불일치 시 누가 중재하는지는
Ossie 명세 밖의 문제다.
조직 내 지표 소유권 구조와 검증 자동화 없이
표준 형식을 도입하면, 불일치가 YAML 파일 안으로 이전될 뿐이다.

### 컨버터 품질이 표준 채택의 실질적 관문이다

Ossie의 레퍼런스 컨버터가 얼마나 완전히 각 플랫폼의 의미론을 보존하는지는
실제 도입 전 검증이 필요한 영역이다.
시맨틱 모델은 단순 필드 매핑이 아니라 계산 로직, 필터 규칙, 집계 방식,
접근 제어까지 포함한다.
이 복잡도가 컨버터에서 손실 없이 처리되는지는
각 플랫폼의 고급 기능을 쓰는 조직에서 가장 먼저 벽에 부딪힐 부분이다.

### 표준 경쟁이 아직 끝나지 않았다

DCAT, OpenTelemetry Semantic Conventions, dbt의 시맨틱 레이어,
Cube의 Semantic Layer 등 이 공간에는 이미 여러 접근이 경쟁 중이다.
Ossie가 이 경쟁에서 어떤 차별점으로 승리할지,
혹은 이 분야가 하나의 표준으로 수렴할 수 있는지는 아직 불확실하다.
Apache 거버넌스만으로는 플랫폼 벤더들의 적극적인 참여를 보장할 수 없다.

## 인사이트

### AI 에이전트가 데이터에 직접 질의하는 미래에서 시맨틱 계층이 인프라가 된다

텍스트-투-SQL, RAG 기반 데이터 분석, 자율 BI 에이전트가 일반화될수록,
SQL 생성 이전 단계인 “무엇을 질의해야 하는가”의 정의가 결과 품질을 결정한다.
시맨틱 계층 없이 에이전트를 데이터에 붙이면,
에이전트의 질문과 데이터의 의미 사이의 간극을 사람이 계속 메워야 한다.
Ossie 같은 표준이 자리 잡으면, AI 에이전트가 공통 시맨틱 모델을 참조해
플랫폼 무관하게 일관된 질의를 생성할 수 있다.
이것은 AI 데이터 스택의 기반 레이어 문제다.

### FinOps, DataOps에서 시맨틱 표준화의 실질적 ROI

시맨틱 불일치의 비용은 추상적으로 느껴지지만,
같은 “전환율”을 두고 마케팅팀과 데이터팀의 수치가 다를 때 발생하는
회의 시간, 에스컬레이션, 정책 결정 지연이 그 실체다.
Ossie 같은 표준이 단일 정의를 강제하면, 이 비용이 줄어든다.
FinOps처럼 “비용 데이터의 표준화”가 거버넌스 압력으로 도입된 사례처럼,
AI 에이전트 도입이 “지표 정의 표준화” 압력으로 작용할 가능성이 높다.

### 표준의 성공은 기여자 다양성으로 측정된다

Ossie의 중요한 신호는 GitHub Trending이 아니라 기여자 조직의 다양성이다.
Apache 재단 산하에 있더라도, 실제 기여가 단일 벤더 중심이라면
그것은 오픈 표준이 아니라 오픈소스로 포장된 벤더 표준에 가깝다.
Ossie가 진정한 생태계 표준이 되려면,
경쟁 관계의 BI 벤더들이 기여하고 자사 형식을 Ossie와 호환시키는
실질적 참여가 있어야 한다.
