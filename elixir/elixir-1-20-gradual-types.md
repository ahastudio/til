# Elixir v1.20: 점진적 타입 언어로의 전환

원문: <https://elixir-lang.org/blog/2026/06/03/elixir-v1-20-0-released/>

HN 토론: <https://news.ycombinator.com/item?id=48388324> (965점, 383개 댓글)

## 요약

Elixir v1.20은 2022년부터 진행된 집합론적 타입 시스템 연구를 완성한 버전이다.
핵심 성과는 “타입 어노테이션 없이도 모든 Elixir 프로그램에 대해 타입 추론과 점진적 타입 검사를 수행”할 수 있게 된 것이다.
기존 코드를 깨뜨리지 않으면서 컴파일러가 죽은 코드와 “검증된 버그(verified bug)”를 자동으로 찾아낸다.

타입 시스템의 설계 목표는 세 가지다.
건전성(soundness) — 추론한 타입이 실제 프로그램 동작과 일치한다.
점진성(gradualness) — `dynamic()` 타입을 통해 기존 동적 코드와 공존한다.
개발자 친화성 — 합집합(union), 교집합(intersection), 부정(negation)이라는 집합 연산으로 타입을 표현한다.

`dynamic()` 타입은 다른 언어의 `any()`와 다르다.
함수 호출 시 제공된 타입과 기대 타입이 겹치는 부분이 있으면 위반을 보고하지 않는다.
완전히 겹치지 않는 서로소 관계일 때만 오류를 발생시킨다.
코드 실행 흐름에 따라 타입이 좁혀진다(narrowing).

Guard와 패턴 매칭에서도 타입 추론이 이루어진다.
`is_list(x) and is_integer(y)` 조건 안에서 `x`는 리스트, `y`는 정수로 추론된다.
`not is_map_key(x, :foo)` 조건 안에서 `x`는 `:foo` 키를 갖지 않는 맵으로 좁혀진다.
`case` 표현식에서 `nil`을 처리하는 절이 있으면 다음 절의 변수는 자동으로 `nil`이 제외된 타입으로 좁혀진다.

컴파일 성능도 개선됐다.
멀티코어 머신에서 빌드 시간이 단축됐으며, 새로운 `:module_definition` 컴파일러 옵션이 추가됐다.
`mix.exs`에서 `elixirc_options: [module_definition: :interpreted]`로 대규모 프로젝트의 컴파일 시간을 추가로 개선할 수 있다.

## 분석

### “검증된 버그”가 의미하는 것

타입 시스템이 찾아내는 버그를 “검증된 버그(verified bug)”라고 부르는 것은 신중한 용어 선택이다.
정적 분석 도구는 거짓 양성(false positive) — 실제로 문제없는 코드를 오류로 보고 — 으로 신뢰를 잃는다.

릴리스 포스트가 제시하는 예시가 이 신중함을 보여준다.

```elixir
def percentage_or_error(value) when is_integer(value) do
  value_or_error =
    if value > 1 do
      value
    else
      "not well"
    end
  
  if value > 1 do
    value_or_error / 100
  else
    String.upcase(value_or_error)
  end
end
```

이 코드에서 `value_or_error`는 정수 또는 문자열이다.
첫 번째 분기(`value > 1`)에서는 정수, 두 번째 분기에서는 문자열로만 사용되어 런타임 오류가 없다.
타입 시스템이 이것을 올바르게 인식해 거짓 양성을 피한다.

반면 `Map.fetch!(value_or_error, :some_key)` 호출은 진짜 버그다.
`Map.fetch!`는 맵을 요구하지만 `value_or_error`는 정수 또는 문자열만 가능하므로
컴파일러가 실제로 실패할 코드임을 확신 있게 보고한다.

### `dynamic()`과 `any()`의 차이

TypeScript의 `any`는 타입 검사를 완전히 우회한다.
`any` 타입의 값에 무엇을 해도 컴파일러가 오류를 내지 않는다.
이것은 점진적 채택을 위한 탈출구이지만, 동시에 타입 시스템을 무력화하는 구멍이다.

Elixir의 `dynamic()`은 다르게 작동한다.
완전히 서로소인 타입에 대한 연산만 오류로 보고하고,
코드 흐름에 따라 타입을 좁힌다.
`dynamic()`은 타입을 모른다는 표시가 아니라 런타임에 결정될 타입의 범위를 추적하는 표시다.

### Dialyzer와의 관계

[GeekNews 댓글](https://news.hada.io/topic?id=30152)에서 Dialyzer와의 비교가 논의됐다.
Dialyzer의 success typing 방식은 “이 코드가 실패하지 않음을 증명할 수 없다”는 보수적 접근을 취한다.
결과적으로 거짓 음성(false negative)은 적지만 분석이 느리고 오류 메시지가 직관적이지 않다.

v1.20의 타입 추론은 컴파일 시간에 즉각적으로 피드백을 제공한다.
Dialyzer를 대체하는 것이 아니라 서로 다른 방식으로 코드를 검증하는 보완 관계다.

## 비평

### 강점: 거짓 양성을 피하는 신중한 설계

“검증된 버그”만 보고한다는 설계 철학은 실용적이다.
거짓 양성이 많은 타입 검사기는 개발자가 경고를 무시하게 만들고,
결국 타입 시스템 자체를 우회하는 패턴으로 이어진다.
서로소 타입에 대해서만 확신 있게 오류를 보고하는 접근은
이 신뢰 문제를 설계 수준에서 해결한다.

### 약점: 재귀 타입과 매개변수화 타입의 미완성

HN에서는 점진적 타입 시스템의 런타임 성능 영향을 우려하는 질문이 제기됐다.
Racket의 점진적 타입 시스템이 특정 조건에서 프로그램을 점근적으로 느리게 만든다는 연구 사례를 들며,
Elixir의 구현이 같은 문제를 갖는지 물었다.[^sestep]
타입 어노테이션 없이 추론만으로 동작하는 이번 구현이 런타임 오버헤드를 만들지 않는다는 점을 공식 문서에서 더 명확히 다룰 필요가 있다.

릴리스 포스트가 향후 과제로 명시한 것들이 있다.
재귀 타입의 효율적 구현, 매개변수화된 타입(generics에 해당)의 구현,
그리고 맵의 키-값 쌍을 열거 가능한 형식으로 순회하는 기술이 아직 미완이다.
타입 서명(type signatures) 도입은 이 과제들이 해결된 이후에나 가능하다.
v1.20이 중요한 이정표이지만 완성된 타입 시스템은 아직 미래의 일이다.

### 보완할 시각: 생태계 라이브러리들의 타입 지원

언어 자체의 타입 시스템이 아무리 좋아도, 핵심 라이브러리들이 타입을 지원하지 않으면 실용적 가치가 제한된다.
Phoenix, Ecto, Absinthe 등 주요 Elixir 라이브러리들이 타입 어노테이션을 추가하는 데 얼마나 걸릴지,
타입 추론이 라이브러리 경계에서 어떻게 작동하는지가 실제 채택의 관건이다.

## 인사이트

### 타입 시스템의 점진적 도입이 언어 생태계를 바꾸는 방식

TypeScript의 성공은 중요한 교훈을 남겼다.
타입 시스템 자체보다 기존 생태계와의 호환성과 점진적 채택 가능성이 언어의 성공을 결정한다는 것이다.

JavaScript에서 TypeScript로의 전환은 강제가 아닌 선택으로 이루어졌다.
`// @ts-check` 하나로 파일 단위로 타입 검사를 켤 수 있었고, `any` 타입으로 점진적으로 타입을 좁혀갈 수 있었다.
수백만 줄의 기존 JavaScript 코드가 타입 없이도 TypeScript 프로젝트에 공존할 수 있었다.

Elixir v1.20은 이 경로를 걷는다.
기존 `.ex` 파일은 변경 없이 동작하고, 컴파일러는 추론 가능한 범위에서 타입을 검사한다.
생태계의 모든 라이브러리가 타입 어노테이션을 갖출 때까지 기다릴 필요가 없다.
[GeekNews 댓글](https://news.hada.io/topic?id=30152)에서 10년 경력의 Elixir 개발자가 “오래 기대해온 것이 실제로 들어온 것이 정말 반갑다”고 표현한 것은 이 호환성에 대한 신뢰에서 나온다.

### BEAM 생태계에서 타입 선택지의 다양화

BEAM(Erlang VM) 위에는 이제 다양한 타입 선호도의 언어가 공존한다.
완전 동적 타입의 Erlang, 점진적 타입의 Elixir, 완전 정적 타입의 Gleam이 같은 VM 위에서 동작한다.

GeekNews 댓글에서 Gleam과의 비교 논의가 이루어졌다.
Gleam은 BEAM 위에서 동작하는 완전 정적 타입 언어로, 타입 안전성만 보면 앞선다.
그러나 Phoenix, Ecto, 매크로 생태계의 성숙도에서 Elixir가 압도적이다.
v1.20은 Elixir의 위치를 명확히 한다.
완전 정적 타입의 엄밀함보다 실용적 점진성과 생태계를 선택한 언어로.

HN에서 Elixir, Go, Rust, Node를 두루 경험한 한 개발자는 결국 OCaml로 이동했다고 밝혔다.[^anonyfox]
"진짜 타입 시스템은 이제 내가 절대 타협하지 않는 요소"라는 그의 입장은 점진적 타입이 모든 개발자를 만족시킬 수는 없음을 보여준다.
Elixir의 BEAM 생태계가 강점이지만, 컴파일러 수준의 완전한 보장을 원하는 개발자에게는 점진성이 오히려 한계로 작용한다.

### 컴파일러가 무료로 찾아주는 버그의 경제학

버그의 발견 시점이 수정 비용을 결정한다.
개발 중 발견하면 수정 비용이 낮다.
코드 리뷰에서 발견하면 조금 더 높다.
프로덕션에서 발견하면 매우 높다.
컴파일 시간에 버그를 잡는 것은 가장 저렴한 시점에 가장 확실한 방식으로 잡는 것이다.

타입 어노테이션 없이 추론만으로 버그를 잡는다는 것은 이 이득을 기존 코드에도 소급 적용한다는 의미다.
레거시 코드를 재작성하지 않아도 컴파일러가 더 많은 문제를 미리 알려준다.

그러나 더 중요한 경제적 함의가 있다.
“검증된 버그”만 보고한다는 설계는 개발자의 주의를 낭비하지 않는다.
거짓 양성으로 가득 찬 경고 목록은 개발자가 경고를 무시하게 훈련시킨다.
경고를 신뢰할 수 있을 때만 경고가 실제로 가치를 만든다.
v1.20의 타입 시스템이 이 신뢰를 확보하는 것이 장기적으로 더 중요한 가치다.

HN에서 현업 Elixir 개발자들이 이 가치를 직접 경험담으로 전했다.
“Elixir 버전을 올리면 아무 변경 없이 컴파일러가 공짜로 버그를 무더기로 찾아준다. 완전히 버릇 들었다”는 반응은[^mrdoops] 이 이득이 이론이 아닌 실제 업그레이드 경험임을 보여준다.
