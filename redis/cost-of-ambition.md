# Redis와 야망의 대가

원문: <https://charlesleifer.com/blog/redis-and-the-cost-of-ambition/>

HN 토론: <https://news.ycombinator.com/item?id=48118553> (36점, 6개 댓글)

## 요약

peewee ORM 개발자이자 lsm-tree 기반 데이터베이스 작성자인 Charles Leifer가
Redis가 어떻게 우아한 집중의 도구에서 과도한 야망의 플랫폼으로 변해갔는지를
분석한 글이다. Redis는 2011년경 “고급 키-값 저장소”이자 데이터 구조 서버로
등장했다. 스스로를 데이터베이스라고 부르지 않았다는 점이 의미심장하다.

저자가 꼽는 초기 Redis의 성공 요인은 세 가지다. 첫째, RESP 프로토콜은 단순하면서도
표현력이 충분해 클라이언트 라이브러리 개발이 쉬웠다. 둘째, 단일 스레드 이벤트
드리븐 아키텍처가 복잡한 동기화 없이 원자적 연산을 보장했다. 셋째, 데이터
구조들이 웹 앱의 실제 필요 — 캐싱, 큐잉, 카운터, 속도 제한, 리더보드 — 에
자연스럽게 들어맞았다.

문제는 Redis가 본격적인 데이터베이스가 되려는 야망을 품으면서 시작됐다. 이후 추가된
기능들은 당시 커뮤니티에서 유행하던 것들을 쫓아간 흔적이 역력하다. MongoDB를
따라 JSON 문서 지원, ElasticSearch를 따라 전문 검색, 그래프 데이터베이스 트렌드에
맞춘 그래프 기능, Kafka를 따라 이벤트 스트리밍, AI 열풍에 맞춘 벡터 저장소까지
추가됐다.

저자의 핵심 비판은 이 확장이 구조적 오류라는 것이다. 전문 기능이 진지하게 필요한
사용자는 목적에 맞게 만들어진 솔루션을 원하지, Redis의 모든 아키텍처적 제약 —
고가용성, 영속성, 프로토콜 복잡성 — 을 짊어진 반쪽짜리 Redis 모듈을 원하지
않는다. 결국 Valkey의 부상은 2011년 Redis 기능 수준으로도 충분한 다수의 사용자들이
내린 시장의 심판이었다.

## 분석

### 정체성 상실의 메커니즘

Redis의 쇠락은 단순히 “기능이 너무 많아진” 문제가 아니다. 저자는 ACL 기능을
예로 들며 그것이 “근본적으로 Redis답지 않다(fundamentally un-Redis)”고 표현한다.
이 진단이 핵심이다. 어떤 도구에 기능이 추가될 때, 그것이 기존의 설계 철학과
정합성을 갖는지가 중요하다.

Redis의 원래 설계 철학은 직교성(orthogonality)이었다. 각 명령이 독립적이고,
데이터 구조들이 서로 간섭하지 않았다. 그런데 모듈 시스템, ACL, 클러스터, 스트림
같은 기능들이 쌓이면서 이 직교성이 무너졌다. 무엇이 가능하고 무엇이 불가능한지,
어떤 제약 조건이 어디에 적용되는지 추적하기 어려워졌다.

### Disque 사례: 솔직한 자기 진단의 역설

Disque는 2015년 Redis 창시자 Antirez가 발표한 메시지 큐 프로젝트다. Antirez
본인이 “우주인 모드(astronaut mode)”로 만들었다고 인정했다 — 실제 사용자 문제를
해결하는 것이 아니라 지적 탐구로 만들었다는 고백이다. 저자는 당시에 이것이
결국 폐기될 것이라 예측했고, 실제로 Disque는 GitHub 스타 8,000개를 받고도
트랙션을 얻지 못한 채 방치됐다.

이 사례가 흥미로운 이유는, 창시자 자신이 자기 과오를 정확히 진단하면서도 패턴이
반복됐다는 점이다. Disque의 교훈을 살렸다면 이후의 기능 확장들도 더 신중했을
것이다. 그러나 기업화 이후 조직의 압력은 개인의 자기 반성보다 강했다.

### Redis-Raft와 구현 품질 문제

저자는 Aphyr의 Redis-Raft 구현 분석을 인용한다. 초기 버전에서 21개의 심각한
문제가 발견됐는데, 여기에는 “장기 가용성 불가(long-lasting unavailability)”와
데이터 손실 시나리오가 포함됐다. 이것은 단순한 버그의 문제가 아니다.

Raft는 분산 합의 알고리즘으로, 정확하게 구현하기 위해 정교한 엔지니어링이 필요하다.
이를 Redis 위에 올리려면 Redis의 단일 스레드 아키텍처, 비동기 복제 모델과 Raft의
동기적 합의 모델을 어떻게 조화시킬지에 대한 깊은 설계 고민이 필요하다. 21개의
심각한 문제는 이 조화가 충분히 이뤄지지 않았음을 보여준다.

그러나 이 인용에는 맥락 누락이라는 문제가 있다. nicwolff는 저자가 Aphyr의
분석을 마치 출시된 제품에 적용된 것처럼 인용했지만, 실제로는 프리릴리스 내부
개발 빌드를 대상으로 한 분석이었음을 지적한다.[^nicwolff] Aphyr 본인도 "실제
프로덕션 사용자가 없으므로 현실적 영향은 무시할 수 있다"고 명시했고, 21개 중
20개는 리뷰 발표 전에 이미 수정됐다. 이 지적은 유효하다 — 저자의 비판이 틀린
것은 아니지만, 인용 방식이 실제보다 훨씬 심각한 인상을 줄 수 있다.

### Valkey와 시장 심판

Valkey는 Redis의 라이선스 변경(BSD → SSPL) 이후 커뮤니티가 포크한 프로젝트다.
저자는 Valkey가 화려한 기능 추가 대신 멀티스레드 성능, 메모리 효율, 클러스터
안정성 같은 “화려하지 않은 엔지니어링”에 집중한다고 평가한다. 이것이 2011년
Redis 기능 수준으로도 충분한 사용자 다수에게 선택받는 이유다.

## 비평

### 긍정적 측면

저자의 분석은 구체적인 기술 판단에 근거한다. “기능이 너무 많다”는 막연한 비판이
아니라 RESP 프로토콜의 단순성, 단일 스레드 아키텍처의 원자성 보장, 직교적 명령
설계 같은 구체적인 속성들이 왜 가치 있었는지를 설명한다. Disque 사례와 Redis-Raft
분석을 근거로 제시하는 것도 주장에 무게를 더한다.

Antirez의 “우주인 모드” 발언을 인용해 Redis 팀 스스로도 이 문제를 인식했다는
점을 보여주는 구성이 효과적이다. 외부 비판이 아니라 내부 자기 진단과 일치한다는
것이다.

### 한계

저자가 Redis의 문제로 진단하는 것들 — JSON 지원, 전문 검색, 벡터 저장소 — 은
실제로 많은 사용자가 요청했던 기능들이다. 특히 벡터 저장소는 AI 애플리케이션
시대에 많은 팀이 기존 Redis 인프라 위에서 임베딩 검색을 하려는 실용적 필요에서
나왔다. “전문 솔루션을 쓰면 된다”는 조언은 현실적으로 가능한 팀에게는 맞지만,
인프라 운영 복잡성을 줄이려는 팀에게는 단순화 유인이 있다.

또한 Redis Labs(현 Redis Inc.)의 기업적 압력을 충분히 다루지 않는다. 오픈소스
프로젝트가 기업 지원을 받으면 어떤 기능이 추가되는지는 기술적 판단만이 아니라
영업 요구사항, 경쟁사 대응, 투자자 기대에 의해 결정된다. 개발자 공동체의 관점에서
보면 “야망의 대가”지만, 기업 입장에서는 “시장 요구에 대한 대응”이었을 수 있다.

HN 토론에서도 저자의 이중 잣대를 지적하는 목소리가 있었다. sc68cal은 “Just use
Postgres”를 권장하는 글들이 HN에 넘쳐나는 상황에서, Postgres가 메시지 큐·분산 락·
JSON 문서 저장소·벡터 DB로 쓰이는 것은 허용되면서 Redis만 비난받는 이유가 무엇이냐고
묻는다.[^sc68cal] 기능 확장 자체가 문제라기보다, 그 도구의 원래 설계 철학과 맞지
않을 때 문제가 된다는 논점을 이 반론이 보완한다. epolanski 역시 핵심 키-값 저장소
기능이 나빠진 것은 아니며 더 많은 데이터 구조를 지원하는 게 왜 나쁜 일인지
불분명하다고 지적한다.[^epolanski]

## 인사이트

### 단순성은 제약이 아니라 설계 결정이다

Redis의 원래 성공은 아무것도 하지 않아서가 아니었다. 매우 구체적인 것들을 매우
잘 해서였다. 이 차이가 중요하다. “단순함”은 수동적인 상태가 아니라 능동적인
선택이다. 무엇을 포함하지 않을지를 결정하는 것이다.

Redis의 5가지 핵심 자료구조(문자열, 리스트, 해시, 집합, 정렬된 집합)는 그냥
생긴 것이 아니다. 웹 애플리케이션의 일반적인 패턴을 추상화한 결과물이다. 캐싱은
문자열로, 메시지 큐는 리스트로, 세션은 해시로, 태그는 집합으로, 랭킹은 정렬된
집합으로 자연스럽게 표현된다. 이 다섯 가지가 나머지를 불필요하게 만들 만큼 충분히
강력했다.

새로운 자료구조를 추가할 때 Redis가 물어야 할 질문은 “이것을 구현할 수 있는가”가
아니라 “이것이 기존 자료구조로 표현할 수 없는 패턴을 해결하는가”여야 했다. 전문
검색, 그래프, 벡터는 이 테스트를 통과하지 못한다. 기존 자료구조로는 불편하지만,
그 불편함의 해결이 Redis의 설계 철학과 맞지 않는 새 복잡성을 도입하는 것보다
낫다.

### 포식자와 먹이의 역설: 경쟁 압력이 정체성을 삼킨다

Kafka를 보고 스트리밍을 추가하고, ElasticSearch를 보고 검색을 추가하고, pgvector를
보고 벡터 저장소를 추가하는 패턴은 경쟁 압력에 대한 반응이다. 이 반응의 문제는
각 추가가 경쟁자를 이길 수 없는 반쪽짜리 구현이라는 점이다.

진지하게 Kafka가 필요한 팀은 Redis Streams로 갈아타지 않는다. Kafka의 소비자
그룹, 파티션, 오프셋 관리, 컨슈머 그룹 재조정 같은 기능들이 실제로 필요하기
때문이다. Redis Streams는 간단한 큐가 필요한 팀에게는 유용하지만, Kafka가 필요한
팀에게는 도달하지 못한다. 결과적으로 Redis는 단순 큐로 충분했던 팀에게 불필요한
복잡성을 더하면서, Kafka가 필요한 팀은 여전히 Kafka를 쓴다.

이 패턴은 소프트웨어 역사에서 반복된다. Microsoft Word가 DTP를 따라 레이아웃
기능을 추가했을 때, InDesign 사용자들을 가져오지 못하면서 Word의 단순성만 희생됐다.
Internet Explorer가 브라우저 점유율을 이용해 HTML 표준을 벗어난 기능을 추가했을 때,
웹 개발자들에게 짐만 됐다. 경쟁자의 시장을 빼앗으려 했지만, 결과는 자신의 사용자를
밀어낸 것이었다.

### 오픈소스 기업화의 구조적 긴장: 기여자와 투자자 사이

Redis Labs의 상업화는 Redis에 두 가지 상충하는 압력을 가했다. 첫 번째는 투자자와
기업 고객의 압력으로, 전통적인 데이터베이스 시장에서 경쟁하기 위한 기능 세트를
요구한다. 두 번째는 오픈소스 커뮤니티의 기대로, Redis가 왜 좋았는지를 아는
사람들은 그 본질을 지키길 원한다.

이 긴장은 필연적으로 발생한다. 투자자는 TAM(Total Addressable Market)을 키우길
원하고, 그것은 더 많은 사용 사례를 커버하는 것을 의미한다. Redis의 원래 TAM은
캐싱, 세션, 큐 같은 보조 데이터 저장소 시장이었다. 이것을 주 데이터베이스 시장으로
확장하려면 Redis가 아닌 무언가가 돼야 한다.

라이선스를 BSD에서 SSPL로 변경한 것도 이 구조적 긴장의 표출이다. 클라우드 제공사
(특히 AWS ElastiCache)가 Redis를 상업적으로 제공하면서 Redis Labs에 기여하지
않는 상황이 지속될 수 없었다. SSPL은 그 수익을 회수하려는 시도였지만, 결과적으로
커뮤니티가 Valkey로 이탈하는 계기가 됐다. 기업이 오픈소스를 통제하려는 순간,
오픈소스의 본질적 강점인 커뮤니티를 잃는다.
