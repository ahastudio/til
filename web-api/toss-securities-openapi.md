# 토스 증권 OpenAPI — LLM 친화적 설계와 개발자 경험

원문: <https://jessyt.tistory.com/480>

## 요약

토스 증권이 자동화 매매를 위한 OpenAPI를 공개하며 단계적 출시를 시작했다.
한 개발자가 직접 API를 통합하고 자동화 매매 시스템을 구축한 경험을 상세히 기록한 글이다.
이전에는 매매 신호를 수동으로 처리했지만,
API를 통해 종목 선정부터 주문 실행까지 전 과정을 자동화했다.

가장 두드러진 특징은 `llms.txt` 파일 제공이다.
API 문서를 LLM이 읽기 좋은 형식으로 정리한 이 파일 덕분에,
Claude Code에 전달하는 것만으로 인증부터 주문까지의 전체 흐름을 단일 파일 Python 클라이언트로 자동 생성했다.
표준 라이브러리만 사용한 클라이언트 생성이 가능했다는 점이 주목할 만하다.

인증은 Client ID와 Client Secret 두 가지 환경 변수만으로 OAuth2 토큰을 발급받는 방식으로 단순하다.
하나의 토큰으로 시세 조회, 계좌 접근, 주문 실행까지 모든 작업이 가능하다.
국내 경쟁사인 키움증권이나 한국투자증권과의 차별점으로 저자는
국내외 주식을 동일한 단일 엔드포인트에서 처리할 수 있다는 점을 꼽았다.

에러 처리 설계가 개발자 경험 측면에서 인상적이다.
`insufficient-buying-power` 같은 읽기 쉬운 에러 코드, 한국어 사람이 읽을 수 있는 메시지,
허용 값 힌트(`allowedValues`), 429 상태에서의 Retry-After 헤더,
중복 주문 방지를 위한 멱등성 키(idempotency key)를 모두 지원한다.

한계도 명확하다.
REST 전용으로 WebSocket 실시간 시세를 지원하지 않으며,
대안으로 1초 폴링을 사용해야 한다.
샌드박스 또는 모의 투자 환경도 제공하지 않아 프로덕션 전 테스트가 어렵다.
현재 사전 신청 후 순차 승인 방식으로만 접근 가능하다.

## 분석

### `llms.txt`가 개발자 경험의 새 기준점을 만든다

저자가 “Claude Code에 문서를 주고 Python 클라이언트를 만들었다”는 한 문장이
이 API의 핵심 가치를 압축한다.
LLM에게 API를 설명하는 문서를 주면 코드가 나온다는 경험은
개발자가 API와 상호작용하는 방식을 근본적으로 바꾼다.

`llms.txt`는 현재 표준화 논의가 진행 중인 비공식 규약으로,
웹사이트나 서비스가 LLM에게 자신을 설명하기 위한 형식이다.
토스 증권이 금융 API에 이를 적용한 것은 상징적이다.
복잡한 인증 흐름, 도메인 특화 용어, 다양한 엔드포인트 조합이 있는 금융 API는
전통적으로 학습 곡선이 가파른 영역이었다.
LLM 친화적 문서화가 이 장벽을 낮출 수 있다는 실증 사례다.

### 에러 설계는 API의 철학을 드러낸다

`{“error”: {“code”: “invalid-request”, “message”: “주문 방향이 올바르지 않습니다.”, “data”: {“allowedValues”: [“BUY”, “SELL”]}}}`

이 에러 응답 구조는 개발자를 “디버깅 파트너”로 대우한다는 신호다.
숫자 에러 코드 대신 사람이 읽을 수 있는 문자열을 사용하고,
한국어 메시지로 도메인 컨텍스트를 제공하며,
`allowedValues`로 다음 단계를 직접 안내한다.

이는 국내 전통 증권사 API가 수십 년 동안 유지해온 설계 관행과 대조적이다.
에러 코드 테이블을 별도로 참조해야 하거나,
에러 메시지가 내부 시스템 오류 코드를 그대로 노출하는 방식은
API를 쓰는 개발자가 아닌, API를 만든 내부 팀의 편의를 위한 것이다.
Retry-After 헤더와 멱등성 키 지원까지 갖추면, 이 API는 분산 시스템 설계 원칙을 금융 도메인에 적용했다고 볼 수 있다.

### 단일 엔드포인트 통합이 주는 아키텍처 이점

국내외 주식을 동일한 엔드포인트에서 처리한다는 것은
클라이언트 코드에서 시장 구분 로직을 제거할 수 있다는 의미다.
경쟁사처럼 국내 API와 해외 API가 분리되어 있으면,
인증 토큰, 에러 처리, 데이터 변환 레이어를 각각 따로 구현해야 한다.

이것은 표면적으로는 작은 차이처럼 보이지만,
자동화 시스템을 구축하는 개발자 입장에서는 복잡성이 제곱으로 늘어나는 문제다.
글로벌 포트폴리오를 운용하는 개인 투자자나 소규모 팀이
처음부터 국내외 통합 시스템을 구축하기 어려운 이유가 바로 이 분절화에 있다.

## 비평

### WebSocket 미지원은 전략적 공백이 아닌 구조적 제약이다

1초 폴링이 “대안”으로 제시되지만, 이는 실시간 트레이딩에서 근본적으로 다른 것이다.
스캘핑이나 변동성 이벤트 대응을 고려하는 사용자에게 1초 지연은 치명적일 수 있다.

그러나 이 한계를 API 설계 실패로 해석하기보다,
토스 증권이 처음부터 타깃으로 삼는 사용자 유형의 신호로 읽어야 한다.
1초 폴링이 허용 가능한 사용 사례 — 일중 추세 추종, 포지션 관리, 리밸런싱 —
에 집중한 API라는 것이다.
WebSocket 지원 없이 출시한 것이 자원 제약인지 의도적 스코핑인지는 불명확하지만,
향후 로드맵에서 이 격차를 어떻게 채울지가 API의 진지함을 가늠하는 지표가 될 것이다.

### 샌드박스 부재가 만드는 채택 장벽

실제 자금이 걸린 API에 샌드박스 없이 코드를 통합한다는 것은
개발자에게 심리적, 실질적 장벽을 동시에 만든다.
저자는 Claude가 생성한 백테스팅 시뮬레이션으로 이를 우회했지만,
이것은 개인 투자자가 직접 해결한 것이지 API가 제공한 것이 아니다.

샌드박스가 없으면 버그가 있는 코드가 실제 주문으로 이어질 수 있다.
이 위험이 개발자가 API 탐색에 투자하는 시간과 노력을 억제하고,
결과적으로 생태계 성장을 늦춘다.
스트라이프나 트위터 API가 풍성한 서드파티 생태계를 만들 수 있었던 이유 중 하나는
안전한 실험 환경을 제공했기 때문이다.

### 사전 신청 방식이 피드백 루프를 차단한다

단계적 출시의 의도는 이해하지만,
대기 목록을 통해서만 접근할 수 있는 API는
커뮤니티 피드백을 늦춘다.
이 글 자체가 그 증거다.
글에서 핵심 링크가 누락되자 nimki가 댓글로 사전 신청 URL과 문서 링크를 보충했고,[^nimki]
qwkjsan99와 recast7838은 접근 승인을 기다리는 상황을 댓글로 공유했다.[^qwkjsan99][^recast7838]

빠른 이터레이션이 중요한 API 초기 단계에서
개발자 커뮤니티의 실제 사용 피드백을 제한하는 것은
장기적으로 API 품질에 불리하게 작용한다.

## 인사이트

### `llms.txt`는 API 문서화의 패러다임 전환을 예고한다

전통적인 API 문서는 사람이 읽도록 설계되었다.
레퍼런스 문서, 튜토리얼, SDK — 이 모든 것이 인간 개발자를 최종 사용자로 가정한다.
`llms.txt`는 이 가정을 뒤집는다.
LLM이 API를 이해하고 코드를 생성할 수 있다면,
문서의 첫 번째 독자는 사람이 아닌 AI다.

이 전환이 완성되면 API 채택의 병목이 달라진다.
“문서가 잘 쓰였는가”에서 “LLM이 이 API를 얼마나 정확하게 추론할 수 있는가”로 이동한다.
이는 API 설계 원칙에도 영향을 준다.
LLM이 잘 추론하는 API는 일관된 명명 규칙, 예측 가능한 구조, 명시적 제약을 갖는다.
역설적으로, LLM 친화적 API는 사람도 이해하기 쉬운 API와 상당 부분 겹친다.

### 증권사 API가 AI 금융 에이전트의 실행 레이어가 된다

토스 증권 OpenAPI의 진짜 의미는
개인 투자자가 자동화 매매 시스템을 구축하기 쉬워진다는 것에 그치지 않는다.
LLM 기반 금융 에이전트가 포트폴리오를 분석하고 리밸런싱 전략을 결정하며
실제 주문을 실행하는 파이프라인의 실행 레이어가 될 수 있다.

이 파이프라인에서 `llms.txt`는 에이전트가 자신이 사용할 도구의 능력과 제약을
정확하게 이해하는 데 필수적이다.
에이전트가 WebSocket이 없다는 사실을 모르고 실시간 데이터를 요청하거나,
멱등성 키 없이 네트워크 오류 시 재시도하면 중복 주문이 발생할 수 있다.
LLM 친화적 문서화는 에이전트 안전성의 인프라이기도 하다.

### 개발자 경험(DX)이 금융 인프라 경쟁에서 차별화 요소가 된다

국내 증권사 API는 오랫동안 기관 투자자와 대형 증권사를 위한 인프라였다.
개인 투자자가 직접 API를 사용하는 수요는 알고리즘 트레이딩에 관심 있는 소수에 한정되었고,
그 수요를 위한 DX 투자는 우선순위가 낮았다.

토스 증권의 OpenAPI는 이 패턴을 깨려는 시도다.
`llms.txt`, 인간 친화적 에러 코드, 통합 엔드포인트 — 이 선택들은
일반 개발자를 첫 번째 사용자로 상정한 설계 결정이다.
이 방향이 성공하면, 다른 금융 기관들도 DX를 경쟁 전략으로 채택하게 될 것이다.
금융 API의 DX 경쟁이 본격화되면
그 수혜자는 개인 개발자와 소규모 핀테크 팀이다.
