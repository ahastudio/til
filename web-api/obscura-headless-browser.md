# Obscura

<https://github.com/h4ckf0r0day/obscura>

## 소개

Rust로 작성된 오픈소스 헤드리스 브라우저로, 웹 스크래핑과 AI 에이전트 자동화를 위해 설계되었다.
V8을 통한 실제 JavaScript 실행과 Chrome DevTools Protocol(CDP)을 지원하며, Puppeteer 및 Playwright와 호환된다.
라이선스는 Apache 2.0이며 GitHub에서 6,400개 이상의 스타를 받았다.

## 성능 비교

헤드리스 Chrome 대비 현저히 가벼운 사용량을 주장한다.

| 항목        | Obscura | 헤드리스 Chrome |
| ----------- | ------- | --------------- |
| 메모리 사용 | 30 MB   | 200 MB+         |
| 바이너리 크기 | 70 MB | 300 MB+         |
| 페이지 로드 | 85 ms   | ~500 ms         |
| 시작 시간   | 즉시    | ~2초            |

## 주요 기능

- V8 기반 실제 JavaScript 실행
- Chrome DevTools Protocol 구현
- Puppeteer-core 및 Playwright-core 호환
- 외부 의존성 없는 단일 바이너리 배포
- 스텔스 모드: 안티 핑거프린팅, 3,520개 도메인 트래커 차단
- 세션별 핑거프린트 무작위화
- 현실적인 User Agent 데이터

## 사용법

세 가지 방식으로 사용할 수 있다.

**CLI**: 단일 페이지 렌더링이나 배치 스크래핑에 사용한다.

```bash
obscura fetch https://example.com
obscura scrape --batch urls.txt
```

**CDP WebSocket 서버**: 프로그래밍 방식으로 제어한다.

```bash
obscura serve --port 9222
```

**Node.js 라이브러리**: 기존 Puppeteer/Playwright 코드를 그대로 활용한다.

```javascript
const browser = await puppeteer.connect({
  browserWSEndpoint: 'ws://localhost:9222'
});
```

## 분석

Obscura의 설계는 두 가지 문제를 동시에 해결하려 한다.
첫째, 헤드리스 Chrome의 무거운 자원 사용이다.
컨테이너 환경이나 서버리스 환경에서 200MB 이상의 메모리와 300MB 바이너리는 실질적인 제약이다.
둘째, 봇 탐지 우회다.
AI 에이전트와 웹 스크래핑의 확산으로 웹사이트들이 봇 탐지를 강화하면서, 스텔스 모드는 점점 필수 기능이 되고 있다.

[GeekNews 댓글](https://news.hada.io/topic?id=28920)에서 주목할 만한 우려가 제기되었다.
GitHub 계정이 한 달 전 생성되었고 외부 가시성이 부족하다는 점이 신뢰성 의문으로 이어졌다.
6,000개 이상의 스타가 있어도 민감한 작업에 바로 사용하기에는 출처가 불명확하다는 경고다.
xguru는 주로 Twitter를 통해 화제가 되었다고 언급했으며, 봇 차단 우회라는 민감한 용도를 감안할 때 신중한 검토가 필요하다는 의견이 지배적이었다.

Rust 선택은 성능과 메모리 안전성 측면에서 이해할 수 있지만, Chromium의 전체 렌더링 엔진을 대체하지 않고 V8만 포함한다는 점에서 CSS 렌더링이나 복잡한 레이아웃 처리에서 한계가 있을 수 있다.

## 비평

### 강점: 경량화와 호환성의 균형

CDP 호환을 통해 기존 Puppeteer/Playwright 코드를 그대로 사용할 수 있다는 점은 실용적 채택 장벽을 크게 낮춘다.
단일 바이너리 배포도 DevOps 관점에서 매력적이다.

### 한계: 신뢰성과 출처 불명확성

봇 탐지 우회, 트래커 차단 등의 기능은 정당한 용도와 악용 사이의 경계에 있다.
이런 도구에서 코드베이스의 신뢰성과 저자의 투명성은 특히 중요하다.
계정 생성 후 한 달 만에 6,000개 스타를 받는 것은 유기적 성장보다는 조직적 홍보의 가능성을 시사한다.

### 실제 정확도 검증 필요

주장된 성능 수치들(85ms 페이지 로드, 30MB 메모리)은 어떤 조건에서 측정되었는지 명확하지 않다.
독립적인 벤치마크 없이는 이 수치들을 그대로 신뢰하기 어렵다.

## 인사이트

### 봇과 웹의 군비 경쟁

Obscura의 스텔스 모드는 웹 자동화와 봇 탐지 사이의 끝없는 군비 경쟁을 반영한다.
웹사이트가 더 정교한 핑거프린팅을 도입하면, 자동화 도구는 더 정교한 위장을 개발한다.
이 경쟁은 웹의 개방성에 대한 근본적인 긴장을 드러낸다.
누구나 공개된 웹 데이터에 접근할 수 있어야 한다는 원칙과, 웹사이트가 무단 대규모 크롤링으로부터 자신을 보호할 권리 사이의 충돌이다.
Obscura 같은 도구는 그 긴장의 현재 상태를 보여주는 지표다.

### AI 에이전트 시대의 브라우저 인프라

AI 에이전트가 웹을 탐색하고 작업을 수행하는 패턴이 일반화되면서, 에이전트를 위한 경량 브라우저 인프라의 수요는 증가하고 있다.
Playwright나 Puppeteer가 인간 테스터를 자동화하기 위해 설계되었다면, Obscura는 처음부터 AI 에이전트를 위해 설계된 도구를 표방한다.
이 방향성은 의미 있다.
에이전트는 사람과 다른 패턴으로 웹을 사용한다.
병렬 처리, 지속적 세션 관리, 최소 자원 사용이 핵심이다.
경량 헤드리스 브라우저 시장은 AI 에이전트 인프라의 필수 구성요소로 성장할 것이다.

### 오픈소스 신뢰의 조건

출처가 불분명한 도구가 빠르게 스타를 모으는 현상은 오픈소스 생태계에서 반복되는 패턴이다.
GitHub 스타는 코드 품질의 지표가 아니다.
특히 보안에 민감한 기능(봇 우회, 트래커 차단)을 포함한 도구라면, 코드 감사, 저자 신원 확인, 독립적 테스트를 거치지 않고 프로덕션에 도입하는 것은 공급망 리스크를 자초하는 행위다.
최근 공급망 공격의 증가를 고려하면, 매력적인 성능 수치와 스타 수가 신뢰의 대리지표가 될 수 없음을 다시 한번 상기해야 한다.
