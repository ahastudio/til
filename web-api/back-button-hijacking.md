# Google, 뒤로 가기 버튼 하이재킹을 스팸으로 지정

원문: <https://developers.google.com/search/blog/2026/04/back-button-hijacking>

## 요약

Google이 “뒤로 가기 버튼 하이재킹(back button hijacking)”을
새로운 스팸 정책 위반 항목으로 지정했다.
2026년 6월 15일부터 시행된다.

이 행위는 사용자가 브라우저의 뒤로 가기 버튼을 눌렀을 때
이전 페이지로 돌아가지 못하게 하고,
원치 않는 광고나 추천 콘텐츠로 리다이렉트하는 것을 말한다.

위반 사이트는 수동 스팸 조치(manual spam action)나
자동 순위 패널티를 받게 된다.
사이트 운영자는 브라우저 탐색 기록을 조작하는 코드를 제거해야 하며,
Search Console의 재검토 요청을 통해 복구할 수 있다.

## 분석

### 기술적 배경

뒤로 가기 버튼 하이재킹은 주로 `history.pushState()`나
`history.replaceState()` API를 악용하여 구현된다.
사용자가 페이지에 진입하는 순간 브라우저 히스토리 스택에
가짜 항목을 삽입하거나,
`popstate` 이벤트를 가로채서 다른 URL로 리다이렉트하는 방식이다.

### 스팸 정책의 구체적 범위

Google은 이번 정책에서 브라우저의 자연스러운 탐색 흐름을
의도적으로 방해하는 모든 행위를 대상으로 한다.
여기에는 히스토리 조작뿐 아니라
뒤로 가기 시 전면 광고를 띄우거나
무한 루프에 빠뜨리는 패턴도 포함된다.

### 시행 방식

수동 조치와 자동 알고리즘 패널티 두 가지 경로가 존재한다.
수동 조치는 Google 검색 품질 팀의 직접 검토를 통해 이루어지며,
자동 패널티는 크롤러가 해당 패턴을 감지했을 때 적용된다.
Search Console을 통한 재검토 요청으로 복구가 가능하다는 점은
기존 스팸 정책과 동일한 구조를 따른다.

## 비평

### 긍정적 측면

오랫동안 사용자를 괴롭혀 온 패턴에 대해
검색 엔진 차원에서 공식적인 제재를 가한다는 점에서 의미가 크다.
GeekNews 댓글에서 “드디어!!!”라는 반응이 나온 것처럼
많은 사용자가 이 조치를 환영하고 있다.

### 한계와 의문점

Google 자체 광고 플랫폼에서 발생하는 유사한 문제에 대해서는
언급이 없다는 점이 지적된다.
“자기 플랫폼의 NSFW 광고부터 해결해야 하는 것 아닌가”라는
비판은 정당한 문제 제기다.

또한 Microsoft QA 사이트처럼 의도치 않게
뒤로 가기 무한 루프를 유발하는 경우와
의도적 하이재킹을 어떻게 구분할 것인지에 대한
기술적 기준이 명확하지 않다.

### 실효성 문제

정책 시행의 실효성은 탐지 기술의 정확도에 달려 있다.
JavaScript 기반의 히스토리 조작은 패턴이 다양하고,
서버 사이드 리다이렉트와 결합되면 탐지가 복잡해진다.
오탐(false positive)으로 인한 피해 가능성도 고려해야 한다.

## 인사이트

### 플랫폼 권력과 웹 생태계 규범의 형성

Google이 스팸 정책을 통해 특정 기술적 행위를 금지하는 것은
사실상 웹 생태계의 규범을 단독으로 제정하는 행위다.
W3C나 IETF 같은 표준화 기구가 아닌
하나의 기업이 “허용되는 브라우저 API 사용법”을 정의하고 있다.

이는 검색 엔진 점유율 90% 이상이라는 시장 지배력에서 비롯된다.
Google 검색에서 패널티를 받는 것은
대부분의 사이트에게 사실상 인터넷에서 사라지는 것을 의미한다.
따라서 이 정책은 기술적 권고가 아니라 사실상의 강제 규범이다.

역사적으로 Microsoft가 IE 시절 웹 표준을 사실상 지배했던 것과
유사한 구조적 패턴이다.
차이가 있다면 당시는 브라우저를 통한 기술적 지배였고
지금은 검색 순위를 통한 경제적 지배라는 점이다.

### 다크 패턴 규제의 전환점

뒤로 가기 하이재킹은 대표적인 다크 패턴(dark pattern) 중 하나다.
Google의 이번 조치는 다크 패턴에 대한 규제가
법적 영역에서 플랫폼 자체 규제 영역으로 확장되고 있음을 보여준다.
EU의 디지털 서비스법(DSA)이나 한국의 전자상거래법보다
실질적으로 더 즉각적인 효과를 발휘할 수 있다.

그러나 플랫폼 자체 규제는 선택적 적용의 위험이 있다.
Google 자체 서비스나 대형 광고주에게는 관대하고
소규모 사이트에게만 엄격하게 적용될 가능성을 배제할 수 없다.
이는 중립적 규제가 아닌 이해관계자에 의한 규제의 본질적 한계다.

Discord, Reddit, LinkedIn 등 대형 서비스에서도
유사한 UX 문제가 보고되고 있다는 Hacker News 논의는
이 문제가 소규모 스팸 사이트만의 문제가 아님을 시사한다.
대형 서비스에 대해서도 동일한 기준이 적용될 것인지가 핵심 쟁점이다.

### History API의 설계적 교훈

`history.pushState()`는 SPA(Single Page Application) 구현을 위해
도입된 강력한 API다.
그러나 이 API가 악용 가능하다는 점은 설계 시점부터 예견된 문제였다.
“강력한 기능은 악용 가능성을 내포한다”는
웹 API 설계의 근본적 딜레마를 다시 한번 드러낸다.

Permission API처럼 사용자 동의를 요구하는 방식이
히스토리 조작에는 적용되지 않았던 이유는
SPA가 대중화되기 전에 설계되었기 때문이다.
브라우저 차원에서 히스토리 조작을 제한하는 것이
근본적 해결책이 될 수 있지만,
기존 SPA 생태계와의 호환성 문제가 걸림돌이다.

이는 웹 플랫폼의 반복되는 패턴이다.
유연한 API를 제공하고, 악용이 발생하면, 사후적으로 제한하는 순서다.
`window.open()` 팝업 차단, `autoplay` 정책 등
같은 경로를 걸어온 사례가 이미 여럿 존재한다.
결국 브라우저 벤더들이 히스토리 API에 대해서도
유사한 제한을 도입할 가능성이 높다.
