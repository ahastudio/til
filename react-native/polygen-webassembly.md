# Polygen으로 React Native에 WebAssembly 가져오기

원문: <https://www.callstack.com/blog/bringing-webassembly-to-react-native-with-polygen>

## 요약

Callstack이 React Native 애플리케이션에서 WebAssembly(이하 WASM)를
네이티브에 가까운 성능으로 실행할 수 있게 하는 툴킷 **Polygen**을
공개했다.
웹 앱을 React Native로 이식할 때 CPU 집약적 연산을 처리하는 방법에
대한 오랜 공백을 메우는 시도다.

기존에는 두 가지 선택지가 있었다.
`react-native-webassembly`는 wasm3 인터프리터를 사용해 WASM 모듈을
런타임에 해석(interpret)한다.
해석 오버헤드 때문에 성능이 민감한 작업에는 적합하지 않다.
Turbo Module은 JSI 브리지를 통해 네이티브 성능을 내지만, 의존하는
라이브러리마다 C/C++ 브리지 코드를 작성하고 유지해야 하는 부담이
크다.

Polygen은 세 번째 경로를 택했다.
빌드 타임에 WASM 모듈을 C 코드로 컴파일(AOT, Ahead-Of-Time compilation)하고,
그 C 코드를 앱 바이너리에 포함시킨다.
결과적으로 런타임 해석 오버헤드가 없고, JIT 컴파일을 금지한 Apple
정책에도 저촉되지 않는다.

사용 방식은 웹 환경과 동일하다.
Metro 번들러 통합 덕분에 `.wasm` 파일을 ES 모듈처럼 import하면 된다.

```javascript
import { add } from './math.wasm';
const result = add(5, 3);
```

현재는 iOS 전용(CocoaPods 기반)이며, Android 지원, WASI 호환, 외부
언어 바인딩 개선이 로드맵에 포함되어 있다.
프로젝트는 공개 GitHub 저장소에서 “활발한 개발 중” 상태로 관리된다.

## 분석

### AOT 컴파일 전략의 트레이드오프

Polygen이 취한 AOT 방식은 WASM → C 변환 단계를 빌드 파이프라인에
끼워 넣는다.
런타임에 인터프리터를 유지할 필요가 없으므로 실행 성능이 향상되고,
코드가 이미 컴파일된 형태이므로 JIT 없이도 작동한다.
이 두 가지 속성이 Apple의 `entitlement` 제한을 피하면서도 성능을
확보하는 핵심이다.

반면 동적 로딩은 불가능해진다.
런타임에 임의의 WASM 모듈을 받아서 실행하는 패턴, 예컨대 서버에서
플러그인을 내려받아 실행하는 아키텍처는 이 접근법과 근본적으로
양립하지 않는다.
이는 기능적 제약이 아니라 설계 원칙의 제약이다.

### JavaScript 스레드 실행 문제

현재 WASM 함수 호출은 JavaScript 스레드에서 일어난다.
CPU 집약적 작업을 WASM으로 옮긴 이유가 성능 확보인데, 그 호출이
JS 스레드를 점유한다면 UI 블로킹 위험이 생긴다.
로드맵에 Web Workers 지원이 언급되어 있지만, 현재 버전에서는 개발자가
스스로 비동기 처리 방법을 마련해야 한다.

### 생태계 포지셔닝

Rust, C++ 등 WASM 타깃 언어로 작성된 라이브러리를 소스 빌드 없이
가져올 수 있다는 점은 중요한 포지셔닝이다.
오픈소스 암호화, 오디오 코덱, 물리 시뮬레이션 라이브러리 중 WASM
빌드를 공식 배포하는 것들이 이미 많다.
이 라이브러리들을 React Native에서 그대로 쓸 수 있는 경로가 생기면,
네이티브 모듈 생태계의 공백을 메우는 대안 경로로 자리잡을 수 있다.

## 비평

### 강점

AOT 컴파일이라는 접근 자체는 기술적으로 설득력이 있다.
“iOS에서 JIT를 쓰지 않고 성능을 내려면?” 이라는 질문에 대한 답으로서
WASM → C 변환은 논리적으로 타당하다.
웹과 동일한 import 문법을 유지한 점도 개발자 경험 측면에서 평가할
만하다.

### 약점

“네이티브에 가까운 성능”이라는 주장은 수치 없이는 빈 약속에 가깝다.
기사에는 벤치마크나 비교 측정값이 전혀 없다.
wasm3 인터프리터 대비 몇 배 빠른지, 실제 네이티브 코드 대비 오버헤드가
얼마나 되는지 수치로 제시하지 않으면, “근접한다”는 표현은 검증할 수
없다.

JS 스레드 블로킹 문제를 인정하면서도 해결책을 “미래 작업”으로 미룬
것은 큰 약점이다.
WASM을 필요로 하는 작업이 대부분 무거운 연산이라는 점에서, 현재
상태의 Polygen은 실제 프로덕션 사용이 어려운 경우가 많다.

iOS 전용이라는 제한도 기사 제목의 “React Native”라는 표현과 온도차가
있다.
React Native의 핵심 가치가 크로스플랫폼임을 감안하면, Android 지원
없이는 절반짜리 솔루션이다.

### 누락된 관점

WASM 모듈 크기가 앱 바이너리에 미치는 영향을 전혀 다루지 않는다.
C로 변환된 코드가 얼마나 커지는지, 빌드 시간이 얼마나 늘어나는지는
실사용 결정에 중요한 변수다.
또한 디버깅 경험, 즉 WASM 코드에서 오류가 났을 때 스택 트레이스가
어떻게 보이는지도 언급이 없다.

## 인사이트

### AOT 컴파일은 “실행 환경의 제약”을 “빌드 파이프라인의 문제”로 전환하는
전략이다

Apple이 JIT 컴파일을 금지한 이유는 동적으로 생성된 코드가 보안 검토를
우회할 수 있다는 우려 때문이다.
iOS 앱 심사 프로세스는 정적으로 제출된 바이너리를 기준으로 동작한다.
JIT는 본질적으로 이 모델과 충돌한다 — 실행 시점에 코드가 생성되면,
심사 시점과 실행 시점의 코드가 달라진다.

Polygen의 AOT 전략은 이 제약을 우회하는 대신 수용하면서, 제약의 무게를
런타임에서 빌드 타임으로 이동시킨다.
사용자의 기기가 아니라 개발자의 CI 서버에서 변환이 일어난다.
이 이동은 “제약 자체를 없애는 것”이 아니라 “제약이 가장 저렴하게
처리될 수 있는 위치를 찾는 것”이다.
소프트웨어 아키텍처에서 반복적으로 등장하는 패턴이다.

역사적 유사사례는 많다.
Emscripten이 C/C++를 JavaScript로 AOT 컴파일해서 브라우저의 언어 제약을
우회했고, Dart의 dart2js도 같은 구조다.
GraalVM Native Image는 JVM 언어를 AOT 컴파일해 JVM이 없는 환경에서
돌린다.
공통점은 모두 “런타임 환경의 제약”을 “빌드 타임 변환 비용”으로
교환했다는 것이다.
Polygen은 이 계보의 모바일 버전이다.

이 패턴의 구조적 한계도 같다.
AOT 컴파일은 동적성을 포기한다.
빌드 타임에 알 수 없는 코드는 포함할 수 없다.
이것이 “설계의 순결함”인가, “유연성의 희생”인가는 사용 맥락에 따라
달라진다.
플러그인 아키텍처, 사용자 정의 로직, 핫픽스 패턴을 필요로 하는
애플리케이션은 이 트레이드오프를 정면으로 마주하게 된다.

### WASM은 “언어의 휴전선”이 아니라 “성능 격리의 경계선”으로 재정의되고 있다

WebAssembly가 처음 설계될 때의 주요 내러티브는 “웹에서 C/C++/Rust를
돌린다”였다.
그러나 WASM의 진짜 가치는 언어 다양성보다 샌드박싱(sandboxing)에 있다.
WASM 모듈은 명시적으로 허가되지 않은 메모리에 접근할 수 없고,
호스트 시스템 콜을 직접 호출할 수 없다.
이 격리 특성은 신뢰할 수 없는 서드파티 코드를 안전하게 실행하기 위한
경계로 WASM을 재위치시킨다.

Polygen이 “모듈 샌드박싱”을 장점으로 나열한 것은 이 흐름을 인식한
결과다.
React Native 애플리케이션에서 외부 라이브러리를 WASM으로 감싸면,
그 라이브러리가 앱의 나머지 메모리나 파일 시스템을 오염시키는 것을
원천 차단할 수 있다.
이는 단순한 성능 최적화가 아니라 보안 아키텍처의 도구로서의 WASM이다.

Cloudflare Workers, Fastly Compute@Edge, Shopify Functions 등이 WASM을
플러그인 실행 환경으로 채택한 것도 같은 이유다.
신뢰할 수 없는 코드를 격리된 컨텍스트에서 돌리되 성능 페널티를
최소화하는 수단으로 WASM이 선택된다.
React Native 생태계에서 이 패턴이 성숙하면, “외부 SDK를 네이티브 모듈
대신 WASM으로 배포”하는 관행이 보안과 배포 유연성을 동시에 높이는
표준 패턴으로 자리잡을 수 있다.

언어 경계보다 성능과 보안의 경계로 WASM을 바라보는 관점은 WASI(WebAssembly
System Interface)와 결합될 때 더 강력해진다.
WASI는 WASM 모듈이 호스트 환경과 통신하는 표준 인터페이스를 정의한다.
Polygen의 로드맵에 WASI 지원이 포함된 것은 단순한 기능 추가가 아니라,
WASM을 플러그인 런타임으로 진화시키려는 방향성의 표현이다.

### 크로스플랫폼 도구의 진짜 경쟁력은 “실행 환경 간 패리티”다

React Native의 가치 명제는 “하나의 코드베이스로 여러 플랫폼”이다.
그런데 Polygen은 현재 iOS만 지원한다.
이 불일치는 단순한 개발 지연의 문제가 아니다.
크로스플랫폼 툴킷에서 플랫폼 패리티(parity, 동등성)가 깨지는 순간,
개발팀은 조건 분기를 관리해야 하고 테스트 매트릭스가 복잡해지며
“플랫폼 X에서만 동작하는 기능”이라는 아키텍처 부채가 생긴다.

React Native 생태계의 역사는 이 문제의 반복이다.
수많은 커뮤니티 라이브러리가 iOS 우선으로 개발되어 Android가 뒤따르거나
영원히 따라오지 않는 패턴을 보였다.
New Architecture 전환 과정에서도 플랫폼별 지원 타임라인이 달랐다.
Polygen이 iOS 우선으로 공개된 것은 애플의 실행 환경 제약이 더 까다롭기
때문에 기술적으로는 이해할 수 있다.
그러나 “React Native용 WASM 솔루션”으로 포지셔닝하려면 패리티 달성이
필수적이며, 그 달성 속도가 채택률을 결정할 것이다.

더 깊은 질문은 “왜 플랫폼 패리티가 항상 뒤처지는가”이다.
모바일 플랫폼의 구조적 이질성 — iOS의 Objective-C/Swift 런타임과
Android의 JVM/ART 런타임 — 이 추상화 레이어를 구축하는 비용을
근본적으로 높인다.
WASM이 약속하는 “한 번 컴파일, 어디서나 실행”이 이 이질성을 흡수하는
레이어가 될 수 있다는 점에서, Polygen의 완성도 있는 크로스플랫폼 지원은
단순한 기능이 아니라 WASM의 가치 명제를 증명하는 리트머스다.

장기적으로 봤을 때, 완전한 크로스플랫폼 WASM 실행 환경이 React Native에
자리잡으면, 네이티브 모듈 생태계의 구조가 달라질 수 있다.
Turbo Module 기반의 플랫폼별 브리지 코드 대신, WASM으로 배포된 모듈이
iOS와 Android를 동시에 커버하는 세계가 가능해진다.
이 세계에서 “네이티브 모듈 유지보수”의 의미는 지금과 완전히 달라진다.
Polygen이 그 세계로 가는 경로인지는 Android 지원이 어떤 완성도로
도착하느냐에 달려 있다.
