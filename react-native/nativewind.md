# NativeWind: React Native용 Tailwind CSS

원문: <https://www.nativewind.dev/>

## 소개

NativeWind는 React Native 앱에서 Tailwind CSS 유틸리티 클래스를 사용해
스타일링할 수 있게 해주는 라이브러리다.
웹 개발에서 익숙한 유틸리티 퍼스트(utility-first) 워크플로를
iOS, Android, 웹 등 모든 플랫폼에서 일관되게 사용할 수 있다.

현재 v4가 프로덕션 버전이며, Tailwind CSS v3을 기반으로 한다.
v5는 프리릴리스 단계로 Tailwind CSS v4와 맞추는 작업이 진행 중이다.

## 작동 원리

### 빌드 타임 컴파일

빌드 단계에서 Tailwind CSS 클래스명을 React Native `StyleSheet.create` 객체로 변환한다.
컴파일러는 `lightningcss`로 재작성되어 v2 대비 유의미하게 빠르다.
`rem` 단위는 기본적으로 빌드 타임에 `px`로 인라이닝(inlining)하여
런타임 계산 비용을 제거한다.

### 런타임 처리

미디어 쿼리, 다크 모드, 디바이스 방향 전환 같은 반응형 스타일은
최소한의 런타임에서 처리한다.
정적으로 결정할 수 있는 스타일은 모두 빌드 타임에 처리하고,
동적으로 바뀌는 스타일만 런타임에 적용해 성능을 최적화한다.

### JSX 트랜스폼 방식 (v4)

v4의 핵심 아키텍처 변화는 Babel 플러그인에서 `jsxImportSource` 트랜스폼으로의
전환이다.
이전 방식은 `className` prop을 빌드 타임에 완전히 제거해버렸지만,
v4는 `className` prop을 컴포넌트 내부에서 그대로 접근할 수 있게 보존한다.
덕분에 `clsx`, `tailwind-variants` 같은 서드파티 라이브러리와 자연스럽게 연동된다.

## 주요 기능

### CSS 변수와 테마

CSS 커스텀 프로퍼티(custom property)를 지원하여
테마 정의, 인라인 스타일, 미디어 쿼리 모두에서 동적 값을 사용할 수 있다.
P3 색상 팔레트와 확장된 셰이드 범위도 포함한다.

### 다크 모드

`dark:` 변형자(modifier)로 다크/라이트 모드 스타일을 선언적으로 지정한다.

### 애니메이션과 트랜지션

`react-native-reanimated`와 연동한 애니메이션 지원.
Tailwind 클래스와 인라인 스타일 모두와 동적으로 작동하는 트랜지션 클래스.

### 컨테이너 쿼리

`@container` 문법으로 부모 컴포넌트 크기에 반응하는 모바일 반응형 설계가 가능하다.

### 플랫폼별 유틸리티

`ios:`, `android:` 접두사로 플랫폼 특화 스타일을 지정할 수 있다.

```jsx
<View className="mx-auto px-8 ios:pt-12 android:pt-8 dark:bg-gray-900">
  <Text className="text-lg font-bold dark:text-white">
    Hello NativeWind
  </Text>
</View>
```

## 생태계

NativewindUI, React Native Reusables, gluestack 등의 컴포넌트 킷과 파트너십을 맺고 있다.
프로덕티비티, 소셜, 피트니스, 게임, 금융 분야의 30개 이상 프로덕션 앱이 도입했다.

## 분석

### 웹↔네이티브 스타일링 간극을 메우는 전략

React Native 스타일링의 가장 큰 마찰 중 하나는
웹 CSS와 React Native StyleSheet API의 인터페이스 차이다.
웹 개발자가 React Native로 전환할 때 `margin: “auto”` 대신
`alignSelf: “center”`, `flex: 1` 같은 새로운 어휘를 배워야 한다.
NativeWind는 이 어휘 전환 비용을 Tailwind 클래스로 흡수한다.
`mx-auto`, `flex-1`은 웹과 네이티브 양쪽에서 같은 의미로 작동하는 것처럼 느껴진다.

실제로는 컴파일러가 각 플랫폼에 맞는 StyleSheet 객체로 변환하기 때문에
“같은 클래스”가 내부적으로 다른 값을 생성할 수 있다.
이 추상화 계층이 학습 비용을 낮추는 동시에
플랫폼별 차이를 은폐한다는 트레이드오프를 갖는다.

### v2 → v4의 아키텍처 전환이 갖는 의미

v2의 `styled()` 래퍼 방식은 NativeWind를 “컴포넌트 라이브러리”처럼 만들었다.
`className`을 지원하려면 반드시 `styled()`로 감싸야 했기 때문이다.
v4의 `jsxImportSource` 트랜스폼은 이 의존성을 끊어
NativeWind를 순수한 “스타일링 라이브러리”로 전환했다.
이 결정은 “프레임워크가 컴포넌트를 소유하지 않는다”는 원칙을 따른 것이며,
서드파티 컴포넌트 라이브러리와의 호환성 문제를 근본적으로 해결했다.

`className` prop 보존이 단순한 편의 기능이 아닌 이유는
헤드리스 UI, Radix, shadcn/ui 같은 생태계가 `className`을 핵심 인터페이스로
사용하기 때문이다. 웹의 컴포넌트 합성 패턴이 네이티브로 이식되려면
`className` 전달 체인이 깨지지 않아야 한다.

### Tailwind v4와의 정렬(v5)

v5가 Tailwind CSS v4를 따른다는 것은
CSS-first 설정(config JS 파일 대신 CSS 파일에서 직접 테마 설정)과
성능 향상된 새 컴파일러를 NativeWind도 수용한다는 의미다.
웹에서 Tailwind v4로의 전환이 가속화될수록
NativeWind v5의 필요성도 함께 높아진다.
그러나 v5가 아직 프리릴리스라는 점은
웹-네이티브 동기화에 항상 시차가 존재한다는 현실을 보여준다.

## 비평

### 강점

유틸리티 퍼스트 스타일링의 생산성 이점을 React Native에 그대로 가져오는 것은
크로스 플랫폼 팀에게 실질적인 가치다.
동일한 디자인 토큰과 클래스 어휘를 웹과 네이티브에서 공유하면
디자인 시스템 일관성 유지 비용이 줄어든다.
JSX 트랜스폼으로 기존 서드파티 컴포넌트 라이브러리와 자연스럽게 연동되는 v4의
아키텍처는 이전 버전의 가장 큰 고통점을 해결했다.

### 약점

React Native와 CSS의 스타일 모델은 근본적으로 다르다.
CSS의 캐스케이드(cascade), 상속, 전역 스타일 같은 개념이
React Native에는 존재하지 않는다.
NativeWind는 이 간극을 추상화로 덮지만,
개발자가 “이 클래스가 왜 예상대로 작동하지 않는가”를 디버깅할 때
추상화 아래 숨겨진 플랫폼 차이를 마주하게 된다.

v2 → v4 마이그레이션의 파괴적 변경 목록—`styled()` 제거, `rem` 기본값 변경,
일부 유틸리티 일시 제거—은 버전 간 마이그레이션 비용이 만만치 않음을 시사한다.
v5로의 전환도 `/migrate-from-v4` 문서가 존재한다는 점에서
추가적인 마이그레이션 부담이 예상된다.

## 인사이트

### “하나의 언어”가 가진 인지적 가치와 함정

NativeWind의 핵심 가치 제안은 기술적인 것이 아니라 인지적이다.
`mx-auto`, `text-lg`, `dark:bg-gray-900`은
웹에서도 네이티브에서도 같은 의미로 읽힌다.
개발자의 작업 기억(working memory)에서 “플랫폼별 스타일 API”를 제거하고
Tailwind 어휘 하나로 통일하는 것이다.
이 인지 부하 감소는 특히 웹 백그라운드를 가진 개발자가
React Native를 시작할 때 진입 장벽을 낮추는 데 효과적이다.

그러나 이 통일된 언어는 플랫폼 차이를 은폐한다.
웹에서 `overflow-hidden`은 CSS `overflow: hidden`이지만,
React Native에서는 `overflow`가 제한적으로 작동하며
`overflow-scroll`은 지원되지 않는다.
숙련된 개발자는 이 차이를 알지만, NativeWind를 통해 진입한 개발자는
“같은 클래스인데 왜 다르게 동작하는가”를 이해하기 위해
결국 플랫폼 내부를 들여다봐야 한다.
추상화는 학습 비용을 낮추지만 이해의 깊이도 함께 낮추는 경향이 있다.

### 크로스 플랫폼 개발의 수렴 압력

NativeWind의 등장은 더 넓은 구조적 압력을 반영한다.
React Native, Expo, Tamagui, Solito가 각각 다른 방식으로
웹과 네이티브의 간극을 줄이려 하고 있다.
이 압력의 원천은 팀 구성의 경제학이다.
웹 개발자와 네이티브 개발자를 분리 운용하는 것보다
하나의 팀이 양쪽을 개발하는 것이 더 효율적이라는 조직적 요구가
기술 선택을 이끈다.
NativeWind는 이 요구에 대한 스타일링 레이어에서의 응답이다.

이 수렴 압력이 강해질수록 “네이티브다움(nativeness)”의 가치가 희석될 위험도 있다.
iOS와 Android가 각각 정교하게 발전시켜온
플랫폼 고유의 상호작용 패턴, 애니메이션 물리, 접근성 처리가
“일관된 크로스 플랫폼 경험”이라는 이름 아래 평준화될 수 있다.
NativeWind의 플랫폼별 변형자(`ios:`, `android:`)가 존재하는 것은
이 딜레마에 대한 탈출구를 제공하지만, 그 탈출구를 얼마나 활용할지는
결국 팀의 철학에 달려 있다.

### 디자인 시스템의 단일 소스 문제

NativeWind가 가장 빛나는 시나리오는
웹과 앱이 동일한 디자인 토큰을 공유하는 조직이다.
`tailwind.config.js`(또는 v4의 CSS 파일)에 정의된 색상, 간격, 타이포그래피가
웹 컴포넌트와 네이티브 컴포넌트 양쪽에 동일하게 반영된다면,
디자인 시스템의 단일 소스(single source of truth)가 실현된다.

그러나 이 이상적인 시나리오에는 전제 조건이 있다.
웹과 앱의 시각 언어가 충분히 유사해야 한다는 것이다.
일부 제품은 웹과 앱에서 의도적으로 다른 경험을 설계한다.
앱에서는 더 큰 터치 타깃, 다른 내비게이션 패턴, 플랫폼 고유의 컨트롤을 사용한다.
이런 경우 “공유된 Tailwind 설정”이 실제로 공유되는 부분은
색상과 타이포그래피 정도에 그치고,
레이아웃과 컴포넌트 스타일은 여전히 플랫폼별로 달라진다.
NativeWind는 공유 가능한 부분을 최대화하지만,
공유 불가능한 부분의 복잡성까지 제거하지는 못한다.
