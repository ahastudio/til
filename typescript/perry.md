GitHub: <https://github.com/PerryTS/perry>

## 소개

Perry는 TypeScript 소스를 LLVM 기반으로 네이티브 실행 파일로 컴파일하는
도구다. macOS, iPadOS, iOS, Android, Linux, Windows, watchOS, tvOS,
WebAssembly, 그리고 Web까지 10개 플랫폼을 단일 코드베이스에서 타깃으로
삼는다. 랜딩 페이지는 “TypeScript → Native”라는 한 줄의 정체성과 함께,
런타임이 필요 없는 작은 바이너리(2~5MB)와 결정론적 빌드를 핵심 가치로
내세운다.

페이지가 제시하는 9가지 핵심 이점은 다음과 같다. 런타임 비의존성,
빠른 컴파일, 2~5MB 수준의 작은 바이너리, 결정론적 빌드, 풍부한 표준
라이브러리, 선택적 V8 런타임, 25개 이상의 네이티브 UI 위젯, 컴파일
타임 플러그인 시스템, 진정한 멀티스레딩, 컴파일 타임 국제화. 플랫폼별
네이티브 프레임워크 — macOS의 AppKit, iOS의 UIKit/SwiftUI, Linux의
GTK4, Windows의 Win32, 그리고 WebAssembly·JavaScript 타깃 — 위에서
직접 위젯을 호출한다는 점을 강조한다.

쇼케이스로는 Perry로 제작된 네 개의 실제 제품이 노출된다. MongoDB GUI인
Mango, 코드 에디터 Hone, 오디오 시각화 도구 dB Meter, JSON 뷰어 Pry.
배포 워크플로우는 컴파일·서명, 앱스토어 배포, 자동 UI 테스트(Geisterhand)
의 3단계로 정리된다. 비교표에서는 React Native, Flutter, Kotlin
Multiplatform 등과 “언어, 네이티브 코드 컴파일, 네이티브 위젯, 런타임
오버헤드” 4축에서 비교되며, “Node.js 대비 최대 18배 빠르다”, “Node의
약 80MB 대비 2~5MB 바이너리”라는 정량 주장이 함께 제시된다.

설치는 Homebrew, APT, winget으로 단순화되어 있다. 지원 기능 목록은
핵심 언어 문법, 함수, 클래스, 타입 시스템, 표준 라이브러리 모듈과
함께 30개 이상의 네이티브 npm 패키지 재구현을 포함한다. 컴파일
파이프라인은 TypeScript → SWC Parser → HIR Transform → LLVM Codegen
→ 2~5MB 실행 파일로 노출된다.

## 벤치마크

M1 Max에서의 측정 결과(v0.5.908):

| 벤치마크          | Perry | Rust | C++  | Go   | Node  | Bun  |
| ----------------- | ----: | ---: | ---: | ---: | ----: | ---: |
| fibonacci (ms)    |   309 |  316 |  309 |  446 |   987 |  518 |
| closure (ms)      |    50 |    — |    — |    — |   304 |   51 |
| binary_trees (ms) |     2 |    — |    — |    — |    10 |    7 |
| string_concat (ms)|     0 |    — |    — |    — |     3 |    1 |

fibonacci 기준으로 Rust, C++와 동일한 수준, Node.js 대비 약 3배 빠르다.
핵심 최적화는 세 가지다. 이스케이프하지 않는 객체를 힙 할당 없이 레지스터로
처리하는 Scalar Replacement, 순수 재귀 수치 함수에 대한 i64 특수화, 이스케이프
객체에 대한 Inline Bump Allocator.

## 명세

### 컴파일 파이프라인

페이지가 노출하는 네 단계 파이프라인은 현대적인 시스템 언어 빌드 흐름과
같다. SWC가 TypeScript를 파싱해 AST를 만들고, 자체 HIR(High-level IR)로
변환한 뒤, LLVM IR로 다시 내려 LLVM 코드 생성기를 통해 플랫폼 네이티브
바이너리를 만든다. SWC를 선택한 이유는 Rust 기반의 빠른 파서라는 점,
LLVM을 선택한 이유는 다중 플랫폼 백엔드를 그대로 활용할 수 있다는
점이다.

### 런타임 모델

기본 모드는 “런타임 없음”이다. JIT나 가비지 컬렉터에 의존하는 런타임
바이너리를 함께 배포하지 않으므로 시작 지연이 짧고 바이너리가 작다.
다만 “선택적 V8 런타임”을 제공해 동적 평가가 필요한 경우 V8을 임베드할
수 있다. 즉 두 가지 배포 모델 — 정적 컴파일 단독 / V8 임베드 — 을
사용자가 선택할 수 있다.

### UI 모델

플랫폼별 네이티브 위젯을 그대로 호출하는 모델이다. 안드로이드는
SwiftUI/Compose가 아니라 AppKit·UIKit·SwiftUI·GTK4·Win32 같은 OS 1차
UI 프레임워크에 직접 바인딩한다. 25개 이상의 네이티브 위젯이 TypeScript
API로 노출된다. Flutter처럼 자체 렌더링 엔진을 그리는 모델이 아니라
React Native에 더 가깝지만, JS 런타임을 거치지 않는다는 차이가 있다.

### 표준 라이브러리와 npm 호환성

표준 라이브러리에 더해 30+ 네이티브 npm 패키지 재구현을 포함한다.
즉 npm 생태계의 일부를 “Perry용으로 다시 작성된 구현”으로 흡수한다.
모든 npm 패키지가 동작하지 않으며, 호환 패키지 목록을 별도로 관리하는
형태로 보인다.

## 분석

### 포지셔닝 — “React Native와 Flutter의 사이”

비교표가 React Native, Flutter, Kotlin Multiplatform을 의식적으로 고른
것은 Perry가 “하이브리드 UI 프레임워크” 카테고리에서 경쟁한다는
선언이다. 그러나 Perry의 메커니즘은 그 어떤 비교 대상과도 다르다.
React Native는 JS 런타임을 두고 위젯을 네이티브로 그리며, Flutter는
자체 렌더링 엔진을 그린다. Kotlin Multiplatform은 JVM/Native 백엔드로
컴파일하지만 UI는 보통 플랫폼별로 따로 작성한다. Perry는 “언어는
TypeScript, 컴파일은 LLVM 네이티브, UI는 플랫폼 1차 위젯”의 조합이다.
이는 React Native의 “언어 친화성”과 Kotlin Multiplatform의 “네이티브
빌드”를 합치려는 시도다.

### TypeScript의 선택 — 시장과 언어 적합성의 충돌

TypeScript를 시스템 언어처럼 사용하려는 시도는 새롭지 않다.
AssemblyScript가 TypeScript-like 문법을 WebAssembly로 컴파일했고,
Static TypeScript(Microsoft MakeCode)가 마이크로컨트롤러용 정적 TS
변종을 만든 적이 있다. 두 사례는 모두 “TypeScript의 모든 의미를 보존하지
않는다”는 트레이드오프를 안았다. Perry가 “TypeScript의 어떤 부분을
지원하는지/포기하는지”는 명세 페이지를 별도로 봐야 하지만, 동적 평가,
프로토타입 체인의 런타임 변경, 임의 타이핑 같은 동적 의미는 컴파일
타임에 정적으로 결정할 수 없으므로 어딘가에서 제한이 들어갈 수밖에
없다.

### LLVM 백엔드의 의미

LLVM을 백엔드로 채택했다는 사실은 두 가지를 시사한다. 첫째,
10개 플랫폼 지원은 LLVM이 이미 지원하는 타깃에 얹은 것이므로 “플랫폼
하나당 백엔드 하나”의 비용을 부담하지 않는다. 둘째, AOT 컴파일의
이점(작은 바이너리, 빠른 시작, 결정론적 빌드)을 자연스럽게 얻는다.
Swift, Rust, Zig가 모두 LLVM을 백엔드로 쓴 이유와 같다. 단 LLVM은
무겁고 컴파일 시간이 길어질 수 있어, 페이지가 “빠른 컴파일”을 강조한
것이 실제로 어느 정도인지가 채택 결정의 핵심 변수다.

### npm 생태계와의 거리

“30+ 네이티브 npm 패키지 재구현”이라는 표현은 정직하다. 즉 npm
생태계와의 호환은 “복사”가 아니라 “선택적 재구현”이다. 이는
Bun이 npm 패키지를 그대로 실행하는 호환성 전략과 정반대다. Perry의
선택은 “네이티브 바이너리, 작은 크기”라는 1차 목표를 지키기 위한
트레이드오프이지만, 동시에 “Perry 생태계는 npm 생태계와 별개”라는
선언이기도 하다.

UI 라이브러리 전체를 정적 링크한다면 “런타임 없음”의 의미도 달라진다.[^evil_buzzard]
25개 네이티브 위젯 바인딩을 정적으로 포함한 바이너리가 2~5MB라는 주장은,
정적 링크된 UI 코드를 포함해도 이 크기가 유지된다는 의미인지, 아니면 위젯
바인딩은 동적 라이브러리로 처리하는지 불명확하다. macOS AppKit 바인딩을 정적
링크하면 그 자체로 수십 MB가 된다. 페이지의 크기 주장이 어떤 링크 전략 위에
있는지를 명시해야 검증 가능한 수치가 된다.

## 비평

### “런타임 없음”은 실제 프로젝트에서 성립하지 않는 주장이다

Perry의 핵심 마케팅 문구 중 하나인 “런타임 비의존성”은 실제 개발 현장에서
즉시 무너진다. Express 서버를 Perry로 만들려면 JS 런타임이 반드시 필요하다.
Perry가 재구현한 30개 npm 패키지 외에 다른 패키지를 쓰는 순간 선택적 V8
임베드 모드로 돌아가야 한다. HN 커뮤니티의 ventana는 이를 “가짜 플라스틱
과일”로 표현했다 — 멀리서 보면 그럴듯하지만 실제로 먹으려 하면 가짜라는
것을 안다.[^ventana]

페이지가 제시하는 선택적 V8 임베드가 있다는 사실 자체가 “런타임 없음”
주장을 자기 부정한다. 바이너리가 작은 이유는 V8을 배제해서인데, V8을
선택한 순간 Node.js와의 차별점이 사라진다. “런타임 없음”이 성립하는
범위는 Perry의 30개 재구현 패키지만으로 완결되는 앱에 한정된다. 쇼케이스
4개(Mango, Hone, dB Meter, Pry)가 이 범위에 들어간다고 가정하더라도,
일반 TypeScript 프로젝트를 Perry로 마이그레이션할 수 있다는 기대는 성립하지
않는다.

GC가 필요하다는 점도 이 주장을 흔든다.[^afavour] 가비지 컬렉션이 없는
시스템에서 TypeScript 객체의 생명주기를 관리하려면 어딘가에서 런타임 지원이
필요하다. Perry가 자체 GC를 구현하거나 힙 관리를 정적으로 처리하는 방식을
선택했다면 그 자체가 런타임이다. 페이지는 이 질문을 회피한다. “런타임 없음”은
“Node.js 런타임 없음”의 줄임말로 읽어야 한다. 그리고 그 마케팅적 단순화는
진입 기대치를 왜곡한다.

### TypeScript 의미론을 보존하면서 C 수준 성능을 얻는 것은 원리적으로 불가능하다

“Node.js 대비 최대 18배 빠르다”는 주장은 무엇을 측정한 수치인지 명시되지
않는다. 단순 시작 시간이라면 AOT 컴파일된 바이너리가 JIT 런타임보다 빠른 건
당연하다. 그러나 compute-intensive 알고리즘을 돌릴 때의 처리량 비교라면 상황이
다르다.

HN의 madanparas는 이를 구체적 수치로 보여줬다.[^madanparas] 이미지 컨볼루션
벤치마크에서 Perry는 Zig보다 1.86배 느리고, 12억 4천만 개의 불필요한 인스트럭션이
NaN-boxing 언박싱 오버헤드에서 나왔다. NaN-boxing은 동적 타입의 숫자 값을 64비트
더블 부동소수점의 남는 비트에 패킹하는 기법인데, TypeScript의 `number` 타입을
정적으로 추론하지 못할 때마다 이 비용이 발생한다. 결론은 명확하다. “TypeScript
의미론을 보존하면서 C 수준 성능을 얻는 것은 불가능하다.” 성능을 얻으려면
TypeScript 의미론의 어떤 부분을 포기해야 하고, 그것은 사실상 TypeScript가 아닌
다른 언어를 만드는 것이다.

tsonic의 저자 jeswin은 이 핵심을 짚었다.[^jeswin] 숫자 타입, 제네릭, 유틸리티
타입이 TS to Native 컴파일러의 가장 어려운 부분이다. Perry 랜딩 페이지는 이
부분에 대한 언급이 없다. “어떤 TypeScript가 성능 페널티 없이 네이티브로 가는가”
— 이 경계를 명시하지 않은 채 “18배 빠르다”를 내세우는 것은 측정 조건을 숨긴
마케팅이다.

### 프로젝트의 신뢰도는 코드와 주석, 두 곳 모두에서 의심받는다

HN 커뮤니티에서 반복적으로 제기된 우려는 프로젝트의 실질적 제작자와
거버넌스다.[^smasher164] 웹사이트 카피, 문서, 이슈 응답이 모두 AI 생성으로
보인다는 지적과 함께, GitHub에서 시간당 15개 안팎의 AI 커밋 패턴이 관측됐다는
보고가 있다.[^afavour] 이 관측이 맞다면 코드베이스의 실질적 이해자가 없다는
것을 의미한다. 컴파일러는 코드를 쓰는 것보다 디버깅하는 것이 훨씬 어렵다.
메모리 손상, GC 버그, 플랫폼별 ABI 차이 같은 문제는 AI가 생성한 커밋으로는
추적이 불가능하다.

거버넌스 정보가 페이지에 없다는 점도 이 우려를 강화한다. 라이선스, 후원
조직, 1.0 도달 여부, 호환성 정책 — 새 컴파일러를 프로덕션 앱에 도입하기 전에
반드시 확인해야 할 항목들이 모두 부재한다. “이걸로 앱을 만들었다가 프로젝트가
멈추면 코드베이스 전체가 잠긴다”는 위험을 정면으로 다루는 거버넌스 문서 없이는,
쇼케이스 4개 앱이 실제로 Perry로 만들어졌는지 여부조차 외부에서 검증하기 어렵다.

LLVM 백엔드 활용 능력도 마찬가지다. LLVM 최적화 패스를 올바르게 구성하고,
ABI를 맞추고, 플랫폼별 인트린식을 다루는 것은 수년의 컴파일러 경험이 필요한
작업이다. 이 작업이 AI 생성 커밋 위에서 이루어졌다면, “Perry로 만든 앱이
프로덕션 환경에서 메모리 손상 버그를 낼 때 누가 디버그하는가”라는 smasher164의
질문은 수사가 아닌 실질적 위험 평가다.

## 인사이트

### TypeScript의 “시스템 언어화”는 언어 진화의 자연스러운 단계다

TypeScript는 2012년 “JavaScript의 정적 타입 확장”으로 시작했다. 그러나
2026년의 TypeScript는 사실상 모든 새 자바스크립트 프로젝트의 기본
언어가 되었고, Vue·Svelte·Next.js·Bun 같은 도구의 1차 입력 형식이
되었다. 이 광범위한 채택이 “TypeScript를 그대로 시스템 영역으로
가져가자”라는 시도를 자연스럽게 만든다. 이미 Bun이 TypeScript를 1급
런타임 언어로 다루고, Deno가 TypeScript를 기본 입력으로 받고,
AssemblyScript가 WebAssembly 영역에서 동일 실험을 했다. Perry는 이
흐름의 데스크톱·모바일 데이터 포인트다.

언어 진화의 역사를 보면 이 패턴이 반복된다. C가 시스템 언어에서 응용
영역으로 확장한 흐름, Python이 스크립트에서 데이터 사이언스로 넘어간
흐름, Rust가 시스템 영역에서 웹 백엔드로 확장한 흐름이 모두 같다.
한 언어가 일정 사용자 임계점을 넘으면 그 언어를 “다른 영역에 가져가는”
시도가 자동으로 생긴다. 사용자는 학습 비용을 절약하고, 도구 제공자는
교차 영역 마켓을 얻기 때문이다.

여기서 두 번째 함의가 나온다. TypeScript의 다음 도전은 언어 자체의
기능이 아니라 “어디까지 따라갈 수 있는가”의 결정이다. 즉 TC39와
TypeScript 팀이 “네이티브 컴파일 친화적 부분집합”을 명시하느냐, 아니면
의미론을 보존하면서 컴파일러가 알아서 해석하느냐의 갈림길에 선다. 전자는
파편화(JavaScript 본진 vs Perry/AssemblyScript 변종)를, 후자는 컴파일러
복잡도를 만든다.

세 번째 함의는 인력 시장에 영향을 준다. TypeScript 개발자가 네이티브
앱, WebAssembly, 임베디드까지 만들 수 있다면 “프런트엔드 개발자 ↔
모바일/데스크톱 개발자”의 경계가 무너진다. React Native가 시작한 흐름이
Perry로 더 강해진다. 이는 [[ai-frontend-lost-decade]]에서 다룬 디스킬링
논의와 같은 구조 — 직무 경계의 분화/통합 — 의 다른 사례다.

### LLVM은 다중 플랫폼 언어의 사실상 운영체제다

Perry가 10개 플랫폼을 지원할 수 있는 이유는 자기 백엔드가 10개라서가
아니라 LLVM이 이미 그 10개를 지원하기 때문이다. Swift, Rust, Zig,
Mojo, MoonBit 같은 21세기 새 언어들이 모두 LLVM을 백엔드로 채택한
것은 우연이 아니다. LLVM은 “언어 개발자가 ISA·플랫폼 디테일을 신경 쓰지
않게 해주는 운영체제”에 가깝다. 즉 LLVM은 컴파일러 인프라가 아니라
새 언어 시장의 메타 플랫폼이다.

이 사실은 새 언어가 등장하는 비용을 극단적으로 낮춘다. 1990년대에는
새 언어를 만들려면 인터프리터, 가비지 컬렉터, 표준 라이브러리, 플랫폼
별 백엔드를 모두 처음부터 만들어야 했다. 2026년에는 SWC 같은 파서, LLVM
같은 백엔드, libc 같은 시스템 인터페이스가 모두 무료로 조립 가능하다.
새 언어를 만드는 비용이 “문법 설계 + 타입 시스템 + HIR 변환”으로 줄어
들었다. 결과적으로 새 언어가 폭발적으로 늘어난다.

여기서 도출되는 미시 경제 함의가 있다. 언어 시장의 차별화 축이 변한다.
20세기 언어 경쟁의 축은 “플랫폼 커버리지, 표준 라이브러리, 도구
체인”이었다. 21세기 언어 경쟁의 축은 “문법 디자인, 타입 시스템,
컴파일러 진단 품질, 에디터 통합”이다. 인프라가 평준화되면 차별화는
사용자 경험으로 이동한다. Rust의 컴파일러 진단 메시지가 압도적인 이유,
Zig의 단순성이 매력적인 이유는 모두 이 변화의 결과다.

세 번째 함의는 LLVM 자체의 거버넌스 위험이다. 모든 새 언어가 LLVM에
의존한다면, LLVM의 방향이 곧 언어 생태계의 방향이다. Mojo가 MLIR로
이동하고, Carbon이 LLVM 후속 작업을 자체 정의하는 흐름은 “LLVM 단일
의존”의 위험을 인지한 결과다. Perry 같은 신생 언어가 LLVM 의존을
어떻게 관리할지가 장기 생존 변수가 된다.

### 작은 바이너리는 다시 1급 가치로 부상한다

2010년대는 “바이너리 크기는 중요하지 않다”의 시대였다. Electron이
100MB 바이너리를 표준으로 만들었고, Node.js 80MB가 받아들여졌다.
SSD 용량은 무한처럼 보였고, 네트워크 대역폭은 점점 빨라졌다. 그러나
2026년에 작은 바이너리가 다시 1급 가치가 되는 흐름이 나타난다. Perry의
2~5MB 약속, Tauri의 채택 증가, Go의 정적 바이너리 인기, Rust의
크기 최적화 옵션의 일상화 — 모두 같은 방향이다.

이 회귀의 원인은 여러 층위로 작동한다. 첫째, 모바일과 임베디드에서
작은 바이너리는 여전히 결정 변수다. 둘째, 클라우드 함수(Lambda,
Cloudflare Workers)는 콜드 스타트 비용을 줄이려고 작은 바이너리를
선호한다. 셋째, CI/CD 파이프라인에서 컨테이너 이미지 크기가 빌드 시간과
배포 비용에 직결된다. 넷째, 사용자가 다시 “설치 안 한 채로 한 번
실행”을 선호하기 시작했다(npx, brew install, gh CLI). 작은 바이너리는
이 모든 영역에서 마찰을 줄인다.

여기서 도출되는 메타 원칙은 “인프라 비용이 충분히 작아 보일 때 기술
결정의 우선순위는 다른 차원으로 이동한다”이다. 메모리가 비쌌던 시절
바이너리 크기 최적화는 필수였고, 메모리가 싸지자 무시되었고, 다시
다른 차원(콜드 스타트, 컨테이너 빌드 시간)에서 재부상한다. 즉
“최적화 대상은 시대가 바꾼다”라는 패턴이 적용된다. 다음 10년의
1급 가치는 “전력 소비”와 “탄소 발자국”이 될 가능성이 높다. 작은
바이너리는 그 가치의 선행 지표다.

세 번째 함의는 Electron의 시대가 저무는 신호다. Electron의 100MB는
2015년에는 수용 가능했지만, 2026년 모바일·웹·에지 컴퓨팅의 분포에서는
부담이다. Tauri, Wails, Perry 같은 “네이티브 바이너리 + 웹 친화적
DX” 조합이 자리를 차지하기 시작한다. 이는 단순한 도구 교체가 아니라
“데스크톱 앱이 다시 OS 시민이 되는” 회귀다. Electron이 데스크톱을
브라우저화했다면, Tauri/Perry는 데스크톱을 다시 데스크톱화한다.

### 멀티 플랫폼 UI의 “자기 그리기 vs 위젯 빌리기”라는 영원한 선택

크로스플랫폼 UI 기술의 역사는 두 진영의 반복이다. Java AWT는 네이티브
위젯을 빌렸고, Java Swing은 자기 그리기로 갔다. Qt는 두 모델 사이를
오갔고, Flutter는 자기 그리기에, React Native는 위젯 빌리기에 베팅했다.
Perry는 React Native와 같은 진영(위젯 빌리기)이다. 이 선택은 영원한
트레이드오프 — 플랫폼 일관성 vs 코드 통일성 — 의 양면을 다시 활성화한다.

위젯 빌리기의 장점은 명확하다. 플랫폼 사용자가 이미 학습한 UX 규약을
그대로 사용하므로 접근성, 다크 모드, 시스템 통합이 “공짜”다. 단점도
명확하다. 플랫폼마다 다른 위젯 모델 때문에 “정확히 같은 화면”을 만들
수 없고, 디자이너의 1픽셀 통제권이 약해진다. 자기 그리기는 정반대 —
1픽셀 통제는 얻지만 플랫폼 통합은 잃는다.

여기서 도출되는 인사이트는 “어떤 UI를 만드는가”에 따라 선택이 달라진다는
점이다. 시스템에 가까운 도구(파일 관리자, 코드 에디터, IDE, 시스템
설정)는 위젯 빌리기가 강하다. Mango, Hone, Pry, dB Meter 같은 Perry의
쇼케이스 앱은 모두 이 카테고리다. 반면 게임 같은 1픽셀 통제가 핵심인
앱, 자체 브랜드 디자인을 강하게 요구하는 앱(Spotify, Slack)은 자기
그리기가 어울린다. Perry의 포지셔닝은 명시적으로 전자에 집중한다.

세 번째 함의는 AI 시대에 다시 갈리는 선택지를 만든다. AI가 UI 코드를
생성할 때 “플랫폼 위젯을 호출하는 코드”와 “캔버스에 1픽셀씩 그리는
코드”의 학습 데이터 양은 크게 다르다. 플랫폼 위젯 API는 한정된 어휘가
있고 잘 문서화되어 있어 AI가 안정적으로 생성한다. 자기 그리기는 무한한
디자인 공간이라 AI 생성이 일관되지 않다. 즉 AI 코딩 도구의 부상은 위젯
빌리기 진영을 우회적으로 강화한다. Perry 같은 도구가 이 흐름의 수혜자가
될 가능성이 있다.
