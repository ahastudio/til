# TypeScript 컴파일러 그래프 MCP가 Claude Code 토큰을 10배 줄인 방법

원문: <https://dev.to/samchon/i-made-ts-compiler-graph-mcp-10x-fewer-tokens-in-claude-code-1aea>

## 요약

Jeongho Nam(samchon)은 TypeScript 컴파일러가 생성한 코드 그래프를 MCP로 제공하는
`@ttsc/graph`를 공개하며, 기존 코드 그래프 도구와 비교한 토큰 사용량 벤치마크를 함께 제시한다.
8개 저장소 × 4개 모델의 중앙값으로 측정했을 때 MCP 없는 베이스라인 대비 약 10배 토큰 절감을
주장한다.

이 주장의 전제는 반직관적이다.
저자가 먼저 기존 도구들(codegraph, codebase-memory-mcp, serena)을 직접 써봤더니 도구 호출
횟수는 줄었지만 토큰 소비량은 오히려 22-93% 늘었다는 것이다.
“코드 그래프 MCP가 토큰을 줄인다”는 기존 도구들의 공통 약속이 실제로는 지켜지지 않았고,
그 원인을 진단한 뒤 설계 원칙을 달리하여 새 도구를 만들었다는 서사 구조다.

기존 도구의 실패는 세 가지 패턴으로 수렴한다.
첫째, codegraph는 심볼 정보와 함께 전체 소스 바디(body)를 반환해 토큰이 폭발한다.
둘째, codebase-memory-mcp는 관계 데이터를 14개 도구 뒤에 감춰 Cypher 쿼리를 써야만
그래프에 접근할 수 있다.
셋째, serena는 약 50개 도구와 150줄짜리 대체 시스템 프롬프트를 요구하며 LSP 콜드 스타트
지연도 동반한다.
공통점은 “인덱스를 제공한다고 말하지만 실제로는 바디를 돌려주거나, 그래프 접근 경로 자체를
과도하게 복잡하게 만들었다”는 것이다.

`@ttsc/graph`의 핵심 설계는 세 원칙으로 요약된다.
인덱스 전용(index only): 이름, 에지, 시그니처, 소스 위치 좌표만 반환하고 소스 바디는 절대
포함하지 않는다.
단일 도구 + CoT 컴플라이언스: 7개 요청 타입을 discriminated union으로 묶은 단일 MCP 도구를
제공하고, typed schema를 통해 `question → draft → review → request` 순서를 강제한다.
컴파일러 기반 해석: `tsconfig` 경로 별칭, 모노레포 교차 참조, 심볼링크, re-export 체인을
TypeScript 컴파일러(ttsc, TypeScript-Go 기반)가 완전히 해석한 결과를 사용한다.

설치는 `npm install -D ttsc @ttsc/graph typescript@rc` 한 줄이며, MCP 서버 항목을 설정
파일에 추가하면 된다.
TypeScript v7 RC가 필요하며 v6.x와는 호환되지 않는다.
브라우저 기반 3D 시각화(`npx @ttsc/graph view`)도 제공한다.

## 분석

### 도구 호출 수와 토큰 수를 혼동한 설계 실수가 낳은 역효과

기존 도구들은 “도구 호출 횟수 감소”를 성공 지표로 삼았다.
codegraph가 내세운 “파일 읽기 횟수 ~0”이나 codebase-memory-mcp의 “120배 토큰 절감”은 모두
이 프레임 안에서 나온 주장이다.
그러나 에이전트가 지불하는 실질적 비용은 도구 호출 횟수가 아니라 처리된 총 토큰 수다.
도구 호출을 한 번으로 줄이더라도 그 호출의 반환값이 전체 소스 파일을 포함한다면, 오히려
필요한 파일만 직접 읽는 것보다 더 많은 토큰이 소비된다.

이 역설은 codegraph에서 가장 명확하게 드러난다.
“파일 읽기를 대체한다”고 약속하면서도 소스 바디를 통째로 반환하기 때문에, 실제 파일 읽기와
토큰 비용이 거의 같거나 오히려 더 크다.
유일하게 줄어드는 것은 도구 호출 횟수인데, 그것은 에이전트가 실제로 지불하는 비용이 아니다.
저자가 Zod 저장소 실험에서 codegraph가 22-27% 토큰을 늘렸다고 보고한 것은 이 구조적 문제의
직접적인 결과다.

### 도구 표면(tool surface) 설계가 에이전트 행동을 결정한다

serena와 codebase-memory-mcp의 실패는 다른 층위에 있다.
이 두 도구는 소스 바디 문제 대신 과도하게 넓은 도구 표면을 가진다.
serena는 약 50개의 도구를 제공하고, 에이전트가 이를 제대로 사용하도록 150줄짜리 시스템
프롬프트를 주입한다.
codebase-memory-mcp는 14개 도구와 Cypher 쿼리 언어 사용을 요구한다.

도구 표면이 넓어질수록 에이전트는 어떤 도구를 선택할지 결정하는 데 추론 비용을 소비하고,
선택 오류 가능성도 높아진다.
시스템 프롬프트의 길이로 이 문제를 해결하려 하면 지시사항 토큰이 늘어나고, 지시사항과
행동 사이의 간극은 오히려 커질 수 있다.
저자가 codegraph의 “Read나 Grep을 먼저 사용하지 말라”는 지시와 serena의 150줄 대체
프롬프트를 실패 사례로 나란히 거론하는 것은 이 지점을 겨냥한 것이다.

`@ttsc/graph`가 단일 도구를 선택한 것은 이 인식에서 나온다.
7개의 요청 타입이 하나의 MCP 도구 안에 discriminated union으로 구현되므로, 에이전트는
도구 선택 대신 요청 타입 선택만 결정하면 된다.
더불어 `escape` 타입으로 그래프가 유용하지 않은 질문에 대한 명시적 탈출구를 제공하여,
도구를 부적절하게 강제 적용하는 상황을 차단한다.

### 프로즈 지시가 아닌 타입 스키마로 추론 순서를 강제한다

저자가 제시한 가장 독창적인 설계 결정은 CoT(Chain-of-Thought) 컴플라이언스를 typed schema로
구현한 것이다.
`question → draft → review → request` 순서는 MCP 도구의 입력 스키마에서 필수 필드로
정의된다.
에이전트는 이 필드들을 채우지 않고서는 요청을 완료할 수 없다.

“An instruction can be read and then ignored, but a required field on a typed schema cannot
be left blank.”
이 문장은 기존 시스템 프롬프트 기반 행동 유도와 스키마 기반 강제의 근본적 차이를 압축한다.
시스템 프롬프트의 지시는 LLM이 다음 토큰을 예측하는 과정에서 확률적으로 무시될 수 있지만,
스키마의 필수 필드는 도구 호출 자체가 거부되므로 우회가 불가능하다.
이것은 단순한 프롬프트 엔지니어링 기법이 아니라 인터페이스 설계 레이어의 변화다.

### 컴파일러 기반 해석이 인덱스 신뢰도를 결정한다

코드 그래프 도구의 핵심 약속은 “파일을 직접 읽지 않아도 코드 구조를 정확히 파악할 수 있다”는
것이다.
이 약속이 성립하려면 반환된 인덱스가 실제 코드 구조를 정확하게 반영해야 한다.
`tsconfig` 경로 별칭, 심볼링크, 모노레포 패키지 교차 참조, re-export 체인 같은 간접 참조
구조를 정적 분석만으로 처리하면 오류가 발생한다.

`@ttsc/graph`는 TypeScript 컴파일러가 이미 완전히 해석한 결과를 사용한다.
컴파일러는 타입 검사를 완료하기 위해 모든 모듈 해석과 타입 추론을 수행하는데, 이 과정의
부산물로 정확한 심볼 그래프가 생성된다.
tree-sitter처럼 파싱만 하는 도구는 이 정보를 갖지 못한다.
저자가 tree-sitter 버전은 약 3배 절감에 그칠 것이라고 추정하는 이유가 여기에 있다.
컴파일러 기반 해석이 없으면 인덱스의 신뢰도가 낮아지고, 에이전트가 인덱스를 신뢰하지 못하면
여전히 소스 파일을 직접 읽으러 돌아간다.

## 비평

### 「TypeScript 전용」이 결함이 아니라는 주장은 시장 조건을 생략한다

저자는 “158개 언어를 얕게 다루는 것보다 하나의 언어를 컴파일러 깊이로 다루는 것이 에이전트에게
더 유용하다”고 주장한다.
이 주장 자체는 타당한 기술적 트레이드오프 서술이다.
그러나 이 서술이 실질적 결함을 중립적인 설계 선택으로 재프레이밍한다는 점에서 정직하지 않다.

실무 프로젝트는 TypeScript 단일 언어로 구성되지 않는 경우가 많다.
인프라 스크립트(Python, bash), 빌드 설정(YAML, Dockerfile), 데이터 처리 파이프라인(Python),
모바일 코드(Swift, Kotlin)가 하나의 저장소에 공존한다.
이 상황에서 TypeScript 심볼만 추적하는 도구는 교차 언어 참조가 발생하는 경계에서 그래프가
끊긴다.
저자도 “TypeScript 외 질문은 escape로 빠져나온다”고 인정하는데, 이것은 곧 다수의 실무 질문에서
`@ttsc/graph`가 직접 답변할 수 없다는 의미다.

“컴파일러 깊이가 쉘로우 브레드스보다 낫다”는 주장은 사용자가 TypeScript 중심 저장소를 사용한다는
가정 위에서만 성립한다.
이 가정은 명시되어야 하며, 그것이 전제가 아닌 한계 조건으로 제시되어야 한다.

### 자가 측정 벤치마크는 10배 주장의 근거로 충분하지 않다

저자는 “저자 본인의 저장소에서 저자 본인의 질문 세트로 측정했다”는 사실을 직접 고백한다.
이 투명성은 신뢰를 높이는 긍정적 신호다.
그러나 이 고백이 벤치마크의 구조적 편향을 해소하지는 못한다.

저자의 저장소는 typia, nestia 같은 TypeScript-first 라이브러리들이다.
이 저장소들은 TypeScript 컴파일러 그래프 접근으로 가장 많은 이득을 얻을 수 있는 구조다.
대규모 모노레포, 다중 언어 혼합 저장소, 복잡한 JavaScript 레거시 코드베이스, 동적 require/import
패턴이 많은 저장소에서 동일한 성능이 나올 것이라는 보장은 없다.
저자의 질문 세트 역시 저자가 만든 도구가 잘 답변할 수 있는 질문들로 자연스럽게 편향될 가능성이
높다.

“10배”라는 수치는 강력하지만, 동일 벤치마크를 외부 저장소와 외부 질문 세트에 적용한 독립적
검증이 없다면 마케팅 수치와 구별하기 어렵다.
저자가 실제로 요청하는 것처럼 외부 사용자의 결과 보고를 바탕으로 이 수치가 교정될 필요가 있다.

### 스키마 강제 CoT가 추론 품질을 높인다는 가정은 검증되지 않았다

`question → draft → review → request` 필드를 typed schema로 강제하면 에이전트가 추론 단계를
생략할 수 없다는 주장은 구조적으로 맞다.
그러나 필드를 채운다는 것과 필드에 담긴 추론이 품질 있다는 것은 다른 문제다.

에이전트는 형식적으로 필드를 채우면서도 `draft`에 “심볼을 찾아봐야 할 것 같다”처럼 의미 없는
내용을 쓸 수 있다.
`review`에 “계획이 적절해 보인다”처럼 실질적 검토 없이 진행할 수도 있다.
스키마는 필드의 존재를 강제하지만 필드의 내용 품질을 보장하지 않는다.

저자는 “사용해보니 실제로 더 잘 작동했다”고 주장하지만, CoT 강제가 성능 향상에 기여한
정도를 인덱스 전용 전략이나 컴파일러 기반 해석과 분리하여 측정한 근거는 제시하지 않는다.
세 설계 원칙이 묶음으로 동작하기 때문에 각 원칙의 기여를 구분하는 제거 실험(ablation study)이
없으면, CoT 강제의 효과를 독립적으로 평가할 수 없다.

## 인사이트

### 타입 스키마를 인터페이스 계약으로 사용하는 AI 도구 설계가 부상한다

`@ttsc/graph`의 CoT 강제 설계는 AI 도구 인터페이스에서 진행 중인 더 큰 전환의 일부다.
초기 AI 도구 설계는 “무엇을 해야 하는지” 텍스트로 설명하는 프롬프트 엔지니어링 방식에 의존했다.
이 방식의 근본 문제는 자연어 지시가 확률적 시스템에 전달되어 불확실한 결과를 낳는다는 것이다.

타입 스키마를 인터페이스로 쓰는 접근은 이 불확실성의 일부를 제거한다.
필수 필드는 채워야 하고, discriminated union은 허용된 상태 공간을 제한한다.
이것은 소프트웨어 공학에서 “make illegal states unrepresentable”이라는 원칙을 AI 도구 설계에
적용한 것이다.
LLM이 텍스트 생성 모델이므로 타입 시스템의 보호를 완전히 누릴 수는 없지만, tool-use API가
JSON Schema 검증을 수행하는 레이어에서는 부분적으로 유효하다.

이 방향이 더 확산되면 MCP 도구 설계에서 schema engineering이 prompt engineering만큼 중요한
역할을 하게 될 것이다.
특히 추론 단계를 명시적으로 분리해야 하는 복잡한 도구일수록, 프로즈 지시보다 스키마 구조로
행동 순서를 제약하는 접근이 더 신뢰할 수 있는 결과를 낸다.
이것은 단순한 프롬프트 기법의 개선이 아니라 인터페이스 설계 패러다임의 변화다.

### 컨텍스트 창 경제학이 MCP 인터페이스 설계를 규율해야 한다

`@ttsc/graph`가 제기하는 더 깊은 문제는 “MCP 도구는 어떤 것을 반환해야 하는가”다.
기존 도구들은 “에이전트에게 더 많은 정보를 주면 더 잘 답변한다”는 직관에서 출발해 소스 바디를
포함했다.
이 직관은 인간 개발자의 경험에서 나온 것이다.
인간에게 더 많은 코드를 보여주면 더 잘 이해하는 것처럼 보인다.

그러나 LLM 에이전트에게는 이 직관이 역방향으로 작동한다.
컨텍스트 창에 올라간 모든 토큰은 처리 비용과 지연을 발생시키며, 무관한 정보가 많을수록
관련 정보에 대한 집중도가 낮아질 수 있다.
“에이전트에게 충분히 작고 정확한 컨텍스트를 주는 것”이 “가능한 많은 정보를 주는 것”보다
실질적 성능이 높다는 역설이 성립할 수 있다.

이 인식은 MCP 도구 설계의 일반 원칙으로 확장된다.
도구가 반환해야 할 것은 에이전트가 다음 결정을 내리기에 충분한 최소한의 구조화된 정보다.
전체 소스 바디는 에이전트가 직접 파일을 읽으면 얻을 수 있으므로, MCP 도구가 굳이 포함할
이유가 없다.
반면 컴파일러만이 생성할 수 있는 교차 모듈 심볼 그래프는 파일 읽기로 대체할 수 없는 가치다.
“도구가 제공해야 할 것은 에이전트가 스스로 할 수 없는 것뿐이다”라는 원칙이 도출된다.

### 코드 그래프 도구의 실패가 드러내는 에이전트 성능 측정의 함정

기존 도구들이 “도구 호출 횟수”를 성과 지표로 삼은 것은 측정 가능성의 함정이다.
도구 호출 횟수는 쉽게 측정되고 가시적으로 줄일 수 있지만, 실제 작업 비용과 직접 연결되지
않는다.
반면 토큰 수는 비용과 직접 연결되지만 도구 호출 횟수만큼 마케팅에 활용하기 쉽지 않다.

이 현상은 Goodhart의 법칙의 변종이다.
“지표가 목표가 되는 순간 좋은 지표이기를 멈춘다.”
도구 호출 횟수 최소화가 목표가 되자, 그것을 최소화하는 방향으로 설계가 최적화되었고
실제 비용 지표인 토큰은 악화되었다.
에이전트 도구 성능을 평가할 때 측정 지표의 선택이 최적화 방향을 결정한다는 교훈이다.

이 교훈은 AI 코딩 도구 전반으로 확장된다.
인간이 직관적으로 이해하기 쉬운 지표(클릭 수, 응답 시간, 도구 호출 횟수)가 실제 에이전트
성능과 괴리될 때, 도구는 보이기 좋은 방향으로 최적화되면서 실질적 가치를 잃는다.
`@ttsc/graph`가 실제 토큰 비용을 주요 지표로 삼기로 결정한 것은, 이 함정을 의식한 결과다.
그러나 저자 자신의 벤치마크도 “총 토큰 수”가 “작업 품질 대비 토큰 효율”과 같지 않다는
한 단계 더 깊은 함정을 여전히 안고 있다.
