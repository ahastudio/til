# TypeScript 7.0 정식 출시: Go로 다시 쓴 네이티브 컴파일러

원문: <https://devblogs.microsoft.com/typescript/announcing-typescript-7-0/>

HN 토론: <https://news.ycombinator.com/item?id=48833715> (517점, 205개 댓글)

## 요약

Microsoft가 2026년 7월 8일 TypeScript 7.0을 정식 발표했다.
핵심은 JavaScript로 작성됐던 컴파일러를 Go로 완전히 다시
작성한 네이티브 포트로, 팀은 이를 “TypeScript 툴셋의 네이티브
시대”라 부르며 “10배 빠른 네이티브 포트”라고 소개한다.

성능 개선 폭은 실제 프로젝트 기준으로 다음과 같다. VS Code는
125.7초에서 10.6초로(11.9배), Sentry는 139.8초에서 15.7초로
(8.9배), Bluesky는 24.3초에서 2.8초로(8.7배), Playwright는
12.8초에서 1.47초로(8.7배) 단축됐다. 메모리 사용량도 6~26%
줄었고, 에디터 경험 역시 크게 개선되어 VS Code에서 오류가
있는 파일을 여는 시간이 17.5초에서 1.3초 이내로 줄었다. 팀은
기존 TypeScript의 로직을 “최대한 충실하게” Go로 옮겨 구조적
일관성을 유지하면서, 네이티브 컴파일 속도와 공유 메모리 기반
멀티스레딩, 전체 빌드 기준 8~12배의 추가 최적화를 확보했다고
설명한다.

7.0은 6.0에서 도입된 새 기본값을 그대로 채택한다.
`strict: true`, `module`은 `esnext`, `types`는 빈 배열(이전에는
모든 `@types` 패키지를 자동 로드)이 기본이며, `rootDir`은 구조
추론 대신 `./`을 기본값으로 삼는다. ES5 타겟, `downlevelIteration`,
AMD/UMD 모듈 시스템 등 오래된 기능은 더 이상 지원되지 않는다.
마이그레이션을 위해서는 `rootDir`과 `types`를 명시적으로 설정해야
한다.

새 기능으로는 병렬 타입 체크와 프로젝트 참조 빌드를 조정하는
`--checkers`(기본값 4)와 `--builders` 플래그, 디버깅을 위해
병렬화를 모두 끄는 `--singleThreaded` 플래그가 추가됐다. 워치
모드는 Parcel 번들러의 기술을 가져와 완전히 새로 구현해, 크로스
플랫폼 파일 감시를 훨씬 적은 리소스로 처리한다. 템플릿 리터럴
타입은 이제 유니코드 코드 포인트를 올바르게 보존해, 이모지를
서로게이트 쌍이 아니라 단일 단위로 처리한다.

VS Code 사용자는 전용 TypeScript 7 확장을 바로 설치할 수 있고,
Visual Studio는 자동으로 활성화된다. LSP(Language Server
Protocol)를 지원하는 대부분의 최신 에디터도 자동으로 동작한다.
다만 Vue, Svelte, Angular, MDX, Astro 같은 임베디드 언어
워크플로는 아직 안정적인 프로그래매틱 API가 없어 TypeScript 7을
쓸 수 없고, 이런 시나리오의 에디터 지원을 위해서는 당분간
TypeScript 6.0을 계속 써야 한다. `@typescript/typescript6`
호환성 패키지를 쓰면 6.0과 7.0을 동시에 실행할 수 있는데, 이는
컴파일러 API에 의존하는 typescript-eslint 같은 도구가 TypeScript
7.1에서 새 API가 나올 때까지 버틸 수 있게 해주는 중요한 장치다.

7.0은 Slack, Vanta, Microsoft 내부 팀(Loop, PowerBI, Teams),
Figma, Linear, Notion 등 주요 기업에서 광범위하게 테스트됐다.
Slack은 CI 타입 체크 시간을 7.5분에서 1.25분으로 줄였고,
Microsoft의 News Services 팀은 빌드 대기 시간을 월 400시간
절감했다고 밝혔다.

## 분석

### “충실한 포팅”이라는 전략이 재작성 리스크를 최소화한다

대규모 컴파일러를 다른 언어로 다시 쓰는 프로젝트는 흔히 이
과정에서 새로운 설계를 시도하려는 유혹에 빠지고, 그 결과
기존 동작과의 미묘한 차이(behavioral drift)가 누적되어 생태계
전체가 깨지는 위험을 안는다. TypeScript 팀이 “기존 로직을
최대한 충실하게” 옮기는 전략을 택한 것은 이런 위험을 의도적으로
회피한 선택이다. 새로운 언어가 주는 성능 이점(네이티브 컴파일,
공유 메모리 멀티스레딩)은 그대로 취하되, 타입 체크 알고리즘
자체의 동작은 바꾸지 않음으로써 수백만 개의 기존 TypeScript
프로젝트가 재작성 이후에도 동일한 타입 오류를 동일하게
보고받도록 보장한다.
HN에서 dimitropoulos는 이 프로젝트의 진짜 이야기는 "인류가 아는
가장 발전된 타입 시스템"을 위해 두 개의 코드베이스를 동시에
유지해 온 팀의 역량에 있다고 평가하며, 왜 Rust 대신 Go를
선택했는지에 대한 논의를 짚었다. 팀이 완전한 재설계가 아니라
"버그까지 포함해 파일 단위로 그대로 옮기는" 번역을 우선시했고,
Rust의 borrow checker가 컴파일러 특유의 순환 데이터 구조에
구조적 변경을 강제했을 수 있다는 점이 Go를 택한 배경으로
제시됐다[^dimitropoulos].

### 6.0을 “브릿지 릴리스”로 설계한 것은 생태계 전환 비용을 미리 분산시킨 결정이다

6.0이 이미 7.0과 동일한 기본값(`strict: true`, `module: esnext`,
`types: []`)과 `--stableTypeOrdering` 같은 호환 플래그를
도입해 뒀다는 점은, 이번 7.0 출시가 갑작스러운 단절이 아니라
오랫동안 준비된 계획의 마지막 단계였음을 보여준다. 생태계
전체가 한 번에 새 기본값과 새 컴파일러를 동시에 받아들여야
했다면 마이그레이션 비용이 훨씬 컸을 것이다. 기본값 변경(6.0)과
런타임 교체(7.0)를 시간차를 두고 분리함으로써, 각 단계에서
발생하는 문제를 독립적으로 진단하고 대응할 수 있게 만들었다.

### 프로덕션 성능 수치를 전면에 내세우는 것은 신뢰 확보 전략이다

VS Code, Slack, Bluesky 같은 실제 대규모 프로젝트의 구체적인
초 단위 수치를 제시한 것은 단순한 벤치마크 그래프보다 훨씬
설득력이 크다. 컴파일러 재작성처럼 리스크가 큰 변경일수록,
추상적인 “N배 빠르다”는 주장보다 이름이 알려진 실제 프로젝트가
이미 프로덕션에서 검증했다는 사실이 채택을 망설이는 팀들에게
더 강한 신호로 작동한다. 이는 대규모 인프라 교체를 발표할 때
반복되는 수사적 전략이다.

## 비평

### “임베디드 언어 워크플로 미지원”은 사소한 예외가 아니라 실질적인 채택 장벽이다

발표 글은 Vue, Svelte, Angular, MDX, Astro에서 TypeScript 7을
아직 쓸 수 없다는 사실을 부수적인 각주처럼 다루지만, 이 프레임워크들은
현재 프런트엔드 생태계에서 결코 주변적이지 않다. 이들을 쓰는
팀은 에디터에서 TypeScript 6.0과 7.0을 동시에 운영해야 하고,
이는 두 버전의 설정을 병행 관리해야 하는 실질적인 부담으로
이어진다. “곧 지원될 것”이라는 로드맵 언급 없이 이 제약을
가볍게 처리한 것은, 해당 프레임워크 사용자층의 실제 전환
경험이 얼마나 매끄러울지에 대한 근거를 충분히 제공하지 못한다.

### 컴파일러 API 의존 도구 생태계의 전환 비용이 과소평가됐다

`@typescript/typescript6` 호환 패키지로 typescript-eslint 같은
도구가 당분간 6.0의 API에 의존한 채 버틸 수 있다는 설명은,
거꾸로 말하면 컴파일러 API에 의존하는 모든 서드파티 도구가
7.1까지는 새 네이티브 컴파일러의 성능 이점을 전혀 누리지 못한다는
뜻이다. 린터, IDE 플러그인, 코드 생성 도구처럼 컴파일러 내부
API를 직접 호출하는 생태계 전체가 사실상 과도기 상태에 놓이게
되는데, 발표 글은 이 전환이 얼마나 오래 걸릴지, 7.1의 새 API가
기존 API와 얼마나 호환될지에 대한 구체적인 일정이나 설계
방향을 제시하지 않는다.
HN에서 afdbcreid는 컴파일러 API 자체가 아직 빠져 있다는 점을
가장 큰 채택 걸림돌로 짚으며, 팀이 이를 7.1로 미루고 있어
프로그래매틱 컴파일러 접근에 의존하는 도구들에 실질적인 마찰을
일으킨다고 지적했다[^afdbcreid]. jakebailey는 WebAssembly 빌드
방향이 아직 불명확하다는 점도 짚었는데, LSP·Monaco 지원, 브라우저
API, 미지원 플랫폼용 CLI 중 무엇을 Wasm으로 제공할지 팀 내에서도
우선순위가 정리되지 않은 상태라고 밝혔다[^jakebailey].
herpdyderp는 TypeScript 자체의 개선과 무관하게 ESLint 패키지 간
버전 불일치가 별도의 업그레이드 장벽으로 남아 있어, 더 넓은
툴체인 조율이 여전히 필요하다고 지적했다[^herpdyderp].

### 벤치마크로 제시된 프로젝트들이 전형적인 워크로드를 대표하는지 불분명하다

VS Code, Sentry, Bluesky, Playwright는 모두 대규모이고 활발히
관리되는 오픈소스 프로젝트다. 이런 프로젝트들은 이미 타입 구조가
비교적 정돈되어 있고 최신 TypeScript 관행을 따를 가능성이 높다.
반면 레거시 코드가 누적된 중소규모 사내 프로젝트, 특히 `types`
기본값 변경이나 `rootDir` 추론 방식 변경에 영향을 받는 프로젝트가
동일한 배수의 성능 향상을 얻을지, 아니면 마이그레이션 과정에서
예상치 못한 타입 오류가 무더기로 드러나 초기에는 오히려 생산성이
저하될지는 이 벤치마크만으로 판단할 수 없다.

## 인사이트

### 컴파일러 재작성의 성공 여부는 결국 하위 호환성 정책에 달려 있다

TypeScript 7.0의 실질적 성패는 Go 네이티브 포트의 성능 수치가
아니라, 기존 생태계(6.0에 의존하는 도구, 임베디드 언어 플러그인,
컴파일러 API 소비자)가 얼마나 매끄럽게 전환하는지에 달려 있다.
이는 Python 2에서 3으로의 전환, 혹은 여러 대규모 언어 런타임
교체 사례에서 반복적으로 확인된 패턴이다. 새 런타임 자체의
기술적 우수성은 전환의 필요조건일 뿐 충분조건이 아니며, 실제
채택 속도는 하위 호환 계층의 완성도와 유지 기간에 의해 결정된다.
TypeScript 팀이 6.0을 브릿지로 설계하고 `@typescript/typescript6`
호환 패키지를 제공한 것은 이 교훈을 이미 반영한 결과로 보이지만,
7.1의 새 API가 실제로 나오기 전까지는 이 하위 호환 전략의
성공 여부를 판단할 수 없다.

### “10배 빠른 컴파일러”는 개발자 워크플로의 병목을 다른 곳으로 옮길 뿐이다

타입 체크가 10배 빨라지면, 그동안 컴파일 대기 시간에 가려져
있던 다른 병목(번들링, 테스트 실행, CI 파이프라인의 다른 단계)이
상대적으로 더 크게 부각된다. Slack이 CI 타입 체크를 7.5분에서
1.25분으로 줄였다는 사례는 인상적이지만, 전체 CI 파이프라인의
총 소요 시간이 동일한 비율로 줄었는지는 언급되지 않는다. 이는
성능 개선이 국소적으로 극적일수록, 그 개선이 실제 체감 생산성
향상으로 이어지려면 주변 시스템(빌드 도구, 배포 파이프라인)도
함께 병목을 해소해야 한다는 일반적인 시스템 최적화의 원리를
보여준다. 컴파일러가 더 이상 병목이 아니게 된 조직에서는, 다음
최적화 대상이 무엇인지에 대한 새로운 질문이 제기될 것이다.
HN에서 dabinat은 흥미로운 관찰을 남겼는데, 실무에서는 tsc가
꾸준히 문제를 일으켜 왔음에도 TypeScript의 성능 불만이 Rust
컴파일러의 느림에 대한 불만보다 오히려 덜 두드러지게
제기돼 왔다는 점이다[^dabinat]. dymk는 여기서 한 걸음 더
나아가 제번스 역설(Jevons Paradox)을 언급하며, 컴파일 속도가
개선될수록 오히려 더 복잡한 타입 패턴(고차 타입 등)을 마음
놓고 쓰게 되어 그 효율 개선분을 다시 소진해버릴 수 있다고
지적했다[^dymk].

### 언어 툴체인의 “네이티브화”는 개발자 도구 전반의 장기 추세를 다시 확인시킨다

TypeScript의 Go 포팅은 최근 몇 년간 반복되어 온 흐름, 즉 스크립팅
언어로 작성된 개발자 도구를 시스템 프로그래밍 언어로 다시 작성해
극적인 성능 향상을 얻는 패턴의 연장선에 있다. Babel 대신 esbuild와
SWC가 등장하고, Webpack 대신 Rspack과 Turbopack이 등장한 것과
같은 궤적이다. 이런 흐름이 시사하는 것은, 개발자 도구 생태계가
이제 “기능의 완성도”를 넘어 “실행 속도”를 핵심 경쟁 축으로
삼는 단계에 들어섰다는 점이다. 다만 이 흐름에는 대가도 따른다.
도구의 핵심부가 JavaScript/TypeScript가 아닌 언어로 옮겨가면서,
해당 생태계에 기여하려는 개발자는 이제 Go나 Rust까지 이해해야
하는 진입 장벽이 새로 생긴다. 편의성과 접근성이라는 오픈소스
