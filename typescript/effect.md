# Effect

<https://effect.website/>

HN 토론: <https://news.ycombinator.com/item?id=40682149> (127점, 74개 댓글)

## 소개

Effect는 “TypeScript로 견고한 앱을 만드는 최선의 방법(The best way to build robust apps in TypeScript)”을 표방하는 TypeScript 표준 라이브러리이자 런타임 시스템이다.
2022 State of JavaScript 조사에서 개발자들이 꼽은 가장 큰 언어 결함—표준 라이브러리 부재, 불변 데이터 구조, 옵저버블, 파이프 연산자, 패턴 매칭—을 한 번에 다루는 것이 목표다.

핵심 런타임은 압축 및 트리셰이킹 후 약 15KB다.
오류를 타입 값으로 추적하고, 지수 백오프 재시도를 제공하고, 타임아웃과 자원 정리를 처리하고,
OpenTelemetry 통합 추적과 메트릭을 내장한다.
같은 비동기 로직을 비교하면 오류 처리 취약한 19줄 코드가 타입 안전 6줄로 압축된다고 소개한다.

Node.js, Deno, Bun, Cloudflare Workers, 브라우저를 모두 지원하며,
React, Solid.js, Vite, Next.js, Tauri와 통합된다.
핵심 라이브러리 외에 불변 데이터 구조, 비동기 큐, pub-sub, 설정 관리, 의존성 관리,
데이터 유효성 검사/직렬화, CLI 프레임워크, HTTP 프레임워크가 생태계를 구성한다.
GitHub 기여자 305명 이상, Discord 커뮤니티 2,000명 이상이다.

## 분석

### TypeScript의 표준 라이브러리 공백을 함수형 효과 시스템으로 채우는 선택

JavaScript는 역사적으로 언어 수준 표준 라이브러리 없이 npm 생태계로 성장했다.
date-fns, lodash, zod, axios 같은 라이브러리들이 각 영역을 분담해 채웠다.
이 방식은 유연하지만 조합의 일관성이 없다.
오류 처리는 프로젝트마다 다르고, 재시도 로직은 팀마다 다시 구현하고,
비동기 자원 정리는 자주 누락된다.

Effect는 이 파편화에 대한 하나의 답이다.
단, npm 생태계 방식(작은 패키지들의 조합)이 아니라
Haskell의 IO 모나드, Scala의 ZIO에서 유래한 함수형 효과 시스템 방식을 선택한다.
오류, 비동기성, 의존성, 자원 관리를 단일 타입 파라미터 체계로 표현하면
컴파일러가 이 모든 관심사를 추적할 수 있다.
이것이 “타입 안전한 오류 처리”가 아니라 “오류 추적을 타입 시스템에 내장”이라고 표현하는 이유다.

### 오류를 Control Flow가 아닌 값으로 다루는 패러다임

JavaScript의 `try/catch`는 오류를 제어 흐름으로 처리한다.
이 방식에서 함수의 시그니처는 어떤 오류를 던질 수 있는지 표현하지 못한다.
호출자는 문서를 읽거나 소스를 분석하거나 런타임에 직접 만나야만 오류 종류를 알 수 있다.
TypeScript의 타입 시스템이 오류를 볼 수 없다.

Effect의 타입은 `Effect<Success, Error, Requirements>` 형태다.
성공 타입, 오류 타입, 필요한 의존성 타입이 모두 함수 시그니처에 드러난다.
오류를 처리하지 않으면 컴파일이 되지 않는 것이 아니라
오류 타입이 전파되어 호출 체인 전체에서 누가 이 오류를 처리하는지가 타입 수준에서 추적된다.
Rust의 `Result<T, E>`, Haskell의 `Either a b`와 같은 패턴을 TypeScript의 타입 추론과 결합한 것이다.

### OpenTelemetry 내장이 말하는 관측성 우선 설계

오픈소스 라이브러리가 OpenTelemetry 추적을 핵심 기능으로 내장하는 것은 드문 선택이다.
대부분의 라이브러리는 관측성을 사용자가 나중에 추가하는 것으로 취급한다.
Effect가 이를 내장하는 것은 “프로덕션에서 무슨 일이 일어나는지 볼 수 없으면 견고한 앱이 아니다”는 판단이다.

이 설계 선택은 Effect의 타겟 사용자가 누구인지를 드러낸다.
토이 프로젝트나 소규모 스크립트가 아니라
추적이 필요하고 재시도 로직이 필요하고 자원 정리가 필요한 프로덕션 서비스다.
라이브러리가 처음부터 프로덕션 복잡성을 가정하고 설계된다는 것은
초기 학습 비용이 올라가는 대신 프로덕션 전환 시 재설계 비용이 낮아진다는 트레이드오프다.

### 의존성 주입을 타입 시스템으로 표현하는 접근

Effect의 세 번째 타입 파라미터 `Requirements`는 이 Effect를 실행하기 위해 필요한 서비스(의존성)를 표현한다.
데이터베이스 연결, 설정 객체, HTTP 클라이언트 같은 의존성을 타입으로 선언하면
컴파일러가 어떤 서비스가 제공됐고 어떤 서비스가 누락됐는지를 추적한다.
이것은 TypeScript 데코레이터 기반 DI(tsyringe, InversifyJS 등)와 다른 접근이다.
데코레이터 DI는 런타임에 오류가 드러나지만, Effect의 DI는 컴파일 타임에 드러난다.

이 방식이 React, Solid.js 통합을 가능하게 하는 이유이기도 하다.
프론트엔드 컴포넌트도 서버 의존성과 같은 방식으로 의존성을 선언하고 테스트에서 대체할 수 있다.
“재사용 가능하고 테스트 가능한 컴포넌트”라는 약속의 기술적 기반이 여기 있다.

## 비평

### “최선의 방법”이라는 태그라인이 학습 비용을 숨긴다

“The best way to build robust apps in TypeScript”는 강한 주장이다.
Effect의 접근이 견고한 앱을 만드는 데 실질적으로 유리하다는 것은 논거가 있다.
그러나 이 주장에는 전제가 있다.
팀이 함수형 효과 시스템의 패러다임을 이해하고 있어야 한다는 것이다.

Effect의 개념 체계는 Haskell과 Scala의 함수형 프로그래밍 전통에서 온다.
Monad, Fiber, Layer, Scope, Effect 타입의 세 파라미터를 이해하는 데
TypeScript 개발자가 투자해야 하는 학습 시간은 상당하다.
“19줄을 6줄로 줄인다”는 비교는 Effect를 이미 아는 사람이 작성한 결과다.
처음 배우는 사람이 같은 로직을 Effect로 작성하는 데는 더 많은 시간이 걸린다.
홈페이지는 이 비용을 다루지 않는다.

AirMax98은 “Effect의 핵심 인사이트는 타입 시스템으로 오류를 추적할 수 있다는 것인데, 홈페이지 어딘가에 묻혀 있고 대신 현대적인 시너지 마케팅 언어가 앞을 채운다”[^AirMax98]고 지적했다.
이 관찰은 비평이자 동시에 진단이다.
가장 설득력 있는 차별점을 가장 눈에 띄는 곳에 놓지 않으면, 이 라이브러리가 단순한 유행어 집합처럼 보인다.

srhtftw는 약 1년 동안 Effect-TS를 사용한 경험을 공유했다.[^srhtftw]
문서가 함수형 프로그래밍 지식이 있는 사람에게는 충분하지만 그렇지 않은 사람에게는 부족하고,
지난 3년간의 API 변경으로 오래된 게시물과 레포지터리가 대부분 작동하지 않는다고 밝혔다.
Discord 채널에 고품질 도움이 있지만 검색이 불가능한 Discord에 지식이 고립되어 있다는 점을 아쉬워했다.
API 불안정성과 지식의 Discord 고립은 학습 비용 외에 별도로 계산해야 할 리스크다.

apozem은 “연산자 체이닝이 RxJS와 비슷하다”고 했다.[^apozem]
RxJS를 프론트엔드 신입 팀원들에게 2년 동안 가르쳐본 경험에서 말하는데, 학습 곡선이 잔인하다는 것이다.
Effect와 RxJS의 비교는 단순한 스타일 유사성이 아니라 확장 시나리오 예측으로도 유효하다.
함수형 리액티브 프로그래밍을 팀에 이식한 사람들이 공통으로 겪은 것은
도구가 아무리 강력해도 팀 전체가 동일한 멘탈 모델을 공유하지 않으면 그 강력함이 오히려 위험이 된다는 점이다.

### 15KB 코어라는 수치가 실제 사용 규모를 가린다

Effect 런타임 코어 15KB는 사실이지만 실제 프로젝트에서의 번들 크기와 다르다.
HTTP 프레임워크, 데이터 유효성 검사, CLI 도구, 설정 관리 등 Effect 생태계의 모듈들은
각각 별도의 크기를 갖는다.
Effect로 실제 서비스를 구축할 때 총 의존성 크기가 얼마인지는
홈페이지에서 정보를 얻을 수 없다.

이 생략은 전략적이다.
“15KB”는 경쟁 프레임워크와 직접 비교될 수 있는 숫자다.
하지만 Effect의 가치 제안은 경량 런타임이 아니라 포괄적 생태계다.
두 메시지가 충돌한다.
“작고 빠르다”와 “모든 것이 있다”를 동시에 강조하면 한쪽은 과장이거나 다른 쪽이 축소된 것이다.

johnfn은 Effect가 제공하는 것을 두 가지로 분류했다.[^johnfn]
첫째는 강 타입 오류, 둘째는 더 나은 표준 라이브러리다.
첫째에 대해 그는 “바닐라 TypeScript로도 `type Result<T> = { type: 'result'; data: T } | { type: 'error'; message: string }`만으로 거의 동일한 것을 얻을 수 있다”고 주장했다.
둘째는 더 설득력 있지만 그 대가로 `Effect.forEach`, `Effect.runPromise`, `Effect.andThen` 같은 비표준 방식으로 코드를 써야 한다는 점에서 비용이 너무 높다고 봤다.
이 시각은 Effect를 이미 사용하는 사람과 그렇지 않은 사람 사이의 인식 차이를 잘 포착한다.
Effect 안에서는 모든 것이 일관되게 보이지만, 밖에서 보면 호환되지 않는 API 세계처럼 보인다.

### Pipe 기반 API 스타일이 팀 구성에 따라 생산성을 낮출 수 있다

Effect의 코드는 pipe를 통한 함수 합성 스타일로 작성된다.
이 스타일은 함수형 프로그래밍에 익숙한 팀에게는 우아하고 표현력이 높다.
그러나 OOP 배경이나 명령형 스타일에 익숙한 팀에게는
같은 코드가 읽기 어렵고 작성하기 어렵다.

“composability”는 Effect가 핵심 가치로 내세우는 개념이다.
그러나 컴포저빌리티는 팀의 패러다임 이해도와 독립적으로 가치를 갖지 않는다.
합성 가능한 코드를 팀이 합성 방식으로 읽고 쓸 수 없다면
실제 프로덕션에서는 단일 거대 함수로 회귀하거나
Effect 패턴을 절반만 적용한 혼합 코드로 귀결된다.
이 현상은 Effect만의 문제가 아니라 고수준 추상을 도입할 때 공통으로 발생한다.
홈페이지는 이 문제를 다루지 않는다.

RussianCow는 이 문제를 언어 수준으로 끌어올렸다.[^RussianCow]
“JavaScript에는 파이프 연산자(`|>`) 같은 함수 합성 문법이 없어서 모든 연산을 메서드 호출과 인수 전달로 합성해야 하고, 이것이 개념적 단순성을 가리는 시각적 노이즈를 만든다.”
그의 결론은 함수형 스타일이 프로젝트 전반에 이익이 된다면
차라리 JavaScript로 컴파일되는 함수형 언어—ReScript, OCaml, F#, Scala—를 쓰라는 것이다.
이 관점은 Effect를 “TypeScript에서 함수형 프로그래밍을 하는 방법”이 아니라
“TypeScript가 함수형 언어가 아니라는 근본적 한계를 라이브러리로 우회하는 시도”로 재프레이밍한다.

ozuly는 개인 스타트업 경험에서 실용적 결론을 공유했다.[^ozuly]
소규모 팀을 직접 채용하는 맥락에서는 함수형 프로그래밍이 가장 생산적이었지만,
팀 규모를 키울 때 함수형 경험이 없는 개발자를 온보딩하기가 너무 어렵다는 것을 발견했다.
결과적으로 Effect 대신 표준 TypeScript를 선택했고, remeda와 ts-pattern 같은 가벼운 함수형 라이브러리를 부분적으로 사용한다.
이 경험은 “작은 팀에서의 생산성”과 “성장하는 팀에서의 채용 가능성”이 다른 최적화 목표임을 보여준다.
Effect가 표방하는 프로덕션 그레이드 복잡성과 실제 팀이 직면하는 채용 풀의 제약 사이의 간극이다.

## 인사이트

### TypeScript 생태계의 두 방향이 동시에 올바를 수 있다

TypeScript 표준 라이브러리 공백을 채우는 방법은 두 가지다.
첫 번째는 현재 지배적인 방향으로, 문제마다 가장 적합한 작은 패키지를 선택한다.
zod로 유효성 검사, date-fns로 날짜 처리, axios나 ky로 HTTP, winston으로 로깅.
두 번째는 Effect가 제안하는 방향으로, 단일 패러다임 안에서 모든 관심사를 일관되게 처리한다.

이 두 방향은 경쟁이 아니라 다른 맥락에 적합하다.
작은 팀이 빠르게 빌드해야 하는 스타트업은 첫 번째 방향이 유리하다.
온보딩한 패키지가 익숙하고 커뮤니티 지원이 넓다.
대규모 팀이 장기 유지보수하는 복잡한 서비스는 두 번째 방향이 유리하다.
일관성이 불일치 오류를 줄이고 트레이싱이 디버깅을 가속한다.
Effect의 채택 패턴이 주로 엔터프라이즈 Node.js 서비스에 집중되는 것은 이 이유다.

### 오류 타입의 타입 시스템 통합이 언어 수준에서 다음 단계를 요구한다

Effect가 보여주는 것은 TypeScript에서 오류 타입을 함수 시그니처로 표현하는 것이 기술적으로 가능하다는 증명이다.
Java가 checked exceptions를 언어 수준에서 강제한 것과 달리,
Effect는 라이브러리 수준에서 opt-in 방식으로 같은 안전성을 제공한다.
Java의 실패 원인이 강제성에 있었다면, Effect의 접근은 그 교훈을 반영한다.

그러나 라이브러리 수준 구현은 한계가 있다.
Effect 타입으로 감싸지 않은 서드파티 함수의 오류는 타입 시스템에서 여전히 불투명하다.
throw하는 라이브러리와 Effect를 함께 쓰면 경계 지점에서 수동 변환이 필요하다.
이 한계는 TypeScript가 언어 수준에서 오류 타입을 지원하지 않는 한 해결되지 않는다.
Effect의 성공이 충분히 넓어진다면, 이것이 TC39나 TypeScript 팀에 대한 압력이 될 수 있다.

nielsbot은 Swift의 throwing functions가 실제로 원하는 것에 더 가깝다고 제안했다.[^nielsbot]
Swift에서 오류를 던지는 함수는 전통적인 예외를 던지지 않는다.
오류를 던지면 함수의 정상 결과 타입 대신 호출 함수에 Error를 반환하는 구조다.
그리고 “던질 수 있는” 코드를 호출할 때 오류를 처리하거나 다시 던지도록 언어가 강제한다.
TypeScript에서 `type ErrorThrowingFunction = <T,>() => T|Error` 패턴을 쓸 수 있지만 언어 기능이 아니라 컨벤션이라 일관성이 없다.
이 관찰은 Effect가 해결하는 문제가 실제로 존재하지만, 라이브러리 수준보다 언어 수준 해법이 더 올바르다는 것을 암시한다.
Effect는 언어가 제공하지 않는 것을 라이브러리로 에뮬레이션하는 것이고, 그 에뮬레이션에 따르는 마찰이 학습 곡선의 상당 부분을 차지한다.

### 함수형 효과 시스템의 TypeScript 이식이 10년 주기로 반복되는 패턴이다

Haskell의 IO 모나드(1990년대) → Scala의 ZIO(2019년) → TypeScript의 Effect(2022년~)로 이어지는 계보는
함수형 효과 시스템이 더 주류에 가까운 언어로 주기적으로 이식되는 패턴이다.
각 이식은 이전보다 더 넓은 청중에 도달하면서
원래 개념에서 멀어지는 단순화와 타협을 동반한다.
ZIO는 Haskell IO보다 접근하기 쉬웠고, Effect는 ZIO보다 TypeScript 개발자에게 더 가깝다.

이 패턴이 시사하는 것은 Effect가 끝이 아닐 수 있다는 점이다.
Effect보다 더 단순하고 TypeScript 관용구에 더 가까운 다음 단계의 라이브러리가
나타날 가능성이 있다.
그 라이브러리는 Effect의 개념을 계승하되 파이프 기반 합성 대신
TypeScript 개발자에게 더 자연스러운 API를 제공할 것이다.
역설적으로 Effect의 성공이 이 다음 단계를 앞당길 수 있다.
Effect가 오류 타입 추적, 의존성 주입, 자원 관리의 가치를 증명할수록
이 개념들을 더 낮은 진입 장벽으로 제공하는 도구가 등장할 이유가 커진다.

### 플랫폼 독립성이 Effect를 “TypeScript용 Java 표준 라이브러리”로 만들려는 시도다

Node.js, Deno, Bun, Cloudflare Workers, 브라우저를 모두 지원한다는 것은
Effect 코드가 실행 환경에 의존하지 않는다는 것을 목표로 한다.
이것은 Java의 “Write Once, Run Anywhere”와 같은 야심이다.
Java가 플랫폼 독립적 표준 라이브러리를 통해 생태계를 통일했듯,
Effect는 런타임 독립적 표준 라이브러리를 통해 TypeScript 생태계를 통일하려 한다.

이 야심이 성공하려면 Effect가 단순한 좋은 라이브러리를 넘어 사실상의 표준이 되어야 한다.
Java의 java.util은 선택이 아니었지만 Effect는 선택이다.
선택 가능한 표준 라이브러리는 경쟁자가 계속 등장하는 한 표준으로 굳어지기 어렵다.
Effect 팀이 생태계 확장(HTTP framework, CLI, HTTP client)에 투자하는 이유가 여기 있다.
더 많은 것을 Effect로 해결할 수 있을수록 대안으로 갈아타는 비용이 높아진다.
이것은 좋은 라이브러리를 만드는 전략이 아니라 플랫폼을 만드는 전략이다.
