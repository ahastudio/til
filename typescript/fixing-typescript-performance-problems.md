# TypeScript 성능 문제 진단 및 해결: 사례 연구

원문: <https://www.viget.com/articles/fixing-typescript-performance-problems>

## 요약

Viget의 Solomon Hawk가 대규모 TypeScript 모노레포에서 겪은 심각한 성능 저하를
체계적으로 진단하고 해결한 과정을 기술한 글이다.
프로젝트는 7개 패키지로 구성된 모노레포로, Prisma, Kysely, ts-pattern, hotscript 같은
타입 계산이 무거운 라이브러리를 사용했다.
이미 프로젝트 참조(Project References)와 증분 컴파일(Incremental Compilation)을 적용했음에도
빌드에 6분 이상이 걸리고 인텔리센스(IntelliSense)가 수십 초씩 멈추는 상태였다.

진단은 4단계로 진행됐다. 먼저 에디터 버전·확장·시스템 자원 같은 기본 항목을 점검하고,
`tsc --listFilesOnly`로 불필요한 파일이 컴파일에 포함되지 않는지 확인한다.
이후 `tsc --extendedDiagnostics`로 파일 수, 타입 수, 메모리 사용량, 체크 시간 등의 수치를 수집하고,
`--generateTrace`로 컴파일러 추적 파일을 생성해 Chrome 트레이스 뷰어나
`@typescript/analyze-trace`로 병목 지점을 찾는다.

추적 분석 결과 `common.ts` 파일의 체크 시간이 80초에 달하는 핫스팟이 발견됐다.
원인은 Kysely 헬퍼 함수 내부에서 `keyof Db` 유니언을 분배하는 복잡한 제네릭 체인이었다.
Db 인터페이스에 30개 테이블, 각 테이블마다 수십 개의 필드가 있어
타입 복잡도가 O(n^x) 수준으로 폭발했다.

해결책은 해당 헬퍼 함수를 제거하고 쿼리를 사용 지점에 인라인으로 작성하는 것이었다.
여기에 순환 의존성 제거, 미사용 타입 삭제, 배럴 파일(Barrel File) 정리,
패키지 버전 통일 등의 작업을 병행했다.
최종 결과로 빌드 시간이 6.2분에서 1.3분으로 79% 단축되고,
메모리 사용량이 7.1GB에서 3.5GB로 절반으로 줄었다.

## 분석

### 체계적 진단 방법론의 가치

이 글의 가장 큰 강점은 막연히 “타입이 복잡하니 단순화하자”는 직관이 아니라
측정 가능한 수치를 기반으로 문제를 좁혀가는 과정에 있다.
`--extendedDiagnostics`로 전체 수치를 확인하고, `--generateTrace`로 정확한 병목 파일과 라인을 특정하고,
`@typescript/analyze-trace`로 인간이 읽기 어려운 추적 데이터를 해석 가능하게 만든다.
이 세 단계는 TypeScript 성능 문제를 다루는 표준 절차로 재사용 가능하다.

### 타입 수준 추상화의 숨겨진 비용

Kysely는 SQL 쿼리를 타입 안전하게 작성할 수 있게 해주는 라이브러리다.
타입 안전성을 제공하기 위해 데이터베이스 스키마 전체를 제네릭 타입 파라미터로 전파하며,
컬럼 이름, 테이블 이름, 조인 관계 등을 유니언 분배로 계산한다.
스키마 규모가 작을 때는 이 비용이 무시할 수 있는 수준이지만,
테이블과 컬럼 수가 증가할수록 복잡도가 지수적으로 늘어난다.
이 사례에서 헬퍼 함수 하나가 전체 빌드 시간의 상당 부분을 차지한 것은
이 비용이 가시화되지 않았기 때문이다.

### 모노레포 위생(Monorepo Hygiene)의 중요성

단일 원인 외에도 순환 의존성, 중복 패키지, 배럴 파일 과다, Node.js 버전 불일치 등
여러 “작은” 문제들이 복합적으로 성능을 갉아먹고 있었다.
각각의 개별 비용은 측정하기 어렵지만 누적 효과는 컸다.
타입 수가 4,605,085개에서 2,303,043개로 절반으로 줄었는데,
이 중 상당 부분은 Kysely 헬퍼 외에 이러한 위생 문제에서 비롯됐다.

## 비평

### 강점

진단 절차가 단계별로 재현 가능하게 기술되어 있어 다른 프로젝트에 바로 적용할 수 있다.
성과 수치를 Before/After 비교표로 명확하게 제시한 점도 글의 신뢰도를 높인다.
“근본 원인 하나”에만 집중하지 않고 병행 개선 사항들도 함께 다뤄,
실제 현장에서 성능 개선이 단일 원인 해결이 아닌 복합 작업임을 정직하게 보여준다.

### 약점과 한계

해결책의 핵심이 “헬퍼 함수를 지우고 인라인으로 작성”이라는 점은 다소 아이러니하다.
타입 안전성을 위해 도입한 추상화가 오히려 개발 경험을 해치는 주범이 됐고,
해결책은 추상화를 포기하는 것이었다.
이는 프로젝트 규모가 커질수록 Kysely 같은 라이브러리의 설계 트레이드오프가
실제로 문제가 될 수 있음을 시사하지만, 글에서는 이 트레이드오프를 깊이 다루지 않는다.

또한 “배럴 파일 정리”나 “순환 의존성 제거”가 각각 얼마나 기여했는지
개별 수치가 제시되지 않아, 핵심 변경과 부수 변경의 상대적 효과를 비교하기 어렵다.

## 인사이트

### 타입 시스템은 런타임과 분리된 또 다른 실행 엔진이다

TypeScript의 타입 시스템은 타입을 “검증”하는 수동적 도구가 아니라
복잡한 타입 계산을 수행하는 능동적 엔진이다.
조건부 타입(Conditional Types), 맵드 타입(Mapped Types), 템플릿 리터럴 타입(Template Literal Types),
유니언 분배(Union Distribution) 등은 타입 수준의 프로그래밍 언어를 구성하며,
이 언어로 작성된 코드는 컴파일 타임에 TypeScript 언어 서버가 실행한다.

이 관점에서 보면 “타입이 복잡하다”는 말은 단순히 코드 가독성 문제가 아니라
“컴파일 타임에 실행되는 코드가 비효율적이다”는 성능 문제다.
런타임 코드에서 O(n^2) 알고리즘을 작성하면 성능 리뷰에서 지적받듯,
타입 수준의 복잡도 폭발도 동일한 기준으로 다뤄야 한다.
그러나 대부분의 개발팀은 런타임 성능에는 민감하면서 타입 수준 복잡도에는 무감각하다.
이 인식 격차가 이 사례처럼 수년에 걸쳐 문제가 쌓이다가 어느 순간 한계에 도달하는 패턴을 만든다.

### 라이브러리 선택은 타입 수준 성능 계약을 포함해야 한다

Kysely, ts-pattern, hotscript 같은 “타입 매직”을 적극 활용하는 라이브러리들은
런타임 번들 크기나 실행 성능이 아니라 컴파일 타임 비용으로 대가를 치른다.
이 비용은 프로젝트 초기에는 거의 느껴지지 않다가 코드베이스가 커질수록 비선형적으로 증가한다.

현재 라이브러리를 평가하는 일반적인 기준은 API 설계, 런타임 성능, 번들 크기, 유지보수 활성도 등이다.
여기에 “이 라이브러리가 프로젝트 규모 X에서 컴파일 타임에 얼마나 기여하는가”라는
타입 수준 성능 계약이 추가되어야 한다.
이를 위해서는 라이브러리 벤치마크가 런타임뿐 아니라 컴파일 타임도 측정해야 하고,
TypeScript 팀이 권장하는 타입 작성 패턴(불필요한 유니언 분배 회피, 조건부 타입 최소화 등)을
라이브러리 설계 원칙에 포함시켜야 한다.

이 기준이 없으면 개발자는 타입 안전성이라는 분명한 이득을 보면서
컴파일 시간 지연이라는 숨겨진 비용을 인식하지 못한 채 라이브러리를 채택하게 된다.

### “개발자 경험 부채”는 기술 부채보다 측정하기 어렵고 더 위험하다

이 사례에서 빌드 시간 6분은 단순히 CI 파이프라인이 느린 문제가 아니다.
개발자가 타입 오류 피드백을 받기 위해 수십 초를 기다려야 하고,
인텔리센스가 응답하지 않아 자동완성 없이 코드를 작성해야 하는 상황은
개발자의 집중력 흐름(Flow State)을 지속적으로 끊는다.
이 비용은 스프린트 속도나 버그 수 같은 지표에 직접 나타나지 않아
경영진이나 PM에게 설명하기 어렵다.

기술 부채는 “이 코드를 리팩토링하지 않으면 기능 추가 속도가 느려진다”고
어느 정도 구체적으로 설명할 수 있다.
그러나 개발자 경험 부채는 측정 지점이 개발자의 주관적 경험에 있어
“느리다”, “답답하다”는 감각으로만 표현되는 경우가 많다.
이 글이 `tsc --extendedDiagnostics`와 `--generateTrace`로 수치를 만들어낸 것의 진짜 의미는
단순히 병목을 찾는 것을 넘어, 개발자 경험 부채를 측정 가능한 형태로 만들어
의사결정자에게 설명할 수 있는 언어를 확보한 것이다.
개발팀이 이 종류의 부채를 관리하려면 측정 가능한 지표를 정기적으로 추적하는 습관이 필요하다.
