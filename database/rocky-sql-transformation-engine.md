# Rocky — SQL 변환 엔진

<https://github.com/rocky-data/rocky>

## 소개

Rocky는 Rust로 작성된 SQL 변환 엔진으로, 데이터 파이프라인에 컴파일 타임 타입 검사를 제공한다.
dbt와 유사한 SQL 모델 기반 워크플로를 지원하면서, 스키마 드리프트 감지와 컬럼 레벨 리니지 추적을 정적 검사 수준에서 수행한다.

지원하는 데이터 웨어하우스는 Databricks(프로덕션), Snowflake·BigQuery·Trino(베타), DuckDB(로컬 테스트)다.
Apache 2.0 라이선스로 배포된다.

## 주요 기능

**타입 안전성**
스키마 드리프트, 컬럼 타입 변경, 누락된 의존성을 런타임이 아닌 검사 시점에 탐지한다.
`E010`(컬럼 타입 불일치), `E011`(필수 컬럼 누락), `E013`(안전하지 않은 타입 변환) 같은 오류 코드로 명확하게 진단한다.

**컬럼 레벨 리니지**
소스에서 목적지까지 컬럼 단위로 데이터 흐름을 추적한다.
`rocky lineage-diff`로 PR 리뷰 시 변경된 리니지를 확인할 수 있다.

**네임드 브랜치 및 리플레이**
격리된 스키마 복사본에서 검사 후 프로모션하거나 롤백하는 실험 환경을 제공한다.

**모델별 비용 추적**
개별 모델 수준에서 쿼리 비용을 추적한다.

**이식성 검사**
웨어하우스별 고유 SQL을 감지한다(예: Databricks 프로젝트에서 Snowflake 전용 구문 사용 시 경고).

**증분 로드**
워터마크 기반 전략으로 신규 행만 처리한다.

**데이터 마스킹**
환경별 PII 보호를 거버넌스 강제와 함께 적용한다.

## CLI

```bash
# 설치 (macOS / Linux)
curl -fsSL https://raw.githubusercontent.com/rocky-data/rocky/main/engine/install.sh | bash

# 프로젝트 생성 및 기본 흐름
rocky playground my-first-project
cd my-first-project
rocky compile && rocky test && rocky run

# PR 리뷰를 위한 리니지 diff
rocky lineage-diff

# 프로덕션 배포
rocky plan && rocky apply

# 순수 SQL 추출 (오프라인 사용)
rocky emit-sql
```

## 아키텍처

단일 정적 바이너리 형태로 Rust로 구현되었다.
Python SDK 바인딩과 VS Code 언어 서버 확장을 통해 개발 중 실시간 타입 검사를 제공한다.
`rocky emit-sql` 명령으로 모든 모델을 순수 SQL로 내보낼 수 있어 벤더 락인 없이 사용 가능하다.

## 분석

### dbt와 비교할 때 Rocky의 차별화 지점

데이터 파이프라인 변환 도구 시장에서 dbt는 사실상의 표준이다.
Rocky가 경쟁하는 핵심 지점은 컴파일 타임 타입 검사다.
dbt는 SQL을 템플릿으로 처리하고 런타임에 오류를 발견하지만, Rocky는 Rust 타입 시스템의 철학을 SQL 파이프라인에 적용해 실행 전에 오류를 잡는다.
이 접근은 “빠른 피드백”을 데이터 엔지니어링의 워크플로에 가져오려는 시도다.

컬럼 레벨 리니지 추적도 dbt와 차별화되는 지점이다.
dbt의 리니지는 모델(테이블) 단위이지만, Rocky는 컬럼 단위 추적을 제공한다.
컴플라이언스 요구 사항이 강한 금융이나 의료 분야에서 컬럼 레벨 데이터 흐름 파악은 감사(audit) 목적에 직접 사용할 수 있다.

### Rust 기반 단일 바이너리의 트레이드오프

dbt의 Python 기반 구현은 커뮤니티 확장성과 플러그인 생태계에서 강점을 가진다.
Rocky의 Rust 기반 단일 바이너리는 성능과 의존성 관리 측면에서 유리하지만, 기여 진입 장벽과 생태계 확장 속도에서 불리할 수 있다.
데이터 엔지니어 커뮤니티는 Python에 익숙하고 dbt 플러그인 생태계에 의존하는 경우가 많아, 생태계 전환 비용이 실질적인 채택 장벽이다.

### 네임드 브랜치 실험 환경의 의미

`rocky plan && rocky apply` 패턴은 Terraform의 인프라 변경 관리 방식을 SQL 파이프라인에 적용한 것이다.
스키마 변경을 격리된 환경에서 검증하고 명시적으로 프로모션하는 워크플로는, 데이터 파이프라인의 “인프라로서의 데이터” 개념을 구체화한다.
이 접근은 특히 여러 팀이 공유 데이터 웨어하우스를 사용하는 환경에서 충돌 방지 효과가 크다.

## 비평

### 초기 단계 프로젝트의 신뢰성 문제

Databricks만이 프로덕션 지원이고 Snowflake·BigQuery·Trino는 베타 상태다.
데이터 엔지니어링 도구는 신뢰성이 최우선이기 때문에, 주요 웨어하우스 지원이 베타 상태인 채로 프로덕션 도입을 설득하기 어렵다.
dbt와 같은 성숙한 도구와 경쟁하려면 광범위한 웨어하우스 지원의 안정화가 선행되어야 한다.

### AI 기반 모델 생성 기능의 위치가 모호하다

자연어로 SQL 모델을 생성하는 AI 기능을 제품 기능으로 포함하고 있지만, 이 기능이 Rocky의 핵심 가치(컴파일 타임 타입 안전성)와 어떻게 연결되는지가 명확하지 않다.
AI가 생성한 SQL이 타입 검사를 통과하면 의미가 있지만, AI 코드 생성은 이미 범용 AI 도구로 처리 가능하다.
이 기능이 Rocky에 통합될 필요성이 있는지 의문이다.

## 인사이트

### SQL 파이프라인의 “컴파일 타임 안전성”은 성숙한 엔지니어링 문화를 전제한다

컴파일 타임 타입 검사의 가치는 런타임 오류의 비용이 높을 때 극대화된다.
데이터 파이프라인에서 런타임 오류는 잘못된 데이터가 다운스트림으로 흘러가 비즈니스 의사결정에 영향을 미칠 때 발생한다.
이 비용을 이해하고 예방에 투자하는 조직은 상대적으로 데이터 엔지니어링이 성숙한 곳이다.
Rocky의 타겟은 데이터 파이프라인을 인프라처럼 관리하는 문화를 가진 팀이다.

### 데이터 파이프라인 도구의 표준화는 Terraform→dbt→Rocky 방향으로 이동 중이다

인프라 관리가 수동 → CFN/Terraform → GitOps 방향으로 발전했듯, 데이터 파이프라인 관리도 수동 SQL → dbt → 타입 안전 변환 엔진 방향으로 이동할 수 있다.
Rocky는 이 흐름에서 다음 단계를 선점하려는 포지셔닝이다.
단, 이 전환의 속도는 데이터 엔지니어 커뮤니티의 타입 시스템 익숙도와 도구 전환 의향에 달려 있어, 소프트웨어 엔지니어링 도구의 전환보다 느릴 가능성이 있다.
