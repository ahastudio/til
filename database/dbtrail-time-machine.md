# dbtrail

<https://github.com/dbtrail/dbtrail>

## 소개

dbtrail은 MySQL을 위한 타임머신을 표방하는 오픈소스 도구다.
“모든 행 변경을 기억한다.
검색 가능하고, 되돌릴 수 있으며, 여러분의 것이다.
잘못된 UPDATE는 2분짜리 수정 작업이 된다”는 문장으로
프로젝트를 소개한다.
잠금이나 스키마 변경, 복원 대기 없이 MySQL을 특정 시점으로
되돌려 조회할 수 있는 시점 복구(point-in-time recovery)를
핵심 목표로 삼는다.
`SELECT * FROM orders WHERE id = 123 AS OF '2026-05-20 14:00:00'`
같은 문법으로 과거 시점의 데이터를 그대로 조회할 수 있다는 점이
프로젝트를 상징적으로 보여준다.

## 아키텍처

dbtrail은 MySQL의 복제 프로토콜을 통해 바이너리 로그를
직접 추적(tailing)하며, 모든 행 변경 내역을 검색 가능한
색인으로 유지한다.
디스크에 바이너리 로그 파일을 별도로 보관할 필요가 없는 구조라서,
Amazon RDS나 Aurora MySQL, Google Cloud SQL 같은 매니지드
클라우드 데이터베이스와도 호환된다.
동작을 위해서는 MySQL 8.0 이상에서 `binlog_format=ROW`와
`binlog_row_image=FULL` 설정이 필요하며, MySQL 본체뿐 아니라
Percona Server, Aurora, RDS 계열에서도 사용할 수 있다.

## 주요 기능

- **변경 이력 조회** — 모든 행에 대해 무엇이 언제 바뀌었는지
  변경 전후(before → after) 값을 함께 확인할 수 있다.
- **정밀 복구** — 손상된 행만 정확히 되돌리는 반대 방향 SQL을
  생성한다.
- **캐스케이드 삭제 복원** — `ON DELETE CASCADE`로 함께 삭제된
  자식 행과 외래 키 관계를 재구성한다.
- **시점 조회(time-travel query)** — 임의의 과거 시점 기준으로
  행 상태를 그대로 조회한다.
- **변경 주체 추적** — 어떤 데이터베이스 사용자, 호스트,
  클라이언트 프로그램이 변경을 일으켰는지 귀속시킨다.
- **복구 사전 검증** — `bintrail verify` 명령으로 복구가
  실제로 성공할지 실행 전에 확인한다.
- **웹 콘솔** — 변경 내역을 탐색하고 복구 작업을 관리하며
  모니터링 대상 서버를 추가할 수 있는 UI를 제공한다.
- **MCP 서버 연동** — Claude를 비롯한 MCP 클라이언트가
  변경 이력을 검색하고 복구 초안을 작성할 수 있다.

## 라이선스 및 상태

Go로 작성된 코드가 전체의 94.4%를 차지하며,
나머지는 웹 콘솔을 위한 JavaScript와 CSS로 구성되어 있다.
Apache-2.0 라이선스로 배포되어 상업적 사용과 프로덕션 사용이
모두 자유롭다.
이 글을 쓰는 시점 기준 670개 이상의 커밋과 107개의 릴리스를
거쳤으며, 최신 버전은 2026년 7월에 나온 v0.29.2다.

## 분석

### 바이너리 로그 추적 방식이 갖는 구조적 의미

dbtrail이 바이너리 로그 파일을 디스크에 직접 보관하지 않고
복제 프로토콜을 통해 실시간으로 추적하는 방식을 택한 것은,
매니지드 클라우드 데이터베이스 사용자를 명확히 겨냥한 설계
결정이다.
RDS나 Aurora 같은 매니지드 서비스는 사용자에게 바이너리 로그
파일에 대한 직접 접근 권한을 주지 않는 경우가 많다.
복제 프로토콜은 애초에 이런 환경에서도 열려 있는 통로이기
때문에, dbtrail은 이 통로를 감사(audit)와 복구라는 새로운
용도로 재활용한 셈이다.

### 시점 복구가 백업과 경쟁하지 않고 보완하는 이유

전통적인 백업-복원 방식은 데이터베이스 전체를 특정 시점으로
되돌리는 데는 강하지만, “특정 행 몇 개만 실수로 잘못
업데이트된” 상황에는 지나치게 무겁다.
dbtrail은 행 단위의 변경 이력을 유지함으로써 이 틈새를
정확히 메운다.
전체 복원이 필요한 재해 상황에는 여전히 백업이 필요하지만,
일상적으로 더 자주 발생하는 “실수로 잘못된 UPDATE 한 줄”
문제에는 dbtrail이 훨씬 빠르고 정밀한 해법을 제공한다.
두 도구는 대체 관계가 아니라 보완 관계로 봐야 한다.

### MCP 연동이 시사하는 복구 워크플로의 방향

dbtrail이 MCP 서버 연동을 지원한다는 점은, 데이터베이스 복구
작업이 점차 자연어 기반의 대화형 인터페이스로 옮겨가고 있다는
신호로 읽을 수 있다.
과거에는 SQL을 직접 작성해 복구 스크립트를 만들어야 했다면,
이제는 “어제 오후 2시 기준으로 이 주문 데이터를 복구해줘”라는
요청을 AI 에이전트가 dbtrail의 이력 조회 기능을 통해 처리하는
흐름이 가능해진다.
이는 데이터베이스 운영 도구가 사람이 아니라 에이전트를
1차 사용자로 상정하기 시작했다는 더 넓은 흐름의 한 사례다.

### 검증 단계를 별도로 둔 설계 철학

`bintrail verify`로 복구 전 사전 검증을 제공하는 것은,
되돌릴 수 없는 실수를 막기 위한 이중 안전장치를 명시적으로
설계에 포함시켰다는 뜻이다.
복구 도구 자체가 새로운 사고의 원인이 되는 것을 막으려면
실행 전 검증이 필수적인데, 많은 유사 도구들이 이 단계를
생략하고 “실행 후 확인”에 의존한다.
dbtrail이 이를 별도 명령으로 분리한 것은 복구 작업의
심리적 부담을 낮추려는 의도적 설계로 볼 수 있다.

## 비평

### 바이너리 로그 설정 요구사항이 진입 장벽으로 작용한다

`binlog_row_image=FULL` 설정은 모든 변경된 행의 전체 이미지를
로그에 남기므로, 이미 운영 중인 데이터베이스의 로그 볼륨을
크게 늘린다.
이미 대규모 트래픽을 처리하고 있는 기존 시스템에 이 설정을
새로 적용하려면 스토리지 비용 증가와 복제 지연 가능성을
감수해야 하는데, README는 이 운영 비용에 대해 구체적인
수치나 가이드를 제공하지 않는다.
“잠금 없이, 스키마 변경 없이”라는 홍보 문구는 사실이지만,
바이너리 로그 설정 변경 자체가 무비용은 아니라는 점을
가볍게 다루고 있다.

### 27개의 스타는 아직 프로덕션 검증이 부족하다는 신호일 수 있다

670개의 커밋과 107개의 릴리스라는 활발한 개발 이력에 비해
스타 수가 27개에 그친다는 것은 다소 이례적이다.
이는 프로젝트가 실제 커뮤니티의 폭넓은 검증을 아직 거치지
못했을 가능성을 시사한다.
데이터 복구라는, 실수가 곧바로 비즈니스 손실로 이어지는
민감한 영역의 도구일수록 커뮤니티 검증의 폭이 신뢰도에
직결되는데, 현재 시점에서는 이 부분이 상대적으로 얇다.

### 캐스케이드 삭제 복원의 한계가 명시되어 있지 않다

`ON DELETE CASCADE`로 삭제된 자식 행을 재구성하는 기능은
매력적이지만, 복원 시점 사이에 새로 생성된 행이나 외래 키가
가리키는 대상 자체가 바뀐 경우처럼 복잡한 케이스에서
어떻게 동작하는지는 문서에 설명되어 있지 않다.
단순한 삭제-복원 시나리오를 넘어서는 복잡한 참조 무결성
상황에 대한 검증 사례가 부족한 채로 이 기능을 핵심 기능으로
내세우는 것은 성급한 일반화로 보인다.

## 인사이트

### 복구 도구의 무게중심이 DBA에서 AI 에이전트로 옮겨가고 있다

dbtrail의 MCP 연동은 단순한 부가 기능이 아니라, 데이터베이스
복구라는 고도로 전문적인 작업의 주된 수행 주체가 바뀌고 있다는
신호로 볼 수 있다.
과거에는 복구 작업이 숙련된 DBA의 판단과 수작업 SQL 작성에
의존했다면, 앞으로는 AI 에이전트가 이력을 조회하고 복구 초안을
작성한 뒤 사람이 최종 승인만 하는 구조로 재편될 가능성이 있다.
이는 데이터베이스 운영 전반에서 “사람이 실행하고 도구가
보조한다”는 구도가 “도구가 실행하고 사람이 검증한다”는
구도로 뒤집히는 더 큰 흐름의 초기 사례다.

### 매니지드 클라우드 데이터베이스의 제약이 새로운 도구 카테고리를 낳는다

dbtrail이 복제 프로토콜을 활용하는 우회 설계를 택한 것은,
클라우드 벤더가 사용자에게 제공하지 않는 저수준 접근 권한을
서드파티 도구가 어떻게든 메워야 하는 상황을 보여준다.
클라우드 데이터베이스가 편의성을 위해 저수준 제어권을
거둬들일수록, 그 거둬들여진 제어권을 다른 방식으로 복원해주는
생태계 도구에 대한 수요는 오히려 커진다.
이는 클라우드 플랫폼의 추상화 수준과 서드파티 도구 생태계의
성장이 반비례가 아니라 정비례할 수 있음을 보여주는 사례다.

### 행 단위 시간 여행은 감사 규제 대응의 부산물로 자리 잡을 가능성이 있다

dbtrail이 제공하는 변경 주체 추적과 시점 조회 기능은
원래 장애 복구를 위해 설계되었지만, 동일한 기능이 금융이나
의료 분야의 감사 로그 요구사항을 충족하는 데도 그대로
활용될 수 있다.
장애 복구 도구와 컴플라이언스 감사 도구가 기술적으로
수렴하는 이 현상은, 앞으로 데이터베이스 인프라 도구들이
단일 목적이 아니라 운영과 규제 대응이라는 두 가지 요구를
동시에 만족시키는 방향으로 설계될 가능성을 시사한다.
