# Instant 1.0 — AI로 작성한 앱을 위한 백엔드 플랫폼

원문: <https://www.instantdb.com/essays/architecture>

## 요약

InstantDB는 AI로 코딩된 애플리케이션을 위한 백엔드 플랫폼이다.
실시간 동기화, 오프라인 모드, 인증, 파일 스토리지를 하나의 플랫폼에 통합했다.
클라이언트 SDK, Clojure 백엔드, PostgreSQL 기반 멀티테넌트 데이터베이스라는
세 가지 핵심 구성요소로 이루어져 있다.

클라이언트 SDK는 IndexedDB를 활용한 로컬 퍼스트 접근으로
오프라인 기능과 낙관적 업데이트(Optimistic Updates)를 처리한다.
데이터를 [entity, attribute, value] 트리플로 저장하고
Datalog 쿼리로 서버 없이 클라이언트에서 쿼리를 평가할 수 있다.

백엔드는 리액티브 쿼리 관리와 멀티테넌트 리소스 분배를 담당한다.
PostgreSQL의 WAL(Write-Ahead Log)을 모니터링하여
데이터 변경 시 영향받는 쿼리만 선별적으로 갱신하는 토픽 기반
무효화(Invalidation) 시스템을 구현했다.

데이터베이스는 앱별 별도 스키마 대신 단일 통합 트리플 테이블에
app_id로 논리적 격리를 수행하는 독특한 설계를 택했다.
이를 통해 락 없는 컬럼 생성, 밀리초 단위의 소프트 삭제 복구,
VM 기동 없는 즉시 앱 프로비저닝을 달성했다.

## 분석

### EAV 모델의 현대적 재해석

Entity-Attribute-Value(EAV) 패턴은 전통적으로 스키마 유연성의
대가로 쿼리 성능을 포기하는 것으로 알려져 있다.
InstantDB는 이 약점을 Count-Min Sketch로 보완한다.
PostgreSQL의 통계 시스템이 테이블과 컬럼 단위로 작동하므로
단일 트리플 테이블에서는 정확한 카디널리티 추정이 불가능한데,
확률적 자료구조로 컬럼별 카디널리티를 근사하여
쿼리 플래너가 효율적인 실행 계획을 생성하도록 돕는다.

### Clojure와 JVM 선택의 전략적 의미

대부분의 모던 백엔드가 Go, Rust, TypeScript를 선택하는 상황에서
Clojure를 택한 것은 흥미로운 결정이다.
실제 스레딩으로 효율적 동시성 처리가 가능하고,
DSL 작성에 강점이 있으며,
Grouped Queue 구현이 215줄로 가능할 만큼 추상화 능력이 뛰어나다.
Google의 CEL 같은 JVM 라이브러리를 샌드박스된 권한 언어로
활용할 수 있다는 점도 실무적 이점이다.

### InstaQL의 설계 철학

별도의 쿼리 언어 문법을 도입하지 않고
JavaScript 객체 구문을 그대로 쿼리 언어로 사용한다.
이로 인해 빌드 단계나 코드 생성 없이 프로그래밍적 쿼리 생성이 가능하다.
`db.useQuery()`와 `db.transact()`만으로 투두 앱에
동기화, 오프라인 지원, 실시간 업데이트를 구현하는 데 25줄이면 충분하다.
이 간결함이 AI 코딩 에이전트와의 궁합이 좋은 이유다.

## 비평

### "AI를 위한 백엔드"라는 포지셔닝의 모호함

AI로 코딩된 앱을 위한 백엔드라는 포지셔닝이 실질적 차별화인지,
아니면 마케팅적 프레이밍인지 불분명하다.
실시간 동기화, 오프라인 지원, 간결한 API는
AI 코딩과 무관하게 모든 애플리케이션에 유용한 기능이다.
"AI가 25줄로 앱을 만들 수 있다"는 것은
개발자 경험(DX)의 우수함이지 AI 특화 기능은 아니다.

### 멀티테넌트 단일 테이블의 리스크

모든 앱의 데이터를 하나의 트리플 테이블에 저장하는 것은
운영 단순성과 확장성 사이의 극단적 트레이드오프다.
Noisy Neighbor 문제가 발생할 수 있으며,
한 앱의 대량 쓰기가 다른 앱의 읽기 성능에 영향을 줄 수 있다.
Grouped Queue로 앱 간 직렬 처리를 보장한다고 하지만,
데이터베이스 수준의 I/O 경합은 큐만으로 해결되지 않는다.

### 소프트 삭제 복구의 범위

에이전트 워크플로우에서 실수로 삭제된 컬럼을
밀리초 단위로 복구할 수 있다는 것은 매력적이지만,
이것이 어디까지 보장되는지 명확하지 않다.
트리플이 물리적으로 삭제된 경우와 논리적 마킹만 된 경우의
구분, 복구 가능 기간, 스토리지 비용에 대한 설명이 부족하다.

## 인사이트

### 트리플 스토어의 부활과 패러다임 순환

InstantDB의 트리플 스토어 접근은 시맨틱 웹 시대의 RDF 트리플과
구조적으로 동일하다.
2000년대 초반 W3C가 추진한 시맨틱 웹은
모든 데이터를 주어-술어-목적어 트리플로 표현하고자 했지만,
복잡성과 성능 문제로 주류가 되지 못했다.

그런데 20년 뒤 같은 구조가 "멀티테넌트 백엔드"라는 다른 맥락에서
부활한 것이다.
시맨틱 웹은 웹 전체의 지식 그래프를 목표로 했기에
범위가 너무 넓었지만, InstantDB는 단일 애플리케이션의
데이터 모델링이라는 좁은 범위에 트리플을 적용함으로써 실용성을 확보했다.

기술의 성패는 종종 기술 자체보다 적용 범위에 의해 결정된다.
XML도 범용 데이터 교환 포맷으로는 실패했지만
구성 파일과 문서 포맷에서는 여전히 살아있다.
트리플 스토어 역시 "모든 데이터를 위한 구조"로서는 과도했지만,
"스키마 없는 멀티테넌트 스토어"로서는 적합할 수 있다.

### 로컬 퍼스트와 서버 권위의 긴장

InstantDB의 클라이언트 SDK는 로컬에서 쿼리를 평가하고
낙관적 업데이트로 즉각적인 UI 반응을 제공하면서도,
서버가 트랜잭션을 거부하면 자동으로 롤백한다.
이는 로컬 퍼스트(Local-First)와 서버 권위(Server Authority)라는
상충하는 두 패러다임을 절충한 설계다.

이 절충은 분산 시스템의 CAP 정리와 맞닿아 있다.
오프라인에서 로컬 변경을 허용하면(가용성)
서버와의 일관성을 즉시 보장할 수 없다.
Pending Queue가 트랜잭션을 서버 확인 전까지 별도로 관리하는 것은
최종 일관성(Eventual Consistency) 모델의 구현이다.

흥미로운 것은 이 패턴이 Git의 작업 모델과 유사하다는 점이다.
로컬에서 자유롭게 커밋하고, push할 때 원격과 충돌을 해소한다.
InstantDB의 클라이언트는 로컬에서 자유롭게 트랜잭션을 적용하고,
서버와 동기화할 때 충돌을 해소한다.
이 패턴의 보편성은 "오프라인 우선, 동기화 나중"이
분산 시스템의 자연스러운 해법임을 시사한다.

### AI 에이전트 시대의 백엔드 요구사항 변화

InstantDB가 "AI를 위한 백엔드"를 표방하는 이면에는
실질적인 변화가 있다.
AI 코딩 에이전트는 스키마를 자주 변경하고,
실수로 컬럼을 삭제하며, 대량의 실험적 앱을 빠르게 생성한다.
전통적인 데이터베이스의 DDL 락, 마이그레이션 절차,
인스턴스 프로비저닝 시간은 이런 워크플로우에 병목이 된다.

InstantDB가 "락 없는 컬럼 생성"과 "밀리초 소프트 삭제 복구"를
강조하는 것은 이 새로운 사용 패턴에 대한 직접적 대응이다.
AI 에이전트가 데이터베이스 스키마를 코드처럼 빠르게 반복할 수 있어야
"바이브 코딩" 워크플로우가 완성된다.

이는 DevOps가 인프라를 코드화(IaC)했듯,
AI 에이전트가 데이터 모델을 코드화하는 것으로 볼 수 있다.
인프라의 불변성(Immutability)이 DevOps의 핵심 원칙이었다면,
스키마의 유연성(Flexibility)이 AI 에이전트 시대 백엔드의
핵심 원칙이 될 수 있다.
