# SQL to ER Diagram

> Free Online ERD Generator from SQL (no signup)

<https://sqltoerdiagram.com/>

GeekNews: <https://news.hada.io/topic?id=30499>

HN 토론: <https://news.ycombinator.com/item?id=48523992> (362점, 73개 댓글)

## 소개

SQL `CREATE TABLE` 구문을 인터랙티브 엔티티-관계(Entity-Relationship) 다이어그램으로 변환하는 브라우저 기반 무료 도구다.
PostgreSQL, MySQL, SQLite, SQL Server 문법을 지원하며, 기본 키·외래 키·제약 조건을 처리한다.
모든 처리는 브라우저 내에서 100% 로컬로 실행되며 스키마가 외부로 전송되지 않는다.
로그인과 설치 없이 사용 가능하다.

## 주요 기능

- **다중 SQL 방언 지원**: PostgreSQL, MySQL, SQLite, SQL Server 문법을 처리한다.
- **인터랙티브 캔버스**: 테이블을 드래그·드롭으로 재배치하고 자동 레이아웃을 적용할 수 있다.
- **내보내기**: PNG, SVG 형식으로 다이어그램을 내보낼 수 있다.
- **공유 가능한 링크**: 다이어그램을 URL로 인코딩하여 공유할 수 있다.
- **로컬 처리**: 스키마가 브라우저 밖으로 나가지 않는다.

## 구현

창작자 robhati는 HN 댓글에서 구현 세부 사항을 공유했다.[^robhati]
성능을 위한 캔버스 기반 렌더링, 정밀 편집을 위한 소스 추적 SQL 파서, URL 기반 스키마 공유를 구현했다.
초기에 Rust/WebAssembly(WASM)를 시도했지만, 최종적으로 약 32KB(gzip 압축)의 순수 자바스크립트를 선택했다.

HN에서 CraigJPerry는 “이 코드베이스는 신선한 공기 같다”며 복잡한 문제를 우아한 해결책으로 압축한 것을 높이 평가했다.[^CraigJPerry]

## 분석

### 개인 도구 카테고리의 성숙

“설치 없이, 로그인 없이, 서버로 데이터를 보내지 않는다”는 원칙은 단순한 기능 목록이 아니라 특정 사용자 그룹에 대한 명확한 포지셔닝이다.
보안에 민감한 환경에서 작업하는 개발자들은 스키마를 외부 서버에 업로드하는 것을 꺼린다.

오픈소스 및 자체 호스팅 가능한 DB 다이어그래밍 도구들이 이 수요를 충족해왔지만, 설치 과정 자체가 마찰을 만들었다.
순수 브라우저 실행 방식은 이 마찰을 완전히 제거한다.

### 정확성의 트레이드오프

HN에서 michaelmior은 기술적으로 중요한 구분을 지적했다.[^michaelmior]
엔티티(entity)와 관계형 테이블(relational table)은 개념적으로 다르며, SQL 스키마만으로는 진정한 ER 다이어그램을 완전히 생성할 수 없다는 것이다.
SQL 스키마는 테이블 구조와 외래 키 제약을 담지만, 개념적 엔티티 간 관계의 의미론적 레이어는 SQL에 명시적으로 표현되지 않는다.

이 도구가 생성하는 것은 엄밀히 말해 ER 다이어그램이 아니라 “스키마 다이어그램”이다.
실용적 목적 — 기존 데이터베이스 구조를 시각화하고 이해하기 — 에는 충분하지만, 정밀한 데이터 모델링에는 한계가 있다.

이 지적에 대한 반론도 설득력 있었다.
eventinbox는 엔티티와 테이블의 구분이 실재하는 것은 맞지만, 기존 스키마를 빠르게 시각화하려는 대다수 개발자에게는 충분하다고 응수했다.[^eventinbox]
대부분은 ORM 추상을 설계하는 것이 아니라 무엇이 무엇과 연결되는지 보고 싶을 뿐이며, 여기서는 “완벽함이 유용함의 적”이라는 것이다.
michaelmior 본인도 자신이 다소 현학적이라는 점을 인정했다.

### 미지 데이터베이스 탐색 도구로서의 가치

HN에서 systems는 가장 실용적인 사용 사례를 지적했다.[^systems]
문서가 없거나 외래 키 제약이 없는 낯선 데이터베이스를 이해해야 할 때, 이 도구가 특히 유용하다.
레거시 시스템을 인수인계받거나, 문서화가 부실한 오픈소스 프로젝트의 데이터베이스를 파악해야 할 때가 대표적이다.

## 비평

### 외래 키 없는 스키마의 처리

많은 레거시 데이터베이스가 외래 키 제약 없이 참조 무결성을 애플리케이션 레이어에서 처리한다.
이 경우 도구는 테이블 간의 관계를 자동으로 추론할 수 없다.
systems가 제안한 것처럼 관계를 수동으로 추가하는 기능이 있다면 이 한계를 부분적으로 해소할 수 있다.

### 대용량 스키마의 가용성

캔버스 기반 렌더링은 성능에 유리하지만, 수백 개의 테이블을 가진 엔터프라이즈 데이터베이스에서의 사용성은 검증이 필요하다.
freeopinion이 제안한 그룹 기반 테이블 드래깅과 관계 선 재형성 같은 기능이 대규모 스키마 탐색을 위해 필요할 수 있다.[^freeopinion]

## 인사이트

### 32KB 구현이 보여주는 기술 선택의 철학

Rust/WASM을 포기하고 32KB 순수 자바스크립트를 선택한 결정은 흥미롭다.
HN에서 written-beyond가 모바일 사용성에 “10/10”을 줄 만큼 부드러운 인터랙션이 가능한 것은 캔버스 렌더링의 덕이다.[^written-beyond]
기술적 복잡성을 추가하지 않고 문제에 맞는 최소한의 도구를 선택하는 철학이 결과물에 반영됐다.

“더 현대적인 기술 스택을 쓰면 더 좋은 결과가 나온다”는 가정을 이 프로젝트는 조용히 반박한다.
도구의 선택보다 도구의 적합성이 더 중요하다는 것을 보여주는 사례다.

### 브라우저가 앱 플랫폼으로서 완성되고 있다

이 도구가 가능한 것은 현대 브라우저가 복잡한 그래픽 렌더링, 충분한 CPU/메모리 처리, 로컬 상태 관리를 모두 담당할 수 있기 때문이다.
10년 전이라면 이런 도구는 데스크톱 앱이었을 것이다.
“설치 없이, 로그인 없이, 서버 없이” 동작하는 복잡한 도구가 브라우저에서 가능해진 것은 개발자 도구의 접근성을 극적으로 높인다.
