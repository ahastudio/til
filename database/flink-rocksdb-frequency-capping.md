# Apache Flink와 RocksDB 튜닝으로 실시간 광고 노출 빈도 제한 구현하기

원문: <https://toss.tech/article/flink-realtime-frequency-capping>

## 요약

토스 데이터 서비스 플랫폼 팀이 Apache Flink와 RocksDB를 활용하여 광고 노출
빈도 제한(frequency capping) 시스템의 실시간 집계 범위를 수 시간에서 7일로
확장한 사례를 다룬다. 기존 시스템은 Head(실시간 Redis), Mid(일별 Spark
사전 집계), Tail(시간별 경계 보정)의 삼중 구조로, 하루 75개 Airflow DAG
실행과 서빙당 최대 4회의 Redis 조회가 필요했다.

새 아키텍처는 세 개의 독립 Flink 애플리케이션(minutes, hours, days)으로
분리하여 각각의 병목에 맞는 RocksDB 튜닝 전략을 적용했다. 핵심 설계
원칙은 State를 유일한 진실의 원천(single source of truth)으로 삼고
Redis를 단순한 투영(projection)으로 격하시키는 것이다.

특히 days 앱은 7일 윈도우로 인해 State가 220~230GB에 달하는데,
Changelog State를 도입하여 체크포인트 시간을 총 State 크기가 아닌
마지막 체크포인트 이후 변경분에만 의존하도록 만들었다. DSTL 업로드는
밀리초~초 단위로 완료되고, 비동기 Materialization은 5~10분 주기로
전체 스냅샷을 캡처한다.

결과적으로 Airflow DAG 실행은 75개에서 25개로, 서빙 쿼리는 단일 Redis
조회로 단순화되었다. 삼중 계층 의존 구조가 통합 실시간 시스템으로
전환되었다.

## 분석

### 삼중 앱 분리 전략의 합리성

1분~7일이라는 넓은 시간 범위를 단일 파이프라인으로 처리하지 않고 세 개의
독립 애플리케이션으로 분리한 것은 핵심적인 설계 결정이다. minutes 앱은
Write Buffer 압력이, hours 앱은 Filter Block 디스크 I/O가, days 앱은
체크포인트 I/O가 병목이었다. 동일한 RocksDB를 사용하면서도 병목 유형이
완전히 다르다는 관찰은, 시간 윈도우 크기가 State 크기를 결정하고 State
크기가 병목 유형을 결정한다는 인과 관계를 명확히 보여준다.

### 2단계 파이프라인의 설계 배경

초기에 히스토리 백필과 실시간 캐치업을 하나의 파이프라인으로 결합하려
했으나, 만료 타이머가 불완전한 데이터에 대해 조기 발동하는 문제가
발생했다. 백필 단계에서는 타이머를 등록하지 않고 카운트만 증가시키고,
캐치업 단계에서 타이머를 활성화하는 분리는 State 초기화와 실시간
처리의 의미적 차이를 존중한 설계다. 이것은 이벤트 소싱(event sourcing)
패턴에서 리플레이와 라이브 처리를 분리하는 것과 같은 원리다.

### RocksDB 튜닝의 깊이

단순한 설정값 조정이 아니라 소스 코드 수준의 조사가 필요했다는 점이
주목할 만하다. Flink가 파티션 필터 활성화 시 `bloom-filter.bits-per-key`를
덮어쓴다는 사실, `withIdleness`가 Bounded Source에서 보이는 동작,
Native Savepoint와 Changelog의 상호작용으로 인한 교착 상태 등은
문서화되지 않은 행동들이다. 이 수준의 디버깅은 프레임워크에 대한
깊은 이해 없이는 불가능하다.

### Changelog State의 전략적 가치

220~230GB State에서 전통적 체크포인트는 실질적으로 불가능하다.
Changelog State가 변경분만 캡처하여 체크포인트 시간을 State 크기에서
분리(decouple)한 것은 대규모 상태 관리의 핵심 돌파구다. 2분 체크포인트
간격, 30초 최소 일시 정지, 10분 Materialization 주기라는 구체적인
설정값은 안정성과 복구 시간 사이의 트레이드오프를 보여준다.

## 비평

### 실전 경험에 기반한 높은 신뢰도

이 글의 가장 큰 강점은 실제 프로덕션 환경에서의 경험에 기반한다는
점이다. Block Cache 적중률 62~64%에서 99~100%로의 개선, Write
Amplification 30에서 20으로의 감소, CPU 프로파일링에서 96.2%가
`ReadBlockContents()`에 소비된다는 관찰 등 정량적 데이터가 주장을
뒷받침한다. Grafana를 통한 메트릭 검증도 엔지니어링 규율을 보여준다.

### 비용 및 운영 복잡성의 누락

삼중 앱 분리로 인해 배포, 모니터링, 장애 대응의 복잡성이 3배로
증가했을 가능성이 있지만 이에 대한 언급이 없다. 12GB 관리 메모리를
할당하는 days 앱의 인프라 비용, K8s 환경에서의 리소스 경합, 그리고
세 앱 간의 정합성 보장 메커니즘에 대한 논의가 있었다면 더 완결적이었을
것이다.

### Native Savepoint 교착 상태의 일반화 가능성

Changelog State와 Native Savepoint의 상호작용 버그는 이 글에서
가장 가치 있는 발견 중 하나다. 그러나 이것이 Flink의 알려진 이슈인지,
특정 버전에 국한된 것인지, 커뮤니티에 보고되었는지가 명시되지 않았다.
이런 종류의 발견은 프레임워크 전체 사용자에게 영향을 미치므로,
업스트림 기여 여부가 궁금하다.

## 인사이트

### State를 진실의 원천으로 삼는 패러다임 전환

이 글의 가장 근본적인 설계 결정은 Redis를 캐시나 주 저장소가 아닌
단순한 투영(projection)으로 격하시킨 것이다. 전통적인 실시간 시스템에서
Redis는 읽기 성능을 위한 핵심 컴포넌트로 취급된다. 그러나 토스 팀은
Flink State를 유일한 진실의 원천으로 삼고, Redis는 서빙 레이어를 위한
물화 뷰(materialized view)에 불과하다는 위치를 부여했다.

이 패러다임은 CQRS(Command Query Responsibility Segregation) 패턴과
정확히 일치한다. 쓰기 모델(Flink State)과 읽기 모델(Redis)을 분리하고,
이벤트 스트림을 통해 동기화한다. 기존 삼중 구조가 실패한 이유는
진실의 원천이 Head, Mid, Tail에 분산되어 정합성 유지 비용이
기하급수적으로 증가했기 때문이다.

이 접근은 다른 실시간 집계 시스템에도 적용 가능한 일반 원리를
제시한다. 캐시를 진실의 원천으로 삼는 순간, 캐시 무효화와 정합성이라는
컴퓨터 과학에서 가장 어려운 문제 중 하나에 직면한다. State를 진실의
원천으로 삼고 캐시를 투영으로 격하시키면, 정합성 문제는 구조적으로
해소된다.

### 병목은 코드가 아니라 저장소 엔진의 물리적 특성에 있다

세 앱의 병목이 Write Buffer 압력, Filter Block I/O, 체크포인트 I/O로
각각 다르다는 관찰은 중요한 교훈을 담고 있다. 애플리케이션 레벨의
로직은 동일한데, 시간 윈도우 크기라는 단 하나의 변수가 완전히 다른
성능 특성을 만들어낸다.

이것은 LSM-tree 기반 저장소 엔진의 물리적 특성이 애플리케이션 성능을
지배한다는 사실을 보여준다. Write Buffer가 차면 L0 파일이 누적되고,
L0 파일이 임계치를 넘으면 Write Stall이 발생한다. Filter Block이
캐시에 올라가지 않으면 매 조회마다 디스크 I/O가 발생한다. SST 파일이
너무 많으면 Compaction이 I/O를 독점한다. 이 모든 것은 애플리케이션
코드와 무관한, 저장소 엔진의 내부 역학이다.

현대 분산 시스템에서 성능 튜닝은 점점 더 저장소 엔진의 물리적 이해를
요구한다. RocksDB가 Flink, Kafka Streams, CockroachDB, TiKV 등
다양한 시스템의 기반으로 사용되는 만큼, RocksDB의 LSM-tree 역학에
대한 이해는 특정 프레임워크를 넘어서는 범용적 기술이 되고 있다.
이 글에서 보여준 수준의 튜닝은 결국 RocksDB 자체에 대한 깊은
이해에서 비롯된 것이다.

### 프레임워크의 문서화되지 않은 행동이 진짜 기술 부채다

Flink가 파티션 필터 활성화 시 bloom-filter 설정을 덮어쓰는 행동,
Native Savepoint와 Changelog의 상호작용으로 인한 `materializedId`
추적 손상 — 이런 문서화되지 않은 행동(undocumented behavior)은
프레임워크 의존도가 높은 시스템에서 가장 위험한 기술 부채다.

오픈소스 프레임워크를 사용할 때 대부분의 팀은 문서와 API 계약을
신뢰한다. 그러나 이 글이 보여주듯, 프로덕션 규모에서는 문서화된
행동과 실제 행동 사이의 괴리가 드러난다. 소스 코드를 읽어야만
발견할 수 있는 이런 행동들은 테스트로도 잡기 어렵다. 소규모
데이터에서는 문제가 드러나지 않고, 프로덕션 규모에서만 발현하기
때문이다.

이는 프레임워크 선택의 숨겨진 비용을 드러낸다. Flink를 선택하면
Flink의 API뿐 아니라 RocksDB의 내부 역학, JVM의 메모리 관리,
K8s의 리소스 제한까지 이해해야 한다. 프레임워크가 제공하는
추상화는 정상 범위에서는 유효하지만, 극한 상황에서는 깨진다.
토스 팀이 CANONICAL Savepoint 포맷으로 우회한 것은 해결이 아닌
회피이며, 이 근본 버그가 수정되지 않는 한 다른 팀도 같은 함정에
빠질 수 있다. 프레임워크 사용자 커뮤니티에 이런 발견을 공유하는
것이 생태계 전체의 기술 부채를 줄이는 길이다.
