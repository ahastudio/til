# 아키텍트가 알아야 할 커넥션 풀의 기본과 함정

원문: <https://www.youtube.com/watch?v=uhMhv8yGAyM>

## 요약

커넥션 풀(Connection Pool)은 데이터베이스 연결을 미리 생성해 두고 재사용하는 기법이다.
연결 생성에는 TCP 핸드셰이크, 인증, 세션 초기화 등의 비용이 따르며,
요청마다 새 연결을 만들면 지연이 누적된다.
풀은 이 비용을 선불로 지불해 두고, 런타임에는 대여(borrow)와 반납(return)만 반복한다.

Spring Boot의 기본 커넥션 풀 구현인 HikariCP(히카리CP)는 Java 생태계에서 가장 빠르다고 알려진 구현체다.
`maximumPoolSize`(기본값 10), `connectionTimeout`(기본값 30s), `maxLifetime`(기본값 30분)이 핵심 파라미터이며,
이 값들을 데이터베이스와 애플리케이션의 특성에 맞게 조정하는 것이 아키텍트의 역할이다.

리틀의 법칙(Little's Law, L = λ × W)은 커넥션 풀 크기를 수학적으로 추론하는 출발점을 제공한다.
L은 동시에 필요한 커넥션 수, λ는 초당 쿼리 요청 수, W는 쿼리 한 건의 평균 처리 시간이다.
초당 100건의 쿼리가 평균 10ms 걸린다면 이론적 필요 커넥션은 1개이지만,
실제로는 분산과 피크 부하를 고려해 더 크게 설정해야 한다.

PostgreSQL은 연결당 독립 프로세스를 생성하는 구조(process-per-connection)라
커넥션 수가 늘어날수록 운영체제 레벨의 컨텍스트 스위칭 비용이 커진다.
이 특성이 PostgreSQL에서 커넥션 풀 설정이 특히 중요한 이유이며,
PgBouncer 같은 서버 사이드 풀러를 추가 계층으로 두는 이유이기도 하다.

함정은 대부분 “더 많으면 좋다”는 직관에서 비롯된다.
풀 크기가 너무 크면 DB 서버가 수백 개의 프로세스를 동시에 유지하며 과부하 상태에 빠진다.
풀 크기가 너무 작으면 `connectionTimeout` 초과 예외가 터지며 서비스가 중단된다.
커넥션 누수(Connection Leak)는 반납되지 않은 연결이 풀을 서서히 고갈시켜
증상이 점진적으로 나타나는 탓에 진단이 어렵다.

## 분석

### 리틀의 법칙이 실무에서 부정확해지는 이유

리틀의 법칙은 안정 상태(steady state)의 시스템에 성립하는 수학적 법칙이다.
λ와 W가 일정하다는 가정 하에서만 L을 정확히 예측한다.
그러나 프로덕션 트래픽은 균등하지 않다.
점심 시간 피크, 배치 작업과의 충돌, 슬로우 쿼리 발생 시 W의 급격한 증가 등이 실제 환경에서 끊임없이 발생한다.

이 법칙을 “정답”이 아닌 “하한선 추정 도구”로 사용하는 것이 올바른 접근이다.
계산 결과에 안전 계수(safety factor)를 곱하고, 실제 모니터링 지표(pool active connections, wait time)로 검증해야 한다.
공식 자체보다 λ와 W를 정확히 측정하는 관측 체계가 더 중요하다.

### PostgreSQL 아키텍처와 커넥션 비용의 비대칭성

MySQL/MariaDB는 스레드 기반(thread-per-connection) 구조라 연결 비용이 상대적으로 낮다.
PostgreSQL은 프로세스 기반(process-per-connection) 구조라 연결당 메모리와 운영체제 자원 소모가 크다.
이 차이는 커넥션 풀 크기 설정에서 데이터베이스를 선택할 때 반드시 고려해야 할 아키텍처 제약이다.

PostgreSQL 공식 문서는 `max_connections` 기본값을 100으로 제시하며,
이를 무분별하게 높이는 것을 권장하지 않는다.
pgBouncer나 pgPool-II 같은 미들웨어가 “애플리케이션 ↔ DB” 사이에서 연결을 중개하면,
애플리케이션에는 커넥션이 충분해 보이지만 DB에는 소수의 연결만 유지된다.
이 분리가 클라우드 환경에서 수평 확장 시 특히 유효하다.

### 스레드 풀과 커넥션 풀의 결합도

Spring Boot 애플리케이션에서 HTTP 요청을 처리하는 스레드 풀(Tomcat 기본 200 스레드)과
DB 커넥션 풀은 직접 연결되어 있지 않지만 강하게 상호작용한다.
스레드가 200개인데 커넥션 풀이 10개라면, 190개의 스레드는 커넥션을 기다리며 블로킹된다.
반대로 커넥션 풀이 200개인데 DB가 그 부하를 감당하지 못하면 쿼리 지연이 길어지고,
리틀의 법칙에 따라 필요한 커넥션 수가 다시 늘어나는 양성 피드백 루프가 형성된다.

이 결합도는 아키텍처 설계 시 두 풀을 독립적으로 튜닝하는 것이 아니라
시스템 전체의 처리량과 지연 목표를 기준으로 함께 설계해야 함을 시사한다.

## 비평

### 멤버십 영상이라는 포맷이 지식 공유의 구조를 왜곡한다

이 영상은 유료 멤버십 전용 콘텐츠로, 커넥션 풀이라는 기술 주제를 다루고 있다.
멤버십 포맷은 제작자에게 지속 가능한 수익을 제공하지만,
기술 지식의 유통 방식에 구조적 왜곡을 만든다.

커넥션 풀 설정 실수로 인한 장애는 특정 산업이나 규모의 팀에만 국한되지 않는다.
소규모 스타트업, 1인 개발자, 학생 프로젝트에서도 동일하게 발생한다.
접근 비용을 지불할 수 없는 개발자는 동일한 실수를 반복하게 되고,
이는 해당 지식이 얼마나 중요한지와 무관하다.

지식의 중요성이 높을수록 제한적 유통의 비용도 커진다는 점에서,
아키텍처 판단에 직결되는 핵심 개념들을 멤버십 뒤에 두는 선택은 재고할 여지가 있다.

### 단일 스택 예시가 일반화 가능성을 제한한다

영상의 태그는 HikariCP, Spring Boot, PostgreSQL로 구성된다.
이 조합은 Java 백엔드 생태계에서는 표준적이지만, 전체 백엔드 개발 지형의 일부일 뿐이다.
Django + psycopg2, Node.js + pg 풀, Go의 `database/sql` 풀은 같은 원칙을 다른 방식으로 구현한다.

리틀의 법칙이나 프로세스-per-커넥션 문제는 스택에 무관한 개념이지만,
HikariCP 파라미터 이름(`maximumPoolSize`, `connectionTimeout`)에 집중하는 설명은
Java 개발자 이외의 청중에게는 노이즈로 느껴질 수 있다.
개념 계층과 구현 계층의 분리가 명확할수록 더 많은 아키텍트에게 유효한 영상이 된다.

### 성능 수치 없는 튜닝 가이드의 한계

커넥션 풀 튜닝 영상에서 가장 유용한 정보는 “어떤 파라미터를 어떻게 바꿨을 때 지표가 어떻게 변했는가”다.
그러나 멤버십 영상 설명만으로는 실제 벤치마크나 사례 수치가 포함되어 있는지 확인할 수 없다.

수치 없는 튜닝 지침은 “무엇을 할지”는 알려주지만 “얼마나 할지”는 여전히 불투명하다.
`maximumPoolSize`를 기본 10에서 얼마로 올려야 하는가에 대한 답은
서비스 특성과 실측 데이터 없이는 근거 있게 제시하기 어렵다.
이 지점에서 영상의 실질적 가이던스는 원리 설명에서 멈출 가능성이 높다.

## 인사이트

### 커넥션 풀 고갈은 장애의 원인이 아니라 증상이다

커넥션 풀이 소진되어 타임아웃이 터졌을 때, 직관적 반응은 풀 크기를 늘리는 것이다.
그러나 풀 고갈의 1차 원인이 슬로우 쿼리라면, W가 커진 것이므로 L도 늘어난다.
풀 크기를 늘려도 슬로우 쿼리가 해결되지 않으면 더 커진 풀도 결국 고갈된다.

진짜 개입 지점은 커넥션 풀 크기가 아니라 W를 줄이는 것, 즉 쿼리 최적화나 인덱스 추가다.
이 사실은 알기 쉽지만, 장애 중 “당장 살아나야 한다”는 압박 아래서는 쉽게 무시된다.
아키텍트가 해야 할 일은 이 함정을 팀이 반복하지 않도록 사후 분석 문화와 관측 체계를 만드는 것이다.

커넥션 풀 지표(`hikaricp.connections.active`, `hikaricp.connections.pending`)를
Prometheus나 Micrometer로 수집하는 체계가 없으면,
고갈은 터진 뒤에야 보인다.
지표 수집은 튜닝에 앞서 반드시 선행되어야 할 인프라다.

### 리틀의 법칙은 병목 탐지 프레임워크다

리틀의 법칙의 진짜 가치는 커넥션 풀 크기 계산이 아니라 병목 위치를 추론하는 데 있다.
시스템의 어느 계층이든 L = λ × W를 적용할 수 있다.
커넥션 풀, 스레드 풀, HTTP 클라이언트 풀, 심지어 큐 기반 시스템까지 모두 이 공식의 범위 안에 있다.

특정 계층에서 L이 포화 상태인데 λ가 낮다면 W가 문제다(슬로우 처리).
λ가 높은데 L이 낮다면 처리는 빠르지만 트래픽이 폭증할 여지가 있다.
이 분석을 각 계층에 반복 적용하면 시스템 전체의 병목 지도를 그릴 수 있다.

단일 계층(커넥션 풀)에서만 이 법칙을 사용하는 것은 도구를 절반만 쓰는 것이다.
이 프레임워크를 내재화한 아키텍트는 APM 대시보드를 볼 때 수치의 나열이 아닌 흐름의 병목으로 읽는다.

### 서버리스와 컨테이너 환경에서 커넥션 풀은 재설계를 요구한다

전통적인 커넥션 풀은 단일 프로세스가 오래 살아있다는 전제에서 동작한다.
AWS Lambda나 컨테이너 오토스케일링 환경에서는 인스턴스가 수초 내에 생성되고 소멸한다.
인스턴스마다 `maximumPoolSize`개의 커넥션을 열면, 100개 인스턴스에 풀 크기 10이면 1,000개의 연결이 PostgreSQL에 동시에 열린다.

이 상황에서 전통적 커넥션 풀은 오히려 문제의 원인이 된다.
AWS RDS Proxy, Supabase의 Supavisor, PlanetScale 등은 이 구조적 문제를 해결하기 위해 등장했다.
서버리스 환경의 DB 연결 관리는 서버 사이드 커넥션 풀러를 기본 전제로 설계해야 한다.

아키텍처 결정의 타임라인을 고려하면, 처음부터 서버사이드 풀러를 도입하는 비용보다
스케일 후에 기존 직접 연결 방식을 교체하는 비용이 훨씬 크다.
“지금은 스케일이 작으니 나중에”라는 선택이 나중에 수백 시간의 마이그레이션 작업으로 돌아온다.

### 커넥션 풀 설정은 개발자 개인이 아닌 팀의 합의 사항이어야 한다

커넥션 풀 파라미터는 대부분 `application.properties`나 환경 변수에 있다.
리뷰를 받지 않거나, 기본값 그대로 프로덕션에 배포되거나,
성능 이슈가 터진 후 급하게 한 명이 수정하는 방식으로 관리되는 경우가 많다.

이 파라미터들은 단순한 설정값이 아니라 아키텍처 결정이다.
변경 시 DB 연결 수, 피크 처리량, 장애 전파 방식이 달라진다.
팀이 이 값을 ADR(Architecture Decision Record)이나 런북(Runbook)으로 문서화하지 않으면,
설정의 이유는 사라지고 숫자만 남는다.

숫자만 남은 설정은 “왜 10인가?”라는 질문에 아무도 답하지 못하는 상태를 만든다.
그 결과, 장애 시 근거 없이 100으로 올리거나, 과도한 보수주의로 기본값을 유지하거나,
두 극단 사이에서 팀이 표류하게 된다.
커넥션 풀 설정을 팀의 지식으로 만드는 것이 아키텍트의 역할이다.
