# murr — ML/AI 워크로드용 서브 밀리초 캐시

<https://github.com/murrdb/murr>

## 소개

murr는 ML/AI 추론 워크로드를 위해 Redis를 대체하도록 설계된 RocksDB 기반 NVMe/S3 캐시다.
배치 파이프라인과 추론 앱 사이의 데이터 서빙 레이어로 동작한다.
Parquet 형식으로 입력받고 Arrow-Flight gRPC로 출력한다.
Rust로 작성되었고(코드베이스의 90.5%), Apache 2.0 라이선스로 공개되어 있다.

적합한 사용 사례는 "데이터가 무겁고 테이블 형태이며, 읽기가 배치로 이루어지고,
비용이 중요할 때"로 명확히 정의된다.

## 아키텍처

**계층형 스토리지**: 자주 사용하는 데이터는 메모리에, 덜 사용하는 데이터는 디스크에,
복제는 S3에 보관한다.

**제로 카피 와이어 프로토콜**: NumPy 배열, Pandas DataFrame, PyTorch 텐서로
직접 변환할 수 있어 변환 오버헤드가 없다.
Arrow IPC(Inter-Process Communication)로 데이터를 직렬화해
열 형식(columnar storage) 기반 네이티브 배치 읽기/쓰기를 지원한다.

**무상태(Stateless) 설계**: 모든 상태를 S3에 유지해 노드 복구 시
자체 부트스트래핑이 가능하다.
운영 복잡성이 줄어든다.

**희소 컬럼 최적화**: 사용하지 않는 컬럼은 0바이트를 차지한다.
packed-blob 방식과 다른 점이다.

### 지원 기능

- 프로젝션 푸시다운(projection pushdown)으로 데이터 전송 최소화
- Python 우선 지원 — Numpy, Pandas, PyTorch 통합
- 배치 읽기/쓰기 네이티브 지원

### 적합하지 않은 사용 사례

OLTP 워크로드, 분석, 범용 캐싱, 피처 스토어 거버넌스 기능이 필요한 경우는
적합하지 않다고 명시되어 있다.

## 성능

100M 행, 10개 float32 컬럼, 1000개 키 조회 기준 벤치마크:

- p50 지연: ~268마이크로초 vs Redis ~815마이크로초
- packed-blob 읽기: Redis 대비 약 3배 빠름
- Feast 스타일 HSET 연산: Redis 대비 약 12배 빠름
- RAM 사용량: 해시 기반 구현 대비 약 1/3

## 라이선스 및 상태

Apache 2.0.
GeekNews 등록 시점(2026년 6월 14일)에 xguru가 직접 소개했으며
게시 직후여서 댓글은 없는 상태다.

## 분석

### 피처 스토어의 서빙 계층 문제를 직접 타깃한다

ML 파이프라인에서 피처 스토어는 훈련 데이터 준비와 추론 시 피처 조회 두 가지 역할을 한다.
훈련 데이터 준비는 대용량 배치 처리가 핵심이고, 추론 시 피처 조회는 저지연이 핵심이다.
이 두 요구사항은 종종 같은 스택으로 해결하기 어렵다.

Feast 같은 기존 피처 스토어의 HSET 기반 서빙은 범용적이지만 배치 ML 워크로드에 최적화되지 않았다.
murr은 "배치 읽기 + 테이블형 데이터 + 비용 효율"이라는 교집합에 특화함으로써 이 틈새를 공략한다.

### Arrow IPC와 제로 카피가 성능의 핵심이다

Arrow IPC 선택은 성능 이상의 의미를 가진다.
Apache Arrow는 데이터 분석 생태계에서 사실상의 표준 메모리 형식이 되어가고 있다.
Pandas, PyTorch, DuckDB, Spark 모두 Arrow를 지원하거나 Arrow 변환을 지원한다.

murr이 Arrow IPC를 내부 직렬화 형식으로 사용하면, 데이터를 ML 프레임워크에 전달할 때 변환 오버헤드가 없다.
이것이 "제로 카피 와이어 프로토콜"의 실질적 의미다 — 저장 형식과 연산 형식이 일치한다.

### 무상태 설계와 S3 의존성

S3를 상태 저장소로 사용하는 무상태 설계는 운영 부담을 줄이는 현명한 선택이다.
클러스터 장애 시 S3에서 상태를 복구할 수 있어 Kubernetes 환경에서 특히 유리하다.

## 비평

### 벤치마크의 선택적 조건

성능 수치는 설득력 있지만 내부 벤치마크다.
100M 행, 10개 float32 컬럼, 1000개 키 조회라는 벤치마크 설정은 murr에 유리한 조건이다.
float32 컬럼 데이터에 최적화돼 있어 문자열 피처가 많은 경우 이점이 다를 수 있으며,
실제 프로덕션 ML 파이프라인은 훨씬 다양한 데이터 타입과 쿼리 패턴을 포함한다.
특히 추천 시스템에서는 user ID, item ID 같은 문자열과 범주형 피처가 많은데,
이런 경우의 성능이 제시되지 않는다.
벤치마크의 최선 시나리오와 실제 사용 환경의 평균 시나리오 사이의 차이를 독립적으로 측정해야 한다.

### 마이그레이션 복잡성

Redis와의 비교에서 프로토콜 차이도 고려해야 한다.
Arrow-Flight gRPC는 Redis 프로토콜보다 클라이언트 통합이 복잡하다.
기존 Redis 클라이언트를 그대로 재사용하기 어렵고, 마이그레이션 경로가 명확하지 않다.
현재 Redis를 feature store로 사용하는 팀이 murr로 전환하려면 클라이언트 코드를 다시 작성해야 한다.
이 전환 비용이 성능 향상으로 정당화되는지는 각 팀의 상황에 따라 다르다.

### RocksDB와 읽기 집중 워크로드

RocksDB를 스토리지 엔진으로 선택한 것은 LSM 트리 기반의 높은 쓰기 처리량을 얻기 위함이지만,
이 선택에는 트레이드오프가 있다.
LSM 트리는 읽기보다 쓰기에 유리한 구조이며,
읽기 집중 워크로드에서는 B-트리 기반 스토리지보다 성능이 낮을 수 있다.
ML 추론에서 feature serving은 일반적으로 쓰기보다 읽기가 훨씬 많다.
RocksDB 선택이 이 읽기 집중 패턴에 최적인지는 별도의 분석이 필요하다.

### S3 의존성의 비용 구조

S3를 상태 저장소로 사용하는 무상태 설계는 운영 편의성을 높이지만 비용 구조를 바꾼다.
S3 API 호출 비용과 데이터 전송 비용이 Redis Cluster 운영 비용보다 낮은지는 사용량에 따라 달라진다.
대규모 ML 파이프라인에서 S3 요청 수가 많아지면 예상치 못한 비용이 발생할 수 있다.
"무상태"라는 장점이 S3 의존성이라는 새로운 종속을 만든다는 점도 고려해야 한다.

프로젝트가 GeekNews에 등록된 시점에 댓글이 없었다는 것은 커뮤니티 검증이 아직 이루어지지 않았음을 의미한다.
오픈소스 초기 단계에서 아직 실사용 사례나 독립적 성능 검증이 없다는 점은,
이 도구를 프로덕션에 도입하려는 팀에게 중요한 리스크 요소다.
Apache 2.0 라이선스로 공개돼 있다는 것은 코드를 검토하고 직접 평가할 수 있다는 뜻이지만,
그 평가 비용 자체가 작지 않다.

### Rust 선택이 Python 중심 ML 생태계에서 가져올 통합 복잡성

murr은 Rust로 작성되었지만 Python 우선 지원을 내세운다.
PyO3 같은 Python-Rust 바인딩 레이어가 이 간극을 메우지만,
이는 배포, 빌드, 의존성 관리 측면에서 순수 Python 라이브러리보다 복잡성을 추가한다.

ML 엔지니어링 팀이 Rust 빌드 환경을 관리하는 것은 흔한 일이 아니다.
AWS Lambda, Google Cloud Functions 같은 서버리스 환경에서의 배포 복잡성도 고려해야 한다.
Python 우선 지원은 사용 경험의 문제이고, 배포 복잡성은 운영 경험의 문제인데
후자에 대한 문서화가 중요해질 것이다.

### 피처 스토어 거버넌스 부재

murr이 명시적으로 배제한 "피처 스토어 거버넌스" 기능 — 피처 카탈로그, 데이터 계보, 접근 제어, 모니터링 — 은
엔터프라이즈 ML 파이프라인에서는 필수 요구사항인 경우가 많다.

서빙 성능이 뛰어나더라도, 이 거버넌스 기능의 부재는 규제 환경이나 큰 조직에서의 채택을 어렵게 만든다.
"서빙 레이어 특화"라는 포지셔닝이 오히려 채택 범위를 좁힌다.

## 인사이트

### 범용 도구의 특수 목적 분화와 운영 비용

ML 추론에서 Redis를 사용하는 것은 범용 도구를 특수 목적에 맞추는 것이다.
컬럼 형식 배치 데이터에 최적화된 캐시 레이어는 feature serving 병목을 해소하는 다른 접근법을 제시하며,
이 방향의 탐색 자체는 의미 있다.
ML 인프라가 성숙해지면서 Redis처럼 수평적으로 사용되던 도구들이
각 사용 사례에 최적화된 전문 도구로 분화되는 추세는 계속될 것이다.

대형 언어 모델 추론 비용이 주목받는 동안,
ML 파이프라인의 다른 비용 — 피처 서빙, 임베딩 저장, 배치 데이터 접근 — 은 상대적으로 덜 주목받았다.
murr이 Redis 대비 1/3의 RAM으로 유사한 성능을 낸다는 것은, 규모가 커질수록 의미 있는 비용 절감이 된다.
특화된 도구가 범용 도구보다 특정 워크로드에서 훨씬 경제적일 수 있다는 원칙이
ML 인프라에서도 구체화되고 있다.

### Apache Arrow 생태계와 데이터 레이어 표준화

Apache Arrow 생태계(pandas, PyTorch, numpy)와의 네이티브 통합은
ML 엔지니어링 스택에서 점점 더 중요한 통합 지점이 되고 있다.
직렬화·역직렬화 없이 데이터를 전달하는 zero-copy 아키텍처는 단순한 성능 최적화를 넘어,
데이터 이동의 개념 자체를 바꾸는 접근법이다.
이 방향이 feature serving 인프라의 표준이 될 수 있는지는
murr처럼 이 가설을 실제 구현으로 테스트하는 프로젝트들이 쌓여가면서 밝혀질 것이다.

Arrow가 메모리 내 데이터의 공통 언어가 되면, 서로 다른 시스템 간의 데이터 이동 비용이 줄어든다.
저장, 서빙, 처리, 훈련이 모두 Arrow 기반이면,
파이프라인의 각 단계는 데이터를 이해하기 위해 역직렬화/재직렬화가 필요하지 않다.

### 자체 벤치마크 전에 할 일

이 도구를 평가하는 팀에게 가장 실용적인 조언은,
자신의 피처 데이터 특성(데이터 타입 비율, 읽기/쓰기 비율, 동시 요청 수)으로
직접 벤치마크를 재현하는 것이다.
내부 벤치마크를 신뢰하기 전에 자신의 조건에서 수치를 확인하는 것이
인프라 결정에서 항상 올바른 접근법이다.
