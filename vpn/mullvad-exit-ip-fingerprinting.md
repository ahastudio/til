# Mullvad VPN 출구 IP 핑거프린팅 취약점과 완화 조치

원문: <https://mullvad.net/en/help/exit-ip-vpn-servers-mitigation-rollout>

HN 토론: <https://news.ycombinator.com/item?id=48269580> (420점, 91개 댓글)

## 요약

2026년 5월, Mullvad VPN은 외부 연구자가 보고한 서버 간 출구 IP 핑거프린팅 취약점을
공개했다. 사용자가 VPN 서버를 전환할 때 같은 기기임을 외부 관찰자가 식별할 수 있는
취약점이었다. 5월 20일 블로그 포스트로 공개된 후, 완화 조치 적용 현황을 실시간으로
추적하는 상태 페이지(5월 25일 기준)가 운영 중이다.

취약점의 기술적 원인은 WireGuard 키와 터널 주소, 출구 IP 사이의 결정론적
연관 관계에 있다. WireGuard는 각 피어를 공개 키로 식별하고, 이 키에서 내부
터널 IP 주소가 파생된다. 서버가 출구 IP를 할당할 때 터널 주소를 기반으로
각 서버의 IP 범위에서 비례적으로 유사한 위치에 있는 주소를 선택했다.
결과적으로 서버 A에서 사용자의 출구 IP가 범위의 30% 지점에 있다면,
서버 B로 전환해도 출구 IP가 그 서버 범위의 약 30% 지점에 위치했다.
이 패턴을 관찰하면 두 연결이 같은 기기에서 왔음을 상관관계로 추적할 수 있었다.

완화 조치는 출구 IP 할당을 무작위화하는 것이다. 터널 주소와 출구 IP 사이의
결정론적 연관을 끊어, 동일 사용자가 다른 서버에서 사용하는 출구 IP 또는
같은 서버에 있는 다른 사용자의 출구 IP와 상관관계를 추론할 수 없게 만든다.
2026년 5월 25일 기준으로 13개 서버에 완화 조치가 적용됐다.
대상에는 호주(멜버른, 시드니), 캐나다(몬트리올), 독일(프랑크푸르트),
핀란드(헬싱키), 프랑스(파리), 아일랜드(더블린), 노르웨이(오슬로),
스웨덴(스톡홀름), 미국(달라스, LA, 뉴욕, 솔트레이크시티)이 포함된다.

Mullvad는 이 취약점을 “중간 수준(medium)” 심각도로 분류했다.
실제 악용이 어려운 이유가 있다. 공격자는 서버 전환 전후의 트래픽을
동시에 관찰할 수 있어야 하고, 여러 서버의 IP 범위 매핑 정보가 있어야 한다.
일반적인 대역 외 관찰자에게는 높은 장벽이지만, 다수의 출구 노드를 운영하거나
광범위한 인터넷 트래픽을 수집하는 국가 수준의 행위자에게는 현실적 위협이다.

## 분석

### WireGuard의 설계 원칙과 핑거프린팅의 충돌

WireGuard는 단순성과 성능을 위해 설계됐다. 공개 키 기반 피어 식별은
OpenVPN의 인증서 체계보다 단순하고, Noise 프로토콜 프레임워크 기반의
핸드셰이크는 빠르다. 그러나 이 단순성은 각 피어가 고정된 공개 키로
일관되게 식별된다는 것을 의미한다. 서버가 클라이언트를 구분해야 할 때,
이 공개 키에서 파생된 값은 결정론적이다.

Mullvad의 취약점은 이 결정론적 특성이 의도치 않게 노출된 사례다.
출구 IP 할당 알고리즘이 내부적으로는 합리적인 설계 — 터널 IP를 기반으로
출구 IP를 결정 — 를 따랐지만, 그 알고리즘이 외부 관찰자에게 정보를 누출했다.
설계자가 “내부 구현”으로 생각한 것이 실제로는 “외부 신호”가 됐다.

### 프라이버시 시스템에서 결정론과 랜덤성의 역할

프라이버시 보호 시스템에서 결정론적 동작은 적이다.
결정론적이라는 것은 입력이 같으면 출력이 같다는 것이고,
이것은 관찰자가 충분한 샘플을 수집하면 입력을 역추론할 수 있다는 의미다.
암호학에서 이 원칙은 잘 알려져 있다.
AES-ECB 모드가 같은 평문 블록을 항상 같은 암호문 블록으로 변환해
패턴이 노출되는 것이 대표적 예다.

Mullvad의 완화 조치 — 출구 IP 무작위화 — 는 이 원칙의 적용이다.
랜덤성을 도입하면 관찰자가 수집한 샘플이 아무리 많아도
입력(터널 IP)과 출력(출구 IP)의 관계를 추론할 수 없다.
랜덤성은 단순한 구현 선택이 아니라 프라이버시 보장의 수학적 기반이다.

### 완화 조치 롤아웃의 투명성 관리

Mullvad는 실시간 상태 페이지를 운영해 어느 서버에 완화 조치가 적용됐는지 공개한다.
이 투명성은 보안 커뮤니케이션의 모범 사례다. 사용자가 자신이 연결하는 서버의
보호 상태를 알 수 있고, 완화되지 않은 서버를 임시로 피할 수 있다.

그러나 투명성은 동시에 공격자에게도 정보를 준다.
어떤 서버가 아직 취약한지 실시간으로 알 수 있다면,
공격자는 취약한 서버에 집중할 수 있다. 이 딜레마에서 Mullvad는
사용자 신뢰를 선택했다. 이것은 가치 판단이다.
사용자가 위험을 알고 선택할 수 있는 것이, 취약점을 숨긴 채 롤아웃을 완료하는 것보다
낫다는 판단이다.

## 비평

### 강점: 책임 있는 공개와 단계적 대응의 모범

Mullvad의 대응 방식은 보안 취약점 처리의 모범에 가깝다.
외부 연구자의 보고를 받아 내부 검증 후 공개 블로그 포스트를 작성했고,
완화 조치 적용 현황을 실시간 추적 페이지로 공개했다.
이 접근법은 사용자에게 정확한 위험 정보를 제공하면서 기술적 수정이 진행 중임을 보여준다.
“우리가 발견했을 때 이미 고쳤습니다”가 아니라
“외부 연구자가 발견했고, 우리는 이렇게 대응하고 있습니다”라는 솔직함이 돋보인다.

m132는 이 신속한 대응에 놀라움을 표한다.[^m132]
“사용자로서 나는 이미 스스로 대처할 준비를 하고 있었다.” 현대 기업의 느린 대응에 익숙한 사용자에게
Mullvad의 빠른 공개가 예상 밖이었다는 것이다. 이 반응 자체가 기업들이 얼마나 투명성을
기본값으로 제공하지 않는지를 역설적으로 보여준다.

### 한계: “중간 심각도” 분류의 맥락 의존성

취약점의 심각도를 “중간”으로 분류한 것이 모든 사용자에게 동일하게 적용되지 않는다.
저널리스트, 인권 활동가, 내부 고발자처럼 국가 수준의 감시를 피해야 하는 사용자에게
이 취약점은 실존적 위협이다. 국가 수준 행위자는 이 공격을 실행할 능력이 있다.
“중간 심각도”는 일반적 위협 모델에서는 적절하지만, 고위험 사용자에게는
즉각적 서버 전환 중단을 권고했어야 할 수도 있다.

### 보완이 필요한 관점: 재연결 동작과 사용자 행동 패턴

취약점이 서버 “전환” 시 발생한다는 점에서, 얼마나 자주 사용자가 서버를 전환하는지,
또는 자동 재연결이 서버를 변경하는지가 실제 노출 범위를 결정한다.
Mullvad 앱이 연결 불안정 시 자동으로 다른 서버에 재연결한다면,
사용자가 의도적으로 서버를 바꾸지 않아도 취약점에 노출될 수 있다.
이 자동 동작과 취약점의 교차점에 대한 설명이 없다.

Cider9986은 Mullvad 브라우저를 사용하면 이 문제가 발생하지 않는다고 지적한다.[^Cider9986]
브라우저는 WireGuard를 사용하지 않고 내장 프록시를 통해 동작하며, 각 사이트마다 다른 IP를 제공하는
랜덤 모드도 있다. VPN 클라이언트와 브라우저 확장의 아키텍처 차이가 취약점 노출 여부를 가른다는 점이다.

## 인사이트

### VPN 출구 IP는 “익명” 주소가 아니다 — 할당 알고리즘이 신원을 만든다

VPN의 핵심 프라이버시 보장은 실제 IP를 출구 IP로 대체한다는 것이다.
그러나 이 보장이 유효하려면 출구 IP와 사용자 사이에 연관이 없어야 한다.
Mullvad의 취약점은 이 연관이 알고리즘적으로 존재했음을 보여준다.
출구 IP를 “익명” 주소로 만드는 것은 단순히 실제 IP를 숨기는 것이 아니라
출구 IP 자체가 다른 식별 정보와 연관되지 않도록 보장하는 것이다.

이것은 프라이버시 시스템 설계의 근본적 원칙과 연결된다.
익명성은 정보의 단순한 치환이 아니라 정보 사이의 연관 고리를 끊는 것이다.
Tor가 다중 홉 라우팅을 사용하는 이유가 여기 있다. 단일 홉으로는
출구 노드 운영자가 원래 IP와 목적지를 동시에 알 수 없지만
연관 패턴이 존재할 수 있다. 다중 홉은 이 연관을 수학적으로 끊는다.

Mullvad의 완화 조치가 랜덤화인 것도 같은 원리다.
랜덤성은 연관을 정보 이론적으로 끊는 가장 강력한 방법이다.
무작위로 할당된 출구 IP는 어떤 관찰 데이터를 모아도 원래 터널 주소를
역추론할 수 없게 만든다. 이것이 “완화”가 아니라 “제거”다.

### 오픈소스 VPN 프로토콜의 투명성은 양날의 검이다

WireGuard는 오픈소스 프로토콜로, 설계와 구현이 공개돼 있다.
이 투명성이 Mullvad의 취약점을 외부 연구자가 발견할 수 있게 했다.
WireGuard의 키-터널IP 관계를 이해하는 연구자가
Mullvad의 출구 IP 할당 방식을 분석해 상관관계를 발견했다.

이것은 “시큐리티 스루 오브스큐리티(security through obscurity)”의 실패를
반대 방향에서 보여주는 사례이기도 하다. 독점 VPN 프로토콜을 사용했다면
같은 취약점이 있어도 외부 연구자가 발견하기 어려웠을 것이다.
그러나 동시에 악의적 행위자도 발견하기 어려웠을 것이다.
오픈소스 프로토콜은 선량한 연구자와 악의적 공격자 모두에게 정보를 준다.

이 딜레마의 결론은 여전히 오픈소스가 낫다는 것이다.
독점 시스템의 취약점은 발견되지 않는 것이 아니라 늦게 발견되고 조용히 악용된다.
오픈소스 시스템의 취약점은 빠르게 발견되고 공개적으로 수정된다.
Mullvad의 이번 대응이 그 차이를 보여준다.
취약점 발견부터 공개, 완화 조치까지의 과정이 투명하게 진행됐다.

### VPN 롤아웃 상태 공개는 새로운 보안 커뮤니케이션 표준을 만든다

보안 취약점 공개의 전통적 접근은 이분법적이다.
패치 전에는 침묵하고, 패치 후에 공개한다.
이유는 명확하다 — 패치 전 공개는 공격자에게 정보를 준다.
그러나 이 접근법에는 비용이 있다. 패치를 기다리는 동안 사용자는
자신이 취약한 상태인지 알 수 없고, 위험 회피 행동을 취할 수 없다.

Mullvad의 접근법 — 취약점 공개 후 서버별 완화 상태를 실시간 추적 —
은 이 딜레마에 대한 중간 경로다. 취약점의 존재는 공개하되,
어느 서버가 아직 취약한지도 공개해 사용자가 행동할 수 있게 한다.
이 접근법이 성공하려면 완화 조치가 빠르게 적용돼야 한다.
상태 페이지가 공격자에게 취약한 서버 목록을 제공하는 기간을
최소화해야 한다.

이 새로운 커뮤니케이션 패턴은 클라우드 서비스의 장애 상태 페이지와 닮았다.
AWS Status Page, GitHub Status처럼 현재 상태를 실시간으로 공개하는 것은
인프라 서비스의 표준이 됐다. 보안 완화 현황을 같은 방식으로 공개하는 것이
VPN 서비스의 새로운 기준이 될 수 있다. 사용자를 정보를 받을 자격이 있는
성인으로 대우하는 것이다. “우리가 알아서 할 테니 걱정하지 마세요”가 아니라
“현재 상황은 이렇고, 우리는 이렇게 대응하고 있습니다”라는 투명성이다.

### 프라이버시 누출은 주로 기능들의 교차점에서 발생한다

WireGuard의 키-주소 매핑도, Mullvad의 출구 IP 할당도 각각은 합리적인 설계다.
문제는 두 시스템이 만나는 교차점에서 발생했다. 내부 시스템의 결정론적 특성이
외부 관찰 가능한 신호로 변환됐다.

이 패턴은 프라이버시 취약점의 보편적 구조다.
브라우저 핑거프린팅도 각 특성 — 화면 해상도, 폰트 목록, 플러그인 — 은
개별적으로 식별 정보가 아니다. 그러나 조합하면 사실상 고유한 핑거프린트가 된다.
타이밍 공격도 마찬가지다. 메모리 접근 시간은 무해하지만,
캐시 히트와 미스의 패턴에서 암호화 키가 노출될 수 있다.

이 구조적 패턴이 주는 교훈은 프라이버시 시스템의 보안 검토가
컴포넌트 단위가 아닌 데이터 흐름 단위로 이루어져야 한다는 것이다.

어떤 정보가 누출되는가”를 물어야 한다.
이 질문은 더 어렵고, 시스템 전체에 대한 이해를 요구한다.
