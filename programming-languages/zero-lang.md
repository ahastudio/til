# Zero — 에이전트를 위한 프로그래밍 언어

<https://zerolang.ai/>

HN 토론: <https://news.ycombinator.com/item?id=48193539> (20점, 7개 댓글)

## 소개

Zero는 Vercel Labs가 개발 중인 실험적 프로그래밍 언어다.
“에이전트가 처음부터 주 사용자로 가정된 환경에서 프로그래밍 언어가 어떤 모습일 수 있는지를 탐구”하는 것이 목표다.
현재 pre-1 상태이며, 문법과 API는 계약이 아니라 실험이다.

## 설계 원칙

Zero의 핵심 설계 원칙은 네 가지다.

**학습 가능성(Learnability)**: 작은 문법 표면적(small surface area)과 규칙적인 문법, 특수 케이스 최소화로 에이전트가 작업 중 점진적으로 학습할 수 있게 한다.

**표준 라이브러리 우선(Standard Library Focus)**: 광범위한 표준 라이브러리 API를 제공해 에이전트가 외부 의존성을 탐색하는 비용을 줄인다.

**도구 검사 가능성(Tool Inspectability)**: 컴파일러가 진단(diagnostics)과 수리 계획(repair plans)을 구조화된 데이터로 노출해 에이전트가 자체적으로 코드를 점검하고 수정할 수 있게 한다.

**명시적 설계(Explicit Design)**: 문법적 편의(syntactic convenience)보다 명시적 패턴을 우선한다. 인간의 편의보다 에이전트의 가독성을 목표로 한다.

## CLI

설치는 단일 명령어로 이루어진다.

```bash
curl -fsSL https://zerolang.ai/install.sh | bash
```

릴리스 바이너리 크기는 289바이트, 컴파일러는 약 800KB 수준으로 매우 작다.

## 주요 기능

에이전트가 코드를 작성하면 컴파일러가 오류를 구조화된 데이터 형태로 반환한다.
에이전트는 이 데이터를 파싱해 스스로 수정 방향을 결정할 수 있다.
기존 언어에서 컴파일러 오류는 인간이 읽는 텍스트 형태였지만, Zero는 에이전트가 파싱 가능한 형태로 설계됐다.

명시적 효과(explicit effects) 가시성, 에이전트 가독 도구 출력, 작업당 하나의 명확한 경로(one obvious path per task), 레거시 호환성 없음이 방향성 목표로 제시되어 있다.

## 분석

### 에이전트를 1등 시민으로 설계하는 언어

기존의 모든 프로그래밍 언어는 인간 개발자를 위해 설계됐다.
에이전트가 언어를 사용할 때 겪는 어려움, 즉 예외적 문법, 암묵적 규칙, 인간 친화적 오류 메시지는 에이전트에게 불필요한 오버헤드다.
Zero는 이 전제 자체를 뒤집는다.
“에이전트가 주 사용자”라는 선언은 단순한 마케팅이 아니라 언어 설계의 모든 결정에 영향을 준다.

[GeekNews 댓글](https://news.hada.io/topic?id=29780)에서 한 독자는 “괄호 없는 리습(Lisp)인데요?”라고 언급했다.
리습의 S-표현식이 프로그램을 데이터로 다루는 동형성(homoiconicity)에서 나온 것처럼, Zero도 에이전트가 코드를 데이터로 처리하기 쉬운 구조를 추구하는 방향성이 있을 수 있다.

### Vercel Labs의 포지셔닝

Vercel은 Next.js와 프론트엔드 배포 플랫폼으로 알려져 있다.
그 Vercel이 프로그래밍 언어를 실험하는 것은 에이전트 기반 개발 패러다임에 대한 장기 베팅으로 읽힌다.
에이전트가 코드를 생성하고 실행하는 환경에서 Vercel의 배포 인프라가 기반이 된다면, 언어 레이어까지 수직 통합하는 그림을 그려볼 수 있다.

## 비평

### Pre-1 경고의 무게

Zero는 “today's syntax and APIs are not a contract”라고 명시하며 보안 취약점도 예상된다고 경고한다.
이것은 솔직한 프로젝트 상태 공개지만, 동시에 실제 사용 가능성이 매우 제한적임을 의미한다.
에이전트를 위한 언어라고 하지만, 에이전트가 실제로 프로덕션에서 사용할 수 없는 언어는 그 의미가 제한된다.

실사용 장벽만이 문제가 아니라는 지적도 있다.
nperez19는 Zero가 기존 언어 대비 설득력 있는 차별점을 제시하지 못하고 있다고 비판한다.[^nperez19]
토큰 소비를 약간 줄이고 오류 출력을 약간 개선하는 것은 생태계 전환을 정당화하기에 부족하다.
언어가 채택되려면 성능, 보안 취약점 감소, 유지보수성 같은 측면에서 측정 가능한 이득이 먼저 제시되어야 한다는 것이다.

### 생태계 부재 문제

[GeekNews 댓글](https://news.hada.io/topic?id=29780)에서 지적된 것처럼, Rust·Go·Zig와 유사한 측면이 있으나 대규모 프로젝트 적용성과 생태계는 미지수다.
에이전트가 외부 라이브러리를 검색하는 비용을 줄이기 위해 표준 라이브러리를 넓힌다는 접근은 좋지만, 결국 현실 세계의 요구는 표준 라이브러리 범위를 벗어나기 마련이다.

에이전트가 Zero를 제대로 쓸 수 있는지 자체를 의심하는 시각도 있다.
sometimelurker는 에이전트가 가장 잘 다루는 언어는 사전 학습 데이터에 가장 많이 등장한 것들,
즉 Python과 Bash라고 지적한다.[^sometimelurker]
Zero처럼 새로 만들어진 언어는 사전 학습 데이터가 사실상 없어 에이전트 친화적 설계만으로는 부족할 수 있다.
그러나 이에 대한 반론도 있다.
kandros는 Svelte의 주요 API 변경 사례에서 LLM이 새로운 패턴에 예상보다 빠르게 적응할 수 있음이 드러났다고 지적한다.[^kandros]
사전 학습 데이터의 비중이 절대적 제약이 아닐 수도 있다는 것이다.

## 인사이트

### 언어 설계의 다음 전선

지난 수십 년간 프로그래밍 언어의 발전 방향은 인간 개발자의 생산성, 안전성, 표현력을 중심으로 이루어졌다.
가비지 컬렉션, 타입 추론, 패턴 매칭 같은 기능들은 모두 “인간이 더 쉽게 쓸 수 있도록”의 방향이었다.
Zero가 제시하는 질문은 근본적이다. 에이전트가 주 소비자인 언어는 어떤 모습이어야 하는가?
이것은 단순한 문법의 문제가 아니라, 컴파일러가 누구(인간 vs. 에이전트)에게 피드백을 주는가의 문제다.

whattheheckheck은 이 질문에 근본적으로 다른 답을 제시한다.[^whattheheckheck]
에이전트를 위한 언어는 범용 프로그래밍 언어가 아니라 특정 도메인에 최적화된 선언적 DSL이어야 한다는 것이다.
사용자가 필요한 자유도를 정확히 표현할 수 있는 도메인 명세를 먼저 설계하고,
이를 기존 범용 언어로 컴파일하는 방향이 더 실용적일 수 있다.
이 관점은 Zero가 제시하는 "에이전트 우선 범용 언어" 경로 외에 또 다른 가능성을 시사한다.
에이전트에게 필요한 것이 새로운 언어인지, 아니면 도메인 특화 인터페이스인지는 아직 열린 질문이다.

### 구조화된 오류 메시지의 혁명적 가능성

현재 에이전트 기반 코딩에서 가장 많은 토큰을 소비하는 작업 중 하나는 컴파일 오류를 파싱하고 이해하는 과정이다.
인간이 읽는 텍스트 형태의 오류 메시지를 에이전트가 이해하려면 자연어 처리 과정이 필요하고, 이는 오류와 오류 해석 사이의 손실을 유발한다.
컴파일러가 구조화된 JSON이나 AST 형태로 진단을 제공한다면, 에이전트는 훨씬 적은 비용으로 정확하게 수정 방향을 결정할 수 있다.
이것이 Zero의 “Tool Inspectability”가 실현된다면, 에이전트 코딩 루프의 효율이 획기적으로 높아질 것이다.

### 에이전트 우선 설계가 인간에게도 좋을 수 있다

역설적으로 에이전트를 위한 언어 설계 원칙들, 즉 명시적 설계, 규칙적 문법, 특수 케이스 최소화는 인간 학습자에게도 유리한 특성이다.
파이썬이 “실행 가능한 의사코드(executable pseudocode)”라는 평가를 받으며 교육 분야에서 성공한 것처럼, Zero의 명시성은 초보자 친화적일 수 있다.
에이전트를 위한 설계가 인간의 인지 부하도 줄인다는 교차점을 발견한다면, Zero는 훨씬 넓은 사용자 기반을 얻게 될 것이다.
