# mq: Markdown을 위한 jq

<https://mqlang.org/>

<https://github.com/harehare/mq>

## 소개

mq는 Markdown 파일을 쿼리·필터·변환하는 커맨드라인 도구다. “마크다운 처리의
jq”를 표방하며, JSON 처리의 jq와 유사한 파이프 기반 문법으로 Markdown 문서를
조작한다. Rust로 작성됐고 2025년 2월에 처음 공개됐으며 2026년 6월 기준 GitHub
스타 818개를 기록 중이다.

기본 문법은 jq와 유사하다. `.h`는 모든 헤딩을, `.h2`는 h2 레벨 헤딩만 선택하고,
`|`로 연산을 체이닝한다. 예를 들어 `cat README.md | mq '.list | to_text()'`는
목록 요소를 텍스트로 변환한다. 세그먼트 셀렉터, 내장 함수, 커스텀 함수 확장이
지원된다. REPL 환경과 온라인 Playground도 제공한다.

핵심 사용 사례로 다섯 가지를 제시한다. LLM 워크플로우(프롬프트와 출력 처리),
LLM 입력 생성(구조화된 Markdown 콘텐츠 생성), 문서 관리(다수 파일에서 콘텐츠
추출·조직), 배치 처리(서브밀리초 속도로 다수 파일 변환), 콘텐츠 분석(특정 섹션이나
패턴 신속 추출)이다.

에코시스템은 외부 서브커맨드 구조로 확장된다. `mq-conv`는 PDF, DOCX, XLSX를
Markdown으로 변환하고, `mq-crawler`는 웹사이트에서 구조화된 콘텐츠를 Markdown으로
추출한다. `mq-tui`는 터미널 Markdown 뷰어, `mq-task`는 Markdown 파일 내 코드
블록을 실행하고, `mq-lsp`는 LSP 서버를 제공한다. `~/.local/bin/`에 `mq-<name>`
형태의 실행 파일을 두면 서브커맨드로 자동 등록된다. VSCode 확장, 타입 체커
(`mq-check`), 실험적 디버거(`mq-dbg`)도 포함한다.

## 분석

### “LLM 입력은 주로 Markdown”이라는 전제가 만드는 포지셔닝

mq의 홈페이지는 LLM 관련 사용 사례를 두 가지나 나열한다. “LLM 워크플로우”와
“LLM 입력 생성”이 각각 별개 항목으로 등장한다. 이 반복은 마케팅 선택이기도 하지만,
실제로 AI 시대에 Markdown의 지위가 달라졌다는 사실을 반영한다. LLM 프롬프트와
출력에서 Markdown은 사실상 표준 포맷이다. Claude, ChatGPT, Gemini 모두 Markdown을
기본 출력 형식으로 사용한다.

이 맥락에서 mq는 기존의 Markdown 도구와 다른 포지셔닝을 갖는다. 기존 도구들—Pandoc,
markdown-it, marked—은 Markdown을 HTML 또는 다른 포맷으로 변환하는 데 집중했다.
mq는 변환이 아니라 쿼리와 추출에 집중한다. LLM이 생성한 긴 Markdown 응답에서
특정 섹션만 뽑거나, 여러 Markdown 파일을 병합해 새 프롬프트를 만드는 것이 전형적인
사용 패턴이다. 이것은 “Markdown을 출력으로 소비”하던 기존 패턴에서 “Markdown을
입력으로 처리”하는 새 패턴으로의 이동을 반영한다.

### jq를 참조점으로 삼는 것의 전략적 선택

“jq의 Markdown 버전”이라는 포지셔닝은 기술적 설명이기도 하지만 커뮤니케이션
전략이기도 하다. jq는 개발자 생태계에서 “JSON 쿼리의 표준”으로 인정받으며,
파이프 기반 변환이라는 유닉스 철학을 JSON에 적용한 것으로 알려져 있다. mq는 이
인지도를 빌린다. “jq처럼”이라는 한 마디가 전체 설계 철학을 전달한다.

이 전략은 반복적으로 등장하는 패턴이다. yq(YAML), xq(XML), htmlq(HTML), dasel
(다중 포맷) 같은 도구들이 모두 jq 인터페이스를 자신의 포맷에 적용했다. 이 패턴의
공통점은 jq의 UX가 “필터 언어”라는 패러다임을 설득력 있게 구현했다는 데 있다.
파이프로 연결하고, 셀렉터로 선택하고, 함수로 변환하는 흐름은 새로운 도메인에도
자연스럽게 전이된다. mq는 이 계보에서 Markdown 대응물의 자리를 노린다.

### 단순 CLI 도구를 넘어 언어로 진화하는 설계

mq의 기능 목록에서 흥미로운 것은 REPL, LSP, 타입 체커, 디버거가 포함됐다는 점이다.
이것들은 단순한 필터 도구가 아니라 프로그래밍 언어에 요구되는 도구 인프라다. LSP
지원은 IDE에서 자동완성과 오류 검사를 제공한다는 의미고, 타입 체커는 쿼리에 타입
시스템이 있다는 뜻이다.

이 방향은 jq와의 차별화이기도 하다. jq는 강력하지만 복잡한 로직을 작성하면 디버깅이
어렵다는 것이 알려진 약점이다. mq는 이 약점을 디버거와 REPL로 보완하면서, 동시에
단순한 필터 이상의 표현력을 가진 언어로 발전하려는 의도를 보인다. `mq-task`로
Markdown 내 코드 블록을 실행할 수 있다는 것은 mq가 일종의 리터럿 프로그래밍(literate
programming) 실행 환경도 지향한다는 것을 시사한다.

### 서브커맨드 생태계가 만드는 확장성 모델

`~/.local/bin/`에 `mq-<name>` 형태의 실행 파일을 두면 mq 서브커맨드가 된다는
설계는 git의 확장 모델에서 차용한 것이다. git이 `git-<name>` 형태의 외부 명령어를
지원하면서 풍부한 플러그인 생태계가 형성된 것처럼, mq도 같은 메커니즘으로 확장성을
제공한다. 공식 서브커맨드(`mq-conv`, `mq-crawler`, `mq-tui`, `mq-task`)는
이 생태계의 씨앗이다.

이 모델의 강점은 핵심 도구를 작게 유지하면서 주변 기능을 분리할 수 있다는 점이다.
PDF·DOCX 변환이 핵심 바이너리에 포함되면 의존성이 복잡해지고 바이너리 크기가
커진다. 서브커맨드로 분리하면 필요한 기능만 설치할 수 있다. Rust로 작성된 핵심
도구의 서브밀리초 성능을 보장하면서, 외부 서브커맨드는 Python이나 다른 언어로도
작성 가능하다.

## 비평

### Markdown의 반구조성이 jq 패러다임과 충돌한다

jq가 JSON에서 우아하게 작동하는 이유는 JSON이 완전히 구조화된 포맷이기 때문이다.
JSON의 모든 요소는 명확한 타입과 경계를 갖는다. 셀렉터 `.foo.bar`가 무엇을 의미하는지
명확하다. Markdown은 근본적으로 다르다. Markdown은 반구조화된(semi-structured)
포맷이다. 헤딩, 목록, 코드 블록 같은 구조적 요소가 있지만, 그 안의 콘텐츠는
자유 형식 텍스트다.

이 차이는 실용적 결과를 낳는다. `mq '.h2 | to_text()'`로 h2 헤딩을 추출하는 것은
단순하다. 그러나 “특정 h2 섹션 아래에 있는 모든 목록 항목을 추출하라”는 요청은
트리 구조 탐색이 필요하고, Markdown의 섹션 경계는 다음 동급 헤딩이 나타날 때까지
암묵적으로 연장된다. jq에서 `.items[]`처럼 명확한 셀렉터가 Markdown에서는 문서
구조에 대한 상당한 가정을 내포한다. 문서마다 구조가 다르면 쿼리를 문서별로 다르게
작성해야 한다는 뜻이다.

홈페이지의 유일한 예시 코드 `cat README.md | mq '.list | to_text()'`가 얼마나
단순한 사례인지는 이 한계를 간접적으로 보여준다. 실제 문서 처리 요구사항은 이
예시보다 훨씬 복잡하다.

### LLM 워크플로우 포지셔닝은 실제 사용 패턴과 거리가 있다

mq가 강조하는 LLM 워크플로우 적용은 실제로 얼마나 일반적인가. LLM 출력을
프로그래밍적으로 처리하는 대부분의 워크플로우는 구조화된 JSON 출력을 사용하거나,
파이썬 라이브러리로 Markdown을 파싱한다. mq가 편리한 상황은 비교적 특수하다.
Markdown을 직접 파싱하기엔 과하지만 Python 스크립트를 쓰기엔 복잡한 중간 지점의
배치 처리 작업이다.

더 근본적으로, LLM 입출력 처리에서 병목은 Markdown 조작 속도가 아니다. LLM API
호출 자체가 수 초에서 수십 초가 걸린다. “서브밀리초 처리”는 전체 워크플로우에서
측정 불가능한 수준의 최적화다. 이 성능 강점은 대용량 문서 파일 배치 처리에서
의미가 있지만, LLM 워크플로우에서는 설득력 있는 차별점이 아니다.

### 서브커맨드 생태계가 형성되기 전까지 가치가 제한된다

mq의 잠재력 중 하나는 `mq-conv`, `mq-crawler` 같은 서브커맨드가 연결되는 통합
파이프라인이다. PDF를 Markdown으로 변환하고, 특정 섹션을 추출하고, LLM 프롬프트로
조립하는 흐름이 하나의 파이프라인으로 연결된다는 비전이다. 그러나 이 비전은 각
서브커맨드가 충분히 성숙하고, 서드파티 기여자들이 추가 서브커맨드를 만들고, 사용자
커뮤니티가 형성됐을 때 실현된다.

현재 상태에서 서브커맨드들은 공식적으로 존재하지만 홈페이지나 문서에서 상세한
사용 예시나 성능 보장이 불충분하다. 예를 들어 `mq-conv`가 PDF 변환에서 어떤 품질을
보장하는지, `mq-crawler`가 어떤 웹사이트 구조를 지원하는지 명확하지 않다.
생태계 전략은 옳지만, 현재는 생태계가 아니라 씨앗 단계다. 이 도구를 프로덕션에
도입하는 결정은 현재 기능이 아닌 미래 가능성에 베팅하는 것이다.

## 인사이트

### Markdown은 AI 시대의 중간 언어가 됐다

HTML이 웹의 표준 포맷이 된 것처럼, Markdown은 LLM 시대의 텍스트 처리 중간 언어로
자리잡고 있다. 이 전환은 Markdown의 설계 의도—인간이 읽기 쉬운 마크업—와 다소
다른 방향이다. Markdown이 인간과 LLM 사이의 인터페이스 언어로 굳어지면서, Markdown을
구조적으로 처리하는 도구에 대한 수요가 생긴다.

mq는 이 수요의 초기 신호다. Markdown을 쿼리 대상으로 보는 시각 자체가 새롭다.
기존 도구들은 Markdown을 “쓰는” 것 또는 “렌더링하는” 것으로 다뤘다. mq는 Markdown을
“읽고 추출하는” 대상으로 다룬다. 이 차이는 Markdown의 역할 변화를 반영한다. 문서
작성 포맷에서 데이터 포맷으로의 이동이다.

2차 효과로, mq가 보편화된다면 Markdown 문서가 쿼리 가능성을 고려해 작성되기
시작할 수 있다. “`.h2[계획]`으로 추출 가능하도록” 헤딩을 구조화하거나, 목록 형식을
통일하는 방향으로 문서 작성 관행이 변할 수 있다. 이것은 문서가 기계 가독성을
고려해 설계되기 시작하는, 스키마 없는 구조화의 시작점이 될 수 있다.

### jq-알라이크 패턴은 도메인 특화 필터 언어의 수렴 진화다

yq, xq, htmlq, dasel, mq가 모두 jq의 UX를 자신의 도메인에 적용한다는 것은
흥미로운 현상이다. 이것은 jq가 “필터 언어”의 적절한 UX를 찾았다는 것을 시사한다.
파이프로 연결하는 선형 변환, 점 표기법으로 경로를 지정하는 셀렉터, 내장 함수로
변환을 적용하는 구조—이 세 가지 조합이 다양한 포맷에 적용 가능한 범용 패턴이다.

이 패턴의 역사적 선례는 SQL이다. 관계형 데이터를 쿼리하는 방법으로 SQL이 표준화된
후, 비관계형 데이터베이스들이 “SQL-알라이크” 인터페이스를 제공하기 시작했다.
MongoDB의 aggregation pipeline, Spark SQL, BigQuery SQL이 모두 SQL의 정신을
가져왔다. jq는 구조화 데이터의 파이프라인 필터 언어 버전의 SQL인 셈이다.

그러나 이 수렴에는 위험이 있다. jq의 UX를 빌리면 익숙함을 얻지만, 해당 도메인에
더 적합한 추상화를 포기할 수 있다. Markdown의 섹션 기반 구조는 JSON의 트리 구조와
다르다. jq 패러다임을 강제하면 Markdown 고유의 구조를 자연스럽게 표현하는 데
제약이 생긴다. CSS 셀렉터가 HTML에 더 자연스러운 이유가 있듯, Markdown에는
jq보다 더 자연스러운 쿼리 모델이 있을 수 있다.

### 리터럿 프로그래밍의 부활과 mq-task의 위치

`mq-task`가 Markdown 파일 내 코드 블록을 실행하는 기능은 1980년대 Donald Knuth가
제안한 리터럿 프로그래밍(literate programming)의 재발명이다. 리터럿 프로그래밍은
코드와 문서를 하나의 파일에 공존시키는 방식으로, Knuth의 WEB 시스템, Emacs Org
Mode의 babel, Jupyter Notebook이 같은 아이디어를 각기 다른 방식으로 구현했다.

mq-task는 이 계보에서 독특한 위치를 차지한다. Jupyter는 웹 인터페이스 기반이고,
Org Mode는 Emacs 생태계에 종속된다. mq-task는 표준 Markdown 파일을 대상으로
커맨드라인에서 코드 블록을 실행한다. GitHub에 올려진 README.md의 예시 코드를
그대로 실행할 수 있다는 것이 잠재적으로 강력한 사용 사례다. 실행 가능한 문서라는
개념이 특수 포맷 없이 표준 Markdown으로 구현된다.

이 방향이 성숙하면 AI 에이전트가 생성한 Markdown 형식의 실행 계획을 mq-task로
직접 실행하는 패턴이 가능해진다. LLM이 작성한 “단계별 명령 가이드”가 곧 실행
가능한 스크립트가 된다. 이것은 mq가 단순한 쿼리 도구를 넘어 AI-human 협업의
실행 레이어가 될 수 있다는 것을 시사한다.

### 도구 생태계의 진입 장벽이 낮아지면서 반짝 도구가 증가한다

818개 스타, 13개 포크, 활발한 업데이트 주기—mq는 건강한 초기 성장을 보인다.
그러나 Rust로 CLI 도구를 만드는 것이 쉬워진 오늘, 비슷한 포지셔닝의 도구들이
계속 등장한다. mq가 이 경쟁에서 지속적으로 생존하려면 단순한 기능 목록이 아닌
차별화된 가치가 필요하다.

이 긴장은 모든 틈새 CLI 도구가 직면하는 트레이드오프다. 좁은 도메인에 집중하면
경쟁 진입은 쉽지만 사용자 기반이 작다. 도메인을 넓히면 사용자가 늘지만 jq, sed,
grep 같은 범용 도구와의 경쟁에 직면한다. mq가 “Markdown 처리의 표준 도구”가 되는
경로는 Markdown이 더 구조화된 데이터로 취급되는 문화적 변화와 함께해야 한다. 즉,
mq의 성공은 도구의 품질뿐 아니라 AI 시대에 Markdown이 어떤 역할을 갖는가라는
더 큰 질문의 답에 달려 있다.
