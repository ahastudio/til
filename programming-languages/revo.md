# Revo: 인간 공학 언어가 에이전트 시대를 만났을 때

<https://gills.pages.dev/revo/>

<https://github.com/if-not-nil/revo>

## 요약

Revo는 Zig로 작성된 동적 타입 프로그래밍 언어로,
저장소 소개 문구를 그대로 따르면 "프로그래밍의 즐거움(the joy of programming)"을 전면에 내세운다.
README는 기술 사양보다 사용 경험을 먼저 보여주는 구성이다.
예시 코드는 `let`, 파이프(`|>`), 패턴 매칭(`match`), 결과 타입 스타일(`(:ok, v)`, `(:err, e)`)을
짧게 제시하면서 언어가 지향하는 표현 방식을 전달한다.

핵심 기능으로는 파이프 기반 데이터 흐름, 태그 튜플 기반 오류 처리 패턴,
`spawn`과 `join` 및 채널(`chan`, `send`, `recv`)을 통한 동시성,
그리고 `test`/`suite` 문법으로 언어 내장 테스트를 제시한다.
특히 "블로킹 코드를 먼저 쓰고, 필요하면 `spawn`을 붙여 비동기화한다"는 메시지는
Revo가 동시성을 고급 이론보다 일상적 리팩터링 대상으로 다루려는 의도를 보여준다.

설치 측면에서 현재 바이너리 릴리스는 제공되지 않으며,
Zig 최신 안정 버전(README 기준 `0.16.0`)을 사용해 소스 빌드하는 방식을 안내한다.
POSIX와 Windows 설치 절차를 분리해 설명하며, 사용법 섹션에서는 CLI 옵션을 꽤 폭넓게 노출한다.
`-e`, `-i`, `-b`, `--test`, `--bench`, `--dis` 등 실행/테스트/벤치/디스어셈블리를
하나의 엔트리포인트에서 다루는 형태다.

Revo는 단순한 인터프리터를 넘어 C 임베딩 API도 제공한다.
README의 예시에서 `erevo_vm_create`, `erevo_compile`, `erevo_run`, `erevo_eval` 호출 흐름을 제시하며,
실패 시 `erevo_vm_last_error`로 오류를 확인하는 패턴을 보여준다.
또한 언어 서버 `revolt`를 저장소에 포함해 `revo --lsp`로 실행 가능하다고 명시한다.
진단, 정의로 이동, 참조 찾기, 심볼 검색 같은 기본 LSP 기능을 제공해
편집기 통합을 염두에 둔 프로젝트임을 드러낸다.

## 분석

### Revo의 핵심 주장은 "표현력"보다 "전환 비용 절감"에 있다

표면적으로 README는 "표현력이 좋은 언어"를 소개하는 문서처럼 보인다.
그러나 구조를 따라가면 Revo의 진짜 주장은 새 패러다임 제안보다
기존 개발 습관의 전환 비용을 낮추는 데 가깝다.
예를 들어 동시성 섹션에서 제시하는 슬로건은 "처음부터 비동기 모델로 설계하라"가 아니라
"일단 동작하는 블로킹 코드를 쓰고 나중에 `spawn`을 붙여라"다.

이 방식은 기능 자체의 참신함보다 채택 경로를 강조하는 설계다.
새 언어가 실패하는 주요 원인은 성능이나 문법 미학이 아니라,
개발팀이 기존 코드와 사고방식을 버리지 못한다는 점이다.
Revo는 이 문제를 정면 돌파하기보다 "기존 습관의 연장선에서 조금씩 이동"하도록 안내한다.
즉, 동시성도 테스트도 "별도 프레임워크 학습"이 아니라 문법 차원에서의 단계적 확장으로 제시된다.

결과적으로 Revo의 설득 포인트는 "이 언어가 원래 없던 것을 가능하게 한다"가 아니라
"지금 하던 일을 덜 마찰적으로 하게 한다"에 있다.
이는 초기 커뮤니티 확보 단계에서 유효한 전략이다.
반면 장기적으로는 "왜 Revo여야만 하는가"라는 질문에 더 강한 기술적 차별화 근거를
지속적으로 제시해야 한다는 과제를 동시에 남긴다.

### README는 언어 문법보다 실행 환경 통합을 먼저 상품화한다

많은 신생 언어 프로젝트는 문법 철학을 길게 설명한다.
Revo README는 반대로 빌드, REPL, 테스트, 디스어셈블리, LSP, C 임베딩이라는 실행 경로를 빠르게 펼친다.
이 구성은 "언어의 아름다움"이 아니라 "도구로서의 완결성"을 구매 포인트로 삼는 접근이다.

특히 `zig build` 기반 빌드 행렬, `--test`와 `--bench`의 단일 CLI 통합,
릴리스 빌드에서 LSP 번들 제공, 별도 `zig build lsp` 경로까지 분기해 둔 점은
운영 관점의 배려를 시사한다.
이는 Revo가 단순한 취미 문법이 아니라
일상적인 개발 루프(코드 작성-실행-테스트-편집기 피드백) 전체를 설계 대상으로 본다는 뜻이다.

이 맥락에서 C 임베딩 API는 언어 채택의 현실적 문을 연다.
조직은 새 언어를 "전면 도입"보다 "부분 삽입"으로 시험한다.
VM 생성, 코드 컴파일, 실행, 오류 조회가 명확한 C 함수 경계로 제공되면,
기존 시스템의 특정 구간에 Revo를 삽입해 ROI를 측정하기 쉬워진다.
README가 이 경로를 초기에 배치한 것은 기술적 기능 소개를 넘어 채택 시나리오를 의식한 편집이다.

### Revo는 "동적 언어"와 "시스템 언어 구현"의 긴장을 전략적으로 활용한다

Revo 자체는 동적 타입 언어인데 구현체는 Zig다.
이 조합은 우연이 아니라 포지셔닝의 결과로 읽힌다.
사용자는 스크립팅 수준의 빠른 피드백과 간결한 문법을 원하고,
구현자는 예측 가능한 성능과 배포 통제를 원한다.
Revo는 이 두 요구를 분리해 충족시키려 한다.

동적 언어는 보통 "생산성은 높고 제어성은 낮다"는 인식이 있다.
반대로 Zig 기반 구현, C API 노출, 디스어셈블리 옵션, 빌드 특성 스위치는 "제어성" 이미지를 끌어온다.
즉 Revo는 사용자 경험 층에서는 유연함을, 런타임/도구 층에서는 시스템 친화성을 주장한다.
신생 언어가 신뢰를 확보할 때 흔히 사용하는 전략적 이중성이다.

문제는 이 긴장이 시간이 갈수록 부채가 될 수 있다는 점이다.
동적 언어 사용자 커뮤니티는 문법 단순성과 런타임 관용성을 기대하고,
시스템 프로그래밍 커뮤니티는 성능 수치와 안정성 계약을 요구한다.
Revo가 양쪽을 동시에 만족시키려면 "언어 철학"과 "실행 플랫폼"의 우선순위를
릴리스마다 일관되게 정리해야 한다.

### 지금의 메시지는 "가능성 데모"에 강하고 "운영 약속"에는 신중하다

README는 예제 중심으로 언어의 감각을 잘 전달한다.
파이프, 패턴 매칭, 채널 루프, 테스트 블록은 Revo를 짧은 시간에 이해시키는 데 효과적이다.
그러나 운영 의사결정자 관점에서 중요한 릴리스 정책, 버전 안정성 계약,
하위 호환 기준, 에러 모델의 장기 계획은 상대적으로 얇다.

물론 이는 초기 단계 프로젝트에서 자연스럽다.
버전 표기가 `0.0.0`이고 nightly 릴리스가 병행되는 상태에서
강한 호환성 약속을 먼저 고정하면 개발 속도를 잃을 수 있다.
따라서 현재 문서 전략은 "먼저 써보게 만들고, 운영 안정성은 점진적으로 축적"하는 모델에 가깝다.

다만 실제 채택 확장은 이 경계에서 결정된다.
개인 개발자는 데모 완성도로 움직이지만 팀 도입은 운영 약속으로 움직인다.
Revo가 다음 단계로 가려면 지금의 매력적인 기능 데모를 유지하면서도
"어떤 변화가 언제 깨질 수 있고, 무엇은 지킨다"는 계약 문서화를 병행해야 한다.

## 비평

### "간결한 동시성" 서사는 실패 모드의 비용을 숨긴다

README의 동시성 메시지는 매우 매력적이다.
블로킹 코드를 먼저 작성하고 `spawn`을 붙여 비동기화하면 된다는 설명은 학습 진입 장벽을 크게 낮춘다.
그러나 이 설명은 동시성의 핵심 난점인 실패 전파, 취소(cancellation),
백프레셔(backpressure), 종료 순서 보장 문제를 의도적으로 뒤로 미룬다.

예제에서 `chan`으로 값을 주고받고 특정 값에서 루프를 `break`하는 패턴은
"동작 시연"으로는 충분하지만,
실제 서비스 코드에서는 고루틴/스레드 누수와 유사한 수명 관리 이슈가 즉시 발생한다.
누가 채널을 닫는지, 생산자가 먼저 종료했을 때 소비자는 어떻게 감지하는지,
부분 실패가 시스템 전체에 어떻게 전달되는지 같은 질문이 빠져 있다.

이 공백은 단순히 문서 분량의 문제가 아니다.
언어가 동시성을 "쉽다"고 브랜딩할수록 사용자 기대치는 높은 추상화를 전제한다.
그런데 실패 모드 모델이 약하면 사용자는 "쉬운 시작" 이후 급격한 복잡성 절벽을 만나게 된다.
결과적으로 "쉬운 동시성" 약속은 초기 온보딩 지표를 올릴 수 있지만, 장기 신뢰 지표를 해칠 위험이 있다.

Revo가 이 비판을 넘어서려면 기능 추가보다 운영 시나리오 문서가 먼저 필요하다.
예를 들어 취소 전파 패턴, 채널 종료 계약, 타임아웃 표준 관용구를 공식 예제로 제시하면
"쉽다"는 메시지가 마케팅 문구가 아니라 학습 경로로 전환된다.

### 테스트 내장 기능은 "있음"을 증명하지만 "검증 철학"은 아직 빈칸이다

`test`와 `suite` 문법 제공은 분명한 장점이다.
언어가 테스트를 1급 시민으로 다룬다는 신호이기 때문이다.
그러나 README 예시는 산술 검증 수준에 머물러 있고,
테스트 러너의 격리 모델, 결정론 보장, flaky 테스트 대응, 실패 재현성 전략 같은 검증 철학을 설명하지 않는다.

언어 채택에서 테스트는 문법이 아니라 계약이다.
팀은 테스트를 "작성할 수 있는가"보다 "신뢰할 수 있는가"를 본다.
예를 들어 동시성 코드 테스트에서 스케줄링 비결정성을 어떻게 통제하는지,
랜덤 시드를 고정할 수 있는지, 실패 시 최소 재현 로그를 어떤 형식으로 남기는지 같은 정보가 필요하다.

현재 문서는 "테스트 가능"을 보여주지만 "테스트 신뢰"를 증명하지 못한다.
초기 단계 프로젝트에서 이는 흔한 한계지만, Revo가 실사용을 겨냥한다면 반드시 보강해야 하는 지점이다.
특히 언어 수준 동시성을 강조하는 프로젝트일수록 테스트 도구의 결정론 설계는 선택이 아니라 필수다.

즉 문제는 기능 부재가 아니라 우선순위의 배치다.
지금처럼 문법 예시 중심 메시지를 유지하더라도,
최소한의 검증 설계 원칙을 README와 별도 문서로 연결하면
"테스트 블록이 있다"에서 "운영 가능한 테스트 체계가 있다"로 인식이 이동할 수 있다.

### LSP 번들 전략은 단기 편의성 대신 장기 결합도 리스크를 키운다

릴리스 빌드에 LSP를 번들하고 `revo --lsp`로 즉시 실행 가능하게 한 결정은 사용자 경험 측면에서 훌륭하다.
설치 직후 편집기 통합이 가능하다는 점은 신생 언어의 치명적 약점인 도구 생태계 부족을 빠르게 보완한다.

그러나 번들 전략은 런타임/컴파일러와 언어 서버의 릴리스 주기를 결합한다.
이는 단기적으로 버전 불일치 문제를 줄이지만,
장기적으로는 어느 한쪽의 변경 속도가 다른 쪽을 발목 잡는 구조를 만든다.
언어 서버는 IDE 프로토콜 변화와 편집기별 상호 운용성 이슈를 자주 맞고,
런타임은 안정성 중심으로 느리게 진화하는 경우가 많다.

두 축을 강하게 묶으면 "빠르게 고쳐야 하는 LSP 이슈"와 "조심스럽게 배포해야 하는 언어 코어"가
하나의 배포 파이프라인에서 충돌할 수 있다.
특히 소규모 팀일수록 이 결합도는 운영 부담을 키운다.
단순히 빌드 타깃을 분리하는 것으로 해결되지 않고, 호환성 매트릭스와 배포 정책까지 분리 설계가 필요하다.

따라서 현재 전략은 "초기 채택 가속"에는 맞지만 "확장 운영" 단계에서는 재검토가 필요하다.
최소한 LSP의 독립 배포 경로와 코어 런타임의 안정 채널을 병행하는 로드맵이 없다면,
편의성의 이익이 시간이 지날수록 유지보수 부채로 전환될 가능성이 높다.

## 인사이트

### 에이전트 시대에는 동적 언어의 가치 기준이 "작성 편의"에서 "경계 명확성"으로 이동한다

Revo는 인간이 빠르게 코드를 쓰기 좋은 문법을 제공한다.
하지만 AI 에이전트가 대량 코드를 생성하는 환경에서는 "쉽게 쓸 수 있는가"보다
"어디서 실패하고 무엇이 계약인지 명확한가"가 더 중요해진다.
동적 언어가 불리하다는 뜻이 아니라, 경쟁 지표가 바뀐다는 뜻이다.

에이전트는 반복 코드를 싫어하지 않는다.
오히려 사람보다 훨씬 많이 생성한다.
이때 시스템 품질을 가르는 것은 문법의 우아함이 아니라 경계의 엄격함이다.
모듈 간 데이터 계약, 오류 표면, 동시성 수명 규칙이 명시적일수록
에이전트 산출물의 검증 비용이 내려간다.

Revo가 이 전환을 선점하려면 타입 체계를 급격히 강화할 필요는 없다.
대신 인터페이스 계약 문법, 실행 시 진단 표준, 실패 전파 기본 규약을 언어 차원에서 정형화하면 된다.
즉 "동적"을 유지하되 "무계약"으로 보이지 않게 만드는 전략이 핵심이다.

이 관점은 다른 신생 언어에도 동일하게 적용된다.
에이전트 시대의 언어 경쟁은 "누가 더 짧게 쓰게 해주나"에서 "누가 더 싸게 검증되나"로 이동하고 있다.
Revo의 다음 성장 단계는 문법 혁신보다 검증 경제성 최적화에서 결정될 가능성이 크다.

### Revo의 C 임베딩은 "새 언어 도입"보다 "레거시 조정 레이어"로서 더 큰 파급효과를 낼 수 있다

README는 C API를 부가 기능처럼 배치하지만, 실제로는 전략적 자산이다.
기업 환경에서 새 언어 도입은 전면 전환보다 기존 시스템 내부에 "작은 실행 엔진"을 삽입하는 방식으로 시작된다.
이때 C ABI를 제공하는 언어는 거의 모든 플랫폼과 접점을 만들 수 있다.

흥미로운 2차 효과는 조직 정치의 변화다.
전면 도입 논쟁은 늘 크게 부딪히지만, 임베딩은 "위험이 작은 실험"으로 승인되기 쉽다.
한두 모듈에서 Revo 스크립팅으로 정책 엔진이나 워크플로를 외부화하면,
성능 병목이나 장애 패턴 데이터를 실제로 축적할 수 있다.
언어의 운명은 철학 논쟁이 아니라 이런 작은 승리의 누적으로 정해진다.

또한 임베딩 전략은 에이전트 운영과 결합될 여지가 크다.
에이전트가 규칙 코드를 생성해 Revo 스크립트로 배포하고,
호스트 애플리케이션은 C API 경계에서 샌드박싱/권한 제어를 수행하는 구조를 만들 수 있다.
이 경우 Revo는 "애플리케이션 언어"보다 "정책 실행 DSL 런타임"으로 먼저 확산될 수 있다.

즉 Revo의 실질 확장 경로는 사용자들이 기대하는 "새 메인 언어"가 아닐 수 있다.
오히려 기존 시스템을 건드리지 않으면서 변화 속도를 높이는 삽입형 런타임으로 자리 잡을 때,
더 빠르게 임계 사용자층을 형성할 가능성이 높다.

### LSP 내장과 단일 CLI 전략은 생태계 분산보다 "제품화된 언어" 모델로 수렴한다

Revo는 CLI 하나로 실행, 테스트, 벤치, 디스어셈블리, LSP 진입점을 묶는다.
이는 도구를 많이 제공한다기보다 "언어를 하나의 제품"으로 패키징한다는 신호다.
역사적으로 이 모델은 초기 성장에는 강하지만,
생태계가 커질수록 커뮤니티 확장성과 긴장 관계를 만든다.

생태계 주도 모델에서는 도구 다양성이 혁신을 만든다.
반면 제품화 모델에서는 일관성이 품질을 만든다.
Revo는 현재 후자에 가깝다.
소규모 핵심 팀이 개발 속도와 사용자 경험을 통제하기에 유리하지만,
외부 기여자가 도구 체인 일부만 독립적으로 발전시키기는 어렵다.

이 트레이드오프는 규모가 커질수록 선명해진다.
언어가 초기 채택을 넘어서려면 "공식 경로의 일관성"과 "비공식 확장의 자유"를 동시에 설계해야 한다.
예를 들어 플러그인 인터페이스, 표준화된 진단 포맷, 버전 호환 정책이 있어야
외부 도구가 안정적으로 공존한다.

Revo가 앞으로 마주할 중요한 선택은 기술 스택이 아니라 거버넌스 모델일 가능성이 높다.
지금의 강한 제품 통합을 유지하면서도,
어디까지를 코어로 고정하고 어디서부터를 생태계 자율성에 맡길지 명확히 선을 그어야 한다.
이 선을 잘 그으면 Revo는 "작지만 매끈한 언어"를 넘어 "확장 가능한 언어 플랫폼"으로 이동할 수 있다.
