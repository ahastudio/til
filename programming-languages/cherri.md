# Cherri - 애플 단축어를 코드로 작성하는 프로그래밍 언어

원문: <https://github.com/electrikmilk/cherri>

## 요약

Cherri는 Apple Shortcuts(단축어)를 텍스트 코드로 작성하고 실행 가능한 Shortcut 파일로
직접 컴파일하는 도메인 특화 언어(DSL)다.
기존에는 iOS/macOS의 시각적 편집기 안에서만 단축어를 만들 수 있었지만,
Cherri를 사용하면 일반적인 소프트웨어 개발 방식으로 단축어를 작성, 버전 관리, 모듈화할 수 있다.

Go와 Ruby에서 영감을 받은 문법을 사용하며, 타입 시스템(타입 추론 포함), 열거형(enum),
옵셔널 타입, 기본값, 함수 정의, 파일 포함(include)을 지원한다.
CLI 도구와 함께 VSCode 익스텐션, macOS 앱, 웹 플레이그라운드를 제공한다.
패키지 관리는 Git 저장소 기반으로 동작하며 자동 업데이트도 지원한다.

기존 iCloud 단축어를 `--import=` 플래그로 Cherri 코드로 변환할 수 있어,
기존 자산을 재활용하거나 분석하는 것도 가능하다.
실행 파일 크기를 최소화하도록 최적화되어 있으며, 디버깅을 위한 `--debug` 플래그는
스택 트레이스와 `.plist` 파일을 출력한다.

2022년 10월 시작된 프로젝트로, 2,000회 이상의 커밋이 누적된 활성 오픈소스 프로젝트다.
Go로 구현되었으며(전체 코드베이스의 99.5%), GPL-2.0 라이선스를 따른다.
Homebrew와 Nix를 통해 설치할 수 있다.

## 분석

### Apple Shortcuts의 한계를 코드로 극복

Apple Shortcuts는 강력한 자동화 플랫폼이지만, 시각적 블록 기반 편집기는
복잡한 로직을 표현하는 데 명확한 한계가 있다.
분기가 많거나 반복이 복잡한 단축어는 화면을 스크롤하며 블록을 추적해야 하고,
변경 이력을 관리하거나 팀과 협업하는 것도 어렵다.
Cherri는 이 문제를 텍스트 기반 개발 도구를 제공함으로써 해결한다.

### 반쪽짜리 부트스트래핑 구조

흥미롭게도 Cherri는 “절반이 부트스트래핑된(half-bootstrapped)” 아키텍처를 가진다.
대부분의 Shortcuts 액션이 Cherri 언어 자체로 정의되어 있어,
언어 자신이 자신의 표준 라이브러리를 정의하는 방식이다.
이는 언어 확장이 핵심 Go 코드를 수정하지 않고도 가능함을 의미하며,
커뮤니티 기여 가능성을 높이는 설계 선택이다.

### 서명 및 배포 메커니즘

컴파일된 Shortcut 파일을 실행하기 위해서는 Apple의 서명이 필요하다.
Cherri는 macOS 로컬 서명, HubSign, 또는 호환 가능한 서버를 통한 서명을 지원한다.
이는 Apple의 폐쇄적 생태계 안에서 오픈소스 도구가 작동하기 위해 필요한 현실적 타협이다.

## 비평

### 강점

실용적인 문제를 해결하는 도구다.
대규모 Shortcut 프로젝트를 유지보수해야 하는 개발자에게 버전 관리, 모듈화, 테스트 가능성은
시각적 편집기로는 얻을 수 없는 핵심 역량이다.
기존 단축어를 임포트하는 기능은 전환 비용을 낮추는 실용적 선택이다.
웹 플레이그라운드를 제공해 설치 없이 언어를 체험할 수 있는 점도 온보딩 경험을 개선한다.

### 약점

Apple 플랫폼에 완전히 종속된다.
Apple이 Shortcuts API를 변경하거나 서명 정책을 바꾸면 Cherri 전체가 영향을 받는다.
또한 현재 macOS 중심이며, iOS/iPadOS에서 직접 컴파일 워크플로우를 지원하지 않는다.
Apple 생태계 외부의 개발자에게는 학습 투자 대비 활용 범위가 제한적이다.

### 커뮤니티 규모의 한계

2022년부터 꾸준히 개발된 프로젝트이지만, 주로 단일 개발자가 주도하고 있다.
“포기된 경쟁 프로젝트들과 달리 안정적이고 활발히 유지된다”는 프로젝트의 자체 포지셔닝은
이 생태계가 아직 넓지 않음을 역설적으로 드러낸다.

## 인사이트

### 시각적 프로그래밍의 확장성 한계는 구조적이다

Apple Shortcuts, Zapier, n8n, Make(구 Integromat) 등 시각적 워크플로우 도구들은
진입 장벽을 낮추지만, 일정 규모 이상에서는 구조적 한계에 직면한다.
복잡성이 커질수록 시각적 표현은 인지 부하를 줄이기는커녕 오히려 늘린다.
Cherri의 존재는 이 한계가 단순한 사용자 불편을 넘어 실제 생산성 문제임을 증명한다.

흥미로운 것은, 시각적 도구가 충분히 강력해질수록 그것을 “코드로 작성하고 싶다”는 니즈도
함께 커진다는 점이다. Terraform이 클라우드 인프라를 코드로, dbt가 데이터 변환을 코드로
만든 것처럼, Cherri는 자동화 워크플로우를 코드로 만드는 흐름의 연장선에 있다.
IaC(Infrastructure as Code)에서 WaC(Workflow as Code)로의 전환이다.

이 패턴이 반복된다면, 시각적 도구의 다음 진화 단계는 항상 텍스트 기반 인터페이스를 향한다.
도구 제공자들은 시각적 편집기와 함께 코드 기반 SDK를 동시에 제공하는 전략을
더 일찍 채택해야 할 것이다.

### 도메인 특화 언어가 생태계 잠금에 대응하는 방식

Cherri는 Apple의 독점 형식(.shortcut 파일)을 오픈 텍스트 형식으로 변환한다.
이를 통해 사용자는 자신의 자동화 자산을 버전 관리하고, 공유하고, 포크할 수 있다.
이는 독점 플랫폼에 갇힌 사용자 자산을 해방시키는 DSL의 전통적 역할이다.

단, 이 해방은 부분적이다. Cherri 코드는 이식 가능하지만,
컴파일된 결과는 Apple 생태계 안에서만 실행된다.
진정한 이식성을 위해서는 Shortcuts 런타임 자체를 추상화해야 하는데,
그것은 Apple의 폐쇄적 API 정책 앞에서 현실적으로 불가능하다.
Cherri의 가치는 “플랫폼 탈출”이 아니라 “플랫폼 안에서의 개발자 경험 개선”에 있다.

### 단일 기여자 오픈소스의 지속가능성 문제

2,000회 이상의 커밋을 거의 혼자 쌓아온 프로젝트는 기술적으로 인상적이지만,
지속가능성 관점에서는 취약하다.
핵심 기여자가 프로젝트를 떠나면 유지보수가 멈추는 “버스 팩터 1” 문제가 있다.

Cherri 스스로 “포기된 경쟁 프로젝트”를 언급한 것은 이 리스크를 인식하고 있음을 보여준다.
사용자 입장에서 Cherri를 핵심 워크플로우에 도입할 때는 이 지속가능성 리스크를
평가해야 한다. 프로젝트가 성숙하려면 커뮤니티 기여를 촉진하는 거버넌스 구조,
문서화, 기여 가이드에 대한 추가 투자가 필요하다.
