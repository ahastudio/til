# IEEE Spectrum 2025 인기 프로그래밍 언어 순위

원문: <https://spectrum.ieee.org/top-programming-languages-2025>

HN 토론: <https://news.ycombinator.com/item?id=45354314> (294점, 457개 댓글)

## 요약

IEEE Spectrum이 발표한 2025년 프로그래밍 언어 인기 순위에서 Python이
Spectrum 기본 순위와 취업 시장(Jobs) 순위 모두 1위를 차지했다.
JavaScript는 2024년 3위에서 2025년 6위로 하락했다.
취업 순위에서는 SQL이 여전히 이력서에 유효한 기술로 자리를 유지한다.

이 보고서에서 가장 주목할 만한 사실은 순위 자체보다 측정 데이터의 변화다.
2025년 Stack Exchange에 올라온 주간 질문 수는 2024년 대비 22%에 불과했다.
프로그래머들이 공개 포럼 대신 Claude, ChatGPT 같은 LLM과의 비공개 대화로
기술 질문을 해결하기 시작했기 때문이다.
이 변화는 언어 인기를 측정하는 기존 방법론 자체의 신뢰도를 흔들었다.

IEEE Spectrum은 Google 검색 트래픽, Stack Exchange 질문 수,
학술 논문 인용, GitHub 오픈소스 저장소 활동을 합산해 순위를 산출한다.
그러나 AI 코드 어시스턴트가 일상화되면서 이 지표들이 실제 언어 사용량을
제대로 반영하는지 의문이 생겼고, 조직은 새로운 측정 지표 개발을 예고했다.

저자는 LLM이 어떤 언어로든 코드를 생성할 수 있게 되면서 프로그래머들이
특정 언어의 세부 문법에 덜 집착하게 될 것이라고 분석한다.
그 결과 프로그래밍 언어의 인기 순위가 “철도 궤간(railway track gauge)의
상대적 인기처럼 모호한 주제”가 될 수 있다고 전망한다.
미래 프로그래머의 역할은 알고리즘 선택, 아키텍처 설계, 시스템 인터페이싱으로
이동하고, 이는 코딩 부트캠프보다 컴퓨터과학 학위의 가치를 높일 수 있다고 본다.

## 분석

### Python 1위 고수의 구조적 이유

Python이 1위를 유지하는 것은 AI·ML 생태계와의 결합 때문이다.
TensorFlow, PyTorch, scikit-learn, Hugging Face Transformers 모두
Python을 기본 인터페이스로 사용한다.
AI 붐이 지속되는 한 Python의 순위는 쉽게 흔들리지 않는다.
LLM 자체가 Python 코드를 가장 많이 학습했기 때문에,
AI 코드 생성 환경에서도 Python은 기본 출력 언어로 선호된다.

### JavaScript 하락의 의미

JavaScript가 3위에서 6위로 떨어진 것은 단순한 순위 변동이 아니다.
Stack Exchange 질문 감소가 JavaScript 생태계에 더 크게 영향을 미쳤을 가능성이 있다.
JavaScript 개발자들은 상대적으로 LLM을 빠르게 채택하는 경향이 있고,
그만큼 공개 포럼 질문이 줄었을 수 있다.
실제 사용량은 여전히 압도적임에도 측정 지표상 하락한 것이라면,
이는 방법론의 한계를 보여주는 사례다.

### Stack Exchange 78% 감소의 함의

2025년 Stack Exchange 질문이 2024년의 22% 수준으로 줄었다는 데이터는
충격적이다.
이 변화는 몇 가지 가능성을 내포한다.
첫째, AI 어시스턴트가 Stack Overflow가 해결하던 질문의 상당 부분을 흡수했다.
둘째, 질문의 복잡도가 높아져 AI로 해결할 수 없는 것만 포럼에 올라오고 있다.
셋째, 개발자 커뮤니티의 지식 공유 채널 자체가 분산됐다.
어느 쪽이든, 이 변화는 오픈소스 지식 생태계에 심각한 영향을 미친다.

## 비평

### 강점: 방법론의 한계를 솔직하게 인정

이 보고서의 강점은 자신의 측정 방법이 흔들리고 있음을 공개적으로 인정한다는 점이다.
많은 랭킹 보고서가 방법론의 결함을 숨기거나 최소화하는 반면,
IEEE Spectrum은 “새로운 지표를 개발할 것”이라고 선언했다.
이는 보고서의 신뢰도를 오히려 높인다.
HN에서 pornel은 Google 검색 트래픽 지표의 한계를 구체적으로 지적했다.
Google 알고리즘 변화나 SGE(생성형 검색 경험) 도입이 언어 인기와 무관하게
검색량을 바꿀 수 있으며, TIOBE 지수처럼 이 방법론도 “가짜 정밀도(false
precision)”를 만들어낼 위험이 있다고 경고했다.[^pornel]

### 약점: 전체 순위 미공개

상위 1위(Python)와 6위(JavaScript), 그리고 SQL 정도만 언급되고
2~5위의 구체적인 언어와 순위가 전달되지 않는다.
“2025 인기 순위” 보고서라는 이름에 비해 실제 순위 정보가 부족하다.
독자가 원문 페이지의 인터랙티브 차트를 직접 확인해야 한다는 점은
접근성 측면에서 아쉽다.
HN에서 kstrauser는 HTML이 프로그래밍 언어로 분류된 것에 의문을 제기했다.
“HTML이 프로그래밍 언어라면, PHP 사용자보다 HTML 사용자가 더 많아야 하지
않느냐”고 반문하며, 무엇을 '프로그래밍 언어'로 볼 것인지에 대한
기준 자체가 모호하다고 지적했다.[^kstrauser]

### 약점: “철도 궤간” 비유의 과도한 비관

“프로그래밍 언어 인기가 철도 궤간처럼 모호해질 것”이라는 전망은
흥미로운 비유이지만 근거가 약하다.
철도 궤간은 표준화로 중요성이 사라졌지만, 프로그래밍 언어는 성능, 생태계,
패러다임 차이가 실질적인 결과 차이를 만든다.
Rust와 Python의 선택은 시스템 성능에 직접 영향을 미친다.
LLM이 문법을 대신 처리해준다고 해서 언어 선택의 중요성이 사라지지는 않는다.

### 약점: 신흥 언어에 대한 LLM 영향 단언

새로운 언어가 LLM 훈련 데이터 부족으로 성장하기 어렵다는 주장은
설득력이 있지만 반례가 있다.
Mojo, Carbon, Gleam 같은 언어들은 소규모 커뮤니티에서도 성장하고 있으며,
LLM이 코드 생성을 지원하기 시작하는 속도도 빨라지고 있다.
Claude나 GPT-4는 훈련 데이터가 적은 언어도 빠르게 학습하고 있다.

## 인사이트

### 지식 공유 플랫폼의 대체와 오픈 생태계의 위기

Stack Exchange 질문이 78% 줄었다는 사실은 프로그래밍 언어 순위보다
훨씬 중요한 신호다.
Stack Overflow는 단순한 Q&A 플랫폼이 아니라, 수십 년간 쌓인 프로그래밍
지식의 공개 데이터베이스였다.
이 데이터는 LLM 훈련에도 사용됐다.
그런데 지금은 개발자들이 LLM에게 질문하고, 그 답변은 비공개로 남는다.

이 순환 구조는 장기적으로 지식 생태계를 고갈시킬 수 있다.
LLM은 과거의 공개 지식으로 훈련됐지만, 지금 생성되는 지식은 공개되지 않는다.
미래의 LLM은 점점 더 빈약한 공개 데이터를 기반으로 훈련될 것이다.
Stack Overflow의 쇠퇴는 AI의 승리처럼 보이지만, 동시에 AI의 장기적 훈련
기반을 무너뜨리는 역설을 내포한다.

역사적으로 비슷한 패턴이 있었다.
오픈소스 운동은 소프트웨어 지식을 공개 공유함으로써 생태계 전체를 성장시켰다.
그러나 AI 어시스턴트가 지식 소비를 비공개 채널로 흡수하면,
공개 지식 생산의 인센티브가 약해진다.
“왜 Stack Overflow에 답변을 쓰는가”라는 질문에 대한 답이 점점 어려워진다.

### 측정 위기가 드러내는 인기 순위의 본질

프로그래밍 언어 인기 순위는 항상 “실제 사용량”이 아니라 “관찰 가능한 신호”를
측정했다.
Google 검색, Stack Exchange 질문, GitHub 저장소 활동은 모두 프록시(proxy)다.
AI 코드 어시스턴트가 이 프록시들을 교란하자, 측정 자체가 무의미해지기 시작했다.

이것은 측정의 역설(Goodhart's Law)의 변형이다.
“측정 대상이 된 지표는 좋은 지표이기를 멈춘다”는 법칙처럼,
언어 인기의 대리 지표들은 개발자 행동 변화에 의해 무력화됐다.
IEEE Spectrum이 새로운 지표를 개발하겠다고 밝힌 것은 이 문제를 인식했다는 의미다.
그러나 어떤 새로운 지표도 동일한 취약성을 가질 것이다.
AI가 어디서나 개입할 수 있게 된 환경에서 “자연스러운” 개발자 행동을
관찰하는 것은 점점 어려워진다.

근본적인 문제는 “인기”라는 개념 자체의 의미가 변하고 있다는 것이다.
LLM이 어떤 언어로든 코드를 생성할 수 있다면, “어떤 언어를 알고 있는가”보다
“어떤 언어의 코드를 LLM이 가장 잘 생성하는가”가 더 중요해질 수 있다.
그것은 훈련 데이터의 양과 질에 달린 문제이고,
결국 Python의 독주 체제를 강화하는 방향으로 작용한다.

### LLM이 언어 생태계를 고착시키는 역설

프로그래밍 언어의 인기는 역사적으로 유동적이었다.
COBOL에서 C로, C에서 Java로, Java에서 Python으로 이동하는 데 수십 년이 걸렸지만,
이 전환은 가능했다.
그러나 LLM이 훈련 데이터에 비례해 특정 언어를 더 잘 처리한다면,
새로운 언어가 생태계에서 자리를 잡기가 구조적으로 더 어려워진다.
HN에서 hackthemack은 이 역설을 정확히 짚었다.
"LLM은 현재 주류 언어들의 훈련 데이터로 가득 차 있기 때문에, 현재의 언어
지형을 수십 년 더 고착시킬 가능성이 있다"고 지적했다.[^hackthemack]

이 관점에서 보면 Python의 독주는 단순히 현재의 인기를 반영하는 것이 아니라,
AI가 매개하는 개발 환경에서 미래의 인기까지 사전에 결정하는 자기 강화 구조다.
LLM이 Python 코드를 가장 잘 생성하기 때문에 개발자들이 Python을 더 많이 쓰고,
더 많이 쓸수록 미래 LLM 훈련 데이터도 Python이 많아진다.
이 순환은 기술적 우열이 아니라 네트워크 효과로 언어의 운명을 결정한다.

### 언어 불가지론 시대와 컴퓨터과학 교육의 르네상스

저자가 “컴퓨터과학 학위가 코딩 부트캠프보다 가치 있어질 것”이라고 전망한 것은
주목할 만하다.
LLM이 문법과 API 호출을 대신 처리해준다면, 남은 것은 알고리즘, 자료구조,
시스템 설계, 수학적 추론이다.
이것들은 전통적으로 컴퓨터과학 학위에서 가르치는 내용이다.

이 전망이 맞다면, 지난 10년간 “코딩 부트캠프로 6개월 만에 개발자가 되자”는
트렌드는 역전될 수 있다.
“코드를 짜는 방법”보다 “무엇을 만들어야 하는지, 왜 이 방식이 더 좋은지”를
아는 것이 핵심 역량이 된다.
이는 소프트웨어 산업에서 “엔지니어”와 “코더”의 차이를 다시 벌리는 방향이다.

그러나 이 전망에는 반론도 있다.
역사적으로 추상화 수준이 높아질수록 더 많은 사람이 참여할 수 있었다.
어셈블리에서 C로, C에서 Python으로 이동하면서 프로그래머의 수는 늘었지,
줄지 않았다.
LLM이 코딩을 더 쉽게 만든다면, 더 많은 사람이 소프트웨어를 만들 수 있게 되고,
그중 일부는 더 깊은 역량을 쌓으려 할 것이다.
“부트캠프 대 학위”의 이분법보다, 양쪽 모두 성격이 변화하는 시나리오가
더 현실적일 수 있다.
