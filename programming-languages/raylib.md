# raylib: 즐겁게 게임을 만들기 위한 C 라이브러리

원문: <https://www.raylib.com/>

참고: <https://github.com/raysan5/raylib>

## 소개

raylib은 비디오 게임 프로그래밍을 즐기기 위해 설계된 단순하고 사용하기 쉬운 C 라이브러리다.
시각적 편집기나 디버그 도구 없이 순수하게 코딩만으로 게임을 만드는
“스파르타적 프로그래머(spartan-programmer)” 방식을 철학으로 삼는다.
Borland BGI 그래픽 라이브러리와 XNA 프레임워크에서 영감을 받았으며,
2013년부터 개발되어 현재까지 활발하게 유지되고 있다.
zlib/libpng 라이선스로 정적 링킹과 상업적 사용이 모두 허용된다.

C99 표준으로 작성되었고 외부 의존성이 없다.
필요한 모든 라이브러리를 번들로 포함하여
`git clone` 한 번으로 바로 빌드할 수 있다.
OpenGL 1.1/2.1/3.3/4.3과 ES 2.0/3.0을 하드웨어 가속으로 지원하며,
커스텀 OpenGL 추상화 레이어(`rlgl`)와 소프트웨어 렌더러(`rlsw`)도 제공한다.

## 주요 기능

### 그래픽

- 2D/3D 렌더링: 스프라이트, 모델, 빌보드, 하이트맵
- 골격 애니메이션: IQM, M3D, glTF 포맷
- 셰이더 시스템: 커스텀 셰이더 및 포스트프로세싱 효과
- PBR(물리 기반 렌더링) 재질
- VR 스테레오 렌더링

### 오디오

WAV, QOA, OGG, MP3, FLAC, XM, MOD 포맷 스트리밍 지원.

### 수학 모듈

`raymath` — 벡터(2D/3D/4D), 행렬, 쿼터니언 연산 라이브러리.

### `rlgl`

OpenGL 버전 차이를 추상화하는 별도 헤더 라이브러리.
OpenGL 1.1 API로 작성된 코드가 3.3이나 ES 2.0에서도 동작하도록 변환한다.

## 플랫폼 및 언어

**플랫폼**: Windows, Linux, macOS, FreeBSD, Raspberry Pi, Android, HTML5(WebAssembly), Chrome OS

**주 언어**: C(C99), 코드베이스의 85.9%

**언어 바인딩**: C#, Go, Lua, Python, Rust, Zig, Odin 등 70개 이상

## 학습 리소스

140개 이상의 코드 예제, 치트시트, 게임 템플릿, GitHub Wiki를 제공한다.
공식 API 문서 대신 치트시트와 코드 예제를 중심으로 한다.
“코딩을 배우는 가장 좋은 방법은 코드를 읽는 것”이라는 철학 때문이다.
Discord, Reddit, YouTube, Twitch, Patreon에서 커뮤니티가 활성화되어 있다.

## 분석

### “의존성 없음”의 설계 결정

raylib이 모든 의존성을 번들로 포함하는 결정은
현대 소프트웨어 개발의 주류와 역행한다.
대부분의 라이브러리는 의존성 관리자(npm, cargo, pip)를 통해
필요한 것을 끌어오는 방식을 따른다.
raylib은 이 접근법이 “시작의 마찰”을 높인다고 본다.
클론하고 빌드하면 즉시 작동하는 경험은
특히 초보자나 교육 목적에서 강력한 장점이 된다.

단점도 명확하다. 번들된 의존성의 보안 패치나 버그 수정이
raylib 릴리스 주기에 종속된다.
OpenSSL 같은 보안 크리티컬 라이브러리를 번들로 포함했다면
이 결정은 위험하지만, 그래픽·오디오·수학 라이브러리는
보안 취약점이 거의 발생하지 않으므로 raylib의 맥락에서는 합리적이다.

### Borland BGI와 XNA의 영감: 두 시대의 교훈

Borland BGI(Borland Graphics Interface)는 1980년대 DOS 환경의 그래픽 라이브러리로,
`initgraph()`, `line()`, `circle()` 같은 단순한 API로
그래픽 프로그래밍에 입문하게 해준 도구였다.
XNA는 2000년대 마이크로소프트가 인디 게임 개발을 위해 만든 프레임워크로,
고수준 추상화와 쉬운 시작을 강조했다.

raylib은 두 전통을 C99 시대에 재합성한다.
BGI의 단순성(“한 줄이면 선 하나 그리기”)과
XNA의 포괄적 게임 개발 지원(“입력, 오디오, 3D 모두 한 라이브러리에서”)을 결합한다.
이 계보는 raylib이 왜 “배우기 쉬운 게임 라이브러리”로 포지셔닝하는지를 설명한다.

### 70개 언어 바인딩의 의미

raylib이 C로 작성되었음에도 70개 이상의 언어 바인딩이 존재하는 것은
raylib의 API 설계가 FFI(외부 함수 인터페이스)에 친화적임을 보여준다.
C99 표준의 단순한 타입 시스템, 명시적 메모리 관리, 헤더 파일 기반 인터페이스는
다른 언어에서 바인딩을 작성하기 쉬운 조건을 만든다.

Rust의 `bindgen`, Python의 `ctypes/cffi`, Zig의 `@cImport`처럼
많은 언어가 C API를 직접 소비하는 메커니즘을 갖고 있다.
raylib이 C로 작성된 것은 단순한 성능 선택이 아니라
“모든 언어의 공통 인터페이스”로서의 C를 의도적으로 선택한 것이다.

## 비평

### 강점

교육적 접근법이 일관성 있다.
API 문서 대신 치트시트와 예제 코드를 중심으로 하는 결정은
“코드를 읽으며 배운다”는 철학과 일치한다.
140개 이상의 예제는 “이 기능이 어떻게 작동하는가”를
실행 가능한 코드로 바로 확인할 수 있게 한다.
외부 의존성 없이 즉시 시작할 수 있는 경험은
초보자와 교육 환경에서 특히 강점이다.

### 약점

“스파르타적 프로그래머” 철학은 특정 개발자 유형을 위한 것이다.
GUI 도구, 비주얼 씬 에디터, 내장 디버거를 원하는 개발자에게는
Unity, Godot, Unreal Engine이 더 적합하다.
raylib은 의도적으로 이 방향을 거부하지만,
이는 동시에 특정 사용자층—비전문가, 교육 목적, 빠른 프로토타입—에 최적화된 것임을 의미한다.

공식 API 문서가 없다는 결정도 논쟁적이다.
치트시트는 함수 시그니처를 보여주지만
매개변수의 세부 동작, 엣지 케이스, 플랫폼별 차이를 설명하지 않는다.
숙련 개발자에게는 소스 코드가 답이 되지만,
초보자에게는 함수가 정확히 무엇을 하는지 알기 위해
예제를 역추적해야 하는 부담이 있다.

## 인사이트

### “즐거움”을 설계 목표로 삼을 때 무엇이 달라지는가

raylib의 첫 문장은 “a simple and easy-to-use library to enjoy videogames programming”이다.
“enjoy(즐기다)”라는 단어는 대부분의 기술 라이브러리에서 찾기 어렵다.
성능, 확장성, 기능 완성도 대신 “즐거움”을 명시적 설계 목표로 삼는다는 것은
라이브러리가 누구를 위한 것인지를 결정한다.

즐거움을 최우선으로 할 때 제거되는 것들이 있다.
복잡한 설정, 긴 학습 곡선, 방대한 문서, 추상화 계층이 줄어든다.
대신 “화면에 무언가를 그리기까지의 시간”이 최소화된다.
이 최소화가 초보자에게는 진입 장벽을 낮추고,
숙련자에게는 프로토타입 속도를 높인다.

이 철학은 게임 개발 교육의 맥락에서 특히 의미 있다.
대학 컴퓨터 그래픽 수업에서 OpenGL을 직접 가르치면
첫 삼각형 하나 그리기까지 수십 줄의 보일러플레이트가 필요하다.
raylib에서는 `DrawTriangle()`을 호출하면 된다.
개념을 배우는 데 집중할 수 있고 설정에 낭비되는 시간이 없다.
즐거움은 사치가 아니라 학습 효율의 요소다.

### C99 라이브러리가 2020년대에도 성장하는 이유

raylib은 C99로 작성된 라이브러리이지만
2020년대 들어 Zig, Odin, Rust 커뮤니티에서도 활발히 사용된다.
이것은 역설처럼 보인다. 새로운 시스템 프로그래밍 언어들은
C보다 안전하고 표현력이 높은 언어로 자신을 내세우는데,
왜 그 커뮤니티들이 C 라이브러리를 선택하는가?

답은 FFI 친화성과 에코시스템 부재에 있다.
Zig와 Odin은 아직 성숙한 게임 개발 라이브러리가 없다.
C로 작성된 raylib은 이들 언어에서 네이티브 수준의 성능으로 호출할 수 있다.
raylib을 쓰면서 언어 자체의 장점(메모리 안전성, 컴파일 타임 보장)을
포기하지 않아도 된다.
C99 표준이 만든 안정적 ABI는 언어 생태계 경계를 넘는 접착제가 된다.

이 패턴은 더 넓은 진실을 가리킨다.
새로운 언어가 등장해도 C 라이브러리 에코시스템은 축소되지 않는다.
오히려 새 언어들이 C 에코시스템을 흡수하며 성장한다.
C의 수십 년 된 라이브러리 자산은 새 언어들의 출발점이 되고,
새 언어들의 바인딩은 C 라이브러리에 새 사용자를 불러온다.

### 단순성은 전략이다

raylib이 “스파르타적”을 자처하며 GUI 도구와 비주얼 에디터를 거부하는 것은
단순히 미학적 선택이 아니라 경쟁 전략이다.
Unity, Unreal, Godot은 풍부한 편집기와 에셋 스토어, 비주얼 스크립팅을 제공한다.
raylib이 같은 방향으로 경쟁하면 훨씬 더 큰 자원을 가진 경쟁자에게 진다.
대신 raylib은 그 경쟁자들이 잘 못하는 것—단순성, 학습 친화성, 코드 직접 제어—에 집중한다.

이 전략은 SQLite가 “서버리스(serverless)” 데이터베이스로 자신을 포지셔닝하여
PostgreSQL, MySQL과 직접 경쟁을 피하는 것과 유사하다.
SQLite가 말하는 “우리는 클라이언트/서버 데이터베이스를 대체하지 않는다”처럼,
raylib도 Unity를 대체하지 않는다.
다른 문제를 해결하는 다른 도구다.
그러나 그 다른 문제가 교육 시장, 인디 게임 개발자, 게임 잼(game jam) 커뮤니티에서
충분히 크다면 지속 가능한 생태계가 된다.

라이브러리의 장기 생존은 종종 “무엇을 하는가”보다
“누구를 위해 존재하는가”를 명확히 하는 것에 달려 있다.
raylib은 “즐겁게 게임을 배우고 싶은 사람”이라는 답을 처음부터 갖고 있었다.
