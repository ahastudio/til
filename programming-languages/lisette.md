# Lisette: Rust 문법 + Go 런타임 프로그래밍 언어

<https://lisette.run/>

<https://github.com/ivov/lisette>

## 요약

Lisette는 Iván Ovejero가 만든 프로그래밍 언어로, Rust에서 영감을 받은 문법으로
작성한 코드를 Go 코드로 컴파일한다. 한 문장 요약: “Rust 문법, Go 런타임.”

타입 시스템의 핵심은 Hindley-Milner 타입 추론, 대수적 데이터 타입(ADT),
패턴 매칭, `Option<T>`와 `Result<T, E>`를 통한 nil 제거다. Go의 nil 포인터
역참조, 불완전한 에러 처리, 누락된 패턴 케이스 같은 런타임 문제들을 컴파일 타임에
잡아준다.

Go 생태계와의 상호운용성을 핵심 가치로 내세운다. Go 표준 라이브러리를 직접 사용할
수 있고, goroutine과 channel 기반 동시성 모델도 그대로 활용한다. `task` 키워드로
goroutine을 생성하고, `Channel<T>` 타입으로 통신한다.

파이프라인 연산자(`|>`), try 블록, recover 블록, 직렬화 어노테이션 등
편의 기능도 갖췄다. 컴파일 결과가 가독성 있는 Go 코드여서 출력을 직접 확인하고
수정할 수 있다. Rust로 작성된 컴파일러, LSP 서버, 린터, 포매터를 포함하며
VSCode, Neovim, Zed를 지원한다. 설치는 `cargo install lisette`.

## 분석

### 언어 설계 전략: 두 언어의 장점 결합

Lisette의 전략은 명확하다. Go의 단점(nil, 불완전한 에러 처리, 반복적인 에러
체크)을 Rust의 타입 시스템으로 보완하되, Go의 장점(빠른 컴파일, 거대한 생태계,
단순한 동시성)은 그대로 취한다.

이것은 소위 “transpiler 언어” 전략이다. TypeScript가 JavaScript의 단점을
보완하면서 JavaScript 생태계를 그대로 활용하는 것과 구조적으로 같다. Go 생태계는
이미 거대하다. 클라우드 인프라 도구(Docker, Kubernetes, Terraform)가 Go로
작성되어 있고, 수십만 개의 Go 라이브러리가 있다. Lisette는 이 생태계를 버리지
않으면서 더 나은 언어를 제공하려 한다.

### 250개 이상의 컴파일러 진단 메시지

“250+ diagnostic messages”라는 수치가 흥미롭다. 단순히 에러를 잡는 것이 아니라,
어떻게 고쳐야 하는지를 안내하는 메시지 품질이 현대 언어 개발의 경쟁 요소가 되었다.
Rust 컴파일러가 이 흐름을 주도했다. 에러 메시지가 “무엇이 잘못되었는가”를 넘어서
“왜 잘못되었고 어떻게 고치는가”를 설명한다. Lisette는 이 철학을 Go 생태계에 가져온다.

### 컴파일 출력의 투명성

“가독성 있는 Go 코드로 컴파일”이라는 설계 결정이 주목할 만하다. 컴파일러 출력이
난독화된 코드가 아니라 개발자가 읽고 이해할 수 있는 Go 코드다. 이것은 두 가지를
의미한다. 첫째, 디버깅이 Go 도구로 가능하다. 둘째, Lisette를 완전히 신뢰하지
않더라도, 생성된 Go 코드를 검토하고 필요하면 수정할 수 있다. 새 언어 채택의
가장 큰 장벽 중 하나인 “블랙박스 신뢰” 문제를 완화한다.

## 비평

### 강점

TypeScript-JavaScript 관계의 성공을 Go 생태계에서 재현하려는 전략은 합리적이다.
Go에서 가장 불만스럽게 여겨지는 부분들(nil, `if err != nil` 반복, 누락된
패턴 매칭)을 정확하게 타겟팅했다. 컴파일러를 Rust로 작성한 것도 적절하다.
Rust 컴파일러의 진단 메시지 수준을 참고하기 쉽고, 성능도 좋다.

Go 1.25+ 요구 사항은 현실적이다. 이미 Go를 사용하는 팀이 타겟이므로, 최신
Go 버전을 요구하는 것은 무리가 없다.

### 약점

아직 712개 스타(2026년 4월 기준)로 비교적 작은 커뮤니티다. 새 언어 채택의
가장 큰 위험은 언어 자체의 품질이 아니라 생태계와 지속 가능성이다. 개인 프로젝트가
대기업 지원 없이 성숙한 언어 생태계를 구축한 사례는 극히 드물다.

또한 “Rust 문법”이라는 표현이 진입 장벽이 될 수 있다. Rust를 모르는 Go 개발자는
Lisette도 배워야 한다. TypeScript가 성공한 이유 중 하나는 JavaScript 개발자가
타입 어노테이션을 점진적으로 추가하면서 배울 수 있었기 때문이다. Lisette는
완전히 새로운 문법을 요구한다.

## 인사이트

### TypeScript 모멘트를 기다리는 Go

TypeScript는 2012년 출시 후 수년간 니치 도구였다가, React와 대형 프로젝트의
복잡성이 증가하면서 사실상 표준이 되었다. 전환점은 “타입이 없어서 생긴 실제 버그”가
누적되면서 생산성 비용이 명확해졌을 때다.

Go 생태계에도 같은 전환점이 올 수 있다. 지금 Go를 사용하는 시스템들이 더 복잡해지고,
nil 포인터 패닉과 무시된 에러로 인한 프로덕션 장애가 축적되면, “Go보다 안전한
Go”에 대한 수요가 생긴다. Lisette는 그 전환점을 노린다. 다만 TypeScript는
Microsoft의 지원과 VS Code와의 통합이라는 강력한 배경이 있었다. Lisette가
같은 임계점에 도달할 수 있을지는 여전히 열린 질문이다.

### 언어 레이어링의 패턴

프로그래밍 언어 역사에서 “기존 언어 위에 레이어를 얹는” 전략은 반복적으로 나타난다.
CoffeeScript → TypeScript → JavaScript, Kotlin → JVM, Scala → JVM,
Clojure → JVM. 이들의 공통점은 기존 생태계(라이브러리, 도구, 런타임)를 활용하면서
언어 수준의 개선을 제공한다는 것이다.

성공 조건도 패턴이 있다. 기존 언어의 명확한 단점을 해결할 것, 기존 코드베이스와의
점진적 통합이 가능할 것, 도구 지원(IDE, 디버거)이 충분할 것. Lisette는 첫 번째와
세 번째를 갖추고 있지만, 두 번째가 도전이다. Go 코드베이스에 Lisette를 점진적으로
도입하는 방법이 얼마나 매끄러운가가 채택 속도를 결정할 것이다.

### Go의 철학과 Lisette의 긴장

Go는 의도적으로 단순하게 설계되었다. Rob Pike를 비롯한 Go 설계자들은 제네릭,
ADT, 패턴 매칭을 오랫동안 거부했다. 단순성이 대규모 팀에서의 유지보수성을
높인다는 철학이었다. Go가 결국 제네릭을 도입했지만, 여전히 “가능한 한 적은
기능”을 지향한다.

Lisette는 이 철학에 정반대 방향에서 접근한다. Rust의 표현력 있는 타입 시스템을
Go 위에 올린다. 이것은 단순성을 포기하고 안전성과 표현력을 선택하는 것이다.
두 철학 중 어느 것이 실제 대규모 시스템 개발에서 더 낫냐는 Go 커뮤니티의 오래된
논쟁이다. Lisette의 채택 여부는 이 논쟁에 데이터를 추가할 것이다.
