# 다크 모드의 여섯 단계 (Six Levels of Dark Mode)

원문: <https://cssence.com/2024/six-levels-of-dark-mode/>

## 요약

Matthias Zöchling의 글은 많은 웹사이트가 아주 낮은 비용으로도 다크 모드를 지원할
수 있음에도 여전히 지원하지 않는다는 관찰에서 출발한다.
저자는 다크 모드 구현의 난이도를 여섯(확장하면 여덟) 단계로 정리한다.
단계별로 기술은 달라지지만 공통점은 “사용자의 시스템 선호를 존중한다”는 방향성이다.

Level 1 Barebone은 HTML에 `<meta name=“color-scheme” content=“light dark”>` 한 줄만
추가하는 접근이다.
브라우저가 사용자 시스템 설정에 맞춰 기본 색을 칠해준다.
Level 2 Basic은 CSS로 이 속성을 루트에 지정한다: `html { color-scheme: light dark; }`.
User agent 기본 스타일에 의존하면서 시스템 색(AccentColor 등)을 선택적으로 사용한다.

Level 3 Benign은 CSS의 `light-dark()` 함수로 간단한 두-색 분기를 쓴다.
`background-color: light-dark(white, black);` 같은 식으로 한 선언으로 양쪽을 처리한다.
Level 4 Bold는 `@media (prefers-color-scheme: dark)` 미디어 쿼리로 가장 세밀한
커스터마이징을 한다.
복잡한 테마 체계, 브랜드 색, 이미지 대체까지 이 단계에서 가능하다.

Level 5 Bisectional은 HTML `<link media=“...”>` 속성을 활용해 테마별 스타일시트를
분리 로딩한다.
대역폭을 아끼는 접근이다.
Level 6 Ballistic은 JS `matchMedia()`로 런타임에 감지·반응하는 동적 처리다.

확장으로 Level 7 Beyond는 자동/라이트/다크 3-상태 사용자 스위처를 제공하고,
Level 8 Beguiling은 `:has()` 선택자로 meta 태그 속성을 쿼리해 클래스나
data 속성 없이 상태를 반영하는 고급 기법이다.

## 분석

### “기본값을 존중하면 대부분 끝난다”는 철학

Level 1만으로 수많은 사이트가 기본적 다크 모드 호환성을 얻을 수 있다.
이 관찰의 함의는 크다.
다크 모드는 “대대적인 재설계”가 아니라 “사용자 시스템 선호에 끼어들지 않기”
문제라는 것이다.
브라우저와 OS는 이미 올바른 기본값을 갖고 있고, 웹사이트가 그 체계를 덮어쓰지만
않으면 된다.

### 웹 플랫폼의 점진적 도구화

Level 1-6의 나열은 지난 10년간 CSS와 브라우저 API가 다크 모드 문제를 얼마나
체계적으로 해결해왔는지 드러낸다.
`prefers-color-scheme` 미디어 쿼리는 2019년 Safari 12.1에서 시작됐고,
`color-scheme` 속성은 이후 표준화됐으며, `light-dark()` 함수는 2024년에서야 광범위
지원에 들어갔다.
즉 단계가 높을수록 더 최근 도구다.
이 연대기 자체가 플랫폼이 유기적으로 추상화 레벨을 높여온 역사다.

### `:has()`가 바꾸는 패턴

Level 8이 제안하는 `html:has(meta[name=“color-scheme”][content*=“dark”])` 식의
선택자는 단순한 쇼케이스가 아니다.
이는 “상태를 데이터 속성이나 클래스로 이중화하지 말라”는 새로운 설계 지침이다.
meta 태그 자체가 상태의 단일 진실 소스가 되고, CSS가 그것을 읽는다.
이는 JS에서 상태를 DOM에 동기화하던 관행을 CSS 레벨로 끌어올린다.

## 비평

### 강점: 난이도 스펙트럼의 명료함

“다크 모드 = 어려운 작업”이라는 통념을 여섯 단계 그라데이션으로 해체한다.
디자이너·PM은 이 글만 보고도 “우리는 Level 3으로 충분하다”는 판단이 가능하다.
기술 문서가 제공해야 할 “의사결정 프레임”을 정확히 제공한다.

### 약점: 접근성과 실사용 이슈의 부재

Level 3의 `light-dark()`는 간결하지만, 중간색·이미지·그림자·그라데이션 같은 요소는
두 값으로 분기하기에 부족하다.
실제 사이트의 다크 모드는 색 반전이 아니라 대비 조정, 이미지 명도 보정, SVG
stroke 색 전환 같은 디테일이 합쳐져야 한다.
이 디테일은 글에서 다뤄지지 않는다.

### 누락된 논의: FOUC와 퍼시스턴스

사용자 스위처(Level 7)를 붙이면 새로고침 직후 “플래시”가 발생하는 문제가 즉시
따라온다.
`<html>` 렌더 전에 localStorage 값을 반영해야 하므로 inline `<script>`가 보통
추천된다.
이 실무적 난관은 레벨 구조 밖에 존재하지만, 실제 구현자가 가장 먼저 부딪히는
지점이다.

## 인사이트

### 웹 플랫폼 설계의 승리 사례

다크 모드는 2010년대 중반까지 각 사이트가 자체 스위처를 만들고 쿠키/로컬스토리지에
상태를 저장하고 JS로 클래스를 토글하던 영역이었다.
지금 이 글이 “Level 1은 meta 한 줄이면 끝”이라고 말할 수 있는 것은 플랫폼이
문제를 정확히 올바른 층위에서 해결했기 때문이다.
OS가 선호를 제공하고, 브라우저가 그것을 읽고, CSS가 그것을 선언형으로 반영한다.

이 구조는 접근성·다국어·색 대비 등 다른 “사용자 환경 선호” 문제에도 같은 패턴을
제안한다.
`prefers-reduced-motion`, `prefers-reduced-data`, `prefers-contrast` 같은 미디어 쿼리는
모두 이 철학의 후속 구현이다.
웹은 “앱이 사용자에게 선호를 묻는” 모델에서 “시스템 선호를 앱이 투명하게 존중하는”
모델로 이동하고 있다.

역사적으로 데스크톱 앱은 오랫동안 자체 UI 설정을 요구했고, 결과는 앱마다 따로
설정을 바꿔야 하는 “설정 파편화”였다.
웹은 이 교훈을 학습해 OS 레벨의 선호를 기본값으로 흡수하는 방향을 택했다.
이는 단순 편의가 아니라 운영체제·브라우저·웹 앱 3층 구조가 올바르게 정렬됐다는
플랫폼 성숙의 증거다.

### 다크 모드는 이제 기본값, 미지원이 부채가 된다

2020년 전후까지는 다크 모드가 “있으면 좋은 기능”이었다.
이제 상황이 뒤집혔다.
OS·브라우저·대부분의 주요 앱이 기본 지원하는 환경에서, 여전히 시스템 선호를
무시하는 사이트는 “현대적이지 않은 사이트”로 인식된다.
특히 개발자·디자이너 커뮤니티에서 이 인식은 강하다.

이는 “구현 여부”가 아니라 “기본값 쪽 부채”라는 역전이다.
예전에는 다크 모드 구현이 투자이자 차별화였다.
이제는 미구현이 누적되는 유지 보수 비용이다.
사용자는 야간에 라이트 모드 사이트에 들어가면 “브라우저 다크 리더”로 임의 반전을
하거나 탭을 닫는다.
두 반응 모두 사이트에 비용이다.

따라서 실무적 함의는 명확하다.
신규 프로젝트는 Level 1-3 사이에서 시작해 점진적으로 올리는 것이 올바른 기본값이다.
“나중에 추가할 기능”으로 미루면 색 팔레트가 라이트 모드 한쪽에만 최적화돼 추후
보수 비용이 커진다.
색 시스템을 처음부터 “라이트·다크 쌍”으로 설계하는 것이 토큰화된 디자인 시스템
시대의 표준이다.

### 선언형 플랫폼의 확장은 JS를 다시 밀어낸다

Level 6 Ballistic(JS `matchMedia`)와 Level 8 Beguiling(`:has()`)의 대비는 중요한
방향성을 보여준다.
과거에는 “상태 기반 UI 변경”이 반드시 JS 영역이었다.
스위처 클릭, 클래스 토글, DOM 동기화가 JS의 본업이었다.
그런데 `:has()`와 시스템 미디어 쿼리 조합은 이런 동기화를 CSS 선언만으로 완결시킨다.

이는 “CSS가 상태 기계를 부분적으로 흡수하고 있다”는 관찰과 일치한다.
`:checked`, `:focus-within`, `:has()`, `@container`, View Transitions 같은
기능들이 누적되며, 과거 JS로만 가능하던 UI 상태 반영의 상당 부분이 CSS로
이동한다.
결과적으로 “사소한 UI를 위해 JS 번들을 키우지 않아도 된다”는 새로운 기본값이 생긴다.

장기적으로 이는 프런트엔드 빌드의 JS 무게 중심을 낮춘다.
React·Vue 같은 프레임워크가 여전히 필요한 영역은 남지만, “라이트/다크 스위처”,
“아코디언”, “탭 전환” 같은 고전적 인터랙션은 CSS로 충분해진다.
개발자는 “이 기능이 정말 JS가 필요한가”를 매번 되묻게 된다.
다크 모드는 이 질문의 가장 명료한 사례이며, 6단계 레벨링은 같은 문제에 대해 JS
의존 대 선언형 의존 사이에서 선택지를 제공하는 지도다.
