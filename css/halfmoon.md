# Halfmoon

<https://www.gethalfmoon.com/>

HN 토론: <https://news.ycombinator.com/item?id=23848198> (7점, 6개 댓글)

## 소개

Halfmoon은 “고도로 커스터마이즈 가능한 드롭인 Bootstrap 대체제(Highly customizable, drop-in Bootstrap replacement)”를 표방하는 CSS 프레임워크다.
Bootstrap과의 문법 호환성을 유지하면서 세 가지 내장 테마와 모든 컴포넌트에서의 다크 모드 지원을 제공한다.
v2.0.2, MIT 라이선스, 개인 개발자 Tahmid Khan이 관리한다.

세 가지 핵심 테마가 있다.
Default는 Apple HIG(Human Interface Guidelines)에서 영감을 받은 고전적 스타일이다.
Modern은 특히 다크 모드에서 현대적 감각을 강조한다.
Elegant는 Edward Tufte의 데이터 시각화 미학을 참조한 정제된 스타일이다.

## 분석

### “Drop-in Bootstrap 대체제”라는 포지셔닝이 전략적으로 제공하는 것

Bootstrap은 웹 UI 프레임워크 시장에서 10년 이상 지배적 위치를 유지했다.
이 기간 동안 수많은 프로젝트가 Bootstrap의 클래스 명명 규칙과 컴포넌트 구조를 기반으로 쌓였다.
이 생태계에서 “드롭인 대체제”를 주장하는 것은 전환 비용을 낮추면서 Bootstrap 사용자 기반 전체를 잠재적 사용자로 만드는 전략이다.
`class=“btn btn-primary”` 같은 익숙한 문법을 그대로 유지하면 학습 비용이 거의 없다.

이 전략은 Bootstrap 대안으로 포지셔닝한 여러 프레임워크들이 선택한 것과 같다.
Bulma, UIKit, Materialize도 각각 다른 방식으로 Bootstrap 사용자를 노렸다.
Halfmoon이 이들과 다른 지점은 호환성을 더 강하게 내세우는 동시에
“커스터마이즈 가능성”을 핵심 차별점으로 삼는다는 것이다.
Bootstrap의 가장 자주 언급되는 단점인 “Bootstrap처럼 보인다”는 문제를 테마 시스템으로 해결하려는 시도다.

### 세 테마의 기준점 선택이 드러내는 의도된 사용자

Default 테마가 Apple HIG에서 영감을 받았다는 것은 단순한 미적 선택이 아니다.
Apple HIG는 디자인 업계에서 높은 품질 기준의 대명사로 인식된다.
이를 참조했다고 밝히는 것은 “우리의 기본 테마는 높은 수준의 디자인 판단에 기반한다”는 신뢰 신호를 빌리는 것이다.
Elegant 테마가 Tufte를 참조하는 것도 같은 맥락이다.
Edward Tufte는 데이터 시각화와 정보 디자인 분야에서 최고의 권위로 꼽히는 이름이다.

이 기준점 선택이 암시하는 것은 Halfmoon의 의도된 사용자가 디자인 감각을 가진 개발자 또는
디자이너가 없는 팀에서 품질 있어 보이는 UI를 빠르게 구축해야 하는 사람이라는 점이다.
“Bootstrap처럼 보이지 않는” 결과물을 최소한의 커스터마이즈로 얻고 싶은 수요를 겨냥한다.

### 다크 모드를 후행 기능이 아닌 설계 원칙으로 다루는 입장

Bootstrap v5가 다크 모드를 지원하기 시작한 것은 2023년 v5.3부터다.
그 이전에도 다크 모드를 원하는 팀은 CSS 변수를 직접 덮어쓰거나 서드파티 라이브러리를 사용해야 했다.
Halfmoon이 모든 테마와 컴포넌트에서 다크 모드를 기본 제공하는 것은
다크 모드가 선택적 기능이 아니라 처음부터 고려된 설계 요건이라는 입장이다.

이 선택은 사용자 경험 측면의 차별화와 기술적 구현 측면의 차별화를 동시에 한다.
다크 모드를 처음부터 설계에 포함하면 단순히 색상을 반전하는 것이 아니라
각 컴포넌트의 명암비, 그림자, 경계선이 다크 모드에서도 의도대로 작동하도록 고려해야 한다.
이것이 실제로 구현됐다면 다크 모드를 나중에 추가하는 방식보다 시각적 일관성이 높을 가능성이 있다.

초기 버전에서 `prefers-color-scheme` 미디어 쿼리를 지원하지 않는다는 지적이 있었다.[^stephenr]
작성자는 JavaScript로 이를 처리한다고 답했다.
이 선택은 눈에 잘 띄지 않는 절충점을 드러낸다.
CSS 미디어 쿼리 방식은 JavaScript 로딩 없이도 OS 설정을 즉시 반영하지만,
JavaScript 방식은 스크립트가 실행되기 전 잠깐 잘못된 테마가 표시되는 플래시 현상(FOUC)이 생길 수 있다.
다크 모드를 “설계 원칙”으로 내세우면서도 구현이 CSS 미디어 쿼리가 아닌
JavaScript에 의존한다는 것은 “처음부터 고려된 설계 요건”이라는 주장을 약화시킨다.

## 비평

### “Drop-in 대체제” 주장의 실제 범위가 명확하지 않다

Bootstrap은 단순한 CSS 파일이 아니다.
JavaScript로 구동되는 컴포넌트들—dropdown, modal, tooltip, popover, carousel—이 Bootstrap 프로젝트의 상당 부분을 차지한다.
Halfmoon이 Bootstrap의 CSS 클래스와 호환된다고 해도
이 JavaScript 컴포넌트들이 같은 방식으로 작동하는지, 혹은 Halfmoon이 자체 JavaScript를 제공하는지가 불명확하다.

또한 “드롭인”이 가능하려면 어떤 버전의 Bootstrap과 호환되는지가 중요하다.
Bootstrap v4와 v5는 클래스 명명 규칙에 차이가 있다.
Halfmoon이 v4, v5 중 어느 버전을 기준으로 하는지, 또는 양쪽 모두와 호환되는지가
홈페이지에서 명확하게 제시되지 않는다.
“드롭인 대체제”라는 주장이 실제로 어느 범위까지인지 사용자가 직접 문서를 파고들기 전까지 알기 어렵다.

HN 토론에서 `112`는 더 근본적인 개념 문제를 제기했다.[^112]
“Halfmoon은 프레임워크가 아니라 기껏해야 라이브러리”라는 지적이다.
“프레임워크”는 애플리케이션 구조를 규정하는 반면, “라이브러리”는 필요한 부분에 선택적으로 사용한다.
이 구분이 사소해 보일 수 있지만, 개발자가 도구를 평가하는 방식에 실질적 영향을 미친다.
Halfmoon을 “프레임워크”로 소개하면서 그것이 정확히 무엇을 제공하는지 명확히 하지 않는 것은
“드롭인 대체제”라는 주장과 마찬가지로 범위의 불명확성 문제다.

`tikkun`은 “드롭인 Bootstrap 대체제”라는 포지셔닝의 핵심 문제를 짚었다.[^tikkun]
“내가 드롭인 Bootstrap 대체제를 원하거나 필요로 하는 이유가 무엇인지” 홈페이지가 설명하지 않는다는 것이다.
Bootstrap을 이미 쓰는 팀이 전환을 고려하려면 “현재 상태보다 무엇이 나아지는가”가 먼저 답해져야 한다.
“이렇게 다르다”가 아니라 “이것 때문에 전환할 가치가 있다”는 설득이 홈페이지에서 이루어져야 한다.

### 개인 개발자 주도 구조가 기업 도입의 장벽이 된다

Halfmoon의 연락처는 개인 이메일과 개인 트위터 계정이다.
이것은 단일 개발자가 프로젝트를 유지하고 있다는 신호다.
MIT 라이선스이기 때문에 법적 사용에는 문제가 없지만
기업 프로젝트에서 CSS 프레임워크를 선택할 때 “장기 유지보수가 보장되는가”는 핵심 질문이다.

Bootstrap은 처음에 Twitter가 만들었고 이후 독립적인 대규모 커뮤니티와 핵심 팀이 생겼다.
Foundation은 Zurb가 상업적 지원을 하고 있다.
이 맥락에서 Halfmoon이 2년 후, 5년 후에도 활발히 유지보수될 것이라는 보장을 찾기 어렵다.
이 문제는 홈페이지가 해결할 수 있는 성격이 아니지만,
채택 결정을 앞둔 팀에게는 실질적인 리스크다.

### 세 테마가 Bootstrap이 갖는 근본 문제를 해결하지 못한다

Bootstrap에 대한 가장 날카로운 비판은 “Bootstrap 같아 보인다”는 것이다.
이 비판의 핵심은 Bootstrap의 기본 디자인 언어가 너무 보편화되어
사용자가 커스터마이즈 없이 쓰면 즉시 Bootstrap임을 알아볼 수 있다는 것이다.

Halfmoon의 세 가지 테마는 이 문제에 대한 답이 될 수 있다.
그러나 내장 테마가 “Halfmoon처럼 보인다”는 새로운 인식이 생기면 같은 문제가 반복된다.
수백만 명이 쓰는 Bootstrap과 달리 Halfmoon의 사용자 풀이 작다면
이 인식이 형성되기 전에 충분한 커스터마이즈가 이루어질 것이다.
그러나 반대로 사용자 풀이 작다는 것은 생태계 성숙도, 서드파티 템플릿, 커뮤니티 지원도 그만큼 작다는 의미다.
이 순환에서 벗어나려면 단순히 더 좋은 기본 테마만으로는 부족하다.

## 인사이트

### CSS 프레임워크 시장이 Tailwind의 등장으로 근본적으로 재편됐다

Halfmoon이 경쟁하는 시장은 2015년의 것이 아니다.
Tailwind CSS가 등장하면서 컴포넌트 기반 프레임워크(Bootstrap, Halfmoon)와 유틸리티 클래스 기반 프레임워크(Tailwind) 사이의 분기가 일어났다.
Tailwind는 미리 만들어진 컴포넌트 대신 작은 단위의 유틸리티 클래스를 조합하는 방식으로 설계의 완전한 통제를 개발자에게 넘겼다.
이 방식이 반응을 얻은 이유 중 하나가 정확히 “Bootstrap처럼 보이지 않는다”는 것이었다.

이 맥락에서 Halfmoon의 경쟁자는 Bootstrap만이 아니다.
Halfmoon을 쓰지 않을 사람들이 선택하는 것이 Bootstrap인지, Tailwind인지, shadcn/ui 같은 headless 컴포넌트 라이브러리인지가 다르다.
Bootstrap을 계속 쓰는 팀은 이미 Tailwind로 넘어가지 않기로 한 팀이다.
이 팀이 Halfmoon으로 이동하기 위한 마찰은 낮지만, 이 팀의 규모 자체가 과거보다 작아지고 있다.

### “기준점의 권위를 빌리는” 포지셔닝이 오히려 제품을 가린다

Apple HIG와 Tufte를 테마 설명에 인용하는 것은 영리한 마케팅이다.
이 이름들은 디자인 품질의 대명사로 인식되며, 이들을 참조했다는 것이
실제 결과물의 품질보다 먼저 인식을 형성한다.
그러나 이 방식에는 역설이 있다.

참조 대상의 권위가 높을수록 결과물에 대한 기대치도 높아진다.
“Apple HIG에서 영감을 받은” Default 테마가 실제로 Apple 제품의 UI 품질에 미치지 못하면
기대치와의 간극이 오히려 실망을 만든다.
Tufte를 참조한 Elegant 테마가 Tufte의 정보 최소주의를 실제로 구현하지 못하면
이름만 빌린 것처럼 보인다.
이 포지셔닝 전략이 성공하려면 참조 대상의 이름값에 걸맞은 결과물이 필요하다.
홈페이지만으로는 이것이 달성됐는지 판단하기 어렵다.

### 오픈소스 CSS 프레임워크의 생명주기가 반복하는 패턴이 있다

웹 UI 프레임워크의 역사는 등장, 성장, 정체, 유지보수 소멸의 반복이다.
Foundation은 활발히 개발되다가 사실상 유지보수 모드로 전환됐다.
Pure.css, Skeleton, Milligram 같은 경량 프레임워크들은 지금도 존재하지만 개발이 멈췄다.
Materialize는 오랫동안 방치됐다가 커뮤니티 포크로 겨우 유지된다.

이 패턴에서 프레임워크가 생존하는 경우는 두 가지다.
첫째는 Bootstrap처럼 충분히 큰 커뮤니티가 형성되어 개인에 의존하지 않게 되는 경우다.
둘째는 Tailwind처럼 상업적 회사가 뒤에 있어 재정적 지속가능성이 있는 경우다.
Halfmoon이 이 두 경로 중 하나를 걷지 못하면 같은 패턴을 따를 가능성이 높다.
이것은 Halfmoon의 기술적 품질과 무관한 구조적 도전이다.
이 맥락에서 Halfmoon을 평가할 때 “좋은 프레임워크인가”만큼 “충분히 오래 살아남을 프레임워크인가”도 중요한 질문이다.
