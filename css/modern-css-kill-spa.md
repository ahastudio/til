# 모던 CSS가 SPA를 죽일 때가 됐다

> 원문: <https://www.jonoalderson.com/conjecture/its-time-for-modern-css-to-kill-the-spa/>

## 요약

### 주장

Jono Alderson은 네이티브 브라우저 기능이 SPA(Single-Page Application)의
핵심 존재 이유를 제거했다고 주장한다.
View Transitions API와 Speculation Rules라는 두 가지 웹 플랫폼 기능이
JavaScript 프레임워크 없이도 매끄러운 페이지 전환을 가능하게 만들었다는 것이다.

### 핵심 기술

View Transitions API는 전체 문서 네비게이션에서 선언적 페이지 애니메이션을 제공한다.
CSS 한 줄(`@view-transition { navigation: auto; }`)로 활성화되며,
Chromium 기반 브라우저에서 네이티브 지원된다.
Speculation Rules API는 링크된 페이지를 사전 렌더링하여
즉각적인 네비게이션 경험을 만들어낸다.

### 4가지 논거

첫째, MPA(Multi-Page Application)가 하이드레이션이 필요한 SPA보다
초기 로딩이 빠르다.
둘째, 상태 관리와 클라이언트 사이드 라우팅의 엔지니어링 오버헤드가 사라진다.
셋째, 스크롤 복원, 페이지 공지, 스크린 리더 지원 등
접근성이 브라우저 기본값으로 처리된다.
넷째, 프레임워크 생태계 대비 인지 부하가 줄어든다.

## 분석

### 논증 구조

이 글의 논증 구조는 "SPA의 장점이었던 것들을 브라우저가 흡수했다"는
플랫폼 수렴 테제에 기반한다.
이는 역사적으로 반복된 패턴이다.
jQuery가 해결한 크로스 브라우저 호환성 문제를 브라우저가 흡수했듯,
SPA가 해결한 UX 문제를 브라우저가 흡수하고 있다는 것이다.

### 브라우저 지원 현실

View Transitions API의 CSS 한 줄 활성화는 기술적으로 인상적이지만,
글이 언급한 브라우저 지원 현황은 중요한 제약이다.
Safari는 아직 완전 지원하지 않으며, Firefox는 폴백이 필요하다.
2025년 7월 기준으로 Chromium 계열에서만 네이티브 지원된다는 것은
프로덕션 환경에서의 채택을 제한하는 현실적 장벽이다.

### 커뮤니티 반응

댓글에서 제기된 서드파티 위젯(채팅, 분석) 상태 유지 문제와
이에 대한 "companion windows" 제안은
MPA 전환에서 해결해야 할 실질적 과제를 보여준다.
42개의 댓글이 달릴 만큼 커뮤니티의 반응이 활발했으며,
Astro가 이 기능들을 폴백과 함께 구현한 프레임워크로 언급되었다.

## 비평

### 강점

SPA의 존재 이유를 하나씩 분해하고
각각에 대응하는 네이티브 기능을 매핑하는 구조가 명확하다.
CSS 코드 예시를 포함하여 주장이 추상적이지 않고 구체적이다.
접근성 측면의 논거는 특히 설득력이 있다.
SPA에서 수동으로 구현해야 하는 것이
MPA에서는 브라우저 기본값으로 제공된다는 점은 부정하기 어렵다.

### 약점

"SPA를 죽인다"는 프레이밍이 과장되었다.
글이 다루는 것은 콘텐츠 중심 웹사이트의 사례이며,
복잡한 상태 관리가 필요한 애플리케이션(Figma, Notion, Google Docs 등)에
MPA가 대안이 될 수 있는지는 논의하지 않는다.
SPA가 해결하는 문제 공간의 일부만 다루면서
전체를 대체할 수 있다고 주장하는 것은 논리적 비약이다.

브라우저 지원 불완전성에 대한 대응이 약하다.
"Chromium에서 되니까 괜찮다"는 접근은
웹 표준의 정신과 충돌하며,
특히 Safari 사용자 비율이 높은 모바일 환경에서는
실용적 조언으로 기능하기 어렵다.

## 인사이트

### 플랫폼의 흡수 주기: "좋은 추상화는 표준이 된다"

jQuery → `querySelector`, Ajax → `fetch`, SPA 전환 → View Transitions.
이 패턴은 웹 생태계의 근본적인 진화 메커니즘을 보여준다.
라이브러리와 프레임워크는 플랫폼의 부족함을 메우는 임시 해결책이며,
충분히 검증된 패턴은 결국 플랫폼에 흡수된다.

이 흡수 주기에서 중요한 것은 "무엇이 흡수되고 무엇이 남는가"다.
View Transitions는 페이지 전환 애니메이션을 흡수했지만,
React의 컴포넌트 모델이나 상태 관리 패러다임은 흡수 대상이 아니다.
Web Components가 10년 넘게 React를 대체하지 못한 것이 그 증거다.
플랫폼이 흡수하는 것은 "UX 패턴"이지 "개발 패러다임"이 아니다.

이 구분을 놓치면 "브라우저가 다 해주니까 프레임워크가 필요 없다"는
과도한 결론에 도달한다.
실제로는 플랫폼 흡수가 프레임워크의 역할을 재정의할 뿐,
제거하지는 않는다.
Astro가 View Transitions를 폴백과 함께 통합한 것은
프레임워크가 "플랫폼 기능의 안정적 배포 채널"로 진화하는 사례다.

### SPA vs MPA는 잘못된 이분법이다

이 글의 가장 큰 맹점은 웹을 SPA와 MPA로 이분하는 프레이밍 자체다.
현대 웹 아키텍처는 이미 이 경계를 넘어섰다.
Next.js의 App Router, Astro의 아일랜드 아키텍처, Remix의 프로그레시브 인핸스먼트는
"필요한 곳에만 JavaScript를 사용하는" 하이브리드 모델이다.

진짜 질문은 "SPA냐 MPA냐"가 아니라
"어떤 상호작용에 클라이언트 상태가 필요한가"다.
블로그 글 읽기에는 View Transitions면 충분하지만,
스프레드시트 셀 편집에는 클라이언트 상태 관리가 필수적이다.
같은 애플리케이션 안에서도 페이지마다 답이 다를 수 있다.

View Transitions API의 진짜 가치는 SPA를 죽이는 것이 아니라,
"SPA가 아니면 안 된다"는 기본 가정을 제거하는 것이다.
콘텐츠 사이트가 페이지 전환 때문에 React를 도입할 필요가 없어진다.
이것은 SPA의 사망이 아니라 SPA의 적용 범위가
본래 필요한 영역으로 축소되는 것이다.

### 선언적 UX의 부상: CSS가 JavaScript의 영역을 잠식하는 구조적 추세

`@view-transition { navigation: auto; }`라는 한 줄이 함축하는 것은
"전환 애니메이션은 선언적으로 표현 가능한 문제"라는 판단이다.
이는 CSS가 레이아웃(Flexbox, Grid)에 이어
인터랙션 영역까지 확장하는 더 큰 흐름의 일부다.

`:has()` 선택자, `@container` 쿼리, `@scope`, `scroll-timeline`,
그리고 이제 `@view-transition`까지.
CSS는 "스타일링 언어"에서 "선언적 UI 로직 언어"로 진화하고 있다.
JavaScript가 명령형으로 해결하던 문제들이
하나씩 CSS의 선언적 모델로 이동하고 있다.

이 추세의 구조적 원인은 성능이다.
브라우저의 렌더링 엔진은 선언적 규칙을 최적화하는 데 수십 년의 투자가 있었지만,
JavaScript 실행은 본질적으로 최적화가 어렵다.
"CSS로 할 수 있으면 CSS로 하라"는 경험칙이
브라우저 벤더의 API 설계 원칙으로 격상된 것이다.

이 방향이 계속되면 프론트엔드 개발의 기술 스택은
"JavaScript를 얼마나 적게 쓸 수 있는가"로 평가될 수 있다.
이는 프론트엔드 엔지니어의 핵심 역량이
프레임워크 숙련도에서 플랫폼 이해도로 이동함을 의미한다.
