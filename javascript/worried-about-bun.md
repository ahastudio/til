# 나는 Bun이 걱정된다

원문: <https://wwj.dev/posts/i-am-worried-about-bun/>

HN 토론: <https://news.ycombinator.com/item?id=48011184> (494점, 318개 댓글)

## 요약

저자는 2025년 12월 Anthropic이 Bun을 인수한 이후 Bun의 미래에 대한 우려를 표명한다.
Bun 자체는 여전히 뛰어난 소프트웨어이며, 저자도 이를 인정한다.
빠른 속도, TypeScript 호환성, 복잡한 툴체인을 단순화하는 실용적 기능들은 여전히 유효하다.
하지만 Bun을 인수한 조직이 어떤 제품 문화를 가졌는지가 문제다.

Anthropic은 인수 당시 오픈소스 개발 지속과 MIT 라이선스 유지를 약속했다.
그 명분으로 내세운 논리는 “Claude Code가 Bun 실행 파일로 수백만 사용자에게 배포된다.
Bun이 망가지면 Claude Code도 망가진다”는 것이었다.
이 논리는 표면적으로 설득력 있어 보이지만, 저자는 정확히 그 논리 구조 자체를 불안 요인으로 읽는다.
Bun이 Anthropic의 제품 이해관계에 종속되는 순간, Bun의 진화 방향은 사용자 커뮤니티가 아니라 Claude Code의 요구사항에 맞춰질 수 있다.

저자가 우려를 근거로 제시하는 것은 Claude Code의 궤적이다.
1년 전만 해도 혁신적이었던 Claude Code가 2026년 들어 문제적인 제품이 됐다는 것이다.
기본 추론 노력의 축소, 코딩 품질에 영향을 미치는 세션 버그, 불투명한 과금 구조, 그리고 OpenClaw 같은 서드파티 하네스에 대한 제한이 그 증거로 제시된다.
특히 git 히스토리에 OpenClaw를 언급하는 것만으로 과금이나 거부 행동이 트리거됐다는 사례는
제품 정책이 사용자에게 투명하지 않은 방식으로 작동하고 있음을 시사한다.

저자는 이 패턴을 코리 닥터로(Cory Doctorow)의 개념인 “엔시티피케이션(enshittification)”으로 명명한다.
제한 강화, 불명확한 가격 정책, 내부 테스트 부재로 보이는 기능 저하가 점진적으로 누적되는 것이다.
결론적으로 저자는 Bun 팀이 좋은 의도를 가졌을 것이라고 인정하면서도,
Anthropic의 문제적 제품 문화에 통합되는 것이 Bun의 발전 궤적을 훼손할 수 있다는 우려를 남긴다.
개인적으로는 pnpm으로 이전하되, Bun이 이 우려를 반증하기를 바란다고 마무리한다.

## 분석

### “자사 제품이 의존하므로 안전하다”는 논리의 한계

Anthropic의 인수 정당화 논리 — “우리도 Bun을 쓰기 때문에 Bun을 잘 관리할 것이다” —
는 표면적으로 합리적으로 보이지만, 실제로는 인센티브 구조를 보장하지 않는다.
자사 제품에서 특정 방식으로 사용하는 한에서만 Bun을 잘 관리한다는 뜻이 될 수 있기 때문이다.
즉, Claude Code가 필요로 하지 않는 기능은 우선순위가 낮아지고,
Claude Code와 충돌하는 기능은 억제될 수 있다.

이것은 인수 이후 오픈소스 프로젝트가 취약해지는 전형적인 경로다.
인수 기업이 오픈소스를 파괴할 의도 없이, 단순히 자신의 제품 로드맵에 맞게 우선순위를 재편하는 것만으로도
커뮤니티가 필요로 하는 기능과 기업이 필요로 하는 기능 사이에 간극이 생긴다.

### Claude Code 사례가 왜 관련성을 갖는가

저자가 Claude Code의 저하를 Bun의 미래에 대한 우려의 근거로 드는 것은 비약처럼 보일 수 있다.
하지만 이 연결은 논리적이다. 두 제품을 관리하는 것은 같은 조직이고,
그 조직이 자신의 플래그십 AI 코딩 도구에서 어떤 제품 결정을 내리는지는
같은 조직이 다른 도구를 어떻게 관리할지에 대한 유의미한 신호다.

OpenClaw 관련 사례 — git 히스토리에 서드파티 도구 이름이 있다는 이유로
과금이나 기능 거부가 트리거됐다는 주장 — 는 단순한 버그가 아니라
제품 정책의 투명성 문제를 드러낸다. 이런 제품 문화에서 오픈소스 프로젝트를 관리하면
어떤 종류의 암묵적 제약이 생길 수 있는지를 상상하는 것은 어렵지 않다.

zarzavat는 이 연결을 더 보편적인 인수 패턴으로 확장한다. "인수 이후 원래 팀원들이 돈을 챙기고 떠나면서 문화가 새 오너의 문화로 교체되는 결말이 반복된다. Anthropic의 CEO는 AI가 프로그래머를 대체한다는 믿음을 Claude Code에 적용했고, 그 결과 Claude Code는 유지보수 불가능한 스파게티 덩어리가 됐다."[^zarzavat] 우려가 추상적 가능성이 아니라 이미 관찰 가능한 패턴 위에 서 있다는 것이다.

0sql은 인수 이후의 실질적 신호 중 하나로, 최근 Bun 기여 중 상당수가 AI가 작성한 것으로 보인다는 점을 지적한다.[^0sql] 핵심 인프라 도구에서 AI 생성 코드 비중이 높아진다는 것은 개발 속도와 기여 품질 사이의 트레이드오프를 새로운 방식으로 제기한다.

### 엔시티피케이션 개념의 적용

코리 닥터로의 엔시티피케이션 프레임은 플랫폼이 세 단계를 거쳐 저하된다고 설명한다.
사용자에게 좋게 → 비즈니스 고객에게 좋게 → 기업 이윤에 좋게.
저자는 Claude Code가 이 사이클의 두 번째에서 세 번째 단계로 이동 중이라고 진단한다.

흥미로운 점은 저자가 Bun에 같은 프레임을 직접 적용하는 게 아니라,
“같은 제품 문화 아래 있게 됐다”는 구조적 이유로 걱정한다는 점이다.
이것은 더 정교한 우려다. Bun이 직접 엔시티파이되는 게 아니라,
Claude Code가 의존하는 한에서만 잘 유지되는 도구가 될 수 있다는 것이다.

## 비평

### 강점: 구체적 사례 기반의 우려

막연한 “대기업이 오픈소스를 망칠 것”이라는 우려가 아니라,
동일 조직의 다른 제품에서 관찰된 구체적 패턴을 근거로 삼는다.
OpenClaw 사례, 과금 불투명성, 세션 버그 등 개인 경험에서 출발한 증거가
주장의 신뢰성을 높인다.

또한 저자가 “Bun 팀은 여전히 좋은 사람들”이라는 점을 인정하면서도 우려를 제기하는 태도는
단순한 감정적 반응이 아닌 구조적 분석임을 보여준다.

### 약점: 인수 이후 경과 데이터 부재

2025년 12월 인수 이후의 실제 Bun 개발 패턴 — 릴리즈 빈도, 커뮤니티 기여 반응 속도,
로드맵 변화 — 에 대한 데이터 없이 우려가 제시된다.
우려 자체는 타당하지만, 우려가 현실화됐다는 증거는 아직 없다.
예방적 이전(pnpm 마이그레이션)을 정당화하기엔 충분하지만,
광범위한 독자에게 같은 선택을 권고하기엔 근거가 약하다.

HN에서 Bun 팀 멤버인 Jarred가 직접 응답했다.[^Jarred] "인수 이후 개발 속도가 오히려 빨라졌다"며 구체적인 로드맵을 제시했다. 바이너리 크기 축소(Windows 17MB, Linux 8MB), `--no-orphans` 플래그, HTTP/3·HTTP/2 실험적 지원, Bun.Image 이미지 처리 라이브러리 등이 준비 중이다. 그는 수익화 압박이 사라져 Claude Code 사용자와 더 넓은 엔지니어링 커뮤니티 모두에 이익이 되는 개선에 집중할 수 있게 됐다고 밝혔다. stopachka도 같은 맥락에서 원문의 접근 방식을 비판한다. "이 글은 Claude Code 경험을 근거로 Bun에 의심을 던지지만 너무 간접적이다. Bun은 오픈소스이므로 더 직접적인 질문은 '인수 이후 Bun이 실제로 어떻게 됐는가'이며, 내가 파악하기로 그들은 이전과 동일한 속도로 사용자에게 반응하고 있다."[^stopachka]

### 약점: 엔시티피케이션 비교의 직접성 부족

Claude Code의 저하와 Bun의 미래를 연결하는 논리는 합리적이지만 간접적이다.
Claude Code는 Anthropic의 직접 수익 제품이고, Bun은 인수된 오픈소스 도구다.
두 제품의 동학은 다르다. Anthropic이 Claude Code에서 과금을 최적화하는 방식과
Bun을 관리하는 방식은 다른 인센티브 구조 위에 있다.
이 차이에 대한 분석이 없으면 비교가 의도보다 강한 인상을 준다.

AntonyGarand는 이 구조적 차이를 명확히 정리한다. "Claude Code는 수익을 발생시키는 제품이며 보안 업데이트를 빠르게 처리해야 한다. Bun은 직접 과금 요소가 없는 JavaScript 런타임이다."[^AntonyGarand] 수익 구조의 차이가 제품 관리 방식에서의 차이를 만든다는 것이다. Anthropic이 Claude Code에서 과금 최적화와 서드파티 제한에 민감한 이유는 제품 문화 이전에 비즈니스 모델 때문이며, 그 논리가 Bun에는 동일하게 적용되지 않는다.

### 보완: 비교 사례의 필요성

오픈소스 프로젝트가 대기업에 인수된 이후 어떤 패턴을 보였는지 비교 분석이 있었다면 더 강한 글이 됐을 것이다.
Sun/Oracle의 MySQL·Java, Facebook의 React/Relay, Google의 Angular — 각각 다른 결과를 보였다.
공통 변수는 인수 기업의 제품 문화가 아니라 오픈소스 기여 구조와 커뮤니티 거버넌스였다.
이 구조가 Bun에 어떻게 설계됐는지가 더 직접적인 예측 변수일 수 있다.

### 보완: 인수 이전의 품질 문제

원문은 Bun이 인수 전에는 뛰어난 도구였다는 전제 하에 우려를 제기하지만,
HN 커뮤니티의 실제 사용 경험은 더 복잡한 그림을 보여준다.
nwienert는 "Bun은 제대로 운영된 적이 없었다. 모든 기능이 버그와 결함으로 가득했고,
대부분의 소프트웨어가 메이저 버전에서나 볼 법한 브레이킹 체인지와 주요 기능들이
패치 릴리즈에 뒤섞여 나왔다"고 진단한다.[^nwienert]
1.1 버전에서 작동하던 기능이 이후 릴리즈에서 아무 문서 없이 수개월간 깨진 채 방치된 사례도 제시됐다.

인수 이전의 Bun이 이미 불안정성 문제를 안고 있었다면,
현재의 우려가 인수 때문인지 원래부터 있던 문제인지를 구분하는 것이 분석의 전제가 되어야 한다.
"인수가 좋은 도구를 망칠 것"이라는 서사는 그 도구가 처음부터 좋았을 때만 성립한다.

## 인사이트

### 오픈소스 지속가능성은 라이선스가 아니라 인센티브 구조의 문제다

MIT 라이선스 유지 약속은 저작권법상 강제력이 있지만,
기능 우선순위, 기여 수용 정책, 릴리즈 타이밍에 대한 약속은 아니다.
어떤 오픈소스 프로젝트든 인수 후 가장 중요한 변수는 “누가 기여를 결정하는가”와
“그 결정이 누구의 이해관계를 반영하는가”다.

Bun이 Anthropic의 내부 팀이 주로 개발하고, 그 팀의 우선순위가 Claude Code와 동기화돼 있다면,
MIT 라이선스는 코드를 포크할 자유를 보장하지만 프로젝트의 발전 방향을 보장하지 않는다.
오픈소스 거버넌스 모델 — 독립 재단, 복수 기업 컨소시엄, 활발한 외부 커미터 비율 —
이 라이선스보다 더 중요한 지속가능성 지표다.

역설적으로, “자사 제품에 의존하기 때문에 잘 관리할 것”이라는 논리는
그 의존성이 사라지거나 약해지면 관리 인센티브도 함께 약해진다는 뜻이기도 하다.
Anthropic이 Claude Code에서 Bun 의존성을 줄이는 방향으로 아키텍처를 바꾼다면,
그 순간 Bun에 대한 투자 명분도 줄어든다.
이것이 인프라 오픈소스를 단일 기업의 제품 이해관계에 묶는 구조적 취약점이다.

### 개발자 도구의 신뢰 자산은 빠르게 소진된다

Claude Code가 1년 만에 “혁신적”에서 “문제적”으로 인식이 바뀐 사례는
개발자 도구 시장의 특성을 드러낸다. 개발자는 도구에 깊이 투자한다.
워크플로우를 재구성하고, 팀 표준을 바꾸고, 지식을 축적한다.
그 투자가 이루어진 후에 도구의 품질이 저하되면, 단순한 불만을 넘어 배신감이 된다.

배신감은 시장 데이터로 잘 보이지 않는다. 많은 개발자가 대안을 즉시 찾지 않는다.
전환 비용이 너무 높기 때문이다. 하지만 새 프로젝트를 시작할 때, 팀을 온보딩할 때,
다음 기술 선택을 할 때 그 불신이 반영된다. 신뢰 손상은 현재 사용자 수에서 드러나지 않고
미래 채택률에서 드러난다.

Bun의 위험은 현재 Bun 사용자를 잃는 것이 아닐 수 있다.
새 프로젝트에서 Bun을 선택하는 개발자 수가 줄어드는 것이다.
“Anthropic 것”이라는 인식 자체가 — Claude Code 경험을 통해 형성된 — 일부 개발자의 첫 선택에서 Bun을 제외시킬 수 있다.
이 효과는 느리고 측정하기 어렵지만, 개발자 생태계에서는 누적된다.

### 독립 오픈소스에서 기업 인수로의 전환은 커뮤니티와의 암묵적 계약을 깬다

Bun이 독립 프로젝트였을 때, 사용자들은 암묵적으로 “이 프로젝트의 방향은 사용자 커뮤니티와 만드는 사람들의 판단에 의해 결정된다”는 가정 하에 투자했다.
Anthropic 인수는 이 계약의 당사자를 바꿨다.
이제 방향 결정에 Claude Code 팀의 이해관계, Anthropic의 법무팀 판단,
투자자의 기대가 포함된다.

이것은 반드시 나쁜 방향을 의미하지 않는다. 기업 자원이 프로젝트를 더 빠르게 발전시킬 수 있다.
하지만 커뮤니티와의 암묵적 계약이 변했다는 사실 자체는 인정되어야 하며,
그 변화를 “MIT 라이선스는 유지된다”는 법적 약속으로 충분히 설명할 수 없다.

저자가 pnpm으로 이전을 결정한 것은 기술적 판단이 아니라 거버넌스 판단이다.
pnpm이 Bun보다 기술적으로 뛰어나서가 아니라, pnpm의 독립성과 커뮤니티 거버넌스 구조가
예측 가능한 방식으로 작동할 것이라는 신뢰 때문이다.
이 신호는 개별적으로는 약하지만, 많은 개발자가 같은 판단을 내릴 때 생태계 수준의 신호가 된다.
오픈소스 생태계에서 거버넌스 신뢰는 기능 우위만큼 중요한 경쟁 요소다.
