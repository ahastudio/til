# Zero-Native: Zig 기반 네이티브 데스크톱 앱 프레임워크

<https://zero-native.dev/>

<https://github.com/vercel-labs/zero-native>

## 소개

Zero-Native는 Vercel Labs가 개발한 Zig 기반 네이티브 데스크톱·모바일 앱
프레임워크다.
시스템 내장 WebView(macOS의 WKWebView, Linux의 WebKitGTK)를 활용해 Chromium을
번들링하지 않고도 웹 UI를 가진 네이티브 앱을 만들 수 있다.
Electron의 무거움과 Flutter/React Native의 커스텀 렌더링 엔진 없이
“작은 바이너리, 최소 메모리, 즉각적인 리빌드”를 목표로 한다.

## 명세

### 아키텍처 구성요소

| 컴포넌트 | 역할                                                      |
| -------- | --------------------------------------------------------- |
| App      | 앱 메타데이터, WebView 소스, 라이프사이클 훅을 담는 Zig 객체 |
| Runtime  | 이벤트 루프, 윈도우, 브리지 디스패치, 플랫폼 서비스 관리    |
| Bridge   | `window.zero.invoke()`를 통한 JavaScript-to-Zig 통신      |
| Manifest | `app.zon`으로 앱 메타데이터, 권한, 보안 정책, 패키징 설정  |

### 웹 엔진 선택지

두 가지 렌더링 경로를 제공한다.

- **시스템 WebView**: WKWebView(macOS), WebKitGTK(Linux). 경량 앱에 적합하며
  Chromium을 번들링하지 않아 바이너리가 작다.
- **CEF(Chromium Embedded Framework)**: 번들 옵션으로 렌더링 일관성이
  필요할 때 선택한다.

### 보안 모델

WebView는 기본적으로 비신뢰(untrusted) 상태로 취급된다.
네이티브 명령 실행, 네비게이션, 윈도우 API 접근은 모두 명시적 권한 부여가
필요한 옵트인(opt-in) 방식이다.
Bridge 통신에는 크기 및 권한 검사가 포함된다.

### 플랫폼 지원

macOS 11 이상, Linux, Windows, iOS, Android를 지원한다.
2026년 5월 현재 v0.1.9까지 출시된 프리릴리즈 소프트웨어다.

## 분석

### Tauri와의 비교: Zig vs Rust

Zero-Native는 Tauri와 동일한 포지션에 있다.
시스템 WebView + 경량 네이티브 셸이라는 접근도 같다.
차이는 셸 언어다. Tauri는 Rust, Zero-Native는 Zig다.
Zig는 컴파일 속도가 빠르고, C 상호운용성이 뛰어나며, 번들 크기가 작다.
그러나 생태계가 Rust보다 훨씬 작고, 아직 1.0 버전이 없는 언어다.
Vercel Labs가 Zig를 선택한 것은 컴파일 속도와 바이너리 크기 최적화에 우선순위를
둔 실험적 결정이다.

### 시스템 WebView의 기회와 위험

시스템 WebView 전략의 핵심 장점은 바이너리 크기다.
Electron 앱은 Chromium을 포함해 최소 100MB를 넘는 반면,
시스템 WebView를 쓰면 셸 바이너리가 수백 KB 수준이 될 수 있다.
그러나 시스템 WebView는 플랫폼마다 다른 버전의 WebKit 또는 EdgeHTML을 쓰기
때문에 렌더링 불일관성이 생긴다.
CSS 기능 지원, JavaScript API 동작, 성능 특성이 다를 수 있다.
이것이 Tauri가 선택적 번들 Chromium을 제공하고, Zero-Native가 CEF를 대안으로
두는 이유다.

### Vercel이 데스크톱 프레임워크를 만드는 이유

Vercel은 웹 호스팅·배포 플랫폼 회사다.
이 회사가 데스크톱 앱 프레임워크를 만든다는 것은 언뜻 뜬금없어 보인다.
그러나 Vercel의 핵심 고객은 웹 개발자이고,
이 개발자들이 점점 더 클라이언트 앱(CLI, 데스크톱 도구, AI 에이전트 UI)을
필요로 하고 있다.
Zero-Native는 웹 개발자가 웹 기술로 네이티브 앱을 만들 수 있는 경로를
제공함으로써 Vercel 생태계를 확장하는 전략으로 읽힌다.

## 비평

### 강점: 보안 우선 기본값

WebView를 비신뢰 상태로 기본 취급하고 모든 네이티브 접근을 옵트인으로 설계한
것은 Electron의 보안 역사에서 교훈을 얻은 접근이다.
Electron은 초기에 렌더러 프로세스에서 Node.js API에 자유롭게 접근할 수 있었고,
이것이 다수의 보안 취약점의 원인이 됐다.
Zero-Native는 이 실수를 반복하지 않으려는 의도가 명확하다.

### 강점: Zig 셸의 빠른 컴파일 속도

Zig의 컴파일 속도는 네이티브 셸 개발 경험을 개선한다.
Rust 기반 Tauri도 Electron보다 빠른 개발 사이클을 제공하지만,
Zig는 Rust보다 컴파일 속도가 더 빠르다.
프론트엔드 변경은 기존 웹 도구로 즉각 반영되고,
네이티브 셸 변경도 빠르게 리빌드되는 개발 경험이 가능하다.

### 약점: 프리릴리즈 + Zig 언어의 불안정성

Zig는 아직 1.0을 출시하지 않았으며 API가 버전 간에 변경될 수 있다.
Zero-Native 자체도 v0.1.9로 프리릴리즈 상태다.
프로덕션 앱에 채택하기 위해서는 언어와 프레임워크 양쪽의 안정성을
기다려야 한다.
Tauri 1.0이 2022년에 출시된 것과 비교하면 채택 준비도가 낮다.

### 약점: Vercel Labs 브랜딩의 지속 가능성 불확실성

“Vercel Labs” 프로젝트는 Vercel의 핵심 제품이 아니다.
실험적 프로젝트로 시작해 중단되거나, 핵심 제품에 흡수되거나,
독립 오픈소스로 분리되는 세 경로 중 하나를 걸을 것이다.
이 불확실성은 장기 의존성으로 채택하기 전에 고려해야 할 리스크다.

## 인사이트

### Electron의 유산이 만든 대안 프레임워크 붐

Electron은 2013년 GitHub Atom 에디터를 위해 만들어졌고,
VS Code, Slack, Discord, Figma 등 수많은 앱이 이를 채택했다.
그러나 Electron 앱의 메모리 사용량과 바이너리 크기는 반복적인 비판의 대상이 됐다.
이 불만이 Tauri, Wails(Go), NeutralinoJS, 그리고 이제 Zero-Native까지
이어지는 “탈 Electron” 흐름을 만들었다.

이 흐름의 공통점은 “시스템 WebView + 경량 네이티브 셸”이라는 전략이다.
Chromium을 포함하지 않으면 바이너리가 작아지지만, 렌더링 불일관성이라는
대가를 치른다.
이 트레이드오프는 앱의 성격에 따라 다르게 평가된다.
내부 도구나 개발자 도구처럼 특정 플랫폼 위주의 앱은 시스템 WebView로 충분하지만,
소비자 앱처럼 모든 플랫폼에서 동일한 경험이 필요하면 번들 Chromium이 낫다.
Zero-Native가 CEF 옵션을 제공하는 것은 이 현실을 인정한 것이다.

역설적으로 Electron의 성공이 이 비판을 가능하게 했다.
Electron이 없었다면 VS Code, Slack 같은 복잡한 크로스플랫폼 앱을 이렇게
빠르게 만들기 어려웠을 것이다.
“너무 무겁다”는 비판은 “충분히 강력하다”는 전제 위에서만 성립한다.

### 시스템 API 브리지의 보안 아키텍처가 드러내는 신뢰 경계

Zero-Native의 `window.zero.invoke()` 브리지와 기본 비신뢰 WebView 모델은
웹과 네이티브의 신뢰 경계를 명시적으로 설계한다.
이것은 단순한 API 선택이 아니라 위협 모델(threat model)의 반영이다.
웹 UI는 외부 콘텐츠를 렌더링할 수 있고, XSS나 악성 스크립트 실행 가능성이 있다.
이 UI가 파일 시스템이나 운영체제 API에 자유롭게 접근할 수 있다면 심각한
보안 위험이 된다.

Electron의 초기 `nodeIntegration: true` 기본값은 이 신뢰 경계를 무시한 설계였고,
CVE가 쌓이면서 뒤늦게 수정됐다.
Zero-Native는 이 역사를 처음부터 반영해서 설계했다.
옵트인 권한 모델은 개발자가 필요한 권한을 명시적으로 선언하게 하고,
최소 권한 원칙(principle of least privilege)을 구조적으로 강제한다.

이 접근은 모바일 앱 권한 모델과 유사하다.
iOS와 Android 모두 앱이 카메라, 위치, 연락처에 접근하려면
사용자에게 명시적으로 요청해야 한다.
데스크톱 앱 프레임워크가 이 모델을 채택한다는 것은, 데스크톱과 모바일의
보안 기대치가 수렴하고 있다는 신호다.

### Zig 언어가 시스템 프로그래밍 생태계에 주는 도전

Zero-Native가 Zig를 선택한 것은 Rust 대비 다른 트레이드오프를 선택한 것이다.
Rust는 메모리 안전성을 컴파일 타임에 강제하는 소유권 시스템을 갖는다.
이것은 강력하지만 학습 곡선이 가파르고, 빌드 시간이 길다는 단점이 있다.
Zig는 소유권 시스템 없이 C에 가까운 제어권을 제공하며, 컴파일이 빠르다.
안전성은 Rust보다 낮지만 C 코드와의 통합이 더 직접적이다.

이 선택은 Vercel Labs의 목표를 드러낸다.
개발자 경험(빠른 컴파일, 작은 바이너리)을 최우선으로 하고,
메모리 안전성보다는 프레임워크 레벨의 설계(브리지 권한 모델)로 보안을
확보하는 전략이다.
Rust가 “언어가 안전성을 강제”라면, Zero-Native는 “아키텍처가 안전성을
구조화”하는 접근이다.

Zig가 Rust처럼 시스템 프로그래밍의 주류 언어가 될지는 불확실하다.
그러나 Bun(JavaScript 런타임)이 Zig로 만들어져 V8 기반 Node.js를 성능으로
압도한 사례처럼, Zig가 틈새 영역에서 강력한 선택지임은 입증되고 있다.
Zero-Native는 그 실험의 연장선이다.
