# Ant: A JavaScript runtime and ecosystem

<https://antjs.org/>

<https://github.com/theMackabu/ant>

HN 토론: <https://news.ycombinator.com/item?id=48875377> (265점, 113개 댓글)

Lobste.rs 토론: <https://lobste.rs/s/n85thm/ant_lightweight_javascript_runtime>

## 소개

Ant는 처음부터 직접 만든 JavaScript 엔진을 기반으로 하는 경량 런타임이다.
Node.js, Bun, Deno와 달리 V8이나 JSC 같은 기존 엔진을 감싸지 않고, “Ant
Silver”라는 자체 엔진을 밑바닥부터 구현했으며, JIT 컴파일러는 저자가 직접
관리하는 MIR 포크를 백엔드로 사용한다.
바이너리 크기는 약 8MB(`-Os` 최적화 시 4.1MB)로, Node(약 120MB), Deno(약
90MB), Bun(약 60MB)에 비해 훨씬 작다.
저자가 공개한 콜드 스타트 벤치마크(Hono 앱을 로드하고 라우트 두 개를
등록한 뒤 종료하는 시나리오, hyperfine으로 10회 워밍업 후 100회 측정)에서는
Ant가 평균 5.5ms로 Bun(10.6ms, 1.93배)·Deno(24.8ms, 4.51배)·Node(28.7ms,
5.22배)보다 빠르다고 보고했다.

Ant는 단일 런타임이 아니라 생태계를 표방한다.
자체 패키지 매니저, `ants.land` 패키지 레지스트리, 애플리케이션 배포·호스팅
플랫폼, 그리고 Electron과 유사한 역할을 하는 데스크톱 앱 빌더 Ant Desktop을
함께 제공한다.
목표는 이 구성 요소들이 하나의 통합 플랫폼으로 동작하면서도 기존
JavaScript/npm 생태계와 호환성을 유지하는 것이다.
저자는 Show HN 게시글에서 이전에도 런타임으로만 Ant를 소개한 적이 있으며,
그 이후 지금의 더 넓은 생태계로 성장했다고 밝혔다.

기술 사양 측면에서 Ant는 Ecma TC55가 만든 WinterTC Minimum Common API
표준을 목표로 하며, `compat-table` 스위트에서는 1511개 테스트 중 100%를
통과했다고 밝혔지만, 훨씬 더 포괄적인 `test262` 스위트에서는 약 64%
통과율에 그친다.
설치는 `curl -fsSL https://antjs.org/install | bash` 한 줄로 이루어지며,
GitHub 저장소는 이 글 작성 시점 기준 546 stars를 보유하고 있다.

## 분석

### “밑바닥부터 만든 엔진”이라는 주장이 프로젝트 전체의 신뢰도를 좌우한다

Ant의 핵심 세일즈 포인트는 기존 엔진을 재활용하지 않고 자체 엔진(Ant
Silver)과 자체 JIT 백엔드를 직접 구현했다는 점이다.
이는 매우 드문 성취로, V8이나 JSC 수준의 JavaScript 엔진을 개인 또는 소규모
팀이 실제로 새로 작성하는 것은 수년 단위의 노력이 드는 작업으로 알려져
있다.
그렇기 때문에 이 주장이 사실인지 여부는 프로젝트의 신뢰도에 결정적인
영향을 미치며, HN 커뮤니티가 이 주장 자체를 검증하려 든 것도 자연스러운
반응이다.
“작은 바이너리, 빠른 콜드 스타트”라는 결과 지표보다, 그 결과를 만들어낸
엔진이 실제로 독자적인 구현물인지가 이 프로젝트를 평가하는 핵심 축이 된다.

### 초경량 런타임 경쟁은 서버리스·엣지 컴퓨팅 흐름과 맞닿아 있다

Ant가 스스로 명시한 타깃 환경 — 서버리스 함수, 엣지 컴퓨팅, 임베디드
시스템, CLI 도구 — 은 최근 몇 년간 JavaScript 런타임 경쟁의 새로운 전선이다.
Cloudflare Workers, Deno Deploy, Vercel Edge Functions 같은 플랫폼이
콜드 스타트 지연을 핵심 성능 지표로 삼으면서, 런타임의 바이너리 크기와
초기화 속도가 곧 상품성으로 직결되는 구조가 만들어졌다.
Ant의 벤치마크가 콜드 스타트에 초점을 맞춘 것은 이 흐름을 정확히 겨냥한
선택이며, Node·Bun·Deno라는 이미 자리 잡은 경쟁자들과 정면으로 비교표를
제시한 것도 이 시장에서의 포지셔닝을 분명히 하려는 의도로 읽힌다.

### 생태계 확장은 신뢰가 검증되기 전에 이루어졌다

패키지 매니저, 레지스트리, 배포 플랫폼, 데스크톱 앱 빌더까지 한 번에
갖추겠다는 로드맵은 야심 차지만, 이 모든 구성 요소는 핵심 런타임 엔진의
신뢰성이 아직 커뮤니티에서 검증되지 않은 상태에서 동시에 제시되었다.
일반적으로 성공한 런타임(Bun, Deno)은 런타임 자체의 신뢰를 먼저 얻은 뒤
패키지 매니저나 배포 도구로 영역을 넓혀 갔다.
Ant는 이 순서를 뒤집어 처음부터 전체 스택을 함께 내놓았는데, 이는 초기
채택자에게 “이 모든 조각이 하나로 묶여야만 가치가 있다”는 진입 장벽을
동시에 지우는 전략이기도 하다.

## 비평

### “밑바닥부터 만들었다”는 핵심 주장이 커뮤니티 검증에서 흔들렸다

HN 댓글 CharlesW는 이 프로젝트가 실제로는 기존의 작은 JS 엔진인
Cesanta의 Elk(<https://github.com/cesanta/elk>, 2019년 공개)를 LLM으로
재작성한 결과물로 보인다고 지적했다[^CharlesW].
이 주장이 사실이라면, README가 내세우는 “밑바닥부터 만든 엔진”이라는
설명과 실제 구현 계보 사이에 상당한 간극이 존재하는 셈이다.
프로젝트의 정체성 자체가 “완전히 새로운 엔진”인지 “기존 엔진의 AI 보조
재작성”인지에 걸려 있는데, 이 구분은 벤치마크 숫자보다 훨씬 근본적인
문제이며, 저자 본인의 응답만으로는 완전히 해소되지 않았다.
비슷한 맥락에서 rererereferred는 이 프로젝트가 “vibecoded”(AI에게 코드를
맡겨 완성한) 방식으로 만들어진 것으로 보인다며, 자체 JS 엔진처럼 정교함이
요구되는 영역에서 그런 개발 방식은 이례적인 선택이라고 꼬집었다[^rererereferred].
한편 guessmyname은 Anthropic이 이미 `ant`라는 이름으로 자체 CLI 도구를
운영하고 있다는 점을 지적하며 이름 자체의 혼동 가능성을 제기했다[^guessmyname].

### AI 생성 코드의 보안 이슈가 별도로 제기됐다

HN 댓글 tekacs는 이 프로젝트에서 3월경 보고된 이슈를 언급하며, 저자가
이후 다시 작성했다고 주장하긴 했지만 “밑바닥부터 손으로 작성했다”는
설명과 실제 개발 방식 사이의 불일치를 지적했다.
런타임처럼 신뢰 경계가 중요한 소프트웨어에서 AI 생성 코드의 비중과 검증
수준은 부차적인 정보가 아니라 채택 여부를 결정하는 핵심 정보다.
저자가 코드의 기원을 명확히 공개하지 않은 채 “새로 만들었다”는 서술만
강조한 것은, 프로젝트의 실제 위험 프로필을 사용자가 정확히 판단하기
어렵게 만든다.
tekacs 본인은 AI로 코딩하는 것 자체에 반대하지 않는다고 밝히면서도, 이
프로젝트의 3월 이슈에서 드러난 “밑바닥부터 손으로 작성”이라는 설명과
실제 개발 방식 사이의 불일치는 별개로 짚어야 할 문제라고 강조했다[^tekacs].

### 벤치마크는 유리한 시나리오만 선택했을 가능성이 있다

README의 콜드 스타트 벤치마크는 Hono 앱 로드와 라우트 등록이라는 매우
좁은 시나리오 하나만 다룬다.
실제 프로덕션 워크로드에서 중요한 것은 콜드 스타트뿐 아니라 지속 실행 시의
처리량, 메모리 사용량, GC 지연, 그리고 무엇보다 test262 통과율이 약 64%에
불과하다는 점에서 드러나는 스펙 준수 수준이다.
저자가 콜드 스타트라는 자신에게 가장 유리한 지표만 전면에 내세운 것은
자연스러운 마케팅이긴 하지만, 이것만으로 “Node 대체 가능한 범용 런타임”을
판단하기에는 근거가 얕다.

## 인사이트

### 신뢰 검증되지 않은 엔진 주장은 프로젝트의 채택 속도 자체를 제한하는 구조적 리스크다

Ant의 사례는 오픈소스 인프라 프로젝트에서 “핵심 주장의 진위”가 기능
자체보다 채택 속도를 좌우한다는 것을 보여준다.
바이너리 크기나 콜드 스타트 숫자가 아무리 매력적이어도, 엔진의 출처에 대한
의심이 해소되지 않으면 프로덕션 환경에 도입하려는 팀은 관망 모드에
머무른다.
이는 신뢰가 한 번 흔들리면 후속 기능(패키지 매니저, 배포 플랫폼)까지
연쇄적으로 신뢰를 얻기 어려워지는 구조적 리스크로 이어지며, Ant처럼 여러
구성 요소를 동시에 내놓는 프로젝트일수록 이 리스크가 더 크게 증폭된다.

### LLM 보조 개발 시대에는 “직접 작성”이라는 표현 자체가 재정의되어야 한다

AI 코딩 도구가 보편화되면서 “밑바닥부터 만들었다”는 표현의 의미가
모호해지고 있다.
과거에는 이 표현이 명확히 “타인의 코드를 참조하지 않고 직접 작성”을
뜻했지만, LLM이 기존 오픈소스 구현(Elk 같은)을 학습 데이터로 흡수한 뒤
유사한 구조의 코드를 생성해내는 상황에서는 “직접 작성”과 “기존 구현의
AI 재구성” 사이의 경계가 실질적으로 흐려진다.
Ant 사례가 던지는 더 큰 질문은 이 프로젝트 하나의 진위 여부를 넘어,
오픈소스 커뮤니티가 AI 생성 코드의 계보를 어떻게 검증하고 표기할
것인가라는 표준의 부재다.
이 표준이 마련되지 않는 한, 비슷한 논란은 다른 프로젝트에서도 반복될
가능성이 높다.

### 초경량 런타임 경쟁은 결국 스펙 준수도의 병목에서 갈릴 것이다

Ant가 콜드 스타트에서 압도적인 숫자를 보여주면서도 test262 통과율이
64%에 머무른다는 사실은, 초경량·고성능 런타임 경쟁의 다음 단계가 어디로
향할지를 시사한다.
새 엔진을 만드는 것은 “빠르게 만드는 것”보다 “표준을 완전히 준수하면서
빠르게 만드는 것”이 훨씬 어렵다.
Bun과 Deno가 초기에 겪었던 호환성 이슈들이 몇 년에 걸쳐 서서히 해소되어
온 과정을 고려하면, Ant 역시 프로덕션급 런타임으로 인정받기까지는 벤치마크
숫자 개선보다 스펙 준수도를 끌어올리는 지루하고 오래 걸리는 작업이 훨씬
크게 남아 있을 것이다.
이는 신생 런타임 프로젝트를 평가할 때 초기 벤치마크 숫자보다 스펙 준수율
추이를 장기 지표로 지켜봐야 한다는 일반적인 교훈으로 이어진다.
