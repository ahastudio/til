# JavaScript의 새로운 기능들: ES2025와 ES2026

원문: <https://neciudan.dev/whats-new-in-javascript>

## 요약

TC39 위원회는 Ecma International 산하에서 JavaScript 표준을 관리하며,
Stage 0부터 Stage 4까지의 단계적 제안 프로세스를 통해 매년 새 기능을 추가한다.
Stage 4에 도달한 기능은 연간 스냅샷에 통합되어 공식 표준이 된다.

ES2025는 이미 주요 브라우저에서 사용 가능한 기능을 포함한다.
이터레이터 헬퍼(Iterator Helpers)는 `.map()`, `.filter()`, `.take()`, `.drop()`,
`.flatMap()`, `.reduce()`, `.toArray()`를 추가해
무한 시퀀스나 스트리밍 데이터에 중간 배열 없이 지연 평가(lazy evaluation)를 적용할 수 있다.
Set에는 `union()`, `intersection()`, `difference()`, `symmetricDifference()` 등
수학적 집합 연산이 추가됐다.
JSON 모듈 가져오기는 `import config from './data.json' with { type: 'json' }` 구문으로
MIME 타입 스푸핑 공격을 방지한다.
`Promise.try()`는 동기·비동기 오류 처리를 통합하고,
`RegExp.escape()`는 사용자 입력의 정규식 특수문자를 안전하게 이스케이프하며,
`Float16Array`는 GPU·ML 워크플로를 위한 반정밀도 부동소수점 배열을 제공한다.

ES2026은 표준화 승인이 완료되어 브라우저 구현이 진행 중이다.
`Math.sumPrecise()`는 Shewchuk 알고리즘으로 부동소수점 누적 오차를 방지한다.
`Uint8Array` 변환 메서드(`.toBase64()`, `.toHex()`, `Uint8Array.fromBase64()`, `Uint8Array.fromHex()`)는
외부 라이브러리 없이 바이너리·텍스트 인코딩을 처리한다.
`Error.isError()`는 iframe, Worker, VM 컨텍스트를 가로질러 신뢰할 수 없었던
`instanceof Error` 검사를 대체한다.
`Iterator.concat()`은 여러 이터레이터를 체이닝하고,
`Map.getOrInsert()`·`getOrInsertComputed()`는 캐싱·카운팅 패턴에서
조회와 삽입을 원자적으로 처리한다.
`Array.fromAsync()`는 비동기 이터러블을 배열로 수집하며,
`JSON.parse()` 소스 텍스트 컨텍스트는 리바이버 함수에서 대형 숫자를 정밀하게 처리할 수 있게 한다.

Stage 4 달성으로 ES2027 대상이 된 Temporal은 `Date` 객체를 완전히 대체하는
불변·타임존 인식 날짜·시간 API다.
`ZonedDateTime`, `PlainDate`, `PlainTime` 등 분리된 타입을 제공하며
직관적인 산술 연산과 파싱을 지원한다.
`using` 키워드(Stage 3)는 스코프 종료 시 `[Symbol.dispose]`를 자동 호출해
try/finally 패턴을 대체하는 리소스 정리 구문이다.
Import Defer(Stage 3)는 `import defer * as heavy from './heavy.js'` 구문으로
모듈 평가를 네임스페이스 접근 시점까지 지연해 초기 로딩 성능을 개선한다.

데코레이터, Records/Tuples, 파이프라인 연산자, 패턴 매칭은 설계 복잡성으로
수년간 낮은 Stage에 머물고 있다.

## 분석

### TC39 프로세스는 안정성을 위해 채택 속도를 희생한다

Stage 0부터 Stage 4까지의 단계는 하나의 기능이 표준이 되기까지
최소 수년이 걸리는 구조다.
두 개의 독립적 구현이 존재해야 Stage 4가 되는 요건은
특정 엔진 벤더의 독점적 영향력을 방지하지만,
동시에 어떤 기능도 단독으로 표준화를 추진할 수 없음을 의미한다.
Temporal이 명백히 필요한 기능임에도 Stage 4에 도달하기까지 10년 가까이 걸린 것은
이 구조의 직접적 결과다.

이 느린 속도는 의도된 것이다.
npm 생태계처럼 불안정한 패키지가 빠르게 보급되는 환경과 달리,
언어 명세 변경은 한번 배포되면 하위 호환성을 수십 년간 유지해야 한다.
`Array.prototype.flatten()` 제안이 MooTools 호환성 문제로 `flat()`으로
이름이 바뀐 “SmooshGate” 사건은 이 제약이 실제임을 보여준다.

그러나 이 구조의 비용이 있다.
데코레이터, 파이프라인 연산자, 패턴 매칭처럼 수년간 Stage 2-3에 머무는 기능들은
커뮤니티가 TypeScript, Babel, 각종 라이브러리로 대안을 구현하게 만든다.
표준화 지연이 파편화를 낳는 역설이다.

### 이터레이터 헬퍼와 Set 연산은 “빠진 기능”을 채운다

Python, Rust, Kotlin 같은 언어들은 이미 오래전부터
컬렉션과 이터레이터에 대한 함수형 파이프라인을 내장했다.
JavaScript는 `Array` 메서드로 유사한 패턴을 지원했지만
이터러블 프로토콜을 따르는 비-Array 컬렉션 — `Set`, `Map`, 제너레이터, Web API 스트림 —
에는 동일한 메서드가 없었다.
이터레이터 헬퍼는 이 불일치를 해소한다.

Set 연산도 마찬가지다.
수학적 집합 연산은 교과서적 자료구조 기능인데도
JavaScript `Set`은 2015년 ES6에 추가된 이후 9년간 이 메서드들이 없었다.
`lodash.union`, `lodash.intersection` 등이 수백만 번 다운로드된 이유가 여기 있다.
표준이 생태계의 부재를 결국 흡수하는 패턴이 반복된 것이다.

### Temporal은 `Date`의 실패를 재설계로 해결한다

JavaScript `Date`는 Java의 `java.util.Date`를 1995년 10일 만에 복사한 것으로
알려져 있다.
그 `java.util.Date`는 이미 2014년 Java 8의 `java.time` 패키지로 대체됐다.
JavaScript는 같은 문제를 2026-2027년에야 해결하는 것이다.

Temporal의 설계가 `Date`와 다른 핵심은 불변성과 타입 분리다.
`PlainDate`, `PlainTime`, `ZonedDateTime`이 분리된 것은
“날짜”와 “시간”과 “타임존을 포함한 절대 시각”이 서로 다른 개념이라는 인식에서 나온다.
기존 `Date`가 하나의 타입으로 이 모든 것을 표현하려 했기 때문에 혼란이 생겼다.

### `Float16Array`와 `Math.sumPrecise()`는 JavaScript의 사용 영역 확장을 반영한다

`Float16Array`는 GPU 컴퓨팅과 머신러닝 추론이 브라우저와 Node.js에서 실행되는
현실에 대응한 기능이다.
WebGPU, WASM, ONNX Runtime Web이 JavaScript 런타임에서 ML 모델을 실행하는 시대에
반정밀도 부동소수점을 언어 수준에서 지원하는 것은 당연한 수순이다.

`Math.sumPrecise()`가 ES2026에 포함된 것도 같은 맥락이다.
부동소수점 누적 오차는 대규모 수치 계산이나 금융 계산에서 치명적 버그를 유발한다.
JavaScript가 서버 사이드, ML 파이프라인, 금융 서비스에 더 많이 사용될수록
이런 수치 정밀도 문제의 중요성이 커진다.

## 비평

### 글이 “쓸 수 있다”와 “써야 한다”를 구분하지 않는다

이 글은 ES2025·ES2026 기능들이 무엇인지를 잘 설명하지만,
각 기능이 어떤 상황에서 기존 패턴보다 실제로 유리한지는 다루지 않는다.
예를 들어 이터레이터 헬퍼가 중간 배열 없이 지연 평가를 한다는 것은 사실이지만,
대부분의 실용적 데이터셋에서 이 차이가 성능에 의미 있게 영향을 미치는가?
`Array` 메서드 체이닝이 이미 대부분의 상황에서 충분히 빠른데
이터레이터로 마이그레이션할 동기가 있는가?

`Math.sumPrecise()`도 같은 문제가 있다.
Shewchuk 알고리즘 기반 정밀 합산이 필요한 맥락은 과학 계산이나 금융 정밀도가
요구되는 특수한 상황이다.
대부분의 JavaScript 개발자가 마주치는 부동소수점 문제는 `0.1 + 0.2 !== 0.3` 같은
비교 오류이며, 이것은 `Math.sumPrecise()`로 해결되지 않는다.
새 기능이 어떤 문제를 해결하는지를 설명하면서
그 문제가 얼마나 흔한지는 다루지 않는 것이다.

### 브라우저 지원 현황을 생략하면 글의 실용성이 떨어진다

글은 ES2025 기능들이 “출시됐다”고, ES2026 기능들이 “표준화 승인됐다”고 서술하지만
실제 브라우저 지원 현황을 제공하지 않는다.
`Float16Array`는 Chrome 120+에서 지원하지만 Firefox 지원이 제한적이다.
이터레이터 헬퍼는 Chrome 122+, Firefox 131+, Safari 17.4+에서 지원한다.
`Math.sumPrecise()`와 `Uint8Array` 변환 메서드는 아직 주요 브라우저에 미배포된 상태다.

“ES2025에 포함됐다”는 것이 “모든 환경에서 안전하게 사용 가능하다”를 의미하지 않는다.
폴리필이 필요한지, 트랜스파일러가 지원하는지, Node.js 최소 버전이 무엇인지
이런 정보 없이 기능을 소개하는 것은 독자를 오도할 수 있다.

### 글이 AI 코딩 도구의 한계를 문제로 설정하지만 해결책은 광고다

글의 말미는 ES2025 이전에 훈련된 AI 코딩 어시스턴트가 낡은 패턴을 생성한다는
실제 문제를 지적한다.
이것은 유효한 관찰이다.
하지만 그 해결책으로 제시하는 “react-tips-skill 플러그인 설치”와
`.cursorrules` 템플릿은 저자의 자체 제품이다.
기술 글쓰기에서 문제 제기와 자사 제품 추천을 같은 글에 결합하는 것은
글의 중립성을 훼손한다.

더 근본적인 비판은 이렇다.
AI 어시스턴트가 ES2025 패턴을 모른다는 문제는
스킬 파일을 설치하는 방식이 아니라 모델 훈련 데이터와 시스템 프롬프트 수준에서
해결되어야 하는 문제다.
개별 개발자가 프로젝트마다 스킬 파일을 설치하는 것은
인프라 문제를 애플리케이션 레이어에서 해결하려는 시도다.

## 인사이트

### 언어 표준 사이클이 짧아질수록 “언어 버전”은 의미를 잃는다

TC39가 매년 새 기능을 추가하는 체계에서 “ES2025를 지원한다”는 선언은
점점 모호해진다.
브라우저는 Stage 4 도달 이전부터 실험적 구현을 배포하고,
일부 Stage 3 기능은 Stage 4 기능보다 먼저 광범위하게 사용된다.
Temporal은 Stage 4가 됐지만 아직 주요 브라우저에 완전히 배포되지 않았고,
`using` 키워드는 Stage 3이지만 TypeScript 5.2에서 이미 지원한다.

이것은 Python 2→3처럼 버전이 명확한 경계를 만들던 방식과 다르다.
JavaScript의 진짜 버전은 사실상 “특정 엔진의 특정 버전”이다.
caniuse.com과 node.green 같은 호환성 매트릭스가 ECMA 스펙 문서보다
실용적으로 중요해지는 이유다.
이 현실에서 “ES2025의 새 기능”이라는 프레임은
실제로 쓸 수 있는 기능의 가이드로는 불완전하다.

### Set 연산과 이터레이터 헬퍼의 추가는 라이브러리 의존성의 수명을 재정의한다

`lodash.union`, `lodash.intersection`, `lodash.flatten`, `lodash.chunk` — 이 함수들은
JavaScript 표준 라이브러리의 부재를 채운 유틸리티였다.
ES2025가 Set 연산을, ES6가 `Array.flat()`을, ES2019가 `Object.fromEntries()`를
추가하면서 lodash의 사용 근거가 조금씩 약해졌다.

이 패턴은 프로그래밍 언어 생태계에서 반복적으로 나타난다.
Go는 거의 모든 것을 표준 라이브러리에 포함해 서드파티 의존성을 최소화했다.
Rust는 `serde`, `tokio`, `rayon` 같은 핵심 생태계를 언어 외부에서 발전시켰다.
JavaScript는 후자에 가까웠지만 점점 전자 방향으로 이동하고 있다.

문제는 이 이동이 기존 코드베이스에 기술 부채를 만든다는 것이다.
`lodash.set()`은 여전히 깊은 객체 경로 설정에 표준 대안이 없다.
표준이 라이브러리를 완전히 대체하는 시점은 오지 않을 수 있으며,
결과적으로 “일부는 표준, 일부는 라이브러리”인 혼재 상태가 장기화된다.

### `using` 키워드는 JavaScript에서 RAII 패턴의 첫 번째 공식 구현이다

C++의 RAII(Resource Acquisition Is Initialization), Rust의 Drop trait,
C#의 `using` 블록, Python의 `with` 문 — 리소스 자동 해제는
시스템 언어와 현대 언어에서 오래된 패턴이다.
JavaScript는 가비지 컬렉션 덕분에 메모리 관리를 추상화했지만
파일 핸들, 데이터베이스 커넥션, WebSocket 같은 외부 리소스는
여전히 수동으로 해제해야 했다.
`using` 키워드는 이 공백을 언어 차원에서 채운다.

흥미로운 것은 타이밍이다.
Node.js가 서버 사이드에서 주요 플랫폼이 된 지 15년이 지났고
파일 시스템, 네트워크, 데이터베이스 리소스 관리는 일상적인 작업이 됐다.
그런데도 이 기능이 이제야 표준화되는 것은
언어 설계자들이 가비지 컬렉션을 제공하는 언어에서 명시적 리소스 관리가
얼마나 중요한지를 늦게 인식했음을 시사한다.
또는 `try/finally` 패턴이 불편하지만 허용 가능한 수준이었기 때문에
우선순위가 낮았을 수도 있다.

`await using`이 비동기 리소스 해제까지 지원하는 것은
JavaScript의 비동기 중심 실행 모델에 맞춘 설계다.
다른 언어의 동기적 RAII와 달리, 비동기 컨텍스트에서 자원을 정리하는 패턴은
JavaScript 고유의 문제였고 `using`이 이를 우아하게 해결한다.
