# JavaScript 날짜 계산의 함정

원문: <https://philna.sh/blog/2026/01/11/javascript-date-calculation/>

## 요약

Phil Nash는 JavaScript에서 날짜에 한 달을 더하는 간단한 연산을 수행하다가
예상치 못한 결과를 마주쳤다.
UTC 기준 2024년 1월 1일에서 한 달을 더했더니 결과가 2023년 3월 4일, 즉 시간이
거꾸로 흘렀다.

원인은 타임존 혼용이었다.
UTC 기준 2024년 1월 1일 자정은 태평양 표준시(UTC-8)로는 2023년 12월 31일
오후 4시다.
`Date` 객체가 UTC로 생성됐어도, `setMonth()` 메서드는 로컬 타임존을 기준으로
동작한다.
로컬 시간이 12월 31일인 상태에서 `setMonth(1)`(2월)을 호출하면 2월 31일이
만들어진다.
JavaScript는 이 존재하지 않는 날짜를 오버플로해 3월 3일로 처리하고,
이를 다시 UTC로 변환하면 3월 4일이 된다.
결과적으로 앞으로 간 것이 아니라 뒤로 간 날짜가 출력된다.

즉각적인 수정 방법은 `setMonth()` 대신 `setUTCMonth()`와 `setUTCDate()`를
사용하는 것이다.
UTC 전용 메서드는 로컬 타임존이 아닌 UTC 표현을 직접 조작하기 때문에
타임존 오버플로가 발생하지 않는다.

근본적인 해결책으로는 JavaScript의 `Temporal` API가 소개된다.
`Temporal.PlainDate`는 시간과 타임존 정보가 없는 순수 날짜 타입으로,
날짜 조작을 예측 가능하고 불변(immutable) 방식으로 처리한다.
저자는 타임존을 다루지 않더라도 JavaScript Date 작업에서는 항상 타임존을
염두에 두어야 한다고 결론짓는다.

## 분석

### 버그 재현 경로의 구조적 문제

이 버그가 발생하는 데는 세 가지 조건이 동시에 맞아야 한다.
첫째, UTC로 날짜를 생성한다.
둘째, 로컬 타임존이 UTC보다 뒤에 있어 날짜가 하루 바뀐다.
셋째, 로컬 날짜 기준으로 다음 달에 해당 일자가 없다(예: 월말).
이 세 조건이 동시에 충족될 때만 버그가 드러나기 때문에, 대부분의 테스트 환경에서
재현되지 않는다.
UTC 타임존으로 설정된 CI 서버에서는 통과하고, 태평양 시간대 개발자 환경에서
특정 날짜에만 실패하는 종류의 버그다.

### setMonth()와 setUTCMonth()의 설계 비일관성

JavaScript `Date` 객체는 내부적으로 UTC 밀리초 타임스탬프를 저장한다.
그러나 `getMonth()`, `setMonth()` 같은 메서드는 로컬 타임존으로 변환한 값을
기준으로 동작한다.
반면 `getUTCMonth()`, `setUTCMonth()`는 UTC 기준으로 동작한다.
같은 객체에 두 가지 다른 기준의 메서드가 공존하는 이 설계는,
어느 메서드를 써야 하는지 매번 의식적으로 선택하도록 강제한다.
이 복잡성은 `Date` 객체가 1995년 Netscape에서 열흘 만에 설계된 역사적 산물이다.

### Temporal API의 타입 설계 철학

`Temporal.PlainDate`는 “날짜”와 “시각”을 분리한다는 명확한 설계 원칙을 갖는다.
날짜만 필요하면 `PlainDate`, 시각이 포함되면 `PlainDateTime`, 타임존까지
필요하면 `ZonedDateTime`을 쓴다.
이 타입 계층은 “지금 내가 다루는 것이 무엇인가”를 명시하도록 강제한다.
기존 `Date` 객체는 모든 것을 하나의 타입으로 처리해서 혼용 오류를 런타임까지
숨겼다.

## 비평

### 강점: 버그 재현 과정의 단계별 추적

이 글의 강점은 버그가 어떻게 발생했는지를 단계적으로 분해한 것이다.
“UTC 자정 → 로컬 전날 → February 31 → 오버플로 → UTC 변환”의 각 단계가 명확하다.
처음에는 마술처럼 보이는 결과가 인과관계로 설명되는 과정이 설득력 있다.

### 강점: 즉각적 수정과 장기적 해결책을 모두 제시

`setUTCMonth()` 같은 즉각 적용 가능한 수정과, Temporal API 같은 미래 지향적
해결책을 함께 제시하는 구조는 실용적이다.
현재 코드베이스가 있는 독자와 새 프로젝트를 시작하는 독자 모두에게 유용하다.

### 약점: Temporal API 호환성 미언급

Temporal API는 2026년 1월 기준으로 여전히 Stage 3 제안(proposal)이다.
주요 브라우저에서의 지원 현황, 폴리필(polyfill) 사용 가능 여부,
Node.js 지원 버전에 대한 정보가 없다.
“더 나은 해결책”으로 소개하면서 실제 사용 가능한 조건을 명시하지 않으면
독자가 잘못된 기대를 가질 수 있다.

### 약점: date-fns, Day.js 같은 라이브러리 대안 미검토

날짜 조작 라이브러리들은 이런 타임존 문제를 오랫동안 해결해 왔다.
`date-fns`의 `addMonths`, Luxon의 `DateTime`, Day.js의 플러그인 시스템은
모두 이 문제를 다룬다.
Temporal API 이전의 현실적 대안으로 이 라이브러리들을 비교했으면
더 완결된 가이드가 됐을 것이다.

## 인사이트

### 타임존 버그는 “맥락 의존적 실패”의 교과서적 사례다

이 버그는 코드가 틀린 것이 아니다.
`setMonth()`는 명세대로 동작한다.
문제는 코드가 실행되는 맥락(로컬 타임존)이 코드 작성자의 의도(UTC)와 달랐다는 것이다.
이런 류의 버그는 코드 리뷰에서 잡기 어렵고, 유닛 테스트에서도 타임존이
고정되지 않으면 재현되지 않는다.

맥락 의존적 실패는 소프트웨어에서 가장 찾기 어려운 버그 유형이다.
플로팅 포인트 정밀도 오류, 로케일별 문자열 정렬 차이, OS별 파일 경로 처리 차이가
같은 계열이다.
이들의 공통점은 코드 자체는 올바르지만, 실행 환경의 암묵적 가정이 다를 때
실패한다는 것이다.
방어 전략은 두 가지다.
암묵적 가정을 명시적으로 만들거나(UTC 메서드 사용), 가정이 없는 타입을
쓰는 것이다(`Temporal.PlainDate`).

JavaScript의 `Date` 객체가 이 문제를 특히 잘 드러내는 것은, 같은 객체가
로컬과 UTC 두 가지 시각을 동시에 표현하기 때문이다.
어느 시각을 기준으로 연산하는지 메서드 이름으로만 구분해야 하며,
IDE나 타입 시스템이 이 실수를 잡아주지 못한다.
이것은 API 설계의 실패이지 사용자의 실수가 아니다.

### 오버플로 허용 설계가 만드는 침묵하는 버그

`new Date(2024, 1, 31)` — 2월 31일 — 은 오류를 던지지 않는다.
JavaScript는 조용히 3월 3일(또는 2일, 윤년에 따라 다름)을 반환한다.
이 “편의 기능”은 `setMonth()`와 조합될 때 타임존 버그를 완전히 숨긴다.
만약 `Date` 생성자가 존재하지 않는 날짜에 예외를 던진다면,
이 버그는 훨씬 일찍 발견됐을 것이다.

오버플로 허용은 달력 라이브러리가 자주 제공하는 기능이다.
“다음 달 말일”을 계산할 때 유용하다.
그러나 예상치 못한 오버플로와 의도적 오버플로를 구분할 방법이 없다면,
버그는 결과값을 직접 검증하지 않는 이상 탐지되지 않는다.
`Temporal` API가 기본적으로 오버플로를 거부(reject)하고 명시적으로 허용해야
하는 설계를 선택한 것은 이 교훈을 반영한다.

이 패턴은 “방어적 기본값(defensive defaults)”이라는 설계 원칙과 연결된다.
Rust의 정수 오버플로는 디버그 빌드에서 패닉을 일으키고, 릴리즈 빌드에서는
명시적 래핑(wrapping)을 요구한다.
Python 3는 정수 나눗셈 결과 타입을 float으로 바꿨다(암묵적 정수 반환 제거).
이런 변화들은 모두 “암묵적 허용”보다 “명시적 선택”을 강제하는 방향이다.

### 30년 된 API의 교체 비용과 Temporal의 도전

`Date` 객체는 1995년에 Java의 `java.util.Date`를 10일 만에 모방해 만들어졌다.
아이러니하게도 Java는 이 설계의 결함을 인식하고 2014년 Java 8에서
`java.time` 패키지(JSR-310)로 교체했다.
JavaScript는 30년이 지난 2026년에야 Temporal로 교체를 시도하고 있다.

이 지연의 이유는 기술적이 아니라 생태계 때문이다.
수십억 개의 웹 페이지, 수백만 개의 NPM 패키지, 수천 개의 프레임워크가
`Date` 객체를 사용한다.
기존 `Date`를 제거하지 않고 `Temporal`을 추가하는 것은 호환성을 유지하는
유일한 방법이지만, 개발자가 두 API 사이에서 선택해야 하는 혼란을 만든다.

Java의 `java.time`도 처음에는 `java.util.Date`와 공존했고, 생태계가 전환하는
데 수년이 걸렸다.
JavaScript의 Temporal도 같은 과정을 겪을 것이다.
“더 나은 API가 있다”는 것과 “생태계가 전환한다”는 것 사이에는 수년의 간극이 있다.
그 전환 기간 동안 `setUTCMonth()` 같은 방어적 코딩 습관이 여전히 중요한
이유다.
