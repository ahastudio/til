# QRCode.js — 의존성 없는 QR 코드 생성 라이브러리

<https://github.com/davidshimjs/qrcodejs>

<https://davidshimjs.github.io/qrcodejs/>

## 소개

QRCode.js는 David Shim이 공개한 순수 JavaScript QR 코드 생성 라이브러리다.
외부 의존성이 전혀 없고, 오래된 Internet Explorer 6부터
Chrome, Firefox, Safari, Opera, 모바일 Safari, Android, Windows Mobile까지
폭넓은 브라우저를 지원한다.

사용법은 단순하다.
`new QRCode(element, “text”)`처럼 대상 DOM 엘리먼트와 문자열만 넘기면 QR 코드가 그려진다.
크기, 색상, 오류 정정 수준(`correctLevel`) 등을 옵션 객체로 지정할 수도 있다.
생성된 QR 코드는 `clear()`로 지우고 `makeCode()`로 다시 그릴 수 있어
동적으로 내용을 바꾸는 UI에도 쓸 수 있다.

핵심 QR 인코딩 로직은 자체 구현이 아니라 이식(移植) 결과물이다.
`qrcode.js` 파일 상단에는 “Copyright (c) 2009 Kazuhiko Arase”라는 표기와
MIT 라이선스 문구가 있고, 원본은 Kazuhiko Arase가 만든
“QR Code for JavaScript” 라이브러리다.
Reed-Solomon 오류 정정을 구현하는 `QRPolynomial`,
QR 매트릭스를 생성하는 `QRCodeModel`,
비트 단위 인코딩을 담당하는 `QRBitBuffer` 클래스가 이 라이브러리의 핵심이다.
렌더링은 SVG, HTML `<table>`, Canvas 세 가지 방식을 지원하며,
브라우저 기능을 감지해 SVG 문서에서는 SVG로,
Canvas를 지원하지 않는 구형 브라우저에서는 테이블로,
그 외에는 Canvas로 자동 전환한다.
Data-URI를 지원하지 않는 구형 Android 기기를 위한 별도 처리도 들어 있다.

라이브러리는 2015년 이후 사실상 유지보수가 멈췄다.
마지막 의미 있는 커밋은 2015년 11월이고,
그 이후로는 새 기능이나 버그 수정 없이 정지된 상태다.
그럼에도 GitHub 스타는 14,000개를 넘고, 포크는 4,700개를 넘는다.
열린 이슈도 238개에 달하지만 응답이 거의 없다.

## 분석

### “충분히 작은 문제”는 유지보수 없이도 오래 산다

QRCode.js가 10년 넘게 방치되고도 여전히 널리 쓰이는 이유는
QR 코드 생성이라는 문제 자체가 시간이 지나도 변하지 않는 유한한 문제이기 때문이다.
QR 코드 표준(ISO/IEC 18004)은 안정적으로 고정되어 있고,
브라우저의 Canvas/SVG/DOM API도 하위 호환성을 강하게 지키는 영역이다.
새로운 기능을 추가할 필요도, 외부 API 변경에 대응할 필요도 없는 라이브러리는
한 번 정확하게 구현되면 유지보수 없이도 계속 작동한다.
이는 프레임워크나 API 클라이언트처럼 외부 환경 변화에 종속된 라이브러리와
근본적으로 다른 생존 조건이다.

### 이식(porting)이 원작자보다 더 널리 알려지는 비대칭

QRCode.js는 사실 David Shim의 독자적인 알고리즘 구현이 아니라
Kazuhiko Arase의 “QR Code for JavaScript”를 브라우저 친화적인 API로 감싼 것이다.
그런데도 정작 널리 알려지고 검색되는 이름은 원작자가 아니라 이식자 쪽이다.
이는 오픈소스 생태계에서 반복되는 패턴을 보여준다.
핵심 알고리즘을 만드는 일과, 그것을 개발자가 쓰기 편한 API로 감싸
접근성을 높이는 일은 완전히 다른 종류의 기여이며,
후자가 채택률과 인지도 면에서 더 큰 보상을 받는 경우가 많다.
저작권 표기가 파일 안에 남아 있다는 사실만으로는
이 비대칭을 사용자 대다수가 인지하기 어렵다.

### 세 가지 렌더링 전략의 계층은 특정 시대의 브라우저 파편화를 그대로 반영한다

SVG, Canvas, HTML 테이블이라는 세 가지 폴백을 순서대로 시도하는 구조는
2012년 당시 IE6 같은 구형 브라우저가 여전히 무시할 수 없는 점유율을 가졌던
브라우저 파편화 시대의 산물이다.
지금 새로 이런 라이브러리를 설계한다면 Canvas나 SVG 단일 경로로도 충분하겠지만,
당시에는 Canvas조차 지원하지 않는 환경을 감안해야 했다.
이 구조는 라이브러리가 만들어진 시점의 웹 생태계를
코드베이스 안에 화석처럼 남겨 놓은 사례다.

## 비평

### 방치된 코드가 여전히 기본 추천으로 남는 것은 생태계의 검색 관성 때문이다

QRCode.js는 유지보수가 끊긴 지 10년이 되어가지만,
“javascript qr code” 같은 검색어를 입력하면 여전히 상위에 노출되고
스택오버플로 답변이나 튜토리얼에서 기본값처럼 인용된다.
이는 라이브러리의 품질이나 활성 유지보수 여부와 무관하게,
먼저 인지도를 확보한 프로젝트가 검색 결과와 커뮤니티 지식 안에서
계속 재생산되는 관성을 보여준다.
qrcode-generator, qrcode(npm), qr-code-styling처럼
더 활발히 유지보수되고 최신 브라우저 API에 맞춰 개선된 대안이 있음에도,
이 관성 때문에 신규 개발자가 굳이 구식 라이브러리를 먼저 접하게 되는 구조적 문제가 있다.

### 238개의 열린 이슈는 “안정적”이라는 평가와 “방치되었다”는 평가를 동시에 뒷받침한다

이 라이브러리를 향한 흔한 평가는 “단순하고 안정적이라 유지보수가 필요 없다”는 것이다.
하지만 열린 이슈가 238개나 쌓여 있다는 사실은 이 평가를 그대로 받아들이기 어렵게 만든다.
버그 리포트와 기능 요청이 응답 없이 누적되는 상태를
“완성되어 더 손댈 곳이 없다”고 해석할지,
“관리자가 떠났다”고 해석할지는 완전히 다른 결론으로 이어진다.
실제로 큰 한글·다국어 문자열 인코딩 문제,
고해상도(Retina) 디스플레이에서의 렌더링 이슈 등
실사용에서 부딪히는 문제들이 이슈 트래커에 방치되어 있다.
“단순해서 안정적”이라는 서사는 유지보수 부재를 정당화하는 데
편리하게 쓰이지만, 실제 사용자가 겪는 문제를 가리는 효과도 있다.

### MIT 라이선스와 상표권 경고가 동시에 존재하는 파일은 법적 모호함을 사용자에게 떠넘긴다

파일 상단에는 MIT 라이선스로 자유로운 사용을 허용한다고 명시하면서,
동시에 “QR Code”가 덴소웨이브(DENSO WAVE)의 등록 상표라는 경고도 함께 적어 놓았다.
이는 코드 자체의 재사용은 자유롭지만
“QR Code”라는 명칭을 상업적으로 사용할 때는 별도의 상표권 문제가 있을 수 있다는
경고인데, 정작 그 경계가 어디인지에 대한 설명은 없다.
라이브러리를 그대로 가져다 쓰는 개발자 입장에서는
이 상표권 조항이 실무에 어떤 영향을 미치는지 판단할 근거가 부족한 채로
그저 파일 안의 문구만 마주하게 된다.

## 인사이트

### “완성된 소프트웨어”라는 개념은 웹 생태계에서 예외적으로만 성립한다

대부분의 소프트웨어는 지속적인 업데이트를 전제로 설계되고 평가받지만,
QRCode.js처럼 문제 영역이 고정되어 있고 외부 API 의존이 거의 없는 소프트웨어는
“더 이상 손댈 필요가 없는 완성 상태”에 실제로 도달할 수 있다.
이런 사례는 드물기 때문에 오히려 소프트웨어 유지보수에 대한 우리의 기본 가정,
즉 “관리자가 응답하지 않으면 죽은 프로젝트”라는 통념을 재검토하게 만든다.
문제는 이 예외적인 안정성이 QR 코드 생성처럼 표준이 고정된 극소수 영역에서만
성립하고, 그 경계를 사용자가 사전에 판단하기 어렵다는 데 있다.
결과적으로 개발자들은 실제로는 방치된 프로젝트와
진짜로 완성된 프로젝트를 구별하지 못한 채
같은 신뢰를 부여하는 경우가 많다.

### 이식 라이브러리의 인지도 역전은 AI 시대에 더 심해질 가능성이 크다

Kazuhiko Arase의 원본보다 David Shim의 브라우저 래퍼가 더 유명해진 현상은
앞으로 LLM이 코드를 추천하는 방식에서 더 증폭될 수 있다.
LLM은 학습 데이터에서 더 많이 언급되고 스타가 많은 저장소를 우선 추천하는 경향이 있는데,
이는 실제 알고리즘 구현의 원천이 아니라 접근성 좋은 래퍼를 계속 재생산하는 결과로 이어진다.
원작자의 기여가 코드 주석 속 저작권 표기로만 남고
생태계 전체의 검색 가능성에서는 사라지는 현상은,
오픈소스 기여의 가치를 “얼마나 인용되는가”로만 측정할 때 발생하는
구조적 왜곡을 보여주는 사례다.

### 방치된 인프라 라이브러리는 공급망 보안의 조용한 위험 지대다

QRCode.js처럼 널리 쓰이지만 관리자가 사실상 부재한 라이브러리는
전형적인 공급망 공격 표적이 되기 좋은 조건을 갖추고 있다.
npm 패키지 탈취, 계정 하이재킹을 통한 악성 코드 주입 같은 공격은
정확히 이런 프로필의 프로젝트, 즉 사용자는 많지만 관리자의 관심은 거의 없는
프로젝트에서 반복적으로 발생해 왔다.
QR 코드처럼 사용자 입력값을 직접 인코딩해 렌더링하는 라이브러리가
공격받는다면 XSS나 피싱 QR 코드 생성 같은 형태로 이어질 수 있는데,
“단순하고 안정적”이라는 인식 때문에 이런 라이브러리는
의존성 감사 대상에서 우선순위 밖으로 밀려나기 쉽다.
활성 유지보수 여부와 무관하게 널리 배포된 코드베이스는
계속해서 감시 대상에 포함되어야 한다는 점을,
이 라이브러리의 방치 상태가 역설적으로 상기시킨다.
