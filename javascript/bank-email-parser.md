# korean-banking-email-parser

<https://github.com/nemorize/korean-banking-email-parser>

## 소개

`korean-banking-email-parser`는 대한민국 은행의 입출금 알림 이메일을 파싱하는
JavaScript 라이브러리다. 현재 NH농협 이메일만 지원한다.

은행 알림 이메일은 암호화된 HTML 파일로 전달된다. 이 라이브러리는 비밀번호(생년월일 6자리
또는 사업자번호 10자리)로 HTML을 복호화한 뒤 계좌 정보와 트랜잭션을 구조화된 객체로 반환한다.

```bash
yarn add @nemorize/korean-banking-email-parser
```

## API

### `parse(html, password, timeout)`

```typescript
parse(html: string, password: string, timeout: number) => Promise<{
    account: {
        accountNumber: string,
        accountHolder: string,
        accountStatus: string,
        balance: number,
        availableBalance: number,
    },
    transactions: {
        transactionDate: string,     // 'YYYY/mm/dd'
        type: 'deposit' | 'withdrawal' | 'unknown',
        amount: number,
        balanceAfter: number,
        branch: string,
        bank: string,
        description: string,
    }[],
}>
```

암호화된 HTML 파일을 받아 복호화 후 필요한 데이터를 추출해 반환한다.

## 분석

### K-금융 이메일 통지 방식의 특수성

이 라이브러리가 존재하는 이유는 국내 은행들이 입출금 알림을 “암호화된 HTML 파일”로
이메일 첨부하는 독특한 방식 때문이다. README의 취소선 주석이 이 점을 잘 포착한다:
“쓸데없이 html 파일을 보내며 고객을 위협하는 K-금융에게서 나 자신을 지키기 위해
충분한 노력을 다해야 합니다.”

이 방식은 보안 의도에서 나왔지만, 자동화를 원하는 개발자에게는 장벽이 된다.
JSON API나 Webhook으로 알림을 받는 것이 불가능하므로, 이메일을 파싱해 구조화된 데이터를
추출하는 레이어가 필요해진다. 이 라이브러리는 그 갭을 채운다.

[GeekNews 댓글](https://news.hada.io/topic?id=29404)에서 calvinsnax는 “국내 은행들이
입출금 내역을 수시로 보내주는 서비스가 없는 것 같아서 시도해보지 못했다”며
NH농협만이 이 기능을 제공하는 것에 주목했다. 실제로 실시간 입출금 알림 이메일 서비스를
제공하는 국내 은행이 극히 제한적이라는 점이 이 라이브러리의 활용 범위를 제약한다.

### 결제 확인 자동화 워크플로우

저장소 설명이 명시하는 용도는 “이메일 라우팅 서비스를 통한 결제 확인 워크플로우 자동화”다.
소규모 사업자가 입금을 수동으로 확인하는 대신, 이메일 라우팅(예: Zapier, Make, 자체 서버)으로
알림 이메일을 받아 이 라이브러리로 파싱하고 주문 처리 시스템에 자동 연동하는 시나리오다.

이는 공식 PG(결제대행사) API 없이 은행 입금 확인을 자동화하는 현실적인 우회 방법이다.

## 비평

### `runScripts: 'dangerously'` 보안 위험

핵심 보안 우려는 복호화 구현 방식이다. JSDOM의 `{ runScripts: 'dangerously' }` 옵션은
HTML 내부의 JavaScript를 Node.js 컨텍스트에서 직접 실행한다. 신뢰할 수 없는 HTML을
넣으면 임의 코드가 실행될 수 있다.

저자도 이를 명시적으로 경고하며 “입출금 알림을 발송한 이메일 주소가 공식 이메일인지,
위변조된 이메일이 아닌지 확인해야 한다”고 권고한다. 이 라이브러리를 프로덕션에서 사용하려면
이메일 발송자 검증 로직이 앞단에 반드시 필요하다. 검증 없이 임의의 HTML 파일을 넣으면
서버 코드 실행, 환경 변수 탈취 같은 심각한 보안 사고로 이어질 수 있다.

### NH농협 단독 지원의 한계

현재 NH농협만 지원한다는 것은 실용 범위를 크게 제약한다. 국민은행, 신한은행, 하나은행,
우리은행 등 주요 은행의 이메일 형식을 지원하지 않으면 대다수 사용자에게 사용 불가다.
저장소가 이메일 파일 샘플 기여를 요청하고 있으므로, 커뮤니티 기여로 지원 은행을 넓혀가는
오픈소스 협업 모델을 기대하는 것으로 보인다.

## 인사이트

### 공식 API 부재가 만들어내는 개발자 생태계

국내 금융 인프라의 개방 API 부재는 이런 파싱 라이브러리를 낳는다. 은행이 공식 Webhook이나
구조화된 데이터 API를 제공하면 이 라이브러리는 필요 없다. 오픈뱅킹 API가 존재하지만
사업자 등록과 심사 과정이 필요해 개인 개발자나 소규모 사업자에게는 여전히 진입 장벽이 높다.

이 현상은 국내 금융 API 생태계의 더 큰 문제를 압축해 보여준다. 공식 채널이 충분히 개방적이지
않을 때, 개발자들은 이메일 파싱, 화면 스크래핑, SMS 파싱 같은 비공식 방법을 동원한다.
이 방법들은 은행이 이메일 형식을 바꾸거나 서비스를 중단하는 순간 깨진다. 근본적으로 불안정한
해결책이지만, 더 나은 옵션이 없으므로 계속 쓰인다.

### 이메일을 구조화된 데이터 소스로 보는 패러다임

이 라이브러리는 “이메일은 사람이 읽는 것”이라는 인식에서 “이메일은 처리해야 할 구조화된
이벤트 스트림”으로의 전환을 보여준다. 이메일 파싱을 통한 자동화는 사실 매우 오래된 기법이다.
주문 확인 이메일 파싱, 배송 알림 처리, 인보이스 자동 입력이 모두 같은 접근법이다.

AI 시대에 이 패러다임은 더욱 강화된다. 구조화된 파싱 코드 없이도 LLM이 이메일 텍스트를
직접 해석해 필요한 정보를 추출할 수 있다. 사실 이 라이브러리의 다음 버전은 암호화 복호화를
기존 방식으로 처리하되, 파싱 로직을 LLM 기반으로 교체해 새 은행 형식을 코드 수정 없이
지원하는 방향으로 발전할 수 있다.

### 보안 위험과 실용성 사이의 현실

`runScripts: 'dangerously'`는 분명한 보안 위험이지만, 이를 피하면서 암호화된 HTML을
복호화하는 방법이 마땅치 않다는 현실도 있다. 은행의 이메일 암호화 구현이 JavaScript 실행에
의존하기 때문이다. 이것은 보안을 위한 암호화가 결국 보안 취약점을 강제하는 아이러니한 상황이다.

이 사례는 레거시 시스템과 상호작용할 때 흔히 마주치는 딜레마를 보여준다. 이상적인 방법으로는
해결할 수 없고, 위험을 인식하고 관리하면서 차선책을 쓰는 것이 현실적 선택이다.
중요한 것은 위험을 숨기지 않고 명시적으로 문서화했다는 점이다. 이 투명성이 사용자가
적절한 방어 조치를 취할 수 있게 한다.
