# Arch Linux의 비트 완전 재현 가능 Docker 이미지

원문: <https://lists.archlinux.org/archives/list/arch-dev-public@lists.archlinux.org/thread/44PW52T547MACXFU5HZ5U7SIPAX3BAXS/>

## 요약

Robin Candau(Antiz)가 2026년 4월 20일 Arch Linux 개발자 메일링 리스트에
Arch Linux Docker 이미지의 비트 완전 재현 가능(bit-for-bit reproducible) 달성을
발표했다. WSL(Windows Subsystem for Linux) 이미지에 이은 두 번째 재현 가능 이미지
마일스톤이며, `repro` 태그로 배포된다.

재현 가능성 달성을 위해 세 가지 핵심 변경이 이루어졌다.
첫째, Dockerfile에서 `SOURCE_DATE_EPOCH`를 설정하고
`org.opencontainers.image.created` 레이블에 반영한다.
둘째, 빌드 간 비결정성을 유발하는 ldconfig 보조 캐시 파일을 최종 이미지에서 제거한다.
셋째, Docker/Podman 빌드 시 `--source-date-epoch`와 `--rewrite-timestamp` 옵션으로
타임스탬프를 정규화한다.

중요한 제약이 하나 있다. 재현 가능성 보장을 위해 pacman 키링(keyring)을
이미지에서 제거해야 했기 때문에, 이 이미지에서는 pacman을 즉시 사용할 수 없다.
컨테이너 내에서 패키지 관리를 사용하려면
`pacman-key --init && pacman-key --populate archlinux`를 먼저 실행해야 한다.
이 제약을 해결하는 방법을 찾는 동안 별도 태그로 첫 번째 마일스톤을 제공하는 형태다.

재현 가능성 검증은 두 방식으로 이루어진다.
빌드 간 다이제스트 동일성 확인과, `diffoci` 도구를 통한 이미지 비교다.
`diffoci`는 파일 차이, 해시 불일치, 타임스탬프 변동을 상세 리포트로 생성한다.
향후 계획으로는 자동화된 재빌더를 통한 주기적 검증과 공개 결과 공유가 예정되어 있다.

## 분석

### 비결정성의 세 가지 원천

Docker 이미지 재현 가능성을 가로막는 요소는 대부분 시간과 상태 의존성에서 온다.

**타임스탬프**: 이미지 레이어의 파일 수정 시각, 이미지 생성 시각은
빌드 시점에 따라 달라진다. `SOURCE_DATE_EPOCH`는 “이 빌드의 기준 시각”을
환경 변수로 고정하여 모든 타임스탬프를 이 기준점으로 정규화한다.
`--rewrite-timestamp` 옵션은 이미 빌드된 레이어의 타임스탬프도
소급 정규화하는 역할을 한다.

**보조 캐시**: ldconfig가 생성하는 동적 링커 캐시 파일은
빌드 환경의 라이브러리 목록에 따라 내용이 달라진다.
이 파일은 런타임에 재생성되므로 이미지에 포함할 이유가 없으며,
제거하면 비결정성 요인이 하나 제거된다.

**암호학적 상태**: pacman 키링은 GPG 키 생성 시 엔트로피를 사용하므로
본질적으로 비결정적이다. 이것이 키링을 이미지에서 제거해야 하는 근본 이유다.
암호학과 재현 가능성은 구조적으로 긴장 관계에 있다.

### WSL 이미지와의 공유 인프라

Docker 이미지 구현이 WSL 재현 가능 이미지의 rootFS 빌드 시스템을 재사용한다는 점은
재현 가능 빌드 인프라가 단일 용도가 아닌 공유 기반으로 설계되었음을 보여준다.
한 번 해결한 문제(타임스탬프 정규화, 비결정적 파일 제거)를
다른 배포 형태에 재적용하는 것이 가능해진 것이다.

### `diffoci`의 역할

`diffoci`는 두 컨테이너 이미지를 레이어 단위로 비교하여
어떤 파일이 다른지, 어떤 메타데이터가 일치하지 않는지를 식별한다.
재현 가능성 달성 과정에서 이 도구는 “무엇이 비결정적인가”를 진단하는 디버거로 작동했을 것이다.
타임스탬프 차이인지, 파일 내용 차이인지, 이미지 레이블 차이인지를
구분할 수 있어야 수정 방향을 잡을 수 있다.

## 비평

### 강점

롤링 릴리스 배포판인 Arch Linux가 재현 가능한 컨테이너 이미지를 달성했다는 것은
눈에 띄는 성과다.
패키지가 지속적으로 업데이트되는 환경에서 특정 시점의 빌드가
동일한 결과를 생성한다는 보장은
공급망 보안(supply chain security) 측면에서 의미 있는 기반을 제공한다.
WSL에 이어 Docker로 이어지는 재현 가능 이미지의 확장도 일관된 방향성을 보여준다.

### 약점

pacman 키링 제거라는 제약은 이 이미지의 실용성을 크게 제한한다.
Arch Linux Docker 이미지의 주요 사용 사례 중 하나는
컨테이너 내에서 패키지를 설치하는 것인데,
`repro` 이미지에서는 사용 전 키링 초기화가 필수 단계가 된다.
이 단계를 모르는 사용자는 혼란을 겪을 것이다.

또한 Hacker News 댓글에서 지적된 것처럼,
“컨테이너를 업데이트하지 않으면 보안 이슈가 생기고,
업데이트하면 재현 가능하지 않게 된다”는 근본적 긴장이 있다.
재현 가능한 이미지는 특정 시점의 스냅샷이므로,
롤링 릴리스의 장점인 최신 패키지 적용과는 구조적으로 충돌한다.
이 긴장은 이미지 자체의 설계 문제가 아니라
컨테이너 재현 가능성의 본질적 한계지만,
사용자에게 명확히 전달되어야 한다.

## 인사이트

### 재현 가능성은 “있으면 좋은 기능”이 아니다

Hacker News 댓글은 이 점을 명확히 표현한다:
“재현 가능한 이미지는 대부분의 경우 감정적 만족을 주는 기능이다—
그러다가 어느 날 그렇지 않게 된다.”
3바이트 타임스탬프 차이가 오후 전체를 디버깅에 써야 했던 사건이 그 예다.

재현 가능성의 가치는 일상에서는 보이지 않는다.
빌드가 항상 같은 결과를 낸다면 아무도 그것을 의식하지 않는다.
가치는 두 가지 위기 상황에서 드러난다.
첫째, “어제와 똑같이 했는데 결과가 다르다”는 디버깅 지옥에서,
동일한 입력으로 동일한 출력을 재현할 수 있다는 것이 문제 원인을 좁히는 데 결정적이다.
둘째, 공급망 공격의 맥락에서,
빌드 산출물의 동일성을 독립적으로 검증할 수 있다면
“이 이미지가 소스에서 정직하게 빌드되었는가”를 확인하는 것이 가능해진다.
SolarWinds나 XZ Utils 사건처럼 빌드 파이프라인이 공격받았을 때
재현 가능성은 변조 탐지의 기반이 된다.

### 타임스탬프는 소프트웨어 재현 가능성의 만성 적이다

`SOURCE_DATE_EPOCH`가 존재한다는 사실 자체가
타임스탬프가 소프트웨어 빌드에서 얼마나 광범위하게 비결정성을 유발하는지를 보여준다.
파일 수정 시각, 아카이브 헤더 타임스탬프, 이미지 생성 레이블,
컴파일러가 삽입하는 `__DATE__`와 `__TIME__` 매크로까지—
소프트웨어 빌드 과정에는 “지금 몇 시인가”를 기록하는 지점이 수십 개 존재한다.
이 모든 지점이 재현 가능성의 적이다.

`SOURCE_DATE_EPOCH`는 이 문제에 대한 커뮤니티 합의 해결책이다.
빌드 시스템이 “현재 시각” 대신 이 환경 변수를 기준 시각으로 사용하도록 약속하면,
동일한 `SOURCE_DATE_EPOCH` 값으로 실행한 모든 빌드는
타임스탬프 측면에서 동일한 결과를 낸다.
이 표준이 Arch Linux Docker 이미지에 적용되었다는 것은
재현 가능 빌드 인프라가 배포판 수준에서 성숙해졌음을 의미한다.

그러나 ldconfig 캐시처럼 타임스탬프가 아닌 비결정성 요인은
케이스별로 발견하고 제거해야 한다.
이 과정에서 `diffoci` 같은 비교 도구의 역할이 중요하다.
재현 가능성 달성은 일회성 선언이 아니라
비결정적 요인을 하나씩 발굴하고 제거하는 반복적 과정이다.

### 컨테이너 이미지와 패키지 관리자의 긴장

Docker 이미지에 패키지 관리자를 포함해야 하는가는 오랫동안 논쟁의 대상이었다.
보안 모범 사례는 “프로덕션 컨테이너에는 패키지 관리자를 두지 말라”고 권고한다.
공격자가 컨테이너를 침해했을 때 `apt`나 `pacman`으로
추가 도구를 설치하는 것을 막기 위해서다.
그러나 현실에서 많은 팀은 기반 이미지에서 패키지를 설치하는 편의성에 의존한다.

Arch Linux의 `repro` 이미지는 이 긴장을 우연히 반영한다.
pacman 키링을 제거함으로써 “즉시 패키지를 설치할 수 없는” 이미지가 만들어졌다.
이것은 제약이지만, 동시에 “이 이미지를 그대로 프로덕션에 올리지 말고
필요한 것만 설치한 파생 이미지를 만들라”는 신호이기도 하다.
멀티 스테이지 빌드 패턴에서 기반 이미지는 도구를 포함하고
최종 이미지는 애플리케이션만 포함하는 것처럼,
재현 가능한 기반 이미지와 그 위에 구축되는 애플리케이션 이미지의 분리가
컨테이너 보안 아키텍처의 자연스러운 방향이다.

pacman 키링 문제의 미래 해결책이 무엇이 될지도 흥미롭다.
결정적 키링 생성이 가능한가? 빌드 타임과 런타임 키링을 분리하는 구조가 가능한가?
이 질문들의 답은 컨테이너 패키지 관리의 더 넓은 설계 문제와 연결된다.
