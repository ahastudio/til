# WSL9x: Linux용 Windows 9x 서브시스템

원문: <https://codeberg.org/hails/wsl9x>

[GeekNews 댓글](https://news.hada.io/topic?id=28796)

## 요약

WSL9x는 Linux 커널 6.19을 Windows 9x 커널 내부에서
협력적으로(cooperatively) 실행하는 실험적 프로젝트다.
하드웨어 가상화나 재부팅 없이 두 운영체제의 기능을 동시에 활용할 수 있다.
GPL-3 라이선스로 공개되었으며, 486+ 프로세서에서 작동한다.

개발자 한 명이 6년에 걸쳐 Windows 9x 내부를 이해하며 완성한 작품으로,
“Proudly written without AI”라는 메시지를 명시하고 있어
GeekNews 댓글에서 별도의 화제가 되었다.

## 아키텍처

세 컴포넌트로 구성된다.

### VxD 드라이버

DOS 인터럽트를 사용해 디스크에서 `vmlinux.elf`를 읽어 커널 매핑을 초기화한다.
16 KiB 스택을 갖는 System VM 스레드를 생성하고,
커널 진입(kernel entry), IRQ 디스패치, 유저스페이스 복귀를 처리하는
이벤트 루프를 실행한다.

### 수정된 Linux 커널

User-Mode Linux를 기반으로 하되,
POSIX 호출 대신 Windows 9x API를 호출하도록 수정했다.
일반적인 ring 3(유저 모드) 실행과 달리 ring 0(슈퍼바이저 모드)에서 실행된다.
핵심 통합 코드는 `arch/um/os-Win95/`에 위치한다.

### WSL.COM 클라이언트

16비트 DOS 프로그램으로 MS-DOS 프롬프트를 통해 터미널 기능을 제공한다.
커스텀 구현 대신 기존 DOS 인터페이스를 재활용한다.

## 핵심 설계 요소

### GPF 핸들러를 통한 시스템 콜

Windows 9x의 인터럽트 디스크립터 테이블은 표준 `int 0x80`을
수용할 수 없다. 따라서 시스템 콜은 일반 보호 결함(General Protection Fault)을
트리거하는 방식으로 처리된다.
GPF 핸들러가 오류 명령어를 검사하여 `int 0x80`이면
명령어 포인터를 전진시키고 Linux 시스템 콜 디스패처로 전달한다.

### 고정 기준 주소

커널은 `0xd0000000`을 하드코딩된 기준 주소로 사용한다.

### 콘솔 통합

출력은 `int 0x29`를 통해 동기화되어
NNANSI 같은 ANSI 드라이버가 터미널 출력을 가로챌 수 있다.

## 빌드 요구사항

```bash
# 필요 도구

```

## 분석

### Windows 9x 미니멀 커널의 역설적 장점

GeekNews 댓글은 흥미로운 역설을 지적한다.
Windows 9x의 단순하고 빈약한 커널 설계가
오히려 현대 OS 기능을 그 위에 올리는 것을 가능하게 했다는 것이다.
NT 커널처럼 강건하게 설계된 시스템은 외부 커널이 침투하기 어려운 반면,
Win9x의 허술한 ring 0 경계가 역설적으로 또 다른 커널을 끼워 넣을 틈을 만들었다.
“취약함이 곧 유연함”이라는 시스템 설계의 아이러니다.

### CoLinux, flinux와의 계보

WSL9x는 진공 속에서 탄생하지 않았다.
GeekNews 댓글은 CoLinux(Windows NT/2000/XP에서 Linux를 협력적으로 실행),
Cygwin, MSYS2, flinux(Wine과 유사하게 Linux 바이너리를 Windows에서 실행)를
선례로 언급한다.
WSL9x가 독특한 점은 이 시도를 Windows 9x—역사적으로 “장난감 OS”로 여겨진—에서
현대 Linux 커널(6.19)을 대상으로 수행했다는 것이다.
기술적 도전의 방향이 실용성보다 극한 탐구에 있다.

### ring 0 실행의 함의

수정된 Linux 커널이 ring 0에서 실행된다는 것은
하드웨어에 대한 완전한 접근 권한을 가진다는 뜻이다.
이는 일반적인 가상화와 근본적으로 다른 협력 모델이다.
두 커널이 ring 0을 공유한다는 설계는
메모리 격리와 보안 경계를 포기하는 대신
가상화 오버헤드 없이 직접 하드웨어 접근을 가능하게 한다.
실험적 맥락에서는 가능한 접근이지만,
이것이 가능하다는 사실 자체가 Win9x의 보안 모델이 얼마나 허술한지를 보여준다.

## 비평

### 강점

순수한 기술적 탐구의 관점에서 이 프로젝트는 인상적인 성취다.
Win9x 내부를 6년에 걸쳐 분석하고,
GPF 핸들러를 시스템 콜 브리지로 활용하며,
현대 Linux 커널을 그 위에서 부팅시키는 창의성은
교과서에서 배울 수 없는 종류의 지식을 요구한다.
이런 프로젝트는 OS 설계, 시스템 프로그래밍,
레트로 컴퓨팅의 교차점에서 드물게 나오는 희귀한 작업이다.

### 약점

486 시스템에서의 메모리 제약, 실제 활용 가능성의 부재,
그리고 Win9x와 Linux 6.19를 동시에 구동할 수 있는
실제 하드웨어 확보의 어려움은 이 프로젝트의 실용적 가치를 제한한다.
“AI 없이 작성됨”이라는 메시지는 개인적 선언으로 이해할 수 있지만,
6년의 투자 대비 외부 기여를 어렵게 만드는 요소이기도 하다.

## 인사이트

### “작동하지 않아야 할 것이 작동하는” 프로젝트의 가치

WSL9x 같은 프로젝트는 실용성의 잣대로 평가할 수 없다.
이 프로젝트의 가치는 “Win9x에서 Linux를 써야 할 이유”가 아니라,
“Win9x의 내부 구조를 이 수준까지 이해한 사람이 존재한다”는 사실에 있다.
GPF 핸들러를 시스템 콜 브리지로 전환하거나,
ring 0 공유로 두 커널을 협력시키는 지식은
현대 OS 설계에서는 접하기 어려운 저수준 통찰이다.
레트로 컴퓨팅 프로젝트는 종종 이런 방식으로 잊혀진 시스템 설계 지식을
살아있는 형태로 보존한다.

역사적으로 “쓸모없는” 탐구에서 나온 기술이 나중에 의외의 곳에 적용된 사례는 많다.
가상화 기술의 상당 부분이 학술적·실험적 OS 연구에서 발전했듯,
WSL9x의 협력 커널 실행 방식에 대한 탐구는
미래의 어떤 시스템 설계에 영감을 줄 수 있다.
오늘의 “장난감 프로젝트”가 내일의 논문 레퍼런스가 되는 것이
컴퓨터 과학사의 반복되는 패턴이다.

### “AI 없이 작성됨”이라는 선언의 맥락

6년짜리 시스템 프로그래밍 프로젝트에 “Proudly written without AI”를
명시하는 것은 단순한 기술 선택 이상의 메시지를 담는다.
이는 깊은 이해 없이 코드를 생성하는 것에 대한 반론이자,
특정 종류의 지식은 수년간의 투쟁 없이는 얻을 수 없다는 주장이다.
Win9x 내부를 이해하는 데 AI는 도움이 되지 않는다.
1990년대 문서, DDK, 역공학, 수많은 BSOD를 거쳐야 하는 지식이기 때문이다.
이 선언은 “AI가 할 수 없는 것”이 무엇인지를 보여주는
프로젝트 자체보다 더 또렷한 증거다.
