# Linux 7.2, strncpy 제거 완료

원문: <https://www.phoronix.com/news/Linux-7.2-Drops-strncpy>

HN 토론: <https://news.ycombinator.com/item?id=48612943> (290점, 318개 댓글)

## 요약

Linux 7.2 커널이 `strncpy()` 함수를 완전히 제거했다.
6년에 걸친 362개 패치의 결실이다.
`strncpy()`는 NUL 종료 실패 가능성과 불필요한 제로 패딩이라는 두 가지 결함으로 인해 오랫동안 버그의 원천으로 지목되어 왔다.
Justin Stitt가 211개 커밋을 단독으로 기여하며 작업을 이끌었고, 70명 이상의 기여자가 함께했다.

## 분석

### strncpy의 구조적 결함

`strncpy()`는 `strcpy()`의 안전한 대안으로 설계되었지만, 정작 그 의도가 모호했다.
목적지 버퍼가 꽉 찰 경우 NUL 종료 문자를 쓰지 않아, 이후 코드에서 버퍼 오버리드를 일으킬 수 있었다.
반대로 소스 문자열이 버퍼보다 짧으면 나머지 공간을 모두 0으로 채우는 불필요한 제로 패딩을 수행했다.
이 두 동작이 결합되면서 함수의 의도를 파악하기 어렵게 만들었고,
올바르게 사용하려면 별도의 NUL 종료 처리가 필요했다.

### 6년에 걸친 대규모 정리 작업

Justin Stitt가 362개 패치 중 211개를 단독으로 기여하며 주도했고, 70명 이상의 기여자가 함께했다.
Google의 커널 자기 보호 프로젝트(Kernel Self Protection Project) 소속 Kees Cook도 이 작업을 지원했다.
6년이라는 시간은 이 함수가 커널 전체에 얼마나 깊이 퍼져 있었는지를 보여준다.
Linux 7.2는 2026년 6월 28일 첫 릴리스 후보(RC)에 진입할 예정이다.

### 대체 API의 설계 의도

`strncpy()` 제거 과정에서 사용 맥락에 따라 여러 대체 함수가 선택되었다.
`strscpy()`는 항상 NUL 종료를 보장하며, `strscpy_pad()`는 남은 공간을 명시적으로 0으로 채운다.
`strtomem_pad()`와 `memcpy_and_pad()`는 원시 메모리 조작이 필요한 경우에 사용된다.
`memcpy()`와 `memset()`은 NUL 종료가 아예 필요 없는 바이너리 데이터 복사 맥락에 쓰인다. [^D-Coder]

[^D-Coder]: D-Coder는 “362개 패치는 `strncpy`가 제거된 사용 위치 수이며, 반드시 버그 수를 의미하지는 않는다”고 지적했다. 패치 수와 실제 취약점 수는 구분해야 한다.

### NUL 종료 문자열의 역사적 맥락

C 언어의 NUL 종료 문자열 방식은 Dennis Ritchie가 1970년대에 선택한 설계다.
Pascal은 문자열 앞에 길이를 저장하는 방식을 채택해 버퍼 오버플로 문제를 구조적으로 회피했다.
Dennis Ritchie 자신도 1990년에 팻 포인터(fat pointer) 타입을 C에 추가할 것을 제안했으나,
C99, C11을 거쳐 C23에 이르기까지 표준에 채택되지 않았다. [^sirwhinesalot]

[^sirwhinesalot]: sirwhinesalot은 Dennis Ritchie가 1990년 팻 포인터 타입을 제안했으나 표준 위원회가 채택하지 않았음을 지적했다. Walter Bright의 2007년 “C's greatest mistake” 글도 같은 방향의 제안이었지만 C11에도 반영되지 않았다. “우리는 지금 C23인데, 아직도 없다. 대신 `_Generic`과 VLA는 있다!”

## 비평

### 정수화된 숫자의 함정

“362개 버그 수정”이라는 프레이밍은 기술 기사에서 쉽게 나타나지만, 정확히는 362개 사용 위치에서의 교체다. [^WalterBright]
그 중 실제 보안 취약점이었던 사례가 몇 건인지는 보도에서 다뤄지지 않는다.
더 정직한 보도는 “버그를 일으킬 수 있는 패턴의 제거”와 “실제 취약점 수정”을 구분하는 것이다.
숫자는 크고 설득력 있어 보이지만, 그 의미를 오해하면 작업의 성격을 왜곡할 수 있다.

[^WalterBright]: WalterBright는 “C 코드를 검토할 때마다 `strncpy`를 찾았고 항상 버그를 발견했다”고 말했다. 반복적인 실수 패턴이었다는 현장 증언이다.

### 점진적 기여의 가시성 문제

362개 패치를 70명 이상이 기여했지만, 대부분의 기여자는 공개적으로 주목받지 못한다.
커널 인프라 정리처럼 기능 추가가 없는 작업은 사용자에게 직접 보이지 않아 평가절하되기 쉽다. [^twothreeone]
이 작업이 6년이 걸렸다는 사실은 단순히 느린 진행이 아니라,
이런 유형의 작업에 할당 가능한 주의 집중의 한계를 반영한다. [^lambdaone]

[^twothreeone]: twothreeone은 “새 기능에는 쉽게 공로를 인정받지만, 나쁜 기능을 제거하는 일이 오히려 커널처럼 근본적인 것에서는 더 중요하다”고 말했다.
[^lambdaone]: lambdaone은 “이런 지루한 갈아내기가 시스템 엔지니어링의 진짜 작업이 이루어지는 곳”이라고 표현했다.

### 언어 설계 실수와 하위 호환성의 비용

`strncpy()`는 C 표준 라이브러리에 수십 년간 남아 있었다.
이는 하위 호환성이라는 가치가 설계 실수의 수정을 얼마나 오래 지연시킬 수 있는지 보여준다.
Linux 커널은 자체적인 결정권이 있었기에 6년 만에 제거할 수 있었지만,
C 표준 자체에서 `strncpy()`는 여전히 사라지지 않았다. [^mrlonglong]

[^mrlonglong]: mrlonglong은 “NUL 종료 문자열이 컴퓨팅 역사에서 가장 큰 실수라고 생각한다. Pascal 방식의 문자열이 훨씬 안전했을 것”이라고 말했다.

## 인사이트

### API 설계에서 명확한 의도의 중요성

`strncpy()`의 문제는 성능이나 효율이 아니라 의도의 모호함이었다.
하나의 함수가 두 가지 서로 다른 동작을 상황에 따라 수행하면, 사용자는 항상 잘못 사용할 위험에 처한다.
`strscpy()`, `strscpy_pad()`, `memcpy_and_pad()` 같은 분리된 API는 사용자가 의도를 명시적으로 선택하도록 강제한다.
이 설계 원칙은 언어와 무관하게 모든 라이브러리 API에 적용된다.

### 문자열 타입 부재가 만들어낸 누적 부채

C 언어는 문자열을 일급 타입으로 정의하지 않았고, 그 결과 문자열 처리는 포인터와 길이의 암묵적 계약에 의존했다. [^cm2187]
이 공백은 `strncpy`, `strlcpy`, `sprintf` 같은 변형 함수들의 난립을 낳았고,
개발자들은 어떤 함수를 써야 하는지 항상 선택해야 했다.
Rust, Go, Python 같은 언어들이 문자열을 일급 타입으로 설계한 이유가 여기 있다.

[^cm2187]: cm2187은 “문자열 데이터타입 하나 없어서 이 많은 고통을 겪어야 했다”고 요약했다.

### NUL과 NULL의 혼동이 드러내는 인지 부하

NUL(ASCII 0x00 문자)과 NULL(void* 0 포인터)은 개념적으로 다르지만, C 코드에서는 모두 0으로 표현된다.
이 혼동은 단순한 명명 문제가 아니라, C 언어가 타입 시스템으로 구분하지 않는 개념을
개발자가 암묵적 지식으로 구분해야 한다는 것을 의미한다. [^rswail]
`strncpy()` 같은 함수의 NUL 종료 버그는 이 혼동이 실제 결함으로 이어진 사례다.

[^rswail]: rswail은 “이 스레드에서 NUL(ASCII 비인쇄 문자, 바이트 값 0)과 NULL(유효한 메모리를 가리키지 않는 포인터, `((void *) 0)`)을 혼동하는 것이 흥미롭다”고 지적했다.

### 잘못 설계된 API 하나의 누적 유지보수 비용

새 API가 설계되어도 기존 코드베이스에 퍼진 사용 위치를 교체하는 데 수년이 걸린다.
362개 패치, 6년, 70명 이상이라는 숫자는 잘못 설계된 API 하나가 만들어내는 누적 유지보수 비용의 척도다.
이 비용은 처음부터 올바른 API를 설계하는 것이 얼마나 중요한지를 역설적으로 보여준다.
그리고 오픈 소스 생태계에서는 이 비용이 수많은 자원봉사자의 시간으로 지불된다.
