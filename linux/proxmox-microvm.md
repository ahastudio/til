# Proxmox VE에서 microVM 운영하기

원문: <https://taoofmac.com/space/blog/2026/06/18/1845>

HN 토론: <https://news.ycombinator.com/item?id=48599555> (225점, 43개 댓글)

## 요약

Rui Carmo가 작성한 이 글은 Proxmox VE에 QEMU의 microvm 머신 타입을 통합하는
`pve-microvm` Debian 패키지를 소개한다.
저자는 수년간 Proxmox 클러스터를 운영하며 LXC 컨테이너와 전체 가상 머신(VM) 사이의
타협이 지겨워졌고, 그 중간 지점을 직접 만들었다.

문제의 핵심은 Proxmox가 제공하는 두 가지 기본 선택지의 한계다.
LXC 컨테이너는 즉시 시작되고 효율적이지만 호스트 커널을 공유해 격리가 약하다.
전체 VM은 KVM/VT-x 하드웨어 격리를 제공하지만 5~10초의 부팅 시간과
에뮬레이션된 레거시 장치들의 오버헤드를 동반한다.
QEMU의 microvm 머신 타입은 BIOS, GRUB, 레거시 장치를 모두 제거하고
virtio 전용 환경에서 직접 커널 부팅을 구현해 300ms 미만 부팅을 달성한다.

`pve-microvm`은 단일 `.deb` 패키지로 Proxmox의 `qemu-server` Perl 모듈을 설치 시 패치한다.
VM 설정에 `machine: microvm`이 지정되면 표준 `config_to_command` 함수가
저자의 `MicroVM.pm` 모듈로 위임되어 완전히 다른 QEMU 명령행을 구성한다.
PCI 브리지도, VGA도, USB도 없다.
게스트는 단일 시리얼 콘솔(Proxmox의 xterm.js가 네이티브로 연결), virtio 블록 장치,
virtio 네트워크 인터페이스만 갖는다.

패키지는 12MB의 커스텀 Linux 6.12.22 커널, 1MB initrd, `pve-microvm-template`(OCI 이미지에서 루트파일시스템 구축),
`pve-oci-import`, Web UI 확장, 부팅 조기 서비스(`pve-microvm-early.service`)를 포함한다.
Debian, Alpine, Fedora, Rocky, NetBSD, Plan9 등 21개 게스트 OS 타입을 지원한다고 밝혔다.

## 분석

### “컨테이너 속도 + VM 격리”는 홈랩을 넘어 AI 에이전트 샌드박싱의 핵심 문제다

저자가 이 프로젝트를 시작한 직접적 동기 중 하나가 “agentic sandbox 관련 하이프 속에서
에이전트 격리 방법을 찾다가”라는 것은 시사적이다.
AI 에이전트가 실제 시스템에서 작업을 실행하는 패턴이 일반화되면서,
에이전트 하나당 격리된 실행 환경을 얼마나 빠르고 저렴하게 만들 수 있느냐가
아키텍처의 핵심 제약이 되고 있다.
LXC는 빠르지만 격리가 약하고, 전체 VM은 격리가 강하지만 5~10초가 지나치게 느리다.

HN 댓글에서 alexellisuk은 SlicerVM을 언급하며 “에이전트 스타일 워크로드에서 중요한 지표는
API 요청 → VM 생성/복제 → 네트워크 정책 적용 → 게스트 에이전트 접근 → exec/shell/cp/forward 작동까지의
전체 시간”이라고 정확히 짚었다.[^alexellisuk]
300ms 부팅이 의미 있는 것은 그것이 에이전트가 기다리는 시간이기 때문이다.

### 커널을 게스트 밖에 두는 설계는 운영 모델을 근본적으로 바꾼다

“하나의 커널, 많은 게스트”는 단순한 최적화가 아니라 운영 철학의 전환이다.
전통적 VM에서 각 게스트는 자신만의 커널을 갖고, 커널 업그레이드는 게스트 내부에서 발생한다.
pve-microvm에서 모든 microVM은 호스트의 `/usr/share/pve-microvm/vmlinuz`를 공유하고,
게스트 디스크는 순수한 루트파일시스템 — 유저랜드, `/boot` 없음, GRUB 없음 — 만 갖는다.

이 설계가 “컨테이너 스타일 커널 일관성 + VM 스타일 격리”라는 결합을 만든다.
호스트에서 `vmlinuz` 파일 하나만 교체하면 모든 microVM이 다음 재시작 때
새 커널을 사용한다.
게스트가 잘못된 커널 패키지를 설치하거나 업그레이드 중에 망가질 위험이 없다.
그러나 이 단일 커널이 특정 모듈을 포함하지 않으면 그 워크로드는 실행 불가능하다는 트레이드오프도 있다.

### Proxmox Perl 내부를 패치하는 접근은 기술적 성취이지만 유지 관리 부채다

저자가 솔직하게 인정하듯, 이 접근의 가장 큰 위험은 Proxmox 업그레이드와의 충돌이다.
`dpkg` 트리거 시스템이 업그레이드 후 자동 재패치를 수행하지만,
`qemu-server`와 `libpve-*` 패키지의 버전 불일치가 Perl 모듈 로딩 실패로 이어지면
`pvedaemon`이 시작되지 못해 VM 관리 자체가 불가능해진다.
저자가 경험한 `/dev/vda` → `/dev/sda` 경로 혼동 문제는 이런 부분적 패치 적용의 구체적 사례다.

HN 댓글의 mtron_은 “300ms 부팅은 훌륭하지만 pvedaemon 패치 접근은 위험하며
모든 업그레이드에서 깨질 수 있다”고 지적했다.[^mtron_]
wereHamster는 NixOS에서 microvm.nix를 사용하면서 명령형 에이전트 microVM 관리가 잘 맞지 않는다는
다른 각도의 문제를 제기했다.[^wereHamster]
공식 지원 없이 서드파티 패치로 인프라를 운영하는 것은 홈랩 수준에서는 수용 가능한 트레이드오프지만,
프로덕션 환경에서는 다른 판단이 필요하다.

## 비평

### 저자가 Proxmox 팀에 기여하지 않는 것은 기술적 선택이 아니라 구조적 장벽의 반영이다

저자는 글 마지막에 “Proxmox 팀과 비공식적으로 연락했더니 메일링 리스트에 참여해 논의하라는 답을 받았지만,
그것은 너무 90년대 방식이고 시간도 없다”고 밝힌다.
이 고백은 오픈소스 인프라 프로젝트에서 혁신이 어떻게 공식 제품에 통합되거나 실패하는지를 드러낸다.

Proxmox의 엔터프라이즈 QA 파이프라인을 거쳐야 한다는 것은 이해할 수 있지만,
그 진입 장벽이 혁신적인 기여를 서드파티 패치로 고립시킨다.
tlamponi는 HN에서 2020년에 microVM을 평가했지만 유지 관리 비용 대비 효과가 불충분하다고 판단했으며,
지금 재평가할 때가 됐다고 언급했다.[^tlamponi]
커뮤니티의 관심과 실용성이 검증된 기능이 공식 제품에 통합되지 못하는 이 간극은
오픈소스 기업 소프트웨어의 고질적 문제다.

### GPU 패스스루 비활성화는 AI 워크로드 에이전트 샌드박싱의 실질적 한계를 드러낸다

저자가 GPU 패스스루를 의도적으로 비활성화했다는 점은 이 도구의 에이전트 샌드박싱 적용 가능성을 제한한다.
apitman의 댓글이 정확히 짚었듯이, “AI 에이전트에게 전체 GPU를 패스스루하지 않고 CUDA 지원을 제공하는 것”이
microVM의 가장 매력적인 미래 기능이지만 현재로서는 해결되지 않은 문제다.[^apitman]

virtio-gpu와 Vulkan은 그래픽에서 작동하지만 CUDA는 NVIDIA의 독점적 계층이 필요하다.
전체 GPU 패스스루는 호스트가 그 GPU를 잃는 것을 의미하며, 여러 에이전트가 GPU를 공유하는 것이
현재 microVM 아키텍처에서는 불가능하다.
저자의 성실한 한계 인정에도 불구하고, GPU를 필요로 하는 에이전트 샌드박싱 사용 사례에서
pve-microvm은 현재 해결책이 아니다.

### 21개 게스트 OS 지원은 인상적이지만 유지 관리 부담의 급증을 의미한다

NetBSD, Plan9, FreeBSD 기반 방화벽까지 지원하는 것은 기술적 호기심의 산물이지만,
각 OS가 고유한 virtio 드라이버 지원, 부팅 시퀀스, 파일시스템 레이아웃을 요구한다.
저자가 QEMU 10.x에서 Linux 게스트의 virtio-mmio 경로 버그를 발견하고
PCIe로 대체하는 과정에서 약 50ms 추가 오버헤드를 수용한 것이 이를 보여준다.

개인 홈랩 프로젝트에서 21개 OS를 지원하는 것은 인상적이지만,
각 QEMU 업그레이드, 각 게스트 OS 업스트림 변경이 회귀 테스트를 요구한다.
저자가 “smith VM이 주기적으로 다른 게스트 OS를 꺼내 회귀 테스트를 수행한다”고 언급한 것은
이 유지 관리 부담을 자동화로 해결하고 있음을 보여주지만,
커뮤니티 기여 없이 이 범위를 장기간 유지하는 것은 도전적이다.

## 인사이트

### microVM의 부상은 컨테이너 이후 격리 모델의 탐색기다

Docker가 프로세스 격리를 대중화한 이후, 보안 요구사항이 높은 워크로드를 위한
“더 강한 격리, 여전히 빠른 시작”의 조합을 찾는 시도가 이어졌다.
Firecracker(AWS Lambda와 Fargate에 사용), gVisor(Google Cloud Run), Kata Containers,
그리고 pve-microvm은 모두 이 문제에 대한 서로 다른 답이다.
이 생태계의 활성화는 “컨테이너 vs. VM” 이분법이 충분하지 않다는 산업 전반의 인식을 반영한다.

AI 에이전트의 확산이 이 탐색에 새로운 압박을 가하고 있다.
에이전트가 실제 명령을 실행하고 파일을 수정하며 네트워크에 접근하는 패턴이 일반화되면,
에이전트 하나당 격리된 환경의 비용이 시스템 설계의 핵심 제약이 된다.
Firecracker가 AWS Lambda의 멀티테넌트 격리를 위해 탄생했듯이,
pve-microvm 같은 도구는 개인 개발자 수준의 에이전트 인프라 구축을 가능하게 한다.

### 하나의 커널, 많은 게스트 패턴은 클라우드 인프라에서 검증된 패턴의 홈랩 재현이다

AWS의 Firecracker, Google의 gVisor, Microsoft의 WSL2는 모두 “경량 하이퍼바이저 + 공유 호스트 커널 또는 제어된 커널 배포” 패턴을 사용한다.
이 패턴이 홈랩에서 개인이 구현 가능한 수준으로 내려온 것은
QEMU/KVM의 성숙과 Linux 커널의 microvm 지원 덕분이다.

흥미로운 것은 저자가 이것을 “귀찮음에서 시작한 해킹”이라고 묘사하지만,
실제로는 클라우드 인프라 수준의 아키텍처 패턴을 단일 Debian 패키지로 구현한 것이라는 점이다.
이 패턴의 확산은 인프라 지식의 민주화를 의미한다.
수년 전에는 AWS 엔지니어링 팀이 구현했을 기술이 이제 홈랩 블로그 포스트와 오픈소스 패키지로 재현된다.

### Proxmox 패치 전략은 공식 지원 전까지의 가교이자 업스트림 설득의 증거 생성 방법이다

저자의 Proxmox Perl 패치 접근은 단순한 해킹이 아니라
“내가 이것이 작동한다는 것을 증명할 테니, Proxmox가 공식적으로 지원해 달라”는 전략으로 읽힌다.
공개 GitHub 저장소, 완성된 패키지, 225점의 HN 스레드, 43개의 기술적 댓글은
Proxmox 팀에 대한 비공식적 압력이 된다.

오픈소스 인프라 도구의 역사에서, 서드파티 패치가 충분한 커뮤니티 관심을 얻으면
공식 제품에 통합되는 패턴은 드물지 않다.
저자가 “언젠가는 누군가 이것을 업스트림해야 할 것이다. 나는 그럴 시간이 없다”고 말할 때,
그것은 패배 선언이 아니라 바통을 넘기는 것이다.
실제 작동하는 구현과 검증된 수요가 있을 때, 업스트림 기여의 장벽은 낮아진다.
