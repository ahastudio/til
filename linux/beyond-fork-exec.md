# fork() + exec()를 넘어서

원문: https://lwn.net/Articles/1016723/

## 요약

이 글은 LSFMM+BPF 2025에서 발표된 Jonathan Cameron의 세션을 다룬다.
주제는 50년 된 Linux 프로세스 생성 모델인 `fork()` + `exec()` 조합의 근본적
비효율성을 개선하기 위한 커널 레벨 제안들이다.

**문제**: `fork()`는 부모 프로세스의 메모리 전체를 복사하지만, 직후 `exec()`가
그 복사본을 모두 폐기한다. Copy-on-Write(CoW)로 실제 물리 메모리 복사는
줄였지만, 페이지 테이블 복사 비용은 여전히 남는다. Redis처럼 수십 GB의 메모리를
가진 프로세스에서 이 비용은 무시하기 어렵다.

**제안된 해법 — Spawn Templates**: `spawn_template_create()`로 실행 파일에 대한
템플릿 파일 디스크립터를 생성해 커널이 실행 정보를 캐시하고,
`spawn_template_spawn()`으로 캐시된 템플릿에서 새 프로세스를 생성한다.
벤치마크에서 약 2%의 성능 개선이 확인되었다.

이 제안은 결국 거부되었지만, 더 폭넓은 논의를 촉발했다. 커뮤니티의 방향은
두 가지로 수렴했다: **pidfd 기반 접근** — 빈 프로세스를 생성하고 `pidfd_config()`
로 설정하는 방식, 그리고 **`posix_spawn()`** 네이티브 구현 — `fork()`/`exec()`의
한계를 우회하는 표준 API의 올바른 구현이다.

## 분석

### fork()/exec()의 우아함과 비효율성의 공존

`fork()` + `exec()` 모델의 설계 미학은 일관성에 있다. `fork()` 이후 자식
프로세스는 일반 Unix 프로세스이므로, 파일 디스크립터 조작, 시그널 마스크 설정,
`chdir()`, `setuid()` 등 표준 API를 자유롭게 사용할 수 있다. 별도의 “프로세스
설정 API”가 필요 없다.

이 유연성이 동시에 문제의 원인이다. `exec()` 전까지 자식 프로세스가 임의의
작업을 할 수 있다는 가정 때문에, 커널은 `fork()` 시점에 완전한 프로세스 상태를
준비해야 한다. Spawn Templates의 핵심 아이디어는 “설정 단계를 템플릿으로
분리해 재사용”하는 것인데, 이는 `fork()`의 유연성 모델과 충돌한다.

### 제안 거부가 의미하는 것

2%의 성능 개선으로는 새로운 커널 API를 정당화하기 어렵다. 커널 API는 한번
추가되면 수십 년간 유지해야 하는 ABI 계약이다. Linux 커뮤니티의 이 거부 결정은
올바른 판단이다. 성능 문제를 해결하더라도, 더 넓은 설계 원칙과 맞지 않는 API는
장기적으로 더 큰 복잡성을 초래한다.

더 흥미로운 것은 이 논의가 촉발한 pidfd 기반 접근이다. `pidfd`는 2019년부터
추가된 프로세스 파일 디스크립터 인터페이스로, 프로세스를 파일 디스크립터로 다루는
보다 현대적인 API다. “빈 프로세스를 만들고 pidfd를 통해 설정하는” 모델은
`fork()`의 암묵적 상태 상속을 명시적 설정으로 대체한다.

## 비평

### 2%는 어떤 맥락에서 중요한가

Redis, PostgreSQL, memcached처럼 대규모 메모리를 가진 서버 프로세스에서의
`fork()` 비용은 실제 문제다. PostgreSQL은 각 연결마다 프로세스를 `fork()`하는
구조이며, 이 비용이 연결 수에 선형 비례한다. “2%”라는 수치는 전체 평균이지만,
특정 워크로드에서는 훨씬 큰 영향을 줄 수 있다.

### “A Fork() in the Road” 논문의 맥락

이 세션의 배경에는 2019년 같은 제목의 논문이 있다. 저자들은 `fork()`를
“1970년대 머신을 위한 영리한 해킹이었지만, 현대 프로그래밍을 위한 좋은
추상화는 아니다”라고 주장했다. 이 논문이 촉발한 학계와 커널 개발자 커뮤니티의
논쟁이 여전히 진행 중임을 이 발표가 보여준다.

## 인사이트

### 50년 된 API의 불변성이 말해주는 것

`fork()` + `exec()`는 1973년 Unix에서 등장한 개념이다. 50년이 지난 지금도
Linux 커널의 표준 프로세스 생성 메커니즘이라는 사실은 두 가지를 시사한다.
첫째, 이 API가 놀랍도록 잘 설계되었다는 것. 단순한 인터페이스로 복잡한 유연성을
제공한다. 둘째, 운영체제 레벨의 ABI 변화가 얼마나 어려운 일인지를 보여준다.
수십 억 개의 프로그램이 이 동작 방식에 의존하고 있다.

이 불변성은 “기술 부채”가 아니다. 안정적인 ABI가 생태계를 가능하게 한 것이다.
동시에, 1970년대의 설계 결정이 2025년의 최적화 논의를 제약한다는 것은 추상화 선택의
장기적 영향을 잘 보여준다.

### pidfd는 Unix 철학의 재적용이다

“빈 프로세스를 만들고 파일 디스크립터를 통해 설정한다”는 pidfd 접근은
Unix 철학 — 모든 것은 파일이다 — 의 재적용이다. 프로세스를 파일 디스크립터로
추상화하면, 기존의 파일 디스크립터 기반 API들(선택적 전달, `SCM_RIGHTS`를 통한
프로세스 간 공유, `select()`/`poll()`로 프로세스 종료 감지)이 자연스럽게 적용된다.

새로운 추상화를 발명하는 대신, 기존의 강력한 추상화를 새 문제에 적용하는 것.
이것이 Unix 철학이 50년간 유효성을 유지하는 이유다.

### 프로세스 생성 비용과 클라우드 아키텍처의 연결

`fork()` 비용이 중요해진 또 다른 이유는 클라우드 환경에서의 프로세스 생성 패턴이다.
컨테이너 런타임, 서버리스 함수, FaaS 플랫폼들은 매초 수천 개의 프로세스를 생성하고
종료한다. 이 환경에서 프로세스 생성 오버헤드는 운영 비용에 직접 영향을 미친다.

1970년대에는 프로세스 생성이 드문 이벤트였다. 2025년에는 그것이 주요 워크로드다.
API가 만들어진 맥락과 사용 맥락이 극적으로 달라졌을 때, 성능 최적화가 필요해지는
것은 자연스러운 귀결이다.
