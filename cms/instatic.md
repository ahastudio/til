# Instatic: Bun 기반 셀프호스팅 비주얼 CMS

<https://instatic.com/>

<https://github.com/CoreBunch/Instatic>

## 소개

Instatic은 Bun 위에서 동작하는 셀프호스팅형 비주얼 CMS다.
“당신의 사이트를 소유하라. 만드는 과정을 사랑하라(Own your site.
Love building it)”는 슬로건을 내세우며, 비주얼 에디터·콘텐츠
엔진·퍼블리셔를 하나의 서버에 통합해 깔끔한 시맨틱 HTML을 생성한다.
개발팀은 README에서 자신들을 “수천 명이 생계를 위해 웹사이트를
만드는 데 쓰는 도구인 Motion.page와 Core Framework를 만든 팀이며,
대부분 WordPress 생태계 안에서 활동해 왔다”고 소개하며, 기존
디자인 토큰 기술을 별도 플러그인이 아니라 핵심 시스템으로
통합했다고 밝힌다.

기술 스택은 서버와 툴링 모두 Bun 위에서 돌아가고, TypeScript로
전체 코드베이스(95.3%)가 작성됐으며, 관리자 인터페이스는 React
Compiler를 적용한 React 19로 만들어졌다. 데이터베이스는 SQLite와
Postgres를 모두 지원하고, 플러그인은 QuickJS-WASM 샌드박스 위에서
실행된다. 발행된 페이지는 대부분 디스크에서 서빙되는 정적 파일
형태이며, 런타임 오버헤드는 약 0.7kB에 불과하다고 설명한다.

주요 기능은 다섯 갈래로 나뉜다. 디자인 시스템 측면에서는 Core
Framework를 내장해 단일 브랜드 컬러로부터 색상 음영을 자동
생성하고, 뷰포트에 반응하는 수학적 타입 스케일과 일관된 간격
체계를 제공한다. 콘텐츠 빌딩 측면에서는 드래그앤드롭 비주얼
모듈(컨테이너·텍스트·이미지·버튼·폼), 타입이 지정된 매개변수를
가진 재사용 가능한 컴포넌트, 컬렉션과 동적 콘텐츠를 위한 반복
기능, 그리고 “실제로 페이지를 편집하는 AI 에이전트”를 제공한다.
이 에이전트는 자연어로 원하는 것을 설명하면 캔버스 위에 실제
편집 가능한 노드로 만들어 준다고 설명하며, 28개의 도구를 갖췄고
“직접 모델을 가져와 쓸 수 있다(bring your own model)”는 방식으로
Claude, OpenAI, OpenRouter, 로컬 Ollama를 지원한다고 밝힌다. 이
밖에도 HTML·정적 사이트를 편집 가능한 페이지로 변환하는 “Super
Import” 기능이 있다.

콘텐츠 관리 측면에서는 페이지·포스트·컴포넌트를 하나의 데이터
저장소로 통합하고, 초안·예약·발행 단계를 갖춘 커스텀 포스트
타입, 폴더 구조와 사용 현황 추적을 갖춘 미디어 워크스페이스,
36개의 세분화된 권한(capability)으로 구성된 역할 기반 접근
제어, 암호화된 비밀키를 쓰는 TOTP 2단계 인증, 전체 관리자
화면을 탐색할 수 있는 커맨드 팔레트(⌘K)를 제공한다. 다만 36개
권한이 왜 필요한지, 어떤 조직 규모나 사용 사례를 염두에 뒀는지에
대한 설명은 README에 없다. 분석 대시보드는 위젯 타일을 드래그로
크기 조절할 수 있고, 모든 관리자 행동을 기록하는 추가 전용
(append-only) 감사 로그를 제공한다. 확장성 측면에서는 플러그인이
파일시스템이나 환경 변수에 접근할 수 없는 샌드박스에서 실행되며,
README는 이를 “플러그인이 여러분의 비밀 정보를 읽거나 몰래
전송할 수 없다. ‘플러그인이 사이트를 다운시키고 데이터베이스를
낯선 사람에게 이메일로 보냈다’는 흔한 이야기는 여기서는 일어나지
않는다”는 문구로 강조한다.

배포는 Railway 원클릭 배포(SQLite 권장, Postgres도 지원), Docker
Compose, Caddy 리버스 프록시 설정, VPS·매니지드 데이터베이스
가이드를 제공한다. 최신 이미지로 재배포해도 첨부된 스토리지의
데이터베이스와 업로드 파일은 보존된다. 빠른 시작은 `git clone` 후
`bun install`, `bun run dev`로 개발 서버(포트 5173, 핫 리로드)를
띄우거나 `bun run start`로 프로덕션 서버(포트 3001의 `/admin`)를
실행한다. 아키텍처 무결성은 `src/__tests__/architecture/`의
실제 테스트로 검증한다고 밝힌다. 라이선스는 MIT이며, README는
“등급 없음, 오픈코어 꼬리표 없음(No tiers, no open-core
asterisks)”이라고 명시하지만, 정작 이 프로젝트의 수익 모델이나
지속 가능성(자금 조달 방식, 유료 플랜 여부)에 대한 언급은 전혀
없다. 이 글을 작성하는 시점 기준 GitHub 스타 2.2k, 포크 178개를
기록하고 있으며, 현재 버전은 0.0.10으로 의도적인 1.0 이전
단계임을 명시한다. 로드맵으로는 자체 프라이버시 존중 애널리틱스,
확장된 모듈·플러그인 생태계, 더 많은 도구를 갖춘 AI 에이전트,
1.0 확정 전 API·워크플로 정리를 꼽는다.

## 분석

### “플러그인이 데이터베이스를 유출했다”는 문구는 WordPress 생태계에 대한 암묵적 고발이다

QuickJS-WASM 샌드박스를 설명하며 든 예시, “플러그인이 사이트를
다운시키고 데이터베이스를 낯선 사람에게 이메일로 보냈다”는 시나리오는
구체적인 실명 언급 없이도 누구를 겨냥한 것인지 명확하다. 개발팀이
스스로 밝혔듯 이들은 수년간 WordPress 생태계 안에서 Motion.page와
Core Framework를 만들어 온 사람들이며, WordPress는 실제로 취약한
서드파티 플러그인을 통한 보안 사고가 반복적으로 보고돼 온
플랫폼이다. 즉 Instatic의 샌드박스 아키텍처 자체가 특정 경쟁
플랫폼에서 겪은 구체적인 고통의 경험을 기술적 해법으로 직접
번역한 결과로 읽힌다. 이는 이 프로젝트가 추상적인 “더 나은
CMS를 만들자”는 목표에서 출발한 것이 아니라, 자신들이 오랫동안
그 생태계 안에서 직접 목격하거나 겪었을 구체적인 실패 사례에
대한 반작용으로 설계됐다는 것을 강하게 암시한다.

### “직접 모델을 가져오라”는 AI 에이전트 설계는 벤더 종속을 피하려는 전략적 선택이다

Instatic의 AI 페이지 편집 에이전트가 Claude·OpenAI·OpenRouter·로컬
Ollama 중에서 사용자가 원하는 모델을 선택하도록 설계된 것은,
많은 AI 기능 통합 제품들이 특정 모델 공급자와 배타적으로
제휴하는 것과 대비된다. 이런 설계는 두 가지 효과를 동시에 낸다.
하나는 사용자가 비용·성능·프라이버시 요구에 맞춰 모델을 고를 수
있는 유연성이고, 다른 하나는 Instatic 자신이 특정 LLM 공급자의
가격 정책이나 API 변경에 종속되지 않는 것이다. 이는 “당신의
사이트를 소유하라”는 프로젝트 전체의 슬로건과 일관된 설계
철학이다. 셀프호스팅 CMS라는 정체성이 데이터 소유권에만 그치지
않고, AI 기능이 특정 벤더의 클라우드 API에 대한 종속으로 이어지지
않도록 하는 데까지 확장되어 있다.

### 페이지·포스트·컴포넌트의 통합 데이터 모델은 CMS 아키텍처의 오래된 이분법을 해체한다

전통적인 CMS는 대개 “고정된 페이지”와 “시간순으로 쌓이는 포스트”를
서로 다른 데이터 모델로 취급해 왔고, 재사용 가능한 컴포넌트는
또 다른 별도의 시스템(위젯, 숏코드, 블록 패턴 등)으로 관리되는
경우가 많았다. Instatic이 이 셋을 “동일한 저장소”에 통합한 것은,
콘텐츠의 형태(페이지냐 포스트냐)와 콘텐츠의 재사용 가능성
(컴포넌트냐 아니냐)을 서로 독립적인 축으로 다루겠다는 설계
결정이다. 이는 최신 헤드리스 CMS와 노코드 사이트 빌더들이 수렴해
온 방향, 즉 콘텐츠를 특정 템플릿 타입에 종속시키지 않고 유연한
구조화 데이터로 다루는 흐름과 궤를 같이한다. 다만 이런 통합
모델은 구현 복잡도가 높아, 왜 0.0.x라는 이른 버전 단계에서도
이런 근본적인 아키텍처 결정을 먼저 내렸는지는 이 팀이 처음부터
프로덕션급 복잡도를 감당할 자신감이 있었거나, 혹은 나중에
갈아엎기 어려운 핵심 결정을 서둘러 굳히려 했다는 두 가지 해석이
모두 가능하다.

## 비평

### “등급 없음”이라는 선언과 지속 가능성에 대한 침묵이 공존한다

README는 라이선스를 소개하며 “등급 없음, 오픈코어 꼬리표 없음”이라고
명시적으로 선언한다. 이는 유료 엔터프라이즈 기능을 잠가 두는
오픈코어 모델을 채택하지 않겠다는 강한 약속처럼 들린다. 그러나
정작 이 프로젝트가 어떻게 지속적으로 개발될 것인지, 즉 자금
조달 방식이나 개발팀의 생계 수단에 대한 설명은 전혀 없다.
Motion.page와 Core Framework라는 기존 상용 도구를 만들어 온
팀이라는 점을 고려하면, Instatic 자체는 수익화 계획이 없는지,
아니면 언젠가 방향을 바꿀 가능성이 있는지 현재로서는 알 수
없다. “등급 없음”이라는 라이선스 차원의 약속과 “그래서 어떻게
먹고살 것인가”라는 지속 가능성의 질문은 서로 다른 층위의
문제인데, README는 전자만 답하고 후자는 답하지 않는다.

### AI 에이전트의 28개 도구가 구체적으로 무엇을 하는지 알 수 없다

“실제로 페이지를 편집하는 AI 에이전트”라는 소개와 “28개의
도구”라는 숫자는 제시되지만, 이 도구들이 각각 무엇을 하는지,
사용자가 잘못된 지시를 내렸을 때 되돌릴 방법이 있는지, 에이전트가
만든 변경 사항을 사람이 승인·거부하는 검토 단계가 있는지에
대한 설명이 없다. 특히 이 에이전트가 “실제, 편집 가능한 노드로”
캔버스에 결과물을 만든다고 강조하는 것은, 결과물이 되돌리기
어려운 상태로 프로덕션 콘텐츠에 직접 반영될 수 있다는 뜻이기도
하다. 36개의 세분화된 권한 체계를 갖춘 CMS라는 점을 감안하면,
AI 에이전트가 그 권한 체계 안에서 어떤 위치를 차지하는지(에이전트
자체에도 별도 권한이 있는지, 에이전트를 실행한 사용자의 권한을
그대로 물려받는지)는 상당히 중요한 질문인데 다뤄지지 않는다.

### 자체 아키텍처 테스트가 곧 외부 검증을 대체하지 않는다

README는 `src/__tests__/architecture/`의 테스트로 “아키텍처
무결성”을 검증한다고 밝히지만, 이는 프로젝트가 스스로 정의한
아키텍처 규칙을 스스로 지키고 있는지 확인하는 내부 일관성
테스트이지, 이 아키텍처 자체가 올바른 설계인지를 외부에서 검증한
것은 아니다. 특히 QuickJS-WASM 샌드박스가 실제로 플러그인의
파일시스템·환경 변수 접근을 완벽히 차단하는지, 알려진 샌드박스
탈출 취약점 클래스에 대한 대응이 이뤄졌는지 같은 보안 관련
주장은 자체 테스트만으로는 검증되지 않는다. “플러그인이
데이터베이스를 유출하는 일은 여기서는 일어나지 않는다”는 단정적
문구는, 독립적인 보안 감사나 침투 테스트 결과 없이 제시된
자신감이라는 점에서 신중하게 받아들여야 한다.

## 인사이트

### WordPress 생태계 출신 팀의 이탈은 플러그인 보안 부채가 새로운 CMS 세대를 만들어내는 원동력이 되고 있음을 보여준다

Instatic 팀의 배경, 즉 WordPress 생태계 안에서 수년간 상용 도구를
만들어 오다가 독립적인 셀프호스팅 CMS를 새로 만들기로 한 결정은
개별 사례로 보이지만, 더 큰 패턴의 일부일 가능성이 있다.
WordPress는 방대한 플러그인 생태계 덕분에 성공했지만, 바로 그
개방성이 오랫동안 보안 사고의 주된 원인으로 지목되어 왔다.
Instatic처럼 WordPress 생태계에서 실력을 쌓은 개발자들이 그
생태계의 구조적 약점(검증되지 않은 서드파티 코드가 전체 사이트에
대한 무제한 접근 권한을 갖는 구조)을 정면으로 겨냥한 샌드박스
아키텍처를 들고 독립하는 흐름이 이어진다면, 이는 WordPress
플러그인 생태계의 보안 부채가 누적된 결과가 마침내 대안 CMS
개발이라는 형태로 표출되고 있다는 신호로 읽을 수 있다. 이런
흐름이 계속되면 WordPress 자체도 플러그인 샌드박싱을 강화하는
방향으로 압박을 받을 가능성이 있다.

### AI 페이지 편집 에이전트는 노코드 툴의 다음 경쟁 축이 “블록 배치”에서 “의도 해석”으로 옮겨가고 있음을 보여준다

기존 비주얼 빌더들의 핵심 경쟁력은 얼마나 다양한 블록·템플릿을
제공하고 그것들을 얼마나 매끄럽게 드래그앤드롭으로 조합할 수
있는가였다. Instatic이 “실제로 페이지를 편집하는” AI 에이전트를
핵심 기능으로 내세운 것은, 사용자가 원하는 결과를 직접 블록으로
조립하는 대신 자연어로 의도를 설명하고 그 해석을 에이전트에게
맡기는 방향으로 상호작용의 축 자체가 옮겨가고 있음을 보여준다.
이는 단순히 “AI 기능을 하나 추가했다”는 수준을 넘어, 비주얼
CMS라는 제품 카테고리가 앞으로 얼마나 정교한 블록 라이브러리를
갖췄는지가 아니라 얼마나 정확하게 사용자의 모호한 자연어 지시를
구체적인 편집 가능한 구조로 변환해 내는지로 경쟁하게 될 가능성을
시사한다. 다만 이 전환은 동시에 새로운 위험을 만든다. 블록
배치는 결과가 눈에 보이는 즉시 검증되지만, 자연어 지시의 해석은
사용자가 의도한 것과 에이전트가 실제로 만든 것 사이의 간극을
매번 확인해야 하는 새로운 인지 부담을 만들어낸다.

### 통합 데이터 모델과 세분화된 권한 체계의 조합은 팀 규모 CMS 운영으로의 확장을 염두에 둔 설계다

페이지·포스트·컴포넌트를 하나의 저장소로 통합하면서 동시에 36개의
세분화된 권한을 제공하는 조합은, 이 CMS가 단순히 개인 블로거나
소규모 사이트를 겨냥한 것이 아니라 여러 명의 편집자·디자이너·개발자가
동시에 협업하는 조직에서의 사용을 염두에 두고 있음을 시사한다.
통합된 데이터 모델만 있고 세분화된 권한이 없다면 여러 사람이
같은 저장소를 건드릴 때 통제가 어려워지고, 반대로 세분화된
권한만 있고 데이터가 파편화되어 있다면 그 권한 체계를 각 데이터
타입마다 따로 관리해야 하는 부담이 생긴다. 두 설계를 함께
채택했다는 것은, 이 프로젝트가 아직 0.0.x 버전임에도 이미 팀
단위의 실무 CMS 운영을 최종 목표로 설계되었음을 보여준다. 이는
개인 개발자를 위한 가벼운 도구로 시작해 점진적으로 기능을
확장하는 대신, 처음부터 조직 규모의 요구사항을 상정하고 만들어진
CMS라는 점에서 Ghost나 Strapi 같은 이전 세대의 헤드리스·모던
CMS들이 밟았던 성장 경로와는 다른 출발점을 취하고 있다.
