# GitHub Copilot 코드 리뷰에 AGENTS.md 지원 추가

원문: <https://github.blog/changelog/2026-06-18-copilot-code-review-agents-md-support-and-ui-improvements/>

## 요약

GitHub이 Copilot 코드 리뷰에 두 가지 업데이트를 발표했다.
첫째, 저장소 루트에 `AGENTS.md` 파일을 두면 Copilot 코드 리뷰가 해당 파일을 자동으로 읽어
프로젝트 규약과 기대치에 맞는 피드백을 생성한다.
둘째, UI 개선으로 드래프트(Draft) PR의 리뷰어 선택 화면에 Copilot 옆에 “Request” 버튼이
새로 생겨 게시 전 코드 리뷰 요청을 간소화했다.

세 번째 변경은 PR 타임라인 정리다.
기존에는 Copilot 코드 리뷰 이벤트가 PR 대화 타임라인에 여러 항목으로 분산 기록되었는데,
이제는 같은 이벤트들이 단일 항목으로 통합되어 타임라인 노이즈가 줄었다.
모든 기능은 정식 출시(Generally Available) 상태다.

이번 업데이트는 단독 공지가 아닌 MAI-Code-1-Flash 확장, Copilot 작성 PR 검색 통합,
Copilot Chat 자동 모드 출시와 같은 날 함께 발표되었다.
분량 자체는 1분 분량의 짧은 체인지로그이지만,
`AGENTS.md`라는 파일 관례의 공식 채택이라는 점에서 의미가 작지 않다.

## 분석

### `AGENTS.md`는 파일 기반 AI 지시 레이어의 표준화 시도다

`AGENTS.md`라는 파일명은 GitHub이 독자적으로 발명한 것이 아니다.
AI 에이전트가 저장소 컨텍스트를 파악하기 위해 루트 파일을 읽는 관행은
Claude Code의 `CLAUDE.md`, OpenAI Codex 계열 도구들의 유사 패턴,
그리고 여러 오픈소스 AI 개발 도구들이 각자 관례를 형성하면서 이미 존재하고 있었다.
GitHub이 `AGENTS.md`라는 이름을 공식 지원 파일로 채택한 것은
그 관례를 플랫폼 레벨에서 흡수하는 움직임이다.

이 선택의 함의는 파일명 그 이상이다.
저장소에 AI 지시 파일이 존재한다는 것은 “이 저장소는 AI 에이전트와 함께 작동하도록 설계되어 있다”는
메타 신호가 된다.
팀이 `AGENTS.md`에 코딩 스타일, 커밋 메시지 관례, 특정 패턴에 대한 주의 사항을 기록하면
이 파일은 단순한 설정 파일이 아니라 팀의 AI 협업 정책 문서로 기능한다.

### 드래프트 PR에 AI 리뷰를 허용하는 것은 피드백 루프의 재설계다

드래프트 PR은 전통적으로 “아직 준비되지 않은 코드”를 나타내는 신호였다.
동료들에게 “검토하지 말고 지켜봐 달라”는 의미를 담고 있었고,
리뷰어 픽커에 Copilot “Request” 버튼이 없었던 것은 그 암묵적 약속과 일치했다.
이번 변경은 그 약속을 AI에 대해서만 선택적으로 해제한다.

이 변경이 묻고 있는 질문은 “AI 리뷰는 언제 시작해야 하는가”다.
인간 리뷰어는 완성된 코드를 보지만, AI는 작업 중인 코드에서도 패턴을 감지할 수 있다.
작성자가 드래프트 단계에서 AI 피드백을 받아 수정하고 완성하면,
최종 PR에는 AI 필터를 이미 거친 코드가 올라온다.
이는 리뷰 프로세스의 앞단을 AI가 담당하는 구조로,
인간 리뷰어는 AI가 걸러낸 이후의 코드를 보게 된다.

### 타임라인 통합은 AI 리뷰 이벤트가 이미 과잉임을 간접적으로 인정한다

체인지로그에서 가장 조용하지만 가장 의미 있는 항목은 타임라인 통합이다.
“여러 Copilot 코드 리뷰 이벤트가 단일 항목으로 통합된다”는 설명은
그 전에 여러 항목이 타임라인을 채우고 있었다는 뜻이다.
이는 Copilot이 하나의 PR에서 다수의 리뷰 이벤트를 발생시킬 만큼 자주 실행되고 있었고,
그것이 타임라인을 읽기 어렵게 만들었음을 시사한다.

UX 개선으로 포장된 이 변경은 실제로는 AI 도구가 만들어내는 이벤트 밀도 문제에 대한 대응이다.
인간 리뷰어는 PR 하나에 한두 번 리뷰 이벤트를 남기지만,
AI는 코드가 변경될 때마다, 또는 요청이 있을 때마다 새 이벤트를 생성할 수 있다.
이 밀도 차이가 협업 도구의 UI 설계에 새로운 과제를 만들고 있다는 점을
이번 업데이트가 조용히 드러낸다.

## 비평

### `AGENTS.md`의 실제 지시 범위와 해석 방식이 공개되지 않았다

체인지로그는 `AGENTS.md`가 “관련 지시를 자동으로 읽어 적용한다”고 하지만,
어떤 종류의 지시가 유효한지, 어떤 형식이 요구되는지, 어떤 내용이 무시되는지는 설명하지 않는다.
팀이 `AGENTS.md`에 작성할 수 있는 것의 범위를 모르는 상태에서
이 파일을 어떻게 활용해야 할지 판단하기 어렵다.

더 근본적인 문제는 “관련 지시”가 무엇인지 Copilot이 어떻게 판단하느냐다.
`AGENTS.md`에 “함수 이름에 동사를 먼저 쓰라”고 적으면 이것이 코드 리뷰에 반영되는가?
“레거시 모듈 A에 대한 의존성 추가를 경고하라”는 지시는 이해되는가?
자연어로 쓰인 지시를 AI가 얼마나 일관되게 해석하느냐에 따라
이 기능의 신뢰성이 크게 달라지는데, 그 정보가 없다.

### 드래프트 PR AI 리뷰는 “미완성 코드의 자기 검열”이라는 부작용을 낳는다

드래프트 단계에서 AI 리뷰를 받는 개발자는 AI 피드백을 내재화하면서 코드를 수정한다.
이 과정이 반복되면 개발자가 코드를 작성하는 방식 자체가 “AI가 지적할 것 같은 것을 미리 피하는”
방향으로 조용히 바뀔 수 있다.
이는 AI가 좋은 코드 작성을 돕는 것처럼 보이지만,
실제로는 AI의 선호 패턴이 개발자의 판단을 대체하는 과정이다.

문제는 AI 리뷰어의 취향이 항상 맞지 않는다는 점이다.
AI는 통계적으로 흔한 패턴을 선호하고, 맥락에 따른 예외나 의도적 비정형을 오류로 볼 수 있다.
드래프트 단계부터 AI 피드백을 받은 코드가 팀에 올라오면,
인간 리뷰어는 이미 AI 취향에 맞게 정제된 코드를 보게 된다.
AI가 걸러내지 않는 창의적이거나 실험적인 접근은 드래프트 단계에서 이미 제거된 후일 것이다.

### 타임라인 통합은 AI 이벤트의 가시성 문제를 해결하지 않고 감춘다

여러 Copilot 이벤트를 단일 항목으로 통합하면 타임라인이 깔끔해지지만,
리뷰 과정에서 AI가 실제로 얼마나 자주 개입했는지 추적하기 어려워진다.
이는 사용자 경험 측면에서는 개선이지만, 감사(audit) 가능성 측면에서는 후퇴다.

PR 히스토리는 코드가 어떤 판단 과정을 거쳐 머지되었는지 기록하는 문서 역할을 한다.
인간 리뷰어의 각 리뷰 라운드가 가시적으로 남는 것처럼,
AI 개입도 가시적으로 추적될 수 있어야 규정 준수나 사고 분석 시 유용하다.
“Copilot이 이 PR에서 총 몇 번 의견을 냈고 어떤 내용이었는가”를 나중에 재구성하기가
더 어려워질 수 있다는 점에서 이 변경은 단순한 UI 개선이 아니다.

## 인사이트

### `AGENTS.md`는 저장소 AI 거버넌스의 시작점이다

지금 당장은 `AGENTS.md`가 Copilot 코드 리뷰의 행동을 커스터마이즈하는 설정 파일처럼 보인다.
그러나 이 파일의 진짜 의미는 팀이 AI 에이전트를 어떻게 다뤄야 하는지에 대한
저장소 레벨의 정책 문서가 코드베이스 안에 존재하게 되었다는 점이다.
AI 도구가 저장소 설정 파일을 읽어 행동을 조정하는 패턴이 정착하면,
이 파일은 점차 “이 저장소에서 AI는 무엇을 해도 되고 무엇을 하면 안 되는가”를 명시하는
거버넌스 문서로 진화할 것이다.

역사적 유사점은 `.editorconfig`나 `.eslintrc`의 등장이다.
이 파일들이 처음 나왔을 때는 단순한 린팅 설정이었지만,
시간이 지나면서 팀의 코드 스타일 정책을 명시하는 표준 문서가 되었다.
`AGENTS.md` 역시 같은 경로를 밟을 가능성이 높다.
그 시점이 오면 “AGENTS.md가 없는 저장소는 AI 리뷰를 신뢰할 수 없다”는
기대치가 생길 수도 있다.

### AI 리뷰의 드래프트 단계 진입은 코드 리뷰의 목적을 재정의한다

코드 리뷰의 전통적 목적은 두 가지였다: 결함 발견과 지식 공유.
결함 발견은 AI가 점점 더 잘 하고 있고, 지식 공유는 아직 AI가 하기 어렵다.
드래프트 PR에 AI 리뷰가 도입되면 결함 발견의 상당 부분이 인간의 시야 밖에서
조용히 처리된다.
인간 리뷰어에게 오는 코드는 이미 1차 필터링된 코드다.

이 변화의 이차적 효과는 인간 코드 리뷰의 역할이 “지식 공유와 설계 판단”으로
더 명확하게 특화된다는 것이다.
AI가 스타일, 잠재적 버그, 명명 관례를 잡아내고 나면,
인간은 “왜 이 설계를 선택했는가”, “이 모듈이 장기적으로 어떤 방향으로 가야 하는가”에
더 집중할 수 있다.
이는 코드 리뷰가 더 전략적이고 덜 기계적이 되는 방향이며,
팀이 의식적으로 설계하지 않으면 자연스럽게 따라오지 않는 변화다.

### AI 이벤트 밀도 문제는 협업 도구 UI 전반에 새로운 설계 원칙을 요구한다

타임라인 통합이 필요해진 것은 Copilot만의 문제가 아니다.
AI 에이전트가 PR, 이슈, 프로젝트 보드, 코드 리뷰 등 협업 도구 전반에 통합될수록
“AI가 생성하는 이벤트를 인간의 이벤트와 어떻게 함께 표시할 것인가”라는
새로운 UI 설계 문제가 생긴다.
인간은 하루에 몇 번 이벤트를 생성하지만, AI는 몇 초마다 이벤트를 생성할 수 있다.

이 밀도 차이는 단순히 타임라인을 통합하는 것으로 해결되지 않는다.
AI 이벤트를 얼마나 집계할 것인지, 어느 맥락에서 보여줄 것인지, 어느 것은 영구 기록에서 제외할 것인지는
설계 원칙이 필요한 결정이다.
GitHub의 타임라인 통합은 그 문제에 대한 최초의 공개된 답변이라는 점에서
다른 협업 도구들이 AI 통합 시 참고할 선례가 될 것이다.
