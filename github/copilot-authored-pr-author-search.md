# Copilot이 작성한 PR이 작성자 검색에 포함된다

원문: <https://github.blog/changelog/2026-06-18-copilot-authored-pull-requests-now-included-in-author-searches/>

## 요약

GitHub이 풀 리퀘스트(PR) 검색 기능을 확장해, Copilot 클라우드 에이전트가 생성한 PR도
`author:[username]` 또는 `author:@me` 검색에 포함되도록 변경했다.
이전에는 Copilot이 생성한 PR과 사용자가 직접 작성한 PR을 별도로 검색해야 했지만,
이제 두 유형이 단일 검색 결과에 통합된다.

PR 작성자 표기 방식도 바뀌었다.
전역 PR 대시보드에서 Copilot이 작성한 PR은 AI 에이전트 단독 작성이 아닌
“username with Copilot” 형태로 표기되어, 해당 사용자의 기여로 귀속된다.
기본 PR 조회 뷰에서도 Copilot 생성 PR이 자동으로 포함되어 별도의 필터 없이 전체 목록을 볼 수 있다.

변경 사항은 현재 github.com UI와 GitHub Mobile에 적용되었다.
REST API와 GraphQL API에 대한 동일한 변경은 2026년 7월 16일에 롤아웃될 예정이다.

## 분석

### “작성자” 개념을 확장하는 이 변경이 담고 있는 귀속 모델

이번 변경의 핵심은 Copilot이 생성한 PR을 “해당 사용자의 PR”로 간주한다는 귀속 결정이다.
`author:@me`로 검색했을 때 Copilot 생성 PR이 나온다는 것은
GitHub이 “이 PR은 당신이 작성한 것”이라고 공식적으로 선언하는 것과 같다.
“username with Copilot”이라는 표기는 AI의 관여를 명시하면서도
책임과 귀속을 사람에게 두는 구조를 선택한 것이다.

이 귀속 모델은 현재 소프트웨어 개발 관행에서 가장 자연스러운 선택으로 보이지만,
실제로는 중요한 전제를 내포하고 있다.
Copilot 클라우드 에이전트가 PR을 생성할 때 사람의 개입 정도는 매우 다양할 수 있다.
사람이 구체적인 명세를 주고 결과를 검토한 경우와,
자동화된 트리거로 Copilot이 독립적으로 작성한 경우가
동일한 “username with Copilot” 표기를 갖는다는 것은 뭉뚱그림이다.

### UI 변경과 API 변경 사이의 한 달 간격이 드러내는 것

변경 사항이 github.com UI에는 즉시 적용되었지만 API는 7월 16일로 미뤄진 데는 기술적 이유가 있다.
UI는 검색 결과를 표시하는 레이어에서 Copilot PR을 포함하도록 조정할 수 있지만,
API는 외부 시스템이 의존하는 데이터 계약이라 동일한 변경이 더 복잡한 하위 호환성 문제를 동반한다.
API를 통해 PR 목록을 가져오는 외부 도구들 — CI 시스템, 팀 대시보드, 생산성 추적 도구 — 이
갑작스러운 결과 변화에 영향받지 않도록 한 달의 유예를 준 것이다.

이 간격은 동시에 이 변경이 조용하지 않다는 신호다.
만약 단순한 표기 변경이었다면 UI와 API 동시 배포가 가능했을 것이다.
API 배포를 분리한 것은 이 변경이 데이터 모델 레벨에서 PR의 저자 필드나 귀속 구조를 다루고 있음을 시사한다.

### PR 검색 통합은 Copilot 기여를 개인 생산성 지표로 편입시키는 구조다

이번 변경 이전에는 Copilot이 생성한 PR과 사람이 작성한 PR이 분리된 검색 결과로 존재했다.
이 분리는 “AI가 생성한 것”과 “내가 만든 것” 사이의 경계를 유지해주는 구조였다.
그 경계가 사라지면, 개발자의 PR 생산성 지표는 암묵적으로 Copilot의 출력을 포함하게 된다.

팀 레벨에서 “이번 분기 PR 수”를 지표로 쓰는 경우, Copilot 사용 빈도가 높은 개발자는
동일한 노력으로 더 높은 PR 수를 기록하게 된다.
이것이 생산성 향상으로 읽힐 수도 있고, 지표 인플레이션으로 읽힐 수도 있다.
체인지로그는 이 문제를 전혀 언급하지 않는다.

## 비평

### “username with Copilot” 표기는 귀속을 명확히 하는 것이 아니라 흐린다

“username with Copilot”이라는 표기는 두 가지 상충하는 목표를 동시에 달성하려 한다:
AI 관여를 인정하는 것과 사람에게 책임을 귀속하는 것.
그러나 이 표기 방식은 실제로는 두 목표 모두 절반씩 달성하면서 새로운 문제를 만든다.

Copilot이 자율적으로 PR을 생성한 경우와, 사람이 코드 초안을 작성하고 Copilot이 정리한 경우,
그리고 사람이 PR을 전적으로 작성한 뒤 Copilot의 제안 한 줄만 포함한 경우가
모두 “username with Copilot”으로 표기될 수 있다.
표기의 균일성이 기여 구조의 다양성을 덮어버린다.
코드 리뷰어 입장에서 이 표기는 “AI가 이 PR에서 얼마나 많은 역할을 했는가”에 대한
어떠한 정보도 제공하지 않는다.

### API 지연 공지는 이 변경에 영향받을 도구에 대한 구체적 대응을 제공하지 않는다

7월 16일 API 변경 예고는 외부 도구 개발자들에게 준비 기간을 주는 것처럼 보이지만,
무엇을 준비해야 하는지에 대한 정보가 없다.
`author:@me` API 쿼리가 Copilot PR을 포함하게 될 때,
기존 도구들이 필터링 방법, 결과 수 변화, 페이지네이션 영향 중 어느 것을 다뤄야 하는지
체인지로그는 설명하지 않는다.

GitHub Actions, Slack 통합, 팀 대시보드처럼 PR 카운트나 목록에 의존하는 자동화는
7월 16일 이후 결과가 달라질 수 있다.
한 달의 유예는 주어졌지만 마이그레이션 가이드, API 응답의 변화 목록, 필터 옵션은
이 체인지로그에 없다.
이는 영향받는 개발자들이 변경 후에야 자신의 도구에서 무엇이 달라졌는지 확인해야 한다는 뜻이다.

### 기본 뷰에서 Copilot PR을 포함하는 결정은 사용자 선택이 아니라 강제 통합이다

체인지로그는 이번 변경을 편의 기능으로 제시하지만,
기본 PR 조회에 Copilot PR이 자동 포함되는 것은 사용자가 선택할 수 없다.
Copilot PR과 직접 작성 PR을 구분해 보고 싶은 사용자 — 코드 리뷰 목적이나 자기 작업 파악 — 에게는
이 통합이 오히려 불편함을 만든다.
체인지로그는 “분리 검색이 더 이상 필요 없다”고 표현하지만, 그 분리를 원하는 사람에 대한 언급은 없다.

분리 검색이 가능한 필터 옵션이 제공되는지, 또는 Copilot PR만 제외하는 쿼리 문법이 있는지
체인지로그는 설명하지 않는다.
통합을 기본값으로 만드는 것과 분리를 불가능하게 만드는 것은 다른 결정이지만,
현재 공지만으로는 두 가지 중 어느 쪽인지 판단하기 어렵다.

## 인사이트

### 이 변경은 GitHub의 AI 기여 귀속 정책의 첫 번째 공개 선언이다

“username with Copilot”이라는 표기 방식을 플랫폼 레벨에서 도입했다는 것은
GitHub이 AI 생성 코드의 귀속 문제에 대한 입장을 공식화했다는 의미다.
이전까지 이 문제는 각 조직이나 팀이 자체적으로 관례를 만들어 다뤄야 했다.
GitHub이 “Copilot이 작성해도 그것은 당신의 PR”이라는 입장을 플랫폼 구조로 못박은 것은
앞으로의 법적·조직적 해석에 영향을 준다.

역사적 유사점은 Git에서 `--author` 옵션과 커밋 공동 저자(Co-authored-by) 트레일러의 도입이다.
쌍 프로그래밍이나 코드 리뷰 과정에서 두 사람의 기여를 어떻게 커밋에 기록할지에 대한
비공식 관행이 있다가, GitHub이 Co-authored-by를 UI에서 인식하기 시작하면서 표준이 되었다.
“username with Copilot”도 비슷한 경로를 밟을 가능성이 높다.
이 표기가 오픈소스 프로젝트의 기여자 통계, 취업 포트폴리오, 코드 감사 도구에서
어떻게 해석되고 처리될지는 앞으로 생태계가 결정할 문제다.

### PR 수 지표는 이 변경 이후 의미가 달라진다

팀 단위로 PR 수를 생산성 지표로 쓰는 조직에서 이번 변경은 측정 대상을 바꾼다.
`author:@me`로 집계한 PR 수에 Copilot 생성 PR이 포함되면,
동일한 업무량으로 PR 수가 늘어날 수 있다.
이는 측정값이 올라간 것처럼 보이지만 실제 개발자의 노력이 늘어난 것은 아닐 수 있다.

이 변화의 이차적 효과는 지표 기반 성과 관리 시스템에서 나타난다.
PR 수를 성과 지표로 삼는 기업에서 Copilot을 활발히 쓰는 개발자는
자연스럽게 더 높은 수치를 기록하게 된다.
이것이 의도된 결과라면 — AI 도구 활용을 장려하는 — 문제가 없다.
하지만 지표의 의미가 변했다는 사실을 관리자나 평가 시스템이 인식하지 못한다면,
Copilot 사용 여부가 성과 평가를 왜곡하는 변수가 된다.
GitHub은 이 구조적 영향을 전혀 다루지 않는다.

### AI 저자성 문제는 PR보다 커밋과 코드 소유권(code ownership)에서 더 복잡해질 것이다

이번 변경은 PR 레벨의 귀속 문제를 다루지만, 더 어려운 문제는 커밋 레벨에 있다.
PR은 사람이 시작하고 승인하는 단위이므로 사람에게 귀속하는 논리가 비교적 자연스럽다.
그러나 커밋은 코드 한 줄 한 줄의 변경 기록이고,
`git blame`은 누가 특정 코드 줄을 작성했는지를 추적한다.
AI가 작성한 코드 줄이 `git blame`에서 사람 이름으로 기록된다는 것은
코드 소유권의 의미를 근본적으로 바꾼다.

보안 취약점 분석, 법적 책임 추적, 코드 리뷰 책임 배분은 모두 “누가 이 코드를 썼는가”에 의존한다.
PR 귀속 문제보다 훨씬 세밀하고 중요한 이 문제들을 GitHub은 아직 공식적으로 다루지 않았다.
“username with Copilot” 모델이 PR에서 커밋으로, 커밋에서 `CODEOWNERS` 파일로 확장될 때
귀속의 논리는 점점 더 복잡해진다.
이번 PR 검색 통합은 그 긴 여정의 시작점에 해당한다.
