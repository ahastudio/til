# Copilot Chat이 에이전트 세션을 보게 됐다

원문: <https://github.blog/changelog/2026-06-10-copilot-chat-now-sees-your-agent-sessions/>

## 요약

GitHub Copilot Chat이 클라우드 에이전트 세션과 통합됐다.
이전까지 에이전트가 수행한 작업의 결과는 풀 리퀘스트나 이슈 같은 결과물로만 확인할 수 있었지만,
이제 Copilot Chat 안에서 실행 중이거나 완료된 에이전트 세션의 상태를 직접 조회할 수 있다.

세션이 종료된 후에는 해당 세션에 대한 후속 질문을 이어 진행하거나 추가 작업을 시작할 수 있다.
이 통합은 에이전트의 자율 작업과 사람의 대화형 상호작용 사이의 전환을 매끄럽게 만드는 데 목적이 있다.

두 가지 새로운 도구가 Copilot Chat에 추가됐다.
첫째, “에이전트 로그 가져오기(Get agent logs)”는 클라우드 에이전트가 풀 리퀘스트에서 수행한 작업의 세션 로그를 조회한다.
어떤 수정이 이루어졌는지, 어떤 검증을 수행했는지, 그 이유가 무엇인지를 대화 안에서 직접 물어볼 수 있다.

둘째, “세션 검색(Session search)”은 주제, 제목, 최신순 등을 기준으로 과거 에이전트 세션을 찾고 요약한다.
이전 작업을 이어받아 진행할 때 연속성을 유지하는 데 유용하다.

이 변경은 풀 리퀘스트 생성, 저장소 리서치 등의 자동화 작업을 에이전트에 맡길 때
개발자 워크플로우의 효율을 높이기 위한 것으로 설명된다.

## 분석

### 에이전트 로그 접근이 해결하는 신뢰 문제

자율 에이전트가 코드베이스를 변경하면, 개발자는 “무엇이 바뀌었는가” 외에 “왜 그렇게 바뀌었는가”를 알고 싶어한다.
풀 리퀘스트의 diff는 전자를 보여주지만 후자를 보여주지 않는다.
에이전트 로그는 이 추론 과정을 사후에 검토할 수 있게 만든다.

이것은 에이전트 채택의 핵심 장벽 중 하나를 다루는 설계다.
개발자가 에이전트가 만든 변경을 리뷰할 때, 에이전트의 의도를 이해하지 못하면 리뷰는 형식적인 절차가 된다.
로그를 대화 맥락 안에서 질문할 수 있다면, 리뷰어는 단순히 코드 변경을 보는 것이 아니라
에이전트의 추론을 검토하는 것이 가능해진다.

이 접근은 AI 에이전트의 설명 가능성(explainability)을 별도의 도구가 아닌 기존 워크플로우 안에 통합하는 방식이다.
별도의 로그 뷰어나 디버거 없이, 이미 사용하는 Copilot Chat 안에서 에이전트의 행동을 질문할 수 있다.
인터페이스 전환 없이 설명 가능성을 제공하는 것이 이 설계의 핵심이다.

### 세션 검색이 드러내는 에이전트 작업의 시간성

세션 검색 기능은 표면적으로는 편의 기능이지만, 더 깊은 전제를 드러낸다.
에이전트가 수행한 작업이 조회하고 검색할 수 있는 영속적인 기록으로 존재한다는 것이다.
이것은 에이전트 작업을 일회성 실행이 아닌 이력(history)으로 다루는 모델이다.

개발자의 작업은 시간적 연속성을 가진다.
한 주 전에 에이전트에게 맡긴 의존성 업데이트 작업을 이번 주에 이어 진행하거나,
비슷한 맥락의 이전 에이전트 세션을 참고해 새 작업을 시작하는 패턴이 생기는 것이다.
세션 검색은 에이전트 작업이 단발적인 명령-응답 사이클이 아니라 지속적인 협업 관계로 발전하는 기반을 만든다.

### 두 도구가 함께 형성하는 상호작용 모델

“에이전트 로그 가져오기”와 “세션 검색”은 독립적인 기능이 아니라 함께 특정 상호작용 패턴을 만든다.
세션 검색으로 관련 과거 세션을 찾고, 에이전트 로그로 그 세션의 추론을 이해한 다음,
새 작업을 시작하는 흐름이다.

이 패턴은 사람이 업무를 인계받을 때 하는 행동과 유사하다.
새 팀원이 이전 작업자의 PR을 보고 커밋 메시지를 읽으며 컨텍스트를 파악하는 것처럼,
개발자는 에이전트 세션 로그를 통해 이전 에이전트 작업을 이해하고 이어받는다.
에이전트가 팀의 또 다른 구성원처럼 작업 이력을 남기는 모델이 실현되는 것이다.

## 비평

### 짧은 공지가 제품 경험을 과도하게 단순화한다

이 공지는 매우 짧다.
두 개의 새로운 도구와 그 기능을 각각 한 문장으로 설명하는 것이 전부다.
그러나 “에이전트 로그를 Copilot Chat에서 질문할 수 있다”는 설명은 실제 경험이 어떻게 작동하는지 거의 전달하지 않는다.

에이전트 로그가 얼마나 상세한가? 에이전트가 틀린 판단을 한 경우 그 추론이 로그에 드러나는가?
세션 검색의 결과는 어떤 형식으로 제공되는가? 검색 범위는 개인 세션에 국한되는가, 조직 범위인가?
이 질문들은 기능의 실질적 유용성을 결정하지만 공지는 어느 것도 답하지 않는다.

공지의 역할이 상세한 문서가 아닌 기능 존재의 알림이라는 것은 이해할 수 있다.
그러나 이 정도의 추상화 수준에서는 독자가 기능을 실제로 사용해보기 전까지 유용성을 판단할 수 없다.
“개선됐다”는 주장이 구체적 사용 시나리오 없이 제시되면, 마케팅 언어와 기술 정보의 경계가 흐려진다.

### 에이전트 로그의 접근 권한 모델이 설명되지 않는다

에이전트가 조직 저장소에서 풀 리퀘스트를 생성하는 작업을 수행했을 때,
그 로그에 접근할 수 있는 사람이 누구인가?
작업을 시작한 개인만 볼 수 있는가, 저장소의 모든 구성원이 볼 수 있는가, 조직 관리자가 볼 수 있는가?

이 질문은 단순한 기능 명세의 문제가 아니다.
에이전트 로그에는 에이전트가 접근한 파일, 실행한 명령, 판단 근거가 포함될 수 있다.
이것은 잠재적으로 민감한 정보다.
특히 에이전트가 보안 취약점을 수정하는 작업을 했다면, 그 로그는 미수정 취약점의 상세 정보를 담을 수도 있다.

공지가 접근 권한 모델을 전혀 언급하지 않는 것은 운영 맥락에서 중요한 공백이다.
개인 프로젝트에서는 문제가 되지 않지만, 조직 환경에서는 도입 전에 반드시 확인해야 할 사항이다.

### 세션 연속성 기능이 에이전트 실패 케이스를 어떻게 다루는지 불분명하다

세션 검색은 “이전 작업을 이어받아 진행할 때 연속성을 유지”하는 데 유용하다고 설명된다.
그러나 이전 에이전트 세션이 실패했거나 부분적으로만 완료된 경우는 어떻게 되는가?

실패한 세션도 검색 결과에 포함되는가?
포함된다면 성공한 세션과 어떻게 구분되는가?
실패 원인을 Copilot Chat에서 질문할 수 있는가?
에이전트가 절반만 완료한 작업을 이어받으려면 어떤 정보가 필요하고, 그 정보가 로그에 있는가?

자동화 도구의 실패는 성공만큼 흔하다.
세션 연속성 기능이 성공 케이스만 다루도록 설계됐다면, 실제 운영에서 가장 필요한 순간(문제 발생 후 복구)에 유용하지 않을 수 있다.
이 케이스를 공지가 다루지 않는 것은 낙관적 시나리오만 제시하는 전형적인 패턴이다.

## 인사이트

### AI 에이전트의 관찰 가능성은 새로운 범주의 개발 인프라가 된다

소프트웨어 시스템의 관찰 가능성(observability)은 로그, 메트릭, 트레이싱의 세 축으로 발전해 왔다.
이 개념이 마이크로서비스 아키텍처의 확산과 함께 인프라의 핵심 요소가 된 것처럼,
AI 에이전트의 확산은 에이전트 관찰 가능성이라는 새로운 범주를 만든다.

에이전트 로그를 Copilot Chat에서 조회할 수 있게 된 것은 이 범주의 첫 번째 사용자 인터페이스다.
그러나 이것은 시작일 뿐이다.
에이전트가 어떤 도구를 몇 번 호출했는지, 어느 단계에서 얼마나 많은 토큰을 소비했는지,
어떤 판단 지점에서 대안적 경로를 선택하지 않았는지 — 이것들이 에이전트 트레이싱의 요소가 될 것이다.

개발 조직이 여러 에이전트를 동시에 운영하게 될수록, 에이전트 관찰 가능성은 선택이 아닌 운영의 전제 조건이 된다.
지금은 “에이전트가 뭘 했는지 Chat에서 물어보기”이지만,
이것이 성숙하면 에이전트 성능 대시보드, 비용 분석, 실패 패턴 탐지로 발전할 가능성이 높다.

### 에이전트 세션 이력은 코드베이스와 별개로 관리해야 할 조직 지식이 된다

코드 변경의 이유는 전통적으로 커밋 메시지와 PR 설명에 기록됐다.
에이전트가 코드를 변경할 때 그 이유는 에이전트 세션 로그에 있다.
이 두 가지는 같은 변경에 대한 두 가지 다른 이야기이며, 둘 다 조직의 지식 자산이다.

문제는 에이전트 세션 로그가 코드 저장소와 별도의 시스템에 저장된다는 점이다.
코드를 다른 호스팅으로 이전하면 커밋 이력은 따라가지만, 에이전트 세션 로그는 그렇지 않을 수 있다.
조직이 GitHub Copilot을 해지하면 에이전트가 수행한 모든 작업의 추론 과정을 잃는다.

이것은 벤더 잠금(vendor lock-in)의 새로운 형태다.
코드 자체는 이동할 수 있지만 “왜 이 코드가 이렇게 됐는가”의 맥락은 특정 플랫폼에 남는다.
조직이 에이전트 세션 로그를 어떻게 보존하고 이식 가능하게 관리할지는,
에이전트 자동화가 확산되면서 점점 더 중요한 아키텍처 결정이 될 것이다.

### 대화형 인터페이스가 에이전트 결과의 검증 방식을 근본적으로 바꾼다

전통적인 코드 리뷰는 변경된 코드를 보고 그것이 올바른지 판단하는 과정이다.
에이전트가 변경한 코드를 리뷰할 때도 동일한 방식을 적용하는 것이 현재의 관행이다.
그러나 에이전트 로그를 대화로 질문할 수 있게 되면 리뷰의 성격이 달라진다.

“이 함수를 왜 이렇게 바꿨어?”라는 질문을 에이전트 로그에 할 수 있다면,
리뷰어는 코드의 결과물뿐 아니라 에이전트의 추론 과정을 검토하게 된다.
이것은 코드 리뷰를 넘어 에이전트 판단 리뷰가 되는 것이다.

이 전환은 리뷰어에게 새로운 역량을 요구한다.
코드가 올바른지 판단하는 능력과, 에이전트의 추론이 타당한지 판단하는 능력은 다르다.
전자는 코드를 읽을 수 있으면 되지만, 후자는 에이전트가 어떤 전제와 제약 아래 작동하는지 이해해야 한다.
Copilot Chat의 에이전트 세션 통합은 이 새로운 역량이 개발자에게 필요해지는 미래를 앞당기고 있다.
