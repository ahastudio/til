# GitHub Stacked PRs (gh-stack)

<https://github.github.com/gh-stack/>

## 문제와 구조

큰 PR은 리뷰가 어렵고, 머지가 느리며, 충돌이 잦다.
GitHub Stacked PRs는 이 문제를 정면으로 겨냥한다.
하나의 거대한 변경을 여러 개의 작은 PR 체인으로 쪼개고,
각 PR이 아래 PR의 브랜치를 타겟으로 삼아 순서대로 쌓이는 구조다.
현재 프라이빗 프리뷰 단계이며, `gh stack` CLI 확장과
GitHub UI의 네이티브 스택 맵을 함께 제공한다.

## 플랫폼 네이티브 통합

GitHub이 이 기능을 직접 만든 것이 핵심이다.
브랜치 보호 규칙이 최종 타겟 브랜치 기준으로 적용되고,
CI도 모든 PR이 메인 브랜치를 타겟하는 것처럼 실행된다.
캐스케이딩 리베이스를 한 번의 클릭으로 처리할 수 있고,
스택 전체 또는 일부를 한꺼번에 머지하면
나머지 PR이 자동으로 리베이스된다.
서드파티 도구로는 불가능했던 플랫폼 레벨의 통합이다.

## AI 에이전트 통합

방향도 눈여겨볼 만하다.
AI 코딩 에이전트가 대규모 diff를 스택 단위로 자동 분할하는 통합을
공식적으로 지원한다. 에이전트가 만들어내는 큰 변경을
사람이 리뷰할 수 있는 단위로 쪼개는 문제를 도구 차원에서 풀겠다는 것이다.

## 분석

### 스택 구조의 설계 원리

Stacked PRs의 핵심은 "의존성 있는 변경의 순차적 리뷰"라는 개념이다.
Git의 브랜치 모델 위에 PR 간 순서 관계를 부여하여, 리뷰어가 전체 변경의
일부분만 집중적으로 검토할 수 있게 한다. 이는 Linux 커널 개발의 패치 시리즈,
Meta의 Phabricator 스택드 디프(Stacked Diffs), Google의 Gerrit 체인지셋과
같은 계보에 놓인다.

### 기존 도구와의 경쟁 구도

Graphite, Aviator, git-branchless 등 서드파티 도구들이 이미 스택드 PR
워크플로우를 제공하고 있었다. GitHub가 퍼스트파티로 이 기능을 도입한 것은
플랫폼 레벨의 통합이라는 결정적 차별점을 만든다. 서드파티 도구들은 GitHub
API 위에서 동작하기 때문에 UI 통합, CI 파이프라인 연동, 브랜치 보호 규칙
적용에서 제약이 있었다.

### 스쿼시 머지와의 호환성

Hacker News 논의에서 가장 뜨거운 쟁점 중 하나는 스쿼시 머지 워크플로우와의
호환성이었다. 스쿼시 머지 시 원본 커밋이 사라지면서 의존 브랜치에 팬텀
컨플릭트(phantom conflict)가 발생하는 문제가 있는데, GitHub은
`git rebase --onto`를 사용하여 이를 처리한다고 밝혔다.

### AI 에이전트 통합의 의미

AI 코딩 에이전트가 대규모 diff를 스택 단위로 자동 분할할 수 있다는 점은
주목할 만하다. 에이전트가 생성하는 코드 변경은 종종 한 번에 많은 파일을
수정하는데, 이를 논리적 단위로 나누는 작업 자체가 에이전트에게 위임될 수 있다.

## 비평

### 근본적인 추상화 계층 논쟁

Hacker News의 ninkendo가 제기한 비판이 핵심을 찌른다. 변경이 직교적이면
(orthogonal) 독립적으로 머지할 수 있고, 직교적이지 않으면 순서대로 리뷰해야
하는데 그건 이미 커밋 단위로 가능하다는 것이다. 스택드 PR이 커밋 단위 리뷰
대비 어떤 본질적 이점을 제공하는지는 여전히 논쟁적이다. jenadine도 커밋
단위의 더 나은 UI 지원이면 충분하다고 주장했다.

이에 대한 반론은 steveklabnik이 제시했다. 스택의 각 레이어에 서로 다른
리뷰어를 지정할 수 있고, 각 PR이 독립적인 논의 공간을 갖는다는 점에서
커밋 단위 리뷰와는 질적으로 다르다는 것이다.

### 히스토리 리라이팅의 위험성

mikeocool이 지적한 것처럼 스쿼시와 리베이스를 반복하는 워크플로우는
히스토리를 지속적으로 재작성한다. 이는 강력하지만 위험한 관행이다.
특히 여러 개발자가 같은 스택의 다른 레이어에서 작업할 때
예기치 않은 충돌이 발생할 수 있다. GitHub이 이 부분을 얼마나 안전하게
추상화했는지가 실사용 만족도를 결정할 것이다.

### Jujutsu 등 대안의 존재

MrJohz가 언급한 Jujutsu(jj)는 리베이스의 이점을 Git의 함정 없이 제공하면서도
GitHub과 호환된다. Git 자체의 한계를 GitHub 레이어에서 패치하는 것보다
버전 관리 시스템 자체를 개선하는 접근이 더 근본적인 해결책일 수 있다.
다만 Jujutsu의 생태계 성숙도를 고려하면 당분간은 GitHub의 네이티브 지원이
현실적 선택이다.

## 인사이트

### 플랫폼의 기능 흡수는 생태계 건강의 신호이자 위협이다

Graphite는 GitHub PR 워크플로우의 고질적 문제를 해결하며 유의미한 사용자
기반을 확보했다. GitHub이 이 기능을 네이티브로 흡수하는 것은 플랫폼
비즈니스의 고전적 패턴이다. 트위터가 서드파티 클라이언트를, 슬랙이 서드파티
봇을, iOS가 서드파티 앱의 기능을 흡수해온 것과 같은 궤적이다.

이는 생태계 건강의 양면을 보여준다. 서드파티가 충분한 수요를 입증해야
플랫폼이 움직이므로, 흡수 자체가 문제 정의의 정확성을 방증한다. 그러나
동시에 서드파티 혁신의 인센티브를 약화시킨다. Graphite가 단순히 스택드 PR만
제공하는 것이 아니라 PR 분석, 팀 메트릭스 등 부가 가치를 쌓아온 만큼
당장 사라지지는 않겠지만, 핵심 가치 제안이 약화된 것은 부정할 수 없다.

개발 도구 생태계에서 이 패턴이 반복될수록 "GitHub 위에서 무엇을 만들어야
하는가"라는 질문이 더 어려워진다. GitHub Actions, Copilot, 그리고 이제
Stacked PRs까지 — GitHub은 워크플로우의 전 영역을 자체적으로 커버하려는
야심을 명확히 하고 있다.

### 리뷰 단위의 세분화는 조직 커뮤니케이션 구조를 반영한다

콘웨이의 법칙(Conway's Law)에 따르면 시스템의 구조는 그것을 만드는 조직의
커뮤니케이션 구조를 반영한다. Stacked PRs는 이 법칙의 역방향 적용이다.
리뷰 단위를 세분화하면 커뮤니케이션의 단위도 세분화되고, 이는 결국
팀의 협업 패턴을 변화시킨다.

거대한 PR 하나를 전체 팀이 리뷰하는 모델에서, 스택의 각 레이어를 해당
도메인의 전문가가 리뷰하는 모델로의 전환은 단순한 도구 변화가 아니다.
이는 코드 리뷰의 책임 분산, 지식의 국소화, 그리고 머지까지의 사이클 타임
단축이라는 조직적 변화를 수반한다.

Linux 커널이 수십 년간 패치 시리즈와 서브시스템 메인테이너 구조로 운영되어
온 것은 우연이 아니다. 거대한 코드베이스를 다루는 조직일수록 리뷰 단위의
세분화가 필수적이며, 그것이 메인테이너십의 계층 구조와 맞물려야 한다.
GitHub Stacked PRs는 이 검증된 패턴을 일반 개발팀에게도 접근 가능하게 만든다.

그러나 모든 팀에 이것이 적합한 것은 아니다. 소규모 팀에서는 스택을 관리하는
오버헤드가 리뷰 효율 향상분을 초과할 수 있다. 도구가 존재한다고 해서
반드시 사용해야 하는 것은 아니며, 팀의 규모와 코드베이스의 특성에 맞는
리뷰 전략을 선택하는 것이 핵심이다.

### 도구의 복잡성과 워크플로우의 복잡성은 트레이드오프 관계다

Git의 설계 철학은 "간단한 기본 요소(primitive)를 조합하여 복잡한 워크플로우를
구성한다"이다. 브랜치, 커밋, 리베이스라는 기본 도구로 스택드 워크플로우를
이미 구현할 수 있었다. 그런데 왜 별도의 추상화가 필요한가?

답은 인지 부하(cognitive load)에 있다. 기본 요소의 조합으로 가능하다는 것과
그것을 매일 안전하게 실행할 수 있다는 것은 다른 문제다.
`git rebase --onto`를 정확히 사용하여 스쿼시 머지 후 팬텀 컨플릭트를
해결하는 것은 가능하지만, 모든 팀원이 이를 오류 없이 수행하리라 기대하기는
어렵다. `gh stack`은 이 인지 부하를 도구 레벨로 흡수한다.

이것은 프로그래밍 언어 설계에서도 반복되는 패턴이다. C로 메모리 안전한
프로그램을 작성할 수 있지만 Rust가 이를 타입 시스템으로 보장하는 것처럼,
`gh stack`은 Git의 저수준 조합 가능성 위에 안전한 고수준 추상화를 제공한다.
다만 모든 추상화가 그렇듯, 내부 동작을 이해하지 못한 채 사용하면
문제 발생 시 디버깅이 더 어려워진다는 양날의 검이기도 하다.
