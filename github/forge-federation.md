# Forge 연합이 필요하다

원문: <https://blog.tangled.org/federation/>

HN 토론: <https://news.ycombinator.com/item?id=47948603> (529점, 334개 댓글)

## 요약

Tangled 블로그에 게시된 이 글은 오픈소스 생태계의 GitHub 단일 의존이 구조적 취약점이라고 진단하며, “forge 연합(federation of forges)”을 해결책으로 제시한다. 저자가 만든 Tangled는 AT Protocol 위에서 분산 Git 호스팅을 구현한 플랫폼으로, “중앙화된 시스템은 항상 무너진다, 그러나 프로토콜은 살아남는다”는 명제에서 출발한다.

글은 코드 협업 도구의 역사를 세 시대로 정리한다. 이메일 시대(Git + 메일링 리스트), GitHub 시대(Git + 웹 기반 상호작용), 그리고 연합 시대(Git + ActivityPub/AT Protocol). ForgeFed는 ActivityPub 위에서 forge 연합을 구현하려는 기존 시도이고, Tangled는 같은 문제를 AT Protocol로 접근한다. 두 접근 모두 저장소, 이슈, PR 같은 협업 이벤트를 플랫폼 경계를 넘어 이동시키는 것이 목표다.

Tangled의 기술 구조에서 Git 저장소는 “knot”라 불리는 분산 Git 서버에 분산된다. AT Protocol은 이슈, PR, 팔로우, 스타, “보증(vouch)” 같은 소셜·개발 이벤트를 인증된 방식으로 서버 간에 전달하는 레이어로 동작한다. SSH 키와 협력자 초대도 AT를 통해 처리된다. 각 서버는 독립적으로 운영되면서도 공유된 소셜 그래프를 참조한다 — Bluesky와 동일한 계정 체계를 공유한다.

## 분석

### AT Protocol vs ActivityPub: 데이터 모델의 근본적 차이

danabramov는 AT Protocol의 데이터 모델이 ActivityPub과 구조적으로 다르다는 점을 설명했다.[^danabramov] ActivityPub 기반 Mastodon은 서버들이 직접 메시지를 주고받는 “메시지 패싱” 모델이다. 반면 AT Protocol은 각 사용자가 자신의 데이터 저장소(Personal Data Server, PDS)를 소유하고, AppView 계층이 여러 PDS에서 데이터를 인덱싱해 피드를 구성하는 “데이터 레이어 분리” 모델이다. 이 차이는 콜드 스타트 문제 해결 방식에 직접 영향을 준다. Mastodon에서 새 서버를 시작하면 기존 피드가 없어 사용자 유입이 어렵지만, AT Protocol AppView 모델에서는 인덱서가 전체 네트워크를 이미 보고 있으므로 새 서버도 즉시 기존 사회 그래프에 접근할 수 있다.[^danabramov]

### “Git은 이미 분산형인데 왜 연합이 필요한가”

ecshafer의 반론이 이 논쟁의 핵심을 찌른다: “Git 자체가 이미 완전히 탈중앙화돼 있다. 대부분 프로젝트에는 소수의 커미터만 있다.”[^ecshafer] 이 반론은 기술적으로 맞다. `git clone`, `git push`, `git fetch`는 프로토콜 수준에서 분산형이다. 문제는 코드가 아니라 코드 주변의 협업 레이어다. 이슈 트래커, PR, CI/CD 결과, 릴리스 아티팩트, 커뮤니티 팔로우 관계는 모두 GitHub라는 단일 플랫폼의 데이터베이스 안에 갇혀 있다. 저장소를 이전하면 코드는 보존되지만 이슈 이력, PR 토론, 기여자 관계는 모두 사라진다. 연합이 필요한 것은 Git 자체가 아니라 이 메타데이터 레이어다.

### 이메일 워크플로의 재평가

noirscape는 forge 연합 대신 이메일 기반 패치 워크플로를 현대화하는 방향을 주장했다.[^noirscape] 스팸 방지, 자동 이메일 포매팅, 더 나은 메일링 리스트 인프라를 갖추면 새 프로토콜 없이 같은 문제를 풀 수 있다는 입장이다. Linux 커널 커뮤니티는 여전히 이 방식으로 운영되며, 코드 배포 자체는 완벽히 탈중앙화돼 있다. 이 시각에서 보면 이슈 트래커와 PR UI의 발명이 실수였다 — 그것들이 개발 협업을 플랫폼에 묶어버렸기 때문이다.

### VC 자금과 장기 지속 가능성

code-blooded는 VC 펀딩이 결국 성장 우선 압박으로 이어진다고 지적했다.[^code_blooded] 오픈소스 분산 플랫폼이 VC를 받으면 인쉬티피케이션(enshittification) 경로를 걸을 수밖에 없다는 비판이다. 저자로 추정되는 icy는 그랜트 자금을 찾기가 현실적으로 매우 어려웠다고 응답했다. willio58는 이 긴장을 인정하면서도 “이 영역에서의 경쟁 자체는 장려해야 한다”고 중재했다.[^willio58] 인프라 비용, 보안 패치, 장기 운영을 담보하는 지속 가능한 비즈니스 모델 없이는 오픈소스 연합 프로젝트도 결국 방치된다.

### 조정과 모더레이션 문제

rafram은 연합 특유의 현실적 문제를 제기했다.[^rafram] 스팸, 정치적 갈등, 모더레이션 불일치로 인한 서버 간 디페더레이션(defederation)이 필연적으로 발생하고, Mastodon 생태계에서 이미 이 패턴이 반복됐다. 개발 협업 네트워크에서 이 문제는 더 복잡하다. 스팸 이슈, AI 생성 PR, 합법적 코드 기여와 악성 코드 기여의 구분은 소셜 피드보다 훨씬 정교한 신뢰 모델을 요구한다. “보증(vouch)” 시스템이 이를 해결하려는 시도이지만, 검증된 방식은 아직 없다.

## 비평

### 강점: 협업 메타데이터와 프로토콜의 분리

이 글의 핵심 기여는 “Git은 이미 분산형”이라는 반론을 정면으로 논박하는 대신, 문제를 코드 레이어가 아닌 협업 메타데이터 레이어로 정확히 재정의한다는 점이다. “knot + AT Protocol” 아키텍처는 저장소 데이터(Git)와 협업 이벤트(AT)를 분리해, 각 레이어에서 이미 성숙한 기술을 활용한다. 프로토콜을 새로 발명하는 대신 Bluesky의 기존 소셜 그래프와 계정 인프라를 재사용하는 점도 현실적이다.

### 약점: “GitHub가 무너지고 있다”는 전제의 과장

글은 “GitHub seems to be crumbling”으로 시작하지만 근거가 빈약하다. yodon이 지적한 것처럼,[^yodon] 최근 GitHub 안정성 문제는 AI 코딩 붐으로 인한 일시적 부하 증가일 가능성이 높다. 독점 플랫폼 의존의 구조적 리스크는 실재하지만, 위기감을 고조시켜 논거를 강화하는 방식은 글의 설득력을 오히려 약화시킨다.

### 약점: 프라이빗 저장소와 엔터프라이즈 사용 사례 부재

NetOpWibby가 지적한 것처럼 Tangled에는 프라이빗 저장소 개념이 없다.[^netopwibby] 오픈소스 커뮤니티 프로젝트에는 충분하지만, 상업용 소프트웨어 개발이나 기업 내부 코드 협업에는 적용이 어렵다. “forge 연합”이 GitHub를 대체한다는 주장이 타당하려면 이 사용 사례를 포함해야 하는데, 현재 아키텍처에서 프라이빗 협업을 분산 방식으로 구현하는 방법은 명확하지 않다.

## 인사이트

### 프로토콜 전쟁의 반복 패턴: 이번에는 다를까

jerojero의 XKCD 표준 비유가 이 상황을 정확히 묘사한다.[^jerojero] “이 문제를 해결하려는 표준이 4개 있다. 너무 많다...” ActivityPub, AT Protocol, ForgeFed, 이메일 패치 워크플로 — 각각 다른 설계 철학을 가진 접근들이 경쟁하고 있다. 역사적으로 이런 프로토콜 경쟁은 두 가지 결말 중 하나로 수렴했다. 하나의 지배적 표준이 나머지를 흡수하거나(HTTP, SMTP), 아니면 모두 비주류로 남아 기존 플랫폼이 계속 지배하거나(XMPP, 이메일 기반 소셜).

이번이 다를 수 있는 이유는 지정학적 맥락의 변화다. 미국 기업 클라우드 의존에 대한 EU 국가들의 우려, CLOUD Act 문제, GitHub·Microsoft·정부 감시에 대한 개발자 커뮤니티의 피로감이 동시에 증폭되고 있다. 단순한 기술적 대안이 아니라 정치·제도적 요구가 연합 forge를 뒷받침하는 방향이다. 네덜란드 `code.overheid.nl`이 Forgejo를 선택한 것, 독일 opencode.de, 노르웨이 정부 저장소가 거의 동시에 등장하는 것은 이 수요가 국가 단위로 구체화됐음을 보여준다.

그러나 과거 프로토콜 전쟁의 교훈은 기술적 우월성이 채택을 보장하지 않는다는 것이다. XMPP는 분산형 메시징의 기술적으로 더 나은 대안이었지만 WhatsApp, Telegram에 졌다. 연합 forge가 GitHub를 뒤흔들려면 기술 우위보다 특정 생태계(유럽 정부 조달, 오픈소스 재단)에서의 티핑 포인트가 먼저 필요하다.

### 데이터 주권의 진짜 단위는 “저장소”가 아니라 “협업 그래프”다

“Git clone으로 언제든 이전할 수 있다”는 논리는 개발자들이 가장 흔히 오해하는 지점이다. 실제로 GitHub에서 이전할 때 잃는 것은 코드가 아니다. 수년간 쌓인 이슈 토론, PR 리뷰 맥락, 기여자 히스토리, 프로젝트를 팔로우하는 커뮤니티 관계가 함께 사라진다. 더 정확히는, 코드는 이전되지만 “사회 자본(social capital)”은 플랫폼에 남는다.

이 관점에서 forge 연합의 진짜 가치는 저장소 이전이 아니라 협업 그래프의 이식성이다. 어느 서버에서 PR을 열어도 기여자의 신뢰 이력이 함께 따라오고, 이슈에 남긴 맥락이 서버 이전 후에도 접근 가능하며, 스타·팔로우 관계가 플랫폼 종속 없이 이동할 수 있다면 — 그것이 진정한 탈중앙화다. AT Protocol의 PDS 모델이 이 목표에 이론적으로 잘 맞는 이유가 여기 있다. 각 사용자가 자신의 기여 이력을 소유하고, 플랫폼은 그 데이터를 인덱싱하는 역할로 분리된다.

단, 이 이상을 현실화하는 과정에서 콜드 스타트 문제(madamelic의 지적)[^madamelic], 스팸과 신뢰 문제, 모더레이션 정책 불일치가 연쇄적으로 등장한다. Mastodon 생태계가 10년간 이 문제들과 씨름해온 것처럼, 연합 forge도 같은 경로를 걸을 것이다.

### 오픈소스의 새로운 공공재 딜레마

d_silin은 “한때 사랑받던 제공자가 독점 업체로 무너진 뒤에야 연합이 미래가 된다”고 압축했다.[^d_silin] 이 관찰은 오픈소스 인프라의 반복되는 비극을 포착한다. 오픈소스 커뮤니티는 편의성을 위해 중앙화된 플랫폼에 이주하고, 플랫폼이 수익화·인수·정책 변경을 겪으면 그때서야 탈출을 시도한다. SourceForge → GitHub 이전이 그랬고, 이제 GitHub → ? 논쟁이 반복되고 있다.

구조적 문제는 탈출 비용이 항상 이주 비용보다 크다는 것이다. 초기에 중앙화 플랫폼으로 이주할 때의 편의성 이득이 수년 후 이전하려 할 때의 데이터 손실과 커뮤니티 분산으로 몇 배로 되돌아온다. 이 비대칭성이 오픈소스 커뮤니티가 반복적으로 “좋은 플랫폼이 망가질 때까지” 머무는 이유다.

forge 연합이 이 패턴을 끊으려면 이전 비용의 비대칭을 제거해야 한다. 협업 메타데이터의 이식성이 보장되고, 새 forge에서도 동일한 기여자 네트워크에 접근할 수 있다면 — 플랫폼 전환의 사회적 비용이 제거된다. 그때 비로소 개발자들은 “좋은 플랫폼이 망가지기 전에” 미리 다양화(diversify)하는 선택을 할 수 있게 된다. 이것이 forge 연합이 추구하는 궁극적 목표다.
