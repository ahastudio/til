# GitFut — your GitHub, rated out of 99

<https://gitfut.com/>

<https://github.com/Younesfdj/gitfut>

HN 토론: <https://news.ycombinator.com/item?id=48780812> (69점, 30개 댓글)

## 소개

GitFut는 GitHub 프로필을 FIFA 스타일의 축구 선수 카드로
바꿔주는 프로젝트로, 활동 데이터를 0~99 점수 척도로 시각화한다.
개발자의 커밋, 스타, 풀 리퀘스트 같은 실제 기여 데이터를
FIFA 얼티밋 팀 카드를 닮은 임베드 가능한 형태로 변환해
GitHub 프로필을 게임화하는 것이 이 도구의 목적이다.

카드는 실시간으로 갱신되며 어디에든(프로필, 포트폴리오
등) 임베드할 수 있고, GitHub 활동이 바뀌면 능력치가
자동으로 재계산된다.
URL 파라미터를 통해 국가 국기를 커스텀할 수도 있다.
카드 등급은 브론즈, 실버, 골드, 인폼(In-Form), 올해의
팀(TOTY), 아이콘 마감 처리로 나뉘며, 원본 능력치는
88점을 상한으로 하고 90점 이상을 받으려면 수년간의
영향력이 필요하다.
포지션과 아키타입(archetype)은 능력치 분포 패턴에서
도출된다.

핵심 동작 방식은 여섯 가지 GitHub 신호를 축구 능력치에
매핑하는 것이다.
PAC(가속력)는 최근 1년간의 커밋 수, SHO(슈팅)는 획득한
스타 수, PAS(패스)는 풀 리퀘스트와 팔로워 수, DRI(드리블)는
저장소에 쓰인 언어의 다양성, DEF(수비)는 리뷰와 이슈
활동, PHY(피지컬)는 평생 누적 기여도에서 가져온다.
데이터는 “GitHub의 GraphQL API에서 직접 가져오며, 설문
조사나 자기 보고는 없다”고 설명한다.

사용법은 `![My GitFut card](https://gitfut.com/YOUR_USERNAME.png)`
형태의 임베드 문법을 지원하며, `gitfut.com/<username>.png`는
카드를 이미지로, `gitfut.com/<username>`는 전체 스카우트
리포트를 제공하고, `?country=XX` 파라미터로 국기를
바꿀 수 있다.
기술 스택은 프론트엔드로 Next.js, TypeScript, Tailwind
CSS를 쓰고, 백엔드는 캐싱에 Redis를 사용하며, GitHub
GraphQL API와 통신한다.
언어 구성은 TypeScript 98.2%, CSS 1.7%, JavaScript
0.1%다.
저장소는 2,100개의 스타, 106개의 포크, 105개의 커밋을
기록하며 활발히 개발되고 있고, 라이선스는 MIT다.

## 분석

### GitFut의 핵심 설계는 “측정 가능한 것”을 “게임화하기 쉬운 것”으로 번역하는 작업이다

GitFut가 하는 일의 본질은 GitHub API가 제공하는 원시
지표(커밋 수, 스타 수, 팔로워 수)를 그대로 보여주는 것이
아니라, 이미 대중에게 익숙한 또 다른 채점 체계인 FIFA
선수 카드의 6각형 능력치(PAC/SHO/PAS/DRI/DEF/PHY)로
재포장하는 것이다.
이 번역 작업이 성립하려면 각 GitHub 지표가 축구 능력치의
은유와 그럴듯하게 맞아떨어져야 하는데, 실제 매핑을
보면(커밋 수 → 가속력, 스타 수 → 슈팅) 은유적 연결이
직관적이라기보다는 “가장 그럴듯하게 들리는 이름을 남은
칸에 배정한” 결과에 가깝다.

이 설계 방식 자체는 새로운 것이 아니다.
Codewars의 “kata” 등급, Duolingo의 “XP”와 리그 시스템,
StackOverflow의 평판 점수 등 개발자 대상 서비스들은
오래전부터 활동 데이터를 게임 메커니즘으로 재포장해 왔다.
GitFut가 다른 점은 그 재포장의 대상이 추상적인 점수
체계가 아니라, 이미 전 세계적으로 통용되는 시각 문법을
가진 FIFA 카드라는 기성 포맷이라는 것이다.
이는 사용자가 새로운 채점 체계를 학습할 필요 없이, 이미
알고 있는 시각적 언어에 자신의 개발 활동을 대입해 즉각
이해할 수 있게 만든다.

다만 이 은유 자체가 정확히 어느 게임을 겨냥한 것인지에
대한 혼선도 있었다.
HN 토론에서 한 사용자는 “카드가 월드컵 카드가 아니다.
전체 사이트 스타일로 미루어 볼 때 이건 파니니(Panini)
스티커 앨범 스타일이라고 생각했다”고 지적했고, 다른
사용자도 “이걸 월드컵 스타일이라고 부르진 않겠다. 이건
FIFA 얼티밋 팀 카드를 모방한 것 아닌가”라고 반문했다.[^thi2]
이 지적은 사소해 보이지만, 프로젝트 이름과 홍보 문구가
내세우는 브랜드(월드컵)와 실제로 시각적으로 모방한
대상(FIFA 얼티밋 팀)이 서로 다른 축구 문화 코드에
속한다는 것을 보여주며, 이는 프로젝트가 정확히 어떤
하위문화의 친밀감을 겨냥하고 있는지 스스로도 완전히
합의되지 않았을 가능성을 시사한다.

### 캡과 등급 구조는 “꾸준함”을 “실력”보다 우위에 두는 가치 판단을 내장한다

원본 능력치가 88점에서 상한(cap)에 걸리고 90점 이상을
받으려면 “수년간의 영향력”이 필요하다는 설계는, 이
시스템이 순간적인 폭발적 기여보다 장기간에 걸친 지속적
기여를 더 높게 평가하도록 의도적으로 설계되었다는 것을
보여준다.
단기간에 코드를 대량으로 작성하거나 스타를 많이 받는
저장소 하나를 만드는 것만으로는 최고 등급에 도달할 수
없고, 시간이라는 변수가 점수 계산에 구조적으로 개입한다.

이 설계 판단은 GitHub 생태계에서 실제로 벌어지는 논쟁,
즉 “한 번의 히트작을 낸 개발자”와 “꾸준히 여러 프로젝트에
기여하는 개발자” 중 누가 더 뛰어난 개발자인가라는 논쟁에서
암묵적으로 후자의 손을 들어주는 셈이다.
이는 중립적인 지표 집계가 아니라, 설계자가 “좋은
개발자란 무엇인가”에 대해 이미 내린 가치 판단을 점수
체계 안에 숨겨 놓은 것이다.

### GitHub API를 유일한 데이터 소스로 삼는 선택은 검증의 정확성과 조작 가능성 사이의 트레이드오프를 만든다

README는 “설문 조사나 자기 보고는 없다”는 점을 강조하며
GitHub GraphQL API만을 데이터 소스로 삼는다는 것을 신뢰성의
근거로 내세운다.
이는 사용자가 직접 점수를 입력할 수 있는 시스템과 비교하면
분명한 장점이다.
그러나 이 선택은 동시에 GitHub 활동 자체가 얼마든지
인위적으로 부풀려질 수 있다는 사실을 그대로 물려받는다는
뜻이기도 하다.

커밋 수는 의미 없는 공백 수정이나 자동 생성 커밋으로
쉽게 늘릴 수 있고, 스타 수는 스타 교환 커뮤니티나 봇을
통해 조작된 사례가 이미 여러 차례 보고된 바 있다.
즉 “설문이나 자기 보고가 없다”는 것이 “조작이 불가능하다”는
뜻은 아니며, GitFut의 신뢰성은 결국 GitHub 플랫폼 자체가
가진 신뢰성의 한계를 그대로 물려받는다.
README는 데이터 출처의 객관성만 강조할 뿐, 그 출처
데이터 자체가 게임화될 수 있다는 가능성은 다루지 않는다.

## 비평

### “객관적 데이터”라는 주장은 매핑 함수 자체의 자의성을 가린다

README는 데이터 수집 과정이 설문이나 자기 보고 없이
이뤄진다는 점을 들어 객관성을 강조하지만, 이 객관성은
원시 데이터 수집 단계에만 해당하는 이야기다.
그 원시 데이터를 여섯 개의 능력치로 변환하는 매핑
함수(커밋 수를 PAC 몇 점으로 환산할지, 언어 다양성을
DRI 몇 점으로 환산할지)는 전적으로 개발자의 주관적
설계 결정이며, 이 변환 함수 자체는 README에 공개되지
않는다.

이는 “데이터는 객관적이지만 결론은 주관적일 수 있다”는
흔한 통계적 함정의 한 사례다.
두 명의 개발자가 완전히 다른 방식으로 기여했더라도(한
명은 적은 수의 대규모 저장소에 깊이 기여하고, 다른 한
명은 많은 수의 작은 저장소에 얕게 기여) 매핑 함수의
설계에 따라 전혀 다른 점수 분포가 나올 수 있다.
README가 “객관적 데이터”라는 문구로 신뢰성을 주장하는
것은, 실제로 점수를 결정하는 것이 데이터가 아니라
비공개된 변환 함수라는 사실을 흐린다.

실제 사용자 경험에서도 이 매핑 함수가 원본 데이터를
제대로 반영하지 못하는 사례가 보고되었다.
한 사용자는 “카드에는 내가 PR을 8개 했다고 나오는데,
실제로는 최소 수천 개는 했다. ‘Pull Shark’ 배지 개수를
보면 대략적인 규모를 짐작할 수 있을 정도다. 커밋 수도
올해 것만 보는 것 같다”고 지적했다.[^jasonjmcghee]
이는 “객관적 데이터”라는 주장이 데이터 수집 단계의
객관성과 무관하게, 그 데이터를 실제로 얼마나 정확히
집계했는지에 대한 구현상의 결함으로 무너질 수 있음을
보여준다.
PR 수를 심각하게 과소 계산하는 버그가 있다면, 이는
매핑 함수의 자의성 문제보다 훨씬 더 기초적인 신뢰성
문제이며, 애초에 “설문이나 자기 보고가 없다”는 주장이
전제하는 “API에서 가져온 숫자는 정확하다”는 가정 자체가
흔들린다.

### 여섯 가지 능력치가 실제로 서로 다른 종류의 기여를 측정하는지 검증되지 않는다

FIFA의 여섯 능력치(PAC/SHO/PAS/DRI/DEF/PHY)는 축구
선수의 실제로 서로 구별되는 신체적·기술적 속성을
반영하도록 오랜 시간에 걸쳐 정교화된 체계다.
반면 GitFut가 매핑한 여섯 개의 GitHub 지표, 즉 커밋 수·
스타 수·풀 리퀘스트와 팔로워 수·언어 다양성·리뷰와 이슈·
누적 기여도는 서로 독립적인 속성이라기보다는 상당 부분
상관관계가 높은 지표들이다.

예를 들어 커밋을 많이 하는 개발자는 대체로 풀 리퀘스트도
많이 만들고 이슈에도 많이 관여하는 경향이 있어, PAC과
PAS와 DEF가 실제로는 하나의 근본적인 요인(전반적인 활동량)에서
파생된 결과일 가능성이 크다.
만약 그렇다면 여섯 각의 “레이더 차트”가 실제로는 서로
다른 여섯 가지 재능을 보여주는 것이 아니라 하나의 활동량
지표를 여섯 개의 다른 이름으로 반복해서 보여주는 착시에
가까울 수 있는데, README는 이 지표들 사이의 상관관계
검증 결과를 전혀 제시하지 않는다.

### 국가 국기를 자유롭게 지정할 수 있는 기능은 검증되지 않은 정체성 주장을 허용한다

README는 URL 파라미터를 통해 국가 국기를 커스텀할 수
있다고 설명하는데, 이는 GitHub 프로필의 실제 등록 정보나
검증된 위치 데이터와 무관하게 사용자가 원하는 어떤
국기든 카드에 표시할 수 있다는 뜻이다.
실제 FIFA 카드에서 국적은 선수의 실제 국적을 반영하는
검증된 정보인 반면, GitFut의 국기는 순전히 사용자가
선택하는 장식 요소에 가깝다.

이 설계 선택 자체가 반드시 문제라고 할 수는 없지만,
카드의 다른 요소들(능력치)이 “객관적 데이터”를 강조하는
것과 대조적으로 국기만큼은 아무 검증 없이 자유롭게 설정
가능하다는 비일관성은 README에서 설명되지 않는다.
카드 전체의 신뢰성을 “데이터 기반”이라고 내세우면서
정작 시각적으로 가장 눈에 띄는 요소 중 하나(국기)는
순수하게 임의적이라는 점은, 이 카드가 실제로 얼마나
“객관적 스카우트 리포트”에 가까운지, 얼마나 “꾸미기용
장난감”에 가까운지에 대한 근본적인 질문을 제기한다.

국적 표시가 사용자 지정이 아니라 자동 추론에 의존하는
경우에도 문제가 발생했다.
한 사용자는 예시로 등장한 Linus Torvalds의 카드를 보고
“Linus는 핀란드 사람인 줄 알았는데”라며 국기 표시가
실제와 다르다는 점을 지적했다.[^kbbgl87]
사용자가 직접 지정하는 국기가 임의적이라는 문제에 더해,
자동으로 추론된 국적 정보조차 부정확할 수 있다는 사실은
“객관적 데이터”라는 카드의 핵심 주장을 국기라는 부수적
요소에서마저 훼손시킨다.

경쟁 지향적 게임화 자체에 대한 근본적인 반대 의견도
있었다.
한 사용자는 “랭킹 알고리즘 자체가 의심스럽고 쉽게 조작될
수 있다. 우리는 GitHub 통계를 굳이 ‘게임화’할 필요가
없고, 그렇게 하는 것은 오히려 상당히 건강하지 못한
일일 가능성이 크다”고 비판했다.[^chank]
이 비판은 앞서 다룬 매핑 함수의 자의성이나 데이터 정확성
문제보다 한 걸음 더 나아가, 애초에 개발 활동을 경쟁적으로
순위 매기는 시도 자체의 바람직함에 의문을 제기한다는
점에서, 이 프로젝트가 안고 있는 가장 근본적인 차원의
반론이라고 볼 수 있다.

## 인사이트

### 개발자 활동의 게임화는 GitHub의 기여도 그래프가 이미 시작한 인센티브 왜곡을 심화시킬 수 있다

GitHub의 기여도 그래프(잔디)는 이미 오래전부터 개발자들이
매일 초록색 칸을 채우기 위해 의미 없는 커밋을 만들거나,
비공개 저장소의 기여를 공개 저장소로 옮기는 등의 행동을
유발한다는 비판을 받아 왔다.
GitFut는 이 기여도 그래프 위에 FIFA 카드라는 훨씬 더
강력한 사회적 비교 메커니즘(레이더 차트, 등급 시스템,
직접적인 능력치 비교)을 얹음으로써, 이미 존재하던 인센티브
왜곡을 한 단계 더 증폭시킬 잠재력을 가진다.

FIFA 카드는 특정 선수를 다른 선수와 직접 비교하고 순위를
매기도록 설계된 포맷이며, 이 비교 지향적 설계가 축구
게임에서 큰 인기를 끈 핵심 이유이기도 하다.
같은 메커니즘이 개발자 커뮤니티에 적용되면, “좋은 코드를
작성하는 것”보다 “카드의 숫자를 올리는 것” 자체가 목표가
되는 굿하트의 법칙(Goodhart's Law) 현상이 GitHub 잔디
문제보다 더 노골적인 형태로 나타날 수 있다.
카드가 임베드되어 프로필과 포트폴리오에 공개적으로
전시된다는 점은 이 사회적 비교 압력을 한층 더 강화한다.

### FIFA라는 은유의 선택은 이 도구가 겨냥하는 세대와 문화적 코드를 드러낸다

GitFut가 굳이 FIFA 얼티밋 팀 카드라는 특정 포맷을 차용한
것은 우연이 아니다.
이는 축구 비디오 게임 문화에 친숙한 특정 세대, 특히
2000년대 이후 출생한 개발자들을 겨냥한 문화적 코드
선택이다.
이 세대에게 “인폼”, “TOTY”, “아이콘” 같은 FIFA 특유의
용어는 별도의 설명 없이도 즉각적인 정서적 반응(희소성,
성취감, 등급 상승의 짜릿함)을 불러일으키는 코드로
작동한다.

이는 개발자 도구가 점점 더 특정 하위문화의 언어를
차용해 친밀감을 형성하려는 더 넓은 흐름의 사례로 볼 수
있다.
과거 개발자 도구들이 SF나 판타지 문화의 은유(길드,
레벨업, 경험치)를 즐겨 차용했다면, GitFut는 축구
게임이라는 또 다른 하위문화를 겨냥함으로써 기존 개발자
문화 코드와는 다른 신규 사용자층, 즉 스포츠 게임 문화에
익숙하지만 전통적인 “개발자 게이미피케이션” 미학에는
둔감했을 수 있는 사용자층까지 끌어들이려는 시도로
해석할 수 있다.

### 캐싱을 위한 Redis 도입은 이 서비스가 API 레이트 리밋과의 싸움을 전제로 설계되었음을 보여준다

README가 백엔드에 Redis를 캐싱 용도로 명시한 것은
사소한 기술 스택 나열이 아니라, GitFut가 처음부터
GitHub GraphQL API의 레이트 리밋 제약과 씨름해야 하는
서비스로 설계되었다는 것을 보여주는 단서다.
카드가 임베드되어 여러 곳(프로필, 포트폴리오, README)에서
동시에 로드될 수 있다는 점을 고려하면, 사용자 하나의
카드가 여러 곳에서 열람될 때마다 GitHub API를 매번 호출하는
것은 API 쿼터를 빠르게 소진시킬 위험이 있다.

이 구조는 GitHub API를 기반으로 하는 다른 서드파티
서비스들(예: shields.io의 배지, GitHub 프로필 통계
카드)이 공통으로 마주하는 문제이며, Redis 캐싱은 이런
서비스들이 대체로 채택하는 표준적인 해법이다.
다만 이 캐싱 계층은 동시에 “실시간 갱신”이라는 README의
약속과 긴장 관계에 있다.
캐시된 데이터는 정의상 최신 상태보다 뒤처질 수밖에
없으며, 캐시 유효 기간을 얼마나 짧게 잡을지는 “실시간성”과
“API 쿼터 절약” 사이의 트레이드오프를 결정하는 핵심
변수인데, README는 이 캐시 유효 기간에 대해서는 전혀
언급하지 않는다.

### 급증하는 트래픽에 대한 대비 없이 바이럴리티를 겨냥한 설계는 서비스 자체의 신뢰도를 갉아먹을 수 있다

임베드 가능성과 소셜 공유를 핵심 기능으로 내세운 서비스는
그 특성상 특정 순간에 트래픽이 폭발적으로 몰리는 바이럴
확산에 취약하다.
실제로 HN에 소개된 직후 한 사용자는 “두들겨 맞고 있는
건가요? 어떤 이름을 넣어도 아무 반응이 없고 예시를
클릭해도 마찬가지입니다”라고 보고했다.[^fredisfred]
다른 사용자는 사이트 전체의 완성도에 대해 “전체적인
스타일로 미루어 볼 때 이건 바이브 코딩(vibecoded)된
것 같다”고 평하기도 했다.[^xinayder]

이 두 반응은 서로 다른 층위의 우려를 가리킨다.
트래픽 폭주로 인한 장애는 Redis 캐싱 설계가 실제 부하
상황에서 얼마나 견고한지에 대한 실증적 시험대가 되며,
바이브 코딩이라는 인상은 이 프로젝트가 얼마나 견고한
엔지니어링 관행 위에서 만들어졌는지에 대한 근본적인
의구심을 제기한다.
바이럴 확산을 유도하는 임베드 가능한 카드라는 설계
목표와, 그 확산이 실제로 일어났을 때 서비스가 감당할
수 있는 안정성 사이의 간극은, README에 나열된 기술
스택만으로는 가늠할 수 없는 프로젝트의 실질적 성숙도를
보여주는 지표다.
