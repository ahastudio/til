# Copilot Usage Metrics API, AI 도입 코호트 분류 추가

원문: <https://github.blog/changelog/2026-05-29-copilot-usage-metrics-api-adds-cohorts-for-ai-adoption/>

## 요약

GitHub이 Copilot Usage Metrics API에 코호트(cohort) 분류 기능을 추가했다.
이 업데이트는 조직이 개발자들의 AI 도구 도입 수준을 더 세밀하게 추적할 수 있게
해준다. 기존의 단순한 활성 사용자(active user) 집계를 넘어, 개발자가 어떤
Copilot 기능을 얼마나 깊이 사용하고 있는지를 4단계 위상(Phase)으로 분류한다.

각 활성 사용자는 28일 사용 패턴을 기준으로 Phase 0부터 Phase 3 중 하나에
배정된다. Phase 0은 특정 기준을 충족하지 못한 사용자이고, Phase 1은 코드 완성
및 IDE 에이전트 모드를 사용하는 개발자다. Phase 2는 클라우드 에이전트, 코드
리뷰, CLI 중 하나의 GitHub 기반 에이전트 표면(agent surface)을 사용하는
단계이며, Phase 3은 두 개 이상의 에이전트 플랫폼 혹은 GitHub Copilot 앱을
활용하는 다중 표면(multi-surface) 사용자다.

기술적으로는 사용자 수준 리포트에 `ai_adoption_phase` 필드가 추가되었고,
엔터프라이즈 및 조직 수준 리포트에는 `totals_by_ai_adoption_phase` 배열이
포함된다. 분류 스키마에는 `version` 필드가 포함되어 향후 기준이 변경되더라도
과거 데이터와의 호환성을 유지할 수 있도록 설계되었다.

각 위상별로 집계되는 지표는 활성 사용자 수(28일 중 최소 2일 활동), 코드 생성
수락률(code acceptance rate), 코드 라인 통계, 풀 리퀘스트 관련 활동, 병합
소요 시간(median time-to-merge) 등이다. 이를 통해 조직은 Copilot 도입
현황을 진단하고, 단계별 맞춤형 교육 및 롤아웃 프로그램을 설계할 수 있다.

## 분석

### 측정의 단위를 “사용 여부”에서 “사용 깊이”로

이번 업데이트의 핵심은 측정 단위의 변화다. 기존 지표는 활성 사용자가 몇 명인지
세는 데 초점을 뒀다면, 위상 분류는 사용자가 AI와 어떤 방식으로, 얼마나 깊이
상호작용하는지를 측정한다. 코드 완성 단계(Phase 1)와 다중 에이전트 활용
단계(Phase 3)는 기술적, 조직적 맥락이 완전히 다르다. 이 구분이 생기면 단순히
“Copilot 도입률 80%”라는 숫자가 아닌 “Phase 3 사용자가 전체의 20%”처럼
AI 도입의 질적 수준을 가시화할 수 있다.

### 에이전트 표면의 계층화

Phase 2와 Phase 3의 구분 기준이 “에이전트 표면(agent surface)의 수”라는 점이
흥미롭다. GitHub은 클라우드 에이전트, 코드 리뷰, CLI를 별개의 에이전트
표면으로 정의하고, 이를 조합하는 방식으로 성숙도를 측정한다. 이는 GitHub이
Copilot을 단일 도구가 아닌 여러 컨텍스트에 걸친 에이전트 생태계로 설계하고
있음을 반영한다. 사용자가 단일 에이전트에 머무르는지, 아니면 여러 에이전트를
넘나들며 작업하는지를 추적하는 것은 복잡한 엔지니어링 워크플로우에서의 AI
통합 수준을 가늠하는 실용적인 지표다.

### `version` 필드의 의미

분류 스키마에 `version` 필드를 포함시킨 것은 작지만 중요한 설계 결정이다.
AI 도입의 기준점 자체가 빠르게 이동하고 있기 때문에, 오늘의 Phase 3가 내년의
Phase 1이 될 수 있다. `version` 필드는 기준이 바뀌더라도 과거 데이터의 의미를
보존하겠다는 의도다. 이는 API 설계 차원에서 장기 데이터 활용 가능성을
고려한 것이며, 조직이 시계열 분석을 안정적으로 수행할 수 있게 한다.

## 비평

### 강점: 실행 가능한 지표

단계별 사용자 분포와 각 단계의 코드 수락률, 시간 절감 효과를 연결할 수 있다면,
조직은 “Phase 2에서 Phase 3으로 이동한 개발자의 생산성 변화”처럼 구체적이고
실행 가능한 인사이트를 얻을 수 있다. 기존의 단순 집계 지표에 비해 훨씬 풍부한
분석이 가능하다.

### 약점: 위상 기준의 불투명성

Phase 기준이 GitHub 내부에서 정의되며 사용자는 이를 변경할 수 없다. 조직마다
Copilot 활용 전략이 다를 수 있는데, 예를 들어 CLI보다 IDE 에이전트에 집중하는
팀이 Phase 2를 달성하지 못했다고 해서 AI 도입이 미흡하다고 볼 수는 없다.
고정된 위상 분류 체계는 다양한 워크플로우를 단일 잣대로 재단할 위험이 있다.

### 약점: “도입”과 “활용 가치” 혼동 가능성

사용 빈도나 에이전트 표면의 수가 실제 가치 창출과 반드시 비례하지는 않는다.
Phase 3 사용자가 Phase 1 사용자보다 비즈니스 가치를 더 많이 창출한다는 보장이
없다. 이 API는 도입 행동(behavior)을 측정하지, 결과(outcome)를 측정하지 않는다.
이 점을 혼동하면 위상 점수를 높이기 위한 형식적 사용이 늘어날 수 있다.

## 인사이트

### AI 도입 측정의 패러다임 전환: “몇 명이 쓰나”에서 “어떻게 쓰나”로

소프트웨어 산업은 오랫동안 도구 도입률을 라이선스 수나 로그인 횟수로 측정해
왔다. 이 방식은 SaaS 시대에도 유효했는데, 도구의 기능이 비교적 균질하고
사용 방식이 단순했기 때문이다. 그러나 AI 에이전트는 다르다. 같은 도구를 써도
어떤 사람은 코드 자동 완성에만 쓰고, 어떤 사람은 에이전트에게 작업 전체를
위임한다. 이 둘 사이의 생산성 격차는 수배에서 수십 배에 이를 수 있다.

GitHub의 코호트 분류는 이 사실을 인정하는 첫 번째 공식 측정 체계다. 단순
도입률이 아닌 사용 깊이를 측정하기 시작했다는 것은, AI 도구 도입이 설치
완료(installation complete)가 아닌 역량 변환(capability transformation)임을
기업이 인식하기 시작했다는 신호다.

이 패러다임 전환은 인사(HR)와 조직 설계에도 파급된다. “AI를 쓰는 개발자”와
“AI와 함께 일하는 개발자”를 구분할 수 있게 되면, 채용, 평가, 교육의 기준이
바뀐다. Phase 3 개발자가 Phase 1 개발자보다 2배 빠르다는 데이터가 쌓이면,
이는 개발자 역량 평가의 새로운 축이 된다.

### GitHub의 에이전트 생태계 전략: 락인(Lock-in)의 새로운 형태

Phase 2와 Phase 3의 구분 기준이 “GitHub 기반 에이전트 표면”이라는 점은
전략적으로 매우 의미심장하다. 클라우드 에이전트, 코드 리뷰 에이전트, CLI
에이전트 모두 GitHub 플랫폼 위에서 동작한다. 즉, 개발자가 더 높은 위상으로
이동할수록 GitHub 에코시스템에 더 깊이 통합된다.

전통적인 플랫폼 락인은 데이터와 워크플로우를 통해 이루어졌다. GitHub은 이미
코드 저장소(repository), 이슈 트래커, CI/CD로 강력한 락인을 형성하고 있다.
여기에 에이전트 표면을 추가함으로써, 이제는 개발자의 인지(cognition) 자체가
GitHub 위에서 작동하게 된다. 코드를 저장하는 것이 아니라 코드를 생각하는
과정이 GitHub에 의존하게 되는 것이다.

이 측정 체계는 그 전략의 실행 도구다. 조직이 위상 분포를 모니터링하고 개선하려
할수록, GitHub 에이전트를 더 많이 사용하게 되고, 이탈 비용은 기하급수적으로
높아진다. 지표 시스템 자체가 플랫폼 심화를 유도하는 피드백 루프(feedback loop)
로 설계된 것이다.

### 측정 가능한 것이 관리된다: 코호트 지표가 만드는 새로운 조직 압력

피터 드러커의 “측정할 수 없으면 관리할 수 없다”는 명제는 AI 도입에도 그대로
적용된다. 위상 분류 지표가 엔터프라이즈 대시보드에 나타나는 순간, 이는 경영진
보고의 KPI가 된다. “Phase 3 사용자 비율을 올 분기까지 30%로 끌어올리자”는
목표가 설정되기 시작한다.

이 압력은 긍정적 결과를 낳을 수도 있다. 조직이 체계적인 교육 프로그램을
설계하고, AI 도입 장벽을 제거하며, 성공 사례를 확산시키는 계기가 된다.
실제로 GitHub은 이 지표를 “맞춤형 교육 및 롤아웃 프로그램”에 활용하라고
권고하고 있다.

그러나 부작용도 예상된다. 위상 점수를 높이기 위한 형식적 사용, 즉 실제
워크플로우에 통합하지 않고 단순히 여러 에이전트 표면을 체크리스트처럼
터치하는 행동이 나타날 수 있다. 측정 체계가 만들어지면 측정 체계를 게임하는
행동도 함께 등장하는 것은 어느 조직에서나 반복되는 패턴이다. GitHub이
실제 가치 창출을 측정하는 아웃컴(outcome) 지표를 함께 제공하지 않는다면,
이 위상 분류는 표면적 도입을 촉진하는 도구로 전락할 위험이 있다.
