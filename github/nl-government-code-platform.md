# 네덜란드 정부 오픈소스 코드 플랫폼 소프트 런칭

원문: <https://www.nldigitalgovernment.nl/news/soft-launch-for-government-open-source-code-platform/>

HN 토론: <https://news.ycombinator.com/item?id=47945918> (520점, 117개 댓글)

## 요약

네덜란드 내무부(BZK) 산하 오픈소스 프로그램 오피스(OSPO)가 공공기관 전용 코드 협업 플랫폼 `code.overheid.nl`을 소프트 런칭했다. 플랫폼 기반 기술은 Forgejo — GitHub·GitLab의 “오픈소스·유럽산·주권 대안”으로 소개된다. 현재는 파일럿 단계로 전체 정부 기관에 개방돼 있지 않으며, DAWO(SSC-ICT), Opensourcewerken, developer.overheid.nl이 공동으로 구축했다.

플랫폼의 목표는 단순한 저장소 호스팅을 넘어 정부 차원의 공유 Git 인프라를 구성하는 것이다. 여러 공공기관이 공동으로 개발하는 오픈소스 프로젝트를 한 곳에 모으고, 코드가 공개적으로 검토·기여될 수 있는 환경을 만든다. 핵심 철학은 “디지털 주권(digital sovereignty)” — 민간 기업, 특히 미국 기반 클라우드 플랫폼 의존도를 줄이고 공공 데이터와 코드에 대한 통제권을 되찾는 것이다.

HN 커뮤니티는 이 소식을 전반적으로 긍정적으로 받아들였지만, 배경에는 더 심각한 긴장이 깔려 있다. 네덜란드 시민 인증 시스템 DigiD의 호스팅 업체 Solvinity가 미국 기반 기업 Kyndryl에 인수되는 과정이 진행 중이다. US CLOUD Act·PATRIOT Act 하에서 사실상 모든 네덜란드 시민 인증 데이터가 미국 정부의 접근 범위에 들어올 수 있다는 우려가 제기됐다. GDPR과의 법적 충돌 가능성도 동시에 지적됐다.

## 분석

### Forgejo 선택의 의미

Forgejo는 Gitea에서 파생된 Go 기반 경량 Git 호스팅 솔루션이다. GitHub의 완전한 기능 집합이나 GitLab의 엔터프라이즈 기능을 모두 갖추진 않지만, EU 기반 Codeberg 재단이 주도하는 완전 오픈소스 프로젝트라는 점이 핵심 선택 이유로 보인다. luplex는 독일이 이미 GitLab 기반의 유사한 포털 opencode.de를 운영 중이며, 강화된 컨테이너 이미지를 container.gov.de에서 제공한다고 밝혔다.[^luplex] 네덜란드는 같은 방향이지만 GitLab 대신 Forgejo를 선택했다. GitLab에 대해서는 jillesvangurp가 “인쉬티피케이션(enshittification)이 착실하게 진행 중이다”라고 평했고,[^jillesvangurp] 오픈소스 기여 기능들이 유료 플랜으로 이동하는 흐름이 정부 플랫폼에 부적절하다는 판단이 작용한 것으로 보인다.

### DigiD 위기와 더 큰 문맥

noirscape는 이번 플랫폼 론칭의 배경이 되는 DigiD 위기를 상세히 설명했다.[^noirscape] DigiD는 네덜란드 정부의 OAuth 기반 단일 인증 시스템으로, 사실상 모든 시민이 정부 서비스 접근에 사용한다. Solvinity(DigiD 호스팅)의 Kyndryl 인수가 완료되면, CLOUD Act에 의해 미국 정부는 DigiD에 저장된 거의 모든 네덜란드 시민 데이터에 접근 요청을 할 수 있는 법적 근거를 얻게 된다. 의회는 호스팅 업체 변경 결의를 통과시켰지만 그 시한이 2028년이고, 인수 완료는 그보다 훨씬 앞이다. 정부가 이미 승인한 인수를 재고할 수 없다는 입장을 고수하는 상황에서, 코드 플랫폼 독립화는 “할 수 있는 것부터 한다”는 신호이기도 하다.

### Common Ground: 342개 지자체의 선행 협력

jibbirish는 네덜란드에 이미 342개 지자체 협력 오픈소스 생태계 Common Ground(commonground.nl)가 존재한다고 소개했다.[^jibbirish] 모든 지자체가 동일한 소프트웨어를 3-4개 벤더에서 구매하는 현 구조를 오픈소스로 대체하는 것이 목표다. 이 선행 경험이 `code.overheid.nl`의 정치적·기술적 토대 역할을 했을 것이다. Nemo_bis는 Standard for Public Code와 Digital Public Goods Network처럼 국가 단위를 넘는 국제 조정 프레임워크도 병행 성장 중이라고 지적했다.[^nemo_bis]

### Microsoft 의존도의 현실

oever는 mxmap.nl을 링크하며 네덜란드 지자체의 대부분이 Microsoft 클라우드에 의존하고 있음을 시각화했다.[^oever] zoobab은 네덜란드 지자체 60%, 벨기에 코뮌 72%가 Microsoft 365를 사용 중이라는 수치를 제시했다. embedding-shape는 반대로 네덜란드가 유럽 정부 중 FOSS 도입에서 선두권이라고 반박했는데, 이 두 주장이 동시에 참일 수 있다 — “선두권”이지만 여전히 Microsoft 의존도가 높다는 것이 FOSS 전환의 어려움을 보여준다.

## 비평

### 강점: 타이밍과 정치적 맥락의 결합

`code.overheid.nl` 론칭이 주목받는 이유는 단순한 기술 선택이 아니라 지정학적 긴장과 맞물렸기 때문이다. US-EU 관계 불확실성, CLOUD Act 우려, Kyndryl 인수 논쟁이 동시에 일어나는 시점에 “유럽산 오픈소스 Git 플랫폼”을 선택한 것은 기술적 결정이면서 정치적 신호이기도 하다. “소프트 런칭”이라는 겸손한 타이밍이 오히려 거버넌스의 진정성을 보여준다.

### 약점: 진단과 처방의 규모 불균형

플랫폼 코드 저장소를 GitHub에서 이전하는 것은 상대적으로 쉽다. 반면 DigiD 인증 시스템, 지자체 클라우드 인프라, 공공 이메일 시스템 같은 핵심 의존성은 플랫폼 하나로 해결되지 않는다. ivolimmen처럼 실제 정부 기관에서 오픈소스를 오래 옹호해온 현장 인력조차 “내 목소리가 통한 적이 없었다”고 토로했다.[^ivolimmen] 코드 플랫폼 독립화가 의미 있는 첫걸음이지만, 60%의 지자체가 Microsoft 365에 의존하는 현실 앞에서는 규모가 맞지 않는다.

### 약점: “규칙 엔진” 잠재력이 과소평가됐다

HN 토론에서 regelrecht 프로젝트가 흥미로운 방향으로 논의됐다. YAML 기반 복잡한 정부 규정 계산 엔진으로, vasco는 이를 확장하면 법 개정 시 영향을 받는 하위 규정을 자동으로 탐지하는 “입법 의존성 그래프”가 될 수 있다고 제안했다. 이런 “정부 로직을 코드로” 방향이 code.overheid.nl의 진짜 잠재력일 수 있지만, 현재 플랫폼 소개에서는 일반 코드 호스팅에 초점이 맞춰져 있다.

## 인사이트

### 디지털 주권은 인프라 계층별로 별도의 전략이 필요하다

`code.overheid.nl`이 보여주는 것은 “디지털 주권”이라는 개념이 단일한 목표가 아니라 계층별로 분리된 문제라는 것이다. 코드 저장소 호스팅, 시민 인증 시스템, 이메일 인프라, 클라우드 컴퓨팅은 각각 다른 수준의 기술적 복잡성, 전환 비용, 정치적 의지를 필요로 한다. 코드 플랫폼은 전환 비용이 낮고 가시성이 높으며 개발자 커뮤니티의 지지를 받기 쉽다. 반면 DigiD 같은 인증 인프라는 시민 수천만 명이 직접 영향을 받고, 전환 실패 시 정치적 대가가 크며, 단기간 내 교체가 사실상 불가능하다.

유럽 각국 정부가 디지털 주권을 회복하려면 “어디서부터 시작할 수 있는가”를 계층 단위로 분석하고, 각 계층에 맞는 타임라인과 기술 선택을 해야 한다. 독일의 opencode.de, 네덜란드의 code.overheid.nl, 노르웨이의 정부 오픈소스 저장소가 거의 동시에 등장하는 것은 유럽 전역에서 “코드 저장소 계층”부터 시작한다는 암묵적 합의가 형성됐음을 시사한다. 이 계층에서 성공 사례와 운영 노하우를 축적한 뒤 더 어려운 계층으로 이동하는 전략이다. 단, 코드 플랫폼 성공이 더 어려운 계층의 전환도 자동으로 이끌 것이라는 기대는 금물이다 — 각 계층은 독립적인 정치 싸움을 필요로 한다.

### “공공 코드”는 새로운 소프트웨어 분배 모델이다

342개 지자체가 같은 소프트웨어를 각자 구매하는 현 구조는 소프트웨어 산업의 기형적 시장 실패다. 동일한 시민 서비스 로직이 수백 번 중복 구매되고, 각 지자체는 같은 벤더에 종속되며, 공개 조달 예산이 민간 기업에 집중된다. Common Ground와 code.overheid.nl이 지향하는 “공공 코드(Public Code)” 모델은 이 구조를 역전시킨다. 소프트웨어가 한 번 개발되면 모든 지자체가 공유하고, 개선사항이 다시 공유 코드베이스로 흘러들어오는 선순환이다.

이 모델이 성숙하면 “정부 소프트웨어 공급망”의 개념 자체가 바뀐다. 현재는 정부가 벤더에서 완성품을 구매하지만, 미래에는 여러 정부기관이 공동으로 소프트웨어를 발주하고 소유하게 된다. 세금이 소프트웨어 비용으로 나가는 것은 동일하지만, 그 비용이 반복 구매가 아닌 공유 자산 개발로 방향을 바꾼다. Standard for Public Code가 이 모델의 조달 기준을 정의하려는 시도고, Nemo_bis가 언급한 Digital Public Goods Network가 글로벌 조정 레이어를 제공하려는 시도다.

이 방향이 성공하려면 기술 선택보다 거버넌스 설계가 더 중요하다. 누가 코드의 방향을 결정하는가, 기여한 지자체와 기여하지 않은 지자체의 접근권이 같은가, 보안 패치의 책임 소재는 어디인가 — 이 질문들에 명확한 답이 없으면 공유 코드베이스는 유지관리되지 않는 쓰레기 창고가 된다.

### GitHub 의존은 공급망 리스크이자 데이터 주권 문제다

andai의 “정부가 OS와 앱 배포 채널까지 통제해야 한다”는 주장은 극단적으로 들리지만,[^andai] 그 방향의 더 완화된 버전이 현실화되고 있다. GitHub가 Microsoft 소유이고 Microsoft가 미국 기업이며 CLOUD Act가 적용된다는 사실은, GitHub에 코드를 올리는 행위가 단순한 버전 관리 선택이 아니라 데이터 관할권 선택이라는 것을 의미한다.

공공 조달 코드, 시민 서비스 로직, 정부 내부 도구의 소스가 외국 기업 서버에 존재한다는 것이 정보 보안상 문제인지는 콘텐츠에 따라 다르다. 그러나 정부가 개발하는 코드의 커밋 이력, 기여자 정보, 이슈 트래킹 내용이 외국 기업의 서비스 이용약관에 종속된다는 것 자체가 거버넌스 문제다. 코드의 저작권이 정부에 있어도 메타데이터와 협업 기록의 통제권은 플랫폼 업체에 있다.

이 논리는 단순히 “외국 기업이 나쁘다”가 아니라 “중요한 공공 인프라가 외부 단일 실패 지점에 의존해서는 안 된다”는 리스크 관리 원칙이다. regexorcist가 “GitHub 신뢰성 문제”를 언급한 것처럼,[^regexorcist] GitHub의 기술적 가용성조차 100%가 아니다. 공개 코드 플랫폼을 자체 운영하면 가용성, 접근 정책, 데이터 보존 방식을 직접 통제할 수 있게 된다.

### 소프트 런칭의 철학: 완성보다 공개가 먼저다

miih0w의 댓글이 이 시도의 정신을 가장 잘 압축했다: “대부분의 정부는 아직 토론 중인데, 네덜란드는 그냥 출시하고 떠났다.”[^miih0w] 정부 IT 프로젝트가 완벽한 기획과 전체 기능 세트를 갖추고 공식 출시하는 전통적 접근과 달리, 이번 론칭은 “아직 모든 기관에 열리지 않은 파일럿”임을 명시하고 개발자 커뮤니티의 참여를 공개적으로 요청했다.

이 “소프트 런칭” 철학은 오픈소스 개발 방법론을 정부 IT 거버넌스에 직접 적용한 것이다. 소프트웨어를 완성하고 공개하는 것이 아니라, 공개 과정 자체를 개발의 일부로 만든다. HN에서 sam_lowry_ 같은 사용자가 저장소 내용이 빈약하다고 비판한 것처럼 초기 인상은 부족할 수 있지만, 이 비판 자체가 공개 개발 과정의 피드백 루프다. 닫힌 정부 IT 프로젝트였다면 이 비판은 외부에 노출되지 않았을 것이고, 내부 개선 기회도 훨씬 느렸을 것이다.
