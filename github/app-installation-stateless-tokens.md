# GitHub App 설치 토큰의 Stateless 전환과 임시 오버라이드 헤더

원문: <https://github.blog/changelog/2026-05-15-github-app-installation-tokens-per-request-override-header/>

## 요약

GitHub은 2026년 5월 15일에 GitHub App 설치 토큰의 새 포맷을 사전 검증할 수 있는 임시 요청 헤더 `X-GitHub-Stateless-S2S-Token`을 공개했다.
이 헤더는 `POST /app/installations/:installation_id/access_tokens` 엔드포인트에서 요청 단위로 토큰 포맷을 강제하는 용도다.

헤더는 세 가지 값을 받는다.
`enabled`는 `ghs_` 접두사를 가진 JWT 형식의 약 520자짜리 stateless 토큰을 강제로 반환한다.
`disabled`는 기존의 짧은 불투명(opaque) 문자열 형태의 stateful 토큰을 반환한다.
값이 비어 있거나 다른 값일 경우 GitHub이 진행 중인 일반 롤아웃 정책을 따른다.

새 stateless 토큰은 `ghs_` 접두사 뒤에 점(`.`) 두 개가 포함된 JWT 구조다.
반면 기존 stateful 토큰은 점이 없는 짧은 문자열이었다.
GitHub은 개발자가 다음을 검증하라고 권한다.
하드코딩된 토큰 길이 제약이 없는지, 정규식이 새 포맷을 수용하는지(권장 `ghs_[A-Za-z0-9\.\-_]{36,}`), 데이터베이스 컬럼이 520자 이상을 저장할 수 있는지, 그리고 토큰을 그 자체로 opaque 문자열로만 다루는지를 확인해야 한다.

이 헤더는 임시이며 향후 사라진다.
적용 범위는 GitHub Enterprise Cloud와 Data Residency 환경이며 GitHub Enterprise Server는 포함되지 않는다.
영향을 받는 토큰은 GitHub App 설치 server-to-server 토큰이며, GitHub Actions의 `GITHUB_TOKEN`도 포함된다.

## 분석

### 헤더의 본질: 마이그레이션을 위한 임시 토글

이 글의 핵심 메시지는 새 토큰 포맷을 알리는 것이 아니다.
새 포맷의 롤아웃은 이미 진행 중이며, 헤더는 그 롤아웃이 자기 시스템에 도달하기 전에 클라이언트가 미리 그 포맷을 강제로 받아 검증할 수 있게 하는 임시 토글이다.
GitHub은 이 헤더를 “deprecated 예정”이라고 명시함으로써 헤더 자체에 의존하지 말라고 신호를 보낸다.

이런 형태의 헤더는 클라이언트가 자신의 통제 아래 “미래 시점”을 한 번 모의 실험할 수 있게 한다.
플랫폼 사업자 입장에서는 강제 롤아웃 전에 거대한 통합 생태계가 새 포맷을 받아들일 수 있는지를 분산된 책임으로 검증받는다.
이 패턴은 점진적 롤아웃의 마지막 안전망에 해당하며, 단순한 feature flag와는 방향이 다르다 — feature flag는 새 동작을 “꺼두는” 도구이고, 이 헤더는 새 동작을 “미리 켜는” 도구다.

### Opaque에서 JWT로: 토큰이 더 이상 “식별자”가 아니다

기존 GitHub App 설치 토큰은 짧은 opaque 문자열이었다.
opaque 토큰은 클라이언트가 그 내용을 해석할 수 없고, 검증은 발급자(서버)만 가능했다.
이 모델에서는 모든 검증 요청이 GitHub의 토큰 저장소(stateful)로 한 번 더 돌아와야 한다.

새 stateless 토큰은 JWT다.
JWT는 그 자체가 서명된 클레임의 묶음이라, 서버가 데이터베이스를 조회하지 않고도 검증할 수 있다.
이 구조 변화의 진짜 함의는 “토큰이 더 이상 단순한 식별자가 아니다”라는 점이다.
토큰이 데이터가 되면서, 토큰 발급/회수의 의미론도 바뀐다.
opaque 토큰을 revoke한다는 것은 “DB에서 삭제한다”였지만, JWT를 revoke한다는 것은 “블록리스트나 짧은 TTL로 우회한다”가 된다.

### 변화의 비용을 클라이언트에 분산시키는 설계

GitHub이 클라이언트에게 요구하는 검증 항목은 모두 “opaque 토큰을 가정한 코드가 깨질 수 있는 지점”이다.
하드코딩된 길이, 좁은 정규식, 길이가 부족한 DB 컬럼, 토큰을 파싱하거나 일부를 추출하려는 시도 — 이 네 가지는 흔한 안티패턴이지만, 그동안 GitHub이 opaque 짧은 토큰을 고수해 왔기에 노출되지 않았던 잠재 결함이다.

이 결함은 GitHub만이 비용을 부담해서는 해소할 수 없다.
오직 각 클라이언트가 자기 코드를 점검하고 수정해야만 해소된다.
헤더는 이 “분산된 비용”을 클라이언트가 자기 시간에 자기 손으로 지불할 수 있게 만드는 장치다.
GitHub이 강제 롤아웃을 일방적으로 하지 않고 헤더라는 우회로를 둔 것은 책임의 분산이라기보다 비용의 분산이다.

### Enterprise Server를 제외한다는 결정의 함의

이번 변경의 적용 범위에서 GitHub Enterprise Server가 명시적으로 빠졌다.
이는 단순한 기능 차이가 아니라 토큰 인프라의 분기가 시작됨을 의미한다.
Cloud와 Data Residency는 JWT 기반 stateless 검증으로 가고, 자가 호스팅 Enterprise Server는 stateful 모델을 유지한다.

이 분기는 향후 GitHub의 멀티-제품 전략과 연결된다.
Cloud는 더 가벼운 검증 비용과 더 빠른 글로벌 분산을 추구할 수 있고, Server는 자가 호스팅 환경의 통제력을 유지한다.
그러나 두 환경 모두에서 동작하는 GitHub App 개발자는 이제 두 토큰 모델을 동시에 지원해야 한다.
이 비용은 글에 명시적으로 등장하지 않지만 통합 도구 생태계에 점진적으로 누적된다.

## 비평

### 글은 “왜 지금 stateless로 가는가”를 설명하지 않는다

changelog 글의 한계는 발표 형식이라는 점에 있지만, 이 변화의 동기를 한 줄도 다루지 않는 것은 의도된 누락이다.
opaque에서 JWT로의 전환은 검증 인프라 비용, 글로벌 분산, 토큰 발급 처리량과 직접 연결된 결정이다.
이 동기를 말하지 않으면 클라이언트는 “왜 우리가 코드를 고쳐야 하는가”에 대한 답 없이 작업을 떠안게 된다.

특히 JWT 전환은 보안 트레이드오프를 동반한다.
JWT는 stateless라서 revoke가 어렵다.
이 문제는 짧은 TTL과 블록리스트로 완화하지만, 완화 메커니즘이 무엇이고 클라이언트가 무엇을 가정해야 하는지 명확하지 않은 채로 토큰 형태만 바뀐다.
글이 침묵하는 동안 클라이언트는 “토큰이 길어졌다” 이상의 변화 신호를 받지 못한다.

### “길이 제약을 풀라”는 권고는 근본 해법이 아니다

GitHub이 제시한 정규식 `ghs_[A-Za-z0-9\.\-_]{36,}`은 “하한 36자 이상”이다.
이는 향후 토큰이 더 길어지더라도 호환되도록 한 선택이다.
그러나 진짜 권고는 “정규식 자체를 토큰 검증에 쓰지 말라”여야 한다.

토큰을 정규식으로 검사하는 행위는 토큰을 식별자로 신뢰한다는 의미다.
opaque 토큰의 핵심 원칙은 “파싱하지 않고 발급자에게 검증을 위임한다”였다.
이 원칙이 흔들리는 이유는 클라이언트들이 토큰을 로그 마스킹, 시크릿 스캐닝 회피, 형식 검증 등 다양한 곳에서 파싱해 왔기 때문이다.
GitHub의 권고는 이 잘못된 관습을 정정하기보다 “이 잘못된 관습이 깨지지 않게 정규식을 느슨하게 잡아라”라고 말한다.
이것은 마이그레이션의 매끄러움을 위한 타협이지만, 토큰을 데이터로 다루지 말라는 원칙을 약화시킨다.

### 헤더의 임시성이 새 부채를 만든다

“이 헤더는 임시이며 향후 deprecated”라는 한 줄은 분명한 신호처럼 보이지만, 실제로는 새로운 운영 부채의 출발점이다.
헤더는 임시지만, 그 헤더를 사용하는 검증 스크립트, CI 파이프라인, 통합 테스트는 코드로 남는다.
조직은 헤더 deprecation 시점까지 이 코드를 추적하고 정리해야 한다.

이 부채는 [Verification Debt](../agentic-coding/verification-debt.md)의 또 다른 형태에 가깝다.
사전 검증을 위한 임시 도구가 자체로 유지보수 대상이 되면서, 한 번의 마이그레이션이 두 번의 청소 작업으로 늘어난다.
GitHub은 헤더 도입 시점만 알릴 뿐 deprecation 일정을 함께 제시하지 않았다.
사용자는 자기 시간을 들여 검증한 뒤 그 시간을 다시 들여 그 검증 흔적을 지워야 한다.

## 인사이트

### Stateless 토큰은 “토큰의 의미”를 분산 시스템 쪽으로 재정의한다

opaque 토큰의 시대에 토큰은 식별자였다.
서버는 토큰을 받고 그 토큰이 누구의 것인지를 자기 저장소에서 조회했다.
이 모델은 단순했지만, 토큰 검증이 항상 발급자에게 왕복해야 했고 발급자가 단일 장애점이 되었다.

JWT로 가는 순간 토큰은 자기 자신을 증명하는 데이터가 된다.
이 변화는 인증을 “저장소 조회”에서 “서명 검증”으로 옮긴다.
이것은 단순한 성능 최적화가 아니라 분산 시스템 설계 원칙의 적용이다 — 상태를 메시지 안에 넣고, 검증 비용을 호출자에게 분산시키는 것.
같은 원칙이 [Harness Engineering](../agentic-coding/harness-engineering.md)이 말하는 “상태를 외부화하라”와 같은 방향의 사고이며, 분산 시스템이 마주한 동일한 트레이드오프(혁신성 vs revocation 복잡성)가 GitHub 인증 인프라에서도 재현된다.

이 재정의는 시크릿 스캐너, 로그 redaction 도구, 토큰 회수 정책 등 토큰 주변 생태계 전체를 흔든다.
점(`.`)이 포함된 520자 문자열이 새로운 “GitHub 시크릿의 모양”이 되면, 이를 매칭하는 모든 도구가 같은 방향으로 함께 이동해야 한다.

### 임시 오버라이드 헤더는 “테스트 인 프로덕션” 패턴의 한 사례다

이 헤더가 흥미로운 이유는 staging 환경이 아니라 프로덕션에서 클라이언트가 직접 검증하게 만들기 때문이다.
GitHub은 “테스트 인 프로덕션(testing in production)” 패턴을 정중한 형태로 강제한 셈이다 — 새 토큰 포맷이 실제 API 응답에서 들어오게 한 뒤, 클라이언트가 자기 코드 경로를 따라가며 발견하는 것.

이 패턴이 작동하려면 클라이언트는 두 가지 가정을 만족해야 한다.
하나, 헤더를 사용한 요청이 자신의 운영 흐름에 영향을 주지 않는다는 점.
둘, 검증 결과(어떤 코드가 깨지는지)를 충분히 빨리 발견하고 수정할 수 있다는 점.
둘 다 클라이언트의 관측성과 배포 속도에 의존하며, 둘 중 하나가 부족하면 헤더는 검증 도구가 아니라 사고 유발 장치가 된다.

이 사례는 플랫폼-클라이언트 관계에서 “마이그레이션 책임의 분배”가 점차 클라이언트 쪽으로 이동하는 흐름을 보여준다.
플랫폼은 더 이상 “모든 클라이언트가 자동으로 안전하게 이동하도록” 책임지지 않는다.
대신 검증 도구를 제공하고, 검증의 실행 책임은 클라이언트에게 위임한다.
이것은 효율적이지만, 비용은 보이지 않게 클라이언트 쪽 운영 예산으로 옮겨간다.

### Enterprise 분기는 통합 생태계의 “두 GitHub” 문제를 강화한다

GitHub은 이미 Cloud와 Enterprise Server가 기능 격차를 갖는 “두 GitHub” 상태에 있다.
이번 토큰 인프라 분기는 이 격차를 단순한 기능 차이에서 인증 모델 차이로 깊게 만든다.
GitHub App이 두 환경 모두에서 동작하려면 이제 토큰 길이, 토큰 형태, 검증 방식, 회수 의미론까지 두 가지 가정 아래에서 코드를 유지해야 한다.

이 차이는 시간이 지날수록 벌어진다.
Cloud는 stateless 모델 위에서 새로운 최적화와 기능을 쌓을 수 있고, Server는 그 최적화 일부를 받지 못한 채 stateful 모델에 남는다.
결과적으로 두 환경을 모두 지원하는 통합 개발자는 “Cloud-only 기능”과 “양쪽 호환 기능”을 명시적으로 구분하는 새 의사결정 축을 떠안는다.

이 분기 효과는 GitHub만의 문제가 아니다.
플랫폼이 자가 호스팅과 SaaS 두 형태를 동시에 운영할 때 마주하는 구조적 긴장이며, 토큰 인프라처럼 코어에 가까운 컴포넌트가 갈라지면 두 제품의 “브랜드는 같지만 본체는 다름” 상태가 깊어진다.
이 글은 단일 변경처럼 보이지만, 장기적으로는 GitHub의 제품 전략이 다시 한 번 “Cloud-first”로 기우는 신호로 읽을 수 있다.
