# Copilot Chat에 자동 모델 선택 모드가 전체 공개되었다

원문: <https://github.blog/changelog/2026-06-17-auto-mode-in-copilot-chat-available-for-all-users/>

## 요약

GitHub이 Copilot Chat의 자동 모델 선택(auto mode)을 github.com과 모바일 앱에서
모든 Copilot 플랜 사용자에게 일반 공개(Generally Available)했다.
이 기능은 사용자의 요청 복잡도와 현재 시스템 가용성을 평가해
Copilot이 가장 적합한 AI 모델을 동적으로 선택하는 방식으로 작동한다.

자동 선택의 라우팅 대상 모델은 Claude Sonnet 4.6, GPT-5.4 mini, GPT-5.4, Haiku 4.5다.
어떤 모델이 선택 가능한지는 실시간 시스템 상태와 작업 복잡도 평가에 따라 달라진다.
사용자는 응답 위에 마우스를 올려 어느 모델이 자신의 요청을 처리했는지 확인할 수 있으며,
응답별로 자동 모드와 특정 모델 간 전환도 가능하다.
사용자와 관리자가 설정한 모든 모델 정책은 자동 선택 과정에서도 그대로 적용된다.

요금 측면에서 유료 구독자는 자동 모드 사용 시 10% 할인이 적용된다.
이번 발표와 동시에 Opus 4.6 (fast) 버전의 deprecated 조치도 함께 공지되었다.

## 분석

### “복잡도 기반 라우팅”이 실제로 의미하는 것은 사용자가 알 수 없다

자동 모드가 “요청 복잡도”를 기준으로 모델을 선택한다는 주장은 직관적으로 들리지만,
“복잡도”의 구체적 정의가 없다.
코드 줄 수, 요청 문장 수, 언어 다양성, 추론 단계 수 중 어느 것이 복잡도를 결정하는지는 공개되지 않았다.
“hover to see which model processed your request”라는 기능은 결과를 보여주지만,
왜 그 모델이 선택되었는지는 알 수 없다.

이것은 사용자에게 결과를 보여주면서 의사결정 과정을 블랙박스로 유지하는 구조다.
같은 요청이 다른 시점에 다른 모델로 라우팅될 수 있고 —
“시스템 가용성”이 두 번째 기준으로 명시되어 있다 —
이 경우 사용자는 같은 입력에 대해 다른 품질의 출력을 받을 수 있다.
자동 모드를 선택하는 것은 모델 품질의 일관성을 시스템 가용성과 맞바꾸는 거래다.

### 10% 할인은 비용 최적화가 아니라 행동 유도 장치다

유료 구독자에게 자동 모드 사용 시 10% 할인을 제공하는 것은 표면적으로 사용자 혜택이지만,
그 구조를 보면 GitHub 입장에서의 이점이 더 명확하다.
자동 모드에서 Copilot은 비용이 낮은 모델(Haiku 4.5, GPT-5.4 mini)로 요청을 라우팅할 수 있다.
사용자가 GPT-5.4나 Claude Sonnet 4.6 같은 고비용 모델을 특정해 사용하는 것보다
자동 모드에서 경량 모델로 처리되는 비율이 높아지면
GitHub의 추론 비용은 10% 할인을 제공하더라도 전체적으로 낮아질 수 있다.

이 구조는 항공사 좌석 업그레이드 할인과 유사하다.
특정 좌석을 요청하는 것보다 “최적 좌석 자동 배정”을 선택하면 할인을 받지만,
항공사는 그 선택이 전체 좌석 최적화에 유리하다는 것을 알고 있다.
10% 할인이 사용자에게 실질적 가치인지는 자동 선택 결과로 얼마나 자주 경량 모델이 배정되는지에 달려 있으며,
그 비율은 공개되지 않는다.

### Opus 4.6 (fast) deprecated와 자동 모드 GA의 동시 발표가 보내는 신호

Opus 4.6 fast 버전의 deprecated와 자동 모드 GA가 같은 날 발표된 것은 우연이 아닐 가능성이 높다.
Opus 4.6 fast는 고성능 모델을 더 빠르게 제공하려는 시도였지만,
더 나은 접근은 “작업에 맞는 모델을 골라 빠르게 제공”이라는 판단이
이번 자동 모드 방향성에 내재되어 있다.

이것은 단일 고성능 모델의 최적화에서 모델 앙상블 라우팅으로의 전략 전환을 보여준다.
Opus 4.6 fast의 deprecated는 “빠른 고성능 모델 하나”라는 접근을 포기하고,
“상황에 맞는 모델을 선택하는 시스템”으로 이동한다는 선언이다.
자동 모드가 이 전환의 사용자 인터페이스가 된다.

## 비평

### 라우팅 기준이 불투명한 시스템에서 “모든 모델 정책이 적용된다”는 주장은 검증 불가능하다

체인지로그는 “사용자와 관리자가 설정한 모든 모델 정책이 자동 선택 과정에서도 적용된다”고 밝히지만,
라우팅 알고리즘이 공개되지 않은 상태에서 이 주장을 외부에서 검증할 방법이 없다.
특정 모델을 사용하지 않도록 정책을 설정한 조직에서
해당 모델이 자동 모드로 선택되지 않는다는 것을 어떻게 확인하는가?

응답 위에 마우스를 올려 모델을 확인하는 것은 사후 확인이다.
엔터프라이즈 보안 정책에서 “어떤 모델이 사용되었는가”보다 중요한 것은
“금지된 모델이 절대 사용되지 않는다는 보장”이다.
정책 적용을 사용자가 각 응답을 수동으로 확인해야만 감시할 수 있는 구조라면,
이것은 기업 AI 거버넌스 요건을 충족하지 못한다.
자동 모드 사용 시 어떤 모델이 얼마나 자주 선택되었는지에 대한 감사 로그가 제공되는지
체인지로그는 설명하지 않는다.

### 모델 선택의 일관성 상실이 가져오는 재현성 문제를 다루지 않는다

자동 모드에서 같은 입력이 항상 같은 모델로 라우팅된다는 보장이 없다.
시스템 가용성이 두 번째 라우팅 기준이라면,
오전 9시에 한 질문이 Claude Sonnet 4.6으로 처리되고
오후 3시에 같은 질문이 Haiku 4.5로 처리될 수 있다.
이 경우 두 응답의 품질이 다르더라도 사용자는 그 차이가 모델 때문인지 다른 이유 때문인지 알기 어렵다.

소프트웨어 개발 컨텍스트에서 이 비일관성은 중요한 문제가 된다.
팀이 특정 코딩 작업에 Copilot을 사용할 때 어제의 결과와 오늘의 결과가 다른 품질을 보인다면,
그 차이가 자동 라우팅의 결과인지 다른 요인인지 추적하기 어렵다.
재현성이 중요한 작업 — 코드 리뷰, 보안 취약점 분석 — 에서
자동 모드의 동적 특성은 일관된 결과를 기대하는 사용자에게 숨겨진 위험이다.

### “응답별로 모델을 전환할 수 있다”는 기능은 실제 사용에서 작동하지 않는 선택지다

자동 모드 사용 중 특정 응답에 대해 모델을 전환할 수 있다는 기능은
이론적으로 사용자 통제권을 보장하는 것처럼 보이지만, 실제 사용 패턴에서는 무의미하다.
사용자가 모델을 전환하려면 먼저 자동 선택 결과를 받아보고, 불만족스러운 경우 다시 요청해야 한다.
그런데 “어떤 응답이 불만족스러운가”를 판단하려면 결과를 읽고 평가해야 한다.
결과를 읽고 평가했다면 이미 그 응답에 시간을 소비한 것이다.

더 근본적인 문제는 사용자 대부분이 어떤 모델이 어떤 작업에 더 적합한지 알지 못한다는 점이다.
Claude Sonnet 4.6이 Haiku 4.5보다 특정 코딩 작업에 더 나은 이유를 모르는 사용자가
“이 응답을 다른 모델로 재시도”를 선택할 근거가 없다.
모델 전환 선택지는 그 선택을 의미 있게 만드는 지식을 사용자가 갖추고 있다고 가정하지만,
자동 모드를 선택하는 사용자 대부분은 그 판단을 시스템에 위임하기 위해 자동을 선택한 것이다.

## 인사이트

### 자동 모드는 모델 선택의 부담을 사용자에서 시스템으로 이전하면서 새로운 형태의 불확실성을 만든다

AI 도구의 초기 단계에서 “어느 모델을 써야 하는가”는 사용자가 스스로 배워야 했던 전문 지식이었다.
GPT-4와 GPT-3.5의 차이, Claude Opus와 Claude Haiku의 적합한 사용 사례를 아는 것이
숙련된 AI 도구 사용자의 특징이었다.
자동 모드는 이 지식의 필요성을 제거한다 — 시스템이 대신 결정하므로 사용자는 알 필요가 없다.

그러나 이 이전은 또 다른 유형의 불확실성을 도입한다.
기존에는 사용자가 모델을 선택했으므로 결과의 품질이 예측 가능했다 — 같은 모델을 쓰면 유사한 성능을 기대할 수 있었다.
자동 모드에서는 결과의 품질이 시스템의 라우팅 결정에 달려 있고,
그 결정이 언제 어떻게 달라질지 사용자는 알 수 없다.
“모델을 몰라도 된다”는 단순화와 “결과가 왜 달라졌는지 알 수 없다”는 불투명성이 함께 온다.

이 이전은 자동변속기의 보급과 구조적으로 같다.
운전자가 기어를 선택하지 않아도 되는 편의성을 얻지만,
차가 왜 특정 기어를 선택했는지는 알 수 없고, 그 결정이 최적인지도 판단하기 어려워진다.
AI 도구에서도 같은 거래가 일어나고 있다: 결정의 편의성 대신 결과의 예측 가능성.

### 다중 모델 라우팅 시스템은 AI 도구를 단일 모델 제품에서 인프라 레이어로 전환한다

Copilot Chat 자동 모드의 모델 목록에 Claude Sonnet 4.6(Anthropic), GPT-5.4(OpenAI), Haiku 4.5(Anthropic)가 함께 있다는 것은
GitHub이 특정 AI 공급업체에 종속되지 않는 라우팅 레이어를 구축하고 있다는 신호다.
단일 모델 제품에서 여러 모델을 오케스트레이션하는 레이어로 이동함으로써,
GitHub은 어떤 모델이 가장 좋은지의 경쟁에서 분리되어
어떤 모델이든 최적으로 라우팅하는 능력의 경쟁으로 이동한다.

역사적 유사점은 클라우드 CDN의 발전이다.
초기 웹 호스팅은 단일 서버에 연결되었지만,
CDN은 요청을 가장 가까운 서버나 가장 덜 바쁜 서버로 라우팅해 성능을 최적화했다.
사용자는 어느 서버에서 콘텐츠를 받는지 알 필요가 없었다.
Copilot 자동 모드는 AI 추론에서 같은 레이어를 구축한다:
요청을 가장 적합한 모델로 라우팅하고, 사용자는 모델의 존재를 추상화 계층 아래로 숨긴다.

이 전환이 완성되면 경쟁의 단위가 바뀐다.
“어느 모델이 더 강력한가”에서 “어느 플랫폼이 더 잘 라우팅하는가”로.
OpenAI, Anthropic, Google이 모델 품질로 경쟁하는 동안,
GitHub은 그 위에서 라우팅 품질로 경쟁하는 다른 레이어에 자리를 잡고 있다.

### 자동 모드의 보급은 “어느 AI를 써야 하는가”라는 개발자 역량 자체를 역사적 지식으로 만들 것이다

2023년부터 2025년까지 AI 도구를 잘 활용하는 개발자의 특징 중 하나는
작업 유형별로 적합한 모델을 아는 것이었다.
긴 코드 분석에는 컨텍스트 창이 큰 모델, 빠른 자동완성에는 경량 모델,
복잡한 추론에는 대형 모델을 선택하는 감각이 전문성으로 인정받았다.

자동 모드가 이 결정을 시스템 레이어로 흡수하면,
“모델을 잘 선택하는 능력”은 점차 불필요한 지식이 된다.
마치 어셈블리 최적화가 컴파일러의 발전으로 대부분의 개발자에게 불필요해진 것처럼,
“모델 선택 감각”은 라우팅 시스템의 발전으로 일반 개발자가 배울 필요 없는 영역이 된다.
이 지식의 소멸은 AI 도구 접근성을 높이지만,
동시에 시스템의 라우팅 결정을 비판적으로 평가할 수 있는 사용자의 비율을 낮춘다.

이것은 이차적 위험을 만든다.
자동 모드가 잘못된 라우팅을 할 때 — 복잡한 보안 분석을 경량 모델로 처리하거나,
단순한 오탈자 수정을 불필요하게 고비용 모델로 처리할 때 —
이를 감지하고 문제를 제기할 수 있는 사용자가 줄어든다.
시스템을 신뢰하는 사용자 집단이 커질수록 시스템의 실패를 감지하는 외부 감시자도 줄어드는 구조다.
자동화는 편의성과 감시 가능성 사이의 이 긴장을 항상 내포하지만,
AI 추론 라우팅에서 이 긴장은 특히 중요하다 —
라우팅의 실패가 코드 품질, 보안 분석, 의사결정 지원의 품질에 직접 영향을 주기 때문이다.
