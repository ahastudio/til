# GitHub Agentic Workflows, PAT 없이 GITHUB_TOKEN으로 실행

원문: <https://github.blog/changelog/2026-06-11-agentic-workflows-no-longer-need-a-personal-access-token/>

## 요약

GitHub Agentic Workflows가 GitHub Actions 내장 토큰인 `GITHUB_TOKEN`을 지원하게 됐다.
이전까지는 에이전틱 워크플로우를 실행하려면 개인 액세스 토큰(PAT, Personal Access Token)이 필요했으나,
이번 변경으로 PAT 없이도 워크플로우를 구성하고 실행할 수 있다.

조직 소유 저장소에서 Actions 토큰을 사용해 에이전틱 워크플로우가 실행되면,
AI 크레딧 비용이 개인 사용자가 아닌 조직에 직접 청구된다.
이를 활성화하려면 조직에서 “Allow use of Copilot CLI billed to the organization” 정책을 켜야 하며,
기존에 “Copilot CLI” 정책이 활성화되어 있었다면 자동으로 켜진 상태다.

사용자 측 설정 변경도 필요하다.
워크플로우 마크다운 파일의 프론트매터 `permissions` 섹션에 `copilot-requests: write`를 추가하고,
업데이트된 lockfile을 컴파일해 푸시해야 한다.
CLI도 최신 버전으로 업그레이드(`gh extension upgrade aw`)해야 이 기능을 사용할 수 있다.

비용 관리 방식도 달라진다.
조직 청구 방식에서는 사용자 레벨 인퍼런스(inference) 예산이 적용되지 않는다.
대신 해당 조직에 cost center를 구성하고, 워크플로우 실행당 토큰 사용량을 모니터링하거나 상한을 설정하는
GitHub Agentic Workflows의 비용 관리 도구를 활용해야 한다.

이 기능은 Free, Pro, Pro+, Business, Enterprise 등 모든 Copilot 플랜에서 이용 가능하다.

## 분석

### PAT 제거가 의미하는 보안 모델의 전환

PAT는 오랫동안 GitHub 자동화의 주된 인증 수단이었지만, 구조적인 보안 약점을 안고 있었다.
PAT는 발급한 개인 계정에 종속되며, 권한 범위가 필요 이상으로 넓어지기 쉽고, 유효 기간이 길어질수록 노출 위험이 누적된다.
에이전틱 워크플로우처럼 자동화된 맥락에서 PAT를 사용하면, 특정 직원의 계정 상태(퇴직, 권한 변경)에 워크플로우의 실행 가능 여부가 의존하게 된다.

`GITHUB_TOKEN`은 GitHub Actions 잡(job)이 실행될 때마다 자동으로 발급되고 잡 종료 시 만료되는 단기 토큰이다.
발급과 폐기가 자동화되어 있어 토큰 수명 관리 부담이 없고, 권한 범위도 워크플로우 파일에 명시적으로 선언된다.
`copilot-requests: write`처럼 필요한 권한만 선언하는 방식은 최소 권한 원칙(principle of least privilege)과 일치한다.

이 변화는 단순한 편의 개선이 아니라 보안 책임의 재배치다.
PAT 방식에서는 개인이 토큰을 생성하고 관리하는 책임을 졌지만,
`GITHUB_TOKEN` 방식에서는 플랫폼이 토큰 생명주기를 관리하고 조직 정책이 사용을 통제한다.
책임이 개인에서 조직 인프라로 이동하는 것이다.

### 청구 단위의 이동이 드러내는 조직화 의도

이전 PAT 방식에서는 에이전틱 워크플로우가 개인 계정에 연결된 AI 크레딧을 소비했다.
이는 자동화 도구가 개인의 할당량을 잠식하거나, 반대로 개인이 조직 목적의 자동화 비용을 부담하는 불균형을 만들었다.
`GITHUB_TOKEN`과 조직 청구를 연계한 것은 이 불균형을 구조적으로 해소하는 설계다.

청구 단위가 개인에서 조직으로 이동한다는 것은, GitHub이 Agentic Workflows를 개인 생산성 도구가 아닌
조직 단위 인프라로 자리매김하고 있음을 시사한다.
비용 센터(cost center), 예산 상한, 조직 정책 설정이 함께 제공되는 구성은 엔터프라이즈 구매자를 겨냥한 설계다.

이 전환은 AI 코딩 도구 전반에서 반복되는 패턴을 따른다.
개인이 먼저 채택하고, 팀과 조직이 표준화하고, 결국 엔터프라이즈 계약으로 귀결된다.
GitHub이 청구 구조를 조직 중심으로 재편하는 것은 이 확장 경로를 의도적으로 가속하는 움직임이다.

### Changelog 포맷이 선택한 것과 숨긴 것

이 글은 GitHub Blog의 Changelog 섹션에 게시된 짧은 공지다.
변경 내용, 설정 방법, 가용 범위를 간결하게 전달하는 데 집중하며, 맥락 설명은 최소화한다.
이런 포맷은 이미 Agentic Workflows를 사용하는 독자를 전제로 쓰인다.

포맷의 선택은 무엇을 전달할지뿐 아니라 무엇을 질문할 수 있는지도 결정한다.
Changelog는 “왜 이제인가”, “이전 PAT 방식의 어떤 구체적 보안 사고가 이 결정을 앞당겼는가”,
“조직 청구와 개인 청구 사이의 전환 비용은 무엇인가” 같은 질문을 다루지 않는다.
이는 포맷의 한계가 아니라 포맷의 의도이며, 독자가 이 공지를 읽을 때 그 경계를 인식해야 한다.

## 비평

### “더 이상 필요 없다”는 표현이 실제 마이그레이션 비용을 가린다

제목은 “Agentic Workflows No Longer Need a Personal Access Token”으로, 문제가 해결된 것처럼 읽힌다.
그러나 기존에 PAT로 운영 중인 워크플로우는 자동으로 전환되지 않는다.
프론트매터 수정, lockfile 재컴파일, CLI 업그레이드, 조직 정책 활성화가 모두 필요하다.

실제 운영 환경에서 이 전환은 단순하지 않다.
저장소가 수십 개이거나, 여러 팀이 각자의 워크플로우를 관리하는 조직이라면
각 워크플로우를 개별적으로 수정하고 재검증해야 한다.
공지는 단계별 절차를 나열하지만, 기존 사용자가 감당할 마이그레이션 규모에 대해서는 침묵한다.

“더 이상 필요 없다”는 표현은 미래 신규 사용자에게는 사실이지만,
현재 운영 중인 사용자에게는 “이제 바꿔야 한다”는 의미다.
이 차이가 흐릿하게 처리된 것은 공지의 논리적 약점이다.

### 조직 청구 전환이 개인 플랜 사용자에게 갖는 의미가 불분명하다

공지는 이 기능이 모든 Copilot 플랜(Free, Pro, Pro+, Business, Enterprise)에서 이용 가능하다고 명시한다.
그러나 조직 청구 활성화는 조직 소유 저장소에서만 의미가 있다.
개인 저장소에서 `GITHUB_TOKEN`을 사용하는 경우 비용은 어디로 청구되는가?
개인 플랜 사용자가 조직 없이 이 기능을 쓸 때의 동작이 공지에서 명확히 설명되지 않는다.

“모든 플랜에서 이용 가능”이라는 표현은 기능의 가용성을 최대한 넓게 제시하는 마케팅적 언어다.
그러나 실제 활성화 경로(조직 정책 설정, 조직 청구)는 조직 플랜을 전제로 설명되어 있다.
개인 사용자에게 이 공지가 실질적으로 의미하는 바가 무엇인지는 추가 문서를 찾아야 알 수 있다.

이런 불일치는 독자가 자신의 상황에 공지를 적용할 수 있는지 판단하기 어렵게 만든다.
“가용하다”와 “유용하다”는 다른 말이며, 그 간격이 클 때 공지의 실질적 가치는 떨어진다.

### 인퍼런스 예산 정책의 변화가 갖는 숨겨진 위험을 다루지 않는다

공지는 조직 청구 방식에서는 사용자 레벨 인퍼런스 예산이 적용되지 않는다고 명시한다.
대신 cost center와 워크플로우 실행당 토큰 상한을 활용하라고 안내한다.
이 변화는 비용 통제 책임을 개인에서 조직 관리자로 이전하는 것이다.

문제는 조직 관리자가 이 새로운 책임을 인식하고 있는가다.
기존에 개인 레벨 예산이 자동 상한 역할을 해왔다면,
조직 청구로 전환하는 순간 그 상한이 사라진다.
cost center와 워크플로우 예산을 설정하지 않은 채로 전환하면 비용이 예상치 않게 급증할 수 있다.

공지는 이 위험을 “cost management tools를 활용하라”는 한 줄로 처리한다.
그러나 “도구가 있다”와 “기본값이 안전하다”는 다른 이야기다.
기본값이 무제한이라면, 마이그레이션 후 첫 달에 예상치 못한 청구서를 받는 조직이 생길 수 있다.
이 위험을 명시적으로 경고하지 않는 것은 운영 맥락에서 중요한 누락이다.

## 인사이트

### AI 워크플로우의 인증 모델은 인프라 권한 모델을 뒤따른다

소프트웨어 인프라의 역사를 돌아보면, 새로운 자동화 계층이 등장할 때마다 인증 모델이 먼저 임시방편으로 출발하고 이후 정식 통합된다.
초기 CI/CD 도구들도 처음에는 개인 계정 자격증명을 환경변수로 주입하는 방식으로 운영됐다.
이후 서비스 계정, OIDC 연동, 단기 토큰 방식으로 점진적으로 성숙해졌다.

Agentic Workflows의 PAT → `GITHUB_TOKEN` 전환은 이 패턴의 반복이다.
AI 에이전트가 인프라의 구성 요소로 자리 잡으면서, 인증도 인프라 수준의 신뢰 모델을 따르게 된다.
단기 토큰, 명시적 권한 선언, 플랫폼 관리 생명주기는 인프라 보안의 기본 원칙이다.

이 추세의 다음 단계는 예측 가능하다.
현재 `copilot-requests: write` 같은 단일 권한이지만, 에이전트가 수행할 수 있는 작업이 다양해질수록
세분화된 권한 범위(fine-grained permissions)가 필요해질 것이다.
GitHub Apps의 권한 모델처럼, 에이전틱 워크플로우에도 리소스별 읽기/쓰기 권한을 선언하는 체계가 생길 가능성이 높다.

### 조직 청구 통합은 AI 자동화의 거버넌스 문제를 수면 위로 끌어올린다

AI 도구 비용이 개인 할당량에서 소비될 때는 사용 규모가 자연스럽게 제한됐다.
개인이 감당할 수 있는 크레딧 한도 안에서 자동화가 이루어졌기 때문이다.
조직 청구로 전환되면 이 자연적 제한이 사라지고, 자동화 규모가 예산과 정책으로만 통제된다.

이것은 기술 문제이기 전에 거버넌스 문제다.
조직에서 누가 에이전틱 워크플로우 사용을 승인하는가?
어떤 워크플로우가 조직 예산을 쓸 수 있는가?
비용 초과 시 누가 책임지는가?
이 질문들은 GitHub의 공지가 다루지 않지만, 실제 조직이 도입을 결정할 때 반드시 답해야 할 것들이다.

GitHub이 cost center와 예산 상한 도구를 함께 제공하는 것은 이 거버넌스 공백을 인식하고 있다는 신호다.
그러나 도구의 존재와 도구의 사용은 다르다.
AI 자동화 거버넌스 정책을 갖추지 않은 채로 조직 청구를 활성화하는 조직은,
기술적으로는 전환했지만 운영적으로는 준비되지 않은 상태에 놓인다.

### 단기 토큰 모델은 에이전트의 신뢰 경계를 새로 정의한다

`GITHUB_TOKEN`은 실행 중인 잡(job)에 바인딩된 단기 토큰이다.
에이전트가 이 토큰으로 실행된다는 것은, 에이전트의 신뢰 수준이 Actions 잡과 동일하다는 의미다.
Actions 잡은 저장소 설정과 브랜치 보호 규칙에 의해 통제되므로,
에이전트의 행동 범위도 동일한 통제 체계 안에 놓이게 된다.

이것은 “에이전트를 어디까지 신뢰할 것인가”라는 질문을 기존 인프라 신뢰 모델로 흡수하는 방식이다.
별도의 에이전트 권한 프레임워크를 설계하는 대신, 이미 검증된 Actions 권한 모델을 재사용한다.
이 접근은 새로운 공격 표면을 최소화하면서도 기존 보안 정책을 그대로 적용할 수 있다는 장점이 있다.

그러나 이 모델에는 내재된 긴장이 있다.
Actions 잡은 사람이 정의한 결정론적 스크립트를 실행하지만,
에이전트는 사람이 명시하지 않은 도구 호출과 코드 생성을 수행할 수 있다.
동일한 토큰으로 실행되더라도 신뢰의 성격이 다르다.
단기 토큰이 에이전트의 수명을 제한하는 것은 맞지만,
그 수명 동안 에이전트가 무엇을 할 수 있는지는 토큰 모델이 아니라 워크플로우 설계가 결정해야 한다.

### GitHub 플랫폼이 AI 에이전트의 기본 실행 환경으로 굳어지고 있다

이 변경은 기술적 세부 사항처럼 보이지만, 더 큰 전략적 움직임의 일부다.
`GITHUB_TOKEN` 통합은 Agentic Workflows를 GitHub Actions 생태계에 더 깊이 뿌리내리게 한다.
Actions의 트리거, 권한 모델, 비밀 관리, 감사 로그가 에이전틱 워크플로우에도 그대로 적용된다.

이는 GitHub 플랫폼 밖에서 에이전트를 운영하는 것을 상대적으로 덜 매력적으로 만든다.
외부 CI/CD 플랫폼에서 AI 에이전트를 실행하려면 별도의 인증, 비용 관리, 감사 체계를 구축해야 하지만,
GitHub Actions 위에서 실행하면 이 모두가 플랫폼으로 제공된다.

소프트웨어 개발 도구의 역사에서 플랫폼 잠금(platform lock-in)은 언제나 편의와 자율의 트레이드오프를 수반했다.
GitHub Agentic Workflows가 Actions 생태계와 더 긴밀하게 통합될수록,
다른 플랫폼으로 이식하는 비용도 커진다.
이 편의의 대가가 무엇인지, 조직이 충분히 인식한 채로 채택 결정을 내리고 있는지는
지금 물어봐야 할 질문이다.
