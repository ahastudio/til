# Enterprise 관리 설정에서 Copilot 권한 우회 모드를 비활성화할 수 있다

원문: <https://github.blog/changelog/2026-06-17-enterprise-managed-settings-now-support-bypass-permission-controls/>

## 요약

GitHub이 Enterprise 관리 설정(Enterprise-Managed Settings)에 새로운 거버넌스 기능을 추가했다.
기업 관리자는 이제 `disableBypassPermissionsMode`를 `“disable”`로 설정함으로써
GitHub Copilot CLI와 VS Code가 권한 프롬프트를 자동으로 건너뛰는 것을 막을 수 있다.
이 자동 승인 동작은 흔히 “yolo mode” 또는 자동 허가(auto-approve)라고 불린다.

설정 파일 위치는 `.github-private/.github/copilot/settings.json`이며,
대안 경로로 `copilot/managed-settings.json`도 지원하고 하위 호환성이 유지된다.
이 설정은 Copilot Business 또는 Copilot Enterprise 라이선스를 통해 기업 계정에서 사용하는 사용자에게 적용된다.
VS Code 1.122 이상 버전은 이미 이 Enterprise 관리 설정을 따른다.

이 기능은 2026년 5월 Copilot CLI, 2026년 6월 VS Code를 대상으로 각각 출시된
Enterprise 관리 플러그인 기능의 연장선이다.
관리자는 Enterprise 설정의 AI 컨트롤 하위 에이전트 페이지에서 적용된 설정을 확인할 수 있다.

## 분석

### “yolo mode 비활성화”가 필요한 이유는 yolo mode가 기본 상태였기 때문이다

`disableBypassPermissionsMode`라는 설정 이름은 “우회 모드를 비활성화하라”는 이중 부정이다.
이 이름이 존재한다는 것은 권한 프롬프트 우회가 기본값이거나 사용자가 켤 수 있는 상태로 존재했다는 뜻이다.
Copilot CLI와 VS Code에서 권한 확인 프롬프트를 자동으로 건너뛰는 “yolo mode”가
편의 기능으로 제공되고 있었고, 기업 환경에서 이를 억제할 수단이 없었던 것이다.

개인 개발자 입장에서 yolo mode는 반복적인 승인 클릭을 없애주는 생산성 도구다.
그러나 기업 입장에서 같은 기능은 코드 실행, 파일 수정, 외부 명령 호출에 대한
사전 인간 검토를 AI가 우회하는 구조적 위험이다.
이번 Enterprise 설정 추가는 그 위험이 현실적으로 인식되기 시작했음을 보여준다.

### 이 기능은 AI 도구의 기업 거버넌스 레이어가 독립 제품으로 성숙하고 있음을 나타낸다

Copilot Enterprise 관리 설정의 진화 경로는 흥미롭다.
처음에는 모델 선택, 제안 필터 같은 기능 설정이 주였지만,
이제는 AI 에이전트의 자율 실행 범위를 제한하는 보안 정책으로 확장되고 있다.
“어떤 AI 기능을 쓸 수 있는가”에서 “AI 에이전트가 사람 확인 없이 무엇을 할 수 있는가”로
관심이 이동하고 있다.

이 패턴은 모바일 기기 관리(MDM, Mobile Device Management)의 진화와 유사하다.
MDM이 처음에는 기기 등록과 앱 배포에서 시작해
나중에는 앱 별 권한, 데이터 분리, 원격 삭제까지 확장된 것처럼,
Copilot Enterprise 관리 설정도 AI 에이전트의 실행 권한을 조직 정책으로 통제하는
방향으로 단계적으로 나아가고 있다.

### 권한 우회 제어가 Enterprise 전용인 것은 보안이 플랜 구분선 위에 있음을 의미한다

이 설정은 Copilot Business 또는 Copilot Enterprise 라이선스 사용자에게만 적용된다.
개인 사용자나 소규모 팀이 쓰는 Pro·Max 플랜에서는 yolo mode 제어가 불가능하다.
기술적으로는 동일한 Copilot CLI와 VS Code를 사용하지만,
보안 거버넌스 기능은 더 비싼 플랜에서만 작동한다.

이것은 개인 개발자와 기업 개발자 사이에 다른 보안 기본값이 적용된다는 것을 의미한다.
기업 개발자는 관리자가 yolo mode를 끄면 강제로 권한 프롬프트를 확인해야 하지만,
같은 도구를 쓰는 프리랜서나 소규모 스타트업 개발자는 그 통제를 받지 않는다.
보안 수준이 라이선스 선택에 따라 달라지는 구조다.

## 비평

### 설정 이름의 이중 부정이 관리자 오류를 유발하기 쉬운 구조다

`disableBypassPermissionsMode: “disable”`은 이중 부정이다.
“우회(bypass)를 비활성화(disable)하는 모드(mode)를 비활성화(disable)하라”라고 읽힐 수 있다.
실제 의미는 “yolo mode 자체를 끄라”이지만, 설정 이름만 보면 직관적이지 않다.

기업 환경에서 설정 파일을 관리하는 사람은 반드시 문서를 읽고 의도를 확인해야 한다는 것은
이미 알려진 문제지만, 보안 관련 설정에서 이름이 혼란스러울 경우
“설정을 했으니 됐겠지”라고 생각하고 검증하지 않는 실수가 생기기 쉽다.
반대로 설정을 안 한 것이 있을 때 의도한 것인지 실수인지 구분하기도 어렵다.

보안 정책 설정에서 명명 관례의 명확성은 운영 신뢰성에 직결된다.
`requirePermissionApproval: true`처럼 긍정적이고 단순한 의미 구조가
같은 효과를 더 명확하게 전달할 수 있었을 것이다.

### 체인지로그는 우회가 실제로 어떤 권한에 적용되는지 구체적으로 설명하지 않는다

“권한 프롬프트를 자동으로 건너뛴다”는 것이 구체적으로 어떤 범위의 권한을 의미하는지
체인지로그는 명시하지 않는다.
파일 읽기와 파일 쓰기가 같은 수준의 권한인지,
로컬 쉘 명령 실행과 외부 API 호출이 같은 카테고리로 묶이는지,
혹은 Copilot의 MCP 도구 실행도 이 우회에 포함되는지는 불분명하다.

기업 관리자가 yolo mode를 비활성화해야 할 이유는 위험한 권한의 자동 실행 방지이다.
그런데 어떤 권한이 자동 실행되는지를 모르면,
이 설정을 켜야 할지 말지에 대한 근거 있는 판단을 내리기 어렵다.
체인지로그는 관리자 행동을 유도하지만, 그 행동의 필요성을 뒷받침할 위험 정보가 빠져 있다.

### Enterprise 관리 설정이 사용자 로컬 설정을 완전히 덮어쓰는지 여부가 불분명하다

Enterprise 관리 설정이 사용자 로컬 설정보다 우선하는가는
이 기능의 실질적 보안 효과를 결정하는 핵심 질문이다.
만약 사용자가 로컬 `.github/copilot/settings.json`에서 yolo mode를 켤 수 있고
Enterprise 설정보다 로컬 설정이 우선한다면, 이 거버넌스 기능은 무력화된다.

체인지로그는 Enterprise 설정이 적용된다고 말하지만,
설정 우선순위 계층(hierarchy)이 어떻게 구성되는지는 별도 문서로 안내한다.
보안 관련 설정에서 우선순위 계층은 부록이 아니라 기능 설명의 핵심이어야 한다.
“관리자가 끈 것을 사용자가 다시 켤 수 있는가”는 기업 IT 보안 팀이 가장 먼저 묻는 질문이다.

## 인사이트

### AI 에이전트의 자율 실행 범위 제어는 기업 보안의 새로운 기본 요건이 되고 있다

과거 기업 보안의 주요 관심사는 네트워크 접근 제어, 데이터 암호화, 인증 정책이었다.
Copilot 같은 AI 에이전트가 코드 실행, 파일 수정, 외부 서비스 호출을 자율적으로 수행할 수 있게 되면서
“AI가 사람 확인 없이 무엇을 할 수 있는가”가 새로운 보안 경계선이 되고 있다.
`disableBypassPermissionsMode`는 그 경계선을 정책으로 설정하는 최초의 공식 메커니즘 중 하나다.

이 변화의 이차적 효과는 기업 보안 팀이 AI 도구 감사(audit) 역량을 갖춰야 한다는 압력이다.
방화벽이나 접근 제어는 감사 로그와 함께 설계되고 운영된다.
Copilot의 권한 우회도 동일한 수준의 감사 가능성을 요구한다 —
“어떤 사용자가 yolo mode를 사용했는가”, “어떤 명령이 자동 승인되었는가” —
이런 정보 없이 권한 정책만 설정하는 것은 잠금장치를 달고 키 사용 기록은 없는 것과 같다.
현재 체인지로그는 이 감사 측면을 다루지 않는다.

### AI 도구의 “편의 기능”은 기업 환경에서 항상 사전에 위험 분류가 필요하다

yolo mode는 개인 생산성 도구로 설계된 기능이다.
반복적인 승인 클릭을 없애 개발 속도를 높이는 것이 목적이다.
같은 기능이 기업 환경에 도입될 때는 용도가 바뀌지 않아도 위험 수준이 달라진다.
개인 개발자가 자신의 로컬 환경에서 yolo mode로 실행하는 것과,
기업 코드베이스와 프로덕션 인프라에 접근할 수 있는 CI 환경에서 yolo mode가 켜져 있는 것은
위험 차원이 다르다.

역사적 유사점은 USB 드라이브 자동 실행(AutoRun) 기능이다.
Windows의 AutoRun은 개인 사용자를 위한 편의 기능으로 설계되었지만,
기업 환경에서는 Stuxnet 같은 악성코드 전파 경로가 되었다.
결국 Microsoft는 기업용 그룹 정책에서 AutoRun을 비활성화하는 옵션을 제공해야 했다.
Copilot의 yolo mode도 같은 경로를 밟고 있다 —
편의 기능에서 보안 정책 관리 대상으로 전환하는 것.
GitHub이 이번에 추가한 Enterprise 설정은 그 전환의 공식 인정이다.

### 권한 제어의 세분화는 AI 에이전트 보안의 다음 전쟁터가 될 것이다

현재 `disableBypassPermissionsMode`는 yolo mode 자체를 켜거나 끄는 이진 설정이다.
그러나 실제 기업 보안 요구는 더 세밀한 수준의 제어를 필요로 한다.
“파일 읽기는 자동 승인하되, 쉘 명령 실행은 반드시 사람이 확인해야 한다”처럼
권한 유형별 정책이 필요한 시나리오가 있다.
현재 설정은 그 세분화를 지원하지 않는다.

AI 에이전트의 권한 모델이 세분화될수록, 정책 설계의 복잡도도 함께 올라간다.
기업 관리자는 어떤 Copilot 작업이 어떤 권한을 요구하는지 이해하고,
그 조합을 조직의 보안 정책에 맞게 구성해야 한다.
이는 전통적인 IT 보안 관리와 AI 시스템 이해가 결합된 새로운 역할을 요구한다.
“AI 에이전트 보안 관리자”라는 역할이 독립적으로 존재하지 않는 지금,
그 공백을 누가 채울 것인가의 문제가 기업마다 다르게 풀리고 있다.
GitHub의 Enterprise 설정 확장은 그 복잡성이 커지기 전에
관리 진입점을 선점하려는 움직임이기도 하다.
