# GitHub Copilot에 Agent Finder가 추가되었다

원문: <https://github.blog/changelog/2026-06-17-agent-finder-for-github-copilot-now-available/>

## 요약

GitHub이 Copilot에 Agent Finder 기능을 출시했다.
Agent Finder는 개발자가 자연어로 작업을 설명하면
그에 적합한 AI 리소스를 자동으로 검색하고 순위가 매겨진 결과를 반환하는 기능이다.
핵심은 사전에 모든 도구를 로드해 둘 필요 없이,
필요한 시점에 필요한 리소스를 온디맨드로 불러올 수 있다는 점이다.

기술 기반은 Agentic Resource Discovery(ARD) 명세다.
이 명세는 GitHub이 단독으로 개발한 것이 아니라
Google, GoDaddy, Hugging Face, Microsoft와 공동으로 개발한 오픈 스탠더드다.
ARD 명세를 채택한 모든 레지스트리와 AI 클라이언트에서 Agent Finder가 작동할 수 있다.

거버넌스 측면에서 세 가지 제어 수단이 제공된다.
첫째, 레지스트리 선택: 조직은 GitHub의 공개 카탈로그나 자체 비공개 리소스 레지스트리를 지정해
Agent Finder의 검색 범위를 정할 수 있다.
둘째, Enterprise 관리 설정: Copilot을 관리하는 동일한 장소에서 에이전트가 검색하고 사용할 수 있는
리소스를 정의한다.
셋째, 수동 설치 통제: Agent Finder는 도구를 찾아줄 뿐 자동으로 연결하지 않으며,
설치 결정은 사용자에게 남아 있다.

이 기능은 모든 GitHub Copilot 플랜에서 사용 가능하며,
ARD 명세를 채택한 어떤 레지스트리나 AI 클라이언트와도 연동된다.

## 분석

### “도구를 사전 로드하지 않아도 된다”는 주장은 에이전트 문맥 관리의 핵심 문제를 건드린다

현재 AI 에이전트 시스템에서 도구 관리의 가장 큰 제약 중 하나는 문맥 창(context window)의 한계다.
MCP 서버를 통해 수십 개의 도구를 에이전트에 연결하면,
각 도구의 스키마 설명이 문맥을 소비한다.
도구가 많을수록 실제 작업에 쓸 수 있는 문맥이 줄어드는 구조적 긴장이 있다.

Agent Finder가 “온디맨드 로딩”을 가능하게 한다는 것은
문맥 창 효율성 문제를 해결하려는 접근이다.
작업 설명에 기반해 실제로 필요한 도구만 런타임에 불러오면,
전체 도구 카탈로그를 사전에 로드하지 않아도 된다.
이것은 이론적으로 훨씬 큰 도구 생태계를 활용 가능하게 하면서
동시에 에이전트의 문맥 효율을 유지하는 방법이다.

그러나 이 접근은 새로운 지연(latency)을 도입한다.
에이전트가 도구를 필요로 하는 시점에 검색하고, 스키마를 가져오고, 도구를 로드하는 과정이
추가되기 때문이다.
“어떤 도구가 필요한지 먼저 파악한 뒤 불러온다”는 이 순서는
에이전트 추론 루프 안에 네트워크 왕복을 삽입한다.
이 지연이 실제 사용 경험에서 어떻게 드러나는지는 체인지로그가 다루지 않는 영역이다.

### ARD 오픈 스탠더드의 공동 개발 명단이 보내는 신호

ARD 명세를 함께 개발한 파트너들이 무작위로 선택된 것이 아니다.
Google은 자체 AI 에이전트 생태계(Gemini, Vertex AI)를 보유하고 있고,
Hugging Face는 오픈소스 AI 모델 허브의 핵심 플레이어다.
Microsoft는 GitHub의 모회사이자 Azure AI 플랫폼의 운영자이며,
GoDaddy는 Agent Finder를 실제 비즈니스 워크플로에 적용할 대형 기업 고객 사례다.

이 조합은 ARD가 GitHub 에코시스템에 국한되지 않는 범용 에이전트 리소스 검색 표준을 목표로 한다는 것을 보여준다.
역사적 유사점은 OAuth와 OpenID Connect의 표준화 과정이다.
인증 프로토콜이 여러 플레이어 간 합의된 표준이 되었을 때 비로소
“어느 서비스에서나 Google 계정으로 로그인”이 가능해졌듯이,
ARD가 표준화되면 “어느 AI 클라이언트에서나 같은 도구 레지스트리를 탐색”하는 것이 가능해진다.

그러나 오픈 스탠더드의 성공은 기술 명세의 품질보다 채택 속도에 달려 있다.
Microsoft와 Google이 함께 참여했다는 것은 초기 채택을 확보할 조건이 갖춰졌다는 신호다.
하지만 두 회사가 각자의 에이전트 생태계에서 ARD를 실제로 기본값으로 구현할지,
아니면 자사 도구 디스커버리 방식을 유지하면서 ARD를 부가 옵션으로 지원할지는 다른 문제다.

### “찾아줄 뿐 연결하지 않는다”는 설계 철학이 해결하는 신뢰 문제

Agent Finder가 도구를 검색하고 순위를 매기지만 자동으로 연결하지 않는다는 설계는
에이전트 시스템에서 중요한 신뢰 레이어를 유지한다.
에이전트가 스스로 필요하다고 판단한 도구를 자율적으로 설치하고 실행하는 시나리오는
보안과 감사 가능성 측면에서 기업 환경에서 수용하기 어려운 모델이다.

“검색은 자동, 설치는 수동”이라는 분리는 에이전트의 능력과 인간의 통제권 사이에
명확한 경계를 유지한다.
이것은 단순한 UX 선택이 아니라 기업 AI 거버넌스의 핵심 원칙을 반영한 것이다.
같은 날 발표된 Enterprise 관리 설정의 `disableBypassPermissionsMode`와 같은 맥락에서
읽힌다: GitHub은 AI 에이전트의 자율성을 확장하면서 동시에 그 자율성의 경계를 명시적으로 설정하고 있다.

이 철학이 일관되게 유지된다면,
Agent Finder는 “에이전트가 도구를 스스로 결정하는 완전 자율 시스템”이 아니라
“에이전트가 추천하고 사람이 승인하는 협력 시스템”의 기반이 된다.
그러나 자동화 압력 — 더 빠른 실행, 더 적은 마찰 — 이 증가함에 따라
이 경계가 앞으로 어떻게 변화할지는 주목해야 할 지점이다.

## 비평

### 체인지로그는 “검색 품질”이라는 가장 중요한 문제를 전혀 다루지 않는다

Agent Finder의 핵심 가치 제안은 “적합한 AI 리소스를 찾아준다”는 것이다.
그런데 “적합함”을 어떻게 정의하고 어떻게 측정하는지에 대한 설명이 없다.
자연어로 작업을 설명했을 때 반환되는 “순위가 매겨진 결과”가
실제로 맥락에 맞는 도구를 상위에 보여주는지는 전적으로 검색 알고리즘의 품질에 달려 있다.

도구 검색의 실패 모드는 세 가지다.
관련 도구를 찾지 못하는 재현율(recall) 실패,
부적합한 도구를 상위에 올리는 정밀도(precision) 실패,
그리고 서로 다른 작업에 대해 동일한 범용 도구를 반복 추천하는 다양성 실패다.
체인지로그는 이 세 가지 문제 중 어느 것도 언급하지 않는다.
“검색 결과가 좋다”는 사용자 경험은 Agent Finder가 이 기능을 실제로 얼마나 잘 수행하는지에
전적으로 의존하는데, 그 품질에 대한 어떠한 정보도 공개되지 않았다.

### 비공개 리소스 레지스트리와 공개 카탈로그의 혼합 시나리오가 다루어지지 않는다

조직이 GitHub의 공개 카탈로그나 자체 비공개 레지스트리를 선택할 수 있다는 설명은
두 가지 옵션을 이진 선택처럼 제시한다.
그러나 실제 기업 환경에서는 공개 도구와 비공개 도구를 함께 써야 하는 경우가 일반적이다.
내부 데이터베이스에 접근하는 비공개 MCP 서버와
공개적으로 사용 가능한 외부 API 도구를 같은 에이전트 세션에서 사용하고 싶은 시나리오가 있다.

Agent Finder가 복수의 레지스트리를 동시에 탐색하는 것을 지원하는지,
공개 카탈로그와 비공개 레지스트리 간의 충돌이나 중복을 어떻게 처리하는지,
비공개 레지스트리에 있는 도구가 공개 카탈로그 결과보다 우선순위가 높은지 낮은지 —
이 모든 실용적 질문이 체인지로그에서 답을 찾을 수 없다.
기업 채택을 목표로 한다면 이 시나리오들이 핵심적인데, 설명이 추상적인 수준에 머물렀다.

### “모든 ARD 채택 클라이언트와 호환”이라는 주장은 검증되지 않은 약속이다

ARD 명세를 채택한 어느 AI 클라이언트와도 Agent Finder가 작동한다는 주장은
오픈 스탠더드의 상호운용성을 전제하지만,
ARD를 실제로 구현한 클라이언트가 얼마나 있는지,
그 구현이 어느 수준의 호환성을 보장하는지는 명시되지 않는다.

오픈 스탠더드의 역사에는 “표준이 있어도 구현이 달라 실제로 호환되지 않는” 사례가 많다.
HTML 표준이 있어도 브라우저마다 동작이 달랐던 시기처럼,
ARD 명세를 따른다고 해도 각 클라이언트의 구현이 실제로 상호운용 가능한지는 별개다.
Google이나 Microsoft가 ARD를 채택했다고 발표한 것과
그 구현이 GitHub의 Agent Finder와 실제로 원활하게 연동되는 것은 다른 이야기다.
“함께 명세를 개발했다”는 것이 “완전히 호환 가능하게 구현했다”를 의미하지는 않는다.

## 인사이트

### Agent Finder는 “무한한 도구 생태계”와 “유한한 문맥 창” 사이의 구조적 긴장을 해결하는 첫 번째 제도적 시도다

AI 에이전트 시스템의 근본적 제약 중 하나는 도구 생태계와 문맥 창 사이의 역설이다.
도구가 많을수록 에이전트가 더 많은 것을 할 수 있지만,
동시에 도구 스키마가 문맥을 소비해 실제 작업에 쓸 수 있는 공간이 줄어든다.
오늘날 많은 팀이 “어떤 MCP 서버를 활성화해야 하는가”를 에이전트 설정의 핵심 결정으로 여기는 것은
이 긴장의 직접적인 표현이다.

Agent Finder는 이 긴장을 “필요할 때만 로드하라”는 방식으로 해결하려 한다.
이것은 운영 체제의 가상 메모리(virtual memory) 개념과 구조적으로 같다.
RAM이 부족할 때 디스크를 가상 메모리로 사용해 더 큰 프로그램을 실행하는 것처럼,
Agent Finder는 도구가 필요한 순간에 레지스트리에서 불러와 에이전트의 “도구 메모리”를 동적으로 확장한다.

이 접근이 성숙하면 에이전트 시스템의 아키텍처가 달라진다.
오늘날 “어떤 도구를 에이전트에 연결할지”를 미리 결정하는 정적 구성(static configuration) 방식에서,
에이전트가 실행 중 필요한 도구를 동적으로 탐색하고 가져오는 방식으로 전환된다.
이 전환이 완성되면 에이전트의 능력은 사전에 구성된 도구 집합이 아니라
접근 가능한 레지스트리의 크기에 의해 결정된다.
Agent Finder는 그 전환을 위한 인프라 레이어다.

### ARD가 표준이 되면 도구 생태계는 앱 스토어 경제와 동일한 역학을 따를 것이다

오픈 스탠더드 기반 도구 레지스트리의 성숙은 AI 도구 생태계를 앱 스토어와 유사한 구조로 변환한다.
앱 스토어에서는 개발자가 표준 API를 따라 앱을 만들어 스토어에 제출하고,
사용자는 검색을 통해 필요한 앱을 찾아 설치한다.
ARD 기반 레지스트리에서는 도구 개발자가 ARD 명세를 따라 MCP 서버를 만들어 레지스트리에 등록하고,
에이전트는 Agent Finder를 통해 필요한 도구를 찾아 로드한다.

이 유사성은 앱 스토어 경제의 역학도 함께 가져온다.
검색 순위 알고리즘이 어떤 도구가 노출되는지를 결정하고,
상위 노출을 위한 최적화(도구 설명 작성 방법, 키워드 선택)가 새로운 마케팅 영역이 된다.
인기 있는 도구가 더 많이 발견되어 더 인기를 얻는 선순환이 생기고,
유사한 기능의 도구 중 어느 것이 “공식” 또는 “검증된” 도구로 인식되는지가 중요해진다.

Agent Finder를 운영하는 GitHub이 어떤 도구가 상위에 노출되는지를 제어하는 위치에 있다는 것은
기존 앱 스토어에서 플랫폼 사업자가 가진 것과 동일한 구조적 권력을 갖는다는 의미다.
오픈 스탠더드라는 표방에도 불구하고,
GitHub의 공식 카탈로그가 레지스트리 시장의 기본값이 되는 순간
이 권력은 가시화된다.

### 자연어 기반 도구 탐색은 개발자가 “무엇이 가능한지”를 알아야 한다는 전제를 뒤집는다

전통적인 개발 도구 환경에서 개발자는 도구의 존재를 먼저 알고, 그다음 사용법을 배우는 순서를 따른다.
어떤 라이브러리가 있는지 모르면 검색할 수 없고,
npm이나 PyPI에서 패키지를 찾으려면 적어도 어떤 종류의 도구가 필요한지 알아야 한다.
이 구조에서 도구 발견(tool discovery)의 장벽은 의식적으로 “무엇을 찾아야 할지”를 아는 것이었다.

Agent Finder는 이 순서를 뒤집는다.
“결과물을 설명하면 어떤 도구가 필요한지를 대신 찾아준다”는 모델에서
개발자는 도구의 존재를 몰라도 그 도구를 사용할 수 있다.
이것은 발견 가능성(discoverability) 문제를 해결할 뿐 아니라
“AI 시스템의 능력 경계”가 개발자의 도구 지식이 아니라 레지스트리의 내용에 의해 결정되는 새로운 구조를 만든다.

이 전환의 이차적 효과는 도구 개발자의 마케팅 전략이 바뀐다는 것이다.
기존에는 개발자가 “이런 도구가 있다”는 것을 알리는 것이 채택의 전제였다.
Agent Finder 세상에서는 도구 설명을 에이전트가 이해하는 방식으로 작성하는 것이 핵심이 된다.
검색 엔진 최적화(SEO)가 웹 페이지의 문서 작성 방식을 바꿨듯이,
에이전트 검색 최적화(AEO, Agent Experience Optimization이라고 부를 수 있을)가
MCP 서버 설명과 도구 스키마 작성 방식을 바꾸는 새로운 전문 영역이 될 수 있다.
