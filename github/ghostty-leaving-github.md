# Ghostty는 왜 GitHub를 떠나는가

원문: <https://mitchellh.com/writing/ghostty-leaving-github>

HN 토론: <https://news.ycombinator.com/item?id=47939579> (2096점, 640개 댓글)

## 요약

Mitchell Hashimoto는 자신이 만든 터미널 에뮬레이터 Ghostty 프로젝트를 GitHub에서
다른 플랫폼으로 이전하겠다고 밝혔다.
Hashimoto는 2008년 2월부터 사용해온 GitHub 계정 번호 #1299를 가진 초기 사용자로,
플랫폼과의 감정적 유대를 솔직하게 인정하면서도 이전 결정을 내렸다.
그는 GitHub 덕분에 경력을 쌓았고, Vagrant를 만든 동기 중 하나가
GitHub 엔지니어링 팀에 깊은 인상을 남기기 위해서였다고 고백한다.

이전의 직접적인 원인은 신뢰성 문제다.
Hashimoto는 한 달 이상 GitHub의 장애를 추적했으며, 거의 매일 자신의 작업에 영향을
미치는 장애가 발생했다고 밝혔다.
문제의 핵심은 Git 자체가 아니라, 이슈·풀 리퀘스트·GitHub Actions 등 주변 서비스다.
2026년 4월 27일 대규모 장애가 발생한 날, 그는 GitHub Actions 장애로 인해
약 2시간 동안 PR 리뷰를 전혀 할 수 없었다.

이 결정이 특정 장애 하나로 인한 충동적 반응이 아님을 그는 강조한다.
수개월에 걸쳐 누적된 판단이었고, 4월 27일 장애는 그저 발표의 시점을 제공했을 뿐이다.
공개적으로 GitHub를 비판해왔던 자신이 “심술궂었다(mean)”고 인정하면서도,
GitHub 직원들의 감정이 상했을 수 있다는 점도 언급한다.

Ghostty는 상업적·오픈소스 대안 플랫폼으로 이전하되, GitHub에 읽기 전용 미러는
유지할 계획이다.
Hashimoto의 개인 프로젝트들은 당분간 GitHub에 남아 있을 것이며,
이전은 점진적으로 진행된다.

## 분석

### 신뢰성 문제의 누적 효과

월별 장애 추적이라는 Hashimoto의 방식은 단순한 불만 토로가 아니라,
엔지니어링적 사고방식의 발현이다.
단일 장애는 허용 가능하지만, 매일 반복되는 장애는 인프라 의존성을 재검토하게 만든다.
GitHub Actions 장애로 인한 2시간 PR 리뷰 불가라는 구체적 수치는,
플랫폼 의존성이 생산성과 직결된다는 현실을 보여준다.

특히 Ghostty처럼 활발한 오픈소스 프로젝트에서 GitHub Actions는 단순한 CI가 아니다.
기여자 리뷰 흐름, 자동화된 릴리즈, 커뮤니티 관리 전반이 Actions에 엮여 있다.
핵심 인프라가 예측 불가능하게 작동하면, 프로젝트 유지 관리자의 인지 부하가
급격히 높아진다.

### 플랫폼 종속과 감정적 비용

이 글에서 주목할 점은 감정적 서사다.
Hashimoto는 GitHub와의 깊은 유대를 숨기지 않는다.
이는 플랫폼 이전이 단순한 기술적 결정이 아님을 드러낸다.
개발자에게 GitHub는 정체성의 일부이며, 포트폴리오이자 네트워크다.
그럼에도 이전을 결정했다는 것은, 신뢰성 문제가 감정적 비용을 초과했다는 의미다.

### Microsoft 인수 이후의 GitHub

Hashimoto가 직접 언급하지는 않지만, 타이밍은 의미심장하다.
Microsoft가 GitHub를 인수한 것은 2018년이다.
이후 GitHub는 Copilot, Azure 통합, 엔터프라이즈 기능 확장에 집중해왔다.
핵심 인프라 신뢰성이 저하되는 동안, 제품 로드맵은 AI·엔터프라이즈 방향으로
흘러갔다는 비판이 Hacker News 스레드에서 반복적으로 등장한다.
Heroku의 Salesforce 인수 이후 쇠락과의 비교도 여러 댓글에서 제기됐다.
nimbius는 “개발자를 잃으면, 21세기의 Xerox가 되는 길을 걷게 된다”고 경고했다.[^nimbius]
Microsoft가 개발자 커뮤니티에 막대한 투자를 해왔음에도, 지금의 상황이 개발자들을
밀어내고 있다는 비판이다.

한편, 신뢰성이 아닌 다른 이유로 일찌감치 GitHub를 떠난 사람들도 있다.
pull_my_finger는 “Copilot 훈련에 허락 없이 코드를 사용한 것이 확실해진 시점부터
GitHub에 진지한 작업을 올리지 않았다”고 밝혔다.[^pull_my_finger]
인프라 문제 이전에 이미 코드 소유권과 라이선스 문제가 이탈의 동인이 됐다는 것이다.
Copilot은 GitHub를 잠금(lock-in) 강화의 수단으로 썼지만, 동시에 일부 개발자들이
플랫폼을 떠나는 계기가 됐다는 역설이 여기서도 드러난다.

### Forgejo·Codeberg의 부상

댓글 스레드에서 Forgejo와 Codeberg가 대안으로 언급된다.
이 도구들은 수년 전만 해도 GitHub 대안으로 진지하게 논의되지 않았다.
Ghostty 같은 고프로필 프로젝트의 이전은 이 대안 플랫폼들에 대한 신호 효과를 갖는다.
단순한 기술 선택이 아니라, 개발자 생태계의 무게 중심이 조금씩 이동하는 징후일 수 있다.

varun_ch는 tangled.org를 흥미로운 대안으로 지목했다.[^varun_ch]
ATProto 기반의 분산 아이덴티티를 활용해, 개발자가 단일 계정으로 여러 플랫폼에 기여하면서
핵심 인프라(Git, CI)는 탈중앙화하는 설계다.

다만 infogulch는 대안 플랫폼 생태계가 오히려 파편화로 이어질 수 있다고 경고한다.[^infogulch]
이슈·PR 이력이 플랫폼 전환마다 유실되고, 어느 플랫폼도 GitHub 규모의 발견·탐색 기능을
갖추지 못하면, “모든 프로젝트가 자기만의 섬”이 되는 바벨탑 시나리오가 펼쳐진다.
Radicle처럼 이슈 자체를 Git 레포 안에 저장하는 접근이 이론적 해법으로 거론되지만,
아직 성숙한 단계가 아니다.

### GitHub 내부의 자기 인식

hmokiguess는 GitHub Staff Research Engineer의 발언을 공유했다.[^hmokiguess]
“내부에서 매우 소수의 사람만이 PR과 이슈가 엔지니어링의 미래를 위한 이상적인 기본 단위라고
믿는다”는 것이다.
이 발언은 신뢰성 문제를 넘어서는 더 구조적인 인식을 드러낸다.
GitHub 스스로 현재의 협업 모델이 한계에 도달했음을 내부적으로 인지하고 있다면,
Hashimoto의 이전 결정은 단순한 불만의 표출이 아니라 다음 세대 도구를 향한 선제적 이동으로
읽힐 수 있다.

## 비평

### 강점

Hashimoto의 글은 감정과 데이터를 균형 있게 활용한다.
월별 장애 추적이라는 구체적 증거, GitHub #1299라는 역사적 맥락,
그리고 자신이 “심술궂었다”는 자기 비판적 인정이 글의 신뢰도를 높인다.
충동적 반응이 아니라 수개월의 숙고 끝에 내린 결정임을 강조하는 것도 설득력을 더한다.

### 약점

글은 GitHub의 인프라 운영 난이도를 과소평가하는 경향이 있다.
수억 개의 저장소와 수십억 건의 Actions 실행을 지원하는 플랫폼에서
99.9% 이상의 가용성을 유지하는 것은 공학적으로 극도로 어렵다.
Hashimoto가 추적한 “거의 매일의 장애”가 실제로 얼마나 심각한 수준이었는지—
전체 서비스 중단인지, 부분 저하인지—에 대한 세부 정보가 없다.

또한 이전할 대안 플랫폼을 구체적으로 밝히지 않은 채 “상업적·오픈소스 옵션을 검토 중”이라고
하는 것은, 대안이 GitHub보다 실제로 더 신뢰할 수 있다는 근거를 제시하지 못한다.

장애 경험의 주관성도 논쟁거리다.
raincole은 자신은 약 6개월에 한 번 꼴로 장애를 경험하며, “거의 매일”이라는 주장은
복잡한 Actions 워크플로를 운영하는 프로젝트에 한정된 경험일 수 있다고 반론한다.[^raincole]
대규모 오픈소스 프로젝트를 운영하는 유지 관리자와 일반 개인 사용자의 장애 노출 빈도가
크게 다를 수 있다는 맥락이 글에는 빠져 있다.

대안이 GitHub를 온전히 대체할 수 있는지도 검증이 필요하다.
dewey는 GitHub의 모바일 앱—스마트폰에서 이슈를 확인하고 PR 알림을 받는 경험—이
종종 간과되는 플랫폼 접착성의 핵심이라고 지적한다.[^dewey]
코드 저장 기능은 Forgejo나 Codeberg로 대체 가능하지만, 앱·탐색·개발자 네트워크로 이루어진
생태계 전체는 훨씬 높은 교체 비용을 요구한다.

### 'GitHub 직원이 읽을 것을 전제한' 서사

글 전반에 GitHub 직원을 의식하는 톤이 흐른다.
“내가 심술궂었다”, “hurt feelings” 언급 등은 논쟁이 아닌 설득을 목표로 한다.
이는 순수한 기술적 평가라기보다 공개 압력(public pressure) 행사에 가깝다.
이 점이 글의 진정성을 해치지는 않지만, 독자는 이 맥락을 고려해야 한다.

### 독점 플랫폼에 대한 감정적 유대 자체가 문제인가

JuniperMesos는 더 근본적인 메타 비판을 제기한다.[^JuniperMesos]
“GitHub는 처음부터 타인이 운영하는 자유롭지 않은 소프트웨어였고, 그것은 처음부터 분명했다”는
것이다.
이메일 계정 요구, 이용약관 동의, 미국 제재 준수를 위한 지역 차단—이런 조건들은 출발부터
존재했다.
이 관점에서 보면, 독점 플랫폼에 깊은 감정적 유대를 형성한 것 자체가 판단 오류이며,
이전의 고통은 처음부터 거리를 뒀더라면 피할 수 있었다.

이 비판은 가혹하지만 무시하기 어렵다.
개발자 커뮤니티 전체가 GitHub의 네트워크 효과와 편의성에 이끌려 선택지를 스스로 좁혀온
것은 사실이다.
“당시에는 대안이 없었다”는 변명이 통하는지는 각자의 판단에 달려 있다.

## 인사이트

### 플랫폼 신뢰성은 감정이 아닌 기회비용으로 계산된다

Hashimoto의 이전 결정에서 가장 주목해야 할 것은 감정이 아니라 기회비용의 계산이다.
그는 18년 이상의 GitHub 사용자이며, 이전에는 실질적인 마이그레이션 비용이 따른다.
저장소 이전, 기여자 워크플로 변경, 이슈·PR 이력 이전, CI 재구성—이 모든 비용보다
GitHub 장애로 인한 생산성 손실이 더 크다는 판단이 이전을 촉발했다.

이것은 플랫폼 의존성에 관한 보편적 교훈을 담고 있다.
플랫폼 전환 비용이 아무리 높아도, 현재 플랫폼의 신뢰성 하락이 그 비용을 넘어서는
순간이 반드시 온다.
조직이나 프로젝트가 플랫폼 의존도를 높일 때는 항상 이 임계점을 고려해야 한다.
고-가용성 인프라가 당연시되던 시대의 종말이 오고 있다면,
모든 팀이 자신의 스택에서 “Ghostty가 GitHub에 느끼는 것”과 같은 감정을
어딘가에서 느끼고 있을 가능성이 높다.

### “탈출 vs. 발언”의 전략적 딜레마

경제학자 앨버트 허시먼(Albert Hirschman)은 1970년 저서 『Exit, Voice, and Loyalty』에서
조직이나 제품이 악화될 때 구성원이 선택할 수 있는 두 가지 전략을 제시했다.
탈출(Exit)—조용히 떠나는 것—과 발언(Voice)—내부에서 변화를 요구하는 것이다.
충성도(Loyalty)가 높을수록 탈출보다 발언을 택하는 경향이 있다.

Hashimoto의 사례는 이 프레임을 흥미롭게 복잡화한다.
그는 공개 비판(발언)을 지속해왔지만, 플랫폼이 반응하지 않자 탈출을 선택했다.
Hacker News에 댓글을 남긴 GitHub 직원 idan은 “GitHub는 신경 쓰는 사람들이
남아서 개선해야 발전한다”고 주장했다—전형적인 발언 전략의 옹호다.
하지만 이 논리에는 함정이 있다.
발언의 효과는 탈출의 위협이 실질적일 때에만 작동한다.
고프로필 프로젝트들이 실제로 떠나지 않는 한, 플랫폼이 변화할 인센티브는 약하다.

역설적으로, Ghostty의 이전은 잔류하는 발언(Voice)보다 더 강력한 발언일 수 있다.
이전을 발표하는 행위 자체가 GitHub에 “이것은 심각하다”는 신호를 보내는 가장 효과적인 방법이다.

### 개발자 인프라의 권력 집중과 그 취약성

GitHub는 현재 세계 최대의 소스 코드 저장소이자 오픈소스 협업의 사실상(de facto) 인프라다.
npm, PyPI, Docker Hub 같은 패키지 레지스트리와 함께, GitHub Actions는
현대 소프트웨어 공급망의 핵심 신경계로 기능한다.
이 집중화는 네트워크 효과를 극대화했지만, 동시에 단일 실패 지점(single point of failure)을
만들었다.

GitHub가 2시간 장애를 겪을 때, 단순히 한 플랫폼이 느려지는 것이 아니다.
수천 개 프로젝트의 빌드 파이프라인이 멈추고, PR 리뷰가 차단되고,
자동화된 보안 검사가 작동하지 않는다.
2020년 SolarWinds 공급망 공격이 보여주었듯이, 개발자 인프라의 집중화는
단순한 불편함을 넘어 보안·가용성·주권(sovereignty)의 문제다.

Ghostty의 이전은 이 집중화에 대한 소규모 저항이다.
하지만 이런 저항들이 누적되면—특히 고프로필 프로젝트들이 대안으로 이동하면—
분산된 개발자 인프라 생태계가 실질적인 대안으로 부상할 수 있다.
ActivityPub이 소셜 미디어 분산화의 가능성을 보여준 것처럼,
Forgejo 같은 연합형(federated) Git 호스팅의 미래가 예전보다 구체적으로 그려지기 시작한다.

### GitHub Actions 딜레마: 탈출을 막는 올가미가 탈출을 유발한다

역설이 하나 있다.
GitHub Actions는 개발자를 GitHub 생태계에 묶어두기 위한 전략적 기능이다.
CI/CD를 플랫폼 내부에 통합함으로써, 이전 비용을 극적으로 높인다.
그러나 Hashimoto의 사례에서 정확히 그 기능—GitHub Actions—이 가장 자주 장애를 일으켰고,
이전을 결심하게 만든 주요 원인이 됐다.

이것은 플랫폼 잠금(platform lock-in) 전략의 아이러니를 드러낸다.
탈출을 막기 위해 설계된 기능이 충분히 불안정해지면, 오히려 탈출의 가장 강력한 동기가 된다.
기능 통합으로 의존성을 높이는 전략은 그 기능의 신뢰성이 전제될 때에만 작동한다.
신뢰성이 무너지면, 통합된 기능의 수만큼 이전의 동기도 함께 커진다.
이는 GitHub뿐 아니라 모든 플랫폼 전략에 적용되는 법칙이다.
