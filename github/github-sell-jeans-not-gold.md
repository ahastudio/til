# GitHub은 금을 캐는 대신 청바지를 팔아야 한다

원문: <https://blog.outsider.ne.kr/1797/>

## 요약

Outsider는 10년 넘게 GitHub을 지지해온 헌신적 팬이다.
GitHub SWAG 전용 공간을 따로 마련할 정도의 애정을 가진 저자는,
2018년 Microsoft 인수 당시에도 우려를 품으면서도 결국 긍정적으로 바라봤다.
그 관점에서 쓴 글이기 때문에 비판의 무게가 다르다.

2021년 GitHub Copilot이 처음 나왔을 때 시장은 GitHub의 독보적 우위를 당연하게 봤다.
OpenAI의 Codex 모델을 기반으로 한 Copilot은 “탭만 누르면 코드가 자동으로 완성”되는 경험을 처음 제시한 제품이었다.
그러나 2026년 현재 판도가 달라졌다.
저자는 Copilot이 “1티어에서 2티어로 내려왔다”고 평가한다.
Claude Code(Anthropic), Codex CLI(OpenAI), Antigravity(Google)가 새로운 1티어를 형성했고,
Cursor, opencode 같은 제품이 Copilot과 같은 2티어를 점유한다.

GitHub의 위치 약화에는 복합적인 원인이 있다.
Nat Friedman CEO 이후 Thomas Dohmke 체제에서 “Microsoft의 분위기가 조금씩 나기 시작”했다고 저자는 진단한다.
빈번한 서비스 장애로 “GitHub이 장애 없이 며칠 지났는지 기록하는 서비스”까지 등장했다.
HashiCorp 공동창업자 Mitchell Hashimoto가 “장애가 너무 많아서 GitHub을 떠나겠다”고 공식 선언한 것은 단순한 불평이 아니라 상징적 사건이다.

역설적으로 GitHub의 성장 수치 자체는 역대 최고다.
2024년에 3,600만 명이 신규 가입해 역사상 최고 성장률을 기록했고, 총 사용자는 1억 8천만 명에 달한다.
저자는 이 숫자가 “10억 명의 개발자”를 향한 청사진으로 제시된 것에 주목하면서도,
성장 수치와 서비스 품질 사이의 간극을 문제로 짚는다.

저자의 핵심 메시지는 “청바지를 팔아라”라는 비유로 수렴한다.
골드러시 시대에 금을 캐러 온 사람들에게 청바지를 판 Levi Strauss가 더 성공했다는 이야기다.
GitHub이 코딩 에이전트(AI 개발 도구) 시장에서 직접 금을 캐려 하기보다,
코드 협업 플랫폼이라는 핵심 위치를 더 잘 지키고 발전시키는 것이 더 유리한 전략이라는 주장이다.
코딩 에이전트가 활성화될수록 코드 리뷰, diff 가독성, 코드 협업 인터페이스의 중요성이 커지기 때문이다.

## 분석

### “청바지 비유”의 전략적 함의

저자가 사용한 골드러시 비유는 전략론에서 “픽앤셔블(picks and shovels)” 투자라고 부르는 개념과 맞닿는다.
기술 변화의 수혜를 직접 경쟁자가 아닌 인프라 제공자로서 누리는 전략이다.
현재 코딩 에이전트 시장은 주요 AI 회사들이 직접 참전해 치열하게 경쟁하는 레드오션이다.
Claude Code, Codex CLI, Antigravity, Cursor가 동시에 경쟁하는 시장에서 GitHub Copilot이 1위를 유지하기는 구조적으로 어렵다.
각 AI 회사가 자사 모델을 최고 성능으로 연결한 도구를 내놓는 것과 경쟁해야 하기 때문이다.

반면 코딩 에이전트들이 모두 코드를 GitHub에 푸시하고, PR을 열고, 코드 리뷰를 받는다면,
GitHub은 에이전트 시장의 승자와 무관하게 이익을 얻는다.
코딩 에이전트가 늘어날수록 GitHub의 트래픽과 가치는 증가한다.
저자가 “에이전트와 협업하는 인터페이스”를 개선하라고 제안하는 것도 같은 방향이다.
도구를 만드는 것이 아니라 도구가 잘 작동하는 환경을 만드는 전략이다.

### 서비스 안정성 문제의 구조적 원인

저자는 장애 빈발을 Microsoft 분위기 유입과 연결짓지만, 더 깊은 원인이 있다.
코딩 에이전트의 확산 자체가 GitHub 인프라에 질적으로 다른 부하를 만들고 있다.
인간 개발자는 하루에 수십 번 커밋하지만, AI 에이전트는 수분 간격으로 커밋을 생성한다.
하나의 기능을 개발하는 과정에서 수백 번의 커밋, 수십 번의 PR 생성, 지속적인 API 호출이 발생한다.
이것은 기존 인프라가 예상하지 못한 사용 패턴이다.
성장하는 수치와 악화되는 안정성이 같은 원인(AI 코딩 확산)에서 비롯된다는 역설이 여기 있다.

### 코딩 에이전트와 코드 협업 플랫폼의 관계 재설정

저자가 제안하는 “에이전트 친화적 인터페이스”는 단순한 UI 개선 요청이 아니다.
코딩 에이전트가 생성한 코드의 diff는 인간이 작성한 diff와 패턴이 다르다.
수백 줄을 한꺼번에 바꾸는 대규모 변경이 잦고, 변경 맥락을 설명하는 커밋 메시지의 품질이 불균일하다.
이 코드를 인간이 리뷰하고 승인하는 과정이 새로운 병목이 됐다.
코드 diff 가독성, AI 기반 코드 리뷰 보조, 에이전트 생성 PR의 맥락 파악 도구는
에이전트 시대의 코드 협업이 필요로 하는 새로운 인터페이스 문제다.
GitHub이 이 문제를 잘 풀면, 에이전트가 누구의 제품이든 GitHub은 필수 레이어가 된다.

## 비평

### 강점: 장기 팬의 시각이 주는 신뢰성

비판은 누구든 할 수 있지만, 오랜 지지자가 실망을 표현할 때의 무게는 다르다.
저자가 GitHub SWAG를 모을 정도의 팬이었다는 사실을 전제로 비판을 읽으면,
이것이 단순한 AI 비관론이나 안티 Microsoft 정서가 아님을 알 수 있다.
“이 회사가 잘됐으면 좋겠어서 하는 말”이라는 톤이 글 전체를 관통한다.
이런 목소리가 익명의 불평보다 회사 내부에서 더 진지하게 받아들여질 가능성이 있다.

### 약점: “2티어” 진단의 근거 부재

Copilot이 2티어로 내려왔다는 진단은 저자의 체감에 의존한다.
시장 점유율 데이터, 사용자 설문, 또는 구체적인 비교 벤치마크가 제시되지 않는다.
코딩 에이전트 시장은 아직 초기 단계라 공인된 랭킹 데이터가 없다는 제약이 있지만,
주장의 근거를 명시적으로 다루지 않은 것은 아쉽다.
GitHub Copilot은 여전히 VS Code 통합에서 압도적인 사용자를 보유하고 있으며,
엔터프라이즈 시장에서의 위치도 다른 요소다.

### 보완할 시각: GitHub의 딜레마를 더 직시할 것

“청바지를 팔아라”는 조언은 외부에서 보면 명확하지만, GitHub 내부에서는 복잡한 딜레마다.
Copilot은 GitHub의 주요 수익원이고 Microsoft의 AI 전략과 깊이 연결되어 있다.
“코딩 에이전트 경쟁은 포기하고 플랫폼에 집중하라”는 것은 단순한 전략 선택이 아니라
수십억 달러의 수익 모델을 재편하는 결정이다.
GitHub이 왜 지금의 방향을 선택할 수밖에 없는지에 대한 구조적 설명이 있었다면
조언의 현실성이 더 높아졌을 것이다.

## 인사이트

### 코딩 에이전트 시장의 승자는 에이전트가 아닌 플랫폼일 수 있다

모든 사람이 코딩 에이전트 레이스에 주목한다.
Claude Code와 Codex CLI 중 어느 것이 이길 것인가, Cursor가 살아남을 것인가.
그러나 에이전트 레이스의 진짜 수혜자는 다른 곳에 있을 수 있다.

역사적으로 기술 전환기의 승자는 새로운 기술 자체가 아니라 그 기술이 의존하는 인프라였다.
인터넷 붐에서 통신 회사들이 성장했고, 클라우드 컴퓨팅 확산에서 AWS가 성장했다.
스마트폰 시대에 앱 스토어 플랫폼이 성장했다.
코딩 에이전트 확산의 인프라 레이어는 코드가 저장되고 리뷰되고 배포되는 플랫폼이다.

GitHub이 에이전트 친화적 협업 플랫폼으로 포지셔닝에 성공한다면,
어느 에이전트가 시장을 지배하든 GitHub은 성장한다.
에이전트가 코드를 더 많이 생성할수록 리뷰와 협업 수요도 비례해 증가하기 때문이다.
문제는 이 전략이 단기 수익(Copilot 구독료)보다 낮아 보이는 장기 전략이라는 점에서
주주 압력을 받는 상장 기업이 선택하기 어렵다는 것이다.

### 플랫폼 신뢰는 성장 수치보다 빨리 무너진다

1억 8천만 명의 사용자와 2024년 역대 최고 성장률은 인상적인 숫자다.
그러나 저자가 지적한 것처럼 Mitchell Hashimoto 같은 영향력 있는 개발자의 공개 이탈은
수백만 명의 신규 가입과 질적으로 다른 신호다.

플랫폼 신뢰는 비선형으로 무너진다.
오랫동안 천천히 쌓이다가 특정 임계점에서 급격히 붕괴하는 패턴이 있다.
Twitter/X의 사례에서 보듯, 핵심 사용자층의 이탈이 시작되면 플랫폼의 가치 자체가 하락하고,
그 하락이 추가 이탈을 유발하는 음의 피드백 루프가 작동한다.
GitHub은 아직 그 임계점에 이르지 않았지만, 저자의 우려는 임계점에 가까워지는 신호를 감지한 것이다.

더 중요한 것은 GitHub의 신뢰 기반이 기술적 신뢰(안정성)와 커뮤니티 신뢰(오픈소스 중립성)
두 가지로 구성된다는 점이다.
장애는 기술적 신뢰를 훼손하지만 회복 가능하다.
반면 “GitHub이 개발자보다 Microsoft의 이익을 앞세운다”는 인식이 퍼지면 커뮤니티 신뢰가 무너지는데,
이것은 서비스 개선만으로는 회복되지 않는다.
저자가 “Microsoft의 분위기”를 언급하는 것은 기술적 문제가 아니라 이 두 번째 신뢰의 균열을 가리킨다.

### 에이전트 시대의 코드 리뷰는 새로운 직업을 만든다

저자가 제안하는 “에이전트와 협업하는 인터페이스”는 표면적으로는 UI 개선 요청이지만,
그 아래에는 더 근본적인 변화가 있다.
코딩 에이전트가 생산하는 코드의 양이 인간이 리뷰할 수 있는 양을 초과하기 시작했다.

이것은 소프트웨어 개발 조직의 구조를 바꾸는 압력이다.
코드를 작성하는 역할이 에이전트로 이동하면, 인간의 핵심 역할은 무엇을 만들지 결정하고
에이전트가 만든 것을 검증하는 것이 된다.
코드 리뷰는 줄어드는 것이 아니라 오히려 더 중요해진다.
에이전트가 생성한 코드는 빠르고 일관성 있지만, 비즈니스 맥락, 보안 함의, 장기 유지보수성을
인간 수준으로 판단하지 못한다.

GitHub이 이 변화를 잘 읽는다면, 에이전트 생성 코드의 리뷰를 돕는 AI 보조 도구,
대규모 diff의 맥락 파악 기능, 에이전트 기여와 인간 기여의 추적 등을 통해
리뷰어의 생산성을 높이는 플랫폼이 될 수 있다.
이것이 저자가 말하는 “청바지”의 구체적인 모습이다.
에이전트 레이스에서 직접 이기려 하지 않고, 에이전트 시대의 모든 개발자에게
필수적인 도구를 파는 것.
