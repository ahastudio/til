# gofumpt

<https://github.com/mvdan/gofumpt>

HN 토론: <https://news.ycombinator.com/item?id=34748475> (98점, 88개 댓글)

Lobste.rs 토론: <https://lobste.rs/s/x2ijwj/gofumpt_a_stricter_gofmt>

## 소개

gofumpt는 `gofmt`보다 더 엄격한 규칙을 적용하는 Go 코드 포매터다.
`gofmt`와 완전히 하위 호환되도록 설계되어, gofumpt로 포맷된 코드는
표준 `gofmt`로 재처리해도 추가 변경이 발생하지 않는다.
mvdan이 관리하며 GitHub에서 4,000개 이상의 스타를 보유하고 있다.
Go 1.25 이상이 필요하며, Go 1.26.0 기준의 `gofmt` 코드를 포크해 유지한다.

```bash
go install mvdan.cc/gofumpt@latest
```

## 추가 규칙

gofumpt는 표준 `gofmt` 위에 다음 규칙들을 추가로 적용한다.

**함수 본문 공백 제거**: 함수 본문의 첫 줄과 마지막 줄에 빈 줄을 허용하지 않는다.

**다중 줄 함수 파라미터**: 파라미터가 여러 줄에 걸칠 때 닫는 괄호와 여는 중괄호를
별도 줄에 배치해야 한다.

**복합 리터럴 일관성**: 복합 리터럴 내 하나의 요소라도 줄 바꿈이 있으면
모든 요소가 같은 패턴을 따라야 한다.

**다중 줄 함수 호출**: 호출이 여러 줄에 걸칠 때 닫는 괄호는 새 줄에서 시작해야 한다.

**빈 필드 목록**: 빈 인터페이스, 구조체, 함수 파라미터 목록은 한 줄 문법을 사용해야 한다.

**임포트 그룹화**: 표준 라이브러리 임포트는 최상단에 별도 그룹으로 분리해야 한다.

**케이스 절**: 짧은 케이스 절은 한 줄에 들어가야 한다.

**최상위 선언**: 다중 줄 선언은 빈 줄로 구분해야 하고, 단일 선언은 그룹 `var` 문법을 사용하지 않아야 한다.

**변수 선언**: 단순한 경우에는 `var` 선언 대신 짧은 대입 문법 `:=`을 사용해야 한다.

**단순화 기본 적용**: `-s` 플래그(불필요한 타입 생성자 제거)가 항상 활성화된다.

**8진수 리터럴**: Go 1.13+ 모듈에서 8진수 숫자에 `0o` 접두사를 사용해야 한다.

**주석 공백**: 지시어가 아닌 주석은 공백으로 시작해야 한다.

**복합 리터럴/타입 공백 제거**: 복합 리터럴과 타입 정의에서 선두/후미 빈 줄을 제거한다.

**불필요한 괄호 제거**: 표현식과 타입을 감싸는 불필요한 괄호를 제거한다.

## `-extra` 플래그 규칙

`-extra` 플래그를 사용하면 추가로 두 가지 규칙이 적용된다.

**파라미터 그룹화**: 인접한 동일 타입 파라미터를 그룹 문법으로 통합한다.

**Naked return 금지**: 명시적 반환값 없이 `return`만 쓰는 naked return 문을 금지한다.

## 에디터 연동

gopls를 통해 주요 에디터와 연동된다.

- **VS Code**: `“formatting.gofumpt”: true` 설정으로 gopls에서 활성화
- **GoLand**: gofumpt 실행 파일을 파일 와처(file watcher)로 구성
- **Vim/Neovim**: vim-go 또는 lspconfig의 gopls 설정
- **Emacs**: lsp-mode 또는 eglot 설정
- **Helix, Sublime Text, Zed**: gopls 연동을 통해 지원

## 분석

### gofumpt는 `gofmt`가 남긴 주관적 결정들을 메우는 2차 레이어다

`gofmt`는 2013년 Go 생태계에 도입됐을 때 혁명적이었다.
“포맷 논쟁은 없다. 도구가 결정한다”는 철학이 코드 리뷰에서 스타일 논의를 제거했다.
그러나 `gofmt`는 의도적으로 완전하지 않다.
빈 줄의 위치, 임포트 그룹화 방식, 변수 선언 스타일 등 수십 가지 결정을
개발자 또는 팀의 자율에 맡긴다.
`gofmt`가 하지 않는 것을 팀 컨벤션이나 린터로 보완하는 패턴이 생겼고,
gofumpt는 그 공백을 표준화된 도구로 채우는 시도다.

이 “2차 레이어” 패턴은 Go 생태계 외에도 반복된다.
Python에서 `autopep8` 이후 `black`이 등장한 것, JavaScript에서 `eslint --fix` 이후
`prettier`가 더 강력한 의견 주도형 포매터로 자리잡은 것이 같은 구조다.
의견 주도형(opinionated) 도구가 더 약한 도구 위에 쌓이는 계층화는
도구 생태계의 전형적인 진화 패턴이다.
gofumpt는 이 관점에서 Go 포매팅 생태계의 두 번째 세대다.

그러나 이 “2차 레이어” 패턴에 회의적인 시각도 있다.
`gofmt`의 진짜 가치는 공식 도구라는 권위에서 온다는 관점이다.[^rickette]
표준에서 벗어나는 것 자체가 “반 걸음 후퇴”라는 주장은,
포매터의 기술적 우수성보다 도구의 공식 지위가 더 중요하다고 본다.
더 날카로운 표현으로, gofumpt가 `gofmt`가 닫아놓은 “판도라의 상자를 다시 연다”는 지적도 있다.[^michiel]
`gofmt`가 해결한 것은 포맷팅 선택지의 기술적 우열이 아니라
“이 도구를 쓴다”는 선택지 자체를 제거한 것이었다.
gofumpt는 그 선택지를 다른 형태로 복원한다.
mvdan 자신도 gofumpt를 “실험”으로 정의하면서
일부 규칙이 공식 `gofmt`에 흡수될 가능성을 열어두고 있는데,[^usrme]
이는 gofumpt가 2차 레이어로 영구히 존재할 계획이 아님을 시사한다.

### 하위 호환성 제약이 gofumpt의 규칙 범위를 구조적으로 제한한다

gofumpt의 핵심 설계 원칙 중 하나는 “gofmt와 절대 의견이 다른 규칙을 추가하지 않는다”는 것이다.
gofumpt로 포맷된 코드를 `gofmt`로 재처리해도 변경이 없어야 한다.
이 원칙을 수식으로 표현하면 `gofmt(gofumpt(code)) == gofumpt(code)`가 성립해야 한다는 것이다.[^parhamn]
이 제약은 gofumpt를 매우 안전하게 도입 가능하게 만들지만,
동시에 gofumpt가 강제할 수 있는 규칙의 범위를 구조적으로 제한한다.

예를 들어, naked return 금지와 파라미터 그룹화는 `-extra` 플래그 뒤에 숨겨져 있다.
이것들은 `gofmt`가 반드시 수용해야 하는 형태는 아니기 때문에 기본 규칙에서 제외됐다.
이 결정은 “gofmt와 충돌하지 않는다”는 조건을 충족하면서도
더 강한 의견을 원하는 팀을 위한 탈출구를 마련한 타협이다.
그러나 이 이분법은 도구를 사용하는 팀이 또 다른 결정 — “extra를 사용할 것인가” — 을
해야 한다는 것을 의미한다.

### gopls 연동이 gofumpt의 실질적 채택 경로를 결정한다

gofumpt가 에디터 독립적 CLI 도구라는 사실보다 gopls(Go Language Server)와의 연동이
더 중요한 채택 요인이다.
gopls가 gofumpt를 공식적으로 지원하고 `“formatting.gofumpt”: true` 한 줄로
VS Code에서 활성화할 수 있다는 사실은,
이 도구의 확산이 개별 개발자의 선택이 아니라
언어 서버 생태계의 표준화를 통해 이루어짐을 의미한다.

이것은 Go 공식 도구체인이 gofumpt를 간접적으로 수용한 것으로 볼 수 있다.
gopls가 gofumpt를 지원하지 않았다면 에디터 연동이 복잡했을 것이고,
채택이 훨씬 더 마찰이 컸을 것이다.
gopls 연동을 통한 채택 경로는 gofumpt를 개인 도구에서
팀 표준으로 전환하는 최소 마찰 경로를 제공한다.

## 비평

### 포맷팅 패키지의 고정 복사본은 장기적으로 유지보수 부채를 만든다

gofumpt는 Go의 포맷팅 패키지(`go/format`, `go/doc/comment`)를
특정 Go 버전에서 고정 복사본으로 유지한다.
이 결정의 이유는 타당하다.
사용자의 설치된 Go 버전과 무관하게 일관된 출력을 보장하기 위해서다.
그러나 이 접근법은 숨겨진 유지보수 부채를 만든다.

Go 언어와 표준 라이브러리가 발전함에 따라 포맷팅 규칙도 변경될 수 있다.
gofumpt는 이 변경을 고정 복사본에 주기적으로 반영(merge)해야 한다.
이것은 지속적인 수작업 병합 작업을 의미하며,
mvdan이라는 단일 메인테이너에게 상당한 유지보수 부담을 준다.
“Go 1.26.0 기준 포크”라는 명시적 표현 자체가
이 부채가 이미 누적되고 있음을 보여준다.
고정 복사본 전략이 사용자 경험을 위한 선택이라면,
동시에 메인테이너 경험을 희생하는 선택이기도 하다.

### `gofmt`와의 하위 호환성 제약이 가장 유용한 일부 규칙을 선택적으로 만든다

naked return 금지는 많은 Go 개발자가 불필요하게 혼란스럽고
가독성을 낮춘다고 동의하는 패턴이다.
하지만 이 규칙은 기본 규칙이 아니라 `-extra` 플래그 뒤에 있다.
이유는 naked return이 `gofmt`와 완전히 호환되기 때문이다.
즉, hgofumpt가 “이것은 나쁜 관행이다”라고 동의하면서도
기본 규칙으로 강제하지 않는다.

이것은 “gofmt 하위 호환성” 원칙의 역설이다.
그 원칙은 gofumpt를 안전하게 만들지만,
동시에 가장 유의미한 코드 품질 개선 규칙들을 선택적 옵션으로 격하시킨다.
팀마다 `-extra`를 사용할지를 결정해야 하고,
그 결정이 일치하지 않으면 코드베이스가 split된다.
결과적으로 하위 호환성 제약이 해결하려는 문제 — “규칙을 하나로 통일한다” — 를
다른 층위에서 재생산한다.

### 함수 본문 빈 줄 제거는 개발자의 의도적 그룹화를 지운다

gofumpt의 가장 논쟁적인 규칙 중 하나는 함수 본문 첫 줄과 마지막 줄의 빈 줄을 제거하는 것이다.
`gofmt`는 개발자가 의도적으로 삽입한 공백을 보존한다.
이 공백은 관련 문장들을 시각적으로 묶거나 코드 블록 사이에 숨 쉴 공간을 만드는 수단이다.[^watt]
gofumpt는 이 판단을 단일 규칙으로 대체한다.

블록 경계에서 빈 줄을 제거하는 규칙은 특히 “코드가 압축된다”는 비판을 받는다.[^KevinMS]
포매팅 규칙이 단순히 스타일을 통일하는 것이 아니라
개발자의 가독성 판단 자체를 덮어쓴다는 지적이다.
`gofmt`가 보존하던 저자의 의도 — 어떤 문장이 함께 읽혀야 하는가 — 를
gofumpt는 더 강한 의견으로 대체한다.
“엄격함”이 언제나 “더 나은 가독성”과 동일하지 않음을 이 논쟁이 잘 보여준다.

### gofumpt의 규칙 추가 프로세스가 불투명하다

gofumpt가 특정 규칙을 채택하고 다른 규칙을 채택하지 않은 기준이
README에서 충분히 설명되지 않는다.
“gofmt가 동의하지 않는 규칙은 추가하지 않는다”는 원칙 외에
어떤 기준으로 규칙이 선정됐는지 알기 어렵다.
이 불투명성은 새 규칙을 제안하는 기여자에게 진입 장벽이 된다.

예를 들어, 임포트 그룹화 규칙은 왜 기본 규칙에 포함됐는가?
표준 라이브러리를 최상단에 두는 것은 관행이지만 `gofmt`가 강제하지 않는 결정이다.
반면 naked return 금지는 왜 `-extra` 뒤에 있는가?
두 결정 모두 “gofmt와 하위 호환”이라는 기준을 충족하는데
하나는 기본이고 하나는 선택적이다.
이 선택의 근거가 명시적으로 문서화되지 않으면
도구의 장기적 방향성과 커뮤니티 신뢰를 유지하기 어렵다.

줄 길이 제한이 여전히 없다는 사실도 같은 문제를 드러낸다.[^djha-skin]
수십 개의 규칙을 추가했음에도 줄 길이 — Go 코드 리뷰에서 가장 빈번하게 언급되는 스타일 결정 중 하나 — 에 대해서는 입장이 없다.
이것이 의도적 선택인지, 기술적 제약인지, 아직 결정하지 않은 것인지 README는 설명하지 않는다.

## 인사이트

### 코드 포매터는 기술적 도구가 아니라 사회적 계약이다

`gofmt`가 해결한 문제는 사실 기술적 문제가 아니었다.
들여쓰기에 탭을 쓸지 스페이스를 쓸지, 중괄호를 어디에 놓을지는
컴파일러가 상관하지 않는 문제다.
그것은 팀 내 코드 리뷰에서 발생하는 사회적 마찰이었다.
`gofmt`는 그 마찰을 기술적으로 해결했다 — 모든 사람이 동일한 도구에 위임함으로써.

gofumpt가 존재한다는 사실은 `gofmt`가 해결한 사회적 합의가 여전히 불완전하다는 것을 보여준다.
함수 본문 첫 줄의 빈 줄, 임포트 그룹화, 변수 선언 스타일은
`gofmt` 이후에도 코드 리뷰에서 논쟁의 여지가 남아 있었다.
gofumpt는 그 잔여 마찰을 추가적인 사회적 계약 — “우리 팀은 gofumpt를 사용한다” — 으로
해결하려는 시도다.
포매터는 코드를 변환하는 도구지만, 그것이 실제로 해결하는 것은
개발팀 내부의 규범 충돌이다.

### 의견 주도형 도구의 생태계 위치는 공식 도구체인과의 거리에 따라 결정된다

gofumpt의 흥미로운 특징은 그것이 비공식 도구임에도 불구하고
공식 gopls를 통해 에디터 생태계에 완전히 통합됐다는 점이다.
이것은 Go 팀이 gofumpt를 묵인하는 수준을 넘어
실질적으로 지원하고 있음을 의미한다.

이 패턴은 언어 생태계가 의견 주도형 도구를 흡수하는 일반적 방식을 보여준다.
도구가 충분한 채택률을 달성하면 언어 서버나 공식 도구체인이 통합 경로를 제공하고,
그것이 다시 채택을 가속화하는 선순환이 발생한다.
Python의 `black`이 VS Code Python Extension에 기본 포매터 옵션으로 들어간 것,
Rust의 `rustfmt`가 공식 도구체인의 일부가 된 것이 같은 패턴이다.
gofumpt는 이 과정의 중간 어딘가에 있다.
공식 도구체인의 일부는 아니지만, 공식 언어 서버가 지원하는 사실상 표준(de facto standard)이다.

### 포맷팅 규칙의 복잡성은 코드베이스 크기에 따라 비선형적으로 증가한다

개인 프로젝트나 소규모 팀에서 gofumpt의 추가 규칙은 마찰 없이 작동한다.
모든 개발자가 같은 에디터 설정을 쓰고, CI에서 포맷 검사를 실행하면 된다.
그러나 수십 명 이상의 개발자가 참여하는 대규모 코드베이스에서는
포맷팅 도구 변경이 대규모 diff를 만든다.

gofumpt를 기존 코드베이스에 도입하면 수천 개의 파일이 변경된다.
이 변경은 논리적 의미가 없고, `git blame`을 오염시키며,
진행 중인 브랜치들과의 머지 충돌을 만든다.
대규모 코드베이스에서 포맷팅 도구 전환의 실질적 비용은
규칙의 장점보다 훨씬 클 수 있다.
이 전환 비용이 대기업 코드베이스에서 gofumpt의 채택이 느린 주요 이유 중 하나다.
gofumpt를 새 프로젝트에 처음부터 도입하는 것과
기존 코드베이스에 도입하는 것은 완전히 다른 결정이다.

개발자 개인이 여러 프로젝트를 오가는 경우에도 파편화가 발생한다.
어떤 프로젝트는 `gofmt`, 다른 프로젝트는 gofumpt를 사용하면
에디터 설정을 전환해야 하고 습관화된 코드 스타일이 충돌한다.[^leighmcculloch]
`gofmt`가 만들어낸 “어디서나 동일한 Go 코드”라는 기대감이
gofumpt 도입으로 인해 퇴행하는 경험이다.
이 파편화가 개인 수준에서 누적되면 gofumpt가 해결하려는 “스타일 논쟁 제거”와
정반대 효과를 만들어낸다.

### `-extra` 이분법은 “의견 주도형 도구”의 근본적 긴장을 드러낸다

의견 주도형(opinionated) 도구의 가치는 결정을 제거하는 것이다.
팀이 규칙에 대해 논쟁하는 대신 도구에 위임한다.
그러나 `-extra` 플래그의 존재는 이 가치 제안을 부분적으로 무효화한다.
기본 gofumpt를 쓸 것인가, `-extra` gofumpt를 쓸 것인가는
여전히 팀이 결정해야 하는 질문이다.

이것은 의견 주도형 도구가 항상 해결하기 어려운 딜레마를 반영한다.
완전히 의견 주도적이 되면 일부 사용자의 컨벤션과 충돌해 채택이 줄어든다.
선택지를 많이 주면 의견 주도형이라는 정체성이 약해진다.
gofumpt의 `-extra` 이분법은 이 딜레마에 대한 실용적 타협이지만,
“규칙을 하나로 통일한다”는 철학의 부분적 후퇴이기도 하다.
더 근본적인 질문은 “얼마나 의견 주도적이어야 좋은가”이고,
도구마다 이 균형점을 다르게 찾는다.
gofumpt는 `gofmt`보다는 더 의견 주도적이지만 `black`(Python)보다는 덜하다.
이 스펙트럼에서의 위치 자체가 하나의 설계 결정이다.
