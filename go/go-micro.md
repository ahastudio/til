## 소개

Go Micro는 Go로 에이전트, 서비스, 워크플로를 만드는 통합 런타임 플랫폼이다.
스스로를 “Go를 위한 에이전트 하네스(Agent Harness)”라고 정의하며,
모델, 메모리, 도구, 계획/위임, 가드레일, 실행 미들웨어를 모든 에이전트에 제공한다.
GitHub 스타 23,000개 이상을 보유하고 있으며 Apache 2.0 라이선스로 배포된다.
Anthropic과 OpenAI가 스폰서로 참여하고 있다.

## 주요 기능

**에이전트 하네스** — 에이전트마다 모델, 메모리, 도구, 플랜/위임, 가드레일, 실행 미들웨어를 일관되게 제공한다.

**서비스를 도구로** — Go로 서비스를 직접 작성하거나 프롬프트로 생성할 수 있고, 엔드포인트가 자동으로 타입이 지정된 도구가 된다.

**내구성 있는 워크플로(Durable Flows)** — 결정론적 작업은 체크포인트가 있는 고정 코드 경로로, 동적 작업은 에이전트에 위임하는 방식으로 구분된다.

**MCP 게이트웨이** — 서비스 엔드포인트가 모델 컨텍스트 프로토콜(MCP)을 통해 AI가 호출 가능한 도구로 동작한다.

**A2A 게이트웨이** — 에이전트가 A2A(Agent-to-Agent) 프로토콜을 통해 다른 프레임워크의 에이전트와도 탐색·호출될 수 있다.

**플러거블 아키텍처** — 모든 추상화가 Go 인터페이스로 구현되어, 코드 변경 없이 어떤 컴포넌트든 교체 가능하다.

## CLI

```bash
micro new     # 서비스나 에이전트 스캐폴딩
micro run     # 핫 리로드와 인터랙티브 콘솔로 서비스 실행
micro chat    # 터미널 기반 에이전트 상호작용
```

## 분산 시스템 인프라

서비스 디스커버리, RPC, 이벤트(pub/sub), 상태 관리, 인증, 관찰 가능성(observability), 배포를 내장 지원한다.
로컬 개발 시 핫 리로드가 지원되며, API 게이트웨이와 에이전트 플레이그라운드, 프로덕션 대시보드를 제공한다.

## 분석

### 마이크로서비스 프레임워크에서 에이전트 하네스로의 피벗

Go Micro는 원래 Go로 마이크로서비스를 구축하는 프레임워크로 출발했다.
23,000개 이상의 GitHub 스타는 대부분 그 이전 시대에 축적된 것이다.
현재 홈페이지는 “에이전트 하네스”를 전면에 내세우며 AI 에이전트 오케스트레이션 플랫폼으로의 재포지셔닝을 선언한다.
Anthropic과 OpenAI의 스폰서십은 이 방향 전환이 마케팅이 아닌 기술적 실체를 가지고 있다는 신호다.

이 피벗이 흥미로운 이유는 프레임워크가 취한 전제 때문이다.
“에이전트는 분산 시스템이다”라는 명제 아래, 서비스 디스커버리, RPC, 메시징, 상태 관리 같은
마이크로서비스 인프라를 에이전트 런타임의 기반으로 재활용한다.
이 관점은 기존 웹 서비스 프레임워크가 에이전트를 단순한 API 클라이언트로 보는 방식과 근본적으로 다르다.

### MCP와 A2A의 동시 지원이 가진 전략적 의미

Go Micro가 MCP와 A2A를 모두 게이트웨이로 제공하는 것은 단순한 기능 추가가 아니다.
MCP는 LLM이 도구를 호출하는 표준이고, A2A는 에이전트 간 통신 프로토콜이다.
두 프로토콜을 동시에 지원한다는 것은 Go Micro가 모델 레이어와 에이전트 레이어 양쪽에서 동시에 작동한다는 뜻이다.

실용적 함의는 이렇다.
기존 Go Micro 서비스는 코드 변경 없이 MCP 도구로 노출될 수 있고,
멀티 에이전트 시스템에서 각 에이전트는 A2A로 서로를 탐색하고 호출할 수 있다.
이것은 기존 레거시 서비스 인프라를 AI 에이전트 생태계에 편입시키는 경로를 제공한다.

### 플러거블 아키텍처가 실제 의미하는 것

“모든 추상화가 Go 인터페이스”라는 설계 원칙은 Go의 문화적 규범과 일치한다.
Go 생태계에서 인터페이스 기반 설계는 테스트 가능성과 교체 가능성의 표준 방법이다.
그러나 플러거블 아키텍처는 동시에 복잡성의 비용을 수반한다.
“어떤 컴포넌트든 교체 가능”하다는 말은 “기본 선택이 올바르지 않을 수 있다”는 의미도 된다.

Durable Flows의 존재는 이 긴장을 잘 보여준다.
결정론적 작업과 비결정론적 작업을 명시적으로 구분하는 것은 올바른 설계이지만,
이 경계를 어디에 그을지는 개발자가 판단해야 한다.
Temporal이나 Durable Execution 분야에서 이미 이 문제가 얼마나 어려운지가 드러났다.

ainar-g[^ainar-g]는 “Go는 관용적으로 프레임워크에 반하고 라이브러리를 지향하는 언어”라고 지적했다.
프레임워크 대신 인터페이스 패키지를 정의해 에코시스템이 그 위에 형성되게 해야 한다는 것이다.
이 비판은 2019년 당시 마이크로서비스 프레임워크 시절의 것이지만,
현재의 “모든 추상화가 Go 인터페이스”라는 설계 원칙은 이 지적을 정면으로 수용한 결과로 볼 수 있다.
피벗 이후 Go Micro는 프레임워크가 아닌 인터페이스 기반 하네스로 재정의되었다는 점에서
커뮤니티의 비판이 방향 전환의 동력 중 하나였을 가능성이 있다.

## 비평

### 에이전트 하네스라는 정의가 범위를 모호하게 만든다

“에이전트 하네스”는 비유적 표현이지 기술적 명세가 아니다.
홈페이지는 모델, 메모리, 도구, 계획, 위임, 가드레일, 미들웨어를 나열하지만
이것들이 어떻게 결합되어 동작하는지의 구체적 메커니즘은 홈페이지에서 보이지 않는다.
“에이전트에 무엇이 들어가는지”를 나열하는 것과 그것을 실제로 어떻게 구현하는지는 다른 문제다.

특히 “가드레일(guardrails)”이 무엇을 의미하는지가 불분명하다.
AI 안전성 문맥의 가드레일인지, 아니면 일반적인 오류 처리 미들웨어인지가 구분되지 않는다.
프레임워크가 에이전트 안전성 문제를 어떻게 다루는지는 실제 사용자에게 중요한 질문인데
홈페이지 수준에서는 답을 찾기 어렵다.

### 23,000 스타는 현재 프레임워크를 반영하지 않는다

GitHub 스타 수는 프레임워크의 현재 상태가 아니라 과거의 명성을 반영한다.
Go Micro의 스타 대부분은 마이크로서비스 프레임워크 시절에 쌓인 것으로,
현재의 에이전트 하네스 방향을 지지하는 커뮤니티 규모와 같지 않다.
이 수치를 신뢰 지표로 제시하는 것은 선택적 정보 제공이다.

더 중요한 지표는 최근 커밋 활동, 이슈 응답 속도, 그리고 에이전트 기능을 실제로 사용하는
프로덕션 사례의 수다.
스타 수와 현재 유지보수 상태 사이의 간격이 얼마나 큰지는
홈페이지만으로는 판단할 수 없다.

신뢰성 문제는 역사적 전례도 있다.
lhopki01[^lhopki01]은 “우리는 이것을 쓰고 있었는데, 저장소가 이름을 바꾸고 이전되면서 온갖 문제가 생겼다”고 경고했다.
저장소 불안정이 실제 사용자에게 직접적 피해를 준 기록이 있는 프로젝트라는 점은,
신뢰 지표로 스타 수만 제시하는 것의 한계를 더욱 두드러지게 한다.
bradhe[^bradhe]도 “서비스 메시가 확산되는 세계에서 이것이 어디에 위치하는지 확신할 수 없다”며
포지셔닝의 모호함을 지적했다.
이 질문은 마이크로서비스 시절에도 해소되지 않았고,
에이전트 하네스로 피벗한 현재의 포지셔닝도 같은 구조적 의문을 다시 유발한다.

boreq[^boreq]는 "지속적인 인터페이스 변경과 불필요한 복잡성" 때문에 회사 전체가 Go Micro를 버리고 plain gRPC로 마이그레이션을 완료했다고 전했다.
저장소 이름 변경보다 더 직접적인 포기 사례다.
스타 수와 실제 프로덕션 채택률 사이의 간극이 단순한 추측이 아님을 보여주는 구체적 경험담이다.

라이선스 불안정도 이 맥락에서 빼놓을 수 없다.
Micro 3.0 출시 당시 dguaraglia[^dguaraglia]는 "매우 이례적이고 장황한 라이선스"와 새로 추가된 "환경(environment)" 요건이 "복잡성 비대화(creeping complexity)"처럼 느껴진다고 경고했다.
danielrheath[^danielrheath]는 당시의 Polyform Shield 라이선스 선택이 AWS의 무임승차를 막는 것이 아니라 "원한다면 비용을 치르게 만드는" 전략이라고 설명했다.
프로젝트는 이후 Apache 2.0으로 돌아왔지만, 라이선스 변동 자체가 방향성 불안정의 또 다른 기록이다.

### Durable Flows의 차별화 근거가 약하다

결정론적 워크플로와 에이전트 위임을 결합하는 아이디어는 이미 Temporal, Prefect, Durable Task 같은
워크플로 오케스트레이션 도구들이 탐색해온 영역이다.
Go Micro의 Durable Flows가 이 도구들과 어떻게 다른지, 무엇이 더 낫거나 나쁜지에 대한 설명이 없다.

특히 에이전트 위임이 비결정론적이라는 점은 내구성 있는 워크플로의 핵심 속성인
재실행 가능성(replayability)과 충돌할 수 있다.
에이전트가 같은 입력에 다른 출력을 낼 때 워크플로를 어떻게 체크포인트하고 재개하는지는
구현 세부 사항이 아니라 아키텍처 핵심 문제다.
이 질문에 대한 답이 홈페이지에는 없다.

## 인사이트

### AI 에이전트는 분산 시스템 문제를 재발명하고 있다

Go Micro의 피벗이 가장 잘 포착하는 것은 AI 에이전트 분야의 근본적 아이러니다.
멀티 에이전트 시스템이 직면하는 문제들 — 서비스 탐색, 메시지 전달, 상태 동기화, 장애 복구 — 은
분산 시스템 커뮤니티가 20년 넘게 다뤄온 문제들과 구조적으로 동일하다.
많은 AI 프레임워크들이 이 교훈을 무시하고 처음부터 다시 만드는 실수를 반복하는 반면,
Go Micro는 기존 마이크로서비스 인프라 위에 에이전트 레이어를 얹는 전략을 택했다.

이 접근은 생산성과 신뢰성 측면에서 실질적 이점을 줄 수 있다.
서비스 디스커버리와 RPC가 이미 검증된 구현 위에 에이전트 오케스트레이션이 구축되면,
에이전트 프레임워크가 처음부터 해결해야 하는 인프라 문제의 범위가 줄어든다.
그러나 이 전략은 Go라는 언어와 마이크로서비스 패러다임을 전제로 하기 때문에
진입 장벽도 함께 높아진다는 역효과가 있다.

runlevel1[^runlevel1]이 2019년에 제기한 mDNS 서비스 디스커버리의 프로덕션 한계 문제는
이 진입 장벽의 성격을 잘 보여준다.
“mDNS는 로컬 개발에는 흥미롭지만 프로덕션에서는 좋지 않다 — 대부분의 클라우드 공급자가 멀티캐스트를 지원하지 않아
mDNS가 작동하지 않는다”는 지적이다.
당시에도 플러그인을 통한 다른 서비스 디스커버리 지원이 있었고,
이것이 바로 플러거블 아키텍처의 필요성을 방증한다.
분산 시스템의 인프라 선택은 기본값이 틀렸을 때의 비용이 크기 때문에,
“어떤 컴포넌트든 교체 가능하다”는 원칙이 있어도 올바른 기본값을 제공하는 것은 별개의 과제다.

### MCP 표준화가 에이전트 프레임워크 경쟁 구도를 바꾼다

MCP가 사실상 표준으로 자리 잡으면서, 에이전트 프레임워크의 경쟁 축이 바뀌고 있다.
과거에는 프레임워크가 자체 도구 호출 방식을 정의했기 때문에
프레임워크 선택이 곧 에코시스템 선택이었다.
MCP 게이트웨이를 통해 기존 서비스를 표준 도구로 노출할 수 있다면,
프레임워크 종속성이 줄어들고 도구 레이어에서의 이식성이 높아진다.

이것은 Go Micro에게 양면적 현실이다.
MCP 지원이 진입 장벽을 낮추는 한편, MCP 표준화는 Go Micro만의 차별점도 약화시킨다.
다른 언어의 에이전트 프레임워크들도 같은 MCP 게이트웨이를 제공하면
“Go 서비스를 AI 도구로 만드는 가장 쉬운 방법”이라는 포지션이 유일하지 않게 된다.

### 레거시 코드베이스의 AI 편입이 진짜 시장이다

Go Micro가 암묵적으로 노리는 시장은 새로 에이전트를 만드는 사람이 아니라
기존 Go 서비스를 AI 생태계에 연결하려는 팀이다.
수천 개의 기업이 Go로 작성된 내부 서비스를 운영하고 있으며,
이 서비스들을 LLM이 호출 가능한 도구로 만들기 위해서는
API 명세를 다시 작성하거나 별도의 래퍼를 만들어야 한다.

MCP 게이트웨이가 이 과정을 자동화한다면, 실질적인 채택 동기는 새 프로젝트가 아닌
기존 레거시 편입에서 나올 가능성이 높다.
그러나 이 시나리오는 Go Micro가 기존 코드베이스에 얼마나 비침투적으로 통합될 수 있는지에 달려 있다.
플러거블 아키텍처를 강조하는 설계 철학은 이 방향과 일치하지만,
기존 서비스의 마이그레이션 경로가 얼마나 매끄러운지는 별도 검증이 필요하다.
