# Go에서 과도한 nil 포인터 검사

원문: <https://konradreiche.com/blog/excessive-nil-pointer-checks-in-go/>

HN 토론: <https://news.ycombinator.com/item?id=48583407> (117점, 95개 댓글)

## 요약

Konrad Reiche가 작성한 이 글은 Go 코드에서 nil 검사가 잘못된 위치에 놓이는 패턴을
분석하고, 올바른 nil 검사의 위치와 이유를 설명한다.
방어적 프로그래밍의 이름으로 추가된 nil 검사가 실제로는 코드의 불변식(invariant)을
무너뜨리고 오류를 조용히 삼키는 문제를 만든다는 것이 핵심 주장이다.

저자는 세 가지 패턴을 분석한다.
첫째, 메서드 내부에서 의존성(dependency)을 nil 검사하는 것이다.
`RateLimiter`가 Redis 클라이언트를 nil 검사하는 코드는 “왜 nil Redis 클라이언트로 RateLimiter를 만드는가”라는
더 근본적인 질문을 피해간다.
nil이라면 그 오류는 이미 생성 시점에 발생한 것이므로, 지금 검사하는 것은 그 오류를 처리하는 것이 아니라
오류가 발생한 상태로 계속 실행하는 것을 정상으로 받아들이는 것이다.

둘째, 생성자에서 nil을 검사해 에러를 반환하는 것도 개선이지만 완전한 해결책이 아니다.
생성자는 nil 포인터가 전달된 곳이 아니므로, 그 오류는 여전히 초기화 지점에서 처리되어야 한다.
오류를 발견한 즉시 처리하고, 초기화 실패 후 nil 포인터로 계속 진행하지 않아야 한다.

셋째, 요청 범위 데이터(request-scoped data)의 nil 검사 위치다.
HTTP 핸들러의 깊은 내부에서 `req`가 nil인지 검사하는 것은 잘못된 위치다.
`req`는 이미 외부 경계(HTTP 핸들러, RPC 디스패치, 큐 컨슈머)에서 시스템에 진입했고,
그 경계에서 검증되었어야 한다.

## 분석

### nil 검사의 위치는 불변식(invariant) 설계의 가시적 증거다

저자의 핵심 통찰은 “nil 검사가 잘못된 위치에 있다는 것은 코드가 자신의 객체 계보를 잃었다는 신호”라는 점이다.
포인터가 어디서 왔는지, 누가 초기화할 책임이 있는지, 어떤 불변식이 nil을 불가능하게 해야 하는지를
알지 못할 때, 모든 곳에 nil 검사가 등장한다.

데이터베이스 제약(constraint)의 비유가 정확하다.
`NOT NULL` 제약은 모든 쿼리에서 null 여부를 재검사할 필요를 없애는 데이터 계층의 불변식이다.
잘 설계된 코드에서 타입 불변식은 같은 역할을 한다.
생성 시점에 한 번 확립된 불변식은 이후 모든 코드가 그것을 신뢰할 수 있게 한다.
불변식 없이 작성된 코드는 모든 함수에서 불변식을 재확인해야 하는 부담을 갖는다.

### 조용한 실패(silent failure)와 명시적 실패(loud failure)의 비대칭 비용

저자가 제시하는 “안전하게 느껴지는 선택 vs. 실제로 안전한 선택”의 구분은 직관에 반한다.
개발자가 nil 검사로 에러를 조용히 삼킬 때, “시스템을 죽이지 않았다”는 느낌을 얻는다.
그러나 저자의 분석대로 조용한 실패는 세 가지 비용을 만든다.
발생 사실을 알 수 없고, 원인과 증상 사이에 시간적 거리가 생기며,
증상을 볼 때쯤 원인을 추적하기 어려워진다.

HN 댓글의 zoogeny는 이 긴장에 현실적 맥락을 추가했다.[^zoogeny]
LLM 에이전트가 코드를 수정하는 시대에 로컬 추론(local reasoning)의 중요성이 더 커졌다는 것이다.
자동 생성 코드가 에러를 조용히 삼키도록 작성되면, 이후 리뷰어와 다른 에이전트가
코드의 실제 상태를 추론하기 더 어려워진다.

### 외부 계층 vs. 내부 계층의 구분은 시스템 설계의 신뢰 경계다

저자의 외부/내부 계층 프레임워크는 신뢰 경계(trust boundary)를 코드 구조로 번역한 것이다.
외부 계층(HTTP 핸들러, RPC 수신, 큐 컨슈머)은 외부에서 들어오는 신뢰할 수 없는 데이터를 받는다.
이 지점에서 nil 검사, 범위 검사, 타입 검증이 수행되어야 한다.
경계를 통과한 데이터는 내부 타입으로 변환되고, 내부 계층에서는 신뢰된 값으로 취급된다.

이 패턴은 도메인 주도 설계(DDD)의 안티코럽션 계층(anti-corruption layer)이나
함수형 프로그래밍의 파싱(parsing)을 유효성 검사(validation)보다 우선하는 원칙과 정확히 일치한다.
경계에서 한 번 파싱해 내부 타입을 만들고, 내부에서는 그 타입을 신뢰하는 것이다.
Go의 타입 시스템이 이 신뢰를 컴파일 타임에 보장하지 못하기 때문에
이 규율은 코드 관행으로 유지되어야 한다는 것이 이 글의 한계이자 핵심이다.

## 비평

### 이 글의 처방은 타입 시스템이 뒷받침하지 않는 규율을 요구한다

HN 댓글에서 가장 공감을 얻은 비판은 diarrhea의 지적이다.[^diarrhea]
“이것은 언어가 옵셔널성(의미론적 개념)과 포인터(기계적 개념)를 혼동한 데서 오는 문제다.
Go에서 '항상 존재하는 오브젝트 참조'는 표현 불가능하다.”
Rust의 `&T`(항상 존재하는 참조)와 `Option<&T>`(없을 수 있는 참조)의 구분이 컴파일 타임에
강제되는 것과 달리, Go에서는 이 구분이 코드 관행과 팀 합의에만 의존한다.

nu11ptr도 같은 맥락을 짚었다.[^nu11ptr]
“불가능하다고 주장하는 상태를 타입 시스템이 표현하지 못한다면,
개발자는 코드베이스 전체에서 nil 가능성을 머릿속으로 추적해야 한다.”
저자의 조언을 따르는 것은 좋은 습관이지만,
컴파일러가 강제하지 않는 규율은 코드베이스가 커지고 팀이 바뀔수록 희석된다.

### 저자의 처방이 실제로 적용 가능한 범위는 그린필드 코드베이스로 제한된다

galkk의 댓글은 현실적인 반론을 제기한다.[^galkk]
생성자를 퍼블릭 API로 보면, 생성자에서 nil을 검사하는 것은 경계 테스트의 표준 관행이다.
호출자가 nil을 전달하는 것은 그 자체로 잘못이지만, 퍼블릭 API는 잘못된 입력에 명확한 오류를 반환해야 한다.
Joker_vD는 저자가 코드베이스 전체를 통제한다고 가정하는 것 같다고 지적했다.[^Joker_vD]
외부 라이브러리나 팀 간 인터페이스에서는 “이 값은 항상 nil이 아니다”라는 불변식을
생성자 검사 없이 신뢰하는 것이 더 위험할 수 있다.

레거시 코드베이스에서 “불변식을 확립하고 검사를 줄여나가자”는 접근은
상당한 리팩토링을 요구하며, 그 과정에서 회귀가 발생할 수 있다.
저자의 조언은 원칙적으로 옳지만, 기존 코드베이스에 적용하는 방법론은 글에서 거의 다루지 않는다.

### “생성된 코드에서 더 자주 보인다”는 관찰은 충분히 탐구되지 않는다

저자는 서두에 “이 패턴을 생성된 코드에서 더 자주 본다”고 언급하지만, 이 관찰의 함의를 탐구하지 않는다.
LLM이 생성하는 코드가 과도한 nil 검사를 선호하는 이유는 명확하다.
훈련 데이터에 방어적으로 작성된 코드가 많고, 안전성을 위해 “더 많이 검사하라”는 일반 원칙을 내면화했기 때문이다.
그러나 LLM은 특정 코드베이스의 불변식을 이해하지 못하고, 맥락 없이 패턴을 적용한다.

이 관찰이 중요한 이유는 AI 코딩 도구의 사용이 늘수록
이 패턴이 코드베이스에 더 빠르게 퍼질 수 있다는 것이다.
저자가 지적한 문제의 규모가 앞으로 커질 것이라는 예측 아래,
이 글이 제시하는 원칙들은 더 시급해진다.

## 인사이트

### nil은 Go에서 세 가지 다른 개념을 하나의 값으로 표현하는 오버로딩된 개념이다

jspdown의 댓글이 정리한 분류는 이 문제의 뿌리를 잘 보여준다.[^jspdown]
Go에서 포인터는 옵셔널성(값이 없을 수 있음), 가변성(변경 가능한 참조가 필요함),
메모리 최적화(복사 비용을 피함)라는 세 가지 다른 의도로 사용된다.
nil 검사의 필요성은 이 세 가지 중 첫 번째 경우에만 해당되지만,
Go 타입 시스템은 이 구분을 표현하지 못한다.

이 오버로딩의 결과로 코드를 읽는 사람은 포인터를 볼 때마다 “이것이 nil일 수 있는가”를
묻어야 한다.
타입 시스템이 답을 줄 수 없으므로 문서, 코드 관행, 팀 규약이 그 역할을 대신해야 한다.
이것이 usrnm이 지적한 “계약 프로그래밍 대 방어적 프로그래밍” 논쟁이 Go에서 특히 첨예한 이유다.[^usrnm]

### 불변식 기반 코딩으로의 전환은 테스트 전략을 바꾼다

저자의 접근을 따르면, 생성자에서 에러를 반환하는 대신 초기화 지점에서 오류를 처리하고
내부 코드에서 nil 검사를 제거한다.
이 변화는 테스트 전략에도 영향을 준다.
nil 처리 분기가 없어지므로 nil 입력에 대한 단위 테스트가 불필요해지는 반면,
초기화 로직의 오류 처리와 경계 계층의 검증이 테스트의 중점이 된다.

Fire-Dragon-DoL의 실용적 가이드라인은 이 전환의 구체적 출발점을 제시한다.[^fire_dragon]
포인터를 쓰지 않고 struct를 값으로 전달하며, 예외는 명시적 이유와 함께 주석을 달고,
옵셔널 파라미터에는 Null Object 패턴을 활용한다.
이 규칙들을 팀 내에서 합의하고 적용했더니 코드 품질이 크게 개선됐다는 증언은,
타입 시스템의 한계를 팀 규약으로 보완하는 현실적 접근이 효과가 있음을 보여준다.

### Go의 nil 문제는 Rust의 소유권 모델이 해결하려 한 문제의 절반이다

diarrhea의 Rust 비교는 언어 설계 트레이드오프의 핵심을 짚는다.
Rust의 `&T`와 `Option<&T>` 구분은 컴파일러가 nil 역참조를 불가능하게 만드는 대신,
`Option`을 다루는 코드의 복잡성을 프로그래머에게 전가한다.
Go는 반대 선택을 했다.
포인터를 단순하게 유지하고, nil 처리의 책임을 프로그래머에게 넘긴다.

이 트레이드오프는 언어의 목표와 연결된다.
Go는 “단순하고 배우기 쉬운 언어”를 지향했고, 타입 시스템의 복잡도보다 코드의 명시성을 우선했다.
Rust는 “메모리 안전성을 컴파일 타임에 보장하는 언어”를 지향해 복잡한 타입 시스템을 선택했다.
어느 쪽도 완전한 해결책이 아니며, 팀과 프로젝트의 특성에 따라 어느 트레이드오프가 더 합리적인지가 다르다.
이 글이 Go 특유의 문제처럼 다루지만, 실제로는 모든 null 허용 언어에서 반복되는 패턴이다.
