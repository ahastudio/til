# Solod: Go can be a better C

원문: <https://antonz.org/solod/>

HN 토론: <https://news.ycombinator.com/item?id=48895199> (208점, 152개 댓글)

Lobste.rs 토론: <https://lobste.rs/s/uqeo3m/solod_go_can_be_a_better_c>

## 요약

Solod(So)는 Go의 엄격한 부분 집합(subset)으로 작성한 코드를 읽기 좋은 C11로 변환하는
트랜스파일러이자 언어다.
저자는 “일반 Go 코드를 쓰면 읽기 좋은 C11을 얻는다”를 핵심 가치로 제시한다.
새 언어를 배우지 않고도 시스템 수준 제어를 얻는 것이 목표다.

핵심 특징은 런타임 없음이다.
가비지 컬렉션, 참조 카운팅, 숨겨진 할당이 없다.
모든 것이 기본적으로 스택에 할당되며 힙은 `so/mem` 같은 표준 라이브러리 패키지를 통해
명시적으로만 접근한다.
구조체, 메서드, 인터페이스, 슬라이스, 맵, 다중 반환값, `defer`를 지원한다.

C 상호운용성은 오버헤드 없이 동작한다.
`so:include`로 C 헤더를 가져오고 `so:extern`으로 외부 C 타입과 함수를 선언하면
CGO 없이 C를 직접 호출한다.
So 코드에서 C를, C 코드에서 So를 모두 호출할 수 있다.

Go 도구 생태계를 그대로 활용할 수 있다는 점이 차별점이다.
구문 강조, LSP, 린팅, `go test`가 추가 설정 없이 작동한다.
저자는 설계 철학을 “Go 개발자가 새 언어 없이 시스템 수준 제어를 얻는 방법”으로 규정하고,
“기능이 적을수록 항상 더 낫다”는 원칙을 명시했다.

현재 지원하지 않는 것은 함수 리터럴, 클로저, 이터레이터다.
제네릭은 제한적으로 지원한다.
동시성은 언어에 내장하지 않고 표준 라이브러리로 제공한다.
저자는 “현재 프로덕션 용도가 아니지만 취미 프로젝트에서 실험하기를 권장한다”고 밝혔다.

## 분석

### Go subset이라는 전략의 핵심 이점

Solod가 완전히 새 언어를 만들지 않고 Go subset을 선택한 이유는 도구 생태계 재사용이다.
새 언어를 만들면 파서, 포매터, LSP, 린터, 테스트 프레임워크를 모두 새로 구현해야 한다.
Go subset이면 기존 Go 도구가 Solod 코드를 이미 이해한다.
`gofmt`로 포매팅하고, gopls로 자동 완성을 받고, `go test`로 테스트한다.

이 접근은 채택 장벽을 낮추는 효과도 있다.
Go 개발자라면 에디터 설정 없이 즉시 Solod 코드를 작성하고 읽을 수 있다.
기존 코드베이스에서 일부 파일만 Solod로 전환해도 나머지 Go 코드와 동일한 리뷰 도구가 적용된다.

단, “Go subset”이라는 표현이 과장일 가능성이 있다.
클로저, 함수 리터럴, 이터레이터 같은 Go의 핵심 기능이 없다면, 실제 Go 코드를 Solod로
그대로 이전하는 것은 대부분 불가능하다.
새로 작성할 때만 “Go처럼 보이는 코드”를 쓸 수 있다는 것에 가깝다.

Lobste.rs에서 alurm은 이 “엄격한 부분 집합” 주장에 구체적인 반례를 제시했다.[^alurm]
Go의 `any` 타입, 빈 슬라이스의 nil 여부, `defer` 실행 시맨틱이 Go와 미묘하게 다르게
동작한다는 것이다.
“같은 코드처럼 보이지만 다르게 동작한다”는 것은 단순한 기능 부재보다 더 위험한 문제다.
Go 개발자가 익숙한 패턴으로 코드를 작성했는데 다른 결과가 나온다면, 이것은 “Go 없이 Go를 쓰는
것”이 아니라 새 언어를 익혀야 하는 것과 다르지 않다.

또한 kristof는 Solod가 “정교한 컴파일러가 아니라 단순 트랜슬레이터”임을 지적했다.[^kristof]
이스케이프 분석 같은 분석이 없어 스택 할당 구조체에 대한 포인터를 반환하면 런타임에
문제가 생길 수 있다는 것이다.
이것은 “Go처럼 쓰면 안전하다”는 기대를 깨는 중요한 제한이다.

### C 상호운용성 설계의 선택

CGO 없는 C 상호운용은 Go 생태계에서 오래된 골칫거리를 해결한다.
CGO는 컴파일 속도를 늦추고, 크로스 컴파일을 복잡하게 하며, 디버깅을 어렵게 한다.
Solod의 접근은 C 헤더를 직접 포함하고 So 코드를 C11로 트랜스파일하므로 이 레이어가 없다.

이 설계는 흥미로운 역방향 사용도 허용한다.
C 코드베이스에 Solod로 새 모듈을 작성하고 링크하는 것이다.
기존 C 프로젝트를 Rust로 점진적으로 마이그레이션하는 패턴과 유사하지만,
Go 문법에 익숙한 팀에게는 진입 장벽이 낮다.

### 스택 우선 메모리 모델의 실용성

Go의 핵심 편의 중 하나는 힙/스택을 컴파일러가 자동으로 결정하는 이스케이프 분석이다.
Solod는 이를 명시적으로 바꾼다.
기본이 스택이고 힙은 명시적 선택이다.

이 변화는 단순히 GC를 제거하는 것 이상의 의미가 있다.
개발자가 객체의 수명을 의식적으로 결정해야 한다.
Go에서 자연스럽게 쓰던 `make([]int, n)` 패턴이 Solod에서는 힙 할당을 명시하는
패키지 호출이 된다.
이것은 더 많은 제어를 주지만, Go를 선택한 이유 중 하나인 메모리 관리 편의성을
되돌리는 것이기도 하다.

## 비평

### “Go 개발자가 C를 쓰는 방법”의 실제 비용

저자가 내세운 장점은 “새 언어 없이”다.
그러나 클로저와 함수 리터럴이 없고, 힙이 명시적이고, 동시성이 없다면 이것은 Go가 아니다.
Go 개발자가 Solod 코드를 쓰려면 Go에서 자연스럽게 쓰던 패턴들을 의식적으로 피해야 한다.
그 비용이 “새 언어를 배우는 것”보다 작은지는 확실하지 않다.

Zig는 처음부터 시스템 언어로 설계됐고, Rust는 소유권 모델로 안전성을 보장한다.
Solod가 “Go처럼 보이는 C”라면, Go 개발자에게 더 친숙한 대안이 될 수 있지만
기존 시스템 언어 생태계에 비해 추가 가치를 증명해야 한다.

HN에서 pjmlp는 전제 자체에 의문을 제기했다.[^pjmlp]
“Go는 이미 C 설계자들이 직접 만든 더 나은 C다”라는 것이다.
Ken Thompson, Rob Pike, Robert Griesemer가 Go를 설계할 때 C, Plan 9, Alef, Limbo에서
얻은 교훈을 반영했다는 점에서, 이미 Go 자체가 더 나은 C를 목표로 했다는 주장이다.
이 관점에서 Solod의 목표인 “Go 문법으로 C를 쓰는 것”은 오히려 Go가 개선하려 했던 C의
복잡성으로 돌아가는 것이 아닌지 물을 수 있다.

### 출력이 읽기 좋은 C인 것의 실질적 의미

“읽기 좋은 C11을 출력한다”는 것이 실용적으로 중요한가는 맥락에 따라 다르다.
디버깅, 감사, C 코드베이스와의 병합 같은 상황에서는 가치 있다.
그러나 대부분의 실용적 사용에서 출력 C 코드를 직접 읽을 필요가 없다면 이 특성의
실용적 가치는 제한적이다.

“읽기 좋은 C”가 더 의미 있는 것은 생성 코드를 수동으로 최적화하거나 기존 C
코드베이스에 통합해야 할 때다.
이런 사용 사례를 명확히 타깃으로 한다면, 문서에서 더 구체적인 사례가 필요하다.

### 프로덕션 전 단계의 한계 투명성

저자가 “현재 프로덕션 용도가 아니다”라고 명시한 것은 정직하다.
그러나 이 단계에서 어떤 것이 불안정한지, 로드맵이 무엇인지, 유지 관리 의지가 얼마나
강한지는 명확하지 않다.
Go subset 컴파일러를 만드는 것이 쉽지 않고, 언어 규격도 아직 안정적이지 않을 수 있어
장기 투자로 Solod를 선택하기에는 정보가 부족하다.

ventana는 언어 트랜스파일러의 구조적 함정을 구체적 예로 보여줬다.[^ventana]
Go 코드에서 `register`라는 변수명을 쓰면 C에서는 예약어와 충돌한다.
이런 충돌을 해소하려면 `so_register` 같은 네임스페이스 프리픽스가 필요한데,
그러면 C 상호운용성이 복잡해진다.
“무한히 계속되는 양파 껍질 벗기기” 같은 문제라는 표현처럼, 언어 간 번역은 표면적인
해결보다 깊은 층의 불일치를 계속 드러낸다.
이것은 Solod가 성숙하는 과정에서 반드시 직면할 종류의 문제다.

## 인사이트

### 언어 설계 대신 생태계를 빌리는 전략

Solod의 접근은 언어 설계의 어려운 부분을 우회한다.
새 문법, 새 도구, 새 에코시스템을 만드는 대신 기존 Go 생태계를 빌린다.
이 “기생적 설계”는 효율적이지만 제약도 크다.
Go 언어가 변화하면 Solod의 subset 정의도 그에 맞춰 갱신해야 한다.

동시에 이 전략은 더 넓은 패턴을 보여준다.
TypeScript가 JavaScript 위에, Cython이 Python 위에, 지금 Solod가 Go 위에 있다.
인기 있는 언어의 타입 시스템이나 에코시스템을 재활용해 더 전문화된 용도로 만드는 것이
새로운 언어 설계의 대안적 경로가 됐다.

### Go의 시스템 프로그래밍 포지셔닝

Go는 오랫동안 “시스템 프로그래밍보다는 서버 프로그래밍”이라는 인식이 있었다.
GC, 런타임, 비교적 큰 바이너리 크기가 임베디드나 커널 수준 작업에서 장벽이었다.
Solod는 Go 문법과 도구를 유지하면서 그 장벽을 낮추려는 시도다.

성공한다면 Go 개발자들이 현재 C, Zig, Rust로 작성하는 시스템 레이어 코드를 Go 환경에서
작성하는 흐름이 생길 수 있다.
이것이 Go 생태계에서 작은 틈새를 메우는 것인지, 아니면 실질적인 패러다임 변화의 시작인지는
Solod가 프로덕션 수준에 도달했을 때 더 명확해질 것이다.

### 트랜스파일러로서의 포지셔닝과 경쟁 관계

Solod를 사용하면 시스템 코드를 C11로 배포할 수 있다.
이것은 기존 C 빌드 시스템, CI 파이프라인, 감사 절차에 별도 변경 없이 통합 가능함을
의미한다.
C를 요구하는 환경—항공, 자동차, 의료 소프트웨어의 인증 체계—에서 이것이 진입 경로가
될 수 있다.

이 포지셔닝에서 Solod의 진짜 경쟁자는 Rust나 Zig가 아니라 기존 C 코드를 쓰던 팀이
“Go 도구를 쓰면서 C 결과물을 내는” 워크플로를 원할 때다.
그 수요가 얼마나 있는지가 프로젝트의 성공 여부를 결정할 것이다.
