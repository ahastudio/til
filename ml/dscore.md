# dscore: 화자 분리 평가 도구

<https://github.com/nryant/dscore>

## 소개

dscore는 화자 분리(speaker diarization) 시스템의 성능을 평가하는 파이썬 CLI 도구다.
“누가 언제 발화했는가”를 판별하는 화자 분리 시스템의 출력(system RTTM)을 레퍼런스 주석(reference RTTM)과 비교해 여러 지표를 산출한다.
파이썬 2.7 이상과 3.6 이상을 모두 지원하며, NumPy·SciPy·intervaltree·tabulate에 의존한다.

## 평가 지표

dscore는 단일 지표가 아니라 상호 보완적인 지표 세트를 동시에 산출한다.

**DER (Diarization Error Rate)** — 화자 오류율, 오탐 발화율, 미탐 발화율의 합산 복합 지표다.
0이 완벽한 성능이며 값이 높을수록 나쁘다.
NIST RT 평가 계획에서 유래한 사실상의 표준 지표이나, 칼라(collar) 설정과 겹침 발화 처리 방식에 따라 수치가 크게 달라진다.

**JER (Jaccard Error Rate)** — Jaccard 유사도 지수 기반 지표다.
헝가리안 알고리즘으로 최적 화자 매핑을 결정하며 0–100% 범위를 갖는다.
극단적 케이스(화자 수가 크게 다를 때)에서 DER보다 안정적인 값을 제공한다.

**B-cubed** — 프레임 단위 정밀도·재현율·F1 점수다.
Bagga & Baldwin(1998)이 상호 참조(coreference) 평가를 위해 제안했으며, 화자 분리에 응용됐다.

**GKT (Goodman-Kruskal Tau)** — 비대칭 연관 지표다.
레퍼런스가 주어졌을 때 시스템 레이블 예측 가능성(τ(ref→sys))과 그 역방향(τ(sys→ref))을 별도로 측정한다.

**정보 이론적 지표** — 조건부 엔트로피 H(ref|sys), H(sys|ref), 상호 정보량(MI), 정규화 상호 정보량(NMI)을 산출한다.
Rosenberg & Hirschberg(2007)의 V-Measure 프레임워크에 기반한다.

## 사용법

### 기본 채점

```bash
python score.py -r ref1.rttm ref2.rttm -s sys1.rttm sys2.rttm
```

### 스크립트 파일 방식

다수의 파일을 일괄 처리할 때 `.scp` 파일에 경로 목록을 기재해 사용한다.

```bash
python score.py -R ref.scp -S sys.scp
```

### UEM으로 채점 구간 지정

UEM(Unpartitioned Evaluation Map)은 채점 대상 구간을 명시해 특정 구간만 평가에 반영한다.

```bash
python score.py -u all.uem -R ref.scp -S sys.scp
```

### 주요 옵션

```bash
python score.py --collar 0.100 --ignore_overlaps -R ref.scp -S sys.scp
```

| 옵션                | 기본값   | 설명                              |
| ------------------- | -------- | --------------------------------- |
| `--collar`          | 0초      | 경계 전후로 제외할 여유 구간(초)  |
| `--ignore_overlaps` | 비활성   | 겹침 발화 구간을 채점에서 제외    |
| `--step`            | 10 ms    | 프레임 단위 계산 간격             |
| `--n_digits`        | 3        | 출력 소수점 자릿수                |
| `--table_format`    | simple   | 표 출력 형식 (tabulate 스타일)    |

### RTTM 유효성 검사

```bash
python validate_rttm.py fn1.rttm fn2.rttm
```

## 파일 형식

### RTTM

10개 공백 구분 필드로 구성된 텍스트 형식이다.
필드 순서는 유형, 파일 ID, 채널 ID, 발화 시작(초), 발화 길이(초), 문자열 필드(`<NA>`), 화자 유형(`<NA>`), 화자 이름, 신뢰 점수(`<NA>`), 신호 선행 시간(`<NA>`)이다.

```text
SPEAKER CMU_20020319-1400_d01_NONE 1 130.430000 2.350 <NA> <NA> juliet <NA> <NA>
SPEAKER CMU_20020319-1400_d01_NONE 1 157.610000 3.060 <NA> <NA> tbc <NA> <NA>
```

### UEM

채점 구간을 정의하는 4개 필드 형식이다: 파일 ID, 채널 ID, 시작(초), 끝(초).

## 분석

### DER은 단순성 덕분에 살아남았지만 그 단순성이 함정이다

DER은 NIST의 RT 평가 계획에서 정의된 지표로, 1990년대 말–2000년대 초 음성 기술 평가의 중심에 놓인 이후 사실상의 표준으로 굳어졌다.
그 생존 이유는 기술적 우월성이 아니라 역사적 관성이다.
NIST가 DER을 사용했기 때문에 연구자들이 DER을 사용했고, DER로 학습된 시스템들이 DER로 비교됐으며, DER 기준의 리더보드가 형성됐다.
지표가 생태계를 만들고, 생태계가 다시 지표를 정당화하는 순환이다.

DER의 핵심 문제는 칼라(collar)라는 매개변수에 있다.
칼라는 화자 전환 경계 전후의 제외 구간으로, 0초로 설정하면 경계 오류에 극도로 민감해지고, 0.25초로 설정하면 상당한 오류를 무시하게 된다.
동일한 시스템이 칼라 설정에 따라 전혀 다른 DER 수치를 보이며, 이 설정이 논문마다 다르면 결과 비교가 의미를 잃는다.
dscore가 기본값을 0초로 두는 것은 보수적 선택이지만, 이를 명시하지 않는 기존 논문들과의 비교를 어렵게 만든다.

### 복수 지표의 공존은 평가 문제의 미해결 상태를 반영한다

dscore가 DER, JER, B-cubed, GKT, 정보 이론적 지표를 동시에 산출하는 것은 편의를 위해서가 아니라, 어떤 단일 지표도 화자 분리의 모든 측면을 포착하지 못하기 때문이다.
DER은 총 오류율을 측정하지만 오류의 원인을 분리하지 못한다.
JER은 Jaccard 유사도로 화자별 오류를 측정하지만 발화 길이를 고려하지 않는다.
B-cubed는 프레임 수준의 정밀도와 재현율을 제공하지만 계산 비용이 높다.
GKT는 두 레이블 집합 간의 예측 방향성을 측정하지만 직관적 해석이 어렵다.

이 지표들의 공존은 화자 분리 평가가 아직 “올바른 지표”를 찾지 못한 영역임을 보여준다.
각 지표는 서로 다른 연구자 그룹의 필요에서 제안됐으며, 커뮤니티가 통합 지표에 합의하지 못한 채 모두를 함께 보고하는 관행이 형성됐다.
이것이 복잡성이지만 동시에 정직성이기도 하다.

### RTTM 형식의 NA 필드는 설계 역사의 흔적이다

RTTM의 10개 필드 중 다섯 개가 `<NA>`로 채워진다는 사실은 이 형식이 원래 더 넓은 목적을 위해 설계됐음을 보여준다.
문자열 필드(orthography), 화자 유형(speaker type), 신뢰 점수(confidence score), 신호 선행 시간(signal lookahead)은 NIST의 RT 평가 계획에서 다양한 작업(음성 인식, 화자 인식, 다중 채널 처리 등)을 포괄하기 위해 정의됐다.
화자 분리 채점에서는 이 필드들이 불필요하지만, 포맷 호환성을 위해 `<NA>` 자리 표시자로 유지된다.

이 과잉 설계된 포맷은 역설적으로 상호 운용성을 보장한다.
RTTM을 출력하는 모든 화자 분리 시스템은 dscore로 바로 평가 가능하며, RTTM을 입력으로 받는 모든 도구와 통합된다.
포맷의 단순성(탭이나 XML이 아닌 공백 구분 텍스트)도 처리 편의성에 기여한다.

## 비평

### DER 기본값 0초 칼라는 실용적 평가와 어긋난다

dscore의 `--collar` 기본값은 0초다.
이는 화자 전환 경계의 모호성을 전혀 허용하지 않는 가장 엄격한 설정이다.
그러나 실제 화자 분리 평가에서 0초 칼라는 주석자의 불일치와 오디오의 물리적 특성(겹침 발화, 음향 잔향) 때문에 체계적으로 불리한 수치를 산출한다.
AMI나 CALLHOME 같은 표준 벤치마크 평가에서 0.25초 칼라가 광범위하게 사용되어 온 이유가 여기에 있다.

0초를 기본값으로 두면 최초 실행 시 예상보다 나쁜 DER이 나와 사용자가 당황하거나, 논문과 비교하기 위해 수동으로 칼라를 조정해야 한다.
더 심각한 문제는 README에 칼라 선택의 영향과 표준 벤치마크에서 관행적으로 사용하는 값이 명시되지 않는다는 것이다.
도구 자체의 신뢰성은 높지만, 사용자가 이 맥락을 알지 못하면 잘못된 비교를 하게 된다.

### OVERALL 집계 방식이 마이크로/매크로 구분 없이 보고된다

dscore의 출력에는 각 파일별 수치와 OVERALL 행이 함께 표시된다.
그러나 OVERALL이 파일 단위 평균(매크로 평균)인지 발화 길이 가중 평균(마이크로 평균)인지가 README에 명시되지 않는다.
이 구분은 파일 길이가 불균등할 때 중요하다.
짧은 파일 10개와 긴 파일 1개를 동일하게 취급하는 매크로 평균은 짧은 파일의 오류에 과도한 가중치를 부여한다.

화자 분리 데이터셋은 파일 길이가 크게 다른 경우가 많다.
TED 강연 데이터셋은 파일당 10분 이상이지만, 전화 통화 데이터셋은 2–3분 파일이 수십 개다.
집계 방식에 따라 OVERALL DER이 수 퍼센트포인트 차이가 날 수 있으며, 이것이 논문 간 비교 오류의 원인이 된다.

### 겹침 발화 처리가 선택적 옵션으로 남겨진 것은 현실과의 간극이다

`--ignore_overlaps` 플래그는 겹침 발화 구간을 평가에서 제외한다.
이 옵션이 존재한다는 것은 겹침 발화가 평가에서 중요한 변수임을 인정한다는 뜻이다.
그러나 대화형 음성에서 겹침 발화는 전체 발화 시간의 10–20%를 차지하며, 이를 제외하면 시스템의 실제 성능이 아닌 “겹침이 없는 깨끗한 구간”에서의 성능만 측정된다.

더 근본적인 문제는 겹침 발화를 RTTM으로 표현하는 방식 자체다.
현재 RTTM 형식은 동일 구간에 여러 화자를 기록할 수 있지만, 대부분의 화자 분리 시스템은 각 프레임에 단일 화자만 할당한다.
겹침을 무시하거나 제외하는 옵션을 두는 것은 이 불일치를 도구 수준에서 해결하는 것이 아니라 사용자에게 선택을 위임하는 것이다.

## 인사이트

### 평가 지표의 선택이 시스템 최적화 방향을 규정한다

dscore가 제공하는 지표들은 같은 시스템에 대해 서로 다른 순위를 매길 수 있다.
DER이 낮은 시스템이 JER은 높을 수 있고, B-cubed F1이 높은 시스템이 GKT는 낮을 수 있다.
연구자가 특정 지표를 최적화 목표로 삼으면, 그 지표에서 잘 작동하는 아키텍처와 손실 함수를 선택하게 된다.
이것은 지표가 중립적 측정 도구가 아니라 기술 발전의 방향을 결정하는 설계 선택임을 의미한다.

화자 분리 연구의 역사에서 DER 최적화에 집중한 결과, 칼라 구간의 모호성을 최소화하는 방향(명확한 발화 경계를 갖는 격식체 회의 음성)으로 연구가 편향됐다는 비판이 있다.
자유 대화, 전화 통화, 잡음 환경에서의 DER 개선은 이 편향 때문에 상대적으로 느렸다.
dscore처럼 복수 지표를 동시에 제공하는 도구는 이 단일 지표 편향에 대한 구조적 견제가 될 수 있다.

새로운 평가 지표가 도입될 때(JER이 DER을 보완하기 위해 제안됐듯이), 기존 시스템의 순위가 바뀌는 현상은 이전 최적화가 특정 지표에 과적합됐음을 드러낸다.
따라서 음성 처리 연구에서 새 평가 도구의 채택은 기술적 선택이 아니라 연구 공동체의 가치 판단이기도 하다.

### 재현 가능한 평가는 도구 표준화에서 시작한다

음성 처리 연구에서 논문 간 비교가 어려운 주요 원인 중 하나는 평가 스크립트의 비일치다.
동일한 DER을 보고하더라도 칼라 설정, 겹침 처리 방식, OVERALL 집계 방법이 다르면 수치가 같아도 의미가 다르다.
dscore처럼 오픈소스로 공유되는 표준 평가 스크립트는 이 비일치를 줄이는 가장 효과적인 수단이다.

그러나 도구 표준화만으로는 충분하지 않다.
같은 dscore를 사용하더라도 `--collar 0.0`과 `--collar 0.25`, `--ignore_overlaps` 유무에 따라 결과가 달라진다.
논문이 dscore를 사용했다고 명시하면서도 어떤 설정을 사용했는지 밝히지 않으면 표준화의 이점이 사라진다.
평가 도구의 표준화와 평가 설정의 보고 규범이 함께 발전해야 진정한 재현 가능성이 확보된다.

이 문제는 음성 처리만의 것이 아니다.
머신러닝 벤치마크 전반에서 동일한 데이터셋과 지표를 사용하면서도 전처리 방식, 데이터 분할 방법, 평가 시점 등이 달라 결과 비교가 불가능한 상황이 반복된다.
dscore 같은 도구가 주석 달린 설정 파일이나 재현 가능한 실험 로그를 기본으로 제공한다면, 평가의 투명성을 도구 수준에서 강제할 수 있다.

### 화자 분리 평가의 다음 과제는 RTTM 형식 자체에 있다

RTTM은 단일 채널, 비겹침, 명확한 화자 레이블을 전제하는 형식이다.
그러나 현실의 음성 처리 과제는 이 전제에서 벗어나고 있다.
다채널 원격 회의, 실시간 전사가 결합된 화자 분리, 감정이나 역할 정보가 포함된 화자 레이블링은 RTTM으로 표현하기 어렵다.

Whisper 같은 엔드투엔드 음성 인식 모델이 화자 정보까지 출력하기 시작하면서, 화자 분리와 음성 인식이 통합된 평가 지표에 대한 수요가 생겼다.
이 통합 지표(cpWER: concatenated minimum-permutation Word Error Rate 등)는 RTTM 기반 평가와 근본적으로 다른 형식과 가정을 갖는다.
dscore가 RTTM 기반 평가의 사실상 표준을 확립했다면, 다음 도전은 이 표준이 새로운 과제들을 어떻게 흡수하거나 대체될지에 있다.

화자 분리 평가 형식의 진화는 단순한 포맷 업그레이드가 아니라, 평가 대상 시스템에 대한 가정의 재설정이다.
RTTM이 정의한 “화자 분리”의 경계가 흐려지면서, 평가 도구가 어떤 과제를 측정하는지에 대한 정의 자체가 다시 쓰이고 있다.
dscore는 기존 과제에서 탁월한 도구이지만, 이 전환의 중심에서 어떤 역할을 할 수 있을지는 아직 열린 질문이다.
