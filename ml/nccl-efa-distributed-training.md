# AWS EFA 환경에서의 NCCL 기반 GPU 간 통신

원문: <https://aws.amazon.com/ko/blogs/tech/nccl/>

## 요약

대규모 딥러닝 모델 학습은 수십~수천 개의 GPU를 필요로 한다.
데이터 병렬 학습에서는 각 GPU가 서로 다른 배치로 그래디언트를 계산한 뒤 전체 GPU에
동기화하는 과정이 매 스텝마다 반복된다.
이 통신 효율이 전체 학습 성능을 결정하므로, GPU 간 통신 인프라의 설계가 핵심이 된다.
이 글은 AWS의 EFA(Elastic Fabric Adapter) 위에서 NCCL이 어떻게 동작하는지를 계층별로
설명한다.

MPI는 CPU 프로세스 간 통신을 위한 범용 표준이고, NCCL은 NVIDIA가 GPU 간 통신을 위해
특화 개발한 라이브러리다.
NCCL은 NVLink, NVSwitch, GPUDirect RDMA 같은 GPU 전용 하드웨어를 최대한 활용한다.
AWS 환경에서는 `aws-ofi-nccl` 플러그인이 NCCL과 libfabric 사이의 브리지 역할을 하며,
이 플러그인 없이는 NCCL이 TCP/IP로 폴백해 성능이 급격히 저하된다.
멀티 노드 아키텍처는 애플리케이션 → NCCL → `aws-ofi-nccl` → libfabric → EFA 드라이버 →
EFA 하드웨어(SRD 프로토콜)의 계층으로 구성된다.

집합 통신(Collective Communication)은 여러 GPU가 동시에 참여하는 통신 패턴이다.
핵심 연산은 세 가지다.
AllReduce는 모든 GPU의 그래디언트를 합산하고 결과를 전체에 공유하며, 매 학습 스텝마다
호출된다.
AllGather는 합산 없이 모든 GPU의 데이터를 모든 GPU에 제공하며, 파라미터 샤딩 기법에서
사용된다.
ReduceScatter는 합산 후 결과를 GPU들이 나눠 갖는 방식으로, FSDP/ZeRO-3의 역전파에서
사용된다.

링(Ring) 토폴로지는 N개의 GPU에 N개의 링크만 필요해, 일대일 통신의 N×(N-1)/2개보다
극적으로 효율적이다(N=64일 때 약 31배 차이).
통신 알고리즘으로는 Ring과 Tree 두 가지가 있다.
Ring은 모든 GPU가 동시에 인접 GPU와 통신해 대역폭을 최대 활용하며 대용량 데이터에
적합하다.
Tree는 계층적으로 데이터를 모아 지연시간이 O(log N)으로 우수하지만 대역폭 활용도가
낮아 소용량 데이터에 적합하다.
NCCL은 초기화 시 GPU 토폴로지(NVLink, NUMA, PCIe), 최적 NIC 매핑 등을 자동으로 감지해
이 두 알고리즘 중 하나를 선택하고 청크 크기와 병렬 채널 수를 결정한다.

단일 노드 내에서는 NVSwitch가 8개 GPU를 완전한 메시로 연결한다.
P5 인스턴스는 NVLink 4세대로 GPU당 900 GB/s, P6는 NVLink 5세대로 GPU당 1,800 GB/s의
양방향 대역폭을 제공한다.
멀티 노드 간 통신은 EFA를 통해 카드당 100 Gbps(p5.48xlarge 기준 총 3,200 Gbps)로,
노드 내 대역폭과 비교하면 수십 배 낮다.
NCCL의 소프트웨어 최적화(파이프라이닝, 청킹)와 EFA의 하드웨어 최적화(OS 우회, GPUDirect
RDMA)가 결합되어 이 격차를 실용적인 수준으로 보완한다.

## 분석

### 계층화된 추상화 스택

멀티 노드 GPU 통신은 명확한 추상화 계층으로 구성된다.

- **애플리케이션 계층** (PyTorch, TensorFlow): NCCL API 호출
- **통신 최적화 계층** (NCCL): 알고리즘 선택, 청크 분할
- **인터페이스 브리지** (`aws-ofi-nccl`): NCCL 요청을 libfabric API로 변환
- **네트워크 추상화** (libfabric): 다양한 네트워크 인터페이스를 단일 API로 노출
- **드라이버 계층** (EFA 드라이버): OS 우회(OS-bypass) 경로 처리
- **하드웨어 계층** (EFA): SRD 프로토콜로 패킷화·전송, GPUDirect RDMA로 GPU 메모리 직접 기록

각 계층이 명확한 책임을 갖고, 하위 계층은 상위 계층에게 표준 인터페이스를 제공한다.
`aws-ofi-nccl`이 없으면 NCCL은 TCP/IP로 폴백해 성능이 급격히 저하된다는 사실은,
플러그인 계층이 이 스택의 핵심 연결 고리임을 보여준다.

### Ring vs. Tree 알고리즘의 적용 기준

두 알고리즘의 선택 기준은 데이터 크기다.
Ring은 대역폭을 최대화하지만 초기 지연이 크고, Tree는 지연을 최소화하지만 대역폭
활용도가 낮다.
수백 MB 이상의 그래디언트 텐서(모델이 클수록 커진다)에는 Ring이, 수 KB~수 MB의 소형
통신에는 Tree가 유리하다.
NCCL이 이 선택을 자동화한다는 것은 개발자가 통신 알고리즘을 수동으로 지정할 필요가
없음을 의미하지만, 최적화 한계를 이해하려면 내부 동작을 알아야 한다.

### AllReduce, AllGather, ReduceScatter의 역할 분담

세 연산은 서로 다른 분산 학습 패러다임에 대응한다.
AllReduce는 데이터 병렬(Data Parallelism)의 그래디언트 동기화에 사용된다.
AllGather + ReduceScatter 조합은 FSDP(Fully Sharded Data Parallel)/ZeRO-3의 핵심이다.
ZeRO-3는 순전파에서 AllGather로 파라미터를 모으고, 역전파 후 ReduceScatter로 그래디언트를
분산 보관함으로써 GPU당 메모리 사용량을 N분의 1로 줄인다.
이 설계는 “통신 비용으로 메모리 비용을 지불하는” 명시적 트레이드오프다.

### 노드 내 vs. 노드 간 대역폭 격차와 함의

NVLink(P6 기준 1,800 GB/s) 대 EFA(100 Gbps ≈ 12.5 GB/s)의 격차는 약 144배다.
이 격차는 학습 전략 선택에 직접 영향을 미친다.
텐서 병렬(Tensor Parallelism)처럼 통신이 매우 잦은 방식은 노드 내 NVLink를 최대한
활용해야 하며, 파이프라인 병렬(Pipeline Parallelism)처럼 노드 간 통신을 최소화하는
방식이 멀티 노드 환경에서 유리하다.

## 비평

### 강점: 계층을 따라가는 구체적 흐름 설명

추상적인 개념을 “애플리케이션이 NCCL API를 호출하면 → aws-ofi-nccl이 변환하면 → EFA가
GPUDirect RDMA로 GPU 메모리에 직접 쓴다”는 데이터 흐름으로 연결한 것이 이 글의 핵심
강점이다.
각 계층의 역할이 명확해지면 성능 병목이 어느 계층에서 발생하는지 진단하는 능력도
생긴다.
P5/P6 인스턴스의 실제 대역폭 수치를 비교한 것도 실무에서 즉시 참조 가능한 정보다.

### 한계: 실패 시나리오와 트러블슈팅 깊이 부족

“최신 소프트웨어 스택을 쓰면 기본값으로 충분하다”는 결론은 입문자에게는 유용하지만,
이미 성능 문제를 겪고 있는 실무자에게는 부족하다.
`NCCL_DEBUG=INFO`를 켰을 때 어떤 로그가 나오고 어떻게 해석하는지, 대역폭이 기대치보다
낮을 때 어느 계층에서 병목인지 판단하는 방법, 잘못된 NUMA 매핑이나 EFA 카드 개수
불일치 같은 흔한 설정 오류에 대한 가이드가 빠져 있다.

### 보완이 필요한 관점: 다른 클라우드 벤더와의 비교

EFA + aws-ofi-nccl 스택이 Google의 TPU + XLA 또는 Azure의 InfiniBand + NCCL 구성과
비교해 어떤 포지셔닝을 갖는지 언급하지 않는다.
AWS가 자사 블로그인 만큼 이는 당연하지만, 독자가 아키텍처 선택을 해야 할 때는 비교
관점이 필요하다.
또한 MoE(Mixture-of-Experts) 모델처럼 비대칭 통신 패턴을 가진 최신 모델 아키텍처에서
현재 스택의 한계를 이 글에서 언급하지 않고 “다음 블로그에서 다룬다”고 미룬 것은
완결성 면에서 아쉽다.

## 인사이트

### 하드웨어 격차를 소프트웨어가 메우는 구조의 한계

노드 간 EFA 대역폭(100 Gbps)과 노드 내 NVLink 대역폭(P6 기준 1,800 GB/s)의 144배
격차는 소프트웨어 최적화로 줄일 수 있는 격차가 아니다.
NCCL의 파이프라이닝과 청킹, EFA의 GPUDirect RDMA는 이 격차 내에서 효율을 높이는 것이지,
격차 자체를 좁히지는 않는다.
이것은 분산 학습의 근본적 제약이다.
GPU 개수를 늘릴수록 노드 간 통신 오버헤드가 증가하고, 어느 순간부터 GPU를 추가해도
학습 속도가 선형으로 증가하지 않는다(Amdahl의 법칙의 통신 버전).
이 제약이 학습 아키텍처 선택을 강제한다.
텐서 병렬은 NVLink 범위 안에서 실행해야 하고, 파이프라인 병렬은 노드 간 통신을
최소화하도록 파이프라인을 끊어야 하며, MoE 모델의 전문가 라우팅은 노드 간 올투올
통신이 폭발적으로 증가하는 문제를 안고 있다.
인프라 제약이 모델 아키텍처 선택의 암묵적 가이드라인이 된다는 패턴은 AI 개발 생태계
전반에서 반복된다.
TPU Pod 아키텍처를 중심으로 설계된 구글의 JAX 에코시스템, NVLink 도메인을 의식한
Megatron-LM의 텐서 병렬 설계가 모두 같은 패턴이다.

### 자동 토폴로지 감지는 투명성의 반대급부다

NCCL이 시스템 초기화 시 NVLink, NUMA, PCIe, NIC 매핑을 자동으로 감지하고 최적 알고리즘을
선택한다는 것은 개발자 경험 측면에서 훌륭한 설계다.
그러나 이 자동화는 문제가 생겼을 때 “왜 이 알고리즘이 선택되었는가”를 이해하기 어렵게
만드는 불투명성을 낳는다.
`NCCL_ALGO=Ring`처럼 수동으로 지정할 수 있지만, 언제 이것이 자동 선택보다 나은지
판단하려면 내부 동작을 알아야 한다.
이것은 높은 수준의 추상화가 항상 갖는 트레이드오프다.
데이터베이스 쿼리 옵티마이저가 인덱스를 자동으로 선택하지만 `EXPLAIN ANALYZE`로 실행
계획을 들여다봐야 할 때가 있는 것, 또는 Go의 GC가 자동으로 동작하지만 GC 튜닝이
필요한 경우가 생기는 것과 같은 구조다.
AI 인프라가 점점 더 자동화되면서, 이 자동화를 이해하는 전문가와 그냥 쓰는 일반
사용자 사이의 격차가 벌어진다.
대규모 학습에서 수백~수천 GPU를 운용하는 MLOps 팀은 이 격차를 메우는 역할을 하며,
이는 새로운 전문화 영역이다.

### “기본값이 충분하다”는 주장의 전제 조건

이 글은 “최신 소프트웨어 스택을 쓰면 기본값으로 충분하다”고 결론 짓는다.
이 주장은 특정 조건 하에서만 성립한다.
AWS가 제공하는 최적화된 AMI(Amazon Machine Image)나 AWS Deep Learning Containers를
사용하고, 인스턴스 타입과 NCCL 버전이 매칭되어 있고, `FI_EFA_USE_HUGE_PAGE=0`처럼
필수 환경 변수가 설정되어 있을 때의 이야기다.
커스텀 환경을 구성하거나, 컨테이너를 직접 빌드하거나, 최신 모델 아키텍처(MoE, 스파스
어텐션 등)를 실험하는 팀에게 “기본값으로 충분하다”는 조언은 성립하지 않는 경우가 많다.
이 주장이 의도하는 바는 “튜닝은 측정 후에 하라”는 합리적 원칙이지만, 그 전제가 충족되지
않는 상황에서는 오히려 진단을 늦추는 잘못된 안도감을 줄 수 있다.
“기본값이 최적이다”와 “기본값이 충분하다”는 다르다.
대규모 클러스터에서 몇 퍼센트의 통신 비효율이 누적되면 수백 GPU-시간의 낭비가 된다.
클러스터 규모가 커질수록 기본값의 충분성 여부는 더 꼼꼼히 검증해야 하며, 이를 위한
프로파일링 도구와 측정 방법론이 이 글에서 다뤄졌다면 더 완결된 가이드가 되었을 것이다.
