# CLX: Compile Lua to Native Executables Through C++20

<https://github.com/samyeyo/clx>

HN 토론: <https://news.ycombinator.com/item?id=48871547> (142점, 32개 댓글)

## 소개

CLX는 Lua 스크립트를 C++20을 중간 표현으로 삼아 독립 실행 가능한 네이티브 바이너리로
변환하는 AOT(Ahead-of-Time) 컴파일러다.
바이트코드 인터프리터 없이 GCC, Clang, MSVC 같은 기존 C++ 컴파일러를 백엔드로 활용해
네이티브 기계어를 생성한다.

주요 특징은 다음과 같다.
Lua 5.5 호환성을 유지하며 코루틴, 메타테이블, 테이블을 지원한다.
8바이트 페이로드와 별도 타입 식별자를 결합한 태그드 값(tagged value) 아키텍처를 쓴다.
6바이트 이하 문자열은 즉시 값(immediate value)으로 저장하는 최적화를 포함한다.
빌드된 실행 파일 크기는 100KB 이하를 목표로 한다.

```bash
clx file.lua             # 기본 컴파일
clx --fast file.lua      # 속도 최적화
clx --debug file.lua     # 디버그 심벌 포함
clx --minimal file.lua   # 최소 라이브러리
```

대표 벤치마크에서 Lua 5.5 인터프리터 대비 피보나치 62.2배, 스펙트럴 노름 10.7배,
배열 합산 4.1배 속도를 기록했다.

현재 베타 상태이며 주요 제한 사항은 다음과 같다.
`load()`, `dofile()` 같은 동적 코드 평가가 순수 AOT 모드에서 지원되지 않는다.
`debug` 모듈은 컴파일된 형태에서 사용 불가다.
전통적인 Lua C API와 호환되지 않아 네이티브 모듈은 CLX 전용 C++ API를 써야 한다.

## 분석

### C++20을 중간 표현으로 선택한 이유

CLX의 핵심 설계 결정은 LLVM IR이나 어셈블리가 아닌 C++20을 중간 표현으로 택한 것이다.
이 선택은 컴파일러 구현 복잡도를 대폭 낮추는 전략이다.
LLVM 백엔드를 직접 구현하려면 IR 생성, 최적화 패스, 코드 생성을 모두 다뤄야 한다.
C++20을 거치면 기존 C++ 컴파일러가 이 모든 작업을 대신한다.

이 접근은 이식성도 함께 해결한다.
C++20 코드는 GCC, Clang, MSVC 모두 컴파일할 수 있으므로, CLX는 세 플랫폼을 별도
백엔드 없이 지원한다.
대가는 빌드 속도다.
Lua 스크립트를 네이티브 바이너리로 만들려면 C++ 컴파일 시간이 추가된다.
JIT처럼 즉각적인 실행이 필요한 상황에서는 적합하지 않다.

### 62배 속도 향상의 의미와 맥락

피보나치 62.2배 향상은 인상적이지만, 마이크로벤치마크 특성을 고려해야 한다.
재귀 피보나치는 인터프리터 오버헤드가 극단적으로 큰 케이스다.
함수 호출마다 스택 프레임 관리, 타입 확인, 메타테이블 조회가 발생한다.
AOT 컴파일이 이 오버헤드를 제거하면 이론적 속도 향상의 상한에 가까워진다.

배열 합산 4.1배는 더 현실적인 수치다.
메모리 접근 패턴이 지배하는 작업에서는 인터프리터 오버헤드 제거의 효과가 줄어든다.
실제 애플리케이션은 이 두 수치 사이 어딘가에 있을 것이다.

### 태그드 값 아키텍처와 트레이드오프

8바이트 페이로드 + 별도 타입 태그 구조는 NaN 박싱(NaN boxing)의 대안이다.
NaN 박싱은 IEEE 754 배정밀도 부동소수점의 NaN 공간을 활용해 타입 정보와 값을
단일 64비트 레지스터에 담는 기법으로, V8과 LuaJIT가 쓴다.
CLX의 분리 저장 방식은 캐시 효율이 낮을 수 있지만 구현이 단순하고 디버깅이 쉽다.

6바이트 이하 문자열 즉시 저장 최적화는 흔한 단일 단어 키, 상수, 짧은 식별자가
별도 힙 할당 없이 처리됨을 의미한다.
Lua 코드에서 문자열 키가 빈번히 쓰이는 패턴에서 눈에 띄는 성능 차이가 날 수 있다.

## 비평

### 동적 언어 AOT의 구조적 한계

Lua의 `load()` 함수를 지원하지 않는다는 것은 작은 기능 누락이 아니다.
`load()`는 Lua 에코시스템에서 DSL(도메인 특화 언어), 플러그인 시스템, 설정 파일 실행,
REPL 구현의 핵심 메커니즘이다.
게임 엔진 스크립팅, 임베디드 설정, 핫 리로드 워크플로에서 이 기능은 Lua를 선택한
이유 자체인 경우가 많다.

CLX가 이 제한을 해소하려면 “혼합 모드” 실행이 필요하다.
미리 컴파일한 코드와 런타임 동적 평가를 혼합하는 것인데, 이렇게 되면 독립 실행 파일이라는
핵심 가치가 흐려진다.
현재 베타 단계에서 이 한계가 어떻게 해소될지가 CLX의 실용적 적용 범위를 결정할 것이다.

### 기존 Lua C API 비호환성의 생태계 비용

전통적인 Lua C API와 호환되지 않는다는 것은 기존 Lua 모듈 생태계를 사용할 수 없다는
의미다.
LuaSocket, LÖVE 프레임워크 바인딩, 데이터베이스 드라이버 같은 C 확장 모듈이
CLX에서 작동하려면 CLX 전용 API로 재작성돼야 한다.

이것은 언어를 쓰는 것이 아니라 비호환 파생 생태계에 진입하는 것에 가깝다.
Lua의 강점 중 하나가 풍부한 C 바인딩 생태계라는 점에서, 이 비호환성은 CLX의 실용적
사용 범위를 스탠드얼론 순수 Lua 프로그램으로 좁힌다.

### 베타 단계에서 측정하기 어려운 안정성

벤치마크와 기능 목록은 명확하지만, 베타 단계 컴파일러의 더 중요한 질문은 엣지 케이스
처리다.
코루틴 재개 중 에러 처리, 복잡한 메타테이블 상속 체인, 약한 테이블(weak table)과
가비지 컬렉터의 상호작용 같은 Lua 특유의 복잡한 시맨틱을 얼마나 정확하게 구현했는지
현재 문서만으로는 평가하기 어렵다.

## 인사이트

### LLVM 없는 컴파일러의 설계 공간

CLX가 제시하는 접근은 “고성능 컴파일러를 만들려면 LLVM이 필요하다”는 통념에 대한
반례다.
기존 C++ 컴파일러를 백엔드로 활용하면 개인이나 소규모 팀도 경쟁력 있는 성능의
AOT 컴파일러를 만들 수 있다.

이 패턴은 다른 동적 언어에도 적용 가능하다.
인터프리터 성능이 한계이지만 LLVM 기반 JIT를 구현할 리소스가 없는 소규모 언어
프로젝트들이 C/C++ transpilation을 고려할 수 있다.
C 출력은 이식성을 확보하고, C++20은 현대 언어 기능으로 소스 생성을 단순화한다.
HN에서 sitkack은 Python-to-C++ 컴파일러인 Shed Skin을 선례로 언급했다.[^sitkack]
Cython이 Python을 C로 컴파일하는 것처럼, 이 계열의 접근이 여러 동적 언어에서
반복적으로 시도되고 있음을 보여준다.
CLX와 Shed Skin의 공통점은 완전한 언어 호환성보다 성능 최적화에 더 집중한다는 점이고,
공통된 한계는 동적 평가(`eval`, `load`)를 완전히 지원하기 어렵다는 것이다.

### 배포 단순화가 언어 채택에 미치는 영향

Lua는 임베디드, 게임, 스크립팅 도구에서 인기 있지만 배포 시 인터프리터 의존성이
항상 부담이었다.
CLX가 이 문제를 해결하면 Lua의 사용 사례가 넓어질 수 있다.
현재 Go나 Rust로 작성하던 단독 배포 CLI 도구를 Lua로 쓸 수 있게 되는 것이다.

HN에서 corysama는 iOS 게임 개발이 특히 유망한 사용 사례라고 지적했다.[^corysama]
iOS에서 LuaJIT는 JIT 모드를 허용하지 않아, Lua로 게임을 만드는 개발자들이 성능 손실을
감수해야 했다.
AOT 컴파일은 이 제약을 우회하면서 네이티브에 가까운 성능을 제공할 수 있다.

이것은 단순한 성능 향상보다 언어 생태계에 더 큰 영향을 줄 수 있다.
배포 복잡도가 낮아지면 Lua로 내부 도구를 만드는 팀이 늘어나고, 그것이 커뮤니티와
라이브러리 생태계 성장으로 이어지는 긍정적 순환이 가능하다.

### 동적 언어 AOT의 실용적 한계선

CLX의 제한 목록은 동적 언어를 AOT 컴파일할 때 어디서 타협이 필요한지를 보여주는
명확한 지도다.
`load()` 제거, C API 비호환, `debug` 모듈 제거는 모두 런타임 동적성에 의존하는
기능이다.

AOT의 핵심 전제는 “컴파일 타임에 알 수 있는 것을 최대화”다.
Lua가 런타임에 허용하는 것들—임의 코드 실행, C 스택 검사, 동적 디버깅—이
정확히 AOT가 처리할 수 없는 부분과 겹친다.
CLX가 성숙해지는 과정에서 “Lua의 무엇을 포기할 때 AOT의 무엇을 얻는가”라는
트레이드오프 지도가 더 선명해질 것이고, 그 지도가 이 기술의 진짜 실용 범위를 정의한다.
