# On Rendering Diffs - 거대한 diff를 브라우저에서 즉시 렌더하기

원문: <https://pierre.computer/writing/on-rendering-diffs>

## 요약

Pierre Computer Company의 amadeus는 코드 리뷰 UI에서 diff 렌더링이라는
좁아 보이는 문제를 극한까지 밀어붙인 경험을 기록한다. 소형·중형 변경은
기존 도구로 충분하지만, 에이전트가 구현·테스트·스냅샷을 한꺼번에
생성하거나 브랜치가 예상보다 큰 영역을 건드리면 리뷰 서피스가 한계를
드러낸다. 한 번에 한 파일만 보여주거나, 파일마다 별도 로딩이 필요하거나,
스크롤 자체가 둔해진다. Pierre는 이 문제를 라이브러리로 외부화하는
선택을 했고, 약 6개월 전 출시한 Diffs 라이브러리의 새 컴포넌트 `CodeView`로
“어떤 diff든 그냥 렌더된다”라는 의도적 불가능 목표를 좇았다.

문제는 세 갈래로 분해된다. 렌더링(DOM 복잡도가 폭발), 처리(파일·diff
연산이 수천 번 곱해짐), 메모리(파싱 데이터 구조가 브라우저 한계를 압박).
처음 출시 때는 단순 가상화와 워커 스레드로의 신택스 하이라이팅 이전이
부분 해법이었으나, O(n×m) 복잡도, 가상화로 인한 빈 화면(blanking),
높은 메모리 사용량이 남았다. `CodeView`는 이 세 영역을 한 문제로 다루는
상위 레이어다.

가상화 측면에서는 “역 sticky 기법(Inverse Sticky Technique)”을 새로
제시한다. 일반 sticky가 콘텐츠 윗 가장자리를 뷰포트 상단에 고정하는
대신, 렌더된 영역의 아래 가장자리를 뷰포트 하단에, 위 가장자리를 뷰포트
상단에 고정한다. 음수 `top`·`bottom` 오프셋(`(contentHeight - viewportHeight) * -1`)
으로 구현되며, JavaScript가 스크롤을 따라가지 못해도 렌더된 영역이
뷰포트의 끝에 “붙어” 빈 공간이 노출되지 않는다. 네이티브 스크롤을
유지하면서 blanking을 사실상 불가능하게 만든다. Safari는 충분히 공격적인
스크롤에서 여전히 빈 공간을 만들 수 있다는 한계를 정직하게 명시한다.

레이아웃 추정은 `lineHeight × splitLineCount + hunkCount × hunkSeparatorHeight`
의 1차 추정 위에 “위치→라인” 체크포인트 캐시와 이진 탐색을 얹어 수십만
라인 hunk에서도 시작점을 빠르게 찾는다. 가상화 환경에서 브라우저 기본
스크롤 앵커링은 작동하지 않으므로 `overflow-anchor: none`으로 끄고, 첫
가시 파일·라인을 자체 앵커로 잡아 DOM 변경 전후로 뷰 안정성을 보장한다.

메모리 영역에서는 더 큰 사례로 Linux v6 → v7 diff(약 700MB 패치)를
다룬다. 첫째, JS 엔진의 substring이 원본 거대 문자열을 살아 있게 만드는
함정을 발견하고, 파싱된 라인 문자열을 명시적으로 분리(detach)하는 복사로
2.4GB → 1.15GB, 파싱 시간 약 80% 단축을 얻었다. 둘째, 파일·diff 컨테이너의
Shadow DOM 셸을 풀(pool)에 보관해 스크롤 시 DOM 처닝과 GC 압력을 줄였다.
셋째, 파일마다 별도 `options` 객체를 두던 모델을 “`CodeView`가 단일
options 상태를 소유하고 각 아이템이 getter로 읽는” 모델로 바꿔, 옵션
변경 비용을 O(n)에서 사실상 O(1)로 줄이고 20~30MB 메모리를 더 절약했다.
넷째, Shiki 기반 신택스 하이라이팅은 워커 풀에서 비동기 처리하고
LRU 캐시·prime API로 곧 보일 파일의 결과를 미리 데워둔다.

저자는 남은 문제도 명시한다. 공격적 스크롤 시 CSS 레이아웃·페인트가
주요 비용이 되는 점, 워커와의 직렬화 비용이 메인 스레드를 압도하기도
한다는 점, 가로 가상화나 극단적으로 긴 한 줄(minified JS/CSS)은 아직
미해결이라는 점. 마지막 P.S.는 Safari에 헌정된 우호적 항의로, sticky
컴포지팅 성능, 60Hz로 제한된 `requestAnimationFrame`, 추적이 어려운
프로파일러, 슬롯 컨테이너 주입 시 발생하는 WebKit 레이아웃 버그
(#308027)를 나열한다.

## 분석

### “라이브러리로 외부화한다”라는 제품 결정의 의미

글의 전제는 “diff 렌더링은 거의 모든 제품에서 1차 가치가 아니지만 모든
팀이 똑같이 다시 짠다”이다. Pierre는 이를 라이브러리로 외부화하는
포지션을 택했다. 이는 코드 리뷰 도메인의 “모노레포 vs 멀티레포”
논쟁과 비슷한 구조다. 각 회사가 자기 사용처에 특화된 작은 diff 렌더러를
계속 다시 짜는 비용을 합치면, 1차 가치도 아닌 영역에 거대한 중복
투자가 누적되어 있다. 한 회사가 그것을 외부화해 OSS/SDK로 풀어주는
순간 산업 전체의 인프라가 한 단계 올라간다.

### 가상화의 “이론적 트레이드오프”를 직접 깨러 간 시도

기존 가상화 기법은 세 갈래(네이티브 스크롤 + 잠재적 blanking / fixed
컨테이너 + rAF 의존 / 완전 에뮬레이션)로 알려져 있고, 각각의 단점이
교과서에 있다. Pierre는 이 트레이드오프 표를 받아들이지 않고 “네이티브
스크롤 유지 + blanking 불가능”을 동시에 만족하는 4번째 점을 찾는다.
“이전 사람들의 트레이드오프를 옳다고 가정하지 말고 한 번 더 의심하라”의
실전 사례다. 결과물은 sticky 포지셔닝의 음수 오프셋이라는 매우 단순한
CSS 트릭이고, 단순함에 도달한 과정 자체가 가치다.

### 메모리 관점이 “DOM 가상화” 못지않게 중요해진다

가상화는 보통 DOM 크기 제어 기법으로 소개되지만, Pierre의 글은 “파싱
문자열의 substring 참조 보존”, “Shadow DOM 셸 풀링”, “options 객체
공유”라는 메모리 디테일이 같은 비중으로 중요해지는 임계 규모를 보여준다.
700MB 패치, 70만 라인 hunk 같은 케이스는 “DOM은 작지만 메모리는 크다”의
함정이 명확해진다. 즉 진짜 가상화는 DOM·메모리·처리의 세 축을 동시에
보는 일이다. [[harness-engineering-encyclopedia]]가 다룬 에이전트
하니스 엔지니어링에서 “한 차원이 풀리면 다음 차원이 보틀넥이 된다”의
구체 사례에 해당한다.

### 에이전트가 만든 diff가 새 워크로드를 정의한다

저자는 “에이전트가 구현·테스트·스냅샷을 생성한 PR”이 리뷰 서피스를
무너뜨리는 시작점이라고 명시한다. 이는 단순한 사례 언급이 아니라
diff 렌더링 도구의 새 워크로드 변화 신호다. 인간이 만드는 PR은 평균
크기와 분포가 안정적이었지만, 에이전트가 만드는 PR은 “수천 파일,
수십만 라인, 수만 hunk”의 꼬리가 두꺼워진다. Linux v6 → v7 같은 케이스가
점점 일상 워크로드가 된다는 뜻이다. CodeView가 “이 diff를 렌더할
수 있다”는 데모(DiffsHub)를 표적으로 삼은 것은 이 변화의 1차 적응이다.

## 비평

### 강점 - 글이 코드, 데모, 라이브러리로 닫혀 있다

이 글의 가장 강한 점은 모든 주장이 검증 가능한 형태로 닫혀 있다는
것이다. CodeView 컴포넌트는 `@pierre/diffs` 패키지로 공개되어 있고,
DiffsHub는 누구나 GitHub URL을 `diffshub`로 바꿔 같은 diff를 직접
체험할 수 있다. 메모리·시간 절감 수치, 사용한 알고리즘(이진 탐색,
sticky 음수 오프셋, Shadow DOM 풀)이 구체적이며, 실패 사례(Safari의
sticky 컴포지팅, rAF 60Hz 캡)도 정직하게 노출한다. 마케팅 글의 형식을
취하면서도 엔지니어링 글의 깊이를 갖춘 드문 사례다.

### 약점 - 정량 데이터의 일관성 부족

메모리 2.4GB → 1.15GB, 파싱 시간 80% 감소, options 공유로 20~30MB 절감
같은 수치는 인상적이지만 측정 환경(브라우저, 머신, 데이터셋)이 한 곳에
모이지 않는다. 같은 PR이 Chrome·Firefox·Safari에서 어떻게 다른지,
주요 경쟁 도구(GitHub의 “Files changed” 탭, GitLab, Phabricator)와의
대조는 어떤지가 빠진다. 라이브러리 채택을 검토하는 팀은 이 비교 매트릭스가
필요하다.

### 약점 - 라이브러리 외 사용처에 대한 단서 부족

CodeView는 React 컴포넌트이고, Shadow DOM 셸을 쓴다. 이는 React 트리에
삽입할 때는 매끄럽지만 Vue, Svelte, Solid, 또는 비-React 페이지에서의
통합 비용은 별도 검토 대상이다. 또한 Tauri를 “일반 타깃”이라고 언급
하면서도, Electron, CEF, 모바일 WebView 같은 환경에서의 검증 상태는
드러나지 않는다.

### 약점 - 비교 대상 부재

기존 OSS 솔루션(diff2html, react-diff-viewer, monaco editor의 diff 모드)
대비 어떤 축에서 우위인지가 글에 명시되지 않는다. “기본 가상화로는
부족했다”라는 표현은 있지만 어떤 라이브러리를 어떤 조건에서 비교했는지가
빠진다. 산업 표준 베이스라인 없이 “우리는 이렇게 만들었다”만 제시되면
독자가 절대 비교를 할 수 없다.

### 약점 - 서버 측 스트리밍의 미루기

저자는 마지막에 “하이라이팅 직렬화는 아마 서버 쪽 스트리밍이 맞다”
라고 인정한다. 그러나 글 전체가 “브라우저에서 끝까지 해보자”라는
챌린지 기조이기 때문에, 어디까지가 브라우저의 임계 영역이고 어디부터는
서버가 답인지의 결정 가이드가 빠진다. 라이브러리 사용자는 “이걸 다
브라우저에 맡길지 서버 쪽으로 옮길지”를 매번 결정해야 하며, 그 가이드가
가장 가치 있는 정보일 텐데 글이 그 단계까지 가지 않는다.

## 인사이트

### “트레이드오프 표는 의심 대상이다”라는 엔지니어링 자세

가상화에는 세 가지 알려진 기법이 있고 각각의 단점이 교과서에 정리되어
있다. 보통의 엔지니어는 이 표를 받아들이고 자기 워크로드에 가장 덜
아픈 하나를 고른다. Pierre는 그 표를 의심하고 4번째 점을 찾았다.
“음수 sticky 오프셋”이라는 최종 해법은 너무 단순해서 누군가는 이미
시도했을 법한 기법이지만, 이 단순함에 도달하기까지는 “트레이드오프가
정말 강제되는가”라는 의심이 필요하다.

이 자세는 엔지니어링 일반에 적용되는 메타 원칙이다. CAP 정리, 동시성
모델의 ACID vs BASE, 가비지 컬렉터의 throughput vs latency — 모두
교과서적 트레이드오프 표가 있고, 그 표 위에서 “선택”하는 것이 표준
교육이다. 그러나 산업의 큰 진전은 종종 “표 자체를 깨러 간” 시도에서
나왔다. CockroachDB는 CAP 표를 그대로 받지 않고 “PC/EC 모델”을 제안했고,
Cloudflare Workers는 cold start vs warm start 표를 isolate 모델로 깼다.

여기서 도출되는 함의는 “표를 받아들이는 학습”과 “표를 의심하는 학습”이
다른 종류라는 것이다. 부트캠프·대학·자격증 교육은 전자에 집중되어
있고, 시니어 엔지니어로의 성장은 후자로의 전환이다. tiny-vllm이 가르치는
“블랙박스를 열어라”의 다음 단계가 “표를 의심하라”다.

세 번째 함의는 AI 시대에 더 무겁다. LLM은 학습 데이터의 평균을 재생산
하므로 “표 안에서 선택”은 잘하지만 “표를 의심”은 거의 못 한다. 즉
LLM이 평범한 가상화 기법을 추천하는 동안, 인간 엔지니어가 가질 수 있는
프리미엄은 “표를 의심하고 4번째 점을 찾는 행위”에 있다. CodeView가
잘 작동하는 이유는 알고리즘이 아니라 이 의심이다.

### 메모리는 “약한 참조”의 영역에서 가장 자주 새는 자원이 된다

저자가 발견한 “parsed substring이 원본 거대 문자열을 살아 있게 한다”는
함정은 JavaScript 엔진의 V8 구현 디테일이다. V8은 substring을 새 메모리
할당이 아니라 “원본의 [start, end) 슬라이스”로 구현해 메모리·시간을
절약한다. 보통은 영리한 최적화이지만, 700MB 원본에서 1KB 라인 만 개를
잘라 보관하는 시나리오에서는 1KB × 만개 = 10MB가 아니라 700MB가 산다.

이는 모든 런타임에 등장하는 일반 패턴이다. Go의 slice도 underlying array를
참조한다. Java의 substring은 자바 7까지 같은 함정이 있었고 7부터 바뀌었다.
Python의 view·memoryview도 같은 의미론이다. Rust의 `&str` 슬라이스는
참조 카운트가 아니라 빌림이지만, `Cow<str>`나 `Arc<str>` 변환 시 비슷한
선택을 만난다. “명시적 복사로 분리하기”는 모든 언어가 가진 패턴이고,
필요할 때를 알아채는 것이 시니어의 신호다.

여기서 도출되는 두 번째 함의는 “기본 메모리 모델의 가정이 깨지는
임계 규모”의 존재다. 작은 데이터에서는 영리한 최적화가 옳고, 큰
데이터에서는 그 최적화가 함정이 된다. 임계점을 인지하지 못하면
“프로토타입은 빠른데 프로덕션은 느리다”의 미스터리가 반복된다.
[[harness-engineering-encyclopedia]]가 다룬 “규모가 차원의 보틀넥을
바꾼다”와 같은 일반 패턴이다.

세 번째 함의는 디버깅 도구의 한계다. 메모리 누수는 “이 객체가 살아
있다”까지는 도구가 보여주지만 “왜 살아 있나”는 코드 의도와 런타임
의미론의 교차점이라 자동 추론이 어렵다. Pierre가 이 함정을 풀어낼 수
있었던 것은 “파서 함수 + 명확한 입출력 + 메모리 측정”이라는 좁은
가설 공간이 있었기 때문이다. 즉 메모리 디버깅의 1차 도구는 좁은
가설을 만드는 도메인 지식이다. AI 어시스턴트가 이 영역에서 보조 역할은
하지만 주역은 아직 인간이다.

### 에이전트가 만드는 PR이 리뷰 도구의 본격 워크로드 변화 시그널이다

“에이전트가 생성한 구현·테스트·스냅샷이 리뷰 서피스를 무너뜨리는 시작
점”이라는 저자의 관찰은 무심코 던지는 한 줄이지만 매우 중요한 신호다.
인간이 만드는 PR의 통계 분포는 수십 년간 안정적이었다. 평균 10~100
파일, 수백~수천 라인. 리뷰 도구는 이 분포에 맞춰 설계되었다. 에이전트가
PR을 생성하면 분포가 바뀐다. 평균이 같더라도 꼬리가 두꺼워지고, 한
PR이 수천 파일·수십만 라인을 건드리는 일이 일상화된다.

이 변화는 코드 리뷰 도구의 1차 가정을 깬다. “파일 단위로 본다”는
가정이 “언제 자료 단위가 너무 커서 본다는 행위가 불가능해지는가”라는
질문으로 바뀐다. 인간이 리뷰할 때는 한 파일을 열어 천천히 보는 시간이
있지만, 에이전트가 생성한 PR에서는 인간이 그 시간을 가질 수 없다.
이는 코드 리뷰의 형식 자체가 “전체 확인”에서 “표본 확인 + 자동 검증”
으로 옮겨감을 의미한다. [[ai-era code review]] 류 글이 다루는 현상의
인프라 측 보완이다.

여기서 도출되는 두 번째 함의는 리뷰 도구의 우선순위 재편이다. 인간
리뷰가 1차였던 시절 도구는 “읽기 좋게” 만드는 데 집중했다. 에이전트가
1차가 되면 도구는 “자동 검증, 패턴 탐지, 위험 시그널 노출”에 집중해야
한다. CodeView가 “어떤 diff든 렌더한다”를 약속하는 것은 이 다음 세대
도구의 인프라 기반을 닦는 행위다. 인간이 다 못 볼 diff를 자동 검증
에이전트가 보고, 그 결과를 인간이 표본 확인하는 워크플로우가 다음
표준이 된다.

세 번째 함의는 코드 호스팅 플랫폼의 미래에 대한 것이다. GitHub의 PR
UI는 인간 중심으로 진화했고, 거대한 diff에서는 의도적으로 일부 파일을
숨기거나 “Load diff” 버튼으로 미루는 식이다. 에이전트 시대에 이 UX는
역행한다. DiffsHub 같은 도구가 “모든 것을 즉시 보여준다”로 차별화하는
이유다. 다음 세대 코드 호스팅 플랫폼이 GitHub 위치를 위협한다면, 그
공격 지점은 기능이 아니라 “에이전트 워크로드에 맞춘 새 UI”일 가능성이
높다.

### Safari와 WebKit이 차세대 웹 인프라의 보틀넥이 된다

저자의 마지막 P.S.는 우호적 톤이지만 내용은 무겁다. Sticky 컴포지팅
성능, 60Hz로 캡된 rAF, 추적이 어려운 프로파일러, 슬롯 컨테이너 주입의
레이아웃 버그. 이 목록은 “WebKit이 다른 두 브라우저 엔진과 같은 등급의
인프라가 아니다”라는 사실을 정확히 짚는다. Pierre 같은 한 회사만의
경험이 아니라 Figma, Linear, Notion, Replit, Vercel 같은 거의 모든 웹
인프라 회사가 공유하는 경험이다.

이 격차는 단순한 “브라우저 호환성”의 문제가 아니라 “플랫폼 사실상
독점의 비용”이다. macOS에서 Safari는 시스템 WebKit을 통해 거의
모든 “네이티브 like” 앱(Tauri, 시스템 WebView)의 엔진이고, iOS에서는
브라우저 엔진 선택의 자유 자체가 (최근 EU DMA 이전까지) 없었다.
이 상태에서 WebKit이 다른 엔진보다 느리면, 그 비용은 “Safari 사용자”
가 아니라 “모든 웹 앱 개발자”와 “Apple 기기 사용자 전체”에게 전가
된다.

여기서 도출되는 인사이트는 “플랫폼 사실상 독점 + 단일 엔진 + 더딘
업데이트”라는 조합이 차세대 웹 인프라의 진짜 보틀넥이 된다는 점이다.
Chrome과 Firefox는 다른 엔진 베더와 경쟁하므로 60Hz rAF 같은 문제는
며칠 안에 픽스가 들어간다. WebKit은 같은 인센티브가 약하고, 메이저
업데이트가 OS 업데이트에 묶여 있다. 이 격차는 시간이 지날수록 누적된다.

세 번째 함의는 정책의 영역이다. EU DMA가 iOS에서 대안 브라우저 엔진을
허용하기 시작했고, 미국·일본·한국에서도 유사한 압박이 늘고 있다.
WebKit 외 엔진이 iOS에 들어오면 시장 인센티브가 작동하기 시작하고,
60Hz rAF 같은 격차가 자동으로 메워질 가능성이 있다. 즉 Pierre 같은
회사의 “Safari, Please” 호소는 단순한 우호적 항의가 아니라 더 큰
정책 시계를 가속하는 데이터 포인트다. 인프라 엔지니어링이 정책과
교차하는 지점이 점점 늘어난다.
