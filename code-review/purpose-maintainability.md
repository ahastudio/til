# 코드 리뷰의 목적은 버그 발견이 아니라 유지보수성 확인이다

원문: <https://mathstodon.xyz/@mjd/115096720350507897>

HN 토론: <https://news.ycombinator.com/item?id=48759870> (338점, 169개 댓글)

## 요약

Mark Dominus(`@mjd@mathstodon.xyz`)는 2025년 8월 27일 Mastodon에 올린 포스트에서
코드 리뷰에 대한 지배적 오해를 정면으로 반박한다.
리뷰어가 버그를 찾아야 한다거나, 버그가 없음을 보장해야 한다는 생각은 착각이라는 것이다.
코드를 읽는 것만으로 버그를 발견하는 일은 “일반적으로 가능하지 않다”는 것이 그의 주장이다.
이 포스트는 1.1K 부스트, 1.6K 즐겨찾기를 기록하며 광범위한 공감을 얻었다.

코드 리뷰의 일차적 목적은 **유지보수하기 어려운 코드를 찾는 것**이라고 그는 말한다.
리뷰어는 코드를 읽으며 “이것이 무엇을 하는가, 어떻게 작동하는가”를 이해하려 한다.
이해가 안 된다면 그것은 미래에 유지보수하기 어렵다는 신호다.
원작자가 아직 코드를 잘 알고 있는 지금, 수정해야 한다.

후속 포스트(Sep 1, 2025)에서 그는 두 접근법의 실무적 차이를 구체화한다.
“이 코드에서 버그를 찾아라”는 지시는 성공이 보장되지 않는다.
얼마나 오래 해야 하는지, 언제 멈춰야 하는지 기준이 없다.
세 개의 버그를 찾았더라도 네 번째를 놓쳤다는 비판을 받을 수 있다.
반면 “이 코드를 이해하라, 이해 안 되는 부분이 있으면 메모하라”는 지시는 다르다.
나쁜 컨디션에도 할 수 있고, 성공이 거의 보장된다.
완료 기준도 명확하다 — 전체를 읽고 이해 안 된 부분을 모두 메모했으면 끝이다.

## 분석

### “버그 발견”과 “유지보수성 확인”은 작업 정의 자체가 다르다

Dominus의 논거의 핵심은 코드 리뷰의 목적 재정의가 아니라
**성공과 실패를 정의하는 방식의 전환**이다.
버그를 찾는 작업은 비대칭적 부담을 가진다.
찾지 못했을 때 “더 열심히 해야 했다”는 비판을 받지만, 찾았을 때 얼마나 찾아야 하는지의 기준이 없다.
실패는 있지만 성공은 불명확한 작업이다.

유지보수성 확인은 구조적으로 다르다.
“이 코드를 이해할 수 있는가”는 이진 질문이 아니라 연속 스펙트럼이다.
이해한 부분은 넘어가고, 이해 못 한 부분은 기록한다.
작업 완료 기준이 명확하고, 성공과 실패가 모두 정의된다.
작업의 심리적 부담이 크게 줄어드는 이유가 여기 있다.

이 차이는 리뷰어가 코드를 보는 시각 자체를 바꾼다.
“버그가 있는가”를 묻는 사람은 코드를 의심스러운 눈으로 훑는다.
“이해가 되는가”를 묻는 사람은 코드를 미래 유지보수자의 눈으로 읽는다.
후자의 시각이 장기적으로 코드베이스에 더 유익한 피드백을 만든다.

### “이해 가능성”이 품질의 대리 지표인 이유

버그를 직접 탐지하는 것과 이해 가능성을 확인하는 것은 측정 전략이 다르다.
버그는 코드의 행동에 관한 것이다 — 특정 입력에 대해 잘못된 출력을 낸다.
코드 읽기만으로 이것을 포착하는 일은 모든 가능한 실행 경로를 머릿속에서 시뮬레이션해야 하므로 어렵다.

이해 가능성은 코드의 표현에 관한 것이다 — 의도가 명확하게 드러나는가.
이것은 읽기만으로도 평가할 수 있다.
그리고 이해하기 어려운 코드는 버그를 숨기기 쉬운 코드라는 상관관계가 높다.
복잡한 조건문, 긴 함수, 불명확한 변수명이 버그의 온상이 되는 것은 우연이 아니다.

이해 가능성 확인이 버그 탐지의 **간접 경로**로 작동하는 셈이다.
리뷰어가 “이 로직이 이해가 안 된다”고 메모하면,
저자가 다시 살펴보면서 버그를 발견하는 경우가 많다.
직접 버그를 지목하지 않아도 코드 품질 향상으로 이어진다.

### 코드 리뷰의 다른 기능들이 이 프레임에서 어떻게 자리를 잡는가

코드 리뷰는 버그 탐지와 유지보수성 확인 외에도 다양한 기능을 한다.
지식 이전(knowledge transfer), 팀 컨벤션 교육, 팀 내 공유 이해 형성이 대표적이다.
tartley(@fosstodon.org)는 유지보수성 확인보다 더 중요한 것이 저자를 팀 컨벤션에 맞게 교육하는 것이라고 주장한다.
같은 실수를 미래 코드에서 반복하지 않도록 하는 효과다.

이 관점들은 Dominus의 프레임과 배타적이지 않다.
“코드를 이해하려 하라”는 지시 안에 지식 이전과 컨벤션 교육이 자연스럽게 포함된다.
리뷰어가 “이 패턴은 우리 팀에서 다르게 쓴다”고 메모하는 것이 대표적이다.
Dominus의 프레임은 버그 탐지를 내리고 이해 가능성 확인을 올리는 재배치지,
다른 기능들을 부정하지 않는다.

sjburt는 이것을 소유권 이전으로 표현한다.
코드 리뷰는 코드가 저자의 소유에서 팀 또는 프로젝트의 소유로 넘어가는 관문이라는 것이다.[^sjburt]
이 프레임은 Dominus의 유지보수성 강조와 자연스럽게 맞닿는다.
유지보수성이 중요한 이유는 결국 코드가 팀 공유 자산이 되기 때문이다.
리뷰어가 이해하지 못하면, 팀 자산으로서의 코드 소유권 이전이 완성되지 않은 것이다.

donatj는 이 맥락에서 '기관 지식 확인(institutional knowledge check)'이라는 표현을 쓴다.[^donatj]
공유 테이블에 접근하는 알려지지 않은 마이크로서비스를 팀원이 리뷰 중에 발견해 잠재적 데이터 문제를 막은 사례다.
이것은 유지보수성도, 버그 탐지도 아닌 세 번째 카테고리다 — 누구도 명시적으로 확인하지 않았던 시스템 맥락이 리뷰를 통해 공유된 것이다.
이런 종류의 지식은 코드 자체에서는 보이지 않으며, 리뷰어의 시스템 전반에 대한 이해가 있어야 포착된다.

## 비평

### “코드 읽기만으로 버그를 찾는 것은 일반적으로 불가능하다”는 주장은 지나치게 강하다

Dominus는 “it is not in general possible to find bugs by examining the code”라고 선언한다.
그러나 이것은 경험과 맞지 않는 극단적 일반화다.
단순한 오타, 경계 조건 누락, 명백한 null 처리 부재는 코드 읽기만으로도 충분히 발견된다.
이런 버그들이 현실에서 코드 리뷰로 잡히는 비율은 무시할 수 없다.

diazona(@techhub.social)가 지적한 것처럼, 버그 탐지를 리뷰의 주목적에서 배제하는 것과
버그 탐지가 불가능하다고 말하는 것은 다른 주장이다.
후자는 전자를 지지하는 논거로 쓰이지만, 전자보다 훨씬 강한 주장이다.
전자(주목적 재정의)는 타당하지만, 후자(불가능성 주장)는 과장이다.

HN 스레드에서 mjd 자신이 이 점을 해명한다.
그는 수학자로서 “in general”이라는 표현을 사용했으며, 이는 수학적 의미의 일반적 불가능성을 뜻한다.[^mjd]
특정 조건에서 버그를 찾는 것이 가능하지만, 코드 검토만으로 버그 부재를 보장하는 절차가 존재하지 않는다는 것이다.
이 해석에서 원 주장은 과장이 아니라 수학적으로 정밀한 언어의 자연스러운 표현이 된다.

그러나 이것이 오해를 막지 못했다는 사실 자체가 표현의 실패다.
수학적 정밀도는 소셜 미디어 청중을 전제하지 않는다.
Dominus가 Mastodon 포스트에서 수학적 언어를 사용했을 때,
독자들은 “일반적으로 불가능하다 = 실질적으로 불가능하다”로 읽었고 반박이 그 지점에 집중됐다.
초기 주장의 설득력은 이 수사적 강도에 기대어 있고, 해명은 논쟁이 불붙은 뒤에야 왔다.

### 보안과 정확성 검토가 유지보수성 프레임에서 밀려나는 위험

batsify(@ruby.social)가 반박한 것처럼, 코드 리뷰는 보안 위험, 성능 문제, 취약한 테스트를
발견하는 실질적 기능을 한다.
이것들은 “이해 가능성”으로 환원되지 않는 독립적 검토 항목이다.
SQL 인젝션 취약점이나 경쟁 조건(race condition)은 코드가 읽기 쉬워도 존재할 수 있다.

“유지보수성 확인”을 일차 목적으로 설정하면,
보안과 정확성 검토는 이차적인 것으로 밀린다.
팀이 이 프레임을 그대로 수용하면, 리뷰어가 보안 항목을 “그건 내 일이 아니다”로 처리할 근거가 생긴다.
이것은 Dominus의 의도가 아닐 것이지만, 프레임이 가져올 수 있는 실무적 위험이다.

역할 분리라는 반론이 있다.
정적 분석 도구, 린터, 자동화 보안 스캐너가 이미 많은 부분을 담당한다는 것이다.
그러나 도구가 없는 팀, 도구가 충분히 설정되지 않은 팀, 도구가 잡지 못하는 맥락 특수적 버그에서
코드 리뷰는 여전히 마지막 방어선이다.
“일차 목적이 아니다”가 “신경 쓰지 않아도 된다”로 해석될 여지는 항상 있다.

### 이 프레임이 전제하는 팀 구성이 보편적이지 않다

Dominus의 논거는 암묵적으로 특정 조건을 가정한다.
리뷰어와 저자가 같은 코드베이스에서 장기적으로 함께 일하며,
유지보수는 팀 내부에서 이루어지고,
자동화 테스트가 충분히 갖춰져 있다는 조건이다.

오픈소스 프로젝트에서 외부 기여자의 PR을 리뷰하는 메인테이너는 다른 상황에 있다.
기여자가 미래에 유지보수에 참여할지 모르고, 코드를 이해하기 어렵더라도 거절할 명확한 이유가 필요하다.
컨설팅이나 외주 형태로 납품된 코드를 발주사가 검수하는 상황도 다르다.
이 맥락들에서는 버그와 정확성 검토가 유지보수성 확인보다 중요도가 높을 수 있다.

“코드 리뷰의 일차 목적”을 하나로 선언하는 시도 자체가
조직의 종류, 팀의 구성, 코드베이스의 성격을 무시하는 일반화다.
Dominus의 주장은 그가 경험한 특정 환경에서는 강하게 타당하지만,
그것이 보편 원칙으로 제시될 때 설득력의 범위를 넘어선다.

brimtown이 제시한 '주인-대리인(principal-agent)' 프레임이 이를 구조화한다.[^brimtown]
고신뢰 환경(소규모 팀)과 저신뢰 환경(대기업, 오픈소스 프로젝트)에서 코드 리뷰의 해법은 달라야 한다.
Dominus의 논거는 암묵적으로 고신뢰 환경을 전제한다.
팀원이 서로의 의도를 신뢰하고, 리뷰는 협력적 개선을 위한 것이며, 코드베이스 전체에 대한 공유 책임이 있다는 가정이다.
저신뢰 환경에서는 버그 탐지, 보안 검토, 정확성 확인이 유지보수성보다 더 긴급한 목적이 된다.

## 인사이트

### 작업을 어떻게 정의하는가가 팀의 책임 구조와 심리적 안전을 바꾼다

“버그를 찾아라”와 “이해하고 메모하라”는 같은 코드를 읽는 행위를 서로 다른 책임 구조에 놓는다.
전자는 리뷰어에게 완전성의 책임을 부과한다.
놓친 버그는 리뷰어의 실패다.
이것은 리뷰어를 방어적으로 만들거나, 리뷰를 두려움의 대상으로 만든다.

후자는 이해 가능성이라는 주관적이고 달성 가능한 기준을 제시한다.
리뷰어는 자신이 이해한 것과 이해 못 한 것을 솔직하게 말할 수 있다.
이것은 심리적 안전(psychological safety)과 직결된다.
팀이 코드 리뷰를 품질 통제 관문이 아닌 협업 도구로 경험하게 하려면,
작업의 정의가 이 방향으로 이동해야 한다.

이 재정의의 효과는 리뷰어에만 머물지 않는다.
저자도 리뷰어가 “이게 뭔지 모르겠다”고 말하는 것을 개인 공격이 아닌 유지보수성 신호로 받아들이게 된다.
“내가 이해 못 하는 것”이 “코드가 나쁘다”가 아니라 “코드가 더 명확해져야 한다”로 번역되는 문화가 만들어진다.

### 정적 분석 도구의 성숙이 코드 리뷰의 역할 이동을 이미 촉진하고 있다

린터, 정적 분석기, 타입 검사기, 자동화 보안 스캐너가 성숙해지면서
리뷰어가 직접 찾아야 했던 버그들이 도구로 이동했다.
미사용 변수, 타입 불일치, 알려진 패턴의 보안 취약점은 이미 도구가 잡는다.
남은 것은 도구가 잡지 못하는 것들 — 비즈니스 로직의 정확성, 설계 의사결정, 유지보수성이다.

Dominus의 주장은 이 기술적 흐름을 앞서 인식한 것으로 읽힌다.
“리뷰어가 버그를 잡아야 한다”는 기대는 자동화 이전 시대의 유산이다.
도구가 할 수 있는 것을 사람에게 시키는 것은 비효율이다.
리뷰어의 고유한 기여는 코드가 **이 팀, 이 맥락, 이 시스템**에서 작동하는 방식을 이해하는 것이다.
그것은 도구가 대체하지 못하는 영역이다.

이 흐름의 자연스러운 귀결은 코드 리뷰의 역할이 시간이 지날수록 Dominus의 주장 쪽으로 실제로 이동한다는 것이다.
논쟁이 어느 쪽이 옳은가 보다, 어느 쪽이 현재의 도구 환경에 맞는가로 재구성돼야 한다.

kasey_junk는 LLM이 이 흐름을 가속한다고 지적한다.[^kasey_junk]
GitHub 스타일의 비동기 PR 리뷰는 자동화가 잡지 못하는 것들을 인간이 직접 보는 경로였다.
LLM이 코드를 즉시 분석하고 문제를 제안할 수 있다면, 비동기 PR 리뷰의 비용 대비 이점이 LLM 이전보다 줄어든다.
남는 것은 LLM도 대체하기 어려운 영역이다 — 이 팀, 이 맥락, 이 시스템에서의 판단.
Dominus의 유지보수성 프레임은 정확히 그 영역을 겨냥하고 있다.

### “완료 기준”의 명확성이 리뷰 문화의 지속 가능성을 결정한다

코드 리뷰가 팀에서 잘 작동하지 않는 경우를 분석해보면,
많은 경우 리뷰어가 얼마나 깊이 파야 하는지 모른다는 문제로 수렴한다.
시간이 없어서 빠르게 LGTM을 누르거나, 완벽주의적으로 파고들어 리뷰가 병목이 된다.
이 두 극단은 리뷰의 “완료”를 정의하지 못한 데서 비롯된다.

“이해하고 이해 못 한 것을 메모한다”는 정의는 완료 기준을 제공한다.
전체를 읽었고, 이해 못 한 부분을 모두 기록했으면 리뷰는 끝이다.
이것은 LGTM 무기력증과 완벽주의적 병목 모두를 해소하는 실용적 기준이다.

더 깊은 함의는 이것이 리뷰의 **반복 가능성**을 만든다는 것이다.
명확한 완료 기준이 있으면 팀이 리뷰를 습관으로 지속할 수 있다.
“버그를 찾아야 한다”는 정의 아래서는 시간이 부족하면 리뷰를 건너뛰게 된다.
“이해하면 된다”는 정의 아래서는 30분짜리 리뷰라도 의미 있는 리뷰가 된다.
지속 가능한 관행은 완벽한 것보다 반복 가능한 것이다.
