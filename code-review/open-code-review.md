# Open Code Review (alibaba/open-code-review)

<https://alibaba.github.io/open-code-review/>

<https://github.com/alibaba/open-code-review>

HN 토론: <https://news.ycombinator.com/item?id=48406358> (284점, 73개 댓글)

## 소개

Open Code Review(`ocr`)는 알리바바 그룹이 내부 시스템에서 출발해 공개한 AI 기반 코드 리뷰 CLI 도구다.
Git diff를 읽어 변경된 파일을 도구 사용(tool-use) 능력을 갖춘 에이전트를 통해 설정 가능한 LLM에 보내고, 라인 단위 정밀도를 가진 구조화된 리뷰 코멘트를 생성한다.
Apache-2.0 라이선스이며, 2026년 6월 스냅숏 기준 GitHub 스타 약 8.2천 개, 포크 507개, 최신 버전은 v1.3.19다.
코드베이스는 Go 74.3%, TypeScript 14.7%로 구성되어 있다.

이 프로젝트의 출발점은 “범용 에이전트로 코드 리뷰를 시키면 생기는 문제들”을 명시적으로 겨냥한다.
저장소는 세 가지 문제를 든다.
첫째, 변경 규모가 커지면 에이전트가 “요령을 피워(cut corners)” 일부 파일만 골라 리뷰하고 나머지를 빠뜨리는 불완전한 커버리지.
둘째, 보고된 이슈의 위치가 실제 코드와 어긋나 라인 번호나 파일 참조가 표류하는 포지션 드리프트.
셋째, 자연어 기반 Skill은 디버깅이 어렵고 사소한 프롬프트 변화에도 리뷰 품질이 크게 출렁이는 불안정성이다.

해법은 결정론적 엔지니어링과 LLM 에이전트를 결합한 하이브리드 구조다.
결정론적 요소(하드 제약)는 어떤 변경도 놓치지 않는 정밀한 파일 선택, 관련 파일을 한 리뷰 단위로 묶는 스마트 번들링, 템플릿 엔진을 통한 세밀한 규칙 매칭, 외부 포지셔닝·리플렉션 모듈을 담당한다.
에이전트 요소(동적 판단)는 코드 리뷰에 맞춰 튜닝된 프롬프트와 프로덕션 데이터 분석에서 추출한 전용 도구셋을 맡는다.

저장소는 50개 오픈소스 저장소, 200개 풀 리퀘스트, 10개 프로그래밍 언어로 구성한 벤치마크를 제시한다.
이 벤치마크에서 `ocr`은 Claude Code보다 정밀도(precision)와 F1 점수가 유의미하게 높으면서 토큰은 약 “1/9”만 소비한다고 주장한다.
설계상 재현율(recall)을 일부 희생하는 대신 정밀도를 택해, 거짓 양성(false positive)을 줄이는 쪽으로 의도적으로 기울였다.
빌트인 규칙셋은 NPE, 스레드 안전성, XSS, SQL 인젝션 등을 점검하며, OpenAI·Anthropic 호환 LLM, GitHub Actions·GitLab CI 통합, 기본 동시성 8의 병렬 리뷰, `localhost:5483`의 웹 UI 뷰어를 제공한다.

## 분석

### 핵심 주장은 “에이전트만으로는 코드 리뷰가 안 된다”는 진단이다

이 프로젝트의 논리적 출발점은 도구 자랑이 아니라 경쟁 접근법에 대한 진단이다.
저자는 Claude Code 같은 범용 코딩 에이전트에게 리뷰를 통째로 맡기면 커버리지 누락, 위치 표류, 품질 불안정이라는 세 가지 실패가 구조적으로 발생한다고 주장한다.
그리고 그 원인을 “자연어와 자율 판단에 모든 것을 위임한 탓”으로 지목한다.

이 진단이 설득력을 갖는 이유는 세 문제가 모두 LLM의 알려진 약점과 정확히 대응하기 때문이다.
긴 컨텍스트에서의 주의 분산은 커버리지 누락으로, 토큰 위치와 소스 위치의 불일치는 포지션 드리프트로, 샘플링의 확률적 성격은 품질 불안정으로 이어진다.
즉 저자는 LLM을 “더 똑똑하게” 만들려 하지 않고, LLM이 잘 틀리는 지점을 결정론적 코드로 둘러싸 가두는 전략을 택했다.

따라서 Open Code Review의 본질은 코드 리뷰 도구이기 이전에 “에이전트를 신뢰할 수 없는 부품으로 간주하고 그 주위에 신뢰할 수 있는 골격을 세운” 엔지니어링 사례다.
이 골격이 바로 결정론적 파일 선택과 외부 포지셔닝 모듈이며, 에이전트는 그 안에서 “이 코드가 문제인가”라는 판단만 맡는다.

### 정밀도를 위해 재현율을 버린 선택은 리뷰 도구의 본질을 건드린다

저자는 재현율을 의도적으로 낮추고 정밀도를 높였다고 명시한다.
이는 단순한 튜닝 결정이 아니라 코드 리뷰 자동화의 가장 어려운 트레이드오프에 대한 입장 표명이다.
거짓 양성이 많은 리뷰 봇은 개발자가 곧 무시하게 되고, 무시되는 순간 도구는 죽는다 — 이른바 “경보 피로(alert fatigue)”다.

이 선택의 배경에는 “놓친 버그보다 잘못된 지적이 더 비싸다”는 가정이 깔려 있다.
잘못된 지적은 개발자의 시간을 직접 빼앗고 도구 전체의 신뢰를 갉아먹지만, 놓친 버그는 어차피 기존의 테스트·QA·다른 리뷰어가 잡을 수 있다는 논리다.
토큰을 1/9만 쓴다는 효율 주장도 같은 철학의 연장선이다 — 모든 것을 다 보려 하지 않고, 결정론적으로 추린 핵심만 에이전트에 보내기 때문에 비용이 낮아진다.

이 균형점은 알리바바 같은 대규모 조직의 맥락에서 합리적이다.
하루에 수천 건의 PR이 올라오는 환경에서 리뷰 봇의 거짓 양성률이 높으면 전사적으로 막대한 시간 낭비가 발생한다.
정밀도 우선 설계는 “자동 리뷰는 보조일 뿐 최종 책임은 사람”이라는 전제 위에서만 정당화되며, 도구는 이 전제를 명시하지 않은 채 깔고 있다.

### 4계층 규칙 우선순위는 조직 단위 거버넌스를 겨냥한다

`ocr`의 규칙 해석은 CLI `--rule` 플래그, 프로젝트 설정(`.opencodereview/rule.json`), 사용자 설정(`~/.opencodereview/rule.json`), 시스템 기본(`system_rules.json`)의 4계층 우선순위로 결정된다.
이 구조는 개인 개발자 도구가 아니라 조직 단위 배포를 전제한다.
시스템 기본 규칙으로 회사 전체의 최소 기준을 깔고, 프로젝트와 사용자가 그 위에 덧칠하는 방식은 ESLint·Checkstyle 같은 전통적 정적 분석 도구의 설정 계층과 정확히 같은 계보다.

이 설계는 Open Code Review가 “AI 코드 리뷰”라는 신기술을 표방하면서도 실제로는 기존 린터의 거버넌스 모델을 그대로 계승하고 있음을 보여준다.
다른 점은 규칙의 판정 주체가 정규식·AST 매칭이 아니라 LLM이라는 것뿐이다.
즉 이 도구는 정적 분석의 운영 모델 위에 LLM이라는 새 판정 엔진을 끼워 넣은 형태다.

다만 HN에서 여러 사용자가 지적했듯, 이 도구의 차별점이 무엇인지부터가 논쟁거리다.
“Codex가 이미 있는데 다른 탭에서 코드 리뷰를 시키면 되지 않나”라는 반문이 나왔고,[^faangguyindia] 실제로 한 사용자는 사내 해커톤에서 노드 이미지에 Claude Code를 설치해 `/review` 류 명령으로 인라인 코멘트를 달고 재실행 시 옛 코멘트를 지우는 봇을 직접 만들었다며 “Git 안에서 AI 코드 리뷰를 하는 것 자체는 어렵지 않고 빠르게 가치를 낸다”고 평가절하했다.[^elpakal] 또 다른 사용자는 Claude Code를 쓰면서 Codex·Cursor·Gemini에게 교차 리뷰를 시키는 `/meta-review` 스킬을 직접 만들어 쓴다고 했다.[^atestu] 이 반응들은 “LLM에 리뷰를 시키는 것” 자체가 이미 누구나 며칠이면 조립하는 상품(commodity)이 되었음을 드러내며, Open Code Review의 진짜 해자는 에이전트 호출이 아니라 결정론적 골격과 규칙셋에 있음을 역설적으로 확인시킨다.

## 비평

### 벤치마크 수치는 출처와 방법이 검증 불가능하다

저자는 50개 저장소, 200개 PR, 10개 언어 벤치마크에서 Claude Code 대비 높은 정밀도·F1과 1/9 토큰을 주장한다.
그러나 이 수치는 도구 제작자가 자사 도구를 평가한 자가 측정이며, 정밀도와 재현율을 가르는 “정답(ground truth)”을 누가 어떻게 정했는지가 핵심인데 그것이 드러나지 않는다.
코드 리뷰에서 “진짜 버그”의 정의 자체가 주관적이므로, 정답 라벨링 주체가 도구 제작자라면 자사 도구가 잡는 이슈를 정답으로 간주하는 순환 논증에 빠지기 쉽다.

비교 대상 설정도 의심스럽다.
Claude Code는 범용 코딩 에이전트이지 코드 리뷰 전용 도구가 아니다.
리뷰 전용으로 튜닝된 도구가 범용 도구보다 리뷰 벤치마크에서 잘 나오는 것은 당연하며, 이는 “전용 도구가 범용 도구를 이긴다”는 동어반복에 가깝다.
공정한 비교라면 같은 리뷰 전용 도구(CodeRabbit, Qodo PR-Agent, GitHub Copilot 리뷰 등)와 겨뤄야 하지만, 저장소는 그 비교를 보여주지 않는다.

토큰 1/9 주장 역시 측정 조건에 크게 의존한다.
결정론적 모듈이 파일을 추려 LLM에 적게 보내면 당연히 토큰이 줄지만, 그만큼 LLM이 보지 못한 코드에서 발생하는 누락은 토큰 효율 지표에 잡히지 않는다.
효율과 커버리지는 같은 동전의 양면인데, 저자는 효율만 수치로 제시하고 그 대가인 누락은 “정밀도 우선”이라는 서사로 정당화한다.

HN의 한 사용자는 이 도구를 제3자 벤치마크(`codereview.withmartian.com`)의 50개 PR 중 10개에 직접 돌려본 결과를 공유했는데, 저장소의 자가 측정과 정반대 그림이 나왔다.[^eranation]
재현율은 약 74%로 “골든 이슈”를 꽤 잡아냈지만 정밀도는 약 12%에 그쳐 거짓 양성이 쏟아졌고, 그 탓에 F1이 약 20%로 주저앉아 “이 표본이 전체 50개로 확장되면 거의 꼴찌, Kilo+Grok보다도 낮을 것”이라는 평이었다.
저장소는 “정밀도를 위해 재현율을 버렸다”고 했는데 독립 측정에서는 정확히 반대 — 재현율은 높고 정밀도는 낮은 — 결과가 나온 것이다.
이는 벤치마크 수치가 측정 주체와 정답 기준에 얼마나 좌우되는지를 단적으로 보여주며, 자가 측정 F1 우위 주장을 신뢰하기 어렵게 만든다.

### “파일을 놓치지 않는다”는 약속과 재현율 포기는 모순처럼 읽힌다

저장소는 결정론적 파일 선택으로 “어떤 변경도 놓치지 않는다”고 강조하면서, 동시에 재현율을 의도적으로 낮췄다고 말한다.
이 두 주장은 표면적으로 충돌한다.
모든 파일을 빠짐없이 본다면서 어떻게 재현율이 낮을 수 있는가.

해소는 “파일 커버리지”와 “이슈 재현율”을 구분하는 데 있다 — 모든 파일을 입력으로 넣되, 그중 실제 결함의 일부만 보고하도록 정밀도 임계값을 높였다는 뜻이다.
그러나 이 구분은 저장소 문구에서 명확히 드러나지 않으며, 마케팅은 “놓치지 않는다”는 강한 메시지와 “적게 보고한다”는 보수적 메시지를 동시에 내세워 독자가 유리한 쪽으로 해석하도록 방치한다.
실제로 중요한 것은 파일을 다 읽었는지가 아니라 진짜 버그를 잡았는지인데, 후자는 정의상 낮춘 재현율에 의해 제한된다.

### 정밀도 우선 설계는 가장 위험한 버그를 놓치는 방향으로 편향된다

거짓 양성을 줄이려 임계값을 높이면, 모델이 확신하지 못하는 미묘한 이슈는 보고되지 않는다.
문제는 가장 위험한 버그일수록 미묘하다는 점이다.
명백한 NPE나 SQL 인젝션은 정규식 린터로도 잡히지만, 경쟁 조건(race condition), 미묘한 인증 우회, 도메인 로직의 논리 오류 같은 고위험 결함은 맥락 의존적이고 확신도가 낮아 정밀도 우선 필터에서 탈락하기 쉽다.

이는 정밀도 우선 설계의 구조적 맹점이다.
도구는 “잡기 쉬운 흔한 버그”에서 높은 정밀도를 보이며 신뢰를 쌓지만, 정작 사람이 놓치기 쉬워 자동화가 가장 필요한 영역에서는 침묵한다.
빌트인 규칙셋이 NPE·스레드 안전성·XSS·SQL 인젝션 같은 “카탈로그화된” 취약점에 집중되어 있다는 사실이 이 편향을 뒷받침한다 — 이들은 모두 패턴이 알려진 결함이지, 그 코드베이스 고유의 논리 오류가 아니다.

결국 정밀도 우선 도구는 “알려진 함정을 자동으로 점검하는 체크리스트”로서는 탁월하지만, “이 설계가 옳은가”를 묻는 진짜 리뷰는 여전히 사람의 몫으로 남긴다.
저장소는 이 경계를 명시하지 않은 채 “코드 리뷰 도구”라는 포괄적 이름을 쓴다.

### 핵심 자산인 규칙셋이 중국어로만 공개된 점은 글로벌 표방과 어긋난다

이 규칙셋을 외부에서 들여다보려는 순간 또 다른 장벽이 드러난다.
HN 사용자들은 핵심 자산인 규칙 파일이 전부 중국어로만 작성되어 있어 번역기를 돌려야 내용을 파악할 수 있었다고 지적했고,[^nutifafa] 급기야 한 사용자는 구글 번역으로 규칙 파일을 영어로 옮긴 별도 저장소를 만들어 공유하기에 이르렀다.[^pramodbiligiri]
4계층 거버넌스로 전 세계 조직의 채택을 노린다면서, 정작 그 거버넌스의 내용물인 규칙은 영어권 사용자가 읽을 수 없는 상태로 공개한 것이다.

이는 단순한 번역 누락이 아니라 설계의 무게중심이 어디에 있는지를 드러내는 균열이다.
규칙은 “무엇이 결함인가”를 정의하는 도구의 두뇌인데, 그 두뇌가 알리바바 내부(중국어 환경)의 관행과 코딩 컨벤션을 전제로 작성되었다면, 영어권·타 문화권 조직이 그대로 가져다 쓸 때 의미가 어긋날 수 있다.
“글로벌 오픈소스”라는 외피와 “내부 도구의 공개”라는 실체 사이의 거리가 바로 이 지점에서 드러난다.

## 인사이트

### LLM을 신뢰 불가능한 부품으로 격하하는 설계가 에이전트 시대의 주류가 된다

Open Code Review의 가장 중요한 메시지는 코드 리뷰 자체가 아니라 “에이전트를 어떻게 가둘 것인가”에 있다.
이 도구는 LLM의 자율성을 늘리는 대신 줄인다 — 파일 선택, 번들링, 포지셔닝을 결정론적 코드가 빼앗고, LLM에게는 “이 좁은 입력이 문제인가”라는 최소 판단만 남긴다.
이것은 “더 강력한 에이전트가 더 좋다”는 2024–2025년의 지배적 서사와 정면으로 어긋난다.

여기에 글이 예상하지 못한 2차 효과가 있다.
이런 “결정론적 골격 + 좁은 LLM 판단” 패턴이 코드 리뷰에서 성공하면, 같은 패턴이 테스트 생성·마이그레이션·문서화 등 다른 코딩 작업으로 번질 것이다.
즉 산업은 “자율 에이전트”에서 “제약된 에이전트”로 회귀하며, LLM은 워크플로의 지휘자가 아니라 잘 정의된 슬롯에 끼워지는 확률적 함수로 자리매김한다.
이는 초기 머신러닝이 “end-to-end 학습”의 환상을 거쳐 결국 피처 엔지니어링과 결합한 역사와 닮았다 — 순수 학습이 아니라 도메인 지식으로 감싼 학습이 실전에서 이긴다.

이 흐름이 굳어지면 “에이전트 프레임워크”의 가치는 자율성이 아니라 구속의 정교함에서 나온다.
가장 좋은 도구는 LLM에게 가장 많은 자유를 주는 것이 아니라, LLM이 틀릴 수 있는 모든 지점을 정확히 식별해 결정론으로 봉쇄한 것이 된다.

### 정밀도 우선 리뷰 봇은 “채워지지 않는 안전감”이라는 함정을 만든다

이 도구는 거짓 양성을 줄여 개발자가 무시하지 않게 만드는 데 성공할 수 있다.
그러나 시간이 지나면서 발생하는 미묘한 트레이드오프가 있다 — 조용하고 정확한 봇은 “통과”를 “안전”으로 오해하게 만든다.
ArchUnit 같은 구조 검증 도구가 초록색 테스트를 “아키텍처 건강”으로 착각하게 만들 듯, 정밀도 우선 리뷰 봇은 “지적 없음”을 “문제 없음”으로 읽히게 한다.

규모와 시간이 누적되면 이 효과는 위험해진다.
팀은 점차 봇이 통과시킨 PR을 사람이 깊이 보지 않게 되고, 봇이 구조적으로 침묵하는 영역(맥락 의존적 고위험 버그)은 이중으로 사각지대가 된다 — 봇도 안 보고 사람도 안 본다.
정밀도를 높일수록 봇의 신뢰가 올라가고, 신뢰가 올라갈수록 사람의 주의가 빠지며, 주의가 빠질수록 봇의 사각지대가 위험해진다.

이 역설은 모든 “믿을 만한 자동 검증”이 공유하는 함정이다.
검증이 신뢰받을수록 인간의 백업이 사라지고, 검증이 못 보는 곳에서 사고가 난다.
정밀도 우선 설계는 단기적으로 채택률을 높이지만, 장기적으로는 팀이 “봇이 안 잡은 영역”을 능동적으로 의식하도록 별도의 문화적 장치를 요구한다 — 그런데 도구는 그 장치를 제공하지 않는다.

### 알리바바의 오픈소스화는 도구가 아니라 “리뷰 규칙 데이터”를 노린 포석일 수 있다

알리바바가 내부 도구를 Apache-2.0으로 공개하고 8.2천 스타를 모은 것은 단순한 선의로만 보기 어렵다.
코드 리뷰 도구의 진짜 자산은 코드가 아니라 “무엇이 결함인가”를 정의하는 규칙셋과, 그 규칙을 다듬는 데 쓰인 프로덕션 리뷰 데이터다.
오픈소스화는 이 규칙셋을 전 세계 개발자가 사용·확장하게 만들어, 알리바바가 자사 내부 데이터만으로는 얻기 어려운 다양한 언어·도메인의 리뷰 시나리오를 사실상 크라우드소싱하는 통로가 된다.

여기서 역사적 패턴이 떠오른다 — 구글의 TensorFlow, 메타의 React, 알리바바 자신의 Vue 후원이 그렇듯, 인프라성 도구의 오픈소스화는 생태계 표준을 선점해 자사 클라우드·상용 제품으로 흐름을 유도하는 전략이었다.
Open Code Review가 OpenAI·Anthropic 호환을 내세우면서도 OpenTelemetry 통합, 4계층 거버넌스, 엔터프라이즈 친화적 CI 통합을 갖춘 것은, 결국 이 도구를 채택한 조직이 알리바바 클라우드의 LLM·관측 서비스로 자연스럽게 연결되도록 설계된 깔때기일 수 있다.

이 관점에서 보면 “1/9 토큰” 효율 주장은 기술 자랑을 넘어 상업적 메시지다.
LLM 추론 비용이 채택의 가장 큰 장벽인 상황에서, 비용을 9배 낮춘다는 약속은 “우리 방식으로 우리 클라우드에서 돌리면 싸다”는 영업 논리로 직결된다.
오픈소스 코드 리뷰 도구의 경쟁은 표면적으로 리뷰 품질을 다투지만, 그 아래에서는 LLM 추론 시장의 트래픽을 누가 가져가느냐를 다투는 대리전이다.
