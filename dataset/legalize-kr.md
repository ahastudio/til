# Legalize KR - 대한민국 법령과 판례를 Git 저장소로 관리

> 원문: <https://legalize.kr/>
>
> 관련 저장소:
>
> - <https://github.com/legalize-kr/legalize-kr> (법령)
> - <https://github.com/legalize-kr/precedent-kr> (판례)

## 요약

### 핵심 개념

Legalize KR은 대한민국의 법령과 판례를 “코드처럼” 관리하는 프로젝트다.
핵심 철학은 명확하다. “모든 법령은 Markdown 파일, 모든 개정은 Git 커밋.”
“모든 판례는 Markdown 파일, 모든 판결은 Git 히스토리.” 법령의 개정 이력을
`git log`로 탐색하고, 개정 내용을 `git diff`로 비교하며, 전체 법령에서
`grep`으로 키워드를 검색한다. PDF를 비교하거나 요약본에 의존할 필요가
없어진다.

### 데이터 출처와 수집

모든 데이터는 국가법령정보센터(law.go.kr) OpenAPI에서 수집된다. 항상 공식
원문을 기반으로 하며, 자체 해석이나 가공이 아니라 원문을 마크다운으로
정제한 형태다. 각 법령 파일에는 YAML Frontmatter가 포함되어 제목, 법령MST,
법령ID, 법령구분, 소관부처, 공포일자, 공포번호, 시행일자, 법령분야, 상태,
출처 URL 같은 메타데이터를 제공한다.

### 법령 저장소(legalize-kr) 구조

법령은 `kr/{법령명}/` 디렉토리에 저장된다. 디렉토리명은 법령명에서 띄어
쓰기를 제거한 형태(예: `민법`, `친일반민족행위자재산의국가귀속에관한특별법`)
로, 법령 홈페이지의 URL 규칙과 동일하다. 각 디렉토리에는 `법률.md`(국회
제정 법률), `시행령.md`(대통령령), `시행규칙.md`(부령), `대통령령.md`
(독립 대통령령)가 들어간다. 관련 법령이 하나의 디렉토리에 함께 관리되어,
법률과 시행령, 시행규칙을 한눈에 볼 수 있다.

### 판례 저장소(precedent-kr) 구조

판례는 사건 종류(민사, 형사, 일반행정, 세무, 특허, 가사, 선거·특별, 기타)
와 법원 등급으로 분류되어 사건번호 단위의 마크다운 파일로 저장된다. 선고
일이 Git Commit Date로 설정되어 `git log`로 시간순 탐색이 가능하다.

### 사용법

Git 명령어를 그대로 활용한다. `cat kr/민법/법률.md`로 민법 현재 내용을
보고, `git log -- kr/민법/`으로 개정 이력을 보고, `grep -r “개인정보” kr/`
로 전체 법령에서 키워드를 검색하고, `git log --before=“2025-01-01” -1 --
kr/민법/법률.md`로 특정 날짜의 법령 상태를 확인한다. 일반적인 개발자
도구(VS Code, GitHub Desktop, SourceTree)로 법령 변경을 추적할 수 있다.

### 영감과 국제 비교

legalize(스페인 법령 Git 프로젝트)에서 영감을 받았으며, legalize.dev라는
글로벌 프로젝트와 연결된다. 즉, 법령을 Git으로 관리한다는 아이디어는 한국
고유의 것이 아니라 국제적 흐름이며, Legalize KR은 이를 한국에 적용한
버전이다.

### 주의사항

법령 파싱 파이프라인이 개선될 경우 전체 히스토리를 재구성하기 위해
force-push가 실행될 수 있으며, 이 경우 모든 commit hash가 변경된다.
fork하거나 참조하는 경우 `git fetch --all && git reset --hard origin/main`
으로 동기화해야 한다.

## 분석

### “법령 = 코드”라는 메타포의 깊이

“법령을 코드처럼 관리한다”는 표현은 단순한 비유가 아니라 구조적 유사성에
근거한다. 법령은 코드와 놀라울 정도로 비슷한 특성을 갖는다. 버전이 있다
(개정). 의존성이 있다(시행령은 법률에 의존). 분기가 있다(부칙, 경과
조치). 테스트가 있다(판례는 법령의 “실행 결과”). 그리고 변경이 엄격히
추적되어야 한다(관보 고시, 시행일).

Git은 이 모든 특성을 자연스럽게 표현할 수 있다. 개정은 커밋, 개정 비교는
diff, 과거 버전 조회는 checkout, 관련 법령 묶음은 디렉토리, 메타데이터는
YAML frontmatter. 이 매핑이 성립하는 이유는 Git이 본래 “시간에 따라
변하는 텍스트 파일 집합”을 관리하기 위해 만들어졌기 때문이며, 법령이
정확히 그런 대상이기 때문이다.

### 국가법령정보센터 대비 차별화

국가법령정보센터(law.go.kr)는 공식 법령 DB이지만, 개발자에게는 불편하다.
웹 UI에서 텍스트를 복사하면 서식이 깨지고, API는 XML 응답이며, 개정
비교는 요약본만 제공된다. Legalize KR은 이 모든 것을 마크다운 파일과
Git diff로 대체한다. 이는 법률 전문가가 아니라 개발자를 1차 사용자로
설정한 디자인이다.

더 중요한 차이는 “로컬 복제”가 가능하다는 것이다. `git clone`으로 전체
법령을 로컬에 가져오면, 인터넷 연결 없이 법령을 검색하고 탐색할 수
있다. 이는 국가법령정보센터의 웹 기반 모델과 근본적으로 다른 접근이며,
특히 AI 도구(Claude, ChatGPT, NotebookLM)에 법령을 투입하려는 사용자
에게 결정적 이점이다.

### 판례 저장소의 의미

법령뿐 아니라 판례까지 Git으로 관리한다는 것은 범위를 크게 확장한다.
판례는 법령의 “실행 결과”에 해당하며, 법 해석의 실제 기준을 제공한다.
판례가 사건번호 단위의 마크다운으로 정리되고, 선고일이 커밋 날짜가
되면, `git log` 한 줄로 특정 법률의 판례 시간순 추이를 볼 수 있다.
이는 법률 리서치에서 매우 강력한 도구다.

### 스페인 legalize 프로젝트와의 관계

Legalize KR이 스페인 legalize 프로젝트에서 영감을 받았다는 것은 이
접근이 국제적으로 검증되고 있음을 보여준다. 법 체계가 대륙법(Civil Law)
기반인 한국과 스페인은 성문법 전통을 공유하며, 법령을 텍스트 파일로
관리하는 것이 자연스럽다. 영미법(Common Law) 국가에서는 판례가 더 중요
하므로 같은 접근이 다소 다르게 적용될 수 있지만, 성문법 전통에서는
법령 Git이 거의 완벽한 표현 수단이다.

## 비평

### 강점

#### 메타포의 실행력

“법령 = 코드”라는 메타포를 단순히 말로만 남기지 않고, 실제로 Git 저장소
를 만들어 데이터를 채운 것은 결정적 차이다. 대부분의 “법과 기술의 만남”
프로젝트가 아이디어 단계에서 멈추는 반면, Legalize KR은 국가법령정보
센터 API를 파싱해 실제 마크다운 파일을 생성하고 Git 히스토리를 구축했다.
이 실행력이 프로젝트의 가장 큰 가치다.

#### 데이터의 공식성

모든 데이터가 국가법령정보센터 OpenAPI에서 수집되어 공식 원문을 기반
으로 한다는 점은 법률 데이터에서 결정적으로 중요하다. 비공식 소스의
법률 데이터는 법적 효력이 없고, 오류가 있으면 심각한 결과를 초래할 수
있다. YAML Frontmatter에 출처 URL을 포함시켜 원문 대조를 가능하게 한
것도 좋은 설계다.

#### 디렉토리 구조의 직관성

법률명을 디렉토리로, 법률/시행령/시행규칙을 파일로 배치한 구조는
법령 체계의 자연스러운 반영이다. 특히 law.go.kr의 URL 규칙과 동일한
디렉토리명을 사용함으로써 “이 디렉토리가 어떤 법령인지”를 URL 경험으로
즉시 알 수 있게 했다.

#### Git diff로 개정 비교

법령 개정을 Git diff로 비교하는 것은 이 프로젝트의 가장 설득력 있는
사용 사례다. 기존에는 개정 전후를 비교하려면 두 버전의 PDF를 나란히
열거나, 국회 의안 정보 시스템에서 비교표를 찾아야 했다. `git diff`는
이를 한 줄 명령어로 해결하며, 추가·삭제·변경이 색상으로 표시되어
시각적으로 즉시 파악 가능하다.

### 약점

#### 법률 비전문가를 위한 안내 부재

사이트와 README는 모두 개발자를 대상으로 한다. `git clone`, `grep`,
`diff` 같은 명령어가 당연시되는데, 실제로 법령을 가장 많이 다루는
사용자(변호사, 법무사, 공무원, 법학생)는 이런 도구에 익숙하지 않을 수
있다. “법률인을 위한 사용 가이드”가 없으면 가장 큰 잠재 사용자 집단에게
접근하지 못한다.

#### 법령 파싱 파이프라인의 불투명성

국가법령정보센터 API에서 데이터를 가져와 마크다운으로 변환하는 과정이
어떻게 이루어지는지 상세한 설명이 부족하다. 파싱 코드가 공개되어 있는지,
어떤 정보가 변환 과정에서 손실되는지(예: 별표, 첨부 서식, 도표), 파싱
오류가 얼마나 발생하는지 등이 다뤄지지 않는다.

#### Force-push 정책의 위험

“파이프라인 개선 시 force-push가 실행될 수 있다”는 공지는 솔직하지만,
이 저장소를 참조하는 프로젝트에는 큰 리스크다. force-push는 모든 커밋
해시를 변경시키므로, 이 저장소의 특정 커밋을 인용한 논문이나 법률
문서는 참조가 깨진다. 안정적 법률 데이터 소스로 사용하기에는 이
정책이 장벽이 될 수 있다.

#### 판례 저장소의 완결성 미확인

precedent-kr 저장소의 판례 범위(대법원 판례만인지, 하급심 판례도
포함하는지), 수집 기간, 총 판례 수 같은 기본 통계가 확인되지 않는다.
법률 리서치에서 “어떤 판례가 빠져 있는지”는 결정적 문제이며, 이에 대한
완결성 보장이 없으면 전문적 용도로 사용하기 어렵다.

#### 라이선스 명시 부재

법령 텍스트 자체는 저작권 대상이 아니지만(한국 저작권법 제7조, 법령은
저작권 보호 대상 제외), 마크다운 파싱 코드, 메타데이터 구조, 디렉토리
설계 같은 파생 작업은 저작권이 발생할 수 있다. 저장소에 명시적 라이선스
가 있는지 확인이 필요하다.

#### 검색 기능의 한계

`grep`으로 전체 법령을 검색할 수 있지만, 이는 순수 텍스트 매칭일 뿐
시맨틱 검색이 아니다. “개인정보”를 검색하면 해당 단어가 포함된 모든
조항이 나오지만, “프라이버시 침해”라는 유사 개념은 잡지 못한다. ir이나
GBrain 같은 시맨틱 검색 도구와 결합하면 이 한계를 넘을 수 있으며,
Legalize KR의 마크다운 파일은 이런 도구에 직접 투입할 수 있는 이상적
형태다.

## 인사이트

### 1. “법령을 Git으로 관리”는 법의 투명성을 기술적으로 보장하는 인프라다

법의 지배(Rule of Law)의 전제조건 중 하나는 “법이 공개되어야 한다”는
것이다. 대한민국 법령은 관보에 게재되어 공개되고, 국가법령정보센터에서
무료로 열람 가능하다. 형식적으로는 투명하다. 그러나 실질적 투명성은
다른 문제다. 특정 법률이 언제 어떻게 개정되었는지를 추적하려면 국가
법령정보센터에서 복잡한 검색을 해야 하고, 개정 전후 비교는 요약본에
의존한다. 이는 법의 공개는 되어 있되, “법의 변화 과정”은 공개되지 않은
상태에 가깝다.

Git 기반 법령 관리는 이 상태를 바꾼다. 모든 개정이 커밋으로 기록되고,
`git diff`로 정확히 무엇이 바뀌었는지 확인할 수 있으며, `git log`로
전체 이력을 시간순으로 볼 수 있다. 이는 형식적 투명성을 넘어 “추적
가능한 투명성(traceable transparency)”을 만든다. 추적 가능한 투명성은
단순히 “정보가 공개되어 있다”를 넘어 “정보의 변화 과정이 구조적으로
기록되고 접근 가능하다”는 의미다.

이 인사이트의 더 깊은 함의는, 법의 투명성이 기술적 인프라의 문제라는
것이다. 법령 텍스트가 PDF로만 공개되면, 개정 추적은 수동 작업이며
소수의 전문가만 수행할 수 있다. 법령이 Git 저장소로 공개되면, 아무나
`git diff`로 변화를 확인할 수 있다. 전자는 “전문가 게이트키핑” 모델이고,
후자는 “민주적 접근” 모델이다. Legalize KR은 후자의 모델을 한국에
도입하는 시도이며, 이는 단지 개발자 편의가 아니라 민주적 참여의 기술
인프라 확장이다.

### 2. “모든 개정은 Git 커밋”이 만드는 법률 데이터 분석의 새 가능성

법령이 Git 히스토리로 존재하면, 코드 분석에서 사용하는 모든 기법을 법령
분석에 적용할 수 있다. 이는 새로운 법률 데이터 분석 방법론을 열어준다.

예를 들어, `git shortlog --summary`로 “가장 많이 개정된 법률 Top 20”을
추출할 수 있다. 이는 “어떤 법률이 가장 불안정한가”를 정량적으로 보여
준다. 자주 개정되는 법률은 사회적 변화나 정치적 압력을 반영할 가능성이
높다. 이 분석을 시계열로 확장하면, “선거 전후에 특정 법률 개정이 집중
되는가” 같은 질문에 답할 수 있다.

또한 `git diff --stat`을 사용하면 각 개정의 규모(추가된 줄/삭제된 줄)를
정량화할 수 있다. “큰 개정”과 “작은 개정”의 분포, “특정 소관부처의 개정
패턴”, “시행령이 법률 개정을 따라가는 속도” 같은 분석이 가능해진다.

판례 저장소와 결합하면 더 깊은 분석이 가능하다. 특정 조항이 개정된
시점과, 그 조항을 인용한 판례의 시간 분포를 비교하면, “법 개정이 실제
법 해석에 얼마나 빠르게 반영되는가”를 측정할 수 있다. 이는 법학, 정치학,
행정학의 연구 주제이며, 지금까지 데이터 접근의 어려움으로 거의 수행되지
못한 분석이다.

이 모든 분석이 가능한 이유는 Legalize KR이 법령을 “데이터 과학이 다룰 수
있는 형태”로 변환했기 때문이다. CSV나 JSON이 아니라 Git 히스토리라는 것이
핵심이다. Git은 “변화의 시간순 기록”을 자연스럽게 표현하고, 이를 다루는
도구가 이미 성숙해 있다. 법률 데이터 분석의 진입 장벽이 “데이터 형태”
에서 “분석 질문”으로 이동한 것이다.

### 3. 법률 RAG의 이상적 데이터 소스가 이미 준비되어 있다

2025-2026년 사이 법률 AI(Legal AI) 시장은 폭발적으로 성장하고 있다.
Harvey, CaseText(Thomson Reuters 인수), 리걸테크 스타트업들이 법률
문서를 LLM에 투입하는 RAG 시스템을 경쟁적으로 구축하고 있다. 한국
에서도 로폼(Lawform), 인텔리콘, 리걸마인드 같은 리걸테크 회사들이
법률 RAG를 만들고 있다.

이런 시스템의 가장 큰 기술적 병목은 “법률 데이터를 청결하고 구조화된
형태로 확보하는 것”이다. 국가법령정보센터 API가 데이터를 제공하지만,
XML 파싱, 개정 이력 추적, 메타데이터 정리 같은 전처리에 상당한 엔지
니어링이 필요하다. 모든 리걸테크 회사가 이 전처리를 중복으로 수행하고
있다.

Legalize KR은 이 중복을 제거할 수 있는 공공재다. `git clone` 한 번으로
전체 법령을 마크다운 형태로 확보할 수 있고, YAML Frontmatter에 메타
데이터가 정리되어 있으며, 개정 이력이 Git 히스토리에 있다. 리걸테크
회사나 개인 개발자가 법률 RAG를 구축하려 할 때, 이 저장소가 첫 번째
데이터 소스가 될 수 있다.

물론 실제 활용에는 주의가 필요하다. force-push 가능성, 파싱 오류의
존재, 판례 데이터의 완결성 미보장 같은 문제가 있다. 그러나 이런 문제가
있더라도, “깨끗한 마크다운 형태의 전체 한국 법령”이라는 데이터셋의
가치는 거대하다. 앞서 다룬 ir이나 GBrain에 이 데이터를 투입하면, 로컬
에서 전체 한국 법령을 시맨틱 검색하는 시스템을 개인이 구축할 수 있다.
이는 Harvey 같은 수십억 달러 스타트업의 핵심 기능을 개인이 재현하는
것이며, 오픈 데이터의 힘을 극적으로 보여주는 사례가 될 것이다.

### 4. “법령 Git”은 국가 인프라가 되어야 한다

현재 Legalize KR은 커뮤니티 프로젝트다. 국가법령정보센터 API에서
데이터를 수집해 Git 저장소로 변환하는 개인/팀의 노력이다. 이 모델은
초기 검증에는 적합하지만, 장기적으로는 국가 인프라로 승격되어야 한다.

왜인가? 첫째, 파싱 파이프라인의 정확성이 법적 신뢰성을 결정한다.
커뮤니티 프로젝트는 파싱 오류가 있을 수 있고, 이를 발견하고 수정하는
메커니즘이 비공식적이다. 국가가 직접 법령 Git을 운영하면, 공식성과
정확성이 보장된다. 둘째, force-push 문제가 사라진다. 국가 인프라로
운영되면 히스토리는 영구적이고 참조 가능해야 한다. 셋째, 판례 데이터의
완결성이 보장된다.

이 제안은 실현 불가능한 것이 아니다. 미국의 경우 의회법(US Code)은
XML 형태로 공식 배포되며, 영국은 legislation.gov.uk에서 법령의 버전
관리를 공식적으로 제공한다. 한국도 국가법령정보센터가 이미 API를
제공하고 있으므로, 그 위에 공식 Git 저장소를 추가하는 것은 기술적
으로 어렵지 않다.

Legalize KR은 이 미래를 민간이 먼저 보여주는 프로젝트다. 국가가
움직이기 전에 커뮤니티가 답을 만들고, 그 답이 충분히 좋으면 국가가
채택한다. 리눅스가 정부 인프라가 되기까지 수십 년이 걸렸지만, 법령
Git은 그보다 훨씬 빠를 수 있다. 왜냐하면 법령 데이터는 이미 공개
되어 있고, 변환 파이프라인은 이미 존재하며, Git이라는 도구는 이미
보편적이기 때문이다. 부족한 것은 정치적 의지뿐이며, Legalize KR 같은
프로젝트가 성공적으로 운영되는 것이 그 의지를 만드는 증거가 된다.

### 5. 법률 데이터의 “포맷 해방”은 한국 HWP 해방과 같은 흐름이다

이 TIL 시리즈에서 앞서 다룬 kimkj.com의 “HWP 1,659개를 마크다운으로 변환”
글과 Legalize KR은 같은 흐름에 속한다. 한국의 디지털 지식 자산이 폐쇄적
포맷(HWP, 국가법령정보센터의 HTML/XML)에 갇혀 있고, 이를 마크다운으로
변환해 AI와 개발자 도구에 투입하려는 움직임이다. HWP에서 마크다운으로,
법령에서 Git으로 ― 두 사례 모두 “한국 특유의 포맷 장벽을 기술로 넘는다”
는 같은 패턴이다.

이 패턴이 일반화되면, 한국의 다른 공공 데이터(판결문 전문, 국회 회의록,
정부 보고서, 특허 명세서)도 같은 방식으로 “마크다운 + Git”으로 변환될
것이다. 각각이 독립적인 커뮤니티 프로젝트로 시작되겠지만, 최종적으로는
“한국 공공 지식의 Git 생태계”라는 더 큰 인프라로 수렴할 가능성이 있다.
Legalize KR은 이 생태계의 첫 번째 조각이며, 가장 성숙한 사례다.
