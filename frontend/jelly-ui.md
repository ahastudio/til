# Jelly UI — Soft Web Components

<https://jelly-ui.com/>

HN 토론: <https://news.ycombinator.com/item?id=48981620> (656점, 201개 댓글)

## 소개

Jelly UI는 네이티브 HTML 폼 컨트롤에 부드러운 물리 효과를 입히는 웹 컴포넌트 라이브러리다.
의존성이 없고, 단일 스크립트 태그 하나로 로드된다.
체크박스, 라디오 버튼, 슬라이더, 토글 같은 실제 폼 엘리먼트를 대체하지 않고,
기존 네이티브 컨트롤 위에 시각적 효과를 추가하는 방식으로 동작한다.

40개의 커스텀 엘리먼트를 제공하며, WCAG AA 색상 토큰을 적용해 접근성 기준을 충족한다.
다크 모드와 RTL(우에서 좌로 쓰는 언어) 레이아웃을 기본 지원한다.
MIT 라이선스를 표방하지만, GitHub 레포지터리는 공개되지 않았다.
소스 코드 접근 방법이 문서에 명시되지 않아 라이선스와 실제 접근성 사이에 간극이 있다.

사용법은 단순하다.

```html
<script type="module" src="https://jelly-ui.com/package.js"></script>
<jelly-theme mode="auto">
  <jelly-button variant="mint">Publish</jelly-button>
</jelly-theme>
```

`<jelly-theme>` 으로 감싸면 내부 컴포넌트에 테마가 적용된다.

## 분석

### 물리 효과와 접근성의 결합이 보기 드문 이유

대부분의 UI 애니메이션 라이브러리는 접근성을 나중에 고려하거나, 완전히 무시한다.
화려한 효과를 먼저 만든 뒤 키보드 탐색이나 화면 낭독기 지원을 덧붙이는 방식은
접근성 지원이 늘 불완전하게 끝나는 원인이다.

Jelly UI가 HN에서 주목받은 이유 중 하나는 물리 효과가 중심이면서도
네이티브 폼 의미론(semantics)을 유지했다는 점이다.
커스텀 엘리먼트가 실제 `<input>`, `<select>`, `<button>` 위에 올라가기 때문에
폼 제출, 레이블 연결, 탭 순서, 화면 낭독기 동작이 그대로 작동한다.
이 설계 선택이 접근성과 시각 효과가 충돌하지 않는 이유다.

### 물리 애니메이션이 폼 컨트롤에 어울리는가

물리 기반 애니메이션은 버튼이나 모달 같은 큰 UI 요소에서는 자연스럽지만,
체크박스나 라디오 버튼 같은 작은 폼 컨트롤에서는 과할 수 있다.
폼 컨트롤은 선택이나 입력의 도구이지, 주의를 끄는 요소가 아니기 때문이다.

그러나 이 프로젝트에서 물리 효과는 “탄성 있게 튀어오르는” 느낌에 가깝다.
버튼을 누를 때 살짝 눌리는 반응, 체크박스가 채워질 때 탱탱한 움직임이 있는 정도다.
사용자가 실제로 무언가를 만지는 듯한 촉각적 피드백을 화면에서 재현하는 시도이고,
그 범위를 절제했다는 점에서 프로덕션에서 쓸 수 있는 수준이다.

### 제로 의존성이 주는 실질적 이점

웹 컴포넌트 생태계에서 제로 의존성은 단순히 번들 크기가 작다는 의미가 아니다.
의존성이 없으면 React, Vue, Svelte 어느 프레임워크와도 충돌 없이 동작한다.
프레임워크 업그레이드나 의존성 트리 충돌 걱정 없이 장기적으로 유지할 수 있다.

## 비평

### MIT 라이선스를 표방하지만 소스 코드가 없다

HN에서 가장 많이 제기된 실용적 문제는 라이선스와 배포 방식의 불일치다.
홈페이지에는 MIT 라이선스라고 명시돼 있지만, GitHub 레포지터리가 없다.[^NiloCK]
소스 코드를 받아 직접 호스팅하거나, npm 패키지로 설치하는 방법이 문서화되지 않았다.

더 심각한 문제는 공식 사용법이 저자 도메인의 스크립트를 직접 로드하는 방식이라는 점이다.[^ChiperSoft]

```html
<script type="module" src="https://jelly-ui.com/package.js"></script>
```

서드파티 도메인 스크립트를 프로덕션 앱에 직접 로드하는 것은 공급망 보안의 기본 원칙에 어긋난다.
저자가 스크립트를 언제든 변경할 수 있고, 도메인 만료나 서버 장애 시 앱이 중단된다.
MIT 라이선스의 의미는 "자유롭게 쓸 수 있다"인데, 소스 코드 없이 자기 서버에만 올라간 스크립트를 로드하는 것은 그 의미와 맞지 않는다.

### `prefers-reduced-motion` 대응 — 작동하지만 문서화가 부족하다

물리 애니메이션 라이브러리에서 가장 먼저 확인해야 하는 것은
`prefers-reduced-motion` 미디어 쿼리 대응이다.
전정 장애(vestibular disorder)가 있는 사용자는 화면의 움직임이 메스꺼움을 유발할 수 있다.

실제로 이 설정을 따른다는 점은 HN 댓글에서 확인됐다.[^TurkTurkleton]
그러나 문서에 이 동작이 어떻게 구현됐는지 명시되지 않았다.
WCAG AA 색상 토큰을 제공할 만큼 접근성에 투자한 프로젝트라면, 모션 감소 대응도 문서화해야 설계 일관성이 완성된다.

### 성능과 UX 일관성 문제가 데모 수준에서도 드러난다

물리 연산은 GPU 가속이 없는 저사양 모바일 기기에서 성능 문제를 일으킬 수 있다.
특히 40개의 커스텀 엘리먼트를 포함한 폼이 저사양 기기에서 동작할 때
애니메이션 드롭 프레임이 오히려 입력 지연처럼 느껴질 수 있다.

데모 사이트에서도 슬라이더가 마우스 커서를 따라가지 못하고 지연되는 문제가 보고됐다.[^JoshTriplett]
물리 효과가 “입력에 즉각 반응한다는 느낌”을 주려면 커서 추종 지연이 없어야 하는데,
정작 슬라이더에서 그 감각이 깨진다는 점은 역설적이다.

UX 일관성 문제도 있다. 버튼은 클릭 후 드래그해 바깥에서 릴리스해도 클릭으로 인식되지만,
체크박스는 같은 방식에서 클릭으로 인식하지 않는다.[^itishappy]
컨트롤마다 다른 인터랙션 모델은 사용자가 물리 효과를 예측하기 어렵게 만든다.

지연 문제의 근본 원인으로 jlukic은 JellyEngine 구현 방식을 지목했다.[^jlukic]
포인터 이벤트 기반으로만 계산하는 대신, 8ms마다 RAF(RequestAnimationFrame) 루프를 실행하면서
페이지의 모든 활성 컴포넌트를 매 프레임 재계산한다.
이 방식은 전체 문서를 계속 리페인트시키고, 코드 주석에서 AI 생성 흔적이 보인다는 점도 지적됐다.
“인간이라면 '이렇게 하면 안 된다'고 바로 잡아냈을 것”이라는 평가다.

### 스타일 커스터마이징 범위가 불명확하다

WCAG AA 색상 토큰을 제공한다는 것은 기본 팔레트가 있다는 의미인데,
그 토큰을 브랜드 색상으로 교체하는 방법이 공개 문서에서 충분히 설명되지 않았다.
디자인 시스템을 가진 팀이 이 라이브러리를 도입하려면 커스터마이징 깊이가
가장 먼저 확인해야 할 사항이다.

## 인사이트

### 네이티브 폼 의미론을 유지하는 것이 웹 컴포넌트의 올바른 방향이다

지난 10년 동안 웹 UI 컴포넌트의 큰 문제는 시각적으로는 폼 컨트롤처럼 보이지만
키보드 탐색, 레이블 연결, 화면 낭독기 동작이 작동하지 않는 커스텀 엘리먼트의 범람이었다.
`<div>` 기반의 가짜 체크박스, 스타일을 입힌 `<span>` 버튼이 대표적이다.

Jelly UI는 이 문제를 다른 방향에서 접근한다.
시각 효과는 네이티브 컨트롤 위에 올리고, 폼 동작은 브라우저에 위임한다.
이 설계가 맞는 방향이라는 증거는 간단하다.
폼 제출, 레이블 클릭, 탭 탐색, 화면 낭독기가 추가 설정 없이 작동한다.

앞으로 웹 컴포넌트 표준이 성숙해지면서,
네이티브 엘리먼트를 대체하는 방향보다 확장하는 방향의 컴포넌트가 더 안정적인 선택이 될 것이다.
Jelly UI는 그 방향의 구체적인 사례다.

rickydroll은 이 라이브러리가 손 운동 능력이 제한된 사용자에게도 잠재적으로 유용할 수 있다고 제안했다.[^rickydroll]
물리 효과로 클릭 영역이 동적으로 반응하거나 타겟 크기가 확장된다면,
작은 폼 컨트롤을 정밀하게 클릭하기 어려운 사용자의 조작 부담을 줄일 수 있다는 관점이다.
접근성이 단순히 색상 대비나 화면 낭독기 지원에 그치지 않고,
운동 능력 차이까지 포괄하는 방향으로 확장될 수 있음을 보여주는 시각이다.

### UI 라이브러리의 차별화가 미감에서 접근성과 성능으로 이동한다

과거의 UI 컴포넌트 라이브러리 경쟁은 주로 미감에서 이뤄졌다.
더 예쁜 버튼, 더 세련된 드롭다운, 더 매끄러운 애니메이션이 선택 기준이었다.

이 경쟁은 이미 충분히 포화 상태다.
Bootstrap, Material UI, Ant Design이 외관 기준에서 거의 동등한 수준에 도달했고,
Tailwind CSS의 등장으로 커스텀 디자인의 진입 장벽도 낮아졌다.

다음 경쟁 축은 접근성과 성능이다.
키보드 탐색이 완벽히 작동하는가, `prefers-reduced-motion`을 지원하는가,
번들 크기가 작은가, 서버 사이드 렌더링과 호환되는가.
Jelly UI가 이 경쟁의 초기 사례가 될 수 있는 이유는
물리 효과보다 접근성과 네이티브 호환성에 먼저 투자했기 때문이다.

### 물리 기반 UI가 모바일 앱 경험의 기준을 웹으로 가져온다

iOS와 Android 앱에서 물리 기반 애니메이션은 이미 오래된 표준이다.
스프링 애니메이션, 탄성 스크롤, 촉각 피드백이 모바일 앱의 자연스러운 느낌을 만든다.
웹 UI는 이 기준보다 항상 조금 딱딱하고 기계적인 느낌을 줬다.

Jelly UI는 모바일 앱 수준의 물리적 반응성을 웹 폼 컨트롤에 가져오려는 시도다.
이 방향이 웹 표준으로 자리 잡으려면 아직 갈 길이 멀지만,
프레임워크 독립적인 웹 컴포넌트로 구현됐다는 점에서 확산 가능성이 있다.
React나 Vue 생태계에 종속되지 않기 때문에,
프레임워크 전환 없이 어떤 프로젝트에도 적용할 수 있다.
