# Foldkit

<https://foldkit.dev/>

<https://github.com/foldkit/foldkit>

## 소개

Foldkit은 TypeScript 기반 프론트엔드 프레임워크다.
“정확성을 위한 프론트엔드 프레임워크(The frontend framework for correctness)”를 표방한다.
Effect-TS 생태계 위에 구축되었으며, Elm Architecture 원칙을 따른다.
v0.118.0 기준 GitHub 492스타, 예제 앱 27개를 제공한다.

핵심 철학은 세 가지로 요약된다.
하나의 불변 모델이 전체 앱 상태를 보유한다.
모든 상태 변경은 단일 update 함수를 통해 이루어진다.
사이드 이펙트는 Commands라는 값으로 표현해 명시적으로 처리한다.
이 구조는 Elm Architecture를 그대로 계승하며, 이를 TypeScript와 Effect-TS 위에 올린 것이다.

설치는 다음 명령으로 시작한다:

```bash
npx create-foldkit-app@latest
```

## 아키텍처

### 상태와 업데이트

Elm Architecture의 중심 개념을 그대로 따른다.
애플리케이션의 전체 상태는 하나의 불변 모델 객체에 담긴다.
상태를 변경하는 유일한 경로는 단일 update 함수다.
이 함수는 현재 모델과 메시지를 받아 새 모델과 Commands 목록을 반환한다.
Commands는 HTTP 요청, 로컬스토리지 접근 같은 사이드 이펙트를 순수한 값으로 표현한다.

Schema 기반 상태 관리로 모델의 직렬화와 유효성 검증이 내장된다.
타입 안전 양방향 라우팅을 제공해 URL 구조가 타입 수준에서 보장된다.
중첩 상태는 Submodels로 구성하고, 범위가 한정된 스트림은 Subscriptions로 처리한다.
브라우저 API 접근은 Managed Resources 추상화를 통해 안전하게 관리한다.

### 테스팅 철학

Story와 Scene이라는 두 가지 테스트 프리미티브를 제공한다.
Story는 상태 기계로서 앱 동작을 테스트한다.
메시지 시퀀스를 적용했을 때 모델이 예상하는 상태로 전이하는지를 검증한다.
DOM이나 렌더러 없이 순수 함수 테스트로 실행된다.

Scene은 UI 인터랙션 테스트다.
DOM을 직접 조작하거나 모킹하는 대신 접근성 로케이터(accessible locators)로 요소를 찾는다.
`role`, `label`, `aria-*` 속성을 기반으로 쿼리하기 때문에
테스트를 통과한 코드는 동시에 접근성 기준도 충족하게 된다.

### 포함된 시스템

라우팅, 접근성이 보장된 UI 컴포넌트(dialog, menu, tabs, listbox),
동기/비동기 필드 유효성 검증, DevTools의 시간여행 디버깅,
그리고 Foldkit 자체로 빌드한 Foldkit이 포함된다.
“자기 자신으로 구현한 도구”라는 선택은 프레임워크의 실용성에 대한 자기 검증이기도 하다.

## 분석

### Elm Architecture가 TypeScript 생태계에서 반복 소환되는 이유

Elm Architecture는 2015년 전후로 React/Redux 패턴에 직접적인 영향을 주었다.
Redux의 reducer와 action 구조는 Elm의 update 함수와 메시지를 거의 그대로 이식한 것이다.
그러나 Redux는 Elm의 핵심인 “사이드 이펙트를 값으로 표현하는 Commands” 개념을
Middleware라는 훨씬 복잡한 추상으로 대체했고,
이 결정이 이후 Redux-Thunk, Redux-Saga, Redux-Observable의 분열로 이어졌다.

Foldkit이 Elm Architecture를 명시적으로 계승하는 선택은
이 분열에 대한 역사적 수정이다.
“사이드 이펙트는 Commands라는 순수한 값이어야 한다”는 명제로 돌아가
거기에 Effect-TS의 타입 추적 능력을 더한다.
이것은 React 생태계가 점진적으로 복잡해진 경로를 거슬러
설계 출발점으로 되돌아가는 시도다.

### Effect-TS 위에 구축하는 전략적 포지셔닝

Effect-TS는 함수형 효과 시스템으로, 사이드 이펙트를 타입 수준에서 추적한다.
오류 타입, 의존성, 비동기성을 모두 타입으로 표현하는 이 라이브러리는
주로 서버 사이드 Node.js 애플리케이션에서 채택되어 왔다.
Foldkit은 이 생태계의 사용자가 프론트엔드에서도 같은 도구로 개발할 수 있게 한다.

“Effect 개발자에게 필요한 프론트엔드 솔루션”이라는 포지셔닝은 의도적이다.
React나 Vue를 쓰는 사람을 설득하는 것이 아니라
이미 Effect-TS를 쓰는 팀에게 프론트엔드 해법을 제공하는 전략이다.
이 선택은 채택 장벽을 높이지만 동시에 제안의 일관성을 높인다.
Effect-TS를 쓰는 팀은 이미 이 프로그래밍 모델에 동의한 상태이므로
Foldkit을 추가로 배우는 비용이 낮다.

### 테스트와 접근성을 단일 구조로 묶는 설계

Scene 테스트가 접근성 로케이터를 사용한다는 선택은
접근성과 테스트 안정성이 같은 방향을 가리킨다는 통찰에서 나온다.
DOM 구조에 의존하는 테스트는 UI를 리팩터링할 때 자주 깨진다.
역할과 레이블에 의존하는 테스트는 의미 구조를 검증하므로
리팩터링에 훨씬 강하고, 동시에 스크린 리더 사용자의 경험을 검증한다.

이 설계는 접근성을 부가 요건이 아니라 아키텍처 불변식(architectural invariant)으로 다룬다.
테스트를 작성하려면 접근성 속성이 있어야 하고,
접근성 속성이 없으면 테스트가 불가능한 구조다.
접근성을 검사 항목이 아닌 설계 제약으로 만드는 방식이다.

## 비평

### Effect-TS 의존성이 채택 비용을 실질적으로 두 배로 만든다

Foldkit의 전제는 “Effect-TS를 이미 알고 있다”는 것이다.
그러나 Effect-TS 자체가 상당한 학습 투자를 요구하는 라이브러리다.
함수형 효과 시스템의 개념적 모델, 파이프라인 API, 타입 추론 패턴을
충분히 익히는 데 몇 주가 걸린다.
Foldkit을 도입하는 팀이 Effect-TS에 익숙하지 않다면
프레임워크와 기반 라이브러리를 동시에 배워야 한다.

홈페이지는 “Effect 개발자를 위한 솔루션”이라고 명시하지만,
기존 Effect-TS 사용자 풀이 얼마나 되는지 묻지 않을 수 없다.
서버 사이드 Node.js에서 Effect-TS를 쓰는 팀이 프론트엔드도 같은 스택으로
가져가려는 맥락에서만 이 프레임워크는 분명한 가치를 제공한다.
이 맥락을 벗어나면 진입 비용 대비 이익이 불분명해진다.

### “정확성”이라는 약속이 어떤 버그를 잡는지 구체화되지 않는다

“The frontend framework for correctness”라는 태그라인은 강한 주장이다.
하지만 어떤 종류의 버그를 예방하는지, 어떤 종류의 버그는 여전히 발생하는지가
명확하게 서술되지 않는다.
타입 시스템이 표현력이 높다는 것은 특정 범주의 오류를 컴파일 타임에 잡는다는 의미다.
그러나 모든 프론트엔드 버그가 그 범주 안에 있지는 않다.
경쟁 조건, 비결정적 네트워크 응답, 서드파티 스크립트 간섭,
접근성 인식 오류 같은 문제들은 타입 시스템으로 잡기 어렵다.

홈페이지의 “이 프레임워크에 적합하지 않은 경우” 목록은 정직한 서술이지만,
“정확성”이 구체적으로 무엇을 의미하는지를 밝히는 것은 다르다.
이 약속이 실제로 어느 범위까지 성립하는지
실제 프로덕션 사례나 비교 데이터 없이는 평가하기 어렵다.

### SSR 부재가 현대 웹 요건과 구조적으로 충돌한다

Foldkit은 서버 사이드 렌더링(SSR)을 지원하지 않는다.
SEO, 초기 로드 성능, 메타태그 동적 생성은 현재 대부분의 프로덕션 웹 애플리케이션에서
기본 요건에 가깝다.
Next.js나 Nuxt가 기본 선택지가 된 이유 중 하나가 바로 SSR 지원이다.

Elm Architecture의 순수 클라이언트 사이드 특성이 SSR과 충돌하는 것은
구조적 문제이지 단순한 누락이 아니다.
불변 모델과 Commands 패턴이 서버와 클라이언트에서 다르게 작동하는 문제를
해결하려면 상당한 아키텍처 확장이 필요하다.
현재 상태에서 Foldkit은 내부 툴, 대시보드, SPA 맥락에만 실용적으로 적용 가능하다.

## 인사이트

### Elm Architecture의 르네상스는 React 피로의 역방향 운동이다

React 생태계는 지난 10년 동안 상태 관리 라이브러리를 주기적으로 교체해 왔다.
Flux → Redux → MobX → Recoil → Jotai → Zustand.
각각의 교체는 이전 솔루션의 복잡성에 대한 반응이었다.
그러나 이 교체의 방향은 “단순화”를 향하지 않고
“다른 종류의 복잡성”을 향했다.
전역 스토어의 복잡성을 피하면 원자 상태의 파편화가 생기고,
파편화를 피하면 다시 전역 스토어로 돌아오는 사이클이다.

Elm Architecture의 재소환은 이 사이클 바깥에 서려는 시도다.
“상태는 하나, 업데이트는 하나”라는 원칙을 타협하지 않으면
사이클이 아예 발생하지 않는다는 주장이다.
이것이 “정확성”이라는 태그라인이 기술적 명세가 아니라
아키텍처 철학 선언인 이유다.
Foldkit은 기능 집합이 아니라 프로그래밍 모델의 선택을 팔고 있다.

### 타입 시스템의 표현력과 의식(ceremony)은 같이 증가한다

Effect-TS와 Elm Architecture를 결합하면 표현 가능한 불변식의 범위가 넓어진다.
사이드 이펙트의 종류가 타입에 인코딩되고,
상태 전이가 검증 가능한 함수로 표현되며,
테스트가 순수 계산 검증으로 축소된다.
이것이 “정확성”이 가리키는 실질적 내용이다.

그러나 표현력이 높아질수록 코드에서 의미를 표현하기 위한 구조가 늘어난다.
타입 선언, Schema 정의, Commands 타입, 메시지 타입 등
실제 비즈니스 로직 외에 작성해야 하는 코드가 많아진다.
이 비용은 프로젝트 초기보다 중기에 회수된다.
타입 시스템이 잡아주는 버그는 초기 개발보다 리팩터링과 기능 확장 단계에서 가치를 발휘한다.
Foldkit의 적합한 사용 맥락은 단기 프로젝트보다
장기 유지보수가 예정된 복잡한 내부 시스템이다.

### 접근성을 아키텍처 불변식으로 만드는 패턴이 프레임워크 수준의 혁신이다

Scene 테스트가 접근성 로케이터를 강제하는 설계는
접근성과 테스트 작성 사이의 인센티브 구조를 바꾼다.
대부분의 프레임워크에서 접근성은 체크리스트 항목이다.
완성된 기능에 `aria-label`을 붙이는 것이 “접근성 작업”이 된다.
이 방식에서 접근성은 항상 후행 작업이다.

Foldkit의 접근 방식에서 접근성 속성이 없으면 테스트를 작성할 수 없다.
테스트 우선 개발(TDD)이 설계를 이끌듯,
접근성 로케이터가 컴포넌트 구조를 이끈다.
이것은 사회적 압력이나 감사 도구가 아니라
개발 워크플로 자체가 접근성을 생성하는 구조다.
프레임워크가 특정 실천을 강제하는 방식 중 가장 효과적인 것은
그 실천 없이는 다른 원하는 것을 할 수 없게 만드는 것이다.
Scene 테스트는 그 원칙의 실현이다.

### 492스타 프레임워크가 보여주는 생태계 분화의 현실

Foldkit은 명확히 틈새 프레임워크다.
React(수백만), Vue(수십만)와 비교하면 492스타는 작은 숫자다.
그러나 이 숫자는 실패의 지표가 아니다.
Effect-TS 사용자 풀 자체가 작고, 그 중 프론트엔드를 Effect-TS로 구성하려는 팀은 더 작다.
이 맥락에서 Foldkit의 존재는 생태계 분화의 자연스러운 결과다.

TypeScript 프론트엔드 생태계는 이제 하나의 주류 프레임워크로 수렴하지 않는다.
Svelte, SolidJS, Qwik, Astro 등이 각각 다른 맥락에서 다른 선택을 제공한다.
Foldkit은 이 중에서 Effect-TS 생태계와 Elm Architecture 원칙의 교차점을 채운다.
이 포지션이 작다는 것은 사실이지만,
그 포지션이 실재한다는 것도 사실이다.
프레임워크가 모든 사람을 위한 것이어야 한다는 전제를 버릴 때
Foldkit 같은 고도로 의견이 강한 도구가 존재할 공간이 생긴다.
