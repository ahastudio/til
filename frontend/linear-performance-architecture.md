# Linear이 왜 빠른가: 기술적 분석

원문: <https://performance.dev/how-is-linear-so-fast-a-technical-breakdown>

HN 토론: <https://news.ycombinator.com/item?id=48437609> (481점, 227개 댓글)

## 요약

Linear의 속도는 어느 한 기법의 결과가 아니라 여러 아키텍처 결정이 서로를
강화하는 시스템의 속성이다. 이 글은 그 결정들을 분해해 각각의 역할과 상호
의존 관계를 설명한다.

가장 핵심적인 결정은 브라우저를 데이터베이스로 삼는 것이다. UI가 읽는
데이터베이스는 서버가 아니라 브라우저의 IndexedDB에 있다. 변경(Mutation)은
먼저 로컬에 적용되고, 이후 서버로 비동기 전송된다. 서버는 변경 사항을
WebSocket을 통해 델타 형태로 다른 클라이언트에 브로드캐스트한다. Linear
공동 창업자 Tuomas는 이를 “스타트업이 보통 하는 것과 달리, 싱크 엔진이
내가 처음 작성한 코드였다”고 설명했다.

번들 크기와 로딩 최적화도 체계적으로 구축됐다. Linear는 Parcel → Rollup →
Vite → Rolldown의 순서로 번들러를 교체해왔고, 각 마이그레이션마다 전송하는
JavaScript 양을 줄이는 것을 목표로 삼았다. 결과적으로 코드 전송량이 50%
감소했고, 압축 후 크기가 30% 줄었으며, 콜드 캐시 페이지 로드가 10~30%
빨라졌다. Safari에서 Issues 뷰의 최초 화면 렌더링 시간이 59% 감소했고,
메모리 사용량은 70~80% 줄었다.

로딩 전략은 여러 기법이 직렬이 아닌 병렬로 작동하도록 설계됐다. `index.html`
에 인라인된 앱 셸이 외부 요청 없이 로딩 상태를 렌더링하고, localStorage에서
테마와 사이드바 설정을 복원한다. 진입 스크립트는 `<link rel=modulepreload>`
태그로 주요 청크를 병렬 프리패치하고, 서비스 워커는 약 1,200개의 해시된
에셋을 백그라운드에서 프리캐싱한다.

인증도 “해피 패스를 가정하고 백그라운드에서 검증”하는 방식이다.
`localStorage.ApplicationStore`가 존재하면 IndexedDB에 워크스페이스 데이터가
있다고 가정해 즉시 렌더링을 시작한다. 실제 세션 토큰은 HttpOnly 쿠키에
있으며, 이후 요청이 실패할 때만 로그인으로 이동한다. 애니메이션은 GPU
합성 속성(`transform`, `opacity`)과 `background-color`, `border-color`만
사용하며, 레이아웃을 유발하는 `width`나 `margin`은 절대 애니메이션하지 않는다.
지속 시간은 기본적으로 100~250ms로, 사용자가 지연을 인지하는 300ms 임계점
아래를 유지한다.

## 분석

### 로컬 퍼스트 아키텍처가 전통적인 CRUD 모델을 역전시키는 방식

전통적인 웹 앱에서 데이터 흐름은 “서버가 진실의 근원”이라는 가정 위에 서있다.
사용자가 무언가를 변경하면 네트워크 요청을 기다리고, 서버 응답을 받아 UI를
갱신한다. 이 구조에서 네트워크 지연은 불가피한 비용이다.

Linear는 이 가정을 뒤집었다. UI가 읽는 데이터베이스는 브라우저의 IndexedDB에
있고, 서버는 이 로컬 데이터베이스의 동기화 대상이다. 변경은 로컬에서 즉시
적용되고, 그 다음에 서버로 전달된다. 전통적 CRUD 앱에서 이슈 업데이트에
~300ms가 걸리는 반면, Linear는 “몇 밀리초”다. 이 차이는 최적화가 아니라
아키텍처의 차이다.

```javascript
// 전통적 방식
async function updateIssue({ issue }) {
  showSpinner();
  const response = await fetch(`/api/issues/${issue.id}`, {
    method: "PATCH",
    body: JSON.stringify({ title: issue.title }),
  });
  const updated = await response.json();
  setIssue(updated);
  hideSpinner();
}

// Linear 방식
issue.title = "Faster app launch";
issue.save();
```

이 패턴은 오프라인 퍼스트 앱에서 오래 검증된 방식이다. 차이는 Linear가 이것을
“보조 기능”이 아닌 “1일차 아키텍처”로 선택했다는 점이다. Tuomas가 싱크 엔진을
“처음 작성한 코드”라고 표현한 것은 이 결정이 후속 모든 결정의 기반이 됐음을
의미한다.

### MobX의 세분화된 반응성이 React의 한계를 어떻게 보완하는가

React의 기본 반응성 모델은 상태 변경이 감지되면 컴포넌트 트리를 재렌더링한다.
최적화 없이는 이슈 목록 하나의 필드가 바뀌어도 전체 목록이 다시 그려진다.

Linear는 MobX를 선택해 이 문제를 속성 수준에서 해결했다. 모델의 모든 속성이
독립적으로 관찰 가능하므로, 이슈 하나의 제목이 바뀌면 그 제목을 읽는 컴포넌트만
재렌더링된다. “50개 이슈 업데이트는 50번의 셀 재렌더링이지, 목록 재렌더링이
아니다.” 이 차이는 대규모 이슈 목록을 다룰 때 결정적이다.

MobX 오브젝트 풀은 커맨드 팔레트(⌘K)가 서버 요청 없이 즉시 검색 결과를 반환하는
것도 가능하게 한다. 메모리에 있는 MobX 풀을 직접 검색하므로, 로컬 퍼스트
아키텍처가 단순히 “데이터 저장” 문제를 넘어 “인터랙션 전체”에 영향을 준다.

### 번들러 교체 전략의 설계 논리

Parcel → Rollup → Vite → Rolldown의 마이그레이션 경로는 번들러를 “바꿀 수 있는
부품”으로 취급한다는 것을 보여준다. 각 번들러로의 이동은 현재 도구가 한계에
도달했을 때 일어났다.

핵심 전략은 개별 npm 패키지를 독립 캐시 청크로 분리하는 것이다. 의존성 하나를
업그레이드해도 다른 패키지 청크의 캐시가 무효화되지 않는다. 21MB의 JavaScript를
수백 개의 경로별 청크로 나눠 현재 경로에 필요한 코드만 전송한다. 레거시 브라우저
지원을 중단해 폴리필과 ES5 트랜스파일을 제거한 것도 이 전략의 일부다.

`<link rel=modulepreload>` 태그로 주요 청크를 병렬 프리패치하는 것은 이 분할의
비용(청크가 많아질수록 요청 수 증가)을 상쇄한다. 순차 워터폴 대신 하나의 배치
작업으로 처리한다.

### 스택의 단순성이 갖는 전략적 의미

Linear의 기술 스택은 React, TypeScript, MobX, PostgreSQL, CDN이다. 엣지 데이터베이스,
React Server Components, 복잡한 프레임워크가 없다. 2024~2026년 웹 개발 트렌드의
맥락에서 이는 의도적인 역선택이다.

이 단순성의 전략적 의미는 “정신 모델 비용의 최소화”다. 팀이 새 추상화를 익히는데
쓸 인지 비용을 제거하고, 대신 핵심 아키텍처 패턴(로컬 퍼스트, 세분화 반응성)을
깊이 파고들었다. 트렌디한 추상화보다 검증된 도구 위에서 독자적 설계를 구축하는
전략이다.

## 비평

### 이 아키텍처는 “실시간 협업 B2B SaaS”라는 특수한 맥락에서만 최적이다

로컬 퍼스트 + 싱크 엔진 아키텍처는 이슈 트래커처럼 “오프라인에서도 작동해야 하고,
다수가 같은 데이터를 편집하며, 변경이 잦은” 앱에 강하게 최적화된 설계다.
이 조건이 모두 성립할 때 이 아키텍처는 강력하다.

그러나 이 설계의 비용은 글에서 거의 다뤄지지 않는다. IndexedDB를 진실의 근원으로
삼으면 충돌 해소(Conflict Resolution) 로직이 필수다. 두 사용자가 같은 이슈를
동시에 편집하면 어느 변경이 우선하는가? CRDT(Conflict-free Replicated Data Type)나
OT(Operational Transformation) 같은 접근이 필요한데, 이 구현 비용은 상당하다.
Tuomas가 싱크 엔진을 “처음 작성한 코드”로 언급한 것은 이 비용의 규모를 암시한다.

Escapado는 이 복잡성을 구체적 시나리오로 펼쳤다.[^Escapado] 두 사용자가 오프라인
상태에서 같은 이슈를 편집하고 재접속하면 어느 변경이 우선하는가? 이슈가 삭제된 후
다른 클라이언트에서 편집되면 어떻게 처리하는가? 목록의 정렬 순서가 클라이언트마다
다르게 바뀌었을 때 최종 순서는 무엇인가? 스키마 변경이 있는 동안 오프라인이었던
클라이언트가 재접속하면 어떻게 되는가? 이 중 어느 하나도 자명한 답이 없다. Zero
(rocicorp)가 이 문제들을 다 해결하는 것을 포기하고 오프라인 쓰기 자체를 지원하지
않기로 결정한 것은 이 복잡성의 크기를 보여주는 사례다.

대부분의 웹 앱 — 전자상거래, 금융 대시보드, 콘텐츠 관리 시스템 — 은 Linear와
사용 패턴이 다르다. 오프라인 작업이 불필요하고, 데이터 정합성 요구가 강하며,
서버가 진실의 근원이어야 한다. throwaway7783는 이 관점을 직접적으로 표현했다 —
싱크 지연으로 인한 불확실성이 프로젝트에 실제 문제를 일으켜왔고, 차라리 서버를
빠르게 만들고 사용자가 네트워크 지연을 감수하게 하는 것이 낫다는 입장이다.[^throwaway7783]
글의 논조는 “Linear의 방식이 빠른 웹 앱의 일반 원칙”처럼 읽히지만, 사실 이것은
특수한 제품 맥락이 만들어낸 특수한 해법이다.

### 글은 “뭘 했는가”에 집중하고 “왜 이게 어려운가”를 숨긴다

최적화 사례 연구의 공통적 한계는 성공 후 서사(Post-hoc narrative)라는 점이다.
“우리는 이걸 했고, 결과적으로 빨라졌다”는 구조는 결정의 어려움과 실패한 시도를
자연스럽게 생략한다.

실제 Linear를 매일 사용하는 ricardobeat는 다른 경험을 보고했다.[^ricardobeat] 검색이
느리고, UI가 어수선하며(보기는 좋지만), Pulse 기능은 소규모에서도 노이즈가 넘치고,
필요한 것을 찾기 어려워 즐겨찾기를 남발하게 된다는 것이다. 아키텍처의 우아함이 곧
사용자 경험의 우아함을 보장하지 않는다는 점을 이 피드백은 드러낸다.

싱크 엔진은 분산 시스템의 가장 어려운 문제들 — 네트워크 분단 처리, 충돌 해소,
클라이언트 상태 복구 — 을 포함한다. 서비스 워커의 1,200개 에셋 캐싱 전략은
캐시 무효화 버그와 오프라인 상태 엣지 케이스를 처리해야 한다. MobX의 세분화
반응성은 순환 의존성과 불필요한 관찰자 누적 같은 새로운 문제를 만들 수 있다.
글은 이 복잡성의 비용을 거의 다루지 않는다.

singjnd는 Linear 싱크 엔진을 리버스 엔지니어링해 정리한 GitHub 저장소를 소개했다.[^simjnd]
이 문서는 글이 제시하는 “델타 동기화” 개요보다 훨씬 깊은 수준에서 실제 동기화 프로토콜이
어떻게 작동하는지를 보여준다. 싱크 엔진이 얼마나 비자명한 설계인지는, 외부 개발자가
그것을 이해하기 위해 리버스 엔지니어링까지 해야 했다는 사실 자체가 증명한다.

특히 “해피 패스 가정” 인증 방식은 보안과 UX의 트레이드오프를 명확히 하지 않는다.
HttpOnly 쿠키가 유효한지 확인하기 전에 데이터를 렌더링하면, 만료된 세션 사용자가
잠깐 데이터를 볼 수 있는 상황이 생긴다. 이 동작이 의도적 설계인지, 수용 가능한
트레이드오프인지, 특정 보안 모델을 전제로 하는지 글에서 다루지 않는다.

### MobX 선택은 React 생태계의 방향과 역행하며 그 비용이 언급되지 않는다

2024년 기준 React 생태계의 주류 방향은 서버 컴포넌트, Suspense, use API 등
“리액트 자체의 동시성 모델”을 활용하는 방향이다. MobX는 React의 렌더링 스케줄러
밖에서 상태를 관리하므로 이 모델과 통합이 복잡하다.

이 선택은 두 가지 비용을 수반한다. 첫째, 신입 개발자가 React 생태계 표준 지식 외에
MobX의 관찰자 패턴을 별도로 익혀야 한다. 둘째, React의 공식 동시성 기능이 발전할수록
MobX 기반 코드베이스의 마이그레이션 부담이 커질 수 있다. Linear가 MobX를 계속
유지하는 것은 “빠른 웹 앱”이라는 목표를 위해 “React 생태계 주류 흐름”을 의도적으로
포기한 선택이다. 그리고 그 선택의 장기 비용은 글에서 논의되지 않는다.

## 인사이트

### 웹 앱 성능의 병목은 네트워크가 아니라 아키텍처 가정이다

전통적인 웹 성능 최적화는 “네트워크 요청을 빠르게 만들기”에 집중한다. CDN,
캐싱 헤더, HTTP/2 멀티플렉싱, 엣지 컴퓨팅 — 모두 “요청은 있지만 더 빠르게”라는
틀 안에 있다.

Linear의 접근은 이 틀을 벗어났다. “네트워크 요청을 사용자에게 숨기는 것이 훌륭한
웹 앱의 비밀”이라는 결론은, 최적화의 목표가 “요청 속도 향상”이 아니라 “요청
존재의 제거”임을 선언한다. 로컬 퍼스트 아키텍처는 네트워크를 최적화하는 것이
아니라 UI 반응성에서 네트워크를 제거한다.

이 인식의 전환은 더 넓은 함의를 갖는다. 데스크톱 네이티브 앱이 “빠른 이유”는
더 좋은 코드가 아니라 데이터가 로컬에 있기 때문이다. Linear는 웹 앱이 이 이점을
취할 수 있음을 보여준다. 그 대가는 싱크 엔진이라는 복잡성이다. 즉 웹 앱의 성능
병목은 네트워크 인프라가 아니라, “서버가 진실의 근원”이라는 아키텍처 가정 자체다.

그러나 이 주장에는 중요한 반론이 있다. jacobgold는 엣지 배포와 최적화된 서버로
전통적 CRUD 앱도 30ms 내외의 응답을 달성할 수 있다고 지적했다.[^jacobgold] 글이
비교 기준으로 삼는 “300ms CRUD”는 최적화되지 않은 서버를 전제로 한다. 즉 로컬
퍼스트가 성능 문제의 유일한 해법이 아니라, 서버를 사용자 가까이 배치하는 것만으로도
지연을 10분의 1로 줄일 수 있다. 선택은 아키텍처 복잡성과 인프라 비용 사이의 트레이드오프다.

이 패턴이 확산될 조건은 이미 갖춰지고 있다. IndexedDB, Service Worker, WebSocket,
Broadcast Channel API 같은 브라우저 기본 기능이 성숙했고, CRDTs를 구현한 라이브러리들
(Yjs, Automerge)이 쓸 수 있는 수준이 됐다. Linear가 2022년에 공개적으로 설명한 패턴을
2026년에는 더 많은 팀이 채택할 수 있는 조건이 됐다.

### 번들러 교체 전략은 장기적 코드베이스 관리의 패턴을 드러낸다

Parcel → Rollup → Vite → Rolldown의 경로는 4번의 번들러 교체다. 일반적으로
번들러 교체는 팀에 상당한 마이그레이션 비용을 요구하는 작업으로 여겨진다.
Linear가 이를 반복적으로 수행했다는 것은, 도구를 “바꿀 수 없는 기반”이 아니라
“교체 가능한 부품”으로 취급하는 관리 방식이 정착됐음을 시사한다.

이 전략의 핵심 전제 조건은 번들러에 종속된 특수 설정을 최소화하고, 번들러
추상화 레이어를 코드베이스와 명확히 분리하는 것이다. 이것이 갖춰지지 않으면
번들러 교체 비용이 지나치게 커져 “현재 도구가 최선이 아니더라도 유지”하는
관성이 생긴다.

이 패턴은 번들러에만 적용되지 않는다. Linear의 스택에서 React는 유지됐지만
상태 관리(MobX)와 번들러는 교체됐다. “무엇을 고정하고 무엇을 교체 가능하게 두는가”
라는 아키텍처 결정이 장기 코드베이스의 진화 가능성을 결정한다. 로컬 퍼스트 싱크
엔진은 교체하기 어려운 기반으로 선택됐고, 번들러는 교체 가능한 부품으로 설계됐다.
이 의도적 분리가 수년에 걸친 대규모 성능 개선을 가능하게 했다.

### 낙관적 UI는 사용자 경험 설계의 패러다임이지 단순한 기술 최적화가 아니다

낙관적 업데이트(Optimistic Update)를 “서버 응답 전에 UI를 먼저 업데이트하는
기법”으로 보면, 이것은 사용자 경험을 속이는 것처럼 보일 수 있다.
그러나 다른 관점에서 보면 이것은 사용자의 인지 모델에 맞춘 설계다.

사용자가 이슈 제목을 변경할 때, 그 행위는 사용자 관점에서 이미 완료됐다.
네트워크 왕복이 완료될 때까지 스피너를 보여주는 것은 사용자의 인지와 UI
상태를 불일치시키는 것이다. 낙관적 업데이트는 사용자의 의도가 실현된 시점을
네트워크 완료 시점이 아닌 사용자 행위 시점으로 정의한다.

이 패러다임 전환의 더 넓은 함의는, 오류 처리(Error Handling)의 설계가 바뀐다는
점이다. 전통적 방식에서는 서버 응답 실패가 “작업 실패”를 의미하고 사용자에게
즉시 알린다. 낙관적 UI에서는 서버 실패가 “비동기 롤백”을 의미하고, 성공 경로가
기본이다. Linear의 설계는 이 충돌이 드문 이벤트이고, 대부분의 경우 사용자의
의도가 서버에 성공적으로 반영된다는 데이터에 근거한 선택이다. “인터넷은
충분히 신뢰할 수 있다”는 가정을 명시적으로 받아들인 설계다.

역사적으로 이 패턴은 성숙한 네이티브 앱 플랫폼에서 기본이었다. macOS 파일
시스템 작업이 “완료됐다고 표시하고 백그라운드에서 처리”하는 방식, iOS 사진
앱이 편집을 즉시 반영하는 방식이 동일 원리다. Linear는 이것을 멀티 플레이어
웹 앱에 적용했다. 그리고 이것이 진짜 어려운 부분 — 네트워크 분단 중 두 사용자가
충돌하는 편집을 어떻게 해소하는가 — 은 싱크 엔진이 감춰진 곳에 있다.

hamandcheese는 낙관적 업데이트가 트렌드가 되면서 생길 위험을 날카롭게 짚었다.[^hamandcheese]
“sad path” — 동기화 실패, 충돌, 롤백 — 에 대한 설계 없이 낙관적 업데이트를 적용하는
팀들이 늘어나면, 새로운 형태의 경쟁 조건과 데이터 불일치 버그가 쏟아질 수 있다.
Linear가 싱크 엔진에 투자한 규모의 엔지니어링 없이 “낙관적 업데이트를 씁시다”는
결론만 가져가는 것은, 이 글이 경계해야 할 잘못된 독해다.

### 성능이 제품 차별화가 되는 시장은 드물고, Linear는 그 예외다

대부분의 B2B SaaS 제품에서 성능은 “충분히 빠르면 된다”의 임계점 문제다.
로딩이 2초가 걸려도 치명적이지 않다. 사용자는 느린 앱을 참고 쓰거나,
느린 경쟁사와 비교해 “이게 업계 표준”으로 받아들인다.

Linear는 성능을 제품 정체성으로 만들었다. “빠르다”는 것이 기능 목록의
항목이 아니라 제품이 주는 감각 자체가 됐다. 이것이 가능했던 조건은 두
가지다. 첫째, 이슈 트래커라는 카테고리에서 기존 도구(Jira, Asana)가 체감
성능에서 낮은 기준을 설정했다. 둘째, 개발자 도구 시장에서 사용자가 속도에
민감하다는 점이다. 개발자는 에디터, 빌드 도구, CLI에서 성능에 직접적으로
반응하는 사용자 집단이다.

이 두 조건이 맞물려 “빠른 이슈 트래커”가 시장 포지셔닝이 됐다. 그러나
이 전략의 일반화 가능성은 제한적이다. 만족도 조사, 헬스케어, 물류 관리
같은 영역에서 사용자는 성능보다 신뢰성, 컴플라이언스, 통합 편의성을
우선시한다. Linear의 아키텍처는 특정 시장 맥락과 사용자 특성이 만든
결과이고, 다른 맥락에 이식할 때는 그 전제를 먼저 검토해야 한다.
