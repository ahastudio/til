# The Svelte Compiler Handbook

원문: <https://lihautan.com/the-svelte-compiler-handbook>

HN 토론: <https://news.ycombinator.com/item?id=22784153> (330점, 131개 댓글)

## 요약

Tan Li Hau가 2020년 4월에 작성한 Svelte 컴파일러 내부 구조 핸드북이다.
Svelte 소스 코드를 직접 읽으려는 개발자를 대상으로,
컴파일 파이프라인을 파싱(parsing), 의존성 추적(tracking), 코드 블록 생성(creating), 코드 출력(generating) 네 단계로 나눠 설명한다.

파싱 단계에서 Svelte는 자체 HTML 파서를 사용해 소스 코드를 AST(Abstract Syntax Tree)로 변환한다.
“Svelte 문법은 HTML의 상위 집합”이며, `{#if}`, `{#each}`, `{:else}` 같은 비표준 구문을 처리하기 위해 커스텀 파서가 필요하다.
`<script>` 태그 안의 JavaScript는 acorn에 위임하고, CSS는 css-tree에 위임한다.
결과 AST는 HTML 프래그먼트, CSS 정보, 인스턴스 스크립트, 모듈 스크립트(`<script context=“module”>`) 네 부분으로 구성된다.

의존성 추적 단계는 `Component` 클래스에서 이루어진다.
`Component` 클래스는 HTML 프래그먼트, 렉시컬 스코프, 인스턴스 변수, 반응형 선언, 슬롯 정보, 경고와 오류, 컴파일 옵션 등 광범위한 메타데이터를 저장한다.
Svelte는 인스턴스/모듈 스크립트를 먼저 순회해 선언된 모든 변수를 파악한 뒤, 템플릿을 순회해 어떤 변수가 참조되거나 업데이트되는지 표시한다.

코드 생성 단계에는 두 렌더러가 존재한다.
DOM 렌더러는 `create_fragment` 함수를 위한 블록을 생성하며, 인스턴스 변수를 `$$.ctx`에 저장한다.
SSR 렌더러는 `add_string()`과 `add_expression()` 헬퍼를 이용해 템플릿 리터럴 방식으로 출력을 생성한다.
최종 렌더링 단계에서 DOM 렌더링은 렌더 트리를 순회하며 각 노드의 `render()` 함수를 호출하고,
SSR 렌더링은 특정 노드 핸들러에 위임한다.

## 분석

### 의존성 추적이 파이프라인 중심에 있는 구조적 이유

파이프라인 4단계 중 가장 중요한 것은 두 번째 단계, 즉 의존성 추적이다.
이 단계가 Svelte가 가상 DOM 없이 작동할 수 있는 이유를 직접적으로 설명한다.
React와 Vue는 “어떤 상태가 바뀌었는가”를 런타임에 계산한다.
가상 DOM은 이 계산을 위한 도구다 — 이전 트리와 현재 트리를 비교해 실제로 바뀐 부분만 DOM에 반영한다.

Svelte의 의존성 추적은 이 질문을 컴파일 타임으로 옮긴다.
“어떤 변수가 어떤 DOM 노드에 영향을 주는가”를 컴파일 과정에서 정적으로 파악해
컴파일 결과물에 직접 인코딩한다.
`count += 1` 같은 코드가 컴파일되면 `count`를 참조하는 DOM 노드를 직접 업데이트하는 코드가 생성된다.
런타임에 “무엇이 바뀌었나”를 계산하는 단계가 존재하지 않는다.

이 설계에서 의존성 추적 단계는 파이프라인 전체의 핵심이다.
이 단계의 정확성이 생성된 코드의 정확성을 결정하기 때문이다.
의존성을 잘못 추적하면 업데이트가 필요 없는 DOM 노드를 업데이트하거나,
업데이트가 필요한 노드를 누락한다.
이 단계가 두 번째에 있는 것은 우연이 아니다 — 이후의 모든 단계가 이 단계의 결과에 의존한다.

### 커스텀 파서 선택이 갖는 아키텍처적 함의

Svelte가 자체 HTML 파서를 구현한 것은 Svelte 특유의 템플릿 구문을 처리하기 위해서다.
표준 HTML 파서는 `{#if}`, `{#each}`, `{:else}` 같은 구문을 알지 못한다.
Svelte가 이 구문을 지원하려면 파서 수준에서 처리해야 한다.
JavaScript(acorn)와 CSS(css-tree)는 각 영역의 표준 파서에 위임하면서
HTML 템플릿만 직접 구현한 것은 책임 범위를 명확히 한 선택이다.

이 선택의 장점은 파서가 Svelte 구문에 완전히 최적화될 수 있다는 점이다.
어떤 구문이 허용되는지, 어떻게 중첩되는지, 어떤 위치에서 어떤 구문이 유효한지를
파서 설계 단계에서 인코딩할 수 있다.
Svelte 특유의 오류 메시지나 문법 검사도 이 단계에서 처리된다.

그러나 이 선택의 비용은 파서의 유지보수 부담이다.
Svelte 팀이 HTML 사양이 업데이트될 때마다 파서를 맞춰야 한다.
새로운 Svelte 문법 기능을 추가하려면 파서도 수정해야 한다.
이 비용은 일반적인 프레임워크 개발에서 보이지 않는 곳에 숨어 있다 — 사용자는 새 기능만 보지만
그 뒤에는 파서 수정이 수반된다.

### DOM과 SSR의 이중 렌더러 구조가 드러내는 설계 철학

컴파일 결과물이 두 가지 렌더러에 의해 생성된다는 것은
Svelte가 단일 런타임 환경을 가정하지 않는다는 의미다.
DOM 렌더러와 SSR 렌더러는 같은 AST와 의존성 정보를 입력으로 받지만 다른 출력을 만든다.
DOM 렌더러는 `document.createElement`, `addEventListener` 같은 DOM API를 직접 호출하는 코드를 생성하고,
SSR 렌더러는 HTML 문자열을 조합하는 코드를 생성한다.

이 분기가 컴파일 타임에 일어난다는 것이 중요하다.
React의 `ReactDOM`과 `ReactDOMServer`는 런타임 라이브러리 수준의 분기다.
같은 컴포넌트 코드가 어떤 환경에서 실행되는지에 따라 다른 런타임 경로를 탄다.
Svelte는 이 분기를 컴파일 시점에 확정한다.
DOM용 번들과 SSR용 번들은 처음부터 다른 코드를 갖는다.
결과적으로 런타임에 “지금 브라우저인가 서버인가”를 확인하는 코드가 없다.

이 구조는 SvelteKit이 파일 시스템 기반 라우팅과 서버 함수를 제공하는 방향의 기술적 기반이 된다.
어떤 컴포넌트가 서버에서만 렌더링되는지, 어떤 컴포넌트가 클라이언트에서 하이드레이션되는지를
빌드 타임에 결정할 수 있는 것은 컴파일러가 두 렌더러를 갖기 때문에 가능하다.

## 비평

### 핸드북이 “어떻게”를 설명하면서 “왜 이렇게”는 건너뛴다

이 글은 Svelte 컴파일러 파이프라인의 각 단계를 충실히 설명한다.
그러나 각 설계 결정의 이유는 대부분 다루지 않는다.
예를 들어 “왜 Svelte는 JSX 대신 자체 템플릿 문법을 선택했는가”,
“왜 파서를 acorn 기반 확장으로 만들지 않고 처음부터 직접 구현했는가”,
“왜 의존성 추적이 두 번째 단계인가 — 파싱과 동시에 처리할 수 없었는가” 같은 질문이 없다.

이 누락은 독자 대상의 선택이기도 하다.
저자는 “Svelte 소스 코드를 읽으려는 개발자”를 대상으로 한다고 명시한다.
이 독자는 “어떻게”를 알면 소스 코드 탐색을 시작할 수 있다.
“왜”는 소스 코드를 충분히 읽은 후에 스스로 추론해야 한다는 것이 이 글의 암묵적 가정이다.

그러나 이 가정이 핸드북의 유용성을 제한한다.
Svelte 컴파일러 소스를 읽으려는 사람은 대부분 특정 기능을 이해하거나 수정하려는 목적이 있다.
그 목적을 달성하려면 “어떻게”만으로는 부족하다.
설계 이유를 모르면 코드를 수정할 때 의도치 않게 불변식(invariant)을 깨는 변경을 만들 수 있다.

HN 토론에서 amelius는 다른 각도의 빈틈을 지적했다.[^amelius]
핸드북을 훑었지만 “Svelte의 핵심 특징, 즉 점진적(incremental) 업데이트가 어떻게 수행되는지”를 찾을 수 없었다는 것이다.
이것은 “어떻게”의 다른 차원이다.
파이프라인이 어떻게 생겼는지는 설명되지만,
그 파이프라인의 가장 중요한 출력 — 변경된 변수가 어떻게 최소한의 DOM 조작으로 이어지는지 — 이 명확히 보이지 않는다.
독자가 컴파일러 구조를 이해했다고 해도
“결국 어떤 코드가 생성되며, 그 코드가 어떻게 DOM을 부분 업데이트하는가”라는 질문에 스스로 답해야 한다.

### 2020년 기준 설명이 현재 Svelte 5 아키텍처와 불일치한다

핸드북은 2020년 4월 Svelte 3 기준으로 작성됐다.
DOM 렌더러의 핵심으로 설명된 `$$.ctx` 배열 기반 상태 관리는
Svelte 3의 반응성 구현 방식이다.
Svelte 5는 Runes 시스템을 도입해 이 내부 구조를 근본적으로 바꿨다.
`$state()`, `$derived()`, `$effect()` 같은 명시적 API가 도입됐고,
컴파일러가 생성하는 코드 구조도 변경됐다.

이 글을 읽고 Svelte 5 소스 코드를 탐색하러 가면
설명된 것과 다른 구조를 만나게 된다.
`$$.ctx` 배열이나 `create_fragment` 함수의 형태가 달라졌기 때문이다.
핸드북 자체에는 이 사실이 표시돼 있지 않다.

이것은 기술 문서의 공통 문제다.
프레임워크 내부를 설명하는 글은 프레임워크 자체보다 빠르게 낡는다.
내부 아키텍처는 공개 API보다 훨씬 자주 바뀌기 때문이다.
핸드북이라는 이름은 지속적으로 참조할 자료임을 암시하는데,
이 기대치와 실제 유효 기간 사이에 간극이 있다.

### SSR 렌더러 설명이 DOM 렌더러 대비 지나치게 간략하다

글의 분량 배분이 균형적이지 않다.
DOM 렌더러는 `create_fragment` 함수, `$$.ctx`, 반응성 업데이트 흐름까지 상세히 설명하지만
SSR 렌더러는 “템플릿 리터럴을 생성하고 `add_string()`과 `add_expression()`을 쓴다”는 수준에서 마무리된다.

SSR이 현대 웹에서 차지하는 위치를 생각하면 이 불균형이 아쉽다.
Next.js, Nuxt, SvelteKit이 모두 SSR을 핵심 기능으로 제공하고,
성능 최적화 논의에서 SSR은 빠지지 않는다.
Svelte의 SSR 렌더러가 어떻게 하이드레이션(hydration)과 연계되는지,
SSR에서 반응성이 어떻게 다뤄지는지는 실제로 SvelteKit 애플리케이션을 이해하는 데 중요하다.

SSR 렌더러가 간략히 다뤄진 이유가 2020년 당시 SSR이 지금만큼 중심 주제가 아니었기 때문일 수 있다.
그러나 핸드북이 현재도 검색을 통해 참조된다는 점에서,
SSR 설명의 부재는 독자가 가장 알고 싶어하는 부분에서 막히는 원인이 된다.

## 인사이트

### 정적 의존성 분석이 소프트웨어 개발 전반의 방향을 예고한다

Svelte 컴파일러의 의존성 추적 단계는 프론트엔드 프레임워크에만 해당하는 아이디어가 아니다.
“런타임에 계산해야 했던 것을 컴파일 타임으로 옮긴다”는 원칙은
소프트웨어 개발의 역사에서 반복적으로 나타나는 패턴이다.

타입 시스템이 이 패턴의 대표적 사례다.
JavaScript에서 런타임에 발생하던 타입 오류를 TypeScript는 컴파일 타임에 잡는다.
프리즈마(Prisma) 같은 ORM은 쿼리의 타입 안전성을 컴파일 타임에 검증한다.
Rust의 빌림 검사기(borrow checker)는 메모리 안전성을 런타임이 아닌 컴파일 타임에 보장한다.
이 모든 것의 공통 패턴은 “나중에 실패할 것을 일찍 확정한다”는 것이다.

Svelte의 의존성 추적도 같은 원칙이다.
“어떤 상태 변경이 어떤 DOM을 업데이트할 것인가”를 런타임이 아닌 컴파일 타임에 확정한다.
이 패턴이 계속 확산되는 이유는 런타임 오류와 런타임 오버헤드를 동시에 줄이기 때문이다.
컴파일러가 더 많이 알수록 런타임은 더 가볍고 안전해진다.

HN 토론에서 nojvek은 이 원칙을 React에 적용하는 아이디어를 제안했다.[^nojvek]
“Babel/TypeScript가 JSX를 분석해 어떤 attribute가 변수에 바인딩됐는지, 어떤 것이 정적인지 파악할 수 있다”는 것이다.
변경이 필요한 attribute만 diffing 대상으로 삼으면 가상 DOM의 비효율을 컴파일 타임 분석으로 줄일 수 있다.
이것은 React Compiler(이전의 React Forget)가 실제로 추구하는 방향과 일치한다.

그러나 spankalee는 이 최적화가 반드시 컴파일러를 요구하지 않는다는 반론을 제기했다.[^spankalee]
lit-html이 런타임에 같은 최적화를 달성한다는 것이다.
태그드 템플릿 리터럴의 정적 구조를 런타임에 한 번 파악해 이후 diffing에서 제외하는 방식이다.
이것은 “컴파일 타임 정적 분석”과 “런타임 정적 분석”의 구분을 만든다.
두 접근법 모두 “정적인 것을 한 번만 처리한다”는 원칙은 같지만 언제 처리하는지가 다르다.
컴파일러 방식은 서버에서 한 번 처리하고 번들에 결과를 담지만,
런타임 방식은 브라우저에서 한 번 처리하고 이후 실행에 재사용한다.
어느 쪽이 더 나은가는 앱의 특성(컴포넌트 수, 초기 로딩 비용, 캐싱 전략)에 따라 달라진다.

### 커스텀 파서를 보유한 시스템의 역사적 궤적

자체 파서를 가진 시스템은 초기에 설계 자유도가 높지만
생태계가 성숙하면서 특유의 압박을 받는다는 역사적 패턴이 있다.
C 컴파일러들이 각자 다른 파서를 갖다가 표준화 압력을 받은 것,
각 브라우저가 다른 HTML 파서를 갖다가 HTML5 파싱 알고리즘이 표준화된 것이 같은 패턴이다.

Svelte의 자체 HTML 파서는 현재 강점이지만 잠재적 병목이기도 하다.
Svelte 커뮤니티가 “왜 이 HTML 구문이 지원되지 않는가”, “왜 이 에지 케이스가 다르게 처리되는가”를 물을 때
답이 항상 “파서가 그렇게 구현됐기 때문”이 된다.
이것은 프레임워크 발전의 방향이 파서 구현 결정에 묶인다는 의미다.

TypeScript가 Babel과 달리 자체 파서를 유지하는 방향을 택한 것, 
그리고 Babel이 플러그인 아키텍처로 파서를 확장 가능하게 만든 것은
같은 문제에 대한 다른 해법이다.
Svelte가 언어 수준의 확장성을 제공하려면 결국 파서 아키텍처에 대한 결정이 필요하다.
Svelte 5의 Runes 도입이 템플릿 문법보다 JavaScript 내부에서 해결하는 방향을 선택한 것은
이 제약을 인식한 결과일 수 있다.

### 컴파일러 복잡도와 사용자 단순함은 비용을 교환하는 것이다

Svelte가 “React보다 코드가 적다”는 것은 Svelte 컴파일러가 더 많은 작업을 한다는 의미다.
이 교환은 명시적으로 언급되지 않는 경우가 많다.
`let count = 0`이 반응성을 갖는다는 “마법”은
컴파일러가 `let` 선언을 분석하고, 그것이 템플릿에서 어떻게 쓰이는지 파악하고,
업데이트 코드를 생성하는 비용으로 지불된다.

이 교환의 함의는 디버깅에서 드러난다.
사용자 코드가 단순하지만 생성된 코드가 복잡하다.
개발자 도구에서 보이는 것은 생성된 코드다.
버그가 생성된 코드에 있을 때, 원인을 찾으려면 컴파일러가 어떻게 소스를 변환했는지 이해해야 한다.
이 이해 없이는 스택 트레이스가 원본 소스와 연결되지 않는다.

이 핸드북이 존재하는 이유도 여기 있다.
컴파일러 내부를 이해해야 하는 개발자가 생겼다는 것은 Svelte 생태계가 성숙했음을 보여주는 신호다.
입문 단계에서는 “마법처럼 작동한다”는 설명으로 충분하지만,
프레임워크의 한계에 부딪히거나 성능을 최적화하거나 기여하려면
마법의 내부를 알아야 한다.
컴파일러 복잡도를 사용자에게 숨기는 것은 입문 경험을 낮추는 대신
심화 단계의 진입 장벽을 높인다.

### 두 렌더러 구조가 “실행 환경을 명시하는” 방향의 선구자였다

2020년 당시 Svelte의 DOM/SSR 이중 렌더러는 비교적 조용한 기술적 선택이었다.
그러나 이후 React Server Components, Astro의 아일랜드 아키텍처, Qwik의 재개가능(resumable) 실행이
모두 “어떤 코드가 어디서 실행되는가”를 컴파일/빌드 타임에 명시하는 방향으로 수렴했다.

이 패턴의 핵심은 “클라이언트와 서버가 동일한 코드를 공유한다”는 isomorphic/universal rendering의 가정을 버리는 것이다.
Isomorphic rendering은 “같은 컴포넌트를 서버에서도 클라이언트에서도 실행한다”는 편리함이 있지만,
클라이언트에 필요 없는 서버 코드가 번들에 포함되거나, 서버에서의 실행 비용이 설계상 고려되지 않는 문제가 있다.

Svelte가 컴파일 타임에 두 렌더러로 분기한 것은
이 문제를 프레임워크 수준이 아닌 컴파일러 수준에서 다룬 것이다.
두 렌더러의 존재는 단순한 기술적 구현 이상으로,
“실행 환경을 선택하는 것은 런타임이 아닌 빌드 타임의 결정이다”라는 아키텍처 관점을 담고 있다.
이 관점이 2020년 이후 웹 개발의 중심 논쟁이 됐다.
