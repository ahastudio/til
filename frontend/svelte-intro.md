# Introduction to Svelte

원문: <https://daveceddia.com/svelte-intro/>

HN 토론: <https://news.ycombinator.com/item?id=20702186> (187점, 108개 댓글)

## 요약

Dave Ceddia가 2019년 7월에 작성한 Svelte 3 입문 글이다.
저자는 Svelte를 React와 비교하며 “React보다 빠르고 쉽게 느껴진다”는 직관적 인상으로 글을 시작한다.
React를 깊이 다뤄온 저자가 Svelte를 처음 접하고 느낀 흥미와 가능성을 전달하는 것이 글의 핵심 동력이다.

Svelte의 가장 중요한 기술적 특성은 컴파일러라는 점이다.
React와 Vue는 런타임에 가상 DOM(Virtual DOM)을 통해 변경사항을 계산하고 실제 DOM에 반영하는 방식을 쓴다.
Svelte는 빌드 타임에 컴포넌트를 순수 JavaScript로 컴파일해 런타임 프레임워크 코드 없이 DOM을 직접 조작하는 코드를 생성한다.
이 방식으로 Hello World 앱의 번들 크기가 2.3KB(gzip 후 1.16KB)에 불과하다.
같은 앱을 Create React App으로 만들면 37.6KB다.
node_modules 규모도 Svelte Hello World는 29MB 242개 패키지, Create React App은 204MB 1,017개 패키지로 차이가 크다.

코드 분량의 차이도 강조된다.
Hello World 기준으로 Svelte는 4줄(script 태그에 변수, HTML 템플릿), React는 6줄(import, JSX, export)이다.
저자는 이 차이가 단순한 줄 수 이상으로 개발 경험의 차이를 만든다고 본다.

반응성(reactivity) 모델은 변수 재할당으로 작동한다.
`items = items.filter(...)` 같은 재할당이 UI 업데이트를 촉발하지만, `items.push()`처럼 변경(mutation)하면 Svelte가 감지하지 못한다.
템플릿 문법은 `{#each items as item}...{/each}` (리스트), `on:click` (이벤트 핸들러), `on:submit|preventDefault` (이벤트 수정자), `bind:checked={item.done}` (양방향 바인딩), `class:done={item.done}` (조건부 클래스)로 구성된다.

스타일은 `<style>` 태그에 작성하며 컴파일러가 자동으로 스코프를 지정해 전역 충돌이 없다.
사용하지 않는 CSS 선택자는 컴파일러가 감지해 에디터에서 경고를 표시한다.
글은 쇼핑 리스트 애플리케이션을 완성하며 마무리되고, 다루지 못한 기능들(멀티 컴포넌트, props, slots, 반응형 구문, async/await, 내장 애니메이션과 전환, 라이프사이클 메서드, Context API, 반응형 스토어)을 짧게 언급한다.

## 분석

### “컴파일러로서의 프레임워크”라는 명제가 노리는 것

Svelte의 핵심 아이디어는 “프레임워크가 런타임에 작동할 필요가 없다”는 것이다.
React는 가상 DOM을 운영하는 런타임이 브라우저에서 계속 실행된다.
Svelte는 이 런타임의 역할을 빌드 타임에 컴파일러로 대체한다.
컴포넌트 코드를 분석해 어떤 변수가 어떤 DOM 노드에 영향을 주는지 정적으로 파악하고,
그 관계를 직접 조작하는 JavaScript 코드를 생성한다.

이 접근법의 함의는 단순히 번들 크기를 줄이는 것이 아니다.
“어떤 상태가 바뀌면 어디가 업데이트되는가”라는 질문을 런타임이 아닌 컴파일 타임에 답하는 것이다.
가상 DOM 방식은 이 질문을 런타임에 매번 다시 계산한다.
Svelte 방식은 답을 컴파일 결과에 고정한다.
이것은 계산 시점의 이동이며, 그 이동이 런타임 성능 향상과 번들 크기 감소를 동시에 가져온다.

이 아이디어는 Rich Harris의 “Rethinking Reactivity” 발표에서 체계화됐고,
글은 이 관점을 그대로 수용한다.
2019년 당시에는 이 방향이 특이한 베팅이었지만,
이후 Solid.js, Qwik, Astro, Million.js 등이 유사한 방향을 채택하면서
“컴파일러 기반 프레임워크”가 프론트엔드 아키텍처의 한 흐름으로 자리잡았다.

### Hello World 비교가 전달하는 것과 전달하지 못하는 것

2.3KB vs 37.6KB, 4줄 vs 6줄 비교는 효과적인 도입부다.
숫자가 직관적이고 차이가 크기 때문이다.
그러나 이 비교는 특정 규모에서만 유효하다.
React의 37.6KB는 앱 크기와 관계없이 한 번 내는 고정 비용이다.
Svelte가 컴파일한 코드는 컴포넌트마다 생성되므로 앱이 커질수록 누적된다.
Hello World에서는 Svelte가 압도적으로 유리하지만, 컴포넌트 수가 늘어날수록 이 차이가 좁혀진다.

글은 이 트레이드오프를 다루지 않는다.
번들 크기는 “앱 규모가 커지면 어떻게 되는가”라는 질문과 분리해서 논의할 수 없다.
React 팀이 가상 DOM을 포기하지 않는 이유 중 하나가 이 스케일링 특성이다.
Svelte의 접근법이 현실의 중간 규모 앱에서 어떤 성능 특성을 보이는지는
Hello World 비교로 알 수 없다.

### 재할당 기반 반응성의 인지 모델이 갖는 특성

`items = items.filter(...)` 재할당이 UI를 갱신하고 `items.push()`는 갱신하지 않는다는 규칙은
처음에는 직관적으로 보인다.
그러나 이 규칙은 JavaScript의 참조 동작 방식을 프레임워크 계층에서 재해석한 것이다.
Svelte 컴파일러가 할당 연산자(`=`)를 감지해 업데이트 트리거를 삽입하기 때문이다.
`push()`는 할당이 아니므로 감지되지 않는다.

이 방식은 React의 불변성(immutability) 규칙과 유사한 인지적 부담을 다른 형태로 요구한다.
React는 “상태를 직접 변경하지 말고 새 객체를 반환해라”를 요구한다.
Svelte는 “변경 후 반드시 재할당해라”를 요구한다.
두 규칙 모두 JavaScript 기본 동작이 아닌 프레임워크 관행이다.
어느 쪽이 더 자연스러운지는 경험과 관점에 따라 다르다.
글은 Svelte 방식이 더 직관적이라고 암묵적으로 가정하지만 이것은 검증 없는 전제다.

### 2019년 맥락이 글의 타당성 범위를 제한한다

이 글은 Svelte 3가 출시된 직후인 2019년 7월에 쓰였다.
당시 Svelte 3는 Svelte 2와의 완전한 단절을 선언하며 새로운 반응성 모델을 도입한 직후였고,
커뮤니티의 흥분이 정점에 있을 때였다.
그 맥락이 글의 톤 — “빠르고 쉽게 느껴진다”는 흥분 — 에 영향을 미쳤을 가능성이 있다.

2024~2025년 현재 Svelte는 Svelte 5로 버전이 올라갔으며,
Runes라는 새로운 반응성 시스템을 도입해 이 글에서 설명한 재할당 기반 반응성 모델에서 상당히 벗어났다.
`$state()`, `$derived()`, `$effect()` 같은 명시적 룬(rune)이 도입됐다.
이 글이 핵심으로 소개한 반응성 작동 방식이 Svelte 5에서는 더 이상 기본 방식이 아니다.

## 비평

### 번들 크기 비교가 상황에 따라 역전되는 구조를 숨긴다

글에서 제시한 2.3KB vs 37.6KB 비교는 빈 앱에서의 측정값이다.
이 숫자는 Svelte의 접근법이 극단적으로 유리한 조건, 즉 컴포넌트가 1개일 때의 결과다.
Svelte의 컴파일 방식은 각 컴포넌트에 대해 DOM 조작 코드를 별도로 생성한다.
React는 런타임 비용을 한 번 치르고 이후 모든 컴포넌트에 재사용한다.

실제 애플리케이션에서 두 방식의 번들 크기가 교차하는 지점이 존재한다.
이 교차점의 정확한 위치는 컴포넌트 복잡도에 따라 다르지만,
일부 벤치마크는 중간 규모 앱(수십 개 컴포넌트)에서 Svelte가 React보다 큰 번들을 생성하는 경우를 보여준다.
글은 이 역전 가능성을 전혀 언급하지 않는다.

이것은 단순한 누락이 아니라 논증 구조의 문제다.
“번들 크기가 작다”는 Svelte의 핵심 주장 중 하나인데,
이 주장이 어떤 조건에서 성립하는지 경계를 정의하지 않는 것은
독자가 주장의 적용 범위를 잘못 이해하게 만든다.

HN 토론에서 playpause는 이 논의에 실제 앱 데이터를 추가했다.[^playpause]
글에서 “29MB”는 컴파일러와 개발 의존성의 크기이지 최종 앱 번들이 아니라는 것이다.
실제 TodoMVC 기준 비교에서는 React 앱이 전송 289KB(압축 해제 1,255KB), Svelte 앱이 전송 28KB(압축 해제 41KB)였다.
이 데이터는 할 일 목록 수준의 앱에서 Svelte의 번들 이점이 실재함을 보여주지만,
“29MB”라는 숫자가 개발 환경에만 해당한다는 점에서 글의 비교 프레임이 맥락 없이 전달되고 있음도 드러낸다.

### 가상 DOM 비판이 단순화된 버전의 Rich Harris 논지를 그대로 수용한다

글은 Svelte가 가상 DOM을 쓰지 않는다는 것을 장점으로 제시하는데,
이 논리는 Rich Harris의 “Virtual DOM is pure overhead” 주장에 기반한다.
가상 DOM을 쓰지 않으면 빠를 수밖에 없다는 결론이 당연한 것처럼 서술된다.
그러나 가상 DOM의 역할은 성능만이 아니다.

가상 DOM은 상태 변경을 배치(batch)로 처리하는 메커니즘을 제공한다.
React 18의 Concurrent Mode와 Transitions는 가상 DOM 위에서 구현된 기능이다.
사용자 상호작용을 우선순위에 따라 중단하고 재개하는 기능, 스트리밍 서버 렌더링, Suspense 같은 기능들은
렌더링을 “가상”으로 표현하는 계층이 있기 때문에 가능하다.
Svelte는 이런 기능들을 다른 방식으로 접근해야 한다.

글은 가상 DOM 비판을 기정사실로 제시하지만,
이 비판이 성립하는 조건과 성립하지 않는 조건이 있다는 점은 다루지 않는다.
비판적 시각이 있는 독자라면 자연스럽게 물어볼 “가상 DOM이 없으면 어떤 기능을 포기하는가”라는 질문이 없다.

### 입문 DX와 프로덕션 채택 사이의 간극이 보이지 않는다

글은 Svelte를 “React보다 빠르고 쉽게 느껴진다”는 인상으로 시작해
쇼핑 리스트 예제를 완성하는 것으로 마무리된다.
이 경험 — 첫 컴포넌트를 만들고 작동하는 것을 보는 경험 — 은 실제로 설득력 있다.
Svelte의 입문 경험이 React보다 단순하다는 것은 근거 있는 주장이다.

그러나 개발자가 개인 프로젝트나 사이드 프로젝트에서 Svelte를 탐색하는 것과
팀이 프로덕션 서비스에 Svelte를 채택하는 것은 매우 다른 결정이다.
글에서 언급만 하고 다루지 않은 기능들 — 멀티 컴포넌트, 라이프사이클, 스토어, Context API — 은
복잡한 앱을 만들 때 핵심이 된다.
상태 관리, 라우팅, 서버 사이드 렌더링, 테스트 전략이 어떻게 다른지는 다루지 않는다.

또한 팀 채용 풀의 문제가 있다.
React 개발자는 찾기 쉽지만 Svelte 경험을 가진 개발자는 드물다.
이 현실적 제약이 팀 규모가 커질수록 중요해진다.
글은 입문자의 첫 경험을 다루기 때문에 이 질문을 다루지 않는 것이 자연스럽지만,
결론이 “더 좋은 프레임워크”라는 인상을 주는 만큼, 이 누락은 결론의 적용 범위를 좁힌다.

HN 토론에서 gedy는 구체적인 사례로 테스팅 인프라를 들었다.[^gedy]
Svelte에서 테스팅, 특히 단위 테스트가 후순위로 다뤄진다는 것이 멀티팀 기업 환경에서의 채택을 막았다는 경험이었다.
흥미롭게도 이 댓글에 Svelte 창작자 Rich Harris 본인이 직접 답했다.[^rich_harris]
“테스팅의 가장 큰 장벽은 방법을 모르는 것이 아니라 무엇을 테스트해야 하는지를 모른다는 것”이라는 답변이었다.
이 답은 Svelte 생태계가 테스팅을 기술적 문제가 아닌 철학적 문제로 접근하고 있음을 드러낸다.
그러나 기업 환경의 팀에게 이 대답은 채택 결정을 연기할 충분한 이유가 된다.

writepub는 이 한계를 다른 각도에서 짚었다.[^writepub]
“쇼핑 리스트보다 복잡한 것을 가지고 Svelte를 테스트 드라이브 해줄 사람이 필요하다 — Svelte 할 일 목록 블로그 글과 튜토리얼은 넘쳐나지만 반복적이고 설득력이 없다”는 것이다.
입문 예제의 과잉과 실제 복잡도를 다루는 사례 연구의 부재는 Svelte 생태계가 2019년 당시 갖고 있던 구조적 공백이었다.
이 공백은 입문 경험이 아무리 우수해도 채택을 설득하지 못하는 이유를 설명한다.

### Svelte 3 기준 설명이 Svelte 5에서 오해를 만든다

이 글의 가장 실질적인 문제는 2019년에 쓰인 글이 업데이트 없이 유통된다는 점이다.
Svelte 5는 Runes 시스템을 도입해 반응성 모델을 근본적으로 변경했다.
`let count = 0; count += 1`처럼 변수 재할당으로 반응성을 표현하던 방식이
`let count = $state(0); count += 1`로 바뀌었다.
`$derived()`, `$effect()`, `$props()` 같은 명시적 API가 추가됐다.

이 글을 읽고 Svelte를 배우려는 사람이 현재의 Svelte 5 문서를 열면
글에서 설명한 것과 다른 패턴을 만나게 된다.
재할당 기반 반응성이라는 핵심 특징이 바뀌었기 때문에
글의 가장 매력적인 부분 — “변수를 재할당하면 UI가 바뀐다” — 이 현재 상태와 다르다.
글 자체에는 이 사실이 표시돼 있지 않다.

2019년 당시에도 이 문제의 전조가 있었다.
HN 토론에서 Vinnl은 Pinafore(Mastodon 클라이언트)를 Svelte로 구축한 실제 앱 사례로 언급하면서도,
v2에서 v3로의 업그레이드가 상당히 큰 변경이었다는 점을 들며
API와 도구가 더 성숙할 때까지 기다리길 권했다.[^Vinnl]
그 조언은 결과적으로 더 적절했다 — Svelte 5의 Runes 도입이 반응성 모델을 근본적으로 바꿨기 때문이다.
메이저 버전 간 단절이 반복되는 프레임워크에서는 “현재 버전으로 배웠다”는 것이 다음 버전에서 부채가 된다.

## 인사이트

### “컴파일러 프레임워크” 방향이 이후 생태계의 예언이 됐다

2019년 Svelte 3의 등장은 단순히 새 프레임워크의 출시가 아니었다.
“프레임워크는 런타임이어야 한다”는 암묵적 전제를 깨는 아키텍처 방향의 선언이었다.
당시에는 React, Vue, Angular가 모두 런타임 방식이었고,
Svelte의 컴파일러 방식은 이단적 선택으로 보였다.

그러나 이후 5년간 프론트엔드 생태계는 컴파일러 방향으로 이동했다.
Solid.js는 Svelte의 철학을 JSX와 결합했다.
Qwik은 컴파일 타임 분석으로 지연 로딩을 극단화했다.
Astro는 “아일랜드 아키텍처”로 서버에서 최대한 처리하고 클라이언트 JS를 최소화했다.
React 자체도 React Server Components와 Compiler(React Forget)로 컴파일 타임 분석을 도입했다.
“빌드 타임에 가능한 것을 런타임에 하지 않는다”는 원칙이 생태계 전반의 방향이 됐다.

이 맥락에서 이 글은 역사적 문서로서의 가치가 있다.
Svelte 3가 이 방향을 가장 명확하게 표현했고,
Dave Ceddia 같은 React 커뮤니티 내부의 사람이 주목했다는 것이
이 아이디어가 주변부 실험에서 주류 논의로 진입하는 신호였다.

### 번들 크기 논쟁이 실제로는 다른 문제를 숨긴다

번들 크기는 2019년 프론트엔드 논쟁의 핵심 지표였다.
Svelte가 “2.3KB”를 내세운 것은 이 지표가 중요하다는 공통 전제 위에서만 설득력이 있다.
그런데 번들 크기가 실제로 무엇을 대리하는가?

번들 크기는 네트워크 전송 비용과 파싱/실행 비용의 대리 지표다.
그런데 이 비용이 실제 사용자 경험에 미치는 영향은 맥락에 따라 크게 다르다.
HTTP/2, 코드 스플리팅, CDN 캐싱, Service Worker 캐싱이 일반화된 현재는
첫 번째 방문에서의 번들 크기 차이가 실질적 UX 차이로 이어지는 경우가 점점 줄어든다.
재방문 사용자에게 번들 크기는 캐시 히트율에 달려 있다.

더 중요한 맥락은 JavaScript 자체의 비용이다.
같은 크기의 이미지와 JavaScript는 브라우저 비용이 다르다.
JavaScript는 파싱, 컴파일, 실행이 필요하기 때문에
번들 크기가 줄어들면 이 모든 비용이 비례해 줄어든다.
이 맥락에서 번들 크기 논쟁은 표면적으로는 프레임워크 선택 논쟁이지만,
실질적으로는 “클라이언트 JavaScript 비용을 어떻게 최소화할 것인가”라는 더 큰 질문의 하위 논쟁이다.

### React의 생태계 독점이 기술적 우위와 독립적으로 작동한다

이 글을 읽는 자연스러운 결론은 “Svelte가 기술적으로 React보다 낫다면 Svelte가 대세가 될 것이다”다.
그러나 이것은 기술 시장이 작동하는 방식에 대한 잘못된 전제다.
React는 2019년 이후에도 시장 점유율을 유지했고, 오히려 확장됐다.

기술적 우위가 시장 우위로 이어지지 않는 구조적 이유가 있다.
첫째, 채용 시장이 React를 기본값으로 선택한다.
React 경험자를 뽑는 것이 Svelte 경험자를 뽑는 것보다 훨씬 쉽다.
이 채용 마찰이 회사의 기술 선택을 제약한다.

둘째, Next.js 같은 메타프레임워크가 React를 묶어 팔기 때문에
React를 선택하면 검증된 전체 스택이 따라온다.
SvelteKit이 이 역할을 하려 하지만 Next.js의 생태계와 채택률에는 미치지 못한다.

셋째, React 팀은 기술적 한계를 인지하고 Concurrent Mode, Server Components, Compiler 같은 방향으로 대응했다.
“React가 더 낫다”는 것이 아니라 “React가 나쁜 상태로 머물지 않는다”는 것이
기존 React 투자를 정당화한다.
Svelte의 기술적 혁신은 실질적으로 React의 발전 방향에 영향을 줬지만,
그 영향의 수혜자는 React 사용자들이었다.

HN 토론에서 dangoor는 Khan Academy의 사례로 이를 구체화했다.[^dangoor]
Khan Academy는 React 출시 직후 채택했고 6년이 지난 2019년에도 전환을 고려하지 않고 있다는 것이었다.
그의 관찰은 “React를 대체하려면 jQuery와 Backbone에서 React로 이동하던 것만큼의 전환이 필요할 것”이라는 것이었다.
그 이전 전환이 가능했던 이유는 jQuery+Backbone이 진짜 문제 — 복잡한 상태 관리, DOM 조작의 혼란 — 를 충분히 풀지 못했기 때문이다.
React가 기존 React 코드베이스에서 그 수준의 진짜 문제를 만들지 않는 한,
같은 수준의 이탈 동기가 발생하지 않는다.
Svelte가 더 낫다는 것과 Svelte로 이동할 이유가 충분하다는 것은 다른 명제다.

### 입문 최적화 설계가 장기 학습 경로를 희생한다

Svelte의 입문 경험이 React보다 단순한 것은 사실이다.
`<script>let count = 0</script>`, `<button on:click={() => count++}>{count}</button>`으로
완성되는 카운터는 프레임워크 개념을 최소한으로 요구한다.
이 단순함은 의도된 설계다.

그러나 이 설계에는 트레이드오프가 있다.
입문 경험을 단순하게 만들기 위해 Svelte는 컴파일러가 많은 것을 암묵적으로 처리한다.
무엇이 일어나는지 명시적으로 드러나지 않는다.
React는 상태 업데이트가 `useState`, 사이드 이펙트가 `useEffect`, 컨텍스트가 `useContext`로 명시적이다.
입문할 때는 이 명시성이 부담이지만, 앱이 복잡해질 때는 명시성이 추론 가능성(reasonability)을 제공한다.

“무엇이 일어나고 있는가”를 코드에서 읽을 수 있는가 없는가는
개인 프로젝트에서는 덜 중요하지만 팀 협업에서는 더 중요해진다.
Svelte 5가 Runes를 도입해 명시성을 높인 것은 이 트레이드오프에 대한 자기 인식을 보여준다.
`let count = 0`이 반응성을 갖는다는 것은 마법처럼 보이지만,
`let count = $state(0)`은 명시적이다.
Svelte 5의 선택은 Svelte 3의 “마법 같은 단순함”이 장기 코드베이스에서 의도치 않은 비용을 만든다는 판단이었다.
그 판단이 옳다면, 이 글이 가장 매력적이라고 소개한 특성은 장기적으로는 약점이었다는 뜻이다.
