# Oracle Cloud 완전 무료 티어로 백엔드 구축하기

<https://yoramilji.kr/blog/394765fe-8b67-81a9-94b6-ceb34a4c353c>

## 요약

이 글은 Oracle Cloud의 영구 무료 티어(Always Free)만으로 월 비용
0원짜리 백엔드 인프라를 구축한 경험을 다룬다. 2026년 기준
인프라 지형을 보면, Oracle Always Free 스펙은 4코어/24GB에서
2코어/12GB ARM으로 축소됐고, Render·Fly.io·Railway 같은 기존
무료 티어 옵션들은 대부분 24/7 상시 운영에는 비현실적인 수준으로
바뀌었다. 저자는 Oracle이 여전히 서버와 데이터베이스를 상시
운영할 수 있는 유일한 무료 옵션이라고 판단한다.

실제 구성은 Caddy(HTTPS 자동화), NestJS API, PostgreSQL 16을
Docker Compose로 오사카 리전의 A1 인스턴스(2코어/12GB RAM)에
올리는 방식이다. DNS는 별도 가입이 필요 없는 sslip.io를 이용해
라우팅한다.

가장 큰 난관은 무료 ARM 인스턴스의 극심한 확보 경쟁이었다.
저자는 OCI CLI로 인스턴스 생성을 반복 시도하는 자동화 스크립트를
만들어 오류를 분류하고, 지터를 넣은 요청 스로틀링으로 속도 제한을
피하는 방식으로 해결했다. 무료 용량이 부족해지자 계정을
종량제(PAYG)로 전환했는데, 핵심은 "Always Free" 한도 안의
리소스는 유료 계정으로 전환해도 여전히 무료로 유지된다는 점이다.
월 1,500 OCPU시간 한도 안에서 사용하면 인스턴스 비용은 계속
0원이다. 예산 알림은 통보만 할 뿐 과금을 막지는 못하기 때문에,
실제 안전장치는 필요한 리소스만 명시적으로 허용하고 나머지는
차단하는 쿼터 정책으로 마련했다.

저자는 "무료의 대가는 시간으로 치렀다"고 요약한다. 설정, 트러블
슈팅, 지속적인 모니터링에 들어간 노력이 상당했다는 뜻이다.

## 분석

### "무료"의 정의가 리소스 한도이지 계정 등급이 아니다

이 글의 핵심 통찰은 Oracle의 Always Free 혜택이 계정의 결제
등급(무료 대 PAYG)이 아니라 리소스 사용량 한도에 묶여 있다는
점이다. 대부분의 사용자는 "무료 계정"과 "유료 계정"을 이분법으로
생각하지만, Oracle의 구조에서는 PAYG로 전환해도 Always Free
한도 안의 사용량은 계속 무료로 남는다. 이 구분을 모르면 무료
인스턴스 확보 경쟁에 밀려 서비스를 포기하거나, 반대로 전환을
두려워해 필요한 용량 확장을 하지 못하는 두 가지 실수를 모두
저지르게 된다.

## 비평

### 자동 재시도 스크립트가 이용약관 경계에 있을 수 있다

무료 인스턴스 확보를 위해 API 호출을 반복 시도하는 자동화는
사실상 리소스 경쟁에서 우위를 점하기 위한 봇 행위에 가깝다.
Oracle이 이런 패턴을 공식적으로 허용하는지, 아니면 묵인하는
수준인지는 글에서 다뤄지지 않는다. 댓글에서 okxrr이 농담처럼
지적했듯, 이런 사용 사례가 널리 알려질수록 Oracle이 무료 티어
정책 자체를 축소할 유인이 커질 수 있다는 점은 실질적인 리스크다.

## 인사이트

### 스펙 축소가 실시간으로 퍼지는 것도 커뮤니티의 역할이다

댓글에서 modac0302는 기존 인스턴스는 4코어 스펙을 유지하며 축소는
신규 인스턴스에만 적용된다고 정정했고, koyokr는 이 정보에 감사를
표했다. 클라우드 사업자의 무료 티어 정책은 공지 없이 조용히
바뀌는 경우가 많아, 실사용자들의 최신 정보 공유가 실제 의사결정에
중요한 역할을 한다. baeba가 제안한 Cloudflare 터널을 이용한
셀프호스팅 대안 역시, 무료 인프라를 확보하려는 개발자들 사이에서
계속 새로운 조합이 실험되고 있음을 보여준다.
