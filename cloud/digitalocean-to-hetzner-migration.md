# DigitalOcean에서 Hetzner로 마이그레이션하기

원문: <https://isayeter.com/posts/digitalocean-to-hetzner-migration/>

## 요약

터키에서 소프트웨어 회사를 운영하는 저자는 인플레이션과 환율 하락으로 인해 인프라 비용
부담이 커지면서 DigitalOcean에서 Hetzner로 이전을 결정했다. 월 $1,432이던 비용이
$233으로 줄어 연간 $14,388을 절감했으며, 오히려 하드웨어 사양은 더 좋아졌다.
Hetzner의 AX162-R 전용 서버는 AMD EPYC 9454P 96논리 CPU, 256GB DDR5 RAM,
2TB NVMe RAID1을 제공한다.

마이그레이션 대상은 30개의 MySQL 데이터베이스(248GB), 34개의 Nginx 가상 호스트,
GitLab EE(42GB), Neo4J 그래프 데이터베이스(30GB), 수십만 사용자가 사용하는 모바일
앱 등이다. 동시에 CentOS 7에서 AlmaLinux 9.7로 OS도 업그레이드했다.

제로 다운타임을 달성하기 위해 6단계 전략을 사용했다. 새 서버에 모든 서비스를 먼저
설치하고, rsync로 웹 파일을 동기화하고, MySQL 리플리케이션으로 데이터를 실시간
복제한 뒤, DNS TTL을 낮추고, 기존 서버를 리버스 프록시로 전환한 후, DNS를 일괄
전환했다.

mydumper/myloader를 32개 병렬 스레드로 사용해 전통적인 mysqldump 대비 수일이 걸릴
작업을 수시간 내에 완료했다. MySQL 5.7에서 8.0으로의 버전 차이로 인한 호환성 문제도
기술하고 있다. 전체 마이그레이션은 약 24시간 내에 사용자 영향 없이 완료되었다.

## 분석

### 클라우드 대 전용 서버의 경제학

이 사례에서 가장 눈에 띄는 것은 비용 차이의 규모다. 동일하거나 더 나은 사양의 하드웨어를
월 $233에 확보했다는 것은 DigitalOcean이 6배 이상의 프리미엄을 부과하고 있었다는 뜻이다.
클라우드의 가치 제안은 탄력성(elasticity)과 관리 편의성이지만, 워크로드가 안정적이고
예측 가능한 경우 전용 서버가 경제적으로 월등히 유리하다.

특히 터키처럼 자국 통화가 약세인 국가에서는 달러 기반 클라우드 비용이 실질적으로 더 빠르게
증가한다. 이는 신흥국 기업들에게 클라우드 탈출(cloud exit)이 생존 전략이 될 수 있음을
보여준다.

### 마이그레이션 아키텍처의 완성도

6단계 전략은 교과서적으로 잘 설계되어 있다. 특히 MySQL 리플리케이션을 활용한 데이터
동기화와 리버스 프록시를 통한 DNS 전파 기간 대응은 실무에서 검증된 패턴이다. DNS TTL을
A/AAAA 레코드만 선별적으로 낮추고 MX/TXT 레코드는 유지한 것도 이메일 중단을 방지하는
세심한 접근이다.

Python 스크립트로 34개 Nginx 설정을 자동으로 리버스 프록시로 변환한 부분은 자동화의
가치를 잘 보여준다. 수작업으로 했다면 실수 가능성이 높았을 것이다.

### MySQL 마이그레이션의 실전 함정

MySQL 5.7에서 8.0으로의 전환에서 발생한 문제들이 실무적으로 가장 유용한 정보다.
`mysql.user` 테이블의 컬럼 수 차이(45 vs 51), `sys` 스키마가 뷰가 아닌 일반 테이블로
임포트되는 문제, SUPER 권한이 `read_only` 플래그를 우회하는 문제 등은 문서화되어 있지
않으면 현장에서 발견하기 어렵다.

`slave_exec_mode = 'IDEMPOTENT'` 설정으로 덤프 중 삽입된 행의 중복 키 에러를 해결한
것도 실무 팁으로 가치가 높다.

## 비평

### 강점

실제 프로덕션 환경에서의 마이그레이션을 상세히 기술한 점이 가장 큰 강점이다. 30개
데이터베이스, 34개 가상 호스트, 수십만 사용자라는 규모는 충분히 현실적이며, 발생한
문제와 해결 방법을 숨기지 않고 공유한다.

모든 마이그레이션 스크립트를 GitHub에 공개한 것도 커뮤니티에 실질적인 기여다.
DNS 관리, 전환 자동화, Nginx 설정 변환, GitLab 웹훅 업데이트 등의 스크립트는
유사한 마이그레이션을 계획하는 엔지니어에게 직접 참고할 수 있는 자료다.

### 약점

단일 서버로의 이전이므로 고가용성(HA)에 대한 논의가 빠져 있다. GeekNews 댓글에서도
지적되었듯이, 하드웨어 장애 시 모든 서비스가 동시에 영향을 받는다. 클라우드에서 전용
서버로 이전할 때 가장 자주 제기되는 우려인데, 이에 대한 저자의 입장이나 대응 전략이
없다.

비용 비교에서 숨겨진 비용을 충분히 다루지 않는다. 전용 서버는 하드웨어 장애 시 교체
시간, 네트워크 문제 시 자체 대응 필요, 백업 인프라 별도 구축 등의 비용이 추가된다.
$14,388 절감이라는 수치는 이러한 운영 비용을 포함하지 않은 것이다.

Hetzner의 IP 주소가 기업 방화벽에서 차단되는 경우가 있다는 댓글도 있었다.
이는 B2B 서비스에서 중요한 고려 사항이지만 글에서 다루지 않았다.

## 인사이트

### 클라우드 탈출은 성숙도의 지표다

클라우드 도입은 초기 성장 단계에서 합리적이다. 인프라 팀이 없고 워크로드가 불확실할 때
탄력적 확장은 진짜 가치를 제공한다. 그러나 워크로드가 안정화되고 예측 가능해진 시점에
클라우드에 머무는 것은 관성이지 전략이 아니다.

이 사례에서 저자의 인프라는 이미 수년간 안정적으로 운영되고 있었다. 192GB RAM 서버가
대부분의 시간을 활용도 낮은 상태로 있었을 가능성이 높다. 이런 상황에서 클라우드의
온디맨드 프리미엄을 지불하는 것은 보험료가 보험 대상보다 비싼 셈이다.

DHH가 2022년에 Basecamp의 클라우드 탈출을 발표한 이후, "Cloud Exit" 움직임이
하나의 흐름이 되었다. 이는 클라우드가 나쁘다는 것이 아니라, 모든 단계에서 최적의
선택이 아니라는 인식의 확산이다. 스타트업이 성장하면서 인프라 전략도 함께 성숙해야
한다.

### 마이그레이션 자동화는 인프라 기술 부채를 청산하는 기회다

저자는 CentOS 7에서 AlmaLinux 9.7로의 OS 업그레이드를 마이그레이션과 함께 수행했다.
이는 마이그레이션이 단순한 서버 이동이 아니라 누적된 기술 부채를 정리하는 기회가 된다는
것을 보여준다.

기존 서버에서 in-place 업그레이드를 시도하면 의존성 문제, 설정 충돌, 다운타임 등의
위험이 있다. 그러나 새 서버에 깨끗하게 설치하면 이런 문제가 자연스럽게 해결된다.
MySQL 5.7에서 8.0으로의 전환도 마찬가지다. 기존 서버에서 했다면 훨씬 더 복잡하고
위험했을 것이다.

마이그레이션을 인프라 현대화의 기회로 활용하는 것은 "어차피 움직여야 한다면 제대로
움직이자"는 전략이다. 이 접근은 마이그레이션의 비용을 정당화하는 추가적인 논거가 된다.

### 개발도상국 인프라 전략의 차별화

이 글의 배경에는 터키의 경제 상황이 있다. 자국 통화 가치가 하락하는 환경에서 달러
기반 클라우드 서비스를 사용하면, 실질 비용이 서비스 자체의 가격 변동과 무관하게
지속적으로 증가한다. 이는 터키만의 문제가 아니라 아르헨티나, 이집트, 나이지리아 등
많은 국가에 해당한다.

유럽에 위치한 Hetzner는 달러가 아닌 유로 기반이므로 환율 리스크가 완전히 사라지지는
않지만, DigitalOcean이나 AWS 대비 절대 비용이 월등히 낮아 환율 충격을 흡수할 여유가
생긴다. 월 $233과 $1,432의 차이는 환율이 2배로 변동해도 여전히 Hetzner가 유리하다는
뜻이다.

개발도상국의 기술 기업들이 글로벌 클라우드 서비스에 의존할수록 환율 변동에 취약해진다.
인프라 비용은 고정비이므로 매출이 줄어도 줄일 수 없다. 이런 맥락에서 전용 서버나
지역 클라우드로의 전환은 기술적 선택이 아니라 경영적 생존 전략이다.
