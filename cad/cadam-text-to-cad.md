# CADAM: 오픈소스 텍스트-투-CAD 웹 앱

<https://adam.new/cadam/>

<https://github.com/Adam-CAD/CADAM>

HN 토론: <https://news.ycombinator.com/item?id=48572553> (184점, 86개 댓글)

## 소개

CADAM(Computer-Aided Design AI Model)은 자연어 설명을 3D CAD 모델로 변환하는
오픈소스 웹 애플리케이션이다.
YC W25 배치 기업 Adam(adam.new)이 2025년 9월에 공개한 프로젝트로,
2026년 기준 GitHub 스타 4,379개, 포크 554개를 기록하고 있다.

사용자가 3D 모델을 자연어로 설명하거나 이미지를 업로드하면
LLM(Claude, GPT, Gemini 등)이 OpenSCAD 코드를 생성하고,
OpenSCAD WebAssembly가 브라우저 내에서 이를 3D 메시로 컴파일한다.
Three.js와 React Three Fiber가 실시간 3D 미리보기를 렌더링하며,
결과물은 `.STL`, `.SCAD`, `.DXF` 형식으로 내보낼 수 있다.
설치 없이 `adam.new/cadam`에서 바로 사용 가능하다.

핵심 차별점은 파라메트릭 제어(parametric controls)다.
LLM이 모델에서 조정 가능한 치수를 자동 식별하고 슬라이더로 노출해,
AI 재생성 없이 치수를 즉시 조정할 수 있다.
BOSL, BOSL2, MCAD 라이브러리를 내장해 기어, 나사산, 허니컴 패턴 같은
복잡한 파라메트릭 형상을 지원한다.

실제 메이커 커뮤니티 반응도 이 타겟팅을 확인해준다.
patja는 알루미늄 파이프 케이블 관통부의 그로밋 씰을 특정 치수와 TPU 소재 요구사항으로 설명하자
빠른 시간 내에 합리적인 결과물이 생성됐다고 보고한다.[^patja]
지정하지 않은 파라미터인 와이어 직경을 모델이 스스로 합리적으로 가정했다는 것이다.
이것이 CADAM이 타겟하는 사용 사례다.

벤치마크에 공개된 생성 결과물은 실용적인 규모다.
V8 엔진(22개 치수, 8개 색상), 9기통 성형 항공기 엔진(15개 치수, 6개 색상),
터보팬 제트 엔진, 축류 터빈 블리스크 같은 복잡한 어셈블리부터
비틀린 육각 화병, 널링 제어 손잡이, M12 육각 볼트와 너트,
NACA 2412 테이퍼드 날개 단면 같은 파라메트릭 부품까지 포함한다.

기술 스택은 React 19 + TypeScript + TanStack Start + Vite(프론트엔드),
Supabase PostgreSQL/Auth/Storage(백엔드), OpenSCAD WASM(CAD 엔진),
Anthropic Claude API + OpenRouter + OpenAI + Google API(AI)로 구성된다.
라이선스는 GPL v3이며 OpenSCAD 파생 코드의 라이선스 연쇄를 따른다.

## 분석

### OpenSCAD를 중간 표현으로 선택한 것은 LLM 친화적 설계다

텍스트-투-CAD의 핵심 설계 결정은 “LLM이 무엇을 생성하게 할 것인가”다.
CADAM은 독자적인 CAD 형식 대신 OpenSCAD의 선언적 코드를 선택했다.
OpenSCAD는 텍스트 기반 스크립트 언어로 3D 형상을 수학적으로 기술한다.
LLM은 이미 대량의 OpenSCAD 코드를 훈련 데이터로 보유하고 있어
복잡한 형상도 비교적 신뢰할 수 있는 코드를 생성할 수 있다.

이 선택은 파라메트릭 제어와도 자연스럽게 결합된다.
OpenSCAD 코드에서 숫자 상수는 모두 잠재적 파라미터다.
LLM이 변수로 선언한 값들을 파싱해 슬라이더로 매핑하는 것은
코드 구조상 직접적이다.
반면 B-rep(경계 표현) 기반 CAD 커널에서 파라미터를 추출하는 것은
훨씬 복잡한 엔지니어링 문제다.

OpenSCAD의 한계도 이 선택에 내재된다.
OpenSCAD는 CSG(Constructive Solid Geometry) 기반이라
필렛, 라운딩, 복잡한 곡면 형상이 어렵다.
FreeCAD, Fusion 360, SolidWorks 같은 전문 CAD 도구가 처리하는
고급 형상 연산의 상당 부분을 표현하기 어렵다.
벤치마크 모델들이 주로 기계 부품과 프린터블 형상에 집중된 것은
이 제약의 반영이다.

### WebAssembly로 CAD 엔진을 브라우저에 올리는 것은 배포 아키텍처를 단순화한다

OpenSCAD WASM을 브라우저에서 실행한다는 것은
CAD 컴파일 연산이 서버가 아닌 클라이언트에서 이루어진다는 의미다.
사용자의 모델 생성은 LLM API 호출 이후 서버 부하 없이 처리된다.
수천 명이 동시에 복잡한 3D 모델을 생성해도
렌더링 서버를 별도로 확장할 필요가 없다.

이 아키텍처는 스타트업의 운영 비용 구조와 직결된다.
비용이 발생하는 주요 지점은 LLM API 호출뿐이며,
이것은 사용량에 비례해 청구된다.
고정 서버 인프라 비용 없이 사용량 기반 과금 모델을 구현할 수 있다.

WASM으로 올릴 수 없는 무거운 CAD 커널이라면 이 아키텍처가 불가능했을 것이다.
OpenSCAD가 비교적 가볍고 WASM으로 컴파일 가능한 C++ 코드베이스를 가졌기 때문에
선택이 가능했다.
CADAM의 아키텍처는 OpenSCAD의 기술적 특성과 부트스트랩 스타트업의
경제적 제약이 교차한 지점에서 탄생했다.

### 벤치마크 공개는 AI 코드 생성 품질 평가의 새로운 방법론을 제안한다

CADAM의 `benchmarks/` 디렉토리는 단순한 데모가 아니다.
각 모델에 정확한 프롬프트, 파라미터 수, 색상 수, GIF 렌더링이 포함된
재현 가능한 테스트 케이스를 제공한다.
이것은 텍스트-투-CAD 시스템의 능력을 투명하게 평가하는 기준점이 된다.

프롬프트 엔지니어링과 LLM 모델 선택의 영향을 비교하려면
동일한 프롬프트에 대한 결과물을 비교할 수 있어야 한다.
CADAM의 벤치마크 형식은 이 비교를 가능하게 한다.
V8 엔진 같은 복잡한 어셈블리 벤치마크는 단순한 마케팅 자료가 아니라
다른 텍스트-투-CAD 시스템과 비교할 수 있는 도전 과제(challenge)다.

## 비평

### GPL v3 라이선스는 상업적 통합의 장벽을 만든다

CADAM이 GPL v3로 배포되는 것은 OpenSCAD 파생 코드의 라이선스 연쇄 때문이다.
GPL v3은 코드를 사용하는 파생 저작물도 GPL v3로 배포할 것을 요구한다.
CADAM을 자신의 독점 제품에 통합하려는 회사는
전체 제품을 GPL v3로 공개하거나 별도 라이선스 계약을 맺어야 한다.

이것이 오픈소스 커뮤니티에는 긍정적이지만,
스타트업이 기업 고객에게 판매하는 시나리오에서는 복잡한 문제를 만든다.
Fusion 360, SolidWorks, CATIA 같은 기존 CAD 도구를 쓰는 기업이
CADAM 기반 솔루션을 내부 시스템에 통합하려 할 때 법적 검토가 필요하다.
“오픈소스로 시작해 기업용 버전을 만든다”는 일반적인 오픈소스 스타트업 전략이
GPL 하에서는 구조적으로 더 어렵다.
Adam이 상업적 성공을 위해 커뮤니티 버전과 엔터프라이즈 버전을 어떻게 분리할지는
아직 명확하지 않다.

### OpenSCAD 기반 접근법은 전문 CAD 사용자의 기존 워크플로를 대체하기 어렵다

CADAM이 생성하는 `.SCAD` 파일은 OpenSCAD 고유 형식이다.
기계 엔지니어링에서 표준 교환 형식은 STEP, IGES, Parasolid이며,
전문 CAD 도구는 이 형식으로 작업한다.
`.STL`은 3D 프린팅용 단방향 출력 형식이고 편집이 불가능하며,
`.DXF`는 2D 도면 형식이다.
전문 엔지니어가 CADAM 결과물을 SolidWorks나 CATIA에서 계속 수정하려면
STEP로의 변환 단계가 필요하고, 이 과정에서 파라메트릭 정보가 손실된다.

이것은 CADAM이 타겟할 수 있는 시장을 제한한다.
전문 기계 엔지니어링 팀의 주요 도구 체인에 통합되기 어렵고,
취미 3D 프린터, 메이커, 프로토타이핑 단계의 스타트업에 더 적합하다.
“AI가 전문 CAD를 민주화한다”는 프레이밍은 타당하지만,
민주화의 대상이 전문 사용자가 아니라 CAD를 처음 접하는 사람들임을 인식해야 한다.

흥미롭게도 c7b가 CadQuery 대신 OpenSCAD를 선택한 이유를 묻자[^c7b]
zachdive는 CADAM의 주요 사용자가 “프린터블 STL 파일”이 필요한 메이커와 티커러라고 명확히 답한다.
전문 CAD 작업에는 adam.new의 별도 플러그인이 build123d를 통해 STEP 파일을 생성한다는 것이다.
이 답변은 CADAM이 의도적으로 전문가 시장을 타겟하지 않음을 확인한다.
OpenSCAD 선택은 기술적 최적해가 아니라 타겟 사용자에 맞춘 결정이었다.

### 프롬프트-투-모델 파이프라인의 신뢰성이 벤치마크에서 드러나지 않는다

공개된 벤치마크는 성공 사례만 보여준다.
V8 엔진이나 터보팬 제트 엔진의 화려한 GIF는 인상적이지만,
같은 프롬프트로 몇 번 시도했을 때 몇 퍼센트가 이 품질을 달성하는지는 알 수 없다.
LLM 기반 코드 생성은 확률적이며 동일한 프롬프트에서도 품질 편차가 크다.
벤치마크가 수십 번의 시도 중 선별된 최상의 결과라면
실제 사용 경험은 상당히 다를 수 있다.

더 중요한 문제는 OpenSCAD 컴파일 오류다.
LLM이 생성한 OpenSCAD 코드가 문법 오류나 논리 오류를 포함하면
컴파일이 실패하고 모델이 생성되지 않는다.
복잡한 어셈블리 프롬프트에서 이런 실패가 얼마나 자주 발생하는지,
오류 복구 메커니즘이 있는지는 README에서 다루지 않는다.
사용자 경험에서 이 실패율은 제품의 실용성을 결정하는 핵심 지표다.

murkt가 지적하는 LLM의 공간 인식(spatial awareness) 한계가 바로 이 문제의 근원이다.[^murkt]
LLM이 3D 공간에서 객체 간 위치 관계, 치수 비율, 접합 조건을 정확히 추론하는 능력은
텍스트-기반 훈련으로 얻기 어려운 능력이다.
jetter는 ModelRift의 OpenSCAD LLM 벤치마킹을 인용하며 “LLM은 여전히 공간 추론에 약하지만 개선 중”이라고 언급한다.[^jetter]
dvh는 실제 부품 번호(AMF-O97L45-DB)로 테스트한 결과를 공유하는데,
데이터시트를 정확히 찾았음에도 핀 피치가 틀리고 핀 위치가 잘못됐으며 핀이 누락됐다고 보고한다.[^dvh]
이것이 “메이커용 프린터블 부품”과 “정밀 공학 부품” 사이의 실용성 경계다.

## 인사이트

### 텍스트-투-CAD는 3D 프린팅 대중화의 마지막 병목을 해소한다

3D 프린팅 하드웨어는 Bambu Lab, Prusa 같은 회사들의 경쟁으로
$200-$500 범위에서 접근 가능해졌다.
소재 비용도 꾸준히 하락하고 있다.
그러나 3D 프린팅의 진짜 장벽은 하드웨어 비용이 아니라 모델 설계 능력이었다.
Thingiverse 같은 모델 라이브러리가 이 문제를 부분적으로 해결했지만,
“내가 필요한 정확한 형상”을 만들려면 여전히 CAD 기술이 필요했다.

텍스트-투-CAD가 이 병목을 제거하면 3D 프린팅의 활용 범위가 질적으로 달라진다.
“기성품 부품을 구매하거나 전문가에게 의뢰한다” 대신
“직접 자연어로 설명해 즉시 프린팅한다”는 패턴이 가능해진다.
가정, 차고, 소규모 제조업에서 맞춤 부품 생산의 문턱이 낮아지는 것은
단순한 편의성 개선이 아니라 제조 분산화의 가속이다.

이 변화의 속도는 LLM의 OpenSCAD 코드 생성 품질과 직결된다.
현재 CADAM이 보여주는 벤치마크 수준이 일관되게 달성 가능해지면,
3D 프린팅은 “취미” 도구에서 “개인 제조 도구”로 재정의될 것이다.

### OpenSCAD를 LLM 코드 생성의 타겟으로 선택하는 것은 프로그래머블 CAD의 부흥을 예고한다

전통적 CAD 패러다임은 GUI 기반 직접 조작이다.
점을 클릭하고, 면을 돌출하고, 치수를 입력하는 인터랙션이다.
OpenSCAD는 이것을 코드로 대체했지만 진입 장벽이 높아 틈새 도구에 머물렀다.

LLM이 OpenSCAD 코드를 생성할 수 있게 되면서 이 방정식이 바뀐다.
코드를 직접 작성하지 않아도 코드 기반 CAD의 파라메트릭 이점을 누릴 수 있다.
더 나아가, 생성된 `.SCAD` 파일은 프로그래머가 직접 편집하거나
버전 관리 시스템에서 추적할 수 있다.
CAD 모델이 코드가 되는 것은 소프트웨어 개발의 Infrastructure as Code와 유사한
“Design as Code” 패러다임의 가능성을 열어준다.

이 패러다임에서 3D 모델은 단순히 “파일”이 아니라
Git으로 버전 관리되고, CI/CD로 렌더링 검증이 자동화되고,
파라미터를 변경하는 PR로 협업하는 소프트웨어 아티팩트가 된다.
CADAM의 `benchmarks/render.sh` 스크립트가 이미 이 방향을 암시한다.

### LLM이 생성하는 중간 언어(intermediate language)로서 도메인 특화 DSL의 가치가 재부상한다

텍스트-투-CAD가 OpenSCAD를 중간 언어로 사용하듯,
AI 코드 생성이 성숙해지면서 다양한 도메인에서
“LLM이 생성하기 좋은 DSL”의 가치가 재조명될 것이다.
SQL이 데이터 조작의 인터페이스로 살아남은 것처럼,
수학적으로 명확하고 학습 데이터가 풍부한 DSL은
AI 시대에 오히려 더 중요해질 수 있다.

CADAM의 성공 여부는 OpenSCAD라는 DSL이 얼마나 잘 선택됐는가에 달려 있다.
OpenSCAD가 표현할 수 없는 형상(유기적 곡면, 복잡한 서피스 모델링)이
LLM의 CAD 생성 능력의 천장이 된다.
더 표현력 있는 CAD 중간 언어 — 예를 들어 CadQuery의 Python API, 
또는 새로운 AI 친화적 CAD DSL — 가 등장한다면
텍스트-투-CAD의 가능성이 한 단계 더 확장될 것이다.

이 관점에서 CADAM은 단순한 제품이 아니라
AI 기반 CAD가 어떤 중간 표현을 채택할 것인가라는 더 큰 질문에 대한
현재의 가장 설득력 있는 답안 중 하나다.

fkilaiwi는 이 질문을 더 근본적으로 던진다.[^fkilaiwi]
텍스트-투-코드-투-CAD가 장기적 전략인지,
아니면 월드 모델이 결국 비디오 대신 3D 공간, 포인트 클라우드, CAD 설계를 직접 생성하게 될지를 묻는 것이다.
이것은 코드 기반 중간 표현이 임시방편인지 지속 가능한 패러다임인지에 대한 질문이다.
코드 기반 접근의 장점은 파라메트릭 편집 가능성과 LLM 훈련 데이터의 풍부함이지만,
공간적 3D 표현을 언어 토큰으로 처리한다는 근본적 비효율이 있다.
월드 모델 기반 접근이 성숙한다면 OpenSCAD 같은 중간 언어는 과도기적 해법으로 판명될 수 있다.
