# Figma Skills — AI 에이전트를 위한 코드-디자인 워크플로 마크다운 명세

<https://www.figma.com/community/skills>

<https://www.figma.com/ko-kr/community/skills>

## 소개

Figma Skills는 AI 모델이 Figma 프리미티브(primitive)와 개념을 다루는 방법을 안내하는
마크다운 파일이다.
각 스킬은 컴포넌트 생성, 레이아웃 생성, 코드와 Figma 사이의 변수 동기화 같은 특정 작업과
컨텍스트를 정의한다.
슬래시 명령(`/skill-name`) 형태로 제공되며, 개발 환경에 설치하면 AI 에이전트가 해당
명세를 읽고 Figma 파일을 조작할 수 있다.

## 공식 스킬

Figma가 직접 제공하는 기본 스킬이다.

| 스킬                      | 설명                                                     |
| ------------------------- | -------------------------------------------------------- |
| `/figma-generate-design`  | 기존 컴포넌트와 변수를 사용해 Figma에 새 디자인 생성    |
| `/figma-generate-library` | 코드베이스에서 Figma 컴포넌트 생성                      |
| `/figma-use-figjam`       | FigJam 보드에 스티키, 섹션, 커넥터, 표 등 생성·수정    |
| `/figma-generate-diagram` | FigJam에서 아키텍처 다이어그램, ERD, 플로우차트 생성    |
| `/generate-project-plan`  | 코드베이스나 문서에서 FigJam 프로젝트 플랜 생성         |

## 분석

### 스킬의 정체: 프롬프트가 아닌 명세

Figma Skills의 핵심은 “AI에게 무엇을 시킬지”를 담은 자연어 프롬프트가 아니라,
“AI가 Figma를 어떻게 다루어야 하는지”를 정의한 구조화된 마크다운 명세라는 점이다.
각 스킬은 Figma 캔버스의 프리미티브(노드, 변수, 스타일, 컴포넌트)와 조작 패턴을
AI 에이전트가 이해할 수 있는 형태로 기술한다.
Claude Code 프로젝트의 `CLAUDE.md` 파일이나 Cursor의 `.cursorrules`와 같은 맥락이지만,
Figma라는 특정 도메인에 특화된 점이 다르다.

### 코드-디자인 양방향 브리지

기존 Figma 워크플로는 “디자이너가 디자인하고 개발자가 구현”하는 단방향이었다.
Figma Skills는 이 흐름을 양방향으로 만들려 한다.
`/figma-generate-library`는 코드베이스에서 Figma 컴포넌트를 만들고,
`/prototype-to-figma`는 코드 프로토타입을 Figma 프레임으로 역변환한다.
MCP 서버와 연동하면 AI가 Figma API를 직접 호출해 파일을 수정할 수 있다.

### 커뮤니티 기여 모델

Figma Skills는 공식 스킬과 커뮤니티 스킬을 함께 큐레이션한다.
Uber의 `/create-voice`처럼 접근성(a11y) 특화 스킬이 있는가 하면, 디자인 시스템
감사 도구인 `/audit-design-system`처럼 엔터프라이즈 워크플로에 특화된 스킬도 있다.
각 스킬은 GitHub 등 외부 소스로 연결되며, 개발 환경에 직접 설치·사용하는 방식이다.
이는 npm 패키지나 VSCode 확장처럼 생태계를 개방형으로 운영하겠다는 의도다.

## 비평

### 강점

마크다운으로 스킬을 정의하는 방식은 AI 에이전트 공급업체에 종속되지 않는다.
슬래시 명령 형태는 개발자 도구(Claude Code, Cursor 등)의 기존 관례와 자연스럽게 맞물린다.
커뮤니티 기여를 수용함으로써 Figma가 커버하지 못하는 도메인별 워크플로를 외부에서
채울 수 있다.

### 약점 및 한계

스킬 파일 자체의 품질이 결과를 좌우하는데, 각 스킬이 어떤 기준으로 작성되고
검증되는지 명확한 명세가 공개되어 있지 않다.
커뮤니티 스킬은 Figma가 직접 유지 관리하지 않으므로, Figma API 변경 시 호환성이
깨질 수 있다.
“MCP clients”가 커뮤니티 카테고리에 포함된 점은 MCP 서버 없이는 일부 스킬이
동작하지 않을 수 있음을 시사하는데, 이 의존성이 진입 장벽이 될 수 있다.

### 관점의 공백

스킬 마크다운의 실제 문법 명세나 스키마가 공개 문서화되어 있지 않다.
어떤 AI 에이전트에서 어떤 스킬이 동작하는지, 호환성 매트릭스도 제시되지 않는다.
`/audit-design-system` 같은 스킬이 감사 결과를 어떻게 반환하고, 개발자가 이를
어떻게 활용하는지에 대한 워크플로 예시가 부족하다.

## 인사이트

### 도구 지식의 마크다운 인코딩이 열어주는 것들

Figma Skills가 마크다운 파일이라는 사실은 단순한 포맷 선택이 아니다.
마크다운은 AI 언어 모델의 훈련 데이터에 가장 많이 등장하는 포맷 중 하나이고,
LLM이 마크다운 구조를 파싱하고 따르는 능력은 이미 충분히 검증되어 있다.
Figma는 이 사실을 이용해 “Figma를 어떻게 다루는지”를 LLM이 이해할 수 있는 형태로
직접 인코딩한다.

이 접근법은 소프트웨어 도구 통합의 새로운 레이어를 정의한다.
기존에는 SDK나 API 클라이언트를 통해 도구를 통합했고, 이를 사람이 코딩해야 했다.
Figma Skills는 “도구 사용 지식 자체를 마크다운으로 패키징”해 AI가 소비하게 한다.
이 패턴이 자리를 잡으면, 모든 SaaS 도구가 “AI 에이전트를 위한 스킬 파일”을 공식
문서처럼 제공하는 시대가 올 것이다.

### 디자이너 없는 Figma 워크플로의 등장

`/figma-generate-design`과 `/figma-generate-library`는 개발자가 디자이너 없이
Figma 파일을 생성하고 수정할 수 있는 가능성을 열어준다.
더 중요한 것은 역방향이다. `/prototype-to-figma`는 코드로 먼저 만들고 나중에
Figma로 문서화하는 워크플로를 지원한다.
이는 “디자인 먼저, 구현 나중”이라는 전통적 순서를 무너뜨린다.

이 변화는 디자이너와 개발자의 협업 방식 자체를 재정의할 수 있다.
디자인 파일이 명세의 출발점이 아니라 결과물 중 하나가 되는 것이다.
코드가 진실의 원천(source of truth)이 되고, Figma는 그 코드의 시각적 표현이 된다.
장기적으로 이는 Figma의 포지셔닝을 “디자인 도구”에서 “코드와 디자인의 동기화 도구”로
이동시킬 수 있다.

### 접근성 스킬이 보여주는 AI 자동화의 경제학

Uber의 `/create-voice`는 Figma 컴포넌트에서 VoiceOver, TalkBack, ARIA 스펙을
자동 생성한다.
이 작업은 기존에 접근성 전문가가 수작업으로 수행하던 것으로, 시간과 전문 지식이
모두 필요했다.
스킬 하나로 이 작업이 자동화된다면, 접근성 스펙 작성의 경제적 장벽이 크게 낮아진다.

이것은 AI 자동화가 고급 전문 작업에 어떻게 침투하는지를 보여주는 사례다.
AI는 처음에는 반복적 단순 작업을 대체한다고 알려졌지만, 실제로는 특정 전문 지식을
요구하는 고단가 작업도 빠르게 자동화한다.
접근성 감사, 디자인 시스템 감사(`/audit-design-system`), 스펙 생성은 모두 이 범주에
속한다.
Figma Skills 생태계가 성숙하면, 지금은 전문가에게 의뢰해야 하는 작업들이 슬래시
명령 하나로 처리되는 시대가 앞당겨질 것이다.
