# Agentic Testing: Where Agents Fit in the E2E Testing Stack

원문: <https://slack.engineering/agentic-testing-where-agents-fit-in-the-e2e-testing-stack/>

## 요약

Slack Engineering의 Staff Software Engineer Sergii Gorbachov가 2026년 6월에 발표한 이 글은,
AI 에이전트 기반 E2E 테스트가 기존 결정론적(deterministic) 테스트 스택 안에서 어디에 위치해야 하는지를 다룬다.
저자는 200회 이상의 자동화 실행 실험을 통해 에이전트 주도 테스트의 신뢰성, 속도, 비용을 측정하고
구체적인 데이터로 결론을 도출한다.

실험은 세 가지 실행 모델을 비교한다.
첫째, Agent + Playwright MCP는 에이전트가 구조화된 브라우저 프리미티브(primitive)를 통해 DOM 컨텍스트를 유지하며 상호작용한다.
둘째, Agent + Playwright CLI는 에이전트가 CLI 커맨드를 단계별로 실행하고 매 단계마다 UI 상태를 재판단한다.
셋째, Generated Playwright Tests는 AI가 자연어 설명을 바탕으로 결정론적 테스트 코드를 생성한다.
두 가지 테스트 시나리오로 검증했는데, Thread Reply(약 15~20 스텝)와 Search Discovery(약 25~30 스텝)다.

신뢰성 결과에서 Playwright MCP가 가장 안정적이었다.
Thread Reply에서 실패율 0%, Search Discovery에서 약 12%를 기록했다.
Playwright CLI는 각각 약 12%, 약 20%를 기록했다.
Generated Tests는 Thread Reply에서 약 8%로 양호했지만 Search Discovery에서 약 48%로 급락했으며,
실패 지점은 대부분 70~80% 진행 후였다.
실행 시간은 MCP가 5~8분, CLI가 9~11분, Generated Tests가 약 3분이었다.

비용 분석은 에이전트 테스트의 현실적 한계를 명확히 드러낸다.
실행당 비용이 $15~30에 달해 기존 테스트 대비 매우 높다.
토큰 소비는 모델 출력이 아니라 컨텍스트 누적 속도와 에이전트 턴(turn) 수에 의해 결정된다.
MCP는 약 40턴, CLI는 약 85턴, Code Gen은 약 70턴을 기록했다.
브라우저 스냅샷과 에러 트레이스가 비용의 주요 원인이었다.

저자는 에이전트 테스트가 결정론적 테스트를 대체하지 않는다고 결론 내린다.
대신 4계층 테스트 피라미드를 제안한다: Unit Tests → Integration Tests → E2E Testing → Agentic Testing.
에이전트는 복잡한 UI 동작 탐색, 불안정한(flaky) 워크플로 디버깅, 프로덕션 버그 재현,
엣지 케이스 검증에 탁월하다.
반면 빠르고 반복 가능한 CI 회귀(regression) 검사에는 여전히 결정론적 테스트가 우월하다.

## 분석

### “여정(journey)”과 “목표(goal)”의 구분이 함의하는 것

글의 핵심 명제는 단순하면서 날카롭다: “Tests enforce journeys. Agents verify goals.”
이 한 문장이 두 패러다임의 철학적 차이를 압축한다.
결정론적 테스트는 특정 행동 시퀀스를 강제하고 그 시퀀스 자체를 명세(spec)로 삼는다.
에이전트는 목표 상태를 받고 그것에 도달하는 경로를 스스로 선택한다.

이 구분은 테스트 도구의 차이에 그치지 않는다.
결정론적 테스트가 암묵적으로 가정하는 것은 “사용자는 우리가 설계한 경로를 따른다”는 것이다.
실제 사용자는 그렇지 않다.
에이전트가 동일한 목표에 대해 약 20%만 동일한 행동 시퀀스를 따른다는 실험 데이터는
UI 테스트의 오랜 허구를 폭로한다.
즉, 우리가 “사용자 시나리오 테스트”라고 불러온 것의 상당 부분은 사실 “개발자가 설계한 경로 테스트”였다.

이 발견은 테스트 전략의 가치 판단을 재정렬한다.
결정론적 테스트는 “이 경로가 깨지지 않았는가”를 보장하고,
에이전트 테스트는 “이 기능이 목표를 달성할 수 있는가”를 검증한다.
두 질문은 다르다. 둘 다 필요하다.

### 비용 구조가 드러내는 컨텍스트 관리의 병목

저자가 강조하는 비용 분석은 단순한 요금 문제를 넘는다.
“비용은 모델 출력이 아니라 컨텍스트 누적과 턴 수에 의해 결정된다”는 관찰은
현재 LLM 에이전트 아키텍처의 근본적 제약을 드러낸다.
Playwright MCP가 CLI보다 훨씬 효율적인 이유는 모델이 더 저렴해서가 아니라
구조화된 브라우저 프리미티브가 컨텍스트를 더 압축적으로 표현하기 때문이다.

이 관찰은 에이전트 테스트 최적화의 방향이 어디를 향해야 하는지를 알려준다.
더 싼 모델보다 더 효율적인 도구 설계가 먼저다.
Haiku 4.5가 Opus 4.6보다 더 많은 토큰(5.7M vs 3.8M)을 소비한 것은 이를 반증한다.
능력이 낮은 모델이 더 많은 턴을 필요로 하면서 오히려 더 비싸진다.

이 구조는 프롬프트 캐싱, 컨텍스트 컴팩션, 스냅샷 빈도 조정이 비용 절감의 핵심 레버임을 시사한다.
저자도 이를 언급하지만 실험에서는 미구현 상태로 남겼다.
즉, 현재 실험 결과($15~30/회)는 최적화 이전 베이스라인에 가깝다.

### 인프라 선택이 모델 선택만큼 중요하다

실험 결과에서 눈에 띄는 것은 모델 차이보다 실행 환경 차이가 신뢰성에 더 큰 영향을 미쳤다는 점이다.
Playwright MCP는 “structured browser primitives and tighter integration with the agent's tool-calling workflow”를 제공해
인증(authentication)과 내비게이션 실패를 CLI 기반 실행보다 현저히 줄였다.
모델을 Opus에서 Sonnet으로 낮춰도 MCP 환경에서는 안정성이 크게 다르지 않았다.

이는 에이전트 시스템 설계에서 흔히 간과되는 원칙을 실증한다.
“어떤 모델을 쓸 것인가”보다 “에이전트에게 어떤 도구 인터페이스를 줄 것인가”가 더 근본적이다.
구조화된 도구 인터페이스는 에이전트의 추론 부담을 줄이고 오류 복구 경로를 명확하게 한다.
비구조화된 CLI 환경은 에이전트가 매 단계마다 현재 상태를 재해석해야 해 턴이 늘고 실패율도 높아진다.

### 4계층 피라미드가 전제하는 조직적 맥락

저자가 제안하는 4계층 테스트 피라미드는 기술적 제안인 동시에 조직적 제안이다.
에이전트 테스트를 “탐색과 디버깅”에 할당하는 것은 그 비용과 비결정론성을 인정한 현실적 타협이다.
하지만 이 모델이 작동하려면 조직이 에이전트 테스트 실행을 언제, 누가, 어떤 기준으로 트리거할지
별도로 설계해야 한다.
“CI에는 결정론적, 탐색에는 에이전트”라는 분리가 명확해 보이지만,
실제 개발 사이클에서 이 경계를 운영하는 것은 별도의 프로세스 설계를 요구한다.

## 비평

### 실험이 답하려는 질문과 실제로 답한 질문이 다르다

이 글은 “에이전트가 E2E 테스트 스택 어디에 맞는가”를 묻는다.
그러나 실험이 실제로 측정한 것은 Slack의 특정 워크플로 두 가지에 대한
특정 도구 조합의 성능이다.
Thread Reply와 Search Discovery는 Slack 특유의 복잡성을 반영하지만,
이 두 시나리오에서 도출한 결론이 다른 제품, 다른 도메인에도 성립한다는 근거는 제시되지 않는다.

더 근본적인 문제는 실험 설계의 범위가 “단일 세션 워크플로”로 명시적으로 제한된다는 점이다.
저자 스스로 크로스 워크스페이스(cross-workspace) 시나리오나 멀티 윈도우 시나리오를 미래 과제로 남겼다.
그런데 실제 SaaS 제품에서 가장 복잡하고 회귀가 잦은 시나리오는 정확히 그 영역에 있다.
즉, 이 실험이 커버하는 영역은 에이전트 테스트가 가장 빛날 수 있는 영역이 아닐 가능성이 높다.

결론인 “에이전트 테스트는 탐색과 디버깅에 적합하다”는 이 실험의 범위 안에서는 타당하다.
그러나 그 결론을 일반화할 근거로서의 실험 설계는 충분하지 않다.
200회 실행은 인상적인 숫자처럼 보이지만, 두 시나리오에 걸쳐 분산되면
각 변수 조합당 표본이 얼마 되지 않는다.

### 비용 분석이 전략적 결론으로 비약한다

저자는 $15~30/실행이라는 비용 데이터를 근거로 에이전트 테스트의 CI 적용이 비현실적이라고 결론 내린다.
이 결론은 현재 시점에서는 타당하지만, 전략적 판단으로서는 과도하게 단정적이다.
저자 스스로 프롬프트 캐싱, 컨텍스트 컴팩션, 스냅샷 최적화가 아직 적용되지 않았다고 밝혔다.
이 최적화들이 적용되면 비용이 얼마나 줄어들 수 있는지 추정조차 제시하지 않는다.

또한 비용 비교 기준이 불균형하다.
결정론적 테스트의 비용을 “최소”로 묘사하지만,
대규모 CI 파이프라인에서 수천 개의 E2E 테스트를 반복 실행하는 실제 인프라 비용,
유지보수에 소요되는 엔지니어링 시간, 불안정(flakiness)으로 인한 재실행 비용은 계상되지 않는다.
에이전트 테스트가 결정론적 테스트의 유지보수 부담을 줄인다면,
총비용(total cost of ownership) 비교는 다른 결론을 낼 수 있다.

이 비교 없이 “비용이 높으니 CI에 부적합하다”는 결론은 근거가 절반뿐이다.

### “보완적 계층”이라는 결론은 결정을 유예한다

글은 에이전트 테스트가 결정론적 테스트의 대체가 아닌 보완이라고 결론 내린다.
이 결론은 맞지만, 실용적 지침으로서는 거의 아무것도 결정하지 않는다.
“언제 에이전트 테스트를 트리거할 것인가”, “어떤 시나리오를 에이전트 테스트로 커버하고
어떤 시나리오를 결정론적으로 커버할 것인가”, “에이전트 테스트 실패를 CI 실패로 처리할 것인가”
같은 운영 질문에 대한 답이 없다.

“탐색, 디버깅, 엣지 케이스”라는 사용 사례 목록은
어떤 팀도 지금 당장 파이프라인에 적용할 수 없는 수준의 추상이다.
논문이나 연구 보고서라면 충분할 수 있다.
그러나 Slack Engineering 블로그의 실무 독자들이 기대하는 것은
“우리 팀에서 내일부터 이렇게 적용할 수 있다”는 구체성이다.
그 구체성이 이 글에는 없다.

### 실험이 측정하지 못한 것: 불안정성(flakiness)의 역설

에이전트 테스트의 핵심 매력으로 제시되는 것 중 하나가 “결정론적 테스트의 flakiness 해결”이다.
그런데 이 실험에서 에이전트 테스트 자체의 재실행 신뢰성은 어떠한가?
동일한 프롬프트, 동일한 상태에서 에이전트가 매번 같은 결론에 도달하는가?
이 질문에 대한 데이터가 없다.

실험은 여러 모델과 실행 환경을 교차 비교하지만,
동일 조건에서의 반복 실행 변동성(run-to-run variance)은 명시적으로 분석하지 않는다.
에이전트 테스트가 LLM 특유의 비결정론성을 내재한다면,
“flakiness를 해결한다”는 주장은 한 종류의 불안정성을 다른 종류로 교체하는 것일 수 있다.
이 가능성을 실험 설계가 배제하지 않는다.

## 인사이트

### 에이전트가 “다른 경로”를 선택한다는 것은 테스트 커버리지 개념을 바꾼다

실험에서 80%의 실행이 동일 목표에 대해 서로 다른 행동 시퀀스를 따랐다.
이 숫자는 단순한 적응성(adaptability) 지표가 아니다.
소프트웨어 테스트에서 우리가 “커버리지”라고 부르던 개념을 근본적으로 재정의한다.

결정론적 테스트의 커버리지는 코드 경로(code path)나 UI 상태 전이의 특정 시퀀스를 커버한다.
에이전트 테스트의 커버리지는 목표 상태 도달 가능성과 그 경로 공간(path space)의 샘플링을 커버한다.
이 두 커버리지는 측정 방법도, 의미도, 보장하는 것도 다르다.
실무에서 이것은 “에이전트 테스트를 몇 회 실행해야 충분한가”라는 새로운 질문을 낳는다.
한 번의 성공은 “이 경로에서 성공했다”를 증명할 뿐, “모든 현실적 경로에서 성공한다”를 보장하지 않는다.

더 나아가, 에이전트가 개발자가 예상하지 못한 경로를 발견하면
그 경로가 지원되어야 하는 것인지, 막혀야 하는 것인지를 제품이 명시해야 한다.
에이전트 테스트는 제품 명세(specification)의 불완전성을 수면 위로 끌어올리는 도구가 된다.

### 비용 구조의 진짜 의미는 “누가 에이전트 테스트를 소유하는가”를 바꾼다

$15~30/실행이라는 비용은 단순히 “비싸다”는 신호가 아니다.
이 비용 수준은 에이전트 테스트가 개발자 개인이 로컬에서 반복 실행하는 도구가 아니라
팀 차원에서 전략적으로 트리거되는 자원임을 의미한다.

결정론적 E2E 테스트는 코스트가 낮아 “커밋마다 실행”할 수 있다.
에이전트 테스트는 “특정 마일스톤, 특정 기능 출시, 특정 회귀 조사”에만 실행된다.
이 차이는 도구 선택의 문제가 아니라 테스트 거버넌스(governance)의 문제다.
누가 언제 에이전트 테스트를 실행할지 결정하고, 그 비용을 어디에 배분할지가
팀 프로세스 설계의 일부가 된다.

역사적 유사점이 있다.
부하 테스트(load testing)와 카오스 엔지니어링(chaos engineering)은 비용 때문에
“항상 실행”이 아니라 “전략적 시점에 실행”으로 운영된다.
에이전트 테스트는 같은 범주로 수렴할 가능성이 높다.
그렇다면 그 조직적 인프라를 어떻게 설계할 것인지가 기술적 선택보다 앞선 질문이다.

### 에이전트 테스트의 진짜 가치는 테스트가 아닌 명세 피드백에 있다

이 글이 강조하지 않지만 데이터에서 읽히는 더 중요한 신호가 있다.
에이전트가 “다른 경로”를 시도하고 그 중 일부가 실패할 때,
그 실패는 버그 신호인 동시에 제품 명세의 공백 신호다.
에이전트가 사용자 관점에서 자연스럽게 시도하는 경로를 제품이 지원하지 않는다면,
그것은 결정론적 테스트가 발견하기 어려운 유형의 문제다.

에이전트 테스트는 이런 의미에서 “테스트 자동화”보다 “탐색적 테스트(exploratory testing)의 자동화”에 가깝다.
전통적으로 QA 엔지니어가 수행하던 탐색적 테스트를 에이전트가 대규모로 수행할 수 있다면,
그 아웃풋은 버그 리포트만이 아니라 UX 명세의 허점 목록이 된다.
이 피드백 루프를 어떻게 제품 개발 사이클에 통합할 것인지가
에이전트 테스트 도입의 핵심 설계 문제다.

### 도구 인터페이스 설계가 AI 시대의 새로운 테스트 인프라 경쟁력이다

이 실험에서 모델보다 실행 환경(MCP vs CLI)이 신뢰성에 더 큰 영향을 미쳤다.
이 발견은 앞으로의 테스트 인프라 투자 방향에 강한 시사를 준다.
더 좋은 모델을 기다리는 것보다 에이전트에게 더 나은 도구 인터페이스를 설계하는 것이
현재 시점에서 더 높은 ROI를 낼 수 있다.

구조화된 브라우저 프리미티브가 에이전트의 추론 부담을 줄이듯,
에이전트 친화적인 도구 설계 원칙이 테스트 인프라 전반에 적용될 것이다.
로그 형식, 에러 메시지 구조, 상태 스냅샷 설계가 “사람이 읽기 쉬운” 것에서
“에이전트가 추론하기 쉬운” 것으로 최적화 기준이 이동할 수 있다.
이것은 테스트 도구만의 문제가 아니라 소프트웨어 관측가능성(observability) 전반의 재설계 방향이다.
