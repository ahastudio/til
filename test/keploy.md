# Keploy — 트래픽 기반 자동 API 테스트 생성기

- 원문: <https://github.com/keploy/keploy>

## 요약

실제 네트워크 트래픽을 기록해 API 테스트와 목(Mock)을 자동으로 생성하는
오픈소스 도구다. 개발자가 테스트를 직접 작성하지 않아도 프로덕션 트래픽에서
테스트 케이스를 뽑아낸다.

Go로 작성됐고, eBPF를 활용해 언어·프레임워크에 무관하게 트래픽을 가로챈다.

## 동작 원리

```text
실제 API 호출 → Keploy가 요청/응답 기록
→ 테스트 케이스 + 의존성 목 자동 생성
→ CI에서 재실행해 회귀 감지
```

외부 DB, 서드파티 API 호출도 기록해 목으로 대체하므로
테스트 실행 환경이 독립적이다.

## 인사이트

### 1) "테스트를 쓰는 비용"이 아니라 "트래픽이 곧 테스트"로 전환

테스트 커버리지가 낮은 이유는 의지의 문제가 아니라 비용의 문제다.
특히 복잡한 API 호출 시나리오를 테스트로 표현하는 것은 원래 코드를
두 번 짜는 것과 같다.

Keploy는 이미 존재하는 트래픽을 테스트로 변환해 그 비용을 거의 제로로
만든다. 베이스라인 커버리지 확보에 특히 효과적이다.

### 2) eBPF 기반 캡처는 언어 무관 통합의 현실적 해법이다

에이전트를 앱에 삽입하거나 SDK를 추가하는 방식은 레거시 코드베이스에서
현실적이지 않다. eBPF로 커널 레벨에서 트래픽을 가로채면 코드 변경 없이
어떤 언어, 어떤 프레임워크에도 붙일 수 있다.

관찰 가능성(Observability) 도구들이 같은 방향으로 진화하고 있다.

### 3) 기록된 테스트는 문서이자 계약이다

자동 생성된 테스트는 "이 API는 이렇게 동작했다"는 스냅샷이다.
API 계약이 암묵적으로 명문화되는 효과가 있다.

변경 후 테스트가 깨지면 "버그"가 아니라 "계약 변경"임을 알려주므로,
의도치 않은 동작 변화를 빠르게 감지할 수 있다.

### 4) 한계: 기록 기반 테스트는 엣지 케이스를 못 잡는다

실제 트래픽에 없는 시나리오—잘못된 입력, 경계값, 오류 경로—는 자동
생성되지 않는다. Keploy는 베이스라인을 채워주지만, 의도적인 엣지 케이스
테스트를 대체하지는 못한다.

도구의 역할을 정확히 이해하고 조합해야 한다.
