## 요약

발표는 “AI가 쓴 코드 누가 검증하나”라는 부제를 달고,
발표자가 속한 네이버 파이낸셜 금융 FE 조직의 실전 경험을 정리한다.
구성은 세 부분이다.
첫째 왜 E2E 테스트가 필요한가,
둘째 테스트를 어떻게 작성·운영하는가,
셋째 그래서 사람은 무엇을 하는가.
아래 요약도 이 3부 구조를 그대로 따른다.

### 문제 제기: 코딩은 빨라졌지만 검증이 병목이 되었다

발표자는 자신이 겪은 실패 사례로 시작한다.
에이전트가 Next.js 버전 업그레이드를 “문제없이 완료”했다고 보고하고
PR까지 올렸으나,
실제로 실행해 보니 특정 페이지에서 뒤로 가기 버튼을 눌렀을 때
기존 동작이 깨져 있었다.
프레임워크 내부 구현이 바뀌며 코드베이스의 특정 기능이 영향을 받았는데,
에이전트는 이를 확인하지 않고 끝났다고 보고한 것이다.
발표자는 “모델은 충분히 똑똑한데 왜 이런 실수를 하는가”라고 묻고,
원인을 모델의 지능이 아니라
“검증할 수단이 처음부터 주어지지 않았다”는 환경의 부재에서 찾는다.

여기서 하네스 엔지니어링(harness engineering)이 화두로 등장한다.
용어를 명명한 미첼 하시모토(Mitchell Hashimoto)의 정의를 빌려,
하네스를 “에이전트가 실수할 때마다 그 실수가 다시는 반복되지 않도록 하는 것”,
구체적으로는 “모델 가중치 외부의 모든 것”으로 설명한다.
`AGENTS.md` 같은 지침 파일, MCP·Skills 같은 도구,
도메인 지식을 인코딩한 LLM 위키, 린트·자동화 테스트 같은 검증 도구가
모두 하네스이며, 그중에서도 이번 발표는 E2E 테스트를 집중적으로 다룬다.

문제의 규모는 PR 통계로 제시된다.
발표자가 주로 작업하는 코드베이스에서 개발자 한 명이 머지한 PR은
AI 에이전트 도구 본격 도입 전 9개월간 227건이었으나,
도입 후 9개월간 348건으로 약 1.5배 늘었다.
그러나 PR이 늘었다고 검증도 그만큼 잘 되는 것은 아니다.
모든 PR을 사람이 로컬에 체크아웃해 직접 돌려보고 리뷰하는 것은
현실적으로 불가능하기 때문이다.
발표자는 이를 “코딩은 빨라졌지만 검증이 병목이 되었다”고 요약한다.
Claude Code를 만든 보리스 체르니(Boris Cherny)가 트위터에서 한
“Claude에게 자기 작업을 검증할 수단을 줘라,
이건 결과를 두세 배 끌어내는 방법이었다”는 말을 인용하며,
사람도 모델도 스스로 짠 코드에는 관대하다는 점을 짚는다.
그래서 팀의 결론은 “모델에게 작업을 검증할 수단을 제공하자”였다.

검증 수단의 후보로 처음 시도한 것은
에이전트가 브라우저를 직접 조작하는 방식이었다(Vercel agent-browser,
Chrome 확장, Playwright MCP).
수동 테스트 부담은 줄었지만,
코드를 수정할 때마다 매번 에이전트에게 시나리오를 시켜야 했다.
시나리오를 마크다운으로 저장해 재생할 수는 있어도,
실행이 느리고 토큰을 많이 쓰며 무엇보다 비결정적이었다.
그래서 매번 해석해 조작하는 비결정적 방식이 아니라
결정론적 테스트 코드가 필요하다는 결론에 이른다.
기존에 Vitest·React Testing Library 기반 단위 테스트는 이미 있었지만
그것만으로는 부족했다.
미들웨어·`getServerSideProps`·컴포넌트의 단위 테스트가 각각 모두 통과해도,
실제 브라우저에서는 데이터 형식이 어긋나고 상태가 넘어오지 않고
특정 환경에서 API 호출이 실패했다.
AI 작업물 오류의 대부분이 이런 “컴포넌트 경계 결함”에서 발생했다.
켄트 C. 도즈(Kent C. Dodds)가 2018년 제시한
테스팅 트로피(static-unit-integration-e2e)를 들어,
아래로 갈수록 비용은 싸지만 확신은 적고
위로 갈수록 비용은 들지만 확신이 크다고 설명하며,
자기 팀에는 맨 위 E2E 자리만 비어 있었다고 진단한다.
이전에는 시간이 들고 깨지기 쉬워 QA 팀에 미뤄 왔던 자리지만,
에이전트 시대에 작성 부담은 줄고 시스템 검증의 중요도는 올라가면서
비용을 감수하고라도 채워야 할 자리가 되었다.
마틴 파울러(Martin Fowler)의
“E2E 테스트가 소프트웨어 동작에 가장 큰 확신을 준다”를 근거로,
팀은 Playwright로 E2E 테스트를 작성하기로 한다.

Playwright를 택한 이유로는 세 가지 장점이 제시된다.
첫째, 실제 브라우저·실제 사용자와 같은 레벨에서 검증한다.
단위 테스트가 Node.js·jsdom 환경에서 함수의 인자와 반환값을 검증한다면,
Playwright는 버튼이 실제로 보이고 클릭되며 다음 화면으로 넘어가는
사용자 경험을 검증한다.
둘째, 그 결과 테스트 코드가 곧 사양 문서가 된다.
“론 페이지에 들어가 한도 조회 버튼을 누르면 한도 조회 결과가 보인다”는 코드는,
다르게 읽으면 그대로 사용자 명세가 된다(“읽으면 명세, 돌리면 검증”의 두 얼굴).
실제 코드와 사양 문서를 따로 관리하며 무엇이 최신인지 혼란스럽던 문제가,
명세가 살아 있는 코드와 한 몸이 되며 해소된다.
셋째, 풍부한 리포트가 에이전트에게 먹일 양질의 데이터가 된다.
Playwright 실행 결과에는 시간순 스크린샷, 실행 비디오, 테스트 액션,
액션 전후 DOM 스냅샷, 테스트 인스펙터, 콘솔 로그,
네트워크 요청·응답이 구조화된 파일로 남는다.
이 데이터를 에이전트에게 주고
“테스트가 실패했으니 원인을 짚어 고쳐라”라고 하면,
에이전트는 사람 개발자가 브라우저 개발자 도구로 디버깅하듯
트레이스를 분석하고 수정안까지 내놓는다.

### 무엇을, 어떻게 짜고 운영하는가

대상 코드베이스는 6년차·17만 줄·220개 페이지 규모의
Next.js 엔터프라이즈 서비스로,
작년 가장 바빴던 달에는 PR이 234건 머지될 만큼
여러 명이 빠르게 작업하는 환경이었다.
이런 상황에서 모든 것을 테스트하는 것은 비현실적이었다.
팀이 원한 것은 “잘못된 수정으로 기존 동작이 깨지는 것을 막는” 것이지,
“코드를 바꿀 때마다 테스트가 깨져 고쳐야 하는” 것이 아니었다.
그래서 모든 기능·분기를 덮는 대신
“가장 중요한 사용자 워크플로는 무엇인가”부터 물었다.
실패하면 매출이 막히거나 데이터가 날아가거나 신뢰가 깨지는 부분,
즉 크리티컬 유저 플로우(critical user flow)에 집중한 것이다.
네이버페이 대출 서비스 기준으로는
홈 진입 → 약관 동의 → 휴대폰 인증·본인 확인 → 결과 조회로 이어지는 플로우,
신청 가능한 상품 목록·상세를 확인하고 신청하는 플로우 등이 핵심이다.
이렇게 추린 흐름으로 지난 5개월간 37개 라우트에 자동화 테스트를 구축했고,
안정 브랜치 기준 최근 CI에서 29회 연속 성공·플래키 없이 운영 중이다.

작성 방법은 “직접 짜지 않는다”를 첫 목표로 삼았다.
팀에 Playwright 경험자가 거의 없고
테스트 작성 자체가 낯선 팀원도 있었기 때문이다.
먼저 Playwright의 공식 도구 Codegen으로 브라우저 조작을 녹화해
코드를 생성한 뒤,
러프한 결과물을 에이전트가 기존 컨벤션에 맞춰 구조화하게 했다(테스트 스텝
분리, 인증 등 초기화 로직 추가, 타입·린트 검증, 실제 동작 확인까지).
더 나아가 Playwright가 공식 제공하는 에이전트 시스템
(`npx playwright init-agents --loop <도구>`)을 도입했다.
이는 플래너(Planner)·제너레이터(Generator)·힐러(Healer)
세 에이전트로 구성된다.
플래너는 코드베이스와 화면을 탐색해
마크다운 테스트 계획서를 만들어 사람이 검토하게 하고,
제너레이터는 그 계획서를 Playwright 코드로 옮기며
각 단계를 실제 브라우저에서 실행해 동작을 확인하고,
힐러는 실패한 테스트를 다시 실행해 원인을 분석하고
통과할 때까지 수정·재실행을 반복한다.
계획과 구현을 굳이 분리한 이유는,
한 LLM에게 한꺼번에 다 시키면 완성도가 떨어지므로
각자 한 가지에만 집중시키기 위해서다.
세 에이전트는 독립적으로도, 순차적으로도,
루프 안에서 계속 도는 형태로도 쓸 수 있다.

그러나 공식 에이전트만 설치한다고 바로 잘 되지는 않았다.
처음에는 인증 셋업을 건너뛰고 테스트를 시작하거나,
익숙한 API 에러의 해결법을 못 찾아 헤매거나,
테스트 계정 사용법을 몰라 매번 사람이 알려 줘야 했다.
에이전트에게 코드베이스의 테스트 전략·도메인 지식·프로젝트 구조에 대한
맥락이 없기 때문이다.
그래서 신입 개발자를 온보딩하듯 “우리만 아는 맥락”을 직접 적어 줘야 했다.
예컨대 “인증은 `getAuthState` 함수를 쓰고,
디테일 코드 90은 인증 만료이니 셋업을 재실행하라,
베이스 URL은 환경 변수와 페이지에 따라 결정하라” 같은 것들이다.
팀이 구축한 하네스는 크게 네 가지다.
(1) `AGENTS.md` — 서비스가 무엇이고 어떻게 구성되며
핵심 플로우·인증·접근 제어가 어떤지 담은 전체 안내 문서,
(2) E2E 테스트 운영·작성 규칙의 단일 출처 문서(SSOT) — 실행 환경,
인증 시나리오, 무엇을 테스트하고 안 할지에 대한 과거 의사결정,
(3) 인증·상태 초기화·데이터 세팅처럼 반복되는 유틸·헬퍼,
(4) MCP 서버 — 기획서·디자인·이슈·PR 같은 사내 컨텍스트에
에이전트가 직접 접근하도록 연결.
이 환경을 깔고 나니 테스트 작성은 몇 줄의 명령만으로 가능해졌다.
플래너에게 “Next.js 내비게이션 깨짐 회귀를 막을 테스트 계획서를 써 달라”고
던지면 관련 코드와 이슈를 MCP로 살펴보고 계획서를 만들고,
검토를 끝낸 계획서를 제너레이터에게 넘기면
Playwright CLI 스킬로 코드를 생성하고 린트·타입·실행까지 확인한다.
결과 코드는 스토리지 스테이트 픽스처로 인증을 분리하고,
테스트 스텝으로 단계를 나누고,
한글 스텝 명을 쓰는 기존 컨벤션을 그대로 지켰다.

운영에서 다른 팀도 참고할 만한 결정 세 가지가 공유된다.
첫째, 분기마다 공통 앞부분(약관 동의·휴대폰 인증)을 반복하면
실행 시간이 누적되고 앞부분이 바뀔 때 모든 테스트가 동시에 실패한다.
Playwright 공식 문서의 “각 테스트는 독립적이어야 한다”를 따라,
테스트용 API로 “이 사용자는 인증까지 끝나 다음 갈래로 진입 가능한 상태”를
미리 만들어 두고 그 갈래부터 검증한다.
둘째, 네이버 인증서·홈택스·전자증명서·네이버 앱 연동 같은 외부 의존성을
E2E에서 그대로 호출하면
느려질 뿐 아니라 외부의 작은 변경에도 테스트가 깨진다.
“통제할 수 있는 것만 테스트하라”는 권고에 따라,
브라우저에서 호출하는 것은 Playwright의 page.route로,
SSR·BFF처럼 서버에서 처리하는 것은
E2E 전용 환경 변수로 고정 응답을 반환하도록 모킹했다.
셋째, 간헐적으로 성공·실패하는 플래키(flaky)는
한 번 발견되면 테스트 결과 전체의 신뢰를 흔들므로
“머지 전 작성 단계에서 잡는다”를 원칙으로 삼았다.
구체적으로 `waitForTimeout` 같은 하드 대기 대신 조건 기반 대기를 쓰고,
CSS 클래스명·DOM 구조 대신 role·name 같은 시맨틱 셀렉터를 쓰고,
작성 직후 같은 테스트를 여러 번 반복 실행(`--repeat-each`, 흔히 번인)해
한 번이라도 실패하면 플래키로 본다.
머지 후 추적하는 비용보다 작성 직후 잡는 비용이 압도적으로 싸기 때문이다.

실패한 테스트의 수정도 에이전트에게 맡긴다.
Playwright 테스트를 실행하면
동작·화면·네트워크·콘솔·에러 스택을 통째로 기록한 트레이스 파일이 남는데,
최근 Playwright 팀이 내놓은 Playwright CLI·Playwright Trace 스킬을 통해
힐러 에이전트가 이 트레이스 집 안을 CLI로 탐색한다.
개발자가 개발자 도구로 콘솔·네트워크를 들여다보듯,
에이전트도 필요한 시점에 필요한 정보를 보며 진단하고,
수정 후 다시 돌려 통과할 때까지 작업한다.
이 흐름은 CI까지 닫힌다.
에이전트가 PR을 올리면 CI에서 E2E를 실행해 검증하고,
실패하면 머지를 차단하며 결과를 코멘트로 남기고
트레이스를 GitHub Actions 아티팩트로 업로드한다.
팀은 CI에서 실패한 PR이나 실행 링크를 주면
트레이스를 로컬에 내려받아 디버깅까지 한 번에 엮어 주는
`e2e-debug` 스킬을 만들었고,
힐러 에이전트가 원인을 분석·수정해 다시 PR에 반영한다.
마치 개발자가 PR을 올리고 CI가 실패하면 로그를 보고 고치는 것과 같은
사이클이다.

### 자가 개선 루프, 그리고 사람의 일

발표는 이 모든 것을 자가 개선 루프(self-improving loop)라는 개념으로 묶는다.
에이전트가 자기 작업을 검증할 수단(센서)과 기존 기능에 대한 가이드,
이 두 가지로 피드백을 받으며 스스로 작업을 개선해 가는 구조다.
루프 안에서는 한 번에 작업을 끝내지 못해도 괜찮다.
틀리면 스스로 고쳐 다시 돌리면 되기 때문이다.
그래서 “자가 개선 루프를 만들 수 있느냐”가
에이전트에게 일을 어디까지 맡길 수 있는지를 가르는 기준이 된다.
루프가 잘 돌려면 두 부품이 필요하다.
행동 전에 방향을 잡아 주는 가이드(`AGENTS.md`, MCP, 스킬)와
행동 후 관찰로 자가 교정을 돕는 센서(테스터, 린터, 타입 체커)다.
그렇다면 Playwright E2E는 어디에 들어가는가.
답은 둘 다다.
실행하면 정상 동작을 검증하는 센서이고,
에이전트가 읽으면 “버튼을 누르면 결과가 보인다”는 명세를 설명하는 가이드다.
한 벌만 짜면 가이드와 센서가 동시에 해결된다.
앤트로픽(Anthropic)이 “Building Effective Agents”에서 제시한
평가자-최적화자(evaluator-optimizer) 패턴
— 답을 만드는 LLM과 채점·피드백하는 LLM을 분리해
피드백을 받아 다시 도는 구조 — 과 닮았지만,
채점자가 LLM이면 비싸고 비결정적이며 느리다.
그래서 그 자리에 결정론적이고 빠르고 저렴한 Playwright를 넣었다.
보리스가 말한 “에이전트에게 줄 검증 수단”이 바로 이 Playwright인 셈이다.

이상적 환경에서는 에이전트가 개발도 하고 테스트도 작성하고
실패하면 디버깅·수정까지 한다.
그렇다면 사람은 무엇을 하는가, 필요 없어지는가.
발표자는 아니라고 답한다.
무엇을 테스트할지, 도메인 지식을 어떻게 인코딩할지,
어떤 데이터·도구를 줄지, 모킹을 어떻게 할지 같은 결정은
아직 에이전트가 한 번에 잘하지 못하며 사람의 직관과 개입이 필요했다.
안드레이 카파시(Andrej Karpathy)의
“사고는 위임할 수 있어도 이해는 위임할 수 없다”를 인용하며,
CI를 다 통과해도 그 PR이 프로덕션에서 사용자에게 어떤 영향을 줄지는
에이전트가 맥락을 모른다고 짚는다.
사람의 일은 크게 셋이다.
(1) 자가 개선 루프 설계 — 플래너·제너레이터·힐러 흐름과 워크플로는
사람이 설계하고 검토한다.
(2) 도구와 맥락 제공 — 코드만 봐선 알 수 없는
도메인 지식·테스트 전략·환경 정보를 정리하고
에이전트가 쓸 도구를 준다.
(3) 모델이 바뀌면 하네스도 진화시키기 — 하네스는 지금 모델이 못 하는 걸
사람이 보강하는 자리이므로,
모델이 좋아지면 일부는 자동화되지만 그만큼 더 어려운 일을 맡기게 되어
새 자리에 다시 사람의 손이 필요하다.
즉 하네스는 사라지는 게 아니라 모양을 바꾸며 함께 자란다.
발표자는 세 줄 요약으로 마무리한다.
테스트 코드는 실행 가능한 센서이자 에이전트에게 주는 명세 문서이고,
이를 구축하면 에이전트가 스스로 작성·개선하는 자가 개선 루프가 돌며,
그 루프가 잘 돌도록 환경을 구축하는
하네스 엔지니어링이 우리의 역할이다.
끝으로 Vercel이 공개한 “에이전트와 책임감 있게 일하기” 원칙
— “에이전트가 올바른 행동을 쉽게 할 수 있도록 환경을 만들어라” —
를 인용하며 발표를 닫는다.

## 분석

### 이 발표의 핵심 주장은 “검증의 비결정성을 제거하라”다

발표 전체를 관통하는 논리는 단순하다.
에이전트는 똑똑하지만 자기 작업에 관대하므로 검증 수단을 줘야 하고,
그 검증 수단은 반드시 결정론적이어야 한다는 것이다.
이 주장의 무게중심은 “검증이 필요하다”가 아니라 “어떤 검증이냐”에 있다.
발표자는 에이전트의 브라우저 직접 조작, 마크다운 시나리오 재생,
LLM 채점자를 차례로 검토한 뒤 모두 기각하는데,
기각의 사유가 일관되게 “비결정적이고 느리고 비싸다”는 점이다.

이 논리 구조가 흥미로운 것은,
일반적인 테스트 담론이 “커버리지”나 “신뢰도”를 중심에 놓는 데 반해
이 발표는 “결정성(determinism)”을 일급 기준으로 끌어올린다는 점이다.
사람이 검증할 때는 비결정성이 큰 문제가 아니다.
사람은 한 번 보고 판단하면 끝이기 때문이다.
그러나 에이전트가 루프 안에서 반복적으로 자기 작업을 검증하려면,
같은 입력에 같은 출력을 보장하는 센서가 없으면 루프 자체가 성립하지 않는다.
즉 이 발표의 진짜 주장은 “E2E 테스트가 좋다”가 아니라,
“에이전트 자가 개선 루프의 채점자 자리는 결정론적 함수여야 한다”는
더 일반적인 명제다.

### 테스팅 트로피와 하네스 엔지니어링을 한 축으로 묶었다

발표는 두 개의 서로 다른 계보를 의도적으로 합류시킨다.
하나는 켄트 C. 도즈의 테스팅 트로피와
마틴 파울러로 대표되는 전통적 테스트 피라미드/트로피 담론이고,
다른 하나는 미첼 하시모토·보리스 체르니·앤트로픽으로 대표되는
2025–2026년의 하네스 엔지니어링 담론이다.
이 둘은 원래 다른 문제를 풀던 도구다.
테스팅 트로피는 “사람이 어디에 테스트 비용을 쓸지”를 다뤘고,
하네스 엔지니어링은 “모델이 일할 환경을 어떻게 구축할지”를 다룬다.

발표자의 기여는 이 둘이 사실 같은 자리를 가리킨다고 본 데 있다.
AI 에이전트 시대가 되면서
“테스트 작성 비용은 줄고 시스템 검증 중요도는 올라갔다”는 관찰이 그 다리다.
트로피의 꼭대기(E2E)는 비용 때문에 비워 두던 자리였는데,
에이전트가 작성을 대신하면서 비용 장벽이 낮아졌고,
동시에 컴포넌트 경계 결함이 늘면서 그 자리의 가치가 올라갔다.
그래서 비어 있던 E2E 자리가 곧
하네스의 “센서” 자리와 정확히 겹치게 된다는 것이다.
이 합류는 깔끔하지만,
뒤에서 비평하듯 두 담론의 전제 차이를 다소 흐린다.

### “스펙이자 센서”라는 이중성이 전체 설계의 주춧돌이다

발표에서 가장 자주 반복되는 명제는
“테스트 코드가 읽으면 명세, 돌리면 검증”이라는 이중성이다.
이는 단순한 수사가 아니라 전체 워크플로의 설계 원리다.
명세와 코드를 따로 관리하면 무엇이 최신인지 혼란스럽지만,
테스트 코드 한 벌이 명세 역할을 겸하면 코드가 바뀔 때 명세도 함께 바뀐다.
이 점은 사양이 충분히 상세해지면 그 자체로 코드가 된다는
최근의 spec-driven 논의와도 맞닿아 있다.

이 이중성은 자가 개선 루프의 두 부품인 “가이드”와 “센서”를
한 벌의 코드로 동시에 충족시킨다.
행동 전 방향을 잡는 가이드와 행동 후 자가 교정을 돕는 센서를 따로 만들면
두 배의 유지보수가 들지만,
Playwright 코드는 같은 텍스트가 양쪽 역할을 한다.
발표 후반의 모든 결론(자가 개선 루프, 평가자-최적화자, CI 게이트)이
이 한 가지 통찰에서 파생된다는 점에서,
이 이중성이야말로 발표의 실질적 중심이다.

### 도메인이 발표의 설득력을 떠받친다

이 발표가 추상적 당위에 그치지 않는 것은 금융 도메인의 구체성 덕분이다.
약관 동의 → 휴대폰 인증 → 본인 확인 → 결과 조회로 이어지는 대출 서비스 플로우,
네이버 인증서·홈택스·전자증명서 같은 외부 의존성,
사용자 상태에 따라 갈라지는 분기 구조는
모두 “실패하면 매출이 막히거나 신뢰가 깨지는”
크리티컬 유저 플로우의 실제 사례다.
이 구체성이 “모든 것을 테스트하지 말고 핵심만”이라는 권고에 무게를 싣는다.

동시에 발표는 이 구체성에서 일반 원칙을 끌어낸다.
상태를 API로 미리 세팅해 각 테스트를 독립 갈래의 시작점에서 출발시키는 방법,
외부 의존성을 호출 위치(브라우저/SSR·BFF)에 따라 다르게 모킹하는 방법,
하드 대기 대신 조건 기반 대기·시맨틱 셀렉터·번인(burn-in)으로
플래키를 작성 단계에서 잡는 방법은
금융 도메인을 넘어 일반화 가능한 E2E 운영 노하우다.
도메인 사례와 일반 원칙을 오가는 이 구성이 발표의 설득력을 만든다.

## 비평

### 1.5배 증가는 “검증이 병목”이라는 진단과 어긋난다

발표의 출발점인 “코딩은 빨라졌지만 검증이 병목”이라는 진단은
PR 227건 → 348건, 약 1.5배라는 수치에 기댄다.
그런데 이 수치를 곧이곧대로 보면 발표의 위기감과 묘하게 어긋난다.
9개월에 걸친 1.5배, 즉 한 달에 PR 십수 건이 더 늘어난 정도는
“검증이 따라잡을 수 없는 폭증”이라기보다
한 명의 개발자가 리뷰·검증 관행을 조금 다듬으면
흡수할 수 있는 규모에 가깝다.
발표는 “PR마다 사람이 로컬에 받아 돌려보고 리뷰하는 것은 불가능하다”고
말하지만,
348건이라는 절대량은 그 불가능성을 스스로 입증하지 못한다.
오히려 수치가 작기 때문에,
진짜 문제는 “양”이 아니라 다른 데 있다는 의심이 든다.

실제로 발표가 든 대표 사례(Next.js 업그레이드 회귀)는 PR 개수와 무관하다.
그것은 PR이 348건이든 3건이든 일어날 수 있는,
“에이전트가 확인하지 않고 끝났다고 보고하는” 신뢰의 문제다.
즉 발표의 설득력은 양적 데이터(1.5배)가 아니라
질적 일화(컴포넌트 경계 결함)에서 나오는데,
정작 도입부는 약한 양적 데이터를 전면에 내세워 인과를 흐린다.
PR이 1.5배 늘었다는 사실로부터 도출되는 것은
기껏해야 “검증을 자동화하면 좋겠다”까지이고,
그 자동화가 왜 하필 5개월·37개 라우트를 들인 E2E여야 하는지는
이 통계가 전혀 떠받치지 못한다.
발표는 데이터로 문을 열지만,
핵심 결론은 데이터가 아니라
일화와 권위 인용(보리스·파울러·하시모토)에 기대고 있다.

### 37개 라우트·플래키 제로는 성공의 조건을 숨긴다

“5개월간 37개 라우트, CI 29회 연속 성공, 플래키 없음”은 인상적인 성과지만,
이 숫자가 성립한 조건이 발표 내내 충분히 드러나지 않는다.
220개 페이지 중 37개 라우트만 골랐다는 것은
약 17%만 E2E로 덮었다는 뜻이고,
나머지 83%에서 발생할 수 있는 회귀는 이 시스템의 사각지대로 남는다.
크리티컬 유저 플로우에 집중한 것은 합리적 선택이지만,
“잘못된 수정으로 기존 동작이 깨지는 것을 막는다”는 목표와
17% 커버리지 사이의 긴장은 해소되지 않은 채 넘어간다.

“플래키 제로” 역시 그 자체로 의심스러운 지표다.
플래키가 없다는 것은 플래키를 제거하는 데 성공했다는 뜻일 수도 있지만,
플래키가 날 만큼 까다로운 플로우(외부 의존성, 타이밍, 비동기 상태)를
애초에 테스트 범위에서 뺐다는 뜻일 수도 있다.
발표가 외부 의존성을 모킹으로 고정하고 상태를 API로 미리 세팅한다고 밝힌 만큼,
실제 통합의 불확실성 상당 부분을 테스트 경계 밖으로
밀어낸 결과일 가능성이 크다.
이는 잘못된 설계가 아니라 합리적 트레이드오프지만,
“플래키 제로”라는 깔끔한 숫자가 그 트레이드오프를 가려 버린다.

### 모킹한 E2E는 “실제 사용자 검증”이라는 약속을 부분적으로 배신한다

발표는 Playwright의 첫 번째 장점으로
“실제 브라우저, 실제 사용자와 같은 레벨의 검증”을 든다.
단위 테스트가 함수를 검증한다면 E2E는 사용자 경험을 검증한다는 것이다.
그런데 같은 발표 후반에서,
외부 의존성(네이버 인증서, 홈택스, 전자증명서)을 고정 응답으로 모킹하고
사용자 상태를 테스트용 API로 미리 세팅한다고 밝힌다.
이 두 진술은 긴장 관계에 있다.

실제 사용자는 약관 동의부터 본인 확인까지 전 과정을 직접 거치고,
실제 외부 시스템과 통신한다.
그러나 테스트는 그 앞부분을 API로 건너뛰고,
외부 통신을 고정 응답으로 대체한다.
그렇다면 이 테스트가 검증하는 것은 “실제 사용자 경험”이 아니라
“우리가 통제하는 경계 안에서의 사용자 경험”이다.
플레이라이트 공식 문서의 “통제할 수 있는 것만 테스트하라”는 권고를
충실히 따른 결과이므로 설계로서는 옳지만,
“실제 사용자와 같은 레벨”이라는 마케팅성 표현은 이 한계를 덮는다.
정작 컴포넌트 경계 결함처럼 “실제로 돌려봐야 드러나는” 버그의 상당수는
외부 연동 지점에서 발생하는데,
바로 그 지점을 모킹으로 비활성화한다면,
E2E의 가장 큰 강점을 스스로 약화시키는 셈이다.

### 자가 개선 루프는 센서의 사각지대를 그대로 증폭한다

발표의 가장 매력적인 결론인
“에이전트가 틀려도 스스로 고쳐 다시 돌리면 된다”는 자가 개선 루프에는
구조적 위험이 내장돼 있다.
루프는 센서가 통과하면 작업이 끝났다고 본다.
그런데 센서(E2E 테스트)가 덮지 못하는 영역,
즉 83%의 비테스트 라우트나 모킹으로 가린 외부 연동에서는
에이전트가 무엇을 망가뜨려도 루프가 초록불을 켠다.

이는 발표 도입부의 Next.js 업그레이드 사례와 정확히 같은 실패 모드다.
당시 에이전트는 “확인하지 않고 끝났다고 보고”했다.
자가 개선 루프는 이 문제를 “확인할 수단”을 줘서 푼다고 주장하지만,
실제로는 “확인할 수단이 덮는 범위 안에서만” 푼다.
센서가 덮지 않는 곳에서는
에이전트가 자신감을 갖고 초록불을 보고 보고하므로,
오히려 사람의 경계심을 떨어뜨려 더 위험할 수 있다.
발표는 카파시를 인용해 “이해는 위임할 수 없다”며 사람의 역할을 남겨 두지만,
루프가 잘 돌수록 사람이 들여다볼 동기가 줄어든다는 역설은 다루지 않는다.

## 인사이트

### 결정론적 센서를 확보하지 못한 도메인에서는 이 워크플로 전체가 무너진다

이 발표의 모든 결론은 “결정론적이고 빠르고 저렴한 센서”의 존재를 전제로 한다.
Playwright가 그 자리를 채울 수 있었던 것은
프런트엔드 웹이라는 도메인 특성 덕분이다.
DOM은 관찰 가능하고, 사용자 동작은 클릭·입력·이동으로 이산화되며,
결과는 화면 텍스트로 결정론적으로 확인된다.
그러나 이 조건이 성립하지 않는 도메인이 훨씬 많다.

추천 시스템의 품질, 생성 모델의 출력, 분산 시스템의 타이밍 의존적 동작,
데이터 파이프라인의 통계적 정합성처럼
“정답이 비결정적이거나 분포로만 정의되는” 영역에서는
발표가 LLM 채점자를 기각한 바로 그 이유(비싸고 비결정적이고 느림)가
센서 자체에 그대로 들러붙는다.
이 경우 자가 개선 루프의 채점자 자리에 넣을
깔끔한 결정론적 함수가 존재하지 않는다.
발표는 “자가 개선 루프를 만들 수 있느냐가
에이전트에 일을 어디까지 맡길 수 있는지를 가른다”고 말하는데,
이 명제를 뒤집으면 더 날카로운 인사이트가 된다.
즉 어떤 도메인이 AI 에이전트에게 잘 맡겨지느냐는 모델의 똑똑함이 아니라
그 도메인에 값싼 결정론적 센서가 존재하느냐로 갈린다.
프런트엔드가 AI 코딩의 최전선이 된 것은 우연이 아니라,
E2E라는 결정론적 센서를 가졌기 때문이다.

### 테스트의 무게중심이 “회귀 방지”에서 “에이전트 사료”로 이동한다

발표가 무심코 드러내는 두 번째 효과는
테스트의 존재 이유 자체가 바뀌고 있다는 점이다.
전통적으로 테스트는 사람을 위한 안전망이었다.
회귀를 막고, 리팩터링에 확신을 주고, 문서 역할을 했다.
그런데 이 발표에서 테스트는 점점 “에이전트에게 먹일 양질의 데이터”로
재정의된다.
풍부한 Playwright 트레이스(스크린샷, DOM 스냅샷, 네트워크, 콘솔)가
강조되는 이유도,
그것이 사람의 디버깅을 돕기 때문이 아니라
에이전트의 자가 교정을 위한 구조화된 입력이기 때문이다.

이 전환은 테스트 설계의 우선순위를 조용히 바꾼다.
사람을 위한 테스트는 가독성과 의도 표현이 중요했지만,
에이전트를 위한 테스트는 실패 시 남기는 데이터의 풍부함과
기계 가독성이 중요해진다.
장기적으로 이는 “사람이 읽기 좋은 테스트”와
“에이전트가 고치기 좋은 테스트”가 갈라지는 결과를 낳을 수 있다.
발표는 두 목적이 한 벌의 코드로 행복하게 양립한다고 보지만,
트레이스 포맷·셀렉터 전략·실패 메시지 설계가
에이전트 최적화 쪽으로 기울면
정작 사람은 자기 코드베이스의 테스트를 점점 읽지 않게 될 수 있다.
테스트가 에이전트의 사료가 되는 만큼,
사람과 코드 사이의 이해의 끈은 한 가닥 더 가늘어진다.

### 하네스가 진화한다는 말은 곧 검증 부채가 이전된다는 뜻이다

발표는 “모델이 좋아지면 어떤 부분은 자동화되지만
그만큼 더 어려운 일을 맡기게 되니
새 자리에 사람의 손이 필요하다,
하네스는 사라지지 않고 모양을 바꾸며 함께 자란다”고 결론짓는다.
이는 위안이 되는 서사지만,
그 이면에는 발표가 말하지 않는 구조적 트레이드오프가 있다.
하네스가 모델을 따라 계속 진화해야 한다는 것은,
하네스 자체가 영구적인 유지보수 부채가 된다는 뜻이다.

37개 라우트의 E2E, `AGENTS.md`, E2E SSOT 문서, 헬퍼·유틸, MCP 서버,
세 종류의 에이전트 설정은
한 번 짜고 끝나는 자산이 아니라,
모델이 바뀔 때마다·도메인이 바뀔 때마다 갱신해야 하는 살아 있는 시스템이다.
모델이 똑똑해질수록 하네스의 어떤 부분은 불필요해지지만,
그 판단(무엇을 덜어내고 무엇을 새로 넣을지) 자체가 고급 인지 노동이다.
발표는 이를 “사람은 높은 수준의 도메인 결정을 한다”고 긍정적으로 표현하지만,
규모가 커지면 이 결정의 총량은 줄지 않고 오히려 늘어난다.

여기서 시간이 지나며 불거질 긴장이 있다.
하네스 구축의 정당화 논리는 “검증이 병목이니 자동화하자”였다.
그런데 하네스 자체가 끊임없는 진화를 요구한다면,
병목은 사라진 것이 아니라
“코드 검증”에서 “하네스 유지보수”로 이전된 것이다.
조직이 220개 페이지를 모두 덮고, 여러 팀이 각자의 하네스를 운영하고,
모델이 분기마다 바뀌는 상황을 상상해 보면,
하네스 엔지니어링은 그 자체로
새로운 전담 직무이자 새로운 종류의 기술 부채가 된다.
발표의 낙관적 결론은 옳지만,
“함께 자라는 시스템”이라는 표현은
그 시스템을 누가 어떤 비용으로 돌볼지를 끝내 비워 둔다.
