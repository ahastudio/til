# 시스템 루비를 쓰지 말라 — 운영체제 소유물과 개발 환경의 경계

<https://dontusesystemruby.com/>

<https://github.com/anothermh/dontusesystemruby.com>

## 요약

“시스템 루비(system Ruby)를 직접 쓰지 말고 버전 매니저로 직접 컴파일해
설치하라.” 이 글의 핵심 주장은 이 한 문장으로 요약된다. 저자는 2013년부터
Yammer, New Relic 같은 대형 루비 샵에서 일해온 엔지니어이며, Stack
Overflow의 ruby 태그에서 반복되는 초보자 질문에 매번 답하는 대신 “여기를
읽고 오라”고 건네주기 위해 이 문서를 만들었다고 밝힌다.

시스템 루비란 운영체제와 함께 설치되어 있거나 `sudo apt install ruby`
처럼 OS 패키지 매니저를 통해 설치되는 루비를 말한다. 반대로 사용자가
직접 다운로드해서 컴파일한 루비는 비(非)시스템 루비이며, 저자는 버전
매니저(asdf-vm, RVM, chruby, rbenv)를 통한 후자의 방식을 권장한다.

시스템 루비를 피해야 하는 이유는 “시스템 루비는 운영체제의 소유물이고
당신의 통제 아래 있지 않다”는 한 줄로 귀결된다. 구체적으로는 OS와 다른
패키지가 사용하라고 깔려 있는 것이지 개발자용이 아니고, 파일 시스템
권한이 제한되며, 버전이 오래되고(예: Ubuntu 18.04의 2.5.1), 업그레이드/
다운그레이드/제거가 불가능하며, 동시에 여러 버전을 쓸 수 없고, OS의
보안 패치로 언제든 덮어써질 수 있다는 것이다.

다만 저자는 예외를 인정한다. Docker 컨테이너 같은 프로덕션 환경에서는
매번 루비를 컴파일하는 것이 비효율적이므로 미리 빌드된 패키지, 즉
시스템 루비 형태가 적합하다. 버전 매니저는 개발 환경의 도구이지
프로덕션 도구가 아니다.

마지막으로 버전 매니저 선택 가이드를 제시한다. 여러 언어를 관리해야
하면 asdf-vm, 루비만 관리하면 RVM/chruby/rbenv 중에서 셸 환경에 대한
숙련도에 따라 고르라는 것이다. 초심자에게는 asdf-vm을 추천한다.

## 분석

### 주장 구조의 단순성

글의 논증은 “전제 → 파생 결과 → 예외 인정” 구조로 매우 간결하다.
전제는 “시스템 루비는 OS의 소유물이다”라는 단 하나이고, 나머지 9가지
근거(권한 제한, 오래됨, 업그레이드 불가 등)는 모두 이 전제에서 파생되는
논리적 결과다. 이 구조는 설득력이 높지만 동시에 전제가 흔들리면 전체가
무너지는 약점도 있다.

### 오랜 커뮤니티 관행의 성문화

이 문서는 새로운 주장을 펼치는 것이 아니라 루비 커뮤니티에서 10년 이상
반복되어 온 조언을 한 곳에 모은 것이다. 참조로 걸린 Thoughtbot의
“PSA: Do not use system Ruby”(2011년경), Chris Herring의 글, Bill
Patrianakos의 “Never Use System Ruby. Ever.”(2014) 등은 이 담론의
계보를 보여준다. 이는 파이썬 생태계의 pyenv/virtualenv, Node의 nvm,
자바의 SDKMAN 담론과 완전히 병렬적이다.

### 버전 매니저 생태계의 파편화

글이 소개하는 네 가지 매니저(asdf-vm, RVM, chruby, rbenv)는 각자
다른 철학을 보여준다. RVM은 셸 함수 오버라이드 기반의 “풍부한” 접근,
rbenv는 PATH 조작 기반의 “단순한” 접근, chruby는 최소주의, asdf-vm은
다중 언어 통합이다. 한 생태계 안에 네 가지 해법이 공존한다는 사실
자체가 루비 커뮤니티가 도구 선택에서 중앙집중적 권위를 거부하는 문화를
반영한다.

## 비평

### “프로덕션 예외”의 모호함

저자는 프로덕션에서는 시스템 루비가 적합할 수 있다고 인정하지만, 그
경계가 모호하다. Docker 이미지의 베이스에서 `apt install ruby`로
설치하는 것과, 컨테이너 빌드 단계에서 rbenv로 컴파일하는 것 중 무엇이
더 권장되는지는 현실 프로젝트에서 자주 논쟁거리다. 공식 `ruby` Docker
이미지는 사실 소스에서 컴파일된 루비를 담고 있다는 점에서, “프로덕션=
시스템 루비”라는 이분법 자체가 단순화된 틀이다.

### macOS 특수성의 희석

macOS의 시스템 루비는 2022년 macOS Ventura부터 기본 제거가 진행되었고,
Apple은 오랫동안 시스템 루비를 업데이트하지 않아 2.6대에 머물렀다.
Homebrew 사용이 보편화된 macOS 환경에서는 이 글의 경고가 더 이상
긴급하지 않다. 반면 Debian/Ubuntu 계열은 여전히 `ruby` 패키지가 일급
시민이라 경고의 강도는 배포판마다 다르다. 글은 이 OS별 차이를 충분히
구분하지 않는다.

### 컴파일 비용의 과소평가

저자는 “기기에 맞게 신선하게 컴파일된다”를 버전 매니저의 장점으로
내세우지만, Apple Silicon이나 저사양 Linux에서 루비 한 버전을
컴파일하는 데 수 분~십수 분이 걸린다는 현실은 언급하지 않는다. 최근
등장한 ruby-install의 바이너리 배포, mise, rbenv의 precompiled
옵션처럼 “컴파일하지 않는 버전 매니저”의 흐름은 이 글의 주장을 부분적
으로 무력화한다. 즉 “버전 매니저=직접 컴파일”이라는 전제 자체가
시대에 뒤진 면이 있다.

### asdf-vm 추천의 편향

글은 신규 사용자에게 asdf-vm을 추천하지만, 2024년 들어 asdf-vm은 Go로
재작성된 v0.14 이후의 호환성 혼란, 그리고 mise(구 rtx)의 부상으로
위상이 흔들리고 있다. 문서가 2023년 시점에 고정되어 있어 최신 흐름이
반영되지 않은 것이다. 특히 mise는 asdf 플러그인 호환성을 유지하면서
훨씬 빠르고 설정이 단순해 많은 개발자가 전환했다는 점이 빠져 있다.

## 인사이트

### “소유권의 전제가 갈등을 만든다” — 패키지 매니저라는 공유지 비극

이 글이 드러내는 가장 근본적인 문제는 기술적인 것이 아니라 소유권의
문제다. 시스템 루비가 거기 있는데 왜 쓰면 안 되는가? 그것이 OS의 것이지
당신 것이 아니기 때문이다. 이 소유권 충돌은 루비만의 문제가 아니라
파이썬의 악명 높은 “externally-managed-environment” 오류, Homebrew가
관리하는 디렉터리에 수동 설치할 때의 깨짐, apt로 설치한 Node 대 nvm으로
설치한 Node 사이의 PATH 전쟁에서 반복적으로 나타난다.

문제의 뿌리는 유닉스 파일 시스템이 본래 다중 사용자 타임셰어링 시대의
모델이라는 데 있다. `/usr/bin`은 관리자가 제공하는 공용 자원이고,
사용자는 자기 홈 디렉터리에서만 권한을 가진다. 그런데 현대 개발자는
자기 노트북의 단일 사용자이면서 동시에 OS와 “자원을 두고 경쟁”하는
모순된 위치에 있다. 버전 매니저는 이 모순을 “OS 공간과 사용자 공간의
분리”로 해결한 우회로다.

더 흥미로운 점은 이 문제가 컨테이너 시대에 해소된 것처럼 보이지만
실은 Docker 이미지 내부로 이동했을 뿐이라는 사실이다. `FROM ruby:3.2`
와 `FROM ubuntu:22.04 && apt install ruby` 사이의 선택은 결국 같은
소유권 논쟁의 반복이다. 소프트웨어 생태계는 문제를 해결하는 것이 아니라
추상화의 층을 한 단계 올려 같은 문제를 다시 마주하게 한다.

### “개발 환경과 배포 환경의 비대칭” — 버전 매니저가 말하지 않는 것

글은 개발에는 버전 매니저, 프로덕션에는 시스템 루비라는 깔끔한
이분법을 제시하지만, 이것이 감춘 비대칭이 있다. 개발자의 로컬은 rbenv로
관리되고, CI는 또 다른 도구로, 스테이징은 Docker 이미지로, 프로덕션은
또 다른 이미지로 관리된다면 “내 머신에서는 되는데” 문제가 폭증한다.
시스템 루비 회피가 로컬의 안전을 샀지만 환경 간 일관성이라는 더 큰
비용을 치르게 한 셈이다.

이 비대칭을 해결하려는 움직임이 최근 도구 생태계의 방향을 규정한다.
Nix와 devcontainer, mise의 `mise.toml`, `.ruby-version`의 보편화는
모두 “개발 환경의 정확한 재현”을 목표로 한다. 시스템 루비 금지는 이
방향으로 가기 위한 첫 단추였지만, 그 자체로는 일관성 문제를 해결하지
못한다. Thoughtbot과 Heroku가 루비 버전 파일을 생태계 표준으로
밀어붙인 10년간의 작업이 이 빈틈을 메워왔다.

역사적 유사성을 찾자면 C/C++ 세계의 시스템 libc 대 glibc/musl 정적
링킹 논쟁, 자바의 JDK 버전 관리, .NET의 글로벌 vs 로컬 SDK 문제가
있다. 모두 “공유된 런타임의 안정성”과 “프로젝트별 재현 가능성” 사이의
갈등이며, 최근 10년의 도구 발전은 점점 후자 쪽으로 기울어왔다. 루비의
버전 매니저 문화는 이 흐름의 선구자 중 하나였다.

### “커뮤니티 문서의 수명과 표준의 파편화” — 누가 진실을 관리하는가

이 사이트의 존재 자체가 흥미로운 사회적 현상이다. 공식 루비 문서
(ruby-lang.org)가 아닌, 한 개인이 CC-BY-SA로 운영하는 단일 페이지
사이트가 Stack Overflow 답변의 기본 링크가 되는 구조. 이것은 오픈소스
생태계의 “사실상의 표준(de facto standard)”이 어떻게 형성되는지를
보여주는 작은 표본이다. 공식 기관의 권위가 아니라 “초보자에게 똑같은
설명을 반복하기 지쳤다”는 실용적 동기에서 만들어진 문서가 커뮤니티의
표준이 된다.

그러나 이 모델의 약점은 수명이다. Firebase Hosting에 올라가 있고,
docsify로 렌더링되며, 한 사람의 GitHub 계정에 의존하는 사이트는
저자의 관심이 떠나면 사라진다. 이 글이 언급하는 Bill Patrianakos의
2014년 글(billpatrianakos.me)은 이미 HTTPS 없이 HTTP로만 서비스되고,
Chris Herring의 글도 개인 블로그에 기록되어 있다. 커뮤니티 문서는
개별적으로는 가볍지만 집단적으로는 링크 부패(link rot)에 취약하다.

이 현상은 위키백과가 등장하기 전의 FAQ 문화, Usenet 시대의 RFC 1855
(넷티켓 가이드라인) 같은 “비공식이지만 규범적인 문서”의 현대적
재현이다. 루비, 파이썬, Node 등 각 언어 생태계마다 이런 “반쯤 공식적인”
가이드가 존재하며, 이들은 공식 문서가 다루기 꺼리는 “하지 말아야 할
것” 담론을 담아낸다. 공식 문서는 본질적으로 “무엇을 할 수 있는가”를
설명하지만, 커뮤니티 문서는 “무엇을 하면 안 되는가”를 말한다. 후자는
실무 지식의 핵심이지만 공식적으로 보장될 수 없기 때문에 늘 주변부에
존재한다.

### “패키지 매니저 전쟁”이 언어 생태계 성숙도의 지표인 이유

루비에 네 개의 버전 매니저가 공존한다는 사실은 일견 혼란스러워 보이지만,
언어 생태계의 성숙도를 측정하는 하나의 지표로 읽을 수 있다. 초기
언어에는 버전 매니저 자체가 없다. 중기에는 하나의 지배적 도구가
등장한다(Perl의 perlbrew, Ruby 초기의 RVM). 후기에는 여러 도구가
경쟁하며 분화한다(현재의 Ruby, Python, Node). 그리고 최근에는 언어를
가리지 않는 통합 도구(asdf, mise, pkgx)로 수렴하려는 움직임이 나타난다.

이 분화-수렴 패턴은 생태계가 단일 언어의 문제에서 폴리글랏(polyglot)
환경의 문제로 이동하는 과정을 반영한다. 한 프로젝트가 루비 백엔드,
Node 프런트엔드, 파이썬 머신러닝 스크립트를 동시에 포함하는 것이 흔한
오늘날, 언어별 매니저 네 개를 유지하는 것은 그 자체로 비용이다. 루비만
위한 RVM이 여전히 살아있지만 점점 주변화되고, 다중 언어 통합 도구로
무게중심이 이동하는 것은 필연적이다.

흥미로운 점은 이 수렴이 역사적으로 OS의 패키지 매니저가 수행했던 역할의
재발명이라는 것이다. apt, yum, brew가 원래 “여러 도구를 하나의 인터페
이스로 관리”한다는 목표였는데, 개발자들은 이것이 “언어 버전 관리에
부적합”하다는 이유로 이탈했다가, 결국 자신들의 레벨에서 같은 역할을
하는 도구(mise 등)를 다시 만들고 있다. 소프트웨어 추상화는 문제를 해결
한다기보다, 문제를 해결할 자격이 있는 사람이 누구인지를 계속 재정의
한다. 시스템 루비를 쓰지 말라는 조언의 밑바닥에는 “이 결정은 OS
관리자가 아니라 개발자의 것이다”라는 주권 선언이 깔려 있다.
