## 소개

Superlog는 OpenTelemetry 데이터를 수집해 AI 에이전트가 인시던트를 자동으로
분석하고 수정 PR까지 여는 오픈소스 옵저버빌리티 도구다. Y Combinator P26 배치
출신으로, “AI 에이전트가 소프트웨어를 자가 치유한다(self-heal)”는 것을 핵심
가치로 내세운다. GitHub 저장소는 2026년 6월 기준 스타 514개, 포크 32개 수준으로
빠르게 성장 중이다.

핵심 작동 방식은 세 단계다. 첫째, OTLP(OpenTelemetry Protocol) 프록시가 트레이스,
로그, 메트릭을 수집한다. 둘째, 워커 프로세스가 노이즈 많은 신호들을 인시던트로
그룹핑한다. 인시던트 그룹핑은 Anthropic Claude 모델(`claude-sonnet-4-6` 기본값)을
활용한 LLM 기반 에이전트 루프로 구현된다. 셋째, 에이전트 러너가 인시던트를
조사해 근본 원인을 찾고 PR을 연다. 에이전트 실행 결과는 `agent_run.completed`
웹훅으로 외부 시스템에 전달된다.

기술 스택은 TypeScript 기반이며 Vite/React 프론트엔드(`apps/web`), HTTP API
(`apps/api`), OTLP 인테이크 프록시(`apps/proxy`), 백그라운드 워커와 에이전트
오케스트레이션(`apps/worker`)으로 구성된다. 데이터 저장은 Postgres(Drizzle ORM)와
ClickHouse 조합으로 처리하며, `packages/fingerprint`가 텔레메트리 핑거프린팅을
담당한다. Docker, Node.js 20+, pnpm 9+가 로컬 실행 전제 조건이다.

에이전트 실행 결과 페이로드는 상세하다. 근본 원인 텍스트와 신뢰도 점수(0-10
숫자 척도), 예상 영향 범위, SEV-1/2/3 심각도 분류, 열린 PR의 브랜치 이름·패치
내용·검증 통과 여부, Linear 티켓 연동 정보가 포함된다. PR 정책이 “never”로
설정된 경우나 에이전트가 노이즈로 판단한 경우에는 `noiseClassification`이나
`resolutionClassification` 필드가 채워진다.

오픈코어(open-core) 모델로 운영된다. 이 저장소에는 웹 앱, API, OTLP 프록시,
워커, Postgres 스키마, ClickHouse 쿼리, 에이전트 러너 인터페이스, `community`
에이전트 러너(로컬 인시던트 요약 기록)가 Apache 2.0 라이선스로 공개되어 있다.
Superlog Cloud는 무료 티어, 사용량 기반 요금제, 월정액 크레딧 팩으로 제공되는
호스팅 에디션이다.

## 분석

### 옵저버빌리티와 AI 에이전트의 접합이 왜 지금인가

전통적인 옵저버빌리티 도구—Datadog, New Relic, Grafana—는 데이터를 수집하고
시각화해 인간이 판단하는 구조를 중심으로 설계됐다. 알람이 울리면 사람이 대시보드를
열고, 트레이스를 추적하고, 원인을 추론한다. Superlog가 제안하는 것은 이 판단 루프를
AI 에이전트가 대체하는 구조다.

이 전환이 지금 가능해진 이유는 두 가지다. 첫째, 코드 이해 능력이 있는 LLM이
실용적 수준에 도달했다. 스택 트레이스, 에러 메시지, 코드 컨텍스트를 함께 읽어
근본 원인을 추론하는 것은 2023년 이전의 모델로는 신뢰성 있게 수행하기 어려웠다.
둘째, OpenTelemetry가 옵저버빌리티 데이터의 표준 포맷으로 자리잡으면서 에이전트가
처리해야 할 데이터 형식이 일관화됐다. OpenTelemetry 없이 Superlog 같은 도구는
각 서비스의 로그 형식을 개별적으로 파싱해야 했을 것이다.

### 에이전트 결과의 신뢰도 점수화가 의미하는 것

Superlog의 결과 페이로드에서 흥미로운 설계 결정은 근본 원인과 예상 영향을 자유
텍스트와 숫자 신뢰도(0-10)의 쌍으로 표현한다는 점이다. “10 = 코드/로그/트레이스/티켓
증거로 뒷받침됨, 0 = 순수 추측”이라는 스펙이 명시적으로 정의되어 있다. 이것은
에이전트 출력이 항상 옳다고 가정하지 않는다는 설계 철학이다.

이 설계는 AI 에이전트를 운영 환경에 통합할 때 반드시 부딪히는 신뢰 문제를 다루는
실용적인 방식이다. 에이전트가 틀릴 수 있다는 것을 인정하고, 신뢰도를 수치화해
인간의 개입 필요성을 판단할 수 있게 한다. 자동으로 PR을 여는 기능이 있어도,
신뢰도가 낮은 경우 인간 리뷰를 유도하는 흐름을 만들 수 있다.

### 오픈코어 전략과 에이전트 러너의 확장 구조

`apps/worker/src/agent-runner-backend.ts`와 에이전트 러너 인터페이스가 플러그어블
(pluggable)로 설계된 것은 오픈코어 전략의 핵심 분기점이다. 오픈소스 커뮤니티 에디션은
`community` 러너가 기본이고, Cloud 에디션은 더 강력한 관리형 에이전트 러너를 제공한다.
에이전트 실행 능력이 무료/유료 경계선이 되는 구조다.

이 구조는 Grafana OSS와 Grafana Enterprise의 경계와 유사하다. 데이터 수집과 시각화는
오픈소스로, 엔터프라이즈 기능(SSO, 고급 알람 등)은 유료로 분리한 것처럼, Superlog는
인프라(수집, 저장, 그룹핑)는 오픈소스로 공개하고 에이전트 실행 품질을 유료화의 레버로
삼는다. 개발자가 오픈소스로 프로덕션에 붙이면서 에이전트 품질이 필요해질 때
Cloud로 전환하는 경로를 의도한 것이다.

### 텔레메트리 핑거프린팅이 인시던트 그룹핑의 기반이다

`packages/fingerprint`가 별도 패키지로 분리된 것은 Superlog의 핵심 기술 중 하나가
텔레메트리 핑거프린팅임을 시사한다. 같은 에러가 여러 서비스에서 반복 발생할 때,
각각을 별개 인시던트로 처리하면 에이전트가 동일한 분석을 반복하게 된다. 핑거프린팅은
유사한 신호를 그룹핑해 에이전트 비용을 절감하고 중복 조사를 막는 핵심 메커니즘이다.

LLM 기반 그룹핑 에이전트(`grouping/agent.ts`)가 후보 인시던트들을 검토해 병합 여부를
결정하는 구조는, 단순 규칙 기반 핑거프린팅의 한계—형식이 조금만 달라도 같은 오류를
다른 것으로 분류하는 문제—를 LLM의 의미 이해로 보완하는 설계다. 인시던트 그룹핑에만
최대 5번의 툴 이터레이션과 `claude-sonnet-4-6` 모델이 투입된다는 것은 이 단계에
상당한 컴퓨팅 자원을 투입한다는 의미다.

## 비평

### 에이전트의 자율적 PR 생성은 프로덕션 신뢰 기준을 충족하지 않는다

Superlog의 가장 공격적인 기능은 에이전트가 자동으로 수정 PR을 여는 것이다. 웹훅
페이로드에 `validationPassed: true`가 있고, 에이전트가 패치를 생성해 브랜치를 열기까지
한다. 그러나 이 흐름이 프로덕션 환경에서 실제로 안전한지는 별개의 문제다.

AI가 생성한 패치는 근본 원인을 정확히 찾더라도 수정 코드가 사이드 이펙트를 낼 수
있다. 특히 코드베이스 전체 맥락을 이해하지 못한 상태에서 국소적 수정을 제안할 때
이 위험이 높아진다. `validationPassed`가 무엇을 검증하는지—컴파일 통과인지, 테스트
통과인지, 단순 문법 검사인지—는 문서에서 명확하지 않다. 이 불투명성이 있는 상태에서
자동 PR 생성을 기본 기능으로 제시하는 것은 “에이전트가 생성한 PR은 반드시 인간이
검토한다”는 암묵적 기대를 희석시킬 위험이 있다.

더 근본적으로, 에이전트가 PR을 여는 것과 그 PR이 실제로 유용한 것은 다르다. 잘못된
수정 제안이 담긴 PR이 쌓이면 개발자는 PR 검토 부담이 늘고, 결국 에이전트 PR을
무시하는 패턴이 형성된다. 이것은 자동화 신뢰의 Goodhart 법칙과 유사하다. “PR을
열면 성공”이라는 지표가 실제 가치 창출과 분리될 수 있다.

### 인시던트 그룹핑의 LLM 비용이 스케일에서 문제가 된다

인시던트 그룹핑에 최대 5번의 툴 이터레이션과 Claude Sonnet 모델이 투입된다. 프로덕션
시스템에서 인시던트는 지속적으로 발생하며, 특히 배포 직후나 장애 시간에는 폭발적으로
증가한다. 모든 새 인시던트마다 LLM 기반 그룹핑 에이전트가 실행된다면, 이 비용이
선형이 아니라 급격히 증가하는 상황이 발생할 수 있다.

코드에서 `ANTHROPIC_GROUPING_MODEL`과 `INCIDENT_GROUPING_TOOL_ITERATIONS`가 환경
변수로 설정 가능하다는 것은 비용 조절의 안전밸브가 설계되어 있음을 보여준다. 그러나
이것이 비용 관리의 책임을 사용자에게 전가하는 것이기도 하다. LLM API 비용이 직접
발생하는 self-hosted 환경에서 인시던트 폭발이 곧 Anthropic API 청구서 폭발로 이어질
수 있다. 이 비용 구조는 Superlog가 self-hosted를 오픈소스로 제공하면서 실질적으로
Anthropic API 의존성을 내장하고 있다는 점에서 “무료 오픈소스”라는 인상과 충돌한다.

### 저장소 공개 범위가 실제 에이전트 능력을 평가할 수 없게 만든다

이 저장소에는 `community` 에이전트 러너가 “로컬 인시던트 요약을 기록한다”고 명시되어
있다. 실제로 PR을 생성하고 근본 원인을 높은 신뢰도로 분석하는 능력은 Cloud 에디션의
관리형 에이전트 러너에 있다는 것을 암시한다. README의 핵심 가치 제안—“AI 에이전트가
소프트웨어를 자가 치유한다”—은 Cloud 에디션을 전제로 한다.

이것은 오픈코어 전략에서 흔한 텐션이다. 오픈소스 저장소가 프로젝트의 얼굴 역할을
하면서 마케팅 가치 제안을 전달하지만, 실제로 그 가치를 경험하려면 유료 서비스로
전환해야 한다. 개발자가 self-hosted로 프로덕션에 붙인 후 “왜 PR이 자동으로 안 열리지?”
라는 질문에 도달했을 때, 그것이 설정 문제인지 Cloud 전용 기능인지 README만으로는
명확하지 않다. 이 경계의 불투명성은 의도적일 수 있지만 사용자 신뢰 비용을 만든다.

## 인사이트

### 옵저버빌리티 에이전트는 알람 피로의 해법이 아니라 새로운 형태의 피로를 만들 수 있다

소프트웨어 팀이 겪는 “알람 피로(alert fatigue)”는 수천 개의 알람 중 어떤 것이
실제로 중요한지 판단하는 인지 부담에서 온다. Superlog는 이것을 에이전트가 그룹핑하고
분류해 해결하겠다고 제안한다. 그러나 에이전트가 생성한 인시던트와 PR이 쌓이기
시작하면 새로운 피로가 발생할 수 있다. “에이전트 PR 피로”—자동으로 열린 PR을
검토해야 하는 부담이다.

이 현상은 GitHub Dependabot이 보안 업데이트 PR을 자동으로 생성하기 시작했을 때
이미 관측됐다. 초기에는 편리했지만, 규모가 커지면서 수십 개의 자동 PR이 쌓이고
개발자들이 이를 무시하거나 일괄 닫는 패턴이 형성됐다. Superlog의 에이전트 PR이
Dependabot과 다른 점은 수정 품질이 훨씬 가변적이라는 것이다. 의존성 업데이트는
의도가 명확하지만, 에이전트가 분석한 근본 원인과 수정 코드의 품질은 인시던트마다
크게 다를 수 있다.

2차 효과로, 에이전트 PR에 익숙해진 팀이 “어차피 Superlog가 고쳐주겠지”라는 기대로
코드 품질 유지에 덜 신경 쓰기 시작할 위험이 있다. 자동화가 안전망이 아닌 변명이
되는 패턴이다.

### AI 에이전트와 코드베이스의 결합이 새로운 종류의 락인을 만든다

전통적인 옵저버빌리티 도구의 락인은 데이터 형식과 대시보드 설정에 있었다. Superlog
같은 도구가 PR을 생성하고 코드를 수정하기 시작하면, 에이전트가 만든 수정 이력과
에이전트의 코드 이해 방식이 코드베이스 자체에 누적된다. 이것은 단순한 설정 락인이
아니라 코드 이력 락인이다.

에이전트가 특정 패턴으로 수정을 제안하면, 그 패턴에 맞게 코드가 점점 변형된다.
이 상태에서 다른 옵저버빌리티 에이전트로 전환하면, 새 에이전트는 이전 에이전트가
만든 수정 패턴에 익숙하지 않을 수 있다. 도구 전환의 비용이 기존 옵저버빌리티 도구
전환보다 훨씬 높아진다.

이 패턴은 역사적으로 인프라 자동화 도구에서 반복됐다. Puppet, Chef, Ansible이
도입될 때마다 인프라 코드가 해당 도구의 패턴으로 누적됐고, 전환 비용이 높아지면서
사실상 락인이 형성됐다. AI 에이전트는 이 락인을 코드 레벨까지 확장한다.

### OpenTelemetry 표준화가 에이전트 레이어의 경쟁을 가속한다

Superlog가 가능한 이유 중 하나는 OpenTelemetry가 텔레메트리 데이터의 표준 형식을
제공하기 때문이다. OTLP를 지원하는 서비스라면 Superlog를 붙이는 것이 코드 변경 없이
가능하다. 이것은 역설적으로 Superlog의 경쟁 진입 장벽을 낮춘다. Superlog가 가능하다면
다른 팀도 같은 구조로 유사한 도구를 만들 수 있다.

OpenTelemetry는 데이터 레이어를 공공재화했다. 기존 옵저버빌리티 업체들이 독점 에이전트
포맷으로 경쟁 장벽을 쌓은 것과 달리, OTLP 기반에서는 수집 레이어가 아닌 분석 레이어—
에이전트가 무엇을 어떻게 보는가—가 차별점이 된다. 이것은 단기적으로는 Superlog에게
기회지만, 장기적으로는 에이전트 레이어에서의 경쟁이 치열해진다는 것을 의미한다.
기존의 Datadog, New Relic이 자체 에이전트를 붙이는 것이 기술적으로 어렵지 않다.

이 경쟁 구도에서 Superlog의 방어 가능한 차별점은 에이전트 품질과 오픈소스 커뮤니티
생태계다. 후자는 `superloglabs/skills`라는 별도 저장소를 통해 구축하려는 시도로 보인다.
커뮤니티가 기여한 스킬이 에이전트의 능력을 확장하는 구조는 플랫폼 네트워크 효과를
노리는 전략이다. 이것이 실제로 작동하려면 에이전트 스킬 생태계가 충분히 성장해야
한다는 전제가 있다.

### 자가 치유 소프트웨어는 장애 대응 문화를 재형성한다

“소프트웨어가 자가 치유한다”는 비전이 현실화된다면, 온콜 문화(on-call culture)의
의미가 근본적으로 바뀐다. 현재 온콜은 장애 알람에 즉각 반응하고 근본 원인을 찾아
수동으로 수정하는 역할이다. 에이전트가 이 루프를 24시간 자율적으로 처리한다면,
온콜 엔지니어의 역할은 에이전트가 열어놓은 PR을 승인하거나 거부하는 것으로 바뀐다.

이 전환이 가져오는 2차 효과는 장애 대응에서 인간의 학습 기회가 줄어든다는 것이다.
온콜 로테이션은 고통스럽지만 시스템에 대한 깊은 이해를 강제하는 훈련이기도 하다.
에이전트가 장애를 처리하면서 인간이 시스템을 직접 디버깅하는 경험이 줄어들면,
전 세대 엔지니어가 가진 “시스템을 몸으로 아는” 지식이 축적되지 않는다. 앞서 분석한
백명석의 “인지적 부채” 개념이 옵저버빌리티 영역에서도 그대로 적용된다.

장기적으로 에이전트 의존도가 높아진 조직은 에이전트가 처리하지 못하는 드문
장애—데이터가 불충분하거나 맥락이 너무 복잡한—에서 인간 개입 능력이 약화될 위험이
있다. 일상적 장애의 자동화는 비일상적 장애 대응 역량의 저하를 비용으로 치를 수 있다.
