# Claude가 “load-bearing”을 그만 말하게 하기

원문: <https://jola.dev/posts/how-to-stop-claude-from-saying-load-bearing>

HN 토론: <https://news.ycombinator.com/item?id=48905248> (595점, 601개 댓글)

## 요약

저자는 Claude가 응답에서 “honest take”나 “load-bearing” 같은 표현을
지나치게 자주 반복해서 쓰는 문제를 지적한다.
이런 표현 자체가 틀린 것은 아니지만, 반복되면 글이 상투적으로 느껴지고
읽는 사람의 신뢰를 갉아먹는다.
문제는 프롬프트로 “이 단어를 쓰지 마”라고 지시해도 모델이 완전히
따르지 않는다는 데 있다.

저자가 택한 해법은 모델의 생성 자체를 바꾸는 대신,
Claude Code의 메시지 표시(display) 단계에 개입하는 hook을 만드는
것이다.
이 hook은 모델이 실제로 무엇을 “생각”했는지는 건드리지 않고,
화면에 출력되기 직전의 텍스트에서 특정 단어나 구문을 사전에 정의한
다른 표현으로 치환한다.

구현은 세 단계로 이루어진다.
먼저 JSON 형식의 입력을 받아 사전형 치환 규칙을 정규표현식으로
적용하는 Python 스크립트를 작성한다.
대소문자 구분 없이 패턴을 매칭하고, 변환된 텍스트를 다시 JSON으로
출력하도록 만든다.
이 스크립트를 `~/.claude/hooks/wordswap.sh` 같은 경로에 저장하고
실행 권한을 부여한 뒤, `~/.claude/settings.json`의 hooks 설정
블록에 등록하면 새 세션이 시작될 때마다 자동으로 로드된다.

저자는 이 방식이 단순히 거슬리는 표현을 없애는 데 그치지 않고,
“spicy doodad”처럼 장난스러운 치환까지 자유롭게 만들 수 있는 여지를
보여준다.
결국 이 글은 모델 출력을 표시 단계에서 후처리하는 작은 자동화가
사용자 경험을 실질적으로 개선할 수 있다는 사례를 제시한다.

## 분석

### 이 해법은 모델 정렬 문제를 UI 계층으로 옮긴다

이 글의 핵심 통찰은 “모델이 특정 표현을 쓰지 않게 만들자”는 문제를
모델 자체의 문제가 아니라 표시 계층의 문제로 재정의한 데 있다.
프롬프트 엔지니어링으로 반복 표현을 없애려는 시도는 모델의 확률
분포에 개입하려는 시도이고, 성공률이 들쭉날쭉하다.
반면 텍스트 치환 hook은 모델이 무엇을 생성하든 상관없이 사용자가
보는 최종 결과물만 통제한다.
이는 “모델을 원하는 대로 만들 수 없다면, 모델과 사용자 사이의
인터페이스를 통제하자”는 실용적 태도이며, LLM 도구 생태계 전반에서
반복되는 패턴이다.

### hook 기반 자동화는 Claude Code의 확장성을 드러낸다

이 글이 가능했던 이유는 Claude Code가 메시지 표시 단계에 사용자
정의 스크립트를 끼워 넣을 수 있는 hook 시스템을 제공하기 때문이다.
이런 구조는 도구 제작사가 모든 사용자 취향을 예측해 기본 동작에
반영할 수 없다는 전제를 인정하는 설계다.
대신 표준화된 개입 지점(hook)을 열어 두고, 개별 사용자가 자신의
불만을 스스로 해결하도록 위임한다.
이는 텍스트 편집기의 플러그인 생태계나 셸의 alias 설정과 비슷한
철학이며, LLM 도구가 성숙해질수록 이런 확장 지점이 늘어날 것이라는
신호로 읽을 수 있다.

### 왜 하필 “load-bearing”과 “honest take”인가

저자가 예로 든 두 표현은 우연이 아니다.
“load-bearing”은 시스템의 특정 구성 요소가 겉보기와 달리 핵심적인
역할을 한다는 것을 강조할 때 쓰는 비유이고, “honest take”는 의견을
꾸밈없이 전달한다는 뉘앙스를 담은 표현이다.
두 표현 모두 모델이 “통찰력 있어 보이는 말투”를 흉내 낼 때 자주
동원하는 수사적 장치다.
즉 이 글이 겨냥하는 것은 개별 단어가 아니라, RLHF 과정에서 학습된
특정 톤의 과잉 사용이라는 더 큰 패턴이다.
HN 댓글 중 한 사용자는 이런 반복 표현의 목록을 직접 채집해 공개했는데,
"projection", "strand", "load-bearing", "frontier", "honest",
"residuals", "rescission" 등이 나열되어 있었다[^arathorn].
이는 개별 단어가 문제가 아니라 특정 톤 전체가 하나의 패턴으로
굳어 있다는 저자의 주장을 뒷받침한다.
또 다른 댓글은 이 현상을 심리학적으로 짚었는데, 사람이 같은 말투를
반복하면 "말투(speaking style)"라고 부르고 넘어가지만 기계가 같은
일을 하면 유독 거슬려한다는 지적이었다[^danielvaughn].

### 반복 표현은 개인 어휘 습관이 10억 배로 증폭된 결과다

한 댓글은 이 현상의 규모를 짚었다.
원래 특정 어휘를 선호하는 습관은 한 사람이 하루에 쓸 수 있는 수천
단어 안에서 나타나는 사소한 개인차였지만, 이제는 같은 모델이 하루에
생성하는 수백억 토큰에 그 편향이 그대로 곱해지면서 훨씬 눈에 띄는
문제가 됐다는 것이다[^infogulch].
이 관점은 저자의 hook 해법이 왜 필요한지를 다른 각도에서 설명한다.
개별 사용자 입장에서는 사소해 보이는 어휘 편향도, 수많은 사용자에게
동시에 적용되는 하나의 모델에서는 인터넷 전체의 문체를 균질화하는
힘으로 작동할 수 있다.

## 비평

### 치환은 원인이 아니라 증상을 다룬다

이 hook은 모델이 왜 특정 표현을 과용하는지에 대해서는 아무것도
설명하지 않고 해결하지도 않는다.
치환 목록에 없는 새로운 상투어가 등장하면 사용자는 또다시 사전을
갱신해야 하고, 이 작업은 끝없이 반복될 수 있다.
근본적으로는 모델의 출력 분포 자체가 특정 수사에 편향되어 있다는
문제가 남아 있으며, 저자는 이 편향이 어디서 비롯됐는지, 왜 여러
사용자가 동시에 같은 표현을 거슬려 하는지에 대한 논의는 하지
않는다.
표시 단계 개입은 빠른 완화책일 뿐, 원인 진단은 아니다.

### 치환이 의미를 왜곡할 위험을 다루지 않는다

정규표현식 기반의 사전형 치환은 문맥을 이해하지 못한다.
“load-bearing”이 실제로 시스템 설계를 설명하는 데 정확하고 필요한
비유로 쓰인 경우에도, hook은 그 문맥을 구분하지 않고 기계적으로
치환해 버릴 수 있다.
이 경우 원래 모델이 의도한 정확한 표현이 사용자가 임의로 정한
대체어로 바뀌면서, 오히려 의미가 부정확해지거나 어색해질 위험이
있다.
저자는 이런 오탐(false positive) 가능성과 이를 완화할 방법에 대해
다루지 않는다.

### 개인 설정이 팀 단위로 확장될 때의 문제를 다루지 않는다

이 글은 개인 사용자가 `~/.claude/settings.json`에 직접 설정을
넣는 시나리오를 전제로 한다.
하지만 팀이나 조직 단위로 이런 치환 규칙을 도입하려면, 누가 어떤
단어를 금지할지에 대한 합의, 규칙의 버전 관리, 그리고 사용자마다
다른 취향 사이의 충돌 조정이 필요하다.
저자는 이 hook을 개인의 생산성 도구로만 제시할 뿐, 여러 사람이
공유하는 설정으로 확장했을 때 생기는 거버넌스 문제는 전혀 건드리지
않는다.
실제로 한 댓글은 자신만의 금지어 목록을 따로 관리하고 있다고
밝혔는데, "smoking gun", "blast radius", "landed", "spine",
"sprint"나 "epic" 같은 애자일 용어까지 포함되어 있었다[^hnarayanan].
이는 "무엇을 금지할 것인가"에 대한 기준이 사용자마다 크게 다르다는
뜻이고, 이런 목록이 팀 단위로 합의되기는 더더욱 어렵다는 것을
보여준다.

### 표시 단계 개입이 모델의 실제 판단력까지 흐릴 수 있다

한 댓글은 치환 hook과는 다른 접근, 즉 모델에게 직접 "em-dash를 쓰지
말라"고 프롬프트로 지시하는 방식의 부작용을 지적했다.
이 사용자는 모델이 작업 중 스스로에게 "no em-dashes"라고 되뇌는
장면을 자주 목격했다고 하며, 이런 자기 지시가 모델이 정작 집중해야
할 작업에서 주의를 분산시키는 것은 아닌지 우려했다[^ksaun].
이는 저자의 hook 방식이 프롬프트 지시보다 나은 이유이기도 하다.
표시 단계에서 치환하면 모델의 실제 추론 과정에는 개입하지 않으므로,
이런 부작용에서는 비교적 자유롭다.
하지만 동시에 이는 저자가 다루지 않은 다른 대안들, 예컨대 "이전에
작성한 원고를 다듬어 달라"고 요청하는 방식이 반복 표현을 줄이는 데
더 효과적일 수 있다는 가능성도 시사한다[^subiculumcode].

## 인사이트

### 표시 단계 개입은 모델 벤더에 대한 사용자 권력의 첫 형태다

모델 제공사가 특정 톤이나 표현 습관을 바꾸려면 파인튜닝이나
시스템 프롬프트 조정처럼 무거운 작업이 필요하고, 이는 전체
사용자에게 일괄 적용된다.
반면 표시 단계 hook은 개별 사용자가 벤더의 결정을 기다리지 않고
스스로 결과물을 조정할 수 있게 해준다.
이는 작지만 중요한 권력 이동이다.
앞으로 LLM 도구가 성숙해질수록, 사용자는 모델 자체를 바꾸는 대신
자신과 모델 사이의 인터페이스 계층을 점점 더 적극적으로 커스터마이즈하려
할 것이고, 이는 브라우저 확장 프로그램이나 애드블로커가 웹
생태계에서 했던 역할과 유사한 자리를 차지할 수 있다.

### 상투어 목록 자체가 모델의 “문체 지문”이 될 수 있다

여러 사용자가 독립적으로 “load-bearing”, “honest take” 같은 특정
표현을 지적한다는 사실은, 이런 반복이 개별 세션의 우연이 아니라
모델의 안정적인 문체적 특징이라는 뜻이다.
이는 마치 특정 작가의 글에서 반복되는 어휘 습관으로 저자를 식별할
수 있는 것처럼, 모델 세대나 파인튜닝 버전마다 고유한 “문체 지문”이
생길 수 있음을 시사한다.
연구자나 도구 제작자 입장에서는 이런 반복 표현 목록을 수집하는 것
자체가 모델 버전을 구분하거나 파인튜닝 효과를 진단하는 신호로
쓰일 수 있다.

### 사용자 커스터마이징이 늘어날수록 “모델의 목소리”라는 개념이 흐려진다

hook 기반 치환이 널리 퍼지면, 같은 모델이라도 사용자마다 실제로
받아보는 텍스트가 달라진다.
이는 모델이 가진 고유한 “목소리”라는 개념 자체를 약화시킨다.
장기적으로는 벤더가 제공하는 기본 출력과, 사용자가 개인화한 최종
출력 사이의 간극이 커지면서, “이 모델이 이렇게 말한다”는 평가나
벤치마크가 실제 사용자 경험을 얼마나 대표하는지에 대한 의문이
생길 수 있다.
이는 검색 엔진 결과가 개인화로 인해 사용자마다 달라지면서 “구글
검색 결과”라는 단일한 기준이 흐려졌던 과거 패턴과 유사한 궤적을
그릴 가능성이 있다.

### 반복 표현은 이미 인간의 글쓰기로 역류하고 있다

한 댓글은 “wire”나 “wiring” 같은 단어가 들어간 커밋 메시지의 비율이
Claude 사용량과 직접 상관관계를 보인다고 언급했다.
일 년 전만 해도 “wired”라는 표현을 쓰는 사람이 거의 없었는데, 지금은
커밋의 약 10퍼센트에서 등장한다는 것이다[^chipoplemoral].
이는 hook으로 사용자가 보는 화면만 정제해도, 모델의 어휘 습관이
이미 인간의 실제 글쓰기 습관에 스며들고 있다는 신호다.
표시 단계 개입이 개인의 경험은 개선할 수 있어도, 모델과 인간
사이에서 어휘가 상호 오염되는 더 큰 흐름은 막지 못한다.
결국 이 인사이트는 앞서 지적한 “모델의 목소리가 흐려진다”는 전망과
정반대의 위험, 즉 모델의 특정 목소리가 오히려 인간의 언어 습관
쪽으로 새어 나가 굳어질 수 있다는 가능성을 함께 짚어준다.
