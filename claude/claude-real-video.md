# claude-real-video — LLM이 실제로 영상을 보게 만드는 도구

<https://github.com/HUANGCHIHHUNGLeo/claude-real-video>

HN 토론: <https://news.ycombinator.com/item?id=48766005> (166점, 59개 댓글)

## 소개

대부분의 AI 도구는 영상을 실제로 “보지” 않는다.
YouTube 링크를 ChatGPT에 붙여넣으면 트랜스크립트(transcript)를 읽을 뿐이고,
Claude는 영상 파일을 아예 받지 않는다.
Gemini는 영상을 직접 읽을 수 있지만 고정 간격(기본 1fps)으로 프레임을 샘플링하므로
빠른 컷 전환이 빠져나간다.

`claude-real-video`(`crv`)는 다르게 작동한다.
URL이나 로컬 파일을 가리키면 실제로 의미 있는 프레임만 추출하고,
중복에 가까운 프레임을 제거하며, 오디오를 전사한 뒤,
어떤 LLM이든 읽을 수 있는 깔끔한 폴더를 만든다.
모든 처리는 로컬에서 실행되며, LLM에 전달되는 것은 이후에 사용자가 직접 선택한
프레임과 텍스트뿐이다.

## 작동 방식

`crv`는 다섯 단계로 처리한다.

**1단계 — 수집**: URL이면 `yt-dlp`로 다운로드하고, 로컬 파일이면 복사한다.
쿠키 파일(`--cookies`)로 로그인이 필요한 콘텐츠도 처리할 수 있다.

**2단계 — 프레임 추출**: `ffmpeg`의 `select` 필터로 씬 체인지(scene change)를
감지하되, 최소 밀도 보장(`--fps-floor`, 기본 1fps)을 함께 적용한다.
빠른 컷이 많은 뮤직비디오와 느린 스크린캐스트를 동시에 처리할 수 있는 이유다.

**3단계 — 중복 제거(dedup)**: 다운스케일된 RGB 픽셀 차이를 슬라이딩 윈도
(`--dedup-window`, 기본 4프레임)와 비교한다.
퍼셉추얼 해시(perceptual hash)가 아닌 실제 픽셀 차이를 사용하므로
단색 배경이나 동일한 밝기의 색조 변화에서도 눈이 멀지 않는다.
A-B-A 컷 구조처럼 이미 본 씬이 다시 등장해도 재전송하지 않는다.

**4단계 — 텍스트**: 로컬 파일에 사이드카 자막(`.srt`/`.vtt`)이 있거나
내장 자막 트랙이 있으면 그것을 그대로 트랜스크립트로 사용한다.
자막이 없을 때만 Whisper로 오디오를 전사하며, 오디오가 없으면 깔끔하게 건너뛴다.

**5단계 — 매니페스트**: `MANIFEST.txt`가 모든 내용을 요약해 LLM이 읽을 수
있는 단일 진입점을 제공한다.

결과로 LLM은 영상을 보고(키프레임), 읽고(트랜스크립트),
`--keep-audio` 옵션을 사용하면 듣기(전체 사운드트랙)까지 할 수 있다.

## 주요 옵션

| 옵션 | 기본값 | 역할 |
| -------------------- | ------ | -------------------------------------------- |
| `--scene` | 0.30 | 씬 체인지 민감도 (낮을수록 프레임 증가) |
| `--fps-floor` | 1.0 | N초마다 최소 1프레임 보장 |
| `--max-frames` | 150 | 프레임 상한 |
| `--dedup-threshold` | 8 | 새 프레임으로 인정할 최소 픽셀 변화율 (%) |
| `--dedup-window` | 4 | 비교 대상 이전 프레임 수 |
| `--grid` | off | 9프레임씩 묶어 컨택트 시트(contact sheet) 생성 |
| `--adaptive` | off | 변화 느린 콘텐츠에 롤링 임계값 적용 |
| `--why` | – | 분석 렌즈 지정 (“가격 전략을 찾아라” 등) |
| `--kb` | – | 분석 결과를 노트 폴더에 날짜별로 저장 |
| `--report` | off | 드롭된 프레임과 keep/drop 시각화 보고서 |
| `--viewer` | off | 로컬 `viewer.html` 생성 (네트워크 불필요) |

## 분석

### 고정 샘플링 대비 씬-어웨어(scene-aware) 추출의 실질적 차이

고정 간격 샘플링의 문제는 단순히 프레임 수가 많다는 게 아니다.
콘텐츠의 밀도와 샘플링 주기가 불일치한다는 구조적 결함이다.
10분짜리 정적 슬라이드 강의를 1fps로 샘플링하면 600개의 거의 동일한 프레임이 생긴다.
반대로 초당 3컷 이상의 광고 영상은 대부분의 시각 변화를 놓친다.
`crv`의 씬 체인지 감지 + 밀도 보장 조합은 이 두 극단을 동시에 처리한다.

슬라이딩 윈도 중복 제거는 A-B-A 구조의 영상에서 특히 가치 있다.
인터뷰 영상에서 두 사람의 얼굴을 번갈아 보여주는 컷 구조를 생각해 보라.
고정 샘플링은 같은 얼굴을 수십 번 전송한다.
`crv`는 이미 본 씬이 다시 나타나면 보내지 않는다.
이는 LLM 컨텍스트 비용과 이해도 모두에 직접 영향을 미친다.

### 로컬 처리와 프라이버시의 연결

README가 “처리는 로컬에서 실행된다”를 여러 번 강조하는 것은 단순한 기술 사양이 아니다.
Gemini 같은 서비스는 영상 데이터 전체를 Google 서버로 전송한다.
크리에이터나 기업 사용자 입장에서 미공개 영상, 내부 교육 자료, 고객 인터뷰를
외부 서버로 보내는 것은 허용하지 못할 리스크다.
`crv`는 원본 영상을 어디에도 보내지 않고, 사용자가 선택한 프레임과 텍스트만
LLM에 전달한다. 이 아키텍처는 민감한 도메인에서의 채택 장벽을 낮춘다.

그런데 bonoboTP는 이 주장에 정면으로 이의를 제기한다.[^bonoboTP]
“`crv`가 추출한 프레임은 결국 Claude API로 전송된다.
데이터가 당신의 머신에 머문다는 주장은 오해를 유발한다”는 것이다.
이 지적은 정확하다. “로컬 처리”는 영상 분해 단계에만 해당하며,
LLM이 분석하는 순간 프레임은 어딘가의 서버로 이동한다.
로컬 VLM(Visual Language Model)을 사용하지 않는 한 완전한 로컬 처리는 불가능하다.
fny가 “아니면 로컬에서 실행되는 LLM을 쓰면 되잖아요”라고 덧붙인 것처럼,[^fny]
실제로 프라이버시를 완결하려면 Ollama 같은 로컬 VLM과의 조합이 필요하다.

### 비용과 모델 선택의 실용적 고려

fzysingularity는 Claude로 영상을 분석하는 것이 비용 면에서 비효율적이라고 지적한다.[^fzysingularity]
대신 Gemini나 로컬 VLM을 쓰는 것이 현실적이며,
vlm-run 같은 라이브러리를 통해 모델을 교체하며 다양한 영상 인코딩 방식을 실험할 수 있다고 제안한다.
이는 `crv`가 출력 폴더를 만드는 방식이 모델 독립적으로 설계된 점과 맞닿는다.
실제로 Claude에 비해 Gemini의 영상 처리 비용이 현저히 낮고,
로컬 VLM은 반복적인 분석에서 비용을 0으로 만든다.
이름은 "claude-real-video"지만 내부 구조는 어느 LLM에도 붙일 수 있는 범용 전처리 도구다.

### Claude Code 스킬로의 통합이 시사하는 방향

README에 Claude Code 스킬로 설치하는 방법이 포함된 것은 주목할 만하다.
`~/.claude/skills/`에 복사하면 Claude Code가 대화 중에 자동으로 영상을 분석할 수 있다.
이는 `crv`가 단순한 CLI 도구를 넘어 에이전트 워크플로의 컴포넌트로 설계됐음을 보여준다.
“에이전트가 영상을 직접 볼 수 있다”는 명제가 실용적으로 구현된 사례다.

## 비평

### 키프레임은 영상이 아니다 — 도구 이름이 과장하는 것

octember는 핵심을 찌른다.[^octember]
"키프레임은 영상이 아니다.
모션, 객체 영속성(object permanence)은 이미지 집합에서 Claude가 추론할 수 없다.
멋진 데모이긴 하지만."
이 비판은 `crv`의 이름("real video")이 과장을 내포한다는 것을 보여준다.
LLM이 정지 프레임 묶음에서 할 수 있는 것과 실제로 영상을 보는 것은 다르다.
공이 어떻게 튀기는지, 사람이 어떻게 걷는지, 카메라가 얼마나 빠르게 패닝하는지는
연속된 프레임에서 픽셀이 어떻게 변화하는지를 직접 처리해야 알 수 있다.
Lerc도 같은 방향의 질문을 던진다.[^Lerc]
"모델이 여러 프레임에서 모션을 추론할 수 있는가?
공이 튀기는 애니메이션 프레임을 주면 어떤 것이 더 현실적인지 알 수 있는가?"
이 질문에 대한 답은 현재로선 "제한적으로만"이다.

gvkhna는 모션 디자인 영역에서 실제로 실험한 경험을 공유한다.[^gvkhna]
그리드 기반 프레임 분석, 시각적 모션 지시자 등 다양한 방식을 시도했지만,
결국 특정 애니메이션을 LLM이 추론하게 하는 것보다
"무슨 일이 일어나는지, 타이밍은 어떤지"를 단순하고 정확하게 설명한 텍스트가
가장 좋은 결과를 냈다고 한다.
이는 키프레임 방식의 한계를 우회하는 실용적 전략이지만,
동시에 `crv`의 핵심 가치 제안("LLM이 영상을 본다")을 약화시키는 반례이기도 하다.

### 프리미엄 모델의 핵심 가치가 무료 도구를 잠식한다

README는 무료 버전이 “LLM이 영상을 보게 한다”고 말하고,
crv Pro는 “LLM이 영상을 이해하게 한다”고 구분한다.
그러나 이 경계선은 점점 흐려지고 있다.
GPT-4o와 Claude 3.5 Sonnet은 키프레임만으로도 카메라 무브, 편집 리듬, 감정을
어느 정도 추론할 수 있다. 무료 버전의 출력물을 GPT-4o에 붙여넣고
“카메라 움직임을 분석해줘”라고 하면 crv Pro가 제공한다고 약속하는 것의
상당 부분을 얻는다.

crv Pro의 차별화 포인트는 그 추론을 자동화하고 구조화한다는 것이다.
그러나 이 자동화의 가치는 사용자가 자주 같은 분석을 반복할 때만 정당화된다.
1회성 영상 분석이 목적이라면 무료 버전 + 강력한 LLM 프롬프트로 충분하다.
유료 버전의 가치가 불분명한 구간이 존재한다.

### `--why`와 `--kb` 기능의 통합이 설계 일관성을 해친다

`--why`와 `--kb`는 영상 처리 파이프라인이 아니라 지식 관리 워크플로의 영역이다.
`--why`는 MANIFEST.txt에 분석 렌즈를 추가하는 것이고,
`--kb`는 결과를 노트 폴더에 저장하는 것이다.
이 기능들은 Obsidian, Logseq, 또는 다른 노트 도구와 통합하기 위한 것이지만,
CLI 도구에 지식 관리 레이어를 직접 얹는 것은 단일 책임 원칙에서 벗어난다.

더 큰 문제는 이 기능들이 특정 워크플로를 가정한다는 것이다.
“왜 보는지”를 CLI 플래그로 전달하는 방식은 탐색적 분석에는 어울리지 않는다.
영상을 보기 전에 무엇을 찾을지 명확히 아는 사용자는 드물다.
실제로는 영상을 보고 나서 무엇이 중요한지 파악하는 경우가 많다.

### 씬 체인지 감지의 한계가 명시되지 않는다

`crv`는 FFmpeg의 씬 체인지 필터를 사용하는데, 이 필터는 픽셀 차이 기반이다.
색감 변화 없이 천천히 패닝하는 씬, 스크롤이 멈추지 않는 화면 녹화,
전체 화면이 서서히 밝아지는 페이드인(fade-in) 등에서 씬 체인지를 제대로 감지하지 못할 수 있다.
`--adaptive` 플래그가 이 문제를 부분적으로 완화하지만,
어떤 콘텐츠 유형에서 어떤 한계가 있는지는 README에 명시되지 않는다.

사용자는 `--scene` 임계값을 조정하며 실험해야 한다.
그러나 이 파라미터의 의미를 직관적으로 이해하기 어렵다.
0.30이 무엇을 의미하는지, 내 영상에 어떤 값이 적합한지를
`--report` 없이는 판단하기 어렵다.
중요한 파라미터인데 설명이 얇다.

## 인사이트

### LLM의 멀티모달 확장은 미들웨어 계층을 만든다

LLM이 영상을 직접 처리할 수 없거나 비효율적으로 처리하는 동안,
`crv` 같은 도구가 그 간극을 채우는 미들웨어(middleware) 계층을 형성한다.
이 패턴은 역사적으로 반복됐다.
1990년대 웹 브라우저가 멀티미디어를 직접 처리하지 못하던 시절
QuickTime 플러그인이 그 역할을 했다.
LLM이 영상을 직접 수용하게 되면 이 미들웨어의 필요성은 줄어든다.
그러나 Gemini가 영상을 직접 읽어도 `crv` 같은 전처리가 여전히 유용한 것처럼,
“더 잘 처리하기 위한 미들웨어”는 모델이 발전해도 새로운 형태로 유지된다.

현재 `crv`의 경쟁 우위는 모델 능력 격차에 기반한다.
그 격차가 좁혀질수록 차별화 포인트는 “더 저렴한 컨텍스트”와 “로컬 프라이버시”로
이동해야 한다. 실제로 README가 이 두 가지를 강조하는 것은 이 전략적 전환을
의식한 포지셔닝으로 읽힌다.

### 중복 제거는 영상뿐 아니라 LLM 컨텍스트 설계의 일반 원칙이다

`crv`의 슬라이딩 윈도 중복 제거는 영상 처리 기법이지만,
LLM 컨텍스트 설계의 더 넓은 원칙을 보여준다.
LLM 컨텍스트는 “더 많은 정보”보다 “더 밀도 높은 정보”가 성능을 높인다.
60프레임의 유사한 이미지를 넣는 것과 26프레임의 실제로 다른 이미지를 넣는 것 중
어느 쪽이 더 나은 분석을 이끌어내는지는 자명하다.

이 원칙은 텍스트 컨텍스트에도 적용된다.
긴 회의 트랜스크립트를 그대로 LLM에 넣는 것보다
의미론적으로 유사한 발언을 제거하고 핵심 전환점만 남기는 것이 더 나은 요약을 만든다.
`crv`는 영상에서 이 원칙을 구현한 도구지만,
동일한 아이디어가 문서 처리, 코드 리뷰, 대화 요약 등에도 적용 가능하다.

### 실제 성능은 마케팅과 거리가 있다

noufalibrahim은 실제 사용 사례를 공유한다.[^noufalibrahim]
DOS 게임 컷씬에서 스프라이트 위치를 식별하려 했는데,
스프라이트 시트와 영상 프레임을 제공했음에도 Claude는 15~16개 중 1~2개만 올바르게 식별하고
나머지는 무작위로 배치했다.
이는 `crv`의 문제라기보다 LLM의 정밀한 공간 추론 한계를 드러낸다.
그러나 이 사례는 "LLM이 영상을 본다"는 명제가 모든 시각 분석 과제에
적용 가능하다는 기대를 경계하게 만든다.
도구가 할 수 있는 것과 할 수 없는 것의 경계가 문서화되지 않으면,
사용자는 실망스러운 경험을 하고 도구를 버린다.

### 오픈소스 핵심 + 유료 프리미엄 모델의 지속 가능성

`crv`의 무료/유료 구분은 흥미로운 비즈니스 설계다.
무료 버전은 HN 프런트 페이지까지 올랐고, 그 바이럴(viral) 노출이 유료 버전의
잠재 고객을 만든다. 그러나 이 모델에는 구조적 긴장이 있다.
무료 버전이 충분히 좋으면 유료 전환율이 낮아진다.
유료 버전이 너무 앞서 있으면 무료 버전의 바이럴 효과가 줄어든다.

이 저장소의 README가 유료 기능을 적극적으로 홍보하는 방식은
오픈소스 마케팅 도구 + SaaS/상용 제품의 하이브리드 전략이다.
창업 초기 1회성 가격($19)을 제공하는 것은 얼리 어답터를 빠르게 확보하고
피드백 루프를 만들기 위한 전략이다.
이 모델이 장기적으로 작동하려면 유료 기능의 가치가 LLM 모델 발전보다
빠르게 진화해야 한다.

### 영상 분석 에이전트 시대의 인프라 도구

`crv`가 Claude Code 스킬로 설치 가능하다는 것은 단순한 통합 편의성이 아니다.
에이전트가 영상을 분석해야 하는 워크플로가 현실화되고 있다는 신호다.
유튜브 채널의 콘텐츠를 자동으로 분석하는 에이전트,
경쟁사 광고 영상의 전략을 추적하는 에이전트,
회의 녹화를 분석해 후속 작업을 도출하는 에이전트 모두
`crv` 같은 전처리 레이어를 필요로 한다.

이 관점에서 `crv`는 최종 사용자 도구이기도 하지만
에이전트 인프라 컴포넌트이기도 하다.
에이전트가 자율적으로 영상을 처리할 수 있게 되면,
“어떤 프레임을 뽑을지” 결정하는 로직 자체가 에이전트의 목표에 따라
동적으로 달라져야 한다. `--why` 플래그는 그 방향으로의 첫 걸음이지만,
완전한 에이전트 통합을 위해서는 훨씬 더 정교한 목표-프레임 연결 메커니즘이 필요하다.
