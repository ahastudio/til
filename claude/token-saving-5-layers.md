# 5계층 스택으로 Claude Code 토큰 사용량 90% 이상 줄이기

원문: <https://medium.com/@abdulgafoorabid/d3f8d2488cd6>

한국어 번역: <https://siosio3103.medium.com/432e41806427>

## 요약

Claude Code는 토큰을 빠르게 소진한다. `cargo test` 한 번에 4,823 토큰,
`git diff HEAD~1`에 21,500 토큰, 500줄짜리 파일 하나를 읽으면 컨텍스트
윈도가 금세 차오른다. 자동 컴팩션이 이르게 발동하고, 세션은 30분을 못
버티는 경우가 많다.

저자는 파이프라인의 서로 다른 지점에 5개 레이어를 쌓아 이 문제를 해결했다.
레이어마다 이전 레이어가 놓친 부분을 잡는다.

```text
프롬프트 / 쿼리
    │
레이어 1: CBM (지식 그래프)       → 구조적 탐색에서 99% 절감
    │
레이어 2: context-mode (출력 가상화) → 큰 출력에서 98% 절감
    │
레이어 3: RTK (셸 압축)           → CLI 출력에서 60–90% 절감
    │
레이어 4: Headroom (API 프록시)   → 남은 페이로드에서 47–92% 절감
    │
Caveman (응답 스타일)              → Claude 자신의 출력에서 50–75% 절감
    │
Anthropic API
```

훅이 이 스택을 강제 적용한다. Claude가 우회하지 못하게 막는다. 세션은
30분에서 3시간 이상으로 늘었다.

### 레이어 1: Codebase Memory MCP (CBM)

저장소: [DeusData/codebase-memory-mcp](https://github.com/DeusData/codebase-memory-mcp)

전체 코드베이스를 tree-sitter AST 파싱으로 영속 지식 그래프에 색인한다.
“이 함수를 누가 호출하는가?”를 물을 때 파일을 읽는 대신 그래프에 쿼리한다.
50개 파일(400K 토큰)을 읽는 대신 50 토큰짜리 구조화된 답을 받는다. 실제
측정에서 다섯 개의 구조적 쿼리가 CBM으로는 3,400 토큰, grep 탐색으로는
412,000 토큰을 써서 99.2% 절감되었다.

훅 두 개가 함께 CBM 사용을 강제한다.

- `cbm-code-discovery-gate` (PreToolUse): 소스 파일에 대한 Read/Grep/Glob을
  차단한다. CBM 호출 후 120초 동안만 Read를 허용한다. 설정, 문서, JSON 파일은
  언제나 허용한다.
- `cbm-mcp-marker` (PostToolUse): CBM 도구가 실행될 때 마커 파일을 생성해
  게이트가 잠금을 풀게 한다.

### 레이어 2: context-mode

저장소: [mksglu/context-mode](https://github.com/mksglu/context-mode)

도구 출력을 샌드박스 서브프로세스에서 실행하고, 전체 출력을 로컬 BM25
지식 베이스에 색인한 뒤, 간결한 요약본만 컨텍스트로 돌려준다. 전체 출력은
검색으로 접근할 수 있다.

주요 도구: `ctx_batch_execute`(여러 명령 실행 + 자동 색인 + 요약 반환),
`ctx_search`(BM25 후속 검색), `ctx_fetch_and_index`(URL 가져와 색인).

실제 측정값.

| 작업                     | 원본      | 적용 후 | 절감 |
| ------------------------ | --------- | ------- | ---- |
| Playwright DOM 스냅샷    | 56.2 KB   | 299 B   | 99%  |
| GitHub Issues (20개)     | 58.9 KB   | 1.1 KB  | 98%  |
| Git log (커밋 153개)     | 11.6 KB   | 107 B   | 99%  |
| 전체 세션                | 315 KB    | 5.4 KB  | 98%  |

### 레이어 3: RTK — Rust Token Killer

저장소: [rtk-ai/rtk](https://github.com/rtk-ai/rtk)

CLI 명령 출력을 가로채 컨텍스트에 들어가기 전에 압축하는 Rust 바이너리다.
보일러플레이트 제거, 유사 항목 묶기, 중복 제거. 단일 바이너리, 10ms 미만
오버헤드, 네트워크 호출 없음. context-mode는 큰 출력(20줄 초과)을 샌드박스
처리하고, RTK는 작은-중간 크기 셸 출력(git status, npm install 등)을
그 자리에서 압축한다. 두 레이어가 겹치지 않는다. RTK는 Headroom 안에 번들로
포함된다.

### 레이어 4: Headroom

저장소: [chopratejas/headroom](https://github.com/chopratejas/headroom)

Claude Code와 Anthropic API 사이에 자리 잡는 API 프록시다. 전체 프롬프트를
머신을 떠나기 전에 압축한다. 레이어 1–3이 컨텍스트 윈도에 들어오는 것을
줄인다면, Headroom은 컨텍스트 윈도를 떠나는 것을 압축한다. 대화 기록,
시스템 프롬프트, CLAUDE.md까지 포함한다.

설정은 셸 함수 하나로 끝난다.

```bash
claude() { command headroom wrap claude "$@"; }
```

`CacheAligner` 기능이 프롬프트 프리픽스를 안정화해 Anthropic KV 캐시
히트율을 높인다.

### 레이어 5: Caveman

저장소: [JuliusBrussee/caveman](https://github.com/JuliusBrussee/caveman)

Claude 자신의 장황한 응답도 컨텍스트 윈도에 카운트된다. “물론이죠! 도와드릴
수 있어 기쁩니다.” 같은 문장 하나하나가 낭비된 토큰이다.

Caveman은 Claude가 압축된 스타일로 응답하게 만드는 Claude Code 플러그인이다.
기술적 내용은 보존하고 군더더기만 제거한다. `/caveman:compress`로 CLAUDE.md를
caveman 형식으로 영구 압축하면 세션마다 절감이 곱셈으로 쌓인다.

## 분석

### 훅 강제 적용 패턴

핵심 통찰은 “제안이 아니라 강제”다. CLAUDE.md에 CBM을 쓰라고 지시해도
Claude는 언제든 Read/Grep으로 돌아간다. 물리적 차단이 강제다.
`bash-ban-raw-tools` 훅은 Bash에서 `cat`, `grep`, `find` 같은 원시 명령을
차단해 Claude를 최적화된 도구로 몰아넣는다.

### 자동 핸드오프 시스템

긴 세션은 자동 컴팩션에 부딪히면 상태를 잃는다. 두 훅이 이를 해결한다.

`handoff-precompact` (PreCompact)가 JSONL 트랜스크립트의 마지막 200KB와
엄격한 JSON 스키마로 `claude -p --bare`를 헤드리스 호출해
`docs/handoff-context.md`를 작성한다. `--bare` 플래그가 중요한데, 훅·스킬·
플러그인·MCP·CLAUDE.md를 모두 건너뛰어 자식 세션을 빠르게 시작하고 중첩
훅 캐스케이드를 방지한다.

`handoff-session-resume` (SessionStart, `compact|resume` 매칭)이 다음
세션에서 handoff 파일을 자동으로 컨텍스트에 인라인한다. Read 도구 왕복이
필요 없다.

### 환경 변수 설정

| 변수                              | 값                | 효과                              |
| --------------------------------- | ----------------- | --------------------------------- |
| `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` | `50`              | 50%에서 컴팩션 발동, 핸드오프 여유 |
| `ENABLE_PROMPT_CACHING_1H`        | `1`               | 캐시 TTL 5분 → 1시간              |
| `BASH_MAX_OUTPUT_LENGTH`          | `10000`           | Bash 출력 상한                    |
| `MAX_MCP_OUTPUT_TOKENS`           | `10000`           | MCP 출력 상한                     |
| `CLAUDE_CODE_SUBAGENT_MODEL`      | `claude-sonnet-4-6` | 서브에이전트를 Sonnet으로 고정    |

### 멀티모델 파이프라인

세션이 3시간으로 늘어나면서 멀티모델 파이프라인이 가능해진다. 플랜은
Opus 4.7, 구현은 Opus 4.7, 리뷰는 `/unleash` 서브에이전트 스웜을 병렬로,
최종 리뷰는 다른 벤더(Codex, Gemini 등)로 크로스-모델 검증한다. 같은
패밀리 모델이 놓치는 사각지대를 크로스-벤더 리뷰가 잡아낸다.

## 비평

### 강점

5개 레이어가 파이프라인의 각기 다른 지점을 담당해 겹치지 않는다는 설계가
명확하다. 실제 수치(99.2%, 98%, 60–90%)를 구체적으로 제시해 신뢰성을 높였다.
“말하지 말고 강제하라”는 원칙은 Claude Code를 쓰는 누구에게나 직접 적용
가능한 통찰이다. 하나의 설치 스크립트로 전체 스택을 구성하는 접근도 실용적이다.

### 약점

각 레이어의 절감률이 독립적으로 측정된 것인지, 스택 전체에서 합산된 것인지
명확하지 않다. 레이어들이 곱셈으로 쌓인다고 주장하지만 이를 검증하는 전체
before/after 측정치가 부족하다. Headroom과 RTK 같은 외부 도구에 대한 의존성이
크고, 특히 Headroom은 API 프록시이므로 보안 검토가 필요하다. CLAUDE.md 전체와
설정 파일이 저장소에 있다고 언급하지만 본문만으로 완전한 재현이 어렵다.

### 적용 맥락

이 스택은 비용을 의식하는 헤비 Claude Code 사용자에게 최적화되어 있다.
소규모 프로젝트나 간헐적 사용자에게는 설정 복잡도가 이득을 초과할 수 있다.
CBM 게이트처럼 Claude의 자율성을 제한하는 훅은 예상치 못한 상황에서 워크플로를
막을 수 있다는 점도 고려해야 한다.

## 인사이트

### 토큰 비용은 행동 설계 문제다

Claude가 토큰을 낭비하는 이유는 모델 자체의 문제가 아니라 가장 저항이 적은
길을 따라가기 때문이다. `grep`으로 파일을 탐색하는 것이 허용되어 있으면,
CBM을 쓰라는 지시가 있어도 Claude는 grep을 선택한다. 이것은 인간 조직의
인센티브 구조와 동일한 원리다. 규칙을 만드는 것만으로는 행동이 바뀌지 않는다.
비효율적인 길을 물리적으로 막고 효율적인 길을 유일한 선택지로 만들어야 한다.

이 통찰은 AI 에이전트 설계 전반에 적용된다. 프롬프트 엔지니어링의 한계는
Claude가 지시를 “이해하지 못하는” 것이 아니라, 지시를 따르는 것보다 다른
경로가 더 쉬울 때 그 경로를 택한다는 데 있다. 훅 기반 강제 적용은 이 문제를
구조적으로 해결한다. 말하지 않고 설계한다.

이는 소프트웨어 아키텍처의 “pit of success” 개념과 동일하다. 올바른 방법이
가장 쉬운 방법이 되도록 시스템을 설계하는 것이다. Claude Code에서 이 원칙을
적용하면, CLAUDE.md의 글줄 수보다 훅 한 줄이 훨씬 강력하다.

### 압축의 복합 효과: 각 레이어가 다른 레이어의 효과를 증폭한다

5개 레이어가 단순히 더해지는 것이 아니라 곱해진다는 점이 중요하다. CBM이
파일 읽기를 줄이면 context-mode가 처리할 출력 자체가 줄어든다. RTK가 셸
출력을 압축하면 Headroom이 전송할 페이로드가 줄어든다. Caveman이 응답을
줄이면 그 짧은 응답이 다음 턴의 대화 기록으로 누적될 때 절감이 이어진다.

이 복합 효과는 /caveman:compress로 CLAUDE.md를 압축하는 것에서 가장 명확하게
드러난다. CLAUDE.md는 매 세션마다 로드되므로, 한 번의 압축이 모든 미래
세션에 영향을 준다. 절감이 시간 축으로도 복리처럼 쌓인다.

소프트웨어 성능 최적화에서도 같은 패턴이 반복된다. 단일 병목을 해결해도
다른 병목이 드러날 뿐이다. 시스템 전체를 동시에 개선해야 복합 효과가 나타난다.
이 아티클의 접근이 단일 도구 소개가 아니라 파이프라인 전체를 다루는 이유가
여기에 있다.

### 세션 수명과 작업 품질의 상관관계

세션이 30분에서 3시간으로 늘어나는 것은 단순한 비용 절감 이상의 의미를 갖는다.
Claude Code에서 긴 세션은 연속적인 작업 컨텍스트를 유지한다는 뜻이다. 30분
세션에서는 매번 컨텍스트를 재구성해야 하고, 이 과정에서 미묘한 맥락이 손실된다.

저자의 멀티모델 파이프라인(플랜→구현→스웜 리뷰→크로스-벤더 리뷰→E2E)은 긴
세션이 없으면 실행 자체가 불가능하다. 컨텍스트 효율화는 단순히 비용을 아끼는
것이 아니라 이전에는 불가능했던 워크플로를 가능하게 만든다. 제약이 제거되면
새로운 작업 방식이 열린다.

자동 핸드오프 시스템도 같은 관점에서 볼 수 있다. 컴팩션이 발생해도 구조화된
핸드오프 문서가 다음 세션에 자동으로 인라인되면, Claude는 “왜 이 결정을
내렸는가”를 포함한 전체 맥락을 이어받는다. 세션 경계가 사라지는 효과다.
이것은 인간 팀에서 교대 근무 시 인수인계 문서를 작성하는 것과 동일한 원리이며,
AI 에이전트에서도 같은 중요성을 갖는다.
