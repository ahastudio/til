# 작업별 맞춤 하네스: Claude Code의 동적 워크플로우

원문: <https://claude.com/blog/a-harness-for-every-task-dynamic-workflows-in-claude-code>

## 요약

Claude Code가 특정 작업에 맞게 최적화된 멀티 에이전트 하네스를 자율적으로 생성하는 동적 워크플로우 기능을 소개한다.
핵심 명제는 “Claude가 이제 작업에 맞게 즉석에서 자신만의 하네스를 직접 작성할 수 있다”는 것이다.
워크플로우는 JavaScript 파일을 실행해 하위 에이전트를 생성하고 조율하며, 각 에이전트는 선택된 모델과 워크트리 격리 설정을 갖는다.

기능이 필요한 배경은 단일 컨텍스트 Claude Code의 세 가지 실패 모드에 있다.
첫째 에이전트적 게으름(agentic laziness)으로, 부분적 진행 후 조기에 작업을 완료했다고 선언하는 문제다.
둘째 자기선호 편향(self-preferential bias)으로, 객관적 평가보다 자신의 출력을 선호하는 문제다.
셋째 목표 표류(goal drift)로, 여러 턴에 걸쳐 원래 작업의 충실도를 잃는 문제다.
워크플로우는 작업을 격리된 컨텍스트들로 분산시켜 이 문제들을 구조적으로 해결한다.

일곱 가지 주요 오케스트레이션 패턴을 제시한다.
유형에 따라 라우팅하는 분류-행동(classify-and-act), 병렬화 후 합성하는 팬아웃-합성(fan-out-and-synthesize), 독립 에이전트가 루브릭으로 출력을 검증하는 적대적 검증(adversarial verification), 복수 솔루션을 생성 후 품질로 필터링하는 생성-필터(generate-and-filter), 최선의 접근을 경쟁시키는 토너먼트(tournament), 완료 조건까지 반복하는 완료까지-루프(loop-until-done), 전용 에이전트가 특정 규칙을 검사하는 구조적 검증(structured verification)이다.

실제 적용 사례로는 대규모 코드 마이그레이션과 리팩터링, 적대적 소스 검증을 통한 심층 리서치, 비교 판단을 통한 대용량 데이터셋 정렬, 분리된 증거 소스에 걸친 근본 원인 분석, 지원 큐 분류 및 중복 제거, 태스크 복잡도에 따른 모델 라우팅이 있다.
단, 워크플로우는 표준 접근보다 토큰을 훨씬 많이 소비하므로 일상적인 코딩보다는 복잡하고 고가치 작업에 적합하다.

## 분석

### 멀티 에이전트의 구조적 해법

단일 에이전트의 세 가지 실패 모드가 모두 컨텍스트 오염에서 비롯된다는 진단이 날카롭다.
에이전트가 오래 실행될수록 초기 지시와의 거리가 멀어지고, 자신이 생성한 코드를 평가할 때 편향이 생긴다.
격리된 독립 컨텍스트로 분산하는 것은 이 오염을 구조적으로 차단한다.
“자기선호 편향”을 적대적 검증으로 해결하는 것은 특히 우아하다. 코드를 쓴 에이전트와 검토하는 에이전트가 다른 컨텍스트에서 시작하므로, 검토자는 생성자의 추론 경로에 물들지 않는다.

팬아웃-합성 패턴은 코드 마이그레이션처럼 서로 의존성이 적은 작업을 병렬화할 때 특히 효과적이다.
수백 개의 파일을 독립적인 에이전트들이 동시에 처리하고, 마지막에 합성하는 방식은 단일 에이전트로는 불가능한 처리량을 가능하게 한다.

### 워크플로우 생성 자체의 재귀성

흥미로운 점은 워크플로우 자체를 Claude가 작성한다는 것이다.
어떤 패턴을 써야 하는지, 몇 개의 에이전트가 필요한지, 어떤 모델을 각 하위 작업에 배정할지를 오케스트레이터가 결정한다.
이것은 메타 수준의 코딩이다. 에이전트가 에이전트 시스템을 설계한다.
이 재귀성이 강력하지만, 동시에 오케스트레이터의 판단 품질에 전체 시스템이 의존한다는 취약성도 만든다.

## 비평

### 토큰 비용의 현실적 제약

“표준 접근보다 토큰을 훨씬 많이 소비한다”는 경고가 사용 범위를 크게 제한한다.
복잡한 마이그레이션이나 심층 리서치처럼 명확히 고가치인 작업은 소수다.
일상적인 코딩 작업에서 워크플로우를 쓰면 비용이 기하급수적으로 늘어난다.
대부분의 개발자에게 “언제 워크플로우를 써야 하는가”의 판단이 어렵고, 과도하게 사용하면 비용이 폭발할 수 있다.

### 오케스트레이션 로직의 투명성

워크플로우가 JavaScript로 작성된다는 것은 원칙적으로 검사 가능하다.
하지만 오케스트레이터가 어떤 워크플로우를 생성했는지, 각 에이전트에 어떤 지시를 내렸는지를 사용자가 실시간으로 이해하기 어렵다.
“나는 내가 볼 수 없는 것을 신뢰할 수 없다”는 원칙이 여기서도 적용된다.
워크플로우 실행의 관찰가능성이 더 발전해야 할 영역이다.

## 인사이트

### 에이전트 오케스트레이션의 설계 언어

이 글이 제시하는 일곱 패턴은 사실 분산 시스템의 설계 패턴과 닮아 있다.
팬아웃은 맵리듀스이고, 적대적 검증은 합의 알고리즘이며, 토너먼트는 경쟁적 실행이다.
에이전트 오케스트레이션이 무르익으면서, 분산 시스템 엔지니어링의 어휘가 에이전트 시스템 설계에도 그대로 적용된다는 것이 명확해지고 있다.
이것은 에이전트 시스템의 신뢰성, 장애 처리, 확장성 문제도 분산 시스템의 방식으로 접근해야 한다는 것을 시사한다.

### 코딩에서 메타-코딩으로

소프트웨어 개발의 진화를 한 단계씩 올라가면서 보면 패턴이 보인다.
기계어 → 어셈블리 → 고수준 언어 → 프레임워크 → AI 보조 코딩.
동적 워크플로우는 다음 단계다. AI가 AI 시스템을 설계하는 메타 코딩이다.
개발자의 역할이 코드를 직접 작성하는 것에서, 문제를 올바르게 분해하고 에이전트들에게 올바른 지시를 내리는 것으로 이동한다.

이 전환은 소프트웨어 엔지니어링의 핵심 역량을 재정의한다.
구현 능력보다 아키텍처 사고, 문제 분해 능력, 에이전트 오케스트레이션 설계가 더 중요해진다.
“어떻게 코딩하는가”보다 “어떻게 에이전트들을 조율하는가”가 시니어 엔지니어의 차별점이 될 수 있다.

### 복잡성의 이전

단일 에이전트 방식에서 멀티 에이전트 워크플로우로의 이동은 복잡성을 없애는 것이 아니라 이전(shift)한다.
개별 태스크의 복잡성은 줄어들지만, 에이전트 간 협력의 복잡성, 오케스트레이션 설계의 복잡성, 디버깅의 복잡성이 증가한다.
10개의 에이전트가 만들어낸 결과물에서 버그가 발생했을 때, 어느 에이전트의 어느 결정이 원인인지를 추적하는 것은 단일 에이전트보다 훨씬 어렵다.

이 새로운 종류의 복잡성은 기존 디버깅 도구로 다루기 어렵다.
분산 추적(distributed tracing)과 에이전트 실행 이력 분석을 위한 새로운 도구 범주가 필요하다.
클라우드 컴퓨팅이 분산 시스템의 복잡성을 숨기면서도 관찰가능성 도구를 진화시킨 것처럼, 에이전트 오케스트레이션도 복잡성을 관리하는 새로운 추상화가 등장할 것이다.
