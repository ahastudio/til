# Claude Code 다중 계정 설정

원문: <https://medium.com/@buwanekasumanasekara/f8769a36d1b1>

## 요약

이 글은 하나의 로컬 머신에서 여러 Claude Code 계정(개인용, 업무용 등)을 동시에
사용하는 방법을 다룬다. 핵심 메커니즘은 `CLAUDE_CONFIG_DIR` 환경 변수를 활용하여
계정별로 독립된 설정 디렉토리를 지정하는 것이다.

설정 과정은 다섯 단계로 구성된다. 먼저 `~/.claude-account1`,
`~/.claude-account2`와 같은 별도의 설정 디렉토리를 생성한다. 그 다음
`type -a claude`로 Claude 바이너리 경로를 확인하고, 셸 설정 파일에 각 계정별
alias를 등록한다. 각 alias는 `CLAUDE_CONFIG_DIR` 환경 변수를 다른 디렉토리로
지정한 채 Claude를 실행하는 방식이다.

alias 등록 후 각 계정별로 `/login` 명령어를 통해 개별 인증을 완료하면 된다. 기본
`claude` 명령어를 리마인더 메시지로 재정의하여, 항상 특정 계정 alias를
사용하도록 유도하는 팁도 제시한다. 프로젝트별 `.env` 파일에
`CLAUDE_CONFIG_DIR`을 설정하여 프로젝트 단위로 기본 계정을 지정하는 방법도
소개한다.

## 분석

### 환경 변수 기반 설정 분리의 구조

`CLAUDE_CONFIG_DIR` 환경 변수는 Claude Code가 세션 데이터와 설정을 저장하는
디렉토리를 결정한다. 기본값은 `~/.claude`인데, 이 값을 변경하면 인증 토큰, 세션
기록, 프로젝트 설정 등이 모두 다른 경로에 저장된다. 이는 Docker의
`DOCKER_CONFIG`나 AWS CLI의 `AWS_PROFILE`과 동일한 패턴으로, Unix 도구들이 널리
사용하는 설정 분리 관행을 따른다.

### 셸 alias를 통한 사용자 경험 최적화

환경 변수를 매번 직접 입력하는 대신 셸 alias로 감싸는 접근은 실용적이다.
`claude-personal`, `claude-work`처럼 의미 있는 이름을 붙이면 어떤 계정으로
작업하는지 즉시 파악할 수 있다. 기본 `claude` 명령어를 가이드 메시지로 덮어쓰는
기법은 실수로 잘못된 계정을 사용하는 것을 방지한다.

### NVM 사용자를 위한 경로 관리 주의사항

Node.js 버전 매니저(NVM)를 사용하는 경우, Node 버전을 변경하면 Claude 바이너리
경로가 달라질 수 있다. 이 점을 명시적으로 경고한 것은 실전에서 자주 발생하는
문제를 짚은 것이다. 다만 이 문제의 해결책까지는 제시하지 않았다.

## 비평

### 강점: 즉시 적용 가능한 실용적 가이드

5단계 과정이 명확하고, 각 단계에서 실행할 명령어가 구체적으로 제시된다.
복사-붙여넣기만으로 설정을 완료할 수 있는 수준의 완성도를 갖추고 있다. Pro Tips
섹션에서 프로젝트별 `.env` 파일 활용이나 의미 있는 alias 네이밍 같은 실전 조언을
추가한 것도 좋다.

### 한계: 보안 고려사항의 부재

여러 계정의 인증 토큰이 같은 머신에 저장되는 상황에서의 보안 고려가 전혀 없다.
디렉토리 권한 설정(`chmod`), 공유 머신에서의 주의사항, 토큰 유출 시 대응 방법
등이 다뤄지지 않았다. 특히 개인 계정과 업무 계정을 같은 머신에서 사용할 때, 업무
환경의 보안 정책과 충돌할 가능성에 대한 언급이 없다.

### 한계: 대안 접근법과의 비교 부족

Claude Code가 공식적으로 다중 계정을 지원하는 방법이 있는지, 혹은 향후 지원
계획이 있는지에 대한 조사가 없다. `/logout`과 `/login`을 번갈아 사용하는 기본
방식과 비교하여 이 방법이 실제로 얼마나 시간을 절약하는지 정량적 비교도 없다.
또한 셸 함수(function)를 사용하여 alias보다 유연한 래퍼를 만드는 방법이나,
direnv와 같은 도구를 활용하는 대안도 다루지 않았다.

## 인사이트

### Unix 철학의 승리 — 환경 변수라는 40년된 인터페이스의 현재성

`CLAUDE_CONFIG_DIR` 환경 변수 하나로 다중 계정 문제를 해결하는 이 패턴은, 1979년
Version 7 Unix에서 도입된 환경 변수 메커니즘이 2026년의 AI 도구에서도 여전히
가장 효과적인 설정 분리 수단임을 보여준다. AWS의 `AWS_PROFILE`, Docker의
`DOCKER_CONFIG`, Kubernetes의 `KUBECONFIG` — 현대의 거의 모든 CLI 도구가 동일한
패턴을 따른다.

이것이 작동하는 근본적인 이유는 Unix의 프로세스 모델에 있다. 환경 변수는
프로세스 생성 시 부모로부터 자식에게 상속되며, 자식 프로세스에서의 변경은
부모에게 전파되지 않는다. 이 격리 특성이 계정별 세션 분리를 자연스럽게 만든다.
별도의 가상화나 컨테이너 없이도, 환경 변수 하나로 완전한 상태 격리를 달성한다.

이 패턴의 지속성은 소프트웨어 설계에 중요한 교훈을 준다. 새로운 도구를 만들 때
기존의 Unix 관행을 따르면, 사용자가 이미 알고 있는 패턴을 활용하여 공식적으로
지원하지 않는 기능도 스스로 구현할 수 있다. Claude Code 팀이
`CLAUDE_CONFIG_DIR`을 문서화했는지와 무관하게, 이 환경 변수가 존재한다는 것
자체가 사용자에게 확장 가능성을 열어둔 설계적 결정이다.

### "비공식 멀티테넌시"가 드러내는 제품 설계의 빈 공간

사용자가 셸 alias와 환경 변수로 다중 계정을 직접 구현해야 한다는 사실은, Claude
Code가 아직 다중 계정 시나리오를 1급 시민(first-class citizen)으로 다루지 않고
있음을 의미한다. 이는 제품 성숙도의 자연스러운 단계이기도 하지만, 동시에 실제
사용 패턴과 제품 설계 사이의 간극을 보여준다.

프리랜서 개발자가 여러 클라이언트의 프로젝트를 작업하거나, 한 사람이 개인
프로젝트와 회사 프로젝트를 넘나드는 상황은 매우 흔하다. IDE들은 이미 프로필
기능(VS Code의 Profile, JetBrains의 Settings Sync)을 통해 이 문제를 해결했다.
Claude Code가 성장하면서 `claude profile switch work`와 같은 내장 명령어나,
디렉토리 기반 자동 계정 전환 기능이 필수적으로 요구될 것이다.

더 넓은 관점에서 보면, 이는 AI 도구가 "개인 도구"에서 "전문 도구"로 전환되는
과정에서 반드시 거치는 단계다. 개인 사용자를 위해 설계된 도구가 전문가
워크플로우에 편입될 때, 계정 분리, 감사 로그(audit log), 접근 제어 같은
엔터프라이즈 요구사항이 부상한다. 이 글이 존재한다는 것 자체가 Claude Code가 그
전환점에 도달했다는 신호다.

### Alias 패턴의 확장 — 계정 분리를 넘어서는 환경 프리셋

이 글에서 제시한 alias 패턴은 다중 계정에만 한정되지 않는다. 같은 구조를
확장하면 동일한 계정 내에서도 용도별 환경을 분리할 수 있다. 예를 들어 하나의
alias는 `CLAUDE_MODEL`을 Haiku로 설정하여 빠른 탐색용으로, 다른 alias는 Opus로
설정하여 깊은 분석용으로 사용할 수 있다.

이 확장은 "도구의 설정을 코드로 관리한다"는 Infrastructure as Code의 개인 도구
버전이다. 셸 설정 파일에 alias를 정의하는 것은 사실상 개발 환경의 선언적
설정이며, 이를 dotfiles 저장소에 보관하면 머신 간 이동 시에도 동일한 개발 환경을
재현할 수 있다.

더 나아가 direnv 같은 도구와 결합하면, 특정 디렉토리에 진입할 때 자동으로 적절한
Claude 계정과 모델이 선택되는 완전 자동화된 환경 전환이 가능하다. 프로젝트
루트에 `.envrc` 파일을 두고 `export CLAUDE_CONFIG_DIR=~/.claude-work`를
설정하면, 해당 프로젝트 디렉토리에 들어가는 순간 자동으로 업무 계정으로
전환된다. 이런 자동화는 수동 alias 실행보다 더 안전하고 편리하며, "올바른 계정을
사용하고 있는지" 확인하는 인지적 부담을 완전히 제거한다.
