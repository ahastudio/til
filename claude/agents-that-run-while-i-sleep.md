# 잠자는 동안 돌아가는 에이전트를 만들고 있다

출처: [I'm Building Agents That Run While I Sleep]
(https://www.claudecodecamp.com/p/i-m-building-agents-that-run-while-i-sleep)
— Abhishek Ray, 2026-03-11

## 한 줄 요약

자율 코딩 에이전트의 결과물을 신뢰하려면,
**프롬프트가 아니라 인수 조건(Acceptance Criteria)을
먼저 써야 한다**.

## 문제 인식

Gastown 같은 도구로 에이전트가 밤새 코드를 작성한다.
변경 사항은 읽어보지 못한 브랜치에 쌓인다.
이 코드가 실제로 올바른지 확인할 방법이 없다.

100명 이상의 엔지니어와 워크숍을 하면서 같은 문제를
확인했다.
Claude를 도입한 팀은 주 10개 PR이 40~50개로 늘었지만,
코드 리뷰에 쏟는 시간도 함께 증가했다.
시스템이 자율화될수록 문제가 심화된다.
어느 순간부터 diff를 읽지 않고
배포를 지켜보며 기도하게 된다.

## 기존 해법이 안 되는 이유

### 리뷰어 추가 고용

충분히 빠르게 채용할 수 없고,
시니어에게 AI 생성 코드 리뷰만 시키는 건 낭비다.

### AI가 스스로 작성한 테스트

Claude가 작성한 코드를 Claude가 테스트하면
"자기 축하 기계(self-congratulation machine)"가 된다.
테스트는 Claude가 해석한 요구사항을 검증할 뿐,
사용자가 실제로 원한 것을 검증하지 않는다.

### AI 간 교차 리뷰

같은 모델 기반이면 "제2의 눈"이 아니다.
같은 맹점을 공유한다.

## TDD가 맞았던 것

TDD의 핵심: 테스트를 먼저 쓰고, 코드를 작성하고,
테스트가 통과하면 멈춘다.

AI 시대에는 속도 병목이 "코드 작성"에서
"결과물이 맞는지 판단"으로 이동했다.
TDD가 정확히 이 문제를 위해 설계된 방법론이다.

차이점: 유닛 테스트 대신 **자연어 인수 조건**을 쓴다.
구현 방식이 아니라 기능이 뭘 해야 하는지 기술한다.

```text
사용자는 이메일과 비밀번호로 인증할 수 있다.
틀린 자격 증명이면 'Invalid email or password'가
표시된다.
성공하면 /dashboard로 이동한다.
세션 토큰은 24시간 후 만료된다.
```

코드 에디터를 열기 전에 쓸 수 있다.
에이전트가 구현하고, 별도 시스템이 검증한다.

## 실제 워크플로우

### 프론트엔드: Playwright 브라우저 에이전트

스펙 파일에서 인수 조건(AC)을 생성한다.
각 AC는 통과/실패가 명확하다.

```text
AC-1: 성공 로그인
- /login에서 유효한 자격 증명 → /dashboard 리다이렉트
- 세션 쿠키 설정됨

AC-2: 틀린 비밀번호 오류
- "Invalid email or password" 정확히 표시
- /login에 머무름

AC-3: 빈 필드 유효성 검사
- 필드가 비면 제출 버튼 비활성화 또는 인라인 오류

AC-4: 속도 제한
- 5회 실패 → 60초 차단, 대기 시간 메시지 표시
```

에이전트가 기능을 구현하면 Playwright 브라우저
에이전트가 각 AC를 검증하고, 스크린샷을 찍고,
기준별 판정 보고서를 생성한다.

### 백엔드: 브라우저 없이 동일 패턴

관찰 가능한 API 동작(상태 코드, 응답 헤더,
오류 메시지)을 명세하고 curl로 검증한다.

### 한계

스펙 자체가 틀리면 검증을 통과해도 기능이 틀리다.
Playwright가 잡는 것은 통합 실패, 렌더링 버그,
실제 브라우저에서 깨지는 동작이다.
"검증 완료"보다 좁은 주장이지만,
코드 리뷰가 안정적으로 잡던 것보다는 넓다.

## 구현: opslane/verify

[github.com/opslane/verify](https://github.com/opslane/verify)
— Claude Skill로 구현.
`claude -p`(헤드리스 모드) + Playwright MCP.
커스텀 백엔드 없음. Claude OAuth 토큰만 필요.

### 4단계 파이프라인

| 단계             | 모델    | 역할                       |
| ---------------- | ------- | -------------------------- |
| Pre-flight       | 없음    | 순수 bash 점검.            |
|                  |         | 서버·세션·스펙 존재 확인.  |
|                  |         | 토큰 소비 전 빠른 실패.    |
| Planner          | Opus    | 스펙 + 변경 파일 읽기.     |
|                  |         | 검증 방법·셀렉터 결정.     |
| Browser Agents   | Sonnet  | AC당 1 에이전트, 병렬 실행.|
|                  |         | Sonnet은 Opus 대비         |
|                  |         | 3~4배 저렴.                |
| Judge            | Opus    | 전체 증거를 읽고           |
|                  |         | AC별 판정: pass / fail /   |
|                  |         | needs-human-review.        |

**워크플로우 요약:**
인수 조건 작성 → 에이전트 구현 → 검증 실행 →
**실패만 리뷰**. diff가 아니라 실패를 본다.

## 인사이트

### 1. 신뢰의 전제 조건은 "완료의 정의"

> 에이전트가 만든 결과물을 신뢰하려면,
> 시작 전에 "완료"가 어떤 모습인지
> 알려줬어야 한다.

인수 조건 없이는 출력을 읽고
맞기를 바라는 것밖에 할 수 없다.

### 2. 인수 조건 작성 > 프롬프트 작성

프롬프트는 "뭘 만들어라"다.
인수 조건은 "이걸 만족해야 완성이다"다.
후자가 더 어렵다.
엣지 케이스를 코드를 보기 전에 생각해야 하기
때문이다.
엔지니어가 TDD를 거부한 이유와 정확히 같다.
**시작이 느리게 느껴진다.**

### 3. AI 자체 검증은 구조적으로 실패한다

작성자와 검증자가 같은 모델이면
같은 오해를 공유한다.
분리해야 한다: 사람이 기준을 쓰고,
기계가 기준에 맞춰 구현하고,
별도 기계가 기준 대비 결과를 검증한다.

### 4. 비용 최적화: 역할별 모델 분리

모든 단계에 Opus를 쓸 필요 없다.
브라우저 클릭은 Sonnet으로 충분하고
3~4배 절약된다.
판단이 필요한 곳만 Opus를 쓴다.

### 5. 리뷰 패러다임의 전환: diff → 실패

자율 에이전트 시대에 모든 diff를 읽는 건
확장되지 않는다.
**실패한 인수 조건만 리뷰**하는 것이
유일하게 확장 가능한 검증 방식이다.

### 6. 스펙이 틀리면 모든 게 무너진다

검증 파이프라인은 "스펙이 올바르다"는
가정 위에 서 있다.
결국 사람의 핵심 역할은
**올바른 스펙을 작성하는 것**으로 수렴한다.
