# Claude Code 작동 원리: 에이전틱 루프와 컨텍스트 시스템

> 원문: <https://www.mintlify.com/VineeTagarwaL-code/claude-code/concepts/how-it-works>

## 요약

Claude Code는 터미널 기반 코딩 에이전트로, 연속적인 루프를 통해 작동한다. 핵심 루프는 6단계로 구성된다. 사용자 입력, 컨텍스트 조립, 모델 추론, 권한 평가, 도구 실행, 루프 계속이다. 모든 처리는 사용자의 터미널 프로세스에서 실행되며, 원격 실행 서버는 없다.

컨텍스트 로딩은 시스템 컨텍스트(git 정보, 캐시 브레이킹)와 사용자 컨텍스트(CLAUDE.md 메모리 파일, 날짜 주입)의 두 블록으로 나뉜다. 4단계 계층 구조의 CLAUDE.md 파일이 로드되며, lodash 메모이제이션으로 대화 중 반복 계산을 방지한다.

도구 실행 모델은 allow(즉시 실행), ask(사용자 확인), deny(거부)의 3가지 권한 결과를 갖는다. 읽기 전용 도구(Read, Glob, Grep)는 모든 모드에서 자동 승인된다. 서브 에이전트는 Task 도구를 통해 생성되며, 각각 격리된 루프와 제한된 도구 집합으로 실행된다.

대화는 `~/.claude/`에 JSON 트랜스크립트로 저장되고, `--resume`으로 재개할 수 있다. 긴 대화는 주기적 압축(compaction)을 거치며, 원본 트랜스크립트는 보존된다.

## 분석

이 문서는 Claude Code의 아키텍처를 외부 개발자 관점에서 정리한 비공식 문서다. Mintlify 기반으로 호스팅되어 있으며, 유출된 소스 코드를 기반으로 작성된 것으로 보인다. 공식 문서에서는 다루지 않는 내부 구현 세부사항(캐시 브레이킹, 메모이제이션 전략, 압축 메커니즘)을 포함한다.

아키텍처의 핵심 설계 결정은 "로컬 실행"이다. 원격 서버 없이 터미널에서 모든 것을 처리하는 선택은 보안(코드가 외부로 나가지 않음)과 지연시간(네트워크 왕복 없음)에서 이점을 제공하지만, 컴퓨팅 자원이 사용자 머신에 제한된다는 트레이드오프가 있다.

`query.ts`를 통한 API 통신 계층이 스트리밍, 도구 디스패치, 토큰 예산, 자동 압축을 모두 처리한다는 점은 이 파일이 시스템의 핵심 병목이자 복잡성 집중 지점임을 시사한다.

## 비평

**강점:** 6단계 에이전틱 루프를 명확하게 도식화한 점이 좋다. 각 단계의 역할과 다음 단계로의 전환 조건이 명시적이다. 인터랙티브 모드와 비인터랙티브 모드의 구분, 서브 에이전트의 격리 모델 등 운영 모드별 차이점도 잘 정리되어 있다.

**약점:** 컨텍스트 조립 과정에서 git 상태를 "2,000자까지" 포함한다는 세부사항은 있지만, 이 제한이 대규모 저장소에서 어떤 영향을 미치는지에 대한 논의가 없다. 압축(compaction) 메커니즘이 "이전 메시지를 요약"한다고만 되어 있는데, 요약 과정에서의 정보 손실이 에이전트 성능에 미치는 영향은 다루지 않는다. 또한 권한 모델의 보안 함의 — 예를 들어 `allow` 모드에서의 위험성 — 에 대한 분석이 부재하다.

문서 자체가 유출된 소스를 기반으로 한 것이므로, 현재 코드와의 정합성이 보장되지 않는다는 점도 주의가 필요하다.

## 인사이트

### 에이전틱 루프의 수렴: 모든 AI 코딩 도구가 같은 구조에 도달하는 이유

Claude Code의 6단계 루프(입력 → 컨텍스트 → 추론 → 권한 → 실행 → 반복)는 GitHub Copilot Workspace, Cursor Agent, OpenAI Codex 등 거의 모든 AI 코딩 도구가 수렴하는 패턴이다. 이 수렴은 우연이 아니라 구조적 필연이다.

자율 에이전트가 외부 세계와 상호작용하려면 관찰-판단-행동(OODA) 루프가 필요하고, 이를 소프트웨어 개발 맥락에 매핑하면 자연스럽게 "코드 읽기 → 변경 계획 → 도구 실행 → 결과 확인"이 된다. 여기에 안전성을 위한 권한 게이트를 추가하면 Claude Code의 구조가 도출된다.

흥미로운 점은 이 구조가 전통적인 REPL(Read-Eval-Print Loop)의 확장이라는 것이다. REPL이 사람의 입력을 받아 평가하는 루프라면, 에이전틱 루프는 LLM의 출력을 받아 도구로 평가하는 루프다. 차이는 루프의 주체가 사람에서 모델로 이동했다는 것이고, 이 이동이 권한 평가라는 새로운 단계를 필수적으로 만들었다. 사람은 자신의 행동에 대한 암묵적 권한을 가지지만, AI 에이전트는 명시적 권한 부여가 필요하기 때문이다.

이 수렴 패턴은 AI 코딩 도구의 차별화 포인트가 루프 구조 자체가 아니라, 각 단계의 품질 — 특히 컨텍스트 조립의 정확성과 모델 추론의 품질 — 에 있음을 시사한다.

### 로컬 실행의 전략적 의미: 신뢰와 통제의 교환

"루프가 전적으로 터미널 프로세스에서 실행된다"는 설계 선택은 기술적 결정인 동시에 신뢰 모델의 선언이다. 코드가 사용자의 머신을 떠나지 않으므로, 기업 고객의 보안 심사를 통과하기 쉽고, 사용자는 자신의 코드가 학습에 사용될 걱정을 덜 수 있다.

그러나 이 선택에는 숨겨진 비용이 있다. 로컬 실행은 서버 측 관찰 가능성(observability)을 포기한다는 뜻이다. Anthropic은 사용자가 어떤 파일을 읽고, 어떤 명령을 실행하고, 어떤 오류를 만나는지에 대한 세밀한 텔레메트리를 수집하기 어렵다. 이는 제품 개선을 위한 데이터 수집과 사용자 프라이버시 사이의 근본적 긴장을 반영한다.

소스 유출에서 드러난 자동 압축의 "일 25만 낭비된 API 호출" 문제는 이 관찰 가능성 부족의 직접적 결과일 수 있다. BigQuery 로그로 문제를 발견했다는 코드 주석은, API 호출 패턴이 유일하게 서버에서 관찰 가능한 시그널이었음을 시사한다. 로컬 실행 아키텍처에서의 디버깅은 본질적으로 더 어렵다.

### 4단계 CLAUDE.md 계층: 컨텍스트의 연방제

CLAUDE.md 파일이 4단계 계층에서 로드된다는 것은 사용자, 프로젝트, 디렉토리, 세션 수준의 컨텍스트가 합성된다는 뜻이다. 이는 Unix의 설정 파일 계층(시스템 → 사용자 → 프로젝트 → 로컬)과 정확히 같은 패턴이다.

이 설계가 해결하는 문제는 "동일한 도구가 다른 맥락에서 다르게 작동해야 한다"는 것이다. Python 프로젝트와 Rust 프로젝트에서 같은 "코드 리뷰" 명령이 다른 규칙을 적용해야 하고, 같은 프로젝트 내에서도 프론트엔드와 백엔드 디렉토리에서 다른 컨벤션을 따라야 할 수 있다.

더 넓게 보면 이것은 AI 에이전트의 "기억" 문제에 대한 하나의 해법이다. LLM은 컨텍스트 윈도우 내에서만 "기억"을 갖지만, 파일 시스템의 계층적 설정 파일은 영속적이고 구조화된 기억을 제공한다. 이는 RAG(Retrieval-Augmented Generation)의 단순한 변형이지만, 검색 대신 파일 시스템의 디렉토리 구조를 인덱스로 사용한다는 점에서 더 예측 가능하고 사용자가 직접 편집할 수 있다. AI 에이전트의 기억 시스템이 궁극적으로 어떤 형태를 취할지는 아직 열린 질문이지만, Claude Code의 접근법은 "사용자가 통제할 수 있는 기억"이라는 중요한 설계 원칙을 보여준다.
