코딩 경험이 전혀 없는 1인 컨설턴트가 Claude Code로
36시간 만에 개인 비서(Chief of Staff) 시스템을 구축한 사례다.
매일 아침 30~45분 걸리던 운영 업무를 한 자릿수 분으로 줄였고,
연간 130~195시간을 절약한다.
200만 뷰를 넘기며 "agentic systems의 가장 좋은
실제 사례 중 하나"라는 평가를 받았다.

코딩 자격증이나 경험은 **전무**하다.

## 4계층 아키텍처

시스템은 순차적으로 쌓이는 4개 레이어로 구성된다.
각 레이어는 이전 레이어의 출력을 입력으로 받지만,
독립적으로도 작동한다.

```
[05:30] Layer 1: Calendar Transit Scanner (자동)
           ↓ 이동 시간 이벤트 생성
[06:00] Layer 2: Inbox Scan (자동)
           ↓ 이메일 → 태스크 변환
[기상 후] Layer 3: Morning Sweep (대화형)
           ↓ 분류 → 6개 에이전트 병렬 실행
[Sweep 후] Layer 4: Time Block (대화형)
           ↓ 남은 태스크 → 시간 블록 캘린더
```

### Layer 1: 캘린더 교통 시간 스캐너 (05:30)

익일 Google Calendar를 스캔해 물리적 위치 미팅을 찾고,
Google Maps Routes API로 실제 이동 시간을 계산해
"Transit →" 접두사가 붙은 이동 시간 이벤트를 생성한다.

**Pre-event transit:**

- 제목: "Transit → [이벤트명]"
- 이벤트 시작 시각에 맞춰 종료
- 소요 시간 = 운전 시간 + 주차 버퍼 5분
- 출발지: 업무 시간(9~17시 평일)이면 사무실,
  그 외는 자택
- 직전 이벤트가 있으면 그 위치를 출발지로 사용

**Post-event transit:**

- 제목: "Transit → [다음 목적지]"
- 이벤트 종료 시각에 시작
- 다음 위치까지의 실제 이동 시간

**스킵 규칙:**

- Zoom/Teams/Meet URL이 있는 가상 미팅
- 사무실 주소의 이벤트
- 운전 5분 미만 이동
- 이미 인접 transit 블록이 있는 이벤트
- 겹치는 이벤트

**기술 구현:**

- Google Cloud 프로젝트에서 Routes API 활성화
  (deprecated된 Distance Matrix API가 아님)
- API 키를 `~/.config/google-maps/api-key`에
  600 퍼미션으로 저장
- Node.js 래퍼: `routes.googleapis.com` 호출,
  npm 의존성 없음, 교통 상황 반영 라우팅,
  파이프 구분 다중 주소, JSON 출력
- macOS: LaunchAgent plist
  (`~/Library/LaunchAgents/`)
- Linux: cron 또는 systemd timer
- `claude -p` (비대화형 모드)로 실행
- `--max-budget-usd $2.00` 비용 상한

### Layer 2: 이메일 트리아지 (06:00)

전일 이메일을 분류하고, 조치가 필요한 항목을 식별해
Todoist에 태스크를 생성한다.

**Phase 1: 이메일 → 태스크**

1. 전일 이메일 검색
   (프로모션, 소셜, 업데이트, 포럼 제외)
2. 스니펫으로 1차 분류:
   자동 알림, 뉴스레터, FYI 스킵
3. 잠재적 액션 이메일만 전문 읽기
4. 태스크 추출: 직접 질문, 할당된 산출물,
   일정 요청, 본인이 한 약속
5. 태스크 매니저에서 중복 검사
   (활성 + 완료 태스크 모두)
6. 태스크 생성:
   - 동사 시작 제목 (예: "수정된 덱 [이름]에게 전송")
   - P1~P4 우선순위
   - 마감일 (추론 또는 합리적 기본값)
   - 소요 시간 추정
   - 이메일 링크가 포함된 설명
   - 올바른 프로젝트 배정

**Phase 2: 태스크 위생**

1. 최근 추가된 태스크 풀링
2. 각 태스크 속성 평가
3. 누락된 우선순위, 소요 시간, 마감일, 설명 보충
4. 이미 잘 정리된 태스크는 그대로 둠

**우선순위 프레임워크:**

| 등급 | 기준 | 테스트 질문 |
|------|------|-------------|
| P1   | 놓치면 돌이킬 수 없는 결과 | "다음 주에 고칠 수 없는 피해가 생기나?" |
| P2   | 시간에 민감, 지연 비용 누적 | "미루면 선택지가 좁아지나?" |
| P3   | 중요하나 유연한 타이밍 | "한 달 밀리면 짜증나지만 며칠은 괜찮나?" |
| P4   | 여유 있을 때 | "2주 사라져도 못 느끼나?" |

이 기준을 자동화의 시스템 프롬프트에 내장한다.

**도구 접근 스코핑:**

- 이메일: 검색 + 읽기 (발송 **절대 불가**)
- 태스크 매니저: 검색, 생성, 수정
- 거부: 캘린더, 파일시스템, 모든 파괴적 작업

**상태 추적:**

타임스탬프 파일로 마지막 성공 실행 시점을 기록해
후속 실행에서 새 이메일과
최근 추가 태스크만 처리한다.
재처리 방지의 핵심 메커니즘이다.

**실행:**

- Layer 1 후 30분 뒤 실행 (기본 06:00)
- `--dangerously-skip-permissions`로 헤드리스 실행
  (승인 터미널 없으므로)
- 도구 스코핑이 대화형 승인을 대체하는 안전 게이트

오전 6:15이면 태스크 시스템이 손대지 않아도
최신 상태가 된다.

### Layer 3: Morning Sweep (대화형)

태스크 + 캘린더 + 최근 미팅 녹취록을 가져와
모든 항목을 4단계로 분류하고,
승인 후 6개 에이전트를 병렬 실행한다.

**Phase 1: Gather**

- Todoist: 오늘 + 내일 + 연체 태스크
- Google Calendar: 오늘 이벤트
- 미팅 녹취록: 최근 48시간

**Phase 2: Classify**

| 색상 | 분류     | 의미 | 예시 |
|------|----------|------|------|
| 🟢   | Dispatch | AI가 완전히 처리 | 이메일 초안, 파일 갱신, 리서치 |
| 🟡   | Prep     | 80% AI, 사람 마무리 | 민감한 이메일, 미팅 준비, 본인 목소리 필요한 콘텐츠 |
| 🔴   | Yours    | 사람의 뇌/대면 필요 | 전략, 라이브 미팅, 관계 민감한 대화 |
| ⚪   | Skip     | 오늘 불가, 사유 기록 | 블로커 있는 태스크, 선행 조건 미충족 |

**핵심 바이어스: 불확실하면 Dispatch가 아닌
Prep으로 분류한다.**

**Phase 3: Present**

구조화된 리스트로 전체 분류 결과를 표시:

- 카테고리 색상 + 라벨
- 태스크명 + 프로젝트
- Dispatch: 담당 에이전트 + 실행 계획
- Prep: AI가 준비할 부분 + 사람이 마무리할 부분
- Yours: AI가 조립할 수 있는 지원 컨텍스트
- Skip: 연기 사유

승인 옵션:

- `go` — 승인된 Green + Yellow 전체 실행
- `go green` — Green만 실행
- 번호로 체리픽

**Phase 4: Dispatch**

승인된 각 태스크에 대해 적절한 에이전트를 스핀업:

| 에이전트 | 역할 | 도구 접근 |
|----------|------|-----------|
| Comms | 이메일 초안, 답장 스케줄링 | 이메일(읽기+초안, 발송 불가), 녹취록, 캘린더 |
| Calendar | 미팅 일정, 가용성 확인 | 캘린더, 태스크 매니저, 이메일 |
| Knowledge | Obsidian 파일 갱신, 회의록 | 파일시스템(지식베이스), 녹취록 |
| Research | 배경 조사, 경쟁 정보 | 웹 검색, 파일시스템 |
| Document | 문서 작성, 보고서 | 파일시스템 |
| Task | 태스크 매니저 정리, 재구성 | 태스크 매니저 |

각 에이전트는 독립된 컨텍스트 윈도우를 받고,
상세 마크다운 지시 파일로 정의된다:

- 역할과 범위
- 사용 가능 도구 (+ 명시적으로 없는 도구)
- 불확실성 처리 방식
- 품질 기준
- 안전 경계

**Comms Agent 안전 규칙 예시:**

- 절대 발송하지 않는다 — 초안만 저장
- 가격, 계약, 법적 사안, 관계 민감 내용은
  완료 처리하지 않고 사람 리뷰로 플래그
- 톤이 불확실하면 더 격식체 쪽으로

완료 후 요약 리포트를 표시한다.
확인된 실행 후에만 태스크를 완료 처리한다.

**원버튼 트리거:**

- `/morning-sweep` 슬래시 커맨드로 저장
- Stream Deck: `claude /morning-sweep` 셸 스크립트
- Raycast, Keyboard Maestro, 직접 입력 모두 가능

### Layer 4: Time Block (대화형)

남은 Todoist 태스크를 시간 블록 캘린더로 변환한다.
"[TB]" 접두사로 다른 시스템의 이벤트와 구분한다.

**일정 제약 정의:**

- 사무실 태스크: 08:30~16:00
- 저녁 태스크(자택만): 19:00~21:15
- 평일 19시 전에 자택 태스크 배치 금지
- 점심: 11:30~13:30 사이 30분 공백
- 운동: 월/수/금, 1시간 + 이동 시간
- 위치 태그: HOME, OFFICE, ERRAND, GYM, ANYWHERE

**스케줄 빌드 순서:**

1. 고정 블록 먼저 배치 (기존 이벤트, 운동, 점심)
2. 퀵윈(≤15분)을 먼저 배치해 모멘텀 확보
3. 나머지를 P1 → P4 순서로 채움
4. 심부름을 지리적 경로 하나로 묶고
   정류장 사이 이동 시간 포함
5. 자택 태스크는 저녁 윈도우에만 배치

**이동 시간:**

Layer 1의 travel time 스크립트를 호출한다.
심부름(정류장 사이), 체육관(사무실에서 왕복),
위치 특정 태스크에 적용.
이동 시간은 캘린더 블록 안에 포함시키고,
별도 이벤트로 만들지 않는다.

**롤오버 규칙:**

| 우선순위 | 처리 |
|----------|------|
| P1       | 강력 경고, 절대 조용히 밀지 않음 |
| P2       | 다음 영업일로 이동 |
| P3       | 5일 내 가장 여유 있는 날로 |
| P4       | 7일 내 가장 여유 있는 날로 |

**부분 완료 태스크:**

원래 태스크를 완료 처리하고,
"(part 2)" 태스크를 남은 소요 시간으로 생성한다.

**시스템 간 충돌 방지:**

Transit Scanner는 "Transit →" 접두사,
Time Blocker는 "[TB]" 접두사를 사용한다.
각 시스템은 상대의 접두사가 붙은 이벤트를 건너뛴다.
이중 이동 시간 계산과 덮어쓰기를 방지하는 핵심이다.

1분이면 하루 일정이 완성된다.

## 모델 선택 전략

야간 자동화와 대화형 작업의 모델을 다르게 쓴다.

| 작업 유형 | 모델 | 이유 |
|-----------|------|------|
| Layer 1~2 (야간) | Sonnet | 구조화된 규칙 따르기, 저렴·빠름 |
| Layer 3~4 (대화형) | Opus | 뉘앙스 있는 판단, 컨텍스트 조립 |

**실행당 비용:**

| 레이어 | 비용 |
|--------|------|
| Calendar Scanner | $0.25~0.50 |
| Inbox Scan | $0.50~1.00 |
| Morning Sweep | $1.00~3.00 (태스크 볼륨 따라) |

모든 야간 자동화에 `--max-budget-usd`를
설정해 폭주 비용을 방지한다.

## 핵심 설계 원칙

### 레이어 간 연결이 시스템을 만든다

각 구성 요소가 다른 구성 요소의 존재를 안다.

- 이메일 스캐너는 AM Sweep이 분류할 수 있도록
  메타데이터를 붙인다.
- AM Sweep은 각 서브에이전트가 독립 컨텍스트에서
  작업할 수 있도록 컨텍스트 패키지를 조립한다.
- Time Blocker는 상류의 모든 출력을 읽고 반영한다.
- 이벤트 접두사("Transit →", "[TB]")로
  시스템 간 경계를 명확히 한다.

어떤 레이어를 제거해도 나머지는 작동하지만,
전체가 결합되면 부분의 합보다 압도적으로 크다.

### 안전 모델: 무엇을 자동화하지 않을지 정한다

시스템의 **가장 중요한 설계 결정**은
무엇을 자동화하지 않을지 정하는 것이다.

**절대 자동화하지 않는 것:**

- 이메일 발송 (초안만 작성)
- Strategic Brief 작성 (핵심 전략 문서)
- 가격 결정
- 관계가 민감한 커뮤니케이션
- 전략적 판단

**안전 장치 체계:**

| 계층 | 메커니즘 |
|------|----------|
| 도구 스코핑 | 야간 자동화는 필요한 도구만 화이트리스트 |
| 접근 거부 | 이메일 발송, 리뷰 없는 캘린더 생성, 범위 밖 파일 수정 |
| 비용 상한 | 자동화 스크립트에 `--max-budget-usd` |
| 상태 추적 | 타임스탬프로 재처리 방지 |
| 분류 바이어스 | 애매하면 Dispatch가 아닌 Prep |
| 에이전트 격리 | 각 에이전트에 독립 컨텍스트 + 스코핑된 도구 |

> 대부분의 사람들이 자동화에서 가장 자주
> 틀리는 결정이다. 너무 소극적이면 그냥
> 멋진 할 일 목록이고, 너무 공격적이면
> 내 목소리가 아닌 이메일이 나가고
> 허가하지 않은 결정이 내려진다.

### 스펙이 코드다

Markdown 파일로 원하는 동작을 상세히 기술했다.
우선순위 프레임워크, 분류 규칙, 에이전트 지시,
스케줄링 제약 — 이것들이 실제 작업 산출물이다.
코드는 이 스펙에서 생성된다.

코드 문법을 이해할 필요는 없었다.
하지만 **아키텍처**는 명확히 알아야 했다.
무엇이 무엇과 대화하고, 각 부분의 책임은 무엇이고,
사람-AI 경계가 어디인지.
이것은 시스템 사고(systems thinking)이지,
소프트웨어 엔지니어링이 아니다.

> 시스템의 품질은 자기 워크플로에 대한
> 사고의 명확성에 정비례한다.
> 어떤 태스크를 dispatch하고
> 어떤 태스크를 prep하고
> 어떤 태스크를 사람에게 남길지
> 명확히 표현할 수 없으면,
> AI도 할 수 없다.

## 구현 전략

### 한 번에 4개를 만들지 않는다

하나의 레이어부터 시작해 충분히 테스트하고,
안정화된 후 다음 레이어를 추가한다.

**권장 구현 순서:**

1. 우선순위 프레임워크 정의
   (모든 레이어의 전제 조건)
2. **Layer 3: Morning Sweep** — 대화형이라
   가장 관대하고, 즉각 가치가 가장 큼
3. Layer 4: Time Block — Sweep을 확장
4. Layer 2: Inbox Scan — 안정적 태스크 구조 필요
5. Layer 1: Calendar Transit — 정제 레이어

### 최소 실행 가능 시스템 (MVP)

1. Claude Code 열기
2. 4단계 분류 프레임워크 정의
3. Morning Sweep 커맨드 작성
4. `/morning-sweep` 슬래시 커맨드로 저장
5. 키보드 단축키 또는 버튼에 바인딩
6. 3일 아침 실행하며 분류 규칙 정제
7. 안정되면 에이전트 하나 추가 (Comms Agent 권장)

### 비개발자를 위한 실전 팁

- Claude Code에 명확하고 구체적인 지시를 준다.
  "이메일 초안 써줘"가 아니라
  톤, 길이, 컨텍스트, 승인 기준, 안전 경계를 명시
- 에이전트를 개별 테스트한 후 병렬 실행한다.
  문제를 격리 상태에서 잡는다
- 의미 있는 접두사를 사용한다.
  캘린더 이벤트에 식별 가능한 접두사를 붙여야
  자동화가 자기 생성물을 인식한다
- 가정을 문서화한다.
  모든 자동화에는 운영에 대한 가정이 내장된다.
  (업무 시간, 위치, 우선순위 기준)
  이것을 적어야 Claude Code가 참조한다
- 하드 비용 한도를 설정한다
- 합성 데이터로 먼저 테스트한다.
  더미 태스크와 이메일로 검증 후 실제 업무에 적용

## 인사이트

### 스펙 주도 개발: 코드가 아닌 스펙을 쓰는 시대

Jim은 코드를 한 줄도 직접 쓰지 않았다.
하지만 전체 아키텍처를 설계했고,
각 레이어의 입출력과 책임을 정의했다.

이 사례가 증명하는 것:
**프로그래밍 능력 없이도 소프트웨어를 만들 수 있다.
단, 시스템을 설계하는 능력은 반드시 필요하다.**

"좋은 스펙"이란 우선순위 프레임워크, 분류 규칙,
에이전트 지시, 스케줄링 제약처럼
의사결정 기준을 명문화한 문서다.
Claude Code는 이 문서를 읽고 구현한다.

이것은 소프트웨어 산업의 근본적 변화다.
가치 창출의 병목이 "코드를 작성하는 능력"에서
"시스템을 구조화하는 사고력"으로 이동했다.
20년간 코딩을 배운 사람과
20년간 조직을 운영한 사람 중,
후자가 더 좋은 자동화를 만들 수 있는 시대가 왔다.

### DPYS: 자동화 경계의 범용 프레임워크

Dispatch-Prep-Yours-Skip 분류 체계는
이 시스템에서 가장 이식 가능한 아이디어다.

핵심은 **바이어스의 방향**이다.
"어디까지 자동화할 수 있는가?"가 아니라
"어디부터 사람이 개입해야 하는가?"를 먼저 정한다.
그리고 **경계선에서는 항상 Prep 쪽으로 기운다.**

이 프레임워크가 강력한 이유:

1. **실수의 비대칭성을 반영한다.**
   Dispatch를 Prep으로 잘못 분류하면 사람이
   5분 더 쓴다.
   Prep을 Dispatch로 잘못 분류하면 내 이름으로
   엉뚱한 이메일이 나간다.
   비용 비대칭이 명백하니
   보수적 바이어스가 합리적이다.

2. **점진적 신뢰 확장이 가능하다.**
   처음에는 대부분 Prep으로 시작하고,
   패턴이 안정되면 하나씩 Dispatch로 승격한다.
   "기본 소극적, 점진적 공격적"이
   안전한 자동화 확장의 공식이다.

3. **어떤 업무에든 적용 가능하다.**
   이메일뿐 아니라 코드 리뷰, 고객 응대,
   보고서 작성, 일정 관리 —
   모든 반복 업무에 이 4단계를 적용할 수 있다.

### 인지 부하 제거가 시간 절약보다 크다

시간 절약(연 130~195시간)도 크지만,
Jim이 강조하는 진짜 가치는 **인지 부하 감소**다.

> 아침에 "수집 모드"가 아닌 "결정 모드"로
> 시작한다. 이메일에 대한 저수준 불안감이
> 사라진 줄도 모르게 사라졌다.

이것은 단순한 생산성 향상이 아니다.
**주의력(attention)의 구조적 재배치**다.

GTD(Getting Things Done)의 핵심 통찰은
"열린 루프가 인지 대역폭을 소비한다"는 것이다.
Jim의 시스템은 야간 자동화로
"수집-처리-정리" 루프를 기상 전에 닫아버린다.
아침에 눈을 뜨면 이미 모든 것이 분류되어 있고,
유일하게 남은 결정은 "go"를 누를지 말지다.

이 패턴을 일반화하면:
**AI 자동화의 진정한 ROI는
대체된 시간이 아니라 해방된 주의력이다.**
시간은 선형적으로 돌아오지만,
인지 부하 감소의 효과는
판단 품질, 창의성, 관계에
복리로 작용한다.

### 에이전트 격리 패턴: 멀티에이전트 설계의 교과서

6개 서브에이전트 설계에서 배울 점:

**1. 컨텍스트 격리:**
각 에이전트가 독립된 컨텍스트 윈도우를 가진다.
Comms Agent는 이메일과 녹취록만 보고,
Research Agent는 웹과 파일시스템만 본다.
이것은 보안만을 위한 것이 아니다.
**좁은 컨텍스트가 더 좋은 출력을 만든다.**
모든 것을 아는 만능 에이전트보다
범위가 제한된 전문 에이전트가
자기 영역에서 더 정확하다.

**2. 도구 스코핑 = 권한 경계:**
각 에이전트가 사용할 수 있는 도구 목록이
곧 그 에이전트의 권한 범위다.
"사용 가능한 도구"뿐 아니라
"명시적으로 없는 도구"를 문서화하는 것이 핵심.
Comms Agent의 지시에는
"이메일 발송 도구가 없다"고 명시한다.

**3. 병렬 실행, 직렬 승인:**
에이전트는 병렬로 실행하지만,
실행 결과의 승인과 최종 적용은
사람이 직렬로 처리한다.
속도와 통제를 동시에 얻는 패턴이다.

**4. 불확실성 처리 정책:**
각 에이전트에 "모르겠을 때 어떻게 하라"는
지시가 있다.
Comms Agent: "톤이 불확실하면 더 격식체로"
Research Agent: "출처가 불확실하면
출처를 명시하고 신뢰도를 표기"
이것은 AI 시스템 설계에서 가장 간과되는 부분이다.

### 이벤트 네임스페이스: 멀티시스템 공존의 열쇠

"Transit →"와 "[TB]"라는 접두사 규칙은
사소해 보이지만 아키텍처의 핵심이다.

**문제:** 여러 자동화가 같은 캘린더를 수정하면
서로의 이벤트를 덮어쓰거나 중복 생성한다.

**해법:** 각 시스템이 고유 네임스페이스를 가지고,
다른 시스템의 네임스페이스를 건너뛴다.

이것은 소프트웨어의 **충돌 방지(conflict
avoidance)** 패턴을 비개발자가
직관적으로 적용한 사례다.
마이크로서비스의 API 경계,
데이터베이스의 스키마 분리와 같은 원리다.

**비개발자가 이런 패턴을 "발견"했다는 사실이
시스템 사고의 보편성을 증명한다.**

### 비용 구조의 파괴: $105 vs $400~$1,000

월 $100(Claude Max) + $5~10(API) vs
파트타임 VA $400~$1,000.

하지만 비용 비교보다 더 중요한 것은
**스케일링 특성**이다.

VA는 업무량이 2배면 비용도 2배다.
이 시스템은 태스크가 10개든 100개든
Morning Sweep 비용이 $1~3이다.
고객이 3곳에서 5곳으로 늘어도
시스템 자체를 바꿀 필요가 없다.
분류 규칙에 새 클라이언트를 추가하면 된다.

이것이 **소프트웨어의 한계 비용 제로** 특성이
비개발자의 1인 운영에도 적용된 사례다.

### "사람이 할 일" 정의가 설계의 90%

Jim의 가장 예리한 관찰:

> 자동화에서 가장 중요한 결정은
> 무엇을 자동화할지가 아니라
> 무엇을 자동화하지 않을지다.

이것은 기술적 결정이 아니라 **가치 판단**이다.
"이 이메일은 내 목소리여야 하는가?"
"이 결정은 관계에 영향을 주는가?"
"이것을 잘못해도 되돌릴 수 있는가?"

이 질문들에 답하려면 기술 역량이 아니라
자기 업무에 대한 **깊은 이해**가 필요하다.
20년 커뮤니케이션 경력이 있는 사람이
"이 이메일은 초안만 만들고 내가 검토"라고
정확히 선을 긋는 것 —
이것이 시스템의 품질을 결정한다.

**자동화의 역설:**
자동화를 잘 설계하려면
자동화하려는 업무를 수동으로 깊이 이해해야 한다.
업무 경험이 곧 자동화 설계 역량이다.

### 테크 버블 밖의 블루오션

Jim의 관찰: 실리콘밸리 밖에서는
AI로 실제 업무 인프라를 구축하는 사람이
거의 없다. 로펌, 회계사, 컨설턴트, 의사, 건축가 —
고도의 전문 지식 + 반복적 운영 오버헤드를
동시에 가진 직종이 가장 큰 기회다.

이 격차는 결국 좁혀질 것이고,
일찍 시작한 사람들이
**시스템적 사고 능력**이라는
복리 자산을 축적한다.

"Siri, but good"을 기다리지 않고
비개발자가 직접 구축한 작동하는 사례다.
36시간이라는 투자 대비 연간 수백 시간의
복리 수익을 만들어낸다.
