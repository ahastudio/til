# My chief of staff, Claude Code

트윗:
<https://twitter.com/jimprosser/status/2029699731539255640>

아키텍처 가이드:
<https://github.com/jimprosser/claude-code-cos>

코딩 경험이 전혀 없는 1인 컨설턴트가 Claude Code로
36시간 만에 개인 비서(Chief of Staff) 시스템을 구축한 사례다.
매일 아침 30~45분 걸리던 운영 업무를 한 자릿수 분으로 줄였고,
연간 130~195시간을 절약한다.

## 저자 배경

Jim Prosser(`@jimprosser`)는 43세 테크 커뮤니케이션 컨설턴트다.
Marin County, California에서 1인 부티크 컨설팅을 운영한다.
고객은 최대 5곳, 전원 레퍼럴, 주니어 스태프 없음.
Google, Twitter, SoFi, Paradigm에서 커뮤니케이션 리더를 역임했다.
코딩 자격증이나 경험은 **전무**하다.

## 시스템 구성

### Layer 1: 캘린더 교통 시간 스캐너 (05:30)

익일 Google Calendar를 스캔해 물리적 위치 미팅을 찾고,
Google Maps API로 실제 이동 시간을 계산해
"Transit →" 접두사가 붙은 이동 시간 이벤트를 생성한다.
현실적인 가용 시간을 보여주고 출발 시각을 알려준다.

### Layer 2: 이메일 트리아지 (06:00)

전일 이메일을 분류하고, 조치가 필요한 항목을 식별해
Todoist에 우선순위·마감일·소요 시간과 함께
태스크를 생성한다.
중복 검사와 상태 추적으로 재처리를 방지한다.
마지막 성공 실행 시점을 기록해
같은 데이터를 두 번 처리하지 않는다.

오전 6:15이면 태스크 시스템이 손대지 않아도
최신 상태가 된다.
야간 자동화는 저렴하고 빠른 Sonnet 모델을 사용한다.

### Layer 3: AM Sweep (Stream Deck 버튼)

태스크 + 캘린더 + 최근 미팅 녹취록을 가져와
모든 항목을 4단계로 분류한다.
대화형이므로 더 강력한 Opus 모델을 사용한다.

| 색상   | 분류     | 의미                              |
|--------|----------|-----------------------------------|
| Green  | Dispatch | Claude가 완전히 처리              |
| Yellow | Prep     | 80% 준비, 사람이 마무리           |
| Red    | Yours    | 판단·대면이 필요                  |
| Gray   | Skip     | 오늘 실행 불가                    |

분류 결과를 확인하고 "go"라고 하면
**6개의 특화 서브에이전트**가 병렬로 실행된다.

| 에이전트         | 역할                           |
|------------------|--------------------------------|
| Comms Agent      | 이메일 초안, 답장 스케줄링     |
| Calendar Agent   | 미팅 일정, 가용성 확인         |
| Knowledge Agent  | Obsidian 파일 갱신, 회의록     |
| Research Agent   | 배경 조사, 경쟁 정보           |
| Document Agent   | 문서 작성, 보고서              |
| Task Agent       | 태스크 매니저 정리, 재구성     |

각 에이전트는 독립된 컨텍스트 윈도우와
범위가 제한된 도구 접근 권한을 가진다.
역할, 사용 가능 도구, 불확실성 처리 방식,
품질 기준, 안전 경계를 정의한 상세 지시 파일을 받는다.

### Layer 4: Time Block (Stream Deck 두 번째 버튼)

남은 Todoist 태스크를 시간 블록 캘린더로 변환한다.
"[TB]" 접두사로 다른 시스템의 이벤트와 구분한다.

- 장소별 분류 (집/사무실/외부)
- 심부름을 지리적으로 묶어 한 번에 처리
- Google Maps 실제 이동 시간 반영
- 운동, 저녁 시간(아이들 취침 후) 배치
- 업무 시간, 고정 블록, 위치 태그 등 제약 반영
- 오늘 안 되면 우선순위별 롤오버:
  P1은 강력 경고, P2~P4는 여유 있는 날에 분배

1분이면 하루 일정이 완성된다.

## 핵심 설계 원칙

### 레이어 간 연결이 시스템을 만든다

각 구성 요소가 다른 구성 요소의 존재를 안다.

- 이메일 스캐너는 AM Sweep이 분류할 수 있도록
  메타데이터를 붙인다.
- AM Sweep은 각 서브에이전트가 독립 컨텍스트에서
  작업할 수 있도록 컨텍스트 패키지를 조립한다.
- Time Blocker는 상류의 모든 출력을 읽고 반영한다.

어떤 레이어를 제거해도 나머지는 작동하지만,
전체가 결합되면 부분의 합보다 압도적으로 크다.

### 사람이 할 일의 경계를 명확히 한다

시스템의 **가장 중요한 설계 결정**은
무엇을 자동화하지 않을지 정하는 것이다.

**절대 자동화하지 않는 것:**

- 이메일 발송 (초안만 작성)
- Strategic Brief 작성 (핵심 전략 문서)
- 가격 결정
- 관계가 민감한 커뮤니케이션
- 전략적 판단

**안전 장치:**

- 야간 자동화는 필요한 도구만 화이트리스트
- 이메일 발송, 리뷰 없는 캘린더 생성,
  범위 밖 파일 수정 접근 불가
- 자동화 스크립트에 최대 지출 한도 설정
- 상태 추적으로 재처리 방지

**애매하면 Prep으로 기본 설정한다.**
80% 준비해 두고 사람이 결정하게 한다.
Dispatch(완전 자동)로 기울지 않는다.

> 대부분의 사람들이 자동화에서 가장 자주
> 틀리는 결정이다. 너무 소극적이면 그냥
> 멋진 할 일 목록이고, 너무 공격적이면
> 내 목소리가 아닌 이메일이 나가고
> 허가하지 않은 결정이 내려진다.

### 구축은 대화였다

Markdown 파일로 원하는 동작을 상세히 기술했다.
분류 기준, 음성 가이드라인, 스케줄링 규칙,
도구 권한 등을 평문으로 작성하고,
Claude Code가 이를 읽고 구현했다.
실제 태스크 리스트로 실행하며 반복 튜닝했다.

코드 문법을 이해할 필요는 없었다.
하지만 **아키텍처**는 명확히 알아야 했다.
무엇이 무엇과 대화하고, 각 부분의 책임은 무엇이고,
사람-AI 경계가 어디인지.
이것은 시스템 사고(systems thinking)이지,
소프트웨어 엔지니어링이 아니다.

## 필요 도구

**필수:**

- Claude Max 또는 Pro 구독
- 항상 켜져 있는 컴퓨터 (야간 자동화용)
- API가 있는 태스크 매니저 (Todoist 권장)
- Google Calendar (프로그래밍 접근 가능)
- Gmail (검색 기능 접근 가능)

**선택:**

- 미팅 녹취 도구 (API 있는 것)
- 지식 관리 시스템 (Obsidian)
- Stream Deck 또는 매크로 패드

**스케줄링:**

macOS는 LaunchAgent,
Linux는 cron/systemd로 야간 자동화를 예약한다.

## 시작 방법

4개 레이어를 한 번에 구축하지 않는다.
하나의 레이어부터 시작해 점진적으로 확장한다.
각 레이어는 독립적으로 작동하도록 설계되어 있고,
서로를 점진적으로 보강하는 구조다.

GitHub 저장소에 각 레이어의 템플릿 스펙이 있다.
자신의 도구, 워크플로, 사람-AI 경계 판단에 맞춰
커스터마이징한다.

## 인사이트

### 시스템 사고가 코딩보다 중요하다

Jim은 코드를 한 줄도 직접 쓰지 않았다.
하지만 전체 아키텍처를 설계했고,
각 레이어의 입출력과 책임을 정의했다.
Claude Code 시대의 핵심 역량은
프로그래밍이 아니라 **시스템 설계**다.

### Dispatch-Prep-Yours-Skip 프레임워크

자동화 수준을 4단계로 나누는 이 분류 체계는
어떤 업무에든 적용할 수 있는 범용 프레임워크다.
핵심은 **애매하면 Prep 쪽으로 기울이는 것**이다.
AI의 실수 비용이 가장 큰 영역(발송, 결정, 관계)을
인간에게 남기는 것이 시스템 신뢰의 기반이다.

### 인지 부하 감소가 진짜 가치

시간 절약(연 130~195시간)도 크지만,
Jim이 강조하는 진짜 가치는 **인지 부하 감소**다.
아침에 "수집 모드"가 아닌 "결정 모드"로 시작한다.
이메일에 대한 저수준 불안감이
사라진 줄도 모르게 사라졌다고 한다.

### 비개발자 자동화의 현실적 모델

"Siri, but good"을 기다리지 않고
비개발자가 직접 구축한 작동하는 사례다.
필요한 것은 코딩 능력이 아니라
자기 업무를 구조화하는 능력이었다.
36시간이라는 투자 대비 연간 수백 시간의
복리 수익을 만들어낸다.

### 테크 버블 밖의 기회

Jim의 관찰: 실리콘밸리 밖에서는
AI로 실제 업무 인프라를 구축하는 사람이
거의 없다. 이 격차는 결국 좁혀질 것이고,
일찍 시작한 사람들이
**시스템적 사고 능력**이라는 우위를 가진다.
