# claude-self-improving-skills — Claude Code에 Hermes Agent식 자기개선 루프를 붙이는 플러그인

<https://github.com/UniM0cha/claude-self-improving-skills>

GeekNews: <https://news.hada.io/topic?id=30327>

## 소개

claude-self-improving-skills는 Nous Research의 Hermes Agent에서 영감을 받아 Claude Code에 절차적 메모리(procedural memory) 시스템을 구현하는 플러그인이다.
복잡한 태스크에서 효과가 있었던 워크플로를 자동으로 `SKILL.md` 파일로 증류(distill)하고, 이후 세션에서 재사용 가능하게 만든다.

```text
복잡한 태스크 → 유용한 것 증류 → 스킬로 저장 → 다음 세션에서 재발견
```

## 동작 방식

```text
Claude Code 세션 종료
  ↓
Stop 훅이 트랜스크립트와 도구 사용 오프셋 파싱
  ↓
충분히 복잡한 작업이 있고 아직 증류되지 않았으면 one-time 블록 반환
  ↓
Claude가 self-improving-skills:skill-distiller 서브에이전트에 위임
  ↓
증류기가 기존 스킬 패치 → 상위 스킬 패치 → 새 스킬 생성 순서로 시도
  ↓
검증 훅이 프론트매터/크기/출처를 검사하고 잘못된 편집은 롤백
  ↓
다음 세션: Claude Code가 스킬을 정상적으로 발견
```

학습된 스킬은 플러그인이 아닌 사용자 디렉토리에 저장된다.
플러그인을 업데이트해도 축적된 절차적 지식이 사라지지 않는다.

## 주요 기능

**자동 증류 트리거**: `Stop` 훅이 마지막 증류 이후 도구 호출 횟수와 파일 편집 수가 임계값을 넘으면 한 번 블록한다.
순수한 리서치 세션이 불필요하게 트리거되는 것을 방지하기 위해 파일 편집 최소 기준이 있다.

**스킬 편집 안전성**: 편집 전 백업, 편집 후 검증, 출처 스탬프, 불량 `SKILL.md`에 대한 자동 롤백이 있다.

**큐레이터 루프**: 오래되고 사용되지 않는 에이전트 생성 스킬을 `.archive/`로 복구 가능하게 보관한다.

**슬래시 커맨드**:
- `/distill-skill`: 수동 스킬 증류 실행
- `/curate-skills`: 스킬 목록 관리
- `/curator-status`: 큐레이터 상태 확인
- `/prune-skills`, `/archive-skill`, `/pin-skill`, `/restore-skill`
- `/propose-plugin-improvement`: 플러그인 자체 개선 제안

**설정 가능한 임계값**:

| 변수                        | 기본값 | 의미                              |
| --------------------------- | -----: | --------------------------------- |
| `SIS_DISTILL_THRESHOLD`     |     12 | 트리거 전 도구 호출 횟수          |
| `SIS_MIN_FILE_EDITS`        |      2 | 최소 파일 편집 수                 |
| `SIS_CURATE_MIN_SKILLS`     |      8 | 자동 큐레이션 시작 스킬 수        |
| `SIS_STALE_AFTER_DAYS`      |     30 | 미사용 스킬을 오래된 것으로 표시  |
| `SIS_ARCHIVE_AFTER_DAYS`    |     90 | 미사용 스킬 아카이브 이동         |

## 분석

### Hermes Agent의 핵심 아이디어를 Claude Code에 이식하는 방식

Hermes Agent의 자기개선 루프는 에이전트가 경험에서 절차적 지식을 추출하고 구조화된 형태로 저장한 뒤 나중에 재활용하는 것이다.
문제는 Claude Code에 Hermes Agent 같은 백그라운드 데몬 스레드가 없다는 점이다.

이 플러그인은 이 제약을 훅 시스템으로 우회한다.
`Stop` 훅을 증류 트리거로, `PreToolUse`/`PostToolUse` 훅을 스킬 편집 검증으로 사용한다.
에이전트의 생애주기 이벤트에 연결하는 것이다.

이 설계의 트레이드오프는 증류가 가시적이고 청구 가능한 서브에이전트 턴을 사용한다는 것이다.
Hermes Agent의 백그라운드 처리와 달리 사용자에게 보이며 비용이 발생한다.
이것을 README에서 솔직하게 “정직한 한계(Honest limitations)”로 명시하는 것은 투명성 있는 문서화 방식이다.

### 절차적 메모리와 사실적 메모리의 구분

이 플러그인이 다루는 것은 절차적 메모리다 — “이런 상황에서 이렇게 하면 효과적이었다”는 워크플로 지식.
개인 정보, 프로젝트 특성, 과거 대화 내용 같은 사실적 메모리는 다른 메커니즘(메모리 시스템, CLAUDE.md)이 담당한다.

이 구분은 중요하다.
“React 컴포넌트 테스트는 이 패턴으로 한다”는 절차적 지식이고, “이 프로젝트는 TypeScript를 사용한다”는 사실적 지식이다.
플러그인이 전자에 집중하고 후자를 다른 시스템에 맡기는 것은 단일 책임 원칙을 메모리 계층에 적용한 것이다.

### 스킬 생태계의 거버넌스 문제

자동 생성된 스킬이 쌓이면서 발생하는 문제들 — 중복 스킬, 더 이상 유효하지 않은 스킬, 충돌하는 스킬 — 은 큐레이터 루프가 해결하려는 것이다.
그러나 스킬의 품질과 관련성을 판단하는 것은 단순한 타임스탬프나 사용 횟수로는 충분하지 않다.

오랫동안 사용되지 않은 스킬이 드문 케이스를 다루는 귀중한 지식일 수 있고, 자주 사용되는 스킬이 잘못된 패턴을 반복하게 만들 수도 있다.
`SIS_STALE_AFTER_DAYS`와 `SIS_ARCHIVE_AFTER_DAYS` 기반의 시간 필터는 이 문제에 대한 근사치일 뿐이다.
플러그인이 스스로 이 한계를 인식하고 보수적인 아카이브 정책을 채택한 것은 적절한 설계 선택이다.

## 비평

### 증류된 스킬의 품질이 근본적 불확실성이다

이 플러그인의 가치는 증류된 스킬이 실제로 재사용 가능하고 정확한지에 달려 있다.
증류 프로세스 자체가 LLM 서브에이전트에 의해 이뤄지므로, 잘못된 일반화나 너무 맥락 특수적인 내용이 스킬로 저장될 수 있다.

예를 들어 “이 프로젝트의 특정 파일에서 효과적이었던 패턴”이 “모든 TypeScript 프로젝트에서 사용할 수 있는 일반 스킬”로 잘못 증류될 수 있다.
검증 훅이 형식적 올바름은 확인하지만 내용의 정확성은 확인하지 않는다.
이것은 플러그인이 해결할 수 없는 근본 한계이지만, 사용자가 생성된 스킬을 신뢰하기 전에 검토하는 절차가 필요하다.

### 플러그인 자체의 자기개선 제안 기능의 아이러니

`/propose-plugin-improvement` 커맨드는 플러그인 자체의 개선을 에이전트가 제안할 수 있게 한다.
이것은 아이디어로는 매력적이지만, 에이전트가 자신을 실행하는 시스템의 변경을 제안한다는 점에서 주의가 필요하다.

플러그인이 자신의 동작 방식을 변경하도록 제안하고 그것이 무비판적으로 적용되면, 의도하지 않은 행동 변화가 발생할 수 있다.
README가 이 기능에 대해 상세히 설명하지 않는 것은 아직 실험적 기능임을 시사한다.
이 기능의 실제 동작 방식과 안전 가드레일이 더 명확하게 문서화될 필요가 있다.

## 인사이트

### 에이전트 자기개선의 첫 번째 실용적 구현 패턴

자기개선 에이전트는 AI 연구의 오랜 목표이지만, 대부분 이론적 논의나 폐쇄형 연구 환경에서 이뤄졌다.
claude-self-improving-skills는 실사용 환경에서 실용적 자기개선 루프를 구현한 사례다.

핵심 통찰은 자기개선의 범위를 절차적 메모리로 제한한 것이다.
에이전트가 자신의 파라미터나 추론 방식을 바꾸는 것은 현재 불가능하지만, 자신이 사용하는 도구(스킬)를 개선하는 것은 가능하다.
이 제한된 형태의 자기개선이 현실적이고 안전하며 측정 가능하다.
Hermes Agent의 아이디어가 Claude Code라는 실용적 환경에서 구현될 수 있었던 것은 이 현실적 제약 덕분이다.

### Claude Code 훅 시스템이 플러그인 생태계의 기반이 되는 방식

이 플러그인이 존재할 수 있는 것은 Claude Code의 훅 시스템 덕분이다.
`Stop`, `PreToolUse`, `PostToolUse`, `SessionStart` 같은 생애주기 이벤트에 외부 스크립트를 연결하는 것이 복잡한 자기개선 루프의 구현을 가능하게 했다.

이것은 Claude Code를 확장 가능한 플랫폼으로 만드는 방향에서 중요한 설계 선택의 결과다.
훅 시스템이 있으면 코어를 변경하지 않고도 새로운 동작을 추가할 수 있다.
이 플러그인은 훅 시스템의 가능성을 탐색하는 동시에, 그 가능성의 경계도 보여준다 — 백그라운드 비동기 처리가 없다는 한계가 설계 결정에 미친 영향처럼.
