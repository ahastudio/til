# Claude 세션 간 Context Handoff: 맥락을 잃지 않는 4계층 전략

원문: <https://codex.epril.com/claude-session-context-handoff-4-layer-strategy>

## 요약

Claude Code의 컨텍스트 윈도우가 200K에서 1M 토큰으로 늘어났지만, 세션이 끝나면 모든 맥락이 사라진다는 본질적 문제는 해결되지 않았다.
오히려 한 세션에서 쌓을 수 있는 것이 많아진 만큼, 세션 종료 시 잃는 것도 늘었다.
이 글은 이 문제에 대한 체계적 대응으로 네 가지 층위의 Handoff 전략을 제시한다.

문제의 층위는 세 가지다.
**Context rot**: 컨텍스트가 길어질수록 중간 정보가 덜 참조되는 “lost in the middle” 현상이 심해지며, 커뮤니티 경험칙으로는 60% 사용률 근처에서 품질 저하가 시작된다.
**Auto-compact의 비대칭성**: 자동 요약은 모델이 가장 덜 명민한 순간에 발동하며, 다음 턴에 필요한 맥락을 잘라내는 역설이 발생한다.
**Context amnesia**: 새 세션은 백지로 시작한다. CLAUDE.md는 불변 규약을 이어주지만, 어제의 결정과 폐기된 가설은 이어주지 못한다.

4계층 전략은 영속성 강도에 따라 Tier 1부터 4까지 구성된다.
Tier 1은 현재 세션 내 관리(`/rewind`, `/compact [focus]`, 서브에이전트 활용), Tier 2는 단기 핸드오프(Document & Clear 패턴), Tier 3는 영구 메모리 계층(CLAUDE.md 계층, Report Registry), Tier 4는 세션 간 오케스트레이션(Spec 기반 개발, ADR, Git 커밋 메시지, Master-Clone 구조)이다.

핸드오프 문서 작성에서 가장 중요한 원칙은 명령형을 피하는 것이다.
“Implement X”가 아니라 “X is not yet implemented; it depends on Y”로 상태를 서술해야 새 세션이 맥락을 검증한 뒤 다음 행동을 스스로 결정한다.
핸드오프 문서는 fact가 아니라 hypothesis로 다뤄야 한다. 이전 세션이 혼동 상태에서 작성했다면 오류가 그대로 이어받아진다.

## 분석

### Tier 1: 세션 내 관리의 저평가된 도구들

`/rewind`(Esc Esc 두 번)는 실패한 시도를 컨텍스트에서 제거하고 다음 프롬프트에 학습을 녹여내는 방식이다.
“그 방식 말고 다른 방법으로”라고 이어 쓰는 것보다 실패 직전으로 돌아가 새 방향을 주는 것이 컨텍스트를 더 깨끗하게 유지한다.
`/compact [focus]`는 포커스 지시 없이 쓰면 결과가 예측 불가능하다. 숙련 사용자들은 아예 `/compact`를 회피하고 Document & Clear로 간다.

서브에이전트는 “main context를 오염시키고 싶지 않은 탐색”에 국한해야 한다.
수십 파일을 읽어도 main context에 돌아오는 건 요약 몇 단락뿐이라는 점이 강점이지만, 과다 생성하면 main agent가 자기 코드의 테스트 방법조차 서브에이전트를 호출해야 알 수 있는 상태가 된다.

### Tier 2: Document & Clear 패턴의 토큰 경제학

핸드오프 문서의 가치 있는 구조는 Summary, Key Decisions, Traps to Avoid, Working Agreements, Relevant Files(라인 번호 포함), Open Work(상태 서술형), Prompt for New Chat으로 구성된다.
Black Dog Labs의 측정에 따르면 이 패턴은 10,000+ 토큰의 지식 전달을 1,000~2,000 토큰으로 압축한다.

“Traps to Avoid” 섹션이 핸드오프 가치의 상당 부분을 담당한다.
성공 정보는 코드에 남지만 실패 정보는 핸드오프에 남기지 않으면 소실된다.
다음 세션이 똑같은 함정을 다시 밟는 비용은 누적된다.

### Tier 3: Registry 패턴의 구조적 통찰

`.claude/reports/_registry.md`는 50줄짜리 인덱스고, 개별 리포트는 필요할 때만 컨텍스트에 불러온다.
Main agent가 5,000줄 전체 리포트가 아닌 50줄 registry만 읽는다는 설계는 정보 접근 비용을 낮추면서 맥락 과부하를 방지한다.
Multi-agent 환경에서 agent A가 만든 리포트를 agent B가 무시하는 “context amnesia” 현상을 구조적으로 차단한다.

### Tier 4: Master-Clone vs Lead-Specialist

Master-Clone 구조에서는 CLAUDE.md에 모든 맥락을 두고, main agent가 동적으로 Task/Explore로 자기 복제본에 위임한다.
Lead-Specialist(커스텀 서브에이전트)를 과다하게 만들면 main agent가 조율 오버헤드에 잠식당한다.
핸드오프 관점에서도 Master-Clone이 단순하다. 전달할 “특화 에이전트 상태”가 따로 없기 때문이다.

## 비평

### 경험칙과 공식 권고의 혼재

글이 “60% 사용률 근처에서 품질 저하”라는 수치를 언급하면서 “Anthropic의 공식 권고가 아니라 커뮤니티의 경험칙”이라고 명시한 점은 정직하다.
그러나 이런 수치가 독자에게 기준으로 각인되면, 정작 자신의 작업 성격에 맞는 임계점을 찾는 실험을 건너뛰게 된다.
“정확한 임계점은 작업 성격에 따라 다르다”는 단서가 강조될 필요가 있다.

### 핸드오프 자체의 유지보수 비용

핸드오프 문서가 “fact가 아닌 hypothesis”라는 원칙은 중요하지만, 이 점을 진지하게 받아들이면 핸드오프 문서 자체를 신뢰하는 데 드는 인지 비용이 생긴다.
새 세션마다 “이 문서의 주장을 코드와 대조해 검증하라”는 지시는 검증 작업을 반드시 수반한다.
특히 잘못된 핸드오프 문서가 쌓이는 장기 프로젝트에서 이 유지보수 비용이 실제로 얼마나 되는지 글은 충분히 다루지 않는다.

### 도구 중심 서술의 한계

전략이 구체적인 도구(`/rewind`, `/compact`, `/context`)와 파일 구조(`.claude/reports/`)에 의존한다는 점은 강점이자 약점이다.
Claude Code 버전이 바뀌면 도구 이름이나 동작 방식이 달라질 수 있다.
글 자체에서도 “2026년 4월 Claude Code v2.1.101부터 커스텀 슬래시 명령이 skills로 통합됐다”는 변경 이력을 언급한다.
이 전략의 핵심 원칙(맥락 관리, 실패 기록, 검증 지시)은 도구에 독립적이지만, 실전 레시피는 툴 의존도가 높다.

## 인사이트

### 세션은 단위가 아니라 경계다

AI 코딩 도구를 사용할 때 “세션”을 대화의 기본 단위로 생각하면, 세션 전환이 곧 맥락 손실이 된다.
그러나 이 글이 제안하는 방식은 세션을 하나의 경계로 보는 것이다.
코드가 파일 시스템에 저장되듯, 결정과 실패는 `.claude/reports/`에 저장된다.
세션은 그 저장소에서 필요한 것을 꺼내 쓰는 작업 창(working window)이 된다.
이 관점의 전환이 핸드오프 전략 전체를 가능하게 하는 인식론적 기반이다.

### 실패 정보는 구조 없이 보존되지 않는다

소프트웨어 개발에서 성공적인 코드는 버전 관리로 보존된다.
그러나 “왜 이 방향을 택했는가”, “어떤 접근을 시도하고 왜 폐기했는가”는 어디에도 자동으로 저장되지 않는다.
ADR(Architecture Decision Record)이 이 문제를 부분적으로 해결하지만, ADR을 작성하는 규율이 없으면 소실된다.
핸드오프의 “Traps to Avoid” 섹션은 이 소실되는 실패 정보를 구조적으로 포획하는 첫 번째 시도다.
소프트웨어 개발이 Git이 코드를 보존한 것처럼 결정과 실패를 보존하는 시스템을 갖추려면, 핸드오프 문서가 ADR로 승격되고 ADR이 CLAUDE.md에서 참조되는 순환 구조가 필요하다.

### 컨텍스트 창이 커질수록 내비게이션이 중요해진다

1M 컨텍스트 윈도우가 와도 “어디에 뭐가 있는지”의 문제는 사라지지 않고 오히려 심해진다.
1M 토큰 안에 있는 정보의 품질과 구조가 맥락의 유효성을 결정한다.
Registry 패턴이 빛나는 이유는 바로 이것이다.
인덱스를 통해 필요한 정보에 도달하는 비용을 낮추는 설계는, 컨텍스트 창이 커질수록 더 중요해진다.
인간의 장기 기억에서 정보를 찾는 것이 정보의 양보다 조직화 방식에 달린 것처럼, AI 컨텍스트도 크기보다 구조가 품질을 결정한다.

### 습관의 경제학

이 글이 반복적으로 강조하는 것은 결국 규율이다.
세션 종료 시 1분을 핸드오프 문서 작성에 쓰는 것, “이 문서의 주장을 검증하라”는 지시를 잊지 않는 것.
End-of-Session 프롬프트를 `/handoff` 슬래시 명령으로 만들어두는 것은 이 규율의 마찰을 낮추는 시도다.
좋은 습관은 마찰이 낮을 때 형성된다. 핸드오프 문서를 1분 프롬프트 한 줄로 만들 수 있다면, 그 습관을 들이는 비용은 거의 0이다.
도구가 습관을 만들고, 습관이 품질을 만드는 연쇄가 여기서도 작동한다.
