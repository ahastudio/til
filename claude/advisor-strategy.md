# Advisor 전략: Opus를 조언자로 활용해 Sonnet의 지능을 끌어올리기

원문: <https://claude.com/blog/the-advisor-strategy>

## 요약

Anthropic이 발표한 Advisor 전략은 작은 모델(Sonnet/Haiku)을 실행자(executor)로,
Opus를 조언자(advisor)로 조합하는 패턴이다. 실행자가 대부분의 작업을 독립적으로
처리하되, 스스로 해결하기 어려운 판단에 직면했을 때만 Opus에 자문을 구한다.

핵심 성과:

- Sonnet + Opus Advisor: SWE-bench Multilingual에서 2.7%p 향상, 작업당 비용
  11.9% 절감
- Haiku + Opus Advisor: BrowseComp에서 19.7% → 41.2%로 2배 이상 향상, Sonnet
  단독 대비 85% 저렴

기술적으로는 단일 API 호출(`/v1/messages`) 안에서 모델 간 핸드오프가 일어나며
추가 라운드트립이 없다. `advisor_20260301` 도구 타입으로 구현되고 `max_uses`로
조언 횟수를 제한할 수 있다. 조언자 토큰은 Opus 요금, 실행자 토큰은 실행자
요금으로 분리 과금된다.

## 분석

### 기존 오케스트레이션의 역전

전통적 멀티모델 패턴은 큰 모델이 작업을 분해하고 작은 모델이 실행하는 톱다운
구조였다. Advisor 전략은 이를 뒤집어 작은 모델이 주도하고 큰 모델은 필요할 때만
개입하는 바텀업 구조를 택한다. 이 역전이 비용 효율을 만드는 핵심이다. Opus의
추론 능력이 전체 작업이 아니라 결정적 분기점에만 선택적으로 투입된다.

### 단일 API 호출이라는 설계 선택

모델 간 핸드오프를 하나의 API 호출 안에서 처리한 것은 개발자 경험과 지연 시간
모두를 고려한 설계다. 별도의 오케스트레이션 레이어 없이 기존 도구 사용 패턴에
자연스럽게 녹아들어간다. 하지만 이는 동시에 Anthropic 생태계 안에서만 작동하는
폐쇄적 구조이기도 하다.

### 비용 구조의 수학

조언자가 생성하는 토큰은 통상 400-700 토큰이다. 실행자의 전체 출력 대비 극히
적은 비율이므로 Opus 요금이 적용되더라도 총비용에 미치는 영향이 작다. "비싼
모델을 조금만 쓴다"는 단순한 원리지만 이를 API 수준에서 원클릭으로 구현한 것이
차별점이다.

## 비평

### 벤더 락인의 우아한 포장

[GeekNews 댓글](https://news.hada.io/topic?id=28370)에서도 지적되었듯이 이
전략은 Anthropic 모델 간의 조합만 지원한다. "Opus가 Sonnet을 돕는다"는
프레이밍은 매력적이지만 실질적으로는 두 개의 Anthropic 모델에 동시에 의존하게
만든다. 혼합 모델 접근(Opus → Gemini → Claude → Qwen 등)이 더 나은 결과를 낸다는
실무 보고가 존재하는 상황에서 단일 벤더 조합의 한계를 인식해야 한다.

### 자문 시점 판단의 블랙박스

실행자 모델이 "언제 조언을 구할지"를 스스로 판단한다. 이 판단의 기준이 투명하지
않다. 너무 자주 자문하면 비용이 증가하고, 너무 적게 자문하면 Advisor 전략의
이점이 사라진다. `max_uses` 파라미터로 상한을 설정할 수 있지만 최적의 자문
빈도를 찾는 것은 여전히 사용자의 몫이다.

### 벤치마크와 실무의 괴리

SWE-bench Multilingual에서 2.7%p 향상은 통계적으로 유의미하지만 실무에서 체감할
수 있는 차이인지는 별개 문제다. 반면 Haiku + Opus의 BrowseComp 2배 향상은
극적이다. 이 차이는 Advisor 전략이 실행자 모델의 기본 능력이 낮을수록 더 큰
효과를 발휘한다는 것을 시사한다. 역설적으로 Sonnet이 충분히 좋아질수록 Advisor
전략의 한계 효용이 줄어든다.

## 인사이트

### "필요할 때만 전문가를 부른다"는 인간 조직의 패턴 복제

이 전략은 소프트웨어 팀의 코드 리뷰 구조와 놀랍도록 닮았다. 주니어 개발자가
대부분의 구현을 하고 막히는 지점에서만 시니어에게 리뷰를 요청하는 패턴이다.
시니어가 모든 코드를 처음부터 작성하지 않듯 Opus가 모든 토큰을 생성하지 않는다.
AI 시스템 설계가 인간 조직 설계의 패턴을 재발견하고 있다는 점이 흥미롭다.

### 모델 간 위계가 API로 공식화되는 전환점

지금까지 모델 간 조합은 개발자가 프롬프트 체이닝이나 외부 오케스트레이션으로
직접 구현해야 했다. Advisor 전략은 이를 API 파라미터 하나로 축약했다. 이는 모델
간 위계(hierarchy)가 비공식적 관행에서 공식 인터페이스로 격상되는 전환점이다.
앞으로 "어떤 모델을 쓸까"가 아니라 "어떤 위계로 모델을 조합할까"가 설계 질문이
될 수 있다.

### 비용 최적화가 아키텍처 패턴을 낳는다

Advisor 전략의 동기는 성능 향상이 아니라 비용 절감이다. Opus 단독 사용이 최고
성능이지만 비용이 문제이므로 "Opus를 조금만 쓰는" 구조가 탄생했다. 역사적으로
비용 제약은 캐싱, CDN, 마이크로서비스 등 중요한 아키텍처 패턴을 낳았다. LLM 비용
제약이 만들어낸 이 패턴도 단순한 최적화 기법을 넘어 새로운 설계 어휘가 될
가능성이 있다.
