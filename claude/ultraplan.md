# Ultraplan - 클라우드에서 계획하기

원문: <https://code.claude.com/docs/en/ultraplan>

## 요약

Ultraplan은 Claude Code CLI에서 시작한 계획(planning) 작업을 Claude Code on the Web 세션으로 전달하여 클라우드에서 초안을 작성하고, 브라우저에서 리뷰 및 수정한 후, 실행 위치를 선택할 수 있는 기능이다.
로컬 터미널에서 `/ultraplan` 명령어나 "ultraplan" 키워드를 사용하면 클라우드 세션이 시작되며, 계획이 작성되는 동안 터미널은 다른 작업에 자유롭게 사용할 수 있다.
계획이 준비되면 브라우저에서 섹션별로 코멘트를 달거나 이모지로 반응하며 수정을 요청할 수 있고, 원하는 만큼 반복할 수 있다.

최종 계획이 만족스러우면 웹에서 바로 구현하거나, 터미널로 보내서 로컬 환경에서 실행할 수 있다.
터미널로 보낼 경우 현재 대화에 주입(Implement here), 새 세션 시작(Start new session), 파일로 저장 후 취소(Cancel) 중 선택할 수 있다.
Research preview 단계이며 Claude Code v2.1.91 이상, Claude Code on the Web 계정, GitHub 리포지토리가 필요하다.

## 분석

### 비동기 계획 작성의 설계

Ultraplan은 계획 작성 과정을 "백그라운드 작업"으로 분리함으로써 개발자의 흐름을 방해하지 않는다.
터미널에서 명령을 실행하면 클라우드 세션이 시작되고, 개발자는 계획이 작성되는 동안 다른 작업을 계속할 수 있다.
상태 표시기(`◇ ultraplan`, `◇ ultraplan needs your input`, `◆ ultraplan ready`)로 진행 상황을 확인할 수 있으며, `/tasks`로 세부 정보와 세션 링크를 열 수 있다.
이는 AI 어시스턴트가 "생각하는 시간"을 별도의 공간으로 격리하여, 개발자가 대기하지 않아도 되도록 설계한 것이다.

### 리뷰 인터페이스의 UX 혁신

터미널의 텍스트 인터페이스와 달리, 브라우저 기반 리뷰 뷰는 섹션별 인라인 코멘트, 이모지 반응, 아웃라인 사이드바를 제공한다.
이는 코드 리뷰 도구(GitHub PR, GitLab MR)의 UX 패턴을 계획 검토 프로세스에 적용한 것으로, 구조화된 피드백을 가능하게 한다.
특히 이모지 반응은 "빠른 승인" 또는 "우려 표시"를 위한 경량 피드백 채널로 작동하며, 전체 코멘트를 작성하지 않고도 의사를 전달할 수 있다.
이는 AI와의 협업을 "비동기 코드 리뷰"로 재해석한 것이다.

### 실행 위치 선택의 유연성

계획 승인 후 웹에서 실행하거나 터미널로 "텔레포트(teleport)"하는 선택지는 서로 다른 작업 컨텍스트에 대응한다.
웹 실행은 환경 설정이 필요 없는 빠른 프로토타이핑에 적합하며, 완료 후 바로 PR을 생성할 수 있다.
터미널 실행은 로컬 환경(데이터베이스, API 키, 개발 서버 등)에 대한 완전한 접근이 필요한 경우에 적합하다.
"Implement here"와 "Start new session" 옵션은 기존 대화 맥락을 유지할지, 새롭게 시작할지에 대한 세밀한 제어를 제공한다.

## 비평

### Remote Control과의 배타성 문제

Ultraplan과 Remote Control이 동시에 작동하지 않는다는 제약은 설계상의 한계를 드러낸다.
두 기능 모두 claude.ai/code 인터페이스를 차지하기 때문에 하나가 활성화되면 다른 하나는 비활성화된다.
이는 "로컬 머신에서 실행하면서 웹 UI로 제어"하는 Remote Control과 "클라우드에서 계획을 작성"하는 Ultraplan이 근본적으로 다른 실행 환경을 요구하기 때문이다.
사용자 입장에서는 워크플로우에 따라 수동으로 전환해야 하는 불편함이 있으며, 향후 두 기능을 통합하거나 동시 사용을 지원하는 아키텍처 개선이 필요하다.

### Research Preview의 불확실성

"Research preview" 명시와 "동작 및 기능이 피드백에 따라 변경될 수 있다"는 경고는 이 기능이 아직 안정화되지 않았음을 의미한다.
장기 워크플로우에 의존하기에는 위험하며, 특히 팀 협업이나 프로덕션 프로세스에 통합하기에는 신중해야 한다.
또한 Bedrock, Vertex AI, Foundry 등 엔터프라이즈 배포 환경에서는 사용할 수 없다는 제약이 있어, Anthropic 클라우드에만 의존해야 한다는 점도 한계다.

### 계획 반복의 효율성 의문

브라우저에서 "원하는 만큼 반복"할 수 있다는 것은 유연성이지만, 동시에 계획 작성이 길어질 수 있다는 우려도 있다.
계획 단계에서 너무 많은 시간을 소비하면 실제 구현이 지연될 수 있으며, "계획의 완벽함"을 추구하다가 오히려 생산성이 떨어질 수 있다.
특히 AI가 작성한 계획이 실제 구현과 괴리될 가능성이 있는데, 계획 단계에서 이를 완벽히 예측하기는 어렵다.
따라서 "적당한 수준의 계획" 후 빠르게 구현으로 넘어가는 것이 더 효율적일 수 있다.

## 인사이트

### 계획과 실행의 공간적 분리

Ultraplan은 "계획"과 "실행"을 단순히 시간적으로만 분리하는 것이 아니라, 공간적으로도 분리한다.
계획은 클라우드(브라우저)에서, 실행은 로컬 또는 클라우드에서 이루어지며, 이는 각 단계에 최적화된 인터페이스와 환경을 제공한다.
이는 전통적인 소프트웨어 개발에서 "설계 문서 작성"과 "코딩"이 물리적으로 다른 도구(워드 프로세서 vs IDE)에서 이루어지는 것과 유사하다.

하지만 AI 시대에는 이 경계가 흐려진다.
계획 자체가 실행 가능한 형태(코드, 구조화된 명세)로 작성되며, 계획을 수정하는 것이 곧 코드를 수정하는 것과 동등해진다.
Ultraplan은 이 새로운 패러다임을 "계획 리뷰"라는 중간 단계로 안착시키려는 시도다.
개발자는 AI가 작성한 계획을 리뷰하고 피드백하며, 최종적으로 실행을 승인하는 역할로 전환된다.
이는 "AI가 작성하고 인간이 검토"하는 새로운 분업 구조를 제도화한 것이다.

### 비동기 협업의 AI 적용

Ultraplan의 상태 표시기(`◇ ultraplan`, `◆ ultraplan ready`)는 비동기 협업 도구(Slack, 이메일, 이슈 트래커)의 알림 패턴을 따른다.
개발자는 AI의 작업이 완료될 때까지 기다리지 않고 다른 작업을 하며, 준비되면 알림을 받아 리뷰한다.
이는 AI를 "대기 시간이 긴 동료"로 취급하는 것으로, 인간-AI 협업의 새로운 모델을 제시한다.

전통적인 동기식 AI 어시스턴트(ChatGPT, Claude 채팅)는 개발자가 질문하면 즉시 응답하길 기대한다.
하지만 복잡한 계획 작성처럼 시간이 오래 걸리는 작업은 동기식 대화로는 비효율적이다.
Ultraplan은 이를 "비동기 작업 큐"로 전환하여, 개발자의 대기 시간을 제거한다.
이는 AI 어시스턴트가 "대화 상대"에서 "백그라운드 워커"로 진화하는 과정이며, 향후 더 긴 작업(전체 코드베이스 리팩토링, 대규모 마이그레이션 등)에도 적용될 수 있는 패턴이다.

### 실행 위치 선택이 드러내는 환경 복잡도

"웹에서 실행" vs "터미널로 텔레포트"라는 선택지는 현대 소프트웨어 개발의 환경 복잡도를 반영한다.
웹 환경은 클린하고 재현 가능하지만, 로컬 환경에 있는 데이터베이스, 환경 변수, 로컬 API 서버 등에 접근할 수 없다.
터미널 환경은 완전한 제어권을 주지만, 환경 설정의 일관성을 보장하기 어렵다.
Ultraplan은 이 두 환경을 "계획 단계에서는 환경 독립적"으로 유지하고, "실행 단계에서만 선택"하도록 설계했다.

이는 계획이 환경에 종속되지 않아야 한다는 철학을 내포한다.
좋은 계획은 어디서든 실행 가능해야 하며, 환경 특수성은 실행 시점에만 고려되어야 한다.
하지만 현실에서는 계획 작성 시점에 이미 환경 제약을 고려해야 하는 경우가 많다(예: "이 라이브러리는 로컬에서만 사용 가능").
Ultraplan의 설계는 이 간극을 "실행 위치 선택"으로 봉합하려 하지만, 계획과 환경의 완전한 분리는 여전히 이상향에 가깝다.

### Plan Mode의 제도화

Ultraplan은 plan mode를 "선택 사항"이 아니라 "필수 단계"로 만든다.
기존에는 plan mode를 사용할지 말지가 개발자의 재량이었지만, Ultraplan을 사용하면 자동으로 plan mode가 적용된다.
이는 "생각 없이 코딩"하는 것을 원천적으로 차단하고, 항상 "계획 → 리뷰 → 실행"의 구조화된 프로세스를 강제한다.

이는 소프트웨어 개발의 품질 관리 관점에서는 긍정적이지만, 빠른 실험이나 프로토타이핑에는 과도한 프로세스일 수 있다.
특히 간단한 변경(오타 수정, 설정 파일 업데이트)에도 계획 단계를 거쳐야 한다면 오히려 생산성이 떨어진다.
Ultraplan의 성공 여부는 "어떤 작업에 계획이 필요한가"를 판단하는 개발자의 메타인지에 달려 있다.
계획이 필요한 복잡한 작업과 즉흥적 실행이 효율적인 단순 작업을 구분하는 능력이 더욱 중요해진다.

### 텔레포트의 의미론

"Teleport back to terminal"이라는 표현은 단순한 파일 전송이 아니라, 작업 컨텍스트의 공간 이동을 의미한다.
계획은 클라우드에서 작성되었지만, 실행은 로컬에서 이루어진다.
이는 마치 "설계는 본사에서, 제조는 공장에서"와 같은 전통적 분업을 연상시킨다.
하지만 소프트웨어 개발에서는 설계와 제조의 경계가 모호하며, 텔레포트는 이 두 세계를 연결하는 포털이다.

텔레포트의 핵심은 "상태 보존"이다.
클라우드에서 작성된 계획이 터미널로 전송될 때, 대화 히스토리, 컨텍스트, 수정 사항이 모두 보존되어야 한다.
이는 단순한 파일 다운로드가 아니라, 세션의 직렬화(serialization)와 역직렬화(deserialization)다.
Ultraplan이 제공하는 3가지 옵션(현재 대화에 주입, 새 세션 시작, 파일로 저장)은 각각 다른 수준의 상태 보존을 의미한다.
이는 AI 어시스턴트의 "작업 컨텍스트"가 파일 시스템을 넘어 네트워크를 통해 이동 가능한 추상화된 객체임을 보여준다.
