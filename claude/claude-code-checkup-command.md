# Claude Code 신규 기능 `/checkup`

원문: <https://twitter.com/bcherny/status/2074997570317779038>

## 요약

Claude Code 팀의 Boris Cherny가 2026년 7월 8일, 새로운 슬래시
명령 `/checkup`을 소개하는 트윗을 올렸다. `/checkup`을 실행하면
다음과 같은 점검과 정리 작업을 수행한다.

1. 사용하지 않는 스킬·MCP·플러그인을 정리해 컨텍스트를 절약한다.
2. 로컬 `CLAUDE.md`를 저장소에 커밋된 `CLAUDE.md`와 비교해
   중복을 제거한다.
3. 루트 `CLAUDE.md`를 중첩된 `CLAUDE.md`와 스킬로 분리한다.
4. 느린 훅(hook)을 끈다.
5. Claude Code를 최신 버전으로 업데이트한다.
6. 자동 모드(auto mode)를 기본값으로 켠다.
7. 자주 거부됐던 읽기 전용 명령을 미리 승인 처리한다.

이 외에도 몇 가지 부가 기능이 더 있다고 언급된다. 가장 중요한
설계 원칙은 `/checkup`이 어떤 변경이든 실행하기 전에 반드시
사용자에게 확인을 구한다는 점이다.

## 분석

### 설정 점검을 하나의 명령으로 자동화하는 발상

Claude Code를 오래 쓴 사용자일수록 `CLAUDE.md`, 훅, MCP, 플러그인,
권한 설정이 시간이 지나며 누적되고 중복되는 문제를 겪는다.
`/checkup`은 이런 누적된 설정 부채를 사람이 하나하나 점검하는
대신 에이전트가 스스로 진단하고 정리하도록 설계됐다. 이는 Claude
Code가 단순히 코드를 작성하는 도구를 넘어, 스스로의 운영 환경을
유지보수하는 메타 도구로 기능 범위를 넓히고 있음을 보여준다.

### 일곱 가지 항목이 겨냥하는 공통 문제는 “컨텍스트 낭비”다

일곱 항목을 관통하는 주제는 컨텍스트 낭비 제거다. 쓰지 않는
스킬·MCP·플러그인은 컨텍스트 윈도우를 잠식하고, 중복된 `CLAUDE.md`
내용은 같은 지침을 두 번 읽게 만들며, 지나치게 비대해진 루트
`CLAUDE.md`는 관련 없는 지침까지 매번 로드하게 만든다. 느린 훅
역시 매 턴마다 실행 시간을 늘려 작업 흐름을 방해한다. 즉
`/checkup`은 개별 기능들의 나열이 아니라, “설정이 커질수록
에이전트가 비효율적으로 작동한다”는 하나의 문제의식에서 나온
일관된 해법이다.

### 확인을 거치는 자동화라는 절충안

일곱 항목 중 상당수(CLAUDE.md 재구성, 훅 비활성화, 자동 모드
전환, 명령 사전 승인)는 사용자의 작업 방식과 보안 태세에 실질적인
영향을 미치는 변경이다. 이런 변경을 완전 자동으로 적용하는 대신
“확인 후 적용”이라는 절충을 택한 것은, 편의성과 통제권 사이에서
후자를 포기하지 않으려는 설계 판단으로 보인다. 이는 자동화의
범위를 넓히면서도 사용자 동의라는 최소한의 안전장치를 유지하려는
Claude Code 팀의 최근 설계 기조와 일치한다.

## 비평

### “확인 후 적용”이 실제로 신중한 검토를 보장하지는 않는다

`/checkup`이 변경 전에 확인을 구한다는 점은 안전장치로 소개되지만,
일곱 항목이 한꺼번에 제안될 경우 사용자가 각 항목을 개별적으로
충분히 검토할지는 별개의 문제다. 특히 “자동 모드를 기본값으로
켠다”거나 “자주 거부됐던 명령을 미리 승인 처리한다”는 항목은
보안·권한 경계에 직접 영향을 미치는데, 여러 변경 사항이 한
번의 확인 흐름에 뭉쳐 제시되면 사용자가 일괄 수락 버튼을 누르듯
승인할 위험이 있다. 트윗은 확인 절차의 세부 UX(항목별 개별
확인인지, 전체 일괄 확인인지)를 명시하지 않아 이 우려를 해소하지
못한다.

### “느린 훅을 끈다”는 항목은 훅이 존재하는 이유와 충돌할 수 있다

훅은 대개 의도적으로 느리게 설계된 경우가 있다. 예컨대 테스트
실행, 보안 스캔, 린트 검사처럼 정확성과 안전성을 위해 시간을
들이는 훅은 “느리다”는 이유만으로 끄면 오히려 원래 목적을
훼손한다. `/checkup`이 느린 훅을 자동으로 식별해 끄자고 제안하는
기준이 순수하게 실행 시간인지, 아니면 훅의 실효성까지 함께
평가하는지 트윗만으로는 알 수 없다. 이 구분이 없다면 사용자는
속도를 위해 실제로 필요한 안전장치를 무심코 제거할 수 있다.

### 로컬과 커밋된 CLAUDE.md의 “중복 제거”가 의도된 차이를 지울 수 있다

로컬 `CLAUDE.md`와 저장소에 커밋된 `CLAUDE.md`가 다른 내용을
담고 있는 경우, 그 차이는 종종 의도적이다. 예를 들어 개인 개발
환경에 맞춘 지침이나, 아직 팀에 공유하지 않은 실험적 워크플로가
로컬에만 있을 수 있다. “중복 제거(dedup)”라는 표현은 두 파일의
차이를 전부 불필요한 중복으로 전제하는 듯한 인상을 주는데, 실제로는
의도된 차이와 우발적 중복을 구분하는 판단이 필요한 작업이다.
트윗은 이 구분을 어떻게 처리하는지 설명하지 않는다.

## 인사이트

### 설정 누적은 도구가 오래 쓰일수록 필연적으로 커지는 부채다

`/checkup` 같은 기능이 등장했다는 사실 자체가, Claude Code
사용자 커뮤니티에서 설정 파일·훅·MCP·플러그인의 누적이 이미
충분히 흔한 문제로 인식됐다는 신호다. 이는 소프트웨어 개발에서
익숙한 “기술 부채”의 패턴이 에이전트 설정 영역에도 그대로
나타나고 있음을 보여준다. 코드에 기술 부채가 쌓이듯, 에이전트를
오래 운영할수록 지침·권한·자동화 설정에도 부채가 쌓이고, 이를
정기적으로 청산하는 절차가 필요해진다는 것이다. 앞으로 에이전트
플랫폼들은 코드용 린터나 정적 분석 도구처럼, 설정 자체를 점검하고
정리하는 도구를 표준 기능으로 갖추는 방향으로 수렴할 가능성이
크다.

### “자동 모드를 기본값으로 켠다”는 제안은 자동화 신뢰의 임계점을 보여주는 지표다

자동 모드를 기본값으로 전환하도록 권장하는 기능이 나왔다는 것은,
Claude Code 팀이 이제 상당수 사용자에게 자동 모드가 충분히
신뢰할 만한 기본값이라고 판단하고 있다는 뜻이다. 이는 에이전트
자율성을 둘러싼 조직의 태도가 “기본은 보수적으로, 필요할 때만
자동화”에서 “기본은 자동화, 필요할 때만 개입”으로 옮겨가는
변곡점을 시사한다. 다만 이런 전환이 모든 사용자와 모든 작업
맥락에 동일하게 적합한지는 별개의 문제이며, 되돌리기 어려운
작업(배포, 프로덕션 변경)이 많은 환경에서는 이 기본값 전환이
오히려 사고 위험을 키울 수 있다는 트레이드오프가 함께 따라온다.

### 명령 사전 승인 기능은 권한 모델을 “거부 이력 기반 학습”으로 바꾼다

“자주 거부됐던 읽기 전용 명령을 미리 승인 처리한다”는 항목은,
권한 관리를 정적인 화이트리스트가 아니라 사용자의 과거 행동
패턴에서 유추하는 방식으로 전환한다는 뜻이다. 이는 사용성을
높이지만, 동시에 권한 경계가 사용자가 명시적으로 설계한 정책이
아니라 누적된 행동 이력에 의해 암묵적으로 결정된다는 의미이기도
하다. 장기적으로 이런 접근이 여러 프로젝트나 팀원 사이에 퍼지면,
왜 특정 명령이 승인되어 있는지 그 근거를 추적하기 어려워지는
투명성 문제로 이어질 수 있다. 권한 모델이 정책 기반에서 이력
기반으로 옮겨가는 흐름은 편의성과 감사 가능성(auditability)
사이의 새로운 트레이드오프를 만들어낸다.
