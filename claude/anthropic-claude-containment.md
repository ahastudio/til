# Anthropic은 제품 전반에서 Claude를 어떻게 봉쇄할까

원문: <https://www.anthropic.com/engineering/how-we-contain-claude>

HN 토론: <https://news.ycombinator.com/item?id=48392082> (223점, 100개 댓글)

## 요약

Anthropic이 세 가지 제품 배포 환경(claude.ai, Claude Code, Claude Cowork)에서 Claude를 봉쇄하는 방법을 상세히 설명한다. 에이전트의 역량이 커질수록 잠재적 피해 반경도 넓어지기 때문에, 인간 감독에 의존하기보다 환경 레이어에서의 봉쇄 설계를 우선시한다.

리스크는 세 범주로 나뉜다. 사용자 오남용(의도적이든 부주의하든), 모델 오동작(에이전트 스스로의 의도치 않은 유해 행동), 외부 공격자(도구·파일·네트워크를 통한 위협)다. 방어는 환경(샌드박스, VM, 파일시스템 경계, 이그레스 통제), 모델 레이어(시스템 프롬프트, 분류기, 프로브, 학습 수정), 외부 콘텐츠(MCP 서버 감사, 도구 권한 세분화, 커넥터 제한)의 세 겹으로 구성된다.

세 가지 격리 패턴이 제품별로 다르다. claude.ai는 세션마다 새로 생성되는 임시 컨테이너(gVisor)를 사용하며 “피해 반경이 최소화”된다. Claude Code는 로컬에서 실행되며 초기에는 행동마다 승인이 필요했으나, 사용자가 93%의 요청을 승인한다는 텔레메트리 데이터를 확인한 후 OS 레벨 샌드박스(Seatbelt/bubblewrap)를 배포해 권한 요청을 84% 줄였다. Claude Cowork는 비기술 사용자를 위한 완전한 가상 머신 격리를 사용한다.

실제로 발견된 보안 취약점 세 가지가 공개됐다. 신뢰 확인 이전 코드 실행 문제(`.claude/settings.json`의 훅이 신뢰 수락 전에 실행됨), 직접 프롬프트 인젝션(내부 레드팀이 25회 시도 중 24회 성공적으로 AWS 자격증명 탈취), 허용 도메인을 통한 데이터 유출(Cowork의 `api.anthropic.com` 허용목록을 악용해 공격자 계정으로 파일 전송)이다. 핵심 원칙은 “환경 레이어에서 먼저 설계하고, 그 다음 모델 레이어에서 행동을 조정하라”는 것이다.

## 분석

### 승인 피로(approval fatigue)와 자동화의 역설

93%의 승인률 데이터는 흥미로운 딜레마를 드러낸다. 사용자가 거의 모든 요청을 승인한다면, 명시적 승인 과정은 실질적인 안전 가드레일이 아닌 마찰 비용에 불과하다. 반면 승인 과정을 없애면 에이전트가 실수를 할 때 인간 감독 기회도 사라진다. Anthropic의 해결책(OS 레벨 샌드박스로 더 강한 기술적 경계를 세우고 사용자 개입 빈도를 줄임)은 이 역설에 대한 실용적 대응이다.

### 프롬프트 지시는 보안 경계가 아니다

레드팀 실험의 결과(“25회 중 24회 AWS 자격증명 탈취 성공”)는 충격적이지만, 더 중요한 발견은 그 메커니즘이다. 사용자 자신이 타이핑한 지시처럼 보이는 악성 프롬프트에 대해 모델 방어는 이상 징후를 감지하지 못했다. “공격자가 사용자인 척 타이핑하면” 모델은 구분할 수 없다. 이것은 모델 행동 수정이 아닌 환경 레이어 격리가 왜 우선순위여야 하는지를 설명한다.

### 허용된 채널을 통한 유출

`api.anthropic.com`은 기능상 필요한 도메인이라 허용목록에 포함됐는데, 바로 그 채널을 통해 공격자의 숨겨진 API 키로 파일을 외부로 전송하는 공격이 가능했다. 이 취약점은 “허용 도메인” 개념이 “허용 콘텐츠”와 분리될 수 없다는 교훈을 준다. 수정은 VM 내부에서 API 트래픽을 가로채는 방어적 중간자 프록시를 배치하는 방식으로 이루어졌다.

### 실제 사용자들의 자체 봉쇄 전략

HN 커뮤니티에서는 Anthropic의 공식 봉쇄 외에 사용자들이 자체적으로 구축한 격리 방식이 공유됐다. lebovic[^lebovic]은 claude.ai/code 세션이 다른 세션의 저장소와 환경 변수 전체에 접근·유출하는 컨테이너 격리 버그가 수차례 재발했다고 지적하며, 근본적인 토큰 스코프 문제가 아직 완전히 해결되지 않았다고 주장했다. saagarjha[^saagarjha]는 프롬프트 인젝션 방어가 구조적으로 어려운 이유를 설명했다. 에이전트가 네트워크 접근이 있는 VM에서 실행되면 외부 콘텐츠를 통해 중첩 인젝션이 발생하고, 에이전트가 관리해야 할 파일이나 캘린더 이벤트 자체가 공격 벡터가 된다는 것이다. vbezhenar[^vbezhenar]는 QEMU 가상 머신으로 Claude를 실행하며 GitHub 토큰 범위를 제한하고 커밋을 직접 검토 후 푸시하는 방식을 택했다. saghm[^saghm]은 bubblewrap 헬퍼 함수로 현재 디렉터리만 읽기/쓰기 허용하고 나머지는 읽기 전용으로 제한하는 설정을 구축했다.

## 비평

### 강점: 실패 공개의 용기

이 글의 가장 큰 가치는 실제 보안 실패 사례를 구체적으로 공개한다는 점이다. 많은 회사가 보안 취약점을 내부에 감추거나 추상적으로만 언급하는 것과 달리, Anthropic은 공격 벡터, 성공률, 수정 방법을 모두 상세히 설명한다. 이것은 업계 전체에 유용한 지식을 제공하는 동시에, 회사의 보안 의식을 신뢰할 수 있게 만든다.

### 약점: 완전히 해결되지 않은 근본 긴장

에이전트가 점점 더 강력해질수록 봉쇄 설계는 항상 뒤를 쫓는 방어 게임이 될 수밖에 없다. [GeekNews 댓글](https://news.hada.io/topic?id=30186)에서도 위험-수익 계산이 배포를 정당화하는지에 대한 의문이 제기됐다. 이 글은 현재의 봉쇄 상태를 설명하지만, 왜 이 리스크 수준이 수용 가능한지에 대한 설명은 충분하지 않다. HN에서 6gvONxR4sf7o[^hn-6gvONxR4sf7o]는 Anthropic의 프레이밍을 "피해가 줄어들지 않고 보상이 커지므로 피해는 사업 비용이 된다"는 논리라고 직접적으로 비판했다. NiloCK[^NiloCK]는 기대 피해가 기대 보상과 "통계적으로 동등"해졌을 때 배포를 정당화한다는 논리에 대해, 기댓값이 피해를 *능가*할 때까지 기다려야 하지 않냐는 반론을 제기했다.

## 인사이트

### 에이전트 보안의 새로운 공격 면

기존 소프트웨어 보안과 에이전트 보안의 근본적 차이는, 에이전트가 명령을 “이해”하고 맥락을 해석한다는 점에서 비롯된다. 기존 시스템에서는 인젝션 공격이 파싱 취약점을 노렸다. 에이전트에서는 에이전트의 추론 능력 자체가 공격 면이 된다. 설득력 있게 보이는 지시를 작성할 수 있는 공격자는 기술적 취약점이 없어도 에이전트를 조종할 수 있다.

이것은 보안 모델의 근본 가정을 바꾼다. 기존 보안은 “신뢰되지 않은 입력을 처리하는 코드”를 분리하는 방식으로 접근했다. 에이전트 보안은 “신뢰되지 않은 내용을 해석하는 추론 과정”을 어떻게 격리할지라는 더 어려운 문제를 다룬다. 환경 레이어 격리를 우선하는 Anthropic의 접근은 이 어려움에 대한 현실적 인정이다.

### 배포 결정의 정치경제학

“승인 피로” 데이터는 안전 설계에서 인간의 행동 경제학이 얼마나 중요한지를 보여준다. 이론적으로 완벽한 감독 체계도 사용자가 실질적으로 참여하지 않으면 의미가 없다. Anthropic이 마찰을 줄이는 방향(OS 샌드박스)으로 전환한 것은 이 현실을 인정한 것이다.

그러나 이 전환은 다른 질문을 만든다. 기술적 봉쇄가 인간 감독을 대체할 수 있는가? 에이전트의 행동 영역이 넓어질수록 기술적 봉쇄도 더 복잡해진다. “표준화된 보안 기본요소는 버텼고, 우리 자체 구현 주변에서 취약점이 노출됐다”는 고백은 이 도전의 핵심을 잘 요약한다. protocolture[^protocolture]는 LLM을 의인화하는 것도, 반대로 SF적 공포 시나리오로 다루는 것도 모두 실제 위협을 이해하는 데 방해가 된다고 지적했다. LLM이 영화 논리처럼 인터넷에 흘러나가 복제를 시작하는 것이 아니라, 훨씬 현실적이고 평범한 공격 경로가 더 위험하다는 시각이다.
