# Claude와 함께한 UI 버그 디버깅 세션

원문: <https://handhand.tistory.com/entry/debugging-session-with-claude>

## 요약

헬스케어·금융·교육 분야에서 10년 이상 경력을 쌓은 시니어 풀스택 개발자가
Next.js 애플리케이션에서 발생한 복잡한 UI 버그를 Claude와 함께 디버깅한 실시간 세션을
기록한 글이다.

문제는 투자 제안서 템플릿 페이지에서 사용자가 새 템플릿을 선택한 후 폼으로 돌아올 때
폼이 응답하지 않는 것이었다.
페이지 자체는 렌더링되지만 폼 옵션과 버튼 등 모든 인터랙티브 요소가 동작을 멈췄다.
페이지를 새로고침하면 기능이 복원됐다.

디버깅 과정은 네 가지 가설을 거쳤다.
① 폼의 disabled 상태 문제(미확인) → ② 로딩 화면의 인터랙션 차단(부분적으로 정확)
→ ③ 모달이 제대로 닫히지 않는 문제(근본 원인에 가까움)
→ ④ 드롭다운-모달 컴포넌트 충돌(실제 원인).
Claude는 `document.body.style`의 `pointerEvents` 속성을 추적하는 코드를 제안해,
`pointer-events: none`이 CSS 규칙이 아닌 인라인 스타일로 설정되고 있음을 밝혔다.

근본 원인은 Radix UI 모달이 열릴 때 body에 `pointer-events: none`을 설정하지만,
드롭다운 메뉴 내에서 모달이 트리거될 때 닫힘 시 이를 제대로 제거하지 않는 버그였다.
해결책은 GitHub 이슈에서 발견됐다.
모달을 트리거하는 드롭다운 메뉴 컴포넌트에 `modal={false}`를 추가하는 것이었다.

저자는 Claude가 코드베이스 패턴 매칭, 디버깅 기법 제안, 대안 가설 빠른 생성에 탁월했다고
평가했다.
반면 좁은 컨텍스트 창, 잘못된 설명에 대한 과신, 근본 원인보다 임시방편을 수용하려는
경향을 한계로 지적했다.
전반적으로 초기 조사 이후 협업 파트너로서 가장 유용했으며, 약 4시간의 시간을 절약했다고
결론 내렸다.

## 분석

### `pointer-events: none` 추적 기법의 가치

Claude가 제안한 `Object.defineProperty(document.body.style, 'pointerEvents', {...})`를
이용한 속성 추적 기법은 이 디버깅의 전환점이었다.
CSS 규칙으로 스타일이 설정되는지, 인라인으로 설정되는지를 구분하는 것은 단순한 브라우저
개발자 도구 검사로는 놓치기 쉬운 차이다.
JavaScript로 스타일 변경을 가로채 콜 스택을 추적하는 방식은, UI 라이브러리 내부에서
일어나는 DOM 조작을 외부에서 관찰하는 우회 전략이다.

### Radix UI의 포커스 트랩과 body 잠금 메커니즘

Radix UI 모달의 `pointer-events: none` 설정은 의도된 접근성 기능이다.
모달이 열려 있을 때 배경 콘텐츠와의 상호작용을 막아 키보드와 스크린 리더 사용자가
모달 안에 집중하도록 하는 포커스 트랩(focus trap)의 일환이다.
문제는 드롭다운 메뉴가 이미 자체적인 포커스 트랩을 운영하는 상태에서 모달이 중첩될 때,
두 컴포넌트의 DOM 정리 순서가 꼬이면서 `pointer-events: none`이 남는 것이다.
`modal={false}`가 해결책이 된 이유는 이 설정이 Radix Dropdown Menu에게 자신이 모달
컨텍스트 안에 있지 않다고 알려, 중복 포커스 트랩 메커니즘을 비활성화하기 때문이다.

### 가설 수렴 과정의 패턴

네 가지 가설이 순서대로 더 정확한 방향으로 수렴하는 과정은 전형적인 이진 탐색식
디버깅이다.
“폼 전체” → “로딩 화면” → “모달 닫힘” → “드롭다운-모달 충돌”로 범위가 좁아졌다.
각 가설이 이전 가설보다 더 구체적인 계층을 지목한다.
이 수렴 과정에서 Claude가 가설을 제안하고, 저자가 코드에서 검증하는 역할 분담이
효과적으로 작동했다.

### “GitHub 이슈에서 발견”의 의미

최종 해결책이 Claude가 아닌 GitHub 이슈에서 나왔다는 사실은 중요한 관찰이다.
Claude는 기존에 학습된 패턴에서 제안할 수 있지만, 특정 라이브러리의 알려진 버그와
그 공식 해결책은 이슈 트래커나 릴리스 노트에 문서화되어 있다.
이 구체적인 버그의 `modal={false}` 해결책은 Radix UI 저장소의 커뮤니티 경험에서 나온
지식이다.

## 비평

### 강점: 실전 경험의 솔직한 기록

성공 사례만 선별하지 않고 Claude가 틀린 가설을 제시한 순간과 과신한 순간을 함께
기록한 점이 강점이다.
“잘못된 설명에 과신”, “임시방편을 수용하려는 경향”이라는 구체적 한계를 명시함으로써,
독자가 Claude를 도구로 사용할 때 어떤 판단을 직접 해야 하는지를 알 수 있다.

### 한계: 실패한 가설의 상세 기록이 부족하다

Claude가 틀린 설명을 과신한 구체적 사례가 무엇이었는지, 어떤 임시방편을 수용하려
했는지의 내용이 요약 수준에서 그쳤다.
그 실패의 구체성이 독자에게 더 실용적인 학습을 제공할 수 있었을 것이다.
어떤 종류의 질문에서 Claude가 틀렸는지를 알면, 비슷한 상황에서 어떤 검증을 추가해야
하는지를 판단할 수 있다.

### 보완이 필요한 관점: 컨텍스트 관리 전략

“좁은 컨텍스트 창”을 한계로 지적하지만, 이를 어떻게 보완했는지에 대한 전략이 없다.
대용량 코드베이스에서 관련 파일만 선별해 제공하는 방법, 이전 대화의 핵심 내용을
요약해 이어가는 방법 등 컨텍스트 관리 전략이 보완됐다면 더 실용적인 가이드가 됐을 것이다.

## 인사이트

### AI 디버깅 파트너의 최적 투입 시점은 “초기 탐색 이후”다

저자가 “초기 조사 이후 협업 파트너로서 가장 유용했다”는 결론은 AI 코딩 어시스턴트의
효과적 사용 패턴에 대한 중요한 통찰이다.
문제를 처음 마주한 순간 AI에게 즉시 질문하는 것과, 스스로 어느 정도 좁혀본 후 AI와
대화하는 것의 결과가 다르다.
초기 탐색 없이 AI에게 질문하면, AI는 모든 가능성을 나열하는 광범위한 응답을 줄 수밖에
없다.
스스로 몇 가지를 확인한 후 질문하면, AI는 더 좁혀진 공간에서 더 정확한 가설을 제안할
수 있다.

이것은 AI와의 협업에서 “컨텍스트 품질이 응답 품질을 결정한다”는 원칙의 실전 사례다.
코드베이스에 대한 맥락, 이미 시도한 것, 관찰된 패턴을 명확히 제공할수록 AI의 가설은
더 정확해진다.
역설적으로, 이 컨텍스트를 잘 정리할 수 있는 시점은 이미 어느 정도 문제를 이해한
이후다.
AI는 처음부터 답을 내놓는 오라클이 아니라, 이미 생각하고 있는 사람이 더 빠르게
생각할 수 있도록 돕는 증폭기에 가깝다.

### UI 라이브러리의 컴포넌트 중첩 문제: 추상화의 대가

Radix UI의 모달과 드롭다운이 각자의 포커스 트랩을 운영하다 충돌한 이 버그는,
추상화(abstraction)의 대가를 보여주는 전형적인 사례다.
각 컴포넌트는 독립적으로 완벽하게 동작한다.
모달만 쓰면 포커스 트랩이 완벽히 동작한다.
드롭다운만 쓰면 포커스 트랩이 완벽히 동작한다.
그런데 둘을 중첩하면 각자의 상태 정리 로직이 서로를 방해한다.

이것은 소프트웨어 컴포넌트가 “조합될 때(composition)” 행동이 달라지는 문제다.
단위 테스트는 개별 컴포넌트를 통과하지만, 조합 테스트는 실패한다.
React 생태계의 헤드리스 UI 라이브러리들이 이 조합 문제를 어떻게 다루느냐는 라이브러리
성숙도의 척도 중 하나다.
`modal={false}` 같은 이스케이프 해치(escape hatch)가 필요하다는 것 자체가,
추상화가 모든 조합 시나리오를 커버하지 못함을 인정하는 것이다.

이 버그가 GitHub 이슈에서 발견됐다는 사실은, 같은 문제를 겪은 사람들이 이미 커뮤니티에서
해결책을 공유했다는 의미다.
추상화가 실패하는 경계 조건은 라이브러리 문서보다 이슈 트래커에 더 정직하게 기록된다.
복잡한 UI 버그를 만났을 때 라이브러리 이슈를 먼저 검색하는 것이 AI 디버깅보다 더 직접적인
경로일 수 있다는 역설이 여기에 있다.

### “과신”은 AI의 버그가 아니라 설계의 긴장이다

저자가 지적한 “잘못된 설명에 대한 과신”은 AI 어시스턴트 설계의 근본적 긴장을 드러낸다.
AI가 확신 없이 답하면 사용자는 불안해한다.
“확실하지 않지만...”이 붙은 답변은 신뢰도가 낮아 보이고, 사용자는 더 명확한 답을 원한다.
그래서 AI는 확신 있게 답하도록 설계되거나 학습된다.
그런데 그 확신이 틀렸을 때, 사용자는 더 큰 혼란을 경험한다.

이 딜레마는 전문가 인간에게도 동일하게 존재한다.
의사가 진단에 확신 없이 말하면 환자는 불안해하고, 확신 있게 말한 진단이 틀리면
더 큰 신뢰 손상이 생긴다.
전문가 시스템은 이 딜레마를 “확신 수준을 명시적으로 보여준다”는 방향으로 해결하려 했다.
“이 진단의 신뢰도는 87%”처럼.
AI 어시스턴트도 이 방향으로 갈 수 있지만, 숫자가 붙는다고 사용자가 그 불확실성을
더 잘 처리하는지는 다른 문제다.
결국 AI와 협업하는 사람이 “이 답변이 틀릴 수 있다”는 태도를 기본으로 장착하는 것이,
도구 자체의 개선만큼이나 중요한 사용자 역량이다.
