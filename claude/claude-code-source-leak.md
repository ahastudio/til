# Claude Code 소스 유출: 가짜 도구, 좌절 감지, 언더커버 모드

> 원문: <https://alex000kim.com/posts/2026-03-31-claude-code-source-leak/>

## 요약

2026년 3월 31일, Anthropic이 npm 패키지에 소스맵(.map) 파일을 실수로 포함시켜 Claude Code CLI의 전체 소스 코드가 유출되었다. 일주일 전 모델 스펙 유출에 이어 두 번째 사고다. 패키지는 이후 제거되었지만, 이미 널리 미러링되고 Hacker News에서 분석되었다.

유출된 코드에서 발견된 주요 내용은 다음과 같다. 첫째, 증류(distillation) 방지 메커니즘으로 가짜 도구 정의를 시스템 프롬프트에 주입하여 API 트래픽을 녹화하는 경쟁사의 학습 데이터를 오염시키는 기능이 있다. 둘째, 언더커버 모드(`undercover.ts`)는 외부 저장소에서 작업할 때 Anthropic 내부 코드명이나 "Claude Code"라는 이름 자체를 언급하지 않도록 모델에 지시한다. 셋째, 정규표현식 기반 사용자 좌절 감지(`userPromptKeywords.ts`)가 욕설과 불만 표현을 패턴 매칭한다.

추가로 네이티브 클라이언트 인증(Zig로 구현된 HTTP 수준 해시), 자동 압축 실패로 인한 일 25만 건의 낭비된 API 호출, 그리고 미공개 자율 에이전트 모드 KAIROS(백그라운드 데몬, GitHub 웹훅, cron 스케줄링 포함)의 스캐폴딩이 발견되었다.

## 분석

글의 구조는 "자극적인 순서"로 발견 사항을 나열하는 전략을 취한다. 이는 기술 블로그보다는 보안 감사 보고서에 가까운 스타일이며, 독자의 관심을 효과적으로 유지한다. 각 발견에 대해 코드 위치(파일명, 줄 번호)를 명시하고, 우회 방법까지 분석한 점은 기술적 깊이를 보여준다.

저자는 OpenCode에 대한 Anthropic의 법적 조치와 연결하여 맥락을 제공한다. 네이티브 클라이언트 인증이 단순한 기술적 선택이 아니라 비즈니스 전략의 기술적 집행이라는 관점은 날카롭다. Bun 인수와 소스맵 버그(oven-sh/bun#28001)를 연결한 분석도 인과관계 추적이 정확하다.

다만 글의 톤이 다소 선정적이고, "스파이시한 순서"라는 프레이밍이 기술적 객관성보다 클릭 유도에 치우친 면이 있다.

## 비평

**강점:** 유출된 소스 코드를 단순히 나열하는 것이 아니라, 각 기능의 비즈니스적 함의와 기술적 한계를 함께 분석한다. 특히 증류 방지 메커니즘의 우회 가능성을 구체적으로 설명하며 "진짜 보호는 법적 수단"이라고 결론 짓는 부분은 정확한 판단이다. 자동 압축 실패 사례(3줄 수정으로 일 25만 API 호출 절약)는 대규모 시스템의 작은 버그가 갖는 경제적 영향을 잘 보여준다.

**약점:** 언더커버 모드에 대해 "AI가 인간인 척 한다"고 프레이밍하지만, 내부 코드명 유출 방지와 AI 사용 은폐는 다른 문제다. 이 구분이 모호하게 처리되어 필요 이상으로 부정적인 인상을 준다. KAIROS에 대해서도 기능 게이트 뒤에 있는 스캐폴딩만으로 "가장 큰 제품 로드맵 노출"이라고 단정하는 것은 과장이다. 또한 Anthropic이 의도적으로 유출했을 수 있다는 음모론을 언급하면서 "아마 아니겠지만"으로 면피하는 것은 책임 있는 보도 태도가 아니다.

에이프릴 풀 기능(타마고치 컴패니언)을 자연스럽게 섞어 넣은 것은 글의 재미를 더하지만, 기술 분석의 신뢰도를 다소 희석시킨다.

## 인사이트

### 증류 방지의 딜레마: 기술적 방어의 구조적 한계

Claude Code의 가짜 도구 주입과 커넥터 텍스트 요약은 모델 증류를 방지하려는 기술적 시도다. 그러나 저자가 지적했듯, 이 방어는 "한 시간 정도 소스를 읽으면" 우회 가능하다. 이는 DRM(디지털 저작권 관리)의 역사적 교훈을 반복한다.

DRM이 실패하는 이유는 "아날로그 홀(analog hole)"이라는 근본적 문제 때문이다. 콘텐츠가 사용자에게 전달되는 순간, 사용자는 그것을 복제할 수 있다. API 트래픽도 마찬가지로, 응답이 클라이언트에 도착한 순간 기록 가능하다. 가짜 도구를 주입해도 트래픽을 기록하는 쪽에서 알려진 도구 목록과 비교하면 가짜를 필터링할 수 있다.

이 딜레마의 핵심은 Anthropic이 두 가지 상충하는 목표를 동시에 추구한다는 점이다. 하나는 개발자에게 강력한 도구를 제공하는 것이고, 다른 하나는 그 도구의 출력이 경쟁사의 학습 데이터로 사용되는 것을 막는 것이다. API 비즈니스의 근본적 긴장 — 접근성과 통제 사이의 균형 — 이 기술적 코드에 직접 반영된 사례다.

장기적으로 증류 방지의 효과적인 전략은 기술적 방어가 아니라, 모델 성능을 지속적으로 향상시켜 증류된 모델과의 격차를 유지하는 것이다. 이는 경주에서 앞서가는 것이지, 뒤쫓는 자의 발목을 잡는 것이 아니다.

### 소스맵 유출의 아이러니: 자신의 도구에 당한 도구 제작자

이 사건에서 가장 주목할 만한 것은 인과 체인이다. Anthropic이 Bun을 인수하고, Claude Code를 Bun 위에 구축했는데, Bun의 알려진 버그(프로덕션 모드에서 소스맵 제공)가 Claude Code의 소스를 유출시켰다. 자신이 만든(인수한) 도구의 결함이 자신의 제품을 노출시킨 셈이다.

이는 소프트웨어 공급망 보안의 더 넓은 교훈을 담고 있다. 현대 소프트웨어는 수백 개의 의존성 위에 구축되며, 각 의존성은 잠재적 취약점이다. Anthropic처럼 기술적으로 정교한 조직도 자신의 도구 체인에 있는 열린 이슈를 놓칠 수 있다. 글에서 언급된 Axios의 악성 버전 사건과 병치하면, 이 문제의 보편성이 더욱 부각된다.

더 넓은 맥락에서, AI 코딩 도구가 코드베이스의 상당 부분을 작성하는 시대에 "AI가 작성한 코드가 AI 도구의 소스를 유출시켰다"는 트위터 댓글은 농담이 아니라 구조적 위험의 징후다. AI 생성 코드의 품질 보증은 아직 해결되지 않은 문제이며, 이 사건은 그 격차의 실제적 결과를 보여준다.

### 정규표현식 vs LLM: 적정 기술의 경제학

좌절 감지에 정규표현식을 사용한 것에 대해 저자는 "LLM 회사가 감성 분석에 정규표현식을 쓴다니"라고 아이러니를 지적한다. 그러나 이것은 아이러니가 아니라 올바른 엔지니어링 판단이다.

Claude Code는 모든 사용자 입력에 대해 실시간으로 반응해야 한다. 감성 분석을 위해 매번 LLM 추론을 실행하면 지연시간과 비용이 발생한다. 반면 정규표현식은 마이크로초 단위로 실행되고 비용이 제로다. 이 맥락에서 필요한 것은 "사용자가 화가 났는지"의 대략적 감지이지, 뉘앙스 있는 감성 분석이 아니다.

이는 기술 선택의 근본 원칙을 보여준다. 문제의 정확한 요구 사항에 맞는 가장 단순한 도구를 선택하라. 자동 압축 실패 사례(3줄 수정으로 일 25만 API 호출 절약)와 합쳐보면, 대규모 시스템의 효율성은 화려한 기술이 아니라 적재적소의 실용적 판단에서 나온다는 것을 알 수 있다.

이 원칙은 특히 AI 도구 생태계에서 중요하다. "AI로 할 수 있다"와 "AI로 해야 한다"는 다른 질문이며, 이 구분을 할 수 있는 것이 엔지니어링 성숙도의 지표다.

### 오픈소스와 유출 사이의 회색 지대

저자는 "Google의 Gemini CLI와 OpenAI의 Codex가 이미 오픈소스"라는 반론을 차단하며, "그들은 에이전트 SDK를 공개한 것이지, 플래그십 제품의 전체 내부 배선을 공개한 것이 아니다"라고 구분한다. 이 구분은 정확하며, AI 도구 기업의 오픈소스 전략에 대한 중요한 통찰을 담고 있다.

SDK 공개는 생태계를 확장하는 전략적 선택이다. 개발자들이 자사 플랫폼 위에 구축하도록 유도한다. 반면 내부 구현의 유출은 통제 불가능한 정보 노출로, 기능 플래그(KAIROS), 보안 메커니즘(클라이언트 인증), 방어 전략(증류 방지)이 모두 드러난다. 전자는 외부 개발자를 향한 인터페이스이고, 후자는 내부 의사결정 과정의 노출이다.

그러나 이 사건이 Anthropic에게 순수하게 부정적이지만은 않을 수 있다. 코드의 보안 검사 수준(23단계 bash 보안, Zsh 특화 위협 모델), 프롬프트 캐시 최적화의 정교함, 터미널 렌더링의 게임 엔진 수준 기법 등은 Anthropic의 기술적 역량을 간접적으로 증명한다. 유출이 결과적으로 기술적 신뢰도를 높이는 역설적 효과를 낳을 수 있다. 물론 이것이 의도된 결과일 가능성은 낮지만, 위기 관리의 관점에서 활용 가능한 측면이다.
