# Claude Code, 프롬프트를 읽기도 전에 33k 토큰을 쓴다

원문: [Claude Code Sends 4.7x More Tokens Than OpenCode Before Reading Your Prompt | Systima Blog](https://systima.ai/blog/claude-code-vs-opencode-token-overhead)

HN 토론: <https://news.ycombinator.com/item?id=48883275> (686점, 378개 댓글)

## 요약

Systima 연구팀은 코딩 하니스와 모델 엔드포인트 사이에 로깅 프록시를
두고, 실제 API로 오가는 JSON 페이로드와 사용량 지표를 그대로 기록하는
방식으로 Claude Code와 OpenCode의 토큰 소비를 비교했다.
Claude Code 2.1.207과 OpenCode 1.17.18을 동일한 초기 설정에서
동일한 작업에 노출시킨 뒤, 지시 파일이나 MCP 서버, 서브에이전트 같은
변수를 하나씩 추가하며 누적 효과를 측정했다.

가장 단순한 작업, 즉 사용자가 “OK”라는 응답만 요구하는 상황에서도
격차는 뚜렷했다.
Claude Code는 사용자 프롬프트가 도달하기도 전에 약 32,800 토큰을
전송했는데, 이는 대부분 시스템 지시문과 27개의 도구 스키마에서
나온다.
반면 OpenCode는 최소화된 시스템 블록과 10개의 도구만으로 약 6,900
토큰을 사용했다.
도구를 아예 비활성화한 상태에서도 시스템 프롬프트 자체가 3배가량
차이 났다.

더 심각한 문제는 캐시 효율이다.
OpenCode는 요청마다 바이트 단위로 동일한 프리픽스를 유지해 캐시를
한 번만 기록하고 이후에는 저렴하게 읽어오는 방식으로 동작했다.
반면 Claude Code는 세션 안에서 서로 다른 세 가지 요청 유형을
만들어내며 각각 별도의 캐시 항목을 발생시켰다.
동일한 요약 작업에서 Claude Code는 작업 도중 캐시를 다시 쓰는
현상까지 포함해 53,839 토큰의 캐시 기록을 만들었지만, OpenCode는
1,003 토큰에 그쳤다.
여러 차례 실행을 반복한 결과 Claude Code의 캐시 기록 비율은
OpenCode 대비 5.9배에서 최대 54배까지 벌어졌다.

72KB 규모의 지시 파일을 추가하면 두 하니스 모두 요청당 2만 토큰
이상이 늘어나 최소 구성 대비 거의 4배로 부풀었고, MCP 서버 5개를
연결하면 약 4,900~6,967 토큰이 추가됐다.
서브에이전트를 2개로 늘리자 토큰 비용은 4.2배로 뛰어 51만 3천
토큰까지 치솟았는데, 이는 서브에이전트 없이 직접 처리했을 때의 12만
1천 토큰과 대비된다.
여러 MCP 서버와 지시 파일을 갖춘 실제 운영 구성에서는 사용자 입력이
도달하기 전부터 이미 75,000~90,817 토큰이 소모됐다.

작성-실행-테스트-수정으로 이어지는 다단계 작업에서는 결과가
반전되기도 했다.
Claude Code의 공격적인 도구 호출 배치 처리 덕분에 초기 테스트에서는
오히려 총 토큰이 더 낮게 나왔지만(12만 1천 대 13만 2천), 최신
모델로 재검증하자 이 우위는 사라지고 Claude Code가 29만 8천 토큰을
쓴 반면 비교 대상은 13만 3천 토큰에 그쳤다.
연구팀은 이 배치 처리 이점이 하니스 자체의 상수가 아니라 모델별
행동 특성에 가깝다고 정리했다.
날짜 유틸리티 구현 같은 별도 품질 벤치마크에서는 두 하니스 모두
5개 테스트를 모두 통과했지만, 동일한 결과를 내기 위해 Claude
Code는 평균 약 268,000 입력 토큰을, OpenCode는 약 72,000 토큰을
사용해 3.7배 차이를 보였고, 소요 시간도 OpenCode의 1~2분에 비해
Claude Code는 4~8분이 걸렸다.

## 분석

### 도구 스키마가 프롬프트 앞에 쌓는 고정비

Claude Code가 사용자 프롬프트를 받기도 전에 32,800 토큰을 쓰는
이유는 27개의 도구 스키마를 매 요청마다 통째로 실어 보내기
때문이다.
이는 모델의 추론 능력과 무관한, 순수하게 하니스 설계에서 비롯된
고정비다.
OpenCode가 10개 도구로 6,900 토큰에 그친다는 사실은 도구 개수와
설명의 장황함이 토큰 소비에 직결된다는 점을 보여준다.

이 고정비는 작업의 복잡도와 무관하게 매 요청마다 반복해서
발생한다는 점에서 특히 문제적이다.
“OK”라고만 답하면 되는 작업에서도 3만 토큰이 넘는 오버헤드가
발생했다는 것은, 하니스가 작업 난이도에 비례하지 않는 구조적 세금을
매기고 있다는 뜻이다.
이는 결국 사용자가 실제로 쓸 수 있는 컨텍스트 창의 크기를 줄이는
결과로 이어진다.

### 캐시 전략의 실패가 진짜 비용을 만든다

토큰 개수 자체보다 더 뼈아픈 지점은 캐시 재사용 실패다.
OpenCode는 프리픽스를 바이트 단위로 고정해 캐시 적중률을
극대화하지만, Claude Code는 세션 중간에 캐시를 다시 쓰는 패턴을
반복한다.
캐시 쓰기는 읽기보다 훨씬 비싼 연산이므로, 5.9배에서 54배에 달하는
캐시 기록 비율 차이는 단순 토큰 수 차이보다 실제 과금액에서 훨씬 큰
격차로 이어진다.

이는 프롬프트 구조의 안정성 문제로 볼 수 있다.
시스템 블록이나 도구 정의가 요청마다 미세하게 달라지면 캐시
프리픽스가 깨지고, 그 순간부터는 캐시가 없는 것과 다름없는 비용을
치른다.
Claude Code에서 세 가지 서로 다른 요청 유형이 관측됐다는 것은
하니스 내부에서 프롬프트 조립 로직이 상황에 따라 달라진다는
뜻이고, 이는 캐시 친화적인 설계 원칙과 충돌한다.

다만 HN에서는 캐시 요금 구조 자체를 짚는 반론도 나왔다.
한 참가자는 32,800 토큰이 최초 호출에서는 비싸더라도 이후 턴부터는
캐시 적중으로 처리될 가능성이 높고, 캐시 적중 토큰은 정가의
10분의 1 수준으로 청구된다는 점을 지적했다.
시스템 프롬프트가 매 턴 새로 발견해야 할 정보를 미리 제공해
작업 성능을 높이거나 에이전트의 자율성을 키워 준다면, 몇 턴 만에
손익분기점을 넘길 수 있다는 것이다.[^eigenblake]
비슷한 맥락에서 다른 참가자는 원문이 스스로 "PUSH_AX의 타당한
지적"이라며 업데이트를 예고한 사실도 언급할 만하다.
계약자 A가 3만 3천 달러를, 계약자 B가 7천 달러를 부른다고 해서
곧바로 B가 더 낫다고 말할 수 없듯이, 원 저자 역시 토큰 수 자체보다
더 깊이 있는 작업과 정성적 결과 비교, 재현 가능한 입출력을
추가하겠다고 댓글로 답했다.[^systima]

### 서브에이전트가 토큰 비용을 기하급수적으로 늘린다

서브에이전트 2개를 추가했을 때 토큰 비용이 4.2배로 뛰었다는 결과는
병렬화가 항상 효율을 뜻하지 않는다는 사실을 드러낸다.
각 서브에이전트가 부모의 시스템 프롬프트와 도구 정의를 다시
상속받아 자신만의 컨텍스트를 새로 구성하기 때문에, 서브에이전트
수가 늘어날수록 고정비도 함께 곱해진다.
이는 HN 스레드에서 언급된 “102개 에이전트가 예기치 않게 생성됐다”는
경험담과 정확히 같은 메커니즘을 가리킨다.

작업을 여러 에이전트로 쪼개는 설계가 유용한 경우도 있지만, 이
글에서 측정한 수치는 그 이득이 토큰 비용 증가를 상쇄할 만큼
확실한지 개별적으로 검증해야 한다는 점을 시사한다.
특히 서브에이전트가 부모와 같은 고비용 모델을 그대로 상속받는
구조라면, 병렬화가 오히려 비용을 곱셈적으로 키우는 결과를 낳는다.

HN 스레드에서도 같은 경험을 전한 사용자가 있다.
한 사용자는 규모가 큰 작업 하나를 맡겼더니 Claude Code가 곧바로
서브에이전트 7개를 실행시켰고, 그중 어느 하나도 끝나기 전에
예산이 소진됐다고 전했다.
다섯 시간 뒤 같은 작업을 다시 시도해도 결과는 동일했지만, 같은
작업을 메인 에이전트가 순차적으로 처리하도록 하면 아무 문제가
없었다고 덧붙였다.
그는 이것이 정말 오케스트레이션 비효율 때문인지, 아니면
서브에이전트를 쓰는 사용자 대부분이 기업 계정에서 토큰당 과금을
하고 있다는 점을 노려 수익을 늘리려는 설계인지 확신할 수 없다고
지적했다.[^mcv]

### 모델 세대 교체가 하니스 비교 결과를 뒤집는다

다단계 작업에서 Claude Code가 처음에는 도구 호출 배치 처리 덕분에
더 적은 토큰을 썼지만, 최신 모델로 재검증하자 오히려 두 배 이상
많은 토큰을 쓰는 결과로 뒤바뀌었다.
연구팀 스스로도 이 우위가 “하니스의 상수가 아니라 모델의 행동
특성”이라고 인정한다.
이는 하니스 비교 실험이 특정 시점의 모델 스냅샷에 강하게 종속된다는
뜻이며, 오늘의 측정치가 다음 모델 업데이트 이후에도 유효하다는
보장이 없다는 뜻이기도 하다.

이 지점은 이 글의 결론을 읽을 때 신중해야 할 이유를 제공한다.
토큰 오버헤드처럼 하니스 설계에 내재한 구조적 비용과, 배치 처리
전략처럼 모델 버전에 따라 출렁이는 행동적 비용을 구분하지 않으면
결론이 쉽게 뒤집힐 수 있다.

## 비평

### 벤치마크 하나로 하니스 전체를 재단한다

이 글의 핵심 수치들은 특정 두 버전, 특정 작업 세트에서 나온
결과다.
Claude Code 2.1.207과 OpenCode 1.17.18이라는 스냅샷은 두 프로젝트
모두 빠르게 버전을 올리는 상황에서 며칠 안에 낡은 자료가 될 수
있다.
실제로 글 안에서도 다단계 작업의 결과가 모델 세대에 따라 뒤집히는
사례를 스스로 보고하는데, 이는 하니스 버전에 따라서도 같은 일이
벌어질 수 있다는 뜻이다.
그럼에도 제목은 “33k 대 7k”라는 단정적인 숫자를 못박아, 마치
영구적인 하니스 특성인 것처럼 읽히게 만든다.

### 프롬프트 크기가 곧 비효율이라는 등식은 성립하지 않는다

HN에서 가장 많은 반박을 받은 지점은 "프롬프트가 작을수록 좋다"는
암묵적 전제다.
한 참가자는 프롬프트 크기만으로 효율을 따진다면 Pi처럼 시스템
프롬프트가 극단적으로 짧은 하니스가 항상 가장 저렴해야 하는데
실제로는 그렇지 않다고 지적하며, 프롬프트 최소주의와 효율을
혼동해서는 안 된다고 짚었다.
그는 도구 품질이 오히려 더 중요한 변수라며, 부실하거나 버그가 있는
도구는 왕복 호출을 늘려 초기의 간결함에서 얻은 이득을 모두
상쇄한다고 덧붙였다.[^godelnumbering]
실제로 pi 하니스의 전체 시스템 프롬프트는 1,000토큰 안팎에
불과하다는 링크도 함께 공유됐다.[^drtournier]

이 반박은 이 글의 분석에 빠진 변수를 정확히 짚는다.
Systima의 실험은 요청 앞단의 토큰 수와 캐시 기록량은 정밀하게
측정했지만, 도구 호출이 실패해서 발생하는 재시도 비용이나 작업
완료까지 걸리는 왕복 횟수는 별도로 통제하지 않았다.
프롬프트가 작아도 도구가 자주 실패하면 왕복이 늘어나 총비용은
오히려 커질 수 있으므로, 초기 오버헤드만으로 하니스의 총비용
효율을 판단하는 것은 성급하다.

### 오픈소스 측정 도구가 객관성을 보장하지는 않는다

연구팀은 측정 방법론과 도구를 공개했다고 강조하지만, 재현
가능성과 결과의 공정성은 다른 문제다.
어떤 작업을 선택하고, 어떤 지시 파일 크기를 기준으로 삼고, 어떤
MCP 서버 조합을 테스트할지는 모두 연구자의 선택이며, 이 선택이
결과의 방향을 얼마든지 좌우할 수 있다.
Systima라는, 지금까지 이 지면에서 언급된 적 없는 신생 조직이 왜
하필 이 비교를 수행했는지, 상업적 이해관계가 있는지에 대한 정보는
글에서 확인되지 않는다.
공개된 측정 도구가 있다는 사실이 곧 편향 없는 실험 설계를
보장하지는 않는다.

한편 HN에는 벤더가 하니스까지 직접 공급하는 구조 자체를 문제
삼는 시각도 있었다.
한 참가자는 모델 제공사가 하니스까지 만드는 것이 이해상충이
아니냐고 물었는데, 사용자의 토큰 비용을 최소화할 동기가 애초에
벤더에게는 없다는 논리다.[^hackingonempty]
비슷하게 다른 참가자는 Claude Code의 토큰 소비가 큰 것은 결국
Anthropic이 그 편이 더 많은 수익을 낳고, 사용자를 자사 구독으로
묶어 두는 효과가 있기 때문이라는 의견을 밝히며, 구독 요금제를 다른
코딩 에이전트에 쓰지 못하게 막는 정책이 그 근거라고 주장했다.[^korrectional]

### 사용자 설정을 하니스 탓으로 돌리는 계산법

72KB 지시 파일과 MCP 서버 5개가 추가하는 토큰은 두 하니스 모두에
거의 동일한 비율로 늘어난다는 점을 글 스스로 인정한다.
그런데도 기사 제목과 결론 문장(“harness sets the floor; your
configuration sets the bill”)은 하니스 간 차이를 전면에 내세운다.
정작 실사용 환경에서 가장 큰 비용 증가는 설정에서 나온다는 것을
스스로 밝혀 놓고도, 헤드라인은 하니스 선택의 문제로 프레이밍하는
모순이 있다.
이는 클릭을 유도하는 제목과 본문 데이터가 가리키는 실제 결론 사이의
간극을 보여준다.

### 캐시 재사용 실패의 원인 진단이 빠져 있다

Claude Code가 세션 중 캐시를 다시 쓰는 이유가 시스템 프롬프트의
동적 삽입 때문인지, 도구 목록이 상황에 따라 바뀌기 때문인지, 아니면
다른 내부 로직 때문인지에 대한 근본 원인 분석은 제시되지 않는다.
숫자로 결과를 보여주는 데는 성공했지만, 그 결과가 의도적 설계
선택인지 최적화되지 않은 버그성 동작인지 구분하지 않은 채
“비효율”이라는 결론으로 건너뛴다.
원인을 모르면 이 문제가 다음 릴리스에서 고쳐질 수 있는 것인지, 아니면
구조적으로 반복될 것인지도 알 수 없다.

## 인사이트

### 사소한 요청조차 과잉 반응하는 현상은 이 글만의 사례가 아니다

HN 댓글 중에는 이 글이 다루는 문제가 Claude Code나 OpenCode에
국한되지 않는다는 지적도 있었다.
한 참가자는 코딩 에이전트 하니스 전반이 사소한 요청에도 점점
공격적으로 도구를 동원하는 경향이 있다며, “안녕”이나 “커밋”
같은 한마디에 30개가 넘는 도구 호출이 발생한 사례를 별도로
측정한 글을 함께 소개했다.
그는 이런 현상을 “토큰플레이션”이라 부르며, 단순 작업이 소비하는
토큰 양이 시간이 갈수록 계속 늘어나는 추세가 뚜렷하다고
평가했다.[^jakozaur]

이 지적은 이 글의 범위를 한정 짓는 동시에 오히려 결론을 강화한다.
Systima의 실험이 Claude Code와 OpenCode 두 하니스에 국한된 것처럼
보이지만, 실제로는 하니스형 코딩 에이전트 전반에서 관찰되는 더 큰
흐름의 한 사례일 가능성이 높다는 뜻이다.
바꿔 말하면 이 글의 개별 수치는 낡아질 수 있어도, “하니스가 작업
난이도와 무관하게 토큰을 과소비하는 경향이 있다”는 더 상위의
패턴은 여러 독립적인 측정에서 반복해서 확인되고 있다.

### 토큰 절약 경쟁이 하니스 시장의 새로운 축이 될 것이다

지금까지 코딩 에이전트 하니스 비교는 주로 작업 성공률이나 응답
품질을 기준으로 이루어졌다.
이 글이 보여주는 것은 동일한 품질(5개 테스트 모두 통과)에서도 비용이
3.7배까지 벌어질 수 있다는 사실이며, 이는 앞으로 하니스 선택
기준에 “토큰 효율”이 품질만큼 중요한 축으로 떠오를 가능성을
시사한다.
특히 EU AI Act 같은 규제 환경에서 감사 가능성과 비용 예측 가능성이
요구되면, 캐시 재사용률이나 요청당 고정비처럼 지금은 부차적으로
취급되는 지표가 조달 기준에 들어갈 수 있다.
현재는 벤치마크 리더보드가 정확도 위주지만, 다음 단계는 정확도 대비
토큰 비용을 함께 보는 방향으로 이동할 가능성이 크다.

### 프록시 계층에서 하니스를 감시하는 관행이 확산될 것이다

이 연구가 사용한 방법, 즉 API 경계에 로깅 프록시를 두고 실제 페이로드를
가로채 감사하는 방식은 이후 조직들이 자체적으로 하니스 비용을
검증하는 표준 관행으로 자리 잡을 가능성이 있다.
지금까지는 각 벤더가 공개하는 토큰 사용량 통계를 그대로 신뢰하는
경우가 많았지만, 하니스가 시스템 프롬프트나 도구 정의를 불투명하게
관리할수록 사용자 조직이 직접 블랙박스를 열어보려는 유인이
커진다.
이는 과거 클라우드 비용 최적화(FinOps)가 발전해 온 경로와
비슷하다.
처음에는 청구서를 그대로 받아들이다가, 비용이 커지자 조직들이 직접
사용량을 계측하고 최적화하는 전문 영역이 생겨난 것처럼, “AI
토큰옵스” 같은 실무 영역이 자리 잡을 수 있다.

### 컨텍스트 창은 공유 자원이고 하니스는 그 지대를 먼저 가져간다

글이 명시적으로 언급하지 않는 긴장 지점은, 도구 스키마와 시스템
지시문이 차지하는 32,800 토큰이 단순히 “비용”만이 아니라 모델이
실제로 사용할 수 있는 유효 컨텍스트 창을 줄이는 효과도 낸다는
점이다.
컨텍스트 창이 20만 토큰이라 해도 하니스가 시작부터 3만 토큰 이상을
가져가면, 사용자가 실제로 쓸 수 있는 몫은 그만큼 줄어든다.
이는 마치 임대 계약에서 관리비가 먼저 공제되고 남는 금액만 실사용
공간에 쓸 수 있는 구조와 비슷하다.
도구를 많이 지원할수록 하니스는 더 유능해 보이지만, 그 유능함을
유지하는 대가로 사용자가 쓸 수 있는 작업 공간 자체가 줄어드는
트레이드오프는 벤더의 기능 목록에는 드러나지 않는다.
결국 “몇 개의 도구를 항상 켜 둘 것인가”라는 설계 결정은 비용
문제이자 동시에 실질적인 추론 능력의 문제이기도 하다.
