# Claude Code 팀이 정리한 세 가지 루프 유형

원문: <https://twitter.com/ClaudeDevs/status/2074208949205881033>

HN 토론: <https://news.ycombinator.com/item?id=48809664> (2점, 1개 댓글)

## 요약

Claude Code 팀 공식 계정(ClaudeDevs)이 2026년 7월 6일에 올린 아티클이다.
“프롬프트하는 대신 루프를 설계하라(designing loops instead of
prompting)”는 말이 X에서 유행하지만, 정작 “루프”가 무엇을 뜻하는지에
대해서는 사람마다 답이 다르다는 문제의식에서 출발한다.

Claude Code 팀은 루프를 “에이전트가 정지 조건이 충족될 때까지 작업
사이클을 반복하는 것”으로 정의하고, 다음 네 가지 기준으로 루프를
분류한다.
어떻게 트리거되는가, 어떻게 멈추는가, 어떤 Claude Code 기본 기능을
쓰는가, 어떤 유형의 작업에 적합한가.

**턴 기반 루프(turn-based loop)** 는 사용자 프롬프트로 트리거되고,
Claude 스스로 작업이 끝났다고 판단하거나 추가 맥락이 필요하다고
판단할 때 멈춘다.
정기적인 일정이 아닌 짧은 작업에 적합하다.
검증 단계를 사람이 매번 확인하는 대신 `SKILL.md`에 절차를 인코딩해
Claude가 스스로 더 많은 부분을 검증하게 만들 수 있다.
글은 프런트엔드 변경을 검증하는 예시 `SKILL.md`를 제시한다.
개발 서버를 띄우고 실제로 화면을 조작해 상태 변화를 확인하고,
콘솔에 새 오류가 없는지 점검하고, Chrome DevTools MCP로 성능
트레이스를 감사하는 4단계 절차이며, 어느 단계든 실패하면 문제를
고치고 1단계부터 다시 시작하도록 명시한다.

**목표 기반 루프(goal-based loop, `/goal`)** 는 실시간 수동 프롬프트로
트리거되고, 목표 달성 또는 최대 턴 수 도달 시 멈춘다.
검증 가능한 종료 조건이 있는 작업에 적합하다.
Claude가 “이 정도면 충분하다”는 판단을 스스로 내리지 않도록,
완료 조건을 사람이 미리 정의하면 별도의 평가자 모델이 매번 그
조건 충족 여부를 판정해 돌려보낸다.
Lighthouse 점수를 90점 이상으로 만들되 5회 시도 후에는 멈추라는
예시가 제시된다.
글은 테스트 통과 개수나 특정 점수 임계값처럼 결정론적(deterministic)
기준일수록 이 루프가 효과적이라고 강조한다.

**시간 기반 루프(time-based loop, `/loop`와 `/schedule`)** 는 지정된
시간 간격으로 트리거되고, 사용자가 취소하거나 작업 자체가 끝날 때
(PR이 병합되거나 큐가 빈다) 멈춘다.
매일 아침 Slack 메시지를 요약하는 것처럼 반복되는 작업이나, PR의
CI 실패 여부처럼 외부 시스템 상태를 주기적으로 확인해야 하는
작업에 적합하다.
간격을 너무 짧게 잡기보다 더 길게 잡거나, 시간이 아닌 이벤트에
반응하도록 설계하라고 권고한다.

글은 모든 작업에 복잡한 루프가 필요하지는 않다고 못박으며, 가장
단순한 해법에서 시작해 필요할 때만 이 패턴들을 선택적으로 적용하라고
마무리한다.

## 분석

### 세 가지 루프는 트리거·정지 조건 두 축의 조합이다

Claude Code 팀의 분류는 언뜻 세 개의 독립된 기능처럼 보이지만,
실제로는 “무엇이 반복을 시작시키는가”와 “무엇이 반복을 멈추는가”라는
두 축의 조합으로 환원된다.
턴 기반은 사용자 프롬프트로 시작해 Claude의 자기 판단으로 멈춘다.
목표 기반은 사용자 프롬프트로 시작해 외부 평가자 모델의 판단으로
멈춘다.
시간 기반은 시계로 시작해 사용자의 취소나 작업 자체의 종결로
멈춘다.

이렇게 축을 분리하면 이 분류가 완전하지 않다는 것도 드러난다.
예를 들어 “이벤트로 트리거되고 평가자 모델로 멈추는” 조합이나
“시계로 트리거되고 Claude 자기 판단으로 멈추는” 조합은 이 셋 중
어디에도 깔끔하게 들어가지 않는다.
실제로 글이 시간 기반 루프의 개선 방법으로 “시간이 아닌 이벤트에
반응하라”고 제안하는 대목은, 이미 네 번째 트리거 축(이벤트 기반)의
필요성을 스스로 인정하는 셈이다.

### 정지 조건을 누가 판단하느냐가 신뢰의 핵심 이동이다

턴 기반 루프에서 목표 기반 루프로 넘어가는 지점에서 벌어지는 가장
중요한 변화는 “충분히 됐다”는 판단의 주체가 바뀐다는 것이다.
턴 기반에서는 작업을 수행한 바로 그 에이전트가 스스로 종료를
선언한다.
목표 기반에서는 별도의 평가자 모델이 그 선언을 매번 검증한다.

이 구조는 소프트웨어 공학에서 익숙한 원칙, 즉 실행자와 검증자를
분리하라는 원칙을 루프 설계에 그대로 적용한 것이다.
같은 모델이 자신의 결과물을 스스로 평가하면 판정 기준이 느슨해질
유인이 생긴다.
작업을 계속하는 것보다 “이만하면 됐다”고 선언하는 쪽이 인지적으로
더 쉬운 경로이기 때문이다.
평가자를 분리하는 순간 이 유인이 사라지고, 종료 기준이 사람이
미리 정한 조건으로 고정된다.
다만 글이 강조하듯 이 구조가 제대로 작동하려면 조건 자체가
결정론적이어야 한다.
평가자 모델도 결국 또 다른 언어모델이므로, 판정 기준이 모호하면
같은 느슨함의 문제가 검증자 쪽으로 자리만 옮길 뿐이다.

### SKILL.md는 검증을 프롬프트에서 절차로 승격시키는 장치다

턴 기반 루프 항목에서 제시되는 `SKILL.md` 예시는 겉보기에는 단순한
체크리스트지만, 실제로 하는 일은 “Claude가 스스로 판단할 여지”를
좁히는 것이다.
“UI 변경을 검증하라”는 지시는 매번 다르게 해석될 수 있지만,
“개발 서버를 켜고 클릭해서 상태 변화를 스크린샷으로 남기고 콘솔
오류를 확인하고 Core Web Vitals를 감사하라”는 지시는 해석의 여지가
거의 없다.

이는 앞서 목표 기반 루프에서 본 것과 동일한 원리를 다른 층위에서
적용한 것이다.
목표 기반 루프가 “언제 멈출지”를 사람이 미리 정해 검증자에게
넘겨주듯, `SKILL.md`는 “무엇을 확인해야 하는지”를 사람이 미리
정해 실행자에게 넘겨준다.
결국 이 아티클 전체를 관통하는 설계 원칙은 하나로 수렴한다.
에이전트의 판단 재량이 클수록 실패 유인도 커지므로, 재량이 필요한
지점마다 사람이 미리 정한 명시적 기준으로 대체하라는 것이다.

## 비평

### “복잡한 루프가 필요 없다”는 조언과 세 가지 루프 체계 자체가 긴장한다

글의 마지막 문장은 “모든 작업에 복잡한 루프가 필요한 것은 아니니
가장 단순한 해법에서 시작하라”고 말한다.
그러나 이 조언은 글 전체가 공들여 쌓아 올린 세 가지 루프 분류
체계, 트리거·정지 조건의 정교한 구분, `SKILL.md` 검증 패턴,
결정론적 조건 설계 같은 내용과 방향이 어긋난다.

한 문단으로 “단순하게 시작하라”고 말한다고 해서, 독자가 실제로
가장 단순한 해법에 머무를 가능성은 낮다.
방금 세 가지 정교한 패턴을 배운 독자는 자연스럽게 그 패턴을
적용할 자리를 찾게 된다.
도구가 있으면 그 도구를 쓸 문제를 찾게 되는 경향은 소프트웨어
공학에서 반복적으로 관찰되는 현상이며, 이 아티클이 그 유인을
줄이는 데 실질적으로 기여하는 장치는 마지막 한 문장뿐이다.

### 결정론적 조건의 예시들은 정작 결정론적이지 않은 영역까지 낙관한다

목표 기반 루프 항목은 “테스트 통과 개수나 점수 임계값처럼
결정론적인 기준일수록 효과적”이라고 말하며 Lighthouse 점수를
예로 든다.
이 예시 자체는 타당하다.
그러나 실무에서 마주치는 대다수의 “목표”는 이렇게 깔끔한 숫자로
환원되지 않는다.
“사용자 경험을 개선하라”, “코드를 더 읽기 쉽게 만들라”, “버그를
찾아 고쳐라” 같은 목표는 결정론적 임계값이 없다.

글은 이런 비결정론적 목표에 대해서는 침묵한다.
평가자 모델이 이런 목표를 판정하는 순간, 앞서 지적한 “같은
느슨함의 문제가 검증자 쪽으로 옮겨가는” 위험이 그대로 재현된다.
결정론적 조건을 설계할 수 있는 작업이 얼마나 되는지, 그리고
그렇지 않은 작업에서는 목표 기반 루프를 어떻게 안전하게 쓸 수
있는지에 대한 논의가 빠져 있다는 점은 이 글의 실용적 가치를
좁히는 공백이다.

### 시간 기반 루프의 정지 조건은 다른 두 루프보다 훨씬 허술하다

턴 기반과 목표 기반 루프는 각각 “Claude의 판단”과 “평가자 모델의
판단”이라는 나름의 정지 메커니즘을 갖는다.
반면 시간 기반 루프의 정지 조건은 “사용자가 취소하거나 작업이
끝난다”로 제시되는데, 이는 사실상 정지 메커니즘이 없다는 뜻에
가깝다.

PR이 병합되거나 큐가 빈다는 조건은 외부 시스템의 상태 변화에
의존하므로, 그 사이 루프는 무기한 반복될 수 있다.
글이 예로 든 “매일 아침 Slack 메시지 요약”처럼 애초에 끝이
없는 작업이라면 이 루프에는 설계상 종료 시점이 존재하지 않는다.
앞의 두 루프 유형이 정교하게 다듬은 “언제 멈출 것인가”라는
질문이, 세 번째 유형에서는 사실상 사람의 수동 개입에 전적으로
떠넘겨진다는 점은 이 분류 체계의 균형이 고르지 않다는 신호다.

## 인사이트

### 세 루프 유형은 사실 하나의 시스템이 성숙해 가는 단계다

이 세 가지 루프를 서로 다른 용도의 병렬적 선택지로 읽기보다,
하나의 작업이 자동화되어 가는 시간적 궤적으로 읽으면 더 많은
것이 보인다.
처음에는 사람이 매 턴 개입하는 턴 기반 루프로 작업을 시작한다.
그 작업의 종료 조건이 명확해지면 목표 기반 루프로 넘겨 사람의
개입 빈도를 낮춘다.
그 작업이 반복되는 패턴임이 확인되면 시간 기반 루프로 넘겨
트리거 자체를 자동화한다.

이렇게 보면 세 루프의 실제 차이는 “어떤 작업에 어떤 루프를
쓸까”가 아니라 “한 작업에 대한 사람의 신뢰와 이해가 쌓일수록
그 작업을 어느 단계까지 자동화 파이프라인에 흘려보낼 수
있는가”라는 질문이 된다.
이는 소프트웨어 배포 파이프라인이 수동 배포에서 스테이징 자동화를
거쳐 완전한 CI/CD로 성숙해 가는 궤적과 구조적으로 닮았다.
루프 유형의 선택은 작업의 본질적 속성이 아니라, 그 작업에 대한
조직의 성숙도를 드러내는 지표에 가깝다.

### 에이전트 자율성 논의의 무게중심이 “능력”에서 “계약”으로 옮겨간다

이 아티클이 실질적으로 보여주는 것은 언어모델의 성능이 아니라
사람과 에이전트 사이의 “계약(contract)” 설계다.
`SKILL.md`로 검증 절차를 명시하는 것, 목표 기반 루프에서 완료
조건을 미리 정의하는 것, 결정론적 임계값을 선호하는 것 모두
에이전트의 지능을 더 끌어올리려는 시도가 아니라, 에이전트가
자유롭게 해석할 수 있는 여지를 계약으로 좁히는 시도다.

이는 에이전트 코딩의 초기 담론이 “모델이 얼마나 똑똑한가”에
집중했던 것과 대비된다.
지금 실무에서 실제로 신뢰성을 개선하는 작업은 모델을 더
똑똑하게 만드는 것이 아니라, 모델이 무엇을 해야 하고 언제
멈춰야 하는지를 사람이 더 명시적으로 계약해 두는 것이다.
이 흐름이 계속되면, 에이전트 도구의 경쟁력은 기반 모델의 성능
차이보다 이런 계약을 얼마나 정교하고 사용하기 쉽게 만들 수
있는가로 옮겨갈 가능성이 크다.

### 루프 설계 담론의 확산은 에이전트 실패의 책임 소재를 재배치한다

“프롬프트 대신 루프를 설계하라”는 표어가 확산되는 현상 자체를
살펴볼 필요가 있다.
프롬프트 시대에는 결과가 나쁘면 “프롬프트를 잘못 썼다”는 개인의
실수로 귀속되기 쉬웠다.
루프 시대에는 결과가 나쁘면 “루프 설계가 잘못됐다”, 즉 트리거·정지
조건·검증 절차 같은 구조적 요소의 결함으로 귀속된다.

이는 실패의 원인을 개인의 순간적 판단에서 재사용 가능한 설계
산출물로 옮기는 변화다.
장점은 명확하다.
설계가 문서화되고 재사용되면 같은 실수가 반복되지 않는다.
그러나 동시에 이 변화는 새로운 위험을 만든다.
루프 설계 자체가 틀렸을 때, 그 루프가 무인으로 반복 실행되며
같은 실수를 사람이 개입하기 전까지 계속 증폭시킬 수 있다는
점이다.
프롬프트 하나의 실패는 그 자리에서 끝나지만, 루프 하나의 실패는
정지 조건이 충족되기 전까지 계속 반복된다.
루프 설계가 정교해질수록, 그 설계의 결함을 조기에 발견하는
장치(예: 이상 반복 횟수 감지, 비용 상한)의 중요성도 함께
커진다는 점을 이 아티클은 다루지 않는다.
