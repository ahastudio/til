# Claude Code와 Codex 설정 변경으로 토큰 절약하기

원문: <https://stdy.blog/posts/how-to-save-tokens-with-claude-code-and-codex-settings/>

## 요약

Claude Opus 4.7 릴리스 노트에서 토큰 소비 증가가 언급되면서, 하네스(harness) 수준의
설정 튜닝으로 실제 토큰 비용을 줄일 수 있는 방법을 정리한 글이다.
저자는 Claude Code 캐시 TTL이 1시간에서 5분으로 단축된 점, Opus 4.7이 4.6 대비 약
1.5배의 토큰을 소비하는 경향, 그리고 캐싱 효율 저하에 관한 기존 연구를 배경으로
제시한다.

토큰 누수의 주된 원천은 세 가지로 지목된다.
첫째, 세션 또는 턴마다 자동으로 덧붙는 안내 텍스트.
둘째, 툴 호출 결과가 대화 히스토리에 길게 남는 현상.
셋째, 검색·커넥터·IDE 등 외부 통합이 유발하는 추가 호출이다.
각 원천은 프롬프트 길이를 비선형적으로 팽창시키며, 캐시 미스와 맞물리면 비용이
급격히 불어난다.

Claude Code(v2.1.114) 최적화 권장사항은 git 지시문과 IDE 자동 연결 비활성화,
bash·파일 읽기·MCP 출력 제한 설정, 환경변수로 MCP 서버·자동 메모리·CLAUDE.md
파일 비활성화, 필요한 도구만 선택적으로 활성화, 불필요한 attribution 필드 정리다.
Codex CLI(v0.121.0) 측은 연결된 MCP 앱과 웹 검색 비활성화, 툴 출력 토큰 제한,
`--profile`, `--json`, `--sandbox read-only`, `--ephemeral` 같은 플래그를
조합해 읽기 전용·임시 실행 모드로 자동화 안전성을 높이는 전략을 제안한다.

GeekNews 댓글에서는 `claude-slim`이라는 보조 CLI가 소개되었다.
미사용 스킬과 비대해진 `CLAUDE.md`를 삭제 없이 스캔·재조직하고 JS-tiktoken으로
토큰 수를 계산해 경량화한다.
한편 “토큰 제한이 너무 빡빡해 하네스 실행이 끊긴다”는 불만도 있었고, 저자 본인도
Claude의 토큰 소비가 부담스러워 Codex를 선호한다고 답했다.

## 분석

### 하네스는 모델이 아니라 비용의 단위다

이 글이 흥미로운 지점은 “모델 선택”이 아니라 “하네스 설정”을 비용 단위로 본다는
점이다.
같은 Opus 4.7이라도 Claude Code 기본값으로 쓰느냐, IDE 연결과 MCP를 모두 끄고
쓰느냐에 따라 실효 토큰 사용량이 수 배 차이가 난다.
즉 모델 가격표는 상한선일 뿐 실제 지출은 하네스 구성이 결정한다.

### 캐시 TTL 단축의 2차 효과

Claude Code의 캐시 TTL이 1시간에서 5분으로 줄면, 겉으로는 “캐시 가격만 약간 오른
셈”이다.
그러나 실제로는 긴 세션에서 캐시 히트가 대부분 5분 벽에 부딪혀 무효화되므로,
프롬프트 재구성 빈도가 급등한다.
여기에 자동 삽입되는 안내문, 긴 툴 출력, 자동 메모리가 얹히면 매 5분마다 풀 프롬프트
재전송에 가까운 비용이 발생한다.
저자가 “자동 메모리·CLAUDE.md·IDE 자동 연결”을 먼저 끄라고 하는 이유는 단순
경량화가 아니라 캐시 재구성 비용 폭발을 막기 위함이다.

### MCP는 양날의 검

MCP는 외부 도구 연결을 표준화해 생산성을 끌어올렸지만, 각 MCP 서버는 툴 설명과
스키마를 시스템 프롬프트에 주입한다.
서버가 10개면 설명도 10개가 붙는다.
필요할 때만 활성화하는 운영이 토큰 예산을 좌우한다.

## 비평

### 강점: 구체적 플래그 레벨 권장

대부분의 “AI 비용 절감” 글은 “프롬프트 짧게 쓰세요” 수준에 머무른다.
이 글은 실제 CLI 플래그와 환경변수, 버전 번호까지 명시해 재현 가능성이 높다.
Codex의 `--ephemeral` + `--sandbox read-only` 조합 같은 조언은 실무에 바로 적용할
수 있다.

### 약점: 절약량 수치가 없다

아쉬운 점은 “몇 % 절약됐다”는 벤치마크가 제시되지 않는다는 것이다.
독자는 각 설정의 효과를 직접 측정해야 한다.
특히 자동 메모리 비활성화는 기능성 저하를 동반하므로 비용 절감 대비 생산성
손실에 대한 정량 비교가 필요하다.

### 누락된 논의: 설정 드리프트

하네스 설정은 업데이트마다 기본값이 바뀐다.
v2.1.114 기준 권장값이 v2.2에서도 유효하다는 보장이 없다.
“한 번 튜닝하고 끝”이 아니라 릴리스 노트를 지속 추적해야 한다는 경고가
빠져 있다.

## 인사이트

### 모델 가격이 아니라 하네스 가격의 시대

그동안 LLM 비용 논의는 “입력 100만 토큰에 얼마”라는 모델 가격표 중심으로
이뤄졌다.
그러나 실무에서 지불하는 금액은 모델 단가 × 실제 전송 토큰이고, 실제 전송
토큰은 하네스가 결정한다.
자동 메모리, CLAUDE.md 자동 로드, MCP 서버 10개, IDE 자동 연결을 모두 켠 상태에서
“Opus 4.7이 비싸다”고 하는 것은 측정 지점을 잘못 잡은 것이다.

이 구조는 클라우드 초창기 EC2 온디맨드 단가만 보고 AWS 비용을 예측하려던
시도와 닮았다.
실제 청구서는 EBS, NAT 게이트웨이, 교차 AZ 트래픽이 합쳐져 예상의 수 배가 되곤
했다.
AI 시대의 “숨은 비용”도 같은 패턴이다.
청구서가 “모델” 라인 하나라고 해서 비용 드라이버도 하나는 아니다.

따라서 앞으로 AI 인프라 팀의 핵심 역량은 모델 선정이 아니라 하네스 감사(audit)와
비용 어트리뷰션이 된다.
어떤 설정이 토큰을 먹는지, 어떤 툴이 출력 길이를 폭발시키는지, 어떤 커넥터가
시스템 프롬프트를 늘리는지 추적할 수 있어야 한다.
FinOps가 클라우드에서 그랬듯, AIOps/LLMOps에서도 세밀한 귀속이 직업의 축이
된다.

### 캐시는 성능 최적화가 아니라 경제의 기반이다

Claude Code의 캐시 TTL이 1시간에서 5분으로 줄었다는 사실만 놓고 보면
“기술적 파라미터 조정”처럼 보인다.
그러나 이는 AI 제품의 단가 구조를 사용자 모르게 흔드는 변화다.
5분을 넘기는 대화나 긴 에이전트 루프는 거의 항상 프롬프트 재전송에 해당하므로,
실질적으로 캐시 할인 혜택이 증발한다.

캐시는 단순 속도 개선 수단이 아니라 토큰당 실효 단가를 결정하는 경제적 레버다.
벤더가 TTL·캐시 키·캐시 크기를 바꾸는 것은 광고 없이 가격을 올리는 행위와
동일한 경제 효과를 낸다.
이 글이 조용히 드러내는 정치학은 “가격은 그대로인데 비용은 올랐다”는 감각이다.

이는 앞으로 LLM 소비자 측에서 “캐시 SLA”를 요구하게 만들 것이다.
모델 가격 고시만으로는 부족하고, 캐시 TTL, 프롬프트 캐시 최소 단위, 캐시 비용
비율이 계약 수준에서 명시되어야 예측 가능한 예산이 가능하다.
현재는 공급자가 일방적으로 변경 가능한 파라미터인데, 이는 기업 고객에게 지속
가능하지 않다.

### “자동으로 추가되는 것”은 항상 비용이다

Claude Code의 자동 메모리, CLAUDE.md 자동 로드, IDE 자동 연결, 자동 attribution
— 저자가 끄라고 권하는 항목들의 공통점은 “자동으로 추가되는” 기능이라는 것이다.
이들은 UX 측면에선 마찰을 줄이지만, 비용 측면에선 사용자가 모르는 사이 프롬프트를
부풀린다.

소프트웨어 역사에서 “자동”은 대부분 숨은 비용의 동의어였다.
자동 색인은 디스크 IO 비용, 자동 동기화는 네트워크 비용, 자동 저장은 스토리지
비용을 발생시킨다.
LLM 시대의 “자동”은 토큰 비용이다.
그리고 토큰은 캐시 미스와 결합할 때 비선형으로 증폭된다.

이 관찰의 실천적 귀결은 명확하다.
새 AI 도구를 도입할 때 가장 먼저 확인할 질문은 “어떤 기능이 자동으로 활성화되어
매 요청마다 프롬프트에 섞이는가”이다.
자동은 편리함이 아니라 계약 조항이다.
읽지 않고 동의 버튼만 누르면 다음 달 청구서에서 비용을 치르게 된다.
