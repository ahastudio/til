# Claude Code Artifacts

<https://code.claude.com/docs/en/artifacts>

## 요약

Claude Code의 Artifacts(아티팩트) 기능은 세션에서 생성된 결과물을 라이브 인터랙티브 웹 페이지로 발행해
조직 내 비공개 URL로 공유할 수 있게 하는 기능이다.
단순히 터미널 텍스트로 전달하기 어려운 시각적 출력 — 주석 달린 diff, 대시보드, 디자인 비교, 진행 타임라인 — 을
한 페이지에 담아 팀원에게 링크로 전달한다.
현재 Team 또는 Enterprise 플랜에서만 이용 가능하며 베타로 제공 중이다.

아티팩트는 Claude Code 세션이 프로젝트 코드베이스와 연결된 도구들을 통해 수집한 데이터를 바탕으로
HTML 또는 Markdown 파일을 작성한 뒤 claude.ai에 발행하는 방식으로 동작한다.
Claude가 스스로 판단해 아티팩트를 생성하기도 하고, 사용자가 직접 요청할 수도 있다.
발행된 URL은 고정되며, 이후 Claude가 같은 파일을 수정하고 재발행하면 같은 URL에서 자동으로 업데이트된다.
페이지를 열어둔 사람은 새로고침 없이 변경 사항을 확인할 수 있다.

공유 범위는 조직 내로 제한된다.
아티팩트를 본 사람은 작성자 아바타와 이름을 확인할 수 있으며, 열람만 가능하고 편집은 불가능하다.
조직 외부로 내용을 공유하려면 Claude에게 HTML 파일 자체를 요청해 별도로 전달해야 한다.
버전 관리도 지원해, Share 컨트롤에서 공유 중인 버전을 선택할 수 있다.

기술적으로 아티팩트는 엄격한 Content Security Policy(CSP)가 적용된 단일 정적 HTML 페이지다.
외부 리소스 요청이 차단되어 CSS와 JavaScript는 인라인으로 삽입하고 이미지는 데이터 URI로 임베딩한다.
백엔드가 없으므로 폼 데이터 저장이나 실시간 API 호출은 불가능하고, 상대 경로 링크도 해결되지 않는다.
파일 형식은 `.html`, `.htm`, `.md`만 허용되며 렌더링된 페이지 크기는 16 MiB 이하여야 한다.

## 분석

### “터미널 텍스트 이상의 것”이라는 포지셔닝의 논리

이 기능의 핵심 전제는 “터미널 텍스트가 잘못된 매체(wrong medium)”가 되는 시점이 존재한다는 것이다.
Claude Code는 원래 CLI 환경에서 동작하지만, AI가 생성하는 결과물의 복잡도가 높아질수록
텍스트 출력만으로는 소통 효율이 떨어진다는 관찰이 이 기능을 만든 동기다.
문서는 이를 “줄 단위로 읽기보다 보고 상호작용하기 쉬운 출력”이라 표현하는데,
이는 Claude Code가 단순한 코드 편집 도구에서 팀 협업 도구로 역할을 넓히려는 방향을 명확히 드러낸다.

이 포지셔닝은 특정 사용자 경험의 병목을 겨냥한다.
PR 리뷰어에게 컨텍스트를 설명할 때, 여러 디자인 안을 비교할 때, 장시간 작업의 진행 상황을 공유할 때 —
이 순간들은 모두 결과를 “전달”하는 문제이지 “생성”하는 문제가 아니다.
Claude Code가 이미 생성한 결과물을 어떻게 팀과 공유하느냐는 문제를 아티팩트가 담당한다.

### 정적 페이지라는 제약을 설계 원칙으로 전환하는 방식

이 기능의 흥미로운 점은 기술적 제약 — CSP, 단일 페이지, 백엔드 없음 — 을 숨기지 않고
기능 설명의 일부로 명시한다는 것이다.
“아티팩트는 작업의 캡처이지 애플리케이션이 아니다(an artifact is a capture of work, not an application)”라는
표현이 이를 잘 보여준다.
이 프레이밍은 사용자가 아티팩트에 “무엇을 기대해야 하는가”를 정확히 설정한다.

CSP가 모든 외부 요청을 차단한다는 설계는 보안과 단순성을 동시에 확보한다.
Claude가 CSS와 JavaScript를 인라인으로 삽입하고 이미지를 데이터 URI로 임베딩하는 방식은
외부 의존성 없이 자기 완결적인 페이지를 만든다는 의미다.
이는 토큰 비용 증가라는 트레이드오프를 수반하지만, 호스팅 인프라를 단순하게 유지할 수 있다는 이점도 있다.

### 세션-아티팩트-팀 루프로 구성된 협업 패턴

문서에서 제시하는 프롬프트 패턴들은 단순한 사용 예시가 아니라
Claude Code를 중심에 둔 팀 워크플로를 묘사한다.
“Copy as prompt” 버튼을 아티팩트에 넣어서 인터랙션 결과를 다시 세션으로 가져오는 패턴은
특히 주목할 만하다 — 아티팩트가 팀원의 결정을 수집하는 인터페이스가 되고,
그 결과가 Claude 세션으로 돌아와 다음 행동의 입력이 되는 루프가 형성된다.

이 루프 구조는 “AI 세션 → 결과 공유 → 팀 피드백 → 다음 세션”이라는
협업 흐름을 Claude Code 안에서 완결하려는 설계 의도를 보여준다.
현재는 수동 붙여넣기를 요구하지만, 이 패턴이 성숙하면
팀의 결정 사항이 세션 컨텍스트에 자동으로 통합되는 방향으로 발전할 수 있다.

## 비평

### 기능의 가용성 제약이 실질적 사용 범위를 크게 좁힌다

문서의 Availability 섹션은 아티팩트를 사용하기 위한 조건을 다섯 가지로 열거한다.
Team 또는 Enterprise 플랜, `/login`으로 인증된 세션, Anthropic API(Bedrock·Vertex·Foundry 불가),
특정 조직 정책(CMEK·HIPAA·ZDR 미적용), Claude Code CLI 또는 데스크탑 앱.
이 다섯 조건이 모두 충족되어야 기능이 활성화된다.

이는 실질적으로 기능을 사용할 수 있는 대상을 매우 좁게 제한한다.
많은 기업들이 데이터 보안이나 규정 준수 이유로 Bedrock이나 Vertex를 통해 Claude를 사용하거나
CMEK·HIPAA 정책을 적용하는데, 이 경우 아티팩트는 아예 사용할 수 없다.
문서는 이 제약을 표로 정리해 투명하게 공개하고 있지만,
기능의 접근성이 이렇게 좁다면 “공유의 편의”라는 핵심 가치를 얼마나 많은 팀이 실제로 누릴 수 있는지 의문이다.

### 아티팩트의 “보안 경계”는 조직 범위를 벗어나는 순간 작동하지 않는다

문서는 아티팩트 공유 범위가 조직 내부로 제한된다는 점을 강조한다.
하지만 “조직 외부 공유는 HTML 파일을 직접 전달하라”는 우회 방법을 공식적으로 제시한다.
이 HTML 파일에는 아티팩트 전체 내용 — 코드, 데이터, 분석 결과 — 이 포함될 수 있다.

즉, 보안 경계는 플랫폼 레벨에서만 작동하고, 파일 레벨에서는 사용자 재량에 완전히 맡겨진다.
PR 리뷰어가 외부 계약자인 경우, 또는 아티팩트에 민감한 코드베이스 정보가 포함된 경우,
“HTML 파일로 공유”라는 방법은 의도치 않은 정보 유출 경로가 될 수 있다.
문서는 이 위험을 전혀 언급하지 않으며, 조직 정책 레벨의 허용 여부도 다루지 않는다.

### 토큰 비용 문제를 사용자 최적화 책임으로 전가한다

문서는 “아티팩트 생성은 일반 응답보다 토큰 집약적”이라고 인정하며
절감 방법으로 SVG 선호, 불필요한 인터랙티브 요소 제거, 대용량 데이터셋 요약을 제안한다.
그런데 이 제안들은 모두 사용자가 프롬프트 설계 단계에서 미리 최적화해야 한다는 전제를 깔고 있다.

실제로 아티팩트가 얼마나 더 많은 토큰을 소비하는지에 대한 구체적인 정보는 없다.
팀이 대시보드 아티팩트를 정기적으로 업데이트하거나 인터랙티브 컨트롤이 많은 페이지를 만들면
토큰 비용이 빠르게 누적될 수 있지만, 문서는 이를 추상적인 주의사항 수준으로만 다룬다.
Compliance API를 통해 아티팩트 목록과 삭제는 지원하지만 사용량 추적 도구는 언급되지 않는다.

## 인사이트

### 아티팩트는 AI 세션의 “출력 형식 혁신”이 아니라 팀 커뮤니케이션 레이어의 확장이다

기능을 표면적으로 보면 HTML 출력 기능처럼 보이지만,
실제 의미는 AI 세션의 출력물이 팀 커뮤니케이션의 일급 시민(first-class citizen)이 되는 전환점이다.
기존에 팀이 코드 리뷰나 디자인 비교를 공유하는 방식은 텍스트 설명, 스크린샷, 별도 문서 도구였다.
아티팩트는 AI 세션에서 생성된 결과물이 별도의 포맷 변환 없이
직접 팀 커뮤니케이션 채널로 진입하는 경로를 만든다.

이 전환의 이차적 효과는 AI 세션의 “결과”에 대한 책임 소재가 명확해진다는 점이다.
아티팩트 헤더에 작성자 아바타와 이름이 표시된다는 설계는 의미심장하다.
“Claude가 만든 것”이 아니라 “사람이 Claude를 통해 만든 것”이라는 프레이밍이
조직 내 AI 출력물의 책임 귀속 방식을 형성한다.
이는 AI 협업 도구가 팀 문화에 통합될 때 어떤 책임 모델이 필요한지를 선제적으로 제안한다.

### 정적 페이지 제약은 현재의 한계가 아니라 설계 선택이며, 그 이유가 중요하다

아티팩트가 정적 페이지로 제한된 것을 기술적 미숙함으로 읽으면 안 된다.
백엔드 없는 단일 페이지 설계는 Anthropic이 추가 인프라 없이 기능을 제공할 수 있게 하고,
CSP 기반 격리는 아티팩트가 claude.ai 도메인에서 실행되면서도 보안 위험을 최소화하는 방법이다.
이는 “무엇이 가능한가”보다 “무엇을 안전하게 운영할 수 있는가”를 기준으로 한 결정이다.

흥미로운 역사적 유사점은 초기 웹 앱의 진화다.
초기 웹 페이지는 정적이었고, 동적 기능은 점진적으로 추가되었다.
아티팩트도 같은 경로를 밟을 수 있다 — 현재는 정적이지만,
팀 내 채택이 충분히 이루어지면 제한된 서버사이드 기능이나 지속적 데이터 연결이
다음 단계로 추가될 수 있다.
그 시점이 언제인지는 조직 내 아티팩트 사용 패턴이 결정할 것이다.

### “세션에서 아티팩트로, 아티팩트에서 세션으로”의 루프가 성숙하면 AI 주도 의사결정 방식이 바뀐다

문서에서 가장 주목할 패턴은 “Copy as prompt” 버튼이다.
사용자가 아티팩트의 트리아지 보드에서 이슈를 드래그해 배치하면,
그 결정을 프롬프트 텍스트로 변환해 다시 Claude 세션에 붙여넣을 수 있다.
이 작은 기능이 담고 있는 암시는 크다: 팀의 의사결정 결과가 구조화된 형태로
AI 세션의 다음 입력이 된다는 피드백 루프다.

현재는 이 루프가 수동 복사-붙여넣기를 요구하지만, 이 패턴이 성숙하면
팀 회의에서 내린 결정이 실시간으로 AI 세션의 컨텍스트로 주입되는 시스템이 가능하다.
이는 단순한 출력 공유 도구를 넘어서, AI 세션이 팀의 집단 의사결정 과정에
능동적으로 연결되는 방향을 제시한다.
이 루프가 완성되는 시점에서 “AI가 팀을 보조한다”는 모델은
“팀이 AI 세션을 중심으로 조율된다”는 모델로 전환될 수 있다 — 이는 관계의 역전이다.
