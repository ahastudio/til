# 대규모 코드베이스에서 Claude Code가 작동하는 방식

원문: <https://claude.com/blog/how-claude-code-works-in-large-codebases-best-practices-and-where-to-start>

HN 토론: <https://news.ycombinator.com/item?id=48144494> (241점, 158개 댓글)

## 요약

Claude Code는 대규모 코드베이스를 탐색할 때 RAG(Retrieval-Augmented Generation) 방식의 인덱스를 구축하는 대신,
파일 시스템 탐색과 grep 검색을 반복하는 “에이전틱 검색(agentic search)” 방식을 채택한다.
이 접근법은 항상 최신 상태의 코드베이스를 직접 참조하기 때문에 인덱스 스테일 문제를 원천 차단한다.

실제 성능을 결정하는 것은 모델 자체가 아니라 모델을 감싸는 “하네스(harness)”라는 것이 핵심 주장이다.
하네스는 `CLAUDE.md` 파일, 훅(hooks), 스킬(skills), 플러그인(plugins), LSP 통합, MCP 서버, 서브에이전트 등
다섯 가지 확장 계층으로 구성된다.
이 계층들을 얼마나 잘 구성하느냐가 Claude Code의 실제 품질을 좌우한다.

`CLAUDE.md` 파일은 프로젝트 루트에서 전체 개요를 제공하고, 하위 디렉터리에서 지역 규약을 보완하는 계층 구조로 관리해야 한다.
파일이 너무 길어지면 성능이 저하되므로, 루트에는 포인터만 남기고 세부 내용은 서브디렉터리 파일에 위임하는 방식이 권장된다.

조직 차원에서 성공한 팀들은 공통적으로 세 가지 패턴을 따른다.
첫째로 코드베이스를 탐색 가능하게 만드는 린(lean)하고 계층적인 `CLAUDE.md` 구조,
둘째로 3~6개월마다 구성을 검토해 이전 모델에 최적화된 지시가 현 모델을 오히려 제약하지 않도록 갱신,
셋째로 구성·권한·플러그인·`CLAUDE.md` 규약을 총괄하는 전담 DRI(Directly Responsible Individual) 또는 소규모 팀의 지정이다.

LSP 통합은 문자열 기반 grep 대신 심볼 기반 탐색을 가능하게 해 정밀도를 크게 높이며,
MCP 서버는 내부 API와 도구를 Claude Code에서 직접 호출할 수 있도록 노출한다.
서브에이전트는 독립적으로 탐색을 수행하고 결과를 메인 에이전트에게 전달해, 병렬 작업과 컨텍스트 격리를 동시에 실현한다.

## 분석

### 하네스 우선 설계의 의미

이 글의 핵심 테제는 “하네스가 모델만큼 중요하다”는 것이다.
이는 모델 자체의 한계를 소프트웨어 엔지니어링으로 보완할 수 있음을 시사한다.
`CLAUDE.md`는 단순한 문서가 아니라, 모델이 코드베이스를 탐색하는 방식을 형성하는 실행 컨텍스트다.
훅은 이 컨텍스트를 작업 도중에도 동적으로 갱신하는 메커니즘으로 기능한다.

### 에이전틱 검색의 트레이드오프

RAG 인덱스 없이 매번 파일 시스템을 탐색하는 방식은 신선도(freshness) 면에서 확실한 이점이 있다.
그러나 동일한 파일을 반복 탐색하는 과정에서 토큰을 낭비할 수 있고,
대규모 모노레포에서는 탐색 비용이 급격히 증가할 수 있다.
`.ignore` 파일로 생성 코드와 서드파티 의존성을 제외하는 것이 이 비용을 제어하는 주요 수단이다.

HN에서는 이 접근법의 타당성을 놓고 의견이 엇갈렸다.
jwilliams는 "소프트웨어 엔지니어처럼 작동한다"는 주장이 자기모순이라고 지적했다.[^jwilliams]
숙련된 엔지니어는 LSP나 자체 구축한 코드베이스 멘탈 모델을 활용하는데,
이는 Claude Code가 회피한다고 주장하는 인덱싱과 본질적으로 같다는 논리다.
반면 lukaslalinsky는 LSP가 보편화되기 전 vim과 grep으로 대형 코드베이스를 탐색하던
자신의 방식이 바로 Claude Code의 방식이라며 낯설지 않다고 반론했다.[^lukaslalinsky]

### 조직적 투자 비용

“에이전트 매니저”라는 역할의 필요성은 AI 도구가 단순 플러그인을 넘어 조직적 인프라가 되었음을 의미한다.
이는 도입 결정이 개발자 개인의 선택에서 팀 또는 조직의 거버넌스 문제로 이동하고 있다는 신호다.

belZaah는 “몇 주 혹은 몇 달 만에 모든 것이 바뀌는 업계에서 어떻게 명확한 패턴이 도출됐고,
그 패턴이 대규모 코드베이스에서 검증까지 됐다고 할 수 있느냐”고 물으며
성공 기준이 무엇인지—프로덕션 DB를 삭제하지 않은 것인지, 팀 속도가 향상된 것인지—를
구체적으로 묻지 않은 채 넘어간 점을 비판했다.[^belZaah]

## 비평

### 긍정적 측면

5계층 하네스 프레임워크는 Claude Code를 단순 도구가 아닌 확장 가능한 플랫폼으로 위치시키는
명확한 아키텍처 서술이다. 에이전틱 검색의 근거를 “스테일 인덱스 문제”로 설명하는 방식은 설득력 있다.
3~6개월 주기 검토 권장은 AI 모델의 급격한 발전 속도를 현실적으로 반영한 실용적인 조언이다.

### 비판적 시각

[GeekNews 댓글](https://news.hada.io/topic?id=29556)에서 제기된 우려처럼,
“소프트웨어 엔지니어처럼 작동한다”는 주장과 실제로 매번 파일 시스템을 탐색해야 한다는 현실은 모순적이다.
숙련된 개발자는 익숙한 코드베이스를 반복 탐색하지 않는다.
또한 중요한 지시를 90% 확률로 무시한다는 사용자 보고는 `CLAUDE.md` 중심 접근법의 실효성에 의문을 제기한다.

HN에서는 이 문제가 더 구체적인 형태로 제기됐다.
bicepjai는 명시적인 지시에도 불구하고 기존 패턴 대신 네 개의 별도 레지스트리를 새로 만든
실제 사례를 들며, 독립적인 기능 유지보수를 Claude Code에 맡기기 어렵다고 결론 내렸다.[^bicepjai]
eithed는 테스트 결과를 파일에 저장하도록 스킬을 정의해두었음에도 Claude Code가 이를 반복 무시하고,
디버깅 시에도 실제 오류를 조사하는 대신 코드를 추측으로 분석하다 15분을 소비한 끝에
틀린 요약을 내놓는다고 보고했다.[^eithed]

jameson은 하네스 구성의 복잡성에 대한 현실적인 어려움을 토로했다.
`CLAUDE.md`, 스킬, 플러그인을 정의해도 LSP의 심볼 리네이밍을 사용하지 않거나
특정 단서가 포함된 프롬프트에서 스킬을 호출하지 않는다며,
“Claude Code가 스스로 코드베이스를 검사해 효과적인 하네스를 생성할 수 없는 이유가
무엇이냐”고 반문했다.[^jameson]

sinsudo는 코드 품질 문제가 더 근본적이라고 지적했다.
개별적으로 합리적인 결정들이 누적되어 전체적인 문제가 만들어지는 것이 실제 소프트웨어의 품질 저하 패턴인데,
Claude Code는 고립된 버그만 수정할 수 있고 이런 누적 문제는 해결하기 어렵다는 것이다.[^sinsudo]
prymitive는 아키텍처가 불분명한 새 프로젝트에서의 경험을 전하며,
“내가 요청한 것을 만들었지만 내가 필요한 것을 만들지는 못했다”고 요약했다.[^prymitive]

전담 에이전트 매니저가 필요하다는 주장은 숨겨진 운영 비용을 과소평가한 것처럼 보인다.
소규모 팀이 이 비용을 감당하기 어렵다면, 이 가이드는 사실상 대기업 중심의 권장사항이 된다.

## 인사이트

### 컨텍스트 엔지니어링이 프롬프트 엔지니어링을 대체한다

프롬프트 엔지니어링이 “모델에게 무엇을 요청하는가”의 문제였다면,
하네스 설계는 “모델이 작동하는 환경 전체를 어떻게 구성하는가”의 문제다.
`CLAUDE.md`, 훅, 스킬, 플러그인, MCP 서버는 모두 모델이 처리하는 컨텍스트를 형성하는 메커니즘이다.
이 관점에서 AI 엔지니어링의 핵심 역량은 프롬프트 작성 능력에서 컨텍스트 설계 능력으로 이동하고 있다.
3~6개월 주기로 `CLAUDE.md`를 검토해야 한다는 권장사항은,
컨텍스트 엔지니어링이 한 번의 설정으로 끝나지 않고 지속적인 유지보수를 요구하는 생산 인프라임을 보여준다.

### AI 도구의 조직화 계층

Claude Code의 플러그인·마켓플레이스 체계는 AI 도구가 단순 개발자 유틸리티에서
조직의 표준화된 인프라 레이어로 진화하고 있음을 시사한다.
과거에 ESLint 규칙이나 사내 린터 설정이 팀 전체에 배포되던 방식처럼,
AI 에이전트의 구성도 중앙에서 관리되고 조직 전체에 일관되게 적용되는 방향으로 발전할 것이다.
이 구조에서 “에이전트 매니저” 역할은 단순 관리 오버헤드가 아니라,
AI 생산성의 편차를 줄이기 위한 핵심 거버넌스 기능으로 이해해야 한다.

### 인덱스 없는 탐색의 역설

파일 시스템을 매번 탐색하는 접근법은 신선도를 보장하지만,
그 비용은 다른 방식으로 누군가가 치른다.
`.ignore` 파일 관리, 코드베이스 맵 구축, LSP 서버 구성이라는 인간의 사전 작업이
인덱스 구축 비용을 대체하는 것이다.
자동 인덱스 대신 인간이 설계한 탐색 힌트에 의존하는 이 방식은,
“AI가 알아서 이해한다”는 기대를 현실적으로 조정하고
개발자가 코드베이스의 내비게이션 구조를 명시적으로 설계해야 함을 의미한다.
결국 AI 도구를 잘 쓰는 팀은, AI가 잘 탐색할 수 있도록 코드베이스 자체를 잘 조직한 팀이다.
