# Claude Code on the Web 제품 발표

원문: <https://claude.com/blog/claude-code-on-the-web>

## 요약

Anthropic은 2025년 10월 20일 Claude Code on the web을 리서치 프리뷰로 발표했다.
브라우저에서 직접 코딩 작업을 위임할 수 있는 새로운 방식으로, Anthropic이
관리하는 클라우드 인프라에서 실행된다. 출시 당시 Pro와 Max 사용자를 대상으로
했으며, 11월 12일 업데이트로 Team 및 프리미엄 시트를 가진 Enterprise
사용자에게도 확대되었다.

핵심 기능은 병렬 작업 실행이다. 터미널을 열지 않고도 GitHub 저장소를 연결하고
작업을 설명하면 Claude가 구현을 처리한다. 각 세션은 격리된 환경에서 실행되며,
실시간 진행 상황 추적과 작업 중 방향 조정이 가능하다. 자동 PR 생성과 변경 요약
기능도 제공한다.

보안 측면에서 모든 작업은 네트워크와 파일시스템 제한이 있는 격리된 샌드박스
환경에서 실행된다. Git 상호작용은 보안 프록시 서비스를 통해 처리되어 인가된
저장소에만 접근할 수 있다. 커스텀 네트워크 설정으로 Claude가 접근할 수 있는
도메인을 선택적으로 허용할 수 있다.

이 발표와 함께 iOS 앱에서도 Claude Code를 사용할 수 있는 모바일 프리뷰가
포함되었다. 클라우드 세션은 다른 모든 Claude Code 사용량과 동일한 rate limit을
공유한다.

## 분석

### 제품 포지셔닝: "위임(Delegation)"이라는 프레이밍

발표문은 "delegate coding tasks"라는 표현을 핵심 메시지로 사용한다.
"보조(assist)"나 "협업(collaborate)"이 아닌 "위임"이라는 단어 선택은, AI 코딩
도구의 역할을 페어 프로그래밍 파트너에서 독립적인 실행자로 재정의하려는 의도를
보여준다. 이는 GitHub Copilot이나 Cursor와 같은 기존 도구들이 사용하는
"copilot", "pair programmer"와 명확히 차별화되는 포지셔닝이다.

### 타겟 유스케이스의 전략적 선택

발표문이 제시하는 주요 유스케이스는 버그 백로그 처리, 루틴 수정, 백엔드 변경 등
상대적으로 위험도가 낮고 잘 정의된 작업들이다. 이는 현재 기술 수준에서 AI가
안정적으로 처리할 수 있는 범위를 현실적으로 반영한 것이다. 특히 "test-driven
development를 통해 변경사항을 검증할 수 있는 백엔드 변경"이라는 표현은, 자동화된
검증이 가능한 영역에서 AI가 가장 효과적이라는 점을 솔직하게 인정한다.

### 보안 아키텍처의 선제적 공개

보안 섹션에서 엔지니어링 블로그와 문서를 별도로 링크하며 샌드박싱 접근법의 "deep
dive"를 제공하는 것은, 기업 고객의 보안 우려에 선제적으로 대응하려는 전략이다.
"격리된 샌드박스", "보안 프록시", "인가된 저장소만 접근" 등의 키워드는
Enterprise 도입 시 보안 검토 과정을 의식한 표현들이다.

## 비평

### 강점: 간결하고 정직한 제품 소개

5분 분량의 짧은 글에서 핵심 가치 제안(병렬 작업, 클라우드 실행, 보안 격리)을
명확히 전달한다. 유스케이스를 "모든 코딩 작업"이 아닌 "버그 수정, 루틴 작업,
백엔드 변경"으로 한정한 것은 정직한 접근이다. 과장 없이 현재 단계("research
preview")를 명시하고 제한사항(rate limit 공유)도 언급한다.

### 한계: 구체적 사례와 성과 데이터의 부재

제품 발표문임에도 불구하고 내부 테스트 결과나 초기 사용자의 피드백이 전혀
포함되지 않았다. "병렬로 작업을 실행하여 더 빠르게 출시(ship faster)"한다고
하지만, 실제로 얼마나 빨라지는지에 대한 정량적 근거가 없다. 경쟁 제품(Devin,
GitHub Copilot Workspace 등)과의 비교도 없어, 이 제품이 기존 솔루션 대비 어떤
차별화를 제공하는지 독자가 스스로 판단해야 한다.

### 한계: 모바일 프리뷰의 모호한 소개

iOS 앱에서의 Claude Code 지원을 "early preview"로 언급하지만, 모바일에서 코딩
작업을 위임하는 것이 실제로 어떤 시나리오에서 유용한지 설명하지 않는다.
모바일에서 코드 diff를 리뷰하거나 PR을 확인하는 구체적인 UX에 대한 언급 없이,
단순히 "개발자가 이동 중에도 코딩할 수 있다"는 막연한 약속만 남긴다.

## 인사이트

### "리서치 프리뷰"라는 출시 전략의 이중 기능

Anthropic이 Claude Code on the web을 "beta" 혹은 "GA"가 아닌 "research
preview"로 출시한 것은 단순한 라벨링이 아니다. 이 용어는 두 가지 전략적 기능을
동시에 수행한다. 첫째, 사용자의 기대치를 관리하여 불완전한 기능에 대한 불만을
완화한다. 둘째, "research"라는 단어를 통해 Anthropic의 연구 중심 정체성을 제품
출시에도 투영한다.

OpenAI가 ChatGPT를 처음 출시할 때 사용한 "research preview" 전략과 동일한
패턴이다. 제품이 충분히 성숙하면 조용히 "preview" 딱지를 떼고 정식 기능으로
전환하면 된다. 실패하면 "연구 목적이었다"고 설명할 수 있는 안전망도 확보된다.

AI 산업 전반에서 이 패턴이 반복되는 이유는, 기존의 소프트웨어 출시
프레임워크(alpha → beta → GA)가 AI 제품의 불확실성을 담아내기에 부족하기
때문이다. AI 제품은 "작동하지만 언제 실패할지 예측하기 어려운" 독특한 상태에
있으며, "research preview"는 이 애매한 상태를 가장 정직하게 표현하는 라벨이다.

### 클라우드 IDE 전쟁의 새로운 차원 — 인간이 아닌 AI를 위한 실행 환경

Claude Code on the web의 클라우드 VM은 기존 클라우드 IDE(GitHub Codespaces,
Gitpod, Cloud9)와 근본적으로 다른 설계 의도를 가진다. 기존 클라우드 IDE는
"개발자가 어디서든 코딩할 수 있는 환경"을 제공하는 것이 목적이다. 반면 Claude
Code의 클라우드 VM은 "AI가 코드를 실행하고 테스트하는 환경"이다. 사용자
인터페이스는 브라우저에 있지만, VM 안에서 작업하는 것은 인간이 아니라 AI다.

이 구분은 클라우드 실행 환경 시장의 새로운 세그먼트를 정의한다. 기존에는
"개발자를 위한 원격 개발 환경"이 전부였지만, 이제 "AI 에이전트를 위한 원격 실행
환경"이라는 카테고리가 생겼다. 이 카테고리의 요구사항은 기존과 다르다 — 키보드
입력 지연(latency)은 중요하지 않지만, 샌드박싱의 견고함과 병렬 인스턴스 관리는
핵심이다.

장기적으로 이 시장은 클라우드 인프라 제공업체들에게 새로운 수익원이 될 수 있다.
AWS나 GCP가 "AI 에이전트 전용 실행 환경"을 제공하거나, Anthropic이 현재 자체
관리하는 VM을 외부 클라우드에 위임하는 시나리오가 모두 가능하다. Anthropic이
현재 직접 인프라를 관리하는 선택은, 보안 통제권을 유지하면서 사용자 경험을 직접
최적화하려는 의도지만, 규모가 커지면 인프라 운영 부담이 핵심 역량(모델 개발)에서
주의를 분산시킬 수 있다.

### Rate Limit 공유가 시사하는 AI 도구의 경제학적 딜레마

발표문 마지막에 조용히 언급된 "클라우드 세션은 다른 모든 Claude Code 사용량과
동일한 rate limit을 공유한다"는 문장은, 이 제품의 가장 근본적인 긴장을 드러낸다.
병렬 작업 실행이 핵심 가치 제안인데, rate limit이 공유되면 병렬로 실행할수록 각
작업의 가용 용량이 줄어든다. "더 많이 사용할수록 더 느려진다"는 이 구조는 제품의
핵심 약속과 정면으로 충돌한다.

이는 AI 도구 전반이 직면한 경제학적 딜레마를 축소판으로 보여준다. AI 추론 비용은
여전히 높고, 사용량 기반 과금(pay-per-use)과 구독 모델(subscription) 사이의
긴장이 존재한다. 구독 모델은 사용자 경험을 단순화하지만, 헤비 유저의 비용을 다른
사용자가 보조하는 구조를 만든다. Claude Code on the web처럼 사용량을 크게
증가시킬 수 있는 기능은 이 보조금 구조를 더 극단적으로 만든다.

Anthropic이 이 딜레마를 어떻게 해결하는지는 AI 도구 시장 전체에 선례를 만들
것이다. 별도의 클라우드 실행 크레딧을 도입할 수도, 티어별 병렬 세션 상한을 둘
수도, 혹은 추론 비용이 충분히 낮아질 때까지 rate limit을 점진적으로 완화할 수도
있다. 어떤 방향이든, "무제한 병렬 작업"이라는 이상과 "유한한 컴퓨팅 자원"이라는
현실 사이의 균형점을 찾아야 한다.
