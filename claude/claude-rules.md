# Claude Code 규칙이 실제로 작동하는 방식 - 발견, 로딩, 스코핑

> 원문: <https://joseparreogarcia.substack.com/p/how-claude-code-rules-actually-work>

## 요약

### 문제 의식의 출발

Jose Parreo Garcia의 Substack 글이다. 저자는 개인 프로젝트에 Claude Code
규칙을 설정하다가 이상한 경험을 했다고 고백한다. 몇 개의 규칙을 추가하고
깔끔하게 정리한 뒤 Claude가 프로젝트가 커질수록 더 예측 가능하게 동작
하리라고 기대했지만, 실제로는 “한순간에는 맞고 다음 순간에는 미묘하게
틀린” 익숙한 패턴이 반복됐다.

그가 내린 결론은 “문제는 Claude가 아니라 내 멘탈 모델”이라는 것이었다.
저자는 규칙을 “Claude가 기억할 지시(instructions)”로 취급하고 있었는데,
실제로 규칙은 “발견되고, 로드되고, 스코프되고, 때때로 무시되는 컨텍스트
(context)”였다. 이 구분이 글 전체의 핵심 테마다.

### 글이 다루는 여섯 가지 질문

저자는 자신이 풀고 싶었던 질문들을 나열한다. Claude Code 규칙이란 무엇
이며 언제 쓰지 말아야 하는가. Claude가 규칙을 발견·로드·우선순위화
하는 방식은 어떤가. Scoped rule을 결정론적으로 사용하는 방법. 자동으로
로드되지 않는 컨텍스트를 Claude에게 읽게 하는 방법. 실제로 메모리에
무엇이 로드됐는지 확인하는 방법. Churn prediction 데이터 사이언스
프로젝트라는 구체적 예시에서 이 모든 것이 어떻게 작동하는가. 어떤
실패 모드가 있고 어떻게 일찍 감지할 것인가.

### 글로벌 CLAUDE.md vs 프로젝트 CLAUDE.md의 역할 분리

저자는 이전 글에서 다룬 메모리 레이어를 먼저 요약한다. 글로벌 CLAUDE.md
는 “Claude와의 작업 관계를 정의하는” 매우 특정한 역할을 갖고 있고, “모든
가능한 작업 선호를 나열하는 곳이 아니다”. 특정 프로젝트의 뉘앙스 ―
churn 모델 만드는 방법, 경영진 요약 쓰는 방법, 딥 리서치 수행 방법 ―
는 프로젝트 CLAUDE.md에 속한다.

문제는 여기서 발생한다. 프로젝트별 파일이 있다는 이유로, 사람들은 모든
프로젝트 디테일을 그 한 파일에 쓰고 싶어진다. 프로젝트 컨텍스트, 아키
텍처 노트, 코딩 컨벤션, 데이터 제약, ML 워크플로, 평가 규칙, 배포 경고,
테스트... 결과는 1,000줄짜리 거대 프로젝트 설명이다. 여기서 컨텍스트
메모리 팽창, 상충하는 지시, 유지보수 어려움 같은 문제가 가지를 친다.
“긴 프로젝트 파일은 가이드가 아니라 부채가 된다.”

### Claude Code 규칙의 정의

저자는 규칙을 “프로젝트 스코프의 메모리”로 정의한다. `.claude/rules/`
아래에 작은 모듈형 마크다운 파일로 저장되며, “이 저장소에서 작업이 어떻게
이루어져야 하는가”를 구조적·합성 가능(composable)·유지보수 가능한 형태로
담는다. 예시 트리는 `python-style.md`, `experiments.md`, `data-privacy.md`,
그리고 `ml/` 서브디렉토리 아래의 `training.md`, `pipelines.md`다. 저자는
규칙을 “프로젝트 메모리의 분해(decomposition)”로 개념화한다. 한 거대한
파일 대신 주제별 작은 파일을 독립적으로 진화시키는 방식이다.

### 기본 동작: 모든 규칙이 시작 시 로드된다

이 섹션이 글의 첫 번째 “반전”이다. 저자는 독자에게 묻는다. `.claude/rules/`
에 5개의 규칙 파일이 있다면, Claude는 모든 것을 세션 시작에 로드하는가?
필요할 때 지능적으로 로드하는가? 그 중간인가?

답은 “기본적으로 Claude는 모든 프로젝트 규칙을 세션 시작에 로드한다”이다.
5개의 마크다운이 있으면 5개 모두가 시작 순간 메모리에 들어간다. 이는
중요한 함의를 갖는다. 아무 트릭 없이, 규칙은 단지 “우리 인간을 위한 조직적
보조 도구”에 불과하다. 규칙을 부주의하게 쓰면 여전히 컨텍스트 메모리가
팽창한다. 이런 “기본 로드” 규칙을 “unconditional rules”라고 부른다.

### Unconditional rules의 적절한 용도

Unconditional rule은 “저장소 안에서 어떤 작업을 하든 적용되어야 하는
제약”을 인코딩할 때 가장 효과적이다. 저자가 따르는 세 가지 예시는 다음과
같다. 첫째, 베이스라인 코딩·품질 표준(포매팅, 타입 기대, 테스트 요건,
로깅 컨벤션)이다. 이들은 안정적이고 논쟁의 여지가 적어서 여기에 속한다.
둘째, 협상 불가능한 제약(보안 요건, 데이터 프라이버시, 컴플라이언스)이다.
셋째, 공유된 프로젝트 가정(예: “이 저장소는 배치 워크로드를 대상으로 한다”,
“이 프로젝트는 반복 속도보다 오프라인 평가 정확성을 우선한다”). 이런
가정은 여러 하위 결정을 좌우하므로 항상 존재해야 미묘한 드리프트를
방지한다.

### Unconditional rules에 두지 말아야 할 것

반면 unconditional rules에 두지 말아야 할 것도 나열된다. 단계별 워크플로,
특정 디렉토리에 묶인 지시, ML 특유의 모델링 선택, 실험적이거나 진화하는
관행. “규칙이 코드베이스의 특정 부분을 만질 때만 적용된다면, 그것은
unconditional이면 안 된다. 편의상 unconditional로 두는 것이 바로 컨텍스트
메모리가 흐려지는 방식이다.”

### Scoped rules의 작동 방식

이 섹션이 글의 두 번째 핵심이다. 진짜 프로젝트는 실험·파이프라인·인프라·
문서가 공존하는 공간이고, 모든 규칙이 어디든 적용되면 다시 컨텍스트
팽창으로 돌아간다. Scoped rule은 YAML frontmatter를 통해 작동한다. 파일
맨 위에 `paths` 필드를 두고 glob 패턴으로 매칭 파일을 지정한다.

저자는 churn 모델 저장소 트리를 예시로 든다. `notebooks/`, `src/features/`,
`src/models/churn/`, `src/pipelines/`, `tests/` 같은 디렉토리가 있고, 각
영역마다 다른 규칙이 필요하다. 노트북 작업은 가볍고 빠르며 “프로토타입
상태임을 명시”해야 한다. 프로덕션 모델 코드는 테스트·재현성·데이터 누수
방지를 엄격히 요구해야 한다.

예시로 제공된 `notebooks.md` 규칙은 `paths: - “notebooks/**”` frontmatter를
갖고, 노트북을 탐색적으로 다루고 프로토타입 상태 표시, 작은 실행 가능
셀 선호, 안정된 코드는 `src/`로 마이그레이션, 비밀/데이터셋 경로 하드
코딩 금지 같은 지침을 담는다. `production-ml.md`는 `src/models/churn/**`과
`src/pipelines/**`를 paths로 갖고, 재현성 보존(고정 random seed, 결정론적
split, 버전 관리된 feature), 미래 정보 누수 방지, 커밋 전 `pytest -q`
실행, 표준 메트릭 사용 같은 더 엄격한 요건을 담는다.

### CLAUDE.md를 인덱스로 사용하기

Scoped rules는 강력하지만 한계가 있다. 그것들은 Claude가 지정된 경로를
읽거나 작업할 때만 트리거된다. 저자는 CLAUDE.md를 “규칙이 무엇에 관한
것인지”를 알려주는 인덱스로 사용하라고 권한다. 프로젝트 CLAUDE.md는
“이 저장소에서 작업하는 법”이라는 섹션을 두고, “탐색적·실험적 작업이라면
`.claude/rules/experiments.md`를 따르고 `notebooks/`에서 작업하라”,
“프로덕션 ML 변경이라면 `src/models/` 아래 scoped rule이 활성화되며
재현성·평가·테스트가 필수”라는 식으로 안내한다.

### 결정론의 두 종류

이 구분이 글의 가장 정제된 통찰이다. “Scoped rules는 적용에서 결정론적
이다(deterministic in application). CLAUDE.md는 기대에서 결정론적이다
(deterministic in expectation).” 또 다른 표현: “Scoped rules는 지역적
실수를 방지한다. CLAUDE.md는 전역적 오해를 방지한다.”

저자는 두 가지 실패 모드를 대비시킨다. Scoped rules만 의존하면 Claude가
기술적으로 제약을 준수하면서도 잘못된 접근으로 과제를 다룰 수 있다.
예컨대 디자인 논의를 원했는데 프로덕션 코드로 뛰어든다거나, 실험을
원했는데 최적화에 집중한다. 반대로 CLAUDE.md만 의존하면 Claude가 의도는
이해하지만 편집을 시작할 때 중요한 가드레일을 놓친다. 두 가지를 함께
사용할 때만 루프가 닫힌다.

### 자동으로 로드되지 않는 컨텍스트

마지막으로 저자는 실제 프로젝트가 CLAUDE.md나 `.claude/rules/`에 들어있지
않은 중요한 컨텍스트를 갖고 있음을 지적한다. README.md, 아키텍처 개요,
디자인 문서, RFC, PRD, 실험 요약, 런북, 운영 노트 같은 문서들이다. Claude
는 기본적으로 이런 파일을 로드하지 않으며, 명시적으로 가리키지 않는 한
아예 읽지 않을 수 있다. 여기서부터 유료 섹션으로 넘어간다(이 TIL에서
다루는 범위는 미리보기까지).

## 분석

### 저자의 멘탈 모델 전환이 글의 축이다

이 글의 구조를 지탱하는 것은 “규칙 = 지시”에서 “규칙 = 컨텍스트”로의
멘탈 모델 전환이다. 저자는 이 전환을 다섯 개의 질문 형태로 풀어낸다.
각 질문은 “내가 당연하다고 가정했던 것이 사실이 아니었다”는 구조를
갖는다. 이는 기술 블로그의 효과적인 수사 장치이며, 독자가 자신의 오해를
동일시하게 만든다.

특히 “규칙은 Claude가 기억하는 것이 아니라 컨텍스트로 로드되는 것”이라는
표현은 LLM 도구 사용 경험이 있는 많은 개발자가 암묵적으로 헷갈리는 지점
이다. LLM은 “기억”이라는 단어와 자주 연관되지만, 실제로는 세션마다 컨텍스트
가 재구성된다. 저자는 이 차이를 명시하고, 그 차이가 실무에 미치는 영향을
구체화한다.

### 결정론의 이원론 ― 글의 가장 강력한 개념

“Scoped rules는 적용에서 결정론적, CLAUDE.md는 기대에서 결정론적”이라는
구분은 이 글이 만든 가장 유용한 개념이다. 이 이분법은 두 메커니즘의
성격을 한 줄로 포착한다.

Scoped rule이 “적용에서 결정론적”이라는 것은, Claude가 매칭 파일을 읽을
때마다 규칙이 반드시 활성화된다는 의미다. 이는 물리적 메커니즘의 결정론
이다. 규칙을 읽지 않는 경우는 없다. CLAUDE.md가 “기대에서 결정론적”이라는
것은, 모든 세션에 반드시 로드되지만 그 안의 지시가 정확히 지켜지리라는
보장은 없다는 의미다. 이는 LLM 확률 분포의 결정론이다. Claude가 매번
“이 프로젝트는 이런 것을 기대한다”는 맥락 안에서 응답한다는 의미에서
결정론적이지만, 응답 자체는 확률적이다.

이 이원론은 두 메커니즘을 어떻게 조합해야 하는지에 대한 명확한 설계
원칙을 제공한다. “로컬 실수 방지는 scoped rule, 전역 오해 방지는
CLAUDE.md”. 이는 “하드 룰 vs 소프트 룰”이라는 전통적 구분의 LLM 시대
버전이다.

### “Churn model 저장소” 예시의 구체성

저자가 churn prediction 프로젝트라는 구체적 예시를 일관되게 사용한 것은
효과적이다. 추상적 원칙 ― “scoped rule을 쓰라” ― 가 구체적 디렉토리 트리
(`notebooks/`, `src/features/`, `src/models/churn/`, `src/pipelines/`,
`tests/`)로 즉시 시각화된다. 독자는 자기 프로젝트에 이 구조를 매핑해볼 수
있다.

특히 데이터 사이언스 프로젝트를 예시로 든 선택은 의도적이다. 데이터 사이
언스 저장소는 “탐색적 코드와 프로덕션 코드의 공존”이라는 문제가 가장 극명
하게 나타나는 영역이다. 같은 저장소에서 노트북의 느슨한 규칙과 프로덕션
파이프라인의 엄격한 규칙을 동시에 적용해야 한다. Scoped rule의 필요성을
가장 잘 드러내는 예시다.

### 글로벌 vs 프로젝트 CLAUDE.md 역할 분리의 논거

저자가 이전 글에서 강조했다는 “글로벌 CLAUDE.md는 작업 관계만, 프로젝트별
뉘앙스는 프로젝트 CLAUDE.md”라는 구분은 본질적이다. 많은 사용자가 이
구분을 지키지 않고 글로벌 파일에 프로젝트 관련 내용을 넣거나, 반대로
작업 관계 기본 원칙을 프로젝트 파일에 넣는다. 둘 다 실패한다. 글로벌에
프로젝트 내용이 들어가면 관련 없는 프로젝트에서 혼란을 만들고, 프로젝트에
작업 관계 원칙이 들어가면 모든 프로젝트에 반복된다.

저자의 권고는 “역할별 분리” 원칙의 표현이다. 이는 소프트웨어 설계의
Single Responsibility Principle을 메모리 레이어에 적용한 것이다. 각
메모리 레이어는 한 가지 책임만 가져야 한다. 글로벌 → 작업 관계, 프로젝트
→ 프로젝트 맥락, Scoped rule → 파일 유형별 행동, Skill → 일회성 워크플로.

## 비평

### 강점

#### 개인 실패 경험에서 출발하는 솔직함

“나는 이렇게 가정했는데 틀렸다”는 솔직한 고백으로 시작한 것은 독자를
즉시 끌어들인다. 기술 문서가 “공식 정답”을 제공하는 반면, 이 글은 “나와
같은 혼란을 겪었을 당신”에게 말한다. 이는 실제로 Claude Code를 써본
사용자 다수가 공유하는 경험이며, 글의 공명이 크다.

#### “적용 결정론 vs 기대 결정론”이라는 명제의 정제

이 한 문장은 글의 가장 큰 기여다. 복잡한 두 메커니즘의 차이를 한 줄로
포착하며, 독자가 앞으로 계속 쓸 수 있는 개념 도구를 제공한다. 좋은 기술
블로그의 특징은 “독자가 며칠 후에도 기억할 수 있는 한 문장”을 남기는
것인데, 이 글은 이를 성공시킨다.

#### 구체적 디렉토리 트리의 제공

추상적 이론이 아니라 실제 디렉토리 구조와 파일 내용을 보여준다. 독자는
`notebooks.md`와 `production-ml.md`의 YAML frontmatter와 마크다운 본문을
그대로 복사해 자기 프로젝트에 적용할 수 있다. 글을 읽고 바로 행동할 수
있게 하는 실용성이 있다.

#### Unconditional rule의 용도와 비용도의 균형 있는 설명

“이런 것은 unconditional에 두고, 이런 것은 두지 말라”는 명확한 가이드는
가장 유용한 부분이다. 베이스라인 표준, 비협상 제약, 공유 가정은 OK,
단계별 워크플로·디렉토리별 지시·ML 모델링 선택·실험적 관행은 NO. 이
대비는 독자가 즉시 자기 규칙을 재분류하게 만든다.

### 약점

#### 유료화(paywall)로 끊어지는 구조

글의 앞부분은 미리보기로 읽을 수 있지만, 실제 “데이터 사이언스 프로젝트
전체 예시”, “검증 방법”, “실패 모드 분석”, “종결 고찰” 같은 약속된 섹션은
유료 구독자에게만 열린다. 이는 글이 제공하는 가치의 상당 부분을 가린다.
미리보기 부분만 보면 “인덱스로서의 CLAUDE.md와 scoped rule의 결합”이라는
핵심 통찰은 얻을 수 있지만, 실전 적용의 디테일은 볼 수 없다.

#### 공식 문서와의 관계 미언급

이 글은 2026년 4월 현재 공식 Claude Code 문서(`code.claude.com/docs/en/
memory`)와 거의 같은 주제를 다루지만, 공식 문서와 어떻게 다른지 또는 왜
보완이 필요한지는 설명하지 않는다. 독자는 공식 문서를 먼저 읽어야 할지,
이 글을 먼저 읽어야 할지 판단하기 어렵다.

#### 검증 메커니즘의 부재

저자는 “메모리에 무엇이 로드됐는지 확인하는 방법”을 약속했지만, 이 부분은
유료 섹션이거나 짧게 다뤄진다. 공식 문서의 `/memory` 명령어와 `Instructions
Loaded` 훅에 대한 언급이 부족하다면, 독자는 실제로 자기 규칙이 로드되고
있는지 어떻게 확인해야 할지 모른다. 이 검증 갭이 저자가 첫머리에서 말한
“확인하지 않고 가정했다”는 실수를 독자에게 그대로 반복시킬 위험이 있다.

#### 규칙 테스트 방법 미언급

Scoped rule이 정말 매칭되는 파일을 열 때만 로드되는지, glob 패턴이 예상
대로 매칭되는지를 테스트할 방법은 제시되지 않는다. “이 규칙이 제대로
작동하는지 어떻게 아는가”라는 질문에 답이 없으면, 규칙 설계는 여전히
기도에 가까워진다.

#### 실제 프로덕션 사용 시 발생하는 실패 사례 부족

저자는 자신의 실패 경험으로 시작했지만, 구체적으로 어떤 규칙이 어떤
상황에서 무시되었는지는 설명하지 않는다. 독자가 “이런 패턴이 실패한다”는
경험적 지식을 얻으려면 실제 사례가 필요한데, 글은 주로 이론적 설명에
머문다.

#### 다른 에이전트 프레임워크와의 비교 부재

Cursor, Windsurf, Aider 같은 다른 AI 코딩 도구들도 유사한 규칙 시스템을
갖고 있다. 이들과 Claude Code의 규칙 시스템이 어떻게 다른지, 이 글에서
배운 교훈이 다른 도구에도 적용되는지는 다뤄지지 않는다. 도구를 넘나드는
개발자에게는 이 비교가 가장 중요한 질문일 수 있다.

## 인사이트

### 1. “규칙은 지시가 아니라 컨텍스트”는 AI 도구 사용의 근본 재프레이밍이다

저자가 글 초반에 명시한 “규칙을 Claude가 기억할 지시로 취급하고 있었다”는
자기 고백은 단순한 용어 혼란이 아니라 AI 도구에 대한 인지 모델 전체의
문제를 드러낸다. 소프트웨어 엔지니어는 수십 년간 “설정 파일”과 “지시”를
같은 것으로 취급해왔다. `.eslintrc.json`에 규칙을 쓰면 린터가 반드시
따르고, `.prettierrc`에 스타일을 쓰면 포매터가 반드시 적용한다. 이 “쓰면
지켜진다”는 결정론적 모델이 너무 깊이 뿌리내려서, LLM 도구의 근본적
다름을 보지 못하게 한다.

AI 도구의 규칙은 “지시”가 아니라 “컨텍스트 조각”이다. 이 차이는 물리적
이다. 전통적 도구는 규칙을 기계가 실행하는 코드로 변환하지만, LLM 도구는
규칙을 프롬프트로 주입한다. 프롬프트는 확률적 입력이며, LLM은 그것을
“고려”할 뿐 “실행”하지 않는다. 이 차이가 인식되지 않으면, 규칙이 “때로
작동하고 때로 작동하지 않는” 현상이 마법처럼 보인다.

이 재프레이밍이 중요한 이유는, 그것이 규칙을 설계하는 방식을 근본적으로
바꾸기 때문이다. 전통적 설정은 “최소한으로, 명확하게” 쓰면 끝이다. AI
도구의 규칙은 “명확성”뿐만 아니라 “주의력 획득”이라는 새로운 차원을
고려해야 한다. 긴 컨텍스트 안에서 특정 지시가 Claude의 주의를 얻지 못
하면, 그것은 사실상 없는 것이다. 따라서 규칙 설계는 프롬프트 엔지니어링의
모든 원칙 ― 강조, 반복, 순서, 대비 ― 을 요구한다.

이 통찰은 AI 도구 문서화의 새로운 장르를 만든다. “어떻게 쓰면 LLM이 잘
따르는가”에 대한 실용적 가이드가 공식 문서의 일부가 되어야 한다. Claude
Code 공식 문서가 “200줄 이하”, “구체적 지시”, “충돌 회피”를 명시한 것은
이 방향이지만, 충분하지 않다. 앞으로 3-5년 내 “CLAUDE.md 린터”나 “규칙
평가 도구” 같은 새 카테고리가 등장할 것이며, 이들은 “이 규칙이 LLM의
주의를 얻을 가능성”을 정량화할 것이다.

### 2. “적용 결정론 vs 기대 결정론”은 LLM 기반 시스템 설계의 보편 원칙

저자가 만든 “scoped rules는 적용에서 결정론적, CLAUDE.md는 기대에서
결정론적”이라는 이분법은 Claude Code를 넘어서는 보편적 원칙이다. 이 원칙은
LLM이 결합된 모든 시스템에서 “무엇이 확실히 일어나고 무엇이 확률적으로
일어나는가”를 구분하는 프레임이다.

“적용 결정론”은 시스템 수준의 메커니즘 결정론이다. 파일이 매칭되면 규칙이
반드시 로드된다, MCP 서버가 요청을 받으면 반드시 응답한다, 훅이 조건에
맞으면 반드시 실행된다. 이 레이어는 LLM이 아니라 전통적 소프트웨어가
담당한다. 따라서 결정론적이다.

“기대 결정론”은 LLM의 주의 분포 결정론이다. 프롬프트에 어떤 내용이 있으면
LLM이 “그 방향으로 기대된다”는 의미에서 결정론적이지만, 실제 응답은 확률적
이다. 이 레이어는 LLM이 담당한다.

이 이분법은 LLM 기반 시스템 설계에 중요한 함의를 갖는다. 시스템의 핵심
안전성·정확성 요건은 “적용 결정론” 레이어에 넣어야 하고, 행동 가이드·맥락
설정은 “기대 결정론” 레이어에 넣어야 한다. 이 구분을 지키지 않으면 두
가지 실패가 발생한다. 첫째, 안전성 요건을 LLM 프롬프트에 넣으면 확률적
으로 무시될 수 있다. 둘째, 행동 가이드를 강제 메커니즘으로 구현하면
과잉 엄격성으로 사용성이 망가진다.

이 원칙은 AI 에이전트 설계에도 적용된다. 에이전트가 금전 거래를 실행할
때는 “적용 결정론”으로 한도를 강제해야 한다 ― 즉 전통적 코드로 검증 로직을
구현해야 한다. 에이전트가 문서 스타일을 결정할 때는 “기대 결정론”으로 충분
하다 ― 즉 프롬프트에 지침을 쓰면 된다. 이 구분을 정확히 하는 에이전트
프레임워크가 앞으로 성공할 것이다.

### 3. “.claude/rules/”는 리포지토리를 “AI 독자 기준”으로 재설계하는 시작이다

저자가 churn 모델 저장소 예시에서 보여준 `notebooks.md`와 `production-ml.md`
의 분리는 단순한 파일 구성이 아니다. 그것은 “리포지토리 구조가 AI 독자의
행동을 결정한다”는 새로운 설계 원칙의 시작이다. 전통적으로 디렉토리 구조는
“사람이 찾기 쉬운 방식”으로 조직되었다. 이제 그것은 “AI가 올바른 규칙을
트리거할 수 있는 방식”으로도 조직되어야 한다.

이 변화는 단순하지 않다. 기존 리포지토리들은 많은 경우 “사람 기준”과
“AI 기준”이 일치하지 않는다. 예를 들어 레거시 프로젝트에서 `src/`에는
프로덕션 코드와 실험 코드가 뒤섞여 있을 수 있다. 이 구조에서는 “프로덕션
ML 규칙”과 “실험적 관행 규칙”을 path로 분리할 수 없다. 규칙을 적용하려면
먼저 디렉토리 자체를 재구성해야 한다.

이는 “AI 도입”이 단순한 도구 사용이 아니라 저장소 아키텍처 변경을 요구
한다는 함의를 낳는다. AI 친화적 저장소는 (1) 디렉토리 구분이 명확하고,
(2) 각 디렉토리의 “용도 구분”이 일관되며, (3) `.claude/rules/`와 같은
메타 파일이 그 구분을 인코딩한다. 이 세 조건을 충족하는 저장소는 AI
협력이 효과적이고, 그렇지 않은 저장소는 AI가 잘못된 규칙을 적용하거나
아무 규칙도 적용하지 못한다.

이 통찰은 코드 리뷰 실무에도 영향을 미친다. 앞으로 리뷰어는 “이 변경이
디렉토리 구조를 AI 친화적으로 유지하는가”를 평가해야 한다. 새 파일을
추가할 때 “이것이 `.claude/rules/`의 어떤 scoped rule 하에 들어가는가”,
“그 규칙이 적절한가”를 물어야 한다. 이는 일종의 “AI 거버넌스”이며, 코드
품질 검토의 새로운 차원이다. 장기적으로 “AI 친화도”는 코드 품질의 한
지표가 될 것이고, 이것이 낮은 저장소는 AI 도입의 이점을 누리지 못할
것이다.

### 4. 지식의 “검색 가능성” vs “자동 로딩”의 긴장 ― Skills의 재발견

저자는 글 말미에서 “자동으로 로드되지 않는 컨텍스트”를 언급하며 README,
아키텍처 문서, RFC, 운영 노트 같은 파일들을 예로 든다. 이는 Claude Code
메모리 시스템의 중요한 한계를 드러낸다. 모든 지식을 CLAUDE.md나 rules에
넣을 수는 없다. 그렇게 하면 컨텍스트가 팽창하고, Claude의 주의력이 분산
된다. 반면 넣지 않으면 Claude가 그 지식을 영영 발견하지 못할 수 있다.

이 긴장은 소프트웨어 문서화의 오래된 문제의 LLM 버전이다. 전통적으로
“찾아야 할 문서는 어디에 두는가”는 개발자의 경험과 직관에 의존했다.
README는 항상 저장소 루트, 아키텍처 다이어그램은 `docs/`, 운영 노트는
`runbooks/`. 이 컨벤션이 있어도 신입 개발자는 몇 주간 “무엇이 어디에
있는지” 배워야 했다.

LLM 도구는 이 문제를 악화시키기도 하고 해결하기도 한다. 악화시키는 면은,
LLM이 자동으로 모든 문서를 로드할 수 없기 때문에 “찾을 수 있어야 한다”는
요건이 더 엄격해진다는 것이다. 해결하는 면은, LLM이 적절한 인덱스가 주어
지면 인간보다 훨씬 빨리 올바른 문서를 찾아낼 수 있다는 것이다.

저자가 말하는 “CLAUDE.md를 인덱스로 사용”은 이 긴장의 해법이다. 모든
지식을 직접 주입하는 대신, “어떤 주제에 대해 질문이 생기면 이 파일을
읽어라”는 포인터를 CLAUDE.md에 두는 것이다. 이는 LLM 도구 사용의 메타
원칙이다. “자동 로드 가능한 크기”에는 맥락과 포인터만 넣고, 상세 지식은
외부 파일로 분리하며, LLM이 필요할 때 발견할 수 있는 경로를 만든다.

이 원칙은 Claude Code의 Skills 메커니즘과 정확히 일치한다. Skills는
“필요할 때만 로드”되는 지식 모듈이며, CLAUDE.md는 “어떤 Skill이 언제
필요한가”를 알려주는 인덱스 역할을 할 수 있다. 저자는 Skills를 명시적
으로 다루지 않지만, 그의 원칙은 Skills의 존재 이유를 설명한다. 결국
“무엇을 자동 로드, 무엇을 검색 가능, 무엇을 온디맨드로 둘 것인가”는
LLM 시대 지식 관리의 핵심 설계 결정이다.

### 5. “AI 시대의 개발 문서화”는 새로운 직무 영역이 될 것이다

저자가 글 전체에서 암묵적으로 주장하는 것은 “Claude Code를 잘 쓰는 개발자
는 단순히 프롬프트를 잘 쓰는 사람이 아니라, 저장소의 메모리 구조를 잘
설계하는 사람”이라는 것이다. CLAUDE.md, `.claude/rules/`, CLAUDE.local.md,
auto memory, Skills ― 이 모든 레이어의 역할을 이해하고 각각에 무엇을
넣을지 결정하는 것은 단순한 도구 사용이 아니라 정보 아키텍처 설계다.

이는 “AI 도구를 쓰는 법”을 가르치는 산업 전체의 방향을 시사한다. 초기
Claude Code 교육은 “프롬프트 잘 쓰기”에 집중했고, 다음 단계는 “에이전트
워크플로 설계”였다. 다음 단계는 “저장소 메모리 아키텍처 설계”가 될 것
이다. 이는 기존의 Information Architecture(IA) 직무와 유사하다. 1990년대
웹사이트가 복잡해지면서 “어떤 콘텐츠를 어떤 페이지에 두는가”를 전문적
으로 설계하는 직무가 생겼듯, 2020년대 저장소의 메모리 구조를 전문적으로
설계하는 직무가 생길 것이다.

이 직무는 “CLAUDE.md 엔지니어” 또는 “AI 컨텍스트 아키텍트” 같은 이름을
갖게 될 것이다. 그들의 일은 다음과 같다. 저장소를 감사하고 중복·충돌·
누락된 규칙을 식별한다. 디렉토리 구조를 scoped rule이 효과적으로 작동
하도록 재조직한다. 글로벌/프로젝트/로컬 레이어 간 역할 구분을 유지한다.
Skills와 rules의 경계를 결정한다. 규칙의 효과를 측정하고 A/B 테스트한다.
외부 문서와 인덱스의 관계를 설계한다.

이 직무가 등장하는 시점은 조직의 AI 성숙도에 따라 다르겠지만, 대규모
엔터프라이즈에서는 이미 2026년 현재 비공식적으로 존재한다. 시니어
엔지니어 한 명이 팀의 “AI 워크플로 유지관리”를 맡는 경우가 많고, 이
역할이 공식화되기까지 1-2년 이내일 것이다. 이 직무의 존재 자체가 AI
도구의 성숙도를 보여준다. 도구가 진짜 인프라가 되면, 그 인프라를 관리
하는 전담 인력이 필요해진다. Claude Code는 이제 그 단계에 들어서고 있다.
