## 한국어 번역

스킬은 Claude Code에서 가장 많이 사용되는 확장 포인트 중 하나가 되었다.
유연하고, 만들기 쉽고, 배포도 간단하다.

하지만 이 유연성 때문에 무엇이 가장 효과적인지 알기 어렵다.
어떤 유형의 스킬을 만들어야 할까? 좋은 스킬을 쓰는 비결은?
다른 사람과 언제 공유해야 할까?

Anthropic에서는 수백 개의 스킬을 활발히 사용하고 있다.
이것은 개발을 가속화하기 위해 스킬을 사용하면서 배운 교훈이다.

### 스킬이란?

스킬에 대한 흔한 오해는 “그냥 마크다운 파일”이라는 것이다.
가장 흥미로운 점은 스킬이 단순한 텍스트 파일이 아니라는 것이다.
스킬은 스크립트, 에셋, 데이터 등을 포함할 수 있는 **폴더**이며,
에이전트가 이를 발견하고 탐색하고 조작할 수 있다.

Claude Code에서 스킬은 동적 훅 등록을 포함한 다양한 설정 옵션도 갖고 있다.
가장 흥미로운 스킬들은 이런 설정 옵션과 폴더 구조를 창의적으로 활용한다.

## 스킬의 9가지 유형

### 1. 라이브러리 & API 레퍼런스

라이브러리, CLI, SDK를 올바르게 사용하는 방법을 설명하는 스킬.
내부 라이브러리든 Claude Code가 어려워하는 일반 라이브러리든 모두 해당한다.
레퍼런스 코드 스니펫 폴더와 주의사항(gotchas) 목록을 포함하는 경우가 많다.

- billing-lib — 내부 빌링 라이브러리의 엣지 케이스, 함정 등
- internal-platform-cli — 내부 CLI 래퍼의 모든 하위 명령과 사용 예시
- frontend-design — Claude의 디자인 시스템 이해도 향상

### 2. 제품 검증 (Product Verification)

코드가 제대로 작동하는지 테스트하거나 검증하는 스킬.
Playwright, tmux 같은 외부 도구와 함께 사용하는 경우가 많다.

검증 스킬은 Claude의 출력이 정확한지 확인하는 데 매우 유용하다.
**엔지니어 한 명이 일주일을 투자해서 검증 스킬을 훌륭하게 만들 가치가 있다.**

Claude가 출력 영상을 녹화하거나, 각 단계에서 프로그래밍 방식으로
상태를 검증하는 기법을 고려해 보라.

- signup-flow-driver — 헤드리스 브라우저에서 가입→이메일 인증→온보딩 실행
- checkout-verifier — Stripe 테스트 카드로 결제 UI 구동 후 인보이스 상태 확인
- tmux-cli-driver — TTY가 필요한 대화형 CLI 테스트

### 3. 데이터 페칭 & 분석

데이터·모니터링 스택에 연결하는 스킬.
자격증명을 포함한 데이터 페칭 라이브러리, 대시보드 ID,
일반적인 워크플로 지침 등이 포함될 수 있다.

- funnel-query — 가입→활성화→결제 퍼널의 이벤트 조인 방법과 canonical user_id 테이블
- cohort-compare — 두 코호트의 리텐션/전환율 비교, 통계적 유의차 플래그
- grafana — 데이터소스 UID, 클러스터 이름, 문제→대시보드 매핑 테이블

### 4. 비즈니스 프로세스 & 팀 자동화

반복적인 워크플로를 하나의 명령으로 자동화하는 스킬.
지침 자체는 비교적 단순하지만 다른 스킬이나 MCP에 대한 복잡한 의존성이 있을 수 있다.
이전 결과를 로그 파일에 저장하면 모델의 일관성과 반성(reflection)에 도움이 된다.

- standup-post — 티켓 트래커, GitHub 활동, 이전 Slack 집계 → 포맷된 스탠드업
- create-\<ticket-system\>-ticket — 스키마 강제(유효한 enum, 필수 필드) + 사후 워크플로
- weekly-recap — 머지된 PR + 닫힌 티켓 + 배포 → 포맷된 주간 요약

### 5. 코드 스캐폴딩 & 템플릿

코드베이스의 특정 기능에 대한 프레임워크 보일러플레이트를 생성하는 스킬.
순수 코드로 커버할 수 없는 자연어 요구사항이 있을 때 특히 유용하다.

- new-\<framework\>-workflow — 어노테이션이 포함된 새 서비스/워크플로/핸들러 스캐폴딩
- new-migration — 마이그레이션 파일 템플릿과 일반적인 주의사항
- create-app — 인증, 로깅, 배포 설정이 미리 연결된 새 내부 앱

### 6. 코드 품질 & 리뷰

조직 내 코드 품질을 강제하고 리뷰를 돕는 스킬.
결정론적 스크립트나 도구를 포함할 수 있으며,
훅이나 GitHub Action의 일부로 자동 실행할 수 있다.

- adversarial-review — 새로운 시각의 서브에이전트가 비평 → 수정 → 지적이 사소해질 때까지 반복
- code-style — Claude가 기본적으로 잘 못하는 코드 스타일 강제
- testing-practices — 테스트 작성 방법과 테스트 대상에 대한 지침

### 7. CI/CD & 배포

코드를 가져오고, 푸시하고, 배포하는 것을 돕는 스킬.
다른 스킬을 참조하여 데이터를 수집할 수 있다.

- babysit-pr — PR 모니터링 → 불안정한 CI 재시도 → 머지 충돌 해결 → 자동 머지 활성화
- deploy-\<service\> — 빌드→스모크 테스트→점진적 트래픽 롤아웃→에러율 비교→회귀 시 자동 롤백
- cherry-pick-prod — 격리된 worktree→체리픽→충돌 해결→템플릿이 포함된 PR

### 8. 런북 (Runbooks)

증상(Slack 스레드, 알림, 에러 시그니처)을 받아서
다중 도구 조사를 수행하고 구조화된 보고서를 생성하는 스킬.

- \<service\>-debugging — 증상→도구→쿼리 패턴 매핑
- oncall-runner — 알림 가져오기→일반적인 원인 확인→발견사항 포맷
- log-correlator — 요청 ID로 관련된 모든 시스템의 로그 수집

### 9. 인프라 운영

루틴 유지보수와 운영 절차를 수행하는 스킬.
파괴적 작업에 가드레일을 제공하여 엔지니어가 모범 사례를 따르기 쉽게 한다.

- \<resource\>-orphans — 고아 Pod/볼륨 탐지→Slack 알림→대기 기간→확인→정리
- dependency-management — 조직의 의존성 승인 워크플로
- cost-investigation — 스토리지/이그레스 비용 급증 원인 조사

## 스킬 제작 팁

### 당연한 것을 말하지 마라

Claude Code는 코드베이스에 대해 많이 알고, Claude는 코딩에 대한 기본 의견이 많다.
지식 중심 스킬을 만들 때는 **Claude의 일반적 사고방식에서 벗어나게 하는 정보**에
집중하라.

frontend-design 스킬이 좋은 예시다 — Anthropic 엔지니어가 고객과 반복하며
Claude의 디자인 취향을 개선했고, Inter 폰트나 보라색 그래디언트 같은
전형적 패턴을 피하도록 만들었다.

### Gotchas 섹션을 만들어라

모든 스킬에서 가장 가치 높은 콘텐츠는 Gotchas 섹션이다.
Claude가 스킬 사용 시 빠지는 일반적인 실패 지점에서 구축해야 한다.
시간이 지나면서 이 주의사항을 업데이트하는 것이 이상적이다.

### 파일 시스템 & 점진적 공개를 활용하라

스킬은 마크다운 파일이 아니라 폴더다. 파일 시스템 전체를 컨텍스트 엔지니어링과
점진적 공개(Progressive Disclosure)의 한 형태로 생각해야 한다.
Claude에게 스킬 안에 어떤 파일이 있는지 알려주면, 적절한 시점에 읽는다.

- 상세 함수 시그니처와 사용 예시를 `references/api.md`에 분리
- 최종 출력이 마크다운 파일이면 `assets/`에 템플릿 파일 포함
- 레퍼런스, 스크립트, 예시 등의 폴더 활용

### Claude를 레일에 가두지 마라

Claude는 지시사항을 따르려 하고, 스킬은 재사용성이 높으므로
지시사항이 너무 구체적이면 안 된다.
Claude에게 필요한 정보는 주되, **상황에 맞게 적응할 유연성**을 줘라.

### 설정(Setup)을 고려하라

일부 스킬은 사용자 컨텍스트로 설정이 필요하다.
예를 들어, 스탠드업을 Slack에 게시하는 스킬이라면
어떤 채널에 게시할지 Claude가 물어봐야 한다.

좋은 패턴: 스킬 디렉토리의 `config.json`에 설정 정보를 저장하고,
설정이 없으면 에이전트가 사용자에게 정보를 요청하도록 한다.

### Description 필드는 모델을 위한 것이다

Claude Code가 세션을 시작하면 모든 사용 가능한 스킬의 description을 스캔하여
“이 요청에 맞는 스킬이 있는가?”를 판단한다.
description은 요약이 아니라 **트리거 조건에 대한 설명**이다.

### 메모리 & 데이터 저장

스킬은 데이터를 저장하여 일종의 메모리를 포함할 수 있다.
단순한 텍스트 로그 파일이나 JSON부터 SQLite 데이터베이스까지 가능하다.

예: standup-post 스킬이 `standups.log`에 게시 이력을 보관하면,
다음 실행 시 Claude가 자신의 이력을 읽고 어제와 달라진 점을 파악한다.

스킬 디렉토리의 데이터는 업그레이드 시 삭제될 수 있으므로,
안정적인 폴더인 `${CLAUDE_PLUGIN_DATA}`에 저장해야 한다.

### 스크립트 저장 & 코드 생성

Claude에게 줄 수 있는 가장 강력한 도구는 코드다.
스크립트와 라이브러리를 제공하면 Claude가 보일러플레이트를 재구성하는 대신
**조합(composition)**과 다음 할 일 결정에 턴을 쓸 수 있다.

예: 데이터 과학 스킬에 이벤트 소스에서 데이터를 가져오는 헬퍼 함수 라이브러리를
포함하면, Claude가 이를 조합하여 “화요일에 무슨 일이 있었나?” 같은 프롬프트에
대한 고급 분석 스크립트를 즉석에서 생성할 수 있다.

### 온디맨드 훅

스킬은 호출될 때만 활성화되고 세션 동안 지속되는 훅을 포함할 수 있다.
항상 실행하고 싶지는 않지만 때때로 매우 유용한 독단적인 훅에 사용한다.

- `/careful` — PreToolUse 매처로 `rm -rf`, `DROP TABLE`, force-push, `kubectl delete` 차단.
  프로덕션을 다룰 때만 원하는 것이지, 항상 켜두면 미칠 노릇
- `/freeze` — 특정 디렉토리 외의 Edit/Write 차단. 디버깅할 때 “로그만 추가하고 싶은데
  관련 없는 것을 자꾸 '고치는'” 문제 방지에 유용

## 스킬 배포

### 두 가지 배포 방식

1. **레포에 체크인** (`./.claude/skills` 하위) — 소규모 팀에 적합
2. **내부 플러그인 마켓플레이스** — 규모가 커지면 팀이 설치할 스킬을 선택 가능

레포에 체크인된 스킬은 모델의 컨텍스트에 조금씩 추가된다.
규모가 커지면 마켓플레이스가 더 나은 선택이다.

### 마켓플레이스 운영

중앙 팀이 결정하지 않고, 가장 유용한 스킬을 유기적으로 찾는 방식이다.
스킬을 시도해 보게 하려면 GitHub의 샌드박스 폴더에 업로드하고
Slack 등에서 홍보한다. 충분한 견인력을 얻으면 마켓플레이스에 PR을 올린다.

나쁘거나 중복된 스킬이 쉽게 만들어질 수 있으므로,
출시 전 큐레이션 방법을 갖추는 것이 중요하다.

### 스킬 조합

스킬 간 의존성은 아직 네이티브로 지원되지 않지만,
다른 스킬을 이름으로 참조하면 모델이 설치된 스킬을 호출한다.

### 스킬 측정

PreToolUse 훅으로 사내 스킬 사용량을 로깅한다.
인기 있는 스킬이나 기대보다 적게 트리거되는 스킬을 발견할 수 있다.

## 분석

### 왜 이 글이 중요한가

이 글은 Skills에 대한 **첫 번째 대규모 실전 운용 보고서**다.
공식 문서가 "무엇이고 어떻게 만드는가"를 다룬다면,
이 글은 "수백 개를 운영하면 무엇을 알게 되는가"를 다룬다.
Anthropic이 자사 제품을 자사 워크플로에 깊이 적용한 결과물이라는 점에서
외부에서 추측하거나 이론적으로 접근하는 것과는 차원이 다르다.

### 9가지 분류 체계의 가치

스킬을 9가지 유형으로 분류한 것은 단순한 정리가 아니다.
조직에서 "우리에게 빠진 스킬이 무엇인가?"라는 질문을 가능하게 하는 프레임워크다.

- **검증 스킬**(2번)은 대부분의 팀이 간과하는 영역이다.
  Claude의 출력이 정확한지 검증하는 체계를 갖추는 것이 품질의 핵심이라는
  통찰이 담겨 있다.
- **런북**(8번)과 **인프라 운영**(9번)은 스킬을 코드 생성 도구가 아니라
  **운영 자동화 도구**로 확장한다.

### 점진적 공개(Progressive Disclosure)의 일관성

"파일 시스템 자체를 점진적 공개의 수단으로 쓰라"는 조언은 스킬의 설계 철학
전체를 관통한다.
메타데이터 → 본문 → 참조 파일의 3단계 로딩과 동일한 원리를
스킬 내부 구조에도 적용하는 것이다.
이것은 컨텍스트 윈도우가 유한한 LLM 환경에서의 정보 아키텍처 원칙이다.

### Gotchas 중심 설계

"당연한 것은 쓰지 말고, Gotchas를 쓰라"는 조언은 직관에 반한다.
보통 문서를 쓸 때 정상 경로(happy path)부터 설명하지만,
LLM에게는 이미 알고 있을 가능성이 높은 정상 경로보다
**실패 경로가 더 높은 정보 가치**를 가진다.
이것은 프롬프트 엔지니어링의 일반 원칙으로 확장할 수 있다.

### 인사이트

**1. 스킬은 “프롬프트 엔지니어링”이 아니라 “컨텍스트 엔지니어링”이다.**
폴더 구조, 점진적 공개, 스크립트 포함이라는 개념은
단순한 프롬프트 작성과 근본적으로 다르다.
파일 시스템 자체를 에이전트의 인지 구조로 설계하는 것이다.
이것은 전통적인 소프트웨어 아키텍처와 에이전트 설계가 만나는 교차점이다.

**2. “검증 스킬에 엔지니어 1주일을 투자할 가치가 있다”는 파격적 주장이다.**
보통 테스트 자동화에 투자하지 스킬 하나에 1주일을 쓰라고 하지 않는다.
이것은 에이전트가 자율적으로 작업할 때 검증이 병목이 된다는
Anthropic 내부의 실전 교훈을 반영한다.
에이전트의 자율성이 높아질수록 검증의 가치는 기하급수적으로 증가한다.

**3. “Claude를 레일에 가두지 마라”는 직관에 반하는 조언이다.**
보통 프롬프트 엔지니어링에서는 구체적일수록 좋다고 하지만,
재사용 가능한 스킬에서는 오히려 유연성이 핵심이다.
이것은 라이브러리 설계의 원칙(범용성 vs 구체성)과 정확히 같은 트레이드오프다.

**4. 온디맨드 훅(`/careful`, `/freeze`)은 에이전트 안전성의 새로운 패러다임이다.**
항상 켜진 제약이 아니라, 맥락에 따라 동적으로 활성화되는 가드레일이다.
이것은 인간 개발자가 “프로덕션 작업할 때만 더 조심하는” 행동 패턴을
에이전트에 이식한 것이다.

**5. 유기적 마켓플레이스 운영은 오픈소스 생태계의 축소판이다.**
중앙 통제 없이 샌드박스→견인력→승격의 흐름은
npm, PyPI의 패키지 성장 과정과 유사하지만,
조직 내부에서 더 빠른 피드백 루프로 동작한다.
다만 큐레이션 없이 스킬이 폭증하면 “dependency hell”의 에이전트 버전이 될 수 있다.

**6. `${CLAUDE_PLUGIN_DATA}`는 에이전트에 상태를 부여하는 것이다.**
스킬이 자신의 실행 이력을 로그로 남기고 다음 실행 시 참조하는 패턴은
에이전트가 단순한 도구에서 **학습하는 시스템**으로 진화하는 첫 단계다.
standup-post가 어제와의 차이를 인식하는 것은
최소한의 메커니즘으로 최대한의 연속성을 확보하는 우아한 접근이다.

## 비평

### 구체적 실패 사례의 부재

9가지 유형과 제작 팁을 나열하지만, **실제로 실패한 스킬**의 사례가 없다.
“여러 유형에 걸치는 스킬이 혼란스럽다”고 언급하면서도 구체적인 안티패턴을
보여주지 않는다.
무엇이 잘 안 되었는지를 보여주는 것이 무엇이 잘 되는지를 보여주는 것만큼
가치 있다.

### 측정 지표의 모호함

“PreToolUse 훅으로 사용량을 측정한다”고 하지만, 어떤 지표로 스킬의 **품질**을
판단하는지는 다루지 않는다.
사용 빈도와 성공률은 다르다.
스킬이 호출된 후 작업이 성공적으로 완료되었는지, 재시도가 얼마나 발생하는지
등의 품질 지표가 빠져 있다.

### 의존성 관리 문제의 경시

스킬 간 의존성을 “이름으로 참조하면 모델이 호출한다”로 처리하는 것은 취약하다.
설치되지 않은 스킬을 참조하면 무슨 일이 일어나는지, 버전 충돌은 어떻게
처리하는지 다루지 않는다.
수백 개 스킬을 운영한다면 이 문제가 실질적으로 발생했을 텐데,
해결책 없이 “아직 네이티브 지원이 없다”로 넘어간다.

### 보안 고려의 부재

`${CLAUDE_PLUGIN_DATA}`에 데이터를 저장하고, 셸 스크립트를 실행하고, 훅을
등록하는 스킬은 **공급망 공격(supply chain attack)**의 벡터가 될 수 있다.
마켓플레이스에서 누군가의 스킬을 설치할 때의 보안 검증 절차에 대한 언급이
전혀 없다.
