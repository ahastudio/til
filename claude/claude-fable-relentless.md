# Claude Fable은 끊임없이 적극적으로 움직인다

원문: [Claude Fable is relentlessly proactive](https://simonwillison.net/2026/Jun/11/fable-is-relentlessly-proactive/)

HN 토론: <https://news.ycombinator.com/item?id=48498573> (761점, 654개 댓글)

## 요약

Simon Willison이 Claude Fable 5로 CSS 스크롤바 버그를 디버깅하면서
"끊임없이 적극적(relentlessly proactive)"이라는 표현을 남겼다.
스크린샷 하나와 짧은 지시만 받고도 Fable은
로컬 개발 서버를 실행하고, 여러 브라우저를 조작하고,
측정 코드를 주입하고, 맞춤형 자동화 솔루션을 직접 구축했다.

## 분석

Fable이 독자적으로 수행한 작업들은 인상적이다.

Firefox와 Safari를 열고 Python과 Quartz 프레임워크로 자체 창 감지 메커니즘을 만들었다.
활성 창을 이름으로 쿼리해 특정 애플리케이션을 찾아 스크린샷을 캡처했다.

Datasette 템플릿에 JavaScript를 주입해 페이지 로드 시 키보드 단축키가 자동으로 실행되도록 했다.
사용자 인터랙션 없이 특정 상태를 시뮬레이션하기 위한 것이다.

POST 요청을 받는 최소 Python HTTP 서버를 직접 작성하고,
페이지 템플릿을 수정해 계산된 CSS와 DOM 측정값을 이 서버로 전송하게 했다.
프로세스 간 통신 채널을 스스로 만든 것이다.

두 줄짜리 CSS 수정을 위해 지출된 토큰 비용은 12.11달러였다.
리소스 한계에 도달한 후 Fable은 Claude Opus로 다운그레이드해 작업을 계속했다.

## 비평

### 보안 위험과 통제 불능의 경계

Willison은 이 능력이 가진 보안 위험을 명시적으로 경고한다.
"코딩 에이전트는 터미널에서 당신이 타이핑할 수 있는 모든 것을 할 수 있다."
이 수준의 자율성은 프롬프트 인젝션이나 손상된 코드 컨텍스트를 통한 악의적인 지시에 그대로 노출된다.
그러나 Willison이 이 경고를 글의 주석처럼 다루는 것은 위험을 과소평가하는 것이다.

HN에서도 이 우려가 강하게 제기됐다.
teraflop은 위험을 알면서도 무방비 상태로 사용하는 것을 "에어백 위험을 경고하면서 대시보드에 발을 올린 채 달리는 것"에 비유했다.[^teraflop]
not_kurt_godel은 Claude Code가 명시적 지시 없이 브라우저 창을 열고 탐색하는 것을 직접 목격했고, 이것이 로컬 머신에서 터미널 기반 LLM을 사용하지 않는 이유라고 밝혔다.[^not_kurt_godel]
tech234a는 Anthropic의 Mythos 시스템 카드에서 언급된 사례, 즉 모델이 샌드박스를 탈출해 연구자에게 이메일을 보낸 일화를 상기시켰다.[^tech234a]

### 비용 효율과 전략 선택 능력

HN에서는 주니어 개발자라면 CSS 클래스를 검색해 한 줄을 추가했을 것이라는 비판이 나왔다.
swingboy는 "이것은 그냥 overflow 문제 아닌가? 이 모델들이 아직 얼마나 갈 길이 먼지, 그리고 기본 CSS를 모르는 사람이 얼마나 많은지를 동시에 보여준다"고 말했다.[^swingboy]
이 비판은 표면적으로 타당하지만 사안의 복잡성을 단순화한다.
CSS 스크롤바 버그가 정확히 어디서 왜 발생하는지를 두 줄의 수정으로 해결하기 위해서는 먼저 원인을 파악해야 한다.
Fable이 취한 경로, 즉 브라우저별 동작 차이를 측정하고 DOM 상태를 캡처하는 방식은 원인 파악을 위한 접근법이다.
문제는 그 원인 파악 방식이 지나치게 정교했다는 것이고, 이것은 에이전트가 "인간이 하듯 간단한 방법을 먼저 시도하는" 메타 전략을 가지고 있지 않음을 보여준다.

paytonjjones도 핵심을 짚었다.[^paytonjjones]
"보안보다 더 신경 쓰인 것은 CSS 두 줄을 고치기 위해 얼마나 많은 토큰을 썼는지"라는 것이다.

12달러라는 비용은 단순히 비싸다는 문제를 넘어 에이전트 설계의 근본적 질문을 제기한다.
에이전트가 작업을 시작하기 전에 예상 비용을 추정하거나 사용자에게 확인을 요청하는 메커니즘이 없다면, 간단한 작업이 언제 12달러짜리 탐험으로 확대될지 예측하기 어렵다.
리소스 한계에 도달해 자동으로 더 저렴한 모델로 다운그레이드하는 것은 한계 관리가 아니라 비용 초과 후 대응이다.

wraptile은 Fable이 복잡한 캐시 업데이트 솔루션을 만든 사례를 공유하며, "실제로는 더 둔하지만 더 적극적인 도구"라고 평가했다.[^wraptile]
AI 발전이 실질적 지능 향상보다는 시스템을 더 공격적으로 만드는 방향으로 가고 있다는 지적이다.

### 자율성의 긍정적 측면

반면 Fable의 자율성이 실제로 유용한 경우도 보고됐다.
BosunoB는 Fable이 UI 변경을 직접 검증하기 위해 CLI로 게임을 실행하고, 결과 프레임을 캡처해 분석하는 과정을 자율적으로 수행했다고 전했다.[^BosunoB]
웰컴 스크린이 방해하자 임시 작업 트리를 만들어 스크린을 삭제하고 다시 실행한 것이다.
Cadwhisker는 Fable 5가 Python 충돌을 디버깅하면서 UI 클릭을 시뮬레이션하는 테스트 하네스를 직접 작성하고, 힙 할당 오버플로를 찾아내 재현 예시와 우회 방법을 만들었다고 전했다.[^Cadwhisker]
이 경험에서 Cadwhisker는 각 명령을 직접 승인하며 토큰 비용을 관리했다는 점을 강조했다.

에이전트가 "가장 그럴듯한 방법을 먼저 시도하는 경향"은 현재 LLM 기반 에이전트의 구조적 특성이다.
LLM은 훈련 데이터에서 본 패턴을 재현하며, 복잡한 접근법이 더 많이 문서화된 경향이 있어 단순한 접근법보다 먼저 생성될 수 있다.
이 편향을 교정하는 것은 프롬프팅만으로 해결하기 어렵고, 에이전트가 스스로의 행동 비용을 평가하는 메타인지 능력을 요구한다.

Willison이 이 경험을 긍정적으로 서술한다는 점도 주목할 만하다.
"인상적이다", "끊임없이 적극적이다"라는 평가는 12달러를 지불하고 두 줄의 CSS 수정을 얻은 경험에 대한 반응치고는 놀랍도록 호의적이다.
이 호의적 프레이밍이 AI 에이전트 능력에 대한 공정한 평가인지, 아니면 새로운 기술에 대한 열정이 비용 효율 판단을 흐리는 것인지를 구분하는 것이 필요하다.

## 인사이트

### 능력과 효율의 간극

Fable의 "끊임없는 적극성"은 능력과 위험의 양면이다.
지시 없이도 맥락을 파악해 필요한 인프라를 구축하는 것은 강력한 도구이지만, 같은 자율성이 샌드박스 없는 환경에서는 통제 불능 위험이 된다.
AI 에이전트의 능력이 커질수록 실행 환경의 경계 설계가 더 중요해진다.

이 사례가 보여주는 더 중요한 패턴은 에이전트의 "능력"과 "효율" 사이의 간극이다.
무엇을 할 수 있느냐와 무엇을 해야 하느냐는 다른 문제다.
두 줄의 CSS 수정을 위해 12달러를 쓰는 에이전트는 능력 있는 에이전트지만 효율적인 에이전트는 아니다.
미래의 코딩 에이전트는 능력뿐만 아니라 상황에 맞는 전략 선택 능력을 갖추어야 한다.

### 평가 기준의 전환

이 글은 기술 데모로서 인상적이지만, 그 인상적임이 어떤 기준에서 나오는지를 명확히 해야 한다.
복잡한 작업을 자율적으로 수행했다는 것은 사실이지만, 같은 작업을 더 단순하게 수행했다면 더 인상적이었을 것이다.
AI 에이전트의 능력을 평가하는 기준이 "얼마나 복잡한 것을 할 수 있느냐"에서 "얼마나 적절한 방식으로 목표를 달성하느냐"로 이동해야 할 시점이다.
