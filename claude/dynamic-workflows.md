# Claude Code 다이나믹 워크플로

<https://code.claude.com/docs/en/workflows>

원문: <https://claude.com/blog/introducing-dynamic-workflows-in-claude-code>

## 요약

Anthropic이 2026년 5월 28일 Claude Code에 다이나믹 워크플로(dynamic workflows)를
발표했다. 단일 세션에서 수십~수백 개의 병렬 서브에이전트(subagent)를 오케스트레이션해
단일 패스로는 불가능한 대규모 작업을 처리하는 기능이다. 기존에 분기(quarter) 단위로
걸리던 프로젝트 타임라인을 며칠로 단축할 수 있다고 한다.

내부적으로 워크플로는 Claude가 작성한 **JavaScript 스크립트**다. 런타임이 이 스크립트를
대화와 분리된 격리 환경에서 실행하고, 중간 결과는 스크립트 변수에 저장된다. 덕분에
Claude의 컨텍스트 윈도우에는 최종 결과만 남아 대화가 계속 응답 가능한 상태를 유지한다.
스크립트는 읽을 수 있고 다시 실행할 수 있으므로, 반복적인 프로세스를 명령어로 저장해
재사용할 수 있다.

대표 사용 사례는 세 가지다. 첫째, 코드베이스 전체를 병렬로 탐색하고 독립적으로
검증하는 버그 헌트와 보안 감사. 둘째, 수천 개 파일에 걸친 프레임워크 교체·언어 포팅
같은 대규모 마이그레이션. 셋째, 납품 전 독립 검증이 필요한 고위험 작업이다.

에이전트들은 독립적으로 동일 문제를 시도하고 결과를 교차 검증(adversarial verification)한
뒤 통합한다. 진행 상황은 자동 저장되어 중단된 작업을 재시작 없이 이어받을 수 있다.
런타임 제약으로 동시 실행 에이전트는 최대 16개, 실행 당 총 에이전트는 최대 1,000개다.

## 사용법

### 워크플로 실행 방법

워크플로를 시작하는 세 가지 방법이 있다.

첫째, 프롬프트에 `workflow` 키워드를 포함하면 Claude가 해당 작업용 스크립트를 작성한다.

```text
Run a workflow to audit every API endpoint under src/routes/ for missing auth checks
```

둘째, `/effort ultracode`로 `ultracode` 모드를 켜면 세션 내 모든 실질적인 작업에
자동으로 워크플로를 적용한다. `ultracode`는 `xhigh` 추론 노력과 자동 워크플로 결정을
결합한 설정이다. 세션이 끝나면 리셋된다.

셋째, 내장 워크플로 `/deep-research`를 직접 실행한다.

```text
/deep-research What changed in the Node.js permission model between v20 and v22?
```

`/deep-research`는 여러 각도로 웹 검색을 분산하고, 소스를 교차 검증하며,
검증을 통과하지 못한 주장을 걸러낸 인용 보고서를 반환한다.

### 실행 모니터링

`/workflows`로 실행 중·완료된 워크플로 목록을 확인하고, 선택하면 진행 뷰가 열린다.

| 키               | 동작                                          |
| ---------------- | --------------------------------------------- |
| `up` / `down`    | 단계 또는 에이전트 선택                       |
| `Enter` / `->`   | 단계 -> 에이전트 드릴다운, 프롬프트·결과 확인 |
| `p`              | 실행 일시정지 / 재개                          |
| `x`              | 선택된 에이전트 또는 전체 워크플로 중단       |
| `r`              | 선택된 에이전트 재시작                        |
| `s`              | 스크립트를 명령어로 저장                      |

### 워크플로 저장과 재사용

실행 후 `s`를 눌러 두 위치 중 하나에 저장할 수 있다.

- `.claude/workflows/` — 프로젝트 공유, 저장소를 클론한 팀원 모두 사용 가능
- `~/.claude/workflows/` — 개인용, 모든 프로젝트에서 사용 가능

저장된 워크플로는 `/` 자동완성에 내장 워크플로와 함께 표시된다.

### 비활성화

개인 비활성화는 `/config` 토글, `settings.json`에 `”disableWorkflows”: true`,
또는 `CLAUDE_CODE_DISABLE_WORKFLOWS=1` 환경변수로 가능하다.
조직 전체 비활성화는 managed settings에서 설정한다.

가용 환경은 Claude Code CLI, Desktop, VS Code 익스텐션이며, Claude API,
Amazon Bedrock, Vertex AI, Microsoft Foundry를 통해서도 접근 가능하다.
Max·Team 플랜은 기본 활성화, Enterprise는 관리자가 설정에서 켜야 한다.
Claude Code v2.1.154 이상이 필요하다.

## 분석

### Bun 포팅 사례의 의미

Bun 개발자 Jarred Sumner가 다이나믹 워크플로로 Bun을 Zig에서 Rust로 포팅한 결과가
핵심 증거로 제시됐다. 75만 줄의 Rust 코드, 기존 테스트 99.8% 통과, 첫 커밋에서
머지까지 11일. 이 수치는 인상적이지만 몇 가지 맥락이 필요하다.

포팅은 기계적 변환이 가능한 작업에서 LLM이 잘하는 영역이다. 동작 사양이 기존
코드베이스에 이미 명시돼 있고, 테스트가 정답을 정의해주며, 개별 파일 단위로 독립적인
변환이 가능하다. “수천 개 파일에 걸친 마이그레이션”이라는 사용 사례 자체가 이런 구조적
조건을 전제로 한다. 맥락 의존성이 높고 판단이 필요한 작업에서의 성능은 별개 문제다.

### 교차 검증(adversarial verification) 구조

단순히 병렬로 실행하는 것이 아니라 에이전트들이 서로의 결과를 검증한다는 점이 주목된다.
이것은 LLM의 자기 확인 한계를 우회하는 아키텍처적 선택이다. 같은 모델이 같은 오류를
생성하고 같은 오류를 놓치는 경향이 있는 문제를, 독립 시도 + 교차 검증으로 완화한다.

다만 이 “독립성”이 얼마나 진짜인지는 미지수다. 동일한 모델 가중치를 공유하는 에이전트들은
같은 코너케이스에서 같은 방식으로 실패할 수 있다. 진정한 독립적 검증을 위해서는 다른
모델이나 다른 접근법이 필요할 수 있다.

### `ultracode`와 자율성의 스펙트럼

`ultracode` 설정은 단순한 편의 기능을 넘어 Claude의 자율성 수준을 높이는 선택지다.
사용자가 워크플로 사용 여부를 직접 결정하는 것이 아니라 Claude가 판단한다.
이것은 “에이전트에게 얼마나 위임할 것인가”라는 근본적인 질문을 사용자 설정 화면에서
다루는 방식이다.

## 비평

### 강점

실제 고객 사례와 정량적 수치를 제시한 점이 신뢰성을 높인다. Bun 포팅의 75만 줄,
99.8%, 11일이라는 수치는 “대폭 향상됐다”는 식의 모호한 주장과 다르다. Klarna,
CyberAgent 같은 기업의 실사용 증언도 구체적인 사용 맥락을 담고 있다.

토큰 소모 경고를 전면에 내세운 점도 정직하다. 많은 AI 기능 발표가 비용 함의를 숨기는
경향이 있는데, 이 발표는 처음부터 “상당히 더 많은 토큰을 소모한다”고 명시했다.

### 약점

한계와 실패 사례가 없다. 어떤 종류의 작업에서 워크플로가 잘 작동하지 않는지,
서브에이전트 간 충돌이나 일관성 문제가 어떻게 처리되는지, 토큰 소모가 “상당히 많다”는
것이 구체적으로 얼마나 되는지 공개되지 않았다. “수십에서 수백 개의 서브에이전트”가
실제로 얼마의 비용인지는 사용자가 직접 경험하기 전까지 알기 어렵다.

교차 검증의 구체적인 메커니즘도 블랙박스로 남는다. 에이전트들이 어떤 기준으로 서로의
결과를 검증하는지, 불일치가 발생했을 때 어떻게 해결하는지 설명이 없다.

## 인사이트

### 복잡성의 단위가 바뀌고 있다

소프트웨어 개발에서 “한 세션에 할 수 있는 일”의 단위는 계속 커져왔다. 처음에는 단일
파일 편집이었고, 그 다음은 여러 파일에 걸친 리팩터링, 그리고 컨텍스트 윈도우 확장으로
전체 코드베이스 인식이 가능해졌다. 다이나믹 워크플로는 이 단위를 “프로젝트 전체의
병렬 변환”으로 끌어올린다.

이것이 의미하는 바는 단순히 “더 빨라진다”가 아니다. 이전에는 엔지니어링 팀의 조율 없이
불가능했던 작업들, 즉 전사 코드베이스 보안 감사, 프레임워크 마이그레이션,
대규모 API 버전 업그레이드 같은 것들이 단일 명령으로 착수 가능한 영역으로 들어온다.
작업의 성격이 바뀌면 누가 그 작업을 소유하느냐도 바뀐다.

시니어 엔지니어의 핵심 역할 중 하나는 이런 복잡한 작업을 분해하고 조율하는 것이었다.
다이나믹 워크플로가 이 조율 기능을 맡기 시작하면, 시니어십의 정의가 달라질 수밖에 없다.

### 병렬 에이전트 오케스트레이션은 분산 시스템 문제다

수십~수백 개의 에이전트가 동시에 같은 코드베이스를 수정할 때, 표면적으로는 편리해
보이지만 근본적으로 분산 시스템의 고전적 문제들이 재등장한다. 충돌(conflict),
정합성(consistency), 순서 의존성(ordering dependency).

Anthropic이 “협업은 대화 외부에서 이루어진다”고 설명하는 것은 에이전트 간 조율이
사용자에게 노출되지 않는 레이어에서 처리된다는 뜻이다. 이것은 복잡성을 추상화한다는
면에서 좋지만, 문제가 생겼을 때 디버깅이 어려워진다는 면에서 위험하다.

분산 시스템에서 “자동으로 잘 처리된다”는 보장은 특정 조건 하에서만 성립한다.
그 조건이 무엇인지 이해하지 못한 채로 워크플로에 전적으로 의존하면,
99.8%가 통과하는 테스트 스위트에서 발견되지 않은 0.2%의 미묘한 불일치가
프로덕션 환경에서 예기치 않게 터질 수 있다.

### 토큰 비용이 새로운 엔지니어링 제약이 된다

전통적인 소프트웨어 개발에서 자원 제약은 컴퓨팅 시간과 메모리였다. 에이전트 시대에는
토큰 소모량이 새로운 제약으로 추가된다. 다이나믹 워크플로가 “상당히 더 많은 토큰을
소모한다”는 경고는 이 제약을 직접 인정하는 것이다.

이것은 기술적 결정이 비용 결정과 분리될 수 없다는 의미다. “이 마이그레이션을
워크플로로 돌려야 하는가”는 단순히 가능성의 문제가 아니라 비용 효율성의 문제가 된다.
엔지니어가 워크플로를 적절히 활용하려면 작업의 복잡성과 예상 토큰 소모량, 그리고
그것이 가져오는 가치를 함께 판단해야 한다.

흥미로운 것은 이 비용 구조가 클라우드 컴퓨팅 초기와 유사하다는 점이다.
EC2가 처음 나왔을 때도 “인스턴스를 언제 키고 끌 것인가”가 엔지니어링 판단의
중요한 요소가 됐다. 토큰 비용을 최적화하는 능력이 에이전트 시대의 인프라 역량이 될 것이다.

### “연구 프리뷰”가 의미하는 것

다이나믹 워크플로가 “연구 프리뷰(research preview)”로 출시됐다는 것은 단순한
마케팅 표현이 아니다. Anthropic이 수백 개의 에이전트가 협력하는 시스템에서
어떤 실패 모드가 나타나는지 아직 완전히 이해하지 못한다는 인정이기도 하다.

대규모 병렬 에이전트 시스템은 새로운 종류의 버그를 만들 수 있다. 단일 에이전트의
실수가 아니라 에이전트들 간의 상호작용에서 발생하는 창발적 오류(emergent error)다.
어느 한 에이전트만 보면 정상이지만, 그들의 결과가 합쳐질 때 비로소 드러나는 불일치.
이런 오류는 기존의 테스트와 코드 리뷰 방법론으로는 포착하기 어렵다.

“연구 프리뷰”를 프로덕션 크리티컬 작업에 사용하는 팀들은 이 점을 인지해야 한다.
Bun 포팅이 성공한 것은 Jarred Sumner가 그 과정을 세밀하게 감독했기 때문이기도 하다.
워크플로의 자율성을 신뢰하되, 그 신뢰를 검증하는 메커니즘을 함께 갖춰야 한다.
