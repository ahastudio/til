관련 글:

## 요약

Claude Code 공식 문서가 정의하는 advisor 도구는 “메인 모델과 함께, 더 강한 advisor 모델을 짝지어 작업의 결정적 순간에만 호출하는” 서버 사이드 도구다.
advisor는 전체 대화(모든 도구 호출과 결과 포함)를 받고 가이던스를 돌려주며, Claude는 그 가이던스를 적용해 작업을 이어 간다.
실험적 기능이고 Claude Code v2.1.98 이상 + Anthropic API 직결 환경에서만 동작한다 — Amazon Bedrock, Google Vertex AI, Microsoft Foundry에서는 사용 불가다.
모델은 advisor 역할만이 아니라 메인 역할도 지원 목록이 정해져 있고(Opus 4.6+, Sonnet 4.6, Haiku 4.5, 그리고 v2.1.170+에서 Fable 5), advisor는 항상 메인보다 같거나 더 유능해야 한다.

설정 경로는 세 가지다 — 세션 내 `/advisor` 명령(선택을 사용자 설정에 영구 저장), 설정 파일의 `advisorModel` 키, 그리고 단발 세션용 `--advisor` 플래그.
세션 안에서 `/advisor opus`처럼 직접 지정하거나, settings에 `{“advisorModel”: “opus”}`로 박아 두거나, `claude --advisor opus`로 일회성으로 켤 수 있다.
플래그가 설정 파일보다 우선하며, 메인 모델이 advisor를 지원하지 않으면 `/advisor`는 선택만 저장하고 비활성 상태로 두지만 `--advisor`는 오류로 종료한다.
서브에이전트는 부모와 동일한 advisor 설정을 상속하되, 자기 모델 기준으로 페어링 적합성을 다시 검사한다.

호출 시점은 모델이 결정한다.
규칙 기반이 아니라 모델 주도로, 접근 방식을 확정하기 전·반복 오류에 막혔을 때·작업 완료를 선언하기 전이 전형적이다.
프롬프트에서 “consult the advisor before you continue” 같은 자연어 지시로 명시 호출도 가능하지만, 횟수 강제·상한을 두는 설정은 없다.
호출 중에는 advisor 모델명이 적힌 `Advising` 라인이 트랜스크립트에 나타나고, 결과가 돌아오면 `Ctrl+O`로 펼쳐 전문을 볼 수 있다.
Claude는 advisor 조언을 대체로 따르지만, 자기 증거(예: 파일 내용·실패한 시도)와 충돌하면 무조건 따르지 않고 충돌을 표면화한다.

비용은 메인 모델 사용량 위에 advisor 토큰이 추가된다.
API 청구에서는 advisor 모델의 입출력 단가로 별도 과금되고, 구독 플랜에서는 플랜의 사용량 한도에 합산된다.
호출이 결정 시점에 한정되기 때문에 ‘메인을 종일 강한 모델로 돌리는’ 것보다 일반적으로 더 싸다.
프롬프트 캐시 측면에서 `/advisor` 토글은 메인 모델 캐시를 무효화하지 않고, advisor의 가이던스는 이후 턴에서 트랜스크립트의 일부로 캐시된다.
다만 advisor 자신의 호출에는 캐시가 적용되지 않아 매번 전체 트랜스크립트를 새로 읽는다.

비활성화 경로도 단계별로 제공된다 — `/advisor off` 또는 picker의 “No advisor” 선택으로 저장된 설정을 비우고, 명령·플래그까지 완전히 막으려면 환경변수 `CLAUDE_CODE_DISABLE_ADVISOR_TOOL=1`을 둔다.
문서 말미는 advisor를 다른 ‘제2 모델 조합 수단’과 정렬해 비교한다 — `opusplan`(플랜 모드에서만 Opus 사용), 서브에이전트(위임된 하위 작업 전체에 강한 모델 사용), `/model`(이후 모든 턴 교체).
advisor는 그 중 “결정 시점에 모델이 스스로 호출”이라는 좌표를 차지한다.

## 분석

### 문서의 핵심 메시지는 ‘모델 주도 호출’이라는 한 줄에 압축된다

이 문서는 새 기능을 설명하는 레퍼런스인 동시에, 매우 명확한 디자인 철학을 전달한다 — “언제 호출할지는 Claude가 결정한다(Claude decides when to call it).”
이 한 문장은 advisor의 모든 다른 결정을 끌어낸다.
설정에 호출 횟수 상한이 없고, 강제 호출 옵션도 없으며, ‘이 단계에서만 호출’ 같은 트리거가 없다.
사용자가 줄 수 있는 신호는 프롬프트의 자연어 지시뿐이다.

이는 같은 제품군의 다른 ‘제2 모델’ 패턴과 의도적으로 구분되는 지점이다.
`opusplan`은 사용자가 플랜 모드 진입을 트리거하고, 서브에이전트는 부모가 위임을 명시하며, `/model`은 사용자가 직접 전환한다.
세 가지 모두 인간 또는 코드가 트리거지만, advisor만 모델 스스로의 ‘의문 신호’가 트리거다.
표 마지막 칸의 “Claude calls it when it needs guidance”는 단순한 사용법 차이가 아니라, advisor가 “에이전트의 내성(introspection)을 인프라화한” 도구라는 신호다.

### advisor 페어링 표는 사실상의 ‘능력 위계’ 선언이다

수용 가능한 페어링 표는 기술 문서로 위장한 모델 위계 발표다 —
Haiku < Sonnet < Opus < Fable.
같은 등급 안에서도 “Opus 4.7 메인 + Opus 4.6 어드바이저는 거부된다”는 명시는, 버전 번호가 단순 식별자가 아니라 ‘능력 순서’를 표현하는 단조 함수임을 공식화한다.
이는 마케팅적 클레임이 아니라 시스템 동작 — 페어링 거부 — 으로 강제된다.

이 위계의 비대칭은 흥미롭다.
Fable 5는 자기 자신만 advisor로 받는다 — 즉 “Fable보다 더 강한 advisor는 존재하지 않는다”는 선언이다.
반대로 Haiku는 advisor로 ‘사용될 수’ 없다.
이로써 능력 스펙트럼은 양 끝(Haiku, Fable)에서 비대칭으로 닫히고, 그 사이 Sonnet/Opus만이 양방향으로 자유롭다.
advisor 표는 Anthropic 모델 패밀리의 능력 순서를 인터페이스 차원으로 응고시킨 사례다.

### 캐싱 정책은 “호출 비용”의 진짜 위치를 노출한다

“`/advisor` 토글은 메인 모델 캐시를 무효화하지 않는다”와 “advisor 자신의 호출에는 캐시가 적용되지 않는다”는 두 문장이 합쳐지면 advisor의 진짜 비용 구조가 드러난다.
메인 모델 측면에서 advisor는 ‘부가 비용 없이 켜고 끌 수 있는’ 토글이지만, advisor 측면에서는 매번 전체 트랜스크립트를 처음부터 읽는 무캐시 호출이다.
즉 비용은 advisor 모델 쪽으로 전가되고, 그 양은 트랜스크립트 길이에 선형적으로 비례한다.

이 구조의 함의는 단순하다 — advisor 호출 비용은 호출 횟수가 아니라 ‘호출 시점까지 누적된 컨텍스트 길이’가 결정한다.
긴 디버깅 세션의 끝자락에서 부르는 advisor는 시작에서 부르는 advisor보다 훨씬 비싸다.
문서는 이를 한 문장으로 흘려보내지만, 실무자에게는 “언제 부를지가 비용을 결정한다”는 운영 원칙의 출처다.

### 서브에이전트 상속 + 페어링 재검사는 작은 인프라 결정이지만 큰 의미를 갖는다

서브에이전트가 부모의 advisor 설정을 상속하되 ‘자기 모델’ 기준으로 페어링을 다시 검사한다는 한 줄은 문서에서 가장 조용히 중요한 부분이다.
이 한 줄은 advisor를 ‘세션 전역의 부가 옵션’이 아니라 ‘각 에이전트의 능력 프로필에 정합한 도구’로 자리매김한다.
예컨대 Opus 메인이 Sonnet 서브에이전트를 띄우면, 서브에이전트는 부모의 advisor를 그대로 받지만 자기 입장에서 적합성을 재검사하므로 페어링이 깨질 수 있다.

이 설계는 “서브에이전트는 부모의 미니 복제본”이라는 단순 모델을 거부한다.
대신 서브에이전트를 ‘자기 능력 프로필을 가진 독립 에이전트’로 다루고, 능력에 맞지 않는 advisor는 자동으로 떼어낸다.
이는 다중 에이전트 시스템의 ‘권한 상속’ 규칙을 advisor라는 새 차원에서 처음 정리한 한 사례다.
앞으로 다른 서버 사이드 도구가 advisor와 같은 페어링 제약을 갖게 되면, 같은 패턴 — 상속 + 자기 기준 재검사 — 이 표준이 될 가능성이 높다.

## 비평

### ‘모델 주도 호출’은 디버그 가능성을 의도적으로 희생한다

문서는 advisor 호출 시점을 “모델이 결정한다”고 짧게 적고 끝난다.
그러나 이 결정에는 큰 운영상 비용이 따른다 — 사용자는 advisor가 *왜* 호출되었는지, *왜* 호출되지 않았는지를 사후 추적할 일관된 신호를 받지 못한다.
‘Advising’ 라인은 호출이 일어난 사실은 보여주지만, 그 직전 어떤 신호가 advisor를 트리거했는지에 대한 모델의 내부 판단은 노출되지 않는다.

이 불투명성은 두 가지 실무 비용을 발생시킨다.
하나는 회고적 분석의 어려움 — 같은 작업을 두 번 돌렸을 때 한 번은 advisor가 호출되고 한 번은 호출되지 않는 비결정성을 설명할 방법이 없다.
다른 하나는 의도적 제어의 어려움 — “이 결정에서는 반드시 advisor를 부르라”는 강제 옵션이 없으므로, 중요한 결정에 advisor가 빠지는 위험을 사용자가 직접 차단할 수 없다.
문서는 “프롬프트에 자연어로 적으라”고 하지만, 이는 보장이 아니라 권유다.

같은 종류의 결정을 인간 조직이 한다고 가정해 보자.
“언제 시니어에게 자문할지는 주니어가 알아서 정한다”는 정책은 시니어를 부른 회수가 적은 주니어를 옳다고 평가할 수도, 일을 망친 책임을 묻기 어렵게 만들 수도 있다.
advisor의 모델 주도 호출은 ‘에이전트의 자율성’을 강조하지만, 동시에 ‘에이전트의 책임 추적’을 어렵게 만든다.
문서가 이 트레이드오프를 노출하지 않은 것은 마케팅 자료가 아니라 레퍼런스 문서의 책임이라는 점에서 더 아쉽다.

### 페어링 규칙의 일관성은 능력 위계가 1차원임을 가정한다

advisor는 메인과 같거나 더 유능해야 한다는 규칙은 단순해 보이지만, 그 기저에는 “모델 능력은 1차원으로 정렬 가능하다”는 강한 가정이 깔려 있다.
실제 모델 능력은 분야별로 비균질적이다 — 한 모델이 코드에서 강하고 다른 모델이 추론에서 강하면, “누가 더 유능한가”는 작업 종류에 따라 답이 다르다.
페어링 표가 이 다차원성을 1차원 순서로 압축하는 순간, 일부 사용 사례에서는 ‘더 유능한 advisor’가 실제로는 ‘다른 결정 편향을 가진 advisor’가 된다.

예를 들어 Sonnet 메인 + Opus advisor 조합은 문서가 권하는 표준이지만, ‘코드 작성 정확도’에 한정해서는 Sonnet이 Opus와 동등하거나 우월한 워크로드가 존재한다는 보고가 흔하다.
이 경우 “Opus의 가이던스를 Sonnet이 따른다”는 흐름이 실제 산출물의 품질을 낮출 수 있다.
문서는 이런 분야별 비균질성을 다루지 않고 “advisor를 따른다, 단 자기 증거와 충돌하면 표면화한다”는 단일 규칙으로만 흐름을 정의한다.

더 깊은 문제는 사용자가 이 비균질성을 advisor 페어링으로 표현할 수 없다는 점이다.
“코드 결정은 Sonnet이, 아키텍처 결정은 Opus가 advise”라는 분야별 advisor 분기는 현재 시스템으로는 불가능하다.
페어링은 1차원 위계로 ‘잠금’되어 있고, advisor는 한 번에 하나만 붙는다.
이는 일관성을 얻기 위해 표현력을 희생한 설계 선택이며, 다음 세대의 advisor 추상화에서 풀어야 할 숙제다.

### advisor 호출 비용을 ‘일반적으로 더 싸다’로 일축한다

문서는 페어링 비용을 짧게 다룬다 — “결정 시점에만 호출되므로 일반적으로 더 강한 모델을 종일 돌리는 것보다 싸다”는 한 줄이다.
그러나 이 진술은 호출 빈도와 트랜스크립트 길이라는 두 변수를 모두 무시한다.
앞서 분석에서 본 것처럼 advisor 호출은 캐시되지 않으므로, 호출 한 번당 비용은 그 시점의 트랜스크립트 전체 길이에 비례한다.

수치를 대입해 보자.
20턴짜리 디버깅 세션에서 advisor가 5번 호출되면, 각 호출이 매번 평균 트랜스크립트 길이만큼의 토큰을 advisor 입력으로 소비한다.
이 5번의 누적 advisor 비용은 “Opus를 메인으로 20턴 돌리되 프롬프트 캐시를 활용한” 비용과 결코 단조적으로 비교 불가능하다.
캐시 적중률, 호출 분포, 트랜스크립트의 출력 토큰 비중 같은 요인에 따라 advisor가 더 비싸지는 시나리오가 존재한다.

문서가 이 시나리오를 다루지 않고 “일반적으로 더 싸다”로 끝낸 것은, 사용자가 비용을 잘못 추정하게 만드는 가장 흔한 함정이다.
실무에서는 `/usage` 출력을 advisor 호출 횟수와 트랜스크립트 길이로 분해해 봐야 진짜 비용을 알 수 있고, 그 분해 방법은 advisor 문서가 아니라 별도의 API 문서로 미뤄져 있다.
레퍼런스 문서의 책임은 정확한 비용 모델을 사용자가 직접 세울 수 있게 하는 것이지, ‘일반적으로 싸다’는 마케팅적 위안을 주는 것이 아니다.

### ‘opusplan vs subagent vs advisor’ 비교 표는 결정 가이드로 충분하지 않다

문서 말미의 비교 표는 깔끔하지만, 실제 선택을 돕는 데 필요한 변수의 절반만 다룬다.
“언제 강한 모델이 도는가”와 “누가 시작하는가”라는 두 축은 분명하지만, “결과물의 품질이 어떻게 다른가”, “디버그 가능성이 어떻게 다른가”, “비용 곡선의 형태가 어떻게 다른가”는 표 어디에도 없다.
사용자는 이 표를 보고 advisor를 선택했다가, 실제로는 opusplan이나 서브에이전트가 더 적합했음을 사후에야 알게 된다.

특히 advisor와 opusplan의 경계는 모호하다.
‘플랜 단계에서만 강한 모델’이라는 opusplan은 advisor의 가장 흔한 호출 패턴(접근 확정 전 호출)과 사실상 겹친다.
두 도구의 차이는 트리거(인간 vs 모델)뿐이고, 산출 품질·비용·디버그 가능성에서의 차이는 문서가 비교하지 않는다.
사용자가 한 도구를 다른 도구로 대체했을 때 실제로 무엇이 바뀌는지를 모른 채 선택하게 된다.

좋은 비교 표라면 “이 워크로드에서는 advisor가, 저 워크로드에서는 opusplan이 우월하다”는 시나리오 매핑을 포함해야 한다.
현재 표는 “기능 X는 트리거 Y를 가진다”는 사실의 나열일 뿐, 의사결정 가이드가 아니다.
이는 ‘비교 섹션의 외양’만 갖춘 채 실제 비교 책임을 사용자에게 떠넘긴다.

## 인사이트

### advisor는 ‘에이전트의 자기 의심’을 인프라화한 첫 사례다

advisor 도구의 진짜 새로움은 ‘더 강한 모델을 부른다’가 아니라 ‘자기 판단을 의심하는 행위를 시스템 호출로 정의했다’는 점이다.
지금까지 에이전트가 자기 출력에 대한 신뢰도를 표현할 통로는 로그 확률, ‘이 답에 대해 얼마나 확신하는가’ 같은 자가 보고용 프롬프트, 또는 사람의 검토 요청뿐이었다.
advisor는 이 ‘의심의 순간’을 호출 가능한 도구로 만들고, 그 호출 결과를 다음 행동으로 직접 연결한다.

이 통찰의 깊은 의미는 도구 호출 추상화의 확장이다.
파일 읽기, 셸 실행 같은 도구들은 ‘외부 세계에 대한 질의’였다.
advisor는 ‘자기 자신의 판단에 대한 질의’를 같은 호출 모델 안에 포섭한다.
이로써 에이전트의 도구 인터페이스는 외부 작업과 내부 성찰을 같은 문법으로 다루게 되고, 미래의 에이전트 설계는 ‘메모리 도구’·‘반박 도구’·‘이전 결정 재검토 도구’ 같은 내부 성찰 도구를 advisor와 같은 패턴으로 추가할 가능성이 높다.

이 패턴이 표준이 되면, 도구 호출의 분류 자체가 바뀐다.
지금은 ‘외부 효과가 있는 도구’와 ‘조회만 하는 도구’로 나뉘지만, 다음 세대에는 ‘외부 도구’와 ‘성찰 도구’의 두 축이 추가될 것이다.
advisor는 그 두 번째 축의 첫 시민이다.

### advisor 호출 시점이 ‘새로운 관측 가능한 지표’가 된다

문서는 advisor 호출 시점을 모델 주도라고만 적지만, 이 호출 시점 자체가 운영 관점에서 매우 값진 신호다.
‘에이전트가 자신의 다음 행동을 의심한 순간’의 분포는 그 에이전트의 자신감 프로파일, 작업 난이도, 모델 학습 데이터의 빈틈을 동시에 드러낸다.
advisor 호출 로그를 시계열로 모으면, 어떤 종류의 결정에서 모델이 자주 막히는지를 후행 분석으로 알 수 있다.

이 신호는 기존의 ‘토큰 사용량’이나 ‘턴 수’ 같은 거친 지표보다 훨씬 정보가 풍부하다.
토큰은 비용을 알려주고 턴은 진행 정도를 알려주지만, advisor 호출은 ‘모델이 인지한 자기 한계’를 알려준다.
조직 단위에서 이 데이터를 모으면, 어느 코드베이스·어느 작업 유형·어느 시점에 모델이 외부 도움을 필요로 하는지의 지도가 그려진다.

이 지도의 두 번째 효과는 평가의 자동화다.
새 모델을 도입할 때 “advisor를 얼마나 자주 부르는가, 그리고 그 빈도가 작업 성공률과 어떻게 상관하는가”라는 메트릭이 자연스럽게 생긴다.
‘부르는 횟수가 적으면서 성공률이 높은’ 모델이 좋은 모델이고, ‘자주 부르면서도 결국 실패하는’ 모델은 능력의 자가 평가 자체가 어긋난 모델이다.
advisor 호출은 모델 평가의 새로운 차원을 열어 준다.

### advisor가 자리잡으면 ‘모델 페어링’이 새로운 SaaS 단위가 된다

advisor 패턴이 일반화되면, AI 도구의 가격·번들·계약 단위가 단일 모델에서 ‘페어링’으로 옮겨갈 가능성이 높다.
현재 Claude·OpenAI의 가격표는 모델 단위로 적혀 있지만, advisor 같은 도구는 “Sonnet 메인 + Opus advisor”라는 페어링을 하나의 운영 단위로 만든다.
이 페어링은 가격, 사용 한도, 캐시 정책, 페어링 호환성 같은 운영 변수를 묶어 두며, 곧 마케팅·계약·SLA의 단위가 된다.

이 변화는 SaaS 가격표의 형태를 바꾼다.
지금은 “모델 X는 입력 1M 토큰당 $3, 출력 $15”라는 평면적 표가 표준이지만, 미래에는 “Sonnet+Opus 페어링, advisor 호출은 평균 0.2 회/턴 가정”처럼 페어링 단위로 가격이 책정될 수 있다.
이는 클라우드 EC2의 인스턴스 타입이 “vCPU + RAM” 묶음으로 표준화된 과정과 구조적으로 닮았다.
초기에는 각 자원을 따로 사다가, 사용 패턴이 안정되면 ‘번들’이 단위가 된다.

이 통찰의 함의는 도구 시장 구조까지 미친다.
페어링이 단위가 되면, ‘Opus를 advisor로 임대한다’는 사업 모델이 등장한다.
어떤 회사가 자기 도메인에 최적화된 메인 모델을 운영하고, advisor만 Anthropic에서 ‘구독’으로 빌려 쓰는 형태다.
advisor 도구의 서버 사이드 실행 구조는 정확히 이런 모델을 가능하게 한다 — 메인은 어디서든 돌고, advisor 호출만 Anthropic 인프라로 라우팅되기 때문이다.
지금 문서가 “Bedrock·Vertex·Foundry에서는 안 된다”고 적은 그 제약이, 미래에는 “advisor는 Anthropic의 SaaS 한정 상품”이라는 비즈니스 경계를 그리는 첫 줄이 된다.

### advisor의 진짜 위험은 ‘권위 위계의 코드화’다

문서가 다루지 않는 가장 큰 트레이드오프는 advisor 도구가 ‘모델 위계’를 시스템 차원에서 코드화한다는 점이다.
페어링 표는 “Opus 4.7 advisor는 Opus 4.6 메인을 받지 않는다”처럼 한 번의 거부로 위계를 강제한다.
이는 단순한 호환성 검사가 아니라, ‘누가 누구를 가르칠 수 있는가’의 규칙을 인프라에 새기는 행위다.
사용자가 “나는 Sonnet의 가이던스가 더 낫다고 본다”라고 판단해도, 페어링은 그 판단을 허용하지 않는다.

이 위계의 1차 효과는 마케팅적 일관성이지만, 2차 효과는 다양성 손실이다.
모델 능력의 다차원성을 1차원 순서로 압축하면, ‘다른 강점을 가진 advisor’가 자리할 공간이 사라진다.
예컨대 ‘보안 전문 모델’, ‘회계 도메인 모델’, ‘저자원 환경 최적화 모델’ 같은 특화 모델이 advisor 역할을 하려면, 현재 페어링 규칙으로는 “일반 능력에서 더 유능해야 한다”는 1차원 기준을 통과해야 한다.
이는 도메인 특화 모델의 advisor 역할 진입을 사실상 봉쇄한다.

장기적으로 이 위계는 ‘메인 모델 = 실행자, advisor = 권위자’라는 사회학적 패턴을 코드화한다.
인간 조직에서 ‘시니어가 주니어를 가르친다’는 규범이 직무·연차·경험의 다차원성을 평탄화하듯, advisor 위계도 모델의 다차원성을 평탄화한다.
이 패턴이 굳어지면, 향후 도전자 모델이 “더 유능하지 않지만 다르게 유능한” 자리로 등장하기 어려워진다.
advisor의 진짜 정치적 함의는 ‘더 강한 모델을 부른다’가 아니라, ‘더 강함의 정의를 인프라가 독점한다’이다.
