# Claude Code Agent View

<https://code.claude.com/docs/en/agent-view>

HN 토론: <https://news.ycombinator.com/item?id=48104699> (5점, 2개 댓글)

## 요약

Agent View는 Claude Code v2.1.139 이상에서 제공되는 기능으로, 여러 백그라운드
세션을 한 화면에서 관리할 수 있는 인터페이스다. `claude agents` 명령으로 진입하며,
현재 실행 중인 모든 세션의 상태를 한눈에 파악하고, 입력이 필요한 세션에만 개입할
수 있다.

각 세션은 슈퍼바이저(supervisor) 프로세스에 의해 터미널과 독립적으로 실행된다.
터미널을 닫거나 Agent View를 종료해도 세션은 계속 작동하며, 나중에 다시 접속해
결과를 확인할 수 있다. 단, 시스템이 절전 모드에 진입하거나 종료되면 실행 중인
세션은 중단된다.

세션 목록은 상태별로 그룹화된다. “Needs input”과 “Ready for review”가 상단에
표시되고, “Working”과 “Completed”가 그 아래 위치한다. 각 행은 세션 이름, 현재
활동 내용, 마지막 변경 시각을 보여주며, 행 요약은 Haiku 계열 모델이 15초 간격으로
자동 갱신한다.

새 세션 생성은 Agent View 하단 입력창에서 프롬프트를 입력하거나, 기존 세션 내에서
`/bg` 명령으로 백그라운드로 전환하거나, 셸에서 `claude --bg “<prompt>”` 형태로
직접 실행할 수 있다. 각 백그라운드 세션은 별도의 git worktree(`.claude/worktrees/`
하위)에서 파일을 편집해 병렬 세션 간 충돌을 방지한다.

세션 관리를 위한 셸 명령도 제공된다. `claude attach <id>`, `claude logs <id>`,
`claude stop <id>`, `claude respawn <id>`, `claude rm <id>` 등을 활용하면 Agent
View를 열지 않고도 스크립팅이나 자동화 파이프라인에서 세션을 제어할 수 있다.

## 분석

### 상태 표현 체계

Agent View는 아이콘의 모양과 색상이라는 두 가지 차원으로 세션 상태를 표현한다.
모양(`✻`, `∙`, `✢`)은 프로세스의 생존 여부와 종류를, 색상(노랑·초록·빨강·회색)은
작업 상태를 나타낸다. 이 이중 인코딩 방식은 두 속성이 서로 독립적으로 변할 수 있기
때문에 필요한 설계 선택이다. 예를 들어 프로세스가 종료(`∙`)됐어도 Completed
상태(초록)일 수 있다.

### Peek 패널과 Attach의 구분

Peek 패널(`Space`)은 전체 대화 내용 없이 최신 출력과 현재 질문만 표시하며,
다중 선택 질문은 숫자 키로 즉시 응답할 수 있다. Attach(`Enter`)는 터미널 전체를
해당 세션으로 전환해 완전한 Claude Code 대화 환경을 제공한다. 이 두 수준의
인터랙션은 “빠른 확인 및 응답”과 “심층 작업”이라는 서로 다른 사용 패턴에
각각 대응한다.

### Worktree 기반 격리

백그라운드 세션은 파일 편집 전 자동으로 독립된 git worktree로 이동한다. 이를
통해 여러 세션이 동일한 저장소에서 병렬로 작업하면서도 서로 간섭하지 않는다. git
저장소가 아닌 경우에는 격리가 이루어지지 않으며, 이때 동일한 파일을 편집하는 병렬
세션은 충돌 위험이 있다.

HN에서는 이 구조에 대해 `claude --worktree` 명령과의 차이를 묻는 질문이 나왔다.
ChatML 개발자 mcastilho는 "컨테이너를 수동으로 실행하는 것과 Compose로 관리하는
것의 차이"[^mcastilho]라고 설명했다. 동일한 기본 기제(primitive)지만, 3~5개의
에이전트를 동시에 실행할 때 대시보드를 통한 오케스트레이션 계층이 가치를 발휘한다는
것이다. nwyin은 유사한 문제를 해결하기 위해 Claude 인스턴스들을 관리하는 데몬 기반
오케스트레이터를 직접 구현한 경험을 공유했다[^nwyin].

### 권한 모드와 모델 설정

권한 모드는 세션을 어떻게 시작했느냐에 따라 결정된다. `/bg`나 `←`으로 기존 세션을
백그라운드로 전환하면 현재 권한 모드가 유지된다. Agent View 입력창이나
`claude --bg`로 신규 디스패치하면 해당 디렉터리의 `defaultMode` 설정이나
서브에이전트의 frontmatter가 적용된다. `bypassPermissions`나 `auto` 모드는
이전에 인터랙티브 세션에서 한 번 수락한 경우에만 백그라운드에서 사용 가능하다.

## 비평

### 강점

Agent View의 가장 두드러진 강점은 “필요할 때만 개입”이라는 비동기 작업 흐름을
명시적으로 지원한다는 점이다. 기존 터미널 멀티플렉서(tmux, screen)가 세션을
단순히 분리해 두는 것과 달리, Agent View는 어떤 세션이 주의를 요하는지 자동으로
분류하고 전면에 노출한다. 이 차이는 에이전트 수가 늘어날수록 더 명확하게 드러난다.

HN의 zknill은 “tmux나 탭으로 충분하다”는 반응이 나올 것을 예상하면서도,
원 프로젝트가 이 문제를 직접 해결하려는 시도 자체를 높이 평가했다[^zknill].
서드파티 도구에 의존하지 않고 기본 경험 안에서 문제를 다루는 접근이 의미 있다는
인정이다.

Worktree 기반 파일 격리는 병렬 AI 세션에서 흔히 발생하는 파일 충돌 문제를
인프라 수준에서 해결한다. 사용자가 명시적으로 분기를 관리하지 않아도 각 세션이
독립적인 작업 공간을 갖는다.

### 약점

리서치 프리뷰 단계이므로 인터페이스와 키보드 단축키가 변경될 수 있다는 점은
주의해야 한다. 도입을 검토하는 팀이라면 워크플로우에 깊이 통합하기 전에 안정화
여부를 확인할 필요가 있다.

구독 사용량(quota)이 세션 수에 비례해 소모된다는 점도 실무적 제약이다. 10개의
에이전트를 병렬로 실행하면 단일 세션 대비 10배의 사용량을 소진한다. 비용과
병렬성 사이의 균형을 의식적으로 관리해야 하며, 현재는 이를 도와주는 자동 조절
기능이 제공되지 않는다.

HN 커뮤니티에서는 병렬성 자체의 가치에 의문을 제기하는 목소리도 있었다.
bob1029는 "하나의 '느린' 에이전트가 실수를 많이 하는 여러 빠른 에이전트를
앞설 수 있다"며 깊이 우선 탐색(DFS) 방식의 직렬 실행을 선호한다고 밝혔다[^bob1029].
100달러어치 토큰을 60초에 소비할 수 있다고 해서 그렇게 해야 하는 것은 아니며,
속도를 추구하는 것이 많은 사람들을 AI에 지치게 만든다는 지적이다.
koliber는 실무적 한계로 주의(attention)의 병목을 언급했다[^koliber]. 병렬 세션을
시도해봤지만 3개 이상은 의미 있게 처리하기 어려웠으며, 여러 에이전트가 서로의
발을 밟을 가능성을 고려하는 인지적 부담 자체가 또 다른 비용이었다는 경험담이다.

또한 gverrilla는 단일 저장소에서 2개 이상의 탭으로 작업하는 경우는 드물다며,
Agent View의 유용성은 7~10개의 병렬 작업을 실제로 구동하는 상황에서만 명확해진다고
지적했다[^gverrilla].

로컬 실행 모델의 한계도 존재한다. 세션은 사용자 기기에서 실행되므로 절전이나
종료 시 중단된다. `claude respawn --all`로 재시작할 수 있지만, 진행 중이던
작업의 컨텍스트 복원이 항상 완벽하지는 않을 수 있다.

## 인사이트

### “주의 큐레이션”으로서의 개발 도구

Agent View가 해결하는 핵심 문제는 기술적 자동화가 아니라 인간의 주의(attention)
할당이다. 기존 개발 도구는 작업의 실행을 도왔지만, 어떤 작업이 지금 사람의 판단을
요하는지를 분류하는 일은 개발자 몫이었다. 터미널 탭을 하나씩 확인하고, 로그를
스크롤하고, 어느 프로세스가 멈춰 있는지 추적하는 인지적 부담은 숨겨진 비용이었다.

Agent View는 이 분류 작업 자체를 자동화한다. “Needs input”과 “Ready for review”를
상단에 고정하는 단순한 구조가 “지금 어디를 봐야 하는가”라는 질문에 대한 시스템의
답변이다. 이는 단순한 UI 편의가 아니라, 개발자가 직접 수행하던 인지 노동을 시스템이
흡수하는 구조적 전환이다.

이러한 관점에서 보면, 향후 AI 에이전트 도구의 경쟁력은 에이전트가 얼마나 많은 작업을
처리하느냐보다 얼마나 정확하게 “지금 사람이 필요한 순간”을 식별하느냐에 달릴
가능성이 높다. 정보 과부하를 줄이고 개입의 타이밍을 제시하는 것, 즉 주의의 큐레이션이
다음 세대 개발 도구의 핵심 역량이 될 것이다.

koliber의 경험은 이 논점을 실무 차원에서 뒷받침한다. 병렬 작업의 병목은 연산
속도가 아니라 사람의 주의였다[^koliber]. 에이전트를 아무리 많이 띄워도 그것을
모니터링하는 인지적 비용이 따르는 한, 주의의 큐레이션은 선택이 아닌 필수다.

### 소프트웨어 개발에서 “감독(supervision)” 패턴의 부상

Agent View에서 가장 주목할 아키텍처 개념은 슈퍼바이저 프로세스다. 사용자 세션과
독립적으로 실행되며, 자동 업데이트 후에도 재연결하고, 자원을 절약하기 위해 유휴
세션의 프로세스를 종료했다가 필요 시 재시작한다. 이 패턴은 Erlang의 OTP 슈퍼바이저
트리나 Kubernetes의 Pod 관리 방식과 구조적으로 닮아 있다.

흥미로운 점은 이 아키텍처가 개발 도구 영역에 적용됐다는 것이다. 운영 환경의
신뢰성 패턴이 개발 도구로 이식되면서, 개발 작업 자체가 장기 실행 서비스처럼
관리된다. `claude respawn --all`이라는 명령은 k8s의 `kubectl rollout restart`와
동형(isomorphic)이다. 개발 세션이 인프라처럼 취급받기 시작한 것이다.

이 전환은 AI 에이전트가 단발성 도구에서 상태를 가진 장기 프로세스로 진화하고
있음을 반영한다. 앞으로 AI 에이전트 플랫폼의 설계는 단일 요청-응답 패턴을 넘어서,
장애 복구, 상태 영속성, 자원 관리 같은 분산 시스템의 문제를 점점 더 깊이
다루게 될 것이다.

### Worktree 격리가 드러내는 협업 모델의 근본적 변화

백그라운드 세션마다 독립된 git worktree를 자동으로 생성하는 설계는 표면적으로는
파일 충돌 방지 기능이지만, 더 깊이 읽으면 AI 에이전트가 인간 협업자와 동일한
워크플로우에 편입되고 있음을 보여준다. 팀 내 여러 개발자가 각자의 브랜치에서
작업하듯, 여러 AI 세션이 각자의 worktree에서 작업한다.

이 구조는 코드 리뷰 문화에도 시사점을 던진다. 세션이 Pull Request를 열면 행 우측에
상태 점(dot)이 표시되고, 점이 초록으로 바뀌면 사람이 검토하고 머지하는 것이
기본 인수인계 흐름이 된다. “AI가 PR을 열고 사람이 검토한다”는 패턴이 예외가 아닌
정상 작업 흐름으로 설계에 내재화된 것이다.

장기적으로 이 모델은 팀의 작업 단위 개념을 바꿀 것이다. 지금까지 “한 사람이 한
브랜치를 담당한다”는 암묵적 규범이 있었다면, 앞으로는 한 사람이 여러 AI 세션을
통해 동시에 여러 브랜치에 기여하는 구조가 일반화될 수 있다. 팀의 속도를 논할 때
“몇 명의 개발자”가 아닌 “몇 개의 세션”이 병렬로 작동하는지가 중요한 변수가
될 것이다. 이는 업무 분배, 코드 리뷰 부담, 팀 구성 방식 전반에 걸쳐 재고를
요구하는 변화다.
