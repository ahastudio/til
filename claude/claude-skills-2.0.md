# Claude Skills 2.0 완전 분석

- 원문: [leopardracer @leopardracer](https://twitter.com/leopardracer/status/2035999459729895493)
- 작성일: 2026-03-23

## 1. 원문 요약

### 핵심 메시지

"상위 99%보다 Claude Skills 2.0을 더 잘 활용하는 법"

다음 세 가지 불만에서 출발한다:

- "매번 AI에게 같은 지시를 반복하는 게 번거롭다"
- "AI가 회사 규칙과 형식을 전혀 기억하지 못한다"
- "팀원마다 AI 활용 방식이 달라, 결국 잘하는 사람만 혜택을 본다"

Anthropic이 이 문제를 해결하는 **Claude Skills**를 업그레이드했다.
단순한 AI 에이전트 업데이트가 아니라, AI에게 비즈니스 프로세스와
전문 지식을 학습시켜 회사 맞춤형 전문가로 성장시키는 차세대 기능이다.

### Skills란 무엇인가?

Skills는 Claude Code가 특정 작업을 수행하기 위해 참조하는
재사용 가능한 지식과 절차의 묶음이다.

- 핵심 파일: `SKILL.md` (Markdown + YAML Front Matter)
- 위치: `.claude/skills/` 하위 폴더
- 동작 방식: 사용자 요청에 맞는 Skill을 자동으로 로드하여 실행

**핵심 가치:** 매 대화마다 선호도와 프로세스를 재설명할 필요가 없다.

### SKILL.md 파일 구조

```yaml
---
name: Your Skill Name
description: Brief description of what this Skill does and when to use it
---
```

```markdown
# Your Skill Name

## Instructions
Provide clear, step-by-step guidance for Claude.

## Examples
Show concrete examples of using this Skill.
```

**메타데이터(YAML)가 핵심이다.** Claude는 시작 시 모든 Skill의 메타데이터를
읽어 시스템 프롬프트에 포함시킨다. 요청이 메타데이터와 매칭될 때만
`SKILL.md` 본문을 파일 시스템에서 로드한다.

### 파일 구조 권장사항

- `SKILL.md`는 **500줄 이하** 유지
- 초과 시 별도 파일로 분리:

```
.claude/skills/my-skill/
├── SKILL.md         ← 핵심 지시 (500줄 이내)
├── templates/       ← 템플릿 파일
└── reference.md     ← 상세 참조 자료
```

**Progressive Disclosure 원칙:** 핵심 지시 먼저, 세부 사항은 필요 시 로드.

### 토큰 효율성 구조 (3계층)

| 계층 | 내용 | 로드 시점 |
|------|------|-----------|
| YAML 메타데이터 | 수십 토큰 | 항상 (시작 시) |
| SKILL.md 본문 | 수백 토큰 | 트리거 시 |
| 참조 파일 | 수천 토큰 | 필요 시 |

### MCP vs Skills

저자가 제시하는 비유: **주방(Kitchen) vs 레시피(Recipe)**

- **MCP**: 프로 주방 — 도구, 재료, 장비를 제공
- **Skills**: 레시피 — 가치 있는 것을 만드는 단계별 지침

MCP 서버만 구축했을 때는 "어떻게 쓸지" 안내가 없어 사용자가 혼란스러워
했다. Skills 도입 후 역할이 명확해졌다:
MCP은 **무엇을 할 수 있는지** 정의하고,
Skills는 **어떻게 하는지** 를 가르친다.

### Skill-Creator (메타 Skill)

Skill을 만들고, 테스트하고, 개선하는 **메타 Skill**이다.

동작 사이클:

1. "어떤 Skill을 만들고 싶은가?" 질문
2. `SKILL.md` 초안 자동 생성
3. 테스트 프롬프트로 실제 실행
4. 결과 평가 및 개선안 제시
5. 만족할 때까지 2~4 반복

### Skills 작성 핵심 원칙

> "Default assumption: Claude is already very smart.
> Only add context Claude doesn't already have."

- Claude는 기본적으로 똑똑하다. 범용 지식, 기초 프로그래밍은 쓸 필요 없다
- 회사 고유 규칙, 특정 라이브러리의 quirk, 도메인 워크플로우에 집중
- 지시의 자유도를 작업에 맞게 조정:
  - **자유도 높음**: 텍스트 기반 지시 (글쓰기 등 다양한 접근이 유효할 때)
  - **자유도 중간**: 파라미터 있는 의사코드/스크립트 (권장 패턴은 있지만
    변형 허용)
  - **자유도 낮음**: 구체적 스크립트 (일관성이 중요하고 실수가 치명적일 때)

### Skills 설치 방법

플러그인 마켓플레이스를 통해 설치한다:

```text
/plugin install document-skills@anthropic-agent-skills
/plugin install example-skills@anthropic-agent-skills
```

공식 Skills 저장소: <https://github.com/anthropics/skills>

커뮤니티 Skills: `skillsmp.com`, `aitmpl.com/skills`

### Skills가 적합한 작업 유형

반복적으로 같은 유형의 작업을 Claude에게 요청하거나,
반복 사용이 필요한 템플릿/자산이 있을 때:

- "회사 템플릿으로 주간 보고서 작성"
- "우리 회사 스타일로 프레젠테이션 생성"
- "특정 형식으로 시장 분석 보고서 정리"

단발성 요청이라면 Skill 불필요 — 그냥 채팅에서 말하면 된다.

## 2. 분석

### 무엇을 해결하는가?

Claude Skills는 근본적으로 **AI의 무상태성(Statelessness) 문제**를
애플리케이션 레벨에서 해결하려는 시도다.

LLM은 대화 세션 간에 아무것도 기억하지 못한다. 이를 보완하기 위해
지금까지는 긴 시스템 프롬프트, 문서 첨부, 반복 설명 등이 사용됐다.
Skills는 이 반복 작업을 **파일 시스템 기반의 모듈형 구조**로 체계화한다.

### 기술적 설계의 핵심

**메타데이터 레이지 로딩 패턴**이 핵심 설계다.

```text
시작 시: 모든 Skill의 메타데이터만 로드 (수십 토큰 × N개)
요청 시: 매칭된 Skill의 본문만 로드 (수백~수천 토큰 × 1개)
필요 시: 참조 파일 추가 로드
```

이는 데이터베이스의 인덱스와 본문 데이터를 분리하는 패턴과 동일하다.
컨텍스트 윈도우를 낭비하지 않으면서 수십 개의 Skill을 운용 가능하게 한다.

### CLAUDE.md와 Skills의 역할 분리

이 프로젝트의 `CLAUDE.md`를 보면 실제로 Skills와 유사한 역할을 수행한다.
차이점:

| 항목 | CLAUDE.md | Skills |
| ---- | --------- | ------ |
| 적용 범위 | 항상 전체 로드 | 트리거 시 선택 로드 |
| 용도 | 전역 규칙 | 특정 워크플로우 |
| 토큰 비용 | 항상 소비 | 필요할 때만 소비 |
| 관리 단위 | 파일 1개 | 폴더 N개 |

이 프로젝트의 `skills/` 디렉터리가 이미 존재하는 것은 우연이 아니다.

## 3. 비평

### 강점

**1. 실용적 추상화 수준**

Skill을 "폴더 + Markdown 파일"로 정의한 것은 탁월한 설계다.
JSON 스키마나 별도 DSL 없이 누구나 만들 수 있다.
진입 장벽을 낮추면서도 구조적 확장성을 확보한다.

**2. Progressive Disclosure는 올바른 아키텍처 패턴**

컨텍스트 윈도우는 RAM과 같다. 불필요한 정보를 미리 올리는 것은
캐시 오염과 같다. 레이지 로딩은 이 문제의 정석적 해법이다.

**3. Skill-Creator의 메타 Skill 개념**

Skill을 만드는 Skill — 이 재귀적 접근은 부트스트래핑 문제를 내부에서
해결한다. 사용자가 Skill 작성법을 배우지 않아도 Claude가 대신 만들어준다.

### 약점 및 한계

**1. 메타데이터 매칭의 신뢰성 문제**

글 본문에도 언급됐다: "실제로 실행되는지 여부는 메타데이터 내용에
크게 의존한다." 이는 결국 프롬프트 엔지니어링 문제다. 메타데이터 `description`을
잘못 작성하면 Skill이 트리거되지 않거나 엉뚱한 Skill이 실행된다.

키워드 매칭인지, 임베딩 기반 시맨틱 매칭인지도 명확하지 않다.
실제 운용 환경에서 Skill 간 충돌 또는 미발동(false negative)이
발생했을 때 디버깅이 어려울 수 있다.

**2. 500줄 제한의 임의성**

"500줄 이하"라는 권장 기준은 토큰 수가 아닌 줄 수 기준이다.
코드 예시가 많은 Skill은 500줄이 금방 초과되고, 반대로 내용이 없는 Skill은
500줄 안에 들어온다. 실질적인 컨텍스트 예산 관리에는 토큰 수 기준이 더 정확하다.

**3. Skills vs RAG의 경계 모호성**

반복적으로 참조되는 지식을 Skill로 패키징하는 것은 RAG(Retrieval-Augmented
Generation)와 기능적으로 겹친다. 두 접근법의 선택 기준이 글에서
명확하게 제시되지 않는다.

- 비구조화된 대용량 지식 → RAG가 적합
- 절차/워크플로우 → Skills가 적합
- 소규모 구조화 지식 → 어느 쪽이든 가능

이 구분이 실무에서 모호해지면 중복 관리 비용이 발생한다.

**4. 팀 협업에서의 버전 관리 문제**

Skill 파일이 로컬 `.claude/skills/`에 존재한다면, 팀 공유는 Git 저장소
병합에 의존한다. 글은 이 부분을 다루지 않는다. 서로 다른 팀원이
같은 Skill을 다르게 수정했을 때의 충돌 해결, 중앙화된 Skill 관리 방안이
명시되지 않는다.

## 4. 인사이트

### 인사이트 1: Skills는 "AI 운영체제의 패키지 매니저"다

npm, pip, apt처럼 Skills는 AI 기능을 패키지 단위로 설치·관리하는 생태계를
만든다. `skillsmp.com` 같은 커뮤니티 마켓플레이스의 등장이 이를 증명한다.
앞으로 "어떤 Skill을 설치하느냐"가 개인의 AI 생산성을 결정하는 중요한
요소가 될 것이다. Skill 품질이 곧 AI 활용 역량의 척도가 된다.

### 인사이트 2: 암묵지(Tacit Knowledge)의 명시화 도구

Skills의 진짜 가치는 토큰 절약이 아니다.
개인이나 팀이 암묵적으로 알고 있는 업무 방식을 명시적 규칙으로
강제 변환하는 데 있다. Skill을 작성하는 과정 자체가 "우리는 이 작업을
어떻게 하는가?"를 문서화하는 프로세스다. AI 도입이 조직의 지식 관리 수준을
높이는 촉매가 된다.

### 인사이트 3: Context Window 경쟁의 대안 전략

OpenAI, Google은 더 긴 컨텍스트 윈도우로 경쟁한다.
Anthropic은 Skills로 **컨텍스트를 효율적으로 쓰는 방법**으로 경쟁한다.
이는 더 적은 자원으로 더 많은 Skill을 운용할 수 있다는 의미다.
무한한 컨텍스트보다 정교한 컨텍스트 관리가 실용적 우위를 가져올 수 있다.

### 인사이트 4: "지시 정확도"가 새로운 기술 격차

글이 언급하듯, 메타데이터 `description`의 품질이 Skill의 발동 정확도를
결정한다. 이는 결국 "AI에게 무엇을 어떻게 말하는가"의 문제다.
프롬프트 엔지니어링이 대화 단위에서 Skill 설계 단위로 확장된다.
앞으로 좋은 Skill을 설계하는 능력이 AI 시대의 핵심 기술 역량 중 하나가 될 것이다.

### 인사이트 5: MCP-Skills 조합이 만드는 "AI 스택"

MCP(도구 레이어) + Skills(워크플로우 레이어) + CLAUDE.md(전역 규칙 레이어)의
3단 구조는 전통적인 소프트웨어 스택과 유사하다:

```text
CLAUDE.md  →  OS / 시스템 설정
Skills     →  애플리케이션 레이어
MCP        →  드라이버 / API 레이어
```

이 구조가 성숙해지면, AI 시스템 구성을 코드로 관리하는
"AI Infrastructure as Code"의 완성형이 된다.
이 프로젝트의 `skills/` 디렉터리는 이미 그 방향으로 가고 있다.

### 인사이트 6: 조직 내 AI 불평등 해소 가능성

글의 시작점이 중요하다: "잘하는 사람만 혜택을 본다." Skills는 고숙련자의
노하우를 Skill로 패키징해 팀 전체에 배포하는 메커니즘이다. AI 활용 능력의
개인차를 조직 레벨에서 평균화할 수 있다. 이는 단순한 생산성 도구를 넘어
**조직 역량의 민주화 도구**다.

## 5. 결론

Claude Skills 2.0은 AI 활용의 패러다임을 **대화 중심**에서
**워크플로우 중심**으로 전환하는 도구다.

핵심을 한 문장으로: **"반복되는 업무 방식을 한 번 문서화하면,
Claude가 영원히 그 방식으로 일한다."**

이 프로젝트에서도 `skills/` 디렉터리를 적극 활용해
코드 리뷰, 명세 작성, 커밋 메시지 생성 등 반복 워크플로우를
Skill로 체계화하는 것을 고려할 만하다.
