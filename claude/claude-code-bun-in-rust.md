# Claude Code가 Rust로 작성된 Bun을 쓴다

원문: <https://simonwillison.net/2026/Jul/19/claude-code-in-bun-in-rust/>

HN 토론: <https://news.ycombinator.com/item?id=48966569> (466점, 625개 댓글)

## 요약

Claude Code v2.1.181부터 JavaScript 구현 대신 Rust 기반 포트인 Bun으로 실행된다.
Bun을 만든 Jarred Sumner의 발표에 따르면 리눅스에서 시작 시간이 10% 개선됐으며,
변경 자체가 대부분 사용자에게 보이지 않게 이뤄진 것을 이상적인 결과로 표현했다.

Simon Willison이 `strings` 명령으로 Claude 바이너리를 직접 분석해 확인했다.
바이너리 안에 Bun v1.4.0이 번들돼 있었으며, 이는 당시 최신 공개 버전인 v1.3.14보다 앞선 미리보기 버전이다.
바이너리에는 Rust 소스 파일 경로가 563개 포함돼 있어 Rust 구현이 완전히 내장됐음을 확인할 수 있다.
`BUN_OPTIONS`를 통해 내장된 Bun 버전을 직접 조회하는 방법도 소개했다.

이 변경은 원래 Zig로 시작한 Bun이 Rust로 재작성한 것이 이미 전 세계 수백만 장치에서
프로덕션 수준으로 실행되고 있음을 보여준다.

## 분석

### Zig에서 Rust로의 전환이 가져온 공학 관리 이점

mrothroc는 원래 발표에서 Jarred가 밝힌 전환 이유를 분석했다.[^mrothroc]
Zig에서는 메모리 생명주기를 수동으로 추적해야 했고 이것이 긴 버그 목록으로 이어졌다.
Rust는 이 과정을 자동화해 관련 오류 계열 전체를 제거한다.
공학 관리 관점에서 런타임 메모리 버그 클래스를 컴파일 타임 오류로 바꾸는 것은
합리적인 트레이드오프다.

mrothroc가 추가로 지적한 점이 흥미롭다.
컴파일러 오류는 코딩 에이전트를 위한 결정론적 가드레일이 된다.
확률적 출력을 확정적 보증으로 바꾸는 검증 표면이 있을 때 에이전트가 잘 작동하는데,
“컴파일 되도록 만들어라”는 목표가 그런 표면을 제공한다는 것이다.

### 커뮤니케이션 방식을 둘러싼 논란

gabrieledarrigo는 기술적 전환 자체보다 과정에서의 커뮤니케이션 방식을 비판했다.[^gabrieledarrigo]
처음에 “이건 내 브랜치, 과민반응”이라는 반응 이후 100만 줄 이상의 PR을 한 달 안에 머지했다는 점에서
TypeScript 팀이 7.0을 다룬 방식과 대조된다는 지적이다.
오픈소스 프로젝트에서 대규모 언어 전환은 기술적 결정만큼 커뮤니티 신뢰 관리가 중요하다.

## 비평

### “지루한 것이 좋다”는 원칙의 검증

Jarred Sumner가 변경이 보이지 않게 이뤄진 것을 이상적이라고 표현한 것은
인프라 변경의 좋은 원칙을 보여준다.
사용자가 눈치채지 못했다는 것은 호환성이 충분히 유지됐다는 의미다.
그러나 Claude Code를 전 세계에서 실행되는 도구에 배포된 실험적 Bun 버전(미리보기)을
번들로 묶은 것은 다른 위험도 있다.
공개 릴리스 전 버전에 의존하면 Bun 측 변경이 예고 없이 영향을 줄 수 있다.

## 인사이트

### 에이전트 코딩 도구의 런타임이 경쟁 요소가 된다

Claude Code가 Bun을 선택한 것, Bun이 Rust로 재작성된 것은
AI 코딩 도구의 로컬 런타임 성능이 제품 경쟁력이 됐음을 보여준다.
시작 시간 10% 개선이 작아 보이지만, 하루 수백 번 실행하는 개발자에게 누적 차이는 실제로 느껴진다.

더 중요한 시사점은 Anthropic이 Bun을 단순히 쓰는 것이 아니라
내부에서 미리보기 버전을 번들로 사용하고 있다는 점이다.
이것은 도구 개발사와 런타임 개발사 사이의 긴밀한 협력을 시사하며,
성능과 기능을 먼저 적용하는 내부 경로가 있음을 보여준다.
