# Claude Code /goal 기능

<https://code.claude.com/docs/ko/goal>

## 요약

`/goal`은 Claude Code에 새로 추가된 세션 범위의 자율 실행 명령이다.
완료 조건을 텍스트로 입력하면 Claude가 해당 조건이 충족될 때까지 여러 턴에 걸쳐
스스로 계속 작업한다. 매 턴 종료 후 작은 빠른 모델(기본값 Haiku)이 조건 충족 여부를
판단하며, “아니오”이면 다음 턴을 이어가고 “예”이면 목표를 자동으로 지운다.

세션당 하나의 목표만 활성화할 수 있고, `--resume`이나 `--continue`로 세션을 재개하면
이전 목표가 복원된다. 조건은 최대 4,000자까지 작성할 수 있으며, 비대화형 모드에서도
`-p “/goal <조건>”`으로 사용할 수 있다.

## 분석

### 자율 워크플로우의 세 가지 접근법 비교

Claude Code는 이제 세 가지 “계속 실행” 메커니즘을 제공한다.

| 방식           | 다음 턴 시작 시기         | 중지 조건                           |
| -------------- | ------------------------- | ----------------------------------- |
| `/goal`        | 이전 턴 완료 시           | 모델이 조건 충족을 확인할 때        |
| `/loop`        | 시간 간격 경과 시         | 사용자가 중지하거나 완료 판단 시    |
| Stop hook      | 이전 턴 완료 시           | 사용자의 스크립트/프롬프트 결정 시  |

`/goal`은 Stop hook의 세션 범위 래퍼(wrapper)다.
Stop hook은 설정 파일에 저장되어 해당 범위의 모든 세션에 적용되지만,
`/goal`은 현재 세션에만 적용된다. 둘 다 매 턴 후 실행된다는 점에서 동일하다.

### 평가자 분리 설계

`/goal`의 핵심 설계 결정은 작업을 수행하는 모델과 완료 여부를 판단하는 모델을 분리한 것이다.
주 모델(Claude)이 “작업 완료”라고 스스로 판단하는 것이 아니라, 별도의 작은 모델(Haiku)이
대화 기록을 보고 조건 충족 여부를 독립적으로 평가한다.

이 설계는 주 모델의 자기기만(self-deception) 문제를 완화한다.
AI 모델이 반복 작업에서 “이 정도면 됐다”고 조기에 판단하거나 조건을 임의로 재해석하는 경향이
있을 때, 독립적인 평가자가 이를 교정한다. 평가자 모델은 도구를 호출하지 않고 대화 기록만 보므로,
Claude가 실제로 검증 가능한 출력을 대화에 남겨야 한다는 제약이 생긴다.

### 효과적인 조건 작성 원칙

문서는 세 가지 요소를 제안한다.
측정 가능한 최종 상태(테스트 통과, 빌드 성공, 파일 수),
검증 방법(`npm test` 종료 코드, `git status` 결과),
중요한 제약 조건(변경되지 않아야 할 파일 등).

“4,000자까지 가능하다”는 것은 복잡한 수용 기준을 고스란히 조건으로 쓸 수 있다는 의미다.
설계 문서의 수용 기준을 그대로 붙여넣어 구현 완료 기준으로 삼는 워크플로우가 가능해진다.

## 비평

### 토큰 소비 우려

[GeekNews 댓글](https://news.hada.io/topic?id=29428)에서 click이 지적한 대로
“codex는 토큰을 엄청 많이 줘서 괜찮지만 claude code에서 돌리다간 순식간에 사용량 다 써버리는 것
아닌가”라는 우려가 실제 사용 장벽이다. `/goal`은 여러 턴을 자동으로 이어가므로
예상보다 빠르게 토큰 예산을 소진할 수 있다.

문서는 “평가 토큰은 일반적으로 주 턴 소비에 비해 무시할 수 있다”고 안심시키지만,
주 턴 자체가 많아지는 것이 문제다. 조건 설계를 잘못하면 목표에 수렴하지 못하고
무한 루프에 가까운 상황이 발생할 수 있다.

### 평가자의 신뢰성

평가자가 대화 기록만을 보고 조건 충족을 판단하므로, Claude가 실제로 작업을 완료하지 않았지만
완료된 것처럼 대화에 표현하는 경우 평가자가 속을 수 있다. 문서가 “Claude의 자체 출력이
입증할 수 있는 것으로 조건을 작성하라”고 권고하는 이유다. 결과적으로 조건 설계의 질이
기능의 신뢰성을 결정한다.

## 인사이트

### 에이전트 루프의 민주화

`/goal`은 복잡한 에이전트 파이프라인 없이 “조건 기반 자율 실행”을 CLI에서 직접 사용할 수
있게 한다. 기존에 이런 워크플로우를 만들려면 LangChain 같은 프레임워크에서 루프를 작성하고,
완료 조건 함수를 정의하고, 오케스트레이션 코드를 관리해야 했다.

`/goal`은 이 복잡성을 자연어 조건 하나로 압축한다. 이것이 강력한 이유는
에이전트 프로그래밍에 익숙하지 않은 개발자도 “조건 기반 자율 실행”의 이점을 즉시 활용할 수
있게 되기 때문이다. 에이전트 기술의 문턱이 낮아질수록 활용 사례가 폭발적으로 증가한다.

장기적으로 이 방향은 “프롬프트 엔지니어링”을 “조건 엔지니어링”으로 진화시킬 것이다.
어떤 프롬프트를 작성하느냐보다, 어떤 완료 조건을 정의하느냐가 AI 작업 품질을 결정하게 된다.
이것은 소프트웨어 엔지니어링의 테스트 주도 개발(TDD)과 구조적으로 유사하다. TDD에서
테스트를 먼저 작성하듯이, AI 작업에서는 완료 조건을 먼저 정의한다.

### 분리된 평가자 패턴의 의미

작업 모델과 평가 모델을 분리하는 이 패턴은 AI 시스템 설계의 중요한 원칙을 담고 있다.
자기 평가(self-evaluation)에 의존하지 않고 독립된 판단 주체를 두는 것이다.

이 패턴은 인간 조직의 권력 분립과 유사하다. 행정부가 자신의 행위를 스스로 적법하다고 판단하는
체계는 신뢰를 받지 못한다. AI 에이전트가 스스로 “임무 완료”를 선언하는 것도 마찬가지다.
독립적인 평가자의 존재가 시스템 전체의 신뢰성을 높인다.

실용적 함의는 명확하다. 중요한 AI 자율 워크플로우를 설계할 때는 “작업 모델”과 “검증 모델”을
분리해야 한다. 같은 모델이 작업하고 검증하는 구조는 편향을 내재화한다.
Claude Code의 `/goal` 설계는 이 원칙의 프로덕션 구현 예시로서 참고 가치가 있다.

### 비대화형 모드와 CI 파이프라인 통합

`claude -p “/goal <조건>”`으로 비대화형 실행이 가능하다는 것은 CI/CD 파이프라인 통합을
열어준다. “PR의 모든 테스트가 통과하고 린트가 깨끗할 때까지”라는 조건으로 자동화된
코드 수정 파이프라인을 만들 수 있다.

이는 기존 CI 철학과 충돌하는 지점도 있다. CI는 인간이 코드를 작성하고 파이프라인이
검증하는 구조였다. `/goal`로 AI가 코드를 수정하고 조건을 만족시키는 구조가 되면,
파이프라인이 코드 변경의 수동적 검증자에서 능동적 자동 수정 트리거로 역할이 바뀐다.
코드 리뷰 프로세스, 변경 이력 관리, 책임 소재 등 기존 엔지니어링 프랙티스를 재검토해야 할
필요성이 생긴다.
