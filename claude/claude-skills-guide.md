# Claude Skills 제작 완전 가이드

> Anthropic이 공개한 33페이지 분량의 Skills 제작 가이드
> 요약 및 인사이트.

출처:
<https://x.com/Meer_AIIT/status/2023763364502008002>

## Chapter 1: 기본 구조

Skill은 Claude에게 특정 작업 방법을 가르치는 폴더다.
매번 워크플로를 설명하는 대신, 한 번 패키징하면
Claude가 학습하고 기억하고 사용한다.

### 폴더 구성

```text
your-skill-name/
├── SKILL.md           ← 필수
├── scripts/           ← 선택 (Python, Bash 등)
├── references/        ← 선택 (문서, API 가이드)
└── assets/            ← 선택 (템플릿, 이미지)
```

### 3단계 점진적 로딩

| 단계    | 로드 시점            | 내용                 |
|---------|----------------------|----------------------|
| Level 1 | 항상 (시스템 프롬프트) | YAML 프론트매터 (훅) |
| Level 2 | 관련성 판단 시       | SKILL.md 본문        |
| Level 3 | 필요할 때만          | 추가 참조 파일       |

컨텍스트를 최소한으로 유지하다가
실제로 필요할 때만 상세 정보를 로드하는 구조다.

### Skills + MCP

MCP는 "무엇을 할 수 있는가"(도구, 연결, 데이터)이고,
Skill은 "어떻게 해야 하는가"(워크플로, 모범 사례)다.

비유하자면 MCP는 작업장의 도구를 제공하는 것이고,
Skill은 설계도와 작업 절차를 제공하는 것이다.

Skill 없이 MCP만 있으면 사용자가
"이걸로 뭘 해야 하지?"라고 묻게 된다.
Skill이 있으면 Claude가 워크플로를 자동으로 알고,
모범 사례가 내장되어 있어 안내 없이 결과를 낸다.

## Chapter 2: 설계 원칙

### 유스케이스 먼저

Skill을 만들 수 있으니까 만드는 게 아니라,
반복되는 문제가 있으니까 만든다.

설계 시 질문할 것:

1. 최종 목표는 무엇인가?
2. 어떤 도구가 필요한가?
3. 가이드 없이 사람들이 어떤 실수를 하는가?
4. 어떤 도메인 지식이 결과를 좋게 만드는가?

### Skill의 3가지 유형

| 유형              | 특징             | 예시                   |
|-------------------|------------------|------------------------|
| 결과물 생성       | 산출물 만듦      | 마케팅 카피, 코드 모듈 |
| 프로세스 실행     | 순서가 중요      | 주간 콘텐츠 기획       |
| MCP 강화          | 도구 + 워크플로  | DB 쿼리 자동화         |

**결과물 생성**: Claude의 내장 능력을 활용해 문서,
시각화, 코드 등을 만든다.

**프로세스 실행**: 단계별 검증이 필요한 다단계 작업을
자동화한다. 각 단계에서 유효성 검사를 거친다.

**MCP 강화**: 원시 도구 접근을 실제 워크플로로 전환한다.
어떤 테이블을 조인하고, 어떤 메트릭을 계산하고,
결과를 어떻게 포맷할지 알려준다.

### 성공 기준

정량적:

- 10번 중 9번 자동으로 Skill이 로드됨
- Skill 없을 때보다 적은 단계로 작업 완료
- API 호출 재시도 없이 성공

정성적:

- 사용자가 "다음에 뭘 하면 되나요?" 질문 불필요
- 세션 간 일관된 결과
- 신규 사용자도 첫 시도에 성공

## Chapter 3: YAML 프론트매터 작성법

### 필수 구조

```yaml
---
name: email-campaign-builder
description: >-
  Creates email campaigns for product launches.
  Use when user wants to draft launch emails,
  announcement sequences, or promotional
  campaigns.
---
```

`name`: 소문자 + 대시. 폴더명과 일치해야 한다.

`description`: 두 부분이 필수다.
1. 이 Skill이 무엇을 하는가
2. 언제 Claude가 사용해야 하는가

### 좋은 description vs 나쁜 description

좋은 예:

```text
Generates SQL queries for the customer
analytics database. Use when user asks about
customer behavior, purchase patterns, retention
metrics, or churn analysis.
```

구체적 트리거 문구가 포함되어 있고
도메인(고객 분석)이 명확하다.

나쁜 예:

```text
Helps with database stuff.
```

너무 모호해서 Claude가 언제 써야 할지 판단 못 한다.

### 지시문 작성 원칙

모호한 표현 대신 구체적으로 쓴다:

- 나쁨: "데이터 무결성을 보장한다"
- 좋음: "`python scripts/check_data.py`를 실행하고
  세 가지 출력을 확인한다: 누락 필드 없음,
  날짜가 YYYY-MM-DD 형식, 중복 ID 없음"

오류 처리도 포함한다:

"API가 429 에러를 반환하면 60초 대기 후 재시도.
재발하면 대시보드에서 Rate Limit을 확인한다."

상세 참조 문서는 `references/` 폴더로 분리하고
SKILL.md는 핵심 워크플로에 집중한다.

## Chapter 4: 테스트와 반복

### 3가지 테스트 방법

| 방법         | 적합한 상황      | 설명                |
|--------------|------------------|---------------------|
| 수동 테스트  | 개인 사용        | Claude.ai에서 직접  |
| 스크립트     | 반복 검증        | Claude Code로 실행  |
| API 테스트   | 대규모 배포      | 체계적 평가 스위트  |

### 3가지 테스트 영역

**1. 트리거 정확성**

- 명확한 요청: "지원 티켓 분석해줘" → 로드되어야 함
- 변형 요청: "지난달 고객 불만 좀 봐줘" → 로드되어야 함
- 무관한 요청: "날씨 어때?" → 로드되면 안 됨

**2. 작업 수행**

전체 워크플로를 실행하고 출력이 정확한지 확인.
빈 데이터, 잘못된 입력 같은 엣지 케이스 테스트.

**3. 개선 효과**

Skill 없이: 12 메시지, 실패 3회, 사용자가 과정 설명.
Skill 있을 때: 3 메시지, 실패 0회, Claude가 과정을 앎.

### 반복 루프

| 문제                      | 원인         | 해결                          |
|---------------------------|--------------|-------------------------------|
| Skill이 로드 안 됨        | description  | 구체적 키워드 추가            |
|                           | 이 너무 모호 |                               |
| 모든 요청에 Skill 로드됨  | description  | 부정 트리거 추가              |
|                           | 이 너무 광범 | "X용. 일반 Y 질문에는 불가"   |
| 로드되지만 지시 미수행    | 지시문이     | references/로 분리,           |
|                           | 너무 길거나  | SKILL.md 5,000단어 이하 유지  |
|                           | 묻혀 있음    |                               |

## Chapter 5: 실전 패턴

### 패턴 1: 순차 워크플로

단계가 반드시 순서대로 실행되어야 할 때.

예: 환불 처리.
거래 확인 → 자격 검증 → 금액 계산 →
환불 실행 → 확인 발송.

핵심: 단계 간 명시적 의존성, 진행 전 유효성 검사,
실패 시 롤백 지시.

### 패턴 2: 다중 도구 조율

여러 서비스를 연계할 때.

예: 블로그 콘텐츠 발행.
콘텐츠 작성(내장) → 이미지 생성(MCP: Midjourney)
→ 발행(MCP: WordPress) → 소셜 예약(MCP: Buffer)
→ 팀 알림(MCP: Slack).

핵심: 단계 간 명확한 인수인계, 데이터가 도구 간 전달,
진행 전 유효성 검사.

### 패턴 3: 반복 개선

출력이 정제를 통해 나아질 때.

초안 생성 → 품질 검사(검증 스크립트) →
개선 → 재검증 → 기준 충족까지 반복.

핵심: 품질 기준을 미리 정의, 가능하면 프로그래밍적
검증, 반복 중단 시점 명확히.

### 패턴 4: 스마트 도구 선택

같은 작업이지만 맥락에 따라 다른 도구가 필요할 때.

예: 파일 저장 시 협업 문서(Google Docs, Notion)이면
클라우드에 유지하고, 로컬 파일이면 직접 저장.

핵심: 도메인 규칙을 로직에 내장, 실행 전 유효성 검사,
결정 사항을 문서화.

## Chapter 6: 배포와 공유

### 현재 배포 방식

- 사용자가 Skill 폴더를 다운로드
- ZIP으로 압축
- Claude.ai > Settings > Capabilities > Skills에서
  업로드
- 또는 Claude Code의 skills 디렉터리에 배치

팀의 경우 관리자가 워크스페이스 전체에 배포 가능.
(2025년 12월 출시)

### 오픈 표준

Skills는 MCP처럼 오픈 표준이다.
한 번 만들면 표준을 지원하는 모든 AI 도구에서 동작.

### 포지셔닝

기능이 아니라 결과로 말한다:

- 좋음: "2시간 걸리는 수동 SQL 작업을 30초로.
  자연어로 질문하면 정확한 쿼리를 받는다."
- 나쁨: "47개의 사전 작성된 SQL 템플릿이
  YAML 메타데이터와 함께 구조화된 형식으로 포함."

## 인사이트

### 1. "점진적 공개"가 핵심 아키텍처다

3단계 로딩은 단순한 최적화가 아니라 설계 철학이다.
시스템 프롬프트에 항상 올라가는 훅(~100 토큰)과
필요할 때만 로드되는 본문을 분리함으로써,
수십 개의 Skill을 동시에 등록해도
컨텍스트 윈도우를 낭비하지 않는다.
이것이 Skill을 단순한 프롬프트 템플릿이 아닌
"확장 가능한 지식 시스템"으로 만드는 차별점이다.

### 2. description이 Skill의 성패를 결정한다

Skill을 만드는 것보다 description을 잘 쓰는 게
더 어렵고 더 중요하다.
Claude가 Skill을 로드할지 말지를 판단하는
유일한 근거가 description이기 때문이다.
"무엇을 하는가" + "언제 사용하는가"를
사용자가 실제로 쓸 문구로 작성해야 한다.

### 3. MCP + Skill = 완전한 솔루션

MCP만으로는 "도구가 있다"까지이고,
Skill이 있어야 "도구를 올바르게 쓴다"가 된다.
이 조합은 SaaS 제품의 API + 문서 관계와 비슷하다.
MCP 서버를 제공하는 쪽이라면 함께 Skill을
만들어 배포하는 것이 사용자 경험의 결정적 차이다.

### 4. 기존 skills.md와의 관계

이 가이드의 기술적 내용(폴더 구조, 프론트매터,
호출 제어 등)은 공식 문서 기반
[Skills 정리](./skills.md)와 동일하다.
이 가이드가 추가로 제공하는 것은:
**설계 프레임워크**(3가지 유형 분류),
**테스트 방법론**(트리거/수행/효과 3영역),
**실전 패턴**(순차/조율/반복/선택 4패턴),
**트러블슈팅 가이드**다.
즉, "무엇이 있는가"에서 "어떻게 잘 만드는가"로의
전환이다.

### 5. 실행 팁: 하나의 어려운 작업부터

Anthropic이 권장하는 접근법은 흥미롭다.
모든 케이스를 커버하려 하지 말고,
하나의 어려운 작업에서 Claude를 성공시킨 뒤
그 과정에서 효과가 있었던 것을 Skill로 추출하라.
이건 소프트웨어 개발의 "Make it work, then
make it right" 원칙과 같다.
