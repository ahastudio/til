# Fairy

<https://fairy.hada.io>

## 소개

Fairy는 GeekNews를 운영하는 Hada Studio가 만든 오픈소스·인디 프로젝트 후원 플랫폼이다.
서버 비용 때문에 좋은 프로젝트가 중단되는 문제를 해결하겠다는 목표로 만들어졌다.
유료 SaaS로 전환하기 어렵거나, 광고를 붙이기 어색하거나,
전통적인 수익화 모델이 맞지 않는 소규모 프로젝트들을 위한 서비스다.

후원자는 프로젝트에 금액과 메시지를 남길 수 있으며,
개발자는 웹훅(webhook)으로 후원자 정보를 수신해
배지 지급이나 베타 접근 같은 혜택을 자동화할 수 있다.
결제·정산 포함 총 수수료는 5.5%다.

Hada Studio는 GeekNews 자체에 Fairy를 먼저 도입했다.
후원 이메일과 계정 이메일이 일치하면 자동으로 연간 서포터 배지를 발급한다.

## 기능

### 후원 페이지 생성

개발자는 자신의 오픈소스 프로젝트, 웹 툴, 앱, 인디 SaaS를 등록하고
후원 페이지를 공유한다.
후원자는 1회성 결제로 지원하며, 익명 후원도 선택할 수 있다.

### 웹훅 통합

후원이 발생하면 개발자가 등록한 웹훅 엔드포인트로 후원자 정보가 전달된다.
이를 통해:
- 후원자 배지 자동 발급
- 베타 기능 접근 권한 부여
- 감사 메시지 자동 발송

등의 자동화가 가능하다.

### 수수료

결제 처리와 정산을 포함한 총 수수료는 5.5%로,
GitHub Sponsors(없음)보다는 높지만 Open Collective(10%)보다 낮다.

## 분석

### “수익화하기 어려운 프로젝트”라는 명확한 타겟

Fairy가 겨냥하는 것은 GitHub Sponsors나 Patreon이 잘 맞지 않는 틈새다.
GitHub Sponsors는 미국 기반 서비스로 한국 개발자에게 접근성이 낮고,
Patreon은 크리에이터 중심 플랫폼으로 개발 툴 후원에는 맥락이 맞지 않는다.
Fairy는 한국어 기반, 국내 결제, 개발자 커뮤니티 중심이라는 세 가지 맥락을 동시에 충족한다.

### GeekNews 커뮤니티를 기반으로 한 콜드 스타트 해결

새로운 결제 서비스의 가장 큰 장벽은 신뢰다.
Fairy는 GeekNews라는 신뢰 기반 커뮤니티 위에서 출발하기 때문에,
첫 사용자 유치와 신뢰 확보를 동시에 해결한다.
GeekNews 자체를 첫 번째 사용 사례로 삼은 것은
“독이 들었는지 먼저 마셔본다”는 전통적인 제품 신뢰 구축 방식이다.

### 웹훅이 핵심 차별화 지점이다

단순 후원 받기를 넘어 웹훅으로 후원자 혜택을 자동화할 수 있다는 것이
플랫폼의 실질적 가치를 높인다.
배지 발급, 접근 권한 부여, 감사 메시지가 자동화되면
후원자 경험이 개선되고 개발자의 유지 비용이 낮아진다.

## 비평

### 강점: 낮은 수수료와 명확한 포지셔닝

5.5%라는 수수료는 국내 결제 시장에서 경쟁력 있는 수준이다.
GeekNews 댓글에서도 “수수료가 저렴해서 마음에 든다”는 반응이 즉각 나왔다.
오픈소스·인디 프로젝트에 특화된 포지셔닝은 다른 결제 플랫폼과의 직접 경쟁을 피한다.

### 강점: 즉시 검증 가능한 실사용 사례

GeekNews 자체에 도입해 동작을 증명한 것은
“우리가 먼저 쓴다”는 신호로 잠재적 사용자에게 신뢰를 준다.

### 약점: 1회성 후원만 지원

현재는 1회성 결제만 지원하며 월정액 구독 후원이 없다.
지속적인 수익이 필요한 프로젝트에게는 한계다.
GitHub Sponsors는 월정액 티어를 지원해 예측 가능한 수입을 만든다.

### 약점: 소규모 플랫폼의 네트워크 효과 한계

후원 플랫폼은 후원자와 프로젝트 양쪽이 모두 있어야 가치가 생긴다.
GeekNews 커뮤니티 안에서는 강하지만,
그 밖의 개발자 커뮤니티로 확산될 수 있는지는 아직 검증되지 않았다.

## 인사이트

### 오픈소스 지속 가능성 문제를 지역화된 인프라로 해결한다

오픈소스 개발자의 번아웃과 프로젝트 중단 문제는 글로벌 이슈지만,
해결책은 지역화가 필요하다.
GitHub Sponsors, Open Collective, Patreon은 모두 영어권·미국 중심 서비스다.
국내 결제 시스템, 한국어 인터페이스, 국내 개발자 커뮤니티에 뿌리를 둔
Fairy 같은 서비스가 필요한 이유가 있다.

지역화된 후원 인프라는 단순한 복제가 아니다.
세금 처리, 정산 주기, 결제 수단, 커뮤니티 문화가 모두 다르다.
Fairy가 국내 시장에서 자리를 잡는다면,
“지역 개발자 생태계의 지속 가능성을 지역 인프라가 지원한다”는 모델의
첫 번째 사례가 된다.

### “개발자 툴”과 “결제 인프라”의 경계가 흐려지고 있다

Fairy의 웹훅 통합은 후원 서비스가 단순한 결제 도구를 넘어
개발자 인프라의 일부가 될 수 있음을 보여준다.
배지 발급, 접근 권한 관리를 API로 연결하면
후원 시스템이 개발자 애플리케이션의 인증·권한 레이어와 통합된다.

이것은 Stripe가 “결제 API”에서 출발해
청구, 세금, 구독 관리 전체를 커버하는 “수익 인프라”가 된 것과 같은 방향이다.
Fairy도 후원 수취를 시작으로 개발자 이코노미의
더 넓은 인프라 레이어를 담당하는 방향으로 진화할 수 있다.

### GeekNews가 자체 도구를 만드는 것이 커뮤니티에 갖는 의미

GeekNews가 단순히 뉴스를 큐레이션하는 플랫폼에서
개발자 생태계를 위한 도구를 직접 만드는 플랫폼으로 진화하고 있다.
이것은 Hacker News가 Y Combinator의 일부이듯,
커뮤니티 플랫폼이 생태계 지원 도구를 만들어가는 패턴이다.

커뮤니티가 도구를 만들면 그 도구는 커뮤니티의 필요에서 출발한다.
외부 VC 자금 없이 커뮤니티 내부의 문제를 해결하는 도구가 탄생하는 것이다.
Fairy가 성공한다면 GeekNews 커뮤니티의 결속력을 높이고,
그것이 다시 더 많은 도구 개발의 기반이 된다는 선순환이 가능하다.
