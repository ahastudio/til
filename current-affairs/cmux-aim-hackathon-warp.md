# CMUX x AIM 해커톤 참가 후기와 AI 협업 개발 이야기

원문: <https://www.youtube.com/watch?v=bxGQ03rrshc>

## 요약

사비츠(SABITS) 팟캐스트의 너블, 섭비큐라, 아웃사이더, 낙교 네 명이 함께한
이번 에피소드는 섭비큐라와 낙교가 CMUX x AIM 해커톤에 참가한 경험을 중심으로
진행된다. 해커톤은 Warp(구 씨웍스, Manaflow 대표 알렉스 왕이 만든 터미널 도구)와
AI Intelligence라는 AI 보안 회사가 공동 주최했으며, 개발 도구, 비즈니스 애플리케이션,
AI 보안 세 가지 트랙으로 구성됐다. 루마(Luma)를 통한 초대장이 계기가 됐고, 약 300명이
참가한 대규모 행사였다.

섭비큐라와 낙교는 비즈니스 트랙에 지원해 당일 아침 현장에서 아이디어를 논의했다.
첫 번째 아이디어는 Cloudflare Access와 터널(Tunnel)을 조합해 바이브 코딩으로 배포한
애플리케이션에 인증 레이어를 간단하게 씌우는 도구였다. 비개발자도 배포 후 보안을
확보할 수 있도록 프록시 앞단에서 인증을 처리하는 구조인데, 이미 Cloudflare Access가
동일한 개념으로 구현돼 있음을 확인하고 방향을 선회했다.

최종적으로 두 사람이 만든 것은 CLI 교육 환경 플랫폼이다. 웹 브라우저에서 가상
터미널처럼 보이는 인터페이스를 제공하고, 그 안에서 CLI 강의 콘텐츠를 따라 직접 실습할
수 있는 구조다. 낙교가 이전에 만들어 놓은 가짜 터미널(실제 쉘 에뮬레이터) 프로토타입과
섭비큐라가 수년 전 작성한 CLI 입문 블로그 글을 결합했다. Claude Code(클로드 코드)를
웹 터미널 환경과 연동해 사용할 수 있다는 점을 전면에 내세웠으며, 해커톤 현장에서
Claude Opus로 대부분의 코드를 생성했다.

심사는 블라인드 1차 심사를 통해 여덟 팀을 선발한 뒤 공개 발표로 진행됐다. 섭비큐라의
발표가 좋은 평가를 받으면서 비즈니스 트랙 1등이 아닌 종합 대상(300만 원)을 수상했다.
비즈니스 트랙 1등은 일렉트론 앱으로 완성도 높은 제품을 만든 다른 팀이 가져갔다. 이후
대화에서는 교육 환경 도구의 잠재적 시장(오프라인 강의 주최자, 해커톤 운영 기관, AWS 교육
파트너 등)과 향후 발전 방향을 논의했다.

에피소드 후반부에는 Warp 터미널이 브라우저를 직접 제어하는 기능(CDP 활용),
크롬 익스텐션으로 콘솔 에러를 감지해 Claude Code에 자동 수정을 요청하는 팀의 사례,
그리고 GL.iNet의 무선 라우터 신제품 Mudi SE7(여행용 듀얼심 LTE/5G 라우터)에 대한
잡담이 이어진다.

## 분석

### 해커톤의 구조와 맥락

이번 해커톤은 Warp(터미널 + AI 에이전트 도구)와 AI 보안 회사 AIM Intelligence가
공동 주관한 행사로, 국내 개발자 커뮤니티 Attention X 등이 운영을 지원했다. 상금은
종합 대상 300만 원, 트랙별 1등 각 100만 원, 학생 특별상 Azure 크레딧 5,000달러로
구성됐다. 참가비 없이 식사와 간식을 제공하며 약 300명을 모은 것은 홍보 전략과
후원사(Vercel, AI NexPotential 등) 지원이 맞물린 결과다.

세 트랙 구성은 주최사의 이해관계를 반영한다. 개발 도구 트랙은 Warp의 에코시스템
확장을 겨냥하고, AI 보안 트랙은 AIM Intelligence의 도메인이다. 비즈니스 트랙은 두 회사와
직접 연관이 없지만 AI 제품 개발 저변을 넓히는 역할을 한다.

### 제작 과정과 AI 협업

두 참가자는 전형적인 해커톤 준비 없이 당일 아침에 아이디어를 결정했다. 핵심은 기존
보유 자산의 재조합이다. 낙교의 가짜 터미널 프로토타입(xterm 기반 간소화 쉘 에뮬레이터),
섭비큐라의 CLI 입문 블로그 글, 그리고 클로코(Cloco)라는 웹 기반 Claude Code 접속
도구에서 얻은 아이디어가 합쳐졌다.

Claude Opus를 활용해 대부분의 UI와 코드를 생성했다. 디자인은 섭비큐라가 색상 팔레트를
제공하고 “색 빼고 외곽선 살려라”는 구체적 지시를 통해 원하는 스타일을 유도했다. Claude
Artifacts의 디자인이 자신만의 감성을 완전히 제거하기 어렵다는 점, 즉 AI 생성 결과에
작업자의 미적 개입이 남는다는 점도 언급된다.

### 교육 도구 시장에 대한 시각

두 사람은 오프라인 개발 교육에서 환경 설정에 소비되는 시간이 실제 교육 시간을 잠식하는
문제를 직접 경험한 배경을 갖고 있다. 약 7년 전 도커 강의 시절 AWS Lightsail 인스턴스를
스크립트로 대량 생성해 수강생마다 접속 정보를 배분하던 방식이 당시로서는 최선이었다.
Claude Code를 통합한 웹 기반 실습 환경은 그 구조적 문제를 해결할 잠재력이 있다고 본다.
타깃 고객으로는 오프라인 강의 운영자, 해커톤 주최 기관, 기업 내부 교육팀을 언급한다.

### 관련 기술 트렌드

Warp의 브라우저 제어 기능은 Chrome DevTools Protocol(CDP)을 CLI로 래핑해 AI 에이전트가
브라우저와 직접 통신하도록 한다. 에이전트 브라우저 도구들이 스크린샷을 극도로 압축해
모델에 전달할 때 발생하는 해상도 문제도 다뤄지는데, 이는 현재 비주얼 컨텍스트 기반 AI
코딩 도구의 실질적 한계다. 개발 도구 트랙에서 주목받은 크롬 익스텐션 사례(콘솔 에러 →
Claude Code 자동 수정)는 AI를 개발 루프에 인라인으로 통합하는 패턴의 구체적 구현이다.

## 비평

### 강점

해커톤 후기로서 실제 아이디어 결정 과정과 AI 협업의 현실적 모습을 담백하게 전달한다.
“아침에 모여 뭐 하죠?”에서 시작해 종합 대상 수상까지 이어지는 흐름은 AI 도구가 실제로
개발 속도를 얼마나 바꿔 놓았는지를 구체적 사례로 보여준다.

교육 환경 문제에 대한 분석이 수년간의 직접 경험에서 나온 것이라 설득력이 있다. 단순한
아이디어 피칭이 아니라 과거의 고통스러운 경험(쿼터 문제, 와이파이 병목, 코드 서버 설정
등)에서 추출된 문제 정의다.

발표 전략에 대한 언급도 흥미롭다. 심사 기준을 PPT에 직접 반영하고 노트북(클로드 디자인)
사용이 평가에 긍정적 영향을 줬다는 회고는 해커톤에서 기술적 완성도 외에 발표 전략이
차지하는 비중을 솔직하게 드러낸다.

### 약점

팟캐스트 특성상 대화가 산만하게 분기된다. Mudi SE7, 트래블 라우터, Kickstarter VIP 패스,
롤 전 프로게이머 너구리 등 본 주제와 거리가 먼 내용이 상당한 분량을 차지한다. 청취
경험으로는 자연스럽지만 정보 밀도 측면에서는 비효율적이다.

만든 제품의 실제 구현 수준에 대한 설명이 부족하다. “가짜 터미널 에뮬레이터”와 “Claude
Code 연동”이 구체적으로 어떻게 작동하는지, 실습 콘텐츠가 어떤 형태인지 청취자가
재현할 수 있는 수준의 기술적 설명이 없다.

경쟁 팀에 대한 언급(일렉트론 앱, 디자이너 별도 보유, 완성도 높음)이 단편적이라 전체
심사 맥락을 파악하기 어렵다. 종합 대상 수상 이유에 대한 심사위원 피드백이 전혀 소개되지
않는다.

## 인사이트

### AI 시대 해커톤의 실질적 변화

전통적인 해커톤 문화에서 “8시간 안에 만들 수 없는 것은 미리 만들어 온 것”이라는 인식이
있었다. 지금은 그 인식 자체가 흔들리고 있다. 당일 아침에 아이디어를 정하고 Claude
Opus로 코드를 생성하면서도 발표 가능한 수준의 제품이 나오는 것이 실제로 가능해졌다.
이것이 해커톤의 평가 기준을 바꿀 것인지, 아니면 준비된 팀과 즉흥적인 팀의 격차가 더
벌어질 것인지는 아직 불분명하다.

더 흥미로운 질문은 해커톤 심사 기준이 AI 보조 개발을 전제로 재설계되고 있는가 하는
점이다. 완성도보다 아이디어의 참신성, 페인포인트의 명확성, 발표 전달력이 더 중요해질
수 있다. 반대로, AI가 코드를 거의 다 만들어주는 환경에서 완성도 자체의 차별화 요소로서
의미가 줄어든다면, 심사 기준 자체를 어떻게 정의해야 하는가라는 문제가 남는다.

이 에피소드에서 수상팀은 기술적 구현보다 문제 정의와 경험(과거 강의 운영 노하우)에서
우위를 가졌다. AI 도구가 기술 구현 진입 장벽을 낮출수록, 도메인 경험과 문제에 대한
공감 능력이 차별화 요소로 부상한다는 패턴을 보여준다.

### 교육 환경 문제의 구조적 반복

오프라인 개발 교육에서 환경 설정 문제는 약 10년 전 도커 시대에도 존재했고, 지금
Claude Code 시대에도 동일한 형태로 재현된다. 도구만 바뀌었지 구조는 같다. 수강생
모두가 동일한 환경에서 시작할 수 없다는 문제, 강사가 환경 설정 트러블슈팅에 시간을
소비하는 문제, 특정 운영체제(특히 윈도우)에서 발생하는 예외 처리 문제.

이 패턴이 반복된다는 것은 문제가 도구의 복잡성이 아니라 교육 설계의 구조적 한계에
있다는 신호다. 클라우드 기반 실습 환경(코드 서버, 라이트세일, 깃허브 코드스페이스)이
이미 존재하지만 완전히 해결되지 않은 이유는 비용, 관리 부담, 콘텐츠와의 통합 부재다.
이 팀이 만든 프로토타입이 그 공백을 겨냥한다.

주목할 점은 이 아이디어가 AI 시대의 새로운 발상이 아니라, 수년간 반복된 고통에서
나왔다는 것이다. 좋은 도구는 대부분 만든 사람이 문제를 가장 오래, 가장 직접적으로 경험한
경우에서 나온다는 교훈을 다시 확인한다.

### 개발자 도구 인터페이스의 전환점

낙교가 “TUI는 잘못된 방향이고 웹 쪽이 희망이 있다”고 말하는 장면은 단순한 취향 표명이
아니다. 지금 개발자 도구 시장에서 진행 중인 분기를 압축한다. 터미널 에뮬레이터 기반
도구들(Warp, Zellij 등)이 주목받는 동시에, 웹 기반 IDE(GitHub Codespaces, VS Code Web)와
AI 에이전트 인터페이스가 빠르게 영역을 넓히고 있다.

Warp이 브라우저를 오른쪽 패널에 띄우고 왼쪽 터미널의 Claude Code가 CDP로 브라우저를
직접 제어하는 구조는 터미널과 브라우저의 경계를 허무는 시도다. 크롬 익스텐션이 콘솔
에러를 감지해 AI에 자동 수정을 요청하는 방식은 반대 방향, 즉 브라우저에서 개발 도구로의
통합이다. 두 방향이 동시에 진행되고 있으며, 어느 쪽이 표준이 될지는 아직 열려 있다.

이 인터페이스 전환이 교육 도구에 미치는 함의는 직접적이다. 터미널이 무서운 검정 화면인
한 CLI 교육은 항상 첫 번째 허들에서 수강생을 잃는다. 웹 인터페이스 안에 터미널 경험을
내재화한다면 그 허들 자체를 없앨 수 있다. 이 팀의 프로토타입은 바로 그 지점을 건드린다.
