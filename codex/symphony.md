# OpenAI Symphony

<https://github.com/openai/symphony>

프로젝트 작업을 격리된 자율 구현 실행으로 변환하는
작업 관리 플랫폼.

> Engineers do not need to supervise Codex;
> they can manage the work at a higher level.

엔지니어가 코딩 에이전트를 감독하는 대신,
**작업 자체를 관리**하는 수준으로 올라가게 해준다.

---

## 핵심 개념: 하니스 엔지니어링의 확장

[Codex를 이해하는 방법](./how-to-think-about-codex.md)에서
다룬 Model + Harness + Surfaces 프레임워크를 떠올려보자.

Symphony는 이 구조의 위에 놓인 **오케스트레이션 레이어**다.

```
Symphony (오케스트레이션)
  └─ Orchestrator: 이슈 폴링 + 에이전트 디스패치
       └─ AgentRunner: 턴 기반 실행 루프
            └─ Codex AppServer: JSON-RPC로 Codex 제어
                 └─ Model + Harness (GPT-5.3-Codex)
```

개별 Codex 에이전트를 "실행"하는 게 아니라,
이슈 트래커에서 작업을 가져와 여러 에이전트를 동시에
돌리고, 결과를 검증하고, PR을 머지하는 전체 파이프라인을
자동화한다.

---

## 아키텍처 분석

### 동작 흐름

1. **폴링**: Linear 보드에서 활성 이슈를 주기적으로 가져옴
2. **디스패치**: 우선순위·생성일·식별자 기준으로 정렬 후
   사용 가능한 슬롯에 에이전트 배정
3. **실행**: 각 에이전트가 격리된 워크스페이스에서 턴 기반
   작업 수행
4. **증거**: CI 상태, PR 리뷰, 복잡도 분석, 워크쓰루 제공
5. **통합**: 승인 후 PR 머지

### 핵심 모듈 (Elixir 참조 구현)

| 모듈 | 역할 |
|---|---|
| Orchestrator | GenServer. 폴링 루프 + 에이전트 라이프사이클 관리 |
| AgentRunner | 이슈 하나의 멀티턴 실행 루프 |
| Codex.AppServer | Codex와 JSON-RPC 2.0 통신 |
| Workflow | YAML 프론트매터 + Solid 템플릿 파싱 |
| PromptBuilder | 이슈 데이터로 프롬프트 렌더링 |
| Workspace | 이슈별 격리된 디렉토리 생성 |
| Tracker | 어댑터 패턴의 이슈 트래커 추상화 |
| Linear.Client | GraphQL API로 Linear 연동 |
| StatusDashboard | Phoenix LiveView 관측성 대시보드 |

## 설계 결정에서 배울 점

### 런타임을 문제에 맞춰라

Elixir/OTP를 선택한 건 트렌드가 아니라 요구사항 분석이다.
Symphony가 풀어야 할 문제를 나열하면:

- 장시간 실행 프로세스의 생존 보장
- 비정상 종료 시 자동 복구
- 수십 개 동시 프로세스의 경량 관리
- 비동기 이벤트(폴링, 종료, 토큰 갱신)의 자연스러운 처리

이것은 **Erlang이 전화 교환기를 위해 설계된 바로 그
요구사항**이다. Supervisor 트리가 에이전트 프로세스를
감독하고, GenServer의 `handle_info`가 `:tick`, `{:DOWN, ...}`,
`{:codex_worker_update, ...}` 같은 비동기 메시지를
패턴 매칭으로 처리한다. Task.Supervisor가 동시 에이전트를
감독 가능한 태스크로 실행한다.

교훈: "어떤 언어가 좋은가"가 아니라
**"이 문제의 구조가 어떤 런타임과 동형인가"**를 물어야 한다.

### 격리 전략은 트레이드오프를 명시하라

Git worktree 대신 디렉토리 복사. 영리하지 않지만 정확하다.

| | Git Worktree | 디렉토리 복사 |
|---|---|---|
| `.git` 공유 | O (잠금 충돌 위험) | X (완전 독립) |
| 속도 | 빠름 | 느림 |
| 보안 | 심볼릭 링크 탈출 가능 | 경로 순회만 차단하면 됨 |
| 병렬 안전성 | Git 잠금으로 직렬화 | 제약 없음 |

병렬 에이전트 환경에서 "빠르지만 때때로 깨지는 것"보다
"느리지만 절대 안 깨지는 것"이 낫다.

교훈: 격리 전략을 고를 때 **정상 경로의 성능이 아니라
실패 모드의 심각도**로 판단하라.

### 통신 프로토콜은 배포 복잡도로 선택하라

Codex와의 통신에 HTTP/WebSocket이 아닌
**stdio 기반 JSON-RPC 2.0**을 채택했다.

```elixir
Port.open(
  {:spawn_executable, executable},
  [:binary, :exit_status, :stderr_to_stdout,
   args: ["-lc", codex_command],
   cd: workspace, line: @port_line_bytes]
)
```

프로토콜 흐름은 LSP와 유사하다:
`initialize` → `initialized` → 스레드 시작 → 턴 실행 →
완료/실패/취소 이벤트 수신.

HTTP는 유연하지만 포트 관리, TLS, 라우팅이 따라온다.
stdio 파이프는 프로세스를 띄우는 것만으로 통신이 시작된다.
네트워크 레이어가 없으니 레이턴시도, 장애 지점도 줄어든다.

교훈: 같은 머신의 프로세스끼리 통신할 때
**네트워크를 경유할 이유가 있는지** 먼저 따져라.

### 추상화 경계는 콜백 개수로 드러난다

Tracker behaviour는 콜백이 **딱 5개**다:

- `fetch_candidate_issues/0`
- `fetch_issues_by_states/1`
- `fetch_issue_states_by_ids/1`
- `create_comment/2`
- `update_issue_state/2`

Linear, Jira, GitHub Issues 중 어느 트래커든
이 5개만 구현하면 Symphony에 붙는다.
콜백이 5개라는 건, 시스템이 트래커에 대해
**5가지만 알면 된다**는 뜻이다.

교훈: 어댑터 인터페이스가 비대하면 추상화가 잘못된 것이다.
**콜백 수는 추상화의 정밀도를 측정하는 지표**다.

### 멀티턴은 컨텍스트 비용과의 싸움이다

AgentRunner의 멀티턴 루프 설계:

- 첫 번째 턴: `PromptBuilder`로 전체 프롬프트 생성
- 이후 턴: "현재 워크스페이스 상태에서 재개하라"는
  간결한 지시만 전달
- 매 턴 후 이슈 상태 확인 → 활성이면 계속, 종료면 중단
- `max_turns`로 무한 루프 방지

두 번째 턴부터 전체 프롬프트를 다시 보내지 않는다.
워크스페이스 자체가 상태를 담고 있으므로,
프롬프트는 "계속하라"면 충분하다.
이것은 [Codex를 이해하는 방법](./how-to-think-about-codex.md)에서
다룬 **컨텍스트 컴팩션**의 실체다.

교훈: 멀티턴 에이전트에서 **매번 전체 컨텍스트를 재전송하면
토큰 비용이 턴 수에 비례해 폭발**한다.
상태를 워크스페이스에 두고 프롬프트를 최소화하라.

### 비용은 1급 시민으로 다뤄라

Orchestrator는 실시간으로 토큰 델타를 집계한다.
`codex_totals`에 세션별 누적량을,
`codex_rate_limits`에 API 제한 정보를 유지한다.

토큰 사용량 추적이 로깅이 아닌 **GenServer 상태의 일부**라는
점이 중요하다. 부가 기능이 아니라 오케스트레이션 로직과
같은 수준에서 관리된다.

교훈: AI 시스템에서 비용 추적을 나중에 붙이면
이미 늦다. **비용 관측성은 아키텍처 초기부터 1급
시민으로 설계**해야 한다.

## 인사이트

### AI가 빠진 AI 인프라

Symphony 코드베이스를 처음 열면 당혹스럽다.
**AI에 대한 코드가 없다.**
프롬프트를 조합하는 `PromptBuilder`와
JSON-RPC로 Codex를 호출하는 `AppServer`를 빼면,
나머지 전부가 프로세스 감독, 상태 머신, 재시도,
격리, 동시성 제어다.

이것은 우연이 아니라 필연이다.
모델의 지능이 올라갈수록, 모델을 감싸는 시스템의
병목은 "얼마나 똑똑한가"에서
**"얼마나 죽지 않는가"**로 이동한다.
Erlang이 전화 교환기를 위해 풀었던 문제—
프로세스가 죽으면 자동 재시작, 장애가 전파되지 않는
감독 트리, 수만 경량 프로세스의 선점적 스케줄링—가
2026년 AI 에이전트 오케스트레이터의 핵심 요구사항과
**글자 그대로 동일하다.**

여기서 도출되는 원칙이 있다.
기술의 가치는 발명된 시점이 아니라
**문제와 만나는 시점**에 결정된다.
OTP는 30년을 기다려 자신의 두 번째 킬러 앱을 만났다.
"낡은 기술"이란 없다.
아직 문제를 못 만난 기술이 있을 뿐이다.

### 추상화의 사다리: 실행에서 의도로

Symphony가 만든 추상화 계층을 역사적 맥락에 놓으면
패턴이 보인다:

```
어셈블리 → C: 레지스터를 변수로
VM → 컨테이너: 하드웨어를 이미지로
kubectl → Helm: 컨테이너를 선언적 워크로드로
Codex → Symphony: 에이전트를 이슈 카드로
```

매번 같은 일이 일어난다.
**실행의 세부사항이 사라지고 의도만 남는다.**
개발자가 Codex를 직접 쓰면 프롬프트를 쓰고,
출력을 읽고, 오류를 고치고, 커밋을 확인한다.
Symphony를 쓰면 Linear에 이슈를 만든다. 끝.

이 전환의 진짜 의미는 효율이 아니다.
**인지 부하의 대상이 바뀐다는 것이다.**
"이 코드가 맞는가?"에서 "이 이슈 정의가 충분한가?"로.
"에이전트가 뭘 하고 있지?"에서
"이 이슈가 어떤 상태지?"로.

이것은 소프트웨어 개발의 본질을 건드린다.
개발자의 핵심 역량이 **구현력에서 명세력으로**
이동하기 시작했다는 신호다.
코드를 잘 짜는 것보다 작업을 잘 정의하는 것이
더 중요해지는 세계가 온다.

### WORKFLOW.md가 여는 판도라의 상자

`WORKFLOW.md`는 단순해 보인다.
YAML 프론트매터로 설정, 마크다운 본문으로
프롬프트 템플릿, Git으로 버전 관리.
하지만 이 단순함이 열어젖히는 가능성은 거대하다.

CI/CD의 역사를 보자.
Jenkins의 Groovy 스크립트는 DevOps 엔지니어의
전유물이었다. GitHub Actions의 YAML은
**모든 개발자의 것**이 되었다.
선언적 포맷이 진입 장벽을 무너뜨린 것이다.

Symphony는 이 패턴을 에이전트 오케스트레이션에
적용한다. `.md` 파일 하나가 에이전트의 행동을
정의한다. 프로그래밍 없이.
프롬프트 엔지니어링이 곧 파이프라인 엔지니어링이
되는 세계에서, **마크다운이 새로운 인프라 언어**다.

하지만 진짜 함의는 더 깊다.
워크플로가 코드가 아닌 문서라면,
PM이 직접 에이전트 파이프라인을 설계할 수 있다.
QA가 검증 워크플로를 자기 손으로 만들 수 있다.
**소프트웨어 개발의 권한 구조 자체가 재편**된다.
코드를 쓸 수 있는 사람만 시스템을 바꿀 수 있던 시대가
끝나기 시작한 것이다.

### 공유 상태 제거라는 설계 철학

Symphony의 기술적 선택들을 나열하면
하나의 강박이 드러난다:

- Git worktree(`.git` 공유) 대신 디렉토리 복사
- HTTP(포트·라우팅 공유) 대신 stdio 파이프
- DB 영속화(스키마 공유) 대신 파일시스템 + 재질의
- 에이전트 간 메시지 패싱 대신 완전한 격리

**공유 상태를 향한 체계적 거부.**

이것은 마이크로서비스에서 배운 교훈의 극단적 적용이다.
마이크로서비스는 "서비스 간 DB를 공유하지 마라"고 했다.
Symphony는 "에이전트 간 아무것도 공유하지 마라"고 한다.
파일시스템도, 네트워크도, Git 객체 저장소도.

왜 이렇게까지 하는가?
에이전트는 사람과 다르기 때문이다.
사람은 공유 자원에서 충돌이 나면 대화로 해결한다.
에이전트는 `.git/index.lock`을 만나면 **멈춘다.**
비결정적 실행 환경에서 공유 상태는 시한폭탄이다.
터질 확률이 에이전트 수의 제곱에 비례해 증가한다.

Symphony가 증명한 원칙:
**에이전트 시스템에서 성능 최적화의 첫 번째 규칙은
공유를 제거하는 것이다.**
느린 복사가 빠른 잠금보다 항상 낫다.

### Tracker behaviour의 5개 콜백이 말하는 것

Tracker behaviour의 콜백은 **딱 5개**다:

- `fetch_candidate_issues/0`
- `fetch_issues_by_states/1`
- `fetch_issue_states_by_ids/1`
- `create_comment/2`
- `update_issue_state/2`

Linear, Jira, GitHub Issues, Notion—
어떤 트래커든 이 5개만 구현하면 Symphony에 붙는다.

이것이 아름다운 이유는 숫자가 아니다.
이 5개가 **에이전트가 이슈 트래커에 대해
알아야 하는 전부**라는 사실이다.
후보 가져오기, 상태 조회, 코멘트 달기, 상태 변경.
이것 외에 에이전트가 트래커에게 요청할 것이 없다.

인터페이스 설계의 본질이 여기 있다.
좋은 인터페이스는 "이것도 되고 저것도 되는" 게 아니라
**"이것만 하면 되는"** 것이다.
콜백이 20개인 어댑터는 추상화가 실패한 것이다.
시스템이 외부에 대해 20가지를 알아야 한다는 뜻이니까.
콜백 수는 추상화의 정밀도를 측정하는 리트머스 시험지다.

### 에이전트 시대의 미해결 문제들

Symphony는 "에이전트를 어떻게 실행할까"에 답했다.
하지만 더 어려운 질문들이 기다리고 있다.

**협업의 부재.**
현재는 이슈 하나에 에이전트 하나.
완전한 격리. 이것은 강점이자 한계다.
프론트엔드 에이전트와 백엔드 에이전트가
하나의 API 계약을 두고 협업해야 한다면?
격리를 유지하면서 협업하는 것은
분산 시스템의 가장 오래된 난제다.
Symphony의 다음 버전이 이것을 어떻게 푸는지가
아키텍처의 성숙도를 결정할 것이다.

**검증의 천장.**
CI 통과가 "좋은 구현"을 의미하는가?
테스트가 없는 코드, 테스트로 검증할 수 없는
설계 결정, 성능 특성, 보안 함의—
이것들을 누가 판단하는가?
에이전트가 만든 코드를 에이전트가 검증하는 순간,
**검증의 독립성이 사라진다.**
이것은 기술 문제가 아니라 인식론적 문제다.

**피드백 루프의 역설.**
PR 리뷰 코멘트를 에이전트가 받아 수정하고,
다시 리뷰를 받는 루프가 자동화되면,
리뷰어는 무엇을 하는가?
에이전트의 코드를 읽는 것인가,
에이전트의 판단을 감사하는 것인가?
리뷰의 목적이 "코드 품질 확인"에서
**"에이전트 행동 감사"**로 바뀐다.
이것은 코드 리뷰라는 관행의 근본적 재정의다.

Symphony는 v0이다.
하지만 이 시스템이 제기하는 질문들은 v1이 아니라
**소프트웨어 공학의 다음 10년**에 속한다.
