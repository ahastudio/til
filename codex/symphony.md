# OpenAI Symphony

<https://github.com/openai/symphony>

프로젝트 작업을 격리된 자율 구현 실행으로 변환하는
작업 관리 플랫폼.

> Engineers do not need to supervise Codex;
> they can manage the work at a higher level.

엔지니어가 코딩 에이전트를 감독하는 대신,
**작업 자체를 관리**하는 수준으로 올라가게 해준다.

---

## 핵심 개념: 하니스 엔지니어링의 확장

[Codex를 이해하는 방법](./how-to-think-about-codex.md)에서
다룬 Model + Harness + Surfaces 프레임워크를 떠올려보자.

Symphony는 이 구조의 위에 놓인 **오케스트레이션 레이어**다.

```
Symphony (오케스트레이션)
  └─ Orchestrator: 이슈 폴링 + 에이전트 디스패치
       └─ AgentRunner: 턴 기반 실행 루프
            └─ Codex AppServer: JSON-RPC로 Codex 제어
                 └─ Model + Harness (GPT-5.3-Codex)
```

개별 Codex 에이전트를 "실행"하는 게 아니라,
이슈 트래커에서 작업을 가져와 여러 에이전트를 동시에
돌리고, 결과를 검증하고, PR을 머지하는 전체 파이프라인을
자동화한다.

---

## 아키텍처 분석

### 동작 흐름

1. **폴링**: Linear 보드에서 활성 이슈를 주기적으로 가져옴
2. **디스패치**: 우선순위·생성일·식별자 기준으로 정렬 후
   사용 가능한 슬롯에 에이전트 배정
3. **실행**: 각 에이전트가 격리된 워크스페이스에서 턴 기반
   작업 수행
4. **증거**: CI 상태, PR 리뷰, 복잡도 분석, 워크쓰루 제공
5. **통합**: 승인 후 PR 머지

### 핵심 모듈 (Elixir 참조 구현)

| 모듈 | 역할 |
|---|---|
| Orchestrator | GenServer. 폴링 루프 + 에이전트 라이프사이클 관리 |
| AgentRunner | 이슈 하나의 멀티턴 실행 루프 |
| Codex.AppServer | Codex와 JSON-RPC 2.0 통신 |
| Workflow | YAML 프론트매터 + Solid 템플릿 파싱 |
| PromptBuilder | 이슈 데이터로 프롬프트 렌더링 |
| Workspace | 이슈별 격리된 디렉토리 생성 |
| Tracker | 어댑터 패턴의 이슈 트래커 추상화 |
| Linear.Client | GraphQL API로 Linear 연동 |
| StatusDashboard | Phoenix LiveView 관측성 대시보드 |

## 설계 결정에서 배울 점

### 런타임을 문제에 맞춰라

Elixir/OTP를 선택한 건 트렌드가 아니라 요구사항 분석이다.
Symphony가 풀어야 할 문제를 나열하면:

- 장시간 실행 프로세스의 생존 보장
- 비정상 종료 시 자동 복구
- 수십 개 동시 프로세스의 경량 관리
- 비동기 이벤트(폴링, 종료, 토큰 갱신)의 자연스러운 처리

이것은 **Erlang이 전화 교환기를 위해 설계된 바로 그
요구사항**이다. Supervisor 트리가 에이전트 프로세스를
감독하고, GenServer의 `handle_info`가 `:tick`, `{:DOWN, ...}`,
`{:codex_worker_update, ...}` 같은 비동기 메시지를
패턴 매칭으로 처리한다. Task.Supervisor가 동시 에이전트를
감독 가능한 태스크로 실행한다.

교훈: "어떤 언어가 좋은가"가 아니라
**"이 문제의 구조가 어떤 런타임과 동형인가"**를 물어야 한다.

### 격리 전략은 트레이드오프를 명시하라

Git worktree 대신 디렉토리 복사. 영리하지 않지만 정확하다.

| | Git Worktree | 디렉토리 복사 |
|---|---|---|
| `.git` 공유 | O (잠금 충돌 위험) | X (완전 독립) |
| 속도 | 빠름 | 느림 |
| 보안 | 심볼릭 링크 탈출 가능 | 경로 순회만 차단하면 됨 |
| 병렬 안전성 | Git 잠금으로 직렬화 | 제약 없음 |

병렬 에이전트 환경에서 "빠르지만 때때로 깨지는 것"보다
"느리지만 절대 안 깨지는 것"이 낫다.

교훈: 격리 전략을 고를 때 **정상 경로의 성능이 아니라
실패 모드의 심각도**로 판단하라.

### 통신 프로토콜은 배포 복잡도로 선택하라

Codex와의 통신에 HTTP/WebSocket이 아닌
**stdio 기반 JSON-RPC 2.0**을 채택했다.

```elixir
Port.open(
  {:spawn_executable, executable},
  [:binary, :exit_status, :stderr_to_stdout,
   args: ["-lc", codex_command],
   cd: workspace, line: @port_line_bytes]
)
```

프로토콜 흐름은 LSP와 유사하다:
`initialize` → `initialized` → 스레드 시작 → 턴 실행 →
완료/실패/취소 이벤트 수신.

HTTP는 유연하지만 포트 관리, TLS, 라우팅이 따라온다.
stdio 파이프는 프로세스를 띄우는 것만으로 통신이 시작된다.
네트워크 레이어가 없으니 레이턴시도, 장애 지점도 줄어든다.

교훈: 같은 머신의 프로세스끼리 통신할 때
**네트워크를 경유할 이유가 있는지** 먼저 따져라.

### 추상화 경계는 콜백 개수로 드러난다

Tracker behaviour는 콜백이 **딱 5개**다:

- `fetch_candidate_issues/0`
- `fetch_issues_by_states/1`
- `fetch_issue_states_by_ids/1`
- `create_comment/2`
- `update_issue_state/2`

Linear, Jira, GitHub Issues 중 어느 트래커든
이 5개만 구현하면 Symphony에 붙는다.
콜백이 5개라는 건, 시스템이 트래커에 대해
**5가지만 알면 된다**는 뜻이다.

교훈: 어댑터 인터페이스가 비대하면 추상화가 잘못된 것이다.
**콜백 수는 추상화의 정밀도를 측정하는 지표**다.

### 멀티턴은 컨텍스트 비용과의 싸움이다

AgentRunner의 멀티턴 루프 설계:

- 첫 번째 턴: `PromptBuilder`로 전체 프롬프트 생성
- 이후 턴: "현재 워크스페이스 상태에서 재개하라"는
  간결한 지시만 전달
- 매 턴 후 이슈 상태 확인 → 활성이면 계속, 종료면 중단
- `max_turns`로 무한 루프 방지

두 번째 턴부터 전체 프롬프트를 다시 보내지 않는다.
워크스페이스 자체가 상태를 담고 있으므로,
프롬프트는 "계속하라"면 충분하다.
이것은 [Codex를 이해하는 방법](./how-to-think-about-codex.md)에서
다룬 **컨텍스트 컴팩션**의 실체다.

교훈: 멀티턴 에이전트에서 **매번 전체 컨텍스트를 재전송하면
토큰 비용이 턴 수에 비례해 폭발**한다.
상태를 워크스페이스에 두고 프롬프트를 최소화하라.

### 비용은 1급 시민으로 다뤄라

Orchestrator는 실시간으로 토큰 델타를 집계한다.
`codex_totals`에 세션별 누적량을,
`codex_rate_limits`에 API 제한 정보를 유지한다.

토큰 사용량 추적이 로깅이 아닌 **GenServer 상태의 일부**라는
점이 중요하다. 부가 기능이 아니라 오케스트레이션 로직과
같은 수준에서 관리된다.

교훈: AI 시스템에서 비용 추적을 나중에 붙이면
이미 늦다. **비용 관측성은 아키텍처 초기부터 1급
시민으로 설계**해야 한다.

## 인사이트

### AI 에이전트 문제는 AI 문제가 아니다

Symphony 코드베이스에서 가장 놀라운 점은
**AI에 대한 코드가 거의 없다**는 것이다.
코드의 대부분은 프로세스 감독, 상태 관리, 재시도 전략,
워크스페이스 격리, 동시성 제어를 다룬다.
이것들은 1990년대 Erlang/OTP가 전화 교환기를 위해
풀었던 문제들과 본질적으로 같다.

AI 에이전트를 "똑똑하게" 만드는 건 모델의 몫이다.
에이전트를 "쓸 수 있게" 만드는 건 분산 시스템 엔지니어링의
몫이다. Symphony는 후자에 올인했다.
Elixir/OTP 선택이 이를 정확히 증명한다.
최신 AI 제품의 핵심 인프라가 30년 된 통신 런타임 위에
서 있다는 사실이 역설적이면서도 합리적이다.

### 감독에서 관리로: 추상화 수준의 도약

개발자가 Codex를 직접 쓰면 "에이전트를 감독"한다.
Symphony를 쓰면 "이슈를 관리"한다.
이 차이는 단순한 편의가 아니라 **추상화 수준의 전환**이다.

Docker가 프로세스를 컨테이너로 추상화했고,
Kubernetes가 컨테이너를 선언적 워크로드로 추상화했듯이,
Symphony는 코딩 에이전트를 이슈 트래커의 카드 한 장으로
추상화한다. 엔지니어는 "이 에이전트가 뭘 하고 있지?"가
아니라 "이 이슈가 어떤 상태지?"를 묻게 된다.

이 패턴이 일반화되면 개발 팀의 역할이 바뀐다.
코드를 짜는 사람에서, 작업을 정의하고
에이전트의 산출물을 검증하는 사람으로.

### WORKFLOW.md: 인프라의 민주화

파이프라인을 마크다운으로 정의한다는 건
**프로그래밍 없이 에이전트 오케스트레이션을 구성**할 수
있다는 뜻이다. YAML 프론트매터로 설정,
본문으로 프롬프트 템플릿. Git으로 버전 관리.
PR로 리뷰. 브랜치로 실험.

이것이 시사하는 바는 크다.
CI/CD 파이프라인을 `.yml`로 정의하면서 DevOps가
민주화되었듯, 에이전트 파이프라인을 `.md`로 정의하면
**에이전트 운영이 민주화**된다.
프롬프트 엔지니어링이 곧 파이프라인 엔지니어링이 되는
세계에서, 마크다운은 새로운 인프라 언어다.

### 격리의 교훈: 단순함이 이긴다

Git worktree가 아닌 디렉토리 복사.
HTTP가 아닌 stdio 파이프.
DB 영속화가 아닌 파일시스템 + 재질의.

Symphony의 설계 결정들에는 일관된 패턴이 있다.
**더 단순하고 더 격리된 쪽을 택한다.**
Git worktree는 `.git`을 공유하므로 잠금 충돌 위험이 있다.
디렉토리 복사는 느리지만 완전히 독립적이다.
HTTP는 유연하지만 네트워크 레이어가 필요하다.
stdio 파이프는 제약이 있지만 배포가 단순하다.

에이전트가 10개, 20개 병렬로 돌아가는 환경에서
"공유 상태"는 곧 장애 지점이다.
Symphony는 공유를 최소화하는 방향으로 모든 결정을 내렸다.
이것은 마이크로서비스 아키텍처에서 배운 교훈과 같다:
**결합을 끊는 것이 성능보다 중요하다.**

### 에이전트 오케스트레이션의 다음 질문

Symphony가 보여주는 건 "어떻게 에이전트를 실행할까"에
대한 답이지만, 동시에 더 큰 질문을 던진다:

- **에이전트 간 협업**: 현재는 이슈 하나에 에이전트 하나.
  여러 에이전트가 하나의 피처를 나눠 구현하려면?
- **검증의 자동화**: CI 통과가 "잘 된 구현"을 보장하는가?
  에이전트 산출물의 품질을 누가, 어떻게 판단하는가?
- **피드백 루프**: PR 리뷰 코멘트를 에이전트가 받아서
  수정하는 루프가 자동화되면, 리뷰어의 역할은 무엇인가?

Symphony는 v0이다. 하지만 이 질문들의 방향을 제시한다.
에이전트 시대의 소프트웨어 개발은 결국
**작업 정의 → 자율 실행 → 검증 → 통합**의 루프이고,
이 루프를 얼마나 빠르고 안정적으로 돌릴 수 있느냐가
팀의 생산성을 결정하게 될 것이다.
