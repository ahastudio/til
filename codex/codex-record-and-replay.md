# OpenAI Codex의 Record & Replay: 워크플로를 한 번 시연하면 자동화로 전환된다

<https://developers.openai.com/codex/record-and-replay>

## 요약

OpenAI Codex의 Record & Replay 기능은 사용자가 워크플로를 한 번 직접 시연하면
그것을 재사용 가능한 스킬로 변환해 주는 기능이다.
“Codex에게 워크플로를 한 번 보여주고 재사용 가능한 스킬로 전환하라”는 설명이 핵심이다.
반복적인 작업에서 성공 기준이 명확한 경우에 가장 잘 작동한다.
경비 청구, 주차 예약, 이슈 생성, 영상 업로드, 보고서 다운로드 등의 사례가 제시된다.

기록(Recording) 과정은 다음 순서로 진행된다.
플러그인을 열고 “스킬 기록”을 선택한 뒤 맥락을 제공하고, 기록 권한을 승인한다.
이후 Mac에서 워크플로를 직접 수행하면 Codex가 행동과 화면 내용을 관찰한다.
기록이 끝나면 Codex가 캡처된 워크플로를 분석해 스킬 문서를 작성한다.
이 문서에는 사용 조건, 필요한 입력값, 절차적 단계, 검증 방법이 포함된다.

재생(Replay) 시에는 사용자가 다른 입력값으로 스킬을 적용해 달라고 요청하면
Computer Use와 브라우저 액션 같은 도구를 사용해 실행된다.
최적의 기록을 위해서는 실제 테스트 입력값을 사용하고, 민감한 데이터를 피하고,
시작 전에 가변 요소에 대한 맥락을 제공하는 것이 권장된다.
기능은 현재 macOS에서만 제공되며, EEA, 영국, 스위스 지역은 제외된다.

팀 전체에 안정적인 패키지를 배포하거나 복수의 스킬을 통합 기능과 함께 묶으려면
별도의 플러그인을 빌드하는 것이 권장된다.

## 분석

### Record & Replay는 프로그래밍의 진입 장벽을 시연으로 대체한다

전통적으로 자동화 스크립트를 만들려면 프로그래밍 지식이 필요했다.
Selenium, Playwright, Automator 같은 도구들은 모두 코드 또는 GUI 기반 워크플로 설계를 요구한다.
Record & Replay는 이 전제를 뒤집는다.
자동화의 명세(specification)를 코드가 아닌 인간의 행동 시연으로 제공하는 방식이다.

이것은 프로그래밍 교육에서 “예제를 통한 학습(learning by example)”과 같은 논리다.
사람이 시스템에게 “이렇게 하면 돼”라고 보여주면, 시스템이 그 의도를 추론해 재현한다.
기존 자동화 도구들이 “어떻게 할지를 기술하라(describe how)”를 요구했다면,
Record & Replay는 “보여주면 내가 기술한다(show me, I'll describe)”로 방향을 바꾼다.

### 스킬 문서 생성은 인간 검토를 위한 설계다

기록 후 Codex가 자동으로 스킬 문서를 생성한다는 점은 중요하다.
이 문서는 사용 조건, 입력값, 절차, 검증 방법을 담고 있어
사용자가 자동화 내용을 검토하고 수정할 수 있게 한다.
단순히 기록을 재생하는 것이 아니라, 인간이 읽을 수 있는 명세로 변환한다.

이 설계 선택은 자동화의 투명성과 통제 가능성을 높이기 위한 것이다.
“나중에 스킬을 다듬어 이름 규칙이나 필드 기본값 같은 선호사항을 반영하라”는 권장 사항은
자동 생성된 스킬이 그대로 배포용이 아닌 출발점임을 전제한다.
인간 검토 단계를 내장함으로써, 악의적으로 기록된 워크플로나 오류가 있는 시연이
무비판적으로 반복 실행되는 위험을 줄인다.

### 이 기능은 Computer Use 능력과 결합해야만 완성된다

Record & Replay는 단독으로 존재하는 기능이 아니다.
재생 단계에서 Codex는 Computer Use와 브라우저 액션을 통해 실제로 화면을 조작한다.
즉, 이 기능의 실제 가치는 Codex가 화면을 보고 클릭하고 입력할 수 있다는 전제에 의존한다.

이 점에서 Record & Replay는 기술적 혁신이라기보다는
Computer Use라는 기반 능력 위에 구축된 사용자 경험 레이어다.
자동화를 위한 API가 없는 서비스, 레거시 데스크톱 소프트웨어,
또는 웹 스크래핑이 어려운 인트라넷 시스템에 특히 유용한 이유가 여기 있다.
코드가 아닌 화면 수준에서 자동화가 이루어지기 때문이다.

## 비평

### “한 번 보여주면 된다”는 약속은 시연의 품질에 숨겨진 전제를 감춘다

“워크플로를 한 번 시연하면 자동화로 전환된다”는 약속은 단순해 보이지만,
실제로는 시연의 품질에 크게 의존한다.
문서가 “실제 테스트 입력값을 사용하고 민감한 데이터를 피하라”고 권장하는 것은
잘못된 입력값으로 기록하면 스킬의 신뢰성이 낮아진다는 뜻이기도 하다.

더 근본적으로, 워크플로가 예외 상황이나 오류 처리를 포함한다면
이를 시연으로 표현하기가 어렵다.
“실패했을 때 어떻게 해야 하는지”를 시연으로 전달하려면
실패 상황을 의도적으로 만들어야 하는데, 이것은 직관적이지 않다.
시연 기반 자동화는 정상 경로(happy path)에 최적화되어 있다.

### macOS 전용 출시는 기업 도입에 구조적 제약을 만든다

현재 macOS에서만 작동한다는 제약은 단순한 플랫폼 한계가 아니다.
기업 환경에서 반복적인 업무 자동화 수요는 Windows가 더 높은 경우가 많다.
경비 청구, 보고서 다운로드, 이슈 생성 같은 사례가 Windows 기반 기업 시스템에서 주로 발생한다.

EEA, 영국, 스위스 제외 역시 규제 이유로 보이지만,
Computer Use 기능이 스크린 캡처와 행동 기록을 포함하는 만큼
GDPR 등 개인정보 규제와의 긴장이 해소되지 않았음을 시사한다.
이 지역 제외는 지리적 제한이 아니라 미해결된 법적 불확실성의 표현이다.

### 팀 공유를 위한 경로가 명확하지 않다

문서는 팀 전체에 배포하려면 “별도의 플러그인을 빌드하라”고 권장한다.
하지만 플러그인 빌드는 Record & Replay가 낮추려는 진입 장벽과 정반대의 요구다.
비개발자가 스킬을 기록하고 팀원들과 공유하려면 결국 개발자의 개입이 필요하다.

이것은 Record & Replay의 핵심 사용 사례와 충돌한다.
경비 담당자나 마케터가 자신의 업무 워크플로를 기록해 팀에서 공유하는 시나리오가
가장 가치 있는 사용 사례일 텐데, 이를 위한 경로가 막혀 있다.
기능의 완성도는 개인 사용자 수준에 머물러 있고, 팀 수준 협업은 미완이다.

## 인사이트

### 시연 기반 프로그래밍은 자동화의 민주화와 책임 분산을 동시에 가져온다

Record & Replay가 성숙하면 자동화를 만드는 주체가 바뀐다.
지금은 IT 부서나 개발자가 자동화 스크립트를 작성한다.
앞으로는 업무 담당자가 직접 자동화를 생성한다.
이것은 자동화 생산성의 민주화이면서 동시에 자동화 품질 책임의 분산이다.

개발자가 만든 자동화는 코드 리뷰, 테스트, 버전 관리를 거친다.
시연으로 만들어진 자동화는 그런 검증 체계 없이 반복 실행될 수 있다.
기업 환경에서 이 자동화들이 누적되면 “보이지 않는 자동화 부채”가 생긴다.
누가 만들었는지, 어떤 가정을 전제하는지, 언제 실패하는지 아무도 모르는 자동화가 쌓이는 것이다.

### 이 기능은 자동화의 단위를 태스크에서 워크플로로 이동시킨다

기존 자동화 도구는 태스크 단위로 작동한다.
“이 API를 호출하라”, “이 파일을 변환하라”처럼 단일 작업을 자동화한다.
Record & Replay는 여러 단계를 포함한 전체 워크플로를 하나의 단위로 자동화한다.
경비 청구 예시는 여러 화면을 탐색하고, 데이터를 입력하고, 제출하는 복합 행동이다.

이 접근법의 두 번째 효과는 자동화의 경계가 인간의 업무 흐름과 일치한다는 것이다.
API 기반 자동화는 시스템의 경계로 자동화를 분해하지만,
워크플로 기반 자동화는 인간이 실제로 인식하는 업무 단위로 자동화를 구성한다.
이것은 자동화의 유지보수 관점에서 더 직관적일 수 있다.

### Computer Use 기반 자동화는 UI 변경에 본질적으로 취약하다

시연 기반 자동화가 화면 요소에 의존하는 이상,
대상 시스템의 UI가 변경되면 스킬이 깨진다.
버튼 위치가 이동하거나, 레이블이 바뀌거나, 로그인 흐름이 변경되면
기록된 스킬은 실패한다.

API 기반 자동화는 UI와 독립적이어서 상대적으로 안정적이다.
Record & Replay는 API가 없는 시스템을 위한 해법이지만,
그 해법은 UI 안정성에 의존하는 구조적 취약점을 내포한다.
기업 환경에서 소프트웨어 업데이트는 정기적으로 발생하고,
그때마다 기록된 스킬들을 재검증하거나 재기록해야 한다.
이 유지보수 비용은 기능 소개에서 드러나지 않지만, 실제 도입 시 중요한 고려사항이다.
