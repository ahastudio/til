# loopndroll - Codex 세션을 무한 연장하는 macOS 메뉴바 앱

> 원문: <https://github.com/lnikell/loopndroll>

## 요약

loopndroll은 OpenAI Codex 에이전트 세션을 자동으로 재시작해 무한 루프로 실행할 수
있도록 돕는 macOS 메뉴바 애플리케이션이다. Swift로 작성되었으며, Codex의 Stop 훅
메커니즘에 글로벌 훅을 설치해 세션이 끝날 때마다 자동으로 재개한다.

핵심 동작 방식은 `~/.codex/config.toml`에 훅 활성화 설정을 추가하고,
`~/.codex/hooks.json`에 Stop 훅을 등록하는 것이다. 설치 후 새로 생성된 Codex
스레드에는 안정적으로 훅이 적용되며, 메뉴바 앱의 Start/Stop 토글로 즉시 제어할 수 있다.

런타임 상태는 `~/Library/Application Support/loopndroll/state.json`에 저장되고,
앱 바이너리는 `--hook` 모드를 지원해 앱 업데이트 이후에도 훅 기능의 연속성이 보장된다.
GitHub 릴리즈에서 사전 빌드된 앱을 내려받거나 `swift build`로 소스에서 직접 빌드할 수 있다.

## 분석

loopndroll의 설계는 Codex가 제공하는 훅 시스템을 활용한다. "에이전트가 멈추면 다시
시작한다"는 단순한 아이디어를 macOS 네이티브 앱으로 구현했다. 메뉴바 위치 선택은
실용적이다. 백그라운드에서 항상 실행되어야 하지만 화면을 차지하지 않아야 하는
유틸리티 앱의 전형적 형태다.

아키텍처는 세 레이어로 명확히 분리된다. `LoopndrollCore`가 설치·상태·훅 결정 로직을
담당하고, `LoopndrollApp`이 SwiftUI 메뉴바 UI를 제공하며, `LoopndrollHook`이
CLI/테스트용 헬퍼 역할을 한다. 훅 바이너리를
`~/Library/Application Support/loopndroll/bin/`에 고정 경로로 두는 방식은 앱이
업데이트되어도 훅 등록을 다시 하지 않아도 되게 하는 영리한 설계다.

한 가지 주목할 제약은 설치 이전에 실행 중이던 Codex 스레드에는 훅이 동적으로
적용되지 않는다는 점이다. 훅은 스레드 생성 시점에 바인딩되므로, 기존 세션을 이어받지
못하고 새 세션부터 적용된다.

## 비평

loopndroll은 좁고 명확한 문제를 해결한다. Codex 세션이 턴 예산 한계나 자연 종료로
끊기는 불편함을 없애주는 것이 전부다. 이 단순함이 장점이자 한계다.

앱의 범용성은 낮다. Codex 전용이기 때문에 Claude Code, Cursor, Copilot 같은 다른
AI 에이전트에는 직접 적용할 수 없다. Codex 훅 시스템에 종속된 구현이므로 OpenAI가
훅 API를 변경하면 전체 기능이 깨질 수 있다.

현재 애드혹 서명만 지원한다는 점도 macOS Gatekeeper를 우회해야 하는 설치 마찰을
만든다. 광범위한 사용자에게 배포하려면 Developer ID 서명과 공증(notarization)이
필요하다.

별점 123개는 이 도구가 실제 불편함을 해소한다는 증거다. 장황한 기능보다 단 하나의
문제를 정확히 해결하는 도구가 커뮤니티에서 받아들여진다는 점을 보여준다.

## 인사이트

### 에이전트 훅(hook)이 새로운 확장 인터페이스가 되고 있다

Claude Code의
`hooks.json`, Codex의 Stop 훅, goose의 `.goosehints`처럼 AI 에이전트들이
하나같이 훅 시스템을 노출하고 있다. 이는 에이전트의 행동을 수정하는 "미들웨어
레이어"가 표준 인터페이스로 자리잡고 있다는 신호다. 과거 웹 프레임워크가 미들웨어
파이프라인을 제공하면서 수많은 플러그인 생태계가 탄생했듯, AI 에이전트 훅 생태계도
같은 경로를 밟을 가능성이 높다. loopndroll은 그 생태계의 초기 사례다.

### "세션 연장"이라는 니즈가 드러내는 에이전트 설계의 맹점

에이전트가 턴 예산에
걸려 멈춰야 한다는 것은 안전 장치지만, 동시에 장시간 작업을 자연스럽게 이어가지 못한다는
UX 결함이기도 하다. loopndroll 같은 도구가 등장했다는 것은 사용자들이 에이전트에게
더 긴 자율권을 원한다는 수요를 보여준다. 에이전트 개발사들이 이 수요를 공식 기능으로
흡수할지, 아니면 외부 도구에 맡길지는 에이전트 설계 철학의 차이를 반영한다.

### 단일 기능 네이티브 앱의 귀환

웹 앱과 Electron이 주류가 된 시대에 Swift로
작성된 macOS 네이티브 메뉴바 앱은 다소 고전적이다. 하지만 "항상 켜져 있어야 하고,
가볍고, 빠르며, 시스템과 깊이 통합되어야 하는" 유틸리티에서는 네이티브 앱이 여전히
최선이다. AI 에이전트와 함께 돌아가는 보조 도구들이 이 유형에 해당할 가능성이 높다.
Electron 번들 100MB짜리 에이전트 세션 관리 앱보다 Swift 메뉴바 앱이 훨씬 적합하다.
