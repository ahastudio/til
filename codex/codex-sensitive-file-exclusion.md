# OpenAI Codex 민감 파일 제외 이슈

<https://github.com/openai/codex/issues/2847>

HN 토론: <https://news.ycombinator.com/item?id=48706714> (175점, 120개 댓글)

## 소개

OpenAI Codex의 GitHub 이슈 #2847은
특정 파일과 경로를 AI 모델에게 읽히거나 전송되지 않도록 제외하는 기능을 요청한다.
2025년 8월 mkusaka가 제기한 이 이슈는 여전히 미해결 상태다.

제안된 방식은 `.codexignore` 파일과 전역 무시 설정의 두 계층이다.
사용 예시로는 `node_modules/`는 빌드 확인용으로 접근 가능하게 두되,
`.env`, `.env.*`, `*.pem`, `id_*`, `.aws/**`, `.ssh/**`는
모델에게 절대 노출되지 않도록 하는 시나리오를 제시했다.
제보자는 이 기능이 결정론적이고 팀 간에 공유 가능해야 한다고 강조했다.

이 이슈가 HN에서 175점을 받은 것은
민감 파일 제어 문제가 단순한 기능 요청을 넘어 보안 철학 논쟁을 촉발시켰기 때문이다.

## 분석

### OS 레벨 vs 앱 레벨 보안의 오래된 논쟁

이 이슈를 둘러싼 핵심 논쟁은 “보안 경계를 어디에 그어야 하는가”다.
TheDong이 가장 명확하게 정리했다.[^TheDong]
파일 권한(`chmod`)을 사용하거나 컨테이너 안에 민감 파일을 마운트하지 않으면
이미 보호가 가능하다.
문제는 코딩 에이전트가 `rg foo` 같은 셸 명령을 실행할 때
민감 파일의 내용이 검색 결과에 포함돼 의도치 않게 모델에 전달될 수 있다는 것이다.
이것은 `read_file` 도구를 막아도 bash 도구를 통해 우회 가능한 구조다.

nikhilsimha는 더 근본적인 입장을 제시했다.[^nikhilsimha]
코딩 에이전트가 접근할 파일은 기본 제외에서 시작해 명시적으로 허용하는 옵트인 방식이어야 한다고 주장했다.
자신은 사용자 설정 기반 폴더를 격리 환경에 복사한 뒤 세션을 시작하는
커스텀 샌드박스 터미널을 직접 구현했다.
이 접근은 옵트아웃보다 옵트인이 보안 기본값으로 더 적합하다는 원칙을 실천한 사례다.

### `.codexignore` 구현의 기술적 함정

mbid가 오픈소스한 devcontainer 기반 에이전트 오케스트레이션 도구는[^mbid]
이 문제에 대한 현실적 해법 방향을 제시한다.
블록리스트는 불완전할 수밖에 없으므로,
에이전트에게 민감한 데이터가 있는 기기에 직접 접근을 허용하지 않는 것이 핵심이다.
Codex의 클라이언트-서버 아키텍처가 이 점에서 강점을 가진다.
서버 컴포넌트가 컨테이너 안에서 동작하기 때문에
환경 분리가 구조적으로 가능하다.

## 비평

### 앱 수준 필터링은 거짓 보안을 만든다

petcat의 날선 비판이 핵심을 찌른다.[^petcat]
LLM의 예측 불가능한 특성상 앱 수준 파일 제외가 실제 보안을 보장할 수 없다.
이 기능이 구현된다면 사용자는 “.env 파일은 안전하다”는 잘못된 확신을 갖게 되고,
실제로는 bash 도구, 환경 변수 누출, 메모리 내 값 접근 등
수십 가지 우회 경로가 여전히 열려 있다.
작동하지 않는 보안 설정은 없는 것보다 위험하다.

ZiiS는 이 점을 더 구체적으로 설명했다.[^ZiiS]
LLM은 이런 종류의 제한을 우회하는 데 매우 능숙하다.
에이전트가 작성하는 코드 자체가 `.env`를 읽어야 동작한다면,
코드 실행 중에 Codex가 해당 값을 메모리에서 보게 되는 시나리오를 막기 어렵다.
디버깅 과정에서 환경 변수 값이 노출되는 것을 “제외 설정”이 막을 수 없다.

### 설정 공유의 팀 신뢰 문제

제보자는 `.codexignore`가 팀 간에 공유 가능해야 한다고 강조했다.
하지만 이는 새로운 위험 요인을 만든다.
팀원 A가 작성한 `.codexignore` 파일이 팀원 B의 환경에서
어떤 파일을 제외하고 노출하는지를 결정하게 된다.
악의적이거나 실수로 작성된 `.codexignore`가
특정 민감 파일을 의도적으로 노출할 수 있는 새로운 공격 벡터가 된다.
“공유 설정”은 “공유 책임”이 아니라 “공유 위험”일 수 있다.

## 인사이트

### AI 에이전트는 신뢰 모델을 재정의한다

skybrian이 지적한 것이 이 논쟁의 본질을 건드린다.[^skybrian]
`.env` 파일에 API 키를 저장하는 관행 자체가 AI 에이전트 시대에 적합하지 않다.
에이전트가 코드베이스 전체를 읽고 셸 명령을 실행한다면
전통적인 “파일 숨기기” 접근으로는 보안을 보장할 수 없다.
SSH 에이전트처럼 동작하는 프록시, Bearer 인증을 대체하는 구조 등
에이전트 친화적인 비밀 관리 패턴이 새롭게 정의되어야 할 시점이다.

### “사용자 실수”라는 프레이밍의 한계

agentdev001이 “코딩 에이전트를 SSH로 접속한 비신뢰 인간처럼 취급하라”고 조언했지만[^agentdev001]
이 프레이밍은 도구 설계 책임을 사용자에게 전가한다.
일반 개발자는 컨테이너 격리, 파일 권한 관리, 에이전트 보안 모델을 깊이 이해하지 못한다.
강력한 기본값이 없는 도구는 사용하기 쉬울수록 더 위험해진다.
OpenAI가 “사용자가 알아서 해야 한다”는 입장을 유지한다면,
Codex는 보안에 민감한 기업 환경에서 채택되기 어려울 것이다.
이 이슈가 수개월째 미해결인 것은 기술적 어려움 때문일 수도 있지만,
OpenAI가 이 문제를 사용자 책임으로 분류했기 때문일 수도 있다.
