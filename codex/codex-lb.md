# codex-lb: 여러 ChatGPT/Codex 계정을 묶는 로드 밸런서

<https://github.com/Soju06/codex-lb>

## 소개

codex-lb는 여러 개의 ChatGPT/Codex 계정을 하나의 풀로 묶어
API 요청을 분산시키는 로드 밸런서 겸 프록시 서비스다.
계정별 사용량, 비용, 28일 추세를 추적하는 대시보드를 제공하며,
OpenAI 호환 엔드포인트(`/v1`, `/backend-api/codex`)를 통해
Codex CLI, OpenCode, OpenClaw, 표준 OpenAI SDK 클라이언트를
그대로 연결할 수 있다.

설치는 Docker(권장), `uvx codex-lb`, Kubernetes Helm 차트 세 방식을
지원한다.
Docker로 실행하면 컨테이너 로그에 일회성 부트스트랩 토큰이 출력되고,
이 토큰으로 초기 비밀번호를 설정한다.
127.0.0.1 로컬 접속은 부트스트랩 절차를 건너뛴다.
데이터는 로컬/`uvx` 환경에서는 `~/.codex-lb/`, Docker에서는
`/var/lib/codex-lb/`에 저장되며, 데이터베이스는 SQLite(기본값)와
PostgreSQL을 선택할 수 있다.

## 주요 기능

### 라우팅 전략

codex-lb는 8가지 부하 분산 전략을 제공한다.

| 전략                | 설명                                    |
| ------------------- | --------------------------------------- |
| Capacity weighted    | 여유 할당량이 많은 계정 우선            |
| Relative availability | 편향 계수를 조절해 강한 계정 위주 분배 |
| Usage weighted        | 최근 소비 패턴에 반응해 분배           |
| Round robin           | 자격 있는 계정을 균등하게 순환         |
| Fill first            | 한 계정을 소진한 뒤 다음으로 이동      |
| Sequential drain      | 고정된 순서로 계정을 순환              |
| Reset drain           | 리셋 시점이 가까운 계정에 가중치       |
| Single account        | 트래픽을 한 계정에 고정해 격리         |

### 인증과 보안

대시보드는 비밀번호와 선택적 TOTP로 보호된다.
인증 모드는 자체 비밀번호 기반의 standard, 역방향 프록시가 인증을
대신하는 trusted_header(Authelia 호환), 외부에서 이미 보안이
확보된 환경을 위한 disabled 세 가지다.
원격 클라이언트를 위한 API 키 인증은 기본적으로 꺼져 있으며,
필요 시 설정에서 켜면 `/v1/*`, `/backend-api/codex/*`,
`/backend-api/transcribe` 경로가 보호 대상이 된다.
API 키에는 만료 시점, 모델 제한, 일/주/월 단위 토큰·비용
레이트 리밋을 개별적으로 지정할 수 있다.

### 클라이언트 연동

Codex CLI는 `model_providers` 설정에서 `base_url`을
`http://127.0.0.1:2455/backend-api/codex`로 지정하고
`wire_api = “responses”`를 쓰면 codex-lb를 프로바이더로 인식한다.
`CODEX_LB_UPSTREAM_STREAM_TRANSPORT=websocket` 환경 변수로
네이티브 WebSocket 스트리밍도 지원한다.
OpenCode, Python OpenAI SDK, OpenClaw도 각각 `baseURL`/`base_url`을
codex-lb 엔드포인트로 바꾸는 방식으로 동일하게 연동된다.

### 다중 인스턴스와 Kubernetes 배포

Helm 차트로 `oci://ghcr.io/soju06/charts/codex-lb`를 설치하면
멀티 레플리카 환경에서 헤드리스 서비스 DNS를 이용해
HTTP 소유권 인계(owner handoff)를 자동으로 구성한다.
멀티 레플리카 배포에서는 재시작 후에도 일관된 동작을 위해
암호화 키를 레플리카 간에 공유해야 한다.

## 분석

### 8가지 라우팅 전략의 존재는 단일 계정 사용의 한계가 이미 구조적이라는 전제를 깐다

codex-lb가 라우팅 전략을 8가지나 제공한다는 것은,
개발자가 이미 다양한 실패 모드를 예상하고 각각에 대응하는
전략을 미리 마련해 두었다는 뜻이다.
Fill first는 한 계정의 할당량을 다 쓸 때까지 밀어붙이는 전략이고
Reset drain은 리셋 시점이 임박한 계정을 우선하는 전략인데,
이 둘은 서로 다른 최적화 목표(단순함 대 리셋 주기 활용)를 가진다.
전략이 이렇게 세분화된다는 것은 “계정 하나로는 충분하지 않다”는
전제가 이미 광범위하게 공유되고 있으며, 사용자마다 그 부족함을
메우는 방식이 다르다는 것을 시사한다.

이 세분화는 동시에 이 도구가 겨냥하는 사용자층의 성격도 드러낸다.
단순히 “요청이 실패하면 다음 계정으로 넘어간다”는 장애 대응
수준을 넘어, 비용과 리셋 주기를 미세 조정하려는 사용자,
즉 여러 구독 계정을 운영하며 그 비용 구조를 능동적으로
최적화하려는 사용자를 전제한다.
이것은 개인 개발자의 편의 도구라기보다 여러 계정을 조직적으로
관리하는 팀이나 운영자를 위한 인프라에 가깝다.

### OpenAI 호환 계층은 도구가 아니라 프로토콜을 재산으로 삼는 전략이다

codex-lb가 `/v1`과 `/backend-api/codex` 양쪽 경로를 모두
지원하며 Codex CLI, OpenCode, OpenClaw, 표준 OpenAI SDK까지
포괄하는 것은, 이 프로젝트가 특정 클라이언트에 종속되지 않고
“OpenAI 호환”이라는 사실상 표준 프로토콜 위에 자신을 배치하는
전략을 취했다는 것을 보여준다.
어떤 클라이언트가 나중에 등장하더라도 OpenAI 호환 API를
말하기만 하면 codex-lb는 별도 어댑터 없이 그 클라이언트를
지원할 수 있다.

이 전략의 이면에는 리스크도 있다.
OpenAI가 API 스펙을 바꾸거나 Codex 전용 백엔드 경로의 동작을
변경하면, codex-lb는 그 변경을 그대로 흡수해야 하는 처지에
놓인다.
즉 이 프로젝트의 안정성은 스스로 통제할 수 없는 업스트림의
API 안정성에 근본적으로 의존한다.
`wire_api = “responses”`처럼 Codex 고유의 전송 방식까지
흉내 내야 하는 지점은, 표준 준수를 넘어 특정 벤더의 비공개
구현 세부사항을 리버스 엔지니어링해 맞춰야 하는 유지보수
부담으로 이어질 수 있다.

### 부트스트랩 토큰과 trusted_header 모드는 셀프 호스팅 인프라의 전형적 신뢰 전이 패턴을 따른다

부트스트랩 토큰이 비밀번호 설정 전까지만 유효하고 로컬 접속은
그 절차 자체를 생략한다는 설계는, “물리적 또는 네트워크적으로
이미 신뢰된 접근”을 별도 인증 없이 통과시키는 흔한 패턴이다.
trusted_header 모드가 Authelia 같은 역방향 프록시와의 호환을
명시한 것도 마찬가지로, 인증 책임을 codex-lb 자신이 아니라
앞단의 프록시에 위임하는 구조다.
이 패턴은 셀프 호스팅되는 오픈소스 인프라 도구 전반에서
반복적으로 나타난다.

이런 신뢰 전이 구조가 성립하려면 전제 조건이 있다.
로컬 네트워크나 리버스 프록시 계층이 실제로 안전하게 격리되어
있어야 한다는 것이다.
`CODEX_LB_FIREWALL_TRUSTED_PROXY_CIDRS`처럼 신뢰할 CIDR 대역을
직접 지정하게 만든 것은 이 전제를 사용자에게 명시적으로
확인시키는 장치이지만, 동시에 그 대역을 잘못 설정했을 때의
위험 부담도 사용자에게 그대로 넘긴다.

## 비평

### 다중 계정 풀링이 서비스 약관과 충돌할 가능성을 문서가 다루지 않는다

codex-lb는 여러 ChatGPT/Codex 계정을 하나의 백엔드에서 풀링해
자동으로 순환시키는 도구다.
그런데 README에는 이런 방식의 계정 운용이 OpenAI의 서비스
약관과 어떤 관계에 있는지에 대한 언급이 전혀 없다.
개인용 구독을 자동화된 방식으로 로드 밸런싱하는 것이
약관상 허용되는 사용 범위인지, 계정 정지 위험이 있는지는
프로젝트가 사용자에게 명시적으로 알려야 할 정보다.

이 누락은 사소하지 않다.
Reset drain이나 Usage weighted 같은 전략은 명백히 여러 계정의
리셋 주기와 사용량 패턴을 조합해 비용 효율을 극대화하려는
목적을 갖는데, 이는 개인 계정 하나를 전제로 설계된 요금제의
의도를 우회하는 방향에 가깝다.
프로젝트가 이 우회의 성격을 스스로 인식하고 있다면 최소한
사용자에게 위험을 고지해야 하고, 인식하지 못하고 있다면
그 자체가 설계 검토의 공백이다.
어느 쪽이든 README에 이 주제가 등장하지 않는다는 사실은
이 도구를 실제 도입하려는 사용자에게 중요한 정보 공백으로
남는다.

### 8가지 전략의 선택 기준이 사용자에게 위임된 채 설명되지 않는다

README는 8가지 라우팅 전략의 동작 방식은 나열하지만,
어떤 상황에서 어떤 전략을 골라야 하는지에 대한 지침은
제공하지 않는다.
Capacity weighted와 Relative availability는 둘 다 “여유
있는 계정을 우선한다”는 점에서 비슷해 보이지만 구체적으로
어떤 지표 차이가 있는지, 언제 하나가 다른 하나보다 낫다고
판단해야 하는지는 설명되지 않는다.

이 공백은 사용자에게 실질적인 의사결정 부담을 떠넘긴다.
계정을 3개 운영하는 사용자와 20개 운영하는 사용자, 비용
최소화가 목표인 사용자와 응답 지연 최소화가 목표인 사용자는
서로 다른 전략을 골라야 할 텐데, 그 대응 관계를 문서가
제시하지 않으면 사용자는 시행착오로 최적 전략을 찾아야 한다.
전략을 8가지나 만든 설계 노력에 비해, 그 전략을 고르는
의사결정 지원은 상대적으로 얇다.

### PostgreSQL 지원과 SQLite 기본값 사이의 마이그레이션 경로가 불분명하다

codex-lb는 SQLite를 기본으로 하되 `CODEX_LB_DATABASE_URL`로
PostgreSQL을 선택할 수 있다고 설명한다.
그러나 SQLite로 시작한 배포를 나중에 PostgreSQL로 옮기는
절차, 혹은 반대로 PostgreSQL 환경에서 SQLite로 되돌리는
절차에 대해서는 언급이 없다.
Helm 차트 설정에 `databaseMigrateOnStartup=true` 옵션이
등장하는 것을 보면 스키마 마이그레이션 체계 자체는 존재하는
것으로 보이지만, 데이터베이스 백엔드 자체를 교체하는
시나리오는 문서 범위 밖에 있다.

이것은 실무에서 흔히 발생하는 전환 경로다.
개인이 SQLite로 시작해 검증한 뒤 팀 단위로 확장하며
PostgreSQL과 Kubernetes로 옮겨가는 흐름은 이 프로젝트가
Docker와 Helm을 모두 지원한다는 점에서 자연스럽게
예상되는 성장 경로인데도, 그 전환 시점에 필요한 데이터
이관 절차가 문서화되어 있지 않다.

## 인사이트

### codex-lb는 AI 코딩 도구 소비 방식이 API 키에서 계정 풀로 옮겨가는 과도기의 산물이다

전통적인 로드 밸런서는 API 키 여러 개를 발급받아 분산시키는
것을 전제로 발전해 왔다.
하지만 ChatGPT/Codex 구독은 API 키가 아니라 사람이 로그인하는
계정 단위로 설계된 상품이며, codex-lb는 바로 이 “계정”을
로드 밸런싱의 단위로 삼는다.
이는 API 상품과 구독 상품 사이의 경계가 실무에서 점점
흐려지고 있다는 신호로 읽을 수 있다.
사용자는 API 요금제의 유연한 종량 과금을 원하면서도 구독
요금제의 예측 가능한 고정 비용을 함께 누리고 싶어 하고,
codex-lb 같은 도구는 그 틈새에서 “구독 계정을 API처럼
쓸 수 있게” 만드는 중개 계층으로 등장한다.

이 현상은 codex-lb만의 특수 사례가 아니라 AI 서비스 소비
생태계 전반에서 반복될 가능성이 크다.
구독형 AI 상품이 늘어날수록, 그 구독을 프로그래밍적으로
제어하고 싶어 하는 개발자 수요도 함께 늘어난다.
벤더 입장에서는 이런 도구가 확산될수록 구독과 API의
가격 차등 전략 자체가 무력화되는 압력을 받게 되므로,
장기적으로는 벤더가 구독 계정의 API 접근을 더 엄격하게
제한하거나, 반대로 이런 수요를 흡수하는 공식 상품을
내놓는 두 방향 중 하나로 대응할 가능성이 있다.

### 셀프 호스팅 로드 밸런서의 확산은 AI 인프라의 관제탑이 벤더에서 사용자 쪽으로 일부 이동하고 있음을 보여준다

과거 API 트래픽 관리, 레이트 리밋, 사용량 분석은 대개 벤더가
제공하는 대시보드의 몫이었다.
codex-lb가 자체 대시보드에서 계정별 토큰, 비용, 28일 추세를
추적하고 자체 레이트 리밋 정책을 사용자가 직접 설정하게
한다는 것은, 이런 관제 기능의 일부가 벤더의 손을 떠나
사용자가 직접 운영하는 인프라 계층으로 옮겨가고 있다는
뜻이다.

이 이동은 사용자에게 더 큰 통제력을 주는 동시에 새로운
책임도 함께 떠넘긴다.
벤더의 공식 대시보드가 제공하지 못하는 세밀한 제어(전략별
라우팅, 커스텀 레이트 리밋)를 얻는 대신, 사용자는 이제
자신이 직접 운영하는 셀프 호스팅 서비스의 보안, 가용성,
데이터 무결성까지 책임져야 한다.
Kubernetes Helm 차트와 멀티 레플리카 지원까지 갖췄다는
것은 이 프로젝트가 이미 “개인 도구”의 범위를 넘어 조직
차원의 인프라로 성장할 준비를 하고 있다는 뜻이며, 이는
곧 AI 사용량 관리라는 영역이 독자적인 인프라 카테고리로
분화되고 있다는 더 큰 흐름의 초기 사례로 볼 수 있다.

### 다중 계정 자동화 도구의 존재 자체가 벤더의 가격 정책 설계에 대한 암묵적 피드백이다

사용자가 codex-lb 같은 도구를 만들어 여러 계정을 자동으로
순환시킨다는 사실은, 단일 계정 요금제의 할당량이나
가격 구조가 실제 수요를 충분히 만족시키지 못한다는 것을
행동으로 드러내는 신호다.
설문이나 요청 대신, 개발자들은 도구를 만들어 문제를 우회하는
방식으로 자신들의 불만을 표출한다.
이것은 소프트웨어 생태계에서 반복적으로 관찰되는 패턴이다.
공식 API가 불편하면 비공식 클라이언트가 등장하고, 요금제가
과도하게 제한적이면 그 제한을 우회하는 도구가 등장한다.

벤더 입장에서 이런 도구의 등장은 무시할 수 없는 시장
신호다.
이 신호를 무시하면 사용자들은 계속 우회 도구에 의존하게
되고, 벤더는 실제 사용 패턴에 대한 통제력과 가시성을
잃는다.
반대로 이 신호에 대응해 API 요금제를 유연하게 조정하거나
팀 단위 계정 풀링을 공식 기능으로 흡수하면, 우회 도구의
존재 이유 자체가 줄어든다.
codex-lb의 존재는 그 자체로 OpenAI를 비롯한 AI 서비스
벤더들에게 “현재 계정 단위 요금제 설계가 실제 사용
패턴과 어긋나 있다”는 것을 보여주는 시장의 무언의
피드백이다.
