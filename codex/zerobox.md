# Zerobox - OpenAI Codex 런타임 기반 경량 프로세스 샌드박싱

<https://github.com/afshinm/zerobox>

## 요약

Zerobox는 OpenAI Codex 샌드박스 런타임에서 독립적으로 추출한 경량 크로스플랫폼
프로세스 격리 도구다. Docker나 VM 없이 약 10ms 오버헤드만으로 명령어를 안전한 환경에서
실행한다. 기본 거부(deny-by-default) 정책으로 동작하며 파일 쓰기, 네트워크 접근,
환경 변수를 명시적으로 허용하지 않는 한 모두 차단한다.

핵심 사용 시나리오는 AI 에이전트가 생성한 코드를 실행할 때다. LLM이 작성한 코드를
그대로 실행해도 파일 손상이나 데이터 유출 걱정 없이 동작하게 한다. API 키를 프로세스에
직접 노출하지 않고 승인된 호스트에만 자격증명을 주입하는 기능도 제공한다.

기술 구현은 플랫폼별로 다르다. macOS에서는 Apple의 Seatbelt(sandbox-exec),
Linux에서는 Bubblewrap + Seccomp + Namespaces를 사용한다. Windows 지원은 계획
중이다. Rust와 TypeScript SDK를 제공하며 라이선스는 Apache-2.0이다.

주요 기능으로는 특정 경로의 읽기/쓰기 허용 또는 거부, 도메인별 아웃바운드 트래픽 제어,
파일시스템 변경사항 추적 및 복원(스냅샷 기능), 권한 분리(LLM 도구 호출마다 다른 권한
적용)가 있다.

```bash
# 기본 실행
zerobox -- node -e "console.log('hello')"

# 현재 디렉터리 쓰기 허용
zerobox --allow-write=. -- node script.js

# API 키를 특정 도메인에만 주입
zerobox --secret OPENAI_API_KEY=sk-proj-123 \
  --secret-host OPENAI_API_KEY=api.openai.com -- node agent.js
```

## 분석

### Codex 런타임 추출의 의미

OpenAI Codex가 내부적으로 사용하는 샌드박스 메커니즘을 독립 도구로 추출했다는 점이
흥미롭다. 이는 Codex가 LLM 생성 코드 실행 시 실제로 어떤 격리 전략을 쓰는지를
간접적으로 보여준다. 프로덕션 AI 에이전트가 검증한 접근법을 범용 도구로 만들었다는
신뢰성이 있다.

기존 샌드박싱 도구들(Docker, Firejail, nsjail)과 비교했을 때 Zerobox의 차별점은
AI 에이전트 워크플로우에 특화된 API 설계다. 자격증명 주입(--secret)과 도메인별
트래픽 제어는 LLM 에이전트가 외부 API를 호출하면서도 키를 노출하지 않아야 하는
특수한 요구사항을 직접 해결한다.

### 보안 모델의 구조적 특징

Zerobox의 보안 모델은 최소 권한 원칙(principle of least privilege)을 CLI 레벨로
구현한다. 프로세스가 실행되는 시점에 필요한 권한 목록을 명시적으로 나열하는 방식은,
전통적인 Unix 파일 권한보다 세밀한 제어를 가능하게 한다.

스냅샷 기능은 특히 유용하다. `npm install` 후 자동으로 변경사항을 복원하거나 기록할 수
있어, 에이전트가 환경을 수정하더라도 원래 상태로 되돌릴 수 있다. 이는 반복적인 에이전트
실행 시나리오에서 환경 오염을 방지한다.

## 비평

### 강점

AI 에이전트 보안이라는 구체적 문제를 다룬다. "LLM이 쓴 코드를 안전하게 실행"이라는
문제는 agentic coding이 확산되면서 실질적으로 중요해지고 있다. 10ms 오버헤드는
대부분의 에이전트 워크플로우에서 허용 가능한 수준이다.

CLI 인터페이스가 직관적이다. 기존 명령어 앞에 `zerobox --` 를 붙이는 방식은 학습
비용이 낮다.

### 약점

Windows 미지원이 치명적 약점이다. Windows 환경에서 Claude Code나 Codex를 사용하는
개발자가 적지 않은데, 이들은 사용할 수 없다.

macOS Seatbelt와 Linux Bubblewrap의 격리 수준이 다르다. 두 플랫폼 간 보안 보장이
동등한지 문서에서 명확히 설명되지 않아 크로스플랫폼 보안 가정이 위험할 수 있다.

`--secret-host` 방식의 자격증명 주입이 네트워크 프록시 없이 어떻게 구현되는지 설명이
부족하다. 실제로 API 키가 어떤 경로로 대상 호스트에만 전달되는지 보안 검토 없이
신뢰하기 어렵다.

## 인사이트

### AI 에이전트 시대의 샌드박싱 재발명

전통적인 샌드박싱은 신뢰할 수 없는 사용자 입력으로부터 시스템을 보호하기 위해 발전했다.
Zerobox가 다루는 문제는 다르다. 에이전트가 호출하는 코드 자체는 악의적이지 않더라도,
그 코드가 의도치 않게 시스템을 변경하거나 민감한 정보를 노출할 수 있다는 불확실성이 문제다.
LLM의 특성상 동일한 프롬프트도 매번 다른 코드를 생성할 수 있고, 그 코드의 부작용을
사전에 완전히 예측할 수 없다. 이것이 AI 에이전트 시대의 새로운 보안 요구사항이다.

전통적 샌드박스가 "차단"에 초점을 맞췄다면, Zerobox는 "제한된 허용"에 초점을 맞춘다.
이 차이가 API 설계에 반영된다. Docker는 무엇을 열지를 설정하지만, Zerobox는 무엇을
허용할지를 실행 시점에 명시한다. 에이전트 워크플로우는 실행마다 다른 권한이 필요할 수
있기 때문에, 정적 컨테이너 설정보다 동적 권한 주입이 더 적합하다.

### 자격증명 격리는 아직 미해결 문제다

AI 에이전트가 외부 API를 호출할 때 API 키를 어떻게 관리할 것인가는 업계 전반의 미해결
문제다. 현재 일반적인 방법은 환경 변수로 키를 주입하는 것인데, 이는 에이전트가 실행하는
모든 서브프로세스에 키가 노출된다는 의미다. LLM이 `os.environ`을 읽어 키를 출력하는
코드를 생성하면 즉시 유출된다.

Zerobox의 `--secret-host` 방식이 이 문제를 해결한다고 주장하지만, 구현 방식에 따라
실제 보안 수준이 크게 달라진다. 네트워크 레이어에서 자격증명을 주입하는 방식(프록시)과
환경 변수 필터링 방식은 보안 속성이 근본적으로 다르다. 이 문제가 제대로 해결된다면
agentic coding 인프라의 핵심 구성요소가 될 수 있다.

### 오픈소스 AI 인프라의 분화

OpenAI가 Codex CLI 내부 구현을 오픈소스로 공개하고, 그것을 제3자가 독립 도구로 추출하는
패턴이 나타나고 있다. 이는 AI 에이전트 인프라가 성숙하면서 공통 컴포넌트가 분리되는
자연스러운 과정이다. 샌드박싱, 도구 호출 관리, 컨텍스트 압축, 메모리 관리 등이 각각
독립 도구로 발전할 것이다.

이 분화는 AI 에이전트 스택이 Web 개발 생태계처럼 레이어드 아키텍처로 발전함을 시사한다.
LLM 레이어, 에이전트 프레임워크 레이어, 실행 환경 레이어, 보안 레이어가 각각 전문화되고
조합되는 방식이다. Zerobox는 실행 환경과 보안 레이어의 교차점에 위치한다.
