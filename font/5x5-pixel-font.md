# 작은 화면을 위한 5×5 픽셀 폰트

원문: <https://maurycyz.com/projects/mcufont/>

## 요약

Maurycy가 설계한 5×5 픽셀 폰트는 극도로 제한된 환경을 위한
350바이트짜리 초소형 모노스페이스 폰트다.
8비트 마이크로컨트롤러(AVR128DA28 등)와
160×128, 128×64 OLED 같은 소형 디스플레이를 주 대상으로 한다.

5×5는 모든 문자를 읽기 가능하게 렌더링할 수 있는 최소 크기다.
4×4로 내려가면 E, M, W 같은 글자의 가로획이 충분히 표현되지 않아
가독성이 무너진다.
소문자는 대문자보다 1픽셀 작게 렌더링하여 시각적 구분을 제공한다.

6×6 그리드 안에 5픽셀 정사각형으로 문자를 그리는 방식이며,
안티에일리어싱과 메가바이트급 코드를 동원한 벡터 폰트보다
실제 소형 디스플레이에서 더 나은 결과를 보인다는 것이 설계자의 주장이다.

## 크기별 가독성 스펙트럼

| 크기  | 가독성              |
| ----- | ------------------- |
| 5×5   | 최소 실용 크기      |
| 3×5   | 여전히 읽을 수 있음 |
| 3×4   | 제약이 심함         |
| 3×3   | 도전적              |
| 2×3   | 거의 읽기 불가      |
| 3×2   | 2×3보다 나음 (가로세로 비율 역전이 오히려 유리) |
| 2×2   | 사실상 비밀 코드    |

## 분석

### 350바이트의 의미

350바이트는 단순한 파일 크기가 아니다.
AVR128DA28의 플래시 메모리는 128 KiB,
SRAM은 16 KiB에 불과하다.
벡터 폰트 렌더링 라이브러리는 수십에서 수백 KiB를 요구하며,
폰트 데이터만도 수십 KiB가 기본이다.
350바이트짜리 폰트 데이터는 마이크로컨트롤러 전체 플래시의 0.27%만 차지한다.
임베디드 시스템에서 폰트에 할당할 수 있는 예산이
이 수준으로 제한된 환경이 실제로 존재하며,
그런 환경을 위한 폰트 설계는 일반 UI 폰트 설계와 완전히 다른 제약 조건을 갖는다.

### 5×5가 마법의 최솟값인 이유

5×5가 “최소 실용 크기”인 것은 라틴 알파벳의 구조적 특성과 맞닿아 있다.
영문자는 획 교차, 닫힌 카운터, 세리프 없는 단순 획으로 이루어지지만,
E(가로획 3개), M(대각선), W(복잡한 V자 반복) 같은 글자는
최소 5열의 픽셀 공간이 있어야 구분 가능한 형태를 갖춘다.
4열이면 M과 N의 구분이 모호해지고,
E와 F도 하단 획 유무로만 구분해야 한다.
5×5는 이 모든 판별 특징을 유지할 수 있는 임계점이다.

### 소문자 1픽셀 축소의 타이포그래피

소문자를 대문자보다 1픽셀 작게 그린다는 설계 결정은
전통적인 타이포그래피의 x-height 개념을 픽셀 그리드에 적용한 것이다.
전통 서체에서 소문자의 높이(x-height)는 대문자보다 낮아
텍스트 블록에서 시각적 리듬과 밀도를 만든다.
5×5 제약에서 이를 구현하는 유일한 방법이 1픽셀 차이다.
타이포그래피 원칙이 극단적 제약 속에서 어떻게 최소 표현으로 증류되는지를 보여준다.

## 비평

### 강점

350바이트라는 수치와 6×6 그리드 설계는 매우 구체적인 제약에서 나온 해결책이다.
“이론적으로 가능한 최소”가 아니라 “실제로 읽을 수 있는 최소”를 추구한 점이
실용적이다.
크기별 가독성 스펙트럼(5×5부터 2×2까지)을 실험하고 문서화한 것은
같은 문제를 고민하는 임베디드 개발자에게 직접적으로 유용한 정보다.

### 약점

[GeekNews 댓글](https://news.hada.io/topic?id=28813)의 한국 사용자가 지적한 것처럼,
한글(CJK 문자)을 5×5에 담는 것은 사실상 불가능하다.
한글은 초성·중성·종성을 하나의 셀에 결합해야 하므로
최소 8×8 또는 16×16 픽셀이 필요하다.
이 폰트는 라틴 알파벳 중심 환경에서만 유효한 해결책이다.
또한 350바이트에 어떤 문자 집합이 포함되는지—
순수 ASCII인지, 확장 문자를 포함하는지—에 대한 설명이 부족하다.

## 인사이트

### 제약이 설계를 순수하게 만든다

5×5 픽셀 폰트는 제약이 설계를 어떻게 정제하는지를 보여주는 좋은 예다.
“어떤 폰트가 아름다운가”라는 질문은 무한한 답을 허용하지만,
“8비트 마이크로컨트롤러에서 350바이트로 모든 ASCII 문자를 읽기 가능하게 표현하라”는
질문은 해결 공간을 극적으로 좁힌다.
이런 수준의 제약은 설계자가 “무엇이 본질인가”를 발견하도록 강요한다.
E, M, W를 구분하는 데 필요한 최소 픽셀 수가 얼마인가—
이 질문에 답하면서 설계자는 라틴 알파벳의 판별 특징 구조를
픽셀 단위로 이해하게 된다.

이 원칙은 폰트 설계를 넘어 일반적으로 적용된다.
임베디드 시스템 프로그래밍, 프로토콜 설계, 데이터 압축 분야에서
극단적 제약은 종종 가장 우아한 해결책을 낳는다.
TCP 헤더가 20바이트로 인터넷의 신뢰성 있는 전송을 구현하고,
QR 코드가 흑백 격자로 수백 바이트의 데이터를 인코딩하듯,
5×5 픽셀 폰트는 “최소 표현”의 미학을 타이포그래피에서 실현한다.

### 임베디드 UI의 보이지 않는 설계

일반 사용자는 스마트폰이나 PC 화면의 폰트를 당연하게 여긴다.
하지만 의료 기기의 소형 LCD, 산업용 센서의 디스플레이,
가전제품의 상태 표시창 같은 곳에도 글자가 있고,
그 글자를 만든 누군가의 설계 결정이 존재한다.
5×5 픽셀 폰트는 이 보이지 않는 설계의 세계를 드러낸다.
현대 폰트 렌더링의 복잡성—힌팅, 서브픽셀 렌더링, 리게이처—이 모두 제거된 자리에
“이 6×6 그리드 안에 E를 그릴 수 있는가”라는 원초적 질문만 남는다.
이 질문에 답하는 350바이트의 데이터가 어딘가의 마이크로컨트롤러에서
사용자에게 정보를 전달하고 있다는 사실은, 작지만 아름다운 공학의 단면이다.
