# Datatype: 텍스트를 차트로 변환하는 가변 폰트

<https://github.com/franktisellano/datatype>

## 소개

Datatype는 자바스크립트 없이 순수 텍스트 표기법으로 인라인 차트를 렌더링하는
OpenType 가변 폰트다. OpenType 합자(ligature) 치환 기술을 이용해 `{b:30,70,50,90}`
같은 텍스트 패턴을 자동으로 시각적 막대 차트로 변환한다. 차트 렌더링에 별도의
라이브러리나 런타임이 필요 없고 폰트 하나만 불러오면 된다.

## 주요 기능

세 가지 차트 타입을 지원한다.

- **막대 차트**: `{b:30,70,90,64,27,42}` — 최대 20개 값
- **스파크라인**: `{l:10,50,30,80,20}` — 최대 20개 포인트
- **파이 차트**: `{p:75}` — 단일 퍼센트 값

```text
{b:30,70,90,64,27,42}
{l:10,50,30,80,20}
{p:75}
```

## 가변 축(Variable Axes)

두 개의 축으로 차트 외형을 조정할 수 있다.

| 축       | 범위      | 기본값 | 설명               |
| -------- | --------- | ------ | ------------------ |
| 너비(wdth) | 50–150  | 100    | 차트 간격 및 폭    |
| 굵기(wght) | 100–900 | 400    | 선 두께 및 막대 굵기 |

CSS에서 `font-variation-settings`로 조정한다.

```css
@font-face {
  font-family: 'Datatype';
  src: url('Datatype.woff2') format('woff2');
}

.chart {
  font-family: 'Datatype';
  font-variation-settings: 'wdth' 120, 'wght' 600;
}
```

## 기술 구조

OpenType 합자 치환이 핵심 메커니즘이다. 합자는 전통적으로 “fi”, “fl” 같은 문자
조합을 하나의 글리프로 대체하는 데 쓰였다. Datatype는 이 메커니즘을 확장해
데이터 패턴 전체를 차트 글리프 시퀀스로 치환한다. 약 10,850개의 글리프와 9개의
폰트 마스터로 구성된다.

파일 크기는 TTF 4.0MB, WOFF2 약 78KB다. WOFF2 압축률이 높아 웹 자체 호스팅이
현실적이다. Google Fonts를 통해 배포된다.

빌드 환경은 Python 3.9+와 fontTools를 사용한다.

```bash
# 전체 빌드
python build.py

# 개발 모드 (빠른 반복)
python build.py --dev

# 로컬 프리뷰 서버
python serve.py
```

## 분석

### 합자 치환의 창의적 재활용

폰트 합자 치환은 수십 년 된 기술이다. Datatype는 이것을 데이터 시각화 렌더링
엔진으로 재활용했다. 텍스트 렌더링 파이프라인 자체가 차트 렌더링 엔진이 되는
셈이다. 이 접근의 영리한 점은 플랫폼 지원이 필요 없다는 것이다. CSS `font-family`
를 지정할 수 있는 곳이라면 어디서든 작동한다. 문서, 테이블, 대시보드, 로그 메시지,
이메일까지 적용 범위가 넓다.

기술적으로는 OpenType GSUB(Glyph Substitution) 테이블을 활용한다. 시스템이 텍스트
를 렌더링할 때 합자 규칙을 적용해 문자 시퀀스를 대체 글리프로 치환한다. 이 과정은
자바스크립트 실행 전에 폰트 레벨에서 이뤄진다.

### 사용 맥락의 확장

[GeekNews 댓글](https://news.hada.io/topic?id=29640)에서 xguru가 실제로 구현해
보인 예시가 흥미롭다. `{b:30,70,90,64,27,42}`와 `{l:10,50,30,80,20}`을 직접
렌더링했고, WOFF2의 작은 용량 덕분에 자체 호스팅이 현실적이라고 언급했다. 이는
외부 서비스 의존 없이 내부 도구에 통합하기 쉽다는 것을 의미한다.

데이터가 자주 업데이트되지 않는 정적 문서나 마크다운 기반 위키에서 특히 유용하다.
JS 기반 차트 라이브러리를 불러올 수 없는 환경에서도 차트를 포함할 수 있다.

## 비평

### 제약된 표현 범위

막대 차트, 스파크라인, 파이 차트 세 가지 타입으로 표현 범위가 고정돼 있다. 축
레이블, 범례, 색상 구분, 복합 차트 같은 정보 밀도가 높은 시각화는 지원하지 않는다.
수치가 “얼마나 큰가”는 표현하지만 “무엇인가”는 텍스트로 별도 표기해야 한다.

최대 20개 값 제한도 실제 데이터 시각화 요구와 맞지 않을 수 있다. 트렌드 시계열
데이터는 보통 이보다 많은 포인트를 가진다.

### 접근성 문제

글리프로 렌더링된 차트는 스크린 리더가 인식하지 못한다. 텍스트 원본이 `{b:30,70,90}`으로
남아 있어 스크린 리더가 읽기는 하지만 이것이 차트라는 맥락이 전달되지 않는다.
데이터 접근성 측면에서 alt text 같은 대안 제공 방법이 없다는 점은 한계다.

### 폰트 렌더링 환경 의존성

합자 치환은 환경마다 다르게 동작할 수 있다. 특정 텍스트 에디터, 터미널, 이메일
클라이언트는 합자를 비활성화하거나 다르게 처리할 수 있다. 폰트가 제대로 로드되지
않으면 `{b:30,70,90}` 텍스트 그대로 노출된다. 폴백 처리가 텍스트 그대로이므로
기능 저하(graceful degradation)는 되지만, 의도한 시각화를 기대한 독자에게는
혼란을 줄 수 있다.

## 인사이트

### 기존 인프라의 창의적 재활용이 최고의 해킹이다

새로운 기능을 구현하는 방법은 새로운 시스템을 만드는 것만이 아니다. 기존 시스템의
예상치 못한 활용이 때로 더 우아한 해결책이 된다. Datatype는 텍스트 렌더링 파이프
라인이 이미 지원하는 글리프 치환 기능을 데이터 시각화에 재활용했다. 이 발상의
전환이 “자바스크립트 없는 차트”라는 새로운 범주를 만들었다.

이 접근 방식의 장점은 플랫폼 지원 확보가 필요 없다는 점이다. CSS를 지원하는 모든
환경이 자동으로 지원 환경이 된다. 새로운 API를 설계하고, SDK를 배포하고, 플랫폼
별 통합을 관리하는 대신 이미 존재하는 인프라 위에 올라탔다.

소프트웨어 설계에서 “기존 메커니즘의 예상치 못한 조합”이 최고의 혁신으로 이어지는
경우가 많다. Unix 파이프, HTTP의 GET/POST를 넘어선 REST 의미론, CSS 애니메이션의
키프레임 재활용 모두 같은 패턴이다.

### 파일 크기가 곧 채택률이다 — WOFF2 78KB의 의미

4.0MB TTF와 78KB WOFF2의 크기 차이는 단순한 압축 효율의 문제가 아니다. 78KB는
웹 폰트로 자체 호스팅하기에 현실적인 임계값이다. 일반적인 웹 폰트 하나가 100–300KB
수준이므로, 차트 기능을 추가하는 비용이 웹 폰트 하나 수준이다. 이 크기 결정이
도구의 실용성을 결정한다.

78KB를 넘었다면 “멋있지만 실제로는 쓰기 어려운” 프로젝트로 남았을 가능성이 높다.
기술적 우아함이 채택으로 이어지려면 사용 비용이 채택 임계값 아래에 있어야 한다.
Datatype는 이 계산을 정확히 맞췄다.

### 단순성의 설계 가치 — 표현력 제한이 오히려 강점이 된다

세 가지 차트 타입과 20개 값 제한이라는 제약은 단점이 아니라 설계 선택이다. 더
많은 기능을 추가할수록 합자 규칙의 복잡도가 급증하고 파일 크기도 늘어난다. 제약을
유지함으로써 “텍스트에 인라인으로 넣을 수 있는 간단한 차트”라는 핵심 사용 사례에
집중할 수 있다.

이 원칙은 도구 설계 일반에 적용된다. 범용 도구는 특수 목적 도구를 이기기 어렵다.
Datatype는 “텍스트 환경에서의 인라인 데이터 시각화”라는 명확히 정의된 문제를 풀고,
그 문제 밖으로 나가지 않는다. 이 명확함이 도구의 정체성을 만들고, 특정 사용 사례에서
대체 불가능한 도구가 되는 기반이 된다.
