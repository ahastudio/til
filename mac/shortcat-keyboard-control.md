# Shortcat — 키보드만으로 맥 전체 제어하기

<https://shortcat.app>

GeekNews: <https://news.hada.io/topic?id=30268>

## 소개

Shortcat은 macOS의 UI를 키보드로 완전히 제어할 수 있게 해 주는 도구다.
마우스 없이 모든 UI 요소에 접근하는 것을 목표로 하며, 앱 메뉴 항목, 버튼, 링크, 창 목록을 타이핑만으로 탐색하고 클릭한다.

인터페이스는 커맨드 팔레트 방식이다.
단축키를 누르면 현재 앱 기준으로 접근 가능한 UI 요소가 퍼지 매칭(fuzzy matching) 방식으로 검색된다.
“ok”를 입력하면 OK 버튼을, “save”를 입력하면 저장 관련 메뉴 항목을 찾아 클릭한다.
우클릭, 더블클릭, modifier 키 조합도 지원한다.

macOS 13 이상, Apple Silicon 및 Intel Mac을 지원한다.

## 주요 기능

**창 전환**: Command+Tab 없이 창 제목으로 직접 전환한다.
현재 열려 있는 창들을 목록으로 탐색하고 원하는 창으로 즉시 이동한다.

**메뉴 접근**: 현재 앱의 메뉴 항목을 퍼지 검색으로 탐색한다.
단축키를 외우지 않아도 메뉴 이름 일부를 입력해 실행할 수 있다.

**이모지 입력**: “poop”, “heart” 같은 키워드로 이모지를 퍼지 매칭해 빠르게 입력한다.

**동의어 지원**: “Delete item”을 검색하면 “remove”, “clear”, “destroy”가 포함된 요소도 매칭된다.

**브라우저 통합**: Safari, Chrome, Firefox, Vivaldi 및 Chromium 기반 브라우저를 지원한다.

**Electron 앱 지원**: VS Code, Home Assistant, Signal, 1Password 8 같은 Electron 앱과 호환된다.

## 분석

### 접근성 API를 생산성 도구로 전환하는 설계

Shortcat이 동작하는 원리는 macOS의 Accessibility API다.
이 API는 원래 장애 사용자를 위한 스크린 리더 등 보조 기술을 위해 설계됐으나, Shortcat은 이를 일반 사용자의 키보드 중심 워크플로에 활용한다.

이 접근이 흥미로운 이유는 새로운 기술을 도입하는 것이 아니라 기존 시스템의 다른 층위를 노출하는 방식이기 때문이다.
모든 macOS 앱은 이미 Accessibility API를 통해 자신의 UI 요소를 운영체제에 알린다.
Shortcat은 이 정보를 검색 가능한 인터페이스로 만드는 것이다.
결과적으로 앱이 별도의 지원 없이도 Shortcat에서 동작하는 경우가 많다.

### 키보드 중심 워크플로의 비용과 이점

키보드 중심으로 작업하는 것은 초기 학습 비용이 있다.
어떤 도구가 있는지, 어떻게 불러내는지를 습관화하는 기간이 필요하다.
단축키 기반 도구들 — Vim, tmux, 각종 키보드 런처 — 이 공유하는 이 진입 장벽은 Shortcat도 피해가지 않는다.

그러나 습관화된 이후의 이득은 맥락 전환 비용 감소다.
마우스로 이동하는 행위는 키보드에서 손을 떼고, 화면으로 시선을 옮기고, 원하는 요소를 찾아 이동하는 여러 단계를 포함한다.
타이핑으로 UI 요소에 직접 접근할 때 이 단계들이 생략된다.
마우스 이동의 물리적 비용보다 인지적 컨텍스트 유지 비용이 더 클 때, 키보드 중심 워크플로의 이득이 크다.

### 유사 도구와의 차별화

[GeekNews 댓글](https://news.hada.io/topic?id=30268)에서 Homerow가 더 완성도 높은 대안으로 언급됐다.
한국어 입력 호환성 문제와 특정 앱 인식 실패 같은 Shortcat의 한계를 경험한 사용자들이 Homerow로 전환한 사례가 있었다.

두 도구는 동일한 접근성 API를 사용하지만 UI/UX 설계와 엣지 케이스 처리 방식이 다르다.
Shortcat은 커맨드 팔레트 인터페이스를, Homerow는 더 시각적인 힌트 기반 인터페이스를 사용한다.
이 차이는 사용 패턴에 따라 선호도가 갈린다 — 항상 타이핑으로 시작하는 사용자에게는 Shortcat이 자연스럽고, 시각적 확인을 선호하는 사용자에게는 Homerow가 맞을 수 있다.

## 비평

### 맥락 의존성이 도구의 신뢰성을 제한한다

Shortcat의 유용성은 앱이 Accessibility API를 올바르게 구현한 정도에 달려 있다.
Warp 터미널 같은 앱이 인식되지 않는다는 보고는 이 의존성의 한계를 보여준다.
접근성 API를 비표준적으로 구현하거나 아예 구현하지 않은 앱에서는 도구가 동작하지 않는다.

이것은 도구의 결함이 아니라 macOS 생태계의 구조적 특성이지만, 사용자 경험에서는 같은 결과다.
Shortcat에 워크플로를 의존하다가 특정 앱에서 작동하지 않는 것을 발견하면, 다시 마우스로 전환해야 한다.
이 불일관성은 “키보드만으로 완전히 제어”라는 약속의 신뢰도를 낮춘다.

### 한국어 입력 호환성 문제는 비영어권 사용자 경험의 체계적 문제다

1.5.1 버전에서 한국어 입력 관련 문제가 보고된 것은 비영어권 사용자를 위한 설계 우선순위 문제를 보여준다.
퍼지 매칭 알고리즘이 한국어 자모 분리 방식과 충돌하거나, IME(Input Method Editor) 상태에서 단축키가 오동작할 가능성이 있다.

키보드 기반 도구들이 공유하는 이 문제는 대부분 영어 사용자가 개발하고 기본 언어로 영어를 가정하기 때문에 발생한다.
이 문제가 구체적으로 어떤 상황에서 발생하는지, 개발팀이 이를 인식하고 있는지가 비영어권 사용자가 구매를 결정할 때 중요한 정보다.

## 인사이트

### 접근성 기술의 주류화는 보조 기술 생태계 전체를 강화한다

Shortcat 같은 도구가 일반 사용자들에게 Accessibility API의 가치를 보여줄 때, 개발자들이 자신의 앱에서 이 API를 올바르게 구현할 동기가 높아진다.
접근성 지원이 장애 사용자만을 위한 것이 아니라 생산성 도구의 기반이 된다는 인식이 퍼지면, 앱 개발 과정에서 접근성 구현의 우선순위가 높아진다.

이것은 키보드 중심 사용자들이 접근성 생태계 개선의 의도치 않은 수혜자이면서 동시에 기여자가 되는 구조다.
Shortcat이 잘 작동하는 환경을 원하는 사용자들은 접근성 API를 제대로 구현한 앱을 선호하게 되고, 이것이 개발자들에게 시장 신호가 된다.

### 커맨드 팔레트 패러다임의 확산이 GUI 설계에 미치는 영향

Spotlight, Alfred, Raycast, VS Code의 Command Palette, Shortcat — 이 도구들은 모두 “검색으로 시작하는” 인터랙션 패러다임을 공유한다.
이 패러다임이 확산됨에 따라, 기능의 발견 가능성(discoverability)에 대한 GUI 설계의 전통적 가정이 변화한다.

전통적 GUI 설계는 사용자가 메뉴 계층을 탐색하며 기능을 발견한다고 가정했다.
커맨드 팔레트 패러다임은 사용자가 원하는 것을 이미 알고 있다고 가정하고, 검색을 통해 직접 접근한다.
이 두 가정이 공존하는 시기에, 앱들은 둘 다를 지원하는 인터페이스를 설계해야 한다.
Shortcat은 앱이 이 두 번째 패러다임을 지원하지 않을 때 그 간극을 메우는 도구로 위치한다.
