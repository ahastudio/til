# macOS VM은 얼마나 빠르고, 얼마나 작아질 수 있을까?

원문: <https://eclecticlight.co/2026/05/02/how-fast-is-a-macos-vm-and-how-small-could-it-be/>

HN 토론: <https://news.ycombinator.com/item?id=47984852> (269점, 101개 댓글)

## 요약

Mac mini M4 Pro 위에서 실행한 macOS 가상 머신(VM)의 성능과 최소 사양을 측정한 글이다.
macOS 26.4.1 환경에서 5개 가상 코어와 16 GB RAM을 할당한 VM에 Geekbench 6.7.1을 실행했다.

싱글 코어 CPU 성능에서 VM이 호스트의 98%를 달성했다.
멀티 코어는 호스트의 57%로 낮아졌는데, 이는 가상 코어 수(5개)가 물리 코어 수보다 적기 때문이다.
Metal GPU 성능은 호스트의 95%를 달성했고, Neural Engine에서만 유의미한 차이가 발생했다.

최소 사양 실험에서 2개 가상 코어와 4 GB RAM으로도 macOS VM이 “충분히 사용 가능한” 수준으로
작동했다.
실제 메모리 사용량은 3.1 GB이고, Safari와 기본 시스템 유틸리티를 무리 없이 실행했다.

스토리지 관점에서는 100 GB로 할당한 VM이 APFS의 희소 파일 기술 덕분에 실제로는
약 54 GB만 차지했다.
시스템 업데이트를 고려한 현실적인 최솟값은 약 60 GB로 제시된다.

## 분석

### 싱글 코어 98% 효율의 의미

Apple Silicon VM에서 싱글 코어 성능이 호스트의 98%라는 결과는 실용적으로 중요하다.
단일 스레드 성능에 의존하는 대부분의 워크로드—GUI 앱, 컴파일의 직렬 단계,
일반 데스크톱 작업—에서 VM 성능이 사실상 네이티브와 동일하다는 의미다.

이는 x86 가상화와 다른 지점이다.
x86에서는 하이퍼바이저 오버헤드가 싱글 코어 성능에도 눈에 띄게 영향을 미쳤다.
Apple Silicon의 하드웨어 가상화 지원과 macOS 네이티브 하이퍼바이저(Virtualization.framework)가
결합되어 이런 결과가 나온다.

fouc[^fouc]는 코어 수를 줄여가며 메모리 변화를 측정한 결과 “코어마다 일정량의 메모리가
묶인다”는 점을 발견했다.
주로 페이지 캐시와 동시성 처리에 할당되는 메모리로,
가상 코어 수가 성능뿐 아니라 실제 메모리 소비에도 직접 영향을 미친다.

### Neural Engine 격차의 배경

Neural Engine에서 VM과 호스트 간 성능 차이가 크게 발생하는 이유는 가상화 구조에 있다.
Neural Engine 접근은 현재 macOS VM에서 완전히 파이어스루(passthrough)되지 않거나,
가상화 레이어를 통한 접근이 직접 하드웨어 접근보다 현저히 느리다.
이는 ML 추론 워크로드를 VM에서 실행할 때 고려해야 할 현실적 제약이다.

[GeekNews 댓글](https://news.hada.io/topic?id=29112)에서는 Apple Silicon 컨테이너 환경에서
PyTorch GPU 가속을 달성하지 못한 경험이 공유되었다.
HN에서 Havoc[^Havoc]은 M5 Air를 포함한 Apple Silicon에서 “PyTorch, GPU 가속, VM/컨테이너
격리를 동시에 달성하는 것은 사실상 불가능하다”고 밝혔다.
virtio-gpu 레이어가 가장 가깝게 도달하지만 그래픽 GPU만 패스스루되고
컴퓨트 GPU는 패스스루되지 않아 PyTorch가 동작하지 않는다는 것이다.
컴퓨트 GPU 가속이 VM에서 구조적으로 제한됨을 구체적으로 확인해주는 사례다.

### 최솟값 실험의 실용적 함의

4 GB RAM이 macOS VM의 현실적인 최소 요건이라는 발견은
Apple Silicon 저가 라인업의 가능성을 시사한다.
128 MB RAM으로 동작하던 초기 iPhone에서 수십 년 만에 데스크톱 OS VM이
4 GB로 구동된다는 것은 macOS의 메모리 효율이 크게 개선되었음을 보여준다.

이 결과는 맥 미니나 맥북 에어 같은 기본 구성(8 GB)에서도 VM을 실용적으로 운용할 수
있다는 가능성을 열어준다.

## 비평

### 강점

실제 수치를 체계적으로 수집하고 맥락과 함께 제시한다는 점이 이 글의 가치다.
Geekbench 수치만 나열하지 않고 “싱글 코어 98%는 실제로 어떤 의미인가”를 설명한다.
최솟값 탐색 실험도 점진적으로 자원을 줄여가는 방식으로 신뢰성을 높였다.

### 약점

벤치마크 환경 재현에 필요한 세부 정보가 부족하다.
사용한 VM 소프트웨어(UTM, Parallels, 네이티브 Virtualization.framework 등),
호스트 macOS 버전, VM에서 실행한 macOS 버전 등이 글에서 명확히 드러나지 않는다.
“macOS 26.4.1”은 현재 공개된 버전과 맞지 않아 베타 또는 미래 버전으로 보이는데,
이 점이 결과의 일반화 가능성을 불분명하게 만든다.

Neural Engine 성능 격차에 대한 분석이 수치 제시에 그치고, 원인 분석이 없다.
“격차가 크다”는 관찰에서 나아가 왜 그런지를 다루었으면 더 유용한 글이 되었을 것이다.

최솟값 수치도 “부팅 직후 유휴 상태”를 기준으로 했다는 한계가 있다.
nottorp[^nottorp]는 “시작 시 5 GB를 사용하더라도 VM 안에서 앱을 실행하면
할당된 전체 8 GB를 요구하지 않겠느냐”고 물었다.
실제 워크로드에서 메모리 압박이 어떻게 변화하는지는 글에서 다루지 않았다.

## 인사이트

### macOS 가상화가 바꾸는 개발 환경

macOS VM의 성능이 네이티브에 근접한다는 사실은 개발 환경 설계를 바꾼다.
Docker, 컨테이너, Linux VM이 macOS 개발자의 표준 도구가 된 것처럼,
경량 macOS VM이 Apple 플랫폼 빌드 파이프라인의 구성 요소가 될 수 있다.
macOS 전용 CI/CD 인프라 비용을 낮추는 방향으로 작용한다.

[GeekNews 댓글](https://news.hada.io/topic?id=29112)에서 에이전트 개발 환경으로서
macOS VM에 대한 관심이 언급된 것도 이 맥락이다.
AI 에이전트가 macOS 환경에서 실행되어야 하는 작업—Xcode 빌드, Apple 기기 시뮬레이터,
macOS 전용 GUI 자동화—을 격리된 VM에서 안전하게 수행할 수 있다는 것은
에이전트 인프라 설계의 새로운 가능성이다.
HN에서 JasonHEIN[^JasonHEIN]은 “왜 아무도 macOS 전용 에이전트 환경을 만들지
않는지 궁금하다”고 적었다.
Linux 샌드박스가 범용 에이전트에는 충분하지만,
Apple 플랫폼 개발을 자동화하는 에이전트는 macOS 고유의 환경이 필요하다.
이 수요와 macOS VM 성능의 성숙이 만나는 지점에서 새로운 도구들이 나올 것이다.

### 희소 파일과 스토리지 추상화의 가치

100 GB로 할당했지만 54 GB만 사용하는 APFS 희소 파일 동작은
사용자가 VM 크기를 고민할 때 잘못된 기준으로 결정을 내리게 만들 수 있다.
“100 GB VM을 만들면 100 GB가 필요하다”는 직관이 틀린 것이다.

이 추상화는 양날의 검이다.
처음에는 공간을 덜 쓰지만, VM 사용 시간이 늘수록 실제 사용량이 늘어나
예상보다 빠르게 디스크를 채울 수 있다.
VM 용량 계획에서 할당 크기가 아닌 성장 속도와 스냅샷 정책이 더 중요한 변수다.

### Apple Silicon VM 생태계의 성숙

M1 출시 당시 Apple Silicon에서의 macOS 가상화는 지원 범위가 좁고 성능도 불안정했다.
4년이 지나 네이티브의 98% 싱글 코어 성능, 2 코어 4 GB 최솟값이 가능해진 것은
하이퍼바이저 생태계의 성숙을 보여준다.

이 성숙의 수혜자는 개인 사용자보다 기업과 개발팀이다.
macOS 앱 테스트 자동화, 보안 샌드박스, CI 파이프라인에서 VM 채택 장벽이 실질적으로
낮아졌다.
Parallels, UTM, Virtualization.framework 기반 솔루션들이 경쟁하며 생태계를 성숙시키는
과정은 Apple이 의도하지 않았더라도 플랫폼 가치를 높이는 결과를 낳는다.
