# WhatCable — USB-C 케이블 능력치를 읽는 macOS 메뉴바 앱

<https://github.com/darrylmorley/whatcable>

HN 토론: <https://news.ycombinator.com/item?id=47972511> (471점, 135개 댓글)

## 소개

WhatCable은 macOS 메뉴바에서 연결된 USB-C 케이블의 실제 스펙을 표시하는 유틸리티 앱이다. USB 2.0 충전 전용 케이블부터 “240W / 40Gbps Thunderbolt 4”까지, 케이블에 내장된 e-마커(e-marker) 칩을 읽어 실제 지원 속도·전력·제조사를 보여준다. Apple Silicon 전용이며 macOS 14 Sonoma 이상이 필요하다.

개발자가 Claude의 도움을 받아 사용자 피드백에 즉각 반응하며 단 7시간 만에 16개 릴리스를 배포한 사례로도 주목받았다.

## 기능

### 케이블 및 포트 정보

- **케이블 식별**: Thunderbolt/USB4, USB 장치, 충전 전용, 저속 케이블 여부
- **케이블 스펙**: 속도(USB 2.0~40Gbps), 최대 전류, 제조사(e-마커 칩 데이터)
- **충전기 프로파일**: Power Delivery 전압/전류 옵션과 현재 협상된 설정
- **장치 인식**: 연결된 허브, 저장장치, 액세서리 벤더명과 제품 유형
- **대안 모드(Option-click)**: IOKit 속성 원시 데이터 조회

포트가 느린 케이블을 감지하거나 충전 병목을 일으키는 요소(케이블 한계 vs 충전기 협상 제한)를 명확히 구분해 표시한다.

## 설치

```bash
# Homebrew
brew tap darrylmorley/whatcable
brew install --cask whatcable
```

또는 GitHub Releases에서 `WhatCable.zip`을 다운로드해 `/Applications`에 옮기면 된다.

## 사용법

### 메뉴바 앱

실행 후 메뉴바 아이콘을 클릭하면 포트별 케이블·충전기·장치 정보가 표시된다. 우클릭으로 새로 고침, 창 고정, 업데이트 확인, 종료 메뉴에 접근할 수 있다.

### CLI

```bash
whatcable             # 사람이 읽기 쉬운 요약 출력
whatcable --json      # JSON 구조화 출력
whatcable --watch     # 실시간 변경 스트리밍
whatcable --raw       # IOKit 속성 포함
```

독립 설치 시 심볼릭 링크가 필요하다:

```bash
ln -s /Applications/WhatCable.app/Contents/Helpers/whatcable /usr/local/bin/whatcable
```

## 분석

WhatCable이 해결하는 문제는 매우 구체적이다. USB-C 케이블의 생김새는 모두 동일하지만 지원 스펙은 천차만별이다. USB 2.0 충전만 되는 케이블, 10Gbps 데이터 전송 케이블, Thunderbolt 4 케이블이 외형상 구분되지 않는다. 기존에는 케이블 포장을 보거나 별도 테스터 하드웨어를 사야 했지만, e-마커 칩이 내장된 케이블은 소프트웨어로 스펙을 읽을 수 있다.

개발 과정도 흥미롭다. 커뮤니티 피드백을 기반으로 7시간 안에 16개 릴리스를 배포한 것은 AI 보조 개발의 실질적인 사례다. 메뉴바 토글, CLI 지원, 알림 기능이 모두 같은 날 반복적으로 추가됐다. 이는 단순히 빠른 개발이 아니라, 사용자 요청을 즉시 구현해 다시 피드백을 받는 루프 속도가 AI 보조로 인해 기존보다 한 자릿수 빨라진 사례다.

시각 장애 사용자가 별도 하드웨어 테스터 구매 필요성이 없어졌다고 언급한 점도 의미 있다. 접근성 관점에서 소프트웨어 기반 케이블 진단이 하드웨어 대체 도구로 기능한 것이다.

## 비평

### 강점

좁고 깊은 문제 해결이 이 앱의 핵심 강점이다. “USB-C 케이블이 왜 느리게 충전되지?”라는 구체적 불편함을 정확히 겨냥했다. macOS 시스템 API(IOKit)를 직접 활용해 추가 드라이버 없이 동작하는 설계도 깔끔하다.

CLI 모드와 `--json` 옵션 제공이 도구의 활용 범위를 넓힌다. 자동화 스크립트나 CI 환경에서 USB-C 연결 상태를 프로그래밍적으로 확인하는 용도로 쓸 수 있다.

### 약점

Apple Silicon 전용이라는 제한이 크다. Intel Mac 사용자는 대상에서 제외된다. Linux 사용자가 `lsusb` 래퍼 형태로 유사 기능을 요청했지만, e-마커 칩 읽기가 플랫폼별 시스템 API에 의존하므로 크로스플랫폼 지원은 단순한 포팅이 아니다.

e-마커가 없는 저가 케이블은 정보를 노출하지 않는다. 정확히 문제가 되는 케이블(미인증 저가 제품)이 앱으로 식별되지 않는 역설이 존재한다.

ricardobeat는 아마존 판매 케이블 상당수가 e-마커 데이터를 실제 스펙보다 높게 허위 신고한다는 분석을 언급하며, WhatCable이 이를 탐지할 수 있는지 물었다[^ricardobeat]. WhatCable은 케이블이 거짓 정보를 신고하는지 직접 판별하지는 못한다. 그러나 Neywiny가 지적했듯이 호스트·케이블·장치·실제 협상값을 나란히 보여주면, 케이블이 주장하는 스펙과 실제 협상된 결과의 불일치를 간접적으로 감지하는 데 도움이 된다.

일부 사용자는 USB-C 포트를 직접 연결하지 않고 모니터 내장 허브를 통해 장치를 연결하는 경우 "USB-C 포트 없음"이 표시된다고 보고했다[^bkummel]. e-마커 칩 읽기가 직접 연결된 포트 기준으로 동작하므로, 허브 경유 연결은 현재 버전에서 감지되지 않을 수 있다.

## 인사이트

### 하드웨어 불투명성을 소프트웨어가 해결하는 패턴

USB-C 케이블의 혼란은 물리적 표준화와 전기적 표준화가 분리된 결과다. 커넥터 형태가 동일해도 내부 스펙이 달라 사용자에게 불투명한 복잡성이 쌓인다. WhatCable은 이 불투명성을 소프트웨어 레이어로 걷어낸다.

이 패턴은 더 넓은 맥락에서 반복된다. HDMI 케이블의 버전 혼란(2.0 vs 2.1), 네트워크 케이블의 카테고리 표시 부재, 오래된 충전기의 실제 출력 불확실성 등 하드웨어 생태계 전반에 유사한 불투명성이 존재한다. e-마커처럼 표준화된 식별 데이터를 읽을 수 있는 곳에서는 소프트웨어 진단 도구가 하드웨어 테스터를 대체할 수 있다.

### AI 보조 개발이 바꾸는 피드백 루프의 속도

7시간에 16개 릴리스는 이전 시대의 개발 관행으로는 불가능하다. 기능 요청 접수, 구현, 테스트, 빌드, 배포 각 단계에 최소 수십 분씩 소요되는데, 이 모든 것을 시간당 두 번꼴로 반복했다는 것은 LLM이 구현 단계를 실질적으로 단축했다는 의미다.

이 속도는 커뮤니티와의 관계를 바꾼다. 사용자가 요청하고 다음 날 새 버전을 받는 대신, 같은 날 반복해서 업데이트를 받으면 프로젝트에 대한 참여 감각이 달라진다. 오픈소스 개발자 한 명이 커뮤니티 전체와 실시간 대화하며 제품을 만드는 경험이 가능해진다. 단기적으로는 흥미롭지만 장기적으로는 각 릴리스의 테스트 깊이와 안정성에 대한 질문이 남는다.

이 파급 효과는 원작자에게만 그치지 않는다. pimeys는 WhatCable 출시 직후 GPT-5.5를 사용해 10분과 2달러로 KDE Plasma 6 플라즈모이드(Plasmoid)를 만들었다고 공유했다[^pimeys]. 원작자가 AI 보조로 macOS 앱을 만든 지 몇 시간 만에, 다른 사용자가 AI 보조로 Linux 데스크톱 위젯으로 포팅한 것이다. AI 보조 개발이 단일 개발자의 생산성만 높이는 게 아니라, 다른 플랫폼 커뮤니티도 즉시 따라 만들 수 있는 재생산 속도를 만든다.

### 접근성은 기능 추가가 아니라 정보 평등의 문제다

시각 장애 사용자가 WhatCable 덕분에 별도 하드웨어 테스터가 필요 없어졌다는 반응은 접근성의 본질을 짚는다. 접근성은 “특수 사용자를 위한 별도 기능”이 아니라 “모든 사람이 정보에 동등하게 접근할 수 있는가”의 문제다.

USB-C 케이블 진단 하드웨어는 특정 사용자층만이 쉽게 다룰 수 있는 물리적 도구다. 소프트웨어 기반 진단이 이 장벽을 없앤다. 표준 인터페이스(macOS 메뉴바, CLI)로 제공되므로 스크린 리더나 보조 기술과 함께 사용하기도 쉽다. 핀셋이나 멀티미터처럼 정교한 손 조작이 필요한 도구를 소프트웨어가 대체할 때, 가장 먼저 혜택을 받는 사람은 그 물리적 도구를 다루기 어려웠던 사람들이다.
