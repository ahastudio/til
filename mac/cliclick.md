# cliclick

<https://github.com/BlueM/cliclick>

## 소개

cliclick(“Command Line Interface Click”)은 macOS에서 마우스와 키보드 이벤트를
셸/터미널에서 실행할 수 있게 해주는 Objective-C 기반 CLI 도구다.
macOS 10.15 이상에서 동작하며, 독일 개발자 Carsten Blüm이 2013년부터
현재까지 유지보수하고 있다. GitHub 기준 스타 1,950개로 macOS 자동화 도구 중
꾸준한 인지도를 유지하는 프로젝트다.

Homebrew로 간단히 설치할 수 있고(`brew install cliclick`), 동작하려면
시스템 환경설정 → 보안 및 개인정보 → 손쉬운 사용에서 터미널 앱에 컴퓨터 제어 권한을
부여해야 한다. 권한이 없으면 cliclick은 아무 동작도 하지 않으며, 5.1 버전부터는
이 경우 stderr에 경고 메시지를 출력한다.

## CLI

### 마우스 명령

| 명령         | 동작                    |
| ------------ | ----------------------- |
| `c:x,y`      | 좌클릭                  |
| `rc:x,y`     | 우클릭                  |
| `dc:x,y`     | 더블클릭                |
| `tc:x,y`     | 트리플클릭              |
| `m:x,y`      | 마우스 이동             |
| `dd:x,y`     | 드래그 시작(버튼 누름)  |
| `dm:x,y`     | 드래그 이동             |
| `du:x,y`     | 드래그 끝(버튼 놓음)    |

좌표 앞에 `+`/`-`를 붙이면 현재 위치 기준 상대 이동이 된다(`m:+50,+0`).
절대 음수 좌표가 필요할 때는 `=`를 접두사로 쓴다(`c:100,=-200`).
좌표 대신 `.`을 쓰면 현재 마우스 위치를 의미한다.

### 키보드 명령

| 명령          | 동작                                            |
| ------------- | ----------------------------------------------- |
| `kp:key`      | 키 누름(key down + key up)                      |
| `kd:keys`     | 수정자 키 누름(alt, cmd, ctrl, fn, shift)       |
| `ku:keys`     | 수정자 키 놓음                                  |
| `t:text`      | 텍스트 입력(공백 포함 시 따옴표로 감싸야 함)    |

`kp`가 지원하는 키 목록은 방향키, 기능키(F1–F16), 미디어 컨트롤
(`play-pause`, `volume-up`, `mute` 등), 숫자 패드 키(`num-0`~`num-9`),
그리고 `return`, `space`, `tab`, `esc`, `delete` 등 일반 키를 포함한다.

### 유틸리티 명령

| 명령       | 동작                                         |
| ---------- | -------------------------------------------- |
| `w:ms`     | 밀리초 단위 대기                             |
| `p[:str]`  | 현재 마우스 좌표 또는 지정 문자열 출력       |
| `cp:x,y`   | 화면 특정 좌표의 픽셀 색상을 R G B로 출력   |

### 전역 옵션

| 옵션          | 설명                                                   |
| ------------- | ------------------------------------------------------ |
| `-r`          | 완료 후 초기 마우스 위치로 복원                        |
| `-m verbose`  | 각 동작 수행 전 설명 출력                              |
| `-m test`     | 동작은 수행하지 않고 설명만 출력(드라이런)             |
| `-e <num>`    | 마우스 이동 이징(easing) 계수 — 높을수록 인간적 움직임 |
| `-f <file>`   | 파일(또는 stdin)에서 명령 읽기, `#`은 주석            |
| `-w <num>`    | 각 이벤트 후 공통 대기 시간(ms), `w` 명령과 합산됨    |

## 분석

### 설계 철학: 조합 가능한 최소 단위 명령

cliclick의 명령 체계는 UNIX 파이프 철학과 닮아 있다.
단일 명령이 하나의 이벤트만 발생시키고, 여러 명령을 공백으로 이어 붙여
시퀀스를 구성한다(`cliclick c:100,200 w:500 t:hello`).
더 복잡한 자동화 스크립트는 `-f` 옵션으로 파일을 읽어 처리하며,
파일 안에서 `#`으로 주석을 달 수 있어 자동화 스크립트를 문서화하기 좋다.

이 구조는 cliclick 자체에 조건 분기나 변수가 없음을 의미한다.
복잡한 흐름 제어는 셸 스크립트나 Python 같은 외부 언어로 처리하고,
cliclick은 저수준 입력 이벤트 에뮬레이션에만 집중한다.
단일 책임 원칙(Single Responsibility Principle)을 CLI 수준에서 실천한 셈이다.

### Accessibility API 의존성

cliclick은 macOS의 Accessibility API 위에서 작동한다.
이는 두 가지 의미를 가진다.
첫째, 사용자가 수동으로 접근성 권한을 허용해야 하므로 무인(headless) 환경에서
완전한 자동 배포가 불가능하다.
둘째, macOS의 샌드박스나 SIP(System Integrity Protection) 제약이 없는
네이티브 앱 수준의 이벤트를 생성할 수 있어, 대부분의 macOS 앱이 cliclick의
입력을 실제 사용자 입력과 구별하지 못한다.

로그인 화면 제어가 불가능한 것은 이 API 아키텍처의 직접적 결과다.
Accessibility API는 세션 내 사용자 공간에서만 동작하므로,
로그인 전 시스템 레벨 UI는 접근할 수 없다.
Citrix Viewer 클릭 불인식 문제도 Citrix가 자체 렌더링 레이어를 쓰기 때문에
발생하는 구조적 한계다.

### `-e` 이징 옵션: 인간 흉내 내기

`-e` 옵션(easing factor)은 단순한 편의 기능처럼 보이지만 흥미로운 함의를 가진다.
기본값 0은 마우스가 순간 이동(teleport)하는데, 이는 자동화 감지 시스템에
비자연적 움직임으로 잡힐 수 있다.
이징 값을 높이면 마우스가 곡선 경로를 따라 이동 거리에 비례한 시간을 소요한다.
이 기능은 cliclick이 단순 UI 테스트뿐 아니라 자동화 감지 우회 시나리오에서도
쓰일 수 있음을 시사한다.

## 비평

### 강점: 2013년부터 살아남은 단순함

cliclick의 가장 큰 강점은 12년 이상 같은 API로 유지돼 왔다는 것이다.
Homebrew, AppleScript, Python의 pyautogui, Hammerspoon 등 여러 macOS 자동화
도구가 존재하지만, 셸 한 줄로 마우스 이벤트를 발생시키는 수준에서
cliclick만큼 간단한 인터페이스를 가진 도구는 드물다.
테스트 모드(`-m test`)와 드라이런 지원도 운영 환경에서 신뢰성 있게 쓸 수 있게 한다.

`cp:x,y`로 픽셀 색상을 읽는 기능은 눈에 띄지 않지만 실용적이다.
이미지 인식 없이 화면 특정 지점의 색상 변화를 폴링해 상태 변화를 감지하는
간단한 시각적 피드백 루프를 만들 수 있다.

### 약점: 절대 좌표 의존성

cliclick의 가장 큰 실용적 약점은 절대 좌표 의존성이다.
`c:512,300`이라고 쓴 스크립트는 해상도나 화면 배치가 달라지면 즉시 깨진다.
창이 다른 위치로 이동해도 같은 문제다.
이 문제를 해결하려면 `p:.`로 현재 위치를 먼저 확인하거나,
외부 도구(예: AppleScript의 `position of window`)로 좌표를 계산해
cliclick에 전달하는 래퍼 스크립트가 필요하다.

대안적으로 Hammerspoon은 앱과 창을 이름으로 쿼리하고 클릭 대상을 찾을 수 있어
좌표 의존성을 추상화할 수 있다. cliclick은 이 수준의 추상화를 제공하지 않는다.
단순함이 강점인 동시에 한계다.

### 미해결 영역: 레티나·다중 모니터

README는 레티나 디스플레이에서 논리 픽셀과 물리 픽셀의 관계를 명시적으로 다루지
않는다. Apple Silicon Mac과 외부 4K 모니터를 혼용하는 환경에서 좌표 체계가
어떻게 동작하는지 문서에서 확인하기 어렵다. 이는 실제 사용자가 맞닥뜨리는
가장 흔한 혼란 지점임에도 불구하고 README에서 빠져 있다.

## 인사이트

### CLI는 언제 “좋은 접착제”가 되는가

cliclick은 macOS 자동화의 복잡한 레이어를 전혀 추상화하지 않고도
수천 명의 사용자를 확보했다. 이유가 있다.
자동화 파이프라인에서 가장 가치 있는 도구는 가장 강력한 도구가 아니라
가장 잘 조합되는 도구다.
셸 스크립트, CI 파이프라인, Python 래퍼, Makefile — 어느 레이어에서도
cliclick은 `cliclick c:100,200`이라는 한 줄로 불러낼 수 있다.

이 조합 가능성(composability)은 도구 설계의 핵심 미덕이다.
Hammerspoon은 더 강력하지만 Lua를 배워야 하고 설정 파일을 관리해야 하며
다른 도구와 연결하려면 별도 인터페이스가 필요하다.
AppleScript는 macOS 네이티브지만 문법이 어색하고 파이프 친화적이지 않다.
cliclick은 그 어떤 학습 비용도 없이 POSIX 파이프 안에 자연스럽게 녹아든다.
이것이 12년을 살아남은 이유다.

2차 효과로, 이 단순함은 cliclick을 AI 에이전트와 브라우저 자동화의 시대에
의외의 관련성을 얻게 한다.
LLM이 macOS를 제어하려 할 때, Accessibility API를 직접 호출하는 복잡한 바인딩보다
`cliclick c:x,y`라는 셸 명령 하나가 훨씬 안정적인 인터페이스가 된다.
단순함은 미래의 통합 비용을 낮춘다.

### 접근성 권한이라는 신뢰 게이트

cliclick이 동작하려면 반드시 사람이 한 번 접근성 권한을 허용해야 한다.
이것은 보안 관점에서 중요한 설계 결정이다.
macOS는 화면 입력을 에뮬레이션하는 능력을 가장 민감한 권한 중 하나로 분류한다.
이유는 명백하다 — 이 권한을 가진 프로세스는 사용자가 할 수 있는 모든 클릭과
키 입력을 대신 수행할 수 있다.

이 신뢰 게이트는 cliclick이 악성 코드로 자동 악용되는 시나리오를 막는
실질적 장벽이다. 하지만 동시에 자동화된 CI/CD 환경에서 cliclick을 활용하는
데 마찰을 만들기도 한다. 특히 GitHub Actions 같은 클라우드 CI 환경에서
macOS 러너에 접근성 권한을 자동 부여하는 것은 까다롭다.
이 트레이드오프는 해결책이 없는 근본적 긴장이다 — 보안과 자동화 편의성은
이 지점에서 충돌한다.

역설적으로, 이 제약 때문에 cliclick은 개발자의 로컬 머신 자동화에
특화된 도구로 남는다. 이는 프로젝트의 범위를 자연스럽게 정의한다.
서버 측 자동화나 헤드리스 테스트가 아닌, 개발자가 직접 보고 확인하는
데스크톱 워크플로우 자동화. 이 명확한 범위는 도구가 오래 살아남는 이유 중 하나다.

### “테스트 모드”가 가르쳐주는 것

cliclick의 `-m test` 옵션은 표면적으로는 단순한 드라이런 기능이다.
그러나 이 기능이 포함된 것은 이 도구가 설계될 때 “되돌리기 어려운 동작”에 대한
의식이 있었음을 보여준다. 마우스 클릭은 undo가 없다.
잘못된 좌표로 클릭 스크립트를 실행하면 열어서는 안 될 창이 열리거나,
삭제 확인 다이얼로그에서 OK를 눌러버릴 수 있다.

이 통찰은 일반적인 자동화 도구 설계 원칙으로 확장된다.
“실행하기 전에 무엇을 할지 보여줄 수 있는가?”라는 질문은
Terraform의 `plan`, Ansible의 `--check`, Kubernetes의 `--dry-run`으로 이어지는
동일한 패턴이다. 비가역적 부작용을 가진 모든 도구는 드라이런 모드가 필요하다.
cliclick이 2013년에 이미 이 패턴을 구현한 것은 주목할 만하다.

더 넓게 보면, 이것은 “관찰 가능성(observability)이 안전의 전제”라는 원칙의
작은 실례다. `-m verbose` 모드로 각 동작을 stdout에 남기는 것,
`p:.`로 현재 위치를 확인하는 것, `-r`로 초기 위치를 복원하는 것 —
이 모든 기능이 자동화 스크립트를 디버그하고 신뢰할 수 있는 도구로 만드는
관찰 가능성 레이어를 형성한다.
