# 알한글 — macOS 네이티브 HWP/HWPX 유틸리티

<https://github.com/postmelee/alhangeul-macos>

## 소개

알한글(alhangeul-macos)은 HWP/HWPX 파일을 macOS에서 미리보고, 앱에서 열고, 저장하고, 공유하고, PDF로 내보낼 수 있게 하는 오픈소스 데스크톱 앱이다.
“Mac에서 한글 파일은 더 이상 이방인이 아닙니다”라는 슬로건이 말해주듯, Finder 단계부터 HWP 파일을 macOS 생태계에 통합하는 것이 핵심 목표다.
파일을 외부 서버에 업로드하지 않고 로컬에서 처리한다는 것을 기본 원칙으로 삼는다.

Rust 기반 [`rhwp`](https://github.com/edwardkim/rhwp) 코어를 macOS 앱, Quick Look 프리뷰, Finder 썸네일, Swift 브리지로 연결하는 구조다.
“닫힌 HWP/HWPX 문서를 더 많은 환경에서 다룰 수 있게 한다”는 `rhwp`의 방향을 Mac 네이티브 경험으로 확장한다는 철학을 명시한다.
현재 공개 릴리즈는 `v0.1.3`이며, `brew install --cask postmelee/tap/alhangeul`로 설치할 수 있다.

앱 이름 “알한글”의 “알(R)”은 `rhwp` 메인테이너가 “알(R), 모두의 한글”이라는 철학에서 붙인 이름이다.
[GeekNews 댓글](https://news.hada.io/topic?id=29692)에서 이 이름이 잘 알려진 보안 소프트웨어 브랜드와 혼동될 수 있다는 우려가 제기되었지만, `rhwp` 메인테이너가 부여한 명칭이라는 배경 설명이 이어졌다.
`cadenzah`를 비롯한 여러 사용자가 프로젝트의 완성도를 높이 평가했고, `jhk0530`은 “안녕히 계세요... 한컴”이라는 반응을 남겼다.

## 아키텍처

현재 릴리즈 라인(`v0.1.x`)은 WebView-backed 구조다.
앱의 뷰어/에디터는 `WKWebView`를 통해 `rhwp-studio`를 렌더링하며, Finder Quick Look과 썸네일, PDF 내보내기는 Swift/Rust 브리지 렌더링 경로를 사용한다.

```text
rhwp (Rust 코어)
    ├── macOS 앱 (Swift + WKWebView)
    ├── Quick Look extension
    ├── Finder Thumbnail extension
    └── Swift bridge
```

각 렌더링 경로가 다를 수 있다는 점을 명시적으로 인정한다.
앱 화면의 뷰어와 Finder Quick Look, PDF 내보내기가 완전히 동일한 시각 결과를 보장하지는 않는다.
장기적으로는 `WebView` 의존도를 줄이고 Rust/rhwp Skia 렌더러와 Swift 편집 UI를 결합한 native macOS 뷰어/에디터로 전환하는 것을 목표로 한다.

## 주요 기능

현재 `v0.1.x` 라인에서 제공하는 기능은 다음과 같다.

- `.hwp`, `.hwpx` Quick Look 프리뷰 및 Finder 썸네일
- `WKWebView` 기반 뷰어/에디터 (찾기, 선택, 복사, 기본 편집)
- HWP 저장 및 다른 이름으로 저장
- PDF 내보내기, 인쇄, macOS 공유 시트
- Signed/notarized universal DMG 배포 (Intel + Apple Silicon)
- Sparkle 업데이트 확인

현재 제한 사항으로는 HWPX 직접 저장 미지원(HWP export 경로 사용), 대용량·손상 문서의 부분 렌더링 미보장, CoreGraphics/CoreText 기반 렌더러의 스타일·이미지 처리 parity gap 등이 있다.

## 분석

### 오픈소스가 채워야 하는 공백

한컴오피스는 독점 포맷 HWP를 통해 대한민국 공공기관과 교육 현장에 깊이 뿌리내렸다.
문제는 이 포맷이 macOS에서 사실상 지원되지 않는다는 것이다.
공식 앱이 없거나 있더라도 수십만 원에 달하는 유료 소프트웨어를 설치해야 했다.
이 공백이 `rhwp` 프로젝트와 알한글 같은 오픈소스 이니셔티브를 탄생시킨 배경이다.

### 렌더링 엔진의 선택

Rust 기반 `rhwp`를 코어로 삼은 것은 성능과 안전성 측면에서 의미 있는 선택이다.
HWP 파일 파싱은 스펙이 완전히 공개되지 않은 복잡한 형식을 다뤄야 하며, 손상된 파일이나 예외적인 문서 구조에 대한 robust한 처리가 필요하다.
Swift로 직접 구현하는 대신 이미 검증된 Rust 코어를 브리지하는 전략은 단기 개발 비용과 장기 유지보수 안정성 사이의 합리적인 트레이드오프다.

## 비평

### 강점: 솔직한 제한 사항 문서화

많은 오픈소스 프로젝트가 자신의 한계를 축소하거나 모호하게 서술하는 데 반해, 알한글은 렌더링 경로의 불일치, HWPX 저장 제한, CoreGraphics parity gap 등을 README에 명시적으로 기술한다.
이는 사용자 기대를 적절하게 조율하고, 기여자들에게 명확한 작업 대상을 제시하는 좋은 관행이다.

### 약점: 단독 메인테이너 리스크

프로젝트가 단일 메인테이너(`postmelee`)에 의존하고 있다는 것은 지속 가능성 측면에서 잠재적 위험 요소다.
공공 문서 포맷을 다루는 유틸리티로서의 사회적 필요와, 개인 프로젝트로서의 현실적 한계 사이의 간극이 존재한다.
`v2.0`의 “Agent-ready 문서 환경”이라는 야심찬 목표가 실현되려면 커뮤니티 기여의 확대가 필수적이다.

## 인사이트

### 독점 포맷은 생태계의 채무다

HWP 포맷이 20년 이상 대한민국 공공 문서 표준으로 자리 잡으면서, 이 포맷을 읽지 못하는 환경은 사실상 특정 행정 서비스에서 배제되는 것을 의미했다.
독점 포맷이 만드는 이 장벽은 단순한 불편함을 넘어 디지털 접근성의 구조적 불평등이다.
macOS 사용자, Linux 사용자, 또는 해외에서 한국 문서를 열어야 하는 사람들이 이 비용을 감당해왔다.

알한글 같은 프로젝트가 등장한다는 것은 이 채무를 오픈소스 공동체가 갚아나가고 있다는 의미다.
그러나 이것이 근본적인 해결책이 아님도 분명하다.
문서 포맷이 공개 표준이 아닌 이상, 오픈소스 구현은 항상 리버스 엔지니어링과 불완전한 호환성이라는 한계 안에서 작동한다.

장기적 해결책은 공공 문서 포맷의 개방 표준화다.
정부가 ODF(Open Document Format)나 HWPX 공개 스펙 제공으로 전환한다면, 알한글 같은 프로젝트의 개발 부담은 대폭 줄어들 것이다.

### 플랫폼 네이티브 경험의 비가시적 가치

macOS에서 스페이스바를 누르면 파일 내용이 미리 보이는 Quick Look은 사소한 기능처럼 보이지만, 실제로는 파일 관리 경험의 상당 부분을 결정한다.
HWP 파일이 Quick Look을 지원하지 않는다는 것은 Finder에서 파일을 구별하기 위해 매번 앱을 열어야 한다는 의미다.
이 마찰이 쌓이면 사용자는 HWP 파일을 피하게 되거나, 변환 도구를 거치게 된다.

플랫폼 통합의 가치는 기능 목록에 나타나지 않는다.
썸네일, Quick Look, 공유 시트, Spotlight 검색 같은 통합은 앱이 플랫폼의 일부처럼 느껴지게 만드는 요소들이다.
알한글이 `v0.2`와 `v0.3`에서 Spotlight 통합과 Shortcuts 연동을 목표로 삼은 것은 단순한 기능 추가가 아니라, 파일 포맷을 플랫폼 생태계에 완전히 통합시키는 작업이다.

이 방향이 완성될 때, HWP 파일은 macOS에서 “이방인”이 아닌 일급 시민이 된다.

### Rust + Swift 조합이 보여주는 시스템 소프트웨어의 미래

알한글의 아키텍처는 Rust가 저수준 파일 파싱과 렌더링 코어를 담당하고 Swift가 macOS UI와 플랫폼 통합을 담당하는 명확한 역할 분리를 보여준다.
이 조합은 시스템 소프트웨어 개발에서 점점 더 일반적인 패턴이 되고 있다.
Rust의 메모리 안전성과 성능, Swift의 플랫폼 친화성과 개발 생산성이 서로의 약점을 보완한다.

특히 HWP 파싱처럼 불완전한 스펙과 악성 파일 가능성을 다루어야 하는 영역에서 Rust의 메모리 안전성은 중요한 보안 이점이다.
취약한 파서는 악성 문서를 통한 코드 실행의 공격 벡터가 될 수 있으며, 이는 공공 문서를 다루는 유틸리티에서 특히 민감한 문제다.
Rust 코어의 선택은 단순한 성능 최적화가 아닌 보안 설계 결정이기도 하다.
