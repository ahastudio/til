# Whispree: 한국어 개발자를 위한 음성 입력 macOS 앱

<https://github.com/Arsture/whispree>

## 요약

Whispree는 한국어-영어 코드스위칭에 특화된 무료 오픈소스 macOS 음성 입력 앱이다.
STT(Speech-to-Text)에 LLM 기반 교정을 결합하여, 기존 음성 인식의 한국어 기술
용어 오인식 문제를 해결한다. 예를 들어 "GitHub에 PR 올려놨어"가 기존 도구에서
"깃펍 펄 올렸어"로 인식되는 문제를 교정한다.

핵심 기능은 단축키(기본 Ctrl+Shift+R)로 음성 입력을 시작하면, STT + LLM 교정을
거쳐 원래 커서 위치에 텍스트를 삽입하는 것이다. 타이핑 대비 3-5배 빠르다고
주장한다. 녹음 시작 시 스크린샷을 자동 캡처하여 시각적 맥락을 교정에 활용하는
Visual Context 기능도 있다.

STT는 Groq(클라우드, 무료), WhisperKit(로컬, ~1.5GB), MLX Audio(로컬, ~1.0GB)를
지원하고, LLM은 OpenAI, Qwen3 시리즈(로컬), GLM-4.7-Flash(로컬)를 지원한다.
교정 모드는 기본, 필러 제거, 구조화(음성을 불릿 포인트로 정리), 커스텀이 있다.

macOS 14+(Sonoma), Apple Silicon 전용이며, Swift로 작성되었다. MIT 라이선스로
Homebrew를 통해 설치할 수 있다. 대학생 개발자의 프로젝트다.

## 분석

Whispree가 해결하는 문제는 구체적이고 실재한다. 한국어 개발자는 일상적으로
한국어와 영어 기술 용어를 섞어 쓴다. 기존 STT는 이 코드스위칭을 처리하지
못한다. "밸리데이션"을 "validation"으로, "리액트"를 "React"로 교정하는 것은
단순한 음성 인식 너머의 도메인 지식이 필요하다.

아키텍처 설계가 영리하다. STT와 LLM을 분리하여 각각 독립적으로 공급자를
선택할 수 있게 했다. 무료 클라우드 STT(Groq)와 기존 OpenAI 구독을 조합하면
추가 비용 없이 사용할 수 있고, 프라이버시가 중요하면 모두 로컬로 전환할
수 있다. 이 모듈성은 사용자의 다양한 요구를 하나의 앱에서 수용한다.

Visual Context 기능은 특히 주목할 만하다. 녹음 시작 시 화면을 캡처하여
LLM에 함께 전달하면, "이 함수 수정해"라는 모호한 음성이 화면에 보이는
코드의 맥락 안에서 해석될 수 있다. 이것은 멀티모달 AI의 실용적 응용이다.

Quick Fix(Ctrl+Shift+D)로 즉석에서 도메인 용어 사전을 구축하는 것도
개발자 워크플로우에 대한 이해를 보여준다. 매번 같은 용어를 오인식하는
짜증을 한 번의 키 입력으로 해결한다.

## 비평

강점은 명확한 문제 정의와 타겟 사용자 이해다. "한국어 개발자의 음성 입력"이라는
구체적인 니치에 집중한 덕분에, 범용 음성 입력 앱이 해결하지 못하는 문제를
정확히 해결한다. GeekNews에서 기존 Handy 앱 사용자가 전환을 시도하겠다는
댓글은 실제 수요를 방증한다.

한계도 있다. 첫째, Apple Silicon + macOS 14 전용이라는 제약이 크다.
Windows와 Linux 개발자는 사용할 수 없고, 인텔 맥도 지원되지 않는다.
타겟 사용자(한국어 개발자) 중 macOS + Apple Silicon 비율이 높다고는
해도, 상당수를 배제한다.

둘째, LLM 교정의 지연 시간이 사용 경험에 미치는 영향이 불명확하다.
STT + LLM 두 단계를 거치므로, 특히 로컬 모델 사용 시 체감 지연이
있을 수 있다. "3-5배 빠르다"는 주장의 측정 조건이 제시되지 않았다.

셋째, 프로젝트 지속성이 불확실하다. 대학생 1인 개발자의 프로젝트로,
37 스타, 7 포크 수준이다. 장기적 유지보수와 macOS 버전 업그레이드
대응에 대한 부담이 있다.

## 인사이트

### STT + LLM 파이프라인: 음성 인식의 새로운 아키텍처 패턴

전통적으로 음성 인식은 단일 모델이 오디오를 텍스트로 직접 변환하는
end-to-end 구조였다. Whispree는 이를 두 단계로 분리한다. STT가 "충분히
좋은" 초벌 텍스트를 만들고, LLM이 도메인 지식으로 교정한다.

이 분리는 단순해 보이지만 근본적인 패러다임 전환이다. STT 모델을
처음부터 한국어 기술 용어에 파인튜닝하는 것은 비용이 크고, 용어가
변할 때마다 재학습이 필요하다. 반면 LLM은 프롬프트나 사전 업데이트만으로
새로운 용어에 적응한다. 문제를 "인식"과 "이해"로 분리함으로써,
각 단계의 최적화가 독립적으로 가능해진다.

이 패턴은 음성 입력을 넘어 일반화할 수 있다. OCR + LLM(문서 인식 후
맥락 교정), 코드 생성 + LLM(생성 후 스타일 교정) 등 "생성 + 교정"
파이프라인은 단일 모델보다 유연하고 적응적인 접근이다.

### 코드스위칭이라는 미해결 AI 과제

Whispree의 존재 자체가 현재 AI의 한계를 드러낸다. Whisper 같은
세계적 수준의 STT 모델도 한국어-영어 코드스위칭을 제대로 처리하지
못한다. 이것은 한국어만의 문제가 아니다. 힌디어-영어(Hinglish),
스페인어-영어(Spanglish) 등 전 세계의 이중 언어 사용자가
같은 문제를 겪는다.

코드스위칭은 단순한 언어 전환이 아니라 사회언어학적 현상이다.
화자가 어떤 단어에서 언어를 전환할지는 맥락, 도메인, 대화
상대에 따라 달라진다. 현재의 STT 모델은 언어 식별을 발화 단위로
하지만, 코드스위칭은 단어, 심지어 형태소 단위에서 일어난다.

이것이 LLM 교정이 필요한 근본 이유이며, 동시에 LLM 교정만으로는
완벽할 수 없는 이유이기도 하다. "밸리데이션"이 validation인지
아닌지는 발화 맥락에 달려 있다. 이 문제의 근본적 해결은 STT
모델 자체가 다국어 코드스위칭을 학습 데이터에 포함해야 가능한데,
이런 데이터셋은 아직 체계적으로 구축되지 않았다.

### 1인 개발자 프로젝트의 전략적 니치 선택

Whispree의 개발자는 "모든 사람을 위한 음성 입력"이 아니라 "한국어
개발자의 macOS 음성 입력"이라는 극도로 좁은 니치를 선택했다.
이것은 제약이 아니라 전략이다.

범용 음성 입력 시장은 Apple(Dictation), Google, Microsoft, 그리고
Superwhisper 같은 스타트업이 이미 점유하고 있다. 1인 개발자가
이 시장에서 경쟁하는 것은 불가능하다. 그러나 "한국어 + 영어
기술 용어 + macOS"라는 교집합에서는 대기업이 움직이지 않는다.
시장이 너무 작기 때문이다.

이 패턴은 오픈소스 생태계에서 반복된다. 대기업 제품의 에지
케이스가 니치 프로젝트의 존재 이유가 된다. 문제는 이 니치가
대기업의 모델이 개선되면 소멸할 수 있다는 것이다. Whisper의
다음 버전이 코드스위칭을 잘 처리하면 Whispree의 핵심 가치
제안이 약해진다. 그때까지 커뮤니티와 기능적 해자를 쌓을 수
있는지가 프로젝트의 장기 생존을 결정한다.
