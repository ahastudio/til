# Trees — 파일 트리 렌더링 라이브러리

https://trees.software

## 소개

Trees는 VSCode 파일 탐색기와 같은 파일/디렉터리 트리 UI를 렌더링하기 위한
오픈소스 라이브러리다. 에디터, 코드 호스팅 도구, 개발 환경 사이드바 구현을
주요 사용 사례로 삼으며, 성능과 커스터마이징 유연성을 핵심 가치로 내세운다.

## 주요 기능

수천 개 항목을 효율적으로 처리하는 **자동 가상화(auto-virtualization)**가
내장되어 있어 대규모 파일 트리에서도 렌더링 성능이 유지된다.

Git 상태 배지(status badge), 드래그 앤 드롭, 검색/필터링, 커스텀 컨텍스트 메뉴를
기본 제공한다. 단일 자식 폴더를 하나의 경로로 납작하게 펼치는(tree flattening)
기능과, Git 변경 항목에 점 마커를 표시하는 기능도 포함된다.

WCAG 2.1 접근성 표준을 준수하며, 키보드 내비게이션과 ARIA 속성을 지원한다.
CSS 변수로 스타일링을 세밀하게 제어할 수 있어 브랜드나 에디터 테마에 맞춘
커스터마이징이 쉽다.

## 분석

### 왜 이런 라이브러리가 따로 필요한가

파일 트리는 단순해 보이지만 구현하기 까다로운 UI 컴포넌트다. 수만 개의
노드가 있어도 스크롤이 부드러워야 하고(가상화), 드래그 앤 드롭 UX는
정교한 상태 관리를 요구하며, Git 상태 표시는 비동기 데이터 갱신을 처리해야
한다. 이를 처음부터 구현하면 상당한 공수가 든다.

기존 범용 트리 라이브러리들은 파일 탐색기에 특화된 기능(Git 통합, hover 시
폴더 확장, 경로 납작화)을 제공하지 않는다. Trees는 이 틈새를 메운다.

### 오픈소스 UI 컴포넌트의 흐름

VSCode가 오픈소스로 공개된 이후, 개발자 도구 UI의 표준은 VSCode의 룩앤필이
되었다. 파일 탐색기, 터미널, 에디터 탭 등 VSCode가 만들어낸 UX 패턴을
자신의 도구에도 재현하고 싶다는 수요가 증가했고, Trees는 그 수요에 직접
대응한다.

## 비평

### 장점

라이브러리가 해결하려는 문제가 명확하고, 개발자 도구 생태계에서 실제로
필요한 컴포넌트를 다룬다. WCAG 2.1 준수와 CSS 변수 기반 커스터마이징은
실용적인 설계 결정이다.

### 한계

소스 URL(trees.software)이 npm 패키지나 GitHub 저장소를 즉시 제공하지 않아
실제 채택 경로가 불분명하다. 라이브러리의 성숙도, 번들 크기, 프레임워크
의존성(React 전용인지 vanilla인지) 등 도입 전 검토해야 할 정보가 공개된 내용만으로는
충분하지 않다.

## 인사이트

### VSCode의 UI 컨벤션은 새로운 표준이 되었다

IDE UI가 사실상 VSCode 하나로 수렴한 결과, 개발자들은 파일 트리의 동작
방식에 대한 매우 구체적인 기대를 갖게 되었다. 단일 자식 폴더 납작화,
Git 변경 파일의 색상 강조, 드래그 앤 드롭으로 파일 이동 — 이것들은 더
이상 “있으면 좋은” 기능이 아니라, 사용자가 당연히 기대하는 기준점이다.

새 도구를 만들 때 이 기대를 충족하지 못하면 즉시 “이 도구는 덜 완성됐다”는
인상을 준다. Trees 같은 라이브러리의 존재는 이 기준을 개별 팀이 처음부터
구현하지 않아도 충족할 수 있게 해준다.

### 가상화는 선택이 아니라 필수

수천 개의 DOM 노드를 실제로 렌더링하는 나이브한 구현은 2025년 기준으로도
여전히 문제가 된다. 가상화(virtualization) — 뷰포트에 보이는 항목만 실제로
렌더링하는 기법 — 는 대규모 목록 UI에서 성능을 유지하는 핵심이다.

Trees가 이를 “auto-virtualization”으로 기본 내장한다는 점은 중요하다.
사용자가 성능 튜닝을 신경 쓰지 않아도 되도록 합리적인 기본값을 제공하는
것이 좋은 라이브러리 설계의 표시다. 복잡성을 라이브러리 안에 숨기고 단순한
인터페이스를 노출하는 것이 관건이다.

### 접근성은 개발자 도구에서도 협상 불가

WCAG 2.1 준수와 키보드 내비게이션 지원을 명시적으로 언급한다는 것은 의미
있다. 개발자 도구는 종종 접근성을 간과한다 — “개발자는 마우스를 쓸 줄 안다”는
가정 아래. 그러나 키보드만으로 개발하는 사용자, 스크린 리더 사용자, 운동
장애가 있는 개발자들에게 이는 결코 선택적 사항이 아니다. 접근성을 라이브러리
레벨에서 보장하면, 이를 사용하는 모든 애플리케이션이 자동으로 혜택을 받는다.
