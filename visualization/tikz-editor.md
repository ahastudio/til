# TikZ Editor

<https://tikz.dev/editor>

<https://github.com/DominikPeters/tikz-editor>

HN 토론: <https://news.ycombinator.com/item?id=48645437> (442점, 74개 댓글)

## 소개

TikZ Editor는 Dominik Peters가 만든 LaTeX TikZ 다이어그램용 WYSIWYG 편집기다.
TikZ 코드를 시각적으로 파싱하고, 캔버스에서 요소를 드래그하거나 변형하면
소스 코드가 실시간으로 업데이트된다.
기존 줄 바꿈이나 공백 같은 포맷은 그대로 유지된다.

웹 앱(tikz.dev/editor)과 데스크탑 앱(macOS, Windows, Linux) 두 가지 형태로 제공된다.
MIT 라이선스로 공개된 무료 오픈소스 프로젝트이며,
2026년 2월 공개 후 약 4개월 만에 GitHub 스타 304개를 기록했다.

## 주요 기능

**캔버스 편집**:
25종 이상의 내장 도형(사각형, 원, 타원, 다이아몬드, 폴리곤, 화살표 등)과
선, 곡선, 호, 격자, Bézier 곡선을 지원한다.
자유 드로잉과 제어점 편집도 가능하다.

**실시간 소스 편집기**:
TikZ 구문 하이라이팅, 자동 완성, 숫자 스크러빙(number scrubbing)을 제공한다.
숫자 위에 커서를 올려 드래그하면 값이 변경된다.

**양방향 동기화**:
캔버스 편집과 소스 코드 편집이 실시간으로 서로 반영된다.

**가져오기/내보내기**:
SVG, IPE, PPTX를 가져올 수 있고, SVG, PDF, PNG로 내보낼 수 있다.

**데스크탑 앱 전용 기능**:
네이티브 파일 다이얼로그, 시스템 클립보드 통합, TikZ 도움을 위한 AI 어시스턴트, 자동 업데이트.

**지원하는 TikZ 구문**:
`\foreach` 루프, 트리(child operations, tree layout, level/sibling 스타일링),
행렬(matrix nodes), 색상, 선 스타일, 채우기 패턴, 셰이딩, 변환 등.
데코레이션, 그래프, 플롯은 부분 지원이고 `let` 연산은 미구현이다.

## 아키텍처

TypeScript로 작성되었으며, 핵심 라이브러리는 4단계 파이프라인으로 구성된다.

1. **Parser** — TikZ를 진단 정보가 있는 손실 없는(lossless) AST로 파싱
2. **Semantic Evaluator** — 스타일, 변환, 좌표, 경로 의미를 씬 그래프(scene graph)로 해석
3. **SVG Backend** — 씬 요소에서 순수 SVG를 생성
4. **Render API** — 소스 → SVG 엔드투엔드 오케스트레이션

앱은 두 가지다.

- **웹 앱**: Vite + React
- **데스크탑 앱**: Tauri v2 (Rust 기반)

테스트 전략으로는 기능 매트릭스 테스트, PGF 매뉴얼 소스 파일을 활용한 코퍼스 회귀 테스트,
Playwright 기반 웹 E2E 테스트, 데스크탑 E2E 테스트,
그리고 TeX 레퍼런스 출력과 자체 SVG 출력을 비교하는 렌더러 비교 스크립트를 사용한다.

## 분석

### 손실 없는 AST가 양방향 동기화를 가능하게 한다

TikZ Editor의 기술적 핵심은 손실 없는(lossless) AST 파서다.
일반적인 파서는 코드에서 의미 있는 구조만 추출하고 공백, 줄 바꿈, 주석을 버린다.
손실 없는 파서는 소스 코드의 모든 토큰을 AST에 보존한다.

이 선택이 양방향 동기화를 가능하게 한다.
캔버스에서 노드 위치를 바꾸면 AST에서 해당 좌표 토큰만 수정하고,
나머지 코드는 원본 그대로 재조합해 출력한다.
기존 라인 포맷, 들여쓰기, 주석이 유지되는 것은 이 설계의 결과다.

반대로, 소스 코드를 직접 수정하면 변경된 토큰만 다시 파싱하고
씬 그래프를 업데이트해 캔버스에 반영한다.
손실 없는 AST는 양방향 동기화의 기반이며,
이것이 없으면 코드와 캔버스는 독립적인 두 표현이 되어 선택적으로 하나만 사용해야 한다.

커뮤니티에서도 이 관점을 지지하는 반응이 있었다.
Littice는 “나에게 킬러 피처는 TikZ를 시각적으로 그리는 것이 아니라,
기존 TikZ 코드를 건드릴 때 자동 생성된 것처럼 보이지 않는다는 점”이라고 지적했다.[^Littice]
저자도 이를 설계 원칙으로 명시했다.
Inkscape에서 SVG를 열고 최소한의 수정 후 저장하면 원본과 거의 관계없는 파일이 나오는데,
TikZ Editor는 정반대를 목표로 한다.[^DominikPeters-inkscape]

### 독자적인 TikZ SVG 렌더러 구현의 도전

LaTeX TikZ를 브라우저에서 시각화하는 가장 쉬운 접근법은
LaTeX를 서버에서 컴파일하고 결과 PDF를 PNG나 SVG로 변환하는 것이다.
TikZ Editor는 이 접근을 취하지 않았다.
대신 TikZ 구문을 직접 파싱하고 독자적인 SVG 렌더러를 구현했다.

이 선택은 인터랙티브성을 위한 것이다.
서버 컴파일 방식은 변경마다 네트워크 왕복이 필요하고,
렌더링 결과가 래스터 이미지라 요소 수준에서 상호작용할 수 없다.
독자적인 SVG 렌더러는 완전히 로컬에서 동작하며 밀리초 단위 반응 속도를 낸다.

그러나 이 선택의 비용은 구현 범위다.
TikZ는 25년 역사의 방대한 매크로 시스템이다.
PGF/TikZ 매뉴얼은 1,300페이지가 넘는다.
“데코레이션 부분 지원”, “그래프 부분 지원”, “`let` 연산 미구현”은
이 방대함에 대응하는 현실적 타협이다.
PGF 매뉴얼 소스 파일을 코퍼스로 삼아 회귀 테스트를 운영하는 것은
이 구현 범위를 체계적으로 관리하려는 시도다.

### 데스크탑 앱 전용 AI 어시스턴트가 던지는 질문

TikZ Editor 데스크탑 앱에는 AI 어시스턴트가 탑재되어 있다.
웹 앱에는 없다.
이 분화는 기술적 이유보다 서비스 모델 이유일 가능성이 높다.

TikZ는 학습 곡선이 가파른 도구다.
전문 용어와 구문이 많고, 원하는 모양을 코드로 표현하는 방법을 찾는 것이 어렵다.
AI 어시스턴트는 이 마찰을 크게 낮출 수 있다.
“화살표가 있는 직사각형 박스 두 개를 연결해줘”라는 자연어 요청을 TikZ 코드로 변환하는 것은
현재 LLM이 잘 할 수 있는 작업이다.

Tauri v2 기반 데스크탑 앱은 Electron 대비 훨씬 가벼운 선택이다.
Rust 런타임 위에서 웹 뷰를 사용하기 때문에
최종 바이너리 크기와 메모리 사용량이 Electron의 절반 이하다.
LaTeX 사용자는 데스크탑 앱을 선호하는 경향이 강하고,
데스크탑 앱에 AI를 탑재함으로써 유료 전환 또는 프리미엄 기능의 문을 열어 두었다.

저자 DominikPeters 스스로 이 점을 수치로 보여줬다.
2026년 2월 이후 약 4개월 동안 Codex를 통해 약 7억 토큰을 사용했으며,
API 요금 기준으로 1만 5천 달러에 달하지만 실제 지출은 ChatGPT 구독료 약 500달러였다.[^DominikPeters-cost]
TikZ처럼 방대하고 지루한 시스템을 재구현하는 작업이 구독형 AI로 가능해진 것이,
이 프로젝트의 탄생 조건이었다.

## 비평

### 자체 렌더러와 LaTeX 실제 출력 간의 괴리는 좁히기 어렵다

TikZ Editor의 SVG 출력이 실제 LaTeX 컴파일 결과와 얼마나 다른가는
이 도구의 실용성을 결정하는 핵심 질문이다.

렌더러 비교 스크립트(`npm run compare:renderers`)의 존재 자체가 이 괴리가 실재함을 인정한다.
폰트 렌더링은 특히 어렵다.
LaTeX는 TeX 폰트 메트릭과 고유한 커닝(kerning) 시스템을 사용하고,
브라우저는 웹 폰트와 다른 텍스트 렌더링 파이프라인을 사용한다.
수식이 포함된 TikZ 노드는 이 불일치가 두드러진다.

패턴, 셰이딩, 화살촉의 정밀한 구현도 마찬가지다.
TikZ가 PGF 레이어를 통해 처리하는 저수준 연산들은
SVG의 표현 모델과 일대일 대응하지 않는다.
“시각적으로 충분히 비슷하다”와 “LaTeX 출력과 동일하다”는 다른 기준이며,
전자는 편집 도구로 충분하지만 후자는 인쇄 품질 논문 제출에 필요하다.

### 지원 범위의 공백이 실제 논문 작업에서 드러난다

25종 이상의 도형, `\foreach` 루프, 트리, 행렬을 지원한다는 것은 인상적이다.
그러나 “데코레이션 부분 지원”과 “`let` 연산 미구현”은
학술 논문에서 자주 사용되는 기능들이다.

TikZ 데코레이션은 경로 위에 패턴을 그리는 기능이다.
지그재그 선, 구불구불한 경계, 표시(ticks), 화살표 패턴이 여기 속한다.
물리학, 전자공학, 알고리즘 다이어그램에서 흔히 쓰인다.
`let` 연산은 좌표 계산을 위한 기능으로,
노드 크기에 의존적인 레이아웃을 만들 때 필수적이다.

이 공백들은 특정 분야의 연구자에게 TikZ Editor를 “보조 도구”로 남게 만든다.
시각적으로 위치를 조정하다가 결국 텍스트 에디터로 돌아가야 하는 시점이 존재한다.
이 시점이 사용자마다 다르고, 예측하기 어렵다는 것이 사용 경험을 불안정하게 만든다.

### 브라우저와 데스크탑의 기능 분화는 오픈소스 신뢰를 약화한다

AI 어시스턴트가 데스크탑 전용이라는 사실은 현재 사소해 보이지만,
장기적으로 이 분화가 어디까지 확장될지가 중요하다.

오픈소스 도구에서 핵심 기능이 특정 배포판에서만 사용 가능해지면,
커뮤니티는 그것을 프리미엄 모델로 가는 전조로 읽는다.
Obsidian이 플러그인 생태계를 열어두면서도 Sync와 Publish를 유료로 운영하는 것,
또는 Docker Desktop이 개인 무료/기업 유료로 분화한 것처럼.
현재 AI 어시스턴트는 “데스크탑 전용”이지만,
향후 “구독 전용”이 될 가능성을 배제하기 어렵다.

MIT 라이선스는 소스 코드를 보호하지만,
서비스로 제공되는 기능은 라이선스가 통제할 수 없다.
학술 도구로서 TikZ Editor의 지속적 신뢰는
핵심 편집 기능이 항상 무료이고 웹에서 접근 가능하다는 보장에 달려 있다.

### 새 요소의 절대 좌표 출력이 숙련된 TikZ 작성자의 기대를 벗어난다

TikZ는 절대 좌표만 쓰는 언어가 아니다.
`right of=A`, `above of=B` 같은 상대적 위치 지정과 명명된 `\coordinate`는
코드의 의미를 보존하고 유지 보수를 쉽게 하는 TikZ의 핵심 기능이다.
gignico는 단일 노드를 배치할 때조차 절대 좌표가 생성된다고 지적했다.[^gignico]
`\node {Hello};`는 경계 박스 중심에 노드를 놓는 완전히 유효한 TikZ 표현이지만,
TikZ Editor는 이를 `(0.5, 2.91)` 같은 하드코딩 좌표로 변환한다.

저자는 이 어려움을 솔직하게 인정했다.[^DominikPeters-coords]
단순한 코드에 갑자기 명명된 `\coordinate`를 삽입하면 사용자를 놀라게 하고,
반대로 상대적 위치 지정을 사용한 코드는 드래그 대상이 어떤 값인지 판단할 수 없어
편집이 제한된다.
어느 쪽을 선택해도 사용자 기대와 충돌한다.

이 딜레마는 WYSIWYG와 선언적 DSL 사이의 근본적 긴장이다.
시각적 조작은 구체적인 수치를 필요로 하지만,
숙련된 TikZ 작성자는 의미론적 관계를 코드에 유지하고 싶어 한다.
이 간극이 좁혀지지 않으면,
TikZ Editor가 생성하는 코드는 “나중에 손으로 정리해야 하는 초안”으로 남는다.

## 인사이트

### WYSIWYG LaTeX 편집기의 반복적 실패가 이번엔 다를 수 있다

LaTeX용 시각적 편집기의 역사는 실패의 역사다.
LyX, Scientific Word, Kile, TeXmacs — 이들은 각자의 방식으로 LaTeX의 시각화를 시도했고,
특정 사용자층을 확보했지만 주류가 되지 못했다.

이전 시도들이 실패한 공통 이유는 LaTeX의 전체를 시각화하려 했다는 것이다.
TikZ Editor는 다른 전략을 취한다.
LaTeX 문서 전체가 아니라 TikZ 그림이라는 단일 구성 요소만 다룬다.
이 범위 제한은 구현 가능성을 현실화한다.

또한 이전 시도들은 데스크탑 네이티브 앱이었고
LaTeX 설치에 의존했다.
TikZ Editor 웹 앱은 LaTeX 설치 없이 브라우저에서 바로 시작할 수 있다.
진입 장벽이 낮아지면 잠재 사용자 풀이 넓어지고,
그 중 일부가 실제로 TikZ를 배우는 동기가 생긴다.

### TikZ의 생존 가능성을 WYSIWYG가 높인다

TikZ는 매우 강력하지만 배우기 어렵다는 평가를 받는다.
Graphviz, Mermaid, D3.js 같은 대안들이 웹 네이티브 생태계에서 자연스러운 선택이 되면서
TikZ는 LaTeX 문서 내에서만 의미 있는 도구로 좁아지는 추세였다.

TikZ Editor 같은 도구는 이 추세를 역전할 잠재력을 가진다.
진입 장벽이 낮아지면 더 많은 사람이 TikZ를 채택하고,
더 많은 채택은 더 많은 학습 자원과 커뮤니티를 만들며,
결과적으로 생태계가 강화된다.
이것은 KaTeX가 MathJax의 성능 문제를 해결함으로써
웹에서의 수식 렌더링 표준을 강화한 것과 유사한 패턴이다.

특히 Overleaf 같은 온라인 LaTeX 플랫폼과 통합될 경우의 영향이 크다.
TikZ 그림을 Overleaf 내에서 TikZ Editor 인터페이스로 편집할 수 있다면,
그 생태계는 한 단계 도약한다.

그러나 이 낙관적 시나리오에는 전제 조건이 있다.
HN 토론에서 복수의 사용자가 Typst와 CeTZ 지원을 요청했다.[^wjholden]
Typst는 LaTeX 문법을 단순화한 현대적 대안으로,
젊은 연구자들 사이에서 빠르게 채택되고 있다.
TikZ Editor가 LaTeX 생태계에만 집중하는 동안
Typst 생태계가 자체적인 WYSIWYG 편집기를 갖추게 된다면,
TikZ Editor의 성공은 TikZ 강화가 아니라 단순한 현상 유지에 그칠 수 있다.

### 손실 없는 파싱이 AI 코드 편집의 새로운 패턴을 시사한다

TikZ Editor의 lossless AST는 AI 코드 생성의 문제를 해결하는 접근 방식으로 읽힐 수 있다.

AI가 코드를 생성할 때의 주요 불만 중 하나는
AI가 코드를 수정할 때 기존 포맷이나 스타일을 망친다는 것이다.
변수 이름을 바꾸거나 값 하나를 수정하는 것이
들여쓰기, 주석, 공백까지 영향을 미치는 경우가 많다.
손실 없는 AST를 통해 필요한 노드만 정밀하게 수정하는 방식은
이 문제의 구조적 해결책이다.

TikZ Editor의 AI 어시스턴트는 이 lossless 파이프라인 위에서 동작한다.
자연어 지시를 받아 TikZ 코드를 생성하되,
기존 코드의 포맷을 보존하면서 필요한 부분만 변경하는 것이 원칙적으로 가능하다.
이 패턴 — lossless 파싱 + 정밀 AST 수정 + AI 오케스트레이션 — 은
다른 DSL(Domain-Specific Language) 편집기에서도 반복될 것이다.

### 학술 도구가 웹 네이티브가 되는 전환의 의미

TikZ Editor는 학술 생태계의 도구가 웹 네이티브로 전환하는 더 넓은 흐름의 일부다.
Overleaf이 로컬 LaTeX 설치를 대체했고,
Jupyter Notebook이 MATLAB을 밀어내기 시작했으며,
Observable이 R Markdown의 웹 대안이 되었다.

이 전환의 공통 패턴은 세 가지다.
첫째, 설치 불필요 — 브라우저로 즉시 시작한다.
둘째, 공유 용이성 — URL로 결과를 공유할 수 있다.
셋째, 협업 가능성 — 실시간 협업이 구조적으로 가능해진다.

TikZ Editor 웹 앱은 첫 번째를 달성했다.
두 번째와 세 번째는 아직 없다.
TikZ 그림 URL을 공유하면 상대방이 즉시 편집할 수 있는 기능,
두 사람이 같은 TikZ 그림을 동시에 편집하는 기능이 추가된다면
이 도구의 위치가 달라진다.
Figma가 디자인 도구 생태계를 재편한 것처럼,
협업 기능이 학술 다이어그램 도구의 판도를 바꿀 수 있다.
