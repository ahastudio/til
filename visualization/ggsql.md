# ggsql: SQL을 위한 그래픽 문법

원문: <https://opensource.posit.co/blog/2026-04-20_ggsql_alpha_release/>

## 요약

ggsql은 Posit(구 RStudio)이 개발한 SQL 기반 데이터 시각화 도구로,
“그래픽 문법(Grammar of Graphics)”의 철학을 SQL 문법으로 구현한 알파 버전이다.
기존의 R `ggplot2`나 Python `plotnine` 같은 그래픽 문법 라이브러리가
프로그래밍 언어 런타임에 의존하는 것과 달리,
ggsql은 SQL 자체를 시각화 선언 언어로 사용해 R/Python 없이 그래프를 생성한다.

기본 구조는 `VISUALIZE ... FROM ...`으로 열을 시각적 속성에 매핑하고,
`DRAW` 절로 기하학적 표현 방식(점, 선, 막대 등)을 지정하는 두 단계로 이루어진다.
예를 들어 산점도는 `VISUALIZE bill_len AS x, bill_dep AS y FROM ggsql:penguins` 뒤에
`DRAW point`만 추가하면 완성된다.
색상 인코딩 추가도 `species AS color`를 선언에 덧붙이는 것으로 끝난다.

ggsql은 Quarto, Jupyter Notebook, Positron, VS Code 등 주요 데이터 과학 환경을
플랫폼으로 지원하며, 선언적 DSL 특성상 악의적 입력에 대해 더 안전하게 관리된다.
현재 알파 버전으로 개발 중이며 커뮤니티 피드백을 적극적으로 수용하고 있다.

SQL의 구조적 명시성(verbosity)은 전통적으로 약점으로 여겨졌지만,
ggsql은 이를 “시각화가 자기 설명적(self-documenting)이 된다”는 강점으로 재해석한다.
코드를 소리 내어 읽으면 어떤 그래프인지 즉시 이해할 수 있는 구조다.

## 분석

### 그래픽 문법(Grammar of Graphics)의 역사와 위치

그래픽 문법은 Leland Wilkinson이 1999년 저서 “The Grammar of Graphics”에서 제안한
데이터 시각화 이론 체계다.
데이터를 기하학적 객체(geom)에 매핑하고, 스케일·좌표계·통계 변환을 레이어로 조합해
임의의 그래프를 조립하는 방식이다.
Hadley Wickham이 이를 R의 `ggplot2`로 구현하면서 폭발적으로 보급되었고,
이후 Python의 `plotnine`, JavaScript의 `Vega-Lite` 등 수많은 파생 도구가 등장했다.

ggsql은 이 계보에서 “실행 환경”을 SQL 엔진으로 내린 실험이다.
R/Python 런타임 없이 SQL만 사용하는 데이터 분석가(예: BI 개발자, 데이터 엔지니어)가
같은 그래픽 문법 철학으로 시각화를 생성할 수 있게 한다.

### 데이터 파이프라인 연속성

SQL 기반 분석의 핵심 장점 중 하나는 데이터 변환과 조회가 하나의 파이프라인 안에서 이루어진다는 점이다.
ggsql은 이 파이프라인에 시각화 단계를 자연스럽게 편입시킨다.
데이터를 추출하고, 집계하고, 필터링한 뒤 별도 도구로 내보내는 단계 없이
SQL 쿼리 끝에 `VISUALIZE ... DRAW`를 붙이는 것만으로 그래프까지 완성된다.

### 모듈식 레이어 조합

ggsql의 설계는 ggplot2처럼 레이어를 조합하는 방식을 지원한다.
`PLACE` 절로 패싯(Facet), `SCALE` 절로 축 스케일, `LABEL` 절로 레이블을 각각
독립적으로 선언할 수 있다.
각 레이어가 명시적으로 선언되므로 시각화 설정이 숨겨진 기본값에 의존하지 않고
완전히 투명하게 관리된다.

## 비평

### 강점

SQL을 이미 알고 있는 분석가에게 학습 장벽이 없다는 점이 큰 강점이다.
R이나 Python 환경 구성 없이 데이터베이스 접속 권한만 있으면 즉시 사용할 수 있다.
선언적 DSL 특성상 시각화 정의가 재현 가능하고 버전 관리하기 쉬우며,
프롬프트 인젝션이나 임의 코드 실행 위험이 낮다.

### 약점과 한계

커뮤니티에서 지적된 가장 큰 문제는 SQL 데이터베이스와의 연결 방식이 불명확하다는 점이다.
ggsql이 기존 데이터베이스에 연결하는 것인지, 아니면 자체 데이터 레이어를 두는 것인지
알파 단계에서는 충분히 설명되지 않는다.
출력 형식(정적 이미지, 인터랙티브 차트, 특정 파일 포맷 등)과
외부 도구 연동에 대한 문서도 보강이 필요하다.

기존 ggplot2 사용자에게 ggsql이 구체적으로 어떤 우위를 제공하는지에 대한
설득력 있는 비교도 아직 부족하다.
R/Python을 사용하지 않는 SQL 전문가에게는 매력적이지만,
이미 ggplot2나 plotnine에 익숙한 사용자를 끌어오기 위한 킬러 기능이 필요하다.

## 인사이트

### SQL은 죽지 않는다 — 진화한다

지난 10년간 데이터 업계는 “SQL을 대체할 것”들을 끊임없이 내놓았다.
NoSQL, Spark DSL, DataFrame API, 데이터 과학 노트북 등이 그 사례다.
그러나 현실에서 SQL은 그 무엇에도 대체되지 않았고, 오히려 영역을 넓혀왔다.
dbt가 데이터 변환을 SQL로 끌어들였고, DuckDB가 인-메모리 분석을 SQL로 민주화했으며,
이제 ggsql이 시각화마저 SQL 안으로 편입하려 한다.

이는 SQL이 “단순히 오래된 언어이기 때문”이 아니다.
SQL의 선언적 특성(무엇을 원하는지만 기술하고 어떻게 처리할지는 엔진이 결정)이
AI 프롬프트 설계, 파이프라인 자동화, 감사 가능성(auditability) 측면에서
현대 데이터 스택이 원하는 속성과 정확히 일치하기 때문이다.
ggsql은 이 흐름의 자연스러운 연장선이다.

### 그래픽 문법의 진짜 혁신은 어휘 제공이었다

ggplot2가 가져온 혁명은 아름다운 기본 테마나 쉬운 문법에만 있지 않았다.
그것은 데이터 시각화에 대한 공통 어휘(vocabulary)를 제공했다는 점이다.
“geom”, “aes”, “scale”, “facet” 같은 용어로 시각화를 설명하면
팀 전체가 동일한 개념 모델로 소통할 수 있게 되었다.

ggsql이 SQL 생태계에 그래픽 문법의 어휘를 이식한다면,
SQL 중심으로 일하는 분석가들이 시각화를 “임시방편 도구”가 아닌
“분석 파이프라인의 일급 시민”으로 다룰 수 있게 될 것이다.
이 어휘 공유가 실현된다면 BI 도구와 데이터 과학 워크플로우 사이의 간극도 좁아질 수 있다.

### Posit의 생태계 전략이 보내는 신호

Posit(구 RStudio)이 ggsql을 만든 것은 단순한 기술 실험이 아니다.
Posit은 최근 수년간 R 전용 회사에서 “데이터 과학 생태계 회사”로 전환하는 중이다.
Quarto(언어 독립적 출판 시스템), Positron(Python/R 통합 IDE),
그리고 이제 ggsql(SQL 시각화)까지, Posit의 행보는 일관된 방향을 가리킨다.
“어떤 언어를 쓰든 좋은 데이터 과학 워크플로우를 지원한다”는 포지셔닝이다.

이는 Tidyverse를 “R만의 자산”으로 두는 대신 그 철학을 더 넓은 생태계로 전파하는 전략이다.
R 사용자 기반이 Python에 밀리는 현실을 직시한 Posit이 생존을 위해 선택한 방향이며,
이 과정에서 SQL 사용자를 새로운 타깃 고객으로 끌어들이려는 의도가 명확하다.
