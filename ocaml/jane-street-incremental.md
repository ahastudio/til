# Incremental — Jane Street의 증분 계산 라이브러리

<https://github.com/janestreet/incremental>

HN 토론: <https://news.ycombinator.com/item?id=48987822> (338점, 67개 댓글)

## 소개

Incremental은 Jane Street가 만든 OCaml 라이브러리로,
입력이 바뀔 때 결과 전체를 다시 계산하지 않고
변경된 부분에 의존하는 계산만 다시 수행하는 증분 계산(incremental computation)을 지원한다.

기본 아이디어는 계산 그래프다.
값들 사이의 의존 관계를 그래프로 추적하고,
입력이 바뀌면 그 입력에 연결된 노드만 무효화해서 필요한 계산만 다시 실행한다.
스프레드시트가 특정 셀이 바뀔 때 그 셀을 참조하는 셀만 다시 계산하는 방식과 같다.

Jane Street는 이 라이브러리를 자사의 금융 시스템에서 실제로 사용한다.
가격 계산, 리스크 집계, 파생 데이터 유지처럼
입력이 자주 바뀌지만 매번 전체 재계산은 너무 비싼 상황이 그 대상이다.

## 주요 개념

**노드(Node)와 의존 그래프**
계산의 각 단위는 노드로 표현된다.
노드는 다른 노드를 입력으로 받아 새 값을 생성하고,
입력 노드가 무효화되면 자동으로 자신도 무효화된다.
관찰되지 않는 노드는 계산을 생략하는 최적화가 적용된다.

**안정화(Stabilization)**
값 변경 후 `stabilize()` 를 호출하면 무효화된 노드들을
의존 순서(위상 정렬 순서)에 따라 재계산한다.
안정화 중에는 상태가 일관되게 유지된다.

**관찰자(Observer)**
노드의 현재 값을 읽으려면 관찰자를 만들어야 한다.
관찰자가 없는 노드는 계산 대상에서 제외되므로
불필요한 계산을 자동으로 제거한다.

**Map, Bind, If**
`map`은 노드 값을 변환하고,
`bind`는 입력 값에 따라 의존 그래프 구조 자체를 바꾸며,
`if_`는 조건에 따라 두 노드 중 하나를 선택한다.
`bind`가 가장 강력하지만 구조가 바뀌므로 비용이 크다.

## 분석

### 증분 계산이 해결하는 문제의 구조

증분 계산이 필요한 상황은 세 조건이 동시에 성립할 때다.
입력이 자주 바뀌고, 계산 비용이 크며, 대부분의 입력 변경이 결과의 일부에만 영향을 준다.
이 세 조건이 모두 성립하지 않으면 일반 재계산이 더 간단하다.

금융 시스템은 이 세 조건의 교과서적 사례다.
시장 데이터는 초당 수천 번 바뀌고,
포트폴리오 전체의 리스크 계산은 비싸며,
개별 자산 가격 변화는 그 자산을 포함한 포지션에만 영향을 준다.

그러나 같은 패턴은 훨씬 넓은 영역에서 나타난다.
가격 계산 엔진, 권한 규칙 평가, 대시보드 집계, 빌드 시스템, UI 상태 관리가 모두 같은 구조다.
React의 가상 DOM 비교도 UI 렌더링에 이 패턴을 적용한 것이고,
Bazel 같은 빌드 시스템도 파일 변경만 다시 빌드하는 방식으로 같은 원리를 사용한다.

이 패턴이 금융 업계에서 오래된 실천이라는 사실도 주목할 만하다.
HN에서 fadesibert는 Goldman Sachs가 약 30년 전 금융 상품 가격 계산에 같은 접근법을 사용했다고 밝혔다.[^fadesibert]
"Node Purpling"이라는 내부 용어로 불렸고, 미분 같은 계산 비용이 큰 연산을 최소한으로 실행하기 위해 쓰였다.
Incremental이 새로운 개념이 아니라 금융 엔지니어링 현장에서 수십 년간 검증된 패턴의 라이브러리화임을 보여준다.

ronfriedhaber는 이 맥락을 더 넓혔다.
Differential Dataflow, Timely Dataflow, DBSP 같은 시스템이 같은 문제 공간을 탐색하고 있으며,
Feldera, Materialize 같은 실제 제품들이 이 위에 구축됐다고 정리했다.[^ronfriedhaber]
Incremental은 그 넓은 생태계의 한 축이지만, 스트리밍 데이터나 분산 처리보다는
단일 프로세스 내의 파생 상태 관리에 초점을 맞춘 위치다.

### 의존 그래프 추적의 비용

증분 계산은 공짜가 아니다.
의존 관계를 추적하고 무효화 전파를 관리하는 비용이 있다.
입력 변경이 드물고 계산이 가벼우면 일반 재계산보다 느릴 수 있다.

`bind`는 특히 주의가 필요하다.
입력에 따라 의존 그래프 구조 자체가 바뀌므로,
`bind`가 발화할 때마다 하위 그래프를 다시 구성하는 비용이 든다.
`bind` 내부에 `bind`가 중첩되면 이 비용이 기하급수적으로 커진다.
`map`으로 표현할 수 있는 계산을 `bind`로 쓰는 것은 흔한 성능 실수다.

또한 관찰되지 않는 노드의 계산을 생략하는 최적화는
관찰자가 잘못 관리되면 역으로 계산이 누락되는 버그로 이어진다.
의존 그래프의 정확성은 의존 관계를 올바르게 선언하는 것에 달려 있고,
이것은 개발자의 책임이다.

### Jane Street 라이브러리 생태계 맥락

Incremental은 Jane Street의 여러 OCaml 라이브러리 중 하나다.
Core, Async, Base, Bonsai와 함께 Jane Street 스타일의 OCaml 시스템을 구성한다.

Bonsai는 이 위에 구축된 UI 프레임워크로,
웹 UI 상태 관리에 Incremental을 활용한다.
함수형 반응형 프로그래밍(FRP)과 유사한 아이디어를 OCaml 타입 시스템으로 표현한 설계다.
osener는 Bonsai가 React의 가상 DOM 접근보다 더 세밀한 수준에서 계산을 건너뛸 수 있다고 설명했다.[^osener]
React는 가상 DOM을 구성하는 비용 자체는 피하지 못하지만,
Bonsai는 가상 DOM 구성 자체를 증분화해서 변경이 없는 부분의 DOM 생성 작업까지 생략한다.

Jane Street가 이 라이브러리들을 오픈소스로 공개한 것은
채용과 생태계 형성을 위한 전략이기도 하지만,
실제 대규모 금융 시스템에서 검증된 설계를 공유한다는 의미이기도 하다.
라이브러리가 학술적 실험이 아니라 운영 시스템에서 나왔다는 점이
설계 결정의 신뢰성을 높인다.

## 비평

### OCaml에 종속된 구현이 더 넓은 채택을 막는다

증분 계산의 아이디어 자체는 언어 독립적이다.
그러나 Incremental 라이브러리는 OCaml 생태계에 종속돼 있다.
같은 아이디어를 다른 언어로 가져가려면 처음부터 다시 구현해야 한다.

JavaScript/TypeScript 생태계에는 MobX, Recoil, Jotai, Solid.js 같은
반응형 상태 관리 라이브러리들이 이미 비슷한 아이디어를 구현하고 있다.
jitl은 이 흐름이 JavaScript에서 "시그널(signals)"이라는 이름으로 표준화되고 있다고 설명했다.[^jitl]
Vue, SolidJS, Svelte, Ember, Angular가 모두 시그널 기반 반응성을 채택했고,
TC39에 공식 표준 제안(proposal-signals)도 제출됐다.
그러나 이 라이브러리들은 UI 상태 관리에 특화돼 있어
백엔드 비즈니스 로직의 파생 데이터 관리에는 잘 맞지 않는 경우가 많다.
eyupucmaz는 이 차이를 명확히 짚었다.
JS 솔루션들과 달리 OCaml 타입 시스템은 의존 관계의 정확성을 컴파일 타임에 보장할 수 있고,
이것이 Incremental이 금융 운영 시스템에서 신뢰할 수 있는 이유 중 하나라는 것이다.[^eyupucmaz]

Incremental의 설계가 범용적인 증분 계산 패턴으로 더 넓게 영향을 주려면,
개념을 문서화하고 다른 언어 구현을 장려하는 노력이 필요하다.
현재 문서는 API 참조와 블로그 글 수준이어서 다른 언어 구현자가 참고하기에는 부족하다.

### 무효화 오류의 디버깅이 어렵다

의존 관계를 잘못 선언하면 오래된 값을 반환하는 버그가 생긴다.
이 종류의 버그는 재현이 어렵다.
특정 입력 변경 순서에서만 나타나고,
의존 그래프 전체를 시각화하지 않으면 원인을 찾기 어렵다.

라이브러리 자체는 이런 디버깅을 위한 도구를 충분히 제공하지 않는다.
의존 그래프의 현재 상태를 덤프하거나,
특정 노드가 왜 무효화됐는지 추적하는 기능이 제한적이다.
운영 시스템에서 증분 계산 버그를 진단하는 것은 일반 계산 버그보다 훨씬 어려울 수 있다.

## 인사이트

### 증분 계산 패턴은 AI 파이프라인에도 적용된다

AI 시스템이 복잡해지면서 증분 계산 패턴이 새로운 맥락에서 필요해지고 있다.
문서가 바뀌면 임베딩을 다시 생성해야 하고,
소스 코드가 바뀌면 의존 파일의 분석을 다시 실행해야 한다.
이것은 정확히 증분 계산이 해결하는 문제다.

그러나 AI 파이프라인의 증분 계산은 일반 계산보다 복잡하다.
LLM 호출은 비결정적이고, 임베딩 모델이 바뀌면 모든 임베딩이 무효화된다.
의존 관계가 코드 의존성뿐 아니라 모델 버전, 프롬프트 버전, 데이터 버전에도 달려 있다.

Incremental 같은 라이브러리가 AI 파이프라인 캐싱과 재계산 관리에
직접 적용되기는 어렵지만, 그 핵심 개념 — 의존 그래프 추적과 선택적 무효화 — 은
AI 파이프라인 관리 도구의 설계 원리로 채용될 수 있다.
LLM 파이프라인 오케스트레이션 도구들이 아직 이 수준의 체계적인 의존성 관리를
제공하지 않는다는 점은 기회이기도 하다.

### 반응형 프로그래밍과 증분 계산의 통합이 백엔드까지 확장될 것이다

지난 10년 동안 반응형 프로그래밍은 프런트엔드 UI 레이어의 주류가 됐다.
React, Vue, Svelte는 모두 입력 변화에 반응하는 UI 업데이트를 선언적으로 표현하는 방향으로 수렴했다.

같은 아이디어가 백엔드 비즈니스 로직으로 확장되는 것은 시간 문제다.
가격 계산, 재고 관리, 권한 평가처럼 파생 상태가 많은 도메인에서
입력이 바뀔 때마다 전체를 재계산하는 방식의 비효율이 점점 눈에 띈다.

이 방향의 확산을 막는 것은 기술적 어려움보다 생태계 부재다.
프런트엔드에서 반응형 프로그래밍이 퍼진 것은 React가 좋은 도구였기 때문이 아니라
React 생태계가 커졌기 때문이다.
백엔드 증분 계산도 언어와 프레임워크에서 지원이 충분해질 때
더 빠르게 퍼질 것이다.
Incremental은 그 방향의 초기 사례이지만, 아직 생태계가 형성되지 않았다.
