# Nubo

<https://github.com/sirini/nubo>

## 소개

Nubo는 Nuxt 4와 GoFiber v3을 결합한 커뮤니티·웹사이트 빌더다.
전작 TSBOARD의 후속 프로젝트로, 데이터베이스 스키마 호환성을 유지해 기존 사용자가
별도의 데이터 변환 없이 마이그레이션할 수 있다.

SSR(서버 사이드 렌더링)을 통한 SEO 최적화, 스킨 기반 UI 커스터마이징,
개선된 검색과 페이지네이션을 핵심 가치로 제시한다.
관리자 대시보드처럼 보호가 필요한 페이지는 보안을 위해 클라이언트 사이드 렌더링으로
유지한다.

## 기술 스택

| 분류       | 도구                                             |
| ---------- | ------------------------------------------------ |
| 프론트엔드 | Nuxt 4, Vue 3, Pinia, shadcn-vue, Tailwind CSS, Tiptap |
| 백엔드     | GoFiber v3 (GOAPI)                               |
| 데이터베이스 | MySQL/MariaDB                                  |

## 스킨 시스템

내장 스킨 아키텍처를 통해 레이아웃, 게시판, 로그인 화면을 코어 로직에 손대지 않고
교체할 수 있다.
스킨 제작에 필요한 TypeScript와 Vue 3 지식을 최소화하여 디자이너도 직접 작업할 수 있도록
설계했다.

## 설치 요구 사항

64비트 CPU 2코어 이상, Linux/macOS, Node.js 24+ LTS, Nginx, MySQL/MariaDB가 필요하다.
저장소 클론 후 `./goapi-linux`로 초기 DB를 설정하고,
`npm install && npm run dev`로 프론트엔드를 실행한다.
프로덕션 환경에서는 PM2로 Node.js와 API 프로세스를 모두 관리한다.

## 라이선스

MIT 라이선스를 적용한다.

## 분석

### TSBOARD에서 Nubo로의 전환

TSBOARD에서 프레임워크를 Nuxt 4로, UI를 shadcn-vue와 Tailwind CSS로 전환한 것은
현재 Vue 생태계의 주류 스택으로의 수렴이다.
shadcn-vue는 Radix Vue 기반의 컴포넌트 라이브러리로, “스타일을 소유한다”는 철학 덕분에
스킨 시스템과 잘 맞는다. 컴포넌트를 복사해 직접 수정하는 방식이 Nubo의 스킨 교체 모델과
자연스럽게 통합된다.

백엔드를 GoFiber v3으로 분리한 것은 Node.js 풀스택 대신 Go의 성능 이점을 취하는
아키텍처 선택이다.
커뮤니티 플랫폼에서 게시물 목록, 검색, 댓글 처리는 동시 요청이 집중되는 지점이고,
Go의 동시성 모델이 이 부분에서 유리하다.

### 데이터베이스 호환성 유지의 전략적 의미

새로운 UI/UX 스택으로 전환하면서도 TSBOARD의 데이터베이스 스키마를 그대로 유지한 것은
기존 사용자 기반을 보호하기 위한 의도적 결정이다.
오픈소스 커뮤니티 소프트웨어에서 마이그레이션 장벽은 채택률의 핵심 변수다.
데이터 변환 없이 Nubo로 전환할 수 있다면, TSBOARD 사용자가 자연스럽게 이주할 동기를
갖는다.

## 비평

### 강점

Nuxt 4 SSR과 Go 백엔드를 분리하는 아키텍처는 각 레이어를 독립적으로 최적화하고
배포할 수 있다는 장점이 있다.
스킨 시스템의 설계 원칙—코어 코드 수정 없이 UI 교체—은 확장성 측면에서 바람직하다.
MIT 라이선스는 상업적 사용과 포크에 제약이 없어 채택 장벽이 낮다.

### 약점

Node.js 24+ LTS를 요구하면서 Nginx와 GoFiber 바이너리까지 설치해야 하는 설정은
단순 배포를 기대한 사용자에게 복잡하게 느껴질 수 있다.
Docker Compose 기반의 원스텝 배포나 클라우드 원클릭 배포 지원 여부가 문서에서
명확하지 않다.

한국어로만 작성된 문서는 글로벌 커뮤니티 참여를 제한한다.
TSBOARD의 레거시를 잇는 프로젝트인 만큼 국내 사용자 기반에 집중하는 전략으로 보이지만,
오픈소스 생태계 기여 관점에서는 아쉽다.

## 인사이트

### Go + Nuxt 분리 아키텍처가 제시하는 방향

Node.js 풀스택(Next.js API Routes, Nuxt server routes)이 주류인 시대에
Go 백엔드를 별도로 운용하는 선택은 의식적인 성능 트레이드오프다.
개발 복잡성은 높아지지만 처리량과 응답 시간에서 이점을 얻는다.

커뮤니티 플랫폼의 트래픽 패턴—읽기 집중, 간헐적 쓰기 급증, 실시간 알림—은
Go의 동시성 강점이 발휘되기 좋은 조건이다.
Node.js API의 이벤트 루프가 충분히 처리 가능한 트래픽 수준에서는 오버엔지니어링이지만,
상당한 규모의 커뮤니티를 목표로 설계한다면 합리적인 시작점이다.

### 한국 오픈소스 커뮤니티 플랫폼의 공백

글로벌 커뮤니티 플랫폼 소프트웨어(Discourse, Flarum, NodeBB)는 한국어 지원이 제한적이고,
한국 인터넷 문화의 특성—파일 첨부, 포인트 시스템, 게시판 중심 구조—에 맞지 않는 경우가 많다.
TSBOARD와 Nubo는 이 공백을 메우는 국산 소프트웨어다.

이 공백이 지속되는 이유는 한국 인터넷 커뮤니티 문화가 글로벌 표준과 다른 요구 사항을
가지기 때문이다.
게시판, 댓글 트리, 파일 첨부, 추천/비추천, 포인트 제도가 기본 기능으로 기대되는 환경은
WordPress 플러그인이나 Discourse 설정으로 해결하기 어렵다.
Nubo가 이 요구 사항에 맞는 현대적 스택으로 제공된다면 틈새 시장에서 확실한 수요가 있다.
