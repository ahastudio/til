# Chromex

<https://github.com/GENEXIS-AI/chromex>

## 소개

Chromex는 Chrome MV3 사이드 패널 어시스턴트로, Chrome과 OpenAI Codex를
로컬 네이티브 브리지를 통해 연결한다.
현재 페이지, 선택한 탭, 업로드한 파일, 음성 입력, 이미지, 브라우저 워크플로우를
다루면서 인증 정보를 익스텐션 스토리지 밖에 보관한다는 것이 핵심 설계 방침이다.
GenexisAI CHOI가 작성해 2026년 4월 28일 공개했고, 2일 만에 745개 스타를 받았다.

주요 기능 범위는 다음과 같다.
현재 웹 페이지, 열린 탭, 스크린샷, 업로드 파일(PDF, DOCX, CSV, XLSX 등),
브라우저 히스토리와 대화할 수 있다.
YouTube, 뉴스, 논문, arXiv 페이지의 콘텐츠를 요약하고 비교할 수 있으며,
Codex 이미지 워크플로우로 이미지를 편집하거나 생성할 수 있다.
음성 전사(voice transcription), 라이브 음성 모드, 페이지 인식 제안,
커스텀 프로필, Codex 스킬도 지원한다.

런타임 경계(Runtime Boundary)는 다음 단방향 체인으로 구성된다.

```text
Chrome Extension → Native Messaging Host → Local Bridge → codex app-server
```

소스 트리는 네 패키지로 분리된다.
`packages/extension`(Chrome MV3 사이드 패널),
`packages/bridge`(Codex app-server 및 멀티모달 워크플로우 로컬 데몬),
`packages/native-host`(Chrome Native Messaging 릴레이),
`packages/shared`(공유 타입·정책·프로필·헬퍼).

## 분석

### Native Messaging으로 API 키를 익스텐션 밖에 두는 이유

Chromex의 가장 독특한 아키텍처 결정은 OpenAI API 키와 OAuth 토큰을
Chrome 익스텐션 스토리지에 저장하지 않는다는 것이다.
Chrome 익스텐션 스토리지(`chrome.storage`)는 권한을 가진 익스텐션이 읽을 수 있고,
악성 익스텐션이나 XSS 공격 시 노출될 수 있다.
Chromex는 인증 흐름 전체를 로컬 브리지 데몬(`packages/bridge`)과
`codex app-server`에 위임해 이 노출 면을 제거한다.

이를 가능하게 하는 것이 Chrome의 Native Messaging API다.
익스텐션이 직접 `localhost` HTTP 서버에 연결하는 대신,
브라우저가 허용한 네이티브 호스트 프로세스(JSON stdio 기반)와만 통신한다.
포트 노출 없이 로컬 프로세스와 대화할 수 있으므로,
로컬 브리지가 일종의 안전한 비밀 관리 게이트웨이가 된다.
설치 시 `install-native-host.mjs`가 Chrome에 허용 네이티브 호스트를 등록하고,
그 이후 익스텐션 ID가 맞아야만 통신이 성립한다.

### MV3 제약을 우회하는 콘텐츠 스크립트 전략

Chrome MV3는 백그라운드 페이지를 서비스 워커로 대체하고,
장기 실행 연결(long-lived connection)을 엄격히 제한한다.
Chromex는 이 제약을 콘텐츠 스크립트와 Native Messaging 결합으로 우회한다.
사이드 패널은 서비스 워커보다 수명이 길고, Native Messaging 포트는
`chrome.runtime.connectNative`로 유지되기 때문에
사용자가 사이드 패널을 열어두는 동안 지속적인 브리지 연결이 가능하다.

브라우저 제어 워크플로우는 콘텐츠 스크립트를 통해 DOM 조작과
페이지 내 활동 표시자(visible in-page activity indicators)를 심는다.
이는 사용자에게 에이전트가 “무엇을 하고 있는지” 보여주는 투명성 장치다.
활동이 화면에 표시되므로 사용자가 워크플로우를 감시하고 중단할 수 있다.

### 멀티모달 라우팅과 사이트 어댑터

Chromex는 사용자 요청을 자동으로 라우팅한다.
`@` 피커로 탭을 선택하거나 첨부 파일을 올리면 요청 유형(페이지·파일·이미지·
히스토리·음성·브라우저 제어)을 분류해 적절한 처리 경로로 보낸다.

사이트 인식(site-aware) 제안 기능은 현재 탭의 URL을 파악해
YouTube, 뉴스, 연구, 메일, 협업, 노트, 태스크 도구, 쇼핑, 여행,
한국 업무 서비스에 맞는 제안을 노출한다.
YouTube 어댑터는 현재 타임스탬프 컨텍스트와 탐색(seek) 액션을 지원한다.
이는 단순한 페이지 텍스트 추출이 아니라 사이트별 시멘틱을 이해하는 레이어다.

Codex 스킬은 사용자의 로컬 `.codex/skills/*/SKILL.md`에서 불러오며,
사용자가 명시적으로 활성화한 경우에만 적용된다.
이는 Claude Code의 `SKILL.md` 컨벤션과 같은 패턴으로,
에이전트 CLI 생태계에서 스킬 파일이 표준 확장 메커니즘으로 수렴하는 흐름을 보여준다.

## 비평

### 강점: 보안 설계가 아키텍처에 반영된 구조

API 키를 익스텐션 스토리지에 넣지 않는다는 원칙이 그냥 문서 조항으로 끝나지 않고
아키텍처 전체를 규정한다는 점이 인상적이다.
대화 히스토리가 기본값으로 세션 한정이고, 영속 저장은 명시적 opt-in인 것,
히스토리·탭·화면 캡처·마이크 권한을 기능이 필요할 때만 요청하는 것 모두
보안이 설계 결정의 출발점이 됐음을 보여준다.

자체 호스트·수정 빌드를 위한 `SECURITY.md`와 `PRIVACY.md`를 별도로 유지하는 것,
네이티브 호스트 자식 프로세스에 환경 변수 허용 목록을 적용하는 것도
운영 환경을 의식한 세심한 처리다.

### 약점: 로컬 브리지 설치 마찰

Native Messaging 기반 아키텍처의 대가는 설치 복잡도다.
`git clone` → `npm install` → `npm run build` → `install-native-host.mjs` →
`chrome://extensions`에서 언팩 로드 — 이 다섯 단계는 일반 사용자에게 높은 장벽이다.
README의 Windows 트러블슈팅 섹션이 길다는 것 자체가 설치 마찰을 방증한다.
`ENOENT Could not read package.json`처럼 폴더 혼동에서 비롯한 오류가
명시적으로 문서화돼 있다는 것은 실제로 이 오류가 자주 발생한다는 뜻이다.

Chrome 웹 스토어 배포가 없으므로 업데이트도 수동 `git pull` + 재빌드다.
보안 설계와 편의성의 근본적 트레이드오프가 설치 경험에 그대로 드러난다.

### 미해결 질문: Codex 의존성 단일화

Chromex는 Codex CLI에 강하게 결합돼 있다.
README의 모든 설치 경로가 Codex를 전제하고, 브리지도 `codex app-server`에
의존한다.
OpenAI가 Codex CLI를 변경하거나 deprecated한다면 Chromex 전체가 흔들린다.
BYOK(OpenAI-compatible proxy 등)를 지원하는지, 다른 에이전트 CLI로
교체 가능한지는 README에서 명확하지 않다.

## 인사이트

### 브라우저 익스텐션이 로컬 에이전트의 UI가 되는 흐름

Chromex가 하는 일을 추상화하면, 로컬에서 실행되는 AI 에이전트에
브라우저 컨텍스트를 공급하는 브리지다.
이 역할 정의는 중요한 패턴 전환을 가리킨다.
기존 AI 어시스턴트는 서버에서 실행되는 모델에 사용자가 질문을 보내는 모델이었다.
Chromex가 보여주는 흐름은 그 역방향이다 — 로컬에서 실행되는 에이전트가
브라우저를 감각 기관(sensory organ)으로 삼아 현실 세계를 인식한다.

이 역전은 “AI 어시스턴트가 어디에 사는가”라는 질문의 답을 바꾼다.
클라우드 서비스에서 사용자의 로컬 머신으로 무게 중심이 이동할 때,
브라우저 익스텐션은 자연스럽게 로컬 에이전트의 눈과 손이 된다.
화면을 캡처하고, 탭을 읽고, DOM을 조작하는 것은 브라우저 익스텐션만이 할 수
있는 작업이다. 이 흐름이 계속된다면 “모델 제공자의 플러그인”이 아니라
“로컬 에이전트의 브라우저 인터페이스”로서의 익스텐션 카테고리가 성장할 것이다.

2차 효과로, 이 구조는 AI 어시스턴트의 개인화 범위를 근본적으로 확장한다.
서버 측 모델은 사용자의 브라우저 히스토리, 열린 탭, 현재 보고 있는 페이지를
알 수 없다. 로컬 에이전트 + 브라우저 익스텐션 구조는 이 맥락 전체를 에이전트에
공급할 수 있고, 개인 정보는 외부 서버로 나가지 않는다.
개인화와 프라이버시가 동시에 향상되는 드문 경우다.

### Native Messaging이라는 잊혀진 보안 프리미티브

Chrome의 Native Messaging API는 2013년부터 존재했지만,
대부분의 개발자에게 잘 알려지지 않은 API다.
익스텐션이 `localhost` HTTP 서버와 직접 대화하는 것보다 훨씬 안전한 이유가 있다.

`localhost` HTTP 포트는 같은 머신의 모든 프로세스가 접근할 수 있고,
CSRF, 포트 스캔, 다른 탭의 JavaScript에서도 접근 가능하다.
Native Messaging은 Chrome이 등록된 네이티브 호스트 목록을 직접 관리하고,
익스텐션 ID가 일치해야만 연결을 허용하며, stdio JSON 통신만 사용하므로
포트가 외부에 노출되지 않는다.

Chromex가 이 메커니즘을 선택한 것은 단순한 기술 선택이 아니라
공격 면(attack surface)을 의식한 보안 결정이다.
AI 에이전트가 로컬에서 실행되는 패턴이 늘어날수록,
에이전트와 브라우저 사이의 통신 채널 보안이 중요해진다.
Native Messaging은 이 문제에 대한 크롬의 답이지만,
설치 복잡도 때문에 잘 쓰이지 않았다.
Chromex는 이 트레이드오프를 명시적으로 받아들인 프로젝트다.

### Codex 스킬 컨벤션의 생태계 수렴

Chromex가 사용자의 `.codex/skills/*/SKILL.md`를 로드한다는 것은 작은 디테일이지만
흥미로운 생태계 신호다.
Claude Code의 `SKILL.md` 컨벤션, Open Design의 스킬 시스템,
그리고 이제 Codex 기반 Chromex까지 — 서로 다른 에이전트 플랫폼이
“스킬을 마크다운 파일로 정의한다”는 동일한 패턴을 독립적으로 채택하고 있다.

이 수렴은 우연이 아니다.
마크다운 파일은 에이전트가 `Read` 도구로 직접 소화할 수 있고,
버전 관리가 되며, 사람이 편집할 수 있고, 공유하기 쉽다.
스킬을 플러그인 패키지나 API 호출로 정의하는 대신 텍스트 파일로 정의하면
에이전트가 스킬 자체를 컨텍스트로 이해할 수 있다 — 이것이 LLM 친화적 확장의
핵심 이점이다.

이 패턴이 더 확산된다면, `.codex/skills/`, `.claude/skills/` 등 에이전트별로
갈라진 스킬 디렉터리가 언젠가 표준화될 가능성이 있다.
마치 `Dockerfile`이 컨테이너 정의의 표준이 된 것처럼,
에이전트 스킬 정의 형식도 수렴 지점을 향해 나아가고 있을 수 있다.
Chromex는 그 수렴 과정의 데이터 포인트다.
