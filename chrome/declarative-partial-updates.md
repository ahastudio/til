# Declarative Partial Updates — 비선형 HTML 전달을 위한 새 API

원문: <https://developer.chrome.com/blog/declarative-partial-updates>

HN 토론: <https://news.ycombinator.com/item?id=48252957> (54점, 44개 댓글)

## 요약

Chrome이 HTML을 비선형(non-linear) 방식으로 전달하기 위한 두 가지 API 묶음을
도입한다.
Barry Pollard와 Noam Rosenthal이 2026년 5월 19일에 발표한 글로,
현대 웹 앱이 콘텐츠를 스트리밍하고 동적으로 갱신하는 방식의 한계를 겨냥한다.
핵심 아이디어는 문서를 위에서 아래로 한 번에 내려보내는 대신,
준비된 조각부터 먼저 보내고 나머지를 나중에 끼워 넣을 수 있게 하는 것이다.

첫 번째 묶음은 순서 없는 스트리밍(out-of-order streaming)을 위한 선언적 API다.
`<template>` 요소와 처리 명령(processing instruction) 플레이스홀더를 함께 쓴다.
전통적으로 XML 기능이던 처리 명령을 HTML 안에서 콘텐츠 교체의 참조점으로 활용한다.
`<?marker name=“placeholder”>`로 자리를 표시하고,
나중에 도착한 `<template for=“placeholder”>`의 내용으로 그 자리를 직접 교체한다.
`<?start>`와 `<?end>`는 그 사이에 임시 콘텐츠(예: “Loading…”)를 감싸 두었다가
해당 템플릿이 처리되는 시점에 교체한다.
이로써 비싼 연산을 기다리느라 페이지 전체를 막지 않고,
중요한 HTML을 먼저 전달한 뒤 덜 중요한 부분을 나중에 흘려보내는
아일랜드 아키텍처(island architecture) 패턴을 구현할 수 있다.
보안을 위해 템플릿은 같은 부모 요소 안의 처리 명령만 수정할 수 있고,
`setHTML`이나 `innerHTML`을 통해 쓸 때는 문서 조각(document fragment) 안에서만
동작하며 기존 DOM을 건드리지 못한다.

두 번째 묶음은 JavaScript에서 HTML을 삽입하고 스트리밍하는 메서드들이다.
파편화된 기존 API를 일관된 명명 규칙으로 대체하자는 제안이다.
요소 내용을 설정하는 `setHTML(html, options)`,
요소를 교체하는 `replaceWithHTML()`,
앞뒤로 삽입하는 `beforeHTML()`·`afterHTML()`,
처음과 끝에 넣는 `prependHTML()`·`appendHTML()`이 정적 메서드로 제공된다.
각 정적 메서드에는 Streams API를 통해 콘텐츠가 준비되는 대로 흘려보내는
스트리밍 변형(`streamHTML()`, `streamAppendHTML()` 등)이 짝으로 존재한다.

기본적으로 살균(sanitization)이 적용되며,
`setHTMLUnsafe()`·`streamHTMLUnsafe()` 같은 `Unsafe` 변형은 이를 끄고
`runScripts: true` 옵션으로 스크립트 실행까지 허용한다.
`options` 인자는 커스텀 `Sanitizer` 객체를 받아 허용 요소를 제한할 수 있다
(예: `new Sanitizer({ elements: [“em”, “i”, “b”, “strong”] })`).
이 기능들은 Chrome 148 이상에서 `chrome://flags/#enable-experimental-web-platform-features`
플래그로 사용할 수 있고,
`template-for-polyfill`과 `html-setters-polyfill` 두 npm 폴리필이 제공된다.
다른 브라우저 벤더로부터 긍정적 피드백을 받았으며 표준화 절차를 밟고 있다.

## 분석

### 이 제안의 핵심 주장은 “전달 순서를 코드가 아니라 마크업으로 결정한다”는 것이다

글의 중심에는 HTML이 본질적으로 스트리밍 포맷이라는 오래된 전제가 있다.
브라우저는 처음부터 도착하는 바이트 순서대로 HTML을 파싱하고 렌더링해 왔다.
그러나 이 스트리밍은 선형적이다.
문서의 5번째 단락을 보내려면 1~4번째 단락을 먼저 보내야 한다.
선언적 부분 갱신은 이 선형성을 깨려는 시도다.
플레이스홀더를 먼저 자리에 두고,
실제 콘텐츠가 준비되는 순서대로 어디든 끼워 넣을 수 있게 한다.

여기서 결정적인 설계 선택은 이 메커니즘을 JavaScript가 아니라 마크업에 둔 점이다.
오늘날 같은 효과를 내려면 React Suspense나 프레임워크의 하이드레이션,
혹은 직접 작성한 DOM 조작 코드가 필요하다.
글은 그 능력을 처리 명령과 `<template>`이라는 선언적 원시 요소로 끌어내린다.
즉 “무엇을 어디에 끼워 넣을지”를 런타임 로직이 아니라 문서 구조가 표현한다.
이것이 가능하면 JavaScript가 실행되기 전에도,
혹은 JavaScript가 전혀 없이도 순서 없는 스트리밍이 동작한다.

HN의 여러 댓글이 이 선언적 접근을 기존 도구에 빗대어 재해석했다.
가장 짧은 반응은 michaelmrose의 “So htmx?”였고,[^michaelmrose]
trueno는 이를 더 풀어 “HTML이 위에서 아래로 스트리밍된다는 점을 생각하면
이 제안은 HTML의 순서 없는 스트리밍이고,
그동안 React·JS로 앞당겨 처리하던 전달·렌더링 문제를 다룬다”고 정리했다.[^trueno]
같은 맥락에서 niutech는 이 기능이 htmx조차 필요 없다며
160바이트짜리 HTMZ를 대안으로 제시했는데,[^niutech-htmz]
이에 대해 설계자 중 한 명인 nomsternom은 HTMZ가 “분명한 영감이지만
이 제안은 덜 단정적(less opinionated)”이라고 답했다.[^nomsternom-htmz]
이 짧은 교환은 글이 명시하지 않은 사실을 드러낸다.
이 제안은 무에서 나온 것이 아니라, 유저랜드에서 이미 검증된 패턴을
플랫폼 원시 기능으로 끌어내린 것이다.

### 두 API 묶음은 사실 같은 문제의 선언적 측면과 명령적 측면이다

글은 두 기능을 나란히 소개하지만 둘의 관계를 명시적으로 묶지는 않는다.
그러나 구조를 보면 분명하다.
처리 명령 기반 스트리밍은 서버가 보내는 초기 HTML 응답을 위한 선언적 경로이고,
`setHTML`/`streamHTML` 계열은 클라이언트 JavaScript를 위한 명령적 경로다.
둘 다 “이미 화면에 있는 콘텐츠를 안전하게 부분 교체한다”는 같은 목표를 향한다.

이 이중 구조는 의도적이다.
초기 로드 시점에는 마크업만으로 처리하고,
이후 사용자 상호작용으로 들어오는 동적 갱신은 JavaScript로 처리한다.
두 경로가 같은 살균 모델과 같은 `Sanitizer` 옵션을 공유한다는 점이
이 통합을 뒷받침한다.
서버 측 처리 명령과 클라이언트 측 `setHTMLUnsafe`가 같은 보안 경계를 따른다.

### 명명 규칙 정리는 기능 추가만큼이나 중요한 의제다

`setHTML`, `replaceWithHTML`, `beforeHTML`, `prependHTML` 표를 보면
이 제안의 절반은 새 기능이 아니라 정리(cleanup)임을 알 수 있다.
오늘날 개발자는 `innerHTML`, `insertAdjacentHTML('beforebegin', ...)`,
`outerHTML`, `append()`, `prepend()` 같은 서로 다른 시대의 API를 섞어 쓴다.
인자 순서도, 살균 정책도, 반환값도 제각각이다.
새 메서드 묶음은 이 파편화를 단일한 `(html, options)` 시그니처로 흡수한다.

이는 웹 플랫폼이 자주 쓰는 패턴이다.
강력한 신기능(스트리밍)을 도입하면서,
그 김에 인접한 낡은 API들을 일관된 표면으로 다시 깐다.
스트리밍 변형을 정적 변형의 단순한 짝으로 설계한 것도 같은 의도다.
한 번 `appendHTML`을 배우면 `streamAppendHTML`은 추가 학습 없이 따라온다.

## 비평

### 글은 “왜 처리 명령인가”라는 가장 큰 의문을 비켜간다

처리 명령(`<?marker>`, `<?start>`, `<?end>`)은 HTML 작성자에게 낯설다 못해
거의 잊힌 문법이다.
이것은 XML/XHTML 시대의 잔재이며,
일반적인 HTML 파서에서는 주석처럼 무시되거나 이상하게 취급되던 토큰이다.
글은 이를 “참조점으로 활용한다”고만 설명할 뿐,
왜 새로운 요소(예: `<slot>`이나 전용 속성)가 아니라 하필 처리 명령을 골랐는지
정당화하지 않는다.

이 선택에는 분명한 트레이드오프가 있다.
처리 명령은 기존 HTML과 충돌하지 않고 기존 파서를 통과한다는 장점이 있지만,
동시에 IDE 지원, 린터, 템플릿 엔진, CDN의 HTML 최적화 도구 대부분이
이를 인식하지 못한다.
개발자가 `<?marker>`를 손으로 쓸 일은 드물고 서버 프레임워크가 생성하겠지만,
글은 그 생태계 공백을 전혀 다루지 않는다.
독자는 “이 문법이 실무에서 어떻게 안전하게 다뤄지는가”라는 핵심 질문에
답을 얻지 못한 채 글을 마친다.

흥미롭게도 HN 토론에서는 글이 제시하지 않은 이 문법의 정당화가 나왔다.
nofriend가 “새 요소 문법이 가장 대담한 선택인데 왜 필요했는지 모르겠다”고
의문을 제기하자,[^nofriend]
dfabulich는 처리 명령이 HTML 요소 트리의 경계를 가로지르는 DOM 범위(range)를
표시하기 위해 필요하다고 답했다.[^dfabulich-syntax]
예컨대 `<em>` 안에서 시작해 그 밖에서 끝나는 구간처럼,
기존 요소 중첩 규칙으로는 감쌀 수 없는 범위를 `<?start>`/`<?end>`가 표현한다는
것이다.
이는 글 본문이 직접 했어야 할 설명이며,
가장 핵심적인 설계 근거가 공식 글이 아니라 댓글에서 보충되었다는 사실 자체가
이 글의 서술 공백을 방증한다.

같은 문법을 두고 jeroenhd는 더 신랄한 역사적 비판을 던졌다.
“XML 지원을 Chrome에서 막 들어낸 직후에 이런 XML 같은 문법을 보니 우습다.
우리는 Google이 XSLT를 실시간으로 재발명하는 걸 지켜보고 있는데,
이번에는 무슨 이유에서인지 엄청난 양의 JavaScript 접착제를 곁들였다”는
것이다.[^jeroenhd]
이 비판은 플랫폼 진화의 모순을 정확히 찌른다.
선언적 변환이라는 XSLT의 이상은 한때 폐기되었지만,
같은 문제(서버가 보낸 마크업을 선언적으로 변형하기)가 다시 떠오르자
플랫폼은 비슷한 모양의 해법으로 회귀하고 있다.

### “긍정적 피드백”이라는 표현이 표준화 현실을 가린다

글은 “다른 브라우저 벤더로부터 긍정적 피드백”을 받았고 표준화 절차를 밟고 있다고
짧게 언급한다.
이는 Chrome 발 기능 제안에서 거의 정형구처럼 반복되는 표현이며,
실제 상호 운용 합의와는 거리가 멀 수 있다.
긍정적 피드백은 “흥미롭다”는 수준일 수도, “구현하겠다”는 약속일 수도 있다.
글은 이 둘을 구분하지 않는다.

이 모호함은 독자의 의사결정에 직접 영향을 준다.
Chrome 148 플래그 뒤에 있는 기능을,
폴리필이 있다는 이유로 프로덕션에 도입할지 판단하려면
다른 엔진의 실제 입장이 중요하다.
처리 명령 기반 스트리밍은 폴리필로 완벽히 재현하기 어려운 파싱 동작에 의존한다.
글이 Firefox나 WebKit의 구체적 입장, 표준 문서 단계(예: WHATWG 이슈 번호)를
제시하지 않는 한,
“긍정적 피드백”은 도입을 정당화하는 근거가 되지 못한다.

바로 이 공백을 HN 댓글이 메웠다.
ComputerGuru가 “이것은 Google 단독 제안인가, Mozilla는 입장을 밝혔는가”라고
묻자,[^computerguru]
dfabulich는 표준 입장 저장소를 직접 가리키며 구체적으로 답했다.
Mozilla는 공식 입장을 내지 않았지만 의사결정자 한 명(hsivonen)이 찬성에 투표했고,
Apple WebKit은 며칠 전 공식적으로 찬성으로 돌아섰다는 것이다.[^dfabulich-positions]
설계자 nomsternom도 “Mozilla·Apple 인사들 및 이 기능을 쓰려는 여러 외부
파트너와 긴밀히 협력해 설계했다”고 덧붙였다.[^nomsternom-design]
이 정보는 글의 “긍정적 피드백”이라는 모호한 표현보다 훨씬 구체적이며,
실제로 도입을 판단하려는 독자에게 결정적이다.
다만 이는 동시에 비판의 정당성을 입증한다.
독자가 도입 가치를 평가하는 데 꼭 필요한 표준화 현실이
공식 글이 아니라 댓글의 외부 링크를 통해서만 확인되기 때문이다.

### 살균을 기본값으로 두면서도 `Unsafe` 경로를 나란히 제공하는 모순

글은 보안을 강조하며 기본 살균과 `Sanitizer` 옵션을 자랑스럽게 소개한다.
그러나 동시에 `setHTMLUnsafe`, `streamHTMLUnsafe`,
그리고 `runScripts: true` 옵션을 제공한다.
스트리밍 콘텐츠는 본질적으로 서버나 외부 소스에서 조각조각 도착하는 데이터다.
바로 그런 데이터에 스크립트 실행을 허용하는 `Unsafe` 경로를 열어두는 것은
가장 위험한 상황에 가장 위험한 도구를 쥐어주는 셈이다.

문제는 명명이 위험을 충분히 전달하지 못한다는 데 있다.
`Unsafe` 접미사는 경고처럼 보이지만,
스트리밍의 편의성(`getWriter().write(...)`로 즉시 쓰는 흐름)이 워낙 매끄러워서
개발자가 안전 변형과 불안전 변형을 무심코 바꿔 쓸 유인이 크다.
글은 살균을 “기본값”이라 안심시키지만,
실무에서 임의 HTML을 흘려보내려는 개발자는 거의 항상 살균에 막혀
`Unsafe` 변형으로 갈아탄다.
이 현실적인 미끄러짐을 글은 인정하지 않는다.

### 예제가 너무 단순해서 비선형 스트리밍의 진짜 어려움을 숨긴다

글의 예제는 단일 플레이스홀더를 단일 템플릿으로 교체하는 깔끔한 경우다.
그러나 순서 없는 스트리밍의 실제 가치는 복잡한 상황에서 나온다.
중첩된 플레이스홀더, 도착하지 않는 템플릿,
같은 마커를 여러 번 갱신하는 경우, 부분 실패 처리 같은 문제다.
“같은 부모 요소 안에서만 수정 가능”이라는 제약은 한 줄로 언급되지만,
이 제약이 실제 레이아웃에서 얼마나 자주 발목을 잡는지는 다루지 않는다.

특히 “도착하지 않는 콘텐츠”는 스트리밍의 근본 문제다.
서버가 `<?start>`로 “Loading…”을 띄웠는데 연결이 끊기면 어떻게 되는가?
타임아웃은 누가 관리하는가, 임시 콘텐츠는 영원히 남는가?
글은 로딩 표시의 우아함만 보여주고 실패 경로를 보여주지 않는다.
실무 채택을 고민하는 독자에게 정작 필요한 것은 행복한 경로의 예제가 아니라
이런 경계 조건의 명세다.

이 서술 부족은 독자의 기대 어긋남으로 이어졌다.
fg137은 “페이지를 훑었지만 왜 이게 필요한지, 어떤 문제를 푸는지,
브라우저가 왜 콘텐츠 갱신을 떠맡아야 하는지 모르겠다”고 토로했고,[^fg137]
BigJono는 더 구체적인 기대 어긋남을 드러냈다.
제목만 보고 “브라우저 개발자가 직접 만든, 데이터가 바뀌면 DOM 일부를
최고 성능으로 갱신하는 간단한 React 같은 API”를 기대했는데
정작 이건 HTML 파싱 순서에 관한 미세 최적화처럼 보인다는 것이다.[^bigjono]
그는 “왜 15~16년이 지나도록 `Element.append`를 '이 데이터가 바뀌면 이 요소를
갱신하라'로 바꾸는 추상을 유저랜드에서 100번씩 다시 만들고 있느냐”고 물었다.
이 반응은 글이 자신의 범위를 명확히 긋지 못했음을 보여준다.
이 제안은 반응형 상태 관리가 아니라 전달 계층의 문제를 풀지만,
글은 그 경계를 분명히 선언하지 않아 독자가 다른 것을 기대하게 만든다.

물론 이런 회의에 대한 반론도 있었다.
nofriend가 “이건 xhr로 해결하도록 확립된 수십 년의 진보를 거스른다”고 하자,[^nofriend]
xg15는 “그렇다, 더 나은 기술이 없었기 때문이다.
순서 없는 로딩을 위해 100만 개의 명령형 임시방편을 짜는 것이
표준화된 해법보다 낫다는 걸 어떤 경험이 증명했단 말인가”라고 반박했다.[^xg15]
이 대립은 글이 다뤘어야 할 핵심 긴장 ─ 명령형 관행 대 선언적 표준 ─ 을
정확히 드러내지만, 정작 원문은 이 논쟁을 회피한 채 기능 소개로만 일관한다.

## 인사이트

### 이 기능은 SSR 프레임워크의 존재 이유 일부를 플랫폼으로 흡수한다

글은 개발자 편의 기능처럼 보이지만,
더 깊은 차원에서 이것은 프레임워크와 플랫폼 사이의 경계 이동이다.
React Server Components, Astro의 아일랜드, Qwik의 재개 가능성(resumability)은
모두 “순서 없이 도착하는 HTML 조각을 안전하게 끼워 넣는” 문제를
프레임워크 런타임으로 풀어 왔다.
선언적 부분 갱신이 표준화되면,
이 능력의 상당 부분이 브라우저 원시 기능으로 내려온다.

이것이 가져올 2차 효과는 프레임워크의 가치 제안 재편이다.
스트리밍 하이드레이션이 프레임워크의 차별화 요소였다면,
그 차별화가 사라진 자리에서 프레임워크는 다른 가치(상태 관리, 라우팅, DX)로
경쟁해야 한다.
과거 `fetch`가 jQuery의 AJAX를,
flexbox가 그리드 레이아웃 라이브러리를 흡수했듯이,
플랫폼이 프레임워크 기능을 흡수하는 익숙한 주기가 반복된다.
글은 이 경쟁 구도를 전혀 언급하지 않지만,
실제 채택을 좌우하는 것은 바로 프레임워크들이 이 기능을 어떻게 포장하느냐다.

### 처리 명령의 부활은 “버려진 문법의 귀환”이라는 역사적 패턴을 따른다

처리 명령을 콘텐츠 교체 참조점으로 재활용하는 선택은 우연이 아니다.
웹 플랫폼은 새 기능에 새 문법을 만드는 비용을 극도로 꺼린다.
새 요소나 속성은 파서 변경, 보안 검토, 하위 호환성 보장을 요구한다.
반면 처리 명령은 이미 HTML 파서가 인식하지만 거의 쓰이지 않는,
말하자면 “비어 있는 문법 슬롯”이다.

이 패턴은 플랫폼 진화에서 반복된다.
`<template>` 요소 자체가 원래 활성화되지 않는 비활성 DOM 컨테이너로 도입되어
나중에 Web Components의 토대가 되었다.
`data-*` 속성도 임의 데이터를 담는 빈 슬롯이었다.
거의 죽어 있던 문법이 새 의미를 얻고 부활하는 것은,
하위 호환성이 절대 명령인 웹에서 혁신이 일어나는 전형적 경로다.
이 렌즈로 보면 `<?marker>`의 어색함은 결함이 아니라
웹이 진화하는 방식 그 자체의 흔적이다.

### 비선형 전달은 캐싱과 관측 가능성이라는 인프라 문제를 끌고 온다

글은 페이지 로드 성능 관점에서만 이 기능을 본다.
그러나 콘텐츠가 순서 없이 도착하면,
그 사이의 모든 인프라 계층이 재검토를 강요받는다.
CDN은 어떤 조각을 캐싱하고 어떤 조각을 통과시킬지 어떻게 판단하는가?
완전한 HTML 문서를 캐싱하던 모델은,
플레이스홀더만 든 골격과 나중에 채워지는 조각으로 분리된 문서 앞에서 깨진다.

관측 가능성도 마찬가지다.
Largest Contentful Paint나 Cumulative Layout Shift 같은 핵심 지표는
콘텐츠가 언제 어디에 나타나는가에 민감하다.
순서 없는 스트리밍은 정의상 콘텐츠 위치를 사후에 바꾼다.
잘 쓰면 LCP를 앞당기지만, 잘못 쓰면 CLS를 악화시킨다.
글이 “성능 최적화”라고 부른 것은 사실 성능을 새로운 방식으로 협상하는 일이며,
그 협상의 비용은 분석 도구, 에러 추적, 서버 로깅 전반으로 번진다.
이 인프라 파급 효과를 이해하지 못한 채 도입하면,
앱은 빨라 보이지만 운영은 더 불투명해진다.

### “안전한 부분 교체”는 결국 신뢰 경계를 마크업으로 옮기는 일이다

가장 깊은 통찰은 보안 모델의 위치 변화에 있다.
전통적으로 HTML의 신뢰 경계는 서버에 있었다.
서버가 완성한 문서를 보내면 브라우저는 그것을 신뢰하고 렌더링했다.
부분 갱신은 이 경계를 문서 안으로 끌어들인다.
같은 문서 안에서 “이미 신뢰된 콘텐츠”와 “방금 도착해 살균이 필요한 조각”이
공존한다.

이 변화는 XSS 방어의 무게중심을 옮긴다.
`Sanitizer` 객체가 살균 정책을 코드가 아니라 데이터로 표현하게 되면서,
보안 검토의 대상이 “어떤 코드가 HTML을 삽입하는가”에서
“어떤 살균 설정이 어떤 삽입 지점에 적용되는가”로 바뀐다.
이는 장기적으로 더 검사 가능한 모델이지만,
전환기에는 두 모델이 뒤섞인 위험한 회색지대를 만든다.
개발자가 `setHTML`과 `setHTMLUnsafe`를 한 코드베이스에서 섞어 쓰는 순간,
신뢰 경계는 더 이상 한눈에 보이지 않는다.
글이 자랑하는 보안 기본값은,
바로 이 회색지대를 관리할 규율이 함께 따라올 때만 의미를 갖는다.
