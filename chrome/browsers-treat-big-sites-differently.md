# 브라우저는 대형 사이트를 다르게 취급한다

원문: <https://denodell.com/blog/browsers-treat-big-sites-differently>

GeekNews: <https://news.hada.io/topic?id=29513>

HN 토론: <https://news.ycombinator.com/item?id=48136949> (67점, 21개 댓글)

## 요약

Safari와 Firefox는 TikTok, Netflix, Instagram, Amazon 같은 특정 도메인에서
렌더링 방식과 API 동작을 변경하는 코드를 탑재하고 있다.
이 사이트별 예외 처리(site-specific overrides)는 공개 코드베이스에서 확인할 수 있으며,
Firefox의 `about:compat` 페이지와 WebKit의 `Quirks.cpp` 파일이 그 증거다.

Firefox의 WebCompat 시스템은 `about:compat`에서 사용자가 직접 확인하고 토글할 수 있다.
각 항목은 특정 웹사이트를 위한 맞춤형 수정으로, 스크롤 동작, 터치 이벤트 처리,
뷰포트 계산, 이미지 MIME 타입 처리 등 근본적인 동작까지 변경한다.

Safari의 `Quirks.cpp`는 더 구체적이다.
TikTok의 잘못된 업그레이드 메시지 처리, Netflix의 팝오버 문제 수정,
Amazon 제품 이미지 패닝 시 터치·마우스 이벤트 변환,
Facebook·X·Reddit의 Picture-in-Picture 비디오 일시정지 문제 등이 하드코딩되어 있다.

Chrome은 이런 예외 처리 파일이 거의 필요 없다.
웹이 이미 Chrome 중심으로 개발되었기 때문이다.
이는 악순환 구조를 만든다. 개발자가 Chrome 기준으로 사이트를 만들면 Chrome에서 최적으로 동작하고,
다른 브라우저에서 문제가 생기면 사용자들은 브라우저를 탓하며 Chrome으로 이동한다.
이 사용자 이탈을 막기 위해 Firefox와 Safari는 더 많은 예외 처리 코드를 추가하고,
예외 처리의 존재가 개발자에게 보이지 않아 악순환이 반복된다.

저자는 사이트 소유자가 정기적으로 Firefox와 Safari에서 직접 테스트하고,
`about:compat`와 `Quirks.cpp` 포함 여부를 확인할 것을 권고한다.
이 예외 처리는 브라우저 오류 로그에 나타나지 않으므로, 개발자는 의도치 않게
브라우저 특화 패치에 의존하고 있음을 모를 수 있다.

## 분석

### 구현이 명세를 대체하는 메커니즘

웹 표준은 명세(specification)와 구현(implementation) 두 층위에서 존재한다.
이상적으로는 명세가 구현을 이끌어야 하지만, 현실에서는 Chrome의 구현이 실질적 명세가 된다.
이 역전은 점진적으로 일어났다. 개별 개발자의 합리적 선택(Chrome에서 테스트하고 배포)이
집합적 결과로 표준 침식을 가속화한다.

Firefox와 Safari의 예외 처리 코드는 이 역전의 물리적 증거다.
표준이 아니라 Chrome의 동작을 기준으로 예외를 만드는 것은,
표준 기관이 아닌 Chrome 팀이 실질적 표준 위원회 역할을 한다는 의미다.
사용자 에이전트 스푸핑(Safari가 일부 사이트에 Chrome UA를 제공하는 것)은
이 현실을 가장 직접적으로 드러내는 사례다.

### 가시성의 비대칭이 만드는 구조적 문제

예외 처리 코드가 브라우저 오류 로그에 나타나지 않는다는 점이 핵심이다.
개발자가 Firefox나 Safari에서 테스트할 때 사이트가 제대로 동작하면,
그것이 표준을 따르기 때문인지 아니면 브라우저가 예외 처리를 적용했기 때문인지 알 수 없다.
이 불투명성이 의존성을 보이지 않게 만들고, 개발자는 자신이 브라우저 특화 코드에
의존하고 있다는 사실을 인식하지 못한다.

가시성의 비대칭은 피드백 루프를 단절시킨다.
Chrome 중심 개발의 비용이 개발자에게 직접 보이지 않기 때문에, 행동을 바꿀 동기가 생기지 않는다.
비용은 Firefox·Safari 팀과 웹 생태계 전체가 부담하지만, 그 비용은 분산되어 있어 체감되지 않는다.

## 비평

### 강점: 공개 코드로 논증을 뒷받침

이 글의 가장 큰 강점은 주장을 공개 코드베이스로 직접 뒷받침한다는 것이다.
“브라우저가 대형 사이트를 다르게 처리한다”는 주장은 직관적으로 받아들이기 어렵지만,
`about:compat`와 `Quirks.cpp`의 실제 코드를 보여줌으로써 반박 불가능한 증거를 제시한다.
음모론이나 추측이 아니라 공개된 사실이라는 점에서 설득력이 높다.

varun_ch는 `Quirks.cpp`에 claude.ai에 대한 예외 처리가 포함되어 있다는 사실을 지적한다.[^varun_ch]
활발히 개발 중인 서비스가 브라우저 레벨의 예외 처리 대상이 된다는 것은,
대형 사이트로 분류되는 순간 브라우저 엔지니어가 그 서비스의 버그를 직접 수정하게 된다는 의미다.
이 역할 역전은 흥미롭다. 웹 표준의 수호자인 브라우저 엔진이 개별 서비스의 유지보수 담당자가 되는 것이다.

tracker1은 브라우저별 예외 처리 문제가 대형 사이트에만 국한되지 않음을 지적한다.[^tracker1]
Linux를 주 데스크톱으로 사용하다 보면 사용자 에이전트 문자열에 “Linux”가 포함되었다는 이유로
작동하지 않거나 다르게 동작하는 사이트를 만나는 경우가 있다.
특정 OS, 특정 브라우저, 심지어 특정 버전을 기준으로 사이트 경험이 달라지는 현상은
`Quirks.cpp`로 공식 문서화된 예외보다 훨씬 광범위하게 존재한다.

### 약점: Chrome의 대안을 논의하지 않는다

글은 Chrome 중심의 악순환을 진단하지만, 이를 어떻게 끊을 수 있는지에 대한 구조적 논의가 없다.
사이트 소유자의 개인적 테스트 습관 개선을 권고하는 것은 집합 행동 문제(collective action problem)에
개인 해법을 제시하는 것이다. Firefox나 Safari의 시장점유율이 없는 상황에서
사이트 소유자가 멀티 브라우저 테스트에 투자할 인센티브는 구조적으로 취약하다.

### 약점: 예외 처리의 긍정적 측면을 다루지 않는다

브라우저의 사이트별 예외 처리는 표준 침식의 증거이기도 하지만, 동시에 사용자 경험을
지키기 위한 실용적 대응이기도 하다. 표준을 지키다가 사용자가 Chrome으로 이동하는 것보다
예외 처리로 사용자를 유지하는 것이 브라우저 다양성에 더 기여할 수도 있다.
이 트레이드오프에 대한 논의가 없어 결론이 다소 일면적이다.

## 인사이트

### 표준의 실질적 권위는 시장점유율에 비례한다

W3C와 WHATWG 같은 표준 기관이 HTML, CSS, JavaScript 명세를 정의하지만,
실질적으로 개발자가 따르는 표준은 가장 많은 사용자가 쓰는 브라우저의 구현이다.
이것은 웹 생태계만의 문제가 아니다.

모든 기술 표준은 명목상 권위와 실질적 권위 사이의 긴장을 내포한다.
RFC가 정의하는 HTTP 표준과 nginx·Apache가 실제로 구현하는 HTTP는 다르다.
SQL 표준과 MySQL·PostgreSQL의 방언은 다르다.
표준은 실질적 권위를 가진 구현이 등장하는 순간, 그 구현을 소급하여 정당화하는 역할로 전락하는 경향이 있다.

Chrome이 웹의 실질적 표준이 된 것은 기술적 우월성보다 Google의 배포 능력과
Android 기본 브라우저 지위, Gmail·YouTube 같은 대형 서비스와의 통합 덕분이다.
표준 경쟁은 기술 경쟁이 아니라 배포 경쟁이다.

### 기술 독점의 비용은 분산되어 보이지 않는다

Chrome의 사실상 독점이 갖는 비용은 집중되지 않고 분산된다.
Firefox 팀이 예외 처리 코드를 유지하는 비용, Safari 팀이 `Quirks.cpp`를 관리하는 비용,
웹 개발자가 브라우저 호환성 버그를 디버깅하는 시간, 사용자가 특정 브라우저에 종속되는 자유 감소.
이 비용들은 누구도 직접 청구하지 않기 때문에 시장 신호로 나타나지 않는다.

이것이 기술 독점이 유지되는 핵심 메커니즘이다.
독점의 비용이 독점 기업에게 귀속되지 않고 생태계 전체에 분산될 때,
독점을 유지하는 것이 집합적으로 비효율적이어도 개별 행위자의 합리적 선택이 독점을 강화한다.
Chrome을 선택하는 개발자와 사용자는 각자의 입장에서 합리적이지만,
그 합리적 선택들의 합이 비합리적 결과를 만든다.

### 브라우저 다양성은 인프라 다양성의 문제다

이 글이 가장 깊이 파고들지 못한 함의는 브라우저 다양성이 단순한 기술적 문제가 아니라는 것이다.
인터넷이 소수의 브라우저 엔진(Blink, WebKit, Gecko)에 의존하는 상황에서,
그 엔진 중 하나가 사실상 표준이 되면 인터넷의 진화 방향이 단일 기업의 결정에 종속된다.

Google이 Privacy Sandbox를 통해 쿠키를 대체하려 했을 때 광고 생태계 전체가 흔들린 것,
AMP(Accelerated Mobile Pages)가 구글 검색 순위와 연동되면서 웹 구조를 변화시킨 것은
이 종속성의 실제 사례다. 브라우저 다양성의 쇠퇴는 미래 인터넷의 진화 경로에 대한
통제권이 한 기업에 집중된다는 의미다. `Quirks.cpp`의 각 항목은 그 집중의 비용을 보여주는 영수증이다.
