# Chrome 2주 릴리스 사이클: 더 빠르게, 더 작게

- 출처: [Get features faster with Chrome's two-week release
  cycle](https://developer.chrome.com/blog/chrome-two-week-release)
- 게시일: 2026년 · Chrome for Developers Blog

---

## 핵심 요약

Chrome이 2026년 9월부터 **4주 릴리스 사이클을 2주로 단축**한다.
Chrome 153 Stable(9월 8일)이 첫 번째 2주 사이클 릴리스다.

릴리스가 두 배 빨라지는 것이 아니다. **릴리스당 변경 범위가 절반이
되는 것이다.** 이 차이가 모든 것을 바꾼다.

---

## 변경 사항

| 항목             | 기존           | 변경 후        |
|------------------|----------------|----------------|
| Stable 주기      | 4주            | 2주            |
| Beta 선행 기간   | 4주 전         | 3주 전         |
| Extended Stable  | 8주            | 8주 (유지)     |
| Dev/Canary       | 변경 없음      | 변경 없음      |
| 적용 플랫폼      | -              | Desktop/Android/iOS 동일 |

- **첫 적용**: Chrome 153 Stable, 2026년 9월 8일
- **Extended Stable**: 엔터프라이즈 관리자·Chromium 임베더를 위해
  8주 사이클 유지

---

## Chrome 릴리스 역사

- **2021년**: 6주 → 4주 단축
- **2023년**: 주간 보안 업데이트 도입, 패치 갭 개선
- **2023년**: Early Stable 도입(릴리스 품질 향상)
- **2026년**: 4주 → 2주

---

## 인사이트

### 1. "더 빠른 릴리스"가 아니라 "더 작은 릴리스"다

릴리스 주기 단축의 본질은 속도가 아니라 **변경 단위의 축소**다.
기능 하나가 stable에 도달하기까지 걸리는 절대 시간은 크게
다르지 않다. 차이는 각 릴리스에 포함되는 변경 수가 절반으로
줄어든다는 것이다. 변경이 작으면 회귀(regression)를 추적하기
쉽고, 롤백 비용도 낮다. **릴리스를 더 자주 하는 게 아니라
릴리스를 더 안전하게 하는 전략이다.**

### 2. 패치 갭(Patch Gap) 문제의 구조적 해결

보안 취약점이 발견되면 코드는 곧 공개되지만(Chromium 오픈소스)
사용자에게 전달되기까지 최대 4주가 걸렸다. 이 기간이 패치 갭이고,
공격자에게는 황금 시간이다. 2주 사이클은 이 창을 이론적으로
절반으로 줄인다. 2023년에 도입한 주간 보안 업데이트가 임시방편이었다면,
**2주 사이클은 구조적 해결책이다.**

### 3. Extended Stable 8주 유지의 의미

엔터프라이즈는 각 업데이트를 내부 테스트하고 승인하는 데 시간이
필요하다. Extended Stable 8주 유지는 Google이 두 고객군을
분리해서 관리하겠다는 선언이다. **소비자 브라우저는 민첩하게,
기업 환경은 안정적으로.** 같은 제품이 다른 릴리스 계약을 맺는
이중 전략이 정착한 것이다.

### 4. Canary/Dev 미변경: 파이프라인 압축이 아니다

Dev와 Canary 채널은 변경이 없다. 이 채널들은 이미 더 빠른
주기로 돌고 있다. 즉 이번 변경은 실험 단계를 압축하는 게 아니라,
**stable로 수렴하는 구간을 좁히는 것**이다. 피처 플래그 뒤에서
익어가는 시간은 동일하고, 문이 더 자주 열릴 뿐이다.

### 5. 개발자에게는 테스트 주기 압박이다

4주마다 beta를 확인하면 됐던 것이 2주마다가 된다. CI에서
Chrome Beta를 대상으로 테스트하지 않고 있었다면 지금이
시작할 때다. **회귀를 사용자보다 먼저 발견하는 유일한
방법은 beta 채널을 파이프라인에 넣는 것이다.**

### 6. 기능 출시 예측 가능성이 높아진다

4주 사이클에서는 피처 플래그를 켜는 타이밍을 놓치면 한 달을
기다렸다. 2주 사이클은 그 대기 시간이 절반이다. 웹 플랫폼
기능에 의존하는 개발 일정의 **불확실성이 줄어든다.** Chrome
Status 로드맵을 보면 어떤 기능이 몇 주 안에 들어오는지
더 정밀하게 계획할 수 있다.

### 7. 릴리스 엔지니어링이 성숙했다는 신호

2021년 6→4주 전환, 2023년 보안 업데이트 주간화, 그리고
2026년 4→2주. Google은 각 단계마다 인프라를 먼저 검증하고
속도를 올렸다. 이번 전환의 전제는 **자동화된 회귀 감지,
빠른 롤아웃 제어, 채널별 인구 관리**가 충분히 신뢰할 수
있다는 Google의 자체 확신이다. 릴리스 엔지니어링을 하나의
경쟁력으로 키워온 결과가 사이클 단축으로 나타난다.
