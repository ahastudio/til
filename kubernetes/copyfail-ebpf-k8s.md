# copyfail-ebpf-k8s

<https://github.com/iwanhae/copyfail-ebpf-k8s>

## 소개

`copyfail-ebpf-k8s`는 CVE-2026-31431(Copy.Fail) 취약점을 Kubernetes 클러스터 전체에
즉시 완화하는 eBPF 기반 DaemonSet이다.
한국 개발자 iwanhae가 2026년 5월 2일에 공개했으며,
같은 날 GeekNews에 소개됐다.

CVE-2026-31431은 `AF_ALG` 암호화 소켓(`algif_*` 커널 모듈)을 통해
읽기 가능한 파일의 캐시된 복사본을 조작할 수 있는 취약점이다.
로컬 권한 상승(root exploit), 컨테이너·샌드박스 탈출로 이어진다.
표준 우회책인 `algif_aead` 커널 모듈 비활성화는 커널에 빌트인된 경우
(Fedora, Oracle Linux, 기타 RHEL 계열)에는 동작하지 않는다.

커맨드 하나로 클러스터 전체를 방어한다.

```bash
kubectl apply -f k8s-daemonset.yaml
```

## 아키텍처

```text
                   Kubernetes Node
 ┌──────────────────────────────────────────┐
 │  DaemonSet Pod (privileged)              │
 │  ┌────────────────────────────────────┐  │
 │  │  copyfail-mitigation (loader)      │  │
 │  │   - BPF LSM 지원 여부 자동 감지    │  │
 │  │   - eBPF 프로그램 로드 및 attach   │  │
 │  │   - BPF ring buffer로 이벤트 로그  │  │
 │  └──────────────┬─────────────────────┘  │
 │                  │ libbpf                 │
 │  ┌──────────────┴─────────────────────┐  │
 │  │  Kernel                            │  │
 │  │  [LSM] lsm/socket_create → EPERM   │  │
 │  │         OR                         │  │
 │  │  [tp]  sys_enter_socket → SIGKILL  │  │
 │  └────────────────────────────────────┘  │
 └──────────────────────────────────────────┘
```

두 eBPF 프로그램 중 커널 능력에 따라 자동 선택된다.

| 프로그램                    | 훅                        | 동작                 | 요건             |
| --------------------------- | ------------------------- | -------------------- | ---------------- |
| `ebpf-alg-socket-filter`    | `lsm/socket_create`       | `EPERM` 반환(거부)   | BPF LSM 활성화   |
| `ebpf-alg-socket-killer`    | `tracepoint/sys_enter_socket` | `SIGKILL` 전송  | 트레이스포인트   |

root를 포함한 모든 UID의 `AF_ALG` 소켓 생성이 차단된다.
커널 내부 소켓은 건너뛴다.
차단 이벤트는 BPF ring buffer로 stdout에 기록된다.

## 분析

### 패치 없는 완화: 배포 현실에 맞춘 설계

CVE-2026-31431의 커널 패치는 존재하지만, 프로덕션 Kubernetes 클러스터에서
커널을 즉시 교체하는 것은 현실적으로 어렵다.
유지보수 창이 필요하고, 노드 드레인과 재부팅이 뒤따른다.
대형 클러스터일수록 롤링 업데이트에 시간이 걸린다.

이 프로젝트는 그 현실 격차를 메운다.
DaemonSet은 재부팅 없이 모든 노드에 즉시 배포되고,
eBPF 프로그램은 커널 재컴파일 없이 커널 내부에서 동작한다.
“즉각 완화, 커널 패치는 별도로”라는 두 단계 접근을 가능하게 한다.

### BPF LSM과 트레이스포인트 폴백: 커널 가변성 처리

동일한 목표를 두 가지 다른 훅으로 구현한 것은 Kubernetes 클러스터의
커널 이질성을 처리하는 실용적 선택이다.
AWS EKS, GKE, 온프레미스 클러스터는 서로 다른 커널 버전과 설정을 가진다.
BPF LSM이 활성화된 최신 커널에서는 `EPERM` 반환으로 깔끔하게 거부하고,
구형 커널에서는 트레이스포인트를 통해 `SIGKILL`로 프로세스를 종료한다.

후자는 더 공격적인 접근이지만, 취약점이 악용되는 것보다는 프로세스가
종료되는 편이 낫다는 판단이다.
로더가 자동으로 최적 프로그램을 선택하므로 운영자가 커널 버전마다 설정을 바꿀 필요가 없다.

### CO-RE: 커널 버전 독립성

eBPF 프로그램이 BTF/CO-RE(Compile Once - Run Everywhere)를 사용한다는 점이
운영 가능성을 높인다. CO-RE 이전 eBPF는 실행 커널 버전에 맞게 컴파일해야 했다.
CO-RE로 빌드된 프로그램은 BTF(BPF Type Format)를 통해 런타임에
커널 내부 구조를 파악하므로, 컨테이너 이미지 하나로 다양한 커널 버전에서 동작한다.
`x86_64`/`amd64`와 `aarch64`/`arm64` 두 아키텍처를 모두 지원한다.

## 비평

### 강점: 단일 커맨드 완화의 운영 가치

`kubectl apply -f k8s-daemonset.yaml` 한 줄은 개념적으로 단순하지만,
실제 보안 운영에서 이 단순함의 가치는 크다.
보안 담당자가 취약점 공지를 받고 몇 분 안에 전체 클러스터를 방어할 수 있다.
README에 검증 명령(`kubectl get daemonset`, `kubectl logs`)이 포함돼 있어
배포 후 작동 여부 확인도 간단하다.

### 약점: 권한 있는 DaemonSet의 위험

이 솔루션은 `privileged: true`와 `/sys/fs/bpf` 마운트를 요구한다.
권한 있는 DaemonSet은 그 자체로 공격 벡터가 될 수 있다.
eBPF 로더가 손상되거나 컨테이너 이미지가 교체되면 오히려 모든 노드에 대한
권한 있는 접근을 공격자에게 제공할 수 있다.
이미지 다이제스트 고정, RBAC 제한, 네트워크 정책 적용이 필수 운영 전제다.

## 인사이트

### eBPF가 보안 완화의 표준 도구가 되는 흐름

이 프로젝트는 보안 취약점 완화에 eBPF를 사용하는 더 큰 흐름의 일부다.
Cilium의 네트워크 정책, Tetragon의 런타임 보안, Falco의 이상 탐지 모두
같은 방향이다 — 커널을 수정하지 않고 eBPF로 커널 내부에서 감시하고 제어한다.

eBPF의 장점은 “커널 패치 없이 커널 수준에서 동작”이다.
이전에는 커널 수준 보안 제어를 위해 커널 모듈 로딩(불안정)이나
커널 패치(재부팅 필요)가 유일한 선택이었다.
eBPF는 그 중간 지점을 만들었다 — 런타임에 안전하게 로드하고, 검증자가 안전성을
확인하고, 재부팅 없이 동작한다.
CVE-2026-31431 완화를 eBPF로 구현하는 것은 이 패러다임의 자연스러운 적용이다.

### 한국 보안 커뮤니티의 오픈소스 기여 패턴

iwanhae는 CVE 공지 당일(또는 직후) eBPF 완화 도구를 오픈소스로 공개했다.
보안 취약점 공개 직후 커뮤니티가 완화 도구를 즉시 만들어 공유하는 이 패턴은
취약점 대응의 집단 지성 모델이다.
공식 배포판의 패치가 나오기까지 걸리는 시간 동안, 커뮤니티 완화 도구가
그 공백을 메운다. RHEL 계열 배포판에서 커널 모듈 비활성화가 불가능하다는
구체적 맥락을 README에 명시한 것을 보면, 실제 프로덕션 환경의 문제에서
출발했음을 알 수 있다.

### 패치 경보 → 즉각 완화 → 커널 업데이트의 3단계 모델

이 프로젝트가 제시하는 운영 모델은 더 일반화할 수 있다.
커널 취약점의 생애 주기는 보통 세 단계다.
(1) 공개 → (2) 임시 완화 → (3) 커널 패치.
기존에는 (1)과 (3) 사이에 할 수 있는 것이 제한됐다.
eBPF는 (2) 단계를 실용적으로 만든다.

Kubernetes 운영자 관점에서, “커널 취약점이 공개되면 eBPF DaemonSet으로
즉각 완화하고, 다음 유지보수 창에 커널을 업데이트한다”는 워크플로우가
표준이 될 가능성이 있다. copyfail-ebpf-k8s는 이 워크플로우의 구체적 사례로서,
비슷한 도구들이 앞으로 취약점마다 더 빠르게 나올 것이라는 신호다.
