# 나는 쿠버네티스를 브라우저로 포팅했다

원문: <https://ngrok.com/blog/i-ported-kubernetes-to-the-browser>

HN 토론: <https://news.ycombinator.com/item?id=48738985> (326점, 102개 댓글)

## 요약

ngrok의 시니어 개발자 에듀케이터 샘 로즈(Sam Rose)가 webernetes라는
프로젝트를 공개했다.
쿠버네티스(Kubernetes)를 TypeScript로 부분 포팅해 브라우저 안에서
클러스터를 실행할 수 있게 만든 것이다.
2개월에 걸쳐 552개의 커밋, 629개의 파일에 걸쳐 거의 10만 줄의 코드를
생성했다.

먼저 저자는 가장 많이 받은 질문에 답한다.
“쿠버네티스를 WebAssembly로 컴파일한 것 아니냐”는 질문이다.
답은 “아니오”다.
“Hello, world!”만 출력하는 Go 프로그램을 WebAssembly로 컴파일해도
gzip 압축 후 약 540KiB인데, webernetes 전체는 약 140KiB에 불과하다.
게다가 쿠버네티스는 브라우저에서 쓸 수 없는 시스템 레벨 API를 호출하기
때문에 컴파일 자체가 되지 않는다.
대신 webernetes는 kubelet 바이너리의 부분 포팅, 파드 스케줄러·네임스페이스
컨트롤러·kube-proxy·디플로이먼트 컨트롤러 등 여러 컨트롤러의 포팅,
브라우저 기반 컨테이너 네트워크 인터페이스(CNI), 브라우저 기반 컨테이너
런타임, 그리고 클러스터와 상호작용하는 API로 구성된다.

크기를 작게 유지하기 위해 Docker Hub 같은 실제 레지스트리에서 이미지를
받아오지 않고, TypeScript API로 이미지를 직접 정의하는 자체 브라우저 기반
레지스트리를 사용한다.
`w8s.BaseImage`를 상속한 클래스로 컨테이너 이미지를 정의하고,
`cluster.apply()`로 실제 쿠버네티스 매니페스트(Deployment 등)를 적용하며,
`kubernetes-client/javascript`와 호환되는 API로 파드 목록 조회, 인포머
(informer) 등록, 요청·응답 이벤트 구독을 할 수 있다.
저자는 webernetes가 프로덕션용 쿠버네티스 배포판이 아니라, 인터랙티브한
쿠버네티스 학습 콘텐츠를 만들기 위한 도구라고 명시한다.
현재 ConfigMap, Secret, 파드 리소스 제한, 퍼시스턴트 볼륨 등은
지원하지 않는다.

이 글의 핵심 주제는 “이것이 슬롭(slop)이 아니냐”는 질문에 대한 답변이다.
코드 대부분을 LLM이 작성했기 때문이다.
저자는 두 가지로 이 우려에 답한다.
모든 코드를 직접 리뷰했다는 것과, 실제 클러스터와 동일하게 동작하는지
검증하는 수백 개의 테스트를 만들었다는 것이다.
LLM이 Go 코드를 TypeScript로 포팅하는 과정에서 반복적으로 저지른 실수는
세 가지 유형으로 분류된다.
지름길(shortcut, 예: 다양한 캐시 구현을 `Map` 하나로 뭉뚱그림),
과도한 친절(요청하지 않은 헬퍼 함수를 만들어 원본과의 대조를 어렵게 만듦),
그리고 누락(테이블 테스트의 일부 케이스를 이유 없이 빠뜨림)이다.

검증을 위해 저자는 `kubernetes-client/javascript`(쿠버네티스 공식
JavaScript 클라이언트)를 채택하고, 동일한 테스트 코드를 k3s 클러스터와
webernetes 양쪽에 대해 실행하는 통합 테스트를 작성했다.
현재 204개의 통합 테스트와 1,855개의 단위 테스트가 있으며,
단위 테스트 대부분은 쿠버네티스 Go 코드베이스에서 직접 포팅한 것이다.
프로젝트 진행 과정을 담은 그래프도 공개했는데,
마지막 주(6월 15일 주간)에 코드 라인이 급증(2만 8천 줄 순증가)했고,
같은 주 LLM 토큰 비용이 API 환산 기준 1,811.64달러로 최고치를 찍었다.
저자는 이 마지막 주에 서브에이전트 팀을 동원해 디플로이먼트 지원을
포팅했다고 밝히며, 작업은 빨랐지만 토큰 효율은 매우 나빴다고 인정한다.
그럼에도 프로젝트 전체에서 가장 비싼 항목은 여전히 저자 자신의
시간이었다고 결론짓는다.

## 분석

### “슬롭이 아니다”라는 주장은 검증 가능성을 증명 수단으로 삼는다

이 글의 논증 구조는 방어적이다.
저자는 처음부터 “사람들이 이것을 조회수를 위한 슬롭 포팅이라고 의심할
것”이라고 선제적으로 인정한 뒤, 그 의심에 반박하는 두 가지 증거,
전체 코드 리뷰와 행동 동등성 테스트를 제시한다.
이는 “LLM이 많이 관여했다”는 사실 자체를 문제 삼는 대신,
그 관여가 검증 가능한 절차를 거쳤는지를 판단 기준으로 삼는 논리다.

이 논증이 흥미로운 지점은, 저자가 자신의 사례를 Anthropic의 C 컴파일러
포팅이나 Bun의 Zig-to-Rust 포팅 사례와 대비시킨다는 점이다.
그 사례들은 이미 존재하는 자동 검증 수단(비교 대상 컴파일러, 신뢰할 수
있는 대규모 테스트 스위트)이 있었지만, 자신에게는 그런 인프라가 없었다고
인정한다.
그래서 검증 인프라 자체를 새로 만드는 것(k3s와의 행동 비교 테스트)이
이 프로젝트의 핵심 작업이 되었다는 논리 전개는, “LLM 코드를 신뢰할 수
있는가”라는 질문이 사실은 “검증 인프라를 얼마나 갖추고 있는가”라는
질문으로 치환될 수 있음을 보여준다.

### 크기 최적화의 근거가 아키텍처 선택 전체를 정당화한다

WebAssembly 컴파일을 하지 않고 선택적 포팅을 택한 이유로 제시된 근거는
단 하나의 숫자다.
“hello world” Go 바이너리의 WebAssembly 컴파일 결과물이 540KiB인데
webernetes 전체가 140KiB라는 것이다.
이 하나의 비교가 이 프로젝트의 근본적인 아키텍처 결정, 즉 컴파일이 아닌
포팅을 선택한 이유를 설명하는 유일한 논거로 제시된다.

이 논리 전개 방식은 이 프로젝트가 학습 콘텐츠 제작이라는 목적에
얼마나 철저하게 최적화되어 있는지를 보여준다.
프로덕션 환경에서라면 크기보다 기능 완결성이나 실제 쿠버네티스와의
API 호환성이 우선시될 수 있지만, 브라우저에서 즉시 로드되어 인터랙티브
데모를 보여줘야 하는 목적에서는 번들 크기가 다른 모든 선택을 지배하는
제약 조건이 된다.
목적이 아키텍처를 결정한다는 원칙이 이 글에서 특히 선명하게 드러난다.

### 테스트가 “무엇을 검증하는가”가 신뢰의 실제 근거다

저자가 강조하는 테스트 전략은 단순히 “테스트를 많이 만들었다”가 아니라
“동일한 테스트 코드를 실제 쿠버네티스 클러스터(k3s)와 webernetes 양쪽에
대해 실행한다”는 점에 있다.
이는 일반적인 단위 테스트가 “코드가 의도한 대로 동작하는가”를 확인하는
것과 다르게, “포팅된 시스템이 원본과 동일하게 동작하는가”라는
행동 동등성(behavioral equivalence)을 직접 검증한다.

이 설계가 성립하려면 `kubernetes-client/javascript`라는 공용 인터페이스가
있어야 한다.
이 공용 API가 있었기에 동일한 테스트 코드를 두 백엔드(webernetes, k3s)에
대해 재사용할 수 있었다.
즉 이 검증 전략은 우연히 만들어진 것이 아니라, 포팅 대상 시스템이 이미
안정적인 공개 API 표준을 가지고 있었기에 가능했던 특수한 조건에
의존하고 있다.

HN 댓글에서도 정확히 이 지점이 지적됐다.
한 사용자는 이 프로젝트가 “말 그대로 대조할 수 있는 명세(spec)가
이미 존재하는 특수한 사례”이며, 대다수의 코딩 작업에는 그런 기회가
주어지지 않는다고 짚었다[^ambicapter].
비슷한 시기에 fd.io VPP를 Codex의 도움으로 Rust로 포팅하고 있다고 밝힌
다른 사용자도, 명확한 인터페이스와 잘 정의된 테스트 케이스가
포팅 성공의 필수 조건이라는 같은 결론에 도달했다[^mintflow].
서로 다른 도메인(쿠버네티스, 네트워크 데이터 플레인)에서 독립적으로
동일한 결론에 이르렀다는 사실은, 이 조건이 webernetes만의 특수 사정이
아니라 LLM 기반 포팅 전반에 적용되는 일반 원칙에 가깝다는 것을
보여준다.

## 비평

### “LLM sucks at porting code”라는 결론은 검증되지 않은 프롬프트 전략 위에서 내려졌다

저자는 스스로 예상하듯 “스킬 이슈 아니냐”는 반박에 직면할 만한 주장을
편다.
지름길, 과도한 친절, 누락이라는 세 가지 실패 유형을 반복적으로 관찰했다고
하면서도, 이 결론에 이르기까지 실제로 어떤 프롬프트 전략을 시도했는지,
컨텍스트를 얼마나 제공했는지, 파일을 얼마나 잘게 쪼개서 작업했는지에
대한 구체적인 방법론은 거의 공개하지 않는다.

저자 스스로도 “완벽한 원샷 프롬프트를 보여주면 좋겠다”고 인정하는
대목에서 이 공백을 알고 있음을 드러낸다.
그러나 이는 자신의 실패 사례를 인정하는 겸손한 태도로 읽히는 동시에,
“LLM은 코드 포팅에 서투르다”는 일반화된 주장의 근거가 사실은 특정
프롬프트 전략, 특정 도구(Codex, Claude), 특정 작업 방식(파일 단위 포팅)의
조합에서 나온 결과일 수 있다는 가능성을 열어 둔다.
방법론이 불투명한 상태에서 “LLM은 이런 실수를 반복한다”는 결론을
일반화하는 것은, 검증되지 않은 변수를 통제하지 않은 실험에서
결론을 내리는 것과 비슷한 위험을 안고 있다.

### 마지막 주의 “서브에이전트 팀” 사례는 저자 자신의 핵심 방법론과 모순된다

이 글이 처음부터 끝까지 강조하는 신뢰의 근거는 “모든 코드를 직접
리뷰했다”는 것이다.
그런데 마지막 주에 대한 서술에서는, 디플로이먼트 지원을 포팅하기 위해
“에이전트 팀을 동원해 의존성 체인을 파악하고, 각 컴포넌트를 더 많은
서브에이전트로 포팅한 뒤, 또 다른 서브에이전트 세트로 전체를
리뷰했다”고 밝힌다.

이는 “내가 모든 줄을 직접 검토했다”는 앞선 주장과 미묘하게 어긋난다.
서브에이전트가 서브에이전트의 결과물을 리뷰하는 구조에서, 저자 본인의
직접적인 라인 단위 검토가 앞선 몇 달 동안의 작업과 동일한 밀도로
이루어졌다고 보기는 어렵다.
저자는 “여전히 마지막에 직접 리뷰를 했다”고 언급하지만, 에이전트가
에이전트를 검토하는 계층을 몇 겹이나 거친 뒤의 최종 리뷰가 처음부터
끝까지 저자가 손수 대조하던 리뷰와 같은 신뢰 수준을 제공하는지는
이 글에서 답하지 않는다.
“토큰 효율이 매우 나빴다”는 인정은 있지만, “검증 밀도가 낮아졌을
가능성”에 대한 인정은 없다.

### “내 시간이 가장 비쌌다”는 결론은 검증 불가능한 진술로 남는다

글은 “내 시간이 프로젝트에서 가장 비싼 항목이었다”는 결론으로 끝맺는다.
이 결론은 사람의 노동력이 여전히 핵심 자원이라는, 안심이 되는 메시지를
전달한다.
그러나 이 비교의 계산 방식은 검증할 수 없다.
저자의 시간당 가치를 어떻게 환산했는지, 리뷰에 실제로 몇 시간이
들었는지에 대한 구체적인 숫자는 전혀 제시되지 않은 채 결론만
서술된다.

반면 LLM 토큰 비용은 API 환산 기준으로 주 단위 그래프까지 그려가며
투명하게 공개된다.
한쪽은 정밀한 숫자로, 다른 한쪽은 뭉뚱그린 주장으로 제시되는 비대칭은
독자가 “정말로 사람의 시간이 더 비쌌는지”를 스스로 검증할 방법을
차단한다.
이 글의 다른 모든 주장이 수치와 테스트로 뒷받침되는 것과 비교하면,
정작 결론부의 핵심 주장 하나만 검증 불가능한 진술로 남아 있다는 점은
이 글의 전반적인 투명성 기조와 어긋난다.

### “브라우저로 포팅했다”는 제목은 실제로 이식된 범위를 과장한다

한 HN 사용자는 이 제목 자체에 근본적인 의문을 제기했다.
쿠버네티스가 오케스트레이션하는 실제 컨테이너 이미지가 브라우저에서
그대로 실행되는 것은 아니지 않느냐는 것이다.
DOOM을 브라우저로 포팅했다면 실제로 그 게임을 브라우저에서 플레이할 수
있어야 하는 것처럼, “쿠버네티스를 브라우저로 포팅했다”는 표현이 성립하려면
실제 백엔드 컨테이너 서비스가 브라우저 안에서 그대로 동작해야 하는데,
이 프로젝트는 그렇지 않다는 지적이었다[^playorizaya].

이에 대한 반박은 본문에 이미 답이 있다는 것이었다.
포팅된 것은 실제 컨테이너 이미지가 아니라 스케줄러, kube-proxy,
디플로이먼트 컨트롤러 같은 제어 평면(control plane)이며, 이 부분은
실제 Go 소스 코드를 그대로 옮기고 동일한 클라이언트 API로 k3s와의
행동 동등성까지 검증했다는 것이다.
파드 사이를 오가는 점으로 시각화되는 “렌더링”은 그 위에 얹힌 데모 앱일
뿐이라는 설명이었다[^firesteelrain].
이 논쟁은 사소한 트집이 아니다.
“포팅”이라는 단어가 정확히 무엇을 가리키는지에 대한 암묵적 동의가
없다면, 이 프로젝트의 성과를 과대평가하거나 과소평가하는 두 가지 오독이
모두 가능해진다.
제목이 만들어 내는 첫인상과, 본문이 신중하게 설명하는 실제 범위 사이의
간극을 저자가 먼저 명확히 정리했어야 할 대목이다.

## 인사이트

### 쿠버네티스의 복잡성을 재현하는 일은 그 복잡성이 필수적인지를 되묻게 한다

이 프로젝트가 쿠버네티스의 핵심 개념(스케줄러, 컨트롤러, CNI, CRI)을
축소판으로나마 재현하는 데 성공했다는 사실은, 쿠버네티스의 복잡성에 대한
해묵은 농담을 다시 소환한다.
HN 댓글 중 하나는 이 시점에서 프레드 브룩스(Fred Brooks)의 필수적
복잡성(essential complexity)과 부수적 복잡성(accidental complexity)
구분을 끌어와, 쿠버네티스가 원래 해결하려는 문제(분산 상태의 조정과
자기 복구)에는 그 정도의 복잡성이 필수적일 수 있지만, 더 단순하게 처리할
수 있는 문제에 쿠버네티스를 갖다 쓰는 순간 그 복잡성은 곧바로
부수적인 것으로 바뀐다고 짚었다[^jaggederest].

webernetes가 역설적으로 이 구분을 시각적으로 증명한다.
브라우저에서 140KiB로 시연 가능할 만큼 핵심 로직을 추출해 낼 수
있었다는 사실은, 쿠버네티스의 필수적 복잡성이 생각보다 작은 핵을
이루고 있고, 실무에서 체감하는 방대한 복잡성 상당 부분은 그 핵을 감싼
운영·확장성·안전장치 계층, 즉 부수적 복잡성에서 온다는 것을
암시한다.
교육용 축소판이 오히려 원본 시스템의 복잡성 구조를 해부해 보여주는
셈이다.

### “검증 인프라를 새로 만드는 비용”이 LLM 코딩의 진짜 병목이 되고 있다

이 글에서 가장 중요한 대목은 저자가 스스로 인정하는 비대칭이다.
Anthropic이나 Bun 팀은 이미 존재하는 검증 수단(비교 컴파일러, 방대한
테스트 스위트) 덕분에 대규모 LLM 생성 코드를 안심하고 병합할 수 있었지만,
저자에게는 그런 인프라가 없었다.
그래서 프로젝트의 실제 작업량 상당 부분이 “쿠버네티스를 포팅하는 것”이
아니라 “포팅 결과를 검증할 수 있는 이중 실행 테스트 하네스를 만드는 것”에
들어갔다.

이는 LLM 코딩의 병목이 코드 생성 속도에서 검증 인프라의 존재 여부로
이동하고 있다는 신호로 읽을 수 있다.
앞으로 “이미 검증 인프라가 갖춰진 도메인”(잘 알려진 언어 컴파일러,
표준화된 API를 가진 시스템)에서는 LLM을 활용한 대규모 포팅이 급속히
확산되겠지만, 검증 인프라가 없는 도메인에서는 그 인프라를 새로 만드는
비용이 LLM이 절약해 주는 타이핑 비용을 상당 부분 상쇄할 것이다.
결국 “LLM으로 얼마나 빨리 만들 수 있는가”라는 질문은 점점 더
“이 도메인에 이미 신뢰할 수 있는 오라클(oracle)이 존재하는가”라는
질문으로 수렴하게 된다.

### 캐시된 토큰의 압도적 비중은 에이전틱 코딩의 경제 구조를 새로 정의한다

저자가 공개한 토큰 소비 그래프에서 가장 두드러지는 특징은 캐시된 입력
토큰이 압도적으로 크다는 점이다.
마지막 주에는 캐시된 입력 토큰이 약 22억 개로, 압축되지 않은 입력
토큰(약 1억 개)이나 출력 토큰(약 640만 개)을 압도적으로 초과한다.

이 비대칭은 에이전틱 코딩 도구의 경제적 구조를 짐작하게 한다.
긴 컨텍스트 창을 계속 채운 채로 반복적으로 작업하는 워크플로에서는,
실제로 “새로 생성되는” 토큰보다 “매 요청마다 다시 참조되는” 컨텍스트가
비용의 대부분을 차지한다.
이는 프롬프트 캐싱 기술이 에이전틱 코딩의 실질적인 비용 구조를
좌우하는 핵심 변수가 되었다는 뜻이며, 앞으로 에이전틱 코딩 도구의
가격 경쟁력은 모델의 추론 능력만큼이나 캐시 히트율과 캐시 유지 정책에
좌우될 것이다.
동시에 이는 장시간 실행되는 에이전트 워크플로일수록, 컨텍스트를 얼마나
효율적으로 압축하고 필요한 부분만 다시 채워 넣는지가 비용 절감의
핵심 레버가 된다는 것을 시사한다.

### 학습 콘텐츠 제작을 위한 “가짜 인프라” 제작은 앞으로 더 흔해질 장르가 될 것이다

webernetes는 실제 프로덕션 쿠버네티스를 대체하려는 시도가 아니라,
교육 콘텐츠를 만들기 위한 도구다.
이 프로젝트가 보여주는 패턴, 즉 복잡한 실제 시스템의 핵심 개념만
선택적으로 재현하고 나머지는 과감히 생략하는 접근은, LLM이 코드 생성
비용을 극적으로 낮추면서 새롭게 가능해진 장르로 볼 수 있다.

과거였다면 “쿠버네티스의 핵심 동작을 브라우저에서 시각화하는 도구”를
만들려면 몇 달에서 몇 년의 전담 엔지니어링 자원이 필요했을 것이고,
그 정도 투자를 정당화할 수 있는 조직은 거의 없었을 것이다.
이제는 한 명의 개발자 에듀케이터가 2개월 만에, 상대적으로 적은 인건비로
이런 결과물을 만들어 낼 수 있다.
이는 기술 교육 콘텐츠 제작의 경제성 자체를 바꾸는 변화이며,
앞으로 복잡한 분산 시스템, 데이터베이스 내부 구조, 네트워크 프로토콜
등을 브라우저에서 시각적으로 체험할 수 있게 하는 “미니어처 재현”
프로젝트들이 유사한 방식으로 늘어날 가능성이 높다.
다만 이런 프로젝트들이 늘어날수록, webernetes가 겪은 것과 같은
“이것이 원본과 진짜 동일하게 동작하는가”라는 신뢰 검증 문제도
함께 반복될 것이다.
