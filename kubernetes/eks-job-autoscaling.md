# EKS에서 Job 워크로드를 위한 노드 오토스케일링 전략

원문: <https://medium.com/daangn/a6a28376d153>

## 요약

당근의 EKS 클러스터에서 Job 워크로드(시작과 끝이 정해진 작업)의 오토스케일링
문제를 해결한 사례다. Job 워크로드는 중단하면 처음부터 재시작해야 하므로
인터럽트가 어렵고, 여러 노드에 분산된 Job 파드가 노드 축소(scale-in)를
방해하여 수동으로 고정 노드 수를 관리해야 했다.

해결 전략은 두 가지다. 첫째, PodAffinity를 이용한 빈 패킹(bin-packing)으로
Job 파드를 소수의 노드에 집중시켜 빈 노드의 축소 가능성을 높였다. 커스텀
스케줄러(Volcano, Yunikorn) 대신 Kubernetes 네이티브 기능과 Kyverno 정책을
활용했다. 둘째, Cluster Autoscaler와 Karpenter에 축출 보호(eviction
protection) 어노테이션을 적용하여 실행 중인 Job의 강제 종료를 방지했다.

프로덕션 전환 시 동시 Job 폭주로 여러 문제가 발생했다. Kubelet 과부하(파드
밀도 초과), 이미지 풀 실패(레지스트리 속도 제한), EBS 볼륨 스로틀링(IOPS
고갈), CNI IP 할당 지연이 동시에 나타났다. 각각 maxPods 제한(234→60),
레지스트리 QPS 상향(5→40), EBS IOPS 확장(3,000→8,000), Host Network Mode
전환으로 대응했다.

결과적으로 평균 노드 수 16% 감소, 파드 Pending 시간 49% 단축, 파드 Running
시간 47% 단축을 달성했다.

## 분석

### 네이티브 기능 우선 전략

Volcano나 Yunikorn 같은 전문 배치 스케줄러를 도입하지 않고 PodAffinity와
Kyverno만으로 빈 패킹을 구현한 것은 운영 복잡성을 최소화하는 실용적 선택이다.
커스텀 스케줄러는 강력하지만 학습 곡선, 업그레이드 호환성, 디버깅 난이도를
수반한다. 기존 Kubernetes 프리미티브를 조합하는 접근은 유지보수 비용 측면에서
유리하다.

### 빈 패킹과 밀도 사이의 긴장

빈 패킹의 목적은 노드 활용률을 높이는 것이지만, 과도한 집중은 Kubelet
과부하를 유발한다. maxPods를 234에서 60으로 줄인 것은 빈 패킹의 이점을
유지하면서 단일 노드의 한계를 존중하는 균형점이다.
`kubelet_pleg_relist_duration_seconds_bucket`이라는 구체적 메트릭으로
Kubelet 부하를 관찰한 것은 엔지니어링 규율을 보여준다.

### 폭포식 장애 패턴

동시 Job 폭주가 Kubelet → 이미지 풀 → EBS → CNI까지 연쇄적으로 장애를
유발한 패턴은 주목할 만하다. 이것은 한 계층의 최적화(빈 패킹)가 다른
계층의 병목을 노출시키는 전형적인 사례다. 각 문제를 개별적으로 해결한 것은
체계적이지만, 이 연쇄 관계를 사전에 예측하기는 어려웠을 것이다.

### Host Network Mode 채택의 대담함

CNI IP 할당 지연을 Host Network Mode로 해결한 것은 보안과 격리를 일부
포기하는 결정이다. 파드가 노드의 IP를 공유하면 네트워크 정책(NetworkPolicy)
적용이 제한되고, 포트 충돌 가능성이 생긴다. Job 워크로드의 특성상
서비스 메시나 정교한 네트워크 정책이 불필요하다는 판단이 전제되어 있다.

## 비평

### 정량적 성과의 명확성

Before/After 메트릭이 표로 제시된 점은 설득력 있다. 노드 수 16% 감소,
Pending 시간 49% 단축, Running 시간 47% 단축은 비용 절감과 성능 개선을
동시에 달성했음을 보여준다. 다만 비용 절감의 절대 금액이나 연간 효과가
없어 비즈니스 임팩트를 가늠하기 어렵다.

### 프로덕션 장애 공유의 가치

알파에서 프로덕션으로의 전환 과정에서 발생한 네 가지 문제를 솔직하게
공유한 것은 이 글의 가장 큰 가치다. 성공 사례만 나열하는 기술 블로그와
달리, 실패와 그 해결 과정을 상세히 기술하여 유사한 환경의 팀에게 실질적
도움을 제공한다.

### Host Network Mode의 장기적 리스크

보안 격리를 포기하는 Host Network Mode가 장기적으로 기술 부채가 될
가능성에 대한 논의가 부족하다. 현재는 Job 워크로드에 한정되지만,
다른 워크로드로 확장될 때 포트 충돌이나 보안 정책 적용의 어려움이
발생할 수 있다.

## 인사이트

### 오토스케일링의 진짜 어려움은 스케일인에 있다

클라우드 오토스케일링은 주로 스케일아웃의 관점에서 논의된다. 수요가
증가하면 노드를 추가하는 것은 상대적으로 단순하다. 그러나 이 글이
보여주듯, 진짜 어려움은 스케일인에 있다. 실행 중인 워크로드를 중단할
수 없을 때, 빈 노드를 만들어야만 축소가 가능하고, 빈 노드를 만들려면
워크로드를 집중시켜야 한다.

이것은 쓰레기 수집(garbage collection)의 압축(compaction)과 구조적으로
같은 문제다. 메모리를 해제하려면 먼저 살아있는 객체를 한쪽으로 모아야
한다. Kubernetes 노드의 스케일인도 마찬가지로, 살아있는 파드를 소수의
노드로 압축해야 빈 노드를 회수할 수 있다. PodAffinity를 통한 빈 패킹은
본질적으로 이 압축 과정을 스케줄링 시점에 선제적으로 수행하는 것이다.

스케일인의 어려움은 Kubernetes에 국한되지 않는다. 데이터베이스 샤드
축소, 마이크로서비스 인스턴스 감소, 심지어 조직 구조 축소에서도 같은
패턴이 반복된다. 제거하려면 먼저 통합해야 하고, 통합에는 비용이 든다.
이 글은 이 보편적 패턴의 Kubernetes 구현체를 보여준다.

### 추상화 계층의 누적이 만드는 튜닝 포인트의 폭발

이 사례에서 튜닝해야 했던 지점을 나열하면 다음과 같다. Kubernetes
스케줄러(PodAffinity), Kubelet(maxPods), 컨테이너 런타임(이미지 풀 QPS),
스토리지(EBS IOPS), 네트워크(CNI vs Host Network), 오토스케일러(Cluster
Autoscaler/Karpenter 어노테이션). 단일 문제를 해결하기 위해 6개 이상의
추상화 계층을 조정해야 했다.

이것은 현대 클라우드 네이티브 스택의 구조적 특성이다. 각 계층은 독립적으로
합리적이지만, 계층 간 상호작용이 예측 불가능한 행동을 만든다. maxPods를
줄이면 파드 밀도 문제는 해결되지만 더 많은 노드가 필요해지고, Host
Network Mode는 CNI 문제를 해결하지만 보안 격리를 약화시킨다.

이 복잡성은 줄어들지 않을 것이다. 서비스 메시, eBPF, GPU 스케줄링 등
새로운 계층이 계속 추가되고 있다. 각 계층의 전문가는 있지만 계층 간
상호작용을 이해하는 전문가는 드물다. 이 글의 가치는 특정 해법보다 계층
간 상호작용을 디버깅하는 과정 자체에 있다.

### Job 워크로드는 Kubernetes의 설계 의도 밖에 있다

Kubernetes는 본질적으로 서비스(항상 실행되는 무상태 워크로드)를 위해
설계되었다. ReplicaSet, Deployment, HPA 등 핵심 추상화는 모두 장기 실행
서비스를 전제한다. Job과 CronJob은 존재하지만, 스케줄링, 오토스케일링,
네트워킹 등 핵심 메커니즘에서 일급 시민(first-class citizen)으로
취급되지 않는다.

이 글에서 PodAffinity로 빈 패킹을 구현하고, 축출 보호 어노테이션을
수동 주입하고, maxPods를 조정하는 등의 작업은 모두 Kubernetes가 Job
워크로드를 네이티브로 지원하지 않기 때문에 필요한 우회(workaround)다.
Volcano나 Yunikorn 같은 프로젝트가 존재하는 것 자체가 이 갭의 증거다.

이것은 기술 선택의 중요한 교훈이다. 플랫폼의 설계 의도와 실제 사용
사이의 간극이 클수록 운영 비용이 증가한다. 당근 팀은 Kubernetes 내에서
문제를 해결했지만, 장기적으로는 Job 워크로드를 위한 전용 인프라(예:
Temporal, AWS Batch)와의 비용 비교가 필요할 수 있다. 범용 플랫폼에서
특수 워크로드를 운영하는 비용이 전용 솔루션의 도입 비용을 초과하는
지점이 반드시 존재한다.
