# deep-teach

<https://github.com/1786329860/deep-teach>

## 소개

deep-teach는 AI 코딩 어시스턴트에 “깊이 있는 교육 모드(深度教学模式)”를 부여하는
Claude Code Skill이다. AI가 코드를 작성하거나 파일을 수정하는 등 프로그래밍 작업을
수행할 때마다, 단순히 결과물만 내놓는 게 아니라 그 결정의 근거와 원리를 구조화된
“Teaching Card”로 함께 출력한다.

핵심 전제는 다음과 같다.

```text
전통적 AI 보조:  AI 실행 → 사용자 결과 수령 → 지식 미성장
deep-teach:     AI 실행 → 결정 근거 해설 → 대안 비교 → 원리 심층 분석
```

`SKILL.md` 파일 하나를 Cursor, Claude Code, Continue 등의 커스텀 지시문에
붙여 넣으면 즉시 활성화된다.

## Teaching Card 구조

매 코드 작업 완료 후 AI가 자동 출력하는 카드는 6개 섹션으로 구성된다.

| 섹션               | 내용                                                     |
| ------------------ | -------------------------------------------------------- |
| ① 사용 기술        | 기술명, 버전, 분류, 현재 단계에서의 역할                 |
| ② 선택 이유        | 프로젝트 제약 → 기술 적합성 → Trade-off 분석             |
| ③ 기술 심층 분석   | 핵심 원리, 구현 세부사항, 코드 예시, 흔한 함정           |
| ④ 대안 비교        | 최소 2개 대안 항목의 다차원 비교표                       |
| ⑤ 우월성 요약      | 핵심 장점(정량 데이터 포함), 생태계 성숙도 평가          |
| ⑥ 지식 연장 및 전이 | 교차 도메인 사상 매핑, 학습 경로, 추천 자료             |

카드는 세 가지 변형으로 자동 선택된다.

- **표준 카드**: 일반적인 코딩 작업에서 6개 섹션 전체 출력
- **Mini 카드**: 의사결정 가치가 낮은 단순 작업(`console.log` 등)에서
  ①②⑤ 세 섹션만 3~5줄로 요약
- **Enhanced 카드**: 프레임워크 선택, DB 선택 등 중요 아키텍처 결정에서
  표준 카드에 결정 트리와 리스크 매트릭스 추가

## 프롬프트 프로토콜 구조 (M1–M5)

`SKILL.md`는 5개 모듈로 구성된 프롬프트 프로토콜이다.

- **M1 역할 정의**: AI에게 “엔지니어 + 교수”의 이중 정체성을 부여하고
  핵심 원칙(대신 생각하지 않고 사고 과정 보여주기, 모든 선택에 근거 제시,
  솔직한 Trade-off 공개)을 명시
- **M2 출력 프로토콜**: Teaching Card 포맷 템플릿(표준 / Mini / Enhanced)
- **M3 트리거 규칙**: 파일 생성·삭제, 의존성 설치·제거, 함수/클래스/컴포넌트
  작성, 설정 파일 변경, DB 작업, 서드파티 서비스 통합, 아키텍처·기술 선택,
  Debug/성능 최적화 등 8가지 자동 트리거와 3가지 병합 규칙
- **M4 품질 기준**: 섹션별 품질 체크리스트, 언어 스타일 규범, 정보 밀도 제어
- **M5 Few-Shot 예시**: 3가지 완전한 고품질 예시(의존성 설치, 코드 작성, 아키텍처 결정)

## 분석

deep-teach의 핵심 메커니즘은 AI 출력에 “왜”의 레이어를 의무적으로 삽입하는
것이다. 기존 AI 코딩 보조 도구들은 작업 완료 속도를 최적화하는 방향으로 설계되어
있다. deep-teach는 반대로, 그 완료 과정에 마찰을 추가하되 그 마찰이 학습 가치를
지니도록 구조화한다.

트리거 자동화와 병합 규칙의 조합은 정보 과부하 문제를 현실적으로 다룬다.
연속적인 동일 유형 작업(예: 여러 의존성 연속 설치)을 하나의 “기술 스택 선택”
카드로 통합하고, 동일 패턴 반복 시 “Step N과 동일 패턴” 방식으로 간소화한다.
이는 교육적 맥락에서 반복 노출과 인지 부하 조절 사이의 균형을 프롬프트 레벨에서
구현한 것이다.

비교 분석표에서 스스로를 대학 강의나 React Query 공식 문서와 비교하는 방식은
포지셔닝 측면에서 흥미롭다. “즉시성 + 맥락 관련성 + 결정 논리 + 대안 시야”를
동시에 제공한다는 주장은 타당하지만, 각 섹션의 실제 깊이는 모델의 응답 품질에
전적으로 의존한다는 점에서 한계가 있다.

## 비평

deep-teach의 접근법은 AI 보조 학습이 가진 구조적 문제 하나를 선명하게 드러낸다.
단지 Task completion(작업 완료)만 측정하는 AI 도구는 사용자의 학습 성장을
측정하거나 촉진하지 않는다. 이 프로젝트는 그 공백을 프롬프트 엔지니어링으로
메우려 한다.

그러나 교육 효과의 검증 방법이 없다는 것이 약점이다. “사용자가 유사 상황에서
동일한 결정을 독립적으로 내릴 수 있는가”를 자가 진단 기준으로 제시하지만,
이는 체계적인 측정이 아니라 AI의 자기 평가에 불과하다. 실제 학습 효과를
측정하려면 세션 이후 사용자 행동 변화를 추적해야 하는데, 프롬프트 Skill 구조에서는
그런 피드백 루프가 존재하지 않는다.

또한 Teaching Card의 정보량이 일관되게 높을 경우, 작업 맥락을 유지하는 컨텍스트
윈도우를 빠르게 소모한다는 실용적 문제도 있다. Enhanced 카드에 결정 트리와
리스크 매트릭스까지 포함한다면, 복잡한 프로젝트에서 카드 자체가 실제 코드보다
더 많은 토큰을 차지하는 상황이 발생할 수 있다.

## 인사이트

### AI 도구 설계에서 “대신 해주기”와 “가르치며 해주기”의 분기

AI 코딩 도구 생태계는 지금 조용한 분기점에 서 있다. 한쪽은 Devin, GitHub Copilot
Workspace처럼 자율적으로 태스크를 완수하는 방향으로 진화하고, 다른 쪽은 사용자의
이해와 역량 증진을 명시적인 목표로 설정하는 방향으로 분기된다.

deep-teach는 후자의 극단에 위치한다. AI가 한 번의 세션에서 코드를 만들어 주더라도,
사용자가 그 코드의 의미를 이해하지 못한다면 다음 세션에서도 동일한 도움이 필요하다.
이를 “AI 의존성 고착화”라고 부를 수 있다. deep-teach의 설계 철학은 이 고착화를
의도적으로 방지하려 한다.

흥미롭게도, 이 분기는 교육학에서 “교사가 대신 해주기 vs. 교사가 비계(scaffolding)
제공하기” 논쟁과 정확히 같은 구조를 갖는다. AI 도구 설계자들이 이 교육학적 긴장을
명시적으로 설계 목표에 반영하기 시작했다는 것 자체가 AI 보조 도구의 성숙도를
보여주는 신호다.

실용적 관점에서는, 두 방향이 서로 다른 사용자 세그먼트를 대상으로 한다.
최대한 빠르게 결과물이 필요한 전문가와, 학습 자체가 목적인 입문자에게 동일한
도구가 최적일 수 없다. deep-teach는 스스로의 타깃 사용자(비전공 학생, 전공 전환자,
초급 개발자)를 명확히 정의했다는 점에서 포지셔닝이 일관적이다.

### 프롬프트 엔지니어링이 교육 설계 방법론을 흡수하는 흐름

M1~M5의 모듈 구성 방식은 단순한 프롬프트 작성이 아니라, 체계적인 교수 설계
(Instructional Design)의 언어를 차용하고 있다. 역할 정의(M1), 출력 형식 명세(M2),
트리거 조건(M3), 품질 기준(M4), Few-Shot 예시(M5)라는 구조는
각각 교육학에서 학습 목표, 수행 기준, 평가 조건, 루브릭, 모델링에 대응한다.

이 프레임은 AI Skill 설계가 점점 더 교육 공학의 방법론을 참조하게 될 것임을 암시한다.
Teaching Card의 6개 섹션 자체도 블룸의 분류법(Bloom's Taxonomy)을 따라가는
구조로 읽힌다. “무엇을 사용했는가(지식)”에서 시작해 “왜 선택했는가(이해)”,
“어떻게 동작하는가(적용)”, “대안과 어떻게 다른가(분석)”, “다른 맥락에서 어떻게
전이되는가(종합)”로 이어지는 계단식 구성이다.

이런 교육 공학적 프레임이 Skill 설계에 적용될수록, AI 프롬프트 엔지니어링은
단순히 “AI가 잘 이해하도록 지시를 쓰는 기술”을 넘어서 “학습 경험을 설계하는
기술”로 영역이 확장된다. Skill 개발자와 교수 설계자의 역할이 수렴하는 현상이
가속화될 것이다.

### 컨텍스트 엔지니어링(Context Engineering)으로서의 Skill

deep-teach는 AI의 출력 내용을 바꾸는 것이 아니라, AI가 컨텍스트를 소비하고
생성하는 방식 자체를 재구성한다. 이것은 최근 논의되는 “컨텍스트 엔지니어링”
개념과 정확히 맞닿는다. 프롬프트를 단순한 지시문이 아니라, AI가 올바른 출력을
생성하기 위해 필요한 구조적 맥락 전체를 설계하는 행위로 보는 관점이다.

M5의 Few-Shot 예시가 단순히 형식을 보여주는 게 아니라 “이 정도 깊이와 이 정도
정량 데이터가 포함되어야 한다”는 품질 기준을 암묵적으로 전달한다는 점은
컨텍스트 엔지니어링의 전형적 패턴이다. 언어 모델은 명시적 규칙보다 시범 사례에서
더 강한 영향을 받기 때문이다.

또한 병합 규칙(동일 유형 연속 작업 통합, 동일 패턴 반복 간소화)은 컨텍스트 윈도우
효율을 고려한 설계다. Teaching Card가 지속적으로 축적될수록 컨텍스트를 잠식한다는
문제를 인식하고, 그 해결책을 Skill 로직 안에 내장한 것이다. 이는 단순한 출력
형식 설계를 넘어 세션 전체의 정보 흐름을 설계하는 수준의 엔지니어링이다.
