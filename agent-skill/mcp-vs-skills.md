# MCP가 Skills보다 여전히 낫다

원문: <https://david.coffee/i-still-prefer-mcp-over-skills/>

## 요약

저자 David는 AI 에이전트 생태계에서 "MCP는 죽었다, Skills가 새 표준이다"라는 주장이
확산되는 것에 반론을 제기한다. 본인은 Claude Code, Codex, Gemini, ChatGPT,
Perplexity를 매일 업무에 사용하는 헤비 유저로서, Notion, DEVONthink, 이메일 관리에
LLM을 적극 활용한다.

MCP(Model Context Protocol)의 강점은 API 추상화 철학에 있다. LLM은 구현 방법을 알
필요 없이 인터페이스만 호출하면 된다(`devonthink.do_x()`). 이를 통해 원격 사용 시
별도 설치 불필요, 서버 업데이트 시 모든 클라이언트 즉시 반영, OAuth 기반 안전한 인증,
기기 무관 이식성, 자연스러운 샌드박싱, 스마트 도구 탐색 등의 이점을 얻는다.

Skills의 문제는 CLI 의존성에서 비롯된다. 순수 지식형 스킬(커밋 메시지 형식, 테스트
작성 방식 등)은 잘 작동하지만, CLI 실행을 요구하는 스킬은 ChatGPT, Perplexity,
표준 웹 Claude처럼 터미널이 없는 환경에서는 아예 동작하지 않는다. 배포 복잡성, 시크릿
관리 문제, 클라이언트별 파편화된 지원, SKILL.md 전체를 컨텍스트 창에 로드해야
하는 비효율도 단점이다.

저자가 제안하는 적절한 역할 분담은 명확하다. 서비스/애플리케이션 연결에는 MCP를,
지식 전달과 컨텍스트 제공에는 Skills를 사용해야 한다. Google Calendar처럼 OAuth
인증이 필요한 서비스는 공식 원격 MCP로, 이미 설치된 `curl`, `git`, `gcloud` 같은
도구 활용법 교육에는 Skills가 적합하다.

저자는 MCP와 Skills의 조합이 최상의 시너지를 낸다고 결론짓는다. 특히 MCP 세션에서
발견한 날짜 포맷 quirk, 검색 함수의 결과 절단 동작, 예상과 다른 도구 이름 등의
gotcha를 Skills에 축적하면, Skills가 MCP의 '치트시트' 역할을 하게 된다.

## 분석

### 두 패러다임의 아키텍처적 차이

MCP와 Skills는 근본적으로 서로 다른 관심사를 해결한다. MCP는 LLM과 외부 시스템
사이의 **연결 계층(connector layer)**이고, Skills는 LLM의 행동 방식을 형성하는
**지식 계층(knowledge layer)**이다. 저자는 둘이 경쟁하는 것이 아니라 상보적
관계여야 한다고 주장한다.

현재 생태계의 문제는 Skills가 연결 계층의 역할까지 맡으려 한다는 데 있다. CLI
wrapper를 통해 서비스에 접근하는 방식은 CLI 설치, 인증 토큰 관리, 환경 의존성이라는
세 가지 문제를 함께 끌어들인다. 반면 MCP는 이를 프로토콜 수준에서 추상화한다.

### "MCP는 죽었다" 담론의 배경

이 담론은 Claude Code, Codex 같은 풀 컴퓨트 환경의 부상과 관련이 깊다. 이런
환경에서는 CLI 실행이 가능하기 때문에 Skills가 충분히 동작한다. 그러나 이는 특수한
환경이고, 대부분의 일반 사용자는 웹 브라우저의 채팅 인터페이스에서 LLM을 사용한다.
저자는 이 맥락의 차이를 놓치고 있다는 점을 지적한다.

### 생태계 파편화 문제

Skills의 가장 큰 현실적 문제는 표준화 부재다. `npx skills`로 설치해도 Codex와
Claude Code에서만 동작하고, Claude Cowork나 표준 Claude에서는 안 된다. OpenClaw
스킬을 Claude에 넣으면 YAML 파싱 오류가 난다. 이런 파편화는 "Skills가 새 표준"이라는
주장과 정면으로 충돌한다.

## 비평

### 강점

저자의 논거는 실제 사용 경험에 기반해 설득력이 강하다. 추상적 주장이 아니라
mcp-server-devonthink, microfn, Kikuyo 같은 직접 구축한 프로젝트를 예시로
들기 때문에 신뢰도가 높다. "MCP는 죽었다"는 트렌드에 맞서 명확한 반론을 제시하는
용기도 돋보인다.

Skills를 완전히 부정하지 않고 순수 지식형 스킬의 가치를 인정하는 균형 잡힌 시각도
장점이다. Skills + MCP 조합을 통해 MCP 세션의 학습을 누적하는 "치트시트" 패턴은
실무에서 즉시 적용 가능한 아이디어다.

### 약점

저자의 사용 환경이 다소 특수하다. DEVONthink, Notion, 원격 MCP 서버를 직접
구축하는 파워 유저의 시각이다 보니, CLI 한 번 설치하는 것을 큰 부담으로 느끼지 않는
개발자 독자에게는 설득력이 다소 떨어질 수 있다.

또한 원격 MCP의 장점을 강조하면서 저자 본인이 만든 MCP Nest를 홍보하는 구조가 이해
충돌(conflict of interest)처럼 보일 수 있다. 독자 입장에서 주장의 중립성에 의문을
품을 여지가 있다.

"Notion은 공식 원격 MCP를 제공하는 것이 옳다"는 주장은 타당하지만, 소규모 서비스나
개인 프로젝트가 MCP 서버를 직접 구축하고 유지하는 비용에 대한 논의가 부족하다.

## 인사이트

### Skills는 LLM 매뉴얼이고 MCP는 커넥터다 — 명칭이 혼란의 근원

저자가 제시한 "LLM_MANUAL.md vs Connectors"라는 리네이밍 아이디어는 표면적으로 가볍게
제안한 것처럼 보이지만, 사실 이 담론 전체의 핵심을 꿰뚫는다. "Skills"라는 단어는
능동적·도구적 뉘앙스를 풍기기 때문에, 마치 LLM이 스킬을 통해 무언가를 할 수 있게
된다는 인상을 준다. 그 결과 서비스 연결이라는 역할까지 Skills에 기대하게 되는 것이다.

언어는 사고를 형성한다. "Skills"라는 이름은 "이걸 배우면 뭔가 할 수 있다"는 기대를
심어주지만, "MANUAL.md"는 "이건 이미 있는 도구의 사용법"이라는 올바른 인식을
유도한다. "MCP"라는 이름은 프로토콜 레이어를 강조해 연결 표준이라는 의미를 잘 전달하지만, 
"Connectors"는 비개발자에게도 직관적이다.

생태계의 혼란은 종종 나쁜 작동 방식이 아니라 나쁜 명명에서 비롯된다. 기술적 아키텍처가
옳더라도 이름이 잘못된 기대를 만들면 오용이 발생한다. 이것은 API 설계에서 SDK 문서에
이르기까지 소프트웨어 전반에 걸친 보편적 교훈이다.

### "MCP는 죽었다"는 담론은 에코 챔버 효과의 산물

저자가 X(트위터)를 언급하며 "아마도 X에서 너무 많은 시간을 보낸 탓"이라고 한 자기
인식은 중요한 메타 관찰이다. 기술 의사결정이 실제 사용성보다 소셜미디어 트렌드에
이끌리는 현상은 최근 AI 도구 생태계에서 특히 두드러진다.

LLM 도구의 "킬러 기능"을 주장하는 글이 바이럴되면, 그 기술의 실제 적합 범위와 무관하게
모든 곳에 적용하려는 시도가 이어진다. MCP는 죽지 않았다 — 특정 클라이언트와 워크플로에서
Skills가 더 편리한 사례가 생겼을 뿐이다. 그런데 그것이 "MCP는 구식"이라는 내러티브로
증폭된다.

이는 기술 의사결정자들이 경계해야 할 패턴이다. 트렌드 언어("legacy", "dead",
"new standard")는 기술의 실제 트레이드오프를 흐린다. 새로운 패러다임이 등장했을 때
"기존 것이 죽었는가"가 아니라 "각각 어떤 문제를 잘 푸는가"를 질문하는 것이 올바른
프레임이다.

### MCP + Skills 조합: 지식 계층의 누적이 LLM의 진짜 경쟁력이 된다

저자가 제안하는 "MCP 세션의 gotcha를 Skills에 축적" 패턴은 단순한 편의성을 넘어선다.
이것은 LLM을 사용하는 팀/개인이 시간이 지날수록 더 스마트한 운영자가 되는 방법이다.

일반적으로 LLM은 매 세션이 stateless하다. 같은 실수를 반복하고, 같은 API quirk를
재발견하고, 같은 컨텍스트를 매번 다시 로드해야 한다. Skills에 누적된 학습은 이 stateless
한계를 극복하는 실용적 방법이다. MCP가 연결을 표준화한다면, Skills는 그 연결 위에서의
인터랙션 패턴을 표준화한다.

장기적으로 이 패턴을 체계화하는 팀은 LLM 활용에서 구조적 우위를 갖게 된다. "AI를
잘 쓰는 팀"과 "AI를 쓰는 팀"의 차이는 결국 이런 지식 계층의 축적 여부에서 갈릴
가능성이 높다. 기술 도입 자체보다 기술 사용 노하우의 제도화가 더 중요한 시대가 오고 있다.
