# Context Is the New Code

원문: <https://www.youtube.com/watch?v=bSG9wUYaHWU>

## 요약

Tessl의 Patrick Debois가 AI Engineer 컨퍼런스에서 발표한 내용으로, 바이브 코딩(vibe
coding) 시대에 코드보다 컨텍스트(context)가 핵심 자산이 됐다는 주장을 펼친다.
프롬프트가 코드를 생성하는 지금, 에이전트에게 무엇을 줄지를 결정하는 컨텍스트 자체가
새로운 코드라는 테제다. 그는 2009년 DevOps를 제창할 때 “ops가 dev처럼 생각하면
어떨까?”라고 물었듯, 이번엔 “컨텍스트를 코드처럼 다루면 어떨까?”를 묻는다.

이 물음에 답하기 위해 Debois는 소프트웨어 개발 생명주기(SDLC)에 대응하는 컨텍스트
개발 생명주기(CDLC, Context Development Life Cycle) 개념을 제안한다. 생성(Generate) →
테스트(Test) → 배포(Distribute) → 관찰(Observe) → 적응(Adapt)으로 이어지는 무한
루프 구조다. 이 루프는 DevOps의 infinity loop를 의식적으로 차용한 것이다.

각 단계에는 코드 세계의 대응물이 있다. 재사용 가능한 프롬프트는 라이브러리, 스킬
레지스트리는 패키지 레지스트리, 컨텍스트 의존성 충돌은 의존성 지옥(dependency
hell), 컨텍스트 스캔은 Snyk 같은 보안 스캐너에 대응된다. LLM 판정자(LLM-as-a-judge)
를 샌드박스 도구와 결합하면 컨텍스트 변경이 실제 동작에 미치는 영향을 엔드투엔드로
검증할 수 있다.

발표의 실용적 결론은 두 가지 루프의 병행이다. 개인·팀이 컨텍스트를 저작하고
테스트하는 “라이브러리 저작 루프”와, 조직 전체가 공유 컨텍스트를 개선해 나가는
“조직 루프”다. LLM과 코딩 에이전트는 단순히 엔진이며, 올바른 연료인 컨텍스트를
공급하지 않으면 엔진 성능은 발휘되지 않는다.

## 분석

### DevOps 유추의 구조

Debois가 DevOps를 만들었다는 배경은 이 발표를 단순한 기술 트렌드 관찰이 아니라
패턴 인식으로 읽게 만든다. 2009년의 핵심 통찰이 “사일로를 허물고 피드백 루프를
만들라”였다면, 2024년의 통찰은 “프롬프트와 컨텍스트를 코드와 같은 엄밀함으로
다루라”는 것이다. CDLC는 SDLC의 메타포가 아니라 실제 공학적 절차로 제안된다.

### 컨텍스트 테스트의 계층 구조

발표에서 제시된 테스트 계층은 코드 테스트 계층과 직접 대응한다.

- 린터(linter): 컨텍스트 형식 검증 — 스킬에 설명이 있는가, 길이 제한을 지키는가
- Grammarly 류: LLM이 컨텍스트를 이해할 수 있는가를 LLM에게 묻는 의미론적 검증
- 유닛 테스트: LLM-as-a-judge로 생성된 코드가 컨텍스트의 규칙을 따르는지 판정
- 엔드투엔드 테스트: 판정자 에이전트에게 도구를 주어 실제 실행 결과를 검증

비결정론적(non-deterministic) 특성 때문에 단일 실행 결과로는 판단할 수 없으며,
에러 버짓(error budget) 개념을 도입해 허용 실패율을 설정하는 방식을 제안한다.
이는 SRE의 에러 버짓 개념을 컨텍스트 평가에 이식한 것이다.

### 배포와 관찰 인프라

스킬이 패키지 형식을 채택하면서 레지스트리, 의존성 관리, 보안 스캐닝, SBOM(Software
Bill of Materials)에 대응하는 AI SBOM 개념이 파생된다. `agent.md`, `skill.md` 등
표준화 움직임도 언급한다. 관찰 단계에서는 에이전트 로그 표준화와 PR 피드백, 프로덕션
장애를 컨텍스트 개선의 신호로 삼는 피드백 루프가 강조된다.

## 비평

### 강점: 타이밍과 유추의 힘

DevOps 창시자가 AI 컨텍스트 관리에 같은 프레임을 적용한다는 점은 설득력이 높다.
추상적 선언이 아니라 구체적 도구 예시(컨텍스트 필터, 에러 버짓, 샌드박스, SBOM)
를 제시하며 실천 가능성을 보여준다. 생성·테스트·배포·관찰이라는 루프는 이미
엔지니어들에게 익숙한 정신 모델이어서 채택 비용이 낮다.

### 약점: 성숙도 격차

발표 자체가 “미완성 사고(unpolished thought)”라고 스스로 인정한다. 스킬 레지스트리의
99.9%가 품질 기준 미달이라고 말하면서도 레지스트리 생태계를 긍정적으로 제시하는
것은 자기모순에 가깝다. 에러 버짓과 LLM-as-a-judge 기반 CI/CD는 개념으로는 매력적이지만,
실제 조직에서 얼마나 안정적으로 운용될 수 있는지에 대한 검증 사례가 없다.

### 누락된 관점

컨텍스트 거버넌스의 정치적 측면이 빠져 있다. 누가 조직의 공유 컨텍스트를 소유하고,
누가 변경 권한을 갖는가? 코드베이스의 소유권 분쟁이 컨텍스트 레이어로 이동할
가능성이 크다. 또한 LLM 모델이 교체될 때 기존 컨텍스트가 얼마나 호환되는지,
모델 특이성(model-specificity) 문제도 다뤄지지 않는다.

## 인사이트

### 컨텍스트는 새로운 설정 파일이다

과거 인프라 세계에서 “인프라 as 코드(Infrastructure as Code)”는 서버 상태를
선언적으로 기술하면서 버전 관리, 리뷰, 테스트를 가능하게 했다. 컨텍스트 as 코드는
같은 전환을 AI 에이전트의 행동 공간에 적용한다. 둘 다 “암묵적이고 변경 추적이
불가능한 상태”를 “명시적이고 버전 관리되는 파일”로 만드는 운동이다. DevOps가
운영팀에 개발 문화를 이식했듯, CDLC는 에이전트 운용에 소프트웨어 공학 문화를
이식하려 한다. 이 유추가 강한 이유는 단순한 수사가 아니라 실제 도구 범주(린터,
CI/CD, 레지스트리, SBOM)가 1:1로 대응되기 때문이다.

### 비결정론이 테스트 철학을 바꾼다

소프트웨어 테스트의 기본 가정은 같은 입력에 같은 출력이 나온다는 것이다. LLM 기반
시스템은 이 가정을 무너뜨린다. 에러 버짓 도입은 단순한 편법이 아니라 테스트 철학
자체의 전환을 의미한다. “통과/실패”의 이진 판정에서 “n회 중 m회 성공”이라는
확률적 판정으로 이동한다. 이는 SRE가 가용성을 “100% 아니면 실패”가 아니라
“99.9% 허용”으로 재정의했던 전환과 구조적으로 동일하다. 컨텍스트 품질도 결국
확률로 측정되는 세계가 될 것이며, 이는 QA 조직의 역할과 도구 전반을 재정의한다.

### 조직 플라이휠이 핵심 경쟁 우위가 된다

개인이 자신의 `CLAUDE.md`를 최적화하는 것과 조직이 공유 컨텍스트를 개선하는 것은
선형이 아닌 지수적 차이를 만든다. 한 팀이 발견한 컨텍스트 결함을 전 조직이 즉시
혜택받는 플라이휠이 작동하면, 조직의 컨텍스트 자산은 시간이 지날수록 경쟁사가
복제하기 어려운 누적 자산이 된다. 이는 오픈소스 코드베이스와 달리 조직 내부 도메인
지식, 관행, 팀 규약이 녹아 있어 공개된다 해도 그대로 이식하기 어렵기 때문이다.
앞으로 AI 생산성의 격차는 어떤 LLM을 쓰느냐가 아니라 얼마나 좋은 컨텍스트 자산을
축적했느냐로 결정될 가능성이 높다.

### 컨텍스트 필터는 새로운 보안 경계다

WAF(Web Application Firewall)가 HTTP 레이어의 악성 패턴을 차단하듯, 컨텍스트
필터는 에이전트에 주입되는 프롬프트 인젝션과 악성 스킬을 차단한다. 그런데 이
경계는 기존 보안 모델과 근본적으로 다르다. 코드의 취약점은 정적 분석으로 탐지
가능하지만, 컨텍스트의 악의적 의도는 의미론적 이해 없이는 판단할 수 없다. Snyk의
컨텍스트 스캐닝이 자격증명 노출이나 서드파티 연결을 탐지하는 것은 시작에 불과하다.
에이전트가 조직 내부 시스템에 더 깊이 통합될수록, 컨텍스트 보안은 코드 보안만큼
혹은 그 이상으로 중요한 공학 분야가 될 것이다.
