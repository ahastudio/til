# Don't Git Afraid

<https://github.com/rubric-dev/dont-git-afraid>

## 소개

Don't Git Afraid는 “바이브 코딩(vibe coding)” 학습자를 위한 Claude Code 에이전트 스킬이다.
Git의 기술적 용어를 한국어 일상 언어와 이모지로 자동 번역해,
Claude Code가 Git 작업을 설명할 때 사용자가 겁먹지 않도록 돕는다.
rubric-dev가 2026년 5월 1일에 공개했다.

AI가 코드 생성은 잘 해주지만, `git rebase origin/main --onto feature/login` 같은
명령이 실행된다고 하면 사용자가 패닉 상태가 된다는 관찰에서 출발했다.
스킬을 설치하면 Claude Code가 Git 용어를 다음과 같이 변환한다.

| Git 용어   | 스킬이 쓰는 말       |
| ---------- | -------------------- |
| commit     | 💾 세이브            |
| push       | ☁️ GitHub에 올리기   |
| pull       | 📥 받아오기          |
| branch     | 🌿 사본              |
| merge      | 🤝 합치기            |
| stage/add  | 🧺 담아두기          |
| stash      | 🗄️ 잠깐 치워두기    |
| rebase     | ⏳ 순서정리          |
| conflict   | 💥 충돌              |

## 설치

```bash
# 프로젝트 전용
cp -r dont-git-afraid 내프로젝트/.claude/skills/

# 전역 적용
cp -r dont-git-afraid ~/.claude/skills/
```

`CLAUDE.md`에 한 단락을 추가하면 끝이다.

```markdown
## Git 규칙
사용자에게 Git 작업을 설명·보고·확인할 때는 반드시
`dont-git-afraid` 스킬의 용어 변환 규칙을 따르세요.
실제 git 명령 실행 시에는 원래 용어를 사용합니다.
```

## 분析

### 위험도 시각화: 되돌리기 어려운 작업의 명시화

스킬의 핵심 장치는 위험도 막대 표시다.

```text
🟢 안전  [█░░░]  — 세이브, GitHub에 올리기
🟡 조심  [██░░]  — 받아오기 (고치던 파일 있을 때)
🟠 주의  [███░]  — 합치기, 순서정리
🔴 위험  [████]  — 강제 덮어쓰기
```

`--force-with-lease` 같은 위험 명령 앞에서 AI가 자동으로 경고를 출력한다.
이는 cliclick의 `-m test` 드라이런 모드와 동일한 철학이다 —
비가역적 동작을 실행하기 전에 사용자에게 선택권을 준다.

### 보안 사고 예방 레이어

`.env` 파일이나 `node_modules`가 스테이징에 올라오려 하면 자동으로 차단한다.
비이너 코더가 실수로 환경 변수를 GitHub에 올리는 것은 실제로 빈번한 사고다.
이 스킬은 Git 지식 없이도 이 종류의 실수를 방어한다.

17개 일반 Git 오류를 평문으로 번역하고 복구 안내를 제공하는 것도 있다.
“코드가 다 날아갔어요!” 패닉 상황에서 차분하게 `git reflog` 경로를 안내하는
패닉 대응 섹션이 포함된다.

### SKILL.md 컨벤션의 실사용 사례

이 스킬은 Claude Code의 `SKILL.md` 컨벤션을 따른다.
폴더 하나를 `.claude/skills/`에 복사하는 것으로 설치가 완료되므로
빌드 도구, 패키지 관리자, 별도 의존성이 필요 없다.
이 단순함이 바이브 코딩 입문자에게 적합한 배포 방식이다 —
스킬 설치 자체도 Git 없이 ZIP 다운로드 방식을 우선 안내한다.

## 비평

### 강점: 도구보다 사용자의 멘탈 모델에 집중

대부분의 Git 학습 자료는 “commit이 무엇인가”를 설명하는 데 집중한다.
이 스킬은 그 접근을 뒤집는다 — 개념을 가르치는 대신, 사용자가 이미 아는 비유
(게임 세이브)로 매핑한다. 학습 없이 안전하게 작업하게 하는 것이 목표다.
이 접근은 바이브 코딩 맥락에서 타당하다. 코드 작성 자체가 AI의 역할이 됐다면,
사용자가 Git의 내부 작동을 알 필요도 없다.

### 약점: 용어 변환이 학습 기회를 차단할 수 있다

Git 용어를 지속적으로 번역해주면 사용자가 원래 용어를 배울 기회가 줄어든다.
“세이브”로만 알고 있던 사람이 다른 도구(GitLab 웹 UI, JetBrains 등)를
사용할 때 “commit”이라는 단어에서 다시 막힌다.
학습 발판이 될 수 있는 입장지만, 의존성을 만들 수도 있다는 이중성이 있다.

### 범위: 영어권 사용자에게는 적합하지 않다

용어 변환이 한국어 기반이므로 영어권 바이브 코더에게는 적용되지 않는다.
영문 README가 따로 있지만, 용어 변환 테이블이 한국어 특화 설계임은 분명하다.
이 범위 한정은 오히려 명확한 타깃 설정이라고 볼 수 있다.

## 인사이트

### 바이브 코딩이 만들어낸 새로운 사용자 집단

“바이브 코딩”의 부상은 코드를 직접 쓰지 않아도 소프트웨어를 만드는 사람들을
만들었다. 이 사용자들은 에디터 키보드 단축키도, Git 명령어도, 터미널 사용법도
알 필요가 없었다 — AI가 다 해줬기 때문이다.
그런데 Git은 예외였다. AI가 코드를 생성하고 파일을 쓰더라도,
“지금 저장하고 GitHub에 올리겠습니다”라는 말은 사용자가 이해하고 승인해야 한다.

이 스킬은 그 간극을 메운다. 흥미로운 것은 이 문제가 기술 문제가 아니라
언어와 불안 문제라는 인식이다. 시스템이 “force push하겠습니까?”라고 묻는 대신
“GitHub에 있는 기록을 내 걸로 완전히 바꾸겠습니다. 🔴 위험”이라고 하면
사용자는 위험을 이해하고 멈출 수 있다. Git을 이해하지 않아도.

2차 효과로, 이 스킬은 에이전트 스킬의 역할이 “능력 추가”에서 “인터페이스 변환”으로
확장될 수 있음을 보여준다. 스킬이 모델에 새 능력을 주는 것이 아니라,
모델이 기존 능력을 표현하는 방식을 바꾼다.
이것은 UX 레이어로서의 스킬이라는 새로운 범주다.

### 스킬 생태계의 롱테일

Don't Git Afraid는 GitHub 스타 2개로 시작했다.
addyosmani의 에이전트 하네스 엔지니어링 스킬, guizang-ppt 스킬처럼
주목받는 스킬과 달리, 이 스킬은 매우 좁은 문제를 매우 구체적인 사용자를 위해
해결한다. 하지만 바이브 코딩 학습자라는 집단이 2026년에 빠르게 성장하고 있다면,
이런 사용자 맞춤 스킬의 수요도 따라 성장한다.

Claude Code 스킬 생태계가 “강력한 스킬 소수”와 “좁은 문제 해결 스킬 다수”로
분화하는 롱테일 구조가 될 가능성이 높다.
VSCode 익스텐션이나 Obsidian 플러그인 생태계가 그런 구조로 성장한 것처럼.
Don't Git Afraid는 그 롱테일의 초기 데이터 포인트다.
