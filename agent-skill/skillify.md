# Skillify - 에이전트 실패를 영구적 스킬로 전환하는 방법론

트윗: <https://twitter.com/garrytan/status/2046876981711769720>

## 소개

Skillify는 AI 에이전트가 반복적으로 실수하는 문제를 일회성 프롬프트 수정이 아니라
구조적 변경으로 해결하는 품질 관리 방법론이다. Garry Tan이 제안하며, gbrain의
내장 기능으로 구현되어 있다.

핵심 발상은 단순하다. 에이전트가 오류를 만들었을 때 그 오류를 “스킬”로 변환한다.
스킬은 마크다운 문서(`SKILL.md`)와 결정론적 스크립트(deterministic script), 자동화
테스트의 조합이다. 한 번 스킬화된 실패는 같은 실수를 다시 유발하지 않는다.

## 두 가지 작업 유형

Skillify의 설계는 작업을 두 종류로 엄격하게 구분한다.

- **잠재적 작업(latent work)**: 판단이 필요한 작업 — LLM 추론이 담당
- **결정론적 작업(deterministic task)**: 정밀도가 필요한 작업 — 코드 실행이 담당

시간대 변환이나 달력 조회처럼 기존 스크립트로 즉시 해결할 수 있는 문제에 LLM
추론을 쓰는 것은 낭비다. 에이전트는 판단이 필요한 영역에만 추론을 써야 한다.

## 10단계 스킬화 프로세스

실패 하나를 스킬로 전환하는 과정은 10단계를 거친다.

1. `SKILL.md` 문서 작성
2. 결정론적 스크립트 생성
3. 단위 테스트(unit test) 작성
4. 통합 테스트(integration test) 작성
5. LLM 평가(LLM evaluation)
6. 리졸버 트리거(resolver trigger) 등록
7. 리졸버 테스트
8. 발견 가능성(discoverability) 감사
9. 엔드-투-엔드 스모크 테스트
10. 브레인 파일링(brain filing) 규칙 적용

이 과정은 “체육관 멤버십이 아닌 운동 루틴”에 비유된다. 도구 자체보다 체계적인
반복 실천이 핵심이라는 뜻이다.

## 분석

### 소프트웨어 엔지니어링의 품질 관리 패턴과의 연결

Skillify는 2005년 전통 소프트웨어 공학이 정립한 패턴을 에이전트 영역에 이식한다.
지속적 통합(CI), 회귀 테스트(regression test), 포스트모템(postmortem) 기반 개선
사이클이 에이전트 실패에 그대로 적용된다.

40개 이상의 스킬을 관리하는 에이전트 시스템에서 지속적 검증이 없으면 스킬 저하
(skill degradation)가 발생한다. 이는 소프트웨어에서 테스트 없는 리팩터링이 회귀를
유발하는 것과 동일한 구조다.

### gbrain과의 통합

GeekNews 댓글에 따르면 Skillify 스킬은 gbrain의 내장 기능으로 제공된다
([GeekNews 댓글](https://news.hada.io/topic?id=28777)). 이는 방법론과
구현체가 같은 생태계 안에 있다는 의미다. gbrain을 사용하는 팀은 별도의 도구 없이
Skillify 사이클을 바로 실천할 수 있다.

### 발견 가능성 문제

10단계 중 “발견 가능성 감사”가 별도 단계로 존재한다는 점이 흥미롭다. 스킬이
잘 만들어졌어도 에이전트가 그 스킬을 찾지 못하면 의미가 없다. 이는 코드 라이브러리
설계에서 API 발견 가능성(discoverability)이 API 기능만큼 중요한 것과 같다.

## 비평

### 강점: 실패를 구조적 자산으로 전환

기존 접근법은 에이전트 실패 후 프롬프트를 수정하는 데 그친다. 수정은 문맥에 묶여
있어 다른 대화나 다른 에이전트에 전달되지 않는다. Skillify는 실패를 테스트 가능하고
재사용 가능한 구조물로 변환함으로써 조직 지식(organizational knowledge)처럼
축적되게 만든다.

### 약점: 10단계는 무겁다

단일 실패를 10단계로 처리하는 것은 overhead가 크다. 실수의 중요도와 빈도를
구분하지 않으면, 드물게 발생하고 영향이 작은 실패에도 동일한 비용을 치러야 한다.
경량 스킬화 경로가 없는 현재 설계는 실천 진입 장벽이 높다.

### 약점: gbrain 생태계 종속성

Skillify가 gbrain의 내장 기능으로 설계되었다면, 이 방법론의 독립적 채택 가능성은
낮다. 방법론 자체는 범용적이지만 구현은 특정 도구에 묶여 있다. 개방형 표준 없이
생태계 락인(lock-in)이 발생할 수 있다.

## 인사이트

### 에이전트 실패는 조직적으로 축적된다

소프트웨어 버그는 코드베이스에 누적된다. 에이전트 실패는 어디에 누적되는가?
현재 대부분의 팀은 에이전트 실패를 대화 맥락 안에서만 처리하고 다음 세션에서는
처음부터 다시 실패를 겪는다. Skillify의 핵심 통찰은 에이전트 실패도 버전 관리되고
테스트되어야 할 구조물이라는 것이다. 이 관점이 보편화되면 에이전트 품질 관리는
소프트웨어 엔지니어링과 동일한 방법론적 체계를 갖추게 된다.

### LLM과 코드의 역할 분리가 시스템 설계의 핵심이 된다

“잠재적 작업은 LLM, 결정론적 작업은 코드”라는 분리는 단순해 보이지만 실전에서
지키기 어렵다. LLM은 날짜 계산도 “할 수 있다” — 하지만 해서는 안 된다. 이 경계를
명시적으로 설계하는 것이 에이전트 시스템 아키텍처의 핵심 과제로 부상하고 있다.
미래의 에이전트 프레임워크는 이 경계를 강제하는 타입 시스템(type system)에 해당하는
메커니즘을 갖춰야 할 것이다.

### 방법론의 스킬화는 문화의 코드화다

Skillify는 숙련된 엔지니어가 포스트모템 후 취하는 행동 패턴을 공식 절차로 만든다.
“다음에는 이런 실수를 안 하겠다”는 개인적 결심이 아니라, 조직이 실패를 자산으로
전환하는 프로세스다. 이는 에이전트 품질 관리를 개인의 프롬프트 공예에서
팀 엔지니어링 실천으로 격상시킨다. 에이전트 시대의 “테스트 문화”가 어떤 모습이어야
하는지를 보여주는 초기 신호다.
