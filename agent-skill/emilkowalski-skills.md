# Skills for Design Engineers (emilkowalski/skills)

<https://github.com/emilkowalski/skills>

## 소개

Vercel과 Linear에서 디자인 엔지니어로 일해온 Emil Kowalski가 공개한 AI 에이전트
스킬 모음이다. [animations.dev](https://animations.dev/)를 운영하며 애니메이션과
UI 디테일을 오래 다뤄온 저자가, 자신이 코드 리뷰에서 반복적으로 지적해 온
판단 기준을 스킬(skill) 형태로 정리해 에이전트에게 이식하려는 시도다. MIT
라이선스로 공개되어 있고, 짧은 기간에 스타 1만 개를 넘기며 눈에 띄게 확산됐다.

핵심 문제의식은 “에이전트에게는 안목(taste)이 없다”는 것이다. 에이전트는
easing 함수를 ease-in과 ease-out 중 아무렇게나 고르거나, 그림자와 색상 값을
근거 없이 결정하는 경향이 있다. 이 저장소는 그런 판단을 명시적 규칙으로
바꿔 에이전트가 참조하도록 만든다.

## 설치법

```bash
npx skills@latest add emilkowalski/skills
```

`skills` CLI를 통해 프로젝트에 스킬을 통합하는 방식이며, 원본 소스 코드를
직접 건드리지 않고 별도 스킬 정의 파일로 추가된다.

## 주요 기능

### emil-design-eng

저자의 애니메이션·디자인 조언을 담은 핵심 스킬이다.

### review-animations

코드베이스에 이미 존재하는 애니메이션을 엄격한 규칙에 따라 검토한다.

### improve-animations

가장 핵심적인 기능으로, 코드베이스 전체를 스캔해 애니메이션을 8개 카테고리로
감사(audit)한다.

- 목적과 빈도 (purpose and frequency)
- Easing과 duration
- 물리적 자연스러움
- 중단 가능성 (interruptibility)
- 성능
- 접근성
- 일관성
- 개선 기회

감사 결과는 `plans/` 폴더에 실행 계획으로 저장되며, 이 계획은 다른(더 저렴한)
에이전트가 이어받아 실행할 수 있도록 설계되어 있다. 사용 예시는 다음과 같다.

```bash
improve-animations quick
improve-animations performance
improve-animations plan add press feedback to all buttons
```

### animation-vocabulary

에이전트가 사용자의 요청을 정확한 용어로 이해하도록 돕는 용어집 성격의
스킬이다.

### apple-design

Apple의 WWDC 디자인 원칙을 웹 환경에 맞게 재해석한 스킬이다.

## 분석

### 감사와 실행의 분리는 shadcn 패턴의 재적용이다

`improve-animations`가 채택한 “상위 모델로 감사하고 저비용 모델로 실행한다”는
구조는 새로운 아이디어가 아니라 shadcn 생태계에서 이미 검증된 패턴을 애니메이션
도메인에 옮겨온 것이다. 비싼 모델은 코드베이스를 훑으며 무엇이 문제인지
판단하는 데만 쓰고, 그 판단을 `plans/` 폴더에 텍스트로 고정한 뒤, 실제 코드
변경은 값싼 모델이 계획을 그대로 따라가며 수행한다. 이 분리가 성립하는 이유는
“어디를 고쳐야 하는가”와 “어떻게 고치는가” 사이에 난이도 격차가 크기 때문이다.
전자는 도메인 판단력이 필요하지만 후자는 대체로 기계적인 패치 작업이다.

이 구조가 흥미로운 지점은 계획 파일이 단순한 로그가 아니라 인터페이스로
기능한다는 것이다. 계획을 파일로 남김으로써 사람이 실행 전에 검토할 기회가
생기고, 서로 다른 세션·다른 모델 간에 작업을 이어받을 수 있다. 자율성을
통째로 위임하는 대신 “판단”과 “실행”이라는 두 단계 사이에 사람이 개입할 수
있는 지점을 만들어 둔 셈이다.

### 스킬은 코드가 아니라 취향의 코드화다

이 저장소가 다루는 대상은 함수나 API가 아니라 “이 상황에서는 ease-out을
쓴다”, “탭 전환에는 200ms가 적절하다” 같은, 문서화되기 어려운 암묵지다.
전통적으로 이런 지식은 시니어 디자인 엔지니어의 코드 리뷰 코멘트로만
전달됐고, 신입이나 다른 팀 엔지니어는 반복된 피드백을 통해 서서히 체득할
수밖에 없었다. 스킬 파일은 이 암묵지를 텍스트 규칙으로 명시화해서, 저자의
코드 리뷰 없이도 동일한 판단 기준을 다른 프로젝트에 적용할 수 있게 만든다.

동시에 이는 위험한 시도이기도 하다. 애니메이션의 “적절함”은 브랜드 톤,
사용자층, 플랫폼 관습에 따라 달라지는 맥락 의존적 판단인데, 저장소는 이를
어느 정도 보편적 규칙으로 일반화하고 있다. 저자 스스로도 “AI는 전문성을
대체하지 않고 증폭한다”고 말하는데, 이는 이 스킬이 전문성이 전무한 상태에서
좋은 결과를 보장하지는 않는다는 암묵적 인정이기도 하다.

### 감사 카테고리 8개는 애니메이션 품질의 체크리스트화다

`improve-animations`가 제시하는 8개 카테고리(목적·easing/duration·물리적
자연스러움·중단 가능성·성능·접근성·일관성·개선 기회)는 개별적으로 보면
새롭지 않은 UI 애니메이션 모범 사례들이다. 그러나 이를 에이전트가 코드베이스
전체에 걸쳐 기계적으로 순회하며 점검하도록 구조화했다는 점이 핵심이다.
사람이 수동으로 수행하면 수십·수백 개의 컴포넌트에 걸쳐 일관되게 적용하기
어려운 체크리스트를, 에이전트의 반복 작업 내성을 이용해 전수 검사로
바꾼 것이다.

이는 코드 린터가 스타일 규칙을 강제하는 방식과 유사한 발상이지만, 대상이
정적 분석으로 판별 가능한 문법 규칙이 아니라 “자연스러움”처럼 맥락적 판단이
필요한 항목이라는 점에서 근본적으로 다른 도전이다. 린터는 규칙 위반을
결정론적으로 찾아내지만, 이 스킬은 LLM의 판단에 의존하므로 같은 코드에 대해
다른 세션에서 다른 결론을 낼 가능성을 안고 있다.

## 비평

### “안목”을 규칙으로 환원하는 순간 저자의 안목만 남는다

저장소의 출발점은 “에이전트에게는 안목이 없다”는 진단이지만, 해법으로
제시된 것은 결국 저자 한 사람의 안목을 코드화한 규칙 집합이다. 이 규칙이
얼마나 보편적으로 타당한지에 대한 근거는 저장소 안에 제시되지 않는다.
Vercel과 Linear라는 특정 회사, 특정 제품군에서 통용되던 판단 기준이 다른
맥락의 프로덕트에도 그대로 적용 가능한지는 검증되지 않은 전제다. 예를 들어
게임형 인터페이스나 데이터 밀도가 높은 대시보드는 Linear류의 미니멀한
마이크로 인터랙션 철학과 다른 애니메이션 관습을 요구할 수 있는데, 이런
도메인 차이를 스킬이 어떻게 처리하는지는 README 수준에서는 확인되지
않는다.

### 8개 카테고리 감사의 결과 신뢰도를 검증할 장치가 없다

`improve-animations`가 감사 결과를 `plans/` 폴더에 저장해 다른 에이전트가
실행하도록 설계했다는 점은 사람의 검토 지점을 만든다는 점에서 긍정적이지만,
동시에 그 계획 자체의 정확성을 검증할 메커니즘은 저장소 설명에 등장하지
않는다. “접근성”이나 “물리적 자연스러움” 같은 판단은 정적 분석으로 자동
검증하기 어려운 영역인데, 에이전트가 잘못된 판단을 계획에 담아도 이를
걸러낼 별도의 검증 단계가 없다면, 계획 파일은 검토 가능한 형태일 뿐 실제로
검토되었다는 보장은 아니다. 사람이 계획을 꼼꼼히 읽지 않고 그대로 실행
에이전트에게 넘긴다면, 파일로 분리한 구조의 장점은 형식적인 것에 그친다.

### “코드를 직접 수정하지 않는다”는 안전성 주장은 범위가 제한적이다

저장소는 원본 소스 코드를 직접 수정하지 않는 방식으로 설계되었다고
설명하지만, 이는 스킬 정의 파일 자체의 설치 방식에 대한 이야기일 뿐,
`improve-animations`가 생성한 계획이 결국 실행되면 프로덕션 코드가
수정된다는 사실과는 별개다. 즉 “안전하다”는 인상을 주는 문구가 실제로는
스킬 설치 단계의 안전성만을 가리키고, 스킬 실행 단계의 안전성(잘못된
애니메이션 변경이 접근성이나 성능을 저해할 가능성)에 대해서는 답하지
않는다. 이 구분이 README에서 명확히 드러나지 않는다면, 사용자는 실행
단계까지 안전하다고 오인할 수 있다.

## 인사이트

### 스킬 생태계는 “감사자-실행자 분리”를 표준 아키텍처로 굳혀가고 있다

이 저장소가 채택한 상위 모델 감사 + 하위 모델 실행 구조는 독립적으로
등장한 아이디어가 아니라, shadcn의 `improve` 계열 도구, 그리고 여러 코드
리뷰 자동화 워크플로에서 이미 반복적으로 관찰되는 패턴이다. 이것이 시사하는
바는 에이전트 오케스트레이션의 다음 단계가 “하나의 강력한 모델에게 전부
맡기기”가 아니라 “역할별로 다른 비용의 모델을 배치하고 그 사이에 파일이라는
고정된 인터페이스를 두기”로 수렴하고 있다는 점이다. 계획 파일은 단순한
중간 산출물이 아니라, 사람의 검토·롤백·재사용을 가능케 하는 감사 추적
(audit trail) 역할을 겸한다. 앞으로 이런 도구들이 늘어날수록, “계획 파일의
스키마를 어떻게 표준화할 것인가”가 스킬 생태계의 상호운용성을 결정하는
숨은 병목이 될 가능성이 높다.

### 안목의 코드화는 필연적으로 특정 시대·특정 취향의 화석화를 동반한다

디자인 트렌드는 시간이 지나면 바뀐다. Material Design의 물리 기반
애니메이션이 한때 표준이었다가 이후 더 절제된 모션으로 트렌드가 옮겨간
것처럼, 이 스킬이 코드화한 easing과 duration 규칙도 몇 년 뒤에는 낡은
관습으로 취급될 수 있다. 문제는 이런 규칙이 사람의 코드 리뷰 코멘트로
존재할 때는 리뷰어 자신이 트렌드 변화에 맞춰 암묵적으로 갱신하지만, 스킬
파일로 고정되는 순간 그 갱신이 저장소 관리자의 의식적인 업데이트에
의존하게 된다는 점이다. 즉 스킬화는 지식 전파의 효율은 높이지만, 그
지식이 시대에 뒤처졌을 때 이를 감지하고 폐기하는 절차는 별도로
설계해야 한다는 부채를 함께 만든다. 이는 코드 스타일 가이드나 린터
규칙이 오래되면 오히려 유해해지는 현상과 같은 계열의 문제이며, 스킬
저장소에도 “규칙의 유효기간”이라는 개념이 필요해질 것이다.

### 애니메이션 감사의 전수 검사화는 리뷰 문화 자체를 바꿀 수 있다

지금까지 애니메이션의 일관성은 대개 시니어 엔지니어가 산발적으로
발견하는 방식으로 관리되어 왔다. 코드베이스가 커질수록 모든 컴포넌트를
사람이 순회하며 검토하는 것은 비용 문제로 사실상 불가능했고, 그 결과
애니메이션 품질은 컴포넌트별로 들쭉날쭉한 채로 방치되는 경우가 많았다.
`improve-animations`처럼 전수 스캔이 가능해지면, 이제 “일관성 부채”를
발견하는 비용이 거의 0에 가까워진다. 이는 팀의 리뷰 관행에도 이차적
영향을 줄 수 있다. 리뷰어가 개별 PR에서 애니메이션 세부사항을 지적하는
대신, 정기적인 전수 감사를 돌려 회귀를 잡아내는 방식으로 역할이
이동할 수 있다는 것이다. 다만 이 변화가 온전히 긍정적이지만은 않다.
전수 스캔이 쉬워질수록 “일단 다 고치자”는 유혹이 커지는데, 애니메이션
변경은 시각적 회귀 테스트 없이는 검증하기 어려운 영역이라, 자동화된
대량 수정이 오히려 사람이 미처 못 본 회귀를 대량으로 만들어낼 위험도
함께 커진다.
