# Loop Library

<https://signals.forwardfuture.ai/loop-library/>

<https://github.com/Forward-Future/loop-library>

## 소개

Loop Library는 AI 에이전트용 재사용 가능한 워크플로 카탈로그다.
단순한 프롬프트 모음집이 아니라, 반복 작업을 루프(loop)라는 구조로 정의하고
에이전트가 결과를 학습해 다음 단계를 결정할 수 있게 만드는 시스템이다.
프로젝트는 두 부분으로 구성된다.
공개 카탈로그 웹사이트와 Claude Code, Cursor, Codex 등 코딩 에이전트에 설치할 수 있는 스킬이다.

2026년 6월 Matthew Berman이 출시를 알리며 "루프를 찾고, 직접 제출하고, tokenmaxx하라"고 밝혔다.[^berman]
tokenmaxx는 토큰을 최대한 효율적으로 소비해 에이전트 성능을 끌어올린다는 의미로 쓰이는 신조어로,
루프 기반 워크플로가 이 목적에 적합하다는 주장을 함축한다.

루프의 핵심 개념은 원샷(one-shot) 프롬프트와의 대비에서 나온다.
원샷 프롬프트는 한 번 실행하고 끝나지만, 루프는 에이전트가 결과를 확인하고 다음에 무엇을 할지 결정한다.
각 루프는 네 가지 질문에 답한다.
목표가 무엇인지, 성공을 어떻게 측정할지, 학습한 내용으로 무엇을 할지, 언제 작업을 멈추거나 에스컬레이션할지다.

카탈로그는 현재 61개의 루프를 다섯 카테고리로 분류해 제공한다.
엔지니어링(코드 최적화, 테스트, 문서화, 아키텍처), 평가(품질 보증, 성능 측정),
운영(릴리스 관리, 데이터 정리, 유지보수), 콘텐츠(글쓰기, 퍼블리싱, 미디어 제작), 디자인(UI/UX 개선)이다.

## 주요 기능

스킬은 다섯 가지 워크플로를 지원한다.

**Discover**는 코드베이스와 코딩 스레드를 분석해 반복되는 작업 패턴을 찾아낸다.
개발자가 의식하지 못하는 사이에 반복하는 작업을 루프로 만들 수 있는 후보를 식별한다.

**Find**는 사용자 요구에 맞는 루프를 카탈로그에서 검색한다.
라이브 카탈로그를 기반으로 동작하므로 새 루프가 추가될 때마다 즉시 반영된다.

**Loop Doctor**는 기존 루프를 감사(audit)하고 구조적 약점을 수리한다.
종료 조건이 불명확하거나 성공 기준이 모호한 루프를 개선하는 데 쓰인다.

**Adapt**는 공개된 루프를 특정 도구와 제약 조건에 맞게 조정한다.
카탈로그의 루프를 그대로 쓰기 어려울 때 팀 환경에 맞게 변형한다.

**Design**은 대화를 통해 새 루프를 처음부터 만든다.
사용자가 원하는 반복 작업을 구조화된 루프 형태로 설계한다.

## 설치

```bash
npx skills add Forward-Future/loop-library --skill loop-library --agent claude -g -y
```

플랫폼별로 `--agent` 값을 `codex`, `cursor`, `claude` 중 하나로 지정한다.
`-g` 플래그는 글로벌 설치를 의미하고, `-y`는 확인 없이 바로 설치한다.

루프 레코드는 GitHub이 아닌 Cloudflare Worker 데이터베이스에 저장된다.
커밋이나 재배포 없이도 카탈로그를 업데이트할 수 있는 구조다.

## 분석

### 루프라는 추상화가 채우는 공백

현재 AI 에이전트 생태계에서 가장 큰 실용적 문제는 “에이전트가 언제 멈춰야 하는가”다.
원샷 프롬프트는 이 문제를 회피한다.
한 번 실행하고 끝나므로 종료 조건이 필요 없다.
반면 자율 에이전트는 반대 극단에 있다.
에이전트 스스로 다음 행동을 결정하다 보면 무한 루프에 빠지거나, 의도하지 않은 방향으로 흘러가는 위험이 있다.

Loop Library의 루프 개념은 이 두 극단 사이의 중간 지점을 정의한다.
에이전트는 스스로 다음 단계를 결정할 수 있지만,
루프가 정의한 종료 조건과 성공 기준 안에서만 움직인다.
“최대 세 번 반복 후 에스컬레이션”처럼 명시적인 경계가 있는 자율성이다.
이는 에이전트 자율성의 폭발이 가져오는 통제 불능 문제를 구조적으로 예방하는 설계다.

### 카탈로그 방식의 전략적 의미

루프를 코드로 커밋하지 않고 Cloudflare Worker 데이터베이스에 저장하는 선택은 기술적 결정이면서 동시에 전략적 결정이다.
사용자는 스킬을 재설치하지 않아도 새 루프를 즉시 쓸 수 있다.
루프 제작자는 배포 파이프라인 없이 카탈로그를 업데이트할 수 있다.
커뮤니티 기여를 리뷰 큐로 받아 관리한다는 점에서 오픈소스 패키지 레지스트리보다 큐레이션된 콘텐츠 플랫폼에 더 가깝다.

이 구조는 스킬 생태계를 “코드 라이브러리”가 아닌 “레시피 북”으로 포지셔닝한다.
npm 패키지처럼 버전 관리와 의존성 해결이 필요한 것이 아니라,
요리책처럼 원하는 레시피를 골라 상황에 맞게 변형하는 것이다.
이 비유는 Adapt 워크플로의 존재가 더 분명하게 만들어준다.

### 인간 통제권의 구조적 보장

루프의 설계 원칙 중 하나는 “자동화는 명시적 승인 없이 발생하지 않는다”는 것이다.
에이전트가 제안하고 인간이 승인하는 구조를 루프 차원에서 강제한다.
단순히 “에이전트를 조심히 써야 한다”는 권고가 아니라, 루프 자체가 에스컬레이션 조건을 포함하도록 설계되어 있다.

이는 AI 에이전트 신뢰 문제에 대한 실용적 접근이다.
AGI 수준의 신뢰를 기다리지 않고, 현재 에이전트의 능력 범위 안에서 안전하게 반복 작업을 위임할 수 있는 구조를 만든다.
“측정 가능한 진행”, “명확한 종료선”, “재사용 가능성”이라는 루프의 세 가지 특성은
인간이 에이전트를 신뢰하기 위해 필요한 최소한의 조건을 명세한 것으로 읽힌다.

## 비평

### 루프 품질의 불균일성을 통제할 수단이 없다

카탈로그에 61개의 루프가 있고 커뮤니티 기여를 받는다.
그러나 루프의 품질을 보장하는 구조적 메커니즘이 공개되어 있지 않다.
“private review queue”가 있다고 명시되어 있지만, 어떤 기준으로 리뷰하는지, 루프가 실제로 잘 동작하는지 검증하는 절차가 있는지 알 수 없다.

이것은 심각한 구조적 문제다.
루프가 나쁜 종료 조건을 가지고 있으면 에이전트는 루프가 끝났다고 판단해 중요한 작업을 건너뛸 수 있다.
성공 기준이 측정 불가능한 방식으로 정의된 루프는 Loop Doctor가 있어도 발견하기 어렵다.
카탈로그가 성장할수록 나쁜 루프가 섞여들어갈 가능성도 커진다.

npm의 패키지 품질 문제가 생태계 성숙에 따라 자연스럽게 나타났듯,
Loop Library도 같은 문제에 직면할 것이다.
평점 시스템, 실행 통계, 에러 레이트 같은 신호가 없으면 사용자는 루프를 골라야 할 때 맹목적으로 선택하게 된다.

### 루프 추상화는 에이전트 차이를 숨긴다

Loop Library는 Claude Code, Cursor, Codex를 같은 `--agent` 플래그로 지원하며 플랫폼 중립적인 루프를 표방한다.
그러나 에이전트마다 도구(tool) 접근 방식, 컨텍스트 창 크기, 에러 처리 방식이 다르다.
Claude Code에서 잘 동작하는 루프가 Cursor에서는 다르게 동작할 수 있다.

루프가 “플랫폼 중립”이라는 전제는 에이전트를 단순한 실행 엔진으로 취급할 때만 성립한다.
실제로 에이전트의 도구 호출 방식, 메모리 관리, 오류 복구 능력은 플랫폼마다 현저히 다르다.
성능 최적화 루프가 “가장 느린 페이지를 찾아라”라고 지시할 때, 어떤 도구로 찾고
어떤 기준으로 느린 것으로 판단할지는 에이전트마다 다른 결과를 낸다.

Adapt 워크플로가 이 문제를 부분적으로 완화하지만, 사용자가 매번 적응 과정을 거쳐야 한다면
“재사용 가능한 루프”라는 핵심 가치가 희석된다.

### 루프 설계 능력이 전제가 된다는 역설

Loop Library의 가장 강력한 기능은 Design 워크플로다.
처음부터 새 루프를 만들 수 있다.
그런데 좋은 루프를 만들려면 무엇이 좋은 루프인지 알아야 한다.
명확한 종료 조건, 측정 가능한 성공 기준, 적절한 에스컬레이션 조건을 정의하는 능력 자체가 전문 지식이다.

이는 라이브러리가 해결하려는 문제와 역설적으로 연결된다.
반복 작업을 루프로 만들지 않아서 비효율적인 사람이 Loop Library의 주요 대상이다.
그러나 그런 사람은 좋은 루프를 설계할 능력도 부족할 가능성이 높다.
Discover 워크플로가 반복 패턴을 찾아주지만, 발견된 패턴을 루프로 변환하는 품질은 사용자 역량에 크게 의존한다.

Loop Doctor가 기존 루프를 개선해주지만, 초기 설계가 근본적으로 잘못된 루프를 고치는 데는 한계가 있다.
루프 설계 능력을 키우는 교육 자원, 예제 설명, 안티패턴 가이드가 카탈로그만큼 중요하지만, 현재는 이 부분이 얇다.

## 인사이트

### 루프는 에이전트 행동의 계약이다

Loop Library가 제시하는 루프 개념을 소프트웨어 엔지니어링 관점에서 재해석하면, 루프는 에이전트와 인간 사이의 계약(contract)이다.
종료 조건은 사후 조건(postcondition)이고, 성공 기준은 불변 조건(invariant)이며, 에스컬레이션 조건은 예외 처리(exception handling)다.
Design by Contract 방법론이 함수 수준에서 정의했던 것을 에이전트 워크플로 수준으로 끌어올린 것이다.

이 관점에서 보면 Loop Library가 해결하는 진짜 문제가 더 선명해진다.
현재 AI 에이전트 프롬프트의 대부분은 계약 없는 코드다.
에이전트가 무엇을 할지 명시하지만, 언제 멈출지, 성공했는지 어떻게 알지, 실패했을 때 무엇을 해야 할지 정의하지 않는다.
루프는 이 공백을 채운다.
에이전트가 성숙해질수록 계약 기반 설계가 에이전트 신뢰성의 표준이 될 가능성이 높다.
Loop Library는 그 방향의 초기 구현체다.

### 카탈로그 경쟁은 콘텐츠가 아닌 피드백 루프에서 결판난다

Loop Library의 성공 여부는 카탈로그의 루프 수나 품질에만 달려 있지 않다.
루프가 실행된 후 피드백이 카탈로그로 돌아오는 루프, 즉 메타 루프(meta-loop)가 존재하는지에 달려 있다.

npm의 성공 요인 중 하나는 다운로드 수와 의존성 그래프라는 신호가 생태계에 피드백을 만들었다는 것이다.
어떤 패키지가 잘 쓰이는지, 어떤 패키지가 문제가 많은지 신호가 흘렀다.
Loop Library가 루프 실행 데이터, 성공률, 사용자 평가를 카탈로그에 연결하면
카탈로그는 단순한 목록을 넘어 자기 개선하는 시스템이 된다.

이 피드백 루프가 없으면 카탈로그는 정체되고, 있으면 네트워크 효과로 빠르게 성장한다.
Cloudflare Worker를 기반으로 한 라이브 카탈로그 아키텍처는 이 피드백을 기술적으로 구현하기에 적합하다.
이것이 실현된다면 Loop Library는 단순한 루프 저장소가 아니라 에이전트 워크플로 품질의 집단 지성이 된다.

### 스킬과 루프의 경계가 허물어지면 에이전트 생태계가 재편된다

현재 Claude Code 생태계에는 스킬(skill)과 루프(loop)라는 두 가지 재사용 단위가 공존한다.
스킬은 에이전트에 새로운 능력을 추가하는 도구다.
루프는 그 능력을 어떤 순서로, 어떤 조건 아래 사용할지 정의하는 워크플로다.

지금은 이 둘이 분리되어 있지만, 루프가 스킬을 호출하고 스킬이 루프를 트리거하는 통합이 이루어지면 에이전트 구성 방식이 바뀐다.
스킬이 원자(atom)이고 루프가 분자(molecule)라면, 복잡한 에이전트 행동은 이 둘의 조합으로 표현된다.
Loop Library가 스킬 생태계와 통합되는 방향으로 발전한다면,
에이전트를 구성하는 방식이 “어떤 스킬을 설치할까”에서 “어떤 루프를 쓸까”로 이동할 수 있다.

이 전환은 에이전트 워크플로의 표준화를 가속한다.
루프가 스킬보다 더 높은 수준의 추상화이므로, 루프 중심의 생태계에서는 에이전트 간 이식성이 더 높아진다.
Loop Library가 이 위치를 선점하면 코딩 에이전트 워크플로의 사실상 표준 레지스트리가 될 수 있다.
