# korean-skills: AI 에이전트를 위한 한국어 스킬 모음

<https://github.com/DaleSeo/korean-skills>

<https://www.skills.sh/daleseo/korean-skills>

## 소개

korean-skills는 Claude Code, Cursor, Windsurf 등 Agent Skills 형식을
지원하는 AI 에이전트에서 쓸 수 있는 한국어 특화 스킬 3종을 제공하는
저장소다.
설치 방법은 `npx skills add`, Claude Code 플러그인 마켓플레이스,
GitHub CLI의 `gh skill install` 세 갈래로 열려 있다.

세 스킬은 각각 다른 층위의 한국어 품질 문제를 다룬다.
**humanizer**는 AI가 생성한 한국어 텍스트의 “AI 티”를 제거해 자연스러운
사람의 글로 바꾼다.
6개 카테고리 40가지 검출 패턴을 사용하며, 문장부호 관련 패턴은 KatFishNet
논문(94.88% AUC) 근거를 명시하고 있다.
문장부호, 띄어쓰기, 품사 다양성, 어휘(대명사·지시관형사 과다, AI 결말
표현, hype 어휘, `~적 N` 추상 체인), 문장 구조, 번역투(조사 번역투, 피동
남용, 미래 단정 등) 총 여섯 카테고리를 검사한다.

**grammar-checker**는 표준 한국어 규칙에 기반해 문법·맞춤법·띄어쓰기·
구두점을 검사한다.
오류를 맞춤법(최고 우선순위), 띄어쓰기(높은 우선순위), 문법 구조(중간),
구두점(낮은 우선순위) 네 단계로 나누고, 확실한 오류와 권장 사항을
구분해 확신도를 표시한다.

**style-guide**는 문서 내부 또는 프로젝트 전체의 작성 스타일 일관성을
검사한다.
어조·격식, 용어 통일, 숫자·단위, 목록 구조, 인용·강조, 날짜·시간,
링크·참조 7개 카테고리를 검사하며, 국립국어원 공문서 작성 지침·대학
학위논문 작성 지침·Kakao Enterprise 기술문서 가이드 등 다층적 권위
출처를 참조 자료로 명시하고 충돌하는 스타일은 다수결로 해소한다.

저장소는 세 스킬을 순차 적용하는 “3-스킬 파이프라인”을 권장 워크플로로
제시한다.
humanizer(AI 티 제거) → grammar-checker(문법 검사) → style-guide(일관성
검사) 순서다.
humanizer가 문장을 크게 바꾸므로 안정된 출력에 대해 grammar-checker가
의미 있는 검사를 할 수 있고, style-guide는 일관성 검사이므로 글이
안정된 상태에서 마지막에 적용된다는 것이 저장소가 설명하는 순서 근거다.

## 분석

### 세 스킬의 역할 분리가 반영하는 언어 품질의 위계

humanizer, grammar-checker, style-guide는 우연히 나열된 세 기능이
아니라 언어 품질을 바라보는 위계를 반영한다.
grammar-checker는 규범 문법(정답이 있는 층위)을 다루고, style-guide는
일관성(문서 내부 상대적 기준)을 다루며, humanizer는 “AI스러움”이라는
훨씬 더 모호하고 통계적인 층위를 다룬다.
문법 오류는 규칙으로 판정 가능하지만, “AI가 쓴 것 같다”는 판단은 확률
분포의 문제다.
이 저장소가 세 스킬을 굳이 분리해서 설계한 것은, 서로 다른 판단 근거를
가진 검사를 하나로 뭉뚱그리면 각각의 정밀도가 떨어진다는 인식을
전제한다.

### 파이프라인 순서 논리는 “안정성”을 기준으로 삼는다

3-스킬 파이프라인의 순서 근거는 명확하다.
가장 많이 바꾸는 작업을 먼저 하고, 변화량이 적은 검사를 나중에 배치해야
뒤 단계가 안정된 대상 위에서 작동한다는 논리다.
이것은 소프트웨어 빌드 파이프라인에서 “가장 자주 바뀌는 레이어를 앞에,
캐시 가능한 안정 레이어를 뒤에” 두는 설계와 구조적으로 동일하다.
humanizer의 출력이 완전히 안정되지 않은 상태에서 grammar-checker를
먼저 돌리면, humanizer가 나중에 새로 만들어낸 문장에 다시 오탈자가
섞여 들어갈 수 있다.
순서를 지정한 것은 이 재작업 낭비를 막기 위한 설계 판단이다.

### 다층적 권위 출처를 명시하는 것의 의미

style-guide가 정부 표준, 학술 지침, 실무 가이드를 나란히 참조 자료로
제시하고 충돌 시 다수결로 해소한다고 밝힌 점은 주목할 만하다.
언어 스타일에는 절대적 정답이 없고, 국립국어원 규범과 실제 기업
기술문서 관행이 종종 어긋난다는 사실을 스킬 설계자가 이미 알고 있다는
뜻이다.
권위 출처를 하나로 단일화하지 않고 여러 개를 병기한 뒤 다수결이라는
명시적 해소 규칙을 둔 것은, 언어 규범이 단일 권위가 아니라 여러
공동체의 관행이 경합하는 영역이라는 현실을 스킬 설계에 반영한
결과다.

### 배포 경로의 다양성이 드러내는 Agent Skills 생태계의 파편화

이 저장소는 설치 경로를 네 가지(`npx skills`, Claude Code 플러그인,
`gh skill`, 직접 클론)로 제공한다.
하나의 스킬 저장소가 이렇게 많은 배포 채널을 동시에 지원해야 한다는
사실 자체가, Agent Skills라는 포맷이 아직 단일 표준 배포 창구를 갖지
못했다는 방증이다.
`npx skills`는 npm 생태계를 빌리고, `gh skill`은 GitHub CLI 확장을
빌리며, 플러그인 마켓플레이스는 Claude Code 고유 체계를 쓴다.
스킬 저자가 어느 하나만 지원하면 특정 도구 사용자를 배제하게 되므로,
현재로서는 여러 경로를 동시에 유지하는 비용을 감수할 수밖에 없는
구조다.

## 비평

### “AI 티” 판정 기준이 검증 가능성 없이 제시된다

humanizer는 KatFishNet 논문의 94.88% AUC라는 구체적 수치를 근거로
내세우지만, 이 수치는 논문이 검증한 탐지 모델의 성능이지 이 스킬이
그 성능을 그대로 재현한다는 보장이 아니다.
논문 기반 패턴과 “커뮤니티 검증 경험적 패턴”이 나열식으로 섞여 있는데,
어느 패턴이 논문 근거이고 어느 패턴이 순수 경험칙인지 README만으로는
구분하기 어렵다.
학술 근거를 인용하는 방식이 신뢰도를 높이는 장치로 쓰였다면, 최소한
어느 항목이 그 근거에 직접 대응하는지는 밝혀야 마땅하다.
근거 인용과 실제 적용 범위 사이의 간극이 불투명한 채로 남아 있다.

### 다수결 해소 규칙은 소수 스타일을 구조적으로 배제한다

style-guide가 충돌하는 스타일을 다수결로 해소한다고 명시한 것은
편의적이지만, 이 규칙은 특정 문서나 저자가 의도적으로 소수 스타일을
택한 경우를 구분하지 못한다.
예를 들어 한 문서가 의도적으로 아라비아 숫자만 쓰기로 정했는데, 문서
대부분이 한글 숫자를 쓴다면 다수결은 저자의 의도적 선택을 “불일치”로
오판할 위험이 있다.
스타일 일관성 검사는 원래 저자 의도를 존중하면서 그 의도 안에서의
일관성을 검사해야 하는데, 다수결이라는 단순 규칙은 의도 자체를
다수 쪽으로 밀어붙이는 부작용을 낳을 수 있다.

### 순서 강제가 되레 유연성을 제약한다

3-스킬 파이프라인 순서를 “이래야 한다”는 식으로 못 박은 것은, 실제
사용 맥락에서는 과도한 제약일 수 있다.
이미 문법적으로 검증된 원고에 humanizer만 적용하고 싶은 사용자,
혹은 style-guide로 용어만 통일하고 humanizer는 건너뛰고 싶은 사용자도
있을 텐데, 저장소 문서는 순서의 근거만 설명할 뿐 “왜 항상 이 순서여야
하는가”에 대한 예외 조건은 다루지 않는다.
humanizer 패턴 24(경어체 균일성)와 style-guide 어조 일관성 검사의
관계를 “글 단위 변주 대 단락 단위 일관”으로 구분해 설명한 부분은
정교하지만, 정작 사용자가 두 스킬 중 하나만 쓰고 싶을 때 무엇을
포기하게 되는지는 설명하지 않는다.

### 설치 경로가 네 갈래로 나뉘면서 버전 관리 책임이 사용자에게 전가된다

`gh skill install --pin`으로 특정 릴리스에 고정할 수 있다는 안내는
있지만, `npx skills add`는 `main` HEAD를 그대로 추적한다고 명시되어
있다.
즉 같은 저장소를 서로 다른 경로로 설치한 두 사용자가 전혀 다른 버전의
스킬을 실행하고 있을 수 있다.
이것은 재현성 문제로 이어진다.
한 사용자가 “이 스킬이 이렇게 동작했다”고 보고해도, 다른 사용자의
환경에서는 설치 경로에 따라 재현되지 않을 수 있다.
저장소가 이 버전 분기를 표로 명시한 것은 투명하지만, 문제 자체를
해결하지는 않는다.

## 인사이트

### 언어 품질 검사 스킬은 코드 린터의 발전 경로를 반복한다

이 저장소의 세 스킬 구조는 소프트웨어 개발에서 코드 품질 도구가
진화해 온 순서와 놀랍도록 닮았다.
초기 린터는 문법 오류(컴파일 에러 수준)만 잡았고, 그다음 스타일
가이드 도구(ESLint, Prettier)가 등장해 일관성을 검사했으며, 최근에는
“이 코드가 AI가 생성한 것처럼 보이는가”를 판별하는 도구까지 나타나고
있다.
grammar-checker는 첫 세대, style-guide는 두 번째 세대, humanizer는
세 번째 세대에 대응한다.
언어 텍스트도 코드와 마찬가지로 “작동하는가”에서 “일관되는가”를 거쳐
“누가 만들었는가를 감추어야 하는가”라는 질문으로 품질 기준이
확장되고 있다는 뜻이다.
이 확장이 코드보다 텍스트에서 먼저, 더 빠르게 일어나는 이유는 텍스트
생성 AI가 코드 생성 AI보다 먼저 대중적으로 보급되었기 때문이다.
앞으로 코드 영역에서도 “이 커밋 메시지가 AI가 쓴 것 같다”를 판정하는
유사 도구가 자리 잡을 가능성이 있다.

### “AI 티 제거”의 보급은 튜링 테스트의 역전을 만든다

전통적 튜링 테스트는 기계가 인간을 속일 수 있는지 묻는다.
humanizer 같은 스킬이 대중화된다는 것은, 이제 인간(정확히는 AI를 쓰는
인간)이 자신의 결과물을 다시 다른 AI에게 맡겨 “더 인간처럼” 다듬는
2차 가공 단계가 표준 워크플로로 자리 잡는다는 뜻이다.
이 구조에서 최종 독자는 텍스트를 읽을 때 그것이 AI가 쓴 것인지, 사람이
쓴 것인지, AI가 쓰고 다른 AI가 사람처럼 다듬은 것인지 구분할 근거를
점점 잃는다.
탐지 모델(KatFishNet 같은)과 회피 도구(humanizer)가 같은 패턴
카탈로그를 놓고 서로 다른 방향으로 학습하는 구도는, 스팸 필터와
스팸 발송자 사이의 군비 경쟁과 같은 형태로 발전할 가능성이 크다.
탐지 정확도가 오르면 회피 도구도 그만큼 정교해지는 순환이 반복되면,
결국 “AI가 썼는지 여부”라는 질문 자체가 실무적으로 무의미해지는
지점에 도달할 수 있다.
그때는 텍스트의 출처가 아니라 내용의 사실성과 유용성만이 유의미한
평가 기준으로 남을 것이다.

### 권위 출처의 병렬 나열은 규범 언어학에서 기술 언어학으로의 이동을 보여준다

style-guide가 국립국어원(정부 규범), 대학 학위논문 지침(학술 규범),
Kakao Enterprise 가이드(기업 실무 규범)를 동등한 층위로 병렬 배치한
것은 사소해 보이지만 함의가 크다.
전통적으로 “표준 한국어”는 국립국어원이라는 단일 권위가 정하는
것이었다(규범 언어학, prescriptive linguistics).
그런데 이 스킬은 실제 언어 사용 관행(기업 기술문서, 학술 관행)을
정부 표준과 같은 무게로 취급하고 다수결로 조정한다.
이것은 언어를 “정해진 대로 써야 하는 것”이 아니라 “실제로 쓰이는
방식들의 합의”로 다루는 기술 언어학(descriptive linguistics)적
접근이다.
AI 에이전트가 언어 스타일 도구를 설계할 때 규범이 아니라 실제 사용
데이터를 권위로 삼기 시작했다는 것은, 언어 표준화의 주도권이 국가
기관에서 실제 텍스트 생산 주체(기업, 학계, 커뮤니티)로 조금씩
이동하고 있다는 더 큰 흐름의 한 사례로 읽을 수 있다.
