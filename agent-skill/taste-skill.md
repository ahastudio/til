# Taste Skill: AI 에이전트용 안티-슬롭 프론트엔드 스킬

<https://www.tasteskill.dev/>

<https://github.com/Leonxlnx/taste-skill>

## 소개

Taste Skill은 AI 에이전트가 생성하는 프론트엔드 UI의 품질을 높이기 위한
포터블 에이전트 스킬 모음이다. “슬롭(slop)”이란 AI가 기본값으로 생성하는
뻔하고 제네릭한 UI를 지칭하는 표현으로, Taste Skill은 이를 방지하는
규칙 집합을 SKILL.md 형식으로 제공한다. 2026년 2월 시작해 스타 22,000개 이상을
받은 프로젝트다. Claude Code, Codex, Cursor 등 주요 AI 코딩 에이전트를 지원한다.

핵심 스킬인 `design-taste-frontend`(v2, 실험적)는 세 개의 다이얼로 동작한다.

| 다이얼            | 범위 | 기본값 | 의미                                        |
| ----------------- | ---- | ------ | ------------------------------------------- |
| `DESIGN_VARIANCE` | 1-10 | 8      | 레이아웃 실험성 (낮음: 대칭/정제 · 높음: 비대칭/모던) |
| `MOTION_INTENSITY`| 1-10 | 6      | 애니메이션 깊이 (낮음: 호버 · 높음: 스크롤/자기)  |
| `VISUAL_DENSITY`  | 1-10 | 4      | 뷰포트당 정보량 (낮음: 여백 · 높음: 대시보드)     |

스킬 목록은 용도별로 세분화돼 있다.

| 스킬                        | 설치 이름                   | 설명                                           |
| --------------------------- | --------------------------- | ---------------------------------------------- |
| `taste-skill`               | `design-taste-frontend`     | v2 기본 스킬. 브리프 추론, 디자인 시스템 매핑 |
| `taste-skill-v1`            | `design-taste-frontend-v1`  | v1 원본. 정확한 동작 의존 시 사용              |
| `gpt-tasteskill`            | `gpt-taste`                 | GPT/Codex용 강화 변형                          |
| `image-to-code-skill`       | `image-to-code`             | 이미지 → 분석 → 코드 파이프라인               |
| `redesign-skill`            | `redesign-existing-projects`| 기존 UI 감사 후 레이아웃/간격/위계 수정        |
| `soft-skill`                | `high-end-visual-design`    | 여백, 프리미엄 폰트, 스프링 모션               |
| `output-skill`              | `full-output-enforcement`   | 에이전트가 출력을 잘라낼 때 강제 전체 출력     |
| `minimalist-skill`          | `minimalist-ui`             | Notion/Linear 느낌, 절제된 팔레트              |
| `brutalist-skill`           | `industrial-brutalist-ui`   | 스위스 타이포, 강한 대비, 실험적 레이아웃      |
| `stitch-skill`              | `stitch-design-taste`       | Google Stitch 호환 규칙                        |

이미지 생성 전용 스킬도 포함된다. `imagegen-frontend-web`, `imagegen-frontend-mobile`,
`brandkit`은 코드가 아닌 디자인 레퍼런스 이미지를 생성하며, ChatGPT Images나
Codex 이미지 모드와 함께 사용한다. 생성된 이미지를 `image-to-code-skill`에 넘겨
코드로 구현하는 파이프라인이 권장된다.

설치는 `npx skills add` CLI 한 줄로 처리된다.

```bash
npx skills add https://github.com/Leonxlnx/taste-skill
npx skills add https://github.com/Leonxlnx/taste-skill --skill "design-taste-frontend"
```

## 분석

### v2 스킬의 핵심: 브리프 추론 우선 원칙

v2의 가장 큰 변화는 섹션 0 “브리프 추론(Brief Inference)”이다.
코드 생성 전에 에이전트가 먼저 브리프를 분석하고
“이 브리프를 다음과 같이 해석합니다: …”라는 한 줄 선언을 출력하도록 강제한다.
예: “B2B SaaS 랜딩, 기술 구매자 대상, Linear 스타일 미니멀리즘, Tailwind v4 + Geist.”

이 단계는 AI 디자인 실패의 근본 원인 — 브리프를 읽지 않고 기본 미학으로 점프하는 것 —
을 차단한다. 섹션 0.D “안티-기본값 규율”은 금지 기본값 목록을 명시한다.
AI-퍼플 그라디언트, 어두운 배경 위 중앙 배치 히어로, 동일한 세 개 피처 카드,
범용 글라스모피즘, Inter + slate-900. 이것들이 LLM 기본값이므로
의도적으로 피해야 한다고 명시한다.

### 구체적 금지 목록: “AI Tells” 섹션의 방법론

v2에서 가장 구체적이고 흥미로운 부분은 섹션 9 “AI Tells” —
AI가 생성한 것임을 드러내는 시각적 신호 목록이다.
Em-대시(`—`) 완전 금지가 첫 번째다. 헤드라인, 버튼 텍스트, 대체 텍스트 어디서도
Em-대시를 쓰면 안 된다. v2 이전 테스트에서 가장 빈번하게 위반된 규칙이었다고 명시한다.

기타 금지 목록은 구체적이다. 섹션 번호 눈썹(`00 / INDEX`, `06 · 작동 방식`),
버전 라벨 히어로(`V0.6`, `INVITE-ONLY PREVIEW`), 사진 크레딧 장식 캡션,
히어로 하단 장식 텍스트 스트립(`BRAND. MOTION. SPATIAL.`),
이미지 위에 겹친 필/라벨, 스크롤 큐(`↓ scroll`), 장식용 상태 점.
이 목록이 흥미로운 이유는 각각이 특정 디자인 트렌드가 과포화됐음을 시사하기 때문이다.

### 디자인 시스템 매핑: 실존 패키지 우선 원칙

섹션 2 “브리프 → 디자인 시스템 맵”은 브리프의 성격에 따라
공식 디자인 시스템 패키지를 사용하도록 안내한다.
Microsoft 스타일이면 `@fluentui/react-components`, Google 스타일이면 `@material/web`,
IBM 스타일이면 `@carbon/react`, Shopify 앱이면 `polaris.js`.
“공식 패키지를 발명하지 말라”는 원칙이다.

동시에 CSS 트렌드와 공식 시스템을 명확히 구분한다.
글라스모피즘, 벤토, 브루탈리즘, 오로라, 키네틱 타이포그래피는
공식 패키지가 없는 미적 경향이므로 웹 표준으로 구현하고
“근사치 구현”임을 명시해야 한다고 규정한다.
Apple Liquid Glass는 공식 패키지가 없으므로 근사치로만 구현 가능하다는 점을 명시한다.

### npx skills add 생태계와 포터빌리티

`npx skills add` CLI는 Vercel Labs의 `agent-skills` 패키지를 기반으로 한다.
GitHub 레포지토리의 `skills/` 폴더를 스캔해 SKILL.md 파일을 프로젝트에 복사한다.
SKILL.md는 에이전트가 자동으로 로드할 수 있는 포터블 지시 파일이다.

이 설계는 스킬의 배포 단위를 GitHub 레포지토리로 만든다.
버전 관리, 포크, 기여가 기존 Git 워크플로로 이루어진다.
설치 이름(`name:` 필드)이 폴더 이름과 다를 수 있어
레포지토리 구조 변경 없이 스킬을 재조직할 수 있다.

## 비평

### 강점: 금지 목록의 구체성이 실제로 작동한다

대부분의 AI 품질 개선 시도는 추상적 원칙을 나열한다.
“더 창의적으로”, “기본값을 피해라”, “좋은 디자인을 해라”.
Taste Skill의 접근법은 반대다. Em-대시 금지, 섹션 번호 눈썹 금지,
스크롤 큐 금지처럼 에이전트가 기계적으로 확인할 수 있는 규칙을 만든다.
이 구체성이 실제 효과를 낸다. 에이전트는 “좋은 디자인”이 무엇인지
추론할 필요 없이 체크리스트를 따르면 된다.

### 한계: 스킬 파일 크기와 컨텍스트 소비

v2 SKILL.md는 1,200줄이 넘는다. 에이전트 컨텍스트 창에서 이 크기가
실제 코드 생성에 쓸 수 있는 공간을 줄인다. 특히 긴 코드베이스를 다루거나
복잡한 컴포넌트를 생성할 때 컨텍스트 압박이 될 수 있다.
스킬이 유용하려면 에이전트가 스킬 내용을 이해한 후에도
실제 구현을 위한 컨텍스트가 충분히 남아야 한다.

### 보완이 필요한 관점: 접근성과 반응형 설계 규칙의 상대적 부재

v2에는 접근성 관련 규칙이 “공공 부문 / 접근성 최우선” 브리프에 대한
다이얼 값 조정(`VARIANCE: 3-4, MOTION: 2-3`) 수준으로만 등장한다.
버튼 대비 체크(WCAG AA), reduced motion 요구사항 등 일부 규칙이 있지만,
시맨틱 마크업, ARIA 레이블, 키보드 내비게이션, 색상 대비 체계에 대한
포괄적 지침은 없다. 프리미엄 시각 디자인과 접근성의 교차점에서
더 강한 규칙이 필요하다.

## 인사이트

### “슬롭”의 명명이 문제를 해결하는 첫 번째 단계다

AI가 생성하는 뻔한 UI에 “슬롭”이라는 이름을 붙인 것은 사소해 보이지만 중요하다.
이름이 없는 문제는 논의하기 어렵다. “AI가 만든 UI는 왜 이렇게 비슷하게 보이지?”라는
막연한 불만이 “슬롭을 어떻게 피하는가”라는 구체적 질문으로 바뀐다.
이 명명이 커뮤니티 내 공통 어휘를 만들고, 문제 해결 방향에 대한 합의를 가능하게 한다.

소프트웨어 개발에서 이 패턴은 반복된다. “기술 부채”라는 표현이 생기기 전에는
코드 품질 저하를 경영진에게 설명하기 어려웠다. “10배 엔지니어”라는 개념이
생산성 격차에 대한 논의를 가능하게 했다(비록 그 개념 자체가 논쟁적이지만).
“바이브코딩(vibecoding)”이라는 표현이 대략적인 프롬프트로 소프트웨어를 만드는
새로운 개발 방식을 포착했다. Taste Skill 레포지토리의 토픽 태그에
`vibecoding`이 포함된 것은 이 연결을 의식한 포지셔닝이다.

AI 출력물에 대한 비판적 어휘가 발전하는 것은 AI 도구 성숙도의 신호다.
초기에는 “AI가 만들었다”는 사실 자체가 주목을 받았다. 이제는 AI 출력물의
품질 차이를 식별하고 명명하는 언어가 필요해졌다. 슬롭은 그 언어의 일부다.

### 에이전트 스킬은 새로운 형태의 소프트웨어 미들웨어다

Taste Skill이 SKILL.md 형식으로 배포되는 방식은 소프트웨어 배포의 새로운 층을 보여준다.
라이브러리는 코드를 배포한다. 패키지 매니저는 의존성을 관리한다.
에이전트 스킬은 에이전트의 행동 방식을 배포한다.
이것은 전례가 없는 소프트웨어 추상화 계층이다.

역사적 유추가 있다. 1990년대 IDE 스니펫이 반복 코드 패턴을 저장하고 공유했다.
2000년대 소프트웨어 패턴(GoF 디자인 패턴 등)이 설계 지식을 명문화했다.
2010년대 IaC(Infrastructure as Code)가 인프라 구성을 코드로 표현했다.
2020년대 에이전트 스킬은 에이전트의 판단 기준을 텍스트로 표현한다.
각 단계는 이전에 암묵적이었던 전문 지식을 명시적이고 공유 가능하게 만들었다.

이 새로운 미들웨어 계층이 가져오는 질문이 있다. 에이전트 스킬의 패키지 생태계가
npm이나 PyPI처럼 성장하면 어떻게 될까. 스킬 품질 보증, 보안 감사,
버전 호환성 문제가 등장할 것이다. `npx skills add`는 npm install과 구조적으로 동일하며,
그 생태계가 겪었던 문제들 — 의존성 지옥, 악의적 패키지, 네임스페이스 충돌 —
을 반복할 가능성이 있다.

### 금지 목록의 진짜 가치는 트렌드의 과포화를 포착한다는 것이다

Em-대시, 글라스모피즘, 섹션 번호 눈썹 같은 구체적 금지 항목들은
단순한 미적 규칙이 아니다. 이것들은 특정 시점에 특정 디자인 트렌드가
AI 생성 UI에서 과도하게 반복됐다는 관찰의 기록이다.
트렌드가 과포화되면 그것을 사용하는 것만으로 “AI가 만든 것”임을 드러낸다.

이 역설은 디자인 역사에서 반복된다. 1990년대 웹의 레인보우 규칙 텍스트,
2000년대의 플래시 인트로, 2010년대의 스큐어모피즘 텍스처,
2015년 이후의 플랫 디자인 카드 레이아웃. 각 트렌드는 차별화 수단으로 시작해
포화점에서 오히려 평범함의 신호가 됐다. 글라스모피즘은 2021년 새로웠고,
2024년에는 AI 생성 UI의 표식이 됐다.

Taste Skill의 금지 목록은 이 포화 시점을 명문화한다. 금지 목록이 정기적으로
갱신되어야 한다는 것이 함의다. 오늘의 “안티-슬롭” 규칙을 따른 결과물이
내년의 새로운 슬롭이 될 수 있다. 다이얼과 브리프 추론이 구체적 금지 목록보다
더 오래 유효할 이유가 여기 있다. 원칙은 트렌드보다 오래 지속된다.

### AI 생성 코드의 품질 보증은 별도의 전문 도메인이 된다

Taste Skill이 존재한다는 사실 자체가 AI 코딩 생태계의 성숙을 보여준다.
에이전트가 코드를 생성할 수 있게 됐지만, 그 코드의 품질을 보장하는 것은
별도의 문제임이 드러났다. 이 공백을 채우기 위해 에이전트 스킬이라는 새로운
소프트웨어 범주가 등장했다. 스킬은 에이전트의 생성 과정에 개입해
출력 품질을 높이는 메타 레이어다.

이 패턴은 소프트웨어 품질 보증의 역사와 유사하다.
컴파일러가 코드를 기계어로 변환할 수 있게 됐을 때,
링커, 디버거, 프로파일러라는 별도 도구가 그 품질을 보장하기 위해 등장했다.
IDEが 코드 작성을 보조할 때, 린터, 포매터, 정적 분석기가 품질 보증을 담당했다.
AI 에이전트가 코드를 생성할 때, 에이전트 스킬이 그 역할을 맡는다.
각 전환점마다 생성 능력과 품질 보증 능력은 별도로 발전했다.

Taste Skill은 프론트엔드 UI 품질에 집중하지만, 같은 원리가 다른 도메인에도 적용된다.
보안 코딩 스킬, 성능 최적화 스킬, 접근성 스킬, API 설계 스킬이
동일한 SKILL.md 형식으로 배포될 수 있다. 에이전트 스킬 생태계는
AI 생성 코드의 품질을 특정 도메인에서 체계적으로 높이는
새로운 전문 분야로 자리잡고 있다.
