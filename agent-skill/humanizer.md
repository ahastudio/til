# Humanizer: AI 글 흔적 제거 오케스트레이터 스킬

<https://github.com/ujuc/agent-stuff/blob/main/claude/skills/humanizer/SKILL.md>

## 소개

Humanizer는 AI가 생성한 한국어/영어 텍스트의 흔적을 찾아 자연스러운 사람의 글로
바꾸는 Claude Code 스킬이다.
`/humanizer` 명령으로 실행하며 한국어 주력, 영어 부수 지원이다.
원본 저장소는 [`epoko77-ai/im-not-ai`](https://github.com/epoko77-ai/im-not-ai) v1.5의
`humanize-korean` 스킬에서 MIT 라이선스로 가져와 개선한 것이다.

핵심 설계는 듀얼 트랙이다.
**Fast 모드(디폴트)**는 `humanize-monolith` 에이전트를 단일 호출해 탐지→윤문→자체검증을
한 번에 처리한다.
5,000자 이하를 2~3분에 완료하는 것이 목표다.
**Strict 모드(`--strict`)**는 4인 파이프라인으로 동작한다.
탐지(humanize-detector) → 윤문(humanize-rewriter) → 의미 감사(humanize-fidelity-auditor)
+ 자연성 리뷰(humanize-naturalness-reviewer) 병렬 검증 구조다.
8,000자 이상이면 Strict 모드로 자동 승급되며 한국어 전용이다.
**Redo 모드**는 이전 실행 결과에서 특정 카테고리나 문단만 재윤문할 때 사용한다.

운영 안전망은 세 가지다.
**변경률 가드**—레벤슈타인 거리 기준 30% 초과 시 경고, 50% 초과 시 강제 중단 후
마지막 안전 버전으로 롤백(`over_polish_aborted: true`).
**Do-NOT list**—고유명사·제품명·수치·날짜·직접 인용문·법률 조문·수식·업계 약어는
탐지와 윤문에서 제외.
**자체검증 6항**—고유명사/수치 보존, 변경률 30% 이하, 장르 이탈 없음, register 보존,
S1/P1 패턴 잔존 0건, 임의 비유 추가 금지—위반 시 해당 edit 롤백 후 재시도.

등급은 A~D로 자동 채점된다.
A는 S1/P1 잔존 0, S2/P2 잔존 2 이하, 변경률 10~25%, 자체검증 6/6 조건을 만족한다.
B 이하면 Strict 모드 권고 안내가 출력된다.

콘텐츠 유형 매트릭스는 장르별로 적용 기준을 다르게 한다.
블로그/에세이·SNS는 개성과 1인칭을 적극 권장하는 “숨결 주입”을 허용하지만,
기술 문서·학술/리포트·코드 주석은 정확성과 건조한 톤을 유지하며 개성 주입을 금지한다.

## 분석

### 4인 파이프라인의 관심사 분리

Strict 모드의 4인 파이프라인은 소프트웨어 설계의 관심사 분리(Separation of Concerns)를
멀티에이전트로 구현한 사례다.
탐지(what to change), 윤문(how to change), 의미 감사(did meaning survive), 자연성 리뷰
(does it sound human)—이 네 역할은 서로 다른 판단 기준을 가진다.
단일 에이전트가 이 모든 판단을 한 번에 처리하면, 어느 하나에 집중할수록 다른 기준이
희생될 수 있다.
역할을 분리하면 각 에이전트가 하나의 기준에 집중할 수 있고, 서로 독립적인 검증이
가능해진다.

### 변경률 가드의 공학적 의미

30%/50% 변경률 가드는 “과윤문(over-polishing)”이라는 개념을 수치로 정의한 것이다.
원문을 너무 많이 바꾸면 본래 의미나 저자의 목소리가 사라진다.
레벤슈타인 거리를 원문 길이로 나눈 비율을 임계값으로 삼는 것은 단순하지만 효과적이다.
50% 초과 시 강제 중단이라는 하드 리밋은, AI가 “더 좋게 고치려다” 원문을 해치는
시나리오를 방어한다.

### 콘텐츠 유형 매트릭스의 실용성

장르별로 다른 기준을 명시적으로 표로 정리한 것은 스킬 설계에서 중요한 결정이다.
블로그에 기술 문서 수준의 건조함을 강요하거나, 학술 보고서에 개성 있는 문체를 주입하면
둘 다 실패다.
“숨결 주입”이 허용되는 장르와 금지되는 장르를 명시함으로써, 에이전트가 장르를
자동 추정한 결과가 틀렸을 때 사용자가 수동으로 오버라이드할 수 있다.
장르 추정 실패를 `AskUserQuestion`으로 확인하는 fallback도 설계되어 있다.

### Do-NOT list의 설계 철학

Do-NOT list가 존재한다는 것은 “모든 것을 바꾸라”가 아니라 “특정 것들은 절대 바꾸지 말라”는
경계를 명시하는 방식이다.
고유명사와 수치는 사실 관계에 해당하고, 직접 인용은 타인의 말이며, 법률 조문은 정확한
문구가 법적 효력을 가진다.
이것들을 AI 티 제거 대상에서 제외하는 것은 단순히 텍스트 처리가 아니라 문서의 법적·사실적
무결성을 보호하는 선택이다.

## 비평

### 강점: 단계적 복잡도와 명확한 에스컬레이션 경로

Fast → Strict → Redo의 세 트랙은 사용자가 원하는 품질 수준과 기다릴 수 있는 시간을
선택할 수 있게 한다.
“8,000자 초과 시 자동 Strict 승급” 규칙은 사용자가 모드를 알지 못해도 합리적인 기본
동작을 제공한다.
등급 B 이하일 때 Strict 모드를 안내하는 것도 자연스러운 에스컬레이션 경로다.

### 한계: 영어 지원의 비대칭성

Strict 모드가 한국어 전용이라는 제약은 영어 사용자에게 명확한 차별이다.
fast 트랙에서 영어를 직접 처리하면서 패턴 카탈로그(`patterns-en.md`)는 존재하지만,
4인 파이프라인의 정밀 검증은 영어에서 사용할 수 없다.
영어 텍스트에 대한 깊이 있는 검증 수요가 있다면, 이 비대칭성이 도구 선택을 제한한다.

### 보완이 필요한 관점: 자동 추정의 실패 모드

장르를 파일 경로나 본문 내용으로 자동 추정하는 로직은 실패할 수 있다.
`README.md`는 대개 기술 문서지만, 개인 저장소의 README는 에세이에 가까울 수 있다.
자동 추정이 틀렸을 때 결과 전체가 잘못된 방향으로 처리될 위험이 있다.
`AskUserQuestion` fallback이 있지만, “판단이 어려우면”이라는 조건이 언제 발동되는지의
기준이 모호하다.

## 인사이트

### “AI 티 제거”는 새로운 문체 편집 카테고리의 탄생이다

교정(proofreading)과 교열(copy-editing)은 수백 년 역사를 가진 전문 직업이다.
문법 교정, 사실 확인, 문체 일관성 유지가 그 핵심 업무였다.
Humanizer가 다루는 “AI 티 제거”는 그 위에 새로운 편집 카테고리가 생겼음을 뜻한다.
AI 생성 텍스트가 공통으로 보이는 패턴—과도한 균형, 지나친 구조화, 감정 없는 나열,
비슷한 문장 리듬—을 탐지하고 제거하는 작업은 기존 교열이 다루지 않았던 영역이다.

이 카테고리가 존재한다는 것 자체가 흥미롭다.
AI 글이 충분히 널리 퍼져서, “이것이 AI가 쓴 글 같다”는 패턴 인식이 독자들 사이에서
형성됐다는 의미이기 때문이다.
문체(style)라는 것은 원래 개인이나 집단의 반복적 표현 습관이 축적되어 인식 가능해지는
것이다.
AI가 쓴 글도 이제 그 “스타일”이 식별 가능한 수준이 되었다.
역설적으로, AI 글을 사람 글처럼 만들기 위해 AI 에이전트를 쓴다는 것은—AI가 자신의
흔적을 지우는 도구가 된다는 것은—기술의 자기참조적 순환이다.

### 멀티에이전트 파이프라인이 “동료 검토”를 구현하는 방식

Strict 모드의 4인 파이프라인은 소프트웨어 코드 리뷰와 구조적으로 유사하다.
코드를 쓴 사람(rewriter)과 코드를 리뷰하는 사람(fidelity-auditor, naturalness-reviewer)을
분리하는 것이 핵심이다.
같은 에이전트가 쓰고 검토하면 자신의 편향을 발견하기 어렵다.
별도의 에이전트가 독립적인 기준으로 검증하면 이 편향을 줄일 수 있다.

이것은 AI 에이전트 설계에서 “자기 자신을 검증하게 하지 말라”는 원칙의 구현이다.
의미 동등성(fidelity)과 자연성(naturalness)을 별개 에이전트에게 분리한 것도 주목할 만하다.
이 두 기준은 종종 상충한다.
의미를 완전히 보존하려면 원문에 가까운 표현을 써야 하고, 자연성을 높이려면 더 자유롭게
바꿔야 한다.
이 긴장을 두 에이전트가 각각 독립적으로 평가하고 최종 판정 테이블로 조율하는 구조는,
이 트레이드오프를 명시적으로 설계에 반영한 것이다.

### “숨결 주입”은 AI와 인간 글쓰기의 경계를 재정의한다

스킬 문서에서 가장 주목할 만한 섹션은 “글에 숨결 불어넣기”다.
“깨끗하지만 무미건조한 글도 AI처럼 보인다”는 문장이 핵심이다.
AI 티 제거가 단순히 특정 패턴을 없애는 것이 아니라, 인간의 불완전함과 개성을 능동적으로
주입하는 것까지 포함한다는 선언이다.

이 관점은 “인간다운 글”의 본질에 대한 흥미로운 주장을 내포한다.
“완벽한 구조는 알고리즘 냄새가 난다”, “곁가지, 여담, 반쯤 정리된 생각은 사람의 것이다”—
이것들은 글쓰기에서 “불완전함”이 신뢰의 신호라는 명제다.
독자는 무의식적으로 너무 완벽한 구조, 너무 균형 잡힌 논거, 너무 적절한 문장 리듬에서
인공적임을 감지한다.
인간은 산만하고, 주제를 벗어나고, 감정적이다.
그것이 글에서 진정성의 시그널로 작동한다.

역설적으로, 이 인사이트는 AI가 더 나은 글을 쓰려면 “의도적으로 덜 완벽해야 한다”는
결론으로 이어진다.
Humanizer가 수행하는 작업은 AI가 학습한 “최적화된 글쓰기”를 비최적화하는 것이다.
이것은 단순한 스타일 변환이 아니라, AI와 인간 글쓰기의 경계가 어디에 있는가에 대한
철학적 질문에 실용적 도구로 답하는 시도다.
