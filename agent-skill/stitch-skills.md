# stitch-skills — Stitch를 위한 에이전트 스킬 모음

<https://github.com/google-labs-code/stitch-skills>

## 소개

Google Labs가 공개한 저장소로, AI 기반 디자인 플랫폼인 Stitch를 코딩
에이전트가 다루도록 만든 스킬(skill)과 플러그인(plugin) 모음이다.
Agent Skills라는 공개 표준을 따르며, Claude Code, Cursor, Codex,
Gemini CLI 같은 여러 코딩 에이전트에 설치해 쓸 수 있도록 설계됐다.

스킬은 `stitch-design`, `stitch-build`, `stitch-utilities` 세 개의
플러그인으로 묶여 있고, 각 플러그인 아래에 개별 스킬이 폴더 단위로
들어 있다. 스킬 하나는 `SKILL.md`(미션 문서), `scripts/`(실행
스크립트), `resources/`(참고 자료), `examples/`(예시 구현)로
구성된다. 동작을 위해서는 에이전트 환경에 Stitch MCP 서버가 설정되어
있어야 한다.

## 주요 기능

### stitch-design — 디자인 생성과 관리

- `code-to-design`: 프런트엔드 코드를 HTML로 추출해 Stitch 디자인으로
  변환한다.
- `generate-design`: 텍스트나 이미지로부터 새 화면을 만들거나 기존
  디자인을 수정한다.
- `manage-design-system`: 디자인 시스템을 다루고 `DESIGN.md` 파일을
  업로드한다.
- `extract-design-md`: 소스 코드에서 포괄적인 `DESIGN.md`를 추출한다.
- `extract-static-html`: 독립 실행 가능한 정적 HTML 스냅샷을 만든다.
- `upload-to-stitch`: 로컬 자산을 Stitch 프로젝트에 업로드한다.

### stitch-build — 디자인을 코드로 변환

- `react-components`: Stitch 화면을 React 컴포넌트 체계로 변환한다.
- `react-native`: HTML 디자인을 React Native 컴포넌트로 옮긴다.
- `remotion`: Stitch 프로젝트로부터 워크스루(walkthrough) 영상을
  생성한다.
- `shadcn-ui`: shadcn/ui 애플리케이션과의 통합을 안내한다.
- `react-vite-dashboard`: 디자인 토큰을 반영한 React + Vite 대시보드를
  만든다.

### stitch-utilities — 품질과 확장

- `design-md`: 프로젝트를 분석해 의미 기반 `DESIGN.md`를 생성한다.
- `enhance-prompt`: 모호한 UI 아이디어를 Stitch에 최적화된 프롬프트로
  다듬는다.
- `stitch-loop`: 단일 프롬프트로 여러 페이지의 사이트를 생성한다.
- `taste-design`: 품질 기준을 강제하는 프리미엄 `DESIGN.md`를
  만든다.

설치는 에이전트마다 다르다.
Claude Code와 Cursor는 `npx plugins add
google-labs-code/stitch-skills --scope project --target claude-code`
같은 npm 기반 명령으로 플러그인 전체를 설치하고,
Codex는 `codex plugin marketplace add`에 `--sparse` 옵션으로 원하는
플러그인 경로만 골라 설치한다.
개별 스킬만 필요하면 `npx skills add
google-labs-code/stitch-skills`로 선택적으로 추가할 수 있다.

## 분석

### 세 플러그인 분리는 파이프라인의 관심사 분리를 그대로 반영한다

`stitch-design`, `stitch-build`, `stitch-utilities`라는 세 갈래 구성은
우연이 아니라 디자인-투-코드(design-to-code) 파이프라인의 자연스러운
단계 구분을 그대로 옮긴 것이다.
디자인을 만들고 관리하는 단계, 그것을 실제 프레임워크 코드로 변환하는
단계, 그리고 품질을 높이거나 반복 작업을 자동화하는 단계는 서로 다른
목적과 다른 실패 양상을 가진다.
이 구분을 플러그인 경계로 못박아 두면, 에이전트는 “지금 필요한 게
디자인 단계인가 빌드 단계인가”를 먼저 판별한 뒤 해당 스킬 집합만
불러오면 된다.

이런 구조는 컨텍스트 관리 측면에서도 이점이 있다.
에이전트가 매번 15개 스킬 전부를 읽어야 한다면 프롬프트 컨텍스트
낭비가 크지만, 플러그인 단위로 필요한 것만 설치하거나 로드하면 실제
작업에 관련된 지식만 에이전트에 전달된다.
`--sparse` 옵션으로 플러그인별 경로만 골라 설치하는 Codex 설치
방식은 이 관심사 분리를 설치 계층까지 끌어올린 사례다.

### MCP 서버 의존은 스킬을 도구 호출의 얇은 래퍼로 만든다

이 저장소의 스킬들은 로직 대부분을 자체적으로 구현하지 않는다.
`SKILL.md`는 절차와 판단 기준을 제공하지만, 실제 디자인 생성이나 코드
변환은 Stitch MCP 서버가 수행한다.
즉 스킬은 “언제, 어떤 순서로, 어떤 파라미터로 MCP 도구를 호출할
것인가”를 정의하는 오케스트레이션 계층이지, 독립적으로 동작하는
프로그램이 아니다.

이 구조는 스킬 생태계에서 흔히 나타나는 두 갈래 패턴 중 하나를
보여준다.
한쪽은 스크립트와 검증 로직을 스킬 자체에 담아 어떤 백엔드 없이도
동작하는 자기완결형 스킬이고, 다른 한쪽은 이 저장소처럼 특정 외부
서비스(MCP 서버)에 강하게 결합된 스킬이다.
후자는 설치와 사용이 간단해 보이지만, Stitch라는 특정 제품이 없으면
스킬 전체가 무용지물이 된다는 대가를 치른다.

### 빌드 타깃의 다양성은 실사용 시나리오를 반영하지만 유지보수 부담을 키운다

`stitch-build` 아래에는 React, React Native, Remotion, shadcn/ui,
Vite 대시보드까지 다섯 개의 서로 다른 빌드 타깃이 있다.
이는 디자인 하나가 웹, 모바일, 영상, 특정 UI 라이브러리 등 다양한
산출물로 이어질 수 있다는 실제 사용 요구를 반영한 것으로 보인다.

그러나 타깃이 늘어날수록 각 타깃의 프레임워크 버전이 바뀔 때마다
스킬을 동기화해야 하는 부담도 함께 커진다.
React Native나 shadcn/ui처럼 자체적으로 빠르게 변화하는 생태계를
대상으로 하는 스킬은, 저장소 관리자가 지속적으로 최신 상태를
따라가지 않으면 금방 낡은 안내를 제공하게 된다.
이는 이 저장소만의 문제가 아니라 특정 프레임워크를 대상으로 하는
모든 코드 생성 도구가 공유하는 구조적 부담이다.

### 품질 강제를 스킬로 표현하려는 시도가 흥미롭다

`taste-design`처럼 “프리미엄 디자인 표준을 강제하는” 스킬을 별도로
둔 것은 눈여겨볼 만하다.
지금까지 디자인 품질은 대체로 디자이너의 안목이나 사내 스타일
가이드처럼 암묵적인 형태로 존재했다.
이를 `DESIGN.md`라는 명시적 문서로 만들고, 그 문서를 생성하는 절차를
스킬로 표준화하려는 시도는 품질 기준을 코드화하려는 흐름의 연장선에
있다.

다만 “프리미엄”이라는 기준 자체가 정성적이라는 점에서, 스킬이 실제로
얼마나 일관된 결과를 만들어내는지는 README만으로는 검증할 수 없다.
품질 기준을 문서화한다고 해서 그 기준이 자동으로 검증 가능해지는
것은 아니다.

## 비평

### Stitch에 대한 종속이 스킬의 범용성을 크게 제한한다

이 저장소가 “Agent Skills 공개 표준을 따른다”고 강조하지만, 실제로는
Stitch MCP 서버 없이는 어떤 스킬도 의미 있게 동작하지 않는다.
표준화된 포맷을 쓴다는 것과 표준화된 방식으로 재사용 가능하다는 것은
다른 이야기다.
다른 에이전트 스킬 저장소들이 로컬 파일 조작이나 범용 워크플로 관리처럼
특정 제품에 묶이지 않는 절차를 제공하는 것과 달리, 이 저장소는 사실상
“Stitch 사용 설명서를 에이전트가 읽을 수 있는 형태로 바꾼 것”에
가깝다.
Stitch를 쓰지 않는 사용자에게는 이 저장소가 통째로 무의미하다.

이런 구조에서 “여러 에이전트를 지원한다”는 것도 실제로는 “여러
에이전트에서 같은 제품 하나를 조작할 수 있다”는 의미로 좁게 읽어야
한다.
Agent Skills 표준을 따른다는 서술이 독자에게 범용성을 과대하게
암시할 여지가 있다.

### 설치 명령의 파편화가 표준화의 취지를 갉아먹는다

Codex, Claude Code, Cursor, Gemini CLI 각각에 대해 서로 다른 설치
명령과 옵션 체계가 존재한다는 점은, “여러 에이전트에서 동작하는
표준 스킬”이라는 주장과 긴장 관계에 있다.
Codex는 `--sparse` 플래그로 세부 경로를 지정해야 하고, Claude
Code와 Cursor는 `npx plugins add`에 `--target` 옵션을 붙이는 방식이
다르며, Gemini CLI는 호환된다고만 언급될 뿐 구체적인 설치 절차가
README에 나오지 않는다.

표준이 진짜 표준으로 기능하려면 최소한 설치 경험 정도는 에이전트에
무관하게 통일되어야 한다.
지금처럼 에이전트마다 다른 CLI 문법을 학습해야 하는 상태는, 스킬
포맷 자체는 표준화됐어도 그 스킬을 얻는 경로는 여전히 파편화되어
있다는 뜻이다.
이는 스킬 생태계 초기 단계에서 흔히 나타나는 문제이지만, “표준”이라는
단어를 전면에 내세우는 저장소라면 이 간극을 더 적극적으로 좁혔어야
한다.

### 15개 스킬로의 세분화가 실제로 필요한지 의문이다

디자인, 빌드, 유틸리티로 나누고 그 안에서 다시 15개 스킬로 쪼갠
구조는 표면적으로는 관심사 분리처럼 보이지만, 그중 상당수는 하나의
스킬로 통합해도 무방해 보인다.
예를 들어 `extract-design-md`와 `design-md`는 이름과 목적이 거의
겹치고, `react-components`와 `react-vite-dashboard`도 결국은 React
코드를 생성한다는 점에서 공통 기반을 공유한다.

세분화 자체는 나쁘지 않지만, 세분화가 지나치면 사용자가 “이 작업에는
정확히 어떤 스킬을 써야 하는가”를 판단하는 데 드는 인지 비용이
스킬을 직접 쓰는 것보다 커질 수 있다.
스킬 개수가 늘어날수록 에이전트가 잘못된 스킬을 고르거나, 여러 스킬을
어떤 순서로 조합해야 하는지 헷갈릴 위험도 함께 늘어난다.
README는 이 조합 순서에 대한 명확한 가이드를 제공하지 않는다.

### 검증 스크립트의 실체가 README만으로는 드러나지 않는다

각 스킬 폴더에 `scripts/`가 있다고 설명하지만, 그 스크립트가 실제로
무엇을 검증하는지, 실패했을 때 에이전트가 어떻게 대응해야 하는지는
README 수준에서 확인할 수 없다.
디자인 시스템이나 생성된 코드의 품질을 자동으로 검증하는 장치가
있는지, 아니면 단순히 MCP 호출을 감싸는 얇은 래퍼에 불과한지가
불분명하다.

에이전트 스킬에서 검증 스크립트의 존재는 종종 신뢰성의 상징처럼
제시되지만, 그 스크립트가 실제로 무엇을 보장하는지 문서화되지 않으면
사용자는 검증이 이뤄지고 있다는 인상만 받을 뿐 실질적인 보증은 얻지
못한다.

## 인사이트

### 디자인 툴 벤더가 스킬 배포를 직접 떠맡는 흐름이 시작됐다

지금까지 에이전트 스킬은 대부분 개인이나 커뮤니티가 특정 워크플로를
정리해 공유하는 형태로 등장했다.
`google-labs-code/stitch-skills`는 이와 달리 제품을 만든 회사(Google
Labs)가 자사 제품을 위한 공식 스킬 세트를 직접 배포한 사례다.
이는 마치 클라우드 서비스 초기에 서드파티가 만들던 SDK를 벤더가
직접 만들어 배포하기 시작한 전환과 유사하다.

벤더가 스킬을 직접 만들면 품질과 최신성은 보장되기 쉽지만, 동시에
그 제품에 사용자를 더 강하게 묶어 두는 효과도 생긴다.
에이전트를 통한 조작이 표준화될수록 사용자는 오히려 그 표준화된
경험을 제공하는 벤더의 제품을 벗어나기 어려워질 수 있다.
스킬이 사용자 편의를 위한 것이면서 동시에 벤더 록인(lock-in)의 새
경로가 될 수 있다는 이중적 성격은, 앞으로 다른 SaaS 벤더들이 비슷한
스킬 저장소를 내놓을 때도 반복될 가능성이 높다.

### MCP와 스킬의 역할 분담이 에이전트 아키텍처의 새로운 관례로 자리잡고 있다

이 저장소는 “MCP 서버가 실제 기능을 제공하고, 스킬은 그 기능을 언제
어떻게 쓸지 안내하는 절차서 역할을 한다”는 역할 분담을 명확히
보여준다.
MCP가 등장하기 전에는 에이전트가 도구를 어떻게 쓸지에 대한 지식이
프롬프트나 코드에 흩어져 있었지만, 이제는 “도구(MCP)”와
“사용법(스킬)”이 분리된 두 개의 배포 단위로 나뉘는 패턴이 자리잡고
있다.

이 분리는 소프트웨어 공학에서 API와 API 문서가 분리되어 왔던 것과
유사한 구조다.
다만 전통적인 API 문서는 사람이 읽고 해석했지만, 스킬은 에이전트가
직접 읽고 실행 절차로 전환한다는 점에서 문서의 역할이 실행
가능한 아티팩트로 격상된다.
이 역할 분담이 정착되면, 앞으로 나오는 모든 MCP 서버가 자체 스킬
세트를 함께 배포하는 것이 사실상의 관례가 될 가능성이 있다.

### 스킬 세분화와 조합 가능성 사이의 트레이드오프는 아직 풀리지 않았다

15개 스킬로 잘게 나눈 이 저장소의 구조는 “각 스킬을 작고 단일
책임으로 유지하면 재사용성과 유지보수성이 높아진다”는 소프트웨어
공학의 오래된 원칙을 그대로 따른다.
그러나 그 원칙이 에이전트 스킬에도 그대로 적용되는지는 분명하지
않다.
함수나 모듈은 컴파일러나 런타임이 자동으로 조합하지만, 스킬은
에이전트가 자연어 맥락에서 “지금 어떤 스킬을, 어떤 순서로 써야
하는가”를 스스로 추론해야 한다.

세분화가 지나치면 조합의 부담이 에이전트에게 전가되고, 반대로
스킬을 크게 묶으면 하나의 스킬 안에 여러 책임이 뒤섞여 유지보수가
어려워진다.
이 저장소는 세분화 쪽으로 크게 기울어 있는데, 이는 README에 조합
가이드가 부족하다는 앞선 비평과 맞물려 있다.
스킬 생태계 전체가 아직 이 최적점을 찾지 못했다는 점에서, 이
저장소의 선택은 하나의 실험적 데이터 포인트로 볼 수 있다.
