# SkillOpt - AI 에이전트의 스킬도 학습시킬 수 있을까

원문: <https://javaexpert.tistory.com/1765>

## 요약

이 글은 Microsoft가 공개한 SkillOpt 프레임워크를 소개한다. SkillOpt는
모델 자체를 미세 조정(fine-tuning)하는 도구가 아니라, AI 에이전트가
반복 작업에서 참고하는 지침 문서(`skill.md`)를 실행 결과와 실패 로그에
근거해 자동으로 개선하는 시스템이다. 더 많은 지침을 프롬프트에 욱여넣어
길이 증가, 내용 중복, 규칙 충돌을 일으키는 기존 방식을 대체하려는
시도다.

저자는 프롬프트와 스킬을 분리해서 정의한다. 프롬프트는 한 번의 요청을
위한 일회성 지시문이고, 스킬은 같은 종류의 작업을 반복적으로 수행하기
위해 보존되는 상시 지침서다. “코드를 수정할 때는 먼저 관련 파일 구조를
파악하라” 같은 표준 절차는 매 대화마다 다시 적을 것이 아니라 스킬
문서로 외부화되어야 한다는 관점이다.

SkillOpt의 작동 사이클은 네 단계로 정리된다. 현재 스킬로 여러 작업을
실행하고, 성공·실패 로그를 수집해 패턴을 분석하고, 분석 결과에 따라
`skill.md` 수정안을 생성한 뒤, 검증 데이터셋에서 성능이 개선되었음이
확인된 버전만 새 스킬로 채택한다. 실패를 채팅 기록 속으로 흘려보내지
않고 자산으로 적립한다는 점이 핵심이다.

활용이 잘 맞는 영역으로 개발 에이전트(테스트 통과, 타입 체크, 린팅처럼
명확한 합격 기준이 있는 작업), 영상 프롬프트 에이전트(시간 분할, 카메라
움직임, 캐릭터 일관성), 교육 콘텐츠 에이전트(어휘 난이도, 연령 적합성,
학습자가 자주 저지르는 오류)가 제시된다. 다만 SkillOpt는 설치만으로
자동 개선이 보장되는 도구가 아니며, 학습 데이터셋·검증 데이터셋·평가
기준의 사전 설계가 전제 조건임을 함께 강조한다.

저자가 글 말미에 도달하는 결론은 명확하다. 앞으로 에이전트를 잘 쓰는
사람은 “질문을 잘하는 사람”이 아니라 “자신의 작업 방식과 실패 패턴을
스킬로 정리해 계속 개선하는 사람”이라는 것이다.

## 분석

### 모델 학습이 아니라 지침 학습이라는 위치 설정

SkillOpt는 의도적으로 가중치(weight) 학습이 아닌 문서(document)
학습이라는 자리를 차지한다. RLHF나 파인튜닝은 비용·시간·데이터 요구치가
크고 일반 팀이 직접 돌리기 어렵다. 반면 스킬 문서는 텍스트이므로
누구나 편집할 수 있고, 버전 관리·diff 검토·코드 리뷰의 대상이 된다.
모델 개선이 아닌 지침 개선을 자동화하면 동일한 모델을 쓰면서도
조직 단위로 누적되는 품질 자산이 생긴다는 발상이다.

### 평가-검증 루프의 이식

execute → reflect → modify → validate라는 사이클은 머신러닝의 학습
루프와 소프트웨어 엔지니어링의 회귀 테스트가 결합된 형태다. 모델 파라미터
대신 문서 텍스트가, 손실 함수 대신 작업 성공률이, 검증 세트 대신
홀드아웃 작업 묶음이 자리를 차지한다. 이는 [[skillify]]가 제시한
SKILL.md + 결정론적 스크립트 + 자동화 테스트 조합과 같은 가족에 속하며,
“에이전트 품질을 통제 가능한 변수로 보자”는 흐름의 연장선이다.

### 평가 기준의 비대칭성

저자가 든 세 영역(개발, 영상, 교육)은 평가 가능성이 동일하지 않다.
개발은 테스트가 즉시 합격·불합격 신호를 주는 반면, 영상 프롬프트의
“카메라 움직임이 자연스럽다”는 사람의 주관 판정을 요구한다.
SkillOpt가 후자에서도 작동하려면 사람 평가를 대신할 LLM-as-judge나
스코어 함수가 필요하고, 이 평가기 자체가 또 다른 학습 대상이 된다.
즉 SkillOpt를 적용한다는 것은 곧 평가 인프라를 함께 구축한다는 뜻이다.

## 비평

### 강점 - 실패를 일회성에서 누적성으로

기존 프롬프트 엔지니어링의 가장 큰 약점은 실패가 대화 안에 갇혀
사라진다는 점이다. 같은 실수가 다음 세션, 다른 사용자, 다른 팀에서
반복된다. SkillOpt는 실패 로그를 분석 대상으로 격상시켜 다음 버전의
스킬에 반영하므로, 한 번 학습된 교훈이 조직 전체로 전파된다. 이는
포스트모템 문화의 자동화에 가깝다.

### 약점 - 비용과 데이터 요구치를 과소평가하기 쉽다

원문은 “평가 기준이 명확하면 잘 동작한다”는 식으로 정리하지만,
검증 데이터셋을 충분한 다양성과 양으로 확보하는 일 자체가 큰 비용이다.
스킬 한 줄을 바꿔도 통계적으로 유의미한 개선인지 판정하려면 작업
수십~수백 개가 필요하고, 작업 자체에 LLM 호출이 들어가면 비용이
빠르게 누적된다. “자동 개선”이라는 표현이 주는 인상보다 실제 운영
비용은 상당히 크다.

### 약점 - 메타 과적합 위험

SkillOpt가 검증 데이터셋의 점수를 올리는 방향으로 스킬을 다듬다 보면,
검증 작업의 분포에는 잘 맞지만 실제 사용 분포에는 잘 맞지 않는 스킬이
선택될 수 있다. 머신러닝의 검증 세트 과적합과 동일한 문제다. 스킬은
인간이 읽고 판단하는 텍스트라는 점에서 사람에 의한 정합성 점검이 더
중요해지지만, “자동 개선”이라는 프레이밍은 그 점검을 생략하게 만들
수 있다.

### 약점 - 스킬 간 충돌과 의존성

스킬이 늘어나면 스킬 사이의 충돌, 중복, 우선순위 문제가 발생한다.
SkillOpt가 단일 스킬을 개선하는 데 집중한다면, 시스템 전체의
일관성은 별도의 관리 메커니즘이 필요하다. 컴파일러의 타입 검사나
의존성 관리자처럼 스킬 그래프 차원의 검증 도구가 따라붙지 않으면,
개별 스킬은 좋아지는데 전체 에이전트는 더 혼란스러워지는 역설이
생긴다.

## 인사이트

### 가중치는 못 바꿔도 지침은 바꿀 수 있다는 비대칭이 제품 전략을 정의한다

대부분의 팀은 프런티어 모델의 가중치에 접근할 수 없다. OpenAI, Anthropic,
Google의 모델을 API로 호출할 뿐이다. 이 비대칭은 “모델을 더 똑똑하게
만드는” 경쟁은 소수의 연구소만 할 수 있고, 나머지는 “이미 똑똑한 모델을
더 잘 쓰게 만드는” 경쟁만 할 수 있음을 의미한다.

SkillOpt는 이 두 번째 경쟁에 정확히 들어맞는 도구다. 모델은 그대로 두고
주변 텍스트 자산만 개선해 성능을 끌어올린다. 이는 [[agent-skills-for-context-engineering]]의
컨텍스트 엔지니어링 흐름, [[context-is-the-new-code]]의 “컨텍스트가 새 코드”
명제와 같은 결을 공유한다. 모델 사용자에게 남은 레버는 가중치가 아니라
컨텍스트와 스킬이다.

여기서 흥미로운 함의가 도출된다. 모델 격차가 벌어지는 동안에도 스킬
운영 격차는 모든 팀이 좁힐 수 있는 영역이다. 같은 GPT-5나 Claude를 쓰는
두 팀의 결과 품질이 갈리는 이유는 점점 더 “누가 더 좋은 스킬을 가지고
있느냐”로 환원된다. 모델 라이선스가 평준화될수록 스킬 자산이 진정한
해자가 된다.

### 스킬 문서는 형상 관리되는 자산이지 메모가 아니다

많은 팀이 “시스템 프롬프트”나 “규칙 문서”를 메모처럼 다룬다. 누군가 한
번 작성하고, 그 후에는 손대지 않거나 임시 패치를 누적한다. SkillOpt가
요구하는 것은 정반대다. 스킬은 디프(diff), 리뷰, 회귀 테스트, 롤백을
가진 형상 관리 대상이다. 이는 코드와 동일한 수명 주기를 가진다는 뜻이다.

이 관점이 자리 잡으면 “프롬프트 엔지니어”는 직무명으로 살아남기 어렵다.
한 번 잘 쓰는 사람보다 지속 개선 체계를 운영하는 사람이 필요하기
때문이다. 그 자리는 “스킬 큐레이터”, “에이전트 SRE”, “컨텍스트 메인테이너”
같은 직무명에 가깝다. 글쓰기 능력 못지않게 평가 설계, 데이터 파이프라인,
실험 관리 능력이 요구된다.

장기적으로는 스킬 저장소가 코드 저장소만큼이나 조직 자산이 된다.
git, GitHub, 코드 리뷰 도구가 코드 협업을 가능하게 했듯, 스킬을 위한
유사 인프라가 등장할 것이다. 이미 [[dont-build-agents-build-skills-instead]]에서
스킬 우선 접근이 제안되고 있고, SkillOpt는 그 자산을 자동으로 다듬는
계층을 추가한 셈이다.

### 자동 개선은 평가가 정의되는 만큼만 자동이다

“자동으로 개선된다”는 표현은 매혹적이지만, 자동화의 경계는 결국 평가
함수에서 끝난다. 합격 기준이 명확하지 않으면 SkillOpt는 무엇이 더 나은
스킬인지 판단할 수 없고, 개선 방향을 잡지 못한다. 다시 말해 SkillOpt를
도입한다는 결정은 평가 함수를 설계한다는 결정과 같다.

이는 머신러닝이 지난 10년간 겪어온 교훈의 재방문이다. 정확도 99%의
모델이 실제로 쓸 수 없는 경우가 흔했던 이유는 정확도라는 지표가
실제 가치를 대리하지 못했기 때문이다. 에이전트 스킬에서도 같은
함정이 반복된다. “테스트 통과율”이 “좋은 코드”와 같지 않고, “문법
정확성”이 “좋은 영상 프롬프트”와 같지 않다. 평가 함수가 가치와
어긋나면, SkillOpt는 잘못된 방향으로 빠르게 수렴할 수 있다.

따라서 SkillOpt 도입의 진짜 작업은 알고리즘이 아니라 메트릭 설계다.
이 단계를 건너뛴 팀은 “자동 개선되는 시스템”을 가졌다고 믿으면서
실제로는 “자동으로 잘못된 방향으로 굳어지는 시스템”을 갖게 된다.
평가가 코드보다 먼저 와야 한다는 [[llm evaluation behavioral gaps]] 류의
교훈은 스킬 자동화 시대에 더 무거워진다.

### 에이전트 품질의 책임이 모델 제공자에서 사용자에게로 이동한다

GPT-3 초기에는 모델 자체의 능력 부족이 결과 품질의 주된 원인이었다.
“모델이 더 좋아지면 해결된다”는 기대가 합리적이었다. 그러나 프런티어
모델이 성숙해질수록 동일 모델로도 결과 품질이 팀마다 크게 갈린다.
이 시점에서 책임 소재는 분명히 이동한다. 결과가 나쁘면 그것은 모델
탓이 아니라 컨텍스트 운영 탓이다.

SkillOpt는 이 책임 이동을 가시화한다. 더 이상 “모델이 못해서”라고 말할
수 없다. 같은 모델로 더 잘 쓰는 팀이 존재한다는 증거가 SkillOpt가
관리하는 스킬 로그에 매일 누적되기 때문이다. 모델 제공자의 마케팅
서사(“다음 버전은 X% 더 똑똑합니다”)와는 별개로, 사용자 측 운영
역량이 품질의 1차 결정 요인으로 부상한다.

이는 클라우드 인프라가 성숙했을 때 일어난 변화와 닮았다. AWS가 충분히
안정적이 된 이후, 서비스 장애의 책임은 “AWS 탓”에서 “우리 SRE 탓”으로
넘어갔다. 모델도 같은 경로를 걷는다. 모델이 충분히 똑똑해진 이후,
에이전트 결과의 책임은 모델 제공자에서 스킬 운영자에게로 넘어간다.
SkillOpt 같은 도구의 등장은 이 책임 이동을 기술적으로 뒷받침하는
신호로 읽힌다.
