# RGB 정규화: 255로 나눌 것인가, 256으로 나눌 것인가?

원문: <https://30fps.net/pages/255-vs-256-division/>

HN 토론: <https://news.ycombinator.com/item?id=48360054> (327점, 138개 댓글)

## 요약

8비트 RGB 색상값을 부동소수점으로 변환할 때 255로 나눌 것인가, 256으로 나눌 것인가라는 작지만 미묘한 질문을 다룬다.
대부분의 이미지 처리 코드는 `value / 255.0`을 사용하지만, `(value + 0.5) / 256.0`을 써야 한다는 주장도 있다.
어떤 방식이 더 “수학적으로 올바른가”를 다루는 글이다.

255 방식은 0을 0.0에, 255를 1.0에 정확히 대응시킨다.
GPU의 UNORM(Unsigned Normalized) 변환과 일치하며, 0과 1.0에서 정확한 극단값을 보장한다.
단, 이 방식은 균일한 분포에서 0과 255가 다른 값들보다 절반의 빈도로 나타나는 좁은 빈(bin)을 만든다.
신호 이론에서 이것은 “mid-tread” 균일 스칼라 양자화기에 해당한다.

256 방식은 `(value + 0.5) / 256.0`으로 각 값을 구간의 중간점에 배치한다.
0은 0.00195에, 255는 0.99805에 대응된다.
모든 빈의 너비가 동일하지만, 순수한 검정(0.0)과 순수한 흰색(1.0)을 직접 표현할 수 없다.
이것은 “mid-riser” 양자화기다.

권고는 명확하다.
외부에서 가져오는 이미지에는 255를 사용하라.
저장과 불러오기를 모두 통제하는 경우에만 256 방식을 고려하라.
실제로는 두 방식의 차이가 대부분의 이미지 처리 작업에서 시각적으로 구분되지 않는다.

## 분석

### 양자화 이론의 관점

이 문제는 더 넓게 보면 양자화 이론의 문제다.
아날로그 연속값을 디지털 이산값으로 변환하거나 역변환할 때 항상 발생하는 반올림과 오차의 문제다.
255 방식과 256 방식의 선택은 양자화기의 “격자점(grid)”을 어디에 놓느냐의 선택이다.
mid-tread는 격자점이 0에 정확히 놓이고, mid-riser는 격자점이 구간 중간에 놓인다.

ADC(아날로그-디지털 변환기) 관점에서도 같은 결론이 나온다.[^fps-hero]
ADC에는 항상 ±1/2 LSB의 양자화 불확실성이 존재하고, 전달 특성은 필연적으로 mid-tread 샘플링이다.
“N개 값, N-1개 구간”이라는 펜스포스트 오류의 관점에서 보면, 분모는 반드시 2^N − 1, 즉 255가 되어야 한다.

GPU의 UNORM 변환이 255 방식을 따른다는 점이 실용적으로 결정적이다.
텍스처 샘플링, 렌더링 파이프라인, 셰이더가 모두 이 기준을 따르기 때문에, 256 방식을 쓰면 GPU가 기대하는 값과 일관성이 깨진다.
이 비호환성의 비용이 256 방식의 이론적 우아함을 상쇄한다.
cbloom의 게임 양자화 분석도 같은 결론을 내린다.[^mwkaufma]
"이론적으로는 두 방식 모두 쓸 수 있는 경우가 있지만, 게임에서는 그렇게 하지 말라. GPU UNORM 색상 표준이 'centered' 양자화를 선택했으므로, 당신도 그것을 따라야 한다."

256 방식을 사용할 때 알파 채널이 포함된 합성 워크플로에서는 실질적인 아티팩트가 생긴다.[^pixelesque]
VFX 업계의 많은 합성·이미지 처리 워크플로는 `0.0`을 "완전 투명", `1.0`을 "완전 불투명"으로 하드코딩한다.
256 방식을 쓰면 255가 1.0 미만, 0이 0.0 초과로 매핑되어, 프리멀티플라이 과정에서 미세하게 반투명한 오브젝트가 생기거나 마스크 임계값이 어긋나는 문제가 발생한다.
이는 픽셀 단위로 확인하거나 특정 조명 조건에서만 드러나는 미묘한 버그다.

### 비트 심도 일반화의 중요성

255 방식의 또 다른 장점은 다른 비트 심도로 자연스럽게 일반화된다는 것이다.
16비트 색상이면 65535로, 12비트 HDR이면 4095로 나누면 된다.
수식이 `value / (2^n - 1)`로 통일된다.
256 방식은 비트 심도에 따라 오프셋을 조정해야 하므로 코드가 입력에 결합된다.
이것이 “저장과 불러오기를 모두 통제하는 경우에만” 256 방식을 고려하라는 이유다.

비트 심도 불변성 논증도 255를 지지한다.[^GuB-42]
1비트 이미지에서 0은 검정, 1은 흰색이므로 자연스럽게 0→0.0, 1→1.0으로 대응된다.
2비트라면 0, 1, 2, 3이 0.0, 0.33, 0.66, 1.0에 대응된다.
반면 256(mid-riser) 방식으로는 1비트 이미지가 [0.25, 0.75]에, 2비트가 [0.125, 0.875]에 매핑되어, 비트 심도가 낮아질수록 출력 범위가 줄어드는 비직관적인 결과가 나온다.
더 많은 비트가 더 섬세한 표현을 의미하되 대비 범위는 동일해야 한다는 기대를 위반하는 것이다.

## 비평

### 실용적 차이의 무시

대부분의 실제 이미지 처리 작업에서 두 방식의 최대 오차는 약 0.002 수준이다.
이것은 8비트 양자화 자체의 오차보다 훨씬 작다.
이미 8비트로 정보를 잃은 상태에서 정규화 방식의 차이를 논하는 것은 얼마나 실용적인가.
신경망 학습이나 의료 이미지 분석처럼 극도로 정밀한 값이 필요한 도메인에서는 의미가 있지만, 대부분의 애플리케이션에서는 과도한 분석일 수 있다.

### 컨텍스트 의존성

어떤 방식이 “더 나은가”는 애플리케이션에 따라 다르다.
순수한 검정과 흰색을 정확히 표현해야 하는 경우(마스킹, 합성)에는 255가 맞다.
균일한 분포를 가정하고 히스토그램 분석을 하는 경우에는 256이 더 수학적으로 깔끔할 수 있다.
글이 단일 권고로 끝맺는 것은 편의를 위한 단순화다.

256 방식이 지지하는 지각적 논거도 있다.[^herf]
낮 장면에서 RGB “0”은 실제 휘도 제로를 뜻하지 않는다 — 최고 밝기의 약 0.001배, 즉 수백만 개의 광자에 해당한다.
인간 지각에서 진정한 제로는 존재하지 않는다.
방송 시스템이 역사적으로 SDR 휘도 범위를 16–235로 제한한 이유도 같다.
“반드시 순수한 제로가 있어야 한다”는 주장은 특정 전제에 편향된 논거다.

하드웨어 구현에서는 정규화 방식과 무관하게 정확한 회색 표현 자체가 불가능한 경우도 있다.[^moefh]
마이크로컨트롤러로 VGA 신호를 생성할 때 8개 색상 출력 핀(R 3개, G 3개, B 2개)을 쓰면,
파란 채널의 전압이 (0V, 0.23V, 0.47V, 0.7V)가 되어 빨강·초록 채널의 전압 눈금과 맞지 않는다.
극단값을 제외하면 어떤 정규화를 써도 순수한 회색을 출력할 수 없다.
이 경우 정규화 선택은 이차적 문제다.

255 빈이 클리핑된 하이라이트를 나타낼 수 있다는 점도 간과되기 쉽다.[^kazinator]
과노출된 픽셀은 255에 고정되지만, 그것이 “1.0에 가까운 값”인지 “범위를 초과한 값”인지는 알 수 없다.
+0.5 이동은 이 모호함을 해결하지 못한다.
근본적 해법은 255를 “클리핑된 데이터”를 표시하는 카나리아 값으로 취급하고, 0–254만 유효 신호 범위로 쓰는 것이다.
이는 원문이 다루지 않는 전혀 다른 접근이다.

## 인사이트

### 사소한 디테일 뒤의 깊은 원칙

255 vs. 256이라는 질문은 사소해 보이지만, 이 안에 수치 표현의 근본 원칙이 들어있다.
이산화, 정규화, 양자화, 부동소수점 표현의 한계, 시스템 간 일관성이 모두 이 작은 선택에 응축된다.
이런 “작은 질문”들이 실제로 이미지 합성의 미묘한 색조 오차, 신경망의 학습 불안정성, 렌더링 아티팩트의 원인이 된다.

프로그래밍에서 “당연히 이렇게 하는 거지”라고 생각하는 관습들에는 종종 이런 배경이 있다.
왜 그렇게 하는지 모르고 따르다가, 다른 맥락에서 같은 관습을 잘못 적용하는 실수가 생긴다.
255로 나누는 것이 관습이 된 이유를 알면, 언제 그 관습을 따르고 언제 벗어나야 하는지를 판단할 수 있다.

### 표준의 기원과 경로 의존성

GPU UNORM이 255 방식을 채택한 것은 어느 날 갑자기 된 것이 아니다.
초기 디스플레이 하드웨어에서의 결정이 소프트웨어 라이브러리로, API로, 표준으로 전파된 결과다.
지금 255가 “맞는” 이유는 그것이 수학적으로 더 우월해서가 아니라 생태계 전체가 그것을 기준으로 맞춰졌기 때문이다.

이것은 기술 표준의 일반적 패턴이다.
최초의 선택이 경로 의존성(path dependency)을 만들고, 시간이 지나면서 그 선택 자체가 “표준”이 된다.
다른 시스템과의 상호운용성이 중요한 분야에서는 수학적 우아함보다 생태계 일관성이 더 중요하다.
이 원칙은 색상 정규화를 넘어 API 설계, 파일 포맷, 프로토콜 선택 모든 곳에 적용된다.

### 수치 코드의 숨겨진 가정

이미지 처리 코드를 작성할 때 `/ 255.0`이라고 타이핑하는 순간, 우리는 암묵적으로 많은 가정을 한다.
입력이 0~255 범위라는 것, 출력이 0.0~1.0 범위를 기대한다는 것, 다운스트림 코드가 이 범위를 가정한다는 것.
이 가정들이 맞지 않는 상황 — 16비트 HDR, scRGB, 음수 색상값을 허용하는 선형 색 공간 — 에서는 `/ 255.0`이 조용히 잘못된 결과를 만든다.

수치 코드에서 “당연한” 상수를 쓸 때 그것이 어떤 가정에 기반하는지 명시하는 습관이 중요하다.
255 대신 `float(MAX_UINT8_VALUE)` 같은 명명 상수를 쓰거나, 최소한 주석으로 전제를 기록하는 것이 유지보수성을 높인다.
이 작은 습관이 나중에 비트 심도를 바꾸거나 다른 색 공간을 지원해야 할 때 버그를 예방한다.
