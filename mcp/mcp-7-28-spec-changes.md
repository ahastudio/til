# AI 애플리케이션 개발자를 위한 MCP 7-28 소개: 무엇이 사라지고, 무엇이 깨질 수 있는가

원문: <https://discuss.pytorch.kr/t/ai-mcp-7-28/11163>

## 요약

이 글은 PyTorchKR 커뮤니티에 2026년 7월 8일 게시된 글로, 2026년
7월 28일경으로 예정된 MCP(Model Context Protocol) 스펙 변경,
일명 “MCP 7-28”을 AI 애플리케이션 개발자 관점에서 정리한다.
MCP는 모델과 외부 도구·데이터·실행 환경을 잇는 공통 규격으로,
Claude Desktop, Cursor, VS Code 등에서 쓰인다.
이번 변경은 새 기능 추가보다 마이그레이션에 초점을 맞춘
릴리스로, 프로토콜 핵심 개념을 명확히 하고 상태·인증 관리를
정비하며 스키마를 표준화하고 기존 기능의 수명 주기를 정리하는
데 목적이 있다.

변경은 크게 두 축으로 나뉜다.
첫째는 사용 중단(deprecated)으로 전환되는 세 기능이다(SEP-2577).
`Roots`는 클라이언트가 서버에 접근 가능한 로컬 시스템의
디렉토리나 파일 경계를 알려주는 기능인데, 런타임 통신 대신
설정이나 환경 변수를 통한 명시적 allowlist 방식으로 옮겨간다.
`Sampling`은 MCP 서버가 클라이언트나 호스트 쪽으로 LLM 응답
생성을 요청하는 기능으로, 대신 LLM 프로바이더 API를 직접
연동하는 방식이 권장된다.
`Logging`은 MCP 서버가 프로토콜 메시지로 구조화된 로그를
MCP 클라이언트에 전달하는 기능으로, OpenTelemetry와 표준
에러 스트림 채택이 권장된다.

둘째는 호환성이 깨지는 변경(breaking changes)이다.
상태 비저장/세션 없는 설계(SEP-2575, SEP-2567)는 프로토콜
수준의 세션 의존성을 제거하고, 암묵적 세션 상태 대신 명시적
상태 핸들을 도입한다.
예를 들어 `create_search(query="term")`가
`{"search_handle": "srch_7f3a"}`를 반환하면, 이후
`get_search_page(search_handle="srch_7f3a", page=2)`로
결과를 조회하는 패턴이다.
HTTP 인가(Authorization Spec Draft)는 OAuth 2.0 기반 흐름을
도입해 Bearer 토큰 검증, 보호된 리소스 메타데이터 검색, 인가
서버 검색, 스코프 챌린지와 audience 검증을 포함한다.
JSON Schema 2020-12(SEP-1613, SEP-2106)는 이 스키마 방언을
기본값으로 채택하고, 파라미터가 없는 도구라도 `inputSchema`를
null이 아니게 요구하며, `oneOf`·`anyOf`·`allOf`·조건부 스키마
같은 복잡한 스키마를 지원하고, `outputSchema`와
`structuredContent`가 객체가 아닌 JSON 값도 지원하도록
확장한다.

글은 이 변경이 실제 서버에 미치는 영향을 두 사례로 대조한다.
질의 중심 서버인 Legalize-KR은 매 호출마다 전체 입력을 받는
쿼리형 도구이고 프로토콜 세션 상태에 의존하지 않으며 `Roots`,
`Sampling`, `Logging` 의존도도 낮아 영향이 상대적으로 작을
것으로 예상된다.
반면 공식 레퍼런스 서버인 `filesystem`은 경계 관리를 위해
`Roots`에 크게 의존하고, `memory`는 영속적인 그래프 저장소를
가진 상태 저장형 서버로 세션 ID 대신 명시적 이름 지정 방식을
쓰는 패턴을 보여준다.
글은 마이그레이션을 위한 9개 항목 점검 목록도 제공한다.
MCP 기능 사용 현황 문서화, 사용 중단 기능 의존성 파악, 세션
상태 구현 위치 확인, 목록 엔드포인트의 세션 독립성 검증, 명시적
핸들 설계 패턴 검토, HTTP 인증 메커니즘 점검, 도구 스키마의
JSON Schema 2020-12 적합성 검증, 공식 서버 구현 참조, 스모크
테스트(스키마 목록화, 인증 실패, 핸들 만료) 준비가 그것이다.
영향이 작은 서버는 완전한 파라미터를 받는 쿼리 중심 도구,
프로토콜 세션 상태 없음, 로컬 stdio 전용 배포, 낮은 인증
복잡도를 특징으로 하며, 영향이 큰 서버는 `Roots` 의존적인 동적
워크스페이스 관리, 세션 범위 상태(브라우저 컨텍스트, 트랜잭션,
워크플로), 맥락 의존적 도구·리소스 목록화, 강한 인증 없이
HTTP로 노출된 경우를 특징으로 한다.

## 분석

### 세 가지 사용 중단 기능의 공통점은 “프로토콜이 대신해 주던 일”을 애플리케이션에 되돌려주는 것이다

`Roots`, `Sampling`, `Logging`이 동시에 사용 중단되는 것은
우연한 나열이 아니다.
세 기능 모두 원래 MCP 프로토콜이 애플리케이션 로직의 일부를
대신 처리해 주던 기능이라는 공통점을 가진다.
`Roots`는 파일 접근 경계 관리를, `Sampling`은 LLM 호출을,
`Logging`은 관측성을 프로토콜 수준에서 해결하려 했다.
이번 변경은 이 세 가지를 모두 프로토콜 바깥, 즉 설정
파일·직접 API 호출·표준 로깅 스택으로 이관한다.

이 패턴은 프로토콜 설계에서 반복적으로 나타나는 궤적이다.
초기 설계는 편의성을 위해 프로토콜이 많은 것을 떠맡지만,
실제 사용 사례가 다양해지면 프로토콜이 떠맡은 기능이 모든
사용 사례에 맞지 않는다는 것이 드러난다.
MCP 팀이 이 세 기능을 걷어내는 것은, 프로토콜의 책임 범위를
“모델과 도구를 연결하는 것” 자체로 좁히고, 파일 접근 정책이나
로깅 방식처럼 애플리케이션마다 다를 수밖에 없는 부분은
애플리케이션에 맡기겠다는 설계 철학의 전환을 보여준다.

### 명시적 상태 핸들 도입은 무상태(stateless) HTTP의 논리를 MCP에 이식하는 작업이다

세션 의존성을 제거하고 `search_handle` 같은 명시적 핸들을
반환하는 패턴은, REST API 설계에서 이미 익숙한 방식이다.
클라이언트가 서버의 암묵적 세션 상태에 의존하지 않고, 서버가
반환한 식별자를 클라이언트가 직접 들고 다니며 후속 요청에
사용하는 방식이다.
이 변경이 필요해진 이유는 MCP가 처음 설계될 당시 전제했던
“하나의 클라이언트가 하나의 서버와 지속적인 연결을 유지한다”는
가정이, 실제 배포 환경에서는 로드 밸런서 뒤에 여러 서버
인스턴스가 있거나 서버가 재시작되는 상황과 충돌하기
때문이다.

이 전환은 MCP가 로컬 stdio 기반의 단순한 도구 연결
프로토콜에서, 여러 클라이언트와 여러 서버 인스턴스가 관여하는
분산 시스템으로 성장하고 있다는 것을 보여준다.
세션이라는 개념 자체가 단일 서버 프로세스를 전제하는데, 이제
그 전제가 깨지는 배포 환경이 늘어나면서 프로토콜이 상태 관리
책임을 서버의 암묵적 메모리에서 클라이언트가 명시적으로
들고 다니는 핸들로 옮기는 것이다.

### 영향도 분류의 기준이 “프로토콜에 얼마나 기대고 있었는가”로 수렴한다

글이 제시하는 두 사례(Legalize-KR과 공식 레퍼런스 서버들)의
대조는 결국 하나의 축으로 요약된다.
프로토콜의 상태 관리·경계 관리 기능에 깊이 의존했던 서버일수록
마이그레이션 부담이 크고, 매 호출이 자기완결적인 쿼리형
서버일수록 부담이 작다는 것이다.
`filesystem` 서버가 `Roots`에 크게 의존하는 이유는 파일
시스템이라는 도메인 자체가 “어디까지 접근 가능한가”라는 경계
설정을 필수로 요구하기 때문이고, `memory` 서버가 상태
저장형인 이유는 그 서버의 존재 이유 자체가 영속적 데이터
보관이기 때문이다.

이 분류는 결국 마이그레이션 난이도가 서버의 코드량이나
복잡도가 아니라 “도메인의 본질적 요구사항이 사용 중단되는
기능과 얼마나 겹치는가”에 따라 결정된다는 것을 보여준다.
파일 시스템처럼 경계 관리가 본질적인 도메인은 `Roots`
사용 중단의 영향을 고스란히 받을 수밖에 없고, 이런 도메인의
서버 개발자들은 단순히 API 호출부를 바꾸는 수준을 넘어
경계 관리 로직 자체를 어떻게 재설계할지 고민해야 한다.

## 비평

### 사용 중단 기능의 대체 경로가 실제로 동등한 기능을 제공하는지 검증되지 않는다

글은 `Roots` 대신 “설정이나 환경 변수를 통한 명시적
allowlist”, `Sampling` 대신 “LLM 프로바이더 API 직접 연동”,
`Logging` 대신 “OpenTelemetry와 표준 에러 스트림”을
대체재로 제시한다.
하지만 이 대체재들이 원래 기능과 정확히 동등한 능력을
제공하는지는 검증하지 않는다.
`Roots`의 원래 장점은 클라이언트가 런타임에 동적으로 접근
가능한 경계를 서버에 알릴 수 있다는 것이었는데, 정적인 설정
파일이나 환경 변수는 이 동적 특성을 포기해야 한다.

사용자가 세션 중에 작업 디렉토리를 바꾸거나 새 프로젝트를
여는 시나리오에서, 정적 allowlist는 매번 재설정을 요구할
가능성이 있다.
글은 이 기능 손실 가능성을 언급하지 않은 채 대체 경로만
소개하는데, 실제로 `Roots`에 크게 의존했던 `filesystem` 같은
동적 워크스페이스 관리 서버가 이 전환으로 어떤 사용자 경험
저하를 겪을 수 있는지에 대한 논의가 빠져 있다.
마이그레이션 체크리스트가 “공식 서버 구현 참조”를 권장하는
것도, 결국 이 구체적인 손실을 각 개발자가 스스로 알아내야
한다는 뜻이다.

### 마이그레이션 체크리스트가 우선순위나 소요 기간을 제시하지 않는다

9개 항목으로 구성된 마이그레이션 체크리스트는 무엇을 점검해야
하는지는 나열하지만, 어떤 항목이 더 시급한지, 각 항목에
얼마나 시간이 걸릴지에 대한 안내는 없다.
세션 상태 구현 확인과 JSON Schema 2020-12 적합성 검증은
난이도와 소요 시간이 크게 다를 수 있는 작업인데, 체크리스트는
이 둘을 동일한 무게로 나열한다.

이 공백은 실제 마이그레이션을 수행해야 하는 개발자에게 실질적인
계획 수립 부담을 떠넘긴다.
7월 28일이라는 구체적 시한이 언급되는 만큼, 개발자들은 제한된
시간 안에 우선순위를 정해 작업해야 하는데, 글은 “무엇을
확인해야 하는가”까지만 정리하고 “무엇을 먼저 해야 하는가”는
다루지 않는다.
영향도가 큰 서버로 분류된 경우일수록 이 우선순위 정보의
부재가 더 큰 문제가 된다.

### “브레이킹 체인지”의 실제 파급 범위가 서버 개발자 관점에만 국한되어 있다

글 전체가 MCP 서버를 만드는 개발자의 관점에서 영향을 분석하지만,
MCP 클라이언트(Claude Desktop, Cursor, VS Code 같은 호스트
애플리케이션)를 만드는 개발자나, 그 클라이언트를 사용하는
최종 사용자에게 이 변경이 어떤 영향을 미치는지는 다루지
않는다.
세션 없는 설계로의 전환이 클라이언트 쪽 구현에도 상당한 변화를
요구할 것이 분명한데(핸들을 저장하고 관리하는 로직이 클라이언트
쪽에도 필요하다), 이 글은 그 부분을 서버 개발자의 시야
밖에 있는 것처럼 다룬다.

최종 사용자 관점에서도, 기존에 동작하던 MCP 서버가 7월 28일
이후 갑자기 작동을 멈추거나 재설정을 요구하게 될 가능성이
있다면 이는 사용자 경험에 직접 영향을 미치는 문제다.
글이 개발자 대상 기술 문서로서 이 범위를 좁힌 것은 이해할 수
있지만, “무엇이 깨질 수 있는가”라는 제목이 암시하는 폭넓은
질문에 비해 실제로 다루는 범위는 서버 구현이라는 한 단면에
그친다.

## 인사이트

### 프로토콜의 “책임 범위 축소”는 생태계가 충분히 성숙했다는 역설적 신호다

`Roots`, `Sampling`, `Logging`을 프로토콜에서 걷어내고
애플리케이션 계층으로 되돌리는 이번 변경은, 얼핏 기능 후퇴처럼
보이지만 실제로는 MCP 생태계가 이미 충분히 성숙했다는 신호로
해석할 수 있다.
프로토콜 초기 단계에서는 이런 기능들을 프로토콜이 직접
제공하는 것이 합리적이다.
아직 표준화된 대안(OAuth, OpenTelemetry 같은)과의 통합
방법이 검증되지 않았고, 개발자들에게 참고할 모범 사례도
부족하기 때문이다.

하지만 생태계가 성숙하면 상황이 바뀐다.
이제는 OAuth 2.0이나 OpenTelemetry 같은 업계 표준이 이미
널리 검증되어 있고, MCP가 굳이 이를 재발명할 필요가 없다는
합의가 형성된다.
이는 마치 초기 웹 프레임워크가 세션 관리, 인증, 로깅을 모두
프레임워크 내장 기능으로 제공하다가, 생태계가 성숙하면서
이런 관심사를 분리된 미들웨어나 외부 표준 라이브러리로
위임하는 흐름과 유사하다.
프로토콜이 “덜 하는 것”을 선택하는 시점은 역설적으로 그
프로토콜을 둘러싼 도구 생태계가 “더 잘하게 된” 시점과
일치한다.

### 세션 없는 설계로의 전환은 MCP가 로컬 도구에서 클라우드 인프라로 이동하고 있음을 보여준다

MCP는 원래 Claude Desktop 같은 로컬 애플리케이션이 로컬
파일 시스템이나 로컬 프로세스와 통신하는 상황을 염두에 두고
설계되었을 가능성이 크다.
이런 환경에서는 하나의 서버 프로세스와 하나의 클라이언트가
안정적으로 세션을 유지하는 것이 자연스러운 전제였다.
하지만 이번 변경이 세션 의존성을 제거하고 로드 밸런서 뒤에
여러 서버 인스턴스가 있는 상황, HTTP 기반 인증이 필요한
상황을 전제로 설계를 바꾼다는 것은, MCP 서버가 이제 로컬
프로세스가 아니라 여러 사용자가 공유하는 클라우드 인프라로
배포되는 경우가 늘어나고 있다는 것을 시사한다.

이 전환은 초기 소프트웨어 도구가 로컬 환경을 전제로
설계되었다가, 사용자 저변이 넓어지면서 다중 사용자·분산
환경을 지원하도록 재설계되는 익숙한 패턴을 따른다.
데이터베이스가 단일 사용자 파일 기반에서 클라이언트-서버
구조로, 다시 분산 클러스터로 진화한 경로와 유사하다.
MCP가 이 경로를 밟고 있다는 것은, AI 에이전트와 도구를 연결하는
인프라 자체가 이제 개인 개발자의 실험 도구를 넘어 조직 차원의
공유 인프라로 취급받기 시작했다는 더 큰 흐름을 보여준다.

### 마이그레이션 중심 릴리스라는 표현 자체가 AI 도구 생태계의 표준화 압력을 드러낸다

글이 “새로운 기능을 추가하기보다는 마이그레이션 중심 릴리스”
라고 이 변경을 규정한 것은, MCP가 이미 상당수 실전 배포
사례를 가진 표준으로 자리 잡았다는 증거다.
아직 초기 단계인 프로토콜이라면 기능 추가가 릴리스의 주된
내용이 되지만, 이미 많은 서버와 클라이언트가 존재하는
프로토콜에서는 하위 호환성을 깨는 변경 하나하나가 생태계
전체에 파급되는 비용을 동반한다.
그럼에도 불구하고 MCP 팀이 이런 파괴적 변경을 감행한다는
것은, 지금 바로잡지 않으면 장기적으로 더 큰 기술 부채가
쌓일 것이라는 판단이 섰다는 뜻이다.

이 판단은 프로토콜 표준화 과정에서 반복적으로 나타나는
긴장, 즉 “지금 당장의 호환성”과 “장기적으로 지속 가능한
설계” 사이의 트레이드오프를 보여준다.
HTTP가 버전 1.0에서 1.1로, 다시 2.0으로 넘어가며 겪었던
진통이나 여러 웹 표준이 초기 설계의 결함을 뒤늦게 고치며
겪은 혼란과 유사한 국면에 MCP도 진입한 것으로 볼 수 있다.
이런 마이그레이션이 반복될수록, 앞으로 MCP 서버를 만드는
개발자들은 프로토콜의 안정성을 처음부터 낮게 잡고 변경에
대응할 수 있는 유연한 아키텍처(이번 사례의 명시적 핸들
패턴처럼)를 선호하게 될 가능성이 크다.
