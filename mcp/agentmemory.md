# agentmemory

<https://www.agent-memory.dev/>

<https://github.com/rohitg00/agentmemory>

## 소개

agentmemory는 AI 코딩 에이전트를 위한 영구 메모리 시스템이다. Claude Code,
Cursor, Gemini CLI 등 다양한 에이전트에서 세션 간 맥락을 유지한다. 세션이
종료될 때마다 스택 구조, 아키텍처 결정, 진행 중인 작업을 처음부터 다시
설명해야 하는 문제를 해결한다.

연간 토큰 사용량 92% 절감(약 170K 토큰)을 목표로 하며, 비용은 연간 약 $10
수준이라고 주장한다. LongMemEval-S 기준 R@5 95.2%의 검색 정확도를 제공한다.

## 아키텍처

메모리 파이프라인은 다음과 같이 동작한다.

```text
PostToolUse → 중복 제거 → 프라이버시 필터 → 원본 저장
→ LLM 압축 → 벡터 임베딩 → BM25/벡터 인덱싱
```

4단계 메모리 통합 구조를 사용한다.

1. **작업용(working)** — 현재 세션의 즉각적 맥락
2. **에피소드(episodic)** — 과거 세션의 구체적 사건
3. **의미론적(semantic)** — 추상화된 사실과 개념
4. **절차적(procedural)** — 반복적으로 사용하는 워크플로우

검색은 BM25, 벡터, 지식 그래프 기반 하이브리드 방식을 사용하며, 12개
라이프사이클 훅으로 도구 사용을 자동 기록한다.

## 주요 기능

- **자동 캡처**: 에이전트의 도구 사용을 자동으로 기록하고 압축
- **프라이버시 보호**: API 키 등 민감 정보 자동 필터링
- **다중 에이전트 지원**: 팀 간 메모리 공유 가능
- **실시간 뷰어**: 세션 탐색, 메모리 브라우저, 지식 그래프 시각화(포트 3113)
- **51개 MCP 도구**: MCP 서버 및 REST API(124개 엔드포인트, 포트 3111) 제공

## 설치 및 사용법

전역 설치 후 서버를 시작한다.

```bash
npm install -g @agentmemory/agentmemory
agentmemory  # 서버 시작 (:3111)
```

샘플 데이터로 빠르게 테스트할 수 있다.

```bash
npx @agentmemory/agentmemory demo
```

에이전트별 연결 방법은 다음과 같다.

- **Claude Code**: `/plugin marketplace add rohitg00/agentmemory`(12 훅 + MCP)
- **Cursor / Claude Desktop / Cline**: `mcpServers` 설정에 MCP 엔트리 추가
- **Codex CLI**: 플러그인 마켓플레이스 등록(6 훅 포함)

## 라이선스 및 상태

Apache-2.0 라이선스. 공식 사이트는 agent-memory.dev이다.
[GeekNews 댓글](https://news.hada.io/topic?id=29754)에서는 post session hook
연결 미비로 그래프 구성이 불가능하고 MCP 서버 응답 속도가 약 3초로 느리다는
실사용 피드백이 있었다. “아직 시기 상조”라는 평가도 나왔다.

## 분석

### MCP 생태계와의 통합 전략

51개 MCP 도구를 제공하는 구조는 agentmemory를 단순한 메모리 저장소가 아닌
에이전트 인프라 계층으로 포지셔닝하려는 의도를 보여준다. MCP 프로토콜은 Claude
Code를 비롯한 여러 에이전트 플랫폼에서 표준 통합 방식이 되고 있으며, 이 흐름에
올라탄 프로젝트다. 그러나 51개라는 숫자 자체가 복잡성을 암시하기도 한다.
실제로 일반 사용자가 필요한 도구 수가 얼마인지는 별도 검토가 필요하다.

### 세션 메모리 문제의 근본 원인

AI 에이전트의 세션 간 맥락 손실 문제는 LLM의 컨텍스트 창 한계에서 비롯된다.
agentmemory의 접근 방식은 중요한 정보를 선택적으로 압축하여 다음 세션에 주입하는
방식으로 이 한계를 우회한다. 그러나 “무엇이 중요한가”를 판단하는 압축 단계에서
LLM을 다시 사용한다는 점은 비용과 지연 시간 측면에서 트레이드오프가 있다.

### iii 엔진 기반 설계의 위험성

프로젝트가 “Postgres, Redis, Express, pm2, Prometheus를 대체”한다고 주장하는 iii
엔진에 강하게 의존하는 점은 주목해야 할 부분이다. 표준 기술 스택 대신 독자적
엔진에 기반한 구조는 생태계 의존성을 만들고, 해당 엔진의 안정성과 지속 가능성이
프로젝트 전체의 신뢰성에 직결된다.

## 비평

### 강점: 실제 개발자 고통 지점 해결

세션마다 맥락을 재설명해야 하는 문제는 AI 코딩 에이전트를 실제로 사용하는 개발자
대부분이 경험하는 고통이다. 이 고통이 크다는 점, 그리고 이를 자동화로 해결하려는
접근 방향 자체는 타당하다. 비용 절감 수치가 실제로 검증된다면 상당한 실용적
가치를 가진다.

### 약점: 초기 상태와 검증 부족

GeekNews 커뮤니티의 실사용 피드백에서 post session hook 미연결, 느린 MCP 응답
속도 같은 기본적인 기능 결함이 지적된 것은 프로젝트가 아직 프로덕션 준비가
되지 않았다는 신호다. R@5 95.2% 같은 인상적인 수치가 어떤 조건에서 측정된
것인지, 실제 사용 환경에서 재현 가능한지 독립 검증이 필요하다.

### 약점: 독점 의존성의 위험

iii 엔진이라는 독자적 인프라에 대한 강한 의존성은 장기적 위험 요소다. 오픈소스
프로젝트임에도 핵심 의존성이 잘 알려지지 않은 독자적 엔진에 있다면, 커뮤니티
기여와 장기 유지보수 가능성에 의문이 생긴다.

## 인사이트

### 에이전트 메모리는 아직 미해결 문제

agentmemory가 등장했다는 사실 자체가 AI 에이전트 생태계에서 세션 지속성 문제가
아직 표준적으로 해결되지 않았음을 방증한다. OpenAI, Anthropic 같은 플랫폼들이
공식적으로 이 문제를 해결하는 방향으로 나아가고 있지만, 현재까지는 각자의 방식으로
해결책을 구축해야 하는 공백이 존재한다. 이 공백이 얼마나 오래 지속되느냐에
따라 agentmemory 같은 서드파티 솔루션의 시장 기회가 결정된다. 플랫폼 레벨에서
표준 메모리 API가 나오는 순간, 이 종류의 도구들은 빠르게 상품화된다.

### 메모리 압축은 정보 손실이다

LLM이 도구 사용을 압축하는 과정에서 어떤 정보가 중요한지 판단한다. 그러나 LLM의
판단이 틀릴 수 있다. 현재 작업과 무관해 보이지만 나중에 중요해지는 맥락, LLM이
이해하기 어려운 도메인 특화 정보, 암묵적 가정 등이 압축 과정에서 손실될 수 있다.
메모리 시스템이 “기억하는 것”만큼 “무엇을 잊었는지”가 에이전트 동작에 영향을 미친다.
이 손실을 최소화하는 압축 전략이 이 분야의 핵심 기술 과제다.

### 팀 공유 메모리의 새로운 가능성

팀 간 메모리 공유 기능은 현재는 부수적 기능처럼 보이지만, 잠재적으로는 가장 중요한
차별화 요소일 수 있다. 개인 에이전트 메모리가 팀 지식 베이스로 확장될 때,
이것은 단순한 세션 지속성을 넘어 조직 지식 관리의 새로운 계층이 된다. 특정 코드베이스에
대한 결정 이력, 반복적으로 등장하는 패턴, 팀의 코딩 관행 등이 자동으로 쌓이는
구조는 온보딩 비용을 줄이고 지식 이전을 가속화할 수 있다.
