# Twitter MCP 서버: Twitter API와 문서를 AI 도구에 연결하는 공식 MCP

<https://docs.x.com/tools/mcp>

## 소개

Twitter(현 X)가 공식 MCP(Model Context Protocol) 서버 두 개를 제공한다.

**X MCP** (`https://api.x.com/mcp`)는 Twitter API 엔드포인트를 직접 호출하는 서버로,
포스트 조회, 전체 아카이브 검색, 사용자 정보 조회, 북마크 관리, 뉴스 스토리 접근,
Article 작성 및 발행 등의 기능을 AI 도구에 제공한다.

**Docs MCP** (`https://docs.x.com/mcp`)는 Twitter 개발자 문서를 검색하고 조회하는 서버로,
`search_x`와 `get_page_x` 두 개의 도구를 제공한다.
인증 없이 URL만으로 설정 가능하다.

## 아키텍처

X MCP는 로컬 브리지 방식으로 동작한다.
MCP 클라이언트가 stdio JSON-RPC로 로컬 브리지(`xurl mcp`)에 연결하면,
브리지가 OAuth2 PKCE 인증과 토큰 갱신을 처리하고,
HTTPS Bearer 토큰을 붙여 `api.x.com`의 호스팅 MCP 서버로 요청을 전달한다.
자격증명은 로컬 머신에만 보관되며(`~/.xurl`), Bearer 토큰만 TLS를 통해 X 서버로 전송된다.

## 설정

개발자 포털에서 OAuth 2.0이 활성화된 X 앱을 생성하고,
콜백 URI(`http://localhost:8080/callback`)를 등록한 뒤 CLIENT_ID와 CLIENT_SECRET을 발급받아야 한다.
Node.js가 필요하며, `xurl`은 Homebrew, npm, 셸 스크립트로 설치할 수 있다.

```json
{
  "mcpServers": {
    "x-mcp": {
      "command": "npx",
      "args": ["@xdevplatform/xurl", "mcp"],
      "env": {
        "CLIENT_ID": "<your-client-id>",
        "CLIENT_SECRET": "<your-client-secret>"
      }
    }
  }
}
```

클라이언트별 설정 위치:

| 클라이언트     | 설정 파일 경로                          |
| -------------- | --------------------------------------- |
| Claude Desktop | `~/Library/Application Support/Claude/` |
| Cursor         | `~/.cursor/mcp.json`                    |
| VS Code        | `.vscode/mcp.json`                      |
| Grok Build     | `~/.grok/config.toml`                   |

## 인증 방식

- **OAuth 2.0 (기본)**: 사용자 계정 권한으로 동작. 토큰은 `~/.xurl`에 캐시되고 자동 갱신된다.
- **Headless 인증**: 브라우저 없는 원격 서버용. `xurl auth oauth2 --headless`로 코드 기반 플로를 먼저 완료한다.
- **App-only 토큰**: 정적 Bearer 토큰으로 URL에 직접 연결. 읽기 전용이며 자동 갱신 없음.
- **다중 계정**: `--app NAME` 또는 `-u USERNAME` 플래그로 앱/사용자를 전환한다.

## 주요 기능

- 포스트 및 참여 지표 조회 (좋아요, 리포스트, 인용 목록)
- 전체 아카이브 포스트/사용자/뉴스 검색
- 사용자 정보 조회, 타임라인 및 멘션 조회
- 북마크 및 북마크 폴더 관리
- 위치 기반 트렌드 및 뉴스 스토리 접근
- Article 초안 작성 및 발행

OpenAPI 명세는 `https://api.x.com/2/openapi.json`에서 제공되며,
클라이언트 생성, Postman 임포트, 커스텀 에이전트 통합에 활용할 수 있다.

## 분석

### MCP를 통한 X API 접근이 기존 SDK 방식과 다른 이유

기존 X API 통합은 개발자가 HTTP 클라이언트를 직접 구성하고,
OAuth 플로를 구현하며, 응답을 파싱하는 코드를 작성해야 했다.
MCP는 이 모든 것을 AI 도구가 자연어 명령으로 호출할 수 있는 도구 인터페이스로 변환한다.
“최근 7일 간 내 멘션 가져와”라는 프롬프트가 X API 엔드포인트 호출로 자동 변환된다.

이 패턴은 X만의 것이 아니다.
GitHub, Stripe, Notion이 각자의 MCP 서버를 출시하면서,
각 플랫폼의 API가 AI 에이전트의 도구 팔레트에 직접 편입되는 방향으로 생태계가 이동하고 있다.
X MCP는 이 흐름에서 소셜 미디어 플랫폼이 어떻게 AI 에이전트 시대에 적응하는지를 보여주는 사례다.

### 로컬 브리지 패턴이 보안과 편의의 균형을 다루는 방식

X MCP의 아키텍처에서 주목할 점은 자격증명이 로컬에 머문다는 것이다.
MCP 클라이언트(Claude Desktop, Cursor 등)는 Bearer 토큰을 직접 보유하지 않는다.
로컬 브리지가 OAuth 협상과 토큰 갱신을 담당하고,
MCP 서버로는 이미 인증된 Bearer 토큰으로 요청한다.

이 구조는 MCP 클라이언트에 비밀 키를 직접 주입하는 방식보다 안전하다.
그러나 로컬 브리지가 단일 장애 지점이 된다.
`~/.xurl` 디렉터리에 저장된 토큰이 노출되거나 로컬 프로세스가 침해되면
X 계정 접근 권한 전체가 위험해진다.
보안 강화를 위해 `~/.xurl`을 비밀처럼 다루고, 최소 권한 스코프만 요청하라는 권고가 이유가 있다.

### Docs MCP와 X MCP의 동시 실행이 가능하게 하는 자가참조 패턴

두 MCP 서버를 동시에 구성하면 AI 에이전트가 실행과 참조를 동시에 할 수 있다.
API 호출이 실패했을 때 Docs MCP로 문서를 조회해 오류 코드를 확인하고,
올바른 파라미터로 재시도하는 루프가 에이전트 내부에서 자동으로 이루어진다.

이것은 개발자가 IDE에서 API 문서를 탭으로 열어두고 코드를 작성하는 방식의 AI 등가물이다.
`search_x`로 엔드포인트 문서를 찾고, `get_page_x`로 상세 명세를 가져온 뒤,
X MCP 도구로 실제 호출을 수행한다.
이 패턴이 성숙하면 에이전트가 신규 API 기능을 문서를 통해 스스로 학습하고 활용하는 것이 가능해진다.

## 비평

### OAuth 2.0 앱 등록 장벽이 일반 사용자 접근성을 막는다

X MCP를 사용하려면 X 개발자 포털에서 앱을 등록하고,
OAuth 2.0을 활성화하고, 콜백 URI를 등록하고, CLIENT_ID와 CLIENT_SECRET을 발급받아야 한다.
이것은 개발자에게는 친숙한 과정이지만, AI 도구를 활용하려는 일반 사용자에게는 진입 장벽이다.

Claude Desktop이나 Cursor 같은 AI 도구의 주 사용자가 반드시 개발자인 것은 아니다.
개발자 포털 접근, 앱 등록, OAuth 설정을 요구하는 MCP가 “AI 도구를 통한 X 접근을 민주화”한다는 주장은 과장이다.
실제로는 개발자 계층에게만 접근 가능한 고급 통합이다.
Docs MCP가 인증 없이 URL만으로 설정 가능한 것과 대조된다.

### npx 의존성이 에이전트 환경에서 안정성 문제를 만든다

모든 클라이언트 설정이 `npx @xdevplatform/xurl mcp` 명령을 통해 실행된다.
이것은 실행 시마다 npm 레지스트리에서 패키지를 가져올 수 있다는 의미다.
네트워크 연결이 없거나 npm 레지스트리에 문제가 생기면 MCP 서버 자체가 시작되지 않는다.

에이전트 환경에서 외부 의존성은 예측 불가능한 장애 지점이다.
버전 고정 없이 항상 최신 버전을 가져오는 경우 패키지 업데이트로 인한 호환성 문제도 생길 수 있다.
프로덕션 환경에서 MCP 서버를 안정적으로 운영하려면 로컬 설치 및 버전 고정이 사실상 필수인데,
공식 문서가 이 점을 명시하지 않는다.

### App-only 인증의 제약이 읽기 전용 사용 사례의 가치를 낮춘다

App-only 토큰 방식은 자동 갱신이 없고 읽기 전용으로 제한된다.
사용자 컨텍스트 없이 공개 데이터만 접근할 수 있다는 의미다.

X API v2의 읽기 전용 접근 범위는 이미 기본 플랜에서 상당히 제한된다.
월 500만 트윗 읽기로 제한된 무료 티어에서 X MCP로 에이전트가 대규모 데이터를 처리하는 것은
현실적으로 불가능하다.
유료 Basic($100/월), Pro($5,000/월) 플랜이 없으면 전체 아카이브 검색 같은
X MCP의 핵심 기능이 사실상 작동하지 않는다.
문서가 요금제에 따른 기능 제한을 명시적으로 언급하지 않는 것은 오해를 불러일으킬 수 있다.

## 인사이트

### 플랫폼 MCP 출시는 API 생태계의 접근 계층 재편을 의미한다

기존에 X API는 HTTP를 이해하는 개발자가 코드로 접근하는 인터페이스였다.
MCP는 이 인터페이스를 AI 에이전트가 직접 소비할 수 있는 도구 레이어로 변환한다.
이것은 단순한 래퍼(wrapper) 추가가 아니라 API 접근 계층의 재편이다.

앞으로 주요 플랫폼 API의 “1차 소비자”는 인간 개발자가 아니라 AI 에이전트가 될 가능성이 있다.
에이전트가 X API를 직접 호출해 콘텐츠를 검색하고, 분석하고, 게시하는 워크플로가 표준화된다면,
X의 API 비즈니스는 인간 개발자 고객보다 AI 에이전트 운영자를 주 고객으로 상정하게 된다.
요금제 설계, 레이트 리밋, 기능 범위가 이 방향으로 재설계될 것이다.

### Docs MCP와 API MCP의 분리 설계가 에이전트 신뢰성의 새 기준을 제시한다

X가 두 MCP를 분리한 것은 실용적 선택처럼 보이지만, 더 깊은 설계 원칙을 담고 있다.
API를 실행하는 서버와 API 문서를 제공하는 서버를 분리하면,
에이전트가 “모르면 먼저 문서를 찾는다”는 행동을 구조적으로 지원할 수 있다.

현재 LLM 기반 에이전트의 한계 중 하나는 학습 시점 이후 업데이트된 API를 모른다는 것이다.
Docs MCP가 항상 최신 문서를 제공하면, 에이전트는 자신의 학습 데이터가 아니라
실시간 문서를 참조해 올바른 API 호출을 구성할 수 있다.
이 패턴이 확산되면 “AI 에이전트가 잘못된 API를 호출하는 문제”의 빈도가 구조적으로 감소한다.

### X MCP의 쓰기 기능은 소셜 미디어 자동화 규범의 새로운 시험대다

X MCP는 포스트를 읽는 것뿐 아니라 Article을 작성하고 발행하는 기능을 포함한다.
AI 에이전트가 X API를 통해 콘텐츠를 자동 생성하고 게시하는 것을 공식적으로 지원하는 구조다.

이것은 소셜 미디어 자동화에 대한 X의 정책 방향을 명확히 보여준다.
과거 X(트위터)는 봇을 규제하고 자동 게시를 제한하려 했다.
그러나 공식 MCP를 통한 AI 에이전트의 게시를 공식 도구로 지원한다는 것은,
X가 AI 생성 콘텐츠와 자동화된 게시를 플랫폼의 미래로 수용한다는 신호다.
이것이 인간 사용자 경험과 플랫폼 신뢰도에 어떤 영향을 미칠지는
X MCP 생태계가 성숙함에 따라 가시화될 것이다.
