# AI시대에는 AdMob 콘솔에 로그인하지 마세요

원문: [AI시대에는 AdMob 콘솔에 로그인하지 마세요 (feat. admob mcp) | by Ted Park | 박상권의 삽질블로그 | Jul, 2026 | Medium](https://medium.com/@gun0912/c2419c38041a)

AdMob MCP Server: <https://github.com/ParkSangGwon/admob-mcp-server>

## 요약

Ted Park(@gun0912, 박상권의 삽질블로그)이 공개한 AdMob MCP 서버 도입기다.
Google AdMob API를 자연어로 조회하고 수정하는 MCP 서버 `admob-mcp-server`를 만든 과정과
그 과정에서 겪은 다섯 가지 삽질을 상세히 기록했다.

`admob-mcp-server`는 AdMob API v1beta 전체를 커버하는 18개의 tool을 제공한다.
리포트 조회 3종(네트워크, 미디에이션, 앱), 쓰기 작업 7개(앱·광고 단위·미디에이션 그룹 생성·수정·조회),
분석 prompt 3종(수익 분석, 이상 탐지, 최적화 제안)으로 구성된다.
Node.js 18 이상 환경에서 `npx admob-mcp-server auth`로 OAuth 인증 후 AI 클라이언트에 연결하면
“이번 달 수익 보여줘”처럼 자연어로 AdMob 데이터를 다룰 수 있다.

기사는 개발 과정에서 마주친 다섯 가지 삽질을 솔직하게 공유한다.
첫째, AdMob API는 서비스 계정(Service Account)을 지원하지 않는다.
OAuth2 사용자 인증만 가능하기 때문에 MCP 서버에서 OAuth 플로를 직접 구현해야 했다.
둘째, `admob.monetization` 스코프가 Google API discovery 문서에 빠져 있다.
공식 문서 검색으로는 찾을 수 없고, 소스를 직접 뒤져야 했다.
셋째, 리포트 API 응답은 단일 JSON이 아니라 header, row, footer 세 종류 청크가
스트림으로 내려오는 구조다.
금액 단위도 micros(100만분의 1 단위)여서 별도 변환이 필요하다.
넷째, 미디에이션 그룹 생성 시 새 미디에이션 라인 ID를 `“-1”`, `“-2”` 같은 음수 문자열 placeholder로 지정해야 한다.
직관에 어긋나는 설계다.
다섯째, 인증이 실패하더라도 MCP 서버를 종료하면 안 된다.
서버가 죽으면 AI 클라이언트가 “tool not found” 오류를 낸다.
lazy auth(지연 인증)와 인증 방법을 안내하는 actionable 에러 메시지로 해결했다.

## 분석

### MCP가 기존 API 통합 방식과 다른 이유

AdMob 콘솔은 웹 인터페이스다.
반복적인 데이터 확인과 수정 작업을 위해 매번 브라우저를 열고, 메뉴를 탐색하고, 필터를 설정해야 한다.
AdMob API를 직접 호출하는 스크립트를 짜는 방법도 있지만,
이는 개발자만 할 수 있는 작업이고 용도가 고정된다.

MCP는 이 중간 지점을 채운다.
API의 유연성과 콘솔의 접근성을 동시에 제공한다.
AI 에이전트가 tool을 선택하고 파라미터를 채워 API를 호출하기 때문에
사용자는 의도를 자연어로 말하기만 하면 된다.
“지난 7일 앱별 수익 비교해줘”라는 요청이 바로 API 호출로 이어진다.

이 구조의 핵심은 tool 설계에 있다.
18개 tool 각각이 AdMob API의 특정 기능과 1:1로 대응하도록 설계됐다.
AI 에이전트가 tool description을 읽고 적절한 tool을 선택하기 때문에,
tool 이름과 설명이 명확할수록 에이전트가 올바른 tool을 선택할 가능성이 높아진다.

### 삽질 다섯 개가 공개 문서인 이유

기사 제목은 마케팅이 아니라 기술 문서다.
AdMob API를 쓰려는 개발자가 반드시 마주칠 다섯 가지 함정을 미리 알려준다.

서비스 계정 미지원과 `admob.monetization` 스코프 누락은 공식 문서에서 알 수 없다.
이런 정보는 실제로 구현해보거나 커뮤니티에서 찾아야 한다.
“삽질 기록”이라는 형식이 이 지식을 공유하는 가장 효과적인 방법이다.
API 설계의 비직관적인 부분(음수 ID, 청크 스트림 응답)을 명시적으로 기록하지 않으면
다음 개발자가 같은 함정에 빠진다.

### 오픈소스 MCP 서버 생태계의 패턴

admob-mcp-server는 특정 외부 API를 MCP로 감싸는(wrap) 패턴의 사례다.
이 패턴이 빠르게 확산 중이다.
Stripe, GitHub, Notion, Slack 등 주요 서비스의 MCP 서버가 이미 공개됐다.

이 패턴의 가치는 두 방향에서 온다.
서비스 제공자 입장에서는 AI 에이전트와 자신의 API를 연결하는 표준 통로가 생긴다.
개발자 입장에서는 API SDK 대신 MCP 서버를 선택해 자연어 인터페이스를 바로 얻는다.
admob-mcp-server가 그 사례다.

## 비평

### OAuth 요구사항이 배포 복잡성을 높인다

기사가 “삽질 1”로 기록한 서비스 계정 미지원 문제는
단순히 구현 난이도의 문제가 아니다.
OAuth는 사용자가 직접 인증 플로를 통과해야 한다.
이는 admob-mcp-server를 완전히 자동화된 환경(서버, CI/CD)에 배포하기 어렵게 만든다.
사용자가 브라우저를 열고 Google 인증을 완료해야 하기 때문이다.

서버 환경에서 MCP를 24/7 실행하려는 사용자에게 이것은 실질적인 장벽이다.
OAuth 토큰을 안전하게 저장하고 refresh token으로 자동 갱신하는 로직이 필요하다.
기사는 이 문제를 단순히 “OAuth를 구현했다”로 처리하지만,
장기 운영 관점에서 token 만료와 재인증이 어떻게 처리되는지는 여전히 열린 질문이다.

### 공식 문서 부재가 비공식 지식에 의존하게 만든다

`admob.monetization` 스코프가 Google API discovery 문서에 빠져 있다는 것은
Google 측 문서화 품질의 문제다.
이 MCP 서버가 이 사실을 기록해 공유한다는 것은 가치 있지만,
기반이 되는 문서 부재가 해결된 것은 아니다.

Google이 이 스코프를 나중에 변경하거나 추가하면,
admob-mcp-server와 이 기사를 참고한 구현체들이 모두 업데이트돼야 한다.
비공식 지식으로 채운 빈틈이 나중에 유지보수 부담으로 돌아온다.
`admob.monetization`처럼 문서화되지 않은 의존성은 외부에서 검증하거나 테스트하기도 어렵다.

### 쓰기 작업의 안전 장치가 보이지 않는다

tool 목록에 쓰기 작업이 7개 포함된다.
앱·광고 단위·미디에이션 그룹을 생성하고 수정하는 작업이다.
이 작업들은 AdMob 수익과 직접 연결되기 때문에 실수가 실질적인 피해로 이어질 수 있다.

기사는 쓰기 tool이 존재한다는 사실을 언급하지만,
AI 에이전트가 잘못된 맥락에서 쓰기 작업을 실행하는 경우에 대한 안전 장치를 다루지 않는다.
읽기-확인-쓰기 패턴을 강제하거나, 위험한 작업에 사용자 확인을 요구하거나,
dry-run 모드를 제공하는 것이 실용적인 배포에서는 필수적이다.
MCP의 자연어 인터페이스는 의도치 않은 쓰기 실행 위험을 오히려 높인다.

## 인사이트

### AI 에이전트 시대의 API 설계 부채

Google AdMob API의 삽질 다섯 개는 API가 인간 개발자를 위해 설계됐다는 것을 보여준다.
서비스 계정 미지원, 문서 누락, 청크 스트림 응답, 음수 placeholder ID 모두
API를 직접 호출하는 개발자가 한 번씩 겪고 넘어가면 되는 문제였다.

AI 에이전트가 API를 중간 개발자 없이 직접 호출하는 시대에서 이런 설계는 증폭된다.
에이전트가 스코프 이름을 잘못 추측하거나, 청크 스트림을 단일 응답으로 오해하거나,
음수 ID가 placeholder인 줄 모르고 실제 ID로 처리하면 조용히 실패한다.
MCP 서버 개발자가 이 모든 불일치를 흡수해 에이전트에게 일관된 인터페이스를 제공해야 한다.
MCP 래퍼 레이어가 API 설계 부채를 청산하는 역할을 맡게 된다.

### “자연어 콘솔” 패러다임의 실용적 진입점

“AI시대에는 AdMob 콘솔에 로그인하지 마세요”라는 제목은 도발적이지만 구체적이다.
특정 웹 콘솔을 자연어 인터페이스로 대체하는 것이 목표다.

이 패러다임의 실용적 가치는 단순한 자동화를 넘는다.
반복적인 조회 작업의 효율화뿐 아니라, 데이터를 보는 방식이 달라진다.
“지난 30일 대비 이번 달 eCPM이 10% 하락한 앱들의 미디에이션 구성 확인해줘”라는 복합 쿼리는
AdMob 콘솔에서 여러 화면을 오가야 답이 나온다.
MCP를 통하면 하나의 자연어 명령으로 처리된다.

이 패러다임이 확산되면 기업용 SaaS 콘솔 자체의 역할이 재정의될 수 있다.
콘솔은 설정 중심의 관리 인터페이스로 남고,
일상적인 조회와 분석은 AI 에이전트를 통하는 구조가 자연스럽게 자리 잡을 것이다.

### 오픈소스 MCP 래퍼가 드러내는 생태계 압력

admob-mcp-server가 공개됐다는 것은
Google이 공식 AdMob MCP 서버를 제공하지 않았다는 뜻이다.
이 빈틈을 커뮤니티가 채운다.

이 패턴은 API 경제의 오래된 흐름과 닮았다.
Twilio, Stripe 같은 API 기업이 SDK를 뒤늦게 공식 지원하기 전에
커뮤니티 라이브러리가 먼저 자리를 잡았던 것처럼,
MCP 서버도 같은 경로를 밟고 있다.
서비스 제공자들이 AI 에이전트 생태계에 적극적으로 진입하지 않으면
커뮤니티 래퍼가 표준을 선점한다.
Google이 공식 AdMob MCP 서버를 내놓는다면 admob-mcp-server가 기여한 설계 패턴이
그 안에 반영될 가능성이 높다.
오픈소스 커뮤니티가 기업 제품 로드맵을 선행하는 구조다.
