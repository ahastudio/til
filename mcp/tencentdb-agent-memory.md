# TencentDB Agent Memory

<https://github.com/TencentCloud/TencentDB-Agent-Memory>

## 소개

TencentDB Agent Memory는 Tencent Cloud가 공개한 AI 에이전트용 장기 기억
시스템이다.
핵심 슬로건은 “Agents remember, Humans innovate”로,
에이전트가 기억을 담당하고 인간은 창의적인 작업에 집중해야 한다는
철학을 내세운다.
외부 API 없이 로컬에서 동작하는 것을 기본값으로 삼으며,
기본 저장소는 `SQLite + sqlite-vec` 조합이다.
클라우드로 확장할 경우 Tencent Cloud Vector Database(TCVDB)를 백엔드로
붙일 수 있다.

이 프로젝트가 겨냥하는 문제는 네 가지다.
사용자가 SOP나 프로젝트 배경, 도구 규칙을 매 세션마다 재설명해야 하는
반복 설명 문제,
장시간 작업 시 대화 이력이 무한정 늘어나는 컨텍스트 폭발,
기존 요약 방식이 추적 가능성을 남기지 않는 손실적 압축,
그리고 벡터 스토어의 조각난 조회가 거시적인 지도를 제공하지 못하는
평면적 메모리 문제다.

## 4단계 점진형 파이프라인

메모리를 두 축으로 계층화한다.
단기 컨텍스트 계층은 원본 도구 출력을 담는 L0(Raw, `refs/*.md`),
단계별 요약인 L1(Summary, `jsonl`),
컨텍스트에만 주입되는 경량 Mermaid 다이어그램인 L2(Canvas)로 구성된다.
장기 개인화 계층은 원본 대화인 L0,
원자적 사실(atomic facts)인 L1,
시나리오 블록(scene blocks)인 L2,
사용자 프로필인 L3(Persona)까지 이어진다.

이 구조의 핵심은 “완전 추적성”이다.
높은 수준의 추상화에서 저수준 증거로 언제든 드릴다운할 수 있는
경로를 항상 열어 둔다.
예컨대 L3 Persona에 요약된 사용자 선호가 실제로 어떤 대화에서
비롯됐는지 L0까지 거슬러 확인할 수 있다.

## 주요 기능

### 상징적 메모리와 오프로딩

대량의 텍스트를 고밀도 Mermaid 그래프로 변환해 토큰 비용을 낮추면서도
정보 밀도를 높이는 방식을 사용한다.
전체 도구 로그는 외부 파일에 저장하고,
컨텍스트에는 경량 Mermaid 태스크 맵만 주입한 뒤
에이전트가 필요할 때 `node_id` 참조로 세부 내용을 즉시 조회하게 한다.
OpenClaw와 통합한 실험에서 토큰 사용량이 61.38퍼센트 줄고
작업 성공률이 51.52퍼센트(상대적) 향상됐다고 밝히고 있다.
WideSearch 벤치마크 기준 성공률은 33퍼센트에서 50퍼센트로,
토큰 사용량은 221.31M에서 85.64M으로 줄었고,
PersonaMem 정확도는 48퍼센트에서 76퍼센트로 올랐다고 소개한다.

### 화이트박스 디버깅

중간 결과물을 사람이 읽을 수 있는 파일로 유지한다는 원칙을 강조한다.
L2 시나리오는 일반 Markdown으로,
L3 페르소나는 `persona.md`에 저장되며,
모든 노드는 `result_ref`와 `node_id`로 연결돼 있어
“메모리는 블랙박스가 아니다”라는 문구로 이 특성을 설명한다.

### 하이브리드 검색

BM25 키워드 검색과 임베딩 기반 의미론적 검색을
RRF(Reciprocal Rank Fusion)로 융합하는 검색 전략을 쓴다.
저수준 증거는 데이터베이스에 저장해 전문 검색을 지원하고,
상위 계층 구조는 Markdown 파일로 저장해 정보 밀도와 검사 용이성을
동시에 확보하는 이중 계층 저장 전략을 취한다.

## 사용법

주 통합 대상은 OpenClaw와 Hermes Agent(Nous Research) 두 플랫폼이다.

OpenClaw에서는 플러그인을 설치하고 게이트웨이를 재시작한 뒤
설정 파일(`~/.openclaw/openclaw.json`)에서 `memory-tencentdb.enabled`를
켜는 방식으로 활성화한다.
기본값은 로컬 `SQLite + sqlite-vec` 백엔드이며,
`offload.enabled` 옵션으로 단기 압축 기능을 추가로 켤 수 있다.

Hermes 통합은 두 경로를 제공한다.
신규 설치라면 Docker 이미지를 빌드해 컨테이너로 띄우는 방식을,
기존 Hermes 환경이라면 플러그인 패키지를 받아
`~/.hermes/config.yaml`에 프로바이더를 선언하고
Gateway를 다시 시작하는 방식을 따른다.

설정 파라미터는 세 단계로 나뉜다.
일상적으로 조정하는 레벨 1에는 시간대, 저장소 종류,
회상 전략(`hybrid`/키워드/임베딩), 회상당 최대 결과 수,
몇 턴마다 L1을 추출할지 등이 포함된다.
레벨 2는 워밍업, 유휴 타임아웃, 추출 중복 제거 같은 고급 튜닝을,
레벨 3은 원격 임베딩 설정이나 독립 LLM 모드 같은 전체 파라미터를 다룬다.

기술 스택은 TypeScript가 84.2퍼센트로 대부분을 차지하고
Python 7.5퍼센트, Shell 7.0퍼센트, JavaScript 1.3퍼센트로 구성된다.
라이선스는 MIT이며, 이 글 작성 시점 기준 GitHub 스타 7.8k,
포크 720개를 기록하고 있다.

## 분석

### 왜 벡터 스토어 대신 계층 구조를 택했는가

이 프로젝트가 기존 RAG 방식 벡터 스토어를 부족하다고 보는 지점은
명확하다.
벡터 유사도 검색은 조각난 청크를 반환할 뿐,
그 조각이 전체 맥락에서 어떤 위치에 있는지 알려주지 않는다.
L0부터 L3까지 이어지는 계층 구조는 이 문제에 대한 정면 대응이다.
L3 Persona는 “이 사용자는 이런 성향이다”라는 요약을,
L1 Atom은 “이 사용자가 이렇게 말했다”라는 사실을,
L0 Conversation은 원본 발화 자체를 보존한다.
이는 정보 검색 분야에서 오래된 아이디어인
계층적 요약(hierarchical summarization)을 에이전트 메모리에
적용한 것에 가깝다.
다만 검색 시 최상위 계층만 사용하는 대신 드릴다운을 허용함으로써,
압축으로 인한 정보 손실을 감내하는 대신 손실을 되돌릴 경로를
남겨두는 절충을 택했다.

### Mermaid를 상태 표현 수단으로 쓰는 이유

L2 Canvas 계층에서 Mermaid 다이어그램을 컨텍스트에 주입하는 선택은
단순한 토큰 절약을 넘어선 의미를 지닌다.
LLM은 이미 대량의 Mermaid 문법을 학습 데이터로 접했기 때문에,
그래프 구조를 텍스트로 직렬화하면서도 모델이 구조를 그대로
이해할 수 있다.
자연어 요약이 관계 정보를 압축하는 과정에서 노드 간 연결을
흐릿하게 만드는 것과 달리,
그래프 표현은 “무엇이 무엇에 의존하는가”라는 구조적 정보를
텍스트 형태로 유지한다.
이는 최근 에이전트 프레임워크들이 텍스트 프롬프트 안에
구조화된 중간 표현(intermediate representation)을 끼워 넣는
흐름과 맞닿아 있다.
동시에 노드가 많아질수록 Mermaid 자체도 장황해질 수 있다는
잠재적 한계를 안고 있다.

### 로컬 우선이라는 선택이 노리는 시장

외부 API 의존 없이 로컬 SQLite로 동작한다는 점을 전면에 내세운 것은
기술적 선택이자 시장 포지셔닝이다.
클라우드 벡터 DB 서비스에 종속되지 않고 즉시 사용 가능하다는 점은
개인 개발자나 프라이버시에 민감한 조직에게 매력적이다.
동시에 Tencent Cloud Vector Database로 확장 가능한 경로를 남겨둔
것은, 로컬 버전을 진입 장벽을 낮추는 미끼로 삼고
실제 수익화는 클라우드 백엔드에서 노리는 오픈소스 기업의
전형적인 전략과 일치한다.
SQLite와 sqlite-vec만으로 실사용 규모의 벡터 검색이 얼마나
버틸 수 있는지는 README만으로 판단하기 어렵고,
이는 실제 채택 규모가 커질수록 검증받게 될 지점이다.

### OpenClaw와 Hermes라는 특정 생태계에 대한 의존

이 프로젝트는 범용 메모리 라이브러리가 아니라
OpenClaw와 Hermes라는 특정 에이전트 플랫폼의 플러그인으로
설계돼 있다.
이는 통합 난도를 낮추고 실제 동작을 보장하는 장점이 있지만,
동시에 이 프로젝트의 가치가 해당 플랫폼들의 생태계 크기에
종속된다는 뜻이기도 하다.
범용 API나 SDK 형태가 아니라 특정 게이트웨이 아키텍처를
전제로 설계된 점은, 다른 에이전트 프레임워크 사용자에게는
직접적인 이식이 쉽지 않을 수 있음을 시사한다.

## 비평

### “외부 API 무의존”이라는 문구는 절반의 진실이다

README는 로컬 실행과 외부 API 무의존을 핵심 장점으로 내세우지만,
실제로는 임베딩 단계에서 “로컬 임베딩 또는 OpenAI 호환 API”를
선택할 수 있다고 명시하고 있다.
즉 기본 설정이 아니라 사용자가 로컬 임베딩 모델(BGE-M3 등)을
직접 준비해야 진정한 무의존이 성립한다.
많은 사용자가 손쉬운 경로로 OpenAI 호환 API를 선택할 가능성이
높다는 점을 고려하면,
“로컬”이라는 표현이 실제 기본 배포 시나리오보다
더 강한 인상을 준다.
프로젝트를 처음 접하는 사람은 이 문구만 보고
완전한 오프라인 동작을 기대할 수 있으나,
품질 좋은 임베딩을 로컬에서 얻으려면 별도의 모델 서빙 인프라를
직접 구축해야 한다는 현실적 부담이 뒤따른다.

### 벤치마크 수치의 일반화 가능성이 검증되지 않았다

WideSearch 성공률 51.52퍼센트 향상, 토큰 61.38퍼센트 감소,
PersonaMem 정확도 59퍼센트 향상이라는 수치는 인상적이지만,
어떤 태스크 분포와 어떤 베이스라인 설정에서 측정했는지에 대한
구체적 방법론이 README 수준에서는 드러나지 않는다.
특히 WideSearch와 PersonaMem은 각각 특정 유형의 작업
(광범위 검색, 개인화 기억)에 최적화된 벤치마크로 보이는데,
이런 특화 벤치마크에서의 개선이 일반적인 장기 대화나
코딩 에이전트 시나리오에도 동일한 폭으로 재현될지는 별개의
질문이다.
자체 벤치마크에서 자체 시스템이 개선을 보였다는 서술 구조는
독립적인 제3자 검증을 거치기 전까지는 참고 지표 이상의
확신을 주기 어렵다.

### 계층적 추상화가 오히려 새로운 실패 지점을 만든다

L0에서 L3까지 자동으로 추출되는 파이프라인은,
추출 단계 각각이 LLM 호출에 의존한다는 것을 의미한다.
즉 원자적 사실을 뽑아내는 L1 추출, 시나리오로 묶는 L2 압축,
페르소나로 정제하는 L3 요약 각 단계에서 LLM이 잘못 판단하면
오류가 상위 계층으로 전파된다.
드릴다운 경로가 있다는 것은 사후에 오류를 추적할 수단은
제공하지만, 애초에 잘못된 추상화가 만들어지는 것을
막지는 못한다.
단일 계층 벡터 검색보다 계층이 많아질수록 파이프라인의
복잡도와 장애 지점이 늘어나는데,
이 트레이드오프에 대한 논의는 README에서 찾아보기 어렵다.

### 특정 게이트웨이 생태계 종속이 확장성을 제약한다

OpenClaw와 Hermes라는 두 플랫폼에 대한 통합만 제공하는 현재 구조는,
범용 메모리 계층으로서의 야심과 실제 구현 범위 사이에 간극을
만든다.
LangChain, AutoGen, CrewAI 같은 더 널리 쓰이는 에이전트
프레임워크에 대한 직접 지원이 없다는 것은,
“AI 에이전트를 위한 장기 기억”이라는 포괄적 슬로건과 달리
실제로는 특정 생태계 사용자만 손쉽게 활용할 수 있는 도구임을
뜻한다.
로드맵에 언급된 “포터블 메모리(크로스 에이전트 마이그레이션)”가
이 간극을 메우려는 시도로 보이지만,
현재 시점에서는 계획 단계에 머물러 있다.

## 인사이트

### 메모리 표준화 경쟁은 결국 데이터 락인 경쟁으로 흐를 것이다

TencentDB Agent Memory가 제안하는 L0-L3 계층 구조나
Mermaid 기반 상징적 메모리는, 지금 시점에는 하나의 설계
제안이지만 여러 벤더가 유사한 시도를 내놓기 시작하면
결국 “에이전트 메모리 포맷” 표준화 경쟁으로 이어질 가능성이 높다.
과거 웹 브라우저 쿠키, 모바일 앱의 로컬 스토리지 포맷이
그랬듯, 초기에는 기술적 우수성을 다투는 것처럼 보이지만
실제로는 어떤 포맷이 먼저 생태계를 장악하느냐의 문제로
귀결된다.
Tencent Cloud가 로컬 버전을 무료로 공개하면서 클라우드
백엔드(TCVDB)로의 자연스러운 이행 경로를 만들어 둔 것은,
표준 경쟁에서 유리한 고지를 점하려는 포석으로 읽을 수 있다.
메모리 포맷이 한번 특정 벤더의 스키마에 맞춰 구축되면,
페르소나와 시나리오 데이터를 다른 시스템으로 이관하는 비용은
생각보다 커질 수 있다.
로드맵의 “포터블 메모리”가 실제로 벤더 중립적인 포맷으로
구현되는지, 아니면 Tencent 생태계 내에서의 이동에 그치는지가
이 프로젝트의 진짜 의도를 가늠하는 시험대가 될 것이다.

### 화이트박스를 표방하는 시스템일수록 감사 부담이 사용자에게 전가된다

“메모리는 블랙박스가 아니다”라는 슬로건은 투명성이라는
긍정적 가치를 내세우지만, 동시에 숨은 함의가 있다.
페르소나와 시나리오가 사람이 읽을 수 있는 Markdown 파일로
저장된다는 것은, 그 파일에 담긴 내용이 정확한지, 편향되지
않았는지, 오래돼 무효화되지 않았는지를 검증할 책임 역시
사용자에게 넘어온다는 뜻이다.
블랙박스 시스템은 애초에 사람이 들여다볼 수 없으니 검증
책임 논쟁 자체가 성립하지 않지만,
화이트박스 시스템은 “읽을 수 있는데 확인하지 않았다”는
새로운 형태의 과실 가능성을 만든다.
장기적으로 에이전트가 자동 생성한 `persona.md` 파일이
수백 개 계정에 걸쳐 쌓였을 때, 이를 주기적으로 감사할
인력이나 도구가 없다면 투명성은 실질적 효용보다는
법적·심리적 안도감을 주는 장치로 전락할 위험이 있다.
이는 로그를 남기는 것과 그 로그를 실제로 검토하는 것
사이의 오래된 간극이 메모리 시스템에도 그대로 반복되는
사례다.

### 토큰 절감형 아키텍처는 모델 성능 향상과 경쟁하며 가치가 줄어들 수 있다

61.38퍼센트 토큰 감소라는 수치는 현재 시점의 LLM
컨텍스트 윈도우 비용 구조를 전제로 한 성과다.
그러나 컨텍스트 윈도우가 지속적으로 커지고 긴 컨텍스트에
대한 추론 비용이 하락하는 추세를 감안하면,
“토큰을 아끼기 위해 정보를 그래프로 압축한다”는 설계
동기 자체가 시간이 지날수록 약해질 수 있다.
이는 과거 데이터베이스 업계에서 디스크 I/O를 아끼기 위한
정교한 인덱싱 기법들이, SSD와 메모리 가격이 떨어지면서
일부는 오버엔지니어링으로 재평가받았던 흐름과 유사하다.
다만 이 프로젝트가 노리는 진짜 가치는 토큰 절감보다
추적 가능한 계층적 구조 자체에 있을 가능성이 크므로,
비용 절감이라는 마케팅 포인트가 희석되더라도
화이트박스 구조와 드릴다운 경로라는 설계 철학은
독자적인 존재 이유를 유지할 수 있다.
결국 이 프로젝트의 장기 생존력은 토큰 비용 논리보다
“에이전트의 판단 근거를 사람이 추적할 수 있어야 한다”는
거버넌스 요구가 얼마나 커지느냐에 더 크게 좌우될 것이다.
