# Context Mode

<https://github.com/mksglu/context-mode>

## 소개

Context Mode는 AI 코딩 에이전트의 컨텍스트 윈도우 소비를 줄이는 MCP 서버다.
MCP 도구 호출 시 원시 데이터가 대화 컨텍스트에 직접 쌓이는 구조적 문제를
샌드박스 실행, 세션 지속성, 코드 기반 분석, 출력 압축이라는 네 가지 전략으로 해결한다.
프로젝트가 제시하는 대표 수치는 “315 KB → 5.4 KB, 98% 절약”이다.

Claude Code, Gemini CLI, VS Code Copilot, JetBrains Copilot, Cursor 등 14개
플랫폼을 지원한다.

## 핵심 전략

### 샌드박스 실행

`ctx_execute`를 비롯한 도구들이 코드를 격리된 서브프로세스에서 실행하고
stdout만 컨텍스트에 주입한다.
Playwright 스냅샷(56 KB), GitHub API 응답(59 KB), 접근 로그(45 KB) 같은
원시 데이터는 대화에 노출되지 않는다.

### 세션 지속성

대화가 압축(compaction)되면 모델은 작업 중이던 파일, 열린 태스크, 미해결 오류를
잃는다.
Context Mode는 파일 수정, Git 작업, 태스크 상태, 오류를 SQLite 데이터베이스에
기록하고, 압축 후 재개 시 2 KB 이하의 구조화된 스냅샷으로 복원한다.
다섯 개 훅(PreToolUse, PostToolUse, UserPromptSubmit, PreCompact, SessionStart)이
협력해 상태를 보존한다.

### 코드 기반 분석

LLM이 50개 파일을 읽어 함수 수를 세는 대신, 분석 스크립트를 작성해 결과만
받는다.
10번의 도구 호출을 1번으로 줄이고 컨텍스트 사용량을 약 100배 절약한다.

### 출력 압축

응답에서 조사, 불필요한 수식어, 인사말을 제거하고 기술적 정확성만 유지한다.
출력 토큰을 65-75% 절약한다.

## 지식 베이스 아키텍처

SQLite FTS5에 BM25 랭킹을 적용하고 Porter 어근 분석으로
“running”, “runs”, “ran”을 동일 어근으로 처리한다.
헤딩에는 5배 가중치를 부여한다.

검색은 Porter 어근 색인과 트라이그램 부분 문자열 검색 결과를
Reciprocal Rank Fusion으로 병합한다.
다중 용어 쿼리는 용어가 인접한 단락을 더 높게 평가하는 근접 재랭킹을 적용한다.
오타 보정도 내장되어 있다(“kuberntes” → “kubernetes”).

## 설치

Claude Code에서는 플러그인 마켓플레이스로 설치할 수 있다.

```bash
/plugin marketplace add mksglu/context-mode
/plugin install context-mode@context-mode
```

훅이 런타임에 자동 주입되고 프로젝트에 파일이 기록되지 않는다.
Gemini CLI, VS Code Copilot 등은 설정 파일을 별도로 작성해야 한다.
Zed처럼 훅을 지원하지 않는 플랫폼은 `AGENTS.md`를 프로젝트에 수동으로 복사한다.

## 분석

### 컨텍스트 소비 문제의 구조적 접근

토큰 절약 도구들은 흔히 “압축”에 집중하지만, Context Mode는 애초에
데이터가 컨텍스트에 들어오지 않도록 막는 방향을 택한다.
샌드박스 실행이 핵심인 이유는 여기에 있다.
원시 데이터를 요약하거나 자르는 것보다 처음부터 컨텍스트 진입을 차단하는 것이
근본적으로 더 많은 공간을 확보한다.

### FTS5와 BM25 조합의 실용성

LLM이 검색 레이어로 활용할 수 있는 SQLite FTS5는 외부 벡터 DB 없이
키워드 기반 검색을 구현한다.
BM25는 TF-IDF보다 문서 길이 정규화가 우수해 짧은 청크와 긴 청크가 혼재할 때
더 공정한 랭킹을 제공한다.
에이전트 실행 환경에서 외부 의존성을 최소화하는 설계 원칙과 일치한다.

### 세션 압축 문제에 대한 현실적 해결책

Claude Code의 `/compact` 기능이나 자동 압축은 현재 많은 에이전트 워크플로에서
“작업 맥락 소실”을 야기하는 실질적 문제다.
Context Mode의 SQLite 기반 스냅샷은 이 문제를 모델 수준이 아닌 인프라 수준에서
우회한다.
실제 효용은 세션 지속 시간이 30분에서 3시간으로 늘어난다는 벤치마크에서 드러난다.

## 비평

### 진짜 98% 절약인가

[GeekNews 댓글](https://news.hada.io/topic?id=29106)에서 recast7838과 shakespeares는
“실질적으로 효과적일지 모르겠다”, “한두 번 더 쓰면 토큰을 더 쓰지 않을까”라고
의문을 제기했다.
정당한 우려다.
`ctx_execute`로 도구를 호출하는 행위 자체가 추가 토큰을 소비하고,
SQLite 조회 결과를 컨텍스트에 주입할 때도 비용이 발생한다.
“315 KB → 5.4 KB”는 Playwright 스냅샷 전체를 직접 주입할 때와의 비교로,
에이전트가 항상 그런 원시 출력을 컨텍스트에 쏟아붓지는 않는다.
워크로드에 따라 절약 효과가 크게 달라질 수 있다.

### Claude mem, caveman과의 유사성

duse0001은 “[GeekNews 댓글](https://news.hada.io/topic?id=29106)에서 Claude mem이랑

## 인사이트

### 컨텍스트 윈도우 경쟁의 역설

모델 제공사들은 컨텍스트 윈도우를 지속적으로 확장하고 있다.
그러나 Context Mode 같은 프로젝트가 존재한다는 것은 이 경쟁의 역설을 드러낸다.
컨텍스트가 클수록 유용하지만, 클수록 비용이 높고 추론 속도가 느리다.
“가능한 모든 것을 컨텍스트에 넣어라”는 naive한 접근은 실제 에이전트 워크플로에서
경제적으로 지속 불가능하다.

컨텍스트 최적화는 모델 성능 최적화와 별개의 엔지니어링 문제가 되고 있다.
하드웨어 세계에서 CPU가 빨라져도 캐시 계층 설계가 여전히 중요한 것처럼,
모델 컨텍스트가 커져도 무엇을 언제 컨텍스트에 넣을지 설계하는 일은
별도의 전문 영역이 된다.

이 최적화의 전문화는 하네스 엔지니어링이 독립적인 기술 스택으로 성장하는 한
신호다.
Context Mode는 그 스택에서 “컨텍스트 관리” 레이어를 담당하는 첫 세대 도구 중 하나다.

### 에이전트 인프라의 로컬 우선 원칙

Context Mode가 “아무것도 머신을 벗어나지 않는다”를 설계 원칙으로 명시한 것은
주목할 만하다.
AI 에이전트 시장에서 현재 지배적인 흐름은 클라우드 기반 오케스트레이션이다.
에이전트 실행 로그, 기억, 세션 상태를 클라우드에 저장하고 분석하는 서비스들이
등장하고 있다.

로컬 SQLite 기반 접근은 이 흐름에 대한 의식적인 대안이다.
기업 보안 요구사항, 개인정보 민감 워크로드, 네트워크 격리 환경에서
클라우드 기반 에이전트 인프라는 현실적으로 채택하기 어렵다.
Context Mode의 “로컬 우선” 설계는 이 시장 세그먼트를 겨냥한다.

장기적으로 에이전트 인프라는 클라우드와 로컬 양 극단에서 발전하고,
그 사이에서 하이브리드 아키텍처가 표준이 될 것이다.

### 훅 기반 에이전트 인프라의 가능성

Context Mode의 세션 지속성은 다섯 개 훅이 없으면 구현할 수 없다.
Claude Code의 훅 시스템(PreToolUse, PostToolUse, PreCompact, SessionStart 등)은
에이전트 행동을 모델 외부에서 제어할 수 있는 강력한 확장점이다.

훅 기반 인프라의 핵심 가치는 “모델을 수정하지 않고 동작을 변경한다”는 것이다.
모델 파인튜닝이나 시스템 프롬프트 수정 없이도 에이전트의 컨텍스트 관리,
상태 저장, 보안 정책을 외부에서 주입할 수 있다.
이 패턴은 소프트웨어 공학의 미들웨어나 관점 지향 프로그래밍(AOP)과 유사하다.

Context Mode는 이 훅 시스템을 최대한 활용하는 예시를 통해,
에이전트 플랫폼이 훅 확장점을 얼마나 풍부하게 제공하는가가
생태계 경쟁력의 핵심 척도임을 보여준다.
