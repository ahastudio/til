# SSE vs Streamable HTTP: MCP 전송 프로토콜 비교

원문: <https://brightdata.com/blog/ai/sse-vs-streamable-http>

## 요약

이 글은 Model Context Protocol(MCP)의 전송 계층이 HTTP+SSE에서 Streamable
HTTP로 전환된 배경과 두 방식의 차이를 비교한다. 핵심 주장은, MCP가 프로토콜
버전 2026-03-26부터 HTTP+SSE를 폐기(deprecated)하고 Streamable HTTP를
표준으로 채택한 데에는 세 가지 한계 — 재개 가능한 스트림(resumable stream)
미지원, 영속 연결(persistent connection)이 요구하는 서버 자원 부담, 단방향
메시징 제약 — 가 있었다는 것이다.

기술적으로 Server-Sent Events(SSE)는 클라이언트가 단일 장기 연결(long-lived
connection)을 통해 서버로부터 자동 업데이트를 받는 메커니즘으로, `text/
event-stream` MIME 타입을 사용해 데이터가 서버에서 클라이언트로 단방향으로
흐른다. 반면 Streamable HTTP는 장기 연결 없이 실시간 통신을 가능케 하는
무상태(stateless) 접근으로, 단일 엔드포인트에서 표준 HTTP POST와 GET 요청을
쓰며 필요에 따라 SSE를 선택적으로 결합해 고급 스트리밍을 지원한다.

진화 과정은 다음과 같다. 2024년 11월 MCP가 HTTP+SSE 전송과 함께 출시됐고,
프로토콜 버전 2024-11-05의 SSE 기반 아키텍처는 두 개의 엔드포인트(스트리밍용
GET, 클라이언트 메시지용 POST)를 요구했다. 서버는 먼저 endpoint URI 이벤트를
보내고, 이후 모든 통신이 그 지정 경로로 라우팅된다. 버전 2026-03-26에서
Streamable HTTP가 도입되며 HTTP+SSE는 폐기됐지만 하위 호환은 유지된다.
Streamable HTTP는 POST와 GET을 모두 지원하는 단일 엔드포인트를 쓰고, 서버가
스트림 기반 ID를 “각 스트림 내의 커서(cursor)”로 부여해 연결 재개와 메시지
재전송(redelivery)을 가능케 한다.

비교상 HTTP+SSE의 강점은 부분 결과(partial results)를 즉시 전송해 지연을
피하고, 클라이언트에게 알리는 비요청(unsolicited) 서버 이벤트를 지원하며,
특별한 프로토콜 없이 표준 HTTP만 쓴다는 점이다. 한계는 단방향이라 별도 POST
요청이 필요하고 열린 연결 유지가 서버 자원을 많이 소모한다는 것이다.
Streamable HTTP의 강점은 상시 장기 연결이 불필요하고, 일반적인 HTTP 미들웨어·
프록시·호스팅 플랫폼과 호환되며, 무상태 서버를 지원하고, 선택적 SSE 업그레이드
경로를 제공한다는 점이다. 글은 또 MCP가 WebSocket을 채택하지 않은 이유로 단순
RPC 호출에 과한 운영 부담, Authorization 헤더를 못 싣는 브라우저 제약, GET
전용 업그레이드 요구가 POST 기반 흐름을 복잡하게 만드는 점을 든다. 결론은
Streamable HTTP가 무상태·인프라 친화적 배포를 가능케 하는 아키텍처적 진보이며,
2024년 11월에 나온 MCP는 아직 “매우 어려서(very young)” 추가 진화가 예상된다는
것이다.

## 분석

### 글의 핵심 주장은 “전송 계층의 무상태화”이고 그 토대는 인프라 적합성이다

이 글의 중심 논리는 기술적 우아함이 아니라 운영 현실에 있다. HTTP+SSE의
폐기 사유로 제시된 세 가지 — 재개 불가, 자원 부담, 단방향 — 중 실질적 무게
중심은 두 번째다. 장기 연결을 유지하는 비용이 문제의 핵심이고, Streamable
HTTP의 강점으로 나열된 항목들(“상시 연결 불필요”, “일반 미들웨어·프록시·
호스팅과 호환”, “무상태 서버 지원”)은 모두 같은 한 가지를 말한다. 기존
HTTP 인프라 위에 그대로 얹힌다는 것이다.

이 주장이 기대는 전제는 “서버리스와 매니지드 인프라가 기본값인 세계”다.
장기 연결은 로드 밸런서, API 게이트웨이, 서버리스 함수의 수명 모델과
충돌한다. AWS Lambda는 실행 시간 제한이 있고, 다수의 PaaS는 idle 연결을
끊으며, 수평 확장 환경에서 특정 인스턴스에 묶인 SSE 연결은 세션 어피니티
문제를 낳는다. 글이 명시적으로 말하지 않지만, Streamable HTTP로의 전환은
“MCP 서버를 클라우드 네이티브 방식으로 배포하고 싶다”는 요구의 직접적
산물이다.

따라서 이 글은 프로토콜 비교를 표방하지만 실제로는 배포 모델의 변화를
서술한다. 전송 방식의 선택이 추상적 기술 토론이 아니라 “어디에 어떻게
호스팅할 것인가”라는 인프라 결정에 종속된다는 점이, 글이 드러내는 가장
중요한 구조다.

### 양방향성이라는 표의 주장은 SSE의 본질과 충돌한다

비교표는 HTTP+SSE를 “Server → Client” 단방향으로, Streamable HTTP를
“Bidirectional”로 분류한다. 그러나 이 분류는 미묘하게 오해를 부른다.
Streamable HTTP의 양방향성은 단일 TCP 연결 위의 전이중(full-duplex)
통신이 아니다. 클라이언트는 여전히 별도의 HTTP POST로 메시지를 보내고,
서버는 그 POST의 응답을 단발 JSON으로 주거나 SSE 스트림으로 업그레이드해
돌려준다. 즉 “양방향”은 하나의 엔드포인트가 양쪽 방향의 요청을 받는다는
의미이지, WebSocket 같은 실시간 양방향 채널을 뜻하지 않는다.

이 구분이 중요한 이유는, Streamable HTTP가 SSE를 버린 것이 아니라 SSE를
선택적 구성 요소로 흡수했기 때문이다. 글 스스로 “선택적 SSE 업그레이드
경로”를 강점으로 든다. 다시 말해 Streamable HTTP는 SSE의 대체재가 아니라
SSE를 포함하는 상위 프레임이다. 서버가 POST 응답에 단발 JSON을 주면 SSE가
전혀 등장하지 않고, 스트리밍이 필요하면 같은 자리에서 `text/event-stream`
으로 응답한다.

그렇다면 “SSE vs Streamable HTTP”라는 제목 자체가 약간의 범주 오류를 안고
있다. 둘은 대등한 경쟁자가 아니라, 후자가 전자를 부분집합으로 거느린다.
글은 이 포함 관계를 강점 목록에 한 줄로 묻어두지만, 사실 이것이 전환의
핵심 설계 결정이다. 기존 SSE 구현을 버리지 않으면서 무상태 경로를 새로
열어준 것이다.

### “Resumability: No / No”라는 표가 전환 명분을 스스로 무너뜨린다

글이 제시한 비교표에서 가장 자기모순적인 칸은 재개 가능성(Resumability)
행이다. HTTP+SSE도 “No”, Streamable HTTP도 “No”로 표기돼 있다. 그런데
글의 도입부는 HTTP+SSE 폐기의 첫 번째 사유로 “재개 가능한 스트림 미지원”을
들었고, Streamable HTTP 아키텍처 설명에서는 서버가 스트림 ID를 “커서”로
부여해 “연결 재개와 메시지 재전송”을 가능케 한다고 명시했다.

본문과 표가 정면으로 어긋난다. 본문의 서술이 MCP 명세에 부합한다.
2026-03-26 명세의 Streamable HTTP는 SSE 이벤트의 `id` 필드와
`Last-Event-ID` 헤더를 활용한 재개를 정의하며, 이것이 바로 HTTP+SSE 대비
개선점으로 내세워진 기능이다. 표의 “No/No”는 명백한 오류다.

이 모순이 분석에서 중요한 이유는, 그것이 글의 신뢰도가 아니라 글의 생산
방식을 드러내기 때문이다. 본문은 명세를 읽고 정확히 서술했는데 요약표는
틀렸다는 것은, 표가 본문과 독립적으로 — 아마도 일반화된 SSE 지식이나
자동 생성에 기대 — 채워졌음을 시사한다. 기술 비교 글에서 요약표는 독자가
가장 먼저, 가장 오래 보는 자산인데, 바로 그 자산이 핵심 차별점을 거꾸로
적었다.

## 비평

### 전환의 진짜 동인은 자원 부담이 아니라 인프라 마찰인데 글은 그 둘을 뒤섞는다

글은 HTTP+SSE의 한계로 “열린 연결이 서버 자원을 많이 소모한다”를 전면에
내세운다. 그러나 이 프레이밍은 2024년 이후의 기술 현실에서 가장 약한
논거다. 단일 서버가 유지할 수 있는 동시 SSE 연결 수는 nginx나 Go 서버
기준으로 수만에서 수십만에 이르고, C10K 문제는 20년 전에 해결됐다. MCP가
다루는 대상이 LLM 에이전트와 도구 서버 간 연결, 즉 수천 단위의 세션이라면
“자원 소모”는 실질적 병목이 아니다.

진짜 동인은 자원량이 아니라 인프라 마찰이다. 장기 연결은 그 자체로 가볍지만,
서버리스 함수의 타임아웃, 로드 밸런서의 idle 타임아웃, 오토스케일링 시의
연결 드레이닝, 스테이트풀 세션의 인스턴스 어피니티 같은 운영 환경의 가정과
충돌한다. 문제는 “연결이 무겁다”가 아니라 “현대 배포 파이프라인이 장기
연결을 일급 시민으로 다루지 않는다”는 것이다. 글이 강점 목록에 적은 “일반
미들웨어·프록시·호스팅과 호환”이 사실 핵심 사유인데, 정작 한계 분석에서는
그것을 자원 소모라는 다른 언어로 바꿔 말한다.

이 혼동은 사소하지 않다. 자원 소모가 사유라면 해법은 더 효율적인 SSE 서버다.
인프라 마찰이 사유라면 해법은 무상태화, 즉 Streamable HTTP다. 글은 후자의
해법을 옹호하면서 전자의 문제 진단을 제시한다. 진단과 처방이 어긋난 채로
결론만 옳은 셈인데, 이 어긋남은 독자가 “내 환경에서도 전환이 필요한가”를
판단할 근거를 흐린다. 자체 서버를 운영하는 팀에게 자원 소모는 무시할 수준일
수 있고, 그렇다면 글의 논리대로라면 전환할 이유가 없어지기 때문이다.

### 보안과 인증을 전송 비교에서 통째로 빠뜨렸다

WebSocket을 다룰 때 글은 “브라우저가 Authorization 헤더를 못 싣는다”는
이유를 든다. 그런데 정작 SSE와 Streamable HTTP의 비교에서는 인증과 보안이
전혀 등장하지 않는다. 이것은 MCP 전송 선택에서 가장 실무적으로 중요한
차원을 누락한 것이다.

원격 MCP 서버의 등장은 인증·인가(OAuth 2.1, 토큰 흐름)와 세션 보안을 전송
선택의 1급 고려사항으로 만들었다. 2026-03-26 명세 자체가 인증 프레임워크를
크게 보강했고, DNS 리바인딩 공격이나 세션 하이재킹 같은 위협이 원격 전송에서
실재한다. SSE의 단일 장기 연결과 Streamable HTTP의 다수 단발 요청은 토큰
갱신, 세션 만료, 재인증 처리 방식이 근본적으로 다르다. 단발 요청 모델은 매
요청마다 인증을 검증할 수 있는 반면, 장기 연결은 연결 수립 시점의 인증이
세션 내내 유지되는 가정에 기댄다.

전송 비교 글이 이 차원을 빠뜨린 것은 단순한 누락이 아니라 비교의 골격을
왜곡한다. 무상태성을 강점으로 칭송하면서, 무상태성이 인증에서 갖는 함의 —
매 요청 인증의 오버헤드, 토큰 재사용 패턴, 세션 ID와 인증 토큰의 분리 —
는 침묵한다. 실무자가 전송을 고를 때 가장 먼저 부딪히는 벽이 인증인데, 글은
그 벽을 보이지 않는 곳에 둔 채 확장성과 인프라 호환성만 비교한다.

### “무상태 우월”이라는 결론은 스트리밍 워크로드의 비용을 숨긴다

글의 결론은 Streamable HTTP가 무상태 배포를 가능케 하는 아키텍처적 진보라는
것이다. 그러나 무상태성은 공짜가 아니다. 장기 연결을 매 요청 단발 모델로
대체하면, 진짜 스트리밍이 필요한 워크로드 — LLM의 토큰 단위 응답 스트리밍,
장기 실행 도구의 진행률 알림, 서버 주도 푸시 — 에서는 결국 SSE 업그레이드로
되돌아가야 한다.

여기서 표면적 단순함과 실제 복잡성이 갈린다. 무상태 단발 요청은 짧은 RPC
호출에는 이상적이지만, MCP의 매력적 사용 사례 다수가 스트리밍이다. 에이전트가
파일을 읽으며 부분 결과를 받고, 긴 작업의 중간 상태를 받고, 서버가 능동적으로
알림을 보내는 시나리오들이다. 이런 워크로드에서 Streamable HTTP는 SSE를
다시 켜고, 그 순간 글이 폐기 사유로 들었던 장기 연결의 문제가 그대로
돌아온다. 무상태는 기본값일 뿐, 스트리밍 사용 사례에서는 상태가 다시
들어온다.

글이 숨기는 트레이드오프는 이것이다. Streamable HTTP는 “필요 없으면 연결을
안 열어도 된다”는 선택지를 줄 뿐, 스트리밍이 필요한 순간의 복잡성을
없애주지 않는다. 오히려 두 모드(단발 JSON 응답 vs SSE 스트림)를 모두
구현하고 둘 사이를 올바르게 전환해야 하므로, 서버 구현의 분기는 늘어난다.
“무상태 우월”이라는 단정은 단발 RPC 중심 워크로드에서만 성립하며, MCP가
지향하는 풍부한 에이전트 상호작용에서는 절반의 진실이다. 글이 결론에서
이 조건을 명시하지 않은 채 일반화한 것은, 독자에게 전환의 이득만 보여주고
구현 비용은 가린 것이다.

## 인사이트

### 전송 프로토콜의 무상태화는 MCP 서버를 “API 상품”으로 만드는 사전 작업이다

이 전환의 2차 효과는 기술이 아니라 시장에 있다. 무상태 HTTP 전송으로의
이동은 MCP 서버를 기존 REST API와 동일한 운영·과금·배포 모델 위에 올려놓는다.
장기 연결은 사용량 기반 과금과 궁합이 나쁘다. 연결이 열려 있는 동안 무엇을
청구할지 모호하기 때문이다. 반면 단발 요청은 호출당 과금, 레이트 리밋,
API 키 관리, 게이트웨이 분석이라는 잘 정립된 SaaS 인프라를 그대로 쓸 수 있다.

이 글이 Bright Data — 데이터 수집을 API로 파는 회사 — 의 블로그에 실렸다는
사실이 이 인사이트를 뒷받침한다. Streamable HTTP가 “일반 미들웨어·프록시·
호스팅과 호환”된다는 강점은, 곧 MCP 서버를 상용 API 게이트웨이 뒤에 세워
판매할 수 있다는 뜻이다. 전송 계층의 무상태화는 MCP를 실험적 에이전트
프로토콜에서 상거래 가능한 API 제품으로 전환하는 인프라적 전제 조건이다.

길게 보면 이것은 MCP 생태계의 분기를 예고한다. 로컬 `stdio` 전송으로 도는
개인용 도구 서버와, 무상태 HTTP로 도는 상용 호스팅 MCP 서버는 점점 다른
세계가 된다. 후자는 인증·과금·SLA를 갖춘 제품이 되고, 전자는 설정 파일에
적는 로컬 바이너리로 남는다. 전송 방식의 선택이 곧 사업 모델의 선택이 되는
것이며, 2026-03-26의 무상태화 결정은 상용화 진영의 손을 들어준 셈이다.

### MCP 전송의 역사는 RPC 프로토콜이 반복해 온 진자 운동을 그대로 답습한다

MCP가 18개월 만에 SSE에서 Streamable HTTP로 옮긴 것은 새로운 사건이 아니라
오래된 패턴의 재현이다. 분산 시스템의 RPC 전송은 “풍부하지만 무거운 상태
유지 연결”과 “단순하지만 제약 있는 무상태 요청” 사이를 반복해 진동해 왔다.
CORBA와 DCOM의 스테이트풀 객체 참조에서 REST의 무상태 자원 모델로, 다시
gRPC의 HTTP/2 스트리밍으로, 그리고 GraphQL subscription의 WebSocket으로.
매번 같은 긴장 — 실시간성과 운영 단순성의 충돌 — 이 다른 이름으로 돌아온다.

이 역사적 렌즈로 보면 MCP의 선택은 REST가 SOAP을 이긴 논리의 반복이다.
SOAP은 기술적으로 더 풍부했지만, REST는 기존 HTTP 인프라에 그대로 올라타는
운영 단순성으로 승리했다. Streamable HTTP가 SSE에 대해 갖는 우위도 정확히
같은 종류다. 더 우아해서가 아니라 더 배포하기 쉬워서다. 프로토콜 설계에서
“기존 인프라와의 마찰 없음”은 거의 항상 기술적 완결성을 이긴다.

이 패턴이 예고하는 것은 다음 진자의 방향이다. 무상태화가 한 번 승리하면,
이후 실시간 협업이나 양방향 에이전트 통신 같은 요구가 커지면서 다시 상태
유지 연결로의 압력이 쌓인다. GraphQL이 무상태 쿼리에서 출발해 subscription을
덧붙였듯, MCP도 무상태 Streamable HTTP를 기반으로 하되 고급 사용 사례에서
점점 더 SSE 업그레이드와 세션 상태에 의존하게 될 것이다. 진자는 멈추지 않고,
글이 말한 “MCP는 아직 어리다”는 다음 진동을 예고하는 문장으로 읽어야 한다.

### 표준을 18개월 만에 폐기한 속도가 생태계 전체의 숨은 비용을 만든다

글은 하위 호환성이 유지되므로 전환이 매끄럽다고 안심시키지만, 이는 명세
수준의 진실일 뿐 생태계 수준의 진실이 아니다. 2024년 11월에 HTTP+SSE로 구현된
MCP 서버와 클라이언트, SDK, 튜토리얼, 블로그 글, 그리고 그것을 학습한 LLM의
지식은 모두 18개월 만에 “폐기된 방식”이 됐다. 하위 호환은 옛 방식이 여전히
작동함을 보장할 뿐, 옛 방식으로 쌓인 지식과 코드가 최선의 경로가 아니게 됐다는
사실은 바꾸지 못한다.

이 빠른 폐기의 숨은 비용은 학습 자료와 도구의 파편화에서 나타난다. 어떤
튜토리얼은 dual-endpoint SSE를, 어떤 것은 single-endpoint Streamable HTTP를
가르친다. 구버전 SDK로 작성된 서버와 신버전 클라이언트의 조합, 또는 그 반대
조합에서 미묘한 호환성 문제가 생긴다. 무엇보다 코드 생성 AI가 두 시대의 코드를
섞어 학습해, 폐기된 패턴을 최신 패턴으로 제안하는 오염이 발생한다. 이 글의
비교표가 본문과 어긋나 재개 가능성을 거꾸로 적은 것 자체가, 그런 지식 오염의
한 증상일 수 있다.

더 깊은 긴장은 프로토콜의 성숙 속도와 생태계의 채택 속도가 어긋난다는 데
있다. 프로토콜은 명세를 한 줄 고쳐 버전을 올리면 되지만, 그 위에 쌓인 수천
개의 구현은 같은 속도로 따라오지 못한다. “아직 어리니까 빠르게 진화한다”는
서술은 설계자의 관점에서는 미덕이지만, 그 위에 제품을 올린 사람의 관점에서는
리스크다. 18개월 주기의 표준 교체가 반복되면, MCP 위에 진지한 투자를 하려는
기업은 “지금 구현한 것이 다음 버전에서 또 폐기되지 않는가”를 매번 계산해야
한다. 빠른 진화는 초기 생태계를 풍부하게 만들지만, 동시에 그 생태계가
프로덕션 신뢰를 얻는 시점을 계속 뒤로 미루는 양날의 검이다.
