# 메모장만으로 MCP 서버 작성하기

원문: <https://hackers.pub/@gnh1201/2026/메모장-만으로-mcp-서버-작성하기>

## 요약

저자 gnh1201은 외부 런타임(Node.js, Python)과
IDE(VSCode, IntelliJ) 설치가 모두 금지된 극단적으로 제한된
기업 클라이언트 환경에서 MCP(Model Context Protocol) 서버를 구축한 경험을 공유한다.
개발 머신에서 사용 가능한 도구는 Windows 메모장과 Notepad++뿐이었으며,
Claude Desktop만 AI 도구 예외로 허용되었다.

핵심 통찰은 MCP가 JSON-RPC 2.0 기반 표준 입출력 통신에 의존한다는 점이다.
이는 외부 도구가 필수가 아님을 의미한다.
Windows에 기본 내장된 cscript 엔진과 WelsonJS JavaScript 프레임워크를 결합하면
별도 런타임 설치 없이 MCP 서버를 구현할 수 있다.

저자는 OS 네이티브 런타임만으로 동작하는 MCP 서버 구현에 성공했다.
이는 외부 개발 도구 설치가 극도로 제한되는 보안 민감 환경이나
엔터프라이즈 클라이언트 환경에서도 MCP를 활용할 수 있음을 보여준다.

## 분석

### MCP 프로토콜의 구조적 단순성

MCP의 핵심 전송 방식은 JSON-RPC 2.0을 stdin/stdout으로 주고받는 것이다.
이는 의도적으로 단순화된 설계로,
"어떤 언어로 어떤 런타임에서 구현하든 무관하다"는 것이 프로토콜의 약속이다.
저자는 이 약속을 글자 그대로 받아들여 가장 제한된 환경에서 검증한 셈이다.

### Windows의 잊혀진 스크립트 호스트

cscript는 Windows Script Host(WSH)의 명령줄 버전으로
Windows 95 시대부터 OS에 기본 내장되어 있다.
JScript, VBScript를 별도 설치 없이 실행할 수 있으며,
COM 객체를 통해 파일 시스템과 시스템 API에 접근할 수 있다.
WelsonJS는 이 WSH 환경 위에 모던 JavaScript 개발 패턴을 얹는 프레임워크다.

### 기업 환경의 현실

GeekNews 댓글에서 kaydash가 "그런 기괴한 회사가 있다고?"라고 반응했지만,
금융, 국방, 정부 기관에서는 외부 소프트웨어 설치가
보안 정책으로 엄격히 통제되는 경우가 흔하다.
이런 환경에서 일하는 개발자에게 OS 내장 도구는 마지막 보루다.

## 비평

### 강점

프로토콜의 본질을 정확히 이해하고 가장 제약된 환경에서 검증했다는 점이 핵심 가치다.
JSON-RPC 2.0과 stdin/stdout이라는 본질적 구성 요소만 있으면
어디서든 MCP 서버가 동작 가능하다는 것을
이론이 아닌 실증으로 보여주었다.

제약을 한계가 아닌 출발점으로 삼는 자세도 주목할 만하다.
"Node.js를 설치하게 해달라"고 요구하는 대신
주어진 도구로 문제를 해결하는 접근은 엔지니어링의 본령에 가깝다.

### 약점

WelsonJS와 cscript의 구체적 사용법이 문서에서 충분히 설명되지 않아,
같은 환경에 처한 다른 개발자가 곧바로 재현하기는 어렵다.
WelsonJS 자체에 대한 사전 지식이 필요하며,
JScript의 한계(현대 JavaScript 기능 부재)도 별도로 학습해야 한다.

또한 이 접근의 실용성에는 한계가 있다.
MCP 도구의 진짜 가치는 외부 API 호출, 파일 처리, 데이터베이스 접근 같은
"기능 풍부한 도구"인데,
JScript/cscript 환경에서는 이런 기능 구현이 매우 제한적이다.
저자가 어떤 종류의 MCP 도구를 구현했는지가 중요한 정보지만 명확하지 않다.

## 인사이트

### 전송 계층 독립성이 만드는 적응력

MCP가 stdin/stdout 위에 JSON-RPC를 얹은 것은
기술적으로는 가장 단순한 선택이지만 전략적으로는 매우 영리한 결정이다.
HTTP, gRPC, WebSocket 같은 더 "현대적인" 전송 계층을 채택했다면
브라우저 환경, 모바일, 서버리스 등에서는 잘 동작했겠지만
저자가 처한 것 같은 제약 환경에서는 사용할 수 없었을 것이다.

stdin/stdout은 모든 OS의 가장 오래된 추상화이자 가장 보편적인 인터페이스다.
유닉스 파이프라인의 전통을 잇는 이 선택 덕분에
Windows cscript, BusyBox, 심지어 임베디드 환경에서도 MCP가 동작할 수 있다.
"가장 단순한 것이 가장 멀리 간다"는 원칙의 사례다.

이 결정은 MCP가 LSP(Language Server Protocol)에서 영감을 받은 부분이기도 하다.
LSP가 다양한 에디터와 언어를 연결할 수 있었던 핵심 이유가
바로 stdin/stdout JSON-RPC라는 최소 요구사항이었다.
MCP가 같은 패턴을 채택한 것은 우연이 아니다.

### 기술적 갈라파고스 환경의 적응 패턴

극도로 제약된 기업 환경은 일종의 "기술적 갈라파고스"다.
외부 생태계의 진화와 단절된 채 OS 내장 도구만으로 진화해야 한다.
이 환경에서 살아남은 도구들(VBScript, batch, PowerShell)은
바깥 세계에서는 거의 잊혀졌지만 안에서는 여전히 핵심 자산이다.

저자의 사례는 이런 갈라파고스 환경에 새로운 기술을 도입할 때의 패턴을 보여준다.
신기술 자체를 들여올 수 없으니
기존 도구를 새로운 방식으로 조합해 신기술의 효과를 재현하는 것이다.
WelsonJS가 WSH 위에 모던 JavaScript 개발 패턴을 재구현한 것도 같은 맥락이다.

이 패턴은 기술 격차가 큰 환경에서 일하는 개발자에게 시사하는 바가 크다.
"우리 회사는 후진적이라 X를 못 쓴다"는 체념 대신
"가진 도구로 X의 효과를 어떻게 달성할까"라는 질문이
실제로 더 흥미로운 엔지니어링 문제를 만들어낸다.
제약은 창의성의 적이 아니라 동력이 되기도 한다.

### OS 내장 런타임이라는 간과된 플랫폼 계층

현대 개발 문화는 "필요한 것은 npm/pip로 설치한다"를 전제한다.
이 전제 때문에 OS가 기본 제공하는 런타임은 거의 잊혀졌다.
Windows의 WSH, macOS의 AppleScript와 osascript,
모든 유닉스의 POSIX shell과 awk는
"실제로 어디서나 동작하는" 진짜 보편 런타임이지만
신규 프로젝트에서 선택지로 거의 고려되지 않는다.

이 망각의 비용은 환경 의존성의 폭증이다.
간단한 자동화 스크립트조차 Node.js 18 이상,
특정 npm 패키지, 특정 Python 가상환경을 요구한다.
그 결과 5년만 지나도 동작하지 않는 스크립트가 쌓인다.
반면 OS 내장 도구로 작성된 것은 수십 년 단위로 동작한다.

저자의 실험은 이 망각된 계층의 가치를 재발견한다.
새로운 프로토콜(MCP)이 가장 오래된 인터페이스(cscript) 위에서 동작한다는 것은
역설이 아니라 디자인의 우아함을 증명한다.
AI 시대에 새로 만들어지는 프로토콜과 도구들이
OS 내장 런타임 호환성을 의식적으로 유지한다면
"설치 지옥" 없는 진정한 보편 도구로 자리잡을 수 있다.
