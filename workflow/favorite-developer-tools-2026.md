# 개발자들이 2026년에 선택한 도구들

원문: <https://lobste.rs/s/2jdvxa/what_are_some_your_favourite_developer>

## 요약

Lobste.rs의 “당신이 좋아하는 개발자 도구는 무엇인가요?” 스레드에서 커뮤니티가 선택한 도구들이 드러났다.
에디터/IDE 영역에서는 Helix, Neovim, Emacs가 강세를 보였다.
Helix는 설정 없이도 좋은 기본값을 제공한다는 점에서 주목받았고,
Emacs는 Magit이라는 Git 인터페이스 하나만으로도 충분하다는 의견이 많았다.

버전 관리와 환경 관리에서는 Jujutsu(`jj`)와 Mise가 주목을 받았다.
Jujutsu는 “git CLI보다 강력하면서도 동시에 더 쉬운” 도구로 평가받았고,
Mise는 `asdf`와 `direnv`를 하나로 합친 대안으로 빠르게 자리를 잡았다.

셸과 터미널 영역에서는 Fish 셸이 압도적인 지지를 받았다.
설정 없이도 탭 완성과 자동 제안이 작동한다는 점이 핵심이었다.
터미널 에뮬레이터로는 Ghostty와 WezTerm이 언급됐다.

유틸리티 영역에서는 `fzf`, `ripgrep`, `jq`, `bat`, `fd`, `delta`, `atuin`이 반복적으로 등장했다.
이 도구들은 기존 Unix 도구들을 현대적으로 재해석한 것들로,
더 나은 기본값과 색상 출력, 빠른 성능을 제공한다.

스레드 전반에 걸쳐 드러난 철학은 하나였다.
“나이가 들수록 끝없이 설정을 만지기보다 의도적으로 좋은 기본값을 가진 도구에 맞추는 편이 낫다.”

## 분석

### 도구 선택의 철학적 분기

이 스레드에서 두 개의 철학이 충돌한다.
하나는 “기본값 우선주의”다. 설정 없이 바로 잘 동작하는 도구를 선호하고,
커스터마이징에 쏟는 시간을 실제 작업에 투자한다는 입장이다.
다른 하나는 “장인 정신”이다. 도구를 자신에게 맞게 깎고 다듬는 것이 장기적으로 더 높은 생산성을 가져온다는 입장이다.

흥미롭게도 두 진영 모두 상대를 인정한다.
기본값 파는 커스터마이징 파가 “한번 제대로 설정하면 유지 보수가 적다”는 점을 인정하고,
커스터마이징 파는 “기본값이 좋은 도구는 그만큼 설계가 뛰어나다”고 말한다.

### CLI 르네상스의 구조

`fzf`, `ripgrep`, `bat`, `fd`, `delta` 같은 도구들은 단순한 유틸리티가 아니다.
이들은 기존 Unix 도구의 핵심 아이디어를 유지하면서 현대적으로 재설계된 것들이다.
`grep` → `ripgrep`, `find` → `fd`, `cat` → `bat`처럼
원형을 유지하면서 속도, 기본값, 출력 품질을 향상시켰다.

이 도구들이 공통적으로 채택하는 원칙이 있다.
색상 출력, `.gitignore` 자동 존중, 사람이 읽기 좋은 오류 메시지, 빠른 속도.
이 원칙들은 이미 2010년대에 존재했지만, 2020년대 들어서야 커뮤니티 표준으로 정착했다.

### 환경 관리의 수렴

언어 버전 관리 도구의 역사는 분열과 수렴의 반복이었다.
각 언어마다 `rbenv`, `pyenv`, `nvm`, `sdkman`이 따로 존재하다가
`asdf`가 이들을 하나로 통합했고, 이제 `mise`가 `asdf`와 `direnv`를 함께 흡수했다.
이 수렴 과정은 언제나 “설정이 더 간단해진다”는 방향으로 움직인다.

## 비평

### 강점

이 스레드는 특정 기술 스택에 편향되지 않은 교차 언어 관점을 제공한다.
Go, Rust, Python, Ruby, Haskell 개발자들이 동시에 참여해 공통분모를 드러낸다.
특히 언어를 넘어 공유되는 도구들(`fzf`, `jq`, `ripgrep`)이 뚜렷이 드러난다는 점에서
“좋은 도구란 무엇인가”에 대한 실용적 정의를 내린다.

### 약점

Lobste.rs라는 플랫폼의 특성상 커뮤니티 편향이 강하다.
Unix/Linux 터미널 지향 개발자가 압도적이고, GUI 도구나 IDE 중심 워크플로는 과소 대표된다.
VS Code가 전 세계 개발자의 주류 에디터임에도 스레드에서 “아직도 쓰나요?” 취급을 받는다.
Windows 개발자나 엔터프라이즈 환경의 개발자 관점은 거의 없다.

AI 기반 개발 도구에 대한 언급이 거의 없다는 점도 인상적이다.
GitHub Copilot, Claude Code, Cursor 같은 도구들이 2026년 개발 환경에서 무시할 수 없는
위치를 차지하고 있음에도 커뮤니티의 관심 밖에 있다.

## 인사이트

### 도구의 “기본값”은 그 도구의 철학을 담는다

기본값이란 단순한 초기 설정이 아니다.
그것은 제작자가 “일반적인 사용자가 가장 많이 원하는 것이 이것이다”라고 내린 판단이다.
`ripgrep`이 `.gitignore`를 기본으로 존중하는 것은 기술적 결정이 아니라 철학적 결정이다.
“개발자는 대부분 Git 저장소 안에서 검색한다”는 관찰을 기본값으로 구체화한 것이다.

도구를 선택할 때 기본값을 보는 것은 그 도구의 세계관을 보는 것과 같다.
기본값이 나의 작업 방식과 일치하면 설정에 쏟는 에너지가 줄어들고,
그 에너지가 실제 문제로 향한다.
이것이 “기본값 우선주의”가 단순한 게으름이 아니라 전략적 선택인 이유다.

### 도구 생태계는 수렴과 분기를 반복한다

언어 버전 관리는 분열 → `asdf` 수렴 → `mise` 재수렴의 경로를 걸었다.
터미널 에뮬레이터는 xterm → iTerm2 → Alacritty/Kitty → Ghostty/WezTerm으로 계속 분기한다.
에디터는 Vim/Emacs 양극 → VSCode 수렴 → Helix/Zed 재분기 중이다.

이 패턴이 반복되는 이유가 있다.
도구가 성숙해질수록 사용자의 요구가 다양해지고,
하나의 도구로 모든 사용자를 만족시키기 어려워진다.
반면 복잡성이 지나치게 증가하면 단순함을 추구하는 새로운 도구가 등장한다.
이 사이클 자체가 오픈소스 생태계의 건강함을 보여주는 지표다.

### 커뮤니티가 선택한 도구는 그 커뮤니티의 가치관을 반영한다

Lobste.rs 스레드에서 AI 보조 도구가 거의 언급되지 않은 것은 우연이 아니다.
이 커뮤니티는 도구에 대한 깊은 이해와 통제를 중요하게 여긴다.
“마법처럼 동작하지만 내부를 알 수 없는 도구”보다
“예측 가능하고 조합 가능한 작은 도구들”을 선호한다.

이 선호는 AI 도구에 대한 거리감으로 이어진다.
AI 도구는 기본적으로 블랙박스이고, 결과를 예측하기 어렵고, 조합하기도 까다롭다.
커뮤니티가 침묵으로 선택한 것은 단순한 무관심이 아니라
자신들의 가치관과 맞지 않는 도구에 대한 조용한 거부일 수 있다.
