# React 컴파일러 18개월의 회고

원문: <https://saschb2b.com/blog/react-compiler-year-in-review>

## 요약

Sascha Becker의 글은 React 컴파일러가 2024년 10월 공개 베타 출시부터 2025년 10월 1.0 릴리스와 그 이후까지 18개월에 걸쳐 어떻게 발전했는지를 추적한다.
컴파일러의 가장 중요한 기여는 “버그 전체 범주를 제거한 것”이라고 저자는 표현한다.
“`useCallback` 의존성을 빠뜨렸다”는 대화가 더 이상 필요 없어졌다는 것이다.
`useMemo`, `useCallback`, `React.memo`를 통한 수동 메모이제이션은 컴파일러가 자동으로 최적화를 처리함으로써 불필요해진다.

채택 흐름은 네 단계로 구분된다.
2024년 말 Next.js, Expo, TanStack Start 같은 프레임워크들이 빌드 파이프라인에 통합하며 새 프로젝트의 기본 설정이 된 출시·통합 단계, 팀들이 보수적으로 채택하면서 코드 리뷰에서 리렌더 버그 감소와 성능 디버깅 축소라는 조용한 개선을 경험한 2025년 상반기, 비호환 라이브러리 문제가 부각되면서 React 규칙이 빌드 시간 제약으로 강제되기 시작한 2025년 하반기의 생태계 재편 단계, 그리고 “그린필드(새 프로젝트)는 해결됐지만 브라운필드(레거시)는 별도 프로젝트”라는 현재 상태다.

컴파일러는 빌드 시간에 React 규칙을 기반으로 메모이제이션을 삽입하는 변환을 수행한다.
손으로 작성된 코드보다 더 세밀한 수준으로 — 전체 객체가 아닌 개별 서브 표현식까지 — 메모이제이션을 적용한다.
실패 유형은 다섯 가지로 정리된다: 렌더 중 프로퍼티 변이, 렌더 중 ref 읽기, 레거시 클래스 컴포넌트, 지원되지 않는 문법 패턴(분해된 props 재할당, try-catch 안의 조건문), 그리고 `“use no memo”` 탈출구.

마이그레이션 권장 절차는 6단계다.
React v19 이상 업데이트, ESLint 플러그인 활성화 및 규칙을 오류로 승격, 리프 컴포넌트에 어노테이션 기반 컴파일러 모드 활성화, 추론 모드 전환, 수동 메모이제이션 제거, strict 모드 고려 순서다.
특히 중요한 발견은, 컴파일러가 특정 문법 패턴이 포함된 컴포넌트를 경고 없이 조용히 건너뛴다는 점이다.
`react-hooks/unsupported-syntax` 규칙을 경고에서 오류로 승격해야 이 회귀를 발견할 수 있으며, 기본값이 꺼져 있고 문서화되지 않은 `react-hooks/todo` 규칙이 가장 넓은 범위의 실패를 포착한다.

향후 개발 방향으로는 컴포넌트별 컴파일 힌트를 통한 세밀한 제어, Server Components와의 통합으로 하이드레이션 비용 감소, 오랜 RFC인 `useEvent` 기본 기능 수렴, React Native 채택 가속화, 컴파일러가 최적화한 내용을 시각화하는 DevTools 도구가 언급된다.

## 분석

### 컴파일러는 개발자 교육 문제를 인프라 문제로 전환한다

React 컴파일러가 해결하는 것은 기술적 문제이기 이전에 인지 부하 문제였다.
`useMemo`와 `useCallback`의 올바른 사용은 React를 처음 배우는 개발자들이 가장 오해하기 쉬운 영역 중 하나였다.
의존성 배열을 잘못 작성하는 것, 너무 적게 메모이제이션하거나 반대로 과도하게 메모이제이션하는 것 — 이 모든 것이 코드 리뷰의 상당한 비중을 차지했다.

컴파일러는 이 인지 부하를 제거하는 대신, 그것을 “React 규칙 준수”라는 다른 형태의 인지 부하로 교환한다.
`useMemo` 의존성을 생각하는 대신, 렌더 중에 프로퍼티를 변이하지 않는지, ref를 읽지 않는지를 생각해야 한다.
이것이 더 쉬운 규칙인 것은 사실이지만, 완전히 부하가 사라진 것은 아니다.
교환의 방향은 “특정 훅의 올바른 사용법”에서 “순수 함수형 패러다임 준수”로 이동했다.

이 전환의 더 깊은 함의는, React 규칙이 이제 선택이 아니라 강제 사항이 된다는 것이다.
이전에는 규칙을 어겨도 성능 문제가 발생할 뿐 코드가 동작하지 않는 것은 아니었다.
컴파일러를 채택한 후에는 규칙 위반이 컴파일 오류 또는 조용한 컴파일 건너뜀으로 이어진다.
이것은 패러다임 이동이다 — React 코드가 어떻게 작성되어야 하는지에 대한 암묵적 합의가 명시적 계약으로 전환된다.

### 채택의 병목은 새 코드가 아니라 기존 생태계다

저자가 “그린필드는 해결됐지만 브라운필드는 별도 프로젝트”라고 표현한 것은 컴파일러 채택의 구조적 문제를 정확하게 짚는다.
새 프로젝트에서는 처음부터 규칙을 지켜 코드를 작성하면 되므로, 컴파일러는 자연스럽게 작동한다.
그러나 기존 프로젝트는 몇 년에 걸쳐 누적된 React 규칙 위반을 모두 수정해야 하는 작업을 요구한다.

더 큰 도전은 서드파티 라이브러리다.
애플리케이션 코드는 팀이 직접 수정할 수 있지만, 외부 라이브러리는 업스트림 메인테이너가 패치할 때까지 기다려야 한다.
라이브러리가 React 규칙을 위반한다면, 그 라이브러리를 사용하는 컴포넌트는 컴파일되지 않거나 `“use no memo”`로 표시되어야 한다.
이것은 “생태계 전체가 동시에 전환되지 않으면 혜택을 온전히 누리기 어렵다”는 구조적 의존성 문제를 만든다.

이 패턴은 TypeScript 도입 초기와 유사하다.
TypeScript도 처음에는 새 프로젝트에서 빠르게 채택되었지만, 기존 JavaScript 생태계의 타입 정의 부재로 인해 `.d.ts` 파일과 `@types` 패키지라는 별도의 생태계 레이어가 필요했다.
React 컴파일러에서도 비호환 라이브러리에 대한 호환성 레이어나 “안전한 래퍼”를 제공하는 유사한 에코시스템이 형성될 가능성이 있다.

### 조용한 건너뜀 문제는 신뢰 위기를 내포한다

컴파일러가 지원되지 않는 문법 패턴을 경고 없이 건너뛴다는 발견은 기사에서 중요하게 다루어지지만, 그 함의는 더 심각하다.
개발자가 컴파일러를 활성화했다고 생각하고 코드를 작성하는데, 실제로는 특정 컴포넌트에서 컴파일러가 작동하지 않고 있다면 이것은 관측 불가능한 실패다.

성능 프로파일링을 통해서만 “이 컴포넌트가 왜 예상보다 느리지?”라는 질문으로 발견할 수 있는 종류의 실패다.
`react-hooks/unsupported-syntax`와 `react-hooks/todo` 규칙의 존재 자체가, 컴파일러가 처리하지 못하는 패턴이 생각보다 광범위함을 시사한다.
기사가 이 ESLint 규칙들의 활성화를 “비판적 발견”으로 강조한 것은 타당하다 — 이 도구들 없이는 컴파일러가 실제로 작동하는지 확인하는 방법이 없다.

## 비평

### 저자는 컴파일러의 장기 유지보수 비용을 과소평가한다

기사는 컴파일러 채택의 장점과 마이그레이션 경로를 설득력 있게 서술하지만, 컴파일러를 사용하는 코드베이스의 장기 유지보수에 어떤 영향을 미치는지에 대한 논의가 부족하다.

컴파일러가 최적화를 자동으로 처리한다는 것은, 개발자가 “이 코드가 실제로 어떻게 실행되는가”와 “컴파일러가 어떻게 변환하는가” 사이의 추가 계층을 이해해야 한다는 뜻이다.
버그가 발생했을 때 — 특히 성능 회귀나 메모이제이션 관련 이상 동작 — 개발자는 컴파일러 출력을 이해하거나 컴파일러 동작을 우회하는 능력이 필요하다.
이것은 기존 React 개발에 없던 새로운 전문 지식 요구사항이다.

`“use no memo”` 주석이 “기술 부채 마커”가 될 수 있다는 것을 저자가 언급하지만, 이것은 단순한 코드 냄새 이상이다.
이 주석이 존재하는 컴포넌트는 왜 컴파일러를 우회해야 했는지, 그 이유가 여전히 유효한지, 해결되었는지 여부를 추적하는 별도의 관리 작업이 필요하다.
컴파일러를 활성화한 대형 코드베이스가 5년 후 어떤 모습일지 — `“use no memo”` 주석이 얼마나 누적되는지, 그것을 관리하는 비용 — 에 대한 전망이 기사에 없다.

### 기사는 컴파일러와 런타임 최적화기의 관계를 미해결 상태로 남긴다

Million.js와 같은 런타임 최적화기와 React 컴파일러의 관계를 “미해결 긴장”으로만 언급하고 지나치는 것은 기사의 중요한 공백이다.
이것은 단순한 철학적 논쟁이 아니다 — 두 접근 방식의 작동 수준이 다르기 때문에 실제로 충돌하거나 중복되는 최적화가 발생할 수 있다.

컴파일러는 빌드 시간에 정적 분석을 기반으로 메모이제이션을 삽입하고, Million.js 같은 도구는 런타임에 DOM 업데이트를 최적화한다.
이 두 레이어가 함께 사용될 때 서로를 보완하는지, 간섭하는지, 또는 어느 한쪽이 다른 한쪽을 불필요하게 만드는지는 실질적인 아키텍처 결정과 직결된다.
기사가 이 관계를 “진행 중인 논쟁”으로 처리하는 것은 결론을 회피하는 방식이다.

### 마이그레이션 6단계 절차의 실제 적용 복잡성이 과소평가되어 있다

저자가 제시하는 6단계 마이그레이션 절차는 간결하고 논리적으로 보인다.
그러나 이 절차가 중간 규모(수만 라인) 이상의 실제 코드베이스에 적용될 때 각 단계가 얼마나 복잡해지는지에 대한 현실적 논의가 없다.

2단계 — “ESLint 규칙을 오류로 승격” — 는 기존 코드베이스에 수백 개의 새 빌드 오류를 발생시킬 수 있다.
이것을 처리하는 것은 “오류를 수정한다”는 간단한 표현으로 처리되기 어려운 작업이다.
특히 ESLint 규칙이 검출하는 위반이 외부 라이브러리에서 발생한다면, 팀은 라이브러리 패치를 기다리거나, fork하거나, 해당 컴포넌트를 `“use no memo”`로 표시하는 중에서 선택해야 한다.
기사가 제시하는 절차는 이상적인 경로이지 일반적인 경로가 아닐 수 있다.

## 인사이트

### 컴파일러는 React를 더 쉽게 만드는 것이 아니라 입문 문턱을 이동시킨다

React 컴파일러의 서사는 대부분 “더 쉬워진다”는 방향으로 구성된다.
`useMemo`와 `useCallback`을 신경 쓰지 않아도 된다는 것은 분명히 특정 복잡성을 제거한다.
그러나 이 감소가 전체적인 단순화를 의미하는지, 아니면 복잡성의 재분배를 의미하는지는 다르게 볼 수 있다.

이전 React의 복잡성은 주로 런타임 동작 이해에서 왔다.
`useCallback`이 없으면 자식 컴포넌트가 불필요하게 리렌더된다는 것은 React의 렌더링 동작을 이해해야 알 수 있는 지식이다.
컴파일러 이후의 복잡성은 컴파일 타임 정적 분석의 제약으로 이동한다.
특정 패턴이 컴파일러를 우회하고 silent fallback이 발생한다는 것을 이해하는 것은 런타임 동작 이해와는 다른 종류의 지식이다.

결과적으로 React 입문이 더 쉬워지는 것은 사실일 수 있지만, 전문적인 React 개발자가 알아야 하는 것의 총량은 줄지 않았을 수 있다.
메모이제이션 지식이 줄고, 컴파일러 동작과 제약 지식이 늘었다.
이것은 새 개발자에게는 낮아진 장벽이지만, 기존 전문가에게는 재학습 비용이다.

### 생태계 전환의 실질적 타임라인은 기사가 암시하는 것보다 훨씬 길다

기사는 채택이 진행 중이고 그린필드 프로젝트에서는 잘 작동한다고 말하지만, 브라운필드 전환과 생태계 전환의 실제 타임라인에 대한 현실적 평가가 없다.

TypeScript 비교가 유용한 데이터 포인트다.
TypeScript는 2012년에 출시되었고, npm의 TypeScript 사용률이 50%를 넘은 것은 약 2020년 전후다.
8년이 걸렸다.
React 컴파일러는 TypeScript보다 영향을 미치는 범위가 더 좁다 — TypeScript가 언어 수준의 변환이었다면, React 컴파일러는 특정 프레임워크 내의 최적화다.
하지만 “기존 라이브러리들이 React 규칙을 준수하도록 업데이트되는” 전환은 인기 있는 라이브러리들의 메인테이너가 변경 사항을 반영하고, 그것이 배포되고, 프로젝트들이 업데이트하는 시간을 필요로 한다.
현실적인 타임라인은 수년 단위다.

이 타임라인이 중요한 이유는, 그 기간 동안 React 생태계가 “컴파일러를 완전히 사용하는 새 프로젝트”와 “컴파일러를 부분적으로 사용하거나 사용하지 않는 레거시 프로젝트”로 이분화된다는 점이다.
이 이분화는 라이브러리 메인테이너, 팀 온보딩, 문서 작성, 커뮤니티 지식 공유 모두를 복잡하게 만든다.
“컴파일러 시대의 React”와 “컴파일러 이전의 React”가 상당 기간 공존할 것이며, 이것은 단순히 “이전 방식은 사라진다”는 서사보다 복잡한 현실이다.

### 컴파일러 채택은 개발 조직의 React 코드 소유권 모델을 변경한다

컴파일러가 자동으로 최적화를 처리하는 세계에서 “누가 성능 최적화를 담당하는가”라는 질문의 답이 바뀐다.
이전에는 시니어 개발자가 메모이제이션 전략을 검토하고 코드 리뷰에서 피드백을 제공하는 역할이 있었다.
컴파일러가 이 결정을 대신하면, 이 역할은 어디로 가는가.

한 가지 결과는 성능 최적화 전문성이 애플리케이션 개발자 레벨에서 컴파일러 개발자 레벨로 이동하는 것이다.
일반 React 개발자는 “컴파일러가 알아서 해주겠지”라는 기대를 갖게 되고, 컴파일러가 예상대로 작동하지 않는 경우 — 조용히 건너뛰거나, 과도하게 메모이제이션하거나 — 를 진단하는 능력이 희소해진다.

이것은 두 번째 질서 효과다.
컴파일러가 일반 사례를 잘 처리할수록, 예외 사례를 이해하는 전문성이 팀 내에서 유지되지 않게 된다.
그리고 컴파일러가 예외 사례를 만났을 때 — 항상 발생한다 — 그것을 진단하고 해결할 수 있는 사람이 조직 내에 없을 수 있다.
자동화가 전문성을 대체할 때 종종 나타나는 패턴이다: 시스템이 잘 작동하는 동안에는 효율적이지만, 실패했을 때 복구 능력이 없다.
