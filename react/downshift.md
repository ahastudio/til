# downshift

<https://www.downshift-js.com/>

<https://github.com/downshift-js/downshift>

## 소개

downshift는 WAI-ARIA 표준을 준수하는 React 자동완성(autocomplete), 콤보박스(combobox), 셀렉트(select) 드롭다운 컴포넌트를 만들기 위한 프리미티브(primitives) 집합이다.
Kent C. Dodds가 2017년에 만들었고, 2026년 6월 기준 GitHub 스타 12,297개를 기록하고 있다.
“UI를 렌더링하지 않고 접근성과 상태 로직만 제공한다”는 헤드리스(headless) UI 접근법의 선구자 중 하나다.

라이브러리는 두 가지 API를 제공한다.
첫 번째는 렌더 프롭(render prop) 방식의 `Downshift` 컴포넌트로, 라이브러리의 원형이다.
두 번째는 React 훅 기반의 `useSelect`, `useCombobox`, `useTagGroup`으로, 현재 권장되는 방식이다.
README는 “새로 시작한다면 훅을 먼저 고려하라”고 명시하며, `Downshift` 컴포넌트는 훅이 성숙하면 완전히 제거될 예정이다.

라이브러리의 핵심 개념은 세 가지다.
`prop getters`는 DOM 요소에 스프레드할 props 객체를 반환하는 함수들로, `aria-*` 속성과 이벤트 핸들러를 자동으로 포함한다.
`stateReducer`는 내부 상태 변화를 가로채고 수정할 수 있는 함수로, Redux 리듀서 패턴을 차용했다.
Control Props는 외부에서 `isOpen`, `selectedItem`, `inputValue`, `highlightedIndex` 같은 내부 상태를 직접 주입하는 제어 컴포넌트(controlled component) 패턴이다.

## 분석

### 헤드리스 UI의 구조적 기여

downshift가 등장한 2017년 React 생태계는 두 가지 방향으로 나뉘어 있었다.
Material-UI나 Ant Design 같은 라이브러리는 완성된 UI를 제공하되 커스터마이징의 폭이 좁았다.
반대쪽에는 직접 구현하거나 날것의 컴포넌트에 많은 props를 추가하는 방식이 있었다.
downshift는 세 번째 길을 제시했다. 로직과 접근성은 라이브러리가 책임지고, UI는 전적으로 사용자가 결정한다.

이 분리는 단순한 스타일 분리가 아니다.
접근성 구현은 올바른 ARIA 역할(`role=“combobox”`, `aria-expanded`, `aria-activedescendant` 등)과
키보드 탐색(방향키, Enter, Escape), 스크린 리더를 위한 라이브 리전(live region) 관리를 포함한다.
이 세 가지를 모두 올바르게 구현하는 것은 숙련된 개발자도 쉽게 실수하는 영역이다.
downshift는 이 복잡성을 흡수하면서 개발자에게는 “어떤 HTML 구조를 사용할지”의 결정권만 남긴다.

Headless UI(Tailwind Labs), Radix UI, React Aria(Adobe) 같은 후속 라이브러리들이 모두 같은 원칙 위에서 만들어졌다.
downshift는 이 패러다임의 직접적인 선구자다.

### Prop Getters 패턴의 설계 의도

`getInputProps()`, `getMenuProps()`, `getItemProps()` 같은 prop getter 함수들은 처음 보면 직관적이지 않다.
`<input {...getInputProps()} />` 대신 `<input />` 에 직접 속성을 쓰는 것이 더 간단해 보인다.

이 패턴의 핵심 이점은 이벤트 핸들러 합성(composition)이다.
사용자가 `onKeyDown`을 전달하면, downshift는 자신의 키보드 핸들러와 사용자의 핸들러를 모두 실행하도록 합성한다.
일반적인 props 스프레드(`{...downshiftProps} onKeyDown={myHandler}`)에서는 나중에 쓴 핸들러가 앞의 것을 덮어쓴다.
prop getter는 이 충돌을 방지하며, 동시에 ARIA 속성이 항상 올바르게 적용됨을 보장한다.

Kent C. Dodds는 이 패턴을 별도의 블로그 포스트로 설명했고, 이후 많은 헤드리스 라이브러리가 같은 설계를 차용했다.
prop getter 패턴은 “라이브러리가 관리하는 props와 사용자 정의 props가 안전하게 공존하는 방법”에 대한 실용적 해답이다.

### 렌더 프롭에서 훅으로의 전환이 의미하는 것

`Downshift` 컴포넌트는 렌더 프롭 패턴의 교과서적 사례였다.
children을 함수로 받아 상태와 헬퍼를 주입하는 방식은 React 16.3 이전, 훅이 없던 시절의 최선이었다.
React 16.8에서 훅이 도입된 이후, 렌더 프롭은 불필요한 컴포넌트 중첩과 “wrapper hell”을 유발한다는 비판을 받았다.

downshift의 훅 전환은 단순한 API 개선이 아니다.
렌더 프롭으로는 불가능했던 구성(composition)이 훅에서는 자연스럽게 된다.
`useCombobox`를 `useMultipleSelection`과 함께 쓰면 다중 선택 콤보박스를 만들 수 있는데,
이 조합은 렌더 프롭 시절에는 두 컴포넌트를 중첩하고 상태를 수동으로 연결해야 했다.

그러나 README에 `Downshift` 컴포넌트 설명이 여전히 남아 있고 훅보다 먼저 등장한다는 점은 레거시 마이그레이션의 현실적 어려움을 보여준다.
“완전히 제거될 예정”이라는 선언과 달리, 수년째 공존 중이다.

### State Reducer 패턴의 제어 역전

`stateReducer`는 downshift의 가장 강력하고 가장 복잡한 기능이다.
컴포넌트의 모든 상태 변화는 이 함수를 통과하며, 개발자는 변화를 가로채고 수정할 수 있다.
예를 들어 항목을 선택해도 메뉴가 닫히지 않게 하려면 `clickItem` 타입의 변화에서 `isOpen`을 현재 상태로 유지하면 된다.

이것은 제어 역전(Inversion of Control, IoC) 패턴의 명확한 적용이다.
일반적인 컴포넌트는 동작을 props로 노출하고 내부에서 처리한다.
`stateReducer`는 처리 흐름 자체를 사용자에게 넘기며, 라이브러리는 “무엇이 변할 수 있는지”만 정의한다.
이 설계는 “수백 개의 props를 추가하지 않고 모든 커스텀 동작을 지원하는 방법”에 대한 우아한 답이다.

## 비평

### 접근성 보장은 라이브러리가 아니라 사용자에게 달려 있다

downshift의 핵심 가치 제안은 WAI-ARIA 준수다.
그러나 prop getter를 사용한다고 자동으로 접근성이 보장되지는 않는다.
`getMenuProps()`가 `role=“listbox”`와 `aria-labelledby`를 주입하지만, 메뉴를 DOM에서 완전히 제거하면(조건부 렌더링으로 `null` 반환) 이 관계가 깨진다.
README 자체가 “메뉴는 항상 렌더링 상태를 유지하고 CSS로만 숨기길 권장한다”고 경고하지만, 이를 어기는 코드는 downshift를 사용하면서도 스크린 리더에서 완전히 망가진 경험을 만든다.

더 근본적인 문제는 ARIA 패턴이 복잡하다는 것이다.
ARIA 1.2의 combobox 패턴은 입력 요소와 팝업 요소의 관계, 포커스 관리, `aria-activedescendant` 갱신 시점을 정확히 구현해야 한다.
downshift가 이 복잡성을 내부화했지만, 잘못된 HTML 구조(예: `<input>` 밖에 `<label>` 배치)나 잘못된 prop getter 적용은 여전히 접근성을 해친다.
라이브러리는 올바른 구현을 쉽게 만들지만, 잘못된 구현을 불가능하게 만들지는 않는다.

### API 표면적이 “단순한 프리미티브”라는 설명과 맞지 않는다

README는 downshift를 “심플하고 유연한 프리미티브”로 소개한다.
그러나 Basic Props, Advanced Props, stateChangeTypes, Control Props, Children Function, prop getters, actions, Event Handlers를 모두 이해해야 기본 사용 사례를 벗어날 수 있다.
`stateReducer`를 올바르게 쓰려면 `stateChangeTypes`의 모든 값을 파악해야 하는데, 이 목록은 17개 항목이다.

단순한 셀렉트 컴포넌트를 만들기 위해 이 모든 개념을 습득하는 비용은,
CSS 프레임워크를 쓰는 것보다 훨씬 높다.
“심플하다”는 설명은 라이브러리 내부의 복잡성이 낮다는 의미이지, 사용 난이도가 낮다는 의미가 아니다.
이 구분을 README가 충분히 명확히 하지 않아서, 처음 접하는 개발자에게 잘못된 기대를 만든다.

### Downshift 컴포넌트 폐기 선언이 실질적 타임라인 없이 수년째 유지된다

“훅이 성숙하면 `Downshift` 컴포넌트를 완전히 제거할 것”이라는 선언은 버전 5(2019년경)부터 있었다.
7년 가까이 지난 2026년 현재도 두 API가 공존하며, 폐기(deprecation) 경고도 없고 구체적인 제거 일정도 없다.

이것은 오픈소스 라이브러리에서 흔한 현상이지만, downshift의 경우 특히 문제가 된다.
README의 구조상 `Downshift` 컴포넌트 문서가 훅 문서보다 앞에 나오며, 표면적으로 더 많은 공간을 차지한다.
새로운 사용자가 README를 순서대로 읽으면 자연스럽게 deprecated 패턴을 먼저 학습하게 된다.
“🚨 훅을 사용하라 🚨” 경고가 있지만, 이미 긴 `Downshift` 설명을 읽은 후에야 나온다.

### 훅 버전은 렌더 프롭 버전의 유연성을 완전히 대체하지 못한다

렌더 프롭 방식의 `Downshift` 컴포넌트는 클래스 컴포넌트에서도 동작한다.
훅은 함수형 컴포넌트에서만 사용 가능하다.
레거시 클래스 컴포넌트가 남아 있는 코드베이스에서는 마이그레이션 없이 훅 전환이 불가능하다.

README는 이 제약을 언급하지 않는다.
“훅을 먼저 고려하라”는 권장 사항이 모든 프로젝트에 적용 가능한 것처럼 서술되어 있다.
실제로는 수백만 줄의 레거시 React 코드를 운영하는 팀에게 이 권장은 이론적 조언에 가깝다.

## 인사이트

### 제어 역전 패턴은 컴포넌트 라이브러리의 API 설계 방향을 전환시켰다

downshift 이전의 컴포넌트 라이브러리들은 대부분 “충분히 많은 props를 제공하면 모든 사용 사례를 커버할 수 있다”는 가정 위에서 설계됐다.
이 접근법의 한계는 예측하지 못한 사용 사례가 생길 때마다 API가 팽창한다는 것이다.
결국 수백 개의 props를 가진 라이브러리가 탄생하고, 그럼에도 원하는 동작을 구현할 수 없어 소스를 포크하게 된다.

`stateReducer`는 다른 관점을 제시한다.
모든 사용 사례를 예측하는 대신, 상태 전환의 흐름을 외부에서 수정할 수 있게 한다.
이것은 Redux의 미들웨어 패턴, React의 `useReducer`, 엘릭서의 `GenServer` 콜백이 공유하는 설계 원칙이다.
“제공되는 것을 사용한다”가 아니라 “흐름에 개입한다”는 모델이다.

이 전환의 영향은 React 생태계 전반에 파급됐다.
React Hook Form의 `Controller`, React Query의 `select`/`transform`, Zustand의 미들웨어 구조가 모두 유사한 제어 역전을 구현한다.
downshift의 `stateReducer`는 이 패턴이 컴포넌트 레이어에 적용된 초기 사례다.

### 헤드리스 UI의 확산은 디자인 시스템 레이어를 재편한다

downshift가 제안한 “로직과 UI의 분리”는 단순한 라이브러리 설계 선택을 넘어, 기업의 프런트엔드 아키텍처에 구조적 변화를 일으키고 있다.

전통적 컴포넌트 라이브러리 스택은 “UI 프레임워크 → 프로젝트 커스터마이징”의 두 층이었다.
헤드리스 UI가 보편화되면서 “헤드리스 프리미티브 → 디자인 시스템 → 프로젝트 구현”의 세 층 구조가 표준이 되고 있다.
Vercel, Stripe, GitHub 같은 팀들이 Radix UI나 React Aria 위에 자체 디자인 시스템을 구축하는 것이 이 패턴의 구체적 사례다.

이 구조 변화의 2차 효과는 접근성 전문성의 집중화다.
과거에는 각 팀이 ARIA 패턴을 독립적으로 구현하며 오류를 반복했다.
이제 접근성 구현은 downshift, Radix, React Aria 같은 소수의 라이브러리에 집중되고, 수천 개의 프로젝트가 이를 재사용한다.
접근성 버그가 수정되면 그 혜택이 모든 사용 프로젝트에 전파된다.
이것은 오픈소스의 공공재적 성격이 접근성이라는 전문 영역에 적용된 효율적 구조다.

### Prop Getter 패턴은 “소유권”과 “합성”의 긴장을 해소한다

컴포넌트 API 설계에서 반복적으로 등장하는 긴장이 있다.
라이브러리가 DOM 요소의 props를 “소유”하면 유저의 커스터마이징이 막히고, 유저가 “소유”하면 라이브러리의 기능이 깨진다.

prop getter는 이 긴장을 다음 방식으로 해소한다.
라이브러리가 필요한 props를 함수로 제공하고, 유저는 그 함수에 추가 props를 인자로 전달한다.
라이브러리는 자신의 props와 유저의 props를 합성하여 반환한다.
결과적으로 어느 쪽도 일방적으로 상대방의 props를 덮어쓰지 않는다.

이 패턴은 “React 컴포넌트가 어떻게 확장 가능해야 하는가”라는 질문에 대한 하나의 답이다.
클래스 상속이 아니라 함수 합성으로, “모든 것을 노출”이 아니라 “결합점(seam)을 제공”하는 방식으로 확장성을 구현한다.
이것은 소프트웨어 설계의 더 넓은 원칙인 “합성이 상속보다 낫다”를 UI 컴포넌트 레이어에서 실현한 형태다.

### 라이브러리의 성숙도는 버전보다 폐기 관리로 측정된다

downshift의 현재 상태—권장되는 훅 API와 폐기 예고된 렌더 프롭 API가 수년째 공존—는 오픈소스 라이브러리의 거버넌스 문제를 드러낸다.

폐기 선언이 실행되지 않는 이유는 항상 같다.
기존 사용자의 마이그레이션 비용, 문서화 부담, 단기적 이익 없는 장기 투자.
그러나 폐기되지 않는 API는 시간이 지날수록 새로운 사용자에게 함정이 된다.
“사용하지 말라”는 API를 학습하는 비용, 내부적으로 두 패러다임을 유지하는 복잡성이 누적된다.

이것은 downshift만의 문제가 아니다.
jQuery UI와 jQuery Mobile의 공존, Angular.js와 Angular의 동시 유지, Python 2와 Python 3의 십 년 공존이 같은 패턴이다.
소프트웨어 생태계에서 “충분히 좋은 레거시”를 폐기하는 것은 기술적 결정이 아니라 사회적 결정이다.
downshift가 이 결정을 어떻게 내리는지는, 미래 라이브러리가 참조할 선례가 된다.
