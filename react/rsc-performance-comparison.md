# React Server Components는 정말 성능을 개선하는가

원문: <https://www.developerway.com/posts/react-server-components-performance>

## 요약

Nadia Makarevich가 CSR(Client-Side Rendering), SSR(Server-Side Rendering),
RSC(React Server Components)를 동일한 앱과 테스트 환경에서 데이터로 비교한 글이다.
측정 지표는 LCP(Largest Contentful Paint), 사이드바 표시 시간, 메시지 표시 시간,
페이지 인터랙티브 시간, 그리고 페이지가 보이지만 JS가 아직 로딩 중인 “무반응 구간”이다.
Chrome DevTools Performance 탭, CPU 6배 슬로우다운, Slow 4G 네트워크 조건에서
측정했다.

CSR은 LCP가 캐시 없을 때 4.1초로 최악이지만, 페이지가 보이는 즉시 인터랙티브하고
페이지 전환이 80ms로 가장 빠르며 인프라 비용이 거의 없다.
SSR(클라이언트 데이터 페칭)은 LCP를 4.1초에서 1.61초로 대폭 개선하지만,
`useEffect`로 데이터를 가져오는 구조는 그대로이므로 사이드바와 메시지 표시 시간은
변하지 않는다.
또한 페이지가 보이고도 JS 다운로드를 기다리는 2.39초짜리 무반응 구간이 생긴다.

SSR에서 서버에서 데이터를 미리 가져오면(서버 데이터 페칭) 사이드바와 메시지가 LCP와
동시에 보이는 장점이 있지만, 모든 데이터를 기다린 후에야 렌더링을 시작하므로 LCP가
오히려 2.16초로 늘어난다.
Next.js Pages Router는 코드 스플리팅 덕분에 무반응 구간을 1.34초까지 줄이는 가장
균형 잡힌 결과를 보인다.

RSC와 스트리밍을 도입하면 이론적으로 “서버에서 데이터를 가져오면서도 LCP를 희생하지
않는다”는 목표를 달성할 수 있다.
단, 두 가지 조건이 필수다.
첫째, 데이터 페칭 코드를 클라이언트 `useEffect`에서 비동기 서버 컴포넌트로 완전히
재작성해야 한다.
둘째, 비동기 서버 컴포넌트를 `<Suspense>`로 반드시 감싸야 한다.
`<Suspense>`를 빠뜨리면 React가 전체 앱을 하나의 청크로 취급해 모든 데이터가 준비될
때까지 렌더링을 블로킹한다. 결과적으로 전통적인 SSR 서버 데이터 페칭과 동일하게
동작한다.
`<Suspense>`까지 올바르게 적용한 최종 결과는 LCP 1.28초, 사이드바 1.28초, 메시지
1.1초, 무반응 구간 0.05초로 모든 지표에서 최고 수준이다.

### SSR의 근본적 긴장: LCP vs. 데이터 완결성

SSR에서 LCP와 데이터 표시 시간은 서로 다른 방향으로 당기는 힘이 있다.
LCP를 빠르게 하려면 가능한 한 빨리 HTML을 보내야 한다.
그러나 서버에서 데이터를 미리 가져오려면 모든 프로미스가 해결될 때까지 렌더링을 블로킹해야
한다.
전통적인 `renderToString`은 동기 방식이므로 이 두 목표를 동시에 달성할 수 없다.
스트리밍은 이 긴장을 해소하기 위해 HTML을 청크 단위로 나눠 보내는 방식을 도입했다.

### Suspense가 RSC 성능의 핵심인 이유

저자가 강조하는 핵심은 RSC 자체가 아니라 스트리밍 + Suspense의 조합이다.
`<Suspense>` 없이 비동기 서버 컴포넌트를 쓰면 React는 트리 전체를 하나의 덩어리로
렌더링하며 모든 `await`가 완료될 때까지 기다린다.
`<Suspense>` 경계가 있어야만 React가 “이 부분은 나중에 보낼 수 있다”고 판단해 나머지
페이지를 먼저 클라이언트에 전송한다.

### App Router의 JS 지연 로딩 트레이드오프

App Router는 CSS가 로딩된 후에야 JavaScript 다운로드를 시작하도록 기본 동작한다.
이것이 LCP를 개선하는 핵심 요인이다(CSS 다운로드 중 JS가 대역폭을 뺏지 않으므로).
그러나 동시에 JavaScript 초기화가 지연되어 무반응 구간이 더 길어지는 트레이드오프가 생긴다.
Pages Router는 CSS와 JS를 병렬로 다운로드해 LCP가 약간 느리지만 인터랙티브 시점이 빠르다.

### RSC 단독 효과: 기대와 현실의 간극

저자가 코드를 점진적으로 `use client`를 제거하며 서버 컴포넌트로 전환해본 결과,
번들 크기는 일부 페이지에서 크게(로그인 페이지 거의 0에 가깝게) 줄었지만,
공유 청크는 변화가 없었고 Inbox 페이지의 성능 지표는 전혀 개선되지 않았다.
“서버 컴포넌트가 번들 크기를 줄여 성능을 개선한다”는 주장은 데이터 페칭 재작성 없이는
현실에서 검증되지 않는 이론이다.

## 비평

### 강점: 재현 가능한 데이터와 점진적 비교

이 글의 가장 큰 강점은 재현 가능성이다.
실험 앱이 GitHub에 공개되어 있고, 각 섹션마다 정확한 재현 절차가 제시된다.
“lift-and-shift” 마이그레이션을 기준점으로 삼아 서버 컴포넌트와 스트리밍의 효과를
분리하는 방법론은 탁월하다.
감에 의존하는 성능 논의가 많은 React 생태계에서 숫자로 말하는 방식이 설득력 있다.

### 한계: 특정 시나리오에 최적화된 벤치마크

Slow 4G + CPU 6배 슬로우다운은 모바일 저사양 기기를 시뮬레이션하는 합리적 설정이지만,
현실의 다양한 환경—빠른 데스크톱, 내부망 서비스, CDN 없는 환경—에서 각 기법의 상대적
우위가 달라질 수 있다.
또한 단일 페이지(Inbox)의 측정이 중심이라, 앱 전체 생애주기에서 캐시 히트율이 높아지는
SaaS 제품의 반복 방문 시나리오에서의 실질적 차이는 충분히 다루지 않는다.

### 보완이 필요한 관점: 개발 복잡도 비용

RSC + 스트리밍 + Suspense 조합이 성능상 최선이라는 결론에는 동의하지만,
그 전환이 요구하는 개발 비용이 과소평가된 느낌이 있다.
저자 스스로 “스트리밍의 실제 구현은 insane하다”고 표현했지만, 이 복잡도가 팀 규모나
숙련도에 따라 성능 이득을 상쇄할 수 있다는 분석이 빠져 있다.
“Suspense를 빠뜨리면 전통 SSR과 같다”는 사실 자체가 이 패턴이 얼마나 오류를 숨기기
쉬운지를 보여준다.

## 인사이트

### “보임”과 “동작함”의 분리는 새로운 UX 계층을 만든다

SSR이 등장하면서 웹 성능의 언어가 바뀌었다.
CSR 시대에는 “언제 보이는가”와 “언제 동작하는가”가 같은 시점이었다.
JavaScript가 다 실행되어야 화면도 보이고 클릭도 된다.
SSR은 두 시점을 분리했다.
페이지가 보이는 시점(LCP)과 인터랙티브해지는 시점(TTI)이 다르다.
이 분리가 “무반응 구간”이라는 새로운 나쁜 UX를 만들었다.
사용자가 보이는 버튼을 눌렀는데 아무 반응이 없는 그 불쾌한 순간이다.
이것은 단순한 기술적 수치가 아니라 사용자의 신뢰 손상이다.
RSC + 스트리밍은 “데이터 도착”이라는 세 번째 시점을 추가해 상황을 더 복잡하게 만들었다.
이제 성능 지표는 LCP, TTI, 각 데이터 영역별 표시 시점으로 분화된다.
프레임워크가 더 똑똑해질수록 개발자가 이해해야 할 성능 모델도 더 복잡해진다는 역설이다.

### Suspense는 API가 아니라 아키텍처 경계다

`<Suspense>`를 빠뜨렸을 때 RSC가 전통 SSR과 동일하게 동작한다는 발견은,
Suspense가 단순한 로딩 UI 컴포넌트가 아님을 보여준다.
Suspense는 React에게 “이 트리의 이 부분은 나중에 채워도 된다”는 청크 경계를 선언하는
아키텍처 지시문이다.
이는 프로그래밍에서 “선언이 동작을 결정한다”는 원칙의 좋은 사례다.
SQL에서 `EXPLAIN`으로 쿼리 플랜을 보면 인덱스 힌트 하나가 전체 실행 경로를 바꾸듯,
`<Suspense>` 경계 하나가 전체 HTML 스트리밍 전략을 바꾼다.
개발자가 이 선언의 의미를 정확히 이해하지 못하면, 성능 목표와 반대 방향으로 앱이
동작하는 상황이 조용히 발생한다.
저자가 실험에서 발견한 것처럼, 올바른 구현과 잘못된 구현의 차이가 코드상으로는
`<Suspense>` 래퍼 하나지만 성능상으로는 전혀 다른 특성을 만들어낸다.
이것은 복잡한 시스템에서 작은 설정이 큰 동작 차이를 만드는 고전적 패턴이다.
React 팀이 이를 명시적 API로 노출한 것은 옳지만, 그 의미가 충분히 직관적이지 않아
실수를 유발하기 쉽다.
스트리밍 경계가 Suspense에 묶여 있다는 사실은 문서에서 강조되어야 할 핵심이지만,
실제로 Next.js 문서는 이를 충분히 전달하지 않는다는 것이 이번 실험에서 드러났다.

### CSR의 재평가: “느린 기술”이 아니라 “다른 트레이드오프”다

CSR이 초기 로드에 4.1초가 걸린다는 수치는 공포스럽지만, 저자가 균형 잡힌 시각을
제공한다.
SaaS처럼 반복 방문이 많은 서비스에서 JS가 캐시되면 800ms로 줄고,
페이지 전환은 80ms로 서버 기반 어떤 방식보다도 빠르다.
무반응 구간이 없다는 것도 중요한 UX 장점이다.
CSR은 “틀린 기술”이 아니라 “다른 트레이드오프를 가진 기술”이다.
이 글이 제공하는 가장 중요한 시각은 어떤 기법도 모든 축에서 우위를 점하지 못한다는
것이다.
LCP를 최적화하면 무반응 구간이 생기고, 무반응 구간을 줄이면 LCP가 악화되거나 구현
복잡도가 폭증한다.
“React Server Components가 성능을 개선하는가?”라는 질문의 올바른 답은 “항상 그렇다”도
“아니다”도 아니라, “어떤 지표를 어떤 사용자 패턴에서 측정하느냐에 따라 다르다”이다.
기술 선택을 단일 벤치마크 수치로 결정하는 것이 얼마나 위험한지,
그리고 성능 측정이 얼마나 맥락 의존적인지를 이 실험은 잘 보여준다.
