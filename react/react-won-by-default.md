# React는 기본값으로 승리했다 — 그리고 그것이 프론트엔드 혁신을 죽이고 있다

원문: <https://www.lorenstew.art/blog/react-won-by-default/>

## 요약

Loren Stewart는 React의 프론트엔드 지배가 기술적 우월성이 아니라 문화적 관성에서
비롯되었다고 주장한다.
팀들이 프로젝트 제약을 검토하는 대신 “모두가 React를 안다”는 이유로 반사적으로
React를 선택한다.
이 자기 강화 순환은 기술적 적합성보다 네트워크 효과를 우선시하고, 결국 생태계
전체를 단일 문화(monoculture)로 고착화한다.

저자는 React의 대안으로 세 프레임워크를 기술적 차별점과 함께 제시한다.
Svelte는 런타임 시그널과 컴파일 타임 최적화를 결합해 가상 DOM 오버헤드를 제거한다.
한 개발자는 React에서 Svelte로 전환해 번들 크기를 187KB에서 9KB로 줄인 사례가
보고된다.
Solid는 가상 DOM 재조정(reconciliation) 없이 세밀한 반응성(fine-grained reactivity)을
제공해 더 효율적인 DOM 업데이트를 가능하게 한다.
Qwik은 하이드레이션 대신 재개성(resumability)을 사용해 현재 인터랙션에 필요한
코드만 로드함으로써 즉각적인 시작을 구현한다.

React 자체의 기술적 부채도 논거로 제시된다.
클래스에서 훅으로, 다시 서버 컴포넌트로 이어지는 React의 진화는 안정성이 아니라
지속적인 복잡도 증가를 보여준다.
특히 `useEffectEvent` 훅의 추가는 “훅 자체가 만들어낸 복잡도에 대한 패치”로
묘사된다.

생태계 차원의 결과도 다룬다.
채용 공고가 “프론트엔드 엔지니어” 대신 “React 개발자”를 요구하는 관행은
스킬 다양성과 조직의 유연성을 제한한다.
저자는 성능 지표, 팀 전문성, 장기 유지보수 비용, 생태계 성숙도와 혁신 가능성을
균형 있게 고려하는 프레임워크 평가 체크리스트를 제안한다.
건강한 기술 생태계는 다양성을 필요로 하며, 반사적 기본값 선택 대신 제약 기반의
의도적 선택이 더 나은 결과를 낳는다는 것이 핵심 주장이다.

## 분석

### 기술 논거와 문화 논거의 혼재

이 글의 논거는 두 층위를 가진다.
하나는 기술적 논거—Svelte의 컴파일 타임 최적화, Solid의 세밀한 반응성, Qwik의
재개성—이고, 다른 하나는 문화적 논거—반사적 선택, 채용 공고 언어, 모노컬처의
관성—다.
두 층위는 서로를 강화하지만, 혼용될 때 논리 구조가 약해진다.
“기술적으로 더 나은 대안이 있다”는 주장과 “사람들이 잘못된 이유로 React를 선택한다”는
주장은 별개이며, 각각 다른 종류의 증거를 필요로 한다.

### 번들 크기 사례의 대표성

187KB에서 9KB로의 번들 크기 감소 사례는 인상적이지만, 단일 사례다.
이 수치는 React 생태계의 라이브러리 의존성—상태 관리, 라우팅, UI 컴포넌트—을
포함한 비교일 가능성이 높다.
Svelte 자체의 기술적 차이만이 아니라, React 프로젝트가 누적한 의존성 부채가 반영된
숫자일 수 있다.
단일 사례가 프레임워크 선택의 일반적 결론을 지지하기에는 불충분하다.

### `useEffectEvent`를 복잡도 패치로 보는 시각

`useEffectEvent`가 “훅이 만들어낸 복잡도의 패치”라는 주장은 특정 관점에서만 옳다.
소프트웨어는 본질적으로 반복적으로 정제되며, 새로운 API가 이전 API의 한계를
보완하는 것은 자연스럽다.
이것을 기술 부채의 증거로 볼 수도 있고, 피드백 루프를 통해 API를 개선하는
성숙한 설계 과정으로 볼 수도 있다.
동일한 사실에 대해 어느 해석을 선택하느냐는 저자의 전제에 이미 결론이 내포되어
있음을 시사한다.

### 모노컬처 비판의 역사적 맥락

웹 기술 생태계에서 모노컬처 비판은 반복되는 패턴이다.
jQuery 지배기에 “jQuery 없이 개발하라”는 운동이 있었고, 실제로 바닐라 JS 르네상스가
뒤따랐다.
Angular 지배기에도 같은 비판이 등장했고, React가 그 지배를 대체했다.
역사적 패턴은 지배적 프레임워크가 결국 교체된다는 것을 보여주지만,
그 교체가 “더 나은 기술”이 아니라 “다른 트레이드오프가 맞는 시대가 왔을 때”
발생한다는 점도 보여준다.

## 비평

### 강점: 구체적 기술 대안과 실용적 평가 프레임

추상적인 비판에 그치지 않고 Svelte, Solid, Qwik 각각의 기술적 차별점을 명시한 점이
강점이다.
독자가 “그래서 대안이 뭔데?”라는 질문에 답을 얻을 수 있다.
프레임워크 평가 체크리스트도 실무적으로 유용하다.
“생태계 성숙도”를 반박하는 논거를 제공해 흔한 방어 논리에 미리 대응하는 구조도
균형적이다.

### 한계: “기본값으로 승리”의 정의가 모호하다

“기본값으로 승리했다”는 핵심 주장이 무엇을 의미하는지 명확하지 않다.
기술적으로 더 나은 선택이 존재함에도 관성으로 선택된다는 뜻인가,
아니면 단순히 가장 흔한 선택지가 되었다는 뜻인가.
전자라면 “어떤 맥락에서 어떤 대안이 더 낫다”는 구체적 증거가 필요하고,
후자라면 이것이 반드시 나쁜 결과를 낳는다는 추가 논거가 필요하다.
이 두 주장이 혼용되면서 논거의 강도가 분산된다.

### 보완이 필요한 관점: React 지배의 실질적 비용

모노컬처의 문제를 이야기하려면 그 실질적 비용이 더 구체적으로 제시되어야 한다.
“프론트엔드 혁신을 죽인다”는 주장을 뒷받침하려면, React 지배가 없었다면 존재했을
혁신이 실제로 억제되었다는 증거가 필요하다.
React 생태계 안에서도 Zustand, TanStack Query, shadcn/ui 같은 혁신적 패키지들이
등장했다는 사실은 모노컬처가 반드시 혁신 부재를 의미하지 않는다는 반례다.
혁신의 단위가 프레임워크 레벨인지, 라이브러리 레벨인지를 구분했다면 논거가
더 정밀해졌을 것이다.

## 인사이트

### 기술 선택의 “기본값”은 암묵적 합의이자 거버넌스다

“모두가 React를 쓴다”는 현상은 단순한 유행이 아니라, 조직들이 명시적으로 논의하지
않고도 도달한 암묵적 합의다.
이것은 네트워크 효과의 결과이지만, 동시에 거버넌스의 실패이기도 하다.
팀이 기술 선택을 의식적으로 평가하지 않으면, 그 선택은 외부의 관성에 위임된다.
외부 관성에 위임된 결정은 팀의 실제 제약과 목표에 맞게 최적화될 이유가 없다.
이 의미에서 “기본값으로 선택”은 결정을 내린 것이 아니라 결정을 포기한 것이다.

소프트웨어 아키텍처에서 “합리적 기본값”과 “비합리적 기본값”을 구분하는 기준은
그 기본값이 실제 트레이드오프를 반영하는가에 있다.
Linux 커널의 CFQ 스케줄러가 기본값이었던 것은 대부분의 워크로드에서 합리적이었기
때문이다.
React가 “기본값”이 된 것이 대부분의 프론트엔드 프로젝트에서 합리적이기 때문인지,
아니면 단순히 먼저 도달한 임계 질량 때문인지를 구분하는 것이 이 논쟁의 핵심이다.
그 구분이 불명확한 한, 논쟁은 계속 순환한다.

### 지배적 기술의 교체는 “더 나은 기술”이 아니라 “제약의 변화”로 촉발된다

기술 역사를 돌아보면, 지배적 기술이 교체될 때 “더 나은 기술이 등장했기 때문”인
경우는 드물다.
jQuery가 쇠퇴한 것은 바닐라 JS가 더 나아서가 아니라 브라우저 호환성 문제가 해소되고
DOM API가 성숙해졌기 때문이다.
jQuery가 해결하던 문제 자체가 더 이상 문제가 아니게 된 것이다.
Flash가 사라진 것은 HTML5가 더 좋아서라기보다 모바일 생태계가 Flash를 지원하지
않는 제약을 강제했기 때문이다.

React가 교체된다면, 그것은 Svelte나 Solid가 “더 낫기” 때문이 아닐 것이다.
웹의 제약이 바뀔 때—예를 들어 WebAssembly가 충분히 성숙해 컴파일 타임 최적화의
장벽이 낮아질 때, 또는 엣지 컴퓨팅이 보편화되어 번들 크기가 임계적 제약이 될 때—
제약에 더 잘 맞는 도구가 자연스럽게 부상할 것이다.
Qwik의 재개성이 주목받는 이유는 이미 모바일 데이터 비용과 코어 웹 바이탈이 새로운
제약으로 부상하고 있기 때문이다.
저자가 주장하는 “의도적 선택”의 진짜 의미는, 현재 제약이 무엇인지 파악하고
그 제약에 맞는 도구를 고르라는 것으로 읽힐 때 더 설득력을 가진다.

### 모노컬처의 비용은 혁신 억제가 아니라 “인지 잠금”이다

저자는 모노컬처가 혁신을 죽인다고 주장하지만, 실제 비용은 다른 곳에 있을 수 있다.
React 생태계는 내부적으로 활발히 혁신하고 있다—TanStack, Zustand, Jotai, shadcn/ui,
Radix UI 모두 React 생태계 안의 혁신이다.
혁신 자체가 억제된 것이 아니라, 혁신의 방향이 React의 프로그래밍 모델 안에 갇혀
있다는 것이 더 정확한 진단이다.

이것을 “인지 잠금(cognitive lock-in)”이라고 부를 수 있다.
개발자들이 React의 사고 모델—컴포넌트 트리, 단방향 데이터 흐름, 훅의 클로저 모델—을
내재화하면, 이 모델 밖에서 문제를 보는 능력이 약해진다.
Solid의 세밀한 반응성이나 Svelte의 컴파일 타임 접근이 “더 나은지”를 평가하는 것
자체가, React 모델에 익숙한 개발자에게는 어려운 인식론적 도전이다.
이것은 언어가 사고를 제한한다는 사피어-워프 가설(Sapir-Whorf hypothesis)의 기술
버전이다.
도구가 사고 방식을 형성하고, 형성된 사고 방식이 새로운 도구를 평가하는 기준이 된다.

이 관점에서 보면, 다양한 프레임워크 경험의 가치는 “각 프레임워크가 더 좋아서”가
아니라 “다른 프로그래밍 모델이 다른 사고 방식을 열어주기 때문”이다.
Elm을 배운 React 개발자가 상태 관리를 더 잘 설계하고, Svelte를 배운 개발자가
번들 크기에 더 민감해지는 것처럼.
모노컬처의 진짜 비용은 특정 혁신의 부재가 아니라, 생태계 전체가 특정 사고 모델의
시야 안에서만 문제를 정의하게 된다는 것이다.
