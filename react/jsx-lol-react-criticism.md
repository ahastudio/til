# jsx.lol: React 비판 발언 큐레이션

<https://jsx.lol/>

HN 토론: <https://news.ycombinator.com/item?id=48274077> (234점, 326개 댓글)

## 소개

jsx.lol은 David Bushell이 만든 React 및 관련 기술에 대한 비판적 발언 모음집이다.
개발자, 엔지니어, 기술 리더들이 React의 지배적 위치에 대해 표명한 우려와 비판을 날짜순으로 정리했다.
수록된 발언은 2023년부터 2026년까지 이어지며, 커뮤니티의 재평가가 지속적으로 진행되고 있음을 보여준다.

주요 발언자와 핵심 주장은 다음과 같다.

- **Craig Cook** (2026-03-24): “React는 모든 것이 못처럼 보이게 만드는 망치가 되었다.”
- **Sérgio Gomes** (2026-02-13): “JS 무거운 프로젝트는 광고한 것보다 느리고 시간이 지날수록 계속 느려진다.”
- **React Team** (2025-12-03): CVSS 10.0 점의 보안 취약점 CVE-2025-55182 — 인증 없는 원격 코드 실행 허용.
- **Jeremy Keith** (2025-11-26): “서버에서 JavaScript로 연산하고, HTML을 보내고, 그 JavaScript를 또 보낸다.”
- **Loren Stewart** (2025-10-28): “React의 모바일 전략은 팀을 플랫폼 포획 방향으로 몰아간다.”
- **David Bushell** (2025-10-23): “React의 혼란스러운 API 설계는 문서를 우유부단하게 만들고 올바른 사용법 논쟁을 끝없이 이어간다.”
- **Loren Stewart** (2025-09-16): “팀은 제약과 기술 적합성을 평가하는 대신 기본값으로 React를 선택한다.”
- **Mario(@mbrizic)** (2025-08-02): “3년 후 React로 돌아오면 생태계 전체를 다시 배워야 한다.”
- **Alex Russell** (2025-06-27): “React 커뮤니티는 깊고 측정 가능한 품질 위기에 직면했다.”
- **Abid Omar** (2025-06-12): “Next.js는 오픈소스 프레임워크로 위장한 Vercel 벤더 락인이다.”
- **Chris Ferdinandi** (2025-05-09): “HTML을 통한 점진적 향상(progressive enhancement)은 느린 연결이나 오류에서도 사용성을 보장한다.”
- **Kelly Sutton** (2025-01-18): “JS 무거운 프론트엔드의 '팻 클라이언트' 시대는 끝나가고 있다.”
- **Keith Cirkel** (2024-11-21): “React는 현대 UI를 거의 감당하지 못한다 — autofocus가 깨져 있고 커스텀 엘리먼트가 작동하지 않는다.”
- **Baldur Bjarnason** (2024-10-08): “React는 거짓 약속과 오해를 부르는 주장들로 가득 찬 비대한 시체다.”
- **Scott Jehl** (2024-10-05): “HTML 프레임워크 사용법: `/node_modules` 폴더를 찾아서 휴지통으로 드래그하라.”
- **Dusty Phillips** (2024-03-20): “React 앱을 3주 만에 Svelte로 재작성한 후 Svelte 진영에 합류했다.”
- **Jeremy Keith** (2024-01-31): “불만은 클라이언트사이드 React가 만들어낸 끔찍한 사용자 경험에 집중된다.”
- **Cassidy Williams** (2024-01-13): “복잡한 프로젝트에 여전히 React를 쓰지만, 기쁘지 않다.”
- **Mayank** (2023-12-29): “React는 Facebook 규모의 문제를 해결하는 레거시 프레임워크로, 대부분의 케이스에 맞지 않는다.”

비판의 주제는 여섯 갈래로 수렴한다.
성능 저하, 생태계 복잡성, 벤더 락인, 개발자 경험 대 사용자 경험의 불균형, 숙련 개발자 부족, 기술 적합성 없는 기본값 채택이다.

## 분석

### 비판의 시간적 축적과 위기 서사

2023년 말부터 2026년 초까지 이어지는 발언들은 단발성 불만이 아니라 커뮤니티의 집단적 재평가가
일어나고 있음을 보여준다.
초기에는 “Facebook 규모에 맞지 않는 도구”(Mayank, 2023)라는 기술적 비판이 중심이었다.
2024년에는 “끔찍한 사용자 경험”(Jeremy Keith)과 “autofocus 미작동”(Keith Cirkel) 같은 구체적 결함 지적으로
좁혀졌다.
2025~2026년에는 “팀의 기본값 선택”(Loren Stewart), “품질 위기”(Alex Russell), “Vercel 락인”(Abid Omar) 같은
생태계·비즈니스 구조 비판으로 확장됐다.
비판의 진화 방향은 기술적 결함 → 사용자 피해 → 구조적·정치적 문제로 이동했다.

### CVE-2025-55182: 기술 비판의 임계점

2025년 12월 React 팀이 CVSS 10.0의 보안 취약점을 공개한 것은 이 컬렉션에서 유일하게 수량화된
객관적 데이터다.
CVSS 10.0은 인증 없는 원격 코드 실행을 의미하며, 가능한 최고 심각도다.
이 항목이 컬렉션에 포함된 것은 “React의 위기”를 의견이 아닌 사실로 앵커링하는 효과를 가진다.
주관적 불만 발언들 사이에 하나의 객관적 사건이 삽입되는 큐레이션 방식은,
전체 컬렉션의 신뢰도를 높이는 동시에 비판의 심각성을 증폭시킨다.

### 비판자 스펙트럼의 다양성

수록된 비판자들은 단일한 대안을 지지하지 않는다.
Svelte(Dusty Phillips), 점진적 향상/바닐라 HTML(Chris Ferdinandi, Scott Jehl), 서버사이드 렌더링 재검토
(Jeremy Keith), 아예 다른 모바일 전략(Loren Stewart) 등 방향이 다양하다.
이것은 “React 비판”이 하나의 대안 진영에서 나오는 것이 아니라 다방면에서 동시에 제기된다는 증거다.
단일 대안 진영의 마케팅이 아닌, 다양한 맥락에서 수렴하는 불만이라는 점이 비판의 무게를 더한다.

### 점진적 향상 대 클라이언트사이드 렌더링의 재부상

Chris Ferdinandi와 Jeremy Keith의 발언은 2010년대에 잠시 유행했다가 React의 부상으로 퇴조한
“점진적 향상(progressive enhancement)” 철학의 귀환을 보여준다.
Web API가 성숙하고 서버사이드 렌더링이 다시 주목받으면서, 브라우저 네이티브 기능을 활용하는
접근법이 재평가받고 있다.
Scott Jehl의 “node_modules 휴지통에 드래그” 발언은 유머 형식이지만,
빌드 툴체인 의존성 없이 HTML과 네이티브 API만으로 충분하다는 입장을 압축한 것이다.

## 비평

### 강점: 시간순 아카이빙이 만드는 집합적 증언의 힘

개별 트윗이나 블로그 포스트는 일회성으로 소비되고 잊힌다.
jsx.lol의 큐레이션은 흩어진 발언들을 하나의 문서로 집결시켜 “많은 사람들이 오랫동안 같은 문제를
지적해왔다”는 집합적 증언을 만든다.
날짜가 표시되어 있어 비판이 일시적 유행이 아닌 지속적 흐름임을 보여주는 것도 효과적이다.

### 한계: 선택 편향(selection bias)의 필연성

큐레이션은 본질적으로 주관적 선택이다.
이 컬렉션에는 React를 긍정적으로 평가하는 발언이 없다.
수십만 명의 React 개발자 중 비판적 목소리만을 선별했기 때문에,
컬렉션 전체가 “React는 나쁘다”는 결론을 향해 구성된 것처럼 보인다.
비판의 내용이 타당하더라도, 반론 없이 한 방향의 발언만 모으는 방식은 논증보다 선동에 가깝다.
독자가 이 컬렉션을 React에 대한 균형 잡힌 평가로 받아들인다면 오해가 생긴다.

fishtoaster는 이 편향을 직접 지적한다.[^fishtoaster]
16년간의 JavaScript 프레임워크 역사를 되돌아보며 React를 “우리가 시도한 다른 모든 것들을 제외하면
최악의 JS 프레임워크”라고 표현한다. Angular 1, Backbone, jQuery, 바닐라 JS 시대와 비교하면
React는 여전히 진보였다는 것이다. pibaker는 시장 선택의 논리를 든다.[^pibaker]
“React를 쓰도록 강제받는 사람은 아무도 없다. JavaScript는 강제되지만 React는 아니다.
그런데도 React가 선택받는다.” 이 사실이 React가 실제로 뭔가를 제공한다는 증거라는 주장이다.

### 보완이 필요한 관점: React가 지배를 유지하는 이유

이 컬렉션은 “왜 React가 나쁜가”를 다루지만, “그렇다면 왜 React는 여전히 지배적인가”를 다루지 않는다.
채용 시장, 생태계 성숙도, 기업의 기술 리스크 회피 성향, Next.js의 풀스택 전환 등
React가 지배력을 유지하는 구조적 이유를 함께 제시했다면 더 완전한 그림이 됐을 것이다.

moojacob은 React가 제공하는 실질적 가치를 구체적 사례로 설명한다.[^moojacob]
받은 편지함 페이지에서 뒤로 가기 버튼을 보여줄지 판단하는 로직을 예로 든다.
“JavaScript가 이전 페이지가 받은 편지함인지 판단하고, 그 값에 따라 JSX를 렌더링하는 것이
모두 하나의 컴포넌트 파일에 있다.” HTMX나 Hotwire로는 이 로직이 분산된다.
manlymuppet은 더 넓은 맥락을 제시한다.[^manlymuppet]
React가 불만스러워도 “코드로 시각적 디자인을 직관적으로 표현하는 문제는 아직 누구도 제대로 풀지 못했다.”
React 비판은 결국 이 미해결 문제의 불편한 현재 근사값에 대한 비판이기도 하다.

## 인사이트

### 큐레이션은 커뮤니티 감정의 압축이다

기술 커뮤니티의 감정을 측정하는 전통적 방법은 설문, 인터뷰, 사용 통계다.
jsx.lol이 시도하는 것은 다른 방법이다.
공개적으로 표명된 불만의 밀도와 연속성으로 커뮤니티 감정의 방향을 보여주는 것이다.
이것은 정량적 측정이 아니라 정성적 아카이빙이지만, 그 자체로 유효한 신호다.

특히 Cassidy Williams의 발언 — “복잡한 프로젝트에 여전히 React를 쓰지만, 기쁘지 않다” — 은
컬렉션에서 가장 울림이 있는 항목 중 하나다.
이것은 React 기피가 아니다.
React를 쓰면서도 그것에 만족하지 못하는 “비자발적 사용자”의 존재를 증언한다.
이 감정 — 쓰지만 좋아하지 않는 — 은 기술의 잠재적 대체 가능성을 가장 정확하게 예측하는 신호다.
도구에 대한 분노는 이미 이탈한 사람의 감정이고, 도구에 대한 무관심은 비사용자의 감정이다.
하지만 “쓰지만 기쁘지 않다”는 현재 사용자의 이탈 직전 감정이다.
이 감정이 충분히 넓게 퍼지면, 전환의 임계점이 가까워진다.

### “기본값의 폭정”은 기술 생태계의 보편 법칙이다

Loren Stewart와 Craig Cook의 비판이 공유하는 핵심은 React가 기술적 적합성이 아닌
관성으로 선택된다는 것이다.
이것은 React 고유의 문제가 아니다.
한 세대 전에는 jQuery가, 그 전에는 Flash가 같은 위치에 있었다.
지배적 기술이 된 도구는 “이게 정말 이 문제에 맞는가”를 묻지 않고 선택된다.
망치를 쥔 사람에게 모든 문제가 못처럼 보이는 것처럼(Craig Cook의 비유),
팀이 React를 기본값으로 쥐고 있으면 모든 프론트엔드 문제가 React 문제처럼 보인다.

이 패턴에는 역설이 있다.
기술이 지배적이 될수록 그 기술을 쓸 이유가 더 강해진다 — 개발자 채용, 문서, 라이브러리 생태계 모두
지배적 기술 주변에 집중되기 때문이다.
그런데 동시에 기술이 지배적일수록 부적절한 맥락에서 남용될 확률도 높아진다.
적합하지 않아도 쓰게 되는 것이다.
지배적 기술의 평균 품질이 시간이 지나도 개선되지 않거나 오히려 악화되는 이유가 여기 있다.
적합한 곳에만 쓰이는 것이 아니라 모든 곳에 쓰이기 때문이다.
jsx.lol은 이 법칙이 현재 React에 적용되고 있음을 실시간으로 기록한다.

### Next.js와 Vercel 락인: 오픈소스 위장의 구조

Abid Omar의 “Next.js는 Vercel 벤더 락인을 위장한 오픈소스 프레임워크”라는 발언은
오픈소스 생태계의 새로운 포획 전략을 정확히 지적한다.
전통적인 벤더 락인은 소유권 기반이었다.
독점 라이선스, 독점 API, 독점 데이터 포맷이 이탈을 막았다.
그런데 새로운 세대의 락인은 오픈소스 코드를 통해 작동한다.

Next.js 자체는 오픈소스이고 누구나 자체 서버에서 실행할 수 있다.
그러나 Vercel이 제공하는 Edge Functions, ISR(Incremental Static Regeneration),
Image Optimization, Middleware 같은 기능은 Vercel 인프라에서 가장 자연스럽게 동작하도록
최적화되어 있다.
자체 호스팅으로 마이그레이션하면 기능 손실이 발생하고, 손실된 기능을 직접 구현하는 비용이 발생한다.
코드는 오픈소스지만 운영 경험은 Vercel에 최적화되어 있는 것이다.

이것은 AWS가 오픈소스 도구(Elasticsearch, Redis, MongoDB)를 관리형 서비스로 감싸서
원 프로젝트의 차별화를 희석시킨 전략과 구조적으로 대칭이다.
AWS가 기존 오픈소스 위에서 클라우드 서비스를 만들었다면,
Vercel은 클라우드 서비스에 최적화된 오픈소스 프레임워크를 만들었다.
방향이 반대지만 결과는 같다 — 프레임워크를 깊이 사용할수록 특정 클라우드에서 이탈하기 어려워진다.
오픈소스 라이선스가 이탈을 기술적으로 막지 않지만, 운영 경험의 최적화가 이탈을 경제적으로 어렵게 만든다.
이것이 오픈소스 시대의 새로운 락인 문법이다.
