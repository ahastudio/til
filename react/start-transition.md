# startTransition은 언제 진짜로 필요한가

원문: <https://tigerabrodi.blog/when-do-you-really-need-starttransition>

## 요약

`startTransition`은 React에게 “이 상태 업데이트는 중요하지만 기다릴 수 있다”는 신호를 보내는 API다.
React 18이 도입한 동시 렌더링(Concurrent Rendering) 모델에서, 렌더링 작업을 작은 청크로 분할하고 각 청크 사이에 더 긴급한 작업이 있는지 확인한다.
`startTransition`으로 감싼 업데이트는 “전환 차선(transition lane)”에 배치되어 입력·클릭 같은 긴급 업데이트에 밀리면 중단(abandon)된다.

이 API가 유용한 상황은 구체적이다.
AI 토큰 스트리밍에서 초당 10~30개 토큰이 도착할 때 매번 마크다운 블록을 재렌더링하면 브라우저가 페인팅할 시간을 확보하지 못한다.
수천 개 항목을 필터링하는 목록에서 `startTransition` 없이 입력에 반응하면 키스트로크가 “얼어붙는다”.
탭 전환 시 탭 하이라이트(긴급)와 콘텐츠 렌더링(전환)을 분리하면 즉각적인 피드백과 매끄러운 콘텐츠 로딩을 동시에 얻는다.

반대로 써서는 안 되는 상황도 명확하다.
16ms 이하로 빠른 렌더링에 쓰면 스케줄링 오버헤드만 추가된다.
사용자 입력값(`<input>` value)을 `startTransition`으로 감싸면 입력이 즉시 반응하지 않아 오히려 나빠진다.
단일 소규모 상태 업데이트에는 이득보다 비용이 크다.

Debouncing과의 차이가 중요하다.
Debouncing은 사용자가 입력을 멈추고 300ms 후에 작업을 시작하는 인위적 지연이다.
`startTransition`은 즉시 작업을 시작하되 더 긴급한 업데이트가 오면 중단한다.
네트워크 요청에는 debouncing을, 후속 렌더링에는 `startTransition`을 쓰는 것이 권장 패턴이다.

`useTransition` 훅은 `startTransition` 함수와 `isPending` 불리언을 함께 제공한다.
전환이 진행 중일 때 로딩 표시를 보여주는 데 쓴다.
`requestIdleCallback`과 달리 React reconciler와 통합되어 있어 렌더링 중단이 가능하다.

## 분석

### 동시 렌더링 모델의 핵심: 중단 가능성

React 18 이전의 렌더링은 동기적이었다.
한번 시작된 렌더링은 끝날 때까지 메인 스레드를 점유했다.
동시 렌더링은 렌더링을 작은 청크로 쪼개고 각 청크 사이에 체크포인트를 둔다.
`startTransition`이 유의미한 것은 이 체크포인트에서 “더 긴급한 업데이트가 있으면 이 렌더링을 버려도 된다”고 표시하기 때문이다.

핵심 단어는 “abandon(버린다)”이다.
새 입력이 들어오면 진행 중이던 전환 렌더는 완료를 기다리지 않고 버려지고 새 렌더가 시작된다.
이것이 debouncing과 근본적으로 다른 이유다.
debouncing은 작업을 지연시키고, `startTransition`은 작업을 즉시 시작하되 중단 가능하게 만든다.

### 레인(Lane) 개념의 실용적 의미

React의 우선순위 시스템은 렌더링 작업을 레인으로 분류한다.
긴급 레인에는 사용자 입력, 클릭, 포커스 이벤트가 들어간다.
전환 레인에는 `startTransition`으로 표시된 업데이트가 들어간다.
레인이 다르면 긴급 레인이 항상 먼저 처리되고, 전환 레인은 긴급 레인이 비워진 뒤에 처리된다.

실제 UI 패턴에 대입하면 이해가 쉽다.
탭 클릭 시 탭의 활성 상태는 긴급(즉시 반응 필요), 탭 내용 렌더링은 전환(약간 늦어도 무방).
검색어 입력은 긴급, 검색 결과 목록 업데이트는 전환.
이 분류가 올바르게 이루어질 때 UI가 반응형(responsive)으로 느껴진다.

### `requestIdleCallback`과의 결정적 차이

`requestIdleCallback`은 브라우저 API로, 메인 스레드가 유휴 상태일 때 콜백을 실행한다.
그러나 React reconciler와 통합되어 있지 않아 렌더링 중단이 불가능하다.
`startTransition`은 React 내부 스케줄러와 연결되어 reconciler 수준에서 중단·재개가 가능하다.
외부 브라우저 API로 React 렌더링을 제어하려는 시도가 왜 작동하지 않는지 이 차이가 설명한다.

## 비평

### 사용 조건이 실제로 얼마나 흔한가

저자가 제시하는 적합한 사례(AI 스트리밍, 수천 개 항목 필터링, 복잡한 탭 콘텐츠)는 모두 특수한 상황이다.
대부분의 React 앱에서 렌더링은 16ms 이내로 완료되고, `startTransition`이 필요한 상황은 생각보다 드물다.
“언제 써야 하는가”를 설명하는 글이면서도 “대부분의 경우 필요 없다”는 점이 충분히 강조되지 않는다.
이 글을 읽고 모든 상태 업데이트에 `startTransition`을 적용하는 개발자가 생길 수 있다.

### 성능 측정 없이 사용하는 위험

`startTransition`이 효과적인지 확인하려면 React DevTools Profiler나 Performance 탭에서 측정이 필요하다.
그러나 글은 “언제 쓰는가”의 원칙만 설명하고, 실제로 적용 전후 성능을 어떻게 검증하는지는 다루지 않는다.
감으로 적용하는 `startTransition`은 의도한 효과 없이 코드 복잡성만 높일 수 있다.

### Server Actions와의 관계 미언급

React 19에서 Server Actions와 `useTransition`의 통합이 강화됐다.
`useTransition`으로 감싼 상태에서 Server Action을 호출하면 `isPending`이 서버 왕복 시간 동안 유지된다.
이 글은 클라이언트 렌더링 비용에만 집중하고, 서버와의 연동에서 `startTransition`이 어떻게 작동하는지 다루지 않는다.
React의 풀스택 방향을 고려하면 이 부분이 점점 더 중요해진다.

## 인사이트

### 사용자 경험은 절대 시간이 아니라 상대적 반응성으로 결정된다

UI 성능 논의에서 흔한 오해는 “빠름”이 절대적 기준이라는 것이다.
300ms 안에 렌더링이 완료되면 빠르고, 그보다 오래 걸리면 느리다는 식의 단순화.
그러나 사용자가 실제로 느끼는 것은 절대 시간이 아니라 자신의 행동과 화면 반응 사이의 지연이다.
입력에 즉시 반응하면서 목록이 약간 늦게 업데이트되는 것은 쾌적하다.
입력도 목록도 동시에 얼어붙으면 300ms도 참을 수 없다.

`startTransition`의 철학은 이 직관을 API로 구현한 것이다.
모든 것을 빠르게 만들 수 없다면, 사용자와 직접 상호작용하는 부분만큼은 반드시 즉시 반응하게 한다.
나머지는 여유 시간에 처리한다.
“빠름”의 기준을 절대 속도에서 상호작용 우선순위로 바꾸는 사고 전환이 이 API의 핵심이다.

### 렌더링 중단 가능성은 새로운 프로그래밍 패러다임이다

기존 동기 렌더링에서 상태 업데이트는 원자적이었다.
시작하면 반드시 끝난다. 중간에 취소할 수 없다.
동시 렌더링의 “abandon” 개념은 이 전제를 깨뜨린다.
렌더링이 완료되지 않고 버려질 수 있다는 것은, 렌더링 함수가 순수(pure)해야 한다는 요구를 강화한다.

중단된 렌더링이 부작용을 남기면 앱 상태가 일관성을 잃는다.
이것이 React가 Strict Mode에서 렌더링 함수를 두 번 호출하는 이유다.
부작용이 있는 컴포넌트를 찾아내기 위해.
`startTransition`을 쓴다는 것은 “이 렌더링은 언제든 버려질 수 있다”는 약속이며, 그 약속을 지키려면 렌더링 순수성이 필수 조건이 된다.

### 우선순위 시스템은 복잡성을 사용자에게 감추는 계층 추상화다

멀티태스킹 운영체제는 프로세스 우선순위를 관리해 사용자가 느끼는 반응성을 유지한다.
음악이 재생되는 동안 파일 복사가 진행될 수 있는 것은 OS가 오디오 재생에 높은 우선순위를 부여하기 때문이다.
React의 레인 시스템은 이 아이디어를 UI 렌더링에 적용한 것이다.

개발자가 `startTransition`으로 “이건 덜 긴급하다”고 표시하면, 나머지 우선순위 관리는 React 내부에서 이루어진다.
개발자는 작업의 상대적 중요도만 선언하고, 실제 스케줄링은 프레임워크에 위임한다.
이것은 선언적(declarative) 프로그래밍 패러다임의 확장이다.
“어떻게 그리는가”를 React에 위임한 것처럼, “언제 그리는가”도 React에 위임하는 방향으로 추상화 수준이 높아지고 있다.
Server Components가 “무엇을 서버에서 렌더링할지”를 선언하는 것과 같은 패턴이다.
React는 점점 더 많은 렌더링 결정을 개발자 대신 내리는 방향으로 진화하고 있다.
