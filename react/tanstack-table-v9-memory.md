# TanStack Table V9: 프로토타입 공유로 메모리 90% 절감

원문: [How an Underrated Refactor Saved 90% Memory Usage | TanStack Blog](https://tanstack.com/blog/tanstack-table-v9-memory-performance)

한국어 번역: <https://ykss.netlify.app/translation/2026/how-an-underrated-refactor-saved-90-percent-memory-usage/>

## 요약

TanStack Table V9는 단 하나의 아키텍처 변경으로 V8 대비 메모리 사용량을 최대 90%까지 줄였다.
핵심은 행(row), 열(column), 셀(cell), 헤더(header) 객체에 공유 프로토타입(shared prototype)을 도입한 것이다.

벤치마크 결과는 데이터 규모가 클수록 격차가 벌어진다.
10만 행 × 8열 기준 245MB, 100만 행 × 8열 기준 2,452MB를 절약했으며
절감률은 각각 90.0%와 90.5%다.
가상화(virtualization)를 적용한 100열 × 1,000 가상화 컬럼 시나리오에서도 65.2% 절감이 확인됐다.
반면 10~80행 수준의 소규모 데이터에서는 차이가 거의 없다.

V8에서는 각 행 객체가 모든 메서드의 복사본을 가지면서, 행별 상태를 캡처하는 클로저 스코프까지 함께 생성됐다.
수백만 개의 행을 만들면 동일한 함수가 수백만 번 복제되는 구조였다.
V9에서는 메서드를 프로토타입에 한 번만 정의하고, `Object.create()`로 생성한 인스턴스가 해당 프로토타입을 참조한다.
인스턴스에는 고유한 값(id, index, original)만 저장되고, 메서드는 `this`를 통해 객체별 컨텍스트를 받는다.

클래스(class)를 쓰지 않은 이유는 TanStack Table의 피처 시스템이 기능을 동적으로 조합하기 때문이다.
예를 들어 열 가시성(column-visibility) 피처를 사용하는 테이블에만 `toggleVisibility()` 메서드가 존재한다.
클래스 기반 믹스인 체인이나 복잡한 상속 구조는 이 “필요한 것만 지불한다(pay only for what you use)” 철학과 맞지 않는다.
대신 각 피처는 `assignRowPrototype()` 훅을 구현해 프로토타입에 메서드를 등록하며,
프로토타입은 테이블 인스턴스당 한 번만 생성돼 재사용된다.

이 변경에는 단 하나의 파괴적 변경(breaking change)이 수반됐다.
메서드를 구조 분해(destructuring)하면 `this` 컨텍스트가 끊어진다는 점이다.
ES 모듈의 기본값인 엄격 모드(strict mode)에서 `this` 없는 메서드 호출은 오류를 낸다.
따라서 `const { getValue } = row` 형태는 V9에서 동작하지 않으며,
반드시 `row.getValue('name')` 형태로 호출해야 한다.
이 하나의 트레이드오프가 이 변경을 V9 메이저 릴리스까지 미룬 이유였다.

## 분석

### 이 최적화가 “단순한 리팩터”로 불릴 수 있는 이유

글의 부제는 “underrated refactor”다.
실제로 변경 사항은 코드 수준에서 단순하다:
`Object.create()`로 객체를 생성하고, 클로저 대신 `this`로 컨텍스트를 전달한다.
패턴 자체는 JavaScript 초기 시절부터 존재했고, ES6 클래스도 내부적으로 동일한 프로토타입 메커니즘을 사용한다.

그럼에도 V8에서 이 구조를 채택하지 못했던 이유는 동적 피처 조합이라는 설계 제약 때문이다.
클래스 상속으로는 “이 테이블 인스턴스에만 이 메서드를 추가”하는 구조를 깔끔하게 표현하기 어렵다.
프로토타입을 직접 조작하는 방식이 라이브러리 내부 구조와 가장 잘 맞았던 것이다.
즉, 성능 최적화가 가능했던 것은 “언제나 가능했던” 일이 아니라,
라이브러리 아키텍처가 그 최적화를 수용할 수 있는 형태로 발전했기 때문이다.

### JavaScript 객체 모델과 메모리의 관계

JavaScript 엔진(V8, SpiderMonkey 등)은 객체 형태(shape, hidden class)를 추적해 최적화한다.
같은 프로퍼티 구조를 가진 객체들을 한 그룹으로 묶어 인라인 캐시를 공유한다.
하지만 클로저가 붙은 함수 객체는 각각 별도의 메모리 할당이 필요하다.
클로저는 외부 스코프의 변수를 캡처하는 “살아있는” 참조를 담고 있기 때문이다.

TanStack Table V8의 행 객체는 본질적으로 “데이터 + 해당 데이터를 다루는 함수들”의 묶음이었다.
이 패턴은 팩토리 함수 방식의 객체 생성에서 자연스럽게 나타나는데,
캡슐화가 깔끔하고 클로저 덕분에 `this` 바인딩 문제가 없다는 장점이 있다.
그러나 수백만 인스턴스를 만들면 같은 함수 코드가 수백만 번 별도 할당된다.
V9의 프로토타입 방식은 이 함수 객체 중복을 제거해 메모리 절감을 이끌어낸다.

### 벤치마크가 말하는 것과 말하지 않는 것

90% 절감이라는 수치는 순수하게 JavaScript 힙 메모리를 기준으로 한다.
Playwright와 Chrome DevTools Protocol을 사용해 GC 이후 잔류 힙을 측정한 방법론은 신뢰할 만하다.
그러나 이 수치가 실제 애플리케이션에서 그대로 재현된다고 단정하기는 어렵다.

실제 앱에서는 TanStack Table 외에도 수많은 객체가 힙을 사용한다.
React의 가상 DOM 트리, 상태 관리 라이브러리의 구독 객체, 이미지/폰트 캐시 등이 공존한다.
100만 행 시나리오에서 테이블 단독으로 2.4GB를 절약한다는 것은 인상적이지만,
애플리케이션 전체 메모리 중 테이블이 차지하는 비율에 따라 체감 효과는 달라진다.

### 동적 피처 조합이라는 설계 철학의 비용

TanStack Table의 “pay only for what you use” 접근법은 번들 크기와 런타임 비용 모두에서 유리하다.
하지만 이 유연성은 타입 시스템과의 통합에서 복잡도를 높인다.
사용하는 피처 조합에 따라 행 객체의 실제 타입이 달라지기 때문이다.
TypeScript에서 이를 정확히 모델링하려면 제네릭과 조건부 타입을 많이 활용해야 한다.

이번 V9의 프로토타입 방식도 마찬가지다.
`assignRowPrototype()` 훅을 통해 런타임에 프로토타입이 구성되므로,
정적 분석 도구가 어떤 메서드가 존재하는지 확신하기 어렵다.
TypeScript 타입 정의가 이 동적 구성을 올바르게 반영하는지가 실제 사용 경험의 핵심이 된다.

## 비평

### 구조 분해 금지라는 제약은 생각보다 광범위하다

글은 파괴적 변경을 “단 하나”로 축소하고, 간단한 코드 예시를 통해 쉽게 해결 가능한 것처럼 제시한다.
그러나 `const { getValue } = row` 패턴이 실제로 얼마나 퍼져 있는지는 과소평가되어 있다.

이 패턴이 문제가 되는 것은 직접 코드에서만이 아니다.
`row`를 인자로 받아 내부에서 구조 분해하는 유틸 함수, 스프레드 연산자로 복사하는 래퍼,
서드파티 라이브러리가 객체를 다루는 방식까지 영향을 받을 수 있다.
특히 `Object.assign({}, row)`나 `{...row}` 같은 얕은 복사는 프로토타입 메서드를 복사하지 않으므로
복사된 객체에서 메서드를 호출하면 실패한다.

글은 마이그레이션 비용을 충분히 논의하지 않는다.
대규모 코드베이스에서 이 패턴을 찾아 수정하는 것은 기계적 치환이 불가능한 경우도 많고,
런타임에서야 드러나는 버그가 될 수 있다.
90%의 메모리 절감은 분명 매력적이지만, 글이 제시하는 것보다 업그레이드 비용이 클 수 있다.

### 벤치마크는 최악의 시나리오를 기준으로 한다

글이 제시하는 수치들은 10만~100만 행 규모에서 나온다.
대부분의 실제 테이블 컴포넌트 사용 사례가 이 규모에 해당하지 않는다.
글 스스로 소규모 데이터(10~80행)에서는 차이가 미미하다고 인정하지만,
이 사실이 제목과 전체 서술의 흐름에서 충분히 부각되지 않는다.

실제로 많은 웹 앱은 페이지네이션이나 서버사이드 렌더링을 통해 DOM에 표시되는 행 수를 수백 단위로 유지한다.
이 경우 V9로의 업그레이드가 체감 가능한 성능 향상을 가져올지는 의문이다.
90% 절감이라는 수치가 마케팅 제목으로서는 강렬하지만,
자신의 사용 사례에 적용 가능한지를 판단하는 정보는 글에서 충분히 제공되지 않는다.

### 프로토타입 직접 조작이 장기적으로 유지보수에 유리한가

글은 클래스를 쓰지 않은 이유를 설명하며 동적 조합의 어려움을 든다.
그러나 `assignRowPrototype()` 훅으로 런타임에 프로토타입을 조작하는 방식은
JavaScript 생태계의 일반적 패턴에서 벗어나 있다.

대부분의 JavaScript 개발자는 프로토타입을 직접 다루는 코드를 마주쳤을 때 당혹스러워한다.
`Object.create()`, `Object.getPrototypeOf()`, 프로토타입 체인 순회는 이해는 되지만
일상적으로 쓰지 않는 API다.
이 패턴을 내부에서 사용한다면 라이브러리 기여자(contributor)의 진입 장벽이 높아지고,
버그 추적 과정에서 예상치 못한 곳에서 막히는 경험을 낳을 수 있다.

클래스 기반 믹스인이 “어색하다(awkward)”고 서술하지만, 그것이 사용자 경험에서
더 어색한지, 아니면 프로토타입 직접 조작이 더 어색한지는 팀의 배경에 따라 달라진다.
TypeScript의 데코레이터나 다른 합성(composition) 패턴도 대안이 될 수 있었을 텐데,
이 방향을 왜 탐색하지 않았는지에 대한 설명이 없다.

## 인사이트

### 클로저 남용은 함수형 스타일의 보이지 않는 세금이다

JavaScript 생태계는 지난 10년 동안 클로저 기반 팩토리 함수 스타일을 선호해왔다.
React 훅, Zustand, Jotai 등 현대적인 상태 관리 라이브러리들이 이 패턴을 권장한다.
클로저는 `this` 바인딩 문제를 회피하고, 코드를 선언적으로 쓸 수 있게 해주며,
테스트하기 쉬운 순수 함수와 잘 어울린다.

그러나 이 스타일이 대규모 인스턴스 생성과 만나면 메모리 문제가 된다.
TanStack Table V8의 사례는 이 패턴의 대표적 피해 사례다.
개발자가 로컬에서 작은 샘플 데이터로 개발하고 테스트할 때는 이 문제가 전혀 드러나지 않는다.
프로덕션에서 수만 행의 실제 데이터를 다룰 때 비로소 메모리 한계가 나타난다.

이 현상은 TanStack Table에만 국한되지 않는다.
가상 DOM 라이브러리, 상태 관리 라이브러리, ORM의 모델 객체 생성 등
반복 생성이 많은 모든 영역에서 같은 함정이 존재한다.
클로저가 편리하다는 사실과, 그것이 대량 인스턴스화 맥락에서 부적절하다는 사실은 서로 독립적이다.
“함수형이니까 메모리도 효율적”이라는 가정은 검토가 필요하다.

### 메이저 버전 업그레이드 전략으로서 “단 하나의 파괴적 변경”

TanStack Table이 이 최적화를 V8에서 선택적 플래그로 도입하지 않고,
V9 메이저 버전의 파괴적 변경으로 처리한 결정은 흥미롭다.
이는 “이전 버전과의 호환성”과 “올바른 기본값”을 맞바꾸는 전략이다.

구조 분해 금지라는 제약이 모든 사용자에게 영향을 주는 것은 아니다.
메서드를 항상 `row.getValue()` 형태로 호출하던 코드는 변경 없이 동작한다.
그러나 호환성 레이어를 유지하면서 최적화를 도입하면 내부 복잡도가 늘어난다.
메이저 버전에서 깔끔하게 끊는 것이 라이브러리 장기 관리 비용을 낮춘다는 판단이다.

이 패턴은 JavaScript 생태계 전반에서 점점 일반화되고 있다.
Vue 3, Angular v18, React Router v7 등 주요 프레임워크가 비슷한 방식을 택했다.
과거 방식과의 완벽한 호환성 대신, 새 버전에서 더 나은 기본값을 선택하고
마이그레이션 가이드를 제공하는 전략이다.
라이브러리 생태계의 무게 중심이 “하위 호환”에서 “점진적 단절”로 이동하고 있다.

### 성능 최적화의 발견 타이밍이 아키텍처 복잡도를 결정한다

이 최적화가 V8이 아닌 V9에서 이루어진 것은, 기술적 불가능 때문이 아니라 설계 제약 때문이었다.
`assignRowPrototype()` 훅이라는 내부 확장 인터페이스가 마련되기 전까지는
프로토타입 공유 방식을 피처 조합 시스템과 통합하기 어려웠다.

이 사례는 성능 최적화를 언제 도입하느냐가 얼마나 중요한지를 보여준다.
설계 초기에 “수백만 행을 다뤄야 할 수도 있다”는 전제를 가졌다면
처음부터 프로토타입 기반 설계를 선택했을 것이다.
그러나 실제로는 라이브러리가 성장하고 사용 사례가 드러난 뒤에야 병목이 확인됐다.

이 경험은 라이브러리 설계에서 “나중에 최적화할 수 있다”는 믿음에 경계를 요구한다.
클로저 기반에서 프로토타입 기반으로의 전환은 API 파괴 없이는 불가능했다.
즉, 초기 설계 선택이 나중의 최적화 옵션을 닫아버렸다.
성능 민감 영역의 라이브러리라면 초기 단계에서 극단적 사용 사례를 시뮬레이션하는 비용이
나중의 메이저 버전 파괴적 변경 비용보다 훨씬 낮을 수 있다.
