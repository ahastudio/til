# React Doctor

원문: <https://github.com/millionco/react-doctor>

웹사이트: <https://www.react.doctor/>

## 소개

React Doctor는 React 코드베이스를 결정론적으로 스캔하여 상태와 이펙트(state & effects),
성능, 아키텍처, 보안, 접근성 전반의 문제를 찾아내는 정적 분석 도구다.
2026년 2월에 공개된 이후 11,000개 이상의 GitHub 스타를 획득했으며,
Next.js, Vite, TanStack, React Native, Expo 등 React 기반 프레임워크와 라이브러리를
모두 지원한다.

태그라인이 직설적이다 — “Your agent writes bad React. This catches it.”
즉, AI 에이전트가 작성하는 React 코드의 품질을 사후 검증하는 도구로 포지셔닝한다.
단순한 린트 도구가 아니라 에이전트 워크플로와의 통합을 명시적으로 설계 목표로 삼는다.

## 사용법

### 빠른 시작

프로젝트 루트에서 아래 명령 하나로 코드베이스 전체를 감사할 수 있다.

```bash
npx react-doctor@latest
```

변경된 파일만 검사하려면 `--diff` 플래그를 사용한다.

```bash
npx react-doctor@latest --verbose --diff
```

### 에이전트용 설치

감사를 완료한 뒤 아래 명령을 실행하면 코딩 에이전트가 발견된 문제를 학습하고
미래에 같은 실수를 반복하지 않도록 스킬(skill)을 설치한다.

```bash
npx react-doctor@latest install
```

Claude Code, Cursor, Codex, OpenCode 등 주요 에이전트를 지원한다.

### CI 통합

GitHub Actions Marketplace에서 재사용 가능한 액션을 제공하며,
PR마다 자동으로 스캔하여 인라인 주석과 어노테이션을 남긴다.

## 아키텍처

프로젝트는 pnpm 워크스페이스 기반의 모노레포로 구성된다.

| 패키지                          | 공개 여부   | 역할                                    |
| ------------------------------- | ----------- | --------------------------------------- |
| `core`                          | 비공개      | 진단 엔진 (Effect v4 기반 스트리밍 오케스트레이터)   |
| `api`                           | 비공개      | 프로그래매틱 `diagnose()` 인터페이스     |
| `react-doctor`                  | 공개        | CLI + 공개 `inspect()` + 바이너리       |
| `oxlint-plugin-react-doctor`    | 공개        | 100개 이상의 규칙을 담은 oxlint 플러그인 |
| `eslint-plugin-react-doctor`    | 공개        | 동일 규칙의 ESLint 미러 플러그인        |
| `website`                       | 비공개      | 문서 사이트                             |

핵심 엔진은 Effect(함수형 이펙트 시스템) v4를 기반으로 구축되었다.
Files, Git, Project, Config, Linter, DeadCode, Score, Reporter, Progress 등
10개의 `Context.Service` 클래스가 의존성 주입 레이어로 구성된다.

## 규칙 시스템

100개 이상의 규칙은 oxlint 플러그인 형태로 구현되며, ESLint 플러그인은 이를 미러링한다.
각 규칙은 한 문장으로 정의 가능해야 하며, 공식 문서 `HOW_TO_WRITE_A_RULE.md`는
규칙 작성의 전체 워크플로를 상세히 기술한다.

규칙의 품질 기준은 다음과 같다.

- **Specific**: 하나의 명확한 문제만 잡는다.
- **Grounded**: 공식 문서, 실제 코드, 이슈, PR, 또는 eval로 검증된 문제를 다룬다.
- **Precise**: 디텍터(detector)가 진단 메시지가 설명하는 동작과 정확히 일치한다.
- **Low-noise**: 거짓 양성(false positive)은 정확성 버그로 취급한다.
- **Tested adversarially**: 테스트는 명백한 잘못된 예시뿐 아니라 경계 사례를 포함한다.
- **Scoped**: v1은 인접한 규칙 아이디어를 섞지 않는다.
- **Readable**: 헬퍼(helper) 이름이 정확한 의미(semantics)를 서술한다.

규칙의 복잡도에 따라 구문(syntax)만 분석하는 단순 규칙부터,
임포트 바인딩을 추적하는 스코프 인식 규칙,
제어 흐름을 따라가는 경로 인식 규칙까지 세 가지 정밀도 수준으로 분류한다.

## 분석

### Effect v4 기반 설계의 의도

핵심 엔진이 Effect v4를 채택한 것은 단순한 기술적 선택이 아니다.
정적 분석 도구는 수십 개의 서비스(파일 시스템, Git, 린터, 리포터 등)가
동시에 동작하면서 오류를 정밀하게 처리해야 한다.
Effect는 이 복잡성을 타입 수준에서 명시적으로 다루게 해준다.

에러 처리 구조가 특히 치밀하다.
`ReactDoctorError`라는 단일 최상위 에러 타입에 `reason: Schema.Union([...])`을 두어,
각 서비스가 자신만의 `Schema.TaggedErrorClass`를 정의하고
렌더러(renderer)가 `error.reason._tag`로 타입 안전하게 디스패치한다.
`error.message.includes(...)` 같은 문자열 파싱은 명시적으로 금지된다.

OTel(OpenTelemetry) 트레이싱도 `Effect.fn(“Service.method”)`와
`Effect.withSpan(...)` 조합으로 무비용 기본 설정을 제공한다.
사용자가 `REACT_DOCTOR_OTLP_ENDPOINT`와 `REACT_DOCTOR_OTLP_AUTH_HEADER`를
설정하면 자동으로 Axiom 등 외부 백엔드로 스팬이 전송된다.

### 에이전트 통합의 방향성

`npx react-doctor@latest install` 명령이 설치하는 것은 파일이 아니라 “스킬”이다.
즉, 에이전트가 다음 코딩 세션에서 같은 패턴을 반복하지 않도록
진단 결과를 에이전트의 컨텍스트 또는 규칙으로 주입한다.
이는 정적 분석 도구를 단발성 검사 도구가 아니라
에이전트의 장기 학습 루프에 통합하려는 설계다.

CI에서 PR마다 스캔하고 인라인 어노테이션을 남기는 것도 같은 맥락이다.
에이전트가 PR을 생성하면 React Doctor가 즉시 피드백을 달고,
에이전트는 그 피드백을 다음 커밋에서 반영한다는 피드백 루프를 상정한다.

### 결정론적 스캔의 위치

React Doctor는 기존 ESLint / oxlint 설정을 대체하지 않는다.
기존 ignore 파일과 린트 설정을 자동으로 인식하며,
ESLint / oxlint 플러그인으로도 설치 가능하다.
포지셔닝은 “에이전트가 놓치는 React 특화 패턴”을 결정론적으로 잡는
보완적 레이어다.

## 비평

### 강점

100개 이상의 규칙이 공식 문서와 실제 OSS 코드(RocketChat, Sentry, PostHog, Grafana,
ToolJet 등)를 근거로 검증된다는 점은 신뢰할 수 있는 강점이다.
`HOW_TO_WRITE_A_RULE.md`에서 요구하는 RDE(React Doctor Evals) 워크플로 —
구현 전 아이디어 검증, 구현 후 OSS 100개 레포 스캔 — 는 거짓 양성을 최소화하는
엄밀한 접근이다.

에러 처리 설계도 눈에 띈다.
`error.message.includes(...)`를 명시적으로 금지하고
`Schema.TaggedErrorClass`와 `Effect.catchReasons`로 타입 안전한 에러 디스패치를
강제하는 것은, 유지보수 과정에서 메시지 문자열이 바뀌어도 버그가 생기지 않는
구조적 보호다.

### 약점과 한계

규칙의 정밀도가 높아질수록 v1 범위를 좁혀야 한다.
`no-mutating-reducer-state` 규칙 예시를 보면, 임포트된 리듀서 함수,
비구조화 할당 별칭, 복잡한 루프 및 try/catch, Immer와 Redux Toolkit 드래프트 리듀서는
모두 v1 비목표(non-goal)로 명시된다.
현실 코드베이스에서 이런 패턴이 얼마나 흔한지를 감안하면,
초기 사용자가 감지되지 않는 버그에 대해 도구를 과신할 위험이 있다.

Effect v4 기반 설계는 강력하지만 진입 장벽이 높다.
`AGENTS.md`에만 Effect 스타일 가이드가 수십 줄에 달한다.
외부 기여자가 새 규칙을 작성하려면 TypeScript, oxlint AST, Effect v4,
그리고 도구 자체의 서비스 레이어를 모두 이해해야 한다.
오픈소스 기여 생태계를 키우기 어려운 구조다.

“결정론적 스캔”을 강조하지만, 경로 인식 규칙은 사실 경로 폭발(path explosion)을
일부 포기하고 보수적으로 모델링한다.
복잡한 제어 흐름은 v1에서 명시적으로 지원하지 않으며,
이는 “결정론적”이라는 표현이 전달하는 완전성과 실제 구현 사이에 간극을 만든다.

## 인사이트

### AI 코드 검토의 무기화 — 도구가 에이전트를 가르친다

React Doctor가 흥미로운 것은 단순히 React 코드를 검사하기 때문이 아니다.
도구의 출력이 에이전트의 다음 행동을 교정하는 피드백 루프를 명시적으로 설계했기 때문이다.

전통적인 정적 분석 도구는 인간 개발자를 독자로 상정한다.
발견된 문제는 티켓이 되고, 인간이 수정하고, 코드 리뷰에서 검토된다.
React Doctor는 이 흐름을 뒤집는다.
에이전트가 코드를 생성하고, React Doctor가 감사하고, 감사 결과가 에이전트의 스킬로 설치되어
다음 생성 사이클에서 같은 실수가 줄어든다.

이 루프가 실제로 작동한다면, 도구의 가치는 한 번의 감사 결과가 아니라
에이전트가 시간이 지남에 따라 더 나은 React 코드를 쓰게 되는 누적 효과에 있다.
이것은 정적 분석 도구가 이전에 한 번도 가져본 적 없는 역할이다.

그런데 이 모델은 에이전트가 스킬을 실제로 흡수하고 적용한다는 전제 위에 있다.
현재 대부분의 코딩 에이전트는 컨텍스트 윈도우 안에서만 작동하며,
스킬 파일이 컨텍스트에 주입되지 않으면 학습 효과는 없다.
에이전트 아키텍처가 지속적 학습을 지원하는 방향으로 발전할수록
이 접근의 가치는 커지고, 그 반대라면 마케팅에 그칠 수 있다.

### 거짓 양성은 정확성 버그다 — 신호 대 잡음비의 신앙 선언

React Doctor의 규칙 작성 가이드에서 가장 눈에 띄는 문장은 다음이다.

“거짓 양성은 정확성 버그로 취급한다.”

이것은 린터 설계 철학에서 근본적인 입장 차이를 드러낸다.
ESLint 같은 범용 린터는 경고를 켜고 끄는 것을 사용자의 책임으로 돌린다.
규칙이 노이즈를 만들더라도 설정으로 억제하면 된다는 태도다.
React Doctor는 그 반대 입장을 취한다.
도구가 보고하는 모든 진단은 실제 문제여야 하며,
아닌 경우 그것은 사용자의 설정 문제가 아니라 규칙의 버그다.

이 입장은 에이전트 통합과 직접 연결된다.
에이전트가 린터 경고를 처리할 때 인간처럼 맥락을 읽고 무시할 수 없다.
노이즈가 많은 도구는 에이전트를 불필요한 수정으로 유도하거나,
경고를 무시하는 규칙 자체를 배우게 만든다.
신호 대 잡음비가 높은 도구만이 에이전트 워크플로에서 신뢰받는 입력이 될 수 있다.

OSS 100개 레포를 스캔하는 RDE 워크플로는 이 철학을 제도적으로 구현한 것이다.
새 규칙이 PR에 들어가기 전, 실제 코드에서 얼마나 노이즈를 만드는지를
수치로 확인하고 기록해야 한다.
`no-mutating-reducer-state` 규칙은 100개 레포 671개 루트디렉토리 스캔에서
단 1건의 진단을 냈다.
이것이 도구가 목표로 하는 신호 밀도다 — 드물지만 확실한 발견.

### 타입 시스템으로 무기화된 에러 처리 — 런타임 보호를 컴파일 타임으로

React Doctor 코어의 에러 처리 설계에서 중요한 패턴은
`error.message.includes(...)`의 금지다.
이것은 단순한 코딩 스타일 규칙이 아니라, 유지보수 안정성에 대한 심층 철학이다.

문자열 패턴 매칭으로 에러를 분기하는 코드는 에러 메시지가 바뀌는 순간 조용히 깨진다.
타입 시스템이 이것을 잡을 수 없기 때문이다.
`Schema.TaggedErrorClass`와 `Effect.catchReasons`는 이 문제를 컴파일 타임으로 끌어올린다.
새 에러 케이스가 추가되면 타입 체커가 처리되지 않은 분기를 즉시 알린다.
삭제된 에러 태그를 참조하는 코드도 즉시 컴파일 에러가 된다.

이 접근은 정적 분석 도구라는 제품의 특성과 깊이 맞닿아 있다.
React Doctor는 다양한 프레임워크와 코드베이스를 스캔하면서 수십 가지 방식으로 실패할 수 있다.
각 서비스가 자신만의 실패 이유를 타입으로 명시하고,
그 이유들이 최상위에서 exhaustive하게 처리된다는 보장은
사용자에게 예측 가능한 에러 메시지를 제공하는 기반이 된다.

더 넓게 보면, 이것은 타입 시스템을 런타임 보호 장치로 사용하는 설계 패턴의
실전 사례다.
TypeScript의 유니온 타입과 Effect의 에러 채널이 결합하면,
에러 처리의 완전성이 컴파일 타임에 검증된다.
이는 “타입은 문서다”라는 약한 주장을 넘어
“타입은 불변 계약이다”라는 강한 주장으로 나아간다.

### 확장 불가능한 복잡성 — 기여 장벽과 오픈소스의 역설

React Doctor는 오픈소스이고 이슈를 환영한다고 명시하지만,
실제 기여 장벽은 상당히 높다.
`AGENTS.md`의 기술 스택 요구사항만 나열해도 — TypeScript, Effect v4,
oxlint AST, ESTree 어휘, Babel 핸들북, OXC 내부 구조, pnpm 워크스페이스,
Turborepo, vite-plus/test — 어느 것 하나 가볍지 않다.

이것은 도구의 설계가 내부적으로 매우 일관성 있고 강력하지만,
그 일관성이 외부에 대해 닫혀 있음을 의미한다.
100개 이상의 규칙을 한 팀이 엄밀하게 유지하는 것이 가능한 이유는
이 높은 기준이 모든 기여자에게 일관되게 적용되기 때문이기도 하다.
하지만 React 생태계가 가진 규칙 아이디어의 롱테일은
이 팀 혼자서 커버하기 어렵다.

비교점으로, ESLint 플러그인 생태계는 수천 개의 서드파티 규칙을 가지고 있다.
품질은 들쭉날쭉하지만, 커버리지는 넓다.
React Doctor가 단일 플러그인으로 고품질을 유지하는 전략을 선택했다면,
커버리지의 폭은 설계상의 트레이드오프다.

궁극적으로 이 도구의 성장 방향은 두 가지다.
팀이 규칙을 계속 추가하며 수직으로 깊어지거나,
기여 파이프라인을 낮춰 커뮤니티가 수평으로 넓히거나.
현재는 전자를 선택한 것처럼 보이며,
이것이 도구의 신뢰도를 높이는 대신 특정 사용 패턴의 공백을 만들 수 있다.
