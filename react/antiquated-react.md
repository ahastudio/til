# Things you forgot (or never knew) because of React

원문: <https://joshcollinsworth.com/blog/antiquated-react>

HN 토론: <https://news.ycombinator.com/item?id=37131802> (526점, 629개 댓글)

## 요약

Josh Collinsworth이 2023년 8월에 작성하고 10월에 업데이트한 React 비판 에세이다.
핵심 주장은 React가 2013년 Facebook의 특수한 제약 조건에 맞게 설계된 레거시 소프트웨어이며,
현재의 대안 프레임워크들이 성능, 단순성, 개발자 경험 모든 면에서 앞서 있음에도
React의 지배력이 만들어낸 “거품” 때문에 개발자들이 이를 인지하지 못한다는 것이다.

저자는 React의 기술적 문제들을 구체적으로 나열한다.
생태계 파편화: React는 웹 플랫폼과의 비호환성 때문에 프레임워크 전용 라이브러리가 필요하지만,
다른 프레임워크들은 일반 JavaScript 패키지나 웹 컴포넌트와 자연스럽게 통합된다.
훅 모델의 한계: React가 훅(hooks)을 대중화했지만 경쟁 프레임워크들 모두 더 빠르고 함정이 적은 방식으로 구현한다.
수동 최적화 부담: React는 개발자가 `useMemo`, `useCallback`, `useEffect` 의존성 배열을 직접 관리해야 하는 반면,
현대 프레임워크들은 반응형 기본 요소(reactive primitives)로 이를 자동화한다.
폼 처리: 단방향 데이터 흐름이 폼 처리를 번거롭게 만든다.

성능 수치를 구체적으로 제시한다.
JS Framework Benchmark 기준으로 React는 Solid보다 약 50% 느리고, Vue보다 약 25% 느리며, Svelte보다 약 40% 느리고, Preact보다 약 35% 느리다.
번들 크기는 다른 프레임워크 대비 최대 10배 크다.
SSR은 한때 React의 강점이었지만 이제 SvelteKit, Nuxt, Fresh, Astro 모두 제공하며 더 나은 구현을 갖추고 있다.
CSS 스코핑은 Vue와 Svelte에 내장됐지만 React는 별도 라이브러리가 필요하다.

저자가 권하는 대안들은 Svelte/SvelteKit, Vue/Nuxt, Solid, Astro, Fresh(Deno 기반), Preact, Qwik이다.
Vue는 React 다음으로 큰 생태계를 보유하며, Svelte는 컴파일러 기반으로 최소한의 JavaScript를 배포한다.
Solid는 React의 JSX 문법을 유지하면서 Signals 기반 반응성으로 높은 성능을 달성한다.
글은 “개발자 경험(DX)은 사용자 경험(UX)보다 앞설 수 없다”는 명제로 요지를 압축하며,
jQuery에서 React로의 전환처럼 React에서 대안 프레임워크로의 전환이 가까웠다는 예측으로 마무리된다.

## 분석

### “React는 레거시 소프트웨어”라는 주장의 논리 구조

이 글의 중심 논거는 “Facebook의 2013년 제약 조건 → React의 설계 결정 → 현재의 기술 부채”라는 인과 사슬이다.
Facebook은 당시 복잡한 단일 페이지 애플리케이션, 빈번한 상태 업데이트, 대규모 팀 협업이라는 특수한 문제를 안고 있었다.
React의 단방향 데이터 흐름, 가상 DOM, 컴포넌트 기반 아키텍처는 이 특수한 맥락에서 나온 해법이다.
저자의 주장은 그 해법이 이제 더 이상 특수한 것이 아니라
범용 웹 개발에서 불필요한 복잡성이 됐다는 것이다.

이 논리가 설득력을 갖는 이유는 기술 부채를 설계 맥락의 변화로 설명하기 때문이다.
단순히 “React가 나쁘다”가 아니라 “React는 다른 문제를 위해 설계됐다”는 주장은
방어하기 어렵다. React 팀이 React의 설계를 합리화할 때 쓰는 논거와 같은 맥락에서
반박을 봉쇄하기 때문이다.
“당신들이 설계 결정을 내린 이유가 바로 지금 문제”라는 구조다.

그러나 이 논리는 암묵적으로 두 가지를 가정한다.
첫째, Facebook의 제약 조건이 일반 웹 개발 맥락과 근본적으로 달랐다는 것.
둘째, 그 제약 조건이 더 이상 의미 없어졌다는 것.
두 가정 모두 검증되지 않는다.
복잡한 상태 관리, 대규모 팀 협업, 빈번한 업데이트는
Facebook만의 문제가 아니라 많은 엔터프라이즈 웹 애플리케이션의 현실이다.

### “DX는 UX를 능가할 수 없다”는 프레임의 전략적 선택

이 명제는 글의 가장 날카로운 수사적 선택이다.
React 사용자들이 React의 복잡성(나쁜 DX)을 감수하는 이유는
충분히 생산적이기 때문이다 — 이것이 묵시적 반론이다.
저자는 이 반론을 “그 복잡성이 최종적으로 사용자에게 부담(나쁜 UX)으로 전가된다”는 방식으로 차단한다.
`useMemo`와 `useCallback`을 빠뜨리면 성능이 떨어지고, 그 결과가 느린 사이트다.
`useEffect` 의존성을 잘못 관리하면 버그가 되고, 그것이 망가진 사용자 경험이다.

이 프레임이 흥미로운 이유는 DX와 UX를 대립 관계로 놓는 것이 일반적이지 않기 때문이다.
보통 “DX가 좋으면 생산성이 높아지고, 그래서 UX도 좋아진다”는 논리가 지배적이다.
저자는 이 논리를 역전한다: “React의 DX가 나쁘기 때문에 DX를 지키려는 노력이 UX 비용을 만든다.”
이 역전이 성립하려면 React의 DX 복잡성이 실제 사용자 경험 저하로 이어지는 경로가 명확해야 한다.
글은 성능 벤치마크와 번들 크기로 이 경로를 주장하지만,
실제 앱에서 이 경로가 얼마나 자주 발동되는지는 다루지 않는다.

실제 개발 경험이 이 경로를 구체적으로 뒷받침하는 경우도 있다.
jQuery, Angular 1, 웹 컴포넌트, Vue를 거쳐 React로 온 개발자 mcv는 대규모 React 프로젝트에서
“`useEffect`가 끝없이 이어지면 무슨 일이 벌어지는지 파악하기 어렵다”는 점이 가장 큰 불만이라고 말한다.[^mcv]
이 경험은 `useEffect` 복잡성이 DX 문제로 그치는 것이 아니라
코드베이스의 가독성, 나아가 유지보수 비용에 영향을 미친다는 것을 보여준다.

### 벤치마크 수치의 수사적 역할

JS Framework Benchmark에서 React가 Solid보다 50% 느리다는 수치는 글에서 핵심 증거로 기능한다.
그러나 이 수치가 “React 앱이 다른 프레임워크 앱보다 50% 느리다”를 의미하지는 않는다.
JS Framework Benchmark는 대형 DOM 리스트 조작, 행 선택, 부분 업데이트 같은
프레임워크의 핵심 경로(critical path)를 격리해 측정하는 도구다.
이 워크로드는 React 가상 DOM의 약점을 최대화하는 조건이다.

실제 웹 애플리케이션은 대부분의 시간을 네트워크 요청 대기, 사용자 입력 처리,
브라우저 레이아웃 계산에 쓴다.
프레임워크의 렌더링 성능이 실제 사용자가 느끼는 속도에 미치는 영향은
이 다른 요소들과의 비율에 달려 있다.
Modern 기기와 빠른 연결에서 동작하는 일반적인 웹 앱에서
React와 Svelte의 렌더링 성능 차이는 사용자에게 체감되기 어렵다.

수치의 효과는 설득이 아니라 충격에 있다.
“50% 느리다”는 것이 얼마나 심각한지 판단하기 위한 기준점을 글은 제공하지 않는다.
대신 독자가 “50%”라는 숫자로부터 스스로 심각성을 추론하게 만든다.
이 방식은 수사적으로 효과적이지만 논증으로서 불완전하다.

### 생태계 성숙도를 무게 없이 다루는 구조

글이 열거하는 대안들 — Svelte, Vue, Solid, Astro, Fresh — 은 기술적 우수성의 차원에서만 평가된다.
각 프레임워크의 NPM 다운로드 수, GitHub 이슈 대응 속도, Stack Overflow 답변 수, 취업 시장에서의 위치는 언급되지 않는다.
이 생략은 의도적이다 — 이 수치들이 여전히 React에 압도적으로 유리하기 때문이다.

“Vue는 React 다음으로 큰 생태계를 가졌다”는 말은 사실이지만, 그 격차를 보여주지 않는다.
React의 npm 주간 다운로드는 Vue보다 6-7배 많다.
이 격차는 특정 라이브러리가 React 전용으로만 유지보수된다는 것을 의미한다.
인증, 결제, 분석, 에디터, 챠트 라이브러리의 React 버전이
Vue나 Svelte 버전보다 더 활발히 유지보수되고, 더 풍부한 기능을 갖는 경우가 많다.
이 실용적 비대칭성이 개발팀의 선택을 제약하는 현실을 글은 다루지 않는다.

## 비평

### 성능 벤치마크에서 “최악 조건”과 “일반 조건”의 혼동

JS Framework Benchmark 수치를 일반적 성능 주장의 근거로 쓰는 것은
논증 구조의 문제다.
이 벤치마크는 수천 행의 DOM 리스트를 생성하고 필터링하고 정렬하는 작업을 측정한다.
이것은 React 가상 DOM이 가장 불리한 조건, 즉 많은 노드가 변경될 때의 상황이다.
반면 가상 DOM은 일부 노드만 변경되는 경우에 오히려 최적화 이점을 가질 수 있다.

더 중요한 점은 대부분의 웹 앱이 이런 대형 리스트 조작을 핵심 인터랙션으로 갖지 않는다는 것이다.
폼, 모달, 탐색, 텍스트 콘텐츠, 대시보드 같은 일반적 UI에서
React와 Svelte의 렌더링 속도 차이가 사용자에게 인식 가능한 영향을 미치는 경우는 드물다.
성능이 실제로 중요한 맥락 — 저사양 기기, 데이터 집약적 시각화, 광고 많은 콘텐츠 사이트 — 이 있지만,
이 맥락들이 React가 주로 쓰이는 맥락과 다르다는 점이 설명되지 않는다.

이 비판이 무효화하는 것은 성능 비교 자체가 아니다.
성능 차이가 실재하고 특정 조건에서 중요하다는 것은 사실이다.
무효화되는 것은 이 수치에서 “React를 쓰면 UX가 나쁘다”는 일반적 결론을 끌어내는 과정이다.

### “jQuery-to-React” 비유가 현재 전환 가능성을 과장한다

jQuery에서 React로의 전환은 능력의 전환이었다.
jQuery는 단일 페이지 애플리케이션 복잡도를 다루는 데 적합하지 않았다.
상태 관리가 수십 개의 `$(“.class”).show()` 호출로 분산되면 유지보수가 불가능해진다.
React는 이 근본적 문제를 해결했기 때문에 전환 동기가 강력했다.

React에서 대안 프레임워크로의 전환은 다르다.
React로 만든 앱이 “Svelte로 만들었어야 작동했을” 앱이 아니다.
차이는 성능과 DX의 정도 차이다.
이 차이가 새 프로젝트 선택에 영향을 줄 수 있지만,
기존 React 코드베이스를 재작성할 동기는 jQuery 재작성 동기와 비교할 수 없다.
글이 예측하는 “주요 전환”은 조용하고 느리게 일어날 수 있지만,
jQuery-React 전환의 속도나 극적인 방식으로는 오지 않는다.

글이 쓰인 2023년 이후 2025년 현재의 데이터를 보면 이 예측의 한계가 드러난다.
Stack Overflow 개발자 설문에서 React는 여전히 가장 많이 사용되는 웹 프레임워크다.
고용 시장에서 React 포지션은 감소하지 않았다.
새 프레임워크들이 등장하고 Svelte, Vue, Solid의 채택이 증가하지만,
React의 절대적 점유율이 하락하는 신호는 뚜렷하지 않다.

더 날카로운 반론은 전환 방향에 관한 것이 아니라 React 내부의 변화 속도에 관한 것이다.
tock은 “React와 Next.js 생태계의 가장 큰 문제는 유지보수”라며,
“3-5년마다 모든 것을 다시 발명하고, 그때마다 이주 작업이 필요하다는 것을 이미 안다”고 쓴다.[^tock]
클래스 컴포넌트 → 함수형 컴포넌트+훅, 클라이언트 렌더링 → 서버 컴포넌트,
수동 메모이제이션 → React Compiler로의 전환들이 그 예다.
이 내부 변화는 “jQuery에서 React로의 전환”이라는 외부 전환 비유와는 무관하다.
대안 프레임워크로의 전환 없이도, React를 계속 쓰는 팀은 React 자체의 패러다임 전환에 대응해야 한다.

### 글의 설득 대상과 실제 독자 사이의 간극

이 글이 설득하려는 대상은 “React 거품 안에 있는 개발자”다.
그러나 이 글을 읽을 가능성이 높은 독자는
이미 대안 프레임워크에 관심이 있거나 React의 한계에 불만이 있는 개발자다.
React를 아무 문제 없이 쓰고 있는 개발자가 이 글의 제목 — “Things you forgot (or never knew) because of React” — 을 클릭할 가능성은 낮다.

이 설득 대상과 실제 독자의 불일치는 글의 논조를 설명한다.
이 글은 React 사용자에게 “당신은 더 좋은 것을 놓치고 있다”고 말하는 형식이지만,
실제로는 이미 회의적인 독자들에게 “당신의 직관이 맞다”고 확인해주는 기능을 한다.
이 구조가 나쁜 것은 아니지만, 글의 논증이 React를 적극적으로 옹호하는 사람을
설득하기에는 부족하다는 것을 의미한다.

실제로 React 팀에게 이 글의 지적 대부분은 알려진 사실이다.
React Forget(Compiler), React Server Components, Concurrent Mode는
글이 비판하는 문제들 — 수동 최적화, 서버 렌더링, 성능 — 에 대한
React 팀의 자체 대응이다.
글은 이 대응들을 “늦었다”고 볼 수 있지만, 대응이 진행 중이라는 사실이 “레거시 소프트웨어”라는 주장을 약화시킨다.

훅을 긍정적으로 평가하는 React 사용자들이 이 간극을 가장 구체적으로 드러낸다.
awestroke는 “훅에 부정적인 사람들은 소수”이며, 훅은 클래스 컴포넌트의 생명주기 메서드보다 훨씬 나은 도구이고
“올바르게 쓰면 매우 깔끔하고 강력한 코드”로 이어진다고 말한다.[^awestroke]
이 개발자들이 이 글을 읽었다면 설득되지 않을 것이다.
훅이 나쁘다는 주장을 일반적 사실처럼 제시하는 것은
정반대의 경험을 가진 React 사용자들과 대화를 시작조차 못 한다.

### 대안 프레임워크들이 공유하는 한계를 인식하지 않는다

글은 각 대안의 기술적 장점을 나열하지만 각각의 실질적 제약은 다루지 않는다.
Svelte는 팀 채용 풀이 좁고, Svelte 3에서 5로의 반응성 모델 변경이 상당한 이주 비용을 만들었다.
Vue는 React보다 TypeScript 통합이 역사적으로 어려웠다(Vue 3에서 개선됐지만).
Solid는 생태계가 작고 특정 써드파티 라이브러리 지원이 부족하다.
Fresh는 Deno 생태계에 묶여 Node.js 기반 인프라와 통합이 복잡하다.
Astro는 인터랙티브 앱보다 콘텐츠 사이트에 최적화됐다.

이 제약들은 기술적 우수성을 부정하지 않는다.
그러나 팀이 React를 선택하거나 유지하는 이유가
단순히 “React 거품” 때문이 아니라
이 제약들을 실용적으로 평가한 결과일 수 있다는 가능성을 글은 닫아버린다.

더 근본적인 반론은 프레임워크 자체의 차이가 팀 생산성에 얼마나 영향을 미치는지에 대한 것이다.
LVB는 React와 Go를 쓰면서 비판적 글을 많이 봤지만,
실제 작업에서 생산성을 떨어뜨리는 것은 “나쁜 요구사항, 나쁜 시스템 설계, 나쁜 경영 결정”이었지
기술 선택이 아니었다고 말한다.[^LVB]
이 관점에서 보면 이 글의 핵심 전제 — 프레임워크 선택이 UX와 DX에 의미 있는 영향을 미친다 — 가 도전받는다.
프레임워크의 성능 차이와 DX 차이가 실제 제품 품질에 반영되려면,
팀이 다른 모든 조건에서 충분히 잘 기능하고 있어야 한다.
프레임워크 선택이 결정적 차이를 만드는 팀은 이미 조직 문제가 없는 팀이다.

## 인사이트

### “React 거품”이라는 프레임이 실제로 비판하는 것은 채용 시장이다

이 글이 기술적 우수성을 이야기하는 것처럼 보이지만
실질적 논증 구조를 따라가면 다른 결론에 도달한다.
React의 기술적 문제들 — 수동 최적화, useEffect 복잡성, 생태계 파편화 — 은
React 팀이 이미 인지하고 있으며 대응하고 있다.
이 문제들이 “React를 쓰지 말아야 할 이유”가 되려면
다른 프레임워크들이 이 문제들을 충분히 더 잘 해결해야 한다.

저자가 명시하지 않은 더 강력한 논거는 채용 시장이다.
React 개발자는 공급이 많기 때문에 비용이 낮지만,
그 공급의 상당 부분은 React가 현대 프론트엔드 개발의 기본값이라는 인식 때문에 형성됐다.
이 인식이 실제로는 과장됐으며, 다른 프레임워크들이 더 나은 선택임에도
채용 시장의 구조가 React 채택을 강제한다는 것이 이 글의 가장 강력한 암묵적 메시지다.
“React 거품”은 기술 선택의 거품이기 이전에 채용 시장의 거품이다.

HN 댓글의 thousand_nights는 이 현실을 간결하게 정리한다:
“그 차트 라이브러리? 이미 있다. 모바일? React Native. 이상한 콤보박스 자동완성 컴포넌트?
누군가 이미 만들었다. 새 팀원들은 어디서 시작할지 안다. 생태계가 곧 왕이다.”[^thousand_nights]
이 논거는 기술적이지 않다.
실력 있는 팀이라면 Svelte나 Vue를 선택해 기술적으로 더 나은 결과물을 만들 수 있다.
그러나 그 선택이 가능하려면 팀 전체가 해당 생태계에 익숙해야 하고,
채용 시 선택지가 좁아지는 것을 감수해야 하며,
지원 라이브러리의 공백을 스스로 채울 역량이 있어야 한다.
React를 “기본값”으로 선택하는 팀은 생산성에 지불하는 보험료가 있다는 것을 알고 있다.

### 기술 비교 글이 프레임워크 생태계에 미치는 실제 영향

이 글은 2023년 출판 이후 상당한 주목을 받았다.
그러나 이 글이 React 채택률에 미친 측정 가능한 영향은 없다.
이것은 글의 실패가 아니라 기술 비교 글이 실제로 작동하는 방식의 특성이다.

이런 글의 효과는 직접적 전환이 아니라 “Overton window”의 이동에 있다.
이 글을 읽은 시니어 개발자는 React를 즉시 버리지 않는다.
그러나 다음 그린필드 프로젝트에서 Svelte나 Vue를 제안할 때
“그런 선택을 지지하는 논리”로 이 글을 참조할 가능성이 높아진다.
효과는 간접적이고, 의사결정자를 통해, 그리고 시간을 두고 나타난다.
기술 에세이의 역할은 설득이 아니라 근거를 제공하는 것이다.

### 지배적 프레임워크의 쇠락이 오는 방식은 항상 다르다

jQuery의 쇠락은 Angular, React의 등장으로 갑자기 왔다.
Flash의 쇠락은 Apple의 iPhone 정책 결정과 함께 왔다.
Backbone의 쇠락은 React의 성능 우위로 왔다.
이 각각은 “더 나은 대안”이 등장한 것이 아니라
패러다임의 전환이 왔다.

현재의 React 상황은 이 역사적 패턴과 다르다.
React의 대안들은 패러다임 전환이 아니라 동일한 패러다임 내에서 개선된 구현이다.
Svelte, Vue, Solid 모두 컴포넌트 기반, 반응형 상태, 선언적 템플릿이라는 React가 확립한 패러다임 위에 서 있다.
이 상황에서 전환의 동력은 “더 이상 현재 도구로 못 하겠다”가 아니라
“현재 도구보다 더 잘할 수 있다”다.
후자는 훨씬 약한 동기다.

React의 쇠락이 온다면 그것은 기술적 열세 때문이 아닐 것이다.
더 가능성 있는 경로는: AI 기반 코드 생성 도구가 프레임워크 선택을 흐리게 만들거나,
서버 컴포넌트와 점진적 향상의 패러다임이 클라이언트 중심 SPA의 전제를 해체하거나,
새로운 플랫폼(웹 어셈블리, 새로운 브라우저 API)이 현재 프레임워크 전체를 구식으로 만드는 것이다.
이 경로들은 이 글이 주장하는 “더 좋은 프레임워크로의 전환”과는 다른 종류의 변화다.

nologic01은 이 논리를 한 단계 더 밀어붙여
“jQuery는 한때 98%의 웹사이트에서 사용됐지만 그 시대는 지났다”며,
앞으로 웹은 하나의 지배적 프레임워크로 수렴하지 않고
“최소한의 반응형 라이브러리(htmx, Alpine.js), WebAssembly 앱, AI가 생성하는 프레임워크”로 삼분될 것이라고 예측한다.[^nologic01]
이 예측이 흥미로운 이유는 “React를 대체할 것이 무엇인가”라는 질문 자체를 틀렸다고 보기 때문이다.
대체가 아니라 분기가 일어난다면, 이 글이 전제하는 “패권 이동” 서사는 성립하지 않는다.
jQuery에서 React로의 전환은 단일 패러다임의 교체였지만,
React 이후에는 단일 패러다임 자체가 사라질 수 있다.

### “레거시”의 정의가 기술적 우열에서 분리되는 지점

이 글은 React를 “레거시 소프트웨어”라고 규정하지만,
레거시의 일반적 의미는 “더 이상 유지보수되지 않거나 구식 기술에 의존하는”이다.
React는 활발히 유지보수되고 있으며, Meta의 전폭적 지원을 받는다.
기술 스택 기준으로 React는 레거시가 아니다 — 최신 JavaScript, WebPack/Vite, TypeScript와 완전히 호환된다.

저자가 실제로 주장하는 것은 “React의 설계 철학이 구시대적”이라는 것이다.
이것은 “레거시”와 다른 주장이다.
가상 DOM이 2023년 기준으로 최선의 선택이 아닐 수 있다는 것은 논쟁 가능하다.
그러나 그것이 “레거시”를 의미하지는 않는다.

이 개념적 혼용이 중요한 이유는 “레거시를 계속 쓰는 팀은 이상하다”는 함의 때문이다.
레거시 프레임워크를 계속 유지보수하는 것은 기술 부채 상환을 미루는 것이고,
그 결정에는 설명이 필요하다.
그러나 기술적 철학이 현재 최선보다 다소 뒤진 프레임워크를 쓰는 팀에게
“설명이 필요하다”는 부담을 지우는 것은 정당하지 않다.
팀이 React를 쓰는 데는 합리적 이유가 있으며,
그 이유가 “거품 안에 있기 때문”으로 환원되지 않는다.
