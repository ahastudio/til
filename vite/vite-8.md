# Vite 8 - Rolldown 통합으로 개발·프로덕션 번들러 단일화

> 원문: <https://vite.dev/blog/announcing-vite8-beta>

## 요약

Vite 8은 Rust 기반 번들러 Rolldown을 채택해 기존에 개발 환경에서 쓰던 esbuild와
프로덕션 빌드에서 쓰던 Rollup을 단일 번들러로 대체한다. 빌드 속도 향상과 함께
개발·프로덕션 환경 동작의 일관성 확보가 핵심 목표다.

실제 사용 사례에서 Linear는 빌드 시간이 46초에서 6초로, Beehiiv는 64%, Mercedes-Benz.io는
최대 38% 단축됐다. Oxc 컴파일러를 활용한 향상된 트리쉐이킹, TypeScript `tsconfig paths`
지원, `emitDecoratorMetadata` 자동 지원도 추가됐다.

마이그레이션 경로는 두 가지다. v8.0.0-beta.0으로 직접 전환하거나, Vite 7 →
rolldown-vite → Vite 8 단계적 경로를 밟을 수 있다. 대부분의 Vite 플러그인은
플러그인 API 호환성이 유지되어 즉시 사용 가능하다.

## 분석

Vite의 가장 오래된 불편함 중 하나는 개발 서버와 프로덕션 빌드가 다른 번들러를
쓴다는 점이었다. esbuild는 빠르지만 Rollup의 트리쉐이킹과 플러그인 생태계를
완전히 대체하지 못했고, 이 때문에 "개발에서는 통과, 프로덕션에서 깨짐" 같은
환경 차이 버그가 반복됐다.

Rolldown은 Rollup 호환 API를 목표로 Rust로 구현되어 이 간극을 좁힌다. 두 번들러를
유지하고 동기화하던 복잡도가 사라지고, 플러그인 생태계도 단일 API 기준으로 수렴하게 된다.

주의해야 할 마이그레이션 포인트는 Rollup이나 esbuild의 특정 옵션에 강하게 의존한
설정이다. Astro, Nuxt, Vitest 같은 프레임워크 사용자는 `package.json`에 명시적
오버라이드가 필요한 경우가 있다.

## 비평

Vite 8은 "더 빠른 Vite"보다 "더 일관된 Vite"가 더 정확한 설명이다. 속도 향상은
번들 규모에 따라 차이가 있고, 기존 설정이 복잡한 프로젝트는 마이그레이션 비용을
먼저 계산해야 한다.

Rolldown이 베타 단계라는 점도 고려해야 한다. 엣지 케이스나 플러그인 호환성 문제가
남아있을 수 있으므로, 대형 프로덕션 프로젝트는 안정화를 기다리는 것이 안전하다.

JS 툴체인이 Rust 엔진으로 이동하는 큰 흐름 — esbuild, SWC, Rolldown, Oxc — 속에서
Vite 8은 프론트엔드 빌드 도구가 "JavaScript API + Rust 엔진" 구조로 안정화되는 과정의
이정표다.

## 인사이트

### 개발·프로덕션 환경 일관성은 번들러 통합 이상의 문제다

두 번들러로 인한 환경
차이 버그는 디버깅 비용이 높고 재현이 어렵다. Vite 8이 이 분기를 제거함으로써
얻는 가장 큰 가치는 벤치마크 수치보다 "프로덕션에서만 재현되는 버그" 범주 하나가
사라진다는 것이다.

### 번들러 전쟁의 2라운드는 속도가 아닌 생태계 호환성이다

esbuild가 속도로
1라운드를 정의했다면, Rolldown은 "Rollup 호환 + Rust 속도"로 2라운드를 정의한다.
속도 경쟁은 이제 충분히 빠른 수준에서 수렴하고, 다음 경쟁 축은 플러그인 생태계
포용성과 마이그레이션 비용 최소화로 이동한다.

### Rust 기반 JS 툴체인의 전환 비용이 설계 철학을 드러낸다

Vite 8의 점진적 마이그레이션
경로 제공은 속도보다 생태계 연속성을 우선한다는 의사결정이다. 기술적 우월함보다
기존 사용자의 전환 마찰을 낮추는 것이 장기 채택에 더 중요하다는 교훈이 여기에도 적용된다.
