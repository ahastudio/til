# OpenMMO

<https://github.com/Julian-adv/OpenMMO>

## 소개

OpenMMO는 AI 에이전트(Agent)와 인간 플레이어를 동등하게 처우하는 오픈소스 MMORPG 프로젝트다.
에이전트와 인간이 동일한 WebSocket 프로토콜로 같은 서버에 접속하며, 서버는 두 종류의 클라이언트를 구분하지 않는다.

제작자는 **송재경**(Jake Song)이다.[^jaksong-tweet]
리니지(1998)의 핵심 개발자이자 넥슨 공동창업자, 엑스엘게임즈(아키에이지) 창업자로 한국 MMORPG 역사를 만들어온 인물이다.
그가 2026년 7월 20일 직접 트윗으로 밝혔다: “저는 회사를 그만두고 쉬다가 집에서 이런거 만들고 있습니다.”
30년 경력의 게임 개발자가 혼자, AI 도구만 들고 집에서 만든 프로젝트다.

“바이브 코딩(vibe-coded)”으로 제작했으며, 현재 실제로 플레이 가능한 서버(<https://openmmo.to.nexus:10004>)를 운영 중이다.
약 280개의 스타와 38개의 포크를 기록했다.
주력 언어는 Rust이며, TypeScript, Svelte, Python, Shell 등이 혼재한다.

## 주요 기능

### 에이전트-인간 동등 접속

이 프로젝트의 핵심 설계 원칙은 단 하나다.
에이전트와 인간은 완전히 동일한 WebSocket 프로토콜을 사용하며, 서버는 어느 쪽인지 알 수 없고 알 필요도 없다.
별도의 API 엔드포인트, 특권 채널, 분리된 인터페이스가 존재하지 않는다.

코드에서 이 원칙은 `AuthenticateNpc` 메시지 하나로 구현된다.
에이전트는 `account_name`과 `npc_token`을 담은 이 메시지로 인증하며, 이후 전송하는 모든 `ClientMessage`는 인간 클라이언트와 동일한 열거형 변형이다.
서버의 `ConnectionState`는 `is_npc: bool` 필드를 갖지만, 이 값은 로깅과 일부 어드민 기능에만 쓰이고 게임 로직에는 관여하지 않는다.

### 와이어 프로토콜: MessagePack

실제 전송 포맷은 JSON이 아닌 MessagePack이다.
`shared/src/messages.rs`의 직렬화 도우미가 `rmp_serde`를 사용해 `ClientMessage`와 `ServerMessage`를 이진 형식으로 변환한다.
`ClientMessage`는 `PlayerMove`, `MonsterAttack`, `ChatMessage`, `OfferDeal` 등 수십 가지 변형을 포함하며, `ServerMessage`는 그에 대응하는 서버 발신 메시지 전체를 담는다.
두 타입은 `shared` 크레이트에서 정의되어 서버, 웹 클라이언트(WASM), 에이전트 클라이언트가 모두 공유한다.

### 에이전트 클라이언트 아키텍처

에이전트 클라이언트는 Rust로 작성되었으며 다중 LLM 백엔드를 지원한다.
`LlmType` 열거형은 `Claude`(CLI 서브프로세스), `Openrouter`(HTTP API), `Codex`(CLI 서브프로세스), `None`을 제공한다.
`LlmBackend` 트레이트는 `send_message(&str) -> Result<String>` 하나만 요구하므로 백엔드 교체가 설정 파일 수정으로 끝난다.

내부 구조는 세 레이어로 나뉜다.

- **LLM**: 고수준 의사결정 (전략, 대화, 판단)
- **에이전트 클라이언트**: 저수준 실행 (A\* 경로탐색, 상태 관리, 이벤트 수집)
- **게임 서버**: 월드 시뮬레이션, 권한 검증

LLM이 내리는 결정은 JSON으로 구조화되어 있다.
`AgentResponse`는 `thought`, `actions`, `memory_update` 세 필드를 가지며, `AgentAction`은 `say`, `attack`, `move`, `offer_deal`, `open_trade`, `wait` 변형을 지원한다.
`extract_json` 함수가 LLM이 마크다운 코드 블록으로 감싸거나 날 JSON을 반환하는 경우를 모두 처리한다.

### LLM 스케줄러

모든 NPC가 하나의 `LlmScheduler`를 공유한다.
`LlmPriority` 열거형은 `Urgent`(전투 피해, 직접 대화, 사망), `Routine`(주기적 활성 폴링), `Idle`(비활성 폴링) 세 단계를 정의한다.
내부적으로 `BinaryHeap`을 사용해 우선순위 큐를 구현하며, 동일 우선순위에서는 제출 시각이 이른 요청을 먼저 처리한다.
`max_concurrent` 설정값만큼만 동시에 LLM을 호출하고, 수신자가 이미 드롭된 고아 요청은 스케줄러가 조용히 건너뛴다.

드라이버 루프는 공격 쿨다운 틱마다 `attack_target`을 확인해 전투 중에는 LLM 응답을 기다리지 않고 즉시 `MonsterAttack` 메시지를 전송한다.
이 분리 덕분에 LLM이 느리게 응답해도 전투 반응성이 유지된다.

### NPC 및 몬스터 AI 시스템

2계층 AI 구조를 채택했다.
몬스터와 기본 NPC 전투는 결정론적 상태 머신으로 처리해 LLM 호출 비용을 발생시키지 않는다.
NPC 대화와 고수준 판단에만 LLM을 사용한다.

몬스터 AI는 `shared/src/monster_ai/` 크레이트에 구현되어 있으며 서버와 에이전트 클라이언트가 동일한 코드를 사용한다.
`MonsterBrain`은 `Idle → Walk/Run → Attack → Hit → Flee → Return → Idle` 상태 전환을 관리하고, 행동 트리(behavior tree)는 `data-src/behavior_trees.json`에서 런타임에 로드된다.
몬스터 AI 틱은 1Hz 간격으로 돌아가는 별도 `tokio::task`에서 실행된다.

몬스터 스폰은 서버가 결정하고 클라이언트가 실행하는 분리 모델이다.
서버가 `SpawnMonsterRequest`를 소유자 클라이언트에 보내면, 클라이언트가 유효한 위치(물/절벽/실내 회피)를 선정해 `RequestSpawnMonster`로 응답한다.
서버가 위치를 검증한 뒤 `MonsterAssigned`와 `MonsterSpawned`를 순서대로 전송한다.

NPC 프롬프트는 3계층으로 구성된다.
역할 공통 프롬프트(템플릿), 개체 고유 정보(인스턴스), 게임 내 경험 기억(메모리)이 런타임에 결합되어 system prompt를 형성한다.
메모리는 LLM 응답에 포함된 `memory_update` 필드를 통해 자동으로 갱신된다.

NPC 스케줄 시스템은 `Day`, `Night`, `Time { hour, minute }`, `Recurring { minute }` 네 가지 조건을 지원한다.
`*:00` 형식의 매시 반복 조건을 지원하며, 조건은 로드 시점에 검증되어 런타임 오류를 사전에 차단한다.

### 세계 시스템

32km × 32km의 절차적 생성 세계를 제공한다.
`shared/src/worldgen/`은 2단계 파이프라인으로 구성된다.
저해상도 전역 맵(예: 4096×4096, 셀당 8m)이 대륙 형태, 바이옴, 강, 정착지, 도로를 담당하고, 고해상도 타일 세부 사항은 전역 맵을 샘플링해 세밀한 노이즈를 더하는 방식으로 요청 시 생성된다.
지형, 강, 해안선, 바이옴이 모두 절차적으로 생성되며 강의 삼각주, 분류(分流) 지류, 하구 블렌딩까지 구현되어 있다.
도로 네트워크는 정착지를 연결하며 강을 가로지르는 자리에는 다리가 자동으로 배치된다.

### LLM 흥정 경제 시스템

가장 독특한 시스템 중 하나다.
NPC 상인은 LLM을 통해 `OfferDeal` 메시지로 가격 조정을 제안할 수 있다.
그러나 LLM이 제안하는 값은 서버에서 반드시 검증과 보정을 거친다.

서버 `deals.rs`의 흥정 처리 흐름은 다음과 같다.

1. **NPC 전용 확인**: 인간 클라이언트가 `OfferDeal`을 보내면 즉시 거부한다.
2. **거리 확인**: 대상 플레이어가 시야 반경(`NPC_SIGHT_RADIUS`) 안에 있어야 한다.
3. **CHA 파생 밴드**: 플레이어의 CHA 능력치가 가격 협상 폭을 결정한다. CHA 10 기준으로 ±10%이며, CHA가 올라갈수록 ±25%까지 확장된다. LLM이 이 밴드 밖의 값을 제안하면 서버가 자동으로 클램핑한다.
4. **일일 예산**: NPC는 하루에 10,000 단위의 할인 예산을 갖는다. 플레이어 역시 하루에 받을 수 있는 총 할인이 4,000 단위로 제한된다.
5. **쿨다운**: 같은 NPC-플레이어 쌍은 30초에 한 번만 흥정할 수 있다.
6. **머니펌프 불변식**: 상인의 판매율과 최대 밴드 폭 조합에서 “싸게 사서 비싸게 파는” 차익이 절대 생기지 않도록 `band_invariant_holds` 함수로 설계 시점에 수학적으로 보장한다.

LLM에게 전달되는 `DealResult` 응답에는 서버가 왜 거부했는지, 클램핑이 있었다면 얼마나 조정됐는지가 포함된다.
LLM이 자신의 제안이 어떻게 처리됐는지 피드백을 받아 다음 결정에 활용할 수 있는 구조다.

### A* 경로탐색: 다층 건물 지원

경로탐색은 `shared/src/pathfinding/astar.rs`에 구현되어 있으며 서버, 브라우저 클라이언트(WASM), 에이전트 클라이언트가 동일한 코드를 사용한다.

핵심 설계는 1m 격자 위의 A*이며, 층(floor) 개념이 핵심이다.
일반 셀은 4방향(동서남북) 카디널 이동을 사용하고, 계단 셀은 별도의 `AStarKey` 타입(`(x, z, floor_key)`)으로 관리된다.
계단 내부의 중간 셀들은 일반 층 키와 다른 중간 값의 `floor_key`를 가지며, 경로 복원 시 일반 층 착지점만 웨이포인트로 내보낸다.

중요한 최적화: 출발지와 목적지가 같은 층이고 계단실 안에 있지 않으면 `confine_to_floor` 플래그를 켜서 계단 탐색을 완전히 건너뛴다.
이 분기 하나로 던전 몬스터 AI처럼 같은 층에서만 움직이는 경우의 탐색 비용이 크게 줄어든다.

최대 노드 수는 기본 2,000개로 제한된다.
탐색이 한계에 도달하면 목표에 가장 가까운 노드까지의 부분 경로를 반환한다(`found: false`).
에이전트는 이를 받아 이동하다가 막히면 상위 레이어(LLM)에 실패를 보고한다.

### 기술 스택

클라이언트는 Svelte + TypeScript + Three.js(Threlte) + WebGPU를 사용한다.
서버는 Rust + Tokio 비동기 런타임 + tokio-tungstenite(WebSocket) + Axum(REST API)으로 구성된다.
에이전트 클라이언트는 Rust + MCP 서버(rmcp) + Tokio로 작성되었다.
공유 크레이트(`shared/`)는 서버, 클라이언트(WASM), 에이전트 클라이언트가 함께 사용한다.

## 분석

### “동등 접속”이 의미하는 것

에이전트-인간 동등 접속은 단순한 마케팅 문구가 아니라 구체적인 아키텍처 결정이다.
서버가 클라이언트 종류를 구분하지 않는다는 말은, 인증, 로드밸런싱, rate limiting 등 기존 서버 인프라가 에이전트와 인간에게 동일하게 적용된다는 뜻이다.

이 설계는 두 가지 함의를 갖는다.
첫째, 에이전트가 인간과 동일한 제약을 받는다. 인간이 할 수 없는 행동은 에이전트도 할 수 없다.
둘째, 에이전트가 인간과 동일한 자유를 누린다. 인간이 할 수 있는 모든 행동을 에이전트도 할 수 있다.
이는 의도적으로 에이전트의 능력을 제한하지 않겠다는 선언이기도 하다.

### LLM과 실행 레이어의 분리

에이전트 클라이언트 설계에서 주목할 점은 LLM과 실행 레이어를 명확히 분리했다는 것이다.
LLM이 매 프레임 좌표를 결정하는 구조는 비현실적이다. LLM 응답 지연이 게임 루프를 가로막고, 호출 비용이 폭발적으로 증가한다.

OpenMMO는 이 문제를 LLM을 고수준 의사결정자로만 사용하고 저수준 실행(경로탐색, 상태 관리)은 결정론적 코드에 맡기는 방식으로 해결했다.
이는 에이전트 시스템 설계의 일반 원칙—LLM은 판단하고 코드는 실행한다—을 게임 도메인에 구체적으로 적용한 사례다.

### 신뢰 경계를 서버가 쥐는 몬스터 스폰 모델

몬스터 AI는 클라이언트(에이전트 또는 인간 브라우저 클라이언트)가 실행하지만, 스폰 결정과 검증은 서버가 담당한다.
서버가 `SpawnMonsterRequest`로 스폰 지역과 몬스터 종류를 지정하면, 클라이언트가 물/절벽/실내를 피한 유효 위치를 골라 `RequestSpawnMonster`로 응답하고, 서버가 위치를 재검증한 뒤 최종 확정한다.

이 흐름은 게임 보안의 고전적 원칙—서버가 권위(authoritative)를 갖고, 클라이언트는 제안(suggest)만 한다—을 에이전트 맥락에서 정확히 구현한 것이다.
에이전트가 악의적으로 요새 한가운데 몬스터를 소환하려 해도 서버의 검증 단계에서 차단된다.
“에이전트와 인간이 동등하다”는 원칙이 신뢰도 동등하다는 뜻은 아님을 코드가 보여준다.

### MCP를 에이전트 브릿지로 활용

에이전트 클라이언트가 MCP 서버로 동작한다는 선택은 흥미롭다.
MCP는 원래 AI 도구 통합을 위한 프로토콜로, Claude, GPT 등 다양한 LLM 클라이언트가 표준 인터페이스로 연결할 수 있다.
즉, 에이전트 클라이언트를 MCP 서버로 만들면 어떤 MCP 호환 LLM도 별도 어댑터 없이 게임에 접속할 수 있게 된다.

이는 특정 LLM 벤더에 대한 의존을 없애고, 미래의 더 나은 모델로 교체할 때 에이전트 클라이언트 코드를 수정하지 않아도 된다는 뜻이다.
게임 클라이언트와 LLM 사이의 결합을 느슨하게 유지하는 설계다.

## 인사이트

### LLM을 신뢰하지 않고 검증하는 흥정 시스템

`deals.rs`의 흥정 구현에서 배울 수 있는 핵심 패턴이 있다.
LLM이 제안하는 모든 값은 서버에서 독립적으로 검증되고 클램핑된다.
LLM이 “50% 할인”을 제안해도 플레이어 CHA가 10이라면 서버가 ±10% 이내로 강제 조정한다.
거절 이유(`why`)는 `DealResult`로 LLM에게 피드백되어 다음 판단에 반영된다.

이 패턴은 LLM 기반 시스템 설계의 근본 원칙을 구현한 것이다.
“LLM을 실행자로 쓰지 말고 제안자로 써라.”
LLM의 출력을 직접 게임 상태에 반영하지 않고, 항상 결정론적 레이어가 중간에서 검증하고 최종 실행한다.
`AgentAction`을 `ClientMessage`로 변환하는 `action_to_command` 함수, 몬스터 스폰 위치를 서버가 재검증하는 흐름, 흥정 할인폭을 CHA 밴드로 클램핑하는 로직이 모두 같은 원칙의 반복이다.

이 설계를 다른 LLM 에이전트 시스템에 적용하면: LLM이 파일을 삭제하려 해도 허용 목록 체크가 막고, LLM이 API 호출 파라미터를 생성해도 타입 검증 레이어가 걸러낸다.
LLM의 출력을 신뢰하는 대신 항상 독립적인 검증 계층을 두는 것이 안전한 에이전트 시스템의 토대다.

### `Arc<Mutex<SharedState>>`가 만드는 경쟁과 그 의미

에이전트 클라이언트의 핵심 설계는 세 태스크가 하나의 `Arc<Mutex<SharedState>>`를 공유하는 구조다.
`rx_task`(WebSocket 수신), `ai_task`(1Hz 몬스터 AI 틱), `llm_task`(비동기 LLM 드라이버)가 동일한 잠금 객체를 놓고 경쟁한다.

이 구조는 상태 일관성(동시에 두 태스크가 상태를 쓰는 일 없음)은 보장하지만, 의미적 일관성(어떤 태스크의 판단이 우선인가)은 보장하지 않는다.
전투 중 LLM이 `move` 액션을 보내는 순간 `ai_task`가 공격 틱을 처리하는 상황이 이론적으로 발생한다.
코드는 이를 특별히 막지 않는다.

여기서 배울 점: 비동기 Rust에서 `Arc<Mutex<T>>`는 데이터 레이스를 막지만, “누가 먼저 결정권을 갖는가”라는 문제는 별도의 우선순위 프로토콜 없이 해결되지 않는다.
LLM 에이전트와 결정론적 AI가 동일한 상태를 공유할 때는 상태 기계(state machine)로 현재 모드를 명시적으로 관리하거나 채널(channel)로 태스크 간 통신을 직렬화하는 편이 더 명확하다.

### 공유 크레이트 하나로 서버-클라이언트-에이전트를 묶는 방법

`shared/` 크레이트는 이 프로젝트의 숨겨진 핵심이다.
`ClientMessage`, `ServerMessage`, `PassabilityCache`, `monster_ai`, `pathfinding` 등이 여기서 정의되어 서버, 브라우저 클라이언트(WASM 빌드), 에이전트 클라이언트가 모두 동일한 코드를 참조한다.

이 구조가 실제로 무엇을 보장하는지 살펴보면:
A* 경로탐색 코드는 서버에서 한 번 작성되면 브라우저 클라이언트와 에이전트 클라이언트에서 동일하게 실행된다.
몬스터 AI 상태 머신이 에이전트 클라이언트에서 버그를 내면 브라우저 클라이언트에서도 동일한 버그가 나온다.
프로토콜 메시지 타입을 바꾸면 컴파일러가 세 곳을 동시에 에러로 잡아준다.

Cargo Workspace + 공유 크레이트 패턴은 이 구조의 기술적 기반이다.
`Cargo.toml`의 `[workspace]` 선언이 `shared/`, `server/`, `agent-client/`, `terrain/`을 묶고, WASM 빌드는 `wasm_api.rs`가 `shared/`의 타입을 재수출하는 방식으로 브라우저에 노출한다.
Rust의 타입 시스템이 “프로토콜 불일치”를 컴파일 타임에 잡아주므로 런타임 역직렬화 오류가 발생할 여지가 줄어든다.

### 애니메이션 데이터로 쿨다운을 결정하는 방법

`combat.rs`에서 공격 쿨다운을 결정하는 방법이 독특하다.

```rust
const ANIMATION_DURATIONS_PATH: &str = "data/animation_durations.json";
// { "combat_melee": { "slash1": 1.533, ... } }
```

런타임에 `slash1` 애니메이션 지속 시간을 읽어서 그것을 공격 쿨다운으로 사용한다.
게임 디자인의 상수가 아니라 실제 애니메이션 파일에서 유도된 값이다.
`tools/extract-animation-durations.mjs`가 GLB 파일에서 이 값을 추출해 JSON으로 만든다.

이 패턴이 의미하는 것: 게임 느낌(game feel)과 AI 타이밍을 동기화하는 가장 정직한 방법이다.
애니메이션 아티스트가 공격 모션을 수정하면 AI의 공격 빈도도 자동으로 따라간다.
하드코딩된 1500ms 폴백(`DEFAULT_ATTACK_COOLDOWN_MS`)이 있지만, 파일을 찾을 수 없을 때만 사용된다.
데이터를 단일 출처(single source of truth)로 관리하는 원칙을 코드 레벨에서 실천한 사례다.

### 30년 경력자가 AI 도구로 혼자 MMORPG를 짓는다는 것의 의미

송재경은 1994년 넥슨 공동창업 때 C로 바람의 나라를 만들었고, 1998년 리니지를 상용화하며 한국 MMORPG 시대를 열었다.
MMORPG를 처음부터 끝까지 손으로 짜본 사람이 몇 안 되는데, 그가 2026년에 혼자 다시 그 일을 하고 있다.

주목할 점은 스택의 선택이다.
C/C++ 서버에서 시작해, Java, Unreal, 지금은 Rust + Tokio + WebSocket이다.
브라우저 클라이언트는 Three.js + WebGPU다.
1990년대에 저수준 소켓 프로그래밍으로 동기화 문제를 손으로 풀었던 사람이 지금은 WASM 크로스컴파일과 비동기 Rust 런타임을 사용한다.
30년 전 지식이 지금도 유효하면서 동시에 완전히 다른 도구를 쓰고 있다.

AI 도구의 역할도 단순한 코드 생성이 아닌 것으로 보인다.
`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`가 저장소에 공존하고, README가 스스로 “바이브 코딩”이라 부른다.
그가 AI와 어떻게 협업했는지는 코드에서 직접 드러나지 않지만, 경험 많은 개발자가 AI를 도구로 사용할 때 어떤 결과물이 나오는지를 이 저장소 전체가 보여준다.

신뢰 경계, 프로토콜 설계, 경제 시스템의 수학적 불변식—이 선택들은 30년 게임 개발 경험 없이는 나오기 어려운 판단이다.
“바이브 코딩”은 아무 생각 없이 짠다는 뜻이 아니라, 이미 머릿속에 시스템 전체가 있는 사람이 AI와 빠르게 구현하는 방식일 수 있다.
