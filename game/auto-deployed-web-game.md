# 유저 의견을 모아 매일 자동 개발·배포되는 웹게임 제작기

원문: <https://blog.frogred8.dev/docs/044_automatic_daily_release_game/>

게임: <https://spiralwave.frogred8.dev/>

GitHub: <https://github.com/frogred8/SpiralWave>

## 요약

이 글은 SpiralWave라는 실험적 웹게임 제작기를 다룬다.
Gemini를 활용한 AI 보조 개발을 시험하기 위한 프로젝트로, 스킬 트리가 있는
자원 수집 게임이며 사용자 제안을 매일 자동으로 반영해 배포한다.

개발에서 얻은 핵심 발견은 다음과 같다.
AI 보조로 구현 속도가 전통적 코딩 대비 약 10배 빨라졌지만, 개별 함수는
잘 만들어진 반면 코드 전체의 가독성이 떨어지고 불필요한 복잡성이 포함됐다.
처음에는 2시간마다 자동 릴리스를 계획했으나 런타임 버그 발생률이 약 25%에
달해 포기하고, 매일 테스트된 빌드를 배포하는 방식으로 전환했다.

워크플로는 Cron 작업으로 피드백 수집 → 코드 생성 → 빌드 → 배포의 주기를
순환한다.
AI는 연쇄적 스킬 취소(cascading skill cancellation) 같은 복잡한 로직에서는
어려움을 겪었지만 기본 구조 구현에는 능했다.
개발 속도를 유지하기 위해 테스트는 의도적으로 생략했다.

## 분석

### "2시간 배포 → 일일 배포"라는 후퇴가 말해주는 것

가장 흥미로운 데이터포인트는 2시간마다 자동 배포하려던 계획이 25% 런타임
버그 발생률 때문에 포기됐다는 점이다.
이는 AI 생성 코드의 속도가 곧바로 배포 빈도로 이어지지 않음을 보여준다.
생성 속도가 10배 빨라져도, 검증되지 않은 코드를 더 자주 배포하면 버그도
그만큼 자주 사용자에게 도달한다.
결국 배포 주기를 결정하는 것은 생성 속도가 아니라 검증 신뢰성이다.

### AI는 구조에 능하고 연쇄 로직에 약하다

AI가 기본 구조 구현에는 능했지만 연쇄적 스킬 취소 같은 상태 의존적 로직에서
어려움을 겪었다는 관찰은, 현 단계 코드 생성 능력의 경계를 정확히 짚는다.
독립적이고 패턴화된 코드(CRUD, UI 컴포넌트)는 잘 만들지만, 상태가 서로
얽혀 부수효과가 전파되는 로직은 모델이 전체 상태 공간을 추적하기 어렵다.
이는 [harness engineering 보고서](../agentic-coding/harness-engineering-comprehensive-report.md)가
지적한 "복잡한 로직에서의 AI 한계"와 일치한다.

## 비평

### 강점: 실패까지 정직하게 기록한 실험 로그

이 글의 가치는 "AI로 10배 빨라졌다"는 자랑이 아니라, 2시간 배포 포기·가독성
저하·테스트 생략 같은 트레이드오프를 솔직하게 드러낸 점이다.
[GeekNews 댓글](https://news.hada.io/topic?id=29985)에서 전 과정이 정말
완전 자동화됐는지 물은 것은, 자동화의 실제 경계가 어디인지가 이 실험의
핵심 질문임을 보여준다.

### 약점: 테스트 생략이라는 근본적 절충

개발 속도를 위해 테스트를 의도적으로 생략한 것은 25% 버그 발생률의 직접
원인일 가능성이 크다.
"AI가 10배 빠르다"는 결론은, 테스트라는 검증 비용을 제거한 상태에서의
속도이므로 공정한 비교가 아니다.
생성 속도와 검증 비용을 함께 계산하면, AI 보조 개발의 진짜 생산성은
다르게 보일 수 있다.
또한 가독성이 떨어지고 불필요한 복잡성이 쌓인 코드는, 장기적으로 유지보수
부채로 돌아와 초기 속도 이득을 상쇄할 위험이 있다.

## 인사이트

### 생성이 싸지면 검증이 병목이 된다

이 실험의 가장 선명한 교훈은, AI로 코드 생성이 10배 빨라져도 배포 주기는
오히려 2시간에서 하루로 늘어났다는 역설이다.
생성 비용이 0에 수렴하면, 소프트웨어 수명 주기의 병목은 "만드는 것"에서
"이게 맞는지 확인하는 것"으로 이동한다.
25% 런타임 버그 발생률은 생성 능력의 한계가 아니라 검증 능력의 부재에서
나온 것이고, 테스트를 생략했기에 그 검증이 사용자에게 떠넘겨진 것이다.

이는 AI 코딩 전반의 구조적 진실을 압축한다.
생성이 commodity가 되면 경쟁 우위는 "얼마나 빨리 만드는가"가 아니라
"얼마나 신뢰성 있게 검증하는가"로 옮겨간다.
behaviour harness — 의도한 기능을 올바르게 수행하는지 판정하는 계층 — 가
미성숙한 한, AI가 아무리 빨리 코드를 쏟아내도 그것을 안심하고 배포할
수 없다.
이 프로젝트가 일일 배포로 후퇴한 것은, 결국 검증 계층의 부재를 사람의
하루치 관찰로 메운 것이다.

### "10배 빠르다"는 검증 비용을 빼고 센 숫자다

테스트를 의도적으로 생략한 상태에서 측정한 10배 속도는, 소프트웨어 개발의
전체 비용 중 일부만 센 수치다.
실제 개발 비용은 생성 + 검증 + 디버깅 + 유지보수의 합이고, AI가 생성
단계만 가속하면 나머지 단계의 비용이 상대적으로 커진다.
25% 버그 발생률과 가독성 저하는 그 미뤄진 비용이 나중에 청구되는 형태다.

이 관점은 "AI가 개발을 N배 빠르게 한다"는 주장을 회의적으로 보게 만든다.
생산성을 정직하게 측정하려면 버그가 사용자에게 도달한 비용, 가독성 낮은
코드를 나중에 이해하는 비용, 누적된 복잡성을 리팩터링하는 비용까지
포함해야 한다.
생성 단계만 떼어 측정한 속도 지표는, 빙산의 수면 위만 보고 항해를 결정하는
것과 같다.

### 자동화의 경계는 "결정론적으로 검증 가능한가"에서 갈린다

이 프로젝트에서 자동화가 성공한 부분(빌드, 배포, 피드백 수집)과 실패한
부분(2시간마다 무인 배포)의 경계는 명확하다 — 결과를 결정론적으로 검증할
수 있으면 자동화되고, 그렇지 못하면 사람의 판단이 끼어든다.
빌드 성공 여부는 밀리초 단위로 판정되지만, "이 게임이 재미있고 버그 없이
동작하는가"는 결정론적 검증 대상이 아니다.
그래서 마지막 게이트에 사람의 하루치 테스트가 남았다.

함의는 자동화 설계의 일반 원칙이다.
파이프라인의 어느 단계를 무인화할 수 있는가는, 그 단계의 출력을 신뢰성
있게 자동 판정할 수 있는가에 달렸다.
연쇄적 스킬 취소 같은 상태 의존 로직처럼 검증이 의미론적(semantic)일수록
자동화는 어려워지고, 그 지점이 바로 인간이 루프에 남아야 하는 곳이다.
완전 자동 개발의 꿈은 모델의 생성 능력이 아니라, 자연어 의도를 검증 가능한
불변식으로 컴파일하는 능력이 결정한다.
