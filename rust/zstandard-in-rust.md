# Rust로 구현한 Zstandard

원문: <https://trifectatech.org/blog/announcing-zstandard-in-rust/>

## 요약

Trifecta Tech Foundation이 `libzstd-rs-sys`를 공개했다.
Zstandard(zstd) 압축 포맷의 순수 Rust 구현으로, 재단의 세 번째 압축 프로젝트다
(이전에는 zlib과 bzip2를 구현했다).
2026년 6월 1일 사전 릴리스가 배포됐다.

Zstandard는 gzip보다 빠르고 압축률도 높다.
Meta가 관리하는 C 레퍼런스 구현이 존재하지만, Contributor License Agreement
요건이 참여를 제한한다.
독립적인 Rust 구현은 오픈소스 생태계를 강화하고 대안적 개발 경로를 제공한다.

구현 방식은 c2rust로 C 코드를 초기 변환한 뒤 광범위한 정리를 거쳤다.
현재 압축 해제와 사전 빌더(dictionary builder)가 완성됐고,
압축(인코더) 구현은 별도 마일스톤(Milestone 4)으로 계획됐지만 아직 자금이
확보되지 않았다.

## 분석

### 성능 — 안전성과의 트레이드오프

기본 모드에서 압축 해제가 C 레퍼런스보다 약 3% 느리다.
이 차이는 Rust의 메모리 안전성이 요구하는 실질적 비용이다.
실험적 기능 플래그를 사용하면 네 개의 경계 검사를 비활성화해 C 수준 성능을
달성할 수 있다.
재단은 “대부분의 사용자에게 3% 성능 저하는 메모리 안전성의 적절한 대가”라고
밝혔다.

### 이식성 이점

순수 Rust 구현은 C 툴체인 의존성을 제거한다.
Windows와 WebAssembly 타깃에서 이 이점은 실질적이다.
C 크로스 컴파일러 설정 없이 `cargo build`만으로 빌드된다.
Rust 생태계 안에서 C FFI 없이 의존성을 관리할 수 있는 것도 장점이다.

### ruzstd와의 차별화

이미 ruzstd라는 Rust 구현이 존재하지만, libzstd-rs-sys는 다른 접근을 취한다.
ruzstd는 압축 로직을 처음부터 다시 구현했고, libzstd-rs-sys는 C 코드를
1:1로 포팅했다.
1:1 포팅은 더 빠르고 완전한 구현을 낳는다는 것이 재단의 주장이다.
두 접근의 공존은 생태계의 다양성을 높인다.

### Meta 의존성 리스크

zstd C 레퍼런스 구현의 핵심 관리자가 Meta라는 사실은 단일 실패 지점을 만든다.
Contributor License Agreement가 기여자를 제한하고, Meta의 우선순위 변화가
프로젝트 방향에 영향을 줄 수 있다.
독립 구현은 이 리스크를 분산시킨다.

## 비평

### 강점 — 검증 방법론

레퍼런스 구현의 테스트 스위트, 퍼즈 테스팅, Miri 분석을 통한 검증 체계는
신뢰성을 높이는 올바른 접근이다.
새 구현이 기존 포맷과 100% 호환되어야 하는 압축 라이브러리에서
테스트 방법론은 성능만큼 중요하다.

Chainguard, Astral, NLnet Foundation, Sovereign Tech Agency의 자금 지원도 주목할
만하다.
오픈소스 보안 인프라에 체계적 자금이 흐르기 시작했다는 신호다.

### 약점 — 인코더 미구현

압축 해제만 완성됐고, 압축(인코더)은 아직 자금도 확보되지 않았다.
실용적 관점에서 많은 사용 사례가 압축-해제 양방향 모두를 필요로 한다.
“인코더는 나중에”는 반쪽짜리 릴리스다.

### 빈 자리 — 딕셔너리 압축의 심층 지원

zstd의 핵심 강점 중 하나인 공유 딕셔너리(shared dictionary)를 활용한 압축은
소규모 파일들을 효율적으로 압축하는 데 결정적이다.
딕셔너리 빌더는 포함됐지만, 딕셔너리 기반 압축의 실제 운영 패턴 지원이 어떤지는
불분명하다.

## 인사이트

### “메모리 안전한 인터넷 인프라”는 체계적 사업이 되고 있다

Trifecta Tech Foundation이 zlib, bzip2, zstd를 순서대로 Rust로 다시 구현하는
것은 개별 프로젝트가 아니라 체계적 사업이다.
인터넷의 근간을 이루는 압축·암호화·직렬화 라이브러리들이 C로 작성된 채
방치되어 왔다는 사실을 직시하고, 하나씩 교체해가는 전략이다.

이 흐름은 미국 CISA와 NSA가 2022년부터 메모리 안전 언어로의 전환을 공식
권고한 것과 맥락이 같다.
“보안 인프라의 Rust 재작성”이 개인 해커의 취미에서 기관 자금을 받는 조직적
사업으로 전환되고 있다.
앞으로 5~10년 안에 C로 작성된 주요 저수준 라이브러리 대부분이 Rust 대안을
갖게 될 것이다.

### 3% 성능 차이가 “허용 가능한가”의 기준은 워크로드에 달려 있다

“3% 느리다”는 주장은 숫자 자체로는 무의미하다.
스트리밍 비디오 서비스에서 압축·해제가 핫패스에 있다면 3%는 인프라 비용에
직접 영향을 준다.
반면 배치 처리나 개발자 도구에서는 전혀 중요하지 않다.

경계 검사를 끄는 실험적 플래그가 있다는 사실은 이중적이다.
프로덕션에서 실험적 플래그를 쓰는 것은 Rust의 안전성 보장을 부분적으로 포기하는
것인데, 그렇다면 Rust 구현의 핵심 이점이 줄어든다.
성능-안전성 트레이드오프를 런타임에 설정으로 제어하는 접근은 기술적으로
영리하지만, 사용자가 올바른 선택을 하도록 충분한 가이드가 필요하다.

### 오픈소스 생태계 강화의 진짜 의미

“오픈소스 생태계를 강화한다”는 표현은 막연하게 들리지만, 구체적 위험을
가리킨다.
Meta가 zstd를 관리한다는 사실은, Meta가 어떤 이유로든 우선순위를 바꾸면
전 세계 인터넷 인프라의 압축 레이어에 영향이 생긴다는 뜻이다.
실제로 OpenSSL이 오랫동안 단 두 명의 전업 관리자에 의존했다는 Heartbleed 이후의
교훈이 반복되지 않으려면 핵심 인프라의 다중 구현이 필요하다.

독립 구현은 “더 좋은 구현”이 목적이 아닐 수 있다.
“특정 기업에 의존하지 않는 구현”이 목적이고, 그것만으로도 존재 이유가 충분하다.
오픈소스 지속 가능성의 관점에서 단일 관리자 리스크를 분산하는 것은 기술 부채가
아니라 인프라 보험이다.
