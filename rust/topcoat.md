# Topcoat: The full full-stack framework for Rust

<https://github.com/tokio-rs/topcoat>

HN 토론: <https://news.ycombinator.com/item?id=48952067> (106점, 42개 댓글)

## 소개

Topcoat은 Tokio 재단이 관리하는 Rust 풀스택 웹 프레임워크다.
“배터리 포함(batteries-included)”을 표방하며 라우팅, 서버 사이드 렌더링, 클라이언트
반응성, 에셋 번들링, Tailwind 통합을 하나의 프레임워크로 제공한다.
조기 단계이며 브레이킹 체인지가 예상된다고 명시돼 있다.
Tokio 창시자인 carllerche는 HN에서 저장소가 예정보다 일찍 공개된 것은 CI 무료 사용량이
소진됐기 때문이라고 밝혔다.[^carllerche]
곧 상세 내용을 담은 블로그 포스트를 올릴 예정이라고 했다.

핵심 설계 아이디어는 `$(...)` 표현식이다.
타입 검사된 일반 Rust 코드를 서버 초기 렌더링에서 실행하면서 동시에 JavaScript로
변환해 브라우저에서도 재실행되게 한다.
WASM 번들도 클라이언트 빌드 단계도 없다.

```rust
view! {
    signal open = false;

    <button @click=$(|_e| open.set(!open.get()))>"What is Topcoat?"</button>
    <p :hidden=$(!open.get())>"A fullstack Rust framework."</p>
}
```

서버 왕복이 필요한 컴포넌트는 `#[shard]`로 선언한다.
`$(...)` 인수가 바뀔 때 서버에서 재렌더링하고 HTML을 교체한다.

```rust
#[shard]
async fn search_results(cx: &Cx, query: String) -> Result {
    view! {
        <ul>
            for product in search_products(cx, &query).await? {
                <li>(product.name)</li>
            }
        </ul>
    }
}
```

모듈 기반 라우팅은 파일 시스템 구조에서 라우트 트리를 자동 유추한다.
`src/app/posts/id.rs`가 `/posts/{post_id}`로 연결된다.
에셋 번들러는 컴파일된 바이너리에서 `asset!()` 호출을 스캔해 파일을 수집하고
콘텐츠 해시 URL로 제공한다.

## 분석

### `$(...)` 표현식이 해결하는 문제

전통적인 서버 사이드 렌더링에서 클라이언트 반응성을 추가하려면 JavaScript 레이어가
필요하고, 두 코드베이스 간의 동기화 문제가 생긴다.
SPA 아키텍처는 이를 클라이언트 전체 렌더링으로 해결했지만 서버 로직과의 분리 비용이 컸다.
Next.js, Remix 같은 메타프레임워크는 두 레이어 사이에 타입 안전한 API 계층을 제공하지만
복잡도는 여전히 높다.

`$(...)`의 접근은 다르다.
“이 Rust 코드가 서버와 브라우저에서 모두 의미 있는 코드”라는 교집합을 정의하는 것이다.
단순 상태 토글 같은 클라이언트 로직은 JavaScript로 변환하고, 데이터베이스 접근이 필요한
경우는 `#[shard]`로 서버 왕복을 표현한다.
개발자는 이 두 가지만 구분하면 된다.

`$(...)` 내 Rust 코드가 실제로 JavaScript로 어떻게 변환되는지는 핵심 기술적 질문이다.
일반 Rust를 JavaScript로 트랜스파일하는 것은 복잡하므로, 실용적으로는 제한된 Rust
부분 집합만 지원될 것이다.
이 제약의 범위가 프레임워크의 표현력을 결정한다.

### Tokio 재단의 포지셔닝

Topcoat이 tokio-rs 조직에 속한다는 것은 의미 있다.
Tokio는 Rust 비동기 생태계의 사실상 표준 런타임이다.
Axum, Tonic 같은 서버 프레임워크들도 tokio-rs 조직에서 관리한다.

풀스택 프레임워크를 Tokio 재단이 직접 만든다는 것은 두 가지를 의미한다.
기술적으로 비동기 런타임과의 통합이 최적화될 가능성이 높다.
생태계적으로 Rust 서버 개발의 표준 스택으로 자리 잡을 수 있다.
React가 Meta에서 관리되며 생태계에 권위를 갖는 것과 유사한 역학이 작동할 수 있다.

### 모듈 기반 라우팅의 절충

파일 시스템 구조에서 라우트를 유추하는 것은 Next.js, SvelteKit, Remix가 정착시킨 패턴이다.
Topcoat은 이것을 Rust 모듈 시스템에 적용한다.
직관적이고 설정이 적다는 장점이 있다.

대가는 유연성이다.
도메인 모델과 URL 구조가 자연스럽게 일치하지 않을 때, 파일 시스템 기반 라우팅은
제약이 된다.
대규모 애플리케이션에서 라우트와 도메인 로직이 얽혀 있을 때 이 구조를 유지하는 것이
어떻게 작동하는지는 더 많은 실전 경험이 필요하다.

## 비평

### “Rust로 JavaScript 없이”의 실행 가능성

`$(...)` 접근의 핵심 약속은 JavaScript를 직접 쓰지 않는 것이다.
그러나 복잡한 UI 상호작용에서 이 약속이 얼마나 유지될 수 있는지는 불명확하다.
애니메이션, 드래그 앤 드롭, 캔버스 조작 같은 기능은 브라우저 API에 직접 접근해야 한다.
Rust 코드가 JavaScript로 변환될 때 브라우저 DOM API를 어떻게 표현하는지는
프레임워크의 현실적 한계를 결정하는 질문이다.

LiveView(Elixir), HTMX, Hotwire도 비슷한 “JavaScript 최소화” 접근을 취하지만,
복잡한 UI에서는 여전히 JavaScript 탈출구가 필요하다.
Topcoat이 이 한계를 어떻게 다룰지가 채택 가능 범위를 결정할 것이다.

### 조기 단계의 안정성 리스크

“브레이킹 체인지가 예상된다”는 명시는 정직하지만, 이것이 실용적 채택을 어렵게 한다.
578 스타의 초기 단계 프레임워크에 의존하는 프로덕션 애플리케이션은 프레임워크가 API를
바꿀 때마다 마이그레이션 비용을 감당해야 한다.

Rust 생태계의 프레임워크는 Tokio와 Axum처럼 오랜 안정화 과정을 거쳐 신뢰를 얻었다.
Topcoat이 tokio-rs 조직 소속이라는 것이 신뢰 신호를 주지만, 풀스택 프레임워크의
복잡도를 고려하면 안정화까지 상당한 시간이 필요할 것이다.

Axum 사용자인 hobofan은 이 점을 더 날카롭게 지적했다.[^hobofan]
tokio 프로젝트가 만든 것이라는 이름값 때문에 실질적 완성도보다 과도한 채택이 이뤄질 수
있다는 우려다.
커뮤니티 영향력 있는 조직이 초기 단계 프레임워크를 공개할 때 따라오는 구조적 위험을
지적한 것이다.

### ORM 부재라는 근본적 공백

HN에서 여러 댓글이 같은 지점을 짚었다.
“배터리 포함” 프레임워크를 표방하지만 ORM이나 데이터베이스 추상화 레이어가 없다는 것이다.

hoistbypetard는 이것을 프레임워크 정의의 문제로 봤다.[^hoistbypetard]
ORM이 없다면 “배터리 포함”이라는 주장을 정당화하려면 최소한 “왜 ORM을 넣지 않았고
그 대신 무엇을 쓰면 되는가”를 명확히 설명해야 한다는 것이다.
frio는 Topcoat이 htmx 스타일의 서버 렌더링을 잘 해결해도 Django가 제공하는 자동 어드민,
인증 프레임워크, 캐싱, 이벤트 처리 같은 기능 없이는 한계가 있다고 지적했다.[^frio]

이것은 “풀스택”이라는 단어의 기준 문제이기도 하다.
서버와 클라이언트를 아우르는 렌더링 모델은 풀스택의 일부이지만,
실제 애플리케이션 개발에서 필요한 데이터 영속성, 인증, 어드민 도구 없이는
프레임워크가 아니라 라이브러리에 가깝다.

### 서버 중심 아키텍처의 트레이드오프

Topcoat은 서버 사이드 렌더링을 기반으로 클라이언트 반응성을 추가하는 모델이다.
이 접근은 SEO, 초기 로드 성능, 데이터베이스 접근 단순화에서 유리하다.

반면 오프라인 동작, 로컬-우선 데이터 동기화, 복잡한 실시간 협업 같은 시나리오에서는
클라이언트 중심 SPA가 더 자연스럽다.
Topcoat이 이런 사용 사례를 커버하려면 추가 추상화가 필요하고, 그러면 단순성이라는
핵심 가치와 충돌할 수 있다.

## 인사이트

### Rust 풀스택의 가능성과 시장

Rust 웹 개발은 오랫동안 “백엔드에는 충분하지만 풀스택으로는 복잡하다”는 인식이 있었다.
Topcoat의 등장은 이 인식을 바꾸려는 시도다.
Next.js나 Rails가 했던 것처럼, 결정을 미리 내려서 보일러플레이트를 줄이는 것이다.

Rust가 시스템 소프트웨어를 넘어 웹 애플리케이션 개발로 영역을 넓히려면 이런 프레임워크가
필요하다.
저수준 성능보다 개발자 경험이 중요한 영역에서 Rust가 채택되려면 진입 장벽이 낮아야 한다.
Topcoat의 방향이 성공한다면 Rust 개발자 풀이 넓어지고, 그것이 생태계 성장의 선순환을
만들 수 있다.

### `view!` 매크로 기반 템플릿의 위험

Rust 매크로는 강력하지만 디버깅과 컴파일 오류 메시지가 어렵다는 오랜 문제가 있다.
`view!` 같은 DSL 매크로에서 오류가 나면 메시지가 매크로 내부를 가리키며 원인 파악이
어려울 수 있다.

Topcoat이 제공하는 `topcoat fmt` CLI가 이 문제를 부분적으로 해소하지만, 근본적으로
매크로 기반 템플릿의 디버깅 경험은 일반 함수 호출보다 어렵다.
이것이 프레임워크 채택의 숨겨진 비용이 될 수 있다.

### Tokio가 Rust의 Spring이 될 수 있는가

valorzard는 HN에서 흥미로운 질문을 던졌다.[^valorzard]
Tokio가 Rust의 Spring(Boot)이 되어가고 있는 것 아닌가 하는 것이다.

Spring/Spring Boot는 Java 생태계에서 한 재단(Pivotal/VMware)이 런타임, 웹 프레임워크,
데이터 레이어, 보안, 배치 처리를 하나의 우산 아래 묶은 사례다.
Tokio는 비동기 런타임에서 시작해 Axum(웹), Tonic(gRPC), Tower(미들웨어)를 거쳐
이제 Topcoat(풀스택)까지 확장하고 있다.
수직적 통합의 경로가 유사하다.

Spring 생태계의 장점은 구성 요소 간 일관성과 공식 지원이다.
단점은 Spring 방식에 묶이면 탈출이 어렵다는 것이다.
Tokio가 같은 경로를 걷는다면, Rust 웹 생태계의 다양성이 줄어드는 대신
기업 채택이 빨라지는 트레이드오프가 생길 것이다.
hobofan이 이름값 기반 채택을 우려한 것도 같은 맥락에서 이해할 수 있다.

### 서버와 클라이언트 경계의 명시화 트렌드

`$(...)` 표현식이 “이 코드는 클라이언트에서 실행된다”를 나타내는 방식은
React Server Components의 `“use client”` 지시어, Next.js의 서버/클라이언트 컴포넌트
구분과 같은 방향이다.
웹 프레임워크가 서버와 클라이언트 경계를 타입 시스템이나 컴파일러 수준에서 명시화하는
트렌드가 강해지고 있다.

Topcoat의 Rust 기반 구현이 이 경계를 타입 안전하게 만들 수 있다면,
“서버에서 실행해야 할 코드가 실수로 클라이언트에 노출되는” 종류의 버그를 컴파일 타임에
막는 것이 가능하다.
이것이 Rust 타입 시스템이 웹 개발에서 실질적 가치를 제공할 수 있는 지점이다.
