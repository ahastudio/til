# Papermake

<https://github.com/rkstgr/papermake>

## 소개

Papermake는 Typst 문서 템플릿을 분산 PDF 렌더링 서비스로
바꿔주는 프로젝트다.
README는 이를 “Typst 템플릿을 PDF API로 바꾸세요. 한 번
게시하면 어디서든 렌더링할 수 있습니다”라는 문구로 소개하며,
서버 사이드 렌더링을 지원하는 템플릿 레지스트리로서 문서
템플릿을 위한 Docker와 비슷한 역할을 한다고 설명한다.

핵심 기능은 다음과 같다.
불변 버전과 가변 태그를 함께 지원하는 버전 관리 방식으로,
`invoice:v1.0.0`처럼 영구적인 버전과 `invoice:latest`처럼
움직이는 태그를 동시에 부여해 재현성과 갱신 가능성을 모두
확보한다.
콘텐츠 주소 지정 저장소(content-addressed storage)를
사용해 파일을 SHA-256 해시로 저장하고 중복을 제거하는데,
이는 Git의 저장 방식을 그대로 본뜬 것이다.
로컬에 Typst를 설치할 필요 없이 HTTP 엔드포인트를 통해
렌더링이 이뤄지는 서버 사이드 렌더링을 지원하며, 각 렌더링마다
입력·출력 해시를 기록해 어떤 PDF든 원본 템플릿과 데이터로
추적할 수 있는 감사 추적(audit trail) 기능을 갖췄다.
시스템 구성은 “Rust 바이너리 하나에 S3와 ClickHouse”로
요약되는 셀프 호스팅 아키텍처다.

프로젝트는 네 개의 Rust 크레이트로 구성된다.
가상 파일 시스템을 갖춘 핵심 렌더링 엔진인 `papermake`(
crates.io에 게시됨), S3를 통한 템플릿 저장과 ClickHouse를
통한 렌더링 이력 관리를 담당하는 `papermake-registry`,
Axum 기반 HTTP API인 `papermake-server`, 큐 기반 렌더링
워커인 `papermake-worker`(실험적 기능)다.
템플릿은 Typst 문법을 사용하며 데이터는 `#data`로 접근한다.
예를 들어 `= Invoice #data.number`, `*Bill To:*
#data.customer.name`, `*Amount:* $#data.amount` 같은
형태다.
워크플로는 멀티파트 업로드로 템플릿을 게시하고, JSON
데이터로 렌더링을 요청한 뒤, 반환된 렌더 ID로 PDF를
다운로드하는 순서로 진행된다.

PDF 표준 지원 측면에서는 보관용 PDF/A-3b와 전자 인보이스
표준인 ZUGFeRD/Factur-X 준수를 포함한 여러 표준을
지원한다.
렌더링은 디스크 접근이 없는 가상 파일 시스템에서 이뤄지며,
이미지·임포트·데이터 파일 등 참조되는 모든 파일은 명시적으로
제공되어야 한다.
시스템 폰트는 자동으로 탐지되며, 추가 폰트가 필요하면
`FONTS_DIR`이 가리키는 디렉터리를 지정하면 된다.
저장소는 서버와 MinIO(S3 호환 스토리지), ClickHouse를
함께 띄우는 Docker Compose 설정을 제공하며, Docker 없이
실행하려면 외부 S3와 ClickHouse 인스턴스를 가리키는 환경
변수를 설정해야 한다.
핵심 렌더러는 독립형 Rust 라이브러리로도 제공되어, 서버
인프라 없이도 직접 PDF를 생성할 수 있다.
저장소는 163개의 스타, 12개의 포크, 두 번의 릴리스(최신
버전 v0.3.0, 2026년 7월)를 기록하고 있으며, Rust가 55.5%,
실험적 웹 UI를 위한 TypeScript가 42.8%를 차지한다.
라이선스는 Apache-2.0이다.

## 분석

### Papermake는 Typst의 “언어”와 “배포”를 분리하려는 시도다

Typst 자체는 로컬에 설치해 컴파일하는 도구이지, 원격에서
호출 가능한 서비스가 아니다.
Papermake가 하는 일은 이 언어 위에 레지스트리, 버전 관리,
HTTP API라는 배포 계층을 얹는 것이다.
이는 프로그래밍 언어의 역사에서 반복적으로 나타난 패턴과
같은 구조다.
언어 자체는 컴파일러나 인터프리터로 존재하지만, 그 언어로
작성된 산출물을 배포하고 버전 관리하고 공유하는 문제는
별도의 생태계 계층(npm, crates.io, Docker Hub)이 해결해
왔다.

Papermake의 설계는 “Docker와 비슷하다”는 README의 비유를
문자 그대로 구현한 것에 가깝다.
Docker가 컨테이너 이미지를 레이어로 나누고 콘텐츠 해시로
식별해 캐싱과 중복 제거를 가능하게 한 것처럼, Papermake는
Typst 템플릿과 그 의존 자산을 SHA-256 해시로 식별해 같은
효과를 노린다.
다만 Docker가 임의의 실행 환경을 패키징하는 범용 도구인
반면, Papermake는 오직 “Typst 템플릿을 렌더링한다”는
단일 목적에 특화되어 있다는 점에서 적용 범위가 훨씬
좁다.

### 감사 추적 기능은 이 프로젝트가 규제 산업을 겨냥해 설계되었음을 보여준다

README가 강조하는 기능 중 “각 렌더링의 입력·출력 해시를
기록해 어떤 PDF든 원본 템플릿과 데이터로 추적할 수 있다”는
설명과 ZUGFeRD/Factur-X 같은 전자 인보이스 표준 지원은
일반적인 개발자 도구에서는 우선순위가 낮은 기능이다.
이런 기능이 핵심 기능 목록에 포함되어 있다는 것은,
Papermake가 처음부터 금융·회계처럼 문서의 출처와 무결성을
증명해야 하는 규제 산업을 염두에 두고 설계되었다는 뜻이다.

이 설계 방향은 앞서 살펴본 `aws/rendering-one-million-pdfs.md`
문서에서 다룬 사례, 즉 Typst와 Papermake를 함께 사용해
금융기관용 거래 확인서를 대량 생성한 사례와 정확히
맞아떨어진다.
그 글에서 Papermake는 데이터 보간과 스키마 검증을 제공하는
라이브러리로만 언급되었지만, 이 저장소를 직접 살펴보면
그 라이브러리가 실은 감사 가능성과 재현성을 1급 시민으로
취급하는 훨씬 더 큰 설계 목표의 일부라는 것이 드러난다.

### 콘텐츠 주소 지정과 불변 버전의 조합은 “재현성”과 “편의성” 사이의 타협이다

콘텐츠 주소 지정 저장소는 완벽한 재현성을 보장한다.
같은 해시는 항상 같은 콘텐츠를 가리키기 때문이다.
그러나 이것만으로는 사용자가 SHA-256 해시를 외워 템플릿을
호출해야 하므로 실용성이 떨어진다.
Papermake는 여기에 `invoice:v1.0.0` 같은 불변 버전과
`invoice:latest` 같은 가변 태그를 얹어 이 문제를 해결한다.

이 조합은 Git이 커밋 해시와 브랜치/태그를 함께 제공하는
방식, 그리고 컨테이너 레지스트리가 이미지 다이제스트와
태그를 함께 제공하는 방식과 본질적으로 동일한 해법이다.
즉 Papermake가 채택한 버전 관리 모델은 새로운 발명이라기보다,
이미 여러 시스템에서 검증된 “해시는 진실을 보장하고 태그는
편의를 보장한다”는 패턴을 문서 템플릿이라는 새 영역에
적용한 것이다.
이런 패턴 재사용 자체는 비판할 이유가 없지만, 그 재사용이
얼마나 성숙하게 구현되었는지는 프로젝트의 두 릴리스와
163개의 스타라는 이른 단계 지표만으로는 판단하기 어렵다.

## 비평

### “한 번 게시하면 어디서든 렌더링할 수 있다”는 슬로건은 서버 인프라 의존성을 가린다

README의 핵심 슬로건은 “Publish once, render anywhere”이지만,
실제로 이 “어디서든”은 Papermake 서버가 배포되어 있고 그
서버가 S3와 ClickHouse에 접근할 수 있는 환경으로 한정된다.
이는 로컬에서 파일 하나로 완결되는 Typst 원본의 “어디서든
컴파일 가능하다”는 성질과는 전혀 다른 종류의 “어디서든”이다.
슬로건이 사용하는 “anywhere”라는 단어는 Docker 이미지가
어떤 호스트에서든 실행되는 이식성을 연상시키지만, 실제로는
렌더링을 요청하는 클라이언트가 아니라 인프라를 갖춘 서버
운영자에게만 해당되는 이식성이다.

이 차이는 사소하지 않다.
핵심 렌더러가 독립형 Rust 라이브러리로도 제공된다는 점을
README가 뒤늦게 언급하는 것도, 이 슬로건이 함축하는 “서버
없이도 어디서든”이라는 기대와 실제 아키텍처의 기본값인
“서버와 S3와 ClickHouse가 있어야 한다”는 요구사항 사이의
간극을 메우기 위한 것으로 보인다.
슬로건과 실제 배포 요구사항 사이의 이 간극은, 사용자가
가벼운 로컬 도구를 기대하고 도입했다가 S3와 ClickHouse
운영 부담을 뒤늦게 마주하게 될 위험을 만든다.

### ClickHouse를 렌더 이력 저장소로 선택한 이유가 검증되지 않았다

Papermake는 렌더링 이력을 저장하는 데 분석용 컬럼 지향
데이터베이스인 ClickHouse를 사용한다.
그러나 렌더 이력이라는 데이터는 전형적으로 “PDF 하나당
레코드 하나”라는 트랜잭션성 쓰기 패턴을 가지며, 대규모
집계 쿼리보다는 특정 렌더 ID로 단건 조회하는 경우가
많을 것으로 예상된다.
이런 접근 패턴은 일반적으로 ClickHouse 같은 분석 특화
데이터베이스보다 PostgreSQL 같은 범용 트랜잭션 데이터베이스에
더 적합하다.

README는 왜 ClickHouse를 선택했는지 근거를 전혀 제시하지
않는다.
가능한 설명은 “100만 개 PDF를 10분 안에 생성한다”는
사용 사례에서 초당 수천 건의 쓰기를 감당해야 하기 때문일
수 있지만, 이는 추측일 뿐 README에서 확인되지 않는다.
만약 실제 이유가 대규모 쓰기 처리량이라면, 이는 Papermake의
설계가 일반적인 문서 관리 도구가 아니라 처음부터 극단적인
처리량을 가정한 틈새 사용 사례에 맞춰졌다는 뜻이며, 그보다
작은 규모의 조직이 이 프로젝트를 도입할 때는 과잉 설계
(over-engineering)로 작용할 수 있다.

### 실험적 컴포넌트와 안정적 컴포넌트가 구분 없이 하나의 저장소로 제시된다

README는 `papermake-worker`를 “실험적(experimental)”이라고
명시하면서도, 이 컴포넌트를 핵심 렌더링 엔진이나 HTTP
서버와 동일한 목록에 나란히 배치한다.
네 개의 크레이트 중 하나가 아직 실험 단계라는 사실은,
프로젝트 전체의 성숙도가 균일하지 않다는 뜻이다.
그러나 README의 서술 방식은 이 차이를 강조하지 않고 마치
네 컴포넌트가 동등한 완성도를 가진 것처럼 나열한다.

두 번의 릴리스와 163개의 스타라는 지표를 고려하면, 이
프로젝트 전체가 상당히 초기 단계에 있다는 합리적 추정이
가능하다.
그런 초기 단계에서 “실험적”이라는 딱지가 붙은 컴포넌트에
의존하는 사용자는 프로덕션 배포 이후에도 API가 크게 바뀌거나
컴포넌트 자체가 폐기될 위험을 감수해야 하는데, README는
이 위험의 구체적인 범위(어떤 API가 안정적이고 어떤 API가
바뀔 수 있는지)를 명시하지 않는다.

## 인사이트

### 문서 템플릿의 컨테이너화는 “문서”와 “소프트웨어 아티팩트” 사이의 경계를 지운다

Papermake가 Typst 템플릿을 SHA-256 해시로 식별하고
불변 버전과 가변 태그로 관리하는 방식은, 문서를 소프트웨어
빌드 산출물과 동일한 방식으로 다루겠다는 선언이다.
전통적으로 문서는 워드 프로세서 파일이나 이메일 첨부파일로
취급되어 왔고, 버전 관리는 파일명에 날짜나 “최종_진짜최종”
같은 수식어를 붙이는 방식으로 이뤄져 왔다.
Papermake는 이 문제를 소프트웨어 엔지니어링이 이미 오래전에
해결한 방법, 즉 콘텐츠 주소 지정과 불변 버전으로 대체한다.

이 접근이 확산되면, 문서 생성 파이프라인도 소프트웨어
배포 파이프라인과 동일한 도구(CI/CD, 이미지 레지스트리에
준하는 템플릿 레지스트리, 롤백 전략)로 관리될 수 있게 된다.
장기적으로 이는 “문서”라는 범주 자체가 소프트웨어 아티팩트의
부분집합으로 재정의되는 흐름의 한 사례로 볼 수 있다.
다만 이 재정의가 성립하려면 문서를 다루는 비개발자(회계
담당자, 법무팀, 컴플라이언스 담당자)도 이 새로운 패러다임
아래에서 작업할 방법을 익혀야 하는데, README는 개발자
워크플로(멀티파트 업로드, JSON 렌더링 요청)만 보여줄 뿐
비개발자용 인터페이스에 대해서는 언급하지 않는다.

### 감사 추적 기능은 규제 요구를 만족시키는 동시에 새로운 데이터 유출 표면을 만든다

각 렌더링의 입력·출력 해시를 기록하는 감사 추적 기능은
“이 PDF가 어떤 데이터로 만들어졌는가”를 증명하는 데
유용하다.
그러나 이는 동시에 렌더링에 사용된 모든 고객 데이터의
지문(fingerprint)이 ClickHouse라는 별도의 저장소에
영구적으로 남는다는 뜻이기도 하다.
원본 PDF는 삭제되더라도 그 PDF를 만든 입력 데이터의 해시
기록은 감사 추적이라는 목적 때문에 삭제되지 않고 남을
가능성이 크다.

이는 감사 가능성과 데이터 최소화(data minimization) 원칙이
충돌하는 지점이다.
GDPR 같은 개인정보 규제는 더 이상 필요하지 않은 개인정보를
삭제할 권리(잊힐 권리)를 요구하지만, 금융 규제는 반대로
문서 생성 이력을 장기간 보존할 것을 요구하는 경우가 많다.
Papermake처럼 감사 추적을 핵심 기능으로 내세우는 시스템을
도입하는 조직은, 이 두 규제 요구가 정면으로 충돌하는 지점을
직접 설계해서 풀어야 하며, 감사 추적 기능이 있다는 사실
자체가 이 문제를 자동으로 해결해주지는 않는다.

### 틈새 인프라 도구의 생존은 상위 언어(Typst)의 성공 여부에 완전히 종속된다

Papermake의 존재 이유는 전적으로 Typst라는 조판 언어의
채택에 달려 있다.
만약 Typst가 `documentation/typst.md`에서 다룬 것처럼
학술 출판 생태계에서 충분한 지지를 얻지 못하고 도태된다면,
Papermake처럼 그 위에 얹힌 인프라 계층도 존재 이유를
잃는다.
이는 기반 기술에 완전히 종속된 “기생적(parasitic)” 생태계
계층이 갖는 근본적인 취약성이다.

이런 종속 관계는 프로그래밍 언어 생태계에서 낯설지 않다.
CoffeeScript가 쇠퇴하면서 그 위에 만들어졌던 빌드 도구와
프레임워크들이 함께 사라진 사례가 대표적이다.
Papermake의 163개 스타와 두 번의 릴리스라는 초기 단계
지표는 이 프로젝트가 아직 Typst 생태계 안에서도 검증되지
않은 실험적 계층이라는 뜻이며, 투자를 고려하는 조직은
Papermake 자체의 완성도뿐 아니라 그 기반이 되는 Typst의
장기 생존 가능성까지 함께 평가해야 한다.
이는 인프라 도구를 선택할 때 자주 간과되는, 그러나 장기적으로
가장 큰 위험이 되는 요인이다.
