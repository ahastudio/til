## 1부. 원칙 (Principles)

### 에이전틱 엔지니어링이란

Willison은 에이전틱 엔지니어링을 “코드를 작성할 뿐 아니라 직접 실행할 수
있는 코딩 에이전트의 도움을 받아 소프트웨어를 개발하는 것”으로 정의한다.
Claude Code, OpenAI Codex, Gemini CLI가 그 예다.
에이전트의 정의는 “목표를 달성하기 위해 루프 안에서 도구를 실행하는 것”이다—
에이전트가 프롬프트와 도구 정의를 담아 LLM을 호출하고,
요청된 도구를 실행한 뒤 결과를 다시 LLM에 먹이는 과정을 목표가 달성될 때까지 반복한다.
코드 실행 능력이 없으면 LLM 출력의 가치는 제한적이지만,
실행이 더해지면 에이전트는 “증명 가능하게 동작하는” 소프트웨어를 향해 반복할 수 있다.

바이브 코딩(vibe coding)은 Andrej Karpathy가 2025년 2월에 만든 용어로
코드의 존재 자체를 무시한 채 LLM에게 코드를 시키는 것이다.
에이전틱 엔지니어링은 그 검토되지 않은 프로토타입 품질의 결과물을
의도적인 검토와 반복으로 프로덕션 품질까지 끌어올린다는 점에서 정반대 끝에 선다.
핵심은 코드 작성이 엔지니어의 유일한 활동이었던 적이 없다는 것이다.
엔지니어는 트레이드오프가 다른 수십 가지 해법 중 무엇을 만들지를 결정한다.
그래서 에이전틱 엔지니어링에서도 사람의 책임은 그대로 남는다:
에이전트에게 적절한 도구를 주고, 문제를 알맞은 수준으로 명세하며,
결과가 문제를 견고히 푼다고 확신할 때까지 검증·반복하고,
배운 교훈을 바탕으로 지시를 의도적으로 갱신하는 것이다.
이 일을 잘하면 더 야심찬 프로젝트를, 더 높은 품질로, 더 큰 문제에 적용할 수 있다.

### 코드 작성은 이제 거의 공짜다

깨끗하고 테스트된 코드 수백 줄을 만드는 데 대부분의 개발자는 하루 이상 걸렸고,
우리의 엔지니어링 습관 전체가 이 핵심 제약 위에 세워져 있었다.
매크로 수준에서는 설계·추정·계획에 시간을 쏟아 비싼 코딩 시간을 효율적으로 썼고,
기능 아이디어를 개발 비용 대비 가치로 평가했다.
마이크로 수준에서는 하루에 수백 번의 트레이드오프를 결정했다—
이 함수를 리팩터링할까, 문서를 쓸까, 이 엣지 케이스에 테스트를 추가할까,
디버그 인터페이스를 만들 가치가 있을까.
코딩 에이전트가 이 비용을 극적으로 낮추면서 기존의 개인적·조직적 직관이 무너진다.
병렬 에이전트를 돌리면 한 명의 엔지니어가 구현·리팩터링·테스트·문서화를 동시에 진행할 수 있다.

다만 “코드 작성”이 싸진 것이지 “좋은 코드”가 공짜가 된 것은 아니다.
Willison이 정의하는 좋은 코드의 아홉 가지 기준은 그대로 코드 리뷰 체크리스트가 된다:

1. **동작한다.** 버그 없이 의도대로 작동한다.
2. **동작함을 안다.** 검증 절차를 거쳤다.
3. **올바른 문제를 푼다.**
4. **에러를 우아하게 처리한다.** 해피 패스만 고려하지 않으며,
   에러 메시지는 미래의 유지보수자가 이해할 수 있어야 한다.
5. **단순하고 최소한이다.** 사람과 기계 모두 이해·유지보수할 수 있는 방식으로 필요한 것만 한다.
6. **테스트로 보호된다.** 지금 동작함을 증명하고 미래의 리그레션을 방지한다.
7. **적절히 문서화된다.** 코드가 바뀌면 문서도 갱신된다.
8. **미래 변경을 수용한다.** YAGNI를 지키되 미래 변경을 불필요하게 어렵게 만들지 않는다.
9. **기타 “-ility”들.** 접근성·테스트 가능성·신뢰성·보안·유지보수성·관측 가능성·확장성·사용성 등.

에이전트가 대부분을 도울 수 있지만,
“이 프로젝트에 필요한 좋음의 수준”을 판단하고 보장하는 책임은 여전히 개발자에게 있다.
그래서 Willison의 실천적 조언은 역설적이다:
“만들 가치가 없다”는 직감이 들 때마다 일단 프롬프트를 날려라.
비동기 에이전트 세션에서 최악의 경우는 10분 뒤에 토큰 낭비였음을 확인하는 것뿐이다.

### 할 줄 아는 것을 비축하라 (Hoard things you know how to do)

소프트웨어 전문성은 무엇이 기술적으로 가능하고 불가능한지 아는 것이다.
“웹페이지가 JavaScript만으로 OCR을 돌릴 수 있는가?
iPhone 앱이 실행 중이 아닐 때도 블루투스 기기와 페어링할 수 있는가?”
Willison의 표현으로, 무언가가 이론적으로 가능함을 아는 것과
직접 해내는 것을 본 것은 같지 않다.

그래서 그는 자신의 개인 지식 저장소를 꾸준히 비축한다.
크게 네 갈래다:
블로그(`simonwillison.net`)와 TIL 블로그(`til.simonwillison.net`)에는 해결한 문제가 글로 남고,
1,000개가 넘는 GitHub 저장소에는 개념 증명과 프로젝트 코드가 쌓이며,
`tools.simonwillison.net`에는 JavaScript와 CSS를 한 페이지에 담은 HTML 도구들이 모이고,
`simonw/research` 저장소에는 에이전트가 문제를 조사해 동작하는 코드와 보고서를 남긴 복잡한 예제가 담긴다.

비축이 어떻게 신규 작업으로 이어지는지는 OCR 도구 사례가 잘 보여준다.
텍스트 레이어가 없는 스캔 이미지 PDF에서 글자를 뽑는 브라우저용 OCR 도구가 필요했을 때,
그는 이미지에서 글자를 뽑는 Tesseract.js(WebAssembly 기반 OCR)와
PDF 페이지를 이미지로 렌더링하는 Mozilla의 PDF.js를 이미 알고 있었다.
두 라이브러리를 쓰는 동작 예제 코드 조각 두 개를 모델에 주고
“PDF 드래그앤드롭 업로드 → JPEG 변환 → OCR → textarea 출력”을 한 HTML 페이지로 합쳐 달라고 했고,
Claude 3 Opus는 몇 분의 반복 만에 작동하는 프로토타입을 내놓았다.

현대 코딩 에이전트는 비축한 예제의 가치를 세 방향으로 증폭한다.
인터넷 접근이 가능하면
“`curl`로 `tools.simonwillison.net/ocr` 소스를 받아서,
PDF 페이지를 골라 Gemini에 넘겨 삽화의 바운딩 박스를 받는 새 도구를 만들어줘”처럼 시킬 수 있다.
로컬 검색으로는
“`~/dev/ecosystem/llm-mistral`이 하는 방식에서 영감을 받아
`~/dev/ecosystem/datasette-oauth`에 모킹된 HTTP 테스트를 추가해줘”가 가능하다.
저장소 클로닝으로는
“`simonw/research`를 GitHub에서 `/tmp`로 클론해 Rust를 WebAssembly로 컴파일하는 예제를 찾아줘”가 된다.

핵심 원리는 이것이다:
“유용한 트릭은 평생 단 한 번만 알아내면 된다.
그 트릭이 동작하는 코드 예제와 함께 어딘가 문서화되어 있으면
에이전트가 그것을 참조해 비슷한 모양의 미래 프로젝트를 풀 수 있다.”

### AI는 더 나은 코드를 만들도록 도와야 한다

AI 에이전트를 쓴다고 코드 품질이 낮아질 이유는 없다.
오히려 품질 기준을 유지하거나 높이는 선택이 가능해진다.
기술 부채의 흔한 원천은 “단순하지만 시간이 걸리는” 일들이다:
수십 곳에 걸친 API 설계 변경, “teams” vs “groups” 같은 명명 불일치,
중복 기능의 통합, 비대한 파일의 모듈화.
“기술 부채에 대한 최선의 대응은 애초에 떠안지 않는 것”이고,
이런 개선의 비용이 너무 낮아져서 이제 사소한 코드 냄새에 무관용 태도를 취할 수 있다.

리팩터링은 에이전트의 이상적 용례다.
Gemini Jules, OpenAI Codex, Claude Code 같은 비동기 에이전트를
백그라운드 브랜치에서 돌려 워크플로를 끊지 않고,
결과를 Pull Request로 평가해 수용·반복·폐기한다.
LLM은 여러 해법을 탐색하는 데도 쓸모가 있다:
명백한 해법을 놓치지 않게 돕고,
학습 데이터에 많은 “지루한 기술(boring technology)”을 제안하며,
저렴한 탐색적 프로토타이핑으로 여러 접근(예: 활동 피드에 Redis가 적합한지)을 동시에 시험한다.
Every의 컴파운드 엔지니어링(compound engineering) 방법론을 빌려,
프로젝트마다 회고하고 성공한 접근을 문서화해 다음 에이전트 실행에 쓰며 코드베이스를 점진적으로 끌어올린다.

### 안티패턴: 피해야 할 것

가장 흔하고 짜증나는 안티패턴은 “검토하지 않은 코드를 협업자에게 떠넘기는 것”이다.
직접 검토하지 않은 수백·수천 줄의 에이전트 생성 코드로 PR을 올리는 것은
가치를 제공하는 게 아니라 일을 다른 개발자에게 위임하는 것이다.
Willison의 못 박는 문장: “당신의 일은 동작하는 코드를 전달하는 것이다.”

좋은 에이전틱 엔지니어링 PR은 네 가지 특징을 갖춘다.
첫째, **동작하는 코드**—개발자가 코드가 동작한다고 확신해야 한다.
둘째, **적절한 범위**—효율적으로 검토할 만큼 작아야 하며, 큰 PR 하나보다 작은 PR 여럿이 낫다.
셋째, **맥락 설명**—상위 목표, 관련 이슈, 명세 정보를 담는다.
넷째, **검증된 설명**—에이전트가 생성한 PR 설명도 직접 검토해야 하며,
읽지 않은 텍스트를 그대로 제출하는 것은 리뷰어에 대한 무례다.

성실히 검토했음을 증명하는 방법으로 Willison은
수행한 수동 테스트에 대한 메모, 특정 구현 결정을 설명하는 주석,
동작을 보여주는 스크린샷이나 영상을 권한다.

## 2부. 코딩 에이전트와 일하기 (Working with coding agents)

### 코딩 에이전트는 어떻게 동작하는가

코딩 에이전트는 “LLM을 위한 하네스(harness)로,
보이지 않는 프롬프트와 호출 가능한 도구로 LLM을 확장하는 소프트웨어”다.
LLM은 텍스트 시퀀스를 완성하며(GPT-5.4, Claude Opus 4.6, Gemini 3.1 Pro, Qwen3.5-35B-A3B 등),
단어가 아니라 정수 토큰으로 작업한다
(“the cat sat on the ”는 `[3086, 9059, 10139, 402, 290, 220]`이 된다).
이것이 중요한 이유는 제공자가 토큰 수로 과금하고 처리 한계도 토큰으로 정해지기 때문이다.
현대 모델은 텍스트 외에 이미지 같은 입력도 받는데,
비전 LLM(vLLM)은 이미지를 토큰으로 변환해 텍스트와 동일하게 처리한다.
입력은 프롬프트, 출력은 컴플리션·응답이라 부른다.

채팅 템플릿 프롬프트가 완성형 프롬프트를 대체했고,
LLM은 무상태(stateless)라 매번 백지에서 시작하므로
새 입력마다 대화 전체를 재생해야 하고 토큰 비용이 누적된다.
토큰 캐싱이 짧은 시간 내 반복 토큰에 할인을 줘 이 비용을 상쇄한다.
도구는 하네스가 제공하는 함수로,
모델이 `<tool>get_weather(“San Francisco”)</tool>` 같은 텍스트로 실행을 요청하면
하네스가 추출·실행해 결과를 다음 프롬프트로 돌려준다.
Bash 명령 실행과 Python 코드 실행이 특히 강력한 도구다.
시스템 프롬프트는 수백 줄에 달하는 숨겨진 초기 지시로,
에이전트의 행동과 사용 가능한 도구를 정의한다(OpenAI Codex의 시스템 프롬프트가 구체적 예다).
2025년에 등장한 추론(reasoning)은 응답 전 내부 사고를 생성하게 해 디버깅에 특히 유용하며,
많은 에이전트가 추론 강도(effort)를 조절할 수 있게 한다.
전체 아키텍처는 “LLM + 시스템 프롬프트 + 도구를 루프 안에서” 돌리는 것이며,
기본 구현은 기존 LLM API 위에 수십 줄이면 되지만 프로덕션 품질의 도구 루프는 훨씬 많은 작업이 든다.

### 코딩 에이전트와 Git 쓰기

버전 관리는 코드가 시간에 따라 어떻게 변하는지 기록하고 실수를 조사·되돌리게 한다.
저장소(변경을 추적하는 폴더), 커밋(메시지·작성자가 붙은 타임스탬프 묶음),
브랜치(병합 전 독립 실험), 클론(전체 이력을 포함한 복제),
리모트(공개된 저장소, GitHub가 대표적)의 개념을 정의한 뒤,
에이전트에게 줄 수 있는 프롬프트를 단계별로 제시한다.
기본 연산은 “Git 새 저장소 시작해줘”, “이 변경 커밋해줘”,
“username/repo를 GitHub 리모트로 추가해줘”, “오늘 만든 변경 리뷰해줘”다.
고급 연산은 “main의 최신 변경을 통합해줘” 또는 “통합 옵션을 논의해줘”,
“이 git 엉망 좀 정리해줘”, “…하는 내 코드를 찾아서 복구해줘”,
“git bisect로 이 버그가 언제 들어왔는지 찾아줘”다.

가이드는 Git 이력을 불변의 기록이 아니라 “의도적으로 저술된 이야기”로 재구성한다.
그래서 커밋을 되돌리거나 다시 쓰고, 특정 파일을 커밋에서 제거하고,
여러 커밋을 더 나은 메시지로 합치고,
큰 저장소에서 이력을 유지한 채 코드를 추출하는 일까지 에이전트에 맡길 수 있다.
에이전트가 Git의 복잡한 연산을 잘하므로 암기 부담 없이 야심차게 쓰라고 권한다.

### 서브에이전트 (Subagents)

서브에이전트는 LLM 컨텍스트 한계를 관리하는 패턴이다.
새 컨텍스트 윈도로 에이전트의 새 사본을 보내 특정 작업을 처리하게 해
부모 에이전트의 귀중한 토큰 예산을 아낀다.
LLM은 약 100만 토큰 한계 안에서 작동하지만 벤치마크는 20만 토큰 아래에서
품질이 더 좋게 나오는 경우가 잦고,
지난 2년간 LLM 능력은 급격히 늘었어도 컨텍스트 한계는 거의 정체였다.

Claude Code의 Explore 서브에이전트가 실용적 예다.
작업에 착수하기 전 저장소 구조를 파악하는데, 과정은 이렇다:
부모 에이전트가 전용 프롬프트를 구성하고,
새 컨텍스트 윈도로 서브에이전트를 디스패치하며,
diff와 관련된 템플릿·Python·JavaScript·CSS를 집중 탐색하고,
발견 사항의 종합 요약을 부모에게 돌려준다.

서브에이전트는 세 유형으로 나뉜다.
순차 서브에이전트는 부모가 멈추고 기다리는 가장 단순한 형태다.
병렬 서브에이전트는 여러 개를 동시에 돌리며 Claude Haiku 같은 빠르고 저렴한 모델로 속도를 높일 수 있어,
상호 의존성이 없는 다중 파일 편집에 유용하다.
전문가 서브에이전트는 시스템 프롬프트나 도구를 맞춘 것으로,
코드 리뷰어(버그·설계 약점 식별), 테스트 러너(실패 격리, 장황한 출력 숨김),
디버거(재현 단계 격리, 근본 원인 규명)가 대표적이다.
원칙은 “루트 에이전트도 토큰만 충분하면 자기 출력을 디버깅·리뷰할 수 있다”는 것이다.
서브에이전트는 OpenAI Codex, Claude, Gemini CLI, Mistral Vibe, OpenCode, VS Code, Cursor에서 지원된다.

## 3부. 테스트와 QA (Testing and QA)

### Red/Green TDD

“Use red/green TDD.” 이 네 단어 프롬프트가 코딩 에이전트의 결과물을 극적으로 개선한다.
가장 엄격한 형태의 테스트 우선 개발은 세 단계다:
테스트를 먼저 작성하고, 테스트가 실패함을 확인하고(Red),
테스트를 통과시키는 구현을 작성한다(Green).

코딩 에이전트의 대표적 위험 두 가지—동작하지 않는 코드를 생성하는 것과
아무도 쓰지 않는 불필요한 코드를 만드는 것—을 테스트 우선 개발이 동시에 방어한다.
테스트가 구현의 정확성을 검증하고, 테스트에 정의된 것만 구현하도록 범위를 제한하기 때문이다.
프로젝트가 커질수록 새 변경이 기존 기능을 깨뜨릴 확률도 높아지는데,
포괄적인 테스트 스위트가 이를 막는 가장 효과적인 수단이다.

Red 단계가 핵심이다.
테스트를 쓴 뒤 실패 확인을 건너뛰면 이미 통과하는 무의미한 테스트를 만들 위험이 있고,
그런 테스트는 새 구현을 실제로 검증하지 못한 채 거짓 안전감만 준다.
에이전트는 테스트와 구현을 동시에 생성할 수 있어 이 위험이 사람보다 훨씬 높으므로,
Red 단계는 에이전트에게 더욱 중요하다.

“red/green TDD”는 그 자체로 프롬프트 엔지니어링 패턴이다.
모델에 이미 내장된 엔지니어링 규율을 네 단어로 활성화한다.
모든 좋은 모델이 이를 “테스트를 먼저 쓰고, 실패를 확인한 뒤, 통과시키는 변경을 구현하라”로 이해한다.
Willison의 전형적인 Python 세션 프롬프트는 다음과 같다:

```text
Run the existing tests with "uv run pytest".
Build using red/green TDD.
```

테스트 실행 방법을 알려주는 것 자체가 “이 프로젝트에서 테스트는 중요하다”는 신호로 작동한다.
에이전트는 기존 테스트를 읽은 뒤 자체 테스트를 작성하므로,
좋은 패턴의 기존 테스트가 있으면 에이전트도 좋은 테스트를 쓸 가능성이 높아진다.
프로젝트의 `CLAUDE.md`(또는 시스템 프롬프트)에 “Build using red/green TDD.” 한 줄을 넣는 것만으로 효과를 볼 수 있다.
주목할 점은 Willison 본인이 커리어 내내 테스트 우선 개발에 회의적이었다가
에이전트가 필요한 코드만 작성하도록 강제하는 가장 효과적인 수단임을 깨닫고 전향했다는 사실이다—
도구의 변화가 방법론에 대한 판단까지 바꾼 사례다.

### 먼저 테스트를 실행하라 (First run the tests)

자동화된 테스트는 코딩 에이전트와 일할 때 사실상 필수가 됐다.
과거엔 빠른 개발 중 시간 낭비로 치부됐지만, 이제 에이전트가 테스트를 빠르게 생성·유지할 수 있다.
새 에이전트 세션은 “First run the tests”(파이썬은 “Run 'uv run pytest'”)로 시작한다.
이 짧은 프롬프트는 세 가지를 한다:
테스트 스위트의 존재를 알리고 실행법을 알아내게 해 이후에도 테스트를 돌릴 확률을 높이고,
테스트 개수로 프로젝트의 복잡도·규모 정보를 주며,
테스트 지향 사고방식을 심어 커버리지 확장을 유도한다.
에이전트는 본래 테스트하려는 성향이 있어서,
기존 테스트 스위트가 있으면 변경이 리그레션을 일으키지 않는지 검증하려는 경향이 강화되고,
코드베이스를 학습할 때도 기존 테스트를 참조한다.
이 패턴은 Red/Green TDD와 짝을 이루며, 모델에 이미 내장된 엔지니어링 규율을 짧은 프롬프트로 활성화한다.

### 에이전틱 수동 테스트 (Agentic manual testing)

코딩 에이전트의 결정적 특징은 자기가 쓴 코드를 실행할 수 있다는 것이다.
“LLM이 생성한 코드는 실행되기 전까지 동작한다고 절대 가정하지 마라.”
자동화된 테스트만으로는 충분치 않다—테스트를 통과해도 의도대로 동작한다는 뜻은 아니며,
에이전트의 수동 테스트는 서버 크래시나 사라진 UI 요소처럼 자동 테스트가 놓치는 문제를 드러낸다.

언어·프레임워크별 접근이 다르다.
Python 라이브러리는 `python -c “... 코드 ...”`로 인터프리터에 코드를 직접 넘겨
영구 파일을 만들지 않고 엣지 케이스를 시험한다.
일반적으로는 `/tmp`에 데모 코드를 써서 실수로 저장소에 커밋되는 것을 막는다.
JSON을 다루는 웹 API는 `curl`로 엔드포인트를 두드리며, 에이전트에 “탐색(explore)”을 지시해 커버리지를 넓힌다.

브라우저 자동화에는 여러 도구가 있다.
Playwright(Microsoft)는 다중 언어 바인딩을 가진 오픈소스 라이브러리로
Chrome·Firefox·Safari를 자동화하고 에이전트용 `playwright-cli`를 제공한다.
agent-browser(Vercel)는 코딩 에이전트 전용으로 설계된 CLI 래퍼다.
Rodney(Willison의 프로젝트)는 Chrome DevTools Protocol을 쓰며
JavaScript 실행, 스크롤, 클릭, 타이핑, 접근성 트리 읽기를 지원한다
(예: “개발 서버를 띄우고 `uvx rodney --help`로 새 홈페이지를 테스트해”).
Showboat는 수동 테스트 워크플로를 문서로 남기는 도구로,
`note`(마크다운 추가), `exec`(명령과 출력 기록—거짓 보고 방지), `image`(스크린샷) 명령을 갖는다.
수동 테스트로 문제를 찾으면 red/green TDD로 영구 자동 테스트 커버리지를 추가한다.

## 4부. 코드 이해하기 (Understanding code)

### 선형 워크스루 (Linear walkthroughs)

“프런티어 모델은 올바른 하네스와 함께라면 코드 동작을 이해시키는 상세한 워크스루를 만들 수 있다.”
Willison은 Claude Code와 Opus 4.6으로 “바이브 코딩”한 SwiftUI 슬라이드 발표 앱 Present를 예로 든다.
2025년 11월부터 2026년 2월까지 이어진 프런티어 모델 발전에 관한 강연용으로 만든 앱인데,
발표 메커니즘 자체가 청중에게 STAR 순간(Something They’ll Always Remember)으로 작동했다.
코드는 GitHub에 올렸지만 정작 그는 작동 방식을 이해하지 못했다.

그래서 Claude Code에 소스를 읽고 선형 워크스루를 계획한 뒤
Showboat로 상세한 설명 문서를 만들라고 시켰다.
Showboat의 `note`는 마크다운 해설을 더하고, `exec`는 셸 명령을 실행해 명령과 출력을 함께 담는다.
환각을 막으려 `sed`·`grep`·`cat`으로 실제 코드 스니펫을 포함하도록 지시했다.
생성된 문서는 여섯 개 Swift 파일을 모두 명확하고 실행 가능한 설명으로 다뤘고,
그는 이 문서를 읽으며 SwiftUI 아키텍처와 Swift 언어 세부를 상당히 배웠다.
이 패턴은 “LLM이 학습을 줄인다”는 우려에 대한 반례로,
짧은 프로젝트도 구조화된 에이전트 설명을 통해 학습 기회가 된다.

### 상호작용 설명 (Interactive explanations)

인지 부채(cognitive debt)는 에이전트가 만든 코드를 충분히 이해하지 못하는 부담이다.
코드가 블랙박스가 되면 애플리케이션을 자신 있게 추론하기 어려워져 기술 부채처럼 진행을 늦춘다.
상호작용 설명은 시각적·인터랙티브한 수단으로 이해를 높여 인지 부채를 줄이는 한 방법이다.

워드 클라우드 알고리즘 사례가 이를 보여준다.
Max Woolf의 글에서 출발해 LLM이 Rust를 어떻게 다루는지 보려고
Claude Code로 비동기 연구 프로젝트를 돌려 워드 클라우드 시각화 도구를 만들었다.
초기 문서는 알고리즘이 “아르키메데스 나선 배치에 단어별 무작위 각도 오프셋”을 쓴다고 했지만,
이 기술적 서술은 직관적 명료함이 없었다.
선형 워크스루는 Rust 구조를 이해시켰으나 알고리즘의 작동 메커니즘까지는 아니었다.
그래서 Claude에게 단어 배치를 애니메이션으로 보여주는 인터랙티브 HTML 페이지를 만들게 했다:
URL 프래그먼트로 입력을 보존하는 텍스트 입력, 애니메이션 속도 슬라이더,
프레임 단위 스텝 제어, 임의 단계에서의 PNG 내보내기를 갖춘 페이지였다.
애니메이션은 알고리즘이 배치 위치를 시도하고 기존 단어와 교차를 검사하며
충돌 시 바깥으로 나선을 그리는 과정을 직관적으로 드러냈다.
좋은 코딩 에이전트는 이렇게 알고리즘 개념을 설명하는 애니메이션과 인터페이스를 즉석에서 만들어낼 수 있다.

## 5부. 주석 달린 프롬프트 (Annotated prompts)

### WebAssembly와 Gifsicle을 쓴 GIF 최적화 도구

Gifsicle(약 30년 된 C 명령행 압축 유틸리티)을 Emscripten으로 WebAssembly(233KB 번들)에 컴파일해
브라우저에서 드래그앤드롭으로 GIF를 압축하고 바로 내려받는 도구를 만들었다.
WebAssembly는 Emscripten 툴체인을 통해 컴파일된 C 코드(Gifsicle)를 브라우저 샌드박스에서 안전하게 실행시키고,
Gifsicle은 변하지 않는 프레임 영역을 식별하며 손실 압축·색 팔레트 축소·스케일링을 지원한다.

프롬프트는 모듈식이었다:
파일명 지정(`gif-optimizer.html`)으로 대상 파일을 정하고,
Gifsicle을 WebAssembly로 컴파일해 드래그앤드롭 인터페이스를 만들며,
파일 크기와 다운로드 버튼이 달린 다중 최적화 프리뷰를 띄우고,
프리셋과 조절 가능한 파라미터를 잇는 “tweak” 링크로 수동 설정을 더하고,
Rodney로 특정 테스트 GIF URL을 검증한다.
후속 프롬프트로 Gifsicle 전체 소스를 커밋하는 대신 빌드 스크립트와 패치를 `lib/`에 두고,
GitHub Pages 배포를 위해 컴파일된 233KB WASM 번들을 포함하며,
원작자 Eddie Kohler에게 GPL v2 라이선스로 출처를 표기했다.
교훈: 확립된 도구를 이름으로 참조하면 에이전트가 기존 코드베이스를 찾아 쓰고,
즉각적 피드백(Rodney)으로 개발 중 버그를 잡으며,
초기 진행을 보며 후속 프롬프트를 대기시켜 방향을 교정하고,
같은 저장소의 기존 도구에서 UX 관행을 명시적 지정 없이 가져온다.

### 블로그-뉴스레터 도구에 새 콘텐츠 타입 추가하기

블로그 콘텐츠를 Substack 배포용 HTML 뉴스레터로 바꾸는 도구에
“beats”라는 새 콘텐츠 타입(프로젝트 릴리스, 도구, 박물관 방문 같은 외부 콘텐츠)을 넣었다.
프롬프트는 단 세 부분이었다:
“`simonw/simonwillisonblog`를 GitHub에서 `/tmp`로 클론해 참조해”,
“블로그의 Atom everything 피드가 동작하는 방식과 비슷하게,
설명이 있는 beats를 포함하도록 `blog-to-newsletter.html`을 업데이트해”,
“`python -m http.server`로 띄우고 `uvx rodney --help`로 테스트해—
뉴스레터에 뜨는 것과 홈페이지에 있는 것을 비교해”.
에이전트는 `note` 컬럼에 내용이 있고 `is_draft = 0`인 beats를 거르는 SQL UNION 절과,
Django 모델 검사에서 유추한 beat 타입→표시명 매핑을 만들었다.
교훈: 참조 코드를 직접 살피게 하고(상세 설명 대신 소스 검사를 시키기),
비교 예시(Atom 피드 로직)로 요구사항을 효율적으로 전달하며,
내장 검증(로컬 서버 + 라이브 비교)으로 짧은 프롬프트를 사람 검토 없이도 완결되는 해법으로 바꾼다.

## 6부. 부록 — 내가 쓰는 프롬프트 (Prompts)

Willison이 반복적으로 쓰는 프롬프트 모음이다.

### Artifacts 커스텀 지시

HTML/JavaScript 도구를 React 없이 만들기 위한 지시다.
React 절대 금지, 순수 HTML·바닐라 JavaScript·CSS만 사용한다.
CSS는 `* { box-sizing: border-box; }`로 시작하는 2칸 들여쓰기,
입력·텍스트영역은 16px, 폰트는 Helvetica,
JavaScript는 2칸 들여쓰기에 모듈 스크립트 타입, 제목은 문장형(sentence case)을 규칙으로 둔다.

### 교정 (Proofreader) 프롬프트

발행 전 블로그 글을 검토하는 다섯 가지 기능을 한다:
오타·문법 오류 식별, 반복되는 표현 지적, 논리·사실 오류 포착,
강화가 필요한 약한 주장 강조, 빈·깨진 링크 점검.

### 대체 텍스트 (Alt text) 프롬프트

이미지의 접근성 설명을 작성한다.
복사하기 쉽게 대체 텍스트를 코드 블록으로 한 줄 형식으로 제시하고,
보이는 텍스트를 그대로 포함하며 이미지 성격을 먼저 기술한다.
Willison은 이 작업에 Claude Opus가 특히 뛰어나다고 말한다.

### 팟캐스트 하이라이트 프롬프트

팟캐스트 트랜스크립트에서 인상적인 인용을 뽑는다.
“전체 주제를 가장 잘 보여주는 인용과 놀라운 아이디어를 소개하는 인용”을 찾되,
명료하고 흥미롭고 기억에 남는 표현을 우선한다.

## 분석

### 가이드의 진짜 주제는 “에이전트”가 아니라 “검증 책임의 이동”이다

표면적으로 이 가이드는 코딩 에이전트 사용법을 가르친다.
그러나 17개 챕터를 관통하는 단 하나의 논리는 “코드 생산은 싸졌고 검증은 비싸졌다”이다.
1부의 “코드는 공짜다”가 전제를 깔고,
3부 전체(TDD, 먼저 테스트, 수동 테스트)와 안티패턴 챕터가 그 전제의 필연적 귀결을 다룬다.
Willison이 반복하는 “실행되기 전까지 동작한다고 가정하지 마라”는 문장은
생산이 아니라 검증이 새로운 병목임을 선언하는 것이다.

이 구조가 중요한 이유는, 대부분의 AI 코딩 담론이 여전히 “얼마나 빨리 만드느냐”에 머물기 때문이다.
Willison은 그 질문을 이미 풀린 것으로 치고 다음 질문—“어떻게 그게 맞다고 확신하느냐”—로 넘어간다.
가이드의 절반 가까이가 테스트·검증·이해에 배정된 것은 우연이 아니라 의도된 무게중심이다.
[병목은 결코 코드가 아니었다](bottleneck-was-never-the-code.md)가 같은 결론에
다른 경로로 도달한다 — 코드 생산 비용이 0으로 수렴하면 진짜 병목인
명세와 검증이 드러난다는 진단이다.
두 글을 겹쳐 읽으면 Willison의 17개 챕터가 “코드는 공짜”라는 거시 진단의
미시 실천 매뉴얼임이 또렷해진다.

### “비축(hoard)”은 RAG의 인간 버전이다

Willison의 비축 패턴은 본질적으로 검색 증강 생성(RAG)을 사람이 손으로 운영하는 것이다.
동작하는 예제를 블로그·TIL·GitHub에 쌓아두고,
에이전트에게 “클론해서 참조해”, “이 프로젝트에서 영감을 받아”라고 시킨다.
이것은 벡터 DB 대신 파일시스템과 git URL을 인덱스로 쓰는 RAG다.
“유용한 트릭은 단 한 번만 알아내면 된다”는 명제는 캐싱의 인식론적 버전이다.

여기서 드러나는 가정은 검색 가능성(retrievability)이 곧 재사용성이라는 것이다.
Willison처럼 1,000개 저장소와 정돈된 도구 모음을 가진 사람에게는 성립한다.
그러나 이 패턴의 가치는 비축의 양이 아니라 정돈의 질에 달려 있는데,
가이드는 그 정돈 비용을 거의 다루지 않는다.

### 에이전트와의 실제 팀 작업에서 드러나는 마찰

HN 댓글에서 mohsen1은 에이전트 사용의 핵심 원리로 “테스트 하네스가 전부”라는 표현을 쓴다.[^mohsen1]
테스트 가능한 소프트웨어는 에이전틱 루프에 완벽하게 맞고,
에이전트를 마이크로매니지하지 않고 반복하며 실험하게 두는 것이 더 나은 결과를 낸다고 강조한다.
문서화가 세션 간 학습을 돕는다는 관찰도 덧붙인다.

winwang은 선형 워크스루를 실행할 때 번호 매긴 트리 구조를 요청하는 방식을 소개한다.[^winwang]
트리 크기로 세분화 수준을 조절하고, 번호 매김 덕분에 논의 시 참조가 단순해진다.
엄격한 타입 분석, 코드 리뷰에 다른 모델을 쓰는 것도 실용적 보완책으로 제시한다.

AlexCalderAI는 병렬 에이전트의 오케스트레이션 상태 관리에 파일 기반 접근법을 제안한다.[^alexcalderai]
단일 `TODO.md`에 “DO NOW”, “BLOCKED”, “DONE” 섹션을 두고 에이전트 유형별 출력 파일을 분리하는 방식이다.
“파일 기반 상태는 결정론적”이어서 충돌 복구 시에도 동일한 입력이 일관된 출력을 낸다고 설명한다.

### 모든 패턴이 “에이전트가 자기 일을 증명하게 만든다”로 수렴한다

Showboat의 `exec`가 “거짓 보고를 막는다”는 설명,
선형 워크스루에서 `cat`·`grep`으로 코드를 포함시켜 “환각을 막는다”는 지시,
수동 테스트의 스크린샷 증거, 안티패턴 챕터의 “검토 증명” 요구가 모두 같은 형태다.
에이전트의 주장(claim)과 실행 증거(evidence)를 분리하고, 증거를 강제하는 것이다.

이는 LLM의 근본 약점—그럴듯하지만 틀린 출력—에 대한 일관된 방어선이다.
Willison은 에이전트를 신뢰할 수 없는 보고자로 취급하고,
모든 패턴을 “실행 가능한 증거를 남기게 하는 장치”로 설계한다.
가이드가 흩어진 팁의 모음이 아니라 하나의 인식론을 가진 이유가 여기 있다.

### 프런티어 모델 의존성이 명세에 새겨져 있다

가이드는 GPT-5.4, Claude Opus 4.6, Gemini 3.1 Pro 같은 구체적 모델명과
233KB WASM 번들, 20만/100만 토큰 같은 수치를 거리낌 없이 쓴다.
“프런티어 모델은 올바른 하네스와 함께라면…”이라는 조건절이 핵심 패턴마다 붙는다.
즉 이 패턴들은 모델 능력의 특정 구간을 전제로 한다.

Willison 스스로 이를 “동작함이 입증되었고 도구가 발전해도 유효할 패턴”이라 표현한다.
이 두 조건—현재 동작과 미래 유효—은 사실 긴장 관계다.
패턴이 특정 모델 세대의 능력에 의존할수록 미래 유효성은 약해진다.
가이드가 “살아있는 문서”임을 강조하는 것은 이 긴장에 대한 정직한 인정이다.

## 비평

### “코드는 공짜”라는 전제가 검증 비용을 가린다

가이드의 토대인 “코드 작성은 거의 공짜”라는 명제는 절반만 맞다.
정확히는 “코드 타이핑이 공짜”가 되었을 뿐, 코드를 맞다고 확신하는 비용은 그대로거나 오히려 늘었다.
Willison도 이를 알기에 가이드의 절반을 테스트에 쓴다.
그러나 그는 이 비용을 “이제 테스트도 에이전트가 싸게 써준다”로 다시 무마한다.
여기에 순환 논리가 있다: 검증을 위해 에이전트가 쓴 테스트를, 누가 검증하는가.

수동 테스트와 PR 검토를 강조하는 챕터들이 사실상 이 질문에 답하려 하지만,
그 답은 “사람이 끝까지 책임진다”로 귀결된다.
그렇다면 절약된 것은 타이핑 시간이지 인지 노동이 아니다.
가이드는 이 재배분을 “비용 하락”으로 일관되게 프레이밍하는데,
실제로는 비용의 이동(생산→검증)에 가깝다.
이 차이를 흐리면 “에이전트로 더 야심찬 일을 하라”는 결론이 위험해진다—
야심의 규모만큼 검증 부채가 쌓이기 때문이다.

HN에서 yieldcrv는 TDD에 대한 회의론을 제기한다.[^yieldcrv]
실패한 테스트가 에이전틱 코딩 도구에게 “잘못된 가정을 했구나”라는 유의미한 신호를 주지 않는다는 것이다.
타입 에러나 린터보다 더 나은 피드백을 주지 않는다는 관찰이다.
jihadjihad도 유사한 우려를 제기하며, 에이전트가 테스트를 통과시키기 위해 테스트 자체에 값을 하드코딩할 수 있다는 위험을 지적한다.[^jihadjihad]
이 두 비판은 가이드의 TDD 챕터가 다루지 않는 실제 에이전트의 패턴에서 나온다.
Red 단계 확인이 에이전트에게 더 중요하다는 가이드의 주장은 이 위험을 인식한 것이지만,
그 확인을 에이전트 스스로 수행한다는 점에서 한계가 여전히 남는다.
[AI 시대의 TDD](tdd-in-ai-era.md)는 같은 위험을 더 정밀하게 분해한다 — Red는
인간의 명세 작업이고 Green만 AI에게 위임 가능한 단계라는 역할 분배는
“에이전트가 자기 테스트를 검증한다”는 순환을 끊기 위한 처방이다.

### 패턴들은 Willison의 특수한 작업 환경에서만 매끄럽다

가이드의 거의 모든 예제는 단일 개발자가 통제하는 작은 도구다:
GIF 최적화 HTML, 슬라이드 앱, 개인 블로그-뉴스레터 변환기, OCR 단일 페이지.
이 맥락에서는 “검토하지 않은 코드를 떠넘기지 마라”가 깔끔하게 작동한다—
검토자도 작성자도 Willison 자신이기 때문이다.

그러나 수백 명이 일하는 코드베이스, 규제 산업, 레거시 시스템에서는
“작은 PR 여럿”, “git 엉망 정리해줘”, “Atom 피드처럼 만들어줘” 같은 처방이 그대로 통하지 않는다.
참조할 깨끗한 예제가 없거나, 도메인 지식이 코드에 없거나,
“동작 확인”이 단위 테스트로 끝나지 않는 경우가 대부분이다.
가이드는 이 일반화 격차를 인정하지 않고 개인 도구의 경험을 보편 패턴으로 제시한다.
비축 패턴이 특히 그렇다—Willison만큼 비축한 사람은 극소수다.

HN에서 slaye는 팀 환경의 구체적 마찰을 기술한다.[^slaye]
동료들이 아키텍처 의미를 이해하지 못한 채 “바이브 코딩”하고, “작동하는데 왜 바꾸냐”는 반응으로 피드백을 거부한다.
점점 커지는 PR 압박 속에서 기준을 타협해야 하는 상황을 토로한다.
이 보고는 가이드의 처방이 팀 역학에서 어떻게 작동하지 않는지를 실제 사례로 보여준다.

gaigalas는 기타 제조 비유를 들어 더 근본적인 재구성이 필요하다고 주장한다.[^gaigalas]
공장식 기타 제조를 가능케 한 것은 장인 방식의 자동화가 아니라, Leo Fender처럼 제품 자체를 공장 생산에 맞게 재설계한 것이었다.
에이전틱 엔지니어링도 기존 소프트웨어 개발 관행을 자동화하는 것으로는 부족하고, 그 관행 자체를 에이전트 중심으로 재구성하는 혁신가가 아직 등장하지 않았다는 것이다.

storus는 LLM 세대 교체 속도가 패턴의 수명을 위협한다고 지적한다.[^storus]
LangChain은 작은 컨텍스트와 약한 모델의 한계를 우회하려고 복잡한 아키텍처를 만들었지만, GPT-3.5가 나오자 무용해졌다.
컨텍스트 크기가 계속 커지고 인컨텍스트 압축이 가능해지면 서브에이전트 패턴 자체가 불필요해질 수 있다는 전망이다.

### 인지 부채를 진단하면서 그 해법이 인지 부채를 만든다

“상호작용 설명” 챕터는 에이전트가 만든 코드를 이해 못 하는 인지 부채를 정면으로 다룬다.
훌륭한 통찰이다.
그러나 해법으로 제시된 것—에이전트에게 애니메이션 설명 페이지를 만들게 하기—은
또 다른 에이전트 생성물이다.
알고리즘을 이해하기 위해 에이전트가 만든 설명을 신뢰하는 셈인데,
그 설명이 정확한지는 누가 보장하는가.

선형 워크스루도 마찬가지다.
Willison은 `cat`·`grep`으로 실제 코드를 포함시켜 환각을 막는다고 했지만,
그 코드에 대한 에이전트의 해설(narration)은 여전히 검증되지 않은 LLM 출력이다.
즉 코드 인용은 사실이어도 설명은 그럴듯한 추측일 수 있다.
가이드는 인지 부채를 “이해 부족”으로 정의하면서,
그 이해를 또 다른 불투명한 생성물로 메우는 자기 참조적 구조를 충분히 경계하지 않는다.

### 부록의 프롬프트는 패턴이 아니라 개인 설정이다

부록 “내가 쓰는 프롬프트”는 가이드에서 가장 구체적이지만 가장 일반화하기 어렵다.
“React 쓰지 마라”, “Helvetica 써라”, “16px로 해라”는 패턴이 아니라 Willison의 취향이다.
이것을 패턴 가이드에 넣은 것은 두 가지를 흐린다:
무엇이 옮길 수 있는 원리이고 무엇이 한 사람의 환경 설정인가.

진짜 옮길 수 있는 교훈은 “재사용 가능한 시스템 프롬프트를 비축하라”는 메타 원리인데,
부록은 그 메타 원리 대신 자신의 구체적 설정을 나열한다.
독자는 이 프롬프트들을 복사하고 싶은 유혹을 받지만,
정작 배워야 할 것은 “왜 이런 프롬프트가 필요한가”를 자기 맥락에서 도출하는 능력이다.
가이드가 원리와 예시를 더 분명히 분리했다면 부록의 교육 가치가 훨씬 컸을 것이다.

## 인사이트

### 검증이 병목이 되면 “비축”의 진짜 자산은 코드가 아니라 테스트다

가이드는 동작하는 코드 예제를 비축하라고 한다.
그러나 검증이 새 병목이라는 가이드 자신의 논리를 끝까지 밀면,
비축해야 할 더 귀한 자산은 코드가 아니라 검증 장치다.
재사용 가능한 테스트 하니스, 골든 데이터셋, 평가 스크립트, 수동 테스트 시나리오야말로
에이전트 시대에 복리로 불어나는 자본이다.

이것이 가이드가 예견하지 못한 2차 효과다.
코드 생산이 공짜가 되면 코드의 한계 가치는 0으로 수렴하지만,
“이 코드가 맞다”를 빠르게 증명하는 능력의 가치는 오히려 오른다.
[에이전틱 엔지니어링 9가지 생존 기술](agentic-engineering-9-survival-skills.md)이
꼽는 “완료 정의(Definition of Done)”와 “관찰 가능성”도 이 자본을
개인 단위에서 적립하는 방법으로 읽을 수 있다.
시간이 지나면 조직의 경쟁력은 코드베이스의 크기가 아니라
검증 인프라의 성숙도로 측정될 것이다.
Showboat 같은 “증거를 남기는 도구”가 가이드에서 차지하는 비중은
이 전환의 초기 신호로 읽어야 한다.

### Showboat는 에이전트 시대의 “감사 추적(audit trail)”이다

Showboat의 `exec`가 거짓 보고를 막는다는 작은 디테일은 가이드에서 과소평가된 핵심이다.
이것은 단순한 문서화 도구가 아니라, 에이전트의 행동에 대한 감사 추적이다.
금융·의료·항공이 신뢰할 수 없는 행위자를 다루기 위해 감사 로그를 발명했듯,
신뢰할 수 없는 LLM을 프로덕션에 들이려면 같은 인프라가 필요하다.

역사적 유비가 분명하다.
복식부기가 상거래의 신뢰 문제를 풀어 자본주의를 가능케 했고,
버전 관리가 협업 코딩의 책임 추적을 풀었다.
에이전틱 엔지니어링은 “기계가 한 일을 기계가 보고하는” 새로운 신뢰 문제를 만들었고,
그 해법은 결국 검증 가능한 실행 기록이다.
지금은 Showboat 같은 개별 도구로 흩어져 있지만,
머지않아 에이전트 실행의 표준 감사 포맷이 등장할 것이고,
규제 산업에서는 그것이 의무가 될 가능성이 높다.

### “코드는 공짜”는 추상화의 가치를 역전시킨다

가이드가 글로싱한 트레이드오프 하나가 장기적으로 가장 크게 작동한다.
지난 50년간 소프트웨어 공학의 핵심 미덕은 DRY—중복 제거와 추상화였다.
코드 작성이 비쌌기 때문에 한 번 쓰고 재사용하는 것이 합리적이었다.
코드가 공짜가 되면 이 계산이 흔들린다.

추상화는 공짜가 아니다—이해하고 유지하는 인지 비용이 든다.
생산이 공짜고 검증이 비싼 세계에서는,
영리하게 추상화된 코드 한 덩어리보다
단순하고 검증하기 쉬운 중복 코드 여러 개가 더 나을 수 있다.
Willison이 “boring technology”와 단순함을 반복 강조하는 것은 이 직관과 맞닿아 있지만,
그는 이것을 명시적 원리로 끌어올리지 않는다.
규모가 커질수록 이 긴장은 폭발한다:
에이전트가 비슷한 코드를 무한히 찍어내면 추상화 부재가 새로운 부채가 되고,
반대로 과도한 추상화는 에이전트와 사람 모두의 검증을 어렵게 한다.
“무엇을 추상화하고 무엇을 중복할 것인가”가
에이전트 시대 아키텍처 설계의 새 중심 질문이 될 것이다.

### 인지 부채는 기술 부채보다 측정 불가능해서 더 위험하다

가이드는 인지 부채라는 개념을 던지지만 그 심각성을 충분히 전개하지 않는다.
기술 부채는 빌드 시간, 버그 수, 변경 리드타임으로 측정할 수 있다.
인지 부채—팀이 자기 코드를 이해하지 못하는 정도—는 측정 지표가 사실상 없다.
측정할 수 없는 부채는 위기가 닥치기 전까지 보이지 않는다.

에이전트가 코드를 빠르게 양산할수록 인지 부채는 생산 속도에 비례해 쌓이지만,
그 누적은 장애·이직·인수인계 같은 순간에야 갑자기 드러난다.
가이드의 해법(워크스루, 상호작용 설명)은 개인 차원의 미봉책이고,
조직 차원에서 인지 부채를 가시화·관리하는 메커니즘은 아직 존재하지 않는다.
이것이 에이전틱 엔지니어링이 향후 몇 년간 풀어야 할 가장 어려운 문제다:
코드를 만든 것은 기계인데 그 코드를 이해해야 하는 책임은 여전히 사람에게 있고,
그 이해의 결손은 부채로 적립되지만 대차대조표 어디에도 기록되지 않는다.

bluemario는 완전 자율 에이전트가 테스트를 통과하면서도 암묵적 불변조건을 위반하는 코드를 만든다는 패턴을 보고한다.[^bluemario]
체크포인트 빈도의 딜레마가 핵심이다.
너무 잦으면 느린 페어 프로그래머가 되고, 너무 적으면 광범위한 디버깅이 필요해진다.
“돌이킬 수 없는 행동 직전”을 체크포인트 기준으로 제안하며,
문서 파일만으로는 아키텍처 관례를 에이전트에게 전달하기 어렵다는 실용적 한계도 제기한다.

### Lobste.rs: 토큰 생성 순간만이 유일한 개입 지점이다

Lobste.rs의 mw는 에이전트 협업과 인간 협업의 근본적 차이를 지적한다.[^mw-lobsters]
AI에게 영향을 미칠 수 있는 유일한 시점은 “토큰이 생성되는 순간”이며, 인간 동료처럼 백그라운드에서 반성하거나 재고하는 일은 일어나지 않는다는 것이다.
중복 코드를 메서드로 추출하라는 지시 같은 패턴 지침이 실제 출력 품질에 최소한의 영향만 미친다는 관찰도 덧붙인다.
TDD 워크플로우에서는 에이전트가 파일을 읽거나 검토된 diff를 적용하는 것만 허용하고 직접 실행은 막는 엄격한 가드레일을 적용한다고 공유한다.
프롬프트 지시가 준수를 “보장”하는 것이 아니라 단지 “장려”할 뿐임을 강조하며, 모델이 명시적 지시를 완전히 무시한 사례를 직접 목격했다고 밝힌다.
이 비판은 가이드가 “에이전트에게 적절한 도구를 주고 지시하라”는 프레임 전체에 대한 구조적 의문이다.
