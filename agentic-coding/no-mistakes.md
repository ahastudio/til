# no-mistakes: AI 기반 Git 푸시 게이트

<https://github.com/kunchenguid/no-mistakes>

HN 토론: <https://news.ycombinator.com/item?id=47838708> (19점, 6개 댓글)

GeekNews 토론: <https://news.hada.io/topic?id=31223>

## 소개

no-mistakes는 Git 푸시를 로컬 프록시 게이트로 가로채
AI 기반 검증 파이프라인을 실행한 뒤 원격 저장소로 보내는 워크플로 도구다.
Go로 작성됐으며(코드베이스의 99.8%), 2026년 7월 기준 GitHub 스타 5.5k, 포크 346개, 릴리스 79개를 기록하고 있다.
최신 버전은 v1.34.0이다.

프로젝트의 슬로건은 “Kill all the slop. Raise clean PR.”이다.
로컬 작업과 원격 푸시 사이에 자리잡아, 코드가 리모트에 도달하기 전에
리뷰·테스트·문서·린트·PR 생성·CI까지 자동으로 처리한다.
검증은 격리된 worktree에서 실행되므로 현재 개발 작업을 방해하지 않는다.

Claude, Codex, RovoDev, OpenCode, Pi, Copilot 등 주요 AI 코딩 에이전트를 지원하며,
`acpx`를 통해 커스텀 타겟도 연결할 수 있다.

## 진입점

도구는 세 가지 방식으로 사용할 수 있다.

- `git push no-mistakes` — Git 원격으로 등록하여 직접 게이트를 통과하는 방식
- `no-mysteries` 명령 — 검증 결과를 검토하는 대화형 TUI
- `/no-mistakes` 스킬 — 코딩 에이전트와 통합하여 헤드리스 방식으로 동작

## 파이프라인 구조

검증은 일회용 worktree에서 순차적으로 실행된다.

```text
review → test → docs → lint → push → PR → CI
```

각 단계는 독립적으로 통과하거나, 처리가 필요한 “finding”을 생성한다.
시스템은 finding을 두 종류로 구분한다.
기계적이고 안전한 수정은 자동으로 적용하고,
인간의 판단이 필요한 변경은 승인을 위해 에스컬레이션한다.
PR 생성과 브랜치 전달은 모든 검사가 통과한 후에만 이루어진다.

## 분석

### 푸시 게이트는 PR 리뷰의 마찰을 업스트림으로 이동시키는 시도다

기존 CI/CD 파이프라인은 푸시 이후 시점에서 문제를 잡는다.
PR이 열리고, 리뷰어가 배정되고, 피드백이 오가는 사이클이 길게 이어진다.
no-mistakes의 핵심 전제는 이 마찰을 개발자의 로컬 환경으로 끌어오는 것이다.
“리모트에 도달하기 전에 이미 클린 상태여야 한다”는 철학이다.

이 접근은 “shift left”라는 소프트웨어 품질 패턴의 연장선에 있다.
테스트를 가능한 한 빠른 시점으로 당겨 피드백 루프를 단축하려는 시도다.
no-mistakes는 이것을 AI 에이전트와 결합해,
사람이 직접 하나씩 체크하지 않아도 되는 자동화된 게이트로 구현했다.

HN 쓰레드에서 snehesht는 “왜 이것을 PR 리뷰 과정의 일부로 실행하지 않느냐”고 물었다.[^snehesht]
커스텀 에이전트로 PR을 스코어링해서 기준을 통과한 것만 병합하면 어떻겠냐는 제안이었다.
저자 akane8은 “슬롭이 PR이 되기 전에 최대한 제거하는 것이 목표”라고 답했다.[^akane8-snehesht]
리뷰어의 시간을 아끼기 위해 게이트를 가능한 한 앞에 두는 설계 의도가 명확히 드러난다.
PR 리뷰 단계에서의 자동화도 가능하지만, no-mistakes는 그보다 더 이른 개발자 로컬 환경을 선택했다.

### “finding” 이분법은 AI 자율성의 경계를 명시적으로 설정하는 설계다

no-mistakes가 finding을 “자동 수정”과 “인간 판단 필요”로 나누는 것은
AI 에이전트가 어디까지 자율적으로 행동할 수 있는지에 대한 명시적 경계 설정이다.

기계적이고 안전한 수정(포매팅, 린트 오류, 명백한 타이포)은 자동 적용하고,
아키텍처적 결정이나 의도가 불분명한 변경은 사람에게 넘긴다.
이 이분법은 실제로 어떻게 구현되는가 — 즉 “안전한 수정”의 경계를 누가 어떻게 결정하는가 — 가
도구의 실질적 유용성을 결정하는 핵심 질문이다.

### 격리된 worktree 실행은 에이전트 안전성 설계의 구체적 구현이다

no-mistakes는 검증을 일회용 worktree에서 실행한다.
이것은 에이전트가 현재 작업 중인 코드베이스를 직접 수정하는 위험을 차단한다.

검증 과정에서 에이전트가 잘못된 변경을 적용하더라도
그 영향은 격리된 worktree 안에 머물고 버려진다.
이 설계는 “에이전트가 실수를 해도 돌이킬 수 없는 결과를 만들지 않는다”는
안전성 원칙을 아키텍처 수준에서 보장한다.

## 비평

### “안전한 수정”의 경계는 실제로 명확하지 않다

no-mistakes는 finding을 자동 적용 가능한 것과 인간 판단이 필요한 것으로 나누지만,
이 경계가 실제로 어떻게 결정되는지는 저장소 README에서 명확하게 설명되지 않는다.

린트 오류 수정이 안전한 수정처럼 보이지만,
lint 룰 변경이 의미 있는 로직 변경을 가리는 경우가 있다.
테스트 추가 역시 “안전”해 보이지만, 잘못된 테스트는 오히려 거짓 안도감을 만든다.
AI 에이전트가 내리는 “이것은 기계적 수정이다”라는 판단 자체가 틀릴 수 있고,
그 오류는 자동 적용을 통해 코드베이스에 조용히 들어온다.

### 프록시 방식의 이점은 pre-commit 훅과 비교해야 비로소 명확해진다

HN에서 csh0은 핵심적인 질문을 던졌다.[^csh0]
“프록시로 이것을 달성하는 것의 pre-commit 훅 대비 이점이 무엇이냐”는 것이다.
pre-commit 훅도 동일하게 로컬에서, 커밋이 원격에 도달하기 전에 실행되며 AI 기반 검증을 붙일 수 있다.
이 질문은 no-mistakes가 해결하는 문제의 본질을 가리킨다.

저자 akane8은 이 질문에 상호작용성의 차이로 답했다.[^akane8-csh0]
에이전트가 7개의 finding을 발견했는데 그 중 4개만 수정하고 싶고,
각 항목에 대해 에이전트에게 구체적인 지침을 주고 싶을 때,
pre-commit 훅에서 이 대화형 주고받기는 매우 어색하다는 것이다.
no-mistakes의 대화형 TUI(`no-mysteries`)는 정확히 이 상호작용을 위해 설계됐다.
즉, 프록시 아키텍처의 차별점은 실행 타이밍이 아니라 검토-수정 루프의 인터랙티브한 설계에 있다.

다만 이 답변이 완전히 문제를 해소하는 것은 아니다.
모든 AI 기반 코드 검토가 이 수준의 상호작용이 필요한 것은 아니며,
단순한 자동화 게이트라면 pre-commit 훅으로 충분할 수 있다.
no-mistakes의 복잡한 아키텍처는 대화형 AI 워크플로를 전제로 할 때 비로소 정당화된다.

같은 질문은 한국 개발자 커뮤니티 GeekNews에서도 독립적으로
제기됐다.
t7vonn은 “git hook이랑 무슨 차이인지 잘 모르겠다”고 짧게
반응했고,[^t7vonn] ng0301은 그 아래에 “새로운 바퀴”라는 냉소적인
한마디를 덧붙였다.[^ng0301]
두 반응 모두 논거를 길게 펴지는 않지만, 겨냥하는 지점은 csh0의
질문과 다르지 않다.
Git 훅이나 pre-commit 프레임워크로 이미 비슷한 일을 할 수 있는데
왜 굳이 프록시가 필요하냐는 것이다.
HN과 GeekNews 양쪽에서 언어권과 상관없이 가장 먼저 나온 반응이
같았다는 사실은, 이 프로젝트를 접한 개발자라면 누구나 가장 먼저
떠올리는 의문이 그것임을 보여준다.

다만 GeekNews 쪽에는 저자가 직접 참여하지 않아,
HN에서는 상호작용성이라는 구체적 답을 얻은 의문이
GeekNews에서는 “새로운 바퀴”라는 결론으로 남아 있다.
저자의 답변(대화형 TUI를 통한 검토-수정 루프)이 실제로 이 냉소를
해소하는지는 README만으로 판단하기 어렵고,
결국 도구를 직접 써 봐야 확인할 수 있는 부분으로 남는다.

### 파이프라인이 길어질수록 느린 피드백이라는 원래 문제가 되돌아온다

no-mistakes의 파이프라인은 review → test → docs → lint → push → PR → CI로 이어진다.
이 모든 단계가 순차적으로 실행된다면, 단순한 수정 하나를 푸시하는 데도
전체 파이프라인이 완료될 때까지 기다려야 할 수 있다.

빠른 피드백을 위해 설계됐지만, 복잡한 파이프라인 자체가 새로운 지연 요인이 된다.
특히 AI 기반 리뷰 단계는 단순한 정적 분석보다 훨씬 느릴 수 있다.
“비차단(non-blocking)” 설계가 이를 부분적으로 완화하지만,
결국 리모트로 푸시되기 전에 모든 단계가 완료되어야 한다는 사실은 변하지 않는다.

### 멀티 에이전트 지원은 이기종 환경에서 일관성 문제를 만들 수 있다

no-mistakes는 Claude, Codex, RovoDev, OpenCode 등 여러 AI 에이전트를 지원한다.
그러나 에이전트마다 코드 리뷰 기준, 수정 스타일, 판단 경향이 다르다.

같은 코드베이스에 팀원 A는 Claude를 쓰고, 팀원 B는 Codex를 쓴다면
“클린 PR”의 기준이 사람마다 달라진다.
도구 레벨의 통일성이 에이전트 레벨에서 무너지는 것이다.
이것은 팀 단위 도입에서 실질적인 마찰 요인이 될 수 있다.

## 인사이트

### Git 푸시 게이트는 코드 품질 책임의 위치를 바꾼다

전통적인 코드 품질 관리는 “PR 리뷰어”라는 역할에 의존한다.
리뷰어가 문제를 잡고, 피드백을 남기고, 작성자가 수정한다.
이 모델에서 품질 책임은 분산되어 있고, 마찰은 리뷰 단계에 집중된다.

no-mistakes는 이 책임을 작성자의 로컬 환경으로 이동시킨다.
리뷰어에게 도달하기 전에 이미 특정 수준의 품질이 보장되어야 한다.
이것은 리뷰어의 부담을 줄이는 동시에, 작성자가 더 오랜 시간을 로컬에서 보내게 만든다.
품질의 총량이 줄어드는 것이 아니라 위치가 바뀌는 것이다.
이 이동이 팀 전체에 이익인지는 팀의 병목이 어디에 있는가에 달려 있다.

### AI 게이트가 일반화되면 “누가 리뷰했는가”의 의미가 달라진다

현재 코드 리뷰는 인간이 인간의 작업을 검토한다는 전제 위에 있다.
no-mistakes 같은 도구가 광범위하게 채택되면,
“PR이 열린 시점”의 코드는 이미 AI 에이전트가 1차 검토를 완료한 상태다.

이것은 인간 리뷰어의 역할을 “전체 검토”에서 “AI가 놓친 것 찾기”로 바꾼다.
리뷰어는 AI가 안전하다고 판단한 변경들을 다시 의심해야 한다는 새로운 인지 부하를 가진다.
“AI가 이미 확인했으니 괜찮겠지”라는 신뢰 전이가 일어날 때,
리뷰어의 실제 검토 깊이가 줄어드는 현상이 나타날 수 있다.
AI 게이트가 품질을 높이는 것인지, 아니면 책임의 소재를 흐리는 것인지는
운용 패턴에 따라 달라진다.

### 프록시 Git 서버 패턴은 코드 품질 이외의 용도로 확장될 수 있다

jmagland는 no-mistakes에서 프록시 Git 서버 아이디어 자체에 주목했다.[^jmagland]
“로컬과 원격 사이에 프록시 Git 서버를 두는 방식을 본 적이 없다”며,
코드 검토 인터셉션 외에 다른 활용 사례를 상상할 수 있는지 물었다.
이 질문은 no-mistakes를 하나의 도구가 아니라 하나의 아키텍처 패턴으로 보게 만든다.

Git 푸시를 가로채는 프록시는 실제로 코드 품질 검증 이외에도 다양한 용도로 확장될 수 있다.
시크릿 탐지(비밀키·토큰 노출 방지), 라이선스 체크, 대용량 파일 차단,
사내 정책 준수 확인, 변경 사항 로깅 등은 현재도 git hooks나 서버 사이드에서 처리하는 작업이다.
이 패턴이 AI 기반 검토와 결합되면, 정책 기반 자동화의 레이어를 로컬 환경 수준으로 끌어내릴 수 있다.

no-mistakes는 코드 품질에 집중하지만, 프록시 아키텍처가 범용화된다면
Git 원격의 “미들웨어 레이어”라는 새로운 개념이 성립할 수 있다.
지금은 no-mistakes가 선구적 사례이지만, 이 패턴 위에 다른 도구들이 구축될 가능성은 열려 있다.

### 푸시 전 검증 도구는 팀 문화와 충돌하는 지점을 만들 수 있다

“빠르게 실험하고 PR에서 논의한다”는 팀 문화는
no-mistakes의 “클린하게 만든 다음 푸시한다”는 철학과 충돌한다.
WIP(Work in Progress) PR, draft PR, 탐색적 커밋을 적극적으로 활용하는 팀에서
모든 푸시에 full 파이프라인을 실행하는 것은 오버헤드가 될 수 있다.

no-mistakes가 가장 잘 작동하는 환경은
코드 기준이 명확하고, 자동화 가능한 검증 항목이 많고,
PR이 최종 결과물에 가까울 때 열리는 팀이다.
이 조건들이 갖춰지지 않은 환경에서는 도구의 이점이 오버헤드를 상쇄하지 못할 수 있다.
