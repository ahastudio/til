# Open Agents

원문: <https://github.com/vercel-labs/open-agents>

## 요약

Vercel Labs가 공개한 오픈소스 레퍼런스 앱으로,
백그라운드에서 동작하는 코딩 에이전트를 구축하고 실행하기 위한 참조 구현이다.
채팅 기반 인터페이스로 코딩 에이전트를 조작하며,
파일, 검색, 셸, 태스크, 스킬, 웹 등 다양한 도구를 제공한다.

아키텍처는 세 계층으로 나뉜다:
Web(Next.js 프론트엔드) → Agent workflow(Workflow SDK) → Sandbox VM(격리 실행 환경).
핵심 설계 원칙은 에이전트가 VM 내부에서 실행되지 않는다는 것이다.
에이전트는 외부에서 실행되며 도구를 통해 샌드박스와 상호작용한다.

Vercel의 Workflow SDK를 사용한 내구성 있는(durable) 다단계 실행,
스냅샷 기반 재개가 가능한 격리 샌드박스,
레포 클론·브랜치 관리·자동 커밋·푸시·PR 생성 등
실무에 필요한 Git 워크플로를 기본 지원한다.
세션 공유, ElevenLabs 음성 입력도 포함되어 있다.

모노레포 구조로 TypeScript 99.3%이며,
최소 PostgreSQL과 JWE 시크릿이 필요하고
전체 기능에는 10개 이상의 환경 변수가 요구된다.
Labs 프로젝트로 API 안정성은 보장되지 않는다.

## 분석

### 세 계층 분리의 설계 의도

Web → Agent workflow → Sandbox VM의 세 계층은
관심사 분리(separation of concerns)의 교과서적 적용이다.
프론트엔드는 사용자 인터랙션만 담당하고,
에이전트 워크플로는 추론과 도구 오케스트레이션을 처리하며,
샌드박스 VM은 코드 실행의 격리를 보장한다.
이 분리 덕분에 각 계층을 독립적으로 교체하거나 확장할 수 있다.

### 에이전트가 VM 외부에서 실행되는 구조

에이전트가 샌드박스 내부가 아닌 외부에서 실행된다는 설계 결정이 중요하다.
에이전트를 VM 안에 넣으면 격리는 강화되지만
워크플로 상태 관리와 도구 접근이 복잡해진다.
외부 실행 방식은 에이전트의 내구성(durability)과
샌드박스의 격리(isolation)를 동시에 확보한다.
도구 호출이라는 명확한 인터페이스를 경계로 두 영역이 분리되는 것이다.

### Workflow SDK를 통한 내구성 확보

Vercel의 Workflow SDK를 사용한 다단계 실행은
에이전트의 장시간 작업을 안정적으로 만든다.
코딩 에이전트는 수십 번의 도구 호출을 거치므로
중간에 실패하면 처음부터 다시 시작해야 하는 문제가 있다.
스냅샷 기반 재개는 이 문제를 해결하되,
상태를 직렬화 가능한 형태로 유지해야 한다는 제약을 부과한다.

### 레퍼런스 앱의 포지셔닝

“레퍼런스 앱”이라는 명칭이 의도적이다.
프레임워크나 라이브러리가 아니라 “이렇게 만들 수 있다”는 참조 구현이다.
API 안정성을 보장하지 않는다는 점이 이를 뒷받침한다.
이는 사용자가 포크해서 자신의 요구에 맞게 수정하는 것을 전제하며,
동시에 Vercel 인프라 위에서 에이전트를 구축하는 패턴을 보급하려는 전략이다.

## 비평

### Vercel 인프라 종속성

가장 큰 한계는 Vercel 인프라에 대한 강한 결합이다.
Workflow SDK, 샌드박스 VM, 배포 파이프라인 모두 Vercel에 의존한다.
“오픈소스”라는 이름에도 불구하고
다른 클라우드나 자체 인프라로 이식하기가 쉽지 않다.
코드를 읽고 패턴을 배울 수는 있지만,
실제로 포크해서 독립적으로 운영하려면 상당한 재작업이 필요하다.

### 환경 변수 10개 이상의 진입 장벽

최소 구성에 PostgreSQL과 JWE 시크릿이 필요하고,
전체 기능을 쓰려면 10개 이상의 환경 변수를 설정해야 한다.
레퍼런스 앱의 목적이 학습과 참조라면
이 진입 장벽은 목적에 어울리지 않는다.
로컬에서 빠르게 실험할 수 있는 최소 모드가 있었다면
더 많은 개발자가 실제로 실행해 보았을 것이다.

### Labs 프로젝트의 지속 가능성

Labs 프로젝트라는 점은 양날의 검이다.
실험적 성격 덕분에 과감한 설계를 시도할 수 있지만,
장기 유지보수와 커뮤니티 성장에 대한 약속이 없다.
API 안정성을 보장하지 않으므로
이 코드베이스 위에 프로덕션 시스템을 구축하는 것은 위험하다.
참조 구현의 가치는 유지보수가 멈추는 순간 급격히 감소한다.

## 인사이트

### 에이전트 인프라는 새로운 클라우드 락인(lock-in) 경로다

클라우드 벤더들은 전통적으로 데이터베이스, 스토리지, 컴퓨팅으로 락인을 만들었다.
Open Agents는 새로운 락인 경로를 보여준다.
에이전트 워크플로, 샌드박스 VM, 내구성 실행이라는 인프라 계층이
그 자체로 이식이 어려운 종속성을 형성한다.

이는 컨테이너 오케스트레이션의 역사와 닮아 있다.
Docker가 컨테이너를 표준화했지만
Kubernetes가 오케스트레이션 계층을 장악하면서
각 클라우드의 관리형 Kubernetes가 새로운 종속 지점이 되었다.
에이전트 인프라도 같은 경로를 따를 가능성이 높다.
에이전트 실행 런타임, 워크플로 SDK, 샌드박스 관리가
“관리형 에이전트 서비스”로 패키징되어
클라우드 벤더의 새로운 차별화 포인트가 될 것이다.

Vercel이 레퍼런스 앱을 오픈소스로 공개한 것은
이 새로운 인프라 계층의 표준을 선점하려는 움직임으로 읽힌다.
코드를 공개하되 인프라는 자사 플랫폼에 최적화함으로써
“개방적이지만 이탈하기 어려운” 구조를 만든다.
이는 Android가 오픈소스이지만 Google 서비스 없이는
실용적이지 않은 것과 같은 패턴이다.

### 에이전트-샌드박스 분리는 마이크로서비스의 반복이다

에이전트가 VM 외부에서 실행되고 도구 호출로 샌드박스와 통신하는 구조는
마이크로서비스 아키텍처의 핵심 원칙을 그대로 따른다.
각 서비스가 자체 프로세스에서 실행되고
잘 정의된 API로만 통신하는 것이다.

이 분리가 가져오는 이차 효과(second-order effect)가 흥미롭다.
에이전트와 샌드박스를 독립적으로 스케일링할 수 있고,
하나의 에이전트가 여러 샌드박스를 동시에 조작하거나
하나의 샌드박스를 여러 에이전트가 순차적으로 사용하는 것이 가능해진다.
스냅샷 기반 재개는 이 분리를 더욱 강화한다.
샌드박스 상태를 스냅샷으로 저장하면
에이전트가 중단되어도 다른 에이전트가 이어받을 수 있다.

그러나 마이크로서비스가 그랬듯이 분산 시스템의 복잡성도 따라온다.
네트워크 지연, 부분 실패, 상태 동기화 문제가 발생한다.
에이전트가 도구 호출을 보냈는데 샌드박스가 응답하지 않으면
에이전트는 재시도할지, 기다릴지, 포기할지 결정해야 한다.
이런 분산 시스템 문제는 모놀리식 에이전트에서는 존재하지 않았던 것이다.

### 레퍼런스 앱 전략은 플랫폼 보급의 고전적 수법이다

“레퍼런스 앱”이라는 포지셔닝은
플랫폼 기업이 개발자 생태계를 구축할 때 반복적으로 사용하는 전략이다.
Apple의 샘플 코드, Google의 Codelabs, AWS의 레퍼런스 아키텍처가 모두 같은 논리다.
“이렇게 만들 수 있다”를 보여주면서
자연스럽게 자사 플랫폼의 사용 패턴을 각인시킨다.

Open Agents가 TypeScript 99.3%이고
Next.js와 Vercel Workflow SDK를 기반으로 한다는 것은
Vercel의 기존 개발자 기반을 정확히 겨냥한 것이다.
이미 Next.js를 쓰는 개발자가 에이전트를 만들고 싶을 때
가장 자연스러운 출발점이 되도록 설계되어 있다.
이는 기술적 선택인 동시에 마케팅 결정이다.

흥미로운 점은 이 전략의 효과가 프로젝트의 수명과 무관할 수 있다는 것이다.
Labs 프로젝트가 중단되더라도
개발자들이 이미 학습한 패턴 — 세 계층 분리, 외부 에이전트 실행,
Workflow SDK 기반 내구성 — 은 Vercel 플랫폼에 대한 친숙도로 남는다.
레퍼런스 앱의 진정한 산출물은 코드가 아니라
개발자의 인지적 종속(cognitive lock-in)이다.
