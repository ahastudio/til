# 사람이 유지보수할 것처럼 코드를 작성하라

원문: <https://unstack.io/write-code-like-a-human-will-maintain-it>

HN 토론: <https://news.ycombinator.com/item?id=48859701> (345점, 299개 댓글)

## 요약

저자 Scott Robinson은 LLM으로 코드를 생성할 때도 높은 코딩 표준을
유지해야 한다고 주장한다.
AI 모델이 이미 코드베이스에 존재하는 패턴으로부터 학습하기
때문에, 오늘 타협한 코드 품질이 내일 더 나쁜 코드를 만들어내도록
모델을 훈련시키는 셈이라는 것이다.

저자는 LLM이 유지보수를 대신 맡아주면서 코딩 표준이 느슨해지기가
얼마나 쉬운지 설명한다.
“같은 긴 조건문을 네 곳의 서로 다른 파일에서 직접 갱신할 필요가
없다”는 이유로 DRY 원칙에 신경 쓰지 않게 됐다는 것이다.

실제 문제는 저자가 접근 권한 검사 조건문을 라우트 핸들러,
백그라운드 작업, API 엔드포인트, 웹훅 등 여러 위치에 중복
작성했다는 사실을 발견하면서 드러났다.
공유 헬퍼 함수로 추출하는 대신, 그는 AI가 생성한 각 버전을 그냥
받아들였다.

핵심 통찰은 패턴이 누적되는 방식에 있다.
“코드베이스에 병합하는 모든 지름길은 '여기서는 일이 이렇게
처리된다'는 신호”라는 것이다.
LLM은 코드를 생성할 때 기존 코드 패턴을 분석하므로, 반복된 실수는
프로젝트 스타일에 대한 모델의 이해 안에 그대로 새겨진다.

저자는 자신이 유지보수를 아웃소싱한 것이 아니라, AI가 점진적으로
더 나쁜 습관을 갖도록 훈련시키고 있었다는 것을 깨달았다고
말한다.
코드 냄새는 누적되고, 이는 이후 리팩터링을 점점 더 어렵게
만든다.
결론적으로 사람이 유지보수해야 한다는 전제로 코드를 작성해야
하는데, LLM은 “당신이 하는 모든 것을 빨아들였다가 그대로
돌려주는 스펀지”처럼 작동하기 때문이다.

## 분석

### 코드 품질 저하가 “게으름”이 아니라 “위임의 부작용”으로 재정의된다

저자의 논지가 흥미로운 지점은, 코드 품질 저하의 원인을 개발자의
나태함이 아니라 위임이라는 행위 자체의 구조적 특성에서 찾는다는
것이다.
전통적으로 나쁜 코드는 시간 압박이나 실력 부족의 결과로
설명됐지만, 저자가 겪은 문제는 정반대다.
그는 반복 작업을 AI에게 맡길 수 있게 되면서 “내가 같은 긴 조건문을
네 곳에서 갱신할 필요가 없다”는 이유로 추상화의 필요성 자체를
느끼지 못하게 됐다.
즉 나쁜 코드가 만들어진 이유는 게을러서가 아니라, 반복의 고통이라는
추상화를 만들게 하는 자연스러운 압력이 AI 위임으로 인해
사라졌기 때문이다.

이 관찰은 HN 댓글에서 실천적 해법으로 확장된다.
cadamsdotcom은 자신의 `/review` 커스텀 명령어 워크플로를
공유하며, 코드 리뷰 체크리스트를 마크다운 파일로 만들어 브랜치와
main의 차이를 검토하게 하는 방식을 설명한다[^cadamsdotcom].
planb 역시 매 큰 변경 후 “이제 최종 코드 점검을 하라.
모든 것이 깔끔하고 컴포넌트들이 관심사 분리 원칙을 따르는가?
모든 것이 이해 가능하고 유지보수 가능한 상태인가?”라고 묻는
프롬프트를 실행한다고 말한다[^planb].
이런 반응들이 보여주는 것은, 저자가 겪은 문제—추상화 압력의
소실—를 해결하려면 사람이 원래 무의식적으로 수행하던 “이거 좀
지저분한데”라는 판단을 의식적인 별도 단계로 재도입해야 한다는
점이다.
즉 AI 위임 시대의 코드 품질 관리는 저절로 되지 않고, 명시적으로
설계해 넣어야 하는 별도의 프로세스가 됐다.

### 추상화 반복 여부가 모델의 “이미 존재하는 패턴 편향”에 좌우된다

davnicwil의 댓글은 저자의 핵심 주장—“다음번에 같은 접근 규칙을
가진 엔드포인트를 요청하면 모델은 처음부터 원칙에 따라 시작하지
않는다.
이미 저장소에 있는 네 개의 복사본에서 시작할 것이다”—에 대해
기술적으로 더 정밀한 반박을 내놓는다.
“이게 얼마나 사실인지는 잘 모르겠다.
기본 구조를 반복하고 재사용하지 않는(추상화를 만들어내는 것은
말할 것도 없고) 상당히 뿌리 깊은 편향이 있는 것 같다.
그래서 기존 패턴이 그렇다면 그 패턴을 유지하는 것처럼
보인다”고 말한다[^davnicwil].
이 미묘한 구분이 중요한 이유는, 저자의 원인 진단(코드베이스의
나쁜 패턴이 학습되어 재생산된다)과 davnicwil의 진단(LLM은
애초에 추상화보다 반복을 선호하는 경향이 있고, 나쁜 패턴은
그 경향이 발현될 계기를 제공했을 뿐이다)이 서로 다른 개입
지점을 시사하기 때문이다.
전자가 맞다면 코드베이스를 깨끗하게 유지하는 것으로 문제가
해결되지만, 후자가 맞다면 코드베이스가 깨끗해도 LLM은 여전히
추상화를 만들어내기를 꺼릴 수 있다는 뜻이 된다.

## 비평

### “프롬프트를 더 쓰면 해결된다”는 대안도 새로운 문제를 낳는다

저자가 제시하는 처방—사람이 유지보수할 것처럼 코드를
작성하라—은 원론적으로 타당하지만, HN 댓글은 이 실천이 실제로
효과가 있는지에 의문을 제기한다.
jwpapi는 “이 안에서 얼마나 많은 엔지니어가 그냥 프롬프트를 하나
더 위에 얹어 쓴다고 말하는지 놀랍다.
내 경험상 그건 상황을 더 나쁘게 만든다.
추상화를 만들기는 하지만 잘못된 추상화를 만든다.
과도하게 주석을 달아서 LLM의 이후 호출을 혼란스럽게
만든다”고 지적한다[^jwpapi].
그는 “좋은 코드베이스에서는 잘 작동하지만, 코드베이스를 점점
더 나쁘게 만들어서 결국 작동하지 않게 될 때까지 계속 쓰다가
결국 대가를 치르게 된다”고 경고한다.
이 반론이 시사하는 바는, 저자가 제안하는 해법—더 나은 리뷰
프롬프트, 더 엄격한 지시—자체가 만능이 아니라는 것이다.
프롬프트로 코드 품질을 강제하려는 시도가 오히려 잘못된
추상화나 과도한 주석 같은 새로운 종류의 코드 냄새를 만들어낼
수 있다는 반박은, 저자의 처방이 문제를 해결하기보다 문제의
형태를 바꿀 뿐일 수 있다는 우려로 이어진다.

### “그냥 사람이 직접 코드를 작성하라”는 반론을 정면으로 다루지 않는다

이 글이 은근히 회피하는 가장 직접적인 대안은 agentultra의 댓글에서
날카롭게 제시된다.
“미친 생각: 그냥 코드를 작성하라.
안다.
이 AI 시대에 믿기 힘든 개념이라는 걸.
코드를 작성한다고?
그게 공룡들이나 하던 거 아닌가?
사람이 그 코드를 읽고 유지보수해야 한다고 예상한다면, 차라리
그들을 위해 직접 작성하는 게 낫다.
지나치게 장황한 복사-붙여넣기 코드를 읽어야 하는 게 짜증날
것이다.
그들도 마찬가지일 거다.
그러니 코드를 직접 작성하라”고 말한다[^agentultra].
Snoopfrogg도 짧게 “아니면 그냥 LLM 없이 코드를 작성하기 시작하면
된다”고 동조한다[^Snoopfrogg].
저자의 글 전체는 “LLM을 계속 쓰되 규율을 지키자”는 전제 위에
서 있는데, 이 전제 자체에 도전하는 반론—애초에 사람이 유지보수할
코드를 사람이 직접 쓰는 것이 왜 안 되는가—을 원문은 전혀 다루지
않는다.
저자가 이 대안을 진지하게 검토하지 않은 채 “LLM 사용은 주어진
조건”으로 전제하고 논의를 시작한다는 점은, 이 글이 실제로는
더 근본적인 질문(“애초에 LLM에게 코드 생성을 맡길 필요가
있는가”)을 피해간다는 약점으로 읽힌다.

## 인사이트

### “사람이 유지보수할 것처럼 쓰라”는 원칙은 LLM 이전에도 이미 있던 원칙의 재발견이다

jtr1의 댓글은 이 글 전체의 논지를 더 오래된 소프트웨어 공학
원칙과 연결 지으며 새로운 통찰을 제공한다.
“대부분의 프로그래밍 모범 사례는 LLM이 직면한 것과 같은
문제—제한된 맥락(context)—를 해결하기 위해 만들어졌다.
잘 만들어진 코드에 관한 원칙들은 저자가 아닌 미래의 에이전트가
그 코드를 수정할 수 있도록 만드는 것을 지향해 왔다.
유지보수 가능한 코드는 그것이 다루는 도메인의 불변 조건 주위에서
안정화되면서, 미래 사용자에게 UI > 설정 > 깔끔하게 추상화된
코드 > 핵심 로직으로 이어지는 수정 가능성의 스펙트럼을
제공한다”고 정리한다[^jtr1].
이 관찰이 중요한 이유는, “사람이 읽고 유지보수하기 쉬운 코드를
쓰라”는 원칙이 LLM 시대에 새로 발명된 것이 아니라, 애초에
“제한된 맥락을 가진 행위자가 코드를 이해하고 수정해야 한다”는
문제를 풀기 위해 이미 수십 년 전부터 존재해 온 원칙이라는
점이다.
사람이든 LLM이든 한 번에 파악할 수 있는 맥락의 크기가 유한하다는
공통 제약을 가진 행위자라는 관점에서 보면, 좋은 소프트웨어
공학 원칙이 LLM에게도 유효한 것은 우연이 아니라 필연이다.
이는 “AI 시대에 맞는 새로운 코딩 원칙이 필요하다”는 성급한
결론 대신, 오히려 기존 원칙이 왜 유효했는지를 더 깊이 이해할
계기를 제공한다.

### 코드 품질 붕괴의 책임 소재가 LLM에서 사람의 오래된 습관으로 되돌아간다

illuminator83의 댓글은 이 논의 전체에 중요한 균형추를 제공한다.
“사람들이 20년간 유지보수해 온 코드 중에는 이런 중복과 그보다
더한 것들로 가득한 경우가 많다.
사실 슬프게도, 사람들이 작성하고 유지보수해 온 코드 대부분이
오늘날 LLM이 만들어내는 것보다 더 나쁘다고 말할 수밖에 없다.
종종 그것은 미숙함 때문이고, 때로는 의도적인 태만 때문이지만,
가장 흔한 원인은 그저 빠듯한 마감과 지금 하고 있는 일을 끝내야
한다는 압박 때문이다.
사람들은 더 잘하는 방법을 알지만, 그럴 시간을 얻지 못한다”고
말한다[^illuminator83].
이 지적이 중요한 이유는, 이 글 전체가 암묵적으로 “LLM 도입 이전에는
코드 품질이 괜찮았는데 LLM이 이를 악화시킨다”는 전제를 깔고
있는데, 실제로는 코드 품질 저하가 LLM 이전부터 존재해 온
훨씬 오래된 문제—시간 압박, 조직의 우선순위, 기술 부채를 감수하는
문화—의 연장선일 수 있다는 것이다.
만약 이 관찰이 맞다면, “LLM에게 좋은 코드를 학습시켜야 한다”는
저자의 프레이밍은 문제의 새로운 원인(LLM)에 집중하느라, 애초에
사람들이 나쁜 코드를 용인해 온 더 오래되고 더 근본적인 조직적
원인—마감과 압박의 문화—을 가릴 위험이 있다.
LLM은 이 오래된 문제를 새로운 속도와 규모로 증폭시키는 도구일
뿐, 문제의 근원이 아닐 수 있다는 것이다.
