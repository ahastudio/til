# 장기 실행 에이전트 — 에이전트가 며칠 동안 실행되면 무엇이 달라지는가

원문: <https://addyo.substack.com/p/long-running-agents>

HN 토론: <https://news.hada.io/topic?id=29153>

## 요약

장기 실행 에이전트(long-running agents)는 단일 대화 세션을 넘어 시간 단위, 혹은 며칠에 걸쳐 지속적으로 실행되는 에이전트를 뜻한다.
상태를 컨텍스트 윈도우 외부에 유지하고, 실패로부터 복구하며, 세션 간 진행 상황을 보존한다.
Anthropic 내부 테스트에서는 Claude가 30시간 이상 자율적으로 코딩을 수행했으며, 그 중 한 번의 실행에서 11,000줄짜리 애플리케이션이 생성됐다.

“장기 실행”이라는 개념은 세 가지 서로 다른 의미를 포함한다.
**장기 수평 추론(long-horizon reasoning)**은 수많은 의존적 단계에 걸쳐 계획하는 능력이며,
METR의 연구에 따르면 프론티어 모델의 시간 수평 지표는 2019년 이후 약 7개월마다 두 배씩 성장했다.
**장기 실행(long-running execution)**은 수천 번의 모델 호출을 포함하는 프로세스로,
주로 모델 역량이 아닌 엔지니어링 문제다.
**지속적 에이전시(persistent agency)**는 세션 간 정체성을 유지하고 메모리를 축적하는 에이전트로,
Google의 Memory Bank 접근 방식이 대표적이다.

장기 실행에는 세 가지 핵심 기술 난제가 있다.
첫째, 유한한 컨텍스트 윈도우다. 백만 토큰 윈도우도 결국 채워지고, 하드 한계에 도달하기 전에 “컨텍스트 부패(context rot)”가 성능을 저하시킨다.
둘째, 영속적 상태의 부재다. 새 세션은 항상 백지 상태에서 시작하며, 이를 “이전 교대 엔지니어의 기억이 없는 교대제 소프트웨어 팀”에 비유할 수 있다.
셋째, 자기 검증 불가능성이다. 모델은 자신의 작업을 평가할 때 긍정 편향을 보이며, 완료 여부를 물으면 실제보다 더 자주 “완료됐다”고 답한다.

주요 구현 패턴으로는 Geoffrey Huntley가 대중화한 **Ralph 루프(Ralph Loop)**가 있다.
`prd.json`(계획), `progress.txt`(메모), `AGENTS.md`(규칙) 같은 파일로 상태를 외부에 보관하고,
미완료 태스크 선택 → 프롬프트 빌드 → 에이전트 호출 → 검증 → 결과 기록 → 상태 갱신의 사이클을 반복한다.
이 접근 방식은 복잡한 인프라 없이도 작동하는 장기 실행 에이전트를 구현할 수 있음을 보여준다.

Anthropic은 **브레인(Brain)·핸즈(Hands)·세션(Session)** 3구성 요소 모델을 개발했다.
세션은 모든 사고, 도구 호출, 관찰을 기록하는 추가 전용(append-only) 이벤트 로그다.
이 분리 덕분에 무상태(stateless) 하네스가 가능해졌으며, 새 컨테이너가 `wake(sessionId)`를 호출해 로그에서 상태를 재구성한다.
Cursor는 **플래너(Planner)·워커(Worker)·저지(Judge)** 역할 분리를 도입하여 조율 병목을 해소했다.

## 분석

### 세션을 이벤트 로그로 취급하는 패러다임 전환

Anthropic의 “세션을 추가 전용 이벤트 로그로” 접근은 분산 시스템 설계에서 이벤트 소싱(event sourcing) 패턴과 구조적으로 동일하다.
상태는 이벤트의 파생물이고, 최신 상태는 로그를 리플레이해 재구성된다.
이 패턴은 에이전트를 무상태로 만들어 컨테이너 재시작, 장애 복구, 독립적 컴포넌트 교체를 가능케 한다.
이를 통해 “time-to-first-token이 p50에서 약 60%, p95에서 90% 이상 감소”라는 실질적 성과가 나왔다.

### 역할 분리가 시스템 성능을 가른다

Cursor의 발견 — “시스템 동작의 놀랍도록 많은 부분이 인프라보다 프롬프팅에서 비롯된다” — 은
장기 실행 에이전트 설계에서 중요한 교훈이다.
또한 “다른 역할에 다른 모델이 맞는다”는 점도 주목할 만하다.
Opus가 장시간 자율 작업에서 GPT 모델보다 성과가 낮았던 이유는
Opus가 완료 기준보다 일찍 태스크를 마무리하려는 경향을 보이기 때문이었다.
모델의 “성격”이 장기 실행 맥락에서는 인프라 선택만큼이나 중요한 변수가 된다.

### 경제적 의미의 전환

10분짜리 에이전트는 질문에 답하지만, 10시간짜리 에이전트는 기능 전체를 소유할 수 있다.
Payhawk가 Google Memory Bank 통합 후 제출 시간이 50% 이상 단축됐다고 보고하듯,
장기 실행이 가능해지면 위임 가능한 작업의 범주가 근본적으로 달라진다.
에이전트가 경쟁사 동향, 테스트 패턴, 사용자 선호도에 대한 맥락적 지식을 축적할 수 있다면,
그것은 더 이상 무상태 도구가 아니라 점점 가치가 높아지는 협업자가 된다.

## 비평

### 강점: 실전에서 검증된 패턴의 통합

Ralph 루프, Anthropic의 Brain/Hands/Session, Cursor의 Planner/Worker/Judge 등
실제 운영에서 검증된 패턴들을 한 자리에 정리했다는 점이 이 글의 가장 큰 강점이다.
추상적 원칙이 아니라 구체적 구현 형태로 제시되어 즉시 적용 가능하다.

### 약점: 비용 현실에 대한 과소 평가

“프론티어 모델로 24시간 자율 실행은 상당한 비용이 든다”고 언급하지만,
이 비용이 실제로 어느 수준인지, 어떤 종류의 작업에서 경제적으로 타당한지에 대한 분석이 부족하다.
24시간 에이전트의 비용이 시니어 개발자 하루 인건비와 비교해 어떤지,
어떤 작업 유형에서 ROI가 나오는지를 명시하지 않으면 실용적 적용 판단이 어렵다.

### 약점: 정렬 드리프트 문제의 과소 평가

[GeekNews 댓글](https://news.hada.io/topic?id=29153)에서 Codex CLI의 `/goal` 기능이
장기 에이전트의 목표 유지 문제를 해결하는 접근으로 언급됐다.
이 기능은 에이전트가 초기 목표를 컨텍스트 압축과 세션 전환 후에도 잃지 않도록 명시적으로 관리한다.
원문은 컨텍스트 압축 시 요약이 충실도를 잃는다고 지적하지만,
이 목표 드리프트가 실제로 어떤 양상으로 나타나고 어떻게 감지하는지에 대한 구체적 방법론이 부재하다.

### 보완점: 인간의 역할 재정의

원문이 “스펙 작성 능력이 점점 중요해진다”고 언급하는 것은 중요한 방향 전환이다.
개발자가 “구현하는 사람”에서 “자율 실행을 가능하게 하는 명세를 작성하는 사람”으로 역할이 바뀐다면,
이것은 기술 스킬셋뿐 아니라 교육 방식과 경력 경로에도 큰 영향을 미친다.
이 변환에 대한 더 깊은 탐구가 필요하다.

## 인사이트

### 장기 에이전트는 분산 시스템 설계 문제다

장기 실행 에이전트의 핵심 과제들 — 상태 영속성, 장애 복구, 비동기 처리, 컴포넌트 간 역할 분리 —
은 모두 분산 시스템 엔지니어링이 수십 년에 걸쳐 다루어온 문제들이다.
이벤트 소싱, CQRS, 사가 패턴, 멱등성 처리 등 분산 시스템 패턴들이
에이전트 아키텍처에서 직접 재현된다.

이것이 의미하는 바는 AI 에이전트 엔지니어링이 결국 소프트웨어 엔지니어링의 기존 지식 체계 위에서
수렴할 것이라는 점이다. “어떻게 신뢰할 수 있는 장기 실행 시스템을 만드는가”라는 질문은
새로운 것이 아니며, 기존 분산 시스템 엔지니어링의 정수가 에이전트 영역으로 이전되는 중이다.
반대로, LLM을 컴포넌트로 포함하는 시스템의 비결정성(nondeterminism)은
기존 분산 시스템보다 상태 관리를 더 복잡하게 만드는 새로운 차원을 추가한다.

### 검증 분리가 장기 에이전트의 신뢰성 핵심이다

“생성자와 평가자를 구조적으로 분리하라”는 원칙은 장기 실행에서 더욱 중요해진다.
Cursor의 Judge 역할과 Anthropic의 랄프 루프에서 진행 상황을 재확인하는 단계가 모두 같은 원리를 구현한다.
단기 대화에서 모델의 긍정 편향은 허용 가능한 수준이지만,
며칠 동안 자율 실행하는 에이전트에서 검증 없이 긍정 편향이 누적되면
결과물이 명세와 완전히 달라진 채 “완료” 상태가 될 수 있다.

이 문제의 깊은 의미는, AI 에이전트 시스템에서 외부 검증자가 선택사항이 아니라 핵심 아키텍처 요소라는 것이다.
인간의 단속적 검토만으로는 며칠의 실행을 충분히 감독할 수 없으며,
자동화된 검증 레이어 — 코드 테스트, 린터, 타입 체커, 전문화된 Judge 에이전트 — 가
신뢰성 있는 장기 실행의 필수 조건이 된다.

### 장기 에이전트의 확산은 소프트웨어 조직의 구조를 바꾼다

10시간짜리 에이전트가 기능을 소유할 수 있다면,
기능 개발의 최소 단위가 개인 개발자의 집중 시간 단위에서
명세-실행-검증의 사이클 단위로 이동한다.
스프린트와 티켓 단위의 계획 방식, 코드 리뷰와 PR 프로세스,
시니어·주니어 역할 분담 구조가 모두 재설계 압력을 받게 된다.

Cursor가 “로컬에서 원격으로의 핸드오프” 문제를 미해결로 언급한 것처럼,
이 전환은 현재 진행 중이며 답이 정해져 있지 않다.
하지만 방향은 분명하다. 에이전트가 더 오래, 더 자율적으로 실행될수록
인간의 역할은 “실행”에서 “명세 작성과 감독”으로, 그리고 점차 “검증 인프라 설계”로 이동한다.
이 전환이 가장 먼저 가시화되는 곳은 작은 팀이 에이전트를 통해 큰 팀의 생산성을 달성하는
스타트업 생태계가 될 것이다.
