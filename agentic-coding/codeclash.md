# CodeClash: 목표 지향 소프트웨어 엔지니어링 벤치마크

원문: <https://codeclash.ai/>

GitHub: <https://github.com/codeclash-ai/codeclash>

## 소개

CodeClash는 AI 시스템이 목표 지향(goal-oriented) 소프트웨어 엔지니어링을 얼마나
잘 수행하는지 평가하는 오픈소스 벤치마크다. Princeton & Stanford의 SWE-bench/SWE-agent
팀이 개발했으며, arXiv 논문(2511.00839)과 함께 공개했다.

기존 코딩 벤치마크(HumanEval, SWE-bench 등)는 과제 지향(task-oriented)이다. 모델에게
명시적인 지시를 주고 단위 테스트로 정확성을 검증한다. 그러나 실제 소프트웨어 개발은
“사용자 유지율을 높여라”, “비용을 줄여라”처럼 고수준 목표에서 시작하는 자기 주도적이고
반복적인 경쟁 과정이다. CodeClash는 이 역동성을 포착하기 위해 설계되었다.

핵심 구조는 코드 아레나(code arena) 토너먼트다. 2개 이상의 언어 모델 에이전트가
다중 라운드 토너먼트에서 경쟁한다. 각 라운드는 두 단계로 구성된다. 편집 단계에서
에이전트들이 자신의 코드베이스를 자유롭게 수정하고, 경쟁 단계에서 수정된 코드베이스를
맞붙인다. 중요한 점은 언어 모델이 게임을 직접 플레이하지 않는다는 것이다. 에이전트의
코드가 경쟁 대리자(competitive proxy) 역할을 한다. 가장 많은 라운드를 이기는
에이전트가 승리한다.

지원하는 아레나는 6가지다: Halite(자원 수집), Poker, CoreWar(메모리 전쟁),
RobotRumble, Robocode(로봇 전투), BattleSnake. 현재 8개 모델이 6개 아레나에서
25,200 라운드를 경쟁한 결과, Claude Sonnet 4.5가 ELO 1385로 1위를 차지했으며
GPT-5, o3가 뒤를 잇는다.

연구에서 드러난 주요 발견은 세 가지다. 모델들이 반복적으로 개선하는 것을 어려워한다는
점, 일부 과제에서 인간 솔루션이 여전히 최상위 모델을 크게 앞선다는 점, 코드베이스가
빠르게 기술 부채(technical debt)를 쌓는다는 점이다.

## 분석

### 과제 지향 vs 목표 지향의 철학적 전환

HumanEval은 “이 함수를 구현하라”고 묻는다. SWE-bench는 “이 GitHub 이슈를 해결하라”고
묻는다. 둘 다 무엇을 해야 하는지가 명확하다. CodeClash는 “이 게임에서 이겨라”고만
한다. 어떻게 이길지, 무엇을 구현할지, 전략을 어떻게 바꿀지는 에이전트 몫이다.
이 차이는 단순한 난이도 차이가 아니라 평가 패러다임의 전환이다. 정해진 정답이
없는 문제에서 AI를 평가한다.

### 코드베이스를 경쟁 대리자로 쓰는 설계

에이전트가 게임을 직접 플레이하지 않고 코드로 대리시키는 구조는 영리하다. 언어 모델의
추론 능력이 아니라 코딩 능력과 전략 설계 능력을 분리해서 측정할 수 있기 때문이다.
게임 결과는 수천 번의 시뮬레이션으로 통계적으로 안정된 신호를 만든다. 단일 테스트
실행이나 judge 모델 평가보다 노이즈가 훨씬 적다. ELO 시스템을 도입한 것도 이
맥락이다. 절대적 점수가 아니라 상대적 강도를 측정한다.

### 기술 부채 축적이라는 발견

“코드베이스가 빠르게 기술 부채를 쌓는다”는 발견은 AI 코딩 에이전트의 현재 한계를
압축적으로 드러낸다. 단기 성과를 위해 장기 유지보수성을 희생하는 경향이 모델에게도
나타난다. 라운드가 진행될수록 초기의 깔끔한 코드가 점점 얽히고, 새 전략을 추가하기
어려워지는 상황이 관찰된다. 이는 현재의 언어 모델이 코드를 생성하는 능력과 코드를
장기적으로 진화시키는 능력이 다름을 시사한다.

### SWE-bench 생태계 확장

CodeClash는 SWE-bench/SWE-agent와 같은 팀의 작품이다. 이 팀은 GitHub 이슈 해결
(SWE-bench) → 에이전트 구현(SWE-agent, mini-swe-agent) → 목표 지향 경쟁(CodeClash)으로
평가 범위를 체계적으로 확장하고 있다. 각 프로젝트가 이전 프로젝트의 한계를 식별하고
다음 벤치마크의 설계 근거로 삼는 구조다.

## 비평

### 강점

평가 설계의 독창성이 돋보인다. 수천 번 시뮬레이션을 통한 ELO 산출은 통계적으로
견고하다. 정해진 정답이 없는 경쟁 설정은 모델 오버피팅을 구조적으로 억제한다.
코드베이스 진화 과정을 추적하는 궤적 뷰어(viewer.codeclash.ai)를 함께 공개한 점도
연구 투명성을 높인다. 2,000개 이상의 토너먼트 결과를 공개했다.

벤치마크가 게임 아레나를 사용한다는 점도 설득력 있다. 게임은 목표가 명확하고,
승패 판정이 객관적이며, 무한히 시뮬레이션할 수 있다. 인간이 설계한 테스트 케이스에
의존하지 않아도 된다.

### 약점 및 한계

6가지 아레나가 소프트웨어 엔지니어링의 실제 스펙트럼을 대표하는지 의문이 든다.
Halite, CoreWar, BattleSnake 같은 경쟁 게임에서 잘하는 능력이 실제 비즈니스
소프트웨어를 개발하는 능력과 얼마나 연관이 있는지는 별도의 검증이 필요하다.
게임 아레나는 규칙이 고정되어 있고, 실제 소프트웨어 개발은 요구사항 자체가
유동적이다.

계산 비용도 진입 장벽이다. 25,200 라운드를 돌리는 것은 상당한 API 비용과 컴퓨팅
자원을 요구한다. Docker 환경이 필수이고 Ubuntu 22.04에서 개발·테스트되었다는
제약도 있다. 개인 연구자나 소규모 팀이 독자적으로 새 모델을 평가하기 어렵다.

또한 ELO 리더보드의 시점 의존성도 고려해야 한다. Claude Sonnet 4.5가 1위를
차지했지만, 모델 버전이 빠르게 바뀌는 현 상황에서 수개월 전 결과가 현재 모델
성능을 대변하지 않을 수 있다. 벤치마크 업데이트 주기가 모델 출시 주기를 따라가기
어렵다는 구조적 문제다.

## 인사이트

### AI 코딩 평가는 “정답이 있는 문제”에서 “목표가 있는 경쟁”으로 이동한다

소프트웨어 엔지니어링 벤치마크의 역사는 측정 대상의 점진적 확장으로 읽힌다.
HumanEval은 함수 구현이라는 가장 작은 단위를 측정했다. SWE-bench는 GitHub 이슈
해결이라는 실제 버그 수정 작업으로 확장했다. CodeClash는 장기적 코드베이스 진화와
전략적 경쟁으로 다시 한 번 확장한다.

이 확장의 방향은 “테스트 케이스가 있는 문제”에서 “테스트 케이스가 없는 문제”로의
이동이다. 단위 테스트를 통과하는 것은 명세를 충족했다는 증거지만, 경쟁에서 이기는
것은 전략적 판단과 반복적 개선 능력의 증거다. 전자는 모델이 주어진 프레임 안에서
작동하는지를 묻고, 후자는 모델이 프레임 자체를 구성할 수 있는지를 묻는다.

이 패러다임 전환은 AI 코딩 도구를 평가하는 방식을 근본적으로 바꿀 수 있다. 현재
기업들이 “우리 제품이 SWE-bench에서 X%다”라고 마케팅하는 것처럼, 향후에는
“우리 에이전트가 목표 지향 벤치마크에서 어떤 ELO를 가지는가”가 중요한 지표가
될 수 있다. 단, 그 벤치마크가 실제 소프트웨어 개발 능력을 대변한다는 신뢰가
먼저 구축되어야 한다.

### “반복적 개선의 실패”가 드러내는 언어 모델의 구조적 한계

모델들이 라운드를 거듭해도 코드베이스를 효과적으로 개선하지 못한다는 발견은
표면적으로는 에이전트 성능의 문제처럼 보인다. 그러나 더 깊이 들여다보면 언어 모델의
설계 방식에 관한 근본적인 질문이 된다.

현재의 언어 모델은 주어진 컨텍스트에서 다음 토큰을 예측하도록 훈련된다. 장기적인
목표를 추적하고, 과거 시도의 실패 원인을 분석하며, 전략을 체계적으로 수정하는 능력은
이 훈련 목표에서 자연스럽게 나오지 않는다. CodeClash에서 에이전트가 매 라운드 새로운
로그를 받아도 개선이 느린 것은 “장기 피드백 학습(long-horizon feedback learning)”이
현재 모델에게 취약한 영역임을 보여준다.

기술 부채 축적은 이 한계의 또 다른 표현이다. 단기적으로 승리 확률을 높이는 코드를
추가하면서 전체 구조의 일관성을 훼손한다. 이는 인간 개발자가 마감 압박 아래 빠른
픽스를 반복할 때 나타나는 패턴과 흡사하다. 차이는 인간은 이 패턴을 의식하고 리팩토링
시점을 판단할 수 있다는 것이다. 현재 언어 모델에게 이 메타 인지적 판단은 아직
도전적인 영역이다.

### 벤치마크 설계 자체가 AI 발전의 방향을 형성한다

Goodhart의 법칙(“측정 지표가 목표가 되면 측정 지표로서의 가치를 잃는다”)은 AI
벤치마크에서 반복적으로 실증된다. HumanEval 점수는 빠르게 포화되었고, SWE-bench는
이미 특정 접근법에 최적화된 에이전트들로 리더보드가 채워지고 있다. 벤치마크가
공개된 순간부터 그것을 향한 최적화가 시작된다.

CodeClash의 경쟁 구조는 이 문제를 부분적으로 완화한다. 에이전트들이 서로를 상대로
경쟁하기 때문에, 특정 정적인 기준을 향해 과적합하는 것이 어렵다. 상대가 발전하면
나도 발전해야 한다. 이는 AlphaGo/AlphaZero가 자기 대국(self-play)을 통해 인간의
한계를 넘어선 방식과 구조적으로 유사하다. 코딩 에이전트 분야에서 이 자기 경쟁
구조가 실질적인 능력 향상으로 이어질지는 흥미로운 미래 연구 질문이다.

더 근본적으로, 어떤 벤치마크가 주목받는가가 연구 커뮤니티의 자원 배분을 결정한다.
SWE-bench 등장 이후 GitHub 이슈 해결 능력이 집중적으로 연구된 것처럼, CodeClash가
주목받을수록 장기 코드베이스 진화와 전략적 경쟁 능력이 연구 우선순위로 올라갈 것이다.
벤치마크는 단순히 현재를 측정하는 도구가 아니라, 미래의 AI 능력 발전 방향을 설정하는
선택 압력(selection pressure)이기도 하다.
