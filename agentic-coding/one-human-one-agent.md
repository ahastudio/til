# One Human + One Agent = One Browser

1인간 + 1에이전트가 수백 에이전트 스웜을 압도한 실험.

원문: <https://emsh.cat/one-human-one-agent-one-browser/>

HN 토론: <https://news.ycombinator.com/item?id=46779522> (324점, 153개 댓글)

## 무엇을 만들었나

LLM 에이전트 하나와 인간 하나가 협업해 **72시간 만에 2만 줄 Rust 코드로
브라우저를 만들었다**.

- HTML/CSS 렌더링
- 서드파티 라이브러리 의존성 제로
- Linux, macOS, Windows 지원
- 스크롤, 뒤로가기, 회귀 테스트

훌륭하진 않다. 좋다고도 못하겠다. 하지만 **작동한다**.

## Cursor 스웜과 비교

Cursor는 GPT-5.2 기반 수백 개 에이전트를 수주간 돌려 브라우저를 만드는 실험을
했다.

|           | 1인간 + 1에이전트 | Cursor 스웜 |
| --------: | :---------------: | :---------: |
|  에이전트 |        1개        |   수백 개   |
|      기간 |      72시간       |    수주     |
| 작동 여부 |      작동함       |  빌드 실패  |
|      비용 |       낮음        |  매우 높음  |

Cursor는 “의미 있는 진전”이라 주장했지만, **컴파일되는 바이너리나 렌더링 결과를
보여주지 못했다**.

## 왜 이런 결과가 나왔나

### 인간이 조정하면 빠르다

인간이 전략을 맡고, 에이전트가 실행에 집중하면 **조정 오버헤드 없이** 빠르게
진행된다. Planner, Worker, Judge 같은 복잡한 역할 분리가 필요 없다.

[멀티 에이전트 오케스트레이션은 왜 잘 안 되는가](multi-agent-orchestration-problems.md)가
분류한 맥락 붕괴(41.8%)와 유령 위임(36.9%)이 정확히 Cursor 스웜이 빌드
실패에 도달한 메커니즘이다.
토큰 5~10배 비용을 들이고도 결과물을 못 낸 사례가 이 실험으로 가시화됐다.

### 병렬화는 조정 비용을 동반한다

에이전트를 늘리면 충돌도 늘어난다. Cursor는 충돌이 적다고 주장했지만 빌드조차 안
됐다. 1:1 협업은 조정 비용이 제로라서 **작동하는 결과물**이 나왔다.

[에이전틱 소프트웨어의 5단계](agentic-software-levels.md)가 “Level 4(멀티
에이전트)로 건너뛰지 말고 Level 1에서 시작하라”고 권한 것의 극단적 실증
사례다.
저자가 “인상적으로 보이는 아키텍처”의 함정을 경고한 그 자리에 Cursor 스웜이
빠졌다.

### 작동이 먼저다

“완벽한” 대신 “작동하는”을 목표로 잡았다. 기능을 점진적으로 추가하면서 회귀
테스트를 즉시 도입해 안정성을 확보했다.

## 실천 지침

1. **에이전트 늘리기 전에 인간 역할부터 명확히 하라.** 스웜은 비용만 높이고
   결과를 보장하지 않는다.

2. **빠르게 작동하는 결과물을 만들어라.** 수주간 “진전”을 주장하면서 작동 증거를
   못 내면 실패다.

3. **단순한 구조를 유지하라.** 1인간 + 1에이전트 루프가 복잡한 계층 구조보다
   효과적이다.

[tmux와 마크다운 스펙으로 4~8개 병렬 코딩 에이전트 운영하기](parallel-coding-agents-with-tmux.md)가
정확히 이 “인간 오케스트레이터” 원칙을 받아들여 4~8개로 상한을 둔다.
스웜이 아니라 인간 인지가 감당할 수 있는 좁은 병렬화가 실제로 작동하는
경계다.

## HN 반응

HN에서 simonw(Simon Willison)는 이 프로젝트를 Cursor의 FastRender보다 훨씬 나은 데모로 평가했다.[^simonw]
코드 크기(20K LOC vs ~1.6M), 의존성 제로, 코드 가독성 면에서 앞선다는 것이다.
자신의 블로그를 렌더링한 스크린샷도 공유하며 레이아웃과 CSS 그라디언트 처리가 인상적이었다고 밝혔다.
“브라우저를 만드는 것이 대규모 병렬 에이전트 설정을 증명할 완벽한 과제라고 생각했는데 내가 틀렸다”고 인정했다.

반면 QuadmasterXLII는 실제 렌더링을 테스트했을 때 링크가 항상 파란색도 아니고 밑줄도 없으며
렌더링이 혼란스러웠다고 비판한다.[^QuadmasterXLII]
“브라우저라고 주장하면 브라우저 기준으로 평가받아야 한다”는 것이다.

fabrice_d는 더 근본적인 비판을 제기한다.[^fabrice_d]
“우리는 브라우저와 거리가 매우 멀다. 기본 렌더러 작성은 사실 그렇게 어렵지 않다.
이것은 70년대부터 쓰여온 수많은 그래픽 툴킷과 비슷한 것이다.”
보다 의미 있는 도전으로 WPT 테스트를 통과시키는 Servo 포크를 제안했다.

jacquesm는 반대로 과정에 초점을 맞춘다.[^jacquesm]
“이 포스트가 같은 주제의 다른 글들보다 훨씬 흥미로운 것은 만들어진 것이 아니라 어떻게 만들어졌는지 때문이다.”

## 핵심 메시지

1인간 + 1에이전트가 수백 에이전트 스웜보다 나은 결과를 냈다면, “에이전트를 더
투입하면 스케일링된다”는 가설은 틀렸다.

**에이전트 스케일링의 정답은 수량이 아니라 협업 품질이다.**
