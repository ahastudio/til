# Artificial Analysis — Coding Agent 벤치마크

<https://artificialanalysis.ai/agents/coding-agents>

## 소개

Artificial Analysis Coding Agent 리더보드는 AI 코딩 에이전트를
“실제 세계 성능(real-world performance)”의 관점에서 평가하는 독립 벤치마크 플랫폼이다.
비용, 토큰 소비, 실행 시간을 성능 지표와 함께 측정하여 단순 점수 순위를 넘어서려는 것이 목적이다.

## 지표 구조

### 복합 인덱스(Composite Index)

핵심 지표는 세 가지 벤치마크의 단순 평균 pass@1이다.
각 벤치마크는 3회 실행의 평균값을 사용한다.

구성 벤치마크:

| 벤치마크           | 과제 수 | 제작      | 특성                          |
| ------------------ | ------- | --------- | ----------------------------- |
| DeepSWE            | 113개   | Datacurve | 소프트웨어 구현 및 버그 수정  |
| Terminal-Bench v2  |  84개   | Laude Institute | 에이전트 셸 환경 탐색   |
| SWE-Atlas-QnA      | 124개   | Scale AI  | 기술 질의응답                 |

총 321개 과제에 걸쳐 에이전트를 평가한다.
채점 방식은 과제 유형에 따라 이진(binary) 또는 부분 점수(partial credit) 방식을 사용한다.

### 비용 및 효율 지표

비용은 API per-token 가격 기준이며, 캐시 쓰기 비용까지 포함한다.
토큰 소비는 세 범주로 분리 보고된다:
입력 토큰(프롬프트·컨텍스트), 캐시 재사용 토큰, 출력 토큰.

실행 시간은 벽시계 기준(wall-clock) 에이전트 런타임이며
추론, 도구 호출, 파일 조작, 모델 레이턴시를 모두 포함하되
환경 설정(environment setup)과 하네스 오버헤드는 제외한다.

캐시 적중률은 “제공자 라우팅(provider routing)”과 백엔드 복제본 상태 불일치로 인해
실행마다 달라질 수 있다고 명시한다.

### 하네스 비교

동일 모델(Claude Opus 4.7)을 사용하면서 코딩 에이전트 하네스만 달리한 비교를 포함한다.

| 하네스      | 기반 모델        |
| ----------- | ---------------- |
| Claude Code | Claude Opus 4.7  |
| Cursor CLI  | Claude Opus 4.7  |
| Opencode    | Claude Opus 4.7  |

모델 능력을 고정하고 도구·인터페이스 차이를 분리 측정하는 구조다.
16개 모델(평가 대상 23개 중)의 결과가 공개되어 있다.

## 분석

### 3개 벤치마크를 단순 평균하는 것이 합성하는 것과 동일하지 않다

DeepSWE, Terminal-Bench v2, SWE-Atlas-QnA는 서로 다른 능력을 측정한다.
DeepSWE는 코드베이스 내 구현과 버그 수정 능력, Terminal-Bench는 셸 환경에서의 에이전트 자율성,
SWE-Atlas-QnA는 기술 질의응답 능력을 각각 테스트한다.
이 세 가지를 단순 평균하면, 한 영역에서 탁월하지만 다른 두 영역에서 평범한 모델이
세 영역에서 고르게 좋은 모델과 동일한 인덱스 점수를 받을 수 있다.

Artificial Analysis 자체도 “인덱스 점수가 높다고 해서 모든 워크플로에서 최선인 것은 아니다”라고 명시하며,
“복합 인덱스는 벤치마크별 차트와 함께 해석해야 한다”고 안내한다.
그러나 리더보드가 단일 숫자로 모델을 순위 매기는 한, 사용자의 시선은 그 숫자에 집중된다.
인덱스 해석에 필요한 세부 데이터는 인터랙티브 차트 뒤에 숨어 있어
단순 비교에 사용되기 쉬운 구조다.

과제 수 분포도 이 불균형을 키운다.
SWE-Atlas-QnA가 124개 과제로 가장 많고, DeepSWE가 113개, Terminal-Bench가 84개다.
세 벤치마크를 동등하게 가중치를 두고 평균하지 않는다면
과제 수가 많은 벤치마크가 인덱스에 더 큰 영향을 미친다.
단순 평균 방식이 이 불균형을 어떻게 처리하는지는 명시적이지 않다.

### 하네스 비교가 드러내는 “도구 마찰(tool friction)” 측정 시도

동일 모델을 세 가지 하네스로 실행한다는 설계는 방법론적으로 중요한 선택이다.
대부분의 벤치마크는 모델을 평가하고 도구는 고정한다.
Artificial Analysis는 반대로 모델을 고정하고 도구를 변수로 만들었다.

이 접근은 “Claude Code로 실행한 Opus 4.7”과 “Cursor CLI로 실행한 Opus 4.7” 사이의 성능 차이가
모델 능력이 아니라 도구 설계에서 나온다는 것을 직접 측정한다.
코딩 에이전트 플랫폼을 선택하는 개발팀에게 이 데이터는 유용하다.
같은 모델을 쓰더라도 어떤 하네스를 사용하느냐에 따라 결과가 달라진다면,
모델 선택만큼 도구 선택이 중요한 결정이 된다.

그러나 이 비교는 Claude Opus 4.7 하나만을 기반으로 한다.
다른 모델(GPT-5.5, Gemini, Kimi 등)을 세 하네스로 실행했을 때 같은 패턴이 나타날지는 알 수 없다.
특정 하네스가 특정 모델과 시너지를 낸다면, 단일 모델 기반 하네스 비교는 그 시너지를 놓친다.

### 캐시 적중률 불안정성이 비용 비교를 불확실하게 만든다

비용 지표의 신뢰성을 가장 크게 위협하는 요인은 캐시 적중률 변동이다.
입력 토큰을 캐시로 재사용할 수 있는지 여부는 실행 비용에 직접 영향을 미친다.
캐시 쓰기 비용까지 포함한다는 것은 장점이지만,
캐시 적중률이 “제공자 라우팅과 복제본 상태 불일치”로 달라진다면
비교 기준이 안정적이지 않다.

벤치마크가 보고하는 비용이 특정 실행 환경에서 측정된 것이라면,
사용자의 실제 환경(다른 지역, 다른 시간대, 다른 사용량 패턴)에서는 다른 비용이 발생할 수 있다.
“API per-token 공개 가격”이라는 기준점은 투명하지만,
캐시 적중률 차이로 인한 실제 비용 편차가 얼마나 될 수 있는지가 공개되지 않으면
비용 순위도 불확실하다.

## 비평

### 3회 평균 pass@1이 충분한 통계적 안정성을 보장하는지 불명확하다

각 벤치마크는 3회 실행의 평균 pass@1을 사용한다.
확률적으로 동작하는 AI 에이전트에서 3회 실행은 통계적으로 취약하다.
특히 DeepSWE 113개, Terminal-Bench 84개처럼 과제 수가 제한된 경우,
개별 실행 간 점수 변동이 클 수 있다.

3회 평균이 측정 노이즈를 얼마나 줄이는지에 대한 분석이 공개되지 않는다.
신뢰 구간이나 표준편차 없이 소수점 한 자리 점수를 순위 기준으로 제시하면,
사용자는 그 차이를 실질적인 성능 차이로 받아들이게 된다.
1~2% 포인트 차이로 순위가 갈리는 리더보드에서 이 문제는 결론에 영향을 미친다.

### 독립 벤치마크를 조합한 구조가 각 벤치마크의 품질 문제를 흡수한다

Artificial Analysis의 인덱스는 세 외부 벤치마크(Datacurve, Laude Institute, Scale AI)를 조합한다.
이 설계는 중립성 측면에서 장점이 있다. 그러나 각 구성 벤치마크의 품질 문제가
인덱스에 그대로 흡수된다는 약점이 있다.

DeepSWE가 특정 모델 가족의 훈련 데이터와 분포가 겹친다면,
그 모델은 DeepSWE에서 과적합된 성능을 낼 수 있고, 인덱스 전체를 끌어올린다.
Terminal-Bench가 특정 셸 환경이나 운영체제에 치우쳐 있다면,
그 환경에서 훈련된 모델이 유리하다.
Artificial Analysis가 구성 벤치마크를 직접 설계하지 않았기 때문에
이 품질 문제를 통제하기 어렵다.

### 상업 서비스 평가 플랫폼과 독립성 사이의 구조적 긴장

Artificial Analysis는 AI 모델 평가 서비스를 판매하는 기업이다.
독립 벤치마크로 권위를 확보하고, 그 권위를 바탕으로 기업 고객에게 유료 서비스를 제공하는 구조다.
이것이 곧 편향을 의미하지는 않는다. 그러나 평가 지표 선택, 포함·제외 모델 결정,
강조하는 수치 등이 잠재적으로 플랫폼의 상업적 이해와 얽힐 수 있다.

“16개 모델(23개 평가 대상 중)”이라는 언급은 7개 모델이 리더보드에 공개되지 않는다는 것을 의미한다.
어떤 모델이 빠졌고 왜 빠졌는지가 명시되지 않는다.
결과가 좋지 않아서 제외된 것인지, 협력 관계 때문인지, 기술적 이유인지 알 수 없다.
독립 벤치마크라면 모든 결과를 공개하거나, 제외 기준을 명시해야 한다.

## 인사이트

### 독립 평가 플랫폼의 등장이 모델 제공업체의 자체 벤치마크를 압박한다

CursorBench, Senior SWE-Bench처럼 서비스 제공업체가 직접 운영하는 벤치마크와 달리,
Artificial Analysis는 제3자 독립 평가를 표방한다.
이 독립성은 시장에서 가치를 갖는다.
자사 모델이 자사 벤치마크에서 잘 나온다는 비판을 받는 제공업체들에게
제3자 평가 결과는 다른 무게를 갖기 때문이다.

역설적으로 이 구조는 새로운 종류의 경쟁을 만든다.
모델 제공업체들은 이제 자사 벤치마크뿐 아니라 제3자 벤치마크에서도 좋은 결과를 내야 한다.
제3자 벤치마크가 표준으로 자리잡으면, 그 벤치마크에 특화된 최적화 압력이 생긴다.
“독립 벤치마크”도 Goodhart's Law를 피할 수 없다.
독립성이 권위가 되는 순간, 그 권위를 목표로 한 최적화가 시작된다.

### 하네스 비교가 코딩 에이전트 시장의 새로운 경쟁 레이어를 드러낸다

동일 모델로 세 가지 하네스를 비교하는 방법론은 중요한 시장 신호를 포함한다.
Claude Code, Cursor CLI, Opencode 모두 같은 Claude Opus 4.7을 쓰지만 성능이 다를 수 있다.
이것이 사실이라면, 에이전트 코딩 시장의 경쟁은 모델 성능 경쟁과 하네스 성능 경쟁 두 레이어로 분리된다.

하네스 경쟁은 코드 에디터, IDE 통합, 컨텍스트 주입 방식, 도구 호출 설계 같은 엔지니어링 선택의 경쟁이다.
같은 모델을 더 잘 활용하는 하네스를 만드는 것은 모델을 더 좋게 만드는 것과 다른 종류의 혁신이다.
이 분리는 스타트업에게 기회를 의미한다.
최고 모델에 API 접근권을 가진 모든 플레이어가 경쟁할 수 있는 레이어가 생기기 때문이다.

장기적으로 하네스 경쟁이 성숙하면, 하네스가 상품화되는 국면이 온다.
어떤 하네스를 쓰든 성능 차이가 미미해지는 시점에 경쟁은 다시 모델로 돌아온다.
IDE 통합과 사용자 경험이 차별화 요소로 남는 한, 하네스 경쟁은 당분간 의미 있는 레이어다.

### 비용·시간·정확성 3축 동시 측정이 요구하는 트레이드오프 인식

단일 점수 리더보드와 달리 Artificial Analysis는 성능, 비용, 실행 시간을 동시에 제공한다.
이 세 지표는 독립적으로 움직이지 않는다.
더 많은 추론 스텝을 밟는 에이전트는 더 나은 성능을 내지만 시간과 비용이 늘어난다.
캐시를 적극 활용하는 에이전트는 비용을 줄이지만 캐시 관리 오버헤드가 발생한다.

Senior SWE-Bench에서 Claude Opus 4.8이 117,100 출력 토큰으로 24%를 달성하고
GPT-5.5가 36,300 토큰으로 16%를 달성한 것처럼,
비용 대비 성능의 트레이드오프는 모델마다 전략이 다르다.
3축 동시 측정이 제공하는 것은 “어떤 모델이 최선인가”가 아니라
“어떤 조건에서 어떤 모델이 최선인가”를 판단할 데이터다.

이 데이터를 실제로 활용하려면 팀의 의사결정 프레임이 바뀌어야 한다.
“가장 점수가 높은 모델을 쓴다”는 단순 규칙 대신,
“이 과제의 실패 비용이 높은가, 시간이 제약인가, 예산이 제약인가”를 먼저 묻고
그에 따라 최적 모델-하네스 조합을 선택하는 결정 구조가 필요하다.

### 복합 인덱스는 시장의 단순화 욕구와 실제 복잡성 사이에서 타협한다

AI 에이전트 성능을 하나의 숫자로 표현하려는 충동은 이해할 수 있다.
기업 의사결정자는 36개 모델의 3개 벤치마크 × 4개 지표를 모두 읽을 시간이 없다.
Artificial Analysis의 복합 인덱스는 이 욕구에 응답한다.
그러나 복잡성을 단순화할 때는 항상 무언가가 지워진다.

지워지는 것은 워크플로 특수성이다.
터미널 작업이 많은 DevOps 팀에게 Terminal-Bench 성능이 중요하고,
코드베이스 구현이 핵심인 제품 팀에게는 DeepSWE가 더 중요하다.
이 두 팀에게 최선인 모델은 다를 수 있는데, 복합 인덱스는 이 차이를 지운다.

벤치마크 플랫폼이 성숙하는 방향은 두 가지다.
더 정교한 단일 인덱스를 만드는 방향과, 워크플로별 특화 인덱스를 제공하는 방향이다.
전자는 채택이 쉽지만 정보 손실이 크고, 후자는 정확하지만 사용하기 복잡하다.
이 긴장은 데이터의 문제가 아니라 사용자 인터페이스 설계의 문제다.
