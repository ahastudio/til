# 프롬프트에서 하네스까지 - AI 에이전틱 패턴 4년의 기록

> 원문: <https://bits-bytes-nn.github.io/insights/agentic-ai/2026/04/05/evolution-of-ai-agentic-patterns.html>

## 요약

2022년부터 2026년까지 AI 에이전트 개발 패러다임이 세 단계로 진화했다는 분석이다.
각 시대의 실패가 다음 시대를 촉발했으며, 글의 중심 논지는 "엄밀함은 사라지지 않았다.
이동했을 뿐(Rigor doesn't disappear — it relocates)"이다.

**1단계 - 프롬프트 엔지니어링 (2022-2024)**: 핵심 질문은 "어떤 말을 해야 하나?"였다.
Chain-of-Thought, ReAct 같은 기법이 발전했고, 정밀하게 작성된 지시문이 성패를
가른다고 믿었다. 한계는 명확했다. 프롬프트가 소비하는 컨텍스트가 불완전하면 실패했고,
프롬프트는 컨텍스트 창 밖의 정보에 접근할 수 없었다.

**2단계 - 컨텍스트 엔지니어링 (2025)**: 핵심 질문이 "어떤 정보를 넣어야 하나?"로
이동했다. KV-캐시 최적화와 구조화된 컨텍스트 관리가 강조됐고, Cursor 같은 도구들이
전체 코드베이스를 이해하는 RAG 기반 시맨틱 검색으로 멀티파일 편집을 구현했다.
그러나 완벽한 컨텍스트도 시스템 설계 자체가 잘못되면 무용지물임이 드러났다.

**3단계 - 하네스 엔지니어링 (2026~)**: 핵심 질문은 "어떤 시스템을 만들어야 하나?"다.
에이전트는 모델과 하네스의 합으로 정의된다. 하네스는 가드레일, 검증, 에러 복구를
담당하는 구조 전체다. 바이브 코딩(Vibe Coding) 시기에 노출된 보안 취약점과 유지보수
문제들이 체계적 하네스 설계의 필요성을 촉발했다.

Fowler/Böckeler가 제시한 2×2 프레임워크는 하네스의 구성 요소를 정리한다.
결정론적 피드포워드(가이드, 컨벤션), 결정론적 피드백(컴파일러, 린터, 타입 체커),
비결정론적 피드포워드(시스템 프롬프트, 지시문), 비결정론적 피드백(LLM-as-a-judge,
코드 리뷰) 네 영역이 하네스를 구성한다.

보안 측면에서 Meta AI의 "Rule of Two"가 제시됐다. 에이전트가 신뢰할 수 없는 입력,
민감한 데이터 접근, 상태 변경 세 가지를 동시에 가지면 반드시 사람의 승인이 필요하다는
원칙이다.

## 분석

### 단계 전환의 구조적 패턴

각 단계의 전환은 동일한 메커니즘으로 작동한다. 현재 추상화 수준에서 해결할 수 없는
실패가 반복되면, 엔지니어들은 한 단계 위의 추상화로 이동해 그곳에서 엄밀함을 재확립한다.
이 패턴은 소프트웨어 공학 역사에서 반복됐다. 어셈블리 → 고급 언어, 절차적 → 객체지향,
단일 서버 → 마이크로서비스로의 이동이 같은 구조다.

### 하네스 2×2 프레임워크의 의미

Fowler/Böckeler의 분류는 기존의 "AI 제어"를 인간 소프트웨어 엔지니어링의 언어로
재해석한다. 컴파일러와 린터를 "결정론적 피드백 하네스"로 보면, 이미 수십 년간 사용된
도구들이 에이전트 하네스의 일부가 된다. 새로운 발명이 아니라 기존 엔지니어링 규율의
에이전트 영역으로의 확장이다.

### 바이브 코딩과 하네스의 관계

바이브 코딩의 "숙취(hangover)"는 하네스 엔지니어링의 직접적인 계기다. AI가 생성한
코드의 의존성이 깊어지고 유지보수 부담이 쌓이자, 엄밀함이 코드 작성에서 시스템 아키텍처로
이동해야 한다는 인식이 확산됐다.

## 비평

### 강점

3단계 프레임워크는 개념적으로 깔끔하고 역사적 흐름을 일관된 논리로 설명한다.
Chad Fowler의 "Relocating Rigor" 개념 차용은 적확하며, 단순한 유행어 나열이 아니라
소프트웨어 공학의 긴 역사 안에 AI 에이전트를 위치시키는 시도가 돋보인다. Fowler/Böckeler
의 2×2 프레임워크와 Meta의 Rule of Two 같은 구체적 도구를 제시한 점도 실용성을 높인다.

### 약점

단계 경계가 인위적이다. 현실에서는 2025년에도 프롬프트 엔지니어링이 지배적이었고,
하네스 엔지니어링이라는 용어가 2026년에 갑자기 등장한 것도 아니다. 사후적으로 정리된
역사화라는 점을 감안해야 한다.

"에이전트가 실수하면 하네스를 고쳐라"는 원칙이 강력하지만, 모델 실패와 하네스 실패를
실제로 어떻게 구분해 진단하는지에 대한 실용적 가이드는 부재하다. 2×2 프레임워크도
분류 체계로는 유용하나, 어느 셀에 무엇을 배치할지 판단하는 기준은 제시하지 않는다.

## 인사이트

### 엔지니어링 무게중심은 항상 이동한다

소프트웨어 개발에서 비즈니스 로직이 데이터베이스 → 애플리케이션 → API → 프론트엔드로
이동했듯이, AI 에이전트에서는 제어 로직이 프롬프트 → 컨텍스트 → 하네스로 이동하고 있다.
이 이동의 법칙은 단순하다. 현재 레이어가 복잡해질수록 다음 레이어가 그 복잡도를 흡수하는
더 나은 추상화가 된다.

모델 자체는 점차 유틸리티 레이어가 되고, 그것을 감싸는 구조가 차별화 요소가 된다.
이것은 클라우드 인프라와 같은 경로다. AWS가 일반화되자 인프라 엔지니어링보다 그 위의
아키텍처 설계가 더 중요해졌다. 모델 성능이 상향 평준화될수록 하네스 설계가 에이전트
품질을 결정하는 주된 변수가 된다.

중요한 점은 각 단계가 이전 단계를 대체하지 않는다는 것이다. 프롬프트 엔지니어링은
하네스 엔지니어링의 서브모듈로 여전히 작동한다. 엄밀함이 이동할 뿐, 소멸하지 않는다.

### 하네스는 AI 버전의 포카요케다

"고장나면 프레임을 고쳐라"는 사고방식은 도요타 생산 방식의 근본 원칙과 닮아 있다.
개별 작업자의 실수를 교육으로 해결하는 대신, 지그와 고정구(jig and fixture)를 바꿔
실수가 구조적으로 불가능한 환경을 만드는 포카요케(poka-yoke) 접근법이다.

프롬프트 시대의 "AI에게 더 잘 설명하기"는 작업자 교육과 같다. 반복 실패가 누적되면
결국 시스템 자체를 고치는 쪽이 훨씬 효율적이라는 결론에 도달하게 된다. 하네스
엔지니어링은 그 전환점을 에이전트 영역에서 공식화한 것이다.

이 관점에서 보면 Meta의 Rule of Two도 다르게 읽힌다. 세 가지 위험 요소를 동시에
갖지 못하도록 아키텍처 수준에서 제약하는 것은, 고위험 작업을 안전하게 만드는 구조적
장치다. 개별 에이전트에게 "조심해"라고 지시하는 것보다 구조가 조심하게 만드는 것이다.

### 이름 붙이기가 패러다임을 만든다

각 시기에 이름을 붙이는 행위 자체가 커뮤니티의 사고를 구조화한다. "컨텍스트 엔지니어링"
이라는 단어가 생기자 그 전까지 산재해 있던 RAG, 메모리, 툴 호출 논의들이 하나의 프레임
아래 통합됐다. 이름은 흩어진 실천을 하나의 전문 분야로 응집시킨다.

"하네스 엔지니어링"이라는 명명이 같은 역할을 할 것인지가 관전 포인트다. AGENTS.md,
Claude Code harness, Cursor rules 같은 실천들이 이미 존재하지만 통일된 이름이 없었다.
이름이 정착되면 교육 과정, 도구, 직무 명세서가 그 이름 주위로 결집한다.

반대 가능성도 있다. "하네스 엔지니어링"이 너무 구현 특정적인 용어라서 더 추상적인
"에이전트 시스템 설계"나 "AI 아키텍처"에 흡수될 수도 있다. 이름 경쟁의 승자는
커뮤니티가 결정한다.

### 메타 하네스: 하네스를 만드는 에이전트

하네스를 설계하는 일 자체가 에이전트에게 위임되기 시작하면, 재귀적 구조가 등장한다.
하네스를 작성하는 에이전트, 하네스를 테스트하는 에이전트, 하네스를 최적화하는 에이전트가
계층적으로 쌓이는 구조다.

이 지점에서 인간의 역할은 목표 정의와 최종 검증으로 수렴한다. 각 레이어의 하네스를
인간이 직접 설계하는 대신, 상위 에이전트가 하위 에이전트의 하네스를 생성하고 관리하는
구조가 된다. 이것이 실현되면 "하네스 엔지니어링"이라는 단어의 주어가 인간에서 에이전트로
바뀌는 순간이 온다.

엄밀함의 이동이 계속된다면, 다음 단계는 메타 하네스를 설계하는 메타 메타 하네스가 될
것이다. 그 끝이 어디인지는 아직 아무도 모른다.
