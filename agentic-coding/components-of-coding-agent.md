# 코딩 에이전트의 구성 요소

원문: <https://magazine.sebastianraschka.com/p/components-of-a-coding-agent>

## 요약

Sebastian Raschka가 코딩 에이전트와 에이전트 하네스(harness)의 전반적 설계를
분석한 글이다.
Claude Code, Codex CLI 같은 도구들이 LLM을 애플리케이션 계층으로 감싸
코딩 작업을 효과적으로 만드는 구조를 설명한다.

먼저 LLM, 추론 모델(reasoning model), 에이전트를 구분한다.
LLM은 다음 토큰 예측 모델이고,
추론 모델은 중간 추론에 더 많은 계산을 투자하도록 훈련된 LLM이며,
에이전트는 모델을 환경 내에서 반복적으로 호출하는 제어 루프다.

핵심은 여섯 가지 구성 요소다:

1. **라이브 저장소 문맥(Live Repo Context)**: Git 상태, 브랜치, 프로젝트 문서 파악.
   안정적 사실을 작업 요약으로 미리 수집.
2. **프롬프트 형태와 캐시 재사용**: 도구 설명과 작업 요약처럼 자주 변하지 않는
   안정적 프롬프트 접두사를 캐시하고, 사용자 요청과 최근 기록만 업데이트.
3. **도구 접근 및 사용**: 임의 명령이 아니라 경계가 명확한 사전정의 도구 목록 제공.
   도구 유효성, 인수 검증, 사용자 승인, 경로 제한을 하네스가 검증.
4. **문맥 팽창 최소화**: 클리핑(긴 출력 단축)과
   전사 축소(세션 기록을 요약으로 변환)의 두 가지 압축 전략.
   최근 이벤트는 풍부하게, 오래된 이벤트는 공격적으로 압축.
5. **구조화된 세션 메모리**: 작업 메모리(작은 추출 상태)와
   전체 전사(JSON으로 저장된 완전한 기록)의 두 계층 분리.
6. **경계가 설정된 하위 에이전트를 통한 위임**: 부수 작업을 하위 에이전트로 분할.
   충분한 문맥을 상속하되 더 엄격한 경계 내에서 실행.

저자의 핵심 주장은 "좋은 코딩 에이전트 설계의 과소평가된 측면은 문맥 품질"이다.
겉으로는 모델 성능처럼 보이는 것이 실제로는 문맥 관리의 결과일 수 있다.

## 분석

### 모델이 아니라 하네스가 차이를 만든다

이 글의 가장 중요한 주장은 마지막에 나온다:
"최신 LLM의 기본 버전들이 유사한 능력을 가지고 있기 때문에,
차이점은 소프트웨어 아키텍처에서 비롯된다."
모델 경쟁이 수렴하는 시점에서 경쟁 우위는 하네스 설계로 이동한다.
Claude Code와 Codex CLI의 차이는 Claude와 GPT의 차이보다
감싸는 소프트웨어의 차이에서 더 많이 나온다.

### 여섯 가지 구성 요소의 계층 관계

Raschka가 나열한 여섯 가지는 동등한 목록이 아니라 계층 구조를 이룬다.
라이브 저장소 문맥과 프롬프트 캐시는 입력 단계,
도구 접근은 실행 단계,
문맥 팽창 최소화와 세션 메모리는 상태 관리 단계,
하위 에이전트 위임은 조직 단계다.
이 순서가 곧 에이전트의 라이프사이클이다.

### 문맥 압축의 두 전략

클리핑과 전사 축소의 구분이 실용적이다.
클리핑은 공간적 압축(한 시점의 출력을 줄임)이고,
전사 축소는 시간적 압축(과거 기록을 줄임)이다.
"최근은 풍부하게, 과거는 공격적으로"라는 원칙은
인간의 기억 구조(작업 기억 vs 장기 기억)와 닮아 있다.

## 비평

### 실증 근거의 부족

글은 아키텍처 패턴을 잘 정리하지만,
각 구성 요소가 실제로 얼마나 성능에 기여하는지 정량적 근거가 없다.
"문맥 품질이 과소평가되었다"는 주장은 직관적으로 설득력 있지만,
프롬프트 캐시를 쓸 때와 안 쓸 때의 성능 차이,
문맥 압축 전략별 정확도 비교 같은 데이터가 있으면 더 강해질 것이다.

### 모델 특화 설계의 부재

여섯 가지 구성 요소는 모델에 독립적(model-agnostic)으로 서술되었다.
하지만 실제로는 모델마다 최적의 하네스 설계가 다를 수 있다.
예를 들어 컨텍스트 윈도우가 큰 모델에서는
문맥 팽창 최소화의 중요성이 줄어들고,
도구 호출 포맷이 다른 모델에서는 도구 접근 레이어가 달라져야 한다.
범용 아키텍처와 모델 특화 최적화의 긴장 관계를 다루지 않은 것이 아쉽다.

### 하위 에이전트 위임의 얕은 다룸

여섯 번째 구성 요소인 하위 에이전트 위임이 가장 복잡한 주제인데
가장 짧게 다뤄졌다.
문맥 상속의 범위, 하위 에이전트 간 통신,
실패 시 복구 전략 같은 실질적 도전이 빠져 있다.
이 부분은 별도의 글로 다룰 만한 깊이가 있다.

## 인사이트

### 코딩 에이전트는 운영체제의 재발명이다

Raschka의 여섯 가지 구성 요소를 운영체제에 대응시키면 구조가 선명해진다.
라이브 저장소 문맥은 파일 시스템,
도구 접근은 시스템 콜 인터페이스,
문맥 팽창 최소화는 메모리 관리,
세션 메모리는 프로세스 상태,
하위 에이전트 위임은 프로세스 스케줄링이다.
코딩 에이전트의 하네스는 사실상 LLM을 위한 운영체제다.
이 유비가 성립한다면, OS 역사에서 검증된 패턴들 —
가상 메모리, 페이지 교체 알고리즘, 프로세스 격리 —이
에이전트 설계에 직접 적용 가능하다는 뜻이다.
"최근은 풍부하게, 과거는 공격적으로" 압축하는 전략은
LRU 캐시 교체 정책의 변형이다.

### "문맥 품질"은 새로운 경쟁 변수다

모델 성능이 수렴하면 경쟁은 다른 차원으로 이동한다.
Raschka의 "문맥 품질" 개념은 그 차원을 명명한 것이다.
같은 모델이라도 어떤 문맥을 어떤 순서로 어떤 형태로 넣느냐에 따라
출력이 극적으로 달라진다.
이는 검색 엔진 역사의 교훈과 닮아 있다.
1990년대 검색 엔진들은 같은 웹을 크롤링했지만
인덱싱과 랭킹 알고리즘의 차이가 Google을 만들었다.
코딩 에이전트에서 문맥 관리는 검색에서의 랭킹 알고리즘이다.
모델이 같아도 문맥 품질이 다르면 결과가 다르다.

### 프롬프트 캐시는 에이전트 경제학의 핵심이다

두 번째 구성 요소(프롬프트 형태와 캐시 재사용)는
기술적 최적화로 보이지만 실제로는 경제적 결정이다.
코딩 에이전트는 반복적으로 모델을 호출하므로
턴당 비용이 세션 비용을 결정한다.
안정적 프롬프트 접두사를 캐시하면 토큰 소비가 줄고,
이는 곧 사용자당 마진에 직결된다.
에이전트 서비스의 수익성은 모델 성능이 아니라
캐시 적중률(cache hit rate)에 의해 결정될 수 있다.
이는 CDN이 웹 서비스의 경제학을 바꾼 것과 같은 패턴이다.
