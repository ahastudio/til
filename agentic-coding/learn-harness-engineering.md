# Learn Harness Engineering

<https://github.com/walkinglabs/learn-harness-engineering>

## 소개

“모델은 똑똑하다. 하네스가 모델을 신뢰할 수 있게 만든다.”
이 한 문장이 이 과정 전체의 핵심이다.
`walkinglabs`가 공개한 하네스 엔지니어링 입문 과정으로, 별 3,988개를 받았다.
한국어를 포함한 6개 언어로 번역됐으며, 12개 강의, 6개 프로젝트, 템플릿 라이브러리로 구성된다.

저자는 Anthropic의 통제 실험을 핵심 증거로 제시한다.
동일한 모델(Opus 4.5), 동일한 프롬프트(“2D 레트로 게임 에디터를 만들어라”).
하네스 없이 실행했을 때: $9, 20분, 작동하지 않는 결과물.
하네스(플래너 + 제너레이터 + 이밸류에이터) 적용 후: $200, 6시간, 실제로 플레이 가능한 게임.
모델은 바뀌지 않았다. 하네스가 달랐다.

하네스는 다섯 가지 서브시스템으로 구성된다.
**Instructions**는 에이전트가 무엇을 어떤 순서로 해야 하는지 정의한다. 하나의 거대한 파일이 아니라 에이전트가 필요할 때 탐색하는 점진적 공개(progressive disclosure) 구조다.
**State**는 무엇이 완료됐고 무엇이 진행 중이며 다음이 무엇인지 추적한다. 디스크에 저장되어 다음 세션이 정확히 이전 세션이 멈춘 곳에서 시작한다.
**Verification**은 테스트 통과만을 완료의 증거로 인정한다. 에이전트는 실행 가능한 증거 없이 “완료”를 선언할 수 없다.
**Scope**는 한 번에 하나의 기능으로 제한한다. 범위 초과, 세 가지를 반쯤 완료하는 행위, 피처 리스트 수정으로 미완료 작업을 숨기는 행위를 막는다.
**Session Lifecycle**은 시작 시 초기화하고 종료 시 정리한다. 다음 세션을 위한 깨끗한 재시작 경로를 남긴다.

프로젝트 파일 구조는 단순하다.

```text
YOUR PROJECT ROOT
├── AGENTS.md              ← 에이전트 운영 매뉴얼
├── CLAUDE.md              ← (Claude Code 사용 시 대안)
├── init.sh                ← 설치 + 검증 + 시작 실행
├── feature_list.json      ← 피처 목록과 완료 상태
├── claude-progress.md     ← 세션별 진행 기록
└── src/                   ← 실제 코드
```

과정의 학습 경로는 6개 페이즈 12주 분량이다.
문제 인식(1) → 저장소 구조화(2) → 세션 연결(3) → 피드백과 범위 설정(4) → 검증(5) → 완전한 하네스 구축(6).
캡스톤 프로젝트는 Electron 기반의 개인 지식 베이스 데스크톱 앱으로, 문서 임포트, 라이브러리 관리, 인덱싱, AI 기반 Q&A, 인용을 포함한 답변 생성 기능을 에이전트로 구축한다.

## 분석

### “하네스 문제 vs 모델 문제” 프레임의 의미

이 과정의 핵심 전략은 에이전트 실패의 원인을 모델 성능에서 하네스 설계로 재귀속시키는 것이다.
“AI가 실패했다”는 진단 대신 “하네스가 충분하지 않았다”는 진단을 제시한다.
이 프레임 전환은 실용적으로 중요하다. 모델은 개발자가 통제할 수 없지만, 하네스는 설계할 수 있다.

Anthropic 실험의 비교($9/broken vs $200/working)는 이 주장을 극적으로 보여준다.
비용이 20배 이상 늘었지만 결과가 질적으로 달라졌다는 사실은, 하네스 투자가 단순한 비용이 아니라 신뢰성 확보를 위한 엔지니어링임을 보여준다.
중요한 것은 모델 비용이 아니라 엔지니어가 청소에 쓰는 숨겨진 비용이다.
[Anthropic의 장기 실행 하네스 글](./harness-design-long-running-apps.md)이 이 실험의 원본을 다루는데, 거기서는 같은 데이터를 비판적으로 재해석한다 — 22배의 지출이 22배의 가치를 낳았는지 입증할 객관적 척도가 부족하다는 지적이다. 두 글을 함께 읽으면 비용 정당화 논쟁의 양면이 드러난다.

### 다섯 서브시스템의 상호 의존성

이 과정이 다섯 서브시스템을 독립적으로 가르치지 않고 단계적으로 쌓아 올리는 이유가 있다.
각 서브시스템은 다른 것들의 전제다.
Verification 없는 Scope는 에이전트가 “완료”를 자의적으로 선언하게 한다.
State 없는 Session Lifecycle은 세션이 끊기면 컨텍스트를 잃는다.
Instructions 없는 State는 추적할 목표가 없다.
이 의존성 때문에 하네스를 부분적으로만 구현하면 효과가 제한적이다.

### 저장소로서의 하네스 파일

`AGENTS.md`, `feature_list.json`, `claude-progress.md`, `init.sh`를 프로젝트 루트에 두는 설계는 하네스를 코드베이스의 일부로 만든다.
이것은 중요한 설계 결정이다. 하네스 파일이 외부 서비스나 프롬프트가 아니라 저장소 안에 있으면, 버전 관리가 되고, 팀원과 공유되고, CI/CD와 통합된다.
에이전트가 읽는 파일과 개발자가 읽는 파일이 같다. 문서화와 에이전트 제어가 동일한 파일에서 일어난다.

## 비평

### 강점: 단계적 구조와 실증적 증거

6개 페이즈가 논리적으로 쌓이고, 각 페이즈가 이전 것의 약점을 명시적으로 해결한다는 점에서 커리큘럼 설계가 탄탄하다.
Anthropic의 실험 데이터를 핵심 증거로 활용한다는 점도 설득력을 높인다.
캡스톤 프로젝트가 각 과제 단계의 해결책에서 다음 단계의 시작점으로 이어지는 진화적 구조도 주목할 만하다.

### 약점: 비용과 복잡성의 현실적 트레이드오프

$200/6시간짜리 완성된 게임이 하네스의 가치를 보여주는 동시에 그 비용도 드러낸다.
모든 프로젝트에 완전한 하네스를 적용하는 것이 합리적인지는 프로젝트의 규모와 반복 빈도에 달려 있다.
간단한 일회성 스크립트에 `feature_list.json`과 `claude-progress.md`를 만드는 것은 오버엔지니어링이다.
과정이 “언제 하네스를 쓸 것인가”보다 “어떻게 쓸 것인가”에 집중하기 때문에, 적용 범위 판단의 기준이 부족하다.
[하네스 엔지니어링 백과사전](./harness-engineering-encyclopedia.md)이 제시한 “얇은 하네스 vs 두꺼운 하네스” 분류와 “고위험 작업은 두껍게, 탐색은 얇게”라는 판단 기준이 이 빈자리를 부분적으로 메운다.

### 보완할 시각: 팀 환경에서의 하네스

이 과정은 주로 1인 개발자 또는 단일 에이전트 세션을 전제한다.
팀 환경에서 여러 개발자가 각자의 에이전트를 사용하면서 동일한 하네스 파일을 공유할 때 어떤 충돌이 발생하는지, `claude-progress.md`의 병합 충돌을 어떻게 처리하는지, `feature_list.json`의 상태 동기화는 어떻게 하는지에 대한 논의가 없다.
하네스가 팀 도구로 진화하려면 이 문제들이 해결되어야 한다.

## 인사이트

### 컨텍스트 윈도우 관리가 곧 엔지니어링이다

에이전트가 멀티세션에서 실패하는 근본 원인 중 하나는 컨텍스트 손실이다.
대화 기반 인터페이스는 이전 세션의 컨텍스트를 자동으로 보존하지 않는다.
`claude-progress.md`와 `feature_list.json`은 컨텍스트를 언어 모델의 메모리가 아닌 파일 시스템에 외부화하는 해법이다.
이것은 컴퓨터 과학의 고전적 패턴이다. 신뢰할 수 없는 메모리를 신뢰할 수 있는 저장소로 보완한다.

이 패턴의 함의는 넓다. 에이전트 엔지니어링의 본질적인 과제 중 하나는 “어떻게 에이전트의 작업 상태를 세션을 넘어 보존하는가”다.
이 과정은 파일 기반 상태 관리라는 단순하지만 강력한 해법을 제시한다.
복잡한 벡터 데이터베이스나 메모리 시스템이 아니라, 에이전트가 읽을 수 있는 마크다운 파일이 상태의 진실 원천(source of truth)이 된다.
이 단순성이 오히려 유지 관리 가능성과 디버깅 용이성을 높인다.

### “완료”의 정의가 시스템의 신뢰성을 결정한다

에이전트가 “완료”를 선언하는 방식은 하네스 설계의 가장 중요한 결정 중 하나다.
검증 없는 완료 선언은 에이전트가 사용자에게 신뢰할 수 없는 결과를 신뢰할 수 있는 것처럼 포장하게 만든다.
이것이 “더 많은 시간 청소에 쓴다”는 악순환의 시작이다.

소프트웨어 공학에서 “완료의 정의(Definition of Done)”는 애자일 방법론의 핵심 개념이다.
팀이 “완료”의 기준을 공유하지 않으면 각자가 다른 기준으로 작업을 완료하고, 통합 시 충돌이 발생한다.
에이전트에게 같은 문제가 존재한다. 에이전트는 자신이 완료했다고 생각하는 것을 완료로 선언한다.
하네스의 Verification 서브시스템은 에이전트의 “완료” 기준을 외부에서 정의하고 강제한다.
테스트를 통과해야 완료다. 린트를 통과해야 완료다. 타입 체크를 통과해야 완료다.
이 강제가 에이전트의 자기 평가와 실제 품질 사이의 간격을 닫는다.
[하네스 엔지니어링 책](./harness-engineering-book.md)은 이 강제를 한 단계 더 끌어올려, validation.md를 Generator보다 먼저 Evaluator가 작성하고 Sprint 시작 시 잠근다 — “완료”의 정의가 구현보다 시간상 앞서 고정되는 구조다.

### 하네스는 기술 부채의 성격을 바꾼다

하네스 없는 에이전트 코딩이 만드는 기술 부채는 측정하기 어렵다.
에이전트가 어떤 결정을 왜 했는지 추적할 수 없고, 세션이 끊기면 맥락이 사라지고, “완료”됐지만 실제로는 작동하지 않는 코드가 축적된다.
이것은 전통적 기술 부채와 다르다. 전통적 기술 부채는 개발자가 빠른 해결책을 선택했다는 인식이 있다. 에이전트 기술 부채는 에이전트가 무엇을 했는지 아무도 정확히 모르는 상태로 쌓인다.

하네스는 이 부채의 성격을 바꾼다.
`claude-progress.md`가 있으면 각 세션에서 무슨 일이 있었는지 추적할 수 있다.
`feature_list.json`이 있으면 어떤 기능이 완료됐고 어떤 것이 미완료인지 명확하다.
Verification이 있으면 “완료”됐다는 주장을 검증할 수 있다.
하네스는 에이전트의 작업을 가시화한다. 가시화된 부채는 관리할 수 있다. 보이지 않는 부채는 관리할 수 없다.
이것이 하네스가 단기적 비용 증가($9 → $200)를 감수할 만한 장기적 이유다.
