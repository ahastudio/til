# 무신사 AI 네이티브 엔지니어 채용 3부작

원문:
[Part 1 — The Philosophy](https://techblog.musinsa.com/c002c2775b3a)
[Part 2 — The Machine](https://techblog.musinsa.com/c2345aab5b7a)
[Part 3 — The Human](https://techblog.musinsa.com/bccc190c9c93)

## 요약

무신사는 2026년 1월, 수백 명의 지원자를 AI 파이프라인으로 처음부터 끝까지 자동
평가하는 AI 네이티브 엔지니어 채용을 진행했다. 문제 설계, 평가 파이프라인,
그 바탕이 되는 철학을 직접 만든 과정을 3부작으로 공개했다.

**Part 1 — The Philosophy**는 “AI를 쓰지 못하게 하려면?”이 아니라 “AI를 진짜 잘
쓰는 사람을 어떻게 찾을 것인가?”라는 질문에서 출발한다. AI 에이전트가 LeetCode
medium을 몇 초 만에 푸는 시대에 기존 코딩 테스트는 의미를 잃었다. 구현이
범용화되면서 병목은 “무엇을 구현할지 아는 것”으로 이동했다.

과제는 대학교 수강신청 시스템이다. “정원이 1명 남은 강좌에 100명이 동시에
신청해도 정확히 1명만 성공해야 한다”는 동시성 요구사항만 명시하고, 락 메커니즘·
성능 요구사항·API 엔드포인트는 일절 지정하지 않았다. 모호함 자체가 테스트다.
3-Tier 평가 모델(Make it Work → Basic Features → Deep Thought)을 설계하고, 모든
지원자에게 동등한 AI 환경을 제공하기 위해 OpenAI와 협업해 무료 에이전트 토큰을
지급했다.

**Part 2 — The Machine**은 400여 명을 처리하는 자동화 파이프라인 구축기다. 한
지원자당 2시간이면 800시간 — 직접 코드를 짜는 대신, 마크다운 지침서를 읽고
실행하는 범용 AI 에이전트를 오케스트레이터로 삼았다. 후보자마다 독립 서브
에이전트를 생성해 컨텍스트 오염을 막는 멀티 에이전트 아키텍처. Security Gate →
Preflight Gate → Quality Gate → Functional Gate → Scoring → Report로 이어지는
7단계 파이프라인을 Markdown으로 구현했다. AI 채점은 파일:라인 수준의 코드 증거를
의무화했다. 빌드 실패율 약 15%를 낮추기 위해 15개 Dockerfile 템플릿을 준비했고,
기능(Base) 100점 + 깊이(Depth) 120점의 220점 체계를 확립했다. Ace / Craftsman /
Hustler / Thinker / Contender / Rookie / Incomplete — 7단계 랭크가 점수의 크기가
아니라 패턴을 반영한다. 채점 모델 17번 변경, 26번 초기화, 36일 1,952 커밋.

**Part 3 — The Human**은 머신이 분류한 100명 넘는 후보자와 20명 이상의 면접관이
5일 동안 마주 앉은 이야기다. AI가 후보자 코드를 분석해 파일:라인 기반 맞춤
면접 질문을 자동 생성했다. 그러나 가장 흥미로운 발견은 같은 랭크(Hustler)로
분류된 사람들이 면접에서 극단적으로 갈라졌다는 것이다. Depth 64점을 받은
후보자가 경력자 면접에서도 보기 드문 수준의 동시성 사고를 보여줬고, 반대로 기능은
완벽하지만 자기 코드의 동작 원리를 설명하지 못하는 사람도 있었다. “코드는 의도의
그림자”라는 인식과 함께 Context Engineering의 중요성을 되새긴다.

## 분석

### AI로 AI 사용자를 평가하는 재귀 구조

이 채용의 가장 선명한 특징은 자기 참조적 구조다. AI 코딩 에이전트를 잘 쓰는
사람을 평가하기 위해, AI 에이전트가 평가 파이프라인을 실행한다. 후보자가 AI를
어떻게 썼는지를 AI가 채점한다. 평가하는 측과 평가받는 측이 같은 도구를 쓴다.

이 구조는 우연이 아니다. Part 2의 핵심 설계 원칙 — “에이전트를 만드는 게 아니라
에이전트가 따를 지침을 설계하는 것” — 이 평가 대상에게도 그대로 적용된다.
후보자가 에이전트에게 무엇을 시켰는가(프롬프트), 에이전트가 따를 지침을
어떻게 설계했는가(에이전트 지침) — 이 두 영역이 합산 35점으로 배점되어 있다.
평가자와 피평가자가 같은 역량을 발휘하고 있는 것이다.

### 모호함의 정밀한 설계

“모호함 자체가 테스트”라는 명제는 단순히 스펙을 덜 쓴 게 아니다. 무엇을
만들지는 알지만(수강신청 시스템) 어떻게는 지원자가 도출해야 하는 구조다.
완전 모호(“유용한 걸 만들어라”)는 신호가 너무 적고, 반쪽 모호(“이 엔드포인트로
REST API를 만들어라”)는 AI가 사고를 건너뛸 수 있다. Sweet spot은 “모호한
요구사항 + 명확한 결과 기대치”다.

이중 락 문제가 이 설계의 효과를 잘 보여준다. 강좌 레벨 락만 구현한 사람은
문제의 절반만 본 것이다. 같은 학생이 동시에 두 요청을 보낼 때 학점 제한이
깨지는 시나리오 — 학생 레벨 직렬화의 필요성 — 를 스스로 발견한 사람이
다섯 명 중 한 명이 채 안 됐다. 스펙이 알려주지 않을 때 무엇을 해야 하는지
파악할 수 있는가, 이것이 “AI 네이티브”와 “AI 의존”의 분기점이다.

### Markdown as Code와 하네스 엔지니어링

평가 파이프라인 전체가 마크다운 지침서로 구현된 것은 주목할 만하다. 코드 컴파일
없이, 수정 즉시 반영된다. Part 2에서 “하네스 엔지니어링”이라 명명한 이 접근 —
에이전트의 능력이 향할 방향을 설계하고 반복 조율하는 것 — 은 소프트웨어 개발의
새로운 레이어다. 코드를 고치지 않고 지침을 수정하면 코드가 다시 만들어진다.

하지만 이 유연함에는 대가가 있다. AI 출력의 비결정성을 JSON Schema의 strict
모드(additionalProperties: false)와 최대 5회 재시도 루프로 제어해야 했다.
마크다운의 자유도와 JSON Schema의 엄격함이라는 긴장이 하네스의 골격을 이룬다.

### Duck Typing의 철학적 확장

Part 1과 Part 2 모두 Duck Typing 원리를 명시적으로 언급한다. 기능 검증에서 “동작하는
시스템처럼 응답하면 동작하는 시스템”이라는 원칙 — 내부 구현이 pessimistic lock이든
optimistic lock이든 결과가 정확하면 된다. 더 나아가 “더 좋은 AI 모델이나 날카로운
프롬프트로 이중 락 인사이트에 도달했다면, 그것도 깊은 사고”라고 규정한다.
도달했다는 사실이 곧 역량이다.

그러나 Part 3에서 이 원칙의 한계가 드러난다. 코드만 보고는 코드 뒤에 누구의
사고가 있는지 알 수 없다. 한 번의 제출물이 역량의 재현이 아니라 AI의 우연일 수
있다. Duck Typing이 기능 검증에서는 합리적이지만, 사람의 역량 판단에서는
충분하지 않다.

## 비평

### 강점: 선제적 실험과 공정성의 설계

동결(freeze)로 대응하는 업계 트렌드 — 빅테크 신입 채용 절반 감소, 국내 IT
신입 공채 67% 감소 — 속에서 먼저 정의하고 먼저 움직인 것은 의미 있는 선택이다.
특히 AI 도구 접근성의 자본 격차를 인식하고 OpenAI와 협업해 모든 지원자에게
동등한 환경을 제공한 것은, 시험의 공정성에 대한 진지한 고민의 산물이다.

캘리브레이션 루프의 설계도 견고하다. 사람이 개별 점수를 검증하는 것이 아니라
구조적 편향 패턴을 발견하고 루브릭 자체를 수정한다. 한 명의 점수가 아니라
400명의 채점 기준을 고치는 것. 머신과 사람의 역할이 명확히 분리되어 있다.

### 약점: 설계 문서 평가의 미흡

Part 3에서 스스로 인정한 가장 큰 맹점이다. “대화를 평가하면서 설계를 놓쳤다.”
프롬프트는 대화 중 휘발된다. 설계 문서 — 왜 이 아키텍처를 선택했는지, 어떤
트레이드오프를 고려했는지 — 는 명시적 문맥으로 남는다. 코드는 의도의 그림자이고,
우리는 그림자를 채점하고 있었다는 고백은 솔직하다. 그러나 이것이 파이프라인을
운영하면서 자연스럽게 하고 있던 일(하네스 엔지니어링)을 채용 평가에 적용하지
못한 실기(失機)라는 점에서 아쉬움이 남는다.

### 약점: 재현성과 단면 평가의 한계

한 번의 3시간 제출물로 역량을 판단하는 것의 구조적 한계는 시리즈 전체에 걸쳐
반복적으로 언급되지만, 해결책은 제시되지 않는다. 특히 AI 에이전트가 우연히
좋은 코드를 만들어준 경우와 후보자가 깊이 이해하고 만든 경우를 기술적으로
구별하는 문제는 면접으로 보완했다고 하지만, 그 면접 역시 AI가 만든 가이드를
AI가 생성한 맥락 안에서 진행된다. 재귀의 탈출구가 완전히 열려 있지 않다.

## 인사이트

### 구현 비용이 0에 수렴할 때 희소해지는 것

Andrew Ng이 언급한 엔지니어 대 PM 비율의 붕괴(8:1 → 1:1)는 단순한 헤드카운트
변화가 아니다. 구현이 범용화되면서 “무엇을 만들지 아는 것”의 가치가 상대적으로
상승한다는 경제적 명제다. 무신사의 과제 설계는 이 명제를 채용 시험으로 구현하려
한 시도다. 명확한 결과 기대치(정원 초과 없이 정확히 1명 성공)는 유지하되,
메커니즘(락 전략)과 요구사항 도출(엣지 케이스, NFR)은 지원자에게 맡긴다.

이 설계가 의미하는 것은 더 깊다. 기존 코딩 테스트가 “구현 능력”을 측정했다면,
이 시험은 “구현이 무엇인지 정의하는 능력”을 측정한다. 시험의 단위가 알고리즘
문제에서 시스템 설계로, 시스템 설계에서 요구사항 도출로 이동한 것이다.
AI 에이전트가 구현을 담당할수록 엔지니어의 가치는 요구사항의 전방으로 이동한다.

이 역설이 가장 날카롭게 드러나는 것은 데이터 설계 함정이다. 학생 10,000명을
만들라는 요구사항에서 “이 데이터로 실제 뭘 할 건지”를 염두에 두고 설계한 사람과
기술적으로만 유효한 데이터를 만든 사람의 차이는, 구현의 문제가 아니라 목적의
이해 문제다. AI가 코드를 만들수록, 코드를 만드는 이유를 이해하는 능력이 더욱
희소해진다.

### 하네스 엔지니어링은 소프트웨어 개발의 새로운 추상화 계층

Markdown as Code로 구현된 평가 파이프라인은 단순한 구현 선택이 아니다. 에이전트의
능력이 이미 주어진 환경에서, 그 능력이 향할 방향을 설계하는 것이 주된 작업이
되는 패러다임의 선언이다. 코드 컴파일 대신 루브릭 수정, 배포 대신 지침서
업데이트. 개발 생산성이 “에이전트를 잘 쓰는 것”에서 “에이전트가 잘 쓰도록
지시하는 것”으로 이동하고 있다.

이 계층의 등장은 소프트웨어 개발의 역사적 추상화 진화와 같은 결을 공유한다.
기계어 → 어셈블리 → 고수준 언어 → 프레임워크 → 클라우드 서비스의 각 단계에서
인간은 한 수준 위로 올라가 “무엇을”에 집중하고 “어떻게”를 기계에 위임했다.
하네스 엔지니어링은 그 다음 계층 — “에이전트에게 무엇을 시킬 것인가” — 이다.

주목할 것은 이 계층이 반드시 Markdown일 필요가 없다는 점이다. JSON, YAML,
자연어, 코드 자체도 모두 지침서가 될 수 있다. 중요한 것은 지침과 실행의 분리다.
지침을 수정하면 코드가 재생성된다. 이 루프의 속도가 곧 하네스 엔지니어의
생산성이다. 무신사 팀이 26번 초기화를 감당할 수 있었던 것은 이 루프가 충분히
빠르도록 설계되어 있었기 때문이다.

### 평가 시스템의 재귀적 아이러니와 그 함의

Part 3의 가장 불편한 질문은 면접관의 자기 참조에서 나온다. AI가 만든 코드를
이해 없이 제출한 후보자를 걸러내려 했는데, 면접관도 AI가 만든 가이드를 따르고
있다. 도구를 이해하며 쓰는 것과 그저 따르는 것 사이의 경계를 후보자에게 묻고
있으면서, 면접관 자신도 같은 경계 위에 서 있다.

이 아이러니는 AI 시대 모든 조직에 일반화될 수 있다. AI가 만든 계획을 따르는 PM,
AI가 생성한 전략을 실행하는 리더, AI가 제안한 아키텍처를 채택하는 엔지니어 —
모두 같은 질문 앞에 선다. 도구가 강력해질수록, 도구를 쓰는 사람의 판단이 결과를
가른다는 명제가 자기 참조적으로 적용된다. 판단 능력 자체도 AI에게 아웃소싱하는
순간, 우리는 무엇을 남기는가.

무신사 채용은 이 질문에 명확한 답을 내리지 않는다. 대신 그 경계가 현재 중요하며,
그 중요함이 얼마나 오래 갈지는 모른다고 솔직하게 인정한다. 네비게이션의 20년
비유는 이 모호함을 잘 포착한다. 경로 계획이 사람의 문제가 아니게 되는 데 20년
걸렸다. 코드 구현이 사람의 문제가 아니게 되는 시간이 얼마나 될지, 지금은 알 수
없다.
