# Dirac — 토큰 효율 극대화 오픈소스 코딩 에이전트

<https://github.com/dirac-run/dirac>

## 소개

Dirac은 코딩 에이전트의 “비용 대비 성과”를 최우선 목표로 삼아 설계된 오픈소스
에이전트다. Cline을 포크(fork)하여 TypeScript로 구현되었으며, 컨텍스트를 얼마나
정밀하게 유지하느냐가 모델의 추론 품질과 직결된다는 전제에서 출발한다. 849개의
GitHub 스타를 받으며 2026년 4월 초 공개 직후부터 주목을 받았다.

Terminal-Bench-2 리더보드에서 `gemini-3-flash-preview`로 65.2%를 기록하며 1위에
올랐다. Google의 공식 기준선(47.6%)과 최상위 클로즈드 소스 에이전트인 Junie
CLI(64.3%)를 모두 앞질렀다. 8개 복잡한 리팩터링 태스크 벤치마크에서 100% 성공률을
달성했고 평균 비용은 $0.18로, 경쟁 에이전트 평균($0.51)보다 64.8% 저렴했다.

## 주요 기법

### Hash-Anchored Edits

기존 코딩 에이전트들이 줄 번호(line number)를 기준으로 편집 위치를 지정하는 데
반해, Dirac은 해시(hash)로 대상 줄을 고정한다. 코드베이스가 변경되어 줄 번호가
밀리더라도 해시 기반 앵커는 정확한 위치를 추적한다. “번역 과정의 손실(lost in
translation)” 문제를 구조적으로 차단한다.

### AST-Native Precision

TypeScript, Python, C++ 등 주요 언어의 문법 구조를 에이전트가 내부적으로 이해한다.
함수 추출이나 클래스 리팩터링 같은 구조적 조작을 텍스트 치환이 아닌 AST(Abstract
Syntax Tree) 수준에서 수행하므로 100% 정확도를 주장한다.

### Multi-File Batching

단일 LLM 라운드트립(roundtrip)으로 여러 파일을 읽고 편집할 수 있다. 파일마다
모델을 별도로 호출하는 방식 대비 레이턴시와 API 비용을 동시에 줄인다.

### High-Bandwidth Context

컨텍스트 크기를 최소화하되 관련성은 최대화하는 큐레이션 전략을 사용한다. 모델이
항상 가장 관련 있는 정보만 받도록 설계하여 긴 컨텍스트에서 발생하는 추론 품질
저하를 억제한다.

## 벤치마크

Cline, Kilo, Ohmypi, Opencode, Pimono, Roo와 비교한 8개 태스크 기준:

| 에이전트    | 성공 수 | 평균 비용 |
| ----------- | ------- | --------- |
| Cline       | 5/8     | $0.49     |
| Kilo        | 5/8     | $0.73     |
| Ohmypi      | 6/8     | $0.51     |
| Opencode    | 8/8     | $0.44     |
| Pimono      | 6/8     | $0.38     |
| Roo         | 6/8     | $0.60     |
| **Dirac**   | **8/8** | **$0.18** |

모든 태스크는 `gemini-3-flash-preview`(thinking: high)로 실행되었으며, huggingface/
transformers, microsoft/vscode, django/django 등 실제 공개 저장소를 대상으로 한다.

## 설치 및 사용

VS Code 익스텐션 또는 CLI로 설치한다.

```bash
npm install -g dirac-cli
dirac auth
dirac "Refactor this module to use async/await"
```

주요 CLI 옵션:

- `-p`: Plan Mode — 실행 전 전략 확인
- `-y`: Yolo Mode — 모든 액션 자동 승인
- `git diff | dirac “Review these changes”`: 표준 입력으로 컨텍스트 주입

## 분석

### 컨텍스트 압축과 비용 절감의 연결고리

Dirac의 비용 우위는 단순히 API를 적게 호출하는 것이 아니다. 컨텍스트 창을 작게
유지함으로써 입력 토큰 수 자체를 줄이는 동시에, 모델이 관련 없는 정보에 주의를
분산하지 않도록 해 성공률을 함께 높인다. 비용과 품질이 통상 트레이드오프 관계인
것을 두 지표 모두 개선한다는 점이 핵심 주장이다.

Hash-Anchored Edits와 AST 조작은 이 전략의 실행 수단이다. 줄 번호 기반 편집은
LLM에게 “어떤 줄을 어떻게 바꿔라”는 전체 맥락을 제공해야 하지만, 해시 기반
편집은 해시 하나로 위치를 특정하므로 컨텍스트 전달 오버헤드가 줄어든다. AST
레이어가 있으면 “함수 전체”를 LLM 응답에 포함할 필요 없이 구조적 식별자만으로
편집을 전달할 수 있다.

### Cline 포크 전략의 의미

Cline은 현재 가장 널리 사용되는 오픈소스 코딩 에이전트 중 하나로, 도구 생태계와
VS Code 통합이 잘 갖춰진 베이스다. Dirac이 바닥부터 새로 만들지 않고 Cline을
포크한 것은 “에이전트 아키텍처”보다 “실행 최적화”에 집중하겠다는 선택이다. 인터페이스
호환성과 사용자 친숙함을 그대로 가져오면서 내부 실행 엔진만 교체하는 접근이다.

이 전략은 Cline에서 발견된 버그(이슈 #10314)에 PR(#10315)을 제출하는 협력 관계로도
이어진다. 경쟁과 협력이 공존하는 오픈소스 생태계의 전형적인 패턴을 보인다.

### MCP 미지원 결정

“Native Tool Calling Only”를 명시하며 MCP(Model Context Protocol)를 지원하지 않는다.
이유는 신뢰성과 성능 극대화다. MCP는 범용 도구 인터페이스를 제공하지만, 그 범용성이
오버헤드를 만든다. Dirac은 모델의 네이티브 도구 호출만 쓰고 그 외 추상화 계층을
제거함으로써 토큰 효율성을 지킨다. 이는 에이전트 생태계 통합보다 단일 태스크 성능을
선택한 명시적 트레이드오프다.

## 비평

### 강점

벤치마크 설계가 신뢰도를 높인다. huggingface/transformers, microsoft/vscode,
django/django 같은 실제 대형 공개 저장소를 대상으로 삼고, 동일 조건(같은 모델,
같은 thinking 설정)에서 6개 경쟁 에이전트와 나란히 비교한다. 재현 가능성을 강조하며
diff 링크를 모두 공개한 점도 투명성 측면에서 긍정적이다.

비용 절감과 품질 향상을 동시에 달성한다는 주장이 데이터로 뒷받침된다. 일반적으로
비용을 낮추면 성능이 희생되기 쉬운데, Dirac의 벤치마크 수치는 반대 방향을 보인다.
이것이 방법론적으로 타당하다면 코딩 에이전트 비용 구조에 실질적 함의가 있다.

### 약점 및 한계

벤치마크 표본이 8개 태스크로 작다. 리팩터링 중심의 태스크 구성이 Dirac의 Hash-Anchored
Edits에 유리한 도메인일 가능성이 있다. 버그 수정, 신규 기능 추가, 테스트 작성 등
다른 유형의 태스크에서 동일한 우위가 유지되는지는 별도 검증이 필요하다.

MCP 미지원은 사용 사례를 제한한다. Cline 사용자 중 MCP 도구를 적극 활용하는 경우,
포크임에도 불구하고 Dirac으로의 전환이 불가능하다. “신뢰성 우선”이라는 설계 철학은
이해할 수 있지만, MCP 생태계와의 단절은 장기적으로 생태계 확장성의 걸림돌이 될 수 있다.

Cline에서 발견된 버그(이슈 #10314)가 벤치마크 비용 수치에 영향을 미쳤음을 README가
직접 인정한다. “약간의 차이”라고 하지만, 경쟁 에이전트와의 비교가 공정하게 이루어졌는지
검토가 남아 있다. 벤치마크 수정 결과가 공개될 때까지 정확한 수치는 잠정적이다.

## 인사이트

### “컨텍스트 큐레이션”이 에이전트 경쟁의 새 전선이다

2024년 코딩 에이전트 경쟁은 주로 “어떤 도구를 제공하는가”와 “어떤 모델을 쓰는가”로
판가름났다. Dirac은 세 번째 차원, “얼마나 정밀하게 컨텍스트를 관리하는가”가 비용과
성능을 동시에 결정하는 핵심 변수임을 실증한다. 같은 모델, 같은 thinking 설정에서도
컨텍스트 전달 방식 하나로 64.8%의 비용 차이가 생긴다.

이 통찰의 파급력은 단순한 비용 절감을 넘는다. LLM의 추론 품질이 컨텍스트 길이에
반비례한다는 연구 결과들이 누적되는 상황에서, 컨텍스트 압축 전략은 곧 모델 성능을
더 효율적으로 끌어내는 방법론이다. 더 강한 모델이 나올수록 컨텍스트 큐레이션의
절대적 가치는 높아진다. 모델이 강할수록 핵심 정보를 잘 받았을 때의 성능 향상 폭이
크기 때문이다.

코딩 에이전트 개발자 관점에서 이 트렌드는 명확한 방향을 제시한다. 다음 세대 에이전트
경쟁력은 프롬프트 품질이나 도구 다양성보다 “무엇을 컨텍스트에서 제거할 것인가”에
달려 있다. 최소 충분 컨텍스트(minimal sufficient context)의 설계가 에이전트
엔지니어링의 핵심 역량으로 부상하고 있다.

### Hash-Anchored Editing이 암시하는 편집의 패러다임 전환

줄 번호 기반 편집은 인간이 코드를 보는 방식을 그대로 모델에게 강요한다. 인간은
IDE에서 줄 번호를 본다. 하지만 LLM에게 줄 번호는 부스러지기 쉬운 좌표계다. 파일이
조금만 바뀌어도 의미가 달라진다.

해시 앵커는 다른 철학에서 출발한다. 코드의 내용 자체가 위치보다 안정적인 식별자라는
아이디어다. 이것은 Git이 콘텐츠 기반 해싱을 쓰는 이유와 같다. Git은 파일 경로가
아닌 콘텐츠의 SHA를 진실의 원천으로 삼는다. Dirac의 Hash-Anchored Edits는 이 원리를
코드 편집 인터페이스에 적용한 것이다.

AST 조작과 결합되면 이 패러다임은 더 강력해진다. 줄이 아닌 “함수”, “클래스”, “블록”
단위로 편집을 추상화하면, LLM이 지시해야 하는 정보량이 줄어들고 정확도가 올라간다.
이 방향이 성숙해지면 미래의 코딩 에이전트는 텍스트를 다루는 것이 아니라 구문 구조를
다루는 시스템이 될 것이다. IDE가 텍스트 편집기에서 리팩터링 도구로 진화한 것처럼,
에이전트도 같은 방향으로 진화하고 있다.

### 오픈소스 코딩 에이전트 생태계의 분화 압력

Dirac의 등장은 Cline 생태계에서 흥미로운 분화 신호다. 기반 코드를 공유하지만
MCP를 버리고 내부 최적화에 집중하는 포크가 단기간에 주목을 받는다는 것은, 사용자
그룹이 서로 다른 우선순위를 가지고 있음을 보여준다. 한쪽은 생태계 통합과 확장성을
원하고, 다른 쪽은 단일 태스크 성능과 비용 효율을 원한다.

이 분화는 오픈소스 에이전트 시장이 성숙기에 접어들면서 필연적으로 일어날 과정이다.
Linux 생태계에서 용도별 배포판이 나뉜 것처럼, 코딩 에이전트도 “범용 + 생태계 통합형”과
“특수 목적 + 성능 최적화형”으로 갈라질 가능성이 높다. Cline이 전자라면 Dirac은 후자의
초기 사례다.

개발자로서 주목해야 할 지점은 이 분화가 에이전트 선택의 기준을 복잡하게 만든다는
것이다. 이전에는 “어떤 에이전트가 더 좋은가”라는 질문에 단일 답이 있었지만, 이제는
“어떤 작업에, 어떤 예산으로, 어떤 생태계와 함께”라는 맥락에 따라 최적 에이전트가
달라진다. 에이전트 선택이 스택 선택처럼 아키텍처 결정의 영역으로 들어오고 있다.
