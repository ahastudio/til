# AI 스페셜리스트와 자동사냥 하네스로 제어하는 AI 파이프라인

원문:
<https://techblog.musinsa.com/6c578f8bd1fb>

관련 글:

- [무신사 AI 네이티브 채용 3부작](./musinsa-ai-native-hiring.md)

## 요약

무신사 테크 블로그가 AI 파이프라인 설계의 핵심 패턴인
“AI 스페셜리스트(AI Specialist)”와 “자동사냥 하네스(Auto-Hunting Harness)”를
소개한다.

무신사가 AI 네이티브 채용 파이프라인(400여 명 처리)을 구축하면서
축적한 설계 원칙을 정제한 글이다.
범용 AI 에이전트 하나가 모든 것을 처리하는 단일 에이전트 방식의
한계를 벗어나, 도메인별로 역할이 특화된 AI 스페셜리스트 에이전트를
자동사냥 하네스로 조율하는 아키텍처를 제안한다.

“자동사냥 하네스”는 MMORPG의 자동사냥 시스템에서 이름을 가져온
은유다. 오케스트레이터가 규칙을 설정하면, 각 스페셜리스트 에이전트가
주어진 역할을 반복 수행하고, 하네스가 전체 루프를 감시·교정한다.
Security Gate → Preflight Gate → Quality Gate → Functional Gate →
Scoring → Report의 단계별 파이프라인은 마크다운 지침서로 구현되어,
배포 없이 수정이 즉시 반영된다.

핵심 통찰은 “AI를 잘 쓰는 것”이 아니라
“AI가 잘 쓰이도록 환경을 설계하는 것”이다.
스페셜리스트 분리, 컨텍스트 격리, 결정론적 검증 루프의 세 원칙이
파이프라인의 신뢰성을 구성한다.

## 분석

### AI 스페셜리스트 분리의 설계 근거

범용 에이전트가 모든 역할을 맡으면 두 가지 문제가 생긴다.
컨텍스트 오염과 역할 혼선이다.
보안 검증을 수행하던 에이전트가 동시에 기능 채점을 하면,
이전 판단이 다음 판단에 누수된다.
스페셜리스트 분리는 이 컨텍스트 간섭을 차단한다.

각 게이트(Security Gate, Preflight Gate, Quality Gate 등)는
독립 서브 에이전트가 담당한다.
후보자 하나당 독립 서브 에이전트를 생성해 컨텍스트 오염을 막는
멀티 에이전트(Multi-Agent) 아키텍처다.
한 에이전트의 출력이 다음 에이전트의 입력으로만 흐르고,
내부 추론 상태는 공유되지 않는다.

스페셜리스트의 또 다른 이점은 교체 가능성이다.
Scoring 에이전트의 루브릭을 수정해도 Security Gate 에이전트는
영향을 받지 않는다.
채점 모델 17번 변경, 26번 초기화가 가능했던 것은
각 스테이지가 독립적으로 교체될 수 있었기 때문이다.

### 자동사냥 하네스의 루프 설계

“자동사냥”이라는 은유가 포착하는 것은 자율 반복성이다.
MMORPG 자동사냥은 캐릭터가 몬스터를 찾고, 공격하고, 아이템을 줍고,
체력이 떨어지면 회복하는 루프를 인간 개입 없이 반복한다.
AI 파이프라인의 자동사냥 하네스도 같은 구조다.
입력(지원자 코드)이 들어오면 각 스페셜리스트가 순서대로 처리하고,
실패가 감지되면 재시도 루프가 돌고, 최종 산출물(보고서)이 나온다.

하네스가 루프를 보장하는 방법은 세 가지다.
첫째, JSON Schema strict 모드(`additionalProperties: false`)로
에이전트 출력 형식을 강제한다.
둘째, 최대 5회 재시도 루프로 비결정성을 흡수한다.
셋째, 각 게이트의 통과·실패 기준이 마크다운 지침서에 명시되어
에이전트의 판단 기준이 일관되게 유지된다.

자동사냥 하네스가 단순 자동화와 다른 점은 피드백 루프다.
단순 자동화는 실패 시 중단한다.
하네스는 실패를 교정 신호로 처리해 재시도하거나, 에스컬레이션
정책에 따라 상위 오케스트레이터에게 위임한다.

### Markdown as Code와 하네스의 관계

파이프라인 전체가 마크다운 지침서로 구현된다는 사실은
하네스 엔지니어링의 핵심 통찰을 드러낸다.
코드 컴파일 없이, 수정 즉시 반영된다.
이것은 지침과 실행의 분리다.

마크다운 지침서가 AI 에이전트의 행동 명세가 되는 이 구조에서,
엔지니어의 주된 작업은 코드 작성이 아니라 지침 설계와 교정이다.
채점 기준을 수정하면 AI가 재채점한다.
게이트의 통과 조건을 바꾸면 AI가 다르게 판단한다.
“코드를 고치지 않고 지침을 수정하면 코드가 다시 만들어진다”는
역전이 하네스 엔지니어링의 핵심 루프다.

### 파이프라인 신뢰성의 세 원칙

무신사 파이프라인이 400여 명을 안정적으로 처리할 수 있었던 것은
세 원칙이 동시에 작동했기 때문이다.

스페셜리스트 분리: 역할 경계를 명확히 해 컨텍스트 오염과
책임 혼선을 차단한다.

결정론적 검증: JSON Schema strict 모드와 재시도 루프로
AI 출력의 비결정성을 흡수한다.
“마크다운의 자유도와 JSON Schema의 엄격함이라는 긴장이
하네스의 골격을 이룬다”는 표현이 이 균형을 잘 요약한다.

증거 기반 채점: AI 채점은 파일:라인 수준의 코드 증거를
의무화한다. 채점 결과가 구체적 근거에 묶여 있어야
캘리브레이션 루프에서 편향을 발견하고 수정할 수 있다.

## 비평

### 강점: 실전 검증된 설계 원칙의 공개

이 글이 추상적 이론이 아니라 400여 명 처리, 36일 1,952 커밋이라는
실제 운영에서 추출된 원칙이라는 점이 가장 큰 강점이다.
“자동사냥 하네스”라는 용어 자체가 실무 현장에서 자연스럽게
생겨난 은유일 가능성이 높다.
실제로 작동한 시스템에서 역산한 설계 원칙은 교과서적 이론보다
훨씬 높은 현장 적용 가능성을 갖는다.

스페셜리스트 분리, 컨텍스트 격리, 결정론적 검증의 조합은
멀티 에이전트 시스템의 실패 모드를 직접 겨냥하고 있다.
단순히 “에이전트를 여러 개 쓰면 좋다”는 수준을 넘어,
왜 분리해야 하는지, 어떻게 격리하는지, 신뢰성을 어떻게
보장하는지를 구체적으로 다룬다.

### 약점: “자동사냥” 은유의 한계

“자동사냥”은 직관적이지만 잘못된 기대를 심을 수 있다.
MMORPG 자동사냥은 환경이 고정되어 있고 몬스터는 예측 가능하다.
실제 AI 파이프라인이 처리하는 입력(지원자 코드, 비즈니스 문서,
고객 요청 등)은 훨씬 다양하고 예측 불가능하다.

자동사냥 하네스가 새로운 입력 분포를 만났을 때 어떻게
degradation을 감지하고 루브릭을 갱신하는지,
즉 “자동사냥 에이전트가 모르는 몬스터를 만났을 때”
어떻게 되는지에 대한 설명이 충분하지 않다.

### 약점: 하네스 자체의 유지보수 비용

마크다운 지침서가 파이프라인이 되는 구조는 강력하지만,
지침서가 복잡해지면 그것 자체가 유지보수 부채가 된다.
채점 모델 17번 변경, 루브릭 26번 초기화는 민첩성의 증거이기도
하지만, 하네스가 지속적으로 진화해야 한다는 것을 의미하기도 한다.
“자동사냥 하네스”가 자동으로 자신을 최적화하지는 않는다.
하네스 자체의 버전 관리와 회귀 테스트 전략에 대한 논의가 있었다면
더 완결된 그림이 되었을 것이다.

## 인사이트

### AI 스페셜리스트 분리는 소프트웨어 설계의 오래된 원칙을 재발견한다

단일 책임 원칙(Single Responsibility Principle)과 관심사 분리
(Separation of Concerns)는 객체지향 설계에서 수십 년간
강조된 원칙이다. AI 스페셜리스트 분리는 이 원칙을 에이전트
수준에 적용한 것이다.

“Security Gate는 보안만 본다”, “Scoring은 채점만 한다”는 선언이
AI 파이프라인에서 동일한 효과를 낸다. 각 에이전트가 단일 역할에
집중할 때, 그 역할의 성능을 측정하고 개선하기가 쉬워진다.
어디서 실패가 발생했는지를 특정 게이트로 격리할 수 있기 때문이다.

흥미로운 것은 이 원칙이 에이전트 간 컨텍스트 격리와 결합될 때
훨씬 강력해진다는 점이다. 소프트웨어 모듈은 메모리를 공유할 수
있지만, 에이전트는 컨텍스트 윈도우라는 인지적 자원을 갖는다.
이 자원이 오염되지 않도록 격리하는 것이 단순한 설계 관습이 아니라
신뢰성의 전제 조건이다.
오래된 원칙이 새로운 제약 조건 위에서 더 날카로워진다.

### 하네스는 예측 불가능성을 흡수하는 구조다

AI 출력은 본질적으로 비결정적이다. 같은 프롬프트에 같은 모델도
다른 응답을 낼 수 있다. 하네스가 해결해야 하는 가장 근본적인 문제는
이 비결정성을 결정론적 파이프라인 안에서 어떻게 처리하느냐다.

무신사의 접근은 두 층위에서 비결정성을 흡수한다. 출력 형식 층위에서는
JSON Schema strict 모드가 형식적 비결정성을 차단한다.
내용 층위에서는 5회 재시도 루프가 출력 실패를 흡수한다.
여기서 주목할 것은 재시도의 철학이다. 실패를 예외 상황이 아니라
시스템 설계의 일부로 취급한다. “AI가 5번 중 한 번은 틀릴 수 있다”는
현실을 하네스가 수용하면서도 전체 파이프라인의 신뢰성을 보장한다.

이 설계 철학은 분산 시스템의 “장애를 가정하고 설계하라(Design for
Failure)”와 같은 결을 공유한다. 서버가 언제든 죽을 수 있다는
가정 위에 분산 시스템이 설계되듯, AI 에이전트가 언제든 잘못된
출력을 낼 수 있다는 가정 위에 하네스가 설계된다. 신뢰성은
개별 구성 요소의 무결성이 아니라 시스템 전체의 복원력에서 온다.

### 자동사냥 은유가 가리키는 자율 운영의 새로운 지평

“자동사냥 하네스”라는 이름이 흥미로운 이유는 자율성(autonomy)에
대한 태도를 암시하기 때문이다. MMORPG에서 자동사냥은 플레이어가
자리를 비워도 캐릭터가 경험치를 쌓는다. AI 파이프라인에서 자동사냥
하네스는 엔지니어가 자리를 비워도 수백 명의 지원자를 처리한다.

이 자율성은 무조건적이지 않다. 게임의 자동사냥처럼, 규칙과 제약이
먼저 설정되어 있어야 한다. 어떤 몬스터를 공격하고, 체력이 얼마 이하면
물약을 쓰고, 가방이 다 차면 귀환한다 — 이 규칙의 집합이 자동사냥을
안전하게 만든다. 마찬가지로 AI 파이프라인의 게이트 조건, 채점 루브릭,
재시도 정책이 자동사냥 하네스를 신뢰할 수 있게 만든다.

주목할 것은 이 자율 운영 모델이 단순히 비용 절감 도구가 아니라는
점이다. 무신사의 경우 400명 처리에 800시간이 필요했을 인력 작업을
파이프라인이 대체했다. 하지만 더 중요한 것은 일관성이다. 인간이
400명을 평가하면 피로, 기분, 편견이 개입한다. 하네스는 동일한
루브릭을 400번 동일하게 적용한다. 스케일에서의 일관성은 인간 평가자가
원리적으로 달성할 수 없는 속성이다.

### 에이전트 조율의 거버넌스 문제는 이제 엔지니어링 과제다

멀티 에이전트 파이프라인이 실용화되면서 새로운 종류의 문제가
등장한다. 각 스페셜리스트 에이전트가 독립적으로 최적화되면,
전체 파이프라인이 부분 최적의 합산이 될 수 있다. Security Gate가
너무 엄격해서 정상 코드를 차단하거나, Scoring이 너무 관대해서
변별력을 잃는 경우다.

이것은 에이전트 하나의 성능 문제가 아니라 시스템 수준의 거버넌스
문제다. 무신사가 채점 모델을 17번 변경하고 26번 초기화한 것은
이 거버넌스 문제를 수동으로 해결한 과정이다. “구조적 편향 패턴을
발견하고 루브릭 자체를 수정한다”는 설계가 그 해법이었다.

앞으로 AI 파이프라인이 복잡해질수록 이 거버넌스의 자동화가
중요해질 것이다. 스페셜리스트 에이전트들의 성능을 모니터링하고,
편향을 자동 감지하고, 루브릭을 자동 수정하는 메타 하네스(Meta-Harness)
개념이 그 방향이다. 현재 무신사의 접근이 수동 캘리브레이션에 의존한다면,
다음 단계는 캘리브레이션 자체를 자동화하는 것이다. 거버넌스의 자동화는
하네스 엔지니어링의 다음 전선이다.
