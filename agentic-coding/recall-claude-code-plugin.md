# Recall — 오프라인 Claude Code 메모리 플러그인

<https://github.com/raiyanyahya/recall>

## 소개

Recall은 Claude Code를 위한 오프라인 메모리 플러그인이다.
외부 API, 추가 모델, 네트워크 통신 없이 세션 로그와 컨텍스트 요약을 로컬에서 유지한다.
Python으로 작성되었고 MIT 라이선스로 공개되어 있으며, GitHub에서 272개의 스타를 받았다.

핵심 가치는 세 가지다.
완전한 오프라인 동작(API 키, 외부 모델, 네트워크 전송 없음),
토큰 효율성(요약에 LLM 호출 대신 TF-IDF + TextRank 알고리즘 사용),
프라이버시 우선(모든 트랜스크립트가 사용자 로컬 머신에만 존재).

## 주요 기능

**두 개의 파일 구조**: `.recall/` 디렉토리에 두 파일을 유지한다.
`history.md`는 프롬프트, 응답, 파일, 명령어를 기록하는 추가 전용(append-only) 세션 로그이고,
`context.md`는 목표, 파일 변경 사항, 다음 단계를 담은 자동 생성 요약(약 1~2K 토큰)이다.

**명령어 세 가지**:
- `/recall:save` — 컨텍스트 요약 생성/갱신
- `/recall:show` — 컨텍스트 파일 표시
- `/recall:log` — 히스토리 파일 표시

**설정**: `recall.config.json`으로 출력 디렉토리, 캡처 동작, 요약 길이, 민감 정보 제거(redaction), git 통합을 커스터마이즈할 수 있다.

**설치**: `/plugin marketplace add raiyanyahya/recall` 한 줄로 설치된다.

## 분석

### 클라이언트 사이드 요약의 기술적 선택

Recall이 요약에 TF-IDF + TextRank를 선택한 것은 제약 조건에 대한 명시적 응답이다.
LLM 호출로 요약하면 더 고품질의 요약이 가능하지만, 추가 토큰 비용과 네트워크 의존성이 발생한다.
알고리즘 기반 요약은 품질이 낮을 수 있지만, 비용이 제로이고 오프라인에서 작동한다.

이 트레이드오프는 플러그인의 사용 맥락과 정확히 맞아떨어진다.
Claude Code 세션이 자주, 짧게 발생하는 워크플로우에서는 매 요약마다 LLM을 호출하는 것이
비효율적이다.
session 간 컨텍스트를 유지하는 것이 목적이라면,
“좋은” 요약보다 “비용 없는” 요약이 더 실용적이다.

### 메모리의 두 가지 다른 문제를 분리한다

AI 코딩 도구의 메모리 문제는 실제로 두 가지 다른 문제를 포함한다.
첫 번째는 세션 내 컨텍스트 창 관리 — 긴 세션에서 초기 컨텍스트가 잘려나가는 문제.
두 번째는 세션 간 지속성 — 오늘 작업한 내용을 내일 새 세션에서 접근하는 문제.

Recall은 두 번째 문제에 집중한다.
`/recall:save`로 생성된 `context.md`를 다음 세션 시작 시 로드하면,
이전 세션의 핵심 정보를 새 컨텍스트 창에 명시적으로 주입할 수 있다.
이것은 정교한 벡터 검색 기반 메모리 시스템과는 다른 접근이다.
단순하고 투명하며 사용자가 직접 내용을 확인하고 편집할 수 있다.

### 민감 정보 제거는 “최선 노력(best-effort)” 수준이다

Recall은 파일 쓰기 전에 민감 정보를 제거한다고 명시하지만,
“최선 노력(best-effort)”이라는 표현이 중요하다.
정규표현식 기반의 패턴 매칭으로 API 키, 토큰, 비밀번호 형태의 문자열을 감지하는 방식은
모든 민감 정보를 포착하지 못할 수 있다.
특히 맥락에 의존적인 민감 정보 — 내부 도메인명, 프로젝트 코드명, 고객 데이터 — 는
패턴 매칭으로 제거할 수 없다.

Bandit 보안 스캔과 CodeQL을 CI에 포함하는 것은 플러그인 코드 자체의 보안을 강화하지만,
기록되는 세션 내용의 민감도는 사용자가 직접 판단해야 한다.
기업 환경에서 이 플러그인을 사용할 때 `history.md`와 `context.md`가 보안 정책을
충족하는지 확인하는 것은 사용자의 책임이다.

## 비평

### 추가 전용 로그는 시간이 지날수록 품질이 저하된다

`history.md`의 추가 전용 설계는 단순성을 위한 선택이지만 장기적 문제를 내재한다.
세션이 쌓일수록 파일 크기가 증가하고, 오래된 컨텍스트가 새로운 요약에 영향을 미친다.
6개월 전의 완료된 작업, 이미 해결된 버그, 삭제된 파일에 대한 참조가
현재 세션의 요약에 노이즈로 혼입될 수 있다.

`context.md`의 자동 재생성이 이를 완화하지만,
TF-IDF 기반 알고리즘이 오래된 항목과 최신 항목의 중요도를 어떻게 구분하는지는 명확하지 않다.
수동으로 `history.md`를 정리하거나 아카이빙하는 워크플로우가 필요할 수 있지만,
현재 문서에는 이에 대한 안내가 부족하다.

### 세션 경계를 명시적으로 정의하지 않는다

Recall은 “세션”을 어떻게 정의하는지 명확하지 않다.
Claude Code를 닫았다가 열면 새 세션인가,
같은 프로젝트 디렉토리에서 작업하면 이전 세션과 연속인가.
`history.md`가 append-only라면 세션 간 구분이 없고,
시간적 맥락 없이 모든 작업이 하나의 스트림으로 기록된다.

현재 작업 중인 기능, 완료된 작업, 보류 중인 작업을 구분하는 것은
효과적인 컨텍스트 관리의 핵심이다.
단순 텍스트 로그 방식은 이 구분을 자동으로 처리하지 못하며,
요약의 품질이 이 구조적 부재로 인해 제한된다.

### 272개 스타의 프로젝트를 프로덕션 워크플로우에 쓰는 것은 위험 부담이 있다

Recall은 아직 성숙한 프로젝트가 아니다.
Claude Code 플러그인 에코시스템 자체도 빠르게 변화하고 있으며,
API 변경에 따른 호환성 문제가 발생할 수 있다.
메모리 시스템에 의존하기 시작하면 — 모든 세션 로그와 요약이 `.recall/`에 쌓이기 시작하면 —
플러그인이 유지 관리되지 않을 경우 데이터가 고립될 수 있다.

`history.md`와 `context.md`가 평범한 마크다운 파일이라는 점은 이 위험을 완화한다.
플러그인 없이도 파일을 읽고 활용할 수 있다.
그러나 이 플러그인을 팀 환경에서 공유하거나,
git에 커밋하거나, CI/CD 파이프라인에 통합하는 것은 추가적인 고려가 필요하다.

## 인사이트

### 오프라인 메모리 도구의 등장은 AI 도구 프라이버시 우려의 반응이다

Recall의 핵심 차별화가 “완전한 오프라인, 외부 API 없음”이라는 사실은
시장 수요를 반영한다.
AI 코딩 도구들이 사용자의 코드와 대화를 서버로 전송하는 것에 대한 불안감이
이 도구의 주요 가치 제안이 되었다.
기업 환경에서 소스 코드를 외부 서버로 전송하는 것은 법적·보안적 제약을 수반하고,
개인 개발자도 자신의 아이디어와 코드가 훈련 데이터로 사용되는 것을 우려한다.

이 트렌드는 더 확장될 것이다.
AI 도구의 기능이 강력해질수록 수집하는 데이터의 민감도도 높아지고,
오프라인 또는 자체 호스팅 대안에 대한 수요도 증가한다.
Recall은 그 수요의 가장 단순한 형태 — 메모리를 로컬 파일로 — 를 구현한다.

### 텍스트 파일 기반 메모리는 AI 도구가 파일 시스템에 의존하는 새로운 패러다임이다

Recall의 설계는 AI 도구의 상태 관리가 파일 시스템으로 이동하는 흐름을 대표한다.
`context.md`와 `history.md`는 데이터베이스나 API가 아니라 평범한 텍스트 파일이다.
이 선택은 단순성과 투명성을 제공하지만, 더 중요한 함의가 있다.
AI 도구의 메모리가 사용자에게 가시적이고 편집 가능해진다.

이것은 AI 도구 설계의 의미 있는 방향성이다.
AI가 무엇을 기억하는지 사용자가 볼 수 없고 통제할 수 없는 블랙박스 메모리와 달리,
텍스트 파일 기반 메모리는 사용자가 직접 내용을 확인하고, 수정하고, 삭제할 수 있다.
CLAUDE.md 파일이 프로젝트 지침을 텍스트로 관리하는 것과 같은 철학이다.
AI 시스템의 상태를 인간이 읽을 수 있는 파일로 표현하는 방향성은
감사 가능성(auditability)과 제어권을 사용자에게 되돌려준다.

### 알고리즘 요약의 한계는 결국 LLM 보조 요약으로 수렴할 것이다

현재 Recall은 TF-IDF + TextRank로 요약을 생성하지만,
이 방식의 한계는 시간이 지날수록 드러날 것이다.
알고리즘 요약은 텍스트의 통계적 특성에 기반하기 때문에,
“중요한 결정을 내린 이유”나 “다음에 해야 할 작업의 맥락” 같은 의미론적 내용을
정확히 포착하지 못한다.

LLM을 호출하지 않는다는 제약이 핵심 설계 원칙이지만,
로컬에서 실행되는 소형 모델(local LLM)의 발전이 이 트레이드오프를 바꿀 수 있다.
Ollama나 llama.cpp로 로컬에서 실행되는 소형 모델을 요약에 사용하면,
오프라인과 프라이버시 원칙을 유지하면서도 의미론적 요약이 가능해진다.
Recall의 현재 설계가 이 발전을 수용할 수 있는 확장 지점을 남겨두는지가
프로젝트의 장기적 가치를 결정할 것이다.
