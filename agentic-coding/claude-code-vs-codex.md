# Claude Code (100시간) vs Codex (20시간) — 14년차 엔지니어의 비교

원문: <https://www.reddit.com/r/ClaudeCode/comments/1sk7e2k/claude_code_100_hours_vs_codex_20_hours/>

## 요약

14년 경력의 Principal/Staff 엔지니어가 8만 줄 규모의 Python/TypeScript 프로젝트(테스트
약 2,800개)에서 Claude Code(Opus 4.6)와 Codex(GPT-5.4)를 비교한 글이다.

Claude Code는 빠르고 인터랙티브하지만 지시를 무시하고 작업을 미완성으로 남기는 경향이
있다. 새 파일을 만들기보다 기존 파일에 함수를 무분별하게 추가한다. 빠른 프로토타이핑에
적합하다.

Codex는 3~4배 느리지만 체계적이고 사려 깊다. AGENTS.md 가이드라인을 엄격히 준수하며,
자발적으로 리팩토링하고 코드 구조를 개선한다. 엔터프라이즈 소프트웨어 개발에 더 적합하다.

핵심 결론: 두 도구 모두 강한 소프트웨어 엔지니어링 역량 없이는 좋은 결과를 내지 못한다.
유능한 감독 없이 성공하는 도구는 없다.

## 분석

### 속도 vs 체계성의 트레이드오프

Claude Code의 빠른 응답과 Codex의 느리지만 체계적인 접근은 근본적으로 다른 설계
철학을 반영한다. Claude Code는 대화형 페어 프로그래밍 모델에 최적화되어 있고,
Codex는 비동기 코드 리뷰 모델에 가깝다. 이 차이는 단순한 속도 차이가 아니라
**인터랙션 패턴의 차이**다. 실시간으로 피드백을 주며 방향을 교정하는 워크플로와,
한 번에 명확한 지시를 내리고 결과를 기다리는 워크플로는 요구하는 엔지니어링 역량
자체가 다르다.

### 가이드라인 준수의 의미

Codex가 AGENTS.md를 엄격히 따른다는 관찰은 중요하다. 이것은 단순한 “지시 잘 듣는
도구”의 문제가 아니라 **하네스 엔지니어링(Harness Engineering)** 의 효과를 보여주는
사례다. 잘 작성된 가이드라인이 있을 때 Codex는 그 가이드라인의 품질만큼
결과물을 낸다. 반면 Claude Code가 가이드라인을 무시한다는 것은, 빠른 컨텍스트
전환 과정에서 장기 지시를 유지하는 능력이 약하다는 뜻이다.

### 코드 구조에 대한 태도 차이

Claude Code가 기존 파일에 함수를 몰아넣고 Codex가 자발적으로 리팩토링한다는 관찰은
**코드베이스에 대한 “이해”의 깊이** 차이를 시사한다. Codex는 파일 구조와 모듈 경계를
인지하고 변경을 거기에 맞추는 반면, Claude Code는 즉각적으로 동작하는 코드를
생성하는 데 집중한다. 이는 단기 최적화 vs 장기 유지보수의 고전적 긴장을 AI 도구
수준에서 재현한 것이다.

### 커뮤니티 반응의 양극화

GeekNews 댓글에서 gpdir16과 master6559는 원문과 정반대의 경험을 보고했다.
shblue21도 Codex가 더 우수하다고 단언하는 한편, tangokorea는 도메인에 따라
결과가 갈린다고 지적했다. 이 양극화는 이런 비교가 **프로젝트 특성, 프롬프트 습관,
가이드라인 품질에 크게 의존**한다는 것을 보여준다. 단일 비교에서 보편적 결론을
끌어내는 것은 위험하다.

## 비평

### N=1의 한계

8만 줄 Python/TypeScript 프로젝트 하나에서의 경험이다. 언어, 프레임워크, 테스트
커버리지, 코드 품질, 아키텍처 패턴이 모두 결과에 영향을 미친다. 원저자는 14년
경력의 시니어 엔지니어이므로 가이드라인 작성 능력이 높고, 이것이 Codex에 유리하게
작용했을 가능성이 크다. 주니어 개발자가 같은 실험을 하면 결과가 달라질 수 있다.

### 사용 시간의 비대칭

Claude Code 100시간 vs Codex 20시간이라는 비대칭은 공정한 비교를 어렵게 한다.
Claude Code에 대한 100시간의 경험은 해당 도구의 한계를 더 선명하게 드러내는 반면,
Codex 20시간은 아직 허니문 단계일 수 있다. 도구에 대한 숙련도(prompting 패턴,
워크플로 최적화)가 결과에 미치는 영향을 통제하지 않았다.

### 모델 버전의 일시성

Opus 4.6과 GPT-5.4의 비교는 글이 작성된 시점의 스냅샷이다. 모델은 빠르게
업데이트되며, gpdir16이 지적한 “Opus 4.6 성능 저하 의심”처럼 같은 모델도
시점에 따라 다르게 동작할 수 있다. 이런 비교는 유통기한이 매우 짧다.

## 인사이트

### 코딩 에이전트는 “더 나은 도구”가 아니라 “다른 종류의 팀원”이다

전통적 도구(IDE, 린터, 포매터)는 개발자의 의도를 보조한다. 개발자가 방향을 정하고,
도구는 그 방향으로 더 빠르게 갈 수 있게 한다. 코딩 에이전트는 다르다.
자체적인 판단을 내리고, 때로는 개발자의 의도를 무시하며, 예상치 못한 방향으로
코드를 변경한다.

이것은 도구가 아니라 팀원의 특성이다. Claude Code는 빠르지만 지시를 무시하는
주니어 개발자에 가깝고, Codex는 느리지만 코딩 컨벤션을 철저히 지키는 시니어
개발자에 가깝다. “어떤 도구가 낫냐”가 아니라 “어떤 팀원이 이 작업에 적합한가”로
질문을 바꿔야 한다.

이 관점 전환은 관리 방식도 바꾼다. 도구는 설정하고 사용하지만, 팀원은 소통하고
감독한다. 원저자가 Claude Code를 “능동적 관리(active management)가 필요하다”고
표현한 것은 이 본질을 정확히 포착한 것이다. 에이전트 시대의 엔지니어링 리더십은
코드 리뷰만이 아니라 **에이전트 관리**를 포함하게 된다.

### 멀티 에이전트 오케스트레이션은 파이프라인이 아니라 조직 설계다

oneforall88의 댓글이 제안한 “Opus 기획 -> Sonnet 구현 -> Codex 리뷰 -> Opus 검증”
워크플로는 단순한 도구 체이닝이 아니다. 이것은 소프트웨어 조직의 역할 분담 — 아키텍트,
개발자, QA, 테크 리드 — 을 AI 에이전트로 재현한 것이다.

역사적으로 소프트웨어 조직은 Conway의 법칙을 따랐다. 시스템 구조가 조직 구조를
반영한다. 멀티 에이전트 오케스트레이션에서도 같은 원리가 작동한다. 기획 에이전트와
구현 에이전트의 인터페이스가 곧 시스템의 모듈 경계가 된다. 에이전트 간 핸드오프의
품질이 곧 코드의 아키텍처 품질을 결정한다.

그러나 이 접근의 위험은 **복잡성의 폭발**이다. 에이전트 하나를 관리하는 것도
비자명(non-trivial)한데, 네 개의 에이전트를 조율하는 것은 소규모 팀을 관리하는 것과
같다. 각 에이전트의 출력을 검증하고, 에이전트 간 컨텍스트 손실을 방지하고, 전체
파이프라인의 일관성을 유지해야 한다. 이것은 결국 **에이전트를 관리하는 에이전트**를
필요로 하며, 그 에이전트를 누가 관리할 것인가라는 재귀적 문제에 도달한다.

### “도메인이 중요하다”는 관찰이 가장 실용적인 결론이다

tangokorea의 “도메인에 따라 다르다”는 댓글은 가장 과소평가된 통찰이다.
픽셀 퍼펙트 렌더링에서 Codex가 약하고, 복잡한 로직에서 Claude가 강하며,
표준적인 웹앱에서는 Codex가 “정신 건강에 좋다”는 관찰은 **도구 선택의 기준이
도구의 절대적 품질이 아니라 작업의 특성**이라는 것을 말해준다.

이것은 프로그래밍 언어 선택의 역사와 평행한다. “Python이 Java보다 나은가?”는
무의미한 질문이고, “이 작업에 Python이 적합한가?”가 유의미한 질문이다.
코딩 에이전트도 같은 성숙 단계에 진입하고 있다. “Claude Code가 Codex보다
나은가?”가 아니라 “이 작업에 어떤 에이전트가 적합한가?”가 올바른 질문이다.

실무적 함의는 명확하다. 프로토타이핑과 탐색적 코딩에는 Claude Code의 빠른
인터랙션이 유리하고, 가이드라인이 확립된 대규모 코드베이스의 유지보수에는
Codex의 체계적 접근이 유리하다. 그리고 bungker가 제안한 것처럼, 한 에이전트로
초안을 잡고 다른 에이전트로 리뷰하는 하이브리드 워크플로가 현재 시점에서 가장
실용적인 전략일 수 있다.

### 에이전트의 가치는 엔지니어의 역량에 비례한다

원저자의 핵심 결론 — “두 도구 모두 강한 소프트웨어 엔지니어링 역량 없이는
성공하지 못한다” — 은 가장 불편하지만 가장 중요한 메시지다.

이것은 자동화의 역설(Automation Paradox)이다. 시스템이 자동화될수록,
남은 수동 개입의 중요성은 더 커진다. 비행기 자동 조종이 고도화될수록
파일럿의 상황 판단 능력이 더 중요해지는 것과 같다. AI 코딩 에이전트가
강력해질수록, 에이전트가 실패하는 지점을 포착하고 교정하는 엔지니어의
역량이 결정적이 된다.

brainer의 “둘 다 human-in-the-loop가 필요하다, 자율 루프를 옹호하지 말라”는
댓글은 이 역설의 실무적 표현이다. 현재 시점에서 코딩 에이전트는 인간
엔지니어의 역량을 증폭하는 도구이지, 대체하는 도구가 아니다. 증폭기는
입력 신호의 품질에 비례하여 출력한다. 약한 신호를 증폭하면 노이즈만
커진다.
