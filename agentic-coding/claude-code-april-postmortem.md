# Claude Code 4월 품질 저하의 진짜 원인: 세 가지 버그

원문: <https://twitter.com/i/status/2048647065740415231>

참조: <https://anthropic.com/engineering/april-23-postmortem>

## 요약

2026년 3월부터 4월 중순까지 Claude Code의 품질이 눈에 띄게 저하됐다는
사용자 보고가 쏟아졌다. 한 GitHub Issue(#42796)에서는 23만 건의 툴 호출
데이터를 분석해 thinking 깊이가 75% 폭락하고, Read:Edit 비율이 6.6에서
2.0으로 떨어졌으며, 수정의 3분의 1이 “블라인드 수정”이라는 사실을 실증적으로
보여줬다. 당시 커뮤니티는 thinking token이 삭감됐다는 결론을 내렸지만,
Anthropic은 침묵했다.

4월 23일, Anthropic이 공식 포스트모텀(postmortem)을 발표하며 세 가지 독립된
버그를 인정했다. 세 버그는 각기 다른 시점에 발생했고 각기 다른 사용자 집단에
영향을 미쳤으며, 전부 4월 20일에 수정됐다. Anthropic은 모든 구독자의 사용량
한도를 리셋한다고 발표했다.

트위터 계정 @sitinme이 이 포스트모텀을 요약하며, 커뮤니티 데이터 분석과
공식 설명을 대조·검증했다. 그 결과 공식 설명이 커뮤니티의 관찰을 상당 부분
뒷받침하면서도, 설명되지 않는 간극이 남아 있음을 지적했다.

세 버그 중 가장 기술적으로 심각한 것은 캐시 최적화 버그다. 이 버그는 세션이
1시간 이상 유휴 상태일 때 thinking 히스토리를 전부 소거하도록 설계된 기능이
매 턴마다 반복 실행되도록 잘못 구현된 것이다. Claude는 계속 실행됐지만,
자신이 왜 그 판단을 했는지 점점 기억하지 못하는 상태가 됐다.

## 분석

### 버그 1: 기본 추론 깊이를 `high`에서 `medium`으로 변경 (3월 4일~4월 7일)

Anthropic은 Opus 4.6을 `high` 모드로 사용할 때 UI가 멈춰 보이는 현상과
비례적이지 않은 토큰 소비를 개선한다는 명목으로 기본값을 `medium`으로 내렸다.
내부 평가에서는 지능 손실이 미미하다고 결론 냈다.

그러나 실제 사용자들은 “바보가 됐다”고 보고하기 시작했다. Anthropic은 곧바로
기본값을 되돌리는 대신 노출 방식을 조정했다. 시작 시 알림을 추가하고, effort
선택기를 노출하고, `ultrathink` 옵션을 복원하는 식이었다. 사실상 “기본값을
바꿨으니 직접 조정하세요”라는 메시지였다. 더 많은 고객 피드백이 쌓인 4월 7일이
돼서야 기본값을 되돌렸다. 현재 Opus 4.7은 `xhigh`, 나머지 모델은 `high`이
기본값이다.

커뮤니티 데이터는 2월 하순부터 thinking 깊이가 2,200자에서 720자로 급감했음을
보여줬는데, 이는 `high`→`medium` 한 단계 변경으로 설명하기 어려운 67% 낙폭이다.
Anthropic의 포스트모텀은 이 수치 차이를 직접 설명하지 않았다.

### 버그 2: 캐시 최적화 버그로 인한 thinking 반복 소거 (3월 26일~4월 10일)

Claude Code는 대화 이력 속에 이전 thinking을 보존해, 모델이 각 라운드에서
자신의 추론 과정을 참조할 수 있도록 설계돼 있다. 3월 26일, 1시간 이상 유휴
상태였던 세션이 재개될 때 오래된 thinking을 제거해 캐시 미스 시의 토큰 전송량을
줄이는 최적화가 도입됐다. `clear_thinking_20251015` API 헤더와 `keep:1`을 사용해
가장 최근 thinking 블록만 보존하는 방식이었다.

문제는 이 소거 연산이 세션당 한 번만 실행돼야 하는데, 실제로는 **매 턴마다**
실행됐다는 것이다. 세션이 유휴 임계값을 한 번이라도 넘으면, 이후의 모든 요청이
“최근 thinking 블록 하나만 남기고 전부 버려라”는 신호를 API에 보냈다. 게다가
Claude가 툴을 실행하는 도중 사용자가 새 메시지를 보내면 새로운 턴이 시작되고,
이 새 턴도 broken flag를 가지고 있어서 현재 진행 중인 thinking까지 폐기됐다.

Claude는 계속 명령을 수행했지만, 그 명령을 왜 수행하는지 점점 잊어갔다.
사용자들이 관찰한 건망증, 반복, 이상한 툴 선택이 바로 이 버그의 증상이다.

thinking 소거는 캐시 미스도 유발했다. 컨텍스트 내용이 바뀌면 이전 캐시가
무효화되고, 96만 토큰 컨텍스트 윈도우에서 캐시 미스는 막대한 비용이다.
커뮤니티에서 별도로 보고된 “캐시 TTL이 1시간에서 5분으로 줄었다”는 현상과
“Pro Max 사용자의 할당량 소진 속도가 비정상적으로 빠르다”는 현상이 모두
이 버그와 연결된다.

이 버그가 두 주 동안 발견되지 않은 이유는, 두 가지 내부 실험이 재현을
방해했기 때문이다. 서버 측 메시지 큐 실험과 thinking 표시 방식 변경이 대부분의
CLI 세션에서 버그 증상을 숨겼고, 내부 테스트 환경에서도 재현이 안 됐다.
다중 코드 리뷰(사람+자동), 단위 테스트, 엔드투엔드 테스트, 자동 검증, 내부 테스트
모두를 통과한 끝에 사용자 피드백으로 겨우 포착됐다. 4월 10일 v2.1.101에서
수정됐다.

### 버그 3: 시스템 프롬프트의 출력 길이 제한 (4월 16일~4월 20일)

Opus 4.7이 출시된 4월 16일, 시스템 프롬프트에 다음 한 줄이 추가됐다.

```text
Length limits: keep text between tool calls to 25 words or less.
Keep final responses to 100 words or less unless the task requires more detail.
```

Opus 4.7의 장황함을 줄이기 위한 조치였다. 수 주간의 내부 테스트와 평가를
통과했다. 그러나 더 광범위한 평가 집합에서 Opus 4.6과 4.7의 코딩 품질이 3%
저하된다는 사실이 발견돼 4월 20일에 롤백됐다. Sonnet 4.6, Opus 4.6,
Opus 4.7 모두 영향을 받았으며, 4일간 지속됐다.

### 세 버그의 합산 효과

각각의 버그는 개별적으로 보면 치명적이지 않다. 그러나 서로 다른 시점에,
서로 다른 사용자 집단에 영향을 미치면서 동시에 겹쳤을 때, 결과는 “3월부터
4월 중순까지 Claude Code가 광범위하고 비일관적으로 나빠졌다”는 인상이 됐다.

비일관성이 핵심이다. 유휴 임계값을 넘기지 않거나, 수동으로 `high` effort를
설정하거나, 새 프롬프트가 포함된 버전으로 업데이트하지 않은 사용자는 아무
영향을 받지 않았다. 세 버그를 모두 맞은 사용자도 있었다. 이 차이가 커뮤니티에서
“나는 변화 못 느꼈다” vs “분명히 나빠졌다”의 논쟁으로 표출됐다.

## 비평

### 공식 설명이 커버하지 않은 수치 간극

커뮤니티 데이터는 thinking 깊이가 67% 감소했다고 보고했다. `high`→`medium`
한 단계 변경으로 설명하기에는 지나치게 큰 낙폭이다. 포스트모텀은 이 수치를
직접 언급하지 않는다. 세 버그의 효과가 중첩됐을 가능성이 높지만, Anthropic은
그 메커니즘을 구체적으로 설명하지 않았다. 실증 데이터를 제출한 커뮤니티에 대한
최소한의 응답으로는 부족하다.

### 탐지 실패의 근본 원인

두 주 동안 캐시 버그가 탐지되지 않은 이유는 “두 개의 내부 실험이 재현을
방해했다”는 것이다. 이 설명은 사실이지만, 더 깊은 문제를 가린다. 내부 빌드와
공개 빌드의 차이가 크면, 내부 테스트는 공개 버전의 동작을 재현하지 못한다.
인프라 수준의 실험이 애플리케이션 레이어 버그의 증상을 숨길 수 있다는 사실은,
단순히 “더 넓은 eval 매트릭스”만으로 해결되지 않는다.

### 대응의 우선순위

버그 1에 대한 Anthropic의 초기 대응, 즉 “기본값을 되돌리는 대신 UI를 조정한다”는
선택은 의문스럽다. “사용자 대부분은 기본값을 바꾸지 않는다”는 사실은 제품 개발의
상식인데, 왜 이를 사용자 피드백을 추가로 수집할 때까지 기다렸는가. 빠른
롤백보다 새 기능 노출을 우선한 판단처럼 보인다.

### 포스트모텀의 가치

단점에도 불구하고, 이 포스트모텀은 AI 기업의 공개 커뮤니케이션으로서 드문
사례다. 버그를 구체적으로 인정하고, 시간 선상을 공개하고, 수정 버전 번호를
명시하고, 사용량 한도를 리셋했다. “더 열심히 하겠습니다” 수준의 사과가 아니라
기술적 사실을 공개한 것은, 신뢰 회복의 출발점으로서 평가할 만하다.

## 인사이트

### 비일관적 품질 저하가 단일 품질 저하보다 훨씬 더 나쁜 이유

버그 하나가 모든 사용자에게 일관되게 영향을 미쳤다면, 진단과 수정이 훨씬 빨랐을
것이다. 사용자 보고가 한 방향으로 수렴하고, 내부 재현도 쉬워진다.

그러나 이번 사태는 세 버그가 서로 다른 트리거 조건(유휴 시간, effort 설정,
모델 버전)을 가지고, 서로 다른 시간대에 발생했다. 어떤 사용자는 버그 1만,
어떤 사용자는 버그 2만, 어떤 사용자는 세 개 모두를 경험했다. 결과는 “Claude Code가
나빠졌다”는 신호가 노이즈에 파묻혔다. 커뮤니티 내부 논쟁(“나는 변화 없다”
vs “분명히 나빠졌다”)이 사실의 검증을 방해하는 소셜 노이즈를 만들었다.

이 구조는 단순히 테스트 커버리지 부족의 문제가 아니다. 실사용 환경에서의
비일관성은 설계 자체의 결함에서 온다. thinking 보존, 캐싱, 추론 깊이가 각자의
로직으로 독립적으로 작동하면서 상호작용 효과가 설계 단계에서 고려되지 않았다.
시스템이 복잡해질수록 단일 기능 테스트가 아니라 기능 간 상호작용 테스트가
필수가 된다.

### “내부 빌드”와 “공개 빌드”의 괴리는 품질 보증의 구조적 맹점

Anthropic은 포스트모텀에서 개선 방안 중 하나로 “더 많은 내부 직원이 공개 버전을
사용한다”를 꼽았다. 이 사실이 암시하는 것은, 기존에 내부 직원들이 주로 사용하던
빌드는 공개 버전과 달랐다는 것이다.

이것은 단순한 개발 편의 문제가 아니다. “개발자가 자신의 제품을 실제로
사용한다(dogfooding)”는 원칙은, 개발팀이 사용자와 동일한 경험을 공유하기 때문에
버그를 사용자와 같은 조건에서 경험한다는 것을 의미한다. 내부 빌드와 공개 빌드가
다르면, 이 원칙 자체가 작동하지 않는다.

AI 코딩 도구처럼 실사용 컨텍스트가 중요한 제품에서, 합성 벤치마크와 내부 eval은
장기 세션, 유휴 후 재개, 실제 프로젝트 파일 구조 같은 실사용 시나리오를
재현하지 못한다. Anthropic이 “더 광범위한 eval 매트릭스”를 개선 방안으로 내세운
것은 맞지만, 구조적 해법은 공개 빌드와 동일한 환경에서 실제로 작업하는 것이다.

### 사용자 신뢰와 투명성 사이의 타이밍 문제

커뮤니티가 23만 건의 툴 호출 데이터를 분석해 버그를 실증적으로 증명하고 한
달이 지난 뒤에야 공식 포스트모텀이 나왔다. 그 한 달 동안 커뮤니티는 추측과
실험을 반복했고, 잘못된 원인 분석이 확산됐으며, “Anthropic이 의도적으로 모델을
나쁘게 만들었다”는 음모론도 등장했다.

투명성의 타이밍은 내용만큼 중요하다. “버그를 완전히 수정한 뒤 설명한다”는
접근은 엔지니어링 팀에게 합리적이지만, 사용자는 설명을 기다리는 동안 대안을
찾거나 신뢰를 잃는다. 특히 유료 구독 서비스에서 품질 저하는 직접적인 비용
문제다. “진행 중인 조사”를 중간에 공개하는 것이 때로는 완벽한 포스트모텀보다
신뢰 유지에 효과적이다.

이 사태는 AI 기업들이 제품 품질 변화에 대해 어떻게 커뮤니케이션해야 하는지에
대한 선례가 됐다. Anthropic의 포스트모텀은 기술적 상세함에서 업계 기준을
높였지만, 타이밍의 문제는 다음 사태에서 개선될 부분으로 남는다.

### 캐시 버그가 드러낸 상태 관리의 설계 원칙

캐시 최적화 버그의 핵심은 “한 번만 실행돼야 하는 연산이 매 턴마다 실행됐다”는
것이다. 이 버그 자체는 단순하다. 하지만 이 버그가 두 주 동안 수많은 테스트를
통과한 이유는, 세션 상태의 올바른 표현이 무엇인지에 대한 가정이 테스트 코드와
프로덕션 코드 사이에서 달랐기 때문이다.

AI 에이전트 시스템에서 상태 관리는 일반 소프트웨어보다 훨씬 미묘하다. 세션의
“논리적 경계”가 코드 레벨의 경계와 일치하지 않는 경우가 많다. 사용자가 인식하는
“하나의 세션”은 여러 API 호출, 여러 툴 실행 턴, 여러 캐시 엔트리에 걸쳐 분산된다.
이 경계를 잘못 정의하면, 정확히 이번처럼 Claude가 “실행은 계속하지만 이유를 잊는”
상태가 된다.

이 사태는 AI 에이전트 시스템의 상태 지속성(state persistence) 설계를 처음부터
명시적으로 테스트해야 한다는 원칙을 실증했다. “기능이 올바르게 작동한다”는
단위 테스트만으로는 부족하다. “세션 경계에서 상태가 예상대로 보존되고 소거된다”는
통합 테스트가 별도로 필요하다.
