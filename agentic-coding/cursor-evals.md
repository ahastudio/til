# Cursor Evals — CursorBench 3.1

<https://cursor.com/evals>

HN 토론: <https://news.ycombinator.com/item?id=48756840> (162점, 99개 댓글)

## 소개

CursorBench 3.1은 Cursor가 공개한 AI 에이전트 코딩 벤치마크 리더보드다.
“실제 Cursor 세션에서 나온 모호하고 멀티파일에 걸친 과제”를 기반으로 에이전트를 평가한다.
점수(%), 과제당 평균 비용($), 토큰 소비량, 스텝 수 네 가지 지표로 모델을 비교한다.

2026년 7월 기준 36개 모델이 등재되어 있으며, 점수 범위는 31.9%에서 72.9%까지다.
과제당 비용은 최저 $0.55에서 최고 $18.02까지 분포한다.
비용은 각 모델의 공개 per-million-token 가격(입력, 캐시 읽기, 캐시 쓰기, 출력)을 기준으로 산출한다.

CursorBench 3.1은 이전 버전(3.0)의 편집, 리팩토링, 버그픽스 과제에
코드베이스 이해, 버그 탐지, 플래닝, 코드 리뷰 문제를 추가했다.
벤치마크 측은 “결과에는 분산이 있으며, 점수의 작은 차이는 통계적으로 유의미하지 않을 수 있다”고 명시한다.

## 분석

### “실제 사용자 세션 기반 과제”가 벤치마크에 부여하는 생태적 타당성과 편향

HumanEval, MBPP, SWE-bench 같은 기존 코딩 벤치마크는 연구자가 설계한 합성 과제를 사용한다.
CursorBench는 실제 Cursor 사용자들이 맞닥뜨린 과제를 사용한다는 점에서 다르다.
과제가 “모호하고(ambiguous)” “멀티파일에 걸쳐 있다(multi-file)”는 설명은
현실의 소프트웨어 엔지니어링 작업과 더 가까운 벤치마크를 지향한다는 뜻이다.

이 “생태적 타당성(ecological validity)”은 실질적인 장점이다.
연구 환경에서 설계된 단일 함수 작성 과제를 잘 푸는 능력이
실제 개발 워크플로에서의 유용성과 다를 수 있다는 비판이 기존 벤치마크에 오랫동안 제기됐다.
Cursor가 실제 사용자 세션을 과제 기반으로 사용하면, 벤치마크와 실제 사용 사례 간의 괴리가 줄어든다.

그러나 “실제 Cursor 세션”은 동시에 편향의 원천이다.
Cursor 사용자는 특정 프로그래밍 언어, 프로젝트 유형, 개발 스타일로 편중될 수 있다.
Cursor의 주요 사용자가 TypeScript/Python 기반 웹 개발자에 집중되어 있다면,
시스템 프로그래밍, 임베디드, 데이터 과학 코드에서의 에이전트 성능은 다르게 측정될 수 있다.

### Effort 레벨 분화가 만드는 비용-성능 최적화의 새로운 레이어

리더보드에서 가장 눈에 띄는 패턴은 동일 모델 가족 내 effort 레벨 분화다.
Fable 5는 Max(72.9%, $18.02)부터 Low(64.2%, $5.70)까지 다섯 단계로 등재됐다.
Opus 4.7은 Max(64.8%, $11.02)부터 Low(48.3%, $1.87)까지, Opus 4.8은 Max(63.8%, $7.59)부터 Low(54.3%, $2.93)까지 분포한다.

이 분화는 “어떤 모델을 쓸 것인가”라는 질문을 “어떤 effort 레벨로 쓸 것인가”라는 질문으로 바꾼다.
Fable 5 Medium(69.8%, $8.27)은 Opus 4.7 Max(64.8%, $11.02)보다 5% 높은 점수를 1/3 낮은 비용으로 달성한다.
모델 선택의 결정이 단순 브랜드 선택이 아니라 effort 파라미터 튜닝 문제가 됐다는 것이다.

실무 관점에서 이 분화는 워크로드별 최적 effort 레벨 탐색이라는 운영 과제를 낳는다.
단순 자동완성과 복잡한 멀티파일 리팩토링에 동일한 effort 레벨을 적용하면
전자에서는 과잉 지출, 후자에서는 성능 부족이 발생할 수 있다.
에이전트 코딩 플랫폼이 과제 복잡도에 따라 effort를 자동으로 조정하는 기능을 갖출 필요성이 여기서 나온다.

실제 고난도 작업 경험도 이 방향을 지지한다.
tekacs[^tekacs]는 매일 "에이전트가 겨우 처리할 수 있는 경계선에 있는 과제"를 수행하면서
최상위 effort 수준을 유지해야 한다고 설명한다.
그는 Fable이 "세심하게 관리하지 않으면 구현물 안에 크고 위험한 구멍을 남긴다"고 지적한다.
벤치마크 점수는 과제 완료율을 측정하지만, 실무에서 더 중요한 것은 예측 가능성이다.
어떤 과제에서 놓치는지 예측할 수 없다면, 인간 검토의 부담은 줄어들지 않는다.

### Composer 2.5의 이례적 효율성과 에이전트 특화 모델의 가능성

Composer 2.5는 리더보드에서 가장 주목할 만한 이례다.
63.2% 점수를 $0.55에 달성하여, 2위 모델(Fable 5 Extra High, $13.74)의 1/25 비용으로 비슷한 수준의 성능을 낸다.
Composer 2($0.56, 52.2%)와 비교하면 버전 업그레이드에서 11% 포인트 성능이 향상됐다.

Composer는 Cursor의 자체 모델로 추정된다.
Cursor의 인터페이스, 컨텍스트 구조, 사용자 워크플로에 최적화된 방식으로 훈련됐을 가능성이 높다.
이 가설이 맞다면, 일반 목적 대형 언어 모델이 아니라 특정 에이전트 환경에 특화된 소형 모델이
비용 효율적으로 경쟁력을 갖출 수 있다는 것을 보여주는 사례가 된다.

## 비평

### Cursor가 자체 벤치마크를 설계하고 자체 모델로 참가하는 구조적 이해충돌

CursorBench는 Cursor가 설계하고, Cursor 데이터를 사용하며, Cursor의 자체 모델(Composer)이 참가한다.
이 세 가지가 겹치는 것은 명백한 이해충돌이다.
벤치마크 설계자가 동시에 참가자인 경우, 과제 선택, 채점 기준, 평가 환경이
자신에게 유리한 방향으로 편향될 가능성이 있다.

이것이 의도적인 조작이어야 할 필요는 없다.
Cursor가 자사 사용자 세션에서 과제를 고르는 과정에서
Cursor 인터페이스와 워크플로에 맞게 과제가 프레이밍될 수 있다.
Cursor의 에디터 컨텍스트, 파일 구조 전달 방식, 커서 위치 정보 같은 환경 요소들이
Cursor 에이전트 프로토콜에 익숙한 모델에게 유리하게 작용할 수 있다.
Composer가 $0.55에 63.2%라는 이례적 효율을 내는 이유 중 일부는 이 환경 최적화일 수 있다.

독립적인 제3자가 과제 선정과 채점을 검증하거나,
Cursor 세션 기반 과제가 다른 코딩 에이전트 플랫폼(GitHub Copilot Workspace, Replit 등)에서도 동등하게 적용 가능한지에 대한 투명성이 없다는 점은 이 벤치마크의 외적 타당성을 제한한다.

famouswaffles[^famouswaffles]는 이 편향의 메커니즘을 직접 지목한다.
Cursor 세션은 Composer 모델이 강화학습(RL) 훈련을 받은 데이터와 사실상 같은 분포라는 것이다.
훈련 분포와 평가 분포가 일치하는 경우, 벤치마크 성능은 독립적인 능력의 증거가 아니라
분포 일치의 결과일 수 있다.

이를 뒷받침하는 반증이 존재한다.
mdasen[^mdasen]은 Artificial Analysis의 DeepSWE 벤치마크 결과를 제시한다.
DeepSWE에서 GPT-5.5 xhigh는 64점, Opus 4.8 max는 56점인 반면
Composer 2.5는 16점에 그친다.
CursorBench에서 세 모델이 62~64% 범위 안에 수렴하는 것과 완전히 다른 양상이다.
두 벤치마크 중 어느 쪽이 더 타당한지는 판단하기 어렵지만,
자사 모델이 자사 벤치마크에서만 유독 효율적으로 보이는 현상이
제3자 벤치마크에서 재현되지 않는다는 사실은 이 비판의 무게를 더한다.

### “통계적으로 유의미하지 않을 수 있다”는 경고가 리더보드의 의미를 흔든다

벤치마크 페이지는 “점수의 작은 차이는 통계적으로 유의미하지 않을 수 있다”고 명시한다.
36개 모델의 세부 순위가 마케팅 자료로 활용될 때 이 경고는 실질적인 효력을 갖기 어렵다.

문제는 “작은 차이”의 범위가 명시되지 않는다는 것이다.
5%? 10%? 어느 임계점까지가 유의미한 차이인지 기준이 없으면
리더보드 사용자는 63.2%와 63.8% 사이의 0.6% 포인트 차이를 실질적인 것으로 오해할 수 있다.
상위 10위 모델 중 8~10위(Opus 4.8 Max 63.8%, Composer 2.5 63.2%, GPT-5.5 High 62.6%)가
서로 1.2% 포인트 범위 안에 있는데, 이것이 실질적인 성능 차이인지 측정 노이즈인지 알 수 없다.

표본 수와 신뢰 구간이 공개되지 않는다는 것은 리더보드를 과학적 참조 자료가 아닌 마케팅 자료로 읽어야 한다는 것을 의미한다.

### 비용 비교가 실제 운영 비용을 반영하지 않는다

비용 계산이 “공개 per-million-token 가격”을 기준으로 한다는 것은 명확히 밝힌다.
그러나 실제 기업 환경에서의 API 비용은 공개 가격과 다르다.
대규모 사용량에 따른 할인, 기업 계약, 배치 API 할인, 캐싱 최적화가 모두 다르게 적용된다.

또한 비용이 과제당 절대값으로 제시되지만,
실제 개발팀의 관심은 “이 비용이 개발자 시간 대비 얼마나 효율적인가”다.
Fable 5 Max($18.02, 72.9%)가 1시간 개발자 작업을 10분으로 단축한다면 비용 효율적일 수 있지만,
GPT-5.5 Extra High($4.37, 64.3%)가 같은 작업을 15분으로 단축한다면 전자가 더 비싸지는 않을 수 있다.
과제당 달러 비용은 유용한 지표지만, 개발자 생산성 향상과의 관계 없이는 의사결정에 충분하지 않다.

BugsJustFindMe[^BugsJustFindMe]는 더 근본적인 비교 구조 문제를 지적한다.
Composer 2.5는 Cursor 월 구독을 통해서만 이용할 수 있기 때문에,
API 단위 토큰 비용 비교 자체가 실질적 의미를 갖지 않는다.
동등한 금액의 OpenAI 구독은 훨씬 높은 성능의 GPT-5.5에 대한 동일한 접근권을 제공한다.
벤치마크의 비용 지표는 "API를 직접 호출할 경우의 이론적 비용"이지,
실제 사용자가 선택하는 구독 패키지 간의 가치 비교가 아니다.

## 인사이트

### 에이전트 벤치마크가 Goodhart's Law에 진입하는 속도가 빨라지고 있다

“측정 기준이 목표가 되면 더 이상 좋은 측정 기준이 아니다”는 Goodhart's Law는
AI 벤치마크에서 반복적으로 나타난다.
HumanEval 점수가 공개되자 모델들이 해당 과제에 과적합되기 시작했고,
MMLU, ARC 같은 벤치마크도 비슷한 궤적을 따랐다.

CursorBench가 Cursor 사용자 세션을 기반으로 한다는 점은 처음에는 오염을 막는 장치처럼 보인다.
그러나 벤치마크가 공개되고 과제 유형이 알려지면,
모델 제공업체들은 “코드베이스 이해, 버그 탐지, 플래닝, 코드 리뷰”에 특화된 훈련 데이터를 투입하기 시작한다.
Cursor가 과제를 계속 갱신하지 않으면 벤치마크 자체가 경쟁의 목표가 되고,
실제 사용자 경험과의 연관성이 희석된다.

이 문제의 해법은 주기적인 과제 갱신과 평가 블라인드 유지다.
그러나 과제 갱신에는 비용이 들고, 블라인드 유지는 투명성과 충돌한다.
에이전트 코딩 벤치마크가 성숙하려면 이 긴장을 어떻게 해소할 것인지에 대한 방법론적 합의가 필요하다.

shadeslayer_[^shadeslayer_]는 이 문제를 실용적 관점에서 요약한다.
"독립적인 벤치마크들은 이미 모델의 훈련 데이터가 되어 있을 가능성이 높다"는 것이다.
그가 내리는 결론은 "모델의 최종 테스트는 자신의 실제 워크로드에서 얼마나 잘 작동하는가"다.
대부분의 모델이 일상적인 과제 대부분을 처리할 수 있는 수준에 이른 지금,
벤치마크 점수보다 사용 마찰(friction)이 낮은 모델이 실질적으로 더 나은 선택일 수 있다.
이는 Goodhart's Law의 실천적 귀결이다.
벤치마크가 오염될수록, 현장의 선택 기준은 측정 지표에서 멀어지고 경험으로 이동한다.

### 모델 가족 내 Effort 레벨 분화가 AI 구매 결정의 단위를 바꾼다

기존 모델 선택 패러다임은 “어떤 모델을 쓸 것인가”였다.
GPT-4 vs Claude 3 vs Gemini 1.5처럼 모델 단위로 결정했다.
Fable 5, Opus 4.7/4.8, GPT-5.5가 모두 5단계 effort 레벨로 등재된 리더보드는
이 패러다임이 “어떤 모델 가족의 어떤 effort 레벨을 어떤 과제에 쓸 것인가”로 이동했음을 보여준다.

이 변화의 실질적 의미는 AI 비용 최적화의 복잡성 증가다.
단일 effort 레벨을 선택하는 것이 아니라, 과제 복잡도를 실시간으로 분류하고
적절한 effort를 동적으로 할당하는 시스템이 필요해진다.
Cursor가 이 할당을 에이전트 플랫폼 레이어에서 자동화한다면,
사용자는 모델과 effort를 직접 선택하지 않고 플랫폼에 위임하게 된다.

이 위임은 플랫폼 종속성을 강화한다.
어떤 모델의 어떤 effort를 어떤 과제에 쓰는지를 Cursor가 결정하면,
사용자는 이 결정의 근거를 알 수 없고 최적화도 직접 할 수 없다.
CursorBench가 이 결정을 위한 공개 데이터를 제공하는 것은 투명성 측면에서 긍정적이지만,
실제 Cursor 제품 내에서 라우팅이 어떻게 이루어지는지는 별개의 문제다.

### 에이전트 특화 소형 모델이 범용 대형 모델의 경쟁 구도를 바꾼다

Composer 2.5($0.55, 63.2%)의 비용 효율은 단순한 벤치마크 이례가 아니다.
에이전트 코딩 과제에 특화된 방식으로 훈련된 소형 모델이
범용 대형 모델과 충분히 경쟁할 수 있다는 것을 보여준다.

이 패턴은 자연어 처리 분야의 경험과 일치한다.
BERT 이후의 도메인 특화 파인튜닝 모델들이 GPT-3 수준의 범용 모델을 특정 과제에서 능가했다.
에이전트 코딩에서도 “특화된 소형 모델이 범용 대형 모델보다 비용 효율적일 수 있다”는 임계점이 존재한다.

Cursor가 Composer를 공개 리더보드에 올리는 것은 이 경쟁력을 시장에 신호하는 행위다.
동시에 이것은 다른 에이전트 코딩 플랫폼들 — Windsurf, GitHub Copilot, Replit 등 —이
자체 특화 모델 개발에 투자할 인센티브를 높인다.
범용 API 모델에 전적으로 의존하는 코딩 에이전트 플랫폼이
장기적으로 자체 모델을 보유한 플랫폼에 비해 비용과 성능 모두에서 불리해지는 경쟁 구도가 형성될 수 있다.

### 코딩 에이전트 벤치마크의 표준화가 이루어지기 전에 선점 효과가 작동한다

AI 코딩 에이전트 벤치마크는 아직 표준화되지 않았다.
SWE-bench Verified, LiveCodeBench, CursorBench가 각각 다른 과제 유형과 채점 방식을 사용한다.
이 분열된 상태에서 Cursor가 자체 벤치마크를 공개 리더보드로 운영하는 것은
“코딩 에이전트 성능 측정의 기준”으로 CursorBench를 자리매김하려는 전략이다.

벤치마크 표준이 되는 것은 단순히 유용한 도구를 제공하는 것 이상의 의미가 있다.
어떤 벤치마크가 업계 표준이 되면, 모델 제공업체들이 그 벤치마크에서 좋은 성적을 내도록 최적화하기 시작한다.
이것은 Cursor가 정의한 “좋은 코딩 에이전트”의 기준이 업계 전체의 개발 방향에 영향을 미친다는 것을 의미한다.
GLUE/SuperGLUE가 자연어 이해 연구의 방향을 형성했듯이,
CursorBench가 성공적으로 표준화되면 에이전트 코딩 모델 개발의 암묵적 사양서가 될 수 있다.
