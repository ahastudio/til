# 코드란 무엇인가

원문: <https://martinfowler.com/articles/what-is-code.html>

## 요약

Thoughtworks의 Distinguished Engineer인 Unmesh Joshi가 쓴 이 글은,
LLM이 코드를 자동으로 생성하는 시대에 “코드란 무엇인가”라는 근본적인 질문을 다시 던진다.
코드는 크게 두 가지 역할을 한다. 첫째는 기계에게 주는 명령(machine instructions)이고,
둘째는 문제 도메인을 개념적으로 모델링한 결과물(conceptual model)이다.
저자는 LLM이 전자를 빠르게 대체하고 있는 지금, 후자의 역할이 오히려 더 중요해졌다고 주장한다.

코드는 단순한 명령문이 아니라, 특정 도메인의 어휘(vocabulary)를 프로그래밍 언어로 번역한 결과다.
은행, 리테일, 헬스케어 같은 도메인 각각에는 고유한 어휘 체계가 있으며, 개발자는
그 어휘를 기술 도메인(웹, DB, 인프라)의 어휘로 옮기는 번역자 역할을 한다.
좋은 코드는 그 번역이 정확하고 일관되어, 코드 자체가 도메인 언어처럼 읽힌다.

Domain-Driven Design(DDD)의 경계 문맥(Bounded Context)과 보편 언어(Ubiquitous Language) 개념이
이 맥락에서 등장한다. 복잡한 도메인일수록 공통으로 쓸 수 있는 상위 프레임워크가 존재하지 않으며,
개발자와 도메인 전문가가 긴밀히 협력해 반복적으로 어휘와 추상을 발견해나가야 한다.
테스트 주도 개발(TDD)은 이 과정에서 이름과 경계를 발견하는 피드백 루프를 제공한다.

프로그래밍 언어 자체도 사고 도구(thinking tool)로 기능한다. Go의 채널, Rust의 소유권 모델,
Java의 OOP는 각각 개발자가 문제를 바라보는 방식을 다르게 형성한다.
저자는 코드를 수동적으로 검토하는 것이 아니라 직접 작성하는 행위 자체가 깊은 사고를 유발한다고 말한다.

LLM 시대의 핵심 위험으로 저자는 “인지 부채(cognitive debt)”를 제시한다.
LLM이 생성한 코드가 컴파일되고 테스트를 통과하더라도, 팀이 그 개념 모델을 이해하지 못한다면
코드베이스는 공유된 이해 없이 어휘만 쌓이는 상태가 된다. 반대로 잘 설계된 코드는
명확한 어휘와 추상을 통해 LLM이 의도를 더 정확하게 구현할 수 있도록 돕는 맥락(context)이자
검증 도구(harness)가 된다.

## 분석

### 두 역할의 분리: 무엇이 상품화되고 있는가

저자가 제시하는 코드의 두 역할 구분은 현재의 AI 코딩 논쟁을 명확하게 정리한다.
기계 명령 생성 측면은 LLM이 빠르게 대체하고 있으며, 이는 되돌릴 수 없는 추세다.
반면 개념적 모델링 측면은 도메인 지식, 맥락 이해, 반복적 피드백 없이는 기계가 수행하기 어렵다.
이 구분은 단순한 관찰을 넘어, “개발자가 무엇을 잘해야 하는가”를 재정의하는 프레임이다.

### DDD와 애자일의 재해석

저자는 DDD의 경계 문맥과 보편 언어, 애자일의 협업과 피드백 원칙을 LLM 맥락에서 재해석한다.
이 개념들은 원래 인간 간 소통을 위한 것이었지만, LLM과의 협업에서도 동일하게 적용된다.
일관된 어휘를 가진 코드베이스는 LLM에게도 더 명확한 신호를 제공하며,
애자일이 강조하는 빠른 피드백 루프는 LLM 생성 코드의 품질 검증에도 그대로 적용된다.

### 코드가 맥락이 된다는 역전

전통적으로 코드는 명세(specification)의 결과물이었다. 그러나 저자는 LLM 시대에
잘 설계된 코드 자체가 가장 중요한 맥락이자 가이드라인이 된다고 주장한다.
프롬프트나 외부 명세보다, 명확한 추상과 어휘를 가진 기존 코드가 LLM 출력의 품질을
더 효과적으로 제어한다는 통찰은, “좋은 코드를 쓰는 것”의 의미를 확장한다.

## 비평

### 강점: 실천적 관점에서의 명확한 재정의

이 글의 가장 큰 강점은 추상적인 철학 논의에 머물지 않고, 실제 개발 실천(TDD, DDD, 페어링)과
LLM 활용을 구체적으로 연결한다는 점이다. “인지 부채”라는 개념은 단순히 LLM 코드를 경계하라는
경고를 넘어, 왜 이해가 중요한지를 구조적으로 설명한다. 실무 개발자가 즉시 적용할 수 있는
방향성을 제공한다는 면에서 실용적 가치가 높다.

### 약점: 검증 과정의 부재

저자의 주장은 설득력 있지만, 대부분이 경험적 관찰과 논리적 추론에 의존한다.
LLM이 일관된 어휘를 가진 코드베이스에서 실제로 더 좋은 결과를 낸다는 주장을 뒷받침할
실증적 데이터나 사례 연구가 없다. 마찬가지로 “인지 부채”가 장기적으로 어떤 구체적 문제를
유발하는지에 대한 증거도 제시되지 않는다. 주장의 방향은 옳을 가능성이 높지만,
실증적 토대가 약하다는 점은 아쉽다.

### 누락된 관점: 어휘 수렴의 어려움

저자는 일관된 어휘의 중요성을 강조하지만, 팀 규모가 커지거나 레거시 시스템이 혼재할 때
어휘를 일관되게 유지하는 것이 얼마나 어려운지는 다루지 않는다. 현실의 대부분 코드베이스는
여러 시대, 여러 팀의 어휘가 충돌하는 혼종 상태다. 이런 환경에서 LLM이 도움이 되는지
방해가 되는지에 대한 논의가 빠져 있다.

## 인사이트

### LLM은 코드의 거울이다

LLM이 코드를 생성할 때 기존 코드베이스의 어휘와 구조를 맥락으로 삼는다는 사실은,
LLM을 일종의 거울로 이해하게 만든다. LLM은 좋은 코드에서 좋은 코드를 만들고,
나쁜 코드에서 나쁜 코드를 증식시킨다. 이는 단순히 “프롬프트를 잘 써야 한다”는 기술적 조언을
넘어, 코드베이스의 설계 품질이 LLM 도구의 효과를 결정하는 근본 변수라는 의미다.

즉, LLM 도입의 ROI는 도구 자체의 성능보다 기존 코드베이스의 품질에 더 크게 좌우된다.
기술 부채가 쌓인 코드베이스에 LLM을 도입하면, LLM은 그 부채를 더 빠른 속도로 증폭시킨다.
반대로 잘 설계된 코드베이스는 LLM을 더 정확하고 예측 가능하게 사용할 수 있게 해준다.
이는 레거시 시스템을 유지하는 조직과 그린필드 개발을 하는 조직 사이의 LLM 활용 격차를
기존보다 훨씬 크게 벌릴 것이다.

결국 LLM 도구는 코드 품질 격차를 줄이는 것이 아니라 확대하는 방향으로 작동할 가능성이 높다.
“좋은 코드를 쓰는 능력”의 경제적 가치는 LLM 시대에 하락하는 것이 아니라,
구조적으로 더 중요해지고 있다.

### 어휘의 지식재산화: 코드베이스가 경쟁 해자가 되는 시대

저자의 주장을 조직 수준으로 확장하면, 잘 설계된 도메인 어휘를 가진 코드베이스는
일종의 지식 자산이자 경쟁 해자가 된다는 결론에 이른다. 도메인 어휘가 명확하게 코드에
녹아 있는 조직은 LLM을 통해 더 빠르고 정확하게 기능을 구현할 수 있다.
반면 어휘가 불명확하거나 일관성이 없는 조직은 LLM이 오히려 혼란을 가중시킨다.

전통적으로 소프트웨어 경쟁력은 알고리즘, 인프라, 인재에서 나왔다. 그러나 LLM이
알고리즘 구현과 인프라 설정을 빠르게 상품화하면서, 경쟁력의 원천이 “도메인 어휘의 품질”로
이동할 수 있다. 이는 DDD를 “좋은 설계 관행” 수준으로 보던 시각에서 벗어나,
DDD가 조직의 전략적 자산을 구성하는 실천으로 재평가받게 될 가능성을 시사한다.

오픈소스 LLM이 코드 생성 비용을 0에 수렴시키는 미래에도,
“어떤 코드를 생성하게 할 것인가”를 결정하는 도메인 모델의 품질은 희소 자원으로 남는다.
결국 코드베이스 자체가 조직의 지적 DNA를 담는 그릇이 되는 셈이다.

### 수동 코드 검토의 역설: 읽기만 해서는 이해할 수 없다

저자가 “코드를 직접 작성하는 행위 자체가 깊은 사고를 유발한다”고 말하는 부분은,
LLM 코드 검토 워크플로우에 근본적인 문제를 제기한다. 많은 팀이 LLM이 생성한 코드를
“검토(review)”하는 방식으로 LLM을 도입하지만, 읽기만으로는 개념 모델에 대한 진짜 이해가
형성되지 않는다는 것이다.

이는 인지 과학적으로도 근거가 있다. 코드를 읽는 것은 수동적 처리이고,
코드를 작성하는 것은 능동적 구성이다. 개념 모델은 능동적 구성 과정에서 내면화된다.
LLM이 생성한 코드를 단순히 승인하거나 거절하는 워크플로우는, 개발자가 코드의 개념 구조를
이해하지 못한 채 코드베이스에 녹아들게 하는 구조적 경로가 된다.

이 역설은 “LLM 네이티브” 개발 방법론이 아직 해결하지 못한 핵심 문제를 드러낸다.
단순한 코드 생성 속도 향상이 아니라, 개발자가 LLM과 협력하면서도 개념적 이해를 유지할 수 있는
새로운 실천 방식이 필요하다. 페어 프로그래밍이 단순한 코드 공유를 넘어 이해를 공유하는
메커니즘이었듯, LLM 시대의 “페어링”은 생성된 코드를 함께 검토하는 것이 아니라
생성 과정에서 함께 개념을 형성하는 새로운 형태가 되어야 할 것이다.
