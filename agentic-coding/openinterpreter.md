# Open Interpreter — 저비용 모델에 최적화된 코딩 에이전트

<https://github.com/openinterpreter/openinterpreter>

## 소개

Open Interpreter는 저비용 언어 모델에서 최고의 성능을 끌어내는 데 초점을 맞춘
코딩 에이전트다.
GitHub Trending 상위권에 올랐으며 65,684개의 star, 5,659개의 fork를 보유하고
있다.
2023년 7월에 처음 만들어진 저장소지만, 지금 배포되는 버전은 OpenAI의 Codex를
포크해 완전히 새로 작성한 Rust 기반 구현이다.
원래의 Python 기반 프로젝트는 별도로 분리되어
`endolith/open-interpreter`에서 커뮤니티가 유지보수하고 있다.

핵심 주장은 “하네스 에뮬레이션(Harness Emulation)”이다.
모델 자체의 성능을 올리는 대신, 저비용 모델이 최상의 결과를 내도록 유도하는
에이전트 하네스(모델을 감싸는 실행 구조)를 갈아 끼울 수 있게 만든 것이다.
`/harness` 명령으로 `native`, `claude-code`, `claude-code-bare`, `zcode`,
`kimi-cli`, `qwen-code`, `deepseek-tui`, `swe-agent`, `minimal` 등 여러
하네스 중 하나를 즉시 전환할 수 있다.
같은 원리로 `/model` 명령을 사용하면 모델 제공자와 모델도 TUI 안에서 바로
바꿀 수 있다.

## 주요 기능

- macOS, Linux, Windows에서 네이티브 샌드박싱을 통해 명령을 실행한다.
- `/model`로 제공자와 모델을 전환하고, `/harness`로 Rust 네이티브 모델
  하네스를 점검하거나 바꾼다.
- 내장 QA 스킬로 웹 앱과 네이티브 앱을 테스트한다.
  [agent-browser](https://github.com/vercel-labs/agent-browser)로 실제
  브라우저를 조작하거나, [trycua](https://github.com/trycua/cua)로 네이티브
  앱을 조작하고 검증한다.
- `interpreter acp` 명령으로 [Agent Client
  Protocol](https://agentclientprotocol.com/) 에이전트로 동작해 에디터와
  연동한다.
- 설정과 세션 상태를 `~/.openinterpreter`에 로컬로 보관한다.
- `exec`, MCP, 스킬, 훅, 권한 체계, `AGENTS.md`를 지원한다.

## 설치 및 사용법

macOS와 Linux는 셸 스크립트로, Windows는 PowerShell 스크립트로 설치한다.

```bash
curl -fsSL https://www.openinterpreter.com/install | sh
```

```powershell
irm https://www.openinterpreter.com/install.ps1 | iex
```

설치 후 터미널에서 `i` 또는 `interpreter`를 입력하면 세션이 시작된다.
라이선스는 Apache-2.0이며, 저장소 언어 구성은 Rust가 대다수(약 41.6MB)를
차지하고 Python, Starlark(Bazel 빌드 파일), TypeScript, Shell,
PowerShell 등이 뒤를 잇는다.
빌드 시스템으로 Bazel을, 패키지 관리로 pnpm 워크스페이스를 사용한다.
최신 릴리스는 `rust-v0.0.25`(2026년 7월 15일)이며, 하루 단위로 패치 릴리스가
이어질 만큼 개발 속도가 빠르다.

## 분석

### “하네스”를 최적화 대상으로 승격시킨 설계

이 프로젝트의 진짜 발명은 모델이 아니라 하네스를 일급 개념으로 만든 것이다.
같은 모델이라도 어떤 프롬프트 구조, 어떤 도구 인터페이스, 어떤 반복 루프로
감싸는지에 따라 벤치마크 점수가 크게 달라진다는 사실은 이미 여러 코딩 에이전트
비교 실험에서 관찰돼 왔다.
Open Interpreter는 이 관찰을 제품 기능으로 바꿨다.
`claude-code`, `qwen-code`, `kimi-cli`처럼 특정 제공자·모델에 최적화된
하네스를 이름째로 이식해, 저비용 모델을 쓰더라도 그 모델을 설계한 곳이
검증한 상호작용 패턴을 그대로 재현하려 한다.
값비싼 모델의 “직관적 추론력”을 하네스 설계로 보완하겠다는 접근이다.

이 설계가 성립하려면 전제가 하나 필요하다.
모델의 성능 차이 중 상당 부분이 모델 자체의 한계가 아니라 하네스와의
불일치에서 온다는 전제다.
만약 저비용 모델의 한계가 정말로 추론 능력 자체의 부족이라면, 하네스를
아무리 정교하게 갈아 끼워도 상한선은 크게 오르지 않는다.
저장소가 이 전제를 얼마나 검증했는지는 README만으로는 확인할 수 없다.

### Codex 포크라는 계보가 말해주는 것

Open Interpreter가 OpenAI Codex의 포크로 다시 태어났다는 점은 단순한 이력이
아니라 전략적 신호로 읽을 수 있다.
Codex는 OpenAI가 자사 모델에 최적화해 설계한 에이전트 하네스다.
그 코드베이스를 가져와 “다른 제공자의 모델에도 잘 맞도록 하네스를
바꿔 끼운다”는 방향으로 재구성한 것은, 최상위 코딩 에이전트의 실행 구조가
특정 벤더에 종속될 이유가 없다는 주장을 담고 있다.
동시에 이는 이 생태계에서 하네스 구현 자체가 경쟁 우위의 원천으로
분리되고 있다는 흐름과도 맞닿아 있다.
모델 API는 갈수록 상품화(Commoditize)되는 반면, 그 모델을 잘 다루는
껍데기 쪽에서 새로운 부가가치가 생겨나는 구도다.

### 로컬 우선(Local-first) 설계와 개발자 신뢰의 관계

설정과 세션 상태를 `~/.openinterpreter`에 저장하고, `AGENTS.md`나 훅,
권한 체계를 지원하는 방식은 최근 코딩 에이전트들이 수렴하는 공통 패턴이다.
샌드박스 안에서 명령을 실행하고 승인 절차를 두는 것도 마찬가지다.
이는 에이전트가 파일 시스템과 셸에 직접 접근하는 도구인 이상, 신뢰를
얻기 위한 최소 조건이 “로컬에서 통제 가능하다”는 감각을 주는 것으로
굳어지고 있음을 보여준다.
반대로 말하면, 이 조건을 갖추지 못한 에이전트 도구는 초기 채택 단계부터
불리해진다는 뜻이기도 하다.

## 비평

### “하네스 전환”의 실제 효과는 README에서 검증되지 않는다

README는 `/harness`로 여러 하네스를 전환할 수 있다는 기능만 설명할 뿐,
전환에 따른 실제 성능 차이를 수치로 제시하지 않는다.
어떤 벤치마크에서, 어떤 모델에, 어떤 하네스를 적용했을 때 얼마나 점수가
올랐는지에 대한 근거가 전혀 없다.
“저비용 모델에 최적화된 코딩 에이전트”라는 프로젝트의 정체성 자체가 이
주장 위에 서 있는데, 정작 그 핵심 주장을 뒷받침할 데이터는 README 수준에서
찾을 수 없다.
스타 수나 릴리스 속도 같은 인기 지표는 이 주장의 타당성과 무관하다.
채택이 활발하다는 사실이 하네스 전환의 실제 효과를 증명하지는 않는다.

### 이식된 하네스는 원본 하네스와 동등하지 않을 가능성이 크다

`claude-code`, `qwen-code`, `kimi-cli` 같은 이름의 하네스를 자체 구현으로
재현했다고 해서 원본 제품이 실제로 사용하는 프롬프트, 도구 정의, 반복
전략까지 동일하게 재현했다고 보장할 수 없다.
각 벤더는 자사 하네스의 세부 구현(시스템 프롬프트, 도구 설명, 오류 복구
로직)을 공개하지 않는 경우가 많고, 공개하더라도 지속적으로 갱신한다.
Open Interpreter가 이름만 빌려온 근사치를 제공하는 것이라면, 사용자는
“claude-code 하네스를 쓴다”는 문구에서 기대하는 성능과 실제로 얻는 성능
사이에 괴리를 겪을 수 있다.
이 괴리 가능성에 대해 README는 아무 언급도 하지 않는다.

### 프로젝트명 승계가 만드는 혼동은 사용자 비용으로 전가된다

기존 `open-interpreter`라는 이름과 브랜드를 가져오면서 원래 프로젝트를
완전히 다른 언어, 다른 아키텍처, 다른 철학의 새 프로젝트로 대체한
결정은 README의 각주 하나로 처리되고 있다.
“Python 버전을 찾는다면 `endolith/open-interpreter`로 가라”는 안내는
친절해 보이지만, 실제로는 기존 사용자·검색엔진·문서·튜토리얼이 가리키던
모든 참조가 하루아침에 다른 대상을 가리키게 됐다는 뜻이다.
이런 승계는 신규 사용자에게는 매끄러운 브랜드 연속성으로 보이지만,
기존 생태계에 뿌리를 둔 사용자에게는 실질적인 전환 비용을 발생시킨다.
이 비용을 누가 부담하는지에 대한 논의가 README에는 없다.

## 인사이트

### 하네스 시장이 형성되면 모델 벤치마크의 의미가 흔들린다

특정 모델의 성능을 이야기할 때, 이제는 “어떤 모델이 강한가”보다 “어떤
모델을 어떤 하네스로 감쌌을 때 강한가”를 물어야 하는 상황이 만들어지고
있다.
Open Interpreter처럼 하네스 자체를 교체 가능한 부품으로 만드는
프로젝트가 늘어나면, 동일한 모델이 하네스에 따라 벤치마크 순위를
오르내리는 일이 흔해진다.
이는 모델 제공자 입장에서 통제하기 어려운 변수가 하나 늘어난다는 뜻이다.
자사가 설계하지 않은 하네스에서 자사 모델이 저평가되거나, 반대로
경쟁사가 자사 모델을 위해 만든 정교한 하네스 덕분에 예상보다 좋은
평가를 받는 상황이 동시에 발생할 수 있다.
결국 벤치마크 리더보드는 “모델 대 모델”이 아니라 “모델-하네스 조합 대
모델-하네스 조합”으로 재편될 가능성이 있으며, 이는 현재의 벤치마크
발표 관행(모델명만 명시하고 하네스는 각주로 처리하는 방식)이 갖는
설명력을 약화시킨다.

### 포크의 정체성 문제는 오픈소스 거버넌스의 오래된 취약점을 재현한다

이름을 유지한 채 구현체를 통째로 교체하는 방식은 새로운 현상이 아니다.
과거에도 유지보수자가 바뀌거나 프로젝트가 인수되면서 이름은 그대로인데
코드베이스와 철학이 완전히 달라진 사례가 여러 차례 있었다.
차이가 있다면 이번에는 그 교체가 “AI 에이전트 하네스”라는, 사용자가
파일 시스템 접근 권한까지 위임하는 민감한 도구에서 벌어졌다는 점이다.
기존 `open-interpreter`를 신뢰해 로컬 환경에 설치해 두었던 사용자가
버전을 갱신했을 때, 자신이 검토했던 Python 코드와는 전혀 다른 Rust
바이너리가 같은 이름으로 실행된다는 사실을 얼마나 명확히 인지할 수
있을지는 별개의 문제다.
이는 패키지 이름의 소유권 이전이 보안 검토 프로세스를 우회하는 통로가
될 수 있다는, 공급망 보안 논의에서 반복적으로 지적돼 온 취약점과
구조적으로 닮아 있다.

### “저비용 모델 최적화”는 모델 가격 하락 속도와 경쟁하는 목표다

이 프로젝트의 존재 이유는 저비용 모델의 부족한 능력을 하네스 설계로
메우는 데 있다.
그런데 모델 제공자들은 매 분기 더 저렴하면서도 더 강력한 모델을
내놓고 있고, 그 격차는 하네스가 메워야 할 간극을 계속 줄이는 방향으로
움직인다.
즉 이 프로젝트가 해결하려는 문제 자체가, 문제를 만든 당사자들(모델
제공자)의 정상적인 발전 속도에 의해 자연스럽게 축소되는 구조다.
장기적으로 하네스 최적화가 갖는 가치는 “저비용 모델을 얼마나 잘
감싸는가”가 아니라 “어떤 하네스 설계든 모델이 바뀌어도 통용되는
범용 실행 패턴을 발견했는가”로 옮겨갈 가능성이 크다.
지금의 하네스 목록이 특정 벤더 이름을 그대로 딴 것은, 아직 그 단계의
범용 패턴을 추출하지 못했다는 방증으로도 읽을 수 있다.
