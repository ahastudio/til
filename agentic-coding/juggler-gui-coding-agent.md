# Juggler: 도구 호출, 분기, 편집 가능한 컨텍스트를 다루는 GUI 코딩 에이전트

<https://github.com/juggler-ai/juggler>

HN 토론: <https://news.ycombinator.com/item?id=48883305> (274점, 118개 댓글)

## 소개

Juggler는 LLM이 코드베이스에 가하는 작업을 사용자가 직접 손으로 다루고
싶어 하는 사람들을 위해 만들어진 AI 코딩 에이전트다.
단순한 채팅 인터페이스 대신 “검사 가능한 도구 호출, 분기되는 스레드,
편집 가능한 컨텍스트”를 갖춘 시각적 작업대(workbench)를 제공한다는 것이
프로젝트의 핵심 표어다.

가장 눈에 띄는 설계는 대화를 선형 채팅 기록이 아니라 편집 가능한 트리로
다룬다는 점이다.
사용자는 대화 중 어떤 지점에서든 하위 스레드로 분기하여 중첩된 논의를
만들 수 있고, 파인더(Finder) 스타일의 밀러 컬럼(Miller columns)으로
탐색한다.
왼쪽에 루트가 있고, 항목을 선택하면 그 속성과 자식 항목이 오른쪽으로
펼쳐지는 구조다.

거의 모든 구성 요소가 자바스크립트 확장(extension)으로 이루어져 있다.
파일 읽기, 텍스트 치환, bash 명령 실행 같은 컨텍스트 항목, “plan”이나
“research” 같은 오케스트레이션 패턴을 담은 전략(strategy), 슬래시
명령까지 모두 확장으로 구현된다.
이 방식 덕분에 터미널 기반 스크립팅 없이도 커스텀 UI와 시각화를 만들 수
있다.

애플리케이션은 로컬 웹서버로 동작하며 하나의 협업 세션을 관리한다.
데스크톱 앱, 브라우저 탭, 원격 머신 등 여러 클라이언트가 동시에 같은
서버에 접속해 모든 화면에서 동기화된 상태를 유지할 수 있다.
서버는 기본적으로 로컬호스트에서만 열리며, `p` 키를 누르면 비밀번호
없이 신뢰할 수 있는 네트워크에서 접근 가능한 LAN 모드를 켤 수 있다.

Claude Code(CLI/API), OpenAI(Codex/API), Gemini, Ollama, OpenRouter,
Z.AI, Deepseek 등 다양한 모델 공급자를 지원하며 추가 공급자를 붙이는
과정도 단순하게 설계되어 있다.
`juggler.studio`에서 macOS용 DMG, Windows용 실행 설치 파일, Linux용
헤드리스 바이너리를 배포한다.
백엔드는 Go로 작성되었고 창 관리에는 Wails를 사용하며, 프론트엔드는
HTML/자바스크립트로 구성된다.
세션 문서는 Yjs로 저장되고 동기화되며, 확장은 TypeScript 컴파일 없이
JSDoc 기반 타입 검사만으로 작성된다.

라이선스는 이중 구조다.
애플리케이션 본체는 AGPL-3.0 이상을 따르지만, 확장 SDK와 번들 확장은
Apache-2.0을 따른다.
이 덕분에 카피레프트 의무 없이 비공개 소스 확장을 만들 수 있다.
저장소는 별 433개, 포크 22개를 기록하고 있으며 최신 릴리스는 v0.3.8이다.

Juggler는 오디오 개발 프레임워크 JUCE와 디지털 오디오 워크스테이션
Tracktion을 만든 개발자 julesrms가 혼자 스페어타임에 만든 프로젝트다.
Hacker News의 Show HN 스레드에서 그는 이 프로젝트가 전적으로 개인
시간에 무료로 만든 것이며, 자금도 없고 앞으로 어떻게 이어갈지에
대한 구체적 계획도 없다고 밝혔다.
사용자가 많아지면 정규직으로 전환할 방법을 찾아보고 싶다는 바람도
덧붙였다.[^julesrms]

## 분석

### “채팅”이 아니라 “작업대”라는 프레이밍이 노리는 것

Juggler가 스스로를 채팅 인터페이스가 아니라 workbench로 규정하는 것은
단순한 마케팅 문구가 아니라 설계 철학의 선언이다.
대부분의 코딩 에이전트 UI는 사람과 모델의 대화를 위에서 아래로 흐르는
로그로 취급하고, 도구 호출과 그 결과는 접혀 있는 상세 패널 뒤로
숨긴다.
이 구조는 “결과가 맞으면 과정은 몰라도 된다”는 암묵적 태도를 반영한다.

Juggler는 정반대의 태도를 취한다.
도구 호출, 승인, 스레드 구조, 항목 속성, 원시 컨텍스트를 모두 접지 않고
밀러 컬럼에 펼쳐 보여주는 것은, 에이전트가 무엇을 근거로 판단했는지를
사용자가 항상 추적할 수 있어야 한다는 전제 위에 서 있다.
이것은 자율성이 높아질수록 검증이 느슨해지는 경향에 대한 구조적
반작용으로 읽을 수 있다.
UI 차원에서 “설명 가능성”을 강제하는 접근이라는 점에서, 이 프로젝트는
에이전트의 자율성을 늘리는 방향이 아니라 사람의 감독 대역폭을 넓히는
방향으로 움직이고 있다.

### 트리형 대화가 해결하려는 실제 문제

선형 채팅 기록의 근본적인 한계는, 대화가 곁가지로 빠졌을 때 그 곁가지를
버리거나 전체 맥락을 다시 설명해야 한다는 데 있다.
코딩 작업에서는 “이 함수를 리팩터링하다가 별개의 버그를 발견해서
잠깐 딴 데를 파야 하는” 상황이 흔한데, 선형 채팅은 이런 분기를 자연스럽게
표현하지 못한다.

Juggler가 도입한 분기 스레드는 이 문제에 대한 직접적인 답이다.
어떤 지점에서든 하위 스레드로 갈라져 나가 독립적으로 논의를 진행하고,
필요하면 그 결과만 상위 컨텍스트로 되돌릴 수 있다.
이는 버전 관리 시스템에서 브랜치와 머지가 하는 역할을 대화 자체에
적용한 것에 가깝다.
코드에 브랜치가 필요했던 이유와 대화에 브랜치가 필요한 이유는 본질적으로
같다.
독립적으로 실험할 수 있는 공간이 없으면 실험 자체를 피하게 된다는
점이다.

HN 스레드에서 gnarlouse는 이 지점을 정확히 짚었다.
LLM과의 상호작용이 등장한 지 3년 가까이 되었는데도 레딧이나 슬랙처럼
본 대화에서 곁가지로 빠지는 "곁다리" 스레드를 만드는 기능조차 제대로
갖추지 못했다는 것이 놀랍다는 반응이었다.[^gnarlouse]
반대로 anigbrowl은 Cline 같은 기존 도구를 쓰면서 가장 답답했던 지점이
바로 "끝없이 아래로 흐르는 스크롤"이었다며, 밀러 컬럼이 그 문제를
정확히 해결한다고 평가했다.
그는 Go 백엔드에 Wails를 써서 Electron을 피한 점, 순수 타입 체크된
JS를 쓴 점, 문서 동기화에 Yjs를 쓴 점까지 기술 스택 전체를 짚으며
지지를 표했다.[^anigbrowl]

### 확장 기반 아키텍처가 감수하는 트레이드오프

컨텍스트 항목, 전략, 슬래시 명령까지 전부 자바스크립트 확장으로 구현한
결정은 유연성을 극대화하지만, 동시에 핵심 로직의 상당 부분을 서드파티
코드의 손에 맡긴다는 뜻이기도 하다.
TypeScript 컴파일 없이 JSDoc만으로 타입을 검사하는 방식은 확장 작성의
진입 장벽을 낮추지만, 대규모 확장 생태계가 형성되었을 때 타입 안정성을
어디까지 보장할 수 있을지는 불확실하다.

이 선택은 Juggler가 “에이전트 프레임워크”보다는 “에이전트를 위한 셸
또는 플랫폼”에 가깝다는 것을 시사한다.
오케스트레이션 전략 자체가 코어에 고정되어 있지 않고 확장으로
분리되어 있다는 것은, 이 프로젝트가 특정한 에이전트 워크플로를
강요하지 않고 사용자가 자신의 워크플로를 조립하도록 설계되었다는
뜻이다.
이는 최근 여러 로컬 코딩 에이전트 도구들이 공유하는 흐름과 맞닿아
있다.
즉 에이전트 자체보다 에이전트를 운용하는 하네스(harness)의 설계가
차별화 지점이 되고 있다는 흐름이다.

solarkraft는 HN 댓글에서 이 구조적 선택의 이유를 CRDT 기반 로컬
웹서버 위에 얹혀 있다는 점에서 찾았다.
그는 하네스를 UI 안에 직접 끼워 넣으려는 프로젝트가 너무 많은데, 이는
시스템을 바꾸거나 그 동작을 추론할 인터페이스 자체를 없애버리기
때문에 잘못된 방향이라고 지적했다.
하네스는 최대한 작고 독립적이어야 하며, 언젠가는 UI를 포함해 이
영역에도 표준 인터페이스가 필요하다는 것이 그의 바람이었다.[^solarkraft]
반면 everforward는 Juggler가 아무리 매력적이어도 자신이 이미 쓰고
있는 Pi 플러그인들을 다시 작성해야 한다는 점이 실제 전환을 막는
장벽이라고 지적하며, ACP(Agent Client Protocol) 지원이 있었다면
진지한 대안으로 고려했을 것이라고 말했다.
이는 확장 기반 아키텍처의 유연성이 기존 생태계와의 호환성 부족이라는
대가를 치른다는 점을 보여주는 사례다.[^everforward]

## 비평

### “가시성”과 “실제 검증”은 다른 문제다

Juggler의 핵심 주장은 도구 호출과 컨텍스트를 전부 화면에 펼쳐 보여주면
사용자가 에이전트를 더 잘 통제할 수 있다는 것이다.
그러나 정보가 화면에 있다는 것과 사용자가 그것을 실제로 검토한다는
것은 전혀 다른 문제다.
밀러 컬럼에 도구 호출이 나열되어 있어도, 세션이 길어지고 호출 수가
늘어나면 사람은 결국 승인 버튼을 누르는 데 익숙해지고 내용을 훑지
않게 된다.
이것은 코드 리뷰에서 흔히 관찰되는 “리뷰 피로” 현상과 동일한 패턴이며,
UI가 정보를 숨기지 않는다고 해서 자동으로 회피되지는 않는다.
결국 가시성은 검증의 필요조건이지 충분조건이 아닌데, README는 이
둘을 사실상 동일시하는 어조로 서술한다.

### LAN 접근 모드는 편의와 위험을 등가교환하지 않는다

`p` 키 하나로 비밀번호 없이 LAN 접근을 열 수 있다는 설계는 “신뢰할 수
있는 네트워크”라는 전제에 기대고 있다.
그러나 실제 환경에서 신뢰할 수 있는 네트워크와 신뢰할 수 없는 네트워크의
경계는 사용자가 생각하는 것보다 훨씬 자주 무너진다.
카페나 공유 오피스의 와이파이, 회사 VPN에 물린 게스트 네트워크, IoT
기기가 뒤섞인 가정용 공유기 등은 모두 “로컬”이라는 이름으로 묶이지만
신뢰 수준은 제각각이다.
게다가 이 서버가 다루는 것은 단순한 파일 공유가 아니라 코드 실행
권한을 가진 에이전트 세션이다.
비밀번호 없는 LAN 노출을 단축키 하나로 켤 수 있게 만든 것은, 편의성을
위해 공격 표면을 늘리는 결정을 사용자의 순간적 판단에 맡긴 셈이다.
기본값이 안전하고 예외적으로 위험을 여는 구조가 아니라, 위험을 여는
행위 자체를 지나치게 쉽게 만들어 둔 것이 문제다.

실제로 HN에서 markasoftware는 원격 연결 기능이 정상 작동하지 않는
것 같다고 보고했다.[^markasoftware]
네트워크 노출 기능 자체가 아직 충분히 검증되지 않았을 가능성을
시사하는 대목이며, 편의 기능의 안전성 설계는 그 기능이 실제로
안정적으로 동작한다는 전제 위에서만 의미를 가진다는 점을 다시
확인시켜 준다.

### 다중 공급자 지원이 감추는 행동 차이

Juggler는 Claude Code, OpenAI, Gemini, Ollama, OpenRouter 등 여러
모델 공급자를 하나의 인터페이스로 통합한다고 소개한다.
그러나 이 모델들은 도구 호출 신뢰도, 컨텍스트 창 처리 방식, 거부
패턴, 코드 생성 스타일에서 상당히 다르게 행동한다.
동일한 밀러 컬럼 UI와 동일한 확장 시스템으로 이 모델들을 감싼다고
해서, 사용자가 체감하는 신뢰성과 예측 가능성까지 동일해지지는
않는다.
README는 “추가 공급자를 붙이는 과정이 단순하다”는 점을 장점으로
제시하지만, 통합의 용이성과 통합된 이후의 행동 일관성은 서로 다른
축이다.
사용자가 공급자를 바꿔가며 같은 확장, 같은 전략을 쓸 수 있다는
기대를 주는 순간, 실제로는 공급자마다 다르게 작동하는 부분을
스스로 파악해야 하는 부담이 사용자에게 넘어간다.

## 인사이트

### 트리형 대화 UI는 결국 버전 관리의 언어를 코딩 에이전트로 되가져오는 흐름의 일부다

Git이 코드에 브랜치와 머지라는 개념을 도입했을 때, 그것은 여러
사람이 동시에 독립적으로 실험하고 나중에 통합할 수 있게 해주는
구조였다.
Juggler의 분기 스레드는 같은 개념을 사람과 에이전트 사이의 대화에
적용한 것이다.
흥미로운 점은 이 흐름이 Juggler만의 독창적 발명이 아니라, 최근 여러
코딩 에이전트 도구들이 독립적으로 수렴하고 있는 패턴이라는 것이다.
세션을 분기하고, 특정 지점으로 되돌리고, 여러 시도를 병렬로 펼친 뒤
좋은 결과만 골라 병합하는 방식은 코드 자체의 버전 관리 관행이 대화
자체의 버전 관리로 확장되고 있다는 신호다.
이것이 의미하는 바는 단순히 UI 편의 기능이 아니다.
에이전트와의 상호작용이 “한 번의 대화”가 아니라 “여러 후보 경로 중
하나를 선택하는 탐색 과정”으로 재정의되고 있다는 뜻이며, 이는 향후
에이전트 도구들이 실험(브랜치), 비교(diff), 선택(머지)이라는 세
가지 동작을 얼마나 잘 지원하느냐로 경쟁하게 될 가능성을 시사한다.

이 흐름에는 대가도 따른다.
HN에서 Max-q는 2년 전 자신도 트리 기반 에이전트를 시도했지만 당시
LLM이 트리를 효과적으로 탐색하지 못해 실패했다고 회고하면서, 지금은
모델도 좋아지고 이를 다루는 이해도 나아졌다고 평가했다.
동시에 그는 분기 구조가 캐시된 데이터를 재사용할 여지를 줄여 토큰
사용량이 훨씬 높아진다는 단점을 지적했다.[^Max-q]
가시성과 탐색 유연성을 얻는 대신 토큰 비용을 지불하는 셈이며, 이
트레이드오프는 트리형 대화 UI가 널리 퍼질수록 더 분명하게 드러날
것이다.
hypfer는 이런 시도 자체가 드물다는 점을 짚으며, 실제로 쓸 만한
하네스의 숫자가 여전히 극소수에 불과하다고 꼬집었다.
겉멋이나 정체성 놀음이 아니라 실제 생산성을 위해 만들어진 소프트웨어를
반긴다는 반응이었다.[^hypfer]

### 가시성 중심 설계는 장기적으로 새로운 형태의 자동 승인 압력을 만들어낼 수 있다

Juggler가 도구 호출과 컨텍스트를 전부 펼쳐 보여주는 것은 단기적으로는
투명성을 높이지만, 장기적으로는 역설적인 결과를 낳을 수 있다.
정보량이 늘어날수록 사람이 모든 것을 검토하는 비용도 함께 늘어나고,
비용이 일정 수준을 넘으면 사람은 검토를 포기하고 습관적으로 승인하는
쪽으로 적응한다.
이것은 보안 경고창이 너무 자주 뜨면 사용자가 내용을 읽지 않고 무조건
“확인”을 누르게 되는 현상과 같은 구조다.
Juggler의 밀러 컬럼이 실제로 가치를 가지려면, 단순히 정보를 노출하는
것을 넘어 어떤 도구 호출이 특별히 위험하거나 비가역적인지를 구분해
주의를 차등적으로 배분하는 장치가 필요하다.
현재 README에 서술된 설계는 “모두 보여준다”에 머물러 있고, “무엇을
먼저 봐야 하는가”에 대한 답은 확장 개발자나 사용자 각자에게 맡겨져
있다.

이 긴장은 HN 댓글에서도 드러난다.
digitaltrees는 완전 자율로 장시간 실행되는 에이전트를 밀어붙이는
최근 흐름에 불만을 표하며, 사람이 중간에 궤도를 수정할 수 없다는
점이 토큰만 태우고 결과는 더 나쁘게 만드는 원인이라고 지적했다.
Juggler처럼 자동화와 사람의 개입 사이에서 균형을 찾으려는 시도를
반기는 반응이었다.[^digitaltrees]
가시성을 높이는 설계가 실제로 이 균형에 기여하려면, 단순히 자동화를
늦추는 것을 넘어 사람이 개입해야 할 순간을 정확히 짚어 주는 장치가
필요하다는 점에서 이 댓글은 앞서 지적한 한계와 같은 방향을 가리킨다.

### 로컬 우선(local-first) 아키텍처는 코딩 에이전트 시장에서 신뢰의 새로운 통화가 되고 있다

Juggler가 로컬 웹서버를 기본 형태로 채택하고, 클라우드 서비스가 아니라
데스크톱 앱과 커맨드라인 서버를 동시에 배포하는 방식은 우연이
아니다.
코딩 에이전트가 다루는 데이터는 회사의 소스 코드, 인증 정보, 내부
설계 문서처럼 민감도가 높은 자산이다.
클라우드에 세션과 컨텍스트가 저장되는 구조는 이런 자산을 제3자
인프라에 위탁하는 것을 의미하며, 많은 조직에서 이는 정책적으로 받아들이기
어렵다.
Juggler가 Yjs 기반의 로컬 문서 동기화와 멀티 클라이언트 접속을
선택한 것은, 클라우드의 편의성을 포기하는 대신 데이터 주권을 사용자
손에 남겨두는 쪽을 택한 것이다.
이는 개별 프로젝트의 선택을 넘어, 코딩 에이전트 시장이 “얼마나
똑똑한가”뿐 아니라 “내 코드가 어디에 머무는가”를 기준으로도
분화되고 있다는 더 넓은 흐름을 보여준다.
장기적으로는 로컬 우선 여부가 기업의 도구 채택 결정에서 성능만큼이나
중요한 변수로 자리 잡을 가능성이 있다.
