## 소개

“니가 알아서 해줘”라는 지시가 AI에게 반복적으로 실패하는 이유, 그리고 그 구조를 이해했을 때 가능해지는 것들을 다루는 책이다.
Planner · Generator · Evaluator 세 역할의 분리 구조, 역할 간 파일 기반 매개체 시스템, 위임 수준 프레임워크, Claude Code 플러그인 구현까지를 플레이북 형식으로 제시한다.

이 책은 세 지적 계보를 통합한다.
Ian Goodfellow의 GAN(2014)이 제시한 “만드는 자와 판정하는 자는 같은 시스템이 겸해서는 안 된다”는 원리.
Anthropic이 제안한 Planner-Generator-Evaluator 3역할 구조.
SDD(Spec-Driven Development)가 정립한 사양 파일 기반 개발 방식.
이 세 흐름을 하나의 실행 가능한 플레이북으로 통합하는 것이 이 책의 핵심 기여다.

## 에이전틱 AI의 진화와 세 가지 구조적 실패

### AI의 세 단계 진화

생성형 AI(2014–2022)는 요청-응답의 단발 구조에 머물렀다.
AI 에이전트(2022–2024)는 도구 사용과 다단계 계획이 가능해졌지만, 방향을 잃거나 자신의 결과를 “완료”로 스스로 평가하는 편향이 반복됐다.
에이전틱 AI(2024–현재)는 긴 컨텍스트 창(최대 200만 토큰), 코드 실행 환경, 멀티 에이전트 오케스트레이션을 통해 장기 자율 실행이 가능해졌다.

그런데 역설이 있다. “AI가 더 자율적이 될수록, 구조 없이 운영했을 때의 위험도 함께 커진다.”

LangChain의 2024년 벤치마크에서 구조화된 역할 분리가 태스크 완료율을 52.8%에서 66.5%로 높였다.

### 세 가지 구조적 실패

**자기 평가 편향(Self-Evaluation Bias)**: 에이전트가 자신의 결과물을 평가할 때 구조적으로 관대해진다.
이는 능력의 문제가 아니라 아키텍처의 문제다.
GAN 원리가 보여주듯, Generator와 Discriminator가 같은 시스템이면 품질 향상이 일어나지 않는다.
특히 주관적 영역(디자인, UX, 창의성)에서 두드러진다.

**컨텍스트 불안(Context Anxiety)**: 세션이 길어질수록 에이전트가 초기에 받은 지시를 잊어버린다.
“배포 완료”가 “코드 작성 완료”로 슬그머니 바뀐다.
완료 정의가 컨텍스트 창이 채워지면서 축소된다.

**역할 혼동(Role Conflation)**: 계획, 실행, 평가 역할이 한 에이전트 안에서 뒤섞이면 구조적 충돌이 발생한다.
Planner가 구현 세부를 결정하면 Generator의 창의적 실행 공간이 사라진다.
Generator가 합격 기준을 정하면 편향이 주입된다.
Evaluator가 구현 방식을 알면 독립성을 잃는다.

### 엔지니어링 패러다임의 세 단계

**프롬프트 엔지니어링**: “AI에게 어떻게 말할 것인가.” 단일 Q&A와 집중 태스크에 강하지만 긴 세션에서 약하다.

**컨텍스트 엔지니어링**: “AI에게 무엇을 보여줄 것인가.” 복잡한 데이터 처리에 강하지만 결과 검증이 약하다.

**하네스 엔지니어링**: “AI가 작업을 완수할 수 있는 구조를 설계하는 것.” 이전 두 패러다임을 포괄하면서 운영 구조를 추가한다. 세 패러다임은 경쟁하지 않고 누적된다. 프롬프트를 잘 써야 한다는 것은 여전히 사실이다.

## PGE 모델: Planner · Generator · Evaluator

### 왜 세 역할인가

GAN의 Discriminator 역할이 코드 생성에서 두 가지로 나뉜다.
기준 설정(Planner)과 기준 적용(Evaluator)이다.
낙관 편향을 차단하려면 기준을 만드는 자와 기준을 적용하는 자 모두 구현자와 분리되어야 한다.

세 역할이 분리된다는 것은 반드시 세 에이전트가 필요하다는 의미가 아니다.
핵심은 관점의 독립성이다.
솔로 개발자도 시간적 분리, 입력 제한, 도구 분리로 동일한 효과를 달성할 수 있다.

### 역할의 정의

**Planner**: “무엇을, 왜”를 정의한다. Constitution(mission.md, tech-stack.md, roadmap.md)과 Feature Spec의 Planner 영역(plan.md, requirements.md, operations.md 상단)을 작성한다. validation.md는 절대 작성하지 않는다.

**Generator**: “동작하는 증분”을 만든다. 코드 작성에서 헬스체크 통과까지 6단계 완료 체인을 책임진다. requirements.md와 validation.md를 읽기 전용 입력으로 받는다.

**Evaluator**: Sprint 전 합격 기준(validation.md)을 작성하고, Sprint 후 3계층으로 판정한다. plan.md를 읽지 않는다. 구현 방식을 모르는 상태에서 기준을 세울 때 더 엄격해진다.

### 한 Sprint의 시간 흐름

1. Planner가 requirements.md 전달
2. Evaluator가 validation.md 및 operations.md 하단 작성 (구현 시작 전)
3. Generator와 Evaluator가 Sprint Contract 협상
4. Generator가 동작하는 증분 생성
5. Evaluator가 3계층으로 판정

Evaluator가 Generator보다 먼저 작업을 시작한다는 점이 핵심이다.

### 에이전트 주도 전제와 HITL

이 책의 전제는 에이전트가 세 역할을 주도하고 사람은 특정 결정 지점에만 개입한다는 것이다.

**과도기적 HITL(Transitional HITL)**: 하네스가 아직 성숙하지 않아 필요한 임시 개입. Evaluator의 문제 발견 기록, Rubric 튜닝, Sprint Contract 협상 패턴이 축적되면 줄어든다. 자동화 가능하다.

**영속적 HITL(Persistent HITL)**: 구조적으로 필요한 개입 영역. 가치·전략 판단, 법적·윤리적 책임, 외부 이해관계자 합의, 되돌릴 수 없는 위험이 해당된다. 에이전트의 입력 공간 밖에 있으므로 모델이 아무리 좋아져도 사라지지 않는다.

### 하네스의 비용

하네스는 공짜가 아니다.
솔로 에이전트보다 20배 이상 비싸다.
파일, 세션, 템플릿, Rubric 관리의 복잡도가 추가된다.
하네스 대부분의 구성 요소는 모델이 좋아지면 감량 가능하다.
단, Generator–Evaluator 분리만은 절대 감량하지 않는다. GAN의 구조적 분리는 모델 성능 향상으로 대체될 수 없다.

## Planner의 역할

### 입력의 비대칭

“일기 앱 만들어줘”처럼 짧은 입력이 거대한 구현 공간을 열어놓는다.
입력이 작을수록 에이전트는 훈련 데이터의 일반적 패턴으로 공백을 채운다.
Planner의 첫 작업은 입력을 확장하는 것이다. 사용자의 맥락을 질문을 통해 끌어내야 한다.

### Constitution: 프로젝트의 헌법

Constitution은 프로젝트 수준에서 한 번만 확정되는 세 가지 문서다.

**mission.md**: 누구의 무슨 문제를 왜 푸는가. 한 페이지 이내. “만들지 않을 것” 목록이 필수다. 배제 조항이 없으면 mission이 모든 피처 요청을 수용하는 고무줄이 된다.

**tech-stack.md**: 무엇으로 만드는가. 선택의 근거를 함께 남기는 것이 핵심이다. 근거 없으면 3개월 뒤 아무도 바꾸지 못한다.

**roadmap.md**: 무슨 순서로 만드는가. 피처가 왜 먼저인지, 어떤 전제가 되는지 기록한다. 구체적 구현 방법은 포함하지 않는다.

### Feature Spec의 Planner 영역

**plan.md**: 어떤 접근을 취할 것인가. 코드가 아니다. 코드 블록이 들어가면 경계를 넘은 것이다. Generator에게 구현의 자유도를 부여한다.

**requirements.md**: 무엇이 되어야 하는가. 모든 문장이 참/거짓으로 판정 가능해야 한다. “안정적으로 작동한다”는 요구사항이 아니다.

**operations.md 상단**: 배포 환경, 배포 절차, 롤백 전략. Full Stack DevOps 시대에는 배포까지가 “완료”다.

### Planner가 넘지 말아야 할 두 선

첫째, 구현 세부사항을 쓰지 않는다. 외부 API 계약은 예외다.

둘째, validation.md를 쓰지 않는다. Planner가 자기 요구사항의 합격 기준을 쓰면 기준이 필연적으로 느슨해진다. 이는 게으름이 아니라 구조적 불가피성이다. Planner는 requirements를 작성하면서 “이렇게 구현될 것”이라는 희미한 가설을 품는다. 이 가설이 무의식적으로 합격 기준의 기준선이 된다.

### Constitution 인터뷰: 영속적 HITL의 대표 사례

mission.md 작성은 영속적 HITL의 전형이다.
발산 단계에서 에이전트가 많은 질문을 던진다. “누가 쓰는가”, “지금은 어떻게 해결하는가”, “만들지 말아야 할 것은”.
수렴 단계에서 에이전트가 초안을 작성하고 사용자가 검토한다.
마지막 질문인 “절대 하지 말 것”이 mission의 배제 조항을 만들고, 이것이 roadmap 우선순위 기준이 된다.

## Generator의 역할

### “동작하는 증분”의 정의

Generator의 완료는 6단계 체인 모두 통과한 상태다.

1. 코드 작성
2. 테스트 통과
3. 로컬 빌드 성공
4. CI 파이프라인 통과
5. 스테이징 배포 성공
6. 헬스체크 통과

배포되지 않은 코드는 검증되지 않은 코드다. 부분 완료는 완료가 아니다.

### 작업 단위 계층

Task(가장 작은 실행 단위) → Feature(사용자에게 가치를 전달하는 단위) → Sprint(여러 Feature를 묶는 기간 단위). 각 단위마다 6단계 완료가 요구된다.

### Generator의 두 입력

requirements.md(Planner 작성)와 validation.md(Evaluator 작성)를 읽기 전용으로 받는다.
기준 변경이 필요하면 Sprint Contract 협상을 통한다. Generator가 직접 수정하지 않는다.

### Red 시작 원리

TDD처럼 실패 조건(validation.md)이 먼저 확정된다.
기준이 구현과 무관하게 독립적으로 설정된다.
이후 Generator가 그 기준을 만족시키기 위해 작업한다.

### 컨텍스트 관리

**리셋**: 새 세션으로 시작. Sprint 경계에서 권장된다.
**컴팩션**: 요약·압축하며 계속 진행. Sprint 내부 기본.
컨텍스트가 70% 채워지면 진행 상황을 기록하고 새 세션으로 이동한다.
Evaluator 채점 전에는 반드시 독립 컨텍스트가 필요하다.

## Evaluator의 역할

### 두 시점

Sprint 전: validation.md 작성(기준 설정).
Sprint 후: 3계층 판정.

첫 번째 시점이 더 중요하다.
Evaluator는 “Sprint 후 판정하는 심판”이 아니라 “Sprint 전 기준을 정의하는 설정자”다.

### 3계층 검증 구조

**Layer 1 — 운영 검증**: 빌드, CI, 배포, 헬스체크, 메트릭. 기계적 판정 가능. DevOps/SRE 영역을 흡수한다.

**Layer 2 — 기능 검증**: Unit, Integration, E2E, Contract 테스트. TDD 피라미드 기반. Given-When-Then 포맷 사용.

**Layer 3 — 주관적 채점**: 디자인, UX, 코드 품질, 창의성. Rubric 기반. 영속적 HITL. 사람의 최종 감수가 필수다.

세 Layer 모두 Given-When-Then 포맷으로 일관성을 유지한다.

### Evaluator의 입력 제약

requirements.md만 읽고, plan.md는 읽지 않는다.
구현 방식을 모르는 상태에서 기준을 쓸 때 더 엄격해진다.
plan.md를 읽으면 Generator의 구현 방식에 맞춰 기준을 완화하는 “낚시”가 발생한다.

### 회의적 튜닝

기본 모드는 “실패할 수 있는 방식 탐색”이다.
Positive/Negative 쌍을 설정하고, 놓친 사례를 축적한다.
AI 슬롭(slop) 패턴을 Rubric 감점 항목으로 등록한다. 과도한 주석, 구현되지 않은 TODO, 사용하지 않는 import, catch-and-ignore 예외, 빈 테스트, 행복 경로만 테스트하는 코드가 해당된다.

### 하네스 성숙도 = Evaluator 성숙도

Evaluator 성숙도가 전체 하네스 성숙도를 결정한다.
축적되는 자산: 놓친 실패 사례, Rubric 개정 이력, Edge case 라이브러리, 반례 카탈로그.
Evaluator 튜닝은 일회성이 아닌 지속적 활동이다.

## 매개체 시스템

### 단일 오너십 원칙

모든 매개체 파일에는 단일 오너가 있다.
오너십은 쓰기 권한의 단일화이지, 읽기 접근성의 제한이 아니다.
모두가 읽을 수 있지만 쓰는 사람은 하나다.
다수 오너가 같은 파일에 쓰면 드리프트(drift)가 시작된다. 어느 문장이 어느 역할의 의도인지, 어느 합의가 여전히 유효한지를 판정할 수 없게 된다.

**오너십 테이블**:

| 매개체            | 오너      | 참조자         | HITL 성격         |
| ----------------- | --------- | -------------- | ----------------- |
| mission.md        | Planner   | 세 역할 모두   | 영속적            |
| tech-stack.md     | Planner   | 세 역할 모두   | 부분 영속적       |
| roadmap.md        | Planner   | 세 역할 모두   | 과도기적          |
| plan.md           | Planner   | Generator 전용 | 과도기적          |
| requirements.md   | Planner   | G + E          | 과도기적          |
| operations.md 상단 | Planner   | G + E          | 과도기적          |
| operations.md 하단 | Evaluator | G + Planner    | 과도기적          |
| validation.md     | Evaluator | Generator      | Layer 3 영속적    |
| Sprint Contract   | G + E 협상 | Planner 통보   | 부분 영속적       |
| ADR               | Planner   | 세 역할 모두   | 영속적            |
| API 계약          | Planner   | G + E(테스트)  | 부분 영속적       |

### Constitution: 프로젝트의 헌법

Constitution은 프로젝트 동안 드물게 바뀐다.
자주 바뀌면 모든 하위 결정이 흔들린다.
Constitution 개정은 보통 Sprint 경계에서, 명시적 합의 뒤에 이루어진다.

### operations.md의 이중 오너십

operations.md는 유일하게 두 오너를 가진 파일이다.
상단(`배포 ·` 접두사 섹션)은 Planner, 하단(`관측 ·` 접두사 섹션)은 Evaluator가 소유한다.
헤더 컨벤션과 HTML 주석으로 경계를 명시한다.

이중 오너십이 원칙의 예외처럼 보이지만, 실제로는 정교화다.
논리적 단위 수준에서는 각 섹션이 단일 오너를 가진다.
배포 환경 변경 시 Evaluator에게 통보하고, Evaluator가 헬스체크와 SLO를 재검토하는 프로토콜이 필수다.

### Sprint Contract: Generator와 Evaluator의 협상 문서

Sprint Contract는 기준을 완화하는 문서가 아니라, 기준 조정을 공식화하고 기록하는 문서다.

5섹션 구조:
1. Sprint 범위
2. 수락된 validation 기준
3. 협상으로 조정된 기준 (근거 필수)
4. 기한과 재개방 조건
5. Planner 통보 채널

Generator는 이의 제기만 하고 validation.md를 직접 수정하지 않는다.
이의 제기에는 기술적 근거가 필수다.
반박된 이의도 Contract에 남는다.
Planner는 협상에 참여하지 않고 결과만 통보받는다.
Sprint 시작과 함께 Contract가 잠긴다. 중간 변경은 공식 재개방 절차를 거친다.

### ADR: 결정의 불변 기록

Architecture Decision Record는 되돌리기 어려운 결정, 외부 합의를 포함한 결정, 관례와 다른 선택에 대해 작성한다.
한번 작성되면 수정되지 않는다. 결정을 뒤집을 때는 새 ADR을 작성한다.
tech-stack.md와 ADR은 양방향으로 링크된다. Constitution은 현재 상태를, ADR은 결정의 역사를 담는다.

### API 계약: 경계면의 특수 매개체

외부 API 계약(OpenAPI, GraphQL, 이벤트 스키마)은 Planner가 오너이지만, Generator의 구현 입력이자 Evaluator의 Contract 테스트 입력이라는 이중 역할을 한다.
내부 함수 시그니처는 Generator 자유 영역이지만, 조직 경계를 넘는 인터페이스는 Planner 수준에서 고정된다.

### 매개체의 두께

모든 프로젝트에서 11개 매개체가 모두 필요하다.
생략되는 것이 아니라 두께가 달라진다.
작은 버그 수정에도 얇은 Feature Spec과 얇은 Sprint Contract가 존재한다.
“없다”와 “얇다”는 다르다. 없으면 드리프트가 시작된다.

## 위임 수준 L1–L4

### 결정하는 두 축

위임 수준은 팀 규모나 일정이 아니라 두 독립적 축이 결정한다.

**하네스 성숙도 (시간 축)**: Evaluator의 작동 품질, Rubric과 반례 카탈로그의 축적 정도. 의도적 투자로 움직인다.

**영속적 HITL 크기 (영역 축)**: 구조적으로 사람 판단이 필요한 결정의 규모. 도메인 특성으로 고정된다. 모델이 좋아져도 줄지 않는다.

100명 팀이라도 Evaluator 분리 없이 운영되면 L1에 갇힌다.
솔로 개발자도 성숙한 하네스로 L3에 도달할 수 있다.

### 네 가지 위임 레벨

| 레벨 | 개입 경계        | 특징                              |
| ---- | ---------------- | --------------------------------- |
| L1   | 각 산출물        | 매 결과마다 사람이 검토하고 개입  |
| L2   | 피처 단위        | 피처 시작·완료 시만 개입          |
| L3   | 스프린트 단위    | 여러 피처 자율 수행, Sprint 완료 시 감수 |
| L4   | 로드맵 단위      | 현재 시점 실험적, 대부분 부적합   |

### 하네스 성숙도 측정: 5가지 신호

절대값이 아닌 추세로 읽는다.

1. 놓친 실패 빈도 — Sprint를 거듭하며 감소하는가
2. Rubric 개정 빈도 — 미세 조정으로 안정화됐는가
3. Layer 1·2 자동화 비율 — 사람 개입이 0에 가까워졌는가
4. Sprint Contract 논점 안정성 — 같은 논점이 재협상되지 않는가
5. 실패 반영 지연 — 놓친 사례가 빠르게 Rubric에 반영되는가

### 피처별 레벨 다변화

같은 프로젝트에서도 피처마다 다른 레벨을 적용한다.
로그인 OAuth는 L3 가능, 결제 처리는 L1 필수, 관리자 대시보드는 L2.
“프로젝트 전체 한 레벨”은 위험한 피처를 과소 관리하거나 자동화 기회를 놓친다.

### 결합 매트릭스

| 구분          | 하네스 미성숙 | 하네스 성숙  |
| ------------- | ------------- | ------------ |
| 영속적 HITL 큼 | L1 고정       | L2 천장      |
| 영속적 HITL 작음 | L1~L2 전환기  | L3 가능      |

### 레벨 이동 규율

올리기는 보수적으로: 3~5 Sprint 연속 안정 신호, 한 번에 한 단계씩.
내리기는 빠르게: 하나의 명확한 신호로 즉시 실행.
잘못 올렸을 때 비용(배포 결함, 규제 위반)이 잘못 내렸을 때 비용(임시 생산성 저하)보다 훨씬 크다.

### 모델 개선의 비대칭 효과

모델 개선은 하네스 성숙도 축을 오른쪽으로 이동시킨다(과도기적 HITL 감소).
영속적 HITL 크기는 고정이다.
결제·의료 도메인은 모델이 좋아져도 L2 천장에 머문다.
모델 개선을 기다리기보다 Rubric·반례 카탈로그 축적 투자가 더 중요하다.

## 프로젝트별 적응

### 불변 원칙과 가변 요소

**불변**: 세 역할의 구조적 분리, 매개체의 단일 오너십, 과도기적/영속적 HITL 구분, Evaluator의 plan.md 접근 제한. 어떤 프로젝트에서도 타협 불가.

**가변**: 매개체의 두께, 역할의 깊이와 HITL 빈도, 운영 주기와 Rubric 재검토 빈도.

### 네 가지 프로젝트 유형

**A형 — 하네스 구축 초기**: L1 고정. Evaluator 기초 체력 구축에 투자한다. 첫 Rubric은 3~5개 핵심 항목으로 시작. Layer 1·2 자동화를 먼저 안정화한다.

**B형 — 하네스 성숙기**: 정기적 감량 실험(분기당 최소 1회). Rubric 버전 관리(Sprint는 미세조정, 분기는 근본 검토). 성숙 정체, Rubric 관성, 감사 방치가 주요 위험이다.

**C형 — 영속적 HITL이 큰 도메인(금융·의료·규제)**: L2가 운영의 천장. BDD를 validation.md 주 포맷으로 사용. ADR 불변성 엄격 적용. Sprint Contract 협상 이력 보존.

**D형 — 영속적 HITL이 적은 도메인(내부 도구·자동화)**: L3 진입이 현실적으로 가능. Edge case 라이브러리에 집중 투자.

### 팀 규모의 간접 영향

소규모 팀(1~5명): HITL 개입 속도가 빠르지만 자기 감사 사각지대가 있다.
중규모 팀(5~20명): HITL 전문성 자연 분화가 강점. Rubric 관리 담당을 지정한다.
대규모 팀(20명+): 유지 작업 전담화 가능. 서브팀별 하네스 분기 위험에 메타 하네스로 대응한다.

## 실패 모드와 회복

### 6가지 실패 카테고리

**경계 침범 실패**: Planner가 validation.md를 작성하거나, Evaluator가 plan.md를 열람하거나, Generator가 기준을 완화하는 경우. “한 경계가 뚫리면 다른 경계도 함께 약해진다.”

**레벨 불일치 실패**: 실제 가능 레벨보다 높은 수준으로 운영하는 오류. 조기 상향, 영속적 HITL 무시, 모델 낙관론이 원인이다.

**Planner 실패**: 경계 침범(validation 작성, plan 세부 지시), 깊이 부족(요구사항 모호성).

**Generator 실패**: 조기 완료(컨텍스트 불안), 기준 회피(테스트 우회), 배포 누락(운영 단계 미완료).

**Evaluator 실패(가장 중요)**: 관대한 채점, Rubric 축 관성, 도메인 전문성 부족, 놓친 실패 반영 지연, plan.md 오염(가장 은밀한 실패). 회복 순서: 오염 차단 → 반영 지연 해소 → 관대 채점 교정 → 축 관성 제거 → 도메인 전문성 보강.

**매개체 실패**: 드리프트(여러 역할의 혼합 수정), operations.md 이중 오너십 경계 붕괴, ADR 변조, Sprint Contract 이력 누락.

**HITL 운영 실패**: Rubber-stamping(형식적 승인), HITL 성격 혼동(영속적을 과도기적으로 취급, 또는 그 반대).

**조직 실패**: 문서 과잉(참조 실효성 저하), 비용 폭발(세션 경계 관리 실패), 신뢰 붕괴(악순환 구조).

### 9가지 건강성 지표 대시보드

놓친 실패 빈도, Rubric 개정 빈도, 자동 판정 비율, Sprint Contract 논점 새로움, 반영 지연 시간, 경계 침범 발생 수, Rubber-stamping 비율, 감량 실험 빈도, 매개체 참조 빈도.

## Claude Code 구현

### 4가지 프리미티브

**Subagent**: 격리된 컨텍스트 윈도우 + 자체 시스템 프롬프트 + 자체 도구 권한. 세 역할의 물리적 분리를 보장한다. `.claude/agents/<name>.md`에 정의한다.

**Skill**: 온디맨드로 로드되는 모듈화된 지시문 번들. `.claude/skills/<skill-name>/SKILL.md`와 보조 파일로 구성된다. A군(사용자 호출)과 B군(역할 규율, 자동 발동)으로 나뉜다.

**Hook**: 도구 호출 라이프사이클의 결정론적 처리. 셸 스크립트가 모델 해석 없이 실행된다. exit code 0은 통과, 1은 정보성 실패, 2는 차단이다. Skill이 모델이 “읽는” 지시문이라면, Hook은 모델 해석 없이 “실행되는” 결정론적 코드다.

**Plugin**: Subagent + Skill + Hook + MCP 서버를 하나의 배포 단위로 묶는다. `/plugin install <name>`으로 설치한다.

### 이중 집행 원칙

Level 1 — Subagent 권한: `tools` 화이트리스트로 도구 자체를 차단(Planner에게는 Bash 없음).

Level 2 — PreToolUse 훅: 경로 수준 검사로 도구 호출 전 차단(Planner의 validation.md 쓰기를 훅이 차단).

한 레벨이 무너져도 다른 레벨이 막는다. 이것이 이중 집행이다.

### 2군 Skill 구조

**A군(workflow 진입점)**: `disable-model-invocation: true`. 사용자가 슬래시 커맨드로 명시적 호출.
- `/init-project`: 새 프로젝트 하네스 초기 세팅
- `/interview-constitution`: Constitution 인터뷰 (4라운드: 발산→수렴→tech-stack→roadmap)
- `/start-feature`: 새 피처 진입
- `/negotiate-contract`: Sprint Contract 양자 협상 자동화
- `/run-sprint`: Generator Sprint 실행
- `/run-evaluation`: 3계층 판정
- `/diagnose-harness`: 자가 진단 (7개 신호 측정)

**B군(역할 규율)**: `user-invocable: false`. 서브에이전트 프리로드 또는 `paths` 글로브로 자동 발동. 메뉴에 표시되지 않는다.
- `planner-discipline`: 편향 방지, 매개체 분리 강제
- `generator-completion-chain`: 6단계 완료 체인 강제
- `evaluator-skepticism`: 회의 모드 자동 강제, AI 슬롭 감점

### Worktree 전략

영속적 HITL 지점과 Worktree 경계를 맞춘다.

| 변경 단위        | Worktree 이름          | 병합 트리거                  |
| ---------------- | ---------------------- | ---------------------------- |
| Constitution 변경 | `constitution-{date}` | 사용자 PR 검토 승인          |
| Feature Spec 작성 | `feat-{id}-spec`      | Planner 완료 + 사용자 검토   |
| Sprint 실행      | `feat-{id}-sprint-{n}` | 3계층 통과 + Layer 3 승인    |
| 회복 시도        | `recover-{type}-{date}` | 회복 검증 + 사용자 확인      |

Sprint worktree는 코드와 sprint-contract.md만 수정한다. plan/requirements/validation은 읽기 전용이다.

### 이식성 평가

**도구 무관 자산**: 역할 정의, Skill 시나리오 프롬프트, 매개체 디렉토리 규약, Worktree 정책, references 패턴.

**Claude Code 의존 메커니즘**: Skill 자동 발동(description matching + paths), Subagent 프리로드, Skill 보조 파일, Worktree 자동 통합.

Codex CLI로 이식 시 B군 자동 발동 기능 상실이 핵심 손실이다.
Gemini CLI로 이식 시 Subagent 격리(서비스 중 가장 심각한 손실)와 Skill 자동 발동 모두 상실된다.

## 기존 방법론과의 관계

### 매핑 관계

GAN의 Generator–Discriminator 비대칭이 이 책의 원리적 토대로 격상된다.
TDD의 Red-Green-Refactor가 역할 레벨로 재배치된다. Red는 Evaluator의 Sprint 전 기준 작성이 되고, Green은 Generator의 6단계 완료 체인이 된다.
BDD의 Given-When-Then이 requirements.md와 세 Layer를 잇는 공용 포맷이 된다.
DevOps/SRE의 CI/CD, SLO, 관측성이 Evaluator Layer 1으로 흡수된다.
SDLC의 BRD/PRD/SRS가 Planner 입력으로 흡수된다.

### 9가지 고유 기여

1. validation.md의 Layer 1/2/3 분할
2. operations.md 이중 오너십 구조
3. 영속적 vs 과도기적 HITL 구분 프레임
4. L1–L4 위임 수준 + 비대칭 원칙
5. 프로젝트 유형 A/B/C/D 적응 프레임워크
6. 실패 모드 6카테고리 분류학
7. Sprint Contract 5섹션 구조와 이력 보존 규약
8. 이중 집행(프롬프트 레벨 + 환경 레벨 분리)
9. GAN 이항 → Planner/Generator/Evaluator 삼항 분화 논증

## 분석

### 역할 분리의 구조적 엄밀성

이 책의 핵심 강점은 역할 분리를 원칙 선언에 그치지 않고 메커니즘으로 구현했다는 점이다.
“Evaluator가 plan.md를 읽지 않는다”는 당위가 PreToolUse 훅의 `block-plan-read.sh` 8줄로 구현된다.
이론과 코드의 거리가 8줄이다.

대부분의 AI 활용 방법론은 “좋은 프롬프트를 써라”나 “역할을 분리하라”는 원칙을 제시하고 멈춘다.
이 책은 원칙을 환경 수준에서 집행하는 단계까지 내려간다.
모델이 지시를 무시하거나 오해하더라도 훅이 차단한다는 결정론적 보장이 이 책을 플레이북으로 만든다.

### PGE 분리의 실용적 의미

GAN의 이항 구조를 소프트웨어 개발의 삼항으로 확장한 논증은 설득력 있다.
Discriminator를 하나로 두면 “기준을 누가 설정하는가”와 “기준을 누가 적용하는가”가 같은 주체에게 돌아간다.
Planner가 validation을 쓰면 낙관 편향이 주입되고, Evaluator가 plan을 읽으면 독립성이 사라진다는 구조적 분석이 명확하다.

특히 “Evaluator의 무지 원칙”이 인상적이다.
Evaluator가 구현 방식을 모를 때 더 엄격한 기준을 세운다는 관찰은 실증적이다.
정보를 덜 가질 때 더 엄격해진다는 것은 역설적으로 보이지만, 구현 가설이 없으면 요구사항 자체에서 기준을 끌어낼 수밖에 없기 때문이다.
[Anthropic의 장기 실행 하네스 글](./harness-design-long-running-apps.md)이 생성자-평가자 분리를 “분업이 아니라 불신의 제도화”로 표현한 것도 같은 통찰이며, 이 책은 그 불신을 plan.md 차단 훅이라는 8줄의 결정론적 메커니즘으로 못 박아 한 단계 더 밀어붙인다.

### 영속적 HITL과 과도기적 HITL의 구분

이 구분이 이 책의 가장 중요한 실용적 기여 중 하나다.
많은 팀이 “AI가 좋아지면 이것도 자동화되겠지”라고 생각하는 항목들이 실제로는 영속적 HITL 영역에 속한다.
가치·전략 판단, 법적·윤리적 책임, 되돌릴 수 없는 위험은 모델이 아무리 좋아져도 인간 판단이 필요하다.
반대로 “이건 사람이 해야 해”라고 여기는 항목들이 실제로는 과도기적 HITL인 경우도 많다.

이 구분이 없으면 자동화 가능한 것을 수동으로 유지하거나, 자동화하면 안 되는 것을 자동화하는 두 방향의 실수 모두 발생한다.

### 매개체 시스템의 완결성

11개 매개체와 단일 오너십 원칙은 “누가 무엇을 쓰고 누가 무엇을 읽는가”를 완전히 명세한다.
이것은 조직 내 정보 흐름의 설계도다.
파일이 소통의 매개체라는 접근이 에이전트 협업에서 특히 유효하다. 에이전트는 인간과 달리 구두 합의나 암묵적 이해를 신뢰할 수 없기 때문이다.
[골빈해커의 18단계 프롬프팅 플로우](golbin-agent-prompting-flow.md)가 단일 에이전트에게
검토 관점을 단계별로 주입해 같은 효과를 노린 것과 비교하면, 매개체 시스템은 그
검토 관점을 파일과 권한 경계로 영구화한 형태다.

“Sprint Contract는 기준 완화 문서가 아니라 기준 조정을 공식화하는 문서”라는 정의가 특히 중요하다.
실제 팀에서 Sprint 중 기준이 은밀히 낮아지는 것은 흔한 문제다. 이 문서가 그 과정을 공개화하고 기록에 남긴다.

## 비평

### 강점: 원칙을 집행 메커니즘으로 구현

대부분의 방법론 서적이 원칙을 선언하고 구현을 독자에게 남기지만, 이 책은 Claude Code 훅 스크립트까지 제시한다.
“Evaluator는 plan.md를 읽지 않는다”가 단순 권고가 아니라 훅으로 강제된다는 것이 이 책을 플레이북으로 만드는 핵심이다.
이론과 코드 사이의 거리를 좁히는 것은 방법론 서적에서 보기 드문 접근이다.

### 강점: 비용과 한계의 명시

“하네스는 공짜가 아니다. 솔로 에이전트보다 20배 이상 비싸다”는 솔직한 선언이 신뢰를 높인다.
이식성 평가에서도 Codex CLI, Gemini CLI로 이식 시 잃는 것을 솔직하게 명시한다.
“no direct support”와 “equivalent”를 혼동하지 않는다.

### 약점: Claude Code 의존성의 깊이

이 책은 도구 무관 원칙을 강조하지만, 실제 집행 메커니즘은 Claude Code에 깊이 의존한다.
Gemini CLI 이식 시 Subagent 격리가 불가능해 GAN 비대칭의 핵심이 무너진다는 것은 작은 문제가 아니다.
Claude Code를 쓰지 않는 팀이 이 플레이북을 도입하려면 상당한 재설계가 필요하다.
“도구 무관”과 “Claude Code 플레이북” 사이의 긴장이 완전히 해소되지 않는다.
[Claude Code vs Codex 비교](claude-code-vs-codex.md)가 도구별 가이드라인 준수도가
극단적으로 다르다고 관찰한 것을 고려하면, 같은 매개체 시스템이라도 어느 에이전트가
실행하느냐에 따라 운영 비용이 달라진다는 점이 더 솔직히 다뤄질 필요가 있다.

### 약점: 비용 정당화의 근거 부족

“솔로 에이전트보다 20배 비싸다”고 하면서도, 언제 이 비용이 정당화되는지의 기준이 명확하지 않다.
개인 프로젝트, 소규모 스타트업, 엔터프라이즈에서 각각 하네스 도입이 정당화되는 임계점이 무엇인가?
비용-편익 분석 없이 “복잡한 프로젝트엔 필요하다”는 수준의 서술에 그친다.
[Learn Harness Engineering](./learn-harness-engineering.md)이 “일회성 스크립트에 `feature_list.json`과 `claude-progress.md`를 만드는 것은 오버엔지니어링”이라 명시한 것과 비교하면, 이 책의 적용 범위 판단 기준은 상대적으로 모호하다.

### 약점: Evaluator의 Layer 3 실용성

Layer 3 주관적 채점은 “사람의 최종 감수가 필수”라고 명시되지만, 이것이 병목이 되는 상황에 대한 대응이 약하다.
매 Sprint마다 사람이 UX, 디자인, 코드 품질을 감수한다면, 빠른 반복을 원하는 팀에게는 현실적이지 않다.
Layer 3의 감수 주기를 어떻게 조율할 것인지에 대한 구체적 지침이 필요하다.

### 보완: 팀 간 협업 시나리오

이 책은 에이전트가 세 역할을 수행하는 구조를 다룬다.
하지만 실제 팀에서는 여러 엔지니어가 세 역할에 부분적으로 관여한다.
“Planner 역할의 에이전트를 감독하는 PM”과 “Generator 역할의 에이전트를 감독하는 엔지니어”가 다른 사람일 때의 협업 프로토콜이 다뤄지지 않는다.

## 인사이트

### 검증의 선행이 개발의 본질을 바꾼다

이 책에서 가장 급진적인 아이디어는 Evaluator가 Generator보다 먼저 작업을 시작한다는 것이다.
전통적 소프트웨어 개발에서 테스트는 구현 후에 온다. “코드를 먼저 짜고, 그 다음 테스트를 작성한다.”
TDD가 이 순서를 뒤집었지만, 실제로 TDD를 실천하는 팀은 드물다. 구현의 유혹이 강하기 때문이다.

하네스 엔지니어링은 이 문제를 구조적으로 해결한다.
Evaluator가 별도의 역할로 분리되어 있고, validation.md가 Sprint 전에 잠기기 때문에 Generator는 미리 정해진 기준에 맞춰 작업할 수밖에 없다.
기준이 먼저 존재한다는 것은 단순한 순서 변경이 아니라 개발의 철학을 바꾼다.
“어떻게 만들지”를 생각하기 전에 “어떻게 하면 완성됐다고 말할 수 있는지”를 먼저 정의하는 것이다.

이 전환의 실용적 효과가 크다.
완료 기준이 모호한 상태에서 구현을 시작하면 에이전트(또는 인간)는 자신이 편한 방향으로 완성도를 정의한다.
기준이 먼저 정해져 있으면 “대략 이 정도면 됐겠지”라는 컨텍스트 불안의 여지가 사라진다.
“AI 배포 완료”가 “코드 작성 완료”로 슬그머니 바뀌는 현상의 근본 원인이 모호한 완료 정의이기 때문이다.

### 구조는 신뢰를 대체한다

에이전트에게 “좋은 코드를 작성해라”고 요청하는 것과, 에이전트가 validation.md의 기준을 통과하지 못하면 훅이 차단하도록 설계하는 것은 근본적으로 다른 접근이다.
전자는 에이전트에 대한 신뢰에 의존한다. 후자는 구조가 신뢰를 대체한다.

이것은 소프트웨어 공학이 오랫동안 내면화해온 원리를 에이전트 시대에 재적용한 것이다.
코드 리뷰, CI/CD, 타입 시스템, 린터는 모두 “개발자를 믿지 말고, 구조가 확인하게 하라”는 철학의 산물이다.
에이전트가 인간보다 낫거나 못하다는 가정 없이, 인간 코드에도 적용하는 동일한 검증 구조를 에이전트 코드에 적용한다는 것이 이 책의 핵심 통찰이다.

역설적으로 이 접근이 에이전트에 대한 더 높은 신뢰를 가능하게 한다.
구조가 없으면 의심이 필요하다. 모든 산출물을 검토해야 한다.
구조가 있으면 구조가 통과한 것은 어느 정도 신뢰할 수 있다.
L1에서 L3로의 위임 수준 상승은 에이전트가 더 똑똑해졌기 때문이 아니라, 구조가 더 신뢰할 수 있게 됐기 때문이다.

### 영속적 HITL이 인간 역할의 새로운 정의다

“에이전트가 세 역할을 주도하고, 사람은 특정 결정 지점에만 개입한다”는 전제는 단순히 에이전트 활용법을 바꾸는 것이 아니다. 소프트웨어 개발에서 인간의 역할이 무엇인가를 재정의한다.

전통적으로 엔지니어의 역할은 코드를 작성하는 것이었다.
AI 보조 도구 시대에는 코드를 작성하되 AI의 도움을 받는 것이었다.
하네스 엔지니어링의 에이전트 주도 전제에서 인간의 역할은 영속적 HITL 지점들이다.

가치·전략 판단, 법적·윤리적 책임, 외부 이해관계자 합의, 되돌릴 수 없는 위험.
이 목록은 소프트웨어 개발의 기술적 측면이 아니라 사회적·조직적 측면을 가리킨다.
코드를 잘 쓰는 것이 아니라 무엇을 만들지를 결정하고, 그것이 조직과 사회에 어떤 영향을 미칠지를 판단하는 것이 인간 역할의 본질이 된다.

이것은 자동화의 일반적 패턴과 같은 방향이다.
회계사가 스프레드시트를 도입했을 때 숫자를 직접 계산하는 역할을 잃었지만, 재무적 판단의 역할은 커졌다.
의사가 영상 진단 AI를 도입했을 때 이미지 판독 역할을 나눠갖게 됐지만, 환자와의 소통과 치료 결정의 역할은 오히려 중요해졌다.
하네스 엔지니어링이 완성된다면 엔지니어는 코드를 쓰는 사람에서 무엇을 만들지를 결정하고 그 결과에 책임지는 사람으로 이동할 것이다.

### 매개체 중심 협업이 지식 관리의 미래다

이 책의 매개체 시스템에서 특히 주목할 것은 파일이 협업의 기록이자 근거가 된다는 점이다.
mission.md, ADR, Sprint Contract 이력은 단순한 문서가 아니라 의사결정의 고고학적 층위다.

“ADR은 한 번 작성되면 수정되지 않는다. 뒤집을 때는 새 ADR을 작성한다”는 규율은 조직 기억의 연속성을 보장하는 메커니즘이다.
ADR-007이 “왜 그때 MongoDB였는가”를 설명하고, ADR-031이 “왜 지금 Postgres인가”를 설명한다.
두 결정이 시간적 맥락 안에서 모두 정당했음을 보여주는 것이 중요하다.

에이전트 주도 개발에서 이 기록의 중요성은 더 커진다.
에이전트가 과거 결정의 근거를 알지 못하면 이미 실패한 접근을 반복한다.
파일 기반 매개체가 에이전트의 조직 기억 접근 채널이 된다.
결국 잘 관리된 매개체 시스템은 새로운 에이전트 세션이 조직의 맥락을 빠르게 흡수할 수 있게 하는 온보딩 시스템이기도 하다.
[하네스 엔지니어링 백과사전](./harness-engineering-encyclopedia.md)이 “메모리 소유권은 제품 전략”이라 표현한 것이 이 관점의 확장이다 — 매개체에 누적되는 결정 이력, 사용자 선호, 실패 사례가 결국 모델보다 강력한 락인 요인이 된다는 점에서, 매개체 시스템 설계는 단순한 협업 도구가 아니라 장기 경쟁 우위의 형식화다.
