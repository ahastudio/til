# AI로 더 좋은 코드를, 더 느리게 작성하기

원문: <https://nolanlawson.com/2026/05/25/using-ai-to-write-better-code-more-slowly/>

HN 토론: <https://news.ycombinator.com/item?id=48272984> (1158점, 426개 댓글)

## 요약

Nolan Lawson은 AI 코딩 도구의 지배적 서사에 도전한다.
AI를 쓰면 코드를 빠르게 생성할 수 있다는 것이 그 서사의 핵심이지만,
그는 LLM을 의도적이고 방법론적인 리뷰 과정에 투입해
속도를 줄이는 대신 코드 품질을 높이는 역방향 전략을 제안한다.

그의 핵심 방법론은 멀티 모델 코드 리뷰다.
풀 리퀘스트에 Claude, Codex, Cursor Bugbot을 동시에 실행하고,
발견된 문제를 심각도(critical/high/medium/low)로 분류한다.
여러 모델의 결과를 교차 검증해 거짓 양성(false positive)을 걸러낸다.
속도가 아닌 포괄적 버그 탐지에 집중한다.

실제 워크플로는 네 단계로 구성된다.
에이전트가 critical과 high 이슈를 반복적으로 수정한다.
medium과 low 이슈는 투입 대비 가치가 낮으면 건너뛴다.
접근 방식이 근본적으로 잘못됐다고 드러나면 PR을 포기한다.
기존 코드베이스의 잠재 버그를 학습 기회로 삼는다.

이 접근법의 실제 효과는 배송 속도 향상이 아니라 잠재적 취약점 발견이었다.
한 댓글 작성자(Ollie)는 이 과정에서 Postgres 정책의 로우 수준 보안(row-level security) 허점을
배포 전에 발견했다고 공유했다.
이런 “사이드 퀘스트”가 개발자로 하여금 시스템의 아키텍처 약점을 학습하게 한다.

속도 이외의 핵심 이점으로는 새 개발자의 코드베이스 이해 가속,
해피 패스가 아닌 실패 모드에 대한 깊은 이해, 지속 가능한 품질 개선,
꼼꼼한 장인 정신을 통한 개발자 만족도 향상이 있다.
Lawson은 동시에 한계도 인정한다. AI 발견 사항을 분류하려면 도메인 전문성이 필수다.
주니어 개발자는 멘토링 없이 이론적 취약점과 실제 위협을 구분하기 어렵다.

## 분석

### 속도 패러다임의 전복

AI 코딩 도구의 마케팅 언어는 거의 예외 없이 속도를 강조한다.
“10배 빠른 개발”, “지루한 작업 자동화”, “더 적은 시간에 더 많이 출시”.
Lawson의 접근법은 이 언어를 의도적으로 뒤집는다. “더 느리게”가 핵심이다.

이 역전이 의미 있는 이유는 속도와 품질의 트레이드오프가 소프트웨어 개발의
오래된 긴장이기 때문이다. AI가 속도를 높이는 방향에서 사용된다면,
이 트레이드오프는 더 빠른 배송을 위해 품질을 희생하는 방향으로 기울 수 있다.
AI를 리뷰 도구로 사용한다면, 같은 기술이 반대 방향 — 속도를 희생해 품질을 높이는 —
으로 트레이드오프를 이동시킨다.
이것은 도구의 사용 방식이 도구 자체보다 더 중요할 수 있다는 점을 보여준다.

### 멀티 모델 교차 검증의 합리성

하나의 LLM에 코드 리뷰를 맡기는 것과 여러 모델을 동시에 실행하는 것은
신뢰도 측면에서 근본적으로 다르다. 단일 모델은 일관된 편향을 가질 수 있다.
특정 패턴의 버그를 잘 잡지 못하거나, 반대로 정상 코드를 버그로 잘못 분류할 수 있다.
여러 모델이 같은 문제를 독립적으로 발견하면 그 문제는 실제일 가능성이 높고,
한 모델만 발견하면 거짓 양성일 가능성을 검토해야 한다.

이것은 통계적 앙상블(ensemble) 방법의 논리와 같다.
모델 예측 불확실성을 줄이기 위해 여러 모델의 결과를 결합하는 것이다.
코드 리뷰에 적용되면 “이 취약점이 실재하는가”에 대한 확신도를 높일 수 있다.
단, 이 접근법의 비용은 시간과 API 비용이다.
여러 모델을 동시에 실행하는 것은 단일 모델보다 몇 배 비싸다.

bottlepalm은 이 접근법을 실제로 수행하는 워크플로를 상세히 공유한다.[^bottlepalm]
설계 단계에서 AI와 먼저 구현 방향을 논의하고, Claude 4.7 Max로 구현한 뒤,
Codex GPT-5.5로 코너 케이스를 찾고, Claude로 수정하는 루프를 반복한다.
그가 발견한 흥미로운 분업은 모델별 강점 차이다. Codex는 버그 탐지와 리뷰에
강하고 성가실 정도로 꼼꼼하지만, Claude는 직관적이고 유지보수하기 좋은 코드를
작성한다. 결과물은 “v1이지만 이미 여러 차례 반복을 거쳐 v3처럼 느껴진다”는 것이다.

### 심각도 분류와 선택적 수정의 실용성

모든 AI 발견 사항을 수정하려는 시도는 종종 비생산적이다.
이것은 소나큐브(SonarQube) 같은 정적 분석 도구의 오래된 교훈이다.
너무 많은 경고를 생성하면 개발자는 경고 피로(alert fatigue)를 겪고
실제 중요한 이슈를 무시하게 된다.
Lawson의 critical/high/medium/low 분류와 medium 이하 건너뛰기는
이 피로를 방지하는 실용적 결정이다.

PR을 포기하는 것도 중요한 결정이다. AI 리뷰가 접근 방식의 근본적 결함을 드러내면,
그 코드를 수정하는 것보다 처음부터 다시 설계하는 것이 낫다.
이 결정을 내리려면 AI의 발견 사항이 기술적으로 타당한지 판단할 도메인 전문성이 필요하다.
이것이 Lawson이 인정하는 한계 — 주니어 개발자가 독립적으로 사용하기 어려운 이유 — 다.

### Postgres row-level security 사례의 중요성

댓글 작성자 Ollie가 공유한 사례는 이 접근법의 실제 가치를 보여주는 구체적 증거다.
Postgres의 로우 수준 보안(RLS) 정책은 잘못 구성하면 인증된 사용자가
다른 사용자의 데이터에 접근할 수 있는 심각한 취약점이 된다.
이런 버그는 해피 패스 테스트로는 발견되기 어렵다. 정상적인 기능 테스트에서
보안 정책은 우회되지 않기 때문이다. AI 코드 리뷰가 이런 종류의 잠재 취약점을
배포 전에 발견할 수 있다는 것이 핵심 가치다.

## 비평

### 강점: 반직관적 가치 제안의 명확한 제시

“AI로 더 느리게”라는 역설적 제안은 기억에 남고 실용적이다.
속도 중심의 AI 도구 담론에서 차별화되며, 실제 경험에서 도출된 방법론이라는 점이
설득력을 높인다. Ollie의 Postgres RLS 사례 같은 구체적 증거가 주장을 뒷받침한다.
코드 품질을 진지하게 고민하는 시니어 개발자와 팀 리더에게 직접적으로 유용한 내용이다.

### 한계: 비용과 확장성 논의의 부재

세 개 모델을 동시에 실행하는 비용이 언급되지 않는다.
Claude, Codex, Cursor Bugbot의 API 비용을 모든 PR에 적용하면
비용이 상당할 수 있다. 소규모 팀이나 개인 프로젝트에서 이 접근법의
경제적 타당성 분석이 없다는 점이 아쉽다.
또한 이것이 모든 PR에 적용 가능한지, 어떤 PR에 우선 적용해야 하는지에 대한
가이드라인도 없다.
TACIXAT은 에이전틱 코딩이 만드는 더 근본적인 문제를 지적한다.[^TACIXAT]
Claude Code 같은 도구가 목표를 향해 너무 빠르게 달려가면서,
개발자가 구현 과정에서 자연스럽게 내리던 마이크로 아키텍처 결정들이 사라진다는 것이다.
“코드를 쓰는 것이 마치 열병 속 꿈처럼 느껴진다”는 표현이 이 감각을 잘 포착한다.
속도가 느려지더라도 개발자가 각 결정에 관여하는 Lawson의 방식은 이 문제에 대한
의식적인 대응으로 읽힐 수 있다.

### 보완이 필요한 관점: 거짓 음성의 위험

멀티 모델 교차 검증은 거짓 양성을 줄이는 데 효과적이지만,
세 모델 모두 놓치는 버그(거짓 음성)에 대한 논의가 없다.
AI 코드 리뷰를 통과한 코드를 “안전하다”고 과신하는 위험이 있다.
특히 도메인 특화 취약점이나 비즈니스 로직 버그는 AI가 발견하기 어렵다.
이 한계를 명시적으로 다루었다면 더 균형 잡힌 권고가 됐을 것이다.

## 인사이트

### 두 가지 AI 활용 방식의 공존

hintymad는 현재 소프트웨어 산업에서 AI 활용이 두 방향으로 갈라지고 있다고 관찰한다.[^hintymad]
한쪽은 Creao 같은 회사로, AI 에이전트가 프로덕션 코드 대부분을 작성하고
“2주 만에 전체 프로덕션 시스템을 재아키텍처했다”는 주장까지 나온다.
다른 쪽은 Lawson이 제안하는 것처럼 AI로 장인 정신을 강화하는 방식이다.
Snowflake의 엔지니어링 팀이 AI 도입 후 처음으로 분기 목표를 조기 달성했다는 사례는
두 접근 방식 모두에서 생산성 향상이 가능함을 보여준다.
그러나 전자는 엔지니어에게 압박을 만들고 후자는 만족도를 높인다는 점에서,
속도 경쟁보다 지속 가능성의 문제가 더 중요할 수 있다.

themanmaran은 이 맥락에서 Lawson의 접근법이 가진 덜 주목받는 이점을 짚는다.[^themanmaran]
AI를 활용하면 빠르게 프로토타입을 만들고, 잘 안 되면 주저 없이 버릴 수 있다.
“수동으로 작성했다면 포기하기 아까웠을” 반쪽짜리 구현을 쉽게 폐기할 수 있게 되면서,
개발자는 더 과감하게 여러 접근 방식을 시도하고 더 나은 최종 설계에 도달한다.
결과적으로 코드량은 줄고 품질은 높아지는 역설적 효과가 나타난다.

### AI 코드 리뷰는 “버그 발견”이 아니라 “가정 도전”이다

코드 리뷰의 전통적 목적은 버그를 찾는 것이다.
그러나 Lawson의 접근법에서 더 중요한 효과는 개발자의 가정에 도전하는 것이다.
PR을 작성하면서 개발자는 수많은 암묵적 가정을 한다.
“이 입력은 항상 유효할 것이다”, “이 쿼리는 항상 한 행을 반환할 것이다”,
“이 정책은 의도한 대로 작동할 것이다”. AI 리뷰는 이 가정들을 명시적으로 검토한다.

이것이 “사이드 퀘스트”를 만드는 이유다. AI가 잠재 취약점을 지적하면,
개발자는 “이게 실제 문제인가”를 확인하기 위해 코드베이스를 탐색하게 된다.
그 탐색 과정에서 관련된 다른 가정들도 검토하게 되고, 때로는 더 근본적인 문제를 발견한다.
이 과정은 버그 수정보다 훨씬 가치 있는 시스템에 대한 이해의 심화다.

Postgres RLS 사례가 그 전형이다. AI가 보안 정책의 잠재적 문제를 지적하자,
개발자는 RLS가 어떻게 동작하는지, 어떤 상황에서 우회될 수 있는지를 깊이 탐구했다.
이 이해는 단순히 그 버그를 수정하는 것을 넘어 향후의 유사한 결정에도 영향을 미친다.

### 코드 리뷰 문화의 위기와 AI의 역할

많은 팀에서 코드 리뷰는 형식이 됐다. PR이 너무 크거나 컨텍스트가 부족하거나,
리뷰어가 너무 바빠 깊은 검토 없이 승인한다.
Google의 연구에 따르면 평균 코드 리뷰 시간은 수 분에 불과하다.
이 상황에서 숙련된 개발자가 발견했을 버그들이 리뷰를 통과해 프로덕션에 도달한다.

Lawson의 접근법은 이 문화적 위기에 대한 기술적 응답이다.
인간 리뷰어의 시간과 주의력이 제한되어 있다면, AI가 그 공백을 채울 수 있다.
인간 리뷰어가 “이 코드가 올바르게 동작하는가”에 집중할 때,
AI는 “이 코드가 모든 엣지 케이스에서 안전한가”를 검토한다.
이 역할 분담이 리뷰의 총체적 품질을 높인다.

그러나 이 접근법의 장기적 위험도 있다. AI 리뷰에 의존하면 인간의 코드 리뷰 역량이
더 빠르게 약화될 수 있다. 인간 리뷰어가 “AI가 잡겠지”라는 가정 하에
더 피상적인 리뷰를 하게 된다면, AI가 놓친 버그는 아무도 잡지 못한다.
AI 리뷰는 인간 리뷰를 대체하는 것이 아니라 보완하는 것이어야 한다.

### “느린 것이 빠른 것이다” — 품질의 복리 효과

소프트웨어 개발에서 속도와 품질의 트레이드오프는 단기적으로만 존재한다.
장기적으로는 빠르게 만들면서 품질을 희생하면 기술 부채가 쌓이고,
그 부채가 미래의 속도를 잡아먹는다. “기술 부채의 이자”가 누적되면
어느 시점에서 새 기능을 추가하는 것이 기존 코드를 유지보수하는 것보다
더 어려워진다.

이 관점에서 Lawson의 접근법 — 지금 느리게, 나중에 빠르게 — 은
기술 부채의 복리 부작용을 줄이는 투자다. 버그를 배포 전에 잡으면
인시던트 대응, 핫픽스 배포, 사후 분석, 고객 신뢰 회복의 비용이 없어진다.
프로덕션 버그의 수정 비용은 개발 중 수정보다 10~100배 높다는 것이
소프트웨어 공학의 오래된 경험칙이다.

AI 코드 리뷰가 PR당 한 시간의 추가 시간을 요구한다면,
그 한 시간이 인시던트 하나를 예방했을 때의 가치는 얼마인가.
Postgres RLS 취약점이 프로덕션에서 발견됐다면 데이터 유출, 법적 책임,
고객 이탈로 이어질 수 있었다. 한 시간의 투자가 그 위험을 제거했다.
이것이 “더 느리게”가 실제로는 더 빠른 이유다.
느린 것과 빠른 것의 비교는 단기 속도가 아닌 장기 속도로 해야 한다.
mlpicker는 이 논의에서 종종 간과되는 심리적 비용을 추가한다.[^mlpicker]
AI가 먼저 제시한 해법을 본 순간, 개발자는 그 출력에 인지적으로 앵커링된다.
“처음부터 새로 작성”이 아닌 “AI 결과를 편집”하는 모드로 자동 전환된다는 것이다.
이것은 개발자의 독립적 설계 능력이 서서히 잠식되는 방식이기도 하다.
Lawson의 접근법 — AI가 먼저 코드를 쓰는 것이 아니라 인간이 쓴 코드를 AI가 리뷰하는 것 — 은
이 앵커링 효과를 구조적으로 회피한다.
