# pi-stuff: Pi 코딩 에이전트용 워크플로 익스텐션

<https://github.com/popododo0720/pi-stuff>

## 소개

`pi-stuff`는 `@mariozechner/pi-coding-agent`를 위한 워크플로 자동화 익스텐션이다.
에이전트의 개발 사이클을 **Plan → Verify → Implement → Verify → Compound → Done** 순서로 강제함으로써
AI가 단계를 건너뛰거나 임의로 코드를 수정하는 것을 구조적으로 막는다.
TypeScript로 작성되었으며, npm 패키지로 설치하거나 소스에서 직접 심링크로 연결할 수 있다.

## 주요 기능

### 워크플로 단계 강제

AI는 비구현 단계(Plan, Verify, Compound)에서 파일 쓰기·편집 도구를 사용할 수 없다.
`WORKFLOW_ACTIVE` 헤더 플래그를 매 프롬프트 턴에 명시적으로 주입해 에이전트가 현재 단계를 신뢰할 수 있게 한다.
각 단계에서 허용되는 도구와 금지되는 도구는 `guard.ts`가 결정한다.

### 병렬 다중 모델 검증

Plan 완료 후와 Implement 완료 후 각각 검증 단계를 거친다.
여러 LLM이 동시에 실행되며, 각 모델은 코드를 무너뜨리려는 적대적 시나리오를 생성한다.
심각도 분류 체계(`🔴 CRITICAL`, `🟡 WARNING`, `🔵 INFO`)에 따라 PASS/FAIL이 결정된다.
Plan 검증에서는 🔴만 실패로, Impl 검증에서는 🔴와 🟡 모두 실패로 처리한다.
VERDICT 키워드가 없으면 항상 FAIL로 간주한다.

### TODO 시스템

대형 작업은 TODO 목록으로 분해된다.
계획은 전체 TODO를 아우르는 단일 문서로 작성되고 한 번만 검증된다.
이후 TODO는 순차적으로 구현되며, 각 TODO마다 Implement → Verify → Compound 사이클이 독립적으로 돌아간다.

### 컨텍스트 압축(Deferred Compaction)

TODO 경계마다 컨텍스트 압축이 발생한다.
압축은 `before_agent_start` 이벤트에서 지연 실행(deferred)되어 도구 실행 경합을 피한다.
`[WF_RESET]` 마커를 이용한 리셋 스타일 압축도 지원한다.

### Repo Map

`web-tree-sitter` 기반 AST 심볼 추출과 임포트 그래프 PageRank를 결합해 파일 중요도를 산출한다.
결과는 토큰 예산(기본 2048토큰) 안에서 시스템 프롬프트에 삽입된다.
TypeScript·JavaScript·Python·Go·Rust 등 18개 언어를 지원하며, 파싱 실패 시 자동으로 생략된다.

### 프로젝트 메모리

`.pi/workflow-memory.json`에 컨벤션·규칙·패턴·결정·고찰 등을 저장한다.
항상 주입되는 핵심 컨벤션과 달리, 과거 해결책·패턴·고찰은 키워드 유사도 Top-k 검색으로 필요할 때만 프롬프트에 삽입된다.

### Git 자동화

TODO 경계와 최종 완료 시점에 자동 커밋·푸시를 수행한다.
워크플로 브랜치와 워크트리를 별도로 관리하며,
최종 푸시 실패 시 워크플로 완료 자체를 차단해 미완료 상태가 원격에 반영되지 않도록 한다.

## 사용법

```bash
# npm으로 설치
pi install npm:workflow-extension

# 소스에서 설치
git clone https://github.com/popododo0720/pi-stuff.git
cd pi-stuff/workflow-extension
npm install
ln -s "$(pwd)" ~/.pi/agent/extensions/workflow-extension
```

주요 명령어:

| 명령어                         | 설명                               |
| ------------------------------ | ---------------------------------- |
| `/workflow <설명>`             | 새 워크플로 시작                   |
| `/workflow-settings`           | 단계별 모델·타임아웃·Git 설정      |
| `/workflow-cancel`             | 활성 워크플로 취소                 |

## 분석

### 에이전트 안정성을 도구 차단으로 해결하다

대부분의 AI 코딩 에이전트 개선 시도는 프롬프트 수준에 머문다.
지시를 더 명확하게 쓰거나, 예시를 추가하거나, 역할을 강화하는 방식이다.
이 프로젝트는 다른 접근을 취한다.
구현 단계가 아닌 상태에서는 파일 쓰기 도구 자체를 막는다.

이 결정의 핵심 전제는 “에이전트는 지시를 따르는 데 신뢰할 수 없다”는 것이다.
프롬프트가 아무리 명확해도 에이전트는 계획 단계에서 코드를 수정하거나,
검증을 건너뛰거나, 승인 전에 구현을 시작할 수 있다.
도구 접근을 환경 수준에서 차단하면 이 가능성 자체가 사라진다.
이는 소프트웨어 보안에서 “최소 권한 원칙”을 에이전트 설계에 적용한 것이다.

### 검증을 독립 프로세스로 분리한 이유

검증 단계에서 동일 에이전트가 자신의 계획을 검토하면 확증 편향(confirmation bias)이 발생한다.
이 프로젝트는 별도 LLM 프로세스를 `Promise.all()`로 병렬 실행하여 검증자와 구현자를 분리한다.
각 검증자는 코드를 깨뜨릴 구체적 시나리오를 찾도록 유도(“adversarial testing”)받는다.

심각도 분류 체계는 구현 검증을 더 엄격하게 처리한다는 점이 주목할 만하다.
Plan 검증에서 🟡 WARNING은 구현자에게 전달되는 메모일 뿐이고,
Impl 검증에서 🟡 WARNING은 실패 사유가 된다.
같은 문제라도 계획 단계에서는 허용하고 구현 단계에서는 막는 비대칭 설계다.

### 컨텍스트 관리의 구조화

긴 세션에서 LLM의 성능 저하는 잘 알려진 문제다.
컨텍스트 창이 가득 차면 초기 지시나 컨벤션이 희석된다.
이 프로젝트는 세 가지 메커니즘으로 이에 대응한다.

첫째, 항상 주입되는 정보를 최소화한다.
핵심 컨벤션만 매 턴에 삽입하고, 나머지는 검색 기반으로 필요할 때만 가져온다.
둘째, TODO 경계마다 컨텍스트를 압축한다.
이전 TODO의 구현 세부사항이 다음 TODO를 오염시키지 않도록 한다.
셋째, `[WF_RESET]` 마커를 활용한 리셋 스타일 압축으로 컨텍스트를 공격적으로 정리할 수 있다.

## 비평

### 도구 차단은 프로젝트 구조에 강하게 결합된다

구현 단계가 아닌 상태에서 파일 쓰기를 막는다는 설계는 단순명쾌하지만,
워크플로가 항상 단일 파일만 수정하는 선형 작업을 가정한다.
현실의 많은 작업은 계획 문서를 수정하면서 동시에 설정 파일을 건드리거나,
검증 과정에서 보조 스크립트를 생성해야 할 수 있다.
도구 차단이 단계별로 전역 적용되기 때문에 이런 예외를 수용하기 어렵다.
결과적으로 사용자는 워크플로를 우회하거나 단계를 억지로 분해해야 하는 상황에 놓일 수 있다.

### 검증 통과는 올바름의 증거가 아니다

병렬 LLM 검증이 “깨뜨릴 시나리오를 찾는” 방식을 취하더라도,
검증자 자신도 같은 훈련 데이터와 같은 추론 패턴을 가진 LLM이다.
특정 유형의 버그, 예를 들어 경쟁 조건(race condition)이나 메모리 모델 의존성 같은 문제는
LLM이 일관되게 놓치는 경향이 있다.
세 모델이 모두 PASS를 반환해도 그 결론은 “세 모델 모두 같은 방식으로 실패했다”는 의미일 수 있다.
README는 이 한계를 명시하지 않으며, 사용자는 자동 검증을 과신할 위험이 있다.

### 심각도 파싱의 취약성

VERDICT와 🔴/🟡/🔵 마커를 텍스트에서 파싱하는 방식은 LLM 출력의 형식 안정성에 의존한다.
README는 부정 처리(`no critical`, `0 warning`, `none`), 중복 제거, 폴백 키워드 스캔까지 구현했다고 밝힌다.
이는 역설적으로 파싱이 충분히 복잡하고 취약하다는 신호다.
LLM이 마커를 다른 맥락에서 사용하거나, 형식을 약간 다르게 출력하거나,
언어를 혼용하면 파서는 오탐(false pass) 또는 미탐(false fail)을 낼 수 있다.
구조화된 출력(structured output) API가 있음에도 텍스트 파싱에 의존하는 이유가 설명되지 않는다.

## 인사이트

### 에이전트 신뢰성은 프롬프트가 아니라 환경 설계의 문제다

이 프로젝트가 보여주는 가장 중요한 통찰은 에이전트를 신뢰하지 않는다는 태도다.
“AI에게 단계를 지키라고 지시하면 지킨다”는 기대를 버리고,
도구 접근 자체를 환경에서 제어하는 쪽을 선택했다.

이 관점은 인프라 보안의 사고방식과 동일하다.
서버에 접근하는 사람에게 “이 디렉터리는 건드리지 마세요”라고 말하는 대신,
파일 시스템 권한으로 접근 자체를 막는다.
에이전트가 얼마나 유능한지와 무관하게, 실행 환경이 제약을 강제해야 한다는 원칙이다.
이 원칙이 일반화된다면, 에이전트 하니스(harness) 설계의 핵심 문제는
“어떤 프롬프트를 쓸까”가 아니라 “어떤 도구를 언제 허용할까”가 된다.

### 검증의 분리는 AI 팀 구성의 원형이다

단일 에이전트가 계획하고 구현하고 검증까지 하는 방식에는 구조적 한계가 있다.
자신이 만든 것을 자신이 검토할 때 확증 편향은 피하기 어렵다.
이 프로젝트가 별도 프로세스로 검증자를 실행하는 것은
“다른 에이전트가 적대적으로 검토한다”는 멀티 에이전트 팀 구성의 단순한 형태다.

이 패턴이 성숙하면 역할 분화가 일어날 것이다.
아키텍처를 검토하는 에이전트, 보안 취약점을 찾는 에이전트, 성능을 측정하는 에이전트가
각각 독립적으로 실행되고, 결과를 통합하는 오케스트레이터가 최종 판단을 내린다.
이 프로젝트는 그 방향의 초기 구현이다.

### 컨텍스트 관리는 에이전트의 장기 기억 문제와 같다

TODO 경계마다 컨텍스트를 압축하는 설계는 인간의 작업 기억 한계와 유사한 문제를 다룬다.
인간도 장기 프로젝트에서 초기 결정을 잊거나, 이전 맥락에 오염되거나, 집중력이 분산된다.
이 프로젝트가 만든 해결책(최소 항상-주입, 필요 시 검색, 경계마다 리셋)은
인간 팀이 스프린트를 나누고, 회고를 하고, 다음 스프린트에서 깨끗하게 시작하는 방식과 구조적으로 같다.

에이전트의 컨텍스트 관리가 곧 에이전트의 조직 설계다.
무엇을 항상 기억하게 하고, 무엇을 검색하게 하고, 무엇을 버리게 할지 결정하는 것이
에이전트 하니스 엔지니어링의 핵심 역량이 될 것이다.
