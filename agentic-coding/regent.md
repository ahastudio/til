# re_gent — AI 에이전트 활동을 위한 버전 관리 시스템

> Version Control for AI agent activity

<https://www.re-gent.dev/>

<https://github.com/regent-vcs/re_gent>

HN 토론: <https://news.ycombinator.com/item?id=48063548> (129점, 67개 댓글)

## 소개

re_gent는 AI 코딩 에이전트의 활동을 추적하기 위해 만들어진 별도의 버전 관리 시스템이다.
Claude Code, OpenAI Codex CLI, OpenCode와 투명하게 통합되어
사용자가 수동으로 커밋하지 않아도 에이전트가 수행한 작업을 자동으로 기록한다.
저장소 구조와 명령어 체계는 Git을 의도적으로 닮았지만,
“커밋”이 아니라 “Step”이라는 단위로 한 번의 에이전트 행위를 캡슐화한다.

핵심 명령은 `rgt log`(세션을 가로지르는 단계 히스토리),
`rgt blame <path>[:<line>]`(특정 줄을 만든 프롬프트 추적),
`rgt show <step>`(도구 호출과 대화를 포함한 전체 컨텍스트 조회),
`rgt sessions`(동시 세션 목록), `rgt status`(저장소 상태)다.
`rgt blame`이 단순한 누가-만들었나가 아니라
“어떤 프롬프트가 이 줄을 만들었나”를 답한다는 점이 Git과 가장 큰 차이다.

저장소는 `.regent/` 하위에 BLAKE3 해시로 콘텐츠 주소화된 객체(`objects/`),
세션별 포인터(`refs/`), SQLite 인덱스(`index.db`), 설정(`config.toml`)을 둔다.
각 Step은 부모 Step 해시, 워크스페이스 트리 스냅샷,
도구 호출의 입출력, 세션 ID, ISO 8601 타임스탬프를 함께 담는다.
Step들은 세션 단위 DAG를 이루어
동시에 진행되는 세션을 충돌 없이 추적하고, 공통 조상을 찾고, 콘텐츠를 자동 중복 제거할 수 있게 한다.

기술 스택은 Go(98.6%) 단일 바이너리에 CLI 프레임워크 cobra,
BLAKE3 해싱(lukechampine), Myers diff(sergi/go-diff), 순수 Go SQLite(modernc.org/sqlite)를 사용한다.
SQLite 인덱싱으로 10ms 미만의 조회를 약속하고,
ACID 트랜잭션과 compare-and-swap 기반 참조 처리로 동시성 안전성을 보장한다고 설명한다.
설치는 Homebrew(`brew tap regent-vcs/tap && brew install regent`),
`go install`, 소스 빌드를 모두 지원한다.

문서상 현재 릴리스는 v1.1.0(2026년 6월 1일)이고,
Claude Code, OpenAI Codex CLI, OpenCode는 완전 지원,
Cursor와 Cline, Continue는 계획 단계로 표시된다.
VSCode 확장은 인라인 blame 주석, 호버 툴팁, 세션 타임라인, 대화 접근을 제공한다.
라이선스는 Apache 2.0이고, 비파괴 rewind/fork, 도구 어댑터 추가, 세션 공유와 머지, GC와 무결성 검증이 로드맵에 올라 있다.

## 분석

### “에이전트는 새로운 1차 작성자”라는 전제

re_gent의 핵심 주장은 단순하지 않다.
“에이전트가 코드를 쓴다”는 사실은 모두가 안다.
하지만 이 프로젝트는 한 발 더 나아가
“에이전트는 사람과 구분되는 별도의 작성자이고,
그 활동은 사람의 작성 활동과 다른 추적 도구를 필요로 한다”고 전제한다.
이 전제 위에서만 “Git이 있는데 왜 또 다른 VCS인가”라는 질문이 사라진다.

이 전제가 받아들여지면 설계의 거의 모든 선택이 따라온다.
단위가 커밋이 아니라 Step인 것,
blame이 사람이 아니라 프롬프트를 가리키는 것,
저장 객체가 코드 변경뿐 아니라 도구 호출과 대화까지 담는 것,
세션이 동시성 모델의 1급 시민인 것 모두 같은 전제의 귀결이다.
즉 re_gent는 “Git의 확장”이 아니라
“에이전트 행위라는 새 작성 단위를 위한 별도 트래킹 레이어”다.
[DeltaDB](deltadb-conversation-as-source.md)가 “대화가 소스다”라는 명제로 같은 공백을 공격하는 것과 비교하면, re_gent는 “Step이 작성 단위다”라는 더 보수적인 베팅을 한 셈이다.

이 입장은 명시적이다.
README가 “Git을 대체하지 않고 보완한다”고 못박는 것은
두 시스템이 같은 차원에 있지 않다는 주장과 같다.
사람의 커밋은 Git, 에이전트의 Step은 re_gent라는 이중 레이어 모델이
프로젝트가 그리는 미래의 작업 흐름이다.

### Git을 모방한 것은 친숙함이 아니라 호환 비용을 위해서다

저장소 레이아웃, 객체-참조 분리, blame이라는 명령 이름까지
re_gent는 Git의 형태를 거의 그대로 가져왔다.
이는 사용자 친숙함을 위한 표면적 선택처럼 보이지만,
실제로는 두 시스템이 한 워크트리에서 공존해야 한다는 제약의 결과다.
`.regentignore`가 `.gitignore`와 호환되도록 설계된 것이 같은 맥락이다.

같은 디렉터리에서 두 VCS가 동시에 돌아가려면
파일 단위 스냅샷, 객체 해시, 무시 패턴 같은 개념이 정합해야 한다.
Git의 모델을 가져오면 사용자 학습 비용이 0에 가까울 뿐 아니라,
Git 도구 생태계(IDE 통합, 코드 리뷰 도구 등)와의 충돌도 줄어든다.
즉 Git을 닮은 것은 마케팅이 아니라 협상이다.

해싱 알고리즘으로 SHA-1/SHA-256 대신 BLAKE3를 고른 것은
이 협상의 한계도 드러낸다.
대량의 도구 호출 로그와 대화 텍스트를 빠르게 해싱하려면 더 빠른 함수가 필요했고,
그 선택은 Git과의 객체 호환을 포기하는 비용을 치른다.
“Git을 따라하되 객체 수준에서는 분리한다”는 결정은
형태와 본질 사이의 타협점이다.

### 세션 단위 DAG는 멀티에이전트 시대의 베팅이다

Git의 브랜치는 “한 사람의 작업 흐름이 여럿 갈라진다”는 모델을 가정한다.
re_gent의 세션 단위 DAG는 그보다 한 단계 더 나간다.
여러 에이전트가 동시에 같은 코드베이스를 만진다는 시나리오를 1급 시민으로 다룬다.
각 세션은 자기 ref를 가지고 독립적으로 진행하다가,
공통 조상을 매개로 통합된다.

이 설계는 지금의 표준 워크플로보다 한 박자 앞을 보고 있다.
대부분의 개발자는 아직 에이전트 하나를 한 번에 하나의 작업에 쓴다.
그러나 “백그라운드에서 여러 에이전트가 다른 이슈를 동시에 처리한다”는 모델은
이미 OpenAI Codex의 클라우드 모드, Devin, Cursor 백그라운드 에이전트에서 부분적으로 등장했다.
re_gent의 세션 DAG는 이 모델이 표준이 될 때 비로소 진가가 드러나는 구조다.
[jcode](jcode.md)의 스웜 모드가 같은 저장소에서 다수 에이전트를 띄우는 시나리오를 1급 시민으로 다루는 것은 이 베팅이 도구 레벨에서도 진행 중임을 보여준다.

문제는 그 시점이 아직 오지 않았다는 데 있다.
지금은 같은 워크트리에서 동시 세션이 만들어내는 충돌을
Git의 워크스페이스 충돌로 처리하는 것이 보통이다.
re_gent는 “곧 이 방식이 무너진다”는 베팅 위에 서 있고,
그 베팅의 회수는 멀티에이전트 도입 곡선에 종속된다.

### “투명한 통합”이라는 약속에 숨은 결합도

re_gent의 가장 강한 약속은 “수동 설정 없이 동작한다”는 것이다.
`rgt init` 한 번이면 Claude Code, Codex CLI, OpenCode와 자동으로 연결된다.
이를 가능하게 하는 것은 각 도구가 노출하는 훅 시스템이다.
세션 시작/종료, 도구 호출 전/후 같은 이벤트에 re_gent가 자기 핸들러를 끼워 넣어 Step을 만든다.

이 접근은 사용자 편의를 극대화하지만,
도구별 훅 API의 안정성에 강하게 결합된다.
Claude Code의 훅 사양이 바뀌면 re_gent가 같이 바뀌어야 하고,
Cursor나 Cline처럼 훅이 약한 도구는 “계획” 상태로 남는다.
“투명함”은 추상의 두께가 아니라 어댑터의 두께로 결정된다.

이 결합은 두 가지 결과를 낳는다.
첫째, 도구 어댑터의 유지보수가 곧 프로젝트의 유지보수다.
둘째, 도구 회사들의 협력 의지가 곧 re_gent의 커버리지다.
재정/거버넌스가 약한 단계에서 이 양쪽이 동시에 흔들리면 “투명한 통합”은 빠르게 깨진다.

## 비평

### “에이전트 활동은 별도 VCS를 요구한다”는 전제는 자명하지 않다

re_gent의 모든 설계가 의존하는 전제 — 에이전트의 활동은 사람의 활동과 다른 추적 도구를 필요로 한다 — 는
글에서 충분히 변호되지 않는다.
오히려 정반대의 입장이 더 현실적이다.
“에이전트가 만든 변경도 결국 사람이 검토하고 커밋한다”는 워크플로가 표준이고,
이 워크플로에서는 에이전트의 중간 단계가 별도 VCS에 남을 필요가 없다.
필요한 것은 “이 커밋을 만들 때 어떤 대화가 있었나”라는 메타데이터일 뿐이다.

이 대안적 시각에서 보면 re_gent의 가치 제안은 좁아진다.
대화 메타데이터는 Git 노트(`git notes`),
PR 본문, 도구별 세션 ID와 외부 저장소로 충분히 표현된다.
별도 객체 저장소, 별도 인덱스, 별도 명령 체계가 필요한 사용자는
“에이전트가 사람의 검토 없이 코드를 커밋한다” 또는
“에이전트 실행 중간의 모든 상태를 사후에 복원해야 한다”는 워크플로를 가진 사람이다.
이 워크플로는 현재 소수이고, 표준이 될지도 불확실하다.

README는 `/compact`로 컨텍스트가 사라지는 문제,
“어떤 프롬프트가 이 줄을 만들었나” 추적 불가 문제,
에이전트 변경을 되돌리기 어렵다는 문제를 근거로 든다.
하지만 이들은 모두 에이전트 CLI 자체가 더 좋은 세션 영속성과 로그를 제공하면 사라지는 문제다.
즉 re_gent가 풀려는 문제 중 상당수는
“별도 VCS”가 아닌 “에이전트 CLI 개선”으로도 해결 가능하다는 점을 글은 인정하지 않는다.

HN 토론에서도 이 반론이 가장 큰 흐름을 이룬다.
한 댓글은 “그냥 Git을 써라.
Claude가 모르면 스킬과 훅으로 해결된다”고 단정한다[^tfrancisl].
또 다른 댓글은 “도구 호출 훅에서 `git add . && git commit`을 별도 `.git` 디렉터리에 떨어뜨리면 끝나는 문제”라고 정리한다[^lifis].
embedding-shape는 “에이전트에게 `~/.codex/sessions`를 뒤지라고 한 줄 적으면 되고,
AGENTS.md에 한 문장 추가하면 자동화된다”는 구체적 대안을 제시했다[^embedding-shape].
bel8은 DeepSeek가 격리된 Docker 안에서 읽기 전용 Git 명령을 수십 번 호출해
특정 기능의 타임라인 리포트를 마크다운으로 만들어냈다는 실제 경험을 보고했다[^bel8].
이 모든 사례는 re_gent의 가치 제안이 “Git을 잘 못 쓰는 사람”을 전제로 한다는 점을 드러낸다.

hombre_fatal의 답은 더 근본적이다.
“사소한 변경이 아니면 prompt → plan → impl 단계를 거쳐야 하고,
plan 파일을 같은 커밋에 포함시키면 의도와 근거가 모두 보존된다”는 워크플로다[^hombre_fatal].
이 입장에서 보면 “왜 이 줄이 만들어졌나”는 별도 VCS가 아니라
계획서가 코드와 함께 커밋되는 관행으로 답할 수 있는 문제다.
re_gent가 풀려는 문제는 워크플로 설계로 사라질 수 있는 문제이기도 하다.

### 이중 레이어 모델은 운영 비용을 숨긴다

“Git을 보완한다”는 표어는 듣기 좋지만,
실제로는 모든 저장소가 두 개의 VCS를 동시에 운영해야 한다는 뜻이다.
`.git/`과 `.regent/`이 같은 워크트리에 공존하고,
`gitignore`와 `regentignore`가 함께 동기화되고,
브랜치와 세션 ref가 별도로 관리되어야 한다.

이 운영 비용은 개인 사용자에게는 보이지 않지만 팀에서는 곧바로 드러난다.
CI는 `.regent/`를 어떻게 다룰지 결정해야 하고,
백업 정책은 두 디렉터리를 모두 포함해야 하고,
대용량 객체 저장소가 둘로 분리되면서 디스크 사용량이 늘어난다.
PR 리뷰 도구는 둘 중 하나만 본다.

더 큰 문제는 “정합성”의 책임 분산이다.
사람의 커밋과 에이전트의 Step이 시간상 어긋날 때
어느 쪽을 진실의 원천(source of truth)으로 보아야 하는지,
re_gent는 명확히 답하지 않는다.
“보완한다”는 표현은 이 충돌의 책임을 사용자에게 넘기는 표현이다.

### “10ms 미만 조회”는 데이터 규모를 가정한 약속이다

SQLite 기반 인덱싱과 BLAKE3 해싱으로 10ms 미만의 lookup을 약속한다는 서술은
벤치마크의 조건이 명시되지 않은 채로 등장한다.
어떤 크기의 저장소에서, 몇 개의 세션과 몇 개의 Step에서 측정된 수치인가가 없다.

이 누락은 사소해 보이지만 핵심을 가린다.
Step에 도구 호출의 입출력과 대화 텍스트가 함께 담긴다면,
저장소는 코드 라인 수가 아니라 토큰 수에 비례해 증가한다.
한 세션이 수십만 토큰의 대화를 남기고,
대형 코드베이스에서 수십 개의 세션이 동시에 돌면
SQLite 인덱스의 크기는 수십 GB 단위로 자랄 수 있다.

이 규모에서도 10ms를 유지할 수 있느냐는 별도의 검증이 필요하다.
GC와 무결성 검증이 “로드맵”에 있다는 표시는,
이 규모 문제가 아직 본격적으로 다뤄지지 않았음을 시사한다.
“SQLite로 빠르다”는 약속은 작은 저장소의 약속이지 큰 저장소의 약속이 아니다.

### 도구 어댑터 의존성은 “투명함”을 일시적 상태로 만든다

re_gent의 “수동 설정 없는 통합”은 각 도구의 훅 시스템에 강하게 결합된다.
Claude Code의 훅 사양, Codex CLI의 이벤트 모델, OpenCode의 통합 지점이 각자 다른 형태로 노출되고,
re_gent는 그 차이를 어댑터 코드로 흡수한다.

이 구조는 어댑터 한 개당 유지보수 부담이 따라붙는다는 뜻이다.
도구가 5개로 늘면 통합 테스트 표면적이 곱셈으로 증가하고,
각 도구의 마이너 업데이트가 회귀 가능성을 만든다.
“Cursor, Cline, Continue는 계획”이라고 적힌 것은
이 곱셈의 비용을 단일 메인테이너 또는 소규모 팀이 아직 감당하지 못한다는 신호다.

더 근본적인 문제는 도구 회사들이 자체적으로 같은 기능을 내장할 수 있다는 점이다.
Claude Code가 자체 세션 영속성과 prompt-to-line 매핑을 제공하기 시작하면,
re_gent의 가치 제안은 그 도구에서는 사라진다.
“투명한 통합”은 결국 도구 제공자가 같은 기능을 내장하지 않는 동안에만 유지되는 일시적 상태다.

HN 토론에서 binyu가 지적한 Cloudflare Artifacts[^binyu]는 이 위협을 구체적으로 보여준다.
“에이전트를 위한 Git”이라는 같은 표어로 인프라 기업이 같은 카테고리에 진입한 순간,
“별도 CLI 도구”의 자리는 빠르게 좁아진다.
도구 회사뿐 아니라 클라우드 사업자, 코드 호스팅 플랫폼, 감사 SaaS가
같은 문제를 자기 제품의 일부로 가져오면 re_gent의 차별성은 어댑터 수에 갇힌다.

## 인사이트

### 에이전트 활동의 감사 가능성은 거버넌스 요구로 곧 강제된다

re_gent의 가장 큰 시장은 개인 개발자가 아니라 규제 산업의 팀이다.
금융, 의료, 공공 분야에서 “에이전트가 코드를 만졌다”는 사실은 곧 감사 대상이 된다.
어떤 모델이, 어떤 프롬프트에 응답해, 어떤 줄을 바꿨는가를
사후에 재현 가능하게 보존하지 못하면 컴플라이언스를 통과할 수 없다.

이 요구는 도구의 편의 기능이 아니라 외부 강제다.
EU AI Act, SOC 2의 변경 관리 요건, 의료기기 소프트웨어 규제가 모두
“코드가 어떻게 만들어졌는가”의 증명을 요구하는 방향으로 가고 있다.
사람의 작성은 Git 커밋과 PR로 충분히 추적되지만,
에이전트의 작성은 지금까지 그 추적 체인 바깥에 있었다.
re_gent 같은 도구가 풀려는 것이 이 공백이다.

흥미로운 점은 이 시장에서 “Git을 닮은 CLI 도구”의 형태가 정답이 아닐 수도 있다는 것이다.
규제 시장은 SIEM, 감사 로그 플랫폼, 변경 관리 시스템과의 통합을 요구한다.
re_gent가 단순한 CLI에서 멈추면 이 시장은 다른 제품 — 예컨대 GitHub의 Copilot Audit이나
별도 SaaS 형태의 에이전트 활동 추적 — 에 빼앗긴다.
re_gent의 진짜 경쟁자는 다른 VCS가 아니라 감사 플랫폼이다.

### Git 모방은 단기 이득과 장기 종속을 동시에 만든다

명령 이름, 저장소 레이아웃, 무시 패턴까지 Git을 모방한 것은
초기 학습 곡선을 거의 0으로 만든다.
하지만 같은 결정이 장기적으로는 종속을 만든다.
“커밋 단위”의 추상은 사람의 작업 흐름에 맞춰 진화한 것이고,
에이전트의 작업 흐름은 본질적으로 다르다.

에이전트의 한 Step은 사람의 커밋보다 훨씬 자주, 훨씬 작게 발생한다.
도구 호출 하나, 파일 한 줄의 수정, 실패한 시도 하나가 모두 Step이 될 수 있다.
이 단위로 보면 “log”, “blame”, “show” 같은 Git의 명령은
잘못된 모델로 데이터를 보여주는 도구가 된다.
사람은 “이 줄을 누가 만들었나”를 묻지만,
에이전트 활동의 진짜 질문은 “이 결과에 도달하기까지 어떤 시도들이 있었나”다.

이 관점에서 보면 HN의 Zambyte가 언급한 Jujutsu(jj) 워크플로[^Zambyte]가 흥미로운 단서를 준다.
jj는 모든 변경을 자동으로 임시 커밋으로 만들고,
한 사람이 여러 프롬프트를 거치며 변경을 다듬다가
“최종 커밋”으로 응축하는 흐름을 자연스럽게 지원한다.
프롬프트 단위 undo와 커밋 단위 의도 사이의 간극을
별도 VCS 없이 같은 트리에서 해결하는 모델이다.
re_gent가 별도 객체 저장소까지 만들지 않고도
“커밋의 단위와 프롬프트의 단위가 다르다”는 문제만 풀고 싶다면,
jj 스타일의 자동 임시 커밋이 더 가벼운 답이 될 수 있다.

User23이 “darcs 쪽이 더 맞다”고 언급한 것[^User23]도 같은 맥락에 있다.
darcs의 패치 대수(patch algebra)는 변경을 시간순 스냅샷이 아니라
교환 가능한 패치의 집합으로 본다.
에이전트의 시도가 순차적이 아니라 병렬적으로 일어난다면,
교환 가능한 패치 단위가 커밋보다 더 자연스러운 추상이 된다.
“Git을 닮을 것인가”라는 질문 자체가 이미 답을 좁히고 있는 셈이다.

Git의 형태에 갇히면 이 질문에 답할 수 있는 새로운 추상 — 시도 트리, 의사결정 그래프, 실패 회로 — 을 만들기 어렵다.
역사적으로 비슷한 사례는 풍부하다.
모바일 OS가 데스크톱 UI를 모방하다가 결국 새 추상으로 옮겨간 것,
LLM의 채팅 인터페이스가 처음에는 IRC를 모방하다가
멀티턴 대화의 새 형태를 만들어낸 것이 그렇다.
re_gent의 다음 단계는 Git을 더 닮는 것이 아니라 Git을 떠나는 것이 되어야 한다.

### “이중 레이어”는 단일 레이어로 통합되거나 한쪽이 흡수한다

re_gent가 그리는 미래 — 사람의 Git과 에이전트의 re_gent가 나란히 돌아가는 — 는
역사적으로 오래 유지되기 어려운 형태다.
두 시스템이 같은 워크트리에서 동시에 운영되면
시간이 갈수록 통합 압력이 쌓인다.
같은 압력이 작용한 사례는 많다.

JavaScript와 TypeScript의 관계, npm과 yarn/pnpm의 관계,
Docker와 Kubernetes의 관계가 모두 같은 패턴을 보여준다.
처음에는 별도 레이어로 시작하지만,
시간이 지나면 통합되거나(TypeScript가 JavaScript 도구 체인을 흡수)
한쪽이 다른 쪽의 기능을 점진적으로 흡수한다(Git이 LFS, Notes, Hooks로 확장된 방식).

re_gent가 풀려는 문제도 같은 경로를 따를 가능성이 높다.
첫째 시나리오는 Git이 “Notes 2.0” 같은 형태로 에이전트 메타데이터를 흡수하는 것이다.
둘째 시나리오는 에이전트 CLI(Claude Code, Codex CLI 등)가 자체 세션 영속성을 강화해
re_gent의 가치를 도구 내부로 가져가는 것이다.
셋째 시나리오만이 re_gent가 살아남는 길인데,
그것은 “감사/거버넌스 플랫폼”으로 카테고리를 바꾸는 경우다.

### 멀티에이전트 동시성은 머지 충돌의 의미를 바꾼다

re_gent의 세션 단위 DAG가 진짜 가치를 가지는 순간은
사람이 PR 리뷰를 하는 시나리오가 아니라
여러 에이전트가 동시에 같은 코드베이스를 만지고
사람은 결과만 검토하는 시나리오다.
이 시나리오에서 머지 충돌은 사람 간 충돌과 다른 의미를 가진다.

사람 간 충돌은 “둘 다 의도가 있었지만 합쳐야 한다”는 문제다.
에이전트 간 충돌은 “둘 다 같은 목표를 다르게 시도했다”는 문제다.
전자는 양쪽 의도를 보존하는 방향으로 해결해야 하지만,
후자는 어느 쪽 결과가 더 나은가의 평가 문제로 환원된다.
이 차이가 새로운 충돌 해결 추상을 요구한다.
[parallel-coding-agents-with-tmux](parallel-coding-agents-with-tmux.md)가 “여러 에이전트가 가설을 병렬 검증하고 최선을 채택한다”는 워크플로를 소개한 것은 이 “세션 선택” 패러다임의 초기 형태다.

re_gent의 로드맵에 “세션 머지”가 올라 있는 것은 이 차이를 인식한 신호다.
하지만 표준 Git 머지의 라인 단위 충돌 해결로는 부족하다.
필요한 것은 “두 세션이 같은 목표를 향했나”를 판별하고,
“어느 시도가 더 나은가”를 자동으로 또는 사람의 한 번의 결정으로 정리하는 도구다.
이 방향이 발전하면 멀티에이전트 코딩의 표준 워크플로 자체가 바뀐다.
“PR 머지”가 아니라 “세션 선택”이 코드 통합의 단위가 된다.
re_gent가 이 추상을 선점할 수 있느냐가 도구의 운명을 결정할 변곡점이다.
