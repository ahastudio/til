# Claude Code의 Dynamic Workflows

원문: <https://javaexpert.tistory.com/1770>

## 요약

Claude Code의 Dynamic Workflows는 에이전트가 주어진 작업에 맞춰 스스로 작업 구조를 설계하고 실행하는 기능이다.
단순히 코드를 작성하는 것을 넘어, 작업을 어떻게 나누고 검증하고 종합할지를 에이전트 스스로 결정하는 “작업용 하네스(task harness)”를 동적으로 생성한다.

이 기능이 등장한 배경에는 단일 에이전트 인스턴스의 한계가 있다.
긴 작업을 단일 Claude 인스턴스에 맡기면 “에이전트 게으름(agentic laziness)”이 발생한다.
컨텍스트가 길어질수록 자기 선호 편향이 생기고, 원래 목표에서 이탈하며, 작업을 완료하지 않고 마무리하는 경향이 나타난다.
Dynamic Workflows는 이 문제를 각 하위 에이전트에게 독립된 목표와 컨텍스트를 부여하는 분업 구조로 해결한다.

적용 범위는 코딩에 국한되지 않는다.
리서치, 보안 분석, 이력서 평가, 기술 문서 검증, 장애 분석 등 복잡한 비기술 작업에도 동일한 구조가 적용된다.
공통점은 “단일 에이전트가 한 번에 처리하기 어려운 복잡한 작업”이라는 것이다.

주요 워크플로 패턴으로는 여섯 가지가 제시된다.
Classify and Act(분류 후 맞춤형 처리), Fan-out and Synthesize(병렬 처리 후 종합),
Adversarial Verification(결과의 공격적 검증), Generate and Filter(다중 생성 후 필터링),
Tournament(경쟁식 비교 판단), Loop Until Done(조건 충족까지 반복)이 그것이다.
워크플로는 JavaScript 기반 파일로 생성되며, 격리된 worktree에서 병렬 실행이 가능하다.

저자는 이 변화가 AI 경쟁력의 이동을 나타낸다고 본다.
개별 답변의 품질에서 “작업 구조 설계 능력”으로 AI의 가치 중심이 이동하고 있다는 것이다.
Skill과 결합하면 팀 차원에서 재사용 가능한 워크플로 라이브러리를 구축할 수 있다.

## 분석

### 에이전트 게으름 문제의 구조적 해법

“에이전트 게으름”이라는 표현은 흥미롭지만, 그 본질은 컨텍스트 윈도우의 물리적 한계와 어텐션(attention) 메커니즘의 특성에서 비롯된다.
트랜스포머 기반 모델은 컨텍스트가 길어질수록 초반부의 정보에 대한 주의(attention)가 분산된다.
“처음에 뭘 하려고 했지”를 잊어버리는 것은 나태함이 아니라 아키텍처의 특성이다.

Dynamic Workflows가 이 문제를 해결하는 방식은 분업의 단위를 모델이 아닌 컨텍스트 윈도우로 재정의하는 것이다.
각 하위 에이전트는 새로운 컨텍스트 윈도우에서 명확한 목표를 가지고 시작한다.
이것은 마이크로서비스가 모놀리스(monolith)의 복잡도 문제를 서비스 경계로 해결하는 것과 구조적으로 같다.

### 여섯 가지 패턴의 계층 구조

제시된 여섯 패턴은 독립적인 것이 아니라 계층을 이룬다.
Classify and Act는 조건 분기, Fan-out and Synthesize는 병렬화, Loop Until Done은 반복이라는 프로그래밍의 기본 제어 구조에 대응한다.
Adversarial Verification과 Tournament는 검증 전략이고, Generate and Filter는 탐색과 선별의 분리다.

이 패턴들은 소프트웨어 아키텍처 패턴과 유사한 방식으로 조합된다.
Fan-out 이후 Adversarial Verification을 적용하거나, Generate and Filter 후 Tournament로 최종 선별하는 조합이 가능하다.
워크플로 설계가 코드 설계와 같은 수준의 추상화 작업이 된다는 의미다.

### Skill과의 통합이 만드는 조직 수준의 함의

개인 워크플로에서 팀 Skill 라이브러리로의 확장은 단순한 편의 기능이 아니다.
“이 유형의 작업은 이 워크플로로 처리한다”는 조직적 합의가 코드로 명시화된다.
개발 문화와 관행이 구두 규범에서 실행 가능한 구조물로 전환되는 것이다.

이것은 소프트웨어 엔지니어링에서 코딩 컨벤션(coding convention)이 린터(linter)로 자동화된 것과 유사한 전환이다.
“좋은 방식으로 하자”는 합의가 “이 워크플로를 쓰면 좋은 방식으로 된다”는 구조로 바뀐다.

## 비평

### 강점: 패턴 언어로의 정리

여섯 가지 워크플로 패턴을 이름 붙이고 분류한 것은 실무적으로 유용하다.
“에이전트한테 시켜봤는데 잘 안 됐다”는 막연한 경험을 “어떤 패턴을 써야 하는가”라는 질문으로 구체화할 수 있다.
패턴 언어(pattern language)는 Gang of Four의 디자인 패턴이 증명한 것처럼, 설계 지식을 전달하는 강력한 형식이다.

### 약점: 워크플로 실패 패턴의 부재

성공 패턴만 제시하고 실패 패턴이나 안티패턴(anti-pattern)을 다루지 않는다.
어떤 상황에서 Fan-out이 비효율적인지, Loop Until Done이 무한 루프가 될 수 있는 조건은 무엇인지,
Adversarial Verification이 오히려 잘 작동하는 결과를 부결시키는 경우는 언제인지가 없다.
패턴을 언제 쓰지 말아야 하는지를 모르면 패턴의 남용이 생긴다.

### 보완할 시각: 비용과 지연 시간의 트레이드오프

병렬 하위 에이전트 실행은 처리 속도를 높이지만 비용은 비례해 증가한다.
Fan-out 패턴에서 N개의 하위 에이전트를 병렬 실행하면 비용도 N배가 된다.
어떤 작업이 병렬화의 비용을 정당화하는지에 대한 기준이 제시되지 않는다.
단순한 작업을 복잡한 워크플로로 처리하는 과잉 엔지니어링 위험이 있다.

## 인사이트

### 에이전트의 능력이 아닌 구조가 결과를 결정한다

“더 좋은 모델을 쓰면 더 좋은 결과가 나온다”는 직관이 흔들리고 있다.
Dynamic Workflows가 시사하는 것은 동일한 모델도 작업 구조에 따라 결과의 품질이 크게 달라진다는 것이다.

이것은 소프트웨어 엔지니어링에서 이미 알려진 사실이다.
뛰어난 개발자도 잘못 설계된 시스템 안에서는 성과를 내기 어렵다.
개발자의 능력보다 시스템 설계가 팀 생산성을 결정하는 경우가 많다.
에이전트도 같은 원칙이 적용된다.
단일 에이전트에게 복잡한 작업을 던지는 것은 한 명의 개발자에게 수십 개의 PR을 동시에 처리하게 하는 것과 같다.

이 인식의 전환은 AI 활용의 핵심 역량이 무엇인지를 바꾼다.
“프롬프트를 잘 쓰는 능력”에서 “작업 구조를 설계하는 능력”으로 이동한다.
워크플로 설계자(workflow architect)라는 새로운 역할이 등장하고 있는 것이다.

### 검증(verification)의 재위치화

Adversarial Verification과 Tournament 패턴이 독립된 패턴으로 제시된다는 점은 중요하다.
검증이 작업의 마지막 단계가 아니라 독립적인 에이전트 역할로 격상된 것이다.

전통적인 소프트웨어 개발에서 테스트와 개발의 분리는 오랜 논쟁이었다.
개발자가 자기 코드를 테스트하면 자기 가정을 그대로 이어가는 경향이 있다.
독립적인 테스터나 코드 리뷰어가 다른 시각에서 결함을 발견한다.
Dynamic Workflows에서 Adversarial Verification은 이 원칙을 에이전트 레벨로 가져온 것이다.
코드를 생성한 에이전트와 다른 에이전트가 독립된 컨텍스트에서 결과를 검증한다.

이것은 AI 생성 결과물의 신뢰성 문제를 해결하는 구조적 접근이다.
단일 에이전트의 출력을 신뢰하기 어렵다는 사실을 인정하고, 불신을 시스템 설계에 내재화한다.
“AI를 믿어라”가 아니라 “AI가 AI를 검증하게 하라”는 설계 철학이다.

### 워크플로가 지식이 되는 조직

팀 Skill 라이브러리로의 확장이 가리키는 더 깊은 변화가 있다.
조직의 노하우가 자연 언어 문서가 아닌 실행 가능한 워크플로로 축적되기 시작하는 것이다.

지금까지 팀의 암묵지(tacit knowledge)는 시니어 개발자의 머릿속에 있었다.
코드 리뷰 기준, 아키텍처 결정의 논리, 특정 상황에서의 대응 방식이 문서화되더라도 자연 언어는 해석에 편차가 생긴다.
실행 가능한 워크플로는 다르다.
“이 상황에서 이렇게 한다”는 규범이 코드로 명시되고, 누가 실행하든 동일한 구조로 작동한다.

장기적으로 이것은 조직 지식의 성격을 바꾼다.
“이 팀에서 오래 일한 사람이 더 나은 판단을 한다”에서 “이 팀의 워크플로를 쓰면 좋은 판단이 구조화된다”로.
암묵지에서 형식지로의 전환이 에이전트 워크플로라는 새로운 형태로 가속되고 있다.
