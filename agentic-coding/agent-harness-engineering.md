# Harness Engineering: AI 에이전트를 실제로 움직이게 하는 인프라 설계

원문: <https://addyosmani.com/blog/agent-harness-engineering/>

## 요약

Addy Osmani는 AI 코딩 에이전트의 성능이 모델 그 자체보다 모델을 감싸는 “하네스(harness)”에
더 크게 좌우된다고 주장한다.
핵심 공식은 단순하다: **코딩 에이전트 = AI 모델 + 하네스**.
“좋은 하네스를 가진 평범한 모델이 나쁜 하네스를 가진 좋은 모델보다 낫다”는 것이
글의 테제다.

하네스는 모델 외부의 모든 인프라 구성 요소를 포함한다.
시스템 프롬프트, 스킬 문서, 도구 정의(MCP 서버 포함), 실행 환경, 오케스트레이션 로직,
미들웨어 훅, 관찰 가능성 인프라가 모두 하네스를 구성한다.
Terminal Bench 2.0 데이터는 최적화된 하네스 안에서 Claude Opus 4.6이
기본 환경보다 유의미하게 높은 성능을 보임을 보여준다.

에이전트 실패를 다루는 “래칫(Ratchet) 원칙”이 핵심 방법론이다.
에이전트가 실수할 때마다 그 실수를 영구적인 신호로 기록하고,
재발을 막는 규칙을 하네스에 추가한다.
실수를 예외가 아닌 데이터 포인트로 취급하는 이 접근은, 하네스를 점진적으로 강화하는
엔지니어링 루프를 형성한다.

하네스의 주요 구성 요소로는 파일시스템·Git(내구적 상태), Bash 실행 환경,
샌드박스(격리된 실행), 메모리·검색(지속적 학습), 컨텍스트 관리(3가지 기법),
장기 실행을 위한 Ralph 루프와 플래너/평가자 분리, 그리고 훅(hooks)이 있다.

산업 수준에서 Claude Code, Cursor, Codex, Aider, Cline 등 경쟁 에이전트들이
서로 다른 모델을 사용하면서도 점점 서로를 닮아가는 수렴 현상이 관찰된다.
하네스 패턴이 표준화되고 있다는 증거이며, 이 스캐폴딩이 에이전트 역량의
핵심 구성 요소임을 시사한다.

## 분석

### “모델보다 하네스” 주장의 근거

이 주장을 뒷받침하는 가장 강력한 근거는 실증적 데이터다.
HumanLayer의 분석에 따르면, 에이전트 실패의 대부분은 “모델 가중치가 아닌 설정 문제”에서
비롯된다.
같은 모델이 하네스 품질에 따라 크게 다른 성능을 보인다는 것은,
모델 선택보다 하네스 설계가 더 높은 레버리지를 갖는다는 의미다.

이 주장은 소프트웨어 공학의 기존 원칙과도 일치한다.
가장 좋은 알고리즘도 나쁜 실행 환경에서는 기대한 성능을 내지 못한다.
AI 에이전트의 경우, 실행 환경이 단순한 하드웨어나 런타임을 넘어
프롬프트·도구·오케스트레이션 전체를 포함하는 것이 특이점이다.

### 래칫 원칙의 엔지니어링적 의미

래칫 원칙은 에이전트 개발을 소프트웨어 테스트와 유사한 반복 프로세스로 정의한다.
에이전트가 테스트 주석 처리를 하면 그것을 막는 규칙을 추가한다.
매 규칙은 과거의 구체적인 실패에서 비롯된다.

이 방법론은 중요한 함의를 갖는다.
하네스는 한 번 구축하고 끝나는 것이 아니라, 에이전트가 실수할 때마다 강화되는
살아있는 시스템이다.
AGENTS.md(Claude Code) 같은 파일이 60줄 이하의 “조종사 체크리스트” 형태를 유지해야 한다는
권고는, 규칙의 질(과거 실패와의 연결)을 양(규칙 수)보다 중시하는 원칙의 표현이다.

### 하네스 표준화와 Harness-as-a-Service

에이전트 간 수렴 현상은 특정 기업의 우위보다 산업 전체의 표준화를 예고한다.
Osmani는 이를 “Harness-as-a-Service(HaaS)”로 명명한다.
LLM API가 완성(completion)을 제공하듯, 하네스 API는 루프 로직·도구·컨텍스트 관리·훅을
번들로 제공한다.
Anthropic의 Claude Code SDK, OpenAI의 Codex CLI가 이 범주에 해당한다.

## 비평

### 강점

추상적 개념을 구체적인 구성 요소로 분해한다는 점에서 글의 구조는 탁월하다.
“에이전트 = 모델 + 하네스”라는 단순한 공식에서 출발해,
각 하네스 구성 요소를 “어떤 행동을 원하는가”에서 역산하는 방식은 실용적이다.
Claude Code를 성숙한 하네스 설계의 예시로 구체적으로 분석한 것도 가치가 있다.

### 마케팅 용어 남발 비판

[GeekNews 댓글](https://news.hada.io/topic?id=28966)에서 kimjoin2는
“점점 마케팅 용어만 엄청 생기는 느낌”이라고 지적했다.
“하네스 엔지니어링”이라는 용어 자체가 기존 개념—CI/CD, DevOps, 프롬프트 엔지니어링,
MLOps—의 재포장에 가깝다는 비판이 가능하다.
파일시스템, Git, Bash 같은 기본 도구를 “하네스의 핵심 구성 요소”로 나열하는 것은,
이미 알고 있는 것에 새로운 이름을 붙이는 것처럼 읽힐 수 있다.

### 이론과 실천의 괴리

kurthong은 Osmani가 자신의 이론을 Google 제품에 직접 구현하는 대신 글을 쓰는 데
집중하는 것을 비판했다.
이는 더 넓은 질문을 제기한다: Karpathy처럼 직접 모델을 훈련하거나 도구를 빌드하는
실천가와, 모범 사례를 정리하고 전파하는 전도사는 다른 역할을 한다.
이 글은 전자보다 후자에 가깝다.

dongho42는 “AI가 명확한 지시에도 확률적으로 무시하는 경우”에 하네스 접근이
실제로 얼마나 효과적인지 의문을 제기했다.
규칙 기반 하네스는 에이전트가 규칙을 따를 때만 작동한다.
확률적 무시는 설정 문제가 아닌 모델 문제이며, 하네스로 완전히 해결되지 않는다.

### 추상화 수준의 불균형

글의 일부는 매우 구체적(Claude Code 아키텍처의 6개 레이어)이고,
일부는 매우 추상적(“하네스는 결국 컴파일러처럼 진화할 것이다”)이다.
이 불균형이 글의 실용성을 다소 희석시킨다.
“어떻게 시작하는가”에 대한 단계적 안내가 부족하다.

## 인사이트

### 에이전트 경쟁의 무게 중심 이동

“좋은 모델 찾기”에서 “좋은 하네스 만들기”로의 이동은 경쟁 지형을 바꾼다.
모델 성능은 Anthropic, OpenAI, Google 같은 소수의 대형 플레이어가 결정하지만,
하네스 설계는 누구나 최적화할 수 있다.
이것은 스타트업과 개인 개발자에게 기회다.
최고의 모델에 접근하는 것이 상품화될수록, 하네스 설계 역량이 차별화 요소가 된다.

Claude Code, Cursor, Codex가 서로를 닮아가는 것은 이 원칙의 실증이다.
하네스 패턴이 표준화되면, 더 이상 하네스 자체가 경쟁 우위가 아니다.
다음 차별화는 도메인 특화 하네스—특정 산업이나 워크플로에 최적화된—에서 나올 것이다.

### AGENTS.md는 조직 학습의 외재화다

래칫 원칙의 핵심은 실수를 문서화해 재발을 막는 것이다.
이것은 소프트웨어 공학에서 포스트모템(post-mortem)이 수행하는 역할과 유사하다.
포스트모템이 인간 팀의 집단 기억을 구조화하듯, AGENTS.md는
에이전트의 “경험에서 학습한 규칙”을 구조화한다.

흥미로운 것은, 이 파일이 인간 팀원에게도 유용하다는 점이다.
AGENTS.md에 기록된 규칙들은 프로젝트의 암묵적 지식—“이 프로젝트에서 무엇을 하면 안 되는가”
—을 명시적으로 만든다.
AI 에이전트를 위해 작성하지만, 결과적으로 인간 온보딩 문서로도 기능한다.

### 하네스의 진화가 모델 훈련에 피드백된다

Osmani가 제시하는 “모델-하네스 훈련 루프”는 장기적으로 매우 중요하다.
하네스 혁신이 제품에 표준화되면, 모델은 그 패턴이 내재화된 채로 훈련된다.
Claude Code의 도구 사용 패턴이 Claude 모델에 내재화되는 것이 한 예다.

이 루프는 AI 개발의 속도를 가속한다.
오늘의 하네스 “최선 사례”가 내일의 모델 기본 역량이 된다.
따라서 하네스 혁신가들은 자신의 발견이 결국 모델에 흡수되어 “보통 수준”이 될 것을
예상하고, 항상 다음 단계를 준비해야 한다.
하네스 엔지니어링은 AI 역량의 프론티어를 계속 앞으로 밀어내는 선행 투자다.
