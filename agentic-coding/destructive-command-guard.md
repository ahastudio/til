# Destructive Command Guard: 에이전트의 손을 붙잡는 Rust 방화벽

<https://github.com/Dicklesworthstone/destructive_command_guard>

## 소개

Destructive Command Guard(이하 dcg)는 코딩 에이전트가 `git reset --hard`,
`rm -rf ./src`, 데이터베이스 드롭 같은 되돌릴 수 없는 명령을 실행하기 직전에
가로채서 막는 Rust 기반 도구다.
Claude Code, Codex CLI, Gemini CLI, GitHub Copilot CLI, Cursor IDE, Hermes
Agent, Grok 등 주요 코딩 에이전트의 `PreToolUse` 훅에 끼어들어 동작한다.
공개 당시 GitHub에서 2,770개의 별과 105개의 포크를 기록했고, 하루 만에
444개의 별이 늘어날 정도로 빠르게 주목받았다.

동작 방식은 단순하다.
에이전트가 셸 명령을 실행하기 직전, dcg에 JSON 형태로 명령 문자열을
표준입력으로 전달한다.
dcg는 이를 정규화하고 패턴과 대조한 뒤, 위험하다고 판단되면 실행을 거부하고
사람이 읽을 수 있는 경고를 표준에러로 출력한다.
핵심은 이 판단이 에이전트의 “의도”가 아니라 명령 문자열 자체에 대한
결정론적 규칙에서 나온다는 점이다.

## 분석

### 4단계 탐지 파이프라인의 설계 의도

dcg의 탐지 과정은 JSON 파싱, 명령 정규화, 빠른 제외(quick reject), 패턴
매칭의 네 단계로 나뉜다.
`/usr/bin/git`처럼 절대 경로로 호출된 명령도 `git`으로 정규화해 우회를
막고, “git”이나 “rm” 같은 키워드가 아예 없는 명령은 정규식 평가 없이 O(n)
문자열 검사만으로 걸러낸다.
README에 따르면 이 단계에서 안전한 명령의 약 99%가 정규식 엔진을 거치지
않고 통과한다.

이 구조는 성능과 정확성을 함께 노린 설계다.
에이전트의 모든 셸 호출에 개입하는 도구는 지연 시간에 매우 민감할 수밖에
없는데, 무거운 정규식 매칭을 먼저 걸어두면 개발자가 체감하는 에이전트
응답 속도 자체가 느려진다.
반대로 검사를 건너뛰면 보호 기능이 무력해진다.
빠른 제외 단계로 절대다수의 안전한 명령을 조기에 통과시키고, 나머지에만
비용이 큰 패턴 매칭을 적용하는 방식은 이 긴장을 실용적으로 절충한
접근이다.

안전 패턴을 먼저 확인해 매치되면 허용하고, 그다음 위험 패턴을 확인해
매치되면 거부하며, 어느 쪽에도 걸리지 않으면 기본적으로 허용한다는
순서도 눈여겨볼 지점이다.
이는 뒤에서 다룰 “실패 시 개방(fail-open)” 철학과 같은 맥락에 있다.

### 헤레독과 인라인 스크립트까지 파고드는 3단계 스캔

dcg가 단순 문자열 매칭 도구와 다른 지점은 헤레독(heredoc)이나 인라인
스크립트 내부까지 분석한다는 점이다.
1단계는 100마이크로초 이내에 헤레독 신호를 RegexSet으로 감지하고, 2단계는
1밀리초 이내에 크기와 줄 수를 제한(최대 1MB, 1만 줄, 명령당 헤레독 10개)해
내용을 추출하며, 3단계는 5밀리초 이내에 tree-sitter 기반 AST 파싱으로
셸 내부 구조를 재귀적으로 분석한다.

이 계층 구조가 필요한 이유는 에이전트가 `bash -c “$(cat <<'EOF' ... EOF)”`
같은 형태로 위험한 명령을 문자열 안에 숨겨서 실행할 수 있기 때문이다.
표면적인 명령줄만 검사하면 이런 우회를 놓친다.
동시에 임의의 셸 스크립트를 완전히 파싱하는 작업은 계산 비용이 크므로,
단계별로 시간 예산을 엄격히 나눠 병목을 막는 구조를 택했다.
AST 분석으로 주석, 인용 문자열, 헤레독 본문에 등장하는 패턴과 실제로
실행되는 명령을 구분하는 것도 오탐(false positive)을 줄이는 데
직접적으로 기여한다.

### 50개 이상의 옵트인 팩과 에이전트별 신뢰 수준

dcg는 파일시스템 삭제, Git 히스토리 재작성, 디스크 포맷처럼 항상 켜져
있는 핵심 규칙 외에, 데이터베이스(PostgreSQL, MongoDB, Redis 등),
컨테이너(Docker, Kubernetes), 클라우드(AWS, Azure, GCP), CI/CD, 시크릿
관리 도구까지 50개 이상의 팩으로 규칙을 분리해 두었다.
`enabled = [“database”]`처럼 카테고리 단위로 지정하면 하위 팩이 자동으로
모두 활성화되는 구조다.

여기에 더해 에이전트별로 다른 신뢰 수준을 설정할 수 있다는 점이 이
도구의 실질적인 차별점이다.
`[agents.{agent-name}]` 설정으로 특정 에이전트에는 더 넓은 허용 목록을
주고, 검증되지 않은 에이전트에는 더 많은 팩과 엄격한 차단을 적용하는
식이다.
이는 모든 코딩 에이전트가 동일한 위험 수준을 갖는다는 전제를 버리고,
운영자가 각 에이전트에 대한 자신의 신뢰도를 직접 정책으로 표현하도록
설계했다는 뜻이다.

## 비평

### “차단”이 아니라 “지연”에 가까운 보호

dcg의 핵심 설계 철학은 fail-open, 즉 파싱이 실패하거나 타임아웃이
발생하거나 자원 제한을 초과하면 명령을 차단하지 않고 허용한다는
것이다.
README는 이를 “절대적 안전보다 작업 연속성을 우선한다”는 말로
정당화하지만, 이는 곧 도구의 보호 범위가 자신이 성공적으로 분석해낸
명령에만 한정된다는 뜻이기도 하다.
공격자나 실수하는 에이전트가 아니라, 단순히 “규칙이 예상하지 못한
형태의 명령”만으로도 보호막을 우회할 수 있다.

헤레독 추출이 1MB, 1만 줄, 헤레독 10개로 제한된다는 점도 같은 문제의
연장선에 있다.
이 한계를 넘는 스크립트는 애초에 분석 대상에서 제외되고, 절대 타임아웃인
200밀리초를 넘기는 명령도 마찬가지로 검사를 건너뛴다.
성능을 지키기 위한 합리적인 트레이드오프이지만, 실사용에서는 “위험한
명령일수록 복잡하고 길어서 검사를 피해간다”는 역설적인 상황이 생길
여지가 있다.

### Codex CLI 사례가 보여주는 훅 기반 보호의 구조적 한계

README는 스스로 Codex CLI에서 모델이 실행 가능한 스크립트를 디스크에
써서 훅을 우회할 수 있다는 점, 그리고 Windows의 `unified_exec` 경로
일부를 `PreToolUse` 훅이 가로채지 못한다는 점을 인정한다.
이는 dcg의 결함이라기보다 “에이전트가 호출하는 도구 실행 지점을 가로채는”
훅 아키텍처 자체의 한계에 가깝다.

문제는 이 한계가 정확히 dcg가 막으려는 시나리오와 맞닿아 있다는
점이다.
에이전트가 의도적으로 위험한 작업을 숨기려는 것이 아니라, 단순히 목표를
달성하는 가장 직접적인 경로로 스크립트 파일을 쓰고 실행하는 흐름을
택했을 뿐인데도 보호가 뚫린다.
즉 회피를 노린 공격이 아니라 정상적인 작업 패턴만으로도 사각지대가
생길 수 있다는 뜻이며, 이는 훅 하나만으로 완결된 방어선을 구축하기
어렵다는 사실을 보여준다.

### 팩이 늘어날수록 커지는 설정 부담

50개 이상의 옵트인 팩과 에이전트별 신뢰 수준 조합은 유연성을 준다는
장점이 있지만, 동시에 운영자가 “우리 조직에는 어떤 팩이 필요한가”를
스스로 판단해야 하는 부담을 떠넘긴다.
새로운 데이터베이스나 클라우드 서비스를 도입할 때마다 관련 팩을
활성화했는지 확인해야 하고, 활성화를 잊으면 보호가 조용히 비어 있는
상태로 남는다.

기본값이 “핵심 팩만 켜고 나머지는 꺼져 있다”는 구조인 이상, 이 도구의
실질적인 보호 범위는 설정 파일을 얼마나 꼼꼼히 채워 넣었는가에
비례한다.
자동으로 프로젝트에서 사용 중인 인프라를 감지해 관련 팩을 제안하는
기능이 없다면, 신규 도입 조직은 자신이 어떤 위험에 노출돼 있는지조차
알기 어려운 구조다.

## 인사이트

### 에이전트 신뢰를 코드로 명문화하면 조직의 암묵적 판단이 드러난다

`[agents.{agent-name}]` 설정으로 에이전트별 신뢰 수준을 다르게 주는
기능은 단순한 편의 기능이 아니다.
지금까지 “이 에이전트는 믿을 만하다”는 판단은 개발자 개개인의 머릿속에
암묵적으로 존재했을 뿐, 조직 차원에서 문서화되거나 검토된 적이 거의
없었다.
dcg처럼 이를 설정 파일에 명시적으로 적어야 하는 도구가 보급되면, 조직은
“우리는 Claude Code를 더 신뢰하고 알려지지 않은 에이전트는 제한한다”는
정책을 코드 리뷰 대상으로 만들 수밖에 없다.
이는 파급 효과가 원래 목적(실수 방지)을 넘어, 에이전트 거버넌스를
암묵지에서 형식지로 끌어올리는 역할을 하게 된다는 점에서 이 도구의
2차 효과다.

### 방화벽의 역사가 반복된다: 시그니처 기반 방어의 익숙한 궤적

dcg의 구조, 즉 알려진 위험 패턴을 정규식과 AST로 매칭해 차단하고
새로운 패턴이 나타나면 팩을 추가하는 방식은 1990~2000년대 네트워크
침입 탐지 시스템(IDS)이나 안티바이러스 소프트웨어가 걸었던 길과
본질적으로 같다.
시그니처 기반 방어는 알려진 위협에는 빠르고 정확하지만, 알려지지 않은
변형이나 우회 기법에는 구조적으로 뒤처질 수밖에 없다.
그 결과 안티바이러스 업계는 휴리스틱, 행위 기반 탐지, 나중에는 머신러닝
기반 이상 탐지로 계속 방어선을 확장해야 했다.

dcg 역시 지금은 명령 문자열과 AST 패턴에 의존하지만, 에이전트가 생성하는
명령의 형태가 다양해질수록 “새로운 팩을 계속 추가하는” 유지보수 부담이
커질 가능성이 높다.
이는 결함이 아니라 시그니처 기반 방어 계열이 공통으로 짊어지는 역사적
숙명에 가깝고, dcg 커뮤니티가 이 패턴을 얼마나 빨리 인식하고 행위 기반
탐지나 사후 감사 같은 보완 계층을 준비하는지가 도구의 장기 생존력을
가를 것이다.

### 로컬 방화벽은 “사후 감사가 필요 없다”는 착각을 부를 수 있다

dcg를 설치하고 나면 개발자는 “이제 위험한 명령은 자동으로 막힌다”는
심리적 안도감을 얻기 쉽다.
그러나 앞서 살펴본 fail-open 철학, 크기·시간 제한을 벗어난 명령의
자동 허용, Codex CLI의 훅 우회 가능성을 종합하면, dcg는 완벽한 차단이
아니라 “가장 흔하고 예측 가능한 실수를 줄이는” 확률적 완화 장치에
가깝다.
이 도구가 있다고 해서 커밋을 자주 하거나, 백업을 유지하거나, 위험한
작업 이후 로그를 감사하는 습관이 불필요해지는 것은 아니다.

README 자체는 이 한계를 솔직하게 명시하고 있지만, 실제 도입 현장에서는
“dcg를 깔았으니 안전하다”는 식으로 도구의 보장 범위가 과장되어
받아들여질 위험이 있다.
로컬에서 실행되는 방화벽 성격의 도구 일반이 안고 있는 이 긴장,
즉 실제 보장 수준과 사용자가 체감하는 안도감 사이의 간극은 dcg만의
문제가 아니라 이런 종류의 안전장치를 도입할 때 항상 함께 짚어야 할
트레이드오프다.
