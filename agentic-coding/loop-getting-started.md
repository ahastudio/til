# 루프 시작하기 — 에이전트 루프를 네 가지 기준으로 분류하기

원문: <https://claude.com/blog/getting-started-with-loops>

## 요약

"루프를 설계하라"는 조언이 코딩 에이전트 커뮤니티에서 프롬프트를
대체하는 화두로 떠올랐다.
그런데 X에서 시간을 들여 "루프"가 정확히 무엇인지 찾아보면, 서로 다른
정의가 뒤섞여 있다는 걸 발견하게 된다.
Claude Code 팀은 이 혼란을 정리하기 위해 루프를 네 가지 기준으로
분류한다.
트리거 방식, 정지 조건, 사용하는 프리미티브, 적합한 태스크 유형이다.

Claude Code 팀은 루프를 "정지 조건을 만족할 때까지 에이전트가
작업 사이클을 반복하는 것"으로 정의한다.

## 분석

### 턴 기반 루프 (Turn-based loop)

사용자 프롬프트로 트리거되고, Claude가 태스크 완료를 판단하거나
더 많은 컨텍스트가 필요하다고 판단할 때 정지한다.
"보내는 프롬프트 하나하나가 매 턴을 사용자가 직접 지시하는
수동 루프를 시작하는 것"이라고 원문은 설명한다.
짧고 반복되지 않는 단발성 태스크에 가장 적합하다.
검증 단계를 스킬로 인코딩해 반복 턴 수를 줄이는 것이 핵심
전략이다.

### 목표 기반 루프 (/goal)

수동 실시간 프롬프트로 트리거되고, 목표를 달성하거나 최대 턴
수에 도달하면 정지한다.
검증 가능한 종료 기준이 있는 태스크에 가장 적합하다.
테스트 통과 개수나 점수 임계값처럼 결정론적 기준으로 완료 상태를
정의할 때 효과가 가장 크다.

### 시간 기반 루프 (/loop, /schedule)

지정된 시간 간격으로 트리거되고, 사용자가 취소하거나 작업이
완료될 때까지 반복한다.
반복 작업이나 외부 시스템 모니터링에 적합하다.
PR 리뷰 여부나 CI 실패를 주기적으로 확인하는 작업이 예시로
제시된다.

### 능동적 루프 (Proactive loop)

이벤트나 스케줄로 트리거되며 사람의 실시간 개입이 없다.
개별 태스크는 목표 달성 시 종료되지만, 루틴 자체는 비활성화될
때까지 계속 실행된다.
버그 분류(triage)나 마이그레이션, 의존성 업데이트처럼 잘
정의된 반복 워크플로에 이상적이다.
`/schedule`, `/goal`, 스킬, 동적 워크플로, 자동 모드를
조합해 구성된다.

## 비평

네 가지 분류는 "루프"라는 모호한 유행어를 트리거·정지 조건·프리미티브·
태스크 적합성이라는 구체적인 축으로 쪼갠다는 점에서 실용적이다.
다만 이 분류는 Claude Code 생태계의 도구(`/goal`, `/loop`, `/schedule`)에
맞춰져 있어, 다른 에이전트 하네스에도 그대로 적용되는 일반 이론이라고
보기는 어렵다.

## 인사이트

글은 루프의 종류를 나열하는 데서 그치지 않고, 코드 품질과 토큰 관리를
위한 운영 원칙도 함께 제시한다.
코드 품질 유지를 위해서는 기존 코드베이스의 패턴을 깨끗하게
유지해 Claude가 따를 수 있게 하고, 표준을 문서화한 스킬로
자기 검증이 가능하게 하며, 프레임워크 문서를 접근 가능하게
유지하고, 편향 없는 검토를 위해 2차 에이전트를 코드 리뷰에
활용하라고 권한다.

토큰 관리 측면에서는 프리미티브와 모델을 작업 복잡도에 맞게
고르고, 명확한 성공·정지 기준을 정의하며, 동적 워크플로는
작은 데이터셋으로 먼저 파일럿 테스트하고, 결정론적인 작업은
스크립트로 처리하고, 루틴 실행 간격을 실제 변경 빈도에 맞추고,
`/usage`·`/goal`·`/workflows` 명령으로 사용량을 주기적으로
점검하라고 조언한다.

시작하는 방법으로는, 자신의 워크플로에서 병목이 되는 작업을
먼저 식별하고, 어떤 검증 요소를 문서화할 수 있는지 판단하고,
목표가 명확한지 확인하고, 작업이 일정한 주기를 따르는지
점검한 뒤 관찰된 결과를 바탕으로 반복 개선하라고 제안한다.

결국 루프를 잘 설계한다는 것은 어떤 트리거·정지 조건을 쓸지 고르는
문제이면서, 동시에 그 루프가 무한정 토큰을 태우지 않도록 관측 가능성을
확보하는 문제이기도 하다.
