# 소프트웨어가 헤드리스로 가는가?

원문: <https://www.a16z.news/p/is-software-losing-its-head>

## 요약

a16z의 Seema Amble은 AI 에이전트 시대가 엔터프라이즈 소프트웨어의 경쟁 구조를
근본적으로 바꾸고 있다고 주장한다. 핵심 신호는 Salesforce의 헤드리스(headless)
제품 출시다. 이것은 “UI가 아니라 데이터 계층에서 가치가 창출된다”는 선언으로
읽힌다.

기존 SaaS의 방어선은 UI에서 나왔다. 사용자가 특정 인터페이스에 익숙해지면
전환 비용이 높아진다. 그런데 에이전트가 인간 인터페이스 없이 데이터베이스에
직접 접근하게 되면, 이 “근육 기억”(muscle memory) 기반의 방어선은 사라진다.
에이전트에게 UI는 그저 하나의 API 엔드포인트일 뿐이다.

저자는 헤드리스 전환에서 살아남는 것과 사라지는 것을 구분한다. 규정 준수
요건과 감사 추적, 미문서화된 업무 절차, 파편화된 시스템 간 연결 복잡성은
단기적으로 유지된다. 반면 UI 숙련도에 기반한 전환 비용, 단순 기록 시스템으로서의
가치는 침식된다.

새로운 경쟁력의 원천으로 저자는 세 가지를 제시한다. 첫째는 독점 데이터 생성
능력이다. 플랫폼을 사용할수록 경쟁자가 복제할 수 없는 데이터가 쌓이는 폐쇄적
피드백 루프를 갖춘 시스템이 강해진다. 둘째는 실제 실행 계층으로의 확장이다.
물류, 현장 운영, 사람 조율 같은 물리적 실행 능력이 데이터 저장보다 방어력이
높다. 셋째는 다자간 네트워크 효과다. 여러 당사자를 조율하는 플랫폼은 단일
사용자 인터페이스보다 에이전트 대체가 어렵다.

## 분석

### 헤드리스가 이미 일어나고 있는 증거

Salesforce의 헤드리스 제품 출시는 저자의 주장을 뒷받침하는 강력한 지표다.
세계 최대 CRM 기업이 자사 UI의 중심성을 스스로 약화시키는 방향으로 움직인다는
것은, 그들 스스로 에이전트 직접 접근이 불가피한 미래를 믿고 있다는 의미다.
이는 예측이 아니라 시장 선도 기업의 전략적 베팅이라는 점에서 무게가 다르다.

### 데이터 모델의 중요성 부상

에이전트 시대에 데이터 모델은 API 이상의 의미를 갖는다. 에이전트가 데이터를
이해하고 조작할 수 있으려면 데이터 구조 자체가 기계 가독성을 갖춰야 한다.
잘 설계된 데이터 모델과 권한 시스템은 에이전트가 작업할 수 있는 표면적을
결정한다. 이는 기술 부채가 많고 스키마가 복잡한 레거시 시스템들이 에이전트
시대에 심각한 경쟁 열위에 놓일 수 있음을 시사한다.

### 프론트엔드 측의 거울상

[컴포넌트가 페이지를 죽일 것이다](components-will-kill-pages.md)는 같은 흐름을
프론트엔드 관점에서 본 글이다. 백엔드가 헤드리스로 가는 동안 프론트엔드는
페이지 단위에서 AI 채팅 안의 컴포넌트 단위로 분해된다. 두 글을 합쳐 보면
SaaS의 UI 방어선이 양방향에서 동시에 무너지고 있다는 그림이 완성된다.

### 수직화의 가속

저자가 제시하는 “실제 실행 계층으로의 확장” 전략은 수직 SaaS의 논리를 에이전트
시대로 투영한 것이다. 특정 산업의 실제 운영 프로세스에 깊숙이 통합된 소프트웨어는
범용 에이전트가 대체하기 어렵다. 건설 현장 관리, 의료 기록, 물류 추적 같은
영역에서 소프트웨어가 데이터 저장소를 넘어 실제 실행의 일부가 될수록 방어선은
강화된다.

## 비평

### 강점: 신호와 노이즈 구분

저자가 Salesforce 사례를 중심으로 논지를 전개하는 방식은 효과적이다. 추상적인
“AI가 모든 것을 바꿀 것이다” 류의 주장 대신, 실제 대형 기업의 전략적 움직임을
근거로 삼는다. 이는 주장의 신뢰성을 높이고 독자가 반박 또는 검증할 수 있는
구체적 기준점을 제공한다.

### 약점: 전환 비용 과소평가

UI 기반 전환 비용이 사라진다는 주장은 지나치게 단순화될 수 있다. 에이전트가
UI를 거치지 않고 데이터에 접근할 수 있게 되더라도, 에이전트 자체의 훈련 데이터,
프롬프트, 통합 로직은 특정 플랫폼에 맞춰 최적화된다. 에이전트 수준에서의 새로운
전환 비용이 생겨날 수 있으며, 이것이 UI 기반 전환 비용을 완전히 대체하는지
아니면 보완하는지는 불명확하다.

### 약점: 중소기업 현실과의 괴리

저자의 분석은 Salesforce, SAP 같은 대형 기업 또는 잘 설계된 데이터 모델을
가진 신생 기업에 초점이 맞춰져 있다. 기술 부채가 쌓인 레거시 시스템으로 운영되는
중소기업 다수에게, 에이전트 친화적 아키텍처로의 전환은 현실적인 시간과 자원을
요구한다. 이들이 어떻게 이 전환을 해나갈 것인지에 대한 논의가 빠져 있다.

## 인사이트

### UI가 사라지면 무엇이 인터페이스가 되는가

소프트웨어 역사에서 인터페이스는 계속 추상화되어 왔다. 명령줄에서 GUI로, GUI에서
웹으로, 웹에서 모바일로. 각 전환에서 이전 인터페이스 전문성은 가치를 잃고
새로운 인터페이스 설계 능력이 부상했다. 에이전트 시대의 “인터페이스”는 프롬프트와
데이터 모델이 될 것이다. 에이전트가 어떤 프롬프트를 받을 때 어떻게 동작하는지,
그 결과가 데이터 모델에 어떻게 반영되는지가 새로운 UX 설계의 영역이 된다.
“헤드리스 UX”라는 개념이 등장할 것이며, 이를 설계하는 능력이 새로운 희소 자원이
될 수 있다.

### 레거시의 역설적 강점

저자가 지적한 “미문서화된 업무 절차”의 단기적 생존 가능성은 흥미로운 역설을 담고
있다. 복잡하고 비효율적인 레거시 시스템일수록, 그 복잡성을 이해하고 운영하는
인적 자본의 가치가 높다. 에이전트가 이 복잡성을 자동화하기 전까지, 레거시를
가장 잘 아는 기업이 역설적으로 가장 강한 방어선을 갖는다. 그러나 이것은 이점이
아니라 시한부 유예일 수 있다. 에이전트가 레거시를 학습하는 순간, 이 방어선도
무너진다.

### 헤드리스 시대의 “에이전트 팀원” 관리

UI 기반 전환 비용이 사라지면, 남는 것은 에이전트와의 API 협상이다.
[Claude Code vs Codex 비교](claude-code-vs-codex.md)가 “코딩 에이전트는 더 나은
도구가 아니라 다른 종류의 팀원”이라 결론지은 것처럼, 헤드리스 SaaS 시대의
사용자는 데이터를 직접 보지 않고 에이전트라는 매개자를 통해 시스템과 소통한다.
이때 어떤 에이전트가 어떤 SaaS와 잘 맞는지가 새로운 도구 선택 기준이 된다.

### 에이전트 조율 계층의 새로운 독과점 가능성

다자간 워크플로우를 조율하는 플랫폼이 에이전트 시대에 더 강해진다는 주장은,
새로운 독과점 구조가 형성될 수 있다는 경고이기도 하다. 복잡한 공급망, 의료
생태계, 금융 거래에서 여러 당사자를 연결하는 플랫폼이 에이전트 통신의 표준
허브가 된다면, 그 허브는 현재의 검색 엔진이나 운영체제 수준의 시장 권력을
갖게 될 수 있다. 헤드리스 전환이 기존 SaaS 독과점을 해소하는 것이 아니라,
더 깊은 인프라 수준의 새로운 독과점을 만들어낼 수 있다.
