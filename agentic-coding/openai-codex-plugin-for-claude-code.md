# OpenAI의 Claude Code용 Codex 플러그인

> 원문: <https://github.com/openai/codex-plugin-cc>

## 요약

OpenAI가 Claude Code 환경에서 Codex를 직접 호출할 수 있는 플러그인을 Apache-2.0
라이선스로 공개했다. Claude Code 내에서 슬래시 명령어로 Codex에 코드 리뷰를
요청하거나, 버그 수정 작업을 위임할 수 있다. OpenAI 직원이 직접 개발했으며,
GitHub 4.8k 스타를 기록했다.

## 핵심 기능

### 슬래시 명령어

| 명령어                     | 기능                                     |
| -------------------------- | ---------------------------------------- |
| `/codex:review`            | 읽기 전용 코드 리뷰                      |
| `/codex:adversarial-review`| 설계 결정을 비판적으로 도전하는 리뷰     |
| `/codex:rescue`            | Codex 서브에이전트에 작업 위임           |
| `/codex:status`            | 백그라운드 작업 진행 상황 확인           |
| `/codex:cancel`            | 활성 작업 중단                           |

### 기술 구성

- 로컬 Codex CLI와 앱 서버를 래핑
- 별도 런타임 불필요, 기존 Codex 바이너리 활용
- `.codex/config.toml`과 호환
- ChatGPT 구독(무료 포함) 또는 OpenAI API 키로 사용 가능

### 설치

```bash
/plugin marketplace add openai/codex-plugin-cc
/plugin install codex@openai-codex
/codex:setup
```

## 분석

이 플러그인의 존재 자체가 2026년 AI 도구 시장의 경쟁 역학을 압축적으로
보여준다. OpenAI가 경쟁사(Anthropic)의 제품 위에 자사 서비스를 올리는 행위는
전통적인 플랫폼 경쟁 논리로는 설명되지 않는다.

해석의 핵심은 **오케스트레이션과 실행의 분리**다. Claude Code가 에이전트
오케스트레이터로서 시장을 지배하고 있는 상황에서, OpenAI는 두 가지 선택지에
직면한다:

1. 자체 오케스트레이터를 밀어붙여 시장을 뺏는다.
2. 상대 오케스트레이터 위에 자사 실행 엔진을 올린다.

1번은 이미 Codex CLI로 시도했지만 Claude Code의 사용자 기반을 넘지 못했다.
2번은 "니 플랫폼에서 내 모델을 쓰게 하겠다"는 전략으로, 오케스트레이션 경쟁을
포기하는 대신 모델 사용량을 확보한다. 기묘해 보이지만 합리적이다.

## 비평

### "잘 만들었다"는 것과 "왜 만들었는가"는 다른 질문이다

Claude Code에 이 플러그인을 리뷰하도록 요청했더니 "really well-engineered
plugin"이라는 평가가 나왔다는 일화는 아이러니의 극치다. 경쟁사 제품이 자사
플랫폼용 플러그인을 칭찬하는 상황. 그러나 기술적 품질과 전략적 의도를 분리해서
봐야 한다.

플러그인이 잘 만들어졌다는 것은 OpenAI가 이것을 장난이나 실험이 아니라
진지한 유통 채널로 보고 있다는 뜻이다. Apache-2.0 라이선스, 4.8k 스타, 공식
계정 배포—이것은 마케팅 스턴트가 아니라 실질적인 제품 전략이다.

### 이중 과금의 함정

GeekNews 댓글에서 지적된 것처럼, Codex 호출 시 Claude 토큰도 소비된다.
사용자는 Claude 구독/API 비용 위에 OpenAI 구독/API 비용을 추가로 지불한다.
두 모델의 조합이 단일 모델보다 나은 결과를 내는지에 대한 정량적 근거가 없는
상태에서, 비용 부담만 확실히 늘어난다.

"두 모델이 협력하면 더 좋다"는 직관적 매력이 있지만, 실제로는 오케스트레이션
오버헤드, 컨텍스트 전환 비용, 출력 형식 불일치 등의 마찰이 발생한다. 이
마찰이 모델 조합의 이론적 이점을 상쇄할 가능성이 있다.

### adversarial-review의 가치

5개 명령어 중 가장 흥미로운 것은 `/codex:adversarial-review`다. Claude가 작성한
코드를 Codex가 비판적으로 검토하는 구조는 **서로 다른 모델 간의 적대적 검증**
패턴이다. 같은 모델이 자기 코드를 리뷰하면 동일한 편향을 공유하지만, 다른 모델이
리뷰하면 편향의 교차 검증이 가능하다.

이것이 실제로 버그를 더 잘 찾아내는지는 경험적 검증이 필요하지만, 원리적으로는
건전한 접근이다. 소프트웨어 리뷰에서 "다른 눈"의 가치는 널리 인정되어 있고,
모델의 "다른 눈"도 같은 논리가 적용될 수 있다.

## 인사이트

### 플랫폼 전쟁에서 프로토콜 전쟁으로

이 플러그인이 시사하는 가장 큰 변화는 **경쟁의 단위가 플랫폼에서 프로토콜로
이동하고 있다**는 것이다. 전통적인 플랫폼 전쟁에서는 사용자를 자사 생태계에
가두는 것이 전략이었다. Apple vs Google, Windows vs Mac. 그런데 OpenAI는
경쟁사 플랫폼 위에 자사 서비스를 올리고 있다. 이것은 플랫폼 전쟁의 논리가
아니다.

새로운 논리는 이렇다: AI 모델은 상품화되고 있으므로, 사용자를 가두는 것보다
**어디에서든 자사 모델이 호출되게 만드는 것**이 더 중요하다. 플러그인,
API, MCP—형태가 무엇이든 호출 경로를 최대화하는 것이 모델 사용량을
최대화하고, 모델 사용량이 수익을 결정한다.

이것은 인터넷 초기의 브라우저 전쟁이 HTTP 프로토콜의 승리로 끝난 것과
평행한다. 개별 브라우저(플랫폼)보다 HTTP(프로토콜)가 이겼다. AI 시장에서도
개별 에이전트 플랫폼보다 모델 호출 프로토콜(MCP, 플러그인 API)이 궁극적
승자가 될 수 있다.

### 경쟁사 플랫폼을 유통 채널로 쓰는 전략의 선례

게임 산업에서 이미 검증된 전략이다. Unity 엔진은 PlayStation, Xbox, Switch 등
경쟁하는 플랫폼 위에서 모두 동작한다. Unity의 가치는 특정 플랫폼에 묶이지
않는 것에서 온다. Spotify는 Apple의 iOS 위에서 Apple Music과 경쟁한다.

OpenAI의 Codex 플러그인은 같은 논리다. Claude Code라는 경쟁사 플랫폼을
Codex의 유통 채널로 활용한다. 이 전략이 작동하려면 두 가지 조건이 필요하다:
플랫폼이 충분히 개방적이어야 하고(Claude Code의 플러그인 시스템),
자사 제품이 플랫폼 없이도 독립적 가치를 가져야 한다(Codex의 코드 리뷰 능력).

### "단일 작업은 잘하고 오케스트레이션은 부족하다"는 자기 인식

GeekNews 댓글의 분석이 예리하다. OpenAI가 이 플러그인을 만든 것은 Codex가
개별 작업(코드 리뷰, 버그 수정)에서는 강하지만 전체 개발 워크플로를
오케스트레이션하는 데는 Claude Code에 뒤진다는 현실 인식의 반영이다.

이것은 더 넓은 AI 시장의 분화를 예고한다. 모든 것을 다 잘하는 단일 에이전트
대신, 특정 작업에 특화된 모델들이 오케스트레이터 아래에서 협업하는 구조.
마이크로서비스 아키텍처가 모놀리식을 대체한 것처럼, 멀티모델 에이전트 아키텍처가
단일 모델 에이전트를 대체할 수 있다.

이 플러그인은 그 미래의 가장 초기 형태다. 아직은 "Claude 위에 Codex를 호출하는"
단순한 구조지만, 궁극적으로는 작업 유형에 따라 최적의 모델이 자동 선택되는
지능적 라우팅으로 발전할 수 있다. 그 세계에서 경쟁 우위는 모델 자체가 아니라
**어떤 작업에 어떤 모델이 최적인지 아는 메타-지식**에서 온다.

### 열린 생태계의 승자는 누구인가

Anthropic이 Claude Code의 플러그인 시스템을 개방한 것은 이런 상황을 초래할
수밖에 없었다. 열린 생태계는 경쟁사에게도 열려 있다. 이것은 약점이 아니라
전략적 선택이다. 플러그인 생태계가 풍부해질수록 Claude Code의 가치가 올라간다.
심지어 경쟁사의 플러그인이 올라와도.

"경쟁사의 성공이 자사 플랫폼의 가치를 높인다"는 역설은 플랫폼 비즈니스의
고전적 특성이다. App Store에 경쟁 앱이 많을수록 iPhone이 매력적인 것과 같다.
Claude Code가 이 역학을 이해하고 플러그인 생태계를 개방했다면, OpenAI의
플러그인은 위협이 아니라 검증이다.
