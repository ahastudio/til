# SDD(Spec-kit) 에이전틱 코딩 실전기 — 카카오페이

원문: <https://tech.kakaopay.com/post/ifkakao-agentic-coding/>

## 요약

카카오페이 AI 플랫폼 팀이 AI 모델 배포 플랫폼을 구축하면서 경험한
에이전틱 코딩의 한계와, 이를 극복하기 위해 도입한 GitHub의
Spec-kit(SDD, Spec-Driven Development) 방법론 실전기다.
프로젝트 초반에는 에이전틱 코딩으로 개발 속도를 높였지만,
후반으로 갈수록 AI가 맥락을 잃고 여기저기 다른 스타일의 코드를 생성하는 문제가
심화됐다. 긴 대화에서 AI가 이전 패턴을 기억하지 못하는 현상이 반복됐고,
생성된 코드를 검토·수정하는 시간이 점점 늘어났다.

Spec-kit은 이 문제를 **스펙 주도 워크플로**로 해결한다.
워크플로는 constitution → specify ↔ clarify → plan → tasks → analyze → implement
의 단계로 구성된다. **Constitution**은 코드 스타일·테스트 커버리지·문서화 요구사항
같은 시스템 최상위 규칙을 정의한다. **Specify**는 개발할 기능의 구체적인 요구사항을
작성하는 단계로, 두서없이 작성해도 spec-kit이 정리해 준다. Figma MCP나 위키 MCP
같은 외부 도구를 이 단계에서 함께 호출할 수 있다.
**Clarify**는 선택 단계로, AI가 스펙을 분석해 최대 5개의 명확화 질문을 제안한다.
**Plan**과 **Tasks**는 constitution·spec을 바탕으로 개발 계획과 구체적 작업 목록을
자동 생성한다. **Analyze**는 constitution 기준으로 spec·plan·tasks의 모순을
점검하고 수정을 제안한다. 마지막으로 **Implement**가 모든 문서를 바탕으로
실제 코드를 생성한다.

실전에서 느낀 핵심 원칙은 “마음에 안 들어도 직접 수정하지 말고 specify로 돌아가라”다.
중간 결과물을 직접 수정하면 스펙 문서와 실제 코드 사이의 불일치가 생기고,
이후 단계에서 팀원들이 혼란을 겪는다. 다만 MCP 주소처럼 specify 수정만으로는
해결이 안 되는 경우도 있어, 이때는 tasks 파일을 직접 참조해 프롬프팅하는 방식을
병행했다. AI가 implement 시 constitution보다 tasks에 더 집중하는 경향이 있으므로,
constitution의 원칙이 tasks에 반영됐는지 반드시 확인해야 한다.

실전 도입 결과, 팀 단위 협업에서 일관성이 크게 향상됐다.
개발자가 스펙 정의에 집중하고 기획자와 함께 요구사항을 세심하게 검토할 수 있으며,
clarify 단계는 기획자가 미처 파악하지 못한 요구사항 구멍을 찾는 데도 유용했다.
단점으로는 단계가 많아 초기 학습 허들이 있고, 간단한 작업에는 오버엔지니어링처럼
느껴질 수 있으며, 파이프라인 전체를 거치면 문서가 많이 생성되어 토큰 소모가
크다는 점이 꼽혔다.

## 분석

### SDD가 해결하는 핵심 문제: 맥락 손실

이 글이 SDD 도입 배경으로 꼽는 가장 구체적인 문제는 “대화가 길어질수록 AI가
이전에 어떤 패턴으로 코드를 작성했는지 기억하지 못하는 현상”이다. 이는 LLM의
컨텍스트 윈도우 한계에서 비롯되는 구조적 문제다. Spec-kit은 이 문제를 대화가
아닌 파일 기반 문서(constitution.md, spec, plan, tasks.md)로 맥락을 외부화함으로써
해결한다. AI의 단기 기억이 아닌 파일 시스템이 프로젝트의 진실 원천(source of
truth)이 된다.

### 스펙-코드 일관성 강제의 구조적 설계

“직접 수정하지 말고 specify로 돌아가라”는 원칙은 단순한 권장사항이 아니라 시스템
설계다. 중간 산출물을 직접 수정하면 스펙과 구현의 불일치가 발생하고, 나중에 어떤
문서가 최신 정보인지 알 수 없게 된다. 이 원칙은 소프트웨어 설계의
단일 진실 원천(Single Source of Truth) 원칙을 개발 프로세스에 적용한 것이다.
스펙이 항상 구현의 상위에 있고, 구현은 스펙에서 파생된다는 의존성 방향을 강제한다.

### Constitution의 역할: 거버넌스의 코드화

Constitution은 단순한 코딩 스타일 가이드가 아니다. 보안 요구사항, 테스트 커버리지,
접근 제어 같은 엔터프라이즈 거버넌스 요소를 AI 워크플로에 주입하는 메커니즘이다.
대규모 조직에서 개별 개발자의 AI 사용 방식이 제각각이면 일관성 없는 결과물이
나오는데, constitution은 팀 전체에 적용되는 공통 규칙을 AI가 반드시 따르도록
강제한다. 엔터프라이즈 AI 거버넌스를 문서 레벨에서 구현한 접근이다.

### Clarify의 숨겨진 가치: 요구사항 완성도 검증

Clarify 단계가 “기획자와 함께 보기에도 유용하다”는 언급은 흥미롭다.
AI가 스펙을 분석해 명확화 질문을 제안하는 과정은, 사실 요구사항의 완성도를 검증하는
역할을 한다. 인간 리뷰어가 놓치기 쉬운 가정(assumption)이나 엣지 케이스를
AI가 체계적으로 짚어낸다. 이는 AI를 코드 생성기로만 보는 시각을 넘어,
요구사항 분석 파트너로 활용하는 관점이다.

## 비평

### 강점: 실패 경험에서 출발한 현장감

이 글의 강점은 “에이전틱 코딩을 잘 쓴 방법”이 아니라 “실패하면서 깨달은 것”에서
시작한다는 점이다. 프로젝트 후반에 검토 시간이 길어졌다는 솔직한 고백,
AI가 맥락을 잃는 구체적 현상 묘사는 같은 고민을 가진 독자에게 신뢰를 준다.
Clarify 단계가 기획자와 함께 하기 유용했다는 관찰처럼, 도구의 의도하지 않은
부가 가치를 포착한 것도 현장 관찰의 깊이를 보여준다.

### 약점: 정량적 성과 데이터 부재

“개발 기간을 크게 단축했다”고 언급하지만, 실제로 얼마나 단축됐는지에 대한
수치가 없다. Spec-kit 도입 전후의 버그 발생률, 코드 리뷰 시간, 릴리스 주기 같은
비교 데이터가 있었다면 더 설득력 있는 사례가 됐을 것이다. 특히 엔터프라이즈 환경
도입을 검토하는 독자에게는 ROI(투자 대비 효과) 근거가 중요하다.

### 약점: 단계 선택 기준의 모호함

“간단한 개발은 어떻게 해야 할지 애매하다”는 단점을 스스로 인정하면서도,
어떤 기준으로 Spec-kit 전체 파이프라인을 쓸지 vs. 직접 에이전트 모드를 쓸지
결정해야 하는지에 대한 가이드라인이 없다. 실무에서 이 판단 기준이 모호하면
팀원마다 다른 방식을 사용하게 되어 오히려 일관성이 깨진다.

### 약점: Constitution-Tasks 불일치 문제의 근본 원인 미탐구

“AI가 implement 시 constitution보다 tasks에 집중한다”는 관찰은 중요한 발견이지만,
왜 그런지에 대한 분석이 없다. 이는 프롬프트 구성상 tasks 파일이 더 가까운 컨텍스트에
위치하기 때문일 가능성이 높다. 이 문제를 해결하기 위해 constitution 내용을
tasks에 명시적으로 포함하는 방식을 시도해 봤는지, 아니면 단순히 수동 검증으로만
대응하고 있는지가 궁금하다.

## 인사이트

### 바이브 코딩의 실패 지점은 항상 “후반부”다

에이전틱 코딩이 초반에는 놀라운 생산성을 보여주다가 후반에 무너지는 패턴은
이 글만의 경험이 아니다. 거의 모든 에이전틱 코딩 실패 사례가 동일한 궤적을 밟는다.
초기에는 AI가 빠르게 구조를 잡고 보일러플레이트를 채워주는 덕분에 속도가 붙지만,
프로젝트가 복잡해질수록 AI가 기존 코드의 패턴을 잊어버리고 새로운 코드에 다른
스타일을 적용한다. 누적 오류가 쌓이면 수정 비용이 생성 비용을 초과한다.

이 패턴의 근본 원인은 AI가 맥락을 컨텍스트 윈도우에만 의존한다는 것이다.
대화가 길어지면 초기 합의 사항이 윈도우 밖으로 밀려나고, AI는 현재 보이는
코드만 기반으로 판단을 내린다. 이것이 Spec-kit이 외부 파일로 맥락을 영속화하는
이유다. 파일은 대화와 달리 잘리지 않는다. 프로젝트의 크기가 얼마나 커지든
constitution.md와 spec 문서는 항상 AI에게 제공될 수 있다.

역설적으로, 바이브 코딩은 가장 잘 작동하는 시점(초반부)에 가장 위험하다.
빠른 진행이 “이 방식이 맞다”는 착각을 심어주기 때문이다. 실패는 항상 나중에
온다. SDD는 이 착각을 예방하기 위해 초반부터 의도적 마찰을 만드는 설계다.
specify, clarify, plan, tasks의 단계를 거치는 오버헤드는 후반부의 붕괴를 막는
보험료다.

### AI 거버넌스는 프롬프트가 아니라 문서 아키텍처로 구현한다

많은 팀이 AI의 출력을 제어하기 위해 프롬프트를 다듬는 데 노력을 집중한다.
더 상세한 지시, 더 구체적인 예시, 더 많은 제약을 프롬프트에 담으려 한다.
그러나 프롬프트는 일회성이다. 매 요청마다 다시 써야 하고, 팀원마다 다르게 쓰며,
대화가 끊기면 사라진다. 엔터프라이즈 환경에서 프롬프트 거버넌스는 확장되지 않는다.

Spec-kit의 constitution이 보여주는 접근은 다르다. 거버넌스 규칙을 파일로 문서화하고,
모든 AI 작업이 그 파일을 컨텍스트로 받도록 워크플로를 설계한다.
개별 개발자가 매번 올바른 프롬프트를 쓰지 않아도, 파일이 규칙을 강제한다.
이것은 소프트웨어 설계에서 “정책을 코드로 표현하라(Policy as Code)”는 원칙의
AI 버전이다.

이 패턴이 확장되면, 조직의 AI 사용 거버넌스는 constitution 파일들의 레지스트리가 된다.
보안팀이 보안 요구사항 constitution을 관리하고, 아키텍처팀이 기술 선택 constitution을
관리하며, 컴플라이언스팀이 규제 요구사항 constitution을 관리하는 방식이다.
AI는 코드를 생성하지만, 그 코드가 따라야 할 규칙은 조직이 문서로 선제적으로 정의한다.

### 개발자 역할의 변화는 '덜 코딩하는 것'이 아니라 '다르게 생각하는 것'이다

글의 마지막이 “개발자의 역할이 '코드를 작성하는 사람'에서 '스펙을 검증하고 테스트를
설계하는 사람'으로 변화할 것”이라는 전망으로 끝나는 것은 의미심장하다.
이 변화는 이미 Spec-kit 워크플로에서 확인된다. 개발자가 가장 많은 시간을 쓰는
지점이 코드 작성에서 specify·clarify·analyze 단계로 이동했다.

그러나 이 전환은 단순히 “코딩을 덜 한다”는 의미가 아니다. 오히려 더 높은 수준의
추상적 사고가 요구된다. 무엇을 만들지(what)를 정확히 기술하는 능력,
요구사항의 모순을 발견하는 능력, AI가 생성한 스펙이 실제 의도와 일치하는지
검증하는 능력이 핵심 역량이 된다. 이는 전통적으로 시니어 개발자나 아키텍트에게
요구되던 역량이다.

역설적으로 SDD는 주니어 개발자에게 더 높은 허들을 만들 수 있다.
AI가 코드를 대신 써주는 환경에서 스펙을 정확히 쓰는 능력이 핵심이 되면,
코드를 직접 작성하면서 배우는 전통적 학습 경로가 약화된다.
constitution을 어떻게 쓰는지, specify에서 무엇을 놓치면 안 되는지는
도메인과 기술에 대한 깊은 이해가 선행되어야 한다. AI가 쓰는 것을 검증하려면
먼저 무엇이 옳은지 알아야 하기 때문이다.
