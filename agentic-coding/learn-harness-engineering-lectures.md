# Learn Harness Engineering 강의 전편

<https://walkinglabs.github.io/learn-harness-engineering/ko/>

하네스 엔지니어링을 체계적으로 가르치는 12강 학습 사이트.
“모델은 똑똑하다. 하네스가 모델을 신뢰할 수 있게 만든다”는 핵심 명제를 각 강의가 하나씩 풀어나간다.

## 강의 구성

### 강의 1: 뛰어난 에이전트도 왜 여전히 실패하는가

핵심 주장은 단순하다. 모델의 역량과 실제 작업 성공률은 별개다.
SWE-bench Verified에서 최고 성능 모델들이 50~60%의 정확도를 보이는 반면, 동일한 모델도 하네스 없이는 무너진다.
Anthropic의 통제 실험이 이를 극적으로 보여준다. 동일한 모델, 동일한 프롬프트로 하네스 없이 실행하면 20분 만에 실패하고, 완전한 하네스를 갖추면 6시간 만에 플레이 가능한 게임이 나온다.

에이전트가 막히는 다섯 지점을 정리한다.

**작업 명세 부족**: “검색 기능 추가”처럼 불명확한 요구사항은 에이전트를 추측에 의존하게 만든다.

**암묵적 컨벤션 미인지**: SQLAlchemy 버전, OAuth 2.0 표준, 코딩 패턴 같은 프로젝트 특정 규칙이 공유되지 않으면 에이전트는 따를 수 없다.

**부실한 실행 환경**: 누락된 의존성, 잘못된 도구 버전은 에이전트가 실제 작업 대신 환경 설정에 컨텍스트를 낭비하게 한다.

**검증 방법 부재**: 테스트나 린트 없이 에이전트는 자체 판단으로 “완료”를 선언한다. 컨텍스트 부족을 감지하면 서둘러 마무리하는 “컨텍스트 불안(Context Anxiety)” 현상도 관찰된다.

**장시간 작업의 상태 손실**: 세션이 끝나면 이전 발견이 사라진다. 30분을 초과하는 작업에서 실패율이 급격히 증가한다.

실패 진단은 “모델이 부족해”가 아니라 다섯 레이어 중 어디인지 정확히 짚는 것에서 시작한다.
`AGENTS.md` 파일 하나가 더 비싼 모델로의 업그레이드보다 효과적일 수 있다.

### 강의 2: 하네스란 실제로 무엇인가

하네스는 “모델 가중치 외부의 모든 엔지니어링 인프라”다.
재료만 있고 가스레인지도, 칼도, 레시피도 없이 레스토랑을 여는 것에 비유한다.
에이전트가 볼 수 없는 것은 실질적으로 존재하지 않는다.

다섯 하위 시스템으로 구성된다.

**지시 하위 시스템**: `AGENTS.md` 또는 `CLAUDE.md`. 프로젝트 개요, 기술 스택 버전, 첫 실행 명령어, 협상 불가 제약을 포함하는 100줄 안팎의 디렉터리 같은 가이드다.

**도구 하위 시스템**: 에이전트가 셸 명령, `pip install` 등에 충분히 접근할 수 있어야 한다. 보안을 과도하게 우려해 기능을 제한하면 안 된다.

**환경 하위 시스템**: 환경 상태가 자기 서술적이어야 한다. `pyproject.toml`, `.nvmrc`, Docker, devcontainers를 활용해 재현성을 확보한다.

**상태 하위 시스템**: 장시간 작업을 위한 진행 추적이 필수다. `PROGRESS.md`로 완료/진행중/차단 사항을 기록하고 매 세션 종료 전 업데이트한다.

**피드백 하위 시스템**: 저자가 “투자 대비 효과가 가장 높은” 요소로 꼽는다. `pytest tests/ -x`, `mypy src/ --strict`, `ruff check src/` 같은 검증 명령이 에이전트에게 기계적 피드백을 제공한다.

TypeScript + React 프로젝트(20,000줄)에서 GPT-4o를 사용한 실험에서, 빈 주방 상태의 성공률 20%가 `AGENTS.md` 추가만으로 60%로, 검증 명령 추가로 80%로, 진행 추적 파일 추가로 80~100%로 상승했다.
모델은 변경하지 않았다.

### 강의 3: 저장소가 단일 진실 원천이 되어야 하는 이유

“저장소에 없는 정보는 에이전트에게 존재하지 않습니다.”
AI 에이전트는 시스템 프롬프트, 작업 설명, 저장소 파일 내용, 도구 실행 결과만 접근할 수 있다.
Slack 기록, Confluence 문서, 팀원의 암묵적 지식은 모두 에이전트의 범위 밖이다.

다섯 가지 개념을 정의한다.

**지식 가시성 격차(Knowledge Visibility Gap)**: 프로젝트 지식 중 저장소 외부에 남아 있는 비율. 이 격차가 크면 에이전트 실패율이 높아진다.

**콜드 스타트 테스트(Cold-Start Test)**: 새로운 에이전트 세션이 저장소 내용만으로 다섯 질문에 답할 수 있는지 검증한다. “이 프로젝트가 무엇인가”, “어떻게 구성되어 있는가”, “어떻게 실행하는가”, “어떻게 검증하는가”, “현재 진행 상황은”. 답하지 못하는 질문이 있으면 지도에 빈 곳이 있다는 의미다.

**발견 비용(Discovery Cost)**: 에이전트가 중요 정보를 찾는 데 소비하는 컨텍스트 예산. 정보가 여러 디렉터리 깊숙이 숨어 있으면 실제 작업에 쓸 예산이 줄어든다.

**지식 부패율(Knowledge Decay Rate)**: 시간이 지나며 낡아지는 지식의 비율. 코드는 변했지만 문서는 업데이트되지 않은 경우, 없는 문서보다 더 해롭다. 에이전트가 잘못된 정보를 믿고 잘못된 방향으로 나아가기 때문이다.

**ACID 유추**: 데이터베이스 원칙을 에이전트 상태 관리에 적용한다. 각 작업이 하나의 커밋으로 이루어지고(원자성), 모든 테스트 통과가 보장되며(일관성), 여러 에이전트의 상태 파일이 경쟁 조건을 피하고(격리성), 중요 지식이 git으로 추적된다(내구성).

네 가지 원칙도 제시한다. 지식은 코드 옆에 있어야 한다. 표준화된 진입 파일을 사용한다. 최소하되 완전하게 갖춘다. 코드와 함께 업데이트한다.
“지식 업데이트를 코드 변경에 바인딩하면 동기화 문제를 줄일 수 있다.”

### 강의 4: 거대한 명령 파일 하나가 왜 실패하는가

컨텍스트 윈도의 10~15%를 초과하는 거대한 단일 명령 파일은 성능 저하를 유발한다.
600줄짜리 파일은 10,000~20,000 토큰을 소비해 실제 작업에 쓸 예산을 잠식한다.

Liu et al.(2023)의 “중간 손실(Lost in the Middle)” 연구가 근거로 제시된다.
LLM은 긴 텍스트의 중간에 있는 정보를 처음이나 끝에 비해 현저히 덜 활용한다.
300번째 줄에 묻힌 보안 제약은 무시될 가능성이 높다.

파일이 비대화되는 과정은 악순환이다.
에이전트 실수 → 규칙 추가 → 일시적 효과 → 새 실수 → 규칙 추가.
기술 부채 축적과 동일한 패턴이다.

권장 아키텍처는 라우팅 파일 기반 분산 구조다.

`AGENTS.md`는 50~200줄로 유지하며 라우터 역할만 담당한다. 프로젝트 개요, 빠른 시작 명령, 전역 하드 제약(최대 15개), 주제 문서 링크와 한 줄 설명만 담는다.

주제 문서는 각 50~150줄로 관심사에 따라 분리한다. `docs/api-patterns.md`, `docs/database-rules.md`, `docs/testing-standards.md` 같은 형태다.

실제 SaaS 팀의 사례에서 600줄 단일 파일을 80줄 라우팅 파일 + 4개 주제 문서로 리팩터링한 후, 작업 성공률이 45%에서 72%로, 보안 제약 준수율이 60%에서 95%로 상승했다.

“진입 파일의 역할은 백과사전이 아닌 라우터여야 한다.”
점진적 공개(progressive disclosure) 원칙으로 필요한 정보만 로드하면, 에이전트의 컨텍스트 예산이 실제 작업에 집중된다.

### 강의 5: 장시간 작업이 왜 연속성을 잃는가

에이전트는 세션이 끝나면 이전 작업의 맥락을 잃는다.
“매일 아침 모든 것을 잊어버리는 장인”에 비유한다.

컨텍스트 윈도 크기 증가(128K, 200K, 1M 토큰)는 근본적 해결책이 아니다.
에이전트는 단순 코드 생성을 넘어 이해, 추론, 도구 활용, 대화 관리를 수행하며 정보량 증가 속도가 빠르다.

Anthropic 연구는 “컨텍스트 불안(Context Anxiety)”을 설명한다.
컨텍스트 고갈을 감지한 에이전트는 검증 단계를 건너뛰고, 최적 해결책 대신 간단한 해결책을 선택하며, 시간 부족 불안으로 작업을 서두른다.
시험에서 시간 부족을 느껴 임의로 답변을 채워넣는 것과 유사하다.

세션 연속성 손실의 네 가지 양상도 정리한다.

**드리프트(Drift)**: 에이전트 이해와 저장소 실제 상태의 괴리. 세션 경계마다 발생하고 누적된다.

**중복 작업**: 이전 작업 완료 여부를 확인할 수 없어 재구현하거나, 절반만 수행 후 기존 구현과 충돌한다.

**요구사항 편탈**: 여러 세션에 걸쳐 원래 목표에서 조용히 벗어나는 전화 게임 효과.

**검증 격차**: 이전 테스트 결과를 모르므로 매 세션마다 모든 검증을 재실행한다.

해결책은 구조화된 상태 영속화다.

`PROGRESS.md`는 최신 커밋, 테스트 통과율, 완료 체크리스트, 진행 중 작업, 알려진 이슈, 다음 단계를 담는 “일지의 핵심”이다.

`DECISIONS.md`는 설계 결정의 논거를 기록한다. 어떤 결정인가, 왜 그 결정인가, 거부된 대안, 제약 조건. 다음 세션이 의도를 이해하게 돕는다.

Git 커밋을 체크포인트로 활용하고, 출근 루틴(PROGRESS.md 읽기 → DECISIONS.md 읽기 → 저장소 일관성 확인 → “다음 단계” 재개)과 퇴근 루틴(PROGRESS.md 업데이트 → 일관성 검증 → 완료 항목 커밋)을 정의한다.

흥미로운 발견이 있다. Sonnet 4.5에서는 컨텍스트 불안이 심각하지만 Opus 4.5에서는 현저히 감소한다.
“하네스 설계는 대상 모델에 대한 구체적 이해가 필요하며, 일률적 템플릿이 아니다.”

### 강의 6: 초기화가 왜 독립된 단계를 필요로 하는가

초기화와 구현을 반드시 분리해야 한다.
기초 공사와 건물 짓기를 동시에 하면 둘 다 망가진다.

두 단계는 근본적으로 다른 최적화 목표를 가진다.
구현은 완성된 기능의 수량과 품질을 극대화한다.
초기화는 이후 모든 작업의 신뢰성과 효율성을 극대화한다.
혼합하면 에이전트가 다중 목표 최적화에 직면하여 보이는 결과(기능 코드)를 선호하고 인프라(장기 가치)는 무시하게 된다.

새 세션이 성공하려면 “부트스트랩 계약(Bootstrap Contract)”의 네 조건이 필요하다.
프로젝트 실행 가능(시작 가능), 검증 프레임워크 작동(테스트 가능), 상태 문서화(진행 추적 가능), 다음 단계 파악(방향 명확).

Anthropic 연구 결과는 명확하다. 전용 초기화를 사용한 프로젝트는 멀티 세션 환경에서 기능 완료율이 31% 향상되었다. 초기화에 투자한 시간은 이후 3~4 세션에서 완전히 회수된다.

세션 1은 비즈니스 기능 코드 없이 초기화만 수행한다. 실행 가능한 환경, 검증 가능한 테스트(최소 1개 통과), 부트스트랩 계약 문서, 작업 분해(순서 있는 목록 + 명확한 인수 기준), Git 체크포인트를 산출물로 남긴다.

완료 기준은 코드 양이 아닌 부트스트랩 계약의 네 조건 충족 여부로 판단한다.

“느린 것이 빠른 것입니다.”

### 강의 7: 에이전트가 왜 과욕을 부리고 미완성으로 끝나는가

에이전트는 “한 가지를 완료한 후 다음을 시작”하지 않고, 관련된 모든 것을 동시에 처리하려는 경향을 보인다.
사용자 인증 추가 요청이 데이터베이스 스키마 수정, 라우트 작성, 프론트엔드 변경, 오류 처리 리팩터링으로 확산되며, 결국 어느 기능도 완성되지 않는다.

수학적으로 설명한다.
에이전트의 컨텍스트 용량 C에서 k개 작업을 동시 활성화하면, 각 작업은 C/k 수준의 추론 자원만 할당받는다.
단일 작업 완료에 필요한 임계값 이하로 떨어지면 모든 작업이 미완성 상태로 남는다.

Anthropic 실험 데이터도 제시한다. “WIP=1 전략 사용 에이전트는 넓은 프롬프트 에이전트보다 37% 높은 완료율을 기록했다.”

두 문제는 공생 관계다.

```text
과욕 → 주의력 희석 → 미완성 → 복잡도 증가 → 더 큰 과욕
```

칸반의 리틀의 법칙(L = λ × W)을 적용하면, WIP가 높을수록 각 작업의 리드 타임이 증가한다.

해결책은 네 가지다.

**WIP=1 강제 적용**: 어느 시점에도 하나의 작업만 `active` 상태. 한 기능이 엔드투엔드 검증을 통과한 후에만 다음 기능을 시작한다.

**완료 증거 명시**: “코드가 작성됨”이 아닌 `curl -X POST /api/register... | jq .status == 201` 같은 실행 가능한 검증 명령으로 완료를 정의한다.

**범위 표면 외부화**: JSON/Markdown 파일로 작업 상태를 기록한다. 활성 작업, 검증 요건, 의존성 관계, 현재 검증 상태를 포함한다.

**검증 완료율(VCR) 모니터링**: `VCR = 검증된 작업 / 활성화된 작업`이 1.0 미만이면 새 작업 활성화를 차단한다.

비교 실험이 역설을 드러낸다. 뷔페 모드(제약 없음)에서 3세션에 8개 기능 중 3개만 완성됐다. 단일 접시 모드(WIP=1)에서 4세션에 8개 기능 중 7개가 완성됐다. 생성 코드량은 전자가 800줄, 후자가 1,200줄로 오히려 많았는데도 결과가 반대였다.

“적게 끝내기가 많이 반만 남기기보다 낫다.”

### 강의 8: 기능 목록이 왜 하네스의 기본 구성요소인가

에이전트가 “완료”의 의미를 이해하지 못해 자신만의 암묵적 기준으로 작업을 끝낸다는 문제에서 출발한다.

기능 목록은 인간을 위한 메모가 아니라 **하네스 시스템의 기본 구성요소(primitive)**다.
스케줄러, 검증기, 핸드오프 보고기 등 모든 하네스 구성 요소가 의존하는 기반 데이터 구조다.

각 기능 항목은 삼중 구조를 가져야 한다.
동작 설명(기능이 무엇을 하는가), 검증 명령(완료를 확인하는 구체적 테스트), 현재 상태(진행 상황 추적). 세 요소 중 하나라도 빠지면 불완전하다.

기능은 네 가지 상태를 순환하는 상태 기계 모델을 따른다.
`not_started` → `active` → `passing`. `blocked`는 진행 불가 상태.
에이전트가 상태를 임의로 변경할 수 없고, 오직 검증 명령 성공을 통해서만 `passing`으로 전환된다.
“한 번 `passing`이 되면 되돌아갈 수 없습니다. 시험에 합격하면 합격한 것.”

기능 목록은 하네스에서 네 가지 역할을 한다.

1. **스케줄러**: 다음 `not_started` 기능 선택
2. **검증기**: 검증 명령 실행 및 상태 전환 승인
3. **핸드오프 보고기**: 기능 목록으로부터 자동 세션 인계 요약 생성
4. **진행 추적기**: 상태 분포 집계 및 프로젝트 건강 지표 제공

세분화 수준도 중요하다. “사용자가 장바구니에 항목을 추가할 수 있다”는 적절하다. “장바구니 구현”은 너무 넓다. “Cart 모델에 name 필드 생성”은 너무 좁다. 한 세션에서 완료 가능한 크기가 기준이다.

구조화된 기능 목록 사용 프로젝트는 완료율이 45% 더 높고 중복 구현이 없다는 측정 결과가 제시된다.

### 강의 9: 에이전트가 왜 너무 일찍 완료를 선언하는가

에이전트는 작업 완료 판단에서 체계적으로 과도한 자신감을 보인다.
“코드가 작성됨”을 “기능이 완료됨”과 혼동하며, 단위 테스트 통과만으로 성공을 선언한다.
2017년 Guo et al.의 ICML 연구가 증명했듯, 신경망은 보고하는 신뢰도가 실제 정확도보다 유의미하게 높다.

세 가지 검증 레이어를 순차적으로 통과해야 완료다.
문법·정적 분석(최소 확인 기준), 런타임 동작(실제 테스트 실행, 앱 시작 확인), 시스템 수준(엔드투엔드 테스트, 사용자 시나리오).
모든 레이어가 통과해야만 완료로 간주된다.

비밀번호 재설정 기능 구현 사례를 제시한다.
에이전트가 스키마 수정, API 작성, 이메일 템플릿 추가, 단위 테스트 통과 후 완료를 선언했다.
실제 문제는 이메일 서비스 설정 누락, 데이터베이스 마이그레이션 실패, 엔드투엔드 흐름 미검증이었다.

에이전트에게 자신의 작업을 평가하도록 요청하면 체계적으로 지나치게 긍정적이다.
Anthropic 연구에서 단일 에이전트(bare run) 20분 $9로 실패했고, 세 에이전트(평가기 포함) 6시간 $200으로 성공했다.
해결책은 “더 객관적인” 에이전트가 아니라 생성자와 검사자의 분리다.

“학생이 자신의 시험을 채점할 수 없듯, 에이전트도 자신의 작업 완료를 판단해서는 안 됩니다.”

### 강의 10: 엔드투엔드 테스트가 왜 결과를 바꾸는가

단위 테스트를 통과해도 컴포넌트 간 상호작용에서 실패할 수 있다.
“각 성부가 개별 연습할 때는 완벽해도 함께 노래하면 엇나간다.”

단위 테스트는 격리 원칙으로 설계되어 네 가지 주요 문제를 놓친다.

**인터페이스 불일치**: 컴포넌트 간 데이터 형식 오류(상대 경로 vs 절대 경로).

**상태 전파 오류**: 캐싱 계층과 데이터베이스 간 동기화 문제.

**리소스 수명 주기**: 파일 핸들, 연결, 소켓의 획득/해제 실패.

**환경 의존성**: 테스트 환경 vs 실제 환경의 불일치.

E2E 테스트가 에이전트의 코딩 행동 자체를 변화시킨다는 주장이 흥미롭다.
검증 결과뿐 아니라 컴포넌트 상호작용을 설계 단계부터 고려하고, 아키텍처 경계 규칙을 준수하며, 예외 처리 및 오류 경로를 충분히 구현하게 된다.

Electron 파일 내보내기 기능 구현 사례에서 파일 경로 형식 불일치, 진행도 상태 전파 불완전, 대용량 파일 메모리 누수, 패키징 환경 권한 문제, 예외 전파 경로 오류 다섯 가지 결함 모두 E2E 테스트로만 탐지되었다.

아키텍처 경계 규칙을 자동화하는 방법도 제시한다.

```bash
# 렌더 프로세스의 직접 파일시스템 접근 검사
grep -r "require('fs')" src/renderer/ && exit 1
```

에이전트 지향 오류 메시지는 “무엇이 잘못됐는지”를 넘어 “왜인지”와 “어떻게 수정하는지”를 포함해야 한다.

```text
ERROR: Direct 'fs' import in src/renderer/App.tsx:12
WHY: Renderer has no Node.js API access for security
FIX: Move to src/preload/file-ops.ts, call via window.api.readFile()
```

피드백 승격 프로세스도 핵심이다. 새로운 에이전트 오류 패턴이 발견될 때마다 자동화된 검사로 전환하면, 시간이 지날수록 하네스가 점진적으로 강화된다.

### 강의 11: 관측 가능성이 왜 하네스 안에 속하는가

에이전트가 작업을 완료했다고 보고하면서도 “왜 실패했는지 모르겠습니다”라고 말하는 상황.
근본 원인은 능력 부족이 아니라 관측 가능성 부재다.

두 가지 관측 가능성 구조를 정의한다.

**런타임 관측 가능성**: 로그, 트레이스, 프로세스 이벤트, 상태 확인 등 시스템 수준 신호. “무엇이 일어났는가”를 기록한다.

**프로세스 관측 가능성**: 계획, 채점 기준, 인수 조건 등 의사결정 산출물. “왜 이 변경이 수용되어야 하는가”를 설명한다.

스프린트 계약(Sprint Contract)은 코딩 시작 전 협상되는 구조화된 약속이다.
수정할 컴포넌트 목록, 각 항목별 검증 기준, 명시적으로 제외된 사항을 포함한다.
생성자와 평가자 간 기대 불일치를 방지한다.

평가자 루브릭은 평가를 주관적 판단에서 구조화된 채점으로 전환한다.
코드 정확성, 아키텍처 준수, 테스트 커버리지 등 차원별로 명확한 등급(A/B/C/D)을 제시해 재현 가능한 평가를 가능하게 한다.

관측 가능성 부재의 실제 비용은 크다.
Anthropic 관찰에 따르면 관측 가능성이 없으면 총 세션 시간의 30~50%를 중복 진단에 소비할 수 있다.

다크 모드 구현 비교 실험이 제시된다.
관측 불가능한 버전에서는 3~4번의 맹목적 재시도, 45분 소요, “느낌이 좋지 않습니다” 같은 모호한 평가가 나왔다.
완전히 관측 가능한 버전에서는 한 번의 반복으로 고품질 결과, 15분 소요였다.
3배 효율성 차이다.

에이전트 신뢰성은 더 나은 모델이 아니라 더 나은 가시성에서 비롯된다.

### 강의 12: 모든 세션이 왜 클린 상태로 끝나야 하는가

에이전트 세션이 깨진 빌드, 실패한 테스트, 임시 파일, 미기록 진행사항을 남기면 다음 세션은 첫 30분을 기존 혼란을 파악하는 데 소비한다.

클린 상태는 단순한 컴파일 성공이 아니라 다섯 차원으로 정의된다.

**빌드 차원**: 오류 없이 빌드되는가.

**테스트 차원**: 모든 테스트(기존 테스트 포함)가 통과하는가.

**진행 상황 차원**: 현재 상태가 기계가 읽을 수 있게 기록되어 있는가.

**산출물 차원**: 오래되거나 모호한 임시 파일이 없는가.

**시작 차원**: 다음 세션이 표준 경로로 시작할 수 있는가.

하나라도 실패하면 세션이 완료된 것이 아니다.

Lehman의 소프트웨어 진화 법칙을 인용하며 능동적 정리 없이는 복잡성이 기하급수적으로 증가함을 설명한다.
실측 데이터에서 12주 후 정리 전략 없는 프로젝트의 빌드 성공률이 68%, 정리 전략 있는 프로젝트가 97%였다.

이중 모드 정리 전략을 제안한다.
즉각적 정리(모든 세션 종료 시)는 임시 산출물 제거, 기능 목록 업데이트, 빌드/테스트 검증이다.
주기적 정리(주간)는 전체 시스템 스캔, 누적된 구조적 문제 해결, 품질 문서 갱신이다.

하네스 단순화도 강조한다. 모델 역량이 향상됨에 따라 기존 하네스 구성 요소는 구식이 된다.
월별로 하나의 구성 요소를 비활성화하고 벤치마크하여 필요 없으면 제거하거나 더 가벼운 대안으로 교체한다.

“클린 상태는 선택 사항이 아니라 완료 정의의 핵심 부분이다.”

## 분석

### 하네스 엔지니어링의 다섯 서브시스템 전체 조감

이 강의 시리즈의 핵심 기여는 하네스를 단일 개념이 아니라 다섯 서브시스템(지시, 도구, 환경, 상태, 피드백)의 상호 의존적 구조로 명확히 분해한 것이다.
각 강의가 하나의 실패 패턴을 선택하고 그 실패가 어느 서브시스템의 결함에서 비롯되는지를 추적하는 구조가 일관적이다.
거대한 명령 파일 실패는 지시 서브시스템 문제이고, 세션 연속성 손실은 상태 서브시스템 문제이며, 조기 완료 선언은 피드백 서브시스템 부재의 결과다.

이 분류 체계는 실무적으로 유용하다.
“에이전트가 실패했다”는 진단 대신 “어느 서브시스템이 부재했는가”를 묻게 되면 수리 방향이 명확해진다.
단순히 더 긴 프롬프트를 쓰거나 더 비싼 모델을 선택하는 대신, 구체적인 파일 하나를 추가하거나 검증 명령 하나를 정의하는 작업으로 좁혀진다.

### “완료”의 정의가 시스템 설계의 핵심이다

이 시리즈에서 가장 반복적으로 등장하는 주제는 “완료란 무엇인가”다.
강의 7은 WIP=1을 제안하고, 강의 8은 기능 목록의 상태 기계 모델을, 강의 9는 세 가지 검증 레이어를, 강의 10은 E2E 테스트를, 강의 12는 클린 상태의 다섯 차원을 제시한다.
이 모든 개념이 하나의 문제, 즉 에이전트가 “완료”를 조기에 잘못 선언한다는 문제의 다른 각도에서의 해법이다.

소프트웨어 공학의 완료의 정의(Definition of Done)는 애자일 방법론의 핵심 개념이다.
이 강의 시리즈는 그 개념을 에이전트 컨텍스트에서 재발명한다.
에이전트가 자신의 시험을 채점할 수 없다는 원칙은 강력하다.
검증은 생성과 분리되어야 하고, 완료의 기준은 에이전트 바깥에서 기계적으로 정의되어야 한다.

## 비평

### 강점: 실증적 데이터와 구체적 처방

이 시리즈의 가장 큰 강점은 각 주장이 구체적인 수치와 실험 결과로 뒷받침된다는 것이다.
“WIP=1 전략 사용 에이전트는 37% 높은 완료율” 같은 수치, Anthropic의 통제 실험 데이터, 팀 성능 개선 비교 등이 설득력을 높인다.
추상적 원칙이 아니라 “진입 파일은 50~200줄로 제한하라”, “기능 목록에 세 요소를 포함하라” 같은 구체적이고 즉시 실행 가능한 처방으로 이어진다.

### 약점: 비용과 복잡성의 트레이드오프 논의 부재

12개 강의 전체를 관통하는 약점은 “언제 하네스를 쓰는가”보다 “어떻게 쓰는가”에만 집중한다는 것이다.
완전한 하네스(AGENTS.md + PROGRESS.md + DECISIONS.md + feature_list.json + init.sh + E2E 테스트 + 스프린트 계약 + 평가자 루브릭)를 모든 프로젝트에 적용하는 것은 오버엔지니어링이다.
간단한 일회성 스크립트에 이 모든 것을 갖추는 비용이 실제 작업 비용보다 클 수 있다.
규모와 반복 빈도에 따른 하네스 선택 기준이 명시되지 않는다.

## 인사이트

### 컨텍스트 예산 관리는 에이전트 시대의 핵심 엔지니어링 과제다

컨텍스트 윈도는 유한한 자원이다.
에이전트가 실패하는 많은 경우가 능력 부족이 아니라 컨텍스트 예산을 잘못 배분한 결과다.
거대한 명령 파일이 예산을 잠식하고, 암묵적 컨텍스트를 재발견하는 데 예산을 낭비하고, 컨텍스트 불안이 발생하면 검증을 건너뛴다.

이 강의 시리즈가 제안하는 거의 모든 구조는 컨텍스트 예산을 실제 작업에 집중하기 위한 설계다.
라우팅 파일 구조는 필요한 정보만 로드하는 점진적 공개다.
`PROGRESS.md`와 `DECISIONS.md`는 세션 재시작 비용을 최소화하는 컨텍스트 외부화다.
기능 목록의 상태 기계는 에이전트가 현재 어디에 있고 다음에 무엇을 해야 하는지 찾는 데 쓰는 시간을 줄인다.

이 관점에서 하네스 엔지니어링은 본질적으로 컨텍스트 예산 관리 엔지니어링이다.
에이전트가 중요한 일에 집중하도록 환경을 설계하는 것이 모델 성능 향상만큼, 어쩌면 그보다 더 중요하다.
100줄의 잘 설계된 `AGENTS.md`가 컨텍스트를 절약하는 방식은, 좋은 API가 클라이언트의 복잡성을 줄이는 방식과 정확히 같다.

컨텍스트 예산 관리를 명시적으로 설계 기준으로 삼으면 많은 결정이 명확해진다.
주제 문서를 왜 분리하는가. 기능 목록을 왜 JSON으로 관리하는가. 초기화 단계를 왜 구현 단계와 분리하는가.
모두 에이전트가 주어진 컨텍스트 안에서 최대한 효과적으로 작동하도록 환경을 구조화하는 결정이다.

### 파일 시스템 기반 상태 관리는 에이전트 세션 경계를 극복하는 가장 단순한 해법이다

에이전트의 근본적 제약 중 하나는 세션이 끝나면 메모리가 사라진다는 것이다.
이 문제를 해결하는 접근법은 복잡한 벡터 데이터베이스, 지속적인 메모리 시스템, 장기 맥락 주입 등 다양하다.

이 강의 시리즈가 제안하는 해법은 놀라울 정도로 단순하다. 마크다운 파일에 쓰라는 것이다.
`PROGRESS.md`는 완료 체크리스트와 다음 단계를 담는다.
`DECISIONS.md`는 왜 특정 결정을 내렸는지를 기록한다.
`feature_list.json`은 기능의 현재 상태를 기계 판독 가능한 형식으로 유지한다.

이 단순성에는 장점이 많다.
에이전트가 읽는 파일과 개발자가 읽는 파일이 같다.
버전 관리가 되고 팀과 공유되며 CI/CD와 통합된다.
디버깅이 쉽다. 마지막 세션에서 무슨 일이 있었는지 파일을 열면 알 수 있다.
외부 서비스 의존성이 없다. 저장소를 복제하면 상태도 함께 온다.

이 설계의 더 깊은 의미는 에이전트 시스템의 신뢰성이 에이전트의 내부 상태가 아니라 외부 저장소에서 와야 한다는 것이다.
에이전트의 기억을 믿는 대신 파일을 믿는다.
파일은 거짓말을 하지 않는다. 에이전트의 자기 평가는 체계적으로 낙관적이다.
이 비대칭성을 인식하고 설계에 반영하는 것이 하네스 엔지니어링의 핵심 통찰 중 하나다.

### 하네스는 시간이 지날수록 강해지는 피드백 루프를 가져야 한다

강의 10의 “피드백 승격” 개념이 이 시리즈 전체에서 가장 중요한 아이디어 중 하나다.
새로운 에이전트 오류 패턴이 발견될 때마다 자동화된 검사로 전환하면, 하네스가 누적적으로 강화된다.
처음에는 에이전트가 특정 실수를 한다. 그 실수를 자동으로 검출하는 규칙을 하네스에 추가한다. 그 실수는 다시는 같은 방식으로 반복되지 않는다.

이 패턴은 소프트웨어 공학의 회귀 테스트와 정확히 같다.
버그가 발견되면 테스트를 추가하여 그 버그가 다시는 같은 방식으로 나타나지 않도록 한다.
하네스도 같은 방식으로 진화할 수 있다.

그러나 강의 12가 지적하듯 하네스 자체도 부채가 된다.
모델 역량이 향상되면 이전에는 필요했던 구성 요소가 더 이상 필요 없어진다.
과도하게 비대해진 하네스는 에이전트의 자율성을 제한하고 유연성을 떨어뜨린다.

이 긴장이 하네스 엔지니어링의 지속적인 과제다.
충분히 구조화되어 에이전트가 신뢰할 수 있게, 그러나 과도하게 제약하지 않아 에이전트가 효과적으로 작동할 수 있게.
이 균형점은 고정되어 있지 않고 모델 역량과 팀의 경험이 쌓일수록 이동한다.
하네스는 한 번 설계하고 끝나는 것이 아니라 코드베이스처럼 지속적으로 관리되어야 하는 살아있는 시스템이다.
