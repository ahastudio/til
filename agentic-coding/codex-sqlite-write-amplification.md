# Codex SQLite 피드백 로그가 연간 640TB를 쓰고 SSD 수명을 소진한다

원문: <https://github.com/openai/codex/issues/28224>

HN 토론: <https://news.ycombinator.com/item?id=48626930> (501점, 269개 댓글)

## 요약

OpenAI Codex CLI의 SQLite 피드백 로그가 연간 약 640TB를 디스크에 쓰면서 일반 소비자 SSD 수명을 수개월 내에 소진시킬 수 있다는 사실이 이슈 #28224를 통해 공개됐다.
보고자 1996fanrui는 21일 운용 후 드라이브에 37TB가 기록된 것을 확인했으며, 이 속도를 연율로 환산하면 640TB에 달한다.
일반 소비자 SSD의 TBW(Total Bytes Written) 보증은 대략 600TBW 수준으로, 단일 도구가 연간 보증 용량을 초과하는 쓰기를 일으키고 있었다.

핵심 증거는 데이터베이스 내부의 10,000배 불일치다.
오토인크리먼트 카운터는 55억(5.5 billion)을 넘었지만 실제로 유지되는 행(row) 수는 약 50만에 불과했다.
15초 관측 구간 동안 36,211개의 행이 삽입됐지만 전체 보유 행 수는 거의 변하지 않았다.
이는 연속적인 삽입-삭제(insert-and-prune) 패턴으로, 실제 데이터는 거의 남기지 않으면서 WAL(Write-Ahead Log) 파일에 막대한 쓰기 부하를 발생시켰다.

로그 구성 분석에 따르면 TRACE 레벨 로그가 보유 로그 볼륨의 70.7%를 차지했으며, 최대 단일 출처는 `codex_api::endpoint::responses_websocket`으로 527.4MiB를 기록했다.
SQLite 피드백 싱크(feedback sink)가 전역 TRACE 기본 레벨을 사용해 의존성 라이브러리 로그와 WebSocket 원시 프로토콜 페이로드까지 포함한 모든 로그를 영속화(persist)한 것이 근본 원인이었다.

OpenAI 팀은 세 개의 PR로 이 문제를 해결했다.
PR #29432(v0.142.0)에서 WebSocket 이벤트 로깅을 중단했고, PR #29457(v0.142.0)에서 잡음이 많은 로그 소스를 필터링했으며, PR #29599(v0.143.0)에서 브릿지(bridged) 로그 이벤트의 영속화를 중단했다.
이 세 변경으로 로그 볼륨의 약 85%가 제거됐다.
이슈 보고 사이에 이미 문제를 파악한 사용자를 위한 임시 해결책으로는 SQLite 트리거를 이용해 새 삽입을 차단하는 방법이 제시됐다.

## 분석

### 삽입-삭제 패턴이 SQLite에서 특히 위험한 이유

SQLite의 WAL 모드는 쓰기 트랜잭션이 WAL 파일에 먼저 기록된 후 체크포인트(checkpoint)를 통해 주 데이터베이스 파일로 병합되는 구조다.
삽입 후 즉시 삭제하는 링 버퍼(ring buffer) 패턴은 데이터베이스 파일 크기 자체는 제한되지만, WAL 파일에는 삽입과 삭제 각각의 트랜잭션 레코드가 모두 기록된다.
즉, 최종 상태는 작아도 그 상태에 도달하는 과정의 모든 쓰기 작업이 실제 I/O로 발생한다.
이것이 55억 오토인크리먼트 대 50만 보유 행이라는 10,000배 불일치의 정확한 원인이다.

이 패턴은 로그 순환(log rotation)을 SQLite 안에서 구현할 때 흔히 발생한다.
오래된 행을 DELETE하고 새 행을 INSERT하는 과정을 반복하면 VACUUM 없이는 파일 크기가 줄어들지 않으며, VACUUM은 그 자체로 대규모 쓰기를 유발한다.
Codex의 경우 WebSocket 이벤트처럼 고빈도 이벤트를 TRACE 레벨로 모두 기록하면서 이 문제가 극단적으로 증폭됐다.

woadwarrior01은 VACUUM을 실행해 SQLite 파일을 27GB에서 73MB로 줄였다고 보고했다.[^woadwarrior01]
이 370배 압축은 실제 보유 데이터가 얼마나 적은지를 극명하게 보여준다.
27GB의 파일 크기는 삭제된 행의 공간이 재사용되지 않고 누적된 결과이며, VACUUM이 이를 재조직해 실제 데이터만 남긴 것이다.
VACUUM 자체도 쓰기를 유발하지만, 이 수치는 삽입-삭제 사이클이 데이터베이스 파일 내부에 얼마나 큰 공동(空洞)을 만들어냈는지를 실증한다.

### TRACE 기본값은 개발 편의가 운영 안전성을 잠식한 사례다

TRACE 레벨 로깅을 피드백 싱크의 전역 기본값으로 설정한 결정에는 나름의 맥락이 있다.
개발 단계에서 TRACE는 버그를 추적하는 데 필수적이며, 기본값을 TRACE로 두면 배포 직후에도 상세한 진단 데이터를 즉시 확보할 수 있다.
특히 AI 에이전트처럼 복잡한 비결정적 시스템에서는 로그가 충분히 상세할수록 디버깅이 수월하다.

그러나 개발 편의를 위한 기본값이 프로덕션(또는 일반 사용자 로컬 환경)에 그대로 유지되는 것은 전형적인 “개발 환경 설정 누수”다.
개발자 머신에서 몇 시간 돌리는 것과 사용자 SSD에서 21일 연속 동작하는 것은 완전히 다른 운용 조건이다.
의존성 라이브러리의 TRACE 로그와 WebSocket 원시 페이로드까지 포함한다는 사실은, 자체 코드와 외부 라이브러리를 동일하게 취급한다는 의미로, 외부 라이브러리가 TRACE 레벨 메시지를 얼마나 생성하는지 사전에 파악하지 않았음을 시사한다.

jofzar는 TRACE 기본값이 고전적인 실수임에도 현대 하드웨어 덕분에 증상이 즉각 드러나지 않는다는 점을 지적했다.[^jofzar]
과거였다면 TRACE 레벨 로그를 기본으로 두는 즉시 애플리케이션이 느려져 다음 업데이트에서 반드시 수정됐을 것이다.
지금은 NVMe SSD가 막대한 쓰기를 소리 없이 흡수하기 때문에 사용자와 개발자 모두 이상을 느끼지 못하고 SSD 수명만 조용히 소진된다.
하드웨어 성능 향상이 소프트웨어 버그의 즉각적 징후를 지워버리는 역설적 효과를 낳은 것이다.

indiv0는 이 버그가 AI 생성 코드만의 문제가 아님을 상기시켰다.[^indiv0]
그는 자신이 AI 이전의 수제(artisan) OSS 프로젝트에서 동일한 유형의 버그를 경험했다고 밝혔다.
유지보수자가 마지막 백업 시각 계산에서 DST(일광절약시간)를 잘못 처리해 앱이 백업을 끝없이 반복 기록하는 상황이었다.
그가 제기한 더 날카로운 질문은 “AI가 왜 우리보다 나은 품질의 소프트웨어를 만들지 못하는가”이며, 학습 데이터를 크게 넘어서는 지능 향상은 기대하기 어렵다는 것이 그의 답이다.
“슬롭 회사가 슬롭을 만들었다”는 조롱보다, 인간 작성 코드에서도 반복되어 온 같은 실수가 AI 시대에도 되풀이된다는 사실이 더 근본적인 문제다.

### 수정이 세 개의 PR에 분산된 것이 드러내는 코드베이스 구조

세 개의 PR이 각각 다른 각도에서 문제를 해결한 것은 근본 원인이 단일 지점이 아닌 아키텍처 전반에 분산되어 있음을 보여준다.
WebSocket 이벤트 중단, 잡음 소스 필터링, 브릿지 이벤트 영속화 중단은 모두 서로 다른 코드 경로를 다루는 독립적인 수정이다.
만약 로깅 구성이 중앙화된 단일 설정 파일에 있었다면 하나의 PR로 충분했을 것이다.

이는 Codex CLI가 빠른 기능 개발 속도를 유지하면서 로깅 정책을 여러 모듈에 걸쳐 분산된 채로 유지해온 결과로 보인다.
v0.142.0과 v0.143.0에 걸쳐 수정이 나뉜 것도 이 분산된 구조를 반영한다.
85% 감소는 인상적이지만, 동시에 나머지 15%가 여전히 어느 정도의 쓰기를 유발하고 있다는 점에서 완전한 해결이 아님을 시사한다.

### 소비자 SSD TBW와 AI 도구의 충돌

SSD의 TBW 보증은 HDD가 대세이던 시절 설계된 소프트웨어가 NVMe SSD 환경에서 실행되는 현실을 반영하지 못하는 경우가 많다.
Codex CLI처럼 상시 실행되는 AI 에이전트 도구가 늘어나면서, 연속 쓰기 부하가 어느 수준인지 사전에 설계 단계에서 고려해야 한다는 요구가 생겼다.
전통적인 소프트웨어는 사용할 때만 실행되고 종료됐지만, AI 에이전트 도구는 백그라운드에서 지속적으로 동작하는 경향이 있다.
이 차이가 SSD 수명에 미치는 영향은 현재 대부분의 도구 개발자들이 인식하는 것보다 훨씬 크다.

## 비평

### 임시 해결책이 일반 사용자에게 지나치게 어렵다

이슈 보고자와 팀이 제시한 임시 해결책은 SQLite 트리거를 직접 작성해 새 삽입을 차단하는 방법이다.
이는 SQLite 파일 경로를 알고, 터미널에서 `sqlite3` 명령을 실행하며, DDL 트리거 구문을 이해하는 사용자를 전제한다.
Codex의 주요 사용자 중 일부는 프로그래머가 아닌 비개발자 파워유저이며, 설령 개발자라도 데이터베이스 관리가 주 역량이 아닌 경우가 많다.

패치가 배포되기 전 수일에서 수주 동안 SSD 수명이 소모되는 상황에서, 유효한 해결책이 SQL 직접 조작이라는 것은 도구의 안전성에 대한 신뢰를 저하시킨다.
더 적절한 임시 해결책은 `--no-feedback-log` 플래그나 환경 변수를 통해 로그 영속화 자체를 비활성화하는 것이었다.
이미 CLI 도구인 만큼 플래그 하나 추가는 SQL 트리거 작성보다 훨씬 접근하기 쉬운 해결책이다.

dundercoder는 임시 해결책을 넘어 더 근본적인 설계 대안을 제시했다.[^dundercoder]
휘발성 로그처럼 재시작 시 유지할 필요가 없는 데이터라면 RAM 기반 tmpfs를 저장 매체로 사용하는 것이 올바른 설계라는 것이다.
tmpfs는 시스템 재시작 시 내용이 사라지므로 SSD 쓰기가 전혀 발생하지 않는다.
피드백 싱크를 SQLite 파일이 아닌 tmpfs 경로에 두었다면 이번 문제 자체가 발생하지 않았을 것이며, 이 관점에서 보면 SQLite 선택 자체가 이 유형의 로그에 맞지 않는 저장 매체였다.

### 수정 후 85% 감소가 “해결”로 프레이밍되는 것에 의문이 있다

세 개의 PR 이후 로그 볼륨의 약 85%가 제거됐다고 보고됐으며 이슈는 종결됐다.
그러나 85% 감소란 원래 640TB/년이었다면 수정 후에도 약 96TB/년이 여전히 기록된다는 의미다.
일반 소비자 SSD가 600TBW 보증을 받는다는 점에서, 연 96TB는 6년 만에 보증을 소진하는 속도다.
이것이 “허용 가능한 수준”인지는 별도의 기준이 필요하다.

이슈 종결이 “완전한 수정”을 의미하는지, 아니면 “충분히 개선됨”을 의미하는지가 명확하지 않다.
피드백 로그의 존재 목적(버그 수집, 원격 분석)을 달성하면서도 SSD에 무해한 쓰기 속도를 정량화한 목표값이 이슈나 PR 어디에도 제시되지 않는다.
수치 목표 없이 “더 나아짐”으로 종결하는 것은 잠재적인 재발 방지 기준이 없다는 의미다.

### 원격 분석 동의와 디스크 쓰기의 관계가 비명시적이다

Codex CLI가 SQLite에 피드백 로그를 영속화한다는 사실이 사용자에게 명확히 고지됐는지가 이슈 어디에도 다뤄지지 않는다.
“피드백 싱크”라는 이름은 원격 전송을 암시하지만, 로컬 SQLite에 저장한다는 것은 다른 함의를 갖는다.
원격 전송하기 전에 로컬에 버퍼링하는 것이라면 사용자 동의 없이 진단 데이터를 수집하는 구조이며, 로컬에만 저장하는 것이라면 그 데이터를 무엇에 사용하는지가 불명확하다.

WebSocket 원시 프로토콜 페이로드가 TRACE 레벨에서 기록됐다는 사실은 통신 내용이 디스크에 남을 수 있다는 뜻이다.
이것이 대화 내용을 포함하는지, API 응답 전체를 포함하는지 이슈에서 명시되지 않으며, 이는 SSD 수명 문제와는 별개의 개인정보 측면에서 논의될 필요가 있다.

## 인사이트

### AI 코딩 도구는 “서버”처럼 설계되어야 하는 “클라이언트”가 됐다

Codex CLI는 사용자 로컬 머신에서 돌아가는 클라이언트 소프트웨어다.
그러나 상시 실행, 백그라운드 로깅, SQLite 피드백 싱크라는 특성은 서버 소프트웨어에 가깝다.
전통적인 클라이언트-서버 구분에서 클라이언트는 사용 중에만 자원을 소비하고 종료하지만, AI 에이전트 도구는 지속적으로 동작하며 운용 부하를 유발한다.
이는 클라이언트 도구 개발에 서버 운영에서 요구되는 수준의 자원 관리와 관찰가능성(observability) 설계가 필요해졌음을 의미한다.

전통적인 CLI 도구는 실행-완료-종료의 생애주기를 가지므로 장기 실행에 따른 자원 누적이 문제가 되지 않았다.
그러나 Codex처럼 세션 간 상태를 유지하고 지속적으로 로그를 생성하는 도구는 사실상 데몬(daemon)에 가깝다.
데몬 수준의 자원 관리—로그 로테이션 정책, 디스크 사용량 상한, 자원 사용 모니터링—를 클라이언트 도구에 적용하는 설계 패러다임이 AI 에이전트 시대의 새로운 요구사항이 된다.

### 기본값 설정의 역사는 보안·성능 사고의 역사다

소프트웨어 기본값이 잘못 설정되어 대규모 문제를 일으킨 사례는 긴 역사를 갖는다.
MySQL이 기본적으로 원격 접속을 허용해 노출된 데이터베이스 서버, Apache가 `Options Indexes`를 기본 활성화해 디렉토리 목록을 노출한 것, Node.js 패키지가 `--verbose` 플래그를 기본으로 두어 민감한 경로를 로그에 남긴 것이 모두 유사한 패턴이다.
이 사례들의 공통점은 “개발 중에는 더 많이 보이는 것이 편리하다”는 합리적 판단이 배포 후 문제가 된다는 것이다.

Codex의 TRACE 기본값은 이 역사적 패턴의 반복이다.
흥미로운 점은 이 실수가 보안이나 기능 분야가 아니라 디스크 I/O라는 물리적 자원 분야에서 나타났다는 것이다.
AI 에이전트 도구의 확산은 소프트웨어 기본값이 단순히 UX나 보안을 넘어 하드웨어 수명에까지 영향을 미치는 시대를 열고 있다.
“기본값은 가장 보수적이어야 한다”는 원칙이 새로운 차원에서 적용돼야 하는 것이다.

### 10,000배 불일치는 모니터링 부재의 증거다

55억 오토인크리먼트 대 50만 보유 행의 10,000배 불일치는 하루아침에 발생하지 않는다.
이 상태는 수주에서 수개월에 걸쳐 점진적으로 형성됐을 것이다.
그럼에도 OpenAI 팀이 외부 사용자 보고로 처음 이 문제를 파악했다는 사실은, 내부적으로 이 지표를 모니터링하는 체계가 없었음을 보여준다.

소프트웨어가 자체 상태를 모니터링하지 않는다면 문제는 사용자 피해가 가시화될 때까지 발견되지 않는다.
이 경우 사용자는 SSD 수명 단축이라는 비가역적 피해를 입은 후에야 이슈가 제기됐다.
AI 에이전트 도구 개발팀은 내부 상태(데이터베이스 행 수 대 오토인크리먼트 값의 비율, WAL 파일 크기, 초당 쓰기 속도)를 지속적으로 모니터링하는 텔레메트리를 도구 자체에 내재화해야 한다.

자체 로그 시스템의 건강 상태를 모니터링하지 않는 것은 “소방서가 화재 경보기 없이 운영되는” 역설과 같다.
로그를 쓰는 시스템은 그 쓰기 행위 자체가 올바른지 감시하는 별도의 메커니즘이 필요하다.
이 원칙은 AI 에이전트처럼 자율적으로 동작하는 도구일수록 더 중요해진다.

i2km은 이슈가 제기된 지 일주일이 넘도록 OpenAI로부터 공식 반응이 없었다는 사실에 주목했다.[^i2km]
“분명히 여러 에이전트를 GitHub에 연결해 이슈를 모니터링하고 수정을 제안하고 있겠지, 맞지?”라는 그의 반어적 질문은 이 침묵의 아이러니를 정확히 포착한다.
AI 에이전트 도구를 판매하는 회사가 자사 도구를 이슈 트래커 모니터링과 자동 수정에 활용하지 않는다는 것은, 자사 제품의 실용성에 대한 신뢰 부재를 드러내거나, 아니면 실용성을 알면서도 내부 운용에는 적용하지 않는 이중성으로 읽힌다.

### SSD 친화적 설계는 AI 시대의 새로운 비기능 요구사항이다

전통적인 소프트웨어 성능 요구사항은 CPU, 메모리, 네트워크 대역폭을 중심으로 정의됐다.
SSD TBW는 하드웨어 수명을 정의하는 지표였지만, 소프트웨어 설계 요구사항으로 포함되는 경우는 드물었다.
AI 에이전트 도구가 상시 실행되면서 이 가정이 바뀌기 시작했다.
연간 640TB 쓰기는 하드웨어 수명을 소프트웨어 요구사항으로 명시해야 할 필요성을 드러낸다.

데이터베이스 집약적 AI 도구 설계에서 “이 작업이 디스크에 얼마나 쓰는가”는 이제 설계 검토의 필수 항목이 돼야 한다.
일반적인 NVMe SSD의 TBW를 365로 나누면 하루 허용 쓰기량 상한을 계산할 수 있으며, 도구의 예상 쓰기 속도를 이 기준과 비교하는 것이 출시 전 체크리스트에 포함돼야 한다.
SQLite를 로컬 저장소로 사용하는 모든 AI 에이전트 도구 개발팀이 이번 이슈를 참고 사례로 삼아 동일한 설계 오류를 반복하지 않기를 바란다.
