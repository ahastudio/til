# DeerFlow 2.0 — ByteDance의 장기 실행 SuperAgent 하네스

<https://github.com/bytedance/deer-flow>

## 소개

DeerFlow 2.0은 ByteDance가 공개한 오픈소스 에이전트 하네스(agent harness)다.
수 분에서 수 시간까지 걸리는 복합 작업을 자동으로 분해하고, 서브에이전트에게 분배하며,
병렬 처리와 장기 실행을 지원한다.
1.0 버전이 “Deep Research 프레임워크”였다면, 2.0은 완전히 재작성되어 범용 SuperAgent 플랫폼으로 탈바꿈했다.

LangGraph와 LangChain을 기반으로 구축되었으며, MIT 라이선스로 공개되어 있다.

## 주요 기능

**서브에이전트 오케스트레이션**: 복합 작업을 리서치, 코딩, 콘텐츠 생성 등의 단위로
자동 분해하고 전문화된 서브에이전트에게 병렬로 위임한다.

**장기 메모리(Long-Term Memory)**: 세션이 종료되어도 사용자 정보와 컨텍스트가 영속적으로
저장된다. 다음 세션에서 이전 작업의 맥락을 이어받아 실행할 수 있다.

**스킬 시스템**: Markdown 파일 기반의 확장 가능한 스킬 시스템을 갖추고 있다.
새로운 스킬을 Markdown으로 정의하면 에이전트가 해당 절차를 따를 수 있다.

**실행 환경 격리**: Docker 컨테이너 또는 로컬 샌드박스에서 독립적으로 실행되며,
서브에이전트 간 격리를 보장한다.

**IM 통합**: Telegram, Slack, Feishu(飞书), WeChat 등 6개 메신저 채널과 통합된다.

**MCP 서버 지원**: MCP(Model Context Protocol) 서버 확장과 Claude Code 연동을 지원한다.

**LLM 프로바이더 다양성**: OpenAI, OpenRouter, vLLM 등 다양한 LLM 백엔드를 지원한다.

## 아키텍처

LangGraph가 오케스트레이션 레이어를 담당한다. 복합 작업을 방향성 비순환 그래프(DAG)로
표현하고, 각 노드가 서브에이전트 또는 도구 호출에 해당한다. 그래프 기반 실행은
병렬 처리와 조건 분기를 자연스럽게 지원한다.

스킬 시스템은 Claude Code의 스킬 구조와 유사한 Markdown 파일 기반이다.
에이전트가 작업을 받으면 관련 스킬 파일을 참조하여 절차를 따른다.
이 설계는 코드 변경 없이 에이전트의 행동을 확장할 수 있게 한다.

## 분석

### 스킬 시스템과 프로세스 외부화

DeerFlow 2.0의 Markdown 기반 스킬 시스템은 에이전트의 행동을 코드가 아닌 문서로 정의한다.
이 설계는 중요한 함의를 갖는다.
개발자가 아닌 도메인 전문가가 에이전트의 행동을 정의하고 수정할 수 있다.
의사, 법률가, 연구자가 자신의 전문 절차를 Markdown으로 작성하면
그 절차가 에이전트의 실행 가능한 행동 목록이 된다.

이것은 소프트웨어 개발에서 “비즈니스 로직을 코드에서 분리한다”는 오래된 목표의
새로운 구현 형태다. 규칙 엔진, 비즈니스 프로세스 관리(BPM), 워크플로 자동화 도구들이
같은 방향을 지향했지만, 자연어 이해 능력의 한계로 완전히 실현하지 못했다.
LLM은 이 한계를 제거한다.

### ByteDance가 오픈소스로 공개한 이유

중국 기업이 내부 에이전트 인프라를 오픈소스로 공개하는 결정은 전략적 계산을 반영한다.
DeerFlow가 생태계 표준이 되면 ByteDance는 그 표준을 정의한 플레이어로서의 지위를 갖는다.
외부 기여자들이 기능을 추가하고 버그를 수정하는 무료 노동력을 얻을 수 있다.
Claude Code 연동 지원처럼 생태계 내 다른 플레이어와의 호환성을 높여 채택률을 올릴 수 있다.

이 전략은 TensorFlow, React, Kubernetes 같은 대형 오픈소스 프로젝트들이 공통적으로
사용한 것이다. 인프라 레이어를 공개하고, 그 위에서의 서비스나 데이터로 수익을 창출한다.

## 비평

### 강점: 장기 실행 작업에 특화된 설계

대부분의 에이전트 프레임워크는 단일 요청·응답 사이클에 최적화되어 있다.
DeerFlow의 장점은 수 시간 걸리는 복합 작업을 처음부터 설계 목표로 삼았다는 것이다.
장기 메모리, 격리된 실행 환경, IM 통합은 모두 이 사용 사례를 위한 설계 결정이다.

그러나 [멀티 에이전트 오케스트레이션은 왜 잘 안 되는가](multi-agent-orchestration-problems.md)가
지적한 맥락 붕괴(41.8%)·유령 위임(36.9%) 문제는 LangGraph DAG 기반의
서브에이전트 분배에서도 동일하게 재현될 수 있다.
DeerFlow의 장기 메모리와 격리 실행이 이 구조적 병목을 얼마나 해소하는지는
별도의 실측이 필요하다.

### 약점: LangGraph 의존성의 복잡성 비용

LangGraph는 강력하지만 학습 곡선이 가파르고, 디버깅이 어렵다.
그래프 기반 실행 흐름은 단순한 선형 파이프라인보다 추론하기 어렵다.
DeerFlow를 커스터마이징하거나 확장하려면 LangGraph의 내부 동작을 이해해야 하는데,
이것이 실제 도입의 장벽이 될 수 있다.

### 약점: 1.0에서 2.0으로의 완전한 재작성

GeekNews 댓글에서 지적되듯, 2.0은 1.0과 완전히 다른 소프트웨어다.
1.0 사용자에게 마이그레이션 경로가 없고, 1.0 생태계에서 개발된 통합이나
플러그인이 재사용되지 않는다. 완전한 재작성은 기술 부채 청산의 기회지만,
커뮤니티 연속성의 단절이기도 하다.

## 인사이트

[하네스 엔지니어링](harness-engineering.md)이 정의한 “제약·지시·검증·교정”의
네 책임을 SuperAgent 플랫폼이 한 번에 묶어 제공하려는 시도로 읽을 수 있다.

### SuperAgent 하네스는 다음 인프라 레이어다

웹 프레임워크가 HTTP 처리를 추상화했듯, 에이전트 하네스는 LLM 오케스트레이션을 추상화한다.
DeerFlow처럼 작업 분해, 서브에이전트 위임, 장기 메모리, 격리 실행을 처리하는 레이어가
표준화되면, 개발자는 “어떻게 에이전트를 오케스트레이션할 것인가”보다
“에이전트로 무엇을 할 것인가”에 집중할 수 있다.

이것은 웹 개발에서 Rails, Django, Express가 등장했을 때와 유사한 전환점이다.
HTTP 파싱과 라우팅을 직접 구현하는 것에서 벗어나, 비즈니스 로직에 집중하게 된 것처럼.
에이전트 하네스가 성숙해지면 에이전트 개발의 생산성이 현재와 비교할 수 없을 만큼
높아질 것이다.

[에이전틱 소프트웨어의 5단계](agentic-software-levels.md) 관점에서 DeerFlow는
Level 4(멀티 에이전트 팀) 위에 Level 5(프로덕션 런타임)를 통째로 얹은
형태다.
저자가 “Level 1에서 시작하라”고 경고한 것과 정반대 방향에서 출발한 셈이라,
도입 비용이 만만치 않을 가능성이 크다.

### 스킬 기반 에이전트와 조직 지식의 코드화

DeerFlow의 Markdown 스킬 시스템은 조직 지식을 에이전트 행동으로 변환하는 인터페이스다.
조직이 수십 년 동안 문서화한 SOP(표준 운영 절차), 체크리스트, 의사결정 트리가
에이전트의 실행 가능한 스킬 목록이 된다.

이것은 지식 관리 분야에서 오랫동안 추구해온 목표의 실현 가능성을 높인다.
조직 지식은 항상 두 형태로 존재했다. 문서화된 명시적 지식과 사람들의 머릿속에 있는
암묵적 지식. LLM이 자연어 절차를 이해하고 실행할 수 있게 되면서,
명시적 지식과 실행 가능한 행동 사이의 간극이 좁혀진다.
이것이 에이전트 기술이 단순한 자동화를 넘어 조직 운영 방식 자체를 바꿀 수 있는 이유다.
