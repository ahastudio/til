# 루프 엔지니어링 — 에이전트를 프롬프트하는 시스템을 설계하기

원문: <https://addyo.substack.com/p/loop-engineering>

HN 토론: <https://news.ycombinator.com/item?id=48514387> (9점, 6개 댓글)

## 요약

저자는 AI 코딩 에이전트와 협업하는 방식이 근본적으로 전환됐다고 주장한다.
이전에는 엔지니어가 에이전트에게 직접 프롬프트를 제공했다.
이제는 엔지니어가 에이전트에게 프롬프트하는 시스템 자체를 설계한다.
이 전환을 “루프 엔지니어링(loop engineering)”이라 부른다.

루프 엔지니어링의 다섯 가지 핵심 구성 요소가 있다.
첫째, 자동화(Automations)는 스케줄에 따라 독립적으로 발견과 분류를 수행하는 태스크다.
CI 실패나 미결 이슈를 분석해 분류함(triage inbox)에 기록한다.
둘째, 워크트리(Worktrees)는 병렬 에이전트 작업을 위한 격리 단위다.
여러 에이전트가 동시에 파일을 수정할 때 충돌을 방지하기 위해 각 에이전트가 별도의 Git 워크트리 위에서 작동한다.
셋째, 스킬(Skills)은 프로젝트 특화 지식을 한 번 문서화하여 세션 간에 재사용하는 단위다.
SKILL.md 파일로 구성되며 반복적인 컨텍스트 설명을 없앤다.
넷째, 플러그인과 커넥터(Plugins and Connectors)는 MCP 표준 위에서 에이전트가 이슈 트래커, 데이터베이스, 외부 API에 접근하게 한다.
다섯째, 서브에이전트(Sub-agents)는 구현 에이전트와 검증 에이전트를 분리하여 자기 채점을 방지한다.

저자는 지속적 메모리(persistent memory)를 별도 지원 인프라로 강조한다.
에이전트는 실행 간에 모든 것을 잊기 때문에, 마크다운 파일이나 Linear 보드 같은 외부 상태 저장소가 필수다.
루프가 진행 상황을 재도출하는 대신 이전 작업을 이어갈 수 있으려면 대화 컨텍스트 밖에 상태가 유지되어야 한다.

글은 세 가지 지속적 위험을 경고로 제시한다.
검증 책임은 여전히 인간에게 있다.
감독 없는 루프는 실수를 대규모로 만들어낸다.
이해 부채(comprehension debt)는 코드가 빠르게 나올수록 엔지니어의 시스템 이해와 실제 시스템 상태 사이의 간극이 벌어지는 현상이다.
인지 포기(cognitive surrender)는 매끄러운 자동화가 수동성을 유발하는 위험이다.
루프는 이해한 것을 가속하는 도구가 되어야 하지, 사고를 회피하는 도구가 되어서는 안 된다.

## 분석

### 루프 엔지니어링의 위치: 추상화 층위의 이동

소프트웨어 엔지니어링의 역사는 추상화 층위가 올라가는 역사다.
어셈블리에서 고급 언어로, 명령형에서 선언형으로, 직접 구현에서 프레임워크 사용으로 이동했다.
루프 엔지니어링은 이 흐름의 연장선에 있다.
직접 코드를 쓰는 것에서 에이전트에게 코드를 쓰게 하는 것으로, 이제는 에이전트를 조율하는 시스템을 설계하는 것으로 이동한다.

각 추상화 이동은 동일한 패턴을 따른다.
이전 층위의 기술이 필요 없어지는 게 아니라, 이전 층위를 이해하는 사람이 상위 추상화를 더 잘 사용하게 된다.
어셈블리를 아는 개발자가 C를 더 잘 쓰듯, 직접 에이전트를 프롬프트해본 사람이 루프를 더 잘 설계한다.
저자는 이 연속성을 “더 깊은 엔지니어링 규율이 필요하다”는 문장으로 인정한다.

### 다섯 가지 구성 요소의 설계 원칙

다섯 가지 구성 요소는 각각 독립된 기능이 아니라 하나의 설계 원칙을 다른 측면에서 구현한다.
그 원칙은 “상태를 외부화하고, 작업을 격리하고, 지식을 재사용하라”다.

자동화는 반복 작업을 외부화한다.
워크트리는 병렬 작업 간의 상태 충돌을 격리한다.
스킬은 프로젝트 지식을 재사용 가능한 형태로 외부화한다.
커넥터는 외부 시스템의 상태를 에이전트가 접근 가능하게 한다.
서브에이전트는 구현과 검증의 역할을 격리하여 자기 참조의 편향을 제거한다.

이 원칙들은 분산 시스템 설계에서 익숙한 것들이다.
루프 엔지니어링은 사실상 에이전트들로 이루어진 분산 시스템을 설계하는 일이다.
[장기 실행 에이전트](./long-running-agents.md)에서 정리한 Anthropic의
Brain·Hands·Session 구조와 Cursor의 Planner·Worker·Judge 분리가 이 원칙의
구체적 산업 사례다 — 다섯 구성 요소가 독립된 발명이 아니라 공통 설계 패턴의
다른 표현임이 드러난다.

### 이해 부채: 가속이 만드는 새로운 기술 부채

shakespeares는 “이해는 방치하면 썩음”이라는 표현을 사용했다.[^shakespeares]
이 표현은 저자가 “이해 부채”라고 부르는 개념의 핵심을 포착한다.
기존의 기술 부채(technical debt)가 코드 품질의 저하라면, 이해 부채는 코드에 대한 엔지니어의 이해 수준의 저하다.

루프가 빠르게 코드를 생성할수록 엔지니어가 결과물을 이해할 시간은 줄어든다.
기술 부채는 리팩토링으로 해소할 수 있지만, 이해 부채는 시간을 투자해 읽고 이해하는 것 외에 다른 해소 방법이 없다.
그리고 루프가 만들어내는 코드의 속도는 이해를 위해 필요한 시간을 구조적으로 앞선다.

cyanydeez는 이 현상을 "AI 정신증(AI psychosis)"이라 부른다.[^cyanydeez]
토큰 생성 속도가 너무 빠른 탓에 아무도 책임질 수 없는 방대한 코드베이스가 생겨나는 것이 그 증상이다.
그가 제안하는 실용적 대안은 속도를 낮추는 것이다.
로컬 LLM을 사용해 생성 속도를 줄이면 엔지니어가 생성 과정을 따라가며 방향을 수정할 수 있다.
이것은 루프 엔지니어링의 생산성 약속이 이해 부채 해소와 구조적으로 긴장 관계에 있음을 시사한다.

aocallaghan17은 이 대안적 접근을 지지하면서 루프 엔지니어링에 대한 직접적인 반론을 제기한다.[^aocallaghan17c]
"프롬프트가 아니라 루프를 만들라"는 주장이 결국 인간을 루프에서 배제하는 방향을 지향한다는 것이다.
그는 프로덕션 시스템에서는 자신이 루프의 운전자가 아닌 상태를 받아들일 수 없다고 말한다.
이 불편함은 개인적 선호의 문제가 아니다.
루프가 자동으로 내린 결정에 대한 책임을 인간이 지는 구조에서, 인간이 루프를 통제하지 못한다는 사실은 책임의 공백을 만든다.

saidnooneever는 더 냉소적인 관찰을 남긴다.[^saidnooneever]
"AI가 풀스택으로 이해를 담당하고 있다. 문제가 생겨도 AI가 처리할 것이니 아무도 신경 쓰지 않는다."
그는 이것을 불편하게 여기지만 이미 많은 팀이 그렇게 움직이고 있다고 인정한다.
이 관찰은 이해 부채가 단순한 위험이 아니라 이미 현재 진행 중인 현실임을 시사한다.

### 검증 분리의 한계

저자는 구현 에이전트와 검증 에이전트를 분리하면 자기 채점의 편향을 줄일 수 있다고 제안한다.
이것은 맞다.
그러나 두 에이전트 모두 같은 베이스 모델에서 나온다면, 같은 종류의 맹점을 공유할 가능성이 있다.
구현 에이전트가 생각하지 못한 케이스를 검증 에이전트도 생각하지 못할 수 있다.

ultimategamer는 “AI 방법론이 프롬프트에서 컨텍스트로, 하네스로, 이제 루프로 확장돼간다”고 관찰했다.[^ultimategamer]
각 새로운 패턴은 이전 패턴의 한계를 해결하지만 새로운 한계를 도입한다.
루프 엔지니어링이 해결하지 못한 한계가 다음 패턴의 출발점이 될 것이다.

aocallaghan17은 비결정성의 문제를 구체적으로 제기한다.[^aocallaghan17b]
팀원 둘이 같은 질문을 에이전트에게 던졌을 때 서로 다른 방법론으로 서로 다른 결과가 나온다면, 그 결과를 기반으로 한 결정은 일관성을 잃는다.
이 질문은 검증 에이전트를 추가한다고 해소되지 않는다.
비결정성은 구현 에이전트와 검증 에이전트 모두에게 동일하게 적용되기 때문이다.
[Anthropic의 장기 실행 하네스 글](./harness-design-long-running-apps.md)이
지적한 “평가자도 같은 모델이라 무한 후퇴가 발생한다”는 문제와 정확히 같으며,
그 글이 few-shot 보정과 인간이 작성한 채점 기준을 “앵커”로 두는 해법을
제시했지만 이 책의 우려가 사라지지는 않는다.

## 비평

### 루프 설계의 진입 장벽은 저자가 인정하는 것보다 높다

저자는 루프 엔지니어링을 비교적 접근 가능한 것으로 묘사하지만, 실제로 효과적인 루프를 설계하려면 분산 시스템 설계, 프롬프트 엔지니어링, 도메인 지식이 동시에 필요하다.
Git 워크트리, MCP 커넥터, 스킬 파일 구조를 모두 올바르게 설계하고 조율하는 것은 상당한 선행 투자가 필요하다.
글은 이 복잡성을 간략히 언급하지만, 루프를 처음 구축하는 사람이 마주할 실제 장벽을 과소평가한다.

특히 ide127이 지적하듯 “하루에 하나씩 새로운 에이전트 시스템이 나온다.”[^ide127]
이 속도는 루프 설계에 투자하는 사람이 다음 주에는 다시 배워야 할 수도 있음을 의미한다.
저자는 이 도구 불안정성(tooling instability)이 루프 투자의 매몰 비용을 어떻게 높이는지 다루지 않는다.

### 자동화가 덮는 문제와 드러내는 문제

루프 엔지니어링은 반복 작업을 자동화하지만, 그 과정에서 문제를 덮는 위험이 있다.
CI 실패를 자동으로 분석하고 수정하는 루프는 같은 실패가 반복되는 패턴을 감지하지 못할 수 있다.
루프가 수정하는 동안 근본 원인은 방치된다.
자동화는 증상을 빠르게 처리하지만, 증상을 통해 근본 문제를 파악하는 인간의 능력을 약화시킨다.

이것은 루프가 나쁘다는 뜻이 아니다.
루프를 설계할 때 “이 루프가 무엇을 보이지 않게 만드는가”를 명시적으로 질문해야 한다는 뜻이다.
저자는 이 질문을 건너뛴다.

### 인간 판단의 역할이 불분명하게 남는다

저자는 분류함(triage inbox)에 미해결 항목이 쌓이면 인간이 검토한다고 설명한다.
그러나 루프가 대부분을 처리한 후 남은 항목들은 가장 어렵거나 애매한 것들이다.
그 항목들을 검토하는 인간은 루프가 이미 처리한 맥락을 모두 파악하지 못한 상태에서 가장 어려운 결정을 내려야 한다.

이것은 “잔여 판단(residual judgment)” 문제다.
자동화가 쉬운 것을 처리하면 할수록, 인간에게 남는 것은 더 어렵고 더 많은 맥락이 필요한 결정들이다.
저자는 루프가 인간의 부담을 덜어준다고 말하지만, 실제로는 인간의 개입이 필요한 순간을 더 어렵게 만들 수 있다.

aocallaghan17은 이 문제를 더 근본적으로 제기한다.[^aocallaghan17]
루프가 대부분의 작업을 처리한다 해도, 프로덕션 시스템에 대한 이해와 책임은 여전히 인간 엔지니어에게 있다.
자신이 책임지고 유지·발전시켜야 하는 시스템을 이해하지 못한 채로 운영하는 것이 과연 가능한가.
이해하지 못한 채로 책임을 지는 상황은 루프가 정상 상태로 정착할 경우 구조적으로 발생한다.

## 인사이트

### 루프 엔지니어링은 소프트웨어 팀의 조직 구조를 재설계한다

루프 엔지니어링이 성숙하면 팀 내 역할 분배가 바뀐다.
코드를 직접 쓰는 역할은 줄고, 루프를 설계하고 유지하는 역할이 늘어난다.
이것은 전통적인 시니어-주니어 구조와 다른 역량 위계를 만든다.
루프 설계 역량은 코딩 능력보다 시스템 사고, 도메인 이해, 품질 기준 설정 능력에 더 의존한다.

이 변화는 채용, 온보딩, 코드 리뷰 방식을 모두 바꾼다.
새로운 팀원은 코드베이스보다 루프 구조를 먼저 이해해야 한다.
코드 리뷰는 코드 품질보다 루프 설계의 안전성과 의도를 검토하는 방향으로 이동한다.
팀이 이 전환에 준비되지 않으면, 루프 도입은 생산성이 아니라 혼란을 늘린다.

pixel_popping은 이 전환의 방향을 더 급진적으로 해석한다.[^pixel_popping]
내일의 코드베이스는 인간이 아니라 AI만이 다루게 될 것이므로, 모든 것을 인간이 아닌 AI를 위해 설계해야 한다는 것이다.
문서는 AI가 문서를 생성하도록 안내하는 포인터만 있으면 충분하다.
이 관점이 맞다면, 루프 설계 역량은 일시적 경쟁 우위가 아니라 소프트웨어 팀 운영의 기본 전제가 된다.

### 이해 부채는 기술 부채보다 더 빠르게 가속된다

기술 부채의 이자는 시간이 지남에 따라 코드베이스가 수정하기 어려워지는 형태로 나타난다.
이해 부채의 이자는 더 즉각적으로 나타난다.
루프가 생성한 코드를 이해하지 못한 채로 다음 루프를 돌리면, 두 번째 루프는 첫 번째 루프의 결과물 위에서 이해되지 않은 결정들을 증폭시킨다.
이 가속은 기술 부채와 달리 리팩토링으로 해소되지 않는다.

이것은 루프 엔지니어링이 반드시 주기적인 이해 복원(comprehension restoration) 사이클을 포함해야 함을 의미한다.
루프를 돌리는 시간과 루프가 만든 것을 이해하는 시간이 균형을 이루지 않으면, 시스템은 점점 더 아무도 이해하지 못하는 코드베이스를 향해 수렴한다.
이 균형을 어떻게 측정하고 유지할 것인가는 루프 엔지니어링이 아직 답을 제시하지 못한 질문이다.
[Harness Engineering](./harness-engineering.md)이 말하는 “엔트로피 관리 에이전트”와 [하네스 엔지니어링 백과사전](./harness-engineering-encyclopedia.md)의 “Garbage Collection” 패턴이 이 문제에 대한 부분적 응답이다 — 코드 생성과 코드 정리를 별개 에이전트로 분리하는 단일 책임 원칙의 에이전트 수준 적용이지만, “이해”의 외재화는 여전히 미결 영역이다.

### 루프는 개인 도구가 아니라 팀 인프라다

저자는 루프 엔지니어링을 개인 생산성 향상 도구로 소개한다.
그러나 루프의 구성 요소들—스킬, 커넥터, 서브에이전트 설정—은 팀 전체가 공유해야 가치를 극대화한다.
개인이 독립적으로 서로 다른 루프를 구축하면 팀의 일관성이 깨진다.
루프 설계가 표준화되지 않으면, 팀원마다 다른 에이전트 동작 기대치를 갖게 되고 이것은 협업의 새로운 장벽이 된다.

이것은 루프 엔지니어링이 개인 역량이 아니라 팀 거버넌스 문제임을 시사한다.
어떤 스킬을 공유할 것인가, 어떤 커넥터를 표준으로 할 것인가, 루프 설계를 어떻게 리뷰하고 승인할 것인가가 모두 팀 수준에서 결정되어야 한다.
루프 엔지니어링이 성숙한 팀은 루프 설계 문서를 코드 아키텍처 문서와 같은 수준으로 관리할 것이다.
