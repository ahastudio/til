## 소개

김동학 저. v1.84(기준일: 2026년 5월 12일). 7,000원.

처음 접하는 사람부터 실무자까지를 대상으로 하는 AI 에이전트 시대의 실전
입문서다. 들어가기(7개 섹션) + 제1~15장 + 맺음말 + 부록 A~O로 구성된다. 각 장
마지막에는 가족 여행·이사·식단·도구 정리 같은 일상생활 주제의 하네스 실습이 붙어
있다. 실습 도구로 Claude Code용 `/harness-lab` 스킬과 Codex CLI용
`$harness-lab`을 제공한다.

이 책이 출발하는 질문은 하나다.

> "AI가 실패하는 이유는 모델의 문제인가, 모델이 일하는 환경의 문제인가?"

책의 한 문장: "좋은 AI 활용은 좋은 질문에서 시작하지만, 오래가는 AI 시스템은
좋은 하네스에서 완성된다."

## 핵심 개념: 하네스

**하네스(Harness)**: "복잡하고 위험할 수 있는 요소를 제멋대로 흩어지지 않게
묶고, 필요한 방향으로 힘과 신호가 흐르게 만드는 구조."

물리적 기원: 등산 안전벨트·말의 마구·자동차 배선 묶음(wiring harness)에서 유추.
소프트웨어 기원: ISTQB의 테스트 하네스 — "테스트 실행에 필요한 stubs와 drivers로
구성된 테스트 환경."

핵심 비유: **"원시 LLM은 운영체제 없는 CPU와 같다."**

| 컴퓨터 구조   | LLM 에이전트 구조       |
| ------------- | ----------------------- |
| CPU           | LLM·모델 가중치         |
| RAM           | 컨텍스트 창             |
| 하드디스크    | 벡터DB·문서·장기 저장소 |
| 장치 드라이버 | 도구 통합·MCP·API       |
| 운영체제      | 에이전트 하네스         |
| 애플리케이션  | 에이전트 행동           |

**에이전트(Agent) = 모델(Model) + 하네스(Harness)**

하네스에 속하는 것: 시스템 프롬프트, 도구, 스킬, MCP와 그 설명,
파일시스템·샌드박스·브라우저, 오케스트레이션 로직, hooks와 middleware.

새로운 기능 배치 기준:

| 기능의 성격             | 배치할 곳 |
| ----------------------- | --------- |
| 안정적으로 보관할 상태  | 메모리    |
| 반복하는 절차·판단 요령 | 스킬      |
| 상호작용 규칙           | 프로토콜  |
| 흐름 통제·되돌리기      | 중재 장치 |

## 프롬프트·컨텍스트·하네스 엔지니어링

| 구분                | 핵심 질문                       | 일상 비유                         |
| ------------------- | ------------------------------- | --------------------------------- |
| 프롬프트 엔지니어링 | 어떻게 말할 것인가?             | 업무 지시서 쓰기                  |
| 컨텍스트 엔지니어링 | 무엇을 보여 줄 것인가?          | 책상 위 자료 정리                 |
| 하네스 엔지니어링   | 어떤 환경에서 일하게 할 것인가? | 사무실·도구·승인·검사 체계 만들기 |

컨텍스트 창의 6가지 재료:

| 재료             | 잘못 설계했을 때                           |
| ---------------- | ------------------------------------------ |
| 시스템 지시      | 규칙이 너무 길거나 서로 충돌               |
| 사용자 요청      | 목표가 모호해 결과가 흔들림                |
| 검색·RAG         | 관련 없는 문서가 섞이거나 중요한 것을 놓침 |
| 장기 메모리      | 잘못된 기억이 반복 오류를 만듦             |
| 사용 가능한 도구 | 도구가 많거나 설명이 모호해 잘못 호출      |
| 출력 형식        | 답변은 맞아도 검토하거나 재사용하기 어려움 |

"Lost in the Middle" 연구: 중요한 정보가 긴 컨텍스트의 중간에 묻히면 성능이
떨어진다. `AGENTS.md`와 `CLAUDE.md`는 백과사전이 아니라 어디를 봐야 하는지 알려
주는 지도여야 한다.

## 에이전트 루프

기본 구조: 관찰 → 계획 → 행동 → 검증 → 기록 → 반복.

7단계 상세:

1. **프롬프트 조립**: 시스템 지시·도구 스키마·메모리·대화 기록·사용자 요청을 한
   입력으로 묶음. 중요한 정보는 앞이나 뒤에 배치
2. **LLM 추론**: 최종 답변·도구 호출·다른 에이전트로의 위임을 판단
3. **출력 분류**: 최종 답변인지, 도구 실행 요청인지, handoff 요청인지 구분
4. **도구 실행**: 인자 검사, 권한 정책 확인, 가능하면 샌드박스 실행. 읽기 전용은
   병렬 가능, 파일 수정은 순차 실행이 안전
5. **결과 패키징**: 도구 결과를 관찰 메시지로 변환. 오류는 숨기지 말고 복구 정보
   제공
6. **컨텍스트 업데이트**: 새 관찰·파일 변경·테스트 결과·진행 기록을 상태에 반영
7. **반복 또는 종료**: 도구 호출 없는 최종 답변이면 종료. 또는 최대 턴 수·토큰
   예산·가드레일·승인 필요·안전 거절 등으로 중단

ReAct vs Plan-and-Execute:

| 전략             | 특성         | 어울리는 상황                   |
| ---------------- | ------------ | ------------------------------- |
| ReAct            | 매 단계 생각 | 불확실한 탐색, 디버깅, 조사     |
| Plan-and-Execute | 계획 후 실행 | 순서가 명확한 반복 업무, 병렬화 |

LLMCompiler 연구: Plan-and-Execute가 ReAct 대비 최대 3.7배 지연시간 개선, 최대
6.7배 비용 절감.

하위 에이전트 오케스트레이션 세 가지 형태:

| 형태     | 설명                               | 주의점                                           |
| -------- | ---------------------------------- | ------------------------------------------------ |
| Fork     | 부모의 컨텍스트를 복사해 별도 작업 | 결과 요약을 잘해야 부모 컨텍스트가 오염되지 않음 |
| Teammate | 별도 터미널·세션에서 동료처럼 일함 | 누가 무엇을 했는지 기록이 필요                   |
| Worktree | 격리된 브랜치·작업공간에서 수정    | merge와 충돌 해결 정책이 필요                    |

## 도구 엔지니어링

도구는 "말 밖의 행동": 검색·계산·파일 읽기·이메일 작성·티켓 조회·데이터베이스
질의·브라우저 클릭·코드 실행.

제품급 도구 레이어의 5가지 책임: 등록 / 스키마 검증 / 권한 확인 / 격리 실행 /
결과 포맷팅.

도구 이름은 목적이 드러나야 한다 (`do_stuff`가 아니라 `search_policy_docs`).

위험도에 따른 도구 분류:

| 위험도 | 도구 예시               | 원칙              |
| ------ | ----------------------- | ----------------- |
| 낮음   | 파일 읽기, 문서 검색    | 자동 허용 가능    |
| 중간   | 파일 수정, 초안 작성    | 작업 후 확인 필요 |
| 높음   | 이메일 전송, 결제, 배포 | 사람 승인 필수    |

도구 폭발 문제: 도구가 많아지면 컨텍스트를 소비하고 잘못된 선택이 발생한다.
해결책: 계층화하고 필요한 도구만 동적으로 노출.

Claude Tool Search: 도구 정의 컨텍스트 85% 이상 절감.

MCP(Model Context Protocol): "AI용 USB-C 포트." 외부 도구와 데이터 소스를
에이전트에 연결하는 공개 표준. 연결 가능 예시: Jira, Sentry, PostgreSQL, Figma,
Slack, Gmail draft.

A2A vs MCP:

| 구분        | MCP                                     | A2A                                 |
| ----------- | --------------------------------------- | ----------------------------------- |
| 핵심 질문   | 에이전트가 어떤 도구를 어떻게 사용할까? | 어떤 에이전트에게 어떤 일을 맡길까? |
| 연결 대상   | 도구·API·데이터 소스                    | 다른 에이전트·원격 에이전트         |
| 하네스 계층 | 도구 연결 계층                          | 에이전트 협업 계층                  |

경쟁 관계가 아니라 함께 쓰인다. 오케스트레이터는 A2A로 하위 에이전트에게
위임하고, 하위 에이전트는 MCP로 도구를 호출한다.

## OpenAI 하네스 구축

Responses API: 모델 응답·도구 호출·built-in tools·function calling·MCP 연결을
하나의 흐름으로 구성하는 통로.

Agents SDK 구성요소:

| 구성요소         | 비서 사무실에서의 모습                              |
| ---------------- | --------------------------------------------------- |
| Responses API    | 모든 요청이 처음 들어오는 접수 창구                 |
| Function calling | 캘린더·메일·예약 앱을 실제로 여는 행동              |
| Built-in tools   | 검색·파일 찾기·계산 같은 기본 도구                  |
| Handoff          | 전문 담당자에게 넘기는 일                           |
| Guardrail        | 결제·외부 발송·민감 정보 처리 전에 멈추는 확인 규칙 |
| Tracing          | 누가 어떤 도구를 썼고 어떤 결과가 나왔는지 기록     |
| Eval             | 비서 업무가 정확했는지 주기적으로 확인하는 점검표   |

모델 선택 기준:

| 작업              | 필요한 모델 특성                       |
| ----------------- | -------------------------------------- |
| 간단 분류         | 빠르고 저렴한 모델                     |
| 깊은 리서치       | 긴 컨텍스트와 추론이 강한 모델         |
| 법무·의료 보조    | 정확성·출처·안전 절차가 중요한 모델    |
| 멀티에이전트 운영 | 도구 호출 안정성과 trace가 중요한 모델 |

"잘 설계된 하네스는 가장 비싼 모델 하나에 의존하지 않는다."

에이전트가 만드는 것은 코드만이 아니다:

| 산출물           | 하네스 관점                                    |
| ---------------- | ---------------------------------------------- |
| 제품 코드        | 사용자가 직접 만나는 기능                      |
| 테스트 코드      | 다음 변경에서도 기능이 깨지지 않게 하는 안전망 |
| 문서             | 다음 에이전트와 사람이 읽을 수 있는 업무 기억  |
| eval 케이스      | 하네스가 좋아졌는지 비교하는 시험지            |
| trace와 대시보드 | 어디서 실패했는지 찾는 계기판                  |

## Claude 하네스 구축

Claude Code 5층 하네스 구조:

| 층  | 구성요소                   | 역할                         | 비유                 |
| --- | -------------------------- | ---------------------------- | -------------------- |
| 1층 | `CLAUDE.md`·auto memory    | 항상 보이는 기본 규칙        | 사무실 벽의 규칙표   |
| 2층 | Skills·templates·scripts   | 필요할 때 열리는 업무 매뉴얼 | 반복 업무 레시피     |
| 3층 | Hooks·settings·permissions | 자동 점검과 차단             | 가스 차단기·출입카드 |
| 4층 | Subagents·Agent Teams      | 일을 나누는 전문 담당자      | 역할별 팀원          |
| 5층 | Plugins·marketplaces       | 배포 가능한 기능 묶음        | 표준 운영 패키지     |

`CLAUDE.md` 6가지 기본 영역: 대화 방식 / 변경 통제 / 사용자와 프로젝트 맥락 /
기억과 연속성 / 개발 작업 안전 / 고위험 행동 차단.

"CLAUDE.md는 마법의 계약서가 아니다. 너무 길거나 서로 충돌하는 규칙이 많으면
오히려 품질이 떨어질 수 있다."

Skill의 5가지 구성요소: YAML 트리거 헤더 / Overview / Workflow / Output Format /
Examples and Edge Cases.

**점진적 공개(Progressive Disclosure)**: 프론트매터는 항상 로드 → `SKILL.md`
본문은 Skill 활성화 시 → references·scripts는 필요할 때만.

## 장시간 실행 에이전트

### Ralph Loop

세션을 넘나드는 근무 교대 시스템이다. 비유: "교대근무 인수인계 없이 출근."

1. **Initializer Agent**: `init.sh`, `progress.md`, `feature_list.json`, 첫 git
   commit 준비
2. **Coding Agent**: `git log` + `progress.md` + `feature_list.json` 읽기 →
   미완료 기능 선택 → 구현 → 테스트 통과 시에만 `passes: true`로 변경 → commit +
   progress 업데이트

`feature_list.json`은 단순 할 일 목록이 아니라 **검증 대장**이다.

세션 시작 절차:

```text
새 세션 시작 → pwd 확인 → progress.md 읽기 → git log 읽기
→ feature_list.json에서 미완료 항목 선택 → init.sh 또는 smoke test 실행
→ 한 기능만 작업 → 테스트 후 상태 업데이트 → commit + progress 기록
```

핵심: "모델이 기억한다"보다 "다음 모델도 읽을 수 있게 남긴다."

### Handoff Artifact

좋은 인계 문서 포함 항목:

```text
- 목표: 현재 프로젝트의 달성 목표
- 완료된 일: 검증된 완료 항목
- 진행 중인 일: 미완료 항목
- 다음 행동: 다음 세션의 최우선 작업
- 주의점: 실패한 접근, 위험 파일, 미해결 질문
```

### Context Anxiety

컨텍스트 창이 거의 차면 에이전트가 서둘러 마무리하려는 경향. 해결책:
`progress.md`·`feature_list.json`·git history 같은 원천 기록과 병행.

장시간 작업에서 자주 생기는 실패:

| 실패 모습                  | 하네스 보완                                    |
| -------------------------- | ---------------------------------------------- |
| 너무 일찍 "전체 완료" 선언 | feature list의 남은 항목 먼저 읽기             |
| 버그 있는 상태로 인계      | 실패 테스트와 오류 로그를 progress file에 기록 |
| 실제 사용 흐름 미확인      | 브라우저 검증·스크린샷·시나리오 테스트 요구    |

## Generator-Evaluator 하네스

Anthropic의 3-agent 아키텍처:

- **Planner**: 큰 목표를 작은 단위로 분할. 범위를 작게 자르고 완료 기준을 분명히
  하되, 구현 세부사항은 Generator에게 맡긴다
- **Generator**: 코드·문서·보고서·이메일 초안 등을 만드는 에이전트
- **Evaluator**: 비판적으로 검사. "친절한 칭찬이 아니라 결함 발견"이 목표

Evaluator 지시 원칙:

```text
너는 Evaluator다.
Generator의 결과를 비판적으로 검토하라.
통과 기준을 충족하지 못하면 구체적 근거와 재현 절차를 작성하라.
수정하지 말고 평가만 하라.
```

자기평가 편향: "사람도 자기 글을 스스로 검토하면 오타를 놓친다. AI도 자기 결과를
평가할 때 지나치게 긍정적일 수 있다."

주관적 품질을 채점 가능하게 만드는 루브릭:

| 기준     | 0점            | 1점            | 2점              | 3점                |
| -------- | -------------- | -------------- | ---------------- | ------------------ |
| 명확성   | 기능만 있음    | 일부 맞음      | 사용 흐름이 명확 | 실제 제품 수준     |
| 일관성   | 완전 불일치    | 부분 일치      | 대체로 일치      | 완벽하게 일치      |
| 접근성   | 글자 너무 작음 | 부분 대비 부족 | 기본 접근성 충족 | WCAG 수준          |
| 오류처리 | 없음           | 기술적 메시지  | 친절한 안내      | 복구 방법까지 제시 |

Sprint Contract: 이번 범위 / 제외 범위 / 완료 기준을 명시한 반복 작업의 약속
문서. 일회용이 아니라 Generator와 Evaluator의 피드백으로 반복 협상된다.

## 평가 하네스

### Eval의 세 가지 요소

- **Trial**: 같은 테스트를 한 번 실행한 결과. 여러 번 실행 필요
- **Grader**: 점수를 매기는 함수나 모델
- **Assertion**: "반드시 만족해야 하는 조건" (예: 이메일을 실제로 전송하지
  않았는가?)

### 세 가지 검증 방식

| 검증 방식      | 잘 잡는 문제            | 한계                     |
| -------------- | ----------------------- | ------------------------ |
| 규칙 기반      | 형식 오류·깨진 기능     | 의미 품질은 놓칠 수 있음 |
| 시각적·환경    | 실제 화면과 상태 불일치 | 검증 도구 준비 필요      |
| 추론 기반(LLM) | 어조·논리·누락·의도     | 지연·비용·평가자 오류    |

"계산형으로 잡을 수 있는 것은 계산형으로 먼저 잡고, 의미 판단이 필요한 부분에만
추론형을 붙인다."

### 회귀 평가와 힐클라이밍

실패 루프:

```text
실패 발생 → 원인 기록 → 테스트 케이스 추가 → 하네스 수정 → 재평가
```

Harnessability(하네스를 걸기 쉬운 환경)의 특징: 타입·스키마·입력 형식이 분명 /
모듈 경계와 책임이 뚜렷 / 테스트와 린터가 이미 존재 / 문서와 예시가 최신 /
로그와 오류 메시지가 친절.

## 안전·거버넌스·운영

### 권한 설계

| 행동                         | 정책      |
| ---------------------------- | --------- |
| 문서 읽기·요약               | 자동 허용 |
| 파일 수정·초안 작성          | 확인 요청 |
| 이메일 전송·결제·삭제        | 사람 승인 |
| 배포                         | 사람 승인 |
| 비밀키 노출·무단 데이터 추출 | 자동 차단 |

"AI에게는 '안 되는 것 빼고 다 해도 된다'보다 '허용된 것만 해도 된다'가 더
안전하다."

### Human-in-the-Loop

사람 승인이 필요한 항목: 돈이 이동하는 행동 / 외부 고객에게 나가는 메시지 /
데이터 삭제·변경 / 법적·의료적 판단 / 회사 평판에 영향주는 공개 게시.

### Prompt Injection 방어

외부 문서에 숨겨진 악성 지시가 AI를 원래 의도와 다르게 행동하게 하는 공격. 방어
원칙:

- 외부 콘텐츠를 신뢰할 수 없는 입력으로 취급
- 민감 행동은 별도 정책 엔진과 사람 승인 거침
- 도구 권한을 작업 범위에 맞게 최소화
- 로그와 trace로 이상 행동 추적

### 품질 Drift와 Garbage Collection

품질 drift: 문서 노후화 / 정책 변경 / 모델 발전 / 도구 추가로 시간이 지나면서
하네스가 낡는 현상.

"Keep quality left" — 빠르고 값싼 검사를 가능한 한 앞에 배치:

| 시점    | 검사 내용                             |
| ------- | ------------------------------------- |
| 작업 전 | 지시 파일·Skill·금지 규칙·출력 형식   |
| 작업 중 | 린터·빠른 테스트·스키마 검사          |
| 통합 전 | 전체 테스트·보안 검사·커버리지 확인   |
| 통합 후 | 비싼 리뷰·성능 테스트·장기 drift 감지 |

### 자동화

#### 패턴 12. 결정론적 수명 주기 훅

"매번 일어나야 하는 일은 지침이 아니라 훅에 있어야 한다."

예시: 파일 수정 뒤 포매터 자동 실행 / 고위험 명령 전 승인 확인 / 새 세션 시작 시
규칙 파일 다시 로드.

### 12가지 패턴 적용 순서

1. 영구 지침 파일 만들기
2. 탐색-계획-실행 기본 습관화
3. 단일 목적 도구 몇 개만 만들기
4. 명령 위험 분류로 allow / ask / deny 정하기
5. 작업이 길어지면 계층형 메모리와 문맥 압축 추가
6. 작업량이 커지면 하위 에이전트와 병렬성 추가
7. 마지막으로 수명 주기 훅으로 반복 절차 시스템화

## 7가지 설계 결정과 3가지 역발상

### 7가지 결정

| 결정          | 선택지 A       | 선택지 B           | 판단 기준                          |
| ------------- | -------------- | ------------------ | ---------------------------------- |
| 에이전트 수   | 단일 에이전트  | 멀티 에이전트      | 복잡도와 분리 필요성               |
| 추론 전략     | ReAct          | Plan-and-Execute   | 탐색적 vs. 반복적                  |
| 컨텍스트 전략 | 강한 압축      | 풍부한 컨텍스트    | 실패 패턴 보인 후 압축 설계로 이동 |
| 검증 방식     | 계산적(테스트) | 추론적(LLM 평가자) | 계산형 먼저, 의미 판단만 추론형    |
| 권한 설계     | 허용적         | 제한적             | 개발 중엔 허용적, 운영엔 제한적    |
| 도구 범위     | 전체 상시 노출 | 단계별 최소 노출   | 항상 최소 노출 기본값              |
| 하네스 두께   | 얇은 하네스    | 두꺼운 하네스      | 고위험 작업은 두껍게, 탐색은 얇게  |

얇은 vs. 두꺼운 하네스:

| 구분          | 얇은 하네스                          | 두꺼운 하네스                             |
| ------------- | ------------------------------------ | ----------------------------------------- |
| 기본 철학     | 모델의 판단을 신뢰                   | 흐름을 코드로 명시                        |
| 장점          | 단순·유연·모델 발전 이점을 빨리 받음 | 예측 가능·감사 가능                       |
| 단점          | 모델 실수에 더 많이 노출             | 설계 무거움·새 모델 능력 가로막을 수 있음 |
| 어울리는 상황 | 탐색·창작·코드 보조                  | 규제 업무·결제·배포·고위험 작업           |

### 3가지 역발상

#### 역발상 1. 도구가 많을수록 좋은 에이전트인가? — 아니다

Vercel d0 사례: 전문 도구 제거 후 파일시스템 방식으로 단순화.

- 실행 시간: 274.8초 → 77.4초 (3.5배 개선)
- 성공률: 80% → 100%
- 토큰 사용량: 37% 감소

"도구를 줄인 것이 아니라 작업 환경을 읽기 좋게 만든 것이다."

#### 역발상 2. ReAct가 항상 더 현대적인가? — 업무 특성에 따라 다르다

LLMCompiler 연구: Plan-and-Execute 방식이 ReAct 대비 최대 3.7배 지연시간 개선,
최대 6.7배 비용 절감.

#### 역발상 3. 허용적인 하네스가 더 빨리 출시되는가? — 개발 중에는 그렇지만 운영에서는 다르다

권한 설계 3단계:

- 낮은 위험(파일 읽기·검색·테스트 실행): 자동 허용
- 중간 위험(파일 수정·내부 문서 업데이트): 로그 + 선택적 확인
- 높은 위험(배포·결제·DB 삭제·외부 발송): 사용자 승인 필요

**비계(Scaffolding) 원칙**: "비계는 건물을 짓지 않는다. 하지만 비계 없이는
작업자가 위층에 닿지 못한다."

잘 설계된 하네스는 모델을 가두는 감옥이 아니라, 모델과 제품이 성장하면 일부를
걷어낼 수 있는 비계다.

## ACI와 에이전트-퍼스트 엔지니어링

Agent-Computer Interface(ACI): 언어 모델 에이전트도 자기 능력과 한계에 맞춘
인터페이스가 필요하다.

| 나쁜 인터페이스              | 좋은 인터페이스                   |
| ---------------------------- | --------------------------------- |
| 검색 결과가 너무 많이 쏟아짐 | 필요한 범위만 좁혀서 보여 줌      |
| 파일 전체를 무작정 읽게 함   | 필요한 위치와 주변 맥락을 보여 줌 |
| 수정 후 오류를 늦게 발견함   | 수정 직후 오류를 알려 줌          |
| 테스트 실패 원인이 불분명    | 실패 위치와 원인을 읽기 쉽게 제공 |

SWE-agent: pass@1 12.5%, HumanEvalFix 87.7% 달성.

하네스 문제 재진단:

| 문제가 보이는 방식                 | 하네스 관점의 질문                      |
| ---------------------------------- | --------------------------------------- |
| 모델이 회사 정책을 반영하지 못했다 | 정책 문서가 컨텍스트에 있었는가?        |
| 도구를 잘못 사용했다               | 도구 설명이 명확했는가?                 |
| 이전 작업을 잊었다                 | 상태 기록이나 progress 파일이 있었는가? |
| 테스트 없이 완료했다고 말했다      | 검증 루프가 하네스에 있었는가?          |
| 위험한 행동을 시도했다             | 권한·승인·샌드박스가 설계되었는가?      |

UI/UX의 세 가지 화면:

| 관점             | 일상 비유              | 설계해야 할 것                                          |
| ---------------- | ---------------------- | ------------------------------------------------------- |
| Human UX         | 손님용 메뉴판          | 요청·검토·승인·취소·수정 경험                           |
| Agent Experience | 직원용 레시피와 주문표 | 에이전트가 상태·의미·다음 행동·오류를 읽을 수 있는 구조 |
| Operator UX      | 점장용 대시보드        | 실행 상태·비용·실패·승인 대기·작업 기록                 |

최소 하네스:

1. `progress.md`: 매 세션 시작 때 읽고, 끝날 때 업데이트
2. `feature_list.json`: 완료 기준을 검증 가능한 항목으로 나눔
3. git commit: 세션 종료 시 깨끗한 상태로 커밋
4. 테스트·린터·브라우저 검증: 에이전트가 결과를 직접 확인

## 하네스와 메모리 소유권

"기억을 소유하지 못하면, 에이전트를 소유하기 어렵다."

단기 vs. 장기 메모리:

| 구분        | 쉬운 설명                     | 예시                                          | 위험                                    |
| ----------- | ----------------------------- | --------------------------------------------- | --------------------------------------- |
| 단기 메모리 | 지금 책상 위에 펼쳐진 자료    | 현재 대화·최근 도구 출력·방금 읽은 파일       | 너무 많으면 모델이 혼란스러워짐         |
| 장기 메모리 | 서랍이나 창고에 보관하는 기록 | 사용자 선호·프로젝트 규칙·과거 결정·실패 사례 | 소유권이 없으면 이전·감사·삭제가 어려움 |

4가지 메모리 소유권 모델:

| 모델               | 특성                             | 주의점                                    |
| ------------------ | -------------------------------- | ----------------------------------------- |
| 내가 소유한 하네스 | 이식성 높음, 구현 부담 큼        | 모델 교체 시 메모리 유지 가능             |
| 상태형 제공자 API  | 단기 스레드 등이 API 안에        | 다른 모델로 이전에 제약 생길 수 있음      |
| 폐쇄형 블랙박스    | 내부 구조 알기 어려움            | 전환 시 무엇을 옮겨야 하는지 모를 수 있음 |
| 관리형 하네스      | 가장 빠르게 시작, 운영 부담 낮음 | 내보내기·감사 로그·삭제 정책 반드시 확인  |

폐쇄형 하네스 사용 시 점검 질문:

| 점검 질문                   | 좋은 신호                        | 위험 신호                 |
| --------------------------- | -------------------------------- | ------------------------- |
| 메모리는 어디에 저장되는가? | 내 DB·파일·표준 내보내기 가능    | 제공자 내부 상태에만 저장 |
| 다른 모델로 옮길 수 있는가? | 모델 독립적 메모리 API·파일 기반 | 특정 모델·스레드에 종속   |
| 삭제와 수정이 가능한가?     | 사용자별 삭제·rollback 지원      | 삭제 정책 불분명          |
| 압축 후 무엇이 남는가?      | 압축 정책과 요약본 확인 가능     | 무엇이 사라졌는지 모름    |

메모리 소유권 실무 원칙:

1. 장기 메모리는 가능하면 내가 소유한 저장소에 둠
2. 메모리 스키마를 문서화 (`memory_id`, `scope`, `type`, `content`, `source`,
   `confidence`, `created_at`)
3. 메모리에도 권한을 둠 (개인·프로젝트·조직·고객별 다른 권한)
4. 에이전트가 메모리를 수정할 때 근거를 남김
5. 압축과 요약을 평가 대상으로 삼음

### 사례별 핵심 교훈

#### OpenAI Codex — 약 5개월, 약 100만 줄 코드, 약 1,500개 PR, 3명의 엔지니어

핵심 실패: 거대한 `AGENTS.md` → 저장소 자체를 에이전트가 읽고 검증하고 수정할 수
있는 기준 기록 시스템으로 전환. "사람이 일을 덜 하는 것이 아니라, 사람이 일하는
층이 바뀐다. 직접 실행에서 환경 설계로."

#### Anthropic long-running agents — initializer agent → coding agent 구조

최소 인계 구조: `goal.md` / `feature_list.json` / `progress.md` / `runbook.md` /
`decisions.md`. "긴 작업을 하는 에이전트에게 가장 중요한 것은 기억력이 아니라
인수인계다."

#### Vercel d0 — text-to-SQL 에이전트

Claude가 YAML·Markdown·JSON 파일 직접 읽기, `grep`·`cat`·`ls` 사용으로 단순화.
실행 시간 274.8초→77.4초, 성공률 80%→100%, 토큰 37% 감소. "도구가 많은 하네스가
강한 하네스는 아니다."

#### Anthropic Research 멀티에이전트 — lead agent + 병렬 subagent

내부 평가에서 단일 Claude Opus 4 대비 90.2% 성능 우위. "멀티 에이전트는 여러
방향을 동시에 조사해야 할 때 의미가 있다."

#### SWE-agent ACI — pass@1 12.5%, HumanEvalFix 87.7%

"좋은 에이전트를 만들려면 모델에게 좋은 작업대를 줘야 한다."

#### GitHub Copilot coding agent — Issue 할당 → 백그라운드 작업 → draft PR → 사람 리뷰

Copilot은 자신의 PR을 Ready for review로 표시하거나 승인·병합 불가(권한 설계).
"에이전트가 일하게 하려면 검토와 승인 흐름 안에 넣는 것이 중요하다."

#### Cursor Background Agents — 격리된 Ubuntu 기반 머신, 인터넷 접근 가능

위험: prompt injection, data exfiltration. "자동 실행은 편리하지만, 그만큼
권한과 감시가 중요해진다."

#### Microsoft Magentic-One — Orchestrator + Task Ledger + Progress Ledger

전문 에이전트: WebSurfer, FileSurfer, Coder, ComputerTerminal. "멀티 에이전트의
핵심은 에이전트 수가 아니라, 조율 구조다."

#### Replit Agent — `.replit`과 `replit.nix`로 실행 명령·의존성·배포 설정 구성

"아이디어를 결과물로 바꾸려면, 모델보다 실행 환경이 먼저 준비되어야 한다."

#### Claude Agent Skills — `SKILL.md` + YAML metadata + scripts + templates

"반복해서 설명하는 업무는 프롬프트가 아니라 Skill로 만들어야 한다."

### 부록 A. 핵심 용어

하네스 엔지니어링에서 반드시 알아야 할 개념어 목록이다. 각 용어는 일상 비유·쉬운
설명·주의점의 형태로 제공된다.

#### 제어 메커니즘

| 용어        | 설명                                                               |
| ----------- | ------------------------------------------------------------------ |
| Feedforward | 일 시작 전 실패 가능성을 줄이는 사전 제어                          |
| Feedback    | 결과를 보고 수정하게 하는 사후 제어                                |
| Guardrail   | 위험하거나 잘못된 행동을 막는 규칙                                 |
| Hook        | 특정 이벤트에서 자동 실행되는 절차                                 |
| Middleware  | 모델 호출 전후나 도구 실행 전후에 끼어들어 검사·수정·기록하는 장치 |

#### 평가 체계

| 용어                 | 설명                                                                  |
| -------------------- | --------------------------------------------------------------------- |
| Computational Sensor | 테스트·린터·타입 검사처럼 빠르고 결정적인 검사                        |
| Inferential Sensor   | LLM 리뷰나 의미 평가처럼 맥락을 읽고 판단하는 검사                    |
| Grader               | 평가 점수를 매기는 함수 또는 모델                                     |
| Rubric               | 좋은 결과와 나쁜 결과를 구분하는 기준표                               |
| Regression Fixture   | 모델이나 Skill이 바뀌어도 기대 행동을 유지하는지 확인하는 테스트 세트 |

#### 메모리·컨텍스트

| 용어                   | 설명                                                                  |
| ---------------------- | --------------------------------------------------------------------- |
| Context Rot            | 오래된 대화와 불필요한 정보가 쌓여 모델 판단을 흐리게 하는 현상       |
| Progressive Disclosure | 처음에는 이름과 설명만 보이고, 필요할 때 본문을 읽는 방식             |
| Compaction             | 오래된 대화와 관찰을 짧게 압축하는 방식                               |
| Context Boundary       | 어떤 정보를 함께 봐야 하고 어떤 정보는 따로 떼어도 되는지 나누는 기준 |
| Fire-and-forget        | Subagent처럼 일을 맡기고 완료 결과만 회수하는 방식                    |

#### 운영

| 용어               | 설명                                                                           |
| ------------------ | ------------------------------------------------------------------------------ |
| Harness Entropy    | 에이전트가 빠르게 만든 임시 코드와 낡은 규칙이 쌓이며 하네스가 복잡해지는 현상 |
| Harness Thickness  | 로직을 모델에 맡길지, 코드로 통제할지의 정도                                   |
| Harnessability     | 하네스를 걸기 쉬운 환경의 특성 (타입·스키마 명확, 모듈 경계 뚜렷, 테스트 존재) |
| Garbage Collection | 더 이상 필요 없는 임시 구조·오래된 지시·중복 문서를 정리하는 운영 루프         |
| Artifact           | 계획서·스크린샷·테스트 결과 같은 검토 가능한 산출물                            |

#### UX

| 용어                  | 설명                                                                  |
| --------------------- | --------------------------------------------------------------------- |
| Human UX              | 사람이 에이전트에게 일을 맡기고, 승인·중단·수정할 수 있게 하는 경험   |
| Agent Experience (AX) | 에이전트가 화면·상태·오류·로그를 의미로 읽고 행동할 수 있게 하는 경험 |

### 부록 D. AI의 기억을 설계하는 법

메모리 설계의 핵심 원칙 8가지:

1. **기억은 하네스 기록장**: AI의 기억은 모델 내부가 아니라 하네스가 관리하는
   기록 시스템
2. **선택적 저장**: 핵심 질문은 "무엇을 저장하지 않을 것인가?"
3. **목차 기반 접근**: 메모리 목차(어떤 기억이 있는지)와 메모리 본문(세부
   규칙)을 분리
4. **정기적 재확인**: 파일 존재 여부·최신 계약과의 충돌·최신 템플릿 변경 사항을
   주기적으로 점검
5. **이전 시 규칙 포함**: 파일만 이전해서는 부족. 스키마·범위·우선순위·검증
   규칙도 포함해야 함
6. **범위별 분리 관리**: 개인 기억 / 프로젝트 기억 / 조직 기억 / 임시 기억 4분류
7. **사용자 승인 초기화**: 초기 기억은 자동 확정이 아니라 사용자 승인으로 시작
8. **보안 중심 장기 관리**: 기존 지시 무시 문장·토큰·외부 지시는 저장 차단

저장하면 좋은 기억: 반복되는 사용자 선호 / 프로젝트 지속 규칙 / 과거 실패 원인 /
재설명 비용이 큰 업무 방식.

저장하지 않을 기억: 일회용 임시 요청 / 문서에 명시된 일반 정보 / 확인되지 않은
추측 / 곧 변할 가능성 높은 선택.

기준: "이 기억이 미래 사용자 시간을 줄여 주는가?"

메모리 스키마 예시: `memory_id`, `scope`, `type`, `content`, `source`,
`confidence`, `created_at`.

### 부록 E. Subagent와 Agent Teams를 구분하는 법

#### 핵심 차이

| 구분             | Subagent          | Agent Teams           |
| ---------------- | ----------------- | --------------------- |
| 기본 성격        | fire-and-forget   | collaborative         |
| 에이전트 간 대화 | 없음              | 있음                  |
| 공유 메모리      | 없음              | 공유 작업 목록 있음   |
| 지속 상태        | 거의 없음         | 시간과 함께 문맥 축적 |
| 부모의 역할      | 위임 후 요약 수신 | 팀 운영 및 결과 종합  |

Subagent는 "독립적 조사형": 각 작업이 따로 끝나고 결과만 합치면 됨. Agent
Teams는 "지속 조율형": 한쪽 결정이 다른 쪽을 계속 변경하고 실시간 상태 공유가
필요.

#### 선택 기준

| 질문                              | Subagent가 맞는 경우    | Agent Teams가 맞는 경우           |
| --------------------------------- | ----------------------- | --------------------------------- |
| 작업이 서로 독립적인가?           | 최종 결과만 합치면 충분 | 한쪽 결과가 다른 쪽 방향을 변경   |
| 중간 발견을 즉시 공유해야 하는가? | 최종 결과만으로 충분    | 중간 발견을 팀원이 바로 알아야 함 |
| 상태 추적이 필요한가?             | 공유할 필요 없음        | 의존성과 차단 상태 계속 확인      |

#### 가장 흔한 실수: 역할 이름으로 나누기

잘못된 방식: "계획 담당", "구현 담당", "테스트 담당"으로 구분. 올바른 방식: 문맥
경계로 나누기 — 두 작업이 같은 정보를 깊게 필요로 하면 한 에이전트가 이어서
담당.

#### 멀티에이전트를 피해야 할 상황

1. 좋은 프롬프트 하나로 충분할 때
2. 에이전트들이 계속 서로의 문맥을 물어봐야 할 때
3. 의존성 관리 비용이 실행 이득보다 클 때
4. 여러 에이전트가 동시에 같은 파일을 수정해야 할 때
5. 결과 개선 측정 방법이 없을 때

실전 판단 절차: 하나의 에이전트로 먼저 해결 시도 → 어디서 실패하는지 확인 →
문맥이 너무 커지면 Subagent로 격리 → 독립 조사가 많으면 Subagent 병렬 사용 →
중간 상태 조율 필요 시 Agent Teams 검토.

### 부록 O. 한 장 치트시트

하네스 7요소:

| 요소        | 확인 질문                                     |
| ----------- | --------------------------------------------- |
| 목표        | 최종 결과물은 무엇인가?                       |
| 컨텍스트    | 어떤 문서와 정보를 먼저 줄 것인가?            |
| 도구        | 어떤 도구가 필요하고, 어디까지 허용할 것인가? |
| 중간 산출물 | 최종 전에 무엇을 먼저 확인할 것인가?          |
| 검증        | 무엇을 만족해야 괜찮은 결과인가?              |
| 권한과 승인 | 발송·게시·제출·삭제 전에 누가 확인하는가?     |
| 기록과 개선 | 무엇을 저장하고 다음번에 고칠 것인가?         |

실패 신호 5가지:

| 신호                            | 의심할 지점                  |
| ------------------------------- | ---------------------------- |
| 같은 요청인데 결과가 매번 다름  | 목표와 기준이 흐릿함         |
| 답은 그럴듯한데 믿기 어려움     | 근거 자료와 검증 기준이 부족 |
| 최종 결과 고치느라 시간이 더 듦 | 중간 산출물이 없음           |
| 위험한 행동을 그냥 진행하려 함  | 권한과 사람 승인 지점이 없음 |
| 다음번에도 같은 실수 반복       | 기록과 개선 루프가 없음      |

작은 하네스 시작 순서:

1. 반복되는 일 하나를 고른다
2. 최종 결과물을 한 문장으로 적는다
3. AI에게 먼저 줄 자료를 모은다
4. 중간 산출물 하나를 정한다
5. 검증 기준 3개를 적는다
6. 사람 승인 지점 하나를 정한다
7. 실행 후 다음번에 고칠 점 하나를 남긴다

마지막 핵심 질문: "나는 지금 AI에게 답만 요구하고 있는가, 아니면 AI가 일할
환경을 만들어 주고 있는가?"

## 분석

### "하네스"라는 프레임의 실용적 가치

저자의 가장 중요한 기여는 책임의 재배치다. "AI가 실패했다"는 진단에서 "하네스가
잘못 설계되어 있다"는 진단으로 이동하면, 문제는 해결 불가능한 모델 선택 문제에서
해결 가능한 엔지니어링 과제로 전환된다.

컨텍스트 창이 부족한 것은 메모리 시스템 설계 문제다. 에이전트가 잘못된 도구를
호출하는 것은 도구 설계 문제다. 결과를 신뢰할 수 없는 것은 평가 하네스의 부재
문제다. 세션이 바뀌면 모든 것을 잊는 것은 인계 구조의 부재 문제다.

### 패턴 카탈로그로서의 의미

GoF의 "Design Patterns"(1994)처럼, 반복되는 문제에 이름을 붙이고 해결책을
표준화하는 접근이다. AI 에이전트 분야는 아직 표준화된 패턴 언어가 발달하지
않았다. 각 조직이 "에이전트"를 각자 다르게 정의하는 현재 상황에서, 이 12가지
패턴은 팀 간 의사소통과 지식 이전의 기반이 될 수 있다.
[코딩 에이전트의 구성 요소](./components-of-coding-agent.md)에서 Raschka가
제시한 6가지 구성 요소(라이브 저장소 문맥, 프롬프트 캐시, 도구 접근, 문맥 압축,
세션 메모리, 하위 에이전트 위임)는 이 12 패턴과 상당 부분 겹치며, 두 분류를
교차 참조하면 어느 패턴이 산업 전반에서 합의를 얻고 있는지 식별할 수 있다.

### 메모리 소유권이 락인으로 이어지는 구조

14장의 메모리 소유권 분석은 이 책에서 가장 독창적인 기여다. 사용자 선호·조직의
의사결정 방식·과거 실패 접근법이 쌓이는 메모리는 제품의 차별화 자산이다.
제공자의 블랙박스 메모리를 쓰는 팀은 기능이 아니라 과거의 축적을 통째로 빌리는
것이다. 이는 모델 성능이 아니라 기억의 전환 비용을 통한 강력한 락인이 된다.

### harness-lab 실습 도구의 설계 철학

`harness-lab`은 자동화 도구가 아니라 "사용자의 일을
Agent·Skill·Orchestrator·Test·Evolution 구조로 바꿔 보게 돕는 연습용 스킬"이다.
하네스 성숙도를 좋은 질문 → 체크리스트 → 역할 나누기 → 테스트 → 청사진 → 실행
하네스로 점진적으로 높이는 구조다. "처음부터 완성된 시스템을 만들 필요 없이 한
계단씩"이라는 저자의 교육 설계 철학을 반영한다.

## 비평

### 강점: 개념적 일관성과 실증된 수치

15개 챕터 전체를 "하네스"라는 단일 메타포로 관통한다. Vercel d0(실행 시간 3.5배
개선·성공률 100%)·Anthropic Research(90.2% 성능 우위)·SWE-agent(pass@1
12.5%)처럼 외부 검증된 수치를 함께 제시해 신뢰도가 높다.
`AGENTS.md`·Skills·Hooks·Subagents·MCP·권한·평가·메모리 소유권까지 2025~2026년
실무를 반영한 구성이다. 개발자가 아니어도 읽을 수 있도록 일상 비유를 철저히
사용하면서도 내용이 가볍지 않다.

### 한계: 메타포의 단순화와 빠른 낡음

"메모리 = 파일 시스템"이라는 메타포는 AI 메모리 시스템의 확률적 특성·컨텍스트
의존성·검색 관련성 문제를 가린다. 입문자에게는 접근 가능한 메타포가 정확한
모델보다 가치 있을 수 있지만, 실무에서 부딪히는 엣지 케이스를 예측하기는 어렵다.
[하네스 엔지니어링 종합 보고서](./harness-engineering-comprehensive-report.md)는
이 단순화의 깊은 정당성을 “에이전트 하니스는 DB·VCS의 검증된 패턴을 LLM 루프에
이식한 것”이라는 관점으로 변호하지만, 동시에 그 이식 과정에서 발생하는
비결정성·환각 같은 LLM 고유 문제는 메타포가 가리는 부분이라는 한계도 함께
드러난다.

"v1.84"라는 판 번호가 시사하듯, 특정 API 이름·도구 목록·구현 세부사항은 빠르게
낡는다. 원칙은 유효하지만 구현 예시는 직접 확인이 필요하다. OpenAI와 Claude 두
플랫폼을 모두 다루는 것이 강점이지만, 두 플랫폼을 동시에 비교하다 보니 어느 쪽도
완전히 깊게 다루지 못하는 부분이 있다.

## 인사이트

### 질문의 전환이 엔지니어링의 전환이다

"어떤 모델이 더 좋은가?"와 "이 모델이 일하는 환경이 제대로 설계되어 있는가?"는
완전히 다른 질문이다. 전자는 구매 결정이고, 후자는 엔지니어링 결정이다. 조직의
역할도 바뀐다. "어떤 AI를 쓸 것인가"를 결정하는 것에서 "AI가 일하는 환경을
어떻게 설계할 것인가"를 결정하는 것으로.

### 도구가 많을수록 좋다는 직관은 틀렸다

Vercel d0 사례가 보여주는 역발상이 충격적이다. 전문 도구를 제거하고 파일시스템
방식으로 단순화했더니 실행 시간이 3.5배 빨라지고 성공률은 100%가 됐다. 도구는 AI
대신 생각하기 위한 것이 아니라 AI가 더 잘 관찰하고 행동하도록 돕는 장치여야
한다. 설계가 단순할수록 에이전트가 읽기 쉬운 환경이 된다.

### "다음 모델도 읽을 수 있게 남긴다"는 원칙의 범용성

Ralph Loop의 핵심 원칙은 AI 에이전트를 넘어 팀 협업 전반에 적용된다. 기억력이
아니라 인계 구조가 장시간 작업의 품질을 결정한다.
`progress.md`·`feature_list.json`·git history의 조합은 "다음 에이전트(또는 다음
사람)가 어디서부터 시작해야 하는지 명확하게 알 수 있게 한다"는 목표를 구조로
구현한 것이다.
[장기 실행 에이전트](./long-running-agents.md)에서 정리한 “장기 수평 추론 / 장기
실행 / 지속적 에이전시”의 3분류 중 두 번째에 정확히 대응하는 패턴이며,
대부분의 “장기 실행” 문제는 모델 역량이 아니라 인계 구조의 엔지니어링 문제임을
이 책과 함께 강하게 시사한다.

### 메모리 소유권은 제품 전략이다

에이전트의 메모리에는 사용자 선호·조직의 의사결정 방식·과거 실패 접근법이
쌓인다. 이것을 외부 제공자의 블랙박스에 맡기면, 모델이 아니라 기억의 축적이 락인
요인이 된다. "어떤 AI를 쓸 것인가"보다 "AI의 메모리를 누가 소유할 것인가"가
장기적으로 더 중요한 설계 결정일 수 있다.

### 하네스는 모델이 좋아질수록 더 중요해진다

맺음말의 핵심 역설: "AI 모델은 계속 좋아질 것이다. 하지만 모델이 좋아질수록
'모델을 어떻게 둘러싸는가'가 더 중요해진다." 좋은 모델에 엉성한 하네스를 씌우면
위험한 장난감이 된다. 모델의 발전은 하네스 설계 능력의 중요성을 줄이는 것이
아니라 오히려 높인다. 경쟁력은 "어떤 모델을 쓰는가"뿐 아니라 "그 모델에게 어떤
하네스를 제공하는가"에서 나온다.
