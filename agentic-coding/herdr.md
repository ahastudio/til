# Herdr: AI 코딩 에이전트 멀티플렉서 터미널

<https://herdr.dev/>

<https://github.com/ogulcancelik/herdr>

HN 토론: <https://news.ycombinator.com/item?id=48714802> (164점, 104개 댓글)

## 소개

Herdr는 여러 AI 코딩 에이전트를 하나의 터미널에서 통합 관리하는
멀티플렉서다. “모든 코딩 에이전트를 하나의 터미널에서 실행하라.
누가 막혔고, 작업 중이고, 끝났는지 한눈에 확인하라(run all your
coding agents in one terminal. see who's blocked, working, or done
at a glance)”는 문구로 프로젝트를 소개한다. 각 에이전트는 시뮬레이션이
아닌 독립적인 실제 터미널을 할당받기 때문에 풀스크린 TUI도 올바르게
렌더링되며, 워크스페이스와 탭 단위로 구성할 수 있다.

핵심 기능은 에이전트 인식 상태 표시로, 각 에이전트가 차단됨(blocked,
🔴), 작업 중(working, 🟡), 완료(done, 🔵), 유휴(idle, 🟢) 중 어떤
상태인지 실시간으로 보여준다. 실제 터미널 환경 위에서 동작하고,
마우스 클릭·드래그를 지원하는 네이티브 인터페이스를 제공하며, 파일
분할과 워크스페이스 관리, 세션을 분리(detach)했다가 다시 연결
(reattach)할 수 있는 지속성을 갖췄다. 단일 바이너리 크기는 약
10MB이며, 별도 의존성 없이 Linux·macOS에서 동작하고 Windows는
베타 상태로 지원된다. Claude Code, OpenAI Codex, Devin CLI, GitHub
Copilot CLI, Pi, Cursor, Kimi, Gemini 등이 검증된 통합 대상으로
명시되며, “커스텀 통합은 소켓 API를 통해 레이블과 상태를 보고할
수 있다(custom integrations can report labels and state over the
socket api)”는 설명으로 임의의 에이전트도 연동할 수 있다고 밝힌다.

설치는 `curl -fsSL https://herdr.dev/install.sh | sh` 스크립트,
Homebrew(`brew install herdr`), Nix(`nix run
github:ogulcancelik/herdr`), Mise(`mise use -g herdr`), Windows
베타용 PowerShell 스크립트 등 여러 경로로 제공된다. `herdr` 명령은
서버를 시작하거나 기존 워크스페이스에 연결하며, `herdr --remote
workbox`로 원격 서버에 접속하고, `herdr integration install
<agent>`로 에이전트별 네이티브 통합을 설치한다. 키바인딩은 Ctrl+B를
프리픽스로 삼아 `c`(새 탭), `v`/`-`(패널 분할), `Shift+N`(새
워크스페이스), `w`(워크스페이스 전환), `q`(detach, 에이전트는 계속
실행), `?`(전체 키맵 표시) 등을 제공한다.

기술적으로는 Rust로 90.1%가 작성됐고, MDX·Python·TypeScript·CSS·HTML이
나머지를 구성한다. 아키텍처는 Unix 소켓 기반 로컬 API와 클라이언트-서버
모델로, 백그라운드 서버가 연속적으로 실행되며 SSH를 통한 원격 접속을
지원한다(다만 인증이나 암호화 방식에 대한 구체적인 설명은 README에
없다). 소켓 API를 통해 에이전트가 워크스페이스를 직접 제어할 수
있고, `npx skills add ogulcancelik/herdr --skill herdr -g` 명령으로
에이전트용 스킬을 설치할 수 있다. 저장소는 977개의 커밋을 가지고
있으며, `cargo build --release`로 빌드하고 `just test`로 유닛
테스트를 실행한다.

라이선스는 오픈소스판(GNU AGPL v3.0 이상)과 상용판을 병행하는
듀얼 라이선스 구조를 취하며, 왜 이 구조를 택했는지에 대한 설명은
없다. 개발 배경이나 저자 소개도 따로 없고, 후원 섹션에 “전업으로,
공개적으로 만들고 있다(built full-time, in the open)”는 한 줄만
언급된다. 이 글을 작성하는 시점 기준 GitHub 스타 9,900개 이상,
포크 579개, 워처 24명, 이슈 38개, 릴리스 66개를 기록하고 있다.
README는 스스로를 tmux 및 Conductor·Cmux·emdash 같은 GUI 에이전트
매니저와 비교하며, tmux 대비 에이전트 인식 상태 표시와 마우스
네이티브 지원을, GUI 매니저 대비 실제 터미널 환경과 가벼운 바이너리
크기를 차별점으로 내세운다.

## 분석

### tmux와 GUI 매니저 사이의 빈 공간을 정확히 겨냥한 포지셔닝

Herdr의 기능 목록 자체는 새로운 것이 거의 없다. 세션 지속성과
detach/reattach는 tmux가 수십 년간 제공해 온 기능이고, 에이전트
상태를 시각적으로 보여주는 것은 Conductor나 Cmux 같은 GUI 도구들이
이미 하고 있는 일이다. Herdr의 실질적인 기여는 새로운 기능이
아니라, 이 두 카테고리 사이에 있던 빈틈을 정확히 짚어낸 포지셔닝에
있다. tmux는 에이전트라는 개념을 전혀 모르기 때문에 어떤 창이
막혀 있고 어떤 창이 작업 중인지 알려면 사용자가 직접 하나씩
확인해야 한다. 반면 GUI 매니저는 에이전트 상태를 예쁘게 보여주지만,
그 안에서 실행되는 것은 종종 시뮬레이션에 가까운 터미널이라
풀스크린 TUI 애플리케이션이 깨지는 경우가 많다. Herdr는 “실제
터미널”이라는 tmux의 강점과 “에이전트 인식”이라는 GUI 매니저의
강점을 하나로 합친 조합으로 스스로를 규정한다.

다만 HN에서는 이 포지셔닝 자체에 의문을 제기하는 목소리도 있었다.
며칠 써 본 한 사용자는 “conductor.build가 체감상 훨씬 낫다.
터미널 '안에서' 실행된다는 것은 과시용 기능이지 실제로 어떤
이점을 주지는 않는다”고 평가했다[^scubbo]. 또 다른 사용자는
“굳이 이렇게 복잡하게 갈 필요가 있나. 이 모든 걸 알아서 관리해
주는 하네스에 좋은 UI를 붙이면 되는 것 아닌가”라고 반문했다
[^hagen8]. 즉 “실제 터미널이라는 정체성”이 Herdr 스스로는 핵심
차별점으로 내세우지만, 실사용자 일부에게는 그 자체가 목적이 아니라
수단이라는 점에서 설득력이 약하다는 반응이 나온 셈이다.

반면 tmux를 직접 커스터마이징해 버티려던 사용자에게는 이 포지셔닝이
정확히 들어맞기도 했다. 일본의 한 개발자는 “cmux로 옮기지 않고
tmux를 커스터마이징해서 버텨 보려던 참에 Herdr를 발견했다.
AI 에이전트 네이티브 터미널 환경을 구현할 수 있어서 매우
만족스럽다”고 트윗했다[^helloyuki_]. tmux 커스터마이징이라는
대안 경로를 이미 고려하고 있던 사용자에게는, Herdr가 “실제 터미널
위에 에이전트 인식을 얹는다”는 조합이 새로 배울 도구가 아니라
커스터마이징 수고를 대신해 주는 지름길로 읽힌 셈이다. 이는 앞서
HN에서 제기된 “실제 터미널이라는 정체성이 실익 없는 과시”라는
비판과 정반대되는 반응으로, Herdr의 포지셔닝이 사용자가 어떤
대안을 이미 저울질하고 있었는지에 따라 설득력이 갈린다는 것을
보여준다.

같은 맥락에서 실사용자가 구체적인 사용 패턴을 공유한 트윗도
있었다. 한 사용자는 “Herdr의 `type = "pane"`을 쓰면 Agent 위에서
탭을 늘리지 않고도 페인을 열 수 있어 매우 편리하다.
Lazygit이나 Vim 같은 도구를 여기서 실행한다”고 소개했다
[^sorafujitani]. 이는 Herdr가 단순히 여러 에이전트를 나란히
배치하는 용도를 넘어, 하나의 에이전트 작업 공간 안에서 Lazygit·Vim
같은 보조 도구를 페인 형태로 붙여 쓰는 워크플로도 지원한다는
것을 보여준다. tmux 사용자에게 익숙한 “창 분할” 감각을 에이전트
탭 구조 위에 그대로 얹을 수 있다는 점에서, 앞서 helloyuki_가
언급한 “실제 터미널 위에 에이전트 인식을 얹는다”는 조합의
구체적인 활용 사례로 볼 수 있다.

일본어권 개발자 커뮤니티에서 영향력 있는 인물로 꼽히는 한
사용자도 짧게 긍정적인 평가를 남겼다. “herdr, 확실히 CLI에서
에이전트를 쓰는 사람에게는 좋다”는 트윗이었다[^naoya_ito]. 앞선
두 트윗이 구체적인 사용 경험이나 기능 활용법을 곁들인 반면, 이
반응은 “CLI로 에이전트를 다루는 사람”이라는 대상을 특정해
Herdr의 가치가 누구에게 성립하는지를 짧게 짚었다는 점에서
눈여겨볼 만하다.

가장 열띤 반응은 “우오오오오 Herdr 정말 좋다!!!!”라는 트윗으로
나타났다[^yusukebe_herdr]. 같은 사용자는 다음 날 곧바로 “그나저나
이 폰트가 마음에 든다(Herdr와는 무관) [UDEV Gothic NF](https://github.com/yuru7/udev-gothic)”라는 트윗을
덧붙였다[^yusukebe_font]. Herdr 자체와는 무관하다고 스스로 선을
그었지만, 터미널 스크린샷을 공유하며 이어진 반응이라는 점에서
에이전트 터미널 환경에 대한 관심이 도구 선택을 넘어 폰트 같은
주변 요소로도 자연스럽게 번지는 모습을 보여준다.

### 소켓 API 중심 설계는 도구 자체를 자동화 가능한 인프라로 재정의한다

일반적인 터미널 멀티플렉서는 사람이 직접 조작하는 것을 전제로
설계된다. Herdr가 Unix 소켓 기반 로컬 API를 중심에 두고, 심지어
에이전트가 이 API를 통해 스스로 워크스페이스를 제어할 수 있게
설계한 것은 중요한 차이다. 이는 “사람이 여러 에이전트를 관리하는
도구”에서 “에이전트들이 서로의 상태를 인지하고 조율할 수 있는
공유 인프라”로 이 도구의 성격이 확장될 수 있음을 시사한다. 후원
섹션에 등장하는 “진짜 에이전트 런타임으로 가는 길(the path to a
real agent runtime)”이라는 표현은 이 방향성을 암시하지만, 구체적인
로드맵은 제시되지 않는다. 즉 현재의 Herdr는 사람을 위한 관리
도구이지만, 설계 자체는 이미 에이전트 간 협업 인프라로 진화할
잠재적 경로를 열어 두고 있다.

### AGPL과 상용 라이선스의 병행은 인프라형 오픈소스 프로젝트의 전형적인 방어 전략이다

이 프로젝트가 GNU AGPL v3.0이라는, 카피레프트 조건 중에서도 특히
강력한 라이선스(네트워크를 통한 서비스 제공에도 소스 공개 의무가
적용되는)를 오픈소스판에 적용하면서 동시에 상용 라이선스를
병행하는 것은 우연이 아니다. AGPL은 클라우드 사업자가 이 코드를
가져다 자체 SaaS 형태로 재판매하면서 소스를 공개하지 않는 것을
막는 데 효과적인 라이선스로, MongoDB나 Grafana 같은 인프라형
오픈소스 프로젝트들이 유사한 방어 목적으로 채택해 온 전략과 같은
계보에 있다. 이는 이 프로젝트가 단순한 취미 도구가 아니라 향후
상업화(엔터프라이즈 지원, 클라우드 호스팅 등)를 염두에 둔 인프라
소프트웨어로 스스로를 포지셔닝하고 있다는 신호다.

## 비평

### “실제 터미널”이라는 핵심 차별점이 원격·보안 설명의 공백을 가린다

README가 가장 강조하는 차별점은 “시뮬레이션이 아닌 실제 터미널”이라는
것이다. 그러나 정작 이 실제 터미널이 SSH를 통한 원격 접속과
결합될 때 어떤 보안 모델을 따르는지에 대한 설명은 전혀 없다.
`herdr --remote workbox` 명령으로 원격 서버에 접속할 수 있다고
설명하면서도, 인증은 어떻게 이뤄지는지, 소켓 API에 대한 접근
권한은 어떻게 제한되는지, 원격 세션의 트래픽이 암호화되는지에
대한 언급이 없다. 여러 AI 에이전트가 실제 셸 접근 권한을 가진
채로 원격 워크스페이스에서 동작한다는 것은, 이 도구가 단순한 UI
편의 기능을 넘어 상당한 공격 표면을 가진 인프라라는 뜻이다. “실제
터미널”이라는 자랑이 기술적 장점을 부각하는 동시에, 그 장점이
수반하는 보안 책임에 대한 설명 부재를 가리는 효과를 내고 있다.

이 공백은 HN 토론에서 실사용자의 질문으로도 확인됐다. 한 사용자는
“각 에이전트가 서로 다른 SSH 세션에 있는 구성도 지원하는가, 아니면
모두 같은 곳에서 실행돼야 하는가”라고 물으며, 원격 Herdr 서버에
SSH로 접속하는 것은 지원되는 것 같지만 “에이전트마다 별도의
샌드박스에 설치돼 있어 먼저 그 안으로 SSH 접속한 뒤 에이전트를
시작하는” 구성이 가능한지는 문서만으로 알 수 없다고 지적했다
[^nok22kon]. 이런 질문에 README나 공식 문서가 명확히 답하지
못한다는 사실 자체가, 원격·분산 실행 시나리오에 대한 설명이
충분히 정리되지 않았음을 보여준다. 또 다른 사용자는 “한 프로젝트
안에서 여러 에이전트를 어떻게 쓰는가, 경쟁 조건(race condition)이
생기지 않는가, 아니면 이건 애초에 두 개의 별도 프로젝트를 동시에
다루기 위한 것인가”라고 물었는데[^fergusargyll], 이 역시 여러
에이전트가 동시에 같은 코드베이스를 건드릴 때의 충돌 방지 전략이
문서화되어 있지 않다는 것을 보여주는 질문이다.

### tmux 대비 우위를 주장하면서도 tmux 사용자가 이미 가진 자산과의 호환성은 다루지 않는다

Herdr는 세션 지속성, detach/reattach 같은 tmux의 핵심 기능을
자신도 제공한다고 주장하며 비교표까지 제시한다. 그러나 수년간
tmux 설정 파일과 플러그인, 커스텀 키바인딩, 스크립트 워크플로를
쌓아 온 기존 tmux 사용자가 Herdr로 전환할 때 무엇을 포기해야
하는지는 다루지 않는다. tmux는 광범위한 플러그인 생태계(tmux
resurrect, tmux-continuum 등)와 성숙한 스크립팅 인터페이스를 갖고
있는데, Herdr가 이런 기존 워크플로와 얼마나 호환되는지, 아니면
완전히 새로 배워야 하는지에 대한 설명이 없다. “tmux보다 낫다”는
주장은 기능 목록 비교로는 성립할 수 있어도, 실제 전환 비용을
고려한 비교로는 불완전하다.

### 듀얼 라이선스의 등장이 프로젝트 궤적에 대한 신호를 주지만 README는 이를 설명하지 않는다

977개의 커밋과 “전업으로 만들고 있다”는 표현은 이 프로젝트가
상당한 개인적 투자를 거쳐 만들어졌음을 시사한다. 그런데 왜
AGPL과 상용 라이선스를 병행하기로 했는지, 상용 라이선스가 정확히
무엇을 다르게 제공하는지(엔터프라이즈 기능, 지원, 아니면 단순히
AGPL 의무 면제인지)에 대한 설명이 없다. 오픈소스 사용자 입장에서는
“이 프로젝트가 언젠가 오픈소스판의 기능을 제한하고 상용판으로
유인하는 방향으로 갈 것인가”라는 질문에 답할 근거가 없다. 인프라형
도구를 프로덕션에 채택하려는 팀에게는 라이선스 정책의 방향성이
기능 목록만큼 중요한 판단 기준인데, 이 부분에 대한 투명성이
부족하다.

## 인사이트

### 에이전트 상태 시각화는 “여러 에이전트를 동시에 돌린다”는 워크플로가 이미 표준이 되고 있다는 증거다

Herdr가 해결하려는 문제, 즉 “누가 막혔고 누가 작업 중인지 한눈에
알기 어렵다”는 문제는 애초에 개발자가 여러 AI 에이전트를 동시에
병렬로 실행하는 것이 일상적인 워크플로가 됐을 때만 성립하는
문제다. 불과 얼마 전까지만 해도 코딩 에이전트는 한 번에 하나씩,
사람이 지켜보며 쓰는 도구였다. 이제는 여러 에이전트를 동시에
띄워 두고 사람이 “감독자” 역할로 오가며 확인하는 패턴이 충분히
보편화되어, 이를 위한 전용 상태 관리 도구가 등장할 만큼 시장이
성숙했다는 뜻이다. 이는 소프트웨어 개발에서 사람의 역할이 코드를
직접 작성하는 것에서 여러 병렬 작업의 진행 상황을 모니터링하는
것으로 옮겨가고 있다는, 더 큰 흐름을 보여주는 구체적인 증거다.

실제로 HN에서 한 사용자는 십여 개의 터미널 창과 탭으로 관리가
엉망이 됐던 상황에서 Herdr를 몇 주간 써 본 경험을 상세히 전했다.
프로젝트별 워크스페이스와 탭으로 정리할 수 있게 됐고, 지속
프로세스로 백그라운드에서 돌아가기 때문에 Tailscale로 원격
머신에 SSH 접속해 활성 세션을 확인하고 디버깅할 수 있으며,
심지어 휴대폰이나 아이패드에서 Prompt·Termius 같은 앱으로도
같은 작업을 할 수 있다고 밝혔다[^ricardobeat]. 반면 다른 사용자는
정반대의 반응을 보였다. “사람들이 정말 여러 개의 에이전트가
필요할까? 프로젝트 두 개에 각각 Codex 세션을 하나씩 띄우는
것만으로도 벌써 벅차게 느껴지는데, 내가 나이가 든 건지도
모르겠다”고 털어놓았다[^mintflow]. 이 두 반응의 병존은 “여러
에이전트를 동시에 돌리는 것”이 아직 모두에게 보편화된 워크플로가
아니라, 특정 작업 스타일과 규모의 팀·개인에게만 명확한 가치를
주는 과도기적 관행일 수 있음을 보여준다.

### 소켓 API 기반 아키텍처는 향후 에이전트 간 직접 조율이라는 다음 단계를 예고한다

현재 Herdr의 소켓 API는 에이전트가 자신의 상태를 보고하는 데
쓰이지만, 이 설계가 한 걸음 더 나아가면 에이전트들이 서로의
상태를 참조해 스스로 작업을 조율하는 형태로 발전할 수 있다.
예컨대 한 에이전트가 특정 파일을 수정 중이라는 신호를 소켓을
통해 공유하면, 다른 에이전트가 그 파일을 건드리지 않고 대기하는
식의 자율적 조정이 가능해진다. “진짜 에이전트 런타임으로 가는
길”이라는 후원 문구가 이런 방향을 암시하는 것으로 보인다. 이렇게
되면 Herdr는 더 이상 “사람이 여러 에이전트를 감독하는 터미널”이
아니라, 여러 에이전트가 서로를 감독하고 조율하는 하부 런타임
계층으로 역할이 바뀔 수 있다. 이는 동시에 사람의 역할이 개별
작업 감독에서, 조율 규칙 자체를 설계하고 예외 상황에서만 개입하는
더 상위의 감독으로 다시 한 단계 물러나게 된다는 뜻이기도 하다.

### AGPL 기반 인프라 도구의 확산은 에이전틱 코딩 생태계에 상업화 압력이 본격화되고 있음을 보여준다

에이전틱 코딩 도구 생태계 초기에는 대부분의 도구가 순수 MIT나
Apache 라이선스로 공개되며 커뮤니티 확산에 무게를 뒀다. Herdr가
방어적 성격이 강한 AGPL과 상용 라이선스를 처음부터 병행하는
것은, 이 생태계가 이미 “무료로 퍼뜨려 생태계를 만드는” 단계를
지나 “누군가는 이 인프라로 돈을 벌 수 있다”는 것이 명확해진
국면에 접어들었음을 시사한다. 특히 에이전트 오케스트레이션처럼
여러 에이전트를 동시에 관리하는 인프라 계층은, 클라우드 사업자가
그대로 가져다 SaaS 형태로 재판매하기 좋은 성격을 가지고 있어
AGPL 같은 방어적 라이선스의 필요성이 실제로 크다. 앞으로 유사한
에이전트 오케스트레이션·관리 도구들이 늘어날수록, 초기 순수
오픈소스 문화에서 벗어나 처음부터 상업화 경로를 함께 설계하는
프로젝트가 표준이 될 가능성이 높으며, 이는 이 생태계가 실험
단계를 지나 실제 시장 가치를 인정받는 단계로 진입하고 있다는
신호로 읽을 수 있다.

### 동시다발적으로 등장한 유사 도구들은 문제의 해법이 아니라 문제의 보편성을 증명한다

HN 댓글 스레드 자체가 이 문제 공간이 얼마나 붐비고 있는지를
보여주는 증거였다. 댓글들 사이에서만 agent-of-empires[^crashabr],
beehive[^mst98], cmux.com[^bazhand], kandev[^iamfreee],
jmux[^jarredkenny], tcode[^wb14123] 같은 유사 목적의 대안 도구들이
저마다 다른 사용자에 의해 소개됐고, 심지어 Zellij에 직접 stop
hook을 붙여 에이전트가 끝났을 때 알림을 보내는 개인 스크립트를
소개한 사용자도 있었다[^joch]. 한 사용자는 Emacs와 agent-shell·
Projectile 조합만으로도 프로젝트 6개를 동시에 잘 다루고 있다며
전용 도구 없이도 충분하다는 반례를 제시하기도 했다[^isityettime].
이렇게 짧은 기간 안에 서로 독립적으로 유사한 해법이 여러 개
등장했다는 사실은, “여러 에이전트를 동시에
다루는 관리 문제”가 특정 회사나 개발자 한 명이 우연히 발견한
틈새가 아니라 이 시점의 개발자 커뮤니티 전반이 동시에 부딪히고
있는 보편적 문제라는 뜻이다. 역사적으로 이런 동시다발적 수렴은
특정 기술의 성숙 신호로 여겨져 왔다. 초기 웹 프레임워크
춘추전국시대나 정적 사이트 생성기 난립기처럼, 결국 몇 개로
수렴되거나 표준 인터페이스(에이전트 상태를 보고하는 공통
프로토콜 같은) 위에서 서로 경쟁하는 형태로 정리될 가능성이
높다.
