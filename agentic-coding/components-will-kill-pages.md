# Components Will Kill Pages

<https://bitsandbytes.dev/posts/components-will-kill-pages>

2026년 2월 11일, Bits & Bytes 블로그에 올라온 글.
AI가 인터넷 인터페이스의 패러다임을 페이지에서 컴포넌트로
바꿀 것이라고 주장한다.

## 한국어 번역

### 컴포넌트가 페이지를 죽일 것이다

인터넷이 변하고 있다. 부정할 수 없는 사실이다.
20년 넘게 유지되어 온 웹사이트라는 형태가 이제
급격하게 바뀌려 한다. 우리는 웹사이트를 하나하나,
페이지를 하나하나 탐색하는 방식에 익숙하다.
최근 ChatGPT 같은 도구가 등장하면서,
틱톡에 길들여진 우리의 주의력은 즉각적인 결과를
요구하고, ChatGPT는 정확히 그것을 제공한다.
창을 열고, 텍스트 입력란에 알고 싶은 것을 아무거나
입력하면 — 짠, 나에게 맞춤화된 답이 나타난다.
남아 있는 뇌세포들이 빠른 도파민 히트를 축하하며
잔을 부딪친다. 최소한의 노력으로 정보에 접근하는 것.
시대가 변하고 있다.

프론트엔드 엔지니어들은 오랫동안 사용자 여정(user journey)
수준에서 문제를 풀어왔다.
사용자가 페이지에서 페이지로 — 기능에서 기능으로 —
어떻게 이동하는지를 최적화해 왔다.
"가입 퍼널에 뭘 넣어야 하나",
"사용자를 가장 빨리 핵심 가치까지 도달시키려면 어떻게 해야 하나"
같은 질문을 20년간 던져왔다.
이제 나는 그 어떤 것도 중요하지 않게 될 것이라고 말한다.
당신의 제품에 가입하는 게 사용자일까, 아니면 사용자를
대신하는 AI일까?

ChatGPT가 사용자들이 인터넷과 상호작용하는 방식을
어떻게 바꾸고 있는지로 돌아오자.
텍스트 상자에 아무거나 입력하면 즉시 정보를 얻을 수 있는
세상에서, 우리는 웹사이트를 방문할 필요 자체를 우회하고 있다.
봇은 당신의 수많은 페이지를 계속 크롤링하겠지만,
사용자는 점점 더 게을러질 것이다.
AI 채팅 앱이 사용자를 외부 웹사이트로 내보내는 것은
당연히 불리하다.
X가 게시물 내 링크에 불이익을 주고, 모바일 앱 개발자가
앱 밖으로의 링크를 거의 넣지 않는 것처럼.
그렇다면 채팅 인터페이스를 통해 사용자에게 시각적
컴포넌트를 제시해서 브랜드와 상호작용하게 할 수 있는
방법이 있다면 어떨까?

Cloudflare를 예로 들어보자.
내가 플랫폼에 배포한 모든 Workflows를 AI가
시각화해 준다면 꽤 멋질 것이다
(단계, 재시도, try/catch 등... 진짜로).
AI가 인터넷을 소유하고 싶다면,
채팅 앱이 해당 웹사이트(예: Cloudflare)에서 노출하는
컴포넌트를 이해하고, 그것을 채팅 응답 안에
인라인으로 호출·렌더링하는 방법이 있어야 한다.
사용자를 앱 밖으로 리다이렉트할 필요도 없고,
최고 수준의 사용자 경험도 제공한다.
AI 채팅 앱도 이기고, 자사 제품의 이야기를
사용자에게 전달하는 방식의 소유권을 유지하는
브랜드도 이긴다.

json-render 같은 라이브러리와 Kumo 같은 컴포넌트
라이브러리를 사용하면, UI 컴포넌트를 JSON으로 표현해서
AI가 생성형 UI를 만드는 방법을 알 수 있게 된다.
애플리케이션은 Kumo의 컴포넌트 풀에서 어떤 컴포넌트가
사용 가능한지 파악하고, json-render를 이용해 AI는
어떤 컴포넌트가 children을 지원하고, 어떤 props가 있고,
어떻게 엮어서 고유한 응답을 만들 수 있는지 이해한다.
예를 들어 내가 앞서 언급한 워크플로를 시각화해 달라고
요청하고 Kumo에 FlowDiagram 컴포넌트가 있다면,
ChatGPT는 이 정보를 이용해 Cloudflare 브랜드가 입혀진
인터랙티브 워크플로 뷰어로 응답할 수 있다.

당신의 제품을 상상해 보라.
5년 뒤에도 사용자가 5페이지 깊이까지 들어가서
필터를 적용하고 데이터를 정렬해서 직접 답을 찾아내길
기대하지는 않을 것이다... 그렇지 않은가?
이 변화를 받아들이고, 사람들이 질문을 던지면
답을 얻을 수 있는 퍼스트파티 경험을 갖추길 바란다.
사용자는 몇 마디 말로 액션을 트리거하고, 몇 초 만에
결과를 얻는다.
AI가 인터넷의 새로운 핸들이 될 것이다.
페이지를 탐색하는 것이 아니다.

지금이 고품질 컴포넌트에 투자할 때다.
AI 퍼스트 컴포넌트 라이브러리를 쓰고 있지 않거나,
AI가 당신의 브랜딩으로 UI를 소비하거나 생성하는 방법을
생각하지 않고 있다면, 지금이 그때다.
(아직) 뒤처진 건 아니지만, 세상이 매달 바뀌는
오늘날에는 앞서 있는 편이 낫다.

## 요약

- ChatGPT 같은 AI 채팅 인터페이스가 웹 탐색의 주요 수단이
  되면서, 사용자가 개별 웹사이트의 페이지를 직접 탐색할 필요가
  사라지고 있다.
- AI 채팅 앱은 X나 모바일 앱처럼 사용자를 외부로 내보내고
  싶지 않다. 따라서 브랜드의 UI 컴포넌트를 채팅 안에서 직접
  렌더링하는 것이 양쪽 모두에게 유리하다.
- json-render와 Kumo 같은 도구를 쓰면 UI 컴포넌트를
  JSON으로 기술할 수 있고, AI가 이를 이해해서 생성형 UI를
  조합할 수 있다.
- 5년 후 사용자가 5페이지를 뒤져 정보를 찾을 것이라고
  기대하지 말라. 지금이 AI 퍼스트 컴포넌트에 투자할 때다.

## 분석

### 핵심 논지 구조

글의 논리는 세 단계로 진행된다:

1. **현상**: 사용자가 ChatGPT를 통해 정보를 얻기 시작했다.
2. **문제**: AI 앱은 사용자를 외부로 내보내지 않으려 한다.
   브랜드는 자기 이야기를 전달할 채널을 잃는다.
3. **해법**: 컴포넌트를 JSON으로 기술해서 AI가 채팅 안에서
   브랜드 UI를 렌더링하게 하라.

### 언급된 도구

| 도구        | 역할                                        |
|-------------|---------------------------------------------|
| json-render | UI 컴포넌트를 JSON 스키마로 표현하는 라이브러리 |
| Kumo        | AI가 이해할 수 있는 형태의 컴포넌트 라이브러리  |

## 비평

### 설득력 있는 부분

- **플랫폼의 폐쇄성에 대한 지적이 정확하다.**
  X가 외부 링크에 불이익을 주고, 모바일 앱이 인앱 브라우저를
  강제하는 것처럼, AI 채팅 앱도 사용자를 가두려 할 것이다.
  이 관찰은 이미 현실이다.
- **"페이지" 패러다임의 한계를 짚었다.**
  5페이지 깊이의 탐색, 필터 적용, 정렬이라는 기존 UX가
  자연어 한 문장으로 대체될 수 있다는 건 직관적으로 맞다.

### 약점과 반론

- **"누가 비용을 지불하는가"가 빠져 있다.**
  브랜드가 컴포넌트를 AI 플랫폼에 제공하면,
  AI 플랫폼이 트래픽(=광고 수익, 전환)을 가져간다.
  브랜드는 "이야기의 소유권"을 지킨다고 하지만,
  실질적으로는 사용자와의 직접 관계를 잃는다.
  Hacker News의 한 댓글이 정확히 이를 지적했다:
  "브랜드는 자기 이야기의 소유권을 지키지만,
  사용자를 잃는다. 남의 앱 안의 위젯 모음이 되는 것이다."
- **알림, 리텐션, 데이터 수집은?**
  페이지 기반 웹의 핵심은 단순한 정보 전달이 아니다.
  푸시 알림, 이메일 수집, 행동 분석, 개인화 —
  이 모든 것이 "자기 플랫폼"에서만 가능하다.
  컴포넌트가 남의 채팅 앱 안에서 렌더링되면,
  이 모든 수단을 잃게 된다.
- **JSON 스키마 표준화 문제를 과소평가했다.**
  ChatGPT, Gemini, Claude가 각각 다른 방식으로
  컴포넌트를 소비한다면, 브랜드는 각 플랫폼마다
  다른 스키마를 유지보수해야 한다.
  이건 "페이지를 만드는 것"보다 더 복잡할 수 있다.
- **모든 웹 경험이 "정보 조회"는 아니다.**
  Figma, Google Docs, Spotify 같은 도구형 제품은
  채팅 인터페이스 안의 컴포넌트로 대체할 수 없다.
  글이 다루는 범위가 정보 조회형 서비스에 한정되어 있다.

## 인사이트

### 1. 이것은 "마이크로프론트엔드의 극단적 확장"이다

마이크로프론트엔드가 하나의 웹앱 안에서 여러 팀의
컴포넌트를 조합하는 것이었다면, 이 글이 제안하는 것은
**여러 회사의 컴포넌트를 AI 플랫폼 안에서 조합하는 것**이다.
웹의 단위가 "페이지 → SPA → 마이크로프론트엔드 → 컴포넌트"로
점점 작아져 온 흐름의 자연스러운 다음 단계일 수 있다.

### 2. "앱 스토어 vs 웹" 전쟁의 재현

2010년대에 네이티브 앱이 웹을 대체할 것이라는 주장이 있었다.
결과적으로는 공존했다.
"AI 채팅이 페이지를 대체한다"는 이 주장도 같은 패턴을
따를 가능성이 높다.
일부 정보 조회형 경험은 AI 채팅으로 이동하지만,
복잡한 인터랙션, 창작 도구, 커뮤니티 경험은
독립적인 웹 앱으로 남을 것이다.

### 3. 진짜 승자는 "컴포넌트 표준"을 만드는 자

json-render, Kumo는 아직 작은 프로젝트다.
진짜 변곡점은 OpenAI, Google, 또는 W3C 수준에서
**"AI가 소비할 수 있는 UI 컴포넌트 기술(description)
표준"**이 등장할 때 올 것이다.
HTML이 웹의 공통어가 된 것처럼,
AI 시대의 UI 공통어가 무엇이 될지가 핵심 질문이다.

### 4. 프론트엔드 엔지니어의 역할이 바뀐다

이 글의 세계관이 실현되면,
프론트엔드 엔지니어는 "페이지를 만드는 사람"에서
**"AI가 조합할 수 있는 고품질 컴포넌트를 설계하는 사람"**
으로 변한다.
디자인 시스템 엔지니어의 가치가 급격히 올라간다.
반면 "페이지를 조립하는" 작업의 가치는 떨어진다.

### 5. 결국 "임베드 경제(Embed Economy)"다

YouTube 임베드, Twitter 임베드, Stripe 결제 위젯처럼
다른 플랫폼 안에 삽입되는 경험은 이미 존재한다.
이 글은 이것을 **모든 웹 경험으로 일반화하자**는
제안이다. 역사적으로 임베드 생태계에서는
플랫폼(YouTube, X)이 크리에이터보다 항상 더 큰
권력을 가졌다.
AI 플랫폼 안의 컴포넌트도 같은 권력 구조를 따를 것이다.
