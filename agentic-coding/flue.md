# Flue — 에이전트 하네스 프레임워크

> Flue — The Open Agent Framework

<https://flueframework.com/>

<https://github.com/withastro/flue>

## 소개

Flue는 Astro 팀(withastro)이 만든 TypeScript 기반 에이전트 하네스 프레임워크다.
“또 하나의 SDK가 아니다(Not another SDK)”라는 슬로건 아래,
자율 에이전트가 실제로 작동하는 데 필요한 환경 전체 — 세션, 도구, 스킬,
지시, 파일시스템 접근, 보안 샌드박스 — 를 제공하는 것을 목표로 한다.

7,000개 이상의 별을 받았으며 코드베이스의 91.1%가 TypeScript로 작성되어 있다.
라이선스는 Apache 2.0이다.

## 주요 기능

### 에이전트와 워크플로

에이전트는 `defineAgent` 함수로 정의한다.
모델, 도구 배열, 스킬, 샌드박스 설정, 지시를 명세하면
프레임워크가 세션과 컨텍스트를 관리하며 자율 실행을 이어간다.
워크플로는 에이전트보다 구조화된 자동화 경로로, 입력에서 출력까지의
흐름을 코드로 명확히 기술한다.

### 샌드박스와 내구성 실행(Durable Execution)

에이전트는 로컬, 가상, 원격 컨테이너 중 하나의 샌드박스에서 도구를 실행하고
파일을 수정한다.
내구성 실행은 장애나 재시작이 발생해도 진행 상태를 보존하므로
장시간 실행 작업에서 처음부터 다시 시작하지 않아도 된다.

### 스킬과 도구

스킬은 마크다운 파일로 패키징된 재사용 가능한 전문성이다.
에이전트가 실행 시점에 동적으로 불러오며,
도메인 특화 지식이나 행동 지침을 코드와 분리해 관리할 수 있다.
도구는 타입 안전한 API 호출 및 제어된 애플리케이션 변경을 정의한다.

### 서브에이전트와 채널

서브에이전트는 상위 에이전트로부터 위임받은 특정 역할을 수행하며,
복잡한 작업을 전문화된 단위로 분해할 수 있다.
채널은 Slack, Teams, Discord, GitHub에서 검증된 이벤트를 수신하는 통합 지점이다.

### 관측성과 배포

OpenTelemetry, Braintrust, Sentry를 통한 텔레메트리를 지원한다.
배포 타깃은 Node.js, Cloudflare Workers, GitHub Actions,
GitLab CI/CD, Daytona, Render다.

## 핵심 패키지

| 패키지                  | 역할                     |
| ----------------------- | ------------------------ |
| `@flue/runtime`         | 핵심 하네스, 세션, 샌드박스 관리 |
| `@flue/cli`             | 개발 도구 및 빌드 유틸리티 |
| `@flue/sdk`             | 배포된 에이전트 클라이언트 |
| `@flue/opentelemetry`   | 트레이싱 어댑터           |
| `@flue/postgres`        | 영속성 레이어             |

## 분석

### “SDK가 아닌 하네스”라는 포지셔닝의 의미

Flue가 스스로를 “SDK가 아닌 하네스”로 규정하는 것은 단순한 마케팅 문구가 아니다.
이 구분은 에이전트 프레임워크 시장의 현재 분열을 정확히 짚는다.
LangChain, CrewAI, AutoGPT 등 대부분의 기존 프레임워크는 LLM API 호출을
추상화하고 체인과 파이프라인을 구성하는 SDK로 출발했다.
에이전트가 실제로 작업하는 환경 — 세션 상태, 파일시스템, 샌드박스, 이벤트 채널 —
은 사용자가 직접 구성해야 했다.

Flue는 그 환경 자체를 프레임워크의 핵심으로 가져온다.
[하네스 엔지니어링](./harness-engineering.md)이 개념으로 제시한
“에이전트가 아니라 에이전트를 둘러싼 시스템이 성패를 결정한다”는 명제를
프레임워크 수준에서 구현하려는 시도다.

### 스킬의 마크다운 패키징이 갖는 구조적 함의

스킬을 마크다운 파일로 패키징하는 설계는 표면적으로는 단순해 보이지만
깊은 아키텍처적 판단을 담고 있다.
코드가 아닌 텍스트로 전문성을 표현함으로써,
스킬이 에이전트 프레임워크에 종속되지 않고 모델과 직접 소통하는 구조가 된다.
프레임워크가 교체되어도 스킬은 재사용 가능하며,
도메인 전문가가 코드 없이 에이전트 행동을 정의할 수 있다.

이 패턴은 이미 Claude Code의 CLAUDE.md, Codex의 AGENTS.md,
그리고 이 TIL 저장소의 `.claude/skills/` 구조에서 검증되고 있다.
Flue는 이 패턴을 프레임워크 수준의 공식 추상화로 격상시킨다.

### Withastro의 선택이 신호하는 것

Astro는 웹 프레임워크로 알려진 팀이다.
아일랜드 아키텍처(Islands Architecture), 정적 사이트 생성, 멀티프레임워크 컴포넌트 지원으로
프론트엔드 개발자 생태계에서 신뢰를 쌓았다.
그 팀이 에이전트 하네스 프레임워크를 만든다는 것은 하나의 신호다.

웹 프레임워크 팀이 공통으로 직면하는 문제 — 배포 환경 추상화,
빌드 도구 통합, 개발자 경험 — 는 에이전트 하네스가 해결해야 할 문제와 구조적으로 닮아 있다.
Cloudflare Workers와 Node.js 양쪽에 동일한 에이전트를 배포하는 문제는
동일한 컴포넌트를 React, Svelte, Vue에서 동작시키는 문제와 같은 층위의 도전이다.
이 팀의 진입이 다른 AI 네이티브 스타트업과 다른 각도의 해법을 가져올 가능성이 있다.

### 내구성 실행이 에이전트 신뢰성의 핵심이 되는 이유

내구성 실행(Durable Execution)은 Temporal과 Cloudflare Durable Objects가
분산 시스템에서 확립한 패턴이다.
이 패턴이 에이전트 하네스에서 특히 중요한 이유는
에이전트 작업의 특성 때문이다.

파일 수정, API 호출, 서브에이전트 위임이 복잡하게 얽힌 장시간 작업에서
중간 실패는 불가피하다.
내구성 실행 없이는 실패 지점부터 재시작하는 것이 아니라 처음부터 다시 시작해야 하며,
이 과정에서 부수 효과(side effect)가 중복 실행될 위험이 있다.
Flue가 이를 프레임워크 수준에서 제공한다는 것은 에이전트의 신뢰성을
애플리케이션 코드가 아닌 인프라가 보장한다는 의미다.

## 비평

### “SDK가 아니다”는 주장이 SDK인 것을 감춘다

Flue는 스스로를 “하네스”로 규정하며 SDK와 선을 긋지만,
`defineAgent`, `@flue/runtime`, `@flue/sdk`라는 명칭들은
이 프레임워크가 여전히 SDK 계층임을 드러낸다.
샌드박스와 세션 관리가 내장된다는 점에서 기존 SDK보다 두텁기는 하지만,
사용자 입장에서는 결국 패키지를 설치하고, API를 호출하며, 추상화에 의존한다.

“하네스”와 “SDK”를 가르는 기준이 무엇인지 문서는 명확히 하지 않는다.
포지셔닝 자체가 마케팅 언어인지 아키텍처적 구분인지 불분명한 상태에서,
이 주장은 경쟁 프레임워크들과 달리 보이고 싶은 욕구의 표현에 더 가깝다.

### 스킬의 마크다운 설계가 규모 확장에서 갖는 문제

스킬을 마크다운으로 관리하는 설계는 소규모에서 우아하지만,
수십, 수백 개의 스킬이 쌓이는 환경에서 취약점이 생긴다.
스킬 간 충돌을 어떻게 감지하는가, 스킬의 버전 관리는 어떻게 하는가,
모순되는 지시를 담은 두 스킬을 동시에 로드하면 에이전트 행동이 어떻게 결정되는가.

타입 시스템으로 잡을 수 없는 텍스트 기반 계약의 한계다.
코드에서는 컴파일러가 인터페이스 충돌을 잡아주지만,
마크다운 스킬 간의 의미론적 충돌을 잡아주는 장치가 없다.
이 문제를 어떻게 해결할 것인지에 대한 설계가 공개되지 않은 상태다.

### 관측성 지원이 표면적 통합에 그칠 위험

OpenTelemetry, Braintrust, Sentry를 지원한다는 것은 강점으로 나열되어 있지만,
에이전트 시스템의 관측성은 웹 서비스의 관측성과 본질적으로 다르다.
요청-응답 추적은 잘 작동하지만, 에이전트가 왜 그 결정을 내렸는가,
어떤 스킬이 어떤 행동에 영향을 미쳤는가, 실패의 근본 원인이
모델 때문인지 하네스 설정 때문인지를 구분하는 것은
표준 텔레메트리 도구가 설계된 문제가 아니다.

`@flue/opentelemetry`가 이 간극을 어떻게 메우는지는 공개된 정보로 확인하기 어렵다.
기존 관측성 도구를 “지원”한다는 것이 에이전트 특유의 가시성을 제공한다는 의미인지,
아니면 단순히 스팬(span)을 내보낸다는 의미인지 명확하지 않다.

### Withastro의 AI 에이전트 영역 신뢰성은 아직 검증 전이다

Astro 팀이 웹 프레임워크에서 쌓은 신뢰는 에이전트 하네스 도메인으로
자동 이전되지 않는다.
웹 프레임워크의 경쟁 우위 — 빌드 속도, 정적 생성, 하이드레이션 전략 —
는 에이전트 하네스의 핵심 문제 — 모델 통합의 깊이, 샌드박스 보안, 내구성 실행의 신뢰성 —
와 겹치는 부분이 적다.
7,000개의 별은 기대의 크기를 보여주지만,
프로덕션 에이전트 시스템을 실제로 운영한 사례가 공개되기 전까지는
이 프레임워크의 실전 신뢰성은 미지수다.

## 인사이트

### 에이전트 프레임워크 전쟁의 새로운 전선

에이전트 프레임워크 경쟁은 지금까지 AI 네이티브 스타트업들 사이에서 벌어졌다.
LangChain, CrewAI, AutoGPT — 모두 LLM 등장 이후 만들어진 조직들이다.
Flue는 이 경쟁에 기존 개발 도구 생태계가 진입하는 것을 상징한다.

이 전환의 의미는 크다.
기존 도구 생태계는 개발자 경험, 빌드 시스템 통합, 배포 파이프라인에서
AI 네이티브 스타트업보다 수년의 노하우를 갖고 있다.
에이전트가 프로덕션 시스템에 통합되는 단계로 넘어갈수록,
이 노하우의 가치는 모델 추상화 능력보다 더 크게 작용할 수 있다.
Withastro의 진입은 이 전환의 신호탄일 수 있다.

### 스킬 마켓플레이스가 다음 경쟁 축이 될 것이다

마크다운 기반 스킬 설계가 성숙하면, 자연스럽게 스킬 공유와 재사용 생태계가 형성된다.
npm이 JavaScript 모듈의, Homebrew가 macOS 도구의 공유 생태계를 만든 것처럼,
에이전트 스킬의 공유 생태계가 등장할 조건이 마련되고 있다.

이 생태계에서 경쟁 우위는 특정 프레임워크를 채택하게 만드는 가장 강력한 유인 중 하나가 된다.
“X 서비스를 위한 스킬 패키지”가 Flue 생태계에서만 작동한다면,
그 스킬들이 사용자를 묶어두는 락인(lock-in) 메커니즘이 된다.
마크다운이 이식성을 높인다는 장점을 가지면서도,
실제로 성숙한 생태계에서는 프레임워크 특화 관례가 쌓이며 이식성을 침식할 것이다.

### Cloudflare Workers 타깃이 에이전트 아키텍처를 바꾼다

Flue가 Cloudflare Workers를 일급 배포 타깃으로 지원한다는 사실은
에지(edge) 컴퓨팅과 에이전트의 결합이 진지하게 고려되고 있음을 시사한다.

전통적인 에이전트 배포는 중앙화된 서버를 전제했다.
에지에서 동작하는 에이전트는 다른 제약과 가능성을 갖는다.
레이턴시가 낮아지고 데이터가 처리되는 위치가 사용자와 가까워지지만,
상태 관리와 내구성 실행의 구현이 훨씬 복잡해진다.
Cloudflare Durable Objects가 이 문제를 해결하는 인프라를 제공하며,
Flue가 이를 프레임워크 수준에서 추상화하려 한다면
에이전트 인프라의 지형을 바꿀 수 있는 조합이다.

### 프레임워크가 사라지면 스킬은 살아남는다

에이전트 프레임워크 생태계는 아직 초기다.
오늘의 선두 주자가 2년 후에도 선두인 보장이 없다.
이 불확실성에서 스킬의 마크다운 기반 설계는 역설적인 보험이 된다.

Flue가 경쟁에서 패하더라도, 마크다운으로 작성된 스킬 라이브러리는
다른 프레임워크로 이식될 수 있다.
CLAUDE.md와 AGENTS.md가 특정 AI 모델에 종속되지 않듯이,
텍스트로 표현된 전문성은 프레임워크 교체 비용을 낮춘다.
이것은 프레임워크가 아닌 스킬에 투자하는 것이
장기적으로 더 안전한 선택일 수 있음을 의미한다.
오늘 Flue를 선택하더라도, 진짜 자산은 Flue 코드가 아닌 마크다운 스킬들이다.
