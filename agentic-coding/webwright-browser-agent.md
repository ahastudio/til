# Webwright: 코딩 모델을 브라우저 에이전트로

프로젝트 페이지:
<https://microsoft.github.io/Webwright/>

GitHub 저장소:

블로그 글:
[Webwright: A Terminal Is All You Need For Web Agents - Microsoft Research](https://www.microsoft.com/en-us/research/articles/webwright-a-terminal-is-all-you-need-for-web-agents/)

## 소개

Webwright는 Microsoft Research AI Frontiers Lab이 공개한 오픈소스 브라우저
자동화 에이전트 프레임워크다. 제목이 핵심을 요약한다: “A Terminal Is All You
Need For Web Agents.” LLM에게 픽셀 좌표 클릭 대신 터미널을 주고, 스스로 브라우저
세션을 띄우며 Python 코드를 작성하게 한다.

전체 구현은 의도적으로 작다. Runner 약 150줄, 모델 엔드포인트 약 550줄, 환경
약 300줄로 총 약 1,000줄이다. 의존성도 `httpx`, `pydantic`, `playwright`,
`typer` 네 가지뿐이다. 숨겨진 프레임워크나 복잡한 오케스트레이션 계층을 의도적으로
배제했다.

## 작동 방식

에이전트 루프는 단순하다. 현재 컨텍스트를 받은 모델이 thinking과 bash 명령을
반환하면 환경이 이를 실행하고 로그, 스크린샷, 오류를 다시 모델에 돌려준다.
이 사이클을 완료 신호가 올 때까지 반복한다.

핵심 설계 결정은 **에이전트와 브라우저를 분리**하는 것이다. 브라우저 세션을
지속적인 상태로 유지하면서 액션 하나씩 예측하는 기존 접근 대신, 브라우저를
일회용 실행 환경으로 취급한다. 지속되는 아티팩트는 브라우저 세션이 아니라
로컬 워크스페이스의 코드와 로그다.

실제 운영 과정에서 두 가지 문제를 발견하고 해결했다. 첫째, 에이전트가 작업을
완료하지 않고 조기에 “완료”를 선언하는 문제다. Self-reflection gate로 해결했다.
완료 선언 전 에이전트가 직접 설정 파일을 생성하고 새 폴더에서 최종 검증 스크립트를
실행해 통과해야만 완료 플래그를 낼 수 있다. 둘째, 긴 작업에서 컨텍스트가 폭발하는
문제다. 20 스텝마다 자동으로 히스토리를 요약(compaction)해 토큰 한계를 회피한다.

## 분석

### 코드-as-액션 패러다임의 부상

기존 브라우저 에이전트는 두 가지 방식 중 하나였다. 좌표 클릭처럼 픽셀 수준에서
직접 DOM을 조작하거나, 사전 정의된 액션 집합(클릭, 입력, 스크롤)에서 하나씩
선택하는 방식이다. 둘 다 근본적인 한계가 있다. 픽셀 좌표는 레이아웃 변경에
취약하고, 이산적 액션 선택은 복잡한 조건 분기나 루프를 표현하지 못한다.

Webwright의 접근은 이 제약을 코드 생성으로 우회한다. Python + Playwright는
완전한 프로그래밍 언어이므로 루프, 조건문, 함수, 예외 처리가 모두 가능하다.
“버튼을 클릭하라”가 아니라 “성공할 때까지 재시도하며 버튼을 클릭하는 함수를
작성하라”는 지시가 가능해진다.

### 워크스페이스 중심 상태 관리

상태를 브라우저 세션이 아닌 로컬 파일 시스템에 두는 결정은 여러 실용적 이점을
만든다. 브라우저 크래시가 치명적이지 않다. 스크립트를 다시 실행하면 된다.
디버깅이 쉽다. 로그와 스크린샷이 디스크에 남아 있다. 재현성이 높다. 같은
스크립트를 다른 환경에서 실행할 수 있다.

무엇보다 “재사용 가능한 자동화”라는 개념이 자연스럽게 탄생한다. 한 번 작성한
스크립트는 파라미터만 바꿔 반복 실행할 수 있는 CLI 도구가 된다.

### 미니멀리즘의 전략적 의미

약 1,150줄의 코드로 state-of-the-art 성능을 낸다는 사실은 단순한 엔지니어링
미덕이 아니다. 이것은 전략적 포지셔닝이다. 복잡한 프레임워크는 기여 장벽이
높고, 내부 동작을 이해하기 어려우며, 연구자들이 새 아이디어를 실험하기 힘들다.
Webwright의 작은 코드베이스는 “이해하고 개선하라”는 초대장이다.

## 비평

### 강점: 패러다임 명확성

코드-as-액션 접근법은 개념적으로 명쾌하다. 기존 브라우저 에이전트 연구의
복잡한 프롬프트 엔지니어링과 특수 액션 공간 설계가 사실은 불필요했을 수
있다는 가능성을 보여준다. 벤치마크 수치가 이를 뒷받침한다.

### 약점: 실제 환경의 비결정성

브라우저 자동화는 근본적으로 비결정적이다. 네트워크 지연, 동적 렌더링, A/B
테스트, 로그인 상태 등 수많은 변수가 존재한다. 스크립트 기반 접근은 이런
변수들을 모두 코드로 처리해야 한다는 부담을 에이전트에게 준다. “단순한”
워크플로가 얼마나 자주 예외 처리 코드로 부풀어 오를지는 실제 배포 경험에서만
검증된다.

### 약점: 스크립트 라이브러리의 유지보수 비용

논문 스스로 인정하듯, 재사용 스크립트가 쌓일수록 검증, 실패 감지, 업데이트
메커니즘이 필요해진다. 세분화가 너무 세면 파편화되고, 너무 굵으면 모놀리스가
된다. 결국 에이전트가 스크립트 라이브러리를 관리하는 또 다른 에이전트를 필요로
할 수 있다. “단순한” 시스템이 운영 단계에서 복잡성을 후불로 치르는 구조다.

## 인사이트

### 코딩 능력이 곧 범용 자동화 능력이다

LLM의 코딩 능력 향상이 코드 생성에만 적용된다는 관점은 점점 좁아지고 있다.
Webwright가 보여주는 것은, 코드를 잘 쓰는 모델은 곧 어떤 소프트웨어 인터페이스도
다룰 수 있는 에이전트가 된다는 것이다. Playwright API를 완벽하게 활용할 수 있는
모델은 사실상 모든 웹사이트를 자동화할 수 있다.

이 논리는 브라우저에 국한되지 않는다. 터미널, API, GUI 애플리케이션, 데이터베이스
쿼리 — 모두 코드로 표현할 수 있는 인터페이스다. 코딩 모델이 강해질수록 범용
자동화 에이전트에 가까워진다. GPT-5 계열 모델에서 86.7%라는 수치는 이미 많은
실무 태스크를 자동화할 수 있는 수준이다. 모델이 계속 발전하면 이 수치는 계속
오른다. 이것은 “브라우저 자동화 도구”의 이야기가 아니라 “모든 소프트웨어
인터페이스가 LLM의 사정권에 들어오는” 이야기다.

이 관점에서 Webwright는 도구가 아니라 증거다. 코딩 능력의 발전이 다른 모든
자동화 영역에 전이된다는 증거. 코딩 벤치마크에서의 점수 향상이 왜 실무 자동화
능력 향상으로 직결되는지를 보여주는 구체적 사례다.

### 에이전트 아키텍처의 수렴 현상

흥미로운 점은 Webwright와 mini-SWE-agent가 서로 다른 문제 도메인(웹 브라우저 vs
코드베이스)을 다루면서 동일한 결론에 도달했다는 것이다. 둘 다 수백 줄의 단순한
루프다. 둘 다 브라우저/셸을 코드로 제어한다. 둘 다 상태를 파일 시스템에 저장한다.
둘 다 최소한의 의존성으로 최대한의 성능을 낸다.

이것은 아마도 수렴이 아니라 발견이다. 에이전트의 “올바른 형태”가 단순한 루프
+ 코드 생성이라는 것을 여러 팀이 독립적으로 발견하고 있다. 복잡한 오케스트레이션
프레임워크들이 점점 더 복잡한 그래프와 체인을 만들어 갈 때, 단순한 구현들이
더 좋은 벤치마크 수치를 내는 역설은 소프트웨어 공학의 오래된 교훈을 반복한다.
복잡성은 성능에서 오지 않는다. 오히려 성능을 갉아먹는다.

에이전트 프레임워크 생태계가 결국 어디로 수렴할지 이 패턴이 암시한다.
복잡한 다중 에이전트 그래프는 특수 목적 시스템에서만 살아남을 것이다.
범용 에이전트는 단순해질 것이다. 모델이 강해질수록 프레임워크는 얇아진다.
프레임워크의 역할은 모델이 못 하는 것을 보완하는 것인데, 모델이 점점 더
많은 것을 할 수 있게 되면 프레임워크가 할 일이 줄어들기 때문이다.

### 웹 에이전트가 웹 접근성 인프라가 된다

블로그가 제안하는 방향 중 가장 흥미로운 것은 접근성(accessibility)과의 연결이다.
Webwright는 웹 페이지를 탐색할 때 접근성 트리(accessibility tree)와 ARIA 메타데이터를
활용한다. 그런데 연구진은 여기서 더 나아가, 강력한 에이전트가 웹을 “수리”할 수 있다고
주장한다. 레이블이 없는 버튼, 혼란스러운 컨트롤, 접근 불가능한 폼을 감지하고
재사용 가능한 수정 스크립트를 생성하는 것이다.

이 발상은 역할을 뒤집는다. 에이전트가 열악한 접근성을 극복하는 피해자가 아니라,
접근성 문제를 발견하고 고치는 인프라가 된다. 수백만 개의 웹 페이지를 인간 접근성
감사원이 일일이 검토하는 것은 불가능하지만, 에이전트가 작업하면서 발견한 장애물을
자동으로 보고하고 수정 패치를 생성한다면 이야기가 달라진다.

이것은 에이전트가 웹을 소비하는 데서 웹을 개선하는 방향으로 전환하는 첫 번째
구체적인 아이디어다. “에이전트가 많이 쓸수록 웹이 더 접근하기 쉬워진다”는 양의
피드백 루프가 가능해진다. 인터넷 인프라의 품질이 사람이 아닌 에이전트의 부산물로
향상되는 세계는 SF처럼 들리지만, Webwright의 접근성 트리 활용이 그 출발점이 될 수
있다.

### 재사용 가능한 자동화가 만드는 새로운 생산성 계층

Webwright가 강조하는 “재사용 가능한 스크립트” 개념은 표면적으로 단순하지만,
그 함의는 깊다. 지금까지 웹 자동화는 대부분 일회성이었다. 특정 작업을 위해
스크립트를 작성하거나, RPA 도구로 레코딩을 했다. 이것들은 웹사이트가 바뀌면
깨졌고, 유지보수가 어려웠다.

에이전트가 직접 스크립트를 작성하고 실행하면서 실패 시 수정까지 하는 루프가
가능해지면, 다른 패턴이 등장한다. 에이전트가 한 번 작성한 스크립트를 나중에
다른 컨텍스트에서 재사용하거나, 파라미터만 바꿔 CLI 도구로 변환할 수 있다.
시간이 지날수록 워크스페이스에는 검증된 자동화 스크립트 라이브러리가 쌓인다.

이것은 개인이나 팀의 자동화 자산이 축적된다는 의미다. 에이전트가 단순히 태스크를
완료하는 것을 넘어, 재사용 가능한 능력을 조직에 기여하는 형태가 된다.
“에이전트를 통해 작업을 처리한다”에서 “에이전트가 조직의 자동화 인프라를
구축한다”로의 전환이다. 이 전환이 실제로 일어나면, 에이전트 도입의 ROI는
태스크 단위 효율이 아니라 누적 자산 형성으로 측정되어야 할 것이다.
