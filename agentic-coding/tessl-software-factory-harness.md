# 하네스 엔지니어링: 에이전트가 코드를 짜고, 개발자는 공장을 만든다

원문: [하니스 엔지니어링: 에이전트가 코드를 짜고, 개발자는 공장을 만든다 | 박재홍의 실리콘밸리](https://wikidocs.net/blog/@jaehong/23991/)

## 요약

Tessl의 제품·디자인 총괄 드루 녹스(Dru Knox)가 AI Engineer 컨퍼런스에서 발표한
“하네스 엔지니어링(harness engineering, 혹은 루프 엔지니어링)”을
박재홍(@jaehong)이 한국어로 해설한 글이다.

핵심 전환은 이렇다: 개발자가 제품을 직접 손으로 만드는 대신,
제품을 찍어내는 “공장”(에이전트 시스템)을 만들고 그 공장의 품질을 끌어올리는 데 집중한다.
소프트웨어 팩토리는 완결된 종착지가 아니라 위임의 폭을 한 겹씩 넓혀가는 여정이다.

성숙도는 세 지표로 측정한다.
첫째, **자율성(autonomy)**: 에이전트가 사람 개입 없이 정답에 도달하는가.
둘째, **자동화(automation)**: 사람의 감독 없이 에이전트를 얼마나 허용하는가.
자율성이 높아도 결과를 믿지 못하면 자동화는 낮다 — 핵심은 **신뢰(trust)**다.
셋째, **품질(quality)**. 순서가 중요하다: 자율성 → 자동화 → 품질.

신뢰를 쌓는 도구는 세 개의 루프다.
이너 루프(PR 전 에이전트 자가 점검), 아우터 루프(PR 시점의 느리고 비싼 검증),
메타 루프(두 루프를 관찰해 자동 개선 — “같은 실수는 두 번 알려주지 않는다”).

Tessl은 두 가지 파격적 규칙을 세웠다: 사람이 손으로 짠 코드 금지,
대화형 코딩 세션 금지.
대신 이슈 트래커에 티켓을 올리면 에이전트가 PR을 만들어 오고,
사람은 그 PR을 코드 리뷰한다.

코드 리뷰는 두 겹이다.
**체인지 리뷰(Change Review)**: 보안·가독성 등 렌즈로 예상 못 한 문제를 잡는다.
**베리파이어(Verifier)**: 작고 빠른 LLM 린트 규칙으로 이분법 검사를 결정론적으로 반복한다.
전자는 처음 보는 문제를, 후자는 이미 겪은 실수의 재발을 막는다.

핵심 통찰: 소프트웨어 팩토리는 “슬롭을 감수하고 속도를 얻는” 거래가 아니다.
에이전트에게는 백로그가 없어 지적당한 문제를 즉시 전부 고치므로,
중장기적으로 코드 품질은 내려가는 게 아니라 오히려 올라간다.

## 분석

### 자율성과 자동화의 구분이 이 프레임워크의 핵심이다

대부분의 팀이 에이전트 도입 초기에 경험하는 것은 자율성의 향상이다.
에이전트가 지시 하나에 30~40분을 혼자 돌린 뒤 손댈 필요 없는 결과를 내놓는다.
그런데 이 단계에서 팀은 여전히 코드를 한 줄씩 직접 검토하고 손으로 검증한다.
자율성은 높지만 자동화는 낮은 상태다.

이 구분이 중요한 이유는, 두 지표를 높이는 방법이 다르기 때문이다.
자율성은 이너 루프를 개선해 높인다 — 에이전트가 스스로 더 잘 판단하도록 피드백 속도를 높이는 것이다.
자동화는 신뢰를 쌓아 높인다 — 아우터 루프가 사람 검토를 대신할 수 있을 만큼 견고해지는 것이다.
두 지표를 같은 방법으로 높이려 하면 둘 다 놓친다.

### “인터랙티브 세션 금지”는 통제권 회수 선언이다

Tessl이 대화형 코딩 세션을 금지했을 때 두 엔지니어가 불편함을 느꼈다는 사실이
이 금지의 본질을 드러낸다.
인터랙티브 세션은 개발자가 에이전트를 실시간으로 조율하는 구조다.
팩토리 모델은 이 조율 권한을 에이전트에게 넘기는 대신 하네스가 구조적으로 대신한다.

불편함의 원인은 기술적인 것이 아니다.
에이전트가 뭔가를 하는 동안 개발자가 “지켜보는” 상태에서 “방관자”가 된다는 심리적 전환이다.
Tessl은 이 불편함을 제거하는 방향이 아니라 그것을 견뎌야 하는 조건으로 설정했다.

### 베리파이어가 스킬의 약점을 보완한다

스킬은 에이전트가 상황에 맞는 스킬을 “활성화”해야 작동하는 불확실성이 있다.
베리파이어는 결정론적이다 — 매번 빠짐없이 실행된다.

베리파이어는 불변식(invariant)을 정의하고 이분법 질문으로 좁힌다.
“이 파일에 ARIA 속성이 없는 프런트엔드 요소가 있는가?”
“이 파일에 사내 로거가 아닌 다른 로거 호출이 있는가?”
질문을 좁혀두면 성공률이 사실상 100%에 수렴한다.

체인지 리뷰가 예상 못 한 문제를 잡는다면,
베리파이어는 지금껏 겪은 모든 문제가 재발하지 않도록 막는다.
“실수는 딱 한 번만”이라는 원칙의 구체적 구현이다.

## 비평

### Tessl 사례의 일반화 범위가 불명확하다

Tessl은 자체 도구를 만드는 회사다.
자체 도구를 만드는 회사가 에이전트 기반 소프트웨어 팩토리를 구축하는 것과
기존 레거시 코드베이스를 운영하는 회사가 같은 구조를 채택하는 것은 전혀 다른 문제다.

소프트웨어 팩토리가 효과적으로 작동하려면 코드베이스가 에이전트 친화적으로 설계돼야 한다.
테스트 커버리지, 명확한 모듈 경계, 기계 검증 가능한 요구사항 명세가 전제 조건이다.
이 조건을 갖추지 못한 조직이 인터랙티브 세션 금지부터 시작한다면
팩토리가 아니라 혼란을 얻는다.

### 컨트롤 플레인 구축 비용이 진입 장벽이다

이슈 트래커 + 오케스트레이터 + 에이전트 실행 환경을 연결하는 것은 생각보다 번거롭다.
신원(identity), 웹훅(webhook), PR 코멘트 반응, 자격증명 갱신 — 이 모든 문제를 직접 풀어야 한다.

Tessl의 마리아(Maria) 일화가 이를 잘 보여준다.
오케스트레이터 첫 버전에 자신의 GitHub 자격증명을 그대로 써서
몇 주 동안 모든 PR이 마리아 이름으로 올라와 기여자 순위 1위로 치솟았다.
이 인프라 문제를 제대로 풀지 않으면 생산성이 아니라 혼란이 생긴다.

### 인터랙티브 세션 금지가 탐색 단계에 맞지 않는다

인터랙티브 에이전트 세션의 가치 중 하나는 탐색적 프로그래밍(exploratory programming)이다.
무엇을 만들어야 할지 아직 모르는 상태에서 에이전트와 함께 가능성을 탐색하는 방식이다.

파이프라인 기반 팩토리는 목표가 명확할 때 효율적이다.
목표 자체가 불명확한 탐색 단계에서는 파이프라인이 오히려 장애가 된다.
녹스도 “생각만큼 극단적이지 않다”고 완화하지만,
어디서부터 금지가 시작되는지, 아이디어 탐색 단계도 포함하는지는 여전히 불명확하다.

## 인사이트

### 소프트웨어 팩토리는 품질 책임의 소재를 바꾼다

전통적 개발에서 코드 품질의 1차 책임자는 코드를 작성한 개발자다.
에이전트 기반 팩토리에서 코드는 에이전트가 작성하지만,
품질의 1차 책임자는 하네스와 검증 시스템을 설계한 사람으로 이동한다.

이 이동은 단순한 역할 변화가 아니다.
버그가 났을 때 “누가 잘못 코딩했는가”에서 “어떤 검증이 빠졌는가”로 질문이 바뀐다.
책임 소재가 개인 역량에서 시스템 설계로 이동하는 것은
소프트웨어 엔지니어링 조직이 어떻게 구성돼야 하는지를 근본적으로 재정의한다.

### 백로그 소멸이 품질 복리 효과를 만든다

Tessl이 5개 리뷰 렌즈를 사람이 작성한 PR에 적용했을 때 엔지니어들이 불평했다:
“피드백이 너무 많다, 다 고칠 시간이 없다.”
지적 자체는 코드 품질을 실제로 높여줄 진짜 문제들이었는데도, 사람에겐 여력이 없었다.

에이전트는 백로그가 없다.
지적당한 문제를 즉시 전부 고친다.
이것이 “중장기적으로 품질이 올라간다”는 주장의 근거다.
“언젠가 고치자”가 영원히 미뤄지던 기술 부채가 자동으로 해소되는 구조 변화다.

### 세 루프 모델은 제조업 품질 관리 이론과 같은 구조다

이너/아우터/메타 루프는 제조업의 PDCA(Plan-Do-Check-Act) 사이클과 구조적으로 같다.
이너 루프는 Do-Check, 아우터 루프는 Act(수정), 메타 루프는 Plan(개선)이다.
토요타 생산 방식의 Jidoka(이상 감지 즉시 정지)가 이너 루프의 자동 검증과 동형이다.

에이전트 개발이 제조업 품질 관리 이론을 재발견하고 있다는 것은,
인간 작업자의 실수를 줄이기 위해 개발된 시스템이
에이전트의 실수를 줄이는 데도 그대로 적용된다는 뜻이다.
60년 역사의 품질 관리 이론이 갑자기 에이전트 시스템 설계의 선례 라이브러리가 됐다.

### “팩토리”라는 은유가 숨기는 것이 있다

공장은 사양이 확정된 제품을 반복 생산하는 곳이다.
소프트웨어 개발의 많은 부분은 아직 사양이 확정되지 않은 문제를 다루는 것이다.

“소프트웨어 팩토리” 은유는 반복 가능한 부분을 자동화하는 데 유용하지만,
창의적 설계와 문제 정의 단계를 은폐하는 위험이 있다.
공장의 강점은 변동성을 줄이는 것이다.
소프트웨어 개발의 가치 있는 부분은 때로 변동성에서 나온다 — 예상치 못한 접근,
기존 제약을 무시하는 해법, 요구사항 자체를 바꾸는 통찰.
팩토리 모델이 이 공간을 어떻게 다루는지가 이 개념의 진짜 한계를 드러낼 것이다.
