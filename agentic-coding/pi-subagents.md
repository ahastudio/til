# pi-subagents: Pi 코딩 에이전트용 서브에이전트 위임 익스텐션

<https://github.com/nicobailon/pi-subagents>

## 소개

`pi-subagents`는 `@mariozechner/pi-coding-agent`(Pi)에서 작업을 전문화된 하위 에이전트에게 위임할 수 있게 해주는 익스텐션이다.
코드 리뷰·탐색·구현·병렬 감사·저장된 워크플로·백그라운드 작업 등 “두 번째, 세 번째 시선”이 필요한 모든 상황에서 활용된다.
TypeScript로 작성되었으며, 2,000개 이상의 별을 받은 활발한 프로젝트다.

## 주요 기능

### 내장 에이전트

별도 설정 없이 즉시 사용할 수 있는 역할 기반 에이전트를 제공한다.

| 에이전트           | 목적                                                                   |
| ------------------ | ---------------------------------------------------------------------- |
| `scout`            | 코드베이스 탐색: 관련 파일·진입점·데이터 흐름·위험 요소 파악           |
| `researcher`       | 웹·공식 문서 조사 및 간결한 리서치 브리핑 작성                         |
| `planner`          | 기존 컨텍스트를 바탕으로 구체적인 구현 계획 수립 (코드 편집 금지)      |
| `worker`           | 계획 구현, 검증, 미승인 결정 에스컬레이션                              |
| `reviewer`         | 코드 리뷰 및 소규모 수정                                               |
| `context-builder`  | 계획 전 컨텍스트 수집 및 핸드오프 자료(`context.md`, `meta-prompt.md`) |
| `oracle`           | 가정에 도전하고 가장 안전한 다음 행동을 추천하는 두 번째 의견 제공     |
| `delegate`         | 부모 세션과 유사하게 동작하는 범용 경량 위임 에이전트                  |

### 실행 모드

자연어 요청만으로 적절한 모드가 선택된다.

- **단일 실행**: 하나의 에이전트에게 작업을 위임한다.
- **병렬 실행**: 여러 에이전트가 동시에 실행된다.
- **체인 실행**: 에이전트가 순차적으로 실행되며 이전 결과를 다음 단계로 전달한다.
- **백그라운드 실행**: `--bg` 플래그로 분리 실행하여 부모 에이전트가 다른 작업을 계속한다.
- **포크 실행**: `--fork` 플래그로 부모 세션의 현재 상태에서 분기된 실제 세션을 생성한다.

### 체인 내 인라인 병렬 그룹

체인 내에서 괄호와 `|`를 사용해 정적 병렬 팬아웃을 선언할 수 있다.

```text
/chain scout "scan" -> (reviewer "A" | reviewer "B") -> writer "fix"
```

### 동적 팬아웃

`.chain.json` 파일에서 이전 구조화 출력 결과를 바탕으로 배열을 확장해 아이템마다 하위 에이전트를 생성한다.
`expand`, 병렬 `parallel` 템플릿, `collect` 조합으로 동적 팬아웃과 팬인(fan-in)을 구성한다.

### 워크트리 격리

병렬 에이전트가 동일 체크아웃을 동시에 수정하면 충돌이 발생한다.
`worktree: true` 옵션은 각 병렬 자식에게 `HEAD`에서 분기된 독립 git 워크트리를 부여한다.

### 수용 게이트(Acceptance Gates)

각 실행에 수용 정책을 설정해 에이전트 출력의 신뢰도를 분류한다.
`auto`, `none`, `attested`, `checked`, `verified`, `reviewed` 수준을 지원하며,
`verified`는 사용자가 지정한 명령(예: `npm test`)을 실제로 실행해 통과 여부를 확인한다.

### pi-intercom 연동

선택적으로 `pi-intercom`을 설치하면 자식 에이전트가 실행 중 부모 세션으로 결정을 요청하거나 진행 상황을 보고할 수 있다.
`contact_supervisor` 도구를 통해 `need_decision` 또는 `progress_update` 이유로 메시지를 전송한다.

## 사용법

```bash
pi install npm:pi-subagents
```

설치 후 자연어로 요청한다.

```text
Use reviewer to review this diff.
Ask oracle for a second opinion on my current plan.
Run parallel reviewers: one for correctness, one for tests, and one for unnecessary complexity.
Have worker implement this approved plan. Afterward, run parallel reviewers, summarize their feedback, and apply the fixes that make sense.
```

직접 명령어를 사용할 수도 있다.

| 명령어                          | 설명                                    |
| ------------------------------- | --------------------------------------- |
| `/run <agent> [task]`           | 단일 에이전트 실행                      |
| `/chain agent1 “t1” -> agent2`  | 순차 체인 실행                          |
| `/parallel agent1 -> agent2`    | 병렬 실행                               |
| `/run-chain <name> -- <task>`   | 저장된 체인 워크플로 실행               |
| `/parallel-review`              | 관점별 신선 컨텍스트 리뷰어 병렬 실행  |
| `/review-loop`                  | 정리될 때까지 리뷰·수정 사이클 반복    |
| `/subagents-doctor`             | 설정 진단                               |

## 분석

### 자연어 인터페이스가 실제 위임 패턴을 바꾼다

에이전트 위임 도구의 전통적 접근은 명시적 API 호출이다.
사용자는 어떤 에이전트를 선택할지, 어떤 모드로 실행할지, 컨텍스트를 어떻게 전달할지 직접 결정해야 한다.
이 프로젝트는 그 결정을 에이전트에게 넘긴다.
“Use reviewer to review this diff”라고 말하면, Pi가 `reviewer` 에이전트를 선택하고, 실행 모드를 결정하고, 컨텍스트를 구성한다.

이 설계의 핵심 전제는 위임 결정 자체도 AI가 잘 내릴 수 있다는 것이다.
어떤 상황에서 `oracle`을 쓰고, 언제 `parallel reviewers`를 쓸지는 규칙으로 명시하기보다 예시로 학습시키는 것이 더 효과적이다.
패키지에 포함된 `pi-subagents` 스킬은 부모 에이전트에게 이 위임 패턴을 가르치는 가이드 역할을 한다.

### 컨텍스트 전파 설계의 비대칭성

부모 세션의 컨텍스트를 자식에게 전달하는 방법으로 `fresh`와 `fork`를 제공한다.
`fresh`는 빈 컨텍스트로 시작하고, `fork`는 부모의 현재 세션 파일에서 분기된 실제 세션을 생성한다.

흥미로운 점은 역할별 기본값이 다르다는 것이다.
`planner`와 `worker`, `oracle`은 `fork`를 기본값으로 사용한다.
계획과 구현은 부모가 이미 파악한 컨텍스트를 필요로 하기 때문이다.
반면 `reviewer`는 `fresh` 맥락에서 더 독립적인 시각을 가질 수 있다.
이 비대칭은 역할의 인식론적 특성을 반영한다.
계획자는 기존 지식에서 출발해야 하고, 검토자는 편견 없이 시작해야 한다.

### 수용 게이트는 에이전트 출력에 신뢰도 층위를 만든다

기존 AI 출력의 문제는 결과가 맞는지 틀린지 알 방법이 없다는 것이다.
에이전트가 “구현 완료”라고 말해도 그것이 진짜 완료인지, 편향된 자기 평가인지 구분하기 어렵다.
수용 게이트는 이 신뢰도를 다섯 단계로 분류한다.

`attested`는 에이전트가 구조화된 수용 보고서를 반환했다는 뜻이다.
`verified`는 사용자가 지정한 명령이 실제로 통과했다는 뜻이다.
`reviewed`는 독립적인 검토자 결과가 있다는 뜻이다.
에이전트의 자기 주장과 외부 검증을 명확히 구분함으로써, 파이프라인 다음 단계가 선택적으로 더 높은 신뢰도를 요구할 수 있다.

## 비평

### 자연어 위임은 예측 불가능성이라는 비용을 치른다

자연어로 위임 결정을 내리는 방식은 유연하지만, 그 결정이 언제 어떻게 바뀔지 예측하기 어렵다.
같은 요청도 컨텍스트에 따라 다른 에이전트를 선택하거나, 다른 실행 모드를 사용할 수 있다.
프로덕션 환경이나 반복 가능한 워크플로가 필요한 상황에서 이 예측 불가능성은 큰 문제다.
저장된 `.chain.md` 파일이 이를 해결하려 하지만, 복잡한 동적 위임은 여전히 런타임 LLM 판단에 의존한다.

### 컨텍스트 경계의 복잡성이 실용적 한계를 만든다

`inheritProjectContext`, `inheritSkills`, `systemPromptMode`, `defaultContext`, `extensions` 등 컨텍스트 전파를 제어하는 옵션이 지나치게 많다.
각 에이전트 프론트매터에서 이 옵션들을 조합하는 경우의 수는 기하급수적으로 늘어난다.
실제로 에이전트가 “왜 이 컨텍스트를 받았는가”를 추적하기 어려워지며,
디버깅은 결국 `/subagents-doctor`와 로그 파일에 의존하게 된다.
이는 복잡성이 기능이 아니라 부채로 바뀌는 전형적인 패턴이다.

### 수용 게이트의 상위 레벨은 실제로 사용되지 않을 가능성이 높다

`verified`와 `reviewed` 수용 레벨은 강력한 개념이지만, 설정 비용이 높다.
`verified`는 검증 명령을 명시해야 하고, `reviewed`는 독립 검토자를 별도로 실행해야 한다.
대부분의 실용적 사용에서는 `auto` 또는 `attested`에 머물 가능성이 높다.
README가 이 레벨들을 대등하게 설명하지만, 실제 채택률은 크게 다를 것이다.
높은 신뢰도 레벨이 실제로 얼마나 활용되는지에 대한 데이터나 예시가 없다는 점은 이 기능의 실용성을 평가하기 어렵게 만든다.

## 인사이트

### 에이전트 팀 구성은 소프트웨어 팀 설계의 문제로 수렴한다

`scout → planner → worker → reviewer`라는 권장 패턴은 소프트웨어 팀의 역할 분화와 같은 구조다.
탐색가가 현황을 파악하고, 설계자가 계획을 세우고, 개발자가 구현하고, 검토자가 품질을 확인한다.
이 병렬성은 우연이 아니다.
팀 설계의 핵심 문제(누가 무엇을 알아야 하는가, 언제 정보를 공유하는가, 누가 최종 결정을 내리는가)가 에이전트 설계에서도 동일하게 나타난다.

이 관점은 에이전트 설계 역량의 위치를 바꾼다.
“어떤 프롬프트를 쓸까”가 아니라 “어떻게 팀을 구성할까”의 문제다.
조직 이론과 소프트웨어 팀 설계에서 검증된 패턴들(역할 분리, 정보 흐름, 에스컬레이션 경로)이 에이전트 시스템 설계에 직접 적용 가능하다.

### 자식 에이전트의 안전 경계가 신뢰할 수 있는 위임의 전제조건이다

이 프로젝트가 공들인 부분 중 하나는 자식 에이전트가 부모 권한을 과도하게 행사하지 못하도록 하는 것이다.
자식은 기본적으로 `subagent` 도구를 받지 않아 중첩 위임을 할 수 없다.
부모 세션의 오케스트레이션 지시는 컨텍스트 포크 시 명시적으로 제거된다.
미승인 결정은 추측 대신 에스컬레이션하도록 프롬프트에 명시되어 있다.

이 경계 설계는 에이전트 보안의 핵심 원칙을 보여준다.
권한은 명시적으로 부여되어야 하며, 상속되어서는 안 된다.
부모가 할 수 있는 것을 자식이 자동으로 할 수 있어서는 안 된다.
이 원칙이 충분히 구현되지 않으면, 단일 에이전트의 실수가 전체 파이프라인을 무너뜨릴 수 있다.

### 동적 팬아웃은 에이전트 시스템과 데이터 파이프라인의 경계를 허문다

`.chain.json`의 `expand` 기능은 이전 에이전트 출력을 데이터 소스로 삼아 새 에이전트 집합을 동적으로 생성한다.
이는 전통적인 ETL 파이프라인의 `flatMap` 연산과 구조적으로 같다.
차이점은 각 아이템의 처리자가 데이터 변환 함수가 아니라 자율적 에이전트라는 점이다.

이 패턴이 성숙하면, 에이전트 시스템과 데이터 파이프라인의 경계가 흐려질 것이다.
아이템 수에 따라 에이전트 수가 동적으로 결정되고, 에이전트 출력이 다음 에이전트의 입력이 되는 구조는
Spark나 Flink 같은 분산 처리 프레임워크의 아이디어를 에이전트 레이어로 가져온 것이다.
이 방향의 다음 단계는 에이전트 스케줄링, 실패 재시도, 체크포인트 같은 데이터 파이프라인의 운영 개념을 에이전트 시스템에 이식하는 것이다.
