# Agent Lightning

> Optimize ANY agent with ANY framework

<https://www.microsoft.com/en-us/research/project/agent-lightning/>

<https://github.com/microsoft/agent-lightning>

## 소개

Agent Lightning은 Microsoft Research가 공개한 AI 에이전트 RL(강화학습) 훈련 프레임워크다.
슬로건은 “Train ANY AI Agents with Reinforcement Learning”이며,
기존 에이전트를 코드 수정 거의 없이 RL로 최적화하는 것을 목표로 한다.
MIT 라이선스, arXiv 논문(2508.03680), 2025년 6월 Microsoft Research 프로젝트 페이지 공개.

핵심 문제 인식: 에이전트를 RL로 훈련하려면 에이전트 코드 자체에 훈련 로직을 심어야 했다.
보상 신호 전달, 멀티턴 크레딧 할당, 학습 상태 관리가 모두 에이전트 코드에 얽혔다.
Agent Lightning은 이 결합을 끊는다.

## 아키텍처

### Training-Agent Disaggregation

에이전트 실행과 RL 학습을 물리적으로 분리하는 설계다.
에이전트는 OpenAI 호환 API를 평소와 동일하게 호출하고,
그 호출이 Lightning Server로 라우팅되어 훈련 데이터로 수집된다.

```text
에이전트 (LangChain / OpenAI SDK / AutoGen / ...)
    ↓ OpenAI 호환 API 호출 (엔드포인트만 변경)
Lightning Server (OpenAI 호환 엔드포인트)
    ↓ agl.emit_xxx() 또는 자동 tracer
LightningStore (tasks, resources, traces 동기화)
    ↓
Algorithm (GRPO / PPO / APO / SFT / ...)
    ↓ 개선된 리소스 (프롬프트 템플릿, 정책 가중치)
Trainer → inference engine 업데이트
```

에이전트 코드에 추가하는 것은 `agl.emit_xxx()` 헬퍼 몇 줄이거나,
자동 tracer를 사용하면 그것도 불필요하다.

### LightningStore

태스크, 리소스, 트레이스를 중앙에서 동기화하는 허브다.
에이전트 실행 중 발생하는 모든 이벤트(프롬프트, 툴 호출, 보상)가
구조화된 스팬(span)으로 저장된다.
알고리즘은 스팬을 읽어 학습하고, 업데이트된 리소스(새 프롬프트 템플릿 또는 정책 가중치)를
다시 LightningStore에 쓴다.

### Credit Assignment Module

멀티턴 인터랙션에서 개별 행동의 기여도를 계산하는 모듈이다.
에이전트의 대화가 여러 턴에 걸쳐 이루어질 때,
최종 보상 신호를 각 행동 단계에 분배하는 역할을 한다.

## 지원 알고리즘

| 알고리즘                        | 특징                                                   |
| ------------------------------- | ------------------------------------------------------ |
| GRPO                            | 그룹 상대 정책 최적화, 별도 가치 네트워크 불필요       |
| PPO                             | 전통적 근위 정책 최적화                                |
| REINFORCE++                     | REINFORCE 강화 버전                                    |
| Supervised Fine-tuning (SFT)    | 지도 학습 기반 미세조정                                |
| APO (Automatic Prompt Optimization) | 모델 가중치 변경 없이 프롬프트 자동 최적화         |

GRPO는 질문당 여러 출력을 샘플링해 그룹 내 상대 성능을 비교하므로,
별도 가치 네트워크 없이 메모리 효율적인 학습이 가능하다.
APO는 가중치 변경이 없어 빠른 프로토타이핑에 적합하다.
서버 인프라는 ByteDance의 verl(벡터 RL 프레임워크) 위에 구축됐다.

## 분석

### 재토크나이제이션 드리프트(Retokenization Drift) 문제

에이전트 RL에서 잘 알려지지 않은 기술적 함정이 있다.
추론(inference) 단계에서 생성된 텍스트를 나중에 훈련 단계에서 다시 토크나이징할 때
토큰 시퀀스가 달라질 수 있다.

원인은 세 가지다:
- **비유일 토크나이제이션**: 같은 문자열이라도 “HAVING”이 `H+AVING` 또는 `HAV+ING`으로
  다르게 분해될 수 있다.
- **툴 호출 직렬화 드리프트**: JSON 파싱·재렌더링 과정에서 포맷이 달라진다.
- **채팅 템플릿 차이**: 프레임워크마다 약간씩 다른 템플릿이 다른 토큰을 생성한다.

이 불일치는 off-policy RL 업데이트를 만들어 학습 곡선을 불안정하게 한다.

Agent Lightning v0.2는 vLLM의 `return_token_ids: true` 파라미터를 활용해 이를 해결한다.
추론 단계에서 텍스트와 함께 정확한 토큰 ID를 직접 수신하므로,
재토크나이징 자체가 발생하지 않는다.
기존 OpenAI 호환 클라이언트와의 호환성을 유지하면서
토큰 수준 정밀도를 확보한 설계다.

### Sidecar 패턴의 RL 훈련 적용

`agl.emit_xxx()` 헬퍼와 자동 tracer는 마이크로서비스의 Sidecar 패턴과 같은 원리다.
서비스 코드를 건드리지 않고 로깅·트레이싱·보안을 주입하듯,
에이전트 코드를 건드리지 않고 훈련 데이터를 수집한다.

이 설계의 핵심은 관찰 지점이 에이전트 내부가 아니라
에이전트가 이미 사용하는 API 경계라는 것이다.
에이전트가 OpenAI API를 호출하는 것 자체가 관찰 가능한 이벤트가 된다.

### verl 채택의 의미

Microsoft Research가 자체 RL 인프라 대신 ByteDance의 verl을 사용하는 것은
“RL 훈련 인프라는 이미 해결된 문제”라는 판단이다.
Agent Lightning의 혁신은 인프라가 아니라 에이전트와 훈련을 잇는 인터페이스 레이어다.
성숙한 오픈소스를 재사용하고, 자신의 기여를 가장 가치 있는 계층에 집중한 선택이다.

## 비평

### 강점: OpenAI 호환 API가 채택 마찰을 최소화한다

대부분의 에이전트 프레임워크는 이미 OpenAI 호환 인터페이스를 사용한다.
엔드포인트 URL 변경 한 줄이 전부라는 것은 과장이 아니다.
LangChain, OpenAI SDK, AutoGen, CrewAI 예제가 모두 지원되며,
Tencent가 128 GPU 규모로 검증한 사례가 있다.

### 강점: 재토크나이제이션 드리프트 해결이 기술적 완성도를 높인다

vLLM 블로그에서 별도 포스트로 다룰 만큼 실질적인 문제였다.
v0.2에서 `return_token_ids` 통합으로 이를 해결한 것은
“작동은 하지만 미묘하게 잘못된” 상태를 벗어나 훈련 안정성을 실질적으로 개선한다.

### 강점: 벤치마크가 구체적이다

“정확도 향상”이라는 막연한 주장이 아니라
특정 데이터셋(Spider), 특정 샘플 수(500개),
특정 모델별 수치(3B, 7B, Llama)를 제시한다.
SQL 에이전트는 검증 가능한 정답이 있어 측정이 용이한 도메인으로,
이것은 재현 가능한 주장이다.

### 약점: SQL 에이전트 외 도메인 검증이 부족하다

Spider 데이터셋은 보상 함수가 자명한 도메인이다.
쿼리 실행 결과가 정답인지 아닌지 명확하다.
열린 도메인(고객 지원, 정보 검색, 코드 리뷰)에서
“이 인터랙션이 얼마나 좋았는가”를 정의하는 방법은
Agent Lightning이 제공하지 않는 보상 설계 문제로 남는다.

### 약점: 프로덕션 환경의 지연 오버헤드가 불명확하다

에이전트의 LLM 호출이 Lightning Server를 경유하면서 추가 레이턴시가 발생한다.
학습 단계에서는 허용 가능하지만,
프로덕션 서빙과 학습을 동시에 진행하는 온라인 학습 시나리오에서의
오버헤드가 문서화되어 있지 않다.

### 약점: 멀티에이전트 시나리오의 크레딧 할당이 복잡하다

단일 에이전트의 멀티턴 크레딧 할당도 어렵지만,
여러 에이전트가 협력하는 시스템에서 특정 에이전트의 기여도를 분리하는 것은
더 어려운 문제다.
README에서 “멀티에이전트 시스템에서 특정 에이전트만 선택적 최적화”를 언급하지만,
이것이 실제로 얼마나 잘 동작하는지는 검증 사례가 적다.

## 인사이트

### 에이전트 RL의 진짜 병목은 코드 복잡성이 아니라 보상 정의다

Agent Lightning이 제거한 것은 “코드 수정”이라는 구현 장벽이다.
이 장벽이 사라지면 다음 병목이 드러난다. 바로 보상 함수 설계다.

SQL 에이전트, 수학 문제 풀기, 코드 생성처럼 검증 가능한 정답이 있는 도메인은
Agent Lightning이 즉시 가치를 제공할 수 있다.
그러나 프로덕션 에이전트의 대부분은 열린 도메인에서 동작한다.
이메일 초안이 “얼마나 좋은지”, 검색 결과가 “얼마나 관련성 있는지”를
수치로 정의하는 것은 Agent Lightning이 해결하지 않는 문제다.

이 도구의 확산은 “보상 설계 패턴 라이브러리”라는 새로운 수요를 만든다.
LLM-as-Judge를 보상 함수로 활용하는 패턴,
간접 지표(태스크 완료율, 사용자 수정 횟수)를 보상으로 변환하는 방법,
RLHF 수집 파이프라인과의 통합이 Agent Lightning 생태계에서 발전해야 할 부분이다.

### API 경계가 관찰 지점이 되는 설계는 더 넓은 패턴이다

Agent Lightning이 OpenAI 호환 API를 훈련 채널로 재활용하는 발상은
소프트웨어 엔지니어링의 오래된 원칙을 ML에 적용한 것이다.
“이미 있는 인터페이스를 관찰 지점으로 삼아라.”

이것은 분산 시스템의 eBPF가 커널 호출을 코드 수정 없이 관찰하는 것과,
서비스 메시의 Envoy Sidecar가 애플리케이션 코드 없이 네트워크 트래픽을 제어하는 것과
같은 계보다. “기존 경계를 투명하게 가로채는” 설계 철학이다.

이 패턴이 에이전트 RL에서 작동한다면, 다른 ML 시스템에도 적용할 수 있다.
추론 서빙 API를 관찰해 자동으로 학습 데이터를 수집하고,
프로덕션 트래픽에서 지속적으로 모델을 개선하는 파이프라인이
Agent Lightning이 열어가는 방향이다.

### “에이전트 개발”과 “에이전트 훈련”의 분업이 가능해진다

Training-Agent Disaggregation은 기술적 분리를 넘어 조직의 역할 분업을 가능하게 한다.

지금까지 에이전트를 RL로 훈련하려면 에이전트 개발자와 ML 연구자가
동일한 코드베이스 위에서 협업해야 했다.
에이전트 개발자는 훈련 로직을 이해해야 하고,
ML 연구자는 에이전트 프레임워크를 이해해야 했다.
서로의 도메인에 들어가야 하는 이 마찰이 에이전트 RL의 실용적 도입을 늦췄다.

Training-Agent Disaggregation은 이 경계를 API 경계로 명확히 만든다.
에이전트 개발팀은 기존 프레임워크와 워크플로우를 그대로 유지하고,
ML팀은 Lightning Server를 독립적으로 운영하며 알고리즘을 실험한다.
두 팀이 공유하는 계약(contract)은 OpenAI 호환 API 스펙 하나다.

프론트엔드와 백엔드를 REST API로 분리하면 독립 배포가 가능해지듯,
에이전트와 훈련을 API로 분리하면 독립 최적화가 가능해진다.
“에이전트 개발자”와 “에이전트 훈련 엔지니어”라는 전문화된 역할이
이 아키텍처 위에서 자연스럽게 분화될 것이다.

### 재토크나이제이션 드리프트는 “숨은 결함”의 전형이다

에이전트 RL 시스템을 구현할 때 대부분의 개발자는
재토크나이제이션 드리프트를 인식하지 못한다.
시스템은 “작동”하고 학습도 “진행”되지만,
off-policy 업데이트로 인해 학습이 예상보다 느리거나 불안정하다.
원인을 찾기 어렵고, 발견하더라도 고치기 복잡하다.

이런 류의 숨은 결함은 모든 ML 시스템에 존재한다.
데이터 전처리 파이프라인의 미묘한 불일치,
학습과 추론의 배치 정규화 동작 차이,
분산 학습의 그래디언트 동기화 타이밍 문제—모두 “작동하지만 잘못된” 상태로 오래 버틴다.

Agent Lightning이 `return_token_ids`로 이 문제를 내부적으로 처리해준다는 것은,
사용자가 알지 못하는 함정을 프레임워크가 미리 막아주는 것이다.
“올바른 기본값(correct defaults)”이 프레임워크의 진짜 가치인 사례다.
