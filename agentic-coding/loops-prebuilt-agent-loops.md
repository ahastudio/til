# loops! — 미리 만들어 둔 에이전트 루프 모음

<https://loops.elorm.xyz/>

## 소개

`loops!`는 “Stop prompting. Start looping.”라는 구호를 내건
코딩 에이전트용 워크플로 모음 사이트다.
elorm이라는 개발자가 만들었고, 핵심 제안은 단순하다.
프롬프트를 한 번 던지고 끝내는 대신,
트리거(trigger)·피드백 게이트(feedback gate)·종료 조건(exit condition)을
갖춘 닫힌 루프(closed-loop) 워크플로를 복사해서 에이전트에게 붙이라는 것이다.
그러면 에이전트가 “스스로 페이스를 조절하며(self-pace)”
작업이 끝날 때까지 반복한다는 발상이다.

사이트는 Cursor, Claude Code, Codex, Gemini CLI, OpenCode 같은
이미 널리 쓰이는 코딩 도구를 대상으로 한다.
각 루프는 복사해 붙일 수 있는 킥오프(kickoff) 텍스트와,
선택적으로 설치 가능한 훅 번들(hook bundle) 형태로 제공된다.
사용자는 루프를 둘러보고(`Browse`), 직접 만든 루프를 제출하고(`Submit`),
번들을 설치하는(`Install`) 흐름을 따른다.

흥미로운 점은 모든 루프가 거의 동일한 골격을 공유한다는 것이다.
`Goal`(목표), `Max iterations`(최대 반복 횟수),
`Between iterations run`(반복 사이에 실행할 명령),
`Exit when`(종료 조건), 그리고 `Step 1`(한 번의 반복에서 할 일)로 구성된다.
마지막에는 거의 모든 루프에 동일한 문구가 붙는다.
“이 루프의 페이스를 스스로 조절하라. 각 반복 후 검사 명령을 실행하고,
출력을 읽고, 종료 조건이 충족되지 않은 경우에만 계속하라.
종료 조건이 통과하거나 최대 반복에 도달하면 멈춰라.
매 반복마다 짧은 상태 업데이트를 주어라.”

## 대표 루프

가장 많이 복사된 루프는 **Ship PR Until Green**(298 copies)이다.
브랜치에서 구현하고, 테스트를 돌리고, 푸시하고, PR을 열고,
CI를 기다린 뒤 모든 체크가 통과해 머지 준비가 될 때까지 반복한다.
검사 명령은 `gh pr checks`이고, 모든 PR 체크가 `success`가 되면 종료하며,
최대 10회 반복한다.

**De-Sloppify Pass**(142 copies)는 구현이 끝난 뒤
정리 패스를 도는 루프다.
디버그 코드를 제거하고, 네이밍을 다듬고, 죽은 분기를 지우고,
프로젝트 컨벤션에 맞춘다.
반복 사이에 `npm run lint && npm test`를 돌리고,
최대 4회 반복한다.

**Spec-First Ship**(66 copies)은 `spec.md` 체크리스트에서 출발한다.
각 반복마다 체크되지 않은 요구사항 하나를 구현하고 검증한 뒤
`[x]`로 표시하고 그 반복을 끝낸다.
`spec.md`에 미체크 항목이 없을 때까지 최대 15회 돈다.

**Build Until Green**(64 copies)은 프로덕션 빌드를 돌려
컴파일·번들링 오류를 고치고 빌드가 성공할 때까지 반복하며,
`npm run build`가 `0`으로 종료될 때까지 최대 10회 돈다.

**PR Babysitter**(60 copies)는 유일하게 interval 방식이다.
`/loop 15m`으로 15분마다 `codex-watch` 라벨이 붙은 열린 PR을 점검해
리뷰어를 재촉하고, CI를 고치고, main보다 뒤처졌으면 리베이스한다.
다른 루프와 달리 명시적 종료 조건이 약하고
“각 PR이 정상이거나 에스컬레이션될 때까지” 도는 감시 루프다.

**Coverage Until Threshold**(51 copies)는
프로덕션 동작을 불필요하게 바꾸지 않으면서
커버리지가 임계값(기본 80%)에 도달할 때까지 집중 테스트를 추가한다.

## 분석

### 루프의 본질은 “프롬프트가 아니라 제어 구조를 표준화”하는 것이다

이 사이트가 파는 것은 프롬프트가 아니다.
정확히는 프롬프트를 감싸는 제어 흐름(control flow)의 템플릿이다.
모든 루프가 `Goal`/`Max iterations`/`Between iterations run`/`Exit when`/`Step 1`이라는
동일한 다섯 칸짜리 틀을 공유한다는 사실이 이를 잘 보여준다.
바뀌는 것은 검사 명령과 종료 조건뿐이고,
그 위를 둘러싼 while 루프의 골격은 고정되어 있다.

이는 의도적인 설계다.
에이전트에게 “잘 해봐”라고 말하는 대신,
“이 명령을 돌려서 그 출력으로 멈출지 말지 판단하라”고
판단 기준을 외부화한다.
다시 말해 자연어 지시를 의사 코드(pseudo-code)에 가깝게 끌어내린다.
`while (exit_condition is false && iter < max) { do_step; run_check; }`라는
전통적인 루프 구조를, 본문이 모호한 채로 에이전트에게 위임하는 셈이다.

여기서 핵심 가정은 “에이전트는 한 번의 응답으로는 끝까지 못 가지만,
검사 명령이 주는 객관적 신호(exit code, CI 상태, 커버리지 수치)를
반복적으로 먹이면 수렴한다”는 것이다.
이 가정은 검증 신호가 명확한 작업(빌드, 테스트, CI)에서는 강력하지만,
신호가 모호한 작업(De-Sloppify의 “slop이 없다”)에서는 흔들린다.
이 차이는 뒤의 비평에서 다시 다룬다.

### 종료 조건의 품질이 루프 전체의 품질을 결정한다

각 루프를 자세히 보면 가치의 90%가 `Exit when` 한 줄에 응축돼 있다.
`Ship PR Until Green`의 “모든 PR 체크가 success”,
`Build Until Green`의 “`npm run build`가 0으로 종료”,
`Coverage Until Threshold`의 “커버리지 임계값 충족”은
모두 기계가 0/1로 판정할 수 있는 단단한 조건이다.
반면 `De-Sloppify Pass`의 “리뷰가 slop을 발견하지 못함”은
판정 주체가 에이전트 자신이라 회로가 닫히지 않는다.

이 구분은 루프를 두 부류로 가른다.
외부 오라클(oracle)이 종료를 판정하는 루프와,
에이전트가 스스로 종료를 판정하는 루프다.
전자는 신뢰할 수 있고 후자는 자기참조적이다.
같은 사이트 안에 두 부류가 섞여 있다는 점은,
“루프”라는 형식이 곧 신뢰성을 보장하지는 않음을 드러낸다.

`Max iterations`가 모든 루프에 빠짐없이 붙어 있는 것도 이 때문이다.
종료 조건을 신뢰할 수 없을 때를 대비한 안전 차단기(circuit breaker)다.
좋은 루프라면 max iterations에 도달하는 일이 드물어야 하고,
자주 max에 부딪힌다면 그것은 종료 조건이 잘못 설계됐다는 신호다.
이 사이트는 그 임계값을 4~20 사이로 손수 튜닝해 두었는데,
이는 곧 “루프는 자동이 아니라 사람이 미리 보정한 자동”이라는
역설을 보여준다.

### interval 루프와 manual 루프는 근본적으로 다른 짐승이다

대부분의 루프는 `manual` 태그를 달고,
“목표 달성까지 가능한 빨리 반복”하는 수렴형 루프다.
반면 `PR Babysitter`만 `interval` 태그와 `/loop 15m`을 달고 있다.
이것은 수렴이 아니라 폴링(polling)이다.
끝나는 게 목적이 아니라 계속 도는 게 목적인,
사실상 cron job에 자연어 판단을 얹은 형태다.

이 차이는 단순한 태그 이상이다.
manual 루프는 “작업 완료”라는 단일 종착점을 향해 달리지만,
interval 루프는 종착점이 없는 상시 감시자다.
후자는 비용 모델과 실패 모드가 완전히 다르다.
15분마다 깨어나 PR 목록을 읽고 판단하는 에이전트는,
잘못 설계되면 무한히 토큰을 태우는 좀비가 된다.

이 둘을 같은 “루프”라는 우산 아래 묶은 것은 마케팅상으로는 깔끔하지만,
개념적으로는 위험하다.
사용자가 manual 루프에 대한 직관을 interval 루프에 그대로 적용하면
(“어차피 종료 조건이 있으니 알아서 멈추겠지”)
비용과 부작용에서 크게 데일 수 있다.

## 비평

### “Stop prompting”은 거짓이다 — 더 정교한 프롬프트로 갈아탔을 뿐이다

사이트의 핵심 카피는 “프롬프트를 멈추고 루프를 시작하라”다.
그러나 실제로 복사되는 것은 정교하게 구조화된 프롬프트다.
`Goal`, `Exit when`, `Self-pace this loop...`로 이어지는 본문은
명백히 프롬프트이고, 오히려 즉흥적인 한 줄 프롬프트보다 훨씬 길고 정형화돼 있다.
이 워크플로는 프롬프팅을 없앤 게 아니라,
프롬프팅에 표준 양식을 부과한 것이다.

이 구분이 중요한 이유는, 카피가 약속하는 “패러다임 전환”이
실은 “프롬프트 템플릿화”라는 훨씬 소박한 현실이기 때문이다.
루프의 본문을 읽어보면 결국 사람이 좋은 검사 명령과 좋은 종료 조건을
손수 골라 넣어야 한다.
`npm test`인지 `npm run build`인지 `gh pr checks`인지를 고르는 일,
임계값을 80%로 둘지 90%로 둘지 정하는 일 —
이 판단들이 루프의 성패를 가르며, 그것은 전적으로 프롬프트 작성 행위다.

따라서 더 정직한 카피는 “프롬프트를 멈추라”가 아니라
“프롬프트를 재사용 가능한 제어 구조로 굳혀라”였을 것이다.
이 사이트의 진짜 가치는 자동화에 있는 게 아니라
좋은 종료 조건의 카탈로그를 제공하는 데 있다.
그런데 정작 그 카탈로그적 가치를 카피가 가리고 있다.

### 자연어 종료 조건은 결정론적 루프의 신뢰성을 흉내 낼 수 없다

전통적인 셸 루프 `until npm run build; do fix; done`은
빌드의 exit code라는 결정론적 신호로 닫힌다.
반면 이 사이트의 루프는 동일한 명령을 돌리되,
그 출력을 “읽고 판단하는” 주체가 LLM이다.
`Build Until Green`처럼 종료 조건이 exit code 0인 경우라면
LLM의 판단이 거의 형식적이지만,
`De-Sloppify Pass`처럼 “slop이 없는가”를 LLM이 판정하면
루프는 결정론을 잃는다.

문제는 LLM이 종료 조건을 평가하는 바로 그 모델이라는 데 있다.
자기가 짠 코드를 자기가 “충분히 깨끗하다”고 판정하는 구조는
이해충돌(self-grading)에 가깝다.
인간 코드 리뷰가 별도의 사람을 두는 이유,
TDD가 코드와 별개로 테스트를 두는 이유가 바로 이 분리에 있다.
루프가 검사 명령(`npm run lint && npm test`)을 끼워 넣어
부분적으로 외부화를 시도하긴 하지만,
린트가 잡지 못하는 “naming을 다듬었는가”의 최종 판정은 여전히 자기 채점이다.

결국 이 루프들은 신뢰도 스펙트럼 위에 놓인다.
한쪽 끝에는 CI/빌드처럼 외부 오라클이 판정하는 신뢰 가능한 루프가,
다른 쪽 끝에는 자기 채점에 의존하는 약한 루프가 있다.
사이트는 둘을 같은 비주얼과 같은 “copies” 카운터로 나란히 진열한다.
사용자는 De-Sloppify의 142 copies를 보고 신뢰하지만,
그 신뢰는 Ship PR Until Green의 신뢰와 질적으로 다르다.

### “copies” 지표는 효능이 아니라 호기심을 센다

이 사이트의 유일한 사회적 증거(social proof)는
각 루프 옆의 “298 copies”, “142 copies” 같은 숫자다.
그러나 복사 횟수는 그 루프가 실제로 작업을 완수했는지를 전혀 말해주지 않는다.
복사는 가장 비용이 낮은 행위이고,
호기심·북마크·실패 후 재시도 모두가 카운트를 올린다.
효능을 측정하려면 “이 루프를 돌린 사람 중 몇 %가 실제로 green에 도달했는가”가 필요한데,
그런 지표는 어디에도 없다.

`Hardened`라는 배지가 `Build Until Green`과 `Coverage Until Threshold`에 붙어 있는 것은
이 공백을 메우려는 시도로 보인다.
하지만 “Hardened”가 무엇을 기준으로 부여되는지,
누가 검증했는지는 드러나 있지 않다.
배지와 복사 카운터는 모두 신뢰의 대용물일 뿐,
루프의 실제 수렴률을 대체하지 못한다.

이 비판이 사소하지 않은 이유는,
잘못 설계된 루프가 실패할 때의 비용이 0이 아니기 때문이다.
종료 조건이 영원히 충족되지 않는 루프는 max iterations까지 토큰을 태우고,
interval 루프는 무기한 토큰을 태운다.
효능 지표 없이 인기 지표만 노출하는 카탈로그는,
가장 그럴듯해 보이는 루프가 아니라
가장 안전하고 검증된 루프를 고르도록 사용자를 돕지 못한다.

## 인사이트

### 루프의 표준화는 “검증 자산”을 코드베이스의 일급 시민으로 만든다

이 사이트가 무심코 드러내는 더 큰 흐름은,
검사 명령과 종료 조건이 점점 더 명시적인 자산이 되어간다는 점이다.
지금까지 `npm test`, `npm run build`, `gh pr checks`는
CI 설정 파일 어딘가에 묻혀 있는 부수적 디테일이었다.
그러나 에이전트 루프의 세계에서 이것들은
루프가 닫히느냐 마느냐를 결정하는 중심 자산으로 승격한다.

이것의 2차 효과(second-order effect)는 코드베이스의 가치 평가가 바뀐다는 것이다.
지금까지 “테스트 커버리지가 높다”는 것은 품질 지표였다.
앞으로는 “에이전트가 자기 작업을 종료할 수 있는 단단한 신호가 풍부하다”는 것이
에이전트 친화성(agent-readiness)의 지표가 된다.
빠르고 결정론적이며 의미 있는 검사 명령을 가진 레포는
에이전트가 효율적으로 수렴하고, 그렇지 못한 레포는 에이전트가 헤맨다.
이는 “AI가 읽기 좋은 코드베이스”라는 기존 논의를 한 단계 더 밀어,
“AI가 종료 신호를 얻기 좋은 코드베이스”라는 새 축을 만든다.

소스는 이 함의를 전혀 다루지 않는다.
루프를 복사하는 행위에 초점을 맞출 뿐,
그 루프가 잘 돌기 위해 코드베이스 쪽이 무엇을 갖춰야 하는지는 침묵한다.
그러나 실무자라면 정반대 순서로 생각해야 한다.
좋은 루프를 찾기 전에, 자기 레포가 좋은 종료 신호를 제공하는지를 먼저 점검해야 한다.

### 이것은 셸 스크립트의 역사가 한 층 위에서 반복되는 것이다

`until`/`while` 루프로 명령을 반복 실행하는 것은
유닉스 셸이 수십 년간 해온 일이다.
`while ! npm run build; do ...; done`은 새로운 발상이 아니다.
이 사이트가 한 일은 그 루프의 본문(do 블록 안)을
결정론적 코드에서 LLM 호출로 바꾸고,
종료 조건의 판정을 exit code에서 자연어 판단으로 확장한 것이다.

이 구조적 패턴은 역사적으로 익숙하다.
makefile이 빌드 절차를 표준화했고,
CI YAML이 파이프라인을 표준화했으며,
이제 루프 템플릿이 에이전트 반복을 표준화하려 한다.
매번 같은 일이 일어난다.
처음에는 사람이 즉흥적으로 하던 일을,
누군가 선언적 양식으로 굳히고, 그것이 공유 가능한 자산이 된다.
`loops!`의 다섯 칸짜리 틀은 사실상 “에이전트 루프의 Makefile 문법”이다.

이 역사적 렌즈는 미래를 예측하게 해준다.
makefile과 CI YAML이 결국 표준 포맷과 린터와 보안 검사로 진화했듯,
에이전트 루프도 곧 검증 가능한 스키마,
종료 조건의 정적 분석, 비용 상한 정책으로 진화할 것이다.
지금의 자유 형식 복사-붙여넣기 카탈로그는 그 진화의 1세대에 불과하다.
이 단계에서 가장 중요한 통찰은,
“좋은 루프 양식”이 곧 표준화 전쟁의 대상이 되리라는 것이다.

### 자율 루프의 진짜 트레이드오프는 “감독 비용”을 시간축에서 다시 배분하는 것이다

소스가 매끄럽게 넘어가는 긴장은 이것이다.
루프는 감독(supervision)을 없애지 않는다.
감독을 작업 중에서 작업 전후로 옮길 뿐이다.
한 줄씩 프롬프트하던 시절에는 매 단계마다 사람이 개입했다.
루프를 돌리면 그 중간 개입이 사라지는 대신,
사전에 종료 조건을 정밀하게 설계하는 비용과,
사후에 N번의 자율 반복 결과를 한꺼번에 검토하는 비용이 생긴다.

이 재배분은 공짜가 아니다.
중간 개입이 사라지면 에이전트가 잘못된 방향으로
10번 반복한 뒤에야 사람이 알게 되는 위험이 생긴다.
`Max iterations`가 이 위험에 대한 유일한 방어선인데,
그것은 “얼마나 헛돌아도 괜찮은가”를 미리 정하는,
본질적으로 손실 한도(stop-loss) 설정이다.
즉 루프는 통제를 자동화한 게 아니라,
통제 시점을 앞당기고 통제 단위를 거칠게 만든 것이다.

이 트레이드오프가 유리하게 작동하는 조건은 분명하다.
한 번의 잘못된 반복 비용이 낮고(되돌리기 쉽고),
종료 신호가 결정론적이며,
사전 설계 비용을 여러 번의 실행으로 분산할 수 있을 때다.
`Build Until Green`은 이 세 조건을 모두 만족한다.
반대로 비가역적 변경, 모호한 종료 조건, 일회성 작업이라면
루프는 감독 비용을 줄이기는커녕 사후 검토라는 새 부채만 쌓는다.
소스는 모든 루프를 똑같이 매력적으로 진열하지만,
실무자는 자기 작업이 이 세 조건을 만족하는지부터 따져야 한다.
그렇지 않으면 “에이전트가 알아서 끝낼 때까지”라는 약속은
“에이전트가 알아서 망칠 때까지”로 조용히 뒤집힌다.
