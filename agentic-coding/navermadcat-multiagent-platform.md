# NaverMadCat: Claude Code 위에 회사를 올리다

원문: <https://www.youtube.com/watch?v=c-loQfGep5g>

## 요약

NAVER ENGINEERING DAY 2026에서 발표된 세션이다.
발표자 신동민은 Claude Code를 기반으로 멀티에이전트 플랫폼인 NaverMadCat을 직접 설계·운영한 경험을 공유한다.
발표는 총 10개 섹션으로 구성되어 있으며, 플랫폼 아키텍처부터 설치·운영까지 전 과정을 다룬다.

NaverMadCat의 출발점은 Claude Code의 세 가지 구조적 한계다.
첫째, 경험 단절: 세션이 끊기면 어제의 맥락이 사라지고, 개발자가 AI를 위해 메모리 역할을 해야 한다.
둘째, PC 단절: 회사 PC와 집 PC 사이에 동기화 개념이 없어 환경을 매번 다시 세팅해야 한다.
셋째, 단일 에이전트의 한계: 복잡한 작업에서 병렬 처리와 전문 영역별 분업이 불가능하다.
이 세 가지 갭을 동시에 메우기 위해 NaverMadCat이 만들어졌다.

NaverMadCat은 네 개의 레이어로 구성된다.
배포 레이어(Layer 4)는 크로스 플랫폼 설치 스크립트와 환경 진단 도구를 담는다.
인프라 레이어(Layer 3)는 Claude Code의 훅 시스템을 활용한 자동화 엔진이다.
조직 레이어(Layer 2)는 비서실장과 10개 부서, 에이전트 로스터, 대시보드, 비용 추적을 포함한다.
성장 레이어(Layer 1)는 Gnosis라는 자율 성장 시스템으로 에피소드를 누적하고 스킬을 관리한다.

조직 구조는 Claude 자신이 비서실장(Chief of Staff) 역할을 맡고, 그 아래 10개 부서에 각 전문 에이전트가 배치된다.
Architecture(Dev), QA, HR, Operations, CX, Design, Cost, Training, Strategy, Consulting 부서로 구성되며
총 11명의 에이전트가 코어 인력으로 지정되어 HR을 통해서만 변경 가능하다.
비서실장은 요청을 심플(단일 처리)과 팀(복합 협업)으로 자동 분류하며, 팀이 필요한 경우에도 전원 소집이 아닌 필요한 부서만 선별 스폰한다.

자동화의 핵심은 Claude Code가 제공하는 라이프 사이클 훅 시스템이다.
Session Start 시점에 Git pull로 최신화하고, settings.json 버전을 비교해 필요 시 재생성하며, Gnosis 컨텍스트를 주입한다.
Pre-Tool Use 시점에 심플/팀 판정을 재확인하고, 신규 에이전트의 헌법을 자동 생성한다.
Post-Tool Use 시점에는 대시보드 활동 기록, 에피소드 저장, 비서실장 버퍼 누적, 로스터 변경 감지 등 네 개의 훅이 동시에 돌아간다.
Session End 시점에 버퍼를 비우고 Git push로 동기화한다.

멀티 PC 동기화는 Git 레포지토리 하나로 해결한다.
각 PC는 자기만의 싱크 브랜치를 가지며, 작업 후 자신의 브랜치에 푸시하고 마스터로 머지한다.
모든 PC는 마스터에서 풀해 최신 상태를 가져온다.
settings.json처럼 Git 추적 밖의 파일은 템플릿 버전 비교 방식으로 동기화하고, ChromaDB 같은 로컬 DB는 JSON 익스포트와 컨텍스트 캐시 마크다운으로 재부트한다.

## 분석

### 발표의 핵심 주장: AI 도구는 쓸수록 더 잘 알아야 한다

발표 전체를 관통하는 명제는 하나다.
“AI는 쓸수록 더 잘 알아야 한다.”
이 명제가 강렬한 이유는 현재의 AI 도구가 그렇게 동작하지 않는다는 역설을 전제하기 때문이다.
Claude Code는 매일 써도 매번 처음인 도구다.
NaverMadCat은 이 역설을 정면으로 공략하는 시스템으로 설계됐다.

발표자가 선택한 전략은 기존 도구를 교체하는 것이 아니라, Claude Code 위에 인프라를 올리는 것이다.
Claude Code의 훅 시스템, 슬래시 커맨드, CLAUDE.md 메커니즘을 그대로 활용하되,
그 위에 조직 구조·자동화·동기화·성장 레이어를 덧씌운다.
이 접근은 Claude Code의 업데이트를 그대로 흡수할 수 있다는 이점이 있다.
기반 도구의 개선이 자동으로 플랫폼 전체에 전파된다.

발표의 논리 구조는 문제 정의 → 해결 원칙 → 구현 증명의 3단계를 따른다.
세 가지 갭을 먼저 명확히 규정한 뒤, 각 갭에 대한 해결 레이어를 대응시키고,
실제 동작 시나리오로 작동을 증명한다.
이 구조는 청중이 시스템의 복잡성에 압도되지 않도록 설계된 것이다.

### 비서실장 설계의 의도: 오버헤드 없는 라우팅

NaverMadCat의 핵심 설계 결정 중 하나는 비서실장을 별도 에이전트로 스폰하지 않고
Claude 자신이 그 역할을 맡게 한 것이다.
이 결정은 단순해 보이지만, 파이프라인 우회를 구조적으로 막는 동시에 심플 작업의 오버헤드를 제거한다.
별도 에이전트를 거쳐야 한다면 단순한 파일 수정 하나도 에이전트 스폰 비용이 붙는다.
Claude가 직접 처리하면 그 비용이 사라진다.

심플/팀 자동 분류 역시 사용자 경험 설계의 핵심이다.
사용자가 “이거 해줘”라고만 하면, 단순한 일은 즉시 처리되고 복잡한 일은 팀이 꾸려진다.
이 투명성이 없다면 사용자는 매번 “이건 팀이 필요한가?”를 스스로 판단해야 한다.
AI 시스템에서 사용자 판단 부담을 제거하는 것은 채택률(adoption rate)에 직결된다.

전원 소집 금지 원칙도 같은 맥락이다.
10개 부서가 있다고 해서 모든 작업에 10명을 부르면 토큰 비용이 선형으로 증가한다.
필요한 부서만 선별 스폰하는 것은 비용 효율과 응답 속도 양쪽을 모두 최적화하는 설계다.

### 동기화 문제를 Git으로 해결한 것의 함의

“어디서든 동일한 환경”이라는 목표를 Git 레포 하나로 달성하는 접근은 우아하다.
Git은 이미 버전 관리와 분산 협업에 검증된 인프라다.
새로운 동기화 시스템을 만드는 대신 Git의 브랜치 모델을 PC 동기화에 재활용한다.

단, settings.json처럼 Git 추적 밖에 있는 파일의 동기화는 별도 메커니즘이 필요했다.
템플릿에 버전 필드를 두고, 사용자 PC의 마지막 적용 버전과 비교해 재생성하는 방식은
설정 파일의 동기화 문제를 버전 비교 문제로 변환한 것이다.
플레이스홀더(Python 경로, 홈 디렉토리 등)를 실제 경로로 치환하는 단계도
크로스 플랫폼 지원의 핵심 메커니즘이다.

이 설계의 한계는 Git 저장소가 모든 상태의 진실 원본(source of truth)이 된다는 점이다.
ChromaDB 같은 로컬 벡터 DB는 Git으로 직접 추적할 수 없어
JSON 익스포트와 컨텍스트 캐시 마크다운으로 재구성하는 간접 방식을 쓴다.
이 재구성 과정에서 정보 손실이 얼마나 발생하는지는 발표에서 명시되지 않았다.

### Gnosis: 자율 성장 레이어의 위상

발표에서 Gnosis는 “가볍게 소개만”이라는 단서와 함께 제시됐다.
그러나 Gnosis는 단순한 부가 기능이 아니라 NaverMadCat의 존재 이유와 직결된다.
“AI는 쓸수록 더 잘 알아야 한다”는 원칙은 에피소드 누적과 스킬 승격 없이는 완전히 실현되지 않는다.
Gnosis 없이는 NaverMadCat은 더 잘 조직된 기억 없는 시스템에 머문다.

에피소드가 쌓이면 반복되는 성공 패턴은 스킬로 승격되고, 실패 패턴은 헌법에 반영된다.
이 피드백 루프가 작동하려면 에피소드의 품질과 스킬 승격 기준이 중요하다.
어떤 에피소드가 의미 있는지, 언제 스킬로 승격할지를 결정하는 로직이
시스템의 실질적 지능 수준을 결정한다.
이 부분이 별도 세션으로 분리된 것은 그 복잡성을 방증한다.

## 비평

### 시스템의 복잡성은 개인 프로젝트 규모를 넘어설 수 있다

NaverMadCat은 10개 부서, 11명 에이전트, 4개 레이어, 7종의 훅, Git 브랜치 전략, Gnosis 레이어까지
단일 개발자가 설계·운영하기에는 이미 상당한 인지 부하를 요구하는 시스템이다.
발표는 이 복잡성을 구조적 강점으로 제시하지만,
복잡성 자체가 새로운 종류의 유지보수 부담을 낳는다는 점은 다루지 않는다.

훅 스크립트가 오작동하거나 Gnosis 루프가 예상치 못한 방향으로 수렴할 때,
문제를 진단하는 사람은 여전히 개발자 본인이다.
NaverMadCat이 해결하려는 문제(매번 처음인 AI)를 NaverMadCat 자체를 이해하고 디버깅하는 데도
동일한 인지 비용이 발생한다.
“파이썬 닥터만 돌려도 대부분 자동 감지”라는 발표의 언급은 이 비용을 최소화하려는 노력이지만,
완전히 제거할 수는 없다.

발표가 네이버 사내 엔지니어링 데이 대상이라는 점도 고려해야 한다.
이 청중은 Claude Code를 이미 능숙하게 쓰는 엔지니어들이다.
NaverMadCat의 진입 장벽이 일반 개발자에게 얼마나 높은지는 발표에서 검증되지 않는다.

### 비용 최적화 주장과 실제 토큰 소비의 간극

발표는 전원 소집 금지와 필요한 부서만 선별 스폰을 비용 효율의 근거로 제시한다.
그러나 동시에 Post-Tool Use 훅만 해도 네 개가 동시에 돌아가고,
심플 작업에도 활동 기록, 에피소드 저장, 버퍼 누적, 로스터 감지가 따라붙는다.

이 자동화 오버헤드의 실제 토큰 비용을 발표는 수치로 제시하지 않는다.
Cost Director가 실시간으로 비용을 분석한다고 했지만,
NaverMadCat 자체를 운영하는 데 드는 기반 비용(baseline overhead)이
절감 효과를 상쇄하는지 여부는 청중이 판단할 수 없다.
“추정치가 아닌 실제 비용”을 추적한다는 강점이 오히려 이 데이터를 공개해야 한다는 기대를 만든다.

토큰 비용 추적 기능이 있다면, NaverMadCat 없는 동일 작업과의 비교 수치가 발표의 설득력을 크게 높였을 것이다.
없이는 비용 효율 주장이 구조적 설계에 대한 신뢰에만 의존하게 된다.

### Claude Code 의존성은 플랫폼 리스크이기도 하다

NaverMadCat은 Claude Code의 훅 시스템, 슬래시 커맨드, CLAUDE.md, settings.json 구조에 깊이 의존한다.
Anthropic이 이 중 하나의 API나 동작 방식을 변경하면, NaverMadCat의 핵심 레이어가 영향을 받는다.

발표는 이 의존성을 “업데이트를 자동으로 흡수한다”는 장점으로 프레이밍하지만,
업데이트가 하위 호환성을 깨는 경우의 리스크는 다루지 않는다.
실제로 Claude Code는 아직 빠르게 진화 중인 도구다.
오늘의 훅 시스템이 6개월 후에도 같은 방식으로 동작한다는 보장이 없다.

이 리스크는 개인 프로덕티비티 도구에서는 감수할 수 있는 수준이다.
그러나 발표의 제목처럼 “회사”를 이 플랫폼 위에 올린다면,
플랫폼 의존성은 단순한 기술적 리스크가 아니라 운영 리스크가 된다.
“어디서든 동일한 환경”을 보장하는 시스템이 외부 요인으로 하루아침에 깨질 수 있다는 점은
아키텍처 결정에서 명시적으로 다뤄져야 했다.

## 인사이트

### Claude Code의 훅 시스템은 아직 충분히 탐구되지 않은 확장 지점이다

NaverMadCat이 가장 잘 보여주는 것은 Claude Code 훅 시스템의 잠재력이다.
Session Start, Pre-Tool Use, Post-Tool Use, Session End라는 네 가지 훅 포인트는
Anthropic이 공식 문서에서 설명하는 것 이상의 활용이 가능하다.

발표자는 이 훅 포인트를 자동화 엔진으로 전환해 조직 운영의 전 라이프 사이클을 연결했다.
이 접근이 시사하는 것은, 훅 시스템이 단순한 알림이나 로깅을 넘어
AI 작업의 상태 관리와 지식 축적을 위한 범용 인프라 레이어가 될 수 있다는 것이다.
훅 시스템을 중심으로 한 Claude Code 확장 패턴은 아직 커뮤니티에서 표준화되지 않았다.
NaverMadCat은 그 초기 사례 중 하나로, 이 공간에서 더 많은 설계 패턴이 등장할 여지가 있다.

### 조직 메타포는 AI 시스템 설계의 강력한 인지 프레임이다

NaverMadCat이 부서, 비서실장, 채용/해고, 헌법 같은 조직 메타포를 쓰는 것은
단순히 재미있는 이름 붙이기가 아니다.
조직 메타포는 시스템의 경계, 책임, 확장 방식에 대한 사고를 구조화한다.

“어떤 에이전트에게 이 작업을 위임해야 하는가?”는 추상적인 질문이다.
“어떤 부서 매니저를 불러야 하는가?”는 직관적인 질문이다.
메타포가 설계자의 판단을 빠르게 만들고, 새로운 에이전트를 추가할 때도 부서 구조 안에서 자연스럽게 위치를 찾게 된다.
아카이브된 헌법이 있으면 재온보딩을 차단하는 규칙처럼, 조직 메타포는 시스템 일관성을 유지하는 사회적 규범을 코드로 변환하는 역할도 한다.

이 패턴은 AI 에이전트 시스템 설계에 더 폭넓게 적용될 수 있다.
기술적 추상화(서비스, 워커, 큐)보다 조직 메타포(팀, 역할, 책임)가
사람이 운영하고 확장하기 쉬운 시스템을 만드는 데 더 효과적일 수 있다.

### 경험 단절 문제는 AI 도구 전반의 미해결 과제다

NaverMadCat이 해결하려는 “매번 처음인 AI”는 Claude Code만의 문제가 아니다.
대부분의 LLM 기반 도구는 세션 간 지속성을 기본으로 제공하지 않는다.
메모리 기능을 갖춘 도구들도 대부분 단순한 요약이나 키-값 저장 수준에 머문다.

Gnosis가 에피소드에서 스킬을 승격하고, 실패 패턴을 헌법에 반영하는 방식은
단순한 메모리를 넘어 경험에서 패턴을 추출하는 과정이다.
이것은 인간의 전문성 형성과 유사한 구조다.
반복 경험이 암묵지(tacit knowledge)를 형성하고, 그 암묵지가 행동 원칙을 만드는 과정.

AI 도구가 진정한 생산성 파트너가 되려면 이 경험 축적 문제가 반드시 해결되어야 한다.
NaverMadCat의 접근은 그 해결을 AI 모델 자체가 아닌 인프라 레이어에서 시도한다는 점에서 독특하다.
모델이 기억하는 것이 아니라, 시스템이 기억을 조직하고 주입한다.
이 접근의 장기적 한계는 모델의 컨텍스트 창이 아무리 커도 누적된 경험의 총량을 결국 따라가지 못한다는 것이다.
경험 요약과 중요도 판단의 품질이 시스템의 성능 상한을 결정할 것이다.

### Git은 AI 에이전트 상태 동기화의 예상치 못한 기반 인프라가 될 수 있다

NaverMadCat이 멀티 PC 동기화에 Git을 사용하는 것은 자연스러운 선택처럼 보이지만,
그 함의는 생각보다 넓다.

Git은 이미 코드의 버전 관리, 협업, 분기 실험을 위한 전 세계 표준 인프라다.
NaverMadCat은 이 인프라를 AI 에이전트의 지식, 경험, 설정 상태를 관리하는 데 재활용한다.
브랜치 전략으로 각 PC의 변경 출처를 추적하는 것은, AI 에이전트가 여러 환경에서 동작할 때
“이 지식은 어디서 왔는가”를 추적할 수 있다는 의미다.

이것이 확장되면, 여러 팀이 공유하는 AI 에이전트 플랫폼에서
각 팀의 에이전트 헌법과 스킬을 Git으로 관리하고, PR 리뷰를 통해 변경을 승인하는 워크플로가 가능해진다.
에이전트의 행동 원칙을 코드로 관리하고 리뷰하는 것은
“AI 거버넌스”라는 추상적 과제를 엔지니어링 워크플로 안으로 끌어들이는 구체적 방법이다.
NaverMadCat은 이 가능성의 초기 형태를 보여주고 있다.
