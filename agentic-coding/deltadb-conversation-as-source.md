# DeltaDB: 대화가 소스가 되는 버전 관리

원문: [Software Is Made Between Commits — Zed's Blog](https://zed.dev/blog/introducing-deltadb)

HN 토론: <https://news.ycombinator.com/item?id=48492533> (316점, 212개 댓글)

## 요약

Zed의 Nathan Sobo가 소개한 DeltaDB는 에이전틱 코딩 시대의 버전 관리를 위해 설계된 새로운 데이터베이스 레이어다.
핵심 주장은 “코드를 생성하는 대화(conversation)가 소프트웨어의 진정한 소스가 되고 있다”는 것이다.
커밋 스냅샷 중심의 전통적 Git 모델은 이 연속적 대화 주도 개발 방식을 지원하지 못한다.

DeltaDB는 커밋 스냅샷 대신 개별 작업의 스트림인 “델타(delta)”로 코드 이력을 표현한다.
각 델타는 안정적인 아이덴티티를 부여받아, 코드가 어떤 진화 시점에 있었는지를 정확히 가리킬 수 있다.
대화는 라인 번호가 아닌 델타에 앵커링되어, 원본 작성 시점이나 현재 시점의 코드 모두에 접근할 수 있다.

실시간 협업도 재정의된다.
커밋 전에도 팀원들이 작업 중인 코드에 주석을 달 수 있고, 풀 리퀘스트와 리뷰 스레드의 의식(ceremony)이 필요 없어진다.
에이전트들도 대화 맥락에 접근하고 이전 에이전트의 구현 결정에 대해 소통할 수 있다.

## 분석

### Git의 근본적 가정이 에이전틱 코딩과 충돌한다

Git은 “의미 있는 변경 단위”를 인간 개발자가 명시적으로 선언하는 구조로 설계되었다.
커밋은 인간의 판단이 개입하는 의미 있는 중단점이다.

에이전틱 코딩 환경에서는 이 가정이 무너진다.
에이전트는 수백 개의 파일을 수백 번 수정하면서 최종 결과를 만들어낸다.
중간 과정의 모든 변경을 커밋으로 기록하면 히스토리가 무의미해지고,
최종 결과만 커밋하면 에이전트가 어떤 경로로 그 결과에 도달했는지를 잃게 된다.

DeltaDB의 델타 스트림 방식은 이 딜레마를 해결하는 하나의 접근법이다.
모든 변경을 기록하되, 커밋처럼 인간이 선언하는 의미 단위가 아니라 작업 자체의 흐름을 따라간다.

### 대화를 코드의 “왜”로 보존한다

소프트웨어 유지보수에서 가장 어려운 질문은 “이 코드는 왜 이렇게 작성되었는가”다.
커밋 메시지, 주석, 문서가 이 질문에 답하려 하지만 항상 불완전하다.

DeltaDB가 제안하는 대화 앵커링은 이 문제에 대한 구조적 해법이다.
에이전트와의 대화 — 요구사항, 설계 결정, 절충안 논의 — 가 코드 변경과 직접 연결되면,
미래의 개발자(혹은 에이전트)가 왜 이 코드가 이런 형태인지를 대화를 통해 추적할 수 있다.

이것은 단순한 편의 기능이 아니라, 코드베이스의 이해 가능성을 높이는 구조적 변화다.

### 협업의 비동기성과 실시간성이 통합된다

전통적인 풀 리퀘스트 기반 협업은 비동기성을 기본값으로 한다.
작업 완료 → 커밋 → PR 생성 → 리뷰 → 머지라는 단계적 흐름이다.

DeltaDB의 실시간 협업 모델은 이 경계를 제거한다.
커밋 전에도 팀원이 주석을 달 수 있다는 것은 작업이 진행되는 동안 실시간 피드백이 가능하다는 의미다.
이는 짝 프로그래밍의 밀도와 비동기 리뷰의 유연성을 동시에 제공하는 새로운 협업 형태다.

## 비평

### Git 생태계와의 호환성이 핵심 장벽이다

DeltaDB가 제시하는 비전이 매력적이더라도, Git 생태계의 깊이는 단기간에 극복하기 어렵다.
GitHub, GitLab, CI/CD 파이프라인, 코드 리뷰 도구, 배포 시스템 모두 Git의 커밋 기반 모델에 의존한다.

델타 스트림 방식을 도입하려면 이 생태계 전체가 새로운 모델을 이해하거나,
DeltaDB가 Git으로의 매핑 레이어를 제공해야 한다.
그 매핑 레이어가 얼마나 자연스럽게 작동하느냐가 실용적 채택의 가장 큰 결정 요인이 될 것이다.

WorldMaker는 Git이 이미 잦은 자동 커밋을 효과적으로 처리하는 메커니즘을 갖추고 있으며, `git merge --no-ff`와 `--first-parent` 플래그를 조합하면 세부 이력과 깔끔한 상위 구조를 동시에 유지할 수 있다고 주장했다.[^WorldMaker]
이 관점에서 DeltaDB는 새로운 솔루션이 아니라 Git 프런트엔드 인터페이스의 부족한 UI를 채우는 것으로 볼 수 있다.
실질적인 필요는 드릴다운 기능을 지원하는 더 나은 Git 도구이지, 새로운 버전 관리 패러다임이 아닐 수 있다.

jmole은 구글이 소스 제어 시스템 Piper를 통해 Ctrl+S 수준의 세분도로 약 10년간 모든 개발자의 전체 이력을 보존해왔다고 밝혔다.[^jmole]
DeltaDB가 참신해 보이는 아이디어가 이미 대형 기업에서 오래전부터 실현되고 있었다는 것이다.

### 대화 보존의 프라이버시와 부피 문제

대화를 코드의 소스로 영구 보존하는 것은 새로운 종류의 문제를 만든다.
코드 작성 과정에서 나누는 대화에는 임시적인 아이디어, 폐기된 방향, 때로는 민감한 맥락이 포함될 수 있다.
이것이 코드베이스와 함께 영구적으로 버전 관리된다면, 기업 환경에서는 프라이버시와 기밀 유지의 문제가 된다.

또한 에이전트와의 대화는 매우 장황할 수 있다.
모든 대화를 보존하면 저장소의 크기가 코드보다 대화 데이터로 인해 더 크게 증가할 수 있다.

tomjakubowski는 이 문제를 개인적 관점에서 제기했다.[^tomjakubowski]
“커밋 사이에 쓰는 코드는 내 생각이다. 코드를 쓰고 지우고 다시 쓰면서 생각한다.”
커밋에 담기는 코드는 다른 사람이 이해할 수 있도록 다듬어진 것이지, 사고 과정 자체를 직렬화해서 버전 관리하고 공개하길 원하지 않는다는 것이다.

jchw는 더 나아가 이 방식이 24시간 돌아가는 화면 녹화와 같다며, 커밋 관행을 저하시킬 것이라고 우려했다.[^jchw]
사람들이 팀원을 위해서는 문서를 작성하지 않지만 LLM에게는 기꺼이 설명하면서 갑자기 `CLAUDE.md`에 문서화가 이루어지는 현상이, 오히려 올바른 방향의 문서화를 왜곡한다는 지적이다.

### 단일 에디터 종속이 생태계를 제한한다

DeltaDB가 Zed의 기능으로 출시된다는 것은 그 혜택을 Zed 사용자만 누릴 수 있다는 의미다.
대화 앵커링이나 실시간 협업의 가치는 팀 전체가 같은 도구를 쓸 때 극대화된다.
팀에 VSCode, Neovim, Cursor 사용자가 혼재한다면 DeltaDB의 협업 기능은 부분적으로만 작동한다.

## 인사이트

### 코드 이해 가능성의 위기가 에이전틱 시대에 심화된다

에이전트가 대량의 코드를 생성할수록, “이 코드는 왜 이렇게 되어 있는가”라는 질문에 답하기가 점점 어려워진다.
인간이 작성한 코드는 작성자가 있고, 커밋 메시지가 있고, 맥락이 있다.
에이전트가 생성한 코드는 중간 과정 없이 결과만 존재할 수 있다.

DeltaDB의 대화 앵커링이 이 문제에 대한 하나의 해법이라면, 더 근본적인 질문은 이것이다:
에이전틱 코딩이 확산되면서 코드베이스의 “설명 가능성”은 어떻게 유지될 것인가?
이 질문은 단순한 도구 선택의 문제가 아니라 소프트웨어 공학의 방법론적 전환을 요구한다.

Lindby는 중간 과정의 잡음을 보존하는 대신 깔끔한 커밋 히스토리를 작성하는 것이 더 나은 접근이라고 반론했다.[^Lindby]
“커밋으로 만들어내는 이야기가 실제 일어난 순서가 아니더라도, 왜 이렇게 되었는지를 설명한다”는 것이다.
이 관점은 DeltaDB가 해결하려는 문제가 새로운 저장 계층이 아니라 커밋 문화의 개선으로도 해결 가능하다는 주장이다.

### 버전 관리는 협업 도구로의 정체성 전환을 요구받는다

Git은 본래 코드의 이력을 관리하는 도구로 시작했고, 협업은 부차적 기능으로 붙었다.
DeltaDB의 방향은 이 순서를 뒤집는다 — 실시간 협업과 대화 보존이 핵심이고, 버전 이력은 그 결과물이다.

이 전환이 의미하는 바는 크다.
미래의 버전 관리 도구는 소스 코드를 관리하는 도구가 아니라, 인간과 에이전트의 협업 과정 전체를 추적하는 도구가 될 수 있다.
DeltaDB는 그 방향으로의 첫 번째 구체적 시도 중 하나다.

### “대화가 소스다”라는 명제는 소프트웨어의 법적 귀속도 바꾼다

코드가 대화의 결과물로 생성된다면, 그 코드의 저작권은 누구에게 귀속되는가?
프롬프트를 작성한 사람, 대화를 이끈 사람, 에이전트를 실행한 사람, 아니면 에이전트 자체?

현행 저작권법은 인간 창작자를 전제로 설계되어 있다.
“대화가 소스다”라는 패러다임이 정착하면, 소프트웨어 귀속과 지식재산의 법적 프레임워크도 함께 재구성되어야 한다.
DeltaDB가 이 대화들을 영구적으로 기록한다면, 그것은 미래의 저작권 분쟁에서 중요한 증거 자료가 될 수도 있다.

prodigycorp는 Zed가 혁신적인 아이디어와 품질로 인해 Anthropic이나 OpenAI에 인수될 것이 불가피하다고 말했다.[^prodigycorp]
만약 그렇게 된다면, DeltaDB에 저장된 모든 대화 이력의 소유권과 통제권이 AI 기업에게 넘어갈 수 있다.
“대화가 소스다”라는 패러다임의 가치가 개발자에게 돌아오려면, 그 데이터가 단일 제공자에 귀속되지 않아야 한다.
