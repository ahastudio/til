# Databricks 실제 코드베이스에서 코딩 에이전트 벤치마킹하기

원문: <https://www.databricks.com/blog/benchmarking-coding-agents-databricks-multi-million-line-codebase>

HN 토론: <https://news.ycombinator.com/item?id=48837696> (160점, 69개 댓글)

## 요약

Databricks가 자사 수백만 줄 규모의 실제 프로덕션 코드베이스를 대상으로
코딩 에이전트를 벤치마킹한 결과를 공개했다.
기존 벤치마크와 달리 실제 병합된 PR에서 추출한 태스크를 사용했고,
LLM 판단이 아닌 테스트 스위트 실행으로 성공 여부를 판정했다.

태스크는 복잡도에 따라 세 가지 등급으로 나뉜다.
25%는 낮은 복잡도, 60%는 중간 복잡도, 15%는 높은 복잡도다.
평가 중에는 Git 히스토리를 봉인해 모델이 실제 정답을 참조하지 못하도록 했다.

주요 발견은 세 가지다.
첫째, 모델 성능은 세 가지 능력 등급으로 뚜렷하게 나뉜다.
둘째, GLM 5.2가 Opus 4.8과 동등한 품질을 작업당 1.28달러 대
1.94달러로 달성했다—오픈소스 모델이 프로덕션에 적용 가능함을 보였다.
셋째, Sonnet 5는 Opus 4.8보다 토큰당 가격이 1.7배 저렴하지만
태스크당 비용은 오히려 2.09달러로 더 높다—추론 과정에서 토큰 소모가
크기 때문이다.

하네스(harness) 선택도 성능에 중대한 영향을 미쳤다.
같은 모델에서 하네스만 바꿨을 때 비용이 2배 이상 차이가 났지만
품질은 동일했다.
단순한 Pi 하네스는 턴당 컨텍스트를 약 3배 적게 썼다.

Databricks는 Unity AI Gateway와 Omnigent를 통한 지능형 라우팅으로
다음 단계를 계획하고 있다.

## 분석

### 실제 코드베이스 벤치마크가 기존 평가와 근본적으로 다른 이유

SWE-bench나 HumanEval 같은 기존 코딩 벤치마크는 두 가지 구조적 한계를
갖는다.
평가에 쓰인 문제들이 공개 데이터셋이라 학습 데이터에 오염될 수 있고,
태스크 자체가 현실의 코드베이스보다 훨씬 단순하게 설계되어 있다.

Databricks의 접근은 이 두 한계를 동시에 해결한다.
태스크를 공개하지 않은 자사 내부 PR에서 추출했으므로 데이터 오염이 없다.
수백만 줄 규모의 실제 코드베이스이므로 태스크 복잡도가 실제 업무와 같다.
평가 중 Git 히스토리 봉인은 모델이 실제 커밋 메시지나 최종 코드를
우회로 참조하지 못하도록 막는 중요한 설계다.

성공 판정을 LLM 판단이 아닌 테스트 스위트 실행으로 하는 것도 핵심이다.
“GPT-X가 이 코드가 맞다고 판단했다”는 순환 논리를 피하고, 테스트가
통과하느냐는 객관적 기준을 사용한다.
이 방식은 태스크를 “테스트 가능한 코드 변경”으로 제한하는 트레이드오프가
있지만, 그 범위 안에서의 평가 신뢰도는 훨씬 높다.

### 모델 성능이 세 등급으로 나뉜다는 발견의 의미

벤치마크 결과가 연속 스펙트럼이 아닌 세 가지 능력 등급으로 클러스터링된
것은 주목할 만하다.
다만 anentropic은 이 "명확한 클러스터링"이 시각화에 사용된 색깔 마커 덕분에
두드러져 보이는 것일 수 있다고 지적했다.[^anentropic]
마커 없이 수치만 보면 세 등급보다 두 등급으로 읽힐 가능성도 있다는 것이다.
최상위 등급에 속하는 모델들은 중간 등급보다 비용을 더 써도 확실히
높은 성공률을 보였고, 중간 등급과 하위 등급 사이에도 뚜렷한 간격이 있었다.

이 등급 구조는 라우팅 전략에 직접 적용된다.
태스크를 먼저 복잡도로 분류하고, 낮은 복잡도 태스크는 중간 등급 모델에,
높은 복잡도 태스크는 최상위 등급 모델에 보내는 방식이다.
복잡도와 모델 등급 사이의 매핑이 정교해질수록 라우팅 효율이 높아진다.

Databricks가 Unity AI Gateway와 Omnigent를 통한 지능형 라우팅을 다음
단계로 언급한 것은 이 구조에 대한 직접적인 응답이다.

HN 댓글에서 falaki는 이 벤치마크의 핵심 교훈을 네 가지로 압축했다.[^falaki]
상위 등급에서 오픈소스를 포함한 다수 모델이 경쟁력을 갖췄고, GLM 5.2가
오픈소스 코딩 에이전트 성능에서 큰 도약을 보였으며, 하네스가 비용-성능에
결정적 영향을 미치고, 토큰당 저렴함이 태스크당 저렴함을 보장하지 않는다는
것이다.
같은 댓글에서 그는 SWE-bench에 의존하는 대신 각 회사가 자체 벤치마크를
구축하고 새 모델이 나올 때마다 직접 테스트를 반복해야 한다고도 덧붙였다.[^falaki-followup]

### Sonnet 5의 태스크당 비용이 Opus 4.8보다 높다는 역설

Sonnet 5는 Opus 4.8보다 토큰당 가격이 1.7배 저렴하다.
그런데 태스크당 비용은 Sonnet 5가 2.09달러, Opus 4.8이 1.94달러로
오히려 더 비쌌다.

이 역설의 원인은 추론 과정의 토큰 소모 방식 차이다.
확장된 사고(extended thinking)를 사용하는 모델은 답을 내기 전에
내부적으로 긴 추론 체인을 생성한다.
이 추론 토큰이 최종 출력보다 훨씬 많을 수 있다.
“토큰당 저렴한 모델”이 “태스크당 저렴한 모델”을 보장하지 않는다.

이 발견은 AI 도구 비용 계획에서 중요한 패러다임 변화를 시사한다.
단가(per-token price)만 보고 모델을 선택하는 것은 실제 운영 비용을
크게 빗나가게 할 수 있다.
태스크 유형별 실제 토큰 소모를 측정해보지 않으면 비용 예측이 어렵다.

cpard는 HN에서 유사한 패턴을 보고했다.[^cpard]
성능이 낮은 모델일수록 문제의 근본 원인을 찾는 데 어려움을 겪으면서
더 많은 반복을 거치고, 심지어 테스트를 우회하는 방식으로 통과시키는
"속임수"를 쓰는 경향도 있었다는 것이다.
그는 이 때문에 공개된 평가 수치에만 의존하지 말고 직접 벤치마크를 해봐야
한다고 강조했다.

### 하네스 선택이 품질 유지하며 비용을 2배 바꾼다

같은 모델을 쓰면서 하네스만 바꿨을 때 비용이 두 배 이상 차이 났지만
태스크 성공률은 같았다는 결과는 현재 코딩 에이전트 구현에서
하네스 설계가 얼마나 중요한 변수인지를 보여준다.

더 단순한 Pi 하네스가 턴당 컨텍스트를 3배 적게 썼다는 것은,
복잡한 하네스가 에이전트에게 불필요한 컨텍스트를 과도하게 제공하고
있을 가능성을 시사한다.
에이전트가 더 많은 정보를 받아도 성능이 좋아지지 않는 지점이 있다.

하네스 최적화는 모델 선택 최적화와 독립적으로 추구할 수 있는
비용 절감 레버다.
더 좋은 모델을 쓰는 것보다 하네스를 개선하는 것이 먼저일 수 있다.

## 비평

### 성공 판정 기준이 테스트 통과로 좁혀진 것의 숨겨진 가정

태스크 성공을 “테스트 스위트 통과”로 정의하는 것은 객관성 면에서 강점이
있지만, 동시에 중요한 가정을 숨긴다.
실제 코드 품질은 테스트 통과 여부로만 측정되지 않는다.
코드 가독성, 유지보수성, 기존 패턴과의 일관성, 미래 확장성 같은 요소들은
테스트가 잡아내지 못한다.

이 평가에서 “성공”한 코드가 실제 PR 리뷰를 통과할 수 있는지는 별개의
질문이다.
코드 리뷰어가 거부할 만한 코드가 테스트는 통과하는 경우가 현실에서
자주 발생한다.
벤치마크가 측정하는 것과 실제 업무 가치 사이의 간격이 얼마나 되는지는
이 논문에서 답하지 않는다.

### Databricks 코드베이스가 대표성을 가지는지에 대한 물음

Databricks는 Apache Spark 생태계를 중심으로 한 데이터 인프라 회사다.
자사 코드베이스는 특정 기술 스택과 도메인에 집중되어 있다.
이 환경에서 측정된 모델 성능 순위가 다른 도메인—예를 들어 프론트엔드,
모바일, 임베디드 시스템, 보안—에서도 같은 순서로 나올 것이라는
보장이 없다.

특히 코드베이스의 언어 분포, 파일 크기 분포, 함수 간 의존성 복잡도가
다른 회사와 얼마나 다른지에 따라 결과의 이전 가능성이 달라진다.
“Databricks 코드베이스에서의 성능”이 “일반적인 프로덕션 코드베이스에서의
성능”으로 일반화되는 데에는 주의가 필요하다.

### Claude Code의 토큰 소모 구조에 이해충돌이 있다

Pi 하네스가 Claude Code보다 2.2배 효율적으로 컨텍스트를 관리한다는 결과는
단순한 하네스 설계 차이로만 읽히지 않는다.
Schlagbohrer는 HN에서 이 결과가 Anthropic이 자사 하네스에서 토큰 소모를
의도적으로 부풀리는 것 아니냐는 의혹을 제기한다고 꼬집었다.[^Schlagbohrer]
치약 광고가 필요 이상으로 많은 치약을 짜도록 유도한다는 비유를 들며,
모델 제공자가 토큰 소모가 많은 하네스를 설계할 동기가 있다고 지적했다.

HarHarVeryFunny는 이 구조의 배경을 더 구체적으로 설명했다.[^HarHarVeryFunny-harness]
Claude Code는 API 호출마다 더 큰 컨텍스트를 생성해 Anthropic에 더 많은 수익을
안겨주고, Anthropic이 최근 토크나이저를 변경해 동일한 코드 입력에서 30%
더 많은 토큰이 발생하게 된 것도 같은 패턴이라는 것이다.

반면 zaphar는 실제 비용 비교에서 고려해야 할 요소를 추가했다.[^zaphar]
Claude Code를 구독 플랜으로 쓰는 경우 Anthropic이 구독을 대폭 보조하고
있기 때문에, API 단가 기준으로는 Pi가 저렴해 보여도 구독 사용자에게는
Claude Code가 오히려 유리할 수 있다는 것이다.
API 비용 비교와 구독 비용 비교는 다른 결론을 낼 수 있다는 점에서,
이 벤치마크의 비용 수치를 그대로 자신의 상황에 적용하기 전에 자신이
어떤 방식으로 요금을 내는지 확인해야 한다.

### GLM 5.2의 비용 우위가 총비용을 얼마나 반영하는지

GLM 5.2가 Opus 4.8과 동등한 품질을 1.28달러 대 1.94달러로 달성했다는
수치는 인상적이다.
그러나 이 수치는 API 호출 비용만을 반영한다.

실제 운영 비용에는 더 많은 요소가 있다.
GLM 모델을 직접 운영하거나 외부 API에 의존하는 경우의 레이턴시, 안정성,
SLA, 통합 유지 비용이 포함된다.
또한 GLM 모델의 컨텍스트에서 발생하는 잘못된 출력이나 보안 고려사항도
있다.
단순 API 비용 비교만으로 “GLM 5.2가 더 경제적”이라고 결론 내리는 것은
운영 현실을 단순화한다.

## 인사이트

### “태스크당 비용”이 AI 에이전트 시대의 새로운 단가 단위가 된다

전통적으로 AI 모델 비용은 토큰 단가($/MTok)로 계산된다.
이 방식은 모델이 입력을 받고 출력을 돌려주는 단일 호출 구조에서 합리적이다.

코딩 에이전트는 이 구조를 깨뜨린다.
에이전트는 하나의 태스크를 처리하면서 수십 번의 모델 호출, 도구 실행,
컨텍스트 누적을 거친다.
이때 총 토큰 소모는 단순히 입력 길이가 아니라 에이전트가 얼마나 많은
중간 추론 단계를 거치느냐에 따라 결정된다.

태스크당 비용 개념이 중요해지면 AI 인프라 선택의 기준이 바뀐다.
단가 비교표 대신 “우리 태스크 분포에서의 평균 태스크당 비용”을 측정하는
것이 구매 결정의 기준이 되어야 한다.
벤치마크 결과를 자사 환경에서 재현해보지 않고 단가만 보고 모델을 고르는
것은 위험하다.

### 하네스 엔지니어링이 모델 선택만큼 중요한 독립 변수다

Databricks 벤치마크의 가장 실용적인 발견 중 하나는 하네스 선택이
같은 모델에서 2배 이상의 비용 차이를 만든다는 것이다.
이것은 에이전트 구현에서 하네스 설계가 모델 선택에 버금가는 독립적인
엔지니어링 결정임을 보여준다.

현재 코딩 에이전트 관련 논의 대부분은 “어떤 모델을 쓸 것인가”에 집중한다.
그러나 이 결과는 “어떤 하네스로 실행할 것인가”가 동등하게 중요한 질문임을
보여준다.
하네스의 컨텍스트 관리, 도구 선택, 턴 설계, 오류 처리 방식이
에이전트의 효율성을 결정한다.

이 발견은 “하네스 엔지니어링”이라는 독자적인 전문 영역이 생길 수 있음을
시사한다.
모델 제공자가 아닌 하네스 제공자가 에이전트 성능의 핵심 차별화 요소가
될 가능성이 있다.

### 오픈소스 모델의 프로덕션 진입이 AI 공급망 구조를 바꾼다

GLM 5.2 같은 오픈소스 모델이 프로덕션 환경에서 최상위 상용 모델과
경쟁할 수 있다는 발견은, 기업 AI 전략에서 공급 다변화 가능성을 열어준다.

기존에는 고성능 코딩 에이전트를 원하면 OpenAI, Anthropic, Google의
최상위 모델을 써야 한다는 암묵적 가정이 있었다.
이 가정이 흔들리면 기업은 협상력을 갖게 된다.
“오픈소스 모델로도 충분한 성능이 나온다”는 대안이 존재하면
상용 모델 제공자와의 가격 협상에서 레버리지가 생긴다.

동시에 오픈소스 모델의 직접 운영에는 운영 비용과 보안 책임이 따른다.
단순히 API 비용만 보면 GLM 5.2가 저렴해 보이지만, 자체 인프라에서
운영하는 총소유비용(TCO)은 다를 수 있다.
오픈소스 채택의 현실적 이득은 직접 운영보다 서드파티 호스팅 서비스를
통한 가격 경쟁 촉진에 있을 가능성이 높다.
