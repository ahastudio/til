# ACE: 에이전트가 경험에서 배우는 학습 루프

<https://github.com/kayba-ai/agentic-context-engine>

HN 토론: <https://news.ycombinator.com/item?id=47289274> (5점, 0개 댓글)

## 소개

ACE(Agentic Context Engine)는 AI 에이전트가 세션 간에 경험을 기억하고 실수를
반복하지 않도록 하는 Python 오픈소스 프레임워크다. Stanford·SambaNova 공동 논문
(arXiv:2510.04618)과 Dynamic Cheatsheet 논문(arXiv:2504.07952)을 기반으로
Kayba가 구현했다. GitHub 스타 2,314개(2026년 6월 기준), 포크 281개로 활발하게
성장 중이며, YC 출신은 아니지만 주목받는 독립 프로젝트다.

핵심 구조는 Skillbook을 중심으로 세 역할이 순환하는 학습 루프다. Agent는
Skillbook에 저장된 전략을 활용해 태스크를 실행하고, Reflector는 실행 트레이스를
분석해 무엇이 작동했고 실패했는지 추출하고, SkillManager는 Skillbook을 관리하며
전략을 추가·수정·제거한다. Recursive Reflector(RR)가 핵심 혁신으로, 단일 패스로
트레이스를 요약하는 대신 샌드박스 환경에서 Python 코드를 작성·실행하며 패턴을
찾고 오류를 격리한다.

벤치마크 결과는 구체적으로 제시된다. Sierra Research의 Tau2 항공 도메인 멀티스텝
태스크에서 Claude Haiku 4.5 에이전트에 15개 전략을 학습시켰을 때 pass^4(4번
독립 시도 모두 성공 확률) 기준 2배 일관성 향상을 달성했다. 보상 신호 없이, 훈련
데이터 없이 달성한 결과다. 브라우저 자동화에서는 10회 학습 커브 후 토큰 비용
49%를 절감했으며, 14,000줄 Python-TypeScript 번역 자율 수행 비용이 $1.50였다.

기술 스택은 PydanticAI 기반으로 100개 이상의 LLM 제공자를 지원한다(LiteLLM
통합). 파이프라인 엔진은 `requires`/`provides` 계약, 불변 컨텍스트, 오류 격리로
설계됐다. 표준 학습 순서는 `AgentStep → EvaluateStep → ReflectStep → UpdateStep
→ DeduplicateStep`이며 `learning_tail()`로 조합한다. 러너 종류로는 기본 LiteLLM
러너, 배치 에포크 지원 Core 러너, 기존 트레이스 분석 TraceAnalyser, browser-use,
LangChain, Claude Code 통합이 있다.

설치는 `uv add ace-framework`로 시작하며 파인튜닝, 학습 데이터, 벡터 데이터베이스
없이 작동한다. MCP 서버(`ace-mcp`)로 IDE 통합도 지원한다.

ACE는 Kayba의 오픈코어 전략의 오픈소스 절반이다. 상용 플랫폼인
[Kayba](kayba.md)는 ACE와 다른 레이어에서 작동한다. ACE가 에이전트의 경험을
전략으로 정제해 다음 실행에 반영하는 사전(proactive) 학습이라면, Kayba 플랫폼은
실패가 발생한 뒤 Sentry·PostHog·OpenTelemetry와 연동해 즉각 수정 PR을 제안하는
사후(reactive) 디버깅이다. 두 제품을 함께 쓰면 같은 실수의 반복은 ACE가 막고,
새로운 실패의 수정은 Kayba가 처리하는 에이전트 신뢰성 스택이 된다.

## 분석

### Skillbook이 해결하는 문제: 세션 망각과 전략 축적의 비대칭

현재 LLM 기반 에이전트의 구조적 약점은 세션 망각이다. GPT-4o든 Claude든 매
대화 시작 시점에 빈 상태로 초기화된다. 반복 태스크를 수행하는 에이전트가 같은
실수를 계속 반복하는 이유다. 이것을 해결하는 전통적인 접근은 시스템 프롬프트에
규칙을 추가하거나, RAG로 문서를 제공하거나, 파인튜닝으로 가중치에 지식을 녹이는
것이었다. 이 모두는 인간이 직접 경험을 텍스트로 변환해야 한다는 병목이 있다.

ACE의 Skillbook은 이 병목을 에이전트 자신이 해결하게 한다. 실행 트레이스에서
Reflector가 전략을 추출하고, SkillManager가 Skillbook을 갱신한다. 인간이 개입
없이 에이전트가 스스로 경험을 전략화한다는 점이 핵심이다. Skillbook은 파인튜닝처럼
가중치를 바꾸지 않고 컨텍스트를 바꾼다. 어떤 모델이든, 언제든 내용을 수정할 수
있으며 투명하게 읽을 수 있다.

[에이전틱 소프트웨어의 5단계](agentic-software-levels.md) Level 3(메모리와
학습)이 “규칙을 따른다 → 규칙을 학습한다”로의 도약이라고 한 것을 ACE는
파인튜닝 없이 컨텍스트 레이어에서 구현한다.

### Recursive Reflector의 코드 실행이 의미하는 것

Reflector가 단순히 트레이스를 읽고 요약하는 것이 아니라 Python 코드를 작성해
샌드박스에서 실행한다는 설계는 흥미로운 함의를 갖는다. LLM이 텍스트를 읽어 패턴을
찾는 것보다, 코드로 패턴을 검색하고 통계를 계산하는 것이 더 신뢰성 있고 반복적인
분석이 가능하다는 전제다.

이것은 “LLM이 코드를 쓰고 실행해서 추론한다”는 패턴—Code Interpreter나 tool-use
기반 수학 문제 풀기에서 이미 검증된—을 반성(reflection) 영역으로 확장한 것이다.
트레이스가 복잡할수록 단순 텍스트 요약의 한계가 드러난다. 1,000개의 브라우저
자동화 액션 로그를 읽어 어디서 실패 패턴이 발생했는지 찾는 것은 LLM 컨텍스트
윈도우와 주의 집중 메커니즘으로는 어렵다. 코드로 검색하면 확실하다.

설계 결정 문서는 이 구현이 PydanticAI 이전에는 ~3,500줄이었던 것을 도구 정의
중심으로 재작성해 ~1,000줄로 줄였다고 밝힌다. 코드 추출에만 200줄 이상의 정규식이
필요했던 것이 PydanticAI의 구조화 출력으로 해결됐다.

### 파이프라인 엔진의 교체 불가능성이 진짜 IP다

README와 아키텍처 문서가 일관되게 강조하는 것은 “핵심 IP는 파이프라인 엔진과
학습 루프”라는 것이다. LLM 클라이언트 3개를 PydanticAI로 교체했을 때 유지한 것이
무엇인지 명확히 밝힌다: `requires`/`provides` 계약, `async_boundary`, 단계별
`max_workers`, `SampleResult` 오류 격리. “어떤 프레임워크도 이 조합을 제공하지
않는다”고 명시한다.

이 판단은 ACE가 LangChain, PydanticAI, LiteLLM 같은 LLM 플러밍 레이어를 소비
가능한 인프라로 취급한다는 의미다. LLM 접근 방법론이 빠르게 바뀌는 시장에서
안정적인 차별점은 오케스트레이션 레이어와 학습 루프 로직이라는 판단이다. 이것은
Superlog가 에이전트 러너를 플러그어블로 설계하고 인프라를 공개한 것과 같은 전략적
논리다.

### `ctx.trace`가 `object | None`인 설계 선택

설계 결정 문서에서 `Trace` 데이터클래스를 거부한 이유가 흥미롭다. browser-use의
`AgentHistoryList`, LangChain 결과 딕셔너리, Claude Code 트랜스크립트가 모두 다른
형태라서 공통 스키마를 강제하면 정보 손실이나 의미 없는 `metadata` 버킷이
불가피하다는 것이다. 대신 `ctx.trace`를 `object | None`으로 두고 Reflector가
무엇을 받든 이해하도록 설계했다.

이것은 “트레이스 형식 표준화” 문제가 ACE 이전에 해결된 것이 아니라 ACE가 다양성을
수용하는 방향으로 설계됐다는 것을 보여준다. Superlog가 OpenTelemetry 표준 위에서
작동하는 것과 대조적이다. ACE는 표준화를 전제로 하지 않고 이질적인 트레이스를
소화하는 반성 루프를 설계했다. 이 선택은 적용 범위는 넓어지지만 Reflector에
더 많은 지능을 요구한다.

## 비평

### 벤치마크의 조건이 실제 사용 환경과 얼마나 다른가

README의 벤치마크—Tau2 pass^4 2배 향상, 토큰 49% 절감, $1.50 번역 비용—는
인상적이지만 조건을 주의 깊게 읽어야 한다. Tau2 벤치마크는 “15개 전략, 보상 신호
없음, 훈련 분할에서 학습, 테스트 분할에서 평가”라는 조건이다. pass^4가 기준이
된 이유는 4번 독립 시도 모두 성공해야 한다는 고난도 기준이며, 이것은 시스템이
안정적일 때 개선이 두드러지는 메트릭이다.

실제 에이전트 배포에서 태스크 분포는 훈련 분할과 동일하지 않다. 프로덕션 환경에서
에이전트가 만나는 태스크는 훈련 중 보지 못한 새로운 변형이 많다. Skillbook에
축적된 전략이 분포 외(out-of-distribution) 태스크에 어떻게 일반화되는지에 대한
데이터가 없다. 오히려 Skillbook이 특정 분포에 과적합(overfit)되어 새로운 태스크에서
잘못된 전략을 적용하는 역효과가 발생할 수 있다는 위험도 논의되지 않는다.

$1.50 번역 비용도 “제로 감독, 빌드 에러 없음, 모든 테스트 통과”라는 조건이 어떤
코드베이스였는지에 따라 재현 가능성이 크게 달라진다. 단순한 유틸리티 라이브러리인지,
복잡한 도메인 로직을 포함하는지, 테스트 커버리지 수준이 어느 정도인지가 명시되지
않는다. ACE 자신의 Python 코드를 TypeScript로 번역했을 가능성이 높은데, 그 경우
번역 대상 코드의 구조적 특성을 학습 루프가 이미 알고 있는 최적 조건이 된다.

### Recursive Reflector의 코드 실행 샌드박스가 보안 위험이다

Reflector가 트레이스를 분석하기 위해 LLM이 생성한 Python 코드를 샌드박스에서
실행한다. README와 아키텍처 문서는 “샌드박스 환경”이라고 명시하지만, 이 샌드박스가
실제로 어떻게 격리되는지는 문서에서 상세히 다루지 않는다.

트레이스 데이터는 에이전트가 처리한 임의의 외부 입력—웹 페이지 내용, API 응답,
사용자 메시지—을 포함할 수 있다. 공격자가 트레이스 데이터에 코드 실행 유도 문자열을
심으면, Reflector가 그 데이터를 분석하는 과정에서 악성 코드를 생성할 위험이 있다.
이것은 간접 프롬프트 인젝션(indirect prompt injection)이 코드 실행 권한과 결합한
형태다. 브라우저 자동화나 외부 API 호출 트레이스를 처리할 때 이 위험이 가장 높다.

이 위험은 ACE만의 문제가 아니라 “LLM이 실행 가능 코드를 생성하고 그 코드가 외부
입력으로 오염될 수 있는” 모든 시스템의 구조적 취약점이다. 그러나 ACE가 이를
충분히 다루지 않으면서 실무 도입을 장려하는 것은 사용자가 인식하지 못하는 공격
표면을 만든다.

[Meta의 HyperAgents](meta-hyperagents.md)가 “수렴인가 모방인가”라는 질문을
받은 것처럼, ACE의 Reflector도 자신이 추출한 전략이 진짜 인과를 잡았는지
아니면 학습 분포에 끼워 맞춘 그럴듯한 패턴인지 구분하기 어렵다.

### Skillbook 오염과 잘못된 전략의 누적

ACE의 가정은 반성(reflection)이 항상 올바른 전략을 추출한다는 것이다. 그러나
Reflector도 LLM이다. LLM이 트레이스를 잘못 해석하거나 인과관계를 혼동할 경우,
잘못된 전략이 Skillbook에 추가된다. 이후 에이전트는 이 잘못된 전략을 활용해
태스크를 실행하고, 더 나쁜 결과를 낸다. SkillManager가 낡은 전략을 제거한다지만,
잘못된 전략이 쌓이는 속도가 제거 속도보다 빠를 수 있다.

이것은 머신러닝에서 레이블 노이즈(label noise) 문제와 유사하다. 학습 데이터에
오염된 레이블이 있으면 모델이 잘못된 패턴을 학습한다. ACE에서는 Reflector의 오류가
학습 데이터(전략)를 오염시킨다. `DeduplicateStep`이 중복을 제거하지만, 잘못된
전략이 올바른 전략과 다른 내용이면 중복 제거가 오염을 막지 않는다.

실제 배포 환경에서 에이전트가 비정상적인 입력이나 예외 상황을 처리할 때 생성된
잘못된 트레이스가 Skillbook을 오염시키는 시나리오는 충분히 가능하다. 이 경우
롤백 메커니즘, 전략별 신뢰도 추적, 인간 검토 인터페이스가 필요하다. `CheckpointStep`이
있지만, 전략 품질 평가와 인간 개입 흐름은 문서에서 상세하지 않다.

## 인사이트

### Skillbook은 모델 간 전이 가능한 새로운 단위의 지식 포맷이다

ACE의 Skillbook이 흥미로운 것은 특정 모델의 가중치에 종속되지 않는다는 점이다.
GPT-4o로 학습한 Skillbook을 Claude Sonnet에 적용할 수 있고, 모델을 바꿔도 전략이
유지된다. 이것은 지식이 가중치에 인코딩되는 파인튜닝과 근본적으로 다른 포맷이다.

이 이식성은 2차 효과를 낳는다. 조직이 LLM 제공자를 교체할 때 학습된 전략을 잃지
않아도 된다. 비용이나 성능 이유로 GPT-4o에서 Haiku로, 다시 Gemini로 전환하더라도
Skillbook은 중립적으로 유지된다. 이것은 LLM 제공자 락인을 완화하는 실용적 가치가
있다. 반면 Skillbook 자체가 새로운 락인 포맷이 된다. kayba.ai 호스팅 서비스로
Skillbook을 관리하기 시작하면, 데이터 포터빌리티와 벤더 종속의 긴장이 다시
나타난다.

역사적으로 지식 표현의 이식성 문제는 반복됐다. 관계형 데이터베이스의 SQL이
비교적 이식 가능한 표준이 됐지만, 각 벤더의 확장 기능에 의존하면 이식성이
깨졌다. Skillbook이 개방형 포맷으로 유지되는가, 아니면 kayba.ai 생태계에 특화된
포맷으로 발전하는가가 장기적 이식성을 결정할 것이다.

### 컨텍스트 엔지니어링은 파인튜닝 없는 전문화의 경로다

ACE가 속한 더 넓은 트렌드는 “컨텍스트 엔지니어링(context engineering)”이다. 모델
가중치를 바꾸는 것이 아니라 모델이 받는 컨텍스트를 정교하게 설계해 행동을 바꾸는
접근이다. Skillbook은 이 컨텍스트 엔지니어링의 자동화다. 인간이 규칙을 수기로
작성하는 것이 아니라 에이전트가 실행 경험에서 컨텍스트를 스스로 생성한다.

이 패턴은 중요한 함의를 갖는다. 파인튜닝은 대규모 학습 데이터와 컴퓨팅 비용이
필요하며 일반 개발자가 접근하기 어렵다. 컨텍스트 엔지니어링 기반 전문화는 추론
비용만 들고 즉시 수정 가능하며 투명하다. 도메인 특화 에이전트를 만드는 진입 장벽이
파인튜닝 대비 크게 낮아진다. ACE의 $1.50 번역 비용이 암시하는 것은 특정 도메인의
반복 태스크에 최적화된 에이전트를 몇 달러로 만들 수 있다는 것이다.

그러나 컨텍스트 엔지니어링의 한계도 있다. 컨텍스트 윈도우 크기가 Skillbook이
담을 수 있는 전략의 수를 제한한다. 전략이 수천 개로 늘면 전략 검색 및 우선순위화
문제가 생긴다. 임베딩 기반 중복 제거(`ace-framework[deduplication]`)가 이 방향의
시작이지만, Skillbook이 수만 개의 전략을 담을 때 어떻게 관리할지는 열린 문제다.

[멀티 에이전트 오케스트레이션 문제](multi-agent-orchestration-problems.md)가
지적한 21.3%의 검증 오류 — 에이전트의 자체 검증 — 가 Reflector 단계에서
구조적으로 재현될 위험이 있다.
Skillbook 오염은 자기 채점이 누적 자산을 만들 때 무엇이 어긋날 수 있는지를
보여주는 사례다.

### 자동화된 반성 루프는 에이전트 행동의 감사 가능성을 바꾼다

현재 LLM 에이전트의 행동을 이해하려면 시스템 프롬프트와 실행 트레이스를 읽으면
된다. Skillbook이 도입되면 세 번째 요소—어떤 전략이 어떤 경험으로부터 형성됐는가—가
추가된다. 에이전트가 특정 방식으로 행동하는 이유를 완전히 이해하려면 Skillbook의
형성 이력을 추적해야 한다.

이것은 AI 시스템의 감사 가능성(auditability)에 새로운 차원을 추가한다. 금융, 의료,
법률 도메인에서 AI 에이전트가 특정 결정을 내린 이유를 설명해야 할 때,
“모델 가중치 + 시스템 프롬프트 + 학습된 전략” 전체를 설명해야 한다. 학습된 전략이 어떤 태스크
실행 경험에서 형성됐는지, 그 경험이 대표성이 있는지, 편향이 없는지를 누가 어떻게
검증하는가는 규제가 강한 영역에서 중요해질 문제다.

2차 효과로, Skillbook 내용 자체가 지적 재산의 문제를 제기한다. 특정 조직의 도메인
전문 지식이 에이전트 경험을 통해 Skillbook에 체계화됐을 때, 그 Skillbook은 누구의
것인가. 직원들의 실무 지식이 에이전트를 통해 조직의 자산이 되는 과정에서 노동과
지식 소유권의 경계가 흐려진다. ACE의 오픈소스 프레임워크 위에서 kayba.ai 호스팅
서비스가 Skillbook을 관리하는 구조는 이 긴장을 더 첨예하게 만든다.

### 학습 루프의 자동화는 에이전트 진화의 감독 공백을 만든다

Reflector와 SkillManager가 인간 개입 없이 Skillbook을 갱신하는 구조는 에이전트가
독립적으로 진화하는 시스템이다. 이것은 효율성 측면에서 강점이지만, 에이전트 행동이
예상치 못한 방향으로 변화할 때 감지하고 개입할 메커니즘이 부족하다는 구조적 문제를
내포한다.

전통적인 소프트웨어 시스템에서 코드는 인간이 명시적으로 변경하며 변경 이력은 Git에
기록된다. ACE의 Skillbook은 에이전트가 자율적으로 변경하는 코드와 유사한 위치에
있다. `CheckpointStep`이 스냅샷을 만들지만, Skillbook 변경의 인과관계를 추적하고
특정 변경이 에이전트 행동에 어떤 영향을 미쳤는지 분석하는 도구는 아직 초기 단계다.

이 감독 공백은 에이전트를 장기 배포할수록 심화된다. 처음 100회 실행 후의
Skillbook과 10,000회 실행 후의 Skillbook은 다른 에이전트를 만들 수 있다. 이
진화 경로가 의도한 방향으로 가고 있는지 모니터링하는 것은 MLOps에서 모델 드리프트
모니터링과 유사한 문제다. 그러나 모델 드리프트는 가중치 변화 없이 데이터 분포
변화로 발생하는 것이고, Skillbook 드리프트는 에이전트 자신이 변화를 만든다는 점에서
더 복잡하다.
