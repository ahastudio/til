# 멀티 에이전트 오케스트레이션은 왜 잘 안 되는가

> 원문: <https://shalomeir.substack.com/p/multi-agent-orchestration-problems>

## 요약

저자는 Paperclip과 Gastown 같은 멀티 에이전트 시스템에 5,000달러를 직접 투자하고
운영한 결과를 바탕으로, 현재 완전 자동화된 에이전트 조직은 실현 불가능하다고
결론짓는다. 토큰 소비는 단일 에이전트 대비 5~10배 증가하지만 생산성은 비례하지
않는다.

UC Berkeley의 MAST 연구를 기반으로 실패의 세 가지 구조적 병목을 분류한다.
맥락 붕괴(41.8%): 에이전트가 전체 목표를 잃어버리고 “왜 이 일을 하는가”를
이해하지 못함. 유령 위임(36.9%): 작업 인수인계 지점에서 연결이 끊어져 미완료
상태로 방치. 검증 오류(21.3%): 에이전트가 부정확한 결과를 자체 검증으로 완료
처리.

핵심 인사이트는 단일 에이전트가 상위 에이전트 역할을 하며 전체 맥락을 유지하는
구조가 다단계 계층보다 훨씬 강건하다는 것이다. 에이전트 간 직접 통신보다
이슈 트래커나 Git 같은 공유 환경(블랙보드 패턴)이 더 효과적이다.

저자는 업무 위임 판별 기준으로 5가지 변수를 제시한다: 오류 비용, 검증 용이성,
암묵지 의존도, 컨텍스트 범위, 피드백 루프 길이. 낮은 점수 영역부터 에이전트 주도로
전환하고 높은 점수 영역은 인간 중심으로 유지해야 한다.

## 분석

### 세 가지 병목의 구조적 해석

41.8%를 차지하는 맥락 붕괴는 LLM의 근본적 한계에서 비롯된다. 각 에이전트는
자신의 컨텍스트 창 안에서만 작동하고, 전체 시스템의 목적은 초기 프롬프트에서만
전달된다. 작업이 여러 에이전트를 거치면서 원래 목적이 희석된다. 인간 조직에서도
같은 문제가 발생한다. 여러 단계의 관리자를 거치면서 CEO의 원래 의도가 왜곡되는
“전화 게임” 현상이다.

유령 위임(36.9%)은 비동기 통신의 취약점이다. 에이전트 A가 에이전트 B에게 작업을
넘겼지만 B가 실패했을 때, A는 이미 다른 작업으로 넘어갔고 아무도 B의 실패를
감지하지 못한다. 이것은 마이크로서비스 아키텍처에서 서비스 간 호출 실패 처리와
동일한 문제다. 분산 시스템에서 가장 어려운 문제 중 하나인 “부분 실패” 처리다.

### 블랙보드 패턴의 재발견

저자가 제안하는 “공유 환경(이슈 트래커, Git)” 방식은 1970-80년대 AI 연구에서
개발된 블랙보드 아키텍처(Blackboard Architecture)의 재발견이다. 여러 전문가
시스템이 공유 데이터 구조에 결과를 쓰고 읽으면서 협력하는 방식이다. 직접 메시지
패싱보다 공유 상태 기반 협력이 더 견고하다는 것은 40년 전에 이미 발견된 원리다.

## 비평

### 강점

5,000달러의 실제 비용을 투자한 1인칭 경험이 글의 신뢰성을 높인다. 학술 연구
(MAST)와 실무 경험을 결합한 분석이 균형 잡혀 있다. 업무 위임 판별 기준 5가지는
즉시 실무에 적용 가능한 구체적 프레임워크다.

특히 “에이전트 간 직접 통신보다 공유 환경이 효과적”이라는 결론은 소프트웨어
아키텍처의 오랜 원칙(느슨한 결합)과 일치하며, 현재 AI 에이전트 설계에 직접
적용할 수 있다.

### 약점

5,000달러 비용과 어떤 구체적 작업에 사용했는지가 불명확하다. “복잡한 소프트웨어
개발”이라는 것만으로는 독자가 자신의 상황에 결과를 적용하기 어렵다.

또한 실패 분류(41.8%, 36.9%, 21.3%)의 측정 방법이 제시되지 않는다. MAST 연구
기반이라고 하지만, 저자 자신의 실험에서 이 수치를 어떻게 측정했는지가 불분명하다.

## 인사이트

### 분산 시스템의 교훈과 AI 에이전트

멀티 에이전트 오케스트레이션의 문제들은 1990년대-2000년대 분산 컴퓨팅이 겪은
문제들과 구조적으로 동일하다. 네트워크 분할(맥락 붕괴), 타임아웃 처리(유령 위임),
분산 트랜잭션(검증 오류). 이 문제들을 해결하기 위해 CAP 정리, 사가(Saga) 패턴,
이벤트 소싱 등의 기법이 개발되었다.

AI 에이전트 오케스트레이션은 같은 길을 걷고 있다. 지금은 “에이전트가 협력하면
좋겠다”는 아이디어 단계지만, 실제 신뢰할 수 있는 시스템을 만들려면 분산 시스템이
30년에 걸쳐 발전시킨 패턴들을 빠르게 재발견하고 적용해야 한다. 차이점은 에이전트는
결정론적이지 않다는 것이다. 서비스는 같은 입력에 같은 출력을 내지만, 에이전트는
그렇지 않다. 이것이 기존 분산 시스템 패턴을 그대로 적용하기 어렵게 만드는 핵심
차이다.

### 자동화 레벨과 신뢰 구축의 순서

저자가 제안하는 “낮은 점수 영역부터 에이전트 주도로 전환”은 자율주행의 자동화
레벨(L0-L5) 접근법과 정확히 같다. 먼저 단순하고 검증 가능한 작업에서 신뢰를
쌓고, 점진적으로 복잡하고 고위험 작업으로 확장한다.

자율주행이 L2(고속도로 자동 조향)에서 수년을 머물다 L3를 건너뛰고 L4로 가려는
움직임이 있는 것처럼, 에이전트도 L2(보조 자동화)에서 L4(특정 도메인 완전 자동화)로
가는 경로가 도메인마다 다를 것이다. 코드 포매팅, 단위 테스트 생성 같은 작업은
이미 L4 수준에 가깝다. 아키텍처 설계, 요구사항 분석은 아직 L1도 안 된다.

### 오케스트레이션 비용의 숨겨진 측면

“토큰 소비가 5-10배”라는 수치는 단순히 비용 문제가 아니다. 토큰 소비 증가는
지연 시간(latency) 증가를 의미한다. 에이전트가 다른 에이전트의 결과를 기다리는
동안 전체 작업이 블로킹된다. 병렬 처리로 일부 해결할 수 있지만, 맥락 의존적인
순차 작업은 병렬화할 수 없다.

또한 오류 디버깅 비용이 단일 에이전트 대비 기하급수적으로 높다. 단일 에이전트의
오류는 하나의 대화 이력에서 추적 가능하지만, 멀티 에이전트의 오류는 여러 에이전트의
이력을 교차 분석해야 한다. 로그 분석, 재현, 수정 모두 훨씬 복잡하다. 실질적인
엔지니어링 생산성 계산에는 이 숨겨진 비용이 포함되어야 한다.
