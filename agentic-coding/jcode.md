# jcode: 차세대 코딩 에이전트 하네스

<https://github.com/1jehuang/jcode>

## 소개

jcode는 Rust로 작성된 고성능 코딩 에이전트 하네스(Coding Agent Harness)다.
Claude Code, OpenCode, Codex CLI 등 기존 에이전트 도구들이 Node.js 또는
Python 런타임 위에서 동작하는 반면, jcode는 Rust의 특성을 살려 메모리 사용량과
부팅 시간을 극단적으로 최소화했다.
2026년 1월 공개 이후 약 5개월 만에 GitHub 스타 6,000개를 돌파했으며,
Claude, OpenAI, Gemini, GitHub Copilot 등 주요 LLM 프로바이더를 모두 지원한다.

## 주요 기능

### 성능

동일한 조건에서 측정한 메모리 사용량(PSS)과 첫 프레임까지의 시간에서
jcode는 경쟁 도구 대비 압도적인 우위를 보인다.

1세션 기준 메모리 비교:

| 도구                | PSS        | 비교          |
| ------------------- | ---------: | ------------: |
| jcode (임베딩 꺼짐) | 27.8 MB    | 기준          |
| jcode               | 167.1 MB   | 6.0×          |
| Claude Code         | 386.6 MB   | 13.9×         |
| OpenCode            | 371.5 MB   | 13.4×         |

첫 프레임까지 시간(10회 평균):

| 도구               | 시간       | 비교          |
| ------------------ | ---------: | ------------: |
| jcode              | 14.0 ms    | 기준          |
| Claude Code        | 3,436.9 ms | 245.5×        |
| Cursor Agent       | 1,949.7 ms | 139.3×        |

10세션을 동시에 운용할 때 차이는 더 두드러진다.
세션 추가 시 추가되는 메모리가 jcode는 약 10 MB인 반면
Claude Code는 약 213 MB, OpenCode는 약 318 MB다.
멀티 세션 워크플로우를 확장할수록 jcode의 효율성은 기하급수적으로 커진다.

### 시맨틱 메모리 시스템

jcode는 매 턴의 응답을 시맨틱 벡터로 임베딩해 그래프 구조의 메모리에 저장한다.
대화가 진행될 때마다 코사인 유사도 검색으로 관련 기억을 자동으로 조회하며,
에이전트가 명시적으로 메모리 툴을 호출하지 않아도 관련 맥락이 자동 주입된다.
메모리 추출·통합은 백그라운드 사이드 에이전트가 주기적으로 수행하며,
오래되거나 충돌하는 기억을 자동으로 정리한다.

### 스웜(Swarm) 모드

같은 저장소에서 두 개 이상의 에이전트를 동시에 실행하면 서버가 자동으로
충돌을 관리한다.
에이전트 A가 수정한 파일을 에이전트 B가 이미 읽은 상태라면
서버가 B에게 변경 알림을 보내고, B는 diff를 확인해 충돌 여부를 판단한다.
에이전트는 DM, 브로드캐스트, 저장소 범위 메시지 등 다양한 채널로 소통할 수 있다.
또한 에이전트가 스스로 하위 에이전트를 스폰해 병렬 작업을 지시하는
코디네이터-워커 패턴도 지원한다.

### 자기 개발(Self-Dev) 모드

jcode는 에이전트에게 자신의 소스 코드를 수정하게 한 다음,
빌드·테스트·바이너리 재로드까지 자동화하는 “자기 개발 모드”를 제공한다.
실행 중인 세션을 종료하지 않고 새 바이너리로 전환할 수 있으며,
다수 세션이 동시 운용 중인 상황에서도 동작한다.

### TUI와 렌더링

사이드 패널에서 파일을 실시간으로 보거나 diff 뷰어로 활용할 수 있다.
Mermaid 다이어그램을 인라인으로 렌더링하는데,
이를 위해 별도의 mermaid-rs-renderer 라이브러리를 제작했다.
브라우저와 TypeScript 의존성 없이 동작하며 기존 대비 1,800배 빠른 렌더링 속도를 달성했다.
정보 위젯(info widgets)은 화면 여백만을 사용해 중요 정보를 표시하고,
공간이 부족하면 자동으로 숨는다.

### 세션 재개 및 크로스 하네스 호환

Claude Code, Codex, OpenCode, pi 등 다른 하네스에서 시작한 세션을
jcode에서 이어받아 계속할 수 있다.
에이전트 오류로 세션이 끊어졌을 때 유용하다.

## 분석

### Rust 선택의 구조적 의미

Node.js나 Go 기반 도구들이 런타임 오버헤드를 안고 가는 반면,
jcode는 Rust의 제로 코스트 추상화(Zero-cost abstraction)를 통해
GC 정지 없이 예측 가능한 지연 시간을 달성한다.
멀티 세션 워크플로우에서 세션당 메모리 증분이 약 10 MB에 불과하다는 점은
단순히 “Rust라서 빠르다”는 마케팅이 아니라
시스템 설계 수준에서 공유 상태(shared state)가 정교하게 관리되고 있음을 시사한다.

### 메모리와 에이전트 협업의 결합

기존 에이전트 도구들은 메모리 관리를 외부 RAG 파이프라인이나
컨텍스트 윈도우 한계에 맡겼다.
jcode는 임베딩·그래프 저장·사이드 에이전트 검증을 하네스 레벨에서 통합함으로써
에이전트가 “기억을 잘 하는 것”과 “응답이 빠른 것”을 동시에 달성하려 한다.
스웜 모드와 메모리 시스템이 결합되면,
여러 에이전트가 같은 기억 그래프를 공유하며 협업하는 구조가 가능해진다.

### 프로바이더 생태계 전략

jcode는 특정 LLM 벤더에 종속되지 않도록
Claude, OpenAI, Gemini, Copilot, Alibaba, Ollama, vLLM 등
30개 이상의 프로바이더를 지원한다.
OAuth 기반 구독 플랜(Claude Max, ChatGPT Pro 등)을 직접 활용할 수 있어
별도 API 키 발급 없이 기존 구독을 재활용하는 경로를 제공한다.
이는 개인 개발자가 진입 장벽을 낮추는 동시에
기업 환경에서 벤더 락인(vendor lock-in)을 피하는 전략이기도 하다.

### 자기 개발 모드의 독창성

에이전트가 자신을 고치는 아이디어 자체는 새롭지 않지만,
“실행 중인 바이너리를 교체하고 세션을 끊지 않는다”는 구현은
Rust의 핫 리로딩 부재를 창의적으로 우회한 것이다.
자기 수정 루프가 프로덕션에서 안정적으로 작동하려면
엄격한 테스트 자동화와 롤백 메커니즘이 필요하다는 점에서,
이 기능이 얼마나 신뢰할 수 있는지는 더 지켜봐야 한다.

## 비평

### 강점

jcode의 가장 두드러진 강점은 성능 수치의 투명성이다.
단순 벤치마크가 아니라 PSS 기준 메모리, 첫 프레임까지 시간, 세션 추가당 증분 등
구체적인 측정 지표와 버전 정보를 함께 공개해 신뢰도를 높였다.
또한 Rust 단일 바이너리로 배포되므로 설치와 제거가 단순하고,
멀티 세션 환경에서 리소스 예측이 가능하다는 점은 실용적 가치가 크다.

### 약점과 한계

README의 문법 오류(“ot”, “reccomended”, “effiecent” 등)와
미완성 섹션(iOS 앱, 자기 개발 데모 누락 등)이 눈에 띈다.
공개 시점이 2026년 1월이고 별이 6,000개를 넘었지만,
단일 개발자 프로젝트 특성상 장기 유지보수 지속성에 대한 불확실성이 있다.
자기 개발 모드는 “프론티어 모델을 사용하지 않으면 코드베이스를 망가뜨릴 수 있다”고
스스로 경고할 만큼 위험 부담이 크다.

스웜 아키텍처는 흥미롭지만,
여러 에이전트가 동일 파일을 동시 수정할 때 발생하는 엣지 케이스가
얼마나 견고하게 처리되는지 실증 사례가 부족하다.
멀티 에이전트 Git 워크플로우가 해결되지 않은 문제임을 README 스스로 인정하고 있다.

### 경쟁 구도

Claude Code, Codex CLI, OpenCode, pi 등이 직접 경쟁 상대다.
이들 모두 비슷한 방향으로 진화하고 있어,
Anthropic이나 OpenAI가 성능 최적화에 집중한 유사 도구를 출시할 경우
jcode의 차별화 포인트가 희석될 위험이 있다.

## 인사이트

### 하네스 경쟁은 에이전트 경쟁보다 더 오래 지속된다

LLM 자체는 빠르게 상향 평준화되고 있다.
GPT-4, Claude 3, Gemini 1.5 등 어느 모델을 사용해도
기본 코딩 작업에서 큰 차이가 느껴지지 않는 시대가 이미 도래했다.
그러나 에이전트가 어떤 환경에서 실행되느냐 즉, 하네스의 설계는
사용자 경험의 품질을 결정하는 층위로 점점 중요해지고 있다.

jcode의 성공은 이 흐름을 정확히 짚는다.
동일한 Claude 모델을 Claude Code로 쓰느냐 jcode로 쓰느냐에 따라
부팅 시간이 245배 차이 난다면, 사용자는 모델 품질보다 하네스 품질을 먼저 느낀다.
IDE가 언어보다 오래 사용되고, 터미널 에뮬레이터가 셸보다 오래 사용되듯이
에이전트 하네스는 LLM 세대 교체를 아래에서 받쳐주는 인프라 층이 된다.

멀티 에이전트 워크플로우가 일상화될수록 이 차이는 선형이 아니라 지수적으로 벌어진다.
세션 10개를 동시에 돌릴 때 메모리 총량이 261 MB(jcode) 대 2,301 MB(Claude Code)라면,
같은 서버에서 운용 가능한 에이전트 수는 8배 이상 차이 난다.
하네스가 플랫폼이 되는 순간, 그 위에서 실행되는 에이전트의 수익 구조와
사용자 획득 비용 자체가 달라진다.

### 에이전트의 자기 수정은 새로운 소프트웨어 개발 패러다임의 전초전이다

jcode의 자기 개발 모드는 단순한 기능이 아니라
소프트웨어가 자신의 유지보수자가 되는 세계의 프로토타입이다.
지금은 “에이전트가 Rust 코드를 수정하고 빌드하고 바이너리를 교체한다”는 수준이지만,
이 루프가 안정화되면 의미는 달라진다.

전통적인 소프트웨어 개발에서 유지보수 비용은 초기 개발 비용의 수 배다.
버그 수정, 의존성 업그레이드, 아키텍처 리팩터링 등이 모두 인간 개입을 요구한다.
에이전트가 이 루프를 스스로 닫을 수 있다면,
소프트웨어의 총 소유 비용(TCO) 계산식이 근본적으로 바뀐다.

물론 현재의 자기 개발 모드는 “프론티어 모델이 아니면 코드를 망가뜨린다”는 경고가 붙는다.
이는 아직 에이전트가 자신의 코드베이스를 충분히 이해하지 못함을 의미한다.
하지만 모델 역량이 올라갈수록 이 임계점은 낮아진다.
자기 수정 루프가 신뢰할 수 있는 수준에 도달하는 시점이
소프트웨어 산업의 고용 구조가 다시 한번 뒤집히는 시점과 일치할 가능성이 높다.

### 멀티 에이전트 협업에서 Git은 이미 병목이다

jcode의 README는 조심스럽게 고백한다.
“에이전트들은 변경 사항이 있는 dirty git 상태에서 커밋하기를 싫어한다.
Git은 멀티 에이전트 워크플로우를 위해 만들어지지 않았으며,
git worktree도 좋은 해결책이 아니다.”

이 문장은 현재 에이전트 협업의 가장 큰 마찰 지점을 정직하게 지목한다.
Git은 인간 개발자가 순차적으로, 또는 적은 수의 브랜치로 협업한다는
암묵적 전제 위에서 설계됐다.
에이전트 10개가 동일 저장소에서 동시에 작업하면
머지 충돌, 락, 커밋 순서 문제가 기하급수적으로 증가한다.

jcode의 스웜 서버가 파일 변경 알림을 에이전트 간에 전파하는 접근법은
이 문제의 응급 처방이다.
근본적인 해결책은 Git과는 다른 버전 관리 원시(primitive)가 필요하다.
jcode가 직접 언급하듯 “새로운 Git 같은 원시”의 탄생을 이 도메인이 요구하고 있다.
멀티 에이전트 시대에 맞는 버전 관리 패러다임은
현재 가장 중요하고 가장 덜 해결된 인프라 문제 중 하나다.
