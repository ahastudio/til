# CodeBoarding

<https://www.codeboarding.org/>

<https://github.com/CodeBoarding/CodeBoarding>

## 소개

CodeBoarding은 코드베이스를 위한 인터랙티브 아키텍처 다이어그램 생성 도구다.
정적 분석(Static Analysis)과 LLM 추론을 결합해 아키텍처 다이어그램, 컴포넌트
수준 문서, IDE와 CI에서 바로 탐색 가능한 출력물을 자동으로 생성한다.
Python, TypeScript, JavaScript, Java, Go, PHP, Rust, C# 등 주요 언어를 지원하며,
OpenAI, Anthropic, Google, AWS Bedrock, Ollama 등 다양한 LLM 프로바이더와
연동된다.

GitHub 스타 약 1,800개를 획득한 오픈소스 프로젝트로, VS Code 익스텐션,
GitHub Action, CLI 형태로 제공된다. 생성된 결과물은 `.codeboarding/` 디렉터리에
Mermaid 다이어그램과 Markdown 문서로 저장되며, PR과 문서에 바로 삽입할 수 있다.

## 작동 방식

CodeBoarding의 내부 구조는 여섯 개 컴포넌트로 구성된다.

- **Application Orchestrator & Repository Manager**: 전체 분석 워크플로를
  오케스트레이션하고 증분 업데이트를 조율한다.
- **LLM Agent Core**: LLM을 활용해 코드 구조를 추론하고 인사이트를 도출한다.
- **Static Code Analyzer**: LSP(Language Server Protocol) 클라이언트를 통해
  코드베이스를 정적으로 분석한다.
- **Agent Tooling Interface**: LLM 에이전트가 코드베이스 및 분석 데이터와
  상호작용할 수 있는 도구 인터페이스를 제공한다.
- **Incremental Analysis Engine**: 변경된 부분만 재분석해 성능을 최적화한다.
- **Documentation & Diagram Generator**: 분석 결과를 Mermaid 다이어그램과
  Markdown 문서로 변환한다.

정적 분석기가 코드 구조를 파악하고, LLM 에이전트가 그 구조에 의미를 부여해
인간이 읽기 좋은 아키텍처 문서를 생성하는 방식이다.

## CLI

```bash
# 소스에서 실행
uv sync --frozen
source .venv/bin/activate
python install.py
python main.py full --local /path/to/repo

# pipx로 설치
pipx install codeboarding --python python3.12
codeboarding-setup
codeboarding full --local /path/to/repo

# 주요 옵션
codeboarding full --local ./my-project --depth-level 2   # 다이어그램 깊이 조정
codeboarding incremental --local ./my-project             # 변경 부분만 재분석
codeboarding partial --local ./my-project --component-id "1.2"  # 특정 컴포넌트만
codeboarding full https://github.com/pytorch/pytorch     # 원격 저장소 분석
```

## 분석

### 정적 분석 + LLM의 결합

CodeBoarding의 핵심 아이디어는 정적 분석과 LLM 추론의 역할 분리다. LSP를 통한
정적 분석이 심볼 참조, 호출 그래프, 타입 정보 등 사실(fact) 계층을 담당하고,
LLM이 그 사실들 사이의 의미와 패턴을 해석한다. 이는 LLM 단독으로 코드를 읽히는
것보다 훨씬 신뢰도 높은 결과를 낸다. LLM이 잘 알려진 라이브러리의 패턴에
과적합되거나 코드 읽기를 건너뛰는 환각을 방지할 수 있다.

### 증분 분석 엔진

전체 코드베이스를 매번 재분석하는 것은 비용이 크다. 증분 분석 엔진은 변경된
파일과 그에 의존하는 컴포넌트만 선택적으로 재분석해 이 문제를 해결한다. CI
파이프라인에서 PR마다 다이어그램을 갱신하는 시나리오를 실용적으로 만드는 핵심
기능이다. GitHub Action과 결합하면 코드 변경과 아키텍처 문서가 항상 동기화된
상태를 유지할 수 있다.

### 에이전트 코딩 시대의 도구

CodeBoarding이 내세우는 슬로건은 “AI가 만들기 전에 무엇을 만드는지 보라(See
what your AI is building before it breaks)”다. 에이전트가 코드를 생성하는
속도가 인간이 검토하는 속도를 앞지르는 현실을 직접 겨냥한 포지셔닝이다. 단순한
코드 문서화 도구가 아니라, 에이전트 코딩 워크플로에서 가시성을 제공하는
레이어로 자신을 위치시키고 있다.

## 비평

### 강점

CodeBoarding의 가장 큰 강점은 800개 이상의 오픈소스 저장소에 적용된 실증적
검증이다. PyTorch, Django 같은 대형 코드베이스에 적용한 예시를 공개함으로써
개념 검증을 넘어 실용성을 입증했다. LSP 클라이언트 기반의 정적 분석은 언어별
파서를 직접 구현하는 것보다 훨씬 넓은 언어 지원을 적은 비용으로 달성하는 영리한
선택이다.

### 약점 및 한계

생성된 다이어그램의 정확성은 LLM의 추론 품질에 크게 의존한다. 정적 분석이
사실 계층을 제공하더라도, LLM이 컴포넌트 간 관계에 잘못된 의미를 부여하거나
중요한 의존성을 누락할 수 있다. 특히 동적 디스패치, 메타프로그래밍, 런타임
의존성 주입처럼 정적 분석으로 포착하기 어려운 패턴에서 취약할 수 있다.

또한 다이어그램의 “신선도” 유지가 관건이다. 개발팀이 CodeBoarding을 CI에
통합하지 않으면 다이어그램은 빠르게 오래된 정보가 된다. 에이전트 코딩 환경에서
코드 변경 속도는 인간 주도 개발보다 훨씬 빠르므로, 증분 분석이 그 속도를
따라잡을 수 있는지가 실제 사용성을 결정할 것이다.

## 인사이트

### 코드 가시성은 다음 엔지니어링 병목이다

소프트웨어 개발의 병목은 시대마다 바뀌었다. 한때는 컴파일 속도였고, 그 다음은
테스트 실행 시간이었으며, 최근에는 배포 파이프라인이었다. 에이전트 코딩이
주류가 되면서 새로운 병목이 등장하고 있다. 코드 생성 속도가 인간의 검토 속도를
압도하기 시작했기 때문이다.

CodeBoarding이 해결하려는 문제의 본질은 바로 이 “가시성 격차(visibility gap)”다.
에이전트가 매 시간 수천 줄의 코드를 쏟아낼 때, 그 변화가 전체 아키텍처에
미치는 영향을 인간이 즉각적으로 파악할 방법이 없다면 기술 부채는 누적 속도가
달라진다. 이전에는 몇 달에 걸쳐 서서히 쌓이던 것이 이제는 며칠 만에 구조적
부패로 이어질 수 있다.

이 도구가 성공한다면, 그것은 단순히 다이어그램이 편리해서가 아니라 가시성이
없으면 에이전트 코딩 자체가 지속 불가능해지는 시대가 오기 때문일 것이다.
CodeBoarding의 포지셔닝은 그 시대를 조금 일찍 내다본 것처럼 보인다.

### LSP를 분석 기반으로 쓰는 것의 의미

Language Server Protocol(LSP)은 원래 IDE와 언어 서버 사이의 통신 표준으로
설계됐다. CodeBoarding은 이 프로토콜을 분석 파이프라인의 기반으로 재활용한다.
이 선택은 표면적으로는 구현 편의성처럼 보이지만, 더 깊은 의미가 있다.

LSP 서버는 이미 수년간 각 언어 커뮤니티가 갈고닦은 파서와 심볼 분석기를 포함한다.
Python의 `pyright`, TypeScript의 `tsserver`, Go의 `gopls` 같은 LSP 서버들은
각 언어의 타입 시스템과 심볼 해석을 가장 정확하게 구현한 소프트웨어다. 이것을
재사용한다는 것은 언어별 파서를 처음부터 구현하는 수년의 작업을 우회하는 동시에,
각 언어 생태계의 투자를 그대로 흡수한다는 뜻이다.

역설적으로, 이 설계는 CodeBoarding이 새로운 언어를 지원하기 쉽게 만든다. 언어용
LSP 서버가 이미 존재한다면, CodeBoarding은 그 서버를 클라이언트로 연결하면
된다. 이것은 범용 분석 플랫폼으로 확장하는 데 있어 상당히 강력한 구조적 이점이다.

### 에이전트를 위한 코드 이해는 인간을 위한 것과 다르다

CodeBoarding의 비전 선언은 흥미롭다. “코드 이해를 위한 오픈 표준: 인간과 에이전트
모두가 사용할 수 있는 정확하고 고수준의 시각적 표현.” 인간과 에이전트 모두를
소비자로 명시한다.

이 지점에서 근본적인 질문이 생긴다. 에이전트를 위한 코드 이해는 인간을 위한
것과 같은 형태여야 할까? 인간에게 좋은 다이어그램은 레이아웃과 색상과 계층적
구조를 통해 직관적으로 의미를 전달한다. 그러나 LLM 에이전트에게는 Mermaid
다이어그램의 시각적 특성이 무의미하다. 에이전트는 텍스트 표현만 읽는다.

CodeBoarding이 진짜로 에이전트 컨텍스트 제공자로 자리잡으려면, 결국 에이전트가
소비하기 좋은 구조화된 데이터 형식(그래프 표현, JSON, 또는 특수 목적 포맷)을
별도로 제공해야 할 것이다. 현재 Mermaid와 Markdown으로 된 출력은 인간을 위한
것이다. 에이전트를 위한 출력은 아직 미완성 과제로 남아 있다. 이 간극을 어떻게
메우는지가 CodeBoarding의 장기적 방향을 결정할 핵심 설계 문제다.

### 아키텍처 문서의 자동화는 조직 정치 문제다

기술적으로 CodeBoarding이 완벽한 다이어그램을 생성한다 해도, 그것이 조직 안에서
실제로 사용되려면 별개의 조건이 필요하다. 아키텍처 다이어그램은 오랫동안 “있으면
좋지만 관리하기 번거로운 것”의 대표였다. 팀이 바쁠수록 문서는 오래되고, 오래된
문서는 신뢰를 잃고, 신뢰를 잃은 문서는 아무도 보지 않는다.

CodeBoarding이 이 악순환을 끊을 수 있는 이유는 자동화 때문이다. CI에 통합된
자동 업데이트는 문서가 최신 상태로 유지되는 유일하게 지속 가능한 방법이다.
그러나 이 자동화가 실제로 채택되려면 팀 리더가 CI 파이프라인 변경을 승인하고,
PR 리뷰 프로세스에 다이어그램 검토를 포함시키는 규범이 생겨야 한다. 이것은
기술 문제가 아니라 조직 변화 관리 문제다.

도구의 성공 여부는 결국 얼마나 낮은 마찰로 도입할 수 있느냐에 달려 있다.
VS Code 익스텐션, GitHub Action, CLI라는 세 가지 진입점을 제공하는 것은
정확히 이 마찰을 줄이려는 전략이다. 개인 개발자가 익스텐션으로 가치를 경험하고,
그 가치를 팀에 전파해 CI 통합으로 이어지는 바이럴 채택 경로를 설계한 것이다.
