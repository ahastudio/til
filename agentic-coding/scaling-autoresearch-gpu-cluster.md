# Karpathy의 AutoResearch를 GPU 클러스터로 확장하면 생기는 일

> 원문: <https://blog.skypilot.co/scaling-autoresearch/>

## 요약

Karpathy의 AutoResearch 에이전트를 단일 GPU에서 16개 GPU 클러스터로 확장했을 때
생기는 병목과 운영 포인트를 SkyPilot 팀이 실제 실험으로 정리했다.

단일 GPU에서는 학습 5분 동안 에이전트가 유휴 상태였고 시간당 약 10개 실험만 가능했다.
16개 GPU로 확장하자 한 번에 10~13개 동시 실험이 가능해져 시간당 약 90개 실험(9배)을
처리했다. 총 비용은 GPU 약 $300, Claude API 약 $9로 8시간에 900개 실험을 수행했다.

주목할 점은 에이전트가 직접 지시받지 않았는데도 H100과 H200의 성능 차이를 발견하고
"저가 H100에서 아이디어 스크리닝 → 상위 후보를 H200으로 승격"하는 2단계 검증 전략을
자율적으로 개발했다는 것이다.

## 분석

핵심 전환점은 순차 실행에서 병렬 실행으로의 이동이다. 단일 GPU 설정에서는 에이전트가
그리디 산악 등반(greedy hill climbing) 방식으로 한 번에 한 방향만 탐색했다. 병렬화하자
요인 격자(factorial grid) 탐색이 가능해져 여러 변수의 상호작용 효과를 동시에 포착할
수 있게 됐다.

5단계 수렴 과정에서 개선폭이 명확히 줄었다(0.022 → 0.004 → ... → 0.001). 에이전트가
탐색 공간을 소진했다는 신호가 정량적으로 나타나 "언제 멈출지"를 판단할 수 있었다.

운영상 발견: 초기 하이퍼파라미터 조정(약 0.022 개선)이 아키텍처 스케일링(약 0.004
개선)보다 훨씬 효과적이었다. 에이전트가 쉬운 개선부터 빠르게 수거하고 점차 어려운
영역으로 이동하는 패턴을 보였다.

## 비평

$300이라는 GPU 비용은 연구 프로토타입 수준이다. 8시간 실험이 아니라 수일간의 실제
연구 워크로드로 확장하면 비용 구조가 달라진다. 또한 AutoResearch가 탐색하는 문제
도메인(NanoGPT 류의 작은 모델 학습)은 클러스터 확장의 효과가 잘 드러나지만, 긴
컨텍스트 의존성이 있는 소프트웨어 엔지니어링 작업에서는 단순 병렬화가 통하지 않는다.

에이전트가 "자율적으로" 2단계 전략을 개발했다는 서술은 인상적이지만, 이것이 일반화
가능한 에이전트 지능인지 아니면 이 특정 도메인에서의 우연한 패턴인지는 구분이 필요하다.

## 인사이트

### 에이전트 성능의 병목은 모델이 아니라 실행 인프라다

단일 GPU에서 에이전트가 유휴 상태로 기다린 5분은 모델 능력과 무관한 낭비다. 에이전트
워크로드를 설계할 때 "모델이 얼마나 똑똑한가"보다 "모델이 기다리는 시간을 얼마나
줄일 수 있는가"가 실제 처리량을 결정한다. AI 인프라의 다음 경쟁 축은 모델 품질이
아닌 실행 오케스트레이션 효율이다.

### 병렬화는 탐색 전략 자체를 바꾼다

순차 실행에서는 이전 결과를 보고 다음 방향을 정하는 그리디 방식이 자연스럽다. 병렬화하면
이전 결과 없이 여러 방향을 동시에 탐색해야 하므로, 에이전트의 계획 능력이 더 중요해진다.
"다음에 무엇을 시도할지"가 아니라 "지금 동시에 무엇들을 시도해야 하는지"로 문제가
바뀐다. 이는 에이전트 설계에서 병렬 실행 인식(parallelism-aware planning)이 새로운
능력 요건이 됨을 의미한다.

### 하드웨어 이종성을 활용하는 2단계 전략은 비용 최적화의 일반 패턴이다

H100에서 스크리닝 → H200으로 승격하는 전략은 "값싼 자원으로 필터링, 비싼 자원으로
검증"이라는 일반 패턴이다. 이는 소프트웨어 테스트에서 빠른 단위 테스트가 통과한 것만
느린 통합 테스트로 보내는 패턴과 구조적으로 같다. 에이전트가 이 패턴을 자율적으로
발견했다는 것은, 비용-성능 트레이드오프가 명시적으로 제약 조건으로 주어지지 않아도
추론 가능한 수준이 됐다는 신호다.
