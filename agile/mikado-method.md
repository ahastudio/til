# 미카도 메서드 (Mikado Method)

<https://understandlegacycode.com/blog/a-process-to-do-safe-changes-in-a-complex-codebase/>

복잡한 코드베이스에서 안전하게 변경하는 체계적 기법.
Ola Ellnestam과 Daniel Brolund가 수년간
레거시 시스템의 기술 부채를 해결한 경험에서
만들었다.

## 이름의 유래

미카도(Mikado)는 막대기 뽑기 게임이다.
빼고 싶은 막대기가 다른 막대기들과 엉켜 있다.
전략은 엉키지 않은 쉬운 막대기부터 빼는 것이다.
점진적으로 엉킴을 풀어가다 보면,
아무것도 건드리지 않고 원하는 막대기를 뺄 수 있다.

## 문제: 늪에 빠지는 리팩터링

30만 줄의 스파게티 코드를 물려받았다.
ORM 의존성을 업그레이드하려고 한다.
업그레이드했더니 프로젝트가 컴파일되지 않는다.
체인지로그를 보니 호출 방식이 바뀌었다.
호출을 고치려면 먼저 추출(extract)해야 한다.
추출하려면 각 호출을 먼저 통일해야 한다.
하나를 고치면 두 개가 새로 터진다.

마지막 커밋이 몇 시간 전이다.
"아직 작업 중이에요, 몇 가지만 더 고치면 돼요"가
며칠째 스탠드업에서 반복된다.

## 프로세스

1. **종이를 꺼낸다** — 로우테크가 답인 순간이 있다.
2. **목표를 설정한다** — 종이 위에 쓴다.
   위쪽이든 가운데든 상관없다.
   다른 항목을 위한 공간을 남긴다.
3. **타임박스 안에 시도한다** — 5분, 10분, 15분.
   짧게 유지한다.
   - **실패하면:**
     1. 되돌린다. 스태시하지 않는다.
        다른 디렉터리에 복사하지 않는다.
        되돌리고 없앤다.
     2. 무엇이 부족한지 생각한다.
        그것이 하위 목표다.
     3. 종이에 쓰고 시도했던 목표에 연결한다.
     4. 하위 목표로 3단계부터 반복한다.
   - **성공하면:**
     1. 커밋한다. 종이에서 체크한다.
     2. 다음 미완료 하위 목표로 돌아간다.
        잎(leaf)부터 처리한다.
     3. 주 목표를 체크할 때까지 반복한다.

## 실전 예시: ORM 업그레이드

### 1단계: 목표 설정

```
  (( ORM 업그레이드 ))
```

이중 원으로 감싼 것이 주 목표다.

### 2단계: 첫 시도 → 실패 → 되돌리기

업그레이드했더니 호출이 깨진다.
되돌린다. 하위 목표를 기록한다.

```
  (( ORM 업그레이드 ))
       /         \
 [호출 A 추출]  [호출 B 추출]
```

### 3단계: 잎에서 시도 → 성공

호출 A 추출은 몇 분 안에 끝난다.
커밋하고 체크한다.

```
  (( ORM 업그레이드 ))
       /         \
 [호출 A 추출]  [호출 B 추출]
      ✓
```

### 4단계: 다음 잎 → 실패 → 더 분해

호출 B 추출을 시도했으나, 호출 형태가 제각각이다.
되돌린다. 먼저 호출을 통일해야 한다.

```
  (( ORM 업그레이드 ))
       /         \
 [호출 A 추출]  [호출 B 추출]
      ✓          /       \
          [호출 B-1    [호출 B-2
           형태 통일]   형태 통일]
```

### 5단계: 잎부터 올라가며 완료

잎을 처리하고, 올라가고, 반복한다.
모든 하위 목표가 체크되면
주 목표는 쉽게 달성된다.

```
  (( ORM 업그레이드 ))  ✓
       /         \
 [호출 A 추출]  [호출 B 추출]  ✓
      ✓          /       \
          [호출 B-1    [호출 B-2
           형태 통일]   형태 통일]
              ✓            ✓
```

## 핵심 원칙

### 되돌리기는 낭비가 아니다

되돌릴 때마다 원점으로 돌아가는 것 같지만,
시작할 때보다 **더 많은 정보**를 가지고 있다.
코드가 항상 컴파일되고 테스트를 통과하는 상태이므로,
IDE의 리팩터링 도구를 자유롭게 쓸 수 있다.

### 잎에서부터 작업한다

그래프에서 선행 조건이 없는 노드,
즉 잎(leaf)부터 처리한다.
하위 목표가 전부 체크되면
상위 목표는 자연히 쉬워진다.

### 언제든 멈출 수 있다

하위 목표를 달성할 때마다 커밋한다.
주 목표에 도달하지 못했더라도,
PR을 열고 배포할 수 있다.
코드는 항상 출하 가능한 상태다.

## 세 가지 실전 조언

1. **타임박스를 짧게 잡는다.** 10분이
   실용적인 타협점이다. 짧을수록 되돌리기가
   쉽고, 매몰 비용의 오류를 피할 수 있다.
2. **목표를 달성하면 즉시 커밋한다.** 주 목표를
   향한 체크포인트다. 아무 때나 멈추고
   PR을 열어 개선 사항을 배포할 수 있다.
3. **야심찬 리팩터링을 시작할 때 쓴다.** 아기
   걸음으로 나아가면서 코드를 출하 가능 상태로
   유지하면, 생산성이 극적으로 올라간다.

## 인사이트

### TDD와의 구조적 유사성

TDD가 "Red → Green → Refactor"로 작은 단위를
반복하듯, 미카도 메서드는
"시도 → 실패 → 되돌리기 → 분해 → 성공 → 커밋"을
반복한다. 둘 다 **되돌리기의 비용을 극단적으로
낮추는 것**이 핵심이다. TDD는 테스트 실패로,
미카도 메서드는 타임박스와 revert로 이를 달성한다.

### 매몰 비용에 대한 체계적 방어

프로그래머가 리팩터링에서 늪에 빠지는 근본 원인은
기술적이 아니라 심리적이다.
"여기까지 했는데 되돌리기엔 아깝다"는 매몰 비용의
오류다. 미카도 메서드는 타임박스와 강제 revert라는
**프로세스 수준의 장치**로 이 심리적 함정을 차단한다.
의지력이 아니라 구조로 해결한다.

### 탐색과 실행의 분리

미카도 메서드의 진짜 힘은 탐색(exploration)과
실행(execution)을 분리하는 데 있다.
처음 시도에서 실패하고 되돌리는 것은 탐색이다.
의존성을 발견하고 그래프에 기록하는 것이 목적이다.
잎에서부터 올라가며 커밋하는 것은 실행이다.
이미 지도가 있으므로 자신 있게 전진한다.
대부분의 리팩터링 실패는 탐색과 실행을
동시에 하려다 발생한다.

### 실패가 제품이다

되돌리기는 낭비가 아니라 **생산**이다.
매번 되돌릴 때마다 그래프에 간선(edge)이 하나
추가된다. 실패할수록 그래프가 완성에 가까워진다.
이것은 반취약성(antifragility)이다.
일반적인 프로젝트에서 실패는 자신감을 깎지만,
미카도 메서드에서 실패는 계획을 강화한다.

### 코드의 결을 따라가는 방법

코드를 몇 시간 읽어도 숨은 의존성을 놓칠 수 있다.
미카도 메서드의 "시도하고 실패하기"는
**숨은 커플링을 강제로 드러내는 탐침**이다.
컴파일 에러와 테스트 실패가
코드가 스스로 자기 구조를 고백하게 만든다.
내가 가정한 구조가 아니라
코드의 실제 구조를 발견하게 된다.

### 과학적 방법의 코드 적용

가설("이 변경을 할 수 있다") → 실험(시도) →
관찰(실패 지점 확인) → 수정(그래프 갱신).
미카도 메서드는 리팩터링에 적용된 과학적 방법이다.
"감으로 리팩터링"에서
"증거 기반 리팩터링"으로 전환한다.

### 복잡성을 시각화하는 도구

미카도 그래프는 단순한 할 일 목록이 아니다.
변경의 **의존성 구조**를 드러낸다.
"이 목표가 왜 어려운가"를 시각적으로 보여준다.
그래프가 깊으면 의존성이 깊다는 뜻이고,
넓으면 영향 범위가 넓다는 뜻이다.
코드를 읽지 않고도 변경의 난이도를 가늠할 수 있다.

### 위상 정렬이라는 수학적 본질

미카도 그래프는 DAG(방향 비순환 그래프)다.
잎에서부터 작업하는 것은 위상 정렬(topological
sort) 순서로 문제를 푸는 것이다.
각 단계가 쉬운 이유는 선행 의존성이
이미 모두 해결된 상태이기 때문이다.
복잡한 문제를 "쉬운 문제의 올바른 순서"로
변환하는 것이 핵심이다.

### 추정의 공포에서 벗어나기

대규모 리팩터링의 추정은 악명 높게 어렵다.
미카도 메서드에서는 전체를 추정할 필요가 없다.
**다음 잎 하나가 타임박스 안에 들어가는가?**
이 질문만 답하면 된다.
전체 소요 시간은 모르지만,
매 커밋마다 진전하고 있다는 것은 안다.
진행 상황은 그래프의 체크 표시가 보여준다.

### 타임박스는 시간 관리가 아니라 판단 장치다

타임박스의 진짜 역할은 효율적인 시간 배분이
아니다. **의사결정 지점을 강제로 만드는 것**이다.
타임박스 없이 작업하면 "조금만 더 하면
될 것 같은데"라는 생각에 끝없이 표류한다.
타임박스는 "지금 진전하고 있는가, 막혀 있는가"를
정기적으로 자문하게 만드는 알람이다.

### 종이의 인지적 기능

디지털 도구 대신 종이를 쓰라는 조언은
단순한 취향이 아니다.
종이는 항상 눈에 보이고,
컨텍스트 스위칭이 없으며,
손으로 그리는 행위가 다른 인지 경로를 활성화한다.
IDE와 종이를 동시에 펼쳐놓으면,
코드(세부)와 그래프(전체)를
**동시에 두 스케일로** 보게 된다.

### 세션 간 지식의 보존

멘탈 모델은 하룻밤 자면 흐려진다.
미카도 그래프는 종이 위에 남는다.
일주일 뒤에 돌아와도 정확히
어디까지 했고 무엇이 남았는지 알 수 있다.
다른 사람에게 넘겨줄 수도 있다.
그래프가 **작업의 직렬화 가능한
스냅샷** 역할을 한다.

### 재작성의 유혹을 구조적으로 차단한다

레거시 코드 앞에서 가장 큰 유혹은
"처음부터 다시 쓰자"다.
미카도 메서드는 **어떤 변경이든 안전한 증분으로
분해할 수 있음**을 실증한다.
"참고 살기" vs "전면 재작성"이라는
거짓 이분법을 깨뜨린다.
세 번째 선택지: 점진적으로, 안전하게, 지금 고친다.

### Strangler Fig 패턴과의 관계

Strangler Fig은 시스템 수준의 점진적 교체,
미카도 메서드는 코드 수준의 점진적 교체다.
둘 다 "기존 것을 작동시킨 채로
새것으로 조금씩 대체한다"는 원칙을 공유한다.
미카도 그래프가 교체 순서를 결정하는
로드맵 역할을 할 수 있다.

### 팀 도구로서의 확장

미카도 그래프는 혼자 쓸 때도 강력하지만,
팀에서 쓸 때 차원이 달라진다.
"다음에 뭘 해야 하지?"라는 질문에
논쟁 없이 답한다 — 잎을 고른다.
페어 프로그래밍의 공유 초점이 된다.
한 사람의 머릿속에만 있던 암묵적 의존성 지식을
**그래프라는 형태로 외부화**한다.

### "리팩터링 중이라 릴리스 못 해요"가 사라진다

미카도 메서드로 작업하면
코드는 항상 출하 가능 상태다.
기술 부채 해소가 비즈니스 일정과 충돌하지 않는다.
이것이 기술적 방법론을 넘어
**조직적 신뢰**를 만드는 지점이다.
신뢰 없이는 경영진의 리팩터링 지원을 얻을 수 없고,
지원 없이는 필요한 리팩터링을 할 수 없다.

### 되돌리기의 문화적 재정의

많은 팀에서 revert는 패배의 인정이다.
"시도했는데 안 됐습니다"라는 고백이다.
미카도 메서드는 revert를 **첫 번째 생산적 행위**로
재정의한다. "못 하겠다"가 아니라
"의존성을 하나 발견했다"가 된다.
이 프레임 전환이 팀의 심리적 안전감을 높이고,
"모른다"를 중간 상태로 정상화한다.
