## 한줄 요약

코드 작성 속도는 소프트웨어 딜리버리의 병목이 아니다.
병목은 불명확한 요구사항, 리뷰 대기열, 배포 공포, 조직 구조에 있다.

## 요약

VP of Engineering이 AI 코딩 어시스턴트 도입으로 "코드 산출량 40% 증가"를
자랑하는 화요일 아침 장면으로 시작한다.
저자의 핵심 질문: "무엇을 향한 속도인가?"

### 골드랫의 제약 이론(Theory of Constraints)

1984년 엘리야후 골드랫의 《The Goal》을 인용한다.
모든 시스템에는 정확히 하나의 제약(병목)이 있고,
시스템 전체의 처리량은 그 병목의 처리량으로 결정된다.
**병목이 아닌 단계를 최적화하면 시스템이 더 빨라지는 게 아니라 더 망가진다.**
병목 앞에 재고(미완성 작업)만 쌓여 대기 시간과 혼란이 증가한다.

### "코드 산출량 3배" 시 실제로 일어나는 일

- PR이 폭증하지만 리뷰어는 늘지 않는다.
- 리뷰 대기 → 컨텍스트 스위칭 → 러버스탬프 승인 → 품질 저하.
- CI 파이프라인, 수동 승인, 스테이징 대기가 겹쳐 WIP 폭발.
- **사이클 타임(커밋~사용자 가치 전달)은 오히려 악화된다.**
- AI가 생성한 코드를 아무도 완전히 이해하지 못한 채 프로덕션에 올라간다.
  → "더 많은 코드, 더 적은 이해. 시한폭탄에 대시보드만 예뻐진 것."

### 실제 병목 5가지

1. **무엇을 만들어야 하는지 모른다** — PM이 사용자와 대화하지 않고,
   요구사항이 슬랙 메시지 전언으로 도착한다.
   코드를 빨리 쓰면 "잘못된 것을 더 빨리 만들 뿐."
2. **코드 "완료" 이후의 모든 것** — 코드 작성은 여정의 20%.
   나머지 80%는 PR 리뷰, CI, QA, 보안 리뷰, 배포 승인 등 대기열이다.
3. **배포 공포의 악순환** — 테스트 불안정, 관측성 부재, 카나리 불신
   → 변경사항을 몰아서 릴리스 → 위험 증가 → 더 무서워짐 → 반복.
4. **출시 후 검증 부재** — 분석도, 사용자 인터뷰도 없이
   "만들고, 출시하고, 모른 척"을 더 빠르게 반복할 뿐이다.
5. **캘린더가 내력벽** — 결정을 위한 회의, 6주짜리 분기 계획,
   한 사람의 승인에 묶인 아키텍처 리뷰.
   조직도가 병목인데 Copilot이 리팩터링해줄 수 없다.

### 대신 해야 할 것

- **가치 흐름을 매핑하라.** 아이디어 → 프로덕션의 모든 단계와 대기 시간 기록.
- **산출량이 아니라 사이클 타임을 측정하라.**
- **대기 상태를 찾아 제거하라.** 페어 프로그래밍, 작은 PR, 비동기 리뷰 등.
- **시작을 멈추고 완료를 시작하라.** WIP 제한 활용.
- **일하는 사람에게 물어보라.** 개발자들은 이미 병목이 어디인지 안다.

## 분석

이 글의 논증 구조는 골드랫의 제약 이론(TOC)이라는 강력한 프레임워크 위에 서 있다.
TOC가 제조업에서 검증된 원리이고, 소프트웨어 딜리버리도 결국 흐름(flow) 시스템이라는
점에서 비유가 정확하게 작동한다.

특히 "병목이 아닌 곳을 최적화하면 시스템이 더 나빠진다"는 핵심 명제를
AI 코딩 어시스턴트라는 시의적절한 사례에 대입한 점이 효과적이다.
PR 폭증 → 리뷰 병목 → 러버스탬프 → 품질 저하라는 인과 사슬이
실무 경험과 강하게 공명한다.

글의 구조도 잘 설계되어 있다.
풍자적 도입(VP의 흥분) → 이론적 근거(골드랫) → 구체적 증상 나열
→ 진짜 병목 5가지 진단 → 처방의 순서로 진행한다.
유머와 냉소가 적절히 섞여 13분 읽기를 지루하지 않게 만든다.

## 비평

### 강점

- **제약 이론의 소프트웨어 적용이 정확하다.**
  리틀의 법칙, WIP 제한, 사이클 타임 같은 린/칸반 개념과 자연스럽게 연결된다.
  이 저장소의 칸반 풀 시스템 문서와도 직접 연결되는 내용이다.
- **실무 체감도가 높다.**
  "PR이 8일 전 코드를 묻는 리뷰 코멘트"나
  "금요일 4:55 승인은 월요일 배포"같은 묘사는 개발자라면 고개를 끄덕일 수밖에 없다.
- **AI 도구에 대한 비판이 도구 자체가 아니라 도입 방식에 집중한다.**
  "AI가 나쁘다"가 아니라 "병목이 아닌 곳에 AI를 투입하는 것이 나쁘다"는 주장이다.
  이 구분이 글의 신뢰도를 높인다.

### 한계

- **병목이 코드 작성인 경우를 완전히 배제한다.**
  초기 스타트업이나 프로토타이핑 단계에서는 실제로 코딩 속도가 병목일 수 있다.
  "거의 절대 아니다(almost never)"라는 한정어를 쓰긴 하지만,
  어떤 조건에서 코딩 속도가 병목이 되는지에 대한 논의가 없다.
- **AI 코드의 이해 불가능성 주장이 과장될 여지가 있다.**
  모든 AI 활용이 "프롬프트 → 훑어보기 → 실행 한 번"은 아니다.
  숙련된 개발자가 AI를 페어 프로그래밍 도구로 쓰는 경우와,
  이해 없이 복붙하는 경우를 구분하지 않는다.
- **"대신 해야 할 것" 섹션이 상대적으로 얕다.**
  가치 흐름 매핑, WIP 제한 같은 처방은 맞지만,
  조직 정치("캘린더가 내력벽")를 어떻게 뚫는지에 대한 구체적 전략이 없다.
  진단은 날카로운데 처방전은 일반론에 머문다.

## 인사이트

### 1. 국소 최적화의 역설은 AI 시대에 더 위험하다

AI 도구의 생산성 향상이 극적이기 때문에,
"이 정도면 전체가 빨라지겠지"라는 착각도 극적으로 강해진다.
40% 코드 산출량 증가라는 숫자가 대시보드에 찍히면
그것이 전체 시스템의 40% 개선이라고 오인하기 쉽다.
**측정 가능한 개선이 보이지 않는 악화를 감추는 함정**이다.

### 2. "더 많은 코드, 더 적은 이해"는 기술 부채의 새로운 형태다

전통적 기술 부채는 "급하게 짠 코드"였다.
AI 시대의 기술 부채는 "아무도 읽지 않은 코드"가 될 수 있다.
코드의 존재와 코드의 이해가 분리되는 현상은
장애 대응(on-call) 능력을 직접적으로 훼손한다.
이것은 단순한 코드 품질 문제가 아니라 **조직 회복력(resilience)의 문제**다.

### 3. 사이클 타임은 "코딩 속도"와 거의 무관하다

글에서 "코드는 오후에 끝났는데 프로덕션까지 2개월"이라는 사례가 등장한다.
이것은 리드 타임의 구성 요소 중 실제 작업 시간(touch time)의 비중이
얼마나 작은지를 극적으로 보여준다.
대부분의 시간은 대기열(queue time)이다.
**대기열을 줄이는 것이 코딩 속도를 높이는 것보다 10배 이상 효과적**일 수 있다.

### 4. "시작을 멈추고 완료를 시작하라"는 직관에 반한다

경영진은 "더 많이 시작하면 더 많이 완료된다"고 가정한다.
그러나 리틀의 법칙(L = λW)은 WIP가 증가하면 리드 타임도 증가한다고 말한다.
AI가 시작을 가속화하면 WIP가 폭발하고,
결과적으로 완료까지의 시간이 오히려 늘어난다.
**AI 도구 도입과 WIP 제한 강화는 반드시 함께 가야 한다.**

### 5. 진짜 경쟁 우위는 "무엇을 만들지 아는 것"이다

저자의 결론이 가장 날카로운 지점이다.
코드를 빨리 쓰는 팀이 이기는 게 아니라,
"무엇을 만들지 파악하고, 만들고, 사용자에게 전달하는" 팀이 이긴다.
AI 시대에 차별화 요소는 코딩 능력이 아니라
**문제 정의 능력과 딜리버리 파이프라인의 효율성**이다.
이것은 기술적 역량이 아니라 조직적 역량이며,
그래서 해결하기가 훨씬 더 어렵고, 그래서 진짜 경쟁 우위가 된다.
