# PRD 템플릿 완전 가이드: 상위 제품 팀 15개 템플릿 분석

원문: <https://www.prodmgmt.world/blog/prd-template-guide>

## 요약

이 글은 Amazon, Intercom, Figma, Basecamp 등 세계 상위 제품 조직에서 실제 사용하는
PRD(Product Requirements Document) 템플릿 15개를 분석한 종합 가이드다.
13개 이상의 회사 PRD를 분석한 결과, 고성과 팀들의 문서에서 세 가지 공통 패턴이
반복된다: **문제 우선(Problem before Solution)**, **명시적 비목표(Non-Goals)**,
**살아있는 문서(Living Documents)**.

5개의 핵심 템플릿이 소개된다. Amazon의 Working Backwards PR/FAQ는 제품 출시를
상상한 보도자료를 먼저 작성하게 강제해, 어떤 고객 가치를 만드는지 명확히 한 뒤
개발을 시작한다. Lenny Rachitsky의 1-Pager는 문제 정의·솔루션·성공 지표·범위·
일정·미결 질문의 6개 섹션으로 가장 광범위하게 채택된 린(Lean) 포맷이다.
Kevin Yien(전 Square)의 PRD 템플릿은 Draft → Problem Review → Solution Review →
Launch Review → Launched의 5단계 워크플로와 단계별 이해관계자 서명을 통해
프로세스 책임을 만든다. Intercom의 Intermission 템플릿은 JTBD(Jobs To Be Done)
철학을 기반으로 한 장짜리 제약을 엄격히 적용하며, “솔루션을 여기에 쓰지 말 것”을
명시한다. Figma의 Coda PRD는 Figma 파일 임베드와 인터랙티브 체크리스트로
문서를 살아있는 프로젝트 허브로 만든다.

린 템플릿으로는 Basecamp의 Shape Up Pitch가 있다. 기능 명세 대신 문제·예산(Appetite)·
대략적 솔루션·위험 요소·비목표의 다섯 가지 요소로 구성하며, “Appetite”라는 개념이
핵심이다. 시간 추정이 아니라 이 작업에 투자할 시간의 상한선을 미리 정해
우선순위를 강제한다. Miro의 PAD(Product Alignment Document)는 2D 캔버스 형식으로
이해관계자가 전체 요소를 한눈에 보고 직접 코멘트할 수 있게 한다.

상위 기업들의 PRD 철학도 소개된다. Stripe는 명세 이전의 “Shaping” 단계를 강조한다.
Linear는 A/B 테스트와 OKR 없이 취향(Taste)과 의견(Opinion)으로 제품을 결정한다.
Spotify는 Data → Insight → Belief → Bet의 DIBB 프레임워크로 전략을 정렬한다.
Meta는 PM이 태스크를 만들지 않고 엔지니어가 프로젝트 관리 대부분을 담당하는
구술 중심(Oral Culture) 조직으로 운영된다.

## 분석

### 문제-솔루션 분리의 보편성

분석된 모든 고성과 템플릿이 문제 섹션과 솔루션 섹션을 명시적으로 분리한다.
Intercom은 “여기에 솔루션을 쓰지 말 것”이라는 문구를 템플릿 자체에 삽입하고,
Asana는 Project Brief(문제) 검토 후에야 Project Proposal(솔루션)로 넘어가는
체크포인트를 강제한다. 이 패턴은 PRD가 해결책 문서가 아닌 문제 이해 도구임을
보여준다. 가장 흔한 제품 실패 원인(잘못된 문제를 완벽하게 해결하는 것)을 구조적으로
예방하려는 공통된 의식이다.

### 비목표(Non-Goals)의 전략적 역할

Kevin Yien, Asana, Shape Up 템플릿은 모두 명시적 비목표 섹션을 포함한다.
범위를 정의하는 것보다 범위 밖을 정의하는 것이 더 중요할 수 있다는 역설이다.
“무엇을 만드는가”보다 “무엇을 만들지 않는가”를 명시함으로써, 개발 중 발생하는
“이것도 해야 하지 않나요?” 논쟁을 사전에 차단한다. Yien의 표현을 빌리면
“솔루션 공간의 경계를 그리면 팀이 그 안을 채우는 방법에 집중할 수 있다.”

### 리빙 문서(Living Document) 패러다임의 부상

Figma의 Coda PRD와 Miro의 PAD는 문서를 스냅샷이 아닌 프로젝트 허브로 재정의한다.
Figma 파일이 직접 임베드되어 디자인이 업데이트될 때 PRD도 자동으로 반영되고,
마일스톤 테이블과 런치 체크리스트가 인터랙티브하게 작동한다. 이는 “문서를 작성한다”는
행위가 “문서를 유지한다”는 지속적 실천으로 전환되고 있음을 의미한다.
노션, Coda, Miro 같은 도구의 확산이 이 패러다임 전환을 가속하고 있다.

### 각 조직 철학의 PRD 투영

PRD 형식은 조직의 의사결정 철학을 반영한다. Amazon의 PR/FAQ는 쓰기 중심 문화와
고객 집착을 구조화한다. Linear의 “취향 기반 결정”과 PRD 없는 운영은 소수 정예
고밀도 팀의 자율성 철학을 보여준다. Meta의 엔지니어 주도 실행은 PM 역할을
전략·비전·파트너십으로 좁히는 사고방식에서 나온다. Spotify의 DIBB는 데이터를
믿음의 근거로, 믿음을 베팅의 이유로 연결하는 인과 사슬을 조직 문화로 만든다.

## 비평

### 강점: 템플릿을 철학으로 설명한다

이 글의 가장 강한 지점은 각 템플릿을 나열하는 데 그치지 않고,
그 템플릿이 탄생한 조직 철학과 연결 짓는다는 것이다.
Intercom의 “한 장 제약”이 “문서가 길수록 읽히지 않는다”는 Paul Adams의 통찰에서
나왔음을 보여주는 방식은, 독자가 형식을 도구가 아닌 사고방식으로 이해하게 돕는다.
실제 사용 가능한 복사 가능 링크를 제공한 점도 실용성을 높인다.

### 약점: “15개 템플릿”이라는 제목과 실제 내용의 불일치

제목은 “15개 템플릿”을 약속하지만 글에서 명시적으로 설명되는 템플릿은 훨씬 적다.
Uber, Asana, Miro 등 일부는 한두 단락으로 약식 처리된다.
“15개”는 과장된 마케팅 헤드라인이며, 실제 심층 분석 대상은 5개 내외다.
독자에게 기대 불일치를 유발할 수 있다.

### 약점: 도입 맥락 없는 도구 추천

Figma PRD는 Coda를 전제하고, Miro PAD는 Miro를 전제하며, Shape Up은
Basecamp 방법론 전체를 전제한다. 각 템플릿이 어떤 도구 생태계, 팀 규모,
개발 문화를 전제하는지 더 명확히 구분했다면 실용적 선택 가이드가 됐을 것이다.
예를 들어 Shape Up은 8주 사이클과 “No Sprint” 문화가 전제되지 않으면
Appetite 개념 자체가 공허해진다는 점이 충분히 강조되지 않는다.

### 약점: 다양한 직종 역할의 비대칭 처리

Meta 사례처럼 PM 역할이 최소화된 조직의 PRD 부재를 소개하면서,
이런 접근이 언제 유효하고 언제 실패하는지의 조건을 다루지 않는다.
Meta 모델은 엔지니어 밀도가 극단적으로 높은 조직에서나 작동하는데,
이를 일반화하면 PRD 자체가 불필요하다는 잘못된 결론을 유도할 수 있다.

## 인사이트

### 템플릿은 사고 순서를 강제하는 설계다

PRD 템플릿을 “빈칸을 채우는 문서”로 보는 시각은 핵심을 놓친다.
Intercom이 “솔루션을 여기에 쓰지 말 것”이라는 문구를 템플릿 안에 삽입한 것,
Asana가 문제 검토 체크포인트를 솔루션 섹션 직전에 배치한 것,
Kevin Yien이 5단계 상태 추적기를 헤더에 넣은 것은 모두 동일한 목적이다.
팀이 자연스럽게 빠지는 “해결책 먼저” 패턴을 구조로 막는 것이다.

소프트웨어의 UI/UX 설계에서 “기본값(Default)”이 사용자 행동을 결정하듯,
문서 템플릿의 구조는 팀의 사고 순서를 결정한다. 문제 섹션이 솔루션 섹션 앞에 오면,
아무리 솔루션 중심으로 사고하는 팀이라도 먼저 문제를 채워야 한다는 마찰이 생긴다.
이 마찰이 행동을 바꾼다. 즉, 좋은 PRD 템플릿은 좋은 사고를 강제하는 인지 설계물이다.

이 관점에서 보면, 팀에 맞는 템플릿을 고르는 것보다 더 중요한 일은 팀이 회피하는
사고 패턴이 무엇인지 먼저 진단하는 것이다. 팀이 범위를 확장하는 경향이 있다면
비목표 섹션이 필요하고, 솔루션으로 먼저 달려드는 경향이 있다면 문제-솔루션 분리가
필요하다. 템플릿은 약점을 교정하는 교정기이지, 모든 팀에 통용되는 정답이 아니다.

### Appetite는 시간 추정이 아닌 우선순위 결정이다

Basecamp의 Shape Up이 도입한 “Appetite” 개념은 소프트웨어 개발의 오랜 관행인
시간 추정(Estimation)에 대한 근본적 비판이다. 전통적 접근은 “이 기능을 만드는 데
얼마나 걸리는가?”를 묻는다. Appetite는 “이 기능이 우리 시간을 얼마나 받을 자격이
있는가?”를 묻는다.

이 차이는 사소해 보이지만 심대한 함의를 가진다. 시간 추정은 기능의 존재를 전제하고
비용을 계산한다. Appetite는 기능의 가치를 먼저 판단하고 그에 맞는 예산을 할당한다.
“2주 Appetite”로 설정된 기능은 어떻게 구현해도 2주를 초과할 수 없다.
이 제약이 엔지니어와 디자이너에게 “가장 단순한 형태로 문제를 해결하는 방법”을
찾도록 강제한다. 복잡성을 후속 개선으로 미루는 의사결정 권한을 팀에 준다.

이 접근이 스타트업 환경에서 특히 강력한 이유는, 초기 제품에서 “완벽한 구현”보다
“작동하는 구현”이 거의 항상 더 가치 있기 때문이다. Appetite는 MVP(최소 기능 제품)
철학을 사고 도구로 코드화한 것이다. 완성도에 대한 논쟁 없이 범위를 자동으로
조정한다.

### 조직의 PRD 문화는 의사결정 권한 구조의 거울이다

Meta에서 PM이 티켓을 만들지 않고 엔지니어가 프로젝트 관리를 담당한다는 사실,
Linear가 PM 타이틀을 가진 사람이 50명 이상의 직원 중 단 1명이라는 사실은
놀라운 이탈처럼 보인다. 그러나 이는 PRD 불필요론이 아니라, 의사결정 권한의
분산에 관한 이야기다.

전통적 PRD 문화에서 PM은 요구사항을 정의하고 엔지니어는 구현한다.
이 분업은 명확하지만 정보 손실을 수반한다. 구현 과정에서 발생하는 판단들이
매번 PM을 경유해야 하기 때문이다. Meta와 Linear 모델은 이 병목을 제거한다.
“왜 만드는가”에 대한 맥락을 엔지니어가 직접 이해하면, 세부 결정을 위한
지속적 커뮤니케이션 비용이 줄어든다.

그러나 이 모델은 전제 조건이 있다. 엔지니어가 비즈니스 맥락을 이해하고,
제품 전략에 대한 공유된 감각이 있으며, 잘못된 구현에 대한 비용 감내력이 있어야 한다.
이 전제가 없는 조직에서 PRD를 없애면 혼돈만 남는다. 즉, PRD의 형식은 조직의
신뢰 구조와 커뮤니케이션 밀도를 반영한다. PRD가 얇아지는 것은 조직이 성숙해진
신호일 수도 있고, 검증 없이 모방한 위험 신호일 수도 있다.

### “문서를 쓴다”는 행위는 생각을 완성하는 과정이다

Amazon의 Werner Vogels가 PR/FAQ 과정을 거치면 “무엇을 만들 계획인지가 훨씬
명확해진다”고 말한 것, Intercom의 Paul Adams가 “한 장 안에 담지 못하면
문제에 대한 충분히 명확한 시각이 없는 것”이라고 말한 것은 같은 통찰을 공유한다.
문서 작성은 이미 명확한 생각을 기록하는 행위가 아니라, 생각을 명확하게 만드는
과정이다.

이 생각은 글쓰기 연구에서도 지지된다. 분산된 아이디어를 선형 텍스트로 변환하는
과정에서 논리의 구멍, 가정의 모순, 놓친 질문이 드러난다. 특히 “문제 설명” 섹션을
작성하는 행위는 팀이 막연히 공유하고 있다고 믿었던 이해가 실제로는 얼마나 다른지
드러내는 리트머스 시험지가 된다.

따라서 PR/FAQ가 “아이디어의 저렴한 검증” 수단으로 기능하는 것은 부수적 효과가 아니라
핵심 가치다. 쓰기 어렵다는 것은 아이디어가 불명확하다는 신호다.
거절당하는 PR/FAQ가 많다는 것이 “버그”가 아니라 “기능”이라는 Amazon의 설명은
이 철학의 정직한 표현이다. PRD는 승인받기 위한 문서가 아니라, 생각하기 위한 도구다.
