# Asio

<https://github.com/chriskohlhoff/asio>

## 소개

asio는 크로스 플랫폼 C++ 네트워크·저수준 I/O 라이브러리다.
Chris Kohlhoff가 2003년부터 관리해온 프로젝트로,
지금은 Boost.Asio와 standalone asio라는 두 갈래로 배포된다.
오늘 GitHub 트렌딩에 올라온 저장소는 standalone 쪽 소스이며,
현재 6,100개 이상의 스타와 1,500개 이상의 포크를 보유하고 있고
오늘 하루에만 76개의 스타가 새로 붙었다.

라이브러리의 핵심 목표는 소켓, 타이머, 시그널, 파일 I/O 같은
운영체제 수준의 비동기 작업을
하나의 일관된 C++ 모델로 감싸는 것이다.
C++11 이상을 타깃으로 하면 별도 의존성 없이 헤더만으로 쓸 수 있고,
Boost 전체를 끌어오지 않아도 되는 `ASIO_STANDALONE` 모드가
바로 이 저장소의 존재 이유다.
최신 릴리스는 1.38.1이며, 릴리스 주기가 여전히 활발하다는 점도
2003년생 라이브러리치고는 이례적이다.

## 분석

### 프록시가 아니라 프로액터라는 설계 선택

asio의 근간은 프로액터(proactor) 패턴이다.
리액터(reactor) 패턴이 “소켓이 준비됐다”는 신호만 알려주고
실제 읽기·쓰기는 애플리케이션이 수행하는 방식이라면,
프로액터는 운영체제나 런타임이 작업 자체를 완료한 뒤
완료 결과를 콜백으로 통지하는 방식이다.
이 차이는 사소해 보이지만 플랫폼 이식성 문제를 정면으로 건드린다.
Windows의 IOCP는 원래 완료 기반으로 설계되어 있고,
반대로 Linux의 epoll이나 BSD의 kqueue는 준비 상태 통지 기반이다.
asio는 이 둘을 하나의 프로액터 인터페이스 아래로 통합하면서,
Linux·BSD 쪽에서는 내부적으로 리액터를 흉내 낸 뒤
완료 형태로 재포장하는 어댑터 계층을 둔다.

이 통합이 왜 중요한가 하면, 애플리케이션 코드가
플랫폼별 이벤트 루프 차이를 전혀 몰라도 되기 때문이다.
`async_read`를 호출하고 완료 핸들러만 넘기면,
그 아래에서 IOCP를 쓰든 epoll을 쓰든 신경 쓸 필요가 없다.
동시에 이 계층은 성능 손실을 감수해야 하는 지점이기도 하다.
epoll 위에 프로액터를 흉내 내려면 read/write 준비 통지를 받은 뒤
실제 시스템 콜을 한 번 더 호출해야 하므로,
IOCP 네이티브 환경보다 한 단계가 더 들어간다.
asio는 이 비용을 두고 이식성과 일관된 API를 택했고,
그 판단이 20년 넘게 유지되고 있다는 사실 자체가
설계의 견고함을 보여주는 증거다.

### io_context와 실행자(Executor) 모델의 분리

초기 asio는 `io_service`라는 단일 객체가 이벤트 루프, 작업 큐,
스레드 안전성 보장을 모두 떠맡는 구조였다.
이후 버전에서 이를 `io_context`로 이름을 바꾸고,
실행 그 자체를 표현하는 실행자(executor) 개념을 분리해냈다.
이 변화는 asio가 C++ 표준화 위원회의 실행자 제안(P0443 계열)과
궤를 같이하며 발전해왔다는 사실을 보여준다.
즉 asio는 단순히 네트워킹 라이브러리에 머무르지 않고,
“비동기 작업을 어디서, 어떤 순서로 실행할 것인가”라는
더 일반적인 문제에 대한 참조 구현 역할을 자처해왔다.

`strand`라는 개념도 이 흐름 위에 있다.
여러 스레드가 하나의 `io_context`를 공유해 실행할 때
특정 핸들러들이 서로 겹치지 않도록 직렬화를 보장하는 장치인데,
락을 명시적으로 걸지 않고도 데이터 경쟁을 피할 수 있게 해준다.
이런 장치들이 쌓이면서 asio는 “소켓 감싸는 라이브러리”에서
“비동기 실행 모델 자체를 제공하는 프레임워크”로 성격이 바뀌었고,
이는 사용자가 배워야 할 개념의 총량도 함께 늘렸다는 뜻이다.

### 코루틴 통합이 보여주는 완료 토큰 추상화의 힘

asio의 비동기 함수는 콜백, `std::future`, Boost.Coroutine,
그리고 C++20 코루틴(`co_await`)까지 동일한 함수 시그니처로
받아들일 수 있다.
이것이 가능한 이유는 “완료 토큰(completion token)”이라는
추상화 계층 덕분이다.
`async_read(socket, buffer, token)`에서 `token` 자리에
무엇을 넘기느냐에 따라 반환 타입과 실행 방식이 통째로 바뀐다.
콜백을 넘기면 즉시 반환하는 fire-and-forget 방식이 되고,
`asio::use_awaitable`을 넘기면 `co_await`로 기다릴 수 있는
awaitable이 반환된다.

이 설계는 API 표면을 늘리지 않으면서도
호출부가 원하는 프로그래밍 모델을 자유롭게 고르게 해준다는 점에서
정교하다.
실제로 C++20이 널리 쓰이기 시작하면서
많은 프로젝트가 콜백 지옥을 벗어나
`co_await` 기반의 순차적 코드로 비동기 로직을 작성하는 쪽으로
옮겨갔고, asio는 언어 차원의 코루틴이 표준화되기 전부터
이미 비슷한 경험을 콜백과 future로 흉내 내고 있었다.
결과적으로 asio 사용자는 언어가 새로운 기능을 얻을 때마다
라이브러리를 바꾸지 않고도 완료 토큰만 바꿔서
자연스럽게 올라탈 수 있었다.

## 비평

### 하나의 저장소, 두 개의 이름이 만드는 혼란

Boost.Asio와 standalone asio는 사실상 같은 코드베이스에서
나오지만, 문서·이슈 트래커·버전 번호가 완전히 통일되어 있지 않다.
어떤 버그는 Boost 쪽 저장소에만 보고되고,
어떤 기능은 standalone 쪽에 먼저 들어간 뒤
다음 Boost 릴리스에 반영되는 시차가 존재한다.
신규 사용자가 “asio를 쓰려면 Boost를 설치해야 하나,
아니면 이 저장소만 받으면 되나”를 구분하는 데
불필요한 학습 비용을 치러야 한다는 점은
이 프로젝트의 오래된 구조적 약점이다.

### 프리미티브 라이브러리라는 정체성과 진입장벽

asio는 HTTP 클라이언트나 웹소켓 서버 같은 완성된 도구가 아니라
소켓·타이머·스트림을 다루는 저수준 프리미티브 모음이다.
그래서 실무에서 asio를 직접 가져다 쓰는 경우보다
Boost.Beast, gRPC, drogon 같은 상위 라이브러리가
asio를 내부 엔진으로 감싸서 쓰는 경우가 훨씬 흔하다.
이 자체는 계층화가 잘 된 설계라고 볼 수도 있지만,
동시에 asio 문서만 읽고 실전 서버를 짜려는 사람에게는
빈틈이 많다는 뜻이기도 하다.
완료 토큰, 실행자, strand, 버퍼 시퀀스 같은 개념을
전부 이해해야 비로소 안전한 코드를 짤 수 있는데,
이 개념들은 서로 얽혀 있어 하나씩 순서대로 배우기가 어렵다.

### 완료 핸들러 생명주기 관리를 사용자에게 떠넘긴다

asio는 스마트 포인터, `shared_from_this`, `strand`를 조합해
비동기 객체의 생명주기를 안전하게 관리하는 패턴을 문서에서
반복해서 권장한다.
바꿔 말하면 객체가 콜백 완료 시점까지 살아있음을 보장하는 책임이
전적으로 사용자 코드에 있다는 뜻이다.
러스트의 소유권 시스템이나 가비지 컬렉션 언어의 런타임과 달리
asio는 이 문제를 언어나 라이브러리 차원에서 강제하지 않는다.
그 결과 use-after-free나 댕글링 콜백 버그는
asio를 다루는 코드베이스에서 반복적으로 발생하는 사고 유형이며,
이는 라이브러리의 성능과 유연성이
안전성과 맞바꾼 대가라는 점을 냉정하게 봐야 한다.

## 인사이트

### 표준 라이브러리에 스며든 프로젝트의 흔적

asio의 실행자·완료 토큰 설계는 C++ 표준 네트워킹 제안과
실행자 제안 양쪽에 실질적인 영향을 미쳤다.
표준화 위원회가 실제로 동작하는 구현체를 참고해
언어 기능을 다듬는다는 점에서,
asio는 “라이브러리가 언어 표준을 앞서 실험하는 실험실” 역할을
오랫동안 해왔다.
이는 초기 Boost 라이브러리들이 `shared_ptr`이나 `function`처럼
훗날 표준 라이브러리로 흡수된 것과 같은 패턴이다.
다만 네트워킹 TS는 결국 C++23에 정식으로 들어가지 못했고,
asio 자체가 사실상 참조 구현이자 최종 목적지로 남아버린 상황은
표준화 과정의 더딘 속도와 실무 라이브러리의 실용주의가
어떻게 어긋날 수 있는지를 보여주는 사례이기도 하다.

### 저수준 프리미티브가 낳는 생태계 레버리지 효과

asio가 직접 노출을 최소화하고 프리미티브에 집중한 결과,
그 위에 Beast(HTTP/웹소켓), cpprestsdk, drogon, gRPC의 일부
전송 계층처럼 서로 다른 목적의 라이브러리들이
독자적인 상위 계층을 쌓아올릴 수 있었다.
이것은 유닉스 철학에서 말하는 “작은 도구를 조합한다”는 원칙이
네트워킹 라이브러리 영역에서도 그대로 재현된 사례로 볼 수 있다.
하지만 이 레버리지에는 대가가 따른다.
asio의 API가 바뀌면 그 위에 쌓인 모든 상위 라이브러리가
연쇄적으로 영향을 받고, 실제로 실행자 모델이 개편될 때마다
Beast를 비롯한 하위 프로젝트들이 대응 릴리스를 내야 했다.
저수준에 자리 잡은 인프라 라이브러리일수록
변경 하나가 만드는 파급 범위가 넓어진다는 점은
asio뿐 아니라 모든 기반 라이브러리가 안고 가는 긴장이다.

### 20년 이상 유지된다는 사실이 주는 신뢰와 그 이면의 정체 위험

asio는 2003년 이후 지금까지 활발히 유지보수되고 있는
몇 안 되는 C++ 인프라 라이브러리 중 하나다.
이 정도 수명은 asio가 특정 프레임워크나 유행에 종속되지 않고
운영체제 I/O라는 변하지 않는 문제 영역에 집중했기 때문에
가능했다고 볼 수 있다.
반면 이렇게 오래 한 사람(Chris Kohlhoff)의 판단에
설계 방향이 크게 의존해온 프로젝트는
메인테이너 리스크에서 자유롭지 않다.
대규모 인프라급 오픈소스가 소수 관리자에게 장기간 의존하는 구조는
curl이나 sqlite에서도 반복적으로 지적되어온 패턴이며,
asio 역시 이 구조적 취약점에서 완전히 벗어나 있지는 않다.
릴리스가 여전히 정기적으로 나온다는 사실이
이 위험을 가려주고 있을 뿐, 근본적으로 해소하지는 못한다.
