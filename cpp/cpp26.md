# C++26 완성 — C++11 이후 최대 업그레이드

> 원문:
> <https://herbsutter.com/2026/03/29/c26-is-done-trip-report-march-2026-iso-c-standards-meeting-london-croydon-uk/>

## 요약

2026년 3월, ISO C++ 위원회가 런던 크로이든 회의에서 C++26의 기술 작업을
완료했다. 24개국에서 약 210명(대면 130명, 원격 80명)이 참석해 411건의 국제
의견을 처리했다. C++11 이후 가장 변혁적인 릴리스로 평가되며, 4대 핵심 기능은
리플렉션, 메모리 안전성 강화, 계약(Contracts), std::execution이다.

## 4대 핵심 기능

### 1. 리플렉션 (Reflection)

C++이 자기 자신을 기술할 수 있는 첫 컴파일 타임 리플렉션 기능이다. 템플릿 발명
이래 가장 큰 업그레이드로 평가된다. 타입 정보를 컴파일 타임에 질의하고 코드를
생성할 수 있으므로, 보일러플레이트 코드를 극적으로 줄일 수 있다.

### 2. 메모리 안전성 강화

두 가지 층위의 개선이 포함된다:

- **미초기화 변수 읽기의 UB 제거**: C++26으로 재컴파일하기만 해도 이 취약점
  범주가 완전히 사라진다. `[[indeterminate]]` 속성으로 기존 UB 동작을 선택적으로
  복원할 수 있다.
- **강화된 표준 라이브러리**: vector, span, string, string_view 등에 경계 안전
  검사를 추가했다. Google 규모의 배포에서 평균 0.3% 성능 오버헤드로 500개 이상의
  버그를 방지했으며, segfault가 30% 감소했다.

### 3. 계약 (Contracts)

언어 수준의 전제 조건(precondition), 사후 조건(postcondition),
`contract_assert`를 도입한다. C의 `assert` 매크로를 언어 기능으로 승격시킨
것이다. 투표 결과 찬성 114, 반대 12, 기권 3으로 채택되었다.

### 4. std::execution (Sender/Receiver)

C++의 비동기 모델로, 동시성과 병렬성을 통일된 방식으로 표현한다. 구조화된
동시성(structured concurrency)을 지원하여 데이터 경쟁을 구조적으로 방지한다.

## 채택 전망

빠른 산업 채택이 예상된다:

- C++11 이후 가장 수요가 높은 기능 세트
- GCC와 Clang이 이미 리플렉션과 계약을 trunk에 병합 완료
- C++29는 타입·메모리 안전성 추가 강화에 집중할 예정

## 분석

C++26은 두 가지 거대한 압력에 대한 응답이다.

첫째, **Rust의 압력**. Rust가 메모리 안전성을 언어 수준에서 보장하면서, C++의
"성능은 좋지만 안전하지 않다"는 인식이 굳어졌다. 미국 백악관(ONCD)이 2024년에
메모리 안전한 언어로의 전환을 권고한 이후, C++은 "안전해질 수 있다"는 것을
증명해야 했다. 경계 검사의 0.3% 오버헤드 데이터는 이 증명의 핵심 근거다.

둘째, **언어 복잡성에 대한 피로**. C++는 매 표준마다 기능이 추가되면서
학습 곡선이 가팔라졌다. C++26은 기능 추가보다 기존 코드의 안전성을
재컴파일만으로 개선한다는 점에서, "기능을 더하되 부담은 줄인다"는
드문 균형을 시도한다.

## 비평

### 리플렉션: 강력하지만 남용이 걱정된다

컴파일 타임 리플렉션은 C++에 메타프로그래밍의 새 차원을 연다. 직렬화, ORM,
GUI 바인딩, 테스트 프레임워크 등에서 보일러플레이트를 제거할 잠재력이 크다.
그러나 C++ 템플릿 메타프로그래밍의 역사가 보여주듯, 강력한 컴파일 타임 기능은
"읽을 수 없는 코드"의 온상이 되기 쉽다.

템플릿이 처음 도입되었을 때도 단순한 제네릭 프로그래밍이 목적이었지만, 결국
Alexandrescu의 "Modern C++ Design" 수준의 암호 같은 코드로 진화했다. 리플렉션이
같은 경로를 밟을 가능성은 높다. "할 수 있다"와 "해야 한다" 사이의 거리가
C++에서는 유독 멀다.

### 계약: Stroustrup의 우려는 타당한가

Bjarne Stroustrup이 계약 기능을 "불완전하고 위원회식으로 부풀려진 설계"라고
비판했다는 Hacker News 댓글은 주목할 만하다. 표준 위원회의 합의 과정은
"모든 사용 사례를 포괄하려는" 방향으로 흐르기 쉽고, 그 결과 개별 사용 사례에
최적화되지 않은 범용적이지만 어색한 설계가 나올 수 있다.

그러나 114 대 12라는 압도적 찬성표는 커뮤니티가 불완전하더라도 계약 기능의
도입을 강하게 원했음을 보여준다. "완벽한 계약"을 기다리다가 영원히 도입하지
못하는 것보다, 불완전하더라도 시작해서 점진적으로 개선하는 것이 현실적이다.

### 0.3%의 의미

경계 검사의 0.3% 오버헤드는 인상적인 수치다. 그러나 이것은 "평균"이다.
실시간 시스템, HFT(고빈도 매매), 게임 엔진의 핫 루프에서 0.3%가
의미하는 바는 일반 서버 애플리케이션에서와 다르다. "평균적으로 안전하다"와
"내 사용 사례에서도 안전하다"는 다른 명제이며, 성능에 민감한 C++ 사용자들은
바로 이 차이를 물을 것이다.

## 인사이트

### C++의 진짜 경쟁 상대는 Rust가 아니라 관성이다

C++26의 메모리 안전성 강화를 "Rust에 대한 대응"으로 읽는 것은 표면적으로는
맞지만, 본질을 놓친다. Rust로 전환하는 프로젝트는 신규 프로젝트가 대부분이다.
수십억 줄의 기존 C++ 코드베이스는 Rust로 재작성될 가능성이 극히 낮다.
C++26의 진짜 전략은 "Rust 대신 C++을 선택하게 만드는 것"이 아니라, "기존
C++ 코드베이스를 재컴파일만으로 안전하게 만드는 것"이다.

이 관점에서 C++26의 가장 혁명적인 기능은 리플렉션이나 계약이 아니라,
**재컴파일만으로 미초기화 변수 UB가 사라지는** 것이다. 코드 한 줄 안 바꾸고
컴파일러만 업그레이드하면 취약점 범주 하나가 통째로 사라진다. 이것이 수십억
줄의 레거시 코드에 미치는 영향은 어떤 새 기능보다 크다.

### 언어 설계의 딜레마: 추가할수록 빼기 어려워진다

C++26의 4대 기능 모두가 순수 추가(addition)이며, 어떤 것도 기존 기능을
제거하거나 단순화하지 않는다. C++의 역사적 약속—"하위 호환성을 깨지
않는다"—은 기능의 단조 증가를 보장한다. 이것은 언어의 수명을 늘리지만
학습 곡선을 끝없이 가팔라지게 만든다.

Rust, Go, Zig 같은 후발 언어가 초기에 "단순함"을 무기로 내세울 수 있는
이유는 역사적 짐이 없기 때문이다. C++은 40년의 역사적 짐을 지고 달리면서도
현대적 기능을 추가해야 하는 독특한 위치에 있다. C++26이 "C++11 이후 최대
업그레이드"라는 수사가 축하인 동시에 경고인 이유다.

### 컴파일러가 표준을 앞서는 시대

GCC와 Clang이 이미 리플렉션과 계약을 trunk에 병합했다는 것은, 표준 확정
전에 구현이 선행되었음을 의미한다. 이것은 C++ 생태계의 중요한 구조적 변화다.
과거에는 표준 확정 → 컴파일러 구현 → 산업 채택이라는 순서였지만, 이제
컴파일러 구현이 표준 확정과 병행되거나 심지어 앞선다.

이 변화의 함의는 C++26 채택 속도가 이전 표준들보다 빠를 것이라는 점이다.
C++11이 확정 후 산업에서 본격적으로 쓰이기까지 3-5년이 걸렸지만, C++26은
표준 확정 시점에 이미 주요 컴파일러가 핵심 기능을 지원하므로 그 시차가
대폭 줄어든다.

### 안전성의 스펙트럼: "안전한 C++"은 가능한가

C++26의 메모리 안전성 접근법은 Rust와 근본적으로 다르다. Rust는 소유권
시스템으로 안전하지 않은 코드를 **작성 불가능하게** 만든다. C++26은 경계
검사로 안전하지 않은 코드의 **결과를 완화**한다. 전자는 예방이고 후자는
완화다.

이 차이가 중요한 이유는 완화 전략에는 항상 빈틈이 있기 때문이다. 경계 검사는
버퍼 오버플로를 막지만, use-after-free, 이중 해제, 데이터 경쟁은 여전히
가능하다. C++29가 "타입·메모리 안전성 추가 강화"를 로드맵에 올린 것은
C++26만으로는 안전성 문제가 완결되지 않음을 위원회 스스로 인정한 것이다.

그럼에도 C++26의 접근법이 현실적으로 더 영향력이 클 수 있다. 수십억 줄의
기존 코드를 Rust로 재작성하는 것은 불가능하지만, C++26으로 재컴파일하는
것은 가능하기 때문이다. 완벽한 안전보다 광범위한 개선이 총 피해를 더 많이
줄일 수 있다. 공중 보건의 논리와 같다—백신 효과가 95%든 70%든, 전 인구에
접종하면 집단 면역이 형성된다.

### C++의 진화가 보여주는 언어 생태계의 동역학

C++26을 단독으로 보면 기술적 성취이지만, 언어 생태계 전체의 맥락에서 보면
**경쟁이 혁신을 강제하는** 동역학의 증거다. Rust가 없었다면 C++이 메모리
안전성을 이 정도로 진지하게 다뤘을까? Java의 리플렉션이 없었다면 C++ 리플렉션이
이 시점에 들어왔을까? D 언어의 계약(contracts)이 선행하지 않았다면?

프로그래밍 언어는 고립된 기술이 아니라 생태계 내에서 서로를 자극하며
진화하는 유기체다. C++26의 4대 기능 모두가 다른 언어에서 먼저 검증된
아이디어의 C++ 번역이라는 점이 이를 증명한다. 혁신은 발명이 아니라
적시적 적용이며, C++26은 이 교훈의 40년짜리 사례다.
