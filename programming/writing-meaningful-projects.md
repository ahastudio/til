# 의미가 보이는 프로젝트 만들기

원문: <https://oborona.zip/post/-c>

## 요약

저자는 AI 시대에 개인 프로젝트를 작성하는 방식에 대한 철학을 `todox`라는 최소한의 TODO 앱을 예시로 설명한다.
`todox`는 D-Bus 알림 시스템과 직접 통합된 간단한 C 프로그램으로, 복잡한 스케줄링 소프트웨어 없이 반복 알람을 관리한다.

저자의 핵심 주장은 명확한 책임 분리다.
핵심 앱은 단순하게 유지하고, 복잡한 논리(달력 변환, ISO8601 포맷팅, 고급 반복 패턴)는 외부 스크립트(Perl, Python)에 위임한다.
이 접근의 장점은 책임 범위의 명확성, 디버깅의 용이성, 프로그래밍의 즐거움 유지에 있다.

구현 규모는 C 1,193줄, Perl 946줄이다.
Perl 래퍼 스크립트 `todoxy.pl`이 사용자 친화적 입력을 표준 타임스탬프로 변환해 `todox`에 전달한다.
이를 통해 에티오피아, 이슬람, 음력 달력 같은 복잡한 달력 시스템도 CPAN 라이브러리를 통해 핵심 C 코드를 수정하지 않고 지원할 수 있다.

저자는 이런 단순한 설계가 AI 시스템에게 현재는 어려운 영역이라고 본다.
모델링과 아키텍처 결정은 AI가 단일 요청으로 생성하기 어려운 인간 판단의 영역이라는 것이다.

## 분석

### 관심사 분리를 Unix 철학으로 구현하는 방식

저자의 설계는 Unix 철학의 핵심인 “한 가지를 잘 하는 작은 프로그램들을 조합하라”를 충실하게 따른다.
`todox`는 알람 저장과 D-Bus 알림만 담당하고, 복잡한 달력 계산은 Perl로 위임한다.
이 분리는 각 컴포넌트를 독립적으로 이해하고, 테스트하고, 교체할 수 있게 만든다.

특히 흥미로운 점은 사용자 인터페이스 레이어를 완전히 별도의 언어(Perl)로 구현한 것이다.
Perl의 CPAN 생태계에 달력 변환 라이브러리가 있다면 C 코드를 건드리지 않고 새로운 달력 시스템을 지원할 수 있다.
이 유연성은 설계 단계의 경계 설정에서 나온다.

### AI 시대에 “이해 가능한 코드”의 가치

저자는 AI가 코드 생성 비용을 낮추면서 아키텍처 결정의 중요성이 높아진다고 암시한다.
AI가 개별 코드 라인을 쉽게 생성하더라도, 어떤 레이어가 어떤 책임을 가져야 하는지의 모델링은 여전히 인간의 판단이 필요하다.
C로 코어를 작성하고 Perl로 래퍼를 작성하는 결정, D-Bus를 직접 사용하는 결정, 외부 스케줄러 없이 자체 알람 관리를 하는 결정 — 이 결정들의 총합이 시스템의 복잡도와 유지보수 용이성을 결정한다.

### 1,193줄의 C와 946줄의 Perl이 보여주는 것

이 규모의 프로젝트는 “장난감 프로젝트”와 “실제 사용 도구” 사이의 적절한 지점에 있다.
너무 크지 않아 전체를 머릿속에 담을 수 있고, 너무 작지 않아 실제 문제를 해결한다.
저자가 프로그래밍의 즐거움을 유지하는 열쇠로 이 규모의 이해 가능성을 든다.
대규모 코드베이스에서 나타나는 “내가 왜 이 코드를 작성했는지 모르겠다”는 감각이 이 규모에서는 발생하지 않는다.

## 비평

### 예시 프로젝트가 주장의 일반화 가능성을 제한한다

`todox`는 단순한 데이터 구조를 가진 개인 도구다.
저자의 설계 원칙이 이보다 복잡한 시스템—여러 사용자가 협업하는 서비스, 분산 상태를 관리하는 시스템, 외부 API와 통합이 많은 도구—에도 동일하게 적용되는지는 논증되지 않는다.
단순한 예시로 복잡한 시스템 설계 원칙을 일반화하는 것은 논리적 도약이다.

### C와 Perl의 조합은 접근성을 낮춘다

C로 코어를 작성하고 Perl로 래퍼를 작성하는 선택은 저자의 역량과 선호를 반영하지만, 이 스택을 유지보수할 수 있는 기여자의 풀이 좁다.
개인 도구라면 문제가 아니지만, 이 설계 철학을 팀 프로젝트에 적용하면 버스 팩터(bus factor)가 낮아질 위험이 있다.
현대적인 대안(Go나 Rust로 단일 바이너리로 구현, Python으로 스크립트 통합)이 동일한 원칙을 더 접근 가능한 방식으로 구현할 수 있다.

### “AI가 어려워하는 설계”라는 주장은 빠르게 구식이 된다

저자는 모델링과 아키텍처 결정이 AI에게 어렵다고 주장하는데, 이 주장은 현재 AI 능력을 고정값으로 가정한다.
AI 코딩 보조 도구의 발전 속도를 고려하면, “AI가 단일 요청으로 만들기 어려운 설계”라는 기준은 빠르게 변한다.
이 주장이 저자의 설계 원칙의 지속적 가치를 뒷받침하기에는 불안정한 근거다.

## 인사이트

### 개인 도구의 가치는 사용 빈도가 아니라 설계 명료성에서 나온다

`todox` 같은 개인 도구를 직접 만드는 것의 가치는 범용 솔루션보다 나은 기능에 있지 않다.
자신의 실제 필요에 정확하게 맞는 도구를 만드는 과정에서 발생하는 설계 명료성, 그리고 그 도구가 어떻게 작동하는지 완전히 이해하는 상태에서 오는 통제감이 핵심이다.
이 통제감은 상용 도구에서는 얻기 어렵고, AI 생성 코드에서도 마찬가지다.
저자의 1,193줄 C 코드는 기능 완성도의 지표가 아니라 “내가 모든 줄을 이해한다”는 상태의 지표다.

### 언어 선택이 설계 경계를 자연스럽게 강제한다

C 코어와 Perl 래퍼의 분리는 언어의 다름 자체가 경계를 명확하게 만드는 부작용을 활용한다.
같은 언어로 전체를 작성하면 경계를 유지하는 의식적 노력이 필요하지만, 언어가 다르면 프로세스 간 통신을 통해 경계가 물리적으로 강제된다.
이 방식은 아키텍처 결정이 코딩 관습보다 강한 방식으로 유지된다는 점에서, 팀 프로젝트에서 마이크로서비스가 모놀리스보다 서비스 경계를 잘 유지하는 이유와 같은 메커니즘이다.

### D-Bus 직접 통합이 의존성 최소화의 실천이다

복잡한 알림 데몬이나 크론 대신 D-Bus를 직접 사용하는 선택은 표준 인터페이스에 직접 연결한다는 의미다.
이 선택은 추가 의존성 없이 시스템 레이어와 통합하는 Unix 전통을 따른다.
AI 도구들이 복잡한 레이어와 추상화를 자동으로 추가하는 경향이 있는 시대에, 시스템 기본 인터페이스를 직접 사용하는 단순함의 가치는 오히려 높아진다.
