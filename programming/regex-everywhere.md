# “어디서나” 동작하는 정규식

원문: <https://www.johndcook.com/blog/2026/06/23/regex-everywhere/>

HN 토론: <https://news.ycombinator.com/item?id=48672232> (97점, 43개 댓글)

## 요약

John D. Cook이 서로 다른 도구와 환경에서 공통으로 동작하는 정규식 부분집합을 찾으려는 시도를 기록한 글이다.
정규식의 가장 큰 좌절은 구현마다 다르다는 점이다.
한 도구에서 지원하는 기능이 다른 도구에서는 아예 존재하지 않거나,
미묘하게 다른 문법을 요구한다.

저자는 “어디서나”를 세 단계로 구분한다.
가장 엄격한 부분집합은 리터럴, 문자 클래스 `[…]`, 그리고 `.`, `*`, `^`, `$` 특수문자뿐이다.
실용적 범위는 GNU 버전의 sed, awk, grep을 `-E` 플래그와 함께 사용하는 것으로,
이때 awk는 `\b`, `\B` 대신 `\<`와 `\>`로 단어 경계를 표현한다.
저자의 개인적 기준은 자신이 가장 자주 쓰는 sed, awk, grep, Emacs 네 도구다.

Emacs는 이 목록에서 이질적인 존재다.
`+`, `?`, `(`, `)`, `{`, `}`, `|` 앞에 역슬래시를 붙여야 awk와 동일한 동작을 한다.
문자 클래스도 다르다.
Emacs에서 `\s`는 공백을 매칭하지 않고 문자 클래스를 시작하며,
공백 클래스는 `\s-`, 구두점 클래스는 `\S.`처럼 표현한다.

저자가 정리한 이 네 도구에서 공통으로 동작하는 기능은 다음과 같다.
기본 앵커(`^`, `$`)와 임의 문자(`.`), 단어 경계, 문자 클래스,
수량자(`*`, `+`, `?`), 역참조, 교대(`|`), 그룹 캡처, 횟수 지정(`{n,m}`)이다.

## 분석

### “어디서나”는 협상의 결과이지 발견의 결과가 아니다

저자가 “어디서나”를 정의하는 방식은 흥미롭다.
보편적 표준에서 출발해 공통 부분을 찾는 것이 아니라,
자신이 실제로 쓰는 도구 목록을 먼저 정하고 그 교집합을 구한다.
이것은 솔직한 접근이다.
진정한 보편성은 없고, 자신의 워크플로에 최적화된 이식성만 있다는 것을 인정하는 셈이다.

이 접근의 핵심은 “어디서나”가 고정된 개념이 아니라는 점이다.
도구 목록이 바뀌면 “어디서나”의 내용도 달라진다.
Python, JavaScript, Go, Rust를 목록에 추가하면
공통 부분집합은 더 줄어들고,
반대로 Emacs를 빼면 `\+`, `\?` 같은 Emacs 특유의 요구사항이 사라지면서 오히려 늘어난다.
이 유동성을 인식하지 못하면 “어디서나”를 절대적 진술로 오해하게 된다.

### Emacs의 역슬래시 요구가 드러내는 역사적 분기

Emacs 정규식이 `+`, `?`, `|`에 역슬래시를 요구하는 것은 단순한 설계 선택이 아니다.
Emacs는 POSIX 확장 정규식(ERE)이 정립되기 전에 개발됐고,
Lisp 전통에서 괄호가 리터럴로 해석되는 환경을 기반으로 했다.
POSIX ERE가 `(`를 기본값으로 메타 문자로 정의한 반면,
Emacs는 `(`를 기본 리터럴로, `\(`를 그룹으로 설계했다.

이 분기는 1980년대 Unix 생태계가 단일 표준 없이 각자 발전하던 시기의 산물이다.
grep, sed, awk가 AT&T Bell Labs에서 Unix 파이프라인용으로 진화하는 동안,
Emacs는 GNU 프로젝트 맥락에서 텍스트 편집기로 발전했다.
같은 “정규식”이라는 이름을 쓰지만 서로 다른 전통을 따랐다.
이 역사적 분기가 수십 년 후 사용자에게 역슬래시 개수를 두 배로 세어야 하는 짐으로 남았다.

### 최소 공통 분모 전략의 실용적 의미

저자의 접근은 정규식 기능 최대화가 아니라 이식성 최대화다.
이것은 “가장 낮은 공통 표준에 맞춰 작성하라”는 소프트웨어 호환성의 오래된 원칙과 같다.
HTML에서 Internet Explorer를 지원해야 했던 시절,
SQL에서 최소 ANSI 표준에 맞춰 여러 데이터베이스를 지원하는 경우와 구조가 같다.

이 전략의 비용은 표현력이다.
룩어헤드(`(?=...)`)、명명된 캡처 그룹(`(?P<name>...)`)、소유적 수량자(`*+`)는
Python, Perl, PCRE 환경에서 코드를 더 명확하게 만들지만
이 “어디서나” 목록에는 들어가지 않는다.
이식성을 택하면 표현력을 잃는 것이 정규식의 오래된 딜레마다.

## 비평

### “어디서나”의 범위가 저자의 워크플로로 좁혀진 것을 명시하지 않는다

이 글의 가장 큰 문제는 “어디서나”라는 표현이 실제로 “저자의 도구 목록 안에서”를 의미하는데
그 제한이 충분히 명시되지 않는다는 점이다.
sed, awk, grep, Emacs는 Unix 커맨드라인 중심 워크플로에서 핵심 도구이지만,
Python, JavaScript, Ruby, Go를 주로 쓰는 개발자에게 이 목록은 낯설다.

프로그래밍 언어 환경에서 “어디서나”를 정의하면 결과가 달라진다.
Python의 `re` 모듈은 PCRE와 유사하지만 완전히 같지 않고,
JavaScript는 PCRE보다 다른 방향으로 발전했으며,
Go는 RE2를 사용해 룩어헤드 자체를 지원하지 않는다.
Unix 커맨드라인 도구 중심과 프로그래밍 언어 중심의 “어디서나”는 서로 다른 교집합을 가진다.
독자가 자신의 컨텍스트를 먼저 정의하지 않으면 이 글의 결론을 잘못 적용할 수 있다.

JdeBP는 이 글이 돌아가면서 말하지만 끝내 명확히 선언하지 못하는 것이 있다고 지적했다.[^JdeBP]
저자가 묘사하는 “어디서나” 부분집합은 실제로 POSIX BRE(Basic Regular Expressions)와 거의 같다.
이 이름을 쓰지 않는 이유가 있을 수 있지만, 이름 없이 설명하면 독자가 표준 문서로 연결하기 어려워진다.
mrbluecoat는 이 글의 내부 모순도 짚었다.[^mrbluecoat]
“다음 기능들은 어디서나 동작한다”고 선언한 직후 “YMMV(상황에 따라 다를 수 있다)”고 덧붙이는 것은
스스로의 주장을 스스로 약화시키는 셈이다.

또한 이 글은 저자가 “어디서나” 목록에 포함시킨 역참조가 실제로 얼마나 보편적인지를 검증하지 않는다.
ok_dad가 지적했듯[^ok_dad] Go 표준 라이브러리의 `regexp` 패키지는 RE2 엔진을 사용하기 때문에
역참조를 매칭에서 지원하지 않는다.
Go 개발자에게 역참조는 “어디서나” 동작하지 않는다.
이것은 “어디서나”가 언어 생태계를 포함하는 순간 다시 좁아진다는 것을 보여주는 구체적 사례다.

### Emacs를 보편 기준에 포함시키는 것은 특이한 선택이다

Emacs를 “어디서나”의 기준 도구에 포함시킨 것은 전략적으로 의문이다.
Emacs 사용자는 개발자 중에서도 소수다.
Emacs가 요구하는 역슬래시 규칙 때문에 다른 세 도구에서는 자연스러운
`+`, `?`, `|`의 직접 사용이 “어디서나” 기준에서 탈락한다.

실용적으로 Emacs를 제외하면 sed, awk, grep `-E` 조합에서 동작하는 집합은
Emacs 역슬래시 규칙 없이 더 자연스럽게 쓸 수 있다.
Emacs 사용자가 정규식을 작성할 때는 Emacs 내에서만 쓰는 경우와
쉘 명령으로 내보내는 경우를 구분해 다른 문법을 쓰는 것이 현실적이다.
하나의 부분집합으로 두 경우를 동시에 만족시키려는 것이 오히려 비효율적일 수 있다.

rtpg는 Emacs 정규식 이스케이핑이 실제로 얼마나 고통스러운지를 토로했다.[^rtpg]
무엇을 이스케이프해야 하는지 “기본적으로 짐작에 의존”한다는 표현이 현실적이다.
`rx`라는 Lisp 기반 대안 문법이 존재하지만 그것도 편하지 않다고 했다.
이것은 Emacs 정규식을 “어디서나” 집합에 포함시키는 것이 실용적이지 않음을 당사자 관점에서 확인해준다.
Emacs 사용자 스스로도 Emacs 정규식 문법을 자연스럽게 쓰지 않는다면,
이것을 이식성 기준으로 삼는 것은 더욱 의문이다.

### 제약의 이유를 설명하지 않아 학습 가이드로서의 가치가 제한된다

이 글은 무엇이 “어디서나” 동작하는지 목록을 제공하지만
왜 그런지는 설명하지 않는다.
역슬래시가 왜 Emacs에서 필요한지, awk가 왜 `\b` 대신 `\<`를 쓰는지,
GNU `-E` 플래그가 무엇을 활성화하는지가 빠져 있다.

이유 없는 규칙 목록은 암기는 가능하지만 응용이 어렵다.
새로운 도구를 접했을 때 그것이 이 목록에 들어가는지 판단하려면
역사적 맥락과 각 도구의 regex 전통을 이해해야 한다.
“어디서나” 집합에서 빠진 기능이 특정 도구에서 동작하는지를 추론하는 능력은
원리를 알아야 생긴다.
이 글은 그 추론 능력을 키워주지 않는다.

MathMonkeyMan은 이 글이 아예 다루지 않은 차원을 지적했다.[^MathMonkeyMan]
어떤 정규식 문법을 쓰는가뿐만 아니라 매칭 범위를 어떻게 정의하느냐가
동작의 일관성에 결정적 영향을 미친다.
부분열 매칭(substring), 접두어 매칭(prefix), 전체 문자열 매칭(whole string), 라인 매칭은
같은 정규식 문법을 쓰더라도 도구마다 기본값이 다르다.
grep은 기본적으로 라인 안의 부분열을 찾지만,
어떤 언어의 regex 함수는 전체 문자열을 매칭한다.
이 차원이 빠지면 “어디서나 동작하는” 정규식 목록은 절반의 정보만 전달하는 셈이다.

## 인사이트

### 최소 공통 분모 학습은 역방향 접근이다

정규식을 배울 때 “어디서나 동작하는 것부터” 시작하는 것은 직관적으로 보이지만
실제로는 학습을 왜곡한다.
최소 집합은 기능이 제한돼 표현력이 낮고,
이 수준의 정규식으로는 실제 문제를 풀기 어려운 경우가 많다.
제한된 도구로 어렵게 풀다가 정규식 자체를 어렵다고 느끼게 될 위험이 있다.

더 효과적인 경로는 PCRE 또는 Python `re`처럼 풍부한 기능을 가진 환경에서 시작해
정규식의 개념을 완전히 익히는 것이다.
그 위에서 특정 환경에 배포할 때 제약을 적용하면 된다.
무엇을 잃는지 알면서 제약을 택하는 것과,
처음부터 제약된 것만 알아서 무엇을 잃는지 모르는 것은 완전히 다른 역량이다.
저자의 “어디서나” 목록은 이미 PCRE를 아는 사람에게는 유용한 가이드지만,
처음 배우는 사람에게는 출발점이 될 수 없다.

### 정규식 단편화는 표준화 실패의 구조적 증거다

HTML은 W3C와 WHATWG를 통해 표준화됐고,
SQL은 ANSI/ISO 표준이 있으며,
CSS도 레벨별 표준이 존재한다.
정규식은 POSIX가 BRE(Basic Regular Expressions)와 ERE(Extended Regular Expressions)를 정의했지만,
Perl이 PCRE를 만들고, 각 언어가 독자적으로 발전하면서 POSIX는 사실상 최소 기준선이 됐다.

이 단편화가 지속되는 이유는 정규식이 “충분히 잘 동작하는” 도구로 오래 살아남았기 때문이다.
Unix 철학의 “작고 한 가지 일을 잘 하는 도구”들이 각자의 정규식 구현을 가지면서
표준화보다 개별 최적화를 선택했다.
Emacs는 Lisp 편집기로서의 정규식, grep은 라인 필터로서의 정규식, awk는 필드 처리용 정규식을 따로 발전시켰다.
이 다양성은 각 도구의 사용 맥락에 최적화된 결과지만,
도구를 넘나드는 사용자에게는 영속적인 인지 부담을 남긴다.

이 문제를 공식적으로 해결하려는 시도가 이미 있었다.
tonyg는 RFC 9485 “I-Regexp: An Interoperable Regular Expression Format”을 소개했다.[^tonyg]
IETF가 2023년 발표한 이 표준은 서로 다른 시스템 간에 상호 운용 가능한 최소 정규식 형식을 정의한다.
저자의 글은 개인 경험에서 귀납적으로 같은 문제를 다루지만,
공식 표준의 존재를 언급하지 않는다.
비공식 경험 공유와 공식 표준화 작업이 독립적으로 같은 결론에 수렴하고 있다는 사실이
이 문제의 심각성과 해결 필요성을 동시에 보여준다.

### AI 코드 생성 시대에 “어디서나” 지식의 위치가 바뀐다

LLM이 정규식을 생성해주는 도구로 자리 잡으면서
이 지식의 실용적 위치가 달라지고 있다.
“Emacs에서 `+` 앞에 역슬래시가 필요하다”는 사실을 암기하는 것과,
“이 환경에서 생성된 정규식이 올바른지 검증하는 능력”은 다른 종류의 지식이다.
전자는 AI가 대신해줄 수 있지만 후자는 그렇지 않다.

이 전환에서 저자의 글이 가진 가치도 변한다.
구체적 문법 목록은 참조 가치가 줄어들지만,
“어디서나”의 기준을 자신의 컨텍스트에 맞게 정의하는 사고방식은 여전히 유효하다.
AI가 생성한 정규식을 받았을 때 “이게 내 환경에서 동작하는가”를 판단하려면
각 도구의 regex 전통과 호환성 지형을 이해해야 한다.
암기 지식보다 판단 능력이 중요해지는 방향으로 이 분야의 필요 역량이 이동하고 있다.
