# Zig Programming Language

<https://ziglang.org/>

한국어: <https://ziglang.org/ko-KR/>

코드 저장소: <https://codeberg.org/ziglang/zig>

## 소개

Zig는 저수준 제어를 중시하는 시스템 프로그래밍 언어다.
메모리 관리, 에러 처리, 빌드, C 연동을 언어와 도구 체계 안에서 일관되게 다루려는 점이 특징이다.

이 언어는 “숨겨진 동작을 줄이고, 컴파일 시점에 더 많은 선택을 드러내자”는 방향으로 설계됐다.
자동 마법을 많이 넣기보다, 호출 지점과 실패 경로, 할당 지점을 개발자가 명시적으로 보게 만든다.

## 핵심 특징

- 명시적 메모리 할당과 해제 흐름을 선호한다.
- 실패 가능성을 값으로 다루는 에러 처리 방식을 제공한다.
- `comptime`으로 컴파일 시점 계산과 제네릭 생성을 적극적으로 활용한다.
- C 코드와의 상호운용성을 중요한 목표로 둔다.
- 언어 기능과 빌드 시스템을 함께 설계해, 작은 프로젝트부터 대형 프로젝트까지 같은 도구로 다루려 한다.

## 읽을 때 볼 점

Zig를 볼 때는 문법보다 설계 철학을 함께 보는 편이 좋다.
이 언어는 편의성만으로 평가하기 어렵고, 제어 가능성, 이식성, 예측 가능성을 어디까지 확보하느냐가 핵심이기 때문이다.

따라서 다음 질문으로 읽으면 도움이 된다.

- 숨겨진 비용이 얼마나 적은가?
- C와 함께 쓸 때 경계가 얼마나 단순한가?
- 컴파일 타임 계산이 실제 생산성으로 이어지는가?
- 생태계의 미성숙이 언어 장점을 얼마나 상쇄하는가?

## 비평

Zig는 설계 목표가 선명하지만, 생태계 성숙도와 도구 안정성이 항상 그 목표를 따라오지는 않는다.
특히 실무에서는 언어 자체의 우아함보다 패키지 관리, 버전 호환성, 문서 품질이 더 큰 비용이 되기도 한다.

또한 명시성을 강조하는 철학은 강점이지만, 모든 개발자에게 같은 수준의 생산성을 주지는 않는다.
저수준 제어를 얻는 대신 더 많은 결정을 직접 내려야 하므로, 팀과 문제 도메인에 따라 체감이 크게 갈린다.

## 인사이트

Zig는 “빠르게 만들기 쉬운 언어”라기보다 “무엇이 일어나는지 분명히 보이는 언어”에 가깝다.
그 점이 맞는 문제에서는 강력하지만, 그 전제를 받아들일 수 없는 상황에서는 부담이 된다.
Zig를 이해하려면 문법보다 먼저, 어떤 비용을 개발자에게 되돌려 주는 언어인지 보는 편이 낫다.
