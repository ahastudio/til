# How Our Rust-to-Zig Rewrite is Going

원문: <https://rtfeldman.com/rust-to-zig>

HN 토론: <https://news.ycombinator.com/item?id=48933149> (524점, 304개 댓글)

Lobste.rs 토론: <https://lobste.rs/s/axdfjx/how_our_rust_to_zig_rewrite_is_going>

## 요약

Richard Feldman이 2026년 7월 15일에 공개한 이 글은 Roc 프로그래밍 언어 컴파일러를
30만 줄의 Rust 코드에서 Zig로 전면 재작성한 487일간의 여정을 다룬다.
핵심 이정표는 기능 동등성 달성이다.
WASM-4 기반 Rocci Bird 게임이 새 컴파일러에서 정상 작동하며, 출력 바이너리 크기도
60KB 이상에서 31KB로 절반 이하로 줄었다.

재작성의 출발점은 기술적 막다른 길이었다.
기존 Rust 구현에서 클로저 캡처를 힙 할당 없이 처리하는 람다 집합 해소(lambda set
resolution) 시스템의 아키텍처 문제를 점진적으로 고치는 것이 사실상 불가능했다.
핵심 기여자들이 “어차피 컴파일러 대부분을 결국 다시 써야 한다”는 판단에 도달했고,
처음부터 다시 쓰는 쪽이 전략적으로 합리적이라고 결정했다.

성능 면에서 결과는 극적이다.
Rust 기준 10초이던 증분 빌드(incremental build)가 Zig 0.17.0에서 35밀리초로 줄었다.
팀은 프리릴리스 버전 채택을 피해 현재 안정 버전인 0.16.0을 사용 중이므로 이 이점은
아직 적용되지 않은 상태다.

메모리 안전성 면에서는 예상과 다른 결과가 나왔다.
Rust 컴파일러 버그 2,596건 대 Zig 컴파일러 버그 431건을 분석한 결과, Zig 버전에서
use-after-free가 두 건 발생했으나 오류 보고 경로에 한정됐다.
대부분의 메모리 손상 보고는 컴파일러 구현 결함이 아닌 잘못된 코드 생성(miscompilation)
문제였고, 두 언어 모두 실용적 안전성 수준에서는 충분했다는 결론이다.

Rust에서 아쉬운 점으로는 테스트 코드의 자동 할당/해제 처리, 매개변수 다형성과
임시(ad hoc) 다형성, 구조체 비공개 필드 강제, 포괄적 데드코드 탐지가 꼽혔다.
반면 Zig에서 재발견한 장점은 뺄셈형 언어 설계 철학, 직접적인 데이터 레이아웃 제어,
매크로 없는 개발, 자연스러운 할당 실패 처리, 뛰어난 빌드 시스템이다.

기술적 혁신 중 하나는 제로 파싱 역직렬화(zero-parse deserialization)다.
포인터 대신 인덱스 기반 배열 접근으로 캐시 데이터를 구조화해, 디스크에서 메모리로
직접 로드할 때 파싱 없이 memcpy 속도로 읽는다.
Roc 0.1.0은 2026년 중 출시를 목표로 하고 있으며, 현재 나이틀리 빌드에서는 안정성과
문서 완성도의 편차가 크다.

## 분석

### 전략적 재작성을 가능하게 한 아키텍처 조건

Roc 팀의 결정이 흥미로운 이유는 재작성이 실패에서 비롯된 것이 아니라 비용 계산의
결과였다는 점이다.
람다 집합 해소 문제는 해결 불가능한 것이 아니었다.
점진적으로 패치할 수 있었지만, 그 비용이 처음부터 다시 쓰는 것보다 크다고 판단했다.

이 계산이 성립하려면 몇 가지 전제가 필요하다.
먼저 팀이 Zig를 충분히 파악하고 있어야 한다.
다음으로 재작성 기간 동안 기존 컴파일러가 계속 작동해야 한다.
그리고 외부 사용자 기반이 아직 작아서 호환성 부담이 낮아야 한다.
Roc은 이 세 조건을 모두 충족했다.

주목할 점은 재작성 결정에서 “언어 선호”가 아닌 “도구 적합성”이 기준이 됐다는 것이다.
팀이 Zig를 선택한 이유로 꼽은 것들은 빠른 빌드 타임, 세밀한 메모리 할당자 제어,
비표준 비트 너비 정수 지원, 컴파일러 구현에 맞는 생태계였다.
임베디드 시스템이나 게임처럼 직접적인 제어가 중요한 도메인이 아니라 컴파일러 자체를
만드는 데 Zig의 특성이 맞았다는 것이다.

HN에서 arthurbrown은 “Rust 자체가 원래 OCaml로 작성됐고, WASM도 마찬가지인데 왜
OCaml은 검토하지 않았는가”라는 질문을 던졌다.[^arthurbrown]
컴파일러 도메인에서 메모리 제어가 실제로 그만큼 중요한지, 성숙한 생태계를 가진
함수형 언어가 더 적합하지 않은지를 묻는 반론이다.
저자가 “왜 Zig인가” 절에서 크로스 컴파일을 주요 이유로 언급하지 않은 것도 이 의문을
강화한다.

### 빌드 속도가 개발 문화에 미치는 영향

35밀리초 증분 빌드라는 수치는 단순한 성능 개선이 아니다.
빌드 피드백 루프가 초 단위에서 35밀리초로 바뀌면 개발 흐름 자체가 달라진다.
3.4초 빌드에서는 “저장 후 결과 확인”이 의식적 행동이다.
35밀리초에서는 그 경계가 사라진다.

이것은 컴파일러 구현에서 특히 중요하다.
컴파일러 코드의 변경 효과는 테스트 케이스를 돌려봐야 알 수 있고, 파이프라인을 따라가며
중간 표현을 검사해야 한다.
빌드가 느리면 이 탐색 과정이 느려진다.
팀이 Zig 빌드 속도를 선택의 핵심 이유로 꼽은 것은 철학이 아니라 일상 생산성의 문제였다.

Lobste.rs 토론에서 zmitchell은 이 차이를 구체적으로 설명했다.[^zmitchell]
35밀리초 빌드에서 `--watch` 모드를 쓰면 파일 저장과 동시에 컴파일이 완료된다.
오류 메시지를 에디터로 전환하기 전에 이미 결과가 준비돼 있다는 것이다.
Arya는 이 패턴의 핵심을 "다음 키 입력 전에 오류 피드백이 도착하는 것"이라 표현했다.[^arya]
LSP가 타이핑 중 오류를 보여주지만, 저장 후 다음 편집까지의 그 틈을 채우는 것이 다르다는 주장이다.
반면 Lobste.rs의 dlisboa는 "3.4초도 충분하다, 서브초 컴파일이 정말 필요한가"라는
반론을 제기했고[^dlisboa], Zig 컴파일러 기여자인 mlugg가 직접 자신의 경험으로 답했다.[^mlugg]
수십 밀리초 빌드가 실제 작업 방식 자체를 바꿨다고 했다.

기술적으로 Zig가 이 속도를 달성하는 이유도 mlugg가 설명했다.
Zig의 증분 링커는 함수 단위 연산과 mmap을 활용해 링크 시간만 약 1밀리초다.
컴파일러와 링커가 긴밀하게 결합된 설계가 이를 가능하게 한다.

흥미로운 것은 이 이점이 언어 설계의 결과가 아닌 컴파일러 구현의 결과라는 점이다.
Zig 언어 자체의 특성이 아니라 Zig 컴파일러가 Rust 컴파일러보다 빠르다는 사실이 이점이다.
언어 선택이 컴파일러 구현 품질까지 포함한다는 것을 보여주는 사례다.

### 버그 수 비교의 함정과 진짜 메시지

Rust 2,596건 대 Zig 431건이라는 버그 수 비교는 표면적으로 Rust가 더 버그가 많다는
인상을 줄 수 있지만, 맥락이 필요하다.
Rust 컴파일러는 훨씬 오래됐고, 더 크고, 더 많이 사용되므로 노출된 코드 경로가 많다.
절대 수치 비교는 안전성 수준의 직접 비교가 아니다.

dminik은 같은 맥락에서 "7년간 잘 테스트된 Rust 구현과 아직 출시되지 않은 1년짜리
Zig 구현을 비교하는 것은 다소 이상하다"고 지적했다.[^dminik]
성숙도가 다른 두 구현의 버그 수를 나란히 놓는 것은 언어 안전성보다 코드 나이를
반영할 가능성이 높다.

Lobste.rs에서 Rust 공식 기여자인 ralfj는 훨씬 더 날카로운 지적을 했다.[^ralfj]
글에서 인용된 "Rust 컴파일러의 unsafe 40,000개"라는 수치가 실제로는 표준 라이브러리를
포함한 것으로, rustc 컴파일러 자체의 unsafe 블록은 2,000개 미만이라는 것이다.
이것은 단순한 맥락 차이가 아니라 비교 기준 자체의 오류다.
Zig의 `unsafe` 없는 설계와 Rust의 pervasive unsafe를 대비하는 글의 핵심 주장이
흔들릴 수 있는 사실 오류다.

글이 실제로 말하는 것은 다르다.
두 언어 모두 컴파일러 구현에서 실용적 안전성 수준에 도달할 수 있다는 것이다.
Zig에서 발생한 두 건의 use-after-free가 오류 보고 경로에만 있었다는 사실은,
일상적 실행 경로에서는 발견되지 않았다는 의미다.
Rust의 빌림 검사기가 없었지만 신중한 설계와 리뷰로 보완할 수 있었다는 시사점이다.

그러나 landr0id는 글이 Zig의 `ReleaseSafe` 모드가 use-after-free를 탐지한다고 암시하는
부분에 의문을 제기했다.[^landr0id]
Zig 공식 문서에서 "use-after-free" 관련 보장을 찾을 수 없으며, `DebugAllocator`가
관련 있을 수 있지만 그조차 신뢰할 만한 탐지 메커니즘인지 불분명하다는 것이다.
만약 이 지적이 정확하다면, "ReleaseFast에서 Rust와 비슷한 안전성" 주장은 과장일 수 있다.

마찬가지로 steveklabnik은 "컴파일러가 머신 코드를 내보내는 데 unsafe가 필요하다"는
표현을 문제 삼았다.[^steveklabnik]
표준 실행 파일 생성에는 unsafe가 필요하지 않으며, hot binary patching이나 코드 리로딩
같은 특수 기능에만 해당된다는 것이다.
이 두 반론은 메모리 안전성 비교 결론이 생각보다 단순하지 않음을 보여준다.

### 비공개 필드 부재가 드러내는 설계 철학 차이

Rust에서 아쉬운 점으로 꼽힌 “비공개 구조체 필드 강제”는 작은 기능처럼 보이지만,
언어 설계 철학의 핵심 차이를 드러낸다.
Rust는 모듈 수준에서 가시성을 강제한다.
`pub` 없으면 외부에서 접근 불가다.

Zig에는 이 메커니즘이 없다.
컴파일러 규모의 코드베이스에서 내부 표현 은닉이 중요한데, Zig는 이를 언어 수준에서
강제하지 않는다.
“뺄셈형 언어 설계”라는 Zig의 철학이 때로는 팀 코드베이스 경계 관리에서 추가 비용으로
나타난다는 구체적 사례다.

## 비평

### 성공 조건이 일반화되지 않는다

487일, 30만 줄, 기능 동등성 달성이라는 결과는 인상적이다.
하지만 이 성공의 전제 조건을 숨기면 재작성 일반론으로 오독될 위험이 있다.

Roc 팀의 재작성이 가능했던 조건은 매우 특수하다.
오픈 소스 프로젝트로 상업적 납기 압력이 낮다.
유료 사용자 기반이 아직 없어 호환성 부담이 없다.
Roc Foundation이라는 비영리 조직이 기여자를 지원한다.
그리고 팀에 언어를 처음부터 설계한 원 저자가 참여한다.

대부분의 상업적 소프트웨어 팀에 이 조건이 없다.
“Roc이 했으니 우리도 할 수 있다”는 추론은 조건의 차이를 간과한다.
글은 이 맥락을 명시적으로 다루지 않는다.

### 아직 끝나지 않은 이야기

글이 현재 시점의 성과를 강조하지만, 0.1.0 출시 전이라는 점이 중요하다.
기능 동등성 달성은 중요한 이정표지만, 새 컴파일러가 프로덕션 언어로 검증되지는 않았다.

Zig 버전에서 아직 드러나지 않은 문제들이 있을 수 있다.
0.16.0을 쓰면서 0.17.0의 성능 이점을 미루고 있다는 것은 안정성을 우선시한 선택이지만,
동시에 도달하지 못한 목표치가 있다는 의미이기도 하다.
487일의 경험이 아직 완성된 이야기가 아니라는 것이다.

### 비용 계산의 불완전한 공개

“어차피 다시 써야 했다”는 판단이 사후 합리화일 가능성을 글이 충분히 검토하지 않는다.
재작성 전에 기존 코드베이스에서 람다 집합 해소 외의 다른 경로를 얼마나 검토했는지,
점진적 마이그레이션이 왜 불가능했는지에 대한 구체적인 근거가 부족하다.

실패한 소프트웨어 재작성 사례들을 보면 “어차피 다시 써야 한다”는 판단이 기술적 분석보다
새 기술을 써보고 싶다는 팀의 동기를 합리화한 경우가 있다.
Roc의 경우 결과가 긍정적이지만, 그것이 결정 과정의 합리성을 소급해서 증명하지는 않는다.

norir는 더 근본적인 의문을 제기했다.[^norir]
팀이 고성능 컴파일러에 저수준 시스템 언어가 필요하다는 것을 실증적으로 확인하지 않았다는
것이다.
매니지드 언어도 같은 알고리즘을 쓰면 2배 이내로 성능을 낼 수 있고, “컴파일러 성능은
알고리즘이 지배한다”는 것이 그의 주장이다.
현재 Roc의 초당 15,000 줄 컴파일 속도는 1998년의 ML 컴파일러와 비교해도 느리다는 지적도
덧붙였다.
재작성 전에 소규모 자기 호스팅 컴파일러를 먼저 작성해 성능 가정을 검증했더라면, 더 나은
아키텍처 판단이 가능했을 것이라는 대안이다.

## 인사이트

### 언어 전환 비용이 0이 되는 조건

소프트웨어 역사에서 언어 재작성은 거의 항상 실패했다.
Joel Spolsky의 “Things You Should Never Do”가 규범이 됐고, Netscape Navigator의 재작성
실패가 대표 사례로 인용된다.
Roc의 성공이 이 통설에 반례가 되는 것처럼 보이지만, 실제로는 다른 이야기다.

전통적인 재작성 실패의 핵심 이유는 두 가지다.
첫째, 기존 코드베이스가 명시적 요구사항 문서보다 많은 것을 담고 있다(암묵적 지식).
둘째, 재작성 기간 동안 원본이 계속 진화한다(이동하는 목표).

Roc의 경우 첫 번째 문제는 원 설계자가 팀에 있어 암묵적 지식이 인간에 있었고,
두 번째 문제는 재작성과 병행해 원본 개발을 의도적으로 동결했을 가능성이 있다.
재작성 성공의 조건은 “언어보다 사람과 프로세스”다.

이것이 시사하는 것은 전략적 재작성이 가능한 조건이 기술의 문제가 아니라는 것이다.
도메인 지식이 팀에 내재화돼 있는지, 요구사항이 안정적인지, 외부 의존이 낮은지가
언어 선택보다 중요하다.

### 컴파일러는 언어의 일부다: 에코시스템 선택으로서의 언어 선택

팀이 Zig를 선택한 이유 중 빌드 속도가 핵심이었다.
그런데 이것은 Zig 언어의 특성이 아니라 Zig 컴파일러 구현의 특성이다.
“Rust vs Zig” 비교가 종종 언어 문법이나 타입 시스템에만 집중하지만,
실무에서는 컴파일러 속도, 에러 메시지 품질, 빌드 도구, 언어 서버 프로토콜(LSP) 지원,
생태계 라이브러리가 동등하게 중요하다.

Roc 팀이 선택한 것은 “Zig 언어”가 아니라 “Zig 에코시스템 전체”였다.
특히 컴파일러 구현에서 빠른 피드백 루프가 중요한 도메인에서는 빌드 속도가 언어 특성보다
더 큰 영향을 줄 수 있다.

이것은 언어 선택 논의의 맹점을 드러낸다.
벤치마크와 기능 비교 중심의 언어 선택 가이드들이 개발 사이클 속도, 도구 생태계,
에러 메시지 품질 같은 일상 경험 지표를 과소평가한다.
Roc의 경험은 이 지표들이 다년간 프로젝트에서 팀 생산성에 더 큰 영향을 준다는
구체적 증거를 제공한다.

### 오픈 소스 컴파일러 작업의 경제학

Roc Foundation이 기여자를 지원하는 비영리 구조와 487일 재작성을 연결하면 흥미로운
패턴이 보인다.
상업적 언어 프로젝트에서 이런 규모의 재작성은 이사회나 투자자의 승인이 필요하고,
ROI를 보여줘야 한다.

오픈 소스 비영리 구조는 이 제약이 다르다.
기여자가 내재적 동기로 참여하고, 장기 기술 부채 해소가 단기 지표보다 우선될 수 있다.
Zig 생태계 자체도 Zig Software Foundation이 비슷한 구조로 운영된다.

이 구조의 장점은 기술적 올바름을 상업적 압력보다 우선할 수 있다는 것이다.
하지만 지속 가능성이 과제다.
GitHub 스폰서와 조직 후원에 의존하는 구조는 핵심 기여자 번아웃이나 자금 감소에 취약하다.
Roc 0.1.0 이후 언어가 더 넓은 사용자 기반을 확보하기 전까지 이 취약성이 남는다.
487일의 성공이 앞으로도 이어지려면 기술적 성취만큼 커뮤니티와 재정 지속 가능성이
중요하다.
