# GitHub Diff 라인 성능 최적화의 험난한 여정

원문: <https://github.blog/engineering/architecture-optimization/the-uphill-climb-of-making-diff-lines-performant/>

## 요약

GitHub 엔지니어 Luke Ghenco와 Adam Shwert는 풀 리퀘스트의 “Files changed” 탭 성능을 개선한 과정을 공개했다.
극단적으로 큰 PR에서 JavaScript 힙이 1 GB를 초과하고, DOM 노드 수가 400,000개를 넘으며, INP(Interaction to Next Paint) 점수가 허용 범위를 벗어나는 심각한 성능 저하가 발생했다는 것이 출발점이었다.

기존 V1 아키텍처는 작은 재사용 가능한 React 컴포넌트 조합에 의존했다.
단일 diff 라인(unified view 기준)에 DOM 엘리먼트 약 10개, React 컴포넌트 8~13개, 이벤트 핸들러 20개 이상이 배치되어 있었다.
split view에서는 DOM 엘리먼트가 라인당 약 15개로 늘어났다.
수천 개의 diff 라인이 렌더링될 때 이 구조는 복합적으로 성능을 악화시켰다.

V2 최적화는 세 가지 축으로 진행되었다.
첫째, find-in-page 같은 네이티브 기능을 유지하면서 주요 diff 경험을 집중 최적화했다.
둘째, 매우 큰 PR에서는 가상화(virtualization)를 통해 반응성을 우선하는 점진적 성능 저하(graceful degradation) 전략을 채택했다.
셋째, 모든 규모의 PR에 혜택을 주는 기반 개선 작업을 수행했다.
핵심 변경 사항으로는 라인 번호 셀의 불필요한 `<code>` 태그 제거, 컴포넌트 수를 라인당 8개에서 2개로 축소, 라인별 이벤트 핸들러를 data attribute를 활용한 단일 최상위 이벤트 핸들러로 교체, O(n) 조회를 JavaScript Map을 활용한 O(1) 조회로 전환이 포함된다.

10,000 라인 PR(split view)을 기준으로 한 측정 결과는 다음과 같다.
렌더링되는 총 컴포넌트 수가 183,504개에서 50,004개로 73% 감소했고, 메모리 사용량은 150~250 MB에서 80~120 MB로 약 50% 줄었으며, INP는 M1 MacBook 4x 슬로우다운 환경에서 450 ms에서 100 ms로 78% 개선되었다.

p95 이상의 초대형 PR(10,000 diff 라인 초과)에 대해서는 TanStack Virtual을 통합해 화면에 보이는 콘텐츠만 렌더링하도록 가상화를 구현했다.
그 결과 JavaScript 힙 사용량이 10배 감소하고, INP가 275~700+ ms에서 40~80 ms로 개선되었다.
추가로 무거운 CSS 선택자를 GPU 가속 transform으로 교체하고, Datadog을 통한 인터랙션 수준의 INP 추적, 서버 사이드 렌더링에서 가시 영역만 하이드레이션하는 최적화, 스마트 백그라운드 페치를 통한 점진적 diff 로딩도 적용했다.

## 분석

### 성능 문제는 컴포넌트 설계 원칙의 역설에서 비롯된다

V1 아키텍처의 문제는 나쁜 설계에서 온 것이 아니었다.
오히려 React 생태계가 권장하는 방식 — 작은 재사용 가능한 컴포넌트, 관심사 분리, 풍부한 이벤트 처리 — 을 충실히 따른 결과였다.
라인당 8~13개의 컴포넌트와 20개 이상의 이벤트 핸들러는 각각이 단독으로는 합리적인 설계였지만, 이것이 수천 개의 라인에 걸쳐 곱해지면서 문제가 발생했다.

이 역설은 소프트웨어 엔지니어링에서 자주 나타나는 “좋은 지역적 결정이 나쁜 전역적 결과를 만드는” 패턴이다.
모듈화와 재사용성이라는 미시적 최적화가 전체 시스템 규모에서 거시적 비효율로 전환된다.
GitHub의 사례는 이 전환점이 diff 라인의 경우 수천 개 단위에서 발생한다는 것을 구체적인 수치로 보여준다.

V2의 핵심 통찰은 “추상화 비용”이었다.
컴포넌트 하나가 두 개의 자식 컴포넌트를 갖는 것은 무해해 보이지만, 이 구조가 183,504번 반복될 때 그 오버헤드는 측정 가능한 수준이 된다.
팀이 도달한 결론 — “단순화: 더 적은 상태, 더 적은 엘리먼트, 더 적은 JavaScript, 더 적은 React 컴포넌트” — 은 프레임워크의 편의성보다 실행 비용을 우선했을 때 얻을 수 있는 이득을 명시적으로 드러낸다.

### 단일 이벤트 핸들러 패턴은 웹의 오래된 지혜를 재발견한 것이다

V2에서 라인별 이벤트 핸들러를 제거하고 data attribute를 활용한 단일 최상위 이벤트 핸들러로 교체한 것은, 이벤트 위임(event delegation) 패턴의 현대적 재적용이다.
이 패턴은 jQuery 시대에 이미 성능 최적화의 정석으로 자리 잡았던 방식이다.
React가 보편화되면서 프레임워크가 이벤트 관리를 추상화해 주었고, 개발자들은 per-element 이벤트 바인딩의 비용을 직접 고민할 필요가 없어졌다.
그러나 충분한 규모에서는 그 비용이 다시 수면 위로 올라온다.

이 관찰은 기술 스택이 특정 문제들을 “해결된 것처럼” 보이게 만들지만, 그 문제들이 사라진 것이 아니라 추상화 계층 아래로 숨어든 것임을 시사한다.
React의 합성 이벤트 시스템은 실제로 루트 수준에서 이벤트 위임을 사용하지만, 사용자 코드 수준에서 생성되는 수만 개의 핸들러 참조는 여전히 메모리와 가비지 컬렉터에 부담을 준다.
프레임워크가 하위 수준 패턴을 추상화할 때, 개발자는 그 패턴의 존재 이유도 함께 잊어버리는 경향이 있다.

### 가상화는 해결책이 아니라 수용(accommodation)이다

TanStack Virtual을 통한 가상화 구현이 p95 이상 PR에서 극적인 성능 개선(힙 사용량 10배 감소, INP 275~700+ ms에서 40~80 ms로)을 가져왔지만, 이것은 근본적인 한계를 숨기는 방식이기도 하다.
가상화는 “10,000개의 diff 라인을 효율적으로 렌더링한다”는 문제를 “사용자가 보는 몇십 개의 라인만 렌더링한다”는 다른 문제로 대체한다.

이 전략이 갖는 트레이드오프는 기사에서 briefly 언급되지만 충분히 탐구되지 않는다.
가상화는 find-in-page, Ctrl+F 검색, 텍스트 선택 같은 브라우저 네이티브 기능과 근본적으로 충돌한다.
팀이 “네이티브 기능을 유지하면서 집중 최적화”를 원칙 중 하나로 명시한 것은 이 긴장을 인지했음을 보여준다.
그러나 가상화된 리스트에서 find-in-page가 어떻게 작동하는지, 또는 작동하지 않는지에 대한 구체적 설명은 없다.

가상화가 “p95 이상” PR에만 적용된다는 사실은 이 긴장을 관리하는 방식이 임계값 기반 분기라는 것을 암시한다.
그러나 10,000 라인이라는 임계값은 어떻게 결정되었는가.
그 임계값 근처의 사용자 경험은 가상화와 비가상화 사이에서 일관성이 있는가.
이런 질문들이 기사에서 다루어졌다면 실제 구현의 복잡성을 더 정확하게 전달할 수 있었을 것이다.

### 측정 체계의 구축이 최적화만큼 중요한 작업이었다

기사에서 주목할 점 중 하나는 Datadog을 통한 인터랙션 수준 INP 추적 구현이다.
이것은 단순한 모니터링 추가가 아니라, 이전에는 존재하지 않던 측정 가시성을 만든 작업이다.
“인터랙션 수준”이라는 수식어가 중요하다 — 페이지 전체의 INP가 아니라, 어떤 특정 인터랙션이 성능 저하의 원인인지를 추적한다는 뜻이다.

이 측정 체계 없이는 최적화의 효과를 검증하거나, 어떤 변경이 어떤 개선을 가져왔는지 인과관계를 파악하기 어렵다.
성능 최적화 작업의 어려움 중 하나는 관측 가능성(observability)의 부재다.
무엇을 측정하지 않으면 무엇을 개선했는지도 모른다.
팀이 최적화 코드와 병행해 측정 인프라를 구축한 것은, 이 작업이 일회성 최적화가 아니라 지속적인 성능 관리 체계를 만드는 프로젝트였음을 보여준다.

## 비평

### 기사는 결과를 서술하지만 트레이드오프의 비용을 숨긴다

74% 컴포넌트 감소, 78% INP 개선이라는 수치는 인상적이다.
그러나 기사는 이 개선이 무엇을 포기함으로써 달성되었는지에 대해 침묵에 가깝다.
코드 줄 수가 2,800에서 2,000으로 27% 줄었다는 것은 단순화의 증거로 제시되지만, 실제로는 더 낮은 수준의 추상화로 이동했다는 것이기도 하다.

라인당 8~13개의 컴포넌트가 2개로 줄었다는 것은 각 컴포넌트가 더 많은 책임을 져야 한다는 의미다.
또는 더 정확히는, 이전에 컴포넌트 분리를 통해 관리되던 복잡성이 다른 방식으로 존재하게 된다는 뜻이다.
이 “단순화”가 유지보수성, 테스트 용이성, 기능 추가 비용에 어떤 영향을 미쳤는지는 기사에서 전혀 다루지 않는다.
성능이 개선되었다는 것과 코드베이스가 더 좋아졌다는 것은 같은 말이 아니다.

data attribute 기반 단일 이벤트 핸들러 패턴도 마찬가지다.
이것은 이벤트 핸들링 로직이 더 중앙화되고, 어떤 엘리먼트가 어떤 동작을 하는지를 컴포넌트 정의가 아닌 DOM 구조에서 추적해야 한다는 것을 의미한다.
이 패턴이 새 기능 추가 시 어떤 인지 부하를 발생시키는지, 버그 디버깅을 어떻게 변화시키는지에 대한 논의가 없다.
기사는 성능 수치의 인상적인 부분만 보여주고, 그 수치를 달성하기 위해 지불한 설계 부채는 보여주지 않는다.

### 수치가 대표하는 시나리오가 실제 사용 분포와 얼마나 겹치는지 불명확하다

10,000 라인 PR을 기준으로 한 측정이 기사의 핵심 비교 데이터다.
그러나 GitHub의 실제 PR 분포에서 10,000 라인 PR이 차지하는 비중은 얼마인가.
기사는 p95 이상이 10,000 라인을 초과한다고 언급하지만, 이것은 5% 미만의 PR이 그 임계값 위에 있다는 뜻이다.

평균적인 PR — 아마도 수백 라인 수준 — 에서 V1과 V2의 차이는 어떤가.
INP 78% 개선이 10,000 라인 PR에서 달성되었다면, 100 라인 PR에서는 사용자가 실제로 차이를 느끼는가.
이 질문은 중요하다.
왜냐하면 최적화 작업의 정당성은 전체 사용자 경험에 미치는 영향의 크기에 달려 있기 때문이다.
기사는 극단적 케이스에서의 극적인 개선을 강조하지만, 그 개선이 실제 사용자 경험의 얼마만큼을 대표하는지는 불명확하다.

이 문제는 성능 최적화 기사에서 반복적으로 나타나는 패턴이다.
가장 극적인 숫자를 만들어 내는 시나리오를 벤치마크로 선택하고, 그 숫자를 전체 성능 개선의 대표치로 제시하는 방식이다.
10X 힙 감소와 78% INP 개선은 모두 사실이지만, 이것이 중간값 사용자의 경험을 대표하지 않을 수 있다는 점은 독자가 스스로 판단해야 한다.

### 서버 사이드 최적화의 역할이 불균형하게 축소되어 있다

기사는 “서버 사이드 렌더링에서 가시 영역만 하이드레이션”하는 최적화를 언급하지만, 이것은 마지막 목록 항목 중 하나로 간략하게 처리된다.
그러나 이것은 클라이언트 사이드 최적화와 완전히 다른 성격의 변경이다.

하이드레이션 전략 변경은 초기 로드 성능, Time to Interactive, 점진적 향상(progressive enhancement)과 직결되는 아키텍처 결정이다.
가시 영역만 하이드레이션한다는 것은 스크롤 아래에 있는 콘텐츠가 초기에는 인터랙티브하지 않다는 뜻이며, 이것은 사용자 경험에 직접 영향을 미친다.
이 변경이 어떤 조건에서 어떻게 작동하는지, 에지 케이스를 어떻게 처리하는지에 대한 설명이 부재하다.

기사의 구조적 선택 — React 컴포넌트 최적화에 집중하고 서버/클라이언트 경계 최적화를 부록 취급 — 은 이것이 팀의 주요 작업이었기 때문일 수도 있다.
그러나 독자 입장에서는 전체 최적화 중 어느 부분이 가장 큰 효과를 냈는지 파악하기 어렵다.
각 최적화 단계별 기여도가 없으면 다른 맥락에서 이 경험을 재적용하기 어렵다.

## 인사이트

### 컴포넌트 추상화의 비용은 사용량이 선형적으로 증가할 때 지수적으로 드러난다

GitHub의 사례가 보여주는 패턴은 단순히 “React는 큰 리스트에서 느리다”는 것이 아니다.
더 정확한 관찰은, 컴포넌트 추상화의 비용이 컴포넌트 인스턴스 수에 비례해 누적되기 때문에, 사용량이 선형적으로 증가하는 UI에서 그 비용이 눈에 보이게 된다는 것이다.

diff 뷰어는 이 패턴의 전형적인 사례다.
라인 수가 n배 증가하면, 총 DOM 노드, 이벤트 핸들러, 메모리 사용량도 n배 증가한다.
이것은 자연스럽게 보이지만, 좋은 사용자 경험을 위한 INP 임계값(200 ms 이하)은 n에 비례해 증가하지 않는다.
그 결과, 어느 시점부터는 “더 많은 콘텐츠 = 더 느린 인터랙션”의 관계가 사용자에게 명확하게 드러나게 된다.

이 관찰이 중요한 이유는 같은 패턴이 다른 많은 UI 컴포넌트에 잠재되어 있기 때문이다.
테이블, 타임라인, 피드, 파일 탐색기, 코드 에디터 — 이 모두가 항목 수에 비례해 인스턴스가 증가하는 컴포넌트다.
성능 문제는 설계 시점이 아니라 사용 규모가 특정 임계값을 넘을 때 발현된다.
따라서 이런 유형의 컴포넌트는 설계 단계부터 “라인당 비용”을 명시적으로 추적하는 방식이 필요하다.

React 생태계가 성숙해지면서 이 문제에 대한 인식도 높아졌다.
React 19의 컴파일러, Server Components, Suspense — 이것들은 모두 부분적으로 이 “스케일에서 나타나는 컴포넌트 추상화 비용” 문제를 해결하려는 시도다.
GitHub의 V2가 도달한 결론 — 컴포넌트를 줄이고, 이벤트 핸들러를 중앙화하고, DOM을 단순화 — 은 프레임워크 레벨의 해결책이 없을 때 애플리케이션 레벨에서 할 수 있는 것의 한계를 보여준다.

### 가상화는 성능 문제를 해결하지 않고 사용자 경험의 경계를 재정의한다

TanStack Virtual을 통한 가상화가 “10X 힙 감소”를 달성했다는 것은 인상적인 수치다.
그러나 이 수치가 의미하는 바를 정확하게 이해하려면, 가상화가 어떤 가정 위에서 작동하는지를 봐야 한다.
가상화는 “사용자는 한 번에 화면에 보이는 콘텐츠만 읽는다”는 가정을 UI 아키텍처에 하드코딩한 것이다.

이 가정이 항상 성립하는 것은 아니다.
Ctrl+F로 파일 내 특정 패턴을 찾는 것, 선택된 텍스트를 복사하는 것, 화면 외부의 특정 라인으로 빠르게 스크롤하는 것 — 이런 사용 패턴은 “전체 콘텐츠가 DOM에 존재한다”는 가정에 의존한다.
가상화는 이 가정을 깨뜨린다.

GitHub가 이 트레이드오프를 “p95 이상” PR에만 적용했다는 것은, 실제로는 “성능이 너무 나빠서 어떤 기능도 작동하지 않는 것보다는, 일부 기능이 제한되더라도 기본 인터랙션이 반응하는 것이 낫다”는 판단이다.
이것은 합리적인 판단이지만, 기사는 이것을 순수한 기술적 성취로 서술한다.
실제로는 이것이 “무엇을 희생할 것인가”의 선택이었다는 맥락이 필요하다.

장기적으로 이 결정은 더 복잡한 문제를 낳을 수 있다.
가상화된 diff 뷰어에서 “전체 파일에서 텍스트 찾기” 기능을 구현하려면 별도의 비가상화 인덱스를 유지해야 한다.
대규모 diff에서 특정 패턴의 변경 사항이 몇 군데 있는지 파악하는 기능 — 코드 리뷰에서 흔히 필요한 작업 — 은 가상화된 구조에서 구현하기 상당히 복잡해진다.
성능을 위한 가상화 결정이 기능 설계의 장기 비용을 높이는 패턴이다.

### 성능 최적화의 진짜 장벽은 기술이 아니라 측정 가능성이다

V1에서 V2로의 전환이 왜 그렇게 오래 걸렸는가 — 기사는 이 질문에 직접 답하지 않지만, Datadog INP 추적 구축이 최적화와 병행해 이루어졌다는 사실이 단서를 준다.
측정할 수 없었기 때문에 문제의 심각성을 정량화하기 어려웠고, 정량화하기 어려웠기 때문에 대규모 리팩터링을 정당화하기도 어려웠을 것이다.

이것은 성능 최적화에서 반복되는 패턴이다.
“페이지가 느리다”는 주관적 관찰은 있지만, “어느 인터랙션이 얼마나 느리고, 그 원인은 무엇이며, 변경 후 얼마나 개선되었는가”를 말할 수 있는 도구가 없는 상태.
이 상태에서는 성능 개선 작업이 “기술 부채 해소”로 분류되어 우선순위 경쟁에서 밀리는 경향이 있다.

인터랙션 수준의 INP 추적을 구축했다는 것은, 이제 GitHub 팀이 특정 인터랙션의 성능 회귀(regression)를 배포 전에 감지할 수 있는 체계를 갖추었다는 의미다.
이것은 V2 최적화 자체보다 장기적으로 더 가치 있는 결과일 수 있다.
최적화는 현재의 문제를 해결하지만, 측정 인프라는 미래의 회귀를 방지한다.
기사가 이 측면을 충분히 강조하지 않는 것은 아쉽다.

프론트엔드 성능 관리의 어려움은 여기서도 나타난다.
백엔드 서비스는 P99 레이턴시, 에러율, 처리량 같은 지표를 오랫동안 추적해 왔다.
프론트엔드에서 이에 상응하는 지표 — 인터랙션별 INP, 컴포넌트별 렌더 시간 — 를 프로덕션 수준에서 추적하는 것은 상대적으로 최근의 일이다.
Core Web Vitals와 INP의 부상은 이 격차를 좁히는 중이지만, GitHub의 사례가 보여주듯 세밀한 인터랙션 수준의 측정은 여전히 커스텀 인프라를 필요로 한다.

### 이 최적화 사이클은 프레임워크 의존성의 장기 비용을 드러낸다

V1에서 V2로의 전환에서 가장 큰 개선 중 하나는 React의 추상화를 우회하는 방향으로 이루어졌다.
단일 이벤트 핸들러, data attribute 기반 상태 추적, DOM 구조 단순화 — 이것들은 모두 React가 제공하는 컴포넌트 추상화를 부분적으로 포기하고 더 저수준(low-level)의 접근 방식으로 돌아가는 것이다.

이 패턴은 프레임워크 사용의 장기적 비용에 대한 질문을 제기한다.
프레임워크는 초기 개발 속도를 높이고, 공통 패턴에 대한 해결책을 제공하며, 팀 간 일관성을 유지하는 데 도움이 된다.
그러나 충분한 규모와 충분한 성능 요구사항에서는, 프레임워크의 추상화 자체가 병목이 되는 지점이 온다.
그 지점에서 선택은 프레임워크를 교체하거나, 프레임워크를 우회하거나, 프레임워크 내에서 비관용적(idiomatic) 방식을 포기하는 것이다.

GitHub가 선택한 것은 세 번째였다.
React를 유지하되, 라인당 2개의 컴포넌트와 단일 이벤트 핸들러라는 비관용적 구조를 채택했다.
이것이 앞으로의 유지보수에 어떤 의미인지 — 새로운 팀원의 온보딩, 기능 추가 시 패턴의 일관성 — 는 기사가 다루지 않는 비용이다.

더 큰 그림에서 보면, 이 사례는 성능이 핵심 요구사항인 UI에서는 프레임워크 선택이 더 신중해야 한다는 것을 시사한다.
diff 뷰어, 코드 에디터, 데이터 그리드처럼 수만 개의 동일 구조 엘리먼트를 렌더링하는 UI는 범용 컴포넌트 프레임워크보다 더 제한적이고 더 최적화된 추상화를 필요로 할 수 있다.
이것은 CodeMirror나 Monaco Editor가 React로 구현되지 않은 이유이기도 하다.
