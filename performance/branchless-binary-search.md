# 8배 빠른 이진 탐색: 컴파일 코드에서 기계 공감까지

원문: <https://pythonspeed.com/articles/branchless-binary-search/>

HN 토론: <https://news.ycombinator.com/item?id=48878996> (70점, 15개 댓글)

Lobste.rs 토론: <https://lobste.rs/s/czbhmr/6_faster_binary_search_from_compiled_code>

## 요약

Itamar Turner-Trauring가 scikit-learn의 그래디언트 히스토그램 부스팅 알고리즘에서
발생한 성능 병목을 CPU 동작 원리를 이해하는 방식으로 해결한 과정을 정리한 글이다.

문제는 부동소수점 배열 값을 균등하게 분포된 버킷(0-254)에 할당하는 작업이었다.
정렬된 버킷 경계에 이진 탐색을 적용했지만 예상보다 느렸다.
근본 원인은 알고리즘 복잡도가 아니라 분기 예측 실패였다.
입력 값이 균등 분포여서 CPU가 어떤 분기 방향으로 갈지 예측할 수 없었고,
원본 코드에서 값 하나당 27개의 분기가 발생하며 16.6%의 예측 실패율을 기록했다.
IPC(명령어 사이클당 처리 수)는 0.7에 그쳤다.

해결책은 분기 없는(branchless) 구현이었다.
핵심은 세 가지다:
`select_unpredictable()`로 조건 분기를 CPU 조건부 명령어로 교체해 분기 자체를 제거하고,
가변 루프 대신 `log₂(버킷_수)` 고정 반복을 사용하며,
안전성 검증 후 `get_unchecked()`로 배열 경계 검사를 제거했다.

최종 최적화는 루프 중첩 순서를 뒤집는 것이었다.
절반화(halving) 단계를 바깥 루프로, 값 처리를 안쪽으로 바꾸면
독립적인 여러 값을 동시에 처리하는 명령어 수준 병렬성(ILP)이 드러난다.
결과적으로 IPC가 2.5에서 4.9로 올라갔다.

최종 벤치마크는 원본 45,200마이크로초에서 5,453마이크로초로 줄어
약 8배 빨라졌다.
주목할 점은 최종 구현이 총 CPU 명령어 수는 오히려 많지만 더 빠르다는 것이다.
분기 예측 실패 비용을 없애고 ILP를 활용한 결과다.

## 분석

### 성능 최적화의 선형 모델이 깨지는 지점

기존의 성능 사고방식은 “명령어 수가 적을수록 빠르다”는 선형 모델을 따른다.
이 글은 그 모델이 현대 CPU에서 성립하지 않는 경우를 보여 준다.
최종 구현은 명령어 수가 원본보다 많지만 8배 빠르다.
IPC가 0.7에서 4.9로 올라갔다는 것은 CPU가 명령어당 더 많은 일을 했다는 의미다.
분기 예측 실패가 파이프라인을 플러시하면 수십 사이클의 지연이 발생하는데,
27개의 분기가 16.6%씩 실패하면 처리량이 지배적으로 줄어든다.
알고리즘 복잡도 O(log n)은 CPU 파이프라인 효율을 설명하지 못한다.

### `select_unpredictable()`이 하는 일의 정확한 의미

이 함수는 단순한 조건부 선택이 아니다.
일반적인 `if-else`는 컴파일러가 조건 분기 명령어로 번역하고
CPU 분기 예측기가 이를 처리한다.
예측이 맞으면 파이프라인이 계속 흐르지만 틀리면 플러시 비용이 든다.
`select_unpredictable()`은 컴파일러에게 “이 분기는 예측 불가능하니
조건부 이동(cmov) 같은 분기 없는 명령어를 쓰라”는 힌트를 준다.
결과는 예측 실패가 없는 대신 항상 두 경로의 비용을 지불하는 방식이지만,
예측 불가능한 분기에서는 이것이 훨씬 낫다.

### 루프 순서 역전이 ILP를 드러내는 이유

값 루프를 바깥으로 두면 각 값이 독립적으로 이진 탐색을 완료하고 다음 값으로 넘어간다.
이 경우 CPU는 현재 값의 결과를 기다려야 다음 연산을 시작할 수 있다.
절반화 단계를 바깥 루프로 뒤집으면 같은 단계에서 여러 값이 동시에 처리된다.
각 값의 처리가 서로 독립적이므로 CPU의 슈퍼스칼라 실행 유닛이 병렬로 처리할 수 있다.
단일 스레드 코드에서 IPC가 2배 이상 오를 수 있다는 것이 핵심이다.
이는 SIMD나 멀티스레딩 없이 명령어 순서만으로 얻는 병렬성이다.

### 이 최적화가 일반화되기 어려운 조건

이 기법이 효과적인 이유는 입력이 균등 분포라 분기가 예측 불가능하다는 전제가 있기 때문이다.
입력이 편향돼 있어 CPU가 분기 방향을 70-80% 이상 맞힐 수 있다면,
조건 분기가 오히려 조건부 이동보다 빠를 수 있다.
또한 버킷 수가 log₂ 반복으로 고정될 수 있어야 한다.
버킷 크기나 분포가 런타임에 달라진다면 고정 반복 접근 자체가 맞지 않는다.
“이 기법은 분기 예측 실패가 병목인 균등 분포 입력에 한해 효과적이다”라는 전제를
파악하지 못하면 잘못된 상황에 적용할 수 있다.

Lobste.rs 토론에서는 글의 저자 itamarst가 Eytzinger 레이아웃을 다룬 관련 글을
직접 공유했다.[^itamarst]
정렬된 순서 대신 BFS(너비 우선 탐색) 순서로 배열을 저장하면 순차적으로 접근되는
캐시 라인이 바뀌어 캐시 지역성이 개선되며, 이 접근은 분기 예측과는 별개로
메모리 계층 구조를 활용하는 최적화 방향이다.
pillmillipedes도 버킷을 Eytzinger 레이아웃으로 배치하면 캐시 지역성을 높일 수 있다고
제안했으며, 보간 탐색(interpolation search)으로 중간점 추정 자체를 개선하는
접근도 덧붙였다.[^pillmillipedes]
이 두 방향은 분기 제거와 독립적으로 조합할 수 있는 최적화 축이다.

## 비평

### 벤치마크가 단일 하드웨어 결과다

8배라는 숫자는 특정 CPU에서 측정한 결과다.
분기 예측기의 정확도, 파이프라인 깊이, ILP 지원 수준은 CPU 아키텍처마다 크게 다르다.
Intel, AMD, ARM 중 어디서 측정했는지, 동일 코드가 각기 다른 아키텍처에서
얼마나 다른 개선을 보이는지에 대한 정보가 없다.
8배는 인상적인 수치지만 다른 환경에서 재현 가능한 수치인지는 별도 검증이 필요하다.

Lobste.rs에서 purplesyringa는 더 근본적인 질문을 제기했다.[^purplesyringa]
이진 탐색 자체를 최적화하기보다 다른 데이터 레이아웃을 선택하면
비슷한 성능을 더 나은 유지보수성으로 얻을 수 있다는 것이다.
저자 itamarst는 실제 변경이 scikit-learn 풀 리퀘스트에서 +54, -15 수준의 소규모였으며
C 코드베이스라는 제약 때문에 현실적인 선택을 했다고 답했다.[^itamarst-lobsters]
성능 최적화는 이론적 최선이 아니라 기존 코드베이스 제약 안의 실용적 판단이다.

### `get_unchecked()`의 unsafe 사용이 실무에서 갖는 맥락

글은 경계 검사 제거를 “안전성 검증 후 적용”이라고 간략히 다루지만,
실무에서 unsafe 코드를 유지하는 데 드는 비용이 설명되지 않는다.
코드 변경 시마다 불변 조건이 여전히 성립하는지 확인해야 하고,
리뷰어가 unsafe 블록의 전제를 이해하고 있어야 한다.
성능 숫자만 제시하고 유지보수 비용을 빠뜨리면 독자가 unsafe를
성능 최적화의 기본 도구로 오해할 수 있다.

### 성능 전문 지식의 진입 장벽을 낮게 서술한다

글의 마지막은 참고 자료 목록으로 마무리되며 “기계 공감을 학습할 수 있다”는
메시지를 전한다.
하지만 perf stat 해석, 파이프라인 스톨 분석, CPU별 마이크로아키텍처 차이 이해는
책 몇 권으로 빠르게 습득하기 어려운 영역이다.
이 최적화를 직접 발견하고 적용할 수 있는 개발자 비율은 상당히 낮으며,
글이 그 진입 장벽을 충분히 표현하지 않는다.

## 인사이트

### 컴파일러는 예측 불가능성을 모른다

컴파일러 최적화는 코드 구조, 타입, 제어 흐름을 분석하지만
런타임 입력의 분포를 알 수 없다.
이 글의 핵심 통찰은 개발자가 “이 분기는 입력 분포상 예측이 불가능하다”는
도메인 지식을 갖고 있다면 컴파일러에게 명시적으로 힌트를 줄 수 있다는 점이다.
`select_unpredictable()`은 그 힌트를 전달하는 인터페이스다.
프로파일 가이드 최적화(PGO)가 런타임 데이터로 컴파일러를 보완하듯,
이 접근은 개발자가 입력 특성을 코드에 직접 반영하는 방식이다.
알고리즘 지식과 CPU 아키텍처 지식이 합쳐질 때만 가능한 최적화다.

### 성능 측정 없이 이런 최적화는 발견되지 않는다

이 최적화는 perf stat으로 분기 수와 예측 실패율을 측정했기 때문에 발견됐다.
코드 리뷰나 정적 분석으로는 “분기 예측 실패가 병목”이라는 사실을 찾을 수 없다.
이는 성능 최적화가 계측(instrumentation) 없이는 눈먼 추측이 된다는 것을 보여 준다.
“느리다고 느껴지는 부분”을 직관으로 최적화하는 것이 아니라
perf, flamegraph, hardware counter 같은 도구로 실제 병목을 찾아야 한다.
측정이 없으면 잘못된 것을 최적화하고 코드만 복잡해지는 경우가 많다.

### ILP는 단일 스레드 코드에서도 병렬성의 원천이다

많은 개발자에게 “병렬화”는 멀티스레딩이나 SIMD를 의미한다.
이 글은 루프 순서 변경만으로 CPU의 실행 유닛이 독립 명령어를 병렬 실행하게 하여
IPC를 2배 가까이 높일 수 있음을 보여 준다.
단일 스레드, 스칼라 코드에서도 명령어 수준 병렬성이 존재하며,
이를 드러내는 코드 구조 선택이 멀티스레딩 도입 없이도 상당한 속도 향상을 만든다.
루프 중첩 순서, 데이터 의존성 최소화, 독립 연산 그룹화는
CPU 아키텍처를 이해하는 개발자가 활용할 수 있는 단일 스레드 최적화 기법이다.
