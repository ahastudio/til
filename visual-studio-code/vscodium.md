# VSCodium: 텔레메트리 없는 VS Code 바이너리

<https://vscodium.com/>

## 소개

VSCodium은 Microsoft VS Code를 소스에서 빌드해 MIT 라이선스로 배포하는 커뮤니티
주도 프로젝트다.
GitHub에서 별 31,000개 이상, 포크 1,700개 이상을 기록하고 있다.
“포크가 아니다”를 명시적으로 강조하는 것이 특징이다.
이 프로젝트는 Microsoft의 `vscode` 저장소를 클론해 빌드 스크립트를 실행하고,
그 결과물을 GitHub 릴리스로 업로드하는 자동화 스크립트의 집합이다.

존재 이유는 VS Code의 구조적 이중성에 있다.
VS Code의 소스 코드는 MIT 라이선스로 공개되어 있지만, Microsoft가 배포하는 바이너리는
독점 라이선스(proprietary license)를 적용하고 텔레메트리를 포함한다.
VS Code 메인테이너가 직접 GitHub 이슈에 남긴 설명에 따르면, Microsoft는 빌드 시
`product.json`을 교체해 텔레메트리, 마켓플레이스 엔드포인트, 로고 등 독점 기능을
추가한다.
수정되지 않은 소스에서 빌드한 결과물은 이 커스터마이제이션이 없는 “clean” 빌드로,
기본 MIT 라이선스가 유지된다.
VSCodium은 이 빌드 과정을 자동화해 사용자가 직접 소스를 내려받아 컴파일하지 않아도
되도록 한다.

설치 방법은 macOS, Windows, Linux 전 플랫폼에 걸쳐 폭넓게 지원된다.
macOS는 Homebrew(`brew install --cask vscodium`), Windows는 WinGet·Chocolatey·Scoop,
Linux는 snap·Flatpak·apt·dnf·zypper·AUR 등을 통해 설치할 수 있다.
안정(stable) 버전과 인사이더(insiders) 버전을 각각 제공한다.

## 분석

### `product.json` 교체 하나가 만드는 법적 차이

VS Code 빌드의 핵심 분기점은 `product.json` 파일이다.
이 파일이 Microsoft 내부 버전에는 텔레메트리 수집 엔드포인트, 독점 마켓플레이스 URL,
로고, 라이선스 식별자가 포함된다.
오픈소스 소스로 빌드할 때는 이 파일이 기본값 상태이며, 결과물은 MIT 라이선스로
유지된다.
동일한 소스코드 저장소에서 빌드한 결과물이 `product.json` 하나의 차이로 법적 지위가
달라진다는 것은 소프트웨어 라이선스 구조의 복잡성을 단적으로 보여준다.

### 빌드 시 다운로드되는 추가 파일

VSCodium 빌드 과정이 완전히 클린하지는 않다는 점도 주목할 만하다.
Microsoft의 빌드 프로세스를 그대로 따르기 때문에, 빌드 중 GitHub에서 프리빌드된
확장(js-debug, js-debug-companion, vscode-js-profile-table)과 Electron 릴리스에서
electron 및 ffmpeg 바이너리를 내려받는다.
“소스로부터 빌드”라는 표현이 완전한 재현 가능성(reproducible build)을 의미하지는
않는다.
이 점은 프로젝트 문서에서 명시적으로 공개하고 있다.

### 마켓플레이스와 확장 호환성

VSCodium은 Microsoft의 독점 마켓플레이스(Visual Studio Marketplace)를 기본으로
사용하지 않는다.
대신 오픈 소스 대안인 Open VSX Registry를 기본 확장 마켓플레이스로 설정한다.
이 때문에 Microsoft 독점 마켓플레이스에만 등록된 일부 확장(특히 Microsoft 자체
확장—Python, Pylance, GitHub Copilot 등)은 VSCodium에서 기본 경로로 설치할 수 없다.
사용자가 직접 `.vsix` 파일을 내려받아 설치하거나, 마켓플레이스 엔드포인트를 수동으로
변경해야 한다.
이것이 실제 사용에서 가장 큰 마찰 지점이다.

### 자동화 스크립트 집합으로서의 정체성

“포크가 아니다”라는 강조는 기술적으로 정확하다.
VSCodium 저장소에는 편집기 코드가 한 줄도 없다.
Microsoft의 `vscode` 저장소를 클론하고, 빌드하고, 바이너리를 릴리스하는 CI/CD
파이프라인 스크립트만 있다.
이 구조는 유지보수 부담을 최소화하면서 최신 VS Code 버전을 지속적으로 따라갈 수
있게 한다.
반면, Microsoft가 `product.json` 외 부분에서 빌드 과정을 변경하면 VSCodium
빌드가 영향을 받을 수 있다는 취약성도 내포한다.

## 비평

### 강점: 문제의 명확한 진단과 최소 마찰 해결책

소스 공개 vs. 바이너리 배포의 라이선스 불일치라는 문제를 정확히 짚고,
그 해결책으로 자동화 빌드 스크립트라는 가장 단순한 접근을 택한 것이 강점이다.
사용자는 소스를 이해하지 않아도 익숙한 패키지 매니저로 설치할 수 있고,
실제 편집 경험은 VS Code와 사실상 동일하다.
“포크가 아니다”라는 투명한 자기 규정도 프로젝트 신뢰도를 높인다.

### 한계: 마켓플레이스 단절이 실용성을 제약한다

Open VSX Registry는 Visual Studio Marketplace에 비해 확장 수와 업데이트 속도 모두
뒤처진다.
Microsoft 독점 확장(Copilot, Pylance, Remote Development 등)은 VSCodium에서 공식
경로로 설치할 수 없다.
현대 개발 워크플로우에서 이 확장들에 의존하는 팀에게 VSCodium은 실질적인 선택지가
되기 어렵다.
텔레메트리 비활성화가 마켓플레이스 단절과 묶여 있어, 하나만 취하는 것이 불가능한
구조다.

### 보완이 필요한 관점: 텔레메트리 비활성화만으로 충분한가

VS Code 자체도 설정에서 텔레메트리를 비활성화할 수 있다(`telemetry.telemetryLevel: off`).
VSCodium이 제공하는 것은 그 비활성화를 코드 레벨에서 제거하는 것으로,
설정 기반 비활성화보다 더 강한 보증을 제공한다.
하지만 VSCodium이 사용하는 확장들 자체의 텔레메트리는 별도 문제다.
Copilot 같은 확장이 없더라도, 사용자가 설치한 서드파티 확장이 독자적으로 데이터를
수집하는 것은 VSCodium이 제어할 수 없다.
“텔레메트리 없는 편집기”가 “텔레메트리 없는 개발 환경”을 보장하지는 않는다.

## 인사이트

### 오픈소스 라이선스의 “이중 문서” 패턴

VSCodium이 존재하게 된 근본 원인은 Microsoft가 소스 저장소와 배포 바이너리에 서로
다른 라이선스를 적용하는 구조다.
소스는 MIT, 바이너리는 독점—이 구조는 법적으로 완전히 합법이지만, 사용자에게는
혼란스럽다.
이것을 “이중 문서(dual artifact)” 패턴이라고 부를 수 있다.
동일한 소스에서 두 가지 다른 법적 지위의 산출물이 만들어지는 것이다.

이 패턴은 VS Code만의 것이 아니다.
Google Chrome과 Chromium, Oracle MySQL과 MariaDB, Sun의 Java와 OpenJDK가 모두
같은 구조를 거쳤다.
공통점은 상업적 가치가 있는 소스를 오픈소스로 공개하되, 배포 바이너리에 추가적인
통제 레이어를 얹는다는 것이다.
그리고 매번 커뮤니티가 해당 바이너리를 “클린하게” 재빌드하는 프로젝트를 만들었다.
VSCodium은 이 반복되는 패턴의 가장 최근 사례다.
이 패턴이 반복되는 이유는, 기업이 오픈소스 생태계의 혜택—개발자 채택, 기여, 신뢰—을
얻으면서 동시에 배포 채널에서 데이터 수집이나 서비스 연결의 이점을 가져가려 하기
때문이다.
법적으로는 라이선스를 분리하는 것이 가장 단순한 방법이다.

### 텔레메트리 논쟁은 신뢰의 문제이지 기능의 문제가 아니다

VS Code의 텔레메트리를 비활성화하는 방법은 이미 존재한다.
설정 파일 한 줄로 가능하다.
그렇다면 VSCodium이 필요한 이유는 무엇인가.
답은 기술적 기능이 아니라 신뢰의 구조에 있다.

설정으로 비활성화된 텔레메트리는 Microsoft 코드가 그 설정을 존중한다는 신뢰에 의존한다.
VSCodium은 그 신뢰를 코드 제거로 대체한다.
“믿어라”가 아니라 “확인하라”는 원칙—이것은 보안 엔지니어링의 핵심 원리인 “신뢰하되
검증하라(trust but verify)”를 넘어 “검증 가능하도록 설계하라”에 가깝다.
오픈소스 소프트웨어의 가치는 코드를 읽을 수 있다는 것이 아니라, 읽힐 수 있다는
구조 자체가 신뢰를 만든다는 데 있다.
독점 바이너리는 그 구조가 없다.

이 프레임에서 VSCodium을 선택하는 것은 “텔레메트리가 싫다”는 감정적 반응이 아니라,
“독립적으로 검증 가능한 소프트웨어만 사용하겠다”는 공급망 보안 정책의 실행이다.
기업 환경에서 이 정책은 구체적인 리스크 관리 요건이 될 수 있다.

### 빌드 스크립트로서의 프로젝트: 유지보수 부담의 최소화 전략

VSCodium이 “포크가 아니다”라고 강조하는 것은 마케팅 언어가 아니라 아키텍처 결정이다.
포크는 원본 저장소와 지속적으로 동기화해야 하는 부담을 낳는다.
병합 충돌, 버전 추적, 패치 재적용—이 비용은 소규모 커뮤니티 프로젝트에서 프로젝트를
죽이는 주요 원인이다.

VSCodium은 이 문제를 근본적으로 회피한다.
편집기 코드를 소유하지 않으므로 동기화할 것이 없다.
`product.json`을 교체하는 최소한의 변경만 유지하면 된다.
이것은 소프트웨어 설계에서 “소유하지 말고 조합하라(don't own, compose)”의 원칙을
인프라 레벨에서 구현한 것이다.
코드를 복사하는 대신 빌드 파이프라인을 구성하고, 결과물만 배포한다.

이 구조의 취약점은 Microsoft가 빌드 과정을 변경할 때 노출된다.
Electron 버전 변경, 빌드 도구 업데이트, `product.json` 구조 변경 모두 VSCodium
파이프라인에 영향을 줄 수 있다.
프로젝트의 지속성이 Microsoft의 빌드 과정 안정성에 종속되는 것이다.
이것은 오픈소스 생태계에서 상류 프로젝트(upstream)에 의존하는 모든 파생 프로젝트가
공유하는 구조적 취약성이며, VSCodium은 그 종속성을 최소화하는 방식으로 설계했지만
완전히 제거하지는 못했다.
