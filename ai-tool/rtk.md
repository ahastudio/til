# rtk - LLM 토큰 소비를 60~90% 줄여주는 CLI 프록시

> 원문: <https://github.com/rtk-ai/rtk>

## 요약

rtk(Rust Token Killer)는 AI 코딩 도구가 CLI 명령어 출력을 LLM에 전달하기 전에
압축·필터링하여 토큰 소비를 60~90% 줄여주는 Rust 기반 단일 바이너리 프록시다.
Claude Code, GitHub Copilot, Cursor 등 10개 이상의 AI 코딩 도구와 호환되며,
100개 이상의 CLI 명령어를 지원한다.

30분 Claude Code 세션에서 약 118,000 토큰을 23,900 토큰으로 줄여 80% 절감한
실측 사례를 제시한다. `git add/commit/push`는 92%, `cargo test`/`npm test`는
최대 90% 절감된다.

## 분석

rtk의 핵심은 4가지 압축 전략이다:

1. **스마트 필터링** — 주석, 공백, 보일러플레이트 코드 제거
2. **그룹화** — 유사 항목 집계(디렉토리별 파일, 오류 유형별 분류)
3. **절단** — 관련 컨텍스트는 유지하면서 중복 내용 제거
4. **중복 제거** — 반복 로그 라인을 카운트 형태로 축약

`rtk init -g` 명령 하나로 Claude Code용 훅과 RTK.md가 설치되며, AI 도구를
재시작하면 Bash 명령이 자동으로 rtk 동등물로 재작성되어 투명하게 동작한다.
오버헤드는 10ms 미만이다.

GeekNews 댓글에서는 컨텍스트 손실 문제가 핵심 우려로 제기되었다. "Claude Code가
명령어 결과에서 필요한 컨텍스트를 확보하지 못하는 경우가 있어 Case by Case"라는
지적과, LLM이 일반 CLI 출력 형식으로 학습되었기 때문에 압축된 출력이 성능 저하를
유발할 수 있다는 우려도 있었다.

## 비평

rtk의 접근법은 본질적으로 손실 압축(lossy compression)이다. 토큰을 줄이기 위해
정보를 버리는 것이므로, 어떤 정보를 버려도 안전한지 rtk가 항상 올바르게 판단한다는
보장이 없다. 실제로 Reddit에서 부작용 사례가 보고되었고, LLM 모델은 원래의 풍부한
CLI 출력을 학습 기반으로 삼기 때문에 압축된 형식은 분포 이탈(distribution shift)을
일으킬 수 있다.

또한 토큰 비용 절감이 모든 상황에서 최적 목표가 아닐 수 있다. 상세한 에러 로그나
스택 트레이스 같은 경우, 토큰이 많더라도 그 정보가 디버깅에 결정적 역할을 할 수
있다. "더 많은 컨텍스트 = 더 나은 결과"인 경우에는 rtk가 역효과를 낼 수 있다.

## 인사이트

**컨텍스트 윈도우의 경제학이 새로운 도구 계층을 만든다.** 토큰이 돈이 되는 세상에서
토큰 최적화 레이어는 필연적으로 등장한다. rtk는 단순한 압축 도구가 아니라 "LLM
인프라의 비용 최적화 계층"의 초기 형태다. 앞으로 이 계층은 더 정교해질 것이다.

**프록시 패턴으로 AI 도구를 보강하는 전략.** rtk처럼 기존 AI 도구를 수정하지 않고
앞단에 프록시를 끼워 동작을 변경하는 방식은 확장성이 높다. 이 패턴은 캐싱,
라우팅, 필터링 등 다양한 용도로 응용될 수 있으며, AI 도구 생태계의 "미들웨어"
계층이 형성되고 있다는 신호다.

**손실 압축과 무손실 압축의 선택 문제.** 이미지 압축에서 JPEG(손실)와 PNG(무손실)
용도가 다르듯, LLM 컨텍스트 압축도 상황에 따라 전략이 달라야 한다. rtk가 장기적으로
가치를 가지려면 "언제 압축하고 언제 하지 말아야 하는지"를 학습하는 지능형
적응 압축으로 발전해야 할 것이다.
