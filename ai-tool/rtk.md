# RTK — Rust Token Killer

> CLI proxy that reduces LLM token consumption by 60-90% on common dev commands.
> Single Rust binary, zero dependencies

<https://www.rtk-ai.app/>

<https://github.com/rtk-ai/rtk>

## 소개

rtk(Rust Token Killer)는 AI 코딩 도구가 CLI 명령어 출력을 LLM에 전달하기 전에
압축·필터링하여 토큰 소비를 60~90% 줄여주는 Rust 기반 단일 바이너리 프록시다.
Claude Code, GitHub Copilot, Cursor 등 10개 이상의 AI 코딩 도구와 호환되며,
100개 이상의 CLI 명령어를 지원한다.

30분 Claude Code 세션에서 약 118,000 토큰을 23,900 토큰으로 줄여 80% 절감한
실측 사례를 제시한다. `git add/commit/push`는 92%, `cargo test`/`npm test`는
최대 90% 절감된다.

rtk의 핵심은 4가지 압축 전략이다.

1. **스마트 필터링** — 주석, 공백, 보일러플레이트 코드 제거
2. **그룹화** — 유사 항목 집계(디렉토리별 파일, 오류 유형별 분류)
3. **절단** — 관련 컨텍스트는 유지하면서 중복 내용 제거
4. **중복 제거** — 반복 로그 라인을 카운트 형태로 축약

## 사용법

`rtk init -g` 명령 하나로 Claude Code용 훅과 RTK.md가 설치되며, AI 도구를
재시작하면 Bash 명령이 자동으로 rtk 동등물로 재작성되어 투명하게 동작한다.
오버헤드는 10ms 미만이다.

에이전트가 압축 사실을 인지하고 원본이 필요한 경우, `RTK_DISABLE=1`
환경 변수로 압축을 끄고 원시 출력을 그대로 받는 우회로도 마련되어
있다.[^ilia-a]

## 관련 글 — 토큰 압축의 환상

원문: <https://mroczek.dev/articles/the-token-compression-illusion-why-im-skeptical-of-rtk>

HN 토론: <https://news.ycombinator.com/item?id=48588755> (119점, 111개 댓글)

Przemek Mroczek가 2026년 6월 18일 발행한 비평 글은 rtk를 향한 회의를 다섯
가지 기술적 논거로 전개한다.
저자의 핵심 주장은 "60~90%"가 실제 API 청구서 절감이 아니라 명령줄 출력(CLI
output)에서 걷어낸 원시 텍스트의 비율일 뿐이라는 것이다.
실제 비용을 좌우하는 요소 — 깊은 파일 읽기, 저장소 컨텍스트, 시스템 프롬프트,
모델의 추론(reasoning) 토큰 — 은 rtk가 건드리지 않는다.
따라서 그 지표는 아키텍처적 최적화가 아니라 "자랑용 스크린샷을 번쩍이기 위해
설계된" 것에 가깝다고 본다.

다섯 가지 논거는 다음과 같다.
첫째, 부풀려진 지표 — 터미널 출력은 전체 토큰 소비의 작은 부분이라 절감 통계가
현실적 API 절감으로 이어지지 않는다.
둘째, 조용한 실패(silent failure) 위험 — rtk가 출력을 압축할 때 LLM은 정보가
제거되었다는 사실을 모른다.
셋째, 정확도 벤치마크 부재 — rtk는 토큰 감소 그래프는 제시하지만 작업 성공률은
빠뜨린다.
넷째, 아키텍처적 취약성 — 압축은 독립 제품이라기보다 기능이며, 주류 CLI 도구가
네이티브 `--compact`나 `--json-stream` 플래그를 추가하면 rtk의 외부 의존성은
무의미해진다.
다섯째, 파싱 취약성 — rtk는 사람이 읽는 stdout/stderr 형식을 파싱하는 데
의존하므로, `git`·`cargo`·`npm`·`grep`의 출력 형식이 바뀌면 파싱 필터가 깨져
명시적 오류 대신 조용한 출력 손상을 일으킬 수 있다.

GeekNews 댓글에서는 컨텍스트 손실 문제가 핵심 우려로 제기되었다. "Claude Code가
명령어 결과에서 필요한 컨텍스트를 확보하지 못하는 경우가 있어 Case by Case"라는
지적과, LLM이 일반 CLI 출력 형식으로 학습되었기 때문에 압축된 출력이 성능 저하를
유발할 수 있다는 우려도 있었다.

## 분석

### 절감 지표와 비용 구조의 불일치

저자의 논증은 단순하지만 강력한 회계적 통찰에 기댄다 — 압축률이 높은 항목과
비용이 큰 항목이 다르다는 것이다.
CLI 출력은 압축하기 쉽지만(반복적이고 보일러플레이트가 많아서) 전체 토큰
비용에서 차지하는 비중은 작다.
반대로 비용의 대부분을 차지하는 파일 내용, 시스템 프롬프트, 추론 토큰은
압축이 어렵거나 rtk의 사정권 밖이다.

이 불일치가 "환상"의 정체다.
rtk는 "출력 토큰의 80%를 줄였다"고 말하고 사용자는 "내 청구서가 80% 줄겠다"고
듣지만, 둘은 같은 80%가 아니다.
출력이 전체 비용의 10%라면 80% 압축은 전체의 8% 절감일 뿐이다.

HN 댓글의 실측이 저자의 산술을 뒷받침했다.
한 사용자는 rtk를 써 봤지만 자신의 컨텍스트 90%를 차지하는 메시지는 압축되지
않아 작은 부분만 줄었다며, `/context`를 보면 토큰이 도구 호출에 쓰이는 게
아님을 알 수 있다고 했다 — "도구 호출을 8배 압축하는 것은 사실이지만 긴 코딩
세션에서는 중요하지 않다."[^tlarkworthy]
또 다른 사용자는 30만 토큰 세션에서 절감된 것이 3천 토큰 남짓이었다고
보고했다.[^Otterly99]

### 회의의 진짜 표적은 "측정되지 않은 품질 손실"이다

다섯 논거 중 가장 무거운 것은 부풀린 지표가 아니라 조용한 실패와 정확도
부재다.
저자가 반복해서 돌아오는 지점은 "토큰을 줄였다"는 측정 가능한 이득과 "작업이
여전히 성공하는가"라는 측정되지 않은 비용의 비대칭이다.
압축이 정보를 버릴 때 그 손실은 즉시 보이지 않고, 나중에 에이전트의 실패로
간접적으로만 드러난다.
측정하기 쉬운 것(토큰)을 최적화하고 측정하기 어려운 것(정확도)을 방치하면,
시스템은 측정된 지표는 좋아지면서 실제 목적은 나빠지는 방향으로 표류한다.

### "기능 대 제품" 논쟁의 한 사례

네 번째 논거(아키텍처적 취약성)는 실리콘밸리의 오래된 질문을 소환한다 —
"이것은 제품인가, 기능인가?"
저자는 토큰 압축이 독립 제품으로 서기엔 너무 얇으며, 결국 CLI 도구나 에이전트
하니스가 네이티브로 흡수할 기능이라고 본다.
주류 도구가 `--json-stream`을 추가하는 순간 외부 압축 프록시는 존재 이유를
잃는다.

HN에서 가장 많은 공감을 받은 댓글은 이 정서를 "LLM 매직 박스 산업"이라는 말로
확장했다 — 개발자가 엔지니어 대신 주문을 외우는 마법사가 되었고, 토큰 절감
퍼센트를 광고하는 GitHub 저장소는 믿지 말아야 하며, "하니스(Codex/Claude)에
들어가지 않은 아이디어는 대체로 그리 좋지 않다"는 경험칙이다.[^cityofdelusion]

## 비평

rtk의 접근법은 본질적으로 손실 압축(lossy compression)이다. 토큰을 줄이기 위해
정보를 버리는 것이므로, 어떤 정보를 버려도 안전한지 rtk가 항상 올바르게 판단한다는
보장이 없다. 실제로 Reddit에서 부작용 사례가 보고되었고, LLM 모델은 원래의 풍부한
CLI 출력을 학습 기반으로 삼기 때문에 압축된 형식은 분포 이탈(distribution shift)을
일으킬 수 있다.

또한 토큰 비용 절감이 모든 상황에서 최적 목표가 아닐 수 있다. 상세한 에러 로그나
스택 트레이스 같은 경우, 토큰이 많더라도 그 정보가 디버깅에 결정적 역할을 할 수
있다. "더 많은 컨텍스트 = 더 나은 결과"인 경우에는 rtk가 역효과를 낼 수 있다.

### 비평 글 자체의 약점 — 검증 없는 회의

앞서 소개한 Mroczek의 비평 글에도 뚜렷한 약점이 있다.
가장 큰 약점은 저자가 자신이 rtk에 요구하는 바로 그 일을 하지 않았다는 것이다.
저자는 rtk에 "SWE-bench 스타일 정확도 평가"가 없다고 비판하지만, 정작 자신도
rtk를 실제 작업에 돌려 정확도가 떨어지는지를 측정하지 않았다.
다섯 논거는 모두 "손실이 일어날 수 있다"는 가능성의 논증이지, "손실이
일어났다"는 증거의 논증이 아니다.

HN의 한 rtk 사용자는 이 약점을 정확히 찔렀다 — 저자가 우려한 "파싱이 깨지면
조용히 손상된 텍스트를 먹인다"는 시나리오에 대해, rtk는 "필터가 실패하면
단순히 원시 출력으로 폴백(fallback)"하도록 설계되어 있어 바로 그 시나리오를
회피하는 것이 핵심 설계 기둥이라고 반박했다.[^compuficial]
또 다른 사용자는 에이전트가 rtk 압축을 인지하고 `RTK_DISABLE=1`로 원본을
복원하는 우회로가 실제로 잘 작동한다고 증언했다.[^ilia-a]
한 댓글은 "이 글은 주장을 뒷받침할 데이터가 사실상 없고 대부분 LLM이 생성한
것처럼 읽힌다"고 일축했다.[^Bnjoroge]

또한 첫 번째 논거 전체가 "CLI 출력이 전체 토큰의 작은 부분"이라는 전제에
의존하는데, 이 전제는 워크로드에 따라 크게 달라진다.
테스트 스위트를 반복 실행하거나, 장황한 빌드 로그를 다루거나, 대용량 로그
파일을 grep하는 에이전트에게는 출력 토큰이 비용의 지배적 부분일 수 있다.
HN의 반응이 이 조건부성을 그대로 입증했다.
도구 사용 출력이 전체 출력에서 큰 비중을 차지한다며 "390만 입력 토큰 중 370만
토큰을 절감했다"는 사용자가 있는가 하면,[^compuficial] 솔로 개발자에게는
불필요할 수 있지만 2천 명 규모 조직에서는 (현재 약 250만 단위의) 비용이 "매운
부분"이라 트레이드오프를 알면서도 도입한다는 운영 관점의 반론도 나왔다.[^ziyasal]

네 번째 논거("기능이지 제품이 아니다") 역시 지나치게 많은 것을 증명한다.
같은 논리로 거의 모든 성공적 개발자 도구를 부정할 수 있기 때문이다.
jq는 셸이 JSON을 네이티브로 다루면 무의미하고, ripgrep은 grep이 빨라지면
무의미하며, Prettier는 언어가 포매팅을 내장하면 무의미하다.
"플랫폼이 흡수할 수 있다"와 "플랫폼이 흡수할 것이다" 사이에는 간극이 있고,
그 공백을 메우는 기간이 바로 도구의 수명이자 가치다.

## 인사이트

**컨텍스트 윈도우의 경제학이 새로운 도구 계층을 만든다.** 토큰이 돈이 되는 세상에서
토큰 최적화 레이어는 필연적으로 등장한다. rtk는 단순한 압축 도구가 아니라 "LLM
인프라의 비용 최적화 계층"의 초기 형태다. 앞으로 이 계층은 더 정교해질 것이다.

**프록시 패턴으로 AI 도구를 보강하는 전략.** rtk처럼 기존 AI 도구를 수정하지 않고
앞단에 프록시를 끼워 동작을 변경하는 방식은 확장성이 높다. 이 패턴은 캐싱,
라우팅, 필터링 등 다양한 용도로 응용될 수 있으며, AI 도구 생태계의 "미들웨어"
계층이 형성되고 있다는 신호다.

**손실 압축과 무손실 압축의 선택 문제.** 이미지 압축에서 JPEG(손실)와 PNG(무손실)
용도가 다르듯, LLM 컨텍스트 압축도 상황에 따라 전략이 달라야 한다. rtk가 장기적으로
가치를 가지려면 "언제 압축하고 언제 하지 말아야 하는지"를 학습하는 지능형
적응 압축으로 발전해야 할 것이다.

**토큰 압축 논쟁의 본질은 비용 절감이 아니라 컨텍스트 큐레이션이다.** 이 도구를
둘러싼 진짜 질문은 비용이 아니라 "에이전트에게 무엇을 보여줄 것인가"다.
압축은 정보를 버리는 행위이고, 무엇을 버릴지 결정하는 것은 곧 컨텍스트
큐레이션이다.
에이전트 비용이 계속 문제가 되는 한, 컨텍스트 큐레이션은 외부 프록시가 아니라
에이전트 자신의 핵심 역량이 되어야 한다.
즉 미래의 에이전트는 "모든 출력을 받은 뒤 압축"하는 것이 아니라, "애초에
무엇을 실행하고 무엇을 읽을지"를 스스로 판단하는 방향으로 진화한다.
rtk 같은 도구는 이 역량이 에이전트에 내재화되기 전의 과도기적 외부 보철물일
수 있다.
이 관점에서 진짜 승부처는 압축률이 아니라 "버린 정보가 필요해질 때 다시
가져오는 능력"이다.
압축이 아니라 점진적 공개(progressive disclosure) — 요약을 먼저 주고 필요할
때 원본을 펼치는 — 가 정답에 가깝다.

**측정 가능한 지표가 측정 불가능한 목적을 밀어내는 굿하트의 법칙.** 토큰 수는
측정하기 쉽고, 작업 성공률은 측정하기 어렵다.
도구가 토큰 절감을 지표로 내세우는 순간, 최적화는 측정 쉬운 쪽으로 쏠리고
측정 어려운 목적(정확도)은 조용히 희생된다.
웹 성능에서 "페이지 로드 시간"을 최적화하다 실제 사용자 체감을 놓친 사례,
코드 커버리지 100%를 좇다 의미 없는 테스트를 양산한 사례처럼, 측정 가능한
대리 지표(proxy metric)가 측정 불가능한 진짜 목적을 밀어낸 경우는 흔하다.
HN 토론은 이 측정 문제의 더 깊은 구조를 드러냈다.
한 댓글은 "AI를 더 좋게 만든다는 도구는 백만 개인데, AI가 더 잘 작동하는지
측정할 방법은 없다"며, 개별 개발자에게는 "전부 바이브(vibes)일 뿐"이라고
토로했다.[^trjordan]
또 다른 개발자는 압축 도구를 만드는 입장에서 "(하니스 × 모델 × MCP/CLI)
조합 때문에 LLM 도구에서 벤치마크를 만드는 것은 지나치게 어렵고 비싸다"고
인정했다.[^stephantul]
측정의 부재가 게으름이 아니라 구조적 난제라는 이 고백은, 굿하트의 법칙이 AI
도구 시장에서 왜 특히 끈질긴지를 설명한다.

**에이전트 비용의 진짜 해법은 압축이 아니라 "실행하지 않을 결정"에 있다.**
비용의 대부분이 출력이 아니라 파일 읽기·추론에 있다면, 비용 절감의
지렛대도 거기 있다.
출력을 압축하는 것은 이미 발생한 비용을 사후에 줄이는 것이지만, 진짜
절감은 "애초에 그 명령을 실행하지 않고, 그 파일을 읽지 않는" 사전 결정에서
온다.
느린 연산을 빠르게 만드는 것보다 그 연산을 아예 하지 않는 것이 언제나
우월하듯, 토큰을 압축하는 것보다 그 토큰을 생성하는 작업을 회피하는 것이
우월하다.
이 통찰은 rtk 같은 도구의 시장을 근본적으로 제한한다.
압축은 에이전트가 "과도하게 실행하고 과도하게 읽는" 현재의 비효율을 전제할
때만 가치가 있다.
에이전트의 계획 능력이 성숙해 실행 자체가 절제되면, 압축할 출력 자체가
줄어들어 이 도구 범주의 시장은 위에서 잠식된다.
한 댓글은 "필요한 것은 압축이 아니라 서브에이전트를 통한 공격적 컨텍스트
관리"라고 정리했고,[^SubiculumCode] 또 다른 사용자는 "전체 컨텍스트 압축"이
아니라 JSON의 장황함만 줄이는 더 현실적인 목표(toon-format)를 선호한다고
했다.[^old_sysadmin]
