## 요약

### 도구 개요와 하이브리드 검색

GBrain은 흩어진 마크다운 파일들을 검색 가능한 “개인 지식 브레인”으로 바꿔주는
오픈소스 도구다. 핵심은 Postgres + pgvector 위에 구축된 하이브리드 검색(벡터 +
키워드 + RRF 융합)이며, 단순한 시맨틱 검색 래퍼와 달리 다중 쿼리 확장(Claude
Haiku), 4단계 중복 제거, 그리고 “오래된 압축 진실(stale compiled truth)”
경고까지 결합되어 있다.

### 데이터 모델: 압축 진실과 타임라인

이 도구의 가장 독특한 지점은 데이터 모델이다. 모든 페이지는 “compiled truth(압축
진실) + timeline(타임라인)” 구조를 따른다. 상단의 압축 진실은 현재 시점의 가장
정제된 이해이며 새 증거가 들어오면 통째로 다시 쓰여진다. 하단의 타임라인은
append-only로 누적되며 절대 수정되지 않는, 증거 흔적이다. 이 패턴은 정보기관의
intelligence assessment 양식에서 차용한 것으로, “결론과 근거”를 물리적으로
분리해 보존한다.

### 에이전트 친화적 운영 모델

운영 모델은 명시적으로 에이전트 친화적이다. AI 에이전트(특히 OpenClaw)가 문서를
받으면, 언급된 모든 엔티티의 페이지를 갱신하고, 크로스 레퍼런스 링크를 생성하고,
타임라인 항목을 append한다. 사용자는 CLI를 직접 만질 일이 거의 없고 자연어로
에이전트에게 “오늘 회의 노트 ingest해줘” 같은 명령을 내린다. 핵심 지능은
바이너리 코드가 아니라 `skills/` 디렉토리의 두꺼운 마크다운 파일에
산다(“intelligence lives in fat markdown skills, not application code”).

### 기술 스택과 운영 비용

기술적으로는 Bun 기반 TypeScript 라이브러리로 시작해 CLI와 MCP 서버는 그 위의
얇은 래퍼다. BrainEngine 인터페이스로 백엔드를 교체할 수 있고(현재
PostgresEngine, SQLiteEngine은 설계만 존재), 청킹 전략은
recursive/semantic/LLM-guided 3가지를 콘텐츠 유형별로 디스패치한다. 약 7,500
페이지 규모의 브레인은 Supabase Pro ($25/월, 8GB)에서 약 750MB를 차지하고 초기
임베딩 비용은 $4-5 수준이다.

## 분석

### 두 가지 검색 패러다임의 정직한 결합

GBrain은 “벡터 검색 vs 키워드 검색” 논쟁에 대한 실용적 답을 제시한다. README는
명시적으로 두 접근의 한계를 인정한다. 키워드 검색은 “ignore conventional
wisdom”이라는 쿼리로 “The Bus Ticket Theory of Genius” 같은 정확히 그 주제를
다루는 글을 찾지 못한다. 반대로 벡터 검색은 정확한 인명/제품명 같은 고유 표현이
주변 텍스트에 희석되면 놓친다.

해법으로 RRF(Reciprocal Rank Fusion)를 채택한 것은 학술적으로 견고한 선택이다.
RRF는 `score = sum(1 / (60 + rank))` 형태로 서로 다른 랭킹 시스템의 결과를 점수
정규화 없이 결합할 수 있어, 2009년 Cormack 논문 이후 검색 시스템 ensemble의
표준에 가깝다. 여기에 Claude Haiku로 다중 쿼리 확장을 추가한 것은 임베딩 모델이
못 잡는 표현 다양성 문제를 LLM으로 보강하는 패턴이며, 최근 RAG 연구에서
HyDE(Hypothetical Document Embeddings)와 같은 계열의 기법이다.

### 압축 진실 + 타임라인이라는 데이터 모델

이 모델은 단순한 노트 앱과 결정적으로 갈라지는 지점이다. Obsidian, Notion,
Logseq는 모두 “한 페이지 = 하나의 가변 문서”라는 가정을 깐다. 새 정보가 들어오면
사용자가 직접 편집해 통합해야 하고, 시간이 지나면 어떤 부분이 언제 무엇 때문에
들어왔는지 추적할 수 없다.

GBrain은 이 두 관심사를 물리적으로 분리한다. 압축 진실은 “지금 시점에서 내가 이
주제에 대해 알고 있는 가장 정제된 형태”이고, 타임라인은 “그 이해에 이르기까지
들어온 모든 증거”다. 이는 정보기관 분석 양식의 핵심인 BLUF (Bottom Line Up
Front)와 evidence trail의 분리를 그대로 따른 것이고, 학술 논문의 abstract와
reference list 관계와도 유사하다. AI 에이전트가 자동으로 이 갱신을 수행할 때, 두
영역의 분리는 결정적이다. 에이전트가 압축 진실을 잘못 다시 써도 타임라인은
그대로 남아 있어 복구가 가능하다.

### 라이브러리 우선 설계의 함의

“CLI와 MCP 서버는 BrainEngine 위의 얇은 래퍼”라는 아키텍처 선언은 흔하지만
실제로 지키는 프로젝트는 드물다. GBrain은 모든 21개 MCP 도구가 CLI 명령을 그대로
미러링하고, “drift test”로 동일한 동작을 검증한다고 명시한다. 이는 이 도구가
처음부터 사람과 LLM을 동등한 사용자로 가정했음을 보여준다. LLM 에이전트가 부르는
도구와 사람이 두드리는 명령이 같은 구현을 공유하면, 사람이 디버깅한 것이
자동으로 에이전트 경로에도 반영된다.

### 스킬을 코드 밖에 두기

`skills/` 디렉토리의 마크다운 파일에 “에이전트가 어떻게 gbrain을 써야 하는가”에
관한 모든 정책이 들어있다는 점은 OpenClaw 생태계의 일관된 철학이다. ingest,
query, maintain, enrich, briefing, migrate, install — 7개의 스킬은 각각 도메인
전문성을 캡슐화하고 있고, 바이너리 업그레이드 없이 사용자가 직접 수정할 수 있다.
이 분리는 “프롬프트 엔지니어링은 소프트웨어 엔지니어링이 아니다”라는 흔한 주장에
대한 반례다. 스킬은 사실상 declarative한 정책 파일이며, 코드와 동일한 형상관리
대상이 된다.

## 비평

### 강점

#### 문제 정의가 정확하다

README의 “당신은 342개의 마크다운 파일이 여러 저장소에 흩어져 있다. GBrain이
이를 검색 가능하게 만든다”라는 첫 문장은 정확히 PKM(Personal Knowledge
Management) 도구를 진지하게 써본 사람이 겪는 문제다. Roam, Logseq, Obsidian
사용자들이 흔히 만나는 “노트는 많은데 찾을 수가 없다”라는 지점을 직격한다.

#### 기술 선택이 보수적이고 견고하다

Postgres + pgvector, HNSW 인덱스, RRF 융합, tsvector — 어느 하나도 실험적이지
않다. 모두 프로덕션에서 검증된 기술이다. 이는 “AI 시대의 새로운 데이터베이스”를
외치는 많은 벡터 DB 스타트업과 대비된다. $25/월 Supabase Pro에서 7,500 페이지가
750MB라는 구체적 견적은 신뢰감을 준다.

#### 라이브러리/CLI/MCP 분리가 깨끗하다

같은 엔진을 사람과 에이전트가 공유한다는 설계는 단순한 기술적 결정을 넘어, “AI
에이전트도 1급 사용자”라는 철학적 입장이다.

### 약점

#### 의존성 사슬이 길다

Supabase + OpenAI + Anthropic 3개의 외부 서비스가 모두 필요하다. 각각 없어도
일부 기능은 동작한다고 명시되어 있지만, 핵심 가치인 시맨틱 검색을 쓰려면
OpenAI가 필수고, 다중 쿼리 확장은 Anthropic이 필수다. “개인 지식 브레인”이라는
이름과 어울리지 않게 데이터 주권은 제한된다. SQLiteEngine은 “설계만
존재(designed)”라고 솔직히 적어두었지만, 이것이 미완성이라는 사실은 자체
호스팅을 원하는 사용자에게 큰 마이너스다.

#### 압축 진실 모델의 운영 비용을 과소평가했다

“AI 에이전트가 자동으로 갱신한다”는 약속은 매력적이지만, 실제로 LLM이 압축
진실을 통째로 다시 쓰는 작업은 환각, 정보 손실, 톤 드리프트를 동반한다. 7,500
페이지 규모에서 매번 ingest마다 여러 페이지의 압축 진실이 다시 쓰여지면,
사용자가 검토하지 않는 한 점진적 열화가 누적될 수 있다. README는 이 위험을 거의
다루지 않는다. `page_versions` 테이블이 스냅샷을 유지하긴 하지만, 잘못된
재작성을 감지하는 메커니즘은 명시되지 않았다.

#### “OpenClaw 권장” 설치가 락인을 만든다

README에서 가장 자세히 설명된 설치 방식이 OpenClaw를 통한 것이고, 7개의 스킬도
OpenClaw 형식이다. 다른 에이전트 프레임워크나 순수 CLI 사용자는 사실상 2등
시민이다. MCP 서버를 통해 Claude Code, Cursor에서 쓸 수 있다고는 하지만, 7개
스킬에 인코딩된 운영 지식을 어떻게 옮길 것인지에 관한 가이드는 부족하다.

#### 평가 방법이 없다

“하이브리드 검색이 실제로 동작한다(that actually works)”는 README의 첫
문장이지만, BEIR, MTEB 같은 벤치마크 결과나 자체 측정한 정밀도/재현율 수치는
어디에도 없다. 4단계 중복 제거의 임계값(0.85 cosine similarity, 60% type
diversity cap)은 어떻게 결정되었는가? 다중 쿼리 확장이 정말 도움이 되는가,
아니면 단지 비용만 늘리는가? 이런 질문에 답할 데이터가 없다.

#### “compiled truth”라는 용어가 무겁다

정보기관 어휘를 가져온 것은 흥미롭지만, 일반 사용자에게는 위압적이고 한국어로
번역하기도 어렵다. “압축 진실”이라는 표현 자체가 인식론적으로 강한 주장을 담고
있어, 사용자가 자신의 노트를 그 정도로 정제된 형태로 유지해야 한다는 부담을 줄
수 있다.

## 인사이트

### 1. RAG의 정직한 종착점은 “하이브리드 + 평가 도구”다

지난 2년간의 RAG 담론은 벡터 데이터베이스 회사들의 마케팅에 크게 휘둘렸다.
Pinecone, Weaviate, Qdrant 같은 전용 벡터 DB는 “임베딩만 잘 저장하면 시맨틱
검색이 마법처럼 동작한다”는 인상을 심었다. 그러나 실제 production RAG 시스템을
운영해본 팀들은 거의 예외 없이 같은 결론에 도달했다. 벡터 검색만으로는 부족하다,
키워드 검색을 합쳐야 한다, 그리고 그 둘을 합치는 방법이 결정적으로 중요하다.

GBrain은 이 학습을 그대로 코드로 옮긴 사례다. 흥미로운 것은 이 결론이 새로운
벡터 DB의 등장이 아니라 `Postgres + pgvector + tsvector`라는 “지루한” 스택의
부활로 이어졌다는 점이다. Supabase가 pgvector 지원을 강화한 것, ParadeDB가
BM25를 Postgres extension으로 제공하는 것, Lantern이 pgvector를 포크한 것 — 모두
같은 흐름이다. 검색 인프라는 새 것이 아니라 오래된 것이 정답이었다.

여기서 더 중요한 2차 효과는, 이제 RAG의 차별화 포인트가 “어떤 벡터 DB를
쓰는가”에서 “어떻게 평가하고 튜닝하는가”로 이동했다는 것이다. GBrain의 가장 큰
빈틈은 바로 이 평가 도구의 부재다. 다음 세대의 PKM/RAG 도구는 RRF 임계값, 청킹
전략, 다중 쿼리 확장 효과를 사용자 데이터에 대해 직접 측정하는 도구를 내장하게
될 것이다. 검색 품질은 더 이상 “마법”이 아니라 “엔지니어링 측정 대상”이다.

### 2. 압축 진실 + 타임라인은 LLM 시대의 데이터 모델 원형이다

전통적인 PKM 도구들은 한 페이지를 “사용자가 손으로 편집하는 하나의 가변 문서”로
가정해왔다. 이 가정은 LLM 시대에 깨진다. LLM은 페이지를 자동으로 갱신할 수 있고,
그 갱신이 정확한지 검증해야 하며, 잘못되었을 때 복구할 수 있어야 한다. 즉,
“지금의 정답”과 “그 정답에 이르기까지의 증거”를 분리해야 한다.

이 분리는 사실 새로운 발상이 아니다. 정보기관의 intelligence assessment,
위키피디아의 본문/토론 페이지, 학술 논문의 abstract/reference, 심지어 이벤트
소싱(event sourcing)에서의 state/event 분리까지 — 모두 같은 패턴이다. 가변
상태와 불변 이벤트를 분리하면, 가변 상태는 자유롭게 다시 계산할 수 있고, 불변
이벤트는 영구적 진실의 원천이 된다.

LLM 에이전트가 데이터를 쓰기 시작하면 이 패턴은 선택이 아니라 필수가 된다.
에이전트가 압축 진실을 잘못 써도 타임라인이 살아 있으면 복구할 수 있다. 반대로
두 개를 섞어 하나의 마크다운 파일에 넣으면, 한 번의 잘못된 자동 편집이 전체를
망가뜨린다. 향후 1-2년 안에 Notion, Obsidian 같은 기존 도구들이 “AI가 쓸 수 있는
노트” 모드를 도입할 때, 그 내부 구조는 거의 확실히 GBrain과 유사한 형태일
것이다. 이 패턴을 가장 먼저 명시적으로 코드화했다는 점이 GBrain의 가장 큰
기여다.

### 3. 스킬을 코드 밖에 두는 것은 새로운 추상화 경계다

소프트웨어 엔지니어링에는 전통적으로 “코드”와 “설정(config)” 이라는 두 가지
추상화 경계가 있다. 코드는 정적으로 컴파일/검증 되고, 설정은 런타임에 읽힌다.
LLM 에이전트의 등장은 여기에 세 번째 경계를 추가한다. “스킬”이다. 스킬은
마크다운으로 쓰여진 자연어 정책 문서이며, LLM이 런타임에 읽고 따른다.

GBrain이 “intelligence lives in fat markdown skills, not application code”라고
선언한 것은 이 새 경계를 가장 명확하게 주장한 사례 중 하나다. 7개의 스킬
파일(ingest, query, maintain, enrich, briefing, migrate, install)은 각각 도메인
전문성을 캡슐화하고, 바이너리 업그레이드 없이 수정 가능하며, git으로 버전
관리된다. 이는 사실상 “정책의 핫 리로드”이며, 전통적 소프트웨어에서는 거의
불가능했던 운영 유연성이다.

이 패턴의 2차 효과는 도구 회사의 비즈니스 모델 변화다. 전통적 SaaS는 “기능”을
팔았다. 스킬 기반 도구는 “스킬 라이브러리” 를 판다. CLI 바이너리는 무료로 두고,
도메인별 스킬 모음 (예: VC 투자 워크플로 스킬, 변호사 사건 관리 스킬)을 유료로
파는 모델이 가능해진다. GBrain의 ClawHub 패키지 매니저 언급은 이 방향을
암시한다. 향후 등장할 LLM 도구들은 점점 더 “엔진은 오픈소스, 스킬은
마켓플레이스”라는 구조로 분화될 것이다. Stripe API가 결제 인프라를 표준화하고 그
위에 무수한 비즈니스 로직이 쌓였듯이, 스킬은 LLM 시대의 “비즈니스 로직 레이어”가
된다.

### 4. PKM의 진짜 병목은 저장이 아니라 인입이다

대부분의 PKM 도구는 “어떻게 저장하고 어떻게 검색하느냐”에 집착해왔다. 그러나
실제로 PKM이 실패하는 가장 흔한 이유는 사용자가 새 정보를 시스템에 넣는 것
자체가 너무 무겁기 때문이다. 회의가 끝나고, 기사를 읽고, 슬랙 대화를 끝낸 직후 —
그 지식을 정제된 형태로 노트에 옮기는 것은 5분이면 충분하지만, 사람들은 그 5분을
거의 항상 미루고, 결국 영영 옮기지 않는다.

GBrain의 “AI 에이전트가 ingest를 담당한다”는 약속은 이 병목을 정면으로 겨냥한다.
ingest 스킬의 설명이 “compiled truth를 갱신하고(rewrite, not append), timeline에
append하고, 모든 언급된 엔티티의 cross-reference link를 만든다”인 것은
의미심장하다. 이는 사람이 손으로 했다면 한 시간이 걸릴 작업이다. 에이전트가
자동으로 한다면 같은 작업이 30초 안에 끝난다.

이 변화의 함의는 PKM의 사용 패턴을 근본적으로 바꾼다는 점이다. 전통적 PKM은
“내가 아는 것을 정리한다”였다. 에이전트 ingest 모델은 “내가 마주친 모든 것을
던지고, 시스템이 알아서 연결한다”가 된다. 두 모델의 차이는 결과적으로 브레인의
크기를 한 자릿수 이상 키운다. 7,500 페이지가 가능한 것은 사람이 7,500개를 손으로
작성해서가 아니라, 자동 ingest로 누적되었기 때문이다. 다음 세대 지식 노동자의
차이는 “얼마나 많이 알고 있느냐”가 아니라 “얼마나 잘 외부화된 브레인을
운영하느냐”에서 갈릴 가능성이 높다. GBrain은 그 미래를 미리 보여주는 도구다.

### 5. “사람과 LLM이 같은 도구를 공유한다”는 설계 원칙

GBrain의 가장 조용하지만 결정적인 결정은 “MCP 도구와 CLI 명령이 동일한 구현을
공유한다”는 점이다. 21개의 MCP 도구가 21개의 CLI 명령과 1:1로 매칭되고, drift
test가 둘의 동등성을 검증한다. 이는 “사람용 인터페이스”와 “에이전트용
인터페이스” 를 분리하지 않는다는 강한 입장이다.

이 원칙의 함의는 깊다. 첫째, 사람이 디버깅한 것이 자동으로 에이전트에도
적용된다. 사람이 `gbrain query “...”`로 실행해 이상한 결과가 나오면, 그것은 곧
에이전트의 query 도구에서도 이상한 결과가 나옴을 의미한다. 둘째, 에이전트의
동작이 재현 가능하고 검사 가능해진다. 에이전트가 무언가를 했다면, 같은 일을
사람이 CLI로 재현할 수 있다. 셋째, 새 기능 추가가 한 번에 양쪽에 반영된다.

이 원칙이 일반화된다면, “사람용 UI”와 “API”의 전통적 분리는 점점 무의미해질
것이다. 둘 다 같은 도구의 표면일 뿐이다. 이는 1990년대 Unix 철학(“도구는 작고,
합성 가능하고, 텍스트로 대화한다”)의 LLM 시대 부활이다. Unix 명령어가 사람과
스크립트를 동시에 섬겼던 것처럼, LLM 시대의 도구는 사람과 에이전트를 동시에
섬긴다. 이 패턴을 받아들이지 못하는 도구 회사들은 사람용 SaaS와 LLM용 API를 따로
만들다가 결국 둘 사이의 일관성을 잃을 것이다. GBrain은 이 함정을 처음부터
회피했다.
