# Hermes Agent — 경험으로부터 스킬을 생성하는 자기 학습형 에이전트

<https://hermes-agent.nousresearch.com/>

<https://github.com/nousresearch/hermes-agent>

HN 토론: <https://news.ycombinator.com/item?id=48586005> (122점, 105개 댓글)

## 요약

Nous Research가 개발한 Hermes Agent는
서버에 상주하며 경험으로부터 스킬을 생성하고 개선하는 자율 AI 에이전트다.
코딩 코파일럿이나 챗봇 래퍼가 아니라,
실행할수록 더 유능해지는 지속형 시스템을 지향한다.

주요 특징은 다음과 같다.

- **자기 학습 루프:** 문제 해결 과정에서 스킬을 자동 생성하고 개선한다.
- **영속 메모리:** Honcho 기반, FTS5 + LLM 요약으로 세션 간 기억을 유지한다.
- **멀티 플랫폼 게이트웨이:** CLI, Telegram, Discord, Slack, WhatsApp, Signal, Email을 지원한다.
- **40개 이상 내장 도구:** 웹 검색, 터미널, 브라우저, 비전, 이미지 생성, TTS 등을 포함한다.
- **실행 환경:** 로컬, Docker, SSH, Modal, Singularity를 지원한다.
- **200개 이상 LLM 모델 호환:** Nous Portal, OpenRouter, OpenAI 호환 API를 통해 연결한다.
- **MIT 라이선스 오픈소스**로 공개되어 있다.

저장소는 2025년 7월에 만들어졌다.
1년이 채 안 되는 기간에 GitHub 스타 21만 8천 개,
포크 3만 9천 개를 넘어섰고, 열린 이슈만 2만 7천 개가 넘는다.
경쟁 플랫폼인 OpenClaw로부터의 자동 마이그레이션 도구를 제공하는데,
이는 후발 주자가 선발 주자의 사용자 기반을 직접 흡수하려는
에이전트 플랫폼 간 경쟁이 격화되고 있음을 보여준다.

## 분석

### “스킬 생성”은 도구 사용을 메타 학습으로 끌어올리려는 시도다

Hermes Agent의 핵심 차별점은 스킬 생성이라는 메타 학습 능력이다.
일반적인 에이전트가 주어진 도구를 그때그때 사용하는 데 그치는 반면,
Hermes Agent는 반복되는 작업 패턴을 스킬로 추상화해
다음 실행에서 더 효율적으로 동작하는 것을 목표로 삼는다.

이는 Claude Code의 스킬 시스템과 표면적으로 유사하지만 방향이 정반대다.
Claude Code에서 스킬은 사용자가 직접 작성하거나
커뮤니티에서 가져오는 정적 자산이다.
반면 Hermes Agent에서 스킬은 에이전트가 자기 실행 이력으로부터
동적으로 생성하는 산출물이라고 소개된다.
사람이 매뉴얼을 써서 도구에 지식을 넣어주는 모델에서,
도구가 스스로 매뉴얼을 쓰는 모델로 방향을 뒤집으려는 시도인 셈이다.
다만 이 뒤집기가 실제로 검증 가능한 형태로 구현되어 있는지는
소개 페이지의 설명만으로는 확인하기 어렵다.

### 멀티 플랫폼 게이트웨이는 접근성과 표면적 취약성을 동시에 넓힌다

CLI, Telegram, Discord, Slack, WhatsApp, Signal, Email까지
7개 채널을 하나의 에이전트로 묶는 설계는 접근성을 극적으로 넓힌다.
Telegram에서 시작한 대화를 Slack에서 이어갈 수 있다는 것은
에이전트가 특정 인터페이스에 종속되지 않는다는 의미이며,
사용자가 이미 쓰고 있는 메신저 습관 위에 에이전트를 얹을 수 있게 한다.

동시에 이 구조는 공격 표면도 그만큼 넓힌다.
채널이 늘어날수록 인증, 권한 위임, 메시지 검증 로직이
채널별로 별도로 필요해지고, 그중 하나만 허술해도
전체 시스템의 신뢰 경계가 흔들린다.
실제로 HN 댓글에서는 소개 페이지가 다루지 않는
백그라운드 네트워크 동작에 대한 우려가 제기됐다.
qingcharles는 Hermes Agent를 포함한 여러 에이전트의 기본 설치본이
사용자 동의 없이 웹 트래픽을 Parallel이라는 서드파티로
조용히 라우팅한다는 Reddit 토론을 링크하며 문제를 제기했다[^qingcharles].
멀티 플랫폼 게이트웨이가 얼마나 매끄럽게 작동하는지와,
그 이면에서 트래픽이 어디로 흘러가는지는 완전히 다른 질문이며,
전자의 매끄러움이 후자에 대한 신뢰를 자동으로 보장하지 않는다.

### GeekNews와 HN 모두에서 반복되는 질문은 “왜 이걸 써야 하는가”다

GeekNews 댓글(theogn)이 지적했듯,
OpenClaw, Ductor, CC-Connect 등 유사 에이전트 플랫폼과의
실질적 차별점이 사용자에게 명확히 전달되지 않고 있다.
기능 목록이 아닌 사용 시나리오로 포지셔닝할 필요가 있다는 지적이다.

HN에서는 이 회의가 훨씬 구체적인 형태로 반복됐다.
andy_xor_andrew는 기본 로드아웃에서 스킬 설정 마법사로
불필요한 스킬을 잔뜩 꺼도 여전히 쓸모없는 스킬 목록이 남아,
그것만 나열하는 데 1만 토큰이 넘는 컨텍스트가 소모됐다고 지적했다.
그는 ACP(Agent Client Protocol)를 지원하는 harness를 직접 만들어
Copilot CLI, OpenClaw 등 어떤 코딩 CLI와도 연동되게 했더니
몇 시간 만에 Hermes Agent와 비슷한 결과를 얻었다며,
결국 “최소한의 코딩 에이전트 코어 + 게이트웨이” 조합이
더 낫다는 인상을 받았다고 말했다[^andy_xor_andrew].
isoprophlex 역시 왜 Codex나 Claude Code, OpenCode를
그냥 Docker 컨테이너에 넣어 쓰지 않는지 이해가 안 된다며
비슷한 의문을 제기했다[^isoprophlex].
두 댓글 모두 Hermes Agent가 제공하는 부가 기능이
“직접 조립하는 것보다 얼마나 나은가”라는 질문에
설득력 있는 답을 주지 못하고 있음을 시사한다.

## 비평

### “표절 논란은 없다”는 반박 자체가 검증 부담을 사용자에게 떠넘긴다

이 프로젝트를 둘러싼 가장 첨예한 논쟁은 표절 의혹이다.
cassianoleal은 Hermes를 둘러싼 “표절 소동”을 알아둘 필요가 있다며
관련 HN 스레드를 링크했다[^cassianoleal].
그 스레드에서 Nous Research 측 관계자로 보이는 segmondy는
“소동이랄 것도 없다”며, 메모리·스킬·진화형 에셋 설계는
이미 2년 넘게 업계 표준 관행이었다고 반박했다[^segmondy].
문제는 이 반박이 구체적인 코드 비교나 커밋 이력 검증이 아니라
“업계에서 흔한 패턴이었다”는 일반론에 머무른다는 점이다.
“어디서나 쓰는 패턴”이라는 방어는 표절이 아니라는 증거가 아니라,
표절 여부를 판단하기 어렵게 만드는 논점 흐리기에 가깝다.
게다가 이 프로젝트는 GitHub 이슈에서 표절 관련 언급을
직접 수정한 이력이 있다는 점도 별도로 보고된 바 있어,
투명한 해명보다 논란 자체를 조용히 지우려는 태도로 비칠 여지가 있다.
사용자는 결국 스스로 원본 프로젝트와 코드를 비교해
진위를 판단해야 하는 부담을 떠안는다.

### “실행할수록 똑똑해진다”는 비전은 검증 메커니즘이 빠진 채로 제시된다

자기 학습 루프가 매력적으로 들리는 것과, 그것이 실제로
품질을 개선하는 방향으로만 작동한다는 것은 다른 문제다.
자동 생성된 스킬이 옳은지, 안전한지, 이전 스킬과 충돌하지 않는지를
검증하는 메커니즘이 소개 페이지 어디에도 구체적으로 설명되어 있지 않다.
잘못된 패턴이 한 번 스킬로 고착되면, 이후 모든 실행에서
그 오류가 반복·증폭될 위험이 있다.
사람이 검토 없이 스스로 규칙을 만들고 스스로 따르는 시스템은
편향과 실수를 흡수하는 구조가 아니라 오히려 증폭하는 구조가 될 수 있으며,
이 위험을 완화하는 구체적 장치가 무엇인지가
“실행할수록 똑똑해진다”는 슬로건만으로는 드러나지 않는다.

### 시장 규모와 수익 구조에 대한 질문에 프로젝트는 아직 답하지 않는다

HN에서는 이런 자율 에이전트 플랫폼 전반의
실질적 시장 규모에 의문을 제기하는 목소리가 이어졌다.
bfeynman은 이런 도구를 실제로 쓰는 사람은
결국 얼리어답터층으로 한정된 것 아니냐고 반문하며,
겉보기의 복잡한 기능도 뜯어보면 “SQLite 파사드를 씌운
LLM 호출 모음”에 불과한데 그 편의성 이상의 가치를
소비자가 체감하기 어렵다고 지적했다[^bfeynman].
rvz는 한 걸음 더 나아가, 사용자가 수천 달러를 쓰는 동안
실제로 돈을 버는 쪽은 호스팅 사업자뿐이고
정작 사용자에게 돌아가는 경제적 이득은 불분명하다고 짚었다[^rvz].
21만 개가 넘는 스타 수는 관심의 크기를 보여주지만,
그 관심이 지속적인 실사용으로, 나아가 지속 가능한 수익 모델로
이어지는지는 완전히 다른 질문이며 아직 답이 없다.
__mharrison__이 코드베이스에 `pyproject.toml` 없이
구식 `setup.py`만 있는 것을 보고 신뢰를 주저하게 됐다고 밝힌 것도[^mharrison]
같은 맥락에서 읽을 수 있다.
바이럴한 관심에 비해 프로젝트 기초 체력이
그에 걸맞게 다져져 있는지에 대한 의문이다.

## 인사이트

### 도구가 사용자에게 적응하는 모델은 “설정”이라는 개념 자체를 지운다

대부분의 소프트웨어는 사용자가 학습 곡선을 올라가야 하는 구조다.
사용자가 도구의 개념 모델을 배우고, 그 모델에 맞춰 자신의 작업을 재구성한다.
Hermes Agent의 자기 학습 루프가 표방하는 방향은 이 관계를 뒤집는 것이다.
도구가 사용 패턴을 관찰해 스스로 적응한다면,
“환경 설정”이라는 행위 자체가 별도의 단계로 존재할 필요가 없어진다.
이 방향이 실제로 성립한다면 파급력은 이 프로젝트 하나에 그치지 않는다.
지금까지 소프트웨어 산업이 온보딩, 튜토리얼, 설정 마법사에
쏟아온 노력 전체가 다른 종류의 문제, 즉
“에이전트가 사용자의 암묵적 신호를 얼마나 정확히 읽어내는가”라는
문제로 대체될 수 있다.
다만 이 재편이 일어나려면 자기 학습이 실제로 안정적으로
검증 가능해야 하는데, 비평에서 지적했듯 그 검증 장치는
아직 공개된 형태로 확인되지 않는다.

### 난립하는 에이전트 플랫폼들은 안드로이드 초기의 파편화를 재연하고 있다

OpenClaw, Ductor, CC-Connect, Hermes Agent 등
유사한 기능을 내세우는 에이전트 플랫폼이 동시다발적으로 등장하는 현재 상황은
2008~2010년 안드로이드 생태계 초기와 구조적으로 닮았다.
당시에도 제조사마다 다른 UI 스킨과 앱 스토어를 내놓았지만,
결국 승부를 가른 것은 개별 기기의 기능 목록이 아니라
개발자 생태계와 배포 채널의 규모였다.
지금의 에이전트 플랫폼 경쟁도 비슷한 궤적을 밟을 가능성이 크다.
기능 목록으로는 차별화가 점점 불가능해지고,
결국 스킬 마켓플레이스, 커뮤니티, 통합 생태계의 크기가 승자를 결정할 것이다.
agentskills.io라는 오픈 포맷의 등장은 이 생태계 전쟁의 첫 수로 볼 수 있는데,
역사적으로 이런 초기 표준화 시도는 최종 승자가 아니라
승자가 흡수하거나 무력화하는 대상이 되는 경우가 더 많았다는 점도
함께 기억할 필요가 있다.

### 표절 논란과 트래픽 라우팅 논란은 같은 구조적 취약점에서 나온다

표절 의혹과 서드파티로의 트래픽 라우팅 문제는
겉보기에는 무관한 별개의 사건처럼 보이지만, 사실 같은 근본 원인을 공유한다.
바이럴하게 성장하는 오픈소스 에이전트 프로젝트는
빠른 기능 출시와 사용자 획득을 최우선 순위로 두는 경향이 있고,
그 과정에서 코드 출처 검증이나 기본 설정의 프라이버시 영향 검토 같은
느리고 눈에 띄지 않는 작업은 뒤로 밀리기 쉽다.
21만 개의 스타와 2만 7천 개의 열린 이슈라는 숫자의 대비는
이 우선순위 구조를 그대로 드러낸다.
관심과 채택은 폭발적으로 늘었지만, 그 관심을 검증하고
정리하는 속도는 따라가지 못하고 있다.
사용자가 결과물의 매끄러움에만 주목하고
그 이면의 데이터 흐름과 코드 출처를 검증하지 않는 소비 패턴이 계속되는 한,
비슷한 성격의 논란은 이 프로젝트뿐 아니라
빠르게 성장하는 다른 에이전트 플랫폼에서도 반복될 가능성이 크다.

### 플랫폼이 아니라 워크플로 습관이 더 오래가는 자산이다

HN에서 jacobgold는 OpenClaw에서 Hermes Agent를 거쳐
결국 자기만의 도구로 정착했다며,
코드 리뷰·이메일 필터링·웹사이트 모니터링·알림 취합처럼
자신이 실제로 자동화하는 작업 목록을 구체적으로 제시했다[^jacobgold].
주목할 점은 그가 플랫폼을 세 번이나 갈아탔음에도
자동화 대상 목록 자체는 거의 바뀌지 않았다는 사실이다.
이는 이런 플랫폼의 진짜 가치가 에이전트 자체의 지능이 아니라,
사용자가 이미 갖고 있던 반복 작업 습관을
얼마나 매끄럽게 얹을 수 있느냐에 달려 있음을 보여준다.
바꿔 말하면, 사용자에게 남는 진짜 자산은
특정 플랫폼에 대한 숙련도가 아니라
“무엇을 자동화할지 아는 습관” 그 자체다.
플랫폼은 언제든 교체될 수 있지만,
이 습관은 플랫폼이 바뀌어도 그대로 이전된다.
에이전트 플랫폼 간 경쟁이 격화될수록,
정작 오래 살아남는 것은 특정 도구의 브랜드가 아니라
사용자 각자가 축적한 자동화 목록이라는 역설이 여기서 드러난다.
