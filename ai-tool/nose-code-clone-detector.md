# nose: 제한적 Type-4 코드 클론 탐지기

<https://github.com/corca-ai/nose>

## 소개

nose는 corca-ai가 Rust로 작성한 코드 클론 탐지기다.
기존 도구들이 텍스트 유사도나 AST 구조 비교에 머무는 데 반해,
nose는 “의미적 동치(semantic equivalence)”를 탐지 대상으로 삼는다.
즉, 코드가 겉으로 얼마나 달라 보이더라도 동일한 동작을 수행한다면 클론으로 판정한다.

핵심 주장은 “동일한 핑거프린트 ⟹ 동일한 동작, 결코 거짓 동치 없음”이다.
이 주장을 형식적으로 보증하기 위해 Lean 4 기반 증명을 사용한다.
증명이 없는 의미적 동치는 플래그하지 않는다는 fail-closed 의미론을 채택한다.

지원 언어는 Python, JavaScript, TypeScript, Go, Rust, Java, C, Ruby, Swift이며
CSS(계산된 스타일 동치), HTML(렌더링된 DOM 동치), Vue/Svelte 컴포넌트 인식도 포함한다.
언어 조합(language composition)은 81.5% Rust, 16.4% Python, 1.1% Lean이며,
현재 pre-1.0 상태다.

## 클론 분류

Roy, Cordy & Koschke(2009)의 클론 분류법을 기반으로 한다.

- **Type-1**: 완전히 동일한 코드 (주석, 공백 무시)
- **Type-2**: 식별자, 리터럴만 다른 코드
- **Type-3**: 문장의 추가, 삭제, 변경이 있는 근사 복제
- **Type-4**: 구문은 달라도 의미가 동치인 코드

nose는 이 중 “제한적(bounded) Type-4”를 탐지한다.
모든 의미적 동치를 탐지하겠다는 주장 대신,
증명 가능한 범위 내에서만 판정하는 방식을 택한다.

세 가지 탐지 채널을 제공한다.

- `syntax` 채널: Type-1/2 (AST 기반 구조 비교)
- `semantic` 채널: Type-4 (형식 증명 기반 의미적 동치)
- `near` 채널: Type-3 (근사 복제)

기본 실행은 세 채널을 모두 활성화한다.

## 팩 아키텍처

의미적 동치 판정의 근거는 “팩(pack)” 시스템이다.
팩은 특정 영역의 의미 규칙을 담은 독립 유닛이다.

- **언어 팩**: 언어별 의미 규칙 (예: JavaScript의 `Array.prototype.flat`)
- **표준 라이브러리 팩**: 표준 API의 동치 관계
- **라이브러리 팩**: 서드파티 라이브러리 패턴
- **프로토콜 팩**: HTTP, REST 등 통신 패턴
- **법률 팩**: 계약 로직, 규제 패턴

각 팩의 형식 증명 의무는 `formal/obligations/` 디렉토리에서 관리된다.
각 항목은 Lean 4 증명 파일과 `meta.toml`을 가지며,
상태는 `proven`, `covered`, `missing`, `empirical-only`, `rejected-counterexample`로 구분된다.

## CLI

```bash
# 현재 디렉토리 클론 탐색
nose query src

# 특정 클론의 전체 세부 정보 출력
nose query src id=<id> full

# main 브랜치 대비 신규 클론만 탐색
nose query src base=origin/main

# JSON 형식으로 출력
nose query src --format json
```

CI 통합을 위한 플래그:

```bash
nose query src --fail-on any      # 클론 존재 시 CI 실패
nose query src --fail-on new      # 신규 클론만 실패 처리
nose query src --baseline          # 현재 상태를 베이스라인으로 저장
nose query src --format sarif      # SARIF 형식 출력 (GitHub Code Scanning 호환)
nose query src --cache-dir .nose   # 캐시 디렉토리 지정
```

결과는 “리팩터링 가치” 기준으로 정렬되며, 추출 스켈레톤(extraction skeleton)을 함께 출력한다.
단일 Rust 바이너리로 제공되며 런타임 의존성과 네트워크 접근 없이 작동한다.

## 분석

### “증명 가능한 것만 플래그한다”는 설계 결정의 구조적 의미

nose의 핵심 설계 결정은 recall(재현율)을 희생하고 precision(정밀도)을 보장하는 것이다.
기존 클론 탐지기들은 휴리스틱 기반 판정으로 false positive를 생산하여
개발자가 직접 판단해야 하는 노이즈를 만든다.
nose는 이 구조를 뒤집는다.
플래그된 것은 반드시 클론이지만, 클론인 것이 반드시 플래그되지는 않는다.

이 트레이드오프가 실용적으로 의미 있는 이유는 클론 탐지의 실제 비용 구조에 있다.
false positive는 개발자의 주의를 빼앗고 도구 자체에 대한 신뢰를 훼손한다.
false negative는 기술 부채를 축적하지만 즉각적인 비용이 없다.
CI 파이프라인에 통합되는 도구라면 false positive의 비용이 훨씬 크다.
이 관점에서 fail-closed 설계는 소음 없는 CI 통합을 위한 합리적 선택이다.

### 팩 시스템은 “의미론의 DefinitelyTyped”를 지향한다

TypeScript 생태계에서 DefinitelyTyped는 커뮤니티가 타입 정의를 기여하는 집합소다.
nose의 팩 시스템은 이 구조를 의미 증명 영역으로 가져온다.
언어별, 라이브러리별, 도메인별 의미 규칙을 누구나 기여할 수 있는 팩으로 표현한다.

이 모델의 강점은 확장성이다.
nose 팀이 모든 언어와 라이브러리의 의미 규칙을 선점할 필요가 없다.
각 생태계의 전문가가 자신의 영역에서 팩을 기여하면 도구의 탐지 범위가 넓어진다.
DefinitelyTyped가 타입스크립트 도입을 가속했듯, 팩 생태계가 성숙하면
nose의 실용적 커버리지는 핵심 팀의 역량과 무관하게 성장할 수 있다.

### 크로스 언어 탐지는 코딩 에이전트 시대의 새로운 문제를 겨냥한다

nose는 Python으로 구현된 알고리즘과 JavaScript로 구현된 알고리즘이 의미적으로 동치임을
탐지할 수 있다고 명시한다.
단일 언어 코드베이스에서 이는 부차적 기능이다.
그러나 코딩 에이전트가 여러 언어로 동시에 코드를 생성하는 환경에서
크로스 언어 클론은 새로운 기술 부채 유형이 된다.

에이전트는 컨텍스트 없이 기존 구현과 유사한 로직을 다른 언어로 재구현하는 경향이 있다.
이 중복은 단일 언어 클론 탐지기로는 보이지 않는다.
nose가 크로스 언어 탐지를 명시적으로 지원하는 것은
코드 생성 도구의 부산물을 관리하려는 의도적 포지셔닝이다.

## 비평

### “제한적” 수식어는 탐지의 핵심 어려움을 회피한다

“bounded Type-4”라는 표현은 전략적 후퇴다.
Type-4 클론의 실용적 의미는 “같은 결과를 내는 다른 구현”인데,
가장 흔하고 리팩터링 가치가 높은 경우는 정렬 알고리즘의 변형,
캐싱 전략의 차이, 재귀 대 반복 구조의 전환이다.
nose는 이 경우들을 탐지하지 않는다고 명시한다.
버블 소트와 퀵 소트는 같은 결과를 내지만 nose의 탐지 범위 밖이다.

이는 마케팅 언어와 실제 기능 사이의 간격이다.
“의미적 동치 탐지”라는 표현은 알고리즘 수준의 동치를 포함한다고 이해되기 쉽다.
그러나 실제 탐지 범위는 동일한 알고리즘이 다른 구문으로 표현된 경우,
즉 코드 스타일의 차이에 가깝다.
이 구별은 README에서 충분히 강조되지 않는다.

### 형식 증명의 실제 커버리지가 공개되지 않는다

nose는 Lean 4 증명을 사용하며 `formal/obligations/`에서 상태를 추적한다고 명시한다.
그러나 현재 몇 개의 증명이 `proven` 상태이고 몇 개가 `missing`이나 `empirical-only`인지
외부에서 확인할 수 없다.
pre-1.0 상태임을 감안하더라도, 형식 증명 커버리지 없이는
“결코 거짓 동치 없음”이라는 주장의 실제 적용 범위를 평가할 수 없다.

실용적 문제는 사용자가 도구를 신뢰할 근거가 투명하지 않다는 것이다.
증명된 규칙 집합의 크기는 도구의 실질적 능력을 결정한다.
100개 규칙 중 3개만 증명되었다면, 나머지 97개는 휴리스틱 수준의 판정이 된다.
이 수치 없이는 “형식적으로 안전하다”는 클레임이 마케팅 문구에 그친다.

### 미검증 메서드 호출 처리 방식은 false positive 경로를 만든다

nose의 알려진 한계 중 “미검증 메서드 호출은 이름만으로 불투명한 동치를 공유한다”는 항목이 있다.
이는 서로 다른 수신자 타입의 같은 이름 메서드를 동치로 취급할 수 있다는 뜻이다.
예를 들어 두 클래스의 `.process()` 메서드가 완전히 다른 동작을 하더라도
이름이 같으면 클론 후보로 묶일 수 있다.

이 경우는 “결코 거짓 동치 없음”이라는 핵심 주장의 예외다.
주장은 “핑거프린트가 같으면 동작이 같다”지만,
미검증 메서드에서는 핑거프린트 계산 자체가 이름 기반 추정에 의존한다.
fail-closed 설계의 보증이 뚫리는 지점이 이미 존재한다는 의미다.

## 인사이트

### 형식 증명 인프라는 정적 분석 도구의 새로운 신뢰 기반이 된다

타입 시스템 이론에서 “건전성(soundness)”과 “완전성(completeness)”은 대립한다.
건전한 시스템은 잘못된 것을 승인하지 않지만, 일부 올바른 것을 거부할 수 있다.
완전한 시스템은 올바른 것을 모두 승인하지만, 잘못된 것도 통과시킬 수 있다.
nose는 명시적으로 건전성을 선택하고 완전성을 포기한다.

흥미로운 점은 이 선택이 Lean 4 증명 인프라로 뒷받침된다는 것이다.
기존 정적 분석 도구들은 건전성 주장을 휴리스틱과 경험적 검증으로 지지했다.
Lean 4를 통한 기계 검증은 이 주장을 형식적으로 검사 가능하게 만든다.
도구의 신뢰성을 “우리가 테스트했다”에서 “수학적으로 증명되었다”로 이동시키는 것이다.
이 패턴이 더 많은 개발 도구에 확산될 경우, 정적 분석의 신뢰 기준 자체가 바뀔 수 있다.

### 의미론 팩 생태계는 오픈소스 기여 구조를 재정의할 수 있다

DefinitelyTyped는 커뮤니티가 도구 자체를 확장하는 구조의 성공 사례다.
타입스크립트 팀은 핵심 언어를 개발하고, 커뮤니티는 타입 정의를 기여하여
도구의 실용적 커버리지를 몇 배로 늘렸다.

nose의 팩 시스템이 동일한 구조를 갖추려면 두 가지가 필요하다.
첫째, 팩을 작성하는 방법이 접근 가능해야 한다.
Lean 4 증명 작성은 전문 지식을 요구하므로, 진입 장벽이 DefinitelyTyped의 TypeScript보다 훨씬 높다.
둘째, 팩의 품질을 검증하는 프로세스가 있어야 한다.
오류 있는 팩은 도구 전체의 신뢰 주장을 훼손한다.
팩 생태계가 성숙할 수 있다면 언어나 프레임워크 전문가들이
자신의 영역에서 의미 규칙을 공식화하는 새로운 기여 유형이 생긴다.

### 코딩 에이전트는 클론 탐지의 수요를 구조적으로 바꾼다

전통적인 코드 클론은 사람이 비슷한 로직을 반복 구현하면서 생긴다.
이는 의식적이거나 기억 실수에서 비롯된다.
코딩 에이전트는 다른 패턴을 만든다.
에이전트는 컨텍스트 창에 들어오지 않는 기존 구현을 모르고 동일한 로직을 새로 작성한다.
각 에이전트 세션은 독립적이며, 세션 간 중복 인식이 없다.

이 구조에서 클론 탐지는 단순한 코드 품질 도구를 넘어 에이전트 출력의 품질 게이트가 된다.
에이전트가 PR을 만들 때 nose 검사를 CI에 통합하면,
에이전트가 모르고 재구현한 중복을 기계적으로 잡아낼 수 있다.
nose가 “코딩 에이전트를 위해 설계되었다”고 명시하는 것은
이 수요 변화를 인식한 포지셔닝이다.

### 탐지보다 추출이 더 어려운 문제다

nose는 클론 탐지와 함께 “리팩터링 가치 기준 정렬” 및 “추출 스켈레톤 출력”을 제공한다.
이 기능이 실질적 가치의 핵심이다.
클론을 탐지하는 것은 문제를 발견하는 것이고, 추출 스켈레톤은 해결 경로를 제안한다.

그러나 의미적 동치인 코드의 공통 추상화 추출은 탐지보다 훨씬 어려운 문제다.
두 함수가 같은 동작을 다른 방식으로 구현한다면,
하나를 다른 것으로 대체하는 것은 쉽지만
둘을 아우르는 새로운 추상화를 만드는 것은 설계 결정이 필요하다.
추출 스켈레톤이 이 결정을 자동화하는지, 아니면 구조적 힌트를 제공하는 수준인지는
pre-1.0 단계에서 아직 명확하지 않다.
이 기능이 얼마나 깊이 구현되는지가 nose의 실용적 가치를 결정할 것이다.
