# Next AI Draw.io

<https://github.com/DayuanJiang/next-ai-draw-io>

## 소개

next-ai-draw-io는 자연어 대화만으로 draw.io 다이어그램을 생성하고 수정하는
Next.js 기반 웹 애플리케이션이다.
사용자가 채팅창에 원하는 다이어그램을 설명하면, LLM이 이를 draw.io XML로
변환하고 화면에는 편집 가능한 다이어그램이 곧바로 렌더링된다.
프런트엔드는 Next.js 16과 React 19로 작성되었고, 다이어그램 렌더링은
`react-drawio` 라이브러리를 사용하며, AI 연동은 Vercel AI SDK를 통해
Claude, GPT, Gemini, DeepSeek, Ollama 등 여러 모델 제공자를 갈아 끼울 수
있도록 설계되어 있다.

이미지나 PDF를 업로드해서 기존 다이어그램을 그대로 재현하는 기능, 클라우드
아키텍처 다이어그램(AWS, GCP, Azure 아이콘)에 특화된 생성 기능, 버전 히스토리
복원, Claude Desktop 및 VS Code에서 사용할 수 있는 MCP 서버 통합까지 지원한다.
공개된 지 1년이 조금 넘은 시점에 이미 3만 3천 개가 넘는 스타와 3천 6백 개가
넘는 포크를 모았고, 오늘 GitHub 트렌딩에 올라오면서 하루 동안 81개의 스타가
추가로 붙었다.

## 분석

### XML을 최종 산출물로 삼은 설계

이 프로젝트의 핵심 판단은 “다이어그램을 이미지가 아니라 draw.io XML로
생성한다”는 데 있다.
LLM이 픽셀이나 SVG를 직접 그리는 대신 draw.io가 이해하는 구조화된 마크업을
출력하게 함으로써, 생성된 결과물은 처음부터 draw.io 캔버스에서 마우스로
드래그하고 다시 고칠 수 있는 편집 가능한 객체가 된다.
이는 “AI가 그림을 그려준다”가 아니라 “AI가 도면 데이터를 작성해준다”에 가까운
구조이며, 결과적으로 생성과 수동 편집이 하나의 파이프라인 안에서 자연스럽게
이어진다.

이 설계는 채팅 기반 반복 수정과도 맞물린다.
사용자가 “이 박스를 오른쪽으로 옮겨줘”라고 요청하면 LLM은 전체 다이어그램을
다시 그리는 대신 기존 XML의 해당 노드 좌표만 바꾸는 식으로 응답할 수 있다.
이미지 생성 모델이었다면 매번 전체 이미지를 재생성해야 했겠지만, XML 기반
구조 덕분에 부분 수정이 가능해진 것이다.

### 모델 제공자를 추상화한 멀티 프로바이더 구조

Vercel AI SDK를 채택한 것은 단순한 라이브러리 선택이 아니라, 특정 LLM
벤더에 종속되지 않겠다는 전략적 결정으로 읽힌다.
README는 AWS Bedrock을 기본값으로 두면서도 OpenAI, Anthropic, Google AI,
Azure OpenAI, Ollama, DeepSeek 등을 나란히 지원한다고 명시한다.
관리자가 서버 측에 여러 모델을 미리 설정해두면 개별 사용자가 자기 API 키를
직접 넣지 않아도 되는 구조도 갖추고 있다.

이런 유연성은 프로젝트가 “어떤 모델이 더 잘 그리는가”라는 질문에 정면으로
답하기보다, 사용자가 직접 비교하고 선택하도록 열어둔다는 뜻이기도 하다.
동시에 README는 “Claude 계열은 클라우드 아키텍처 로고가 포함된 draw.io
다이어그램으로 학습되어 있다”고 특정 모델의 강점을 명시적으로 언급하는데,
이는 모델마다 학습 데이터의 편향이 실제 산출물 품질에 그대로 드러난다는
사실을 프로젝트 스스로 인정하고 있다는 점에서 흥미롭다.

### 입력 경로의 다양화: 텍스트를 넘어선 이미지·PDF 인식

텍스트 프롬프트만이 아니라 이미지나 PDF를 업로드해서 기존 다이어그램을
재현하는 기능은, 이 프로젝트를 “제로에서 다이어그램을 만드는 도구”에서
“이미 존재하는 도면을 편집 가능한 형태로 되살리는 도구”로 확장시킨다.
많은 조직이 오래된 아키텍처 다이어그램을 PNG 스크린샷이나 PDF로만 갖고
있고 원본 편집 파일을 잃어버린 경우가 흔한데, 이 기능은 그런 자료를
다시 편집 가능한 XML로 복원하는 실질적인 쓸모를 만든다.

이 기능이 성립하려면 멀티모달 LLM이 이미지 속의 도형, 화살표, 텍스트
레이블을 정확히 인식해서 구조화된 XML로 옮겨야 한다.
즉 다이어그램 생성 품질이 텍스트 프롬프트 이해력뿐 아니라 모델의 시각 인식
정확도에도 좌우되는 구조로, 별도 영역의 두 가지 역량이 한 파이프라인에
동시에 걸려 있는 셈이다.

### MCP 서버로 확장된 사용 맥락

Claude Desktop과 VS Code에서 쓸 수 있는 MCP 서버 통합은 이 프로젝트를 단순한
웹 서비스에서 벗어나게 만든다.
브라우저에서 채팅창을 열어야만 다이어그램을 만들 수 있는 게 아니라, 이미
코딩 작업을 하고 있는 에이전트 워크플로 안에서 “이 시스템 구조를 다이어그램으로
그려줘”라고 요청하면 곧바로 draw.io XML이 만들어지는 흐름이 가능해진다.
이는 다이어그램 생성이 독립된 작업이 아니라 코드 작성, 문서화, 설계 논의
같은 더 큰 작업 흐름의 부산물로 자연스럽게 딸려 나오는 형태로 도구의
위치를 재정의한다.

## 비평

### “그림을 그린다”는 착시가 검증 부담을 가린다

이 도구가 만들어내는 결과물은 시각적으로 완성된 다이어그램처럼 보이지만,
실제로는 LLM이 추측한 XML 마크업의 결과물일 뿐이다.
클라우드 아키텍처 다이어그램에서 컴포넌트 사이의 연결선이 실제 데이터
흐름과 일치하는지, 로그가 올바른 서비스를 가리키는지는 LLM의 텍스트 이해
정확도에 전적으로 의존한다.
그림이 그럴듯하게 그려질수록 사용자가 세부 내용을 검증하지 않고 그대로
신뢰할 위험이 커지는데, 이는 다이어그램이라는 매체 특성상 잘못된 부분을
한눈에 알아채기 어렵다는 문제와 맞물린다.

특히 시스템 설계 문서나 온보딩 자료로 이 다이어그램이 쓰인다면, 잘못된
화살표 하나가 실제 아키텍처에 대한 오해를 조직 전체에 퍼뜨릴 수 있다.
README 어디에도 생성된 다이어그램의 정확성을 사람이 검토해야 한다는
권고는 명시적으로 등장하지 않으며, 이는 도구의 편의성과 신뢰성 사이의
간극을 사용자 스스로 메워야 한다는 뜻이다.

### 모델 종속성이 사실상 숨은 표준이 되고 있다

멀티 프로바이더를 표방하지만, README가 굳이 “Claude가 클라우드 아키텍처
로고 학습이 잘 되어 있다”고 짚는 순간 이 프로젝트의 실질적 품질 기준은
특정 벤더의 학습 데이터에 종속된다.
다른 모델을 선택할 자유는 있지만, 실제로 만족스러운 결과를 얻으려면
사실상 권장된 소수의 모델(Claude Sonnet, GPT-5.1, DeepSeek V3.2)로
좁혀지는 구조다.
이는 “여러 프로바이더를 지원한다”는 표면적 개방성과 “특정 모델이 아니면
품질이 떨어진다”는 실질적 폐쇄성이 공존하는 상태다.

프로젝트 운영진이 의도했든 아니든, 사용자 대부분은 권장 모델을 그대로
따를 가능성이 높고, 이는 도구의 품질이 결국 소수 벤더의 사전학습 데이터에
얼마나 draw.io 관련 자료가 포함되어 있었는지에 좌우된다는 뜻이다.
멀티 프로바이더 아키텍처가 실제로 보장하는 것은 벤더 락인 해소가 아니라
장애 발생 시 대체 경로 정도로 보는 편이 더 정확하다.

### 편집 가능성이 곧 신뢰성을 뜻하지는 않는다

XML을 산출물로 삼아 draw.io 캔버스에서 즉시 편집할 수 있게 한 설계는
분명 실용적이지만, 이 편의성이 “AI가 만든 초안은 대충 만들어도 사람이
고치면 된다”는 식의 안일한 사용 습관으로 이어질 위험도 함께 만든다.
편집이 쉬워질수록 초기 생성 결과의 정확도에 대한 기대치가 낮아지고,
결과적으로 사람이 매번 전체 구조를 처음부터 다시 검토해야 하는 부담이
줄어들지 않을 수 있다.
즉 “쉽게 고칠 수 있다”는 사실이 “고치지 않아도 된다”는 착각으로 바뀌는
경계가 모호하다.

## 인사이트

### 다이어그램 생성 도구가 문서화 부채를 다른 형태로 옮길 뿐일 수 있다

이 도구는 “다이어그램을 그리는 수고”를 없애주지만, 그 수고를 완전히
사라지게 하는 것이 아니라 “생성된 다이어그램이 맞는지 검토하는 수고”로
옮겨놓을 가능성이 크다.
기존에는 사람이 직접 다이어그램을 그리면서 시스템 구조를 다시 한번
검토하는 과정 자체가 이해를 다지는 역할을 했는데, 이 과정이 생략되면
다이어그램은 빠르게 만들어지지만 그것을 만든 사람의 이해도는 오히려
얕아질 수 있다.
이는 코드 자동완성 도구가 보급된 이후 개발자가 자신이 작성한 코드를
얼마나 깊이 이해하고 있는지에 대한 논쟁과 같은 구조를 갖는다.
결국 문서화 부채는 사라지는 것이 아니라 “누가, 언제 검증하는가”의
문제로 형태만 바뀌어 남는다.

### MCP 통합은 다이어그램을 에이전트 워크플로의 부산물로 만든다

MCP 서버 지원은 단순한 부가 기능이 아니라, 다이어그램 생성이 독립된
목적 행위에서 다른 작업의 부수 효과로 위치가 바뀌는 흐름을 보여준다.
과거에는 다이어그램을 그리기 위해 사용자가 별도로 draw.io나 다른 도구를
열어야 했지만, 이제는 코딩 에이전트가 작업 도중 필요에 따라 다이어그램을
즉석에서 생성해 문서에 끼워 넣을 수 있다.
이는 마치 스프레드시트 프로그램에 매크로 자동화가 들어오면서 “표를
만드는 행위”와 “표를 만들라고 지시하는 행위”가 분리되었던 변화와 비슷하다.
다만 이런 자동 생성이 늘어날수록, 다이어그램이 실제로 어떤 근거로
만들어졌는지 추적하기 어려워지는 두 번째 문제가 함께 자라난다.
생성 이력이 채팅 로그 어딘가에 묻혀버리면, 나중에 왜 이 구조로
그려졌는지 되짚어보기 어려워지는 것이다.

### 오픈소스 스타 수가 곧 실무 채택을 보장하지는 않는다

3만 3천 개가 넘는 스타와 하루 81개씩 늘어나는 성장세는 이 프로젝트가
개발자 커뮤니티의 관심을 크게 끌고 있다는 신호이지만, 이것이 곧 프로덕션
환경에서의 광범위한 채택을 뜻하지는 않는다.
“AI로 다이어그램을 그린다”는 발상 자체가 시연하기 좋고 공유하기 좋은
성격을 가지고 있어서, 실제 업무에 정착시키기보다 한 번 시도해보고
스타를 누른 뒤 떠나는 사용자 비율이 높을 수 있다.
과거 여러 “AI로 무엇이든 만든다” 계열 도구들이 초기 트렌딩 이후 실사용
지표에서는 기대에 못 미쳤던 사례들과 비슷한 패턴을 그릴 가능성을 열어둘
필요가 있다.
스타 수는 관심의 지표이지 신뢰성이나 정확도의 지표가 아니라는 점을
구분해서 볼 필요가 있다.

### 클라우드 벤더 아이콘 재현이 만드는 저작권·정확성의 회색지대

Claude 계열이 AWS, GCP, Azure 아이콘이 포함된 다이어그램에 강점을
보인다는 서술은, 이 도구가 각 클라우드 벤더가 공식으로 배포하는
아이콘 세트와 표기 규칙을 학습 데이터를 통해 흡수했다는 뜻이기도 하다.
이는 실무적으로는 편리하지만, 생성된 아이콘과 배치가 각 벤더의 최신
공식 가이드라인과 정확히 일치하는지, 혹은 학습 시점의 낡은 스타일을
그대로 반복하고 있는 것은 아닌지는 검증되지 않은 채로 남아 있다.
아키텍처 다이어그램이 실제 설계 검토나 보안 감사 자료로 쓰일 경우,
이런 사소해 보이는 표기 오차가 의사소통 오류로 이어질 수 있다는 점은
README에서 다뤄지지 않는 트레이드오프다.
