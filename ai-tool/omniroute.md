# OmniRoute: 236개 이상 AI 프로바이더를 묶는 오픈소스 게이트웨이

<https://omniroute.online/>

<https://github.com/diegosouzapw/OmniRoute>

## 소개

OmniRoute는 여러 AI 프로바이더를 하나의 엔드포인트로 통합하는 무료
오픈소스 AI 게이트웨이다. README는 236개(다른 곳에서는 231개로도
표기되는 등 수치가 문서 내에서 일관되지 않다) 이상의 AI 프로바이더에
대한 접근을 하나의 엔드포인트로 중앙화하며, 지능적 라우팅과 자동
장애 조치(auto-fallback), 토큰 최적화 기능을 제공한다고 소개한다.
구독 한도가 월간 초기화되기 전에 사용량을 최대화하는 쿼터 관리,
레이트 리밋으로 인한 서비스 중단을 막는 프로바이더 간 자동 전환,
RTK와 Caveman이라는 압축 엔진 스택을 통한 15~95%의 토큰 절감,
Claude Code·Cursor·Cline·Copilot 등 16개 이상의 코딩 도구에서 쓸
수 있는 단일 설정을 핵심 가치로 내세운다.

라우팅 기능으로는 우선순위·비용 최적화·라운드로빈·가중 분산·
컨텍스트 릴레이·퓨전 패널 등 17가지 전략을 지원하며, 헬스·쿼터·
비용·지연시간·성공률 등 9개 요소로 점수를 매기는 “스마트 오토
콤보 엔진”을 갖췄다고 설명한다. 무료 티어 집계 기능은 풀링된 무료
프로바이더로부터 월 약 16억 개의 무료 토큰을 확보한다고 주장하며,
Kiro·Qoder·Pollinations·LongCat·NVIDIA NIM·Cerebras·Cloudflare AI
등 11개 프로바이더가 영구 무료 접근을 제공하고, 문서화된 무료
티어를 가진 프로바이더가 50개 이상이라고 소개한다. 압축 파이프라인은
Session-Dedup, CCR, RTK, Headroom, Caveman, LLMLingua-2, Lite,
Aggressive, Ultra라는 9개의 조합 가능한 엔진으로 구성되며, RTK와
Caveman을 함께 적용하면 혼합 페이로드에서 78~95%의 절감을 달성한다고
밝힌다.

프로토콜 측면에서는 OpenAI·Claude·Gemini API 변환을 지원하고, 87개
도구·3가지 전송 방식·30개 스코프를 갖춘 MCP(Model Context Protocol),
SSE 스트리밍을 지원하는 A2A(Agent-to-Agent) JSON-RPC 2.0 프로토콜,
Codex·Devin·Jules 같은 클라우드 에이전트 연동까지 아우른다. 보안
측면에서는 클라우드 중개자 없는 로컬 우선 아키텍처, 저장된 자격
증명에 대한 AES-256-GCM 암호화, 레드팀 테스트를 거친 프롬프트
인젝션 가드레일, IP 필터링과 API 키 스코핑, 기본값으로 텔레메트리를
수집하지 않는다는 점을 강조한다. 흥미로운 점은 “언제 어디서든 AI를
쓸 수 있게 한다”는 명목으로 JA3/JA4 방식의 TLS 지문 스푸핑
(wreq-js 기반)을 적용해 지역 차단을 우회하는 기능이 포함돼
있다는 것인데, 이에 대한 서비스 약관(ToS) 준수 여부는 README에서
논의되지 않는다.

배포 방식은 npm 전역 설치, 멀티 아키텍처 Docker 컨테이너, Electron
기반 데스크톱 앱(Windows·macOS·Linux), Android용 Termux, 오프라인
지원 PWA, 소스 빌드까지 다양하게 제공된다. 기술 스택은 Node.js
22.x/24.x LTS, TypeScript 6.0(핵심 모듈은 100% TypeScript이며 v2.0
이후 `any` 타입을 사용하지 않는다), Next.js 16과 React 19, Tailwind
CSS 4, better-sqlite3(LowDB JSON 폴백 지원)이며, 517개 파일에 걸쳐
14,965개의 테스트 케이스를 갖췄다고 밝힌다. 회복탄력성 아키텍처로는
서킷 브레이커, 커넥션 쿨다운, 모델 격리(lockout), 결손 라운드로빈
스케줄링 기반의 쿼터 공유 라우팅을 갖췄다. 이 프로젝트는 어크놀리지먼트
섹션에서 9router의 포크로 시작했으며, Go로 작성된 CLIProxyAPI를
TypeScript로 이식한 것이라고 명시한다. 라이선스는 MIT이며, 이 글을
작성하는 시점 기준 GitHub 스타 10,000개 이상, 포크 1,500개 이상,
271개 이상의 릴리스, 42개 언어로 현지화됐다고 소개한다.

## 분석

### 무료 티어의 “집계”는 각 프로바이더가 개별 사용자를 전제로 설계한 한도를 재해석하는 행위다

OmniRoute의 핵심 가치 제안은 여러 프로바이더의 무료 할당량을 하나로
“풀링”해 월 16억 토큰이라는 규모를 만들어낸다는 것이다. 그러나 각
프로바이더가 제공하는 무료 티어는 애초에 “한 명의 개인 사용자가
가볍게 시험 삼아 써 보는” 시나리오를 전제로 설계된 한도다. 여러
계정과 여러 프로바이더에 걸친 무료 한도를 자동으로 조합해 하나의
연속적인 대용량 서비스처럼 재구성하는 것은, 개별 한도의 총합은
맞더라도 각 프로바이더가 애초에 의도한 사용 패턴과는 근본적으로
다른 형태의 트래픽을 만들어낸다. 이는 항공사의 마일리지 적립
최적화나 신용카드 리워드 해킹처럼, 개별 약관은 지키면서도 그
약관이 전제한 사용자 행동 모델 자체를 우회하는 최적화 기법의
소프트웨어 버전에 가깝다.

### 압축과 라우팅이라는 두 축의 결합은 비용 최적화 도구가 인프라 계층으로 진화하고 있음을 보여준다

이 프로젝트가 단순한 API 프록시에 머무르지 않고 9개의 압축
엔진, 17가지 라우팅 전략, MCP·A2A 프로토콜 지원까지 아우르는
것은 “저렴하게 쓰기”라는 최초의 동기가 점차 “AI 워크로드 전체를
관리하는 오케스트레이션 레이어”로 확장되고 있음을 보여준다. 압축
기술(LLMLingua-2 같은 의미 기반 가지치기)과 라우팅 전략(9요소
스코어링 기반 오토 콤보)을 결합한 설계는, 더 이상 “어느 프로바이더가
저렴한가”를 수동으로 고르는 문제가 아니라 “주어진 워크로드에
어떤 조합의 압축·라우팅·프로바이더가 최적인가”를 자동으로 푸는
문제로 프레임이 바뀌었다는 뜻이다. 이는 클라우드 컴퓨팅 초기에
단순 리버스 프록시였던 로드밸런서가 점차 오토스케일링·서킷
브레이커·헬스체크를 갖춘 서비스 메시로 진화했던 패턴과 구조적으로
유사하다.

### 지역 차단 우회 기능은 “제약 없는 접근”이라는 프레이밍 뒤에 약관 위반 가능성을 감춘다

TLS 지문 스푸핑을 통한 지역 차단 우회는 README에서 “언제 어디서든
AI를 쓸 수 있게 한다”는 접근성의 언어로 서술된다. 그러나 이 기술이
실제로 하는 일은 프로바이더가 의도적으로 설정한 지역별 접근
제한을 클라이언트 단에서 우회하는 것이다. 이는 검열 회피라는
정당한 목적과, 지역별 가격 차별화나 규제 준수를 위한 프로바이더의
정책을 무력화하는 목적이 기술적으로 구분되지 않는다는 근본적
특성을 갖는다. “접근성”이라는 우호적인 프레이밍으로 소개되는
이 기능이, 실제로는 각 프로바이더의 서비스 약관과 정면으로
충돌할 수 있는 기술이라는 점을 README가 명시적으로 인정하지
않는다는 사실 자체가 분석 대상이다.

## 비평

### 프로젝트를 소개하는 핵심 수치들이 문서 내에서 서로 어긋난다

README는 “236개 이상의 프로바이더”라는 숫자를 프로젝트의 정체성
전면에 내세우지만, 같은 문서 안에서 “231개 AI 프로바이더”, “50개
이상의 무료 프로바이더”처럼 다른 수치가 함께 등장한다. 무료 티어
“풀링”이 정확히 어떤 메커니즘으로 이뤄지는지도 “40개 이상의
프로바이더 풀·500개 이상의 모델을 대시보드에 집계한다”는 모호한
서술 이상으로 구체화되지 않는다. 프로젝트의 핵심 셀링 포인트가
바로 이 “규모”인 만큼, 숫자가 문서 내에서조차 일관되지 않는다는
것은 사소한 오타 이상의 문제다. 이는 실제 프로바이더 수를 정확히
추적·검증하는 프로세스보다 인상적인 숫자를 여러 곳에서 반복
언급하는 마케팅이 앞서 있다는 인상을 준다.

### 지역 차단 우회 기능에 대한 서비스 약관 논의가 통째로 빠져 있다

JA3/JA4 TLS 지문 스푸핑이라는, 명백히 탐지 회피를 목적으로 하는
기술을 프로젝트의 정식 기능으로 포함하면서도, README 어디에도
이것이 각 AI 프로바이더의 서비스 약관을 위반할 수 있다는 경고나
논의가 없다. 이는 단순한 누락이 아니라 구조적인 공백이다. 사용자
입장에서는 이 기능을 켜는 순간 자신의 계정이 이용약관 위반으로
정지될 위험을 감수하게 되는데, 그 위험을 사용자가 스스로 판단할
근거조차 프로젝트가 제공하지 않는다. 오픈소스 프로젝트가 이런
회색지대 기능을 제공하는 것 자체는 드문 일이 아니지만, 최소한
“이 기능은 프로바이더 약관과 충돌할 수 있다”는 고지조차 없다는
것은 사용자 보호보다 기능 목록의 화려함을 우선시한 결과로 보인다.

### 무료 티어 남용이 프로바이더 생태계 전체에 미치는 영향은 논의되지 않는다

이 프로젝트가 성공적으로 확산되어 다수의 사용자가 동시에 여러
무료 프로바이더의 한도를 자동으로 소진하기 시작하면, 이는 그
무료 티어를 제공하는 프로바이더 입장에서는 예상치 못한 비용
급증이나 서비스 품질 저하로 이어질 수 있다. 개별 사용자가 수동으로
여러 무료 계정을 오가며 쓰는 것과, 자동화 도구가 대규모로 이를
최적화해 확산시키는 것은 프로바이더가 감당해야 하는 리스크의
규모가 질적으로 다르다. README는 이 프로젝트가 프로바이더
생태계에 미칠 수 있는 부작용, 예컨대 무료 티어 자체의 축소나
폐지로 이어질 가능성에 대해서는 전혀 언급하지 않는다. 이는 마치
개별 소비자의 관점에서는 합리적인 최적화가, 그 최적화가 대규모로
확산됐을 때는 애초에 그 최적화를 가능하게 했던 자원 자체를
고갈시키는 공유지의 비극 구조를 그대로 재현할 위험이 있다는 점을
간과한 것이다.

### 계보를 인정하면서도 원본 프로젝트와의 관계는 불투명하게 남겨 둔다

README는 이 프로젝트가 9router의 포크로 시작했고, Go로 작성된
CLIProxyAPI를 TypeScript로 이식했다고 명시한다. 그러나 이 계보가
현재 코드베이스에서 정확히 어떤 비중을 차지하는지, 원본 프로젝트
대비 어떤 부분이 완전히 재작성됐는지, 그리고 두 원본 프로젝트의
라이선스가 현재의 MIT 라이선스와 어떻게 양립하는지에 대한 설명은
없다. 오픈소스 생태계에서 포크와 이식은 흔하고 정당한 개발
방식이지만, 파생 관계를 명확히 밝히지 않은 채 “236개 프로바이더를
지원하는 독자적인 게이트웨이”로만 자신을 소개하는 것은, 이 프로젝트가
쌓아 올린 실제 코드 자산과 원본 프로젝트가 이미 마련해 둔 기반을
독자가 구분하기 어렵게 만든다.

## 인사이트

### 무료 티어 재정거래(arbitrage) 자동화 도구는 결국 무료 티어 자체를 소멸시키는 촉매가 될 수 있다

이런 종류의 무료 티어 집계·최적화 도구가 대중화될 때 나타나는
전형적인 패턴이 있다. 처음에는 소수의 얼리어답터가 여러 무료
계정을 오가며 이득을 보지만, 자동화 도구가 이 행위를 대중화하는
순간 프로바이더가 감당하는 무료 트래픽의 총량과 패턴이 예측
불가능해진다. 항공사 마일리지 크레딧카드 최적화(“처칭”)가 결국
카드사의 정책 강화로 이어졌던 것처럼, 클라우드 서비스들이 프리
티어 남용에 대응해 신용카드 인증이나 전화번호 인증을 강화해 온
것처럼, OmniRoute 같은 도구가 널리 쓰이면 프로바이더들은 무료
티어 자체를 축소하거나, 더 엄격한 본인 인증을 요구하거나, API
사용 패턴 탐지를 통한 차단을 강화하는 방향으로 대응할 가능성이
크다. 즉 이 도구의 존재 자체가, 이 도구가 딛고 서 있는 자원
(다양한 무료 AI 티어의 존재)을 스스로 잠식하는 역설적 결과를
낳을 수 있다.

### 압축 기술의 발전은 토큰 가격 하락을 상쇄하는 역설적 결과를 낳을 수 있다

RTK와 Caveman을 결합해 78~95%의 토큰을 절감한다는 주장은 인상적이지만,
이런 압축 기술이 산업 표준으로 자리 잡으면 프로바이더 입장에서는
가격 정책을 재고할 유인이 생긴다. 토큰 단가 인하 경쟁이 치열한
현재 시장에서, 실사용 토큰이 대폭 줄어드는 방향으로 사용자 행동이
바뀌면 프로바이더의 매출은 압축 기술 채택률에 비례해 줄어든다.
이는 경제학에서 말하는 제본스의 역설(Jevons paradox, 효율이
올라가면 총 소비가 오히려 늘어나는 현상)과는 반대 방향의 압력,
즉 압축 기술이 효율을 높일수록 프로바이더가 이를 상쇄하기 위해
토큰 단가를 올리거나 압축된 요청에 대한 별도 요금 체계를 도입할
가능성을 낳는다. 압축과 가격 정책은 결국 서로를 겨냥한 군비
경쟁으로 흐를 수 있으며, 이 구도에서 최종 승자는 압축 기술을
가장 잘 만드는 오픈소스 프로젝트가 아니라, 가격 정책을 통제하는
프로바이더 쪽일 가능성이 높다.

### 게이트웨이 계층의 비대화는 개별 AI 프로바이더의 브랜드 차별화를 무력화한다

OmniRoute처럼 프로토콜 변환·자동 라우팅·압축까지 아우르는 게이트웨이가
표준적인 개발 워크플로에 자리 잡으면, 최종 사용자(개발자)는 점점
더 어떤 프로바이더의 모델을 실제로 쓰고 있는지 의식하지 않게 된다.
9요소 스코어링 기반 자동 콤보 엔진이 매 요청마다 최적의 프로바이더를
선택해 주는 세계에서는, 특정 모델의 브랜드나 고유한 특성보다 게이트웨이
계층이 제공하는 통합 경험이 개발자의 실질적인 접점이 된다. 이는
클라우드 컴퓨팅에서 CDN이나 로드밸런서 계층이 원본 서버의 브랜드를
사용자로부터 가려 왔던 것과 유사한 구조다. 장기적으로 이런 게이트웨이가
표준이 되면, 개별 AI 모델 프로바이더들은 최종 사용자에게 직접
브랜드 충성도를 쌓기 어려워지고, 게이트웨이 계층에서의 노출과
순위(라우팅 전략에서 어떤 프로바이더가 우선적으로 선택되는가)를
둘러싼 새로운 형태의 경쟁, 즉 검색엔진최적화(SEO)의 AI 게이트웨이
버전이 등장할 가능성이 있다.
