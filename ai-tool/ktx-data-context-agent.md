# ktx: AI 데이터 에이전트를 위한 웨어하우스 컨텍스트 레이어

<https://github.com/Kaelio/ktx>

Docs: <https://docs.kaelio.com/ktx>

## 요약

ktx는 Claude Code, Codex 같은 AI 데이터 에이전트가 승인된 메트릭 정의와
회사 지식을 사용해 데이터 웨어하우스를 정확하게 쿼리하도록 돕는 컨텍스트 레이어다.
에이전트가 비즈니스 용어를 자의적으로 해석하지 않고
조직이 정의한 메트릭을 기준으로 SQL을 실행하도록 하는 것이 핵심 목표다.

두 가지 핵심 동작이 있다.
수집(Ingestion) 단계에서는 데이터베이스, BI 도구, 모델링 코드, 문서를
컨텍스트 엔진으로 처리해 검색 가능한 위키 마크다운과 시맨틱 레이어 YAML 파일로 변환한다.
제공(Serving) 단계에서는 에이전트가 MCP(Model Context Protocol)를 통해 ktx를 쿼리하면,
결합된 지식 베이스를 검색해 “웨어하우스에 실행되는 읽기 전용 SQL로 컴파일된 승인된 메트릭”을 반환한다.

주요 기능으로는 dbt, Looker, Metabase, Notion에 분산된 비즈니스 지식 자동 학습,
조인 가능한 컬럼 탐지 및 데이터베이스 조인 복잡성 해소,
원시 테이블과 재사용 가능한 메트릭 정의를 결합한 시맨틱 레이어 구축,
에이전트 실행을 위한 CLI와 MCP 도구 이중 인터페이스,
웨어하우스 안전을 보장하는 읽기 전용 설계가 있다.

PostgreSQL, Snowflake, BigQuery, ClickHouse, MySQL, SQL Server, SQLite를 지원하며,
npm 설치 후 `ktx setup`과 `ktx status`로 시작한다.
TypeScript(82.9%), Python(9.3%), JavaScript로 구현되며 Apache 2.0 라이선스다.

## 분석

### ktx는 “AI가 비즈니스 데이터를 틀리게 쿼리하는 문제”를 구조적으로 해결한다

AI 에이전트에게 데이터 쿼리를 위임할 때 가장 흔한 실패는
용어 해석의 불일치다.
“매출”이 총매출인지 순매출인지, “활성 사용자”가 7일 기준인지 30일 기준인지,
“전환율”의 분자와 분모가 무엇인지는 회사마다 다르다.
에이전트가 이 정의 없이 SQL을 작성하면 문법적으로 올바르지만 비즈니스적으로 틀린 쿼리가 나온다.

ktx의 접근법은 이 정의들을 에이전트가 쿼리 시점에 참조할 수 있는 형태로 미리 인코딩하는 것이다.
“승인된 메트릭”이라는 표현이 핵심이다.
쿼리를 실행하기 전에 메트릭 정의가 조직적으로 검토되고 승인됐다는 것을 전제한다.
이것은 단순한 기술 솔루션이 아니라 데이터 거버넌스 프로세스와의 통합을 요구한다.

### MCP를 통한 에이전트 통합은 도구의 확장성 전략이다

ktx가 MCP를 서빙 인터페이스로 선택한 것은 에이전트 생태계의 흐름을 읽은 결정이다.
MCP는 Claude Code, Codex 같은 주요 에이전트들이 외부 도구와 통합하는 표준 프로토콜이다.
ktx를 MCP 서버로 노출함으로써, 에이전트가 ktx를 다른 도구와 동일한 방식으로 호출할 수 있다.

이 설계의 장점은 특정 에이전트에 종속되지 않는다는 것이다.
Claude Code를 Codex로 교체하거나, 새 에이전트를 추가해도
ktx는 MCP 인터페이스를 통해 동일하게 서비스된다.
에이전트 시장이 아직 안정화되지 않은 상황에서 특정 에이전트에 묶이지 않는 것은
도구 지속성 측면에서 현명한 전략이다.

### 위키 마크다운과 시맨틱 레이어 YAML의 이중 출력은 두 독자를 동시에 겨냥한다

ktx가 수집 결과로 두 가지 형태를 생성하는 것은 의도적이다.
위키 마크다운은 인간이 읽고 수정하기 쉽다.
시맨틱 레이어 YAML은 에이전트가 파싱하고 실행하기 쉽다.
동일한 지식이 두 형태로 공존함으로써 인간 검토와 에이전트 실행이 같은 소스에서 나온다.

이 설계는 비즈니스 지식의 단일 진실 원천(single source of truth)을 만들려는 시도다.
dbt, Looker, Metabase, Notion에 분산된 정의들이 ktx에 통합되면,
앞으로 정의를 변경할 때 한 곳만 수정하면 된다.
이것은 기술 도구이면서 동시에 데이터 문화(data culture) 도구이기도 하다.

## 비평

### “자동 학습”이 실제로 얼마나 정확한지 검증 방법이 제시되지 않는다

“dbt, Looker, Metabase, Notion에서 자동으로 학습한다”는 주장은 매력적이지만,
그 학습의 정확도와 커버리지가 어느 수준인지 문서에서 확인할 수 없다.
실제 환경에서 BI 도구와 문서는 불일치하는 경우가 많다.
Looker에 정의된 메트릭과 Notion에 문서화된 메트릭이 다를 때
ktx는 어느 것을 우선하는가? 충돌을 어떻게 해소하는가?

메트릭 정의가 모호하거나 불완전한 원본 소스에서 학습했을 때
ktx가 생성하는 시맨틱 레이어의 신뢰성은 원본의 품질에 종속된다.
“정확하지 않은 데이터를 자동으로 학습하면 정확하지 않은 메트릭이 생성된다”는
GIGO(Garbage In, Garbage Out) 문제가 여기서도 적용된다.

### 읽기 전용 설계는 웨어하우스 안전을 보장하지만 활용 범위를 제한한다

“웨어하우스 안전을 위한 읽기 전용 설계”는 책임 있는 선택이다.
에이전트가 데이터를 삭제하거나 수정하는 사고를 방지한다.
하지만 이 제약은 동시에 ktx의 활용 범위를 분석(analytics)으로 한정한다.

데이터 파이프라인 자동화, ETL 작업, 데이터 정제 같은 쓰기 작업을 에이전트에게 위임하려면
ktx를 사용할 수 없다.
이것이 명시적 설계 결정이라면 이해할 수 있지만,
“안전”이라는 이름으로 제한을 자연스럽게 보이게 하는 것은
도구의 적용 범위를 불명확하게 만든다.

### 설정 복잡성이 진입 장벽에 대한 설명 없이 생략되어 있다

“npm 설치 후 `ktx setup`으로 시작”이라는 설명은 단순해 보이지만,
실제 엔터프라이즈 환경에서 Snowflake나 BigQuery 같은 웨어하우스에 연결하고,
dbt와 Looker를 통합하며, LLM 프로바이더를 설정하는 과정은 상당한 설정 작업을 요구한다.

특히 엔터프라이즈 데이터 환경에서는 VPC, IAM 권한, 네트워크 정책, SSO 통합이
모두 별도로 처리되어야 한다.
이 설정들이 얼마나 복잡한지, 어떤 권한이 필요한지, 초기 설정에 얼마나 걸리는지에 대한
현실적 설명 없이 도구를 평가하기 어렵다.

## 인사이트

### 비즈니스 메트릭의 기계 판독 가능한 형식화가 AI 데이터 에이전트의 전제 조건이다

ktx가 해결하려는 문제는 기술 문제가 아니라 표현(representation) 문제다.
비즈니스 지식이 자연어로 존재하는 한, AI 에이전트는 그것을 부정확하게 해석한다.
ktx는 그 지식을 기계가 정확하게 실행할 수 있는 형태로 변환하는 중간 레이어다.

이것은 더 일반적인 패턴의 구체적 사례다.
AI 에이전트가 특정 도메인에서 신뢰할 수 있게 작동하려면,
그 도메인의 지식이 기계 판독 가능한 형태로 코드화되어야 한다.
데이터 도메인에서 ktx가 이 역할을 하듯,
법률 도메인에서는 조문 파서가, 의료 도메인에서는 임상 온톨로지가 같은 역할을 해야 한다.
에이전트 능력의 한계는 도메인 지식의 형식화 수준에 의해 결정된다.

### 데이터 거버넌스 성숙도가 AI 데이터 에이전트 도입의 숨겨진 전제 조건이다

ktx를 효과적으로 사용하려면 “승인된 메트릭”이 이미 존재해야 한다.
이것은 데이터 거버넌스가 어느 수준 이상 성숙한 조직을 전제한다.
메트릭 정의가 없거나, 여러 팀이 서로 다른 정의를 사용하거나,
데이터 문서화가 불완전한 조직에서 ktx는 효과를 발휘하기 어렵다.

역설적으로, AI 데이터 에이전트의 혜택이 가장 필요한 조직
(데이터 분석 역량이 부족한 조직)은
AI 에이전트를 신뢰할 수 있게 만드는 전제 조건(데이터 거버넌스)을 갖추지 못한 경우가 많다.
반면 데이터 거버넌스가 잘 갖춰진 조직은 이미 BI 도구와 데이터 팀으로 분석 역량을 보유하고 있다.
ktx 같은 도구는 데이터 역량의 격차를 줄이는 것이 아니라
이미 역량이 있는 조직의 효율성을 더 높이는 방향으로 작동한다.

### MCP 기반 컨텍스트 레이어는 에이전트 신뢰성 문제의 표준 해법이 될 수 있다

“에이전트가 도메인 지식 없이 자의적으로 행동한다”는 문제는
데이터 쿼리에만 해당하지 않는다.
코드 생성, 문서 작성, 이메일 응답 등 에이전트가 조직 맥락에서 작동하는 모든 영역에서
같은 문제가 발생한다.

ktx의 패턴은 이 문제의 일반적 해법을 암시한다.
조직 지식을 MCP 서버로 노출하고,
에이전트가 행동 전에 그 지식을 참조하도록 하는 것이다.
데이터 도메인에서 ktx가 웨어하우스 컨텍스트를 제공하듯,
코딩 도메인에서 코드베이스 컨텍스트가, 비즈니스 도메인에서 정책 컨텍스트가
MCP 레이어로 에이전트에게 제공될 수 있다.
이것은 에이전트 시스템 아키텍처에서 “컨텍스트 레이어”라는 새로운 역할의 부상을 예고한다.
