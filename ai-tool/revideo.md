# Revideo

<https://github.com/redotvideo/revideo>

## 소개

Revideo는 TypeScript로 비디오를 프로그래밍 방식으로 편집하는 오픈소스 프레임워크다.
Motion Canvas를 포크(fork)해 만들어졌으며,
“Create Videos with Code”를 슬로건으로 내세운다.

개발자가 TypeScript로 비디오 템플릿을 작성하고,
동적 입력값으로 렌더링 API 엔드포인트를 배포하거나
브라우저에서 실시간 프리뷰를 제공하는 React 플레이어 컴포넌트를 제공한다.
MIT 라이선스로 완전한 오픈소스다.

GitHub 스타 3,700여 개를 기록하고 있으며,
2024년 3월 출시 이후 v0.10.4(2025년 2월)까지 꾸준히 업데이트됐다.
다만 개발팀은 현재 Revideo 엔진 위에 구축한 상용 제품 Midrender로 중심을
이동한 상태다.

## Motion Canvas 포크의 배경

Revideo 팀은 원래 Motion Canvas 사용자였다.
Motion Canvas 위에 비디오 편집 앱을 구축하려 했으나
두 가지 근본적인 한계에 부딪혔다.

첫째, Motion Canvas 유지보수팀은 이 프로젝트를
[독립형(standalone) 에디터](https://github.com/orgs/motion-canvas/discussions/1015)로만
유지하겠다는 입장을 명확히 했다.
라이브러리로 사용하는 것은 의도한 용도가 아니었다.

둘째, 플러그인 시스템으로 초기 버전을 구현해봤지만,
Motion Canvas API 호환성을 유지하면서 원하는 변경을 가하기 어렵다는 것을 깨달았다.

그래서 직접 포크를 택했다. Motion Canvas의 핵심 애니메이션 엔진은 그대로 가져오되,
세 가지를 새로 구현했다: 헤드리스 렌더링, 병렬화 렌더링, 오디오 지원.

## 아키텍처

Revideo는 generator 함수 기반의 애니메이션 패러다임을 Motion Canvas로부터 계승한다.
`yield*`로 애니메이션을 시퀀스하고, `all()`로 병렬 실행하는 방식이다:

```tsx
import { makeScene2D } from '@revideo/2d';
import { all, chain, createRef, waitFor } from '@revideo/core';

export default makeScene2D('scene', function* (view) {
  const logoRef = createRef<Img>();

  yield* waitFor(1);

  yield* chain(
    all(logoRef().scale(40, 2), logoRef().rotation(360, 2)),
    logoRef().scale(60, 1),
  );
});
```

렌더링 파이프라인은 다음과 같다:

```text
TypeScript 씬 → Vite 번들링 → 헤드리스 Chrome
→ 병렬 프레임 캡처 → FFmpeg 비디오 프레임 추출 → MP4
```

Vite를 번들러로 사용하며, FFmpeg 기반 자체 비디오 프레임 추출기로
HTML `<video>` 태그의 `seek()` 연산을 대체해 렌더링 속도를 높였다.

## 주요 패키지

| 패키지              | 역할                                              |
| ------------------- | ------------------------------------------------- |
| `@revideo/core`     | 애니메이션 엔진, 씬 관리, 타이밍 유틸리티         |
| `@revideo/2d`       | 2D 렌더링 컴포넌트 (Img, Video, Audio 등)         |
| `@revideo/renderer` | 헤드리스 렌더링 서버 (클라이언트 + 서버)          |
| `@revideo/player`   | 브라우저 플레이어 웹 컴포넌트                     |
| `@revideo/player-react` | React 플레이어 컴포넌트                       |
| `@revideo/ffmpeg`   | FFmpeg 바인딩 및 비디오 프레임 추출               |
| `@revideo/vite-plugin` | Vite 통합 플러그인                             |
| `@revideo/cli`      | CLI 도구 (렌더링 엔드포인트 노출 포함)            |

## Motion Canvas와의 구체적 차이점

| 항목             | Revideo                              | Motion Canvas                     |
| ---------------- | ------------------------------------ | --------------------------------- |
| 렌더링 방식      | 함수 호출 / API 엔드포인트           | UI 버튼 클릭 필요                 |
| 렌더링 속도      | 병렬화, FFmpeg 프레임 추출           | 순차 렌더링                       |
| 오디오           | `<Audio/>` 태그 + 내보내기 지원      | 제한적                            |
| 사용 목적        | 라이브러리 / 앱 구축                 | 독립형 에디터                     |
| 클라우드 배포    | Google Cloud Run 예제 제공           | 없음                              |

## 분석

### 포크를 정당화한 세 가지 기술적 결정

Motion Canvas가 “독립형 에디터”로 남기로 결정한 순간,
그 위에 앱을 구축하려는 개발자들에게는 진공 상태가 생겼다.
Revideo는 이 진공을 채우는 세 가지 구체적 변경으로 자신의 존재를 정당화한다.

헤드리스 렌더링은 가장 근본적인 변경이다.
UI 버튼 클릭으로만 렌더링할 수 있는 도구는 자동화 파이프라인에 끼워 넣을 수 없다.
`renderVideo()` 함수 호출로 렌더링을 노출한 것은,
비디오 렌더링을 API 서비스로 배포하는 길을 열었다.

병렬화 렌더링과 FFmpeg 기반 비디오 프레임 추출기는 속도 문제를 다룬다.
개인이 자신의 비디오를 만드는 것과,
서비스에서 수백만 사용자의 비디오를 생성하는 것은 요구하는 성능이 다르다.
“앱을 만들 때 렌더링 속도가 중요하다”는 README의 문장은
이 도구가 누구를 위한 것인지를 명확하게 드러낸다.

### generator 함수 패러다임의 표현력

Revideo가 계승한 Motion Canvas의 generator 기반 애니메이션 모델은
Remotion의 `frame → React 트리` 모델과 근본적으로 다른 추상화다.

`yield*`로 애니메이션을 서술하는 방식은 명령형(imperative) 코드처럼 읽힌다.
“1초 기다린 후, 로고를 2초 동안 40배 키우면서 360도 회전시켜라”는
순서가 코드 구조에 그대로 반영된다.
이것은 시간을 인자로 받는 순수 함수보다 인간의 직관에 더 가깝게 느껴진다.

## 비평

### 강점: MIT 라이선스와 Motion Canvas 생태계 계승

MIT 라이선스는 Remotion의 소스 공개 모델에 비해 상업적 도입 장벽이 전혀 없다.
스타트업이 렌더링 서비스를 구축하거나,
에이전시가 고객 프로젝트에 내재화할 때 법무 검토가 필요하지 않다.

Motion Canvas가 축적한 애니메이션 엔진, 씬 모델,
2D 컴포넌트 시스템을 그대로 계승한다는 점도 강점이다.
Motion Canvas 커뮤니티의 지식과 예제가 상당 부분 Revideo에도 적용된다.

### 약점: 팀이 상용 제품으로 이동했다

가장 큰 리스크는 프로젝트의 현재 상태다.
개발팀은 Revideo 엔진 위에 Midrender라는 상용 비주얼 에디터를 구축하고
그쪽에 집중하고 있다.
README 페이지가 Midrender 랜딩 페이지로 리다이렉트되는 것에서
이 전환을 단적으로 확인할 수 있다.

최근 엔진 개선 사항이 오픈소스 저장소에 반영되지 않고 있다는 것도 문제다.
오픈소스 프로젝트가 상용 제품의 “쇼케이스”로 전락할 때
커뮤니티가 의존하는 라이브러리의 품질은 정체되거나 저하된다.

### 약점: Remotion 대비 생태계 격차

GitHub 스타 3,700개는 Remotion의 44,000개 대비 12분의 1 수준이다.
패키지 다양성, 문서 품질, 커뮤니티 규모 모두 Remotion이 압도적이다.
분산 렌더링(Lambda/Cloud Run)도 예제 수준이며,
Remotion처럼 프로덕션 레디 패키지로 추상화되어 있지 않다.

## 인사이트

### 업스트림의 “우리는 라이브러리가 아니다”가 포크를 낳는다

Motion Canvas 팀의 결정이 흥미롭다.
그들은 npm 패키지로 배포하면서도, 라이브러리로 사용하는 것을 의도하지 않는다고 밝혔다.
이것은 오픈소스 생태계에서 흔히 나타나는 긴장 관계다.

“이 도구로 무언가를 만들고 싶다”는 사용자와,
“이 도구는 이런 목적으로만 써야 한다”는 유지보수팀 사이의 간극이 포크를 만든다.
Revideo는 이 간극에서 태어났고,
그 포크의 정당성이 세 가지 구체적인 기술적 변경으로 입증됐다.

이 패턴은 반복된다. Redis, Elasticsearch, MongoDB가 오픈소스 기반에서
라이선스를 변경하자 Valkey, OpenSearch, FerretDB 같은 포크가 탄생했다.
라이선스 변경이 아니라 “사용 의도의 제한”도 포크의 충분한 이유가 된다.
오픈소스 라이선스는 사용자의 포크 권한을 보장하기 때문에,
유지보수팀이 특정 사용 방식을 거부할수록 포크 생태계가 번성하는 역설이 발생한다.

### OSS 프로젝트가 상용 제품의 도구로 전환될 때 일어나는 일

Revideo의 현재 상태는 오픈소스 지속 가능성의 교과서적 딜레마를 보여준다.
팀은 오픈소스로 시작해 커뮤니티를 형성하고,
그 과정에서 상용 제품에 대한 아이디어를 검증했다.
그리고 상용 제품인 Midrender로 피벗했다.

이 시나리오에서 오픈소스 프로젝트는 세 가지 운명 중 하나를 맞는다.
첫째, 팀이 OSS와 상용 제품을 병행 유지하는 경우(HashiCorp, Elastic의 초기).
둘째, OSS 프로젝트가 커뮤니티에 의해 살아남는 경우(OpenTofu).
셋째, OSS 프로젝트가 서서히 방치되는 경우.

Revideo는 현재 세 번째 경로에 가장 가까워 보인다.
최근 커밋이 드물고, 주요 기능 개선은 Midrender 쪽에서 이루어지고 있다.
이것은 개발팀의 잘못이 아니다. 수익 없이 오픈소스를 무한정 유지하는 것은
개인이나 소규모 팀에게 지속 가능하지 않다.
그러나 이 프로젝트에 의존하는 개발자들에게는 의존성 위험이 된다.

### generator 기반 애니메이션과 frame 함수 기반 애니메이션의 철학적 차이

Revideo(Motion Canvas)의 `yield*` 패러다임과
Remotion의 `frame → React 트리` 패러다임은 단순한 API 차이가 아니다.
애니메이션을 어떻게 모델링하는가에 대한 근본적으로 다른 철학이다.

`yield*` 모델은 애니메이션을 “시간이 흐르는 절차”로 본다.
코드는 “무엇을 한 다음에 무엇을 하라”는 서사(narrative)로 읽힌다.
이것은 After Effects의 키프레임 타임라인보다 오히려 스크립트나 연극 지문에 가깝다.
프로그래머가 직관적으로 이해하기 쉽고,
복잡한 시퀀스를 읽기 쉬운 코드로 표현하는 데 유리하다.

반면 `frame → React 트리` 모델은 애니메이션을 “프레임 인덱스의 함수”로 본다.
임의의 프레임으로 시크(seek)하는 것이 자연스럽고,
렌더러가 비선형 방식으로 프레임을 처리하는 데 유리하다.
이것은 Lambda 병렬 렌더링처럼 순서와 무관하게 프레임을 분산 처리할 때 핵심 전제가 된다.

두 모델 중 어느 것이 더 낫다고 단정하기 어렵다.
“읽기 쉬운 애니메이션 코드”를 원하면 generator 모델이,
“확장 가능한 렌더링 파이프라인”을 원하면 frame 함수 모델이 유리하다.
이 철학적 차이가 두 도구가 각자의 생태계에서 공존할 수 있는 이유다.
