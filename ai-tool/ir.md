# ir - Rust 기반 로컬 마크다운 하이브리드 검색 엔진

> 원문: <https://github.com/vlwkaos/ir>

## 요약

### 무엇인가

vlwkaos의 ir은 마크다운 지식 베이스를 위한 로컬 시맨틱 검색 엔진이다.
Shopify의 Tobi Lütke가 만든 [qmd](https://github.com/tobi/qmd)를 Rust로
포팅한 프로젝트이며, 세 가지 핵심 차이점이 있다. 컬렉션별 독립 SQLite
(공유 글로벌 인덱스 없음), 상주 데몬(모델이 쿼리 간에 로드 유지), 이중
LLM 캐시(확장기 출력과 리랭커 점수 모두 지속 저장)다. BEIR 4개 데이터셋
벤치마크에서 리랭킹이 순수 벡터 대비 최대 +14.5% nDCG@10 향상을 보인다.

### 검색 파이프라인

검색은 다단계로 이루어진다. `Query → BM25 probe → score fusion
(0.80·vec + 0.20·bm25) → reranking`. 강신호 바로가기(BM25 점수 ≥ 0.85,
격차 ≥ 0.10)가 작동하면 모든 LLM 작업을 건너뛴다. 확장기(expander)
모델이 있으면 쿼리를 lex/vec/hyde 하위 쿼리로 확장한 뒤 RRF 융합 → 상위
20개 리랭킹한다. 모든 LLM 출력은 SQLite에 캐시되어 반복 쿼리는 추론을
완전히 건너뛴다.

### 사용 모델

모델은 HuggingFace Hub에서 자동 다운로드된다. 임베딩에 EmbeddingGemma
300M(GGUF), 확장+리랭킹 통합에 Qwen3.5-0.8B 또는 2B(GGUF), 리랭킹 전용에
Qwen3-Reranker 0.6B, 확장 전용에 qmd-query-expansion 1.7B가 사용된다.
한국어 대안으로 BGE-M3 568M도 지원된다. BM25 검색만으로는 어떤 모델도
필요 없고, `ir update`만으로 즉시 동작한다.

### 설치와 사용

macOS에서는 `brew install vlwkaos/tap/ir`로 설치하거나, Rust 1.80+에서
`cargo install --path .`로 빌드한다. macOS에서는 llama.cpp가 Metal을
자동으로 사용한다. 사용법은 단 4줄이다:

```
ir collection add notes ~/notes   # 컬렉션 등록
ir update notes                   # 파일 스캔 → FTS5 인덱스 구축 (BM25)
ir embed notes                    # 청킹 → 임베딩 모델 → 벡터 저장
ir search "memory safety in rust" # 검색 (데몬 자동 시작)
```

### qmd 대비 성능

M4 Max macOS에서 같은 모델과 쿼리로 측정한 벤치마크에서, cold 시작(캐시
없음)은 ir이 3.0초 대 qmd 9.5초(3배 빠름), warm 상태(데몬 + 캐시 hot)에서
ir이 30ms 대 qmd 840ms(28배 빠름)를 보인다. Cold 차이는 리랭킹 후보를
20개로 제한한 결과이고, warm 차이는 qmd가 매 호출마다 ~800ms의 프로세스
스폰 + JS 런타임 비용을 지불하는 데 반해 ir은 상주 데몬 왕복만 필요한
구조적 차이다.

### 한국어(CJK) 지원

한국어 BM25 검색을 위해 lindera 전처리기가 내장되어 있다. `ir preprocessor
install ko`로 설치하면 ko-dic 사전이 포함된 네이티브 바이너리가 다운로드
된다. Decompose 모드를 사용해 “이스탄불의” 같은 교착어 토큰을 형태소
단위로 분리한다. MIRACL-Korean 벤치마크(213개 쿼리)에서, 전처리기 없는
BM25의 nDCG@10은 0.0009이지만 lindera 적용 시 0.0460(50배 향상), lindera
+ 하이브리드 + 리랭킹에서 **0.8411**(2,835개 패시지에서 거의 천장)을
달성했다. 복합어 분해(compound decompounding) 테스트에서도 lindera가
nDCG@10 0.6326을 보여 한국어 형태소 토큰화의 효과를 실증했다.

### MCP 서버 지원

Claude Desktop과 Claude Code를 위한 MCP 서버가 내장되어 있다. `ir mcp`
명령으로 실행하면 search, status, update 세 가지 도구가 노출된다. HTTP
모드(`ir mcp --http 3620`)도 지원하지만, 인증이 없으므로 신뢰할 수 있는
네트워크에서만 사용해야 한다.

### 스토리지 구조

각 컬렉션 데이터베이스(`~/.config/ir/collections/<name>.sqlite`)에 content
(SHA-256 content-addressed), documents(경로, 제목, 해시, active 플래그),
documents_fts(FTS5 가상 테이블), vectors_vec(sqlite-vec kNN), content_vectors
(청크 메타데이터), llm_cache(리랭커 점수 캐시), meta(컬렉션 메타데이터)
테이블이 들어간다. 글로벌 캐시(`~/.config/ir/expander_cache.sqlite`)는
확장기 출력을 저장한다.

## 분석

### qmd에서 ir로의 진화

qmd는 Shopify CEO Tobi Lütke가 “자기 마크다운 노트를 검색”하기 위해 만든
TypeScript 프로젝트다. ir은 이를 Rust로 포팅하면서 세 가지 구조적 변화를
가했다. 첫째, 스토리지를 단일 SQLite에서 컬렉션별 독립 SQLite로 바꿨다.
이는 “rm name.sqlite”로 컬렉션 삭제가 가능하고, 독립 WAL 저널로 동시
쓰기 경합이 없다. 둘째, 프로세스 모델을 쿼리별 스폰에서 상주 데몬으로
바꿨다. 이 변경만으로 warm 쿼리가 840ms에서 30ms로 28배 빨라졌다.
셋째, 캐시를 리랭커 점수만에서 확장기 출력까지 확장했다. 반복 쿼리는
추론을 완전히 건너뛰게 된다.

이 세 변화 모두 “개인 데스크탑에서 반복 사용하는 검색 도구”라는 사용
시나리오에 최적화된 결정이다. 서버 환경에서는 단일 SQLite가 더 관리
하기 쉬울 수 있지만, 개인 사용자의 “컬렉션 추가/삭제/이동”에서는 독립
파일이 유리하다. 상주 데몬은 서버에서는 당연한 패턴이지만, CLI 도구에서는
새로운 결정이다. 이 결정이 28배 성능 향상으로 이어진 것은 qmd의 JS
런타임 비용이 얼마나 컸는지를 보여준다.

### 하이브리드 검색의 설계 선택

ir의 검색 파이프라인은 GBrain(앞서 다룬 Postgres + pgvector 기반 개인
지식 브레인)과 흥미로운 대비를 이룬다. 둘 다 BM25 + 벡터 + 리랭킹의
하이브리드 검색을 사용하지만, 구현 방식이 다르다. GBrain은 Supabase
Postgres 위에서 RRF 융합을 쓰고, ir은 SQLite + sqlite-vec 위에서 가중
합산(0.80·vec + 0.20·bm25)을 쓴다.

주목할 점은 ir의 벤치마크 결과에서 “BM25 융합은 순수 벡터 대비 통계적
으로 유의미한 향상을 제공하지 않는다(paired t-test)”고 밝힌 것이다. 이는
GBrain이 RRF를 핵심 차별화 포인트로 내세우는 것과 대비된다. ir 저자는
그럼에도 하이브리드를 유지하는데, 이는 리랭킹 단계에서 BM25 결과가
후보 풀의 다양성을 높이기 때문으로 추정된다.

### 한국어 벤치마크의 중요한 수치

lindera 없는 BM25에서 nDCG@10이 0.0009라는 것은 한국어 검색에서 형태소
분석이 선택이 아니라 필수라는 것을 정량적으로 증명한다. lindera 적용 후
0.0460(50배)은 형태소 토큰화의 효과를, 하이브리드 + 리랭킹 후 0.8411은
전체 파이프라인의 효과를 보여준다. 이 벤치마크는 MIRACL-Korean(213개
쿼리, 2,835개 패시지)이라는 표준 데이터셋에서 수행되어 재현 가능하다.

### MCP 통합의 의미

ir이 MCP 서버를 내장한 것은 이 도구가 단독 CLI 도구에서 “AI 에이전트의
검색 백엔드”로 포지셔닝을 확장했음을 보여준다. Claude Desktop이나 Claude
Code에서 `ir mcp`를 설정하면, 에이전트가 자연어로 사용자의 마크다운 노트를
검색할 수 있다. 이는 GBrain의 MCP 서버 통합과 같은 방향이지만, ir은
완전 로컬이라는 점에서 데이터 주권 측면이 더 강하다.

## 비평

### 강점

#### 벤치마크의 정직함

BEIR 4개 데이터셋에 대한 벤치마크, qmd 대비 cold/warm 성능 비교,
MIRACL-Korean 벤치마크까지 제공한 것은 검색 도구로서 드문 수준의 투명성
이다. 특히 “BM25 융합이 통계적으로 유의미한 향상을 주지 않는다”고 스스로
밝힌 것은 정직하다. 대부분의 검색 도구는 하이브리드의 모든 구성요소가
필수적이라고 주장한다.

#### Homebrew 지원과 제로 설정 모델

`brew install vlwkaos/tap/ir`로 설치하고, 모델이 첫 사용 시 자동 다운
로드되며, 데몬도 첫 검색에 자동 시작된다는 것은 마찰을 극한까지 줄인
설계다. BM25만 쓸 때는 모델이 전혀 필요 없다는 점도 진입 장벽을 낮춘다.

#### CJK 전처리기의 플러그인 아키텍처

한국어/일본어/중국어 전처리기를 stdin/stdout 프로토콜로 분리하고 “어떤
실행 파일이든 이 프로토콜을 따르면 등록 가능”하게 만든 것은 확장성이
뛰어나다. `ir preprocessor install ko`로 사전 내장 바이너리를 설치하는
UX도 깔끔하다.

#### SQLite + sqlite-vec의 실용성

Postgres + pgvector를 요구하지 않고 SQLite + sqlite-vec를 정적 컴파일해
단일 바이너리에 포함시킨 것은 로컬 도구로서의 핵심 설계 결정이다.
외부 데이터베이스 의존이 제로이며, `rm name.sqlite`로 데이터를 완전히
제거할 수 있다.

#### Content-addressed 저장소

같은 컬렉션 내에서 SHA-256으로 동일 파일을 중복 제거하는 것은 대규모
마크다운 저장소에서 디스크 사용을 줄이고, 인덱스 업데이트 시 변경된 파일
만 재처리하는 효율성을 보장한다.

### 약점

#### README 구조의 과밀함

기술적 내용이 풍부하지만, 거의 모든 것이 `<details>` 접이식 섹션에
담겨 있어 초보 사용자가 처음 방문했을 때 “Quick start” 4줄과 Features
목록만 보이고 나머지는 접혀 있다. 중요한 정보(한국어 지원, 벤치마크,
MCP 통합)가 숨겨져 있어 발견하지 못할 수 있다.

#### 윈도우 미지원

README에서 지원 플랫폼은 macOS(Intel + Apple Silicon)와 Linux(x86_64,
aarch64)뿐이다. 윈도우 사용자는 WSL을 통해야 하는데 이에 대한 안내가
없다. 한국의 윈도우 사용 비율을 고려하면 접근성의 빈틈이다.

#### 데몬 관리 도구 부재

상주 데몬이 핵심 성능 이점인데, 데몬 시작/중지/재시작/상태 확인을 위한
명시적 명령어가 README에서 잘 드러나지 않는다. `ir status`로 건강 상태는
볼 수 있지만, 데몬 OOM이나 크래시 시 어떻게 되는지, 모델 로드가 실패
하면 어떻게 되는지 등의 실패 모드가 다뤄지지 않는다.

#### 경쟁 제품과의 명확한 위치 결정 부재

qmd와의 비교는 상세하지만, GBrain, Obsidian, Raycast AI, macOS Spotlight
같은 “개인 지식 검색” 경쟁자와의 비교는 없다. “왜 ir이 필요한가”라는
질문에 qmd를 모르는 사용자는 답을 얻기 어렵다.

#### FTS5 injection-safe 주장의 검증 부족

“All user input escaped before FTS5 query construction”이라고 명시했지만,
보안 감사 결과나 퍼즈 테스트 데이터는 없다. 사용자 입력을 직접 FTS5에
넘기는 것이 아니라 이스케이프한다는 것은 좋은 관행이지만, 검증 없이
안전하다고 주장하는 것은 조심해야 한다.

## 인사이트

### 1. “28배 성능 차이”의 진짜 원인은 Rust가 아니라 아키텍처다

ir과 qmd의 warm 쿼리 성능 차이(30ms vs 840ms, 28배)를 보면 “Rust가
TypeScript보다 빠르다”는 결론을 내리기 쉽다. 그러나 실제 원인은 언어가
아니라 프로세스 모델이다. qmd는 매 쿼리마다 새 프로세스를 스폰하고 JS
런타임을 초기화한다(~800ms). ir은 상주 데몬에서 이미 로드된 모델로 즉시
응답한다(30ms). 이 차이의 대부분은 프로세스 스폰 비용이며, qmd를 Node.js
데몬으로 재구현해도 상당 부분 좁힐 수 있다.

이 관찰은 더 큰 교훈을 담고 있다. CLI 도구의 성능을 극적으로 향상시키는
가장 쉬운 방법은 언어를 바꾸는 것이 아니라 프로세스 모델을 바꾸는 것이다.
“매번 새로 시작”에서 “항상 떠 있는 서비스”로 전환하면, 대부분의 CLI
도구에서 자릿수 단위의 개선이 가능하다. ripgrep이 grep보다 빠른 것도
부분적으로 같은 원리다(mmap과 병렬화 외에도 프로세스 최적화 기여).

이 아키텍처 패턴은 AI CLI 도구에서 특히 중요하다. LLM 모델 로딩에 수 초가
걸리기 때문에, 매 호출마다 모델을 로드하면 사용이 불가능할 정도로 느려
진다. ir의 데몬 모델은 이 문제의 정답이다. 앞으로 등장할 AI CLI 도구
(RAG 검색, 코드 분석, 문서 요약)도 동일한 데몬 패턴을 채택할 것이다.
“CLI이지만 실은 로컬 서비스”라는 하이브리드 형태가 AI 도구의 새 표준이
될 가능성이 높다.

### 2. SQLite + sqlite-vec는 “로컬 벡터 검색의 정답”이 되고 있다

ir이 Postgres + pgvector 대신 SQLite + sqlite-vec를 선택한 것은 “서버리스
벡터 검색”이라는 새로운 카테고리를 대표한다. GBrain은 Supabase Pro($25/월)
를 요구하지만, ir은 로컬 파일 하나로 모든 것을 해결한다. 이 차이는 단순한
편의성이 아니라 데이터 주권, 인터넷 의존성, 운영 비용에서 결정적이다.

sqlite-vec(Alex Garcia의 프로젝트)는 SQLite의 가상 테이블로 kNN 벡터 검색을
제공하며, 별도의 서버나 서비스 없이 SQLite 파일 안에 벡터를 저장한다. ir이
이를 정적으로 컴파일해 단일 바이너리에 포함시킨 것은 “설치 의존성 제로”를
실현한다.

이 패턴의 확산은 주목할 만하다. Obsidian의 로컬 AI 플러그인, Apple의 Core
ML 기반 온디바이스 검색, 그리고 이제 ir까지 ― 모두 “벡터 검색을 클라우드가
아니라 로컬에서” 실행하는 방향이다. 2024년까지 벡터 검색은 거의 전적으로
“클라우드 벡터 DB” 담론이었지만(Pinecone, Weaviate, Qdrant), 2025-2026년
에는 로컬 벡터 검색이 빠르게 성장하고 있다. SQLite + sqlite-vec는 이
카테고리의 표준 스택이 될 가능성이 높다.

### 3. MIRACL-Korean 벤치마크가 증명하는 “한국어 검색의 특수성”

ir이 제공한 MIRACL-Korean 벤치마크 수치는 간단하지만 강력하다. 형태소
분석 없는 BM25의 nDCG@10이 0.0009라는 것은 “사실상 아무것도 찾지 못한다”
는 뜻이다. 이는 영어와 한국어의 구조적 차이 때문이다. 영어는 공백으로
단어가 분리되므로 FTS5의 기본 토크나이저(porter)가 합리적으로 작동하지만,
한국어는 “이스탄불의”가 하나의 토큰으로 처리되어 “이스탄불”이라는 쿼리에
매칭되지 않는다.

lindera(Decompose 모드) 적용 후 0.0460(50배 향상)은 형태소 토큰화의 가치를
극명하게 보여준다. 그리고 하이브리드 + 리랭킹 후 0.8411은 “올바른 전처리 +
올바른 검색 파이프라인”이 결합되면 소규모 패시지 컬렉션에서 사실상 거의
완벽한 검색이 가능하다는 증거다.

이 수치는 한국어 RAG 시스템을 구축하는 모든 개발자에게 필수 참조다. “한국어
BM25는 형태소 분석 없이는 쓸모없다”는 것이 정량적으로 증명되었으며, 벡터
검색과 결합하면 소규모 컬렉션에서 nDCG@10 0.84라는 실용적 수준에 도달할
수 있다. 이는 GBrain이 제공하지 않은 정량적 증거이며, ir의 가장 큰 기여 중
하나다.

### 4. “개인 검색 도구”는 AI 시대의 새로운 인프라다

qmd를 Tobi Lütke(Shopify CEO)가 만들고, 그것을 vlwkaos가 Rust로 포팅하고,
GBrain을 Garry Tan(Y Combinator CEO)이 만드는 패턴에서 무엇을 읽을 수
있는가? 테크 리더들이 “자기 마크다운 노트를 검색하는 도구”를 직접 만들고
있다는 것이다. 이는 개인 지식 검색이 2026년 시점에서 “해결되지 않은
문제”임을 보여준다.

Obsidian, Notion, Roam 같은 노트 앱은 기본 검색을 제공하지만, 시맨틱
검색이나 하이브리드 검색은 아직 성숙하지 않았다. Apple의 Spotlight는
파일 이름과 메타데이터 검색에 강하지만 본문 시맨틱 검색은 약하다. 이
틈새에서 qmd, ir, GBrain 같은 도구가 등장하는 것이다.

ir이 이 시장에서 갖는 고유한 위치는 “완전 로컬, 제로 의존성”이다. GBrain은
Supabase + OpenAI + Anthropic이라는 클라우드 삼중 의존이 있고, qmd는 모델
로딩이 느리며, 기존 노트 앱의 검색은 시맨틱이 아니다. ir은 로컬 SQLite +
GGUF 모델 + Metal GPU로 “인터넷 없이, 외부 서비스 없이, 30ms 안에” 검색을
완료한다. 이 조합은 프라이버시 민감 사용자(법률, 의료, 금융)에게 특히
가치가 있다.

장기적으로 “개인 검색 도구”는 OS 수준에 통합될 것이다. Apple이 Core ML +
on-device 검색을 강화하고, Microsoft가 Copilot + Windows 검색을 결합하는
흐름이 이미 시작됐다. ir 같은 독립 도구는 이 OS 통합이 완성되기 전의
과도기적 솔루션이면서, 동시에 OS 통합이 놓치는 파워 유저 니즈(커스텀 모델,
CJK 전처리, MCP 통합)를 채우는 역할을 한다. 이 역할이 이 도구의 가장
지속 가능한 가치다.

### 5. “공개 벤치마크”가 검색 도구의 새로운 표준이 되어야 한다

ir의 README에서 가장 인상적인 부분은 벤치마크의 투명성이다. BEIR 4개
데이터셋의 nDCG@10, qmd 대비 cold/warm 성능, MIRACL-Korean 수치, 복합어
분해 벤치마크 ― 모두 재현 가능한 조건(`research/experiment.md` 참조)과
함께 제공된다.

이 투명성은 GBrain이 제공하지 않는 것이다. GBrain은 “하이브리드 검색이
실제로 동작한다(that actually works)”고 주장하지만 어떤 벤치마크 수치도
공개하지 않았다. 이는 검색 도구 선택에서 중요한 차이다. 사용자는 벤치
마크 없이는 “이 도구의 검색이 얼마나 좋은가”를 알 수 없고, 도구 간
비교도 불가능하다.

앞으로 검색 도구(RAG 포함)는 BEIR, MTEB, MIRACL 같은 공개 벤치마크에서
측정한 수치를 README에 필수적으로 포함해야 한다는 사회적 기대가 형성될
것이다. LLM 분야에서 MMLU, HumanEval, GPQA 같은 벤치마크가 모델 선택의
표준이 된 것처럼, 검색 도구에서도 nDCG@10이 비교의 단일 축이 될 것이다.
ir은 이 방향을 가장 먼저 실천한 개인 검색 도구 중 하나이며, 이 패턴이
다른 도구들에도 확산되면 사용자의 도구 선택이 직관이 아니라 데이터에
기반하게 된다. 이는 전체 카테고리의 품질 향상으로 이어질 것이다.
