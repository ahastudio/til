# OpenHarness - Claude Code 아키텍처를 재현한 오픈소스 AI 에이전트 프레임워크

원문: <https://github.com/HKUDS/OpenHarness>

## 요약

OpenHarness는 홍콩대학교 데이터 사이언스 연구소(HKUDS)가 만든 오픈소스 Python
프레임워크로, AI 에이전트를 구축하기 위한 인프라를 제공한다. 도구 사용, 스킬,
메모리, 멀티 에이전트 조정 등 핵심 기능을 갖추고 있으며, 언어 모델에 손과 눈과
기억과 안전 경계를 부여하는 것을 목표로 한다.

10개의 통합 서브시스템을 제공한다. 에이전트 루프 엔진, 43개의 통합 도구(파일
입출력, 셸, 웹 검색, MCP 프로토콜), 마크다운 기반 온디맨드 스킬 시스템(40개
이상 내장), 다단계 안전 모드의 권한 시스템, MEMORY.md를 통한 세션 간 영속
메모리, 멀티 에이전트 스폰과 팀 관리, MCP 클라이언트 지원, 54개의 내장 명령어,
React/Ink 기반 터미널 인터페이스를 포함한다.

Ohmo라는 개인 에이전트 애플리케이션도 OpenHarness 위에 구축되었다. Feishu,
Slack, Telegram, Discord에서 동작하며, 자율적으로 브랜치를 포크하고 코드를
작성하고 테스트를 실행하고 PR을 여는 기능을 갖추고 있다.

Anthropic, OpenAI, Copilot, Moonshot/Kimi, Gemini, Groq, Ollama 등 다수의
프로바이더를 지원한다. GitHub 기준 10k 스타, 1.7k 포크, 290개 이상의 커밋,
MIT 라이선스로 공개되어 있다.

## 분석

### 아키텍처 구조

OpenHarness의 핵심은 query에서 stream, tool-call 사이클로 이어지는 에이전트
루프다. 지수 백오프(exponential backoff)와 병렬 실행을 지원한다. 이 구조는
Claude Code의 에이전트 루프와 거의 동일한 패턴이다. 도구 호출 결과를 스트리밍
으로 받아 다음 행동을 결정하는 반복 구조가 핵심이다.

### Claude Code와의 유사성

MEMORY.md를 통한 세션 간 영속 메모리, 마크다운 파일 기반 스킬 시스템(Anthropic의
스킬 호환을 명시), /commit이나 /plan 같은 슬래시 명령어, 다단계 권한 시스템,
React/Ink 기반 터미널 UI까지 Claude Code의 아키텍처를 거의 그대로 재현하고 있다.
이는 우연의 일치가 아니라 명시적인 역설계(reverse engineering)의 결과로 보인다.

### MCP 통합과 생태계

Model Context Protocol 클라이언트를 지원하는 것은 Anthropic이 주도하는 MCP
표준을 따르겠다는 의미다. 이를 통해 기존 MCP 서버 생태계를 그대로 활용할 수
있으며, 도구 확장성을 확보한다. 43개의 통합 도구는 파일 시스템 조작부터 웹
검색까지 범용적인 에이전트 작업을 커버한다.

### 테스트와 품질

114개의 pytest 테스트와 6개의 E2E 테스트 스위트를 갖추고 있다. 오픈소스
에이전트 프레임워크 중에서 테스트 커버리지를 명시적으로 공개하는 것은 드문
편이며, 프로젝트의 성숙도를 보여주는 지표다.

## 비평

### 강점

아키텍처 설계가 실전에서 검증된 Claude Code의 구조를 따르고 있어, 설계 자체의
타당성은 이미 입증되어 있다. MIT 라이선스로 공개되어 상업적 활용이 자유롭다.
다수의 LLM 프로바이더를 지원하므로 특정 벤더에 종속되지 않는다. Ohmo라는 실제
애플리케이션이 프레임워크 위에 구축되어 있어 단순한 개념 증명을 넘어선다.

### 약점

“영감을 받은” 수준을 넘어 명시적으로 Claude Code의 구조를 복제하고 있다는 점은
독자적인 설계 철학의 부재를 드러낸다. MEMORY.md, 스킬 시스템, 슬래시 명령어
등의 네이밍과 구조가 거의 동일하다. Claude Code가 진화할 때 이 프레임워크가
독자적으로 발전할 방향성을 가지고 있는지는 불확실하다.

학술 연구소(HKUDS)가 주도하는 프로젝트라는 점도 양날의 검이다. 연구 목적의
프로젝트는 논문 발표 후 유지보수가 중단되는 경우가 많다. 290개의 커밋과 10k
스타는 현재 시점의 관심도를 보여주지만, 장기적 지속성을 보장하지는 않는다.

### 품질 불확실성

GeekNews 댓글에서 언급된 것처럼 HKUDS는 흥미로운 프로젝트를 여러 개 만들어
왔다. 그러나 흥미로운 것과 프로덕션에서 쓸 수 있는 것은 다르다. 에이전트
프레임워크는 엣지 케이스 처리와 안정성이 핵심인데, 이 부분의 검증은 아직
부족해 보인다.

## 인사이트

### 오픈소스 역설계가 의미하는 AI 도구의 상품화

Claude Code의 아키텍처가 오픈소스로 재현될 수 있다는 것은, 에이전트 오케스트
레이션 레이어가 더 이상 경쟁 우위의 원천이 아닐 수 있음을 시사한다. 도구 호출
루프, 메모리 시스템, 권한 관리, 스킬 시스템은 구현 패턴이 표준화되고 있다.

이는 웹 프레임워크의 역사와 유사하다. Ruby on Rails가 MVC 패턴을 대중화하자
Django, Spring Boot, Express 등이 같은 패턴을 채택했다. 패턴 자체는 상품화
되었고, 차별화는 생태계와 개발자 경험으로 이동했다. AI 에이전트 프레임워크도
같은 경로를 밟고 있다.

Claude Code의 진정한 경쟁 우위는 아키텍처가 아니라 그 위에서 동작하는 모델의
품질, Anthropic의 안전성 연구, 그리고 지속적인 개선 속도에 있다. 아키텍처가
복제 가능하다는 것은 오히려 Anthropic의 핵심 가치가 모델 레이어에 있다는
것을 증명한다.

### 학술 연구소가 상용 도구를 역설계하는 현상

HKUDS 같은 학술 기관이 상용 AI 도구의 아키텍처를 분석하고 재현하는 것은
AI 도구 생태계에서 새로운 현상이다. 과거에는 학술 연구가 산업을 선도했지만,
AI 에이전트 분야에서는 산업이 앞서고 학술이 이를 역설계하는 구도가 형성되고
있다.

이 현상은 두 가지 방향으로 해석할 수 있다. 첫째, 상용 도구의 아키텍처가
충분히 성숙하여 학술적 분석 대상이 되었다는 의미다. 둘째, 학술 연구소가
새로운 아키텍처를 제안하기보다 기존 구현을 재현하는 데 집중하고 있다면,
이 분야의 근본적 혁신이 산업에서 주도되고 있다는 방증이기도 하다.

nanobot 등 HKUDS의 다른 프로젝트와 함께 보면, 이 연구소는 실용적인 AI
도구 구현에 초점을 맞추고 있다. 논문보다 코드를 우선하는 이런 접근은 학술과
산업의 경계가 흐려지는 추세를 반영한다.

### 에이전트 아키텍처의 수렴과 표준화

OpenHarness, OpenHands, goose, Devin 등을 관찰하면 AI 에이전트의 아키텍처가
하나의 패턴으로 수렴하고 있음을 알 수 있다. 에이전트 루프, 도구 호출, 메모리
영속화, 권한 관리, MCP 통합이라는 다섯 가지 핵심 요소가 사실상 표준이 되었다.

이 수렴은 두 가지 시사점을 갖는다. 단기적으로는 에이전트 프레임워크 간 전환
비용이 낮아지며, 개발자는 특정 프레임워크에 종속되지 않는다. 장기적으로는
프레임워크가 아니라 그 위의 에이전트 경험(UX, 도구 품질, 모델 최적화)이
차별화 요소가 된다.

운영체제의 역사가 참고가 된다. UNIX의 핵심 아키텍처(프로세스, 파일 시스템,
파이프)는 표준화되었고, 경쟁은 커널 성능, 패키지 관리, 사용자 경험으로
이동했다. AI 에이전트도 아키텍처 표준화 이후의 경쟁이 본격화되는 단계에
진입하고 있다. OpenHarness의 등장은 그 전환점을 보여주는 사례다.
