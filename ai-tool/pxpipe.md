# pxpipe — 텍스트 컨텍스트를 이미지로 변환해 Claude Code 토큰 비용을 줄이는 프록시

<https://github.com/teamchong/pxpipe>

HN 토론: <https://news.ycombinator.com/item?id=48776464> (302점, 98개 댓글)

## 소개

pxpipe는 Claude Code의 API 요청을 중간에 가로채 대용량 텍스트 컨텍스트를 PNG 이미지로
변환하는 로컬 프록시다.
이미지 토큰 비용은 텍스트 내용이 아니라 픽셀 크기에 고정되기 때문에,
고밀도 텍스트(코드, JSON, 도구 출력)를 이미지로 변환하면 토큰 사용량을 대폭 줄일 수 있다.
실측 기준 문자당 약 3.1자/이미지-토큰 대 1자/텍스트-토큰으로,
엔드투엔드 청구액 기준 약 59~70% 절감이 보고됐다.

```bash
npx pxpipe-proxy                                  # 127.0.0.1:47821 포트에 프록시 실행
ANTHROPIC_BASE_URL=http://127.0.0.1:47821 claude  # Claude Code가 프록시를 통하도록 설정
```

기본 지원 모델은 `claude-fable-5`와 `gpt-5.6`이다.
Opus 4.8은 렌더링된 이미지를 약 7% 비율로 오독하고,
GPT 5.5는 이미지 컨텍스트에서 성능이 떨어지기 때문에 기본값에서 제외돼 있다.
대시보드(`http://127.0.0.1:47821/`)에서 절감된 토큰, 텍스트→이미지 변환 내역,
킬 스위치, 모델 선택 칩을 실시간으로 확인할 수 있다.

프록시는 요청만 압축하고 응답에는 절대 손대지 않는다.
최근 턴은 항상 텍스트로 유지되며, 시스템 프롬프트와 도구 문서, 오래된 히스토리 벌크만
이미지로 변환된다.
변환 여부는 수익성 게이트(profitability gate)가 요청별로 결정하며,
희소한 산문(~3.5자/토큰)은 이미지 변환이 오히려 손해이므로 텍스트로 통과시킨다.

## 기술 구조

이미지 변환 파이프라인은 단순하다.

```text
tool_result 문자열 ──► 1928px 너비로 줄 바꿈 ──► 페이지당 ~92,000자 압축 ──► PNG[]
```

1928×1928 이미지는 약 4,761 비전 토큰 비용이 들면서 약 92,000자를 담는다.
텍스트 대비 이미지가 유리한 손익분기점은 ~19자/토큰이며,
실제 Claude Code 트래픽 기준(N=391)으로 ~1.91자/토큰이므로 대부분의 경우 이미지가 유리하다.

프록시는 `/v1/messages` POST를 가로채면서 동시에 원본 비압축 요청에
`count_tokens` 프로브를 병렬로 보내 반사실(counterfactual) 토큰 수를 측정한다.
실제 청구 사용량과 반사실 값이 같은 행의 `~/.pxpipe/events.jsonl`에 기록되므로,
턴 수나 실행 간 혼동 없이 정확한 절감량을 재현할 수 있다.

압축 대상은 세 종류다: (1) 6k자 이상의 대형 `tool_result` 바디(파일 읽기, 명령 출력, 로그),
(2) 라이브 테일 뒤쪽의 오래된 히스토리, (3) 정적 시스템 프롬프트 + 도구 문서 슬랩.
프롬프트 캐싱은 계속 작동한다(정적 접두사 보존).
캐시 가격은 양쪽에 동일하게 적용되므로 캐시 할인이 절감액으로 이중 계산되는 일이 없다.

라이브러리 모드도 지원한다.

```ts
import { renderTextToImages, transformAnthropicMessages } from "pxpipe-proxy";
const { pages } = await renderTextToImages(toolResultText);
const { body, applied, info } = await transformAnthropicMessages({
  body: requestBytes, model: "claude-fable-5",
});
```

`options.keepSharp(block)`으로 특정 블록을 텍스트로 고정할 수 있으며,
`options.emitRecoverable`로 이미지화된 블록의 원본을 반환받을 수 있다.

## 분석

### 이미지 토큰 가격 구조의 비대칭성을 이용한 아비트라지

pxpipe의 핵심 아이디어는 Anthropic의 비전 토큰 가격 구조가 텍스트 가격 구조와 다른 척도로
책정됐다는 관찰에서 출발한다.
텍스트 토큰은 내용의 복잡성에 비례하지만,
이미지 토큰은 픽셀 면적에 비례한다.
이 비대칭성이 “같은 정보를 다른 모달리티로 전달하면 더 싸게 전달할 수 있는” 공간을 만든다.

이것은 Anthropic이 의도하지 않은 가격 차익거래(price arbitrage)다.
모델이 이미지에서 텍스트 내용을 충분히 읽을 수 있다면, 비전 토큰 가격이
텍스트 토큰 가격보다 효율적인 경우 이미지 경로가 항상 우선된다.
Fable 5가 100/100 정확도로 렌더링된 이미지를 읽을 수 있다는 것은,
Anthropic이 의도적으로 이 격차를 허용했거나 아직 인지하지 못했다는 의미다.
이 격차가 가격 정책 변경으로 닫힐 가능성이 있으며, pxpipe의 가치는 그 시점까지만 유효하다.

HN 토론에서는 이것이 정말 "루프홀"인지를 두고 견해가 갈렸다.
ricardobeat는 루프홀이 아니라 정보를 광학 토큰으로 인코딩하는 쪽이
텍스트로 인코딩하는 쪽보다 본질적으로 효율적인 것뿐이라고 반박했다[^ricardobeat].
dimitropoulos는 이 기법이 pxpipe의 독자적 발견이 아니라
DeepSeek의 "Contexts Optical Compression" 백서에서 이미 다뤄진 접근이라는 점을
지적했다[^dimitropoulos].
즉 텍스트를 시각 토큰으로 눌러 담아 컨텍스트를 압축하는 아이디어 자체는
학계에서 먼저 검증된 것이고, pxpipe는 이를 Claude Code 프록시 형태로
실용화한 사례에 가깝다.

수익성 게이트의 설계도 주목할 만하다.
단순히 “모든 텍스트를 이미지로” 바꾸는 게 아니라,
요청별로 손익분기점을 계산해 이익이 될 때만 변환한다.
N=391 실제 트래픽 기반으로 보정된 이 게이트는,
희소한 산문에서 이미지 변환이 오히려 비용을 높인다는 사실을 고려한다.
이 설계는 “모든 토큰을 절감하는 도구”가 아니라
“수익성 있는 경우에만 압축하는 금융 도구”라는 자기 인식을 보여준다.

### Fable 5의 비전 능력이 가져온 새로운 컨텍스트 전달 경로

기존 LLM 프록시 최적화(프롬프트 압축, 요약, 청킹)는 정보를 버리거나 변환한다.
pxpipe는 다르다—정보를 버리지 않고 모달리티만 바꾼다.
이미지 안의 텍스트는 여전히 모델이 읽을 수 있으며,
정보 손실 없이 토큰 비용만 줄인다(정확히 말하면, 읽기 오류가 발생하는 경우를 제외하면).

이 접근은 Fable 5의 비전 능력이 “사진이나 다이어그램을 이해하는 것”을 넘어
“밀도 높게 렌더링된 코드와 JSON을 읽는 것”에도 충분히 강하다는 사실을 전제로 한다.
100/100 산술 정확도와 98/98 정보 회상이 이를 지지한다.
그러나 Opus 4.8에서 12자리 hex 정확 회상이 0/15로 완전히 실패한다는 관찰은,
이 전제가 모델마다 크게 다르다는 것을 보여준다.
비전 능력의 고도화가 특정 모델에서만 나타나는 불균등 발전이라는 점이 드러난다.

SWE-bench 결과가 흥미롭다.
Lite에서는 10/10 동일, Pro에서는 14/19 ON vs 15/19 OFF—실질적으로 동등하다.
이는 에이전트가 파일을 편집하기 전에 다시 읽기 때문에 이미지의 부정확한 회상이
실제 코딩 오류로 이어지지 않는다는 메커니즘으로 설명된다.
“에이전트는 행동 전에 파일을 다시 읽는다”는 보호 메커니즘이 있는 한,
이미지의 손실성(lossiness)이 코드 정확성에 미치는 영향은 제한적이다.

### 이 저장소 자체가 pxpipe를 통해 만들어졌다

README의 마지막 FAQ에 이 사실이 명시돼 있다.
“이 저장소의 코드와 문서 대부분은 자체 압축된 히스토리를 이미지 페이지로 읽으면서
작업한 Opus/Fable 에이전트 세션이 작성했다.”
이것은 자기 참조(self-reference)이면서 동시에 실사용 증명이다.
도구가 자신의 개발 과정에 사용됐다는 것은, 개발자가 도구의 실용성에 충분히 확신이 있었다는 뜻이다.

이 사실은 한편으로는 강력한 검증이지만, 다른 한편으로는 문서의 정확성에 대한 질문을 제기한다.
README 자체가 이미지화된 컨텍스트를 읽는 에이전트가 작성했다면,
정보 회상 오류가 문서에 조용히 스며들었을 가능성이 있다.
“한 번 실제 실패: 채팅 히스토리의 이미지에서 이름을 잘못 회상했다”는 고백은
바로 이 가능성을 시인한다.

HN 댓글에서는 이 자기 참조가 오히려 읽기 부담으로 드러났다는 지적이 나왔다.
sebmellen은 README의 LLM 특유의 설명 방식이 읽기 고통스럽다며,
캐비어트를 나열하는 문장 하나를 예로 들어 이해에 두 배의 노력이 든다고
꼬집었다[^sebmellen].
trueno는 한발 더 나아가, 만든 사람이 결과물을 자기 언어로 설명하지 못한다는 것은
그 사람이 자신이 조립한 것을 충분히 이해하지 못했다는 신호라고 지적했다[^trueno].
에이전트가 자신의 도구를 사용해 자신의 문서를 쓰는 루프가 신뢰의 근거이자
동시에 의심의 근거가 될 수 있다는 이중성을 이 반응들이 보여준다.

## 비평

### 수익성 주장의 재현 가능성이 좁은 조건에 의존한다

59~70%라는 절감 수치는 인상적이지만, 이 수치가 얼마나 보편적인지는 의문이다.
이 수치는 특정 워크로드(토큰 밀도가 높은 코드/JSON 집중 세션, N=391),
특정 가격 구조(Fable 5 현재 리스트 가격), 특정 캐시 히트율 조건에서 측정됐다.
희소한 산문이 많거나 짧은 대화 위주의 세션에서는 프록시가 “이미지화하지 않음”을 선택하고,
그 경우 절감은 0%이거나 오히려 프록시 오버헤드로 약간 증가한다.

“수익성 게이트”가 있다는 것 자체가, 워크로드에 따라 pxpipe가 전혀 효과적이지 않은
시나리오가 존재함을 인정하는 것이다.
특히 Claude Code를 코드 작업보다 대화형 질의응답에 주로 사용하는 팀에게는
절감 효과가 미미할 수 있다.
“코드 집중 세션”이라는 전제가 명시되지 않은 채 헤드라인 절감률이 제시된다면,
이는 독자를 오도할 수 있다.

이 재현 가능성에 대한 회의는 HN에서 구체적인 반례로 뒷받침됐다.
lpellis는 작년에 OpenAI 모델을 대상으로 같은 방식을 직접 시도했는데,
프롬프트 토큰은 줄었지만 완성 토큰(completion token)이 훨씬 늘어나
결과적으로 더 비싸고 느려졌다고 증언했다[^lpellis].
__hugues도 같은 맥락에서, 입력 토큰은 출력 토큰보다 저렴하기 때문에
이미지에서 텍스트를 다시 뽑아내는 과정(사고 과정을 통한 OCR)이 실제로는
입력 토큰 절감분을 출력 토큰 증가로 상쇄할 수 있다는 정보이론적 의문을
제기했다[^hugues].
pxpipe의 수익성 게이트가 청구 사용량과 반사실 값을 나란히 기록한다는 점은
이런 우려에 대한 답이 될 수 있지만, 게이트가 완성 토큰 증가분까지
정확히 반영하는지는 README만으로는 확인하기 어렵다.

### 손실성(lossiness)의 실제 위험이 충분히 강조되지 않는다

README는 손실성을 솔직하게 인정하면서도, “코딩 세션에서는 에이전트가 파일을 다시 읽기
때문에 문제없다”는 논리로 위험을 상쇄한다.
그러나 이 보호 메커니즘은 코딩 에이전트의 작업 방식을 전제로 한다.
Claude Code를 코드 작성이 아닌 다른 방식—긴 문서 분석, 멀티턴 계획 수립,
정확한 참조가 필요한 리서치—으로 사용하는 경우에는 이 보호가 없다.

12자리 hex 정확 회상이 Fable 5에서 13/15라는 것은, 15번 중 2번은 아무 오류 없이
잘못된 해시나 ID를 사용한다는 의미다.
“오류가 아니라 그럴듯한 잘못된 값”이라는 실패 모드는,
오류가 조용히 전파될 때 특히 위험하다.
SHA, UUID, API 키, 버전 태그 같은 정확성이 핵심인 값이 이미지화된 컨텍스트에 있다면,
pxpipe의 보호 장치(최근 턴 텍스트 유지, keepSharp 옵션)를 의식적으로 사용해야 한다.
이 의식적 설정이 기본값이 아니라는 점에서, 기본 설정으로 사용하는 사용자에게는
위험이 전가된다.

### 가격 정책 변경에 취약한 구조적 리스크를 과소평가한다

pxpipe의 가치 제안 전체가 “비전 토큰이 텍스트 토큰보다 정보 밀도 대비 저렴하다”는
가격 비대칭성에 기반한다.
Anthropic이 이 격차를 인식하고 비전 토큰 가격을 조정하거나,
프롬프트 캐싱 정책을 변경하거나, 이미지 내 텍스트 처리 방식을 바꾼다면
pxpipe의 절감 효과는 하루아침에 사라질 수 있다.

README는 “가격은 움직이고 워크로드는 다르므로, 지속적인 수치는 토큰 절감량 자체”라고
솔직하게 인정한다.
그러나 이 위험의 크기를 과소평가한다.
도구 자체의 코드 복잡성을 생각하면—프록시 설정, keepSharp 관리, 모델 범위 제어—
가격 구조가 바뀌면 이 모든 복잡성이 비용 대비 가치를 잃는다.
pxpipe를 프로덕션 인프라에 통합한 팀에게는 이 전환 비용이 무시할 수 없다.

가격 구조 변화 못지않게 검증하기 어려운 위험도 있다.
troglodytetrain은 절감 효과가 실제로는 프록시가 조용히 더 저렴하고
덜 유능한 모델로 전환한 결과일 수 있다고 경고했다.
겉보기에는 Fable처럼 보이지만 실제로는 아니라는 것이며,
이 경우 사용자는 이미지 변환의 이득 없이 모델 품질 저하라는
추가 비용만 떠안게 된다[^troglodytetrain].
이 의심이 사실인지는 README의 벤치마크만으로는 판별할 수 없고,
사용자가 직접 모델 응답의 품질을 추적해야 확인할 수 있는 문제라는 점에서
검증 부담이 다시 사용자에게 전가된다.

## 인사이트

### 모달리티 차익거래는 AI API 과금 구조 변화의 선행 지표다

pxpipe의 존재는 Anthropic의 텍스트-비전 가격 구조가 정보 밀도를 충분히 반영하지 않는다는
시장의 신호다.
텍스트와 이미지 모두 동일한 처리 비용이 든다면—결국 같은 어텐션 연산을 수행한다—
가격이 달리 책정된다는 것은 역사적 관행과 현재의 과금 모델이 충돌하는 것이다.
pxpipe 같은 도구가 등장하면 Anthropic은 두 가지 중 하나를 선택해야 한다:
비전 토큰 가격을 올리거나, 이 최적화를 공식 기능으로 흡수하거나.

역사적 패턴을 보면, 아키텍처 비효율을 이용하는 최적화 도구는
플랫폼이 그 기능을 공식화하면서 사라지는 경향이 있다.
이 패턴은 클라우드 비용 최적화 도구(AWS Savings Plans 이전의 RI 최적화 도구),
데이터베이스 쿼리 최적화(ORM의 N+1 문제를 우회하는 캐싱 레이어)에서 반복적으로 나타났다.
pxpipe의 접근법이 충분히 효과적이라면, 결국 Anthropic이 프롬프트 압축이나
자동 컨텍스트 이미지화를 API 레이어에서 제공하는 방향으로 이어질 수 있다.

이 과정에서 pxpipe는 시장 실험이자 제품 로드맵 신호다.
AI API 과금 구조는 아직 안정화되지 않았으며,
비용에 민감한 사용자들이 만든 최적화 도구들이 플랫폼의 다음 가격 모델을 암시한다.

다만 HN 반응 중에는 이 아비트라지를 신호가 아니라 낭비로 보는 시각도 있었다.
anigbrowl은 이 방식이 영리하고 실제로 작동하지만,
결국 가격 실패(pricing failure)를 이용하는 편법이라고 비판했다.
독을 가진 뱀에 현상금을 걸었더니 사람들이 뱀 사육을 시작한 사례에 빗대,
이런 아비트라지가 확산되면 낭비를 조장할 뿐이라고 지적했다[^anigbrowl].
가격 비대칭성이 시장 신호라는 해석과, 그것이 조장하는 낭비적 사용이라는
비판은 서로 배타적이지 않다.
플랫폼이 격차를 흡수하기 전까지, 그 격차를 이용하는 도구들이 늘어날수록
불필요한 연산과 지연이 함께 늘어난다는 부작용은 신호로서의 가치와
별개로 존재한다.

### 손실성을 투명하게 측정하는 방식이 AI 도구 신뢰의 새 기준이 될 수 있다

pxpipe가 다른 AI 비용 최적화 도구와 구별되는 점은 손실성의 정량화 방식이다.
“12자리 hex 정확 회상: Fable 5 기준 13/15, Opus 기준 0/15”처럼 구체적인 수치로
실패 모드를 공개하고, 모든 실패가 예측 가능한 글리프 혼동 행렬로 설명된다는 것은
이례적인 투명성이다.

대부분의 AI 도구는 정확도 수치를 강점으로 제시하고 실패 사례를 숨기거나 희석한다.
pxpipe는 반대다—실패를 측정하고, 실패 패턴을 문서화하며, 실패가 예측 가능한 경우와
그렇지 않은 경우를 구분한다.
“verbatim-risk guard is not built yet”이라는 문장은 미완성을 솔직히 인정한다.
이 수준의 투명성은 사용자가 도구를 어느 조건에서 신뢰할 수 있는지 스스로 판단하게 만든다.

이 접근 방식이 확산된다면 AI 도구 평가의 기준이 높아진다.
“정확도 몇 퍼센트”라는 단일 수치보다 “어떤 조건에서 어떤 방식으로 실패하는가”가
더 중요한 평가 기준이 될 것이다.
pxpipe의 FINDINGS.md와 LEGIBILITY-AUDIT 문서는 그 기준의 구체적 형식을 제안한다.

### 에이전트가 자신의 개발 도구를 사용하는 루프가 도구 설계에 미치는 영향

pxpipe가 자체 pxpipe를 통해 개발됐다는 사실은 흥미로운 피드백 루프를 만든다.
에이전트가 이미지화된 히스토리를 읽으면서 코드를 작성했다면,
에이전트가 경험한 불편함—회상 실패, 포맷 불일치—이 코드에 반영됐을 가능성이 높다.
“한 번 실패: 채팅 히스토리에서 이름을 잘못 회상했다”는 실제 실패 사례가 문서화된 것은
이 피드백 루프가 작동했다는 증거다.

도구 개발에 도구 자체를 사용하는 패턴—bootstrap 또는 dogfooding—은
컴파일러 개발(GCC가 GCC로 컴파일됨), 운영체제 개발(Linux 커널 개발에 Linux 사용)에서
오랜 전통이 있다.
AI 에이전트 도구에서 이 패턴이 등장한 것은, 에이전트가 복잡한 상태를 유지하면서
자기 개선 루프에 들어갈 수 있는 수준의 능력에 도달했다는 신호다.
pxpipe의 경우 이 루프가 의도적으로 설계된 것이 아니라 비용 절감의 부산물로 나타났다는 점이
더 흥미롭다.

이 패턴이 확산되면 에이전트 도구의 개발 사이클이 빨라진다.
도구를 사용하면서 도구의 한계를 직접 경험하는 에이전트가 그 한계를 즉시 수정한다면,
인간 개발자의 피드백 루프보다 훨씬 짧은 주기로 개선이 이뤄진다.
물론 에이전트가 “조용한 실패”를 인식하지 못한다면 이 루프는 편향된 방향으로 수렴할 위험이 있다.
