# pdf-inspector: Firecrawl의 Rust PDF 분류·추출 라이브러리

<https://github.com/firecrawl/pdf-inspector>

## 소개

pdf-inspector는 Firecrawl이 개발한 Rust 라이브러리로, PDF를 OCR 없이
200밀리초 이내에 분류하고 텍스트를 추출해 마크다운으로 변환한다.
핵심 문제 의식은 명확하다. PDF의 약 54%는 이미 텍스트를 포함하고 있어
OCR이 필요 없지만, 기존 파이프라인은 모든 PDF를 비용이 높은 OCR 서비스로
전송하는 경우가 많다. pdf-inspector는 이 분류를 먼저 수행해 불필요한
OCR 호출을 차단한다.

분류 결과는 네 가지다. `TextBased`(텍스트 기반), `Scanned`(스캔본),
`ImageBased`(이미지 기반), `Mixed`(혼합). 분류는 PDF 전체를 파싱하지 않고
xref 테이블과 페이지 트리만 읽은 뒤 콘텐츠 스트림에서 텍스트 연산자(`Tj`/`TJ`)와
이미지 연산자(`Do`)의 존재 여부를 샘플링해 10~50밀리초에 완료한다.
300페이지 이상의 대용량 PDF도 밀리초 단위로 처리한다.
분류 결과에는 0.0~1.0 신뢰도 점수와 OCR이 필요한 특정 페이지 목록이 포함돼
전체 문서가 아닌 페이지 단위 OCR 라우팅이 가능하다.

텍스트 추출은 위치 인식 방식으로 동작한다. 폰트 정보, X/Y 좌표, 다중 컬럼
레이아웃 자동 감지, RTL 텍스트 지원을 포함한다. 마크다운 변환은
폰트 크기 비율로 H1~H4 헤딩을 감지하고, 직사각형 기반 및 텍스트 정렬
기반 이중 모드 테이블 감지, 굵기/기울임 서식, 하이퍼링크 변환,
페이지 번호 필터링 등을 처리한다. CID 폰트의 ToUnicode CMap 디코딩도 지원한다.

Python(`pip install maturin` + PyO3), Node.js(`@firecrawl/pdf-inspector` npm 패키지),
Rust 의존성, CLI(`pdf2md`, `detect-pdf`) 네 가지 인터페이스를 제공한다.
외부 서비스나 ML 모델 없이 `lopdf` 단일 의존성만 사용한다.

벤치마크에서는 200개 PDF 코퍼스 기준 4초로 직접 텍스트 추출 엔진 중 가장 빠르다.
종합 점수 0.78로 opendataloader(0.84)에 이어 2위이며, pymupdf4llm(0.73),
markitdown(0.58)을 앞선다. OCR·ML 기반 엔진(docling, marker, mineru)은
0.83~0.88이지만 같은 코퍼스에 2~180분이 걸린다.

## 분석

### PDF 분류가 먼저인 아키텍처의 경제학

pdf-inspector의 설계 핵심은 분류-우선(classification-first) 파이프라인이다.

```text
PDF 도착
  → 분류 (~20ms)
  → TextBased + 높은 신뢰도?
      YES → 로컬 추출 (~150ms), 완료
      NO  → OCR 서비스 (2-10s)
```

이 설계가 의미 있는 이유는 PDF 처리에서 OCR과 비OCR의 비용 차이가 크기 때문이다.
클라우드 OCR 서비스는 요청당 수 센트에서 수십 센트 수준이다.
분류 비용이 거의 0에 가깝고, PDF의 54%가 텍스트 기반이라면
전체 비용을 절반 이상 줄일 수 있다는 계산이 나온다.

분류 결과가 페이지 단위(`pages_needing_ocr` 목록)를 반환하는 것도 중요하다.
혼합 PDF에서 텍스트 페이지는 로컬 처리, 스캔 페이지만 OCR로 전송하는
세밀한 라우팅이 가능해진다. 이것이 단순 전체/부분 분기와 다른 지점이다.

### Rust 선택과 단일 의존성 철학

ML 모델 없이 순수 Rust로 구현하고 `lopdf` 단일 의존성만 사용하는 선택은
운영 환경에서 중요한 의미를 갖는다. 파이썬 기반 PDF 처리 라이브러리들은
종종 수십 개의 의존성과 큰 Docker 이미지 크기를 요구한다.
Rust 바이너리는 컴파일 후 단일 실행 파일이 되고, PyO3와 napi-rs로
Python과 Node.js 바인딩을 만들어 기존 스택과 통합한다.

“단일 문서 로드(single document load)” 원칙도 주목할 만하다.
분류와 추출 단계가 동일한 파싱 결과를 공유해 I/O를 한 번만 수행한다.
파이프라인에서 같은 PDF를 분류기와 추출기가 각각 열어 파싱하는
중복 작업을 피하는 설계다.

### 테이블 감지의 이중 모드 전략

테이블 감지에 직사각형 기반(rectangle-based)과 휴리스틱(heuristic) 두 모드를
동시에 운영하는 전략은 PDF 테이블의 다양성에 대한 현실적 대응이다.

전자 문서로 생성된 PDF는 테이블 셀 경계를 그리기 연산(drawing ops)으로
명시적으로 표현하는 경우가 많다. 직사각형 기반 감지는 이를 활용한다.
반면 많은 PDF가 테이블을 단순히 열 정렬된 텍스트로 표현한다.
이 경우 텍스트 X좌표의 통계적 정렬을 분석하는 휴리스틱이 필요하다.
두 모드를 병행해 커버리지를 높인다.

벤치마크에서 테이블 TEDS 점수 0.59가 OCR 기반 엔진에 비해 낮은 이유도
여기서 나온다. OCR 엔진은 시각적 테이블 구조를 인식할 수 있지만,
직접 텍스트 추출 엔진은 PDF 내부 구조에 의존한다.
표가 텍스트 정렬로만 표현된 경우 한계가 있다.

### 스캔 전략의 유연성

네 가지 스캔 전략(`EarlyExit`, `Full`, `Sample(n)`, `Pages(vec)`)은
사용 컨텍스트에 따른 속도-정확도 트레이드오프를 명시적으로 제공한다.
기본값인 `EarlyExit`는 텍스트 페이지를 빠르게 확인하고 싶은 파이프라인에 최적이다.
대용량 PDF에서 `Sample(n)`은 일부 페이지만 검사해 처리 시간을 줄인다.
`Pages(vec)`는 호출자가 이미 어떤 페이지를 확인해야 하는지 알 때
불필요한 스캔을 피한다. 이 유연성은 라이브러리가 범용 도구임을 보여준다.

## 비평

### 강점: 벤치마크의 정직성

README의 벤치마크가 특히 신뢰할 만한 이유는 약점을 숨기지 않는다는 점이다.
헤딩 감지(0.57)가 opendataloader(0.74)에 비해 뒤처지는 이유,
테이블 감지가 OCR 기반 엔진에 뒤지는 이유를 구체적으로 설명한다.
“볼드 텍스트가 본문과 같은 크기로 헤딩 역할을 하는 PDF”,
“미묘하게 큰 텍스트가 헤딩인 경우”처럼 실제 실패 사례를 명시한다.
약점을 안다는 것은 적절한 사용 사례를 판단하는 데 필수적이다.

### 한계: 헤딩 감지의 구조적 제약

폰트 크기 비율로 헤딩을 감지하는 접근법은 PDF 헤딩 표현 방식의
다양성을 완전히 처리하지 못한다. 많은 PDF가 헤딩과 본문에 동일한 크기의
폰트를 사용하면서 색상, 간격, 위치로 구조를 표현한다.
색상 정보나 단락 간격을 보조 신호로 활용하는 기능이 없는 것은
헤딩 감지 개선의 주요 장벽이다.

### 보완이 필요한 관점: 언어별 성능 차이

벤치마크가 영어 PDF를 주로 다루는 opendataloader-bench 코퍼스를 사용한다.
CJK(한국어, 중국어, 일본어), 아랍어, 히브리어 PDF에서의 성능이
명시적으로 다루어지지 않는다. CID 폰트 지원과 RTL 텍스트 지원이 언급되지만,
이 언어들의 실제 처리 품질에 대한 벤치마크는 없다.
다국어 PDF를 처리하는 파이프라인에서의 실제 성능은 별도 검증이 필요하다.

## 인사이트

### PDF는 문서 형식이 아니라 렌더링 명령어 집합이다

PDF를 처리하는 것이 왜 어려운지 이해하려면 PDF의 본질을 알아야 한다.
PDF는 “이 텍스트를 이 위치에 이 폰트로 그려라”라는 렌더링 명령어의 집합이다.
논리적 구조 — 헤딩, 단락, 테이블, 목록 — 는 PDF 명세에 존재하지 않는다.
다시 말해, PDF를 마크다운으로 변환하는 것은 렌더링 결과에서
문서 구조를 역공학(reverse engineering)하는 작업이다.

이 역공학이 얼마나 어려운지는 pdf-inspector의 아키텍처에 반영돼 있다.
헤딩 감지는 폰트 크기 통계를, 테이블 감지는 그리기 연산과 텍스트 정렬을,
다중 컬럼 감지는 X좌표 분포를 분석한다. 이 모든 것이 “렌더링 정보에서
의미를 추론”하는 과정이다. PDF/UA(Universal Accessibility) 명세가
태그된 PDF 구조를 요구하는 이유도 여기 있다. 태그 없는 PDF는
접근성 도구에도, 텍스트 추출 도구에도 동일하게 어렵다.

이 구조적 한계는 OCR 기반 엔진이 직접 텍스트 추출 엔진보다
테이블과 헤딩에서 더 좋은 점수를 받는 이유를 설명한다.
OCR 엔진은 렌더링 결과 이미지를 보고 시각적으로 구조를 인식한다.
텍스트 추출 엔진은 렌더링 명령어를 보고 구조를 추론한다.
시각적 인식이 논리적 추론보다 실제 문서 구조에 더 가까운 역설이다.

### 분류-우선 아키텍처는 AI 파이프라인의 비용 구조를 재설계한다

AI 애플리케이션에서 비용 최적화의 핵심 패턴 중 하나는
“비싼 연산을 마지막에”다. 모든 입력을 동일하게 처리하는 것이 아니라
먼저 분류해 처리 경로를 결정한다. pdf-inspector의 사용 사례가
이 패턴의 교과서적 구현이다.

이 패턴의 확장 가능성을 생각해보면, PDF 분류가 시작점이다.
이미지에서도 동일한 논리가 적용된다. 텍스트가 없는 순수 사진,
텍스트가 포함된 스크린샷, 도표, 표 이미지는 다른 처리 경로가 최적이다.
오디오에서도 음성 구간과 비음성 구간을 먼저 구분하면
전사(transcription) 비용을 줄일 수 있다. 모달리티 불문하고
“분류 먼저, 처리 나중”이라는 원칙은 일반화 가능하다.

더 중요한 점은 분류의 정확도 요구사항이 처리보다 낮다는 것이다.
분류가 틀려 OCR이 필요 없는 PDF를 OCR로 보내도 결과의 정확도는 높아진다.
단지 비용이 낭비된다. 반대로 OCR이 필요한 PDF를 로컬 처리로 보내면
결과가 나빠진다. 이 비대칭성 때문에 분류기의 오류 방향이 중요하다.
pdf-inspector가 신뢰도 점수를 제공하는 이유도 호출자가 이 트레이드오프를
직접 조정할 수 있게 하기 위해서다.

### 오픈소스 PDF 처리 생태계의 취약성과 Rust의 역할

파이썬 생태계에서 PDF 처리는 오랫동안 묵은 문제였다.
PyPDF2, pdfminer, pdfplumber, pymupdf(fitz) — 이 라이브러리들은
각각 다른 품질의 추출 결과를 내고, 유지보수 상태가 불안정하며,
C 라이브러리에 의존하는 경우 배포 복잡성이 높다.

Rust가 이 공간에 진입하는 것은 자연스럽다. PDF 파싱은 바이트 수준의
정밀한 처리가 필요하고, 메모리 안전성이 중요하며, 바인딩으로
여러 언어에서 사용할 수 있어야 한다. PyO3와 napi-rs는 Rust 라이브러리를
파이썬과 Node.js에서 네이티브 확장처럼 사용할 수 있게 만드는
성숙한 도구다. pdf-inspector는 이 조합의 실용적 증거다.

흥미로운 것은 Firecrawl 같은 상업적 회사가 핵심 인프라를 오픈소스로
공개한다는 사실이다. 경쟁 우위가 PDF 파싱 자체가 아니라
그 위에 구축된 서비스에 있기 때문이다. 이 공개가 커뮤니티 개선을
유도하고, 생태계 표준이 되는 선점 효과를 노린다.
pdf-inspector의 벤치마크가 경쟁 도구들과의 비교를 투명하게 공개하는 것도
이 전략의 일부다. 표준이 되려면 측정 기준도 함께 제시해야 한다.
