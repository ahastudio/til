# OpenTag

<https://github.com/CopilotKit/opentag>

## 소개

OpenTag는 앤트로픽의 Claude Tag를 Slack에서 사용할 수 있게 해주는
오픈소스 대안이다.
CopilotKit이 만든 봇 SDK 위에 구축되었으며,
“러너타임을 직접 소유하고, 원하는 모델을 붙이고,
원하는 도구에 연결하라”는 자체 호스팅 철학을 내세운다.
사용자가 Slack 채널에서 봇을 태그하면 에이전트가 스레드를 읽고,
응답을 생성하고, 도구를 실행하고, 그 결과를 대화 안에 그대로
렌더링해서 보여준다.

## 아키텍처

OpenTag는 두 개의 프로세스로 구성된다.
`@copilotkit/runtime`으로 구동되는 에이전트 백엔드가 LLM과 도구
실행을 담당하고, `@copilotkit/bot-slack` 어댑터를 사용하는
봇 서비스가 Slack과의 연결을 담당한다.
핵심 패키지는 `@copilotkit/bot`, `@copilotkit/runtime`,
`@copilotkit/bot-ui`, 그리고 플랫폼별 어댑터로 나뉘어 있으며,
이 구조 덕분에 같은 코드베이스로 Slack뿐 아니라 Discord, Telegram,
WhatsApp에도 동일한 에이전트를 붙일 수 있다.
스레드 상태를 재시작 이후에도 유지하고 싶다면 Redis 기반
영속 저장소를 선택적으로 붙일 수 있다.

## 주요 기능

- **제너레이티브 UI** — 표나 차트 같은 분석 결과를 대화 안에
  직접 렌더링한다.
- **휴먼인더루프(human-in-the-loop)** — 티켓 생성처럼 민감한 작업을
  실행하기 전에 사용자 승인을 거치는 게이트를 둘 수 있다.
- **스레드 기반 대화** — 채널 대화의 맥락을 유지한 채 작업을 이어간다.
- **커스터마이징** — `runtime.ts` 안의 시스템 프롬프트 하나로
  에이전트의 동작 방식을 제어한다.
- **멀티플랫폼 어댑터** — 같은 코드베이스로 여러 메신저 플랫폼을
  지원한다.

## 사용법

1. api.slack.com에서 매니페스트를 이용해 Slack 앱을 생성한다.
2. Bot Token, App Token, LLM API 키 세 가지를 `.env`에 설정한다.
3. `pnpm install` 이후 `pnpm --filter slack-example runtime`과
   `pnpm --filter slack-example dev`를 각각 실행해
   에이전트 백엔드와 봇 서비스를 띄운다.
4. Slack에서 `@OpenTag [요청 내용]`으로 호출한다.

## 라이선스 및 상태

MIT 라이선스로 배포되며, 이 글을 쓰는 시점 기준 606개의 스타와
72개의 포크, 22개의 커밋을 보유하고 있다.
CopilotKit은 직접 호스팅을 원하지 않는 사용자를 위한 매니지드
서비스도 대기자 명단 형태로 준비 중이라고 밝히고 있다.

## 분석

### 이 프로젝트가 겨냥하는 지점은 무엇인가

OpenTag의 존재 이유는 Claude Tag가 앤트로픽의 서비스에 강하게
결합된 구조라는 점에서 나온다.
Claude Tag를 쓰려면 앤트로픽의 인프라와 요금제, 데이터 처리 정책을
그대로 받아들여야 한다.
OpenTag는 같은 사용자 경험, 즉 “Slack에서 태그하면 에이전트가
알아서 처리해준다”는 경험을 유지하면서도 모델 선택권과 데이터
소유권을 사용자에게 되돌려주는 것을 목표로 한다.
이는 상용 서비스가 먼저 시장을 검증한 뒤 오픈소스 진영이
같은 경험을 재현하는, 오픈소스 생태계에서 반복적으로 나타나는
패턴이다.

### 두 프로세스 분리 구조가 갖는 의미

에이전트 백엔드와 봇 서비스를 별도 프로세스로 분리한 설계는
단순한 코드 정리 이상의 의미를 갖는다.
봇 서비스는 플랫폼(Slack, Discord 등)에 특화된 연결 로직만
담당하고, 실제 추론과 도구 실행은 플랫폼과 무관한 에이전트
백엔드에서 이뤄진다.
이 분리 덕분에 새로운 메신저 플랫폼을 지원하려면 어댑터 하나만
추가하면 되고, 에이전트의 핵심 로직은 건드릴 필요가 없다.
이는 결국 “봇 프레임워크”가 아니라 “에이전트 런타임 플러스
어댑터 계층”이라는 아키텍처를 지향하고 있다는 뜻이다.

### 휴먼인더루프 게이트가 자체 호스팅과 만나는 지점

승인 게이트 기능은 상용 서비스에서도 흔히 제공하지만,
자체 호스팅 환경에서는 그 의미가 다르다.
상용 서비스에서는 벤더가 정한 정책 안에서 승인 흐름이 동작하지만,
자체 호스팅에서는 조직이 직접 어떤 작업에 게이트를 걸지,
누가 승인 권한을 가질지를 코드 수준에서 결정할 수 있다.
이는 규제가 엄격한 조직이 상용 AI 에이전트 서비스 대신
자체 호스팅형 대안을 찾는 이유와 정확히 일치한다.

### 606개의 스타와 22개의 커밋이라는 숫자가 말해주는 것

짧은 커밋 이력에 비해 스타 수가 상대적으로 많다는 것은,
이 프로젝트가 오랜 기간 유기적으로 성장했다기보다
Claude Tag 발표 직후의 화제성에 힘입어 빠르게 주목받았을
가능성을 시사한다.
초기 관심이 실제 프로덕션 채택으로 이어지는지는
커밋 이력과 이슈 대응 속도가 앞으로 어떻게 쌓이는지를
지켜봐야 판단할 수 있다.

## 비평

### “오픈소스 대안”이라는 프레이밍은 실제 격차를 가린다

OpenTag를 Claude Tag의 오픈소스 대안으로 소개하는 것은
마케팅으로서는 효과적이지만, 실제 기능 격차를 흐릴 위험이 있다.
Claude Tag는 채널 대화에서 암묵적 지식을 축적하는 시간 기반
학습이나 태그 없이도 개입하는 앰비언트 주도성 같은 기능을
갖추고 있는데, OpenTag의 README에는 이런 고급 기능에 대한
언급이 보이지 않는다.
“대안”이라는 표현이 두 제품을 동급으로 보이게 만들지만,
실제로는 기본적인 태그-응답 흐름만 재현한 초기 단계 구현에
가깝다.

### 자체 호스팅의 책임을 과소평가하고 있다

README는 “러너타임을 소유하라”는 장점만 강조하고,
그 소유에 따르는 운영 부담은 충분히 다루지 않는다.
LLM API 키 관리, 두 프로세스의 가동 상태 모니터링, Redis
영속성 계층의 장애 대응까지 모두 사용자 조직의 몫이 된다.
상용 서비스가 흡수해주던 운영 복잡도를 자체 호스팅으로 가져오는
선택에는 반드시 그만한 인력과 운영 역량이 뒷받침되어야 하는데,
이 트레이드오프에 대한 설명이 문서에서 충분히 드러나지 않는다.

### 멀티플랫폼 지원은 아직 검증되지 않은 약속에 가깝다

Discord, Telegram, WhatsApp 어댑터가 언급되어 있지만,
실제 예제와 문서는 Slack 중심으로만 구성되어 있다.
아키텍처가 멀티플랫폼을 지향한다는 것과, 실제로 각 플랫폼에서
프로덕션 수준으로 검증된 어댑터가 존재한다는 것은 다른 문제다.
멀티플랫폼 지원을 핵심 가치로 내세우려면 최소한 하나 이상의
비Slack 플랫폼에 대한 실사용 예제가 필요한데, 현재로서는
그 근거가 부족하다.

## 인사이트

### 오픈소스 대안의 등장 속도가 상용 AI 에이전트 서비스의 해자를 시험한다

Claude Tag 공개 이후 짧은 시간 안에 오픈소스 대안이 등장했다는
사실 자체가 중요한 신호다.
이는 상용 AI 에이전트 서비스의 핵심 가치가 독자적인 기술이 아니라
“통합의 매끄러움”에 있을 때, 그 매끄러움을 오픈소스 커뮤니티가
얼마나 빠르게 재현할 수 있는지를 보여주는 사례로 읽을 수 있다.
앞으로 상용 서비스들은 단순한 통합 경험이 아니라, 시간에 따라
누적되는 학습이나 조직 전체의 맥락 이해처럼 쉽게 복제되지 않는
기능으로 차별화 지점을 옮겨갈 가능성이 크다.

### 자체 호스팅 선택은 데이터 주권 논의와 직결된다

기업이 OpenTag 같은 자체 호스팅 대안을 선택하는 이유는
비용 절감보다 데이터 주권 문제일 가능성이 높다.
Slack 대화 내용, 특히 내부 코드베이스나 민감한 업무 정보가
외부 벤더의 인프라를 거치는 것을 꺼리는 조직에게는,
기능이 다소 부족하더라도 데이터가 자사 인프라를 벗어나지 않는다는
사실 자체가 결정적 이점이 된다.
이는 AI 에이전트 서비스 시장이 기능 경쟁뿐 아니라
데이터 거버넌스 경쟁으로도 분화되고 있음을 보여주는 사례다.

### 봇 프레임워크의 표준화 경쟁이 다음 전장이 될 수 있다

CopilotKit의 어댑터 기반 아키텍처는 특정 메신저 플랫폼에
종속되지 않는 “에이전트 런타임” 개념을 제시한다.
만약 이런 구조가 업계 표준으로 자리 잡는다면, 향후 경쟁은
개별 챗봇 기능이 아니라 “어떤 런타임이 더 많은 플랫폼
어댑터와 도구 생태계를 갖추고 있는가”라는 플랫폼 경쟁으로
옮겨갈 가능성이 있다.
이는 초기 웹 프레임워크들이 특정 기능이 아니라 미들웨어와
플러그인 생태계의 크기로 경쟁했던 역사와 유사한 구조다.
