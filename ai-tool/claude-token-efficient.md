# claude-token-efficient — CLAUDE.md 한 파일로 출력 장황함을 줄이는 프로젝트

<https://github.com/drona23/claude-token-efficient>

HN 토론: <https://news.ycombinator.com/item?id=47581701> (471점, 162개 댓글)

## 소개

CLAUDE.md 파일 하나를 프로젝트 루트에 놓으면 Claude의 응답이 간결해진다는 전제로 만들어진 저장소다.
코드 변경 없이 파일 하나만 추가하는 드롭인(drop-in) 방식이다.
2026년 3월 30일 생성돼 5,773 스타, 439 포크를 기록하고 있다.

이 파일이 겨냥하는 Claude Code의 기본 행동 8가지는 다음과 같다:
맥락을 읽기 전에 코딩을 시작하는 것, 장황한 응답, 대형 파일 불필요한 재작성,
같은 파일 반복 읽기, 검증 없이 완료 선언, 아첨성 채팅(sycophantic chatter),
과도하게 추상화된 해결책, 프롬프트 충돌 시 혼란.

기본 사용법은 단순하다.

```bash
curl -o CLAUDE.md https://raw.githubusercontent.com/drona23/claude-token-efficient/main/CLAUDE.md
```

프로필은 5종이다: 범용(`CLAUDE.md`), 벤치마크(`CLAUDE.benchmark.md`),
개발 프로젝트(`CLAUDE.coding.md`), 자동화 파이프라인(`CLAUDE.agents.md`),
데이터 분석(`CLAUDE.analysis.md`).
버전별 설정 세트도 3가지가 있다: J-drona23-v5(멀티파일 구조, 툴콜 50회),
K-drona23-v6(원샷 실행, 50회), M-drona23-v8(초경량 최소 턴, 20회).

## 분석

### 헤드라인 63% 절감과 실측 4~12% 사이의 간극이 담긴 맥락

README에는 두 개의 벤치마크가 나란히 있다.
하나는 5개 프롬프트에서 63% 단어 감소라는 초기 수치고,
다른 하나는 haiku ~4%, sonnet ~12%, opus ~7% 출력 토큰 감소라는 재현 가능한 벤치마크다.
이 두 수치의 차이가 이 저장소의 핵심 진실이다.

63%는 엄격한 규칙 프로파일로 측정한 단어 수 감소이고,
4~12%는 현재의 최소화된 CLAUDE.md로 측정한 실제 출력 토큰 감소다.
더 중요한 것은 시맨틱 평가 결과다: “현재 모델 기준 베이스라인이 이미
preamble 0%, sycophancy 0%, 'as an AI' 0%, smart quotes 0%를 보인다.
이런 행동을 겨냥한 규칙은 입력 비용만 들이고 출력을 바꾸지 못한다.”
이 발견은 이 저장소가 존재해야 할 이유 중 상당 부분이 이미 해소됐다는 것을 의미한다.

이 시맨틱 평가 결과가 README의 깊은 곳에 묻혀 있다는 점은 중요하다.
저자가 이 발견을 숨기지 않고 공개한 것은 정직함이지만,
헤드라인과 주요 섹션은 여전히 63% 수치를 중심으로 구성돼 있다.
독자가 이 벤치마크를 얼마나 깊이 읽느냐에 따라 완전히 다른 인상을 받을 수 있다.

### CLAUDE.md 컴포지션 패턴이 소프트웨어 설정 관리의 확장이다

이 프로젝트가 무심코 제안하는 가장 흥미로운 패턴은 규칙 파일의 컴포지션이다.
Claude는 전역(`~/.claude/CLAUDE.md`), 프로젝트 루트, 하위 디렉토리 CLAUDE.md를
동시에 읽는다. 이것은 `.editorconfig`, `.gitignore`, `.env` 파일이 계층적으로
적용되는 방식과 동형이다.

이 패턴의 실용적 의미는 크다.
전역 파일에는 “tone, format, ASCII rules” 같은 범용 선호도를,
프로젝트 파일에는 “`/config` 수정 전 확인 요청” 같은 프로젝트 제약을,
하위 디렉토리 파일에는 작업별 규칙을 넣으면 단일 파일 비대화를 막을 수 있다.
“규칙을 적용 지점에 가깝게 유지한다”는 원칙은 소프트웨어 공학의
관심사 분리(separation of concerns)를 AI 행동 제어에 적용한 것이다.

이 설계는 하나의 거대한 CLAUDE.md가 아니라 계층적 설정 시스템을 향한다.
각 수준이 다른 수준을 상속하거나 재정의하는 구조는,
팀이 공유하는 기준과 개인/프로젝트별 커스터마이제이션을 동시에 관리할 수 있게 한다.

### 커뮤니티 GitHub 이슈를 규칙 소스로 활용하는 집단 행동 교정

이 저장소의 References 섹션은 독특하다.
각 규칙이 어떤 GitHub 이슈나 커뮤니티 보고에서 왔는지 명시하고 있다:
“Claude Code 지나친 동조 동의” (350+ upvotes), “sycophantic 행동 이슈”,
“--quiet 플래그 요청”, “도구 출력 장황함 이슈” 등.
규칙의 출처가 모델 제작사의 설계 결정이 아니라 사용자 고통점이다.

이것은 모델 정렬(model alignment)의 바텀업 버전이다.
Anthropic의 공식 가이드라인이나 RLHF가 탑다운 방식으로 모델 행동을 교정한다면,
이 저장소는 사용자들이 경험한 구체적 실패를 규칙으로 코딩해 프롬프트 레이어에서
교정을 시도한다.
이 접근의 한계는 규칙이 확률을 바꿀 뿐 행동을 보장하지 못한다는 것이지만,
GitHub 이슈 -> 규칙 -> 버전 관리 -> 커뮤니티 기여 파이프라인이 형성됐다는 점 자체가
새로운 집단적 모델 행동 관리 방식의 실험이다.

## 비평

### 헤드라인 수치와 실측 수치의 간극이 독자를 오도할 수 있다

5-프롬프트 63% 단어 감소는 방법론 주석에서 “단방향 지표이며 통계적으로 통제된 연구가
아니다”라고 인정한다. 그러나 이 수치가 제목, 첫 번째 표, Before/After 사례의 헤드라인을
장식한다. 더 신뢰할 수 있는 수치(haiku 4%, sonnet 12%, opus 7%)는 세 번째 벤치마크
섹션에 묻혀 있다.

HN에서 monooso는 이 프로젝트의 전제 자체를 흔드는 데이터를 제시했다.[^monooso]
OpenRouter 프로그래밍 카테고리의 실사용 데이터에서 출력 토큰은 전체의 4.0%에 불과하고,
입력 토큰이 93.4%, 추론 토큰이 2.5%를 차지한다.
“출력 토큰을 줄여 비용을 절감한다”는 이 프로젝트의 핵심 전제가
실제 비용 구조와 얼마나 일치하는지 재검토가 필요하다.

niklassheth는 벤치마크 설계 자체를 비판했다.[^niklassheth]
“정확도 비교 없이 출력 토큰만 비교한다. '항상 한 단어로 대답하라'는 프롬프트로도 이 벤치마크를
압살할 수 있다.” Asmod4n은 Issue #1을 인용하며 “아무 지시 없을 때 토큰 효율이 가장 높다”는
외부 측정 결과를 언급했다.[^Asmod4n]
저자가 정직하게 공개했다는 점은 인정할 만하지만,
구조상 독자가 이 발견에 도달하려면 능동적으로 깊이 읽어야 한다.

### “답을 먼저, 추론은 나중에” 규칙이 모델의 작동 원리에 역행한다

이 저장소의 규칙 중 “답은 항상 첫 번째 줄, 추론은 그 다음”이라는 지시가 여러 댓글에서
집중 비판을 받았다.
xianshou은 LLM의 자기회귀(autoregressive) 특성을 지적했다.[^xianshou]
“LLM은 이미 나온 내용을 완성하는 방식으로 작동한다. 답을 먼저 쓰게 하면
그 답에 의해 이후 추론이 확증 편향으로 고정된다.”
adastra22도 동일한 관찰을 공유했다: “답을 먼저 쓰게 하면 만들어낸 답을 나중에 정당화한다.
추론을 먼저 하게 하면 진짜 브레인스토밍 후 합리적 답에 도달한다.”[^adastra22]

btown은 이보다 더 근본적인 질문을 제기했다.[^btown]
“코드를 작성하면서 왜 그렇게 하는지 설명하는 장황함이, 실제로는 세션이 커질수록
컨텍스트 일관성을 유지하게 해서 전체 토큰을 절감할 수도 있지 않을까?”
장황한 자기 설명이 비용이 아니라 투자일 수 있다는 것이다.
이 가설은 이 저장소가 다루지 않은 채로 남겨둔 질문이다.

### 규칙 파일의 실제 효과가 모델 버전 의존성이 높다는 사실을 충분히 강조하지 않는다

시맨틱 평가가 드러낸 것처럼, 2026년 현재 모델(haiku/sonnet/opus)은
이미 preamble, sycophancy, smart quotes를 기본적으로 0%로 보인다.
이것은 Anthropic이 모델 훈련을 통해 이미 이 문제들을 상당 부분 해결했다는 뜻이다.
그렇다면 이 규칙들을 CLAUDE.md에 넣는 것은 입력 토큰 비용만 들이는 죽은 코드다.

이 함의는 중요하다: 모델이 업데이트될수록 CLAUDE.md 규칙의 일부는 효과가 없어지거나
오히려 비용만 드는 쓰레기가 된다. 규칙 파일도 버전 관리와 주기적 감사가 필요하다.
README는 “필요없는 규칙을 잘라내라”는 권고를 포함하지만,
어떤 규칙이 현재 모델에서 여전히 효과적인지 확인하는 방법론을 제시하지 않는다.
`benchmark/run.py`를 직접 실행해야 하는데, 이 작업을 할 사용자는 많지 않다.

### 확장 시나리오의 절감 수치가 실질적이지 않다

“100 프롬프트/일 → 월 $0.86 절감”이라는 수치는 설득력이 약하다.
이 계산이 ~4~12% 출력 토큰 감소에 기반한다면, 그리고 sonnet 기준 월 $8.64라는
수치도 큰 규모의 프로덕션 파이프라인 기준이라면,
이 파일을 관리하고 적용하는 엔지니어링 비용 대비 ROI가 명확하지 않다.

특히 “CLAUDE.md 자체가 모든 메시지에 입력 토큰을 소비한다”는 경고와 함께 보면,
실제 네트 절감은 계산보다 낮다. 프롬프트 캐싱이 잘 작동하는 환경에서는
CLAUDE.md의 입력 비용이 상쇄되지만, 세션이 자주 초기화되는 파이프라인에서는
캐시 이점이 줄어든다. 이 조건들이 명시되지 않은 채 절감 수치만 제시된다.

## 인사이트

### 모델 능력 향상이 프롬프트 엔지니어링 자산을 빠르게 구식으로 만든다

이 저장소가 보여주는 가장 중요한 패턴은 기술적 해결책이 아니라 그 구식화(obsolescence) 속도다.
2025년 중반에 실제 문제였던 행동들이 2026년 현재 모델에서는 기본값으로 수정됐다.
Claude Code GitHub 이슈 #3382(“You're absolutely right about everything”에 350+ upvotes)가
실제 고통점이었다면, 그 문제는 이미 사라졌다.
그러나 그 문제를 겨냥해 만들어진 규칙은 여전히 CLAUDE.md에 남아 있고
매 메시지마다 입력 토큰을 소비한다.

이것은 프롬프트 엔지니어링 자산의 부채화(technical debt)다.
코드에서 죽은 코드(dead code)가 축적되듯, CLAUDE.md 규칙도
더 이상 필요없는 행동 제약이 축적된다.
모델이 빠르게 개선될수록 이 부채 축적 속도가 빨라진다.
“프롬프트 엔지니어링”이라는 분야 자체가 이 구식화 문제를 내포하고 있으며,
어떤 프롬프트가 현재에도 유효한지 지속적으로 검증하는 메커니즘이 필요하다.

이 관찰은 더 일반적인 원칙으로 확장된다.
모델 능력이 불균등하게 발전하기 때문에, 6개월 전에 유효했던 프롬프트 전략이
지금은 오히려 노이즈가 될 수 있다. 따라서 프롬프트 엔지니어링 모범 사례에는
“이 규칙이 아직 유효한가”를 주기적으로 평가하는 회귀 테스트가 포함되어야 한다.

### CLAUDE.md 계층 구조는 AI 행동 거버넌스의 분산 모델을 제안한다

전역, 프로젝트, 하위 디렉토리 CLAUDE.md의 계층 구조는
AI 행동을 팀과 조직 수준에서 관리하는 거버넌스 모델이 될 수 있다.
개인 선호도는 전역 파일에, 팀 규범은 프로젝트 파일에, 작업별 제약은
하위 디렉토리에 두는 구조는 권한과 범위가 명확하다.

이 패턴이 확산되면 “AI 거버넌스”가 중앙화된 모델 설정보다
분산된 설정 파일 생태계로 실현될 수 있다.
각 팀이 자신의 컨텍스트에 맞는 행동 규범을 유지하고,
조직 수준에서 공통 기준을 강제하되 팀별 커스터마이제이션을 허용하는 구조다.
이것은 소프트웨어 개발에서 린터 설정(ESLint, Prettier)이 계층적으로 작동하는 방식과 유사하다.

중요한 차이는 코드 린터와 달리 CLAUDE.md 규칙이 확률적으로 작동한다는 것이다.
“사용자 지시가 항상 이긴다”는 override 규칙은 이 확률성의 솔직한 인정이다.
그러나 감사 가능성(auditability)과 일관성 보장이 없는 설정 파일 기반 거버넌스는
규정 준수가 중요한 환경에서 한계를 가진다.

### 사용자 고통점 집계가 모델 행동 교정의 새로운 피드백 채널이 됐다

이 저장소의 References 섹션이 보여주는 것처럼, 350+ upvotes를 받은 GitHub 이슈가
CLAUDE.md 규칙의 원천이 됐다. 이 과정은 Anthropic의 공식 RLHF 파이프라인과 별개로
작동하는 사용자 주도 행동 교정 채널이다.

이것은 모델 회사와 사용자 커뮤니티 사이의 피드백 루프를 가시화한다.
GitHub 이슈 → 커뮤니티 규칙 저장소 → 광범위한 채택 → Anthropic의 모델 훈련 반영이라는
경로가 실제로 존재했을 가능성이 있다. 실제로 현재 모델이 sycophancy를 0%로 보인다는
시맨틱 평가 결과는, 커뮤니티가 충분히 큰 소리로 불만을 제기했을 때
모델 회사가 반응한다는 경험적 증거다.

이 피드백 루프의 의미는 두 가지다.
첫째, 사용자들이 경험한 실패를 구체적으로 보고하고 집계하는 커뮤니티 활동이
실제 모델 개선에 영향을 준다. 
둘째, 모델이 문제를 수정하면 그 문제를 겨냥한 프롬프트 규칙은 쓸모없어진다—
즉 성공적인 피드백 루프는 자신이 만들어낸 도구를 구식화시킨다.
이 역설이 프롬프트 엔지니어링 커뮤니티가 지속적으로 직면하는 구조적 조건이다.
