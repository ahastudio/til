# Weave Router: 코딩 에이전트를 위한 모델 라우팅 프록시

<https://github.com/workweave/router>

HN 토론: <https://news.ycombinator.com/item?id=48688700> (148점, 89개 댓글)

## 소개

Weave Router는 Anthropic, OpenAI, Gemini API를 하나의 엔드포인트로
통합하고, 각 요청을 가장 적합한 모델로 자동 라우팅하는 프록시다.
Claude Code, Codex, opencode, Cursor 같은 코딩 에이전트 도구에
`npx @workweave/router` 한 번으로 붙일 수 있는 드롭인 대체
엔드포인트로 설계됐다.

기술적으로는 라우팅 로직과 API 프록시를 담당하는 Go 백엔드(85.1%),
설정과 상태를 저장하는 PostgreSQL, CLI 설치·설정을 담당하는
TypeScript/Node.js 유틸리티로 구성된다. 라우팅 결정은 휴리스틱이
아니라 Avengers-Pro 연구에서 파생된 클러스터 스코어링 기법을 써서
온디바이스 임베딩으로 이뤄지며, 50ms 이내에 완료된다. Anthropic
Messages, OpenAI Chat Completions, Gemini 네이티브 API를 모두
지원하고, OpenRouter를 통해 DeepSeek·Kimi·GLM·Qwen·Llama·Mistral
같은 오픈소스 모델도 라우팅 대상에 포함할 수 있다. 스트리밍, 툴
사용, 비전 기능은 모든 제공자에 걸쳐 그대로 지원되며, API 키는
기본적으로 저장 시 암호화된다(BYOK). `/v1/route` 엔드포인트로
라우팅 결정 자체를 직접 조회할 수도 있고, OTLP 기반 관측성 통합으로
Weave·Honeycomb·Datadog·Grafana 같은 도구에 트레이스를 보낼 수
있다.

셀프호스팅은 `make full-setup` 한 번으로 Postgres와 라우터를
localhost:8080에 띄우는 방식이며, 호스팅형 버전은 앞서 언급한
`npx` 명령으로 Claude Code·Codex·opencode 설정을 자동으로
구성해준다. 라이선스는 Elastic License v2(ELv2)다. 작성자는 내부
사용에서 토큰 비용을 40~70% 절감했다고 밝혔다.

## 분석

### "프록시"가 아니라 "판단 계층"이라는 위치 선정

Weave Router가 스스로를 차별화하는 지점은 단순 로드밸런싱이나
장애 조치(failover)가 아니라, 프롬프트 내용 자체를 임베딩으로
분석해 어떤 모델이 적합한지 "판단"한다는 데 있다. 이는 API
게이트웨이(Kong, Envoy 같은)가 라우팅 규칙을 정적으로 설정하는
방식과 근본적으로 다르다. 코딩 에이전트가 수행하는 작업의 성격이
"파일 탐색"부터 "복잡한 아키텍처 설계"까지 스펙트럼이 넓다는 점을
전제로, 작업 성격에 맞는 모델을 실시간으로 고르겠다는 발상이다.
다만 이 판단의 정확도가 코딩 에이전트의 최종 결과물 품질에 어떤
영향을 미치는지는 저장소 설명만으로는 검증되지 않는다.

### BYOK와 셀프호스팅 선택지가 신뢰 문제를 우회한다

에이전트의 모든 프롬프트가 거치는 프록시라는 위치는 필연적으로
민감한 코드와 요청 내용에 대한 접근 권한을 갖는다는 뜻이다. 이
프로젝트가 API 키 암호화 저장과 완전한 셀프호스팅 경로를 함께
제공하는 것은, 라우팅이라는 부가 기능을 위해 코드베이스 전체를
제3자 SaaS에 노출해야 한다는 우려를 정면으로 겨냥한 설계로 보인다.

## 비평

### 40~70% 절감이라는 수치의 재현성이 불투명하다

작성자가 밝힌 비용 절감 폭은 넓은 범위(40~70%)로 제시되며, 어떤
워크로드 구성에서 이 수치가 나왔는지 구체적인 벤치마크나 방법론이
공개되어 있지 않다. 라우팅에 따른 비용 절감은 작업 분포(단순
조회 대 복잡한 추론의 비율)에 크게 의존하므로, 팀마다 실제
절감폭은 크게 달라질 수 있다.

### 모델을 자주 바꾸는 대가는 캐시와 일관성 손실

프롬프트 캐싱은 동일한 모델에 반복 요청을 보낼 때 가장 큰 효과를
낸다. 요청마다 다른 모델로 라우팅되면 프롬프트 캐시 적중률이
떨어지고, 모델마다 미묘하게 다른 응답 스타일과 실패 모드가 섞이면서
디버깅과 재현성이 오히려 복잡해질 수 있다. 저장소 설명에는 이
트레이드오프에 대한 언급이 없다.
HN 댓글에서도 같은 우려가 반복적으로 제기됐다.
port11은 프롬프트 캐시의 TTL이 5분에 불과한 상황에서 라우터가
매번 다른 모델로 요청을 보내면 캐시 미스가 늘어날 수밖에 없다고
지적했고[^port11], ai_slop_hater 역시 상태 없는(stateless) 라우팅이
캐시 손실 때문에 오히려 비용을 늘릴 수 있다고 같은 논리를
폈다[^ai_slop_hater].
_pdp_는 여기서 한 걸음 더 나아가, 저렴한 모델로 라우팅된다는
것 자체가 실질적인 절감으로 이어지는지 의문을 제기했다.
작은 모델은 작업을 너무 일찍 멈추거나, 오류를 내거나, 루프에
빠지는 경향이 있어 결국 재시도 비용으로 상쇄될 수 있다는
것이다[^_pdp_].

### 모델별로 프롬프트를 다르게 쓰는 관행과 충돌한다

g00k는 자신이 이미 어떤 모델을 쓰느냐에 따라 프롬프트 작성
방식을 바꾸고 있다고 언급하며, 라우터가 사용자의 의도와 맥락
뉘앙스를 이해하지 못한 채 모델을 선택하면 오히려 최적이 아닌
결과로 이어질 수 있다고 지적했다[^g00k]. 이는 "프롬프트 내용만
보고 최적 모델을 고른다"는 이 프로젝트의 핵심 전제가, 실제
숙련된 사용자의 작업 방식과 어긋날 수 있음을 보여준다.

### 프록시를 거치는 구조 자체의 프라이버시 리스크

bijowo1676은 코드와 프롬프트, 출력물 전체가 제3자 프록시 서비스를
거친다는 점에서 상업적 개발 환경에 적용하기 어렵다는 우려를
제기했다[^bijowo1676]. 이는 앞서 다룬 BYOK·셀프호스팅 선택지가
정확히 겨냥하는 우려이지만, 셀프호스팅을 하지 않고 호스팅형
버전을 그대로 쓰는 사용자에게는 여전히 유효한 문제로 남는다.

## 인사이트

### 에이전트 비용 최적화가 "모델 선택"이라는 별도 계층으로 분리되고 있다

에이전트 도입 비용이 커지는 조직일수록 "항상 최고 성능 모델 하나만
쓴다"는 단순한 전략에서 벗어나, 작업 단계별로 저렴한 모델과
고성능 모델을 구분해 쓰는 운영 과제가 부상하고 있다. Weave Router
같은 프로젝트는 이 판단을 애플리케이션 코드가 아니라 별도의
라우팅 계층에 위임함으로써, 모델 선택 로직을 에이전트 구현과
분리하는 아키텍처 패턴을 제시한다. 이런 계층이 표준화되면, 향후
프롬프트 캐시·실패 재시도·평가 데이터까지 함께 다루는 좀 더 포괄적인
"에이전트 런타임" 기능으로 흡수될 가능성이 있다.
HN 댓글 중에는 구현 세부사항에 대한 흥미로운 지적이 있었고, 작성자
본인도 일부 내용을 정정했다고 밝혔다.[^weave-hn]

### 대형 모델 사업자가 직접 이 기능을 흡수할 가능성

asdev는 이런 라우팅 기능을 결국 대형 모델 회사들이 직접 만들고,
가격 경쟁력에서 우위를 가진 채로 더 잘 만들 가능성이 크다고
내다봤다[^asdev]. 독립 라우터 서비스는 결국 모델 제공자가 자체
플랫폼에 라우팅 기능을 내장하는 순간 존재 이유가 옅어질 수 있다는
뜻이다.
nikcub는 여러 갈래로 문제를 제기했다.
에이전트가 이미 내부적으로 모델 라우팅에 준하는 판단을 하고
있는데 프록시 계층이 추가로 무엇을 개선하는지, 모델이 실패했을
때 피드백 루프가 어떻게 개선되는지, 그리고 라우팅 결정에 쓰이는
임베딩이 새 모델이 빠르게 나오는 속도를 따라가지 못해 오히려
학습 데이터가 금방 낡아버리는 것은 아닌지를 지적했다[^nikcub].
