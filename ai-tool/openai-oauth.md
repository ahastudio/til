# openai-oauth - ChatGPT 계정으로 OpenAI API를 무료 사용하는 비공식 도구

원문: <https://github.com/EvanZhouDev/openai-oauth>

## 요약

openai-oauth는 기존 ChatGPT 계정 인증 정보를 활용하여
OpenAI API에 무료로 접근할 수 있게 해주는 비공식 도구다.
로컬호스트 프록시 서버를 생성하여
OpenAI 내부 Codex 엔드포인트(chatgpt.com/backend-api/codex/responses)에
인증 요청을 중계하는 방식으로 동작한다.

`npx openai-oauth` 명령으로 localhost(http://127.0.0.1:10531/v1) 서버를 시작하면
OpenAI API 포맷과 호환되는 엔드포인트를 별도 API 키 없이 사용할 수 있다.
Vercel AI SDK 프로바이더 패키지로도 제공된다.

지원하는 엔드포인트는 /v1/responses, /v1/chat/completions, /v1/models이며,
스트리밍, 도구 호출(tool calling), 추론 트레이스(reasoning traces)를 지원한다.
로컬 ChatGPT 인증 캐시에서 OAuth 토큰을 재사용하는 구조다.

제약 사항으로는 Codex가 지원하는 모델만 사용 가능하고(구독 티어에 따라 상이),
로그인에 별도 명령(`npx @openai/codex login`)이 필요하며,
상태 유지 대화가 불가능하여 매 요청마다 전체 히스토리를 전송해야 한다.
라이선스는 AGPL-3.0이며 OpenAI와 무관한 비공식 프로젝트다.

## 분석

### 기술적 구조

openai-oauth의 핵심 메커니즘은 OAuth 토큰 재사용이다.
사용자가 `npx @openai/codex login`으로 인증하면
로컬에 저장되는 인증 파일을 openai-oauth가 읽어
ChatGPT 백엔드의 Codex API에 요청을 전달한다.
이 프록시는 OpenAI의 공식 API 포맷을 그대로 노출하므로
기존 OpenAI SDK를 사용하는 코드에서 base URL만 변경하면 동작한다.

### 작동 가능한 이유

이 도구가 존재할 수 있는 배경에는
OpenAI의 Codex CLI가 OAuth 기반 인증을 허용한다는 사실이 있다.
GeekNews 댓글에서 heycalmdown이 지적했듯,
Anthropic의 Claude는 opencode와 hermes-agent에서
OAuth 인증을 제거한 반면 OpenAI는 OpenClaw 같은 도구를 허용하고 있다.
이는 두 회사의 API 접근 통제 전략이 근본적으로 다르다는 것을 보여준다.

### 제약과 경고

인증 파일이 패스워드와 동등한 보안 수준을 가진다는 점,
계정 정지 위험이 있다는 점,
개인 로컬 사용만 권장된다는 점이 명시되어 있다.
또한 Codex API가 지원하는 기능만 사용 가능하므로
ChatGPT에서 가능한 이미지 생성 등은 이 도구로 접근할 수 없다.

## 비평

### 강점

기술적 구현이 단순하고 실용적이다.
`npx` 한 줄로 프록시를 실행할 수 있고,
OpenAI API 호환 포맷을 제공하므로 기존 코드와의 통합이 쉽다.
Vercel AI SDK 프로바이더까지 제공하는 것은
개발자 경험(DX)에 대한 세심한 배려다.

AGPL-3.0 라이선스 선택은 의미가 있다.
이 도구를 서비스 형태로 재배포하는 것을 사실상 차단하여
"개인 로컬 사용" 취지를 라이선스 차원에서 강제한다.

### 약점

가장 근본적인 문제는 지속 가능성이다.
OpenAI가 내부 Codex 엔드포인트의 인증 방식을 변경하거나,
토큰 재사용을 탐지하여 차단하면 즉시 무용지물이 된다.
이 도구의 수명은 전적으로 OpenAI의 관용에 달려 있다.

보안 리스크도 무시할 수 없다.
인증 파일이 "패스워드와 동등"하다는 경고가 있지만,
`npx`로 실행하는 서드파티 패키지가 이 인증 파일에 접근한다는 것 자체가
공급망 공격(supply chain attack)의 잠재적 벡터가 된다.
npm 패키지 하이재킹이 현실적 위협인 시대에
민감한 인증 정보를 다루는 비공식 패키지를 신뢰하는 것은
개인의 위험 감수 판단에 해당한다.

## 인사이트

### API 경제의 회색지대와 플랫폼의 전략적 선택

openai-oauth 같은 도구의 존재는
API 과금 모델과 구독 모델 사이의 구조적 긴장을 드러낸다.
ChatGPT Plus/Pro 구독자는 월정액을 지불하고
웹 인터페이스를 통해 모델에 접근하지만,
같은 모델을 API로 호출하면 별도 과금이 발생한다.
사용자 입장에서 "이미 돈을 내고 있는데 왜 또 내야 하는가"라는
불만은 자연스럽다.

OpenAI가 이런 도구를 묵인하는 것은
생태계 확장 전략의 일환일 수 있다.
Codex CLI의 무료 사용을 허용하고,
OAuth 기반 접근을 열어둠으로써
개발자들이 OpenAI 생태계에 더 깊이 진입하게 만든다.
단기적인 API 수익 손실보다
장기적인 개발자 록인(lock-in)이 더 가치 있다는 판단이다.

반대로 Anthropic이 OAuth 접근을 차단한 것은
API 수익 모델을 엄격히 보호하겠다는 의지의 표현이다.
두 전략 모두 합리적이지만,
어느 쪽이 장기적으로 더 큰 생태계를 만드는지는
아직 결론이 나지 않았다.

### 비공식 API 도구의 역사적 패턴

비공식 API 접근 도구는 플랫폼 역사에서 반복되는 패턴이다.
트위터 초기에 서드파티 클라이언트가 번성했고,
유튜브 다운로더(youtube-dl)는 수년간 법적 공방을 거쳤으며,
스포티파이의 비공식 API 래퍼들은 주기적으로 차단과 우회를 반복했다.

이 패턴에서 일관된 교훈은
비공식 도구가 사용자의 실제 필요를 대변한다는 점이다.
openai-oauth가 해결하는 문제는
"API 키 발급과 별도 과금 없이 프로그래밍적으로 모델에 접근하고 싶다"는 것이고,
이는 상당수 개발자의 진짜 니즈다.

역사적으로 플랫폼은 두 가지 경로를 택했다.
하나는 비공식 도구를 차단하고 공식 채널을 강화하는 것(트위터의 API v2)이고,
다른 하나는 비공식 도구의 기능을 공식 제품에 흡수하는 것이다.
OpenAI가 Codex CLI를 무료로 제공한 것 자체가
후자의 경로를 이미 걷고 있음을 시사한다.

### 인증 토큰 재사용의 보안 함의와 신뢰 경계

openai-oauth의 작동 원리인 "로컬 인증 토큰 재사용"은
보안 관점에서 신뢰 경계(trust boundary)의 문제를 제기한다.
Codex CLI가 저장한 인증 토큰은
해당 CLI만 사용할 것이라는 암묵적 가정 하에 로컬에 저장된다.
서드파티 도구가 이 토큰을 읽는 것은
기술적으로 가능하지만 의도된 사용이 아니다.

이 패턴이 위험한 이유는 확장 가능성에 있다.
openai-oauth 자체는 선의의 도구일 수 있지만,
같은 메커니즘을 악용하는 도구가 등장하면
사용자는 구분할 수 없다.
npm 생태계에서 타이포스쿼팅(typosquatting)이나
패키지 하이재킹으로 악성 코드가 배포된 사례는 이미 다수 존재한다.

더 근본적인 질문은 로컬 파일 시스템의 인증 정보를
누구까지 신뢰할 것인가 하는 것이다.
브라우저의 쿠키, CLI 도구의 토큰, SSH 키 등
로컬에 저장된 인증 자산을 다른 프로세스가 읽는 것은
운영체제의 파일 권한 모델에서는 허용되지만
애플리케이션 보안 모델에서는 위반이다.
macOS의 키체인이나 Linux의 시크릿 서비스 같은
자격 증명 관리 시스템이 존재하는 이유가 바로 이것이다.
