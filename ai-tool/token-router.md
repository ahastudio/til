# token-router — 로컬 모델로 라인 범위를 추출하고 클라우드 모델에 증거만 전달하는 라우터

<https://github.com/sleeplesshan/token-router>

## 소개

`token-router`는 대용량 로그, 대형 소스 파일, 에이전트 컨텍스트 참조 문서를 클라우드
LLM에 전달하기 전에 로컬 Ollama 모델로 관련 라인 범위를 추출하는 Python 라우터다.
“손실 없는 로컬 라인 라우팅(lossless local line routing)”을 표방하며, 로컬 모델이
요약이나 해석 없이 JSON 라인 좌표만 반환하면 라우터가 원본 파일에서 해당 행을 그대로
잘라내 클라우드 모델에 전달한다.

기본 로컬 라우팅 모델은 `gemma4:e2b-it-q4_K_M`(Ollama)이며, 클라우드 추론 대상은
GPT-5.5, o3 등이다.
Codex 스킬과 Claude Code 통합을 모두 지원한다.
별(star) 77개, 포크 12개의 초기 단계 프로젝트이며, 2026년 6월 생성되어 MIT 라이선스로
공개되어 있다.

## 아키텍처

라우팅 파이프라인은 세 단계로 분리된다.

| 단계         | 담당 엔진      | 역할                              | 출력           |
| ------------ | -------------- | --------------------------------- | -------------- |
| 검색 · 분류  | 로컬 Ollama    | 관련 라인 좌표 탐색               | JSON 범위      |
| 증거 추출    | Python 라우터  | 원본 파일을 라인 번호로 슬라이싱  | 원본 텍스트    |
| 추론         | 클라우드 LLM   | 디버그 · 설명 · 패치 · 리뷰       | 최종 응답      |

로컬 모델은 분석자가 아니라 라우터다.
원본 텍스트를 보거나 수정하지 않으며, 오직 “몇 번 줄부터 몇 번 줄까지가 관련 있다”는
JSON을 반환하는 역할만 한다.

## 라우팅 모드

세 가지 모드를 제공한다.

- **`error_log`**: 배포 로그, 스택 트레이스, CI 출력에 최적화.
  키워드 스캔과 tail-window 스캔을 결합한다.
- **`heavy_code`**: 대형 소스 파일과 국소적 코드 조사에 최적화.
  쿼리 히트, 의심스러운 코드 마커(TODO/FIXME/raise/assert), head/tail 미리보기 순서로
  우선순위를 정한다.
- **`agent_context`**: `AGENTS.md`, `CLAUDE.md`, `.cursorrules` 같은 긴 에이전트 참조
  문서에 최적화.
  쿼리 히트, frontmatter, 제목, 필수 규칙, workflow/tool/security 키워드, head/tail 순서로
  범위를 선정한다.

## 분석

### 로컬·클라우드 역할 분리가 새로운 유형의 비용 구조를 만든다

`token-router`가 선택한 아키텍처의 핵심은 “탐색 비용은 로컬에서, 추론 비용은 클라우드에서”라는
분리다.
이 분리가 경제적으로 성립하려면 세 조건이 동시에 충족되어야 한다.
로컬 모델 실행 비용이 무시할 수 있을 만큼 낮을 것, 로컬 모델의 라인 선택이 충분히 정확할 것,
그리고 클라우드 모델에 전달하는 슬라이스가 충분히 작아야 비용 절감 효과가 분명할 것.

이 조건이 충족되는 상황은 뚜렷하다.
배포 실패 로그는 오류 증거가 특정 구간에 집중되고, 나머지는 정상 상태를 반복하는 노이즈다.
레거시 소스 파일에서 특정 함수만 디버그할 때도 전체 파일의 대부분은 관련이 없다.
이런 “국소성이 높은” 작업에서 99% 수준의 토큰 절감이 가능하다.
반대로 “전체를 봐야 판단할 수 있는” 작업에서는 라우팅 자체가 정보 손실을 유발할 수 있으며,
저자도 이 점을 명시적으로 인정하여 아키텍처 리뷰나 보안 감사에는 사용하지 말 것을 권고한다.

### 「손실 없음」은 추출에 대한 주장이지 선택에 대한 주장이 아니다

README의 “lossless”는 주의 깊게 읽을 필요가 있다.
이 단어는 라인 추출 단계의 정확성을 주장하는 것이지, 라인 선택 단계가 옳다는 주장이 아니다.
로컬 모델이 반환한 행 범위를 Python 라우터가 원본 파일에서 정확히 잘라낸다는 의미에서
“손실 없는”이다.
그러나 로컬 모델이 선택한 범위가 처음부터 올바른 범위인지는 별개의 문제다.

저자는 이 구분을 “선택 단계는 불완전할 수 있지만 추출된 증거 자체는 손실 없다”고 표현하며,
“클라우드 모델이 더 넓은 범위가 필요하다고 판단하면 더 요청할 수 있다”는 반복 워크플로를
제안한다.
그러나 반복적으로 범위를 확장하는 워크플로는 단순한 파일 직접 읽기보다 복잡하고, 경우에
따라 총 토큰 비용이 더 클 수 있다.
“손실 없음”이라는 표현은 기술적으로 정확하지만 마케팅적으로 오해를 유발할 여지가 있다.

### 에이전트 컨텍스트 라우팅은 파일 구조 리팩터링을 전제한다

`agent_context` 모드는 긴 AGENTS.md, CLAUDE.md, .cursorrules 파일을 라우팅하는 기능이다.
이 기능이 작동하려면 사용자가 이미 “항상 로드되는 루트 지시 파일은 짧게, 작업별 세부 지침은
별도 참조 파일로”라는 아키텍처를 구현해 두어야 한다.

README가 명시하는 것처럼, 플랫폼이 이미 긴 루트 지시 파일을 프롬프트에 주입했다면
`token-router`는 그 토큰 비용을 소급해서 제거할 수 없다.
이것은 기능의 한계가 아니라 아키텍처적 전제 조건이다.
에이전트 컨텍스트 라우팅의 효과를 누리려면 프로젝트의 지시 파일 구조 자체를 라우팅을 염두에
두고 재설계해야 한다.
이 진입 장벽은 “설치 후 즉시 사용 가능한” 도구로서의 접근성을 제한한다.

## 비평

### 로컬 모델의 JSON 신뢰도가 핵심 변수인데 측정되지 않는다

`token-router`의 전체 파이프라인은 로컬 모델이 올바른 JSON 라인 범위를 반환한다는 가정 위에
서 있다.
README는 “매우 작은 로컬 모델이 복잡한 로그나 비정형 기호가 있는 파일에서 가끔 잘못된 JSON을
반환할 수 있다”고 인정하며, 코드 펜스 노이즈 정리와 안전한 폴백을 구현했다고 밝힌다.

그러나 “가끔”이 어느 정도인지, 어떤 입력 유형에서 오류율이 높아지는지에 대한 체계적인 측정은
제공되지 않는다.
벤치마크 보고서도 존재하지만, JSON 오류율이나 폴백 발동 빈도는 포함되지 않는다.
라우터가 잘못된 범위를 선택하거나 폴백이 발동되는 경우 클라우드 모델은 불완전한 증거로
추론해야 하며, 이것이 오답을 유발한다면 토큰 절감의 실질적 가치가 크게 하락한다.
“JSON 오류가 잦으면 더 큰 모델을 써보라”는 권고는 필요하지만, 신뢰도를 측정 가능한 지표로
제시하지 않는다는 점에서 사용자가 안전한 사용 범위를 스스로 파악해야 한다.

### 벤치마크는 도구가 가장 유리한 케이스만 측정한다

제시된 세 케이스는 모두 99% 수준의 절감을 보여준다.
sparse 인프라 로그, 레거시 버그 소스, 키워드 없는 구조적 소스가 그것이다.
이 케이스들은 `token-router`의 설계가 최대한 효과적인 입력 유형이다.
오류 증거가 국소화된 로그, 특정 함수 버그가 있는 소스가 바로 이 도구가 겨냥하는 워크플로다.

독립 검증이 없는 상황에서 세 케이스의 99% 절감 수치는 인상적이지만 대표성이 낮다.
전체 구조를 봐야 하는 리팩터링 작업, 교차 파일 참조가 많은 쿼리, 에러가 여러 컴포넌트에
분산된 로그에서의 성능은 측정되어 있지 않다.
README의 “언제 사용하고 언제 우회할 것인가” 표가 이 한계를 솔직하게 열거하고 있는 것은
긍정적이지만, 그 경계 조건에서의 실제 토큰 비용이 어떻게 되는지는 알 수 없다.

### 도구를 도구로만 사용할 때 발생하는 반복 확장 비용이 계산에 없다

저자가 설계 원칙으로 제시하는 “반복적 라우팅(route, reason, expand when needed)” 워크플로는
실제로 얼마나 많은 반복이 발생하는지에 따라 총 비용이 크게 달라진다.
클라우드 모델이 “더 넓은 범위가 필요하다”고 요청하면 두 번째 라우팅이 필요하고, 그것도
부족하면 세 번째가 필요하다.
각 라운드마다 로컬 모델 실행 비용(시간 포함)과 클라우드 API 호출 비용이 발생한다.

벤치마크는 단일 라우팅 라운드의 토큰을 측정한다.
그러나 실제 디버깅 세션에서 평균적으로 몇 번의 반복이 필요한지, 반복이 발생할 때 총 비용이
직접 파일 읽기 방식과 비교해 어떻게 되는지에 대한 측정은 없다.
“라우팅 → 이유 → 확장” 사이클의 평균 반복 횟수가 1.5번만 되어도 99% 절감 주장의 실질적
의미가 상당히 달라진다.

## 인사이트

### AI 코딩 스택에 “전처리 레이어”라는 새로운 범주가 등장한다

지금까지 AI 코딩 도구의 최적화는 주로 두 방향에서 이루어졌다.
프롬프트 엔지니어링(무엇을 어떻게 요청하는가)과 모델 선택(어떤 모델을 사용하는가)이다.
`token-router`는 세 번째 방향을 제시한다.
모델에 전달하기 전에 입력 자체를 가공하는 “전처리 레이어”다.

이 전처리 레이어의 핵심은 모델 외부에서 실행된다는 것이다.
모델의 능력이나 프롬프트 설계에 의존하지 않고, 독립된 Python 스크립트와 로컬 모델이 입력을
재구성한다.
이것은 AI 코딩 도구가 단순한 “LLM 호출 래퍼”에서 “입력 최적화 파이프라인을 포함한 시스템”으로
진화하는 신호다.
`pxpipe`가 텍스트를 이미지로 변환해 토큰 비용 구조를 우회하는 것처럼, `token-router`는
라우팅으로 컨텍스트 크기를 줄이는 다른 형태의 전처리다.
이 흐름이 계속되면 AI 코딩 워크플로에서 “컨텍스트 엔지니어링” 도구군이 하나의 레이어로
자리 잡을 것이다.

### 유닉스 파이프 철학이 LLM 아키텍처로 재현된다

`token-router`의 설계는 유닉스 파이프 철학의 반복이다.
“작고 단일 책임을 가진 프로그램을 파이프로 연결한다”는 원칙이 “로컬 모델은 라인 좌표만,
Python 라우터는 슬라이싱만, 클라우드 모델은 추론만”이라는 분리로 표현된다.

유닉스에서 이 철학이 성공한 이유는 각 단계의 출력이 표준화된 형식(텍스트 스트림)이었기
때문이다.
`token-router`는 JSON 라인 범위를 그 표준 형식으로 사용한다.
흥미로운 점은 이 JSON 인터페이스 자체가 로컬 모델의 약한 고리라는 것이다.
작은 모델이 JSON을 안정적으로 생성하지 못하는 경우가 있고, 라우터는 이를 정규화하고
폴백한다.
유닉스 파이프에서 프로그램 간 통신이 실패하면 파이프라인 전체가 중단되는 것처럼, 이 JSON
인터페이스의 신뢰도가 전체 시스템의 신뢰도를 결정한다.
이 병목이 해결될 때 비로소 파이프라인 전체의 가치가 실현된다.

### 컨텍스트 국소성 가정이 도구 적용 범위를 제한하는 숨은 전제다

`token-router`가 효과적인 모든 케이스는 “관련 정보가 파일 내 특정 구간에 집중되어 있다”는
공통 가정을 공유한다.
배포 로그의 오류 스택, 소스 파일의 특정 함수, AGENTS.md의 특정 작업 관련 규칙 모두 이 가정에
부합한다.

그러나 소프트웨어 엔지니어링의 많은 중요한 질문은 이 가정에 부합하지 않는다.
“이 코드베이스 전반에서 인증 로직이 어떻게 분산되어 있는가”, “이 변경이 전체 API 계약에
미치는 영향은 무엇인가”, “이 시스템의 오류 처리 일관성은 어떠한가” 같은 질문은 파일 내
특정 구간이 아니라 전체적인 패턴 파악을 요구한다.
이런 질문에 `token-router`를 적용하면 관련 정보가 잘려나가 클라우드 모델이 오답을 내놓을
위험이 있다.

이 한계는 도구의 결함이 아니라 전제의 경계다.
“컨텍스트 국소성이 높은 작업”과 “전체적 패턴 이해가 필요한 작업”을 구분하는 판단은 결국
사용자가 내려야 한다.
그 판단을 지원하는 시스템적 장치(자동 국소성 측정, 라우팅 신뢰도 점수 등)가 현재는 없으며,
이것이 도구의 안전한 적용 범위를 스스로 알아내야 하는 사용자의 책임으로 남아 있다.
