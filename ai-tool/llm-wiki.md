# LLM Wiki: LLM이 유지관리하는 개인 지식 베이스 패턴

- 전편: <https://twitter.com/karpathy/status/2039805659525644595>
- 후편: <https://twitter.com/karpathy/status/2040470801506541998>

## 요약

Andrej Karpathy가 두 편의 트윗으로 공유한
**LLM Wiki** 패턴과 **아이디어 파일(Idea File)** 개념이다.

전편(4월 3일)은 실제 사용 경험을 공유한다.
소스 문서를 raw/에 모으고, LLM이 이를 마크다운 위키로 "컴파일"한다.
Obsidian을 프론트엔드로, LLM을 백엔드로 쓴다.
~100개 기사, ~40만 단어 규모에서 RAG 없이도 질의가 잘 된다.
출력물(답변, 슬라이드, 차트)을 다시 위키에 파일링해서 지식이 복리로 쌓인다.

후편(4월 5일)은 전편이 바이럴(1,300만 조회)되자
아이디어를 체계화한 "아이디어 파일"을 Gist로 공유한다.
RAG와의 차이(매번 재검색 vs 지속적 축적),
아키텍처 세 레이어(원본 소스, 위키, 스키마),
오퍼레이션 세 가지(수집, 질의, 린트)를 정리한다.
아이디어 파일이라는 새 공유 방식도 제안한다 —
코드/앱 대신 아이디어를 공유하고, 받는 사람의 에이전트가 맞춤 구현한다.

## 원문 번역

### 전편 트윗 (2026년 4월 3일)

#### LLM 지식 베이스

최근 매우 유용하게 쓰고 있는 것:
다양한 리서치 관심 주제에 대해 LLM으로 개인 지식 베이스를 구축하는 것.
이런 식으로, 최근 제 토큰 사용량의 상당 부분이
코드 조작이 아니라 지식 조작(마크다운과 이미지로 저장됨)에 들어가고 있습니다.
최신 LLM은 이런 작업을 꽤 잘합니다. 그래서:

#### 데이터 수집

소스 문서(기사, 논문, 레포, 데이터셋, 이미지 등)를 raw/ 디렉터리에 인덱싱하고,
LLM을 사용해 위키를 점진적으로 "컴파일"합니다.
위키는 디렉터리 구조의 .md 파일 모음일 뿐입니다.
위키에는 raw/에 있는 모든 데이터의 요약, 백링크가 포함되고,
데이터를 개념별로 분류하고 글을 쓰고 모두 링크합니다.
웹 기사를 .md로 변환할 때 Obsidian Web Clipper 확장을 쓰고,
핫키로 관련 이미지를 모두 로컬에 다운로드해서
LLM이 쉽게 참조할 수 있게 합니다.

#### IDE

Obsidian을 IDE "프론트엔드"로 사용합니다.
원본 데이터, 컴파일된 위키, 파생된 시각화를 볼 수 있습니다.
중요한 점: LLM이 위키의 모든 데이터를 작성하고 유지하며,
저는 거의 직접 손대지 않습니다.
다른 방식으로 데이터를 렌더링하고 보기 위해
몇 가지 Obsidian 플러그인을 써봤습니다(예: 슬라이드용 Marp).

#### Q&A

흥미로워지는 지점은 위키가 충분히 커졌을 때입니다
(예: 제 최근 리서치 위키는 ~100개 기사, ~40만 단어).
LLM 에이전트에게 위키에 대해 온갖 종류의 복잡한 질문을 할 수 있고,
에이전트가 알아서 답을 조사합니다.
멋진 RAG를 써야 할 줄 알았는데,
LLM이 인덱스 파일과 모든 문서의 간략한 요약을 자동으로 유지하는 게 꽤 잘 되어서
이 ~소규모에서는 중요한 관련 데이터를 꽤 쉽게 읽습니다.

#### 출력

텍스트/터미널로 답을 받는 대신,
마크다운 파일이나 슬라이드 쇼(Marp 형식)나 matplotlib 이미지를
렌더링해달라고 하고, 다시 Obsidian에서 봅니다.
쿼리에 따라 다양한 시각적 출력 형식을 상상할 수 있습니다.
종종 출력물을 위키에 "파일링"해서 이후 쿼리를 위해 강화합니다.
그래서 제 탐색과 쿼리가 항상 지식 베이스에 "쌓입니다".

#### 린팅

위키에 대해 LLM "상태 점검"을 돌려봤습니다.
예를 들어 불일치하는 데이터 찾기, 빠진 데이터 추정(웹 검색으로),
새 글 후보를 위한 흥미로운 연결 찾기 등으로
위키를 점진적으로 정리하고 전체 데이터 무결성을 강화합니다.
LLM은 더 질문하고 조사할 것을 제안하는 데 꽤 능숙합니다.

#### 추가 도구

데이터를 처리하기 위한 추가 도구를 개발하게 됩니다.
예를 들어 위키 위에 작고 나이브한 검색 엔진을 바이브 코딩했는데,
직접 사용하기도 하고(웹 UI로), 더 자주는 더 큰 쿼리를 위한
도구로 CLI를 통해 LLM에 넘겨줍니다.

#### 추후 탐구

레포가 커지면서 자연스러운 욕구는
합성 데이터 생성 + 파인튜닝을 고려하는 것입니다.
LLM이 컨텍스트 윈도우가 아니라 가중치에서 데이터를 "아는" 상태로 만드는 거죠.

#### 요약 (원문)

주어진 수의 소스에서 원본 데이터를 수집하고,
LLM이 .md 위키로 컴파일하고,
LLM이 다양한 CLI로 Q&A를 수행하며 위키를 점진적으로 강화하고,
이 모든 것을 Obsidian에서 봅니다.
위키를 수동으로 쓰거나 편집하는 일은 거의 없습니다, 그건 LLM의 영역입니다.
스크립트의 해키한 모음이 아니라 대단한 새 제품이 될 여지가 있다고 생각합니다.

### 후편 트윗 (2026년 4월 5일)

와, 이 트윗이 엄청 퍼졌네요!

트윗의 약간 개선된 버전을 "아이디어 파일" 형태로 공유하고 싶었습니다.
아이디어 파일의 개념은, LLM 에이전트 시대에는
특정 코드나 앱을 공유할 필요성이 줄어들었다는 거예요.
그냥 아이디어만 공유하면, 상대방의 에이전트가
자신의 필요에 맞게 커스터마이징해서 만들어줍니다.

그래서 아이디어를 Gist 형식으로 올렸습니다:
<https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f>

이걸 에이전트에게 주면 자신만의 LLM 위키를 만들고
어떻게 사용하는지 안내해줄 거예요.
방향이 너무 다양해서 의도적으로 조금 추상적/모호하게 작성했습니다.
물론 사람들이 아이디어를 수정하거나 Discussion에 자신의 아이디어를
기여할 수도 있는데, 그것도 멋진 일이죠.

### Gist: LLM Wiki

#### 핵심 아이디어

LLM과 문서를 다루는 대부분의 경험은 RAG처럼 생겼습니다:
파일 모음을 올리면 LLM이 질의 시점에 관련 청크를 검색해서 답변을 생성하죠.
이 방식은 작동하지만, LLM이 매번 처음부터 지식을 재발견합니다.
축적이 없어요. 다섯 개 문서를 종합해야 하는 미묘한 질문을 하면,
LLM은 그때마다 관련 조각을 찾아 이어붙여야 합니다.
쌓이는 게 없어요.
NotebookLM, ChatGPT 파일 업로드, 대부분의 RAG 시스템이 이렇게 작동합니다.

여기서 제안하는 아이디어는 다릅니다.
질의 시점에 원본 문서에서 검색하는 대신,
LLM이 **지속적인 위키를 점진적으로 구축하고 유지관리**합니다 —
당신과 원본 소스 사이에 위치하는,
구조화되고 상호 연결된 마크다운 파일 모음입니다.
새 소스를 추가하면, LLM은 나중에 검색하려고 인덱싱하지 않습니다.
읽고, 핵심 정보를 추출하고, 기존 위키에 통합합니다 —
엔티티 페이지를 업데이트하고, 주제 요약을 수정하고,
새 데이터가 기존 주장과 어디서 모순되는지 기록하고,
진화하는 종합을 강화하거나 도전합니다.
지식은 한 번 컴파일되고 나서 *최신 상태로 유지*됩니다,
매번 쿼리마다 재도출되는 게 아니라.

이것이 핵심 차이점입니다: **위키는 지속적이고 복리로 쌓이는 산물입니다.**
상호 참조가 이미 거기 있습니다.
모순은 이미 플래그가 달렸습니다.
종합은 이미 읽은 모든 것을 반영합니다.
위키는 소스를 추가할 때마다, 질문을 할 때마다 더 풍부해집니다.

위키를 직접 작성하는 일은 없거나 거의 없습니다 — LLM이 전부 씁니다.
당신은 소싱, 탐색, 올바른 질문에 집중합니다.
LLM은 나머지 잡일을 다 합니다 —
요약하기, 상호 참조하기, 파일링하기,
지식 베이스가 시간이 지나도 실제로 유용하게 만드는 장부 정리.
실제로 저는 한쪽에 LLM 에이전트를, 다른 쪽에 Obsidian을 열어둡니다.
LLM이 대화에 기반해 편집을 하면, 저는 실시간으로 결과를 탐색합니다 —
링크를 따라가고, 그래프 뷰를 확인하고, 업데이트된 페이지를 읽으면서.
Obsidian은 IDE, LLM은 프로그래머, 위키는 코드베이스입니다.

다양한 맥락에 적용할 수 있습니다. 몇 가지 예:

- **개인**: 자신의 목표, 건강, 심리, 자기계발 추적 —
  일기, 글, 팟캐스트 메모를 파일링하고 자신에 대한 구조화된 그림을 쌓아가기.
- **리서치**: 몇 주 또는 몇 달에 걸쳐 주제를 깊이 파고들기 —
  논문, 기사, 보고서를 읽으며 진화하는 논제가 있는 포괄적 위키 구축.
- **책 읽기**: 챕터마다 파일링하며 인물, 주제, 플롯 실마리와
  그 연결 페이지를 구축하기. 끝에는 풍부한 컴패니언 위키가 생깁니다.
  [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page)처럼요 —
  커뮤니티 자원봉사자들이 수년에 걸쳐 수천 개의 연결 페이지를 쌓은 것.
  LLM이 상호 참조와 유지관리를 다 해주면서 혼자서도 그런 걸 만들 수 있습니다.
- **비즈니스/팀**: Slack 스레드, 회의 녹취, 프로젝트 문서, 고객 통화로
  피드되는 LLM 유지관리 내부 위키.
  업데이트를 검토하는 인간이 루프에 있을 수도 있습니다.
  아무도 하고 싶어하지 않는 유지관리를 LLM이 해주므로 위키가 최신 상태를 유지합니다.
- **경쟁 분석, 실사, 여행 계획, 강의 노트, 취미 심화** —
  시간이 지나며 지식을 축적하고 흩어지지 않게 정리하고 싶은 모든 것.

#### 아키텍처

세 가지 레이어:

**원본 소스** — 큐레이션된 소스 문서 모음. 기사, 논문, 이미지, 데이터 파일.
불변(immutable) — LLM이 읽지만 수정하지 않습니다. 진실의 원천입니다.

**위키** — LLM이 생성한 마크다운 파일 디렉터리.
요약, 엔티티 페이지, 개념 페이지, 비교, 개요, 종합.
LLM이 이 레이어를 완전히 소유합니다.
페이지를 만들고, 새 소스가 오면 업데이트하고, 상호 참조를 유지하고, 일관성을 지킵니다.
당신은 읽고; LLM은 씁니다.

**스키마** — LLM에게 위키 구조, 규칙, 소스 수집·질문 답변·위키 유지관리 시
따를 워크플로를 알려주는 문서
(Claude Code용 CLAUDE.md, Codex용 AGENTS.md 등).
이게 핵심 설정 파일입니다 — LLM을 범용 챗봇이 아닌
훈련된 위키 유지관리자로 만드는 것이 이겁니다.
무엇이 효과적인지 파악하면서 당신과 LLM이 함께 진화시킵니다.

#### 오퍼레이션

**수집(Ingest).** 새 소스를 원본 컬렉션에 넣고 LLM에게 처리를 지시합니다.
예시 흐름: LLM이 소스를 읽고, 핵심 내용을 논의하고, 위키에 요약 페이지를 쓰고,
인덱스를 업데이트하고, 위키 전반의 관련 엔티티·개념 페이지를 업데이트하고,
로그에 항목을 추가합니다. 하나의 소스가 10~15개 위키 페이지에 영향을 줄 수 있습니다.
개인적으로 저는 소스를 하나씩 수집하며 관여하기를 선호합니다 —
요약을 읽고, 업데이트를 확인하고, 강조할 내용을 LLM에게 안내하면서요.
하지만 감독을 줄이고 많은 소스를 한 번에 배치 수집할 수도 있습니다.
자신의 스타일에 맞는 워크플로를 개발하고 스키마에 문서화해서
다음 세션에 활용하세요.

**질의(Query).** 위키에 질문합니다.
LLM이 관련 페이지를 검색하고 읽어서 인용과 함께 답변을 종합합니다.
질문에 따라 답변 형태가 달라질 수 있습니다 —
마크다운 페이지, 비교 표, 슬라이드 덱(Marp), 차트(matplotlib), 캔버스.
중요한 인사이트: **좋은 답변은 위키에 새 페이지로 다시 파일링할 수 있습니다.**
요청한 비교, 분석, 발견한 연결 — 이것들은 가치 있고
채팅 히스토리 속으로 사라져서는 안 됩니다.
이렇게 하면 탐색도 수집한 소스처럼 지식 베이스에 복리로 쌓입니다.

**린트(Lint).** 주기적으로 LLM에게 위키 상태 점검을 요청합니다.
확인 항목: 페이지 간 모순, 최신 소스가 대체한 오래된 주장,
인바운드 링크 없는 고아 페이지, 언급되었지만 전용 페이지 없는 중요 개념,
누락된 상호 참조, 웹 검색으로 채울 수 있는 데이터 공백.
LLM은 조사할 새 질문과 찾아볼 새 소스를 제안하는 데 능숙합니다.
위키가 성장함에 따라 건강하게 유지해줍니다.

#### 왜 작동하는가

지식 베이스 유지관리의 지루한 부분은 읽기나 생각이 아닙니다 — 장부 정리입니다.
상호 참조 업데이트하기, 요약을 최신 상태로 유지하기,
새 데이터가 기존 주장과 언제 모순되는지 기록하기,
수십 개 페이지에 걸쳐 일관성 유지하기.
유지 부담이 가치보다 빠르게 커지기 때문에 사람들은 위키를 포기합니다.
LLM은 지루해하지 않고, 상호 참조 업데이트를 잊지 않으며,
한 번에 15개 파일을 손댈 수 있습니다.
유지관리 비용이 거의 0에 가깝기 때문에 위키가 유지됩니다.

인간의 역할은 소스 큐레이션, 분석 방향 잡기, 좋은 질문 던지기,
전체가 무엇을 의미하는지 생각하기입니다.
LLM의 역할은 그 외의 모든 것입니다.

이 아이디어는 Vannevar Bush의 Memex(1945) — 문서 간 연상 경로가 있는
개인적이고 큐레이션된 지식 저장소 — 와 정신적으로 관련됩니다.
Bush의 비전은 웹이 된 것보다 이쪽에 가까웠습니다:
사적이고, 적극적으로 큐레이션되고, 문서 간 연결이 문서 자체만큼 가치 있는.
그가 해결하지 못한 부분은 누가 유지관리를 하느냐였습니다.
LLM이 그것을 처리합니다.

## 분석

### 경험담에서 패턴으로

전편에서 후편으로의 진화가 흥미롭다.
전편은 "내가 이렇게 쓰고 있다"는 경험담이고,
후편은 경험을 추상화한 패턴 문서다.
전편이 바이럴되자 구체적 구현 대신 재현 가능한 아이디어로 정리한 것이다.
이 과정 자체가 "아이디어 파일" 개념의 실증이다.

### 유지관리 비용이라는 병목

LLM Wiki의 핵심 통찰은 "지식 관리의 병목은 유지관리 비용"이라는 진단이다.
인간은 위키를 만들 수 있지만 유지하지 못한다.
교차 참조 업데이트, 모순 해소, 일관성 유지는 지루하고 반복적이다.
LLM은 이 작업을 지치지 않고 수행한다.

### 역할 분담의 비유

Karpathy의 비유가 명쾌하다:
"Obsidian은 IDE, LLM은 프로그래머, 위키는 코드베이스."
이 프레임은 역할 분담을 정확히 정의한다.
인간은 무엇을 만들지 결정하고, LLM은 실제로 만든다.

### RAG 없이도 된다

전편에서 "멋진 RAG를 써야 할 줄 알았는데" 안 써도 된다는 발견이 실용적이다.
~100개 기사 규모에서는 index.md와 요약만으로 충분하다.
벡터 DB, 임베딩, 청킹 같은 RAG 인프라 없이 마크다운 파일만으로 작동한다.
단순한 것이 이겼다.

### 아이디어 파일이라는 공유 방식

아이디어 파일 개념은 오픈소스의 진화 형태다.
기존 오픈소스는 코드를 공유한다.
아이디어 파일은 의도와 패턴을 공유하고, 구현은 각자의 에이전트에 맡긴다.

## 비평

### 규모 확장의 한계

전편의 "~소규모(~100 기사, ~40만 단어)"라는 단서가 중요하다.
이 규모에서 RAG 없이 작동한다는 것은 설득력 있지만,
규모가 10배로 커지면 index.md 기반 탐색은 한계에 부딪힌다.
Karpathy 자신도 검색 엔진을 바이브 코딩했다고 말하는데,
이 시점에서 결국 RAG와 비슷한 인프라가 필요해진다.

### 세팅 비용의 과소평가

LLM Wiki는 여전히 세팅 비용이 있다.
스키마를 정의하고, LLM이 일관되게 작동하도록 훈련하고,
처음 몇 개의 소스를 수집하며 워크플로를 확립하는 데 시간이 필요하다.
"LLM이 다 해준다"는 메시지가 진입 장벽을 실제보다 낮게 보이게 할 수 있다.

### 컨텍스트 윈도우의 휘발성

"합성 데이터 생성 + 파인튜닝"이라는 전편의 마지막 언급은
아직 탐구 단계임을 보여준다.
컨텍스트 윈도우에 의존하는 현재 방식은 근본적으로 휘발적이다.
세션이 끝나면 LLM은 위키 내용을 "잊는다".
매번 index.md를 읽어야 하는 것은 인간이 매일 아침 업무일지를 다시 읽는 것과 같다.

### 아이디어 파일의 재현성 문제

아이디어 파일 개념은 흥미롭지만, 아직 공유 방식의 합의가 없다.
어떤 형식으로, 어느 수준의 추상도로 공유해야 하는지,
받는 쪽 에이전트가 다를 때 재현성을 어떻게 보장하는지는 열린 문제다.

## 인사이트

### 위키 실패의 진짜 원인은 의지가 아니라 구조였다

"왜 팀 위키는 항상 방치되는가?"라는 질문의 답은 "귀찮아서"가 아니다.
인간의 주의는 유한하고, 유지관리는 보상이 없는 작업이다.
새 문서를 쓰는 것은 창조의 즐거움이 있지만,
기존 문서의 교차 참조를 업데이트하는 것은 순수한 비용이다.
위키는 이 구조적 문제 때문에 실패했다.
LLM이 이 비용을 0에 가깝게 만들면 방정식이 바뀐다.
Confluence나 Notion이 해결하지 못한 문제를
LLM이 처음으로 실질적으로 해결하는 것이다.

### "컴파일"이라는 비유가 드러내는 패러다임 전환

Karpathy는 위키를 "컴파일"한다고 표현한다.
소스 코드(원본 문서) → 컴파일러(LLM) → 바이너리(위키).
이 비유는 단순한 수사가 아니다.
컴파일러가 소스를 최적화하듯, LLM은 원본을 읽기 좋은 형태로 변환한다.
컴파일러가 링킹을 하듯, LLM은 교차 참조를 만든다.
컴파일러가 타입 체크를 하듯, LLM은 린트로 모순을 찾는다.
소프트웨어 엔지니어링의 빌드 시스템이
지식 관리에 그대로 적용되는 것이다.
"지식 컴파일러"라는 프레임은 LLM의 역할을
대화 상대가 아니라 빌드 도구로 재정의한다.

### 아이디어 파일은 소프트웨어 공유의 패러다임 전환을 예고한다

전통적으로 소프트웨어 공유는 코드의 공유였다.
오픈소스는 "실행 가능한 코드"를 공유 단위로 삼았다.
아이디어 파일은 "의도와 패턴"을 공유 단위로 만든다.
코드는 특정 언어, 런타임, 의존성에 묶이지만,
아이디어 파일은 어느 에이전트, 어느 언어로도 구현될 수 있다.
이것이 성숙한다면 GitHub 같은 코드 공유 플랫폼과 별개로
아이디어 파일 전용 공유 생태계가 등장할 수 있다.
레시피가 완성된 음식이 아니라 만드는 방법을 공유하는 것처럼.

### Vannevar Bush의 Memex가 80년 만에 실현된다

Bush가 1945년에 상상한 Memex는 문서 간 "연상 트레일"로 연결된
개인 지식 저장소였다.
웹은 이 비전의 변형이었지만 공개적이고 비큐레이션적이었다.
LLM Wiki는 Bush가 원했던 것에 더 가깝다:
개인적이고, 적극적으로 큐레이션되고, 연결이 내용만큼 가치 있는.
Bush가 상상하지 못한 것은 Memex를 유지하는 존재가
인간이 아닐 수도 있다는 점이었다.
