# Skyvern: LLM과 컴퓨터 비전으로 브라우저를 자동화하는 에이전트

<https://github.com/Skyvern-AI/skyvern>

## 소개

Skyvern은 LLM과 컴퓨터 비전을 이용해 브라우저 기반 워크플로우를 자동화하는 오픈소스
프레임워크다.
GitHub 스타 21,000개 이상, 포크 2,000개 이상을 보유한 이 프로젝트는 Playwright와
호환되는 SDK를 제공하면서 그 위에 AI 기능을 얹는 방식으로 설계되었다.
주요 언어는 Python이며 TypeScript SDK도 함께 지원한다.

기존 브라우저 자동화는 DOM 파싱과 XPath/CSS 셀렉터에 의존해, 웹사이트 레이아웃이
변경될 때마다 스크립트가 깨지는 구조적 취약점을 안고 있었다.
Skyvern은 셀렉터 대신 Vision LLM이 화면을 보고 필요한 요소를 직접 식별하고 상호작용하는
방식을 택한다.
이를 통해 한 번도 본 적 없는 웹사이트에서도 동작하고, 레이아웃 변경에 강인하며,
동일한 워크플로우를 여러 웹사이트에 적용할 수 있다.

SDK의 핵심은 Playwright `page` 객체에 네 가지 AI 커맨드를 추가한 것이다.

- `page.act(prompt)`: 자연어로 동작을 수행한다.
- `page.extract(prompt, schema)`: 구조화된 데이터를 추출한다.
- `page.validate(prompt)`: 페이지 상태를 확인해 불린 값을 반환한다.
- `page.prompt(prompt, schema)`: LLM에 임의 프롬프트를 보낸다.

기존 Playwright 셀렉터 방식, AI 자연어 방식, AI 폴백(셀렉터 실패 시 AI로 전환) 방식의
세 가지 상호작용 모드를 혼합해 사용할 수 있다.

벤치마크 성능은 WebVoyager 평가에서 85.8%, WebBench에서 64.4%로 현재 최고 수준을
기록하고 있다.
클라우드 버전은 안티봇 감지 우회, 프록시 네트워크, CAPTCHA 해결 기능을 번들로 제공한다.

## 분석

### Playwright 확장으로서의 설계 철학

Skyvern이 Playwright를 대체하는 것이 아니라 확장하는 방식을 택한 것은 전략적으로
중요한 선택이다.
Playwright는 이미 검증된 브라우저 자동화 인프라를 갖추고 있고, 개발자들에게 친숙하다.
Skyvern은 그 위에 AI 레이어를 얹어, 기존 Playwright 코드베이스를 완전히 버리지 않고도
점진적으로 AI 기능을 도입할 수 있는 마이그레이션 경로를 제공한다.
특히 AI 폴백 모드(`page.click(“#selector”, prompt=“...”)`)는 셀렉터가 유효할 때는
빠른 전통 방식을 쓰고, 실패 시에만 AI를 호출함으로써 비용과 속도 균형을 맞춘다.

### 에이전트 스웜 아키텍처

Skyvern은 BabyAGI, AutoGPT에서 영감을 받은 Task-Driven 자율 에이전트 설계를 따른다.
단일 LLM 호출이 아니라 에이전트 스웜(swarm)이 협력해 웹사이트를 이해하고, 계획을 세우고,
행동을 실행한다.
이 구조는 복잡한 다단계 워크플로우에서 단순한 프롬프트 엔지니어링보다 더 안정적인
결과를 낳을 수 있다.

### 크리덴셜 통합과 실사용 시나리오

`page.agent.login(credential_type, credential_id)` API는 Skyvern, Bitwarden,
1Password와 연동해 저장된 크리덴셜로 자동 로그인을 수행한다.
사용자의 기존 Chrome 브라우저를 CDP(Chrome DevTools Protocol)로 연결해 쿠키, 로그인
상태, 확장 프로그램을 그대로 활용하는 기능도 제공한다.
이는 “이미 로그인되어 있는 사이트를 자동화하고 싶다”는 실무 요구사항을 직접 해결한다.

### 로컬/클라우드 이중 전략

`pip install skyvern`으로 시작하는 로컬 모드(SQLite 기본)부터 Docker Compose를 이용한
Postgres 기반 구성, 관리형 클라우드 서비스까지 단계적으로 확장 가능하다.
TypeScript SDK(`@skyvern/client`)도 제공해 Node.js 기반 백엔드나 프론트엔드 도구와의
통합도 지원한다.

## 비평

### 강점: 낮은 진입 장벽과 점진적 채택

기존 Playwright 코드를 버리지 않아도 된다는 점이 가장 큰 강점이다.
`pip install skyvern`으로 시작해 필요한 부분에만 AI 커맨드를 추가할 수 있어,
팀이 점진적으로 도입 범위를 넓힐 수 있다.
로컬 SQLite 기본 설정도 초기 셋업 마찰을 최소화한다.

### 한계: LLM 호출 비용과 응답 지연

Vision LLM을 통한 요소 식별은 CSS 셀렉터보다 느리고 비용이 높다.
AI 폴백 모드로 일부 완화할 수 있지만, 페이지 로드마다 스크린샷을 LLM에 보내는 구조는
대량 자동화 시나리오에서 지연과 비용이 문제가 될 수 있다.
벤치마크 성능(WebBench 64.4%)은 인상적이지만, 실제 환경의 엣지 케이스—다국어 사이트,
복잡한 SPA, 인터랙티브 캔버스 요소—에서의 성능은 별도로 검증이 필요하다.

### 보완이 필요한 관점: 쓰기 전용 최적화

WebBench의 WRITE 태스크(폼 입력, 로그인, 파일 다운로드)에서 최고 성능을 보이지만,
읽기 중심 작업(대규모 데이터 스크래핑, 실시간 모니터링)에서의 효율성은 아직 검증이
부족하다.
또한 안티봇 감지 우회, CAPTCHA 해결은 클라우드 버전에만 번들되어 있어, 로컬 환경에서
접근성이 떨어지는 사이트 자동화에는 제한이 있다.
터널링을 통해 로컬 브라우저를 클라우드에 노출하는 기능은 편리하지만, 보안 문서를 별도로
숙지해야 하는 운영 부담이 있다.

## 인사이트

### “AI 폴백” 패턴은 새로운 소프트웨어 레이어의 원형이다

Skyvern의 `page.click(“#selector”, prompt=“Click the Submit button”)` 인터페이스는
표면적으로는 작은 기능 추가처럼 보이지만, 더 큰 소프트웨어 설계 패턴의 원형이다.
“전통 방식을 시도하고, 실패하면 AI로 폴백”이라는 구조는 단순히 브라우저 자동화에만
적용되는 것이 아니다.
이것은 결정적 알고리즘과 확률적 AI를 계층으로 결합하는 새로운 소프트웨어 아키텍처
패턴이다.
데이터베이스 쿼리 플래너가 통계를 바탕으로 실행 계획을 고르되 실패 시 더 보수적인
계획으로 전환하는 것, 컴파일러가 최적화 패스를 적용하되 안전하지 않으면 원본을 유지하는
것과 구조적으로 같다.
AI가 확산되는 시대에 “AI 우선, 전통 방식 폴백”이나 “전통 방식 우선, AI 폴백” 중 어느
방향이 더 실용적인지는 도메인마다 다를 것이다.
Skyvern이 “전통 방식 우선, AI 폴백”을 택한 것은 성능과 비용 균형에서 합리적이지만,
AI 모델의 추론 속도가 빨라질수록 이 트레이드오프의 최적점은 AI 쪽으로 이동할 것이다.

### 브라우저 자동화의 취약성은 웹의 “구조 vs. 표현” 분리 실패에서 온다

기존 XPath/CSS 셀렉터 기반 자동화가 레이아웃 변경에 취약한 근본 원인은,
웹이 “의미(semantics)”와 “표현(presentation)”을 충분히 분리하지 못했기 때문이다.
HTML의 `<button>`, `<input>`, `<form>` 같은 시맨틱 태그가 있지만, 실제 웹 개발에서는
`<div class=“btn”>`, `<span onclick=“”>` 같은 표현 중심 코드가 남발된다.
이 때문에 자동화 도구는 시맨틱 의미가 아니라 DOM 구조나 CSS 클래스에 의존하게 되고,
디자인 리뉴얼마다 스크립트가 깨진다.
Vision LLM이 스크린샷을 보고 “이것이 로그인 버튼이다”를 인식하는 것은, 인간이 시각적
문맥으로 UI를 해석하는 방식을 모방한다.
이 접근은 웹의 시맨틱 결핍을 시각적 추론으로 우회하는 것이다.
장기적으로 보면, 웹이 더 풍부한 시맨틱 마크업(ARIA, 시맨틱 HTML)을 채택할수록
Vision LLM의 역할이 줄어들고 시맨틱 기반 자동화가 더 신뢰성 있게 동작할 것이다.
역설적으로, Skyvern 같은 도구의 성공은 현재 웹의 시맨틱 부채를 드러낸다.

### RPA의 역사는 반복되고 있다: 이번에는 AI가 접착제다

브라우저 자동화 도구의 역사는 반복되는 패턴을 보인다.
Selenium(2004) → Cypress/Playwright(2016~2020) → AI 기반 에이전트(2023~)로 이어지는
흐름에서, 각 세대는 이전 세대의 취약성을 줄이는 방향으로 진화했다.
Selenium은 브라우저 종속성과 불안정성이 문제였고, Playwright는 안정성을 높였지만
셀렉터 의존성은 여전했다.
Skyvern은 시각적 추론으로 셀렉터 의존성을 줄이려 한다.
이것은 기업 RPA(Robotic Process Automation) 시장에서 수십 년간 반복된 이야기이기도 하다.
UiPath, Automation Anywhere 같은 엔터프라이즈 RPA 도구들도 “레코딩 기반 자동화 → 선택자
의존 → 깨지기 쉬운 봇”이라는 사이클에 빠졌다.
이제 AI가 그 접착제 역할을 맡아 취약한 부분을 보정하려 한다.
하지만 이 세대도 새로운 취약점을 가질 것이다.
LLM의 환각(hallucination), 비결정적 동작, 프롬프트 인젝션 공격 가능성이 그것이다.
자동화 도구의 진화는 취약성을 없애는 것이 아니라 취약성의 종류를 바꾸는 과정이다.
다음 세대의 도구는 아마 AI의 비결정성을 다루는 새로운 메커니즘을 중심으로 등장할 것이다.
