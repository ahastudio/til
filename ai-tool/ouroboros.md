## 소개

“프롬프트를 멈추고, 사양을 시작하라(Stop prompting. Start specifying).”
Ouroboros는 Claude Code 플러그인으로 동작하는 사양 우선(specification-first) AI
개발 시스템이다. AI 코딩 실패의 근본 원인이 코드 생성 능력이 아니라 인간의
요구사항 명확성에 있다고 진단하고, 소크라테스식 문답을 통해 숨겨진 가정을
드러낸 뒤 불변 사양으로 결정화하는 진화적 루프를 구축한다.

## 핵심 아이디어

### 우로보로스 진화 루프

자기 참조적 순환이 핵심이다.
뱀이 자신의 꼬리를 먹는 상징처럼, 시스템의 출력이 다음 세대의 입력이 된다.

1. **Interview** — 소크라테스식 질문으로 12개 이상의 숨겨진 가정을 표면화
2. **Seed** — 답변을 불변 사양(immutable specification)으로 결정화
3. **Execute** — 더블 다이아몬드 분해로 산출물 생성
4. **Evaluate** — 3단계 검증 (기계적 → 의미적 → 합의)
5. **Evolve** — 평가 출력이 다음 세대의 입력으로 회귀

이것은 반복(iteration)이 아니라 진화(evolution)다.
매 세대마다 사양 자체가 성숙한다.

### 더블 다이아몬드 구조

디자인 씽킹에서 빌려온 2단계 패턴이다.

| 다이아몬드 | 발산              | 수렴              |
| ---------- | ----------------- | ----------------- |
| 1단계      | 소크라테스식 질문 | 존재론적 명확성   |
| 2단계      | 설계 옵션 탐색    | 검증된 산출물     |

이전 단계를 완료하지 않으면 다음 단계로 진행할 수 없다.
이해하지 못한 것을 설계할 수 없다는 원칙이다.

### 수학적 게이트

주관적 판단이 아니라 정량적 기준으로 단계를 통제한다.

#### 모호성 점수 (Interview → Seed 관문)

```text
Ambiguity = 1 − Σ(clarityᵢ × weightᵢ)
```

임계값 ≤ 0.2 이하에서만 다음 단계로 진행한다.
80% 명확성에 도달하면 나머지 미지 요소는 아키텍처 추측이 아닌 코드 수준
결정이 된다.

#### 존재론적 수렴 (진화 종료 기준)

```text
Similarity = 0.5 × name_overlap + 0.3 × type_match + 0.2 × exact_match
```

유사도 ≥ 0.95에 도달하면 수렴으로 판정한다.
정체, 진동, 반복적 피드백 같은 병리적 패턴도 감지한다.
안전장치로 최대 30세대 하드 캡이 있다.

## 아키텍처

### 9개 전문 에이전트

각 에이전트는 고유한 인지 모드를 대표하며 온디맨드로 로딩된다.

| 에이전트                | 역할                         |
| ----------------------- | ---------------------------- |
| Socratic Interviewer    | 질문 전용 모드               |
| Ontologist              | 본질적 속성 식별             |
| Seed Architect          | 사양 결정화                  |
| Evaluator               | 검증 오케스트레이션          |
| Contrarian              | 가정 도전                    |
| Hacker                  | 비관습적 제약 돌파           |
| Simplifier              | 복잡성 축소                  |
| Researcher              | 증거 기반 조사               |
| Architect               | 구조적 근본 원인 분석        |

Socratic Interviewer의 설계가 특히 인상적이다.
절대로 구현을 약속하지 않고, 반드시 질문으로 끝내며, 기존 코드베이스가 있으면
발견한 패턴을 검증 대상 가정으로 프레이밍한다.
질문할 수 없으면 도구가 실패해도 질문을 유지한다.

### 기술 스택

```text
Python 3.12+ · 18 패키지 · 166 모듈 · 20 서브모듈
├── anthropic + claude-agent-sdk — Claude 통합
├── litellm — 100+ 모델 어댑터
├── sqlalchemy + aiosqlite — 이벤트 소싱 영속화
├── textual + typer — TUI 모니터링 대시보드
├── pydantic — 스키마 검증
└── stamina + filelock — 복원력과 동시성
```

PAL Router가 3-tier 비용 전략(1x/10x/30x 에스컬레이션)으로 모델 능력을
최적화한다.
실패 시 강한 모델로 올리고, 성공 시 약한 모델로 내린다.

### 명령어

Claude Code 세션 내에서 실행하는 `ooo` 접두사 명령이다.

| 명령어         | 기능                                       |
| -------------- | ------------------------------------------ |
| `ooo setup`    | 초기 MCP 서버 등록                         |
| `ooo interview`| 소크라테스식 질문 시작                     |
| `ooo seed`     | 사양 생성                                  |
| `ooo run`      | 더블 다이아몬드 실행                       |
| `ooo evaluate` | 검증 게이트 실행                           |
| `ooo evolve`   | 수렴까지 연속 루프                         |
| `ooo ralph`    | 백그라운드 지속 진화                       |
| `ooo unstuck`  | 5개 측면 사고 페르소나로 막힌 세션 타개    |
| `ooo status`   | 드리프트 감지와 세션 추적                  |

## 코드 분석

### 잘 설계된 점

#### 이벤트 소싱 영속화

SQLAlchemy + aiosqlite 기반으로 전체 이력을 보존한다.
세션 중단 후 재개가 가능하고, 진화 계보(lineage)를 추적할 수 있다.
상태 비저장(stateless) 진화를 가능케 하는 핵심 설계다.

#### 에이전트 분리 원칙

10개 에이전트가 각각 마크다운 파일로 정의되어 있고, 허용/차단 도구가 명시적으로
선언된다.
Socratic Interviewer는 Read/Glob/Grep만 허용하고 Write/Edit/Bash를 차단한다.
역할 경계가 코드가 아니라 선언으로 강제되는 구조다.

#### 브라운필드 인식

15개 설정 유형, 12개 이상의 생태계를 스캔하여 기존 코드베이스를 분석한 뒤
사양을 작성한다.
그린필드 가정으로 시작하지 않는다는 점이 실전적이다.
브라운필드 모드에서는 모호도 채점에 Context Clarity(15%) 차원이 추가된다.

#### 드리프트 측정

목표(50%) + 제약(30%) + 존재론(20%) 가중치로 드리프트를 정량화한다.
에이전트가 원래 의도에서 벗어나는 것을 수학적으로 감지한다.
Claude Code 플러그인에서는 Write/Edit 시 훅이 자동으로 드리프트 자문을 제공한다.

### 모듈 구조

```text
src/ouroboros/
├── agents/        — 에이전트 구현
├── bigbang/       — 초기 부트스트래핑
├── cli/           — 명령줄 인터페이스
├── config/        — 설정 관리
├── core/          — 핵심 기능
├── evaluation/    — 평가 로직
├── events/        — 이벤트 처리
├── evolution/     — 진화/적응 메커니즘
├── execution/     — 실행 엔진
├── mcp/           — Model Context Protocol 통합
├── observability/ — 모니터링과 로깅
├── orchestrator/  — 오케스트레이션
├── persistence/   — 데이터 영속화
├── plugin/        — 플러그인 시스템
├── providers/     — 프로바이더 구현
├── resilience/    — 내결함성과 복구
├── routing/       — 요청 라우팅
├── secondary/     — 보조 기능
├── strategies/    — 전략 구현
└── verification/  — 검증/밸리데이션
```

166개 모듈, 20개 서브패키지는 이 규모의 프로젝트치고 상당히 크다.
95개 테스트 파일이 존재한다.

### 프롬프트 vs 코드 분업 원리

Ouroboros 코드베이스가 실증하는 경계선이 있다.
어떤 관심사를 프롬프트로 다루고 어떤 것을 코드로 강제해야 하는가에 대한 답이다.

| 관심사                                        | 프롬프트에 | 코드에 |
| --------------------------------------------- | ---------- | ------ |
| 창의적 판단 (질문 생성, 코드 작성, 평가 의견) | ✅         |        |
| 페르소나·관점 (인터뷰어, 악마의 변호인)       | ✅         |        |
| 워크플로 안내·라우팅 선언                     | ✅ (얇게)  |        |
| 규칙의 강제 (거부권, 허용목록, 보호 AC)       |            | ✅     |
| 상태·영속성·재개                              |            | ✅     |
| 측정·관문 수치 (모호도, 드리프트, 수렴)       |            | ✅     |
| 프로세스 관리 (스폰, 취소, 타임아웃, 병렬)    |            | ✅     |
| 프로토콜 (MCP, CLI, 스트림 파싱)              |            | ✅     |
| 보안 (소독, 상한, 원자적 쓰기, fail-closed)   |            | ✅     |
| 예산·비용 라우팅                              |            | ✅     |

판별 질문은 하나다.
“이것이 확률적으로 지켜져도 되는가, 결정론적으로 보장되어야 하는가?”
전자는 프롬프트로, 후자는 코드로.
LLM에게 “테스트를 조작하지 마”라고 프롬프트에 쓰는 것과,
모든 승인 경로가 지나는 단일 관문에서 `if reward_hacking_risk >= 0.7: final_approved = False`를
실행하는 것은 완전히 다른 보증 수준이다.
22만 줄은 후자의 목록이 생각보다 훨씬 길다는 증거다.

이 원리는 Claude Code 활용에도 그대로 적용된다.
CLAUDE.md의 지시(프롬프트)는 에이전트가 따르려 노력하는 것이고,
hooks·권한 설정·CI 게이트(코드)는 구조적으로 강제되는 것이다.
중요한 규칙일수록 프롬프트에서 코드로 옮겨야 한다.

## 비평

### 강점

#### “인간을 고치라”는 진단은 정확하다

대부분의 AI 코딩 도구가 프롬프트 엔지니어링(더 나은 입력)에 집중할 때,
Ouroboros는 사양 엔지니어링(더 나은 정의)에 집중한다.
[코드 리뷰가 사양 리뷰로 전환](./reviews-dead.md)되어야 한다는 논의,
[Spec-Driven Development](./spec-driven-development.md)의 흐름과
정확히 같은 방향이다.
프롬프트는 휘발성이지만 사양은 축적된다.

#### 소크라테스식 접근의 형식화

“질문만 하고, 절대 구현을 약속하지 않는” 에이전트를 별도 역할로 분리한 것은
탁월하다.
대부분의 AI 도구는 사용자의 첫 문장에서 바로 코드를 생성하려 한다.
12개 이상의 숨겨진 가정을 드러내겠다는 목표는 야심적이지만 올바른 방향이다.

#### 수학적 게이트는 실용적이다

“충분히 명확한가”를 주관적으로 판단하지 않고 모호성 점수로 정량화한다.
80% 명확성 임계값은 과도한 분석 마비(analysis paralysis)를 방지하면서도
아키텍처 수준의 불확실성은 제거하는 균형점이다.

### 약점과 우려

#### 복잡성 대비 효과의 검증 부재

20개 서브모듈, 166개 모듈, 10개 에이전트, 수학적 게이트, 이벤트 소싱까지
동원하지만, 이 모든 복잡성이 “사양을 먼저 잘 쓰자”는 결론 대비 과잉이
아닌지 증명할 벤치마크가 없다.
사양 문서 하나와 Claude Code의 기본 기능만으로 동일한 품질을 달성할 가능성을
배제할 수 없다.

#### Python 3.12+ 의존성의 과도함

18개 패키지, 특히 anthropic + claude-agent-sdk + litellm 세 가지 AI SDK를
동시에 의존한다.
Claude Code 플러그인이라는 컨텍스트에서 litellm(100+ 모델)까지 필요한
이유가 불분명하다.
의존성이 많을수록 파손점이 늘어난다.

#### 진화 루프의 실전 비용

“수렴까지 연속 루프”는 이론적으로 우아하지만, 매 세대마다 LLM API 호출이
발생한다.
최대 30세대 × 다단계 에이전트 호출은 상당한 토큰 비용을 의미한다.
PAL Router의 3-tier 에스컬레이션이 비용을 얼마나 절감하는지 실측 데이터가
없다.

#### “존재론”이라는 용어의 과잉

프로젝트가 “존재론적 명확성”, “존재론적 수렴” 같은 철학 용어를 빈번히
사용하지만, 실제로 하는 일은 “요구사항을 잘 정리하자”다.
학술적 용어가 실질을 가리면 커뮤니케이션 비용이 올라간다.
소프트웨어 엔지니어링에서 “ontology”는 이미 특정 의미(데이터 모델링)를
가지고 있어 혼동을 유발할 수 있다.

#### 인터뷰가 제안이 아닌 심문이 된다

실제 사용자 피드백이 이 문제를 명확히 지적한다.
이슈 #1650(“RFC: the interview should propose, not interrogate”)은
인터뷰 시스템의 부담이 역전돼 있다고 비판한다.[^issue1650]
Ouroboros가 가장 필요한 사람—요구사항을 명확하게 말하기 어려운 사람—이
오히려 매 턴마다 스스로 답을 생성해야 하는 구조다.
시스템이 먼저 프롬프트를 분석해 이해 초안을 제시하고 사용자가 수정하는 방식이어야 하는데,
현재는 거꾸로 돼 있다.
모호도 점수와 게이트 같은 절차적 세부사항이 항상 노출되는 것도 인지 부하를 높인다.

소크라테스식 인터뷰라는 은유가 여기서 발목을 잡는다.
소크라테스는 이미 무엇을 알고 있는 사람에게 질문을 던진다.
하지만 Ouroboros의 주요 사용자는 무엇을 모르는지조차 모르는 사람이다.
그들에게 연속적인 열린 질문은 도움이 아니라 장벽이다.

#### `ooo auto`가 Seed 계약을 사용자 동의 없이 변경한다

이슈 #1678은 `ooo auto` 실행 중 Seed에 명시된 수용 기준이 시스템에 의해 묵묵히 확장되는 버그를 보고한다.[^issue1678]
6개 명시 기준을 가진 Seed가 `GAME_2D` 태스크 클래스로 분류되자,
관련 기본 기준 3개가 사용자 확인 없이 앞에 추가됐다.
“불변 Seed”라는 핵심 약속이 자동화 파이프라인 안에서 깨진 사례다.

이 버그가 드러내는 것은 단순한 결함이 아니다.
Seed의 불변성은 Ouroboros 전체 철학의 기반이다.
명세가 수렴 과정에서 조용히 변하면 드리프트 측정 기준선도 오염된다.
검증된 기준선이 신뢰할 수 없다면 수렴 판정도 의미를 잃는다.
불변이어야 할 것이 변할 때 그 위에 쌓인 모든 수학적 게이트가 흔들린다.

## 인사이트

### 사양의 반감기 문제

Ouroboros가 해결하지 않는 핵심 질문이 있다.
사양은 코드와 함께 진화하는가, 아니면 코드가 작성된 뒤 화석이 되는가?
이벤트 소싱으로 사양의 계보를 추적하지만, 코드 변경이 사양을 역으로
업데이트하는 메커니즘은 보이지 않는다.
코드와 사양의 일치를 지속적으로 보장하지 못하면, 결국 “오래된 문서” 문제의
더 정교한 변형에 불과해진다.

### 진짜 병목은 질문의 품질이다

Socratic Interviewer가 12개 숨겨진 가정을 드러낸다고 하지만, AI가 묻는
질문의 품질은 결국 AI 모델의 도메인 이해도에 의존한다.
인간 요구사항 엔지니어가 고객에게 좋은 질문을 하려면 해당 도메인의 경험이
필요하듯, AI 인터뷰어도 같은 한계를 가진다.
형식(소크라테스식)이 내용(도메인 지식)을 대체할 수 없다.

### 복잡성의 역설

Ouroboros는 AI 개발의 복잡성을 줄이기 위해 존재하지만, 그 자체가
20개 서브모듈의 복잡한 시스템이다.
사용자가 Ouroboros의 진화 루프, 수학적 게이트, 9개 에이전트의 역할을
이해하는 데 드는 인지 비용이 “그냥 사양을 잘 쓰는 것”보다 높다면,
도구가 해결하려는 문제를 재생산하게 된다.

### Safe-but-Wrong이 가장 위험한 실패다

Ouroboros가 명시하는 AI 코딩의 진짜 위험은 오류가 아니라 Safe-but-Wrong이다.
에이전트가 실행 가능한 도구를 만들어달라는 요청에 인수인계 문서 뭉치를 내놓거나,
재사용 가능한 워크플로 대신 일회성 체크리스트를 생성하거나,
결측 데이터를 정직하게 표기하지 않고 OK, 0, 빈 값으로 채우는 사례들이다.
이 실패는 테스트를 통과하고, 리뷰를 통과하고, 배포까지 이어진다.
발견되는 시점은 실제 사용 중이다.

Safe-but-Wrong을 막는 Ouroboros의 접근은 산출물 계약을 명세에 명시하는 것이다.
“실행 가능한 도구”인지 “문서”인지를 Seed에 고정해두면
에이전트가 비슷해 보이는 대안으로 표류할 때 드리프트 측정이 이를 감지한다.
이것은 “완료됐는지”를 묻기 전에 “무엇이 완료여야 하는지”를 먼저 못 박는 순서의 문제다.

### 완료≠승인: 검증 비용 사다리의 설계

“됐다”는 주장을 어떻게 증거로 바꾸는가가 Ouroboros의 두 번째 핵심 원칙이다.
3단계 검증 파이프라인은 비용을 기준으로 설계됐다.
기계적 검사(\$0) → 의미론적 평가 → 다중 모델 합의의 순서로,
비싼 검증은 싼 검증을 통과한 것에만 적용된다.

리워드 해킹 방어도 같은 맥락이다.
평가를 통과하도록 산출물을 조작하는 에이전트 행동을 막기 위해
모든 승인 경로가 지나는 단일 코드 관문(`REWARD_HACKING_VETO_THRESHOLD = 0.7`)을 둔다.
이것은 CLAUDE.md 규칙이 아니라 코드다.
“규칙을 지키도록 지시하는 것”과 “규칙을 어길 수 없게 만드는 것”의 차이를
코드베이스 전체가 보여준다.

### 깊이 있는 개발자의 사고가 코드로 구현될 때

Threads의 @_petercha는 Ouroboros를 “사고의 깊이가 깊은 개발자가 자신의 생각을 Harness로 구현해버리면 어떤 일이 일어나는지 알게됐다”고 표현했다.[^petercha]
이 문장은 Ouroboros의 성격을 정확히 짚는다.
Ambiguity Score, 드리프트 측정, 4패턴 정체 감지, 배심원 독립성 보장—이 모든 개념은
숙련된 소프트웨어 엔지니어링 경험에서 나온 직관을 수치와 코드로 외재화한 것이다.

이것이 Ouroboros의 진짜 가치이자 진입 장벽이다.
이 도구를 최대한 활용하려면 도구가 포함한 원칙들을—모호도 게이트가 왜 필요한지,
드리프트 임계값 0.3이 무엇을 의미하는지, Safe-but-Wrong이 왜 가장 위험한 실패인지—이미
이해하고 있어야 한다.
경험 있는 엔지니어에게 Ouroboros는 자신이 이미 아는 것을 자동화하는 도구다.
그 경험이 없는 사람에게는 이해하기 어려운 복잡한 시스템이다.

### SDD 생태계의 수렴

[Spec-Driven Development](./spec-driven-development.md),
[코드 리뷰의 사양 리뷰 전환](./reviews-dead.md),
그리고 Ouroboros까지 — 모두 같은 방향을 가리킨다.
“코드가 아니라 사양을 리뷰하라”는 합의가 형성되고 있다.
Ouroboros의 가치는 이 합의를 실행 가능한 도구로 구체화했다는 데 있다.
다만 그 구체화가 최소한인지 최대한인지는 아직 판단하기 이르다.

## 결론

Ouroboros의 철학은 옳다.
AI 코딩의 병목이 코드 생성이 아니라 요구사항 명확성이라는 진단,
소크라테스식 질문으로 숨겨진 가정을 드러내겠다는 접근,
모호성을 정량화하겠다는 엔지니어링 감각 — 모두 올바른 방향이다.

그러나 구현의 무게가 철학의 날카로움을 무디게 할 위험이 있다.
166개 모듈과 10개 에이전트라는 무게가 “사양을 잘 쓰자”는 메시지의
접근성을 떨어뜨린다.
가장 날카로운 도구는 가장 가벼운 도구일 때가 많다.
