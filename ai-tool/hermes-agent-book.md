# Hermes Agent: 성장하는 AI 에이전트 실전 가이드

원문: <https://wikidocs.net/book/19414>

## 요약

NousResearch가 2026년 2월 26일에 MIT 라이선스로 공개한 오픈소스 AI 에이전트 Hermes Agent를
한국어로 안내하는 전자책이다.
지은이는 필명 “채찍피티”를 쓰는 최용이고, 위키독스에서 무료로 공개되어 있다.
2026년 7월 23일까지 갱신되었으며 v0.19.0을 기준으로 삼는다.
책 첫머리에 “이 책은 AI를 활용하여 작성되었다”는 고지가 붙어 있고,
본문의 거의 모든 절 앞에 `⚠️ 미검증` / `⚠️ 부분 검증` / `ℹ️ 실측 사례` 같은 검증 등급이 명시되어 있다.

구성은 13개 장과 부록 두 편, 총 80개 문서다.
1장에서 도구의 정체성과 경쟁 도구 비교를 다루고,
2~3장에서 설치와 첫 대화, 4장에서 도구(Tools) 시스템,
5장에서 네 가지 기억 장치, 6장에서 `SOUL.md`·`AGENTS.md`,
7장에서 스킬 시스템, 8장에서 MCP, 9장에서 22개 메시징 게이트웨이,
10장에서 크론과 영속 목표, 11장에서 5계층 보안,
12장에서 내부 아키텍처와 자기 진화, 13장에서 네 개의 실전 프로젝트를 다룬다.
부록은 Claude Code에서 Hermes로 작업을 넘기는 방법과 로컬 LLM 가이드다.

기술적 뼈대는 ReAct 루프 위에 하네스 기능을 쌓아 올린 구조다.
`run_agent.py`의 `AIAgent.run_conversation()`이 핵심 루프이고,
API 호출 경로는 트랜스포트 추상화(`ChatCompletions`·`Responses`·`Anthropic`·`Bedrock`)로 분리되어 있다.
기억은 네 갈래로 나뉜다.
매 턴 시스템 프롬프트에 들어가는 자체 메모리(`MEMORY.md` 2,200자 + `USER.md` 1,375자),
Honcho·Mem0·Holographic·Hindsight를 비롯한 8개 외부 메모리 백엔드,
SQLite FTS5 기반 세션 검색, 그리고 현재 세션 한정의 컨텍스트 압축이다.
스킬은 `agentskills.io` 표준을 따르는 마크다운 문서이고,
번들 스킬 71개(v0.17.0 기준)가 함께 설치되며 3단계 지연 로딩으로 토큰을 아낀다.

책이 반복해서 강조하는 차별점은 “성장”이다.
도구 호출이 일정 횟수 누적되면 백그라운드 리뷰 에이전트가 대화를 되짚어 스킬을 만들거나 고치고,
7일 주기로 백그라운드 큐레이터가 라이브러리 전체를 훑어 비슷한 스킬을 합치고 안 쓰는 스킬을 아카이브로 옮긴다.
`/goal`은 판정 에이전트가 매 턴 완료 여부를 확인하며 목표 달성까지 턴을 이어 가는 Ralph loop이고,
크론은 게이트웨이 프로세스 안의 60초 스레드가 돌린다.
DSPy + GEPA 기반 자기 진화 시스템은 메인 저장소가 아니라
`NousResearch/hermes-agent-self-evolution` 별도 저장소에 있다는 사실도 소스 검색으로 확인해 밝혀 둔다.

저자가 이 책의 특징으로 내세우는 것은 공식 문서 번역이 아니라는 점이다.
실제로 상당 부분이 공식 문서와 소스 코드를 대조해 어긋난 지점을 잡아내는 데 쓰인다.
공식 문서가 말하는 “읽기 전용 루트 파일시스템”이 소스에는 `--read-only` 플래그로 존재하지 않는다는 지적,
`/personality`가 문서상 세션 한정인데 실제로는 설정 파일에 저장되어 다음 세션까지 이어진다는 지적,
로컬 LLM에서 `execute_code`가 `\n` 이스케이프 처리 문제로 10회 연속 실패했다는 실측 기록이 그런 예다.
동시에 “2026년 3월 기준 Qwen3 14B는 한국어 도구 호출에 부족하다”,
“MCP는 같은 작업에 CLI 대비 약 30배 토큰을 쓴다” 같은 부정적 결론도 그대로 싣는다.

## 장별 내용

### 1장 — 소개와 골디락스 존

Hermes Agent를 OpenClaw·NanoClaw·NemoClaw·Claude Code·OpenCode·Aider·Gemini CLI·OpenHands와
프로필 카드 형태로 비교한다.
2026년 7월 기준 GitHub 스타는 OpenClaw 384k, Hermes 219k, OpenCode 188k, Gemini CLI 106k로 적는다.
Hermes의 약점도 따로 절을 두어 밝힌다 — 출시 5개월의 얕은 문서, Claude Code 수준의 IDE 통합 부재,
한국어 로컬 모델 품질 한계, SaaS가 아니어서 직접 져야 하는 인프라 운영 부담.

1-4절 “에이전트의 골디락스 존”이 이 장의 백미다.
“중간 판단이 필요한가”와 “완료를 객관적으로 확인할 수 있는가” 두 축으로 작업을 세 구간으로 나눈다.
분기 없는 반복 작업은 n8n이나 스크립트가 낫고(결정적 파이프라인),
두 질문에 모두 “예”인 구간이 에이전트가 빛나는 골디락스 존이며,
“문장을 자연스럽게 다듬어줘”처럼 완료 기준이 주관적인 작업은 자율 실행이 아니라 보조 도구로 써야 한다고 정리한다.
Microsoft Research와 Stanford의 공동 연구를 인용해
에이전틱 코딩이 일반 코드 채팅 대비 토큰을 수백~수천 배 쓰지만
정확도는 중간 비용에서 정점을 찍고 포화된다는 점도 근거로 든다.

### 2~3장 — 설치, 프로바이더, 첫 대화

v0.18에서 도입된 지원 등급이 정리되어 있다.
Tier 1은 macOS(Apple Silicon)·Native Windows·Linux/WSL2·Docker,
Tier 2는 Termux와 Nix, 미지원은 macOS Intel·PyPI·Homebrew·AUR이다.
저자의 검증 환경은 2015년형 12인치 맥북(Core M·8GB), ThinkPad X230, i9+RTX 4090 데스크톱 세 대인데,
세 대 모두 Hermes 본체는 원격 모델로 구동한다.

프로바이더 결정 우선순위는 CLI 플래그 → `config.yaml`의 `model.provider` → 환경 변수 → auto 탐지다.
“`.env`에 API 키만 추가해도 프로바이더는 바뀌지 않는다”를 초보자가 가장 자주 빠지는 함정으로 지목한다.
기본 프로바이더는 OpenRouter이고, 실습은 무료 모델
`nvidia/nemotron-3-super-120b-a12b:free`로 진행하되
무료 엔드포인트는 프롬프트와 응답이 로깅된다는 경고를 반복한다.
`hermes model`(셸, 영구)과 `/model`(세션 중, 임시)의 차이를 표로 갈라 둔 부분이 실용적이다.

### 4장 — 도구 시스템

도구는 세트 단위로 묶여 있고 `tools/registry.py`에 등록된다.
웹 3개, 터미널 2개, 파일 4개, 브라우저 12개, 미디어 5개, 오케스트레이션 5개,
메모리·검색 2개, 스킬 관리 3개, 자동화 2개, 칸반 9개 등으로 나뉜다.
`moa`·`homeassistant`·`video`·`x_search` 등은 기본 비활성이다.

터미널 백엔드는 `local`·`docker`·`ssh`·`singularity`·`modal`·`daytona`·`vercel_sandbox` 일곱 가지다.
4-4절의 Codex App-Server 런타임은 결이 다르다 —
터미널 백엔드가 “명령을 어디서 실행할지”를 바꾸는 것이라면
이쪽은 도구 루프 자체를 OpenAI Codex CLI에 위임한다.
대신 `delegate_task`·`memory`·`session_search`·`todo`는 쓸 수 없게 된다.
에이전트 루프의 상태에 접근해야 하는 도구는 무상태 MCP 콜백으로 대체할 수 없기 때문이다.

4-3절 실습에서 로컬 LLM(Qwen 2.5 14B) 실측 결과를 표로 남긴다.
터미널·파일 읽기·패치는 안정적, `execute_code`는 불안정, `web_search`는 실패였다.

### 5장 — 기억

| 장치             | 저장 위치              | 지속 범위     | 작동 방식                  |
| ---------------- | ---------------------- | ------------- | -------------------------- |
| 자체 메모리      | `~/.hermes/memories/`  | 세션 간 영구  | 매 턴 시스템 프롬프트 주입 |
| 외부 메모리      | 외부 서비스            | 세션 간 영구  | 매 턴 주입 + 도구 호출     |
| 세션 검색        | `~/.hermes/state.db`   | 세션 간 영구  | 에이전트가 도구로 검색     |
| 컨텍스트 압축    | 메모리 내(휘발)        | 현재 세션     | 토큰 임계값 도달 시 자동   |

자체 메모리는 합계 약 3,500자로 의도적으로 작게 설계되어 있다.
한도를 넘으면 새 정보가 거부되고, 자동 정리 기능은 없으며,
“정리해줘”라고 시켜도 실제로는 삭제되지 않은 채 “정리했습니다”라고 보고하는 경우가 있다고 경고한다.

외부 백엔드 8종 중 넷을 자세히 다룬다.
Honcho는 변증법적 사용자 모델링(사용자의 사고방식을 이해),
Mem0는 서버 측 LLM 팩트 추출(사실을 정확히 기억),
Holographic은 추가 LLM도 임베딩도 없이 SQLite FTS5 + HRR 대수로만 도는 로컬 팩트 저장소,
Hindsight는 원시 사실 위에 관찰 층을 쌓고 성찰(reflect)로 합성하는 지식 그래프다.
백엔드 사이 데이터 마이그레이션 기능이 없다는 점을 반복해서 강조한다.

컨텍스트 압축은 세 단계다 —
LLM 없이 오래된 도구 출력 가지치기, 앞 3개와 뒤 20개 메시지 경계 보호,
그 사이 중간 턴을 보조 모델로 구조화 요약.
재압축 시 이전 요약을 다시 만들지 않고 누적 갱신한다.

### 6장 — 퍼스널리티와 컨텍스트

`SOUL.md`는 정체성(전역), `AGENTS.md`는 프로젝트 지침(지역)이다.
중요한 함정은 `SOUL.md`가 기본 정체성을 **추가하는 게 아니라 완전히 대체**한다는 점이다.
“도움이 되는 AI”라는 지시를 빠뜨리면 그 특성이 사라진다.
30~80줄이 적정 범위이고, 성격 서술보다 행동 규칙이 잘 먹힌다고 권한다.

프로젝트 컨텍스트 파일은 우선순위대로 한 종류만 로딩된다 —
`.hermes.md`/`HERMES.md`(git root까지 상위 탐색) → `AGENTS.md`(cwd만) →
`CLAUDE.md`(cwd만) → `.cursorrules`(cwd만).
과거의 재귀적 계층 탐색은 제거되었다.
시스템 프롬프트 슬롯은 9칸이고 `SOUL.md`가 1번, 컨텍스트 파일이 7번이다.

Claude Pro/Max 구독 OAuth로 붙이면 Hermes가 요청 앞에 “You are Claude Code…”를 끼워 넣어
커스텀 정체성이 흐려진다는 관찰도 실려 있다.
저자는 이 경로를 약관상 회색지대로 규정하고 API 키 경로를 권한다.

### 7장 — 스킬

스킬은 `~/.hermes/skills/` 아래 평평하게 놓이는 마크다운 문서다.
3단계 지연 로딩이 핵심이다 —
Level 0은 시스템 프롬프트의 인덱스(스킬당 약 50토큰),
Level 1은 `skills_list`로 트리거와 예시까지,
Level 2는 `skill_view`로 전문을 읽는다.
설치된 모든 스킬은 자동으로 슬래시 명령이 된다.

자동 생성은 “도구 5회 호출” 같은 단순 트리거가 아니다.
에이전트 루프 반복이 기본 10회 누적되면 리뷰 플래그가 켜지고,
사용자 응답이 끝난 뒤 데몬 스레드에서 리뷰 에이전트가 최대 16회 반복으로 검토한다.
v0.12에서 이 리뷰 포크가 정비되었다 —
자유 형식 판단 대신 루브릭 평가, 직전에 로드한 스킬 우선 갱신,
부모 대화의 프로바이더·모델 상속, `memory`·`skills` 도구 세트로 한정.

7-6절 백그라운드 큐레이터는 이 책에서 가장 흥미로운 절 중 하나다.
7일 주기로 깨어나 1단계에서는 LLM 없이 마지막 사용 시각으로 상태를 전이시키고
(30일 미사용 → `stale`, 90일 → `archived`),
2단계에서 보조 모델이 비슷한 스킬을 합치거나 가치 낮은 스킬을 솎아낸다.
매 실행마다 `run.json`과 사람이 읽는 `REPORT.md`를 남기고,
실행 직전 `tar.gz` 스냅숏을 자동으로 만들며,
`hermes curator rollback` 한 명령으로 되돌릴 수 있다.
허브 스킬은 항상 제외되고, 핀한 스킬은 자동 전이를 받지 않는다.

### 8장 — MCP

Hermes의 도구를 3계층으로 정리한다 — 빌트인 도구, 스킬, MCP.
MCP를 쓰지 말아야 할 경우를 먼저 밝히는 구성이 특이하다.
셸 명령은 LLM이 이미 알고 있어 감싸면 손해이고,
반복 워크플로는 스킬이 간단하며,
ScaleKit 벤치마크 기준 같은 작업에 CLI는 약 1,400토큰, MCP는 약 44,000토큰을 쓴다고 인용한다.
판단 기준은 “이 작업을 터미널 명령어 한 줄로 할 수 있는가”다.

도구 필터링(`tools.include`/`exclude`)을 중요 항목으로 다룬다.
GitHub MCP 서버 하나가 30개 넘는 도구 스키마를 컨텍스트에 올리기 때문이다.
8-4절은 반대 방향 —`hermes mcp serve`로 Hermes 자신을 MCP 서버로 노출해
Claude Code나 Cursor에서 Hermes의 메신저 대화에 접근하게 하는 구성을 다룬다.
메시지 읽기는 승인 없이, 전송은 승인을 거치도록 설계되어 있다.

### 9장 — 메시징 게이트웨이

v0.14.0 기준 22개 플랫폼을 지원한다.
Telegram·Discord·WhatsApp·Slack·Signal·Mattermost·Matrix·Email·SMS·DingTalk·WeCom·
WeChat·Feishu·iMessage·QQBot·Yuanbao·Teams·LINE·SimpleX Chat·HomeAssistant·API Server 등이다.
게이트웨이는 상시 실행 프로세스이고, systemd 유저 서비스나 launchd로 등록하는 것이 권장된다.

Discord 연동 절은 실제 연결 기록에 근거해 상당히 촘촘하다.
Message Content Intent를 켜지 않으면 `PrivilegedIntentsRequired`로 접속조차 못 한다는 것,
“Hermes”라는 사용자명은 이미 한도가 차서 등록할 수 없다는 것(2026-07 확인),
허용 목록을 비워 두면 봇이 온라인이어도 아무에게도 응답하지 않는다는 것까지 적어 둔다.

### 10장 — 크론과 영속 목표

크론은 OS의 `crontab`이나 `launchd`를 쓰지 않는다.
게이트웨이 프로세스 안의 백그라운드 스레드가 60초마다 훑는다.
따라서 해상도는 1분이고, 게이트웨이가 꺼져 있으면 아무것도 돌지 않는다.
각 작업은 격리된 세션에서 실행되고 크론 실행 중에는 `cronjob` 도구가 비활성화된다(재귀 방지).
`workdir`을 지정하면 그 디렉터리의 `AGENTS.md`가 주입되고,
`context_from`으로 다른 작업의 최근 성공 출력을 이어받을 수 있으며,
`[SILENT]` 마커로 이상이 있을 때만 알림을 받는 패턴을 만든다.

10-3절은 이 책에서 유일하게 `ℹ️ 실측 사례`로 표시된 절이다.
네이버 카페 스터디 모니터링 크론의 비용이 이상해 보여 파고든 기록인데,
같은 잡의 AM 세션(25회 호출, $0.49)과 PM 세션(37회 호출, $1.02)을 세션 JSONL까지 파싱해 비교한다.
차이의 원인은 Linear GraphQL mutation 스키마를 매번 새로 시행착오로 발견하느라
디버깅 6라운드가 들어간 것이었고,
결론은 “시행착오를 영구화하라”였다.
에이전트에게 던진 첫 질문이 “지난 24시간 호출 패턴 보고 어디서 토큰을 많이 쓰는지 짚어줘”라는
평범한 한 문장이었다는 점을 저자는 강조한다.

`/goal`은 목표 텍스트가 첫 턴 사용자 메시지로 들어가고,
매 턴 끝에 판정 에이전트가 `{“done”: true/false, “reason”: “...”}`를 반환하는 구조다.
기본 턴 예산은 20턴이고, 판정 모델은 `auxiliary.goal_judge` 슬롯에
저렴한 모델을 따로 지정하라고 권한다.
소스에서 이 기능은 “Ralph loop”으로 불리며 OpenAI Codex CLI의 동명 기능에서 영감을 받았다.

### 11장 — 보안

공식 문서는 7계층을 말하지만 책은 핵심 5계층으로 정리한다 —
사용자 인증, 위험 명령 승인, 컨테이너 격리, MCP 자격증명 필터링, 컨텍스트 파일 스캔.

인증은 기본값이 “거부”다.
DM 페어링은 8자리 코드를 솔트 + SHA-256으로 저장하고 상수 시간 비교로 검증하며,
1시간 만료·10분당 1회·실패 5회 시 1시간 잠금 같은 제약이 붙는다.
승인은 `DANGEROUS_PATTERNS`(승인 요청)와
v0.11에서 추가된 `HARDLINE_PATTERNS`(승인 모드 무관 차단, `/yolo`로도 우회 불가)로 나뉜다.

Docker 백엔드의 실제 플래그는 소스에서 확인해 나열한다 —
`--cap-drop ALL` 후 `DAC_OVERRIDE`·`CHOWN`·`FOWNER`만 복원,
`no-new-privileges`, `--pids-limit 256`,
`/tmp` 512MB·`/var/tmp` noexec 256MB·`/run` 64MB tmpfs.
여기서 공식 문서의 “읽기 전용 루트 파일시스템”이 실제로는 없다고 밝힌다 —
패키지 매니저가 작동해야 하기 때문이다.

11-6절 프로파일은 한 머신에서 여러 Hermes 인스턴스를 돌리는 가장 가벼운 격리 수단이다.
`config.yaml`·`.env`·`SOUL.md`·메모리·세션·스킬·크론이 프로파일별로 갈리고,
코드와 번들 스킬 갱신만 공유된다.

### 12장 — 아키텍처와 자기 진화

이터레이션 버짓은 부모와 서브에이전트가 공유하며 기본 90이다.
`execute_code`로 소비된 이터레이션은 실제 추론 턴이 아니므로 환불된다.
도구는 조건이 맞으면 최대 8워커로 병렬 실행되는데,
`clarify`는 절대 병렬 대상이 아니고 파일 도구는 경로가 겹치지 않을 때만 병렬을 허용한다.

12-2절이 이 책의 검증 태도를 가장 잘 보여 준다.
저자는 메인 저장소에서 “GEPA”와 “self-evolution”을 검색해 결과가 전혀 없음을 확인하고,
자기 진화 시스템이 별도 저장소에 있다는 사실을 밝힌다.
나아가 README의 “ICLR 2026 Oral”이 Hermes Agent가 학회에 발표되었다는 뜻이 아니라
Stanford DSPy 팀의 GEPA 논문을 가리킨다는 점까지 바로잡는다.

### 13장과 부록

실전 프로젝트는 코드 분석 에이전트, 일일 뉴스 브리핑, 팀 협업 봇, Mac Mini 홈 허브 넷이다.
부록 「Claude Code에서 Hermes로 인계하기」는 흥미로운 역할 분담을 제안한다 —
설계·프로토타입·디버깅은 Claude Code로, 반복 유지보수와 24/7 자율 실행은 Hermes로.
부록 「로컬 LLM 가이드」는 Ollama·LM Studio 연동과 VRAM별 모델 추천을 다루고,
2026년 4월 출시된 Gemma 4(26B)가 한국어 품질과 도구 호출에서 처음으로 쓸 만해졌다고 기록한다.
다만 RTX 4090에서 32K 컨텍스트로 돌리면 VRAM을 넘겨 CPU 오프로딩이 발생하고,
체감상 “속도는 Opus급, 품질은 Haiku급”이 된다고 적는다.

## 분석

### 검증 등급 표기가 이 책의 진짜 발명이다

이 책에서 가장 눈에 띄는 것은 내용이 아니라 형식이다.
거의 모든 절 앞에 그 절의 서술이 어디에 근거하는지가 명시되어 있다.
`⚠️ 미검증`은 공식 문서만 보고 정리했다는 뜻이고,
`⚠️ 부분 검증`은 소스 코드를 대조했지만 실행 테스트는 하지 않았다는 뜻이며,
`ℹ️ 실측 사례`는 실제로 돌린 기록이라는 뜻이다.
어떤 파일을 대조했는지(`tools/approval.py`, `agent/curator.py`)까지 적는 경우도 많다.

이 장치는 “AI를 활용하여 작성되었다”는 첫머리 고지와 짝을 이룬다.
AI가 초안을 쓰는 기술 문서에서 가장 위험한 것은 그럴듯한 서술과 확인된 서술이 구별되지 않는다는 점이다.
검증 등급은 그 구별을 문서의 1급 요소로 끌어올린다.
독자는 한 문단을 읽으면서 그 문단을 어느 정도로 믿어야 하는지 함께 알게 된다.

효과는 즉각적이다.
“읽기 전용 루트 파일시스템”이 소스에 없다는 지적,
`/personality`가 문서와 달리 영구 저장된다는 지적,
GEPA가 Hermes의 성과가 아니라 인용한 외부 기법이라는 정정 —
이런 발견은 “공식 문서를 믿지 않고 소스를 연다”는 규율에서 나온다.
그리고 그 규율은 검증 등급 표기가 강제한다.
등급을 적으려면 무엇을 대조했는지 스스로 답해야 하기 때문이다.

### 도구 선택 기준을 세 겹으로 반복해서 세운다

책의 논지는 “Hermes를 쓰세요”가 아니라 “언제 무엇을 쓸지 판단하세요”에 가깝다.
같은 형태의 판단 프레임이 세 겹으로 반복된다.

첫째는 1-4절의 골디락스 존이다 — 에이전트를 쓸 것인가 스크립트를 쓸 것인가.
둘째는 8장의 3계층 구조다 — 빌트인 도구인가 스킬인가 MCP인가.
셋째는 5장의 네 기억 장치와 백엔드 비교다 — 무엇을 항상 보이게 두고 무엇을 검색으로 꺼낼 것인가.
세 경우 모두 “가장 단순한 것부터, 측정 가능한 개선이 있을 때만 복잡도를 더하라”는
Anthropic의 에이전트 구축 가이드 원칙을 변주한다.

이 반복은 우연이 아니다.
에이전트 도구의 세일즈 포인트는 대개 “무엇이든 할 수 있다”인데,
그 주장을 그대로 받아들이면 결정적 파이프라인에 LLM을 붙여 토큰만 태우게 된다.
책은 이 함정을 알고 있고, 도구를 소개할 때마다 “쓰지 말아야 할 경우”를 먼저 배치한다.
MCP 장이 MCP의 44,000토큰 오버헤드로 시작하는 구성이 대표적이다.

### 하네스와 모델을 분리하는 서사가 책 전체의 뼈대다

1-2절의 “모델이 아니라 하네스에 가치가 쌓인다”가 이 책의 중심 명제다.
모델은 뇌이고 하네스는 실행 환경이며,
같은 모델이라도 하네스에 따라 성능이 크게 달라지므로
모델은 갈아 끼우고 메모리·스킬·워크플로는 남는다는 주장이다.

이 명제는 책의 구성 자체를 설명한다.
검증 환경 세 대가 각각 OpenAI·Anthropic·OpenRouter로 프로바이더만 다른 것도,
2-3절이 모델명의 슬래시 접두사가 라우팅과 무관하다는 점을 집요하게 반복하는 것도,
트랜스포트 추상화를 아키텍처 장의 앞자리에 두는 것도 같은 주장의 다른 표현이다.
“어떤 모델을 쓰느냐”를 설정 항목으로 격하시키는 것이 이 책의 서술 전략이다.

동시에 이 서사는 오픈소스 후발 주자의 포지셔닝이기도 하다.
Hermes는 자체 모델 계열을 갖고 있지만 그것을 앞세우지 않는다.
Claude Code가 Anthropic 모델에, NanoClaw가 Claude Agent SDK에,
NemoClaw가 NVIDIA 생태계에 묶여 있다는 점을 비교 카드마다 “한계”로 적는 것과 짝을 이룬다.
모델 종속을 경쟁 도구의 약점으로 규정하면, 모델을 갖지 않은 쪽이 유리해진다.

### 자율성의 대가를 계층마다 다시 계산한다

책이 다루는 기능은 대부분 자율성을 높이는 장치다.
자동 스킬 생성, 백그라운드 큐레이터, 크론, `/goal`, 메시징 게이트웨이가 모두 그렇다.
그리고 매번 그 대가가 따라붙는다.

자동 스킬 생성은 로컬 LLM에서 불필요한 스킬을 만들거나 기회를 놓친다.
큐레이터는 정리를 자동화하는 대신 잘못 정리할 위험을 안고,
그래서 자동 백업·핀·롤백·보고서라는 네 겹의 되돌리기 장치가 붙는다.
`/goal`은 사람이 Enter를 누르지 않아도 되게 만드는 대신
20턴 예산과 Ctrl+C 즉시 정지와 별도 판정 모델 비용을 만든다.
크론은 무인 실행을 주는 대신 게이트웨이 상시 가동을 요구한다.

이 패턴을 관통하는 것은 1-4절의 두 번째 축 — “완료를 객관적으로 확인할 수 있는가”다.
자율성이 올라갈수록 검증 지점이 사라지고, 사라진 검증 지점을 기계적 장치로 메워야 한다.
큐레이터의 `REPORT.md`도, `/goal`의 판정 에이전트도,
크론의 `[SILENT]` 패턴도 결국 사라진 사람의 눈을 대신하려는 시도다.
책은 이 점을 명시적으로 이론화하지는 않지만, 기능마다 되풀이해 보여 준다.

## 비평

### “완전한 데이터 주권”이라는 문구가 책 자신의 기록과 충돌한다

1-2절은 Hermes를 선택할 이유의 두 번째로 “완전한 데이터 주권”을 든다.
로컬 LLM과 연동하면 프롬프트와 작업 데이터가 외부로 나가지 않는다는 것이다.
그런데 이 문장을 책의 나머지 기록과 나란히 놓으면 남는 것이 거의 없다.

첫째, 저자의 검증 환경 세 대가 모두 원격 모델로 돈다.
RTX 4090이 달린 데스크톱조차 기본 모델이 OpenRouter 경유 `deepseek/deepseek-v4-pro`다.
둘째, 부록은 2026년 3월 기준 Qwen3 14B의 한국어 도구 호출이 부족하다고 적고,
개선되었다는 Gemma 4(26B)조차 RTX 4090에서 32K 컨텍스트를 못 담아 CPU로 흘러넘친다고 적는다.
셋째, 외부 메모리 백엔드 중 Honcho·Mem0·Hindsight는 모두 서버 측 LLM을 별도로 호출한다.
대화 내용이 관찰 추출을 위해 또 다른 LLM에 전달되는 구조이니,
메모리를 쓰는 순간 데이터 경로가 하나 더 늘어난다.

즉 “완전한 데이터 주권”은 한국어 사용자에게 사실상 성립하지 않거나,
성립시키려면 품질·속도·기억 기능을 동시에 포기해야 하는 선택지다.
책은 이 사실을 흩어진 형태로 정직하게 적어 두었으면서도,
정작 도구를 고르는 근거를 제시하는 자리에서는 그 조건을 지운 채 장점만 내세운다.
선택 이유를 다루는 절이야말로 조건을 함께 적어야 할 자리다.

### 골디락스 존을 세워 놓고 자기 도구에는 적용하지 않는다

1-4절은 이 책에서 가장 잘 벼려진 부분이다.
그런데 그 기준으로 13장의 실전 프로젝트를 재 보면 결과가 불편해진다.

13-2절 일일 뉴스 브리핑은 “매일 8시에 키워드로 검색하고 3줄 요약해서 Telegram으로 보낸다”이다.
단계가 고정되어 있고 분기가 없다.
1-4절의 정의를 그대로 적용하면 이것은 결정적 파이프라인이고,
저자 자신이 “RSS 피드를 읽어서 정해진 템플릿으로 요약하고 슬랙에 올리기”를
에이전트를 쓰지 말아야 할 예시로 든 것과 거의 같은 작업이다.
13-4절의 야간 루틴(조명 끄기, 잠금 확인, 미리알림 알려주기)도 마찬가지다.

책은 이 충돌을 언급하지 않는다.
언급했다면 오히려 좋은 절이 되었을 것이다 —
“이 작업은 골디락스 존 바깥이지만, 자연어 한 줄로 등록할 수 있다는 진입 비용 때문에 에이전트가 이긴다”는
설명이 가능하고, 그것이 에이전트 도구의 실제 가치 명제에 가깝다.
대신 책은 판단 기준을 1장에 두고, 그 기준을 쓰지 않은 채 13장을 쓴다.
가장 좋은 도구를 만들어 놓고 서랍에 넣어 둔 셈이다.

10-3절 카페 모니터 사례는 이 대비를 더 선명하게 만든다.
그 절의 작업 — 로그를 파싱하고 원인을 짚고 재편안을 설계하는 일 — 은
중간 판단이 필요하고 결과를 객관적으로 확인할 수 있는 전형적인 골디락스 존 작업이다.
책에서 가장 설득력 있는 절이 하필 기준에 정확히 들어맞는 절이라는 사실은 우연이 아니다.

### 대부분이 “부분 검증”인데 표지는 실습서를 표방한다

검증 등급 표기는 이 책의 미덕이지만, 등급의 분포를 세어 보면 다른 그림이 나온다.
실행 테스트를 거친 절은 손에 꼽는다 —
Intel Mac 원라인 설치, macOS 데스크톱 앱 설치, Ubuntu Discord 연결,
himalaya Gmail 연동, 로컬 LLM 도구 호환성 테스트, 카페 모니터 크론 진단 정도다.
나머지 대다수는 `⚠️ 부분 검증`, 즉 “소스 코드를 읽고 정리했으나 실행하지 않았다”이다.
보안 장, 아키텍처 장, 스킬 자동 생성 장, 큐레이터 장이 모두 여기 속한다.

그런데 책 소개는 “공식 문서의 단순 번역이 아닌, 실습 경험을 생생히 전달합니다”로 시작하고,
목차에는 “실습:”으로 시작하는 절이 여섯 개 있다.
그중 6-3절 「실습: 나만의 에이전트 만들기」는 `⚠️ 미검증`으로,
“실습 내용은 소스 코드 분석과 커뮤니티 사례를 바탕으로 설계했으며,
실제 테스트는 추후 진행 예정”이라고 적혀 있다.
설계된 실습과 수행된 실습은 다른 것이다.

이것이 문서의 가치를 없애지는 않는다.
소스 대조는 그 자체로 값진 작업이고, 공식 문서보다 정확한 서술을 여러 번 만들어 냈다.
문제는 장르 표기다.
이 책은 실습서라기보다 “소스 코드를 읽고 쓴 독해 노트에 일부 실측을 곁들인 문서”에 가깝다.
그렇게 소개했다면 검증 등급 표기의 정직함과 책 소개의 표현이 어긋나지 않았을 것이다.
가장 정직한 장치를 갖춘 책이 자기 소개에서만 그 정직함을 유지하지 못한 셈이다.

### 빠르게 변하는 수치를 판단 근거로 삼는 구조

1-3절의 비교 카드는 GitHub 스타 수를 각 도구의 “현황” 첫 줄에 놓는다.
OpenClaw 384k, Hermes 219k, OpenCode 188k, Gemini CLI 106k, OpenHands 82k, Aider 48k, NanoClaw 30k.
책은 이 수치가 빠르게 변한다는 경고를 절 앞에 달아 두었다.
그러나 경고는 문제를 인식했다는 표시일 뿐 해결이 아니다.

더 근본적인 문제는 스타 수가 판단 근거로서 거의 아무것도 말해 주지 않는다는 점이다.
책 자신도 그것을 안다 —
같은 절에서 Hermes가 스타는 OpenClaw의 절반 남짓이지만
2026년 5월 OpenRouter 일일 토큰 처리량 1위로 실사용에서는 앞섰다고 적는다.
두 지표가 반대 방향을 가리키는데도 스타 수가 여전히 첫 줄을 차지한다.

같은 문제가 모델 가격표에도 있다.
10-4절은 여섯 모델의 입출력 단가를 표로 싣고 “가격은 수시로 변동된다”는 주석을 단다.
2-3절도 Claude 4종의 단가를 싣는다.
전자책이라 갱신이 가능하다는 점이 완충 장치이긴 하나,
갱신 부담을 저자 혼자 지는 구조에서 이런 표는 결국 낡는다.
판단의 축(도구 호출 안정성, 컨텍스트 처리, 구조화 출력 정확도)만 남기고
수치는 확인 방법으로 대체했다면 문서의 수명이 훨씬 길었을 것이다.

## 인사이트

### 하네스에 가치가 쌓인다는 명제에는 유효기간이 있다

“모델이 아니라 하네스에 가치가 쌓인다”는 주장은 지금은 참이다.
같은 모델이라도 도구 정의, 컨텍스트 관리, 승인 체계에 따라 결과가 크게 달라진다.
그런데 이 주장이 참인 이유를 뒤집어 보면 유효기간도 함께 보인다.
하네스가 가치를 갖는 것은 모델이 아직 못 하는 일이 있기 때문이다.

책 안에서 이미 침식의 증거가 나온다.
4-4절의 Codex App-Server 런타임은 도구 루프 전체를 OpenAI의 런타임에 넘긴다.
그 순간 Hermes는 터미널 실행도 파일 편집도 샌드박싱도 하지 않는다 —
세션 관리, 메모리, 스킬 리뷰, 게이트웨이, 크론이라는 외곽만 남는다.
프론티어 랩이 자기 CLI를 앱 서버로 노출하기 시작하면
하네스의 안쪽 절반은 모델 제공자에게 흡수된다.
`hermes proxy`가 OAuth 프로바이더를 외부 도구에 중계하는 얇은 층인 것도 같은 방향의 신호다.

그렇다면 남는 질문은 “하네스의 어느 부분이 모델에 흡수되지 않는가”다.
책의 서술을 근거로 추리면 답은 좁다 —
사용자 개인의 축적물(메모리와 스킬), 조직 경계(인증·허용 목록·격리), 시간축(크론과 영속 목표)이다.
셋 모두 모델 안에 넣을 수 없는 것들이다.
사용자에게 속하거나, 조직에 속하거나, 프로세스에 속하기 때문이다.
반대로 도구 정의, 루프 구현, 컨텍스트 압축, 프롬프트 조립처럼
“잘 만든 스캐폴딩”에 해당하는 부분은 다음 세대 모델에서 기본 제공될 가능성이 높다.
에이전트 하네스에 투자하는 쪽이라면 이 경계 어디에 자기 노력이 놓이는지 따져 볼 일이다.

### 자동 지식 관리는 사내 위키가 죽은 자리에 다시 서 있다

7장의 구조를 한 발 물러서서 보면 낯익은 곡선이 보인다.
에이전트가 작업하며 절차서를 자동으로 만든다 → 절차서가 쌓인다 →
비슷한 절차서가 흩어지고 안 쓰는 것이 섞인다 → 라이브러리가 잡초밭이 된다 →
7일마다 도는 큐레이터를 붙여 합치고 솎아낸다.

이것은 사내 위키의 역사 그대로다.
문서를 쉽게 만들 수 있게 하면 문서가 폭증하고,
폭증한 문서는 검색 결과를 오염시키며,
그 시점에 조직은 “위키 가드너”를 지정하거나 정리 스프린트를 돌린다.
그리고 대부분의 조직에서 그 노력은 지속되지 못한다.
사람이 하는 정리는 티가 안 나고 급하지도 않기 때문이다.

Hermes의 답은 정리마저 자동화하는 것이다.
그러나 이 답은 새로운 종류의 문제를 만든다 —
사람이 읽지 않는 문서를 기계가 만들고, 기계가 합치고, 기계가 버린다.
`REPORT.md`와 롤백 스냅숏은 사람이 감시할 자리를 남겨 둔 장치이지만,
큐레이터가 7일마다 조용히 도는 이상 그 보고서를 매번 읽을 사람은 없다.
결국 “안 쓰는 스킬이 정리되고 있다”는 믿음만 남고 검증은 사라진다.

여기서 나오는 실천적 함의는 핀(pin) 기능의 위상이다.
책은 핀을 자동 정리 방지 장치로 담담하게 소개하지만,
자동 큐레이션 체제에서 핀은 “사람이 책임지는 문서”를 선언하는 표시다.
사람이 손으로 다듬은 소수의 핀 스킬과,
기계가 만들고 기계가 치우는 다수의 비핀 스킬로 라이브러리가 두 층으로 갈리는 것 —
이것이 자동 지식 관리의 실제 정착 형태일 가능성이 높다.
그리고 실제 가치는 거의 전적으로 위층에 있을 것이다.

### 3,500자 메모리 상한은 결함이 아니라 정책이고, 그 정책이 락인을 만든다

`MEMORY.md` 2,200자와 `USER.md` 1,375자라는 상한은 처음 보면 인색해 보인다.
책도 한도 초과 시의 문제를 여러 줄에 걸쳐 적는다 —
새 정보 거부, 자동 정리 없음, 정리 실패를 성공으로 보고하는 현상.

그러나 이 상한은 매 턴 시스템 프롬프트에 들어간다는 조건에서 나온 필연이다.
3,500자는 약 1~2천 토큰이고, 그것이 모든 요청에 곱해진다.
게이트웨이와 크론까지 돌리면 하루 수백 회다.
상한을 두 배로 늘리는 설정(`memory_char_limit`)이 있는데도 기본값이 작은 이유가 여기 있다 —
이것은 저장 공간의 문제가 아니라 반복 비용의 문제다.

흥미로운 것은 이 제약이 만들어 낸 시장이다.
용량이 부족하니 외부 백엔드가 필요하고,
Honcho·Mem0·Hindsight·RetainDB·Supermemory 같은 서비스가 그 슬롯을 두고 경쟁한다.
그리고 책은 조용히 결정적인 한 줄을 덧붙인다 —
백엔드 사이에 데이터 마이그레이션 기능은 없다.

이 조합이 문제다.
동시에 하나만 활성화할 수 있고, 옮길 수 없으며, 축적될수록 가치가 커진다.
전환 비용이 시간에 비례해 커지는 구조는 정의상 락인이다.
로컬 LLM으로 모델 종속을 피하고 하네스에 가치를 쌓겠다고 시작한 사용자가
메모리 백엔드 선택 한 번으로 다른 종류의 종속에 들어가는 것이다.
Holographic이 유일하게 로컬 SQLite 파일 하나로 도는 백엔드라는 점은,
기능 비교표에서보다 이 관점에서 훨씬 중요해진다.
데이터 주권을 진지하게 따진다면 “어느 백엔드가 똑똑한가”보다
“어느 백엔드가 내 파일 시스템 위에 있는가”가 먼저 와야 한다.

### 문단 단위 출처 표기는 AI 시대 기술 문서의 최소 규범이 될 것이다

이 책의 검증 등급 표기는 개인의 성실함으로 읽히기 쉽지만,
그보다는 새로운 문서 규범의 초기 형태로 보는 편이 유용하다.

기술 문서의 전통적인 신뢰 구조는 저자 단위였다.
누가 썼는지, 어느 출판사인지, 몇 쇄인지가 신뢰의 단위였고,
독자는 문서 전체를 하나의 신뢰 등급으로 소비했다.
이 구조는 저자가 쓴 문장과 저자가 확인한 사실이 대체로 일치한다는 가정 위에 서 있었다.

AI가 초안을 쓰면 그 가정이 깨진다.
같은 문서 안에 실행해 본 것, 소스를 읽은 것, 문서를 옮긴 것,
모델이 그럴듯하게 채운 것이 문체 차이 없이 섞인다.
독자는 문체로 그것을 구별할 수 없다 — 이것이 AI 생성 기술 문서의 핵심 위험이다.
그래서 신뢰의 단위가 문서에서 문단으로 내려와야 한다.

이 책이 보여 주는 형태는 소박하다.
절 앞의 한 문단, 대조한 파일 경로, 확인 날짜, 무엇을 하지 않았는지에 대한 명시.
그러나 이 소박한 장치가 실제로 오류를 잡아냈다는 사실이 중요하다.
`--read-only` 플래그의 부재도, `/personality`의 영속성도, GEPA의 출처도
“등급을 적으려면 근거를 대야 한다”는 규율이 없었다면 나오지 않았을 발견이다.
형식이 내용을 만든 사례다.

한 걸음 더 나가면, 이 규범은 기술 문서를 넘어 코드베이스에도 적용될 수 있다.
AI가 생성한 코드와 사람이 검토한 코드,
테스트로 확인된 동작과 추정된 동작을 같은 색으로 칠해 두면
언젠가 비용을 치른다.
문단마다 인식론적 지위를 적어 두는 습관은 결국
“이 줄을 누가 어떤 근거로 믿고 있는가”를 코드 리뷰의 1급 질문으로 만드는 방향과 이어진다.
