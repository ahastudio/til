# HackerRank Hiring Agent

<https://github.com/interviewstreet/hiring-agent>

HN 토론: <https://news.ycombinator.com/item?id=48713832> (967점, 406개 댓글)

## 소개

Hiring Agent는 HackerRank(interviewstreet)가 오픈소스로 공개한 AI 기반 이력서 평가 시스템이다.
PDF 이력서를 구조화된 데이터로 변환하고, GitHub 활동 데이터로 보강한 뒤,
LLM을 통해 채점과 설명을 생성한다.
로컬 추론 엔진인 Ollama(기본값: `gemma3:4b`)와 Google Gemini API 두 가지 백엔드를 지원한다.
MIT 라이선스로 공개되어 있으며, 공개 시점 기준 3,800개 이상의 스타를 받았다.

## 아키텍처

처리 흐름은 5단계 파이프라인으로 구성된다.

1. **PDF 추출** — `pymupdf_rag.py`가 PDF 페이지를 마크다운 텍스트로 변환한다.
2. **섹션 파싱** — `pdf.py`가 Jinja 템플릿을 통해 각 섹션(기본 정보, 경력, 학력, 기술, 프로젝트, 수상 등)을 구조화된 JSON으로 추출한다.
3. **GitHub 보강** — `github.py`가 이력서에서 GitHub 사용자명을 찾아 프로필과 레포지토리를 가져오고, 프로젝트를 유형별로 분류하며, 최소 커밋 수를 기준으로 상위 7개 기여를 선별한다.
4. **평가** — `evaluator.py`가 오픈소스 기여, 개인 프로젝트, 실무 경험, 기술 깊이, 보너스/감점 항목에 걸쳐 채점을 수행한다.
5. **출력** — `score.py`가 전체 파이프라인을 조율하고 결과를 내보낸다.

프롬프트는 `prompts/templates/` 디렉토리의 Jinja 템플릿 파일로 관리된다.
섹션별로 별도 템플릿이 있으며(`awards.jinja`, `work.jinja`, `projects.jinja` 등),
채점 로직은 `resume_evaluation_*.jinja`에 담겨 있다.
데이터 표현은 JSON Resume 표준으로 정규화된다.

개발 모드(`DEVELOPMENT_MODE = True`)가 활성화되면 중간 처리 결과가 `cache/` 디렉토리에 JSON으로 저장되고, 평가 결과는 `resume_evaluations.csv`에 누적된다.

## 분석

### 추출과 평가의 분리가 갖는 공학적 의미

5단계 파이프라인의 가장 중요한 설계 결정은 PDF 파싱과 LLM 평가를 별도 단계로 분리한 것이다.
많은 유사 도구들이 원시 PDF 텍스트를 LLM에 직접 던지는 방식을 택한다.
이 도구는 Jinja 템플릿으로 섹션을 먼저 구조화된 JSON으로 변환한 뒤 평가한다.
이 분리는 디버깅 가능성과 캐싱을 가능하게 한다.
어느 단계에서 오류가 발생했는지 추적할 수 있고, 파싱 결과를 재사용할 수 있다.

그러나 이 구조는 오류가 누적되는 경로이기도 하다.
PDF 파싱이 잘못되면 이후 모든 단계가 오염된 데이터를 처리한다.
섹션 파싱이 GitHub 사용자명을 인식하지 못하면 GitHub 보강 단계 전체가 생략된다.
Dan Kinsky의 실험에서 발견된 GitHub 프로필 인식 실패는 이 두 번째 단계의 오류다.
파이프라인이 깊을수록 첫 단계의 실수가 최종 점수에 미치는 영향이 증폭된다.

### Jinja 템플릿이 채용 의사결정을 코드베이스에 담는다

이 프로젝트의 핵심은 `prompts/templates/` 디렉토리다.
채점 기준, 공정성 제약, 가중치 설명이 모두 `.jinja` 파일에 있다.
이것은 채용 정책이 Git 저장소로 버전 관리된다는 의미다.
“학벌과 GPA를 평가하지 말라”는 지침이 `resume_evaluation_system_message.jinja`에 하드코딩되어 있다.

이런 방식은 투명성의 측면에서 기존 블랙박스 ATS보다 낫다.
코드를 읽으면 도구가 무엇을 평가하는지 알 수 있다.
그러나 이 투명성은 동시에 취약점이 된다.
시스템이 무엇을 보상하고 무엇을 벌하는지 누구나 읽을 수 있다면,
지원자들은 채점 루브릭에 맞게 이력서를 최적화할 수 있다.
평가 기준이 공개될수록 그 기준이 측정하려는 것과 측정 결과 사이의 간격이 벌어진다.

seanieb는 이 구조적 문제를 다른 각도에서 비판한다.[^seanieb]
우수한 엔지니어 한 명에게 30만 달러 이상을 쓰는 회사가
채용 담당자에게는 아무 지원도 하지 않고, 엔지니어링·데이터 팀의 검토도 없이
Gmail 연동 기능 같은 것을 보고 ATS를 선택한다는 것이다.
이 도구의 템플릿 품질이 낮다면, 그것은 단순히 소프트웨어 품질의 문제가 아니다.
채용 인프라를 누가 결정하고 누가 책임지는지에 관한 조직 구조의 문제다.

### GitHub 보강이 파이프라인의 중심에 있는 것

전체 파이프라인에서 GitHub 보강 단계는 특별한 위치를 차지한다.
단순한 데이터 수집이 아니라 LLM이 상위 7개 기여를 “선별”하는 판단을 포함한다.
이 7개 기여가 평가의 핵심 입력 데이터가 된다.

최소 커밋 수 필터가 존재한다는 것도 주목할 만하다.
“커밋 수가 충분하지 않은 기여는 고려하지 않는다”는 기준이 내재되어 있다.
이것은 1회성 버그 수정이나 문서 기여를 무시할 수 있다는 의미다.
코드를 많이 작성하는 것이 좋은 기여자임을 뜻하지 않는데도,
커밋 수는 여전히 스크리닝 기준으로 작동한다.

## 비평

### “공정성 제약”은 내용 없는 레이블이다

README는 이 시스템이 “fairness-constrained scoring”을 수행한다고 명시한다.
그러나 README 어디에도 공정성(fairness)이 무엇을 의미하는지, 어떻게 구현되는지 설명이 없다.
학벌과 GPA를 제외하는 것이 공정성인가.
오픈소스 기여에 65점을 배정하는 것은 공정한가.

공정성은 기준을 정의하지 않으면 주장이 되지 않는다.
어떤 인구통계 집단을 기준으로 공정성을 측정했는가.
어떤 데이터셋으로 편향을 검증했는가.
이 질문들에 대한 답이 없다면, “공정성 제약”은 마케팅 문구에 가깝다.
오히려 학벌/GPA 제외가 오픈소스 활동 가중치의 편향을 숨기는 방향으로 작동할 수 있다.
두 가지 모두 동등하게 사회경제적 배경을 반영하는 신호이기 때문이다.

학벌과 GPA 제외 조항은 HN 커뮤니티에서도 의문을 낳았다.
kailpa1은 `resume_evaluation_system_message.jinja`의 해당 지침을 직접 인용하며 왜 이 항목들을 제외하는지 묻는다.[^kailpa1]
이에 대한 답변으로 swiftcoder는 법적 이유를 든다.[^swiftcoder]
“MIT 졸업생만 뽑는 것은 좋아 보일 수 있지만, HBCU(흑인 전통 대학) 출신 지원자를 자동 탈락시키면 소송감이다.”
LLM이 학교 명성에 과도하게 의존하는 것을 방지하고, 몇 학점 차이가 수년 후 업무 성과와 상관관계가 없다는 인식도 반영됐다.
이 설명은 타당하지만, 학벌 편향을 피하려다 GitHub 편향으로 이동하는 것이 공정성의 진전인지는 여전히 불분명하다.

### GitHub 의존성은 편향을 없애지 않고 이동시킨다

GitHub 데이터를 핵심 입력으로 삼는 설계는 “학벌보다 실력”이라는 철학을 담고 있다.
그러나 GitHub 활동도 중립적인 능력 측정이 아니다.
오픈소스 기여자 중 여성의 비율은 남성보다 현저히 낮다고 여러 연구가 보고한다.
개발도상국의 개발자들은 여가 시간 코딩보다 생계를 위한 실무에 집중할 가능성이 높다.
GitHub 프로필이 없거나 활동이 적은 개발자가 GitHub 활동이 많은 개발자보다 역량이 낮다는 증거는 없다.

이 도구는 학벌 편향을 GitHub 활동 편향으로 교체한다.
두 편향 모두 특정 집단에게 불리하게 작동하지만, 후자는 더 정당해 보이는 외관을 갖는다.
“코드로 증명하라”는 메시지는 공정해 보이지만,
코드를 공개적으로 작성할 수 있는 환경 자체가 특권이라는 사실을 가린다.

### 개발 모드 설계는 프로덕션 사용을 전제한다

`config.py`의 `DEVELOPMENT_MODE = True` 설정은 캐싱과 CSV 내보내기를 활성화한다.
이 설계는 이 도구가 반복적인 대규모 이력서 처리를 위해 설계됐음을 보여준다.
`resume_evaluations.csv`에 결과가 누적된다는 것은 이 도구가 채용 결정의 데이터베이스가 된다는 뜻이다.

그런데 Dan Kinsky의 실험이 보여준 것처럼, 동일 이력서도 66~99점이라는 큰 범위로 변동한다.
CSV에 기록된 점수는 어떤 실행의 결과인가.
같은 이력서를 두 번 처리하면 다른 점수가 저장될 수 있다.
그 두 점수 중 어떤 것이 채용 결정에 쓰이는가.
이 질문들이 해결되지 않은 채 프로덕션 스케일 사용을 염두에 둔 설계가 존재한다.

gs17은 더 근본적인 질문을 던진다.[^gs17]
이 ATS를 실제로 채용에 쓰는 회사가 있기는 한가.
없다면 이것은 "ChatGPT에게 점수를 매겨달라고 하는 것"과 무슨 차이가 있는가.
3,800개의 스타와 772개의 포크가 실제 채용 현장 배포를 의미하는지는 알 수 없다.
그러나 오픈소스로 공개된 이상, 누군가는 검증 없이 배포할 가능성이 있고,
그 가능성이 오픈소스 공개의 책임 문제를 더 날카롭게 만든다.

## 인사이트

### 채점 루브릭을 읽는 사람이 시스템을 이긴다

Jinja 템플릿이 공개됐다는 것은 이제 누구나 이 도구의 채점 기준을 읽을 수 있다는 의미다.
지원자 입장에서 이것은 시험 문제지가 사전에 공개된 것과 같다.
어떤 오픈소스 프로젝트가 높은 점수를 받는지, 어떤 기술 키워드가 유리한지,
GitHub 기여가 어떤 형태일 때 7개 선별에 포함되는지 알 수 있다.

이것은 Goodhart의 법칙이 채용 시스템에 적용되는 지점이다.
측정 기준이 알려지면, 그 기준 자체가 목표가 된다.
이력서 최적화 서비스나 AI 채용 코치가 이 레포지토리의 템플릿을 훈련 데이터로 쓰는 것은
기술적으로 완전히 가능하다.
오픈소스 채용 도구의 역설은 투명성이 높을수록 그 기준에 맞춘 가공된 이력서가 늘어난다는 것이다.

rsanek은 이 관점에서 한 걸음 더 나아가 실용적인 시각을 제시한다.[^rsanek]
콜드 지원(cold-apply)은 AI 이전에도 항상 블랙홀이었다는 것이다.
LLM 기반 ATS가 상황을 악화시켰을 수 있지만, 근본적으로 다르게 만들지는 않았다.
네트워크를 통한 추천, LinkedIn에서 채용 담당자 직접 연락, 서드파티 헤드헌터는 여전히 작동한다.
역설적으로 이 도구의 Jinja 템플릿을 읽고 이력서를 최적화하는 시간이,
같은 시간을 인적 네트워크에 투자하는 것보다 낮은 수익을 낼 수 있다.

### 백엔드 선택이 점수의 의미를 무력화한다

이 도구는 `gemma3:4b`부터 `gemini-2.5-pro`까지 다양한 모델을 지원한다.
그런데 모델이 다르면 동일한 이력서에 다른 점수가 나온다는 것은 명확하다.
Kinsky의 실험도 `gemma3:4b`와 Gemini Flash Lite에서 전혀 다른 분포를 보였다.

이것은 점수의 절대값이 의미가 없다는 뜻이다.
A사가 `gemma3:4b`로 평가한 90점과 B사가 `gemini-2.5-pro`로 평가한 90점은 같은 의미가 아니다.
심지어 같은 회사가 어느 시점에 모델을 바꾸면, 이전 지원자와 이후 지원자의 점수를 비교할 수 없다.
채용 결정이 점수에 기반한다면, 점수가 모델 선택에 의존한다는 사실은 그 결정의 정당성을 의심하게 만든다.

### 코드가 채용 정책이 되는 순간 책임의 소재가 달라진다

전통적인 채용에서 불공정한 결정의 책임은 채용 담당자나 임원에게 있다.
ATS 도구를 쓰더라도, 최종 결정을 내린 인간이 법적·도의적 책임을 진다.
그런데 채용 기준이 Jinja 템플릿에 있고, 그 템플릿이 오픈소스 레포지토리에 있으면,
책임의 소재가 복잡해진다.

이 템플릿을 그대로 쓴 기업이 차별 소송을 당한다면, HackerRank가 책임지는가, 기업이 책임지는가.
MIT 라이선스는 명시적으로 어떤 보증도 하지 않는다.
그러나 법원이 “우리는 그냥 오픈소스 도구를 썼을 뿐”이라는 항변을 받아들일지는 다른 문제다.
채용 알고리즘의 책임 귀속은 아직 법적으로 정착되지 않은 영역이며,
이 도구를 사용하는 기업은 자신이 어떤 법적 지형 위에 있는지 파악해야 한다.

0xbadcafebee는 이 상황이 더 깊은 문제의 증상임을 지적한다.[^0xbadcafebee]
기술 업계에는 형식적인 교육 요건도, 의무적인 훈련도, 도제 과정도, 건축 기준 같은 코드도, 전문직 단체도 없다.
이 상황에서 이력서는 원래부터 직무 성과의 좋은 예측 변수가 아니었다.
AI 기반 ATS는 처음부터 불완전한 신호를 처리하는 새로운 불완전한 방법일 뿐이다.
이 관점에서 보면 Hiring Agent는 기술 채용의 근본적 문제를 해결하는 것이 아니라,
해결하지 못한 채 자동화하는 것이다.

### 이 레포지토리는 Kinsky 실험 결과의 근본 원인을 설명한다

Dan Kinsky가 발견한 점수 변동성의 원인은 이 코드베이스를 보면 명확해진다.
`resume_evaluation_*.jinja` 파일들이 루브릭을 얼마나 상세하게 정의했는지가 곧 일관성을 결정한다.
기술 스택 항목이 일관된 이유는 체크리스트 형식의 Jinja 템플릿이 있기 때문이다.
프로젝트와 오픈소스 기여 항목이 변동성이 높은 이유는 LLM에게 주관적 판단을 위임하는 템플릿이기 때문이다.

오픈소스로 공개된 코드는 “왜 이렇게 작동하는가”를 설명하는 동시에,
“어떻게 고쳐야 하는가”의 출발점이 된다.
루브릭을 강화하면 일관성이 높아진다.
그러나 루브릭을 강화하는 순간 “무엇이 좋은 개발자인가”라는 가치 판단이 코드에 명시적으로 새겨진다.
그 판단이 옳은지 물을 책임은 코드를 쓴 HackerRank에게도, 코드를 쓰는 기여자들에게도 있다.
