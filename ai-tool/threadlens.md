# ThreadLens — 로컬 AI 세션 검색·백업·정리 도구

<https://github.com/hanityx/threadlens>

GeekNews: <https://news.hada.io/topic?id=29012>

## 소개

ThreadLens는 Codex, Claude, Gemini, Copilot 등 여러 AI 플랫폼에 흩어진 대화 이력을 하나의 인터페이스에서 검색·조회·백업·정리할 수 있는 오픈소스 로컬 퍼스트(Local-First) 도구다. 계정이나 클라우드 업로드 없이 완전히 로컬에서 동작한다.

개발자(hanityx)가 직접 밝힌 동기는 “여러 AI 플랫폼의 대화 이력이 흩어져 있고, 삭제 과정이 번거로워서” 만들었다는 것이다. GeekNews에 올라오자마자 minhoryang이 “최고에요! 지금 정말 필요하던 거였습니다!!”라고 첫 댓글을 남겼다.[^minhoryang] 개발자 hanityx는 “UI에서 한국어도 됩니다. i18n 기반 다국어입니다. 앞으로 더 열심히 깎아보겠습니다!”로 화답했다.[^hanityx]

## 주요 기능

- **통합 검색** — Codex, Claude, Gemini, Copilot 세션을 단일 키워드로 검색
- **전체 대화 열람** — 각 프로바이더 폴더를 직접 탐색하지 않고 트랜스크립트를 바로 열람
- **안전한 삭제** — 백업, 드라이 런(Dry-run), 삭제 전 확인 단계를 통해 의존성이 있는 세션을 실수로 제거하는 것을 방지
- **스레드 검사** — Codex 스레드의 범위, 연관 세션, 감사 이력 확인
- **프로바이더 상태 모니터링** — 세션 탐색 흐름 및 설정 진단
- **터미널 인터페이스(TUI)** — 키보드 중심의 TUI로 웹/데스크톱과 동일한 워크플로우 제공

## AI 세션 데이터의 구조

각 AI 플랫폼이 로컬에 세션 데이터를 저장하는 방식은 제각각이다. ThreadLens가 파싱해야 하는 데이터 구조를 이해하면, 이 도구의 기술적 도전이 명확해진다.

Claude Code는 `~/.claude/projects/` 하위에 프로젝트별 디렉토리를 만들고, 각 세션을 JSONL 파일로 저장한다. 각 줄은 하나의 메시지 이벤트를 담은 JSON 객체이며, 타임스탬프, 역할(human/assistant), 콘텐츠 블록 배열이 포함된다. 멀티모달 콘텐츠(이미지, 툴 결과 등)도 이 구조 안에 인라인으로 들어간다.

OpenAI Codex는 다르다. 클라우드 API 기반이기 때문에 로컬 세션 파일 자체가 제한적이다. 로컬에 캐시된 데이터와 API를 통해 가져오는 데이터를 구분해서 처리해야 한다. GitHub Copilot은 VS Code Extension의 로컬 스토리지에 세션 데이터를 저장하는데, 이는 Electron의 IndexedDB 또는 파일 시스템 기반 스토리지를 사용한다. Gemini는 Google 계정에 묶인 클라우드 세션이 기본이지만, 로컬 CLI 도구를 통한 세션은 별도의 형식으로 저장된다.

ThreadLens는 이 이기종 데이터 포맷들을 통합 스키마로 정규화해야 한다. 각 플랫폼의 세션 파일 위치, 파일 포맷(JSON, JSONL, SQLite, IndexedDB), 메시지 구조가 모두 다르기 때문에, 플랫폼별 파서(Parser)를 유지관리하는 것이 핵심 기술 부채가 된다. 플랫폼이 파일 저장 방식을 바꿀 때마다 ThreadLens의 파서도 업데이트해야 한다. 로드맵의 v0.3.x가 “프로바이더 신뢰성”에 집중하는 이유가 여기에 있다.

## 아키텍처

TypeScript 89.8%, CSS 7.9%, JavaScript 2.1%로 구성된 모노레포(pnpm workspaces)다. 웹, 터미널, 데스크톱 인터페이스가 공유 컨트랙트(Shared Contracts) 위에서 동작하며, 로컬 API를 통해 일관된 워크플로우를 제공한다.

### 로컬 퍼스트 아키텍처의 기술적 구현

ThreadLens의 로컬 퍼스트 설계는 단순히 “서버가 없다”는 의미가 아니다. 세 가지 인터페이스(웹 UI, TUI, 데스크톱)가 공유 컨트랙트(Shared Contracts) 위에서 동작한다는 점이 핵심이다.

공유 컨트랙트는 TypeScript 인터페이스로 정의된 데이터 모델과 API 계약이다. 웹 UI는 이 컨트랙트를 HTTP API로 접근하고, TUI는 동일한 컨트랙트를 직접 임포트해서 사용하고, Electron 데스크톱 앱은 IPC(Inter-Process Communication)로 접근한다. 렌더링 레이어만 다르고 비즈니스 로직은 단일 코드베이스에서 관리된다.

로컬 API 서버는 개발 환경에서 포트 8788에서 실행되며, 파일시스템 접근과 데이터 파싱을 담당한다. 웹 UI(포트 5174)는 이 로컬 API를 호출한다. 이 구조 덕분에 클라우드 서버 없이도 브라우저 기반 UI를 사용할 수 있다. Cloudflare Workers 개발 도구인 Wrangler를 활용한 것으로 보이며, 이를 통해 향후 엣지 배포 가능성도 열어둔 설계다.

TUI는 별도의 `pnpm dev:tui` 명령으로 실행되며, 서버 없이 파일시스템에 직접 접근하는 방식으로 더 빠른 응답성을 제공한다. Electron 데스크톱 앱은 세 가지 중 가장 완성된 배포 형태로, macOS(.dmg), Windows(.exe), Linux(.AppImage)로 패키징되어 있다.

**사용 가능한 플랫폼:**

- macOS (.dmg), Windows (.exe), Linux (.AppImage)
- 소스 코드 직접 실행: Node.js 22.12+, pnpm 10.33.2+

```bash
git clone https://github.com/hanityx/threadlens.git
cd threadlens
pnpm install && pnpm dev
```

개발 명령어는 `pnpm dev`(웹 UI), `pnpm dev:tui`(터미널), `pnpm dev:desktop`(Electron) 세 가지다.

### 멀티 프로바이더 통합의 기술적 도전

여러 AI 플랫폼을 단일 인터페이스에 통합하는 작업은 겉보기보다 훨씬 복잡하다. 각 플랫폼의 세션 데이터는 파일 시스템 위치, 포맷, 메타데이터 구조가 모두 다르다. 여기에 더해 플랫폼이 내부 저장 방식을 변경할 때마다 ThreadLens의 파서도 업데이트해야 한다.

두 번째 도전은 세션 간 관계(Dependency Graph)를 파악하는 것이다. Codex의 경우 하나의 스레드가 다른 스레드를 참조하거나, 연속된 컨텍스트를 공유하는 경우가 있다. ThreadLens의 “삭제 전 의존성 분석”은 이 관계 그래프를 탐색해서 “이 세션을 삭제하면 저 세션이 깨진다”는 사전 경고를 주는 기능이다. 이것은 단순한 파일 삭제 도구를 넘어서는 기능이며, 구현 난도도 상당히 높다.

세 번째 도전은 세션 데이터의 대용량 처리다. AI 코딩 도구를 매일 사용하는 개발자라면 수백, 수천 개의 세션이 쌓인다. 각 세션이 수만 토큰의 대화를 담고 있다면, 전체 세션 인덱싱과 전문 검색(Full-Text Search)은 상당한 처리 부하를 요구한다. 로컬에서 이를 효율적으로 처리하려면 SQLite 기반의 인덱싱, 증분 업데이트, 백그라운드 프로세싱 등의 기술이 필요하다. 현재 v0.3.x가 안정성에 집중하는 것은 이 기반 작업을 탄탄히 하는 단계임을 시사한다.

## 분석

ThreadLens가 해결하는 문제는 AI 코딩 도구 사용이 늘어나면서 자연스럽게 발생한 **AI 세션 관리 공백**이다. Codex, Claude, Gemini, Copilot 각각 자체 인터페이스를 제공하지만, 이들을 동시에 사용하는 개발자는 대화 이력을 통합적으로 볼 방법이 없었다. ThreadLens는 이 공백을 채우는 “AI 세션 레이어” 역할을 한다.

특히 “삭제 전 의존성 분석” 기능은 단순 파일 삭제 도구와 차별화된다. AI 세션들이 서로 참조하거나 연속된 컨텍스트를 공유하는 경우, 잘못된 삭제가 워크플로우를 망가뜨릴 수 있다. ThreadLens는 이 위험을 Dry-run 단계로 미리 확인하게 한다.

다국어 UI 지원(i18n 기반)은 영어권 외 개발자를 고려한 설계다. 한국어 지원 역시 포함돼 있어, GeekNews에서 즉각적인 반응이 나온 배경이기도 하다. hanityx가 개발자이자 한국어권 사용자로서 직접 경험한 불편함을 해결하기 위해 만든 도구라는 점이 그 공감을 만들어낸 것으로 보인다.

### AI 세션 관리가 새로운 카테고리인 이유

AI 코딩 도구가 보편화되기 전에도 “대화 관리” 도구는 존재했다. 이메일 클라이언트, 채팅 아카이버, 슬랙 검색 도구 등이 그것이다. ThreadLens가 이들과 다른 이유는 AI 세션의 성격이 근본적으로 다르기 때문이다.

이메일이나 채팅은 주로 커뮤니케이션 기록이다. 반면 AI 코딩 세션은 의사결정 과정의 기록이다. 어떤 버그를 어떤 접근법으로 분석했는지, 어떤 아키텍처 선택지를 검토했는지, 어떤 코드 패턴이 왜 채택되거나 버려졌는지가 담겨 있다. 이것은 코드베이스의 `git log`만큼이나 중요한 맥락 정보다.

두 번째 차이점은 멀티 프로바이더 문제의 심각성이다. 이메일은 수십 년간 SMTP/IMAP이라는 표준으로 통합됐다. AI 세션에는 아직 표준이 없다. 각 플랫폼이 독자적인 저장 형식을 사용한다. ThreadLens는 이 표준 부재의 공백을 로컬 통합 레이어로 메우는 것이다. 플랫폼이 표준화 API를 제공하기 시작하면 ThreadLens의 파서 의존성 문제는 해소되지만, 그 전까지 ThreadLens 같은 도구가 유일한 해결책이다.

## 비평

### 강점: 로컬 퍼스트 설계

클라우드 업로드 없이 로컬에서만 동작한다는 점은 기업 환경이나 보안에 민감한 프로젝트에서 AI 도구를 사용하는 개발자에게 중요한 신뢰 요소다. AI 대화 이력에는 코드, 설계 내용, 민감한 비즈니스 로직이 담겨 있을 수 있기 때문에 로컬 퍼스트 아키텍처는 선택이 아닌 필수에 가깝다. 세션 관리 도구 자체가 새로운 데이터 유출 경로가 되는 아이러니를 피하기 위한 올바른 선택이다.

### 약점: 초기 단계의 안정성

현재 로드맵(v0.3.x)이 “안정성과 프로바이더 신뢰성에 집중”이라고 밝힌 것은, 역으로 현재 버전이 아직 충분히 안정적이지 않을 수 있음을 시사한다. v0.4에서 추가 예정인 세션 탐색 및 영향 분석 기능이 핵심 가치임에도 아직 출시되지 않았다. “지금 정말 필요한 도구”라는 즉각적 공감이 있어도, 실제 도입과 일상 사용으로 이어지려면 이 안정성 단계를 빠르게 통과해야 한다.

### 약점: 프로바이더 의존성

지원하는 AI 플랫폼들이 로컬 세션 데이터를 저장하는 방식이나 경로가 변경되면, ThreadLens의 파싱 로직도 함께 업데이트해야 한다. 특히 Codex처럼 클라우드 기반 세션을 주로 사용하는 플랫폼은 로컬 접근 자체가 어려울 수 있다. 플랫폼들이 로컬 저장 방식을 변경할수록 ThreadLens의 유지관리 부담은 커지고, 오픈소스 단일 개발자 프로젝트로서 이 속도를 따라가기가 어려울 수 있다.

## 인사이트

### AI 세션이 “새로운 코드베이스”가 되고 있다

AI 코딩 도구를 매일 사용하는 개발자에게 AI 대화 이력은 단순한 로그가 아니다. 어떤 문제를 어떻게 해결했는지, 어떤 접근법이 실패했는지, 어떤 컨텍스트가 특정 코드 결정에 영향을 줬는지가 담긴 **지식 자산**이다. ThreadLens가 이 세션들을 검색·분류·백업할 수 있게 한다는 것은, AI 세션을 코드베이스처럼 관리할 수 있다는 가능성을 열어준다.

코드베이스는 `git log`로 누가 언제 왜 어떤 변경을 했는지 추적할 수 있다. AI 세션은 그 결정의 배경에 있는 탐색 과정을 담고 있다. “왜 이 함수가 이렇게 구현됐는가”는 `git blame`으로 코드 변경자를 찾을 수 있지만, “그 개발자가 어떤 AI 대화 끝에 이 구현을 선택했는가”는 AI 세션에만 남아 있다. 앞으로 AI 세션 관리 도구는 `git`만큼 필수적인 개발 인프라가 될 수 있다. ThreadLens는 그 인프라의 초기 형태다.

더 나아가, AI 세션 데이터가 충분히 쌓이면 이것은 개인의 문제 해결 패턴을 보여주는 데이터셋이 된다. 어떤 종류의 문제에서 어떤 AI를 사용했는지, 어떤 접근법이 자주 실패했는지, 어떤 컨텍스트 패턴이 좋은 해결책을 만들어냈는지. 이 메타데이터는 개발자 자신의 AI 사용 패턴을 개선하는 데 활용될 수 있다. ThreadLens의 장기 로드맵에 이런 분석 기능이 포함된다면, 단순한 세션 관리 도구를 넘어서는 역할을 할 것이다.

### 멀티 프로바이더 시대의 통합 레이어 필요성

개발자들이 Copilot, Claude, Gemini, Codex를 작업 유형에 따라 혼용하는 것이 보편화되면서, 각 플랫폼의 사일로(Silo)된 이력 관리는 점점 불편해진다. ThreadLens의 접근은 각 플랫폼 위에 독립적인 “세션 관리 레이어”를 만드는 것이다. 이는 과거에 여러 이메일 서비스를 하나의 클라이언트로 통합했던 패턴과 유사하다. AI 도구가 더 많아질수록 이 통합 레이어의 가치는 높아진다.

그러나 이 비유는 한계도 드러낸다. 이메일은 IMAP/SMTP라는 표준 프로토콜 덕분에 통합이 가능했다. AI 세션 데이터는 표준이 없다. ThreadLens가 단순히 각 플랫폼의 비공개 로컬 저장 형식을 역공학(Reverse Engineering)으로 파싱하는 방식에 머문다면, 플랫폼 업데이트마다 도구가 깨지는 불안정성에서 벗어나기 어렵다. 장기적으로 AI 세션 데이터의 표준화 필요성이 생긴다. ThreadLens가 그 표준화 논의를 주도하거나 참여할 수 있다면, 단순한 도구를 넘어 플랫폼으로 성장할 수 있다.

현실적으로 AI 플랫폼들이 자발적으로 세션 데이터 표준을 만들 동기는 약하다. 세션 이력은 플랫폼 고착(Lock-in)의 핵심 요소이기 때문이다. “내 모든 AI 대화 이력이 Claude에 있다”면 Claude를 떠나기 어렵다. ThreadLens 같은 도구가 이 고착을 완화하는 역할을 할 수 있다. 이것이 플랫폼들이 ThreadLens를 반길 이유가 없는 이유이기도 하다.

### 오픈소스 단일 개발자 프로젝트의 지속 가능성

ThreadLens는 개발자 개인의 불편함에서 시작된 프로젝트다. GeekNews 댓글에서 hanityx 본인이 “앞으로 더 열심히 깎아보겠습니다”라고 한 것처럼,[^hanityx] 현재는 단일 개발자가 열정으로 유지하는 초기 단계다. 이 단계의 오픈소스 프로젝트에는 공통적인 지속 가능성 도전이 있다.

첫째, 유지관리 부담이다. 4개 AI 플랫폼을 지원하고, 3개 인터페이스(웹, TUI, 데스크톱)를 유지하면서, 플랫폼 업데이트에 대응하고, 버그를 수정하는 것은 단일 개발자에게 상당한 부담이다. 커뮤니티 기여자가 늘어나지 않으면 로드맵이 지연되거나 지원 플랫폼이 축소될 수 있다.

둘째, 비즈니스 모델이다. MIT 라이선스로 완전히 무료 공개된 상태에서 지속 가능한 개발을 이어가려면 수익화 전략이 필요하다. GitHub Sponsors, 유료 호스팅 버전, 엔터프라이즈 라이선스 등의 경로를 조기에 탐색하는 것이 장기 지속 가능성을 위해 중요하다. “지금 정말 필요한 도구”라는 공감이 실제 기여나 후원으로 이어질 수 있는 구조를 만드는 것이 ThreadLens의 다음 과제다.
