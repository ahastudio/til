# OpenFlowKit

<https://openflowkit.com/>

<https://github.com/Vrun-design/openflowkit>

## 소개

OpenFlowKit은 MIT 라이선스로 공개된 로컬 퍼스트(local-first) AI 다이어그래밍 스튜디오다.
아키텍처 다이어그램과 플로우차트를 중심으로, 시각 캔버스 편집, 양방향 코드 동기화,
AI 생성, 자동 아이콘 할당, 애니메이션 내보내기를 하나의 도구에서 제공한다.

React 19 + TypeScript 5 + Vite 6 스택 위에서 React Flow(XYFlow)로 캔버스를 구성하고,
ELK.js로 자동 레이아웃을 처리하며, Zustand로 상태를 관리한다. 저장소는 IndexedDB를
사용하므로 서버가 전혀 없고, 협업은 WebRTC P2P로 처리한다.

주요 기능은 다음과 같다.

- **Mermaid → 아이콘 자동 할당**: 8가지 Mermaid 다이어그램 패밀리를 붙여넣으면
  노드 레이블을 의미론적으로 분류해 1,600개 이상의 기술 아이콘(개발자·AWS·Azure·GCP·CNCF)을
  자동으로 매핑한다.
- **Flowpilot AI 생성**: Google Gemini, OpenAI, Anthropic Claude, Groq 등 9개
  프로바이더를 지원하며 BYOK(Bring Your Own Key) 방식으로 서버를 경유하지 않고
  브라우저에서 직접 호출한다.
- **OpenFlow DSL**: Mermaid 호환 문법의 양방향 라이브 코드 패널로, 캔버스 수정이
  코드에 즉시 반영되고 코드 수정이 캔버스에 즉시 반영된다.
- **시네마틱 내보내기**: 노드와 엣지를 순서대로 그려가는 애니메이션을 WebM/MP4로
  브라우저 안에서 렌더링한다. 외부 업로드 없이 순수 로컬 처리다.
- **구조화된 다이어그램 패밀리**: 플로우차트, 아키텍처, ER, 클래스, 시퀀스, 마인드맵,
  사용자 여정, 상태 머신 8종을 지원한다.

## 분석

### 포지셔닝 전략

OpenFlowKit은 경쟁 도구들 각각의 결핍을 명시적으로 정리해 자신을 “타협 없는 선택”으로
포지셔닝한다. Excalidraw/tldraw는 DSL 없음, Mermaid는 시각 캔버스 없음, Draw.io는
오래된 UX, Lucidchart/Miro는 클라우드 종속, PlantUML은 서버 의존이라는 약점을 각각
지목한다. 이 전략은 기존 도구 사용자를 설득하는 데 효과적이지만, 모든 기능을 하나에서
제공하려는 시도가 각 영역에서의 깊이를 희생할 가능성도 내포한다.

### 로컬 퍼스트 설계 원칙

서버 저장소 없음, 계정 없음, AI 키 서버 미전달, WebRTC P2P 협업이라는 네 가지
원칙이 일관성 있게 설계 전체를 관통한다. 이는 단순한 기능 선택이 아니라 프라이버시와
자율성을 핵심 가치로 선언하는 아키텍처 철학이다. 순수 정적 SPA로 `dist/` 폴더만으로
자체 호스팅이 가능하다는 점은 이 철학의 실용적 귀결이다.

### AI 통합 방식

9개 프로바이더를 지원하되 키를 서버에 보내지 않는 구조는, AI 기능을 제공하면서도
벤더 종속성과 프라이버시 위험을 동시에 회피하는 설계다. BYOK 모델은 사용자에게
인증 부담을 주는 대신 도구 공급자에 대한 신뢰 의존도를 낮춘다. Anthropic Claude
`claude-sonnet-4-6`을 기본 모델 중 하나로 포함한 점도 눈에 띈다.

### 아이콘 자동 할당 파이프라인

Mermaid를 붙여넣으면 의미론적 분류기 → 아이콘 매처(정확 일치·별칭·부분 문자열 순)
→ 인리처 → ELK 레이아웃 순서로 처리된다. 이 파이프라인은 기술 스택 다이어그램 작성
시간을 크게 단축하는 실용적 기능이지만, 아이콘 매핑의 정확도는 레이블 표준화 수준에
크게 의존한다.

## 비평

### 강점

“어디서도 없는” 차별화 기능을 실제로 복수 보유하고 있다는 점이 강점이다. 시네마틱
애니메이션 내보내기는 오픈소스 다이어그래밍 도구 중 독보적으로 주장되며, 아이콘 자동
할당 파이프라인도 경쟁 도구에서는 찾기 어렵다. MIT 라이선스 + 서버 불필요 + 계정
불필요의 조합은 기업 내부 보안 정책이 엄격한 환경에서의 채택 장벽을 낮춘다.

### 약점

기능의 폭이 넓은 만큼 각 영역의 깊이가 아직 검증되지 않았다. 시퀀스 다이어그램,
클래스 다이어그램, ER 다이어그램 같은 형식 모델링에서 PlantUML이나 전문 UML 도구
수준의 정밀도를 제공하는지는 README만으로는 확인하기 어렵다. 실시간 협업(WebRTC P2P)과
AI 생성이 모두 Beta 상태이며, 로드맵에서 “GIF 내보내기”와 “GitHub 저장소 → 다이어그램”
기능이 아직 미완성임을 솔직히 인정하고 있다.

README의 경쟁 비교표는 자사에 유리한 항목만 선별했을 가능성이 높다. 예를 들어
Excalidraw의 협업 생태계나 draw.io의 방대한 스텐실 라이브러리는 언급되지 않는다.
비교 항목 선택 자체가 마케팅 전략의 일부다.

### 생태계 의존성

React Flow(XYFlow), ELK.js, Yjs, Framer Motion 등 외부 라이브러리에 대한 의존도가
높다. 이는 빠른 개발을 가능하게 하지만, 핵심 의존성의 라이선스 변경이나 유지보수
중단 시 대응 비용이 증가한다. React Flow는 과거 라이선스 정책을 변경한 전례가 있어
장기 프로젝트에서는 리스크 요인이 된다.

## 인사이트

### “로컬 퍼스트”는 더 이상 틈새 철학이 아니다

OpenFlowKit이 채택한 로컬 퍼스트 아키텍처는 SaaS 피로감(SaaS fatigue)의 직접적인
반응이다. 지난 10년간 소프트웨어 업계는 모든 것을 클라우드로 올리는 방향으로 달렸지만,
그 결과로 구독료 피로, 데이터 주권 상실, 인터넷 연결 의존, 서비스 종료 위험이 쌓였다.
Obsidian, Logseq, Local-First의 Ink & Switch 연구 등 여러 지점에서 같은 반동이
일어나고 있다.

흥미로운 것은 로컬 퍼스트가 기술적으로도 더 단순한 아키텍처를 만들어낸다는 점이다.
OpenFlowKit은 “백엔드 없음, 데이터베이스 없음, 환경 변수 없음”을 자랑으로 삼는다.
이는 운영 복잡도의 제거이면서, 동시에 기여자 진입 장벽의 제거이기도 하다. 오픈소스
생태계에서 자체 호스팅 가능한 순수 정적 앱은 포크, 실험, 커스터마이징이 가장 쉬운 형태다.
로컬 퍼스트는 단지 프라이버시 선택이 아니라, 협력적 오픈소스 개발에 최적화된 아키텍처
선택이기도 하다.

앞으로 AI 기능이 필수화될수록, “AI 키를 서버에 보내지 않는” 설계 원칙의 가치는 더욱
커질 것이다. 사용자는 자신의 Anthropic/OpenAI API 사용 패턴이 도구 공급사에 노출되길
원하지 않는다. BYOK + 브라우저 직접 요청 모델은 이 우려를 구조적으로 차단한다.

### 다이어그램 as Code의 미완성 혁명

Mermaid의 등장은 다이어그래밍을 개발자 워크플로에 통합하는 데 큰 기여를 했지만,
“코드로만” 다이어그램을 그리는 방식은 비개발자를 배제하고 복잡한 레이아웃 제어를
어렵게 만드는 한계를 남겼다. OpenFlowKit이 추구하는 “양방향 라이브 동기화”는 이
미완성을 완성하려는 시도다.

양방향 동기화는 구현이 생각보다 까다롭다. 시각 편집과 텍스트 편집을 동시에 일관된
상태로 유지하려면 두 표현 사이의 양방향 변환이 손실 없이 이루어져야 하고, 충돌 해결
전략도 필요하다. OpenFlowKit이 이를 얼마나 완성도 있게 구현했는지는 실제 사용 경험이
필요하지만, 이 방향 자체는 다이어그래밍 도구의 자연스러운 진화 방향이다.

더 넓게 보면, 다이어그램 as Code는 소프트웨어 아키텍처 문서화 문제의 핵심 해결책이다.
PowerPoint나 Visio에 그려진 다이어그램은 코드와 함께 버전 관리가 안 되고, 시스템이
바뀌어도 업데이트되지 않아 금방 거짓말이 된다. Mermaid를 Git 저장소 안에 두고,
PR 리뷰 때 다이어그램도 함께 리뷰하는 워크플로가 가능해질 때, 아키텍처 문서는 처음으로
“살아있는 문서”가 될 수 있다.

### AI 생성 다이어그램의 출력 품질 문제

OpenFlowKit이 로드맵에서 “GitHub 저장소 → 다이어그램” 기능을 “내부 베타이나 실제
코드베이스에서 출력 품질이 일관되지 않아 아직 출시 못 함”이라고 솔직하게 밝힌 부분은
흥미롭다. 이는 AI 다이어그램 생성의 핵심 난제를 드러낸다.

텍스트를 요약하거나 코드를 생성하는 AI는 출력의 품질이 다소 들쭉날쭉해도 사용자가
쉽게 보정할 수 있다. 그러나 다이어그램은 다르다. 노드 간 관계, 레이아웃, 계층 구조가
전체적으로 맞아야 유용하며, 부분적으로 틀린 다이어그램은 부분적으로 틀린 코드보다
오히려 더 위험할 수 있다. 틀린 아키텍처 다이어그램을 보고 시스템을 이해했다고 착각하는
상황이 생기기 때문이다.

이 품질 문제는 결국 “AI가 코드베이스의 의미론적 구조를 얼마나 깊이 이해하는가”의 문제다.
단순한 import 관계나 함수 호출 그래프는 정적 분석으로 추출할 수 있지만, 실제 아키텍처의
경계와 책임을 AI가 올바르게 추론하는 것은 훨씬 어려운 과제다. OpenFlowKit이 이 문제를
해결하는 방식이 이 도구의 장기적 성패를 결정하는 핵심 기술 과제가 될 것이다.

### 시네마틱 내보내기가 여는 새로운 커뮤니케이션 형식

“다이어그램을 동영상으로”라는 기능은 표면상 화려한 부가 기능처럼 보이지만, 실은 기술
커뮤니케이션의 미해결 문제를 건드린다. 정적 아키텍처 다이어그램은 복잡한 시스템의
전체 구조를 한 화면에 보여주려다 오히려 인지 과부하를 일으킨다. 발표나 온보딩에서
“이 다이어그램의 이 부분부터 설명하겠습니다”라고 하며 화살표로 가리키는 행위 자체가,
정보의 점진적 공개(progressive disclosure)가 필요하다는 신호다.

노드와 엣지를 순서대로 그려가는 애니메이션은 바로 이 점진적 공개를 자동화한다. 발표자가
직접 클릭하며 설명하지 않아도, 영상 자체가 “이 시스템은 이렇게 구성된다”는 내러티브를
가진다. 기술 문서에 첨부하거나, Confluence 페이지에 임베드하거나, Slack에 공유할 때
정적 이미지보다 훨씬 효과적으로 복잡한 아키텍처를 전달할 수 있다.

흥미로운 점은 이 기능이 “서버 없이, 브라우저 안에서” 가능하다는 것이다. 기존에 다이어그램
애니메이션을 만들려면 화면 녹화 소프트웨어나 전문 모션 그래픽 도구가 필요했다. 브라우저
네이티브 렌더링으로 이를 대체한다는 아이디어는 기술적으로도 흥미롭고, 접근성 측면에서도
의미 있는 진전이다. 이 방향이 성숙하면 “다이어그램 첨부”가 아니라 “아키텍처 영상 첨부”가
기술 문서의 새로운 표준이 될 수 있다.
