# md-to-naver-blog - 마크다운을 네이버 블로그 호환 HTML로 변환

원문: <https://github.com/jjlabsio/md-to-naver-blog>

## 요약

md-to-naver-blog는 마크다운 문서를 네이버 블로그 에디터에 바로 붙여넣기 가능한
HTML로 변환하는 라이브러리다.
네이버 블로그 에디터가 마크다운을 지원하지 않는 문제를 해결하기 위해 만들어졌다.

제목, 볼드/이탤릭, 코드 블록(구문 강조 포함), 테이블, 리스트, 인용문, 이미지 등
주요 마크다운 문법을 지원한다.
YAML 프론트매터에서 태그를 추출해 네이버 블로그 해시태그로 자동 변환하는 기능과
URL 도메인 치환 기능도 제공한다.

npm 패키지(`@jjlabsio/md-to-naver-blog`), CLI 도구, 웹 앱(`https://mtnb.dev`)
세 가지 인터페이스를 제공하며, Node.js와 브라우저 환경 모두에서 동작한다.
TypeScript로 작성되었고, Playwright/Puppeteer 통합을 위한
클립보드 유틸리티 함수도 포함한다.

작업 흐름은 마크다운을 스크립트로 변환한 뒤 CLI 프리뷰 웹 앱에서 확인하고,
완성된 HTML을 네이버 블로그에 복사·붙여넣기하는 방식이다.

## 분석

### 네이버 블로그 에디터의 구조적 한계를 우회하는 접근

네이버 블로그는 자체 WYSIWYG 에디터만 제공하며 마크다운 입력을 지원하지 않는다.
이는 마크다운으로 콘텐츠를 관리하는 개발자·기술 블로거에게 진입 장벽이 된다.
md-to-naver-blog는 에디터 자체를 바꾸지 않고 변환 레이어를 두어 이 문제를 해결한다.

### 멀티 플랫폼 퍼블리싱 파이프라인

하나의 마크다운 원본으로 자체 블로그, 네이버 블로그 등 여러 플랫폼에
동시 발행하는 워크플로우를 가능하게 한다.
YAML 프론트매터 파싱과 URL 치환 기능이 이 멀티 플랫폼 전략의 핵심 요소다.

### 플랫폼 정책을 존중하는 설계 철학

GeekNews 댓글에서 자동 포스팅 도구(viruagent-cli)가 제안되었으나,
제작자는 "플랫폼 가이드라인을 존중하면서 콘텐츠 제작을 최적화한다"는 원칙을 밝혔다.
자동 포스팅이 아닌 변환·프리뷰·수동 붙여넣기라는 의도적 설계 선택이다.

## 비평

### 좁지만 명확한 문제 해결

네이버 블로그 마크다운 변환이라는 매우 구체적인 문제를 정확히 해결한다.
범용 마크다운-HTML 변환기와 달리 네이버 블로그 에디터의 호환성에 특화되어 있어
실용적 가치가 높다.

### 지속가능성의 불확실성

네이버 블로그 에디터의 HTML 파싱 방식에 의존하므로,
네이버가 에디터를 업데이트하면 변환 결과가 깨질 수 있다.
3 stars, 2 forks라는 소규모 커뮤니티에서
이런 변화에 빠르게 대응할 수 있을지는 미지수다.

### 완전 자동화와의 경계

수동 붙여넣기를 고수하는 것은 플랫폼 정책 준수라는 점에서 바람직하지만,
대량 콘텐츠 운영 시에는 병목이 될 수 있다.
Playwright/Puppeteer 클립보드 유틸리티를 제공하면서도
자동 포스팅은 하지 않는 선은 다소 모호하다.

## 인사이트

### 폐쇄적 플랫폼 시대의 "변환 레이어" 패턴

네이버 블로그처럼 표준 포맷을 지원하지 않는 플랫폼이 존재하는 한,
변환 레이어는 필연적으로 등장한다.
이는 RSS가 쇠퇴하고 각 플랫폼이 독자적 포맷을 고집하면서 생긴 구조적 문제다.

과거 웹은 HTML이라는 공통 언어 위에서 콘텐츠가 자유롭게 유통되었다.
그러나 플랫폼들이 사용자를 가두기 위해 독자 에디터와 포맷을 채택하면서,
콘텐츠 생산자는 플랫폼마다 별도의 변환 도구를 만들어야 하는 상황이 되었다.
md-to-naver-blog는 이 파편화의 증상이자 현실적 해법이다.

마크다운이 개발자 커뮤니티의 사실상 표준이 된 지 오래지만,
주요 블로그 플랫폼들의 마크다운 지원은 여전히 부족하다.
이 간극이 좁혀지지 않는 한 변환 도구의 수요는 계속될 것이다.

### "콘텐츠 원본 소유권"이라는 개발자 블로거의 핵심 가치

md-to-naver-blog의 워크플로우는 마크다운 원본을 Git 저장소에 보관하고,
네이버 블로그는 배포 채널 중 하나로 취급한다.
이는 콘텐츠의 원본 소유권을 플랫폼이 아닌 작성자가 갖겠다는 선언이다.

전통적으로 블로그 플랫폼에 직접 작성하면 콘텐츠는 해당 플랫폼에 종속된다.
플랫폼이 서비스를 종료하거나 정책을 변경하면 콘텐츠를 잃을 수 있다.
마크다운 원본을 로컬에 보관하는 방식은 이 위험을 원천적으로 제거한다.

이 패턴은 Jamstack, 정적 사이트 생성기, Git 기반 CMS의 연장선에 있다.
"소스 코드처럼 콘텐츠를 관리한다"는 개발자 문화가
블로그 퍼블리싱 파이프라인에까지 확장된 사례다.

### 자동화의 윤리적 경계선

제작자가 자동 포스팅 제안을 거절하고 수동 붙여넣기를 고수한 것은
기술적 한계가 아닌 의도적 선택이다.
"할 수 있다"와 "해야 한다"의 구분이 점점 흐려지는 자동화 시대에
이 경계선은 주목할 만하다.

플랫폼 자동 포스팅은 스팸과 SEO 어뷰징의 온상이 되어왔다.
네이버가 자동 포스팅을 금지하는 데는 이런 역사적 맥락이 있다.
도구 제작자가 기술적으로 가능한 자동화를 스스로 제한하는 것은
플랫폼 생태계의 건강성을 유지하는 데 기여한다.

GeekNews 댓글에서 드러난 "개인 기록용"과 "프로페셔널 브랜딩용"의 차이는
같은 기술이라도 맥락에 따라 윤리적 판단이 달라짐을 보여준다.
자동화 도구를 만들 때 "누가, 어떤 목적으로 쓸 것인가"를 고려하는 것이
단순한 기능 구현보다 중요한 설계 결정이 될 수 있다.
