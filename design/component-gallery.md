# The Component Gallery

<https://component.gallery/>

디자인 시스템의 UI 컴포넌트를 한데 모아 비교할 수 있는
레퍼런스 사이트. 프론트엔드 개발자 Iain Bean이 2019년부터
운영하고 있다. Smashing Magazine 메인에 소개되었고,
CSS-Tricks 선정 2021년 17번째 "Hottest Frontend Tool"이다.

## 핵심 가치

"이름 짓기는 어렵다(Naming things is hard)."
어떤 컴포넌트를 뭐라고 불러야 할지 모르겠을 때,
다른 사람들이 비슷한 것을 뭐라고 부르는지 보는 게
가장 좋은 출발점이다.

디자인 시스템은 프로젝트에 참여하는 모든 팀이 컴포넌트에
대한 공통 언어(a common language)를 갖도록 돕는다.
The Component Gallery는 그 공통 언어를 찾는 도구다.

## 규모

- 약 63개 컴포넌트 카테고리
- 92개 이상의 디자인 시스템 수록
- 2,500개 이상의 개별 컴포넌트 예시

Fluent UI, Atlassian Design System, GOV.UK 등
실제 운용 중인 디자인 시스템에서 가져온 사례들이다.

## 수록 컴포넌트 예시

| 컴포넌트    | 설명                                  |
|-------------|---------------------------------------|
| Accordion   | 접고 펼 수 있는 수직 헤딩 스택        |
| Alert       | 중요한 변경을 사용자에게 알림         |
| Avatar      | 사용자의 시각적 표현 (사진, 이니셜)   |
| Badge       | 상태나 속성을 나타내는 작은 라벨      |
| Breadcrumb  | 탐색 계층에서 현재 위치를 보여줌      |
| Button      | 폼 제출 등 액션을 트리거              |
| Card        | 단일 엔티티를 표현하는 콘텐츠 컨테이너 |
| Carousel    | 여러 슬라이드를 순환하며 보여줌       |
| Combobox    | Select + 자유 텍스트 입력             |
| Modal       | 다른 콘텐츠 위에 나타나는 인터페이스  |
| Toast       | 푸시 알림처럼 레이어 위에 뜨는 알림   |

| Tree View   | 디렉터리 구조 같은 중첩 계층 표시     |

## 이름의 스펙트럼

각 컴포넌트 페이지에는 대안 이름(alternative names)이
함께 표시된다. 이것이 이 사이트의 가장 큰 가치다.

Accordion 하나만 봐도:

> Arrow toggle, Collapse, Collapsible sections,
> Collapsible, Details, Disclosure, Expandable,
> Expander, ShowyHideyThing

9개의 다른 이름이 존재한다. 누군가는 Disclosure라
부르고, 누군가는 Collapsible이라 부르고, 누군가는
ShowyHideyThing이라 부른다. 같은 것을 두고.

Button처럼 이름이 완전히 표준화된 것도 있지만,
대부분의 컴포넌트는 이런 이름의 스펙트럼 위에 있다.

## 영감: A Pattern Language

Christopher Alexander의 1977년 저서
*A Pattern Language*에서 영감을 받았다.
이 책은 건축과 도시 설계의 253가지 "패턴"을 기술한다.
"양지바른 곳(Sunny Place)", "두 면에서 들어오는
빛(Light on Two Sides of Every Room)" 같은 식으로,
특정 맥락에 맞는 형태를 제안한다.

1987년 Ward Cunningham과 Kent Beck이 이 책에서
영감을 받아 소프트웨어 디자인 패턴 개념을 만들어냈다.
Gang of Four의 디자인 패턴 책, Wiki의 발명,
오늘날의 디자인 시스템까지 이 흐름 위에 있다.

웹에서도 이런 패턴이 존재하며, 흔히 "컴포넌트"라 부른다.
컴포넌트 라이브러리와 달리 바로 갖다 쓰는 게 아니라,
자신만의 해결책을 설계하도록 영감을 주는 것이 목적이다.

## 인사이트

### 1. 이름이 곧 소통 비용이다

디자이너가 "Drawer"라 부르는 걸 개발자가
"Side Panel"이라 부르면, 매 대화마다 번역이 필요하다.
이 비용은 눈에 보이지 않지만 누적되면 거대하다.

Figma에서 쓰는 이름과 Storybook에서 쓰는 이름이
일치해야 한다. 디자인 시스템 도입의 첫 단계는
컴포넌트에 대한 공통 이름을 합의하는 것이다.
The Component Gallery는 업계 전체의 작명 현황을
조망함으로써 그 합의의 근거를 제공한다.

실천 방법: 팀에서 컴포넌트 네이밍 워크숍을 열어라.
기존에 가진 모든 컴포넌트를 감사(audit)하고,
이 사이트를 근거 자료로 활용하라.

### 2. 인기 ≠ 정답

사이트 자체가 경고한다:
"의견이나 기법이 인기 있다고 해서 올바른 것은 아니다.
정보 폭포(information cascade)를 경계하고,
가능하면 실제 리서치에 기반한 결정을 내려라."

92개 시스템 중 80개가 같은 패턴을 쓴다고 해서
그게 우리 맥락에 맞는 건 아니다.
좋은 디자인 시스템은 결정의 이유를 리서치에서 찾는다.
GOV.UK 디자인 시스템이 존경받는 이유가 여기에 있다.
그들은 모든 컴포넌트 결정에 사용자 리서치 근거를 붙인다.

### 3. 패턴은 해결책이 아니라 영감이다

*A Pattern Language*의 패턴은 설계도가 아니다.
"두 면에서 빛이 들어오는 방은 좋다"는 원칙이지,
"창문을 여기 놓아라"는 지시가 아니다.

마찬가지로 The Component Gallery의 컴포넌트는
복사해서 붙여넣을 코드가 아니다.
92개 시스템이 Accordion을 어떻게 만들었는지
살펴본 뒤, 우리 맥락에 맞는 Accordion을 직접
설계하라는 것이다.

shadcn/ui 같은 컴포넌트 라이브러리는 "가져다 쓰는 것"
이고, 이 사이트는 "보고 배우는 것"이다.
용도가 다르다.

### 4. 바퀴를 다시 발명하지 말 것

인터페이스 문제 중 상당수는 이미 해결되어 있다.
Sparkbox의 연구에 따르면, IBM Carbon 디자인 시스템을
활용해 폼을 만든 경우 처음부터 만든 것 대비
개발 시간이 크게 단축되었고, 일관성과 접근성도
더 높았다.

Forrester 리서치에 따르면 디자인 시스템에 투자한
기업은 671%의 ROI를 보였다.
개발자 생산성 30% 향상, 디자이너 생산성 42% 향상,
출시 주기 20~50% 단축.

새 프로젝트를 시작할 때마다 처음부터 만들 필요 없이,
기존 디자인 시스템의 검증된 패턴을 참고하자.

### 5. 디자인 시스템은 웹사이트가 아니다

성공적인 디자인 시스템은 컴포넌트 몇 개를 보여주는
웹사이트가 아니다. Iain Bean도 "디자인 시스템은
코드 이상의 것"이라고 말한다.

색상 팔레트, 타이포그래피, 프로세스, 접근성, 도구,
톤 앤 보이스를 아우르는 공유된 실천의 집합이며,
끊임없이 진화한다. Salesforce의 Lightning Design
System은 이 접근으로 생산성 60% 향상, CSS 70%
감소를 달성했다.

The Component Gallery에 수록된 92개 시스템도
컴포넌트 카탈로그는 빙산의 일각일 뿐이다.

### 6. 57개의 구멍에 담기지 않는 1,000개의 조각

사이트에는 기존 57개 카테고리에 깔끔하게 들어맞지
않는 약 1,000개의 컴포넌트 예시가 미분류 상태로
있다. 이것이 시사하는 바가 크다.

현실의 UI는 교과서적 분류를 거부한다.
Accordion인 동시에 Navigation인 컴포넌트,
Modal이면서 Drawer인 컴포넌트가 존재한다.
분류 체계는 이해를 돕지만, 현실은 항상 분류를
넘어선다.

### 7. 건축에서 소프트웨어로, 다시 디자인으로

Alexander → Beck & Cunningham → GoF 디자인 패턴
→ 컴포넌트 라이브러리 → 디자인 시스템.

건축가의 "패턴 언어"가 소프트웨어 엔지니어의
"디자인 패턴"이 되었고, 이제 다시 디자인 영역으로
돌아와 "디자인 시스템"이 되었다.
이 순환이 흥미롭다.

Alexander가 말한 "이름 없는 속성(the quality that
has no name)" — 전체가 살아있는 느낌 — 은
좋은 디자인 시스템에서도 느껴진다.
개별 컴포넌트가 아니라 그것들이 조화롭게 결합된
전체가 만드는 경험이 핵심이다.
