# Claude Code is Re-shaping My Design Process

Jeff Zych가 자신의 디자인 워크플로가 Claude Code로
어떻게 변했는지 기록한 글의 요약과 분석이다.

<https://jlzych.com/2026/02/20/claude-code-is-re-shaping-my-design-process/>

![Git commits 추이](https://jlzych.com/images/2026-cc-process/git-commits.png?1771613024)

커밋 수의 급증이 Claude Code로 디자인하고
출하한 결과다. Figma 사용은 크게 줄었다.

## 요약

### 핵심 전제

Claude Code에게 의도한 사용자 경험 — 유스케이스,
기대 동작, 기능의 진화 방향 — 을 설명하면
구현 방법을 알아서 찾는다.
Zych는 이것이 엔지니어와 일하던 방식과 본질적으로
같다고 말한다.

> "This is basically how I've always worked
> with engineers. Only I'm in control of the
> engineer and they'll do whatever I say."

프롬프팅이 자연스러운 이유도 여기에 있다.
커리어 내내 의도를 설명하고 맥락을 제공해 온
디자이너의 기존 역량이 그대로 전이된다.

### 작은 기능은 Figma를 건너뛴다

검색 필드 추가, QoL 개선처럼 기존 컴포넌트와 패턴을
재활용하는 작업은 Figma 없이 Claude Code로 직행한다.
와이어프레임도, 핸드오프도 없다.
적절한 프롬프팅만 있으면 Claude가 올바른 컴포넌트와
패턴을 골라 쓴다.

Opus 4.6에 대해서는 "코드베이스를 이해하고
명시하지 않아도 올바르게 행동하는 능력이
미친 듯이 뛰어나다(crazy good)"고 평가한다.

### 큰 기능은 단계별로 도구를 나눈다

솔루션 공간이 넓은 기능은 여전히 사전 탐색을 한다.
다만 Claude Chat과 Claude Code를 구분해서 쓴다.

1. **문제 정의** — 문제, 유스케이스, 좋은 솔루션의
   "형태(shape)"를 글로 정리한다.
   예: "빠르고 마찰 없는 느낌", "사용자에게
   추가 작업을 시키지 않는 것".
2. **아이디어 확장** — Claude Chat(CC가 아닌
   일반 대화)에 아이디어를 더 요청한다.
   생각 못 한 흥미로운 방향이 나오기도 하고,
   맥락에 안 맞는 것도 섞인다.
3. **감 잡기** — 마음에 드는 방향을 Claude의
   인터랙티브 아티팩트로 만들어 직접 느끼고,
   팀과 공유해 의견을 모은다.
4. **Lo-fi 보완** — 레이아웃이나 단순한 아이디어는
   Figma에서 상자 몇 개 옮기는 것이 프롬프트를
   쓰고 기다리는 것보다 빠를 때가 있다.
5. **코드에서 완성** — 팀 논의와 사용자 피드백으로
   방향이 확정되면 Claude Code에서 직접 빌드하고
   다듬는다. 모든 상태를 맞추고, 인터랙션을 튜닝한다.

### Mid-fi 목업 단계가 사라졌다

Mid-fi를 완전히 건너뛰고 바로 Claude Code로 간다.
좋은 프롬프팅이면 몇 분 만에 상당히 괜찮은 버전이
동작한다.
실제 데이터가 들어간 동작 버전 위에서 편집하면
정렬, 필터링, 데이터가 너무 길거나 짧을 때 등
정적 목업에서 잡지 못하는 UX 개선점이 빠르게 드러난다.

### Figma가 여전히 이기는 영역

색상 조합, 타이포그래피 옵션, 새로운 컴포넌트 같은
비주얼 탐색은 Figma가 압도적이다.
나란히 비교하고, 확대/축소하고, 요소를 자유롭게
옮기는 작업은 코드에서 하기 고통스럽다.
다만 기존 디자인 시스템이 있는 제품에서 일하는 비중이
크기 때문에 이 영역의 비중은 줄어들고 있다.

### 변하지 않은 것

사용자 중심 사고, 문제 이해, 다양한 옵션 탐색,
반복, 테스트, 다듬기.
이 본질은 그대로다.
달라진 것은 이 과정의 중간 구간이 Claude Code
안에서 이루어지고, 코드가 사용자 손에 닿는 것까지
본인이 직접 가져갈 수 있다는 점이다.

> "No handoffs. No wasteful artifacts.
> Just working in the final medium
> and firing off a PR."

## 분석

### 디자인 프로세스의 구조적 변화

전통적 더블 다이아몬드 모델에서 두 번째 다이아몬드
(솔루션 구체화)의 매체가 바뀌었다.
목업 도구 대신 코드가 표현 수단이 된 것이다.
이 변화의 전제 조건은 두 가지다.

1. AI 코딩 도구가 디자이너의 의도를 코드로
   변환할 수 있을 만큼 성숙했다.
2. 디자이너가 "무엇을 만들어야 하는지"를 글로
   명확히 표현할 수 있다.

두 번째 조건이 핵심이다.
Zych가 "엔지니어와 일하던 방식 그대로"라고 말한 것은
프롬프팅이 새로운 기술이 아니라
디자이너의 기존 역량이 형태를 바꾼 것임을 뜻한다.

### Claude Chat과 Claude Code의 역할 분리

Zych의 워크플로에서 두 도구는 명확히 분리된다.

| 단계         | 도구        | 목적               |
| ------------ | ----------- | ------------------ |
| 아이디어     | Claude Chat | 발산, 대안 탐색    |
| 프로토타입   | Artifacts   | 감 잡기, 팀 공유   |
| Lo-fi 스케치 | Figma       | 빠른 레이아웃 탐색 |
| 구현·다듬기  | Claude Code | 최종 매체에서 작업 |

Chat은 발산, Code는 수렴.
이 구분이 흥미로운 이유는 같은 모델이라도
인터페이스(대화 vs 코드 에디터)에 따라
디자이너가 기대하는 출력이 달라지기 때문이다.

### 핸드오프 제거가 만드는 효과

핸드오프가 사라지면 세 가지가 동시에 일어난다.

1. **맥락 손실 제거** — 의도를 가진 사람이
   직접 최종 매체에서 작업한다.
2. **피드백 루프 단축** — 목업 → 스펙 → 구현 →
   리뷰 사이클이 프롬프트 → 결과 → 수정으로 줄어든다.
3. **낭비 산출물 감소** — 구현 후 버려지는 목업,
   레드라인, 스펙 문서가 사라진다.

이것은 [Direct Design](./direct-design.md)의
핵심 논지와 정확히 일치한다.

### Lo-fi는 남고 Mid-fi가 사라진 이유

Lo-fi 스케치는 사고 도구다.
머릿속 아이디어를 외부로 꺼내 빠르게 비교하는 용도로,
"때로는 상자 몇 개를 직접 옮기는 게 더 빠르다."
반면 Mid-fi 목업은 커뮤니케이션 도구다.
"이렇게 만들어 달라"는 메시지를
개발자에게 전달하기 위해 존재했다.
AI가 구현까지 수행하면 이 전달 단계가 불필요해진다.

## 인사이트

### 1. 디자이너의 가치가 판단력으로 이동한다

실행(그리기, 스펙 작성, 핸드오프)이 자동화되면
남는 것은 판단력이다.
무엇을 만들지, 왜 만드는지, 어떤 방향이 더 나은지
결정하는 능력이 디자이너의 핵심 가치가 된다.

Roger Wong도 같은 결론에 도달했다.
Claude Code로 자사 디자인 시스템 컴포넌트를 조합해
화면을 생성한 뒤 이렇게 말했다.

> "수년간 해 온 일의 전 과정을
> 건너뛰었다는 것을 깨달았다.
> 디자인 도구에서 그림을 그리고
> 다른 사람에게 넘기는 것."

— [Roger Wong, "Product Design Is Changing"](https://rogerwong.me/2026/02/product-design-is-changing/)

### 2. "엔지니어와 일하던 방식 그대로"의 의미

Zych의 비유에서 Claude Code는 새로운 도구가 아니라
완벽하게 순종하는 엔지니어다.
이것은 단순한 농담이 아니다.

디자이너-엔지니어 협업의 마찰은
의도 전달의 손실과 해석 차이에서 발생한다.
Claude Code는 이 마찰을 제거하지만,
전제가 있다 — 의도를 명확히 전달하는 능력.
잘못된 지시를 거부하거나 더 나은 대안을 제안하는
인간 엔지니어와 달리, AI는 시킨 대로 한다.
따라서 디자이너의 판단 품질이
결과물 품질을 직접 결정한다.

### 3. 도구가 아니라 매체가 바뀌었다

Figma에서 Claude Code로 도구가 바뀐 것이 아니다.
디자인의 표현 매체가 목업에서 동작하는 코드로 바뀌었다.
매체가 바뀌면 사고 방식이 바뀐다.
정적 화면 단위로 생각하던 것이
상태, 인터랙션, 실제 데이터 단위로 바뀐다.

이것이 "정렬, 필터링, 데이터가 너무 길거나 짧을 때
같은 것들이 빠르게 드러난다"는 Zych의 관찰을
설명한다.
매체의 충실도(fidelity)가 높을수록
발견의 해상도도 높아진다.

### 4. 디자인 시스템이 AI 시대의 인프라가 된다

Zych가 "기존 컴포넌트와 패턴을 사용하는 작업에서
Claude가 잘 동작한다"고 말한 것은
디자인 시스템이 잘 갖춰져 있기 때문이다.
Opus 4.6이 "명시하지 않아도 올바르게 행동"하는 것도
코드베이스에 일관된 패턴이 존재하기 때문이다.

AI가 생성하는 코드의 품질은
디자인 시스템의 정리 수준에 비례한다.
컴포넌트가 명확하고, 패턴이 문서화되어 있고,
사용 예제가 풍부할수록 AI의 출력이 일관된다.
디자인 시스템은 더 이상 "일관성 도구"가 아니라
AI가 올바른 코드를 생성하기 위한
인프라(infrastructure)다.

### 5. Taste가 컴파일러를 얻었다

[Direct Design](./direct-design.md) 문서의 인용구
"Taste now has a compiler"가 이 글의 핵심을
한 문장으로 요약한다.
취향과 판단력이 있는 사람이
실행 도구(AI)를 통해 직접 결과물을 만들 수 있게 되었다.

실행 비용이 극적으로 낮아진 세계에서
맛(taste)을 가진 사람의 영향력은 커지고,
실행만 하는 사람의 자리는 좁아진다.
