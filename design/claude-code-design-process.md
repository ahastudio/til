# Claude Code가 디자인 프로세스를 바꾸고 있다

Jeff Zych가 자신의 디자인 워크플로가 Claude Code로
어떻게 변했는지 기록한 글의 요약과 분석이다.

<https://jlzych.com/2026/02/20/claude-code-is-re-shaping-my-design-process/>

## 요약

### 작은 기능은 Figma를 건너뛴다

검색 필드 추가, QoL 개선처럼 기존 컴포넌트와 패턴을
재활용하는 작업은 Figma 없이 Claude Code로 직행한다.
와이어프레임도, 핸드오프도 없다.
적절한 프롬프팅만 있으면 Claude가 올바른 컴포넌트와
패턴을 골라 쓴다.

### 큰 기능은 디스커버리 후 Claude로 간다

솔루션 공간이 넓은 기능은 여전히 사전 탐색을 한다.
다만 방식이 달라졌다.
문제, 유스케이스, 좋은 솔루션의 "형태(shape)"를
글로 정리한 뒤 Claude에게 아이디어를 더 요청한다.
마음에 드는 방향이 나오면 인터랙티브 아티팩트로
만들어 팀과 공유한다.

### Mid-fi 목업 단계가 사라졌다

이전에는 Lo-fi → Mid-fi → Hi-fi → 구현 순서였다.
지금은 Lo-fi 스케치 후 바로 Claude Code에서
동작하는 코드로 이동한다.
실제 데이터가 들어간 동작 버전 위에서 편집하면
정적 목업이나 클릭스루 프로토타입에서 잡지 못하는
UX 개선점이 빠르게 드러난다.

### Figma가 여전히 이기는 영역

색상 조합, 타이포그래피 옵션, 새로운 컴포넌트 같은
비주얼 탐색은 Figma가 압도적이다.
나란히 비교하고, 확대/축소하고, 요소를 자유롭게
옮기는 작업은 코드에서 하기 고통스럽다.

### 변하지 않은 것

사용자 중심 사고, 문제 이해, 다양한 옵션 탐색,
반복, 테스트, 다듬기.
이 본질은 그대로다.
달라진 것은 이 과정의 중간 구간이 Claude Code
안에서 이루어지고, 결과물이 바로 코드가 된다는 점이다.

> "No handoffs. No wasteful artifacts.
> Just working in the final medium
> and firing off a PR."

## 분석

### 디자인 프로세스의 구조적 변화

전통적 더블 다이아몬드 모델에서 두 번째 다이아몬드
(솔루션 구체화)의 매체가 바뀌었다.
목업 도구 대신 코드가 표현 수단이 된 것이다.
이 변화의 전제 조건은 두 가지다.

1. AI 코딩 도구가 디자이너의 의도를 코드로
   변환할 수 있을 만큼 성숙했다.
2. 디자이너가 "무엇을 만들어야 하는지"를 글로
   명확히 표현할 수 있다.

두 번째 조건이 핵심이다.
Zych 본인도 "커리어 내내 의도를 설명하고 맥락을
제공해 온 경험 덕에 프롬프팅이 자연스럽다"고 말한다.
즉, 프롬프팅 능력은 새로운 기술이 아니라
디자이너의 기존 역량이 형태를 바꾼 것이다.

### 핸드오프 제거가 만드는 효과

핸드오프가 사라지면 세 가지가 동시에 일어난다.

1. **맥락 손실 제거** — 의도를 가진 사람이
   직접 최종 매체에서 작업한다.
2. **피드백 루프 단축** — 목업 → 스펙 → 구현 →
   리뷰 사이클이 프롬프트 → 결과 → 수정으로 줄어든다.
3. **낭비 산출물 감소** — 구현 후 버려지는 목업,
   레드라인, 스펙 문서가 사라진다.

이것은 [Direct Design](./direct-design.md)의
핵심 논지와 정확히 일치한다.

### Lo-fi는 남고 Mid-fi가 사라진 이유

Lo-fi 스케치는 사고 도구다.
머릿속 아이디어를 외부로 꺼내 빠르게 비교하는 용도로
프롬프트를 쓰고 기다리는 것보다
상자 몇 개를 직접 그리는 것이 빠를 때가 있다.
반면 Mid-fi 목업은 커뮤니케이션 도구다.
"이렇게 만들어 달라"는 메시지를
개발자에게 전달하기 위해 존재했다.
AI가 구현까지 수행하면 이 전달 단계가 불필요해진다.

## 인사이트

### 1. 디자이너의 가치가 판단력으로 이동한다

실행(그리기, 스펙 작성, 핸드오프)이 자동화되면
남는 것은 판단력이다.
무엇을 만들지, 왜 만드는지, 어떤 방향이 더 나은지
결정하는 능력이 디자이너의 핵심 가치가 된다.

Roger Wong도 같은 결론에 도달했다.
Claude Code로 자사 디자인 시스템 컴포넌트를 조합해
화면을 생성한 뒤 이렇게 말했다.

> "수년간 해 온 일의 전 과정을
> 건너뛰었다는 것을 깨달았다.
> 디자인 도구에서 그림을 그리고
> 다른 사람에게 넘기는 것."

— [Roger Wong, "Product Design Is Changing"](https://rogerwong.me/2026/02/product-design-is-changing/)

### 2. 오케스트레이션이 새로운 핵심 역량이다

Wong의 표현을 빌리면 병목은 모델이 아니라
키보드 앞의 사람이다.
무엇을 요청할지, 어떻게 작업을 모델이 처리할 수 있는
단위로 분해할지, 언제 결과를 거부할지 아는 것이
오케스트레이터의 역량이다.

이것은 Claude Code Best Practice에서 말하는
"프롬프트 엔지니어링보다 운영 엔지니어링"과
같은 맥락이다.
좋은 문장 하나가 아니라
좋은 컨텍스트, 좋은 제약, 좋은 검증 루프가
결과를 만든다.

### 3. 도구가 아니라 매체가 바뀌었다

Figma에서 Claude Code로 도구가 바뀐 것이 아니다.
디자인의 표현 매체가 목업에서 동작하는 코드로 바뀌었다.
매체가 바뀌면 사고 방식이 바뀐다.
정적 화면 단위로 생각하던 것이
상태, 인터랙션, 실제 데이터 단위로 바뀐다.

이것이 "동작하는 버전 위에서 편집하면
UX 개선점이 빠르게 드러난다"는 Zych의 관찰을
설명한다.
매체의 충실도(fidelity)가 높을수록
발견의 해상도도 높아진다.

### 4. 디자인 시스템이 AI 시대의 인프라가 된다

Zych가 "기존 컴포넌트와 패턴을 사용하는 작업에서
Claude가 잘 동작한다"고 말한 것은
디자인 시스템이 잘 갖춰져 있기 때문이다.
Wong 역시 자사 디자인 시스템 저장소를 Claude Code에
연결해 화면을 생성했다.

AI가 생성하는 코드의 품질은
디자인 시스템의 정리 수준에 비례한다.
컴포넌트가 명확하고, 패턴이 문서화되어 있고,
사용 예제가 풍부할수록 AI의 출력이 일관된다.
디자인 시스템은 더 이상 "일관성 도구"가 아니라
AI가 올바른 코드를 생성하기 위한
인프라(infrastructure)다.

### 5. Taste가 컴파일러를 얻었다

[Direct Design](./direct-design.md) 문서의 인용구
"Taste now has a compiler"가 이 글의 핵심을
한 문장으로 요약한다.
취향과 판단력이 있는 사람이
실행 도구(AI)를 통해 직접 결과물을 만들 수 있게 되었다.

실행 비용이 극적으로 낮아진 세계에서
맛(taste)을 가진 사람의 영향력은 커지고,
실행만 하는 사람의 자리는 좁아진다.
