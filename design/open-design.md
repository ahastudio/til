# Open Design

<https://github.com/nexu-io/open-design>

HN 토론: <https://news.ycombinator.com/item?id=47985750> (119점, 71개 댓글)

## 소개

Open Design(OD)은 Anthropic의 Claude Design에 대한 오픈소스 대안이다.
2026년 4월 17일 공개된 Claude Design이 바이럴을 일으키고 나서 불과 11일 만에
등장한 프로젝트로, 출시 1주일 만에 13,000개가 넘는 스타를 획득했다.
Claude Design이 클로즈드 소스, 유료, 클라우드 전용, Anthropic 모델 고정이었던 것에 반해,
Open Design은 로컬 퍼스트(local-first), 셀프 호스팅 가능, BYOK(Bring Your Own Key)를 모든
계층에서 지원하는 방식으로 설계됐다.

핵심 기치는 “에이전트를 직접 제공하지 않는다”는 것이다.
Claude Code, Codex, Cursor Agent, Gemini CLI, OpenCode, Qwen Code, GitHub Copilot CLI,
Hermes, Kimi, Pi, Kiro CLI 등 11개 코딩 에이전트 CLI를 시스템 `PATH`에서 자동 감지해
디자인 엔진으로 활용한다.
CLI가 없는 경우 OpenAI 호환 BYOK 프록시로 같은 파이프라인을 사용할 수 있다.

기능 범위는 넓다. 31개의 컴포저블 스킬(Skill)과 72개 이상의 브랜드 수준 디자인 시스템을
내장하며, 웹·데스크톱·모바일 프로토타입, 슬라이드 덱, 이미지, 비디오, HyperFrame을
생성할 수 있다. 산출물은 샌드박스 iframe에서 미리보기 가능하고, HTML·PDF·PPTX·MP4로
내보낼 수 있다. 저장소로는 로컬 SQLite(`better-sqlite3`)를 사용하고 대화, 파일, 탭,
템플릿 상태를 영속한다.

설계 철학은 여섯 가지 핵심 아이디어로 압축된다.
(1) 에이전트는 네 것을 써라,
(2) 스킬은 플러그인이 아니라 파일이다,
(3) 디자인 시스템은 테마 JSON이 아닌 포터블 마크다운이다,
(4) 인터랙티브 질문 폼이 리다이렉트의 80%를 막는다,
(5) 데몬이 에이전트에게 실제 로컬 파일시스템을 준다,
(6) 프롬프트 스택이 곧 제품이다.

## 분석

### 아키텍처: 데몬 + 프론트엔드 분리 구조

아키텍처는 Next.js 16 App Router 프론트엔드와 Node 24 + Express 로컬 데몬의
두 층으로 나뉜다. 데몬은 유일하게 권한을 가진 프로세스로, 코딩 에이전트 CLI를
`child_process.spawn`으로 실행하되 작업 디렉터리를 `.od/projects/<id>/`로 설정해
에이전트가 실제 파일시스템을 읽고 쓸 수 있게 한다.
CLI별로 타입 이벤트 파서가 따로 구현돼 있어
(`claude-stream-json`, `acp-json-rpc`, `pi-rpc`, `plain` 등)
에이전트마다 서로 다른 통신 프로토콜을 흡수한다.

BYOK 프록시(`POST /api/proxy/stream`)는 OpenAI 호환 엔드포인트로 SSE를 통과시키며,
루프백·링크 로컬·RFC1918 주소는 데몬 단에서 차단한다(SSRF 방어).
Electron 데스크톱 셸은 선택적 레이어로 제공되며, 사이드카 IPC
(`STATUS · EVAL · SCREENSHOT · CONSOLE · CLICK · SHUTDOWN`)를 통해 E2E 테스트도 지원한다.

### 스킬 시스템: 파일이 곧 플러그인

스킬은 `SKILL.md` + `assets/` + `references/` 폴더 하나로 구성되며,
Claude Code의 `SKILL.md` 컨벤션을 그대로 따른다.
`od:` 프론트매터에 `mode`, `platform`, `scenario`, `preview.type`,
`design_system.requires` 등을 기술하면 데몬이 파싱해 피커에 노출한다.
폴더 하나를 `skills/`에 드롭하고 데몬을 재시작하면 끝이다.

31개 내장 스킬은 `prototype`(27개)과 `deck`(4개) 두 모드로 나뉘고,
`scenario` 필드(`design · marketing · operation · engineering · product · finance · hr · sale · personal`)로
그루핑된다.
`guizang-ppt` 스킬은 [`op7418/guizang-ppt-skill`](https://github.com/op7418/guizang-ppt-skill)을
원본 라이선스 그대로 번들해 제공한다는 점이 주목할 만하다.
오픈소스 저작물을 단순 참조가 아니라 저작권을 보존한 채 포함시킨 방식은
커뮤니티 신뢰를 구축하는 적절한 선택이다.

### 디자인 시스템: 마크다운 토큰

72개 이상의 디자인 시스템은 JSON 테마가 아니라 색상·타이포그래피·간격·레이아웃·컴포넌트·모션·보이스·브랜드·안티패턴
9개 섹션으로 구성된 `DESIGN.md` 마크다운 스키마로 정의된다.
[`VoltAgent/awesome-design-md`](https://github.com/VoltAgent/awesome-design-md)에서
파생된 이 방식은 에이전트가 파일 시스템 도구(`Read`, `grep`)로 직접 토큰을 읽을 수 있도록 한다.
Linear, Stripe, Vercel, Anthropic, Apple, Figma 등 시장 주요 브랜드 시스템이 포함되어 있다.

### 프롬프트 스택: 레이어드 컨텍스트

전송 프롬프트 구성이 독창적이다.
단순 `system + user`가 아니라 DISCOVERY 지시문, 아이덴티티 차터,
활성 `DESIGN.md`, 활성 `SKILL.md`, 프로젝트 메타데이터, 스킬 사이드파일이
모두 합성된다.
특히 첫 번째 턴에서 반드시 `<question-form id=“discovery”>`을 먼저 보내
표면·타깃·톤·브랜드 컨텍스트·규모·제약을 30초 안에 확정하는 방식은
[`huashu-design`](https://github.com/alchaincyf/huashu-design)의
“주니어 디자이너 모드”를 공식화한 것이다.
모델이 즉흥으로 픽셀을 그리기 전에 방향을 잠그는 이 패턴은 리다이렉트 비용을 극적으로 낮춘다.

## 비평

### 강점: 조립성과 개방성의 조화

가장 강력한 설계 결정은 에이전트 에코시스템을 추상화 레이어 안에 가두는 대신
있는 그대로 통과시키는 것이다.
11개 CLI 어댑터를 각각 구현한 것은 분명 많은 공수지만,
사용자가 이미 선호하는 에이전트와 투자한 컨텍스트를 그대로 활용할 수 있게 해준다.
이는 Figma가 서드파티 플러그인 생태계를 전면 개방했을 때 얻었던
“우리 도구가 생태계 중심이 된다”는 효과와 유사하다.

마크다운 기반 디자인 시스템은 LLM 친화적이다.
에이전트가 JSON 스키마를 파싱하는 대신 `grep hex`로 색상 토큰을 읽고,
`Read DESIGN.md`로 전체 브랜드 가이드를 소화할 수 있다는 것은
프롬프트 컨텍스트 활용 측면에서 명확한 이점이다.

그러나 README 자체의 글쓰기 방식이 논란이 됐다.
ricardobeat은 “Claude-세일즈맨 스타일”이라고 정확히 짚었다[^ricardobeat].
실제로 무엇을 하는지 보여주기보다 가치 명제를 과장하는 마케팅 언어가
README 전반에 퍼져 있다는 것이다.
MSaiRam10은 “세일즈 덱 같다”며 “도구가 정말 좋다면 무엇을 하는지만 보여줘라”고 했다[^MSaiRam10].
1주일에 14,000개 스타도 유기적 발견보다는 홍보 활동의 결과로 읽힌다는 의심도 덧붙였다.
AI 도구들이 AI로 작성된 마케팅 언어로 가득 차면서, 독자는 README만 읽어도
그 프로젝트가 실제로 만들어진 것인지, AI가 쓴 홍보문인지 점점 구별하기 어려워진다는
메타적 아이러니가 생긴다.

### 약점: 유지보수 부채와 스킬 품질 편차

11개 CLI 어댑터 유지는 상당한 부채다.
각 에이전트 CLI가 업데이트될 때마다 파서도 따라가야 하는데,
소수 컨트리뷰터 구조에서 이 속도를 따라가는 것은 쉽지 않다.
특히 ACP(Agent Client Protocol) 기반 어댑터(Hermes, Kimi, Kiro)는
해당 프로토콜 자체가 아직 초기 단계라 인터페이스 변동 위험이 크다.

31개 스킬의 품질 균일성도 의문이다.
`guizang-ppt`처럼 잘 검증된 스킬과 새로 추가된 스킬 간 품질 편차가
누적될수록 “31개 스킬”이라는 숫자가 과대 대표될 수 있다.
스킬 등록이 “폴더 하나 드롭”으로 너무 쉬운 것은 생태계 확장성에 유리하지만,
품질 문호가 낮아지는 이중성을 가진다.

hmokiguess는 이 복잡성 문제를 다른 각도에서 제기했다[^hmokiguess].
도구 안에 좋은 것이 많더라도 진입 곡선이 너무 가파르면, 결국 원작자만 완전히 이해하는
“괴물 같은 워크플로우”가 만들어진다는 것이다.
Napster에 비유했는데, 좋은 콘텐츠가 있어도 익히기까지의 마찰이 대중화를 막는다.
31개 스킬, 72개 디자인 시스템, 11개 CLI 어댑터는 숫자로 보면 풍성하지만,
처음 들어온 사용자에게는 어디서부터 시작해야 할지 모르는 공간이 될 수 있다.

Saline9515는 더 직접적인 비교를 들었다[^Saline9515].
Claude Design 방식 자체가 토큰을 낭비하고 전체 웹사이트를 빌드해야 하는 시간이 걸리며,
결과물도 스타일 일관성이 부족하다는 것이다.
ChatGPT Image 2는 UI 프로토타이핑에서 더 저렴하고 빠르다는 게 그의 평가다.
Open Design이 Claude Design의 아키텍처 방식을 그대로 따른다면,
이 효율 문제도 함께 상속하는 셈이다.

### 미해결 질문: 에이전트 컨텍스트 한계

각 에이전트가 실제 파일시스템을 읽고 쓴다는 것은 장점이지만,
프롬프트 스택이 커질수록 컨텍스트 창 한계 문제가 발생한다.
`DISCOVERY 지시문 + DESIGN.md + SKILL.md + 프로젝트 메타데이터 + 사이드파일`의
합산 토큰이 몇 천을 넘으면, 정작 사용자 브리프와 히스토리가 압축되는 트레이드오프가 생긴다.
이 문제를 어떻게 핸들링하는지는 README에서 명확히 언급되지 않는다.

## 인사이트

### 클로드 디자인의 역설: 오픈소스를 자극한 클로즈드 소스

Anthropic이 Claude Design을 클로즈드 소스, 클라우드 전용으로 출시한 것은
단기적으로는 수익 모델을 보호하지만 장기적으로 역설적 결과를 낳는다.
출시 11일 만에 오픈소스 대안이 등장하고 1주일 만에 13,000개 이상의 스타를 받은 것은
“충분한 수요 + 충분한 기술 장벽 부재 = 즉각적 오픈소스화”라는 패턴을 다시 확인해준다.
역사적으로 Stable Diffusion이 DALL-E 2 출시 직후 등장했고,
Llama가 GPT-4 대비 접근성 압력에 응답했던 것처럼,
유명 클로즈드 AI 제품은 오픈소스 대안의 존재 이유가 된다.

문제는 이 역학이 Anthropic의 전략적 선택을 제약한다는 점이다.
Claude Design이 API 위에서만 동작하는 클라우드 서비스라면,
로컬 에이전트를 이미 보유한 사용자 집단은 오픈소스 대안을 선택할 유인이 강하다.
Anthropic 입장에서 Claude Design의 차별화는 “Anthropic 모델의 성능”이 되어야 하는데,
Open Design이 Claude Code를 에이전트 중 하나로 지원하므로
모델 성능 차별화조차 제거되는 구조다.
클로즈드 소스 전략이 방어해야 했던 바로 그 시장을 스스로 열어준 셈이다.

### “에이전트를 직접 제공하지 않는다”는 선택의 전략적 함의

Open Design의 가장 영리한 선택은 에이전트를 직접 제공하지 않는 것이다.
이는 단순히 개발 비용 절감이 아니라 포지셔닝 전략이다.
자체 에이전트를 번들하면 Claude Code, Codex, Gemini CLI 등 기존 에이전트를 이미
사용하는 개발자들이 “왜 다른 에이전트로 갈아타야 하지?”라는 질문에 맞닥뜨린다.
하지만 “네 에이전트를 그대로 써라”는 메시지는 기존 사용자 습관을 온전히 포용한다.

이 전략은 멀티클라우드 시대의 인프라 툴링이 걸어온 길과 닮아 있다.
Terraform이 AWS SDK 대신 클라우드-어그노스틱 레이어를 제공해 생태계 중심이 된 것처럼,
Open Design은 “에이전트-어그노스틱 디자인 오케스트레이터”로 자리를 잡는다.
에이전트 생태계가 더 파편화될수록 이 포지셔닝의 가치는 높아진다.
11개 CLI 어댑터 유지 부채는 바로 이 포지셔닝의 비용이다.

2차 효과로, 이 선택은 Open Design이 특정 모델 공급자의 에코시스템 정치에 종속되지 않게 한다.
Anthropic이 Claude Code API를 변경하더라도 11개 중 하나일 뿐이고,
OpenAI가 Codex를 deprecate하더라도 대안이 10개 남는다.
의존성 분산은 프로젝트의 생존력을 높인다.

### 프롬프트 스택이 제품이라는 명제의 역설

“프롬프트 스택이 제품이다”라는 Open Design의 여섯 번째 아이디어는
솔직하면서도 위험한 주장이다.
모든 레이어가 파일로 공개되고 편집 가능하다는 것은 투명성의 극대화지만,
동시에 진입 장벽이 거의 없다는 고백이기도 하다.
경쟁자가 `apps/web/src/prompts/system.ts`와 `discovery.ts`를 읽고
더 나은 버전을 만드는 데는 몇 시간이면 충분하다.

jshaqaw는 더 큰 질문을 던졌다[^jshaqaw].
디자인 도구의 가치는 근본적으로 신호 기능(signaling)에서 온다 — 이것이
얼마나 잘 만들어졌는지, 얼마나 노력이 들어갔는지를 보여주는 것이다.
1990년대 데스크톱 퍼블리싱이 처음 나왔을 때도 인상적이었지만,
곧 누구나 쓸 수 있는 도구가 되면서 그 신호 가치가 사라졌다.
AI 디자인도 같은 궤적을 따를 것이다 — “무료, 즉각적, 무한 생성 가능”이 되면
디자인 산출물은 배경 소음이 된다.
이 예측이 맞다면 Open Design이 성공적으로 확산될수록 그 가치를 스스로 희석시키는
역설에 빠질 수 있다.

하지만 이 역설에는 반전이 있다.
오픈소스 생태계에서 코드 자체가 모트(moat)가 된 적은 거의 없다.
리눅스 커널은 공개돼 있지만 Red Hat/Ubuntu의 가치는 생태계와 지원에서 나온다.
리액트는 누구나 포크할 수 있지만 메타의 커밋 속도와 커뮤니티 관성이 대안을 제압한다.
Open Design의 실제 모트는 31개 스킬 카탈로그, 72개 디자인 시스템 라이브러리,
커뮤니티 기여 속도, 그리고 사용자들이 쌓아가는 `SKILL.md` 생태계다.
“프롬프트 스택이 제품이다”라는 명제의 진짜 의미는
“우리가 공개한 프롬프트 스택을 베이스라인으로 커뮤니티가 더 나은 스킬을 만들어라”는 초대장이다.
이것이 성공한다면 오픈소스 모델의 전형적 승리 패턴이 된다.
