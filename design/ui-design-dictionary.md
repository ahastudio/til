# UI Design Dictionary — 108가지 UI 패턴 사전

<https://ui-design-dictionary.pages.dev/>

## 요약

UI Design Dictionary는 108가지 UI 패턴을 카테고리별로 정리한 시각적 참조 사전이다.
Navigation, Layout, Forms & Input, Data Display, Feedback, Content, Actions, Mobile, Social & Communication, Onboarding & Guidance, Media, Commerce, Advanced Patterns, Authentication, Error & System의 15개 카테고리로 구성된다.
각 패턴은 영문 명칭, 일본어 명칭, 한 줄 설명, 적용 맥락 태그, 그리고 실제 작동하는 미니 데모로 구성된다.

사전은 일본의 디자이너 KAWAI가 제작했으며, 일본어와 영어 두 언어를 지원한다.
주요 대상은 UI/UX를 학습 중인 디자이너와 프론트엔드 개발자로, 패턴의 명칭과 개념을 시각적으로 연결하는 데 초점을 맞춘다.
검색 기능과 카테고리 필터를 제공해 원하는 패턴을 빠르게 찾을 수 있다.

수록된 패턴의 범위는 Hamburger Menu, Tab Bar 같은 기초 컴포넌트부터 Virtual Scroll, Micro-interactions, Command Palette 같은 고급 패턴까지 포괄한다.
각 패턴은 Web/Mobile/Desktop, 사용 맥락(Admin, EC, SNS 등) 태그를 달아 어떤 상황에 적합한지 빠르게 파악할 수 있게 한다.
Bento Grid, Sticky Sidebar처럼 최근 디자인 트렌드를 반영한 항목도 포함되어 있다.

## 분석

### 패턴 언어로서의 UI 사전

크리스토퍼 알렉산더의 패턴 언어(Pattern Language) 개념은 건축에서 소프트웨어로 이어졌다.
Gang of Four의 디자인 패턴이 코드 수준의 반복 문제를 명명하고 공유한 것처럼, UI Design Dictionary는 인터페이스 수준의 반복 해법을 명명하고 공유한다.
이 사전의 본질적 기여는 패턴 자체의 발명이 아니라 **이미 존재하는 관행에 공유 가능한 이름을 부여하는 것**이다.
Hamburger Menu를 몰라도 그것을 쓸 수 있지만, 이름을 알면 팀 안에서 “햄버거 메뉴는 모바일에서만”처럼 논의할 수 있다.

### 카테고리 구조가 드러내는 UI의 관심사

15개 카테고리는 현대 UI가 다루는 핵심 관심사를 그대로 보여준다.
Navigation(14), Forms & Input(17), Data Display(13)가 가장 많은 패턴을 포함한다는 사실은 사용자 이동, 데이터 입력, 데이터 표현이 UI의 핵심 과제임을 방증한다.
반면 Onboarding & Guidance(3), Media(3), Commerce(3)는 도메인 특화 패턴으로 범용성이 낮아 더 적은 수가 수록되었다.
이 분포 자체가 “어떤 종류의 문제가 UI 설계에서 가장 자주 등장하는가”를 답하는 데이터가 된다.

### 적용 맥락 태그의 실용성

각 패턴에 붙은 태그(Mobile, Web, Desktop, iOS, Android, EC, SNS, Admin 등)는 단순 분류를 넘어 **설계 결정의 컨텍스트**를 명시한다.
예컨대 Infinite Scroll에는 Mobile/SNS/Feed 태그가, Pagination에는 Web/List 태그가 붙는다.
이것은 “두 패턴 중 어느 것을 선택할 것인가”라는 질문의 답이 기술적 구현 난이도가 아니라 플랫폼과 사용 맥락에 달려 있음을 시각적으로 보여준다.

## 비평

### 설명 깊이의 한계

각 패턴 설명이 한 줄 정의에 그친다는 것이 이 사전의 구조적 한계다.
Kanban Board를 “카드를 카라무 간에 드래그 이동하는 태스크 관리 UI”로 설명하는 것은 패턴을 식별하기에는 충분하지만, 패턴을 **적용**하기에는 부족하다.
언제 Kanban을 선택하고 언제 Table을 선택하는지, 각 패턴의 장단점과 안티패턴은 무엇인지에 대한 안내가 없다.
참조 사전으로서의 역할에는 충실하지만, 학습 자료로서의 역할에는 미흡하다.

### 접근성과 반응형 관점의 부재

패턴의 접근성(Accessibility) 고려사항이 전혀 언급되지 않는다.
Hamburger Menu는 스크린 리더 사용자에게 ARIA 레이블 없이 접근 불가한 패턴이 될 수 있다.
Date Picker는 키보드 전용 사용자에게 까다로운 컴포넌트다.
Modal/Dialog는 포커스 트랩(focus trap) 없이는 접근성 위반이 된다.
시각적 미니 데모에만 집중하고 접근성 구현 요건을 다루지 않는 것은 실무 적용 시 간과되기 쉬운 공백을 남긴다.

### 패턴 선택 가이드의 부재

유사한 패턴들(Toast vs. Alert, Modal vs. Drawer vs. Bottom Sheet, Tabs vs. Segmented Control) 사이의 선택 기준이 없다.
이 패턴들은 맥락에 따라 교체 가능해 보이지만 실제로는 명확한 선택 원칙이 존재한다.
Toast는 비파괴적 일시 알림에, Alert는 지속적 경고에, Modal은 필수 확인이 필요한 작업에 쓰인다.
이 구분 없이 시각적 데모만 제공하면 학습자가 외양만 보고 패턴을 오용할 위험이 있다.

## 인사이트

### 공유 어휘가 팀의 설계 역량을 결정한다

프로젝트 초기에 팀이 공유하는 UI 어휘의 크기가 설계 논의의 효율을 결정한다.
“로그인 화면 만들어주세요”와 “OTP Input, Password Strength Meter, Social Login 버튼이 포함된 Login Form 만들어주세요”는 동일한 목표를 가리키지만, 후자는 모호성 제거에 드는 왕복 커뮤니케이션 비용을 절약한다.
디자이너와 개발자가 같은 패턴 이름을 쓸 때 Figma 컴포넌트, 코드 컴포넌트, 디자인 토큰이 동일한 개념을 가리키게 되고, 이것이 Design System의 근간이 된다.
UI Design Dictionary가 제공하는 공유 어휘는 개인 역량의 문제가 아니라 팀 협업 인프라의 문제다.

공유 어휘의 부재는 조직의 성장 단계에서 두드러지게 드러난다.
초기에는 소수가 전체 UI를 결정하므로 어휘 공유 없이도 일관성이 유지된다.
그러나 팀이 커지고 여러 피처 팀이 독립적으로 UI를 만들기 시작하면, 같은 기능을 서로 다른 패턴으로 구현하는 파편화가 일어난다.
어떤 팀은 Toast를 쓰고 어떤 팀은 Alert를 쓰며, 어떤 팀은 Modal을 쓰는데 다른 팀은 Drawer를 쓴다.
이 파편화를 교정하는 것이 Design System 도입의 첫 번째 과제인데, 그 전에 공유 언어가 먼저 있어야 한다.

AI 코딩 도구의 확산은 이 어휘의 중요성을 다른 차원으로 높인다.
“버튼 아래에 뜨는 추가 선택지”를 프롬프트에 쓰는 것보다 “Split Button 컴포넌트”를 쓰는 것이 더 정확한 결과를 만들어낸다.
패턴의 이름을 아는 것이 도구를 더 효과적으로 다루는 능력이 된다.
UI Design Dictionary는 AI 네이티브 워크플로우에서 프롬프트 어휘집이기도 하다.

### 패턴은 맥락을 포함한다

Software Design Pattern이 “의도, 동기, 적용 가능성, 구조, 협력자, 결과, 구현”을 명시하듯, UI 패턴도 단순한 시각적 솔루션이 아니라 **특정 사용자 문제와 맥락의 해법**이다.
Action Sheet가 존재하는 이유는 iOS에서 Context Menu의 hover 인터랙션이 불가능하기 때문이다.
Infinite Scroll이 SNS Feed에 적합한 이유는 목적지 없는 브라우징 행동 패턴과 맞기 때문이다.
Breadcrumb이 필요한 이유는 깊은 계층 구조에서 사용자가 현재 위치를 잃기 때문이다.
패턴을 외우는 것이 아니라 패턴이 해결하는 문제를 이해할 때 패턴의 선택과 변형이 가능해진다.

이 맥락 이해는 패턴이 안티패턴으로 전환되는 지점을 알아보는 능력이기도 하다.
Infinite Scroll은 콘텐츠에 목적지가 있을 때(법률 문서, 전자상거래 카탈로그)는 사용자가 “어디까지 왔는지” 파악하기 어려워 Pagination이 더 적합하다.
Hamburger Menu는 모바일에서는 표준이지만 데스크톱 B2B 관리 도구에서는 탐색성을 저해한다.
같은 패턴이 어떤 맥락에서는 최선이고 다른 맥락에서는 안티패턴이 된다는 사실은, 패턴 사전의 최종 목표가 암기가 아니라 판단력 형성에 있음을 보여준다.

108가지가 완전한 목록이 아니라는 점도 중요하다.
UI 패턴은 플랫폼 진화, 사용자 행동 변화, 새로운 인터랙션 패러다임의 등장에 따라 계속 생겨나고 소멸한다.
Voice UI, Gesture, AR/VR 인터페이스는 이 사전의 범위 밖에 있다.
Chat UI가 패턴으로 수록된 것처럼, AI 대화형 인터페이스나 에이전트 UI도 곧 독자적인 패턴 언어를 필요로 하게 될 것이다.
사전은 완성형이 아니라 살아있는 문서여야 한다.

### 시각적 데모가 문서보다 강하다

인터랙티브 데모는 정적 스크린샷이나 텍스트 설명보다 패턴을 이해시키는 데 압도적으로 효과적이다.
Skeleton Screen을 텍스트로 설명하면 “로딩 중 레이아웃 골격 표시”인데, 직접 보면 즉시 이해된다.
Pull to Refresh는 제스처 인터랙션이 핵심인데, 텍스트만으로는 그 경험을 전달할 수 없다.
UI Design Dictionary가 제공하는 것은 정의가 아니라 **경험 가능한 레퍼런스**다.

이것은 기술 문서 설계 전반에 시사점을 준다.
코드 예제가 설명보다 강하듯, UI 패턴은 데모가 설명보다 강하다.
Storybook, Chromatic, CodePen 같은 도구들이 인기를 얻은 배경도 같은 원리다.
“보여주는 것이 말하는 것보다 낫다”는 원칙은 UI 학습 맥락에서 특히 날카롭게 적용된다.

반대로, 데모에만 의존하는 학습의 위험도 있다.
시각적으로 익숙해졌다고 해서 패턴의 적용 원칙을 이해한 것은 아니다.
Walkthrough가 어떻게 생겼는지는 데모로 알 수 있지만, 언제 Walkthrough가 역효과를 내는지(사용자가 스킵하고 싶은데 못 하게 할 때, 기능을 모두 익히기 전에 배우게 할 때)는 경험과 고찰이 필요하다.
최선의 학습은 데모가 촉발하는 질문에 텍스트가 답하는 구조다.
이 사전은 그 전반부를 잘 제공하지만, 후반부는 학습자 스스로 채워야 한다.
