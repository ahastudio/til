# DESIGN.md — Google Labs의 AI 에이전트용 디자인 시스템 명세 형식

<https://github.com/google-labs-code/design.md>

## 명세

DESIGN.md는 Google Labs가 제안한 파일 형식으로, AI 코딩 에이전트에게 시각적 아이덴티티를
일관되게 전달하기 위한 구조화된 명세다.
YAML 프론트매터(front matter)에 기계 판독 가능한 디자인 토큰을 담고,
마크다운 본문에 인간이 읽을 수 있는 디자인 근거를 작성하는 두 레이어 구조를 취한다.
토큰이 정확한 값을 제공하고, 산문이 “왜 이 값인가”와 “어떻게 적용하는가”를 설명한다.

형식 명세는 색상(Color), 치수(Dimension), 토큰 참조(Token Reference), 타이포그래피
(Typography) 등 네 가지 타입을 정의한다.
컴포넌트 토큰은 `{colors.primary}`처럼 중괄호 참조 문법으로 다른 토큰을 가리킬 수 있다.
섹션 순서도 규정되어 있으며, Overview, Colors, Typography, Layout, Elevation & Depth,
Shapes, Components, Do's and Don'ts의 8개 섹션이 이 순서로 등장해야 한다.

## CLI

CLI 도구 `@google/design.md`는 lint, diff, export, spec의 네 가지 하위 명령을 제공한다.
`lint`는 구조 검증과 WCAG 대비율 확인을 수행하고, `diff`는 두 버전 간 토큰 수준의 변경을
감지한다. `export`는 Tailwind 테마 설정 또는 W3C Design Tokens Format Module(DTCG)으로
토큰을 내보낸다. `spec`은 명세 전문을 출력해 에이전트 프롬프트에 주입할 수 있게 한다.

린터는 7개 규칙을 실행한다. `broken-ref`(오류), `missing-primary`, `contrast-ratio`,
`orphaned-tokens`, `missing-typography`, `section-order`(경고)와 `token-summary`,
`missing-sections`(정보) 수준으로 나뉜다. 프로그래매틱 API도 제공되어 라이브러리로
직접 임포트해 `lint(markdownString)` 형태로 사용할 수 있다.
현재 형식은 `alpha` 버전으로 활발히 개발 중이다.

## 분석

### 두 레이어 구조의 설계 원리

DESIGN.md의 핵심은 기계 판독 가능한 토큰과 인간이 읽는 산문을 한 파일에 공존시킨다는 점이다.
YAML 프론트매터는 컴파일러가 소비하는 정확한 값이고, 마크다운 본문은 LLM이 맥락을 이해하는
재료다. 이 이중 구조는 정적 분석과 의미 이해를 동시에 지원한다.

토큰만으로는 디자인 의도가 전달되지 않는다. `#B8422E`라는 값만 주면 에이전트는 이것이
유일한 강조색(accent)이고 CTA 버튼에만 써야 하며 “Boston Clay”라는 이름에 정체성이 담겨
있다는 사실을 알 수 없다. 산문이 이 의도를 채운다. 반대로 산문만 있으면 확률적 해석의
여지가 생기고 자동 검증이 불가능해진다. 두 레이어의 조합이 각각의 한계를 보완한다.

### 토큰 참조 문법과 단일 진실 원천

`{colors.tertiary}` 같은 참조 문법은 CSS 커스텀 프로퍼티나 Design Tokens 커뮤니티 그룹
(DTCG) 명세의 aliasing 개념과 구조적으로 동일하다.
컴포넌트 토큰이 기본 토큰을 참조하도록 강제함으로써, 색상이 한 곳에서만 정의되고
컴포넌트들이 그것을 가리키는 단일 진실 원천(single source of truth) 구조가 성립한다.
`broken-ref` 규칙이 에러 수준으로 실행되는 이유는 참조가 깨지는 순간 이 구조 자체가
무너지기 때문이다.

### diff 명령과 디자인 시스템의 버전 관리

`diff` 명령은 두 DESIGN.md 간의 토큰 수준 변경을 감지하고, “after” 파일의 오류와 경고가
늘었을 때 exit code 1을 반환한다. 이는 CI 파이프라인에 삽입 가능한 설계다.
디자인 시스템 변경이 접근성 저하나 구조 회귀를 유발하는지를 자동으로 게이팅할 수 있다.
코드 품질 게이팅과 동일한 패턴을 디자인 시스템에 적용한 것이다.

## 비평

### 강점

CLI의 exit code 설계가 일관되고 실용적이다.
`lint`는 오류가 있을 때만 1을 반환하고, `diff`는 회귀가 감지될 때 1을 반환한다.
CI 통합을 염두에 둔 설계이며, 복잡한 파싱 없이 쉘 스크립트 수준에서 통합할 수 있다.

`spec` 명령이 흥미로운 자기 참조 구조를 만든다.
`npx @google/design.md spec`을 실행해 명세 전문을 출력하고 이를 에이전트 프롬프트에 주입하면,
에이전트가 DESIGN.md 형식을 스스로 생성하거나 검증하도록 유도할 수 있다.
도구가 자신의 명세를 도구로서 출력한다는 메타적 설계다.

### 약점

타이포그래피 토큰 구조가 과소 명세되어 있다.
색상은 hex 값으로 명확히 정의되지만, 타이포그래피는 fontFamily, fontSize, fontWeight 등
여러 속성이 조합된 객체다. “Public Sans 3rem”과 “Public Sans 3rem / 1.2 line-height”는
렌더링 결과가 다르지만 토큰 스키마에서 필수/선택 구분이 모호하다.
이로 인해 에이전트마다 타이포그래피 해석이 달라질 수 있다.

알파 상태에서의 format churn 리스크도 있다.
“Expect changes to the format as it matures”라는 문구는 기존 DESIGN.md 파일이
스펙 변경으로 무효화될 수 있음을 암시한다. VoltAgent의 awesome-design-md 같은
생태계 파일들이 구버전 스펙을 따르는 경우 호환성 문제가 생긴다.
버전 필드가 선택 사항으로 정의된 것도 마이그레이션을 어렵게 만드는 요소다.

### WCAG 검증의 실질적 가치와 한계

`contrast-ratio` 규칙이 컴포넌트 토큰의 backgroundColor/textColor 쌍을 검증한다는 점은
접근성을 형식 명세에 내장시킨 드문 사례다. 대부분의 디자인 토큰 도구가 값의 정합성만
확인하는 반면, DESIGN.md는 접근성 기준을 린팅 규칙으로 포함시킨다.
그러나 대비율 계산은 컴포넌트 토큰에 정의된 쌍에만 적용되며,
인라인 텍스트나 중첩 컴포넌트, 반투명 배경 같은 복잡한 합성에는 적용되지 않는다.

## 인사이트

### 디자인 시스템이 컴파일 타깃이 되는 시대

소프트웨어 개발의 역사에서 “중간 표현(intermediate representation)”은 거듭 등장했다.
C가 어셈블리로, TypeScript가 JavaScript로, JSX가 DOM으로 컴파일된다.
각 변환에는 타깃 실행 환경에 최적화된 표현으로 변환하는 과정이 있다.

DESIGN.md는 디자인 의도를 UI 코드로 변환하는 파이프라인에서 중간 표현으로 자리잡으려 한다.
디자이너가 작성한 DESIGN.md는 AI 에이전트에 의해 React 컴포넌트, CSS 변수,
Tailwind 설정으로 “컴파일”된다.
이 관점에서 DESIGN.md의 스펙 정밀도는 컴파일러 명세의 정밀도와 같은 의미를 갖는다.
모호한 중간 표현은 컴파일 결과를 예측 불가능하게 만들듯,
모호한 DESIGN.md는 에이전트의 UI 생성 결과를 발산시킨다.

흥미로운 점은 이 변환이 결정론적이지 않다는 것이다.
기존 컴파일러는 동일한 입력에 동일한 출력을 보장하지만,
LLM 기반 “컴파일”은 확률적이다. DESIGN.md가 린팅과 WCAG 검증 같은 정적 분석을
파이프라인에 포함시킨 것은 이 비결정성을 사전에 줄이려는 시도다.
스펙이 충분히 엄밀하면 에이전트의 해석 공간이 좁아지고 출력의 분산이 줄어든다는 가설이다.

### W3C 토큰 명세와 도구 생태계 경쟁의 지형

W3C Design Tokens Format Module(DTCG)은 디자인 토큰의 표준화를 목표로 하는 명세다.
Style Dictionary, Theo, Token Pipeline 같은 도구들이 이 명세를 중심으로 생태계를 형성했다.
DESIGN.md의 `export --format dtcg` 명령은 이 생태계와의 상호 운용성을 확보하려는 선택이다.

그러나 DESIGN.md와 DTCG는 근본적으로 다른 독자를 전제한다.
DTCG는 도구 체인을 독자로 설계된 기계 판독 전용 형식이다.
DESIGN.md는 LLM과 인간을 동시에 독자로 설계한 하이브리드 형식이다.
export 명령이 DTCG로의 변환을 지원한다는 것은 DESIGN.md가 DTCG를 대체하려는 것이 아니라,
AI 에이전트 워크플로우의 진입점으로 포지셔닝하고 있음을 보여준다.
기존 도구 체인과 단절이 아닌 연결을 택한 것이다.

Google Stitch, Figma의 AI 기능, Vercel v0 같은 AI UI 생성 도구들이 각자의 디자인 명세
형식을 암묵적으로 가지고 있는 상황에서, DESIGN.md가 오픈 스탠다드 포지션을 선점하려는
전략이 보인다. 성공하면 AI 에이전트 UI 생성 파이프라인의 사실상 표준이 되고,
실패하면 생태계가 파편화된 채로 각 플랫폼의 독점 형식이 경쟁하는 구도가 이어진다.

### 접근성의 인프라화와 기본값의 힘

대부분의 개발 워크플로우에서 WCAG 접근성 검증은 추가 단계다.
별도 도구를 설치하고, CI에 추가 스텝을 구성하며, 팀이 의식적으로 실행해야 한다.
이 “추가 단계”라는 위치 때문에 접근성 검증은 마감 압박 하에서 가장 먼저 생략된다.

DESIGN.md는 접근성 검증을 형식 명세 자체에 내장한다.
`contrast-ratio` 규칙이 `lint` 명령의 기본 규칙으로 포함되어 있으므로,
디자인 명세를 검증하는 것과 접근성을 확인하는 것이 동일한 행위가 된다.
별도의 의식이나 추가 설정 없이 DESIGN.md를 lint하면 대비율이 확인된다.

이 패턴은 소프트웨어 품질 관행에서 반복적으로 등장한 전략이다.
ESLint가 코드 스타일을 개발 워크플로우에 내장시켰듯, Prettier가 포맷팅을 저장 시점에
실행하게 만들었듯, 좋은 관행을 “기본값”으로 만드는 것이 채택율을 결정한다.
접근성이 오픈소스 디자인 명세 형식의 기본 린팅 규칙으로 들어가는 것은
접근성의 인프라화가 진행되는 신호다.
장기적으로 접근성 미준수가 명시적인 경고나 오류로 시각화되는 도구들이 늘어날수록,
접근성은 의식적 노력에서 자동화된 품질 게이트로 전환된다.
