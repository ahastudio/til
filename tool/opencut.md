# OpenCut

<https://github.com/OpenCut-app/OpenCut>

HN 토론: <https://news.ycombinator.com/item?id=44553752> (447점, 151개 댓글)

## 소개

OpenCut은 CapCut의 대안을 표방하는 오픈소스 비디오 편집 플랫폼이다.
오늘 기준 66,328개의 스타와 6,987개의 포크를 보유하고 있으며
하루 만에 1,229개의 스타가 늘어날 정도로 GitHub 트렌드 상위에 올라 있다.

저장소는 현재 대대적인 아키텍처 재작성 작업을 진행 중이다.
기존 버전은 `opencut-app/opencut-classic`으로 분리되어 유지되고 있고,
새 버전은 `new.opencut.app`에서 개발되고 있다.
README에 따르면 새 아키텍처는 플러그인 우선 설계를 지향하며,
에디터 API를 통한 서드파티 확장, 웹·데스크톱·모바일에 걸친
공유 Rust 코어 기반 네이티브 지원, AI 에이전트를 위한 MCP 서버 통합,
헤드리스 자동화, 에디터 내 스크립팅 기능을 계획하고 있다.

기술 스택은 TypeScript가 97.2%로 대부분을 차지하고
CSS 2.1%, Rust 0.7%로 구성된다.
개발 도구로는 툴체인 관리를 위한 proto와
web·API·데스크톱 등 여러 앱을 아우르는 모노레포 오케스트레이션 도구인
moon을 사용한다.
라이선스는 MIT이며 fal.ai가 스폰서로 이름을 올리고 있다.
다만 현재는 아키텍처 재설계가 진행 중이라는 이유로 외부 기여를 제한하고 있고,
커뮤니티 참여는 Discord나 GitHub 이슈를 통한 논의로 한정된다.

## 분석

### 스타 수와 실제 완성도 사이의 간극

66,328개의 스타는 이 프로젝트가 이미 완성된 제품이라는 인상을 준다.
하지만 README를 읽어 보면 정작 저장소는 재작성이 진행 중이라고
스스로 밝히고 있다.
기존 버전은 별도 저장소로 옮겨졌고 새 버전은 아직 개발 초기 단계다.
즉 오늘 트렌드에 오른 이유는 완성된 결과물 때문이 아니라
“CapCut 대안”이라는 서사와 재작성 발표가 만들어낸 관심 그 자체에 가깝다.
스타 수라는 지표가 코드의 성숙도와 얼마나 쉽게 분리될 수 있는지를
보여주는 사례다.
실제로 이 프로젝트가 처음 화제가 되었던 HN 스레드에서도
같은 지적이 여러 차례 나왔다.
greatgib는 그렇게 많은 스타를 받았는데도
GitHub나 공식 웹사이트, 트위터 어디에서도
스크린샷 한 장을 찾을 수 없었다고 지적했고[^greatgib],
lpln3452는 스크린샷 없는 홍보성 README 자체를 신뢰하지 않는다며
“Project Structure” 절이 단순 디렉터리 목록에 불과해
LLM이 생성한 것처럼 보인다고 꼬집었다[^lpln3452].
Imustaskforhelp는 실제로 도커 컴포즈를 내려받아 실행해 봤더니
로컬호스트에서 열리는 것은 실제 편집기가 아니라
대기자 명단 등록 페이지뿐이었다며,
스타를 눌렀던 것을 취소하고 싶을 정도로 실망스러웠다고 전했다[^imustaskforhelp].

이런 간극은 오픈소스 생태계에서 드문 일이 아니다.
매력적인 콘셉트, 유명한 상용 제품을 겨냥한 포지셔닝,
활발한 커뮤니티 논의만으로도 스타는 얼마든지 쌓일 수 있다.
문제는 그 스타가 실제 사용자 저변이나 프로덕션 준비 상태를
보장하지 않는다는 점이다.
OpenCut을 평가할 때는 스타 수가 아니라
새 아키텍처가 실제로 얼마나 진척되었는지를 따로 확인해야 한다.

### 플러그인 우선 설계라는 승부수

새 아키텍처의 핵심은 에디터 API를 통한 서드파티 확장이다.
비디오 편집기를 처음부터 플러그인 구조로 설계한다는 것은
단순한 기술적 선택이 아니라 제품 전략의 재정의에 가깝다.
CapCut 같은 폐쇄형 상용 앱은 기능을 자체적으로 채워 넣어야 하지만,
플러그인 생태계가 자리 잡으면 커뮤니티가 기능을 채워 넣는 방식으로
개발 속도를 분산시킬 수 있다.

동시에 이 전략은 웹·데스크톱·모바일에 걸친 공유 Rust 코어라는
또 다른 승부수와 맞물려 있다.
TypeScript가 97%를 차지하는 현재 코드베이스에서 Rust는 아직 0.7%에
불과하지만, 이 비율은 재작성이 진행될수록 뒤집힐 가능성이 높다.
편집 로직을 Rust로 옮기고 각 플랫폼에서 이를 감싸는 방식은
성능과 이식성을 동시에 노리는 선택이지만,
동시에 두 가지 큰 기술적 도전(코어 재작성과 플러그인 API 설계)을
병행해야 한다는 뜻이기도 하다.

### MCP 통합이 가리키는 편집기의 다음 형태

README는 AI 에이전트를 위한 MCP 서버 통합과 헤드리스 자동화를
새 아키텍처의 계획된 기능으로 명시하고 있다.
이는 비디오 편집기를 사람이 마우스로 조작하는 도구에서
에이전트가 프로그래밍적으로 호출할 수 있는 도구로 확장하겠다는
의도로 읽을 수 있다.
에디터 내 스크립팅 기능까지 더해지면
OpenCut은 단순한 CapCut 대체재를 넘어
자동화 파이프라인의 한 구성 요소로 자리매김할 가능성이 있다.

이런 방향은 최근 여러 크리에이티브 도구가 공통적으로 걷고 있는
경로이기도 하다.
이미지 편집, 디자인 툴, 문서 작성 도구들이 잇따라 API와 에이전트
연동을 추가하는 흐름 속에서, OpenCut은 비디오 편집이라는
상대적으로 무거운 도메인에서 같은 흐름을 시도하고 있다.
다만 헤드리스 자동화와 MCP 통합은 README에 계획으로만 적혀 있을 뿐
아직 구현 여부나 구체적인 인터페이스는 공개되지 않았다.

### 모노레포와 moon이 말해주는 조직 규모의 야심

web, API, 데스크톱 앱을 하나의 모노레포에서 moon으로 오케스트레이션한다는
설정은 이 프로젝트가 단일 웹 앱 수준을 넘어서는 규모를 염두에 두고 있음을
보여준다.
proto로 툴체인 버전을 고정하고 moon으로 태스크를 조율하는 구성은
여러 앱과 여러 기여자가 동시에 작업하는 것을 전제로 한 선택이다.
이는 개인 프로젝트나 소규모 오픈소스에서는 잘 채택하지 않는
무거운 도구 조합이다.

역으로 보면 이 구성 자체가 프로젝트의 야심을 드러내는 신호이기도 하다.
단순히 웹에서 동작하는 편집기 하나를 만드는 것이 아니라
데스크톱과 모바일까지 아우르는 크로스 플랫폼 제품을
처음부터 모노레포 체제로 준비하고 있다는 뜻이다.
문제는 이런 규모의 인프라를 유지하려면 상당한 엔지니어링 리소스가
필요하다는 점이고, 현재 외부 기여가 제한된 상태에서
이 부담을 핵심 팀이 얼마나 감당할 수 있을지는 README만으로는
가늠하기 어렵다.

## 비평

### “오픈소스”라는 이름표와 닫힌 기여 문 사이의 모순

OpenCut은 MIT 라이선스로 코드를 공개하고 있지만
현재 외부 기여는 받지 않는다고 명시한다.
코드를 읽고 포크할 수는 있지만 풀 리퀘스트로 참여할 수는 없는 상태다.
이는 라이선스상의 개방성과 실질적인 협업 개방성이 다르다는
점을 그대로 드러낸다.
스타를 누른 6만 명이 넘는 사용자 대부분은
정작 이 프로젝트의 방향에 아무런 영향력을 행사할 수 없다.

이런 상태가 재작성이라는 일시적 사정 때문이라는 설명은 납득할 수 있지만,
그 “일시적” 기간이 얼마나 지속될지는 아무도 보장하지 않는다.
많은 오픈소스 프로젝트가 “재설계 중이라 기여를 잠시 닫는다”는
공지를 낸 뒤 그 상태가 수개월에서 수년간 이어진 전례가 있다.
OpenCut이 그런 전철을 밟지 않을 것이라는 근거는
현재 README 어디에도 없다.

HN 스레드에서는 이런 폐쇄성을 오픈소스 명분과 상업적 동기가
뒤섞인 정황으로 보는 시각도 나왔다.
arnaudsm는 오픈소스 프로젝트인데 왜 대기자 명단을 운영하고
자체 분석 도구인 Databuddy 탑재를 홍보하는지 물으면서,
사실은 오픈소스를 마케팅 수단으로 쓰다가 나중에 수익화할
스타트업이 아니냐고 되물었다[^arnaudsm].
alexfromapex 역시 대기자 명단이 곧 유료화를 뜻하는 것인지
같은 의문을 제기했다[^alexfromapex].
이런 반응들은 “MIT 라이선스인데 왜 참여를 막느냐”는 질문에서 한 걸음 더 나아가
“애초에 오픈소스라는 이름표를 붙인 목적이 무엇이냐”는
더 근본적인 의심으로 이어진다.

### 계획의 나열과 구현의 부재

새 아키텍처가 약속하는 항목들, 즉 에디터 API, 공유 Rust 코어,
MCP 서버, 헤드리스 자동화, 인앱 스크립팅은 하나하나가
독립적으로도 상당한 개발 공수가 필요한 기능이다.
이 다섯 가지를 동시에 계획으로 내걸었다는 것은
로드맵으로서는 매력적이지만, 실행 가능성 검증으로서는
아직 아무것도 보여주지 않았다는 뜻이기도 하다.

비디오 편집기는 프레임 정확도, 코덱 호환성, 렌더링 성능처럼
눈에 잘 띄지 않지만 검증에 오랜 시간이 걸리는 요구사항이 많은 도메인이다.
이런 도메인에서 플러그인 API와 크로스 플랫폼 코어를 동시에
새로 설계하겠다는 계획은, 각 기능이 서로의 완성을 전제로 하는
의존 관계에 놓일 위험이 크다.
Rust 코어가 안정되지 않으면 플러그인 API도 확정하기 어렵고,
플러그인 API가 확정되지 않으면 MCP 통합도 임시방편에 머물 수밖에 없다.
README는 이 순서와 의존성에 대해 아무런 언급을 하지 않는다.

### CapCut 대안이라는 포지셔닝의 실체 검증 필요성

CapCut은 숏폼 영상 편집에 최적화된 완성도 높은 상용 앱으로,
자동 캡션, 템플릿, AI 기반 편집 보조 기능까지 갖추고 있다.
OpenCut이 스스로를 그 대안으로 소개하는 것은 마케팅 관점에서는
효과적이지만, 현재 저장소가 재작성 중이라는 사실을 감안하면
이 표현은 “지금 당장 CapCut을 대체할 수 있다”는 뜻이 아니라
“CapCut이 채운 시장 공백을 오픈소스로 채우겠다”는 지향에 가깝다.

이 차이를 인식하지 못한 사용자가 당장 대체재를 찾아 저장소를 방문했다가
아직 미완성인 새 버전과 별도로 분리된 클래식 버전을 마주하게 되면
혼란을 겪을 수 있다.
포지셔닝 문구와 실제 제품 상태 사이의 시차는
이 프로젝트가 마케팅 언어를 관리하는 데 있어
좀 더 신중해야 할 지점이다.

실제로 사용해 본 이용자들의 반응도 이 시차를 뒷받침한다.
xnx와 vrthink는 CapCut을 대체할 오픈소스 편집기라는 발상 자체는
반갑지만, Bun과 Docker, Docker Compose, Node.js를 미리 설치해야 하는
현재 진입 장벽은 캐주얼한 CapCut 사용자 대부분을 걸러낼 것이라고
지적하며 Blender나 Shotcut, OpenShot, Kdenlive 같은 기존 오픈소스
편집기가 지금 시점에는 더 현실적인 선택지라고 평가했다[^xnx].
jchw는 실제로 몇 초간 사용해 본 소감을 남겼는데,
정작 사용해 보려면 저장소가 아니라 `opencut.app/projects`라는
별도 주소를 찾아가야 했고, 대부분의 기능이 아직 TODO 상태였지만
텍스트를 영상 위에 얹는 정도는 가능했다고 전했다[^jchw].
CapCut 페이지 안에 "왜 CapCut을 쓰지 않는가"라는 이름의
비교 페이지를 별도로 만들어 둔 점도 눈에 띄었는데,
eddythompson80과 bravesoul2는 그 페이지의 도발적인 어조를 지적하며
씁쓸한 반응을 남겼다[^eddythompson80].
이는 포지셔닝이 단순한 설명을 넘어 경쟁사를 직접 겨냥하는
공격적인 톤으로 짜여 있다는 뜻이기도 하다.

## 인사이트

### 재작성 발표가 기여자 커뮤니티에 미치는 이차 효과

외부 기여를 닫고 재작성을 진행하는 결정은 단기적으로는
아키텍처 일관성을 지키는 데 도움이 되지만,
장기적으로는 기존에 형성되어 있던 기여자 풀을 흩어지게 만드는
이차 효과를 낳을 수 있다.
클래식 버전에 이미 기여했던 개발자들은 자신의 작업이
새 아키텍처에서 어떻게 이어질지 알 수 없는 상태에 놓인다.
기여 문이 다시 열렸을 때 이들이 돌아온다는 보장은 없고,
그사이 다른 오픈소스 비디오 편집 프로젝트로 관심이 옮겨갈 가능성도 있다.

스타 수 증가가 실제 기여자 증가로 이어지지 않는 구조가
당분간 고착될 수 있다는 점도 눈여겨볼 부분이다.
관심은 쌓이는데 참여 경로는 막혀 있는 상태가 길어지면
프로젝트에 대한 기대감이 실망으로 바뀌는 시점이 찾아올 수 있고,
이는 트렌드에 오른 프로젝트일수록 더 빠르게 나타나는 경향이 있다.

### 대형 상용 소프트웨어의 오픈소스 재현이라는 익숙한 서사

CapCut의 오픈소스 대안을 만들겠다는 시도는
과거 여러 도메인에서 반복된 패턴과 구조적으로 닮아 있다.
상용 데이터베이스에 맞서 등장한 PostgreSQL 생태계의 파생 프로젝트들,
Photoshop에 맞서 성장한 GIMP, Slack에 맞서 등장했던 여러 오픈소스
협업 도구들처럼, 지배적인 상용 제품이 존재할 때
그 빈틈을 오픈소스로 채우려는 시도는 꾸준히 반복되어 왔다.

이런 역사적 사례들이 공통적으로 보여주는 교훈은
초기의 관심과 스타 수보다 중요한 것이 장기간에 걸친
꾸준한 엔지니어링 투자와 커뮤니티 유지라는 점이다.
GIMP는 살아남았지만 Photoshop과 동등한 위치에 오르지는 못했고,
많은 Slack 대안들은 관심이 식으면서 유지보수가 멈췄다.
OpenCut이 66,328개의 스타로 시작한 오늘의 관심을
몇 년 뒤에도 유지할 수 있을지는 재작성이 실제로 완료되고
플러그인 생태계가 형성되는지에 달려 있다.

### 플랫폼 확장과 코어 안정성 사이의 트레이드오프

공유 Rust 코어로 웹·데스크톱·모바일을 동시에 지원하겠다는 계획은
매력적이지만, 이 선택에는 잘 드러나지 않는 트레이드오프가 있다.
하나의 코어로 여러 플랫폼을 지원하면 유지보수 비용은 줄어들지만,
플랫폼마다 다른 성능 특성이나 파일 시스템, GPU 가속 방식의 차이를
추상화 계층 뒤에 숨겨야 하는 부담이 커진다.
비디오 편집처럼 하드웨어 자원을 많이 쓰는 도메인에서는
이 추상화 비용이 성능 저하로 직접 드러날 위험이 있다.

README는 이 트레이드오프에 대해 별다른 언급을 하지 않고
크로스 플랫폼 지원을 단순히 장점으로만 제시하고 있다.
하지만 실제로 코어를 공유하면서도 각 플랫폼에서 만족스러운 편집 성능을
내려면, 플랫폼별 최적화 경로를 따로 마련하거나
일부 기능을 플랫폼 특화로 남겨두는 타협이 필요할 가능성이 크다.
이런 현실적인 조정 과정이 로드맵에 아직 반영되어 있지 않다는 점은
이 프로젝트를 지켜볼 때 함께 확인해야 할 지점이다.
