# Hackney – 우버·리프트 가격 한눈에 비교하는 라이드셰어 앱

<https://hackney.app/>

HN 토론: <https://news.ycombinator.com/item?id=48893550> (60점, 47개 댓글)

## 소개

Hackney는 Uber, Lyft, Waymo, Curb, Empower 등 여러 라이드셰어 서비스의
가격과 대기 시간을 한 화면에서 비교하는 iOS/Android 앱이다.
목적지를 입력하면 각 서비스의 예상 요금과 도착 시간을 나란히 보여주며,
원하는 앱으로 연결해 예약을 완료하는 방식이다.

보안 구조에서 차별점을 내세운다.
데이터 처리를 클라이언트(기기) 측에서 수행하고, 각 라이드셰어 회사의
서버에 직접 연결하며, 사용자의 로그인 정보와 데이터를 Hackney 서버에
보내지 않는다.
“Your login is private”와 “Works client-side”를 강조한다.

서지 요금(surge pricing) 회피 기능도 제공한다고 밝히지만,
그 구체적인 작동 방식은 홈페이지에서 설명하지 않는다.
Hackney Carriage, LLC가 개발했으며, 어떤 라이드셰어 회사와도 제휴 관계가
없다고 명시한다.

## 분석

### 라이드셰어 비교 앱의 핵심 가치는 “전환 마찰 제거”다

라이드셰어 시장에는 여러 앱이 존재하지만, 사용자가 습관적으로 하나만
쓰는 이유는 비교의 마찰이 크기 때문이다.
Uber 앱에서 요금을 확인하고 Lyft 앱으로 넘어가 다시 입력하는 과정은
번거롭다.
Hackney는 이 전환 마찰을 없애 비교를 기본 경험으로 만든다.

이 가치 제안은 Kayak, Google Flights가 항공권 시장에서 한 것과 구조가
같다.
예약 자체는 각 서비스 앱에서 완료되지만, 선택 결정은 비교 레이어에서
이루어진다.
비교 레이어가 습관이 되면, 개별 앱의 충성도는 약해진다.

### 클라이언트 측 처리가 핵심 기술 차별점이다

라이드셰어 비교 앱의 기술적 어려움은 각 서비스의 실시간 요금을 가져오는
방식에 있다.
공식 API를 제공하지 않는 서비스의 경우, 데이터를 중간 서버를 통해 프록시하면
이용약관 위반 위험이 있고, 서버 비용도 발생한다.

Hackney가 “각 라이드셰어 회사 서버에 직접 연결”하고 “클라이언트 측 처리”를
강조하는 것은 이 문제에 대한 기술적 해법을 암시한다.
앱이 사용자 기기에서 직접 각 서비스에 연결하면, Hackney 측 서버가 중간에
끼지 않아도 된다.
이 구조는 서버 비용을 줄이고, “우리는 사용자 데이터를 보지 않는다”는
프라이버시 주장도 더 쉽게 뒷받침한다.

개발자 griffinli는 HN에서 구체적인 구현 방식을 직접 설명했다.[^griffinli-hn]
라이드셰어 앱들을 역공학해 서버와 통신하는 방식을 파악했고, 인증 토큰은
사용자 기기에 저장되어 각 서비스에 직접 요청을 보낸다는 것이다.
일부 앱은 인증서 피닝(certificate pinning)으로 네트워크 요청 열람을 막고,
일부는 원래 필드명이 없는 Protobuf 형식으로 통신해 구현이 기술적으로 복잡하다고도 밝혔다.

### 서지 요금 회피의 실제 의미

서지 요금은 수요가 높을 때 Uber나 Lyft가 요금을 평소보다 높게 책정하는
방식이다.
Hackney가 “서지 요금을 건너뛸 수 있다”고 말하는 것은, 한 서비스가 서지
요금을 적용할 때 다른 서비스는 적용하지 않을 수 있다는 점을 활용하는 것이다.
비교 기능 자체가 서지 요금 회피의 핵심 메커니즘이다.

이것은 사용자에게 실질적인 금전적 이득이다.
출퇴근 시간이나 행사가 끝난 직후처럼 서지가 강하게 붙는 상황에서,
여러 서비스를 빠르게 비교해 서지가 덜한 쪽을 선택할 수 있다.

## 비평

### 공식 API 없는 비교 앱의 지속 가능성 문제

Uber와 Lyft는 라이드셰어 비교 앱을 위한 공개 API를 제공하지 않는다.
실제로 개발자가 HN에서 직접 밝혔듯, Uber는 공식 API를 제공하지만 가격 비교 목적
사용을 금지하며, 과거에 공식 API를 가격 비교에 쓴 앱의 API 접근을 차단한 전례가 있다.[^griffinli-hn]
Hackney는 이 때문에 역공학을 통해 내부 API를 활용하는 방식을 택했다.

gonzalohm은 Uber가 내부 API를 막는 조치를 취할 가능성을 물었다.[^gonzalohm]
개발자는 “서드파티 클라이언트를 완전히 차단하기는 어려우며, API가 바뀌어도
대개 우회할 수 있다”고 답했다.[^griffinli-api]

한편 f3408fh는 앱스토어 심사에서 역공학에 기반한 앱이 통과되는지를 물었다.[^f3408fh]
개발자는 앱스토어 심사가 문제 삼지 않았으며, 통합 메시징 앱이나 드라이버용
라이드셰어 집계 앱 등 이미 유사한 앱들이 다수 존재한다고 밝혔다.[^griffinli-appstore]

### 비교 기능의 실제 효용은 지역과 시장 구조에 달려있다

Hackney의 가치는 여러 라이드셰어 서비스가 경쟁하는 시장에서만 발휘된다.
미국 주요 도시에서는 Uber와 Lyft가 모두 강하게 경쟁하고 Waymo도 일부
지역에서 운영되므로 비교 가치가 있다.

그러나 한 서비스가 독점적인 지역에서는 비교할 선택지가 없다.
한국 등에서는 Uber가 제한적이고 카카오T가 지배적인 구조라 Hackney의
비교 가치가 거의 없다.
앱의 효용은 사용자의 지리적 위치와 시장 구조에 전적으로 의존한다.

## 인사이트

### 라이드셰어 비교 레이어는 플랫폼의 가격 결정력을 약화시킨다

Uber와 Lyft가 비교 앱을 공식 지원하지 않는 이유가 있다.
비교 레이어가 보편화되면, 두 서비스는 가격 경쟁에서 서로를 능가하기
어려워진다.
실시간 비교가 이루어지는 환경에서 서지 요금을 높게 책정하는 서비스는
고객을 빠르게 잃는다.

이는 항공권 시장에서 Kayak·Google Flights가 등장한 이후 항공사들이 가격
투명성에 노출된 것과 같은 구조다.
비교 인프라가 성숙할수록 플랫폼의 가격 결정력이 줄어드는 경향이 있다.
라이드셰어 시장에서도 이 방향이 가속화되면 서비스 간 차별화가 요금
외의 요소—안전성, 운전자 품질, 취소 정책—로 이동할 수 있다.

### "계정 잠금 위험"이 사용자 확보의 심리적 장벽이다

xnx는 HN에서 "Uber 계정이 잠길까봐 시도하기가 두렵다"고 밝혔다.[^xnx]
이 우려는 사용자 입장에서 합리적이다.
라이드셰어 계정이 잠기면 일상적인 이동 수단을 잃게 되는데, 비교 기능 몇
달러 절감이 그 리스크를 감수할 만큼 가치 있는지 판단하기 어렵다.

그러나 dantemoon은 수개월 이상 iOS와 Android 양쪽에서 사용하면서 계정이
잠기거나 차단된 적이 없다고 밝혔다.[^dantemoon]
실제 사용 경험이 이 두려움이 실질적 위험보다 더 크게 느껴진다는 것을
보여준다.

또한 josefrichter는 Obi라는 앱이 같은 기능을 몇 년 전부터 제공하고 있으며
전 세계적으로 동작한다고 언급했다.[^josefrichter]
Hackney가 완전히 새로운 카테고리를 개척하는 것이 아니라, 기존 비교 앱보다
더 나은 프라이버시와 미국 특화 서비스(Waymo, Tesla Robotaxi 포함)로 차별화를
노리는 것임을 시사한다.

### 프라이버시 중심 설계가 경쟁 우위가 되는 카테고리

위치 정보와 이동 패턴은 민감한 데이터다.
어디서 어디로, 언제 이동하는지는 생활 패턴 전반을 드러낸다.
Hackney가 “클라이언트 측 처리, 서버 없음, 로그인 비공개”를 전면에 내세우는
것은 이 민감성에 대한 대응이다.

기존 라이드셰어 앱은 이동 데이터를 수집하고 광고나 서드파티 파트너십에
활용한다.
Hackney처럼 중간 서버 없이 작동하는 구조는 이 데이터 수집 고리를 끊는다.
프라이버시를 이유로 추가 앱 설치를 꺼리는 사용자들에게 이 구조는 실질적인
설득 포인트다.
