# StackRender – 데이터베이스 스키마 다이어그램 생성 도구

<https://www.stackrender.io/>

<https://github.com/stackrender/stackrender>

## 소개

StackRender는 오픈소스 데이터베이스 스키마 다이어그램 생성 도구다.
TypeScript와 React Flow(xyflow)로 구현되었으며, 드래그 앤 드롭
UI에서 스키마를 설계하고 SQL DDL로 내보낼 수 있다.
PostgreSQL, MySQL, MariaDB, SQLite, Oracle, SQL Server(MSSQL)를
지원하며, AGPLv3 라이선스로 배포된다.

GitHub 스타 373개로, Docker Compose로 실행하거나 `npm run dev`로
로컬 개발 환경을 띄울 수 있다.

## 주요 기능

- **스키마 다이어그램 편집**: 드래그 앤 드롭으로 테이블과 컬럼 구성
- **DDL 가져오기/내보내기**: SQL 스크립트를 임포트하거나 다이어그램에서
  DDL 생성
- **순환 의존성 감지**: 외래키 순환 참조 탐지
- **인덱스 최적화 제안**: 성능 개선을 위한 인덱스 권장
- **AI 지원 스키마 생성**: 클라우드 버전에서 스키마 보강 및 문서 자동 생성

## 분석

### “사양에서 프로덕션 데이터베이스로”라는 가치 제안의 실제 범위

StackRender의 슬로건은 사양(specification)에서 바로 프로덕션 준비된
데이터베이스로 이어진다고 말한다.
그러나 이 흐름에서 StackRender가 실제로 담당하는 범위는 스키마 다이어그램
→ DDL 생성까지다.
마이그레이션 관리, 인덱스 튜닝, 파티셔닝, 복제 설정 같은 프로덕션 환경의
추가 작업은 포함되지 않는다.

그럼에도 시각적 스키마 설계 → DDL 자동 생성 파이프라인 자체의 가치는
충분하다.
Lucidchart나 dbdiagram.io처럼 다이어그램만 그리고 DDL은 직접 작성해야
하는 도구와 비교하면, StackRender는 설계와 구현 사이의 싱크를 유지한다.
특히 여러 DBMS를 지원하면서 같은 다이어그램에서 각 방언에 맞는 DDL을
생성할 수 있다면, 멀티 데이터베이스 환경에서 유용하다.

### AI 기능이 클라우드 버전에만 있는 구조의 의미

AI 지원 스키마 생성은 클라우드 버전(`stackrender.io`)에만 있다.
오픈소스 셀프 호스팅 버전에는 없다.
이 분리는 전형적인 오픈코어(open-core) 전략이다.
핵심 기능은 오픈소스로 제공해 개발자 커뮤니티를 모으고,
고부가가치 기능은 클라우드 서비스로 묶어 수익화한다.

AGPLv3 라이선스도 이 전략과 맞물린다.
AGPL은 네트워크를 통해 소프트웨어를 제공하는 경우에도 소스 코드
공개를 요구하기 때문에, 경쟁 SaaS가 StackRender를 그대로 클라우드로
서비스하기 어렵게 만든다.
개발자가 직접 쓰는 셀프 호스팅은 허용하되, 상업적 SaaS 운영은
차단하는 구조다.

### React Flow(xyflow) 기반 다이어그램 구현의 트레이드오프

StackRender는 React Flow(현재 이름: xyflow)를 기반으로 다이어그램 UI를
구현한다.
React Flow는 노드-엣지 그래프 편집기를 빠르게 구현할 수 있는 라이브러리로,
많은 다이어그램 도구가 채택한다.

이 선택의 트레이드오프는 커스터마이징 자유도와 성능에 있다.
React Flow는 유연하지만, 수백 개의 테이블이 있는 대규모 스키마에서
렌더링 성능이 어떻게 나오는지는 직접 테스트해봐야 한다.
엔터프라이즈급 데이터베이스 스키마는 테이블이 수백 개를 넘기도 하므로,
이 규모에서의 성능이 실용성의 핵심 기준이다.

## 비평

### 순환 의존성 감지가 얼마나 유용한지는 사용 방식에 달려있다

순환 의존성 감지는 설계 단계에서 외래키 순환을 잡아내는 기능으로,
이론적으로는 유용하다.
그런데 실제로 스키마 설계에서 순환 참조가 발생하는 경우는 많지 않으며,
발생한다면 대부분 설계 자체의 문제로 도구의 자동 감지 이전에 인지할 수
있는 경우가 많다.
이 기능이 실제 워크플로에서 얼마나 자주 쓰이는지는 불분명하다.
반면 자주 필요한 기능인 “기존 DB에서 스키마를 리버스 엔지니어링하기”나
“스키마 변경 이력 비교”는 소개 자료에서 강조되지 않는다.

### AGPLv3 라이선스가 기업 채택을 제한할 수 있다

AGPLv3는 상업적 SaaS를 막는 데 효과적이지만, 동시에 기업 내부 사용에도
제약을 줄 수 있다.
많은 기업의 법무팀은 AGPL 소프트웨어 내부 배포에 대해 신중한 입장을
취한다.
이는 개발자가 개인적으로 유용하다고 판단해도 기업 환경에서의 공식 도입이
지연되거나 거부될 수 있음을 의미한다.
MIT나 Apache 2.0에 비해 AGPL은 기업 채택 장벽이 높다.

## 인사이트

### 데이터베이스 스키마 도구 시장은 AI 통합으로 재편되는 중이다

전통적인 ER 다이어그램 도구(Lucidchart, draw.io, dbdiagram.io)는 수동
설계를 전제로 한다.
StackRender처럼 AI로 스키마를 자동 생성하거나 보강하는 기능을 추가하는
방향이 이 시장의 다음 경쟁 축이 되고 있다.

자연어 요구사항에서 스키마를 생성하는 AI 기능이 충분히 신뢰할 수 있는
수준이 된다면, 스키마 설계의 첫 단계는 “다이어그램 그리기”에서 “AI 초안을
검토하고 수정하기”로 이동한다.
이 변화에서 시각적 편집기의 역할은 초안 작성에서 검토·수정으로 바뀐다.
StackRender의 AI 기능이 이 흐름을 얼마나 잘 지원하는지가 클라우드 버전의
경쟁력을 결정할 것이다.

### 오픈소스 데이터베이스 도구의 지속 가능성 모델

StackRender가 채택한 오픈코어 + AGPLv3 모델은 오픈소스 데이터베이스
도구가 수익화를 시도하는 패턴 중 하나다.
GitLab, Metabase, Redash가 같은 패턴을 따른다.
핵심 기능은 오픈소스로 커뮤니티 신뢰를 얻고, 클라우드/엔터프라이즈 기능으로
수익을 낸다.

이 모델의 성공 조건은 클라우드 버전에서만 사용할 수 있는 기능이
충분히 가치 있어야 한다는 점이다.
AI 스키마 생성은 그런 기능이 될 수 있지만, AI 기능 자체가 빠르게
상품화되는 지금 이 차별성이 얼마나 지속될지는 불확실하다.
