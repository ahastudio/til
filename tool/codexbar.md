# CodexBar — 로그인 없이 AI 코딩 사용량을 모아 보는 메뉴바 앱

<https://codexbar.app/>

<https://github.com/steipete/CodexBar>

## 소개

CodexBar는 macOS 14(Sonoma) 이상에서 동작하는 경량 메뉴바 앱이다.
개발자 Peter Steinberger(steipete)가 Swift로 만들었고,
Codex, Claude Code, Cursor, Gemini, Copilot 등
57개가 넘는 AI 코딩 제공자의 사용량 한도와 리셋 시각을 한 화면에서
보여준다.
핵심 가치 제안은 “로그인 없이”다.
OAuth 토큰, 브라우저 쿠키, API 키, 로컬 CLI 설정 파일 등
사용자가 이미 갖고 있는 인증 정보를 재사용하기 때문에
CodexBar 자체에 비밀번호를 새로 입력할 필요가 없다.
라이선스는 MIT이고, GitHub Releases, Homebrew, AUR을 통해 배포된다.

## 명세

CodexBar가 제공자별로 읽어들이는 인증 방식은 제각각이다.

- Codex: OAuth API 또는 로컬 Codex CLI, 선택적으로 OpenAI 웹 대시보드 쿠키
- Claude: OAuth API, 브라우저 쿠키, 또는 CLI PTY 폴백
- Cursor, Windsurf, Zed 등: 브라우저 세션 쿠키 또는 로컬 SQLite 캐시
- OpenAI, ElevenLabs, OpenRouter, DeepSeek 등: API 키 기반 사용량 조회
- AWS Bedrock: AWS 액세스 키 또는 named profile(SSO/assume-role)

macOS 권한 요구 사항도 문서화되어 있다.
Full Disk Access는 Safari 쿠키를 읽을 때만 필요하고,
Keychain 접근은 Chromium 계열 브라우저의 쿠키를 복호화하거나
Claude OAuth 부트스트랩에 캐시된 자격 증명이 없을 때만 발생한다.
Screen Recording이나 Accessibility 권한은 요구하지 않는다고 명시한다.

## 사용법

설치 경로는 세 가지다.

```bash
# Homebrew (macOS)
brew install --cask codexbar

# Homebrew (Linux 배포용 CLI)
brew install steipete/tap/codexbar

# AUR (Arch Linux)
yay -S codexbar-cli
```

GitHub Releases에서 `.app` 번들이나 플랫폼별 CLI 타르볼을
직접 내려받을 수도 있다.
설치 후에는 Settings → Providers에서 사용할 제공자를 켜고,
각 제공자가 요구하는 인증(CLI 로그인, 브라우저 세션, API 키 등)을
연결하면 된다.
API 키가 필요한 제공자는 CLI로도 등록할 수 있다.

```bash
codexbar config providers
codexbar config enable --provider grok
printf '%s' "$ELEVENLABS_API_KEY" | \
  codexbar config set-api-key --provider elevenlabs --stdin
```

번들 CLI는 `codexbar cost --provider codex` 같은 명령으로
CI 파이프라인에서도 사용량을 조회할 수 있게 해준다.
Merge Icons 모드로 여러 제공자를 하나의 메뉴바 아이템으로
합쳐서 볼 수도 있다.

## 분석

### “로그인 없음”은 인증을 없앤 것이 아니라 위임한 것이다

CodexBar의 핵심 홍보 문구는 로그인이 필요 없다는 것이지만,
실제로는 인증 자체를 생략한 게 아니라
사용자가 이미 다른 도구(브라우저, CLI, OAuth 앱)에 해 둔 인증을
재사용하는 구조다.
즉 CodexBar는 새로운 신뢰 관계를 만드는 대신,
기존에 존재하던 57개의 서로 다른 신뢰 관계를 읽어내는
파서와 집계기 역할을 한다.
이 설계는 온보딩 마찰을 줄이는 대신,
각 제공자의 쿠키 포맷·토큰 위치·CLI 설정 파일 구조가 바뀔 때마다
CodexBar가 그 변화를 따라잡아야 하는 유지보수 부담을 안는다.
57개 제공자 문서(`docs/*.md`)가 각각 별도로 존재한다는 사실 자체가
이 부담의 크기를 보여준다.

### 사용량 추적 도구의 존재는 AI 코딩 요금제의 불투명성을 반증한다

CodexBar 같은 도구가 필요하다는 사실 자체가,
각 AI 코딩 제공자가 자체 대시보드에서 사용량과 리셋 시각을
충분히 투명하게 보여주지 못하고 있다는 방증이다.
세션·주간·월간 윈도우가 제공자마다 다르고,
어떤 곳은 API로 조회 가능하지만 어떤 곳은 브라우저 쿠키를
훔쳐보듯 읽어야만 확인할 수 있다.
이는 개별 제공자의 실패라기보다,
AI 코딩 도구 시장이 짧은 기간에 급증하면서
사용량 공개 표준이 아직 수렴하지 않았다는 구조적 현상에 가깝다.
“Inspired by ccusage”라는 크레딧 표기도 같은 맥락을 보여준다.
Claude Code 사용량을 추적하는 ccusage가 먼저 등장했고,
CodexBar는 그 문제의식을 다중 제공자로 확장한 것이다.

### 커뮤니티 이식판의 확산은 이 도구가 채운 공백의 크기를 보여준다

README 하단에는 Windows판(Win-CodexBar), Waybar 모듈,
GNOME Shell 확장, KDE Plasma 위젯, SketchyBar/tmux/Zellij용
플러그인까지 CodexBar의 CLI를 기반으로 만들어진
제3자 이식판이 다섯 개 넘게 나열되어 있다.
원 저장소가 macOS 전용임에도 불구하고
이렇게 빠르게 여러 플랫폼으로 재구현되었다는 것은,
“메뉴바에서 AI 사용량을 한눈에 보고 싶다”는 수요가
macOS 사용자에게 국한되지 않음을 보여준다.
동시에 이는 CodexBar 본체가 감당하지 못하는 플랫폼 확장을
커뮤니티가 대신 떠맡고 있다는 뜻이기도 하다.

## 비평

### “디스크를 스캔하지 않는다”는 해명은 사용자 경험의 문제를 해결하지 못한다

저장소 issue #12에서 한 사용자는 최신 버전이
자신의 여러 폴더를 순차적으로 훑으며 접근 권한을 요구한다고
불만을 제기했다.
개발자는 README의 프라이버시 노트에서
“파일시스템을 무차별로 훑지 않으며, 관련 기능이 켜졌을 때만
알려진 소수의 위치(브라우저 쿠키/로컬 스토리지, 제공자 설정 파일,
로컬 JSONL 로그)를 읽는다”고 해명한다.
그러나 사용자 입장에서 실제로 목격한 현상은
macOS가 폴더 접근을 반복적으로 물어보는 상황이었고,
이는 “스캔하지 않는다”는 설명과 별개로 실제 UX 문제다.
57개 제공자 각각이 서로 다른 로컬 경로를 확인하려 들면,
사용자에게는 그 총합이 광범위한 디스크 접근 시도로 보일 수밖에 없다.
설계 의도의 정당성과 사용자가 체감하는 신뢰도는 다른 층위의 문제다.

### 제공자 수가 늘수록 공격 표면과 감사 부담이 함께 늘어난다

57개 제공자를 지원한다는 것은 57개의 서로 다른 인증 정보 소스,
즉 브라우저 쿠키 복호화 로직, OAuth 토큰 캐시, API 키 저장소,
로컬 CLI 설정 파일 파서가 하나의 앱 안에 공존한다는 뜻이다.
이 중 하나라도 잘못 구현되면 - 예를 들어 토큰을 로그에 남기거나
평문으로 디스크에 쓰면 - 전체 도구의 신뢰가 흔들린다.
README는 “restrictive file permissions”로 설정 파일을 보호한다고
언급하지만, 57개 제공자 각각의 구현이 이 원칙을 얼마나 일관되게
지키는지는 오픈소스 코드를 직접 감사하지 않는 한 사용자가
검증하기 어렵다.
오픈소스라는 사실이 감사 가능성을 열어 두기는 하지만,
실제로 감사가 이루어졌다는 것과는 다른 이야기다.
제공자 목록이 계속 늘어나는 성장 전략은
곧 검증되지 않은 코드 경로가 계속 늘어나는 것과 같은 속도로
진행된다.

### Keychain 접근을 완전히 끄는 옵션은 기능 저하를 감춘 절충안이다

README는 Keychain 접근 프롬프트가 계속 뜨는 경우
“Settings → Advanced → Keychain access”에서 아예 꺼버릴 수 있다고
안내하면서, 이 경우에도 “Claude/Codex OAuth via CLI는
`~/.codex` / `~/.claude` 설정 파일을 읽으므로 계속 동작한다”고
설명한다.
이는 마치 기능 손실이 없는 것처럼 서술되어 있지만,
실제로는 브라우저 쿠키 기반 제공자(Cursor, Windsurf, Zed 등
상당수)가 이 옵션을 켜는 순간 조용히 동작을 멈춘다는 뜻이다.
사용자가 반복되는 Keychain 프롬프트에 지쳐 이 옵션을 켰다가
일부 제공자의 사용량이 더 이상 갱신되지 않는 이유를
스스로 알아채기는 쉽지 않다.
편의 기능을 끄는 스위치가 정확히 무엇을 희생시키는지는
UI 안에서 더 명확하게 드러나야 한다.

## 인사이트

### 사용량 집계 도구는 결국 인증 정보의 중앙 저장소가 된다

CodexBar는 스스로 새로운 계정 체계를 만들지 않겠다고 표방하지만,
결과적으로는 57개 서비스에 대한 자격 증명이나 그 파생물(쿠키,
토큰, API 키)에 접근할 수 있는 단일 지점이 된다.
개별 서비스를 개별적으로 신뢰하던 사용자가
어느 순간 “이 모든 것을 한 번에 읽을 수 있는 하나의 앱”을
신뢰하는 구조로 자신도 모르게 이동하는 셈이다.
이는 비밀번호 관리자가 초기에 겪었던 신뢰 이전과 근본적으로
같은 패턴이며, 유틸리티가 여러 서비스를 가로질러 편의를
제공하려는 순간 항상 발생하는 구조적 긴장이다.
차이가 있다면 비밀번호 관리자는 처음부터 보안 도구로 설계되고
감사받는 반면, CodexBar 같은 생산성 유틸리티는 그런 기대치를
안고 출발하지 않는다는 점이다.

### 오픈소스 이식판의 급증은 플랫폼 종속 도구의 수명을 예고한다

Windows, Linux 데스크톱 환경, 상태 바 도구까지 CodexBar의 CLI를
기반으로 한 파생 프로젝트가 짧은 시간에 여러 개 등장한 현상은,
원 도구가 사실상 “사양”의 역할을 하고 각 플랫폼 커뮤니티가
자체적으로 프런트엔드만 새로 만드는 생태계로 진화하고 있음을
보여준다.
이런 구조에서는 CodexBar 본체의 CLI 인터페이스가 바뀌는 순간
다섯 개가 넘는 파생 프로젝트가 동시에 깨질 수 있다는 위험이
따라온다.
제공자가 늘어나고 API가 안정화될수록 원본 프로젝트는
사실상 비공식 표준 노릇을 하게 되지만,
그 표준을 유지·관리할 책임은 여전히 개인 개발자 한 명에게
남아 있다.

### AI 코딩 한도 추적 자체가 하나의 산업 신호다

Codex, Claude Code, Cursor를 비롯한 57개 제공자가
모두 사용량 한도와 리셋 주기를 갖고 있고,
그것을 추적하는 전용 메뉴바 앱이 만들어질 만큼 수요가 있다는
사실은, AI 코딩 도구 사용이 이미 “무제한처럼 느껴지지만
실제로는 세밀하게 계량되는” 자원 소비 모델로 굳어졌음을
보여준다.
개발자가 다음 작업을 시작하기 전에 리셋 시각을 확인해야 하는
상황은, 클라우드 컴퓨팅 초기에 사용자들이 인스턴스 과금을
의식하며 작업을 계획하던 패턴과 닮아 있다.
차이는 이번에는 계량 대상이 컴퓨팅 자원이 아니라
모델 추론이라는 점이며, 이 계량이 개발자의 작업 리듬 자체에
영향을 미치기 시작했다는 점에서 단순한 편의 도구를 넘어서는
산업적 신호로 읽을 수 있다.
