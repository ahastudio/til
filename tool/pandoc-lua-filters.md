# Pandoc Lua Filters

<https://pandoc.org/lua-filters.html>

HN 토론: <https://news.ycombinator.com/item?id=48773079> (141점, 12개 댓글)

## 소개

Lua 필터는 Pandoc 2.0부터 지원되는 문서 변환 훅이다.
Pandoc은 입력 문서를 파싱해 추상 구문 트리(AST)를 만들고,
이를 원하는 출력 형식으로 다시 쓰는 두 단계로 동작하는데,
Lua 필터는 그 사이에 끼어들어 AST를 직접 조작한다.
가장 큰 특징은 Lua 5.4 인터프리터가 Pandoc 실행 파일 안에 내장되어
있다는 점이다.
JSON 필터처럼 별도 프로세스를 띄우고 AST를 JSON으로 직렬화·역직렬화할
필요가 없으므로 외부 의존성 없이 바로 실행된다.

Pandoc 공식 매뉴얼을 HTML로 변환하는 벤치마크에서 이 구조적 차이가
드러난다.
필터 없는 순수 변환이 1.01초, Haskell 필터가 1.36초,
Python 필터가 1.40초인 데 비해 Lua 필터는 1.03초로 사실상 필터가
없는 경우와 차이가 없다.
JSON 마셜링 오버헤드를 건너뛴 결과다.

## 사용법

### 필터 함수 등록

Lua 필터는 요소 이름을 키로, 그 요소를 처리할 함수를 값으로 하는
테이블이다.

```lua
return {
  Strong = function (elem)
    return pandoc.SmallCaps(elem.content)
  end,
}
```

최상위 함수로 선언해도 동일하게 동작한다.

```lua
function Strong (elem)
  return pandoc.SmallCaps(elem.content)
end
```

필터 함수는 반드시 값을 반환해야 한다.
`nil`을 반환하면 요소가 그대로 유지되고,
같은 타입의 pandoc 객체를 반환하면 그 요소로 교체되며,
리스트를 반환하면 그 리스트가 부모 위치에 펼쳐져 들어간다.
빈 리스트를 반환하면 요소가 삭제된다.

### 순회 순서와 `walk` 메서드

기본 순회 방식은 `typewise`다.
Inline 함수들, `Inlines` 리스트 필터, Block 함수들, `Blocks` 리스트
필터, `Meta`, `Pandoc` 순서로 고정되어 처리된다.
`traverse = 'topdown'`으로 설정하면 루트에서 잎 노드 방향으로
깊이 우선 순회하며,
함수가 두 번째 반환값으로 `false`를 주면 해당 요소의 하위 내용을
더 이상 순회하지 않고 건너뛴다.

```lua
traverse = 'topdown'

function Note (n)
  return n, false
end
```

특정 요소 내부에만 필터를 적용하고 싶을 때는 `walk` 메서드를 쓴다.
`Pandoc`, `Block`, `Inline`, `Blocks`, `Inlines` 타입 모두에서
사용할 수 있다.

```lua
return pandoc.Para('Hi'):walk {
  Str = function (_) return 'Bye' end,
}
```

### 리스트 필터와 전역 변수

`Inlines`, `Blocks`라는 이름의 함수는 개별 요소가 아니라 인라인 또는
블록 리스트 전체를 인자로 받는다.
이 함수는 `nil`이나 올바른 타입의 리스트만 반환할 수 있고,
단일 요소를 반환하면 오류가 난다.

Pandoc은 필터 실행 환경에 몇 가지 전역 변수를 미리 채워둔다.
`FORMAT`은 출력 형식 문자열(`html5`, `latex` 등),
`PANDOC_VERSION`은 Pandoc 버전 객체,
`PANDOC_API_VERSION`은 AST API 버전,
`PANDOC_READER_OPTIONS`와 `PANDOC_WRITER_OPTIONS`는 각각 파서와
라이터에 전달된 옵션,
`PANDOC_STATE`는 읽기 전용 공유 상태,
`PANDOC_SCRIPT_FILE`은 현재 실행 중인 필터 파일 경로를 담는다.
문자열 패턴 매칭이 필요하면 `lpeg`, `re` 라이브러리도 기본 제공된다.

### 외부 프로그램 실행과 흔한 실수

`pandoc.pipe`를 쓰면 필터 안에서 외부 프로그램을 실행하고 그 출력을
받아올 수 있다.

```bash
sed -e 's/a/b/'
```

```lua
local output = pandoc.pipe("sed", {"-e", "s/a/b/"}, "abc")
```

가장 흔한 실수는 요소를 수정하고도 반환값을 빠뜨리는 것이다.

```lua
-- 틀린 예: 반환값 없음, 아무 효과도 없다
function Str (str)
  str.text = string.upper(str.text)
end

-- 올바른 예
function Str (str)
  str.text = string.upper(str.text)
  return str
end
```

또한 Lua 표준 `string` 라이브러리는 바이트 단위로 동작하므로
유니코드 문자열을 다룰 때는 `pandoc.text` 모듈을 대신 써야 한다.

## 분석

### JSON 필터와의 성능 격차는 아키텍처 차이에서 나온다

JSON 필터는 Pandoc이 AST를 JSON으로 직렬화해 표준 입력으로 넘기고,
필터 프로세스가 이를 파싱해 처리한 뒤 다시 JSON으로 돌려주는 구조다.
이 과정에서 프로세스 생성 비용, 직렬화·역직렬화 비용,
파이프 통신 비용이 누적된다.
Lua는 Pandoc 실행 파일 자체에 인터프리터로 내장되어 있으므로
이 모든 단계가 사라지고 AST를 메모리 상의 Lua 테이블로 직접 다룬다.
공식 벤치마크가 보여주는 1.03초 대 1.36~1.40초의 격차는 단순한
언어 성능 차이가 아니라 프로세스 경계를 없앴다는 구조적 이점에서
비롯된 것이다.

HN 토론에서는 이 벤치마크 자체보다 Pandoc이라는 도구 전체의 성격을
두고 논쟁이 벌어졌다.
`lifthrasiir`는 10년 전에 Lua 필터를 써봤던 경험을 떠올리며 지금의
문서가 그때와 딴판이고, Pandoc이 점점 비대해지고 있다는 인상을
받는다고 지적했다[^lifthrasiir].
이에 `zenoprax`는 "범용 문서 변환기"라는 태그라인 자체가 복잡성을
필연적으로 동반하며, Rclone이나 ZFS처럼 특정 목적을 위해 유닉스
철학을 깨는 도구들도 그 대가로 다른 곳에서 시너지를 얻는다고
반박했다[^zenoprax].
Lua 필터가 성능 손실 없이 AST 조작이라는 새 기능을 얹을 수
있었던 것도, 이 논쟁이 가리키는 것처럼 도구가 커지는 대신 실행
경로를 영리하게 설계해 복잡성을 국소화한 결과로 볼 수 있다.

이 설계는 필터 작성자에게도 실질적인 함의를 남긴다.
외부 프로세스 필터는 언어를 자유롭게 고를 수 있는 대신 배포
환경에 그 언어의 런타임이 있어야 하지만,
Lua 필터는 Pandoc 자체만 있으면 실행되므로 배포 단순성과 성능을
동시에 얻는다.
이는 Pandoc이 범용 문서 변환 도구로서 다양한 환경에 설치되어야
한다는 제약과 맞물려, Lua를 내장 언어로 고른 선택이 우연이 아니라
배포 전략의 연장선임을 보여준다.

### 두 가지 순회 모델은 서로 다른 필터 작성 습관을 요구한다

`typewise`가 기본값이라는 사실은 대부분의 필터가 “이 타입의
요소를 만나면 이렇게 바꿔라”는 국소적 규칙의 집합으로 작성된다는
가정을 반영한다.
타입별로 처리 순서가 고정되어 있으면 필터 작성자는 순회 경로를
신경 쓰지 않고 각 요소 타입에 대한 변환 규칙만 독립적으로 작성할 수
있다.

반면 `topdown`과 두 번째 반환값 `false`의 조합은 트리 구조 자체를
아는 필터를 요구한다.
주석(`Note`)처럼 내부에 별도의 블록 트리를 담는 요소를 다룰 때,
바깥쪽 함수가 이미 처리를 끝냈다면 안쪽까지 순회할 필요가 없다는
최적화 신호를 필터가 직접 판단해서 내려줘야 한다.
즉 `topdown`은 단순히 순서를 바꾸는 옵션이 아니라,
필터가 AST의 부모-자식 관계를 인식하고 순회를 능동적으로 제어할 수
있게 만드는 별도의 프로그래밍 모델에 가깝다.

### 리스트 필터는 문맥 의존적 변환을 위한 탈출구다

개별 요소 함수만으로는 “앞뒤 요소를 함께 봐야 판단할 수 있는” 변환을
표현하기 어렵다.
예를 들어 연속된 여러 개의 강조 인라인을 하나로 합치거나,
문단 경계를 넘나드는 패턴을 인식하는 작업은 단일 요소 단위 함수로는
불가능하다.
`Inlines`와 `Blocks`라는 리스트 전체를 받는 필터가 별도로 존재하는
이유가 여기에 있다.

이 설계는 Pandoc의 전체 필터 아키텍처가 “요소 단위 치환”이라는
가장 흔한 경우를 기본으로 최적화하면서도,
문맥이 필요한 소수의 경우를 위한 별도 확장 지점을 남겨둔 사례로
읽을 수 있다.
단일 인터페이스로 모든 경우를 욱여넣는 대신 두 개의 병렬 인터페이스를
제공한 것이다.

실사용 후기도 이 확장 지점이 어디서 가장 요긴하게 쓰이는지 보여준다.
`kalcode`는 Markdown을 DOCX로 변환할 때 생기는 문제를 Lua 필터로
해결해온 반면, PDF 쪽은 WeasyPrint로 엔진을 바꾼 뒤로는 필터가 거의
필요 없어졌다고 전했다[^kalcode].
이는 Lua 필터가 모든 출력 형식에 고르게 필요한 것이 아니라,
포맷별 변환 규칙이 유독 까다로운 지점(DOCX의 스타일·문단 처리 등)에
집중적으로 쓰인다는 실무적 신호로 읽을 수 있다.

## 비평

### “반환값을 빠뜨리면 조용히 실패한다”는 설계는 오류를 숨긴다

문서 자체가 별도 절로 이 실수를 경고할 만큼 흔한 함정이라는 점은
역설적으로 이 API 설계의 약점을 드러낸다.
`str.text = string.upper(str.text)`처럼 객체를 직접 변경한 뒤
반환을 빠뜨리면 Pandoc은 오류를 던지지 않고 그냥 원본 요소를 그대로
둔다.
즉 필터가 잘못 작성되어도 실행은 성공하고, 결과물만 조용히
기대와 달라진다.
런타임 경고나 정적 검사 없이 “문서에 적어뒀으니 조심하라”는 방식으로
문제를 떠넘기는 것은 사용자 실수를 언어·API 차원에서 방지하지 못한
설계의 한계를 그대로 인정하는 셈이다.

### `pandoc.pipe`는 Lua 필터의 무의존성이라는 장점을 스스로 깬다

Lua 필터의 핵심 매력은 외부 프로세스나 런타임 없이 Pandoc 하나만
있으면 필터가 동작한다는 점이다.
그런데 `pandoc.pipe`로 `sed` 같은 외부 프로그램을 호출하는 순간
이 장점은 사라진다.
필터가 특정 외부 명령에 의존하게 되면, 그 필터를 다른 환경에
배포할 때 다시 그 외부 프로그램의 존재 여부와 버전 차이를
신경 써야 하는, JSON 필터가 안고 있던 것과 동일한 배포 복잡성이
되돌아온다.
문서는 이 기능을 유용한 확장으로만 소개할 뿐,
이것이 애초에 Lua 필터를 선택한 이유(외부 의존성 제거)와 정면으로
충돌한다는 점은 짚지 않는다.

### `typewise`와 `topdown`의 차이를 이해하지 않고 섞어 쓰면 순서 버그가 난다

문서는 두 순회 모드를 각각 독립적인 옵션처럼 설명하지만,
실제로는 같은 필터 파일 안에 `Str`, `Emph` 같은 개별 함수와
`Inlines` 리스트 함수를 동시에 두었을 때 그 실행 순서가
`traverse` 값에 따라 완전히 달라진다.
`typewise`에서는 개별 함수들이 다 끝난 뒤 리스트 필터가 실행되지만,
`topdown`에서는 순회 경로 자체가 달라지므로 같은 이름의 함수 조합이
다른 결과를 낼 수 있다.
필터를 작성하다가 `traverse` 옵션 하나를 바꾸는 것만으로 기존에
잘 동작하던 다른 함수들의 상호작용이 깨질 수 있는데,
이는 옵션이 단순 설정값이 아니라 실행 모델 자체를 바꾸는 것이라는
점에서 문서의 안내만으로는 충분히 예측하기 어렵다.

## 인사이트

### 언어 내장형 필터는 확장 생태계의 파편화를 낳을 수 있다

JSON 필터 시절에는 어떤 언어로든 필터를 작성할 수 있었으므로
Python, Haskell, JavaScript 등 각 커뮤니티가 자기 언어로 필터를
만들고 공유했다.
Lua가 사실상 표준 확장 언어로 자리 잡으면 성능과 배포 단순성은
얻지만, 다른 언어에 익숙한 사용자는 새로 Lua 문법과 API를 익혀야
하는 진입 장벽이 생긴다.
이는 결과적으로 필터 생태계가 다양한 언어 공동체로 분산되는 대신
Lua API에 익숙한 소수의 관리자와 저장소로 수렴하게 만드는
2차 효과를 낳을 가능성이 있다.
문서가 성능 수치로 Lua 필터의 우위를 강조할수록, 신규 필터
작성자는 성능을 이유로 Lua를 선택할 유인이 커지고
JSON 필터 생태계는 점점 레거시로 밀려날 수 있다.

### 내장 스크립팅 언어를 고르는 문제는 확장 가능한 도구 전반의 오래된 패턴이다

애플리케이션에 스크립팅 언어를 내장하는 결정은 Pandoc만의 것이
아니다.
Nginx가 LuaJIT을 임베드해 OpenResty 생태계를 만든 사례,
Redis가 Lua를 스크립팅 언어로 채택한 사례,
게임 엔진들이 Lua를 즐겨 쓰는 사례 모두 “가볍고 임베드하기 쉬우며
C로 작성된 호스트 프로그램과 자연스럽게 맞물리는 언어”라는 동일한
요구에서 나온 선택이다.
Pandoc이 Haskell로 작성되어 있음에도 필터 언어로는 Haskell이 아닌
Lua를 택했다는 사실은, 임베드 언어를 고르는 기준이 호스트 언어와의
친화성이 아니라 “가볍고 빠르게 내장할 수 있는가”라는 별도의
축이라는 점을 보여준다.
이는 도구 설계자가 핵심 구현 언어와 확장 언어를 분리해서 사고해야
한다는 일반 원칙의 구체적 사례다.

### 조용한 실패를 허용하는 API는 결국 커뮤니티 문서로 그 부담을 떠넘긴다

반환값 누락이라는 흔한 실수를 공식 문서가 별도 절로 다룬다는 것은,
언어·API 차원에서 해결하지 못한 문제를 문서화로 메우고 있다는
신호다.
이런 패턴은 Lua 필터에 국한되지 않는다.
정적 타입이 없는 언어로 확장 API를 설계할 때는 거의 예외 없이
“흔한 실수” 절이 문서에 등장하고, 시간이 지나면 커뮤니티 위키나
블로그 글이 그 역할을 대신 떠맡는다.
장기적으로 이는 API 설계 자체의 결함을 사용자 교육 비용으로
전가하는 구조이며, Pandoc Lua 필터 생태계가 성숙할수록 타입
어노테이션이나 린터 같은 별도 도구가 등장해 이 공백을 메우려 할
가능성이 크다.

### AST를 반응형으로 다루려는 시도는 필터 모델의 다음 단계를 암시한다

HN 댓글 중에는 지금의 일회성 변환 모델을 넘어서려는 아이디어도
있었다.
`chaoxu`는 Markdown을 Pandoc AST로 바꾼 뒤, 원본이 바뀌면 AST가
전체를 다시 만들지 않고 증분으로만 갱신되는 반응형 구조가 가능하지
않을지 궁금해했고, 최근의 LLM 도구라면 이런 구조를 실제로 시도해볼
수 있겠다고 덧붙였다[^chaoxu].
현재 Lua 필터는 문서 전체를 한 번에 순회하며 변환하는 일회성
파이프라인을 전제하는데, 편집기에서 실시간 미리보기를 갱신하거나
대규모 문서에서 일부만 바뀐 경우를 반복 변환하는 상황에서는 이
전제가 비효율의 근원이 된다.
`typewise`와 `topdown`이라는 두 순회 모델이 이미 AST를 다루는
방식에 대한 선택지를 열어둔 것처럼, 증분 갱신이라는 요구가 쌓이면
필터 API 자체가 “매번 전체를 순회하는 함수”에서 “변경된 부분만
다시 호출되는 함수”로 진화할 여지가 있다.
이는 이 문서가 다루는 현재의 Lua 필터 API가 고정된 종착점이 아니라
Pandoc의 AST 처리 모델이 앞으로 나아갈 수 있는 여러 방향 중 하나의
스냅샷에 불과하다는 점을 보여준다.
