# Jina Reader API — URL을 LLM 친화적 마크다운으로 변환

<https://jina.ai/reader/>

## 소개

Jina Reader API는 URL을 LLM이 소비하기 좋은 마크다운으로 변환하는 개발자 인프라다.
URL 앞에 `r.jina.ai/`를 붙이는 것만으로 웹페이지 본문이 정제된 마크다운으로 반환된다.

```bash
curl https://r.jina.ai/https://example.com/article
```

웹 검색도 지원한다.
`s.jina.ai`에 검색어를 보내면 상위 5개 결과를 마크다운으로 받는다.

```bash
curl https://s.jina.ai/LLM+agent+tool+calling
```

PDF, 동적 JavaScript 렌더링 콘텐츠, 로컬 HTML 파일도 처리한다.
CSS 선택자로 원하는 부분만 추출하거나, 커스텀 JavaScript를 실행해 페이지를 조작한 뒤 추출할 수 있다.
이미지는 자동으로 캡션이 달린다.

## API 엔드포인트

| 엔드포인트 | 용도 |
| ----------------------- | ------------------------------ |
| `https://r.jina.ai`     | URL 콘텐츠 → 마크다운          |
| `https://s.jina.ai`     | 웹 검색 결과 → 마크다운        |
| `https://api.jina.ai/v1/embeddings` | 텍스트 임베딩     |
| `https://api.jina.ai/v1/rerank`     | 문서 재랭킹       |

## 요금 및 제한

신규 사용자에게 1000만 토큰을 무료로 제공한다.
무료 티어는 Reader API 기준 20 RPM, 검색 API는 요청당 최소 10,000 토큰이 소비된다.

유료 티어는 토큰 기반 종량제로, Stripe를 통해 신용카드·Google Pay·PayPal을 지원한다.
유료 API 키는 500 RPM / 2M TPM, 프리미엄 키는 5,000 RPM / 50M TPM이다.

5분 이내 동일 URL 요청은 캐시에서 반환된다.
동시 처리 용량은 약 4,000 요청이다.

API 요청 내용은 모델 학습에 사용되지 않는다고 명시한다.
안티봇 시스템 우회는 지원하지 않는다 — “웹 표준 클라이언트로 동작”한다는 것이 공식 입장이다.

## 분석

### URL 프리픽스 하나로 RAG 파이프라인의 크롤링 단계를 대체한다

RAG 시스템을 구축할 때 가장 번거로운 부분 중 하나는 웹 콘텐츠 수집이다.
Playwright나 Puppeteer로 동적 페이지를 렌더링하고, BeautifulSoup으로 본문을 추출하고,
마크다운으로 변환하는 파이프라인은 유지보수 비용이 높다.

Jina Reader는 이 과정을 URL 프리픽스 하나로 대체한다.
`r.jina.ai/` 앞에 붙이면 끝이다.
에이전트가 웹 페이지를 읽어야 할 때 별도의 크롤러 없이 도구 하나로 해결된다.
웹 검색 결과를 컨텍스트에 넣는 것도 `s.jina.ai`로 단순화된다.

### 동일 API 키로 임베딩과 재랭킹까지 연결하는 수직 통합

Reader API 키가 임베딩(`api.jina.ai/v1/embeddings`)과 재랭킹(`api.jina.ai/v1/rerank`)에도 쓰인다.
RAG의 세 단계 — 수집, 임베딩, 검색 후 재랭킹 — 를 하나의 공급업체와 API 키로 처리하는 구조다.

이것은 단순한 편의 기능이 아니라 잠금(lock-in) 전략이다.
한 API 키가 파이프라인 전체에 스며들면 공급업체를 바꾸는 비용이 높아진다.
Jina가 Reader를 무료로 넉넉하게 제공하는 것은 임베딩·재랭킹 유료 사용자를 확보하기 위한 유인책일 수 있다.

### 프리픽스 방식은 기존 코드 수정 없이 통합된다

`fetch(“https://r.jina.ai/” + url)` 한 줄이면 된다.
SDK 설치, 라이브러리 임포트, 설정 파일이 필요 없다.
기존 에이전트 코드에 URL 프리픽스만 추가하면 통합이 완료된다.

이 설계는 Cloudflare Workers의 `fetch` API 활용 패턴과 유사하다.
복잡한 추상화 없이 HTTP 요청 하나로 기능을 제공하는 방식이다.
LLM 도구 정의에서 웹 검색 도구를 `s.jina.ai/{query}`로 단순하게 구현할 수 있다.

## 비평

### 안티봇 우회 미지원은 실제 사용 범위를 좁힌다

공식 입장은 “웹 표준 클라이언트로 동작하며 안티봇 시스템을 우회하지 않는다”는 것이다.
Cloudflare, Akamai, Imperva 같은 봇 방어 시스템이 붙은 사이트에서는 콘텐츠 추출이 실패한다.

실제 웹에서 가장 가치 있는 콘텐츠들 — 뉴스 사이트, 전문 데이터베이스, 금융 정보 플랫폼 — 은
대부분 강력한 봇 방어를 적용하고 있다.
이 제약이 있는 한 “모든 URL을 LLM 친화적으로”라는 약속은 공개 웹의 일부에만 성립한다.

### 5분 캐싱은 실시간 정보가 필요한 용도에 맞지 않는다

같은 URL 요청이 5분 내에 반복되면 캐시가 반환된다.
뉴스 페이지, 실시간 가격 데이터, 라이브 스포츠 결과처럼 빠르게 바뀌는 콘텐츠를 다루는 에이전트에서는
캐시된 결과가 이미 오래된 정보일 수 있다.

캐시 무효화 옵션이 제공되는지, `no-cache` 헤더가 동작하는지 문서에 명확하지 않다.
실시간 데이터를 다루는 용도에서는 이 제약이 설계상 결함이 된다.

### 토큰 계산 기준이 불투명하다

검색 API는 “요청당 최소 10,000 토큰”이 소비된다.
그런데 토큰을 “출력 응답 토큰 기준으로 계산”한다는 설명과 “최소 10,000 토큰” 사이에 불일치가 있다.

짧은 응답이 나와도 10,000 토큰이 차감되는 것인지,
응답 길이에 따라 달라지는 것인지 분명하지 않다.
비용 예측이 어렵고, 대규모 운영 환경에서 예상치 못한 청구가 발생할 가능성이 있다.

## 인사이트

### 웹 접근 도구가 에이전트 인프라의 필수 레이어가 된다

에이전트가 실제 세계와 상호작용하려면 웹을 읽어야 한다.
Jina Reader 같은 서비스는 이 필요를 “클롤러를 직접 구현하라”에서 “API를 호출하라”로 바꾼다.

이 전환이 중요한 이유는 에이전트 구성 방식을 바꾸기 때문이다.
이전에는 웹 읽기 능력이 에이전트 내부에 구현되어야 했다.
이제는 도구 정의 하나로 외부에서 가져올 수 있다.
에이전트 개발이 인프라 구축에서 도구 조합으로 이동하는 방향을 Jina Reader가 가속한다.

### 프리픽스 API 패턴은 기존 인터넷 인프라와 정합성이 높다

`r.jina.ai/https://...` 형식은 프록시와 같은 구조다.
HTTP 프록시, CDN, 역방향 프록시가 원본 URL을 변환 없이 통과시키는 것처럼,
Jina Reader는 원본 URL을 마크다운 변환 프록시를 통과시킨다.

이 패턴의 장점은 기존 HTTP 클라이언트, 에이전트 프레임워크, 도구 정의와 즉각 호환된다는 것이다.
새 SDK나 라이브러리 없이 URL 문자열만 바꾸면 된다.
프리픽스 패턴이 충분히 퍼지면, AI 에이전트가 웹을 읽는 사실상 표준 방식이 될 수 있다.

### 콘텐츠 추출 품질이 RAG 전체 품질의 천장을 결정한다

RAG 파이프라인에서 임베딩과 재랭킹 알고리즘이 아무리 정교해도,
입력 텍스트 품질이 낮으면 검색 결과는 나빠진다.
Jina Reader의 마크다운 변환 품질 — 불필요한 내비게이션 삭제, 본문 추출 정확도, 이미지 캡션의 적절성 — 이
그 위에 쌓이는 임베딩·재랭킹의 성능 상한을 결정한다.

Jina가 Reader와 임베딩·재랭킹을 수직 통합하는 것은 이 점에서 합리적이다.
수집 품질이 낮으면 임베딩 성능을 제어하기 어렵고, 수집 품질이 높으면 임베딩·재랭킹의 가치가 명확해진다.
공급업체로서 파이프라인 전체를 제어하면 품질 책임이 분산되지 않는다.
