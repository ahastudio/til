# Clypra

<https://github.com/AIEraDev/Clypra>

## 소개

Clypra는 “전문적인 영상 편집을 무료로, 그리고 영구적으로 오픈소스로”라는
슬로건을 내건 크로스플랫폼 영상 편집기다.
Tauri v2, React 19, Rust, FFmpeg를 조합해 만들어졌으며
GitHub Trending에서 2.7k(README 기준 2.8k) 스타, 284개 포크를 기록하며
주목받고 있다.
라이선스는 MIT이고 핵심 편집기, 이펙트 엔진, 오디오 도구,
익스포트 파이프라인은 워터마크나 기능 제한 없이 전부 무료로 제공된다.

핵심 편집 기능은 MP4, MOV, WebM, MKV 등 다양한 영상 포맷과
MP3, WAV 같은 오디오, JPG, PNG 같은 이미지를 아우르는
멀티포맷 미디어 임포트, 밀리초 단위의 프레임 정확 트리밍,
멀티트랙 타임라인, 100단계 히스토리 스택을 갖춘 언두/리도를 포함한다.
오디오 쪽에서는 피크와 RMS를 함께 보여주는 고충실도 파형 표시와
프레임 정확한 AV 싱크를 지원하고,
시각 효과 쪽에서는 하드웨어 가속 필름스트립 썸네일과
커스텀 폰트·스타일·애니메이션을 적용할 수 있는 텍스트 오버레이를 제공한다.

성능 최적화가 README에서 특히 강조되는 대목이다.
FFmpeg를 통한 GPU 디코딩(macOS의 VideoToolbox, Windows의 D3D11VA,
Linux의 VAAPI)으로 하드웨어 가속을 적용하고,
디코더를 미리 초기화해 두어 프로젝트 로드 시 첫 프레임 지연을
10ms 이내로 줄인다고 설명한다.
썸네일 생성은 웹 워커 풀로 병렬화해 렌더링 속도를 2~4배 높이고
메인 스레드 CPU 사용량을 60% 낮췄다고 주장하며,
LRU 기반으로 최대 20개의 동시 디코더를 유지하는 캐싱 풀과
30종 이상의 성능 지표를 실시간으로 추적하는 모니터링도 갖췄다.

아키텍처는 프론트엔드(React 19 + TypeScript strict mode, Zustand,
Vite)와 백엔드(Rust + Tauri v2 + FFmpeg 바인딩인 ffmpeg-next,
동시성 자료구조 DashMap)로 나뉜다.
프론트엔드의 타임라인 UI, 프리뷰 캔버스, 필름스트립 캐시는
Tauri IPC를 거쳐 백엔드의 디코더 풀과 익스포트 파이프라인으로 연결된다.
설계 원칙으로는 브라우저 제약을 없앤 네이티브 성능,
데스크톱을 우선하되 Capacitor를 통해 iOS·Android로 이식 가능한 구조,
네이티브 FFmpeg 하드웨어 디코더에 직접 접근하는 방식,
WebCodecs나 MSE 같은 브라우저 API에 의존하지 않는 점을 내세운다.
소스 코드는 `src/`(컴포넌트, 도메인별 Zustand 스토어, 엔진 로직)와
`src-tauri/`(Tauri 명령, FFmpeg 디코더 풀을 담은 thumbnail_engine)로
나뉘어 있고, 익스포트 파이프라인은 `src-tauri/commands/export.rs`에서
H.264, H.265, ProRes 포맷을 처리한다.

비즈니스 모델은 오픈 코어 방식이다.
핵심 편집기는 MIT 라이선스로 무료 공개하되,
자연어 편집(“일시정지 제거”, “자막 추가” 같은 명령),
자동 자막, 인스타그램·틱톡·유튜브 포맷에 맞춘 스마트 리프레임,
장면 감지, 오디오 향상, 향후 출시 예정인 음성 복제와 다국어 더빙 같은
AI 기능은 Pro 요금제($10/월, 무료 티어는 월 100회 AI 호출)로 유료화한다.
설치는 macOS는 Homebrew나 유니버설 DMG, Windows는 MSI,
Linux는 AppImage로 배포되며,
소스 빌드에는 Node.js 18+, Rust 1.70+, FFmpeg 6.0+ 개발 라이브러리가
필요하다.
언어 구성은 TypeScript 91.4%, Rust 7.3%가 대부분을 차지한다.

## 분석

### 브라우저 회피가 아키텍처 전체를 규정한다

Clypra README가 가장 힘주어 설명하는 지점은 “브라우저 무의존”이라는
설계 원칙이다.
WebCodecs나 MediaSource Extensions 같은 브라우저 API를 쓰지 않고
Rust와 FFmpeg로 직접 디코딩 파이프라인을 구축한 이유는,
웹 기반 영상 편집기가 브라우저 샌드박스와 코덱 지원 범위,
메모리 한계에 묶여 있다는 문제의식에서 출발한다.
이 선택은 단순한 기술 취향이 아니라 프로젝트의 정체성 자체를 규정한다.
Tauri를 골라 데스크톱 셸 위에 네이티브 프로세스를 얹은 것,
디코더 풀을 Rust 쪽에 두고 LRU 캐시로 관리하는 것,
하드웨어 가속을 플랫폼별 API로 직접 붙인 것 모두
“브라우저를 거치지 않겠다”는 하나의 결정에서 파생된 하위 결정들이다.

이 구조는 성능 지표로 뒷받침된다.
콜드 스타트 10ms, CPU 사용량 60% 감소, 렌더링 2~4배 향상 같은 수치는
브라우저 API를 우회했을 때 실제로 어떤 이득이 나는지를 구체적으로
보여주려는 시도다.
동시에 이 수치들이 벤치마크 방법론이나 비교 대상 없이 제시된다는 점은
분석의 여지를 남긴다.
어떤 브라우저 기반 편집기와 비교했는지, 어떤 하드웨어에서 측정했는지가
README에 명시되지 않은 채 결과만 나열되어 있다.

### 오픈 코어 모델이 로드맵의 우선순위를 결정한다

핵심 편집 기능은 완전히 무료로 공개하면서 AI 기능만 유료화하는 구조는
최근 오픈소스 개발 도구들이 흔히 채택하는 패턴이다.
Clypra의 로드맵을 보면 이 패턴이 실제 개발 순서에도 반영되어 있다.
데스크톱 쪽에서는 모바일 앱, 색 보정, 멀티카메라 편집,
플러그인 시스템이 계획되어 있는 반면,
Pro AI 기능 쪽은 2026년 3분기부터 2027년 2분기까지
자연어 명령·자동 자막·스마트 리프레임·음성 복제·다국어 더빙 순으로
분기별 일정이 구체적으로 잡혀 있다.
무료 기능의 로드맵은 상대적으로 느슨하고,
수익화와 직결되는 AI 기능의 로드맵만 촘촘하다는 비대칭은
오픈 코어 모델에서 “코어”가 어떤 위치에 놓이는지를 보여준다.

### FFmpeg 라이선싱이 배포 방식을 제약한다

README는 FFmpeg 사용에 따른 라이선스 조건을 별도로 명시한다.
기본 빌드는 LGPL 2.1 이상을 따르고,
GPL 전용 컴포넌트를 포함하면 GPL 2 이상으로 전환된다는 설명이다.
바이너리 배포판은 LGPL 조건을 따른다고 밝히고 있는데,
이는 Clypra 자체가 MIT 라이선스를 표방하더라도
실제 배포 바이너리의 법적 조건은 의존성인 FFmpeg의 빌드 방식에
좌우된다는 뜻이다.
영상 편집기처럼 코덱 의존도가 높은 소프트웨어에서
“오픈소스”라는 라벨이 단일 라이선스로 깔끔하게 정리되지 않는다는
사실을 이 프로젝트가 보여준다.

## 비평

### 성능 수치는 검증 가능한 형태로 제시되지 않는다

콜드 스타트 10ms, CPU 60% 감소, 렌더링 2~4배 향상이라는 수치들은
모두 README 안에서만 존재하고 외부에서 재현할 방법이 없다.
벤치마크 스크립트나 측정 환경, 비교 기준선이 공개되어 있지 않기 때문에
이 수치들은 사실상 마케팅 문구와 동일한 신뢰 수준을 갖는다.
오픈소스 프로젝트가 성능을 핵심 차별점으로 내세울 때는
벤치마크 재현성이 코드 공개 자체만큼 중요한데,
Clypra는 코드는 공개하면서도 성능 주장의 근거는 공개하지 않는
비대칭적인 투명성을 보이고 있다.
스타 수가 2.7k에 달할 정도로 관심을 받았다는 사실이
이 수치들의 진위를 검증했다는 뜻은 아니다.

### 크로스플랫폼과 모바일 지원은 선언과 현실 사이에 거리가 있다

README는 “크로스플랫폼(데스크톱 + 모바일)”을 핵심 특징으로 내세우지만,
로드맵을 보면 모바일 앱 출시는 아직 계획 단계다.
언어 구성에서 Swift 비중이 0.1%에 불과하다는 점도
iOS 지원이 초기 단계를 넘어서지 않았음을 시사한다.
Capacitor를 통해 모바일로 이식 가능한 구조라는 설계 원칙과
실제로 모바일에서 동작하는 제품 사이에는 상당한 거리가 있고,
현재 시점의 Clypra는 사실상 데스크톱 전용 편집기에 가깝다.
“크로스플랫폼”이라는 표현이 이미 갖춘 능력이 아니라
지향하는 방향을 가리키고 있다는 점은 사용자가 혼동하기 쉬운 지점이다.

### 무료 티어의 지속 가능성은 검증되지 않은 가정에 기대고 있다

핵심 편집 기능을 워터마크나 제한 없이 완전히 무료로 제공하면서
Pro AI 기능만으로 수익을 낸다는 모델은
AI 호출 비용이 충분히 낮거나 유료 전환율이 충분히 높다는 가정을
전제로 한다.
자연어 편집이나 음성 복제, 다국어 더빙 같은 기능은
외부 AI 모델 호출 비용이 발생할 가능성이 높은 영역인데,
월 100회라는 무료 티어의 한도가 실제 운영 비용과
얼마나 맞아떨어지는지는 README만으로 알 수 없다.
오픈 코어 모델을 표방한 프로젝트가 핵심 기능을 무료로 유지하면서도
장기적으로 수익을 내지 못하면, 결국 무료 기능의 유지보수 우선순위가
떨어지거나 유료 기능 쪽으로 개발 자원이 쏠리는 결과로 이어질 수 있다.
현재 README는 이 지속 가능성에 대한 근거를 제시하지 않는다.

### 커뮤니티 참여 구조가 프로젝트 성장 경로를 좁힌다

이슈 12개, 커밋 1,026개라는 수치는 스타 2.7k에 비해
저장소 자체의 활동량이 상대적으로 조용하다는 인상을 준다.
기여나 논의 채널이 GitHub Issues, Discussions, Wiki로 한정되어 있고
별도의 기여 가이드라인이 두드러지게 제시되어 있지 않다는 점도
외부 기여자가 프로젝트에 참여하는 진입 장벽을 낮추지 못한다.
스타 수로 대표되는 관심과, 실제 코드베이스에 기여자가 늘어나는
성장 경로는 별개의 문제인데, Clypra는 아직 전자에서 후자로
넘어가는 지점을 명확히 보여주지 않는다.

## 인사이트

### 오픈 코어 영상 편집기는 코덱 라이선스라는 별도의 규제 층을 감당해야 한다

일반적인 오픈소스 SaaS 도구는 라이선스 설계가 비즈니스 모델의 문제로
끝나지만, 영상 편집기는 여기에 코덱 라이선스라는 층이 하나 더 얹힌다.
FFmpeg의 LGPL·GPL 이원 구조, H.264나 H.265 같은 코덱 자체에 걸린
특허 라이선스 문제는 Clypra 팀이 통제할 수 없는 외부 제약이다.
오픈 코어 모델을 설계할 때 보통은 “어떤 기능을 무료로 둘 것인가”만
고민하면 되지만, 영상 편집 도구는 여기에 더해
“이 배포판이 어떤 코덱 라이선스 조건 아래 놓이는가”까지 함께
설계해야 한다.
이 이중 제약은 향후 Clypra가 특정 국가에서 배포하거나
특정 코덱을 기본값으로 바꿀 때마다 반복해서 마주칠 문제이고,
README에 명시된 라이선스 설명은 그 복잡성의 일부만 드러낸 것에 가깝다.

### 성능 벤치마크의 비검증성은 GitHub Trending 생태계의 구조적 유인에서 나온다

Clypra만의 문제가 아니라, GitHub Trending에 오르는 프로젝트들이
공통으로 마주하는 유인 구조가 있다.
스타 수는 README 첫인상에 크게 좌우되고,
README는 구체적인 숫자를 담을수록 신뢰감을 주는 것처럼 보인다.
이 유인은 실제 벤치마크를 재현 가능하게 공개하는 노력보다
인상적인 숫자를 먼저 적어 넣는 쪽으로 프로젝트를 이끈다.
결과적으로 트렌딩 저장소의 성능 주장은 검증 절차 없이
그대로 확산되는 경우가 많고, 이는 개별 프로젝트의 정직성 문제라기보다
플랫폼이 만들어낸 인센티브 설계의 부산물에 가깝다.
Clypra의 수치들을 비판적으로 읽어야 하는 이유도
이 프로젝트가 유독 과장이 심해서가 아니라
같은 유인 구조 안에 있는 모든 프로젝트에 동일하게 적용되는 회의가
필요하기 때문이다.

### 브라우저 회피 전략은 웹 표준의 성숙도에 대한 시차 있는 반응이다

Clypra가 WebCodecs와 MSE를 의도적으로 피한 선택은
현재 시점에서는 합리적이지만, 이 판단은 웹 표준이 정지된 상태를
전제로 한다.
WebCodecs는 아직 발전 중인 API이고, 하드웨어 가속 디코딩 지원 범위도
브라우저와 플랫폼마다 계속 넓어지고 있다.
과거 데스크톱 앱들이 “네이티브가 웹보다 빠르다”는 이유로
Electron이나 웹 기술을 피했다가, 몇 년 뒤 웹 표준이 따라잡으면서
그 선택의 근거가 약해진 사례는 여러 차례 반복되었다.
Clypra의 네이티브 우선 전략도 같은 시차 문제를 안고 있다.
지금은 브라우저 API의 한계가 명확하지만,
Tauri와 Rust로 쌓아 올린 아키텍처는 웹 표준이 따라잡았을 때
쉽게 되돌릴 수 없는 매몰 비용이 된다.
장기적으로 이 프로젝트가 마주할 질문은 “브라우저를 피할 것인가”가
아니라 “언제 다시 브라우저로 돌아갈 여지를 열어둘 것인가”에
가까울 수 있다.
