# Nativ — Mac에서 오픈 모델을 로컬로 실행하는 앱

<https://blaizzy.github.io/nativ/>

<https://github.com/Blaizzy/nativ>

HN 토론: <https://news.ycombinator.com/item?id=48982681> (371점, 128개 댓글)

## 소개

Nativ는 Mac에서 오픈 웨이트 AI 모델을 로컬로 실행하는 오픈소스 애플리케이션이다.
클라우드 의존성, 계정, 구독 없이 동작하며, Apple Silicon에 최적화된 MLX-VLM 프레임워크 위에 구축됐다.

언어, 비전, 비디오, 코드, 오디오를 하나의 앱에서 지원한다.
실시간으로 토큰/초, 메모리 사용량, 열 상태, 첫 토큰 도달 시간을 측정해 표시한다.
MIT 라이선스로 공개됐다.

개발자 Prince Canuma는 LM Studio 등 여러 앱이 의존하는 MLX-VLM 라이브러리의 메인테이너이기도 하다.[^erikkahler]
앱 본체는 주로 Swift로 작성됐으며, 이 구조는 향후 iPad·iPhone 포팅을 용이하게 한다.[^erikkahler]

지원 모델 예시:

| 모델 | 제공사 | 크기 | 컨텍스트 | 기능 |
| ---- | ------ | ---- | -------- | ---- |
| Gemma 4 E2B | Google | 10.28 GB | 128K | 비전 + 오디오 |
| North Mini Code | Cohere | 19.38 GB | 500K | 코드 + 도구 |
| LFM2.5-VL 1.6B | Liquid AI | 3.20 GB | 128K | 비전 + 언어 |

M1 이상의 Apple Silicon이 필요하고, Metal 가속을 사용한다.
번역 레이어 없이 M 시리즈 통합 메모리에 직접 최적화됐다.

## 분석

### 로컬 AI 도구의 분화: 속도 vs 프라이버시

로컬 모델 실행 도구들이 두 가지 방향으로 분화되고 있다.
하나는 Ollama처럼 CLI 중심으로 개발자가 직접 제어하는 방식이고,
다른 하나는 Nativ처럼 일반 사용자를 대상으로 설치하고 실행하는 경험을 단순화하는 방식이다.

Nativ는 후자의 방향이다.
CLI 없이 앱을 설치하고 모델을 선택해 바로 쓸 수 있는 경험을 제공한다.
이것은 클라우드 AI 앱과 같은 사용자 경험을 로컬에서 재현하려는 시도다.

이 접근의 핵심 가치는 프라이버시다.
데이터가 외부로 나가지 않고, 오프라인에서 동작하며, API 키 관리가 필요 없다.
이 가치를 데스크톱 앱 형태로 쉽게 접근하게 만드는 것이 Nativ의 포지션이다.

### Apple Silicon이 만든 로컬 AI의 현실적 가능성

M 시리즈 칩의 통합 메모리 아키텍처는 CPU와 GPU가 같은 메모리 풀을 공유한다.
이것이 로컬 AI 실행에서 의미하는 바는 크다.
128GB 통합 메모리를 가진 MacBook Pro는 대형 모델도 램에 올릴 수 있고,
GPU와 CPU 사이의 데이터 전송 지연이 없다.

Ollama 같은 도구들이 이미 이 생태계를 개척했고,
LM Studio, Jan 같은 GUI 도구들이 뒤따랐다.
Nativ는 이 흐름에서 멀티모달 지원과 성능 모니터링에 초점을 맞춘 포지션이다.

MLX 지원 자체를 가장 핵심 차별점으로 보는 시각도 있다.
대부분의 로컬 실행 도구가 여전히 GGUF 포맷에 머물러 있어서, MLX를 기본으로 지원하는 도구는 아직 드물다.[^satvikpendem]
Nativ를 계기로 로컬 AI 러너 시장에 경쟁자가 늘어나는 것을 환영하는 목소리도 있다.[^rvz]

## 비평

### LM Studio, Jan 같은 경쟁 도구와의 차별점이 불명확하다

Nativ가 공개됐을 때 HN에서 자연스럽게 나온 질문은 “LM Studio나 Jan과 무엇이 다른가”였다.
멀티모달 지원, 성능 모니터링, MLX 최적화가 차별점으로 제시되지만,
LM Studio도 이미 비슷한 기능을 제공한다.

새로운 도구가 성장하려면 기존 도구에서 반복적으로 불편한 점을 해결하거나,
완전히 다른 사용자 집단을 겨냥해야 한다.
현재 공개된 내용으로는 이 차별점이 명확하지 않다.

D13Fd는 랜딩 페이지가 LM Studio나 Open WebUI 같은 기존 도구를 아예 언급하지 않는 점을 지적했다.[^D13Fd]
"Native"라는 이름을 내세우면서도 Python 런타임을 통째로 번들링하는 아이러니를 꼬집는 의견도 있었다.
다만 Electron까지 포함한 LM Studio보다는 덜 무겁다는 점에서 상대적 우위는 인정받는다.[^woadwarrior01]

### 모델 크기와 실용성 사이의 간극

19.38 GB짜리 North Mini Code 모델을 500K 컨텍스트로 실행하려면
최소 32GB 이상의 통합 메모리가 필요하다.
이 요구사항을 충족하는 Mac은 상위 라인업에 한정되고, 가격이 높다.

“로컬 AI”의 진입 장벽이 소프트웨어에서 하드웨어로 이동했다.
이것은 Nativ만의 문제가 아니라 로컬 AI 전반의 구조적 제약이지만,
앱이 어떤 모델을 기본 추천하고 어떤 하드웨어 요구사항을 명시하는지가 사용자 경험을 좌우한다.

Gemma 4 E2B처럼 작은 모델도 실제로 쓸 만한지에 대한 의구심도 있다.
Gemma 4 12B는 충분히 유용하지만 E4B로만 내려가도 기본적인 작업에서 실패한다는 경험이 공유됐다.[^Daunk]
18GB 통합 메모리의 M3 Pro에서도 기본 모델 실행 시 팬이 돌고 기기가 뜨거워지는 문제를 호소하는 사용자도 있었다.[^sajithdilshan]

## 인사이트

### 로컬 AI는 프라이버시 민감 업무의 기본 선택지가 된다

지금까지 로컬 AI 실행은 “클라우드 API가 너무 비쌀 때”의 대안이었다.
앞으로는 반대가 될 수 있다.
개인 파일, 회사 코드, 의료 데이터처럼 외부로 보내기 어려운 정보를 다루는 업무에서
로컬 실행이 기본값이 되고, 클라우드 API가 대안이 되는 방향이다.

이 전환이 일어나려면 두 가지가 필요하다.
로컬에서 실행 가능한 모델의 품질이 충분히 높아야 하고,
로컬 실행 환경이 클라우드 앱만큼 쉬워야 한다.
Apple Silicon의 성능 향상과 Nativ 같은 앱들이 이 두 조건을 동시에 충족시켜가고 있다.

### 성능 모니터링이 로컬 AI 도구의 필수 요소가 된다

Nativ가 토큰/초, 메모리 사용량, 열 상태를 실시간으로 표시하는 것은 단순한 디버깅 기능이 아니다.
로컬 AI를 실용적으로 쓰려면 이 모델이 내 장비에서 얼마나 빠르게 실행되는지,
배터리와 발열에 어떤 영향을 주는지 알아야 한다.

이 정보가 없으면 사용자는 모델 선택과 사용 패턴을 최적화할 수 없다.
로컬 AI 앱이 성숙해지면서 이런 관측성 기능이 없는 도구는 전문 사용자에게 선택받기 어려울 것이다.
Nativ가 이 기능을 기본으로 제공한다는 점은 설계 방향이 맞다.

한편 로컬 소형 모델의 실제 활용처가 무엇인지 묻는 시각도 나왔다.
"장난감 프로젝트 외에 실제 업무에 쓰기엔 신뢰하기 어렵다"는 솔직한 의문도 있었고,[^JosNun]
이에 대해 커뮤니티에서는 프라이버시가 중요한 작업, 오프라인 환경, 빠른 초안 작성 등을 현실적 용도로 꼽았다.
