# Mindwalk

<https://github.com/cosmtrek/mindwalk>

HN 토론: <https://news.ycombinator.com/item?id=48878682> (162점, 63개 댓글)

GN 토론: <https://news.hada.io/topic?id=31569>

## 소개

Mindwalk는 AI 코딩 에이전트의 세션 로그를 3D 맵으로 시각화하고 재생하는 도구다.
저장소를 야간 지도처럼 렌더링해, 에이전트가 탐색·읽기·수정한 파일 위치는 빛으로
표현하고 손대지 않은 영역은 어둡게 남긴다.
"세션 로그를 읽기 어렵다"는 문제를 "에이전트가 어느 부분을 탐색했는지 빛으로 보여준다"는
방식으로 해결한다.

## 주요 기능

**시각화 방식.**
저장소를 방사형 트리 또는 트리맵 평면 두 가지 뷰로 표현한다.
파일 상태는 색상으로 구분된다.
탐색만 한 파일은 초록색, 읽은 파일은 파란색, 편집한 파일은 주황색, 손대지 않은 파일은
어두운 색으로 표시한다.
타임라인에는 컨텍스트 압축(◇), 서브에이전트 시작(○), 사용자 전환(›) 마커가 표시되며
클릭 가능하다.

**재생 기능.**
세션을 재생하면 에이전트가 탐색하고 편집한 위치가 시간 순서대로 맵에 나타난다.
재생 속도 조절과 WebM 형식의 영상 내보내기를 지원한다.
영상 내보내기는 클라이언트 측에서 처리된다.
서브에이전트도 별도로 추적할 수 있다.

**평가 기능.**
로컬 에이전트 CLI로 세션을 판단한다.
탐색 범위, 코드 수정 범위, 검증 과정을 분석한다.
평가는 명시적으로 요청할 때만 실행되며, 파일 경로와 이벤트 요약만 모델로 전송한다.
파일 내용은 전송하지 않는다.

## CLI

```bash
curl -fsSL https://raw.githubusercontent.com/cosmtrek/mindwalk/master/scripts/install.sh | sh
```

주요 명령어:

```bash
mindwalk serve [--port N] [--no-open]  # UI 서버 실행
mindwalk open <session.jsonl>          # 특정 세션 열기
mindwalk map                           # 세션 없이 저장소 맵만 표시
mindwalk analyze <session>             # 세션 평가
mindwalk trace                         # JSON 생성
```

## 아키텍처

세 가지 독립적인 구성 요소로 이루어진다.

- **Trace**: 세션 로그를 파일 터치 이벤트로 정규화한다.
- **Citymap**: 저장소의 결정론적 레이아웃을 생성한다. 같은 저장소는 항상 동일한 맵을 보여준다.
- **Report**: LLM을 이용한 증거 기반 판단 결과를 생성한다.

Go 백엔드가 React + Three.js 프론트엔드를 서빙하는 구조다.
단일 Go 바이너리로 배포되며 완전히 로컬에서 실행된다.

## 분석

### 세션 로그 가독성 문제를 공간적으로 해결한다

에이전트 세션 로그는 수천 줄에 달하는 텍스트 파일로 쌓인다.
어느 파일을 얼마나 읽었는지, 어떤 순서로 탐색했는지를 텍스트 로그에서 파악하려면
많은 시간이 걸린다.
Mindwalk는 이 시간적·선형적 데이터를 공간적·시각적으로 변환한다.
"3번 읽히고 2번 수정된 파일"은 로그에서는 찾아야 하지만 맵에서는 밝은 주황색으로
바로 눈에 들어온다.
thunfischtoast는 이 불편함을 직접 표현했다.[^thunfischtoast]
"터미널의 'Read file: xyz' 언급은 추적하기 어렵다.
LLM이 어디서 정보를 가져오는지 쉽게 볼 수 있으면 좋겠다."
실시간 로그를 보면서도 에이전트의 탐색 경로를 이해하기 어렵다는 점이 Mindwalk가
풀려는 핵심 고통이다.

### 제작 동기: LLM 능력 비교의 새로운 방법

원작자 cosmtrek은 HN에서 Mindwalk를 만든 진짜 이유를 밝혔다.[^cosmtrek]
"LLM의 작업 해결 능력을 에이전트의 궤적 시각화를 통해 비교하는 것이 원래 동기였다.
LLM과 에이전트 시스템의 역량을 검증하는 대안적 방식을 제공하고 싶었다."
단순한 세션 재생 도구가 아니라, 서로 다른 모델이 동일한 문제를 어떻게 탐색하는지
비교하는 평가 도구로 기획됐다는 것이다.
alansaber는 이 방향을 구체화했다.[^alansaber]
"두 모델이 같은 문제를 처리할 때 어떤 파일을 건드리고 무엇을 했는지 공간적으로
비교하면 흥미로울 것"이라고 제안했다.
모델 선택의 근거를 정성적 감상이 아니라 탐색 패턴의 시각적 비교로 마련할 수 있다는
발상이다.

### 로컬 우선 설계

세션 데이터에는 코드베이스의 구조와 에이전트의 작업 패턴이 담겨 있다.
Mindwalk는 모든 처리를 로컬에서 수행하고, 평가 단계에서도 파일 경로와 이벤트 요약만
외부 모델로 전송한다.
파일 내용을 전송하지 않는다는 설계 원칙은 기업 환경에서 채택 가능성을 높인다.

## 비평

### 강점: 명확한 문제 정의

"세션 로그를 읽기 어렵다"는 고통은 Claude Code나 Cursor를 써본 사람이라면 공감한다.
Mindwalk는 이 문제를 직접 겨냥하며, 해결 방식도 직관적이다.
3D 맵이라는 시각적 은유가 "에이전트가 어디를 탐색했는가"라는 질문에 잘 들어맞는다.

### 약점: 3D 시각화의 실용성

3D 맵은 인상적으로 보이지만, 실제로 디버깅에 얼마나 유용한지는 검증이 필요하다.
저장소가 수천 개의 파일로 구성된 경우 3D 공간에서 특정 파일을 찾기가 오히려 어려울 수 있다.
텍스트 기반 필터나 검색 기능이 없다면 대형 코드베이스에서는 오히려 2D 트리맵 뷰가
더 실용적일 수 있다.
almog는 핵심적인 질문을 던졌다.[^almog]
"미적으로는 분명히 멋지지만, 실질적인 용도가 무엇인가?"
altmanaltman은 Community TV 시리즈의 에피소드를 비유로 들었다.[^altmanaltman]
학장이 VR 시스템에서 파일 하나를 꺼내려고 미로를 걷고, 여러 장애물을 넘어야 하는 장면이다.
"코드 변경을 검토하고 정리하는 더 빠르고 편리한 방법들이 이미 있지 않은가?"
시각적 인상과 실제 작업 효율 사이의 간극에 대한 비판이다.
esafak은 더 구체적으로 지적했다.[^esafak]
시각화 문제를 먼저 정의한 뒤 그에 맞는 방식을 골라야 하는데, 현재의 3D 맵이 어떤
구체적인 질문에 답하는지 불분명하다는 것이다.
트리맵은 공간 할당을 이해하는 데 적합하지만 에이전트의 탐색 경로를 추적하기에는
다른 형태가 더 나을 수 있다.

## 인사이트

### 에이전트 세션은 새로운 종류의 관찰 대상이다

전통적인 개발 도구는 코드, 빌드 결과물, 런타임 로그를 관찰한다.
에이전트 세션은 이와 다른 종류의 데이터다.
"무엇을 만들었는가"가 아니라 "어떤 과정으로 만들었는가"를 담는다.
Mindwalk가 다루는 것은 결과물이 아니라 에이전트의 인지 과정이다.
이 관점에서 Mindwalk는 에이전트 세션 관찰을 위한 새로운 도구 범주의 시작점으로 볼 수 있다.
에이전트가 코드를 어떻게 탐색하는지 이해하면, 더 효과적인 프롬프트와 더 나은
저장소 구조를 설계하는 데 도움이 된다.
cududa는 더 넓은 방향을 제시했다.[^cududa]
"에이전트와 상호작용하는 최적의 UI는 장기적으로 공간적인 형태가 될 것이라고 점점
더 확신하게 된다."
코드가 파일 트리라는 공간에 존재하는 이상, 그것을 탐색하는 에이전트의 행동도
공간적으로 표현하는 것이 자연스럽다는 시각이다.
Mindwalk는 그 가능성의 초기 탐색이다.
