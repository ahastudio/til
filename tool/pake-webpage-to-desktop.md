# Pake — 웹페이지를 데스크톱 앱으로

<https://github.com/tw93/Pake>

## 소개

Pake는 웹페이지를 macOS, Windows, Linux용 데스크톱 애플리케이션으로 변환하는 오픈소스 도구다.
Rust와 Tauri 프레임워크를 기반으로 하며, GitHub에서 5만 6,700개의 스타를 받은 인기 프로젝트다.
GPL-3.0 라이선스로 공개되어 있으며, 빌드 결과물에 대해서는 GPL 제약이 면제되는 출력 예외(output exception)를 적용한다.

핵심 가치 제안은 Electron 기반 앱 대비 크기 우위다.
“인스톨러가 Electron 패키지보다 거의 20배 작으며 일반적으로 10MB 미만”이라고 강조한다.
이는 Electron이 Chromium과 Node.js 전체를 번들링하는 것과 달리,
Tauri가 운영체제의 네이티브 웹뷰(macOS의 WebKit, Windows의 WebView2)를 활용하기 때문이다.

## 주요 기능

**CLI 사용**: `pake-cli`를 전역 설치하면 URL과 이름만으로 앱을 만들 수 있다.

```bash
pake https://github.com --name GitHub
```

**사전 빌드 패키지**: ChatGPT, Gemini, DeepSeek, Grok 같은 AI 도구,
YouTube, Twitter, WeChat 대안, Excalidraw 같은 생산성 도구의 사전 빌드 패키지를 제공한다.

**커스터마이징**: 아이콘, 창 크기, 스타일 수정, 키보드 단축키, 전체 화면 모드, 개발자 도구를 지원한다.

**GitHub Actions 빌드**: 로컬 개발 환경 없이 GitHub Actions로 온라인 빌드가 가능하다.

**개발 요구사항**: Rust ≥1.85, Node ≥22(LTS 권장), Tauri 프레임워크가 필요하다.

## 분석

### Tauri의 네이티브 웹뷰 활용은 크기 vs. 일관성의 트레이드오프다

Pake가 작은 이유는 기술적으로 명확하다.
Electron은 Chromium 전체를 번들에 포함해 어디서나 동일한 렌더링 엔진을 보장하지만,
그 결과 100MB 이상의 인스톨러가 된다.
Tauri는 운영체제가 이미 갖고 있는 웹뷰를 활용해 앱 크기를 극소화하지만,
macOS의 WebKit, Windows의 WebView2, Linux의 WebKitGTK가 각각 다른 동작을 할 수 있다.

이 트레이드오프는 Pake의 사용 사례와 잘 맞아떨어진다.
단순히 웹사이트를 독립 창에서 띄우는 것이 목적이라면 크로스 플랫폼 렌더링 일관성이 덜 중요하다.
그러나 CSS 애니메이션, 폰트 렌더링, 일부 JavaScript API의 플랫폼별 차이가
특정 웹앱에서 레이아웃 깨짐이나 기능 불일치를 만들 수 있다.

### 5만 6천 개의 스타는 “Electron 앱 크기 피로감”의 직접적 표현이다

Discord, Slack, VS Code, Notion, Figma — 개발자의 데스크톱에는 Electron 앱이 가득하다.
각 앱이 자체 Chromium 인스턴스를 실행하며 수백 MB의 메모리를 점유한다.
이 문제는 오랫동안 알려져 있었고, 수많은 대안들이 제안되었다.

Pake가 5만 6천 개의 스타를 받은 것은 이 피로감이 실제 수요로 표출된 결과다.
특히 개발자들이 Claude, ChatGPT, Gemini 같은 AI 도구를 데스크톱 앱으로 갖고 싶어 하는 수요와
공식 데스크톱 앱이 없거나 Electron 기반으로 무거운 현실이 Pake의 주요 사용 사례를 만들었다.
사전 빌드 패키지 목록에 AI 도구들이 상단에 위치한 것은 우연이 아니다.

### GPL-3.0과 출력 예외는 상업적 활용의 법적 문을 열어둔다

Pake 자체는 GPL-3.0이지만, Pake로 만든 데스크톱 앱은 GPL 의무를 적용받지 않는다.
이 출력 예외(output exception)는 GCC 컴파일러와 유사한 방식으로,
도구의 사용이 결과물을 오염시키지 않도록 명시적으로 규정한다.

이 라이선스 구조는 중요한 실용적 함의를 갖는다.
기업이나 개인이 Pake를 활용해 내부 도구나 상업적 앱을 만들 때
소스 코드 공개 의무 없이 자유롭게 배포할 수 있다.
그러나 Pake 자체를 수정해 별도 도구로 배포하려면 GPL 조건을 따라야 한다.
이 구분을 명확히 이해하지 못한 채 상업적으로 활용하다가
나중에 라이선스 분쟁에 노출될 수 있다.

## 비평

### “웹사이트를 앱으로”라는 접근은 근본적으로 웹앱이 제공하지 못하는 것을 해결하지 않는다

Pake가 만드는 것은 엄밀히 말해 “웹사이트를 네이티브 창에 담은 것”이다.
오프라인 작동, 로컬 데이터 저장, 시스템 통합(파일 시스템, 알림, 키보드 단축키, 메뉴 바) 같은
진정한 네이티브 앱 기능은 Pake 래퍼로 자동으로 얻어지지 않는다.

이 구분이 중요한 이유는 사용자가 “데스크톱 앱”이라는 표현에서 기대하는 것과
실제로 제공되는 것 사이의 간극 때문이다.
ChatGPT를 Pake로 만든 앱은 브라우저 탭에서 열린 ChatGPT와 기능적으로 동일하다.
독립 창과 약간의 UX 개선이 전부다.
이것이 10MB짜리 앱을 받아 설치하는 것과 “브라우저 탭을 고정하는” 것의 실질적 차이가 얼마나 되는지는
개인의 사용 패턴에 따라 다르다.

### 빌드 요구사항이 “단일 명령어” 약속을 배신한다

Rust ≥1.85와 Node ≥22를 요구하는 것은 Rust와 Node 개발 환경이 없는 사람에게
상당한 설치 부담이다.
`pnpm install -g pake-cli` 한 줄로 시작하지만, 그 이전에 시스템에 Rust와 Node가 있어야 한다.
이 의존성이 명확히 문서화되어 있지 않거나, 버전 불일치가 발생하면
“단일 명령어로 앱 만들기”라는 약속이 깨진다.

GitHub Actions 온라인 빌드가 이 문제의 우회책으로 제시되지만,
이것은 진입 장벽을 낮추는 것이 아니라 다른 방향의 복잡성(GitHub 계정, Actions 설정, 저장소 설정)으로 대체한다.
비개발자가 원하는 웹사이트를 Pake로 데스크톱 앱으로 만드는 것이 실제로 얼마나 쉬운지는
프로젝트의 목표 사용자가 누구인가에 달려 있다.

### 플랫폼별 네이티브 웹뷰의 버전 불일치는 사전 빌드 패키지의 신뢰성을 약화한다

사전 빌드 패키지를 배포한다는 것은 특정 시점의 웹사이트와 특정 버전의 웹뷰로 빌드된 앱을
사용자에게 제공한다는 의미다.
웹사이트가 업데이트되면 사전 빌드 앱이 새 UI나 기능을 반영하지 못할 수 있다.
Tauri가 의존하는 시스템 웹뷰가 OS 업그레이드와 함께 변경되면 렌더링 이슈가 발생할 수 있다.

Pake의 관리자가 수십 개의 사전 빌드 앱을 최신 상태로 유지하는 것은
지속적인 유지 관리 부담이다.
이 부담이 커뮤니티 기여에 의존한다면, 인기 있는 앱과 그렇지 않은 앱 사이에
유지 관리 품질의 불균일성이 발생할 것이다.

## 인사이트

### Pake의 성공은 PWA(프로그레시브 웹앱)의 실패를 드러낸다

PWA는 웹 기술로 네이티브 앱 수준의 경험을 제공하겠다는 약속으로 탄생했다.
오프라인 지원, 설치 가능, 알림, 백그라운드 동기화를 브라우저에서 가능하게 하는 표준이 만들어졌다.
그러나 많은 개발자들이 PWA 대신 Pake 같은 도구로 웹사이트를 래핑하는 것을 선택한다.

이 선택의 배경에는 PWA의 구조적 한계가 있다.
iOS에서 PWA는 지속적으로 제한적인 지원을 받았고,
Windows의 PWA 설치 경험은 일관되지 않으며,
일부 브라우저 API는 설치된 PWA에서도 제한된다.
Pake는 이 표준의 실패를 메꾸는 비공식 해결책이다.
표준이 충분히 작동했다면 Pake는 필요 없었을 것이다.

### Tauri 생태계의 성장은 Rust가 앱 개발 영역으로 진입하는 경로다

Pake가 Rust와 Tauri를 기반으로 한다는 것은 단순한 기술 선택이 아니라
Rust 생태계가 시스템 프로그래밍을 넘어 애플리케이션 개발로 확장되는 흐름의 일부다.
Tauri는 Electron 대체재로 포지셔닝하면서, Rust의 메모리 안전성과 성능을 UI 레이어에 가져온다.

Pake의 5만 6천 개 스타는 이 생태계에 의미 있는 검증이다.
비개발자도 결과물을 사용하고, 개발자들이 Rust/Tauri 기반 도구에 스타를 누르는 것은
이 기술 스택이 실용적 가치를 인정받고 있음을 보여준다.
더 많은 도구가 Tauri를 채택할수록 운영체제 내장 웹뷰의 품질과 API 지원도 압력을 받아 개선될 것이다.

### “앱 하나당 독립 창”의 수요는 멀티태스킹 패러다임의 변화를 반영한다

탭 기반 브라우저가 표준이 된 이후, 많은 웹 서비스들이 “브라우저 탭”으로 존재한다.
그러나 생산성 도구, AI 어시스턴트, 협업 앱처럼 지속적으로 사용하는 서비스들은
탭의 일시적 성격 — 실수로 닫히고, 다른 탭에 묻히며, 브라우저 종료 시 사라지는 — 이 불편하다.

Pake의 수요는 “이 서비스는 독립 앱으로 항상 떠 있어야 한다”는 사용자 요구를 반영한다.
Alt+Tab으로 앱 전환, 독(dock)에 고정, 멀티모니터에서 전용 화면 배치 같은 OS 수준의 창 관리가
브라우저 탭 관리보다 일부 사용자에게 더 자연스럽다.
이 수요는 웹이 네이티브 앱을 대체하지 못한 영역, 즉 운영체제와의 통합에서
여전히 간극이 있음을 보여준다.
