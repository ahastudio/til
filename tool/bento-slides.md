# Bento Slides

<https://bento.page/slides/>

HN 토론: <https://news.ycombinator.com/item?id=49008211> (726점, 163개 댓글)

## 소개

Bento Slides는 단일 HTML 파일 하나로 덱(deck), 뷰어, 에디터를 모두 담은 프레젠테이션 도구다.
파일을 브라우저에서 열면 발표 자료를 보여주고, `Esc`를 누르면 같은 파일 안에 있는 에디터가 나타난다.
`⌘S`를 누르면 현재 파일 자체를 덮어쓰는 방식으로 저장한다.
서버가 없고, 계정이 없고, 설치가 없다.

슬라이드 콘텐츠는 `<script>` 블록 안에 JSON으로 저장된다.
이미지와 폰트는 data URI로 파일 안에 내장되므로,
파일 하나만 보내면 받는 사람이 그대로 열어볼 수 있다.
업데이트 기능이 있어서 실행 시 서명된 매니페스트를 확인하고 자동으로 파일을 갱신하는데,
이 기능은 끌 수 있다.

## 주요 기능

**모프 전환**: 슬라이드 간에 같은 `id`를 공유하는 요소는 위치·크기·색상·그래디언트까지 자동으로 애니메이션된다.
PowerPoint나 Keynote의 “매직 무브”와 같은 개념이다.

**라이브 차트**: 막대·파이·산포도 차트를 슬라이드 안에 넣을 수 있다.
툴팁, 줌, 차트 형태 전환이 발표 중에도 실시간으로 작동한다.
테이블 셀을 편집하면 연결된 차트가 즉시 반영된다.

**인터랙티브 스테이트**: 슬라이드 안에 버튼을 두고 다른 “장면(scene)”으로 이동하는 흐름을 만들 수 있다.
각 장면은 동일한 요소를 다른 배치로 보여주는 평범한 슬라이드다.

**마크다운 타이포그래피**: 텍스트 블록에 `**굵게**`, `*기울임*`, `` `코드` `` 등 마크다운 문법을 쓰면 즉시 렌더링된다.

**발표자 뷰**: `S` 키로 발표자 노트와 다음 슬라이드를 별도 창에서 볼 수 있다.

## 분석

### 파일이 소프트웨어라는 철학

Bento Slides의 핵심 명제는 슬라이드 1에 직접 적혀 있다: “The file is the software.”
일반적인 SaaS 프레젠테이션 도구(Canva, Pitch, Tome 등)는 파일이 서버에 있고 앱이 그것을 열어준다.
Bento Slides는 반대다.
파일 안에 런타임, 에디터, 데이터가 모두 들어 있고,
브라우저가 그것을 실행하는 역할만 한다.

이 설계는 Notion이나 Figma 같은 협업 중심 도구가 지배하는 흐름에 역행한다.
대신 1990년대 HyperCard나 2000년대의 HTA(HTML Application)가 했던 시도를 현대 웹 기술로 재현한다.
단일 파일 앱은 배포가 간단하고, 오프라인에서 작동하며, 계정 없이도 완전히 기능한다.
이 접근의 실용적 강점은 영구 보관성이다.
“10년 후에 열어도 런타임이 파일 안에 있어서 작동한다”는 슬라이드 10의 문구가 그 의도를 정확히 담는다.
calebm은 이 흐름을 개념으로 정착시키자고 제안했다.
“Single-File Web Apps”를 Wikipedia 독립 항목으로 만들자는 것이다.[^calebm]
단일 파일 앱이 아직 이름을 갖지 못한 범주라면, 이름을 붙이는 것 자체가 개념을 확산시키는 방법이다.
llagerlof는 TiddlyWiki를 연상했다.[^llagerlof]
수십 년 전부터 단일 HTML 파일 위키를 운영해온 프로젝트다.
Bento Slides는 이 계보의 프레젠테이션 도구 버전에 해당한다.

### 에디터와 뷰어의 경계 제거

대부분의 프레젠테이션 도구는 “편집 모드”와 “발표 모드”가 명확히 분리된다.
Bento Slides는 `Esc` 하나로 전환한다.
발표 중에 에디터를 열어 실시간으로 내용을 수정하거나,
라이브 데이터 차트를 직접 조작하며 보여줄 수 있다.

이 경계 해소는 도구의 투명성을 높인다.
“view-source honest — your document is readable JSON”이라는 문구가 상징하듯,
파일의 내부 구조가 숨겨지지 않는다.

### 인터랙티브 프레젠테이션의 제약

인터랙티브 스테이트 기능은 하나의 슬라이드 안에서 여러 배치를 전환하는 방식이다.
Pitch나 Tome처럼 AI가 콘텐츠를 생성하거나 외부 데이터를 가져오는 기능은 없다.
인터랙티브 요소도 버튼과 장면 이동으로 제한돼 있어,
복잡한 비선형 스토리텔링이나 청중 반응 수집 같은 기능을 구현하기 어렵다.
trentor는 Claude와 Qwen으로 단일 파일 HTML 덱을 직접 만들어온 경험을 공유하면서,
"프레임 안에 프로토타입이나 데모를 통합해 발표 흐름과 함께 보여주는 기능"을 원한다고 밝혔다.[^trentor]
덱 안에 실제 동작하는 코드를 넣어 발표 중 라이브 데모를 보여주는 방향이다.
현재 Bento Slides에는 이 기능이 없지만, 단일 파일 아키텍처 위에서 구현 가능한 확장이다.

## 비평

### 단일 파일 방식은 협업의 구조적 한계를 내포한다

Bento Slides가 선택한 단일 파일 모델은 영구 보관성과 이식성에서 강하지만,
협업에서는 근본적인 제약이 따른다.
두 사람이 같은 파일을 동시에 편집하면 충돌한다.
버전 관리도 git처럼 파일 기반 도구를 외부에서 사용해야 한다.
Figma나 Google Slides가 실시간 협업을 기본값으로 만든 이후,
단일 파일 도구는 “혼자 쓰는 도구”로 포지션이 좁아졌다.
이 선택이 의도적인지, 아니면 극복할 기술적 한계인지는 공개 문서에서 확인하기 어렵다.
작성자 starfallg는 HN 스레드에서 내부 구조를 공개했다.[^starfallg-arch]
슬라이드 데이터는 파일 상단의 JSON 블록에 있고,
이미지는 base64 압축(DecompressionStream)으로 파일 크기를 최소화한다.
협업은 CRDT 기반으로 Cloudflare Durable Objects 위에서 구현했다.
ECDSA 서명 검증, GSAP/Flip 애니메이션, 자체 제작 차트 라이브러리도 포함된다.
외부 의존성을 최소화하기 위해 차트 기능을 직접 구현했다는 설명이다.
실시간 협업이 이미 CRDT로 지원되고 있다는 것은, 문서에서 드러나지 않은 기능이다.

### 자동 업데이트 기능은 파일 영속성의 약속과 충돌한다

“파일 하나만 있으면 10년 후에도 작동한다”는 약속과,
“실행 시 서명된 매니페스트를 확인해 파일을 갱신한다”는 자동 업데이트 기능은 방향이 다르다.
업데이트를 통해 파일이 변경되면, 원본과 달라진다.
끌 수 있다고 하지만, 기본 동작이 자동 갱신이라면
“파일이 소프트웨어”라는 명제에서 기대하는 결정론적 영속성은 약해진다.

### 라이브 차트의 데이터 소스가 파일 내부에 한정된다

라이브 차트와 라이브 데이터를 강조하지만,
실제로 “라이브”는 파일 내부의 JSON을 편집했을 때 즉시 반영된다는 의미다.
외부 스프레드시트, API, 데이터베이스에 연결되는 기능은 없다.
“차트에 맥박이 있다(Charts with a pulse)”라는 문구가 실시간 외부 데이터를 연상시키지만,
실제 동작은 정적 데이터의 인터랙티브 시각화에 가깝다.
접근성도 빠진 항목이다.
jimmar는 이미지에 대체 텍스트(alt text)를 추가할 방법이 없다는 점을 지적했다.[^jimmar]
이로 인해 스크린 리더 사용자는 슬라이드 내 이미지 콘텐츠에 접근할 수 없다.
접근성을 우선순위로 두지 않은 설계라는 비판이다.

## 인사이트

### 로컬 파일 도구가 다시 의미를 갖는 조건

SaaS 도구가 표준이 된 시대에 단일 HTML 파일 도구가 나온 것은 역류처럼 보이지만,
맥락을 보면 합리적인 반응이다.
서버 의존 도구는 서비스가 종료되면 접근이 불가능하다.
Notion, Pitch, Tome 등은 이미 무료 요금제를 변경하거나 기능을 제한한 사례가 있다.
Bento Slides의 영속성 약속은 이 불안에 직접 답한다.
파일 기반 도구가 다시 설득력을 얻는 조건은 클라우드 서비스에 대한 신뢰가 낮아질 때다.
SaaS 피로(SaaS fatigue)라는 말이 생겨난 것은 이 흐름의 징후다.
stackghost는 이 점을 가장 직접적으로 표현했다.[^stackghost]
"설치도 없고, 클라우드 로그인도 없고, 오프라인에서 모두 작동한다.
기본 덱이 560KB 정도고, 한 번 받으면 아무것도 가져올 필요가 없다. 이것이 최고의 기능이다."
HN 댓글에서 이 반응이 공감을 얻었다는 것은,
기업 환경에서 HTML 프레젠테이션으로 전환하는 실제 흐름과도 맞닿아 있다.
georgeburdell은 자신의 회사에서 이미 팀들이 기존 프레젠테이션 소프트웨어의 버그와 기능 부재에 지쳐
HTML/JavaScript 대안으로 이전했다고 전했다.[^georgeburdell]
IT 부서가 이 흐름을 인지해야 한다는 말도 덧붙였다.

### 브라우저가 런타임이 되는 설계의 확장 가능성

Bento Slides가 보여주는 패턴—HTML 파일 안에 에디터, 뷰어, 데이터, 런타임을 모두 넣는 방식—은
프레젠테이션 이외의 도구에도 적용 가능하다.
Obsidian이 마크다운 파일을 에디터 없이도 읽을 수 있도록 만든 것처럼,
도구와 데이터를 분리하지 않는 설계는 이식성과 영속성을 동시에 달성한다.
TiddlyWiki가 수십 년간 이 접근을 유지해온 것이 선례다.
브라우저 엔진이 사실상 어디에나 내장된 지금,
브라우저가 런타임인 단일 파일 앱은 과거보다 훨씬 넓은 환경에서 실행 가능하다.

### 인터랙티브 프레젠테이션의 실제 수요

발표 도구 시장에서 인터랙티브성은 오래된 약속이다.
Prezi는 비선형 탐색으로 주목받았고, Mentimeter는 청중 참여를 넣었다.
Bento Slides의 인터랙티브 스테이트는 이 계보에서 가장 절제된 구현이다.
복잡한 분기 로직이나 외부 입력 없이,
같은 요소를 다른 배치로 보여주는 것만으로도 대부분의 발표 시나리오를 커버한다는 판단으로 보인다.
“모든 장면은 평범한 슬라이드”라는 설명이 이 의도를 담는다.
복잡성을 숨기지 않고 아예 복잡성 자체를 줄인 설계다.
