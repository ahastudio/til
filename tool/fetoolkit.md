# FE Toolkit

<https://www.fetoolkit.io/>

## 소개

FE Toolkit은 프론트엔드 개발자의 반복 작업을 하나의 대시보드에 모아주는 서비스다.
색상 변환, SVG 최적화, JSON 포맷팅, 이미지 압축 등 개발 중 자주 쓰는 유틸리티 13가지를
커스터마이즈 가능한 위젯 형태로 제공한다.
광고 없는 깔끔한 환경을 표방하며, 무료 계정으로 이용할 수 있다.

## 주요 기능

### 위젯 목록

| 위젯 | 기능 |
| ---- | ---- |
| Color Converter | RGB/HEX/HSL 상호 변환 |
| SVG Converter | 최적화 후 JSX, TSX, React Native, CSS, Base64 출력 |
| Base64 Converter | 인코딩/디코딩 |
| CSS Converter | 압축(minify) 및 정리(beautify) |
| URL Converter | URL 인코딩/디코딩 |
| Regex Tester | 구문 강조 포함 JavaScript 패턴 테스트 |
| NPM Package Size | BundlePhobia 연동 의존성 크기 분석 |
| JSON Converter | 압축, 포맷팅, TypeScript 인터페이스 생성 |
| JS/TS Converter | AI 보조 JavaScript → TypeScript 변환 및 코드 포맷팅 |
| Image Converter | 브라우저 기반 JPG, PNG, WebP, GIF 압축 |
| CSS Triangle Generator | 커스터마이즈 가능한 CSS 삼각형 생성 |
| CSS & SVG Symbols | 아이콘/심볼 컬렉션 라이브러리 |
| Embedded Websites | 외부 사이트를 대시보드에 직접 삽입 |

마크다운 노트와 코드 스니펫 저장 기능도 포함된다.

### 대시보드 관리

프로젝트나 팀별로 여러 대시보드를 만들 수 있다.
각 대시보드는 비공개, 비밀번호 보호, 공개 중 하나로 설정하고
직접 링크로 공유할 수 있다.

## 분석

### 파편화된 개발자 유틸리티 시장의 통합 시도

프론트엔드 개발자가 일상에서 쓰는 유틸리티 도구들은 대부분 별도 사이트에 흩어져 있다.
JSON 포맷팅은 jsonformatter.org, 정규식 테스트는 regex101.com, SVG 최적화는 svgomg.app.
각 도구는 해당 기능에 특화되어 있지만 브라우저 탭을 여러 개 열어야 하고
작업 컨텍스트가 분산된다.

FE Toolkit은 이 파편화에 대한 한 가지 답이다.
도구들을 모아놓는 것 자체가 새로운 가치를 만든다는 가정에서 출발한다.
“자주 쓰는 도구를 한 곳에 두면 맥락 전환 비용이 줄어든다”는 명제는
단순해 보이지만 실제로 개발자의 작업 흐름을 이해한 포지셔닝이다.
Notion이 노트/위키/태스크를 한 공간에 모아 성공한 것과 같은 방향의 직관이다.

### 13가지 위젯 선택이 반영하는 현재 프론트엔드 환경

포함된 위젯들을 보면 현재 프론트엔드 개발 환경의 단면이 드러난다.
JSON → TypeScript 인터페이스 자동 생성은 TypeScript 전환이 완전히 일상화된 맥락에서만
일상적 반복 작업이 된다.
SVG → JSX/TSX 변환은 React 생태계가 SVG 처리를 컴포넌트 방식으로 다루기 때문에 필요하다.
NPM 패키지 크기 분석은 번들 크기 최적화가 프론트엔드의 핵심 관심사가 되었음을 보여준다.

반면 포함되지 않은 도구들도 선택의 윤곽을 드러낸다.
Git diff 뷰어, API 응답 모킹, HTTP 헤더 분석 같은 도구들은 없다.
위젯 목록은 “무엇이든 다 넣겠다”가 아니라
일정한 기준으로 범위를 좁힌 결과처럼 보인다.
그 기준이 명시되지 않는다는 점이 이후에 살펴볼 비평 지점이기도 하다.

### 대시보드 공유와 팀 도구로서의 잠재성

개인 도구를 팀과 공유하는 기능은 단순한 편의 기능 이상의 함의를 갖는다.
“우리 팀이 쓰는 도구 셋”을 하나의 대시보드로 구성해 신규 팀원에게 공유하면
온보딩 체크리스트의 일부를 대체할 수 있다.
“이 도구로 색상 변환하고, 이 도구로 SVG 처리한다”는 맥락이
대시보드 구성 자체에 인코딩되기 때문이다.

이 잠재성은 현재 홈페이지에서 충분히 강조되지 않는다.
대시보드 공유가 가능하다는 것은 언급되지만
팀 온보딩이나 프로젝트별 도구 표준화 용도로 어떻게 쓸 수 있는지는
서술되지 않는다.

## 비평

### “AI-enhanced” 변환이 무엇을 뜻하는지 설명하지 않는다

JS/TS Converter에 “AI-enhanced”라는 수식어가 붙어 있지만
어떤 AI 모델이, 어떤 방식으로, 어떤 부분에 관여하는지 설명이 없다.
JavaScript를 TypeScript로 변환하는 작업에서 AI가 추가하는 가치가
단순한 타입 추론 이상인지, 실제로 어떤 사례에서 효과적인지를
사용자가 판단할 수 없다.

“AI-enhanced”는 2020년대 중반의 마케팅 언어로 이미 의미가 희석되어 있다.
AI가 포함된다는 사실이 이 위젯을 선택할 이유가 되려면
어떤 문제를 AI가 해결하는지 구체적으로 보여줘야 한다.
예를 들어 “타입 단언이 필요한 위치를 AI가 추론한다”거나
“React prop 타입을 자동으로 인식한다”는 식의 설명이 없다면
이 레이블은 신뢰보다 의구심을 먼저 낳는다.

### CSS Triangle Generator의 존재가 위젯 선택 기준을 모호하게 만든다

CSS Triangle Generator는 CSS의 `border` 속성을 이용해 삼각형을 만드는 오래된 기법을 보조하는 도구다.
이 기법은 `clip-path`와 SVG가 보편화된 현재 프론트엔드 작업에서 사용 빈도가 높지 않다.
13개 위젯 목록에서 이 도구의 존재는 “반복 작업”이라는 FE Toolkit의 표방 기준이
실제로 어떻게 적용됐는지 의문을 일으킨다.

더 자주 필요한 도구들 — 색상 팔레트 생성, 타이포그래피 스케일 계산,
반응형 브레이크포인트 시각화, CSS 변수 관리 — 이 빠진 자리에
CSS Triangle Generator가 있다는 것은
위젯 선택이 “현재 개발자에게 가장 자주 필요한 것”보다
“만들기 쉬운 것”이나 “이미 아이디어가 있었던 것”에 더 가까울 수 있음을 시사한다.

### 브라우저 기반 처리 주장이 모든 위젯에 일관되게 적용되는지 불분명하다

Image Converter에 대해 “브라우저 기반 압축”임을 명시하는 것은
이미지가 서버로 전송되지 않는다는 프라이버시 보장이다.
이것은 중요한 정보다.
그러나 다른 위젯들, 특히 “AI-enhanced” JS/TS Converter에 대해서는
처리가 어디서 이루어지는지 설명이 없다.

AI 변환이 서버 측에서 처리된다면 코드가 외부 서버로 전송된다.
회사 코드베이스를 이 도구에 붙여 넣는 개발자는
코드가 어디로 가는지 알아야 한다.
이미지에 대해서는 명시하고 코드에 대해서는 명시하지 않는 구조는
정보 공개의 일관성 문제이자 보안 의식의 문제다.

## 인사이트

### 개발자 유틸리티 대시보드는 IDE와 웹 서비스 사이의 고유한 위치를 점유한다

Color Converter, Regex Tester, JSON Formatter 같은 도구들은
VS Code 확장으로도 구현할 수 있다.
실제로 이 도구들의 VS Code 버전은 이미 존재한다.
그렇다면 FE Toolkit 같은 웹 서비스가 존재하는 이유는 무엇인가.

브라우저 기반이라는 점이 결정적 차이다.
VS Code 확장은 IDE가 열려 있는 동안, 해당 OS에서만 동작한다.
웹 서비스는 회사 컴퓨터, 개인 노트북, 클라이언트 사무실 PC에서
로그인 하나로 같은 환경을 재현한다.
또한 코드 에디터와 독립된 공간에서 도구를 쓰는 것이
작업 흐름상 더 자연스러운 경우도 있다.
이 포지션은 IDE 확장보다 낮은 통합도를 갖지만
플랫폼 독립성이라는 장점으로 다른 사용자 집합을 포착한다.

### 위젯 커스터마이제이션이 사용자의 도구 사용 패턴 데이터가 된다

대시보드에 어떤 위젯을 배치하는지는 개별 사용자의 도구 사용 패턴을 드러낸다.
어떤 위젯이 자주 추가되고, 어떤 위젯이 제거되는지를 집계하면
다음 위젯을 무엇으로 추가할지에 대한 신호가 된다.
대시보드 공유 기능은 이 신호를 팀 단위로 확장한다.
“이 팀은 어떤 도구를 표준으로 쓰는가”라는 데이터다.

이 데이터는 FE Toolkit이 단순한 도구 모음을 넘어
프론트엔드 개발 생태계의 도구 사용 트렌드를 관찰하는 창이 될 수 있다는 것을 의미한다.
어떤 색상 형식이 가장 많이 변환되는지, 어떤 이미지 형식이 압축 요청이 많은지는
디자인 시스템과 프론트엔드 기술 스택의 현황을 간접적으로 보여준다.
이 관찰 능력이 서비스의 장기 가치를 결정할 수 있다.

### 프론트엔드 개발자 도구의 파편화는 “최선의 도구”를 알기 어렵게 만든다

JSON formatter를 한 번만 쓸 개발자는 첫 번째 검색 결과를 쓴다.
자주 쓰는 개발자는 선호하는 사이트 하나를 기억한다.
이 선택은 합리적 비교가 아니라 습관이다.

FE Toolkit이 노리는 것은 이 습관의 교체다.
“어차피 여러 도구를 쓴다면 한 곳에서 쓰는 습관”으로 전환시키는 것이 목표다.
이 전환이 성공하려면 대시보드에 들어오는 순간부터
각 위젯이 사용자가 기억하고 있던 기존 도구보다 나아야 한다.
도구의 품질이 통합의 가치를 뒷받침하지 못하면
“모아놨지만 쓰지 않는” 대시보드가 된다.
FE Toolkit의 핵심 도전은 통합 경험을 파는 동시에
각 위젯이 해당 카테고리의 전문 도구와 비교되는 위치에 있다는 점이다.

### 팀 대시보드가 개발 문화의 표준화 도구로 진화할 수 있다

팀이 공유 대시보드를 갖는다는 것은 “우리 팀은 이 도구들을 이 방식으로 쓴다”는 합의를
시각화하는 것이다.
신규 팀원이 대시보드 링크를 받으면 온보딩 첫날부터 팀의 도구 셋에 접근할 수 있다.
어떤 색상 형식을 주로 사용하는지, SVG를 어떤 형태로 변환해서 쓰는지가
대시보드 구성에 반영된다.

이것은 문서화와 다르다.
“우리는 HSL 색상을 쓴다”는 문서보다
HSL 변환 위젯이 올라온 대시보드가 더 즉각적이고 실행 가능하다.
도구가 관행을 인코딩하는 방식이다.
FE Toolkit이 이 방향으로 발전한다면
단순한 유틸리티 모음에서 팀 개발 문화의 접점으로 성격이 바뀔 수 있다.
그 전환을 의도적으로 설계하고 있는지는 현재 홈페이지만으로 알 수 없다.
