# transcribe.cpp

원문: <https://workshop.cjpais.com/projects/transcribe-cpp>

HN 토론: <https://news.ycombinator.com/item?id=48963879> (733점, 156개 댓글)

## 요약

transcribe.cpp는 ggml 기반 크로스 플랫폼 로컬 음성 전사(ASR) 라이브러리다.
whisper.cpp와 거의 호환되면서 더 넓은 모델 지원과 나은 성능을 제공하는 것을 목표로 한다.
16개 ASR 패밀리, 60개 이상의 모델을 지원하며
Vulkan, Metal, CUDA, TinyBLAS를 통한 GPU 가속을 갖추고 있다.

작자는 Handy라는 크로스 플랫폼 STT 앱을 배포하면서
기존 배포 옵션이 whisper.cpp와 ONNX밖에 없다는 현실에서 이 라이브러리를 만들었다.
실시간보다 빠른 전사를 SOTA 모델로 몇 와트 수준의 전력으로 실행하는 것이 목표이고,
클라우드 의존을 없애는 로컬 추론에 집중한다.

모든 모델은 수치 검증과 단어 오류율(WER) 테스트를 거쳐
레퍼런스 구현과의 출력 일치를 보증한다.
Python, JavaScript/TypeScript, Rust, Objective-C/Swift 공식 바인딩을 제공하며
Mozilla AI, ggml 기여자, Hugging Face 등의 지원을 받아 개발됐다.

## 분석

### whisper.cpp 이후 배포 공백을 채우는 시도

로컬 음성 전사 생태계는 whisper.cpp가 사실상 표준으로 자리 잡은 뒤
배포 편의성과 다양한 모델 지원 면에서 발전이 더뎠다.
whisper.cpp는 Whisper 계열 모델에 특화돼 있고,
새 ASR 아키텍처가 나와도 빠르게 지원하기 어려운 구조다.

transcribe.cpp는 이 공백을 다모델 지원과 공식 다언어 바인딩으로 채우려 한다.
simonw는 Python 바인딩이 이미 PyPI에 있지만 바이너리 휠 배포는 아직 준비 중임을 지적했다.[^simonw]
배포 형태까지 완성되면 로컬 STT를 Python 생태계에 통합하는 진입 장벽이
크게 낮아질 수 있다.

### 로컬 STT의 실제 사용 시나리오와 한계

abdullahkhalids는 현재 STT 도구들의 공통 한계를 정확히 짚었다.
전사 정확도는 충분히 높아졌지만, 커서 위치에 실시간으로 받아쓰는 스트리밍 입력 모드가
부족하다는 점이다.[^abdullahkhalids]
녹음 후 일괄 전사는 받아쓰기 워크플로에는 맞지 않고,
연속적인 저지연 스트리밍이 돼야 실제 문서 작성 보조로 쓸 수 있다.
이것은 transcribe.cpp만의 문제가 아니라 로컬 STT 생태계 전반의 과제다.

## 비평

### 배포 편의성이 완성되기 전까지는 틈새 도구다

“60개 이상의 모델 지원”은 강점이지만, 바이너리 배포가 완성되기 전까지는
일반 개발자가 직접 쓰기 어렵다.
ctypes로 별도 설치된 네이티브 라이브러리를 연결하는 방식은
의존성 관리 문제를 사용자에게 넘기는 구조다.
프로덕션 파이프라인에 넣으려면 배포 자동화와 버전 고정이 선행돼야 한다.

이 문제는 whisper.cpp 생태계도 겪었고 faster-whisper 같은 래퍼 라이브러리가
그 부분을 채워 주목받은 전례가 있다.
transcribe.cpp가 직접 이 경험을 제공하지 않으면,
편의성을 앞세운 대안 라이브러리가 같은 자리를 차지할 수 있다.

### WER 검증이 실제 사용 언어 다양성을 반영하는가

모든 모델에 WER 테스트를 했다는 점은 신뢰도를 높이지만,
어떤 데이터셋과 언어로 검증했는지가 중요하다.
rmunn이 지적한 것처럼, 소수 언어 음성 연구나 국제 음성 기호(IPA) 전사 같은
특수 사용 사례에서 WER이 어떻게 나오는지는 알 수 없다.[^rmunn]
“SOTA 모델”의 성능이 영어 중심 벤치마크에서 나온 것이라면,
다국어 환경에서의 실제 품질은 별도 검증이 필요하다.

## 인사이트

### 로컬 STT는 아직 배포 문제가 모델 품질만큼 중요하다

음성 전사 모델의 품질이 충분한 수준에 도달했음에도 로컬 STT 채택이 느린 이유는
배포와 통합의 어려움이 크다.
whisper.cpp가 성능 측면의 표준을 만들었다면,
그 다음 도전은 “설치하고 바로 쓰는” 경험을 만드는 것이다.
transcribe.cpp가 이 방향을 명시적으로 목표로 삼은 것은 올바른 판단이다.

로컬 STT가 개발자 툴체인에 자연스럽게 통합되면,
회의 전사, 코드 설명 음성화, 접근성 도구, 언어 학습 앱 같은
클라우드 API 없이는 구현하기 번거로웠던 기능들이
훨씬 낮은 비용으로 실험 가능해진다.

### 로컬 추론 생태계의 공급망 문제

ggml 기반 로컬 추론 생태계는 whisper.cpp, llama.cpp, transcribe.cpp 등
개별 라이브러리 단위로 발전하고 있다.
각 라이브러리가 모델 지원, 플랫폼 가속, 바인딩을 독립적으로 구현하는 구조는
중복 작업이 많고 생태계 파편화를 낳는다.
Mozilla AI 같은 후원이 이 공백을 메우는 방향으로 쓰인다면
공통 인프라 계층이 생길 가능성이 있다.
