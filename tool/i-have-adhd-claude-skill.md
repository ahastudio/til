# i-have-adhd — 코딩 에이전트를 위한 ADHD 친화적 출력 스킬

<https://github.com/ayghri/i-have-adhd>

## 소개

i-have-adhd는 Claude Code, Codex 같은 코딩 에이전트에게 설치하는 슬래시 커맨드 스킬이다.
에이전트가 답을 뒤에 묻어버리는 방식 대신, 핵심 행동을 먼저 제시하도록 출력 방식을 바꾼다.
GitHub 별 7,700개 이상을 받았으며, MIT 라이선스로 공개됐다.

Claude Code에서는 다음 명령으로 설치한다.

```bash
claude plugin marketplace add ayghri/i-have-adhd
claude plugin install i-have-adhd@i-have-adhd
```

설치 후 `/i-have-adhd`를 입력하면 스킬이 활성화된다.
매 세션마다 자동 적용하려면 `touch ~/.claude/.i-have-adhd-always`를 실행한다.

스킬의 핵심은 10개 규칙이다.

1. 다음 행동을 맨 앞에 제시한다
2. 여러 단계 작업은 번호를 매긴다
3. 구체적인 다음 단계 하나로 마친다
4. 곁가지 내용을 억제한다
5. 매 응답마다 현재 상태를 재진술한다
6. 시간 추정은 구체적으로 (분 단위, “잠깐”은 금지)
7. 완료된 항목을 가시화한다
8. 오류는 담담하게 사실로 서술한다
9. 목록은 최대 5항목
10. 도입부 없음, 요약 없음, 마무리 멘트 없음

## 분석

### 코딩 에이전트 출력의 구조적 문제를 정면으로 다룬다

LLM 기반 코딩 에이전트의 출력은 흔히 “답을 묻어버리는” 구조다.
질문에 먼저 공감을 표현하고, 맥락을 설명하고, 여러 접근 방식을 나열한 뒤, 마지막에 실제 행동을 제시한다.
이것은 모델이 훈련받은 “도움이 되는 대화”의 패턴이지만, 코딩 작업에서는 역효과를 낸다.

개발자가 “이 버그를 고쳐줘”라고 했을 때 필요한 것은 즉각적인 행동이다.
어떤 파일 몇 번째 줄을 어떻게 바꾸고, 어떤 명령을 실행하고, 결과가 어떠면 성공인지다.
이 스킬은 모델의 기본 출력 패턴을 행동 우선으로 재조정한다.

### 규칙 9번이 핵심이다 — 목록 5개 제한

10개 규칙 중 가장 실질적 영향이 큰 것은 목록 최대 5항목 제한이다.
LLM은 “고려해야 할 것들”을 열거할 때 자연스럽게 과잉 생성한다.
7개, 10개, 때로는 15개 항목을 나열하는데, 이 중 실제로 행동 가능한 것은 2–3개다.

5개 제한은 모델이 우선순위를 강제하게 만든다.
“지금 당장 해야 하는 것 5개” 안에 들어오지 않으면 나열하지 않는다.
이것이 인지 부하를 줄이는 핵심 메커니즘이다.

### *ADHD 도구키트*에서 빌려온 원칙의 LLM 적용

저자는 이 스킬이 J. Russell Ramsay와 Anthony L. Rostain의 *The Adult ADHD Tool Kit*에서 느슨하게 영감을 받았다고 밝힌다.
ADHD 관리 전략 — 즉각적 행동 피드백, 상태 재진술, 완료 가시화 — 을 인간이 자신을 조직하는 방법이 아니라, LLM이 응답하는 방법으로 재해석했다.

이 전환이 흥미로운 이유는, ADHD 친화적 설계가 집중력 문제가 없는 사람에게도 유용하다는 점이다.
“답을 먼저, 설명은 나중에” 원칙은 누구에게나 시간을 절약한다.
보편적 설계(universal design)가 접근성을 넘어 일반적 사용성을 높이는 것과 같은 원리다.

## 비평

### 스킬이 모델의 기본 경향을 얼마나 오래 억제할 수 있는가

이 스킬은 프롬프트 레벨에서 모델의 출력 스타일을 바꾼다.
그러나 대화가 길어지면 모델은 훈련 시의 기본 패턴으로 돌아가는 경향이 있다.
단기 요청에서는 효과가 분명하지만, 긴 코딩 세션에서 스킬의 효과가 얼마나 지속되는지는 사용자 보고에 따라 다르다.

“매 세션 자동 적용”(`touch ~/.claude/.i-have-adhd-always`) 기능이 이 문제를 부분적으로 다루지만,
세션 내 지속성보다 세션 시작 시 재활성화에 의존하는 구조다.

### 10개 규칙이 서로 충돌할 수 있다

규칙 3번(구체적인 다음 단계 하나로 마친다)과 규칙 2번(여러 단계는 번호를 매긴다)은 상황에 따라 긴장 관계가 생긴다.
복잡한 디버깅 시나리오에서 “다음 단계 하나”가 적절하지 않을 때, 모델이 어느 규칙을 우선하는지 명확하지 않다.

규칙 간 충돌 해결 방침이 `SKILL.md`에 명시되지 않은 점은, 포크해서 커스터마이징하는 사용자에게 가장 먼저 마주치는 모호함이다.

## 인사이트

### 코딩 에이전트 출력 설계가 독립적인 관심 영역이 됐다

이 프로젝트가 7,700개 별을 받은 것은 단순한 유용성 때문이 아니다.
“에이전트가 어떻게 말해야 하는가”가 독립적인 설계 문제로 인식되기 시작했다는 신호다.

모델 능력이 충분해진 시점에서 사용자 경험을 결정하는 것은 점점 더 출력 형식이다.
정확한 코드를 내놓더라도 개발자가 그것을 찾아 읽는 데 시간이 걸리면 생산성이 낮아진다.
이 스킬의 성공은 “출력 형식 최적화”가 모델 파인튜닝 없이 프롬프트 레벨에서도 의미 있게 개선될 수 있다는 것을 보여준다.

### Claude Code 플러그인 마켓플레이스가 에이전트 스킬 생태계의 진입점이 된다

i-have-adhd가 `claude plugin marketplace add`로 설치된다는 것은,
Claude Code 플러그인 마켓플레이스가 에이전트 스킬 유통 채널로 작동하고 있음을 보여준다.
개발자가 스킬을 만들어 마켓플레이스에 올리고, 다른 개발자가 한 명령으로 설치한다.

이 패턴이 확산되면, “에이전트 출력 스타일”이 개인화 레이어가 된다.
ADHD 친화적 출력, 교육용 상세 출력, 코드 리뷰어 모드 — 각각 다른 맥락에 최적화된 스킬을 필요에 따라 전환하는 방식이다.
이것은 모델 선택과 별개로 에이전트 경험을 조율하는 새로운 차원이다.
