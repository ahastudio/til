# wikimap – Zero-LLM 지식 금고 색인 도구

<https://github.com/dhha22/wikimap>

<https://pypi.org/project/wikimap/>

## 소개

wikimap은 마크다운, HTML, PDF, 이미지로 구성된 지식 금고(knowledge vault)를
LLM 없이 색인하는 Python 도구다.
MIT 라이선스, 의존성 제로, 단일 파일로 배포된다.
`pipx install wikimap` 으로 설치하고 지식 금고 디렉터리에서
`wikimap update`를 실행하면 된다.

핵심 원칙은 “구조는 지금 파싱하고, 의미는 나중에 배운다(parse the structure
now, learn the meaning later)”다.
문서를 처음 수집할 때 제목, 헤딩, 링크, ID 같은 구조 요소를 추출하고,
LLM 호출이 필요한 의미 처리는 실제 쿼리가 들어왔을 때로 미룬다.
이 접근 방식으로 색인 비용을 $0으로 유지하면서도 자연어 검색을 지원한다.

## 주요 기능

- **준실시간 증분 업데이트**: 콘텐츠 해싱으로 변경 감지, 단일 문서 업데이트
  0.07초, 0 토큰
- **자연어 검색**: 결정론적 $0 랭킹, 검색 지연 0.15초
- **링크 발견**: 공유된 희귀 용어와 폴더 근접도 같은 무료 신호 사용
- **캐시된 의미 레이어**: 원본 파일이 변경되면 자동으로 만료
- **에이전트 통합**: `wikimap install`로 Claude Code, GitHub Copilot 등
  에이전트 도구에 등록

262개 문서 이중 언어 금고 기준 벤치마크에서 전체 색인 빌드 0.28초/$0,
71개 질문 회상 벤치마크(recall@5)에서 0.887을 달성했다.

## 분석

### “LLM 없는 색인”이 가능한 이유와 그 구조적 선택

wikimap이 Zero-LLM을 유지하면서도 자연어 검색을 제공할 수 있는 핵심은
“비용 발생 시점을 색인에서 쿼리로 이동”한 설계에 있다.
경쟁 도구들은 문서 수집 시점에 LLM으로 임베딩이나 지식 그래프를
만들기 때문에 문서 수가 늘수록 사전 비용이 쌓인다.
wikimap은 구조 추출(무료)만 미리 하고 의미 해석(유료)은 쿼리 시점으로
미루기 때문에, 색인 자체는 항상 $0이다.

이 선택의 트레이드오프는 쿼리 응답 시간에 있다.
쿼리가 의미 처리를 요구할 때 LLM 호출이 발생하므로, 대규모 금고에서
복잡한 의미적 질문을 자주 한다면 쿼리 비용이 누적될 수 있다.
반면 자연어 쿼리를 거의 하지 않고 구조 검색(키워드, 링크 탐색)이
주 용도라면 비용이 거의 발생하지 않는다.
개인 노트나 소규모 팀 지식 금고처럼 쿼리 빈도가 낮은 환경에 최적화된
설계다.

### 콘텐츠 해시 기반 캐시 무효화가 해결하는 오래된 문제

지식 금고 도구들이 공통으로 겪는 문제 중 하나는 “오래된 정보가 조용히
에이전트 결정을 오염시키는 것”이다.
문서가 변경됐는데 캐시가 남아 있으면, 에이전트는 이미 없어진 내용이나
바뀐 사실을 기반으로 답변한다.

wikimap은 모든 캐시 데이터를 원본 파일의 콘텐츠 해시에 연결한다.
파일이 변경되면 해시가 달라지고, 이전 캐시는 자동으로 만료된다.
이 메커니즘은 단순하지만 신뢰도 측면에서 중요한 보장이다.
특히 에이전트 시스템에서 “내가 참조한 정보가 현재 상태를 반영한다”는
신뢰는 도구 선택의 핵심 기준이 된다.

### Claude Code 통합이 노리는 에이전트 시장

wikimap의 GitHub 주제 태그에 `claude-code`가 포함되어 있고, `wikimap
install`이 Claude Code 통합을 지원한다.
이는 개인 개발자 대상 유틸리티보다 “AI 에이전트의 메모리 레이어” 포지션을
겨냥하고 있음을 보여준다.
Claude Code가 로컬 파일을 탐색하고 분석하는 세션에서, wikimap으로 색인된
지식 금고는 에이전트가 “이 파일과 연결된 다른 개념은 무엇인가”를 빠르게
파악하는 데 도움이 된다.

에이전트 도구 생태계에서 “로컬 지식 금고 색인”은 아직 표준화된 솔루션이
없는 영역이다.
Obsidian, Logseq, 기타 마크다운 기반 노트 도구 사용자들이 AI 에이전트와
자신의 로컬 지식을 연결하려 할 때, 이 공백을 채우는 것이 wikimap의
기회다.

## 비평

### 벤치마크 비교 대상인 graphify가 무엇인지 알 수 없다

README의 성능 수치는 “graphify”와의 비교로 제시된다.
전체 색인 빌드 0.28초/$0 대 “분(minutes) 이상 + 추출 비용”,
단일 문서 업데이트 0.07초/0토큰 대 약 95초/46,000 토큰이라는 차이는
인상적이다.
그러나 graphify가 어떤 도구인지, 어떤 설정에서 측정했는지, 262개 문서가
어떤 종류의 문서인지가 명확하지 않다.

이 벤치마크가 공정한 비교인지를 외부에서 검증하기 어렵다.
자체 제작 벤치마크에서 자사 도구가 좋은 결과를 내는 것은 흔한
마케팅 패턴이다.
recall@5 0.887이 “좋은” 수치인지도 문서 도메인과 쿼리 유형에 따라
크게 달라지므로, 이 수치만으로 실제 유용성을 판단하기는 어렵다.

### 단일 파일 의존성 제로가 제공하는 신뢰와 그 한계

“단일 파일, 의존성 제로”는 설치 편의성 측면에서 강력한 가치 제안이다.
설치 실패, 버전 충돌, 환경 문제가 없다는 뜻이기 때문이다.
그러나 의존성 제로는 동시에 “단일 파일 안에 모든 것을 구현해야 한다”는
제약이기도 하다.
PDF 파싱, 이미지 처리, 자연어 검색을 단일 파일에서 구현하려면 각 기능의
복잡성을 감출 방법이 필요하고, 이는 외부 의존성을 완전히 제거하기보다
Python 표준 라이브러리 범위 안에서 제한된 구현을 선택하는 것을 의미할 수
있다.
PDF나 이미지의 복잡한 구조를 얼마나 정확하게 처리할 수 있는지는 실제
테스트 없이 판단하기 어렵다.

## 인사이트

### “비용 발생 시점 지연”은 AI 도구 설계의 새로운 패러다임이다

wikimap의 핵심 아이디어는 AI 관련 비용(토큰, 지연, 인프라)이 발생하는
시점을 최대한 뒤로 미루는 것이다.
이 패러다임은 소프트웨어 엔지니어링의 “지연 평가(lazy evaluation)”를
AI 비용 구조에 적용한 것으로 볼 수 있다.
모든 것을 미리 처리하는 “이거 전부 LLM 돌려놔” 방식은 비용이 예측 불가능하게
쌓이는 문제가 있는데, 필요한 순간에만 처리하는 방식은 비용을 실제 사용과
연동시킨다.

이 접근은 AI 사용 비용이 여전히 높고 예측하기 어려운 현재 환경에서
특히 유용하다.
앞으로 토큰 비용이 내려가면 이 최적화의 필요성이 줄겠지만, 그 전까지
“필요할 때 쓴다”는 설계 원칙은 합리적이다.

### 로컬 지식과 AI 에이전트의 통합이 아직 해결되지 않은 이유

개인 개발자들이 수년간 쌓아온 마크다운 노트, 위키, 문서들은 AI 에이전트가
활용하기 어렵다.
이유는 크게 두 가지다.
첫째, 로컬 파일을 실시간으로 에이전트에게 제공하는 표준화된 방법이 없다.
둘째, 전체 파일 내용을 컨텍스트로 넣으면 토큰이 부족하고, 어떤 파일이
관련 있는지를 에이전트가 파악하기 어렵다.

wikimap은 두 번째 문제의 해결을 노린다.
전체 파일 대신 구조화된 색인을 에이전트에게 제공함으로써, 에이전트가
“어떤 파일이 이 쿼리와 관련 있는가”를 빠르게 파악하고 필요한 파일만
읽을 수 있게 한다.
이 “에이전트를 위한 색인” 문제는 MCP(Model Context Protocol) 생태계가
성숙하면서 더 많은 도구들이 뛰어들 가능성이 높은 영역이다.
