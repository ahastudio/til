# HyperDesk — 창 스왈로잉으로 여러 원격 세션을 한 창에 담기

<https://github.com/qetqet910/HyperDesk>

## 소개

HyperDesk는 Hyper-V 가상 머신과 RDP 원격 데스크톱 세션을 하나의 창에서
관리하고 모니터링하는 Windows 전용 데스크톱 애플리케이션이다.
Tauri v2와 Rust를 기반으로 하며,
프런트엔드는 React 19와 TypeScript로 작성됐다.
현재 Hyper-V(VMConnect)와 RDP는 공식적으로 검증됐고,
VMware/Omnissa 지원 코드는 존재하지만 프로덕션 환경에서 검증되지 않았다.

라이선스는 PolyForm Noncommercial License 1.0.0이다.
비상업적 사용은 제한 없이 허용되지만,
상업적 배포에는 별도의 라이선스 계약이 필요하다.

## 동작 방식

HyperDesk의 핵심 기술은 "SwallowGrid™"라고 부르는 창 스왈로잉(Window
Swallowing) 방식이다.
RDP나 VMConnect가 띄운 외부 프로세스의 창을 스크린샷이나 미리보기로
대체하는 대신, `SetParent`, `SetWindowPos`, `EnumWindows`, `SetWindowRgn`
같은 Win32 API를 사용해 그 창 자체를 HyperDesk 창 안에 재배치한다.

이 과정에서 몇 가지 세부 구현이 눈에 띈다.

- 메시지 큐를 독립적으로 유지해 `AttachThreadInput`을 사용하지 않는다.
  이를 통해 자식 프로세스가 인증 대화상자 같은 걸 띄울 때 발생할 수 있는
  데드락 위험을 피한다.
- 창 위치 동기화는 `requestAnimationFrame`과 백엔드의 델타 필터링을
  조합해 처리한다.
- VMConnect가 제거할 수 없는 리본 UI는 `SetWindowRgn`으로 영역을
  잘라내 가린다.

이렇게 재배치된 세션은 최대 4개까지 동시에 유지되며,
연결을 끊지 않고 `Alt+1`부터 `Alt+4`까지의 단축키로 전환할 수 있다.
전체화면으로 전환하는 몰입 모드에서는 UI 크롬이 사라지고,
커서가 화면 상단에 닿을 때만 헤더가 나타난다.
이 모드에서는 Win 키나 Alt+Tab 같은 시스템 단축키도 활성 VM 세션으로
바로 전달된다.

## 사용법

개발 환경에서 실행하려면 다음이 필요하다.

- Rust 1.80 이상
- Node.js(LTS 권장)
- WebView2 Runtime(Windows 10/11에는 기본 설치되어 있음)
- 관리자 권한(외부 창 프로세스를 제어하려면 필수)

```bash
npm install
npm run tauri dev
```

주요 단축키는 다음과 같다.

| 동작          | 단축키           |
| ------------- | ---------------- |
| 세션 전환     | `Alt+1` ~ `Alt+4` |
| 명령 팔레트   | `Ctrl+K`          |
| 사이드바 토글 | `Ctrl+B`          |
| 몰입 모드 종료 | `ESC`            |

이 외에도 VM 시작/중지/저장/재개/일시정지, CPU와 메모리 할당 조정,
스냅샷 생성·복원·삭제, 레지스트리 기반 RDP 연결 기록 자동 감지,
자산별 메모와 태그, CPU·메모리·디스크·IP·가동시간 실시간 추적,
지수 백오프 방식의 크래시 감지 및 자동 재연결,
Hyper-V 가상 스위치와 이벤트 로그 조회 기능을 제공한다.
테마는 다크·라이트·레트로 세 가지를 지원한다.

알려진 제약도 있다.
구조적으로 여러 창을 동시에 그리드로 재배치하는 것은 불가능하고,
VMware/Omnissa Horizon 지원은 대시보드 기반 아키텍처 특성상
창 캡처가 특히 까다로워 검증되지 않은 상태로 남아 있다.

## 분석

### 창 재배치는 미리보기 방식보다 근본적으로 다른 절충이다

화면 미리보기나 스크린샷 기반 원격 데스크톱 도구는 구현이 단순하고
플랫폼 종속성이 낮다.
반면 HyperDesk가 택한 방식은 실제 창을 그대로 옮겨 붙이기 때문에
세션이 살아있는 프로세스로 계속 동작하고, 사용자는 지연이나 프레임
손실 없이 원본 애플리케이션을 다루는 느낌을 받는다.
그 대가로 Win32 API에 깊이 의존하게 되고, 다른 프로세스의 창 핸들을
조작할 수 있는 관리자 권한이 필수가 된다.

`AttachThreadInput`을 의도적으로 배제한 결정도 이 절충을 잘 보여준다.
스레드 입력을 붙이는 방식이 구현하기는 더 간단할 수 있지만,
자식 프로세스가 모달 대화상자를 띄우는 순간 전체 UI가 멈추는
데드락 위험을 안는다.
독립 메시지 큐를 유지하는 대신 위치 동기화를 직접 구현해야 하는
복잡성을 떠안은 셈이다.

### VMConnect 리본을 지우는 방식은 임시 봉합에 가깝다

`SetWindowRgn`으로 VMConnect의 리본 UI를 가리는 방식은 결과적으로
깔끔한 단일 슬롯을 만들어내지만, 이는 VMConnect가 애초에 제공하지
않는 "리본 없는 모드"를 API 차원에서 흉내 낸 것이다.
Microsoft가 VMConnect의 UI 구조를 바꾸면 이 영역 클리핑 좌표가 어긋나
리본이 다시 드러나거나 콘텐츠 일부가 잘릴 수 있다.
이런 방식은 대상 애플리케이션의 UI 변경에 항상 취약하다.

## 비평

### "최대 4개, 그리드 불가"라는 제약은 사용 시나리오를 좁힌다

README는 구조적 제약으로 인해 여러 창을 동시에 그리드로 재배치할 수
없다고 명시한다.
즉 여러 세션을 동시에 보면서 작업하는 대시보드형 사용은 지원되지
않고, 오직 전환식 사용만 가능하다.
다중 모니터에서 여러 VM을 동시에 감시해야 하는 운영 환경에는
이 제약이 근본적인 한계로 작동할 수 있다.

### 관리자 권한 요구가 채택 장벽을 높인다

외부 프로세스의 창을 제어하려면 관리자 권한이 필수라는 점은
Win32 창 스왈로잉의 구조적 한계이지 HyperDesk만의 결함은 아니다.
그러나 기업 환경에서는 관리자 권한으로 실행되는 도구 자체가
보안 정책상 배포 승인을 받기 어려운 경우가 많다.
개인 프로젝트로서는 자연스러운 선택이지만,
바로 이 지점이 실제 업무 환경에서의 채택을 가로막을 수 있다.

### VMware 지원 미검증은 "지원 목록"의 신뢰도를 흐린다

README가 VMware/Omnissa 코드의 존재를 명시하면서도 프로덕션 미검증을
반복해서 경고하는 것은 정직한 태도이지만, 동시에 이 프로젝트가
Hyper-V/RDP 조합 바깥에서는 아직 실사용 단계가 아님을 보여준다.
제작자 스스로도 GeekNews에서 Omnissa의 대시보드 기반 아키텍처가
창 캡처를 특히 어렵게 만든다고 언급했는데,
이는 단순한 시간 부족이 아니라 아키텍처 차이에서 오는 구조적 난제일
가능성을 시사한다.

## 인사이트

### Win32 창 스왈로잉은 레거시 API가 여전히 유효한 해법을 제공하는 사례다

Tauri, React, Rust 같은 현대적인 스택 위에서 애플리케이션을 만들면서도
핵심 기능은 `SetParent`처럼 수십 년 된 Win32 API에 의존한다는 점이
흥미롭다.
운영체제가 공식적으로 지원하는 최신 창 관리 API가 아니라,
윈도우 초기부터 존재하던 저수준 핸들 조작이 여전히 가장 실용적인
해법으로 남아 있다는 사실은 플랫폼 API의 진화가 실제 개발자 수요를
항상 따라가지 못한다는 점을 보여준다.

### "탭"이 아니라 "실제 세션"을 유지하려는 요구가 이 프로젝트의 동기다

브라우저 탭이나 원격 데스크톱 클라이언트의 화면 미리보기 방식은
이미 널리 쓰이고 있지만, HyperDesk 제작자는 굳이 더 복잡한 창 재배치
방식을 택했다.
이는 사용자가 단순히 "화면을 보는 것"이 아니라 "실제로 살아있는
세션을 그대로 다루는 것"을 원한다는 신호로 읽을 수 있다.
백그라운드에서 세션이 계속 실행되고, 전환 시 지연이나 재연결이
없어야 한다는 요구는 다중 VM·RDP를 다루는 운영자나 개발자에게는
단순한 편의를 넘어 실질적인 생산성 차이로 이어진다.

### 개인 프로젝트가 상용 VDI 클라이언트의 틈새를 메운다

Citrix Workspace나 VMware Horizon Client 같은 상용 제품은 이미
다중 세션 관리 기능을 제공하지만, Hyper-V 환경을 개인 또는 소규모
팀 단위로 다루는 사용자에게는 과한 도구이거나 라이선스 비용이
부담일 수 있다.
HyperDesk는 이런 틈새에서 Rust와 Tauri라는 비교적 가벼운 스택으로
핵심 기능만 구현한 대안을 제시한다.
제작자 스스로 이것이 사이드 프로젝트이며 한계를 인지하고 있다고
밝힌 점은, 이 도구의 위치가 상용 제품의 완전한 대체재가 아니라
특정 워크플로를 가진 사용자를 위한 실용적 보완재임을 분명히 한다.
