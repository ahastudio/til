# LeafWiki – A wiki for people who think in folders, not feeds

> Self-hosted wiki.
> Single Go binary, SQLite, Markdown on disk.
> No external database required.

<https://leafwiki.com/>

<https://github.com/nicholasgasior/leafwiki>

GN 토론: <https://news.hada.io/topic?id=31510>

## 소개

LeafWiki는 폴더 구조로 사고하는 사람을 위한 셀프 호스팅 위키 애플리케이션이다.
단일 Go 바이너리 하나로 실행되며, 콘텐츠는 디스크의 마크다운 파일로 저장된다.
Node.js, Redis, PostgreSQL 같은 별도 서버나 데이터베이스가 필요 없고,
메타데이터 관리에는 SQLite를 사용한다.
MIT 라이선스로 배포된다.

런북, 홈랩 문서, 팀 노트, 개인 지식 기반처럼 폴더 트리 구조로 정리하는
문서 작업에 최적화되어 있다.

## 주요 기능

편집 환경은 분할 화면 구조로 마크다운과 렌더링 결과를 동시에 확인할 수 있으며,
자동 저장, Ctrl+V 이미지 붙여넣기, 옵시디언 호환 `[[wikilink]]` 문법을 지원한다.
네비게이션은 트리 구조 기반이며 전체 텍스트 검색, 태그 필터링,
빠른 이동(`Ctrl+Alt+P`)을 제공한다.

고급 기능으로는 수정 이력 추적(`--enable-revision` 옵션),
자동 링크 업데이트, 손상된 링크 감지, Mermaid 다이어그램, KaTeX 수식이 있다.

## 인증 모드

세 가지 운영 방식을 지원한다.

- **비공개 위키**: 편집과 읽기 모두 로그인 필수
- **공개 문서 사이트**: 읽기는 자유, 편집만 인증 필요
- **로컬 노트패드**: 인증 없음

Authentik, Authelia, Keycloak 같은 외부 인증 시스템과 연동할 수 있다.

## 플랫폼

Linux, macOS, Windows, Raspberry Pi에서 동작한다.
Docker 컨테이너, Ubuntu/Raspberry Pi OS용 스크립트,
각 플랫폼 바이너리 직접 다운로드 방식으로 설치할 수 있다.

## 분석

### "무거운 스택 없는 진짜 위키"라는 포지셔닝이 겨냥하는 시장

LeafWiki의 슬로건 "A real wiki app without the heavy stack"은
기존 위키 소프트웨어(Confluence, MediaWiki, Notion 등)가 운영하기
너무 복잡해진 현실을 겨냥한다.
대부분의 위키는 데이터베이스 서버, 캐시 레이어, 별도 웹 서버,
의존성 패키지 관리 등을 요구하는데, 홈랩이나 소규모 팀에서 이 모든
것을 관리하기에는 부담이 크다.
단일 Go 바이너리 + 마크다운 파일이라는 스택은 이 복잡성을 근본적으로
제거하는 선택이다.

마크다운 파일이 그대로 디스크에 저장된다는 사실은 또 다른 이점을
만든다.
콘텐츠가 특정 데이터베이스 포맷이나 앱 포맷에 갇히지 않으므로,
LeafWiki를 쓰다가 다른 도구로 이전할 때 마크다운 파일을 그대로
가져갈 수 있다.
`grep`으로 직접 검색할 수 있다는 점도 개발자 친화적인 특성이다.

### 옵시디언 호환 위키링크는 두 도구 사이의 이동 비용을 낮춘다

`[[wikilink]]` 문법을 지원한다는 것은 옵시디언 사용자가 별도
변환 없이 LeafWiki로 이전하거나, 두 도구를 같은 파일셋으로 함께
쓸 수 있다는 의미다.
옵시디언은 강력한 개인 지식 관리 도구이지만 팀 공유와 웹 접근이
불편하고, LeafWiki는 반대로 팀 협업과 웹 접근에 강하다.
같은 마크다운 파일을 두 도구로 읽을 수 있다면 각자의 강점을
겹치지 않게 활용할 수 있다.

## 비평

### 실시간 협업 편집 부재는 명시적 한계다

LeafWiki는 동시 편집을 지원하지 않는다고 명시한다.
이는 Notion이나 Confluence처럼 여러 명이 같은 문서를 동시에
편집해야 하는 환경에서는 선택지가 되지 못한다는 뜻이다.
소규모 팀이나 개인 사용에서는 큰 문제가 아니지만,
"팀 운영 설명서"를 표방하면서 이 제약을 갖는 것은 사용 맥락이
제한된다는 신호다.

### v0.x 단계와 커뮤니티 크기

GN 토픽 기준으로 댓글이 없었다는 사실은 아직 한국어권 개발자 사용
사례가 충분히 쌓이지 않았음을 보여준다.
단일 개발자 프로젝트의 특성상 장기 유지보수 지속성이 불확실하며,
버전이 올라가면서 마크다운 파일 포맷이나 폴더 구조가 바뀔 경우
마이그레이션 부담이 생길 수 있다.

## 인사이트

### 마크다운 파일 기반 스택은 도구 종속을 거부하는 설계 원칙이다

LeafWiki처럼 콘텐츠를 평범한 마크다운 파일로 유지하는 설계는
"도구가 바뀌어도 데이터는 남는다"는 철학을 구현한다.
이는 SaaS 위키처럼 서비스가 종료되거나 가격이 오르면 데이터를
빼내기 어려운 구조와 대척점에 있다.
지식 관리 도구 선택에서 "내 데이터를 얼마나 쉽게 꺼낼 수 있는가"를
기준으로 삼는다면, 마크다운 + Git 조합이 가장 낮은 탈출 비용을
제공한다.

### 단일 바이너리 배포는 인프라 관리 부담을 최소화하는 트렌드를 반영한다

Go로 작성된 단일 바이너리 배포는 LeafWiki만의 특성이 아니라
Caddy, Litestream, Fly.io의 flyctl 등 최근 인프라 도구에서
반복해서 나타나는 패턴이다.
설치, 업그레이드, 이전이 바이너리 하나의 교체로 끝나는 이 방식은
운영 복잡성을 극적으로 줄인다.
Raspberry Pi처럼 자원이 제한된 환경에서도 동작하도록 설계했다는
점은 이 도구의 주요 사용자가 클라우드보다 로컬 혹은 엣지에
가깝다는 신호다.
