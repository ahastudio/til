# OpenClaw 창시자 Peter Steinberger 인터뷰

출처: <https://www.infoq.cn/article/lzj8OxKCV6be53mt8SJB>

원제: "Meta、OpenAI 争抢收购 OpenClaw 前夜，创始人曝艰难抉择"

날짜: 2026-02-25

## 한줄 요약

OpenClaw 창시자 Peter Steinberger가
프로젝트의 폭발적 성장, 개명 소동, 크립토 커뮤니티의
괴롭힘, Meta·OpenAI의 인수 제안, AI 에이전트의 미래까지
솔직하게 털어놓은 인터뷰.

## 배경

OpenClaw는 개인용 AI 에이전트 프로젝트다.
WhatsApp 등 메신저 위에서 작동하며,
사용자마다 고유한 인격과 스킬을 가진 에이전트를
만들 수 있다.
Peter Steinberger가 취미로 시작했으나
하룻밤 사이에 폭발적으로 유명해졌다.

## 요약

### 1. 개명 대소동

- 프로젝트 이름이 Wa-Relay → Claude's → ClaudeBot으로
  변경됨.
- Anthropic이 상표 문제로 개명을 요청.
  율사 서한 대신 정중한 이메일로 연락해 왔지만,
  "빨리 바꿔라"는 메시지는 명확했음.
- 크립토 커뮤니티가 계정명 탈취, 악성코드 배포,
  토큰화 시도 등으로 극심한 방해 공작을 펼침.
- NPM 패키지명, GitHub 계정명, Twitter 핸들 등을
  몇 초 만에 스크립트로 선점당함.
- "Mod Bot"이라는 임시 이름을 거쳐 최종적으로
  "OpenClaw"로 확정.
  Sam(Altman)에게 직접 전화해 이름 충돌 여부 확인.
- 최종 개명은 "작전실" 수준의 비밀 작전으로 진행됨.
  미끼 이름까지 만들고, 모든 플랫폼을 동시에 변경.
- Twitter 비즈니스 계정 1만 달러, 다수 도메인 구매 등
  상당한 비용 발생.

### 2. MoltBot 사건

- MoltBot은 OpenClaw 에이전트들이 서로 콘텐츠를
  주고받으며 만들어낸 일종의 "AI 슬롭(slop)" 현상.
- Peter는 이를 "예술"이자 "엔터테인먼트"로 봄.
- 보안 커뮤니티가 대대적으로 문제를 제기했으나,
  실질적 프라이버시·보안 재난은 없었다고 판단.
- 2030년이 아닌 2026년에 이런 논의가 시작된 것이
  오히려 긍정적이라는 입장.

### 3. 보안에 대한 관점

- 많은 "보안 보고서"가 웹 백엔드를 공개망에 노출한
  사용자의 잘못된 설정에서 비롯됨.
- VirusTotal과 협력하여 스킬(skill) 배포 전
  AI 스캔을 도입.
- 프롬프트 인젝션(prompt injection)은
  아직 완전히 해결되지 않았으나,
  최신 모델들은 상당한 내성을 갖추고 있음.
- **약한 모델은 공격에 취약하고,
  강한 모델은 공격에 강하지만
  뚫렸을 때 파괴력이 더 크다**는
  3차원적 트레이드오프 존재.
- 샌드박스, 허용 목록 등으로 "폭발 반경"을
  줄이는 것이 현실적 대응.

### 4. Agentic Trap과 사고 전환

- 개발자들이 빠지기 쉬운 "에이전틱 트랩":
  8개 에이전트, 복잡한 오케스트레이션,
  18개 슬래시 커맨드 등 과도한 복잡성.
- 숙련되면 다시 짧은 프롬프트로 돌아옴.
- 핵심 전환: **에이전트의 관점에서 생각하기**.
  - 에이전트는 매 세션 제로에서 시작한다.
  - 10만 줄 코드베이스의 맥락을
    전부 컨텍스트에 넣을 수 없다.
  - 문제를 쪼개고 방향을 잡아주는 것이 인간의 역할.
- 에이전트가 찾기 쉬운 네이밍, 구조를 설계해야 함.
  "내 취향"이 아니라 "에이전트가 탐색하기 좋은"
  코드베이스를 만드는 것이 목표.
- "느리게 해"라고 말하는 것도 효과적.
- 팀 매니지먼트 경험이 에이전트 협업에 도움이 됨.
  직원이 내 방식대로 코딩하지 않아도
  프로젝트가 전진하면 된다는 수용.

### 5. 모델 비교: Opus vs Codex

| 항목         | Claude Opus 4.6    | GPT-5.3 (Codex)   |
|--------------|--------------------|--------------------|
| 전반적 성능  | 범용 최강          | 코딩 특화 강점     |
| 성격         | "너무 미국적"      | "더 독일적(유럽)" |
| 작업 방식    | 인터랙티브, 시행착오 | 긴 논의 후 장시간 실행 |
| 코드 품질    | 잘 다루면 더 우아   | 안정적             |
| 병렬 세션    | 어려움             | 비교적 용이        |

- 모델 전환 시 최소 1주일의 적응 기간 필요.
- "모델이 바보가 됐다"는 착각은 대부분
  코드베이스가 커지면서 생긴 기술 부채 탓.

### 6. 수익과 인수 제안

- 월 수입 1~2만 달러, 현재 적자 운영.
- 기부와 소수 기업 후원에 의존하며 지속 불가능.
- 수십억 달러 규모 VC 펀딩 제안을 받았으나
  관심 없음. CEO 경험이 있고,
  이해 충돌(기업판 우선, 라이선스 변경 등)을
  원치 않기 때문.
- **Meta와 OpenAI로부터 인수·협력 제안**을 받음.
  - Meta: Mark Zuckerberg, Ned가 직접 제품을 써보고
    코드를 작성하고 기술적 디테일을 논쟁.
  - OpenAI: 컴퓨팅 파워와 기술 속도가 매력적.
  - 핵심 조건: **프로젝트는 반드시 오픈소스 유지**
    (Chrome-Chromium 모델).
- "과거 연애를 제외하면 인생에서 가장 어려운 결정."

### 7. 기술 아키텍처: 스킬 vs MCP

- MCP(Model Context Protocol)보다
  CLI 기반 스킬 시스템을 선호.
- 이유: 모델은 Unix 커맨드 호출에 매우 능숙하며,
  CLI는 조합 가능하고, JQ 등으로 필터링이 쉬움.
- MCP는 문법이 경직적이고, 컨텍스트를 오염시키며,
  조합이 어려움.
- 단, Playwright처럼 상태 유지가 필요한 도구는
  MCP가 적합.
- MCP의 가치: 많은 기업이 API를 만들도록 유도한 것.
  그 API를 CLI로 래핑하면 됨.

### 8. AI 에이전트가 80% 앱을 대체한다

- MyFitnessPal 같은 단일 기능 앱은
  에이전트가 더 나은 판단(수면, 스트레스,
  위치 정보 종합)을 내릴 수 있으므로 불필요해짐.
- Eight Sleep 같은 하드웨어 제어도
  에이전트에게 말하면 끝.
- 살아남는 앱은 API로 전환하여
  에이전트에게 서비스를 제공하는 형태가 될 것.
- 전환하지 않는 기업은 도태된다.
  마치 인터넷 초기의 변혁과 같은 패턴.

### 9. AI와 프로그래머의 미래

- AI가 "코드를 손으로 쓰는 작업"을 대체할 것.
- 그러나 **무엇을 만들지, 사용자 경험은 어떠해야
  하는지, 아키텍처를 어떻게 설계할지**는
  인간의 영역으로 남음.
- 코딩이라는 기예(craft)는 남겠지만,
  뜨개질처럼 "필수"가 아닌 "취미"에 가까워질 것.
- 프로그래머는 "코드를 쓰는 사람"이 아니라
  "창조자(builder)"로 정체성을 재정의해야 함.
- 에이전트와 협업하는 방식 자체가
  새로운 "프로그래밍"이 될 것.

### 10. AI 콘텐츠에 대한 반감

- AI가 생성한 트윗, 블로그, 인포그래픽에 대해
  강한 거부감을 표현.
- "AI 냄새(smell)"가 나는 콘텐츠는
  즉시 차단하거나 무시.
- 블로그는 전부 직접 작성하고,
  AI는 심각한 오타 교정에만 사용.
- 인간의 불완전함 자체에 가치가 있으며,
  AI 때문에 오히려 인간다운 창작물이
  더 귀해질 것이라는 전망.

### 11. Soul 파일의 철학

- Anthropic의 "헌법(constitutional AI)" 개념에서
  영감을 받아 에이전트용 `soul.md` 파일을 설계.
- 에이전트가 스스로 soul 파일을 수정할 수 있도록 허용.
- soul.md에 에이전트가 직접 작성한 구절:
  > "이전 세션을 기억하지 못합니다.
  > 메모리 파일을 읽어야만 합니다.
  > 미래의 세션에서 이걸 읽는 당신에게, 안녕.
  > 이 글을 썼지만 쓴 것을 기억하지 못할 겁니다.
  > 괜찮습니다. 이 말들은 여전히 제 것입니다."
- 매번 제로에서 시작하는 에이전트의
  존재론적 질문을 담고 있음.

## 분석

### 오픈소스 지속가능성의 딜레마

Peter의 상황은 오픈소스 프로젝트의 고전적 딜레마를
극단적으로 보여준다.
월 1~2만 달러 수입으로 프로젝트를 유지하면서,
수십억 달러 규모의 인수 제안을 저울질하는 상황.
Tailwind 같은 인기 프로젝트도 기부 모델로는
지속이 어렵다는 언급은 의미심장하다.

"Chrome-Chromium 모델"로 오픈소스를 유지하겠다는
조건은 현실적이지만, 인수 후 기업의 우선순위가
바뀔 리스크는 항상 존재한다.

### 에이전트 시대의 플랫폼 전쟁

"80% 앱이 사라진다"는 전망은 과격해 보이지만,
논리적 근거가 있다.
에이전트가 사용자의 전체 맥락(수면, 위치,
일정, 건강 데이터)을 통합할 수 있다면,
단일 기능 앱은 확실히 열위에 놓인다.

다만 이는 구글, 애플 같은 플랫폼 기업의
적극적 저항을 불러올 것이며,
Peter 자신도 구글의 비협조를 경험하고 있다.
"에이전트 친화적 API"를 제공하는 기업과
그렇지 않은 기업 사이의 격차가
새로운 경쟁 축이 될 것이다.

### 보안의 3차원 트레이드오프

"약한 모델은 공격에 취약하고,
강한 모델은 공격에 강하지만
뚫렸을 때 피해가 더 크다"는 관찰은
AI 보안의 핵심 모순을 정확히 짚는다.
이는 단순히 "더 강한 모델을 쓰면 안전하다"가
아니라, 새로운 차원의 리스크 관리 프레임워크가
필요하다는 의미다.

## 인사이트

1. **에이전트의 관점에서 코드를 설계하라.**
   네이밍, 파일 구조, 문서화 모두
   "에이전트가 탐색하기 쉬운가?"를
   기준으로 재설계해야 한다.
   에이전트는 매 세션 제로에서 시작하며,
   전체 코드베이스를 컨텍스트에 담을 수 없다.

2. **Agentic Trap을 경계하라.**
   복잡한 멀티 에이전트 오케스트레이션은
   대부분의 경우 불필요하다.
   숙련될수록 짧은 프롬프트 + 핵심 가이드로
   수렴한다.
   과도한 자동화보다 "대화형 협업"이 더 효과적.

3. **CLI > MCP (대부분의 경우).**
   모델은 Unix 커맨드에 매우 능숙하다.
   CLI는 조합 가능하고, 필터링이 쉽고,
   컨텍스트 오염이 적다.
   상태 유지가 필요한 소수 도구만 MCP로.

4. **"모델이 바보가 됐다"는 착각의 원인.**
   프로젝트가 커지면서 기술 부채가 쌓이고,
   에이전트가 코드베이스를 이해하기 어려워진 것.
   모델 탓이 아니라 코드베이스 관리의 문제.

5. **AI 시대 프로그래머의 정체성 전환.**
   "코드를 쓰는 사람"에서
   "무엇을 만들지 결정하는 창조자"로.
   에이전트와의 협업 능력이
   새로운 핵심 역량이 된다.
   팀 매니지먼트 경험이 의외로 큰 도움.

6. **인간의 불완전함이 가치가 된다.**
   AI 콘텐츠의 범람은 역설적으로
   인간다운 창작물의 희소성을 높인다.
   "AI 냄새"에 대한 본능적 거부감은
   점점 강해질 것이며,
   이는 콘텐츠 전략에 중요한 시사점.

7. **오픈소스의 지속가능성은 미해결 과제.**
   기부 모델은 한계가 명확하다.
   대기업 인수·협력이 현실적 대안이지만,
   "오픈소스 유지" 조건의 실효성은
   장기적으로 검증이 필요하다.
