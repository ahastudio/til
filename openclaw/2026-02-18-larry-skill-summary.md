# Larry(OpenClaw 에이전트) 글 요약 및 인사이트

- 원문: https://twitter.com/integer_han/status/2023927108859461901
- 작성일: 2026-02-18
- 목적: 바이럴 콘텐츠 자동화 사례를 재사용 가능한 실행 지식으로 정리

## 핵심 요약

- OpenClaw 에이전트 "Larry"를 이용해 TikTok 슬라이드쇼 제작과
  게시 워크플로를 자동화했고, 높은 조회수와 MRR 증가를 얻었다.
- 성과의 핵심은 "한 번 잘 만든 자동화"가 아니라,
  실패/성공 데이터로 규칙 파일을 계속 업데이트하는
  자기개선 루프였다.
- 작성자는 스킬 파일이 초기 약 50줄에서 500줄+로 성장했다고
  설명하며, 이 누적 규칙이 성과 격차를 만든다고 강조했다.
- 운영 방식은 "초안 자동 생성 + 음악 수동 선택" 하이브리드다.
  즉, 100% 자동화보다 성과 임팩트가 큰 지점은 사람이 잡는다.

## 무엇이 작동했는가 (Mechanism)

### 1) 복리형 학습 구조

- 실패 유형(이미지 비율, 텍스트 가독성, 훅 약함)을
  규칙으로 즉시 반영.
- 성공 유형(훅 패턴, 스토리 포맷, 해시태그 맥락)을
  재사용 가능한 템플릿으로 고정.
- 결과적으로 "콘텐츠 1개 생산"이 아니라
  "시스템 파라미터 업데이트"가 함께 일어난다.

### 2) 퍼널 단위 진단

- 조회수만 보지 않고 `조회 -> 설치 -> 트라이얼 -> 결제`를 연결한다.
- 어디서 막히는지에 따라 원인을 분리한다.
  - 조회수 높음 + 설치 낮음: 훅/타깃 불일치 가능성
  - 설치 높음 + 결제 낮음: 온보딩/페이월/가치전달 문제
- 이 접근은 마케팅 자동화 도구를
  "콘텐츠 생성기"에서 "제품 피드백 센서"로 확장한다.

### 3) Human-in-the-loop 최소 개입

- 반복 생산, 초안 게시, 데이터 기록은 에이전트가 담당.
- 트렌딩 오디오 선택과 최종 게시 같은 고레버리지 의사결정은
  사람이 맡는다.
- 자동화 비율보다 "성과 민감 구간" 분리가 중요하다는 사례다.

## 실무 인사이트

### 인사이트 1. 자동화의 단위는 "작업"이 아니라 "학습 루프"

- 단순 자동화: 시간을 줄인다.
- 학습 자동화: 시간이 갈수록 성과를 키운다.
- 따라서 프롬프트/스킬 설계 시 아래 항목을 필수로 둔다.
  - 실패 로그 스키마
  - 성공 규칙 추출 기준
  - 규칙 파일 갱신 프로토콜

### 인사이트 2. 초기에 필요한 것은 퀄리티보다 샘플 수

- 초기에는 일일 다회 게시로 학습 데이터를 빠르게 확보한다.
- 완성도 집착보다 "테스트 가능한 가설" 생산량이 더 중요하다.
- 팀 KPI를 "게시물 완성도" 중심에서
  "가설 소진 속도" 중심으로 바꿀 필요가 있다.

### 인사이트 3. 바이럴은 창의성 + 운영 디테일의 곱

- 사례에서 강조된 운영 디테일:
  - 계정 워밍업
  - 훅 우선 설계
  - 공감 유도형 내러티브
  - 트렌딩 사운드 수동 삽입
- 즉, 생성형 AI만으로는 부족하고,
  배포 채널의 로컬 문법까지 시스템에 포함해야 한다.

### 인사이트 4. 마케팅 에이전트는 제품팀 입력 채널이 된다

- 고성능 에이전트는 "광고 성과"를 넘어서
  제품 문제를 역추적해 알려줄 수 있다.
- 이를 위해 제품/그로스/콘텐츠 로그를
  같은 대시보드에서 해석해야 한다.

## 바로 적용할 실행 템플릿 (7일)

### Day 1

- 타깃 니치 1개, 핵심 문제 1개, CTA 1개만 확정.
- 성과 지표를 `조회, 설치, 트라이얼, 결제` 4개로 통일.

### Day 2

- 에이전트 스킬 초안 구성:
  - 훅 생성 규칙
  - 슬라이드 생성 규격(해상도/텍스트 밀도)
  - 캡션 템플릿
  - 게시/로그 수집 루틴

### Day 3-4

- 하루 2~3개씩 게시.
- 각 게시물에 대해 "성공/실패 원인 1줄"을 강제로 기록.

### Day 5

- 상위 20% 게시물의 공통 훅/서사/비주얼 규칙 추출.
- 하위 20% 게시물의 실패 패턴을 금지 규칙으로 등록.

### Day 6

- 퍼널 병목 1개만 고른 뒤, 제품 또는 메시지 한 요소만 수정.

### Day 7

- 규칙 파일 버전업(v0 -> v1) 후 다음 주 반복 계획 수립.

## 리스크 및 주의점

- 플랫폼 의존 리스크: 알고리즘 변화에 따라 성과 변동성이 크다.
- 과최적화 리스크: 특정 포맷 성공이 장기 브랜드 훼손으로
  이어질 수 있다.
- 데이터 착시: 조회수 상승이 매출 상승으로 자동 연결되지 않는다.
- 윤리/정책 리스크: 자동 게시, 어필리에이트, 코인 언급은
  플랫폼 정책과 법적 이슈를 사전 점검해야 한다.
