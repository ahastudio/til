# Obsidian CEO의 Obsidian 사용법

<https://stephango.com/vault>

Obsidian CEO Steph Ango가 공개한 자신의 Vault 운영법.
단순한 설정 가이드가 아니라, 수년간 노트를 쓰며
다듬어온 사고 체계이자 인생 운영 시스템이다.

## 핵심 철학: File over App

> If you want to create digital artifacts that last,
> they must be files you can control, in formats that
> are easy to retrieve and read.

Obsidian에서 "Vault"는 그냥 파일 폴더다.
이것이 중요하다. 내 파일을 내가 소유하고,
쉽게 꺼내고, 어디서든 읽을 수 있다.
Obsidian은 그 자유를 보장한다.

→ CEO가 자기 제품의 유한함을 인정하는 것
자체가 제품 철학의 진정성을 보여준다.
앱은 사라지지만 파일은 남는다.

→ 1,500개 이상의 플러그인 생태계를 가진
플랫폼의 CEO가 쓰는 플러그인이 이게 전부다.
도구의 힘은 양이 아니라 일관성에 있다는 걸
몸소 보여준다.

## 스타일 가이드: 8가지 규칙

| 규칙                               | 이유                      |
|------------------------------------|---------------------------|
| 여러 Vault로 쪼개지 않는다         | 맥락 단절 방지            |
| 폴더로 정리하지 않는다             | 하나의 노트가 여러 영역에 |
|                                    | 속할 수 있다              |
| 비표준 마크다운을 피한다           | 이식성 확보               |
| 카테고리와 태그는 항상 복수형      | 네이밍 고민 제거          |
| 내부 링크를 아낌없이 건다          | 발견 가능성 극대화        |
| 날짜는 어디서나 YYYY-MM-DD         | 정렬과 검색의 일관성      |
| 평점은 7점 만점 정수               | 상위 구간의 세밀한 구분   |
| 주간 할 일 목록은 하나만           | 집중력 유지               |

일관된 스타일은 미래의 수백 가지 결정을
하나로 축약한다. 예를 들어 태그를 항상
복수형으로 쓰면 새 태그 이름을 고민할 일이
없다. 자기에게 편한 규칙을 고르고, 적어 두고,
나중에 바꿔도 된다.

→ 규칙이 많아 보이지만, 실은 매순간의
판단 비용을 제로로 만드는 장치다.

## 폴더 구조

Ango는 폴더를 최소화하지만 완전히 없애지는
않는다. 하위 폴더는 쓰지 않는다.
파일 탐색기도 거의 쓰지 않는다.
Quick Switcher, 백링크, 노트 내 링크로
탐색한다.

노트 정리는 주로 categories 속성으로 한다.
Obsidian Bases 기능으로 카테고리별 개요를
볼 수 있다.

### 루트 폴더: 나의 세계

대부분의 노트는 Vault 루트에 있다.
저널, 에세이, 에버그린 노트, 개인 노트.
루트에 있으면 "내가 쓴 것" 또는
"나와 직접 관련된 것"이라는 뜻이다.

### 참조 폴더

```
References/    내 세계 밖의 것들.
               책, 영화, 장소, 인물, 팟캐스트 등.
               항상 제목으로 파일명을 짓는다.
               (예: Book title.md, Movie title.md)

Clippings/     다른 사람이 쓴 글.
               에세이, 아티클 등을 저장한다.
```

### 관리 폴더

```
Attachments/   이미지, 오디오, 비디오, PDF 등.
Daily/         일일 노트. 파일명은 YYYY-MM-DD.md.
               여기에 직접 쓰지 않는다.
               다른 노트에서 링크하기 위해서만
               존재한다.
Templates/     노트 템플릿.
```

→ 폴더는 "종류"를 나누는 데만 쓰고,
"주제"는 링크와 태그로 해결한다.
이것이 "폴더를 피한다"의 진짜 의미다.
폴더를 없앤 게 아니라 폴더의 역할을 재정의했다.

→ 일일 노트에 아무것도 쓰지 않는다는 점이
특히 흥미롭다. 날짜 노트는 "앵커"일 뿐이다.
실제 기록은 Unique Note로 하고, 거기서
날짜 노트를 링크한다. 시간축과 내용을
분리하는 설계다.

## 링크 전략

> 첫 번째 언급에는 항상 링크를 건다.

저널 항목은 의식의 흐름처럼 최근 사건을
기록하고, 사물 사이의 연결을 찾는다.
아직 노트가 없는 곳에도 링크를 건다.
미해결 링크(unresolved link)는 미래의
연결을 위한 빵 부스러기다.

### 예시

```markdown
영화 [[Perfect Days]]를 [[Aisha]]와
[[Vidiots]]에서 봤고, [[Little Ongpin]]에서
필리핀 음식을 먹었다. Perfect Days의
이 대사가 좋았다:
[[Next time is next time, now is now]].
에세이 ...가 떠올랐다.
```

- 영화, 극장, 레스토랑 → References 폴더
- 의미 있는 인용문 → 루트의 에버그린 노트
- 남이 쓴 에세이 → Clippings 폴더

→ 이 무거운 링크 스타일은 시간이 지날수록
강력해진다. 아이디어가 어떻게 출현했는지,
어떤 갈래를 만들었는지 추적할 수 있다.

→ "자기 생각"과 "남의 생각"을 물리적으로
분리하면서 링크로는 연결한다. 출처를 잃지 않되,
자기 사고에 섞이지 않게 하는 설계다.

## 프랙탈 저널링과 랜덤 재발견

> Fractal journaling and randomization are how
> I tame the wilderness that a knowledge base
> can grow into.

### 포착

하루 중 생각이 떠오르면 Unique Note 단축키를
누른다. `YYYY-MM-DD HHmm` 접두사가 자동으로
붙은 노트가 생성되고, 아이디어를 설명하는
제목을 추가할 수도 있다.

### 리뷰 사이클

```
매일의 조각 → 며칠마다 정리 → 월간 리뷰
→ 연간 리뷰 (템플릿 사용)
```

짧은 생각 조각들을 며칠마다 리뷰하며
핵심을 추린다. 그 리뷰를 월간으로 다시
리뷰하고, 월간 리뷰를 연간으로 다시 리뷰한다.
결과는 인생의 "프랙탈 웹"이다.
다양한 해상도로 확대/축소하며 볼 수 있고,
하나의 생각이 어디서 시작되어 어떤 주제로
성장했는지 역추적할 수 있다.

### 랜덤 재발견

몇 달마다 시간을 내어 랜덤 노트 단축키로
Vault를 무작위 탐색한다. 로컬 그래프를 얕은
깊이로 펼쳐 관련 노트를 살핀다.
오래된 아이디어를 재방문하고, 빠진 링크를
만들고, 과거의 생각에서 영감을 얻는다.
스타일 가이드에 맞게 포맷을 고치는
유지보수 시간이기도 하다.

### "이해를 위임하지 마라"

> People have asked me if this could be automated
> with language models but I do not care to do so.
> I enjoy this process. Doing this maintenance
> helps me understand my own patterns.
> Don't delegate understanding.

→ AI 시대에 가장 역설적인 조언이다.
AI가 할 수 있는 일을 일부러 직접 하라는 것.
정리와 리뷰의 목적은 깔끔한 결과물이 아니라,
그 과정에서 생기는 패턴 인식이기 때문이다.

## 속성과 템플릿

거의 모든 노트가 템플릿에서 시작한다.
게으르게 정보를 추가하면서도 나중에 찾을 수
있도록 돕는 장치다. 카테고리마다 고유한
템플릿이 있고, 상단에 속성을 넣는다.

### 캡처하는 속성들

- 날짜: 생성일, 시작일, 종료일, 발행일
- 인물: 저자, 감독, 아티스트, 출연진,
  호스트, 게스트
- 주제: 장르, 유형, 토픽, 관련 노트
- 위치: 동네, 도시, 좌표
- 평점

### 속성 운영 규칙

- 속성명과 값은 카테고리 간 재사용을 목표.
  `genre`를 모든 미디어 타입에 공유하면
  SF 책, 영화, 드라마를 한곳에서 볼 수 있다.
- 템플릿은 조합 가능해야 한다.
  Person과 Author는 별도 템플릿이지만
  같은 노트에 추가할 수 있다.
- 짧은 속성명이 빠르다.
  `start-date`가 아니라 `start`.
- 값이 여러 개가 될 가능성이 조금이라도
  있으면 text 대신 list 타입을 기본으로 쓴다.

→ 속성 설계 자체가 하나의 데이터 모델링이다.
정규화(재사용 가능한 속성명), 유연성(list
타입 기본), 조합성(템플릿 합성)이라는
데이터베이스 설계 원칙이 그대로 적용되어 있다.

## 평점 시스템: 왜 7점인가

| 점수 | 의미                                      |
|------|-------------------------------------------|
| 7    | 완벽. 반드시 경험해야 한다. 인생이 바뀜   |
| 6    | 훌륭. 반복할 가치가 있다                  |
| 5    | 좋음. 일부러 찾아갈 정도는 아니지만 즐김  |
| 4    | 무난. 급할 때 쓸 만하다                   |
| 3    | 나쁨. 가능하면 피하라                     |
| 2    | 최악. 적극적으로 피하라. 역겨움           |
| 1    | 악. 인생이 나쁜 쪽으로 바뀜               |

4점이나 5점 만점은 상위 구간이 너무
뭉뚱그려지고, 10점은 너무 세밀하다.
7점은 좋은 경험을 세밀하게 구분하는 데
최적화된 스케일이다.

→ 평점의 목적은 정밀한 측정이 아니라
"다시 할 것인가?"의 빠른 판단이다.
5점에서는 4점과 5점의 차이가 모호하지만,
7점에서는 5, 6, 7이 각각 뚜렷한 의미를 가진다.

## 퍼블리싱 파이프라인

사이트는 Obsidian에서 직접 작성, 편집, 발행한다.
이를 위해 자신의 규칙 하나를 깬다.
사이트용 별도 Vault를 사용한다.

```
Obsidian에서 작성
    ↓
Obsidian Git으로 GitHub에 푸시
    ↓
Jekyll이 마크다운을 HTML로 컴파일
    ↓
Netlify가 자동 배포
```

설정은 좀 기술적이지만 사용은 쉽다.
사이트 레이아웃의 모든 측면을 제어하고
싶기 때문이다. 그런 제어가 필요 없다면
Obsidian Publish가 더 간단하다.
컬러 팔레트는 자체 제작한 Flexoki.

→ "규칙을 만든 사람이 규칙을 깨는" 모습이
인상적이다. 교조적이지 않다는 선언과 일치한다.
규칙은 도구지 족쇄가 아니다.

## 인사이트

### 1. 시스템의 핵심은 "안 하는 것"이다

Ango의 시스템에서 가장 눈에 띄는 것은
"하지 않기로 한 것들"이다.

- 폴더로 분류하지 않는다
- 여러 Vault로 나누지 않는다
- 복잡한 플러그인 스택을 쌓지 않는다
- 비표준 문법을 쓰지 않는다
- 일일 노트에 직접 쓰지 않는다

생산성 시스템은 보통 "무엇을 하느냐"로
정의되지만, Ango의 시스템은 "무엇을
하지 않느냐"로 정의된다. 제약이 곧 자유다.

### 2. 구조는 만드는 게 아니라 발견하는 것이다

폴더와 카테고리로 먼저 구조를 만들고
거기에 노트를 끼워 넣는 "탑다운" 방식 대신,
노트를 먼저 쓰고 링크가 자연스럽게 구조를
드러내게 하는 "바텀업" 방식을 택했다.

→ 소프트웨어 개발의 "진화적 설계"와
같은 원리다. 미리 완벽한 아키텍처를 만들지
말고, 코드를 쓰며 패턴이 드러나면 그때
리팩터링하라. 노트에서도 같은 원리가 작동한다.

### 3. 프랙탈 리뷰는 "생각의 컴파일러"다

일간 → 수일 → 월간 → 연간 리뷰 사이클은
단순한 정리가 아니다. 각 단계에서 추상화
수준이 한 단계씩 올라간다.

- 일간: 원시 데이터 (생각 조각)
- 수일: 패턴 인식 (관련 생각 묶기)
- 월간: 테마 추출 (큰 그림 파악)
- 연간: 방향성 점검 (인생 수준 반성)

→ 데이터 파이프라인의 ETL과 같다.
Extract(포착) → Transform(리뷰/정제)
→ Load(상위 리뷰에 적재). 인간의 사고를
데이터 처리 관점에서 최적화한 시스템이다.

### 4. 게으름을 설계에 포함시켰다

"속도와 게으름에 최적화된 시스템"이라고
직접 표현한다. 폴더 선택 없이 즉시 기록,
분류 고민 없이 링크만 추가, 태그 이름 고민
없이 복수형 규칙 적용.

→ 좋은 시스템은 의지력에 의존하지 않는다.
마찰이 적을수록 지속 가능성이 높다.
UX 디자인의 핵심 원칙이기도 하다.
"사용자(자기 자신)가 게을러도 작동하는
시스템을 만들어라."

### 5. CEO가 자기 규칙을 깨는 유연함

"폴더를 피하라", "Vault를 나누지 마라"는
규칙을 만들어 놓고, 사이트 퍼블리싱을 위해
별도 Vault를 사용한다. 이것은 모순이 아니라
실용주의다.

→ 규칙은 맹목적으로 따르는 교리가 아니라,
기본값(default)이다. 더 나은 이유가 있으면
깨도 된다. 이것이 "교조적이지 않다"는
첫 문장의 진짜 의미다.

### 6. "이해를 위임하지 마라"의 깊은 의미

AI로 리뷰를 자동화할 수 있냐는 질문에
"그럴 생각이 없다"고 답한다.
유지보수 과정이 자기 패턴을 이해하는
과정이기 때문이다.

→ AI 시대의 가장 중요한 구분선이 여기 있다.
"AI가 할 수 있는 것"과 "AI에게 맡겨야 하는
것"은 다르다. 효율의 관점에서는 자동화가
맞지만, 성장의 관점에서는 직접 해야 한다.
Ango는 노트 정리를 "생산성 작업"이 아니라
"사고 훈련"으로 본다.

### 7. 의도적 세렌디피티의 힘

랜덤 노트 탐색은 단순한 향수가 아니다.
정리된 시스템의 약점은 "예상한 것만
찾게 된다"는 것이다. 검색은 이미 뭘
찾을지 아는 사람을 위한 도구다.

→ 랜덤 탐색은 "무엇을 모르는지 모르는 것"을
발견하게 한다. 기계 학습의
Exploration vs Exploitation 트레이드오프와
같다. 가끔 무작위로 탐색해야 전역 최적해에
가까워진다.

### 8. 미해결 링크는 "미래의 씨앗"이다

아직 존재하지 않는 노트에 링크를 거는 것은
Ango 시스템의 핵심 습관이다.
이것은 단순한 메모 미완성이 아니라,
미래에 탐구할 주제를 의도적으로 심어두는 것.

→ 소프트웨어의 TODO 주석과 비슷하지만
더 강력하다. TODO는 잊히지만, 미해결 링크는
다른 노트에서 참조될 때마다 다시 떠오른다.
시스템 자체가 "이거 아직 안 했어"라고
알려주는 구조다.

### 9. 속성 설계는 개인 온톨로지 구축이다

`genre`를 모든 미디어에 공유하고,
Person과 Author 템플릿을 조합하고,
list 타입을 기본으로 쓰는 것은
개인 지식의 온톨로지(존재론)를 만드는 것이다.

→ 이것은 데이터베이스 정규화와 같은 원리다.
잘 설계된 스키마가 미래의 쿼리를 가능하게 하듯,
잘 설계된 속성 체계가 미래의 발견을
가능하게 한다. "SF"라는 장르로 책, 영화,
드라마를 한 번에 볼 수 있는 것은
처음부터 속성을 공유했기 때문이다.

### 10. 노트 시스템은 결국 사고 시스템이다

Ango의 Vault는 단순한 정보 저장소가 아니다.
포착 → 리뷰 → 연결 → 재발견의 사이클은
인간의 사고 과정 자체를 외부화한 것이다.

→ 이것이 "세컨드 브레인"의 진짜 의미다.
뇌가 하는 일(기억, 연결, 패턴 인식, 잊기)을
시스템이 대신하거나 보조한다.
그리고 AI 시대에 이 외부화된 사고 시스템은
AI의 컨텍스트가 된다. 과거의 내가 쓴 노트가
현재의 AI를 더 똑똑하게 만드는 구조다.
단, "이해를 위임하지 마라"는 전제 위에서.
