# What Requests Taught Me About Marriage

- 원문: <https://kennethreitz.org/essays/2026-03-06-what_requests_taught_me_about_marriage>
- 저자: Kenneth Reitz (Python `requests` 라이브러리 제작자)
- 날짜: 2026-03-06

## 요약

Python `requests` 라이브러리를 만든 Kenneth Reitz가 API 설계 철학을
결혼 생활에 적용한 에세이. "for humans" 철학이 소프트웨어에만 국한되지
않고 인간 관계의 본질을 꿰뚫는다는 주장을 6가지 원칙으로 풀어낸다.

## 핵심 주제 분석

### 1. 합리적인 기본값 (Sensible Defaults)

`requests`는 SSL 검증, 리다이렉트 처리, 인코딩 자동 감지 등을 기본값으로
제공한다. 사용자가 명시하지 않아도 대부분의 경우에 올바르게 동작한다.

결혼에서도 마찬가지다. 부부는 암묵적 협의 없이 서로 다른 기본값을 가지고
살다가 충돌한다. 저자는 감정 처리 방식에서 충돌을 겪었다.

- 저자: 혼자 생각 정리 후 답변 (내부 처리)
- 배우자 Sarah: 실시간 대화로 풀기

해결책은 어느 한쪽이 기본값을 바꾸는 게 아니라, **공유된 새 기본값**을
설계하는 것이었다. "1시간 후에 다시 얘기하자"는 타임아웃과 콜백이 포함된
프로토콜이 탄생했다.

> 대부분의 부부 싸움은 실제 문제가 아니라 일치하지 않는 기본값에서 온다.

### 2. 명확한 계약 (Clear Contracts)

`requests`의 API는 명확하다. `response.status_code`, `response.ok`,
`response.json()` — 무엇을 반환하는지 알 수 있다. urllib2처럼 12단계
의식을 치를 필요가 없다.

암묵적 기대는 관계의 urllib2다. "왜 내가 화난 걸 몰랐어?"는 상태 코드 없는
응답을 파싱하라는 요구와 같다.

Sarah는 필요한 게 있으면 직접 말한다고 저자는 서술한다. "오늘 밤 아이
재우는 거 도와줘. 나 지쳤어." — 파라미터가 명확한 요청이다.

저자의 계약: **말한 것을 의미하고, 의미하는 것을 말한다.**
"Explicit is better than implicit"은 파이썬의 선(禪)이면서 결혼 지침이다.

### 3. 우아한 오류 처리 (Graceful Error Handling)

`requests`는 구체적인 예외를 던진다. `ConnectionError`, `Timeout`,
`HTTPError` — 무엇이 잘못됐는지, 어떻게 대응해야 하는지 알 수 있다.

잘 설계된 관계의 오류 표현:

```
"저녁 먹을 때 폰 보면 상처받아. 내려놓아줘."
→ 감정(hurt) + 트리거(폰) + 필요(내려놓기)
```

스택 트레이스 덤프 방식:

```
"항상 그래! 2019년에도 그랬고, 시어머니도 똑같아!"
→ 밤 11시에 파싱 불가능
```

저자는 조현정동장애(Schizoaffective disorder)를 가지고 있어 스트레스 상황에서
오류 처리가 저하된다고 솔직하게 고백한다. Sarah는 시스템이 안정될 때까지
기다릴 줄 안다 — 이것 자체가 사랑의 형태다.

### 4. 하위 호환성 (Backwards Compatibility)

`requests` 2.x 전환 시 기존 API를 최대한 유지했다. 파괴적 변경(breaking
change)은 배신이다. 사람들이 기대하던 인터페이스가 갑자기 사라지는 것이기
때문이다.

결혼에서도 마찬가지다. 배우자는 당신의 특정 패턴과 약속에 의존하며 삶을
설계한다. 필요한 변화가 있다면 **deprecation warning**을 먼저 보내야 한다.

> "요즘 커리어 전환을 생각하고 있어. 아직 결정 안 했는데 알아는 줘야 할 것
> 같아서."

이것이 deprecation warning이다. 준비할 시간을 준다.

저자는 중요한 결정을 이미 내린 후 통보하는 실수를 반복했다고 고백한다.
결정 자체가 맞아도, **프로세스의 배신**이 신뢰를 더 크게 훼손한다.

### 5. 사람이 읽을 수 있는 응답 (Human-Readable Responses)

`response.content`(raw bytes)와 `response.text`(디코딩된 문자열)의 차이.
라이브러리가 번역 작업을 대신해준다.

저자의 내면 경험은 종종 raw bytes로 도착한다 — 이름 붙이지 못한 불안,
미분화된 짜증. 그것을 그대로 던지면 Sarah가 파싱할 수 없다.

감정적 성숙의 작업: **내가 직접 디코더가 되는 것**.

```
raw:     "뭔가 잘못됐는데 모르겠어"
decoded: "내일 미팅이 걱정돼서 예민해진 것 같아.
          너한테 화가 난 건 아니야."
```

Sarah가 저자의 raw output을 잘 읽어내는 능력을 가졌다고 해서,
raw bytes를 계속 보내도 된다는 뜻은 아니다.

> 사랑은 게으른 소통의 면죄부가 아니다. 오히려 더 열심히 할 이유다.

### 6. 커넥션 풀링 (Connection Pooling)

`Session` 객체는 TCP 연결을 재사용한다. 매번 핸드셰이크, SSL 협상을
처음부터 하지 않아도 된다. 공유 컨텍스트 덕분에 통신이 효율적이다.

결혼은 Session 객체다. 이미 핸드셰이크를 마쳤다. 신뢰가 수립됐다.
공유된 헤더 — 가치관, 전제, 내부 언어, 수년간 쌓인 맥락 — 가 있다.

Sarah가 "오늘 화요일 같은 날이야"라고 하면 저자는 무슨 뜻인지 즉시 안다.
저자가 "뇌가 그 상태야"라고 하면 Sarah는 어떤 상태인지 알고 무엇이
필요한지 안다. 이게 커넥션 풀링이다.

실패 모드: 세션을 유지하지 않는 것. 모든 대화를 새 연결로 시작하면
공유 컨텍스트가 없어진다. 오래된 부부가 같은 싸움을 반복하는 이유다
— 지난 번 해결의 맥락을 이어받지 못하기 때문이다.

> 결혼의 작업은 세션 유지다. 지쳐도, 화가 나도, 새로 시작하고 싶어도,
> 컨텍스트를 의식적으로 이어가는 선택이다.

## 인사이트

### "for humans" 철학의 확장

저자의 설계 철학은 HTTP에서 시작했지만 그 핵심은 **시스템이 인간에게
맞춰야 한다**는 것이다. 인간이 시스템 논리에 맞춰야 하는 것이 아니라.

이 원칙은 소프트웨어, 관계, 의사소통 모두에 적용된다.

### 기술적 사고의 감정 번역

기술자들이 감정 언어에 약하다는 고정관념이 있다. 이 에세이는 반대를
보여준다. API 설계 원칙으로 감정적 성숙을 설명하는 것이 오히려 더 정확하고
실용적일 수 있다. 같은 언어로 말할 때 이해가 깊어진다.

### 재귀적 영향력

저자의 아들 Malachi(3살)는 부모의 상호작용 패턴을 보며 기본값을 학습한다.
부부가 유지하는 코드베이스를 아들이 fork해서 평생 수정하며 살아갈 것이다.

> 우리가 심는 기본값이 다음 세대의 초기 코드가 된다.

### 자기 인식의 솔직함

저자는 자신의 정신 질환, 과거의 실수, "나는 좋은 API 설계자지만 좋은
파트너가 되기는 더 어렵다"는 고백을 숨기지 않는다. 이 솔직함 자체가
에세이의 신뢰성을 높인다.

### urllib2 비유

> "Implicit expectations are the urllib2 of relationships."

이 문장 하나가 에세이 전체를 압축한다. 300명만 완전히 이해한다고 했지만,
그 300명에게는 어떤 심리학 용어보다 강렬하게 와닿는 비유다.
