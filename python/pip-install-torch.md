# Wheel Next - `pip install torch`를 한 줄로 끝내는 패키징 혁신

원문: <https://talkpython.fm/episodes/show/544/wheel-next-packaging-peps>

## 요약

Wheel Next는 NVIDIA, Astral, Quansight 등이 주도하는
Python 패키지 배포 시스템 현대화 이니셔티브다.
현재 Python 휠(wheel) 포맷이 CPU 명령어 셋(AVX2, SSE4)이나
GPU 요구사항(CUDA 버전)을 표현할 수 없어
패키지가 2009년 수준의 하드웨어 능력을 타깃으로 한다는 문제를 해결한다.

가장 극단적인 사례가 PyTorch다.
PyTorch 휠은 다양한 GPU 아키텍처용 CUDA 라이브러리를 모두 번들링하기 때문에
거의 1GB에 달한다.
설치하려면 사용자가 특수 인덱스 URL을 직접 지정해야 하는
"퍼즐북" 같은 문서를 따라가야 한다.

NumPy는 런타임 CPU 감지와 동적 디스패치로 이 문제를 회피한다.
여러 CPU 패밀리(Haswell, Skylake)용으로 코드를 여러 번 컴파일하고
하나의 확장 모듈로 병합하는 방식이다.
"팻 바이너리"가 만들어지지만 동작은 한다.
다만 이 접근은 Intel과 ARM 전담 엔지니어가 필요할 정도로 부담이 커
대부분의 패키지에는 비현실적이다.

Wheel Next는 PEP 817과 PEP 825를 통해 "휠 변형(wheel variants)"이라는 일반화된 시스템을 제안한다.
패키지가 임의의 하드웨어 속성을 PEP를 통해 선언하면,
설치 도구의 플러그인이 사용자 하드웨어를 동적으로 감지해
적절한 빌드를 자동 선택한다.
사용자는 그저 `uv pip install torch`라고 입력하면
설치 도구가 GPU를 감지해 200~250MB의 슬림한 휠을 받아 설치를 끝낸다.

NVIDIA, AMD, Intel, Google, Meta, Red Hat, Quansight, Astral이 협력하는
이례적 산업 협력이며,
pip, PyPI, setuptools, uv 전반에 걸쳐 작동하는 프로토타입이 이미 구축되어 있다.
PyTorch와 JAX는 이미 참여하고 있어
표준 확정 후 수 주 내에 변형 지원을 활성화할 수 있을 것으로 전망된다.

## 분석

### 패키지 포맷이 표현할 수 없는 것들

기존 휠 포맷은 OS, Python 버전, ABI, 플랫폼 태그(예: `manylinux2014_x86_64`)만 표현한다.
CPU 명령어 셋, GPU 모델, CUDA 버전 같은 하드웨어 속성은 표현할 수 없다.
플랫폼 태그를 무한히 확장하는 대신,
임의의 하드웨어 속성을 선언할 수 있는 일반화된 변형 시스템을 도입하는 것이
Wheel Next의 핵심 설계 결정이다.

### 플러그인 기반 하드웨어 감지

설치 도구가 모든 하드웨어를 직접 감지하는 것은 비현실적이다.
NVIDIA GPU 감지는 NVIDIA가, AMD GPU 감지는 AMD가 가장 잘 안다.
Wheel Next는 벤더가 자사 하드웨어 감지 플러그인을 제공하고
설치 도구가 이를 호출해 적절한 변형을 선택하는 구조를 채택한다.
산업 협력이 단순한 정치적 협력이 아니라 기술 아키텍처 자체에 내재되어 있는 셈이다.

### NumPy의 런타임 디스패치라는 우회로

런타임 디스패치는 패키징 한계를 코드 수준에서 우회하는 방식이다.
동작하지만 비용이 크다.
패키지 크기가 부풀고, 빌드가 복잡해지고,
새 CPU 아키텍처가 나올 때마다 별도 작업이 필요하다.
Wheel Next는 이 우회로를 패키징 계층의 표준으로 끌어올린다.

## 비평

### 표준화 속도와 생태계 적응의 간극

PEP 채택과 PyPI 적용까지의 행정적 시간은 수개월에서 수년이 걸린다.
프로토타입이 이미 동작해도
모든 사용자가 새 설치 도구를 채택할 때까지의 마이그레이션 기간이 길다.
2026년 이후로 완전한 생태계 적응이 미뤄진 것은 이런 현실의 반영이다.

### 플러그인 신뢰성 문제

설치 도구가 임의의 벤더 플러그인을 실행해
하드웨어를 감지하는 모델은 보안 측면에서 새로운 공격 표면을 만든다.
플러그인 서명, 샌드박싱, 권한 모델 등 부수적 인프라가 필요한데
초기 PEP에서 이 부분이 충분히 다뤄지는지는 확인이 필요하다.

### 한국 AI 생태계의 참여 부재

GeekNews 댓글에서 지적되었듯,
한국 기업 RabbitUp 외에는 한국 AI 회사의 후원이나 참여가 거의 없다.
Python 패키징 표준은 향후 AI 인프라의 기반이 되는데,
한국 기업이 표준 형성 과정에서 배제되는 것은 장기적 경쟁력에 영향을 줄 수 있다.

## 인사이트

### Python 패키징의 "플랫폼 전략화"

Wheel Next는 단순한 기술 개선이 아니라
Python 패키징이 "플랫폼 전략의 영역"으로 진입했음을 보여준다.
NVIDIA, AMD, Intel이 같은 테이블에 앉은 이유는
자사 하드웨어가 Python 생태계에서 정당하게 평가받지 못하면
AI 개발자들이 자사 하드웨어를 선택할 이유가 줄어들기 때문이다.

지난 10년간 NVIDIA가 CUDA로 구축한 해자(moat)는
"PyTorch가 NVIDIA에서 잘 동작한다"는 사실에 크게 의존했다.
이 잘 동작함의 상당 부분은 패키징 우회와 문서 노력으로 만들어진 것이다.
AMD ROCm이나 Intel oneAPI가 같은 수준의 사용자 경험을 제공하려면
패키징 표준 자체가 GPU 다양성을 표현할 수 있어야 한다.

이 맥락에서 Wheel Next에 대한 산업 협력은
표면적으로는 협력이지만 실질적으로는 경쟁의 다른 형태다.
NVIDIA는 자사의 우위를 지속하려면 표준화에 참여해야 하고,
AMD/Intel은 표준화 없이는 영원히 후발 주자에 머물 수밖에 없다.
패키징 포맷이 산업 권력 구조를 결정하는 시대가 된 것이다.

### 1GB 휠의 숨은 비용

PyTorch 휠이 1GB라는 사실은 단순한 불편함을 넘어선다.
CI/CD 파이프라인에서 매 빌드마다 1GB를 다운로드한다면
연간 수만 GB의 트래픽과 수 시간의 빌드 시간이 누적된다.
컨테이너 이미지에 PyTorch를 포함하면 이미지가 거대해져
배포 속도, 스토리지 비용, 콜드 스타트 시간 모두 악화된다.

200~250MB로 줄어든다는 것은
이런 누적 비용이 4분의 1 수준으로 감소한다는 의미다.
대규모 ML 인프라를 운영하는 회사 입장에서
이는 수억 원 단위의 인프라 비용 절감으로 직결된다.
PyPI 자체의 스토리지·대역폭 비용도 비례해서 줄어든다.

더 중요한 것은 진입 장벽 효과다.
1GB 다운로드가 부담스러운 환경(개발도상국 인터넷, 학교 네트워크)에서
PyTorch는 사실상 사용하기 어려운 도구였다.
패키지 크기가 줄어들면 PyTorch 사용자 기반 자체가 확장되며,
이는 PyTorch 생태계의 장기적 우위에 결정적 요인이 된다.

### 하위 호환성의 저주에서 벗어나는 법

Python 휠이 2009년 CPU 명령어 셋을 타깃으로 하는 것은
"가장 오래된 환경에서도 동작해야 한다"는 하위 호환성 원칙의 결과다.
이 원칙은 안전하지만 발전을 가로막는다.
모든 사용자가 17년 전 하드웨어를 보호하기 위해 비용을 지불하는 셈이다.

소프트웨어 산업은 이런 하위 호환성의 저주를 반복적으로 경험해왔다.
IE6를 지원하기 위해 웹 개발이 10년간 정체된 사례,
Windows XP 지원 종료에 대한 저항,
Python 2의 끈질긴 생존 등이 모두 같은 패턴이다.
하위 호환성을 깨기 어렵기 때문에 점진적 우회로가 누적되고,
결국 시스템 전체가 무거워진다.

Wheel Next의 접근은 흥미롭다.
하위 호환성을 깨는 대신 "변형"을 도입해
새로운 능력을 표현할 수 있는 공간을 옆에 만든다.
기본 휠은 여전히 2009년 CPU에서도 동작하지만,
변형 시스템을 통해 현대 하드웨어용 빌드가 함께 제공된다.
이는 깨지 않는 진화의 모범적 패턴이며,
HTTP/2, IPv6, ES6 모듈 같은 다른 표준들의 접근과도 일맥상통한다.

### Astral과 uv가 가져오는 권력 이동

Wheel Next 협력에 Astral이 포함된 것은 의미심장하다.
Astral은 uv를 만든 신생 회사로,
Python 패키징 도구의 사실상 표준이 되어가고 있다.
NVIDIA, Meta, Google이 한 신생 스타트업과 같은 테이블에 앉는 것은
이 회사가 이미 충분한 영향력을 가졌음을 의미한다.

Python 패키징은 오랫동안 PyPA(Python Packaging Authority) 같은
커뮤니티 기반 거버넌스 아래 있었다.
pip의 발전 속도는 보수적이었고,
새로운 PEP의 채택은 신중하게 이뤄졌다.
uv는 이 흐름을 흔들었다.
Rust로 다시 작성하고, 빠른 속도와 적극적인 기능 추가로
사용자가 빠르게 이동하는 기반을 만들었다.

이는 패키징 거버넌스의 권력 이동을 시사한다.
PyPA의 합의 모델 대신
uv가 먼저 구현하고 표준이 따라오는 패턴이 자리잡고 있다.
Wheel Next 프로토타입이 uv에서 먼저 동작한다는 사실은
"표준 → 구현"이 아니라 "구현 → 표준" 순서로
Python 패키징이 진화하고 있음을 보여준다.
이 변화는 효율적이지만,
하나의 회사가 표준 형성에 과도한 영향력을 갖게 되는 위험도 동반한다.
