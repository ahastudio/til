# CPython 3.15의 초저오버헤드 인터프리터 프로파일링

원문: [Python 3.15’s Ultra-Low Overhead Interpreter Profiling Mode | Ken Jin’s Blog](https://fidget-spinner.github.io/posts/ultra-fast-tracing.html)

HN 토론: <https://news.ycombinator.com/item?id=48918140> (156점)

## 요약

Ken Jin이 CPython 3.15 JIT 컴파일러를 위해 개발한 초저오버헤드 인터프리터 프로파일링 기법을 설명한 글이다.
JIT 컴파일러는 실행 중 인스트럭션 스트림을 기록해야 하는데,
기존 프로파일링 방식은 성능 저하가 크다는 문제가 있었다.

전통적인 두 접근 방식은 각각 한계가 있다.
첫째, 별도의 실행용/프로파일링용 인터프리터를 두는 방식은
computed goto 인터프리터에서 약 6% 성능 저하를 일으켰다.
둘째, 조건부 불리언으로 프로파일링을 활성화하는 방식은 코드 팽창은 줄이지만
여전히 실행 속도를 늦춘다.
PyPy의 메타 추적 방식은 900–1000배의 성능 저하를 초래한다.

이 글에서 제안하는 해법은 디스패치 테이블 교체다.
두 개의 디스패치 테이블을 유지한다. 하나는 일반 실행용, 하나는 프로파일링용이다.
프로파일링 인스트럭션은 모두 하나의 기록 동작으로 집결되고,
실제 실행은 일반 인터프리터에 위임한다.
모드 전환은 디스패치 테이블 변수를 교체하는 것만으로 이뤄진다.

```text
ENTER_TRACING() → DISPATCH_TABLE_VAR = TRACING_DISPATCH_TABLE
LEAVE_TRACING() → DISPATCH_TABLE_VAR = DISPATCH_TABLE
```

결과적으로 프로파일링 오버헤드가 최대 4.5배 수준으로 낮아졌다.
기존 방식의 900–1000배와 비교하면 획기적인 개선이다.

## 분석

### JIT 컴파일을 위한 트레이싱의 구조적 딜레마

JIT 컴파일러는 “자주 실행되는 코드 경로”를 찾아 최적화한다.
이를 위해 어떤 인스트럭션이 얼마나 실행됐는지 추적해야 하는데,
이 추적 자체가 실행 속도를 늦추는 역설이 생긴다.

추적 비용이 크면 JIT의 이점을 상쇄한다.
PyPy의 1000배 오버헤드가 왜 문제인지가 여기서 명확해진다.
추적 중에는 일반 Python 실행 속도로도 느린데,
추적 비용까지 더해지면 JIT 없는 CPython보다 느려질 수 있다.

디스패치 테이블 교체 방식은 이 딜레마를 구조적으로 푼다.
프로파일링 로직을 실행 경로에서 분리해 별도 테이블로 옮기고,
전환 비용을 포인터 교체 수준으로 낮춘 것이다.

### CPython의 JIT 방향과 이 기법의 위치

CPython 3.13에서 시험적 JIT가 처음 도입됐고,
3.14, 3.15로 개선이 계속되고 있다.
이 프로파일링 기법은 그 JIT 파이프라인의 핵심 부품이다.

JIT가 최적화할 코드 경로를 찾으려면 실행 추적이 필요하고,
그 추적이 가볍지 않으면 JIT 전체의 가치가 떨어진다.
4.5배 오버헤드는 여전히 작지 않지만, 추적이 항상 켜져 있지 않고
JIT가 필요하다고 판단할 때만 활성화된다는 점에서 실용적이다.

### 인터프리터 프로파일링과 일반 Python 프로파일링의 차이

`cProfile`, `py-spy`, `Pyroscope` 같은 일반 Python 프로파일러와 이 기법은 목적이 다르다.
일반 프로파일러는 개발자가 코드의 병목을 찾을 때 쓴다.
이 기법은 런타임 자체가 JIT 최적화 대상을 식별하기 위해 내부적으로 사용한다.

그러나 이 기술의 응용 가능성은 JIT를 넘어선다.
낮은 오버헤드로 인터프리터 수준의 실행 추적이 가능하다면,
프로덕션에서 항상 켜놓을 수 있는 프로파일링이 가능해진다.
이것은 APM(Application Performance Monitoring) 도구들이 오랫동안 원해온 능력이다.

## 비평

### 4.5배 오버헤드가 “초저”라고 부를 수 있는가

제목의 “ultra-low overhead”는 비교 대상이 있는 주장이다.
PyPy의 900–1000배와 비교하면 4.5배는 극적인 개선이다.
그러나 프로덕션 서비스 관점에서 4.5배 성능 저하는 여전히 허용하기 어렵다.

이 기법이 “항상 켜져 있는 프로파일링”을 목표로 한다면,
4.5배 오버헤드 구간이 얼마나 짧게 발생하는지가 실용성을 결정한다.
글에서 이 점을 명확히 설명하지 않는 것은 아쉬운 부분이다.

### 구현 복잡도가 유지 비용으로 전환된다

저자 스스로 “이 접근의 복잡성이 성능 향상을 정당화하는가”라는 질문을 던진다.
두 개의 디스패치 테이블을 유지하고, 모드 전환 로직을 관리하는 것은
CPython 내부 구현의 복잡도를 높인다.

CPython 기여자들이 이 코드를 이해하고 유지하는 비용이 있다.
언어 런타임의 핵심 부품일수록 단순성이 가치 있는데,
이 트레이드오프가 장기적으로 어떻게 판단될지는 실제 CPython 채택 여부가 보여줄 것이다.

## 인사이트

### 런타임 내장 관측성이 외부 APM을 대체하는 방향으로 나아간다

지금까지 Python 애플리케이션의 성능 관측은 주로 외부 도구에 의존했다.
`py-spy` 같은 샘플링 프로파일러를 주기적으로 실행하거나,
Datadog, New Relic 같은 APM이 에이전트를 삽입하는 방식이다.

이 방식의 한계는 오버헤드와 샘플링 간격이다.
항상 켜놓기에는 비싸고, 주기적 샘플링은 짧은 성능 이상을 놓친다.

런타임 수준의 초저오버헤드 추적이 가능해지면 이 한계가 사라진다.
Python 3.15가 이 방향을 제공한다면, 외부 APM 에이전트 없이도
런타임 자체에서 실행 프로파일을 수집하는 “내장 관측성”이 현실이 된다.

이것은 DevOps 관점에서 의미있는 변화다.
에이전트 설치, 버전 호환성, 에이전트 자체의 오버헤드 관리 같은 운영 부담이 줄어든다.

### CPython JIT의 성숙이 Python 성능 선입견을 바꿀 수 있다

Python이 느리다는 인식은 대부분 사실이다.
GIL, 동적 타입 시스템, 인터프리터 오버헤드가 C, Rust, Go보다 느린 이유다.

그러나 JIT가 성숙하면 이 인식이 부분적으로 바뀔 수 있다.
자주 실행되는 경로가 JIT 최적화되면, Python 코드가 네이티브 수준에 가까운 속도로 실행되는 케이스가 늘어난다.

이것이 왜 중요한가 하면, Python은 이미 AI/ML, 데이터 처리, 백엔드 개발의 주류 언어가 됐기 때문이다.
Python의 성능이 개선되면 “성능이 중요한 부분은 C 확장으로”라는 패턴의 필요성이 줄고,
순수 Python 코드로 더 많은 영역을 커버할 수 있게 된다.
이 방향의 마일스톤으로 CPython 3.15의 JIT 개선은 추적할 가치가 있다.
