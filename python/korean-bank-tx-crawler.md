## 소개

korean-bank-tx-crawler는 국내 개인 은행 거래내역을 파이썬으로 자동화해 가져오는 라이브러리다.
beomi의 simple_bank_korea에서 포크됐으며, KB국민은행과 우리은행을 지원한다.

```python
from simple_bank_korea import get_transactions

txs = get_transactions(
    bank_name='woori',
    bank_num='1002360090945',
    birthday='990429',
    password='1427',
    days=30
)
```

## 사용법

### 설치

```bash
pip install -U korean_bank_tx_crawler
```

### 전제 조건

각 계좌에서 은행 인터넷 뱅킹의 **빠른조회** 서비스를 사전 등록해야 한다.
Chrome 또는 Brave 브라우저가 시스템에 설치되어 있어야 한다(가상 키패드 처리용).

### 반환 값

```python
[
    {
        'date': datetime.datetime(2026, 6, 9, 13, 28, 15),
        'amount': -10000,      # 양수: 입금, 음수: 출금
        'balance': 0,
        'transaction_by': '김철수'
    },
    ...
]
```

각 거래는 날짜(`date`), 금액(`amount`), 잔액(`balance`), 거래처 이름(`transaction_by`)을 포함한다.

### 지원 은행

| 은행          | `bank_name` 값      |
| ------------- | ------------------- |
| KB국민은행    | `'kb'` 또는 `'kookmin'` |
| 우리은행      | `'woori'`           |

## 분석

### 이 라이브러리가 해결하는 문제

국내 은행들은 공식 오픈 API를 개인 계좌에 대해 일반적으로 제공하지 않는다.
거래내역을 프로그래밍 방식으로 가져오려면 인터넷 뱅킹 화면을 자동화해야 한다.
엑셀 내보내기를 수동으로 반복하거나 화면 크롤링을 직접 구현하는 대신, 이 라이브러리는 그 자동화를 추상화한다.

가상 키패드 처리가 기술적 핵심이다.
국내 은행들은 보안 목적으로 비밀번호 입력에 이미지 기반 가상 키패드를 사용한다.
이것은 일반 텍스트 입력 자동화를 막기 위한 것으로, 라이브러리는 이미지 매칭(Pillow)으로 이를 우회한다.
`requirements`에 `pillow`, `selenium`, `webdriver-manager`가 포함된 이유가 여기에 있다.

### 개인 가계부 자동화의 기술적 접근

월별 가계부를 수동으로 입력하는 대신 이 라이브러리로 거래내역을 자동 수집하면, 파이썬으로 지출 분류, 통계, 시각화를 구현할 수 있다.
`days` 파라미터로 최근 N일의 거래를 가져오므로 정기적 실행으로 증분 수집도 가능하다.

Pandas, Matplotlib 같은 데이터 분석 라이브러리와 조합하면 개인 재무 분석 파이프라인을 빠르게 구성할 수 있다.
국내 개인 뱅킹 API의 공백을 채우는 실용적 해법이다.

### 지원 은행 범위 제한이 실용성의 핵심 제약이다

KB국민은행과 우리은행만 지원한다는 것은 신한, 하나, 농협, 카카오뱅크, 토스뱅크 같은 다른 주요 은행을 사용하는 사람에게는 사용 불가 라이브러리라는 의미다.
은행별로 인터넷 뱅킹 구조가 다르고, 가상 키패드 구현 방식도 다르기 때문에 각 은행을 추가하는 것은 상당한 추가 작업이다.

프로젝트가 포크 기반이고 스타 3개라는 것은 현재 활발히 유지보수되는 프로젝트가 아닐 가능성을 시사한다.
새 은행 추가나 인터넷 뱅킹 업데이트 대응이 즉시 이뤄지지 않을 수 있다는 점을 사용 전 고려해야 한다.

## 인사이트

### 공식 API의 공백이 만드는 크롤링 생태계

국내 금융 기관들이 개인 계좌 접근 API를 공개하지 않는 구조적 이유는 보안과 책임 귀속의 문제다.
API를 통한 제3자 접근이 발생했을 때 보안 사고의 책임이 은행에 귀속될 수 있고, 자동화된 접근이 사기나 자금 세탁에 활용될 위험이 있다.

이 공백이 이런 라이브러리를 만들어내지만, 이것은 공백에 대한 임시 해법이지 지속 가능한 기반이 아니다.
인터넷 뱅킹 UI가 변경될 때마다 라이브러리가 깨지고, 은행이 크롤링 방지 조치를 강화하면 라이브러리가 동작을 멈춘다.
금융결제원의 오픈뱅킹 API가 확대되면 이런 라이브러리의 필요성이 줄어든다 — 공식 API가 충족하지 못하는 영역이 좁아질수록 크롤링의 대안 가치도 줄어든다.

### 가계부 자동화 요구가 보여주는 개인 재무 데이터 접근의 문제

많은 사람이 자신의 지출 패턴을 이해하고 관리하고 싶어 하지만, 이 정보에 프로그래밍 방식으로 접근하는 것이 어렵다.
Toss나 카카오페이 같은 앱이 이 니즈를 UI 차원에서 충족하지만, 데이터를 원하는 방식으로 분석하거나 다른 시스템과 통합하려면 API가 필요하다.

이것은 개인 데이터 주권의 문제이기도 하다.
은행에 있는 내 거래 데이터에 내가 원하는 방식으로 접근하는 것이 왜 어려운가?
EU의 PSD2(Payment Services Directive 2)는 은행이 제3자 서비스에 고객 계좌 데이터 접근 API를 제공하도록 의무화했다.
한국에서 오픈뱅킹이 확장되는 방향이 결국 이 라이브러리 같은 크롤링 해법을 불필요하게 만드는 방향이다.
