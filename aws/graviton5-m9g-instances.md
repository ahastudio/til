# AWS Graviton5: EC2 M9g/M9gd 인스턴스 출시

원문: [AWS Graviton5 프로세서 기반 Amazon EC2 M9g 및 M9gd 인스턴스 정식 출시 | Amazon Web Services 한국 블로그](https://aws.amazon.com/ko/blogs/korea/now-available-amazon-ec2-m9g-and-m9gd-instances-powered-by-new-aws-graviton5-processors/)

HN 토론: <https://news.ycombinator.com/item?id=48481086> (8점, 1개 댓글) · 관련 Graviton5 발표 토론 <https://news.ycombinator.com/item?id=46191993> (31점, 23개 댓글)

## 요약

AWS가 Graviton5 프로세서 기반의 Amazon EC2 M9g 및 M9gd 인스턴스를 출시했다.
Graviton 플랫폼 8년 연속 투자의 다섯 번째 세대로, Graviton4 대비 컴퓨팅 성능이 최대 25% 향상되었다.

실제 사용 사례에서의 성능 향상이 두드러진다.
ClickHouse는 코드 변경 없이 M8g 대비 36% 향상, Honeycomb은 관측 가능성(observability) 워크로드에서 코어 처리량 36% 향상, HubSpot은 MySQL 쿼리 시간을 최대 60% 단축했다.

기술 사양으로는 코어 192개, L3 캐시 5배 확장, 코어 간 지연 최대 33% 감소, DDR5-8800 메모리 지원이 특징이다.
이 DDR5-8800은 “클라우드의 모든 프로세서 인스턴스 중 가장 빠른 메모리”라고 AWS는 설명한다.
네트워크 대역폭은 평균 15%, EBS 대역폭은 평균 20% 향상되었다.

보안 측면에서는 Nitro 격리 엔진이 수학적으로 검증된(formally verified) 보안 보장을 제공한다.
인스턴스 대역폭 구성(IBC) 기능으로 EBS와 VPC 네트워킹 자원을 유연하게 배분할 수 있다.

현재 4개 AWS 리전에서 사용 가능하며, Savings Plans, 온디맨드, 스팟, 전용 인스턴스 방식으로 구매할 수 있다.

## 분석

### Graviton 세대 누적이 보여주는 AWS의 칩 전략

Graviton5의 출시는 단일 제품 발표가 아니라 8년에 걸친 전략적 투자의 결과다.
AWS가 자체 칩을 설계하는 이유는 단순히 성능 향상 때문이 아니다.
x86 기반 Intel과 AMD 칩의 가격 결정권은 칩 제조사에 있지만, ARM 기반 자체 설계 칩은 AWS가 비용 구조를 직접 제어할 수 있다.

Graviton 고객에게 x86 대비 가격 성능비 우위를 제공하는 것은, 고객이 AWS 생태계에 더 깊이 의존하게 만드는 전략이기도 하다.
Graviton에 최적화된 워크로드는 다른 클라우드로 이전할 때 재최적화 비용이 발생한다.
성능 향상이 곧 플랫폼 고착화(lock-in)의 강화다.

HN에서도 이 발표문의 성격을 두고 비슷한 지적이 나왔다.
spwa4는 이 글이 고객을 설득하는 문서라기보다 새 칩 투자를 정당화하기 위해 AWS 경영진을 겨냥한 문서처럼 읽힌다고 평했다.[^spwa4]
이에 locknitpicker는 “AWS 경영진이야말로 고객”이라고 받아쳤다 — 컴퓨팅 밀도 향상과 인프라 비용 절감은 데이터센터의 판매 포인트이며,
서버리스나 Lambda만 쓰는 외부 고객은 CPU 아키텍처가 아니라 실행 비용에만 관심을 두기 때문이다.[^locknitpicker]
이 교환은 Graviton 전략의 진짜 수혜자가 누구인지를 날카롭게 드러낸다.
자체 칩의 1차 이득은 AWS의 마진이고, 고객의 가격-성능비 개선은 그 마진을 일부 환원한 결과다.

### 코드 변경 없이 36% 향상이라는 클레임의 의미

ClickHouse와 Honeycomb의 성능 향상이 코드 변경 없이 달성되었다는 점은 중요한 의미를 가진다.
새로운 하드웨어로 업그레이드하면서 소프트웨어 재최적화 비용이 0이라면, 인스턴스 교체의 ROI는 순수하게 가격-성능비 개선이 된다.

그러나 이 수치는 특정 워크로드 — 데이터베이스 쿼리, 관측 가능성 데이터 처리 — 에서의 결과다.
메모리 집약적이지 않거나 단일 코어 성능에 의존하는 워크로드에서는 이 수치가 적용되지 않을 수 있다.
HubSpot MySQL 쿼리 60% 단축은 특히 MySQL의 특성과 HubSpot의 쿼리 패턴에 의존하는 결과다.

### DDR5-8800과 L3 캐시 확장이 시사하는 병목 지점

Graviton5의 주요 성능 향상은 메모리 대역폭(DDR5-8800)과 캐시 크기(L3 5배)에 집중되어 있다.
이것은 현재 클라우드 워크로드의 병목이 연산 속도보다 메모리 접근 속도임을 AWS가 판단했다는 의미다.

ML 추론, 데이터베이스, 분석 워크로드는 모두 대용량 데이터를 메모리에서 빠르게 읽는 것이 핵심이다.
코어 수를 늘리는 것보다 캐시와 메모리 대역폭을 늘리는 것이 이 워크로드들에서 더 효과적이라는 설계 판단이 반영되어 있다.

## 비평

### 가용 리전 4개는 전 세계 배포에서 병목이 된다

Graviton5 인스턴스가 현재 4개 리전에서만 사용 가능하다는 것은 글로벌 배포 시 문제가 된다.
다수 리전에 배포된 워크로드를 Graviton5로 마이그레이션하려면 일부 리전은 여전히 이전 세대를 사용해야 한다.
이 불균형은 운영 복잡성 — 리전별 인스턴스 타입 관리, 성능 차이 모니터링 — 을 추가한다.

AWS가 신규 하드웨어를 단계적으로 출시하는 것은 공급망 현실을 반영하지만,
글로벌 규모의 기업에게는 리전 가용성이 채택의 실제 장벽이 된다.

다만 이 제약은 시간이 해결하는 종류이기도 하다.
HN에서 nodesocket이 t 계열도 t5g로 갱신되느냐고 묻자, jng는 AWS가 보통 대부분의 인스턴스 타입을 새 Graviton 세대로 업그레이드하며 다만 전체 롤아웃에 시간이 걸릴 뿐이라고 답했다.[^jng]
즉 4개 리전과 일부 인스턴스 패밀리라는 현재의 한계는 영구적 설계가 아니라 출시 초기의 일시적 상태에 가깝다.
문제는 그 “시간”이 글로벌 배포를 계획하는 팀의 의사결정 주기와 맞지 않을 때 비로소 실제 장벽이 된다는 점이다.

### “수학적으로 검증된 보안”이라는 표현의 범위

Nitro 격리 엔진이 “수학적으로 검증된” 보안을 제공한다는 설명은 인상적이지만,
이것이 어떤 위협 모델에 대한 보장인지가 중요하다.

수학적 검증(formal verification)은 특정한 속성 — 예컨대 VM 간 메모리 격리 — 에 대해서는 강력한 보장을 제공하지만,
전체 보안 스택의 모든 구성요소가 검증되지는 않는다.
“수학적으로 검증된”이라는 표현이 어디까지의 보장을 의미하는지 명확히 이해하고 사용해야 한다.

### 헤드라인 성능 수치는 실제 가격-성능비로 자동 환산되지 않는다

“최대 25% 향상”이나 “36% 향상” 같은 수치는 인상적이지만, 고객이 실제로 얻는 것은 성능 자체가 아니라 가격-성능비다.
HN에서 eugmai86은 Vantage의 현재 데이터 기준으로 m9g.large가 m8g.large보다 온디맨드 가격이 약 9% 비싸다고 지적했다.[^eugmai86]
25% 성능 향상이 실제 웹·데이터베이스 워크로드에서 그대로 나온다면 이 가격 차이는 합리적인 거래지만,
그 전제가 충족되지 않으면 그저 더 비싼 인스턴스로 갈아탄 것에 불과할 수 있다.
같은 댓글은 더 근본적인 함정을 짚는다 — 소규모 SaaS 앱은 원시 CPU 성능에 도달하기 한참 전에 DB·스토리지·네트워크에서 먼저 병목에 걸린다.
이 경우 25% 빠른 코어는 청구서만 늘릴 뿐 체감 성능을 거의 바꾸지 못한다.

발표문이 강조하는 수치들이 얼마나 구체적인 기준 위에 있는지도 따져볼 만하다.
hulitu는 “192 코어”가 AMD Epyc과 다를 바 없다고 보고, “5배 큰 캐시”라는 표현에 대해 무엇보다 5배 큰 것이냐고 물었다.[^hulitu]
crest가 “이전 세대인 Graviton4의 같은 캐시 레벨보다 크다는 뜻”이라고 답하면서 의미가 분명해졌지만,[^crest_cache]
이 짧은 교환은 마케팅 수치가 기준점을 생략한 채 배수만 제시할 때 독자가 얼마나 쉽게 오해하는지를 보여준다.

## 인사이트

### ARM 아키텍처의 클라우드 지배가 가속화된다

Intel x86이 수십 년간 지배했던 서버 시장에서 ARM 아키텍처가 점유율을 늘려가는 추세는,
Graviton 시리즈의 성공과 함께 더 빠르게 진행되고 있다.
AWS Graviton, Apple Silicon, Qualcomm Oryon의 성공이 ARM의 데이터센터 침투를 실증하고 있다.

이 전환은 소프트웨어 생태계에도 영향을 미친다.
ARM 네이티브 컴파일이 x86 에뮬레이션보다 성능이 훨씬 좋기 때문에, ARM을 지원하지 않는 소프트웨어는 점점 경쟁력을 잃게 된다.
Graviton5로의 마이그레이션을 고려하는 팀이라면 의존성 스택의 ARM 지원 현황을 먼저 파악해야 한다.

이 전환이 단순한 점유율 통계가 아니라 개발자 경험의 변화임을 보여주는 현장 증언도 있다.
HN에서 quesomaster9000은 Graviton과 Nitro 4 조합이 상당히 쾌적하다고 전했다.
Rust의 aarch64 musl 정적 타깃과 rust-lld로 단일 ELF 바이너리를 빌드하면 안드로이드 기기에 `adb push`로 올려 실행하든 AWS에서 돌리든 똑같이 동작하고,
Nitro v3+의 TPM 지원 덕에 Amazon CA로 VM 상태를 원격 증명(attestation)할 수도 있다는 것이다.[^quesomaster9000]
binfmt와 qemu-user 덕에 자신이 어떤 아키텍처에서 빌드·실행·테스트하는지조차 잊을 때가 많다는 그의 말은,
ARM이 더 이상 “이식 비용을 감수해야 하는 대안”이 아니라 x86과 구별되지 않는 기본값으로 성숙했음을 시사한다.
crest가 정리했듯 Graviton5는 ARM Neoverse V3 코어 기반으로 최신 고성능 모바일 Cortex X4 코어와 매우 유사하며,[^crest_neoverse]
데이터센터 칩과 모바일 칩이 같은 코어 계보를 공유한다는 사실 자체가 이 수렴을 압축적으로 보여준다.

### IBC 기능이 예고하는 네트워크-스토리지 통합 관리의 미래

인스턴스 대역폭 구성(IBC)은 EBS와 VPC 네트워킹 자원을 유연하게 배분한다는 점에서, 인프라 자원 관리의 새로운 방향을 보여준다.
전통적으로 네트워크 대역폭과 스토리지 대역폭은 별도의 고정 자원이었다.

워크로드의 특성에 따라 — 데이터 집약적 작업은 EBS에, 마이크로서비스 통신은 VPC에 더 많이 배분 — 유연하게 조정할 수 있다면, 인프라 비용 최적화의 새로운 레버가 생긴다.
이 방향이 계속되면 미래의 클라우드 인스턴스는 컴퓨팅, 메모리, 스토리지, 네트워크 자원을 워크로드 패턴에 따라 동적으로 재배분하는 방향으로 발전할 수 있다.
