# 서버리스 Rust로 10분 안에 PDF 100만 개 생성하기

원문: [Generating 1 Million PDFs in 10 Minutes with Serverless Rust](https://www.ersteiger.com/posts/rendering-one-million-pdfs/)

GN 토론: <https://news.hada.io/topic?id=20705> (24P, 댓글 7개)

## 요약

저자 Erik Steiger는 AWS 서버리스 아키텍처를 이용해 100만 개의
PDF 문서를 10분 안에, 전통적인 EC2 기반 접근보다 약 97% 낮은
비용으로 생성하는 파이프라인을 구축한 과정을 소개한다.
글은 금융기관이 거래 확인서, 세금 명세서, 분기 보고서 같은
개인화 문서를 규제가 정한 시간 안에 대량으로 생성해야 하는
문제 상황에서 출발한다.
저자는 실제로 어느 금융 회사가 6개월과 다섯 명의 정규 엔지니어를
투입하고도 레거시 온프레미스 시스템에서 벗어나지 못했다는
사례를 언급하며, 목표를 “초당 1,667개의 PDF, 즉 PDF 한 개당
0.6ms”로 구체화한다.
일반적인 PDF 생성기가 개당 약 1초가 걸린다는 점을 고려하면
같은 작업에 11.5 컴퓨트 일(compute day)이 필요하다는 것이
문제의 규모를 보여준다.

아키텍처는 API Gateway, SQS, Lambda, S3로 구성된다.
API Gateway가 요청을 받으면 첫 번째 Lambda 함수가 배치 렌더링
요청을 SQS 큐에 넣고 HTTP 202(Accepted)와 작업 ID를 즉시
반환해 비동기 처리를 가능하게 한다.
두 번째 Lambda 함수는 SQS에서 작업을 꺼내 S3에 저장된 Typst
템플릿을 가져오고, Papermake 라이브러리로 PDF를 렌더링한 뒤
S3에 업로드한다.
두 함수 모두 Rust로 작성되어 네이티브 ARM64 바이너리로
컴파일되며, 이는 JVM이나 Node.js 환경에서 흔히 발생하는
런타임 의존성과 콜드 스타트 지연을 없앤다.

PDF 생성 엔진 선택에서 저자는 Puppeteer(1~2초, 헤드리스
브라우저 오버헤드), Crystal Reports(750~900ms, 레거시 Windows
의존성과 콘텐츠 오버플로 문제), LaTeX(500~800ms, 큰 컴파일
패키지와 높은 메모리 사용량)을 검토한 뒤 Typst를 선택했다.
최적화된 상태에서 Typst는 약 35ms의 지연시간을 보였으며,
저자는 이를 “빠르고 유용한 오류 메시지를 제공하는 매우
유망한 새 조판 시스템”이라고 평가한다.
템플릿은 회사 브랜딩과 고객 정보, 동적 거래 내역 테이블,
수수료·세금·합계가 포함된 계산 요약, 법적 고지와 결제 정보로
구성되며 `#data.customer.name`처럼 Papermake를 통한 데이터
보간(interpolation)을 사용한다.
렌더링 전에는 필수 필드가 모두 채워졌는지 검증한다.

성능 최적화는 세 축으로 이뤄진다.
Lambda 동시성은 최대 1,000개 인스턴스까지 오토스케일링하도록
설정하고, 기본 가용성을 위한 최소 프로비저닝 인스턴스 5개를
유지하며, 목표 사용률 75%와 30초 스케일아웃 쿨다운을
적용했다.
캐싱은 세 계층으로 나뉜다.
호출 간 메모리에 Typst 템플릿을 유지하는 템플릿 캐시,
렌더링 사이에 부분 컴파일 결과를 재사용하는 월드(world)
캐시, 핫 스타트 사이에 AWS SDK의 S3/SQS 연결을 유지하는
클라이언트 캐시다.
스레드 안전한 싱글턴 리소스 관리에는 Rust의 `OnceCell`과
`RwLock`을 사용했다.
배칭 전략으로는 캐시 효율을 극대화하기 위해 템플릿별로
렌더링 요청을 묶고, SQS 배치 처리로 API 호출 수를 줄였다.

동시 호출 10개로 진행한 예비 테스트에서는 1,000개의 요청을
11초 만에 처리해 초당 약 91개의 처리량과 개당 35ms의
지연시간(핫 스타트, 캐시 적용 상태)을 기록했다.
동시 인스턴스 60개로 확장한 예상 시나리오에서는 초당 1,667개
처리량, 10분 이내 완료, 개당 0.6ms라는 목표를 달성한 것으로
계산된다.
100만 개 PDF 생성 비용은 컴퓨트 8,750 GB초에 대해 0.15달러,
100만 번의 호출에 대해 0.20달러로 총 0.35달러이며, 이는
c6i.32xlarge EC2 인스턴스 13대를 10분간 사용하는 전통적
접근의 13.45유로와 비교해 약 97% 절감된 수치다.
저자는 프로덕션 배포를 위해서는 템플릿 ID별 SQS 라우팅,
실시간 모니터링과 재시도 로직, 다중 리전 배포, 문서 서명과
암호화, 포괄적 오류 처리와 데드레터 큐 같은 추가 요소가
필요하다는 점도 인정한다.

## 분석

### 이 글의 핵심 주장은 “언어 선택”이 아니라 “계층별 지연시간 제거”다

글의 표면적인 결론은 “Rust와 Typst가 빠르다”이지만, 실제
논증 구조를 뜯어보면 핵심은 병목이 될 수 있는 모든 계층에서
지연시간의 근원을 하나씩 제거해 나간 과정이다.
콜드 스타트는 네이티브 컴파일로, 템플릿 재로딩은 메모리
캐시로, 컴파일 반복 비용은 월드 캐시로, 연결 재수립 비용은
클라이언트 캐시로 각각 없앤다.
이 네 가지 해법은 서로 다른 계층(런타임, 애플리케이션,
조판 엔진, 네트워크)에 위치하며, 어느 하나만으로는 목표
지연시간에 도달할 수 없다.

이 구조가 중요한 이유는, 만약 저자가 Rust 대신 Node.js를
쓰면서 나머지 세 가지 캐싱 전략을 그대로 적용했어도 상당한
성능 개선을 얻었을 것이기 때문이다.
반대로 Rust를 쓰고도 캐싱을 하나도 적용하지 않았다면 목표에
도달하지 못했을 것이다.
즉 “서버리스 Rust”라는 제목은 가장 눈에 띄는 변수 하나를
전면에 내세운 것일 뿐, 실제 성과는 계층별 최적화의 합이라는
점이 글의 실제 논증 구조에서 드러난다.

### “100만 개, 10분”이라는 숫자는 특정 정의 위에서만 성립한다

글의 핵심 성과 지표인 “10분 안에 100만 개”는 예비 테스트에서
직접 측정된 값이 아니라, 동시 호출 10개로 측정한 초당 91개
처리량을 동시 인스턴스 60개로 선형 외삽(extrapolate)한
계산값이다.
저자 스스로도 “동시성 60에서의 예상 성능”이라고 표현하며
이것이 실측이 아니라 추정이라는 점을 명시한다.
이는 감추어진 결함이 아니라 저자가 스스로 밝힌 한계이지만,
글의 제목과 요약이 이 추정치를 마치 검증된 결과처럼
전면에 내세운다는 점에서 표현의 무게와 실제 근거 사이에
간극이 있다.

Lambda의 동시성 확장은 선형적이라는 보장이 없다.
AWS 계정 동시성 쿼터, 콜드 스타트가 재발하는 스케일아웃
구간, SQS 처리량 한계, S3 요청 속도 제한 등 여러 요인이
동시 인스턴스 수가 늘어날수록 비선형적으로 작용할 수 있다.
저자가 실제로 “테스트가 AWS 계정 동시성 쿼터 증가를 기다리는
중이라 제한되었다”고 밝힌 대목은, 60개 동시성에서의 실측이
아직 이뤄지지 않았다는 뜻이며, 외삽의 신뢰도를 낮추는
직접적인 증거이기도 하다.
GeekNews 댓글에서 savvykang은 단위를 일관되게 맞춰 이 외삽을
다시 계산했는데, 목표치인 초당 1,666.7개에 비해 예비 테스트의
동시성 10개 처리량은 초당 29.4개에 그쳐 목표를 달성하려면
동시 인스턴스 570개가 필요하다는 결과를 내놓았다[^savvykang].
이는 글이 제시한 “동시성 60개면 충분하다”는 계산과 크게
어긋나는 수치이며, 외삽에 사용된 처리량 산출 방식 자체를
재검토해야 할 필요성을 보여준다.

### Typst 선택은 “문서 생성”이라는 특정 사용 사례의 요구에 맞춰진 결정이다

저자가 Typst를 선택한 이유는 조판 품질이나 문법의 우아함이
아니라 Lambda라는 실행 환경의 제약, 즉 짧은 실행 시간과
제한된 메모리에 맞는 가벼움과 빠른 컴파일이었다.
LaTeX이 배제된 이유도 미학적 열세가 아니라 “큰 컴파일 패키지와
높은 메모리 사용량”이라는 서버리스 환경 특유의 제약
때문이다.
이는 Typst의 강점이 맥락 독립적인 절대적 우수성이 아니라,
서버리스처럼 자원이 제한되고 반복 호출이 잦은 환경에서
특히 부각되는 상대적 우수성이라는 점을 시사한다.

이 구분은 Typst를 다른 맥락, 예컨대 데스크톱에서 대규모
학술 문서 하나를 편집하는 상황에 적용할 때는 그대로
성립하지 않을 수 있다.
글이 “PDF 생성 엔진 선택”이라는 프레임 안에서만 Typst의
우수성을 논증하고 있다는 점을 분명히 인식하는 것이,
이 사례를 다른 맥락에 일반화하기 전에 필요한 전제다.

GeekNews 댓글에서 crypto는 이 선택을 실무에서 고려할 만한
대안들과 견주어 뒷받침한다[^crypto].
LaTeX은 Docker 이미지로 실행하면 Typst 수준의 속도를 얻기
어렵고, Google Docs는 API를 통한 자동화 편집이 의외로
자유롭지 않다는 점에서, 서버리스 환경의 PDF 생성이라는
좁은 요구를 만족하는 옵션이 실제로 많지 않았다는 정황을
보탠다.

## 비평

### 비용 비교는 서로 다른 신뢰 수준의 숫자를 나란히 놓는다

글이 제시하는 “97% 절감”이라는 결론은 극적이지만, 비교
대상이 되는 두 숫자의 성격이 다르다.
서버리스 파이프라인의 0.35달러는 AWS의 공개된 Lambda
과금 단가에 GB초와 호출 수를 곱한 정적 계산값으로, 실측이
아니라 이론적 산식이다.
반면 전통적 접근의 13.45유로는 c6i.32xlarge 인스턴스 13대를
10분간 가동한다는 가정 위에서 나온 숫자인데, 이 가정 자체가
검증되지 않았다.
왜 13대인지, 왜 c6i.32xlarge인지에 대한 근거가 글에 제시되지
않으며, 실제 레거시 시스템이 그 정도 인스턴스로 같은
작업을 완료할 수 있는지도 확인되지 않는다.

더 근본적인 문제는 비교 대상에서 누락된 비용이다.
전통적 EC2 접근에는 일반적으로 상시 가동되는 유휴 용량,
로드밸런서, 모니터링 인프라, 운영 인력 비용이 포함되지만,
글의 계산은 순수 컴퓨트 비용만 비교한다.
반대로 서버리스 쪽에도 API Gateway 요청 비용, SQS 메시지
비용, S3 저장·요청 비용, Terraform과 CI/CD 파이프라인
유지보수 비용이 빠져 있다.
두 숫자 모두 “가장 유리하게 보이는 항목만 골라 계산한
부분 비용”일 가능성이 있으며, 97%라는 절감률은 이 누락이
그대로 반영된 결과다.

### “6개월, 다섯 명의 엔지니어”라는 서사는 대조군 역할을 하기에는 근거가 부족하다

글은 도입부에서 실제 금융 회사가 6개월과 다섯 명의 정규
엔지니어를 투입하고도 마이그레이션에 실패했다는 사례를
언급하며, 이를 통해 자신의 해법이 상대적으로 얼마나 효율적인지
암시한다.
그러나 이 사례는 회사명, 실패의 구체적 원인, 시도했던
접근법이 전혀 특정되지 않은 채 제시된다.
그 팀이 실패한 이유가 기술 선택의 문제였는지, 조직 내부의
승인 절차나 레거시 시스템과의 통합 문제였는지 알 수 없다면,
이 사례는 저자의 해법이 우수하다는 근거가 아니라 단지
극적인 대비 효과를 위한 수사적 장치에 가깝다.

이 점이 중요한 이유는, 실제 금융기관의 마이그레이션 실패는
대개 기술적 난이도보다 규제 승인, 감사 요구사항, 기존
시스템과의 데이터 정합성 검증처럼 훨씬 느리고 기술 외적인
과정에서 발생하기 때문이다.
저자가 제시하는 해법은 순수한 렌더링 파이프라인의 기술적
증명(proof of concept)에 그치며, 그 회사가 겪었을 조직적
장벽에는 답하지 않는다.
따라서 “우리는 6개월과 다섯 명이 걸린 문제를 해결했다”는
암묵적 주장은 검증되지 않은 비교다.

### 프로덕션에 필요한 요소들을 나열하는 것과 그 요소들이 비용·성능 수치에 미칠 영향을 계산하는 것은 다르다

저자는 글의 말미에서 프로덕션 배포에 필요한 추가 요소로
템플릿 ID별 SQS 라우팅, 실시간 모니터링과 재시도 로직,
다중 리전 배포, 문서 서명과 암호화, 데드레터 큐를
나열한다.
이는 정직한 태도이지만, 이 요소들이 앞서 제시한 0.35달러와
10분이라는 핵심 수치에 어떤 영향을 미치는지는 전혀
계산하지 않은 채 목록으로만 남겨둔다.

문서 서명과 암호화는 렌더링 파이프라인에 추가 연산 단계를
더하고, 다중 리전 배포는 데이터 전송과 인프라 복제 비용을
발생시키며, 재시도 로직과 데드레터 큐는 최소한 일부 요청이
두 번 이상 처리됨을 의미해 실질 처리량을 낮춘다.
즉 “프로덕션에는 이런 것들이 더 필요하다”는 인정과 “그래도
핵심 수치는 유효하다”는 결론 사이에는 실제로 검증되지 않은
큰 간극이 있으며, 글은 이 간극을 명시적으로 인정하지 않은 채
넘어간다.

## 인사이트

### 서버리스 성능 최적화 사례는 벤더 종속을 기술적 미덕으로 재포장하는 경향이 있다

이 글의 아키텍처는 Lambda, SQS, S3, API Gateway라는 AWS
고유 서비스에 강하게 결합되어 있으며, 성능과 비용 우위는
바로 이 결합에서 나온다.
그런데 그 결합은 동시에 벤더 종속(vendor lock-in)이기도
하다.
같은 파이프라인을 다른 클라우드로 옮기려면 SQS를 GCP Pub/Sub나
Azure Service Bus로, Lambda를 각 벤더의 서버리스 함수
서비스로 다시 구현해야 하며, 캐싱 전략 중 상당수(특히 웜
컨테이너 재사용에 의존하는 클라이언트 캐시)는 벤더별 콜드
스타트 동작 차이에 따라 성능이 달라질 수 있다.

이 문제는 글에서 전혀 논의되지 않지만, 금융기관처럼 규제
준수와 장기 운영 안정성이 중요한 조직에게는 성능 수치
못지않게 중요한 고려사항이다.
서버리스 아키텍처를 다루는 기술 글들이 대체로 성능과 비용
지표는 상세히 제시하면서 이식성 상실이라는 대가는 언급하지
않는 것은 이 사례에 국한된 현상이 아니라, 서버리스 담론
전반에서 반복되는 패턴이다.
결국 “97% 비용 절감”이라는 수치는 그 절감을 특정 벤더에게
장기간 종속되는 대가로 얻은 것이라는 사실과 함께 읽혀야
한다.

### 콜드 스타트 제거 기법은 규제 산업의 감사 가능성(auditability) 요구와 충돌할 수 있다

글이 강조하는 핵심 최적화 중 하나는 컨테이너를 웜 상태로
유지해 재사용하는 캐싱 전략이다.
그런데 금융 문서 생성처럼 규제 대상 산업에서는 각 문서
생성 과정이 감사 가능해야 하는 경우가 많다.
같은 실행 환경(월드 캐시, 템플릿 캐시)이 여러 요청에
걸쳐 재사용된다는 것은, 이전 요청의 상태가 완전히
초기화되지 않은 채 다음 요청에 영향을 줄 가능성이
이론적으로 남아 있다는 뜻이기도 하다.

이는 소프트웨어 성능 최적화의 오래된 긴장 관계를 재현한다.
데이터베이스의 커넥션 풀링이 성능을 위해 상태를 재사용하지만
동시에 격리(isolation) 보장을 어렵게 만드는 것과 유사한
구조다.
저자는 성능 수치에 집중한 나머지 이 캐싱 전략이 규제
감사관이 요구할 수 있는 “요청 간 완전한 격리” 원칙과 어떻게
공존하는지는 다루지 않는다.
프로덕션 도입을 고려하는 팀이라면 캐시 재사용과 감사
요구사항 사이의 이 긴장을 직접 검증해야 하며, 이는 글이
제시한 목록에 없는 항목이다.

### 극단적인 처리량 목표는 종종 실제로 필요한 요구사항이 아니라 마케팅 가능한 벤치마크로 설정된다

“10분 안에 100만 개”라는 목표는 인상적인 벤치마크지만,
실제 금융기관이 왜 하필 10분이라는 시간 제약을 갖는지에
대한 규제적 근거는 글에 제시되지 않는다.
분기 보고서나 세금 명세서는 대개 몇 시간에서 며칠 단위의
처리 기한을 가지며, 10분이라는 숫자는 실제 업무 요구사항에서
역산된 것이 아니라 “인상적인 벤치마크를 만들기 위해 선택된
숫자”일 가능성이 있다.

이는 성능 최적화 기술 글에서 흔히 나타나는 패턴이다.
실제 필요보다 훨씬 공격적인 목표를 설정하고 그것을
달성하는 과정을 보여줌으로써, 정작 그보다 훨씬 완화된
요구사항(예: 1시간 안에 100만 개)에서는 캐싱이나 동시성
설정을 그만큼 정교하게 다듬을 필요가 없어진다는 사실이
가려진다.
결과적으로 독자는 “100만 개, 10분”이라는 극단적 조합을
자신의 실제 문제에 그대로 적용해야 한다고 오인하기 쉽지만,
실제로는 훨씬 느슨한 시간 제약 안에서 더 단순한 아키텍처로도
충분할 가능성이 크다는 점을 이 글은 언급하지 않는다.
GeekNews 댓글에서 fortune도 같은 지점을 짚으며, 금융 업계가
왜 이런 대량 문서를 몇 분 안에 생성해야 하는지 그 배경이
궁금하다는 반응을 남겼다[^fortune].
글은 BaFin 같은 감독기관의 벌금 위험을 근거로 들지만, 구체적으로
어떤 규제 조항이 몇 분 단위의 처리 시간을 요구하는지는 밝히지
않아 이 의문은 해소되지 않은 채 남는다.
