# LLVM IR 읽기 — 성능 엔지니어를 위한 입문

원문: <https://mcyoung.xyz/2023/08/01/llvm-ir/>

## 요약

컴파일러 엔지니어 Sunny가 2023년에 작성한 글로, LLVM IR을 컴파일러 전문가가 아닌 성능 지향 프로그래머의 관점에서 소개한다.
Andrew Gallant의 트윗(“어셈블리 대신 LLVM IR을 보는 것이 성능 작업에 유용하다”)을 계기로 쓰였다. 어셈블리가 “프로세서가 무엇을 하는지” 보여준다면, LLVM IR은 “컴파일러가 코드에 대해 어떻게 생각하는지”를 보여준다는 것이 글의 핵심 전제다.

LLVM IR은 함수 정의(`define`), 기본 블록(Basic Block), 명령어로 구성된다.
각 함수는 라벨과 명령어의 목록이며, 모든 블록은 제어 흐름 이동 명령어(`br`, `ret`, `unreachable`)로 끝난다.
`%` 시길(sigil)은 로컬 레지스터, `@` 시길은 전역 또는 함수를 나타낸다. 정수 타입은 `i32`, `i64`처럼 `iN` 형식이며, 부호(signed/unsigned) 구분은 타입이 아닌 명령어 수준에서 한다(`sdiv`/`udiv`, `ashr`/`lshr` 등).

SSA(Static Single Assignment)는 LLVM IR의 핵심 제약이다.
각 레지스터는 함수 내에서 정확히 한 번만 정의된다. 뮤테이션이 필요할 때는 `alloca`로 스택 슬롯을 만들고 `load`/`store`로 접근하거나, `phi` 명령어로 다른 블록에서 도달한 값을 선택한다.
Clang과 Rust 모두 코드젠 단계에서 스택 변수를 `alloca`로 변환하고, 이후 `mem2reg` 패스가 `phi` 노드를 갖춘 순수 SSA 형식으로 승격시킨다.

타입 시스템은 정수(`iN`), 부동소수점(`float`, `double`, `bfloat`), 포인터(`ptr`), 배열(`[n x T]`), 구조체(`{T1, T2, ...}`), 벡터(`<n x T>`)를 지원한다.
집합 타입 연산에는 `extractvalue`/`insertvalue`(값 수준 접근), `getelementptr`(포인터 산술)이 있다. 특히 GEP(getelementptr)는 배열·구조체 내 특정 필드의 포인터를 계산하며, `inbounds` 어노테이션 없이는 순수 산술 연산이다.

Undefined Behavior는 poison 값 개념으로 구현된다.
poison 값은 “어떤 값이든 될 수 있는” 값으로, 최적화에 편리한 값을 선택할 수 있다. poison을 포인터로 역참조하거나, 나눗셈의 피제수로 사용하거나, 분기에 전달하면 UB가 된다.
`add nsw`(no signed wrap), `udiv exact`, `getelementptr inbounds` 등 많은 명령어가 조건 위반 시 poison을 생성하는 “덜 정의된” 버전을 가진다.
UB가 발생시키는 것이 아니라 poison이 사용될 때 UB가 되는 이 의미론은 대부분의 연산을 순수(pure)하게 유지해 자유로운 재정렬을 허용한다.

## 분析

### 독자 설정이 결정하는 설명 방식

이 글은 컴파일러 개론이 아니라 “성능 작업을 하는 일반 프로그래머”를 위한 글이다.
이 독자 설정이 설명 방식 전체를 결정한다. 어셈블리를 먼저 읽으라는 선수 조건, 최적화 이론보다 “읽는 법”에 집중하는 구성, Rust 예제를 통한 실제 컴파일러 출력 분석이 모두 이 관점에서 나온다.

컴파일러 교과서가 SSA를 알고리즘(Dominance Frontier, SSA Construction) 관점에서 접근한다면, 이 글은 SSA를 “읽기 위해 알아야 할 것” 관점에서 접근한다.
phi 노드를 이해하려고 Rust의 `pow` 함수를 `alloca` 버전으로 먼저 작성하고, `opt -p mem2reg`를 실행해 phi 노드가 생성되는 과정을 직접 보여주는 방식은 이 관점의 구체적 실천이다.

### poison 모델의 정밀한 설명

poison 값의 설명은 이 글의 기술적 하이라이트다. “UB가 즉시 발생”하는 C 의미론과 달리, LLVM의 “poison을 생성하지만 사용될 때만 UB”라는 의미론이 왜 더 유리한지를 명확하게 설명한다.

핵심 논점은 최적화의 재정렬 가능성이다.
C에서 `signed integer overflow`는 즉시 UB다. 즉, UB를 일으키는 연산이 사이드 이펙트를 가진 것처럼 취급해야 해서 재정렬이 어렵다.
LLVM에서 `add nsw`는 오버플로우 시 poison을 생성하지만, 이 poison이 “witnessed”(사용)되지 않으면 단순히 무시된다. 이렇게 되면 대부분의 연산이 순수 함수처럼 동작해 패스 간 자유로운 재정렬이 가능하다.
“UB를 만드는 것이 아니라 poison을 만드는 것”이라는 구분이 이 최적화 자유도의 원천이다.

### 실제 컴파일러 출력 분석의 교육적 가치

글 말미의 Rust `square` 함수 컴파일 출력 분석은 추상적 설명을 구체화한다.
디버그 모드에서 Rust가 `x * x`를 `@llvm.smul.with.overflow.i32`로 컴파일하는 것, `@llvm.expect.i1`로 패닉 분기를 “콜드”로 표시하는 것, `unreachable`로 블록을 종료하는 것을 순서대로 추적한다.

이 분석이 보여주는 것은 “컴파일러가 코드를 어떻게 생각하는지”다. Rust의 정수 오버플로우 패닉이 IR 수준에서 어떻게 구현되는지, `llvm.expect`가 옵티마이저에게 어떤 힌트를 주는지, 함수 속성(attributes)이 어떤 정보를 전달하는지를 직접 볼 수 있다.
“어셈블리를 읽으면 결과를 알 수 있고, IR을 읽으면 과정을 알 수 있다”는 글의 전제가 이 분석으로 완성된다.

### CFG 시각화 도구의 실용성

`opt -p dot-cfg`로 CFG를 `.dot` 파일로 내보내고 `dot` 명령으로 SVG를 생성하는 방법은 매우 실용적인 팁이다.
복잡한 Rust 함수(`parse_hex`)의 최적화 전후 CFG를 비교하는 것은 이론보다 직관적이다. `-O0`에서 모든 변수가 `alloca`인 것과 `-O2`에서 `alloca`가 사라지고 phi 노드만 남는 것을 시각적으로 비교하는 것은 최적화 패스의 효과를 한눈에 이해하게 한다.

## 비평

### 커버하지 않는 것을 명시하지 않는 곳이 있다

글이 부동소수점, SIMD, 예외 처리를 다루지 않는다고 언급하지만, 이것들이 실제 Rust 코드의 IR에 얼마나 자주 등장하는지는 언급하지 않는다.
Rust에서 `async`/`await` 코드, 클로저, trait 객체 디스패치를 컴파일하면 복잡한 IR이 나오는데, 이 글의 “입문” 수준으로 그 IR을 읽기는 어렵다.
“이 글을 읽은 다음 무엇을 보면 되는가”에 대한 답이 없다. 다음 단계로 연결되는 내용이 “LangRef를 읽어라”뿐인 것은 너무 성긴 안내다.

### phi 노드 설명의 순서

phi 노드를 설명하는 방법이 약간 역방향이다. `alloca` 버전을 먼저 작성하고 `mem2reg`를 실행해 phi가 나오는 것을 보여주는 접근은 직관적이지만, phi의 의미를 “최적화의 결과물”로 먼저 접하게 만든다.
phi가 왜 필요한지 — SSA에서 루프 변수의 두 버전(초기값과 업데이트된 값)을 합칠 방법이 없기 때문 — 를 먼저 설명하고, 그 해결책으로 phi를 소개하는 순서가 더 명확했을 것이다.

각주에서 MLIR이 phi 대신 “블록 인자(block arguments)” 방식을 사용한다고 언급하는 부분이 오히려 더 직관적인 대안을 보여주는데, 이것이 본문에 있었으면 phi 이해에 도움이 됐을 것이다.

### GEP 설명의 난이도

GEP(getelementptr)는 LLVM IR에서 가장 혼란스러운 명령어 중 하나다. “모든 포인터는 무한 크기 배열에 대한 포인터”라는 관점, 첫 번째 인덱스가 배열 원소 이동이고 이후 인덱스가 구조체 필드 접근이라는 구분은 짧은 예제 하나로 전달하기 어렵다.
LLVM 공식 FAQ 링크를 “helpful(sarcasm)”이라고 주석 처리하는 유머는 실제로 GEP FAQ가 이해하기 어렵다는 고백이기도 하다.
GEP가 포인터 역참조를 포함하지 않는다는 점 — 순수 주소 계산이라는 점 — 이 명확하게 강조되지 않아, 처음 읽는 독자는 `load`/`store` 없이 GEP만으로 메모리 접근이 된다고 착각할 수 있다.

## 인사이트

### IR을 읽는 것은 컴파일러와의 대화다

어셈블리 읽기가 “CPU와의 대화”라면, IR 읽기는 “컴파일러와의 대화”다.
이 구분은 단순한 비유가 아니다. 어셈블리는 특정 아키텍처의 명령어 집합을 알아야 하고, 최적화 결과물이라 원래 코드 의도를 추적하기 어렵다.
IR은 아키텍처 독립적이고, 최적화 전후를 모두 볼 수 있으며, 컴파일러가 어떤 가정을 했는지가 어노테이션으로 명시된다. `nsw`, `inbounds`, `noundef`, `willreturn` 같은 어노테이션들이 컴파일러의 추론 과정을 드러낸다.

성능 문제를 디버깅할 때 어셈블리를 먼저 보는 것이 관행이지만, 어셈블리는 “무엇이 됐는가”를 보여주고 “왜 그렇게 됐는가”를 설명하지 않는다.
컴파일러가 특정 최적화를 적용하지 않은 이유, 예상한 인라이닝이 일어나지 않은 이유, 불필요한 분기가 남아 있는 이유를 이해하려면 IR을 봐야 한다.
IR은 컴파일러의 “의사결정 과정”이 담긴 문서다. 어셈블리는 결과물이고, IR은 결과물이 나오기 전의 중간 의도다.

이 관점에서 Godbolt의 “LLVM IR 출력” 옵션은 단순한 디버깅 도구가 아니라, 컴파일러의 시각으로 코드를 보는 창이다. 같은 코드를 `-O0`와 `-O2`의 IR로 비교하는 것이 어셈블리를 보는 것보다 “컴파일러가 무엇을 하고 있는지”를 더 직접적으로 드러낸다.

### poison 모델이 보여주는 “정의되지 않음”의 설계

UB를 “즉시 폭발”로 처리하는 C 의미론과 LLVM의 poison 모델 사이의 차이는 단순한 구현 세부사항이 아니라 의미론 설계의 근본적 선택이다.
C에서 UB는 “이 코드가 실행되면 우주가 폭발할 수 있다”는 선언이다. 컴파일러는 UB에 도달하는 코드가 실행되지 않는다고 가정하고 역방향으로 추론해 다른 코드까지 제거할 수 있다. 이것이 “time-traveling UB”다.
LLVM의 poison은 더 국소적(local)이다. 잘못된 연산이 poison을 생성하지만, 그 poison이 “위험한” 방식으로 사용되지 않으면 단지 임의의 값으로 처리된다.

이 설계의 핵심 이점은 최적화의 독립성이다. C의 즉시-UB 모델에서 최적화 A가 UB-유발 코드를 이동시키면, 이후 최적화 B가 다른 코드를 제거하기 위해 UB를 사용할 때 A가 UB의 위치를 바꿨다는 사실을 알아야 한다. 즉, 최적화들이 글로벌 UB 상태를 공유해야 한다.
LLVM의 poison 모델에서 각 최적화 패스는 “poison을 사용하지 않으면 문제 없다”는 단순한 규칙만 따르면 된다. 패스들이 상호 독립적으로 동작할 수 있다.

소프트웨어 설계 관점에서 이것은 “에러를 즉시 폭발시키는 것”과 “에러 상태를 포함한 채로 계속 진행하다가 에러 상태를 실제로 관찰할 때 폭발하는 것”의 차이다. 컴파일러 최적화 맥락에서는 후자가 각 패스의 독립성을 보장하는 더 나은 선택이다.

### 어셈블리와 IR 사이의 추상화 레벨 선택

어셈블리를 읽는 사람과 IR을 읽는 사람은 다른 질문을 한다.
어셈블리를 읽는 사람은 “이 코드가 어떤 명령어를 실행하는가?”를 묻는다. 캐시 라인 정렬, 분기 예측, SIMD 명령어 선택이 관심사다.
IR을 읽는 사람은 “컴파일러가 내 코드에 대해 무엇을 알고 있는가?”를 묻는다. 어노테이션, 최적화 기회, 컴파일러가 가정하는 불변 조건이 관심사다.

성능 최적화 작업에서 어느 레벨을 봐야 하는지는 문제의 성격에 달려 있다.
“이 함수가 왜 느린가?”를 이해하려면 어셈블리가 필요하다. 실제 실행되는 명령어, 메모리 접근 패턴, 분기 패턴이 거기 있다.
“컴파일러가 왜 이 최적화를 적용하지 않는가?”를 이해하려면 IR이 필요하다. 컴파일러가 어떤 정보를 갖고 있고 어떤 가정을 했는지가 거기 있다.

이 두 레벨을 자유롭게 오가는 능력이 깊은 성능 최적화 작업의 필수 역량이다.
Sunny가 “나는 IR을 읽으면서 컴파일러에 입문했다”고 말하는 것은 개인적 경험을 넘어 컴파일러를 이해하는 진입점으로서 IR의 가치를 말하는 것이다. 컴파일러의 IR은 블랙박스의 내부를 들여다보는 창이며, 그 창을 열 수 있는 사람은 컴파일러와 협력하는 것이 아니라 컴파일러를 이해하는 수준에 도달한다.
