# 프로그래밍 언어 개발의 감초 — LLVM의 이해와 활용

원문: <https://www.itworld.co.kr/article/3563713>

## 요약

InfoWorld의 Serdar Yegulalp이 2023년 8월에 작성한 글로, LLVM을 언어 개발자 관점에서 실용적으로 소개한다. 새로운 언어를 만들거나 기존 언어 컴파일러를 강화하려는 개발자를 독자로 상정한다.

LLVM은 기본적으로 “프로그래밍 방식으로 머신 네이티브 코드를 생성하는 라이브러리”다. 개발자가 API를 통해 IR(Intermediate Representation)을 생성하면, LLVM이 이를 독립형 바이너리로 컴파일하거나 JIT 컴파일로 실행한다. Swift, Rust, Clang, Mono의 네이티브 컴파일 옵션, Kotlin Native가 LLVM을 핵심 백엔드로 사용한다. LLVM은 함수, 전역 변수, 코루틴, C FFI 같은 언어 공통 구조를 IR 레벨에서 이미 제공하므로, 언어 개발자는 언어 고유의 부분에 집중할 수 있다.

LLVM의 가장 큰 이점은 이식성이다. LLVM의 IR은 처음부터 이식 가능한 어셈블리로 설계됐다. 정수 형식이 기반 하드웨어의 비트 폭에 제한되지 않아 128비트 정수도 표현 가능하고, 특정 프로세서 명령어 집합에 맞게 출력을 가공하는 작업은 LLVM이 처리한다. IBM이 z/OS, Power 기반 Linux, AIX를 지원하기 위한 코드를 LLVM에 기여한 것이 이 설계의 결실이다.

주요 활용 패턴은 네 가지다. 첫째, Clang처럼 AOT(Ahead-Of-Time) 컴파일러로 사용하는 것이다. 둘째, JIT 컴파일로, Julia는 REPL 상호작용성을 위해, Numba는 Python 함수의 수치 연산 가속을 위해, PostgreSQL은 쿼리 컴파일로 최대 5배 성능 향상을 위해 사용한다. 셋째, 자동 코드 최적화다. 함수 인라이닝, 데드 코드 제거, 루프 언롤링을 비롯한 공격적인 최적화를 선택적으로 활성화할 수 있다. 넷째, 도메인 특화 언어(DSL) 개발로, Emscripten이 LLVM IR을 JavaScript/WebAssembly로 변환하고 NVIDIA CUDA 컴파일러가 GPU 네이티브 지원을 언어에 추가한다.

LLVM이 하지 않는 것도 명확하다. 문법 파싱(lex/yacc, ANTLR 등 별도 도구를 써야 한다), 패키지 관리와 툴체인 업그레이드, 가비지 수집 메커니즘은 LLVM의 범위 밖이다. 단, GC를 구현하기 위한 메타데이터 표시 도구는 제공한다. 개발 언어로는 C, C++가 기본이지만, Rust, Haskell, OCaml, Node.js, Go, Python 등 C 라이브러리를 호출할 수 있는 언어에서도 바인딩을 통해 사용 가능하다. Python에서는 Numba 팀이 만든 `llvmlite`가 사실상의 표준이다.

## 분析

### 실용적 범위 설정의 명확함

이 글은 LLVM의 내부 구조보다 “무엇에 쓸 수 있는가”와 “어떻게 시작하는가”에 집중한다. 활용 패턴(AOT, JIT, 최적화, DSL)을 구체적 사례(Julia, Numba, PostgreSQL, Emscripten, CUDA)로 보여주는 구성은 언어 개발자가 자신의 상황에 LLVM이 맞는지 판단하는 데 직접적으로 도움이 된다.

특히 “LLVM이 하지 않는 것” 섹션이 실용적 가치가 높다. 도구를 선택할 때 할 수 없는 것을 아는 것이 할 수 있는 것을 아는 것만큼 중요하다. 문법 파싱, 패키지 관리, GC가 LLVM 범위 밖이라는 사실을 명시해 “LLVM이면 다 된다”는 오해를 방지한다.

### Python 바인딩 생태계 분석

Python에서 LLVM을 사용하려는 개발자에게 네 가지 선택지(`llvmlite`, LLVM 공식 C API 바인딩, `llvmpy`, `llvmcpy`)를 유지보수 상태와 함께 비교하는 것은 시간을 절약해 주는 실용적 정보다.
`llvmpy`가 2015년, `llvmcpy`가 2019년 이후 유지보수가 중단됐다는 사실은 단순한 상태 정보가 아니다. LLVM이 6개월마다 메이저 릴리스를 내놓는 빠른 개발 속도를 감안하면, 오래된 바인딩은 최신 API와의 불일치로 사실상 사용 불가 상태일 가능성이 높다.

`llvmlite`가 LLVM의 전체 API 중 일부만 구현한다는 점도 중요한 정보다. “대다수 LLVM 사용자에게 필요한 것들”이라는 설명은 정확하지만, 고급 사용자가 구현되지 않은 기능이 필요할 때 `llvmlite`로는 한계가 있음을 암시한다.

### MLIR 언급의 위치와 비중

MLIR(Multi-Level Intermediate Representation)을 DSL 개발 섹션에서 간략히 언급한다. “일부 DSL을 LLVM IR로 변환하기 위해 프론트엔드에서 많은 노력이 필요하다는 점”을 해결책으로 소개하지만, MLIR이 어떻게 그 문제를 해결하는지(Dialect 개념, Lowering 체인)는 설명하지 않는다.

AI/ML 컴파일러 시대에 MLIR의 중요성은 LLVM IR에 버금간다. TensorFlow XLA, PyTorch torch-mlir, OpenAI Triton이 모두 MLIR 위에 구축됐다는 사실을 고려하면, MLIR을 한 문단 각주 수준으로 다루는 것은 2023년 시점에서도 과소평가다.

## 비평

### 깊이 없는 사례들

Julia, Numba, PostgreSQL 사례는 JIT 컴파일의 동기를 설명하지만, LLVM JIT을 실제로 구성하는 복잡성은 전혀 다루지 않는다. LLVM의 ORC JIT, MCJIT 중 어떤 것을 써야 하는지, JIT 컴파일된 코드의 수명 관리가 어떻게 이루어지는지 같은 실제적인 질문에 대한 답이 없다.

Kaleidoscope 자습서를 학습 자원으로 안내하는 것은 좋지만, 자습서가 다루지 않는 영역(예외 처리, GC 통합, 디버그 정보 생성)에 대한 안내가 없다. 실제 언어를 만들려는 개발자에게는 자습서 이후의 gap이 핵심 장벽이다.

### 이식성 주장의 한계 미언급

LLVM의 이식성이 가장 큰 이점이라는 주장은 맞지만, 한계를 언급하지 않는다.
LLVM의 티어별 플랫폼 지원 수준이 다르다. x86_64, AArch64는 최고 품질이지만, 덜 일반적인 아키텍처는 지원 수준이 낮거나 활발히 유지되지 않는다. “현재와 미래의 모든 종류의 하드웨어를 쉽게 지원한다”는 표현은 과장이다.

또한 이식성이 뛰어나더라도 플랫폼 ABI(Application Binary Interface) 차이, 운영체제별 시스템 콜 규약, C 런타임 링킹 방식 차이는 언어 프론트엔드가 직접 처리해야 한다. LLVM이 이 부분을 “알아서 처리”하지는 않는다.

### GC 설명의 불완전함

“LLVM은 가비지 수집 메커니즘을 제공하지 않지만, GC를 쉽게 작성할 수 있게 해주는 메타데이터를 코드에 표시할 수 있다”는 설명은 부정확하다.
LLVM이 제공하는 것은 GC 루트(스택에서 포인터를 찾는 정보), Statepoints(GC가 안전하게 동작할 수 있는 지점 표시), GC 전략 인터페이스다. “쉽게 작성할 수 있게 해주는” 수준이 아니라, GC를 올바르게 구현하려면 LLVM의 GC 인프라를 깊이 이해해야 한다. 이 부분이 실제로 많은 언어 개발자에게 큰 장벽이다.

## 인사이트

### “라이브러리로서의 컴파일러”가 가진 생태계 전략

LLVM이 컴파일러가 아니라 “컴파일러를 만드는 라이브러리”로 설계됐다는 사실은 단순한 아키텍처 결정이 아니라 생태계 전략이다.
컴파일러를 하나의 모놀리식 프로그램으로 만들면, 그것을 개선하거나 새 언어에 적용하려면 전체를 이해해야 한다. 라이브러리로 만들면, 각 언어 팀이 자신이 필요한 부분만 가져다 쓰고 나머지는 LLVM의 구현에 위임할 수 있다.

이 설계의 두 번째 효과는 투자의 집중이다. Swift, Rust, Clang, Julia, Kotlin Native가 모두 같은 옵티마이저를 사용하면, 옵티마이저 개선이 모든 언어에 동시에 이익이 된다. 한 언어 팀이 루프 벡터화를 개선하면 다른 모든 언어가 혜택을 받는다.
GCC의 모놀리식 설계와 비교하면 이 차이가 명확하다. GCC를 특정 목적으로 재사용하려면 GPL 라이선스 의무와 단단히 결합된 내부 구조를 다뤄야 한다. LLVM의 라이브러리 모델과 자유로운 라이선스(Apache 2.0)가 결합해 재사용 비용을 낮췄고, 그것이 현재의 LLVM 중심 생태계를 만들었다.

### JIT와 AOT의 경계가 흐려지는 추세

Julia, Numba, PostgreSQL이 JIT를 사용하는 이유를 보면 공통 패턴이 있다. 컴파일 시점에 알 수 없는 정보(실제 타입, 실제 입력 분포, 실제 쿼리 패턴)를 런타임에 활용하려는 것이다.

이것은 소프트웨어의 최적화 전략이 “컴파일 시점 vs. 런타임”의 이분법에서 “컴파일 시점 + 프로파일 + 재컴파일” 연속체로 이동하고 있다는 신호다. PGO(Profile-Guided Optimization)가 AOT 컴파일러에 적용되고, JIT 컴파일러가 “충분히 뜨거운” 코드를 감지해 더 공격적으로 최적화하는 방식이 이 연속체의 구현이다.

LLVM이 AOT와 JIT 모두를 같은 IR로 지원한다는 사실은 이 연속체를 구현하는 데 구조적으로 유리하다. Clang이 `-O0`에서 시작해 `-O3`까지 동적으로 최적화 수준을 조정하거나, 런타임에 수집된 프로파일을 사용해 재컴파일하는 것이 같은 코드베이스 위에서 가능하다.

### LLVM이 제공하지 않는 것들의 공통점

LLVM이 제공하지 않는 것들 — 문법 파싱, 패키지 관리, 가비지 수집 — 을 나열하면 공통점이 보인다. 모두 “언어마다 답이 다른” 문제들이다.

문법 파싱은 언어의 가장 본질적인 개성이다. 같은 기능도 언어마다 완전히 다른 문법으로 표현된다. 패키지 관리는 언어 생태계의 문화와 규범을 담는다. npm과 cargo와 pip은 기술적으로 유사한 문제를 풀지만 철학이 다르다. GC는 언어의 메모리 모델을 결정하는 핵심이다. 레퍼런스 카운팅, 마크앤스윕, 세대별 GC 중 어느 것이 맞는지는 언어의 목표에 달려 있다.

LLVM이 이 부분들을 제공하지 않는 것은 능력의 부재가 아니라 영역 설정의 선택이다. “이식 가능한 코드 생성”이라는 핵심 문제에 집중하고, 언어 고유의 문제는 각 언어 팀에 맡기는 것이다. 이 경계 설정이 LLVM이 수십 개의 언어에 채택될 수 있었던 이유이기도 하다. 만약 LLVM이 특정 GC 방식이나 파싱 전략을 강제했다면, 훨씬 적은 언어가 LLVM을 선택했을 것이다.

### 언어 바인딩 생태계의 생존 법칙

Python LLVM 바인딩의 역사 — `llvmpy`(2015년 중단), 공식 바인딩(유지보수 중단), `llvmcpy`(2019년 중단) — 는 오픈소스 생태계에서 바인딩 라이브러리의 생존 조건을 보여준다.

LLVM은 6개월마다 메이저 버전을 내놓고, 각 버전이 API를 변경한다. 바인딩 라이브러리는 이 변화를 지속적으로 추적해야 한다. 이것은 상당한 유지보수 부담이다. `llvmpy`와 `llvmcpy`가 중단된 이유는 기술적 결함이 아니라 이 지속적 유지보수의 부담을 감당할 수 없었기 때문이다.

`llvmlite`가 살아남은 것은 Numba 프로젝트라는 실제 사용자와 그 자금 지원이 있었기 때문이다. 유지보수 지속성은 라이브러리의 기술적 품질보다 “이 라이브러리가 중요한 프로젝트의 생존에 필수적인가”에 달려 있다. 이것은 오픈소스 생태계에서 반복되는 패턴이다. 독립 프로젝트로 출발해 살아남는 라이브러리는 대개 큰 프로젝트의 하위 구성요소로 흡수되거나, 기업의 지원을 받거나, 활발한 커뮤니티가 형성된 경우다.
