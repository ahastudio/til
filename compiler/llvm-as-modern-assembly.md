# LLVM — 현대의 어셈블리어

원문: <https://www.linkedin.com/pulse/low-level-programming-llvm-revolution-universal-all-ayman-alheraki-wbtff/>

## 요약

Ayman Alheraki가 2025년 10월에 LinkedIn에 게시한 글로, 어셈블리 프로그래밍의 역사적 맥락에서 출발해 LLVM이 저수준 프로그래밍을 어떻게 재정의했는지를 설명한다.

프로그래밍 초기에는 개발자가 x86, ARM, MIPS 등 각 아키텍처의 레지스터와 하드웨어 명령어를 직접 다뤘다. 한 기계를 위한 코드를 다른 기계에서 실행하려면 처음부터 다시 작성해야 했다. C, C++ 같은 고수준 언어가 이 세부사항을 추상화했지만, 컴파일러는 여전히 각 하드웨어 타입마다 별도의 코드 생성 로직을 구현해야 했다. ISA(Instruction Set Architecture) 다양성, 엔디언 차이, 레지스터 레이아웃, 호출 규약의 차이가 컴파일러 개발의 핵심 장벽이었다.

LLVM은 이 문제를 IR(Intermediate Representation)로 해결했다. IR은 고수준 소스코드와 타깃 머신 코드 사이에 위치하는 아키텍처 중립적인 저수준 언어다. 프론트엔드 컴파일러들(C, C++, Rust, Swift, Fortran 등)이 소스를 LLVM IR로 변환하면, LLVM 백엔드가 이 IR을 ARM, x86, RISC-V, MIPS, PowerPC 등 어느 아키텍처의 네이티브 코드로든 변환한다. 이 모듈식 설계가 LLVM을 수많은 언어의 범용 백엔드로 만들었다.

LLVM 이전의 GCC는 모놀리식이었다. 각 언어 프론트엔드와 백엔드가 단단히 결합돼 있어 새로운 언어나 새 아키텍처를 추가하는 데 큰 노력이 필요했다. LLVM은 프론트엔드와 백엔드를 분리해 이 패턴을 깼다. 핵심 이점은 다섯 가지다. 이식성(코드 수정 없이 다른 아키텍처로 컴파일), 최적화(소스 언어나 타깃 하드웨어에 독립적인 IR 레벨 최적화), 재사용성(새 언어가 성숙한 백엔드를 처음부터 만들 필요 없이 가져다 쓸 수 있음), JIT 컴파일(동적 코드 생성과 실행), 크로스 컴파일(플래그 하나로 다른 아키텍처 타깃팅)이다.

저자의 핵심 테제는 “LLVM IR은 현대의 어셈블리어가 됐다”는 것이다. 특정 CPU에 묶이지 않으면서도 모든 저수준 연산을 표현할 수 있을 만큼 구체적이다. “한 번 쓰면 어디서나 실행”이라는 약속을 가상 머신이나 바이트코드 인터프리터가 아닌 진정한 하드웨어 수준의 통합으로 실현한다는 것이다.

## 분析

### 역사적 서사로 기술적 혁신을 설명하는 방식

이 글은 LLVM의 기술적 구조보다 “왜 LLVM이 혁명적인가”를 역사적 서사로 설명한다. 어셈블리 시대의 고통(아키텍처마다 코드를 다시 써야 하는 문제)에서 출발해 GCC의 모놀리식 한계를 거쳐 LLVM의 모듈식 해법으로 이어지는 서사 구조다.

이 서사 방식은 독자가 “LLVM이 무엇인가”를 이해하기 전에 “LLVM이 왜 필요했는가”를 먼저 이해하게 만든다. 기술 문서에서 드문 접근이다. 대부분의 LLVM 소개 글이 구조와 기능에서 시작하는 것과 대조된다.

### “현대의 어셈블리어”라는 프레임의 함의

“LLVM IR은 현대의 어셈블리어”라는 테제는 기술적으로 정확하지 않지만 개념적으로 유용하다.
전통적 어셈블리는 특정 아키텍처의 명령어 집합과 1대1로 대응한다. LLVM IR은 아키텍처 독립적이므로 어셈블리와 다르다. 그러나 “인간이 읽을 수 있는 최저수준 표현”이라는 의미에서 어셈블리의 현대적 역할을 한다고 볼 수 있다.

이 프레임은 “write once, run everywhere”를 Java가 JVM으로 구현하는 방식과 LLVM이 네이티브 코드 생성으로 구현하는 방식의 차이를 명확하게 한다. Java는 런타임에 추상화 비용을 지불하지만, LLVM은 컴파일 시점에 아키텍처 특화 코드를 생성해 런타임 비용이 없다.

### GCC 모놀리식 vs LLVM 모듈식 대비

GCC의 모놀리식 설계와 LLVM의 모듈식 설계를 명시적으로 대비하는 것은 이 글의 강점이다. “프론트엔드와 백엔드의 분리”가 컴파일러 설계의 패러다임 전환이었다는 점을 구체적으로 짚는다.

그러나 GCC도 이후 내부를 모듈화하는 방향으로 발전했고, LLVM도 완전히 독립적인 모듈은 아니라는 점은 다루지 않는다. GCC가 GPL 라이선스로 기업 재사용에 제약이 있었던 반면 LLVM은 Apache 2.0을 선택한 것도 채택에 결정적이었는데, 이 맥락이 없다.

## 비평

### 내용의 얕은 깊이

이 글은 LLVM의 존재와 이점을 소개하는 수준에서 멈춘다. 각 이점(이식성, 최적화, JIT 등)이 어떻게 구현되는지, 실제로 언어를 만들 때 무엇이 필요한지는 다루지 않는다. 7554 단어를 넘긴 mcyoung의 입문 글과 비교하면 이 LinkedIn 글은 600여 단어 분량에 불과하다.

“GCC는 모놀리식이었고 LLVM은 모듈화했다”는 설명은 정확하지만, 그 모듈화가 실제로 어떻게 동작하는지 — 프론트엔드가 IR을 어떻게 생성하고, 옵티마이저가 어떤 패스를 실행하고, 백엔드가 어떻게 기계 코드를 선택하는지 — 를 알 수 없다.

### 검증되지 않은 주장들

“write once, run everywhere의 약속을 실현했다”는 결론은 과장이다. LLVM이 같은 IR에서 다른 아키텍처의 바이너리를 생성하는 것은 맞지만, 플랫폼 ABI 차이, 운영체제별 시스템 콜, 라이브러리 가용성 차이로 인해 “수정 없이 모든 곳에서 실행”은 현실과 거리가 있다. Rust나 C++로 작성한 코드를 x86에서 RISC-V로 단순 재컴파일한다고 바로 동작하지는 않는다.

또한 “AI 프레임워크에서 JIT 컴파일이 중요하다”는 언급이 있지만, 실제로 TensorFlow, PyTorch 같은 프레임워크가 LLVM JIT을 어떻게 사용하는지(또는 MLIR을 통해 사용하는지)는 설명되지 않는다. 결론이 내용보다 크다.

### 다이어그램의 역할

ASCII 다이어그램으로 프론트엔드 → IR 옵티마이저 → 백엔드 흐름을 시각화하는 시도는 좋지만, LinkedIn 포스트 형식에서 ASCII 아트는 렌더링이 깔끔하지 않다. 의도는 좋지만 실제 읽기에는 불편하다.

## 인사이트

### 저수준 프로그래밍의 재정의: 아키텍처에서 의미론으로

LLVM이 “저수준 프로그래밍을 재정의했다”는 주장의 핵심은 저수준의 기준이 바뀌었다는 것이다.
어셈블리 시대에 “저수준”이란 특정 아키텍처의 레지스터와 명령어를 직접 다루는 것이었다. 저수준 프로그래머는 x86 레지스터 이름(`eax`, `rbx`)과 ARM 레지스터 이름(`r0`, `sp`)을 각각 알아야 했다.

LLVM IR이 등장하면서 저수준의 기준이 “아키텍처 종속적 세부사항 다루기”에서 “프로그램의 의미론을 직접 조작하기”로 이동했다. LLVM IR에서 변수는 레지스터가 아니라 무한한 가상 레지스터(`%1`, `%2`, ...)이고, 연산은 아키텍처 명령어가 아니라 `add`, `load`, `store` 같은 의미론적 연산이다.

이 전환은 저수준 프로그래밍의 진입 장벽을 낮추는 동시에 새로운 추상화 계층을 만들었다. 과거에는 어셈블리를 모르면 컴파일러를 이해할 수 없었지만, 이제는 LLVM IR을 이해하면 컴파일러 최적화를 이해할 수 있다. 저수준 지식의 단위가 아키텍처에서 IR로 대체됐다.

### “쓰면 어디서나 실행”의 두 가지 경로와 그 차이

“write once, run everywhere”를 실현하는 경로는 역사적으로 두 가지였다. 하나는 Java가 선택한 방식 — 가상 머신 위에서 바이트코드를 런타임에 해석하거나 JIT 컴파일. 다른 하나는 LLVM이 제안하는 방식 — 컴파일 시점에 각 타깃에 최적화된 네이티브 코드를 미리 생성.

이 두 경로는 성능, 시작 시간, 이식성의 트레이드오프가 다르다. Java JVM 방식은 런타임에 타입 정보를 활용한 최적화가 가능하지만 JVM 설치와 시작 시간 오버헤드가 있다. LLVM 방식은 런타임 비용이 없지만, 컴파일 시점에 타깃 아키텍처를 알아야 한다.

흥미롭게도 현대에는 이 두 경로가 수렴하고 있다. GraalVM이 Java 코드를 네이티브 이미지로 AOT 컴파일하고, LLVM의 ORC JIT이 동적 컴파일을 지원한다. “이식성의 두 경로”가 LLVM이라는 공통 인프라 위에서 만나고 있다는 것이 현재의 상황이다.

### 컴파일러 혁신이 언어 혁신을 가능하게 하는 구조

Rust, Swift, Kotlin Native, Zig, Mojo가 모두 10년 내에 등장할 수 있었던 것은 이들이 LLVM이라는 성숙한 컴파일러 인프라 위에서 만들어졌기 때문이다.
LLVM 이전에 새 언어를 만드는 것은 컴파일러 전체를 처음부터 만드는 것이었다. 파서, 시맨틱 분석, 코드 생성, 최적화, 어셈블러, 링커 — 모두 새로 구현해야 했다. 이것은 수십 명의 컴파일러 전문가와 수년의 시간을 요구했다.

LLVM 이후에는 파서와 시맨틱 분석(프론트엔드), 그리고 IR 생성만 구현하면 성숙한 최적화기와 모든 아키텍처의 백엔드를 자동으로 얻는다. 새 언어 설계의 가장 흥미로운 부분에 집중할 수 있게 됐다.

이것은 “플랫폼이 혁신을 가능하게 한다”는 패턴의 전형이다. iOS 앱 생태계가 iOS 플랫폼 없이 불가능했듯, 현대 언어 생태계의 풍요로움은 LLVM 플랫폼 없이 불가능했다. LLVM은 언어 디자인의 진입 장벽을 낮춤으로써 수십 개의 실험적 언어가 등장하고 도태되고 살아남는 진화적 과정을 가능하게 했다. Rust가 메모리 안전성이라는 아이디어를 실험할 수 있었던 것도 컴파일러 인프라를 처음부터 만들지 않아도 됐기 때문이다.
