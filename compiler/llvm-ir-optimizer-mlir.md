# LLVM Deep Dive — IR, Optimizer, Backend, MLIR 완전 정복

원문: <https://www.youngju.dev/blog/culture/2026-04-15-llvm-compiler-infrastructure-ir-optimizer-backend-mlir-deep-dive-guide-2025>

## 요약

Youngju Kim이 2026년 4월에 작성한 글로, LLVM 컴파일러 인프라의 전체 구조를 프론트엔드부터 MLIR까지 깊이 다룬다.
LLVM은 2000년 Chris Lattner가 석사 논문으로 시작해 Apple의 투자를 거쳐 현재 Clang, Rust, Swift, Julia, Zig, Mojo, CUDA nvcc가 사용하는 사실상의 표준 컴파일러 인프라가 됐다.

LLVM의 핵심 혁신은 “3단 아키텍처”다. 프론트엔드(언어 → IR), 옵티마이저(IR → IR), 백엔드(IR → 머신 코드)로 분리함으로써 N개 언어 × M개 타겟이 필요하던 N×M 문제를 N+M으로 줄였다.
LLVM IR은 이 구조의 공통 계약이다. IR은 SSA(Static Single Assignment) 형식으로, 각 가상 레지스터가 정확히 한 번 정의된다는 규칙이 데이터 흐름 분석과 대부분의 최적화를 단순하게 만든다.
IR은 인메모리 C++ 객체, 텍스트 형식(`.ll`), 직렬화 비트코드(`.bc`) 세 가지 형식으로 존재한다.

옵티마이저는 수백 개의 Pass가 순차적으로 실행되는 Pass Manager 구조다.
SROA/Mem2Reg(스택 할당 → SSA 변환), InstCombine(패턴 기반 피프홀 최적화), GVN(전역 값 번호 매기기로 중복 계산 제거), LICM(루프 불변식 이동), Inliner(함수 인라이닝), DCE(죽은 코드 제거)가 핵심 패스다.
Pass 순서가 중요하다. InstCombine이 상수를 접으면 DCE가 더 많은 코드를 제거할 수 있는 식으로 패스들이 상호작용한다.

백엔드는 IR을 머신 코드로 변환한다. SelectionDAG(전통적, 트리 패턴 매칭 기반)와 GlobalISel(최신, MI 위에서 직접 instruction selection) 두 경로가 있다.
2025년 기준 AArch64는 GlobalISel이 기본이고 x86_64는 여전히 SelectionDAG를 사용한다.
레지스터 할당은 가상 레지스터를 물리 레지스터에 매핑하는 가장 복잡한 단계로, Greedy(live range 분석 + 그래프 채색), Fast(-O0용 단순 할당), PBQP(최고 품질, 느림) 전략을 지원한다.

LTO(Link-Time Optimization)는 파일 경계를 넘는 최적화를 가능하게 한다.
Full LTO는 모든 IR을 한 덩어리로 모아 최적화해 품질이 가장 높지만 빌드가 느리다.
ThinLTO는 모듈별 병렬 최적화와 작은 요약 교환으로 이득의 80~90%를 얻으면서 빌드 시간을 대폭 줄인다. Chrome, Firefox, Rust 모두 ThinLTO를 기본으로 사용한다.
PGO(Profile-Guided Optimization)는 계측 빌드 → 대표 워크로드 실행 → 프로파일 수집 → 재빌드의 사이클로 핫 함수 인라이닝, 분기 예측, 루프 언롤 크기를 실측 데이터 기반으로 결정한다. Chromium은 PGO로 6~10% 성능 향상을 얻었다.

MLIR(Multi-Level IR)은 “LLVM IR이 너무 저수준”이라는 문제에서 출발한다.
2019년 Chris Lattner가 Google에서 설계한 MLIR의 핵심은 Dialect 개념이다. 한 IR 안에 TensorFlow Dialect, Linalg Dialect, Affine Dialect, LLVM Dialect가 공존하며 Lowering을 통해 점진적으로 저수준화된다.
TensorFlow XLA, PyTorch torch-mlir, OpenAI Triton, Mojo가 MLIR 위에 구축됐다.

## 분析

### SSA 형식이 컴파일러 최적화를 단순하게 만드는 구조적 이유

SSA(Static Single Assignment)는 “각 변수는 정확히 한 번 정의된다”는 단순한 규칙이지만, 이 규칙 하나가 컴파일러 분석의 복잡도를 근본적으로 낮춘다.
전통적인 IR에서 “이 시점에 `x`의 값은 무엇인가?”라는 질문은 모든 이전 대입을 추적해야 하는 도달 정의(reaching definition) 분석을 요구한다.
SSA에서는 질문 자체가 사라진다. `%x`는 단 한 곳에서 정의되므로 정의 위치가 곧 값이다.

GVN(Global Value Numbering)이 좋은 사례다. “이 두 표현식이 같은 값인가?”를 묻는데, SSA에서는 같은 연산에 같은 피연산자가 있으면 같은 번호를 부여하는 것만으로 충분하다.
SSA가 없으면 메모리 접근이나 별칭(aliasing) 분석 없이는 같은 값임을 판단하기 어렵다.
GCC도 결국 GIMPLE/SSA로 이동했고, Rust의 MIR, Swift의 SIL 모두 SSA를 채택했다. SSA가 현대 컴파일러 IR의 표준이 된 것은 이 분석 단순화 때문이다.

분기가 있는 코드에서는 Phi 노드가 필요하다. `if (a > b) return a; else return b;`에서 반환값이 어느 경로에서 왔는지 추적하는 것이 Phi 노드다. Phi 노드는 SSA의 단순성을 유지하면서 제어 흐름을 처리하는 필수 보완재다.

### 3단 아키텍처의 실질적 효과

N×M에서 N+M으로의 전환은 단순한 수학이 아니라 생태계 구조의 변화다.
LLVM 이전에는 새 언어가 모든 타겟 아키텍처에 대한 백엔드를 구현해야 했다. x86_64, ARM, MIPS, PowerPC, RISC-V... 각 언어가 이 작업을 반복했다.
LLVM 이후에는 프론트엔드 하나만 만들면 모든 백엔드를 자동으로 얻는다. Zig, Crystal, Mojo 같은 최근 언어들이 빠르게 여러 플랫폼을 지원할 수 있는 이유다.

역방향도 마찬가지다. 새 CPU 아키텍처가 나오면 LLVM 백엔드 하나를 추가하는 것만으로 모든 언어가 그 CPU를 지원하게 된다.
RISC-V가 학술 연구에서 시작해 임베디드 시장까지 빠르게 침투할 수 있었던 데는 LLVM 백엔드 지원이 결정적이었다.

### LTO와 PGO의 결합이 만드는 최적화의 층위

LTO와 PGO는 서로 다른 정보 부재 문제를 해결한다.
LTO는 컴파일 시점의 “파일 경계”가 가리는 정보를 드러낸다. 컴파일러가 `foo.c`를 컴파일할 때 `bar.c`의 함수 정의를 모르는 문제를 링크 시점에 해결한다.
PGO는 컴파일 시점의 “런타임 동작” 정보 부재를 해결한다. 코드가 실제로 어떻게 실행되는지를 프로파일로 수집해 최적화에 반영한다.

두 기법을 결합하면 컴파일러가 “전체 프로그램의 정적 구조”와 “실제 실행 경로” 두 가지 정보를 모두 갖게 된다.
Chrome은 이 조합(ThinLTO + PGO)으로 바이너리 크기 7% 감소, 시작 시간 8% 개선을 달성했다.
AutoFDO는 프로덕션 `perf record` 데이터를 프로파일로 사용해 “실제 사용자 워크로드”를 최적화에 반영하는 실용적 접근이다.

### MLIR Dialect 설계의 의미

MLIR 이전에는 각 AI 프레임워크가 LLVM IR과의 경계에서 도메인 개념을 잃었다.
“텐서 행렬 곱”이라는 고수준 개념이 LLVM IR로 내려오는 순간 수천 개의 스칼라 연산으로 분해되고, 행렬 구조 자체를 활용하는 타일링(tiling)이나 fusion 최적화는 불가능해진다.

MLIR의 Dialect는 이 도메인 정보 손실 문제를 “정보가 사라지는 시점을 뒤로 미루는” 방식으로 해결한다.
TF Dialect에서 Linalg Dialect로, Linalg에서 Affine Dialect로, Affine에서 LLVM Dialect로 단계적으로 내려오면서 각 레벨에서 그 레벨에 적합한 최적화를 적용할 수 있다.
Triton이 GPU 커널 최적화를 Dialect 수준에서 표현하고, Mojo가 Python 유사 문법으로 고성능 ML 코드를 작성할 수 있는 것은 이 설계 덕분이다.

## 비평

### 깊이와 범위 사이의 타협

이 글은 LLVM의 거의 모든 측면을 다루지만 각 주제를 진정한 깊이로 파고들기엔 분량이 부족하다.
레지스터 할당의 Greedy 알고리즘이 “live range 분석 + 그래프 채색”이라고 소개하지만, 그래프 채색이 NP-완전 문제임에도 어떻게 실용적인 시간 안에 “충분히 좋은” 해를 찾는지를 설명하지 않는다.
Phi 노드를 설명하는 예제 코드가 있지만, 실제 루프에서 Phi 노드가 어떻게 생성되는지, SSA Construction Algorithm이 어떻게 동작하는지는 없다.

레퍼런스로는 훌륭하다. 하지만 어떤 개념을 진짜 이해하고 싶을 때 이 글 하나로는 부족하고, 글에서 언급한 “Engineering a Compiler”나 LLVM Tutorial(Kaleidoscope)을 직접 따라가야 한다는 점을 글이 명시하면 더 정직할 것이다.

### 실행 시간 데이터의 출처 불명확성

“Chromium: PGO로 6-10% 성능 향상”, “ThinLTO로 바이너리 크기 7% 감소” 같은 수치들이 인용 없이 제시된다.
이 수치들이 어느 버전, 어느 빌드 설정, 어느 벤치마크에서 나왔는지 알 수 없다.
실제로 성능 향상 수치는 워크로드에 따라 크게 달라지며, “Chrome” 전체의 평균인지 특정 마이크로벤치마크인지에 따라 의미가 전혀 다르다.
수치가 너무 깔끔하게 긍정적이어서, 이것이 최선의 경우인지 대표값인지 확인할 방법이 없다.

### Undefined Behavior 섹션의 예제 결함

“UB 최적화” 예제로 제시한 코드가 설명과 정확히 일치하지 않는다.
“`p`가 null이면 return 0하고, 그 후 `*p = 5`를 역참조한다”는 코드에서, 옵티마이저가 “`p`가 null이 아님”을 가정해 `if`를 제거한다고 설명한다.
그러나 `*p = 5`는 `if (!p) return 0;` 이후에 있고, null이면 이미 리턴했으므로 이 흐름에서 `p`가 null이라는 것은 불가능하다. 이것이 UB를 통한 최적화인지, 아니면 단순한 도달 불가능 코드 제거인지가 불명확하게 설명됐다.
실제 LLVM UB 최적화 예제는 `signed integer overflow`, `strict aliasing violation`, `out-of-bounds access` 같은 더 직관적인 사례로 설명하는 것이 적합하다.

## 인사이트

### 중간 표현(IR)은 지식의 분리 경계다

LLVM IR의 역할을 “공통 계약”이라고 부를 수 있지만, 더 정확하게는 “지식의 분리 경계”다.
프론트엔드는 언어의 의미론(semantics)을 안다. 백엔드는 하드웨어의 특성을 안다. 옵티마이저는 둘 다 모른다. 그리고 오직 IR만 알면 된다.
이 분리가 소프트웨어 공학적으로 갖는 의미는 깊다. 각 레이어가 자신의 영역에서만 전문성을 가지면 되고, 경계를 넘어선 복잡한 상호작용을 차단한다.

소프트웨어 아키텍처의 일반 원칙으로 확장하면, 좋은 인터페이스는 두 레이어가 서로에 대해 “얼마나 적게 알아도 되는가”를 결정한다.
GCC의 모놀리식 설계는 언어, 최적화, 코드 생성이 서로의 내부를 아는 구조였다. LLVM의 IR 기반 설계는 각 레이어가 이웃 레이어의 내부를 몰라도 동작하게 만들었다.
“분리된 지식”이 모듈성의 본질이며, LLVM IR은 그 원칙의 성공적인 구현이다.

이것은 컴파일러 설계를 넘어 시스템 설계의 보편적 교훈이다.
마이크로서비스에서 API가 하는 역할, 하드웨어 추상화 계층(HAL)이 하는 역할, 운영체제 시스템 콜 인터페이스가 하는 역할이 모두 같다.
레이어 간의 계약을 명확하게 정의하면, 각 레이어는 독립적으로 발전할 수 있다.

### SSA가 가르쳐주는 “불변 조건 강제”의 위력

SSA의 “한 번만 정의” 규칙은 제약처럼 보이지만, 이 제약이 오히려 분석을 자유롭게 만든다.
제약이 없는 변수는 “어떤 값이든 될 수 있다”는 가능성의 공간을 열어두어 분석기가 모든 경우를 고려해야 한다.
제약이 있는 변수는 가능성 공간이 닫혀 있어 분석기가 “이것은 무조건 이렇다”고 단정 지을 수 있다.

함수형 프로그래밍의 불변성(immutability), Rust의 소유권 시스템, 타입 시스템의 제약이 모두 같은 원리다.
프로그래머에게 표현의 자유를 제한하는 대신, 그 제한 덕분에 컴파일러나 분석 도구가 더 많은 것을 확신할 수 있게 된다.
“제약을 추가하면 가능성이 사라지는 것이 아니라 확실성이 생긴다”는 역설이 SSA에서 가장 잘 구현됐다.

소프트웨어 설계에서 불변 조건(invariant)을 강제하는 것의 가치가 여기 있다.
“이 필드는 초기화 이후 변하지 않는다”, “이 함수는 항상 정렬된 리스트를 반환한다” 같은 불변 조건이 코드 수준에서 강제되면, 그 불변 조건에 의존하는 코드는 방어적인 체크 없이 더 단순해진다.
SSA가 컴파일러 분석을 단순하게 만드는 것처럼, 설계 수준의 불변 조건이 코드를 단순하게 만든다.

### MLIR이 보여주는 “추상화 레벨의 보존”이라는 설계 원칙

MLIR의 Dialect 체계가 해결한 문제는 기술적으로는 “정보 손실”이지만, 설계 원칙으로 보면 “추상화 레벨의 조기 하강” 문제다.
도메인 개념을 너무 일찍 저수준으로 내리면, 그 개념이 가진 구조적 정보가 사라져서 나중에 그 정보를 활용하는 최적화가 불가능해진다.

이 원칙은 소프트웨어 아키텍처의 “조기 추상화” 문제와 대칭이다.
조기 추상화는 도메인 개념을 너무 일찍 일반화해서 도메인 특수성을 잃는 문제다.
MLIR이 보여주는 것은 반대 방향의 함정, 즉 조기 하강(premature lowering)이다. 도메인 개념을 도메인 레벨에서 최적화하기 전에 저수준 IR로 내리면 그 기회를 영원히 잃는다.

“각 레벨에서 그 레벨의 최적화를 완료하고 내려가라”는 MLIR의 Lowering 철학은 컴파일러를 넘어 시스템 설계에 적용된다.
데이터베이스 쿼리 옵티마이저가 물리적 실행 계획을 만들기 전에 논리적 쿼리 레벨에서 최적화하는 것, API 게이트웨이가 개별 마이크로서비스로 라우팅하기 전에 요청 레벨에서 인증/인가를 처리하는 것이 같은 패턴이다.
각 레이어는 자신이 가진 정보를 최대한 활용한 후에야 다음 레이어로 내려야 한다.

### Chris Lattner 한 사람의 영향력이 만든 컴파일러 생태계

LLVM(2000), Clang(Apple 시절), MLIR(Google 시절), Mojo(Modular) 모두 Chris Lattner가 설계를 주도했다.
현대 컴파일러 인프라의 핵심 레이어 여러 개가 한 사람의 일관된 철학으로 설계됐다는 사실은 “플랫폼 설계자” 역할의 영향력을 보여준다.

이것은 오픈소스 생태계에서 “조직”보다 “개인의 비전”이 더 오래 지속되는 사례다.
LLVM을 만든 University of Illinois도, Apple도, Google도 Lattner가 떠난 후에도 그가 설계한 구조 위에서 계속 작업한다.
좋은 아키텍처는 설계자보다 오래 산다. 그리고 그 아키텍처가 생태계 전체를 수렴시킨다.

역설적으로, LLVM의 성공은 후속 언어들의 선택지를 좁혔다.
새 언어가 자체 컴파일러 백엔드를 만드는 것은 LLVM 생태계의 도구, 최적화, 플랫폼 지원 전체를 포기하는 것을 의미한다.
대안적 컴파일러 인프라를 만드는 것이 기술적으로 어려운 것이 아니라, LLVM이 구축한 생태계에서 이탈하는 비용이 너무 크다는 네트워크 효과가 진입 장벽이 됐다.
Cranelift가 Rust의 디버그 빌드라는 좁은 영역에서만 대안으로 자리 잡을 수 있었던 이유도 이 때문이다.
