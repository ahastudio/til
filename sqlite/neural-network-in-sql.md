# SQL로 신경망 구현하기

<https://github.com/xqlsystems/xarray-sql/blob/claude/xarray-sql-mnist-demo/benchmarks/nn.py>

GitHub 저장소: <https://github.com/xqlsystems/xarray-sql>

HN 토론: <https://news.ycombinator.com/item?id=48897975> (78점, 11개 댓글)

## 소개

`xarray-sql`은 과학 데이터에서 흔히 쓰이는 배열(Xarray) 데이터셋을
SQL로 질의할 수 있게 해주는 라이브러리다.
쿼리 엔진으로는 DataFusion을 사용하며,
Xarray의 청크(chunk)를 Arrow의 RecordBatch로 변환하는 방식으로
배열 데이터를 데이터베이스 파티션처럼 다룰 수 있게 만든다.
저장소의 핵심 주장은 기후과학·지리정보 분야에서 흔히 “배열 연산”으로
여겨지는 기후값 계산, 이상치 탐지, 스펙트럼 지수 계산, 재투영 같은
작업들이 사실은 `GROUP BY`, `JOIN`, 윈도 함수, `CASE`로 표현 가능한
관계형 연산이라는 것이다.

이 저장소의 `benchmarks/nn.py`는 그 주장을 극단까지 밀어붙인 데모다.
784→196→32→10 구조의 3계층 신경망을 Fashion-MNIST 데이터셋으로 학습시키는
전체 파이프라인을, 순전파·역전파·경사하강법 갱신까지 포함해서
전부 SQL 쿼리로 표현했다.
데이터는 S3에 있는 Zarr 포맷의 Fashion-MNIST를 xbatcher로 불러오며,
오프라인 테스트를 위한 합성 데이터 경로도 마련되어 있다.
학습은 700개 샘플(70% 학습, 30% 테스트), 학습률 0.5, 60스텝,
250개 단위의 배치 청크로 구성된다.

## 분석

### 행렬곱을 관계형 연산으로 재정의하기

이 데모의 근본적인 아이디어는 행렬곱이 특별한 배열 연산이 아니라
`JOIN` 다음에 오는 그룹별 `SUM`이라는 것이다.
신경망의 순전파에서 `z = x @ W + b`를 계산할 때,
활성화 값 텐서와 가중치 텐서를 공통 차원 기준으로 조인한 뒤
그 차원으로 그룹화해서 곱의 합을 구하면 행렬곱과 동일한 결과가 나온다.
이후 바이어스를 더하는 별도의 조인을 거쳐 활성화 함수(은닉층은 tanh,
출력층은 선형)를 적용한다.
이 관점이 성립하는 이유는 행렬곱 자체가 원래 두 인덱스 집합의 데카르트
곱에서 한 축을 접어(marginalize) 축소하는 연산이기 때문이다.
SQL의 조인-집계 패턴은 정확히 이 구조와 일치한다.
HN 스레드에서 한 댓글은 "처음엔 'SQL로 신경망이라니' 하고 눈을
굴렸지만, 코드를 읽고 나니 인상 깊었다"며, 핵심은 관계형 대수를
중간 표현(IR)으로 써서 데이터베이스 옵티마이저가 텐서 프로그램을
추론하게 만드는 것이라고 정리했다[^0xnyn].
같은 스레드의 다른 댓글은 이 관찰을 수학적으로 더 밀어붙여서,
아인슈타인 합(einsum)과 데이터베이스 조인이 사실 같은 연산이며
차이는 그 연산이 어떤 세미링(semiring) 위에서 도는가뿐이라고
설명했다.
einsum은 실수 위에서, 관계형 조인은 불리언 위에서 도는 동일한
구조라는 것이다[^sporkl].

역전파도 같은 틀로 자연스럽게 확장된다.
출력층의 오차는 소프트맥스와 원핫 인코딩된 정답의 차이로 계산되고,
각 은닉층의 델타는 다음 층의 델타를 가중치의 전치와 곱한 뒤
활성화 함수의 도함수를 곱하는 체인룰로 역방향 전파된다.
그래디언트는 순전파 활성화의 전치와 델타를 곱해 배치 크기로 나눈 값이며,
파라미터 갱신은 `weight - lr * gradient` 형태의 SGD를 모든 레이어에 대해
하나의 통합 쿼리로 표현한다.
결국 신경망 학습에 필요한 네 가지 국면(순전파, 손실 계산, 역전파, 갱신)이
전부 조인과 집계의 조합으로 환원된다는 것을 실제로 동작하는 코드로
보여준 셈이다.

### 희소성을 SQL 쿼리 최적화로 다루기

이 데모에서 흥미로운 지점은 `SKIP_ZERO_PIXELS`라는 플래그다.
Fashion-MNIST 이미지는 배경이 검은색인 픽셀이 많아서 입력 벡터의 약
50%가 0값이다.
행렬곱을 조인-집계로 표현하면, 값이 0인 행을 조인 이전에 걸러내는 것만으로
곱셈 자체를 건너뛸 수 있다.
이는 배열 연산 관점에서는 특별한 스파스 매트릭스 라이브러리가 필요한
최적화지만, 관계형 관점에서는 그냥 `WHERE val != 0` 하나로 끝나는
평범한 술어 푸시다운(predicate pushdown)이다.
저자가 보고한 수치는 스텝당 2.56초에서 1.45초로, 약 1.8배의 속도 향상이다.

이 지점이 이 데모가 단순한 재미로 그치지 않는 이유다.
SQL 엔진은 수십 년간 술어 푸시다운, 조인 순서 최적화, 인덱스 활용 같은
쿼리 최적화 기법을 발전시켜 왔다.
연산을 SQL로 표현하면 이런 최적화 인프라를 신경망 연산에도 공짜로
가져다 쓸 수 있다는 뜻이 되고, 이 데모는 그 잠재력을 희소 입력이라는
아주 구체적인 사례로 증명한다.

### 완결성보다 개념 증명에 무게를 둔 설계

700개 샘플, 60스텝, 3계층 소형 네트워크라는 규모는 이 코드가 실전
학습 파이프라인이 아니라 개념 증명임을 스스로 드러낸다.
실제 딥러닝 프레임워크와 경쟁하려는 목적이 아니라, “이 연산도 SQL로
표현될 수 있는가”라는 질문에 대한 최소 반례를 만드는 데 집중한
설계다.
합성 데이터 폴백 경로를 남겨둔 것도 같은 맥락으로 읽을 수 있다.
S3 접근이 없는 환경에서도 개념이 성립함을 보여주려는 배려이지,
프로덕션 등급의 데이터 파이프라인을 목표로 한 흔적은 아니다.

## 비평

### SQL로 “표현 가능하다”와 “SQL로 하는 것이 낫다”는 다른 문장이다

이 데모가 증명하는 것은 딱 “신경망 연산이 관계형 대수로 인코딩 가능하다”
까지다.
그 이상을 주장하지 않는다는 점은 저장소 저자 스스로도 인정하는 부분이지만,
Show HN류 제목은 종종 그 경계를 흐리게 만든다.
실제로 SQL 엔진에서 신경망을 학습시키는 것이 numpy나 PyTorch보다 빠르거나
간결하다는 근거는 이 코드 어디에도 없다.
오히려 조인 기반 행렬곱은 텐서 연산 전용으로 설계된 BLAS 커널에 비해
구조적으로 불리한 위치에서 출발한다.
가능성의 증명과 실용성의 증명 사이의 거리를 좁히지 않은 채로 두면,
독자는 “SQL이 신경망에 적합하다”는 과장된 결론으로 건너뛰기 쉽다.

### 벤치마크 수치 하나로 일반화하기엔 조건이 너무 좁다

1.8배 속도 향상이라는 수치는 딱 하나의 조건, 즉 입력의 절반이 0인
Fashion-MNIST라는 특수한 데이터셋에서 나온 값이다.
이 최적화는 희소성이 낮은 데이터셋이나, 애초에 0을 걸러낼 축이 뚜렷하지
않은 은닉층 사이의 연산에는 적용되지 않는다.
게다가 비교 대상이 “SKIP_ZERO_PIXELS를 켰을 때와 껐을 때의 같은 SQL
파이프라인”이지, 업계 표준 프레임워크와의 절대적인 속도 비교가 아니다.
즉 이 수치는 SQL 접근법 내부의 상대적 개선을 보여줄 뿐, SQL 접근법
자체의 경쟁력을 뒷받침하는 근거로 쓰기엔 표본이 하나뿐이다.

### 역전파 전체를 쿼리로 욱여넣는 것의 디버깅 비용은 가려져 있다

델타 전파, 그래디언트 계산, 파라미터 갱신을 “하나의 통합 쿼리”로
표현했다는 설명은 코드의 우아함을 강조하지만, 동시에 실무적인 대가를
숨기고 있다.
일반적인 딥러닝 프레임워크는 중간 텐서 값을 손쉽게 찍어보고,
그래디언트 폭주나 소실을 레이어 단위로 추적할 수 있는 디버깅 도구를
갖추고 있다.
반면 여러 조인과 집계가 중첩된 대형 SQL 쿼리에서 특정 레이어의 그래디언트가
왜 이상해졌는지 추적하는 일은 훨씬 번거롭다.
쿼리 플래너가 실행 순서를 재배치하면 어느 시점의 중간값을 봐야 할지조차
불분명해질 수 있다.
이 데모는 “표현할 수 있다”는 가능성에 집중한 나머지, 그 표현 방식이
장기적으로 유지보수 가능한지에 대해서는 침묵한다.

## 인사이트

### 관계형 엔진의 최적화 자산이 새로운 워크로드로 이전될 수 있다

이 데모가 보여주는 가장 실질적인 시사점은, 연산을 SQL로 재구성하는
순간 그 연산이 자동으로 수십 년간 축적된 쿼리 최적화 기법의 적용
대상이 된다는 점이다.
`SKIP_ZERO_PIXELS`가 별도의 스파스 텐서 라이브러리 없이 `WHERE` 절
하나로 구현된 것이 단적인 예다.
이는 조인 순서 최적화, 파티션 프루닝, 비용 기반 플래너 같은 기존
인프라가 앞으로 더 다양한 배열·텐서 워크로드에 자연스럽게 적용될 수
있음을 시사한다.
다만 이 이전이 공짜는 아니다.
관계형 엔진의 최적화기는 텐서 연산의 구조(예: 배치 차원과 특징 차원의
구분)를 알지 못하므로, 신경망에 특화된 최적화(예: 커널 퓨전, 메모리
재사용 패턴)는 오히려 SQL 표현으로 옮기면서 잃을 가능성이 크다.

### 데이터베이스와 텐서 컴퓨팅의 경계가 흐려지는 흐름의 연장선

이 데모는 고립된 실험이 아니라, DuckDB나 다른 분석 엔진들이 배열·텐서
연산을 관계형 질의로 흡수해온 흐름과 같은 방향을 가리킨다.
역사적으로 보면 1970년대 관계형 모델이 처음 등장했을 때도 비슷한
논쟁이 있었다.
계층형·네트워크형 데이터베이스 지지자들은 “모든 걸 테이블과 조인으로
표현하는 건 비효율적”이라고 반박했지만, 결국 관계형 대수의 표현력과
최적화 가능성이 승리했다.
지금 텐서 연산을 SQL로 표현하려는 시도들도 비슷한 질문을 던진다.
텐서 전용 실행 모델(예: XLA, PyTorch의 오토그라드)이 갖는 특화된
효율성과, 관계형 모델이 갖는 범용성·조합 가능성·기존 인프라 재사용
사이에서 어느 쪽이 장기적으로 이길지는 아직 열려 있는 문제다.
이 데모는 그 논쟁에 신경망이라는 아주 구체적인 사례를 하나 추가한
것이다.

사실 관계형 데이터베이스에 머신러닝을 통합하려는 시도 자체는 이번이
처음이 아니다.
HN 댓글 하나는 확장 기능을 통해 SQL 안에 신경망 구현을 넣었던
MADlib 같은 오픈소스 프로젝트와, 2005년 무렵부터 존재했던 SQL
Server의 상용 ML 확장을 언급하면서, 이번 데모가 진짜로 새로운 것은
"관계형 구조의 어떤 특성"이 이 결과를 가능하게 했는지에 대한
설명이라고 지적했다.
같은 댓글은 다차원 배열을 위한 SQL 표준인 ISO/IEC 9075-15:2023(SQL/MDA)
과의 관계도 물었는데, 저장소 저자는 배열을 컬럼 타입으로 다루는
그 표준 방식 대신 Michael Stonebraker의 SciDB처럼 배열 자체를
테이블로 취급하는 접근을 택했다고 답했다[^sixdimensional].
즉 이 데모는 "SQL과 ML을 묶는다"는 아이디어의 재발명이 아니라,
그 오래된 아이디어를 관계형 대수 자체를 실행 모델의 중심에 두는
방식으로 다시 짠 것에 가깝다.

이런 "X가 Y를 실행한다" 계열의 데모가 갖는 재미 요소를 지적한 반응도
있었다.
한 댓글은 이 프로젝트가 신경망 버전의 "X runs Doom" 데모 같다고
평했고, 다른 참가자는 실제로 같은 시기에 올라온, 재귀적 CTE로
레이 트레이싱을 구현한 SQLite 기반 DOOM 데모를 언급하며 화답했다[^simonw].
이런 비교가 시사하는 바는, 이 데모의 사회적 기능이 실무 채택
설득이 아니라 "이 실행 모델의 표현력이 어디까지 뻗어나가는가"를
보여주는 과시적 벤치마크에 가깝다는 점이다.

### “SQL로 할 수 있다”는 증명이 “SQL로 해야 한다”는 처방으로 잘못 읽히는 순간의 위험

Show HN 형태의 개념 증명이 도달하는 흔한 함정은, 커뮤니티가 그
증명을 실제 처방으로 오독하는 순간이다.
이 코드 자체는 소규모 데이터셋에서 원리를 보여주는 데 그치지만,
제목의 임팩트 때문에 “SQL 엔진에서 신경망을 실전 학습시켜야 한다”는
식의 과도한 해석으로 이어지기 쉽다.
저자가 명시한 트레이드오프, 즉 조인 기반 행렬곱이 BLAS 커널보다
구조적으로 느릴 수밖에 없다는 사실은 벤치마크 수치의 화려함에 가려
쉽게 넘어가는 지점이다.
이런 개념 증명을 읽을 때 중요한 건 “이 사람이 정말 이 방식을 실전에
쓰라고 권하는가”를 구분하는 일이다.
이 데모의 진짜 가치는 신경망을 SQL로 대체하라는 제안이 아니라,
관계형 모델의 표현력이 우리가 생각했던 것보다 훨씬 넓다는 것을 보여주는
교육적 사례라는 데 있다.
