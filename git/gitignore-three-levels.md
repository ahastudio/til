# .gitignore 말고도 파일을 무시하는 세 가지 방법

원문: <https://nelson.cloud/.gitignore-isnt-the-only-way-to-ignore-files-in-git/>

HN 토론: <https://news.ycombinator.com/item?id=48583356> (327점, 108개 댓글)

## 요약

Git에서 파일을 무시하는 방법이 `.gitignore` 하나만이 아니라는 것을 설명하는 글이다.
Git은 세 가지 수준에서 파일 무시 규칙을 처리한다.

첫 번째는 모두가 아는 `.gitignore`다.
저장소 루트(또는 하위 디렉터리)에 위치하며 Git으로 커밋되어 팀 전체에 공유된다.
여러 파일로 분산할 수 있고 디렉터리별로 다른 규칙을 적용할 수 있다.

두 번째는 `.git/info/exclude`다.
모든 Git 저장소의 `.git` 디렉터리 안에 존재하지만 Git으로 추적되지 않는다.
팀에게 영향을 주지 않고 개인 워크플로에서만 필요한 파일 — 예를 들어 로컬 메모 파일 —
을 무시할 때 적합하다.
변경 내용이 커밋에 포함되지 않으므로 팀원들의 저장소에는 영향을 주지 않는다.

세 번째는 `~/.config/git/ignore`다.
머신 전체에 적용되는 글로벌 무시 파일로 모든 저장소에서 공통으로 적용된다.
macOS의 `.DS_Store`, Windows의 `Thumbs.db`, 에디터별 설정 파일처럼
운영체제나 개발 환경에 특화된 패턴을 저장하기에 적합하다.

글로벌 파일의 위치는 커스터마이징할 수 있다.
`git config --global core.excludesFile ~/.gitignore_global` 명령으로
원하는 경로를 지정하거나,
`git config --global --unset core.excludesFile`로 기본값으로 되돌릴 수 있다.
어떤 규칙이 특정 파일을 무시하고 있는지 확인하려면
`git check-ignore -v <파일명>`을 사용하면 어느 ignore 파일의 어떤 규칙이 적용되었는지
출력된다.
출력이 없으면 해당 파일은 어떤 메커니즘으로도 무시되지 않고 있다는 의미다.

## 분석

### Git의 무시 파일 계층 구조는 세 가지 독립적 관심사를 분리한다

Git의 세 가지 무시 파일은 각각 다른 범위와 공유 대상을 가진다.
`.gitignore`는 저장소 범위이고 팀과 공유된다.
`.git/info/exclude`는 저장소 범위이지만 개인 전용이다.
`~/.config/git/ignore`는 머신 범위이고 개인 전용이다.

이 세 가지 축 — “어떤 범위인가”와 “누구와 공유하는가” — 이
독립적으로 조합된 결과다.
범위(저장소 vs 머신)와 공유(팀 vs 개인)가 직교(orthogonal)하게 설계되어 있어서,
각 사용 사례에 정확히 맞는 도구를 선택할 수 있다.

이 설계가 실용적으로 중요한 이유는 팀 작업과 개인 워크플로의 충돌을 방지하기 때문이다.
`.DS_Store`를 `.gitignore`에 넣으면 팀 모두에게 적용되는데,
이것이 macOS를 쓰지 않는 팀원에게는 무의미한 항목이다.
글로벌 파일에 넣으면 각자의 환경에서 각자가 처리할 수 있다.
마찬가지로 로컬 메모나 실험적 스크립트를 `.gitignore`에 넣으면 팀의 무시 파일이
개인 작업 흔적으로 오염된다.

kevincox는 이 문제가 실제 현장에서 얼마나 흔한지 증언했다.[^kevincox]
“글로벌/유저 수준 제외 파일은 더 널리 알려져야 할 기능이다.
IDE, OS, AI 도구 파일을 모든 프로젝트의 `.gitignore`에 추가하는 변경사항을 PR로 올리는 사람들을
자주 본다.”
이 증언은 글로벌 파일의 부재가 단순한 편의 문제가 아니라 협업 마찰의 원인임을 보여준다.

### `.git/info/exclude`의 존재가 잘 알려지지 않은 것은 Git 문서화의 구조적 문제를 반영한다

`.git/info/exclude`는 모든 Git 저장소에 기본으로 존재하는 파일이다.
`git init`을 실행하면 자동으로 생성된다.
그런데 이것을 모르는 Git 사용자가 많다.

이유는 Git의 공식 문서가 기능 중심으로 조직되어 있고
“이런 상황에서는 이 파일을 써라”는 시나리오 중심 가이드가 약하기 때문이다.
`git help gitignore`는 세 가지 파일을 모두 언급하지만,
대부분의 입문 튜토리얼은 `.gitignore`만 가르친다.
결과적으로 많은 개발자가 개인 파일을 팀의 `.gitignore`에 넣거나,
더 나쁜 경우 `.gitignore`에 넣지도 못해 실수로 커밋하는 상황이 발생한다.

이것은 더 넓은 패턴을 보여준다.
Git은 매우 강력하지만 그 기능의 상당 부분이 잘 노출되지 않는다.
`git reflog`, `git bisect`, `git worktree`, `git stash apply` vs `git stash pop`의 차이,
그리고 이 세 가지 무시 파일 — 이것들은 모두 “알면 삶이 달라지지만 모르면 몰랐던 기능”이다.
이런 기능들이 입문 자료에서 일관되게 빠지는 것은 Git 학습의 구조적 정보 공백이다.

### `git check-ignore -v`는 디버깅 도구이자 Git 내부 메커니즘의 창이다

`git check-ignore -v <파일>`은 단순히 “이 파일이 무시되는가”를 확인하는 것을 넘어
어느 파일의 어떤 줄에 있는 규칙이 적용되는지를 정확하게 보여준다.
이것은 Git이 무시 규칙을 처리하는 내부 로직을 직접 들여다보는 창이다.

이 명령의 존재는 Git의 설계 원칙을 반영한다.
Git은 “관찰 가능성(observability)”을 중요하게 여기는 시스템이다.
`git log`, `git status`, `git diff`, `git reflog` — 이것들은 모두
Git의 내부 상태를 인간이 이해할 수 있는 형태로 출력하는 도구들이다.
`git check-ignore -v`도 같은 원칙 하에 설계되었다.

실용적으로 이 명령은 “왜 이 파일이 무시되는가” 또는 “왜 이 파일이 무시되지 않는가”를
추적하는 데 필수적이다.
복잡한 `.gitignore` 중첩 규칙, 팀 규칙과 개인 규칙의 충돌,
글로벌 규칙과 저장소 규칙의 우선순위 — 이런 상황에서 `git check-ignore -v`가 없으면
시행착오에 의존해야 한다.

hungryhobbit은 이 글이 다루지 않은 또 다른 파일 처리 메커니즘을 제시했다.[^hungryhobbit]
`.gitattributes`에 `package-lock.json -diff`를 추가하면 파일을 추적하면서도 diff 출력에서 제외할 수 있다.
파일을 “무시”하는 것이 아니라 “추적하되 노이즈를 숨기는” 다른 문제를 해결하는 방법이다.
파일이 변경되었다는 사실 자체는 Git이 기록하지만, `git diff`나 PR 리뷰에서 그 내용이 출력되지 않는다.
이것은 세 가지 무시 파일과는 직교하는 별개의 Git 파일 제어 메커니즘이다.

## 비평

### 글은 세 가지 메커니즘을 나열하지만 우선순위와 충돌 해결을 설명하지 않는다

이 글의 핵심 정보인 세 가지 무시 파일이 동시에 존재할 때 어떤 순서로 적용되는지,
같은 파일에 대해 서로 다른 규칙이 있을 때 무엇이 우선하는지를 다루지 않는다.

Git의 실제 처리 순서는 `.gitignore` → `.git/info/exclude` → 글로벌 파일 순이 아니라
더 복잡하다.
가장 구체적인 경로의 `.gitignore`가 우선하며,
같은 파일에 대해 여러 규칙이 있으면 마지막으로 매칭된 규칙이 적용된다.
부정 패턴(느낌표로 시작하는 `!pattern`)으로 무시 취소도 가능하다.
이 정보 없이 “세 가지 방법이 있다”고만 아는 것은 실제 문제 상황에서 불충분하다.

예를 들어 글로벌 파일에서 `*.log`를 무시하도록 설정했는데,
특정 저장소의 `.gitignore`에서 `!important.log`로 무시 취소를 했다면 어떻게 되는가.
또는 `.git/info/exclude`와 `.gitignore`에 같은 패턴이 있다면 중복인가 충돌인가.
이런 질문에 답할 수 없다면 세 가지 파일을 아는 것의 실용적 가치가 반감된다.

### 개인 무시 파일이 팀 워크플로에 미치는 비가시적 영향이 다루어지지 않는다

`.git/info/exclude`와 글로벌 파일이 “팀에 영향을 주지 않는다”는 설명은
단방향으로만 사실이다.
이 파일들의 규칙이 다른 팀원에게 커밋되지 않는 것은 맞다.
그러나 이 파일들로 인해 발생하는 문제가 팀에 영향을 줄 수 있다.

시나리오: 개발자 A가 `debug-output.txt`를 글로벌 파일에서 무시하도록 설정했다.
나중에 A가 실수로 이 파일을 포함하는 디렉터리를 `git add .`로 스테이징할 때
글로벌 규칙 덕분에 그 파일이 포함되지 않는다.
이것은 좋은 일처럼 보이지만, 개발자 B는 같은 글로벌 설정이 없으므로
`git add .`를 했을 때 그 파일이 포함된다.
같은 저장소에서 같은 명령어가 다른 결과를 낸다는 것은 잠재적 팀 혼란이다.

이것은 “개인 설정은 개인에게만 영향을 준다”는 단순한 모델의 한계를 보여준다.
공유 저장소에서 작업하는 팀은 팀원마다 다른 무시 설정이 있다는 사실이 예상치 못한
행동 차이로 나타날 수 있다는 것을 알아야 한다.

hk1337은 `.git/info/exclude`의 또 다른 실용적 한계를 지적했다.[^hk1337]
이 파일은 저장소를 재클론할 때마다 다시 만들어야 한다.
`.gitignore`가 저장소와 함께 영구히 공유되는 것과 달리,
`.git/info/exclude`는 로컬 `.git` 디렉터리에만 존재하므로
저장소를 새로 클론하거나 CI/CD 환경에서는 존재하지 않는다.
이것은 이 파일이 진정한 “개인 워크플로” 설정이지 “저장소 로컬 설정”이 아님을 의미한다.

wccrawford는 글로벌 파일에 맡기는 대신 `.gitignore`에 넣는 것을 선호하는 반론을 제시했다.[^wccrawford]
“항상 프로젝트의 `.gitignore`에 추가한다 — 무지한 다른 사람이 그것들을 프로젝트에 추가하지 않도록 막기 위해.”
이것은 글로벌 설정이 팀원 각자의 자율에 맡겨지는 방식이 새 팀원이나 자동화 시스템에서
예상치 못한 파일이 커밋되는 위험을 감수한다는 현실적 반론이다.

### 글로벌 무시 파일의 위치 커스터마이징 기능은 유용하지만 복잡성을 숨긴다

`core.excludesFile`을 커스터마이징할 수 있다는 것은 유용한 정보지만,
이것을 사용하는 이유나 시나리오를 설명하지 않는다.
왜 기본 위치(`~/.config/git/ignore`)가 아닌 다른 위치를 원하는가.

실제 사용 사례는 있다.
dotfiles 관리 도구(chezmoi, stow 등)를 사용할 때 글로벌 Git 설정을 특정 경로에서 관리하고 싶을 수 있다.
또는 여러 머신 간 동기화를 위해 Dropbox나 iCloud에 저장하고 싶을 수도 있다.
그러나 이런 설명 없이 커맨드만 제시하는 것은
독자가 “언제 이것이 필요한가”를 스스로 파악해야 한다는 것을 의미한다.

또한 `git config --global --unset core.excludesFile`로 설정을 제거하면
기본 위치로 돌아가는 것인지,
아니면 글로벌 무시 기능 자체가 비활성화되는 것인지도 글에서 명확하지 않다.
실제로는 기본 위치(`~/.config/git/ignore`)로 돌아가지만,
“unset”이라는 표현이 “제거”처럼 읽힐 수 있다.

### 글은 “글로벌”을 머신 전체로 설명하지만 Git의 실제 용어와 다르다

wpollock은 글의 용어 사용에서 중요한 오류를 짚었다.[^wpollock]
“Git에서 '글로벌'은 머신 전체가 아니라 per-user를 의미한다.
머신 전체 설정은 Git에서 '시스템(system)'이라고 불리며 `/etc` 아래에 위치한다.”

이것은 단순한 용어 문제가 아니라 Git이 실제로 네 가지 수준의 무시 메커니즘을 가진다는 것을 뜻한다.
이 글은 세 가지만 다루지만 실제로는 `/etc/gitconfig`나 `/etc/gitignore_global` 같은
시스템 수준의 설정도 존재한다.
이것은 공유 서버나 컨테이너 환경에서 모든 사용자에게 공통으로 적용할 무시 규칙이 필요할 때 관련된다.

글이 `~/.config/git/ignore`를 “머신 전체(machine-wide)”라고 설명하는 것은
Git 공식 문서의 per-user 정의와 다르다.
MacBook을 혼자 쓰는 경우에는 실질적 차이가 없지만,
여러 사용자 계정이 있는 공유 환경에서는 이 구분이 중요해진다.
“글로벌”이 어디까지인지를 정확히 이해해야 예상치 못한 범위 문제를 피할 수 있다.

## 인사이트

### Git의 세 가지 무시 수준은 소프트웨어 도구 설계의 “관심사 분리” 원칙을 실천한다

`.gitignore`, `.git/info/exclude`, 글로벌 파일이라는 세 가지 메커니즘은
각각 “프로젝트의 관심사”, “개인과 프로젝트의 교차 관심사”, “개인의 관심사”를 담당한다.
이 분리는 단순히 Git 내부의 구현 세부사항이 아니라
협업 도구 설계에서 반복적으로 나타나는 원칙이다.

같은 패턴이 다른 도구에서도 나타난다.
VS Code는 `.vscode/settings.json`(저장소 공유), `.vscode/settings.json`의 일부 설정(개인 오버라이드),
사용자 설정(전역)이라는 세 가지 수준의 설정을 가진다.
npm은 `package.json`(저장소 공유), `.npmrc`(저장소 로컬), `~/.npmrc`(전역)을 가진다.
이 패턴의 공통점은 “무엇을 공유하고 무엇을 개인화할지를 선택할 수 있는 권한”을 사용자에게 준다는 것이다.

이 설계의 이차 효과는 팀 협업에서 불필요한 마찰을 줄이는 것이다.
모든 개인 설정이 공유되거나 모든 것이 개인 설정으로만 처리된다면
팀은 항상 합의가 필요하거나 항상 각자가 재설정해야 하는 상황에 놓인다.
세 수준의 분리는 “합의할 것”과 “개인이 결정할 것”의 경계를 도구 수준에서 명시한다.

### 알려지지 않은 Git 기능의 존재는 도구 복잡성이 문서화 전략에 패배하고 있다는 신호다

`.git/info/exclude`가 모든 저장소에 존재하지만 대부분의 개발자가 모른다는 사실은
도구의 기능이 얼마나 풍부해도 사용자가 아는 것만 사용한다는 것을 보여준다.
Git은 1.0부터 이 파일을 지원해 왔고, 공식 문서에도 기술되어 있다.
그러나 입문 튜토리얼의 외부로 나오지 않으면 존재 자체를 모른다.

이것은 개인 블로그 포스트가 공식 문서보다 더 효과적으로 기능을 알린다는 역설을 만든다.
Nelson이 이 블로그 포스트를 쓰기로 결정하고,
그것이 HN 같은 커뮤니티에 공유되어야 비로소 더 많은 개발자가 알게 된다.
공식 Git 문서는 포괄적이지만 검색 가능성과 읽기 흐름에서
시나리오 기반의 개인 블로그에 뒤진다.

이 패턴은 소프트웨어 산업에서 “숨겨진 전문가 지식”이 생성되는 방식을 보여준다.
도구의 기능은 공개되어 있지만 그것을 발견하는 경로는 공개되어 있지 않다.
Stack Overflow 답변, 트위터/X 스레드, 개인 블로그 포스트 — 이것들이
공식 문서가 하지 못하는 “언제 이것을 써야 하는지”를 알려준다.
이것은 기술 지식 전파의 구조적 특성이기도 하다.

### 세 가지 수준의 무시 파일이 있다는 것은 팀 온보딩에서 중요하게 다루어야 할 주제다

새로운 팀원이 합류할 때 전달해야 하는 지식 중에 “우리 팀은 `.DS_Store`를 어떻게 다루는가”가
포함되어야 한다.
팀마다 다르게 처리한다 — 어떤 팀은 `.gitignore`에 넣고, 어떤 팀은 글로벌 파일에 맡긴다.
어떤 선택이 옳은지는 팀의 OS 다양성과 규약에 따라 다르다.

이 지식이 공유되지 않으면 팀원이 실수로 운영체제 특화 파일을 커밋하거나,
팀의 `.gitignore`가 개인 워크플로 항목으로 오염되는 상황이 반복된다.
새로운 팀원이 왜 이 파일들이 있는지, 왜 그 파일들이 `.gitignore`에 있지 않은지를
이해하려면 세 가지 수준의 메커니즘을 알아야 한다.

이것을 팀 온보딩 문서나 `README.md`에 포함하는 것이 실용적 권장사항이다.
“우리 팀은 OS 특화 파일은 개인 글로벌 설정에서 처리합니다.
`.gitignore`는 프로젝트 코드와 직접 관련된 생성 파일만 포함합니다”와 같은
간단한 규약 하나가 수많은 PR 리뷰 댓글을 예방할 수 있다.
Git의 세 수준 무시 파일은 기능의 문제이기도 하지만 팀 규약의 문제이기도 하다.

judofyr는 글로벌 무시 파일을 창의적으로 활용하는 관례를 소개했다.[^judofyr]
`attic`을 글로벌 무시 파일에 추가하면, 어느 프로젝트에서든 `attic/` 디렉터리를 만들어
커밋되지 않을 임시 파일이나 실험적 코드를 안전하게 보관할 수 있다.
“이 디렉터리명을 실제로 사용하는 저장소를 아직 만난 적이 없다.”
글로벌 파일이 단순히 `.DS_Store` 같은 방어적 규칙을 담는 것을 넘어
개인 워크플로를 설계하는 도구가 될 수 있다는 사례다.
