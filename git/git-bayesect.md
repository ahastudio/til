# git-bayesect - 비결정적 버그를 베이지안으로 bisect하기

> 원문: <https://github.com/hauntsaninja/git_bayesect>

## 요약

`git-bayesect`는 flaky test처럼 결과가 확률적으로 변하는 버그를 찾는 데 특화된
베이지안 git bisect 도구다. 기존 `git bisect`가 결정론적 성공/실패를 전제로 하는
것과 달리, git-bayesect는 "어느 커밋에서 실패 확률이 변했는가"를 베이지안 추론으로
좁혀나간다.

엔트로피 최소화를 통한 커밋 선택과 베타-베르누이 켤레 성질을 이용해 미지의 실패율을
처리한다. `pip install git_bayesect`로 설치하며, 커밋 메시지·diff 내용·파일명
패턴을 사전 확률(prior)로 주입해 탐색을 가중할 수 있다.

```bash
git bayesect start --old $COMMIT
git bayesect run python flaky.py   # 자동 탐색
git bayesect status                # 현재 확률 분포 확인
```

## 분석

기존 `git bisect`의 전제는 "특정 커밋 이전은 항상 통과, 이후는 항상 실패"다.
CI에서 간헐적으로만 터지는 버그, 확률이 낮아서 몇 번 실행해야 재현되는 레이스
컨디션, 통계적 회귀 테스트의 임계값 변화 같은 경우에는 이 전제가 성립하지 않는다.

git-bayesect는 각 커밋에 사후 확률 분포를 유지하고, 다음 테스트할 커밋을 엔트로피
최소화 기준으로 선택한다. 즉 결과에 따라 불확실성을 가장 많이 줄이는 커밋을
다음 체크포인트로 고른다.

사전 확률 주입 기능이 실용적이다. `priors_from_filenames`로 의심스러운 파일을
수정한 커밋에 가중치를 주거나, `priors_from_text`로 커밋 메시지에 특정 키워드가
있는 커밋을 우선 탐색할 수 있다. 직관적 의심을 탐색 전략에 인코딩하는 방식이다.

## 비평

베이지안 접근의 전제는 실패가 특정 커밋 이후 **일정하게 더 자주** 발생한다는
것이다. 외부 환경 변화(서드파티 서비스, 인프라 상태, 타임존 의존성)가 원인인
경우처럼 실패율 변화가 커밋 히스토리와 무관하다면 이 도구도 정답을 찾지 못한다.

또한 샘플 수가 적거나 테스트 환경 노이즈가 크면 추정이 흔들린다. `git bayesect run`이
자동으로 반복 실행하지만, 각 커밋당 충분한 샘플을 확보하려면 시간 비용이 크다.

그럼에도 이 도구가 다루는 문제 — 비결정적 버그의 원인 커밋 추적 — 는 실제로 많은
팀이 수작업으로 감당하거나 포기하는 영역이다. 존재 자체로 가치가 있다.

## 인사이트

### 결정론 가정이 무너지는 영역이 늘고 있다

분산 시스템, 비동기 처리, LLM 기반
테스트처럼 결과가 본질적으로 확률적인 컴포넌트가 코드베이스에 늘수록 기존 디버깅
도구의 전제가 깨진다. git-bayesect는 이 흐름의 작은 징표다. 앞으로 디버깅 도구
전반이 확률적 추론, 반복 샘플링, 신뢰 구간 기반 리포트를 기본으로 갖추는 방향으로
발전할 것이다.

### 사전 확률 주입은 인간 직관을 알고리즘에 인코딩하는 인터페이스다

"이 파일이
수정된 커밋이 의심스럽다"는 직관을 `priors_from_filenames`로 탐색 전략에 녹이는
패턴은, AI 에이전트가 인간의 도메인 지식을 힌트로 받아 탐색 공간을 좁히는 방식과
구조적으로 같다. 탐색 도구에 "prior 주입 인터페이스"를 두는 설계 패턴은 더 많은
곳에서 나타날 것이다.

### flaky test는 기술 부채가 아니라 측정 부채다

많은 팀이 flaky test를 "때때로
실패하는 테스트"로 관리하며 재시도로 처리한다. git-bayesect가 전제하는 것은
"어느 시점부터 실패율이 올랐는가"라는 더 정밀한 측정이다. flaky test를 무시하거나
격리하는 대신 실패율 변화 지점을 추적하는 문화가 정착되면, 테스트 신뢰성이 코드
품질 지표로서 다시 의미를 가질 수 있다.
