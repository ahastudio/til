# sem — Git 위에 얹는 함수/클래스 단위 시맨틱 버전 관리

<https://github.com/Ataraxy-Labs/sem>

HN 토론: <https://news.ycombinator.com/item?id=47294924> (95점, 24개 댓글)

## 소개

sem은 Git의 라인 기반 diff를 함수·클래스 같은 코드 엔티티 단위 diff로
바꿔주는 도구다.
"42~67번째 줄이 바뀌었다" 대신 "함수 `authenticateUser`가 수정됐다",
"클래스 `UserManager`가 이름이 바뀌었다"처럼 의미 단위로 변경을
설명한다.
특히 구조적 변경을 정확히 이해해야 하는 AI 코딩 에이전트에게
유용하도록 설계됐다.

## 아키텍처

Rust로 작성된 애플리케이션으로 Git 위에 얹혀 동작한다.

- tree-sitter로 코드를 AST로 파싱
- git2로 저장소 조작
- rayon으로 파일 처리를 병렬화
- xxhash로 구조적 해시를 계산해 이름 변경(rename) 탐지

기본적으로 로컬에서 동작하며 저장소 밖에 SQLite 캐시를 둔다.
대형 모노레포를 위한 선택적 클라우드 가속 옵션도 제공해, 사전
빌드된 의존성 그래프를 서빙할 수 있다.

## 주요 기능

**엔티티 단위 분석**은 TypeScript, Python, Go, Rust, Java 등 32개
이상의 언어에서 함수·클래스·메서드·타입 정의를 전부 추출한다.

**3단계 매칭**으로 정확히 일치, 구조적으로 동등, 유사도 기반 퍼지
매칭을 순서대로 적용해 단순 추가/삭제가 아닌 이름 변경과 이동을
탐지한다.

**의존성 그래프 분석**은 특정 엔티티가 바뀌었을 때 무엇이 깨지는지
보여줘, 변경의 파급 범위(blast radius)를 파악하는 데 쓰인다.

**AI 에이전트 연동**은 Model Context Protocol을 통해 코딩 어시스턴트가
시맨틱 정보를 직접 조회할 수 있게 한다.

## CLI

- `sem diff` — 이름 변경 탐지와 인라인 하이라이트가 포함된
  엔티티 단위 변경 사항
- `sem impact` — 특정 엔티티 변경이 무엇에 영향을 주는지 보여주는
  크로스 파일 의존성 분석
- `sem blame` — 함수나 클래스를 마지막으로 수정한 사람 추적
- `sem log` — 엔티티가 git 히스토리를 거치며 어떻게 변해왔는지 추적
- `sem context` — 토큰 예산에 맞춰 엔티티와 그 의존성을 LLM 컨텍스트에
  담아주는 기능

GitHub Actions와 연동해 PR에 변경된 엔티티를 주석으로 달아주는
기능도 제공한다.

## 라이선스 및 상태

MIT 또는 Apache-2.0 듀얼 라이선스로 배포된다.
2026년 7월 기준 3.2k 스타, 55개 릴리스, 567개 커밋으로 활발히
유지보수되고 있으며, 최신 안정 버전은 0.21.0이다.

## 분석

### 라인 diff의 한계와 엔티티 diff의 필요성

기존 `git diff`는 텍스트 라인을 기준으로 변경을 보여주기 때문에,
함수 내부에서 몇 줄만 바뀌어도 전체 함수가 바뀐 것처럼 보이거나,
반대로 함수가 통째로 이동/이름 변경됐을 때 이를 삭제+추가로
오인하기 쉽다.
sem은 AST 기반 구조적 해시로 이런 문제를 우회한다.
aliveornot이 GeekNews 댓글에서 "이런 게 가능했다고?"라며 놀라움을
표하면서도 "안정적이려나요"라고 안정성에 의문을 표한 것처럼[^aliveornot],
코드 구조를 해석하는 정교한 파싱 계층이 얼마나 다양한 실제
코드베이스(리팩터링 패턴, 언어별 관용구)에서 견고하게 동작하는지가
이 도구의 실질적 가치를 좌우할 것으로 보인다.

HN에서도 비슷한 지적이 나왔다.
mellosouls는 sem이 함수 이름과 그 변화를 텍스트 처리로 매핑할
뿐이라 "시맨틱"이라는 이름이 다소 과장됐다고 지적하면서, 저장소를
열어보기 전까지는 LLM을 이용해 의미론적 설명을 뽑아내는 도구일
것으로 예상했다고 밝혔다[^mellosouls].
그는 AI 코딩이 확산될수록 diff 자체가 AI로 한층 더 추상화될
것이라고 내다봤다.
7777777phil은 "사람이 모든 줄을 직접 쓸 때는 라인 단위 diff로
충분했지만, 에이전트가 세션당 수백 건씩 커밋하는 지금은 어떤 줄이
아니라 어떤 함수가 바뀌었는지 알아야 한다"며 sem의 존재 이유를
AI 코딩 시대의 필연으로 짚었다[^phil7777777].

경쟁 도구를 둘러싼 반응도 있었다.
gritzko는 "sem이 지난주엔 'graft', 그 전주엔 'got'이라는 이름이
아니었냐"며 비슷한 컨셉의 프로젝트가 이름만 바꿔 반복적으로
등장하는 데 냉소적인 반응을 보였고, 자신이 만든 실험적 AST 버전
관리 시스템 Beagle을 대안으로 소개했다[^gritzko].
반면 shubhamintech는 이 문제가 코드 버전 관리에 국한되지 않는다고
짚었다.
그는 AI 에이전트 관측성(observability) 분야에서도 서로 다른 대화
흐름이 동일한 결과를 내거나, 반대로 비슷해 보이는 흐름이 다른
결과를 내는 경우가 있다며, "행동이 실제로 바뀌었는가"를 포착하는
정규화 단계가 두 분야 모두에서 가장 어려운 부분이라고
평가했다[^shubhamintech].

### AI 에이전트 워크플로와의 결합

`sem context`처럼 토큰 예산에 맞춰 관련 엔티티만 골라 담는 기능은
LLM 컨텍스트 윈도우를 절약하려는 최근 도구들의 흐름과 맞닿아 있다.
`sem impact`로 파급 범위를 파악하는 것도, 에이전트가 코드를 수정하기
전에 어디까지 건드려도 안전한지 스스로 판단하게 하려는 목적으로
읽힌다.
즉 sem은 사람을 위한 코드 리뷰 도구라기보다 에이전트가 코드베이스를
구조적으로 이해하고 안전하게 변경하도록 돕는 인프라에 가깝다.

## 인사이트

라인 기반 버전 관리가 코드를 "텍스트"로 다뤄왔다면, sem은 코드를
"구조를 가진 엔티티의 집합"으로 다룬다.
이 전환은 사람보다 AI 에이전트가 코드베이스를 대량으로, 반복적으로
읽고 수정하는 시대에 더 절실해진 문제다.
사람은 함수 이름과 파일 위치로 어느 정도 맥락을 유추할 수 있지만,
에이전트는 명시적인 구조 정보 없이는 "이 변경이 무엇을 의미하는지"를
매번 라인 단위로 재해석해야 한다.
다만 aliveornot의 우려처럼, 이런 도구가 실제로 신뢰받으려면 다양한
리팩터링 시나리오에서의 정확도와 대규모 저장소에서의 성능이 함께
검증되어야 할 것이다.
