# 코드를 읽기 전에 실행하는 Git 명령들

원문: <https://piechowski.io/post/git-commands-before-reading-code/>

## 요약

Ally Piechowski가 코드베이스의 건강 상태를 커밋 이력으로 진단하는 다섯 가지 Git
명령을 소개한다. 코드를 한 줄도 읽지 않고 몇 분 만에 프로젝트의 위험 지점, 팀
구조, 개발 추세를 파악할 수 있다.

1. **가장 많이 변경된 파일**: 1년간 변경 횟수 상위 20개 파일을 추출한다. "목록
   맨 위 파일은 거의 항상 사람들이 경고하는 파일이다."
2. **기여자 분석**: 커밋 수 기준 기여자 순위로 버스 팩터를 평가한다. 한 사람이
   60% 이상을 차지하면 위험 신호다.
3. **버그 클러스터링**: fix, bug, broken 등의 패턴으로 반복 수정된 파일을
   찾는다. 변경 빈도와 버그 빈도가 모두 높은 파일이 최고 위험 코드다.
4. **프로젝트 모멘텀**: 월별 커밋 수 추이로 개발 속도를 시각화한다. 급격한
   하락은 팀 이탈과 상관관계가 있다.
5. **소방 빈도**: revert, hotfix, emergency, rollback 관련 커밋을 카운트한다.
   배포 신뢰성 문제와 프로세스 불안을 드러낸다.

## 분석

### 코드 고고학의 체계화

이 접근은 Adam Tornhill의 "Your Code as a Crime Scene"에서 제시한 코드 포렌식
개념을 실무자가 즉시 쓸 수 있는 원라이너로 응축한 것이다. Tornhill은 변경 빈도와
복잡도의 교차점이 기술 부채의 핵심이라 주장했고, 이 글의 1번(변경 빈도)과
3번(버그 집중) 명령이 정확히 그 교차점을 잡아낸다.

### squash-merge 워크플로와의 충돌

[GeekNews 댓글](https://news.hada.io/topic?id=28324)에서도 지적되었듯이
squash-merge를 쓰는 팀에서는 이 명령들의 정확도가 크게 떨어진다. 한 PR에 수십 개
파일 변경이 포함되어도 커밋 하나로 압축되므로 파일별 변경 빈도가 왜곡되고 기여자
분포도 실제와 달라진다. 이 한계를 인식하지 않으면 진단 자체가 오도할 수 있다.

### 커밋 메시지 품질 의존성

3번과 5번 명령은 커밋 메시지의 grep에 의존한다. "fix typo"와 "fix critical race
condition"을 동일하게 카운트하므로 팀의 커밋 메시지 관행이 결과의 신호 대
잡음비를 결정한다. 커밋 메시지가 체계적이지 않은 프로젝트에서는 오탐률이
높아진다.

## 비평

### 정량 지표의 맹점

안정적인 프로젝트는 커밋 빈도가 낮고 변경이 적다. 이 명령들만으로는 "건강하게
안정된 프로젝트"와 "방치된 프로젝트"를 구분할 수 없다. 4번 명령의 커밋 수 감소가
팀 이탈인지 성숙인지는 맥락 없이 판단할 수 없다.

### 자동화 파이프라인의 왜곡

CI/CD 봇의 자동 커밋, dependabot의 의존성 업데이트, 코드 생성기 출력 등이 통계를
왜곡한다. 특히 기여자 분석에서 봇 계정이 상위에 오르면 실제 인간 기여자 비율이
가려진다. --no-merges 옵션만으로는 이 문제를 해결할 수 없다.

### 방법론의 가치

이런 한계에도 불구하고 이 접근의 진짜 가치는 "코드를 읽기 전에 어디를 읽어야
하는지 안다"는 점이다. 완벽한 진단 도구가 아니라 방향 설정 도구로 이해해야 한다.
10분 투자로 수십 시간의 코드 리딩 방향을 잡는다면 정확도의 한계는 충분히 수용
가능하다.

## 인사이트

### Git 이력은 코드베이스의 열화상 카메라다

건물의 열화상 카메라가 벽 뒤의 단열 결함을 보여주듯, Git 이력 분석은 코드 표면
아래의 구조적 문제를 드러낸다. 중요한 것은 열화상이 건물 구조 자체를 보여주지
않듯 Git 통계도 코드의 품질을 직접 측정하지 않는다는 점이다. "이 벽 뒤에 뭔가
있다"는 신호를 주는 것이지 "이 벽이 무너질 것이다"를 알려주는 것이 아니다. 이
구분을 놓치면 통계적 신호를 인과적 결론으로 오독하게 된다.

### 버스 팩터 진단은 조직 문제의 기술적 프록시다

2번 명령(기여자 분석)이 드러내는 것은 코드 품질이 아니라 조직 구조다. 한 사람이
60%를 차지한다는 것은 코드 리뷰 문화, 온보딩 프로세스, 지식 공유 관행의 부재를
동시에 암시한다. 이 진단은 기술 부채가 아니라 조직 부채를 가리킨다. 기술적
도구로 조직 문제를 발견하는 이 역전이 오히려 경영진에게 설득력 있는 근거를
제공한다. "코드가 위험합니다"보다 "이 사람이 퇴사하면 이 시스템 전체가
멈춥니다"가 의사결정자에게 훨씬 강력한 메시지다.

### 소방 빈도는 배포 파이프라인의 심리적 안전도를 측정한다

5번 명령의 hotfix/rollback 빈도는 단순한 기술 지표가 아니다. 소방이 잦은 팀은
배포를 두려워하고, 배포를 두려워하는 팀은 변경을 미루며, 변경을 미루는 팀은 한
번에 큰 변경을 배포한다. 이 악순환이 다시 소방 빈도를 높인다. revert 카운트
하나에 팀의 심리적 안전, 테스트 커버리지, 배포 자동화 수준, 롤백 절차의 성숙도가
모두 응축되어 있다.
