# deff

<https://github.com/flamestro/deff>

HN 토론: <https://news.ycombinator.com/item?id=47169518> (120점, 67개 댓글)

## 소개

`deff`는 Rust로 작성된 터미널 UI(TUI) 도구로, git diff를 나란히(side-by-side) 배치해 파일 단위로 인터랙티브하게 리뷰할 수 있다.
파일 간 이동, 수직·수평 독립 스크롤, 신택스 하이라이팅, 추가/삭제 라인 색조 표시를 제공한다.
`cargo install deff`로 설치하며, 문법 정의를 바이너리에 번들링해 별도 설정 파일 없이 즉시 사용할 수 있다.

## CLI

```bash
deff                                          # 기본: 로컬 브랜치 vs 업스트림 비교
deff --strategy upstream-ahead                # 기본 전략 명시
deff --strategy range --base origin/main --head HEAD
deff --strategy range --base origin/main --include-uncommitted
deff --only-uncommitted                       # 작업 트리 + 미추적 파일 vs HEAD
deff --theme dark|light|auto
```

## 주요 기능

**비교 전략**

| 전략                | 설명                                          |
| ------------------- | --------------------------------------------- |
| `upstream-ahead`    | 로컬 브랜치 커밋을 업스트림과 비교 (기본값)   |
| `range`             | `--base` / `--head`로 명시적 범위 지정        |
| `--include-uncommitted` | 작업 트리와 미추적 파일까지 포함          |
| `--only-uncommitted`    | 작업 트리만 HEAD와 비교                   |

**탐색**

- Vim 키(`h`/`j`/`k`/`l`, `g`/`G`, `Ctrl+u`/`Ctrl+d`)와 마우스 휠 모두 지원
- 양쪽 패널이 독립적인 수평 스크롤 오프셋을 유지
- `/`로 현재 파일 내 검색, `n`/`N`으로 매치 이동

**리뷰 상태 추적**

- `r`로 파일을 리뷰 완료/미완료로 토글
- 상태는 `.git/deff/reviewed/`에 저장되며, 비교 범위와 파일 내용 해시를 키로 사용
- 저장소를 다시 열어도 리뷰 상태가 유지됨

**신택스 하이라이팅**

- `syntect` 기반으로 언어 인식 하이라이팅 제공
- Sublime Text `.sublime-syntax` 파일을 `assets/syntaxes/` 또는 `.deff/syntaxes/`에 추가해 확장 가능
- 빌드 시 번들링되므로 런타임 파일 의존성 없음

## 분석

### `git diff` 출력의 인지 부하 문제를 정면으로 다룬다

터미널에서 `git diff`를 그대로 보는 것은 컨텍스트가 부족하다.
unified diff 형식은 좌우 비교 없이 `+`/`-` 접두사로만 변경을 표현하므로, 큰 변경에서 원래 코드와 새 코드를 동시에 추적하기 어렵다.
`deff`는 side-by-side 레이아웃으로 이 문제를 해결한다.
두 패널을 독립적으로 스크롤할 수 있다는 점은 특히 가로로 긴 코드 라인에서 유용하다.

### `.git` 디렉토리를 상태 저장소로 활용하는 설계

리뷰 상태를 `.git/deff/reviewed/`에 저장하는 선택은 흥미롭다.
gitignore에 자동으로 포함되지 않아 실수로 커밋될 위험이 없고, 저장소마다 독립적으로 상태를 유지한다.
파일 내용 해시를 키로 사용해 파일이 변경되면 리뷰 상태가 자동으로 초기화된다.
이 방식은 외부 상태 저장 없이 저장소에 종속된 지속성을 구현하는 실용적 패턴이다.

### 커밋 접두사 기반 자동 버전 관리

`feat:` 커밋은 마이너 버전을, `fix:`/`chore:`/`docs:` 커밋은 패치 버전을 올리는 GitHub Actions 워크플로우가 내장돼 있다.
Conventional Commits 규약을 버전 관리 자동화에 연결하는 이 패턴은 소규모 오픈소스 프로젝트의 릴리스 오버헤드를 크게 줄인다.

## 비평

`deff`는 명확한 문제를 해결하는 도구지만, 경쟁 도구들과 비교해 차별화 지점을 명시하지 않는다.
`delta`(side-by-side diff 출력), `difftastic`(구문 인식 diff), `lazygit`(TUI git 클라이언트)가 이미 유사한 영역을 커버한다.
`deff`가 이들과 어떤 트레이드오프를 선택했는지 README에서 설명하면 사용자가 도구를 선택하기 더 쉬울 것이다.

파일 리뷰 상태 추적은 유용하지만, 팀 단위 리뷰에는 적합하지 않다.
`.git/deff/reviewed/`는 로컬 상태이므로 다른 개발자와 공유되지 않는다.
GitHub PR 리뷰, Gerrit, Phabricator 같은 협업 코드 리뷰 도구의 대체재가 아니라 개인 리뷰 보조 도구로 포지셔닝이 명확하다.

신택스 번들링은 편의를 주지만 비용도 있다.
문법 파일이 바이너리에 포함되면 바이너리 크기가 커지고, 새 언어 지원을 위해 재빌드가 필요하다.
반면 런타임 의존성 없이 `cargo install`로 즉시 사용할 수 있다는 장점은 개발자 경험 측면에서 큰 가치다.

HN 스레드에서는 새 도구 채택의 마찰을 다루는 현실적 관점이 제기됐다.
rileymichael은 git이 외부 pager를 지정할 수 있어 `delta` 같은 대안을 기존 git 프론트엔드에 연결할 수 있다고 지적했다.[^rileymichael]
사용자가 이미 익숙한 워크플로우를 유지하면서 시각화만 교체할 수 있다는 것은 `deff`처럼 전용 TUI 진입이 필요한 도구에 비해 채택 문턱이 낮다.

AI 에이전트 워크플로우에서 `deff` 같은 도구가 채울 수 있는 역할에 대한 수요도 관찰됐다.
llbbdd는 에이전트 기반 개발에서 diff의 특정 라인 범위에 타깃 피드백을 달아 에이전트에게 전달하는 기능이 필요하다고 언급했다.[^llbbdd]
VSCode는 diff 뷰에서 라인 범위 코멘트를 제공하지만 터미널 기반으로는 이 기능이 부재하다.
`deff`가 이 방향으로 발전할 수 있는지는 미정이지만, AI 에이전트 워크플로우와 터미널 diff 검토의 결합이라는 새로운 사용 시나리오가 생겨나고 있다는 신호다.

## 인사이트

### 터미널 도구는 마찰이 낮을수록 실제로 쓰인다

코드 리뷰 도구의 역설은 리뷰가 필요한 상황에서 리뷰 도구를 설정하기가 귀찮다는 것이다.
GitHub PR 리뷰는 브라우저를 열고 로그인해야 한다.
`git diff`는 언제나 즉시 쓸 수 있지만 가독성이 낮다.
`deff`가 노리는 위치는 이 두 극단 사이다. 터미널에 이미 있는 개발자에게 `deff` 하나로 구조화된 리뷰 환경을 제공한다.

이 마찰 감소 철학은 도구 설계의 핵심 원칙으로 작동한다.
`cargo install deff`로 시작하고, 저장소에서 `deff`를 실행하면 즉시 동작한다.
테마, 신택스 파일, 외부 의존성 없이 바이너리 하나가 전부다.
마찰이 낮은 도구는 습관적으로 사용되고, 습관적으로 사용되는 도구는 실제 품질을 높인다.

### `.git` 디렉토리는 저장소 로컬 상태의 자연스러운 저장소다

개발 도구들은 대부분 사용자 홈 디렉토리나 별도 설정 파일에 상태를 저장한다.
`deff`의 `.git/deff/reviewed/` 선택은 다른 사고방식을 보여준다.
이 상태는 저장소에 종속된 것이므로, 저장소의 관리 공간인 `.git/` 안에 두는 것이 논리적으로 일관성 있다.

이 패턴은 다른 도구들에도 적용할 수 있다.
저장소별 설정, 임시 분석 결과, 에이전트 핸드오프 문서처럼 “이 저장소에만 유효한 상태”는 `.git/` 안에 저장하는 것이 자연스럽다.
gitignore에 자동으로 포함되고, 저장소 삭제 시 함께 사라지며, 팀원과 공유되지 않는다.
상태의 수명과 범위가 저장소와 정확히 일치한다.

### Rust TUI 생태계가 개발자 도구의 새로운 기준을 세우고 있다

`ratatui`, `crossterm`을 기반으로 한 Rust TUI 애플리케이션이 빠르게 늘고 있다.
`deff`, `gitui`, `bottom`, `zellij`처럼 Rust로 작성된 터미널 도구들은 공통된 특성을 갖는다.
낮은 메모리 사용량, 빠른 시작 시간, 단일 바이너리 배포, 크로스 플랫폼 지원.
이 특성들은 개발자 도구가 실제로 쓰이기 위한 조건과 정확히 일치한다.

Python이나 Node.js로 작성된 CLI 도구는 런타임 의존성 관리, 느린 시작 시간, 버전 충돌이라는 만성적 문제를 안고 있다.
Rust는 이 문제들을 언어 레벨에서 해결한다.
신택스 문법을 바이너리에 번들링하고, 외부 의존성 없이 배포하는 `deff`의 접근 방식은 Rust 생태계의 철학을 그대로 반영한다.
개발자 도구의 배포 단순성이 사용자 채택률에 직접 영향을 미친다는 교훈은 다른 언어로 도구를 만드는 개발자들도 참고할 만하다.
