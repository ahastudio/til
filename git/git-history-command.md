# `git history`, `jj`의 대안이 될 수 있을까

원문: [The git history command deserves more attention - Lalit Maganti](https://lalitm.com/post/git-history/)

HN 토론: <https://news.ycombinator.com/item?id=48901010> (194점, 113개 댓글)

Lobste.rs 토론: <https://lobste.rs/s/tb3el5/git_history_command_deserves_more>

## 요약

저자 Lalit Maganti는 병렬로 여러 변경사항을 다루는 Git 작업이 고통스럽다고
말한다.
브랜치와 커밋을 동시에 관리하다 보면 `rebase -i`처럼 위험한 명령을
자주 쓰게 되고, 조금만 실수해도 트리가 반쯤 망가진 상태에 빠진다.
이런 문제의 대안으로 `jj`(Jujutsu)가 자주 거론되지만,
저자는 지난 1년 반 동안 3개월에 한 번씩 `jj`를 워크플로에 넣어보려
시도했고, 매번 며칠 쓰다가 다시 Git으로 돌아왔다고 고백한다.

이 글은 그 대안으로 Git 2.54와 2.55에 걸쳐 실험적으로 추가된
`git history` 명령어를 조명한다.
`reword`와 `split`은 2.54(4월)에, `fixup`은 2.55(6월)에 추가되었는데
비교적 적은 관심 속에 지나갔다는 것이 저자의 문제의식이다.
그는 이 명령어가 `jj`가 각광받는 이유의 상당 부분을
워크플로 전체를 갈아엎지 않고도 이미 제공한다고 주장한다.

세 서브커맨드는 각각 다른 역할을 한다.
`git history fixup <commit>`은 스테이징된 변경사항을 오래된 커밋에
접어 넣고, 그 커밋에서 파생된 모든 로컬 브랜치를 자동으로 리베이스한다.
`git rebase --update-refs`와 달리 현재 체크아웃된 범위를 넘어
그 커밋을 포함하는 모든 로컬 브랜치를 찾아 재작성한다는 점이 다르다.
`git history reword <commit>`은 커밋 메시지만 수정하고 그 위에 쌓인
커밋들을 재구성하는데, 작업 트리나 인덱스를 건드리지 않고 순수하게
커밋 그래프 위에서만 동작하기 때문에 체크아웃하지 않은 브랜치도
수정할 수 있다.
`git history split <commit>`은 `git add -p`와 비슷하게 헝크 단위로
선택해 하나의 커밋을 둘로 나눈다.

세 명령어가 공유하는 핵심 속성은 원자성이다.
충돌을 유발할 수 있는 연산은 아예 거부함으로써 트리가 절대
반쯤 깨진 상태로 남지 않는다.
다만 저자는 이 도구가 `jj`의 모든 것을 대체하지는 못한다는 점도
분명히 한다.
병합 커밋은 다룰 수 없고, `jj`처럼 충돌을 1급 상태로 취급해
리베이스 중에도 들고 다니는 기능이나, 되돌리기가 가능한 연산
로그, 작업 복사본을 커밋으로 모델링하는 방식은 아직 없다.
그럼에도 저자는 이것이 Git 코어 배포판에 이미 들어와 있는
“큰 진전”이라며, 앞으로의 개선을 기대하는 것으로 글을 맺는다.

## 분석

### `jj`를 설득 도구가 아니라 요구사항 목록으로 읽는다

이 글의 흥미로운 지점은 `jj` 자체를 다루는 글이 아니라는 것이다.
저자는 `jj`를 1년 반 동안 반복해서 시도하고 포기한 사람으로서,
`jj`가 매력적인 이유를 몇 가지 구체적인 기능으로 분해한다.
브랜치 자동 전파, 메타데이터만 건드리는 리워드, 원자적 실패 처리 같은
것들이다.
그렇게 분해하고 나면 “`jj`로 갈아타야 하는가”라는 질문이
“Git이 이 기능들을 개별적으로 제공할 수 있는가”라는 질문으로
바뀐다.

이 재구성은 도구 선택 논쟁에서 흔히 생략되는 단계를 드러낸다.
새 도구를 전면 도입할지 말지를 묻기 전에,
새 도구가 매력적인 이유를 구성 요소로 쪼개는 작업이 먼저 필요하다.
`git history`는 그 구성 요소 중 상당수를 기존 Git 위에
증분적으로 얹은 결과물이라 할 수 있다.

### `fixup`이 메꾸는 빈 자리

Git 2.54에서 먼저 등장한 `reword`와 `split`은
`git rebase -i`가 “할 수는 있지만 번거로운” 작업을 단일 명령으로
압축한 것이었다.
`fixup`은 여기에 브랜치 전파라는 새로운 축을 더한다.
`git commit --fixup`과 오토스쿼시 리베이스를 조합해도 되지만,
그 조합은 현재 체크아웃된 브랜치 하나만 갱신한다.
여러 기능 브랜치가 같은 조상 커밋을 공유하는 상황에서 `fixup`은
그 조상에서 파생된 모든 로컬 브랜치를 찾아 함께 재작성한다.

이것은 “병렬로 여러 변경사항을 다루는 고통”이라는 글 서두의 문제와
정확히 맞아떨어진다.
스택형 브랜치 워크플로에서 아래쪽 커밋을 고치면 위쪽 브랜치들이
전부 뒤틀리는 문제는 오랫동안 수작업 리베이스로 해결해왔다.
`fixup`은 그 수작업을 명령어 하나로 대체하려는 시도다.

### 원자성이라는 이름의 범위 제한

세 명령어가 공유하는 원자성은 사실 “충돌 가능성이 있으면 아무것도
하지 않는다”는 매우 보수적인 전략이다.
이는 안전하지만 동시에 적용 범위를 스스로 좁힌다.
병합 커밋을 다루지 못하고, 충돌이 예상되는 리베이스는 애초에
시작조차 하지 않는다.
저자도 이 한계를 인정하면서 문서에 나온 표현을 인용한다.
Git이 1급 충돌 개념을 도입하면 이 제약이 풀릴 수 있다는 것이다.

이 설계는 새로운 명령어를 만들 때 흔히 마주치는 트레이드오프를
보여준다.
기존 명령어(`rebase -i`)가 표현력은 높지만 위험한 자리를
차지하고 있을 때, 새 명령어는 표현력을 낮추는 대신 안전을
얻는 쪽을 택할 수 있다.
`git history`는 그 선택을 명시적으로 한 사례다.
HN에서 nine_k도 이를 정확히 요약했는데,
`rebase --interactive`의 가장 흔한 세 가지 사용 패턴을
저마찰(low-friction) 명령어로 분리한 것일 뿐이며
충돌이 없을 때만 동작한다는 제약이 핵심이라고 짚었다[^nine_k].

다만 이 원자성에는 구멍이 하나 있다.
`chandlerswift`는 서명된 커밋에 `git history reword`를 실행하면
GPG 서명이 조용히 사라진다는 것을 직접 재현해 보여줬다.
`git log --show-signature`로 서명을 확인한 뒤 `reword`를 실행하면
재작성된 커밋에는 서명이 붙지 않는다는 것이다.
그는 이 때문에 서명을 유지해야 하는 워크플로에서는 결국
`rebase -i`로 돌아갈 수밖에 없었다고 말했다[^chandlerswift].
“충돌만 없으면 안전하다”는 전제는 데이터 무결성의 다른 축인
서명 보존까지 포함하지는 못한 셈이다.

`fixup`이 브랜치 자동 전파에서 정확히 어디까지 갱신하는지도
HN에서 논쟁거리가 됐다.
`shepmaster`는 원문의 브랜치 다이어그램을 구체적인 예시로
재구성하며, `rebase --update-refs`는 현재 리베이스 범위 밖에
있는 브랜치(`E`)는 건드리지 않는 반면 `git history fixup`은
그 브랜치까지 갱신하는지 질문했고,
동일한 동작을 기존 `git rebase`만으로 재현할 방법이 있는지
궁금해했다[^shepmaster].
이 질문에 명확한 답은 달리지 않았는데,
이는 “모든 로컬 브랜치를 찾아 재작성한다”는 문서 표현이
실제 그래프 위상에서 정확히 어떤 경계를 갖는지가
문서만으로는 완전히 드러나지 않는다는 뜻이기도 하다.

## 비평

### “적은 관심 속에 지나갔다”는 진단은 근거가 얇다

저자는 `git history`가 별다른 주목을 받지 못하고 지나갔다고
말하지만, 이 진단을 뒷받침하는 구체적인 근거(다운로드 수,
소셜 미디어 언급량, 이슈 트래커 활동 등)는 제시하지 않는다.
실험적 명령어가 릴리스 노트 한 줄로 소개되고 넘어가는 일은
흔하다.
이것이 유독 저평가되었다는 주장은 저자 개인의 관찰에 기반한
인상에 가깝고, 다른 실험적 기능과 비교한 근거는 없다.

### `jj`와의 비교가 기능 목록에 머문다

글은 `jj`가 제공하는 기능을 나열하고 `git history`가 그중
일부를 제공한다고 정리하지만, 두 도구의 근본적인 설계 차이는
깊이 다루지 않는다.
`jj`는 작업 복사본 자체를 커밋으로 모델링하고 충돌을 1급
상태로 다루는 반면, `git history`는 기존 Git의 커밋-워킹트리
분리 모델 위에서 안전한 연산만 추가한 것이다.
이 차이는 단순한 기능 격차가 아니라 데이터 모델의 격차이므로,
`git history`가 아무리 발전해도 `jj`와 동일한 경험에
도달하기는 구조적으로 어려울 수 있다.
글은 이 구조적 차이를 인정하면서도 “격차를 완전히 좁히지는
못한다”는 정도로만 짚고 넘어간다.

### 개인의 반복된 실패가 도구 비교의 근거로 충분한가

저자는 `jj`를 1년 반 동안 여러 번 시도했다가 포기한 경험을
핵심 근거로 제시한다.
이 경험 자체는 진솔하지만, 왜 매번 실패했는지에 대한 구체적인
분석(어떤 워크플로가 안 맞았는지, 어떤 명령어가 낯설었는지)은
제공되지 않는다.
개인의 습관이나 팀의 관행이 원인이었을 가능성과, `jj` 자체의
설계 문제였을 가능성을 구분하지 않은 채 “`git history`가 더
낫다”는 결론으로 넘어가는 것은 성급해 보인다.

### “히스토리를 깔끔하게 관리해야 한다”는 전제 자체가 논쟁적이다

글은 히스토리를 정교하게 다듬는 작업이 당연히 가치 있다는
전제 위에 서 있지만, HN에서는 이 전제부터 반박하는 목소리가
적지 않았다.
`paxys`는 개별 커밋을 나중에 다시 읽는 사람은 거의 없으니
병합 전에 그냥 스쿼시해버리면 그만이라고 주장했다[^paxys].
`hahahaa`는 회계사가 장부를 고치지 않고 새 분개(journal
entry)를 추가하듯, 히스토리를 편집하지 말고 고치는 커밋을
새로 쌓아야 한다는 정반대 원칙을 제시했다[^hahahaa].
`krick`은 한 걸음 더 나아가 “`rebase -i`가 트리를 반쯤
망가뜨린다”는 서두의 묘사 자체를 과장이라 부르며,
자신은 브랜치 작업 중 수십 번씩 `rebase -i`를 돌려도
문제가 난 적이 없다고 반박했다[^krick].
이 반응들은 글이 암묵적으로 전제한 “정교한 히스토리 관리는
좋은 것”이라는 가치판단이 보편적 합의가 아니라
하나의 워크플로 취향에 가깝다는 것을 보여준다.

## 인사이트

### 실험적 기능은 경쟁 도구의 압력 아래에서 더 빨리 다듬어진다

`git history`가 2.54와 2.55 두 릴리스에 걸쳐 빠르게 서브커맨드를
추가한 것은 우연이 아닐 가능성이 높다.
`jj`라는 대안이 실제로 사용자를 흡수하고 있다는 압력이
Git 코어 팀의 우선순위에 영향을 줬다고 보는 것이 자연스럽다.
이는 오픈소스 생태계에서 반복적으로 관찰되는 패턴이다.
브라우저 엔진 경쟁이 웹 표준의 발전 속도를 끌어올렸던 것처럼,
경쟁하는 대안의 존재는 기존 도구의 개선 속도를 높이는
2차 효과를 낸다.
사용자 입장에서는 `jj`로 갈아타지 않아도 `jj`의 존재 자체가
Git을 더 낫게 만드는 셈이다.

### 표준 도구가 틈새 기능을 흡수하는 오래된 패턴

`git history`가 `jj`의 기능 일부를 흡수하는 과정은
새로운 현상이 아니다.
`git worktree`, `git switch`, `git restore`, `git sparse-checkout`
모두 서드파티 스크립트나 워크플로로 먼저 검증된 뒤
Git 코어에 흡수된 기능들이다.
더 오래된 사례로는 유닉스 셸 도구들이 여러 서드파티 유틸리티의
기능을 표준 유틸리티(`grep`, `sed`, `awk`)로 흡수해 온 역사도
있다.
표준 도구는 처음부터 완결된 설계를 제시하기보다,
주변 생태계에서 검증된 아이디어를 관찰한 뒤 보수적으로
받아들이는 방식으로 진화하는 경우가 많다.
`git history`가 `jj`의 아이디어를 통째로 이식하지 않고
“안전하게 할 수 있는 부분”만 골라 흡수한 것도 이 패턴에
들어맞는다.

### 표현력을 낮추고 얻는 안전은 결국 누군가의 몫을 다른 곳으로 옮긴다

`git history`가 충돌 가능성이 있는 연산을 아예 거부하는 설계는
사용자에게는 안전하게 느껴지지만, 그 복잡성이 사라진 것은
아니다.
충돌이 생길 수 있는 상황은 여전히 존재하고, 그 상황은
`git history`가 처리를 거부하는 순간 다시 `git rebase -i`나
수작업으로 넘어간다.
즉 이 도구는 복잡성을 없앤 것이 아니라 “쉬운 80%”와
“어려운 20%”를 명확히 분리해서, 쉬운 부분만 안전하게
자동화한 것이다.
이 트레이드오프 자체는 합리적이지만, 글은 이 20%가 실제
워크플로에서 얼마나 자주 등장하는지는 다루지 않는다.
스택형 브랜치를 많이 쓰는 팀일수록 병합 커밋이나 충돌이
잦을 수 있고, 그런 팀에게는 `git history`가 감당하는 비중이
저자의 경험보다 훨씬 작을 수 있다.
실제로 HN에서 `smcameron`은 수천 개의 커밋을 스택으로 다뤄야
하는 규모에서는 `git history`나 맨 `git`보다
`stgit` 같은 패치 스택 관리 도구가 훨씬 낫다고 주장했다.
그의 표현을 빌리면 몇십 개 커밋 수준에서는 “그냥 Git으로도
되는데 왜 필요하냐”는 반응이 나오지만, 규모가 커지면
없어서는 안 될 도구가 된다는 것이다[^smcameron].
이는 `git history`가 메꾸려는 빈 자리가 실은 새로운 것이
아니라 이미 서드파티 도구들이 각자의 방식으로 채워온
자리라는 점을 시사한다.

한편 `skydhash`는 애초에 충돌을 겁내는 태도 자체를 문제
삼았다.
충돌은 두 커밋의 의도가 실제로 다르다는 신호이므로
피해야 할 사고가 아니라 두 의도를 어떻게 합칠지 고민할
기회로 봐야 한다는 것이다.
그는 충돌을 두려워하는 사람은 대개 자신이 만지는 커밋
중 하나를 제대로 이해하지 못한 경우가 많다고 지적하며,
문제는 Git의 인터페이스가 아니라 코드베이스에 대한
이해도라고 정리했다[^skydhash].
이 시각에서 보면 `git history`가 충돌을 아예 거부하는 설계는
안전장치인 동시에, 충돌이 요구하는 이해의 과정을
건너뛸 구실을 제공하는 것일 수도 있다.
