# Git 2.54 주요 변경사항

원문: <https://github.blog/open-source/git/highlights-from-git-2-54/>

## 요약

Git 2.54가 릴리스되었다.
세 가지 굵직한 변경이 핵심이다:
`git history` 실험적 신규 명령어, 설정 파일 기반 훅, 기하학적 재팩킹의 기본값 채택.

`git history`는 워킹 트리와 인덱스를 건드리지 않고 히스토리를 수정하는 실험적 명령어다.
현재 `reword`(커밋 메시지 수정)와 `split`(커밋 분할) 두 서브커맨드를 제공한다.
`git history reword <commit>`은 에디터를 열어 메시지를 수정하고
하위 브랜치를 자동으로 재작성한다.
`git history split <commit>`은 `git add -p`와 유사하게 헝크(hunk)를 선택해
커밋을 분할한다.
병합 커밋이나 충돌을 유발하는 연산은 거부하여 단순한 케이스에 집중한다.

훅을 `.git/hooks/` 실행 파일 대신 Git 설정 파일에 정의할 수 있게 되었다.
`~/.gitconfig`나 `/etc/gitconfig`에 훅을 중앙화할 수 있고,
이벤트 하나에 여러 훅을 등록하거나 `hook.<name>.enabled = false`로 개별 비활성화가 가능하다.
`git hook list`로 등록된 훅 목록을 확인할 수 있다.

기하학적 재팩킹(geometric repacking)이 `git maintenance`의 기본 전략이 되었다.
Git 2.52에 도입된 이 전략은 모든 팩파일을 하나로 합치는 비싼 연산 대신
팩파일을 점진적으로 결합하여 저장소 효율을 높인다.

## 주요 개선사항

**`git add -p` 개선**: 이미 승인/건너뛴 헝크에 시각적 표시가 추가되고,
`--no-auto-advance` 플래그로 파일 탐색을 수동 제어할 수 있다.

**`git replay` 개선**: 참조 업데이트를 기본적으로 원자적으로 수행하고,
`--revert` 모드 추가, 빈 커밋 자동 삭제, 루트 커밋 처리 지원.

**HTTP 429 처리**: “Too Many Requests” 응답에 재시도 로직이 추가되었다.
`http.retryAfter`, `http.maxRetries`, `http.maxRetryTime`으로 설정한다.

**`git log -L` 개선**: 픽스액스 옵션(`-S`, `-G`)과 패치 포맷팅과 호환되어
줄 범위 추적 안에서 내용 기반 필터링이 가능해졌다.

**`git rebase --trailer`**: 리베이스된 모든 커밋에 트레일러를 추가하는 플래그.

**GPG 서명 표시 수정**: 이미 만료된 키로 서명된 경우 경고 색상 없이 올바르게 유효 서명으로 표시.

**`git blame --diff-algorithm`**: blame 계산에 사용할 diff 알고리즘(histogram, patience, minimal)을 선택 가능.

**유니코드 별칭**: 서브섹션 문법을 사용해 비ASCII 별칭 이름을 지원한다.

**히스토그램 diff 개선**: 앵커 라인 이동 시 변경 그룹이 잘못 배치되는 문제 수정으로 출력 품질 향상.

**`git backfill` 개선**: 실험적 부분 클론 명령어가 리비전 범위와 패스스펙 인자를 지원.

**MIDX 압축**: 증분 멀티팩 인덱스가 레이어 압축을 지원해 체인 길이를 관리 가능하게 유지.

**`status.compareBranches`**: 여러 브랜치와 동시 비교를 허용하는 새 설정 옵션.

오브젝트 데이터베이스 내부가 플러그 가능한 백엔드 설계로 대규모 리팩터링되어
대안적 스토리지 구성과 향후 확장성의 기반을 마련했다.

## 분석

### `git history`: `git rebase -i`의 한계를 인정한 설계

`git history`는 `git rebase -i`가 “할 수는 있지만 불편한” 작업을
별도 명령어로 분리하려는 시도다.
`git rebase -i`로 커밋 메시지 하나를 수정하려면
인터랙티브 에디터에서 `reword`를 선택하고, 커밋을 찾고, 다시 에디터를 열어야 한다.
`git history reword <commit>`은 이 과정을 단 하나의 명령어로 압축한다.

흥미로운 설계 결정은 병합 커밋과 충돌 케이스를 의도적으로 거부한다는 것이다.
`git rebase -i`는 충돌 처리를 사용자에게 위임하여 강력하지만 위험하다.
`git history`는 “안전하게 할 수 있는 것만” 하겠다는 선언이다.
이는 초보 사용자에게 안전한 진입점을 제공하는 동시에
고급 케이스는 여전히 `git rebase -i`로 처리해야 한다는 트레이드오프를 수반한다.
“실험적” 상태는 API가 안정화되지 않았다는 신호이므로 스크립트에서의 사용은 주의가 필요하다.

### 설정 기반 훅: 단일 저장소 제약의 해소

기존 `.git/hooks/` 방식은 두 가지 근본 문제가 있었다.
첫째, `.git/` 디렉터리는 버전 관리되지 않으므로 팀 공유가 어렵다.
둘째, 사용자가 여러 저장소에서 동일한 훅을 쓰려면 각 저장소마다 설치해야 했다.

설정 기반 훅은 두 번째 문제를 해결한다.
`~/.gitconfig`에 정의한 훅은 사용자의 모든 저장소에 자동으로 적용된다.
`/etc/gitconfig`에 정의하면 시스템 전체에 적용된다.
`hook.<name>.enabled = false`로 특정 저장소에서만 비활성화하는 것도 가능하다.
첫 번째 문제(팀 공유)는 여전히 해결되지 않는다.
팀 공유 훅은 `.githooks/` 디렉터리를 버전 관리하고
`core.hooksPath`로 지정하는 기존 관행이 여전히 필요하다.

### 기하학적 재팩킹의 기본값 채택

Git 저장소는 시간이 지남에 따라 팩파일이 늘어난다.
기존 `gc` 방식은 모든 팩파일을 하나로 합치는(all-into-one) 연산이라
저장소가 클수록 시간이 오래 걸린다.
기하학적 재팩킹은 팩파일 크기의 기하학적 비율을 기준으로
작은 팩파일만 선택적으로 병합한다.
매번 전체를 재팩킹하는 대신 증분(incremental) 작업만 수행하므로
유지보수 오버헤드가 균등하게 분산된다.
이것이 기본값이 된 것은 이 전략이 충분히 검증되었다는 신호다.

## 비평

### 강점

`git history`의 “안전한 부분집합” 접근법은 사용자 경험 측면에서 올바른 방향이다.
Git의 강력함은 종종 복잡성과 동의어였다.
자주 쓰이는 단순한 케이스를 별도 명령어로 추출하면
Git을 배우는 진입 장벽이 낮아진다.
HTTP 429 처리, GPG 서명 표시 수정 같은 실용적 개선들도 현장 피드백이 반영된 결과다.

### 약점

`git history`가 “실험적” 상태로 출시된 만큼
서브커맨드 이름이나 플래그가 향후 변경될 수 있다.
`reword`와 `split` 두 서브커맨드만으로는 `fixup`, `drop`, `exec` 같은
`git rebase -i`의 다른 연산을 대체하지 못한다.
사용자가 두 명령어의 경계를 명확히 이해하지 못하면
“왜 `git history`로 이건 못 하지?”라는 혼란이 생길 수 있다.

설정 기반 훅은 강력하지만 디버깅 복잡성을 높인다.
훅이 설정 파일, `core.hooksPath`, `.git/hooks/` 세 곳에서 올 수 있게 되면
“왜 이 훅이 실행되는 거지?” 또는 “왜 실행이 안 되는 거지?”를
진단하는 것이 어려워진다.
`git hook list`가 이를 부분적으로 해소하지만,
우선순위와 결합 규칙이 명확히 문서화되어야 한다.

## 인사이트

### Git의 사용성 개선은 항상 안전성과 표현력의 긴장 위에 있다

Git은 수십 년간 “강력하지만 실수하기 쉬운” 도구로 알려졌다.
`git rebase -i`는 이 긴장의 대표 사례다.
원하는 모든 것을 할 수 있지만, 잘못 쓰면 히스토리를 망가뜨릴 수 있다.
`git history`는 이 긴장을 해소하는 새로운 시도다.
안전한 연산만 허용하는 대가로 표현력을 일부 포기한다.

이 접근법이 성공하려면 “안전한 부분집합”의 경계가 사용자에게 직관적이어야 한다.
병합 커밋을 거부하는 이유, 충돌을 유발하는 연산을 거부하는 이유가
오류 메시지에 명확히 설명되어야 한다.
그렇지 않으면 사용자는 제한에 부딪힐 때 좌절하고
“결국 `git rebase -i`를 쓸 수밖에 없다”는 결론에 이를 것이다.
새로운 명령어의 성공은 그것이 무엇을 할 수 있는가만큼
무엇을 할 수 없는가를 어떻게 설명하는가에 달려 있다.

Git의 역사를 보면 실험적 명령어가 핵심 기능이 된 사례가 많다.
`git worktree`, `git sparse-checkout`, `git switch`, `git restore` 모두
한때 실험적이었다.
`git history`가 같은 경로를 걷는다면, 향후 버전에서
`fixup`, `drop` 같은 추가 서브커맨드가 붙고 안정화될 것이다.

### 설정의 계층화는 강력함이자 복잡성이다

Git 설정은 시스템, 전역, 로컬, 워크트리 네 계층이 있고
이제 훅도 이 계층에 참여한다.
이 계층화는 “모든 저장소에 공통으로 적용”과 “이 저장소에만 예외”를
우아하게 표현할 수 있게 한다.
그러나 계층이 깊어질수록 “무엇이 실제로 적용되는가”를 추론하기 어려워진다.

설정 기반 훅이 `.git/hooks/`와 공존한다는 것은
두 시스템이 다른 규칙으로 동작할 때 개발자가 혼란을 겪을 수 있음을 의미한다.
Ansible의 변수 우선순위, CSS의 명시도(specificity), 자바의 클래스 로더 위계처럼
계층적 설정은 강력한 추상화이지만 디버깅 시 “어디서 이 값이 오는가”를
추적하는 도구 없이는 유지보수 부담이 된다.
Git이 앞으로 설정 디버깅 도구(`git config --debug`, `git hook list` 확장 등)를
계속 개선해야 하는 이유가 여기에 있다.

### 오브젝트 데이터베이스 플러그 가능 백엔드: 보이지 않는 변화가 만드는 미래

이번 릴리스에서 가장 눈에 띄지 않지만 장기적으로 중요한 변화는
오브젝트 데이터베이스(ODB)의 플러그 가능 백엔드 리팩터링이다.
현재 Git의 ODB는 파일 시스템 기반 팩파일이 유일한 스토리지 구현이다.
이 리팩터링은 “ODB 위에 다른 스토리지 백엔드를 얹을 수 있는” 구조를 만드는 작업이다.

이것이 열어주는 가능성은 크다.
데이터베이스 기반 ODB는 고빈도 커밋 환경에서 파일 시스템 inode 한계를 우회할 수 있다.
오브젝트 스토어를 클라우드 스토리지에 분리하면 대규모 모노레포의 복제 비용을 낮출 수 있다.
Git이 대형 조직의 모노레포에서 겪는 성능 한계의 상당 부분은
파일 시스템 기반 ODB의 구조적 제약에서 온다.
Microsoft의 GVFS, Meta의 Sapling, Google의 내부 Piper가 각각 다른 방식으로
이 문제를 해결해온 것처럼, 공식 Git에서 플러그 가능 ODB가 가능해진다면
이런 벤더별 포크의 필요성이 줄어들 것이다.
지금은 기반 공사 단계이지만, 이 리팩터링이 완성되는 미래의 Git은
지금과 다른 스토리지 패러다임 위에서 작동하게 될 수 있다.
