# Core AI: 애플 실리콘을 위한 새 온디바이스 추론 프레임워크

<https://developer.apple.com/documentation/coreai/>

HN 토론: <https://news.ycombinator.com/item?id=48449665> (358점, 106개 댓글)

## 소개

Core AI는 애플이 공개한 새 온디바이스 AI 추론 프레임워크다. “Run AI models in
your app on Apple silicon”이라는 한 줄 설명대로, 앱 안에서 AI 모델을 빌드, 실행,
배포하는 것을 목표로 한다. iOS 27.0+, iPadOS 27.0+, macOS 27.0+, Mac Catalyst
27.0+, tvOS 27.0+, visionOS 27.0+, watchOS 27.0+에서 베타로 제공되며, CPU, GPU,
Neural Engine을 모두 활용해 최신 모델 아키텍처와 추론 기법을 지원한다고
소개한다. Swift API는 일반적인 작업을 단순하게 만들면서도, 필요할 때는 모델
특수화(specialization), 캐싱, 추론 성능에 대한 세밀한 제어를 제공한다.

프레임워크 본체 외에 모델 준비, 통합, 디버깅을 위한 도구가 함께 제공된다. Core
AI Optimization으로 모델을 애플 실리콘에 맞게 준비하고, Core AI PyTorch
Extensions(Python 패키지)로 `.aimodel` 포맷으로 변환한다. Core AI Debugger
앱은 시각화와 수치 디버깅을 지원하며, 모델 구조를 검사하고 텐서 값을 Python
소스 코드까지 역추적할 수 있다. Xcode 통합도 포함되어 Core AI 디버그 게이지와
인스트루먼트로 추론 성능을 모니터링·프로파일링하고, `coreai-build` 커맨드라인
도구로 모델을 빌드 타임에 미리 컴파일(AOT)할 수 있다. 결정 트리나 테이블 형식
피처 엔지니어링 같은 신경망 외 모델 타입은 기존 Core ML을 사용하라고 안내한다.

API 표면은 명확하게 구획되어 있다. 핵심 타입은 디바이스에서 추론을 실행하는
특수화된 모델인 `AIModel`과 특수화 이전의 소스 모델 애셋인 `AIModelAsset`이다.
추론 계층에는 `InferenceFunction`(입력을 받아 출력을 생성하는 함수),
`InferenceFunctionDescriptor`(함수 시그니처 기술), `InferenceValue`(입출력 값),
`ImageDescriptor`(이미지 차원·픽셀 포맷 기술), `ComputeStream`(비동기 작업
스트림)이 있다. 다차원 배열은 `NDArray`와 `NDArrayDescriptor`가 담당하고, 구성
계층에는 `AIModelCache`(특수화된 모델 아티팩트 캐시), `ComputeUnitKind`(하드웨어
연산 유닛 종류), `SpecializationOptions`가 있다.

통합 가이드에 따르면 워크플로는 다음과 같다. `.aimodel` 파일을 Xcode 프로젝트에
번들하거나 네트워크로 다운로드한다. 빌드에는 기본 설치되지 않는 Metal
Toolchain이 필요하며, 없으면 Metal 컴파일러 누락 오류로 빌드가 실패한다.
Xcode 모델 뷰어에서 파라미터 수, 저장 크기, 연산 정밀도(compute/storage 분리),
연산 분포, 함수 시그니처를 확인할 수 있다. 코드에서는 `AIModel(contentsOf:)`로
모델을 로드하는데, 이때 Core AI가 현재 디바이스의 가용 연산 유닛을 고려해 최적
조합으로 모델을 특수화한다. 이 초기화는 비동기이며 모델 크기에 따라 상당한
시간이 걸릴 수 있다. 이후 `loadFunction(named:)`로 `InferenceFunction`을 얻고,
모델이 기대하는 형태와 타입에 맞는 `NDArray`를 만들어 `mutableView(as:)`로
데이터를 쓴 뒤 `run(inputs:)`로 추론을 실행한다. 이미지 입출력은
`CVMutablePixelBuffer`를 사용하고, 동적 차원은 `?`로 표시되어 런타임에 결정된다.

## 분석

### Core ML의 계승이 아니라 교체다

문서가 가장 조심스럽게, 그러나 분명하게 전달하는 메시지는 Core AI와 Core ML의
역할 분담이다. “결정 트리나 테이블 형식 피처 엔지니어링 같은 신경망 외 모델
타입은 Core ML을 보라”는 안내는 표면적으로 공존을 말하지만, 실질적으로는 Core
ML을 레거시 클래식 ML 전담으로 밀어내는 선언이다. 2017년 등장한 Core ML은
지난 9년간 애플 온디바이스 ML의 정문이었는데, 신경망이라는 본진을 새
프레임워크에 넘겨주는 것이다. HN에서도 같은 문장이 곧바로 증거로 인용됐다.
`bensyverson`이 “이게 Core ML을 완전히 대체하느냐”고 묻자 `earthnail`은 문서의
“신경망 외 모델 타입은 Core ML을 보라”는 구절을 그대로 따와 “그렇다”고
답했다.[^earthnail] 커뮤니티는 이 한 문장을 공존이 아니라 대체의 신호로 읽었다는
점에서 본문의 해석과 일치한다.

이 교체가 필요했던 이유는 Core ML의 설계 시점에 있다. Core ML은 CNN과 소형
분류기의 시대에 설계되었고, 모델을 불투명한 번들로 변환해 넣으면 프레임워크가
알아서 실행하는 “변환하고 기도하기” 모델이었다. 트랜스포머와 생성형 모델
시대에 이 추상화는 한계에 부딪혔다. 개발자는 어떤 연산이 Neural Engine에서
도는지, 특수화에 왜 오래 걸리는지, 메모리가 어디서 새는지 알 수 없었다. Core
AI의 API 표면 — `ComputeUnitKind` 노출, `SpecializationOptions`, AOT 컴파일,
연산 분포 시각화, 텐서 값 역추적 — 은 정확히 이 불투명성에 대한 답이다.

흥미로운 것은 새 이름이다. “Core ML 2”나 “Core ML Next”가 아니라 Core AI라는
별도 브랜드를 만든 것은 하위 호환의 부담 없이 API를 처음부터 다시 설계하겠다는
의지다. 동시에 “ML”에서 “AI”로의 개명은 시대 구분이기도 하다. 머신러닝이
분류·회귀를 의미하던 시대의 프레임워크와, AI가 생성형 모델을 의미하는 시대의
프레임워크를 이름으로 가른 것이다.

### PyTorch를 공식 소스 오브 트루스로 인정했다

변환 도구의 이름이 모든 것을 말한다. Core AI PyTorch Extensions. 기존
coremltools가 TensorFlow와 PyTorch를 모두 지원하는 중립적 변환기를 표방했던
것과 달리, Core AI는 PyTorch를 유일한 공식 입구로 명시한다. 한 걸음 더 나아가
Core AI Debugger는 디바이스에서 실행되는 텐서 값을 “Python 소스 코드까지”
역추적한다. 애플이 자사 디버깅 도구의 추적 종점을 타사 생태계의 언어로 잡은
것이다.

이것은 ML 생태계의 현실에 대한 항복이자 실용주의다. 연구와 모델 개발은
PyTorch에서 일어나고, 애플이 아무리 Swift를 밀어도 모델을 만드는 사람들은
Python을 떠나지 않는다. 애플의 선택은 그 현실을 바꾸려는 대신 경계를 명확히
긋는 것이다. 모델 제작은 Python/PyTorch의 세계, 모델 소비는 Swift의 세계,
그리고 `.aimodel` 파일이 두 세계의 계약(contract)이 된다.

`InferenceFunctionDescriptor`로 런타임에 입출력의 이름, 타입, 형태를 검사할 수
있게 한 설계도 같은 맥락이다. 모델이 앱과 독립적으로 배포·갱신될 수 있다는
전제, 즉 모델 파일이 사실상 버전 관리되는 API라는 전제가 깔려 있다. 문서는
“모델의 입출력이 배포 간에 바뀌어도 코드 변경 없이 동적으로 적응”하는
시나리오를 명시적으로 언급한다.

### 특수화와 캐싱이 1급 개념으로 승격됐다

Core AI의 핵심 타입 이름들에서 눈에 띄는 것은 `AIModelAsset`(특수화 전)과
`AIModel`(특수화 후)의 구분이다. 디바이스별 컴파일 — 같은 모델도 M1 Mac과
A19 iPhone에서 다른 실행 계획을 갖는다는 사실 — 이 API 타입 시스템 수준으로
올라왔다. Core ML에서 백그라운드에서 암묵적으로 일어나던 일이 명시적 개념이
된 것이다.

이 승격은 비용의 인정이기도 하다. 문서는 `AIModel(contentsOf:)` 초기화가
비동기이며 “모델 크기에 따라 상당한 시간이 걸릴 수 있다”고 명시하고, 이를
관리하기 위한 문서(특수화·캐싱 관리, AOT 컴파일)를 Essentials 바로 다음
섹션으로 배치했다. `coreai-build`로 빌드 타임에 미리 컴파일하거나
`AIModelCache`로 특수화 결과물을 관리하는 선택지를 주는 것은, 수 GB짜리
모델의 첫 실행 컴파일이 사용자 경험을 좌우하는 시대의 요구를 반영한다.

Metal Toolchain이 빌드 필수 요건이라는 점도 구조를 드러낸다. `.aimodel`은
결국 Metal 셰이더 컴파일 파이프라인을 통과하는 아티팩트이며, Core AI는 Metal
위에 세워진 추론 계층이라는 뜻이다. 그래픽 API였던 Metal이 애플 컴퓨트
스택의 공통 기반으로 확장되어 온 흐름의 연장선이다.

## 비평

### “최신 모델 아키텍처”라는 주장과 공개된 API의 간극

개요는 “최신 모델 아키텍처와 추론 기법”을 지원한다고 주장하지만, 공개된 문서
범위에서 보이는 API는 고전적인 텐서 입력-텐서 출력 모델이다. `NDArray`를
만들고, 데이터를 쓰고, `run(inputs:)`을 호출하고, 출력을 꺼낸다. 이것은 이미지
분류기나 세그멘테이션 모델의 호출 패턴이지, 2026년에 “최신”이라 부를 만한
자기회귀 생성 모델의 패턴이 아니다.

LLM 추론의 핵심 요구사항 — KV 캐시 같은 상태 유지, 토큰 단위 스트리밍, 동적
배칭, 추측 디코딩(speculative decoding) — 은 개요와 통합 가이드 어디에도
등장하지 않는다. 정작 Core ML은 2024년에 stateful model을 도입하며 이 방향으로
움직였는데, 그것을 대체한다는 새 프레임워크의 첫 문서에서 상태 관리가 보이지
않는 것은 기묘하다. 베타 문서의 미완성일 수 있으나, 그렇다면 “최신 추론 기법”
이라는 마케팅 문구는 현재 문서가 입증하는 범위를 넘어선다.

동적 차원을 `?`로 지원한다는 언급이 시퀀스 길이 가변성의 단서일 수는 있다.
하지만 매 토큰마다 `run(inputs:)`을 호출하며 전체 컨텍스트를 다시 넘기는
구조라면 LLM 추론은 실용적이지 않다. 이 간극이 문서의 누락인지 설계의
한계인지가 Core AI의 실제 가치를 가르는 질문인데, 문서는 그 답을 주지 않는다.

### 또 하나의 독점 포맷, 더 좁아진 입구

`.aimodel`은 애플 생태계에 추가되는 또 하나의 독점 모델 포맷이다. 그리고
공식 입구는 PyTorch 하나뿐이다. coremltools가 TensorFlow, ONNX(초기),
PyTorch를 받던 것과 비교하면 입구는 오히려 좁아졌다. JAX로 학습된 모델,
ONNX로 배포되는 모델, GGUF로 유통되는 양자화 모델은 모두 PyTorch를 경유하는
우회로를 찾아야 한다.

더 큰 문제는 애플 자체 스택 내부의 분열이다. 개발자 앞에는 이제 Foundation
Models 프레임워크(애플 자체 LLM 사용), Core AI(자체 신경망 반입), Core
ML(클래식 ML), MLX(연구·학습), MPS Graph와 BNNS(저수준)가 나란히 놓인다.
문서는 Core ML과의 경계만 한 문장으로 언급할 뿐, MLX와의 관계 — 둘 다 애플
실리콘에서 PyTorch 모델을 실행하는 길이다 — 는 설명하지 않는다. “어느
프레임워크를 써야 하는가”라는 개발자의 첫 질문에 답하는 문서가 없다는 것은
신규 플랫폼 문서의 명백한 결함이다. HN의 반응이 이 진단을 그대로 입증한다.
`pzo`는 “Core AI와 MLX, coremltools 사이에서 정말 혼란스럽다. 애플이 셋의
장단점과 기능 동등성(feature parity)을 더 잘 설명해야 한다”고 토로했다.[^pzo]
`wtallis`는 한발 더 나아가 이 분열이 애플만의 문제가 아니라 업계 전반의
구조라고 짚는다. 그는 비(非)애플 플랫폼에서는 “실리콘 벤더 수에 2를 더한 만큼의
AI 프레임워크”를 신경 써야 하며, 엔비디아는 모든 학습·추론을 CUDA로 끌어들이려
NPU의 유용성을 부정하고, NPU를 만드는 모든 업체가 자기 아키텍처에 맞춘 별도
프레임워크를 두고 있어 파편화가 사라질 기미가 없다고 지적했다.[^wtallis] 애플의
Core AI는 그 파편화 목록에 한 줄을 더 보탠 셈이다.

이 분열은 일시적 과도기가 아니라 구조적 비용이 될 가능성이 높다. 각
프레임워크는 각자의 포맷, 변환 도구, 디버거, 성능 특성을 갖는다. 한 팀이
모델을 MLX로 프로토타이핑하고 Core AI로 배포하려면 두 변환 경로의 수치
일치성을 스스로 검증해야 한다. Core AI Debugger의 “Python 소스 역추적”이
필요해진 이유 자체가, 변환 경계마다 수치가 어긋나는 사고가 그만큼 흔하다는
방증이다.

### 27.0+ 전용이라는 조건이 채택 곡선을 수년 뒤로 민다

Core AI는 모든 플랫폼에서 27.0 이상 전용이다. 실제 앱이 최소 지원 버전을 최신
OS로 올리기까지 보통 2~3년이 걸린다는 점을 고려하면, 대부분의 상용 앱에게
Core AI는 당분간 “쓸 수 없는 권장 경로”다. 그동안 신경망 모델을 온디바이스로
돌려야 하는 앱은 Core ML을 계속 써야 하는데, 문서는 Core ML 신경망 모델에서
Core AI로의 마이그레이션 경로를 전혀 다루지 않는다. HN에서 `wahnfrieden`이
“OS 27+를 요구하므로 Core ML은 하위 호환을 위해 여전히 유용하다”고 간단히
정리한 것은[^wahnfrieden] 이 조건이 현장 개발자에게 곧바로 떠올리는 첫
제약임을 보여준다. 권장 경로가 새로 생겨도 실제 배포 타깃은 한동안 옛 경로에
묶여 있다는 뜻이다.

이 침묵은 개발자를 곤란한 위치에 놓는다. 지금 Core ML에 투자하면 곧
레거시가 될 스택에 투자하는 것이고, Core AI를 기다리면 수년간 신기능 없이
버텨야 한다. 애플은 Metal이 OpenGL을 대체할 때도, SwiftUI가 UIKit을 잠식할
때도 같은 패턴을 보였다. 새 스택을 발표하고, 옛 스택의 개선을 조용히 멈추고,
전환 비용은 생태계가 흡수한다. mlmodel/mlpackage 자산과 그 주변에 쌓인 변환
파이프라인을 가진 팀일수록 이 비용은 크다.

베타 단계의 문서에 마이그레이션 가이드가 없는 것 자체는 이해할 수 있다.
그러나 “신경망이면 Core AI, 아니면 Core ML”이라는 구분을 첫 문단에 박아 넣은
이상, 기존 Core ML 신경망 사용자에 대한 안내 의무는 이미 발생했다. 그 의무를
다하지 않은 채 역할 분담만 선언한 것은 무책임한 순서다.

### 단순함의 약속 뒤에 숨은 운영 복잡성

“Swift API는 일반적인 작업을 단순하게 만든다”는 약속과 달리, 통합 가이드를
따라가면 개발자가 떠안는 운영 부담이 차곡차곡 쌓인다. Metal Toolchain을
별도로 설치해야 하고, 특수화 시간은 “상당할 수 있으며”, 캐시는 앱의 저장 공간
사용량(storage footprint)에 영향을 주고, AOT 컴파일은 빌드 파이프라인에 새
단계를 추가한다. 단순한 것은 호출 코드 다섯 줄이고, 복잡한 것은 그 주변
전부다.

특히 첫 실행 특수화는 사용자가 직접 체감하는 비용인데, 문서는 이를
“비동기이므로 기다려라”로 처리한다. 수 분이 걸릴 수 있는 디바이스 컴파일을
앱의 첫 인상 시점에 배치하는 설계에서, 진행률 보고나 백그라운드 사전 특수화
같은 UX 도구가 있는지는 공개된 문서 범위에서 확인되지 않는다. “저장 공간
줄이기” 문서가 별도로 존재한다는 사실은, 캐시 비대화가 이미 알려진 문제임을
시사한다.

이는 온디바이스 AI 전반의 정직한 비용이지 Core AI만의 잘못은 아니다. 그러나
프레임워크 소개가 “common tasks simple”을 전면에 내세우면서 비용은 하위
문서에 분산 배치한 구성은, 평가 단계의 개발자가 총소유비용을 과소평가하게
만드는 구조다.

## 인사이트

### 추론 한계비용 제로는 앱의 형태를 바꾼다

통합 가이드의 한 문장 — “추론당 비용이 개발자에게도 사용자에게도 없다” — 은
조용하지만 가장 파괴적인 경제적 선언이다. 클라우드 LLM API 위에 지은 기능은
호출마다 비용이 발생하므로, 개발자는 기능을 아껴 쓰도록 설계한다. 버튼 뒤에
숨기고, 횟수를 제한하고, 구독으로 회수한다. 한계비용이 0이 되면 이 설계
문법이 뒤집힌다. AI가 항상 켜져 있고, 모든 입력에 반응하고, 백그라운드에서
상시 추론하는 기능이 경제적으로 합리화된다. HN에서 `dvt`는 이를 “무한 토큰”
이라는 표현으로 요약했다. “내 M1 맥북 프로가 할 수 있는 게 그거고, 내 RTX3090이
할 수 있는 게 그거다. 한 달에 수백 달러를 낼 필요가 없고, 다른 누구도 그럴
필요가 없다”는 것이다.[^dvt] 다만 `ip26`은 곧바로 현실의 상한을 덧붙였다.
“10 tok/s로 제한된 무한 토큰은 한 달에 2,600만 토큰”이라는 계산은,[^ip26]
한계비용이 0이라도 처리량(throughput)이라는 다른 자원의 천장이 남는다는 점을
환기한다. 이는 본문이 강조한 “항상 켜진 AI”의 경제적 합리성에 물리적 제약이라는
단서를 단다.

`an0malous`는 이 흐름을 산업 구조의 차원에서 읽었다. “AI 회사들이 IPO를
서두르는 이유가 이것이다. 내년 말이면 대부분의 AI를 디바이스에서 돌리게 된다.
그들에게는 해자(moat)가 없고, 스케일링의 한계에 도달했으며, 마법의 대부분은
더 작은 모델로 증류될 수 있다는 걸 그들도 안다”는 주장이다.[^an0malous]
온디바이스 추론의 한계비용 제로가 단지 앱의 형태만이 아니라 프런티어 랩의
사업 모델까지 흔든다는, 본문보다 한 단계 더 공격적인 전망이다.

2차 효과는 앱의 물리적 형태에 나타난다. 모델이 경쟁력이 되면 앱들은 더 큰
모델을 싣고, 앱 크기는 수 GB 단위로 부푼다. 문서가 모델의 네트워크 다운로드를
1급 시나리오로 언급하고 저장 공간 관리 문서를 따로 둔 것은 애플이 이를
예견했다는 뜻이다. 그 끝은 게임 업계가 먼저 도착한 곳이다. 본체는 작게,
애셋은 나중에. 모델 가중치가 게임 애셋처럼 CDN으로 배포되고 버전 관리되는
세계에서, 디바이스 저장 공간은 새로운 희소 자원이 되고 OS의 모델 캐시 관리
정책이 앱 간 경쟁의 심판이 된다.

이 지점에서 애플의 이해관계가 드러난다. 추론 비용이 0인 대신 그 비용은
하드웨어 가격에 선불로 포함되어 있다. 온디바이스 AI가 좋아질수록 상위
칩셋의 구매 이유가 강해지고, 이것이 클라우드 사업자가 아닌 하드웨어
판매자로서 애플이 온디바이스에 투자하는 구조적 동기다. Core AI는 기술
스택이면서 동시에 실리콘 업그레이드 수요를 만드는 장치다.

### Metal 대 OpenGL의 재연 — 이식성을 버리고 수직 통합을 산다

Core AI의 등장 구도는 2014년 Metal이 OpenGL을 대체하던 구도와 정확히
겹친다. 범용성과 이식성을 지향하던 옛 계층(OpenGL/Core ML)이 하드웨어의
진화를 따라가지 못하자, 애플은 표준이나 호환성 대신 자사 실리콘에 밀착한
수직 통합 계층을 새로 짓고 옛 계층을 단계적으로 고사시켰다. “Designed with
Apple silicon in mind”라는 첫 문장은 이 선언의 반복이다. 실제로 Core AI가
Metal Toolchain 위에 서 있다는 사실은 이 계보를 물리적으로도 잇는다.

이 패턴의 교훈은 양면적이다. Metal의 수직 통합은 실제로 성능을 끌어냈고,
애플 실리콘 전환의 기반이 되었다. 동시에 게임 업계의 크로스 플랫폼
타이틀들이 Mac을 건너뛰는 비용도 치렀다. Core AI에서 같은 트레이드오프가
재연된다면, 애플 디바이스에서의 추론 성능은 동급 최고가 되겠지만,
크로스 플랫폼으로 모델을 배포하는 팀 — 즉 오늘날 거의 모든 ML 팀 — 은
ONNX Runtime이나 ExecuTorch 같은 이식 계층과 Core AI 사이에서 선택을
강요받는다. 역사가 반복된다면 대형 앱은 자체 추론 엔진을 들고 다니고, Core
AI의 주 사용자는 애플 생태계 전업 개발자로 수렴할 것이다.

차이점도 있다. 그래픽과 달리 모델 생태계에는 PyTorch라는 압도적 단일
중심이 있고, 애플은 이번에 그 중심과 싸우는 대신 공식 다리(PyTorch
Extensions)를 놓았다. Metal 시대에는 없던 양보다. 이 다리의 품질 — 변환
실패율, 연산 커버리지, 수치 일치성 — 이 Core AI가 Metal의 성공을 재연할지,
아니면 변환기가 항상 한 발 늦는 coremltools의 전철을 밟을지를 결정할 것이다.

HN의 `ankit219`은 이 다리가 단순한 호환 장치가 아니라 영향력의 지렛대가 될 수
있다고 봤다. 애플이 활성값 양자화(`w4a8`, `w4a16`)까지 작업 중이며, 이것이
제대로 동작한다면 — 그는 “큰 if”라고 단서를 달았다 — 시장 도달 범위를 고려할 때
애플이 “100B 미만 모델이 학습되고 서빙되는 방식을 상당 부분 좌우할 수 있다”는
것이다.[^ankit219] 수직 통합이 성능을 넘어 모델 제작 관행 자체를 끌어당기는
중력이 될 수 있다는 관찰이다. 한편 `ABS`는 애플이 의도적으로 경계를 긋고 있다고
짚었다. 그는 아직 어디서도 부각되지 않은 것으로 Thunderbolt 5 위의
`JACCL`을 통한 맥 간 분산 추론과 OpenAI 호환 `mlx_lm.server`를 들면서,
“애플은 MLX(가중치 직접 반입)를 Foundation Models / Core AI와 별도로
유지한다”고 정리했다.[^ABS] 본문이 지적한 “MLX와의 관계 설명 부재”가
누락이 아니라 의도적 분리일 수 있음을 시사하는 단서다.

### 모델 파일이 API 계약이 되면 조직 구조가 따라온다

`InferenceFunctionDescriptor`를 통한 런타임 시그니처 검사, 함수 이름으로
모델 내 기능을 조회하는 설계, 모델만 독립적으로 갱신하는 배포 시나리오의
명시. 이 셋을 합치면 `.aimodel` 파일은 단순한 데이터 번들이 아니라 버전
관리되는 인터페이스, 즉 API 계약이 된다. protobuf 스키마가 서비스 간 계약인
것처럼, 모델 파일이 ML 팀과 앱 팀 간 계약이 되는 것이다.

콘웨이의 법칙은 이 계약선이 조직 구조선이 될 것을 예고한다. Python으로 모델을
만드는 팀과 Swift로 앱을 만드는 팀은 이미 다른 도구, 다른 배포 주기, 다른
문화를 갖고 있고, Core AI는 그 분리를 봉합하는 대신 공식화한다. 디버거가
텐서 값을 Python 소스까지 역추적해 주는 기능은, 계약 위반(수치 불일치)이
발생했을 때 책임 소재를 가리는 법의학 도구인 셈이다. 잘 작동하면 두 팀은
계약만 보고 독립적으로 움직일 수 있다.

그러나 API 계약의 역사는 계약 명세만으로 충분하지 않다는 것을 보여줬다.
스키마가 있어도 시맨틱 변경 — 같은 형태의 입력이지만 다른 정규화를 기대하는
모델 — 은 잡히지 않는다. 입력이 `[3, 4]` float32라는 검사는 통과하지만 RGB
순서가 BGR로 바뀐 것은 런타임 어디서도 걸리지 않는다. 모델 계약에는 아직
시맨틱 버저닝도, 호환성 테스트 표준도 없다. Core AI가 만든 계약 인프라는
필요조건이지 충분조건이 아니며, 그 공백을 메우는 검증 관행 — 모델 회귀
테스트, 골든 입출력 세트 — 이 따라오지 않으면 “모델만 갱신하는 배포”는
조용한 품질 사고의 통로가 될 것이다.

### 제어권의 노출은 미래 실리콘에 대한 부채다

`ComputeUnitKind`로 연산 유닛 종류를 노출하고 `SpecializationOptions`로
특수화를 제어하게 한 것은 Core ML의 불투명성에 대한 직접적인 응답이고,
개발자들이 요구해 온 것이다. 그러나 제어권의 노출은 양날의 검이다. 개발자가
특정 칩 세대의 특성에 맞춰 손튜닝을 시작하는 순간, 그 튜닝은 다음 칩
세대에서 기술 부채가 된다. “이 모델은 Neural Engine에 강제 배치”라는 오늘의
최적화가, NPU 구조가 바뀐 2년 뒤 디바이스에서는 비관화(pessimization)가
되는 식이다.

애플은 이 긴장을 알고 있다. 기본 경로는 여전히 “Core AI가 가용 유닛을 고려해
최적 조합을 선택한다”는 자동 스케줄링이고, 제어는 옵션으로 밀어 두었다.
하지만 성능이 곧 기능인 AI 앱에서 개발자들이 옵션을 옵션으로 남겨둘
가능성은 낮다. 웹이 User-Agent 스니핑으로, 게임이 GPU 모델별 분기로 겪었던
일 — 하드웨어 식별 기반 분기가 누적되어 플랫폼의 진화를 제약하는 상황 — 이
연산 유닛 선택에서 재연될 수 있다.

이 트레이드오프의 균형점은 결국 자동 스케줄러의 품질이 결정한다. 자동
선택이 손튜닝 대비 95%의 성능을 내면 아무도 수동 제어를 쓰지 않을 것이고,
80%에 그치면 모두가 쓸 것이다. Core AI의 장기적 성패는 화려한 API 표면이
아니라, 문서에 한 줄로 언급된 “최적 조합을 선택한다”는 스케줄러가 침묵
속에서 얼마나 잘 작동하느냐에 달려 있다.
