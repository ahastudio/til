## 1장 — 에이전트 기초

### 에이전트 = LLM + 컨텍스트 + 도구

저자는 에이전트를 세 구성요소로 정의한다.

- **LLM(대뇌)**: 의도 파악, 단계 분해, 판단 수행. 사전 훈련과 사후 훈련이 합쳐진 결정 핵심
- **컨텍스트(눈)**: 환경 정보, 사용자 기억, 도메인 지식, 작업 진행 상황. 결정 시점에 에이전트가 볼 수 있는 모든 정보
- **도구(손발)**: API 함수, Skills, 동적 코드 생성, 서브 에이전트 위임. 결정을 실제 세계 변화로 전환하는 수단

이 공식이 강한 이유는 강화학습(RL)의 세 개념과 정확히 대응하기 때문이다.
LLM은 정책(Policy), 컨텍스트는 관찰 공간(Observation Space), 도구는 행동 공간(Action Space)이다.

### 도구의 다섯 가지 유형

1. **감지 도구**: 정보를 가져온다. 웹 검색, 파일 읽기, 지식베이스 검색
2. **실행 도구**: 세계를 바꾼다. 코드 실행, 파일 쓰기, 이메일 발송
3. **협력 도구**: 서브 에이전트나 인간과 협업한다. 서브 에이전트 생성/메시지/취소
4. **이벤트 트리거 도구**: 에이전트가 등록하고 외부가 실행을 유발한다. 타이머, 모니터링, 외부 채널 연결
5. **사용자 소통 도구**: 에이전트가 사용자에게 능동적으로 정보를 전달한다. 메시지 회신, 카드 발송, 알림

### ReAct 루프

에이전트가 작업을 수행하는 핵심 패턴이다.
“생각(Reasoning) → 행동(Acting) → 관찰(Observing)” 세 단계가 반복된다.
이 루프에서 에이전트의 컨텍스트는 **정적 접두어(시스템 프롬프트 + 도구 정의) + 궤적(동적 메시지 기록)**으로 구성된다.

컨텍스트 구성요소 소거 실험(실험 1-1)에서 각 구성요소의 필수성이 증명됐다.
도구 정의가 없으면 에이전트는 도구를 호출할 수 없다.
도구 실행 결과가 없으면 이전 단계를 보지 못해 무한 루프에 빠진다.
추론 과정이 없으면 전후 결정이 모순된다.
메시지 기록이 없으면 이미 완료한 단계를 반복한다.

### 하네스(Harness)와 “모델이 곧 에이전트” 패러다임

저자는 선진 모델이 도구 호출 능력을 내재화하면서 “모델이 곧 에이전트(Model as Agent)” 시대가 왔다고 본다.
그러나 모델이 강해질수록 하네스가 더 중요해진다고 강조한다.

하네스란 컨텍스트 관리, 도구 인터페이스, 안전 제약, 검증과 교정 등의 공학적 외피다.
Rich Sutton의 “쓴맛의 교훈(The Bitter Lesson)” — 인간의 선험적 지식을 코딩한 시스템은 결국 범용적 방법에 진다 — 에 비춰, 저자는 **방향에는 동의하되 속도에는 현실적**이어야 한다는 입장을 취한다.
모델이 아직 안정적으로 하지 못하는 것을 하네스가 보완하고, 모델이 능력을 내재화할 때마다 하네스는 그 층을 걷어내고 다음 능력 전선으로 이동한다.

### 에이전트의 세 가지 학습 패러다임

저자는 에이전트 학습을 세 가지로 분류한다.

- **사후 훈련(Post-training)**: 강화학습으로 경험을 파라미터에 고정. 가장 강하지만 업데이트 비용이 높다
- **컨텍스트 내 학습(In-Context Learning)**: 주의(Attention) 메커니즘으로 패턴을 빠르게 적응. 세션이 끝나면 사라진다. 새로운 규칙 발견이 아니라 이미 본 패턴의 매칭에 가깝다
- **외부화 학습(Externalized Learning)**: 지식과 프로세스를 지식베이스·실행 코드로 외부화. 지속적이고 해석 가능하다

세 패러다임은 시간 규모에서 상보적이다.
사후 훈련은 기초 능력, 컨텍스트 학습은 빠른 적응, 외부화 학습은 신뢰성과 효율성을 담당한다.

## 2장 — 컨텍스트 공학

### 컨텍스트가 능력의 상한을 결정한다

저자는 컨텍스트 공학이 기술 문제이기 이전에 **조직 문제**라고 본다.
핵심 지식이 암묵적으로만 존재하는 팀에서는 아무리 좋은 에이전트도 힘을 쓰지 못한다.
원격 근무 친화적이고 문서 중심의 소통 문화를 가진 팀이 에이전트 친화적이다.
Linux 커널 프로젝트처럼 모든 논의가 공개되고 결정마다 기록이 남는 환경이 이상적인 예다.

OpenAI 연구원 옹자이(翁家翌)의 말을 저자는 인용한다.
“사람과 모델 모두에게 가장 중요한 것은 컨텍스트다.”
“팀 협업의 가장 큰 문제는 컨텍스트의 불일치다.”
“AI가 단기간 내 인간을 대체하지 못하는 가장 큰 이유도 컨텍스트다 — AI는 인간과 같은 환경에 있지 않기 때문이다.”

### API 수준에서 본 컨텍스트 구조

LLM API 호출 시 메시지에는 네 가지 역할이 있다.

- **system**: 개발자가 작성하는 시스템 프롬프트. 에이전트의 정체성과 행동 규칙
- **user**: 최종 사용자의 입력
- **assistant**: 이전 모델 회신. 텍스트(content), 추론 과정(reasoning), 도구 호출 요청(tool_calls) 세 부분으로 구성
- **tool**: 도구 실행 결과. `tool_call_id`로 해당 호출 요청과 연결

도구 정의(tools)는 메시지가 아니라 별도 필드로 전달되며, 에이전트가 사용할 수 있는 도구와 파라미터 형식을 명시한다.

### KV 캐시와 정적 접두어

KV 캐시(Key-Value Cache)는 동일한 토큰 시퀀스를 반복 계산하지 않도록 캐싱하는 메커니즘이다.
정적 접두어(시스템 프롬프트 + 도구 정의)가 자주 변하지 않을수록 KV 캐시 적중률이 높아져 비용이 줄어든다.
대화가 길어질 때는 요약, 선택적 삭제, 외부 저장 후 검색 같은 압축 전략으로 컨텍스트 창 초과를 방지한다.

## 3장 — 사용자 기억과 지식베이스

### 기억 시스템의 목표

기억 시스템의 목표는 대화 기록을 그대로 저장하는 것이 아니다.
상호작용을 통해 사용자에 대한 예측 모델을 구축하는 것이다.
분산된 대화 기록에서 핵심 정보를 추출·압축하고, 다음 번에 바로 활용할 수 있는 형태로 유지한다.

항공권 예약 예시에서 저자가 설명하는 추출 과정의 세 특징이 중요하다.

- **선택성**: 임시 정보(검색 결과 3건 반환됨)는 저장하지 않는다
- **추상화**: “이번 항공권에 창가 자리”가 아니라 “창가 선호”로 일반화한다
- **구조화**: 유형(선호, 제약, 계정 번호)을 태깅해 나중에 검색 가능하게 한다

### 기억 능력의 3단계 평가 프레임

저자가 제안하는 평가 기준이다.

1. **1단계 — 기초 회상**: 사용자가 직접 제공한 구조화된 정보를 정확히 저장·검색한다
2. **2단계 — 다중 세션 검색**: 다양한 시점과 채널에서 온 여러 세션을 통해 관련 정보를 찾아 추론한다. “내 차 점검 예약해줘”라고 하면 차가 두 대임을 알고 어떤 차인지 확인한다
3. **3단계 — 능동 서비스**: 명시적 지시 없이 과거 기억에서 연관성을 발견해 선제적으로 돕는다. 국제 항공권 예약 시 몇 달 전 저장된 여권 만료일 접근을 확인하고 경고한다

### 기억의 계층 구조

기억은 단일 저장소가 아니라 계층으로 설계한다.

- **궤적(Trajectory)**: 현재 실행 세션의 완전한 기록. 시간순으로 추가만 하고 수정은 없다(append-only)
- **세션 기억**: 여러 세션에 걸쳐 유지되는 사용자별 요약 정보
- **지식베이스**: 모든 사용자가 공유하는 집단 지식. RAG로 검색해 컨텍스트에 주입한다

RAG(검색 증강 생성)는 질문을 벡터로 변환해 의미적으로 유사한 문서를 검색하고 컨텍스트에 주입한다.
사용자 기억과 지식베이스는 스케일만 다를 뿐 동일한 기반 기술을 공유하며, 정보 충돌·지식 노후화·검색 부정확이라는 동일한 도전을 공유한다.

## 4장 — 도구

### 전용 도구 vs Skill + 범용 실행기

저자는 에이전트의 능력을 두 가지 형태로 표현할 수 있다고 설명한다.

- **전용 코드 도구**: 구조화된 함수 호출. 결정론적이고 테스트 가능하지만, 각 도구가 수백 토큰을 차지하고 수가 늘어나면 KV 캐시를 깨뜨린다
- **Skill + 범용 실행기**: 자연어로 작성된 Skill 문서로 작업 절차를 기술하고, 에이전트가 셸이나 코드 인터프리터로 실행한다. 적은 수의 범용 도구로 많은 시나리오를 커버한다

배포 자동화를 예로 들면, `npm run build → docker build → kubectl apply` 순서를 Skill 문서로 작성하면 에이전트가 `bash` 도구로 단계별 실행한다.
각 단계를 전용 도구로 만들 필요가 없다.

선택 기준 세 가지: 파라미터 복잡도(중첩 객체나 복합 검증이 있으면 전용 도구), 변경 빈도(자주 바뀌면 Skill), 모델 능력(약한 모델일수록 구조화된 스키마가 필요).

### 도구 설계의 통용 원칙

**전용 도구보다 범용 도구를 우선한다.** 보안, 권한, 성능상 명확한 이유가 없는 한 그렇다.
`code_interpreter`는 십수 개의 전용 계산기를 대체하며, 더 적은 토큰으로 더 유연하다.

**도구 수가 너무 많으면(100개 이상) 최상위 모델도 선택 오류를 낸다.**
유사한 기능을 통합하는 기준은 입출력 형태와 사용 시나리오의 중첩도다.
PDF/DOCX/PPTX 텍스트 추출은 `read_document` 하나로 통합할 수 있다.
그러나 이미지 OCR과 영상 키프레임 추출은 파라미터 형태와 지연 특성이 달라서 통합하면 오히려 인터페이스가 모호해진다.

**도구 설명에서 가장 중요한 것은 “언제 쓰지 말아야 하는가”다.**
도구 호출 실패의 근본 원인 대부분은 모델이 도구가 할 수 있는 것을 몰라서가 아니라, 할 수 없는 것을 몰라서다.

**파라미터 설명에는 구체적인 예시를 써야 한다.**
“`timestamp`: RFC3339 형식”보다 “`timestamp`: RFC3339 형식, 예시 `2024-03-15T14:30:00Z`”가 낫다.
복잡한 작업을 동시에 처리할 때 모델의 주의가 분산되기 때문에, 예시가 없으면 파라미터 형식을 쉽게 틀린다.

## 5장 — 코딩 에이전트

### 코딩이 에이전트의 기초 능력인 이유

저자의 핵심 주장은 **코드 생성이 일부 에이전트의 전문 기능이 아니라 모든 에이전트의 기초 능력**이라는 것이다.
코드는 도구 상자의 도구 하나가 아니라 런타임에 동적으로 새 도구를 만드는 **메타 능력**이다.

코드의 가치는 두 차원이다.
**사고의 엄밀성**: “나이 18세 이상이고 실명 인증된”이라는 조건을 자연어로 서술하면 여러 해석이 가능하지만, `age > 18 and is_verified`는 모호하지 않다.
**표현의 객관성**: 실행 가능한 코드 자체가 논리 일관성의 증명이고, 실행 결과가 객관적인 성공 기준을 제공한다.

### 코딩 에이전트의 7개 핵심 도구

저자는 기초 코딩 에이전트에 필요한 도구를 7개로 정리한다.

1. **Code Interpreter**: 격리된 샌드박스에서 Python 코드를 안전하게 실행
2. **Bash Shell**: 터미널에서 명령 실행. 테스트 실행, 특수 형식 파일 처리
3. **파일 읽기**: 코드, 설정, 문서, 로그 읽기
4. **파일 쓰기**: 새 파일 생성 또는 기존 파일 완전 덮어쓰기
5. **파일 편집**: 기존 파일의 부분 수정. 코드 유지보수의 핵심
6. **파일명 검색(Glob)**: 패턴 매칭으로 파일 위치 탐색. 예: `**/*.py`
7. **파일 내용 검색(Grep)**: 파일 내 텍스트 패턴 검색

이 7개 도구로 TODO 수집, 코드 리팩터링, 데이터 분석, 자동화 스크립트 작성 같은 다양한 작업을 조합으로 수행할 수 있다.

### 통용 에이전트 아키텍처 — OpenClaw

저자는 Manus 같은 통용 에이전트 제품과 오픈소스 OpenClaw 프로젝트를 분석해, **코딩 에이전트 + 파일 시스템**이 개방형 태스크 통용 에이전트의 핵심 기술 기반임을 논증한다.

왜 코딩이 핵심인가 하면, 거의 모든 효율적인 콘텐츠 생성이 결국 코드로 귀결되기 때문이다.
PPT는 OOXML 코드, PDF 보고서는 코드로 생성, 데이터 분석은 Python 스크립트다.
성공한 브라우저 조작 순서도 RPA(Robotic Process Automation) 코드로 고정화할 수 있다.
Deep Research의 웹 요청과 파싱도 코드로 구현된다.

OpenClaw의 파일 시스템은 단순 저장소가 아니다.
에이전트의 기억, 지식, 능력의 허브다.
장기 기억은 `MEMORY.md`와 날짜별 Markdown 로그에 저장된다.
벡터 데이터베이스가 아닌 Markdown을 선택한 이유가 명확하다.
사용자가 직접 열어서 읽고 수정할 수 있고, Markdown은 시간 순서를 자연스럽게 보존하며, Git으로 버전 관리와 롤백이 된다.

에이전트는 파일 쓰기 능력으로 스스로를 진화시킨다.
은행에 전화했을 때 개설 지점 주소가 필요하다는 새 사실을 발견하면, 그 경험을 지식베이스에 기록한다.
다음에 같은 태스크를 실행할 때 자동으로 로드된다.
이것이 8장에서 심화 논의하는 외부화 학습의 구체적 실천이다.

코딩 에이전트 중심 아키텍처의 적용 범위는 **개방형 태스크**다.
태스크 경계가 불확실하고 산출물 형태가 다양한 심층 조사, 콘텐츠 생성, 데이터 처리 시나리오에 적합하다.
수직 도메인 고객 서비스나 음성 어시스턴트처럼 태스크 공간이 닫혀 있는 에이전트에서는 아키텍처 중심이 고정된 비즈니스 흐름과 도메인 도구에 있다.
그러나 이 경우에도 코딩은 필수 기초 능력이다.

## 인사이트

### 하네스 공학과 “쓴맛의 교훈” 사이의 긴장

저자가 1장에서 가장 솔직하게 다루는 지점이 여기다.
Rich Sutton의 “쓴맛의 교훈”은 70년간 AI 역사의 패턴을 요약한다.
인간의 선험 지식을 코딩한 시스템은 단기에는 효과적이지만, 장기적으로는 범용적 방법(탐색과 학습)에 패배했다.

하네스가 이 패턴의 반복이 되지 않으려면, 하네스는 모델이 아직 불안정하게 하는 것을 보완하는 임시 교량이어야 하고, 모델이 한 층을 내재화할 때마다 그 층을 걷어내야 한다.
저자가 “방향 동의, 속도 현실적”이라고 표현한 것이 이 입장이다.
도구 호출과 장기 계획이 한때 외부 오케스트레이션에 의존했다가 지금은 모델의 원시 능력이 된 것이 방향의 증거다.
그러나 이 “흡수”가 직관보다 훨씬 느리기 때문에, 지금 이 시점의 모델 능력 경계가 하네스의 가치다.

### 컨텍스트 공학은 기술보다 조직 변화가 먼저다

저자가 2장에서 강조하는 관점이다.
에이전트가 효과를 발휘하려면 필요한 정보가 명시적이고 구조화되어 있어야 한다.
정보가 특정 사람의 머릿속에만 있거나 비공개 채팅에 흩어져 있으면, 에이전트는 그 정보를 참조할 수 없다.

“AI 원시 팀을 만드는 것은 새 도구를 도입하는 것이 아니라 문서화 운동이다”라는 저자의 주장은 기술서에서 보기 드문 관점이다.
에이전트 도입 전에 팀이 어떻게 지식을 관리하는지 점검해야 한다는 것이다.
