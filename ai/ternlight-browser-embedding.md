# Ternlight — 5~7MB WASM 번들에 넣은 삼진 가중치 임베딩 모델

<https://ternlight-demo.vercel.app>

<https://github.com/soycaporal/ternlight>

HN 토론: <https://news.ycombinator.com/item?id=48811644> (148점, 40개 댓글)

## 소개

Ternlight는 문장을 384차원 L2 정규화 벡터로 바꾸는 임베딩 모델을,
엔진·모델·토크나이저를 통째로 하나의 WASM 파일에 담아 배포하는
프로젝트다.
`all-MiniLM-L6-v2`를 교사 모델로 삼아 BitNet b1.58 방식의
양자화 인식 학습(QAT)으로 증류했고,
결과물은 `@ternlight/base`(7.2MB, quality tier)와
`@ternlight/mini`(5.0MB, small tier) 두 패키지로 npm에 배포된다.
라이선스는 MIT다.

핵심 설계는 세 가지로 요약된다.

- **삼진 가중치**: 모든 가중치가 `-1`, `0`, `+1` 중 하나이며,
  추론은 곱셈 없는 덧셈·뺄셈으로만 이뤄진다.
  학습 단계부터 삼진 모델로 훈련하기 때문에 사후 양자화보다
  품질 손실이 적다.
- **단일 번들**: 모델, BERT 토크나이저, 추론 엔진을 하나의 `.wasm`에
  넣어 별도의 postinstall이나 런타임 fetch가 없다.
- **SIMD 추론 엔진**: Rust로 작성해 WASM SIMD로 컴파일한 엔진이
  덧셈·뺄셈 연산을 CPU 벡터 명령으로 처리한다.

Node 18 이상, 브라우저, Cloudflare Workers, Vercel Edge, Deno, Bun을
모두 지원하며, 패키지가 환경에 맞는 로더를 자동으로 고른다.

## 아키텍처와 벤치마크

두 모델 모두 최대 128토큰(약 95단어)을 입력받아 384차원 벡터를
출력하지만, 내부 구성은 다르다.

| 항목                       | mini         | base         |
| -------------------------- | ------------ | ------------ |
| 레이어 수 / d_model / 헤드 | 2 / 256 / 4  | 2 / 384 / 6  |
| 파라미터 수                | 약 950만     | 약 1,540만   |
| p50 지연(1회 임베딩)       | 2.5ms        | 5.1ms        |
| 단일 스레드 처리량         | 약 400 emb/s | 약 195 emb/s |
| MiniLM 교사 대비 Spearman  | 0.820        | 0.844        |
| SciFact NDCG@10            | 0.439        | 0.465        |

내부 구조는 표준 Pre-LN 트랜스포머다.
어텐션과 FFN 서브레이어의 `nn.Linear`는 모두 BitLinear로 교체되고,
최종 프로젝션(256→384 또는 384→384) 레이어만 float32로 남는다.
BitLinear의 순전파는 입력에 파라미터 없는 LayerNorm을 적용한 뒤
활성화는 토큰별 AbsMax 기준 int8로,
가중치는 행렬별 AbsMedian 기준 삼진으로 양자화한다.
평가 문서(`docs/eval-methodology.md`)는 모델 품질, 양자화 손실,
런타임 성능, 크기, 호환성, 경쟁 비교라는 6개 축으로 릴리스 전
검증 항목을 정의해두고 있다.

## 학습과 평가 파이프라인

저장소는 `training/`에 Python 기반 증류·QAT·패킹 파이프라인을,
`eval/`에 품질·성능 벤치마크를 각각 별도 디렉터리로 분리해뒀다.
평가 항목 중 Recall@K는 “일반 질의 20개, 기술 질의 20개”로 구성된
수작업 테스트셋을 각각의 코퍼스에 대해 검색하는 방식으로 측정한다.
teacher alignment와 STS-B 계열 지표(AUC, Spearman)는 MS MARCO
2,000개 질의 홀드아웃과 STS 벤치마크 1,379개 문장 쌍이라는
공개된 표준 데이터셋을 사용해 비교적 널리 통용되는 기준을 따른다.

양자화 손실은 임베딩 테이블 양자화, BitLinear 가중치 양자화,
활성화 int8 양자화 세 단계로 나눠 각각 fp32 기준과의 품질 차이를
따로 측정한다.
이렇게 단계를 분리한 이유는 삼진 제약이 실제로 어느 구성 요소에서
가장 큰 비용을 발생시키는지 짚어내기 위해서다.

HN 댓글 중에는 비슷한 문제의식을 가진 다른 소형 임베딩 프로젝트를
언급한 것도 있었다.
`dmezzetti`는 1~4MB, 25만~95만 파라미터 규모의 BERT Hash 임베딩
모델을 직접 만든 경험을 공유했고[^dmezzetti],
`jbellis`는 CPU에서 돌릴 수 있는 파인튜닝 베이스라인으로
3천만 파라미터짜리 Granite r2 small을 대안으로 제시했다[^jbellis].
두 댓글 모두 Ternlight가 던진 "임베딩 모델을 얼마나 작게 만들
수 있는가"라는 질문이 이미 여러 팀이 각자 다른 크기·아키텍처로
탐색하고 있는 영역임을 보여준다.

## 비평

### Recall@K 벤치마크는 표본 40개로 일반화를 감당할 수 없다

teacher alignment나 STS-B 지표는 각각 2,000개, 1,379개라는
충분한 규모의 표준 데이터셋을 쓰지만,
정작 실사용과 가장 가까운 지표인 Recall@K는 일반 질의 20개와
기술 질의 20개, 합쳐서 40개짜리 수작업 테스트셋에 의존한다.
이 정도 표본으로는 신뢰구간 자체가 의미 있게 좁혀지지 않는다.
질의 하나가 오답으로 바뀌는 것만으로 Recall 수치가 5%p씩
흔들릴 수 있는 규모이기 때문에,
0.439와 0.465 같은 두 tier 사이의 SciFact NDCG@10 차이는
표준 벤치마크에서 나온 값이니 신뢰할 수 있어도,
Recall@K 수치는 릴리스 노트에 적힌 그대로 받아들이기보다
실제 도메인 코퍼스로 재현해봐야 하는 참고치에 가깝다.

### 작은 모델일수록 도메인별 품질 편차가 커지는데 검증 축이 없다

950만 파라미터, 2레이어짜리 모델은 애초에 표현력의 여유가 크지
않다.
이런 모델이 특정 도메인에서 잘 작동한다고 해서 다른 도메인에서도
비슷한 품질을 낼 것이라고 보장할 근거는 약하다.
그런데 저장소가 공개한 평가 축 여섯 가지 중 어느 것도
“도메인 간 편차”를 직접 측정하지 않는다.
Recall@K를 일반/기술로 나눈 것이 가장 근접한 시도이지만,
법률, 의료, 다국어, 코드 같은 다른 도메인은 아예 다루지 않는다.
증류 교사인 MiniLM-L6 자체가 범용 목적으로 학습된 모델이기 때문에
그 편향이 그대로 상속됐을 가능성이 높고,
여기에 삼진 양자화로 인한 표현력 손실까지 더해지면
도메인 특이적 실패가 어디서 얼마나 발생하는지는
사용자가 직접 검증하기 전까지 알 수 없다.

### “교사 대비 유사도”가 실사용 품질의 대리 지표로 과도하게 쓰인다

여섯 개 평가 축 중 세 개(teacher alignment, STS-B AUC, STS-B
Spearman)가 사실상 “교사 모델과 얼마나 비슷한가”를 측정하는
지표다.
이는 증류가 잘 됐는지를 보는 데는 적합하지만,
교사 모델인 MiniLM-L6 자체의 한계 — 예를 들어 긴 문맥에서의
성능 저하나 특정 언어에 대한 취약점 — 을 학생 모델이 그대로
물려받았는지는 이 지표들로 걸러낼 수 없다.
결국 “교사를 얼마나 닮았는가”와 “실제로 쓸만한가”는 다른 질문인데,
현재 평가 체계는 전자에 편중돼 있고 후자를 직접 겨냥한 지표는
Recall@K 하나뿐이며 그마저 표본이 지나치게 작다.

## 인사이트

### 온디바이스 임베딩은 검색 인프라의 위치를 옮기는 것이 아니라 삭제한다

기존 임베딩 서비스는 서버에 API 호출을 보내고 응답을 기다리는
구조였기 때문에, 성능 개선은 늘 “더 빠른 API”, “더 가까운 리전”,
“더 나은 캐싱” 같은 인프라 최적화 문제였다.
Ternlight처럼 7MB 파일 하나로 임베딩이 끝나면 이 최적화 대상 자체가
사라진다.
브라우저 확장 프로그램이나 정적 사이트가 백엔드 없이 의미 기반
검색을 제공할 수 있다는 것은 단순한 속도 개선이 아니라,
“검색 기능을 배포하려면 서버가 필요하다”는 전제 자체를 무너뜨리는
변화다.
이 변화가 본격화되면 임베딩 API를 파는 사업 모델은 저지연·고정밀이
필요한 상위 구간으로 밀려나고,
저용량 온디바이스 모델이 하위 구간의 수요 대부분을 흡수할 가능성이
있다.

`kamranjon`은 이런 흐름을 정적으로 호스팅된 parquet 파일에
HTTP range 쿼리로 접근하는 portable-hnsw 프로젝트와 연결지으며,
이런 조각들이 모이면 대형 사업자가 통제하지 않는 개방적이고
분산된 검색 생태계로 이어질 수 있다고 내다봤다[^kamranjon].
반면 `CobrastanJorji`와 `rvz`는 정반대 방향의 우려를 제기했다.
웹사이트가 사용자 동의 없이 CPU를 빌려 쇼핑 습관 같은 걸 추론하는
데 쓰거나[^cobrastanjorji], 페이지 방문만으로 모델이 자동
다운로드되는 방식이 악성 코드 배포나 과도한 메모리 점유의
통로가 될 수 있다는 지적이다[^rvz].
서버 인프라가 사라진다는 것은 동시에 사용자 동의 없는 자동
다운로드·실행을 막던 서버 쪽 통제 지점도 함께 사라진다는 뜻이므로,
개방형 생태계에 대한 낙관과 브라우저 리소스 남용에 대한 경계는
같은 구조 변화의 양면이라고 볼 수 있다.

### 삼진 가중치는 예전에 폐기된 저정밀 연산의 재부상이다

곱셈 없이 덧셈·뺄셈만으로 추론한다는 아이디어는
1990년대 고정소수점 DSP나 초기 신경망 하드웨어에서 흔했던 접근과
본질적으로 다르지 않다.
당시에는 부동소수점 연산 자원이 부족해서 채택한 타협이었는데,
지금은 반대로 연산 자원은 넘치지만 모델을 브라우저로 전송하는
대역폭과 다운로드 크기가 새로운 병목이 되면서 같은 저정밀 연산이
다시 유용해졌다.
제약의 성격이 완전히 바뀌었는데도 해법은 회귀했다는 점은,
하드웨어·네트워크 환경이 바뀔 때마다 한때 폐기됐던 저수준 기법이
다른 이유로 재조명되는 패턴이 반복될 수 있음을 보여준다.

### 학습 파이프라인 공개는 검증 부담을 사용자에게 이전하는 결과를 낳는다

`training/`과 `eval/`을 저장소에 통째로 공개한 것은 투명성 측면에서
긍정적이지만, 동시에 “이 모델이 우리 데이터에서도 잘 작동하는지는
직접 재학습하거나 재평가해서 확인하라”는 암묵적 메시지이기도 하다.
파이프라인이 공개되지 않았다면 사용자는 벤치마크 수치를 그대로
믿거나 아예 안 쓰거나 둘 중 하나를 선택했겠지만,
공개된 지금은 “검증 가능하니 필요하면 직접 하라”는 책임 이전이
성립한다.
문제는 대다수 사용자가 실제로 재학습이나 도메인별 재평가를 수행할
여력이 없다는 점이다.
결국 공개된 파이프라인은 소수의 정교한 사용자에게는 실질적 검증
수단이 되지만, 다수의 일반 사용자에게는 “검증은 가능했지만
하지 않은 채로 채택한” 상태를 정당화하는 명분으로만 남을 위험이
있다.
