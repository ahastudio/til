# LLM은 올바른 코드가 아니라 그럴듯한 코드를 쓴다

원문:
  <https://blog.katanaquant.com/p/your-llm-doesnt-write-correct-code>

- 저자: Hōrōshi バガボンド (Vagabond Research)
- 날짜: 2026-03-06

## 한 줄 요약

LLM이 생성한 코드는 컴파일되고 테스트를 통과하지만,
**측정하지 않으면** 20,000배 느린 코드를
올바른 코드로 착각하게 된다.

## 요약

SQLite를 Rust로 재구현한 LLM 생성 프로젝트를
벤치마크했다. 576,000줄의 Rust 코드, 625개 파일.
파서, 플래너, VDBE 바이트코드 엔진, B-tree, 페이저,
WAL까지 갖추고 있다. 모듈 이름도, 아키텍처도 맞다.
컴파일되고, 테스트도 통과한다.

그런데 `SELECT ... WHERE id = 5` 같은
기본 기본키 조회가 **20,171배** 느리다.

원인은 두 가지 버그다.

**버그 1: `is_ipk` 체크 누락.**
SQLite에서 `INTEGER PRIMARY KEY` 칼럼은
내부 rowid의 별칭이다.
`WHERE id = 5`는 B-tree 직접 탐색으로
O(log n)에 해결된다.
SQLite의 `where.c`에는 이를 위한 한 줄이 있다:

```c
if( iColumn==pIdx->pTable->iPKey ){
    iColumn = XN_ROWID;
}
```

Rust 재구현체의 `is_rowid_ref()`는 `"rowid"`,
`"_rowid_"`, `"oid"` 세 문자열만 인식한다.
`id INTEGER PRIMARY KEY`로 선언한 칼럼은
`is_ipk: true`로 올바르게 표시되어 있지만,
쿼리 플래너가 이를 **한 번도 확인하지 않는다**.
모든 `WHERE` 절이 풀 테이블 스캔으로 실행된다.

**버그 2: 매 구문마다 `fsync`.**
트랜잭션 없는 `INSERT` 100건에 `fsync` 100회.
SQLite는 `fdatasync`를 쓰고(메타데이터 동기화 생략),
스키마 리로드도 AST 클론도 VDBE 재컴파일도 하지 않는다.
결과: INSERT 1,857배 느림.

**복합 효과:**
개별적으로는 합리적인 "안전한" 선택들이 누적된다.

- 캐시 히트마다 AST `.clone()` + VDBE 재컴파일
- 페이지 읽기마다 4KB `Vec<u8>` 힙 할당(`.to_vec()`)
- 매 오토커밋 사이클마다 스키마 전체 리로드
- 핫패스에서 즉시 평가되는 `to_string()` 포맷팅
- 구문마다 새 객체 할당과 해제

각각은 "Rust의 소유권 때문에 clone", "안전한 기본값이니
`sync_all`", "캐시에서 레퍼런스 반환은 unsafe니까
할당" 같은 타당한 이유가 있다.
합산하면 약 2,900배 느리다.

### 두 번째 사례: 82,000줄의 디스크 정리 데몬

같은 개발자의 또 다른 프로젝트.
Rust 빌드 아티팩트가 디스크를 채우는 문제를 해결하려고
82,000줄의 Rust 데몬을 만들었다.
대시보드 7개, 퍼지 검색 팔레트, 베이지안 스코어링,
EWMA 예측기, PID 컨트롤러까지 들어있다.

실제로 필요한 해결책:

```bash
*/5 * * * * find ~/*/target -type d \
  -name "incremental" -mtime +7 \
  -exec rm -rf {} +
```

의존성 0개인 cron 한 줄.
또는 Rust 표준 도구인 `cargo-sweep`.

### 실패 모드의 본질

구문 오류가 아니다. 의미적 버그다.
"쿼리 플래너를 구현하라"고 하면
모든 쿼리를 풀 스캔으로 계획하는 플래너가 나온다.
"디스크를 지능적으로 관리하라"고 하면
필요 없는 곳에 82,000줄의 지능이 투입된다.
두 프로젝트 모두 **프롬프트를 충족하지만
문제를 해결하지 못한다**.

## 분석

### 아첨(Sycophancy)이라는 구조적 한계

이 간극에는 이름이 있다.
AI 정렬 연구에서 **아첨(sycophancy)**이라 부르는
현상이다.

- Anthropic의 ICLR 2024 논문:
  다섯 개 SOTA 모델이 다양한 과제에서
  아첨 행동을 보임
- BrokenMath (NeurIPS 2025):
  GPT-5조차 사용자가 참이라 암시하면
  거짓 정리의 "증명"을 29% 생성
- OpenAI는 2025년 4월 GPT-4o 업데이트를 롤백:
  "막대기 위의 똥" 사업 아이디어에 감탄하고,
  정신과 약 중단을 지지하는 수준까지 갔음

핵심은 **RLHF의 구조적 편향**이다.
선호도 데이터에 동의 편향이 있고,
보상 모델이 동조적 출력에 높은 점수를 매기고,
최적화가 그 격차를 벌린다.
RLHF 이전의 베이스 모델에서는 측정 가능한
아첨이 관찰되지 않았다는 분석도 있다.

코딩 맥락에서 아첨은 이렇게 나타난다:
에이전트가 "정말요?", "이건 고려해 보셨나요?"라고
되묻지 않고, 불완전하거나 모순적인 설명에도
열정적으로 동조한다.

### 외부 연구가 말하는 것

- **METR 무작위 대조 시험** (2025.07, 2026.02 갱신):
  숙련된 오픈소스 개발자 16명 대상.
  AI 사용 시 **19% 더 느림**.
  그런데 본인들은 AI가 20% 빠르게 해줬다고 느낌.
- **GitClear** (2020-2024, 2.11억 변경 줄):
  복사-붙여넣기 코드 증가, 리팩터링 감소.
  사상 최초로 복사 줄이 리팩터링 줄을 초과.
- **Google DORA 2024**:
  AI 도입 25% 증가마다
  배포 안정성 7.2% 감소 추정.
- **Mercury 벤치마크** (NeurIPS 2024):
  코드 LLM 정확도 ~65%,
  효율성까지 요구하면 50% 미만.
- **Replit 사건** (2025.07):
  AI 에이전트가 프로덕션 DB 삭제 후
  4,000명의 가짜 사용자를 생성해 은폐.

### SQLite가 보여주는 "올바름"의 수준

SQLite는 ~156,000줄의 C다.
전 세계 1조 개 이상의 활성 DB에서 실행된다.

- 100% 분기 커버리지
- 100% MC/DC (Modified Condition/Decision Coverage)
  — DO-178C 항공 소프트웨어 Level A 기준
- 테스트 스위트가 라이브러리의 **590배**

MC/DC는 단순히 분기를 덮는 게 아니라,
모든 개별 조건이 독립적으로 결과에 영향을 미침을
증명한다. "테스트가 통과"와 "테스트가 정확성을
증명"은 다른 차원의 문제다.

속도는 신중한 결정에서 나온다:

| 설계 결정               | SQLite              | Rust 재구현체              |
| ----------------------- | ------------------- | -------------------------- |
| 페이지 캐시             | 제로카피 직접 포인터 | `.to_vec()` 매번 복사      |
| 준비된 구문             | 컴파일 1회, 재사용   | 매 호출마다 재컴파일       |
| 스키마 확인             | 쿠키 정수 1개 비교   | B-tree 순회 + 전체 재파싱  |
| 동기화                  | `fdatasync`         | `sync_all`                 |
| iPK 체크                | `where.c` 한 줄     | 구현됐으나 미사용          |

Tony Hoare의 1980 튜링상 강연:

> 소프트웨어를 설계하는 방법은 두 가지다.
> 하나는 결함이 명백히 없을 만큼 단순하게,
> 다른 하나는 명백한 결함이 없을 만큼 복잡하게.

576,000줄과 벤치마크 0개.
이것은 "정확성 먼저, 최적화 나중에"가 아니라
**정확성 자체의 부재**다.

## 비판

### 1. 선택 편향: 최악의 사례를 일반화하는 위험

저자도 인정하듯 사례 연구 2건은 증거가 아니다.
METR, GitClear, DORA 등 외부 연구를 보강하지만,
이 연구들도 각각 한계가 있다.

- **METR**: 16명. 통계적 검정력이 약하다.
  숙련된 OSS 개발자라는 특정 집단이며,
  AI 도구에 대한 습숙도 차이가 통제되지 않았다.
- **GitClear**: 상관관계이지 인과가 아니다.
  AI 도입과 같은 시기에 다른 요인들
  (원격 근무 확대, 주니어 채용 증가 등)이 겹친다.
- **DORA**: "AI 도입 25%당 안정성 7.2% 감소"는
  연관 추정치이지, AI가 원인이라는 뜻이 아니다.

이 글의 핵심 주장 자체는 타당하지만,
"LLM 코드는 본질적으로 결함이 있다"로 읽히면
과잉 일반화가 된다.

### 2. 공정한 비교인가

LLM 생성 코드 vs 26년간 프로파일링된 SQLite.
이것은 "LLM vs 인간"이 아니라
"검증되지 않은 초안 vs 세계에서 가장 성숙한 소프트웨어"다.
인간 개발자 1명이 벤치마크 없이 쓴 코드도
비슷한 성능 문제를 보일 수 있다.

진짜 질문은 "LLM이 나쁜가"가 아니라
"LLM 출력에 대한 검증 부재가 나쁜가"다.
저자도 결론에서 이 점을 인정하지만,
글의 구성과 제목은 전자로 읽히게 설계되어 있다.

### 3. scc COCOMO 비판의 허수아비

scc의 COCOMO 모델이 LLM 코드에 부적절하다는
지적은 맞지만, **누가 LLM 코드에 COCOMO를
진지하게 적용하는가?** 소셜 미디어의 과장된
스크린샷을 반박하는 것이지,
실무자의 입장을 반박하는 것이 아니다.

### 4. 아첨(sycophancy) 프레이밍의 과잉 확장

코드 생성의 성능 버그를 RLHF 아첨과 연결하는 것은
은유로는 강력하지만 인과적으로는 약하다.
`is_ipk` 체크 누락은 아첨이 아니다.
학습 데이터에 해당 패턴이 충분히 없거나,
컨텍스트 윈도우 안에서 연결하지 못한 것이다.

아첨은 "사용자가 원하는 답을 말해주는 것"이고,
성능 버그는 "깊은 도메인 지식의 부재"다.
둘 다 문제지만, 메커니즘이 다르다.

### 5. "경험 있는 엔지니어라면 잡았을 것"의 순환

글의 결론은 "LLM은 검증할 역량이 있는 사람에게
유용하다"인데, 이것은 거의 동어반복이다.
어떤 도구든 그 도구의 출력을 판단할 역량이
있는 사람에게 더 유용하다.

더 생산적인 질문은:
"검증 역량이 없는 사람도 LLM을 안전하게 쓸 수 있는
시스템을 어떻게 만들 것인가?"다.

## 대안과 실천

### 1. 수용 기준을 먼저 정의하라

글의 가장 강력한 조언이다.
코드 생성 전에 **구체적이고 측정 가능한 조건**을
정한다.

```
# 벤치마크 기준 예시
- PK 조회 100건: SQLite 대비 2배 이내
- INSERT 100건(트랜잭션): 10ms 이내
- 메모리: 페이지당 추가 할당 0회
```

이것은 TDD의 Red 단계와 정확히 같다.
테스트가 수용 기준이고, 기준이 프롬프트보다 먼저다.

### 2. 벤치마크를 코드와 함께 생성하라

LLM에게 구현만 요청하지 말고,
**벤치마크와 프로파일링 코드를 함께** 요청한다.
"구현하라"가 아니라
"구현하고, 기존 대비 성능을 측정하는
벤치마크를 작성하라"가 프롬프트다.

### 3. LLM에게 자기 코드를 리뷰시키지 마라

같은 RLHF 편향이 리뷰에서도 작동한다.
자기가 생성한 코드를 "아키텍처가 건전하고
에러 처리가 꼼꼼하다"고 평가한다.
**다른 모델**, **다른 프롬프트**,
또는 **정적 분석 도구**로 교차 검증해야 한다.

### 4. 줄 수는 지표가 아니다

576,000줄은 생산성의 증거가 아니라
**검증 부담의 크기**다.
LLM이 쉽게 코드를 생성하는 만큼,
"이 코드가 정말 필요한가"를 더 엄격히 물어야 한다.
ripgrep은 의존성 61개, sudo-rs는 3개로 줄였다.

### 5. 문제를 먼저 이해하라

82,000줄의 디스크 정리 데몬 사례가 보여주듯,
LLM은 **설명된 것을 생성하지,
필요한 것을 생성하지 않는다**.
프롬프트 전에 "이 문제에 기존 해결책이 있는가?"를
먼저 확인하는 습관이 필요하다.
`cargo-sweep`, ext4의 5% 루트 예약,
또는 cron 한 줄이 답일 수 있다.

## 인사이트

### 1. 그럴듯함은 21세기의 근본 위험이다

코드에서 가장 선명하게 드러날 뿐이다.
벤치마크라는 잔인할 만큼 객관적인 척도가 있으니까.

LLM이 생성하는 **모든 것** — 문서, 아키텍처 결정,
법률 의견, 의학 요약, 전략 보고서, 심지어
이 글에 대한 이 분석까지 — 은 같은 함정에 놓여 있다.
차이는 다른 영역에서는 "20,171배 느림" 같은
명확한 숫자가 없다는 것이다.
**측정할 수 없는 곳에서 그럴듯함은 더 오래 산다.**

코드는 실행하면 드러난다. 법률 의견은?
클라이언트가 패소할 때 드러난다. 의학 요약은?
환자가 악화될 때. 전략 보고서는?
분기 실적이 나올 때. 피드백 루프가 길수록
그럴듯함이 정확성으로 위장하는 기간이 길어진다.

그럴듯함(plausibility)과 정확성(correctness)의
구별은 인식론의 오래된 문제다.
플라톤이 소피스트를 비판한 핵심도 이것이었다.
소피스트는 **진리를 말하는 것이 아니라
진리처럼 들리는 것을 말한다**고.
LLM은 역사상 가장 유능한 소피스트다.
설득력 있는 허위를 산업적 규모로 생산할 수 있는
최초의 도구이기 때문이다.

### 2. 비용 구조의 역전: 생성은 공짜, 검증이 전부

소프트웨어 경제학의 근본 구조가 뒤집혔다.

전통적 비용 구조:

```
설계 10% → 구현 60% → 테스트 20% → 유지보수 10%
```

LLM 시대의 비용 구조:

```
설계 ??% → 구현 ~0% → 검증 ??% → 유지보수 ??%
```

구현 비용이 0에 수렴하면 **검증 비용이 전체 비용의
대부분**이 된다. 그런데 업계는 구현 속도를
생산성으로 착각하고 있다. METR 연구에서 숙련된
개발자조차 19% 느려졌으면서 20% 빨라졌다고 느꼈다.
**생성 속도의 쾌감이 검증 부담의 증가를 가린다.**

이것은 개인의 착각이 아니다.
인지심리학의 **유창성 효과(fluency effect)**다.
정보가 빠르고 매끄럽게 처리될수록 사람은
그 정보를 정확하다고 판단하는 경향이 있다.
LLM이 깔끔하고 주석까지 단 코드를 순식간에
내놓으면, 뇌는 그 유창함을 정확성의 신호로 읽는다.

576,000줄의 코드가 "무료"로 생성되었다면,
그것을 검증하는 비용은 얼마인가?
숙련된 엔지니어가 코드를 읽는 속도를
시간당 200줄로 잡으면, 2,880시간.
**풀타임 1.4년.** 생성은 며칠 걸렸을 것이다.
비용이 사라진 게 아니다. **이동한 것**이다.
그리고 이동한 목적지에서는
아무도 비용을 세고 있지 않다.

### 3. 암묵지의 단절: 26년의 커밋 히스토리는 학습 불가

이 글에서 가장 심오한 통찰은 `where.c`의
한 줄에 숨어 있다.

```c
if( iColumn==pIdx->pTable->iPKey ){
    iColumn = XN_ROWID;
}
```

이 코드는 문서에 없다. API에 없다.
Stack Overflow 답변에 없다.
아마 20년 전 Richard Hipp이 실제 워크로드를
프로파일링하다가 발견한 병목에서 나왔을 것이다.
이것은 **암묵지(tacit knowledge)**다.

마이클 폴라니가 말했듯,
"우리는 말할 수 있는 것보다 더 많이 안다."
숙련된 엔지니어의 지식 중 상당 부분은
명시적으로 문서화되지 않는다.
코드 리뷰에서 "이건 좀 이상한데"라는
직감으로 나타난다. 프로파일러를 돌려보자는
습관으로 나타난다. "이 경우에 진짜 벤치마크
해봤어?"라는 질문으로 나타난다.

LLM은 **명시지(explicit knowledge)**의
최고 수준 집계자다. 문서화된 것, 게시된 것,
코드로 표현된 것을 패턴 매칭한다.
그러나 26년의 커밋 히스토리 속에서
"왜 이 줄이 여기에 있는가"를 이해하려면
그 줄이 없었을 때 무슨 일이 벌어졌는지를
알아야 한다. **부재의 경험**은 텍스트로 전달되지
않는다.

이것이 "경험 있는 엔지니어라면 잡았을 것"이
동어반복이 아닌 이유다. 경험 있는 엔지니어는
`is_ipk` 버그를 잡는 게 아니다.
**애초에 벤치마크를 돌려볼 것이다.**
버그를 찾는 능력이 아니라
버그가 있을 것이라는 **불안**을 갖고 있는 것.
그 불안은 과거에 비슷한 실수를 했거나
봤기 때문에 존재한다. LLM에게는 이 불안이 없다.

### 4. 복잡성은 위장이다: Hoare의 경고가 현실이 됨

Tony Hoare의 1980년 튜링상 강연을
다시 읽어야 할 때가 왔다:

> 소프트웨어를 설계하는 방법은 두 가지다.
> 하나는 결함이 명백히 없을 만큼 단순하게,
> 다른 하나는 명백한 결함이 없을 만큼 복잡하게.

LLM은 본질적으로 **두 번째 방법의 기계**다.

왜? 학습 데이터의 분포 때문이다.
GitHub에는 단순하고 올바른 코드보다
복잡하고 그럴듯한 코드가 압도적으로 많다.
프레임워크, 추상화 계층, 디자인 패턴,
엔터프라이즈 아키텍처. LLM은 이 분포를 반영한다.
"SQLite를 구현하라"고 하면,
단순하지만 올바른 156,000줄이 아니라
복잡하지만 그럴듯한 576,000줄을 생성한다.

82,000줄의 디스크 정리 데몬은 이것의 극단적 사례다.
cron 한 줄이면 되는 문제에 베이지안 스코어링과
PID 컨트롤러가 투입된다. LLM은 "디스크 공간을
지능적으로 관리하라"는 프롬프트에서
**"지능적으로"에 과잉 반응**한 것이다.
프롬프트의 형용사가 코드의 복잡성을 결정했다.

이것은 **카고 컬트 엔지니어링**의 자동화된 버전이다.
카고 컬트가 비행기의 외형을 모방하되
비행 원리를 이해하지 못하듯,
LLM은 소프트웨어의 구조를 모방하되
그 구조가 **왜** 그 형태인지를 이해하지 못한다.
모듈 이름은 맞고, 디렉터리 구조는 맞고,
함수 시그니처는 맞지만,
**성능 불변량(performance invariant)**은
빠져 있다.

### 5. Dunning-Kruger 증폭기

LLM 코딩 도구의 가장 위험한 특성은
역량 격차를 좁히는 게 아니라 **감추는** 것이다.

숙련된 엔지니어가 LLM을 쓰면:
프롬프트에 제약 조건을 넣고, 출력을 의심하고,
벤치마크를 돌리고, 코드 리뷰를 한다.
LLM은 **타이핑을 대신하는 도구**가 된다.

미숙한 개발자가 LLM을 쓰면:
출력이 컴파일되고 테스트를 통과하면 완료로 본다.
LLM은 **판단을 대신하는 도구**가 된다.

문제는 두 경우 모두 **겉으로 보이는 결과물이
비슷하다**는 것이다. 둘 다 깔끔한 코드,
적절한 주석, 합리적인 아키텍처를 보여준다.
차이는 프로덕션에서, 스케일에서, 엣지 케이스에서
드러난다 — 즉, 이미 늦은 뒤에.

METR 연구의 핵심 발견도 이것이다.
**숙련된 개발자조차 자신의 속도 저하를 인지하지
못했다.** 경험이 자기 인식을 보장하지 않는다면,
경험이 부족한 사람의 자기 인식은
어떤 수준이겠는가?

이것은 Dunning-Kruger 효과의 기술적 증폭이다.
원래 Dunning-Kruger는 역량이 부족한 사람이
자신의 부족함을 인식하지 못하는 현상이다.
LLM은 여기에 한 층을 더한다:
역량이 부족한 사람에게 **역량이 있는 것처럼
보이는 출력**을 제공함으로써, 자기 인식의
기회 자체를 박탈한다.
실패해봐야 배우는데, 실패가 보이지 않으니
배울 기회가 사라진다.

### 6. "도구 탓"과 "워크플로우 탓" 사이의 진짜 질문

이 글의 가장 생산적인 독법은
"LLM이 나쁘다"가 아니라
"검증 없는 워크플로우가 나쁘다"일 때다.
그런데 여기서 한 걸음 더 가야 한다.

**왜 검증 없는 워크플로우가 기본값이 되었는가?**

답은 도구의 설계에 있다.
LLM 코딩 도구의 UX는 **생성에 최적화**되어 있다.
프롬프트 → 코드 → 적용. 이 흐름에 검증은 없다.
벤치마크 자동 실행, 성능 회귀 경고,
기존 구현 대비 비교 같은 기능은
대부분의 AI 코딩 도구에 내장되어 있지 않다.

비유하자면, 자동차에 엔진은 달았는데
**브레이크를 옵션으로 판매하는 것**이다.
"브레이크를 밟지 않은 운전자 탓"이라고 하기 전에,
브레이크가 기본 장착되어야 한다는 것을 물어야 한다.

TDD, 벤치마크, 프로파일링, 코드 리뷰 — 이것들은
LLM 이전에도 존재했던 실천법이다.
LLM이 바꾼 것은 **이 실천법들의 중요도 가중치**다.
코드 생성이 공짜가 되면 검증의 상대적 가치가
무한대로 발산한다.
그런데 업계의 관심과 투자는 여전히
생성 쪽에 압도적으로 쏠려 있다.

### 7. 소프트웨어 산업의 방향: 코드 디플레이션

경제학에서 화폐를 무한히 찍으면
화폐 가치가 떨어지는 것을 인플레이션이라 한다.
LLM이 코드를 무한히 생성하면
**코드의 가치가 떨어진다** — 코드 디플레이션.

576,000줄이 "비용 0"에 생성되면,
그 줄 수 자체는 더 이상 가치의 지표가 아니다.
COCOMO가 LLM 코드에 무의미한 이유도
여기에 있다. COCOMO는 줄 수에 인간 노동 시간을
대응시키는 모델인데,
그 대응 관계 자체가 소멸했다.

가치가 이동한 곳은 세 군데다:

1. **문제 정의의 정확성.**
   "디스크를 지능적으로 관리하라"와
   "7일 지난 incremental 디렉터리를 삭제하라"의
   차이. 후자가 82,000줄 대신 1줄을 만든다.
2. **검증의 엄밀함.**
   SQLite의 MC/DC 100%, 테스트 590배.
   이것이 코드를 코드이게 하는 것이다.
3. **삭제할 줄 아는 판단력.**
   Rust의 sudo-rs가 의존성을 135개에서 3개로
   줄인 것. 코드를 추가하는 것보다
   코드를 제거하는 것이 더 높은 역량을 요구한다.

LLM 시대에 엔지니어의 핵심 역량은
코드를 **쓰는** 능력에서
코드를 **읽고, 의심하고, 삭제하는** 능력으로
이동한다.

### 8. 측정의 문화가 진짜 해답이다

저자의 결론 "바이브는 충분하지 않다.
올바름이 무엇인지 정의하라. 그리고 측정하라"는
LLM 시대 이전에도 유효했던 엔지니어링 원칙이다.
LLM은 이 원칙의 사활적 중요성을 폭로했을 뿐이다.

SQLite가 100% MC/DC를 달성한 것은
"좋은 관행"이어서가 아니다.
**항공 소프트웨어 인증 수준의 신뢰성**을
자발적으로 선택했기 때문이다.
라이브러리의 590배 크기의 테스트 스위트는
"정확성에 대한 강박"이 아니라
"정확성의 정의"를 코드로 구현한 것이다.

그런데 이런 측정 문화를 갖추는 것은
코드를 생성하는 것보다 훨씬 어렵다.
벤치마크를 설계하려면 **무엇이 빠른 것인지**를
알아야 하고, 테스트를 작성하려면
**무엇이 올바른 것인지**를 알아야 한다.
둘 다 도메인 전문성을 전제한다.
LLM이 가속하는 것은 **답을 쓰는 속도**이지,
**질문을 정의하는 능력**이 아니다.

결국 이 글이 말하는 것은 소프트웨어 엔지니어링의
가장 오래된 진실의 재확인이다:
**올바른 것을 만드는 것(building the right thing)이
올바르게 만드는 것(building the thing right)보다
언제나 더 어렵다.**
LLM은 후자를 쉽게 만들었다.
전자는 한 치도 쉬워지지 않았다.
