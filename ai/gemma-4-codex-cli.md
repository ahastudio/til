# Gemma 4를 Codex CLI에서 로컬 모델로 실행한 결과

## 원문

<https://blog.danielvaughan.com/i-ran-gemma-4-as-a-local-model-in-codex-cli-7fda754dc0d4>

## 요약

한 개발자가 비용, 프라이버시, 안정성을 이유로 Gemma 4 모델을 클라우드 대신
로컬에서 실행하는 실험을 진행했다. 테스트 환경은 Mac M4 Pro와 NVIDIA GB10
두 대다.

핵심 결과는 도구 호출(tool calling) 성공률이다. Gemma 3의 성공률이
6.6%였던 반면, Gemma 4는 86.4%로 급등했다. 로컬 모델이 실질적으로
코딩 에이전트에 사용 가능한 수준에 진입했음을 보여주는 수치다.

속도 대 품질의 트레이드오프도 흥미롭다. Mac은 토큰 생성 속도가 5.1배
빨랐지만 재시도가 필요했고, 최종 완료 시간은 30%만 빨랐다. GB10은
약 7분으로 느렸지만 첫 시도에 성공했다. Mac이 10번의 도구 호출을
필요로 한 것과 대조적이다.

저자의 결론은 하이브리드 접근법이다. 반복적 작업은 로컬에서, 복잡한
작업은 클라우드에서 처리하는 것이 현실적인 전략이라고 권장한다.
셋업 가이드로 Apple Silicon에서는 llama.cpp와 --jinja 플래그, 32K
컨텍스트, KV 캐시 양자화를, NVIDIA에서는 Ollama v0.20.5와 SSH 포트
터널링을 권장한다.

## 분석

### 도구 호출 성공률의 의미

6.6%에서 86.4%로의 도약은 단순한 수치 개선이 아니라 질적 전환이다.
6.6%는 실패가 기본값인 시스템이고, 86.4%는 성공이 기본값인 시스템이다.
에이전틱 코딩에서 도구 호출은 모델이 외부 세계와 상호작용하는 유일한
수단이므로, 이 성공률이 임계점을 넘느냐가 로컬 모델의 실용성을
결정한다. Gemma 4는 그 임계점을 넘었다.

### 속도와 신뢰성의 분리

Mac이 5.1배 빠르지만 최종 완료 시간이 30%만 빠른 이유는 재시도
오버헤드 때문이다. 이는 토큰 생성 속도(tokens/second)가 사용자
체감 생산성과 직선적으로 비례하지 않음을 보여준다. GB10이 느리지만
첫 시도에 성공한다는 것은, 실제 워크플로에서는 추론 속도보다
추론 정확도가 총 소요 시간에 더 큰 영향을 미친다는 뜻이다. GeekNews
댓글에서 "모델 품질이 토큰 속도보다 중요하다"는 지적이 이 데이터와
정확히 일치한다.

### 양자화 수준의 비선형 효과

Q6_K+ 양자화를 권장하고 Q4 수준에서는 성능 저하가 발생한다는 보고는
양자화와 성능의 관계가 선형이 아님을 시사한다. 특히 도구 호출처럼
정확한 JSON 형식 출력이 필요한 작업에서 양자화 손실이 불균등하게
영향을 미친다. 자연어 생성에서는 무시할 수 있는 수준의 정보 손실이
구조화된 출력에서는 파싱 실패로 이어질 수 있다.

### 하이브리드 전략의 현실성

로컬과 클라우드를 작업 유형에 따라 나누는 하이브리드 접근법은 개념적으로
합리적이지만, 실무에서는 작업 경계를 사전에 판단하는 비용이 발생한다.
"반복적 작업"과 "복잡한 작업"의 구분이 항상 명확하지는 않으며, 단순해
보이는 작업이 예상치 못한 복잡성을 드러내는 경우가 빈번하다. 라우팅
로직 자체가 새로운 엔지니어링 과제가 된다.

## 비평

### 단일 벤치마크의 한계

86.4%라는 도구 호출 성공률이 어떤 벤치마크에서 측정되었는지, 작업의
난이도 분포가 어떠한지, 반복 실행 시 분산이 얼마인지에 대한 정보가
부족하다. 한 명의 개발자가 특정 환경에서 수행한 테스트를 일반화하기에는
표본이 작다. 특히 도구 호출의 유형(파일 읽기, 코드 실행, API 호출 등)에
따른 성공률 차이가 있을 수 있으나 분석에서 구분되지 않는다.

### 비용 분석의 부재

클라우드 비용을 절약한다는 동기를 제시하면서도, 하드웨어 비용(M4 Pro,
GB10)의 감가상각, 전력 소비, 셋업에 투입한 시간의 기회비용을 포함한
총소유비용(TCO) 분석은 없다. Mac M4 Pro와 NVIDIA GB10 모두 저렴한
장비가 아니며, 셋업에 llama.cpp 컴파일, 양자화, KV 캐시 설정 등
상당한 기술적 노력이 필요하다. 클라우드 API를 사용했을 때의 실제
비용과의 직접 비교가 없으면 비용 절감 논거가 설득력을 잃는다.

### H100 사용자 피드백의 맥락

GeekNews에서 H100 사용자가 "소규모 빠른 작업에 충분하다"고 평가한
것은 긍정적 신호이면서 동시에 한계의 고백이다. "충분하다"는 표현이
"뛰어나다"가 아닌 점, "소규모"라는 범위 한정이 있는 점을 간과해서는
안 된다. 로컬 모델이 실용적이 된 것은 맞지만, 클라우드 모델을
대체하는 것이 아니라 보완하는 위치에 있음을 시사한다.

## 인사이트

### 도구 호출 임계점과 로컬 AI 생태계의 분기점

Gemma 3에서 4로의 도구 호출 성공률 변화(6.6% → 86.4%)는 점진적
개선이 아니라 위상 전이(phase transition)다. 이 전환이 중요한 이유는
로컬 모델의 사용 가능 영역을 근본적으로 재정의하기 때문이다. 도구
호출 성공률이 낮을 때 로컬 모델은 텍스트 생성 도구에 불과했다. 높은
도구 호출 성공률은 로컬 모델을 에이전트로 격상시킨다.

역사적으로 기술의 위상 전이는 양적 변화가 아닌 임계점 돌파에서
발생했다. 인터넷 전화의 품질이 "통화 가능" 수준을 넘은 순간 PSTN이
위협받기 시작했고, 전기차 배터리의 에너지 밀도가 특정 임계점을 넘은
순간 내연기관의 시장 점유율이 꺾이기 시작했다. Gemma 4의 86.4%가
바로 그 임계점일 수 있다.

이 임계점 돌파의 2차 효과는 로컬 AI 도구 생태계의 활성화다. 도구
호출이 작동하지 않으면 llama.cpp나 Ollama 같은 로컬 추론 엔진 위에
에이전틱 애플리케이션을 구축할 유인이 없다. 임계점을 넘은 순간
개발자들이 로컬 에이전트를 만들기 시작하고, 생태계가 형성되고,
생태계가 추가 최적화를 낳는 양성 피드백 루프가 가동된다. LM Studio로
65536 컨텍스트를 안정적으로 운용한다는 M4 Max 사용자의 보고는 이
루프의 초기 징후다.

3차 효과는 클라우드 AI 제공업체의 가격 전략에 대한 압박이다. 반복적
작업이 로컬로 이동하면 클라우드에 남는 것은 복잡하고 고가치인 작업뿐이다.
이는 클라우드 AI의 ARPU(사용자당 평균 수익)를 높일 수도 있지만,
총 API 호출량의 감소는 규모의 경제에 영향을 미친다.

### 추론 정확도가 추론 속도를 지배하는 구조적 이유

Mac의 5.1배 빠른 토큰 생성이 30%의 완료 시간 우위로 축소되는 현상은
에이전틱 워크플로의 구조적 특성을 드러낸다. 전통적 LLM 사용(채팅,
텍스트 생성)에서는 토큰 속도가 곧 사용자 경험이었다. 에이전틱
워크플로에서는 그렇지 않다. 에이전트가 도구를 호출하고, 결과를 받고,
다음 행동을 결정하는 루프에서 실패와 재시도는 토큰 생성보다 훨씬 큰
시간 비용을 발생시킨다.

이 구조는 컴퓨터 아키텍처에서 오래전에 관찰된 패턴과 동형이다.
CPU 클럭 속도가 높아도 분기 예측(branch prediction) 실패율이 높으면
파이프라인 플러시로 인해 실질 성능이 떨어진다. 에이전틱 워크플로에서
도구 호출 실패는 분기 예측 실패와 같은 역할을 한다. 실패할 때마다
컨텍스트가 오염되고, 재시도 비용이 누적되며, 최악의 경우 무한
루프에 빠진다. GeekNews에서 "불필요한 루프를 피하는 것이 생산성의
핵심"이라는 의견이 정확히 이 메커니즘을 포착한다.

이 통찰의 실무적 함의는 하드웨어 투자의 우선순위를 재정의하는 것이다.
에이전틱 코딩에서 최적의 하드웨어는 가장 빠른 하드웨어가 아니라
가장 정확한 추론을 가능하게 하는 하드웨어다. GB10이 Mac보다 느리지만
첫 시도에 성공하는 패턴이 이를 증명한다. 양자화 수준을 Q4로 낮춰
속도를 높이는 것보다 Q6_K+를 유지하여 정확도를 보존하는 것이
합리적인 이유도 같은 논리다.

이 원리는 개인 하드웨어를 넘어 인프라 설계 전반에 적용된다.
API 게이트웨이에서 로컬과 클라우드를 라우팅할 때, 단순히 지연시간이
아니라 작업 유형별 첫 시도 성공률을 기준으로 라우팅하는 것이
총 소요 시간을 최소화하는 전략이 될 수 있다.

### 로컬 추론의 부상이 AI 권력 구조에 미치는 영향

Gemma 4의 실험 결과를 개인 개발자의 비용 절감 사례로만 읽으면
본질을 놓친다. 로컬 추론이 실용적 수준에 도달한다는 것은 AI 능력의
분산화가 기술적으로 가능해졌다는 뜻이다. 지금까지 에이전틱 코딩은
OpenAI, Anthropic, Google의 API에 의존했다. 이 의존은 기능적인
동시에 정치적이다. API 제공업체가 모델을 제거하거나, 가격을 올리거나,
이용 약관을 변경하면 전체 워크플로가 영향을 받는다.

로컬 추론은 이 의존 관계를 완화한다. 모델 가중치가 사용자의 하드웨어에
있으면 제공업체의 일방적 결정에서 벗어난다. 이는 오픈소스 소프트웨어가
상용 소프트웨어에 대한 대안을 제공한 것과 구조적으로 동일하다. 리눅스가
윈도우를 완전히 대체하지 않았지만, 리눅스의 존재 자체가 마이크로소프트의
가격 정책과 라이선스 전략에 제약을 가했다. 마찬가지로 로컬에서 실행
가능한 Gemma 4가 클라우드 API를 완전히 대체하지 않더라도, 그 존재
자체가 클라우드 제공업체의 행동 범위를 제한한다.

더 넓은 맥락에서, 이 분산화는 AI 규제 논의의 전제를 흔든다. 현재
대부분의 AI 규제 프레임워크는 소수의 대형 제공업체가 AI 능력을
중앙에서 통제한다는 가정에 기반한다. 로컬 추론이 에이전틱 수준에
도달하면 이 가정이 무너진다. 규제 대상이 모델 제공업체에서 하드웨어
제조업체, 양자화 도구 개발자, 오픈소스 모델 배포 플랫폼으로 분산되며,
규제의 복잡도가 급격히 상승한다.

Google이 Gemma를 오픈 가중치로 공개하는 전략적 의미도 이 맥락에서
재해석된다. 이는 단순한 개발자 커뮤니티 지원이 아니라, 클라우드 AI
시장에서의 경쟁 구도를 자신에게 유리하게 재편하려는 시도일 수 있다.
OpenAI와 Anthropic이 폐쇄형 API 모델에 의존하는 반면, Google은
오픈 가중치 모델로 생태계를 구축하면서 동시에 자사 클라우드 인프라의
사용을 촉진하는 이중 전략을 취하고 있다.
