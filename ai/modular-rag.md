# Modular RAG: 파이프라인에서 의사결정 시스템으로

출처:
<https://twitter.com/ihtesham2005/status/2028900581084131420>

참고 논문:
[Retrieval-Augmented Generation for Large Language Models:
A Survey](https://arxiv.org/abs/2312.10997)
(Gao et al., 2024)

## 요약

RAG의 진화를 세 단계로 구분한다.
Naive RAG → Advanced RAG → Modular RAG.

핵심 통찰은 단순하다.
검색(retrieval)을 고정된 파이프라인의 "단계"로
취급하지 말고, 모델이 동적으로 내리는
"의사결정"으로 전환하라는 것이다.

Modular RAG는 실제 프로덕션 태스크에서
30~40% 정확도 향상을 보인다.
복잡한 멀티홉(multi-hop) 질문에서
Advanced RAG 대비 성능 격차가 더 벌어진다.

## 세 가지 RAG 패러다임

| 패러다임     | 구조              | 특징                   |
| ------------ | ----------------- | ---------------------- |
| Naive RAG    | 검색 → 삽입 → 생성 | 고정 파이프라인        |
| Advanced RAG | 전처리 + 후처리   | 쿼리 최적화, 리랭킹   |
| Modular RAG  | 모듈 조합         | 동적 라우팅, 자기 비판 |

### Naive RAG의 한계

검색 → 컨텍스트 삽입(stuff) → 생성.
매번 이 순서를 강제한다.

문제점:

- 정밀도 낮음: 관련 없는 청크가 섞임
- 재현율 낮음: 필요한 문서를 놓침
- 한 번의 잘못된 검색이 전체 답변을 무너뜨림

### Advanced RAG의 개선

검색 전(pre-retrieval)과
검색 후(post-retrieval) 단계를 최적화한다.

- 검색 전: 쿼리 확장(expansion),
  쿼리 재작성(rewriting)
- 검색 후: 리랭킹(reranking) 알고리즘으로
  검색 결과를 정제

파이프라인 구조 자체는 여전히 고정적이다.

### Modular RAG의 전환

고정 파이프라인을 해체하고 독립 모듈을 조합한다.

확장 모듈:

- **Search**: 다양한 소스에서 검색
- **Memory**: 이전 검색·생성 결과를 기억
- **Fusion**: 여러 쿼리 결과를 통합
- **Routing**: 질문 유형에 따라 경로 결정
- **Predict**: 검색 없이 직접 예측 시도
- **Task Adapter**: 태스크별 맞춤 조정

Naive RAG와 Advanced RAG는 Modular RAG의
특수한 경우(고정 모듈 조합)에 해당한다.

## Self-RAG: 자기 비판 메커니즘

Modular RAG의 핵심 기법 중 하나다.
모델이 스스로 출력을 비판하고,
확신도(confidence)가 낮으면 추가 컨텍스트를
검색하도록 한다.

동작 원리:

1. 검색이 필요한지 스스로 판단
2. 검색한 문서가 관련 있는지 평가
3. 생성한 답변이 문서에 근거하는지 검증
4. 확신도가 낮으면 추가 검색 수행

전문 연구자의 사고방식과 일치한다.
분석가는 모든 것을 미리 검색하지 않는다.
무엇을 가져올지, 언제 더 가져올지,
이미 충분한지를 판단한다.

## 핵심 수치

논문에서 보고된 결과:

- **30% 정확도 향상**: 제어된 노이즈를 추가해
  모델이 신호를 필터링하도록 학습시킨 결과
- 복잡한 멀티홉 질문에서 Modular 시스템이
  Advanced RAG를 상회
- 여러 소스를 종합해야 하는 태스크에서
  성능 격차가 더 확대

## Naive RAG vs Modular RAG 비유

> Naive RAG made AI retrieve like a search engine.
> Modular RAG makes it retrieve like a researcher.

검색 엔진은 쿼리를 받으면 무조건 결과를 반환한다.
연구자는 질문에 따라 전략을 바꾼다.
어떤 소스를 볼지, 몇 번 검색할지,
어떤 형식으로 가져올지를 능동적으로 결정한다.

## 관련 기법

| 기법                    | 설명                         |
| ----------------------- | ---------------------------- |
| Self-RAG                | 자기 비판 기반 적응형 검색   |
| FLARE                   | 생성 중 능동적 검색 트리거   |
| RAG-Fusion              | 다중 쿼리 결과 통합(RRF)    |
| RQ-RAG                  | 멀티홉 쿼리 분해            |
| Adaptive Retrieval      | 검색 필요성 동적 판단        |
| Knowledge Graph + RAG   | 구조화된 지식과 결합         |

## 인사이트

### 1) 검색은 "단계"가 아니라 "판단"이다

Modular RAG의 가장 중요한 전환점이다.
기존 RAG는 검색을 파이프라인의 한 단계로 고정했다.
Modular RAG는 검색 자체를 모델의 의사결정으로
전환한다.
검색할지 말지, 몇 번 할지, 어디서 할지,
어떤 형식으로 할지를 모델이 결정한다.

이것은 소프트웨어 아키텍처에서
하드코딩된 절차를 플러그인 시스템으로
전환하는 것과 같다.

### 2) 실패 격리가 품질을 만든다

Naive RAG에서는 한 번의 잘못된 검색이
전체 답변을 오염시킨다.
Modular RAG는 각 모듈이 독립적이므로
한 검색의 실패가 전체를 무너뜨리지 않는다.
마이크로서비스 아키텍처의 장애 격리와
동일한 원리다.

### 3) 노이즈가 오히려 강화 신호가 된다

제어된 노이즈를 학습에 추가하면
모델이 관련 없는 정보를 필터링하는 능력이
향상된다.
30% 정확도 향상의 핵심 메커니즘이다.
이것은 드롭아웃(dropout)이나
데이터 증강(data augmentation)과 같은
정규화 기법의 철학과 맥을 같이한다.

## 적용 아이디어

- 프로덕션 RAG 시스템에 검색 필요성 판단
  모듈을 추가하는 것만으로도 불필요한 검색을
  줄이고 정확도를 높일 수 있다
- Self-RAG 패턴을 도입해 생성 결과의 확신도에
  따라 추가 검색을 트리거하는 피드백 루프를
  구현할 수 있다
- 질문 유형에 따라 다른 검색 전략을 라우팅하는
  것은 비교적 간단하면서도 효과가 크다
