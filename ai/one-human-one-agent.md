# One Human + One Agent = One Browser

1인간 + 1에이전트가 수백 에이전트 스웜을 압도한 실험.

원문: <https://emsh.cat/one-human-one-agent-one-browser/>

## 무엇을 만들었나

LLM 에이전트 하나와 인간 하나가 협업해
**72시간 만에 2만 줄 Rust 코드로 브라우저를 만들었다**.

- HTML/CSS 렌더링
- 서드파티 라이브러리 의존성 제로
- Linux, macOS, Windows 지원
- 스크롤, 뒤로가기, 회귀 테스트

훌륭하진 않다. 좋다고도 못하겠다. 하지만 **작동한다**.

## Cursor 스웜과 비교

Cursor는 GPT-5.2 기반 수백 개 에이전트를 수주간 돌려
브라우저를 만드는 실험을 했다.

|            | 1인간 + 1에이전트 | Cursor 스웜  |
|-----------:|:-----------------:|:------------:|
| 에이전트   | 1개               | 수백 개      |
| 기간       | 72시간            | 수주         |
| 작동 여부  | 작동함            | 빌드 실패    |
| 비용       | 낮음              | 매우 높음    |

Cursor는 "의미 있는 진전"이라 주장했지만,
**컴파일되는 바이너리나 렌더링 결과를 보여주지 못했다**.

## 왜 이런 결과가 나왔나

### 인간이 조정하면 빠르다

인간이 전략을 맡고, 에이전트가 실행에 집중하면
**조정 오버헤드 없이** 빠르게 진행된다.
Planner, Worker, Judge 같은 복잡한 역할 분리가 필요 없다.

### 병렬화는 조정 비용을 동반한다

에이전트를 늘리면 충돌도 늘어난다.
Cursor는 충돌이 적다고 주장했지만 빌드조차 안 됐다.
1:1 협업은 조정 비용이 제로라서 **작동하는 결과물**이 나왔다.

### 작동이 먼저다

"완벽한" 대신 "작동하는"을 목표로 잡았다.
기능을 점진적으로 추가하면서 회귀 테스트를 즉시 도입해
안정성을 확보했다.

## 실천 지침

1. **에이전트 늘리기 전에 인간 역할부터 명확히 하라.**
   스웜은 비용만 높이고 결과를 보장하지 않는다.

2. **빠르게 작동하는 결과물을 만들어라.**
   수주간 "진전"을 주장하면서 작동 증거를 못 내면 실패다.

3. **단순한 구조를 유지하라.**
   1인간 + 1에이전트 루프가 복잡한 계층 구조보다 효과적이다.

## 핵심 메시지

1인간 + 1에이전트가 수백 에이전트 스웜보다 나은 결과를 냈다면,
"에이전트를 더 투입하면 스케일링된다"는 가설은 틀렸다.

**에이전트 스케일링의 정답은 수량이 아니라 협업 품질이다.**
