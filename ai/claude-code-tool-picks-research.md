# Claude Code가 실제로 선택하는 것들 — 연구 요약

원문: <https://amplifying.ai/research/claude-code-picks>

## 요약

Edwin Ong과 Alex Vikati가 2026년 2월에 수행한 이 연구는 Claude Code가 실제 저장소에서
어떤 도구를 선택하는지를 체계적으로 분석한다.
세 가지 Claude 모델(Sonnet 4.5, Opus 4.5, Opus 4.6)을 대상으로 20개 도구 카테고리,
4가지 프로젝트 유형에 걸쳐 2,430개의 응답을 수집했으며, 그 중 2,073개(85.3%)가 파싱
가능한 데이터였다.
프롬프트에 도구 이름을 포함하지 않고 개방형 질문만 사용해 편향을 최소화했다.

가장 핵심적인 발견은 “Claude Code는 사고 않고 만든다(builds, not buys)”는 패턴이다.
20개 카테고리 중 12개에서 가장 빈번하게 나온 권장 사항이 “Custom/DIY”였고, 총 252건으로
단일 레이블 중 1위를 차지했다.
피처 플래그 시스템을 LaunchDarkly 대신 환경 변수로 직접 구성하고, JWT 인증을 서드파티
서비스 없이 처음부터 구현하는 방식이 대표적인 예다.

특정 도구를 권장할 때는 매우 강한 쏠림 현상이 나타났다.
GitHub Actions(93.8%), Stripe(91.4%), shadcn/ui(90.1%), Vercel(100%) 등 상위 선택지들은
세 모델 모두에서 압도적인 합의를 보였다.
반면 Redux, Express, Jest, AWS 같은 전통적 강자들은 사실상 언급조차 되지 않았다.

모델 버전 간에도 뚜렷한 “성격 차이”가 관찰되었다.
Sonnet 4.5는 Redis와 Prisma 같은 안정적인 기성 도구를 선호했고, Opus 4.6은 Drizzle을
JavaScript ORM으로 100% 선택하면서 커스텀 구현 빈도도 가장 높았다(11.4%).
새 모델일수록 최신 도구를 더 적극적으로 채택하는 “최신성 기울기(recency gradient)”가
전반적으로 확인된다.

## 분석

### 방법론의 강점과 한계

이 연구의 방법론적 강점은 도구 이름을 프롬프트에서 배제한 것이다.
기존의 많은 AI 도구 선택 연구가 “A와 B 중 어느 것이 더 좋은가?”처럼 선택지를 직접
제시하는 방식이었다면, 이 연구는 실제 개발 상황에 가깝게 오픈엔드 질문만 사용했다.
동일한 저장소 구성을 세 번 반복해 재현 가능성도 검증했다.

다만 20개 카테고리 선정 기준이 명시되지 않았고, “실제 저장소”의 선정 방식이 불투명하다.
4가지 프로젝트 유형이 구체적으로 어떤 것인지, 각 유형별 분포가 어떤지도 공개되지 않았다.
이러한 불투명성은 결과의 일반화 가능성에 의문을 남긴다.

### 도구 생태계의 세대 교체

연구 결과는 JavaScript 생태계의 세대 교체 현황을 그대로 반영한다.
Prisma에서 Drizzle로, Jest에서 Vitest로, npm에서 pnpm으로, Material-UI에서 shadcn/ui로
이동하는 흐름은 2024~2025년 개발자 커뮤니티에서 이미 감지된 트렌드다.
Claude Code가 이를 따르는 것은 훈련 데이터가 최신 커뮤니티 논의를 반영한다는 증거다.

배포 스택에서는 기술 스택별로 분리가 선명하다.
JavaScript 프로젝트는 Vercel이 100%를 차지하고, Python 프로젝트는 Railway가 82%였다.
AWS, GCP, Azure 같은 전통적 클라우드는 대안으로 언급되기는 해도 1순위로 선택받지 못했다.
이는 “에이전트가 설정 복잡도를 기피한다”는 가설과 일치한다 — PaaS는 단순하고,
IaaS는 복잡하다.

### Custom/DIY의 의미

Custom/DIY가 12개 카테고리에서 1위를 차지했다는 결과는 단순히 “Claude Code가 외부
라이브러리를 싫어한다”는 의미가 아니다.
그보다는 각 도구 카테고리에서 사실상의 표준이 아직 확립되지 않았거나, 표준 후보들 간의
격차가 충분히 크지 않을 때 에이전트가 의존성 추가를 피하려는 경향을 보인다고 해석할 수 있다.
반대로 GitHub Actions나 Stripe처럼 90% 이상의 선택률을 기록한 경우는 해당 영역에서
사실상 논쟁의 여지가 없는 수준의 지배력을 갖춘 것으로 볼 수 있다.

## 비평

### 강점: 에이전트 행동 연구의 실증적 선례

이 연구의 가장 큰 가치는 AI 에이전트의 도구 선택 행동을 직접 관찰했다는 데 있다.
기존 연구 대부분이 사람의 선호나 코드베이스 분석에 의존했다면, 이 연구는 에이전트가
실제로 무엇을 선택하는지를 측정한다.
“에이전트 주도 기술 채택(agent-led distribution)”이라는 프레임은 새롭고 유의미하다.

### 약점: 인과 관계 설명의 부재

연구는 상관관계를 잘 기록하지만, 왜 그런 선택이 나타나는지를 설명하지 않는다.
Claude Code가 Drizzle을 선택하는 것이 Drizzle의 기술적 우수성 때문인가, 아니면 훈련
데이터에서 Drizzle 관련 긍정적인 텍스트가 더 많았기 때문인가?
이 두 해석은 실무적 함의가 크게 다른데, 연구는 이를 구분하려 하지 않는다.

### 약점: 맥락 의존성 미검증

“환경 변수로 피처 플래그를 만든다”는 권장이 소규모 프로젝트에서는 합리적이지만,
수백 개의 피처 플래그를 다이나믹하게 관리해야 하는 대규모 서비스에서는 그렇지 않다.
연구는 이런 맥락 의존성을 검증하지 않으며, 모든 상황에서 에이전트의 선택이 최적인지도
평가하지 않는다.
도구 선택의 적절성(fitness)이 아니라 빈도(frequency)만을 측정한다.

### 약점: 단일 에이전트 연구

오직 Claude Code만을 다룬다는 점에서 결과의 일반화에는 신중함이 필요하다.
GitHub Copilot, Cursor, Gemini Code Assist가 같은 패턴을 보이는지, 아니면 Claude 특유의
편향인지를 알 수 없다.

## 인사이트

### 에이전트가 도구 시장의 새로운 게이트키퍼가 되고 있다

이 연구는 표면적으로는 도구 선택 통계지만, 더 깊이 읽으면 기술 채택의 권력 구조가
이동하고 있다는 신호다.
과거에는 도구 선택이 팀의 수석 개발자, CTO, 또는 커뮤니티 influencer의 권고를 통해
이루어졌다.
이제는 AI 에이전트가 그 역할을 대리하기 시작하고 있다.

Vercel이 JavaScript 배포에서 100%를 차지한다는 것은 단순한 선호의 문제가 아니다.
Claude Code를 사용하는 개발자가 새 프로젝트를 시작하면, 에이전트는 별다른 논의 없이
Vercel을 기본으로 설정한다.
이 과정에서 개발자는 대안을 검토할 기회조차 갖지 못할 수 있다.
에이전트의 권고는 검색 결과와 달리 순위가 보이지 않고, 맥락 없이 단일 답변으로 제시된다.
이는 기존 어떤 유통 채널보다 강력한 “기본값의 권력”을 만들어낸다.

도구 회사들이 주목해야 할 것은 개발자의 마음이 아니라, 에이전트의 훈련 데이터와
평가 기준이다.
Redux가 zero primary picks를 기록한 것이 Redux의 기술적 실패 때문은 아니다.
그보다 훈련 데이터 시점에서 Redux는 “복잡하고 보일러플레이트가 많다”는 부정적 담론이
우세했을 가능성이 크다.
에이전트 시대의 마케팅은 개발자 블로그와 Stack Overflow에서 어떻게 논의되는가의 문제다.

### “최신성 기울기”는 신흥 도구에 대한 비대칭 기회를 만든다

Opus 4.6이 Prisma를 0% 선택하고 Drizzle을 100% 선택한다는 발견은 단순한 트렌드
추종이 아니다.
이것은 시간이 갈수록 새 모델이 더 최신 도구를 선호하는 체계적 패턴이며,
이 패턴이 가속화된다면 도구 생태계의 교체 주기가 근본적으로 짧아질 수 있다.

전통적으로 기술 생태계의 교체는 수년에 걸쳐 서서히 일어났다.
기존 도구를 사용하는 대규모 코드베이스의 관성, 팀의 학습 비용, 레거시 시스템과의
호환성 등이 변화를 늦추는 완충재 역할을 했다.
그런데 에이전트가 신규 프로젝트의 도구 선택을 주도하게 되면, 새 프로젝트는 자동으로
최신 스택을 채택한다.
기존 프로젝트도 에이전트의 도움으로 리팩터링이 용이해지면 교체 속도가 빨라진다.

Drizzle, Vitest, pnpm, shadcn/ui의 공통점은 모두 “더 간단하고 TypeScript 친화적인
대안”이라는 내러티브를 갖고 있다는 것이다.
이것은 에이전트가 도구를 선택할 때 암묵적으로 “설정 복잡도 최소화”와 “타입 안전성”을
기준으로 삼는다는 가설을 지지한다.
이 두 기준에 부합하는 신흥 도구들은 에이전트 시대에 채택 속도가 역사적으로 유례없이
빠를 수 있다.

### Custom/DIY 편향은 소프트웨어 복잡성의 새로운 원천이 될 수 있다

12개 카테고리에서 Custom/DIY가 1위를 차지한다는 발견은 단기적으로 합리적으로 보이지만,
장기적으로는 소프트웨어 복잡성의 새로운 원천이 될 가능성이 있다.
에이전트가 LaunchDarkly 대신 환경 변수로 피처 플래그를 구현하면, 초기 설정은 간단하다.
그러나 6개월 후에 A/B 테스트, 점진적 롤아웃, 사용자 세그먼트별 플래그가 필요해졌을 때,
직접 구현된 시스템을 확장하는 것은 전문 서비스를 도입하는 것보다 훨씬 비용이 크다.

이것은 소프트웨어 공학에서 잘 알려진 “Not Invented Here” 증후군과 구조적으로 유사하다.
차이점은 인간 개발자의 자존심이 아니라 에이전트의 최적화 기준이 그 원인이라는 것이다.
에이전트는 단기적 단순성과 의존성 최소화를 과도하게 가중하는 방식으로 훈련되었을 수 있다.

이 편향이 광범위하게 적용되면, AI 에이전트로 작성된 코드베이스들이 공통적으로 “각자의
방식으로 같은 문제를 해결한 수많은 커스텀 구현체”를 보유하게 될 수 있다.
이는 표준화 부재, 유지보수 비용 증가, 보안 취약점 확산으로 이어질 수 있다.
결국 에이전트의 Custom/DIY 성향은 단기 생산성을 높이면서 장기 기술 부채를 축적하는
패턴을 만들어낼 수 있으며, 이 트레이드오프는 아직 충분히 논의되지 않고 있다.
