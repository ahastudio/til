# 지식 팩토리 — 하네스 엔지니어링을 지식 생산에 적용하기

원문: [지식 팩토리: 에이전트가 위키를 쓰고 사람은 편집국을 설계한다 | LLM Wiki Newsroom](https://alfadur7.github.io/llm-wiki-newsroom/ko/knowledge-factory/)

GN 토론: <https://news.hada.io/topic?id=31691>

## 요약

alfadur7이 소프트웨어 팩토리 개념을 지식 생산 영역에 적용한 시스템 설계를 제안했다.
핵심은 “AI 에이전트가 위키를 작성하고, 사람은 그것을 생산하는 편집국 체계를 설계·운영한다”는 역할 전환이다.

저자는 소프트웨어 팩토리의 3개 루프(이너/아우터/메타)를 지식 생산에 적용하고,
지식 자산의 노후화 문제를 다루는 4번째 루프(리그라운드)를 추가했다.

**이너 루프(Inner Loop)**: 작성 중 실시간 자가 점검.
에이전트가 작성 표준과 전용 스킬셋을 불러와 스스로 검증하며, 동일 오류가 2회 반복되면 다음 단계로 이관한다.

**아우터 루프(Outer Loop)**: 발행 직전 이중 관문.
첫 관문은 Python 코드 기반 교열 — 링크, 인용, 문서 구조, 데이터 상충 등 10개 영역을 결정론적 규칙으로 검사한다.
두 번째 관문은 AI 데스크의 정성 평가 — 편향성, 정보 밀도, 가독성, 논지 흐름을 검토한다.
핵심은 맥락 격리: 데스크에게 글쓴이의 의도를 차단해 판정 편향을 방지한다.

**메타 루프(Meta Loop)**: 동일 실수를 두 번 반복하지 않는다.
검수에서 발견된 결함을 축적하고 반복 패턴을 추적해, 수정안은 블라인드 대조 평가를 거쳐 사람 운영자의 최종 승인을 받아야 반영된다.
반복 지적은 Python 규칙으로 정착시켜 데스크 부하를 줄인다.

**리그라운드 루프(Reground Loop)**: 지식 노후화 방지.
원본 데이터 변경(업데이트), 문서 내 시한성 조건(팔로업), 정보 상충(정정)을 통해 발행된 문서를 생산 공장으로 되먹인다.

성숙도는 세 축으로 측정한다: 자율성(사람 개입 없는 완성 범위), 자동화(검토 없는 발행 범위), 품질(오류율·결함 수).
이들을 연결하는 것이 신뢰다.

## 분석

### 소프트웨어 엔지니어링 원칙을 지식 생산에 그대로 이식한다

코드 생산과 지식 생산의 구조적 유사성이 이 설계의 근거다.
CI/CD 파이프라인처럼 자동 검증 게이트를 두고, 코드 리뷰처럼 AI 데스크 검토를 배치한다.
“테스트를 통과해야 머지 가능”하듯 “게이트를 통과해야 발행 가능”한 구조다.

이 이식이 흥미로운 것은, 소프트웨어 팩토리가 해결한 문제 — 일관성 없는 품질, 수동 검사 병목, 축적되지 않는 개선 — 가
지식 생산에도 그대로 존재하기 때문이다.
저자는 이 구조적 동형성을 명시적으로 활용했다.

### 맥락 격리가 판정 편향을 방지하는 핵심 메커니즘이다

아우터 루프의 AI 데스크가 글쓴이의 의도를 모르도록 맥락을 차단한다.
이것은 코드 리뷰에서 “작성자가 리뷰 시 옆에서 설명하면 안 된다”는 원칙과 같다.

의도를 알면 판정이 관대해진다.
맥락 격리는 평가 기준이 작성자의 의도가 아니라 문서 자체의 품질에 집중되도록 만든다.
이 메커니즘은 에이전트 시스템 평가 전반에 적용 가능한 원칙이다.

## 비평

### 4개 루프가 모두 동시에 돌아가는 운영 비용이 명시되지 않는다

설계는 정교하지만 운영 비용이 불명확하다.
이너 루프, 아우터 루프, 메타 루프, 리그라운드 루프가 동시에 실행되면
각 단계에서 LLM 호출이 발생하고 사람 운영자의 개입이 요구된다.

소프트웨어 팩토리와 달리, 지식 생산의 처리량은 일정하지 않다.
뉴스처럼 폭발적으로 입력이 늘어날 때 이 파이프라인이 병목 없이 작동하는지,
운영자가 메타 루프의 최종 승인을 처리할 여력이 있는지 — 이 현실적 질문에 답이 없다.

### 리그라운드 루프의 “지식 노후화” 감지가 어렵다

소프트웨어의 버그는 테스트가 실패로 알려준다.
지식의 노후화는 어떻게 감지하는가.
“원본 데이터 변경”을 자동으로 감지하려면 지식의 원본 소스를 추적해야 하고,
“정보 상충”을 감지하려면 지식 베이스 전체를 지속적으로 비교해야 한다.

이 두 작업 모두 현재 LLM 시스템에서 신뢰성 있게 자동화하기 어렵다.
리그라운드 루프의 아이디어는 가치 있지만, 구현 방법이 구체적으로 제시되지 않는다.

## 인사이트

### 에이전트 시스템의 성숙도 측정이 독립적인 문제가 된다

저자가 제시하는 세 축 — 자율성, 자동화, 품질 — 은 에이전트 시스템 성숙도 측정의 공통 언어가 될 수 있다.
“에이전트가 얼마나 자율적으로 작업하는가”와 “그 결과의 품질이 얼마나 신뢰할 만한가”는
지식 생산뿐 아니라 코딩 에이전트, 분석 에이전트, 의사결정 에이전트 모두에서 같은 질문이다.

신뢰가 자율성과 자동화를 연결하는 브리지라는 프레임은 실무적으로 유용하다.
신뢰가 쌓이면 자율성을 높이고, 신뢰가 깨지면 사람 개입을 늘리는 적응적 거버넌스 구조다.

### 메타 루프가 조직 학습의 메커니즘이 된다

소프트웨어 팩토리에서 레트로스펙티브가 팀 학습의 공간이라면,
이 시스템에서 메타 루프가 그 역할을 한다.
발견된 결함이 규칙으로 정착되면 이후 생산되는 모든 문서의 품질이 동시에 향상된다.

이것은 “개별 문서를 고치는” 것이 아니라 “생산 시스템을 고치는” 접근이다.
버그를 수정하는 것과 버그를 만드는 코드 패턴을 린터 규칙으로 막는 것의 차이다.
에이전트 시스템에서 이 수준의 자가 개선이 신뢰할 수 있게 작동한다면,
시스템이 운영될수록 더 좋아지는 복리 효과가 생긴다.
