# Optio: AI 코딩 에이전트를 위한 워크플로 오케스트레이터

<https://github.com/jonwiggins/optio>

## 요약

Optio는 AI 코딩 에이전트의 작업을 자동화하는 오픈소스 워크플로 오케스트레이터로,
태스크 수신부터 PR 머지까지의 전체 개발 라이프사이클을 관리한다. GitHub Issues,
Linear, Jira, Notion에서 태스크를 받아 Kubernetes 파드에서 AI 에이전트를 실행하고,
CI 결과와 코드 리뷰 피드백에 따라 자동으로 에이전트를 재실행하여 PR을 완성한다.

6단계 파이프라인으로 동작한다. Intake(태스크 수신) → Provisioning(Kubernetes
파드 할당 및 격리된 git worktree 생성) → Execution(AI 에이전트 실행) →
PR Lifecycle(30초 간격 CI 상태 폴링) → Feedback Loop(CI 실패나 리뷰 코멘트 시
자동 재실행) → Completion(자동 머지 및 이슈 종료).

Claude Code, OpenAI Codex, GitHub Copilot을 에이전트로 지원하며, 리포지토리별로
모델, 프롬프트, 리소스 제한을 설정할 수 있다. Fastify + Next.js + PostgreSQL +
Redis + Kubernetes + BullMQ 기반이며, TypeScript 98.2%로 작성되었다. 실시간
대시보드에서 로그 스트리밍, 파이프라인 시각화, 비용 분석을 제공한다.

## 분석

Optio가 해결하려는 문제는 "AI 에이전트의 마지막 1마일"이다. Claude Code나
Codex가 코드를 생성하는 것은 시작에 불과하다. PR 생성, CI 통과, 코드 리뷰
반영, 머지까지의 과정에서 사람의 개입이 필요한 지점이 많다. Optio는 이
개입 지점들을 자동화한다.

Pod-per-Repo 아키텍처는 핵심적인 설계 결정이다. 각 리포지토리가 전용
Kubernetes 파드를 갖고, 그 안에서 격리된 git worktree를 생성한다. 이는
동시에 여러 태스크를 병렬 처리할 수 있게 하면서도, 리포지토리 간의
간섭을 방지한다. 유휴 파드는 자동 정리된다.

피드백 루프가 가장 야심찬 부분이다. CI가 실패하면 에이전트가 자동으로
오류를 분석하고 수정을 시도한다. 코드 리뷰어가 변경을 요청하면
에이전트가 이를 반영한다. 이것은 "한 번 생성하고 끝"이 아니라
"수렴할 때까지 반복"하는 접근이다.

기술 스택은 현대적이고 잘 짜여 있다. Turborepo 모노레포, Drizzle ORM,
BullMQ 작업 큐, Helm 차트 배포 등은 프로덕션 수준의 인프라를 시사한다.

## 비평

Optio의 비전은 매력적이지만, 현실적 우려가 있다. 첫째, 운영 복잡도가
높다. Kubernetes 클러스터, PostgreSQL, Redis, Docker가 모두 필요하다.
로컬 개발에도 Docker Desktop + Kubernetes + Node.js 22+ + Helm이
전제 조건이다. 이것은 "AI 에이전트를 쉽게 쓰기 위한 도구"의
진입 장벽으로는 상당히 높다.

둘째, 자동 머지의 위험이다. CI 통과와 코드 리뷰 반영 후 자동으로
머지하는 것은, AI가 생성한 코드의 품질을 CI와 자동 리뷰에 전적으로
의존한다는 뜻이다. 테스트 커버리지가 충분하지 않은 프로젝트에서는
자동 머지가 위험할 수 있다.

셋째, 비용 추적 기능이 있다는 것은 이 시스템이 상당한 API 비용을
발생시킬 수 있음을 암시한다. 피드백 루프가 수렴하지 않고 반복되면
비용이 급증할 수 있다. 반복 횟수 제한이나 비용 상한 설정이 중요하다.

넷째, 아직 초기 프로젝트(6포인트)로 실제 프로덕션 사용 사례가
검증되지 않았다. GitHub Issues, Linear, Jira, Notion 통합을 모두
주장하지만, 각 통합의 깊이와 안정성은 미지수다.

## 인사이트

### AI 코딩의 진화: 생성에서 운영으로

AI 코딩 도구의 1세대는 코드 생성이었다(GitHub Copilot의 자동완성).
2세대는 에이전트적 코딩이었다(Claude Code, Cursor의 컨텍스트 인식
코드 작성). Optio는 3세대를 보여준다. AI가 코드를 "생성"하는
것을 넘어 소프트웨어 개발의 "운영"을 수행하는 것이다.

PR 생성, CI 모니터링, 리뷰 대응, 머지는 코딩이 아니라 운영이다.
이것들은 반복적이고 규칙 기반이며 컨텍스트 전환 비용이 높다.
개발자의 시간 중 실제 "생각하고 코드 쓰는" 시간 비율이 낮다는
연구와 일치한다. Optio는 이 운영 오버헤드를 에이전트에게 넘긴다.

그러나 이것은 "개발자가 코드만 쓰면 된다"가 아니라 "개발자가
오케스트레이터를 관리해야 한다"로의 전환이다. 자동화의 레이어가
하나 더 추가되었을 뿐, 복잡도가 사라진 것은 아니다.

### 피드백 루프의 수렴 문제: 언제 멈출 것인가

Optio의 피드백 루프는 "CI 실패 → 에이전트 재실행 → CI 확인"의
순환이다. 이것이 수렴할 것이라는 보장은 없다. AI 에이전트가
한 버그를 고치면서 다른 버그를 만들면, 루프는 발산한다.

이것은 자동화된 시스템의 고전적 문제다. 제어 이론에서 피드백
루프의 안정성은 시스템의 이득(gain)과 지연(delay)에 달려 있다.
AI 에이전트의 "이득"(한 번의 수정으로 얼마나 개선하는가)이
불안정하면, 루프는 진동하거나 발산한다.

실용적으로, 반복 횟수 제한, 비용 상한, 그리고 "사람 개입 요청"
에스컬레이션이 필수다. 완전 자동화의 유혹은 크지만, 현재 AI
에이전트의 신뢰성 수준에서는 "사람이 승인하는 관문"을 제거하는
것이 위험하다. Optio가 이 균형을 어떻게 설계하느냐가 프로젝트의
실용성을 결정할 것이다.

### 개발 워크플로의 표준화 압력

Optio 같은 오케스트레이터가 작동하려면, 개발 워크플로가 표준화되어
있어야 한다. "이슈 → 브랜치 → PR → CI → 리뷰 → 머지"라는
GitHub Flow가 전제된다. 이 흐름을 따르지 않는 팀에서는 Optio가
작동하지 않는다.

이것은 도구가 프로세스를 강제하는 현상이다. Kubernetes가 배포
방식을 표준화했듯, AI 코딩 오케스트레이터는 개발 워크플로를
표준화할 압력을 만든다. 자동화할 수 있으려면 예측 가능해야
하고, 예측 가능하려면 표준화되어야 한다.

긍정적으로 보면 이것은 개발 프로세스의 성숙을 촉진한다.
부정적으로 보면 창의적이거나 탐구적인 개발 방식을 위축시킨다.
모든 작업이 "이슈 → PR" 형태로 환원될 수 있는 것은 아니다.
프로토타이핑, 탐색적 리팩터링, 아키텍처 실험 같은 작업은
이 프레임워크에 잘 맞지 않는다. 오케스트레이터의 채택이
"자동화 가능한 작업"만 하게 되는 왜곡을 만들지 않는지
주의할 필요가 있다.
