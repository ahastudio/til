# Hands-on AI Agents: 2026 Guide

<https://github.com/agulli/atlas-agents>

## 소개

“Hands-on AI Agents”는 2025-2026년 최신 프레임워크와 프로토콜을 사용해
자율(autonomous) 및 반자율(semi-autonomous) AI 에이전트를 구축하는
실습 중심 가이드다.
저자는 “Atlas”라는 이름의 에이전트를 챕터마다 점진적으로 발전시키면서
개념을 코드로 연결한다.

레포지토리는 21개 챕터로 구성되며, 각 챕터에 독립 실행 가능한 Python
예제 코드가 딸려 있다.
주요 기술 스택은 LangGraph, CrewAI, PydanticAI, MCP(Model Context Protocol),
OpenAI Swarm, LangSmith, Phoenix다.

## 분석

### 커리큘럼 구조

챕터 1에서 프레임워크 없이 순수 Python으로 ReAct 루프를 구현하는 것으로
시작하는 방식이 눈에 띈다.
`atlas_v01.py`는 DuckDuckGo 검색과 HTML 파싱을 직접 구현하며, OpenAI
클라이언트만 의존한다.
“추상화를 걷어내고 동작 원리부터 이해하라”는 교육 철학이 뚜렷하다.

이후 챕터는 LangGraph(상태 그래프, 휴먼-인-더-루프), CrewAI(역할 기반
멀티에이전트), PydanticAI(타입 안전 파이프라인) 순으로 추상화 수준을
높여간다.
챕터 8에서 MCP와 A2A(Agent-to-Agent) 프로토콜을 다루고, 챕터 10에서는
Claude Code 자체를 에이전트 런타임으로 사용하는 “Antigravity” 접근을
소개한다.

후반부(챕터 14-21)는 가드레일, 하네스(harness), 상시 구동 에이전트,
관리형 에이전트, 평가, 배포, 캡스톤, 미래 전망으로 이어진다.
에이전트를 “만드는 것”에서 “운영하는 것”으로 관심이 이동하는 흐름이
명확하다.

### 선언적 스킬(Declarative Skill) 모델

챕터 9의 스킬 라이브러리가 특히 주목할 만하다.
각 스킬은 `SKILL.md` 파일에 YAML 프론트매터, 프로세스 단계, 검증
체크리스트를 담고, 런타임은 `skill_loader.py`가 디렉터리를 탐색해
카탈로그를 구성한다.
코드-리뷰, API 설계, git 커밋, 데이터베이스 마이그레이션, 인시던트 대응
등 12개 프로덕션 스킬이 포함되어 있다.
에이전트가 “무엇을 할 수 있는가”를 파일 시스템 구조로 선언하는 방식은
Claude Code의 `CLAUDE.md` / `.claude/skills/` 패턴과 구조적으로 동일하다.

### AGENTS.md — 행동 계약(Behavioral Contract)

챕터 10의 `AGENTS.md`는 에이전트에게 허용·금지 행동, 품질 게이트,
에스컬레이션 조건을 명시한다.
“테스트 없이 완료를 선언하지 마라”, “비밀키를 발견하면 즉시 멈춰라”
같은 규칙은 안전한 에이전틱 코딩의 최소 기준을 제시한다.
이는 단순한 프롬프트 엔지니어링이 아니라 에이전트를 위한 소프트웨어
계약(software contract) 개념에 가깝다.

## 비평

### 강점

실습 코드가 챕터마다 독립적으로 실행 가능하다는 점이 강점이다.
프레임워크 의존성을 최소화한 챕터 1의 접근은 에이전트 동작 원리를
이해하는 데 효과적이다.
MCP, A2A, Claude Code Antigravity처럼 2025-2026년에 등장한 최신 패턴을
포괄적으로 다루는 교재는 드물다.

### 약점

README만으로는 각 챕터의 구체적 내용을 파악하기 어렵다.
`TOC.md`가 존재하지 않거나 아직 미작성 상태로 보인다.
21개 챕터를 모두 소화하려면 상당한 시간이 필요하며, 어떤 챕터를 먼저
읽어야 하는지 우선순위 안내가 없다.

또한 OpenAI 모델을 기본으로 사용하는 예제가 많아, Anthropic 또는 로컬
모델 중심 독자에게는 추가 설정이 필요하다.
챕터 7이 LiteLLM과 Ollama를 다루지만, 전체 예제의 기본값은 OpenAI다.

## 인사이트

### ReAct를 직접 구현해야 하는 이유

에이전트 교육에서 “프레임워크부터 가르치기” 대 “원리부터 가르치기”는
오래된 논쟁이다.
이 책이 챕터 1에서 ReAct 루프를 순수 Python으로 구현하게 하는 선택은
단순한 교육적 친절이 아니다.
현장에서 에이전트를 운영하다 보면 프레임워크 추상화가 숨기는 지점에서
버그가 발생한다.
어떤 단계에서 LLM이 호출되고, 어떤 단계에서 도구가 실행되며, 루프는
언제 종료되는지를 직접 작성해본 사람은 LangGraph의 `StateGraph`가
어떤 문제를 해결하는지 직관적으로 이해한다.

이는 데이터베이스를 배울 때 SQL을 먼저 배우고 ORM을 배우는 순서와
같다.
추상화의 가치는 추상화가 없을 때의 고통을 경험한 뒤에야 체감된다.
에이전트 프레임워크 생태계가 빠르게 바뀌고 있는 지금, 원리를 아는
개발자는 새 프레임워크가 나와도 빠르게 적응할 수 있다.

### 선언적 스킬은 에이전트 운영 체제의 패키지 관리자다

챕터 9의 스킬 모델은 에이전트에게 능력(capability)을 런타임에 플러그인처럼
추가하는 구조다.
이 패턴이 성숙하면 에이전트 생태계에 “스킬 마켓플레이스”가 자연스럽게
등장한다.
개발자가 `npm install`로 라이브러리를 추가하듯, 에이전트가 `skill install
code-review`로 능력을 추가하는 세계가 가시화된다.

현재 Claude Code의 `.claude/skills/` 디렉터리와 Anthropic의 MCP 생태계가
이 방향으로 수렴하고 있다.
스킬 파일이 YAML 프론트매터와 마크다운 프로세스로 구성된다는 점은
에이전트가 자신의 능력을 “읽고 이해할 수 있는” 형식으로 기술된다는
의미다.
에이전트가 스스로 필요한 스킬을 검색하고 로드하는 메타-에이전트
구조도 기술적으로 가능하다.

이 구조의 핵심 긴장은 “스킬의 범위 통제”다.
스킬이 너무 광범위하면 예측 불가능한 부작용이 발생하고, 너무 세분화되면
조합 폭발(combinatorial explosion)이 일어난다.
챕터 14의 가드레일이 이 문제를 다루는 이유가 여기에 있다.

### AGENTS.md는 계약이자 감사 로그의 명세다

소프트웨어 계약(Design by Contract) 개념을 에이전트에 적용하면
`AGENTS.md`는 사전조건(precondition), 사후조건(postcondition), 불변식
(invariant)을 정의하는 문서가 된다.
“테스트 없이 완료 선언 금지”는 사후조건이고, “비밀키 발견 시 즉시
정지”는 불변식이다.

이 문서가 코드베이스에 체크인된다는 사실이 중요하다.
에이전트의 행동 규칙이 버전 관리 시스템에서 추적되고, 규칙 변경이
diff로 남는다.
이는 에이전트의 행동을 소급해서 감사(audit)할 수 있게 하는 기반이다.
규제 산업(금융, 의료)에서 AI 에이전트를 운영하려면 “이 에이전트가
그 시점에 어떤 규칙 아래 동작했는가”를 증명해야 한다.
`AGENTS.md`는 그 증명의 시작점이 될 수 있다.

인간 조직의 직무기술서(job description)와도 유사하다.
직무기술서가 없는 직원은 모든 일을 해도 되거나 아무 일도 안 해도 되는
모호한 상태가 된다.
에이전트에게 `AGENTS.md`가 없다면 같은 모호함이 발생하고, 이는 곧
예측 불가능한 행동으로 이어진다.
에이전트 시스템의 안정성은 코드 품질 못지않게 행동 계약의 명확성에
달려 있다.

### 에이전트 교육의 커리큘럼이 소프트웨어 엔지니어링 교육을 반복하고 있다

이 책의 챕터 순서는 소프트웨어 엔지니어링 교육의 역사적 궤적을
압축해서 반복한다.
프로시저럴 코드(챕터 1) → 라이브러리/프레임워크(챕터 2-8) →
모듈화/스킬(챕터 9) → 멀티 에이전트/협업(챕터 6, 17) → 관찰
가능성(챕터 18) → 배포(챕터 19).
인간이 소프트웨어를 잘 만드는 법을 배우는 데 수십 년이 걸렸고, 에이전트
시스템은 그 교훈을 빠르게 흡수하고 있다.

흥미로운 점은 에이전트 시스템이 인간 개발자의 협업 패턴을 모방한다는
것이다.
CrewAI의 역할 기반 멀티에이전트는 팀의 역할 분담을 모델링하고, 챕터
16의 상시 구동 에이전트는 온콜(on-call) 엔지니어를 모델링한다.
인간 조직의 동작 방식을 이해하는 것이 에이전트 시스템 설계의 중요한
입력이 되고 있다.

역설적으로 이 커리큘럼이 가장 잘 가르치는 것은 “에이전트가 얼마나
빠르게 복잡해지는가”다.
챕터 1에서 200줄이던 코드가 챕터 20의 캡스톤에서는 수십 개 파일로
확장된다.
에이전트 시스템의 복잡성 관리는 전통적 소프트웨어 엔지니어링의 핵심
문제와 동일하며, 따라서 소프트웨어 엔지니어링의 지식이 여전히
유효하다는 결론으로 이어진다.
