# M4 24GB에서 로컬 모델 실행하기

원문: <https://jola.dev/posts/running-local-models-on-m4>

HN 토론: <https://news.ycombinator.com/item?id=48089091> (563점, 172개 댓글)

## 요약

저자는 24GB M4 MacBook Pro에서 일상 개발 작업에 활용할 수 있는 로컬 LLM 설정을
찾아 여러 모델을 실험한 경험을 공유한다. Qwen 3.6 Q3, GPT-OSS 20B,
Devstral Small 24B는 메모리에 올릴 수 있었지만 실용적이지 않았고,
Gemma 4B는 도구 사용에서 한계를 보였다. 최종 선택은 `qwen3.5-9b@q4_k_s`로,
약 40 tok/s 속도, 추론 모드, 도구 사용, 128K 컨텍스트를 동시에 만족했다.

추론에 최적화된 파라미터는 `temperature=0.6, top_p=0.95, top_k=20,
min_p=0.0, presence_penalty=0.0, repetition_penalty=1.0`으로,
LM Studio를 통해 Pi와 OpenCode에 통합했다. 상호작용형 단계별 가이드 워크플로우에서
강점을 보이며 Elixir 린팅 수정 같은 실용적 작업에 성공했다. 다단계 명령 실행이나
복잡한 문제의 지속적 해결에는 한계가 있음을 솔직하게 인정한다.

저자가 꼽은 로컬 모델의 장점은 인터넷 연결 불필요, 구독료 없음, 환경적 이점이다.

## 분석

### 24GB M4의 실질적 한계선

M4 24GB는 로컬 LLM 실험에서 “충분히 재미있지만 충분히 불편한” 지점이다.
메모리에 올릴 수 있는 모델의 범위는 넓지만, 실제 도구 사용과 긴 컨텍스트를 동시에
만족하는 모델 조합을 찾는 데 상당한 시행착오가 필요하다. 저자의 실험은 이 과정을
투명하게 보여준다는 점에서 가치 있다.

[GeekNews 댓글](https://news.hada.io/topic?id=29385)에서 사용자들은 M5 Pro 64GB에서도
여전히 제한적이라는 현실적 평가를 내놓았다. 이는 로컬 모델의 문제가 단순히 “더 큰 메모리”로
해결되지 않음을 시사한다. 모델 품질 자체가 클라우드 프론티어 모델과의 격차를 만들어내며,
메모리 증가는 더 큰 모델을 올릴 수 있게 하지만 이 격차를 완전히 좁히지는 못한다.

HN 토론에서 nl은 9B 모델의 능력을 Sonnet 3.6 수준으로 구체적으로 비교했다.[^nl]
자동완성과 소규모 함수는 처리하지만, 큰 문제를 이해하며 추적하는 데 한계가 있다는 것이다.
이는 저자의 경험과 일치하며, 모델 크기와 실용성 사이의 한계선을 능력 측면에서도 명확히 한다.

### 도구 통합 설정의 복잡성

Pi의 `models.json` 수동 설정과 OpenCode 설정 파일 작성이 요구된다는 점은
로컬 모델 통합의 현재 마찰을 보여준다. OpenAI API 호환 엔드포인트가 표준화되었음에도
클라이언트마다 미묘하게 다른 설정 방식을 요구한다.

[GeekNews 댓글](https://news.hada.io/topic?id=29385)에서는 OpenRouter 같은 API 서비스로
전환하는 사용자가 늘고 있다는 언급도 있었다. 로컬 설정의 복잡성이 “저렴한 클라우드 API”의
편의성 앞에서 경쟁력을 잃는 순간이 온다. 이 지점이 로컬 모델 채택의 핵심 마찰이다.

## 비평

### 낙관적 서사와 실제 한계

저자는 “오프라인, 구독료 없음, 환경 이점”을 강조하지만 이 이점들은 조건부다.
40 tok/s 속도는 단순 텍스트 생성에는 충분하지만, 복잡한 코딩 작업에서 응답을 기다리는
시간 비용을 고려하면 클라우드 API 대비 생산성 손실이 발생할 수 있다.

[GeekNews 댓글](https://news.hada.io/topic?id=29385)에서 핵심 문제는 결국 “돈”이라는
평가가 있었다. 월 20달러짜리 Claude Pro 구독과 비교하면, 전기세와 설정 시간을 감안한
로컬 모델의 실제 비용 효율성은 생각보다 낮을 수 있다. 특히 여러 복잡한 작업을 처리하는
전문 개발자에게는 더욱 그렇다.

### Gemma 4 31B를 언급하지 않은 것

[GeekNews 댓글](https://news.hada.io/topic?id=29385)에서 Gemma 4 31B가 새로운 기준이
되고 있다는 언급이 나왔다. 저자의 실험 시점에서 이 모델이 없었거나 24GB에서 실용적이지
않았을 가능성이 있지만, 로컬 모델 생태계는 매우 빠르게 변화하고 있어 몇 달 전 결론이
이미 구식이 될 수 있다는 한계가 있다.

HN 토론의 soganess는 Gemma 4 31B가 이전 로컬 모델과 달리 "과학 실험"처럼 느껴지지
않는 새로운 기준선이라고 주장했다.[^soganess] M5 Max 128GB 시스템에서 실제로 약 70GB의
RAM을 사용하며 256K 컨텍스트 창을 활용할 수 있다는 구체적인 수치를 공유했다.
32GB 시스템에서는 32K 컨텍스트로 간신히 실행 가능할 수 있다고 추정했다.
이는 24GB가 Gemma 4 31B 실행에 사실상 부족하다는 것을 수치로 뒷받침한다.

## 인사이트

### 40 tok/s가 만드는 새로운 사용 패턴

40 tok/s는 읽는 속도보다 빠르다. 이것이 단순 텍스트 생성에는 충분하지만, 코드 생성이나
복잡한 분석에서는 다르다. 저자가 “상호작용형 워크플로우”에서 강점을 보인다고 한 것이 핵심이다.
로컬 모델이 잘 맞는 작업은 빠른 왕복(quick back-and-forth)이 필요하고, 각 응답이 비교적
짧으며, 실수를 즉시 수정할 수 있는 대화 흐름이다. 반면 긴 출력이 한 번에 필요한 작업에서는
클라우드 모델의 속도 이점이 두드러진다.

이 특성은 로컬 모델의 최적 사용 시나리오를 정의한다. 코드 설명 요청, 짧은 함수 작성,
오류 메시지 해석, 명령어 옵션 확인 같은 “보조 도구” 패턴에서는 로컬 모델이 충분히 실용적이다.
반면 “전체 기능을 처음부터 구현해줘” 같은 장기 작업은 여전히 클라우드 모델의 영역이다.
이 구분을 명확히 인식하고 두 모드를 전략적으로 조합하는 것이 현실적인 접근법이다.

HN 토론에서 PAndreew는 복잡한 코딩 작업에 집중된 비판이 놓치는 부분을 지적했다.[^PAndreew]
화이트칼라 오피스 업무의 상당 부분은 Excel 작업, 파일 처리, 건조한 법적 문서 번역,
이메일 초안, 프레젠테이션 작업이며, 이런 작업들은 30-35B 이상의 모델로 충분히 처리 가능하고
회사 데이터를 외부에 보내지 않는다는 이점도 있다. 로컬 모델의 실용적 활용 영역이 코딩
이외의 오피스 업무로 더 넓게 열려 있다는 관점이다.

### 온도 파라미터와 모델 특성의 관계

저자가 제시한 파라미터 설정(`temperature=0.6, top_p=0.95, top_k=20`)은 Qwen3.5 모델의
추론 모드에 최적화된 값이다. 이 파라미터들은 단순히 “창의성 조절”이 아니라 모델의
추론 체인 품질에 직접 영향을 미친다.

높은 `temperature`는 추론 과정에서 탐색 범위를 넓히지만 일관성을 떨어뜨리고,
낮은 값은 반복적이고 예측 가능한 추론 경로를 강제한다. `top_k=20`으로 후보 토큰을
20개로 제한하는 것은 양자화 모델에서 발생하는 엉뚱한 토큰 선택을 줄이는 데 도움이 된다.

이 파라미터 튜닝이 필요하다는 사실 자체가, 로컬 모델 실용화의 숨겨진 비용을 보여준다.
클라우드 API는 모델 제공자가 이미 최적 파라미터를 설정해두었지만, 로컬 모델은 사용자가
직접 실험을 통해 최적값을 찾아야 한다. 이 과정은 기술적으로 흥미롭지만, 도구로서의
마찰이기도 하다.

### 프라이버시와 성능의 트레이드오프를 다시 생각하다

저자가 언급한 “프라이버시 보호”는 종종 과장되는 로컬 AI의 이점이다. 코드베이스나
개인 문서를 클라우드에 보내는 것에 대한 우려는 실제로 많은 개발자가 느끼는 것이다.
기업 환경에서 소스 코드를 외부 API에 전송하는 것은 법적 또는 정책적 제약이 있을 수 있다.

하지만 이 프라이버시 이점의 실제 가치는 사용 맥락에 따라 크게 다르다. 개인 학습 목적이나
공개 저장소 코드 작업에서는 프라이버시 우려가 크지 않다. 반면 의료, 법률, 금융 분야에서
민감한 데이터를 다루는 경우 로컬 모델의 프라이버시 이점이 성능 손실을 충분히 정당화할 수 있다.

결국 로컬 대 클라우드 선택은 단순한 비용-성능 계산이 아니라, 데이터 민감도, 워크플로우 유형,
네트워크 의존성 허용 수준을 종합한 의사결정이다. 저자의 경험은 이 판단을 위한
하나의 데이터 포인트로서 가치 있다.

HN 토론의 ThomasBb는 모델 품질 외에 추론 엔진 최적화 영역에서 아직 큰 성능 향상이 가능하다고
지적했다.[^ThomasBb] Dflash, MRT, turboquant 같은 기법들이 특정 사용 사례에서 속도를 수배로
끌어올릴 수 있으며, DeepSeek 4 Flash용 모델별 최적화 커널처럼 극적인 결과도 나오고 있다.
이는 동일한 하드웨어에서도 추론 엔진의 발전으로 실용성이 계속 향상될 수 있음을 시사한다.
로컬 AI의 실용성 판단은 모델 품질뿐 아니라 추론 엔진의 성숙도까지 고려해야 한다.
