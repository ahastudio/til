# Writing Code vs. Shipping Code

원문: [Writing Code vs. Shipping Code: Productivity Effects Across Generations of AI Coding Tools | NBER](https://www.nber.org/papers/w35275)

## 요약

이 NBER 워킹 페이퍼는 AI 코딩 도구가 가져오는 생산성 향상이 실제 소프트웨어 출시로 얼마나 이어지는지를 추적한다.
저자들은 GitHub 개발자 10만 명 이상의 데이터와 AI 도구 사용 텔레메트리를 결합해
코드 작성 단계의 개선이 최종 출시 단계까지 전달되는 비율을 측정했다.

연구는 AI 코딩 도구를 세 세대로 구분한다.
자동완성(Autocomplete), 대화형 코딩 에이전트(Interactive coding agents), 자율 코딩 에이전트(Autonomous coding agents)가 그것이다.
각 세대가 커밋 수에 미치는 영향은 순서대로 40%, 140%, 180% 증가로, 도구가 진화할수록 코드 작성 단계의 생산성 향상이 극적으로 커진다.

그러나 이 효과는 파이프라인 하류로 갈수록 크게 약해진다.
프로젝트 생성 수준에서는 누적 효과가 50%로 줄고, 실제 릴리스에서는 30%까지 떨어진다.
자율 에이전트가 커밋을 180% 늘려도 출시 수는 30%밖에 늘지 않는다는 뜻이다.

저자들은 이 현상을 “약한 고리 가설(weak-link hypothesis)”로 설명한다.
AI와 인간 노력 사이의 대체 탄성(elasticity of substitution)을 0.25로 추정했는데,
이 수치는 두 투입 요소가 대체 관계가 아니라 강한 보완 관계에 있음을 뜻한다.
AI가 특정 단계를 빠르게 처리하더라도, 파이프라인에서 인간이 담당하는 병목이 존재하는 한 전체 산출은 그에 맞춰 제한된다.

연구는 네 개의 주요 앱 마켓플레이스 데이터로 검증 작업도 진행했다.
AI 도구 도입 이후 신규 앱 수는 소폭 늘었지만, 앱 전체 사용량 지표는 증가하지 않았다.
코드가 더 많이 작성되고 더 많은 앱이 등록되어도, 실제로 사용되는 소프트웨어의 총량은 별로 달라지지 않았다는 결론이다.

## 분석

### 생산성 지표의 위치가 결론을 결정한다

AI 코딩 도구 연구 대부분은 코드 작성 속도, 코드 수락률, 개발자 만족도 같은 지표를 성과로 삼는다.
이 연구의 핵심 기여는 측정 위치를 파이프라인 끝으로 옮긴 것이다.
커밋이 늘었다는 사실과 릴리스가 늘었다는 사실은 서로 다른 주장이고, 두 주장을 혼동하면 AI 투자 효과를 과대평가하게 된다.

코드 작성이 개발자 업무의 전부가 아니라는 점은 누구나 알고 있다.
그러나 AI가 등장하면서 코드 작성 속도가 유독 부각됐고, 마치 그것이 소프트웨어 출시 속도와 직결되는 것처럼 논의가 흘렀다.
이 논문은 그 가정에 데이터를 들이댄다.
대체 탄성 0.25라는 수치는 AI와 인간 노력이 거의 대체될 수 없음을 뜻하는데,
이는 “코드를 빨리 쓰면 제품이 빨리 나온다”는 직관과 정반대다.

### 약한 고리 가설이 시사하는 병목의 위치

약한 고리 이론은 생산 체인에서 가장 느린 단계가 전체 속도를 결정한다는 논리다.
AI가 코드 작성을 가속하면 병목이 다른 곳으로 이동한다.
코드 리뷰, 테스트, 제품 기획, 배포 인프라, 사용자 온보딩 등이 후보다.

흥미로운 점은 이 병목이 반드시 인간의 “느림” 때문만이 아닐 수 있다는 것이다.
AI가 생성한 코드는 검증이 어렵다.
리뷰어가 같은 시간에 처리할 수 있는 PR 수가 늘지 않으면, 코드를 빨리 만들어도 리뷰 대기열만 쌓인다.
탄성이 낮은 이유 중 하나는 AI 출력의 신뢰도가 충분하지 않아 인간 검증 단계를 건너뛸 수 없기 때문일 수 있다.

### 자율 에이전트 세대와 180% 증가의 의미

커밋 180% 증가라는 수치는 눈에 띄지만, 그 내용을 따져볼 필요가 있다.
자율 에이전트가 만드는 커밋은 자동완성이 돕는 커밋과 성격이 다르다.
에이전트가 많은 코드를 한꺼번에 생성하면, 작은 단위로 쌓인 커밋보다 검토 부담이 클 수 있다.

출시 효과가 30%에 그친다는 결과는 자율 에이전트 세대에서 특히 의미심장하다.
이 세대는 커밋 기여가 가장 크지만, 추가로 생성한 코드 중 출시로 이어지는 비율은 앞 세대보다 낮을 가능성이 있다.
논문이 세대별로 이 전환율을 분해해서 보여주는지는 확인이 필요하지만, 만약 그렇다면 에이전트가 많은 코드를 만들수록 선별 비용도 함께 커진다는 가설을 뒷받침할 것이다.

### 앱 마켓플레이스 데이터가 보완하는 부분

GitHub 데이터만으로는 “만들어진 코드”와 “배포된 소프트웨어” 사이의 간극을 포착하기 어렵다.
앱 마켓플레이스 데이터는 그 간극 너머, 즉 실제 사용자가 선택한 소프트웨어를 본다.
신규 앱 수는 소폭 늘었지만 사용량이 늘지 않았다는 결과는 두 가지로 읽힌다.

첫째, AI 덕분에 더 많은 사람이 앱을 만들 수 있게 됐지만, 시장에서 선택받는 앱의 수는 기술적 생산 능력과 별개로 결정된다.
둘째, 비슷비슷한 앱이 많아지면 파이가 나뉠 뿐 시장 총량은 커지지 않는다.
이 두 번째 독해는 AI 도구가 가져오는 경쟁 구조 변화에 관한 더 넓은 질문으로 이어진다.

## 비평

### 측정한 것이 “출시”인지 “코드 생성”인지 구분이 흐릿하다

논문은 GitHub 커밋을 코드 작성의 지표로, 릴리스를 출시의 지표로 사용한다.
그러나 GitHub에서의 “릴리스”가 무엇을 의미하는지는 맥락에 따라 다르다.
리포지터리에 태그를 붙이는 행위와 실제 사용자에게 소프트웨어를 전달하는 행위는 같지 않다.
오픈소스 프로젝트, 사내 도구, 상업적 SaaS가 모두 같은 “릴리스” 지표로 묶이면, 측정이 실제 출시 활동을 얼마나 반영하는지 불분명해진다.

앱 마켓플레이스 검증이 이 문제를 부분적으로 보완하지만, 네 개의 마켓플레이스가 어디인지, 어떤 기준으로 선택했는지는 초록만으로는 알 수 없다.
모바일 앱 마켓이라면 모바일 개발자 생태계에 특화된 결과이고, 웹 플랫폼이라면 또 다른 편향이 있을 수 있다.
결론의 일반화 가능성은 이 데이터 선택에 크게 의존한다.

### 약한 고리 가설 자체가 아직 논쟁 중인 틀이다

약한 고리 가설은 생산 과정을 일종의 직렬 체인으로 본다.
그러나 현대 소프트웨어 개발은 많은 부분이 병렬로 진행된다.
코드 리뷰, 테스트 자동화, 배포 파이프라인이 동시에 돌아가는 환경에서는 가장 느린 단계가 전체를 막는다는 가정이 항상 성립하지 않는다.

더 중요한 것은 대체 탄성 0.25라는 추정치가 어떤 가정 아래 도출됐는지다.
이 수치가 AI 도구 도입 전후의 패널 데이터에서 나왔다면, 도구가 아직 성숙하지 않은 시기를 포착한 값일 수 있다.
자율 에이전트가 코드 리뷰 자체를 보조하거나, 배포 파이프라인을 직접 구성하는 도구가 확산되면 이 탄성은 달라질 것이다.
논문의 결론이 특정 시기의 기술 수준에 국한될 가능성을 저자들이 얼마나 명시적으로 다뤘는지가 핵심 검증 지점이다.

### 개발자 간 이질성이 평균에 묻힌다

10만 명의 GitHub 개발자를 하나의 표본으로 다룰 때, 그 안의 이질성은 매우 크다.
처음 코드를 배우는 학습자, 개인 프로젝트를 즐기는 취미 개발자, 스타트업에서 제품을 만드는 개발자, 대기업 내 레거시 시스템을 유지하는 엔지니어는
AI 도구를 사용하는 맥락과 출시라는 개념 자체가 완전히 다르다.

자동완성이 커밋을 40% 늘린다는 결과가 어느 집단에서 나오는지에 따라 해석이 달라진다.
취미 개발자에게 커밋이 40% 늘어난 것은 출시와 무관할 수 있고, 스타트업 개발자에게는 실제 제품 속도와 연결될 수 있다.
집단별 분해 없이 평균을 내면, 어느 맥락에서 AI 도구가 실질적인 출시 효과를 낳는지 알기 어렵다.

### 사용량 정체가 공급보다 수요 문제일 수 있다

앱 수가 늘어도 총 사용량이 늘지 않았다는 결과를 두고 논문은 AI 도구의 실질 효과가 제한적이라는 근거로 쓴다.
그러나 이 해석에는 전제가 있다.
총 앱 사용량이 늘지 않은 이유가 소프트웨어 공급 품질의 문제가 아니라, 그냥 사람들이 앱을 더 쓸 이유가 없어서일 수도 있다.
수요가 포화된 시장에서 공급을 아무리 늘려도 소비는 늘지 않는다.
이 대안 설명을 통제하지 않으면, 사용량 데이터는 AI 도구 효과에 관한 근거로 쓰기 어렵다.

## 인사이트

### 측정 위치를 옮기면 다른 문제가 보인다

이 논문이 제기하는 가장 중요한 물음은 “AI가 생산성을 높이는가”가 아니라
“어디서 측정한 생산성인가”다.
코드 작성 속도는 개발자 경험의 일부를 포착하지만, 그것이 제품 출시 속도나 사용자 가치와 어느 정도 연결되는지는 별개의 측정 문제다.

이 구분은 AI 도구만의 문제가 아니다.
과거에도 IDE 자동완성, 고수준 프레임워크, 코드 생성 도구가 등장할 때마다 같은 질문이 반복됐다.
기술이 개발자 손에서 코드가 만들어지는 속도를 높인다 해도, 그 코드가 제품이 되고 사용자에게 가치를 주는 경로는 훨씬 길고 복잡하다.
AI 도구의 의의를 평가하려면, 파이프라인 전체를 단위로 보는 측정 설계가 필요하다.

### 병목은 이동하지, 사라지지 않는다

AI가 코드 작성을 가속하면 파이프라인에서 코드 작성의 비중이 줄어든다.
그 자리에 생기는 병목은 코드 검토, 제품 기획, 사용자 조사, 마케팅 같은 영역이다.
이것들은 AI가 아직 자동화하기 어려운 인간 판단 영역이고, 조직의 구성과 역할 배분과 깊이 연결돼 있다.

역사적으로 비슷한 패턴을 찾자면 산업 생산의 자동화가 좋은 비유다.
조립 공정을 자동화하면 품질 검사, 공급망 관리, 재고 최적화에 새로운 병목이 생겼다.
자동화가 가져온 이득은 새로운 병목을 어떻게 처리하느냐에 따라 실현됐다.
소프트웨어 개발도 마찬가지다.
AI가 코드를 더 많이 만들수록, 그 코드를 검토하고 선별하고 통합하는 역량이 새로운 병목이 된다.

### 앱 마켓 사용량 정체가 가리키는 진짜 질문

AI 도구 덕분에 앱을 만드는 비용이 내려가면, 공급은 늘어난다.
그런데 수요가 같다면, 공급이 늘수록 앱 하나당 평균 사용량은 줄어든다.
이것은 AI가 소프트웨어 시장에서 진입 장벽을 낮추지만, 동시에 경쟁을 심화시킨다는 뜻이다.

더 근본적으로는, 사용자가 원하는 소프트웨어의 종류와 양이 무한하지 않다는 현실이 있다.
AI 도구는 코드 작성 비용을 줄이지만, 어떤 제품을 만들어야 하는지, 어떤 사용자 문제를 해결해야 하는지를 결정하는 비용은 줄이지 못한다.
앱 수가 늘어도 사용량이 늘지 않는 결과는, AI 도구가 아직 그 판단의 영역에 들어오지 못했음을 보여준다.

### 자율 에이전트 시대에 더 중요해지는 인간의 역할

자율 에이전트 세대에서 커밋이 180% 늘어난다면, 개발자는 더 이상 코드를 직접 치는 사람이 아니라
AI가 만든 코드를 평가하고 방향을 정하는 사람이 된다.
이 전환은 지금 당장 진행 중이지만, 그에 맞는 역량과 프로세스는 아직 자리를 잡지 못했다.

코드 리뷰의 의미가 달라진다.
사람이 직접 쓴 코드보다 훨씬 많은 양을, 더 짧은 시간에 검토해야 하고,
AI가 만들어낸 미묘한 오류는 기존 패턴 인식으로 잡기 어렵다.
테스트, 검증, 배포 전략도 이 변화에 맞게 재설계돼야 한다.
대체 탄성 0.25라는 낮은 수치는 지금 그 재설계가 충분히 이뤄지지 않았음을 반영하는 수치로도 읽힌다.
AI 도구의 출시 효과가 제한적인 이유 중 일부는 기술의 한계가 아니라
그 기술을 흡수할 조직과 프로세스가 아직 준비되지 않았기 때문일 수 있다.
