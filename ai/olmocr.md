# olmOCR: AI2의 오픈소스 문서 OCR 툴킷

<https://github.com/allenai/olmocr>

HN 토론: <https://news.ycombinator.com/item?id=43174298> (313점, 40개 댓글)

## 소개

olmOCR은 Allen Institute for AI(AI2)의 AllenNLP 팀이 개발한 오픈소스
문서 OCR 툴킷으로, PDF와 이미지 기반 문서를 비전 언어 모델(VLM)을
활용해 깔끔한 평문·마크다운으로 변환한다. Qwen2.5-VL을 기반으로
한 7B 파라미터 VLM이 핵심 엔진이며, PDF·PNG·JPEG를 입력으로 받아
수식·표·손글씨·복잡한 레이아웃까지 처리하고, 헤더·푸터를 제거하며
자연스러운 읽기 순서를 복원한다고 밝힌다. README는 “백만 페이지당
200달러 미만”이라는 비용 효율성을 핵심 셀링 포인트로 제시하지만,
이 수치가 어떤 하드웨어·조건에서 산출됐는지에 대한 계산 근거는
제시하지 않는다.

자체 벤치마크인 olmOCR-Bench에서 82.4±1.1점을 기록했다고 소개하며,
수학 공식 인식 82.3점, 표 인식 84.9점, 멀티컬럼 레이아웃 47.7점
등 세부 항목별 점수를 함께 제시한다. 이 벤치마크 자체는 AI2가
2025년 5월 v0.1.68 릴리스와 함께 직접 만들어 공개한 것으로,
제3자가 독립적으로 설계한 벤치마크는 아니다. 벤치마크 비교
대상으로는 Mistral OCR API, Marker, MinerU, DeepSeek-OCR,
Nanonets-OCR2-3B, PaddleOCR-VL, Infinity-Parser, Chandra OCR 등이
포함되지만, AWS Textract나 Google Document AI 같은 대형 클라우드
사업자의 상용 OCR 서비스, 혹은 GPT-4o 기반 OCR과의 비교는
포함되지 않는다.

설치는 원격 추론용 경량 설치(`pip install olmocr`), 로컬 GPU용
설치(`pip install olmocr[gpu]`), AI2 자체 클러스터인 Beaker용
설치, 벤치마크 스위트용 설치로 나뉜다. Ubuntu·Debian 환경에서는
poppler-utils와 추가 폰트가 필요하고 Python 3.11을 요구한다.
`olmocr ./localworkspace --markdown --pdfs sample.pdf` 같은
명령으로 로컬에서 단일·다중 PDF를 변환할 수 있고, `--server`
옵션으로 원격 vLLM 서버를 지정해 처리할 수도 있다. 추론 엔진은
과거 sglang을 사용하다가 v0.1.75부터 vLLM 기반 파이프라인으로
전환했는데, README는 이 전환 사실만 언급할 뿐 구체적인 이유
(성능, 안정성, 생태계 지원 등)는 설명하지 않는다. 기본 양자화는
FP8이며, Cirrascale·DeepInfra·Parasail 같은 외부 추론 제공자를
통해 토큰당 요금을 지불하고 쓸 수도 있다.

기술 스택은 Python이 86.5%, Shell이 8.7%를 차지하며, Docker
이미지는 모델을 포함해 약 30GB에 달한다. AWS S3 기반 작업 큐를
활용한 멀티노드 처리와 AI2의 Beaker 클러스터 실행을 지원한다.
저장소에는 툴킷 코드 외에도 GPT-4o 기반 프롬프팅 전략(silver
데이터 생성용), 언어·스팸 필터링, SFT 파인튜닝 코드, GRPO 강화학습
트레이너, 합성 데이터 생성 스크립트가 함께 포함되어 있다. GRPO
훈련은 README에서 사용 사실만 언급되며 구체적인 목표나 보상
함수 설계는 설명되지 않지만, 참조된 논문 제목(“olmOCR 2: Unit
Test Rewards for Document OCR”)은 OCR 결과를 “유닛 테스트”처럼
검증 가능한 형태로 채점하는 방식으로 보상을 설계했음을 시사한다.
라이선스는 Apache 2.0이며, 이 글을 작성하는 시점 기준 GitHub 스타
18.5k, 포크 1.5k를 기록하고 있고 최신 버전은 v0.4.27(2026년
3월)이다. v1 논문(arXiv:2502.18443)과 v2 RL 훈련 논문
(arXiv:2510.19817)이 함께 공개되어 있다.

## 분석

### “유닛 테스트 보상”이라는 발상은 채점 불가능했던 OCR 품질을 검증 가능한 신호로 바꾼 것이다

전통적으로 OCR 품질 평가는 정답 텍스트와의 문자열 유사도(edit
distance 등)에 의존해 왔는데, 이는 레이아웃이 복잡한 문서에서는
“정답”이 하나로 정의되기 어렵다는 근본적 한계를 가진다. olmOCR
2의 논문 제목이 암시하는 “유닛 테스트 보상”이라는 접근은, OCR
결과가 만족해야 할 개별적이고 검증 가능한 속성들(예: 특정 숫자가
특정 표 셀에 존재하는가, 읽기 순서가 논리적인가)을 마치 소프트웨어
유닛 테스트처럼 정의해 강화학습의 보상 신호로 삼는 방식으로
보인다. 이는 정답이 모호한 생성형 태스크에 강화학습을 적용할 때
공통적으로 부딪히는 “보상을 어떻게 설계할 것인가”라는 문제를,
검증 가능한 하위 속성들로 분해해 우회하는 전략이다. 코드 생성
모델을 실제 유닛 테스트 통과 여부로 보상하는 방식이 이미
성공적으로 자리 잡은 것과 유사한 발상을, 텍스트가 아닌 문서
구조라는 다른 도메인에 적용한 것이라 할 수 있다.

### sglang에서 vLLM으로의 조용한 전환은 추론 엔진 생태계의 세력 판도를 보여주는 신호다

README가 이유를 설명하지 않은 채 추론 엔진을 sglang에서 vLLM으로
바꿨다는 사실 자체가 분석 대상이다. 두 프로젝트는 모두 LLM 서빙을
위한 고성능 추론 엔진으로 경쟁해 왔는데, AI2처럼 인프라 자원과
기술 역량을 갖춘 조직이 이미 한 번 채택했던 엔진을 나중에 다른
엔진으로 갈아탄다는 것은 상당한 마이그레이션 비용을 감수했다는
뜻이다. 이런 결정 뒤에는 대개 생태계 성숙도(문서, 커뮤니티,
플러그인), 특정 모델 아키텍처(VLM)에 대한 지원 수준, 또는
안정성·성능 이슈가 있기 마련이다. 설명 없이 이런 전환이 이뤄졌다는
것은, 프로젝트 관리자 입장에서는 사용자에게 왜 바뀌었는지보다
“이제 이렇게 동작한다”는 사실만 전달하면 충분하다고 판단했다는
뜻이며, 이는 오픈소스 인프라 도구에서 흔히 나타나는 설명 부채
(documentation debt)의 한 사례다.

### VLM 기반 OCR이라는 전제 자체가 자명하지 않다

README는 왜 전통적인 OCR 대신 VLM을 써야 하는지 그 필요성을
당연한 전제로 깔고 시작하지만, 이는 자명한 선택이 아니다.
HN에서 한 사용자는 “왜 굳이 이게 AI여야 하는가? OCR은 AI 붐이
있기 전부터 존재했고 대체로 잘 동작한다. 과한 것 같다”고 반문했다
[^xz18r]. 다른 사용자는 아예 “PDF에서 텍스트를 뽑아내는 도구를
기대했지, 신뢰할 수 있는 OCR인 척하는 또 다른 LLM을 기대한 게
아니다”라며 VLM 기반 OCR이라는 카테고리 자체에 의문을
표했다[^zardoz84]. 이에 대한 반론도 있었다. Tesseract 같은 전통
OCR은 텍스트를 통째로 뽑아내지만 읽기 순서를 복원하려는 시도조차
하지 않고, 다이어그램 안의 텍스트 조각까지 섞여 나오는 경우가
많다는 지적과 함께, 특히 학술 논문처럼 수식·표·복잡한 구조가
많은 문서에서는 전통 OCR의 구조 파괴와 수식 인식 실패가
심각하다는 반박이 이어졌다[^rahimnathwani-tesseract]. 즉 “왜
AI가 필요한가”라는 질문에 대한 답은 균일하지 않으며, 문서
유형(정형 텍스트 vs 복잡한 학술·혼합 레이아웃)에 따라 전통
OCR로 충분한 경우와 VLM이 실질적 우위를 갖는 경우가 갈린다는
점을 README는 명시적으로 구분해 설명하지 않는다.

### 상용 클라우드 OCR과의 비교 회피는 벤치마크의 설계된 경쟁 구도를 드러낸다

olmOCR-Bench의 비교 대상이 대부분 오픈소스이거나 VLM 기반 신생
OCR 도구들(Marker, MinerU, DeepSeek-OCR 등)로 구성되고, AWS
Textract나 Google Document AI 같은 이미 수년간 프로덕션에서
검증된 대형 클라우드 OCR 서비스는 빠져 있다는 점은 우연이 아닐
가능성이 높다. 이 벤치마크는 “VLM 기반 오픈소스 OCR 도구들끼리의
경쟁”이라는 카테고리 안에서 olmOCR의 우수성을 보여주도록 설계된
것으로 보이며, 이는 실제로 이 도구를 프로덕션 OCR 파이프라인
대체재로 고려하는 기업 사용자가 가장 궁금해할 비교(기존 상용
서비스 대비 정확도·비용)와는 다른 질문에 답하고 있다는 뜻이다.

## 비평

### 자체 제작 벤치마크에서 자체 모델이 1위를 하는 구조는 독립적 검증이 아니다

olmOCR-Bench는 AI2가 직접 설계하고 공개한 벤치마크이며, olmOCR
자신이 바로 그 벤치마크에서 최고 점수를 받았다고 소개하는 구조다.
벤치마크 설계자와 벤치마크에서 우승한 모델의 개발자가 동일한
조직이라는 사실은, 이 점수가 독립적인 제3자 평가가 아니라는
근본적 한계를 갖는다. 벤치마크의 세부 카테고리(수식, 표,
멀티컬럼 등)를 어떤 기준으로 골랐는지, 평가 문서 세트가 어떻게
수집됐는지에 따라 특정 모델에 유리하게 설계될 여지는 항상 존재한다.
물론 벤치마크가 공개되어 있어 제3자가 재현·검증할 수 있다는 점은
긍정적이지만, README가 이 벤치마크 점수를 마치 객관적 우수성의
증거처럼 제시하는 방식은, 벤치마크 자체의 출처가 갖는 이해상충을
독자가 스스로 알아채야만 파악할 수 있게 되어 있다.

이 우려는 추측이 아니라 실제로 확인됐다. HN에서 경쟁 도구
Marker의 개발자가 직접 이 벤치마크의 구체적인 결함을 지적했다.
처리 속도 비교에서는 Marker의 API 비용과 olmOCR의 로컬 추론
비용을 나란히 비교하는 방식 자체가 공정하지 않으며, 실제로
H100 한 대에서 Marker는 로컬로 초당 20~120페이지를 처리하는
반면 olmOCR은 같은 조건에서 초당 0.4~4페이지에 그쳤다고
밝혔다. 정확도 평가 역시 약 2,000개 문서 중 불투명한 기준으로
추려낸 75개 샘플만으로 승률을 매겼고, 그마저 도구 쌍마다 다른
샘플을 썼다는 문제를 지적했다. Marker 팀이 자체 데이터셋
1,107개 문서에 LLM을 심판으로 써서 다시 평가했을 때는 Marker의
승률이 56%로 나왔다고 밝혔다[^vikp]. 다른 사용자도 “며칠 전에도
이 도구가 정확도가 형편없고, 상당한 텍스트를 누락하며, 알려진
OCR 솔루션보다 한 자릿수 이상 느리고 훨씬 더 많은 에너지를
쓴다는 테스트 결과와 함께 링크된 적이 있지 않았나”라고
지적했다[^kennyblanken]. 즉 자체 벤치마크의 이해상충이라는
구조적 문제가, 실제 경쟁자의 독립적인 재평가에서 구체적인 수치
차이로 드러난 셈이다.

### “백만 페이지당 200달러”라는 핵심 수치가 검증 불가능한 채로 제시된다

이 프로젝트의 가장 강력한 셀링 포인트인 비용 효율성 주장은,
정작 계산에 필요한 어떤 변수도 공개하지 않은 채 결론만 제시된다.
어떤 GPU를 몇 시간 사용했는지, 클라우드 임대 비용인지 자체
하드웨어 상각 비용인지, 페이지당 평균 처리 시간이 얼마나 걸렸는지,
FP8 양자화를 적용한 조건인지 아닌지조차 알 수 없다. 이런 정보
없이는 이 수치를 자신의 환경에 그대로 적용할 수 있는지 판단할
방법이 없다. 특히 문서의 복잡도(스캔 품질, 언어, 레이아웃
복잡도)에 따라 처리 시간과 비용이 크게 달라질 수 있는 OCR
워크로드의 특성을 고려하면, 단일 수치로 제시된 비용 주장은
실제 도입을 검토하는 조직에게 실질적인 판단 근거가 되기 어렵다.

### 실사용 테스트는 VLM 기반 OCR 특유의 환각 문제를 재확인한다

HN에서 여러 문서 유형으로 직접 테스트해 본 사용자는 대출
신청서에서는 필기와 체크박스는 인식했지만 많은 양식 필드를
놓쳤고, 스캔 상태가 좋지 않은 청구서에서는 실제 이름 대신
발음이나 철자가 비슷한 다른 이름을 만들어내는 환각을 관찰했으며,
복잡한 병합 표가 있는 투자 펀드 팩트시트에서는 여러 행과 열을
임의로 누락시켰다고 보고했다[^constantinum]. 다른 사용자는 로컬
Nvidia GPU만으로 동작하고 다단 레이아웃에서도 올바른 읽기 순서로
출력한다는 장점을 인정하면서도, 다이어그램 안에 있는 텍스트는
추출하지 못한다는 한계를 지적했다[^rahimnathwani]. 또 다른
사용자는 이런 멀티모달 기반 “OCR” 도구들의 공통적인 문제로
환각과 비결정적(nondeterministic) 동작을 꼽으며, 100% 정확도가
필요한 “중요한” 문서에는 아직 적합하지 않다고 평가했다
[^chad1n]. README가 제시하는 벤치마크 점수만으로는 드러나지
않는 이런 실패 사례들은, 정량적 점수와 실제 신뢰도 사이에
간극이 있을 수 있음을 보여준다.

### GPU 필수 요구사항이 “저렴함”이라는 프레이밍과 부딪힌다

이 도구는 7B 파라미터 VLM을 기반으로 하며 GPU가 필수적이다.
비용 효율성을 앞세우는 서술과 별개로, 이는 애초에 상당한 초기
인프라 투자나 클라우드 GPU 임대 비용을 감당할 수 있는 조직에게만
“저렴한” 선택지라는 뜻이다. 소규모 팀이나 개인 개발자가 대량
문서를 처리하려 할 때, GPU 인스턴스를 직접 운영하는 것과 AWS
Textract 같은 완전관리형 서비스를 종량제로 쓰는 것 사이의 실질적
총소유비용(TCO) 비교가 없다면, “백만 페이지당 200달러 미만”이라는
숫자는 그 비용을 실제로 감당할 수 있는 인프라를 갖춘 조직에게만
의미 있는 수치다. README는 이 경제성 주장이 어떤 규모, 어떤
조직 유형에 적용 가능한지에 대한 맥락을 제공하지 않는다.

## 인사이트

### 검증 가능한 부분 보상 설계는 강화학습이 “정답이 모호한” 도메인으로 계속 확장되고 있다는 신호다

수학 문제 풀이나 코드 생성처럼 정답이 명확히 검증 가능한 도메인에서
강화학습이 성공을 거둔 이후, 다음 단계는 정답이 모호하거나
연속적인 도메인에 이 방법론을 어떻게 적용할 것인가였다. olmOCR의
“유닛 테스트 보상” 접근은 OCR이라는, 얼핏 정답이 모호해 보이는
도메인에서도 검증 가능한 하위 속성들로 분해하면 강화학습을 적용할
수 있다는 것을 보여주는 사례다. 이는 앞으로 번역, 요약, 이미지
캡셔닝처럼 “정답이 여러 개일 수 있는” 다른 생성형 태스크들에서도,
전체를 하나의 모호한 보상으로 다루는 대신 검증 가능한 부분
속성들로 쪼개 보상을 설계하는 접근이 표준적인 방법론으로 자리
잡을 가능성을 시사한다. 강화학습의 적용 범위가 “정답이 명확한
문제”에서 “정답의 특정 속성만 명확한 문제”로 계속 확장되고 있는
흐름의 연장선이다.

### 문서 OCR의 고품질화는 LLM 학습 데이터 파이프라인의 병목을 해소하려는 움직임의 일부다

AI2가 OCR 도구를 자체 개발하고 지속적으로 투자하는 이유는 단순히
사용자 편의를 위한 것만은 아닐 가능성이 크다. 대규모 언어 모델을
학습시키려면 방대한 양의 텍스트 데이터가 필요한데, 웹에 이미
존재하는 정제된 텍스트의 공급은 점점 한계에 다다르고 있고, PDF나
스캔 문서 형태로만 존재하는 방대한 지식(학술 논문, 법률 문서,
역사적 기록, 정부 문서)은 여전히 기계 판독 가능한 형태로 변환되지
않은 채 남아 있다. olmOCR가 헤더·푸터 제거와 자연스러운 읽기
순서 복원처럼 “LLM 학습 데이터로 쓰기 좋은 형태”에 최적화된
기능을 강조하는 것은, 이 도구가 최종 사용자를 위한 문서 변환기라기보다
AI2 자신을 포함한 LLM 개발자들이 겪고 있는 고품질 학습 데이터
확보 병목을 해소하기 위한 인프라 도구로 설계됐음을 시사한다.
이런 관점에서 보면 olmOCR의 오픈소스 공개는 이타적 기여인 동시에,
전체 오픈소스 LLM 생태계의 데이터 공급망을 확장해 결과적으로
AI2 자신이 속한 개방형 AI 연구 생태계 전체의 경쟁력을 높이는
전략적 행위이기도 하다.

### 자체 벤치마크의 증식은 OCR 성능 비교를 점점 더 불투명하게 만들 위험이 있다

olmOCR-Bench처럼 모델 개발사가 자체적으로 벤치마크를 만들고
공개하는 관행이 OCR 분야 전반에 확산되면, 서로 다른 도구들이
서로 다른 벤치마크에서 각자 1위를 주장하는 상황이 벌어지기 쉽다.
이는 이미지 생성 모델이나 코딩 에이전트 분야에서 이미 관찰되는
패턴과 같다. 각 개발사가 자사에 유리한 벤치마크를 설계하고,
그 벤치마크에서의 우수성을 마케팅에 활용하는 구조가 굳어지면,
정작 최종 사용자는 “내 문서 유형에 어떤 도구가 실제로 가장
정확한가”라는 질문에 답하기 위해 벤치마크 점수가 아니라 자신의
실제 데이터로 직접 비교 테스트를 해야 하는 상황에 놓이게 된다.
공개 벤치마크와 공개 코드 자체는 검증 가능성을 높이는 긍정적
요소이지만, 벤치마크 설계자와 최고 성능 모델 개발자가 계속
동일한 조직으로 수렴하는 한, 벤치마크 점수의 마케팅 가치와
실질적 신뢰도 사이의 괴리는 오히려 커질 수 있다.

### 벤치마크가 포착하지 못하는 좁고 깊은 가치가 VLM 기반 OCR의 진짜 강점일 수 있다

olmOCR-Bench 같은 종합 벤치마크는 표·수식·멀티컬럼처럼 일반적인
문서 구조를 기준으로 점수를 매기지만, VLM 기반 OCR의 실질적
가치는 오히려 이런 범용 카테고리로 포착되지 않는 좁고 특수한
영역에서 드러날 수 있다. HN에서 한 사용자는 라틴어·중세 필사본
서체를 인식하는 AI 비전 도구는 olmOCR이 처음이었다며, 공공
도메인에 속한 중세 서적을 원래의 삽화·레이아웃까지 살려 재출판용
PDF로 옮기는 작업에 쓰고 있다고 소개했다. Google Vision과 o1을
조합한 기존 방법으로는 첫 시도에 제대로 되지 않았던 장식
이니셜 문자까지 문맥을 활용해 올바르게 인식하거나 교정해 냈다는
것이다[^fschuett]. 이는 표준 벤치마크의 점수표에는 애초에
카테고리조차 존재하지 않는 초저빈도·고난도 특수 사용 사례
(역사 문헌 디지털화, 희귀 서체 인식 등)에서, 검증된 종합
벤치마크 순위보다 개별 사용자의 실제 경험담이 도구의 진짜
가치를 훨씬 더 정확하게 보여줄 수 있다는 것을 시사한다. 앞으로
이런 도구들의 가치를 평가할 때는 종합 순위표 못지않게, 특정
틈새 영역에서의 실사용 사례 축적이 중요한 판단 근거가 될
것이다.
