# Karpathy's Claude Coding Notes

Andrej Karpathy의 트윗 (2026년 1월 27일):
<https://twitter.com/karpathy/status/2015883857489522876>

Claude로 몇 주간 코딩한 후 남긴 여러 가지 메모.

**Coding workflow.** LLM 코딩 능력의 최근 향상으로 인해
2025년 11월에는 80% 수동+자동완성 코딩, 20% 에이전트였던 것이
12월에는 80% 에이전트 코딩, 20% 편집+수정으로 바뀌었다.

> I really am mostly programming in English now,
> a bit sheepishly telling the LLM what code to write... in words.

즉, 이제는 대부분 영어로 프로그래밍하고 있다.
약간 쑥스럽게 LLM에게 어떤 코드를 작성할지... 말로 지시하면서.

자존심이 좀 상하지만, 큰 "코드 액션" 단위로 소프트웨어를 다루는 힘은
너무나 유용하다. 특히 적응하고, 설정하고, 사용법을 배우고,
무엇을 할 수 있고 없는지 파악한 후에는 더욱 그렇다.

**지난 20년간 프로그래밍하면서 기본 코딩 워크플로우에서 가장 큰 변화였고,
이것이 불과 몇 주 만에 일어났다.**

**IDEs/agent swarms/fallibility.** "IDE가 더 이상 필요 없다"는 과대광고와
"에이전트 스웜" 과대광고는 현재로서는 과도하다.

모델은 여전히 실수를 한다.
실제로 중요한 코드가 있다면 IDE를 옆에 크게 띄워놓고 매처럼 지켜봐야 한다.

실수의 양상이 변했다:
- 더 이상 단순한 문법 오류가 아니다
- 약간 부주의하고 성급한 주니어 개발자가 할 법한 **미묘한 개념적 오류**
- 가장 흔한 건 당신을 대신해 **잘못된 가정**을 하고 확인 없이 진행하는 것

모델이 하지 않는 것들:
- 혼란을 관리하지 않음
- 명확화를 요청하지 않음
- 불일치를 표면화하지 않음
- 트레이드오프를 제시하지 않음
- 필요할 때 반박하지 않음
- 여전히 약간 아첨하는 경향

모델이 하는 나쁜 습관들:
- 코드와 API를 **과도하게 복잡하게** 만듦
- 추상화를 부풀림
- 죽은 코드를 정리하지 않음
- 1000줄의 비효율적이고 부풀려진 취약한 구조를 구현
  - "이렇게 하면 안 되나요?"라고 하면 "물론이죠!"라며 100줄로 줄임
- 작업과 무관한 주석과 코드를 부작용으로 변경/삭제

`CLAUDE.md`의 지시로 고치려 해도 이런 문제들이 발생한다.
그럼에도 불구하고 여전히 순이익은 크며,
수동 코딩으로 돌아가는 것은 상상하기 어렵다.

현재 워크플로우: 왼쪽에 ghostty 창/탭에서 몇 개의 CC(Claude Code) 세션,
오른쪽에 IDE로 코드 보기 + 수동 편집.

**Tenacity.** 에이전트가 끈질기게 작업하는 것을 보는 건 흥미롭다.
지치지 않고, 낙담하지 않고, 사람이라면 오래전에 포기했을 곳에서 계속 시도한다.

30분 동안 고군분투하다가 결국 성공하는 것을 보면 "AGI를 느끼는" 순간이다.
**체력(stamina)이 작업의 핵심 병목**이며, LLM으로 인해 이것이 극적으로 증가했다.

**Speedups.** LLM 지원의 "속도 향상"을 측정하기 어렵다.
원래 하려던 것은 훨씬 빨라진 느낌이지만, 주요 효과는:

1. 예전에는 코딩할 가치가 없던 것들을 코딩할 수 있음
2. 지식/기술 부족으로 접근 못했던 코드를 다룰 수 있음

속도 향상이기도 하지만, **확장(expansion)**에 더 가깝다.

**Leverage.** LLM은 특정 목표를 달성할 때까지 **루프를 도는 것에 뛰어나다**.
여기서 대부분의 "AGI를 느끼는" 마법이 발생한다.

> Don't tell it what to do, give it success criteria and watch it go.

- 테스트를 먼저 작성하게 하고 통과시키게 하라
- 브라우저 MCP와 루프에 넣어라
- 매우 높은 확률로 정확한 나이브 알고리즘을 먼저 작성하고,
  정확성을 유지하면서 최적화하게 하라
- **명령형에서 선언형으로** 접근 방식을 바꿔
  에이전트가 더 오래 루프를 돌게 하고 레버리지를 얻어라

**Fun.** 에이전트와 함께하면 프로그래밍이 **더 재미있다**.
빈칸 채우기 노가다가 제거되고 창의적인 부분만 남는다.

- 막히거나 갇힌 느낌이 줄어듦 (재미없는 상태)
- 긍정적인 진전을 이룰 방법이 거의 항상 있어서 **용기**가 더 생김

반대 의견도 있다:
LLM 코딩은 **코딩을 좋아하는 엔지니어**와
**빌딩을 좋아하는 엔지니어**를 나눌 것이다.

**Atrophy.** 수동으로 코드를 작성하는 능력이 천천히 퇴화하기 시작했다.

**생성(코드 작성)**과 **판별(코드 읽기)**은 뇌에서 다른 능력이다.
프로그래밍의 작은 문법적 세부 사항들 때문에,
코드 작성은 힘들어도 코드 리뷰는 잘 할 수 있다.

**Slopacolypse.** 2026년은 **슬롭아포칼립스의 해**가 될 것이다.
GitHub, Substack, arXiv, X/Instagram, 일반적으로 모든 디지털 미디어에서.

실제 개선과 함께 AI 과대광고 생산성 극장도 훨씬 더 많이 보게 될 것이다.

**Questions.** 몇 가지 질문들:

- **10X 엔지니어**는 어떻게 되나? 평균과 최대 엔지니어 간 생산성 비율이
  **많이** 커질 가능성이 있다.
- LLM으로 무장하면 **제너럴리스트가 스페셜리스트를 점점 능가**하나?
  LLM은 빈칸 채우기(마이크로)가 대전략(매크로)보다 훨씬 낫다.
- 미래의 LLM 코딩은 어떤 느낌일까?
  스타크래프트? 팩토리오? 음악 연주?
- 사회의 얼마나 많은 부분이 디지털 지식 노동에 병목되어 있나?

**TLDR** LLM 에이전트 능력(특히 Claude & Codex)이
2025년 12월 즈음에 어떤 일관성의 임계값을 넘어
소프트웨어 엔지니어링과 관련 분야에서 **상전이**를 일으켰다.

지능 부분이 갑자기 나머지 모든 것보다 상당히 앞서 느껴진다:
통합(도구, 지식), 새로운 조직 워크플로우의 필요성, 프로세스, 일반적인 확산.

**2026년은 업계가 새로운 능력을 소화하면서 고에너지의 해가 될 것이다.**
