# 데이터 사이언티스트의 역습

원문: <https://hamel.dev/blog/posts/revenge>

## 요약

Hamel Husain이 2026년 3월 26일 발표한 글로,
LLM 시대에 데이터 사이언티스트가 쓸모없어졌다는 서사에 반박한다.
핵심 주장은 "모델 학습은 업무의 대부분이 아니었다"는 것이다.
실험 설계, 지표 측정, 복잡한 시스템 디버깅, 데이터 직접 관찰이
데이터 사이언티스트의 본질적 업무이며
이것은 LLM API로 대체되지 않는다.

글은 AI 시스템의 평가(evaluation) 파이프라인에서
팀들이 반복적으로 빠지는 5가지 함정을 제시한다.
첫째, 일반적 지표(generic metrics)를 쓰면서
실제 실패 모드를 분석하지 않는 것.
둘째, LLM 판정자(judge)를 검증 없이 배포하는 것.
셋째, 합성 테스트 데이터로 실험을 설계하는 것.
넷째, 도메인 전문가가 아닌 사람이 라벨링하는 것.
다섯째, 데이터 직접 관찰을 자동화로 대체하는 것이다.

이 모든 함정은 데이터 사이언스의 기본 실천에 매핑된다.
트레이스 분석은 탐색적 데이터 분석(EDA),
판정자 검증은 모델 평가,
대표 테스트 구성은 실험 설계,
도메인 전문가 라벨링은 데이터 수집,
프로덕션 모니터링은 프로덕션 ML이다.
"항상 데이터를 직접 보라"가 글의 핵심 처방이다.

OpenAI의 Codex 하니스 엔지니어링,
Andrej Karpathy의 auto-research 프로젝트를 근거로
자율 AI 시스템일수록 관측 가능성 인프라(로그, 지표, 트레이스)가
필수적이며, 이것이 데이터 사이언스 역량이라고 주장한다.

## 분석

### "하니스가 곧 데이터 사이언스"라는 핵심 논제

글의 가장 강력한 논거는 OpenAI Codex와 Karpathy의 사례다.
자율 에이전트가 스스로 최적화하려면
검증 지표에 대해 반복적으로 측정·평가하는 하니스가 필요하고,
이 하니스를 설계하는 것이 데이터 사이언스라는 연결이다.
에이전트가 똑똑해질수록 하니스의 중요성이 올라가므로
데이터 사이언스의 수요도 함께 증가한다는 논리 구조다.

### 5가지 함정의 체계성

각 함정이 독립적이지 않고 인과적으로 연결된다.
일반적 지표를 쓰면(1번) → 판정자 검증을 건너뛰고(2번) →
합성 데이터로 실험하고(3번) → 비전문가가 라벨링하고(4번) →
결국 자동화에 의존하게 된다(5번).
이 연쇄는 "데이터를 직접 보지 않는" 한 가지 근본 원인에서 파생되며,
글의 구조가 이 점을 효과적으로 드러낸다.

### GeekNews 댓글의 반론

GeekNews에서는 회의적 시각이 우세했다.
"데이터 사이언스 역량이 필요하다는 것이
데이터 사이언티스트 직무의 지속을 보장하지 않는다"는 반론이 핵심이다.
엔지니어가 평가 실천을 빠르게 습득할 수 있다면
별도 직무가 아니라 엔지니어링 역량으로 흡수될 수 있다.

## 비평

### "역량은 필요하다"와 "직무는 살아남는다"는 다른 주장이다

글의 논증은 "데이터 사이언스 역량이 필수적"이라는 것이지
"데이터 사이언티스트라는 직무가 유지된다"는 것이 아니다.
그러나 제목("역습")과 톤은 후자를 암시한다.
실험 설계, 지표 설정, 데이터 관찰이 중요하더라도
그것이 독립 직무로 남을지, 엔지니어링 팀에 흡수될지는
별개의 질문이며, 글은 이 구분을 흐리고 있다.

### 5가지 함정이 "데이터 사이언스 고유"인지 의문

제시된 함정들은 소프트웨어 엔지니어링의 테스트 원칙과
상당 부분 겹친다.
"대표적 테스트 데이터 구성"은 테스트 엔지니어링,
"판정자 검증"은 테스트 오라클 문제,
"프로덕션 모니터링"은 SRE의 영역이기도 하다.
이를 데이터 사이언스의 고유 영역으로 프레이밍하는 것은
기존 엔지니어링 실천과의 겹침을 과소평가한다.

### 구체적 사례의 깊이 부족

OpenAI Codex와 Karpathy 프로젝트를 인용하지만
실제로 데이터 사이언티스트가 그 하니스를 설계했는지,
엔지니어가 했는지에 대한 구체적 증거가 없다.
"이런 류의 작업이 필요하다 → 이것은 데이터 사이언스다"라는
연결이 순환 논증에 가깝다.

## 인사이트

### "평가 엔지니어링"이 AI 시대의 새로운 핵심 역량으로 부상한다

Husain의 글이 정확히 짚은 것은
AI 시스템에서 "무엇을 만드느냐"보다
"어떻게 측정하느냐"가 더 어려워졌다는 점이다.
LLM이 코드를 생성하고 텍스트를 요약하는 것은 쉬워졌지만,
그 결과물이 "충분히 좋은지"를 판단하는 것은
오히려 더 어려워졌다.

이는 "평가 엔지니어링(Evaluation Engineering)"이라는
새로운 전문 영역의 탄생을 시사한다.
전통적 QA가 결정론적 시스템의 기능 검증이었다면,
평가 엔지니어링은 확률적 시스템의 품질 보증이다.
데이터 사이언스, 소프트웨어 테스팅, 도메인 전문성이
교차하는 지점에 이 역할이 위치한다.

직무명이 "데이터 사이언티스트"로 남을지는 불확실하지만,
이 역량 조합에 대한 수요는 확실히 증가한다.
Husain의 진짜 통찰은 직무의 생존이 아니라
이 역량 조합의 필수성이다.

### "항상 데이터를 보라"는 AI 시대에 역설적으로 더 어려워진다

Husain의 핵심 처방은 "항상 데이터를 직접 보라"인데,
AI 시스템의 규모가 커질수록 이것이 물리적으로 불가능해진다.
하루에 수만 건의 에이전트 트레이스가 생성되는 시스템에서
사람이 직접 데이터를 관찰하는 것은
샘플링 전략 없이는 실행 불가능하다.

이 역설이 만들어내는 수요는
"어떤 데이터를 봐야 하는지"를 결정하는 시스템이다.
이상 탐지, 능동 학습(active learning), 대표 샘플 추출 등
"관찰해야 할 데이터를 선별하는 자동화"가 필요해진다.
데이터 관찰을 자동화하지 말라는 Husain의 조언은 옳지만,
관찰 대상의 선별은 자동화해야 한다.
이 구분이 실무에서 가장 중요한 설계 결정이 된다.

### Criteria Drift는 AI 제품의 숨은 품질 저하 메커니즘이다

글에서 인용된 Shreya Shankar의 "criteria drift" 연구는
AI 제품에서 가장 과소평가된 문제 중 하나를 짚는다.
사용자가 원하는 것이 시간에 따라 변하고,
평가 기준도 암묵적으로 이동하는 현상이다.

전통적 소프트웨어에서는 요구사항이 명시적으로 변경되지만,
AI 제품에서는 "좋은 결과"의 기준 자체가 사용하면서 바뀐다.
초기에 "문법적으로 정확한 요약"이면 충분했던 것이
시간이 지나면 "핵심 인사이트를 포착하는 요약"으로 기대가 올라간다.
이 이동을 감지하지 못하면 모델은 동일하지만
사용자 만족도는 떨어지는 "보이지 않는 품질 저하"가 발생한다.

Husain이 강조하는 "도메인 전문가 라벨링"은
이 drift를 지속적으로 캘리브레이션하는 메커니즘이다.
라벨링 과정 자체가 "현재 기준에서 무엇이 좋은 결과인지"를
재정의하는 행위이기 때문이다.
