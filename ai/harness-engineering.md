## 한 줄 요약

2025년이 AI 에이전트가 코드를 쓸 수 있음을 증명한 해였다면,
2026년은 **에이전트가 아니라 하네스가 어려운 부분**이라는
걸 깨달은 해다.

## 요약

### 하네스 엔지니어링이란

"하네스(harness)"는 마구(馬具)에서 온 은유다.
고삐, 안장, 재갈 — 강력하지만 예측 불가능한 동물을
올바른 방향으로 이끄는 장비 전체를 가리킨다.

- **말** = AI 모델. 빠르고 강력하지만 혼자서는 어디로
  갈지 모른다.
- **하네스** = 인프라. 제약, 가드레일, 피드백 루프가
  모델의 힘을 생산적으로 채널링한다.
- **기수** = 인간 엔지니어. 방향을 제시하지, 직접 뛰지
  않는다.

하네스 없는 AI 에이전트는 탁 트인 벌판의 서러브레드다.
빠르고 인상적이지만, 실제로 쓸모는 없다.

공식적 정의: 하네스 엔지니어링은 다음을 설계·구현하는
분야다.

1. 에이전트가 할 수 있는 것을 **제약**한다
   (아키텍처 경계, 의존성 규칙).
2. 에이전트가 해야 할 것을 **알려준다**
   (컨텍스트 엔지니어링, 문서).
3. 에이전트가 올바르게 했는지 **검증**한다
   (테스트, 린팅, CI 검증).
4. 에이전트가 틀렸을 때 **교정**한다
   (피드백 루프, 자기 수리 메커니즘).

Martin Fowler는 이를 "AI 에이전트를 제어하는 도구와
관행"이라 설명했지만, 단순한 안전장치가 아니다.
**좋은 하네스는 에이전트를 더 제어하는 게 아니라
더 유능하게 만든다.**

### 왜 지금 중요한가

#### 모델은 상품, 하네스가 해자(moat)

LangChain이 결정적으로 증명했다.
코딩 에이전트의 Terminal Bench 2.0 점수가
52.8%에서 66.5%로 — Top 30에서 Top 5로 뛰었다.
**모델은 그대로, 하네스만 바꿨다.**

| 변경 사항               | 내용                                   | 효과                               |
| ----------------------- | -------------------------------------- | ---------------------------------- |
| 자기 검증 루프           | 완료 전 체크리스트 미들웨어 추가       | 제출 전 오류 포착                   |
| 컨텍스트 엔지니어링     | 시작 시 디렉토리 구조 매핑             | 에이전트가 코드베이스를 즉시 파악   |
| 루프 감지               | 반복 파일 편집 추적                     | "둠 루프" 방지                     |
| 리즈닝 샌드위치         | 계획/검증은 고추론, 구현은 중추론 배분 | 시간 예산 내 품질 향상             |

같은 모델. 다른 하네스. **극적으로 다른 결과.**

#### OpenAI의 100만 줄 증거

OpenAI Codex 팀의 실험:

- 5개월 개발
- 최종 산출물 100만 줄 이상
- **인간이 직접 작성한 코드 0줄** — 전부 Codex 에이전트 생산
- 인간 대비 약 1/10 기간
- 내부 일일 사용자와 외부 알파 테스터 보유
- 배포, 장애, 수정 — 모두 하네스 안에서 에이전트가 수행

엔지니어의 역할? **하네스 설계, 의도 명세, 피드백 제공.**
코드 작성이 아니다.

### 하네스 엔지니어링의 3대 기둥

#### 1. 컨텍스트 엔지니어링

에이전트가 적절한 시점에 적절한 정보를 갖도록 보장한다.

**정적 컨텍스트:**

- 리포지토리 로컬 문서 (아키텍처 명세, API 계약, 스타일 가이드)
- AGENTS.md / CLAUDE.md — 프로젝트별 규칙 인코딩
- 린터로 검증되는 상호 연결된 설계 문서

**동적 컨텍스트:**

- 관측성 데이터 (로그, 메트릭, 트레이스) — 에이전트가 접근 가능
- 에이전트 시작 시 디렉토리 구조 매핑
- CI/CD 파이프라인 상태와 테스트 결과

핵심 원칙: **에이전트 관점에서 컨텍스트 안에 없는 것은
존재하지 않는다.** Google Docs, Slack, 사람 머릿속의 지식은
시스템에 보이지 않는다.
리포지토리가 단일 진실 공급원(single source of truth)이어야
한다.

#### 2. 아키텍처 제약

전통적 AI 프롬프팅과 가장 날카롭게 갈라지는 지점.
"좋은 코드 작성해"라고 말하는 대신,
좋은 코드가 무엇인지 **기계적으로 강제**한다.

**의존성 레이어링:**

```
Types → Config → Repo → Service → Runtime → UI
```

각 레이어는 왼쪽 레이어만 임포트 가능.
제안이 아니라 구조 테스트와 CI 검증으로 **강제**한다.

**제약 시행 도구:**

- 결정론적 린터 — 위반을 자동 플래그
- LLM 기반 감사자 — 다른 에이전트의 코드를 리뷰
- 구조 테스트 — ArchUnit 같은 것, AI 생성 코드용
- Pre-commit 훅 — 커밋 전 자동 검사

역설: **솔루션 공간을 제약하면 에이전트가 더 생산적이
된다.** 무엇이든 생성 가능하면 막다른 길 탐색에
토큰을 낭비한다. 하네스가 명확한 경계를 정의하면
에이전트가 올바른 해에 더 빨리 수렴한다.

#### 3. 엔트로피 관리 ("가비지 컬렉션")

가장 과소평가된 구성요소.
시간이 지나면 AI 생성 코드베이스에 엔트로피가
누적된다 — 문서가 현실과 괴리되고, 명명 규칙이
분기되고, 죽은 코드가 쌓인다.

주기적 정리 에이전트로 대응한다:

- **문서 일관성 에이전트** — 문서와 현재 코드의 일치 검증
- **제약 위반 스캐너** — 이전 검사를 빠져나간 코드 탐지
- **패턴 강제 에이전트** — 확립된 패턴에서 벗어난 부분 식별·수정
- **의존성 감사자** — 순환/불필요 의존성 추적·해결

이 에이전트들은 일간, 주간, 또는 특정 이벤트 트리거로
실행되며, 인간 리뷰어와 미래 AI 에이전트 모두를 위해
코드베이스를 건강하게 유지한다.

### 실전 사례

#### OpenAI: 인간 코드 제로

| 역할           | 전통적                    | 하네스 엔지니어링                           |
| -------------- | ------------------------- | ------------------------------------------- |
| 코드 작성     | 주요 업무                 | 절대 안 함                                   |
| 아키텍처 설계 | 업무의 일부               | **주요 업무**                               |
| 문서 작성     | 뒷전                       | **핵심 인프라**                             |
| PR 리뷰       | 코드 리뷰                 | 에이전트 출력 + 하네스 효과성 리뷰           |
| 디버깅         | 코드 읽기                 | 에이전트 행동 패턴 분석                     |
| 테스트         | 테스트 작성               | 에이전트가 실행할 테스트 **전략** 설계       |

#### Stripe: 대규모 Minions

Stripe의 내부 코딩 에이전트 Minions —
주당 1,000건 이상의 병합 PR 생산:

1. 개발자가 Slack에 작업 게시
2. Minion이 코드 작성
3. Minion이 CI 통과
4. Minion이 PR 오픈
5. 인간이 리뷰·병합

1단계와 5단계 사이에 **개발자 개입 없음.**
테스트 실행, CI 검증, 스타일 준수, 문서 업데이트 —
모두 하네스가 처리한다.

#### LangChain: 미들웨어 우선

하네스를 조합 가능한 미들웨어 레이어로 구조화:

```
Agent Request
  → LocalContextMiddleware (코드베이스 매핑)
  → LoopDetectionMiddleware (반복 방지)
  → ReasoningSandwichMiddleware (추론 비용 최적화)
  → PreCompletionChecklistMiddleware (검증 강제)
  → Agent Response
```

각 미들웨어 레이어가 코어 에이전트 로직 수정 없이
특정 기능을 추가한다. 모듈식이라 테스트와 진화가 가능하다.

### 하네스 구축 3단계

#### Level 1: 기본 하네스 (개인 개발자)

- CLAUDE.md / .cursorrules에 프로젝트 관례 기록
- 린팅·포매팅용 Pre-commit 훅
- 에이전트가 자기 검증할 수 있는 테스트 스위트
- 일관된 명명의 명확한 디렉토리 구조

설정 시간: 1-2시간. 가장 흔한 에이전트 실수를 방지한다.

#### Level 2: 팀 하네스 (3-10인)

Level 1에 추가:

- 팀 전체 관례가 담긴 AGENTS.md
- CI로 강제하는 아키텍처 제약
- 공통 작업용 프롬프트 템플릿 공유
- 린터로 검증하는 문서화-as-코드
- 에이전트 생성 PR 전용 리뷰 체크리스트

설정 시간: 1-2일.

#### Level 3: 프로덕션 하네스 (엔지니어링 조직)

Level 2에 추가:

- 커스텀 미들웨어 (루프 감지, 추론 최적화)
- 관측성 통합 (에이전트가 로그·메트릭 읽기)
- 스케줄된 엔트로피 관리 에이전트
- 하네스 버전 관리와 A/B 테스트
- 에이전트 성능 모니터링 대시보드
- 에이전트 막힐 때의 에스컬레이션 정책

설정 시간: 1-2주.

### 흔한 실수 5가지

**1. 제어 흐름 과잉 설계.**
다음 모델 업데이트가 시스템을 깨뜨린다.
하네스를 "뜯어낼 수 있게(rippable)" 만들어라 —
모델이 똑똑해지면 "영리한" 로직을 제거할 수 있어야 한다.

**2. 하네스를 정적으로 취급.**
새 모델이 추론을 개선하면, 기존 추론 최적화
미들웨어가 오히려 역효과를 낸다.
주요 모델 업데이트마다 하네스 구성요소를 재검토하라.

**3. 문서 레이어 무시.**
가장 임팩트 있는 하네스 개선은 종종 가장 단순하다:
**더 나은 문서.**
AGENTS.md가 모호하면 에이전트 출력도 모호하다.

**4. 피드백 루프 없음.**
피드백 없는 하네스는 가이드가 아니라 **감옥**이다.
자기 검증 단계, 테스트 실행, 작업 유형별 성공률 메트릭이
필요하다.

**5. 인간 전용 문서.**
아키텍처 결정이 사람 머릿속이나 에이전트가 접근 불가한
Confluence에만 있으면 하네스에 구멍이 생긴다.
에이전트에게 필요한 모든 것이 리포지토리 안에 있어야 한다.

### 관련 개념 비교

| 개념                     | 범위                 | 초점                                     |
| ------------------------ | -------------------- | ---------------------------------------- |
| 프롬프트 엔지니어링     | 단일 상호작용         | 효과적인 프롬프트 작성                   |
| 컨텍스트 엔지니어링     | 모델 컨텍스트 윈도우 | 모델이 보는 정보                         |
| **하네스 엔지니어링**   | **전체 에이전트 시스템** | **환경, 제약, 피드백, 생애주기**     |
| 에이전트 엔지니어링     | 에이전트 아키텍처     | 내부 에이전트 설계와 라우팅             |
| 플랫폼 엔지니어링       | 인프라               | 배포, 스케일링, 운영                     |

하네스 엔지니어링은 컨텍스트 엔지니어링을 포함하고
프롬프트 엔지니어링을 활용하지만, 더 상위 수준에서
작동한다 — 단일 상호작용의 입력이 아니라
에이전트를 신뢰할 수 있게 만드는 **완전한 시스템**이다.

### NxCode의 실전 패턴

1. **리포지토리 우선 문서화:**
   모든 아키텍처 결정, 명명 규칙, 배포 프로세스가
   리포에 있다. Slack이나 Google Docs에 없다.

2. **점진적 제약 구축:**
   기본 린팅으로 시작해 패턴이 드러나면 아키텍처 제약을
   추가한다. 완벽한 하네스를 선설계하지 않는다.

3. **에이전트 특화 리뷰 체크리스트:**
   AI 생성 코드는 인간 코드와 다른 실패 모드를 갖는다.
   과도한 추상화, 불필요한 에러 핸들링, 문서 드리프트
   같은 에이전트 특유 패턴을 점검한다.

4. **멀티 프로바이더 하네스 설계:**
   Claude, GPT, Gemini 모두에서 작동하는 하네스.
   프로바이더 불가지론적 설계로 모델 교체 시 전체 시스템을
   재구축할 필요가 없다.

## 분석

이 글은 2026년 초에 급부상한 "하네스 엔지니어링"이라는
새로운 분야의 **산업 전체적 조감도**를 제공한다.
OpenAI, LangChain, Stripe, Martin Fowler, NxCode의
사례를 종합해 개인부터 조직 수준까지의 실천 프레임워크를
제시한다.

핵심 논점의 구조는 명확하다:

1. **문제 정의**: 에이전트 자체가 아니라 에이전트를
   둘러싼 시스템이 성패를 결정한다.
2. **증거 제시**: LangChain의 벤치마크(모델 불변, 하네스만
   변경으로 Top 30→5), OpenAI의 100만 줄 실험.
3. **프레임워크**: 3대 기둥 (컨텍스트, 제약, 엔트로피 관리).
4. **실행 가이드**: 3단계 성숙도 모델.

특히 "에이전트 = 모델 + 하네스"라는 공식은
[에이전틱 엔지니어링](./agentic-engineering-9-survival-skills.md)에서
다룬 컨텍스트 설계, 완료 정의, 실패 복구와 정확히
맞물린다.

## 비평

### 강점

**산업 증거의 밀도가 높다.**
하네스 엔지니어링을 주장만으로 내세우는 게 아니라
LangChain의 벤치마크 숫자(52.8→66.5%),
OpenAI의 100만 줄 실험, Stripe의 주당 1,000 PR이라는
구체적 데이터로 뒷받침한다.

**3단계 성숙도 모델이 실용적이다.**
개인 개발자의 CLAUDE.md 하나부터 조직 수준의
엔트로피 관리 에이전트까지 점진적 도입 경로를 제시해,
"하네스 엔지니어링은 대기업 전용"이라는 오해를 방지한다.

### 약점

**OpenAI 실험의 검증 불가능성.**
"100만 줄, 인간 코드 0줄"은 강렬한 헤드라인이지만,
내부 실험이라 외부에서 코드 품질, 유지보수성,
기술 부채를 검증할 방법이 없다. 100만 줄이
[그럴듯하지만 올바르지 않은 코드](./llm-writes-plausible-code-not-correct-code.md)일
가능성에 대한 논의가 빠져 있다.

**실패 사례의 깊이 부족.**
Vercel의 "도구 과잉 → 간소화" 사례를 언급하지만,
하네스 자체가 실패하는 경우 — 제약이 너무 엄격해
에이전트가 유효한 해를 찾지 못하는 경우, 엔트로피 관리
에이전트가 올바른 코드를 "정리"해버리는 경우 — 에 대한
분석이 없다.

**"rippable" 하네스의 모순.**
"뜯어낼 수 있게 만들라"고 조언하면서 동시에
구조 테스트, CI 검증, 미들웨어 체인이라는 상당한
인프라 투자를 권장한다. 뜯어냄의 비용과 기준에 대한
구체적 가이드라인이 없다.

**프롬프트 엔지니어링과의 경계 불분명.**
AGENTS.md에 "정확하고 기계 판독 가능한 문서"를
작성하라는 조언은 결국 정교한 시스템 프롬프트 작성이다.
하네스 엔지니어링이 프롬프트 엔지니어링의 상위 집합이라고
주장하지만, 실제 예시에서 그 경계가 흐릿하다.

## 인사이트

### 1. 소프트웨어 엔지니어링의 본질 재정의

하네스 엔지니어링의 진짜 혁명성은 기술이 아니라
**직업 정체성의 전환**에 있다.
"코드를 쓰는 사람"에서 "코드가 쓰여지는 환경을 설계하는
사람"으로. 이것은 단순한 도구 변화가 아니라
소프트웨어 엔지니어라는 직업의 **존재론적 재정의**다.

제조업에서 장인(craftsman)이 공정 설계자(process
engineer)로 전환된 역사와 평행한다.
손으로 물건을 만들던 사람이 물건이 만들어지는
시스템을 설계하는 사람이 된 것처럼,
코드를 쓰던 엔지니어가 코드가 쓰여지는 시스템을
설계하는 엔지니어가 되고 있다.

### 2. "제약이 창의성을 낳는다"의 공학적 증명

솔루션 공간을 제약하면 에이전트가 **더** 생산적이라는
발견은 직관에 반한다. 하지만 인간 창의성 연구에서
오래전부터 알려진 사실이다 — 소네트의 14행 제약이
무한한 자유형보다 더 강렬한 시를 만들 듯,
의존성 레이어링의 엄격한 규칙이 무제한 임포트보다
더 정확한 코드를 만든다.

이것은 **"모든 것이 가능한 AI"라는 환상을 깨는**
가장 실용적인 통찰이다.
에이전트에게 자유를 줄수록 결과가 나빠진다.
이것이 바이브 코딩과 하네스 엔지니어링의 본질적 차이다.
바이브 코딩은 자유를 주고, 하네스 엔지니어링은
자유를 빼앗는다. 그리고 후자가 이긴다.

### 3. 문서가 코드가 되는 세계

"AGENTS.md가 모호하면 에이전트 출력도 모호하다"는
문장은 단순한 조언 이상의 함의를 갖는다.
전통적으로 문서는 코드의 **부산물**이었다.
하네스 엔지니어링에서 문서는 코드의 **원인**이 된다.

이것은 **인과 관계의 역전**이다.
코드 → 문서 (전통)에서 문서 → 코드 (하네스)로.
AGENTS.md는 더 이상 README가 아니라
**실행 가능한 사양서(executable specification)**다.

이 역전이 함의하는 바: 문서 작성 능력이
코딩 능력보다 **직접적으로** 산출물 품질에 영향을
미치는 시대가 왔다. "글을 잘 쓰는 엔지니어"가
"코드를 잘 쓰는 엔지니어"보다 더 높은 레버리지를
갖게 된다.

### 4. 엔트로피는 AI의 아킬레스건

3대 기둥 중 "엔트로피 관리"가 가장 과소평가됐다는
원문의 지적에 전적으로 동의한다.
AI가 생성한 코드는 인간 코드보다 **더 빠르게**
부패한다. 왜냐하면 AI는 로컬 최적화에 탁월하지만
글로벌 일관성에 취약하기 때문이다.

파일 하나는 완벽하지만 프로젝트 전체는 방향이
다른 100개의 "완벽한" 파일로 이루어진 혼돈.
이것이 [LLM이 그럴듯하지만 올바르지 않은 코드를
쓴다](./llm-writes-plausible-code-not-correct-code.md)는
문제의 **시스템 수준 발현**이다.

가비지 컬렉션 에이전트라는 해법이 우아하다.
코드를 쓰는 에이전트와 코드를 청소하는 에이전트를
분리하는 것. 생성과 정리의 관심사 분리.
이것은 결국 소프트웨어의 오래된 원칙 — 단일 책임
원칙 — 을 **에이전트 수준**에 적용한 것이다.

### 5. "모델은 상품, 하네스가 해자"의 전략적 함의

이 명제가 사실이라면 (LangChain의 벤치마크가
강하게 시사한다), AI 시대의 경쟁 우위는
**어떤 모델을 쓰느냐가 아니라
어떤 하네스를 구축했느냐**에서 온다.

이것은 클라우드 시대 초기와 닮았다.
2010년대 초, "어떤 서버를 쓰느냐"가 중요했지만,
결국 "어떤 아키텍처를 구축했느냐"가 승부를 갈랐다.
서버는 상품화됐고 아키텍처가 차별화 요소가 됐다.

지금 모델이 상품화되고 있다.
OpenAI, Anthropic, Google의 최상위 모델 간 차이는
줄어들고 있다.
하네스 — 제약, 피드백, 문서, 생애주기 관리 — 가
새로운 차별화 요소다.

### 6. 하네스 엔지니어링의 재귀적 본질

이 글이 직접적으로 언급하지 않지만 가장 흥미로운 측면:
**하네스 자체를 AI로 구축할 수 있다.**

AGENTS.md를 AI가 작성하고, 아키텍처 제약을 AI가
설계하고, 엔트로피 관리 에이전트를 AI가 만든다면?
하네스 엔지니어링 자체가 하네스 안에 들어가는
**재귀적 구조**가 된다.

이 재귀의 끝은 어디인가?
인간이 마지막으로 남는 역할은 무엇인가?
원문은 "방향 제시"라고 말하지만,
방향 제시(의도 명세)조차 AI가 할 수 있게 되면?
하네스 엔지니어링은 **인간-AI 협업의 마지막 보루**인
동시에, 그 보루마저 자동화될 가능성을 내포한다.

### 7. 이 리포지토리 자체가 하네스다

이 TIL 리포지토리의 CLAUDE.md, AGENTS.md,
`.agent/guides/` 디렉토리 구조는 정확히 하네스
엔지니어링의 Level 1에 해당한다.
작성 가이드라인, Git 워크플로우, 커밋 메시지 규칙이
AI 에이전트의 행동을 제약하고 안내한다.

원문이 말하는 "가장 임팩트 있는 하네스 개선은
종종 가장 단순하다: 더 나은 문서"라는 통찰이
바로 여기서 실현되고 있다.
