## 요약

코드 diff를 리뷰하지 말고, 사양과 인수 기준을 리뷰하라. 코드 리뷰 병목의 해법은
리뷰 폐지가 아니라 리뷰 대상의 전환이다.

---

## 원문 논지

### 문제: 리뷰 병목의 구조적 붕괴

AI 코딩 도구를 사용하는 팀은 21% 더 많은 태스크를 완수하고 PR 머지가 98%
증가한다. 그런데 아이러니하게도 **PR 리뷰 시간은 91% 늘어난다.** 10,000명 이상의
개발자 데이터에서 도출된 수치다.
[코드 생성 비용이 사실상 0에 수렴하는](./agentic-engineering-patterns-1-1-code-is-cheap.md)
세계에서, 리뷰는 여전히 인간의 속도로 남아 있다.

코드 리뷰가 “원래부터 보편적이지 않았다”는 지적도 제기한다. 업계 표준이 된
시점은 2012~2014년경이며, 그 이전 수십 년간 팀들은 라인-바이-라인 심사 없이도
소프트웨어를 만들어왔다. 에이전트가 기하급수적으로 코드를 생성하는 세계에서
인간이 모든 변경을 리뷰한다는 것은 수학적으로 불가능하다.

### 해결책: 리뷰 대상을 코드에서 사양으로

저자의 핵심 처방은 한 문장으로 압축된다. **코드 diff를 리뷰하지 말고, 사양
(specification)과 인수 기준(acceptance criteria)을 리뷰하라.** 코드가 존재하기
전에 인간이 먼저 “성공이 무엇인지”를 정의하고, 그 정의가 올바른지를 검토하는
것이다. 기계는 기준을 집행하고, 인간은 기준을 설계한다.

이 전환을 구조화한 것이 5계층 신뢰 모델이다.

| 계층                 | 내용                                                             |
| -------------------- | ---------------------------------------------------------------- |
| 1. 복수의 경쟁 옵션  | 여러 에이전트가 동일 태스크를 각자 다르게 시도하여 서로 경쟁시킴 |
| 2. 결정론적 가드레일 | 테스트, 타입 체크, 계약 검증 — 통과/실패로 명확한 판정           |
| 3. 인수 기준         | 코드 생성 전에 인간이 성공 기준을 정의                           |
| 4. 권한 시스템       | 에이전트의 접근 범위를 아키텍처 수준에서 제한                    |
| 5. 적대적 검증       | 충돌하는 인센티브를 가진 별도 에이전트가 작업을 상호 검증        |

3번(인수 기준)이 모델의 중심이다. BDD(행동 주도 개발)는 이 패러다임의 실천
형태다. “인간이 저작한 인수 기준, 기계가 검증” — 리뷰어가 묻는 질문이 “이 코드가
올바른가?”에서 “이 기준이 올바른가?”로 바뀐다.

> “빠르게 배포하고, 모든 것을 관찰하고, 더 빠르게 되돌려라.” (ship fast, observe
> everything, revert faster)

인간의 가치는 코드 생성 이후의 검사(downstream inspection)에서 생성 이전의
의사결정(upstream decision-making)으로 이동한다. 그 upstream의 구체적 산출물이
사양과 인수 기준이다.

## HN 커뮤니티 반응

원문보다 오히려 HN 토론이 더 날카로운 지점을 짚는다.

### 저자 본인의 해명이 제목을 뒤집는다

저자 ankitdce는 HN에서 “지식 공유는 코드 리뷰가 아닌 **설계 사양 리뷰**를 통해
계속돼야 한다”고 직접 밝혔다. 원문 제목(“코드 리뷰를 죽이는 법”)과 달리, 저자
자신은 리뷰 자체를 없애는 게 아니라 리뷰의 **대상**을 바꾸자는 입장이다.
헤드라인이 주장보다 급진적이다.

### 리뷰의 타이밍 문제: PR이 너무 늦다

thomascountz의 발언이 가장 예리하다.

> “PR이 진짜로 머지될 준비가 됐을 때는, 이미 사실상 rubber-stamp이어야 한다.”

코드 리뷰가 병목인 것은 리뷰 자체의 문제가 아니라 **리뷰가 일어나는 시점**의
문제다. 리뷰는 PR 머지 직전이 아니라 설계와 스펙 단계에서 이뤄져야 한다. 이것은
원문의 핵심 처방 — 사양과 인수 기준을 먼저 정의하고 그것을 리뷰하라 — 과 같은
방향을 훨씬 간결하게 표현한 것이다.

### 자동화 테스트 소유권 문제

deterministic은 코드 리뷰를 **팀원이 작성하는 포괄적 자동화 테스트**로
대체하자고 제안한다. 원문의 “결정론적 가드레일”과 방향은 같지만, 에이전트가 아닌
인간 팀원이 테스트를 소유해야 한다는 점에서 다르다. 테스트는 인수 기준의 실행
가능한 형태이며, 누가 그것을 저작하는가가 품질을 결정한다는 통찰이다.

### SDLC의 점진적 붕괴

편집자 swyx는 “SDLC가 천천히 무너지고 있다(the SDLC is slowwwly breaking
down)”고 표현한다. 급변이 아니라 점진적 붕괴라는 인식이다.

## 분석

### 논증의 강점

**데이터 기반의 선명한 진단.** 21%/98%/91%라는 수치는 단순한 직관이 아니라 실측
데이터다. PR 머지 98% 증가 대비 리뷰 시간 91% 증가라는 비대칭은 “AI가 코드를
만들어도 검토는 인간이 한다”는 조직 현실을 수치로 드러낸다.

**역사적 맥락 활용.** 코드 리뷰가 2012~2014년에야 보편화됐다는 지적은 영리하다.
“코드 리뷰는 불변의 공학적 진리”가 아니라 “특정 시대의 관행”임을 상기시키며,
변화에 대한 심리적 저항을 약화시킨다.

**처방의 구체성.** “리뷰를 없애라”가 아니라 리뷰의 대상을 코드에서 사양과 인수
기준으로 전환하자는 제안은 실행 가능하다. 특히 “적대적 검증(adversarial
verification)” — 서로 충돌하는 인센티브를 가진 에이전트가 검증 — 은 LLM의 확증
편향(sycophancy)을 조직 설계로 제어하려는 시도로, 단순하지만 강력하다.

### 논증의 취약점

**[검증 부채](./verification-debt.md) 문제를 과소평가.** 원문의 진짜 맹점은
여기에 있다. AI가 생성한 코드는 테스트를 통과하면서도 실측 환경에서 기본 쿼리가
**20,000배 느릴 수 있다.** 이것은 구문 오류가 아니라 도메인 지식의 부재에서
비롯된다. “결정론적 가드레일”이 이런 종류의 오류를 잡으려면 측정 기준과
벤치마크를 미리 정의해야 하는데, 그 정의가 가장 어렵다. 자동화 테스트는 우리가
이미 아는 실패만 잡는다.

**타이밍 문제를 구조 문제로 오진.** HN의 thomascountz가 정확히 지적했듯, 코드
리뷰가 병목인 것은 리뷰의 본질적 문제가 아니라 PR 직전이라는 타이밍의 문제다.
“리뷰를 없애라”가 아니라 “리뷰를 앞당겨라”가 더 정밀한 처방일 수 있다.

**5계층 모델의 전제 조건을 과소평가.** 이 모델은 테스트 커버리지, BDD 인프라,
복수 에이전트 운영 비용, 타입 시스템 등 높은 기술 역량을 전제한다. 대다수
기업에서는 기본 테스트 커버리지조차 부족하다. 사양과 인수 기준을 잘 정의할 수
있는 팀이라면 이미 좋은 엔지니어링 문화를 가진 팀이다.

**BDD의 오랜 실패를 무시.** BDD는 2003년에 등장해 “자연어 스펙 → 자동 검증”을
약속했지만, Gherkin 등 사양 언어의 유지 비용과 자연어의 모호함 때문에
제한적으로만 쓰였다. LLM이 이 갭을 메울 수 있다는 주장은 흥미롭지만 검증이
필요하다.

**기준 실패의 책임 구조 부재.** 인수 기준을 인간이 정의한다고 해도, 그 기준
자체가 잘못됐을 때 누가 잡는가? “나쁜 기준을 통과한 코드”는 리뷰 없이 어떻게
걸러지는가? 코드 리뷰가 잘못된 사양을 검출하는 마지막 안전망 역할을 했다는
사실을 저자는 다루지 않는다.

## 비평

### 저자의 이해관계

Ankit Jain은 Aviator의 창업자다. Aviator는 PR 관리 자동화 플랫폼으로, “코드
리뷰의 비효율”이 커질수록 그들의 시장이 커진다. 이 글은 통찰이기도 하지만 동시에
시장 내러티브 구축이다. 데이터와 결론이 제품 방향과 정확히 일치하는 경우, 논증을
더 엄격하게 읽어야 한다.

### 제목의 수사적 과잉

“인간이 작성한 코드는 2025년에 죽었다”는 선언은 자극적이지만 거짓이다. AI가 코드
생산성을 높이는 것과 인간이 코드를 작성한다는 사실은 양립한다. 저자 본인도
HN에서 이를 수정했다 — 헤드라인이 주장보다 급진적으로 쓰인 것이다.

### 현실은 반대 방향을 가리킨다

글이 발행된 같은 달, 아마존은 AI 보조 코드 변경에
[**시니어 엔지니어 sign-off 의무화**](./amazon-senior-engineer-signoff-ai-changes-2026.md)
정책을 시행했다. 2024년 12월과 2026년 3월, AI 코드로 인한 대규모 장애 이후의
대응이다. “코드 리뷰의 죽음”을 예측하는 글이 나온 순간, 가장 큰 테크 기업 중
하나는 반대로 리뷰를 강화했다. 이 긴장을 이론으로만 해소하는 것은 충분하지 않다.

### 코드 리뷰의 사회적 기능을 범주 오류로 처리

코드 리뷰의 진짜 가치는 diff 검사만이 아니다. 지식 공유, 팀 내 공통 이해 형성,
주니어 개발자 성장, 암묵지(tacit knowledge) 전파 — 이 사회적 기능들이 사라질 때
생기는 공백을 저자는 다루지 않는다. 사양 리뷰가 이 기능들을 대체할 수 있다는
근거가 없다. “기술적 검증”을 “조직적 리뷰”와 동일시하는 것은 범주 오류다.

## 인사이트

### 1. 이 글의 진짜 이름은 “검증 부채”다

원문이 진단하는 문제의 정확한 이름은
**[검증 부채(Verification Debt)](./verification-debt.md)**다. 기술 부채가 마찰을
통해 자신을 드러낸다면, 검증 부채는 **거짓 자신감**을 낳는다. 코드는 깔끔해
보이고 테스트를 통과하며 PR도 정상으로 보이지만, 실제로는 충분히 검증되지 않은
상태다. 코드 리뷰 병목은 검증 부채가 누적되는 증상이지, 근본 원인이 아니다.

### 2. 검증 비용의 비대칭이 구조적 전환을 만든다

```
Before AI: 작성(병목) → 테스트 → 리뷰 → 배포
After AI:  생성(빠름) → 검증·리뷰(병목) → 배포
```

코드 생성 비용이 0에 수렴하면 검증 비용이 전체 엔지니어링 비용을 지배하게 된다.
“리뷰 병목”은 일시적 적응 문제가 아니라 생산-검증 비율이 구조적으로 역전되는
징후다. 이 전환은 되돌릴 수 없다.

### 3. 자동화 테스트는 우리가 아는 실패만 잡는다

[LLM은 그럴듯한 코드를 쓴다. 올바른 코드가 아니라.](./llm-writes-plausible-code-not-correct-code.md)
테스트를 통과하면서도 도메인 지식 부재로 인해 기본 쿼리가 20,000배 느린 코드를
만들 수 있다. “결정론적 가드레일”이 이런 오류를 잡으려면 측정 기준을 미리
정의해야 한다 — 그 정의 자체가 가장 어렵다. 자동화는 우리가 이미 아는 실패만
잡는다는 근본적 한계가 있다. 이것이 사양과 인수 기준을 코드보다 먼저 정의해야
하는 이유다 — 모르는 실패는 기준으로 만들 수 없지만, 아는 실패는 기준으로
만들어야 한다.

### 4. 인간의 역할은 판단 위치의 재배치다

인간의 가치는 “이 코드가 올바른가”를 판단하는 데서 “우리가 올바른 것을 만들고
있는가”를 판단하는 데로 이동한다. 이것은 코드 리뷰의 죽음이 아니라 **인간
판단력의 추상화 수준 상승**이다. 리뷰 대상이 코드에서 **사양과 인수 기준**으로
바뀐다. 아마존의 시니어 엔지니어 sign-off 정책조차 실질적으로는 이 방향이다 —
“이 AI 코드를 라인 바이 라인 검토했는가?”가 아니라 “이 변경이 의도한 동작을
하는가?”를 시니어가 책임지는 구조다.

### 5. 사양과 인수 기준을 정의하는 능력이 새로운 핵심 역량

AI 시대의 엔지니어링 역량은 코드를 잘 쓰는 것보다 **사양과 인수 기준을 잘
정의하는 것**으로 이동한다. 에이전트는 기준이 명확할수록 강력해지고, 기준이
모호할수록 위험해진다. [TDD의 Red 단계](./tdd-in-ai-era.md) — 실패하는 테스트를
먼저 쓰는 행위 — 는 사양을 실행 가능한 명세로 굳히는 과정이다. 이것이 AI 시대의
가장 인간적인 작업이 된다.

### 6. AI는 엔지니어링 역량의 증폭기다

역설적으로, AI 시대에 앞서가는 팀은 이미 좋은 엔지니어링 문화를 가진 팀이다. 이
모델이 작동하려면 테스트, 타입 시스템, 명세 문서화 등의 기반이 있어야 한다.
**기술 부채가 있는 조직에게 AI는 구원이 아니라 가속기다.** AI는 좋은 팀을 더
강하게 만들고, 나쁜 팀을 더 빠르게 무너지게 한다.

## 결론

Ankit Jain의 글은 **진단은 날카롭고 처방은 불완전하다.** 코드 리뷰의 병목이
구조적임은 맞다. 그러나 처방이 “리뷰를 없애라”에 가까운 반면, 현실(아마존)은
“리뷰를 강화하되 대상을 바꿔라”는 방향을 택하고 있다. 이론과 현장의 간극이다.

HN 논의가 덧붙인 핵심 수정은 “타이밍”이다. 코드 리뷰가 문제가 아니라 PR
직전이라는 타이밍이 문제다. 인간의 판단을 diff 검사에서 사양 정의와 인수 기준
설계로 옮기되, 그 이동을 개발 사이클의 가능한 한 앞단으로 당겨야 한다.

**무엇을 리뷰하는가가 아니라 언제 리뷰하는가가 진짜 질문이다.** 그리고 그
“언제”는 코드가 존재하기 전 — 사양과 인수 기준으로 의도를 굳히는 순간 — 이어야
한다.
