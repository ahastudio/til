# 하네스 엔지니어링 커리큘럼 — 핵심 개념 정리

<https://harness-curriculum.vercel.app/>

## 소개

하네스 엔지니어링 커리큘럼(HEC)은 LLM 에이전트 공학 지식을 대학 전공 교과과정처럼
선수과목 체계가 있는 14개 과목으로 정리한 한국어 학습 자료다.
저자 Dongkyu가 Anthropic 엔지니어링 블로그, arXiv 논문, MCP 스펙 등 1차 출처 30건 이상을
직접 대조해 만들었고, GitHub(`Dongkyu-Es/harness-curriculum`)에서 CC BY 4.0으로 공개됐다.

커리큘럼 전체를 관통하는 공식은 하나다.

> **에이전트 = 모델 + 하네스**

모델(Claude, GPT-4 등)은 판단만 한다.
하네스는 컨텍스트 조립, 도구 실행, 권한 게이팅, 상태 관리, 루프 제어를 담당한다.
Anthropic 공식 글로서리에 따르면 “하네스는 파일 접근, 셸 실행, 권한 게이팅, 메모리 로딩,
행동을 연쇄시키는 루프를 공급하는 소프트웨어 비계(scaffolding)”다.
“Claude Code가 하네스이고, Claude는 그 안의 모델이다”가 가장 직접적인 예시다.

## 1. 하네스와 에이전트 루프

### 하네스의 본질은 평범한 코드다

하네스는 마법이 아니다.
“모델 주변을 감싸는 평범한 코드(ordinary code wrapped around the model)”로서
계획, 도구 실행, 메모리, 안전 제어를 담당한다.
모델은 다음 행동만 결정하고, 나머지는 모두 하네스의 몫이다.

이 분리가 핵심이다.
모델은 확률론적이다 — 같은 입력이 같은 출력을 보장하지 않는다.
하네스는 결정론적이다 — 모델의 협력 여부와 무관하게 허용 목록을 검사하고 로그를 기록한다.
“모델이 판단하고, 하네스가 실행한다”는 분리가 에이전트 시스템 설계의 제1원리다.

### 에이전트 루프의 3상 사이클

에이전트 루프는 세 단계가 반복된다.

1. **컨텍스트 수집(gather context)** — 시스템 프롬프트, 도구 정의, 대화 이력, 파일 등을 조립한다.
2. **행동 실행(take action)** — 모델이 다음 행동(도구 호출 또는 완료 텍스트)을 출력하고, 하네스가 실행한다.
3. **결과 검증(verify results)** — 실행 결과를 대화에 추가하고 루프로 돌아가거나 종료한다.

기술적 골격은 `while True` 루프다.
모델 호출 → `tool_use` 응답 → 하네스가 도구 실행 → `tool_result`를 대화에 추가 → 재호출.
`stop_reason`이 `tool_use`이면 루프를 계속하고, 텍스트 완료이면 종료한다.

이 루프의 지적 계보는 ReAct(2022-10, Yao et al.)다.
ReAct는 추론 트레이스(Thought)와 행동(Action), 관찰(Observation)을 인터리빙하는 프롬프트 패턴으로,
“모델이 이유를 쓰면서 행동을 선택한다”는 원형을 확립했다.
오늘날 에이전트 루프는 이 골격을 API 계층으로 구현한 것이다.

### 루프 엔지니어링의 제1원리

“하네스의 모든 부품은 모델이 혼자 할 수 없는 것에 대한 가정을 인코딩한다 — 그 가정은 모델이 좋아질수록 낡는다.”

실천적 함의는 세 가지다.
첫째, 모델이 이미 아는 범용 도구(bash, 파일 편집)에 기대고 좁은 전용 도구를 줄인다.
둘째, 새 모델이 나올 때마다 “무엇을 그만할 수 있나”를 물어 죽은 비계를 걷어낸다.
셋째, 보안·UX·관측 경계만 신중하게 하네스에 남긴다.

## 2. 툴콜링의 역사와 표준화

### 발산기 (2021~2023 상반기): 네 가지 서로 다른 답

같은 문제(“LLM이 외부 세계를 조작하게 하려면?”)에 네 가지 접근이 경쟁했다.

| 접근 | 대표 | 방식 |
| ---- | ---- | ---- |
| 단일 도구 파인튜닝 | WebGPT (2021-12) | 브라우저 하나에 모델을 통째로 파인튜닝. 확장 불가. |
| 라우터 아키텍처 | MRKL (2022-05) | LM 밖의 라우터가 전문 모듈로 위임. |
| 프롬프트 파싱 | ReAct + LangChain (2022-10) | few-shot 프롬프트로 `Thought/Action/Observation` 유도, 출력을 정규식으로 파싱. |
| 자기지도 학습 | Toolformer (2023-02) | 학습 데이터에 API 호출 토큰을 삽입해 모델 가중치에 내재화. |

3번 접근(프롬프트 파싱)의 고질병은 형식 이탈 시 파싱 실패였다.
이 취약성이 네이티브 함수호출의 존재 이유가 된다.

### 네이티브 함수호출 (2023-06-13): 파싱에서 계약으로

OpenAI가 2023년 6월 13일 `gpt-4-0613`과 `gpt-3.5-turbo-0613`에 `functions` 파라미터를 도입했다.
JSON Schema로 함수를 기술하면 모델이 “함수 시그니처에 부합하는 JSON”을 출력하도록 파인튜닝됐다.

이 시점에 성립한 패러다임:

> **도구는 JSON Schema로 선언하고, 호출은 구조화 메시지로 받고, 실행은 개발자가 하고, 결과를 대화에 되돌려준다.**

중요한 사실: **도구 정의는 API 메타데이터가 아니라 프롬프트의 일부다.**
OpenAI 공식 문서: “함수 정의는 모델이 학습된 문법으로 시스템 메시지에 주입되며, 컨텍스트 한도를 차지하고 입력 토큰으로 과금된다.”
Anthropic도 `tools` 파라미터 사용 시 자동으로 특수 시스템 프롬프트가 추가되며, 모델에 따라 290~675 토큰이 소비된다.

Anthropic의 현행 API 형태:

- `tools[].input_schema`에 JSON Schema로 도구 정의
- 모델이 `stop_reason: “tool_use”` + `tool_use` 콘텐츠 블록(`id`, `name`, `input`) 반환
- 개발자가 실행 후 `role: “user”` 메시지 안의 `tool_result` 블록으로 결과 주입

이후 Structured Outputs(2024-08-06)가 마지막 퍼즐을 채웠다.
“스키마에 대체로 맞는 JSON”을 제약 디코딩으로 “항상 맞는 JSON”으로 바꿨다(`strict: true`).

### MCP (2024-11-25): 앱과 도구 사이의 표준화

함수호출 API가 표준화한 것은 모델↔개발자 코드 사이였다.
도구 구현체는 앱마다 다시 짜야 했다 — N개 앱 × M개 도구 = N×M 커스텀 통합.
MCP(Model Context Protocol)는 이를 N+M으로 줄이는 개방 표준이다.

MCP의 공식 비유: “AI 애플리케이션을 위한 USB-C 포트.”

구조는 JSON-RPC 기반 스테이트풀 세션 프로토콜이다.
Host(코디네이터·보안 경계 집행) → Client(서버당 1:1 연결) → Server(기능 제공).
서버 프리미티브 3종: **tools**, **resources**, **prompts**.

2025년 3월 26일 리비전에서 기존 HTTP+SSE 전송이 Streamable HTTP로 대체됐다.
같은 날 OpenAI가 채택했고, 2025년 4월 9일 Google이 채택하면서 사실상 업계 표준이 확정됐다.

## 3. 컨텍스트 엔지니어링

### 컨텍스트는 가장 희소한 자원이다

컨텍스트 윈도우는 유한하고, 휘발성이며, 비싸다.
Karpathy의 비유: LLM의 RAM.
트랜스포머의 셀프어텐션은 시퀀스 길이의 제곱(n²)으로 비용이 증가한다.
길수록 각 토큰이 모든 다른 토큰을 참조해야 하므로, 주의 예산(attention budget)은 빠르게 소진된다.

컨텍스트가 길어질수록 한계 효용이 체감한다.
오래된 내용이 묻히고, 최신 정보가 상대적으로 더 강하게 모델에 영향을 미친다.

하네스의 절반은 사실상 컨텍스트 관리 장치다.

### 컨텍스트 관리 기법 4종

### 1. 압축(Compaction)

오래된 대화나 중간 결과를 요약해 토큰을 줄인다.
의미적 압축(핵심 정보만)과 구문적 압축(형식 축소)을 상황에 따라 선택한다.
단, 압축은 정보 손실을 수반한다 — 무엇을 버리고 무엇을 유지할지가 핵심 판단이다.

### 2. 구조적 노트 테이킹

파일시스템이 에이전트의 장기 기억이다.
`CLAUDE.md`, `SKILL.md`, `NOTES.md` 같은 파일에 컨텍스트 윈도우 밖의 정보를 보존한다.
세션이 끊겨도 파일은 남는다 — 모델의 기억이 아닌 파일시스템에 의존하는 이유다.

### 3. 서브에이전트 격리

탐색이 넓어지면 서브에이전트를 분기한다.
서브에이전트는 격리된 컨텍스트를 가지며, 요약만 부모에게 돌아온다.
주 컨텍스트를 탐색 결과로 채우지 않아도 된다.

### 4. 적시 검색(Just-In-Time Retrieval)

모든 정보를 처음부터 로드하지 않는다.
필요할 때 검색해 로드한다.
스킬 카탈로그에서 한 줄 요약만 상시 유지하고, 필요한 스킬의 전체 내용을 호출 시 로딩하는 것이 대표적 패턴이다.

### 컨텍스트 구성 요소의 분류

하네스가 매 턴마다 컨텍스트를 재조립할 때 들어가는 것들:

| 위치 | 내용 | 수명 | 특성 |
| ---- | ---- | ---- | ---- |
| 컨텍스트 윈도우 | 시스템 프롬프트, 도구 정의, 대화 이력, 실행 결과 | 세션 한정 | 휘발성, 고비용 |
| 파일시스템 | `CLAUDE.md`, `SKILL.md`, 로그, 출력, git 이력 | 영속 | 에이전트 장기 기억 |
| 하네스 코드 | 루프 로직, 게이트, 훅, 압축 트리거, 로깅 | 결정론적 | 모델 설득 불가 |
| 격리된 컨텍스트 | 서브에이전트 탐색, 독립 평가자 판단 | 격리 | 요약만 부모로 복귀 |

**설계 원칙: 모델은 판단하고, 파일시스템은 기억하고, 하네스는 집행하고, 파일은 영속한다.**

## 4. 도구 설계

### 좋은 도구는 에이전트의 어포던스에 맞춰 설계된다

Anthropic 엔지니어링 블로그 “Writing effective tools for agents”(2025-09-11)가 도구 설계론의 준정전 문서다.

### 원칙 1: 통합(Consolidation)

`list_users`, `list_events`, `create_event` 세 도구보다 `schedule_event` 하나가 낫다.
에이전트 관점의 워크플로 단위로 묶어야 컨텍스트 소비와 중간 단계가 줄어든다.
“API를 래핑하는 것”이 아니라 “에이전트의 어포던스에 맞춰 재설계하는 것”이다.

### 원칙 2: 네임스페이싱

`asana_search`, `jira_search`처럼 접두사로 그룹화한다.
도구가 많을수록 모델이 잘못된 도구를 선택할 확률이 높아진다.
OpenAI 공식 문서는 “턴 시작 시 20개 미만 도구를 목표로” 권고한다.
20개를 초과하면 설명 축약 또는 tool search(지연 로딩)를 고려해야 한다.

### 원칙 3: 의미 있는 에러

에러 코드가 아니라 “다음에 무엇을 고쳐 재시도할지”를 자연어로 알려준다.
에이전트는 에러를 보고 다음 행동을 결정해야 하므로, 실행 가능한 정보(actionable improvements)가 필요하다.

### 원칙 4: 토큰 효율

페이지네이션, 필터, 절단 기본값을 설정한다.
UUID 같은 저신호 식별자를 응답에서 제거한다.
`response_format: DETAILED|CONCISE` 같은 열거형으로 응답 상세도를 제어한다.

### 원칙 5: 설명 작성

파라미터명은 `user`가 아니라 `user_id`처럼 명확하게 쓴다.
암묵적 컨텍스트를 명시로 바꾼다.
“새 팀원에게 설명하듯” 쓰는 것이 기준이다.

### 원칙 6: 평가

실제 워크플로 기반 평가 태스크를 만든다.
정확도, 툴콜 수, 토큰, 에러율을 계측한다.
에이전트(Claude Code)로 도구 설명을 자동 최적화하는 루프를 구성할 수 있다.

## 5. 워크플로 패턴

Anthropic “Building effective agents”(2024-12-19)가 정리한 5가지 패턴.
가장 단순한 패턴에서 시작해 복잡성이 정당화될 때만 상위 패턴으로 이동하는 것이 원칙이다.

### 1. 프롬프트 체이닝(Prompt Chaining)

작업을 순차적 단계로 분리한다.
각 단계의 출력이 다음 단계의 입력이 된다.
단계 사이에 게이트를 두어 품질을 확인할 수 있다.

### 2. 라우팅(Routing)

입력을 분류해 적합한 처리 경로로 보낸다.
코딩 질문, 일반 질문, 결제 질문을 각각 다른 모델이나 프롬프트로 처리한다.

### 3. 병렬화(Parallelization)

독립적인 작업을 동시에 실행한다.
동일 작업을 여러 에이전트가 실행해 결과를 투표로 결정하는 “앙상블” 변형도 포함된다.

### 4. 오케스트레이터-작업자(Orchestrator-Workers)

오케스트레이터가 작업을 분해해 작업자에게 위임한다.
작업자들이 병렬로 실행하고 결과를 오케스트레이터가 집계한다.
복잡한 코딩 작업, 리서치 파이프라인에 적합하다.

### 5. 평가자-최적화자(Evaluator-Optimizer)

생성자와 평가자를 분리한다.
평가자가 기준을 충족할 때까지 생성자가 반복한다.
번역 품질 개선, 코드 검색 정확도 향상에 효과적이다.

## 6. 장기 하네스 설계

### 멀티 컨텍스트윈도우 작업의 핵심 문제

시간 단위 혹은 주 단위로 실행되는 에이전트는 다른 설계를 필요로 한다.
Anthropic “Effective harnesses for long-running agents”(2025-11-26)의 핵심 비유:
“이전 교대 기억이 없는 신입 엔지니어.”

세션 경계를 넘으면 이전 컨텍스트가 사라진다.
에이전트는 매 재개마다 현재 상태를 파일시스템에서 재구성해야 한다.

### 해법은 소프트웨어 엔지니어링 관행이다

### 초기화(Initializer)

작업 시작 시 피처 리스트 JSON, 진행 로그, git 커밋으로 상태를 명시한다.

### 증분 단위 실행

1 피처 단위로 작업을 쪼개고, 완료 후 체크포인트를 기록한다.

### 테스트 후 완료 처리

각 단계의 완료를 테스트가 검증한다.
“통과”가 완료의 정의다.

### 클린 스테이트 유지

중간 상태를 파일에 명시적으로 기록하고, 재개 시 로드한다.

이것들은 새로운 AI 개념이 아니라 소프트웨어 엔지니어링 관행이다.
에이전트 시스템이 복잡해질수록 기존 SE의 원칙들이 더 중요해진다.

## 7. 스킬 시스템

### 스킬은 에이전트의 절차 지식이다

SKILL.md 형식은 2025년 10월 16일 Anthropic이 발표하고,
2025년 12월 18일 `agentskills.io`에서 오픈 표준으로 공개됐다.

스킬의 비유는 “신입사원 온보딩 가이드”다.
에이전트가 알아야 하는 절차적 지식을 파일에 담아 필요할 때 로드한다.
MCP가 외부 서비스와의 연결을 표준화한다면, 스킬은 절차 지식을 표준화한다.

### 3단 점진적 공개(Progressive Disclosure)

```text
── 메타데이터 (YAML frontmatter, 항상 컨텍스트에 있음)
   └── 본문 (Markdown, 필요 시 로드)
        └── 참조 파일 (외부 파일, 필요 시 로드)
```

컨텍스트에는 한 줄 요약만 상시 유지한다.
스킬이 호출되면 전체 본문을 로드한다.
파일시스템 기반이므로 크기 제한이 없다.

MCP(외부 연결·실시간 데이터)와 스킬(절차 지식·재사용 가능 패턴)은 상보적이다.

## 분석

### LLM OS 비유가 포착하는 것과 놓치는 것

Karpathy의 비유: “LLM은 새로운 운영체제의 커널 프로세스.”
Model = CPU(연산 처리), Context = RAM(128K 토큰), Tools = 주변장치/시스템 콜.

이 비유가 포착하는 것은 컨텍스트의 세 속성이다.
유한하고(크기가 고정됨), 휘발성이며(세션이 끊기면 사라짐), 고가다(토큰 비용).
RAM의 세 속성과 정확히 대응한다.

그러나 비유는 한계가 있다.
CPU는 결정론적이지만 LLM은 확률론적이다 — 같은 “명령”이 다른 출력을 낼 수 있다.
RAM은 주소로 접근하지만 컨텍스트는 어텐션 메커니즘으로 접근한다 — n² 비용 구조는 RAM에 없다.
KV 캐시 경제학은 OS 비유로 설명되지 않는다.

비유는 직관을 준다.
설계 결정의 근거는 될 수 없다.

### 하네스 공학이 독립 영역이 되는 조건

커리큘럼은 “하네스 엔지니어”라는 역할이 응용 개발자나 ML 엔지니어와 구별되는 전문직으로
자리 잡을 수 있다는 전제로 설계됐다.

이 전제가 성립하려면 하네스의 복잡성이 계속 증가해야 한다.
지금은 성립하는 것처럼 보인다.
모델이 강해질수록 더 긴 컨텍스트, 더 많은 도구, 더 복잡한 오케스트레이션이 쓰인다.
하네스의 역할이 줄어드는 것이 아니라 더 중요해지고 있다.

그러나 루프 엔지니어링의 제1원리가 말했듯, 모델이 좋아질수록 하네스의 가정들은 낡는다.
지금 필요한 하네스 설계가 1년 후에도 필요할지는 열린 질문이다.

## 비평

### 커리큘럼이 “판단력”을 전달하는 방법이 없다

14개 과목이 다루는 것은 패턴과 구조다.
그러나 실제 에이전트 시스템 개발에서 가장 어려운 순간은 패턴을 아는 것이 아니라
패턴들 사이에서 판단을 내리는 것이다.

컨텍스트가 부족할 때 무엇을 버리는가.
에이전트가 루프에 갇혔을 때 어떻게 감지하는가.
서브에이전트로 분기할 시점은 언제인가.

이 판단들은 캡스톤 과제로 가르칠 수 없다.
HE402(설계 철학)가 이 영역을 다루지만, 원칙 나열을 넘어서지 못한다.
교과과정이 전달할 수 없는 부분이 있다고 솔직하게 명시하는 것이 정직한 설계일 것이다.

### 검증 엔지니어링이 300단계에 배치된 것은 설계 순서와 맞지 않는다

HE304(검증 엔지니어링)는 300단계다.
루프를 구현하고(HE201), 도구를 설계하고(HE202), 컨텍스트를 관리한(HE203/204) 뒤에야 검증을 배운다.

에이전트 시스템에서 검증은 처음부터 설계에 포함되어야 한다.
나중에 추가하면 검증을 고려하지 않은 구조 위에 얹는 것이 된다.
이 학습 순서는 현실의 설계 순서와 반대다.

### 평가가 구현 역량에 치우쳐 있다

“5개 도구 스위트를 설계하라”는 캡스톤은 구현 역량을 측정한다.
에이전트 공학에서 더 중요한 역량은 “이 설계의 어느 부분이 모델을 실수하게 만드는가”를 찾는 것이다.
비판적 설계 검토를 평가하는 과제가 포함된다면 교육 효과가 높아질 것이다.

## 인사이트

### 권한 게이팅은 운영체제 커널과 같은 원리다

운영체제는 프로세스가 직접 하드웨어에 접근하는 것을 막는다.
모든 자원 접근은 시스템 콜을 통해 커널이 중재한다.

하네스의 권한 게이팅은 같은 구조다.
모델이 직접 도구를 실행하지 않고, 하네스가 실행 전에 허용 목록을 확인하고 감사 로그를 기록한다.
최소 권한 원칙, 감사 가능성, 격리 — 운영체제가 수십 년에 걸쳐 발전시킨 개념들이
에이전트 시스템에서 재발견되고 있다.

비유의 차이: 운영체제의 프로세스는 결정론적이고, 에이전트의 모델은 확률론적이다.
모델이 권한 게이팅을 우회하는 방법을 “발견”할 수 있다는 점은 커널 보안과 다른 위협 모델이다.

### 도구 역사의 3막 구조가 표준화의 패턴을 보여준다

발산 → 수렴 → 프로토콜 표준화는 여러 기술 영역에서 반복된 패턴이다.
이더넷 이전의 LAN 기술 난립, HTTP 이전의 다양한 하이퍼텍스트 시스템이 같은 경로를 걸었다.

LLM 도구 호출도 같은 경로를 걷다.
프롬프트 파싱(방언 시대) → JSON Schema 함수호출(사실상 표준 성립) → MCP(프로토콜 표준화).
OpenAI와 Google이 모두 MCP를 채택한 2025년 4월이 표준 확정 시점이다.

이 패턴의 함의: 지금 “사실상 표준”이 된 것은 앞으로도 바뀔 수 있다.
MCP 스펙은 이미 두 차례 주요 리비전을 거쳤다.
특정 API 형태가 아니라 “왜 이 구조가 선택됐는가”의 원리를 이해하는 것이 더 오래 유효하다.

### 에이전트 시대에 소프트웨어 엔지니어링 원칙이 더 중요해진다

에이전트가 자율적으로 코드를 쓰고 시스템을 수정하게 될수록,
소프트웨어 엔지니어링의 기본 원칙들 — 테스트, 점진적 변경, 클린 스테이트, 검증 — 이
더 중요해진다.

아이러니하게도 AI 도구가 발전할수록 그 도구를 안전하게 사용하기 위한 인간 공학 역량이 높아져야 한다.
장기 하네스 설계에서 해법이 “소프트웨어 엔지니어링 관행”으로 귀결되는 것은 우연이 아니다.
에이전트는 새로운 패러다임이지만, 그것을 제어하는 것은 오래된 원칙이다.
