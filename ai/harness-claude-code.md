# Harness — Claude Code 에이전트 팀 & 스킬 아키텍트

- 원문: <https://github.com/revfactory/harness>

## 요약

Claude Code용 에이전트 팀과 스킬을 설계하고 운용하는 플러그인이다.
여러 에이전트를 역할별로 구성해 팀처럼 협업시키고, 스킬을 모듈 단위로
정의해 재사용할 수 있도록 한다.

GeekNews에서 55포인트를 기록하며 오늘 가장 높은 관심을 받은 글 중 하나다.

## 핵심 개념

### 에이전트 팀

단일 에이전트 대신 역할이 다른 에이전트들을 팀으로 구성한다.
예: 기획 에이전트, 코드 작성 에이전트, 리뷰 에이전트.

### 스킬 아키텍처

재사용 가능한 스킬을 모듈로 정의하고, 에이전트가 필요에 따라 호출한다.
스킬은 프롬프트 + 도구 + 컨텍스트의 묶음이다.

## 인사이트

### 1) 단일 에이전트의 한계를 팀 구조로 돌파한다

하나의 에이전트가 모든 작업을 담당하면 컨텍스트가 오염되고,
역할 혼선으로 품질이 떨어진다.

팀 구조는 각 에이전트가 자신의 전문 영역에 집중하게 만든다.
코드 리뷰 에이전트는 리뷰만 하고, 문서 에이전트는 문서만 본다.
책임 분리(Separation of Concerns)를 에이전트 레이어에 적용한 것이다.

### 2) 스킬 재사용성이 에이전트 조직의 핵심 경제다

매번 프롬프트를 새로 짜는 대신 검증된 스킬을 재사용하면,
에이전트 팀의 운영 비용이 선형이 아닌 점진적으로 줄어든다.

스킬 라이브러리가 쌓일수록 새 작업 구성이 빨라진다.
이것은 함수·라이브러리 재사용과 동일한 경제 논리다.

### 3) Claude Code 생태계가 플러그인 레이어로 확장되고 있다

Claude Code 자체가 확장 포인트를 제공하면서 그 위에 메타 도구가
생겨나고 있다. Harness는 그 흐름의 초기 사례다.

IDE 플러그인 생태계처럼, Claude Code 플러그인 생태계가 형성될 가능성이 있다.
먼저 좋은 플러그인을 내놓는 팀이 워크플로 표준을 선점한다.

### 4) 에이전트 조직화는 소프트웨어 아키텍처 문제다

에이전트 수가 늘수록 의존성, 인터페이스, 상태 공유 문제가 발생한다.
이것은 마이크로서비스를 설계할 때와 같은 종류의 문제다.

역할 경계를 명확히 하고, 에이전트 간 통신 프로토콜을 정의하고,
장애 전파를 막는 설계가 필요하다. 에이전트 팀도 결국 아키텍처다.
