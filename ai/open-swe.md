# Open SWE - 오픈소스 사내 코딩 에이전트 프레임워크

<https://github.com/langchain-ai/open-swe>

## 요약

LangChain이 만든 오픈소스 코딩 에이전트 프레임워크.
Stripe, Ramp, Coinbase 같은 회사들이 사내에서 운영하는
코딩 에이전트 시스템을 누구나 구축할 수 있게 해준다.
LangGraph + Deep Agents 위에 구축되어 있으며,
Slack, Linear, GitHub에서 호출하면 샌드박스 환경에서
코드를 수정하고 PR을 열어주는 구조다.

MIT 라이선스. 6.9k 스타. Claude Opus 4.6을 기본 모델로 사용한다.

## 아키텍처

### 전체 흐름

```
[Slack/Linear/GitHub 웹훅]
        │
        ▼
[FastAPI 서버] ─── 시그니처 검증 ─── 이벤트 필터링
        │
        ▼
[LangGraph 에이전트]
  ├─ 시스템 프롬프트 (prompt.py + AGENTS.md)
  ├─ 도구 ~15개 (execute, fetch_url, commit_and_open_pr 등)
  ├─ 미들웨어 4개 (에러 처리, 메시지 큐, PR 안전망 등)
  └─ 서브에이전트 (병렬 하위 작업)
        │
        ▼
[샌드박스] ─── Modal / Daytona / Runloop / Local
        │
        ▼
[GitHub PR 생성]
```

### 핵심 컴포넌트

| 컴포넌트             | 역할                                          |
| -------------------- | --------------------------------------------- |
| `agent/server.py`    | CLI 진입점, 샌드박스 관리, 에이전트 초기화     |
| `agent/webapp.py`    | FastAPI 웹훅 서버 (Slack/Linear/GitHub)        |
| `agent/prompt.py`    | 시스템 프롬프트 구성 및 컨텍스트 주입          |
| `agent/tools/`       | 도구 6개 (PR, HTTP, URL, Slack, Linear, GitHub)|
| `agent/middleware/`   | 미들웨어 4개 (에러, 큐, 빈 메시지, PR 안전망) |
| `agent/integrations/` | 샌드박스 백엔드 (Modal, Daytona, Runloop 등)  |

### 도구 목록

Deep Agents 내장 도구 (파일 조작, 검색, grep, todo, 서브에이전트 생성)에
더해 커스텀 도구 6개를 추가:

- `execute` — 셸 명령 실행 (기본 타임아웃 5분)
- `fetch_url` — 웹 콘텐츠 가져오기
- `http_request` — API 호출
- `commit_and_open_pr` — Git 커밋 + PR 생성
- `linear_comment` / `slack_thread_reply` / `github_comment` — 협업 알림

### 미들웨어 스택

1. **ToolErrorMiddleware** — 도구 실행 에러를 우아하게 처리
2. **check_message_queue_before_model** — 모델 호출 전 큐에 쌓인 메시지 주입
3. **ensure_no_empty_msg** — 빈 메시지 방지
4. **open_pr_if_needed** — 에이전트가 PR을 안 열었으면 미들웨어가 대신 생성

`open_pr_if_needed`는 특히 인상적이다.
에이전트가 `commit_and_open_pr` 도구 호출에 실패하더라도,
미들웨어가 커밋되지 않은 변경사항과 푸시되지 않은 커밋을 감지해서
자동으로 PR을 생성하는 안전망 역할을 한다.

### 컨텍스트 엔지니어링

- **AGENTS.md** — 저장소별 지시사항을 시스템 프롬프트에 주입
- **소스 컨텍스트** — 이슈/스레드 전체 이력을 선행 제공
- **메시지 큐** — 에이전트 실행 중 도착하는 후속 메시지를 비동기 주입

### 호출 방식과 스레드 관리

| 플랫폼   | 호출 방법                           | 시그니처 검증       |
| -------- | ----------------------------------- | ------------------- |
| Slack    | 봇 멘션 + `repo:owner/name`        | 타임스탬프 기반     |
| Linear   | 이슈에 `@openswe` 코멘트           | HMAC-SHA256         |
| GitHub   | PR 리뷰에서 봇 태그                 | X-Hub-Signature-256 |

결정론적 스레드 ID로 후속 메시지를 같은 에이전트에 라우팅한다.
스레드가 활성 상태이면 새 실행 대신 메시지 큐에 추가하여
동시 실행 충돌을 방지한다.

## 코드 분석

### Deep Agents 의존

자체 LangGraph 그래프를 직접 정의하지 않고
`deepagents` 라이브러리의 `create_deep_agent()`를 사용한다.
이는 LangGraph의 상태 머신 패턴을 Pregel 실행기를 통해 추상화한 것이다.
덕분에 노드 연결이나 라우팅 로직을 직접 작성할 필요가 없고,
도구와 미들웨어만 등록하면 된다.

```python
# server.py의 에이전트 생성 (핵심 구조)
agent = create_deep_agent(
    model="anthropic:claude-opus-4-6",
    tools=[http_request, fetch_url, commit_and_open_pr, ...],
    middleware=[
        ToolErrorMiddleware(),
        check_message_queue_before_model,
        ensure_no_empty_msg,
        open_pr_if_needed,
    ],
    max_tokens=20_000,
)
```

### 샌드박스 생명주기

```python
SANDBOX_BACKENDS = {}  # 스레드별 샌드박스 캐시

# 1. 기존 샌드박스 재연결 시도
# 2. 실패 시 180초 타임아웃으로 새 샌드박스 생성
# 3. 1초 간격 폴링으로 준비 상태 확인
# 4. 레포 클론 또는 pull (언커밋 변경사항 감지)
```

스레드 단위로 샌드박스를 재사용하므로
후속 메시지에서 이전 작업 상태가 보존된다.

### 프롬프트 설계

시스템 프롬프트의 핵심 지시:

- "매 턴마다 반드시 도구를 호출하라" — 세션 유지를 위해
- "commit_and_open_pr 호출 전에 절대 사용자 확인을 구하지 마라" — 자율 동작
- 변경은 최소한으로, 린터와 관련 테스트만 실행
- 백업 파일 생성 금지, Git 저장소 내에서만 작업

## 비평

### 강점

1. **미들웨어 패턴이 훌륭하다.**
   에이전트의 불완전한 실행을 보완하는 안전망 설계가 실용적이다.
   `open_pr_if_needed`가 좋은 예시다.

2. **호출 채널 추상화가 깔끔하다.**
   Slack, Linear, GitHub 세 채널을 통일된 스레드 모델로 추상화했다.
   결정론적 스레드 ID로 후속 라우팅을 해결한 것도 깔끔하다.

3. **메시지 큐로 비동기 상호작용을 해결했다.**
   에이전트가 돌고 있는 동안 사용자가 추가 지시를 보내면
   큐에 쌓았다가 다음 모델 호출 전에 주입하는 방식이 실전적이다.

4. **샌드박스 재사용으로 상태 연속성을 보장한다.**
   매번 새 환경을 만들지 않고 스레드별로 재사용하므로
   후속 대화에서 이전 작업 결과가 유지된다.

### 약점

1. **Deep Agents에 강하게 결합되어 있다.**
   핵심 오케스트레이션을 `create_deep_agent()`에 위임하므로
   Deep Agents의 내부 동작을 이해하지 않으면 커스터마이징이 제한적이다.
   "상류 개선을 가져올 수 있다"고 하지만,
   동시에 상류 변경에 취약하다는 의미이기도 하다.

2. **검증이 프롬프트 의존적이다.**
   린트/테스트 실행을 프롬프트로 지시하고,
   실패 시 미들웨어가 PR을 강제 생성하는 구조다.
   에이전트가 테스트를 건너뛰거나 실패한 테스트를 무시할 가능성이 있다.
   프로그래매틱한 게이트가 더 안전할 것이다.

3. **모델 의존도가 높다.**
   Claude Opus 4.6 하드코딩에 토큰 제한 20,000.
   다른 모델로 전환하거나 토큰 예산을 조정하려면 코드 수정이 필요하다.
   설정 파일로 빼는 게 자연스럽다.

4. **에러 복구 전략이 단순하다.**
   샌드박스 연결 실패 시 재생성하는 것 외에
   도구 실행 실패나 모델 할루시네이션에 대한
   체계적인 복구 전략이 보이지 않는다.

## 인사이트

### 1. 미들웨어는 에이전트 신뢰성의 핵심이다

LLM 에이전트는 본질적으로 비결정적이다.
"에이전트가 완벽하게 동작할 것"을 가정하지 않고,
미들웨어로 실패를 보완하는 패턴이 프로덕션 에이전트의 핵심 설계 원칙이다.
`open_pr_if_needed`는 이 원칙의 교과서적 구현이다.

### 2. 도구는 적을수록 좋다

~15개로 의도적으로 제한했다고 명시한다.
"축적된(accumulated)" 도구가 아니라 "큐레이션된(curated)" 도구라는 표현이 인상적이다.
도구가 많으면 모델의 선택 부담이 커지고 오용 가능성이 높아진다.

### 3. 컨텍스트 엔지니어링 = AGENTS.md + 소스 컨텍스트 + 메시지 큐

세 가지 레이어로 컨텍스트를 구성한다:

- **정적 컨텍스트**: AGENTS.md (저장소 규칙)
- **초기 컨텍스트**: 이슈/스레드 전체 이력 (선행 제공)
- **동적 컨텍스트**: 메시지 큐 (실행 중 주입)

이 세 가지를 분리해서 관리하는 것이
단순히 "프롬프트를 잘 쓰는 것"보다 체계적인 접근이다.

### 4. 스레드 기반 상태 관리가 핵심이다

결정론적 스레드 ID → 샌드박스 매핑 → 메시지 큐 라우팅.
이 단순한 패턴으로 "대화형 코딩 에이전트"의
상태 연속성 문제를 깔끔하게 해결했다.

### 5. "매 턴마다 도구 호출" 패턴

에이전트가 도구 없이 텍스트만 생성하면 세션이 끊길 수 있다.
"매 턴마다 반드시 도구를 호출하라"는 지시는
에이전트의 행동 루프를 강제하는 실전적 테크닉이다.
