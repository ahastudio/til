(2026-02-24)

## 요약

50개 이상의 MVP를 출시한 Harshil Tomar가 정리한 바이브 코딩 실전 가이드.

핵심 메시지는 "최고의 바이브 코더는 코딩을 가장 잘하는 사람이 아니라, 무엇을
만들지 **않을지** 아는 사람"이라는 것이다.

인증, 결제, 파일 업로드, 실시간 시스템 등을 직접 구축하지 말고 검증된 도구를
활용해 핵심 기능 개발에 에너지를 집중하라는 실용적인 조언을 18가지 DO's와
대응하는 DON'Ts로 구성했다.

## 18가지 규칙 (DO's)

## DON'Ts (하지 말아야 할 것)

각 DO에 대응하는 안티패턴이 있다. 핵심만 정리하면:

- 인증을 직접 구축하지 말 것 — MVP에서 가장 큰 시간 낭비
- Raw CSS 작성하지 말 것 — 2026년에 Tailwind 없이 CSS를 쓰는 건 비효율
- 상태 관리를 과도하게 설계하지 말 것 — 사용자 10명 전에 1000만 명용 아키텍처
  불필요
- 커스텀 REST API를 조기에 만들지 말 것 — 검증 전 인프라 구축은 낭비
- 수동 배포하지 말 것 — 휴먼 에러의 원천
- Raw SQL 남용하지 말 것 — 유지보수와 보안 면에서 ORM이 유리
- 자체 결제 시스템 만들지 말 것 — PCI 컴플라이언스만 수개월
- 자체 검색 엔진 만들지 말 것 — Algolia, Typesense, Meilisearch 활용
- 로깅/모니터링 건너뛰지 말 것 — 눈 감고 운전하는 것과 같음
- API 키 하드코딩하지 말 것 — GitHub이 자동 스캔하여 키 무효화
- 파일 업로드 직접 구현하지 말 것 — 프로덕션에서 예측 불가능하게 깨짐
- main에 직접 푸시하지 말 것 — 솔로 개발자도 feature branch 사용
- 실시간 시스템 혼자 만들지 말 것 — Supabase Realtime, Pusher, Partykit 활용
- 성능 무시하지 말 것 — 느린 앱은 죽어가는 앱
- 사용자가 알아서 이해할 거라 가정하지 말 것 — 온보딩 없으면 이탈
- 리팩터링 미루지 말 것 — 기능 2~3개마다 정리
- 의사결정을 기억에 의존하지 말 것 — 문서화 필수
- 완벽주의에 빠지지 말 것 — 불완전해도 출시가 우선

## 분석

### 대상 독자와 맥락

저자는 미국, 인도, 두바이, 호주의 클라이언트를 대상으로 50개 이상의 MVP를 출시한
프리랜서/에이전시 개발자다. 이 글의 대상은 **혼자 또는 소규모 팀으로 빠르게
MVP를 출시해야 하는 인디 해커, 창업자, 프리랜서**다. 엔터프라이즈 개발이나
대규모 팀 환경과는 맥락이 다르다.

### 기술 스택의 편향

18가지 규칙이 사실상 **Next.js + Vercel 생태계 원스톱 가이드**로 수렴한다.

- 프레임워크: Next.js (Server Components, Server Actions)
- 스타일링: Tailwind + shadcn/ui
- 상태: Zustand
- API: tRPC
- ORM: Prisma
- 배포: Vercel
- 인증: Clerk / Supabase Auth
- DB: Supabase / Neon / Railway (모두 Postgres)

이것은 2024~2026년 인디 해커 커뮤니티의 **지배적 기술 스택(dominant stack)**을
그대로 반영한다. 다른 생태계(Rails, Laravel, Django, Go 등)에서도 동일한 원칙이
적용되지만 구체적 도구 추천은 달라진다.

### "무엇을 만들지 않을지 아는 것"의 의미

이 글의 핵심 통찰은 개발이 아니라 **의사결정**에 관한 것이다. 코드를 작성하기
전에 내리는 결정이 프로젝트의 속도와 생존을 결정한다는 주장이다.

이는 소프트웨어 엔지니어링의 오래된 원칙 — "가장 좋은 코드는 작성하지 않은
코드다(The best code is no code at all)" — 을 바이브 코딩 시대에 재해석한
것이다.

## 비평

### 강점

1. **실전 경험 기반**: 이론이 아니라 50개 이상의 MVP 출시 경험에서 나온 구체적
   조언이다. "2주 동안 인증 만드는 창업자를 봤다" 같은 사례가 설득력을 준다.

2. **DO/DON'T 대칭 구조**: 같은 주제를 양면에서 다뤄 메시지를 강화한다. 특히
   DON'T 섹션에서 "왜 안 되는지"를 구체적으로 설명하는 점이 좋다.

3. **실용적 시간 감각**: "Figma에서 작동하는 UI까지 2~3시간", "Stripe 통합에
   45분", "README 작성에 20분" 등 구체적 시간 추정이 의사결정에 도움을 준다.

### 약점

1. **생존자 편향(Survivorship Bias)**: 50개 이상의 MVP를 출시했다는 것은 빠르게
   만들고 넘어간다는 의미다. 이 중 몇 개가 지속 가능한 비즈니스가 되었는지는
   언급하지 않는다. MVP를 빨리 만드는 것과 성공하는 제품을 만드는 것은 다른
   문제다.

2. **도구 의존성의 위험**: "도구를 신뢰하라(trust the tools)"는 메시지가
   반복되지만, 외부 서비스에 대한 과도한 의존은 그 자체가 리스크다. Clerk의 가격
   변경, Vercel의 요금 정책 변화, 서비스 중단 등에 대한 논의가 없다. "Build vs
   Buy" 결정에서 Buy의 장점만 강조한다.

3. **확장성에 대한 무관심**: "사용자 10명 전에 1000만 명용 아키텍처를 만들지
   말라"는 맞지만, 초기 아키텍처 결정이 나중에 마이그레이션 비용으로 돌아오는
   경우도 많다. tRPC에서 REST API로의 전환, Zustand에서 더 복잡한 상태 관리로의
   전환 비용에 대한 현실적 논의가 부족하다.

4. **"바이브 코딩 2.0"이라는 명명**: 원래 Andrej Karpathy가 정의한 바이브 코딩은
   "코드를 신경 쓰지 않고 AI에 맡기는 것"이다. 이 글은 사실상 **도구 선택
   가이드**이지, AI와의 협업 방식에 대한 논의가 아니다. "바이브 코딩"이라는
   이름을 빌려왔지만 실제 내용은 기존의 "올바른 도구 선택" 조언에 가깝다.

5. **마케팅적 성격**: "상위 1% 빌더" 같은 표현, cal.com 링크로 끝나는 구조 등
   콘텐츠 마케팅의 전형적 패턴이다. 정보 자체는 유용하지만, 에이전시 홍보 목적이
   있음을 인식할 필요가 있다.

## 인사이트

### 1. "의사결정 프레임워크"로 읽어야 한다

이 글의 진짜 가치는 개별 도구 추천이 아니라, **"에너지를 어디에 쓸 것인가"라는
의사결정 프레임워크**에 있다. 도구는 바뀌지만, "차별화되지 않는 부분에 시간을
투자하지 않는다"는 원칙은 지속된다. AWS가 이를 "Undifferentiated Heavy
Lifting"이라 불렀던 것과 같은 맥락이다.

### 2. 바이브 코딩의 진화 방향이 보인다

Karpathy의 바이브 코딩(2025.1) → a16z의 소비자 바이브 코딩 분석(2025) → 이
글(2026.2)로 이어지는 흐름을 보면, 바이브 코딩 담론이 **"AI가 코드를
생성한다"에서 "어떤 코드를 생성하게 할 것인가"로 이동**하고 있다. 코드 생성
자체는 당연한 것이 되었고, 이제 문제는 아키텍처 결정과 도구 조합이다.

### 3. "45분 통합"의 함정

저자가 반복하는 "45분이면 된다"식 시간 추정은 **해피 패스(happy path)**
기준이다. 실제로는 엣지 케이스 처리, 기존 시스템과의 통합, 에러 핸들링 등으로 수
배의 시간이 든다. 이 간극을 인식하는 것이 중요하다.

### 4. MVP 이후가 진짜 전쟁터

이 글이 다루지 않는 가장 큰 주제는 **MVP 이후**다. 빠르게 출시한 뒤
Product-Market Fit을 찾는 과정, 기술 부채 청산, 팀 확장 시의 아키텍처 전환 등이
실제로 더 어렵고 중요한 문제다. MVP 출시 속도 최적화만으로는 "상위 1% 빌더"가 될
수 없다.

### 5. Kent Beck의 Augmented Coding과 대비

Kent Beck이 제안한 [Augmented Coding](./augmented-coding.md)은 코드 품질,
테스트, 복잡성 관리에 집중한다. 이 글은 정반대로 "완벽보다 출시"를 강조한다. 두
접근법은 서로 다른 단계에 적합하다 — 검증 단계에서는 이 글의 접근이, 제품이
시장에 맞은 후에는 Beck의 접근이 맞다. **둘 다 필요한 것이지 하나만 선택하는
것이 아니다.**

### 6. 도구 목록의 유통기한

2024년에는 이 목록에 없던 도구들(Clerk, UploadThing 등)이 2026년에는 당연한
선택이 되었다. 반대로 2024년의 인기 도구 중 일부는 사라졌다. **이 글의 구체적
도구 추천은 6~12개월의 유통기한**을 가진다. 원칙은 남지만 도구는 바뀐다.
