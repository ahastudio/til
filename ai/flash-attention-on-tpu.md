# Flash Attention을 TPU에 강제로 올려보고 배운 것들

원문: <https://archerzhang.me/forcing-flash-attention-onto-a-tpu>

Archer Zhang의 LLM 내부 시리즈 5편.
Part 4에서 Triton으로 GPU에 올린 Flash Attention 커널을, 이번에는 JAX로 TPU에
포팅한다. "알고리즘 알고, 커널 짜봤고, JAX는 numpy 컴파일 버전이니까 금방
끝나겠지"라는 예상은 완전히 빗나간다.

## 요약

### JAX/XLA 프로그래밍 모델

Triton은 포인터 산술로 메모리를 직접 제어한다.
JAX는 한 단계 위 추상화다. `matmul`, `exp`, `where` 같은 연산을 표현하면
XLA 컴파일러가 하드웨어 매핑을 결정한다.

`jax.jit` 호출 시 흐름:

1. Python 함수를 추상 값으로 한 번 실행해 연산을 트레이싱
2. 트레이스 결과가 HLO(High-Level Operations) 그래프로 변환
3. XLA가 elementwise 연산 시퀀스를 단일 커널로 퓨전
4. GPU는 PTX, TPU는 VLIW 명령어로 컴파일

핵심 차이: JAX 배열은 불변(immutable)이다.
Triton의 `tl.store(ptr, val)`처럼 원하는 곳에 쓸 수 없다.
`jax.lax.fori_loop`는 상태를 받아 새 상태를 반환하는 순수 함수 본문을 요구하고,
`dynamic_update_slice`는 슬라이스가 교체된 새 배열을 반환한다.

### TPU에서의 Flash Attention 구현

알고리즘 자체(running max, running sum, accumulator의 타일별 갱신)는 동일하다.
바뀌는 것은 JAX의 함수형 모델이 코드를 형성하는 방식이다.

| Triton (Part 4)                       | JAX (이번 글)                                       |
| ------------------------------------- | --------------------------------------------------- |
| `tl.store(out_ptrs, acc, mask=...)`   | `lax.dynamic_update_slice(out, tile, (start, 0))`   |
| `for kv_start in range(...)`          | `jax.lax.fori_loop(0, num_k_blocks, k_body, state)` |
| 가변 `acc += tl.dot(weights, v)`      | 새 상태 반환: `return (new_max, new_sum, new_acc)`   |
| 포인터 산술로 타일 주소 계산          | 컴파일러가 데이터 이동 처리                         |

### 벤치마크: 충격적인 결과

TPU v5e, JAX 0.7.2, float32, 단일 헤드 (n, 64) 기준:

| n     | fused 표준(ms) | flash(ms) | 비율      |
| ----- | -------------- | --------- | --------- |
| 512   | 0.076          | 0.082     | 1.1x 느림 |
| 1024  | 0.055          | 0.133     | 2.4x 느림 |
| 4096  | 0.072          | 2.509     | 35x 느림  |
| 16384 | 4.445          | 89.567    | 20x 느림  |

Flash Attention이 fused 표준 어텐션보다 **35배 느리다.**
GPU에서는 존재 이유 그 자체였던 최적화가, TPU에서는 참혹하게 진다.

### vmap으로 45배 개선

원인: `fori_loop`가 Q 블록 간 독립성을 컴파일러에게 숨겼다.
각 Q 블록은 동일한 K/V를 읽고, 독자적 softmax 상태를 유지하며,
서로 다른 출력 행에 쓴다. 데이터 의존성이 전혀 없다.

`jax.vmap`으로 외부 Q 루프를 대체하면, XLA에게 "이 블록들은 독립적"이라고
직접 알려주는 셈이 된다.

| n     | fori(ms) | vmap(ms) | 개선  |
| ----- | -------- | -------- | ----- |
| 512   | 0.074    | 0.065    | 1.1x  |
| 4096  | 2.510    | 0.178    | 14.1x |
| 16384 | 89.538   | 1.997    | 44.8x |

같은 알고리즘, 같은 타일, 같은 수학. 유일한 차이: 컴파일러에게 전달한 정보.

### TPU 하드웨어 구조

TPU v5e 칩 하나:

- **MXU** 4개: 128×128 시스톨릭 어레이. GPU의 텐서 코어가 "부가 가속기"라면,
  MXU는 주 연산 엔진이다. 16,384개 곱셈-누적 셀이 매 클럭 데이터를 흘려보낸다.
- **VPU** 1개: elementwise 연산, 리덕션 담당. MXU 대비 ~10배 느리다.
- **스칼라 유닛**: 제어 흐름, DMA 조율.
- **VMEM** ~128 MB: 칩 전체 공유 온칩 SRAM.
  GPU SM의 공유 메모리 ~164 KB 대비 800배.

스레드가 없다. GPU는 워프 전환으로 메모리 지연을 숨기지만,
TPU는 파이프라이닝으로 해결한다. MXU가 현재 타일을 연산하는 동안
DMA 엔진이 다음 타일을 프리페치한다.

### 시스톨릭 어레이의 작동 방식

C = A @ B에서 (Weight-Stationary 방식):

- B가 셀에 고정되고, A가 왼쪽에서 스트리밍
- 부분합이 아래로 흐름
- **스태거(stagger)**: A[m][k]가 k번째 행에 k 사이클 지연 진입.
  위에서 내려오는 부분합 도착 타이밍과 정확히 동기화하기 위해서다.
- 총 사이클: M + K + N - 2. 파이프라인이 항상 가득 차고, 버블이 없다.

저자는 틱 기반 에뮬레이터를 직접 구현해 이를 검증했다.

### Pallas: 컴파일러를 이기려면

Splash Attention(Google의 TPU 최적화 Flash Attention)은 Pallas로 작성된다.
순수 JAX로는 표현할 수 없는 세 가지:

1. **DMA 파이프라이닝**: 연산과 메모리 전송의 오버랩(이중 버퍼링)
2. **MXU 매칭 타일링 + 인과 스킵**: 2D 그리드로 마스크된 타일 완전 건너뛰기
3. **명시적 VMEM 배치**: BlockSpec 선언으로 프리페치 제어

## 분석

### 왜 Flash Attention이 TPU에서 실패하는가

GPU에서 Flash Attention이 필수인 이유: SM 공유 메모리 ~164 KB에 n×n 스코어
행렬이 절대 안 들어간다. n=256부터 이미 초과한다.

TPU는 상황이 근본적으로 다르다:

- VMEM ~128 MB → 스코어 행렬이 n=8192까지 온칩에 들어감
- XLA가 자동으로 퓨전 → 중간 행렬이 HBM에 내려가지 않음
- MXU가 원래 타일 프로세서 → 소프트웨어 타일링이 하드웨어 기본 동작을 재구현

즉, GPU에서 "소프트웨어가 반드시 해야 하는 일"을 TPU에서는
"하드웨어+컴파일러가 이미 하고 있는 일"이었다.

### fori_loop vs vmap: 정보 이론적 관점

이것은 단순한 성능 팁이 아니다.
컴파일러 최적화의 본질에 대한 통찰이다.

`fori_loop`는 XLA에게 "순차적으로 실행하라"고 말한다.
캐리 상태(carried state)가 있으니 컴파일러는 반복 간 독립성을 추론할 수 없다.

`vmap`은 "이 함수를 배치에 매핑하라"고 말한다.
독립성이 추상화 자체에 내장되어 있다.

같은 수학, 45배 차이.
**컴파일러에게 전달하는 의도(intent)가 곧 성능이다.**

### MXU 활용률이 말해주는 것

에뮬레이터 분석에서 flash attention의 MXU 활용률은 ~20%,
표준 어텐션은 ~94%다.

flash의 타일별 matmul 내부 차원이 최대 128이라 시스톨릭 파이프라인이
128 스텝만 돈다. 표준 어텐션의 weights @ V는 내부 차원이 n(=512)이라
파이프라인이 512 스텝 동안 유용한 일을 한다.

큰 matmul 하나가 작은 matmul 여럿보다 하드웨어를 훨씬 효율적으로 쓴다.
이것이 시스톨릭 어레이의 근본 특성이다.

## 비평

### 강점

1. **실패를 통한 학습 서사**: "GPU에서 먹혔으니 TPU에서도 되겠지"라는 자연스러운
   가정이 완전히 깨지는 과정을 솔직하게 보여준다. 벤치마크 숫자가 무자비하다.

2. **에뮬레이터 구현**: 시스톨릭 어레이를 말로 설명하는 글은 많지만,
   틱 기반 에뮬레이터를 직접 구현해 스태거 타이밍까지 검증한 것은 드물다.
   추상적 이해와 실제 이해의 간극을 보여준다.

3. **vmap 인사이트의 일반성**: JAX/TPU에 국한되지 않는다.
   "컴파일러에게 의도를 명시적으로 전달하라"는 원칙은 모든 컴파일러 기반
   프레임워크(TorchScript, TVM, MLIR 등)에 적용된다.

### 약점

1. **HLO 덤프 부재**: "XLA가 아마 이렇게 할 것이다"라는 추론이 반복되지만,
   실제 HLO를 덤프해서 검증하지 않았다. 저자도 이를 인정하지만,
   "educated guess"와 "verified fact"는 다르다.

2. **단일 헤드, d=64 한정**: 실제 트랜스포머는 multi-head(d=128, 32~128 heads)를
   쓴다. VMEM 128 MB를 여러 헤드가 공유하면 크로스오버 포인트가 상당히 내려간다.
   이 점을 언급은 하지만 실험하지 않았다.

3. **Pallas 실험 미완**: "시도했으나 코드가 살아남지 못했다"고 한다.
   이해는 가지만, Splash Attention과의 정량적 비교 없이는 "어디까지 순수 JAX로
   갈 수 있는가"의 한계를 정확히 그을 수 없다.

## 인사이트

### 1) 최적화는 하드웨어 컨텍스트 없이는 의미가 없다

Flash Attention은 "좋은 알고리즘"이 아니라 "GPU SRAM이 작다는 제약에 대한
해법"이다. 제약이 달라지면(TPU VMEM ~128 MB) 해법의 가치도 달라진다.

이것은 소프트웨어 엔지니어링 전반에 적용되는 원칙이다.
캐시 최적화, 배치 처리, 연결 풀링 등 모든 최적화 기법은 특정 병목에 대한 응답이다.
병목이 없는 곳에 최적화를 적용하면 복잡성만 늘어난다.

### 2) 추상화 수준의 선택이 성능을 결정한다

Triton: 프로그래머가 컴파일러다. 무엇이 어디에 사는지 직접 지정한다.
JAX: 컴파일러와 대화한다. 의도를 잘 표현할수록 좋은 코드가 나온다.

`fori_loop` → `vmap` 전환은 코드 구조의 변경이 아니라
**의사소통 방식의 변경**이다.
이것은 "선언적 vs 명령형" 논쟁의 가장 극적인 실례다.

### 3) 하드웨어가 이미 하고 있는 일을 소프트웨어로 재구현하지 마라

TPU의 MXU는 128×128 시스톨릭 어레이다. 이것 자체가 타일 프로세서다.
Flash Attention이 소프트웨어로 하는 일(타일, 스트림, 누적)을
하드웨어가 매 클럭 자동으로 수행한다.

교훈: 최적화를 적용하기 전에 "이것을 이미 누가(하드웨어, 컴파일러, 런타임)
하고 있지 않은가?"를 먼저 확인하라.

### 4) 컴파일러의 시야(visibility)가 곧 최적화 상한이다

XLA는 데이터플로우 분석, 연산자 퓨전, 메모리 계획을 수행한다.
그러나 `fori_loop`의 캐리 상태를 통해서는 반복 간 독립성을 추론할 수 없다.
`vmap`은 독립성을 타입 수준에서 보장한다.

**컴파일러가 볼 수 없는 것은 최적화할 수 없다.**
이것이 45배 차이의 근본 원인이다.

### 5) "같은 추상화, 다른 주체"라는 프레임

타일링은 어디에나 있다. 차이는 누가 하느냐다.

- TPU: 하드웨어(MXU)가 타일 연산, 컴파일러(XLA)가 스케줄링, 컴파일러가 메모리 관리
- GPU: 소프트웨어(Triton/CUDA)가 타일 연산, 프로그래머가 스케줄링, 프로그래머가 메모리 관리

이 프레임은 새로운 하드웨어를 평가할 때 즉시 적용할 수 있다:
"이 칩에서 타일링/퓨전/메모리 관리의 주체는 누구인가?"

### 6) 실패의 교육적 가치

저자가 명시적으로 말한다: "왜 졌는지 이해하는 것이 GPU에서 이긴 것보다
두 아키텍처 모두에 대해 더 많이 가르쳐줬다."

이것은 단순한 겸손이 아니다.
성공은 "이 방법이 작동한다"만 확인시켜 주지만,
실패는 "왜 이 방법이 작동하는가"의 전제 조건을 드러낸다.
Flash Attention의 전제 조건이 "온칩 SRAM이 작다"였음을 TPU에서의 실패가
비로소 명확히 한 것이다.
