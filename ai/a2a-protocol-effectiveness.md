# Google A2A 프로토콜, 실제로 효과가 입증된 사례가 있는가

## 요약

Google이 2025년 4월 발표한 A2A(Agent2Agent) 프로토콜은 1년 만에
150개 이상 기관의 지원을 받고 Microsoft, AWS 같은 주요 클라우드
플랫폼에 통합되며 2026년 초 v1.0을 정식 출시했다.
그러나 “실제로 효과가 입증된 도입 사례”를 기준으로 조사한 결과,
성장 지표는 사실이지만 정량적 성과나 구체적 기업 사례로
뒷받침되는 “효과 입증”은 확인되지 않았다.
오히려 인터넷에 떠도는 구체적 성공 사례 다수가 검증 과정에서
근거 없음으로 드러났다.

이 문서는 `deep-research` 워크플로(6개 검색 각도, 22개 출처,
86개 주장 추출, 상위 25개에 대한 3표 반박 검증)를 통해 얻은
결과를 정리한 것이다.

## 확인된 사실

### 성장세는 실재한다

- 지원 기관이 출시 시점 50개 이상에서 1년 만에 150개 이상으로
  늘었고, GitHub 스타 22,000개 이상, 5개 언어 SDK를 갖췄다.
- Microsoft는 Azure AI Foundry와 Copilot Studio에, AWS는 Bedrock
  AgentCore Runtime에 A2A를 직접 통합했다.
- 2026년 초 서명된 에이전트 카드(signed Agent Card)를 포함한
  v1.0이 정식 출시됐다.

이 수치들은 Linux Foundation 공식 보도자료, Microsoft·AWS 자체
블로그, 위키백과 등 1차 소스에서 서로 일치해 신뢰도가 높다.
다만 이는 “플랫폼이 A2A라는 기능을 지원한다”는 사실이지, 특정
고객사가 A2A로 실제 업무를 처리하고 있다는 사실은 아니다.

### “프로덕션 사용” 주장은 업종 단위에 머문다

Linux Foundation 보도자료는 공급망, 금융, 보험, IT 운영 분야에서
프로덕션 사용이 이뤄지고 있다고 주장하지만, 이를 직접 확인한
결과 구체적인 기업명이나 비용 절감률·처리 시간 단축 같은 정량적
성과는 전혀 제시되지 않았다.
DigitalOcean과 Merge.dev의 독립 기술 블로그도 각각 “A2A는 아직
초기 채택 단계”라거나 “제시된 활용 사례가 가상의 시나리오에
불과하다”는 점을 스스로 인정했다.

### MCP와의 차별점은 정설이지만, 성과로 이어진 증거는 없다

“MCP는 에이전트와 도구를 수직으로 연결하고, A2A는 에이전트와
에이전트를 수평으로 협업시킨다”는 구도는 a2a-protocol.org 공식
비교 문서를 포함해 업계 전반이 반복하는 설명이다.
그러나 이 구도가 실제 비즈니스 성과 차이로 이어졌음을 보여주는
독립적 사례나 벤치마크는 이번 조사에서 하나도 발견되지 않았다.

### 개발자 커뮤니티의 실제 반응은 회의적이다

추가 조사에서 확인한 Hacker News 스레드
“[Is anyone using the A2A protocol?](https://news.ycombinator.com/item?id=48582679)”에는
실제 사용 경험을 묻는 질문에 뚜렷한 긍정 사례가 달리지 않았다.
초기 발표 직후 올라온
“[Do you think Google's A2A protocol will catch on?](https://news.ycombinator.com/item?id=43650326)”
스레드에서도 회의적 반응이 두드러졌다.
엔터프라이즈 인프라 스타트업 Credal.ai가 정리한
“[What happened to A2A Protocol?](https://www.credal.ai/blog/what-happened-to-a2a-protocol)”는
A2A의 채택 부진 원인을 두 가지로 짚는다.
하나는 사용 사례 자체가 MCP로도 이미 충분히 처리된다는 점이고,
다른 하나는 구현 부담이다.
A2A가 내세운 장기 실행 태스크 관리, 상태 유지 통신, 엔터프라이즈급
보안, 에이전트 디스커버리 같은 기능은 실제로는 MCP도 이미 상당
부분 지원하고 있었다는 것이다.
반면 MCP는 개발자가 Claude를 Notion, Jira, GitHub에 10분 안에
연결할 수 있을 만큼 학습 곡선이 낮았던 반면, A2A는 통신 계층
전체를 직접 구축해야 하는 복잡한 오케스트레이션 개념을 요구했다.
이 글은 심지어 A2A가 원래 목표로 삼았던 엔터프라이즈 시장에서도
정작 보안·인프라 장치 없이 MCP를 더 빠르게 채택하는 역설적
상황을 지적한다.
Medium에 실린
“[Everything wrong with Agent2Agent (A2A) Protocol](https://medium.com/@ckekula/everything-wrong-with-agent2agent-a2a-protocol-7e5ae8d4ab2b)”과
“[A2A Is a Nonsense Protocol](https://medium.com/@balajibal/a2a-is-a-nonsense-protocol-49a0c2a0f84c)”
같은 개인 블로그성 비판 글도 다수 발견됐는데, 이런 글들은
저자 개인의 의견이라는 한계가 있어 근거로 채택하기보다는
“채택 실패”라는 정서가 개발자 커뮤니티에 널리 퍼져 있다는
정황 증거 정도로만 취급하는 것이 적절하다.

### Agent Card 구현 자체의 기술적 결함

GitHub 이슈
“[Proposal: Agent Identity Verification for Agent Cards](https://github.com/a2aproject/A2A/issues/1672)”는
스펙 저장소에 실제로 제기된 문제로, Agent Card가 발급 주체를
암호학적으로 검증할 프로토콜 차원의 방법이 없다는 점을 지적한다.
Red Hat Developer의
“[How to enhance Agent2Agent (A2A) security](https://developers.redhat.com/articles/2025/08/19/how-enhance-agent2agent-security)”도
같은 문제를 다루면서, Agent Card 관리가 버전 관리 부재·갱신
지연·부실한 인증·민감한 메타데이터 노출 같은 실무적 결함을
안고 있다고 짚는다.
A2A의 디스커버리 방식이 풀(pull) 방식이라 클라이언트가 카드를
다시 가져와 버전이 달라졌는지 스스로 확인해야 하는 구조인데,
SDK 자체는 카드가 바뀌어도 비교하거나 경고하지 않고 받은 JSON을
그대로 파싱해 서빙한다.
또한 A2A가 점대점(point-to-point) HTTP 연결에 의존하는 구조라서
에이전트 수가 늘어날수록 연결 수가 N제곱에 가깝게 증가할 수
있다는 지적도 있는데, 이는 스펙이 아직 대규모 다자간 토폴로지를
충분히 고려하지 않았다는 뜻으로 읽힌다.

## 반박된 주장들

검증 과정에서 다음과 같은 구체적 성공 사례 주장들이 3표 중
2표 이상의 반박을 받아 근거 불충분으로 기각됐다.

- “A2A가 150개 기관의 프로덕션 환경에서 서로 다른 플랫폼 간
  실제 태스크를 라우팅하고 있다” (1-2)
- “Microsoft, AWS, Salesforce, SAP, ServiceNow가 A2A를 프로덕션
  환경에서 운영 중이다” (0-3)
- “Comparus가 IBM watsonx.ai와 A2A를 통합해 처리 속도와 서비스
  품질을 개선했다” (0-3)
- “50개 이상의 기술 파트너가 A2A 파트너십을 발표했다”는 주장은
  실제로는 프로토콜 지지 선언일 뿐 검증된 프로덕션 구현이 아니다
  (1-2)
- Microsoft Semantic Kernel의 TravelManager 에이전트 사례처럼
  유일하게 구체적으로 언급된 사례조차 검증 단계에서 근거
  불충분 판정을 받았다 (0-3)

이 중 다수는 언론 기사(TheNextWeb)나 벤더 블로그(TrueFoundry)에서
유래한 주장으로, 원출처를 추적했을 때 뒷받침하는 근거가
확인되지 않았다.

## 알려진 보안 취약점

추가 조사에서는 A2A 스펙 자체를 겨냥한 보안 연구자들의 구체적
공격 시나리오가 다수 확인됐다.
이는 기존 조사가 다루지 않았던 영역으로, “효과 입증 사례가
없다”는 결론에 더해 “위험이 이미 문서화돼 있다”는 사실도 함께
짚을 필요가 있다.

### 에이전트 신원 위조와 태스크 하이재킹

Cloud Security Alliance의
“[Threat Modeling Google's A2A Protocol with the MAESTRO Framework](https://cloudsecurityalliance.org/blog/2025/04/30/threat-modeling-google-s-a2a-protocol-with-the-maestro-framework)”는
A2A 도입 직후부터 가짜 에이전트 광고(fake agent advertisement)와
무단 등록 같은 프로토콜 특유의 위협을 지적했다.
공격자가 위조된 Agent Card를 만들어 신뢰받는 에이전트를 사칭하면,
이를 신뢰한 클라이언트가 가짜 서버에 태스크를 위임하게 되고
그 결과 태스크 하이재킹, 데이터 유출, 에이전트 사칭으로 이어질
수 있다는 것이 여러 보안 블로그(Legion Security, Solo.io,
SecureW2)에서 공통으로 지적하는 구조다.
근본 원인은 Agent Card의 신원 검증이 프로토콜 내부가 아니라
외부 메커니즘에 위임돼 있다는 설계 자체에 있다.

### Agent Session Smuggling — 실증된 공격

Palo Alto Networks Unit 42가 공개한
“[When AI Agents Go Rogue: Agent Session Smuggling Attack in A2A Systems](https://unit42.paloaltonetworks.com/agent-session-smuggling-in-agent2agent-systems/)”는
이론적 위협 모델링을 넘어 실제로 작동하는 공격을 시연했다는
점에서 이번 조사에서 확인된 가장 구체적인 보안 증거다.
연구진은 Google의 Agent Development Kit로 두 가지 개념 증명
공격을 구현했다.
첫 번째는 악성 리서치 어시스턴트 에이전트가 금융 어시스턴트
클라이언트 에이전트를 속여 시스템 프롬프트, 도구 설정, 대화
기록 같은 민감 정보를 무해해 보이는 후속 질문을 통해 빼내는
방식이었다.
두 번째는 한 단계 더 나아가, A2A 프로토콜의 정상 기능인
input-required 상태(원격 에이전트가 클라이언트에게 추가 정보를
요청하는 절차)에 숨긴 지시를 실어 보내 금융 어시스턴트가
승인되지 않은 주식 거래를 실행하도록 유도했다.
연구진은 이 공격이 프로토콜의 결함을 이용하는 것이 아니라
에이전트 간에 이미 존재하는 암묵적 신뢰 관계를 무기화하는
것이라고 설명하며, 하나의 손상된 에이전트가 4시간 안에
하위 의사결정의 87%를 오염시켰다고 보고했다.
이 수치가 실험실 환경의 개념 증명에서 나온 것이라는 한계는
있지만, “프로덕션 성공 사례는 검증되지 않았어도 프로덕션 위험
시나리오는 이미 검증됐다”는 비대칭은 그 자체로 주목할 만하다.

## 분석

### A2A의 실제 경쟁 상대는 MCP가 아니라 사내 통합 코드다

A2A가 해결하려는 문제, 즉 서로 다른 벤더가 만든 에이전트끼리
표준화된 방식으로 통신하는 문제는 지금까지 조직 대부분이 사내
통합 코드나 벤더별 SDK로 임시방편으로 풀어왔다.
이 경우 A2A의 진짜 경쟁 상대는 다른 에이전트 프로토콜이 아니라
“이미 동작하고 있어서 굳이 바꿀 이유가 없는 자체 구현”이다.
새 표준을 도입하는 비용(마이그레이션, 학습, 검증)이 기존 통합
코드를 유지하는 비용보다 낮아야 채택이 일어나는데, 이번 조사에서
확인된 “익명화된 업종 단위 사례”들은 이 비용 대비 편익 계산이
아직 조직 차원에서 뚜렷하게 검증되지 않았다는 정황과 맞물린다.

### 오픈소스 표준의 초기 채택 지표는 항상 벤더 발표에 오염돼 있다

A2A의 성장 수치(지원 기관 수, GitHub 스타, 파트너 발표)는 전부
Google이 주도하거나 파트너사가 자발적으로 발표한 수치다.
이는 표준 자체의 문제가 아니라, 신생 오픈소스 표준이 겪는
구조적 문제다.
독립적인 제3자 조사기관이나 학계가 채택 실태를 측정하기까지는
시간이 걸리므로, 초기 국면에는 표준을 제안한 당사자와 그
파트너사가 내놓는 수치가 사실상 유일한 정보원이 된다.
이 시기에 나오는 모든 “성장세” 서술은 이해관계자가 이해관계자를
인용하는 구조에서 벗어나기 어렵다.

### 표준 전쟁의 역사적 패턴에서 A2A의 위치

“대기업이 주도한 오픈 표준”이 실제로는 생태계 지배력을 강화하는
도구로 쓰인 전례는 드물지 않다.
SOAP는 원래 Microsoft가 주도해 만든 표준이었고, 이전 세대의
CORBA/DCOM 표준 전쟁에서 이어진 연장선에 있었다.
[The History of the Web의 SOAP vs REST 정리](https://thehistoryoftheweb.com/soap-rest-odds/)에
따르면 Salesforce, eBay, Yahoo 같은 초기 오픈 API 제공자들이
SOAP를 채택했지만, 이후 Roy Fielding을 중심으로 한 REST 진영이
“더 단순하고 웹에 더 잘 맞는다”는 논리로 SOAP의 복잡성을
공략하면서 결국 REST가 사실상 표준 지위를 얻었다.
이 역사에서 얻을 수 있는 교훈은, 대기업이 주도하고 컨소시엄이
승인한 표준이라고 해서 반드시 승리하지 않으며, 오히려 구현
부담이 낮은 경쟁 표준에 자리를 내주는 경우가 반복돼 왔다는
점이다.

A2A와 MCP의 관계는 이 패턴과 흥미롭게 겹친다.
Credal.ai의 분석이 지적하듯, A2A는 스펙 차원에서 더 많은 기능
(장기 실행 태스크, 상태 유지, 엔터프라이즈 보안, 디스커버리)을
내세웠지만 정작 시장에서 더 빠르게 퍼진 쪽은 구현이 쉬운 MCP였다.
이는 SOAP(기능은 풍부하지만 무겁다)와 REST(기능은 적지만
가볍다)의 구도를 연상시킨다.
다만 결정적 차이도 있다.
SOAP-REST 전쟁은 두 표준이 같은 문제(원격 서비스 호출)를 놓고
경쟁한 것이었지만, A2A와 MCP는 공식적으로는 다른 층위(에이전트
간 협업 vs 에이전트-도구 연결)를 겨냥한다고 알려져 있다.
그럼에도 실제 개발자들이 MCP만으로 A2A가 약속한 기능 상당수를
충족하고 있다는 관찰(Credal.ai, HN 코멘트)은, 두 표준이 층위
분리라는 공식 설명보다 훨씬 더 직접적으로 경쟁하고 있을
가능성을 시사한다.

Google이 A2A를 통해 얻는 전략적 이익을 이 틀에서 보면, 표준
자체의 승패와 무관하게 최소 두 가지가 확보된다.
첫째, “에이전트 상호운용성 표준을 최초로 제안한 기업”이라는
서사적 지위이며, 이는 향후 에이전트 플랫폼 경쟁에서 기술
리더십을 주장하는 근거로 계속 쓰일 수 있다.
둘째, Google Cloud와 Vertex AI/Agentspace 생태계에 A2A를
먼저 통합함으로써, 설령 시장 전체가 A2A를 채택하지 않더라도
Google 자체 생태계 안에서는 이미 이익을 실현한 상태라는 점이다.
표준을 제안한 기업이 표준 자체의 성패와 무관하게 선점 효과를
누리는 구도는 과거 표준 전쟁에서도 반복적으로 관찰된 패턴이다.

### 상호운용성 문제는 지금 시급한가, 아니면 선제적 표준화인가

A2A가 해결하겠다고 나선 문제, 즉 서로 다른 벤더의 자율 에이전트가
프로덕션에서 대규모로 협업하는 상황은 이번 조사에서 확인된
바로는 아직 널리 존재하지 않는다.
현재 대부분의 “멀티 에이전트 시스템”은 같은 조직, 같은 프레임워크
안에서 오케스트레이션되는 경우가 많고, 서로 다른 회사가 만든
자율 에이전트끼리 실시간으로 협상하고 작업을 위임하는 시나리오는
사례 조사에서도, Credal.ai 등 업계 분석에서도 뚜렷하게 확인되지
않는다.
이는 A2A가 아직 도래하지 않은 미래의 문제, 즉 “에이전트 경제”가
실제로 형성된 이후에나 절실해질 문제를 미리 표준화하려는
시도일 가능성을 뜻한다.
선제적 표준화 자체가 나쁜 것은 아니다.
HTTP나 TCP/IP도 문제가 대규모로 발생하기 전에 설계된 표준이다.
그러나 선제적 표준화가 성공하려면 표준이 예측한 미래상이
실제로 그 형태로 도래해야 하는데, 지금 시점에는 “에이전트 간
수평 협업”이 정말 조직 경계를 넘어 일어날 것인지, 아니면
조직 내부의 도구 연결(MCP가 이미 처리하는 문제)로 대부분
수렴할 것인지가 아직 불확실하다.

### Linux Foundation 이관은 중립성의 실질인가 상징인가

Google은 2025년 6월
[A2A를 Linux Foundation에 공식 기증](https://developers.googleblog.com/en/google-cloud-donates-a2a-to-linux-foundation/)했고,
[Linux Foundation 보도자료](https://www.linuxfoundation.org/press/linux-foundation-launches-the-agent2agent-protocol-project-to-enable-secure-intelligent-communication-between-ai-agents)는
AWS, Cisco, Google, IBM Research, Microsoft, Salesforce, SAP,
ServiceNow가 참여하는 기술운영위원회(Technical Steering
Committee)가 스펙을 관리한다고 밝혔다.
2025년 8월에는 IBM 주도의 ACP까지
[A2A와 통합](https://lfaidata.foundation/communityblog/2025/08/29/acp-joins-forces-with-a2a-under-the-linux-foundations-lf-ai-data/)되며
거버넌스 참여 폭이 더 넓어졌다.
형식적으로는 이것이 다중 대기업 합의체 구조이므로 Google
단독 통제에서는 벗어난 것이 맞다.
그러나 실질적 중립성과 상징적 중립성을 구분해서 봐야 한다.
첫째, A2A의 핵심 설계(Agent Card 구조, 태스크 상태 머신,
스트리밍 방식)는 이미 Google이 기증하기 전에 확정된 것이며,
이관 이후 위원회가 근본 아키텍처를 재설계했다는 증거는 확인되지
않는다.
즉 거버넌스가 이관된 시점은 “설계는 끝난 뒤, 홍보와 정당성
확보가 필요한 시점”과 일치한다.
둘째, 위원회에 이름을 올린 기업들은 각자 자사 플랫폼에 A2A를
통합하며 이미 이해관계를 갖게 된 파트너들이라, 표준을 근본적으로
재검토하기보다는 유지·보수에 무게가 실릴 유인이 크다.
이는 Linux Foundation 이관 자체가 거짓이라거나 무의미하다는
뜻이 아니라, “누가 다음 버전을 결정하는가”라는 실질적 권력
질문과 “누가 스펙을 소유한다고 공식적으로 표기되는가”라는
상징적 질문을 분리해서 평가해야 한다는 뜻이다.
현재까지 공개된 정보만으로는 후자가 확인됐을 뿐 전자가
근본적으로 바뀌었다는 증거는 부족하다.

### MCP·ACP·AGNTCY와의 경쟁 구도가 수렴하고 있다

2026년 초 시점에는 MCP, A2A, ACP가 모두 Linux Foundation 계열
거버넌스 아래 놓이면서(MCP는 Agentic AI Foundation, A2A와
ACP는 LF AI & Data) 표준 경쟁이 완전한 승자독식으로 끝나기보다
“MCP는 에이전트-도구, A2A는 에이전트-에이전트”라는 2계층
참조 아키텍처로 수렴하는 흐름이 여러 분석
([Zylos Research](https://zylos.ai/research/2026-03-26-agent-interoperability-protocols-mcp-a2a-acp-convergence/),
[AppScale](https://appscale.blog/en/blog/mcp-vs-a2a-vs-acp-agent-interop-standards-2026))에서
공통으로 관찰된다.
그러나 이 수렴 서술 자체도 검증이 필요하다.
“두 계층이 상호 보완적으로 자리 잡았다”는 서술과 “한쪽(MCP)이
다른 쪽(A2A)이 약속한 기능 상당수를 흡수하며 사실상 승리했다”는
서술은 표면적으로 비슷해 보이지만 함의는 전혀 다르다.
이번 조사에서 확인된 개발자 커뮤니티의 반응과 Credal.ai의
분석은 후자에 가까운 그림을 그린다.

## 비평

### “채택 지표”와 “효과 입증”은 다른 질문이다

A2A를 둘러싼 대부분의 보도와 블로그 글은 지원 기관 수, GitHub
스타, 파트너십 발표 같은 채택 지표를 근거로 성공을 암시한다.
그러나 이 지표들은 프로토콜이 “얼마나 알려졌는가”를 말해줄 뿐
“실제로 도입한 조직이 이전보다 나은 결과를 얻었는가”를 말해주지
않는다.
스펙을 지지한다고 선언하는 것과 그 스펙으로 실제 워크로드를
운영하는 것 사이에는 큰 간극이 있는데, 현재 A2A를 둘러싼
담론은 이 간극을 의도적으로 흐리는 경향이 있다.

### 벤더 발표를 인용하는 2차 보도의 검증 실패

TheNextWeb이나 여러 기술 블로그가 인용한 “150개 기관이 프로덕션
운영 중”이라는 식의 문장은 출처를 거슬러 올라가면 Linux
Foundation 보도자료의 업종 나열을 과장해 재서술한 결과로 보인다.
1차 소스는 “공급망, 금융, 보험, IT 운영 분야에서 사용되고
있다”는 정도만 말했을 뿐 “150개 기관 모두가 프로덕션에서
운영 중”이라고 말하지 않았는데, 이 문장이 반복 인용되는
과정에서 근거의 강도가 부풀려졌다.
이는 오픈소스 프로토콜의 초기 홍보 국면에서 흔히 나타나는
왜곡으로, 채택 지표가 성과 지표로 둔갑하는 경로를 보여준다.

### 조사 자체의 시점 한계를 인정해야 한다

이 조사는 v1.0 정식 출시 직후에 이뤄졌다.
표준이 막 안정화된 시점에는 조직들이 아직 파일럿을 검토하는
단계일 가능성이 높고, 정량적 성과가 나오기까지는 통상 분기
단위의 시간이 더 필요하다.
따라서 “지금 이 시점에 효과 입증 사례가 없다”는 이번 조사의
결론을 “A2A가 근본적으로 효과가 없다”는 결론으로 확장하는 것은
성급하다.
이는 표준의 실패를 뜻하는 것이 아니라, 아직 증거가 축적될
시간이 지나지 않았다는 뜻에 가깝다.
다만 그 구분 자체가 현재 유통되는 낙관적 서술에서는 잘 드러나지
않는다는 점이 문제로 남는다.

### 이번 조사 방법론 자체의 지역적 한계

기존 조사와 이번 추가 조사 모두 검색어를 영어로 구성했고,
검색 결과 역시 영어권 매체(TheNextWeb, Medium, Hacker News,
벤더 블로그)와 미국 기업 중심 발표에 크게 치우쳐 있다.
중국의 바이두, 알리바바, 텐센트나 일본의 대형 SI 기업들이
사내 또는 산업 컨소시엄 차원에서 A2A를 실험하거나 도입했을
가능성은 이번 조사 범위 밖에 있다.
특히 중국 시장은 자체 에이전트 프레임워크 생태계가 별도로
발달하고 있어, A2A 채택 여부가 영어권 보도와 무관하게 움직일
수 있다.
이번 조사에서 “구체적 사례가 없다”는 결론은 어디까지나 영어권
공개 소스를 기준으로 한 것이며, 비영어권 소스를 포함한 조사가
이뤄진다면 결론이 달라질 여지를 열어둬야 한다.

### “반박됐다”는 것과 “사례가 존재하지 않는다”는 것은 다르다

이번 조사와 앞선 `deep-research` 조사 모두 3표 검증에서 반박된
주장을 “근거 불충분”으로 처리했는데, 이는 방법론적으로 올바른
처리이지만 그 함의를 과도하게 확장하지 않도록 주의해야 한다.
반박은 “공개된 근거로 이 주장을 뒷받침할 수 없었다”는 뜻이지
“그런 사례가 세상에 존재하지 않는다”는 뜻이 아니다.
엔터프라이즈 파일럿 프로젝트, 특히 금융·보험처럼 이번 조사에서
언급된 산업의 파일럿은 대개 계약상 비밀 유지 조항의 대상이 되고,
성과가 검증되기 전까지는 보도자료조차 나오지 않는 경우가
흔하다.
즉 공개 웹 검색이라는 조사 방법 자체가 구조적으로 “비공개로
진행 중인 성공 사례”를 발견할 수 없게 설계돼 있다.
이 맹점은 이번 조사뿐 아니라 웹 검색에 의존하는 모든 기술
채택 조사가 공유하는 한계이며, “효과 입증 사례가 없다”는
결론을 읽는 독자는 이것이 “존재하지 않음의 증명”이 아니라
“발견하지 못했음의 보고”라는 점을 구분해서 받아들여야 한다.

### A2A vs MCP라는 이분법 자체가 단순화일 수 있다

이번 조사와 다수의 업계 분석([Zylos Research](https://zylos.ai/research/2026-03-26-agent-interoperability-protocols-mcp-a2a-acp-convergence/) 등)이
반복하는 “MCP는 수직, A2A는 수평”이라는 구도는 실무에서는
두 프로토콜이 계층적으로 함께 쓰이는 경우가 많다는 사실을
가릴 위험이 있다.
즉 A2A로 연결된 원격 에이전트가 내부적으로는 MCP 클라이언트를
통해 자신의 도구를 호출하는 조합이 참조 아키텍처로 거론되는데,
이 경우 “A2A가 채택됐는가”라는 질문 자체가 이분법적으로
답하기 어려운 질문이 된다.
어떤 시스템은 외부에는 A2A 인터페이스를 노출하면서 내부
구현에는 A2A 고유 기능을 거의 쓰지 않고 사실상 MCP 위에
얇은 A2A 껍데기만 얹었을 수 있다.
이런 경우 “A2A 지원”이라는 발표와 “A2A의 핵심 가치(에이전트
간 자율적 협상과 위임)를 실제로 활용하고 있다”는 사실 사이에는
큰 차이가 있는데, 이번 조사를 포함해 대부분의 채택 지표는 이
차이를 구분하지 않고 집계된다.
따라서 “A2A 지원 기관 150개”라는 수치도, 그중 몇 개가 A2A의
수평적 협업 기능을 실제로 활용하고 몇 개가 표면적 호환성만
갖췄는지는 알 수 없다.

## 인사이트

### 프로토콜의 성숙도와 생태계의 성숙도는 별개로 움직인다

A2A는 스펙 차원에서는 명백히 성숙해지고 있다.
v1.0 출시, 서명된 에이전트 카드, 다국어 SDK, 대형 클라우드
플랫폼 통합은 모두 실질적인 기술적 진전이다.
그러나 그 스펙 위에서 실제로 검증 가능한 성과를 낸 도입 사례는
아직 축적되지 않았다.
이는 표준 자체의 완성도와, 그 표준이 실제 조직의 문제를
해결했다는 증거 사이에 시차가 존재함을 보여준다.
새로운 상호운용성 표준이 등장할 때 반복되는 패턴으로, 표준이
갖춰진 다음에도 그것을 실제로 프로덕션에 걸고 성과를 측정하는
조직이 나타나기까지는 추가적인 시간이 필요하다.

### “익명화된 성공 사례”는 검증 불가능한 주장의 신호다

Linux Foundation 보도자료가 업종만 언급하고 기업명을 밝히지
않은 것은 흔한 관행이지만, 그 결과 외부에서는 이 주장을 독립적
으로 검증할 방법이 없다.
어떤 조직의 어떤 워크로드가 어떤 성과를 냈는지 특정할 수 없는
“익명 성공 사례”는 그 자체로 회의적으로 다뤄야 할 신호다.
에이전트 프로토콜처럼 아직 표준화 초기 단계에 있는 기술을
평가할 때는, 이름이 명시되고 제3자가 검증 가능한 사례가
나타나기 전까지는 “도입되고 있다”는 서술과 “효과가 입증됐다”는
서술을 분리해서 받아들일 필요가 있다.

### 에이전트 상호운용성 표준 전반에 적용할 수 있는 평가 기준이 필요하다

A2A뿐 아니라 MCP를 포함한 에이전트 프로토콜 전반이 지금 비슷한
국면에 있다.
채택 지표는 빠르게 쌓이지만, 그 채택이 실제로 어떤 조직의 어떤
문제를 해결했는지 제3자가 검증 가능한 형태로 공개되는 사례는
아직 드물다.
이런 상황에서는 프로토콜을 평가할 때 “몇 개 기관이 지원을
선언했는가”보다 “이름이 명시된 몇 개 조직이 몇 개월간 실제
운영하며 어떤 지표가 바뀌었는가”를 기준으로 삼는 편이 안전하다.
이번 A2A 조사에서 후자에 해당하는 사례가 사실상 전무했다는
점은, 앞으로 등장할 다른 에이전트 프로토콜들의 초기 홍보 자료를
읽을 때도 동일한 검증 기준을 적용해야 함을 시사한다.

### 오픈소스 프로토콜의 생애주기에서 A2A가 있는 지점

기술 하이프 사이클의 논리를 그대로 빌리면, A2A는 “기대의
정점”을 이미 지나 “환멸의 계곡” 초입에 들어선 것으로 보인다.
발표 직후에는 50개 이상의 파트너 발표와 낙관적인 보도가
쏟아졌고(기대의 정점), 이번 조사와 Credal.ai의 분석이 보여주듯
지금은 “구현 부담 대비 실질 이득이 불분명하다”는 회의적 평가가
개발자 커뮤니티에서 확산되는 국면(환멸의 계곡 초입)에 있다.
동시에 v1.0 정식 출시, 서명된 Agent Card, Microsoft·AWS의
플랫폼 통합처럼 스펙과 인프라 차원의 성숙은 계속 진행 중인데,
이는 하이프 사이클에서 “계몽의 사면”으로 이어지는 다음 단계의
재료가 아직 준비되고 있다는 뜻으로 해석할 수 있다.
다만 SOAP의 사례처럼, 환멸의 계곡에서 회복하지 못하고 더 가벼운
경쟁 표준에 자리를 내주는 결말도 하이프 사이클의 흔한 변주라는
점을 함께 기억해야 한다.
지금 시점에서 A2A가 다음 단계로 올라갈지, SOAP처럼 REST 격의
경쟁자(이 경우 MCP)에게 자리를 내줄지는 이번 조사로는 결정할
수 없고, 앞으로 1~2년의 실사용 데이터를 지켜봐야 판단할 수
있는 질문으로 남는다.

### 상호운용성보다 먼저 풀어야 할 문제들

A2A가 정말 해결하려는 최종 목표, 즉 서로 다른 조직이 만든
자율 에이전트들이 자유롭게 협상하고 거래하는 “에이전트 경제”가
실현되려면 통신 프로토콜의 표준화만으로는 부족하다.
이번 보안 조사에서 확인된 Agent Session Smuggling이나 신원
위조 취약점은, 에이전트 간 통신 방식을 표준화하는 것보다
먼저 “이 에이전트가 정말 그 에이전트가 맞는가”를 신뢰할 수
있는 신원 증명 체계, 그리고 에이전트가 대신 내린 결정에 대한
책임을 누가 지는가라는 법적·계약적 프레임워크가 갖춰져야
한다는 점을 보여준다.
과금 체계 역시 마찬가지다.
서로 다른 회사의 에이전트가 실시간으로 작업을 위임하고
받는다면, 그 작업에 대한 대가를 어떻게 측정하고 정산할
것인가는 통신 프로토콜의 스키마 설계보다 훨씬 더 어려운
문제이며 A2A 스펙 자체는 이 문제를 다루지 않는다.
즉 A2A는 “에이전트들이 대화하는 방법”은 정의했지만 “그
대화를 신뢰할 수 있는 근거”와 “그 대화의 경제적 결과를
정산하는 방법”은 정의하지 않았는데, 후자 둘이 해결되지 않은
상태에서는 상호운용성 프로토콜이 아무리 정교해져도 실제
프로덕션에서 조직 경계를 넘어 쓰이기 어렵다.
이렇게 보면 이번 조사에서 “효과 입증 사례가 없다”는 결과는
A2A라는 특정 표준의 완성도 문제라기보다, 에이전트 경제 자체가
아직 신뢰·과금·책임이라는 훨씬 근본적인 전제 조건을 갖추지
못한 단계에 있다는 신호로 읽는 편이 더 정확하다.

### AI가 AI를 검증하는 재귀적 구조의 함의

이 문서 자체가 하나의 사례가 된다.
`deep-research` 워크플로는 AI 에이전트가 AI 에이전트에 대한
과장된 주장을 다른 AI 에이전트의 반박 검증으로 걸러내는
구조이며, 이번 확장 작업 역시 AI가 AI 프로토콜(A2A)의 실패
서사와 보안 취약점을 웹 검색이라는 매개를 통해 다시 AI가
정리하는 형태를 띤다.
이 재귀 구조에는 명확한 이점이 있다.
사람 혼자서는 22개, 86개에 달하는 출처와 주장을 일관된 기준으로
반박 검증하기 어렵지만, 자동화된 다표 검증은 이 작업을 확장
가능하게 만든다.
그러나 동시에 위험도 함께 커진다.
검증에 사용되는 AI 자체가 학습 데이터에 포함된 벤더 홍보성
서술을 “정설”로 흡수했을 가능성이 있고, 검증 대상 주장의
출처 대부분이 결국 다른 AI가 요약하거나 재구성한 콘텐츠
(Medium 블로그, 벤더 블로그, AI가 초안을 잡았을 가능성이 있는
2차 보도)라면, “독립적 반박”이라는 절차가 실제로는 같은 근원의
착각을 다른 표현으로 재확인하는 결과를 낳을 수도 있다.
이번 조사에서 발견한 것처럼 다수의 구체적 성공 사례 주장이
2차 보도의 과장에서 비롯됐다는 사실은, 그 2차 보도 다수가
AI 보조 도구로 작성됐을 가능성이 낮지 않은 시점에서 특히
눈여겨볼 대목이다.
결국 이런 재귀적 검증 구조를 신뢰하려면, 검증 절차의 산출물이
아니라 검증 절차가 실제로 1차 소스(공식 보도자료, GitHub 이슈,
학술 논문, 재현 가능한 벤치마크)까지 거슬러 올라갔는지를
사람이 주기적으로 감사하는 습관이 함께 필요하다.
