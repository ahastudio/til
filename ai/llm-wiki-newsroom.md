# LLM Wiki Newsroom — 다중 에이전트 지식 관리 시스템

<https://github.com/alfadur7/llm-wiki-newsroom>

## 소개

LLM Wiki Newsroom은 alfadur7이 만든 다중 에이전트 지식 관리 시스템이다.
문서를 흡수해 교차 연결된 구조화 위키로 변환하며,
“에이전트가 위키를 쓰고 사람은 편집국 체계를 설계·운영한다”는 역할 분리를 구현한다.

Claude Code를 주 에이전트로 삼고, Python 도구로 그래프 빌드·린트·쿼리를 처리한다.
MIT 라이선스이며, 별점 57개에 릴리스 v0.3.0까지 나온 활성 프로젝트다.

## 아키텍처

저장소는 세 레이어로 구성된다.

- **Layer 1** (`raw/`): 원본 소스 문서
- **Layer 2** (`wiki/`): 에이전트가 관리하는 위키 문서
- **Layer 3** (`CLAUDE.md` + `.claude/`): 에이전트 운영 규칙

에이전트 역할은 다섯 가지로 분리된다: Reporter(수집), Columnist(집필),
Copy Editor(교열), Desk(검토), Editor-in-Chief(최종 승인).
글쓴이와 검토자를 분리하는 것이 품질 편향 방지의 핵심이다.

## 슬래시 커맨드

Claude Code 기반 슬래시 커맨드로 시스템을 운영한다.

| 커맨드          | 기능                               |
| --------------- | ---------------------------------- |
| `/wiki-ingest`  | 문서 흡수 및 연관 페이지 갱신      |
| `/wiki-query`   | 그래프 순회 기반 질문 응답         |
| `/wiki-graph`   | 지식 그래프 시각화 생성            |
| `/wiki-lint`    | 위키 상태 점검 및 자동 수정        |
| `/wiki-discover`| 예상치 못한 연결 탐색              |
| `/wiki-trail`   | 저장된 읽기 경로 구성              |

## 주요 기능

문서 하나를 추가하면 연관된 기존 페이지 10~15개가 자동으로 갱신된다.
지식 그래프를 HTML5 물리 시뮬레이션으로 인터랙티브하게 시각화한다.
3단계 모순 추적 시스템이 출처 간 상충 주장을 표시한다.
Leiden 알고리즘으로 토픽 클러스터링을 자동 수행한다.

검색은 외부 API 없이 로컬 임베딩(QMD)으로 처리한다.
Obsidian과 Git에 호환되는 순수 마크다운으로 출력한다.
메타 루프가 반복 실패를 수집해 작성 규칙 수정안을 제안하고,
블라인드 A/B 테스트로 검증한 뒤 반영한다.

## 분석

### 편집국 모델을 그대로 복제한 역할 분리가 핵심이다

전통적인 신문사 편집국 구조 — 기자, 칼럼니스트, 교열부, 데스크, 편집장 — 를
에이전트 시스템에 그대로 이식했다.
이 구조가 존재하는 이유는 단순하다: 글쓴이가 자신의 글을 편집하면 판단이 흐려진다.
같은 문제가 에이전트 시스템에도 그대로 성립한다.

“에이전트 A가 쓰고 에이전트 B가 검토한다”는 분리는
기술적 선택이 아니라 인식론적 선택이다.
어떤 지식이 위키에 들어갈 자격이 있는가를 결정하는 관문이
단일 에이전트의 자기 평가가 아닌 외부 검토를 거친다는 뜻이다.

### 메타 루프가 시스템 자가 개선의 메커니즘이다

반복 실패를 수집하고, 규칙 수정안을 만들고, A/B 테스트로 검증하는 메타 루프는
단순한 자동화가 아니라 학습 체계다.
“이 규칙이 실제로 품질을 높이는가”를 데이터로 검증하는 구조가
인간 편집국의 레트로스펙티브와 동형이다.

이 루프가 안정적으로 작동한다면, 시스템은 운영될수록 나빠지는 대신 나아진다.
지식 관리 시스템이 갖춰야 할 복리 효과를 기술적으로 구현한 시도다.

### 캐스케이딩 갱신이 지식 그래프의 일관성을 유지한다

문서 하나 추가에 10~15개 연관 페이지가 갱신되는 캐스케이딩 업데이트는
지식 베이스의 고질적 문제 — 새 정보가 들어와도 이전 문서가 업데이트되지 않는 정체 — 를
구조적으로 공략한다.

이것은 데이터베이스의 외래 키 제약과 같은 역할을 한다.
한 노드가 바뀌면 연결된 노드가 그 변화를 반영해야 한다는 강제가
명시적 설계로 구현된다.

## 비평

### 캐스케이딩 갱신의 비용이 검증되지 않았다

문서 하나에 15개 페이지가 갱신된다면, 100개 문서를 추가하면 최대 1,500번의
LLM 호출이 발생한다.
이 비용이 실제로 어느 정도인지, 갱신 품질이 새로 작성한 것과 비교해 어떤지 — 이
현실적 질문에 저장소의 README는 답하지 않는다.

소규모 코퍼스(현재 15개 노드 예시)에서 검증된 패턴이
100개, 1,000개 노드 규모에서도 작동하는지는 전혀 다른 문제다.
“자동으로 갱신된다”는 기능 설명이 “감당할 수 있는 비용으로 갱신된다”는 것을 보증하지 않는다.

### A/B 테스트 기반 규칙 검증이 과도하게 낙관적이다

메타 루프에서 블라인드 A/B 테스트로 규칙 수정안을 검증한다는 설계는
원칙적으로는 타당하다.
그러나 “어떤 규칙이 더 나은가”를 판단하는 A/B 테스트의 평가 기준이 무엇인지,
그 기준 자체를 누가 정하는지 명확하지 않다.

평가 기준이 또 다른 LLM이라면 순환 논증이다.
사람 운영자가 최종 승인한다고 해도, 운영자가 검토해야 할 비교 쌍의 수가
코퍼스 크기에 비례해 늘어나면 병목이 된다.

### 로컬 임베딩 검색은 규모에서 한계가 있다

QMD 기반 로컬 임베딩 검색은 외부 API 없이 작동한다는 장점이 있다.
그러나 임베딩 품질은 모델 크기에 비례하며,
로컬에서 실행할 수 있는 임베딩 모델은 API 기반 모델보다 일반적으로 성능이 낮다.

수천 개 이상의 문서를 다루는 기업 수준의 지식 베이스에서
검색 품질이 외부 API 수준과 얼마나 가까운지 — 이것이 실제 사용 가능 여부를 가른다.
“외부 API 없이”는 자급 자족을 강조하지만, 동시에 품질의 상한을 낮추는 선택이기도 하다.

## 인사이트

### 에이전트 역할 분리는 조직 설계를 소프트웨어로 번역하는 작업이다

LLM Wiki Newsroom의 핵심 통찰은 기술적인 것이 아니라 조직적인 것이다.
편집국이 수백 년에 걸쳐 개발한 글쓰기 품질 관리 구조 — 역할 분리, 편집 관문, 사실 확인 — 를
에이전트 시스템으로 번역했다는 점이 이 프로젝트를 단순한 “LLM + 위키” 도구와 구별한다.

이 번역의 함의는 넓다.
에이전트 시스템 설계자가 참고해야 할 선례가 소프트웨어 엔지니어링에만 있는 것이 아니라는 뜻이다.
법률 사무소의 파트너-어소시에이트 구조, 과학 저널의 이중 맹검 검토, 군사 조직의 지휘 체계 — 이 모든 것이
에이전트 역할 설계의 선례 라이브러리가 된다.

### 마크다운 + Git이 에이전트 지식 시스템의 기반이 되는 이유가 있다

Obsidian 호환, Git 호환 순수 마크다운을 고집하는 것은 기술적 편의가 아니다.
에이전트가 생성한 지식이 인간이 읽고 수정할 수 있어야 한다는 설계 철학이다.

에이전트 전용 포맷이나 독점 데이터 구조는 에이전트가 정상 작동할 때는 문제가 없다.
에이전트가 오작동하거나 모델이 바뀌거나 시스템이 바뀔 때, 사람이 개입해서
지식 베이스를 점검하고 수정할 수 있어야 한다.
마크다운 + Git은 이 개입 가능성을 보장하는 선택이다.

### 이 프로젝트가 실험인지 제품인지의 경계가 불분명하다

57개 별, 8번 릴리스, 15노드 예시 코퍼스 — 이 숫자들이 말하는 것은
활발한 실험이지 검증된 제품이 아니라는 것이다.

편집국 모델, 메타 루프, 캐스케이딩 갱신 같은 설계 아이디어는 가치 있지만,
실제 규모의 코퍼스에서 안정적으로 작동하는지는 아직 증명되지 않았다.
이 프로젝트를 “영감의 소스”로 보는 것과 “프로덕션 의존성”으로 채택하는 것 사이에는
큰 차이가 있다.
지식 관리 시스템의 신뢰성은 소규모 데모가 아니라 장기 운영에서 드러난다.
