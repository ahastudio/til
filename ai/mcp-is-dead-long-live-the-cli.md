# MCP is dead. Long live the CLI

- 원문:
  <https://ejholmes.github.io/2026/02/28/mcp-is-dead-long-live-the-cli.html>

## 요약

이 글의 핵심 메시지는 명확하다.
"MCP를 부정"하는 것이 아니라, 실제 에이전트 제품 개발에서는
CLI가 더 강한 기본 인터페이스라는 주장이다.

MCP는 도구 연결 표준으로 유용하지만,
복잡한 상호작용(스트리밍, 승인, 장기 세션, 복구, 분기)까지
포괄하려고 하면 추상화가 빠르게 무거워진다.

반면 CLI는 운영체제의 프로세스 모델과 stdin/stdout,
exit code, 파이프라인 같은 검증된 구성 요소를 그대로 활용한다.
결과적으로 구현 난이도, 디버깅, 배포, 호환성에서 이점을 얻는다.

즉, 글의 메시지는 "표준 vs 비표준"의 이분법이 아니라,
"문제에 맞는 인터페이스 선택"에 가깝다.

## 분석

### 1) MCP의 문제라기보다, 책임 과적재의 문제

MCP 자체는 툴 디스커버리와 호출 계약에 강점이 있다.
문제는 팀이 MCP 하나로 에이전트 런타임의 모든 계층을
해결하려고 할 때 발생한다.

- 도구 호출 계약
- 세션/상태 지속성
- 사용자 승인 루프
- 스트리밍 UI 동기화
- 에러 복구와 재시도

이 다섯 책임은 결이 다르다.
하나의 프로토콜에 모두 얹으면 설계가 비대해지고,
경계가 흐려져 구현체마다 해석 차이가 커진다.

### 2) CLI는 "가난한 인터페이스"가 아니라 "강한 최소 코어"

CLI는 표현력 부족처럼 보이지만,
에이전트 시스템에서는 오히려 장점이 된다.

- 어디서나 실행 가능(로컬, CI, 컨테이너, 원격 SSH)
- 로깅/관찰 가능성 우수(텍스트 스트림 기반)
- 조합 용이(`|`, `xargs`, `jq`, 리다이렉션)
- 실패 모델 명확(exit code, 시그널)

즉 CLI는 기능이 적어서 약한 게 아니라,
운영 관점의 불확실성을 줄여서 강하다.

### 3) 제품 팀 관점: "통합 속도"가 아키텍처를 이긴다

에이전트 제품의 병목은 모델 성능만이 아니다.
현실의 병목은 통합과 운영이다.

- 기존 툴체인에 얼마나 빨리 붙는가
- 장애 시 누가 어디를 볼 수 있는가
- 팀 간 계약이 얼마나 단순한가

CLI 우선 전략은 이 병목을 줄인다.
특히 초기/성장 단계 제품에서 의사결정 속도를 높인다.

## 인사이트

### 1) "MCP 종료"가 아니라 "MCP의 계층 재배치"가 본질이다

이 논쟁을 승패 프레임으로 보면 오판한다.
더 정확한 해석은 계층 분리다.

- 상위 계층: 에이전트 경험/세션/승인/워크플로우
- 하위 계층: 도구 호출 계약(MCP), 프로세스 실행(CLI)

즉 MCP는 사라지는 것이 아니라,
"도구 표준"이라는 제자리를 찾아가는 과정이다.

### 2) 2026년 에이전트 스택의 실전 패턴은
"CLI 퍼스트 + 선택적 MCP"가 될 가능성이 높다

운영 가능한 시스템은 대개 복잡도를 늦게 도입한다.
처음엔 CLI로 end-to-end를 고정하고,
필요한 지점에만 MCP를 얹는 방식이 생존 확률이 높다.

이 패턴의 장점:

- 초기 전달 속도 빠름
- 디버깅 단순
- 장애 격리 쉬움
- 팀 온보딩 비용 낮음

결국 "아름다운 표준"보다
"고장 나도 복구 가능한 파이프라인"이 이긴다.

### 3) 프로토콜 경쟁의 승부처는 기능이 아니라
"운영 인지부하(operational cognitive load)"다

엔지니어 조직이 감당하는 진짜 비용은
스펙 문서 분량이 아니라 운영 시 인지부하다.

- 이 문제가 누구 소유인지 즉시 아는가
- 로그 한 번으로 경로를 재구성할 수 있는가
- 재현 시나리오를 5분 안에 만들 수 있는가

CLI는 이 질문에 강하게 답한다.
"단순성"이 미학이 아니라, MTTR을 줄이는 실무 무기다.

### 4) Human-in-the-loop의 핵심은 UI가 아니라 경계 정의다

승인 UX를 화려하게 만드는 것보다 중요한 건
"어디서 자동화가 멈추고 사람이 결정하는지"를
명확히 선언하는 것이다.

CLI 중심 설계는 이 경계를 명령 단위로 드러낸다.
에이전트 동작을 작은 불변 계약으로 쪼개고,
위험 작업 앞에서 중단점을 강제하기 쉽다.

이는 보안/감사/컴플라이언스 요구가 있는 조직에서
특히 강력한 장점이 된다.

### 5) 장기적으로는 "CLI가 API를 이긴다"가 아니라
"CLI가 API 설계를 정련한다"가 맞다

초기에는 CLI가 빠르다.
하지만 규모가 커지면 API/프로토콜이 필요해진다.

중요한 점은 순서다.
CLI로 먼저 실전 워크플로우를 검증하면,
나중에 만드는 API는 "가정"이 아니라
"실측된 사용 패턴"을 반영한다.

즉 CLI 퍼스트는 종착지가 아니라,
더 좋은 프로토콜로 가는 압축 학습 단계다.

## 실무 적용 체크리스트

- 에이전트 핵심 루프를 먼저 CLI로 고정했는가?
- 실패 모델(exit code, timeout, retry)을 계약으로 문서화했는가?
- 승인 지점(human gate)을 명령 경계로 정의했는가?
- 로그/트레이스를 텍스트 기반으로 재현 가능하게 남기는가?
- MCP 도입 목적을 "표준 준수"가 아니라
  "명확한 통합 이득"으로 증명했는가?

위 다섯 질문에 "예"가 많을수록,
도구 유행과 무관하게 오래가는 에이전트 시스템이 된다.
