# 멀티 에이전트 오케스트레이션 특유의 실패

[← 에이전트형 코딩의 실패/함정 패턴](index.md)

지금까지 다룬 문제(검증 부채, 감독의 역설, 스펙/TDD)는 단일 에이전트에도
적용된다. 이 주제는 자율성을 여러 에이전트로 확장할 때만 새로 발생하는
구조적 실패를 다룬다 — 여러 노트가 결론을 공유한다: "복잡할수록 좋아
보이지만, 실제로는 명시적 단순함이 이긴다."

## MAST 연구가 분류한 세 가지 병목

[multi-agent-orchestration-problems.md](../../agentic-coding/multi-agent-orchestration-problems.md)의
저자는 5,000달러를 직접 투자해 멀티 에이전트 시스템(Paperclip,
Gastown)을 운영한 뒤, UC Berkeley MAST 연구를 인용해 실패를 세 가지로
분류한다.

- **맥락 붕괴(41.8%)**: 에이전트가 전체 목표를 잃고 "왜 이 일을
  하는가"를 이해하지 못하게 된다. 작업이 여러 에이전트를 거치며 원래
  목적이 희석되는, 인간 조직의 "전화 게임"과 같은 구조다.
- **유령 위임(36.9%)**: 에이전트 A가 B에게 작업을 넘겼는데 B가
  실패하면, A는 이미 다른 작업으로 넘어가 아무도 그 실패를 감지하지
  못한다. 분산 시스템의 "부분 실패" 처리 문제와 동일하다.
- **검증 오류(21.3%)**: 에이전트가 부정확한 결과를 스스로 검증해
  완료 처리한다. [agent-responsibly.md](../../agentic-coding/agent-responsibly.md)가
  경고하는 "기계가 작성한 코드를 기계가 검증하는 순환 — 신뢰의 앵커가
  사라지는" 문제가 여기서 정확히 재현된다.

토큰 소비는 단일 에이전트 대비 5~10배 늘지만 생산성은 그에 비례하지
않는다는 것이 저자의 핵심 관찰이다. 이는 단순한 비용 문제가 아니다.
토큰 소비 증가는 지연 시간 증가를 의미하고(에이전트가 다른 에이전트의
결과를 기다리며 블로킹), 오류 디버깅 비용은 여러 에이전트의 이력을
교차 분석해야 하므로 기하급수적으로 늘어난다.

## 블랙보드 패턴의 재발견

`multi-agent-orchestration-problems.md`가 제안하는 해법 — 에이전트 간
직접 통신보다 이슈 트래커나 Git 같은 공유 환경을 쓰는 것 — 은 사실
1970~80년대 AI 연구에서 나온 블랙보드 아키텍처(Blackboard
Architecture)의 재발견이다. 여러 전문가 시스템이 공유 데이터 구조에
결과를 쓰고 읽으며 협력하는 방식으로, 직접 메시지 패싱보다 공유 상태
기반 협력이 더 견고하다는 원칙은 이미 40년 전에 확립되어 있었다. 다만
결정적 차이가 있다 — 기존 분산 시스템은 결정론적(같은 입력에 같은
출력)이지만 에이전트는 그렇지 않다. 이 비결정성이 기존 분산 시스템
패턴을 그대로 적용하기 어렵게 만드는 핵심 요인이다.

## Level 4의 솔직한 경고: 동적 팀보다 명시적 워크플로

[agentic-software-levels.md](../../agentic-coding/agentic-software-levels.md)는
에이전트형 소프트웨어를 5단계(도구 → 저장소/지식 → 메모리/학습 →
멀티 에이전트 팀 → 프로덕션 시스템)로 정리하면서, Level 4(멀티
에이전트 팀)에 이례적으로 솔직한 경고를 담는다 — "팀 리더도 LLM이라
위임 결정이 들쭉날쭉하다. 신뢰성이 중요한 프로덕션 시스템에서는 동적
팀 대신 명시적 워크플로우를 선호하라." 대부분의 에이전트 프레임워크
소개 글이 멀티 에이전트를 이상적 목표로 그리는 것과 대비된다.

이 경고의 핵심 메시지는 소프트웨어 개발의 오래된 원칙(YAGNI)과
겹친다 — 대부분의 팀은 멀티 에이전트 오케스트레이션이 데모에서
인상적으로 보이기 때문에 Level 1(단일 에이전트 + 좋은 지시문)을
건너뛰고 Level 4로 직행하며, 단순한 방법으로 피할 수 있었던 조율
실패를 몇 달씩 디버깅하게 된다. "단순한 방법이 명백히 실패했을
때만 다음 레벨로 이동하라"는 것이 이 노트의 결론이다.

## 자율주행과의 평행 — 신뢰는 단계적으로만 쌓인다

`multi-agent-orchestration-problems.md`가 제시하는 업무 위임 판별
기준(오류 비용, 검증 용이성, 암묵지 의존도, 컨텍스트 범위, 피드백 루프
길이)은 자율주행의 자동화 레벨(L0~L5) 접근법과 구조적으로 같다. 낮은
위험의 검증 가능한 작업(코드 포매팅, 단위 테스트 생성)은 이미
자동화에 가깝지만, 아키텍처 설계나 요구사항 분석 같은 고위험·저검증
작업은 아직 초기 단계에 머문다는 것이다. 이 단계적 신뢰 구축 원칙은
[어디까지 자율을 주고 어디서 개입해야 하는가](where-to-draw-the-line.md)에서
다루는 판단 기준의 한 축이 된다.

## 정리: 복잡성 추가는 검증 부담을 줄이지 않는다

이 주제의 세 노트가 공유하는 결론은 명확하다. 멀티 에이전트로
확장하면 처리량은 늘어날 수 있지만, [자율성이 검증을 잠식하는
구조](autonomy-erodes-verification.md)에서 다룬 문제(검증이 생성
속도를 못 따라간다)가 사라지지 않고 오히려 새로운 실패 모드(맥락
붕괴, 유령 위임, 상호 검증의 순환 참조)로 증폭된다. 에이전트 수를
늘리는 것은 검증 병목에 대한 해법이 아니라, 그 병목을 여러 에이전트
사이의 조율 문제로 형태만 바꾸는 것에 가깝다.
