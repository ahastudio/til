# AudioMass — 백엔드 없는 브라우저 기반 오픈소스 오디오 편집기

<https://github.com/pkalogiros/AudioMass>

<https://audiomass.co/>

## 소개

AudioMass는 자바스크립트로 작성되어 브라우저 안에서 완전히 동작하는 웹
기반 오디오 편집기다.
백엔드 서버 없이 클라이언트에서만 돌아가며, 라이선스는 MIT다.
파일 크기도 작아 자바스크립트가 약 98KB, CSS가 약 10KB에 불과하다.

## 주요 기능

DAW(디지털 오디오 워크스테이션) 수준의 효과를 제공한다 — 트리밍, 컷,
복사·붙여넣기, 페이드, 컴프레서, 리버브, 피치 시프트.
그래프 기반의 피치·속도 편집으로 세밀한 조작이 가능하고, 자동 비트 감지와
메트로놈, 비트에 맞춘 스냅 편집을 지원한다.

크로스페이드 조정과 제로 크로싱(zero-crossing) 스냅으로 이음매 없는 루핑이
가능하며, 멀티트랙 모드에서 클립을 드래그앤드롭하면 자동으로 크로스페이드된다.
세션은 `.amss` 파일로 저장되는데, 원본 오디오와 JSON 레이아웃을 LZMA로
압축한 형식이다.
WAV·MP3로의 오프라인 렌더링과 실행 취소(undo) 스택도 지원한다.

## 분석

### Web Audio API의 성숙을 보여주는 사례

AudioMass는 브라우저가 더 이상 단순 문서 뷰어가 아니라 본격적 오디오
편집 플랫폼이 되었음을 보여준다.
컴프레서·리버브·피치 시프트 같은 DSP 처리를 백엔드 없이 클라이언트에서
수행한다는 것은, Web Audio API와 자바스크립트 성능이 데스크톱 DAW에 근접한
수준에 도달했다는 증거다.
98KB라는 작은 번들 크기로 이를 달성한 것은 특히 인상적이다.

### 코드 스타일은 구형이지만 견고하다

[GeekNews 댓글](https://news.hada.io/topic?id=30005)에서 관찰됐듯, 코드는
안전한 클로저와 순차적 `var` 선언 같은 구형 자바스크립트 패턴을 따른다.
이는 의존성 없이 오래 동작하도록 만든 의도적 선택으로 볼 수 있다 —
프레임워크에 묶이지 않은 바닐라 코드는 빌드 도구의 변천에 영향받지 않고
오래 산다.

## 비평

### 강점: 백엔드 없음이 곧 영속성이다

AudioMass의 가장 큰 미덕은 서버가 없다는 점이다.
백엔드가 없으면 운영 비용도, 데이터 유출 위험도, 서비스 종료 위험도 없다.
사용자의 오디오가 서버로 업로드되지 않으니 프라이버시가 보장되고, 정적
호스팅만으로 무한히 동작한다.
[GeekNews 댓글](https://news.hada.io/topic?id=30005)에서 Audacity보다 낫고
"투명하면서도 단단한 느낌"이라는 호평이 나온 것은 이 단순함 덕분이다.

### 약점: 협업과 대용량의 한계

그러나 백엔드 부재는 곧 한계이기도 하다.
댓글에서 클라우드 트랙 협업, DAW 프로젝트 버전 관리 같은 기능이 요청된
것은, 서버 없는 구조가 협업과 영속 저장에서 본질적 제약을 갖기 때문이다.
또한 브라우저 메모리에 오디오를 올려 처리하므로, 장시간 녹음이나 대용량
멀티트랙에서는 데스크톱 DAW의 스트리밍 처리를 따라가기 어렵다.

## 인사이트

### "백엔드 없음"은 제약이 아니라 아키텍처 철학이다

AudioMass가 백엔드를 두지 않은 것은 기능 부족이 아니라 명확한 설계 철학이다 —
복잡한 처리를 클라이언트로 밀어내면 서버의 모든 부담(비용·보안·운영·종료
위험)이 사라진다.
이는 로컬 퍼스트(local-first) 소프트웨어 운동과 같은 정신이다.
사용자의 데이터가 사용자의 기기를 떠나지 않으면, 프라이버시와 영속성이
구조적으로 보장된다.

함의는 "모든 것을 클라우드로"라는 시대정신에 대한 반례다.
SaaS 모델이 모든 기능을 서버로 끌어가는 동안, AudioMass는 정반대 방향 —
처리를 엣지(브라우저)로 밀어내는 길 — 이 종종 더 단순하고 견고함을 보여준다.
브라우저의 연산 능력이 커질수록, 서버가 꼭 필요한 기능(협업, 동기화)과
클라이언트에서 충분한 기능(편집, 처리)의 경계가 후자 쪽으로 이동한다.

### 작은 번들은 그 자체로 기능이다

98KB라는 번들 크기는 부차적 디테일이 아니라 핵심 가치다.
가벼운 코드는 빠르게 로드되고, 의존성이 적어 빌드 도구의 변천에 견디며,
전체를 한 사람이 이해할 수 있다.
오늘날 수 MB짜리 자바스크립트 번들이 흔한 시대에, 본격적 오디오 편집기를
98KB로 만든 것은 절제의 공학(engineering of restraint)이다.

이는 [SQLite의 50년 코드 철학](../sqlite/sqlite-no-ai-code.md)과 통하는
지점이 있다 — 작고 의존성 없는 코드가 오래 산다.
프레임워크와 의존성을 쌓아 올리는 것이 기본값이 된 생태계에서, "바닐라
자바스크립트로 충분히 작게"라는 선택은 장기 유지보수성에 대한 투자다.
번들 크기를 줄이는 것은 성능 최적화일 뿐 아니라, 소프트웨어의 수명을
늘리는 일이다.

### 오픈소스 데스크톱 대체재의 새 표준

AudioMass가 Audacity보다 낫다는 평가는, 웹 기술이 전통적 데스크톱 오픈소스
소프트웨어를 대체하기 시작했음을 시사한다.
설치가 필요 없고, OS를 가리지 않으며, 링크 하나로 즉시 쓸 수 있는 웹앱은
배포 마찰(distribution friction)이 0에 가깝다.
Figma가 디자인 도구에서, 그리고 수많은 웹 IDE가 개발 도구에서 그랬듯,
오디오 편집도 브라우저로 옮겨가는 흐름의 한 사례다.

이 전환의 핵심 동력은 배포 마찰의 제거다.
데스크톱 앱은 다운로드·설치·업데이트·OS 호환성이라는 마찰을 동반하지만,
웹앱은 URL 하나로 모든 것을 우회한다.
기능이 동등해지는 순간, 마찰이 적은 쪽이 이긴다 — AudioMass는 그 전환점이
오디오 편집 영역에도 도달했음을 보여주는 신호다.
