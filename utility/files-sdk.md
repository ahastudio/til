# Files SDK — 모든 blob 스토리지를 위한 단일 API

<https://files-sdk.dev/>

<https://news.hada.io/topic?id=30064>

## 소개

Files SDK는 AWS S3, Cloudflare R2, Google Cloud Storage, Azure Blob Storage를
비롯한 40개 이상의 blob 스토리지 백엔드를 단일 인터페이스로 다루는
JavaScript/TypeScript 라이브러리다.
“한 번 작성하고 어디서나 저장(Write Once, Store Anywhere)”이라는 철학 아래,
스토리지 제공자를 교체해도 애플리케이션 코드는 설정 변경만으로 충분하다.

## 지원 백엔드

40개 이상의 어댑터를 제공한다.

- **클라우드 메이저**: AWS S3, Cloudflare R2, Google Cloud Storage, Azure Blob
- **특화 서비스**: Vercel Blob, Netlify Blobs, Supabase, UploadThing
- **자체 호스팅**: MinIO, DigitalOcean Spaces, Dropbox, Google Drive, OneDrive, Box

## API 설계

모든 어댑터에서 동일한 메서드로 동작한다.

```typescript
upload()   download()   head()   exists()
copy()     move()       list()   delete()
```

배열을 넘기면 배치 처리를 제한된 동시성으로 자동 수행하며,
`list()`는 대용량 파일 컬렉션을 메모리 효율적으로 순회하는 async iterable을
반환한다.

## 주요 기능

### 멀티파트 업로드

대용량 파일이나 스트림을 자동으로 청크로 분할해 병렬 업로드하며,
청크 크기와 동시성 수준을 설정 가능하다.

### 바이트 수준 진행률 콜백

단건·배치 작업 모두에서 전송 계층 세부 사항 없이 바이트 단위 진행 지표를 전달한다.

### HTTP 206 범위 다운로드

범위 요청을 HTTP 206 응답으로 직접 매핑해 전체 파일 없이 비디오 탐색,
다운로드 재개, 헤더 전용 읽기를 지원한다.

### 라이프사이클 훅

`onAction`, `onRetry`, `onError` 세 가지 훅이 모든 작업·어댑터에서 일관되게
발생해 중앙집중식 메트릭, 로깅, 에러 보고에 활용된다.

### 백엔드 동기화

`sync()` 함수는 변경된 파일만 업로드하고 동일 콘텐츠를 건너뛰며 선택적으로
삭제된 항목을 정리한다.
Dry-run 모드로 실행 전 변경 내용을 미리 확인할 수 있다.

## AI 통합

Vercel AI SDK, OpenAI Agents, Claude용으로 즉시 사용 가능한 파일 툴을 제공한다.
`listFiles()`, `downloadFile()`, `uploadFile()`, `deleteFile()` 래퍼는
승인 게이트와 읽기 전용 모드를 내장해 에이전트의 안전한 스토리지 접근을 지원한다.

## CLI

모든 SDK 메서드가 셸 명령에 그대로 매핑된다.

```text
files-sdk upload ./image.png --provider s3 --bucket my-bucket
files-sdk download s3://my-bucket/image.png | cat
files-sdk list --provider r2
```

`--provider` 플래그로 제공자를 전환하고, 기본 JSON 출력으로 스크립팅과 CI/CD
파이프라인에 통합 가능하다.
Node.js 서버 없이 일회성 작업에 바로 쓸 수 있다.

## 분석

### 어댑터 패턴과 의존성 최소화

각 제공자는 별도 어댑터 모듈(예: `files-sdk/s3`, `files-sdk/r2`)로 구현되며,
네이티브 SDK는 선택적 피어 의존성이다.
사용하는 제공자의 SDK만 설치하면 된다.
이 설계는 번들 크기를 제공자 수에 비례하지 않게 유지하는 핵심 결정이다.

### “결정 연기” 전략

Files SDK가 제공하는 가장 큰 가치는 스토리지 아키텍처 결정을 나중으로
미룰 수 있다는 점이다.
초기에는 무료 Cloudflare R2로 시작하고, 규모가 커지면 S3나 GCS로 전환해도
애플리케이션 코드를 건드리지 않아도 된다.
이는 특히 스타트업에서 제품-시장 적합성을 찾는 동안 인프라 의존성을 낮게
유지하는 실용적 접근이다.

### MCP 통합과 에이전트 스토리지

AI 에이전트 툴 지원을 명시적으로 포함한 것은 LLM 시대의 스토리지 추상화
수요를 반영한다.
에이전트가 파일을 읽고 쓰는 작업은 앞으로 더 빈번해질 텐데,
어댑터 교체만으로 클라우드 백엔드를 바꿀 수 있다는 점은 에이전트 인프라
구성에서 상당한 유연성을 제공한다.

## 비평

### 강점 — 실용적 추상화 수준

Files SDK의 API는 최소한의 공통 분모를 선택했다.
`upload`, `download`, `list`, `delete` 같은 기본 연산은 40개 이상의 제공자가
모두 지원하므로, 이 추상이 실제로 작동한다.
지나치게 많은 공통 분모를 찾으려다 추상 자체가 무력해지는 함정을 피한 것이다.

### 약점 — 제공자별 고급 기능의 노출

S3의 객체 태그, R2의 사용자 지정 도메인, GCS의 서명 URL처럼 제공자별 고유
기능은 통합 API로 노출되지 않는다.
이 기능이 필요한 시점에는 결국 제공자 SDK를 직접 사용해야 하고, 그러면
추상화의 이점이 퇴색된다.
Files SDK가 “90%의 사용 사례”를 위한 도구인지, “모든 사용 사례”를 위한 도구인지를
명확히 하지 않으면 기대치 불일치가 발생한다.

### 빈 자리 — 에러 처리 일관성

40개 어댑터 각각이 서로 다른 에러 코드와 상황을 낸다.
통합 API에서 에러를 어떻게 정규화하는지, 예를 들어 “파일 없음”이 S3의
`NoSuchKey`와 GCS의 404를 어떻게 동일한 에러 타입으로 노출하는지가 문서에서
명확하지 않다.
이 부분이 실제 마이그레이션에서 가장 많은 버그를 낳는 지점이 될 수 있다.

## 인사이트

### 스토리지 추상화가 필요한 진짜 이유는 공급자 교체가 아니다

Files SDK의 마케팅은 “공급자를 쉽게 교체”에 초점을 맞추지만, 실제 가치는
다른 곳에 있다.
대부분의 프로젝트는 S3에서 R2로 이전하는 경우보다 개발 환경(로컬 MinIO)과
프로덕션 환경(S3)을 동시에 지원해야 하는 경우가 훨씬 많다.
테스트에서는 인메모리 어댑터, CI에서는 로컬 스토리지, 프로덕션에서는 S3를
설정 한 줄로 전환하는 능력이 진짜 일상적 이득이다.

이것은 인터페이스 추상화의 고전적 가치인데, 많은 팀이 스토리지 레이어에는
이 추상을 적용하지 않는다.
데이터베이스 ORM이 DB를 교체하기 위해서가 아니라 테스트를 쉽게 하기 위해
쓰이듯이, 스토리지 SDK도 같은 역할을 한다.

### 에이전트 시대의 파일 시스템은 blob 스토리지다

전통적인 파일 시스템(POSIX)은 에이전트 환경에서 신뢰할 수 없다.
여러 에이전트 인스턴스가 동일 파일에 쓰거나, 컨테이너가 재시작되면 로컬
파일이 사라지거나, 분산 환경에서 파일 경로를 공유하는 것 자체가 불가능하다.
Files SDK가 에이전트 툴을 명시적으로 지원하는 것은 이 필요를 정확히 읽은
것이다.

앞으로 에이전트 프레임워크에서 “파일에 저장”은 자연스럽게 blob 스토리지를
의미하게 될 것이다.
에이전트가 작업 결과를 유지하고, 다른 에이전트나 사람이 그 결과에 접근하려면
결국 위치 독립적인 스토리지가 필요하다.
Files SDK 같은 추상 레이어가 에이전트 인프라의 표준 컴포넌트로 자리잡을
가능성이 있다.

### “Write Once, Store Anywhere”는 WORA가 남긴 역사적 교훈을 반복한다

1990년대 Java의 “Write Once, Run Anywhere”는 플랫폼 독립성의 약속이었다.
실제로는 플랫폼마다 미묘하게 다른 동작을 낳았지만, 그 추상화 자체는 개발자
생산성을 높이는 데 실질적으로 기여했다.
Files SDK의 “Write Once, Store Anywhere”도 같은 패턴이다.
40개 어댑터가 완벽하게 동일하게 동작한다는 보장은 없지만, 80~90%의 일상적
작업에서 마찰을 없애는 것만으로도 충분한 가치가 있다.

진정한 이식성보다 충분한 이식성이 실용적이라는 교훈은, 추상화 레이어를
평가하는 올바른 기준이 “완벽한 호환성”이 아니라 “마찰 감소의 실질적 가치”임을
시사한다.
