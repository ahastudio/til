# 개발자 대부분이 모르는 ARIA 속성들

원문: <https://dev.to/web_dev-usman/99-of-developers-dont-know-these-aria-attributes-exist-19j0>

## 요약

ARIA(Accessible Rich Internet Applications)는 보조 기술(스크린 리더 등)이 웹 UI의
의미와 상태를 이해할 수 있도록 돕는 HTML 속성 집합이다.
전 세계 10억 명 이상의 장애인이 접근성에 의존하며, ARIA는 시맨틱 HTML만으로 표현하기
어려운 동적 UI 패턴을 보조 기술에 전달하는 핵심 수단이다.
저자는 “시맨틱 HTML이 항상 우선”이라는 원칙을 글 전체의 기반으로 강조하며, ARIA는
네이티브 HTML 요소만으로 부족한 경우의 보완 도구임을 명확히 한다.

ARIA는 크게 네 범주로 나뉜다.
첫째, **역할(Roles)**은 요소의 기능적 유형을 정의한다.
랜드마크 역할(`banner`, `navigation`, `main`, `search`)은 페이지 구조를,
위젯 역할(`button`, `checkbox`, `tab`, `slider`)은 인터랙티브 컨트롤을,
라이브 영역 역할(`alert`, `log`, `status`)은 동적으로 업데이트되는 콘텐츠를 나타낸다.
둘째, **위젯 속성(Widget Attributes)**은 컨트롤의 상태와 값을 기술한다.
`aria-checked`, `aria-expanded`, `aria-pressed` 같은 상태 속성,
`aria-valuemin`/`aria-valuemax`/`aria-valuenow` 같은 값 속성,
`aria-required`, `aria-readonly`, `aria-invalid` 같은 입력 속성이 포함된다.

셋째, **라이브 영역 속성(Live Region Attributes)**은 동적 콘텐츠 변경을 스크린 리더에
알리는 방식을 제어한다.
`aria-live=“polite”`는 현재 읽는 내용이 끝난 후에 알리고,
`aria-live=“assertive”`는 즉시 인터럽트해서 알린다.
`aria-atomic`, `aria-relevant`, `aria-busy`로 업데이트 범위와 종류를 세밀하게 제어할 수 있다.
넷째, **관계 속성(Relationship Attributes)**은 요소 간의 시맨틱 연결을 만든다.
`aria-labelledby`와 `aria-describedby`는 레이블 및 설명 연결에,
`aria-controls`는 한 요소가 제어하는 대상을,
`aria-owns`는 DOM 구조와 다른 논리적 부모-자식 관계를 표현한다.

실전 예시로는 토글 버튼(`aria-pressed`), 모달 대화상자(`role=“dialog”`, `aria-modal`),
폼 유효성 검사(`aria-invalid`, `aria-describedby`), 확장 가능 섹션(`aria-expanded`),
라이브 검색 결과(`aria-live`, `role=“status”`) 구현이 제시된다.
테스트 방법으로는 Chrome DevTools 접근성 인스펙터, NVDA/JAWS/VoiceOver 같은 실제
스크린 리더, axe DevTools 같은 자동화 도구, 키보드 전용 탐색, 실제 사용자 테스트를 권장한다.

## 분석

### 네 가지 ARIA 범주의 상호 보완 구조

ARIA의 네 범주는 독립적이지 않고 층위를 이룬다.
역할(Roles)이 요소의 “정체”를 선언하고, 위젯 속성이 “현재 상태”를 전달하며,
라이브 영역 속성이 “변화 방식”을 제어하고, 관계 속성이 “맥락”을 연결한다.
모달 대화상자 하나에 `role=“dialog”`, `aria-modal=“true”`, `aria-labelledby`, `aria-describedby`를
함께 적용해야 완전한 접근성이 확보되는 것처럼, 범주를 조합하는 방식이 실제 구현의
핵심이다.

### 라이브 영역의 인터럽트 계층 설계

`aria-live`의 세 가지 값(`off`, `polite`, `assertive`)은 사용자 흐름 방해 정도에 따른
계층이다.
`assertive`는 현재 읽는 내용을 중단시키므로 진정한 긴급 상황—오류, 보안 경고—에만
적합하다.
`polite`는 알림, 검색 결과, 상태 업데이트의 기본값이다.
이 계층을 무시하고 모든 동적 콘텐츠를 `assertive`로 처리하면, 모든 알림이 경보처럼
동작해 스크린 리더 사용자 경험을 망친다.

### 관계 속성과 DOM 구조의 분리

`aria-owns`는 특히 주목할 만하다.
DOM 구조상 부모-자식이 아닌 요소들을 논리적으로 연결할 수 있기 때문이다.
드래그 앤 드롭으로 재정렬 가능한 목록이나, CSS 포지셔닝으로 시각적 위치와 DOM 순서가
다른 UI에서 보조 기술이 올바른 순서로 콘텐츠를 해석하도록 안내한다.
이는 “시각적 표현”과 “의미적 구조”를 분리해 관리할 수 있음을 보여준다.

### 흔한 실수와 그 원인

저자가 나열한 실수들—시맨틱 HTML 불필요한 대체, ARIA 상태와 실제 UI 상태 불일치,
보조 기술에서 인터랙티브 콘텐츠 숨김—은 공통 패턴을 가진다.
개발자가 ARIA를 “장식”으로 추가하고 실제 동작과 연결하는 로직을 빠뜨리는 경우가 많다.
예를 들어 `aria-expanded=“false”`를 HTML에 작성하고, 아코디언을 열 때 JavaScript로
이를 `true`로 업데이트하지 않으면 스크린 리더는 항상 닫혀 있다고 읽는다.

## 비평

### 강점: 범주별 체계화와 실전 코드 예시

개념이 추상적으로 느껴지기 쉬운 ARIA를 네 범주로 나눠 체계화한 구조가 좋다.
토글 버튼, 모달, 폼 유효성 검사 등 실무에서 자주 마주치는 패턴을 구체적인 코드와 함께
제시해 즉시 적용 가능하다.
테스트 섹션에서 자동화 도구와 실제 스크린 리더를 함께 언급한 것도 균형 잡힌 관점이다.

### 한계: “99%가 모른다”는 클릭베이트 제목

제목의 “99%가 모른다”는 주장은 검증되지 않은 과장이다.
`aria-labelledby`, `aria-describedby` 같은 관계 속성은 접근성 관련 글에서 빠지지 않는
기본 내용이다.
실제로 소개된 속성들이 “모르는” 것이라기보다, “알지만 제대로 사용하지 않는” 것에 가깝다.
제목이 내용의 가치를 오히려 낮춘다.

### 보완이 필요한 관점: 성능과 복잡도 트레이드오프

`aria-live=“polite”`로 표시된 영역은 DOM 변경마다 스크린 리더가 감지해야 하므로,
잘못 배치하면 불필요한 공지가 폭주할 수 있다.
또한 `aria-owns`로 복잡한 가상 트리를 만들면 유지보수가 어려워진다.
ARIA 남용(ARIA over-engineering)의 위험성과 “가능한 한 네이티브 HTML을 쓰라”는 원칙을
코드 예시로 대비해 보여줬다면 더 실용적인 글이 되었을 것이다.

## 인사이트

### ARIA는 실패의 보상재다

ARIA가 필요한 근본 원인은 웹 개발 생태계가 시맨틱 HTML을 충분히 사용하지 않는다는 것이다.
`<div>`와 `<span>`으로 버튼, 드롭다운, 탭을 만드는 관행이 수십 년간 이어지면서,
보조 기술은 이 “의미 없는” 요소들을 해석할 방법이 없었다.
ARIA는 이 실패를 사후에 수리하는 패치 레이어다.
만약 모든 개발자가 처음부터 `<button>`, `<select>`, `<details>` 같은 네이티브 HTML을
적극 사용했다면, ARIA의 상당 부분은 필요하지 않았을 것이다.
이 역사는 기술 부채의 전형적 패턴이다.
“빠르게 만들기” 위해 의미를 생략하고, 나중에 더 복잡한 레이어로 그 의미를 복원한다.
CSS-in-JS가 캐스케이드의 복잡함을 피하려다 다른 복잡함을 만든 것, 또는 GraphQL이
REST API의 오버페칭을 해결하려다 다른 문제를 도입한 것과 구조적으로 닮았다.
올바른 추상화를 처음부터 쓰는 것이 나중에 패치하는 것보다 항상 낫지만, 생태계가 한번
잘못된 방향으로 굳어지면 그 방향을 되돌리기보다 패치 레이어를 쌓는 것이 현실적 선택이
된다.

### “상태 동기화”는 접근성 버그의 핵심 패턴이다

ARIA 관련 버그의 가장 흔한 원인은 속성값과 UI 상태의 불일치다.
`aria-expanded=“false”`가 HTML에 선언되어 있는데 아코디언이 열려 있거나,
`aria-checked=“true”`인데 실제 체크박스가 해제되어 있는 경우다.
이는 더 넓은 소프트웨어 공학 문제인 “여러 곳에서 진실의 원천(source of truth)을 관리할
때 생기는 일관성 문제”의 접근성 버전이다.
React, Vue 같은 현대 프레임워크가 상태를 단방향으로 흐르게 설계한 것은 이 문제를
해결하기 위해서다.
ARIA 속성도 같은 원칙이 적용된다.
UI 상태가 변할 때 ARIA 속성을 자동으로 업데이트하는 단일 경로를 만들어야 한다.
헤드리스 UI 라이브러리(Radix UI, Headless UI 등)가 접근성 커뮤니티에서 주목받는 이유도
이것이다.
이런 라이브러리들은 ARIA 상태 동기화를 개발자 대신 처리해줌으로써, 접근성 버그의 가장
흔한 원인을 아예 제거한다.
“접근성을 나중에 추가한다”는 접근이 실패하는 이유도 여기 있다.
나중에 추가한 ARIA는 UI 상태와 동기화되지 않는 죽은 속성이 되기 쉽다.

### 접근성은 가장자리 기능이 아니라 설계의 척도다

ARIA가 10억 명의 장애인을 위한 것이라는 설명은 도덕적으로는 강력하지만, 개발자들이
접근성을 우선순위에 올리는 데는 충분하지 않은 경우가 많다.
더 설득력 있는 논거는 접근성 있는 UI가 비장애인에게도 더 좋다는 것이다.
키보드 전용으로 동작하는 모달은 포인터 장치 없이 일하는 전문가에게도 필수다.
`aria-live`로 동적 콘텐츠를 관리하는 습관은 느린 네트워크 환경에서 로딩 상태를 명확히
전달하는 코드로 이어진다.
시맨틱 HTML을 쓰는 개발자는 SEO, 성능, 유지보수성도 자연스럽게 좋아진다.
접근성 요구사항이 좋은 설계의 강제 함수(forcing function) 역할을 한다는 관점에서 보면,
ARIA와 시맨틱 HTML에 투자하는 것은 장애인만을 위한 것이 아니라 더 견고하고 유지보수
가능한 코드를 만드는 훈련이다.
반대로 생각하면, 접근성이 나쁜 코드베이스는 대부분 다른 품질 지표도 낮다.
DOM 구조가 시각적 표현에 종속되어 있고, 상태 관리가 분산되어 있고, 인터랙션 로직이
일관성 없이 흩어져 있는 코드에서 접근성을 나중에 추가하는 것은 사실상 불가능하다.
접근성은 “체크리스트 항목”이 아니라 “코드 품질의 결과물”이다.
