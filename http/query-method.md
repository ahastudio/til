# RFC 10008: HTTP QUERY 메서드

<https://www.rfc-editor.org/rfc/rfc10008.html>

HN 토론: <https://news.ycombinator.com/item?id=48568502> (365점, 153개 댓글)

## 요약

RFC 10008은 HTTP 메서드 레지스트리에 `QUERY`를 추가하는 표준 트랙 명세다.
저자는 J. Reschke(greenbytes), J.M. Snell(Cloudflare), M. Bishop(Akamai)이며 2026년 6월 발행되었다.
QUERY의 핵심은 요청 본문에 질의 파라미터를 담으면서도 GET처럼 안전(safe)하고 멱등(idempotent)한 메서드를 제공하는 것이다.
POST는 이 두 속성을 보장하지 않으므로 자동 재시도, 캐싱, 조건부 요청이 어렵다.
QUERY는 이 간극을 메운다.

`Content-Type` 요청 헤더는 필수다.
서버는 헤더가 없거나 일치하지 않으면 요청을 실패시켜야(MUST) 한다.
지원하지 않는 미디어 타입에는 415(Unsupported Media Type),
일관성 없는 콘텐츠에는 400(Bad Request),
유효한 질의가 존재하지 않는 리소스를 참조하면 422(Unprocessable Content)를 반환한다.
`Accept-Query` 응답 헤더 필드는 해당 URI에서 지원하는 질의 미디어 타입을 광고하며,
클라이언트가 OPTIONS로 사전 탐색할 수 있다.

캐싱에서는 캐시 키에 요청 본문과 관련 메타데이터가 반드시(MUST) 포함되어야 한다.
캐시는 콘텐츠 인코딩, 미디어 타입 접미사 같은 의미론적으로 무관한 차이를 정규화할 수 있다.
클라이언트는 `no-transform` 지시자로 이 정규화를 방지할 수 있다.
리다이렉트 처리에서 POST-to-GET 변환 예외는 QUERY에 적용되지 않으며,
303(See Other)은 GET으로 결과를 가져오도록 안내하는 데 사용된다.

QUERY는 “동치 리소스(equivalent resource)” 개념을 도입한다.
서버는 선택적으로 질의 정의와 결과 각각에 URI를 할당할 수 있다.
`Location` 헤더는 질의 자체를 나타내는 리소스를, `Content-Location`은 결과를 담은 리소스를 가리킨다.
이를 통해 클라이언트는 나중에 같은 결과를 GET으로 재조회할 수 있다.

CORS와 관련하여 QUERY는 사전 허용(safelisted) 메서드가 아니므로 프리플라이트 요청이 필요하다.
`Accept-Query` 헤더 필드는 Structured Fields 구문을 사용하며 와일드카드는 `*/*`와 `type/*` 패턴만 지원한다.
메서드 이름 QUERY는 PROPFIND, REPORT, SEARCH 후보들을 제치고 선택되었다.
WebDAV의 기존 의미론적 부담을 피하고 URI 질의 컴포넌트와의 의미론적 관계를 명확히 하기 위해서다.

## 분석

### GET-with-body 문제는 20년 된 미해결 과제였다

HTTP/1.1이 설계될 때 URI는 충분히 긴 질의 파라미터를 수용할 수 있다고 가정했다.
그러나 복잡한 검색 질의, GraphQL 오퍼레이션, ElasticSearch DSL, OData 필터는
URL 길이 제한(대부분 브라우저/서버 구현에서 2KB~8KB)을 초과한다.
현장의 해결책은 항상 POST였지만, POST는 안전하지도 멱등하지도 않다.
이는 중간 계층(프록시, CDN, 캐시)이 POST를 불투명하게 취급하게 만들었고,
자동 재시도를 막았으며, 캐싱을 불가능하게 했다.

WebDAV는 2004년 `SEARCH` 메서드(RFC 5323)로 이 문제를 해결했지만
WebDAV 컨텍스트에 묶여 일반 HTTP 생태계로 확산되지 못했다.
ElasticSearch는 GET 요청에 본문을 허용하는 비표준 관행을 채택했고,
일부 클라이언트 라이브러리와 로드 밸런서가 이를 거부하는 운영 문제가 반복적으로 발생했다.
QUERY는 이 오래된 해결책 없는 합의된 문제에 표준 메커니즘을 제공한다.

HN에서 piterrro는 이 비표준 관행이 얼마나 광범위하게 퍼졌는지를 단적으로 보여줬다.[^piterrro]
"수년간 GET 요청에 본문을 넣어 전송하고 있었다"는 것이다.
IETF 워킹 그룹이 GET-with-body를 검토했지만 역사적 상호운용성 문제로 기각한 배경이
현장에서는 이미 사실상 표준처럼 사용되는 관행으로 해결되고 있었음을 보여준다.

### 안전+멱등의 조합이 만드는 새로운 가능성

HTTP에서 “안전”하다는 것은 요청이 대상 리소스의 상태를 변경하지 않는다는 의미론적 약속이다.
“멱등”하다는 것은 동일 요청을 여러 번 보내도 결과가 같다는 보장이다.
이 두 속성의 조합이 QUERY를 POST와 근본적으로 다르게 만든다.

안전+멱등 요청은 네트워크 오류 시 자동으로 재시도할 수 있다.
CDN과 캐시 레이어가 응답을 저장하고 재사용할 수 있다.
조건부 요청(`If-Modified-Since`, `ETag`)으로 불필요한 전송을 피할 수 있다.
범위 요청으로 결과의 일부만 가져올 수 있다.
이 인프라는 GET에서 수십 년간 구축된 것이며, QUERY는 이를 요청 본문이 있는 질의에 개방한다.

mlhpdx는 QUERY가 표준이 되기 전부터 수년간 이 메서드로 구축해왔다고 밝혔다.[^mlhpdx]
특히 Range 헤더와의 조합을 페이지네이션에 활용하는 패턴이 자연스럽다고 언급했다.
RFC는 질의 형식에 내장된 페이지네이션 메커니즘(SQL의 `FETCH FIRST` 등)을 권장하지만,
"QUERY로 집합을 정의하고 Range로 부분집합을 가져온다"는 패턴도 실제 사용에서 유용하다는 것이다.

### 동치 리소스 개념은 조용히 REST 원칙을 보강한다

REST의 핵심 원칙 중 하나는 리소스 식별 가능성이다.
복잡한 POST 기반 질의는 결과에 URI가 없어 북마크, 공유, 재사용이 불가능했다.
QUERY의 `Location` 헤더는 질의 자체에 URI를 할당하는 선택지를 서버에 준다.
`Content-Location`은 결과 집합에 URI를 할당한다.

이 메커니즘은 질의 결과를 첫 번째 클래스 리소스로 만든다.
“이 질의로 찾은 결과”를 영구 링크로 공유하거나 GET으로 재조회하는 것이 가능해진다.
SPARQL 엔드포인트나 GraphQL에서 요청별 식별자가 없어 복잡했던 결과 캐싱 패턴이
표준 HTTP 메커니즘으로 해결될 수 있다.

## 비평

### 캐시 구현자에게 의미론적 지식을 요구하는 설계는 현실적이지 않다

명세는 캐시가 의미론적으로 무관한 차이를 정규화할 수 있다고 허용한다.
같은 SQL 질의의 다른 포맷팅, 동치인 JSONPath 표현 등이 예시다.
이는 캐시가 특정 질의 언어의 의미론을 이해해야 한다는 의미다.

그러나 HTTP 캐시는 전통적으로 애플리케이션 로직을 모른다.
Varnish, Nginx, Cloudflare의 캐시 레이어는 HTTP 헤더를 파싱하지, SQL 파서를 내장하지 않는다.
이 정규화를 구현하려면 질의 언어별 플러그인이나 애플리케이션 계층 캐시가 필요하다.
실제로는 대부분의 캐시가 정규화 없이 본문을 그대로 캐시 키로 사용할 것이다.

더 심각한 문제는 잘못된 정규화가 캐시 포이즈닝으로 이어질 수 있다는 점이다.
의미론적으로 다른 두 질의를 동치로 판단하면 잘못된 결과가 캐시에서 반환된다.
명세가 이 위험을 보안 고려사항에서 언급하지만, 구현자가 정규화를 잘못 구현할 경우의
데이터 노출 시나리오는 충분히 탐구되지 않는다.

100ms는 이 문제를 더 근본적인 각도에서 지적했다.[^100ms]
요청 본문 기반 캐시 키는 "사용자가 제어하는 무제한 캐시 키"를 만든다는 것이다.
대규모 JSON 필터나 바이너리 데이터를 본문에 담은 QUERY 요청이라면
캐시 키 자체가 수십 킬로바이트가 될 수 있다.
헤더 기반 캐시 키는 크기와 형식이 HTTP 스펙으로 제한되지만, 요청 본문은 그렇지 않다.
대안으로 POST 요청에 `Vary: request-body`처럼 캐싱 의미론을 부여하는 새 헤더를 추가하는 방식이
역방향 호환성을 유지하면서 이 문제를 더 좁게 해결할 수 있다는 제안도 나왔다.

### Accept-Query 헤더의 URI 경로 단위 적용은 실제 API 구조와 맞지 않는다

`Accept-Query` 헤더 필드는 동일한 경로를 공유하는 모든 URI에 적용되며,
질의 컴포넌트는 무시된다고 명세는 정의한다.
즉 `/api/search?type=user`와 `/api/search?type=product`는 같은 `Accept-Query` 값을 가진다.

그러나 현실의 API에서 같은 경로의 다른 리소스 타입이 다른 질의 형식을 지원하는 경우가 많다.
사용자 검색은 SQL-like 필터를 지원하고, 상품 검색은 ElasticSearch DSL을 지원하는 상황이다.
명세의 경로 단위 적용은 이 구분을 표현할 방법을 제공하지 않는다.
결과적으로 실제 구현에서는 `Accept-Query`가 가장 낮은 공통분모를 광고하거나,
실제 지원 능력보다 넓게 선언하는 방식을 선택하게 된다.
이는 탐색 가능성(discoverability)을 위한 헤더가 실제로는 신뢰할 수 없게 만드는 설계다.

### 핵심 주장이 스스로를 논박한다 — “긴 URL”과 “QUERY 미지원”은 같은 문제다

paulddraper는 QUERY의 근거로 자주 인용되는 “URL 길이 제한” 주장을 정면으로 반박했다.[^paulddraper]
“긴 URL을 지원하지 않는 시스템은 QUERY도 지원하지 않는다. 이 논거는 자기 모순이다.”
실제로 URL 길이 제한에 걸리는 환경이라면 — 오래된 로드 밸런서, 레거시 프록시, 구형 브라우저 —
새로운 HTTP 메서드를 인식하지 못할 가능성이 훨씬 높다.

이 주장이 완전히 틀린 것은 아니다.
QUERY는 긴 URL 문제보다 POST-as-query의 의미론적 오용을 해결하는 데 더 집중한다.
캐싱, 멱등성, 안전 속성을 명확하게 표현하는 것이 핵심 가치이며
URL 길이 제한은 부차적 동기다.
그러나 명세와 많은 소개 자료가 URL 길이 제한을 주요 동기로 제시하는 방식은
“URL이 짧으면 QUERY가 필요 없다”는 오해를 만들어, 실제 핵심 가치를 가리는 마케팅 문제가 있다.

### “안전” 속성의 의미론적 약속은 구현 강제력이 없다

HTTP에서 “안전”하다는 것은 프로토콜 수준의 보장이 아닌 의미론적 약속이다.
RFC 9110은 명시적으로 안전성이 서버 구현의 관찰 가능한 부작용 없음을 보장하지 않는다고 적시한다.
QUERY를 처리하면서 로그를 기록하고, 통계를 업데이트하고, 감사 레코드를 만드는 것은 허용된다.

문제는 클라이언트와 중간 계층이 이 약속에 의존하여 자동 재시도를 결정한다는 것이다.
만약 서버가 QUERY를 안전하지 않게 구현한다면 — 예를 들어 조회 카운터를 증가시키거나
사용자 활동 로그에 기록하는 방식으로 — 자동 재시도가 의도치 않은 부작용을 누적시킨다.
POST에서 이 문제는 이미 잘 알려져 있어 개발자들이 주의한다.
QUERY는 GET처럼 안전하다는 새로운 기대를 심으면서 같은 구현 위험을 가진다.
이 기대 차이가 오히려 더 교묘한 버그를 만들 수 있다.

## 인사이트

### GraphQL의 기술적 차별화 근거 중 하나가 약화된다

GraphQL이 REST에 제기한 핵심 기술 주장 중 하나는 단일 엔드포인트에서
타입 안전하고 복잡한 질의를 POST로 보내는 것이 REST보다 표현력이 뛰어나다는 것이었다.
REST가 복잡한 질의를 URI로 표현해야 한다는 제약은 GET의 한계에서 비롯되었다.

QUERY는 이 제약을 제거한다.
임의의 질의 언어를 안전+멱등한 방식으로 HTTP에 실을 수 있게 되면,
REST 엔드포인트가 QUERY 메서드로 GraphQL-like 표현력을 얻는다.
GraphQL 생태계가 REST보다 우위를 가졌던 캐싱 어려움, 서버 지정 엔드포인트 구조,
네트워크 전송 최적화 문제들이 QUERY의 표준 캐싱 의미론으로 해소될 수 있다.
이것이 GraphQL을 대체하지는 않겠지만, 기존 REST API를 GraphQL로 마이그레이션할
기술적 동기를 줄인다.

inlined은 QUERY가 더 즉각적으로 해결할 수 있는 현재 문제를 지적했다.[^inlined]
스트리밍 AI 질의다.
JavaScript의 `EventSource` API는 서버-전송 이벤트(SSE) 스트리밍에 사용되지만
GET 메서드만 지원한다는 제약이 있다.
AI 추론 요청처럼 요청 본문이 필요하면서 응답을 스트리밍해야 하는 패턴에서,
모든 팀이 POST + `text/event-stream`을 사용하지만 이는 상태 변경이 없는 조회 작업에 POST를 오용하는 것이다.
QUERY가 `EventSource`에서 지원된다면 이 오용이 사라지고,
SSE 스트리밍 응답이 안전+멱등한 의미론을 가지게 된다.

### 표준화 속도는 채택보다 느리지 않다 — 채택이 표준을 기다리지 않는 게 문제였다

QUERY가 해결하는 문제는 20년 이상 존재했다.
ElasticSearch는 GET-with-body를, GraphQL은 POST-as-query를, OData는 `$filter`를
각자 해결책으로 채택했다.
이 각각의 해결책은 호환되지 않으며, 미들웨어와 도구가 이 비표준 패턴들을 별도로 처리해야 한다.

표준화 지연이 낳은 다른 결과는 생태계 파편화다.
지금 QUERY가 표준이 되어도 기존 시스템들이 즉시 마이그레이션할 이유는 없다.
GraphQL 서버는 POST를 계속 지원할 것이고, ElasticSearch는 GET-with-body를 계속 허용할 것이다.
표준화의 가치는 새 시스템에서 발현된다.
HTTP 표준이 미들웨어 계층(로드 밸런서, API 게이트웨이, CDN)에서 QUERY를 올바르게
처리하도록 구현자를 안내한다는 것이 가장 즉각적인 효과다.

### 요청 본문의 캐시 키 포함은 캐시 아키텍처의 근본 전환을 요구한다

기존 HTTP 캐시 구현은 URI + 선택적 헤더를 캐시 키로 사용한다.
Varnish, Nginx, CDN들의 기본 캐싱 인프라는 이 가정 위에 수십 년간 최적화되었다.
QUERY는 요청 본문을 캐시 키에 포함시켜야 한다는 요구사항으로 이 아키텍처에 도전한다.

요청 본문은 헤더보다 크고, 구조화되어 있으며, 파싱이 필요하다.
현재 대부분의 HTTP 캐시 레이어는 QUERY를 만나면 두 가지 중 하나를 선택할 것이다.
캐싱하지 않거나(성능 손실), 본문을 불투명한 해시로 취급하여 정규화 없이 캐싱하는 것이다.
명세가 의도하는 의미론적 정규화를 실현하려면 각 질의 언어를 이해하는 특화된 캐싱 레이어가 필요하다.
이는 애플리케이션 계층 캐시(Redis, Memcached를 쿼리 결과 저장에 직접 사용)가
표준 HTTP 캐시보다 실제 QUERY 캐싱 구현에서 더 실용적인 선택이 될 것임을 시사한다.

### 동치 리소스 패턴은 검색 엔진과 링크 공유의 미래를 바꿀 수 있다

현재 복잡한 검색 결과는 공유가 어렵다.
POST 기반 질의의 결과에는 URL이 없어 직접 링크로 공유할 수 없다.
GET 기반 복잡한 질의는 URL이 너무 길거나, 시스템이 URL에 민감한 파라미터를 노출한다.

QUERY의 `Location` 헤더를 통한 동치 리소스 할당은 이 문제를 해결할 구조를 제공한다.
서버는 QUERY 요청을 받아 결과를 생성하고, 그 결과에 영구 URI를 할당하여 반환할 수 있다.
이후 같은 URI로 GET 요청이 오면 동일한 결과를 즉시 반환한다.
이 패턴은 검색 엔진이 동적 질의 결과를 인덱싱하고, 사용자가 복잡한 필터가 적용된 결과를
북마크하며, API 응답을 CDN에서 캐싱하는 새로운 가능성을 열어준다.
표준화가 이 패턴의 생태계 지원을 촉진할 수 있다면, 웹의 주소 지정 가능성이 의미 있게 확장된다.

CodesInChaos는 HTML 폼과의 통합 가능성을 제기했다.[^CodesInChaos]
`<form method="query">`가 지원된다면 POST 폼 제출 후 새로 고침 시 나타나는
"양식을 다시 제출하시겠습니까?" 경고가 사라진다.
QUERY는 멱등하므로 브라우저가 재제출 확인 없이 결과를 재조회할 수 있기 때문이다.
이는 검색 폼, 필터 폼, 복잡한 조회 UI 모두에 해당한다.
HTML 스펙이 이를 채택한다면, 사용자 경험의 오랜 마찰 중 하나가 프로토콜 수준에서 해결된다.
