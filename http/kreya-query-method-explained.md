# 새로운 HTTP QUERY 메서드 설명

원문: <https://kreya.app/blog/new-http-query-method-explained/>

HN 토론: <https://news.ycombinator.com/item?id=48640974> (217점, 171개 댓글)

## 요약

Kreya 블로그의 Manuel이 작성한 이 글은 RFC 10008이 정의한 HTTP QUERY 메서드를
실무 개발자 관점에서 설명한다.
GET 파라미터로 복잡한 질의를 표현할 때 생기는 문제들 — URL 길이 초과, 특수문자 인코딩,
배열·중첩 구조의 비표준 표현, 서버 로그에 파라미터가 노출되는 프라이버시 우려 —을
출발점으로 삼는다.
HTTP 스펙이 GET 요청 본문을 명시적으로 금지하지는 않지만,
클라이언트·프록시·웹서버 구현마다 이를 다르게 처리(거부하거나, 본문을 버리거나, 오해석)하기 때문에
실무에서 GET-with-body는 신뢰할 수 없다.

현재의 우회책은 POST다.
POST는 본문을 허용하지만 멱등하지 않고 리소스 생성·처리 의미론을 가지므로,
자동 재시도가 불가하고, 미들웨어가 이를 읽기 전용 작업으로 인식하지 못한다.
QUERY는 GET처럼 안전(safe)하고 멱등(idempotent)하면서 본문을 지원하도록 설계됐다.
이를 통해 캐싱, 조건부 요청, 자동 재시도 인프라를 활용할 수 있다.

글은 QUERY 채택 전 고려할 주의사항(Gotchas)으로 마무리된다.
브라우저와 서버의 지원이 아직 제한적이고 광범위한 채택까지 수년이 걸릴 수 있다.
단순한 필터링에는 GET 파라미터가 여전히 적합하다.
QUERY 요청 결과는 북마크 가능한 URL로 공유할 수 없다.
GET보다 캐시 구현이 복잡하다.

## 분석

### 글의 구조는 기술 소개가 아닌 설득을 위해 설계됐다

이 글은 문제-우회책-해결책의 서사 구조를 따른다.
GET의 한계를 상세히 나열하고, POST 우회책의 의미론적 문제를 부각한 뒤,
QUERY가 두 문제를 모두 해결한다고 제시한다.
이 구조는 독자가 QUERY를 “당연히 필요한 것”으로 받아들이도록 유도하는 마케팅 논리다.

그런데 HN 토론에서 드러나는 개발자 커뮤니티의 반응은 이 서사 구조가 해결하지 못한 균열을 보여준다.
grugdev42는 “그냥 GET 요청에 선택적 본문 지원을 추가했으면 훨씬 단순했을 것”[^grugdev42]이라고 말했고,
veltas는 “표준은 관행을 따라야 한다 — 이미 작동하는 GET+body 방식을 새로운 문법으로 대체하는 것은
QUERY가 보편화된 후 기존 방식 지원이 사라지는 더 큰 호환성 문제를 낳을 수 있다”[^veltas]고 반론했다.
이 글은 POST 우회책의 단점만 부각할 뿐, GET+body를 표준화하는 대안이 왜 선택되지 않았는지 설명하지 않는다.
RFC 설계 과정에서 이 논쟁은 이미 있었고 QUERY라는 선택이 나왔다.
하지만 블로그 포스트는 그 선택의 근거를 생략함으로써 반론의 여지가 없는 것처럼 제시한다.

Kreya는 gRPC·REST·GraphQL API 테스트 도구다.
새 HTTP 메서드가 보편화되면 Kreya가 이를 지원하는 도구로서 가치가 높아진다.
기술 블로그가 자사 제품의 가치 상승 조건을 설명하는 패턴은 흔하며,
그 자체가 결함은 아니지만 글의 프레이밍이 어디서 과장됐는지 가늠하는 데 유용하다.

### 실무 가이드로서의 접근이 RFC 독해를 대체하고 있다

RFC 10008 자체는 기술적 정밀도를 위해 작성된 명세다.
이 글은 명세의 핵심을 개발자가 이해하기 쉬운 언어로 압축하면서,
“왜 이 방법인가”를 실무 맥락으로 재구성한다.
`Content-Type` 필수, `Accept-Query` 헤더, 동치 리소스 개념 같은 명세의 세부사항은
언급되지 않거나 단순화된다.

이 접근은 독자층을 명세 독자가 아닌 “API를 설계하는 개발자”로 설정한다.
그 결과 글의 유용성은 높지만, 실제 구현에 필요한 정보를 모두 담지는 않는다.
명세 대신 이 글을 읽고 QUERY를 구현하면 빠뜨리는 부분이 생길 수 있다.

### Gotchas 섹션은 문제를 인정하면서 채택을 권장하는 균형 전략이다

“주의사항”을 명시적으로 나열하는 글 구성은 신뢰를 쌓는 효과가 있다.
독자는 저자가 단점을 숨기지 않는다고 느낀다.
동시에 Gotchas를 짧고 완화된 언어로 제시함으로써,
잠재적 채택 장벽을 심각하게 보이지 않도록 관리한다.

“광범위한 채택까지 수년이 걸릴 수 있다”는 언급은 사실이지만,
그것이 실제로 무엇을 의미하는지 — 레거시 프록시, 엔터프라이즈 방화벽,
CDN 지원 부재 — 를 구체화하지 않는다.
주의사항의 무게가 균등하게 전달되지 않는다.

## 비평

### “북마크 가능한 URL 불가”는 동치 리소스 개념을 빠뜨린 불완전한 서술이다

글은 “QUERY 요청으로 북마크 가능한 URL을 얻을 수 없다”를 Gotcha로 나열한다.
이 진술은 RFC 10008의 핵심 메커니즘 중 하나를 누락한 결과다.
RFC는 `Location` 헤더를 통해 서버가 질의 자체에 URI를 할당하는 “동치 리소스(equivalent resource)” 개념을 정의한다.
서버가 이를 구현하면 QUERY 결과를 GET으로 재조회할 수 있는 URI를 클라이언트에 돌려줄 수 있다.

이 생략은 우연이 아닐 수 있다.
동치 리소스 패턴은 서버 측 구현이 필요하며, 클라이언트 도구 입장에서는 서버가 지원하는지 여부가 불확실하다.
Kreya 같은 API 테스트 도구가 “이 기능을 설명하면 서버 미지원 시 실망이 더 크다”고 판단해 생략했을 가능성이 있다.
그러나 이 생략은 QUERY가 GET보다 공유 가능성이 근본적으로 열등하다는 오해를 심는다.

역설적으로, “공유 불가”라는 특성이 오히려 악용될 수 있다.
IshKebab은 “사이트들이 이를 링크 공유를 막는 용도로 사용하기 시작하면 어떻게 될까”[^IshKebab]라고 물었다.
스트리밍 서비스, 유료 컨텐츠 플랫폼, 폐쇄적 소셜 네트워크가 QUERY를 의도적으로 채택해
결과 페이지를 외부에서 접근 불가능하게 만드는 시나리오는 충분히 현실적이다.
“북마크 불가”를 Gotcha로 나열한 글은 이를 결함으로 보지만,
특정 서비스 운영자에게는 이 “결함”이 오히려 기능이다.
표준이 악용 가능성을 내포하고 있을 때, 그것을 Gotcha로 완화해 소개하는 방식은 개발자에게 충분한 경고가 되지 않는다.

### 단순-복잡 경계의 기준이 없어 실무 판단을 돕지 못한다

글은 “단순한 필터링에는 GET을, 복잡한 질의에는 QUERY를”이라고 조언한다.
이 조언이 유용하려면 경계가 어디인지 기준이 있어야 한다.
URL 길이 제한을 기준으로 제시하지만, 이 제한은 브라우저·서버·프록시마다 다르다.
2KB인지 8KB인지 32KB인지에 따라 GET으로 충분한 질의 범위가 달라진다.

경계 기준의 부재는 실무 개발자가 “지금 내 상황에서 QUERY를 써야 하나?”라는
가장 중요한 질문에 답하지 못하게 만든다.
“복잡하다고 느껴지면 QUERY를 고려하라”는 것은 조언이 아니라 책임 회피다.
글이 실제로 도움이 되려면 구체적인 결정 기준 — 파라미터 수, 중첩 깊이, URL 길이 임계값 —을 제시해야 한다.

### 채택 장벽을 “수년”으로 요약하는 것은 실제 위험을 과소평가한다

“광범위한 채택까지 수년이 걸릴 수 있다”는 언급은 기다리면 해결되는 문제처럼 들린다.
그러나 HTTP 메서드의 생태계 채택은 단순한 시간 문제가 아니다.
WAF(웹 방화벽), SSL 인터셉트 프록시, 엔터프라이즈 네트워크 장비 중 일부는
화이트리스트에 없는 HTTP 메서드를 능동적으로 차단한다.
이 장비들의 교체 주기는 “수년”이 아니라 “10년 이상”인 경우가 많다.

QUERY를 프로덕션에 배포한다는 것은 “내 서버가 QUERY를 지원한다”는 의미가 아니다.
클라이언트에서 서버까지의 모든 네트워크 경로가 QUERY를 올바르게 처리해야 한다.
이 경로에 하나의 미지원 장비가 있으면 요청 전체가 실패하거나, 더 나쁘게는 조용히 오동작한다.
글이 이를 “주의사항” 수준으로 처리하는 것은 실제 배포 위험을 개발자에게 전달하지 못한다.

HN에서 Asmod4n은 더 구체적인 공포를 지목했다.
“수천 달러짜리 SSL 인터셉트 프록시 어플라이언스를 구매한 기업들이 있는데, 이 장비들은 HTTP/1.1만 지원한다.
새 HTTP 동사를 만나면 그 회사의 실적을 볼 때 DoS 수준의 장애를 일으킬 가능성이 높다.”[^Asmod4n]
이 지적은 추상적인 “방화벽 문제”를 구체적인 장비와 장애 시나리오로 좁힌다.
실제로 SSL 인터셉트 프록시는 HTTPS 트래픽을 복호화해 검사하는 방식으로 작동하기 때문에,
인식하지 못하는 메서드를 받으면 요청을 누락하거나 오류 응답을 내보낼 수 있다.

tosti는 또 다른 각도에서 이 문제를 찔렀다.
“GET-with-body가 기업 방화벽 때문에 안 된다고 했는데, QUERY도 당분간은 같은 이유로 안 된다.”[^tosti]
이 지적은 글의 논리 구조를 무너뜨린다.
글은 GET-with-body의 단점으로 “기업 프록시에서 거부당할 수 있다”는 점을 들며 QUERY를 대안으로 제시하지만,
QUERY도 지원하지 않는 프록시라면 같은 이유로 거부당한다.
두 방법 사이의 실질적 차이는 “미래에 표준화됐을 때”에만 발생한다 — 그 미래가 언제인지는 아무도 모른다.

## 인사이트

### 도구 회사의 기술 블로그가 RFC보다 먼저 읽히는 시대

HTTP QUERY를 처음 접하는 개발자 대부분은 RFC 10008 명세를 직접 읽지 않는다.
Kreya 블로그, MDN, 스택오버플로 답변, 유튜브 설명 영상이 먼저 소비된다.
이 현상은 기술 표준의 실질적 확산 경로가 공식 문서에서 도구 생태계로 이동했음을 보여준다.

이 구조에서 도구 회사는 기술 표준의 대중화 주체가 된다.
Kreya가 QUERY를 지원하고 이를 설명하는 블로그를 쓰는 행위는,
QUERY 생태계의 성장에 기여하면서 동시에 자사 도구의 필요성을 강화한다.
표준 기관(IETF)과 도구 회사(Kreya) 사이에 새로운 공생 관계가 형성된다.
이 관계에서 도구 회사의 설명이 표준을 단순화하고 오해를 만들 수 있다는 위험이 있지만,
동시에 표준이 실제 채택으로 이어지도록 돕는 다리 역할을 한다.

HN에서 1vuio0pswjnm7은 이 관계를 더 근본적으로 해체하는 관점을 제시했다.[^1vuio0pswjnm7]
“HTTP를 세 가지로 정의할 수 있다.
1) 가장 인기 있는 HTTP 서버가 실제로 받아들이는 것.
2) 가장 인기 있는 HTTP 클라이언트가 실제로 보내는 것.
3) RFC가 말하는 것.
나는 1번을 선택한다.”
이 관점에서는 RFC 10008이 공식 표준이 됐다는 사실보다,
nginx·Apache·Caddy가 QUERY를 실제로 처리하는 방식이 더 중요하다.
Kreya의 블로그는 RFC 권위를 바탕으로 QUERY를 정당화하지만,
실무에서 HTTP는 RFC대로 작동한 적이 거의 없었다.
HTTP/1.1 명세가 GET 요청 본문을 금지하지 않음에도 프록시들이 이를 거부한 것이 그 증거다.
QUERY가 “표준”이 된다는 것과 QUERY가 “작동한다”는 것은 같은 주장이 아니다.

### Gotchas 패턴은 신뢰를 쌓는 동시에 비판의 날을 무디게 만든다

기술 블로그에서 “주의사항”을 명시적으로 나열하는 패턴은 최근 몇 년간 표준화됐다.
단점을 숨기지 않는다는 인상을 주어 독자의 신뢰를 얻으면서,
동시에 단점을 저자가 선택한 프레이밍과 분량으로 제시하는 통제권을 유지한다.
이 글의 Gotchas는 네 가지인데, 각각 한두 문장으로 처리되고 “그럼에도 써라”는 암묵적 결론으로 이어진다.

독자가 이 패턴을 인식하면 Gotchas를 다르게 읽어야 한다.
“채택 장벽이 있다”는 문장에서 멈추지 말고, 그 장벽이 내 상황에서 얼마나 높은지 독립적으로 평가해야 한다.
이 글이 언급하지 않은 Gotcha — 동치 리소스 서버 구현 부담, CORS 프리플라이트 추가 요청,
엔터프라이즈 WAF 차단 위험 — 가 어떤 팀에게는 실제 도입 불가 요인이 될 수 있다.

### 실용 가이드의 단순화가 만드는 두 번째 오해 경로

RFC를 읽지 않고 실용 가이드만 읽는 개발자는 두 종류의 오해를 갖게 된다.
첫 번째는 이 글처럼 긍정적 단순화에서 오는 오해 — QUERY가 GET보다 공유 가능성에서 열등하다는 오해.
두 번째는 부정적 단순화에서 오는 오해 — QUERY가 단순히 “본문 있는 GET”이라는 오해.

두 번째 오해가 더 위험하다.
QUERY를 “본문 있는 GET”으로만 이해하면,
`Content-Type` 필수 요건, `Accept-Query` 헤더의 탐색 가능성 메커니즘,
동치 리소스 할당, CORS 프리플라이트 요구사항을 모두 놓친다.
실용 가이드는 진입 장벽을 낮추지만, 그 장벽이 다음 단계에서 더 높게 나타나는 결과를 낳을 수 있다.
이 글을 읽고 QUERY를 도입하기로 결정했다면, RFC 10008 전문을 읽는 것이 다음 단계여야 한다.
