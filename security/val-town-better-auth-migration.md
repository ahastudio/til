# Val Town의 Clerk에서 Better Auth로 인증 마이그레이션

원문: <https://blog.val.town/better-auth>

## 요약

Tom MacWright는 Val Town이 인증 서비스를 Clerk에서 Better Auth로 이전한 과정을 기록한다.
Val Town은 2023년 이미 Supabase에서 Clerk으로 한 번 이전한 바 있다.
2회 연속 인증 마이그레이션은 벤더 의존성을 잘못 선택했을 때 치르는 비용을 생생하게 보여준다.

Clerk의 핵심 문제는 두 가지였다.
첫째, 유저 테이블 대체품으로서의 심각한 한계다.
Clerk은 유저 세션 테이블 역할을 하도록 설계됐지만, 계정 전체에 초당 5회 요청 제한이 있었다.
Val Town은 다른 유저의 콘텐츠와 아바타를 표시하는 소셜 기능을 갖고 있어, 자신의 데이터만 보는 일반적인 앱보다 훨씬 빈번하게 유저 데이터를 조회해야 했다.
결국 Clerk과 내부 유저 테이블을 이중으로 운영하고 웹훅으로 데이터를 동기화하는 복잡한 구조가 됐다.
둘째, 단일 장애점 문제다.
모든 세션 갱신이 Clerk 서버 호출을 필요로 했기 때문에 Clerk 장애가 발생하면 로그인된 사용자조차 서비스를 이용할 수 없었다.
2025년 5월 이후 Clerk의 가동률은 “2~3 나인(99% 내외)”에 머물렀다.

그럼에도 3년간 이전을 미룬 이유가 있다.
벤더를 반복 교체하는 것 자체가 개발 속도를 손상시키고, Clerk의 Remix/Fastify/Express 지원 SDK 품질은 좋았다.
단순한 애플리케이션에는 Clerk이 잘 작동했으며, 대안이 마땅치 않았다.

Better Auth는 오픈소스 코어를 가지고 있고 세션 관리를 서드파티 가용성과 독립적으로 처리한다.
대시보드 같은 편의 기능은 선택적 플러그인으로 제공되어 핵심 인프라에 포함되지 않는다.
마이그레이션은 Better Auth와 Clerk을 전환 기간 동안 병렬 운영하는 방식으로 진행했다.
사용자들이 새 세션 방식을 점진적으로 채택하도록 하면서 리스크를 최소화했다.

## 분석

### 소셜 플랫폼과 인증 서비스의 구조적 불일치

Clerk 문제의 근원은 초당 5회 요청 제한이 아니라, Clerk의 설계 전제가 Val Town의 사용 패턴과 맞지 않는다는 것이다.
Clerk은 “유저가 자신의 데이터만 본다”는 전제로 설계됐다.
로그인한 유저가 자신의 프로필을 조회하고 자신의 세션을 갱신하는 패턴이다.
Val Town은 소셜 기능이 있어 다른 유저들의 데이터를 빈번하게 조회한다.
이 패턴 불일치가 Clerk을 “유저 테이블”로 쓰는 것을 구조적으로 실패하게 만들었다.

이것은 “기능 목록”이 아닌 “사용 패턴”으로 서비스를 평가해야 한다는 교훈을 준다.
Clerk의 기능 목록은 Val Town의 요구사항을 표면적으로는 충족했다.
문제는 실제 운영에서 드러나는 규모와 패턴이었다.

### 단일 장애점 설계의 위험

인증 서비스가 단일 장애점이 되는 구조는 일반적인 SaaS 아키텍처의 맹점이다.
“인증은 복잡하니 전문 서비스에 위임한다”는 결정이 자연스럽게 이루어지지만,
그 서비스가 장애를 겪으면 자체 시스템이 완벽해도 서비스 전체가 중단된다.

“시스템의 신뢰성은 핵심 의존성들의 최솟값과 같다”는 원칙은 단순하지만 실천적으로 간과되기 쉽다.
SaaS 스택이 복잡해질수록, 각 서비스의 개별 가동률이 아니라 전체 체인의 결합 가동률을 계산해야 한다.
99.9% 가동률의 서비스 세 개가 체인으로 연결되면 가동률은 99.7%가 된다.

### 점진적 마이그레이션 전략의 설계

Better Auth와 Clerk을 병렬 운영하는 전환 전략은 인증 마이그레이션의 특수한 어려움을 해결한다.
인증 시스템을 교체할 때 가장 큰 위험은 “모든 사용자의 세션을 일시에 무효화”하는 강제 전환이다.
병렬 운영으로 사용자들이 자연스럽게 새 세션으로 전환하도록 하면, 전환 시 사용자 경험의 중단이 없다.

이 전략의 기술적 요건은 두 인증 시스템을 동시에 검증할 수 있는 미들웨어 설계다.
요청이 들어왔을 때 새 세션(Better Auth)과 구 세션(Clerk) 모두를 확인하고, 어느 쪽이든 유효하면 통과시키는 구조다.
이것은 인증 레이어의 추상화가 제대로 되어 있을 때 가능한 접근이다.

## 비평

### 강점: 실제 운영 경험에서 나온 구체성

이 글의 가치는 “이론상 좋아 보이는” 아키텍처 조언이 아니라, 3년간 실제로 겪은 구체적인 문제에서 비롯됐다는 점이다.
“초당 5회 요청 제한”, “2~3 나인의 가동률”, “2023년 이전 후 3년”이라는 구체적인 데이터가 설득력을 만든다.
실패의 과정을 숨기지 않고 드러내는 방식이 유사한 상황의 개발팀에게 실질적인 준거를 제공한다.

### 약점: Better Auth의 장기 지속 가능성 미검토

Better Auth로의 이전 결정을 지지하는 근거가 충분히 제시되지만, Better Auth 자체의 지속 가능성에 대한 분석이 없다.
오픈소스 인증 라이브러리는 수없이 등장했다가 사라졌다.
유지보수 인력, 보안 취약점 대응 속도, 커뮤니티 규모가 장기적으로 중요한데, 이 글은 현재 시점의 코드 품질에 주로 집중한다.
“오픈소스이므로 우리가 포크할 수 있다”는 논리가 암묵적으로 전제되지만, 인증 라이브러리를 직접 유지보수하는 비용은 상당하다.

### 약점: 마이그레이션 비용의 과소 서술

“세 번의 인증 마이그레이션”이라는 상황의 실제 비용 — 개발 공수, QA 비용, 보안 감사, 사용자 혼란 — 이 충분히 다뤄지지 않는다.
이 글이 마이그레이션을 비교적 합리적인 결정으로 서술하지만, 마이그레이션 자체의 리스크와 비용이 구체화되지 않으면 유사한 결정을 하는 팀들이 비용을 과소평가할 수 있다.

### 보완: 인증 서비스 선택 프레임워크의 부재

Clerk이 왜 처음에 좋아 보였는지, 어떤 기준으로 선택했는지, 소셜 기능 요구사항이 처음부터 있었는지 등의 맥락이 더 있었다면 유용했다.
사후 진단은 명확하지만, “처음에 어떻게 평가했어야 했는가”에 대한 프레임워크가 없으면 독자가 같은 실수를 반복하기 쉽다.

## 인사이트

### 벤더 의존성은 기능이 아니라 운영 패턴으로 평가해야 한다

인증 서비스를 포함한 SaaS 벤더 선택은 보통 기능 목록 비교로 이루어진다.
“OAuth 지원 여부”, “MFA 제공 여부”, “SDK 품질” 같은 체크리스트다.
Val Town의 경험은 이 방식의 맹점을 드러낸다. Clerk은 체크리스트를 통과했지만 실제 운영 패턴에서 실패했다.

올바른 평가 방법은 운영 패턴 시뮬레이션이다.
“하루에 유저 데이터를 몇 번 조회하는가?”, “피크 시간에 초당 인증 요청이 얼마나 발생하는가?”, “인증 서비스가 5분 다운될 때 서비스에 무슨 일이 일어나는가?”를 미리 시뮬레이션해야 한다.
이것은 단순히 성능 테스트가 아니라 의존성 실패 모드를 분석하는 것이다.

더 근본적으로, 벤더 선택 시 “이 서비스가 없어지면 어떻게 되는가”를 먼저 물어야 한다.
답이 “서비스 전체가 멈춘다”라면, 그 의존성은 핵심 인프라가 되어선 안 된다.
외부 의존성이 핵심 인프라가 될 때는 그 서비스의 장애를 자체적으로 처리할 수 있는 설계 — 캐싱, 폴백, 독립적 세션 검증 — 가 함께 있어야 한다.

### 추상화의 깊이가 마이그레이션 비용을 결정한다

Val Town이 병렬 운영 전략을 선택할 수 있었던 것은 인증 레이어가 충분히 추상화되어 있었기 때문이다.
인증 로직이 애플리케이션 전반에 분산되어 있었다면, 두 시스템을 동시에 지원하는 미들웨어를 만드는 것이 불가능하거나 매우 어려웠을 것이다.

이것은 소프트웨어 설계에서 추상화의 실용적 가치를 보여준다.
“지금 당장 필요 없는데 왜 추상화 레이어를 만드는가”라는 질문에 대한 답이 여기 있다.
추상화 레이어는 현재 유연성이 아니라 미래 변경 비용을 낮추기 위한 투자다.
인증, 결제, 이메일 서비스 같은 외부 의존성이 높은 영역을 추상화하면, 벤더 교체 시 교체 범위가 추상화 경계 안으로 제한된다.

반대로 외부 서비스의 API가 직접 코드베이스 전체에 퍼져 있으면 마이그레이션 비용이 기하급수적으로 증가한다.
Clerk SDK 메서드를 직접 호출하는 코드가 100곳에 있다면 Better Auth로 전환할 때 100곳을 모두 수정해야 한다.
추상화가 있다면 한 곳만 바꾸면 된다.
이 차이가 “3주 마이그레이션”과 “6개월 마이그레이션”을 가른다.

### 오픈소스 인프라로의 귀환이 가속되는 구조적 이유

Val Town이 Clerk 같은 관리형 서비스에서 Better Auth 같은 오픈소스로 이전하는 흐름은 개별 사례를 넘어 업계 트렌드를 반영한다.
2010년대 중반 이후 “SaaS로 모든 것을 위임하라”는 경향이 지배적이었다.
인증은 Clerk, 결제는 Stripe, 이메일은 SendGrid, 로그는 Datadog.
각각의 서비스가 탁월했고, 직접 구현하는 것보다 명백히 빠르게 시장에 나갈 수 있었다.

그러나 성숙한 제품이 될수록 이 SaaS 스택의 비용 구조와 의존성 위험이 드러난다.
요청량이 많아질수록 SaaS 비용이 직접 구현 비용을 초과한다.
서비스가 복잡해질수록 SaaS의 제약이 필요한 커스터마이제이션을 가로막는다.
스타트업에서 성장 단계로 넘어가면서 “빠른 시작”을 위한 선택이 “확장의 장벽”이 된다.

동시에 오픈소스 생태계가 성숙했다.
2020년대 중반의 Better Auth, Zod, tRPC, Drizzle 같은 라이브러리들은 5년 전보다 훨씬 나은 개발자 경험을 제공한다.
“직접 구현”이 더 이상 “수개월의 고통”을 의미하지 않을 때, SaaS 벤더에 지불하는 비용과 의존성 위험을 감수할 이유가 줄어든다.
Val Town의 마이그레이션은 이 균형점이 특정 규모와 복잡성 이상에서 역전되는 현상의 사례다.
