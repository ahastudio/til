# Hacking Google with A.I. for $500,000

원문: <https://brutecat.com/articles/hacking-google-with-ai/>

HN 토론: <https://news.ycombinator.com/item?id=48496803> (16점, 0개 댓글)

## 요약

Arvin Shivram은 공동 연구자 Michael과 함께 AI 보조 퍼징(fuzzing)으로 Google 인프라에서 취약점 24건을 발견하고 총 50만 달러의 버그 바운티를 수령했다.
연구의 핵심은 Google이 제공하는 “디스커버리 문서(discovery document)”를 API 명세서로 활용하고, Claude AI를 통합한 커스텀 도구로 1,500개 이상의 엔드포인트를 체계적으로 테스트한 것이다.

공격 표면 수집 단계에서 두 사람은 APKMirror에서 Android APK 6만 1,200개를 스크레이핑하고, Chrome Debugger API를 활용한 확장 프로그램으로 Google 도메인 2,800개 이상의 네트워크 트래픽을 가로채 API 키 3,600개를 모았다.
Google의 Cloud Marketplace API로 각 키가 Google 프로젝트에 속하는지 검증했으며, iOS 앱 복호화와 소스맵(sourcemap) 누출에서 퍼스트파티 인증(FPA) v2 라이브러리인 `gapix`를 역공학해 인증 헤더 생성 방식을 파악했다.

디스커버리 문서는 Swagger/OpenAPI와 유사한 머신리더블 API 명세서다.
공개 문서는 YouTube Data API처럼 누구나 접근 가능하지만, 일부는 유효한 API 키가 있어야 열람된다.
연구팀은 `GOOGLE_INTERNAL` 같은 가시성 레이블(visibility label)을 사용하면 문서에 숨겨진 엔드포인트가 드러난다는 사실을 발견했다.
이를 통해 외부에 노출되지 않아야 할 내부 API 수백 개가 테스트 대상에 포함됐다.

AI 통합 측면에서 연구팀은 Claude와 MCP(Model Context Protocol)로 연결된 커스텀 도구 세 가지를 제작했다.
`probe_api`는 키 제한 헤더를 자동으로 처리하며 API 요청을 실행하고, `report_vulnerability`는 검증용 작업 ID를 첨부해 취약점을 제출하며, `get_endpoint_schema`는 요청·응답 스키마를 조회한다.
시스템 프롬프트는 오탐(false positive)을 줄이는 방향으로 반복 정제됐으며, 열거(enumeration) 자체는 취약점이 아니라 실제 데이터 노출만 보고 대상으로 명시적으로 구분했다.

주요 취약점으로는 접근 제어가 전혀 없는 Google Voice API를 통한 계정 탈취($2만), AdExchange 스테이징 환경이 프로덕션 데이터를 그대로 사용하면서 미인증 엔드포인트를 노출해 전 세계 광고주 계정에 관리자를 추가할 수 있었던 권한 상승($3만), 내부 개인정보 영향 평가 플랫폼 Eldar의 공개 접근($2만 6,674달러), Widevine DRM 포털의 조직·PGP 키·AES 서명 키 노출($1만 6,004달러), Translation Hub의 GCS 버킷 유출 가능 교차 테넌트 접근($3만 6,500달러), Vertex AI Search의 미인증 프롬프트 인젝션 공격 가능($3만) 등이 있다.
Google 취약점 보상 패널은 여러 보고서에 “탁월한 품질” 평가를 부여했다.

## 분석

### 공격 표면의 민주화: 머신리더블 명세서가 바꾼 보안 지형

전통적인 API 취약점 발굴은 대규모 수동 탐색이나 Burp Suite 트래픽 분석에 의존했다.
디스커버리 문서는 이 등식을 근본적으로 바꾼다.
API 명세서를 기계가 읽을 수 있는 형태로 제공하면 개발자 생산성이 높아지지만, 동시에 공격자에게 체계화된 공격 계획서를 건네는 셈이 된다.
OpenAPI 사양이 외부 공격자가 Swagger UI를 통해 엔드포인트를 탐색하도록 돕는 것처럼, Google의 디스커버리 문서는 인증 우회, 가시성 레이블 조작과 결합해 훨씬 더 넓은 내부 표면을 열어줬다.
이는 “문서화 = 노출”이라는 구조적 긴장을 보여준다.

이 구조가 AI와 결합하면 자동화된 공격 확장성이 생긴다.
연구팀이 1,500개 이상의 엔드포인트를 테스트할 수 있었던 것은 Claude가 스키마를 이해하고 적절한 요청을 생성하는 능력 덕분이다.
사람이 수행했다면 동일한 범위의 탐색에는 몇 배의 시간이 필요했을 것이다.

### 스테이징 환경이 프로덕션 위협이 되는 구조

이번 연구에서 반복적으로 등장하는 패턴은 스테이징 환경의 접근 제어 불일치다.
AdExchange 취약점은 스테이징 도메인이 프로덕션 데이터베이스를 가리키면서도 인증 검증을 생략했기에 성립했다.
Cloud Console GraphQL 취약점도 스테이징 환경에서 쿼리 서명 검증을 건너뛰면서 발생했다.
이는 단순한 구성 오류가 아니라, “스테이징은 실제 데이터를 쓰지 않는다”는 암묵적 가정이 무너진 결과다.

대형 조직에서 스테이징과 프로덕션 간 데이터 분리는 개발 편의와 보안 사이의 만성적 긴장 지점이다.
프로덕션 데이터를 스테이징에서 테스트하면 실제 조건과 동일해 버그 재현이 용이하지만, 그 순간 스테이징의 보안 수준이 프로덕션과 동일해야 한다는 요구가 발생한다.
Google처럼 수천 개의 팀이 독립적으로 서비스를 운영하는 환경에서 이 일관성을 유지하기란 구조적으로 어렵다.

### AI가 보안 연구의 레버리지 도구로 전환된 맥락

연구팀이 만든 MCP 통합은 AI를 “생성 도구”가 아니라 “레버리지 도구”로 사용하는 방식을 보여준다.
Claude는 스키마를 받아 적절한 테스트 요청을 생성하고, 응답을 분석해 이상 여부를 판단하며, 취약점 보고서를 자동으로 제출하는 파이프라인의 중심에 놓인다.
이 접근의 핵심은 AI가 판단하는 것이 아니라 인간 연구자의 판단 속도를 증폭한다는 점이다.

시스템 프롬프트의 반복 정제 과정—오탐 감소, 열거와 실제 노출의 구분—은 AI 보조 보안 도구를 실용화하는 데 있어서 프롬프트 엔지니어링이 핵심 기술임을 드러낸다.
같은 테스트 파이프라인이라도 프롬프트 설계에 따라 결과의 신호 대 잡음비가 크게 달라진다.

### 내부 API 노출의 구조적 원인

Eldar 플랫폼이나 Google Voice API가 공개 접근 가능했던 것은 단순한 실수가 아니다.
대형 조직에서 내부 API는 프론트엔드 레이어(Identity-Aware Proxy, IAP 등)가 보호한다는 전제하에 개발되는 경우가 많다.
그러나 API가 프론트엔드 없이 직접 접근 가능한 경로를 갖게 되면, 백엔드 자체의 인증·인가 로직이 없을 때 완전히 노출된다.
이는 “경계 보안(perimeter security)”의 고전적 한계로, 경계가 우회되면 내부는 무방비 상태가 된다.

## 비평

### 방법론의 재현 가능성은 의도적으로 제한된다

이 연구가 보안 커뮤니티에 제시하는 방법론은 실제로 일반적으로 재현하기 어렵다.
API 키 3,600개를 수집하기 위해 APK 6만 개를 처리하고, Chrome Debugger API로 도메인 2,800개의 트래픽을 가로채며, iOS 앱을 복호화하는 과정은 상당한 엔지니어링 투자를 전제한다.
연구 결과물이 “AI를 활용하면 이런 것이 가능하다”는 메시지를 전달하지만, 실제 진입 비용은 AI 도입 이전의 전문성과 인프라 구축에 있다.
AI는 마지막 마일을 자동화했을 뿐, 전제 조건은 전통적인 보안 연구의 그것과 크게 다르지 않다.

이 점에서 이 글은 AI의 역할을 약간 과장한다.
Claude가 기여한 것은 엔드포인트 테스트의 자동화와 속도 증폭이지, 취약점의 발견 자체가 아니다.
공격 표면 정의, 인증 메커니즘 역공학, 가시성 레이블 조작 아이디어는 모두 인간 연구자에게서 나왔다.
AI 보조 연구의 정확한 역할과 한계를 명시하는 것이 기술적 정직성을 높인다.

### 윤리적 책임 소재가 모호한 공개 방식

이 글은 취약점 수십 개의 기술적 세부 사항을 상당히 구체적으로 공개한다.
Gaia ID 열거와 Play Books API를 조합한 Nest 기기 소유자 신원 확인 방법이나, YouTube Content ID 자산 명명 규칙을 통한 비공개 영상 ID 유출 방법은 현재 Google이 패치했다고 가정하더라도 유사한 패턴이 다른 곳에 존재할 가능성을 열어둔다.
버그 바운티 보고와 패치 완료를 확인한 후 공개하는 것이 책임 있는 공개(responsible disclosure)의 표준이지만, 이 글은 각 취약점의 패치 완료 여부를 명시적으로 밝히지 않는다.

Widevine DRM 키 노출 취약점의 경우, 조직 열거·PGP 키·AES 서명 키 접근과 미인증 복호화 엔드포인트를 상세히 설명한다.
DRM 생태계의 특성상 이 정보가 실제로 어떤 파급을 가질 수 있는지에 대한 논의가 없는 것은 아쉽다.

### 성공 사례 편향이 방법론의 효율성을 가린다

이 연구는 취약점을 발견한 성공 스토리다.
그러나 1,500개 엔드포인트 중 취약점으로 이어진 것이 얼마나 되는지, 오탐률이 얼마였는지, 같은 파이프라인으로 테스트했지만 취약점이 없었던 API 군의 비율은 어땠는지에 대한 정보가 없다.
이는 방법론의 실제 효율성을 평가하기 어렵게 만든다.
50만 달러의 바운티를 받은 것은 결과가 좋았다는 증거이지, 방법론이 최적이었다는 증거가 아니다.

오탐을 줄이기 위해 시스템 프롬프트를 반복 정제했다는 언급은, 초기 파이프라인이 상당한 노이즈를 만들어냈음을 간접적으로 시사한다.
그 정제 과정에서 놓친 진짜 취약점은 없었는지도 알 수 없다.

## 인사이트

### AI 보조 보안 연구는 방어보다 공격이 먼저 이득을 본다

AI를 활용한 취약점 탐색의 자동화는 공격자(또는 연구자)가 방어자보다 훨씬 빠르게 생산성 향상을 얻는 구조를 만든다.
방어자는 모든 엔드포인트에서 취약점이 없어야 하지만, 공격자는 하나만 찾으면 된다.
AI가 1,500개 엔드포인트를 자동으로 탐색하는 파이프라인은 공격자에게 비대칭 이점을 증폭한다.

Google의 경우 버그 바운티 프로그램이 이 비대칭성의 일부를 흡수한다.
외부 연구자를 “계약된 공격자”로 활용해 내부 팀이 발견하지 못한 취약점을 찾게 하는 것이다.
그러나 이 메커니즘이 기능하려면 바운티 금액이 취약점을 악의적으로 사용할 때의 기대 수익보다 높아야 한다는 전제가 필요하다.
AdExchange 전 세계 광고주 계정 관리자 추가 취약점($3만)을 악의적으로 활용하면 얻을 수 있는 가치가 바운티보다 훨씬 크다는 점에서, 이 균형이 항상 성립하는지는 의문이다.

더 큰 문제는, AI 보조 공격 파이프라인을 구축하는 데 필요한 기술 문턱이 낮아지면 버그 바운티 프로그램이 없는 조직들이 더 큰 위험에 처한다는 것이다.
방어 측에서 동일한 AI 도구를 사용해 자체 API를 퍼징하는 “내부 레드팀” 역량 구축이 시급해지는 이유가 여기에 있다.

### 경계 보안에서 심층 방어로의 전환을 가속하는 압력

이번 연구가 반복적으로 드러낸 것은 “내부 API는 외부에서 접근 불가하다”는 가정에 기댄 보안 설계의 취약성이다.
Eldar 플랫폼은 프론트엔드 보호가 있었지만 API 자체에는 인증이 없었고, Google Voice API는 접근 제어가 전무했다.
이는 제로 트러스트(Zero Trust) 아키텍처가 단순한 유행어가 아니라 필수 설계 원칙이 되는 근거를 실증적으로 보여준다.

API 키 3,600개를 수집하는 연구팀의 방법론은 공격자가 얼마나 다양한 경로로 내부 API에 접근할 수 있는지를 보여준다.
Android APK 스크레이핑, 브라우저 트래픽 가로채기, iOS 앱 복호화라는 세 가지 경로 중 어느 하나를 막아도 나머지 두 경로가 존재한다.
단일 경계 보호가 얼마나 취약한지를 방증한다.

마이크로서비스 아키텍처에서 각 서비스가 자체 인증·인가를 구현해야 한다는 원칙은 개발 비용을 높인다.
그러나 이번 연구가 보여준 것처럼, 그 비용을 아끼면 훨씬 더 큰 보안 비용으로 돌아온다.
Google은 이번 연구 한 건으로만 50만 달러를 지불했으며, 각 취약점의 잠재적 피해 규모를 고려하면 이는 최소 비용에 불과하다.

### 머신리더블 명세서의 보안 설계가 새로운 과제로 부상한다

OpenAPI, GraphQL 인트로스펙션, Google 디스커버리 문서와 같은 머신리더블 API 명세는 개발자 경험을 대폭 개선하는 한편, 자동화된 공격의 진입 비용을 낮춘다.
연구팀이 가시성 레이블 `GOOGLE_INTERNAL`을 사용해 숨겨진 엔드포인트를 발견한 것처럼, 명세서 자체가 공격 표면 확대의 매개가 된다.

이 긴장은 단기적으로 해소되기 어렵다.
API 명세를 완전히 비공개로 하면 개발 생태계의 이점이 사라지고, 공개하면 공격 표면이 체계화된다.
중간 경로로 등장하는 것이 맥락 기반 접근 제어다.
누가 어떤 범위의 명세를 볼 수 있는지를 세밀하게 제어하면 노출 표면을 줄일 수 있지만, 그 자체가 복잡성과 유지 비용을 증가시킨다.

AI 시대에 더 중요한 함의는, API 명세서가 AI 에이전트의 공격 계획서로 직접 활용될 수 있다는 점이다.
인간 보안 연구자가 1,500개 엔드포인트의 명세를 읽고 테스트를 설계하는 데는 수개월이 걸리지만, AI는 이를 수 시간으로 단축한다.
API 명세의 보안 설계—어떤 정보를 포함하고 어떤 정보를 제외할지—가 API 설계 자체만큼 중요해지는 시점이 빠르게 다가오고 있다.
