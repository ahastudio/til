# WAF와 프레임워크 어댑터를 통한 React·Next.js 취약점 대응

원문: <https://developers.cloudflare.com/changelog/post/2026-05-06-react-nextjs-vulnerabilities/>

HN 토론: <https://news.ycombinator.com/item?id=46236924> (346점, 225개 댓글)

## 요약

2026년 5월 7일, Cloudflare는 React 팀과 Vercel이 공개한 다수의 취약점에 대한
보안 완화 조치를 발표했다. 이 취약점들은 React Server Components와 Next.js
애플리케이션에 영향을 미치며, DoS(서비스 거부), 미들웨어 우회, SSRF(서버 사이드 요청
위조), XSS(크로스 사이트 스크립팅), 캐시 포이즈닝 등 다양한 공격 벡터를 포함한다.

총 12개의 취약점이 공개됐으며, 심각도에 따라 고위험·중위험·저위험으로 분류된다.
고위험 취약점으로는 Server Components의 DoS, 미들웨어 우회, WebSocket 업그레이드를
통한 SSRF가 포함된다. 중위험은 XSS와 추가 DoS 벡터, 저위험은 캐시 포이즈닝 시나리오에
해당한다.

패치된 버전은 다음과 같다. React 패키지(`react-server-dom-webpack`,
`react-server-dom-parcel`, `react-server-dom-turbopack`)는 19.0.6, 19.1.7,
19.2.6 버전으로 업데이트해야 하며, Next.js는 15.5.16 또는 16.2.5로 업데이트해야
한다. Cloudflare는 즉각적인 의존성 업데이트를 강력히 권고하고 있다.

Cloudflare WAF(웹 애플리케이션 방화벽)는 DoS 취약점에 대한 두 가지 기존 규칙을
통해 CVE-2025-55184, CVE-2026-23864를 기본 차단 상태로 제공하며, 이 규칙들이
CVE-2026-23870에도 적용된다. 그러나 나머지 고위험 취약점 세 가지는 정상적인 애플리케이션
기능을 손상시키지 않으면서 WAF 수준에서 안전하게 막을 수 있는 방법을 아직 조사 중이라고
밝혔다.

프레임워크 어댑터 측에서는 Vinext(Vite 플러그인 기반 Next.js 재구현체)가 PPR resume
프로토콜 미포함, Pages Router 데이터 라우트 엔드포인트 부재, 내부 헤더 제거 등
아키텍처 차이를 통해 취약점 노출을 원천 회피했다. OpenNext on Cloudflare는 직접적인
React DoS CVE 영향을 받지 않으나, Next.js 버전 업데이트와 함께 강화된 어댑터 버전을
배포했다.

## 분석

### 방어 계층의 구조적 분리

이번 공개는 현대 웹 보안이 단일 계층 방어로는 충분하지 않다는 사실을 다시 확인시켜
준다. Cloudflare는 WAF를 통해 일부 DoS 벡터를 차단하면서도, 미들웨어 우회나 SSRF처럼
정상 트래픽 패턴과 구분이 어려운 취약점은 WAF 규칙만으로 안전하게 막기 어렵다고
인정했다. 이는 CDN·WAF 계층이 네트워크 수준 공격에는 효과적이지만, 애플리케이션
의미론(semantics)에 의존하는 취약점에는 한계가 있음을 보여준다.

### React Server Components의 공격 표면 확대

React Server Components(RSC) 아키텍처는 서버와 클라이언트의 경계를 프레임워크가
관리하게 만든다. 이 추상화는 개발 편의성을 높이지만, 동시에 서버 측에서 처리되는
데이터 스트림이 새로운 공격 표면이 된다. DoS 취약점이 Server Components에서 발생한
것은 직렬화·역직렬화 과정이나 PPR(Partial Prerendering) resume 프로토콜 같은
프레임워크 내부 메커니즘이 외부 입력에 의해 악용될 수 있음을 의미한다.

simonw는 HN에서 이 문제를 정확히 짚었다[^simonw]: “RSC는 JavaScript 코드를 보고
어떤 부분이 클라이언트에서 실행되고 어떤 부분이 서버에서 실행되는지 파악하기 어렵게
만든다.” 여기에 깊은 직렬화 RPC 메커니즘이 추가되면서 개발자에게는 투명성이 부족하고,
보안 측면에서도 취약점이 생긴다는 것이다. 그가 불편함을 느낀 바로 그 구조적 불투명성이
이번 취약점 묶음의 근원이기도 하다.

RSC와 HTMX를 대비한 관점도 주목할 만하다[^delifue]. delifue는 “RSC는 프론트엔드가
백엔드를 '먹으려는' 시도이고, HTMX는 백엔드가 프론트엔드를 '먹으려는' 시도”라고
표현했다. RSC는 서버·클라이언트 경계를 프레임워크가 통합 관리함으로써 백엔드 보안을
약화시키는 반면, HTMX는 경계를 명확히 유지하되 XSS 위험이 프론트엔드에 집중된다.
두 접근법은 보안 위험의 위치를 다르게 배분하는 설계 철학의 차이이기도 하다.

### 프레임워크 어댑터의 취약점 격리 전략

Vinext가 취약점을 “아키텍처 차이로 자연스럽게 회피했다”는 점은 흥미롭다. PPR resume
프로토콜을 포함하지 않거나 Pages Router를 지원하지 않는 등, 기능 범위를 의도적으로
좁힘으로써 공격 표면을 줄였다. 이는 호환성보다 보안을 우선한 설계 선택이며, 생태계
분화(fragmentation)의 예기치 않은 보안상 이점이기도 하다.

### CVE 번호 체계와 공개 타이밍

CVE-2025-55184는 2025년에 번호가 부여됐고, CVE-2026-23864·CVE-2026-23870은
2026년 번호다. 이 혼재는 취약점 발견·보고·공개 타이밍이 모두 다름을 보여준다. Cloudflare
WAF 규칙이 기존의 일반적인 공격 패턴 탐지 방식으로 CVE-2026-23870까지 커버할 수
있었다는 사실은, 패턴 기반 방어가 구체적인 CVE 식별 전에도 어느 정도 효과적임을 시사한다.

## 비평

### 강점

Cloudflare의 공시는 기술적 구체성과 실행 가능한 권고 사항을 함께 제공한다는 점에서
높이 평가할 수 있다. 어떤 WAF 규칙 ID가 어떤 CVE를 커버하는지 명시하고, 패치된 버전
번호를 정확히 기재했으며, 완화할 수 없는 취약점에 대해서는 솔직하게 한계를 인정했다.
이는 많은 보안 공시가 모호한 권고만 남기는 것과 대비된다.

### 한계와 빠진 관점

그러나 이 문서는 Cloudflare 자사 제품의 보호 범위를 홍보하는 성격이 강하다. 12개의
취약점 목록은 요약 형태로만 제시됐고, 각 CVE의 기술적 세부 사항이나 익스플로잇(exploit)
조건이 생략됐다. “현재 조사 중”이라는 표현은 세 가지 고위험 취약점에 대한 WAF 수준
완화책이 없다는 사실을 희석시킨다.

또한 “즉각 업데이트”라는 권고가 대규모 프로덕션 환경에서 얼마나 현실적인지에 대한
논의도 없다. 대기업 환경에서 Next.js 메이저 버전을 빠르게 올리는 것은 광범위한 회귀
테스트를 수반하며, 그 기간 동안의 임시 완화 전략에 대한 조언이 부족하다.

WatchDog의 관찰은 이 한계를 더 날카롭게 드러낸다[^WatchDog]: RSC는 복잡한데도
관련 문서가 현저히 부족하고, Next.js는 이를 이미 광범위하게 채택했으며, 내부 빌드
시스템과 벤더드(vendored) 의존성 때문에 디버깅조차 어렵다는 것이다. 그는 앞으로도
보안 취약점이 추가로 발견될 것이라고 예측했다. “즉각 업데이트”가 어려운 이유 중
하나는 업데이트 자체가 복잡한 빌드 시스템을 통과해야 하기 때문이기도 하다.

yread의 질문도 짚고 넘어갈 필요가 있다[^yread]: “지난주 패치가 나오기 전까지 RSC를
충분히 많은 사람이 검토하지 않은 것 아닌가?” RSC가 Next.js를 통해 빠르게 확산됐음에도
불구하고, 그 보안 분석이 충분히 이루어지지 않은 채 프로덕션에 사용됐다는 지적이다.
이는 “오픈소스라 많은 눈이 검토한다(Linus's Law)”는 가정이 복잡한 프레임워크
내부에서는 언제나 작동하지 않는다는 사실을 상기시킨다.

### OpenNext와 Vinext 언급의 맥락

OpenNext와 Vinext가 언급된 방식은 이들이 Cloudflare 생태계와 밀접하게 연관된
프로젝트임을 고려할 때, 독립적인 평가라기보다는 파트너 프로젝트 홍보에 가깝다.
Cloudflare Workers 위에서 Next.js를 실행하는 대안적 스택이 보안 면에서 유리하다는
메시지를 내포하고 있어, 일정 수준의 편향을 감안하고 읽어야 한다.

## 인사이트

### 추상화 계층이 공격 표면이 된다

React Server Components는 개발자가 서버·클라이언트 경계를 명시적으로 관리할 필요를
없애 주는 강력한 추상화다. 그런데 이 취약점 묶음은 그 편의성이 보안 비용을 수반함을
드러낸다. 프레임워크가 내부적으로 처리하는 직렬화, 스트리밍, 라우팅 로직이 외부 입력에
노출되는 순간, 그것은 공격 표면이 된다.

역사적으로 비슷한 패턴이 반복된다. Java의 직렬화 취약점(Apache Commons Collections,
Log4Shell의 JNDI 조회)이나 PHP의 `unserialize()` 계열 취약점이 모두 “프레임워크가
대신 처리해 주는 것”에서 비롯됐다. RSC의 PPR resume 프로토콜이 DoS의 원인이 됐다는
사실은, 성능 최적화를 위한 서버 사이드 상태 복원 메커니즘이 악의적인 페이로드에 취약하다는
것을 의미한다.

이 패턴에서 얻을 수 있는 원칙은 단순하다: 프레임워크가 “마법처럼” 처리하는 영역일수록,
그 내부 동작을 이해하지 못한 채 사용하면 취약점 발생 시 대응이 늦어진다. 개발자는
자신이 사용하는 추상화의 한 층 아래를 이해해야 한다.

kpozin의 한 줄은 이 모순을 가장 날카롭게 요약한다[^kpozin]: “클라이언트와 서버에서
실행되는 코드의 경계를 흐리도록 설계된 프레임워크가, 클라이언트와 서버에서 실행되는
코드의 구분을 잊어버렸다.” 아키텍처적 목표(경계 통합)와 보안 요구사항(경계 구분)이
정면으로 충돌하는 지점이다.

역사적 선례도 있다. hbbio는 15년 전 Opa라는 언어를 소개했는데[^hbbio], OCaml
기반의 이 언어는 RSC보다 먼저 자동 클라이언트·서버 코드 분리를 구현하고 JSX 유사
문법도 갖췄었다. 그러나 개발진은 결국 자동 코드 분리보다 더 명확한 관심사 분리가
낫다는 교훈을 얻었다. 10년 넘게 같은 아이디어가 반복 시도되고, 매번 비슷한 구조적
위험에 직면한다는 사실은, 이 접근법의 근본적인 한계가 기술적 세부구현의 문제가 아닐
수 있음을 시사한다.

### WAF의 의미론적 한계는 애플리케이션 보안의 책임 경계를 재정의한다

Cloudflare가 세 가지 고위험 취약점을 WAF로 막을 수 없다고 인정한 것은 작은 발언이
아니다. 많은 조직이 CDN·WAF를 보안 완충재로 삼아 애플리케이션 패치를 미루는 전략을
취한다. 이번 공시는 그 전략의 한계를 공개적으로 확인시켜 준다.

미들웨어 우회나 SSRF는 HTTP 헤더·URL 패턴 수준에서 정상 요청과 구분하기 어렵다.
웹소켓 업그레이드를 통한 SSRF는 특히 그렇다. WAF 규칙으로 이를 막으려면 모든 웹소켓
업그레이드 요청을 차단해야 할 수도 있는데, 그것은 정상 기능을 파괴한다. 결국 애플리케이션
의미론(semantics)을 이해하는 방어는 애플리케이션 계층에서만 가능하다.

이는 “WAF가 있으니 괜찮다”는 인식이 얼마나 위험한지를 보여준다. 보안 책임은 CDN
벤더에게 위임할 수 없으며, 취약한 의존성을 최신 상태로 유지하는 것은 여전히 개발 팀의
핵심 책임이다. 특히 오픈소스 프레임워크의 보안 업데이트를 신속하게 반영하는 내부 프로세스—
의존성 자동 업데이트, 취약점 알림 구독, 패치 주기 단축—가 없는 조직은 구조적 위험에
노출돼 있다.

### 생태계 파편화가 보안에 미치는 양면성

Vinext가 Next.js의 일부 기능을 구현하지 않음으로써 취약점을 “자연스럽게” 피했다는
사실은 아이러니하다. 일반적으로 생태계 파편화는 호환성·유지보수 비용 측면에서 부정적으로
평가된다. 그러나 공격 표면의 관점에서는, 기능 범위를 좁힌 구현체가 더 안전할 수 있다.

이것은 “적을수록 많다(less is more)”의 보안 버전이다. 사용하지 않는 기능은 패치할
필요가 없고, 존재하지 않는 코드 경로는 취약점이 될 수 없다. OpenBSD의 “안전한 기본값과
최소한의 코드”라는 철학이 같은 맥락에 있다.

그러나 이 관점을 과도하게 일반화하면 위험하다. Vinext 같은 재구현체는 원본 프레임워크의
버그 수정과 보안 패치를 자동으로 받지 못한다. 자체적인 취약점 대응 능력과 커뮤니티 규모가
충분하지 않으면, 지금은 유리해 보이는 “작은 코드베이스”가 나중에는 “방치된 의존성”이
될 수 있다. 선택의 기준은 단순한 공격 표면 크기가 아니라, 보안 유지 능력의 지속 가능성이어야
한다.

### 12개 취약점 동시 공개가 시사하는 보안 연구의 집중 패턴

한 번에 12개의 취약점이 공개됐다는 것은 단순한 개별 버그 발견이 아니다. 이는 React
Server Components 아키텍처 전반을 표적으로 한 집중적인 보안 감사(security audit)가
있었음을 시사한다. CVE 번호가 2025년과 2026년에 걸쳐 있다는 점을 보면, 취약점 연구가
최소 수개월에 걸쳐 진행됐고 조율된 공개(coordinated disclosure) 과정을 거쳤을 가능성이
높다.

이런 패턴은 신기술이 주류로 채택될 때 보안 연구자들의 관심이 집중된다는 점을 보여준다.
React Server Components는 2023년부터 Next.js를 통해 빠르게 보급됐고, 이제 그
아키텍처에 대한 심층 보안 분석이 나오고 있다. Log4Shell도 Log4j가 광범위하게 사용된
지 몇 년 후에 발견됐다. 기술의 보급 곡선이 정점을 넘을 무렵, 그것을 겨냥한 보안 연구가
급증한다.

이것이 의미하는 바는 명확하다: 새로운 프레임워크나 패러다임을 조기에 채택할수록 초기에는
보안 연구가 덜 집중돼 있지만, 채택이 확산될수록 공격 연구도 함께 성숙한다. “새로운
기술이라 아직 공격이 적다”는 인식은 착각이다. 보안 부채는 채택 확산과 함께 이자가 붙는다.
