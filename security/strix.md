# Strix

<https://www.strix.ai/>

<https://github.com/usestrix/strix>

## 요약

Strix는 AI 에이전트가 실제 해커처럼 애플리케이션의 취약점을 찾고,
PoC(Proof of Concept)로 검증한 뒤, 수정 PR까지 자동 생성하는
오픈소스 자율 보안 테스트 플랫폼이다.

Graph of Agents 아키텍처를 채택하여
특화된 여러 에이전트가 협력·병렬 실행하며 보안 테스트를 수행한다.
코드 분석, HTTP 프록시, 브라우저 자동화, 터미널, Python 런타임 등
풀 해커 툴킷을 갖춘 에이전트 시스템이다.

IDOR, SQL injection, XSS, SSRF, 인증 우회, 비즈니스 로직 결함 등
광범위한 공격 벡터를 다루며,
발견된 취약점마다 실제 공격 증명(PoC)을 생성해
거짓양성(false positive) 없이 재현 가능한 결과만 보고한다.

Python 3.12 기반이며 Docker 샌드박스에서 실행된다.
LiteLLM을 통해 OpenAI, Anthropic, Google, Azure 등 다양한 LLM을 지원한다.
Apache 2.0 라이선스로 공개되어 있고,
Alpha 단계(v0.8.3)임에도 GitHub 스타 약 23,000개를 기록했다.

## 분석

### 거짓양성 제로라는 가치 제안

Strix의 핵심 가치 제안은 "거짓양성 제로"다.
기존 정적 분석 도구(SAST)는 코드 패턴만 보고 경고를 던지므로
오탐률이 높아 보안팀이 분류(triage)에 과도한 시간을 쓴다.
Strix는 실제로 공격을 실행하고 성공 여부를 확인하는 방식으로
이 문제를 구조적으로 우회한다.

### 탐지에서 수정까지 한 루프

자동 수정 PR 생성은 기존 보안 도구와 가장 큰 차별점이다.
전통적 보안 도구는 "여기에 문제가 있다"까지만 알려주고
수정은 개발자의 몫이었다.
Strix는 탐지부터 수정까지 한 루프 안에서 처리하며,
개발자의 기존 워크플로(PR → 리뷰 → 머지)에 자연스럽게 끼워넣는
shift-left 접근을 택했다.

### PentAGI와의 포지셔닝 차이

같은 카테고리인 PentAGI와 비교하면 접근이 다르다.
PentAGI는 네트워크/인프라 침투 테스트를 Planner-Executor 루프로
자동화하는 방향이고,
Strix는 애플리케이션 취약점 탐지와 자동 수정에 집중한다.
PentAGI가 전통적 펜테스트 절차의 자동화라면,
Strix는 개발 파이프라인 안에서 보안을 해결하려는 도구다.

## 비평

### 거짓양성 제로의 이면: 미탐 리스크

PoC 기반 검증이라는 접근 자체는 강력하지만,
"거짓양성 제로"라는 주장은 마케팅 표현에 가깝다.
PoC가 성공하지 못한 취약점은 보고하지 않으므로
거짓양성이 아니라 미탐(false negative)이 실제 리스크다.
공격 PoC를 만들 수 없지만 실제로 위험한 취약점이
빠질 가능성에 대한 언급이 없다.

### Alpha 단계 도구의 신뢰 문제

Alpha 단계라는 점도 간과하기 어렵다.
보안 도구는 "작동하지 않을 때"의 결과가 치명적이다.
스캔 결과가 깨끗하게 나왔을 때 그것이
"정말 안전한 것"인지 "도구가 미성숙해서 못 찾은 것"인지
구분할 수 없다면 도구의 실무 가치가 제한된다.

### LLM 의존에 따른 비용과 재현성

LLM 의존도가 높다는 것은 비용과 재현성 모두에 영향을 준다.
동일한 코드베이스를 두 번 스캔했을 때 같은 결과가 나오는지,
모델 버전이 바뀌었을 때 탐지 성능이 유지되는지에 대한
벤치마크나 보장이 공개되지 않았다.

### CI 환경에서의 Docker 마찰

Docker 필수 요구사항은 CI 환경에서 마찰을 일으킨다.
GitHub Actions의 기본 러너에서 Docker-in-Docker 구성이 필요하고,
보안 정책이 엄격한 조직에서는 권한 문제가 발생할 수 있다.

## 인사이트

### "탐지 → 수정" 루프의 완결이 보안 도구의 다음 기준선이 된다

전통적 보안 도구는 "발견"과 "수정"이 분리된 두 단계였다.
Snyk, SonarQube 같은 도구도 수정 제안을 하지만
실제 PR을 만들어 머지 가능한 상태로 넘기지는 않았다.
Strix가 수정 PR 자동 생성을 전면에 내세운 것은
코딩 에이전트(Cursor, Claude Code 등)가 보여준
"AI가 코드를 직접 쓰는" 패러다임을 보안 영역에 적용한 것이다.

이 접근이 성공하면 보안 도구의 평가 기준 자체가 바뀐다.
"얼마나 많은 취약점을 찾았는가"가 아니라
"얼마나 많은 취약점을 자동으로 고쳤는가"가 핵심 지표가 된다.
보안팀의 역할도 "취약점을 찾아 개발팀에 전달하는 사람"에서
"자동 수정의 품질을 검증하는 리뷰어"로 이동할 수 있다.

### Graph of Agents는 보안 테스트에 구조적으로 적합하다

단일 에이전트 루프(Plan → Act → Observe)는
침투 테스트의 탐색적 성격과 맞지 않는 지점이 있다.
실제 침투 테스터는 정찰, 인증 테스트, 인젝션 테스트 등을
병렬로 진행하면서 한 경로의 발견이 다른 경로의 전략을 바꾼다.

Graph of Agents 구조는 이 패턴을 자연스럽게 모델링한다.
각 에이전트가 특정 공격 벡터에 특화되면서도
그래프를 통해 정보를 공유하고 전략을 조율할 수 있다.
PentAGI의 단일 Planner-Executor 루프와 비교했을 때
병렬성과 전문성 분리에서 이점이 있다.

다만 이 구조의 약점은 복잡성이다.
에이전트 간 통신 오버헤드, 중복 작업, 상충하는 판단 등
멀티 에이전트 시스템의 고전적 문제가 보안이라는
고위험 도메인에서 어떻게 관리되는지가 성패를 가른다.

### PoC 검증은 "신뢰할 수 있는 보안 테스트"의 핵심 요건이 된다

보안 도구 시장에서 가장 큰 불만은 거짓양성이다.
경고 100개 중 실제 위험한 것이 3개라면
나머지 97개를 검토하는 비용이 도구의 가치를 상쇄한다.
Strix의 PoC 기반 접근은 이 문제에 정면으로 답한다.

이 방식이 업계 표준으로 자리 잡으면
"스캔 결과를 보고서로 제출"하는 기존 컴플라이언스 관행도 바뀐다.
감사인이 "취약점 목록"이 아니라 "재현 가능한 PoC 세트"를
요구하는 시대가 올 수 있다.
보안 검증의 증거 기준(evidence bar)이 높아지는 것이다.

동시에 이는 양날의 검이다.
PoC를 생성한다는 것은 실제 공격 코드를 만든다는 뜻이므로,
이 코드의 보관, 접근 제어, 유출 방지가
새로운 보안 관리 대상이 된다.
"보안 도구가 만든 공격 코드"의 거버넌스는
아직 업계에서 합의된 프레임워크가 없는 영역이다.
