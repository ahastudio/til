# AI 바이브 코딩 호러 스토리 - 환자 관리 앱의 재앙적 보안 실패

원문: <https://www.tobru.ch/an-ai-vibe-coding-horror-story/>

## 요약

스위스의 한 의료 전문가가 AI 코딩 에이전트를 활용해 환자 관리 시스템을 직접
구축하고 인터넷에 공개했다.
기존 환자 기록까지 임포트한 이 애플리케이션은 30분 만에 전체 환자 데이터에
대한 읽기·쓰기 접근이 가능한 상태로 드러났다.
모든 데이터가 암호화 없이 공개되어 있었고, 인증이나 행 수준 보안(Row-Level
Security) 같은 기본적인 보안 제어가 전혀 없었다.
진료 녹음이 환자 동의 없이 미국 소재 AI 서비스로 전송되어 전사(Transcription)
처리되고 있었다.

## 분석

### 기술적 결함의 구조

애플리케이션은 인라인 JavaScript와 CSS를 포함한 단일 HTML 파일로 구성되었다.
보안 로직이 전부 클라이언트 사이드 JavaScript에 존재했으며, 백엔드 데이터베이스
에는 인증도, 접근 제한도 없었다.
이는 `curl` 명령어 한 줄이면 모든 데이터를 가져올 수 있다는 뜻이다.
AI 코딩 도구가 생성한 코드가 기능적으로는 동작했지만 보안 아키텍처라는 개념
자체가 부재했음을 보여준다.

### 법적·규제적 위반

환자 데이터가 데이터 처리 계약(Data Processing Agreement) 없이 미국 서버에
저장되었다.
이는 스위스 데이터 보호법(nDSG)과 직업상 기밀 유지 의무를 위반할 소지가 크다.
의료 데이터는 대부분의 법체계에서 가장 엄격한 보호 등급에 해당하며, 이를
위반할 경우 형사 처벌까지 가능하다.

### AI 도구의 보안 맹점

AI 코딩 에이전트는 사용자가 요청한 기능을 구현하는 데 집중한다.
"환자 관리 시스템을 만들어줘"라는 프롬프트에 보안 요구사항이 명시되지 않으면
인증, 암호화, 접근 제어 같은 비기능적 요구사항은 누락되기 쉽다.
도구는 코드가 동작하는지 여부만 판단할 뿐, 코드가 안전한지 여부를 판단하는
메커니즘이 기본적으로 내장되어 있지 않다.

## 비평

### 개인의 무지가 아닌 구조적 문제

이 사건을 단순히 "비전문가의 실수"로 치부하기 쉽지만, 근본 원인은 더 깊다.
AI 코딩 도구의 마케팅은 "누구나 개발자가 될 수 있다"를 전면에 내세운다.
그러나 보안, 규정 준수, 아키텍처에 대한 이해 없이 코드를 생성하는 것은
자동차 면허 없이 도로에 나가는 것과 다르지 않다.
도구 제공자와 교육 콘텐츠 제작자 모두 이 격차에 대한 책임이 있다.

### 의료 분야 특수성

GeekNews 댓글에서 지적된 것처럼, 한국 의료법도 환자 기록의 무단 접근을
엄격히 금지한다.
대형 병원은 네트워크 분리(Network Segmentation)로 이를 방지하지만, 인터넷에
직접 연결되는 소규모 의원은 취약점이 어디에 있는지조차 파악되지 않는다.
이 사건은 스위스에서 발생했지만 동일한 위험은 전 세계 1차 의료기관에 존재한다.

### 보안 감사 부재의 위험

저자가 30분 만에 전체 접근 권한을 획득했다는 사실은, 전문적인 공격자라면 훨씬
빠르게 악용할 수 있음을 의미한다.
문제는 이런 애플리케이션이 별도의 보안 감사나 침투 테스트 없이 배포된다는
점이다.
전통적인 소프트웨어 개발에서는 코드 리뷰와 QA 프로세스가 이런 결함을 걸러
내지만, 바이브 코딩에는 그런 관문이 존재하지 않는다.

## 인사이트

### 민주화의 역설: 접근성이 높아질수록 위험도 높아진다

AI 코딩 도구는 소프트웨어 개발의 진입 장벽을 극적으로 낮추었다.
이는 긍정적인 면이 분명히 있지만, 동시에 보안·규정 준수·아키텍처에 대한
이해 없이 프로덕션 시스템을 배포하는 사람이 늘어난다는 뜻이기도 하다.
역사적으로 비슷한 패턴이 있다.
1990년대 웹의 대중화는 SQL 인젝션과 XSS 같은 취약점의 대량 발생으로 이어졌고,
업계가 OWASP 같은 보안 표준을 정립하기까지 수년이 걸렸다.

바이브 코딩은 이 사이클을 더 빠르고 더 넓은 범위에서 반복하고 있다.
차이점은 이번에는 코드를 작성하는 사람이 개발자가 아니라는 것이다.
개발자에게는 보안 교육을 제공할 수 있지만, 의사나 변호사에게 보안 아키텍처를
가르치는 것은 현실적이지 않다.
이는 도구 자체에 보안 가드레일이 내장되어야 한다는 요구로 이어질 수밖에 없다.

### 보이지 않는 것의 위험: 비기능적 요구사항의 소멸

소프트웨어 공학에서 보안, 성능, 확장성 같은 비기능적 요구사항(Non-Functional
Requirements)은 항상 간과되기 쉬운 영역이었다.
기능이 동작하면 완성된 것처럼 보이기 때문이다.
AI 코딩 도구는 이 문제를 극단적으로 증폭시킨다.
프롬프트 기반 개발에서는 사용자가 명시하지 않은 요구사항은 존재하지 않는 것과
같다.

GeekNews 댓글 중 woung717의 "비전문가는 그런 것이 존재하는지, 왜 필요한지조차
모른다"는 지적이 핵심을 찌른다.
이는 던닝-크루거 효과(Dunning-Kruger Effect)의 전형적인 발현이다.
자신이 무엇을 모르는지 모르는 상태에서 AI가 생성한 결과물이 완전해 보이면,
부족한 부분을 인식할 방법 자체가 없다.
전통적 개발에서는 팀의 시니어 개발자나 보안 엔지니어가 이 역할을 했지만,
1인 바이브 코딩에는 그런 안전망이 없다.

### 규제 공백과 제도적 대응의 시급성

현행 데이터 보호법은 전문 개발 조직이 소프트웨어를 구축한다는 전제 위에
설계되어 있다.
nDSG든 GDPR이든 한국 개인정보보호법이든, 규제는 "데이터 처리자"와 "데이터
컨트롤러"를 구분하고 각각에 의무를 부과한다.
그러나 의사가 AI로 직접 만든 앱에서는 이 구분이 무너진다.
개발자, 운영자, 데이터 처리자, 의료 서비스 제공자가 모두 한 사람이다.

이는 기존 규제 프레임워크가 예상하지 못한 상황이다.
GitHub에서 노출된 OpenAI API 키가 "수없이 많다"는 GeekNews 댓글은 이 문제가
개별 사건이 아니라 구조적 현상임을 보여준다.
유럽 전역에서 유사한 사례가 보고되고 있다는 Hacker News 논의도 같은 맥락이다.
AI 코딩 도구 사용에 대한 업종별 가이드라인, 최소 보안 기준, 그리고 민감
데이터를 다루는 애플리케이션에 대한 배포 전 검증 의무화가 시급하다.
