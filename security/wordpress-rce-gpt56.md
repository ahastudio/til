# GPT-5.6으로 25달러에 찾은 WordPress RCE

원문: [Exploit brokers pay $500,000 for a WordPress RCE. I found one with GPT5.6 Sol Ultra and $25 › Searchlight Cyber](https://slcyber.io/research-center/exploit-brokers-pay-500000-for-a-wordpress-rce-i-found-one-with-gpt5-6/)

HN 토론: <https://news.ycombinator.com/item?id=48975665> (402점, 226개 댓글)

## 요약

보안 연구자 Adam Kues가 LLM 에이전트를 이용해 WordPress에서 원격 코드 실행(RCE)
체인을 발견한 과정을 공개했다.
익스플로잇 브로커 시장에서 50만 달러에 거래되는 수준의 취약점을
약 25달러의 API 비용으로 찾았다.

취약점 체인은 여러 단계로 구성된다.
WordPress 5.6에서 도입된 배치(batch) API가 유효성 검사와 실행 단계에서
서로 다른 배열 상태를 사용한다는 점이 출발점이다.
이 불일치를 이용하면 파라미터 검증을 우회할 수 있다.

두 번째 단계는 SQL 인젝션이다.
`author__not_in` 파라미터에 배열 대신 문자열을 넘기면 이스케이프 없이
쿼리에 직접 삽입된다.
재귀적 배치 API 호출로 GET 요청 제한도 우회할 수 있다.

세 번째 단계는 WordPress의 포스트 캐싱 메커니즘 악용이다.
UNION 기반 SQL 인젝션으로 조작된 가짜 포스트를 메모리 캐시에 올리고,
`customize_changeset` 포스트 타입을 통해 관리자 권한으로 사이트 설정을 변경한다.
마지막으로 포스트 상태와 타입 조합을 조작해 `parse_request` 훅을 호출함으로써
새로운 관리자 계정을 생성하는 데까지 이른다.

연구자는 OpenAI의 순환 이중 커버(Cycle Double Cover) 추측 해결 방법론을 바탕으로
프롬프트를 설계했다.
최대 4개의 동시 에이전트가 서로 다른 취약점 클래스를 탐색하도록 지시했고,
6시간 이상 실행했다.
약 500만 개의 WordPress 인스턴스가 영향을 받는다.

## 분석

### AI가 취약점 발견에서 하는 일이 달라지고 있다

전통적인 보안 연구에서 AI는 코드 검색 속도를 높이거나 패턴 매칭을 도와주는 보조 도구였다.
이 연구는 그 역할이 바뀌었음을 보여준다.
에이전트가 다단계 익스플로잇 체인을 스스로 구성했다는 점이 핵심이다.

각 단계의 취약점 — 배치 API 불일치, SQL 인젝션 지점, 캐시 조작, 훅 남용 — 은
개별적으로는 찾기 어렵지 않을 수도 있다.
그러나 이 단계들을 하나의 실행 가능한 체인으로 연결하는 작업이
수동 연구에서 가장 시간이 많이 드는 부분이다.
연구자가 “보안 연구자 혼자서는 10시간 안에 이 체인을 찾아 완성할 수 없었을 것”이라고
밝힌 것은 이 점을 가리킨다.

에이전트의 기여는 각 취약점을 발견한 것이 아니라,
광범위한 코드베이스에서 연결 가능한 취약점들을 병렬로 탐색해 체인을 완성한 것이다.
이것은 AI가 보안 연구에서 할 수 있는 일의 질적 변화를 나타낸다.

### 비용 비대칭이 위협 지형을 바꾼다

익스플로잇 브로커 시장에서 WordPress RCE는 50만 달러에 거래된다.
그 취약점을 25달러의 API 비용으로 발견했다는 것은 단순한 홍보 문구가 아니다.
이것은 공격 비용과 방어 비용 사이의 비대칭이 극단적으로 벌어지고 있음을 보여준다.

방어 측은 WordPress 코드베이스 전체를 지속적으로 감사해야 한다.
코어, 플러그인, 테마, API 레이어를 모두 포함하면 방어 비용은 선형적으로 증가한다.
반면 공격 측은 한 번만 성공하면 되고, 이제 그 탐색 비용이 급격히 낮아졌다.

오픈소스 프로젝트와 그 생태계는 이 비대칭을 구조적으로 받아들여야 하는 상황에 처해있다.
인기 있는 오픈소스 프로젝트의 취약점 발견 속도가 빨라진다는 것은
보안 패치 출시와 사용자 업데이트 사이의 시간이 더 중요해진다는 의미이기도 하다.

### 연구 방법론 자체가 재현 가능한 패턴이 됐다

이 연구에서 주목할 또 다른 점은 연구 방법론이 공개됐다는 것이다.
OpenAI의 수학 추측 해결 방법론을 보안 연구에 적용한 프롬프트 구조,
동시 에이전트로 서로 다른 취약점 클래스를 병렬 탐색하는 방식은
재현 가능한 패턴이다.

이 패턴은 다른 대형 오픈소스 프로젝트에도 그대로 적용될 수 있다.
WordPress만이 아니라 Drupal, Joomla, 주요 PHP 프레임워크,
그리고 충분히 큰 코드베이스를 가진 모든 프로젝트가 같은 방식의 탐색 대상이 될 수 있다.
방법론의 공개는 지식 공유이기도 하지만, 동시에 공격 역량의 민주화이기도 하다.

## 비평

### 비용 비교는 오해를 유발하도록 구성됐다

“50만 달러짜리 취약점을 25달러로 발견했다”는 표현은 강렬하지만,
두 숫자의 의미가 다르다.
50만 달러는 익스플로잇 브로커 시장에서 완성된 익스플로잇의 거래 가격이고,
25달러는 연구자의 API 비용이다.

인건비, 연구자의 도메인 지식, 검증 시간, 책임 있는 공개 절차의 비용은 포함되지 않았다.
무엇보다 25달러로 찾을 수 있다는 것은 공격자도 같은 비용으로 탐색할 수 있다는 의미이지,
방어자가 같은 비용으로 모든 취약점을 탐지할 수 있다는 의미가 아니다.
숫자의 대비가 주는 직관적 충격이 실제 함의보다 과장된 방향으로 읽힌다.

HN에서도 같은 비판이 나왔다.
"이 글에서 마케팅 글쓰기가 끝이 없다. 맞아, 25달러로 찾았겠지. 어디를 봐야 하는지,
어떻게 탐색해야 하는지에 관한 수년간의 업계 도메인 지식도 함께 들어간 25달러로."[^hmokiguess]
더 나아가, 이 취약점을 실제로 50만 달러에 사려는 브로커가 있다는 증거도 없다는 지적도 있었다.[^Zsfe510asG]
저자가 AI 자동 스캔 제품을 판매하는 회사에 재직 중이라는 이해충돌도 함께 거론됐다.

### 2026년에 문자열 연결 SQL 인젝션이 있다는 사실 자체가 문제다

취약점 체인의 기술적 핵심 중 하나는 `author__not_in` 파라미터의 SQL 인젝션이다.
배열 대신 문자열을 넘기면 이스케이프 없이 쿼리에 직접 삽입된다.
HN의 progbits는 이를 짧게 짚었다.
"2026년에 문자열 연결 SQL 인젝션."[^progbits]
실제 WordPress GitHub 커밋도 링크했다.

AI 에이전트가 이 취약점을 발견했다는 사실보다, 이 종류의 취약점이 2026년에도 존재했다는 사실이 더 충격적일 수 있다.
코드 리뷰와 SAST 도구가 이 취약점을 왜 잡지 못했는지는 글에서 다루지 않는다.[^lillesvin]
에이전트가 취약점을 찾았다는 것은 에이전트의 능력 증거이기도 하지만,
WordPress 코어의 보안 검토 프로세스에 구멍이 있다는 증거이기도 하다.

또한 ameliaquining은 기술적으로 중요한 질문을 제기했다.
SQL 인젝션이 있다면 왜 바로 사용자 테이블에 관리자 계정을 삽입하지 않고
복잡한 캐시 조작과 훅 남용 체인이 필요한가?[^ameliaquining]
이 질문에 대한 답은 글에 명시되지 않았고,
WordPress의 내부 구조상 직접 삽입이 어떤 이유로 막혔는지가 체인의 복잡성을 이해하는 데 핵심이다.

### 에이전트의 역할이 과장됐을 가능성이 있다

연구자는 에이전트가 체인을 완성했다고 설명하지만,
에이전트가 실제로 어디까지 독립적으로 추론했고
어디서부터 연구자의 가이드가 개입했는지는 명확하지 않다.

“6시간 이상 탐색”이라는 묘사는 에이전트가 자율적으로 탐색했음을 암시하지만,
초기 프롬프트 설계와 탐색 방향 결정에 연구자의 도메인 지식이 얼마나 깊이 반영됐는지가
에이전트 기여도를 평가하는 데 결정적이다.
현재 공개된 내용만으로는 에이전트가 발견한 것인지,
연구자가 에이전트를 이용해 확인한 것인지 구분하기 어렵다.

### 책임 있는 공개 이후의 생태계 반응이 빠진다

이 글은 취약점 발견과 공개에 초점을 맞추지만,
실제 WordPress 생태계에서 패치가 어떻게 배포될 것인지는 다루지 않는다.
WordPress 코어 업데이트는 관리형 호스팅에서 자동으로 이뤄지지만,
플러그인과 테마 업데이트는 사용자의 수동 행동에 의존하는 경우가 많다.

취약점을 빠르게 발견할 수 있게 됐다는 사실은
패치 배포와 적용 속도가 따라가지 못할 때만 문제가 된다.
에이전트 기반 취약점 발견 속도가 빨라진다면,
생태계 수준의 패치 배포 속도도 같이 빨라져야 한다.
이 부분에 대한 논의가 없으면 발견의 의미가 절반만 전달된다.

## 인사이트

### 보안 감사는 LLM 에이전트를 기본 도구로 포함해야 하는 시점이 됐다

이 연구 이전에도 LLM을 코드 감사에 활용하는 사례는 있었다.
그러나 대부분은 단일 파일이나 함수 수준의 리뷰 지원이었다.
이 연구가 보여준 것은 에이전트가 코드베이스 전체를 가로지르는
다단계 취약점 체인을 탐색할 수 있다는 것이다.

이것은 보안 감사의 설계를 바꿔야 함을 의미한다.
기존 SAST 도구와 수동 코드 리뷰는 단일 지점의 취약점 탐지에 최적화돼 있다.
에이전트 기반 탐색은 서로 다른 레이어에 흩어진 취약점들이 연결될 수 있는지를 탐색한다.
두 접근 방식은 탐지하는 취약점의 종류가 다르므로 보완적으로 사용해야 한다.

실무적으로는 핵심 의존성과 자체 플러그인을 대상으로
LLM 에이전트 기반 감사와 기존 SAST, 퍼징, 수동 리뷰를 조합한
반복 검증 워크플로를 도입할 필요가 있다.
이것은 선택 사항이 아니라 공격자가 같은 도구를 쓰기 시작한 이상의 필수적인 대응이다.

### 오픈소스 생태계의 취약점 발견-패치-배포 사이클이 병목이 된다

오픈소스 프로젝트의 보안 모델은 “많은 눈이 보면 버그가 없다”는 가정에서 출발했다.
이 가정은 취약점 발견에 사람의 노력이 필요하다는 전제에 기대고 있었다.
에이전트가 대규모 코드베이스를 저비용으로 탐색할 수 있게 되면,
그 전제가 흔들린다.

취약점 발견 속도가 빨라지면 패치 개발과 배포도 빨라져야 한다.
그러나 오픈소스 프로젝트의 패치 과정은 여전히 사람의 리뷰, 테스트, 릴리스,
그리고 수백만 사용자의 업데이트 적용에 의존한다.
이 사이클의 어느 단계도 에이전트 발견 속도에 맞게 자동화되지 않았다.

다음 10년 안에 오픈소스 생태계가 해결해야 할 핵심 문제 중 하나는
이 사이클의 각 단계를 얼마나 빠르게 만들 수 있는가다.
자동 패치 생성, 회귀 테스트 자동화, 관리형 배포 기본값화는
취약점 발견 가속화에 대응하는 방향이지만,
각각이 새로운 위험을 도입한다는 점도 함께 고려해야 한다.

### 익스플로잇 브로커 시장 가격도 하락 압력을 받는다

익스플로잇 발견 비용이 낮아지면 브로커 시장 가격도 내려갈 수밖에 없다는 논리가 HN에서 나왔다.
"발견하는 데 25달러가 든 익스플로잇에 그보다 훨씬 많은 돈을 쓰는 것은 비효율적이다."[^barbazoo]
공급이 늘고 발견 비용이 내려가면 브로커 가격도 따라서 내려가는 것은 시장 논리상 자연스러운 방향이다.

이 압력이 실제로 작동한다면, 익스플로잇 시장의 구조 변화가 생긴다.
현재 브로커 시장에서 높은 가격이 형성된 이유 중 하나는
고가 취약점을 찾을 수 있는 연구자가 소수라는 공급 제약이다.
AI 에이전트가 이 제약을 완화하면, 시장 가격과 함께 공격 도구 접근성도 변한다.

한편 실제로 이 취약점을 통한 공격을 경험했다는 댓글도 달렸다.[^secretslol]
해당 사이트에 관리자 계정 2개가 생성됐고, 웹 셸과 백도어가 심어졌으며,
WordPress 버전 번호를 패치된 버전처럼 보이도록 조작하는 파일까지 포함됐다고 한다.
이 사례는 취약점이 이미 실제 공격에 사용됐음을 시사한다.

### AI 보안 연구는 공개 방식의 새로운 기준이 필요하다

이 연구는 책임 있는 공개(responsible disclosure) 절차를 따랐다.
그러나 에이전트 기반 취약점 발견이 일반화되면,
개별 연구자 수준의 책임 있는 공개로는 충분하지 않을 수 있다.

하나의 방법론이 공개되면 다른 사람들이 같은 방식으로 다른 취약점을 찾는다.
25달러의 탐색 비용은 진입 장벽을 크게 낮춘다.
악의적 행위자가 같은 방법론을 써서 동시에 여러 프로젝트를 탐색하는 상황이 올 수 있고,
