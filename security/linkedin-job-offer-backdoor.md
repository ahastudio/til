# LinkedIn 채용 제안에 숨겨진 백도어

원문: <https://roman.pt/posts/linkedin-backdoor/>

GeekNews: <https://news.hada.io/topic?id=30526>

HN 토론: <https://news.ycombinator.com/item?id=48546294> (1022점, 195개 댓글)

## 요약

개발자 로만 이만쿨로프(Roman Imankulov)가 LinkedIn을 통해 접근한 가짜 리크루터의 공격 시도를 직접 분석한 글이다.
리크루터는 크립토 스타트업을 대표한다고 소개하며, 퍼블릭 GitHub 저장소의 “구버전 Node 모듈 문제”를 코드 리뷰해달라고 요청했다.

악성 코드는 `app/test/index.js` 파일 안에 숨겨져 있었다.
약 250줄의 이 파일은 겉보기에 테스트 코드처럼 보이지만, 도메인 컴포넌트를 조각내어 URL을 조립하고 서버가 전송하는 임의의 코드를 실행하도록 설계되어 있었다.

트리거 메커니즘은 정교하다.
`app/index.js`가 테스트 파일을 `require`하고, `package.json`의 `prepare` 스크립트가 `node app/index.js`를 실행한다.
npm은 `prepare` 스크립트를 설치 시 자동으로 실행한다.
즉, 피해자가 단순히 `npm install`을 실행하는 것만으로 백도어가 작동된다.

신원 도용은 다층적으로 이루어졌다.
저장소 커밋은 실제 존재하는 풀스택 개발자의 정체를 도용했으며, 그 개발자는 자신이 해당 코드를 작성한 적이 없다고 확인했다.
리크루터 프로필 역시 실제 언론인의 신원을 도용한 것이었다.

이만쿨로프는 로컬 실행 대신 읽기 전용 AI 에이전트로 코드를 분석하여 감염을 피했다.
에이전트는 즉시 의심스러운 패턴을 식별했다.

## 분석

### npm 생명주기 스크립트의 구조적 취약점

이 공격의 핵심은 npm의 정상적인 기능을 무기화한다는 점이다.
`prepare` 스크립트는 npm이 패키지 설치 후 자동으로 실행하는 합법적 메커니즘이다.
빌드 단계, 타입 컴파일, 자산 생성 등에 사용된다.
공격자는 이 메커니즘이 존재한다는 사실 자체를 이용한다.

HN에서 gleenn은 “npm이 의존성 설치 중 임의 코드를 실행할 수 있다는 것은 설계상 명백한 취약점이며 폐기되어야 한다”고 주장했다.[^gleenn]
aykutseker는 이 공격의 사회공학적 정교함을 지적했다.[^aykutseker]
실제 채용 면접 프로세스에는 저장소를 검토하고 `npm install`을 실행하는 것이 표준 절차다.
공격자는 이 정상적인 절차를 그대로 이용한다.

### 신원 도용의 다층 구조

이 공격이 특히 효과적인 이유는 신뢰 체계의 여러 층을 동시에 공략하기 때문이다.
첫 번째 층은 LinkedIn 자체다.
pants2가 HN에서 지적했듯, “LinkedIn은 기업이 자사 직원을 사칭하는 가짜 계정을 제거할 수단을 제공하지 않는다.”[^pants2]
리크루터가 실제로 해당 회사 소속인지 검증하는 메커니즘이 없다.

두 번째 층은 GitHub 커밋 서명이다.
커밋이 특정 개발자 계정과 연결되어 있다고 해서 그 개발자가 실제로 코드를 작성했다는 의미가 아니다.
세 번째 층은 기업 도메인이다.
회사 이름, 도메인, 소셜 프로필이 모두 합법적으로 보이도록 조작될 수 있다.

### 공급망 공격의 사회공학화

HN에서 denysvitali는 이메일을 통해 유사한 공격을 직접 경험했으며 북한(DPRK) 소행으로 추정된다고 언급했다.[^denysvitali]
HN의 clemailacct1은 이 패턴을 라자루스 그룹(Lazarus Group), 구체적으로 Famous Chollima(북한 연계 행위자)로 귀속시켰고,[^clemailacct1]
abhisek는 같은 “contagious interview”(전염성 면접) 캠페인의 일종이며 이들이 IDE 설정이나 Claude 훅을 악용한 악성 코드 실행까지 시도해왔다고 덧붙였다.[^abhisek]

이 공격이 공급망 공격의 사회공학화된 버전이라는 점에서 중요하다.
전통적인 공급망 공격은 널리 사용되는 오픈소스 패키지에 악성 코드를 삽입한다.
이 변형은 개인을 타겟으로 삼아 특별히 설계된 저장소를 검토하도록 유도한다.
훨씬 좁은 타겟이지만, 숙련된 개발자를 대상으로 고도화된 공격이 가능하다.

이 패턴이 얼마나 일상화됐는지를 보여주는 증언이 댓글에 쏟아졌다.
크립토 업계에서 일하는 atraac는 이 사기가 거의 격일로 벌어진다며, LinkedIn에서 web3·crypto를 내건 모르는 사람은 모두 거절한다고 밝혔다.[^atraac]
dantodor는 지난 6개월간 세 번 겪었고 프로필과 코드가 점점 정교해지지만, 자기네 코드를 로컬에서 실행하라고 고집하는 순간 경보가 울린다고 했다.[^dantodor]
개별 사건이 아니라 조직적이고 지속적인 캠페인이라는 점이 분석에서 중요하다.

## 비평

### 개인 주의가 구조적 문제를 가린다

글쓴이는 “읽기 전용 AI 에이전트로 분석해서 감염을 피했다”는 개인적 해결책을 제시한다.
하지만 모든 개발자가 의심스러운 저장소를 분석하기 위한 격리된 환경을 갖출 것을 기대하는 것은 비현실적이다.

실제 문제는 npm의 자동 코드 실행 기능이 안전하지 않은 기본값(default)을 갖는다는 것이다.
`--ignore-scripts` 플래그가 존재하지만, 이것을 기본값으로 사용하는 팀은 드물다.
개별 개발자의 경계심을 해결책으로 제시하는 것은 시스템적 취약점을 개인화된 책임으로 전가한다.

### 경고 신호가 명확했다는 전제의 문제

글은 공격의 신호들이 회고적으로 명확해 보인다.
하지만 시간 압박, 피로, 흥미로운 기회처럼 보이는 상황에서는 같은 신호들이 쉽게 무시된다.
이만쿨로프 자신도 이 점을 언급한다.
교육적 글쓰기에서 “신호가 명확했다”는 서술은 “이 상황에서 어떻게 하면 더 잘 대응할 수 있었는가”라는 실용적 질문으로 이어지지 않는다.

이 실용적 공백을 HN의 한 댓글이 메웠다.
f055는 한때 주당 2~3건의 수상한 크립토 제안을 받았는데, 공식 이메일·공식 채용 링크·업무 범위·계약 조건 같은 다단계 검증을 요구하고
“채용 과제”를 위해 어떤 코드도 실행하거나 패키지를 설치하지 않겠다는 단호한 거부를 AI로 자동 응답하기 시작하자 제안이 끊겼다고 밝혔다.[^f055]
회고적 “신호 읽기”보다 사전에 합의해 둔 거부 규칙이 더 실효적인 방어라는 점을 보여준다.

### 플랫폼이 신고를 받고도 움직이지 않는다

이만쿨로프는 저장소를 GitHub에, 리크루터를 LinkedIn에 신고했지만 아무것도 바뀌지 않았고 악성 코드는 그대로 남아 있다고 적었다.
HN에서도 같은 좌절이 반복됐다.
한 사용자는 GitHub(마이크로소프트)의 무대응을 짧게 꼬집었고,[^BobAliceInATree]
jhancock은 침해된 유명 보안 연구자의 신원이 채용에 도용된 사례를 LinkedIn과 GitHub에 신고했지만 아무 조치가 없었다고 증언했다.[^jhancock]
또 다른 사용자는 사이버 범죄를 신고하고 도움받을 수 있는 “911” 같은 창구가 사회에 없다는 점을 지적하며,
조직적 범죄에는 조직적 방어가 필요하다고 주장했다.[^jmward01]
개인의 경계심을 강조하는 결론은 이 구조적 공백 앞에서 한계가 분명하다.

## 인사이트

### npm 설치 = 코드 실행이라는 인식이 보편화되어야 한다

오픈소스 커뮤니티에서 “신뢰할 수 없는 코드를 실행하지 마라”는 원칙은 오래됐지만, `npm install`이 코드 실행임을 인식하는 개발자는 여전히 많지 않다.
패키지 설치가 파일 복사와 동일하게 느껴지는 추상화 레이어가 이 인식 격차를 만든다.

개발 환경 격리(sandbox, VM, 컨테이너)가 단순한 성능 최적화가 아니라 보안 필수 요소로 자리매김해야 한다.
이미 이 방향으로 움직이는 도구들이 있지만, 기본 개발 워크플로에 통합되지 않는 한 채택률은 낮을 것이다.

### 채용 사기가 기술적 공격의 벡터로 진화하고 있다

LinkedIn 가짜 리크루터는 오래된 현상이지만, 그 목적이 개인 정보 수집에서 시스템 침투로 이동하고 있다.
개발자를 대상으로 하는 이 공격은 특별히 위험하다.
개발자는 일반 사용자보다 더 많은 시스템에 접근하고, 더 많은 비밀 정보를 다루며, 회사 내 다른 시스템의 진입점이 된다.

이것은 보안 교육의 범위가 “이메일 피싱”에서 “채용 사기 + 코드 리뷰 요청”까지 확장되어야 한다는 것을 의미한다.
사회공학 공격이 기술적 전문성을 가진 목표를 위해 기술적으로 정교해지고 있다.

### 신원 검증 인프라의 부재가 공격 비용을 낮춘다

이 공격이 가능한 근본 이유 중 하나는 온라인 신원 검증 인프라가 미성숙하다는 것이다.
LinkedIn 계정, GitHub 커밋, 기업 이메일 도메인 — 이 세 가지를 모두 갖고 있어도 실제 신원은 검증되지 않는다.

Web3 커뮤니티가 주장하는 자기주권 신원(Self-Sovereign Identity) 같은 개념이 이 문제에 대한 이론적 해답을 제시한다.
그러나 기존 플랫폼들이 이런 시스템을 채택할 인센티브는 크지 않다.
LinkedIn이 가짜 계정을 제거하기 위한 강력한 도구를 제공하면 플랫폼 상의 “활동적 사용자”가 감소할 수 있기 때문이다.
