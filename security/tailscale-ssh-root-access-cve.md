# TS-2026-009: Tailscale SSH의 잘못된 인자 처리로 인한 root 권한 획득

원문: [Security Bulletins · Tailscale](https://tailscale.com/security-bulletins)

HN 토론: <https://news.ycombinator.com/item?id=48915004> (212점, 145개 댓글)

## 요약

Tailscale은 2026년 7월 13일 보안 공지 TS-2026-009를 통해 Tailscale SSH의
취약점을 공개했다.
이 취약점은 Linux 환경에서 하이픈(`-`)으로 시작하는 사용자 이름을
그대로 받아들이던 처리 방식에서 비롯됐다.
Tailscale SSH는 사용자 이름을 조회할 때 내부적으로 `getent(1)` 명령을
호출하는데, 하이픈으로 시작하는 사용자 이름이 이 호출의 인자로 그대로
전달되면서 명령줄 플래그로 해석될 여지가 생겼다.

구체적으로, 공격자가 `-i`라는 사용자 이름으로 SSH 접속을 시도하면 이
값이 `getent`의 `--no-idn` 플래그로 해석됐다.
그 결과 `getent`는 사용자 이름 필터링 없이 `passwd` 파일 전체를
root부터 출력했고, 이 과정에서 ACL 정책이 특정 사용자에게 root 접근을
허용하지 않았음에도 root 세션으로 로그인이 가능해지는 결과로 이어졌다.
즉 인자와 값 사이의 경계가 무너지면서, 접근 제어 계층을 완전히
우회하는 권한 상승이 발생한 것이다.

영향을 받는 버전은 1.98.9 이전의 모든 Tailscale 버전이며, 수정은
1.98.9에서 이루어졌다.
수정 방식은 근본적인 파싱 로직 개선이 아니라, 하이픈으로 시작하는
사용자 이름 자체를 거부하는 입력 검증 추가였다.
이 취약점은 Anthropic과 Ada Logics에 의해 발견되어 책임 있는 방식으로
Tailscale에 보고됐다.

같은 날 Tailscale은 TS-2026-008(Serve/Funnel의 경로 처리 무한 루프로
인한 서비스 거부)과 TS-2026-007(Services의 인바운드 패킷 필터링
미비로 인한 loopback 접근)도 함께 공개했으며, 앞서 6월에는
TS-2026-006(숫자 UID를 통한 root ACL 우회)과 TS-2026-004/005(Unix
소켓 프록시 및 심볼릭 링크 관련 권한 문제) 등 SSH와 권한 경계에 관련된
공지가 연달아 이어졌다.

## 분석

### 이 취약점은 고전적인 인자 주입 패턴의 재현이다

`-i`가 `getent`의 플래그로 해석되는 구조는 셸 프로그래밍에서 수십 년간
반복돼 온 “인자 인젝션(argument injection)” 패턴과 정확히 같은 형태다.
`rm -- “$file”`처럼 `--`로 옵션 종료를 명시하지 않으면, 사용자가
제어하는 문자열이 파일 이름이 아니라 플래그로 둔갑할 수 있다는 것은
POSIX 유틸리티를 다루는 코드에서 잘 알려진 함정이다.
Tailscale SSH의 개발자들이 이 함정을 몰랐을 가능성은 낮다.
오히려 이 사례가 흥미로운 지점은, 사용자 이름이라는 값이 “신뢰할 수
있는 내부 데이터”로 취급되면서 외부 입력 검증 대상에서 누락됐다는
점이다.
사용자 이름은 SSH 연결 요청에 실려 오는 시점에는 분명히 외부 입력인데,
이후 시스템 콜 체인을 따라 내려가면서 어느 지점에서인가 “이미 검증된
값”이라는 암묵적 신뢰를 얻은 것으로 보인다.
HN 토론에서 tptacek는 이 버그 계열이 최소한 AIX 3까지 거슬러 올라가는
“유서 깊고 오래된” 클래스라고 지적했다[^tptacek].
jcarrano 역시 같은 맥락에서, 사용자 제어 값을 서브프로세스 호출의
인자로 넘기는 대신 처음부터 `getpwnam` 같은 실제 API·시스템 콜을
쓰는 것이 정석이라고 짚었다[^jcarrano].
cyberax는 한 걸음 더 나아가 Tailscale이 택한 수정 자체를
문제 삼았는데, “제대로 된 수정은 `--`로 인자를 구분하는 것”이라며
하이픈 거부는 근본 해법이 아니라고 비판했다[^cyberax].
eviks도 비슷하게, “왜 권한 문제 자체를 고치지 않고 사용자를 제한하는
쪽을 택했는가”라고 반문하며 같은 공지 묶음의 TS-2026-006(숫자 UID
거부)에도 동일한 의문을 제기했다[^eviks].

### root로 실행되는 프로세스가 외부 유틸리티를 호출하는 구조 자체가 위험 표면이다

Tailscale SSH 데몬은 root 권한으로 동작하며, 사용자 인증과 세션 생성을
위해 `getent`라는 별도 프로세스를 실행한다.
이 구조는 필연적으로 두 개의 신뢰 경계를 만든다.
하나는 SSH 프로토콜 계층에서 사용자 이름을 받아들이는 경계이고,
다른 하나는 그 값을 셸 명령줄 인자로 넘기는 경계다.
문제는 이 두 경계가 서로 다른 데이터 모델을 전제한다는 것이다.
SSH 계층에서 사용자 이름은 단순한 문자열이지만, 명령줄 인자 계층에서
같은 문자열은 위치와 접두사에 따라 값도 되고 플래그도 될 수 있는
이중적 존재다.
이 경계를 넘나드는 순간 데이터의 의미가 암묵적으로 바뀌는데, 이런
전환 지점이야말로 권한 상승 취약점이 반복적으로 발생하는 자리다.
같은 공지 묶음에 등장하는 TS-2026-006(숫자 UID로 root 지칭)과
TS-2026-004(심볼릭 링크를 통한 권한 우회) 역시 근본적으로는 같은
계열이다.
SSH 계층이 사용자를 식별하는 방식과, OS 계층이 실제 권한을 판단하는
방식 사이의 간극을 공격자가 파고든 사례들이다.

### 짧은 시간에 몰린 다발적 공지는 목표를 정한 감사(audit)의 산물로 보인다

TS-2026-004부터 TS-2026-009까지, 두 달이 채 안 되는 기간에 Tailscale
SSH와 그 주변 권한 처리 로직에서만 6건의 공지가 나왔다.
이는 무작위로 발견된 개별 버그들의 우연한 집중이라기보다, 누군가
Tailscale SSH의 권한 경계를 체계적으로 훑어본 결과로 읽는 편이
합리적이다.
실제로 TS-2026-009의 발견자로 명시된 곳은 Anthropic과 Ada Logics인데,
Ada Logics는 오픈소스 프로젝트의 보안 감사를 전문으로 하는 회사로 잘
알려져 있다.
이런 조합은 특정 서브시스템(여기서는 SSH 권한 위임 경로)을 표적으로
삼아 유사 취약점을 연쇄적으로 찾아내는 감사 방법론의 전형적인 결과물에
가깝다.
한 번의 감사에서 같은 클래스의 취약점이 여러 개 나온다는 사실은,
Tailscale SSH의 사용자 매핑 계층 전체가 구조적으로 취약한 설계 위에
서 있었다는 뜻이기도 하다.
공지문의 감사 표기를 두고 HN에서 mintflow는 “Anthropic도 Tailscale을
쓰는 것인지, 아니면 특정 모델이 발견한 것인지” 궁금해했는데[^mintflow],
이는 최근 AI 모델을 활용한 자동화된 취약점 탐색이 늘어나면서 보안
공지의 발견 경위 자체가 더 이상 자명하지 않게 됐다는 사실을 드러낸다.

## 비평

### “하이픈 거부”라는 수정은 증상 치료이지 원인 치료가 아니다

Tailscale이 선택한 수정 방식은 하이픈으로 시작하는 사용자 이름을
거부하는 것이다.
이 방식은 지금 알려진 `-i` 벡터는 막지만, `getent`의 인자 파싱
규칙에서 파생될 수 있는 다른 형태의 문제를 원천적으로 차단하지는
않는다.
예를 들어 공백이 포함된 값, 특정 로케일에서 특수하게 해석되는 문자,
혹은 `getent`가 아닌 다른 외부 유틸리티를 새로 호출하게 될 때 똑같은
클래스의 문제가 재발할 여지가 남는다.
근본적인 해법은 사용자 제어 값을 외부 프로세스의 인자로 넘길 때
`--` 구분자를 강제하거나, 애초에 셸/인자 파싱을 거치지 않는
시스템 콜(getpwnam 계열의 라이브러리 함수 등)로 대체하는 것이다.
공지문은 “하이픈으로 시작하는 사용자 이름을 거부한다”는 조치만
언급할 뿐, 이런 근본적 아키텍처 변경을 계획하고 있는지에 대해서는
아무런 언급이 없다.
이는 이번 벡터를 막았다는 확인일 뿐, 같은 계열의 다음 벡터가 나오지
않으리라는 보장은 아니다.
HN에서도 정확히 같은 지적이 나왔다.
cyberax는 공지문의 수정 설명을 그대로 인용하며 “정말로 그게
수정이냐”고 반문했고[^cyberax], s_ting765는 수정 방향 이전에 배포
프로세스 자체를 문제 삼아, 공지를 낸 시점에도 Docker Hub 이미지가
8일째 갱신되지 않은 상태였다며 “제때 패치를 배포하지 않을 것이면
공지를 내는 의미가 무엇이냐”고 비판했다[^s_ting765].

### 6건의 연쇄 공지에도 재발 방지를 위한 구조적 설명이 빠져 있다

두 달 사이 SSH 권한 경계에서 6건의 공지가 나왔다는 사실 자체가
이례적인데, Tailscale의 공지문 어디에도 이 클러스터링에 대한 메타적
설명이 없다.
외부 감사를 받았다는 사실, 그 감사의 범위가 무엇이었는지, 감사가
완료됐는지 아니면 진행 중인지에 대한 언급이 전무하다.
사용자 입장에서는 “이번 수정으로 안전해졌다”는 확신을 갖기 어렵다.
같은 서브시스템에서 이미 여섯 번 문제가 나왔다면, 일곱 번째가 없을
것이라고 믿을 근거가 공지문 안에는 없기 때문이다.
보안 공지의 목적이 투명성이라면, 개별 버그의 기술적 설명뿐 아니라
“우리가 이 영역을 얼마나 더 살펴봤고, 앞으로 무엇을 다르게 할
것인가”에 대한 답도 함께 제공돼야 한다.
이 부분이 빠져 있다는 것은 공지의 완성도에 대한 명확한 결함이다.

### CVE 미부여는 심각도 판단을 사용자에게 떠넘긴다

이 공지에는 CVE 식별자가 부여되지 않았다.
Tailscale SSH를 통해 ACL을 우회해 root 세션을 얻을 수 있었다는 것은
권한 상승 취약점의 정의에 정확히 들어맞는데도, 표준화된 식별자 없이
자체 번호 체계(TS-2026-009)만으로 공지가 이루어졌다.
CVE가 없으면 조직들이 사용하는 취약점 스캐너나 자산 관리 도구가 이
문제를 자동으로 추적하기 어렵고, 심각도(CVSS) 점수도 공식적으로
매겨지지 않는다.
결과적으로 각 조직의 보안 담당자가 공지문을 직접 읽고 심각도를 스스로
판단해야 하는 부담을 떠안게 되는데, 이는 원격 root 권한 상승이라는
사안의 무게에 비해 지나치게 안일한 처리로 보인다.
HN에서 kbumsik도 같은 의문을 제기하며 “왜 자체 번호 체계이고
CVE가 아닌가”라고 물었는데[^kbumsik], 이 질문에 대한 명확한 답은
스레드 안에서도, 공지문 안에서도 나오지 않았다.

## 인사이트

### 신뢰 전이(trust transfer)는 코드 리뷰로 잡아내기 어려운 결함이다

이 취약점의 핵심은 “검증되지 않은 입력을 검증했다”는 데 있지 않다.
사용자 이름은 SSH 인증 과정에서 이미 한 차례 처리를 거친 값이고,
개발자 입장에서는 그 시점에 이미 “안전한 문자열”이라는 심리적 표식이
붙었을 가능성이 크다.
문제는 그 표식이 다음 계층, 즉 명령줄 인자로 넘어가는 순간에는
전혀 다른 의미를 갖는다는 점을 코드만 봐서는 알아채기 어렵다는
것이다.
코드 리뷰어는 “이 값은 이미 SSH 계층에서 검증됐으니 안전하다”는
전제를 그대로 물려받기 쉽고, 그 전제가 새로운 문맥에서는 성립하지
않는다는 것을 알아차리려면 데이터가 거쳐 온 전체 경로를 추적해야
한다.
이런 종류의 결함은 단위 테스트나 정적 분석 도구가 잡아내기 어렵고,
오직 “이 값이 다음 시스템 경계를 넘을 때 무엇으로 재해석되는가”를
의도적으로 묻는 위협 모델링을 통해서만 드러난다.
Ada Logics 같은 외부 감사 업체가 반복적으로 이런 유형의 문제를
찾아낸다는 사실은, 내부 개발팀의 역량 문제라기보다 이런 결함이 애초에
내부자 시점에서는 구조적으로 보이지 않는 사각지대에 있다는 것을
시사한다.

### 원격 접속 도구는 “그 자체가 root로 동작하는 게이트키퍼”라는 이중 위험을 안는다

Tailscale SSH처럼 OS 수준 인증을 대체하거나 우회하는 원격 접속
계층은 구조적으로 두 배의 위험을 진다.
하나는 이 계층 자체가 뚫리면 침입 경로가 된다는 전통적 위험이고,
다른 하나는 이 계층이 root로 동작하며 OS의 네이티브 권한 시스템을
대신 판단해 준다는 데서 오는 위험이다.
후자는 특히 위험한데, 왜냐하면 OS가 수십 년에 걸쳐 다듬어 온 권한
경계(uid, gid, PAM 등)를 우회해 SSH 계층이 자체적으로 “누가 root인가”
를 다시 판단해야 하기 때문이다.
이는 이미 검증된 안전장치를 새로 재발명하는 것과 같고, 이번 취약점을
포함해 TS-2026-004, 006에서 반복적으로 나타난 패턴이 바로 이
재발명 과정에서의 결함이다.
제로 트러스트 네트워킹을 표방하는 도구일수록 이런 “메타 권한 계층”을
자체 구현하게 되는데, 이 계층의 신뢰성이 결국 그 도구가 지키겠다고
약속한 보안 모델 전체의 신뢰성을 좌우한다는 역설이 여기에 있다.
HN에서 RossBencina는 이 지점을 정확히 짚었다.
“SSH 로그인 프로세스가 기본적으로 root 로그인을 가능하게 할 수 있는
권한을 왜 갖고 있어야 하는가”라는 그의 질문은[^rossbencina], 이번
버그의 파급력이 인자 파싱 실수 하나 때문이 아니라 그 실수가 발생한
프로세스 자체가 애초에 root 권한을 쥐고 있었기 때문에 증폭됐다는
사실을 드러낸다.
farfatched는 이를 더 냉소적으로 요약했는데, “Tailscale을 통한 root
경로가 또 하나 늘었다”면서 “그들의 (권한) 범위가 계속 커지고 root로
그렇게 많은 것을 실행한다면, 이번이 마지막은 아닐 것”이라고
지적했다[^farfatched].
luciana1u는 여기서 한 걸음 더 나아가 아키텍처 선택 자체를 비판했다.
“25년간 실전에서 검증된 코드베이스(OpenSSH)를 스타트업의 Go
재작성물로 대체해 놓고, 버그가 나오면 놀라는 척한다”는 그의
평가는[^luciana1u] 검증된 시스템을 대체할 때 감수해야 할 위험을
직설적으로 짚은 것이다.

### 짧은 주기의 다발적 공지는 사용자에게 역설적으로 피로감을 남길 수 있다

두 달 사이 6건의 관련 공지가 이어지는 것은 보안 프로세스가 건강하게
작동하고 있다는 신호로 읽을 수도 있지만, 동시에 사용자 커뮤니티에는
“이 제품의 핵심 기능(SSH)이 근본적으로 불안정한 것 아닌가”라는 피로와
불신을 남길 위험도 있다.
실제로 이 공지에 대한 Hacker News 토론에서도 이런 반응이 나타났다.
Tailscale의 오랜 사용자라고 밝힌 doublepg23는 “Tailscale 자체는
꽤 신뢰하지만, Tailscale SSH 기능은 한 번도 써본 적이 없다”면서
“OpenSSH의 보안 이력은 사실상 따라잡기 어려운데 왜 굳이 이렇게
보안에 민감한 기능을 바꿔야 하는지 모르겠다”고 밝혔다[^doublepg23].
modeless는 Tailscale SSH가 22번 포트를 그대로 점유해 다른 문제를
일으킨 적이 있다며 애초에 선호하지 않는다고 덧붙였고[^modeless],
drnick1은 아예 “나는 100% 자체 호스팅한 WireGuard 구성을
고수하겠다”며 이런 통합형 도구 자체에 대한 회의를 드러냈다[^drnick1].
이런 반응들은 개별 버그의 기술적 심각도와는 별개로, 반복되는 공지가
기능 자체에 대한 신뢰를 갉아먹을 수 있음을 보여준다.
보안 공지가 투명하게, 자주 나오는 것은 원칙적으로 바람직한 일이지만,
그 빈도가 사용자의 신뢰 계좌에서 인출로 작용할지 입금으로 작용할지는
공지의 형식과 맥락 설명에 크게 좌우된다.
개별 버그 하나하나를 성실히 공개하는 조직이, 정작 “왜 같은 서브시스템
에서 이렇게 많은 문제가 한꺼번에 나왔는가”라는 상위 질문에는 답하지
않는다면, 투명성이라는 미덕이 오히려 근본 원인에 대한 침묵을 가리는
결과로 이어질 수 있다.
이것은 비단 Tailscale만의 문제가 아니라, 정기적인 보안 공지 체계를
운영하는 모든 인프라 소프트웨어 업체가 마주하는 커뮤니케이션 설계의
문제다.
