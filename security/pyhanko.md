# pyHanko — PDF 서명·스탬핑 Python 라이브러리

<https://github.com/MatthiasValvekens/pyHanko>

<https://docs.pyhanko.eu/en/latest/>

## 소개

pyHanko는 PDF 파일의 디지털 서명, 스탬핑, 암호화를 처리하는 Python 라이브러리 및 CLI 도구다.
MatthiasValvekens가 개발했으며 MIT 라이선스로 공개되어 있다.
2025년 7월 기준 GitHub 스타 738개, 포크 103개다.
공식 문서에서 “현재 베타 상태이며 아직 프로덕션 준비가 되지 않았다”고 명시한다.

Python 3.10 이상을 요구하며, 선택적 의존성으로 PKCS#11 토큰, 이미지 처리, OpenType 폰트,
QR 코드 생성을 지원한다.

```bash
pip install 'pyHanko[pkcs11,image-support,opentype,qr]' pyhanko-cli
```

## 주요 기능

### 서명(Signing)

- 비가시적 서명 또는 시각적 표현을 포함한 서명
- PAdES 기준선 프로파일 B-B, B-T, B-LT, B-LTA 지원 (장기 검증 가능 서명)
- RFC 3161 타임스탬프 서버 연동
- 증분 업데이트(incremental update) 방식으로 단일 문서에 다중 서명 지원
- 알고리즘: RSA(PKCS#1 v1.5, RSASSA-PSS), DSA, ECDSA, EdDSA(Ed25519/Ed448), ML-DSA(격자 기반 후양자)
- PKCS#11 하드웨어 토큰 지원, Belgian eID 카드 전용 편의 래퍼 포함
- 원격·비대화형 프로세스를 위한 중단형 서명(interrupted signing) 모드

### 검증(Validation)

- 암호학적 무결성 검증
- X.509 신뢰 체인 인증
- 다중 서명 문서에서 서명 이후 변경 사항 차이 분석(difference analysis)
- AdES 검증, EU 신뢰 목록(trusted list) 연동

### 스탬핑(Stamping)

- 텍스트 기반 스탬프 (폰트 설정 가능)
- QR 코드 스탬프
- TrueType/OpenType 임베디드 폰트 지원

### 암호화(Encryption)

- PDF 2.0 암호화 방식 전체 지원
- ISO/TS 32003 및 32004 기반 인증 암호화
- RSAES-OAEP, 임시-정적 ECDH(X9.63 키 파생) 추가 모드

지원 표준: PDF 2.0 암호화, PAdES B-B~B-LTA, ISO/TS 32001·32002·32003·32004, RFC 3161.

## 분석

### PAdES 장기 검증 프로파일이 요구하는 기술적 복잡성

PAdES(PDF Advanced Electronic Signatures)는 단순한 서명 포맷이 아니라
검증 가능성의 수명을 연장하는 체계다.
네 개의 기준선 프로파일은 누적 구조를 이룬다.

- **B-B**: 서명자 인증서와 서명 알고리즘을 포함한 기본 서명
- **B-T**: B-B에 RFC 3161 타임스탬프를 추가하여 서명 시각을 증명
- **B-LT**: B-T에 검증에 필요한 인증서·CRL·OCSP 응답을 문서에 임베딩
- **B-LTA**: B-LT에 아카이브 타임스탬프를 추가하여 장기 보존 가능

B-LTA까지 구현하려면 PDF 내부 구조, CMS(Cryptographic Message Syntax),
X.509 PKI, OCSP/CRL 프로토콜을 동시에 이해해야 한다.
이것이 Python 생태계에서 pyHanko 이전에 완전한 PAdES 구현이 없었던 이유다.

### 증분 업데이트 서명이 다중 서명을 가능하게 하는 메커니즘

PDF의 증분 업데이트(incremental update)는 기존 바이트 스트림을 수정하지 않고
파일 끝에 변경 사항을 추가하는 방식이다.
이 구조가 다중 서명을 기술적으로 가능하게 한다.

서명자 A가 서명한 문서에 서명자 B가 추가 서명할 때,
B의 서명은 A의 서명 이후에 추가된 증분 업데이트 위에 적용된다.
검증 시 A의 서명이 포함한 바이트 범위와 B의 추가 변경 사항을 분리하여
A의 서명이 여전히 유효한지 확인해야 한다.

pyHanko의 차이 분석(difference analysis)은 이 분리를 자동화한다.
어떤 변경이 서명 이후에 이루어졌는지, 그 변경이 정책상 허용된 것인지를 판단한다.
예를 들어 서명 후 폼 필드 채우기는 일반적으로 허용되지만,
서명된 본문 내용의 수정은 허용되지 않는다.

### ML-DSA 지원이 나타내는 후양자(Post-Quantum) 전환 준비

ML-DSA(Module-Lattice-based Digital Signature Algorithm)는
NIST가 2024년 표준화한 격자 기반 서명 알고리즘이다.
RSA나 ECDSA와 달리 대형 양자 컴퓨터의 공격에 저항성을 갖도록 설계됐다.

pyHanko가 ML-DSA를 지원한다는 것은 라이브러리 수준에서 후양자 전환을 준비하는 것이다.
현재 Certificate Authority(CA) 생태계, Adobe Approved Trust List(AATL),
EU Trusted List는 ML-DSA 기반 인증서를 아직 발급하지 않는다.
즉 지금 당장 ML-DSA로 서명한 PDF는 표준 PDF 뷰어에서 검증되지 않는다.

그러나 PKI 인프라가 후양자로 이행하는 시점에 라이브러리가 이미 준비되어 있다는 것은
전환 비용을 낮추는 선제적 선택이다.
“Harvest Now, Decrypt Later” 공격이 현실화되는 환경에서
서명 도구의 후양자 지원은 기술 채무 관점에서 의미가 있다.

## 비평

### 베타 상태가 법적 구속력 있는 문서에 미치는 위험

pyHanko가 “not yet production-ready”를 명시하는 것은 정직하지만,
이 경고가 실제 사용 맥락에서 얼마나 반영될지는 불분명하다.
738 스타와 103 포크는 상당한 채택이 이루어졌음을 시사한다.

법적 구속력 있는 문서(계약서, 공증 문서, 규제 제출물)에 베타 소프트웨어를 사용하면
기술적 하자가 법적 분쟁으로 이어질 수 있다.
서명 검증이 나중에 실패하거나 프로파일 구현에 오류가 발견됐을 때,
“베타”라는 경고가 법적 보호를 제공하지 않는다.
Python 생태계에서 대안이 없기 때문에 사용자들이 이 경고를 무시하고 채택하는 상황이 발생한다.

### Belgian eID 특화 래퍼가 드러내는 지역 편중

PKCS#11 지원은 범용적이지만, Belgian eID 카드 전용 편의 래퍼의 포함은
이 프로젝트가 EU 법적·기술적 맥락에서 출발했음을 보여준다.
PAdES 자체가 EU의 eIDAS 규정에 기반한 표준이며,
“EU trusted list” 연동은 EU 이외 지역에서는 직접적 의미가 없다.

한국의 국가공인인증서(NPKI) 체계, 미국의 ESIGN Act 및 WebTrust PKI,
일본의 공적개인인증서비스(JPKI)는 각각 다른 신뢰 체계를 사용한다.
pyHanko가 이 지역 신뢰 체계들과 얼마나 호환되는지는 문서에서 명확하지 않다.
“국제 표준을 구현한다”는 것이 “어디서나 법적으로 유효하다”는 의미가 아님을 사용자가 알아야 한다.

### 차이 분석의 허용 범위가 사용 맥락에 따라 달라진다

차이 분석(difference analysis)은 서명 이후 변경이 “허용 가능한가”를 판단한다.
그런데 “허용 가능”의 기준은 문서 유형과 서명 정책에 따라 다르다.

의료 기록, 법적 계약, 규제 제출 문서는 서명 이후 어떤 변경도 허용하지 않는 엄격한 정책이 필요하다.
반면 계약 체결 과정에서 여러 당사자가 순차적으로 서명하고 일부 필드를 채우는 워크플로는
특정 범위의 변경을 허용해야 한다.
pyHanko의 차이 분석 규칙이 이 다양한 정책 요구사항을 얼마나 세밀하게 표현할 수 있는지,
그리고 기본값이 어떤 보안 가정을 담고 있는지는 문서 표면에서 파악하기 어렵다.

## 인사이트

### PDF 서명의 기술적 유효성과 법적 유효성은 별개의 문제다

pyHanko로 PAdES B-LTA 서명을 생성하면 기술적으로는 최고 수준의 장기 검증 가능 서명이 된다.
그러나 이 서명이 법적으로 유효한지는 전혀 다른 질문이다.

법적 유효성은 세 가지 조건이 충족되어야 한다.
첫째, 서명에 사용된 인증서가 해당 국가·지역에서 공인된 CA로부터 발급되어야 한다.
둘째, 서명 방식이 해당 법률(eIDAS, ESIGN Act, 전자서명법 등)의 요구사항을 준수해야 한다.
셋째, 분쟁 시 법원이 전자서명의 법적 효력을 인정해야 한다.
개인 CA로 자체 서명한 PAdES B-LTA는 기술적으로 완벽하지만 법적으로는 아무 의미가 없다.

pyHanko는 기술 표준 구현을 제공하지만 법적 유효성 체계와의 연결은 사용자의 책임이다.
이 구분이 명확하지 않으면 “표준을 준수하는 서명 = 법적으로 유효한 서명”이라는 오해가 생긴다.

### Python 생태계에서 오픈소스 PDF 서명 도구의 공백이 생긴 배경

PDF 서명을 제대로 구현하는 오픈소스 도구가 드문 이유는 기술적 장벽 때문만이 아니다.
시장 구조가 오픈소스 대안 발전을 억제했다.

iText(구 iTextSharp)는 오랫동안 Java/C# 생태계에서 PDF 조작의 사실상 표준이었다.
그러나 iText 5에서 iText 7으로 전환되면서 AGPL 라이선스로 이동했고,
상업적 사용을 위해서는 유료 라이선스가 필요해졌다.
Python에서는 reportlab, fpdf2, pikepdf 등이 있지만 디지털 서명을 제대로 지원하지 않는다.
pyHanko는 이 공백 속에서 등장했다.

738 스타가 베타 경고에도 쌓인 것은 수요가 공급을 앞서고 있다는 신호다.
엔터프라이즈 시장에서 PDF 서명 수요가 증가하는 동시에
Python 자동화 파이프라인에서 처리하려는 요구가 늘어났지만,
신뢰할 수 있는 오픈소스 선택지가 없었다.

### LTV 임베딩이 “오류의 장기 지속성” 문제를 낳는다

LTV(Long-Term Validation)의 핵심은 검증에 필요한 정보를 문서 안에 포함시키는 것이다.
타임스탬프 서버, CA, OCSP 응답자가 나중에 사라지더라도 문서를 독립적으로 검증할 수 있게 한다.
이것은 명백한 장점이다.

그러나 LTV 임베딩은 서명 시점에 수집한 검증 정보를 영구적으로 문서에 기록한다.
만약 OCSP 응답이 잘못됐거나(인증서가 이미 폐기됐는데 유효로 응답), CA 인프라에 오류가 있었다면,
그 오류가 B-LT 또는 B-LTA 서명을 통해 문서에 영구적으로 임베딩된다.
“장기 검증 가능성”이 “잘못된 검증 정보의 장기 고착”으로 이어질 수 있다.

이 문제는 LTV 자체의 결함이 아니라 신뢰의 수집 시점 문제다.
pyHanko가 OCSP/CRL 응답을 임베딩할 때 그 응답의 정확성을 어떻게 검증하는지,
그리고 사용자가 이 과정을 어떻게 감사할 수 있는지는 중요한 운영 질문이다.

### 후양자 전환이 PDF 서명 생태계 전체에 미치는 영향

ML-DSA 지원 라이브러리가 등장하는 것과 실제 후양자 PDF 서명이 법적으로 유효해지는 것 사이에는
생태계 전체의 전환이 필요하다.

브라우저, PDF 뷰어(Adobe Acrobat, PDF.js), CA, AATL, 국가별 신뢰 목록,
그리고 검증 라이브러리까지 모두 ML-DSA를 지원해야 한다.
이 전환은 2000년대 초 SHA-1에서 SHA-2로의 이행이나
RSA 1024비트에서 2048비트로의 이행과 구조적으로 유사하다.
그 전환들도 표준이 먼저 나오고 생태계 전체가 따라오는 데 10년 가까이 걸렸다.

pyHanko가 ML-DSA를 지원하는 것은 선도적이지만 고립된 선택이다.
현재 ML-DSA 서명 PDF를 정확히 검증할 수 있는 다른 도구가 없다면,
후양자 준비는 실험적 의미 이상을 갖기 어렵다.
그러나 라이브러리 수준에서 미리 지원하지 않으면 생태계 전환이 더 늦어진다는 역설도 있다.
