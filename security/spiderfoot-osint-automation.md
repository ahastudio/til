# SpiderFoot — OSINT 자동화 플랫폼

<https://github.com/smicallef/spiderfoot>

## 소개

SpiderFoot은 OSINT(Open Source Intelligence, 공개 출처 정보) 수집을 자동화하는
오픈소스 플랫폼으로, 2012년부터 개발되어 현재 GitHub에서 약 1만 9천 개의 스타를 받은
성숙한 프로젝트다.
Python 3.7+로 작성되었으며 MIT 라이선스로 공개되어 있다.
웹 UI와 CLI 두 가지 인터페이스를 제공하며,
단일 타깃에서 출발해 연관 데이터를 자동으로 확장 수집하는 방식으로 작동한다.

핵심 가치 제안은 200개 이상의 통합 모듈을 통해 수십 개의 외부 데이터 소스를
단일 인터페이스에서 조율한다는 점이다.
IP 주소, 도메인, 이메일, 전화번호, 사용자명, 비트코인 주소 등 다양한 타깃 유형을 지원하며,
서브도메인 열거(enumeration), 이메일/연락처 추출, 위협 인텔리전스 조회,
소셜 미디어 정찰, S3/클라우드 버킷 발견, 포트 스캔, 유출 데이터 검색 등을 수행한다.

## 주요 기능

**모듈 시스템**: 200개 이상의 모듈이 각각 특정 데이터 소스나 분석 유형을 담당한다.
대부분의 모듈은 API 키 없이 작동하며, 고급 기능이 필요한 경우 외부 서비스의 무료 티어를
활용할 수 있다.

**상관관계 엔진(Correlation Engine)**: YAML로 설정 가능한 상관관계 엔진이
37개의 사전 정의된 규칙을 통해 수집된 데이터 간의 연관성을 자동 분석한다.

**내보내기 및 저장**: CSV, JSON, GEXF 형식으로 결과를 내보낼 수 있으며,
SQLite 백엔드를 통해 직접 쿼리가 가능하다.

**서드파티 도구 통합**: DNSTwist, Nmap, CMSeeK 등 외부 도구와 연동되며,
TOR 통합을 통한 다크웹 연구도 지원한다.
Docker 배포를 공식 지원한다.

**SpiderFoot HX**: 클라우드 기반의 유료 버전으로, 공격 표면 모니터링,
다중 사용자 협업, API 접근, Splunk/Elasticsearch 기본 통합 기능을 제공한다.

## 분석

### OSINT 자동화의 핵심 긴장: 깊이 vs. 너비

SpiderFoot의 설계 철학은 단일 전문 도구의 깊이보다 통합 플랫폼의 너비를 선택한다.
이 선택은 OSINT 작업의 실제 병목이 어디에 있는지에 대한 판단에서 비롯된다.
OSINT 분석가들이 시간을 가장 많이 소비하는 곳은 단일 소스에서의 깊은 탐색이 아니라,
수십 개의 서로 다른 서비스를 순차적으로 조회하고 그 결과를 수동으로 연결하는 과정이다.

이 구조는 타협을 내포한다.
각 도메인에서 최고의 전문 도구와 비교하면 개별 모듈의 깊이는 떨어질 수 있다.
그러나 SpiderFoot의 진정한 가치는 모듈 하나하나가 아니라,
한 모듈의 출력이 다른 모듈의 입력으로 자동 연결되는 파이프라인 효과에 있다.
도메인 조회 결과로 발견된 이메일이 자동으로 유출 데이터베이스 조회 입력이 되고,
거기서 발견된 패스워드 해시가 다시 크래킹 시도의 대상이 되는 연쇄 구조가
수동 작업의 수십 시간을 대체한다.

### 2012년부터 현재까지: 오픈소스 보안 도구의 생존 방정식

SpiderFoot은 2012년에 시작해 10년 이상 활발히 유지되고 있다.
보안 도구 생태계에서 이는 드문 성취다.
Maltego 같은 상업적 대안이 시장을 지배하는 상황에서,
오픈소스 OSINT 도구가 장기 생존하는 경로는 크게 두 가지다.
하나는 상업적 대안과 직접 경쟁하기보다 접근성을 차별화하는 것,
다른 하나는 상업 버전(SpiderFoot HX)을 통해 수익을 확보해 개발을 지속하는 것이다.

SpiderFoot은 두 경로를 모두 취한다.
무료 오픈소스 버전은 보안 연구자, 학생, 소규모 팀에게 접근성을 제공하고,
상업 버전은 기업 고객에게 추가 기능과 서비스를 판매해 개발 자금을 조달한다.
이 오픈코어(open-core) 모델은 오픈소스 지속 가능성의 검증된 패턴이지만,
커뮤니티와 상업적 이해관계 사이의 긴장을 내재한다.

### GDPR 이후 OSINT 도구의 데이터 소스 변화

프로젝트 제작자가 HN 댓글에서 직접 언급했듯이, WHOIS는 GDPR 이후 OSINT 소스로서
빠르게 가치를 잃어가고 있다.[^smicallef-whois]
도메인 등록 정보에서 개인 식별 정보가 제거되면서,
OSINT 도구들이 수십 년간 의존해온 핵심 데이터 소스 하나가 사실상 무력화되었다.

이 변화는 SpiderFoot 같은 도구에 구조적 도전을 제기한다.
규제 환경이 특정 데이터 소스의 접근성을 제한할 때,
도구의 가치는 해당 소스에 대한 접근 능력이 아니라
규제를 준수하면서도 필요한 정보를 다른 경로로 수집하는 능력에 달리게 된다.
200개 이상의 모듈을 유지하는 구조는 개별 소스의 가치가 떨어질 때
전체 시스템의 가치가 유지될 수 있다는 방어적 다각화이기도 하다.

### 위협 인텔리전스와 공격 표면 매핑의 이중 용도

SpiderFoot은 두 가지 뚜렷이 다른 사용 맥락을 동시에 타깃으로 삼는다.
방어적 맥락(자신의 조직의 공격 표면을 파악)과
공격적 맥락(타깃에 대한 정보 수집).
이 이중 용도는 단순히 마케팅 선택이 아니라 도구 설계에 반영된다.

방어적 사용에서는 정기적 스캔과 결과 비교를 통한 새로운 노출 면 감지가 핵심이고,
공격적 사용에서는 타깃에 대한 포괄적 프로파일 구성이 목표다.
같은 도구가 두 역할을 하는 것은 기술적으로는 자연스럽지만,
법적·윤리적으로는 사용 맥락에 따라 완전히 다른 의미를 갖는다.
SpiderFoot HX의 기업 버전이 강조하는 “공격 표면 모니터링”은
방어적 프레이밍을 전면에 내세우는 포지셔닝 선택이다.

## 비평

### “가장 완전한” OSINT 도구라는 주장은 과장이다

프로젝트 설명은 포괄성을 강조하지만, HN 커뮤니티의 지적처럼 실제 범위는
웹 도메인 OSINT에 크게 편중되어 있다.[^KarlKemp]
역방향 이미지 검색, 정부 기록, 법원 문서, 영상 분석 같은 OSINT 도메인은
200개 모듈에도 불구하고 거의 다루어지지 않는다.

이는 단순한 누락이 아니라 도구의 본질적 한계를 드러낸다.
SpiderFoot이 잘 하는 것은 구조화된 API나 웹 크롤링으로 접근 가능한 데이터 소스를
자동화하는 것이다.
비구조화된 실세계 데이터 — 법원 기록, 부동산 등기, 기업 공시 문서 등 — 는
API가 없거나 있더라도 단순 조회 이상의 맥락 해석이 필요하다.
“200개 이상의 모듈”이라는 숫자가 실제 커버리지의 너비를 오해하게 만드는 효과가 있다.

### 자동화는 분석가의 판단을 대체하지 못한다

HN 댓글에서 경험 있는 OSINT 분석가 josephmosby는 수동 작업의 장점을 명확히 짚었다.[^josephmosby]
분석가가 직접 수행할 때는 데이터의 출처를 추적하고, 이동하면서 소스를 인용하고,
특정 사용 사례에 맞게 조정할 수 있다.
SpiderFoot이 자동화하는 것은 데이터 수집이지, 데이터 해석이 아니다.

이 구분은 SpiderFoot의 실용적 한계를 정의한다.
결과로 생성된 방대한 데이터 그래프는 출발점이지 결론이 아니다.
거짓 양성(false positive)이 다음 모듈의 입력이 되어 무관한 정보를 눈덩이처럼 불리는
“정보 오염” 문제는 자동화 파이프라인 구조에서 필연적으로 발생한다.
OSINT 도구의 자동화가 “패닉 상황에서 두 시간 안에 보고서가 필요할 때” 유용하다는 것은
역으로, 정밀한 분석이 필요한 상황에서는 수동 검증 없이 결과를 신뢰하기 어렵다는 의미다.

### 코드 품질 문제는 도구의 신뢰성에 대한 질문을 제기한다

HN 댓글에서 dr_zoidberg는 플러그인 코드에서 발견한 비효율적인 패턴을 지적했다.[^dr_zoidberg]
`dict()`를 기본값으로 사용하는 것은 Python에서 잘 알려진 변경 가능 기본 인수 문제를
내포하며, 루프를 통한 딕셔너리 업데이트는 `update()` 한 줄로 대체 가능하다.
Python 2/3 호환성 관용구가 곳곳에 남아 있다는 지적도 있었다.

200개 이상의 모듈을 단일 개발자 혹은 소규모 팀이 유지한다는 현실에서,
일관된 코드 품질을 보장하기는 구조적으로 어렵다.
보안 도구에서 코드 품질은 미적 문제가 아니다.
모듈이 예상치 못한 방식으로 상태를 공유하거나, 에러를 조용히 삼키거나,
외부 API 응답을 충분히 검증하지 않을 경우, 부정확한 결과가 분석의 토대가 된다.
이 문제는 오픈소스 기여 문화로 완화할 수 있지만,
기여된 모듈의 품질 게이팅(gating) 없이는 오히려 불균일성이 심해질 수 있다.

## 인사이트

### OSINT 자동화는 분석가의 역할을 없애는 것이 아니라 재정의한다

SpiderFoot 같은 도구의 확산이 가져오는 2차 효과는
OSINT 분석가 직군의 필요성을 줄이는 것이 아니라,
요구되는 역량의 중심을 이동시키는 것이다.
과거에 OSINT 분석가의 핵심 역량은 어떤 소스를 조회할지 알고 그 과정을 효율적으로 수행하는
절차적 지식이었다.
도구가 그 절차를 자동화하면, 경쟁 우위는 수집된 데이터를 해석하고
노이즈에서 신호를 추출하는 능력으로 이동한다.

이 전환의 아이러니는, 자동화가 오히려 고급 분석가의 가치를 높일 수 있다는 것이다.
누구나 SpiderFoot을 실행해 방대한 데이터 그래프를 얻을 수 있게 되면,
그 그래프를 의미 있는 인텔리전스로 변환하는 능력의 희소성이 증가한다.
반면 저급 반복 수집 작업은 도구에 의해 대체된다.
보안 산업에서 자동화는 역설적으로 최상위 역량의 프리미엄을 높이는 경향이 있다.

### 모든 보안 도구는 공격과 방어의 이중 속성을 갖지만, 그 경계는 제도적으로 결정된다

SpiderFoot은 보안 도구의 이중 용도 문제를 날것으로 드러낸다.
같은 도구가 자사 인프라의 취약점을 발견하는 데 쓰일 수도 있고,
타인의 정보를 무단으로 수집하는 데 쓰일 수도 있다.
기술 자체는 이 구분을 하지 않는다.

역사적으로 이 이중성은 도구 배포의 법적 환경에 의해 다루어졌다.
독일의 해킹 도구 소지 금지법, 미국의 CFAA(컴퓨터 사기 및 남용법),
EU의 사이버범죄 협약 등은 동일한 기술적 행위의 법적 지위를 맥락에 따라 다르게 정의한다.
SpiderFoot이 “위협 인텔리전스와 공격 표면 매핑”을 공식 사용 사례로 내세우는 것은
단순한 마케팅이 아니라 도구의 법적 허용성을 보호하기 위한 프레이밍이다.
이 경계가 어떻게 해석되느냐는 도구 자체가 아니라 각국의 법 집행 실무에 달려 있다.

### 오픈코어 모델은 커뮤니티와의 신뢰 계약을 필요로 한다

SpiderFoot HX라는 상업 버전의 존재는 오픈코어 전략의 전형적인 긴장을 만든다.
오픈소스 버전은 커뮤니티가 기여하고 개선하는 반면,
그 개선의 일부는 상업 버전에 통합되어 수익을 창출한다.
이 구조가 지속 가능하려면 커뮤니티가 오픈소스 버전이 충분히 유용하며,
가장 가치 있는 기능이 상업 버전 뒤에 잠겨있지 않다는 인식을 유지해야 한다.

HN 댓글의 gatewaynode 지적은 이 긴장을 드러낸다.[^gatewaynode]
더 나은 시각화, 정렬 옵션, 보고서, 차이 비교(diffing) 같은 기능들이
상업 버전에서만 제공된다면, 오픈소스 버전의 실용적 가치는 제한된다.
커뮤니티 기여자들이 자신의 기여가 결국 유료 기능의 기반이 된다는 사실을 인식하면서도
기여를 지속하는 것은, 오픈소스 프로젝트에 대한 강한 신뢰와 공유 목적의식이 있을 때만 가능하다.
이 균형이 무너지면 커뮤니티 기여는 멈추고 프로젝트는 사실상 상업 제품의 무료 티어로 전락한다.

### 장기 오픈소스 보안 프로젝트는 기술 부채와 레거시의 긴장을 필연적으로 축적한다

SpiderFoot은 2012년에 Python 2 시대에 시작되어 지금까지 활발히 개발되고 있다.
10년 이상의 연속적 개발은 기능의 축적을 의미하는 동시에,
초기 아키텍처 결정과 코딩 관행이 현재의 코드베이스 전반에 퍼져 있음을 의미한다.
HN 댓글에서 지적된 Python 2/3 호환성 관용구의 흔적이 그 단적인 예다.

이 문제는 SpiderFoot에만 국한되지 않는다.
Metasploit, Nmap, Wireshark 같은 장수 보안 도구들은 모두 같은 문제를 갖는다.
기술 부채를 해소하는 대규모 리팩토링은 기여자와 유지 관리자의 에너지를
새 기능 개발에서 돌려야 하며, 기존 사용자의 워크플로를 깨뜨릴 위험을 수반한다.
결과적으로 장수 도구들은 새로 시작하는 경쟁자와 비교할 때
“이미 작동하는 것을 왜 바꾸느냐”는 보수적 압력과
“현대적 코드베이스로 재작성해야 한다”는 진보적 요구 사이에서
항상적인 긴장 속에 놓인다.
SpiderFoot이 다음 10년을 어떻게 헤쳐 나가는지는 이 긴장을 어떻게 관리하느냐에 달려 있다.
