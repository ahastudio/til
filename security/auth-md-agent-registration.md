# auth.md — 에이전트 등록을 위한 오픈 프로토콜

<https://workos.com/auth-md>

HN 토론: <https://news.ycombinator.com/item?id=48239369> (7점, 0개 댓글)

## 소개

auth.md는 AI 에이전트가 전통적인 회원 가입 폼 없이 사용자를 대신해 서비스에 등록·인증할 수 있게 하는 오픈 프로토콜이다.
WorkOS가 설계했으며, WorkOS 계정 없이도 독립적으로 구현할 수 있는 오픈 표준이다.

프로토콜의 핵심은 도메인 루트에 호스팅되는 마크다운 파일이다.
`https://yourapp.com/auth.md` 경로에 게시하면,
에이전트가 이 파일을 읽고 지원되는 등록 방식, 제공 가능한 권한 범위(scope), 등록 절차를 파악한다.

## 등록 플로우

두 가지 등록 방식을 지원한다.

**에이전트 검증 플로우(Agent Verified Flow)**는 에이전트의 신원 공급자(identity provider)가 사용자를 보증하는 방식이다.
인간 개입 없이 에이전트가 신원 어설션(identity assertion)을 발급하고,
애플리케이션이 이를 신뢰해 등록을 완료한다.

**사용자 확인 플로우(User Claimed Flow)**는 에이전트가 사용자에게 코드를 보여주고,
사용자가 직접 서비스에 로그인해 코드를 확인하는 방식이다.
신원 공급자 없이도 구현할 수 있어 진입 장벽이 낮다.

두 플로우는 동시에 지원할 수 있으며, 에이전트가 상황에 맞는 방식을 선택한다.

## OAuth 표준 구성

프로토콜은 두 기존 표준을 조합한다.

- **Protected Resource Metadata**: 보호 리소스에 대한 메타데이터를 제공하는 표준
- **ID-JAG(Identity Assertions for Agents)**: 에이전트가 사용자 신원을 어설션하는 표준

토큰은 표준 OAuth 위에서 발급된다.
이미 API 인증에 OAuth를 쓰는 서비스라면 별도 인프라 없이 재사용할 수 있다.
발급된 토큰은 특정 사용자에게 묶이고(scoped), 단기 유효하며(short-lived), 취소 가능하다(revocable).

## 도입 현황

Cloudflare, Resend, Monday.com 등이 이미 채택했다.
WorkOS는 AuthKit을 통한 원클릭 활성화를 얼리 액세스로 제공하고 있다.

## 분석

### 에이전트 시대가 만든 새로운 인증 문제

전통적인 웹 인증은 인간이 직접 폼을 채우고, CAPTCHA를 풀고, 이메일을 확인한다고 전제한다.
AI 에이전트가 사용자를 대신해 서비스를 사용하는 시대에 이 전제가 무너진다.
에이전트는 폼을 채울 수 있지만, CAPTCHA나 이메일 확인 루프를 자연스럽게 처리하지 못한다.

이 문제에 대한 임시방편은 세 가지였다.
첫째, 사용자가 에이전트에게 직접 자격증명을 넘기는 방법 — 심각한 보안 위험이다.
둘째, 에이전트가 브라우저 자동화로 로그인하는 방법 — 서비스 약관 위반이자 불안정하다.
셋째, 에이전트마다 전용 API 키를 발급하는 방법 — 서비스별로 별도 구현이 필요해 확장성이 없다.
auth.md는 이 세 가지 임시방편을 모두 대체하는 표준화된 접근이다.

### robots.txt 유추: 기계 가독성의 반복되는 패턴

auth.md의 설계는 robots.txt와 구조적으로 동일하다.
도메인 루트에 기계가 읽을 수 있는 파일을 놓고, 기계(크롤러 또는 에이전트)가 행동 방식을 결정하게 한다.
robots.txt는 크롤러에게 “어디에 접근하지 말 것”을 알리고,
auth.md는 에이전트에게 “어떻게 등록할 것”을 알린다.

이 유추는 auth.md의 채택 가능성을 시사한다.
robots.txt는 1994년에 비공식 컨벤션으로 시작해 사실상 표준이 됐다.
기계가 읽는 도메인 루트 파일이라는 형식은 단순하고, 비용이 낮고, 네트워크 효과가 강하다.
에이전트 생태계에서 동일한 패턴이 반복될 조건이 갖춰져 있다.

### WorkOS의 전략적 포지셔닝

프로토콜이 오픈 표준이라는 점과 WorkOS가 이것을 설계했다는 점은 동시에 사실이다.
이 구조는 Auth0(Okta에 인수)나 Stripe의 결제 표준화 전략과 유사하다.
표준을 선점하면 구현 레퍼런스가 되고, 엔터프라이즈 수요가 생길 때 자연스럽게 프리미엄 서비스로 전환된다.
AuthKit을 통한 원클릭 활성화는 무료 진입점 역할을 하면서 워크플로를 WorkOS 생태계 안으로 끌어들인다.

이것은 오픈 코어(open core) 전략의 변형이다.
프로토콜은 오픈 소스이지만, 관리형 인프라와 기업용 기능은 유료다.
표준이 확산될수록 WorkOS의 포지션이 강화되는 구조다.

## 비평

### 신원 공급자 없는 에이전트의 보안 신뢰 문제가 해결되지 않는다

에이전트 검증 플로우는 “에이전트의 신원 공급자가 사용자를 보증한다”고 설명하지만,
어떤 신원 공급자를 신뢰할 것인가라는 질문에 답하지 않는다.
서비스가 ChatGPT 플러그인, Claude 에이전트, 임의의 오픈소스 에이전트를 모두 신뢰해야 하는가.
악의적인 에이전트가 신원 공급자를 자처하면 어떤 방어 메커니즘이 작동하는가.

프로토콜이 ID-JAG 표준을 사용한다고 하지만,
신원 공급자의 신뢰성을 검증하는 메커니즘이 페이지에서 명시되지 않는다.
이 공백은 프로토콜이 아직 보안 위협 모델을 충분히 발전시키지 못했다는 징후일 수 있다.
사용자 확인 플로우는 인간이 직접 확인하므로 상대적으로 안전하지만,
이것이 에이전트 자동화의 이점을 상당 부분 희석시킨다.

### 프로토콜 발견 가능성이 에이전트 생태계 발전에 의존한다

auth.md 파일이 존재해도, 에이전트가 이 파일을 먼저 확인하는 동작을 구현하지 않으면 프로토콜은 작동하지 않는다.
현재 주요 에이전트 프레임워크(LangChain, AutoGPT, Claude 에이전트 등)가 auth.md를 기본으로 확인하는지 명시되지 않는다.
서비스 측이 auth.md를 게시하더라도 에이전트 측이 이를 읽지 않으면 표준의 의미가 없다.

닭과 달걀 문제가 있다.
에이전트 개발자들이 auth.md를 지원하는 이유는 서비스들이 이것을 게시하기 때문이고,
서비스들이 auth.md를 게시하는 이유는 에이전트들이 이를 지원하기 때문이다.
Cloudflare, Resend, Monday.com의 도입이 이 문제를 부분적으로 완화하지만,
에이전트 측 지원 없이는 여전히 표준이 공중에 뜬다.

### 토큰 범위와 권한 모델의 구체성이 부족하다

“사용자에게 묶인 스코프 토큰”이라는 설명은 어느 수준의 권한을 부여하는지 명확하지 않다.
에이전트가 사용자를 대신해 데이터를 읽는 것과 쓰는 것, 또는 결제를 실행하는 것은 전혀 다른 권한 수준이다.
auth.md 파일이 “사용 가능한 스코프”를 명시한다고 하지만,
표준화된 스코프 네이밍 컨벤션이나 최소 권한 원칙의 강제 메커니즘이 제시되지 않는다.

에이전트가 과도한 스코프를 요청하거나, 서비스가 스코프를 너무 넓게 정의하면
이 프로토콜이 오히려 기존 API 키보다 더 큰 보안 취약점을 만들 수 있다.
권한 모델의 세부 명세가 공개 프로토콜 문서에서 명확하지 않다는 점은 실제 도입 전에 검토가 필요한 지점이다.

## 인사이트

### 에이전트 인증은 기계 신원(machine identity)의 새로운 레이어를 만든다

전통적인 신원 체계는 두 레이어였다.
사람 계정(human account)과 서비스 계정(service account, API 키).
auth.md는 세 번째 레이어를 제안한다 — 에이전트 계정(agent account).

이 세 번째 레이어는 기존 두 레이어와 근본적으로 다른 특성을 가진다.
에이전트는 특정 사용자를 대신해 행동하지만, 그 에이전트 자체도 신원을 가진다.
에이전트의 행동은 사용자의 의도와 일치해야 하지만, 사용자가 모든 행동을 사전에 승인하지는 않는다.
이 새로운 신원 레이어는 감사 로그, 책임 귀속, 권한 취소의 기준을 재정의한다.
에이전트가 한 행동이 잘못됐을 때, 책임은 사용자에게 있는가, 에이전트 개발사에 있는가, 서비스에 있는가.

### 마크다운 파일이 프로토콜 설명자가 되는 패턴이 확산 중이다

auth.md는 llms.txt, agents.txt, robots.txt, humans.txt의 계보를 잇는다.
도메인 루트에 기계(또는 AI)를 위한 힌트 파일을 놓는 컨벤션이 빠르게 확산되고 있다.
이 패턴이 매력적인 이유는 배포 비용이 사실상 0이고, 중앙 등록 없이도 작동하기 때문이다.

이 추세는 웹의 기계 가독성이 새로운 인프라 레이어로 성숙하고 있음을 보여준다.
1990년대 웹이 인간을 위한 HTML을 표준화했다면,
2020년대 웹은 기계를 위한 의미론적 파일 컨벤션을 표준화하는 과정에 있다.
auth.md가 성공한다면, 에이전트가 사이트와 상호작용하는 방식 전체를 선언적으로 기술하는
더 넓은 에이전트 메타데이터 레이어의 일부가 될 가능성이 있다.

### 오픈 프로토콜을 선점하는 것이 아이덴티티 시장의 새 경쟁 전선이다

AWS Cognito, Auth0, WorkOS는 기업 인증 시장에서 수렴하는 경쟁을 벌였다.
이제 에이전트 인증이라는 새로운 세그먼트가 열리면서 경쟁이 재편되고 있다.

WorkOS가 auth.md를 오픈 표준으로 공개한 것은 선점 전략이자 방어 전략이다.
표준을 먼저 제안하면 구현 레퍼런스가 되고, 상호운용성을 강조하면서도
자사 관리형 서비스(AuthKit)로 수익화하는 구조를 만들 수 있다.
경쟁사들이 auth.md 호환성을 주장하려면 결국 WorkOS의 사양을 따르거나
직접 대안 표준을 제안해야 한다.
표준 경쟁에서 첫 번째로 진지하게 제안된 것이 갖는 유리함은 상당하다.
그러나 에이전트 생태계의 주요 플레이어(Anthropic, OpenAI, Google)가 독자적 표준을 선호한다면
이 유리함은 빠르게 희석될 수 있다.
