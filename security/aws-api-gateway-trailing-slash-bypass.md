# 후행 슬래시로 AWS API Gateway 인증 우회: $12,000 버그 바운티

원문: <https://vechron.com/2026/04/i-bypassed-aws-api-gateway-auth-with-a-trailing-slash-got-12k-bounty/>

HN 토론: <https://news.ycombinator.com/item?id=48277537> (87점, 39개 댓글)

## 요약

한 보안 연구자가 핀테크 서비스의 AWS HTTP API에서 JWT 인증을 우회하는 취약점을 발견했다.
엔드포인트 URL 뒤에 슬래시 하나를 추가하는 것만으로 인증을 건너뛸 수 있었다.
책임 있는 공개(responsible disclosure)와 상세한 기술 문서화로 $12,000 버그 바운티를 받았다.

핵심 현상은 다음과 같다.

| 요청                    | 응답                     |
| ----------------------- | ------------------------ |
| `GET /v1/accounts`      | 401 Unauthorized         |
| `GET /v1/accounts/`     | 200 OK + 계정 전체 데이터 |
| `GET /v1/accounts//`    | 200 OK                   |
| `GET /v1/accounts?foo=bar` | 401 Unauthorized      |
| `GET /v1/accounts%2f`   | 404 Not Found            |

취약점의 기술적 근본 원인은 세 가지 결함이 연쇄된 결과다.
첫째, AWS HTTP API의 탐욕적 경로 매칭(greedy path matching)이다.
후행 슬래시가 붙은 경로 `/v1/accounts/`가 `/v1/accounts` 라우트의 접두사로 매칭됐다.
둘째, 경로 재작성 중 인증 컨텍스트 손실이다.
API Gateway가 `/v1/accounts/`를 `/v1/accounts`로 재작성하는 과정에서
Lambda 권한 부여자(authorizer)가 설정한 인증 컨텍스트를 잃어버렸다.
셋째, 백엔드의 누락 값 검증 부재다. 백엔드 통합 레이어는 권한 부여자가 설정하는
`userId` 값을 기대했는데, 컨텍스트가 삭제되자 `userId`가 `undefined`가 됐고,
백엔드는 이 누락 값을 검증하지 않은 채 데이터를 반환했다.

영향 범위는 심각했다. `GET /v1/accounts/`로 유효한 JWT 없이 전체 계정 데이터를 조회했다.
더 위험하게는 `POST /v1/transfers/`로 송금 기능도 우회됐다.
연구자는 시스템 계정을 사용해 $0.01 테스트 송금을 성공시켰다.

수정은 세 가지로 구성됐다. HTTP API에서 REST API로 전환해 더 엄격한 경로 매칭을 적용했다.
모든 Lambda 함수에 `userId` 검증을 추가했다.
권한 부여자 레이어 외에 서버 측 인증 검사를 별도로 구현했다.

## 분석

### 탐욕적 경로 매칭과 권한 부여자의 신뢰 경계

이 취약점의 구조는 보안 시스템의 전형적인 경계 문제다.
AWS HTTP API의 탐욕적 경로 매칭은 유연성을 위해 설계된 기능이다.
개발자가 `/v1/accounts/{proxy+}` 같은 와일드카드 라우트를 설정하면
다양한 하위 경로를 하나의 핸들러로 처리할 수 있다.
그런데 이 유연성이 인증 레이어와의 경계에서 예상치 못한 동작을 만들었다.

Lambda 권한 부여자는 들어온 요청을 검증하고 “허용(allow)” 정책을 반환했다.
그러나 API Gateway가 경로를 재작성하면서 그 허용 결정의 컨텍스트 — `userId` — 를
삭제했다. 권한 부여자는 정상적으로 동작했고, 백엔드도 나름의 로직대로 동작했다.
두 레이어 사이의 경계에서 컨텍스트가 사라진 것이다.

이것은 “A가 B를 신뢰하고, B가 C를 신뢰할 때, A→C 신뢰 경로가 자동으로 성립하는가”의 문제다.
API Gateway는 권한 부여자의 결정을 신뢰했고, 백엔드는 API Gateway가 전달한 컨텍스트를 신뢰했다.
하지만 그 신뢰 체인의 중간에서 정보가 유실됐다.

### HTTP API vs REST API의 경로 처리 차이

AWS는 두 종류의 API Gateway를 제공한다. REST API(v1)와 HTTP API(v2)다.
HTTP API는 낮은 비용과 높은 성능을 강점으로 2019년 출시됐다.
그러나 이번 사례에서 드러나듯 경로 매칭 동작이 REST API와 다르다.

REST API는 경로를 더 엄격하게 매칭한다. `/v1/accounts`와 `/v1/accounts/`는
서로 다른 리소스로 취급된다. HTTP API는 후행 슬래시를 더 유연하게 처리한다.
이 차이는 AWS 문서에 기술되어 있지만, 인증 컨텍스트 손실이라는 보안 함의로
연결된다는 점은 문서화되지 않았다. 수정 방안 중 하나가 REST API로의 마이그레이션이라는 점이
이 구조적 차이를 증명한다.

### 후행 슬래시 변형의 경계 테스트

연구자의 PoC(개념 증명)에서 눈에 띄는 점은 변형 실험의 체계성이다.
슬래시 추가(`/accounts/`), 이중 슬래시(`/accounts//`), 쿼리스트링(`/accounts?foo=bar`),
URL 인코딩된 슬래시(`/accounts%2f`)를 모두 시도했다.
이 중 `/accounts%2f`가 404를 반환한 점은 중요한 경계를 보여준다.
API Gateway가 URL 인코딩은 디코딩 전에 정규화하므로 매칭 규칙이 다르게 적용된 것이다.

이 체계적 테스트는 취약점의 정확한 메커니즘을 이해하는 데 필수적이다.
“슬래시가 문제”라는 것만으로는 부족하고, “어떤 형태의 슬래시가, 어떤 처리 단계에서,
왜 다르게 동작하는가”를 밝혀야 완전한 이해가 된다.

### 방어의 심층성(Defense in Depth) 실패

이 취약점이 최종적으로 데이터 노출로 이어진 것은 방어 계층들이
독립적으로 검증하지 않았기 때문이다. 권한 부여자가 인증을 담당하고,
백엔드는 권한 부여자를 믿었다. 그 신뢰가 깨졌을 때 백엔드에는 두 번째 방어선이 없었다.
`userId`가 `undefined`임에도 데이터를 반환한 것이 그 증거다.

방어의 심층성(Defense in Depth) 원칙은 “하나의 방어가 실패해도
다음 방어가 막아야 한다”는 것이다. 이 경우 권한 부여자가 사실상 우회됐고,
백엔드의 두 번째 방어선이 없어 데이터가 노출됐다.

## 비평

### 강점: 연쇄 실패의 정밀한 해부

이 보고서의 가장 큰 가치는 단순한 취약점 발견이 아니라 연쇄 실패 구조의 해부다.
“슬래시 하나로 인증 우회”라는 단순한 서술 뒤에 HTTP API 경로 매칭,
권한 부여자 컨텍스트 전달, 백엔드 검증 부재라는 세 레이어의 결함이 있음을
명확하게 보여준다. 이 구조적 이해가 수정 방안의 다층성으로도 이어진다.

### 한계: 재현 가능성과 일반화의 불명확성

이 취약점이 특정 회사의 구성 문제인지, AWS HTTP API 자체의 일반적 동작인지가
명확하지 않다. 모든 HTTP API + Lambda 권한 부여자 조합에서 재현되는지,
아니면 특정 라우팅 구성에서만 발생하는지가 중요한 질문이다.
독자가 자신의 시스템을 점검하려면 이 재현 조건이 명확해야 한다.

### 보완이 필요한 관점: AWS의 공식 입장

AWS가 이 동작을 버그로 분류하는지, 아니면 문서화된 설계 동작(documented behavior)으로
보는지에 대한 언급이 없다. AWS가 HTTP API의 경로 매칭 동작을 수정했는지,
또는 이것을 개발자가 처리해야 할 구성 문제로 남겨두었는지가 명확하지 않다.
AWS의 입장이 포함됐다면 독자가 플랫폼 수준 수정을 기대할 수 있는지 판단하는 데
도움이 됐을 것이다.

HN 댓글 중 가장 많이 지적된 관점은 “AWS API Gateway를 우회한 것이 아니라
잘못 구성된 회사의 설정을 우회한 것”이라는 해석이다.[^me551ah][^sammy2255]
me551ah는 “API Gateway가 아니라 회사가 API Gateway를 잘못 설정한 것을 깬 것”이라고 직접 말한다.
이것은 취약점의 귀속(attribution) 문제다. 플랫폼 동작이 예상대로였다면
이것은 구성 오류에 대한 버그 바운티인가, 플랫폼 설계 결함에 대한 바운티인가.
praptak는 넓은 시야를 제공한다.[^praptak]
“블랙리스트 우회를 위해 뭔가를 뒤에 붙이는 것은 영원한 수법이다.”
수십 년 전 쿼리 파라미터를 붙여 방화벽 필터를 우회한 경험을 공유한다.
후행 슬래시 기반 우회는 이 고전적 수법의 한 변형이다.

## 인사이트

### 보안 경계는 컴포넌트가 아니라 컴포넌트 사이에 있다

소프트웨어 보안의 전통적 접근법은 각 컴포넌트를 독립적으로 분석한다.
Lambda 권한 부여자가 올바르게 동작하는가? 백엔드 로직이 올바른가?
API Gateway 설정이 올바른가? 각 질문에 대한 답이 “예”여도 취약점이 존재할 수 있다.

이 사례는 보안 결함이 컴포넌트 내부가 아닌 컴포넌트 사이의 인터페이스에 있었음을 보여준다.
API Gateway가 경로를 재작성하면서 권한 부여자 컨텍스트를 삭제하는 동작은
각 컴포넌트를 독립적으로 검토할 때는 보이지 않는다. 통합 테스트, 더 구체적으로는
경계 조건 통합 테스트가 이런 종류의 취약점을 발견하는 데 필요하다.

이 패턴은 마이크로서비스 아키텍처에서 더 심각해진다. 서비스가 많을수록
인터페이스의 수도 늘어나고, 각 인터페이스에서 이런 종류의 컨텍스트 손실이
발생할 가능성도 높아진다. “컴포넌트가 각자 올바르게 동작한다”는 것과
“시스템 전체가 올바르게 동작한다”는 것은 근본적으로 다른 주장이다.
분산 시스템의 보안 검토는 반드시 통합 관점을 포함해야 한다.

### “없으면 안 된다”가 아닌 “있어야 한다”로 설계하라

이 취약점에서 백엔드의 결정적 실수는 `userId`가 `undefined`일 때 데이터를 반환한 것이다.
백엔드는 “권한 부여자가 이미 검증했으므로 `userId`는 반드시 있다”고 가정했다.
이것이 “부재가 불가능하다(cannot be absent)”는 가정이다.

더 안전한 설계 원칙은 반대다. “`userId`가 존재하고 유효한 경우에만 진행한다.”
이것이 “존재가 확인돼야 진행한다(must be present)”는 원칙이다.
두 원칙의 차이는 미묘하지만 결과는 크다. 전자는 외부 보장에 의존하고,
후자는 자체 보장을 요구한다. 외부 보장이 실패했을 때 전자는 무너지고 후자는 버틴다.

이 원칙은 제로 트러스트(zero-trust) 보안 모델의 핵심이기도 하다.
“신뢰하되 검증하라(trust but verify)”가 아니라 “검증하지 않으면 신뢰하지 마라(never trust, always verify)”.
API Gateway가 올바른 `userId`를 전달한다고 신뢰하지 말고, Lambda 함수 자체에서
`userId`의 존재와 형식을 독립적으로 검증해야 한다. 이 추가 검증 비용은 적지만,
우회 가능성을 원천 차단한다.

### 단순함이 최고의 방어다: 경계 조건의 명시적 처리

후행 슬래시는 웹 개발에서 가장 오래된 엣지 케이스 중 하나다.
URL `/path`와 `/path/`가 같은 리소스인가 다른 리소스인가는
HTTP 명세에서도 애매하고(RFC 3986은 이 두 URI를 다른 것으로 취급),
웹 프레임워크마다 처리 방식이 다르다. Django는 `APPEND_SLASH` 설정으로 리다이렉트하고,
Express는 기본적으로 다른 경로로 취급하며, Rails는 무시한다.

이 불일치가 오랫동안 다양한 보안 문제의 원천이었다.
경로 순회 공격(path traversal), 캐시 포이즈닝, 인증 우회가 후행 슬래시나
이중 슬래시를 활용한 사례들이 있다.
AWS HTTP API의 이번 사례는 이 고전적 문제의 새로운 변형이다.

방어 전략은 단순할수록 좋다. 인증이 필요한 엔드포인트에서는
경로를 정규화(normalize)한 후 인증을 적용하거나, 경로 변형을 허용하지 않거나,
허용하더라도 인증 컨텍스트가 정규화 이후에도 유지되도록 보장해야 한다.
가장 단순한 방어는 인증 레이어와 통합 레이어가 같은 경로 표현을 사용하게 만드는 것이다.
이 단순함을 유지하는 것이 복잡한 탐지 로직을 추가하는 것보다 훨씬 효과적이다.
