# Goodbye innerHTML, Hello setHTML — Firefox 148의 Sanitizer API

원문: <https://hacks.mozilla.org/2026/02/goodbye-innerhtml-hello-sethtml-stronger-xss-protection-in-firefox-148/>

## 요약

Firefox 148이 표준화된 Sanitizer API를 최초로 출시했다.
이 API의 핵심 메서드 `setHTML()`은 신뢰할 수 없는 HTML을 DOM에 삽입하기 전에
안전하게 정제(sanitize)하는 방법으로, XSS(Cross-Site Scripting) 공격을 방지하기 위한
대안이다. XSS는 약 10년 가까이 웹 취약점 상위 3위를 벗어나지 못했다.

기존의 `innerHTML`을 사용하는 패턴은 개발자가 직접 입력값을 검증하지 않으면
스크립트 삽입이 가능하다. `setHTML()`은 이 문제를 브라우저 수준에서 해결한다.

```javascript
document.body.setHTML(`<h1>Hello my name is <img src="x" onclick="alert('XSS')">`);
// 결과: <h1>Hello my name is</h1>
// onclick 속성과 img 요소가 자동으로 제거됨
```

기본 설정은 잠재적으로 위험한 요소와 속성을 자동으로 제거하며,
필요에 따라 커스텀 설정으로 허용할 요소와 속성을 명시적으로 지정할 수도 있다.

Sanitizer API는 Trusted Types와 결합하면 더 강력해진다.
Trusted Types는 HTML 파싱과 삽입에 대한 제어권을 중앙 집중화한다.
`setHTML()`이 채택되면 복잡한 커스텀 정책 없이도 Trusted Types 강제 적용이 쉬워진다.
Mozilla는 다른 브라우저들도 곧 동일한 기능을 출시할 것으로 기대한다.

## 분석

### innerHTML의 설계 결함과 20년간의 회피

`innerHTML`은 HTML 4 시대부터 사용된 비표준 프로퍼티가 사실상의 표준이 된 사례다.
브라우저들이 경쟁적으로 구현하고 개발자들이 편리함 때문에 광범위하게 사용하면서,
DOM 조작의 사실상 표준으로 굳어졌다.
문제는 `innerHTML`이 HTML을 파싱해 즉시 DOM에 삽입하면서
스크립트 실행 가능한 콘텐츠도 그대로 처리한다는 설계 결함이다.

지난 20년간 업계는 이 결함을 다양한 방식으로 “회피”해왔다.
DOMPurify 같은 서드파티 라이브러리, CSP 헤더 설정, 프레임워크 수준의 자동 이스케이핑이
대표적이다. 그러나 이 접근들은 모두 `innerHTML` 자체를 고치지 않고 주변에
방어막을 치는 방식이다. `setHTML()`은 처음으로 삽입 메커니즘 자체에 정제를 내장한다.

### CSP의 한계와 Sanitizer API의 포지셔닝

Mozilla는 2009년 CSP(Content Security Policy)를 선도했다.
CSP는 브라우저가 어떤 리소스를 로드할 수 있는지 제한하는 강력한 메커니즘이지만,
“충분한 채택을 얻지 못했다”고 스스로 인정한다.
이유는 명확하다. CSP는 기존 사이트에 상당한 아키텍처 변경을 요구하고,
보안 전문가의 지속적 검토가 필요하기 때문이다.

Sanitizer API는 이 교훈을 반영한 접근이다.
“전담 보안 팀이나 대규모 구현 변경 없이 XSS를 방지”할 수 있다는 목표는
CSP가 실패한 지점을 정확히 겨냥한다. `innerHTML`을 `setHTML()`로 교체하는 것은
아키텍처 변경이 아니라 메서드 치환이다. 진입 장벽이 낮다.

### Trusted Types와의 시너지

Trusted Types는 DOM 조작 지점에서 신뢰할 수 있는 타입 객체만 허용하도록 강제하는 API다.
`setHTML()`이 반환하는 정제된 HTML은 Trusted Types가 요구하는 신뢰 타입 조건을 만족한다.
두 API의 조합은 “방어 심화(defense in depth)” 패턴을 코드 수준에서 구현한다.

Sanitizer API가 먼저 위험한 콘텐츠를 제거하고,
Trusted Types가 정제되지 않은 문자열이 DOM에 도달하는 경로 자체를 차단한다.
이 조합이 조직 전체에 적용되면 XSS 취약점이 발생할 수 있는 경로가 구조적으로 줄어든다.

## 비평

### 강점

진입 장벽이 낮다는 점이 가장 큰 강점이다.
기존 코드에서 `innerHTML =` 를 `setHTML()` 로 바꾸는 것은 최소한의 변경이다.
라이브러리 의존성 없이 브라우저가 직접 제공하므로 번들 크기도 늘어나지 않는다.
“안전한 기본값(secure by default)” 원칙을 API 설계에 구현한 드문 사례이기도 하다.

브라우저 네이티브 솔루션이라는 점도 중요하다.
DOMPurify 같은 라이브러리는 HTML 파싱 로직을 JavaScript로 재구현한다.
브라우저의 파서와 라이브러리의 파서가 서로 다르게 동작하는 파싱 차이(parser differential)가
보안 취약점의 원인이 될 수 있다. `setHTML()`은 브라우저가 자체 파서로 직접 정제하므로
이 위험이 없다.

### 약점

“Firefox 148이 최초”라는 표현은 실제 배포 가능성을 제한한다.
다른 브라우저가 지원하지 않는 동안에는 기능 감지와 폴백을 구현해야 하며,
이 과도기에는 오히려 코드가 복잡해진다.
“곧 다른 브라우저들도 따를 것”이라는 낙관적 전망이 실현되기까지의 기간이
현실적 채택의 병목이다.

커스텀 설정이 얼마나 직관적인가에 대한 정보도 부족하다.
기본 설정이 너무 엄격한 경우 개발자들이 커스텀 설정을 작성하게 되는데,
이 과정에서 실수가 발생하면 의도치 않게 위험한 콘텐츠를 허용할 수 있다.
커스텀 설정의 복잡도가 곧 DOMPurify를 사용하는 것과 비슷해진다면
도입 유인이 줄어든다.

### 기존 라이브러리와의 공존 문제

DOMPurify는 이미 광범위하게 사용되고 있다.
Sanitizer API가 표준화되더라도 기존 코드베이스의 DOMPurify를 제거하는 것은
별도의 마이그레이션 작업이며, 보안 관련 라이브러리를 교체하는 것은
테스트 부담이 크다.
새 프로젝트에서는 빠르게 채택되겠지만, 레거시 코드베이스에서는 오랫동안 공존할 것이다.

## 인사이트

### 보안의 인프라화: 방어 로직이 언어/플랫폼 계층으로 내려가는 흐름

소프트웨어 보안의 역사는 보안 로직이 점점 낮은 계층으로 내려가는 과정이다.
초기 웹에서 XSS 방어는 전적으로 애플리케이션 개발자의 책임이었다.
서버 측에서 HTML 이스케이핑을 직접 구현해야 했고, 놓치면 취약점이 생겼다.
그 다음에 프레임워크들이 이 책임을 흡수했다.
React의 JSX가 기본으로 이스케이핑하고, Django의 템플릿이 자동으로 이스케이핑한다.
이제 브라우저가 그 책임을 한 단계 더 아래로 가져가고 있다.

이 흐름의 의미는 단순한 편의성 향상이 아니다.
보안 로직이 플랫폼 계층에 들어갈수록, 개발자가 “실수로 비활성화”하기 어려워진다.
프레임워크의 이스케이핑은 `dangerouslySetInnerHTML`처럼 의도적으로 우회할 수 있다.
브라우저의 Sanitizer API도 우회 방법이 존재하겠지만,
우회하려면 명시적이고 의도적인 코드가 필요하도록 설계되었다.
보안의 기본값이 불안전에서 안전으로 이동하는 것이 이 흐름의 핵심이다.

역사적 유추가 있다.
메모리 안전성은 C/C++에서 개발자의 책임이었다가, Rust가 언어 수준에서 강제하기 시작했다.
타입 안전성은 동적 언어에서 옵션이었다가, TypeScript가 생태계 수준에서 확산시켰다.
XSS 방어가 브라우저 네이티브 API로 들어가는 것은 이 흐름의 웹 버전이다.
다음 단계는 아마도 브라우저가 Trusted Types를 기본값으로 강제하는 것일 수 있다.

### 파서 차별성 공격(Parser Differential Attack)과 네이티브 솔루션의 우위

XSS 방어의 가장 교묘한 공격 벡터 중 하나는 파서 차별성이다.
HTML 파서는 사양보다 실제 브라우저 구현이 더 관대하고,
브라우저마다 엣지 케이스 처리가 다르다.
DOMPurify는 JavaScript로 HTML 파싱을 재구현하는데,
이 구현과 실제 브라우저 파서가 특정 입력에 대해 다르게 반응하면
DOMPurify는 안전하다고 판단했지만 브라우저는 실행하는 상황이 생긴다.

실제로 이 유형의 우회 공격이 DOMPurify를 포함한 여러 정제 라이브러리에서
발견된 적이 있다. 정제 라이브러리가 아무리 정교해도 브라우저의 내부 파서를
완벽하게 모델링하는 것은 불가능하다.

`setHTML()`이 브라우저의 자체 파서로 정제한다는 것은 이 공격 표면이 사라진다는 의미다.
정제에 사용하는 파서와 DOM 삽입에 사용하는 파서가 동일하기 때문에,
파서 차이로 인한 우회가 구조적으로 불가능하다.
이것이 네이티브 솔루션이 라이브러리 솔루션보다 근본적으로 우월한 이유다.
편의성의 문제가 아니라 신뢰 모델의 문제다.

### “보안 기능의 채택률”이라는 오래된 문제

Mozilla가 2009년 CSP를 발명하고 17년이 지난 지금도 “충분한 채택을 얻지 못했다”는
자기 비판적 평가는 보안 업계의 고질적 문제를 드러낸다.
아무리 좋은 보안 메커니즘도 채택되지 않으면 의미가 없다.
CSP는 강력하지만 채택률이 낮고, 그보다 약한 메커니즘들이 더 널리 사용된다.

Sanitizer API의 설계 철학은 이 교훈을 반영한다.
“전담 보안 팀이나 대규모 구현 변경 없이”라는 문구는 마케팅 문구가 아니라
채택률에 대한 솔직한 인식이다. 진입 장벽이 낮을수록 채택률이 높아지고,
채택률이 높을수록 실질적 보안이 개선된다.
완벽하지만 복잡한 솔루션보다 충분히 좋고 간단한 솔루션이 실제 보안을 높인다.

이 통찰은 보안을 넘어 소프트웨어 설계 전반에 적용된다.
사용하기 어려운 올바른 방법보다 사용하기 쉬운 올바른 방법이 더 많은 올바른 코드를 만든다.
API 설계에서 “pit of success(성공의 함정, 올바른 방법이 가장 쉬운 방법인 상태)”는
보안 영역에서 특히 중요하다. `setHTML()`이 `innerHTML`보다 타이핑이 더 많지도 않고,
기본적으로 안전하다면, 개발자는 추가 노력 없이 안전한 선택을 하게 된다.
이것이 보안 기능 설계의 이상적 형태다.
