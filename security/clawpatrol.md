# Claw Patrol — 에이전트를 위한 보안 방화벽

<https://clawpatrol.dev/>

<https://github.com/denoland/clawpatrol>

## 소개

Claw Patrol은 Deno 팀이 공개한 “에이전트(agent)를 위한 보안 방화벽(security
firewall)”이다.
LLM 기반 에이전트와 운영(prod) 시스템 사이에 자리잡아, 에이전트가 발생시키는
트래픽을 와이어(wire) 수준에서 파싱(parse)하고, HCL로 작성된 규칙에 따라 각
요청을 차단하거나 사람의 승인 단계로 보류한다.

차단의 단위는 “API 호출”이 아니라 “프로토콜 의미”다.
Postgres·ClickHouse라면 SQL 동사(verb)와 테이블 이름, Kubernetes라면
리소스(resource)·동사·네임스페이스, HTTP라면 메서드·경로·헤더·바디까지
프로토콜별로 게이트웨이가 추출한 사실(fact)을 CEL(Common Expression
Language) 식으로 평가한다.
예시 규칙은 `k8s.resource == 'secrets'`인 호출에 `deny`를 내려, 시크릿 값이
에이전트 경로로 클러스터를 떠나지 못하게 막는다.

## 명세

### 규칙(rule) 형식

규칙은 HCL 블록이며 `endpoint`, `condition`, `verdict`, `reason` 네 필드를
가진다.
`condition`은 게이트웨이가 트래픽에서 뽑아낸 프로토콜 사실에 대한 CEL 식이고,
`verdict`는 `deny`처럼 단순 차단부터 사람 승인 보류까지 확장될 수 있다.
운영 환경의 실제 규칙이 그대로 README에 예시로 박혀 있다는 점에서, “차단
정책 자체가 코드로 리뷰되는 산출물”이라는 입장을 분명히 한다.

### 추출되는 와이어 사실(wire-level fact)

- Postgres / ClickHouse: SQL verb, table name
- Kubernetes: resource, verb, namespace
- HTTP: method, path, headers, body

즉 Claw Patrol은 “HTTP 프록시”가 아니라 “프로토콜 게이트웨이”다.
각 백엔드의 의미론을 직접 파싱해야 하므로, 지원되는 프로토콜의 수가 곧
제품의 커버리지가 된다.

## CLI

세 가지 배포 형태가 제공된다.

```text
clawpatrol gateway config.hcl   # 프록시 자체를 띄움
clawpatrol join <gateway-url>   # 게이트웨이에 호스트를 합류시킴
clawpatrol run claude           # 특정 에이전트 프로세스 트리만 감쌈
```

- `clawpatrol run`은 Linux에서는 네트워크 네임스페이스(netns), macOS에서는
  NetworkExtension을 이용해 “그 프로세스만” 게이트웨이로 보낸다.
- `clawpatrol join`은 WireGuard 터널로 호스트 전체 트래픽을 라우팅한다.
- `clawpatrol gateway`는 HCL 설정을 로드하고 WireGuard 또는 Tailscale로
  들어오는 클라이언트를 받는 단일 바이너리다.

설치는 `curl -fsSL https://clawpatrol.dev/install.sh | sh`로 한 줄이며,
소스 빌드는 `make`(Go·Node.js 필요)다.
라이선스는 MIT다.

## 분석

### 위치 선정 — 호스트가 아니라 네트워크 경계

대다수의 “에이전트 안전장치”는 두 곳에 산다.
하나는 모델 안쪽의 가드레일(guardrail) 프롬프트와 분류기, 다른 하나는 호스트
위의 샌드박스(seccomp, Capsicum, 컨테이너).
Claw Patrol은 셋째 위치, 즉 “에이전트가 외부 시스템에 도달하기 직전의
네트워크 게이트웨이”를 선택했다.
모델 출력이 무엇이든, 셸 명령이 무엇이든, 결국 운영 시스템에 닿으려면 와이어
위에 프로토콜로 표현되어야 하기 때문에, 이 위치는 “관찰 가능성과 강제력이
동시에 성립하는 마지막 지점”이다.

### CEL + HCL 조합의 의미

규칙 본문은 HCL로 선언하고, 조건식만 CEL로 작성하는 분리는 Kubernetes의
ValidatingAdmissionPolicy, GCP IAM Conditions와 같은 계보다.
구조(어디에 붙는 규칙인가)는 정적인 DSL로, 술어(언제 적용되는가)는 표현력이
있는 식 언어로 가른다.
이 패턴은 “정책을 코드로 리뷰 가능하게 유지하면서도 임의의 분기를 허용한다”
는, 정책 엔진 설계에서 이미 검증된 절충안이다.

### 배포 모델의 세 갈래

`run`·`join`·`gateway`는 단순히 “편의 옵션”이 아니라 “신뢰 경계(trust
boundary)를 어디에 그을 것인가”라는 질문에 대한 세 가지 답이다.
- `run`은 신뢰 단위가 “프로세스 하나”다 — 같은 호스트의 다른 도구는 제약을
  받지 않는다.
- `join`은 신뢰 단위가 “호스트 전체”다 — 어떤 프로세스가 띄워지든 게이트를
  통과해야 한다.
- `gateway`는 신뢰 단위가 “네트워크 세그먼트”다 — 여러 호스트가 한 게이트
  뒤에 모인다.

이 셋은 운영 조직이 에이전트를 “도구로 빌려 쓰는 단계 → 워크스테이션에
상주시키는 단계 → 자체 클러스터로 운영하는 단계”로 성숙해가는 경로와 거의
일대일로 대응한다.

## 비평

### 강점 — 솔직하고 좁은 약속

README가 약속하는 범위는 매우 좁다.
“에이전트 트래픽을 와이어에서 파싱하고, HCL 규칙으로 게이트한다.” 그게
전부다.
프롬프트 인젝션 탐지, 모델 정합성, 에이전트 행동 분석 같은 거대한 단어가
없다.
보안 도구의 가장 흔한 실패는 “모든 위협을 막는다”는 약속인데, Claw Patrol은
의도적으로 그 길을 피한다.
정책 엔진 + 프로토콜 게이트웨이라는 잘 알려진 두 부품을 “에이전트 트래픽”
이라는 새 컨텍스트에 정직하게 합친 제품이다.

### 약점 — 와이어 파싱의 가시성 한계

치명적인 한계는 “TLS로 암호화되어 흐르는 호출”이다.
Postgres·Kubernetes는 보통 mTLS, HTTP는 거의 항상 TLS다.
이 게이트웨이가 “리소스 이름”을 본다는 것은, 사실상 클라이언트가 TLS를 종단
(terminate)하도록 강제하거나, 게이트웨이가 MITM 인증서를 갖고 복호화한다는
뜻이다.
README는 이 트레이드오프를 명시하지 않는다 — 운영자는 “데이터 평면의 평문
가시성을 제3의 컴포넌트에 양도한다”는 결정을 사실상 강요받는다.
이는 작은 문제가 아니라, “보안을 위해 새 보안 경계를 만든다”는 고전적
재귀(recursion) 문제다.

### 약점 — 사람 승인의 운영 비용

`pause until a human approves it`는 데모에서는 멋지지만, 운영에서는 가장
비싼 부분이다.
누가 승인 채널을 운영하고, SLA는 얼마이고, 승인자가 자리에 없을 때 에이전트
는 어떻게 행동해야 하는가?
이 모든 운영 질문이 “verdict”라는 한 단어로 압축되어 있다.
실제로 효과를 보려면 PagerDuty·Slack·Teams 등과의 통합, 승인 권한의 RBAC,
승인 로그의 무결성 확보 등 또 한 켤레의 시스템이 필요하다.
Claw Patrol은 정책 엔진을 제공할 뿐, 휴먼-인-더-루프(human-in-the-loop)
워크플로우 그 자체를 제공하지는 않는다.

### 빈 자리 — 에이전트 ID와 출처(provenance)

룰은 “어떤 호출이 일어났는가”에만 반응한다.
“누가(어떤 에이전트 세션이, 어떤 사용자 권한 위임을 받아) 그 호출을
일으켰는가”는 와이어에 자연스레 드러나지 않는다.
에이전트 보안의 진짜 어려운 문제 — 위임된 권한의 추적, 세션 간 데이터 누설,
다중 에이전트 협업의 책임 분리 — 는 프로토콜 게이트 한 켜로는 풀리지
않는다.

## 인사이트

### 에이전트 보안의 무게중심은 “출력 검사”에서 “행동 게이팅”으로 옮겨가고 있다

지난 2년의 에이전트 안전 담론은 압도적으로 “모델의 출력을 어떻게 더 안전하게
만들 것인가”였다.
RLHF, 가드레일 분류기, 시스템 프롬프트 강화, 모더레이션 API — 모두 “토큰이
생성된 다음, 그것이 나쁜지 판정한다”의 변주다.
Claw Patrol은 이 무게중심을 한 칸 뒤로 옮긴다.
모델이 무엇을 말했는가는 더 이상 핵심이 아니다.
그 말이 결국 운영 시스템에 어떤 동사로 도달하는가, 즉 “행동”만이 평가
대상이다.

이 전환은 단순한 구현 디테일이 아니라 위협 모델의 변화다.
“악의적 출력”이 아니라 “정상 출력이 누적되어 만들어내는 위험한 행동 패턴”이
일급 시민이 된다.
프롬프트 인젝션을 100% 막을 수 없다는 사실을 받아들이고, 그 다음 단의 방어선
을 와이어 수준에 설치한다.

흥미로운 점은, 이 자세가 “신뢰할 수 없는 입력은 신뢰할 수 없는 출력을
낳는다, 그러나 신뢰할 수 없는 출력도 신뢰할 수 있는 부수효과만 일으키도록
좁힐 수 있다”는, 능력 기반 보안(capability-based security)의 오래된 원칙과
정확히 같은 구조라는 점이다.
모델 시대에 가장 신뢰받는 방어선이 1970년대 OS 보안 원리의 부활이라는 사실은
시사적이다.

### Deno가 이 도구를 내놓은 사실 자체가 가장 큰 신호다

`denoland` 조직이 새 패키지 매니저나 런타임이 아닌 “에이전트 방화벽”을
공개했다는 사실은, 런타임 벤더가 보는 시장 방향에 대한 거의 가장 정직한
신호다.
Deno는 등장 때부터 “기본적으로 안전한 런타임(permissioned runtime)”을 정체성
으로 삼아왔다 — `--allow-net`, `--allow-read`로 권한을 명시적으로 켜는
모델이다.
그런 회사가 “런타임 권한 모델로는 부족하다, 외부에 별도의 게이트가 필요하다”
는 결론에 도달했다는 것은, 에이전트 워크로드가 기존의 “한 프로세스 = 한 권한
세트” 모델에 들어가지 않는다는 자백에 가깝다.

에이전트는 한 세션 안에서도 권한의 컨텍스트가 동적으로 바뀐다.
처음에는 “리포 읽기”였다가, 다섯 턴 뒤에는 “프로덕션 DB 쓰기”로 미끄러진다.
이 미끄러짐은 단일 프로세스 안에서 일어나기 때문에, 프로세스 단위 권한 모델
로는 잡히지 않는다.
유일하게 잡히는 곳이 “외부 시스템에 도달하기 직전의 네트워크”다.

즉 Claw Patrol은 Deno라는 회사가 자신의 핵심 베팅(런타임 권한 모델)을
스스로 보완해야 한다고 인정한 제품이다.
이런 종류의 자기 부정은 보통 시장에 늦지 않다 — 같은 깨달음이 다른 런타임
벤더, 클라우드 벤더, 보안 벤더에서 동시다발로 일어날 가능성이 높다.

### HCL + CEL은 “정책 엔진의 표준 페어”가 되어가고 있다

10년 전 정책을 작성한다고 하면 사람들은 XACML, OPA Rego, 자체 DSL을
떠올렸다.
지금은 점점 더 많은 새 도구가 HCL(구조) + CEL(술어)이라는 같은 페어를
선택한다 — Kubernetes의 ValidatingAdmissionPolicy, GCP IAM Conditions,
HashiCorp Sentinel의 후속 흐름, 그리고 Claw Patrol.

이 수렴에는 이유가 있다.
HCL은 “사람이 읽기 좋은 구조”에 강하고 CEL은 “타입 안전한 식 평가”에 강하다.
Rego가 갖고 있던 “표현력은 강하지만 학습 곡선이 가파르다”는 약점을, HCL+CEL
은 “구조와 표현을 분리”함으로써 우회한다.
구조는 정적 도구로 검증되고, 표현은 샌드박스 안에서 결정적으로 평가된다.

장기적으로 이는 “정책 코드”가 “애플리케이션 코드”와 같은 도구체인에 들어간
다는 뜻이다.
정책에 대한 PR 리뷰, 정책의 테스트 커버리지, 정책의 단계적 롤아웃 — 인프라
스트럭처-as-코드가 지난 10년에 걸쳐 인프라에 가져온 변화가, 다음 10년에
정책에 일어날 가능성이 크다.
Claw Patrol을 평가할 때 정말 봐야 할 것은 “오늘 막을 수 있는 호출의 종류”가
아니라, “조직이 이 규칙들을 어떻게 리뷰·테스트·배포하는가”라는 운영 루틴
이다.

### “Approve before reach”의 진짜 적은 지연이 아니라 피로

`pause until a human approves`는 보안 도구가 가장 자주 약속하는 기능이지만,
실패의 90%는 같은 곳에서 일어난다 — 알림 피로(alert fatigue).
승인 요청이 시간당 한 번 오면 사람들은 본다.
하루에 100번 오면 본능적으로 “Approve” 버튼을 누르는 근육이 생긴다.

에이전트는 이 실패 모드를 구조적으로 가속한다.
사람은 1초에 한 번 행동하지 않지만 에이전트는 그렇게 한다.
“민감한 호출만 사람에게 보낸다”는 정책은, 에이전트의 행동 빈도가 충분히
높아지면 결국 “민감한 호출도 사람이 자동으로 승인한다”로 수렴한다.

따라서 Claw Patrol 같은 도구의 진짜 평가 기준은 “deny 룰의 정밀도”가 아니라
“승인 요청 자체의 양을 시간에 따라 어떻게 감쇠시키는가”가 되어야 한다.
초기에는 보수적으로 잡고, 운영 신호(false positive 통계, 승인 응답 시간,
승인자 분산)를 보고 자동으로 룰을 더 좁혀나가는, 일종의 폐쇄 루프(closed
loop)가 필요하다.
이 루프가 없는 휴먼-인-더-루프 시스템은, 도입 첫 달에는 안전하고 셋째 달에는
무력하다.
정책 엔진의 다음 경쟁축은 표현력이 아니라, 이 피로 곡선을 어떻게 관리하는가
가 될 것이다.
