# Copilot CLI에 전용 보안 리뷰 명령 /security-review 추가

원문: <https://github.blog/changelog/2026-06-10-dedicated-security-review-command-now-available-in-copilot-cli/>

## 요약

GitHub Copilot CLI에 `/security-review` 슬래시 커맨드가 실험 기능(Experimental)으로 Public Preview에 추가됐다.
개발자가 터미널에서 코드를 프로덕션에 커밋하기 전, 로컬 코드 변경사항에 대한 보안 분석을 실행할 수 있다.

이 명령은 취약점을 심각도(severity)와 신뢰도(confidence) 기준으로 평가한 결과를 제공하며,
터미널을 벗어나지 않고 적용할 수 있는 실질적인 수정 권고안을 함께 제시한다.

스캔이 탐지하는 취약점 범주는 13가지다.
인젝션 취약점, XSS(Cross-site Scripting), 접근 제어 오류, 경로 순회(Path Traversal),
SSRF(Server-side Request Forgery), 역직렬화 취약점 및 프로토타입 오염,
약한 암호화 구현, 하드코딩된 자격증명, 민감 데이터 노출, 인증 및 CORS 실패,
보안 설정 오류, 고정되지 않은 의존성을 포함한 공급망 위험,
그리고 LLM 통합 애플리케이션을 대상으로 한 크로스 프롬프트 인젝션 공격이다.

이 도구는 GitHub의 기존 코드 스캐닝(Code Scanning), Dependabot, 시크릿 스캐닝과는 독립적으로 동작한다.
공지에서는 기존 도구들과 상호 보완적인 “경량(lightweight) 온디맨드 리뷰” 방식으로 설명한다.

사용하려면 Copilot CLI에서 실험 모드를 활성화한 후 `/security-review`를 실행해 현재 프로젝트 변경사항을 스캔한다.

## 분석

### 개발자 터미널에 보안 리뷰를 통합하는 것이 갖는 의미

보안 도구는 오랫동안 개발 워크플로우와 별도의 파이프라인에 존재했다.
SAST(정적 분석 도구)는 CI/CD 단계에서 실행되고, 취약점 스캐너는 빌드 이후에 돌아가며,
코드 리뷰는 PR을 열어야 시작됐다.
이 구조에서 보안 피드백은 항상 지연됐다.

`/security-review`가 “커밋 전” 로컬 변경사항을 분석한다는 것은 이 피드백 루프를 크게 단축한다.
개발자가 코드를 작성하는 시점과 보안 피드백을 받는 시점 사이의 거리가 줄어들수록
수정 비용이 낮아진다는 것은 보안 공학의 기본 원칙이다.
Shift Left Security의 구체적 구현이 터미널 슬래시 커맨드 형태로 나타난 것이다.

이 접근은 보안을 개발자의 일상 도구 안으로 가져오려는 전략이기도 하다.
보안 팀이 아닌 개발자가 자신의 코드를 직접 스캔하고 수정할 수 있는 환경을 만든다.
보안 전문가 없이 초기 취약점을 거르는 1차 방어선을 개발자 스스로 구성하게 되는 것이다.

### 크로스 프롬프트 인젝션이 취약점 목록에 포함된 것의 의미

13개 취약점 범주 중 마지막 항목인 “크로스 프롬프트 인젝션 공격(Cross-prompt Injection)”은
나머지 12개와 성격이 다르다.
인젝션, XSS, SSRF는 수십 년간 알려진 웹 보안 취약점이지만,
크로스 프롬프트 인젝션은 LLM 통합 애플리케이션에서 새롭게 등장한 공격 벡터다.

이 범주의 포함은 Copilot CLI가 이미 LLM 기반 애플리케이션을 작성하는 개발자를 주된 사용자로 상정한다는 신호다.
AI 코딩 도구로 AI 기반 앱의 보안 취약점을 분석하는 구조다.
LLM 통합 앱이 표준적인 개발 산출물이 되면서, 그 취약점도 표준적인 보안 점검 목록에 들어가야 한다는 인식의 반영이다.

크로스 프롬프트 인젝션 탐지가 정적 분석으로 얼마나 효과적으로 이루어질 수 있는가는 열린 질문이다.
이 공격의 많은 변형은 런타임에 외부 입력과 결합할 때 발생하며, 정적 코드만으로는 탐지 범위가 제한적이다.
그러나 알려진 취약 패턴(예: 사용자 입력을 시스템 프롬프트에 직접 삽입하는 코드)은 정적으로 탐지 가능하다.

### 기존 GitHub 보안 도구들과의 관계

공지는 `/security-review`가 Code Scanning, Dependabot, 시크릿 스캐닝과 독립적이며 상호 보완적이라고 설명한다.
이 설명은 기존 도구들을 대체하지 않는다는 메시지지만, 동시에 중복 가능성을 인정하는 것이기도 하다.

각 도구의 역할은 다르다.
Code Scanning은 코드베이스 전체를 정적으로 분석하고 CI에서 실행된다.
Dependabot은 알려진 취약점이 있는 의존성을 추적한다.
시크릿 스캐닝은 커밋된 자격증명을 탐지한다.
`/security-review`는 현재 변경사항에 집중하며 터미널에서 즉각적인 피드백을 제공한다.

이 분업은 의도적이지만 불완전하다.
개발자 입장에서 “어떤 도구를 언제 써야 하는가”는 점점 복잡해진다.
보안 도구의 분화가 보안 관행의 일관성을 해치는 역설적 상황이 생길 수 있다.

## 비평

### “경량 온디맨드”라는 포지셔닝이 실제 사용 패턴과 맞지 않을 수 있다

공지는 `/security-review`를 “경량(lightweight)”으로 설명한다.
그러나 13개 취약점 범주를 LLM 기반으로 분석하는 작업이 실제로 얼마나 빠른지는 공지에서 알 수 없다.

보안 리뷰 도구의 채택에서 가장 큰 장벽 중 하나는 속도다.
개발자가 코드를 작성하는 흐름 중에 수십 초 이상 기다려야 한다면, 도구를 건너뛰는 것이 기본 행동이 된다.
“경량”이라는 표현이 응답 시간을 보장하지 않는다면, 이것은 마케팅 언어에 가깝다.

CI/CD에서 실행되는 SAST 도구들이 느리다는 이유로 개발자들이 로컬에서 우선 실행하지 않는 현상은 이미 알려진 문제다.
`/security-review`가 이 문제를 해결하려면 속도가 핵심이며, 그 속도에 대한 정보가 공지에 없다.

### 심각도와 신뢰도 평가의 기준이 투명하지 않다

`/security-review`는 취약점을 심각도(severity)와 신뢰도(confidence)로 평가한다.
그러나 이 평가가 어떤 기준으로 이루어지는지, LLM 기반 분석에서 이 수치가 얼마나 일관적인지는 설명되지 않는다.

보안 취약점 평가의 표준 체계로 CVSS(Common Vulnerability Scoring System)가 있지만,
`/security-review`가 이를 따르는지, 아니면 독자적인 기준을 사용하는지 알 수 없다.
심각도와 신뢰도가 기준 없이 제시되면, 개발자는 이 수치를 어떻게 해석해야 할지 판단 근거가 없다.

더 심각한 문제는 거짓 음성(false negative)이다.
LLM 기반 보안 분석은 알려지지 않은 취약점 패턴이나 맥락 의존적 취약점을 놓칠 수 있다.
`/security-review`가 아무 문제도 찾지 못했다는 결과가 “안전하다”는 의미인지,
“탐지 범위 내에서 문제가 없다”는 의미인지가 개발자에게 명확히 전달되지 않으면
도구에 대한 과신(over-trust)으로 이어질 수 있다.

### 실험 기능 상태가 프로덕션 사용에 주는 신호를 모호하게 처리한다

`/security-review`는 실험 기능(Experimental)으로 출시됐다.
보안 도구가 실험 상태라는 것은 특별한 주의가 필요하다.
탐지 정확도, 오탐율, 보안 판단의 신뢰도가 아직 검증 중인 도구를 프로덕션 코드에 적용할 때의 위험이 있다.

공지는 실험 상태의 의미나 그것이 보안 판단에 미치는 영향을 설명하지 않는다.
“실험 모드를 활성화하고 실행하라”는 안내는 이 도구의 결과를 어느 수준으로 신뢰해야 하는지에 대한 맥락을 제공하지 않는다.

보안 도구에서 실험 상태는 특히 중요한 표시다.
실험 상태의 도구가 취약점을 놓쳤을 때, 개발자는 그 결과를 어느 수준의 확신으로 믿었는가?
이 책임의 경계가 명확하지 않은 채로 보안 도구를 출시하는 것은 잘못된 안전감(false sense of security)을 심어줄 위험이 있다.

## 인사이트

### 터미널 기반 보안 리뷰는 보안과 개발의 책임 경계를 다시 그린다

소프트웨어 보안의 역사에서 보안 리뷰는 전문가 영역이었다.
침투 테스터, 보안 엔지니어, 코드 감사자가 개발자가 만든 코드를 검토하는 구조다.
`/security-review` 같은 도구는 이 경계를 흐리기 시작한다.

개발자가 자신의 코드를 직접 보안 분석할 수 있게 되면, 보안 리뷰의 1차 책임이 개발자에게 이동한다.
이것은 Shift Left의 논리적 귀결이지만, 동시에 보안 전문가의 역할이 “개발자가 접근하지 못하는 심층 분석”으로 좁아지는 것을 의미한다.

이 전환은 조직의 보안 팀 구성에도 영향을 미친다.
기본 취약점 탐지를 도구가 처리한다면, 보안 팀은 도구가 잡지 못하는 로직 버그, 아키텍처 수준의 취약점,
제로데이에 집중할 수 있다.
그러나 “도구가 통과시켰으니 괜찮다”는 잘못된 안전감이 조직에 퍼질 경우, 보안 팀의 감소가 위험 증가로 이어질 수 있다.

### AI가 보안 취약점을 만들고 동시에 탐지하는 순환 구조가 형성된다

AI 코딩 도구는 코드를 빠르게 생성하지만, AI가 생성한 코드는 보안 취약점을 포함할 가능성이 있다는 연구가 이미 다수 존재한다.
특히 AI는 학습 데이터에 포함된 취약한 패턴을 그대로 재현하거나,
보안 컨텍스트를 고려하지 않은 채 빠른 해결책을 제시하는 경향이 있다.

`/security-review`는 AI가 생성했을 가능성이 높은 코드를 AI가 보안 분석하는 구조다.
이것은 순환(circular) 구조다.
동일한 모델이나 유사한 학습 데이터에서 나온 취약점을 같은 계열의 모델이 탐지한다면,
시스템적 맹점(blind spot)이 공유될 가능성이 있다.

이 한계는 AI 보안 도구 전반의 근본적 문제다.
해결책은 여러 독립적인 분석 방법의 조합이다.
정적 분석(SAST), 동적 분석(DAST), 의존성 스캐닝, 그리고 LLM 기반 분석이 각자의 맹점을 상호 보완할 때
전체적인 취약점 탐지율이 높아진다.
`/security-review`가 가치 있으려면 기존 도구들의 보완재로 사용되어야 하며, 대체재로 사용되어서는 안 된다.

### 공급망 취약점과 크로스 프롬프트 인젝션의 동시 등장은 위협 지형의 변화를 보여준다

13개 취약점 범주의 구성은 현재 보안 위협 지형을 반영한다.
고정되지 않은 의존성(unpinned dependencies)과 크로스 프롬프트 인젝션이 목록에 함께 있다는 것은 의미심장하다.

전자는 SolarWinds, XZ Utils 사건 이후 공급망 보안이 주류 보안 관심사가 됐음을 반영한다.
의존성 버전을 고정하지 않으면 악성 업데이트가 자동으로 프로덕션에 들어갈 수 있다.
후자는 LLM 통합 앱이 새로운 공격 표면을 만들었음을 반영한다.
사용자 입력이 AI 에이전트의 판단에 영향을 줄 수 있다면, 그 경계를 코드 수준에서 검사해야 한다.

이 두 범주는 2020년대 중반의 소프트웨어 개발이 직면한 두 가지 새로운 위협을 상징한다.
전통적인 OWASP Top 10과 달리, 이것들은 소프트웨어 공급망과 AI 통합이라는 현재의 개발 패러다임에서 파생됐다.
보안 취약점 범주가 개발 패러다임의 변화를 뒤따라 진화하고 있으며,
개발자가 사용하는 도구도 이 진화를 반영해야 한다는 것을 `/security-review`의 범주 목록이 보여준다.
