# Bumblebee — 개발자 엔드포인트 공급망 스캐너

<https://github.com/perplexityai/bumblebee>

## 소개

Bumblebee는 Perplexity AI가 공개한 읽기 전용 인벤토리 수집기다.
macOS와 Linux 개발자 엔드포인트의 패키지, 확장 프로그램, 개발 도구 메타데이터를
온디스크(on-disk)에서 읽어 NDJSON으로 출력한다.

이 도구가 답하려는 질문은 하나다.
“어드바이저리가 특정 패키지나 버전을 지목했을 때, 지금 이 순간 어떤 개발자 머신에 그것이 설치되어 있는가?”

SBOM은 “무엇이 배포됐는가”를 답하고,
EDR(Endpoint Detection and Response)은 “무엇이 실행됐거나 네트워크에 접근했는가”를 답하지만,
공급망 대응에는 다른 관점이 필요하다.
락파일, 패키지 매니저 메타데이터, 확장 프로그램 매니페스트, 개발 도구 설정 파일에 퍼진
어지러운 온디스크 상태를 실시간으로 파악하는 것이다.
Bumblebee는 그 공백을 채운다.

Go 1.25+로 작성된 단일 정적 바이너리이며 비표준 라이브러리 의존성이 없다.
패키지 매니저 실행(`npm ls`, `pip show`, `go list` 등)이나 소스 파일 읽기를 수행하지 않는다.
Apache 2.0 라이선스로 공개되었고, 2026년 5월 공개 후 약 한 달 만에 GitHub 스타 4,634개를 기록했다.

## 커버리지

| 생태계           | `ecosystem` 값      | 주요 스캔 대상                                                   |
| ---------------- | ------------------- | ---------------------------------------------------------------- |
| npm              | `npm`               | `package-lock.json`, `node_modules/<pkg>/package.json`           |
| pnpm             | `npm`               | `pnpm-lock.yaml`                                                 |
| Yarn             | `npm`               | `yarn.lock` (Classic + Berry)                                    |
| Bun              | `npm`               | `bun.lock`, `bun.lockb`                                          |
| PyPI             | `pypi`              | `*.dist-info/METADATA`, `*.egg-info/PKG-INFO`                    |
| Go modules       | `go`                | `go.sum`, `go.mod`                                               |
| RubyGems         | `rubygems`          | `Gemfile.lock`, `*.gemspec`                                      |
| Composer         | `packagist`         | `composer.lock`, `vendor/composer/installed.json`                |
| MCP              | `mcp`               | `claude_desktop_config.json`, `~/.claude.json` 등 호스트 설정   |
| Agent skills     | `agent-skill`       | `~/.agents/.skill-lock.json`, `skills-lock.json`                 |
| 에디터 확장      | `editor-extension`  | VS Code, Cursor, Windsurf, VSCodium 매니페스트                   |
| 브라우저 확장    | `browser-extension` | Chromium, Firefox 프로파일                                       |
| Homebrew         | `homebrew`          | 포뮬라 `INSTALL_RECEIPT.json`, cask `.metadata`                  |

## 스캔 프로파일

Bumblebee는 원샷(one-shot) 스캐너로, 실행 주기는 호출자(cron, launchd, MDM 등)가 관리한다.

| 프로파일   | 스캔 범위                                       | 용도                                 |
| ---------- | ----------------------------------------------- | ------------------------------------ |
| `baseline` | 전역/사용자 패키지 루트, 툴체인, 확장, MCP 설정 | 외부 러너를 통한 주기적 경량 인벤토리 |
| `project`  | 구성된 개발 디렉토리 (`~/code`, `~/src` 등)     | 알려진 프로젝트 워크스페이스 인벤토리 |
| `deep`     | 명시적 `--root` 경로, `$HOME` 포함              | 사고 대응, 캠페인 조사용 온디맨드 검사 |

`baseline`과 `project`는 홈 디렉토리 루트를 거부하며, `deep`만 허용한다.

## 사용법

```bash
# 기본 전역 인벤토리
bumblebee scan --profile baseline > inventory.ndjson

# 특정 에코시스템만 스캔
bumblebee scan --profile baseline --ecosystem npm,pypi

# 어드바이저리 카탈로그로 노출 검사
bumblebee scan --profile deep \
  --root "$HOME" \
  --exposure-catalog ./catalog.json \
  --findings-only

# 스캔 루트 미리 확인
bumblebee roots --profile baseline

# 설치 검증
bumblebee selftest
```

노출 카탈로그는 `(ecosystem, name, version)` 정확 매칭 방식의 JSON 형식이다.
`threat_intel/` 디렉토리에는 최근 공급망 캠페인을 기반으로 Perplexity Computer가 조립하고
PR로 유지 관리하는 카탈로그가 포함되어 있다.

## 분석

### SBOM과 EDR 사이의 새로운 갭

공급망 보안의 전통적인 도구 두 가지는 서로 다른 시점을 본다.
SBOM은 배포 시점의 스냅샷이고, EDR은 실행 시점의 행동을 추적한다.
그러나 개발자 엔드포인트는 이 두 도구 모두가 잘 보지 못하는 공간이다.

개발 머신에는 배포된 적 없는 패키지, 실행된 적 없는 스크립트, 설치되어 있지만 활성화되지 않은 확장이 공존한다.
락파일은 프로젝트마다 다르고, 동일한 패키지의 서로 다른 버전이 여러 디렉토리에 존재할 수 있다.
이 상태는 SBOM이 포착하지 못하고 EDR이 추적하지 않는 잠재적 노출이다.

XZ Utils 사태(2024)나 tj-actions 공급망 공격처럼, 특정 버전이 악성으로 밝혀졌을 때
“우리 개발자 중 누가 이것을 설치했는가”를 즉시 파악하는 것은 사고 대응에서 핵심이다.
그 질문에 답하기 위해 기존에는 별도의 스크립트를 즉흥적으로 만들거나
MDM 에이전트의 제한적 쿼리에 의존해야 했다.
Bumblebee는 이 임시방편을 구조화된 도구로 대체한다.

### MCP와 에이전트 스킬을 포함한 공격 표면 확대

Bumblebee의 커버리지 테이블에서 가장 주목할 항목은 `mcp`와 `agent-skill` 생태계다.
이 두 항목은 기존 공급망 보안 도구가 전혀 다루지 않는 영역이다.

MCP(Model Context Protocol) 호스트 설정 파일은 어떤 MCP 서버가 어떤 도구를 노출하는지를 담고 있으며,
`env` 블록에 자격증명을 포함할 수 있다.
Bumblebee는 이 파일을 읽어 서버 인벤토리를 수집하되,
자격증명 값 자체는 출력하지 않도록 설계되어 있다.
그러나 어떤 MCP 서버가 설치되어 있는지를 파악하는 것만으로도
공급망 공격의 첫 번째 질문 — “어떤 환경이 노출됐는가” — 에 답할 수 있다.

`agent-skill` 에코시스템은 더욱 새롭다.
`~/.agents/.skill-lock.json`이나 프로젝트 수준 `skills-lock.json`을 스캔하는 것은
AI 에이전트 스킬 생태계가 이미 보안 감시의 대상이 되어야 할 만큼 성숙했음을 인정하는 것이다.

### Perplexity AI가 이 도구를 만든 이유

Perplexity는 검색과 AI 제품 회사다.
그들이 공급망 보안 스캐너를 만든 것은 표면적으로는 이질적으로 보인다.

그러나 `threat_intel/` 디렉토리가 Perplexity Computer로 조립된다는 사실에서 의도를 읽을 수 있다.
이것은 제품 데모이기도 하다.
공개된 위협 인텔리전스를 AI가 구조화된 카탈로그로 변환하는 파이프라인을 공개함으로써,
Perplexity가 자사 AI 제품의 실용적 사용 사례를 보여주는 것이다.

동시에 MCP와 에이전트 스킬에 대한 커버리지는 Perplexity 자신이 AI 에이전트 생태계의
보안 위험을 직접 다루어야 하는 당사자임을 보여준다.
Perplexity Assistant나 Perplexity Computer가 MCP 서버와 통합된다면,
그 서버들의 공급망 상태는 Perplexity에게도 실질적인 보안 문제다.
이 도구는 내부 필요에서 시작했을 가능성이 높다.

## 비평

### 온디스크 스냅샷은 실제 위험이 아니라 잠재적 노출만 본다

Bumblebee의 핵심 제약은 설계의 핵심이기도 하다.
이 도구는 “무엇이 설치되어 있는가”만 말하고 “무엇이 실행됐는가”는 말하지 않는다.

그러나 공급망 침해에서 중요한 것은 패키지가 설치됐느냐가 아니라 악성 코드가 실행됐느냐다.
XZ Utils의 경우, 악성 버전이 설치되어 있어도 특정 조건(systemd 통합, root 실행)을 충족하지 않으면 공격이 실행되지 않는다.
Bumblebee는 취약한 버전이 락파일에 있다는 것은 알려주지만,
그것이 실제로 로드됐는지, 악성 조건이 트리거됐는지는 알 수 없다.

이것이 “잠재적 노출 파악”에는 유용하지만, 실제 침해 확인에는 부족하다는 것을 의미한다.
사고 대응팀은 Bumblebee의 findings를 출발점으로 삼아야 하며, 결론으로 받아들여선 안 된다.
이 제약은 README에 명시되어 있지만, findings를 “노출”이라고 표현하는 방식이
실제 침해로 혼동될 여지가 있다.

### 위협 인텔리전스 카탈로그의 품질은 단일 소스에 의존한다

`threat_intel/` 디렉토리의 카탈로그가 Perplexity Computer로 조립된다는 것은
AI가 생성한 위협 인텔리전스를 보안 결정에 직접 사용하는 것이다.

공개된 위협 인텔리전스 보고서를 AI가 구조화된 `(ecosystem, name, version)` 튜플로 변환하는 과정에는
오류와 오해석의 가능성이 있다.
특히 패키지 이름의 철자, 버전 범위의 해석, 에코시스템 분류에서 AI가 실수할 수 있다.
잘못된 카탈로그 항목은 false positive(실제로 안전한 패키지를 위험으로 표시)나
false negative(실제로 위험한 버전을 놓치는 것)를 낳는다.

PRs를 통해 카탈로그가 업데이트된다고 하지만, 리뷰 프로세스의 엄격성은 공개되어 있지 않다.
OSV(Open Source Vulnerabilities) 데이터베이스나 NVD처럼 검증된 외부 데이터 소스와의 통합이
신뢰성을 높이는 자연스러운 다음 단계일 것이다.

### MCP 설정 파싱이 가져오는 새로운 위험

MCP 호스트 설정 파일을 스캔한다는 것은 `env` 블록에 포함된 자격증명에 접근한다는 것이다.
Bumblebee는 “이 값들을 출력 레코드에 포함하지 않는다”고 명시한다.
그러나 이것은 소프트웨어의 의도적 설계이지, 기술적 불가능성이 아니다.

Bumblebee 바이너리 자체가 공급망 공격의 대상이 될 경우,
악성으로 교체된 바이너리는 MCP 설정 파일을 파싱하면서 자격증명을 유출할 수 있다.
공급망 보안 도구가 공급망 공격에 취약하다는 아이러니는 단순한 역설이 아니다.
Bumblebee 설치 방법(`go install github.com/perplexityai/bumblebee@latest`)은
최신 태그를 신뢰하는 것을 기본으로 삼는데, 그 신뢰 체인이 얼마나 견고한지가 핵심이다.
바이너리 재현 가능 빌드(reproducible builds)와 서명 검증이 없으면,
이 도구는 자신이 탐지하려는 위협과 같은 범주의 위험을 내포한다.

## 인사이트

### 개발자 엔드포인트가 새로운 공격 벡터로 공식화됐다

Bumblebee의 등장은 기술적 도구의 출현이기도 하지만,
“개발자 머신이 공급망 공격의 중요한 진입점”이라는 위협 모델의 공식화이기도 하다.

과거의 공급망 공격은 주로 빌드 파이프라인이나 배포된 패키지를 표적으로 삼았다.
그러나 개발자 엔드포인트는 다르다.
코드 서명 키, 내부 API 자격증명, 프로덕션 인프라 접근권이 개발자 머신에 존재한다.
악성 패키지가 개발자 머신에서 실행될 경우, 배포 인프라까지 이동 경로가 짧다.

Bumblebee 같은 도구의 존재는 이 위협 모델이 이제 조직적 대응 수준으로 올라왔음을 의미한다.
단발적인 사고 대응 스크립트가 아니라, 플릿(fleet) 수준에서 지속적으로 모니터링하는 인프라가 필요해졌다는 인식이다.
이 인식이 확산되면 개발자 엔드포인트에 대한 보안 요구 사항이 높아지고,
결과적으로 개발자의 작업 환경에 더 많은 제약이 생길 것이다.

### MCP 에코시스템의 보안 인프라가 형성되기 시작했다

Bumblebee가 MCP를 에코시스템으로 다루는 것은,
MCP 서버가 이제 패키지 레지스트리와 유사한 공급망 위험을 가진다는 인식이 생겼음을 보여준다.

npm 생태계가 성숙하면서 `npm audit`, `snyk`, `socket.dev` 같은 보안 도구가 등장했다.
MCP 생태계는 이제 막 형성되는 중이지만,
이미 보안 도구가 그 뒤를 따르기 시작했다.
이 속도는 이전 에코시스템보다 빠르다.
npm이 보안 도구를 갖추기까지 약 5-7년이 걸렸다면,
MCP는 에코시스템 형성과 보안 도구 등장이 거의 동시에 일어나고 있다.

이것이 좋은 것만은 아니다.
에코시스템이 아직 표준화되지 않은 시점에 보안 인프라가 등장하면,
그 인프라 자체가 에코시스템의 발전 방향을 왜곡할 수 있다.
“어떤 MCP 서버가 안전한가”를 정의하는 카탈로그는
누가 그 카탈로그를 관리하느냐에 따라 MCP 서버 생태계의 게이트키퍼가 될 수 있다.

### AI가 자신의 보안 인프라를 만들기 시작했다

Perplexity Computer가 위협 인텔리전스 카탈로그를 조립한다는 사실은
기술적 세부사항이 아니라 구조적 신호다.

보안 분야의 위협 인텔리전스는 전통적으로 인간 분석가의 고부가가치 작업이었다.
공개된 어드바이저리와 사고 보고서를 읽고, 핵심 정보(패키지 이름, 버전, 에코시스템)를 추출하고,
구조화된 형식으로 변환하는 것은 보안 전문가의 몫이었다.
Perplexity가 이 과정을 AI로 자동화하는 것은, AI가 자신의 생태계 보안 인프라를 스스로 구축하는 방향으로의 첫 걸음이다.

이 방향이 계속된다면, 보안 인텔리전스의 생산 비용이 급격히 낮아지고
더 많은 위협이 더 빠르게 구조화된 카탈로그로 변환될 것이다.
그러나 동시에 AI가 생성한 위협 정보의 정확성 검증이 새로운 병목이 된다.
AI가 만든 카탈로그를 AI가 검증하는 순환이 생기면,
그 검증의 독립성은 어디서 오는가.

### 읽기 전용 설계는 신뢰 문제를 해결하지 못한다

Bumblebee가 “읽기 전용”임을 강조하는 것은 배포 승인을 받기 위한 중요한 속성이다.
개발자 엔드포인트에 에이전트를 설치하려면, 그 에이전트가 변경을 가하지 않는다는 보증이 필요하다.

그러나 “읽기 전용”은 신뢰의 충분 조건이 아니다.
읽기만 해도 민감한 정보(자격증명, 설치된 소프트웨어 목록, 개발 환경 구성)가 유출될 수 있다.
Bumblebee는 NDJSON 출력을 파일이나 HTTPS 엔드포인트로 전송할 수 있다.
이 데이터가 누구에게 전송되고 어떻게 저장되는지는 Bumblebee 자체의 문제가 아니라 운영자의 문제다.

그러나 도구가 “읽기 전용”이라는 강조는 이 데이터 전송의 의미를 가리는 효과가 있다.
어떤 패키지가 어떤 버전으로 어떤 머신에 설치되어 있는지를 중앙에서 수집하는 것은,
그 자체로 매우 민감한 인텔리전스가 된다.
공격자가 이 데이터에 접근하면, 어떤 머신에 어떤 취약점이 있는지를 먼저 알 수 있다.
“읽기 전용 수집기”의 출력 데이터가 새로운 공격 표면이 되는 역설을 직시해야 한다.
