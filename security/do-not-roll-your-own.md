# 직접 만들지 말라

원문: <https://susam.net/do-not-roll-your-own.html>

HN 토론: <https://news.ycombinator.com/item?id=48252192> (139점, 115개 댓글)

## 요약

“자체 암호화를 구현하지 말라(Don't roll your own crypto)”는 보안 업계의 오랜 격언이 있다.
저자는 이 원칙이 암호화를 넘어 웹 UI 전반에도 적용되어야 한다고 주장한다.
브라우저가 수십 년간 다듬어온 기본 동작들, 즉 스크롤, 링크 내비게이션, 텍스트 선택, 복사/붙여넣기, 비밀번호 입력, 날짜 선택기를 개발자가 직접 재구현할 때 일어나는 문제를 구체적 사례로 제시한다.

저자가 지적하는 핵심 문제는 재구현의 품질 문제가 아니라 재구현 자체의 불필요함이다.
브라우저가 이미 잘 동작하는 기능을 JavaScript로 가로채거나 대체하면 자동완성, 접근성 도구, 비밀번호 관리자 같은 외부 시스템과의 협력이 깨진다.
GitHub의 링크 내비게이션을 JavaScript가 가로채는 바람에 현재 탭 로딩이 새 탭보다 느려지는 사례처럼, 재구현은 종종 원래보다 못한 결과를 낳는다.

보안 관점에서도 자체 비밀번호 필드 구현은 위험하다.
일반 텍스트 필드를 마스킹하는 방식은 브라우저가 비밀번호로 인식하지 못해 운영체제 수준의 보안 처리가 누락된다.
페루의 은행들처럼 실제 산업 환경에서도 이 패턴은 반복된다.

저자는 사용자 경험 전문가가 아닌 한 사람의 웹 사용자로서 관점을 제시하며 결론짓는다.
UI 변경은 사용자가 매번 재학습해야 하는 부담을 낳는다.
특히 새로운 인터페이스에 적응이 어려운 고령 사용자에게 이 비용은 불균형적으로 크다.

## 분석

### 암호화 원칙의 일반화

“자체 암호화를 구현하지 말라”는 원칙은 암호화가 검증하기 어렵고 실수의 파급 효과가 크기 때문에 생겨났다.
저자는 이 논리 구조를 웹 UI에 적용한다.
브라우저 API도 수십 년간 수천만 명의 사용자가 검증한 구현체이며, 재구현의 리스크가 실익을 초과하는 경우가 많다.

[GeekNews 댓글](https://news.hada.io/topic?id=29811)에서 제기된 우려 중 흥미로운 것은 CSR(Client-Side Rendering) 필요성에 관한 반론이다.
“링크 내비게이션에는 강하게 반대한다”는 댓글 의견처럼, 단일 페이지 애플리케이션(SPA)에서 링크 처리를 JavaScript가 담당하는 것은 불가피한 경우가 있다.
저자의 주장은 재구현 자체를 금지하는 것이 아니라 불필요한 재구현을 경계하라는 것이지만, 이 경계선은 맥락에 따라 달라질 수 있다.

incr_me는 이 긴장의 구조적 원인을 짚는다.[^incr_me]
브라우저가 정적 콘텐츠 표시를 넘어 애플리케이션 호스트 역할을 떠맡게 된 것이 문제의 뿌리라는 것이다.
이 역할 확장은 되돌릴 수 없다.
브라우저가 문서 뷰어로 설계될 때 상정하지 않았던 상호작용 패턴들을 개발자가 구현해야 하는 현실이 재구현 충동의 근본 배경이다.

### 한국의 맥락

[GeekNews 댓글](https://news.hada.io/topic?id=29811)에서 한 독자는 “우리나라처럼 자체 암호화, 자체 보안 프로그램이 많은 나라가 없을 것 같다”고 언급했다.
Active X 시대부터 이어진 한국의 자체 보안 플러그인 문화는 이 원칙의 반면교사다.
인터넷뱅킹에서 요구하던 키보드 보안 프로그램들은 브라우저의 표준 보안 메커니즘과 충돌하며 오히려 호환성과 보안 문제를 낳았다.

### 네이티브 UI의 한계도 존재한다

날짜 선택기나 날짜 범위 입력처럼 `<input type=“date”>`의 표현력이 부족한 경우가 있다.
항공권 예약 화면처럼 날짜와 가격 정보를 함께 보여주는 달력은 네이티브로 구현이 불가능하다.
“캘린더 위젯을 네이티브 날짜 선택기 옆에 병렬로 제공”하는 절충안처럼, 원칙의 적용에는 경우의 수가 있다.

## 비평

### 재구현의 이유를 다루지 않는다

개발자들이 브라우저 기본 UI를 재구현하는 이유는 대부분 디자인 일관성과 크로스브라우저 호환성 때문이다.
`<input type=“date”>`는 브라우저마다 UI가 다르고 스타일링이 제한적이다.
저자는 재구현의 부작용을 잘 정리하지만, 왜 개발자들이 그 선택을 하는지에 대한 이해와 공감이 부족하다.

awongh는 더 날카로운 개념 구분을 제시한다.[^awongh]
글의 진짜 주장은 "직접 만들지 말라"가 아니라 "UI 관행을 깨지 말라"에 더 가깝다는 것이다.
네이티브 브라우저 기능을 대체하는 것이 문제라면, 무거운 외부 npm 패키지 대신 소규모 자체 컴포넌트를 만드는 것은 오히려 합리적인 선택일 수 있다.
npm 공급망 보안 위협이 커진 지금, 매년 주요 버전이 바뀌는 외부 라이브러리의 유지보수 비용과 자체 구현의 비용을 단순하게 비교하기 어렵다.

### 대안 제시가 약하다

“직접 만들지 말라”는 주장의 자연스러운 다음 단계는 “그럼 무엇을 쓰라”는 안내다.
잘 유지되는 오픈소스 컴포넌트 라이브러리나 표준 준수 방법론을 제시했다면 더 실용적인 글이 됐을 것이다.
원칙 제시에서 멈추는 것은 독자를 절반만 돕는 셈이다.

## 인사이트

### 플랫폼 신뢰의 경제학

개발자가 브라우저 기본 동작을 재구현하는 순간, 그 구현의 버그 책임과 유지보수 비용을 모두 떠안게 된다.
반면 브라우저 엔진이 처리하면 구글, Mozilla, Apple의 수천 명 엔지니어가 지속적으로 개선하는 코드를 무료로 사용하는 셈이다.
이것은 단순히 “더 좋은 코드”의 문제가 아니라 경제적 선택이다.
자체 구현의 유지보수 비용을 인식하는 팀은 더 신중하게 재구현 결정을 내릴 것이다.

pvillano는 이 원칙에 유용한 개념 프레임을 더한다.[^pvillano]
브라우저 UI 요소들은 "비다이어제틱(non-diegetic)"하다는 것이다.
탭, 주소창, 스크롤바, 컨텍스트 메뉴는 웹사이트의 일부가 아니라 브라우저 자체의 일부다.
사이트 디자인과 일치하지 않아도 아무도 이상하게 여기지 않는다.
따라서 브라우저 UI를 사이트 스타일에 맞게 커스터마이징해야 한다는 전제 자체가 존재하지 않는 필요를 만들어낸 것일 수 있다.

### 접근성은 공짜가 아니다

브라우저 기본 컴포넌트는 스크린 리더, 키보드 내비게이션, 고대비 모드, 텍스트 확대 같은 접근성 기능을 내장한다.
자체 구현한 비밀번호 필드나 날짜 선택기는 이 모든 기능을 처음부터 다시 구현해야 한다.
실제로 대부분의 커스텀 컴포넌트는 접근성 지원이 부실하고, 결과적으로 장애가 있는 사용자를 배제한다.
“직접 만들지 말라”는 원칙은 접근성의 언어로 번역하면 “이미 접근 가능한 것을 접근 불가능하게 만들지 말라”가 된다.

ethin은 이 실패의 구체적 메커니즘을 지목한다.[^ethin]
커스텀 컨트롤을 만들 때 ARIA 모범 사례를 제대로 따르는 개발자가 거의 없다는 것이다.
스크린 리더를 위한 올바른 시맨틱 마크업과 역할 속성을 갖추는 것은 생각보다 훨씬 복잡하다.
그 결과 커스텀 컴보박스, 라디오 버튼, 입력 필드들이 장애인 사용자에게는 작동하지 않는 인터페이스가 된다.

### 재구현 욕구는 실력의 함정이다

역설적으로 재구현 욕구는 종종 실력 있는 개발자에게서 강하게 나타난다.
“내가 더 잘 만들 수 있다”는 자신감이 기존 시스템이 수십 년간 누적한 엣지 케이스 처리와 플랫폼 통합의 가치를 과소평가하게 만든다.
암호화 분야의 “똑똑한 개발자가 만든 자체 암호화가 가장 위험하다”는 격언은 UI에도 적용된다.
플랫폼이 처리하는 엣지 케이스의 목록은 보이지 않기 때문에, 쉬워 보이는 것이 사실은 깊은 문제다.
