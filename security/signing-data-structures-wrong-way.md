# 구조화된 데이터를 잘못 서명하는 방법 - 도메인 분리 취약점

> 원문: <https://blog.foks.pub/posts/domain-separation-in-idl/>

## 요약

서로 다른 의미를 가진 두 데이터 구조가 동일한 바이트열로 직렬화될 수 있을 때,
한 타입에 만들어진 서명을 다른 타입에 재사용하는 공격이 가능하다. 이를 도메인
분리(domain separation) 문제라고 한다.

예를 들어 `TreeRoot`(투명성 트리 루트 캡슐화)와 `KeyRevoke`(키 폐기)가 같은 필드
구성을 가지면 동일한 바이트로 직렬화된다. 공격자는 서명된 `TreeRoot`의 서명을
`KeyRevoke`에 붙여 위조된 키 폐기 메시지를 만들 수 있다.

실제 피해 사례는 Bitcoin 거래 가변성(transaction malleability), Ethereum DEX 주문
재사용, TLS, JWT, AWS 등에서 발생했다.

Snowpack의 방어 전략은 64비트 난수 도메인 분리자를 IDL에 직접 포함시키고 타입
시스템이 이를 강제하는 것이다. `GetUniqueTypeID()` 메서드 구현 없이는 서명·검증
함수를 아예 호출할 수 없어 실수 자체를 불가능하게 만든다.

## 분석

도메인 분리 취약점이 반복되는 이유는 직렬화 형식(protobuf, JSON, msgpack 등)이
타입 정보를 바이트열에 포함하지 않기 때문이다. 두 메시지가 같은 스키마로 인코딩되면
구분할 수 없다. 서명은 바이트열만 보고 의미를 모른다.

방어의 핵심 원칙은 두 가지다.

첫째, 서명 대상 바이트열에 **타입 식별자를 포함**시킨다. `"KeyRevoke" || serialize(data)`
처럼 타입을 나타내는 문자열을 앞에 붙이거나, 고유 상수를 해시에 포함시킨다.

둘째, **타입 시스템이 도메인 분리를 강제**하도록 설계한다. Snowpack처럼 인터페이스
수준에서 고유 ID 제공을 요구하면 개발자가 잊어버릴 수 없다. 코드 리뷰가 아닌
컴파일러가 막아야 한다.

## 비평

이 취약점은 암호화 API의 추상화 수준 문제를 드러낸다. `sign(bytes)` 형태의 저수준
API는 "어떤 바이트에 서명하는가"에 대한 맥락을 요구하지 않는다. 사용자가 올바른
전처리를 직접 구현해야 하므로 실수가 발생한다.

AI 코드 생성이 확산될수록 이 위험은 커진다. AI는 기존 코드 패턴을 학습해 상수나
타입 태그를 무심코 재사용할 수 있다. 도메인 분리 값의 중복 검사를 자동화하지 않으면
코드 리뷰만으로는 막기 어렵다.

## 인사이트

### 암호화 API의 인터페이스가 올바른 사용을 강제해야 한다

`sign(bytes)`는 편리하지만 위험하다. "사용하기 쉬운 올바른 방법, 사용하기 어려운
잘못된 방법(make the right thing easy, wrong thing hard)"이라는 API 설계 원칙이
암호화 라이브러리에 특히 중요하다. Snowpack의 접근처럼 타입 ID 없이는 서명 함수를
호출조차 못하게 만드는 것이 방어의 최선이다.

### 직렬화 형식의 타입 정보 부재는 보안 가정에서 자주 무시된다

protobuf, JSON, msgpack은 의도적으로 타입 정보를 최소화해 크기를 줄인다. 이 설계
결정이 암호화 문맥에서는 취약점이 된다. 직렬화 형식 선택이 보안 속성에 영향을
준다는 인식이 시스템 설계 단계에서 필요하다.

### 역사적 선례가 반복되는 이유는 교육이 아닌 도구 문제다

Bitcoin 거래 가변성은 2014년에 이미 대규모로 악용됐다. 10년이 넘었지만 같은 패턴의
취약점이 새 시스템에서 반복된다. 이는 개발자 교육의 실패가 아니라 올바른 패턴을
강제하는 도구의 부재다. 암호화 primitives를 직접 조합하는 저수준 인터페이스 대신,
도메인 분리가 내장된 상위 추상화가 기본값이 되어야 한다.
