# ChatGPT가 타이핑을 허용하기 전에 Cloudflare가 React 상태를 읽는다

원문: <https://www.buchodi.com/chatgpt-wont-let-you-type-until-cloudflare-reads-your-react-state-i-decrypted-the-program-that-does-it/>

## 요약

보안 연구자가 ChatGPT를 보호하는 Cloudflare Turnstile 프로그램 377개를 복호화해 내부 동작을 분석했다.
겉보기엔 불투명해 보이는 암호화가 실제로는 XOR 연산에 불과했고, 키마저 동일 데이터 스트림 안에 포함돼 있었다.
암호화의 목적이 분석 방지가 아니라 임의 검사를 어렵게 만드는 정책적 선택임이 드러났다.

복호화 과정은 두 단계다.
먼저 prepare 요청의 `p` 토큰으로 `turnstile.dx` 필드의 base64 디코딩 결과를 XOR해 89개 VM 명령어를 얻는다.
그 안에 있는 19KB 암호화 블롭은 바이트코드에 부동소수점 리터럴로 포함된 XOR 키(예: `97.35`)로 풀린다.
50개 요청 샘플에서 이 키가 100% 일치했다.

Turnstile의 봇 탐지는 세 계층으로 나뉜다.
첫째, 브라우저 핑거프린트 계층은 WebGL 파라미터, 화면 속성, 하드웨어 정보, 폰트 측정, DOM 탐사, 스토리지 등 55개 속성을 수집한다.
수집 결과는 `6f376b6560133c2c` 키로 localStorage에 영속적으로 저장된다.
둘째, Cloudflare 네트워크 인텔리전스 계층은 에지 서버가 주입한 도시·위도·경도·IP·리전 헤더를 확인한다.
이 값이 없으면 Cloudflare 네트워크를 통과하지 않은 것으로 판단한다.

셋째 계층이 가장 중요하다.
Turnstile은 `__reactRouterContext`, `loaderData`, `clientBootstrap` 같은 React 내부 상태를 직접 검사한다.
이것은 브라우저 환경의 핑거프린트가 아니라 특정 React 애플리케이션이 완전히 부팅됐는지를 확인하는 애플리케이션 계층 탐지다.
JavaScript를 실행하지 않거나 React 번들을 수화(hydration)하지 않은 헤드리스 브라우저는 이 단계에서 탐지된다.

별도로 Signal Orchestrator라는 모듈이 271개 명령어로 행동 생체 인식(behavioral biometrics)을 수행한다.
`keydown`, `pointermove`, `click`, `scroll`, `paste`, `wheel` 이벤트 리스너를 설치하고 36개의 `window.__oai_so_*` 속성에 키스트로크 타이밍, 마우스 속도, 스크롤 패턴, 유휴 시간을 기록한다.
Proof of Work는 25개 필드 핑거프린트와 SHA-256 해시캐시로 구성되며, 난이도는 400K~500K 균일 난수다.

## 분석

### XOR 암호화의 실제 목적

암호화 강도가 낮은 XOR을 사용하면서도 키를 동일 데이터 안에 포함한다는 설계는 역설적으로 보인다.
저자는 이를 “암호화는 인과관계가 아닌 정책 결정”이라고 표현한다.
임의로 Turnstile 프로그램을 열어 분석하는 것은 어렵게 만들되, 작정하고 분석하는 연구자는 어차피 막지 못한다는 현실적 선택이다.
대신 프로그램 자체가 서버 측에서 동적으로 생성되므로 “자동화된 수정 없이 항상 변경될 수 있다”는 운영상 유연성이 핵심 방어 메커니즘이다.

### React 상태 검사의 의미

전통적인 봇 탐지는 브라우저 환경을 흉내 내면 통과할 수 있었다.
User-Agent 스푸핑, WebDriver 속성 숨기기, 캔버스 핑거프린트 조작 등이 그 수단이다.
React 내부 상태 검사는 이 전제를 깨뜨린다.
단순히 브라우저처럼 보이는 것만으로는 부족하고, 특정 React 애플리케이션의 전체 부팅 과정을 실제로 실행해야 통과할 수 있다.
이것은 봇 탐지의 방어선이 “브라우저인가”에서 “이 특정 애플리케이션을 실제로 실행하고 있는가”로 이동했다는 신호다.

### 계층별 탐지의 전략적 설계

세 계층은 각기 다른 우회 비용을 요구한다.
브라우저 핑거프린트는 도구로 우회 가능하다.
Cloudflare 네트워크 헤더는 Cloudflare 프록시를 경유해야 해 비용이 높아진다.
React 상태 검사는 완전한 애플리케이션 실행 환경을 요구한다.
Signal Orchestrator의 행동 생체 인식은 장기 학습이 필요해 단기 우회가 사실상 불가능하다.
각 계층이 서로 다른 위협 모델을 타깃으로 하는 심층 방어(defense in depth) 설계다.

## 비평

### 공개 분석의 역설

이 글은 Turnstile의 내부 동작을 상세히 공개한다.
이 정보가 봇 제작자에게 유용할 수 있다는 점에서 보안 연구의 책임 공개(responsible disclosure) 문제가 제기될 수 있다.
그러나 저자의 핵심 주장, 즉 “암호화가 분석을 막지 못하며 실제 방어는 동적 변경 가능성에 있다”는 점이 맞다면, 이 분석 자체가 방어력을 크게 약화시키지 않는다.
오히려 이 분석은 Cloudflare의 봇 탐지 철학이 “비밀 유지”가 아니라 “실행 비용 증가”에 있다는 것을 확인한다.

### 377개 샘플의 대표성

50개 요청 샘플에서 XOR 키 일치율이 100%라고 보고하지만, 이것이 모든 시나리오에서 동일한지는 확인되지 않는다.
Cloudflare가 지역, 클라이언트 유형, 리스크 점수에 따라 다른 프로그램을 제공할 가능성은 배제할 수 없다.
377개 샘플 분석이 전체 Turnstile 동작의 대표 샘플인지, 특정 조건에서만 관찰된 것인지 글은 명확히 하지 않는다.

### 행동 생체 인식의 설명 부재

Signal Orchestrator가 수집하는 36개 속성의 실제 서버 측 처리 방식은 분석되지 않았다.
데이터를 어떻게 점수화하고, 어떤 임계값으로 봇을 판정하는지가 이 탐지 계층의 핵심인데, 클라이언트 사이드 코드 분석만으로는 접근할 수 없다.
글의 분석이 클라이언트 측 수집 메커니즘에 집중한 것은 방법론적으로 한계이지 결함이 아니지만, 독자는 이 차이를 인식해야 한다.

## 인사이트

### 봇 탐지의 군비 경쟁이 애플리케이션 계층으로 이동했다

초기 CAPTCHA는 이미지 인식을 인간과 기계를 구분하는 기준으로 사용했다.
AI 이미지 인식이 발전하자 행동 분석으로 이동했다.
헤드리스 브라우저가 행동을 흉내 내자 핑거프린트로 이동했다.
이제 Cloudflare는 특정 React 애플리케이션의 완전한 실행 여부를 검사한다.
이 진화의 방향은 명확하다. 탐지 기준이 “무엇처럼 보이는가”에서 “무엇을 실제로 실행하고 있는가”로 심화되고 있다.
다음 단계는 특정 사용자 행동 패턴의 장기 일관성 검사나 디바이스 신뢰도 체인일 가능성이 높다.

이 군비 경쟁의 승자는 탐지를 불가능하게 만드는 쪽이 아니라, 탐지 우회 비용을 경제적으로 지속 불가능하게 만드는 쪽이다.
Cloudflare의 설계가 바로 이 경제적 논리를 따른다.
Proof of Work의 400K~500K 난이도는 개인 봇에는 부담이지만, 대규모 봇팜에는 여전히 감당 가능한 수준이다.
결국 봇 탐지는 완전한 해결책이 없는 비용 증가 게임이다.

### 암호화는 신뢰가 아닌 마찰을 제공한다

이 글이 보여주는 가장 중요한 교훈은 암호화의 실제 역할에 관한 것이다.
XOR 암호화에 키를 같은 데이터에 포함하는 설계는 보안 교과서의 관점에서는 잘못됐다.
그러나 Cloudflare의 실제 목표는 “분석 불가능한 코드”가 아니라 “동적으로 변경 가능하면서 임의 검사를 어렵게 만드는 코드”였다.
이 목표에서 XOR 암호화는 완벽하게 기능한다.

암호화가 “비밀을 지키는 것”이 아니라 “마찰을 만드는 것”으로 사용될 수 있다는 관점은 보안 설계에서 중요한 통찰이다.
완벽한 비밀이 불가능한 환경, 즉 클라이언트 측 코드가 결국 실행 환경에 노출되는 웹의 특성상, 암호화의 목적은 변할 수밖에 없다.
이 사례는 보안 목표를 명확히 정의하지 않은 채 “암호화를 추가하면 더 안전하다”는 막연한 믿음이 설계 오류로 이어질 수 있다는 경고이기도 하다.

### 프론트엔드 코드는 언제나 공개된 인터페이스다

웹 애플리케이션의 클라이언트 사이드 코드는 원칙적으로 분석 가능하다.
난독화, 최소화, 암호화 모두 분석 비용을 높일 뿐 분석 가능성을 없애지 않는다.
이 사실을 설계 전제로 받아들이면, 보안 아키텍처의 방향이 바뀐다.
클라이언트 사이드에 비밀을 두는 대신, 서버 측 검증과 동적 변경을 방어의 핵심으로 삼아야 한다.

ChatGPT의 Turnstile 구현이 바로 이 원칙을 따른다.
Turnstile 프로그램 자체가 분석되더라도 Cloudflare는 그 프로그램을 언제든 바꿀 수 있다.
분석에 들어간 연구자의 시간이 Cloudflare의 한 번 배포로 무효화될 수 있다.
클라이언트 코드의 공개성을 인정하면서 서버 측 동적성으로 방어하는 이 패턴은, 퍼블릭 인터넷에서 봇 탐지가 작동하는 방식의 본질적 모습이다.
