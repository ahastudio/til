# What xAI's Grok Build CLI Actually Sends to xAI

원문: <https://gist.github.com/cereblab/dc9a40bc26120f4540e4e09b75ffb547>

HN 토론: <https://news.ycombinator.com/item?id=48877371> (274점, 126개 댓글)

## 요약

저자는 xAI의 Grok CLI(버전 0.2.93)가 실제로 어떤 데이터를 xAI 서버에
전송하는지 확인하기 위해 `mitmproxy`를 이용한 와이어 레벨 트래픽 분석을
수행했다.
macOS 로그인 키체인에 mitmproxy CA 인증서를 설치하고 Grok의 HTTPS 트래픽을
프록시로 우회시켜 복호화한 뒤, 파일에 고유한 “canary” 마커를 심은 테스트
저장소를 만들어 데이터가 실제로 전송되는지 추적했다.
`~/.grok/upload_queue`에 쌓이는 업로드 대기 아티팩트를 검사하고, git
bundle 분석으로 저장소 전체가 실제로 전송되는지도 검증했다.

첫 번째 발견은 시크릿 파일이 두 경로를 통해 검열 없이 그대로 전송된다는
것이다.
모델 턴 채널(`POST /v1/responses`)에서는 요청 본문에 파일 내용이 그대로
포함되는데, 48KB 크기의 캡처된 요청 본문에는 `.env` 자격 증명이
`API_KEY=CANARY7F3A9-SECRET-should-not-leave`,
`DB_PASSWORD=CANARY7F3A9-DBPASS` 형태로 저장된 그대로 담겨 있었다.
저장 채널(`POST /v1/storage`)에서도 동일한 내용이 `session_state`
아카이브에 포함되어 HTTP 200 응답과 함께 백엔드 저장소로 전송되었다.
대조 실험에서는 읽지 않은 파일이 모델 턴 채널(채널 A)로는 새지 않는다는
점을 확인했지만, 이는 저장 채널(채널 B)의 전체 저장소 스냅샷에서까지
제외된다는 뜻은 아니었다.

두 번째 발견은 저장소 전체가 에이전트가 실제로 읽은 파일과 무관하게
업로드된다는 것이다.
12GB 규모의 테스트 저장소를 사용했을 때 `/v1/storage` 엔드포인트를 통해
약 75MB씩 73개의 순차적 청크가 생성되었고, 캡처를 종료하기 전까지 총
5.10GiB가 모두 HTTP 200으로 성공적으로 업로드되었다.
같은 세션에서 모델 턴 채널로는 겨우 192KB만 오갔는데, 이는 27,800배에
달하는 비율 차이로, 기가바이트 단위 전송이 모델이 읽은 파일이 아니라
저장소 콘텐츠 자체임을 뒷받침한다.
결정적으로, 와이어 트래픽에서 캡처한 git bundle을 실제로 clone해 보니
“한 번도 읽지 않음”으로 표시한 파일의 고유 마커가 그대로 담겨 있었고,
4개의 커밋과 47개의 파일로 이루어진 전체 git 히스토리도 그대로 복구됐다.
이는 서로 다른 두 개의 실제 코드베이스에서 독립적으로 검증됐다.

세 번째 발견은 저장소 스냅샷이 `grok-code-session-traces`라는 이름의
특정 Google Cloud Storage 버킷으로 전송된다는 것이다.
바이너리 문자열 상수에 이 버킷 이름이 그대로 노출되어 있었고, 스테이징된
`metadata.json` 파일에는 `gs://grok-code-session-traces/repo_changes_dedup/v2/supplemental/sha256_...`
형태의 파일별 전송 대상이 기록되어 있었으며, `storage.googleapis.com`
도메인으로의 멀티파트 PUT 요청도 직접 관측됐다.
추가로 Mixpanel과 xAI 자체 이벤트 엔드포인트를 통한 서드파티 트래킹도
확인됐고, 계정 설정에서 “Improve the model” 기능을 꺼도 저장소 업로드는
막히지 않았으며 옵트아웃 이후에도 `/v1/settings` 응답은 여전히
`trace_upload_enabled: true`를 반환했다.
저자는 CLI 설치 스크립트와 퀵스타트 문서를 살펴봤지만 `grok-code-session-traces`
업로드 메커니즘에 대한 명시적 고지는 찾지 못했다고 밝히면서도, 이것이
전체 문서에 대한 포괄적 감사는 아니었다고 명시했다.
저자는 전송된 데이터가 실제로 학습에 쓰인다는 증거는 없다는 점, 3GB
규모의 `storage.googleapis.com` 직접 PUT 캡처는 쿼터 소진으로 보존하지
못했다는 점, 12GB 캡처는 5.1GiB 지점에서 중단됐다는 점 등 조사 범위의
한계도 함께 밝혔다.

## 분석

### 이 분석의 설득력은 “말”이 아니라 “패킷”에서 나온다

이 글이 다른 유사한 프라이버시 폭로 글들과 구별되는 지점은, 도구의 문서나
약관을 근거로 삼지 않고 실제 네트워크 트래픽을 캡처해 검증했다는 데
있다.
canary 마커를 심어 “읽지 않은 파일도 전송되는가”를 실험으로 확인하고,
캡처한 git bundle을 실제로 clone해서 파일 내용을 복구하는 절차는 반박하기
어려운 수준의 증거를 제공한다.
AI 코딩 도구에 대한 신뢰 문제는 종종 “약관에 뭐라고 쓰여 있는가”의
논쟁으로 흐르기 쉬운데, 이 글은 그 논쟁을 우회하고 “실제로 무엇이 전송
되는가”라는 검증 가능한 사실의 층위로 끌어내렸다는 점에서 구조적으로
강하다.

### 모델 턴 채널과 저장 채널의 분리는 우연이 아니라 설계다

저자가 발견한 27,800배의 데이터 비율 차이는 단순한 버그의 흔적이 아니라,
애초에 두 개의 독립된 파이프라인이 설계되어 있음을 보여준다.
모델이 실제로 응답을 생성하는 데 필요한 데이터(모델 턴 채널)와, 별도로
저장소 전체를 스냅샷 형태로 축적하는 데이터(저장 채널)가 분리되어 있다는
것은, 이 업로드가 단순한 기능 부산물이 아니라 의도적으로 구축된 데이터
수집 인프라라는 뜻이다.
이런 이중 채널 구조는 사용량 기반 과금(모델 턴)과는 별도의 트래픽으로
저장 업로드가 집계된다는 사실과도 맞물리는데, 이는 저장 업로드가
비즈니스 지표상으로도 별도 관리되는 대상임을 시사한다.

### “옵트아웃이 작동하지 않는다”는 사실이 이 글의 핵심 뇌관이다

이 글에서 가장 파급력이 큰 발견은 전송 규모(5GiB)가 아니라, “Improve the
model” 설정을 꺼도 저장소 업로드가 계속되고 서버가 여전히
`trace_upload_enabled: true`를 반환한다는 사실이다.
데이터가 많이 전송된다는 것 자체는 놀랍지 않을 수 있지만, 사용자가
명시적으로 선택한 설정이 실제 동작과 어긋난다는 것은 단순한 프라이버시
정책의 문제가 아니라 사용자 동의의 유효성 자체를 무너뜨리는 문제다.
이 지점에서 글의 성격은 “얼마나 많은 데이터를 보내는가”에서 “사용자에게
약속한 통제권이 실제로 존재하는가”로 바뀐다.
HN 댓글 goos도 정확히 이 지점을 짚었다.
“Improve the model” 토글이 켜져 있든 꺼져 있든 저장소 전체가 동일하게
업로드된다는 서술을 두고, 다른 AI 업체들도 데이터 수집에 옵트인한
사용자에게는 비슷한 일을 할 수 있겠지만 옵트아웃한 사용자에게까지 이를
강행하는 것은 “diabolical”(악질적)이라고 표현했다[^goos].

## 비평

### 학습 사용 여부에 대한 침묵이 결론의 무게를 스스로 낮춘다

저자는 “업로드된 데이터가 실제로 학습에 쓰인다는 증거는 없다”고 명시적으로
밝혔는데, 이는 정직한 태도이지만 동시에 이 글의 결론이 갖는 실질적
위협의 크기를 스스로 제한한다.
많은 독자에게 “코드가 전송된다”는 사실보다 “그 코드가 모델 학습에
재사용되는가”가 훨씬 중요한 질문인데, 이 글은 정확히 그 질문에 답하지
못한 채 전송 사실만 확정한다.
결과적으로 이 글은 “우려할 만한 강력한 정황 증거”를 제공하지만 “확정된
피해”를 입증하지는 못하며, 독자는 이 공백을 각자의 신뢰 수준에 따라
자의적으로 채워 넣게 된다.

### 단일 버전, 단일 저자 검증이라는 재현성의 한계

이 조사는 Grok CLI 0.2.93이라는 특정 버전을 대상으로, 한 명의 저자가
수행한 결과다.
xAI가 이 폭로 이후 동작을 변경했을 가능성, 혹은 애초에 이 동작이 특정
빌드나 지역 설정에서만 나타났을 가능성을 배제할 근거가 이 글 자체에는
없다.
소프트웨어의 네트워크 동작은 버전마다 크게 달라질 수 있고, 특히 이런
폭로성 조사 이후에는 벤더가 조용히 동작을 바꾸는 경우가 흔하다는 점을
고려하면, 이 글의 발견을 “현재도 유효한 사실”로 일반화하려면 별도의
재현 검증이 필요하다.
저자 스스로도 이를 포괄적 감사라 주장하지 않았지만, 글이 확산되는
과정에서는 이 단서가 쉽게 탈락한다.

### 다른 AI 코딩 도구와의 비교가 빠져 있어 문제의 특수성이 불분명하다

이 글은 Grok Build CLI 하나만을 대상으로 하며, GitHub Copilot, Cursor,
Claude Code, Codex 같은 경쟁 도구들이 유사한 상황에서 어떻게 동작하는지는
전혀 다루지 않는다.
저장소 전체를 업로드하는 방식이 Grok만의 독특한 설계인지, 아니면 AI
코딩 에이전트 전반에 걸쳐 흔한 패턴인지에 따라 이 글이 시사하는 위험의
성격은 크게 달라진다.
만약 유사한 패턴이 업계 전반에 퍼져 있다면 이는 개별 벤더의 문제가
아니라 AI 코딩 도구 카테고리 자체의 구조적 문제이고, 반대로 Grok만의
특이 사례라면 xAI를 특정해 문제 삼는 이 글의 프레이밍이 정확하다.
이 비교가 빠진 채로는 독자가 “이 도구를 피해야 하는가” 아니면 “이
카테고리의 도구를 쓸 때는 원래 이런 위험을 감수해야 하는가”를 판단할
근거가 부족하다.
HN 댓글 gitgud는 정확히 이 지점에서, Claude Code나 Codex 같은 네이티브
독점 코딩 에이전트 러너 전반이 구조적으로 비슷한 위험을 안고 있다고
지적하며, 다음 업데이트에서 어떤 “비밀 소스”가 추가될지 사용자가 알
방법이 없다는 점이 문제의 핵심이라고 짚었다.
그는 대안으로 opencode처럼 API를 통해 모델만 불러오는 방식이 프라이버시
측면에서는 더 안전하지만, 네이티브 에이전트 러너만큼의 성능은 포기해야
하는 트레이드오프가 있다고 덧붙였다[^gitgud].
반대로 Karmakosmik과 Geee는 에이전트가 실행된 디렉터리에 대해서는 어느
정도 전권을 갖는 것이 애초에 자연스러운 전제이며, `.env` 같은 파일에
실제 시크릿을 두는 것 자체가 잘못된 관행이라는 반박도 제기했다[^Geee].
이 반박은 저자가 채택한 위협 모델(로컬 저장소는 안전하게 보호되어야 할
경계)과, 일부 사용자가 실제로 채택하고 있는 위협 모델(에이전트에게 넘긴
디렉터리는 이미 신뢰 경계 밖) 사이의 근본적인 인식 차이를 드러낸다.

## 인사이트

### 저장소 업로드는 곧 AI 코딩 도구의 새로운 공급망 위험 표면이 된다

이 글이 직접 언급하지 않는 결과는, 저장소 전체 업로드라는 관행이 소프트웨어
공급망 보안의 새로운 취약 지점이 된다는 점이다.
과거의 공급망 공격은 의존성 패키지나 CI 파이프라인을 노렸지만, AI
코딩 CLI가 로컬 저장소 전체를 벤더 서버로 실시간 전송한다면, 그 벤더의
저장 인프라(`grok-code-session-traces` 같은 버킷) 자체가 공격자에게
매력적인 단일 실패 지점이 된다.
버킷 접근 권한이 잘못 설정되거나 유출되면, 그 버킷 하나를 통해 수많은
기업의 소스 코드와 시크릿이 한꺼번에 노출되는 시나리오가 성립한다.
이는 AI 코딩 도구를 도입하는 조직이 그 도구의 코드 품질뿐 아니라, 벤더의
저장 인프라 보안 태세까지 실사 대상에 포함해야 한다는 뜻이다.
HN 댓글 outloudvi가 던진 발상 — GCP의 사용 약관(AUP)에 위배되는 콘텐츠를
저장소에 넣어 Grok Build가 이를 업로드하게 만든 뒤, 그 버킷을 Google에
직접 신고하면 어떻게 되는지 — 은 농담처럼 보이지만, 벤더가 통제하는
업로드 파이프라인이 사용자의 의도와 무관하게 제3자 서비스의 약관 위반
콘텐츠를 실어 나르는 통로로 악용될 수 있다는 새로운 공격 표면을 정확히
짚어낸다[^outloudvi].
이는 저장소 업로드 파이프라인이 유출 경로일 뿐 아니라, 반대로 벤더의
인프라를 표적으로 삼는 공격의 매개체도 될 수 있음을 보여준다.

### 옵트아웃 불이행은 규제 대응 비용을 만들어낼 구조적 씨앗이다

“Improve the model”을 꺼도 저장소 업로드가 계속된다는 사실은, 단기적으로는
평판 문제이지만 장기적으로는 GDPR이나 CCPA 같은 개인정보 규제 프레임워크
아래에서 법적 문제로 번질 수 있는 씨앗이다.
사용자가 명시적으로 옵트아웃한 설정이 서버 측에서 무시된다는 것은, 동의를
철회할 권리(right to withdraw consent)가 기술적으로 보장되지 않는다는
증거로 규제 기관에 제출될 수 있는 유형의 사실이다.
이 글 자체는 법적 분석을 시도하지 않지만, AI 서비스 벤더들이 향후
“설정 UI와 실제 서버 동작의 일치”를 별도로 감사받아야 하는 규제 압력에
직면할 가능성을 이 사례가 앞당겨 보여준다.

### 트래픽 분석은 AI 도구에 대한 신뢰를 벤더 주장에서 독립시키는 실용적 방법론이 된다

이 글의 진짜 가치는 폭로된 사실 자체보다, mitmproxy와 canary 마커를
결합한 방법론이 재현 가능하고 다른 도구에도 적용 가능한 검증 절차라는
점에 있다.
지금까지 AI 코딩 도구의 데이터 처리 방식은 대부분 벤더의 문서나 공개
성명에 의존해 평가되어 왔는데, 이 글은 개별 개발자나 보안팀이 직접
검증할 수 있는 구체적 절차(프록시 설치, 마커 삽입, 업로드 큐 검사, git
bundle 복구)를 제시함으로써 신뢰 검증을 벤더의 자기 신고에서 독립시킨다.
앞으로 기업의 보안팀들이 신규 AI 코딩 도구를 도입하기 전에 이런 종류의
자체 트래픽 감사를 표준 온보딩 절차로 채택하는 흐름이 생길 가능성이
높으며, 이는 “도구 설명서를 믿는다”에서 “도구의 실제 동작을 검증한다”로
AI 도구 채택 문화 자체를 바꾸는 계기가 될 수 있다.
실제로 HN 댓글 phaseleza는 이런 신뢰 검증을 넘어 사전 방어 체계를
구축한 사례를 공유했다.
bubblewrap으로 코딩 도구를 샌드박싱해 작업 디렉터리 이외에는 읽을 수
없게 하고, `.git`은 읽기 전용으로 마운트하며, 민감한 디렉터리는 빈
디렉터리로 가려 보이지 않게 하고, 네트워크는 Unix 소켓 위의 HTTP
프록시를 거쳐 특정 LLM 제공자 호스트로만 나가도록 격리했다는 것이다[^phaseleza].
이는 벤더의 선의나 설정 UI를 신뢰하는 대신, 애초에 도구가 위반할 수 없는
운영체제 수준의 경계를 사용자 스스로 구축하는 접근으로, 이 글이 제시한
트래픽 분석보다 한 단계 더 나아간 실용적 대응이라 할 수 있다.
