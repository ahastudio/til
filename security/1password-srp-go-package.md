# 1Password/srp — Go로 구현한 SRP 프로토콜 패키지

<https://github.com/1Password/srp>

## 소개

1Password(구 AgileBits)가 공개한 Go 언어용 SRP(Secure Remote Password) 구현 패키지다.
RFC 2945(SRP 인증 및 키 교환 시스템)와 RFC 5054(TLS 인증을 위한 SRP 프로토콜)를
구현하지만, 해싱과 패딩 방식이 이 표준들과 호환되지 않는다.
1Password 팀의 인증 과정 일부를 지원하기 위해 개발됐으며,
일반적인 용도로도 사용할 수 있도록 오픈소스로 공개됐다.

## SRP 프로토콜 작동 방식

SRP는 서버가 사용자의 비밀번호를 알지 못해도 인증할 수 있게 하는 프로토콜이다.
서버와 클라이언트가 서로 장기 비밀(long-term secret)을 알고 있음을 증명하고,
공유 세션 키(K)를 도출한다.

핵심 개념은 다음과 같다.

- **x**: 클라이언트 장기 비밀. KDF(키 유도 함수)를 통해 사용자 비밀번호에서 파생한다.
- **v**: 서버 검증자(verifier). 클라이언트가 `v = g^x`로 계산해 등록 시 서버에 전송한다.
  서버는 비밀번호 대신 `{I, s, v}`를 저장한다.
- **A, B**: 에피머럴(ephemeral) 공개 키. 클라이언트가 `A = g^a`를, 서버가 `B = kv + g^b`를 전송한다.
- **u**: `H(A, B)`로 계산하는 무작위 스크램블링 파라미터.

세션 키 계산:

```text
클라이언트: S = (B - kg^x)^(a + ux) → K = H(S)
서버:       S = (Av^u)^b            → K = H(S)
```

양측이 동일한 K를 도출한 뒤, M() 증명 교환으로 키 일치를 검증해야 인증이 완료된다.

## 주요 API

패키지의 핵심은 `SRP` 타입이다. 서버와 클라이언트 양쪽에 사용한다.

```go
// 서버 측
server := NewSRPServer(KnownGroups[RFC5054Group4096], v, nil)
server.SetOthersPublic(clientA)  // 반드시 에러 확인
serverKey, err := server.Key()

// 클라이언트 측
client := NewSRPClient(KnownGroups[RFC5054Group4096], x, nil)
client.SetOthersPublic(serverB)  // 반드시 에러 확인
clientKey, err := client.Key()
```

DH 그룹은 RFC 5054 부록 A에서 정의한 `RFC5054Group2048`부터 `RFC5054Group8192`까지
지원하며, 최소 그룹 크기 기본값은 2048비트다.

## 보안 책임 사항

패키지 문서는 사용자(consumer)가 직접 책임져야 할 네 가지를 명시한다.

1. **모든 에러 반환 확인**: 특히 `Key()`와 `SetOthersPublic()`. `A mod N == 0`인
   에피머럴 공개 키는 인증 실패로 처리해야 한다.
2. **클라이언트의 KDF 선택**: x를 파생할 때 비밀번호 크래킹에 저항하는 KDF를 사용해야 한다.
   패키지가 제공하는 `KDFRFC5054`는 권장하지 않으며 Argon2 같은 전용 KDF를 권장한다.
3. **서버의 v 보호**: v는 비밀번호 해시와 동등한 방식으로 보호해야 한다.
   탈취된 v로 서버 위장 또는 패스워드 크래킹이 가능하다.
4. **키 일치 증명**: 세션 키 K를 사용하기 전에 양측이 M() 증명으로 키 일치를 확인해야 한다.

## 분석

### 비밀번호를 서버가 알 필요 없다는 설계의 함의

SRP의 핵심 보장은 “서버가 비밀번호를 저장하지 않아도 된다”는 것이다.
전통적인 인증에서 서버는 비밀번호 해시를 저장하고, 클라이언트가 보낸 비밀번호를
해시해 비교한다. 이 방식은 서버 데이터베이스가 침해될 때 해시가 노출되고,
충분히 약한 비밀번호는 크래킹된다.

SRP는 다른 모델을 사용한다. 서버는 v(검증자)를 저장하는데, v도 비밀번호로부터
파생되므로 서버 침해 시 v가 노출되는 문제는 동일하다.
그러나 인증 과정에서 비밀번호 자체나 그 해시가 네트워크를 통해 전달되지 않는다.
MITM(중간자 공격)이 진행 중인 상황에서도 패시브 도청자는 x나 비밀번호를 얻을 수 없다.
이는 TLS를 통해 비밀번호를 전송하는 방식과 다른 보안 속성이다.

1Password의 맥락에서 이 설계가 특히 의미 있는 이유는,
사용자의 금고(vault) 복호화 키가 비밀번호에서 파생되기 때문이다.
서버가 비밀번호를 알게 되면 서버 운영자가 사용자 금고에 접근할 수 있다.
SRP는 이 신뢰 관계를 끊는다.

### RFC 비호환 선택이 드러내는 설계 트레이드오프

`doc.go`에 `BUG` 주석으로 표시된 RFC 비호환성은 솔직한 자기 공개다.
표준과의 비호환을 “버그”라고 부르는 것은 이 선택이 의도적인 트레이드오프임을 인정한다.
RFC 5054의 패딩·해싱 방식 대신 자체 방식을 쓰는 이유는 문서에 명시되지 않지만,
1Password의 세션 ID를 k 파라미터에 통합하는 커스터마이징에서 이유를 추론할 수 있다.
표준 방식(`k = H(N, g)`)을 그대로 쓰면 세션별 k를 만들 수 없다.

이 선택의 비용은 명확하다. 이 패키지로 만든 클라이언트는 RFC 5054 호환 서버와
통신할 수 없다. 반대도 마찬가지다. 1Password 생태계 내에서만 통용되는 구현이다.
일반적인 SRP 라이브러리를 찾는 개발자에게 이 패키지는 적합하지 않다.

### 보안 책임의 명시적 위임이 갖는 실용적 가치

패키지 문서가 사용자 책임을 네 가지로 명시하는 것은 암호화 라이브러리 설계에서
드문 정직함이다. 대부분의 암호화 라이브러리는 “올바르게 사용하면 안전하다”는
암묵적 가정을 문서에 묻어 놓는다. 이 패키지는 반대로 “이것은 당신이 해야 한다”를
먼저 강조한다.

KDF 선택을 사용자에게 맡기는 것은 의도적이다.
패키지는 `KDFRFC5054`를 제공하지만 Argon2 같은 더 강한 KDF를 권장한다.
비밀번호 해싱 강도는 배포 환경과 위협 모델에 따라 달라지므로,
라이브러리가 특정 KDF를 강제하는 것은 유연성을 희생한다.
그러나 이 유연성이 “기본값이 안전하지 않은” 문제로 이어질 수도 있다.

## 비평

### v 보호 문제는 SRP가 해결하지 못하는 서버 침해 시나리오를 드러낸다

패키지 문서는 v를 “비밀번호 해시처럼 보호해야 한다”고 명시한다.
이는 SRP의 핵심 보장—서버가 비밀번호를 알 필요가 없다—이 서버 침해에 대해
예상보다 약한 보호를 제공함을 암시한다.

v가 유출되면 공격자는 두 가지를 할 수 있다.
첫째, v로 서버를 위장해 사용자를 속일 수 있다(서버 가장 공격).
둘째, v를 오프라인에서 비밀번호 크래킹의 출발점으로 사용할 수 있다.
`v = g^x`이고 x가 비밀번호에서 파생되므로, 충분히 약한 비밀번호는 v에서 역산 가능하다.
이는 전통적인 비밀번호 해시 유출과 본질적으로 같은 시나리오다.

SRP가 제공하는 것은 전송 중 비밀번호 노출을 방지하는 것이지,
서버 침해에서 비밀번호를 보호하는 것이 아니다.
이 구분이 1Password의 아키텍처 문서에서는 명확히 설명되지만,
이 패키지의 README만 보면 놓치기 쉽다.

### RFC 비호환성의 실제 비용이 문서에서 과소평가된다

`doc.go`가 비호환성을 `BUG` 주석으로 표시하고 “언젠가 RFC 5054 모드를 추가할 수도 있다”고
말하는 것은 이 결정의 실제 비용을 과소평가한다.
이 패키지를 일반 SRP 구현으로 사용하려는 개발자는 표준 호환 서버와 통신할 수 없다.
README에서 이 제약이 첫 번째 경고로 나오지 않는다는 것은 설계 선택에 대한 솔직함이 부족하다.

GitHub 스타 392개는 적은 수지만, 오픈소스 암호화 라이브러리로서 의미 있는 채택이다.
이 패키지를 사용하는 개발자 중 얼마나 많은 사람이 RFC 비호환성의 의미를
완전히 이해하고 선택하는지는 의문이다. “1Password가 만들었으니 안전하겠지”라는
인식이 주의 깊은 평가를 대체할 위험이 있다.

### 최소 그룹 크기 기본값이 현재 권장 사항에 뒤처진다

패키지의 최소 DH 그룹 크기 기본값은 2048비트다.
이는 `MinGroupSize` 변수로 조정 가능하지만, 기본값이 2048비트인 것은
현재 암호학 커뮤니티의 권장 사항(3072비트 이상)에 비해 보수적이지 않다.
NSA의 Commercial National Security Algorithm Suite 2.0은
3072비트 이상을 권장하며, 장기적으로 4096비트를 목표로 한다.

물론 2048비트 DH는 현재 대부분의 위협에 대해 충분하다.
그러나 암호화 라이브러리의 기본값은 최소한의 안전이 아닌 권장 안전을 제공해야 한다.
기본값을 3072비트로 설정하고 성능이 중요한 경우에만 낮추도록 하는 것이
더 안전한 기본값 원칙에 부합한다.

## 인사이트

### 오픈소스 암호화 구현은 신뢰의 전이 메커니즘이다

1Password가 내부 SRP 구현을 오픈소스로 공개한 것은 단순한 기술 공유가 아니다.
신뢰 구축 전략이다. 사용자 인증 프로토콜의 핵심 구현을 공개하면
보안 연구자들이 이를 감사할 수 있고, 취약점이 발견되면 책임이 따른다.
이 투명성은 “우리를 믿어라”에서 “직접 확인하라”로 신뢰의 기반을 이동시킨다.

이 전략의 역설은 공개함으로써 오히려 더 안전해진다는 것이다.
폐쇄형 구현은 외부 감사를 받지 않아 취약점이 오래 유지될 수 있다.
오픈소스 구현은 취약점이 더 빠르게 발견되고 수정된다.
암호화 분야에서 “모호성을 통한 보안(security through obscurity)”이 작동하지 않는다는
원칙이 여기에 적용된다. 1Password의 공개 선택은 이 원칙의 실천이다.

### SRP의 부활은 패스워드리스 인증의 과도기적 성격을 드러낸다

SRP는 1990년대에 설계됐지만 2020년대에 재조명받고 있다.
이는 역설적이다. 패스워드리스 인증(FIDO2, WebAuthn, 패스키)으로의 전환이
진행 중인 시점에 왜 비밀번호 기반 프로토콜을 강화하는가?

답은 전환 기간의 현실에 있다. 패스워드리스 인증이 보편화되는 데는 수년에서 수십 년이
걸린다. 그 기간 동안 수십억 명의 사용자는 여전히 비밀번호를 사용한다.
SRP는 비밀번호를 없애는 것이 아니라, 비밀번호를 사용하는 동안 그 위험을 최소화한다.
이 과도기적 역할이 1Password 같은 비밀번호 관리 서비스에서 특히 중요하다.
SRP는 비밀번호 매니저의 비밀번호를 보호하는 프로토콜이라는 재귀적 구조를 갖는다.

### 사용자 책임 명시는 API 설계 철학의 근본적 선택이다

암호화 라이브러리 설계에서 “사용하기 쉽게 만들되 잘못 사용하기 어렵게 만들라”는
원칙과 “올바른 사용을 위해 사용자가 이해해야 한다”는 원칙이 충돌한다.
이 패키지는 후자를 선택했다.

이 선택은 옳다. KDF 선택, v 보호 방식, 에러 처리 전략은 배포 환경에 따라 다르며,
라이브러리가 이를 완전히 추상화하면 사용자는 중요한 결정을 인식하지 못한 채
내리게 된다. 명시적인 보안 책임 문서는 사용자가 이 결정을 의식적으로 내리도록 강제한다.

그러나 이 철학의 위험은 문서를 읽지 않는 개발자다.
암호화 라이브러리의 잘못된 사용은 조용히 실패한다.
컴파일 오류가 발생하지 않고 런타임에 즉시 오류가 나지 않으며,
나중에 보안 사고가 발생하고서야 잘못된 사용이 드러난다.
이 패키지의 명시적 경고가 실제로 개발자의 행동을 바꾸는지는
설계 의도만으로는 알 수 없다.
