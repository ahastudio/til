# Chromium 148의 Math.tanh, OS를 구분하는 지문이 되다

원문: [Your Browser Does Math Differently on Every OS, and Anti-Bot Systems Read the Bits · scrapfly.dev](https://scrapfly.dev/posts/browser-math-os-fingerprint/)

HN 토론: <https://news.ycombinator.com/item?id=48884853> (425점, 216개 댓글)

## 요약

Chromium 148부터 자바스크립트의 `Math.tanh` 호출 결과가 운영체제를 구분하는
지문으로 쓰일 수 있게 되었다.
IEEE 754 부동소수점 표준은 삼각함수나 쌍곡선함수 같은 초월함수에 대해
정확한 반올림을 요구하지 않는다.
그래서 Linux의 glibc, macOS의 `libsystem_m`, Windows의 UCRT(Universal C
Runtime)는 저마다 다른 근사 계수와 룩업 테이블, 축소(reduction) 상수를 써서
`tanh`를 구현하며, 전체 입력값의 약 25%에서 최하위 비트(ULP, unit in the
last place) 하나 정도의 차이를 만들어낸다.

Chrome 147까지는 V8이 Sun Microsystems가 만든 이식성 높은 수학 라이브러리인
fdlibm을 내부에 포함시켜 사용했기 때문에, 어떤 OS에서 실행하든 `Math.tanh`는
항상 동일한 비트 값을 반환했다.
그런데 V8 커밋 `c1486295ae5`가 이 내부 구현을 표준 라이브러리 호출인
`std::tanh`로 교체하면서, 이제 V8은 운영체제가 제공하는 `libm`을 그대로
호출하게 되었다.
이 변경은 V8 14.8.57에 처음 반영되었고, 이는 곧 Chrome 148에 해당한다.
결과적으로 `Math.tanh(0.8)` 같은 단순한 호출 하나만으로도 Linux, macOS,
Windows를 구분하는 신호가 생겼다.

원문은 이 차이를 표로 제시한다.
예를 들어 `Math.tanh(0.5)`는 세 플랫폼에서 모두
`0.46211715726000974`로 일치하지만, `Math.tanh(0.8)`은 Linux에서
`0.6640367702678491`, macOS에서 `0.664036770267849`, Windows에서
`0.6640367702678489`로 서로 다르다.
공격자 입장에서는 복잡한 수학 계산이 필요 없다.
미리 만들어둔 참조 테이블과 결과값을 한 번 비교하는 것만으로 어떤 OS의
Chrome이 접속했는지 판별할 수 있다.
이 지문 채널은 `Math.tanh` 하나에 그치지 않고, Blink 레이아웃 엔진이 축소된
값에 대해 플랫폼의 `std::sin`을 그대로 호출하는 CSS 삼각함수(`sin`, `cos`,
`tan` 등)와, macOS에서 스칼라 `libsystem_m`을 사용하는 Web Audio의
DynamicsCompressor 샘플 연산으로도 확장된다.
ARM과 x86 사이에는 FMA(fused-multiply-add) 처리와 NaN 부호 전파 방식의
차이까지 더해져, OS 수준을 넘어 아키텍처 수준의 구분도 가능하다.

원문을 쓴 회사 Scrapfly는 웹 스크레이핑 및 안티봇 우회 서비스를 제공하는
곳으로, 이 지문 채널을 실제로 스푸핑(위장)하는 방법도 상세히 설명한다.
대상 `libm`에서 계수를 추출해 FMA 연산까지 그대로 재현한 이식 가능한 C
코드로 옮기거나, Windows의 `ucrtbase.dll`을 메모리에 직접 매핑해 위치 독립
코드를 그대로 호출하는 방식이다.
Windows x64 호출 규약을 맞추기 위해 `__attribute__((ms_abi))`를 사용하고,
타이밍 기반 탐지를 피하기 위해 하드웨어 FMA를 활성화한 채 컴파일한다.
검증은 릴리스마다 87만 1천 개의 입력값을 실제 macOS 기기의 스칼라·Accelerate
결과, 그리고 디버깅 프로토콜로 얻은 진짜 macOS Chrome 결과와 대조하는
방식으로 이루어진다.
다만 macOS 안에서도 Apple Silicon의 스칼라 `libsystem_m`과 Accelerate
프레임워크의 벡터 루틴(`vvsin`, `vvtanh`)은 서로 다른 코드여서, 백만 개
입력값 기준으로 10~89%가 서로 어긋난다는 점이 스푸핑을 더 어렵게 만든다.

## 분석

### 성능 최적화가 프라이버시 경계를 뚫는 구조

이번 사례의 핵심은 V8 개발팀이 보안이나 프라이버시를 훼손하려는 의도가
전혀 없었다는 점이다.
`std::tanh`로 교체한 것은 십중팔구 코드 유지보수를 단순화하거나 성능을
끌어올리기 위한 선택이었을 것이다.
운영체제가 이미 최적화해 둔 `libm`을 재사용하면 V8이 자체 구현을 유지·검증할
필요가 없어진다.
하지만 바로 이 선택이 “OS마다 결과가 달라도 상관없다”는 전제를 깔고 있고,
그 전제가 브라우저 지문 회피라는 전혀 다른 요구와 충돌한다.
성능 개선과 크로스 플랫폼 일관성은 애초에 양립하기 어려운 목표이며, 이번
사건은 그 긴장이 실제로 터진 사례다.

### 지문 채널은 늘 컴포넌트 경계에서 새어 나온다

`Math.tanh`가 문제인 이유는 함수 자체의 결함이 아니라, 자바스크립트 엔진이라는
추상화 계층과 OS 수학 라이브러리라는 하위 계층 사이의 경계에서 정보가
새어 나오기 때문이다.
지금까지 알려진 굵직한 지문 채널들, 캔버스 렌더링, 오디오 컨텍스트, 폰트
렌더링 등도 전부 같은 구조를 공유한다.
브라우저가 표준을 지키면서도 세부 구현을 하위 시스템에 위임하는 지점마다
플랫폼 차이가 새어 나올 잠재적 통로가 생긴다.
`Math.tanh`는 그 목록에 추가된 최신 항목일 뿐, 근본 구조가 새로운 것은
아니다.

### 비대칭적 노출이 만드는 탐지 가능성

원문이 강조하듯, 스푸핑을 어렵게 만드는 요인은 `Math.tanh`만 `libm`을
호출하고 나머지 `Math.*` 함수 대부분은 여전히 번들된 llvm-libc를 쓴다는
비대칭성이다.
공격자가 `Math.tanh` 결과만 골라서 다른 OS인 척 위장하면, 같은 브라우저
안에서 `Math.tanh`는 macOS 신호를 내면서 다른 수학 함수는 Linux 신호를
내는 모순이 발생한다.
이 모순 자체가 새로운 탐지 신호가 된다.
즉 지문 채널 하나를 막으려는 시도가 오히려 더 정교한 두 번째 탐지 계층을
만들어내는 역설이 생긴다.

### 안티봇 업계의 이해관계와 공개의 이중성

이 글을 쓴 Scrapfly는 웹 스크레이핑 서비스를 판매하는 회사이며, 지문 회피는
그들의 핵심 상품 경쟁력과 직결된다.
따라서 이 글은 순수한 보안 연구라기보다, 자사가 이미 대응책을 구현했다는
것을 보여주는 마케팅 성격의 기술 문서에 가깝다.
동시에 이 공개는 안티봇 벤더와 탐지 회피 업체 사이의 군비 경쟁이 얼마나
세밀한 수준(부동소수점 최하위 비트)까지 내려갔는지를 드러내는 자료이기도
하다.
공개함으로써 방어 측(브라우저 벤더, 안티봇 서비스)도 같은 정보를 얻게
되므로, 결과적으로 이 채널의 수명은 오히려 짧아질 가능성이 크다.

## 비평

### “취약점”이라는 프레이밍은 과장이다

이 글의 제목과 논조는 마치 보안 취약점을 발견한 것처럼 읽히지만, 실제
내용은 IEEE 754가 애초에 보장하지 않는 동작에 의존한 지문 채널일 뿐이다.
`Math.tanh`의 반환값이 OS마다 다른 것은 명세 위반이 아니라 명세가 애초에
허용하는 범위 안의 차이다.
V8 팀 입장에서는 고칠 “버그”가 없다.
고치려면 다시 fdlibm 같은 고정 구현으로 되돌려야 하는데, 이는 성능
회귀라는 실질적 비용을 감수해야 하는 트레이드오프다.
HN 댓글에서는 오히려 근본 해법으로 "정확하게 반올림되는(correctly
rounded)" 초월함수 구현을 밀어붙여야 한다는 의견이 나왔고, 최근 학계
연구(ARITH 학회)에서 이 문제가 사실상 풀렸다는 지적도 있었다[^sjrd].
실제로 최신 glibc는 CORE-MATH 프로젝트의 정확한 반올림 `tanh`를 쓰기
때문에, 원문이 인용한 Linux 수치 자체가 이미 최신 버전에서는 달라졌다는
반박도 나왔다[^fweimer].
다만 그 반박을 한 당사자조차 다른 초월함수들은 여전히 저마다 고유한
지문을 남긴다는 점은 인정했다.

### 실전 효용은 저자의 주장보다 훨씬 제한적이다

원문은 OS 판별이라는 프레임을 앞세우지만, HN 댓글에서 지적되었듯 이 기법의
실질적 가치는 OS 자체를 알아내는 데 있지 않다.
User-Agent 문자열만 읽어도 OS 정보는 이미 노출되어 있기 때문이다.
진짜 쓸모는 User-Agent를 위조한 봇이나 헤드리스 브라우저가 실제로 주장하는
OS에서 실행되고 있는지를 검증하는 데 있으며, 이는 훨씬 좁고 구체적인
용도다.
제목이 “OS를 구분할 수 있다”는 인상을 주는 것과 실제 활용 시나리오
사이에는 상당한 간극이 있다.
HN에서 가장 많은 지지를 받은 댓글도 정확히 이 지점을 짚었는데, “OS를
macOS라고 주장하면서 Linux 수치를 반환하면 스스로 User-Agent와 모순되는
것”이라는 지적과 함께, 대다수 지문 인식 시스템은 OS 자체보다 준-고유한
신호의 조합에 관심이 있을 뿐이라는 점을 상기시켰다[^aurornis].
게다가 재현 검증 과정에서도 허점이 드러났는데, 한 참가자가 Windows용
Firefox에서 `Math.tanh(0.8)`을 직접 실행해 보니 원문이 Linux 값이라고
표로 제시한 결과가 나왔다며, 왜 크로미움 계열 브라우저 하나만으로
검증했는지 의문을 제기했다[^hotsalad].

### 필자가 스스로 이해관계자라는 사실을 가볍게 다룬다

Scrapfly는 안티봇 우회를 파는 회사이고, 이 글은 사실상 “우리가 이 지문
채널을 이미 뚫었다”는 역량 과시에 가깝다.
그런데 글에는 책임 있는 공개(responsible disclosure) 절차나 V8 팀과의
사전 협의를 거쳤다는 언급이 전혀 없다.
방어책을 먼저 벤더에 알리고 패치 시간을 준 다음 공개하는 절차 대신, 곧바로
스푸핑 구현 세부사항까지 공개해 버렸다.
이는 안티봇 탐지를 우회하려는 다른 스크레이핑 업체들에게도 즉시 실전 매뉴얼을
제공한 셈이며, 자사의 우회 능력을 알리려는 동기가 공개 방식 자체를
좌우했다고 볼 수 있다.
HN에서도 이 지점이 가장 날카롭게 지적되었는데, 한 참가자는 "모든 지문
인식 기법을 AI로 분석해 공개하고, 여론의 반발로 벤더가 고치게 만들어
자사 스크레이핑 사업을 더 키우려는 영리한 수"라고 꼬집었고, 지문 인식
회사(fingerprint.js 등) 쪽에서 나오는 글이 차라리 동기가 명확해서 낫다고
평했다[^jeroenhd].
Scrapfly 자체 소유의 스크레이핑 브라우저 "Scrapium"이 OS별로 수학 함수
결과를 위조한다는 사실을 원문에서 직접 언급한 대목을 짚으며, "이 회사가
없었다면 지문 인식이 이렇게까지 만연하지 않았을 것"이라고 비판한
댓글도 있었다[^zombot].

### 글쓰기 품질이 내용의 신뢰도를 갉아먹는다

HN 댓글들은 이 글이 LLM으로 작성되었다는 사실이 최초 게시 이후에야
슬그머니 고지되었다는 점을 지적했다.
한 참가자는 "흥미로운 취재 내용이 LLM 특유의 티가 나는 문체 때문에
얼룩졌다"며 "당신은 ~을 만드는 것이 아니라 ~을 하는 것이다" 같은 전형적인
LLM식 대구 표현을 예로 들었다[^dmitrygr].
다른 참가자는 아예 "이것은 AI 슬롭(AI slop)이며, 웹을 망치는 데 일조하는
회사를 위해 쓰인 글"이라고 더 신랄하게 평했다[^simondotau].
반대로 원문의 첫 문장("One tanh call on the right input is a per-OS
signature")을 그대로 인용하며 "고맙다, Claude"라고 반응한 댓글도 있어[^retr0id],
LLM 저작 여부에 대한 반응 자체가 극과 극으로 갈렸다는 점도 흥미롭다.
기술적으로는 흥미로운 내용을 담고 있음에도, 이런 서술 방식과 뒤늦은
고지는 독자가 원문의 다른 주장들, 예컨대 87만 1천 개 입력값 검증 같은
수치를 얼마나 그대로 신뢰해야 할지에 대한 의구심을 키운다.

## 인사이트

### 지문 채널의 발견과 패치는 군비 경쟁의 한 라운드일 뿐이다

`Math.tanh` 지문 채널은 발견되는 즉시 두 갈래 반응을 촉발한다.
브라우저 벤더는 fdlibm으로 되돌리거나 결과를 정규화하는 패치를 검토하고,
안티봇 업체는 이미 스푸핑 구현을 완성해 배포한다.
어느 쪽이 이기든 이 라운드가 끝나면 다음 채널이 등장할 뿐이다.
캔버스 지문, 오디오 지문, 폰트 렌더링 지문이 차례로 등장하고 각각에 대한
방어와 회피가 반복되어 온 역사가 이미 있다.
`Math.tanh`는 그 목록에 이름 하나가 추가된 것에 가까우며, 근본적인 구조를
바꾸지 않는 한 이런 사이클은 계속된다.
HN에서도 이 싸움 자체가 이길 수 있는 싸움인지에 대한 회의적인 시각이
나왔는데, 실행 시간 비교와 반올림 오차 같은 신호를 충분히 많이 조합하면
결국 OS는 물론 정확한 기기, 심지어 같은 기기에서 동시에 돌아가는 다른
작업까지 추론할 수 있을 것이라는 전망과 함께, "이건 현관문의 자물쇠 같은
것이다.
자물쇠는 침입을 막지 못한다.
몇 명을 단념시키고 몇 명을 지연시킬 뿐이다"라는 비유가 제시되었다[^jmward01].

### 웹 표준의 “구현 자유”가 프라이버시 비용으로 되돌아온다

IEEE 754와 ECMAScript 명세가 초월함수의 정확한 반올림을 강제하지 않은
이유는 원래 성능과 이식성을 위한 실용적 타협이었다.
1980~90년대에는 이 자유가 각 플랫폼이 하드웨어에 맞게 수학 라이브러리를
최적화할 수 있게 해주는 합리적 설계였다.
그런데 브라우저가 모든 사람의 범용 실행 환경이 된 오늘날, 같은 자유가
정반대의 효과, 즉 플랫폼 식별 신호로 작동한다.
이는 오래된 표준의 설계 목표(이식성과 성능)와 오늘날의 요구(프라이버시와
균일성)가 정면으로 부딪히는 구조적 예시이며, POSIX 시스템 콜의 미묘한 동작
차이가 VM 탐지에 쓰이는 것과 같은 계열의 문제다.
표준을 지금 다시 쓴다고 해도 성능 손실 없이 이 긴장을 완전히 없앨 수는
없다.

### 완벽한 지문 회피는 일관성이라는 새로운 비용을 요구한다

이 글이 드러내는 가장 흥미로운 트레이드오프는, 지문 하나를 위장하려면
그와 연결된 다른 모든 신호까지 함께 위장해야 한다는 점이다.
`Math.tanh`만 다른 OS인 척 꾸미면 나머지 `Math.*` 함수나 CSS 삼각함수와
불일치가 생겨 오히려 더 강력한 탐지 신호가 된다.
그래서 Scrapfly의 대응은 단일 함수 패치가 아니라, 여러 코드 경로(JS
Math, CSS 삼각함수, Web Audio)를 모두 일관되게 재현하는 전체 시스템
수준의 작업이 되었다.
이는 지문 회피가 국소적 땜질로는 통하지 않고, 갈수록 원본 환경 전체를
통째로 시뮬레이션해야 하는 방향으로 수렴한다는 것을 보여준다.
장기적으로는 이 비용 곡선이 회피 기법의 실효성 자체를 제한하는 요인이 될
수 있다.
