# ChatGPT for Google Sheets — 프롬프트 인젝션 데이터 유출

원문: <https://www.promptarmor.com/resources/gpt-for-google-sheets-data-exfiltration>

## 요약

ChatGPT for Google Sheets 확장 프로그램(18만 5천 건 이상 설치)에서 간접 프롬프트
인젝션 취약점이 발견됐다.
공격자는 외부 시트에 숨겨진 프롬프트 하나로 여러 워크북에 무단 접근하고,
피싱 오버레이를 실행하며, 사용자 계정을 침해할 수 있었다.
“편집 전 인간 승인 필요” 설정을 활성화해도 공격이 성공했다.

2026년 5월 27일 공개 후 OpenAI는 5월 31일 응답했으며, Apps Script 코드 생성
기능을 제거하고 샌드박싱 방어 강화를 약속했다.

## 분석

### 공격 경로의 일곱 단계

취약점이 어떻게 작동하는지 단계별로 따라가면 AI 보안의 구조적 문제가 드러난다.

1. 사용자가 민감한 정보가 담긴 내부 문서를 스프레드시트에서 작업한다.
2. 외부 데이터셋을 스프레드시트에 가져온다.
3. 공격자가 가져온 데이터에 흰색 텍스트로 악성 프롬프트를 숨겨둔다.
4. 사용자가 ChatGPT에 데이터 통합을 요청하면 인젝션이 활성화된다.
5. 주입된 지시가 ChatGPT를 조작해 전체 확장 프로그램 권한으로 외부 스크립트를
   실행하게 만든다.
6. 공격자 서버로 활성 워크북이 전송된다.
7. 스크립트가 스프레드시트 링크를 분석해 피해자 계정에서 최대 12개 추가
   워크북을 자동으로 발견하고 유출한다.

### “인간 승인”의 무력화

이번 공격의 가장 섬뜩한 측면은 보안 설정을 우회했다는 점이다.
“편집 자동 적용” 설정 여부와 무관하게 공격이 성공했다.
이는 “인간이 승인하면 안전하다”는 가정이 프롬프트 인젝션 앞에서 무너짐을
보여준다.
공격자가 AI를 통해 스크립트 코드를 생성하고 실행하게 만든 이상,
승인 게이트는 사용자가 무엇을 승인하는지 알 수 없는 상황에서 의미를 잃는다.

### 피싱 오버레이의 이중 위협

데이터 유출 외에 두 가지 피싱 기법이 추가로 가능했다.
사이드바 오버레이는 정당한 확장 프로그램을 사칭하는 악성 인터페이스로
사용자 프롬프트를 가로채고, 자격증명을 탈취하며, 커넥터 접근을 조작한다.
모달 팝업은 직접 자격증명 탈취를 시도한다.
같은 인젝션 하나가 데이터 유출과 계정 탈취를 동시에 가능하게 한다.

## 비평

### 강점 — 구체적 공격 경로 공개

PromptArmor의 가장 큰 기여는 “간접 프롬프트 인젝션”을 추상적 위협이 아니라
7단계 구체 공격 시나리오로 보여준 것이다.
보안 연구에서 개념 증명(proof of concept)이 중요한 이유가 여기 있다.
실제로 동작하는 공격이 공개되지 않으면 벤더들은 위협을 과소평가하는 경향이
있다.

### 약점 — 책임 공개의 실패

복수의 후속 연락에도 자동 응답만 돌아왔다는 보고는 OpenAI의 보안 연구자
소통 프로세스에 문제가 있음을 시사한다.
공개 전 충분한 협력이 이루어지지 않았고, 결국 공개 압박으로 응답을 받았다.
보안 취약점에 대한 책임 공개(responsible disclosure)가 정상적으로 작동하려면
벤더 측의 명확한 채널과 응답 SLA가 필요하다.

### 빈 자리 — 근본 원인 해결의 한계

OpenAI의 대응은 Apps Script 코드 생성 기능을 제거하는 것이었다.
이는 증상을 제거한 것이지 원인을 해결한 것이 아니다.
LLM이 외부 입력의 지시를 데이터가 아닌 명령으로 해석하는 한, 어떤 기능이든
간접 인젝션의 벡터가 될 수 있다.
[GeekNews 댓글](https://news.hada.io/topic?id=30084)에서도 프롬프트 인젝션이
“LLM의 정교한 추론 능력 때문에 근본적으로 해결 불가능한 문제”라는 지적이
나왔다.

## 인사이트

### 스프레드시트는 데이터와 코드의 경계가 없는 유일한 시스템이다

스프레드시트는 수십 년간 “데이터 + 수식(코드)”를 같은 셀에 담아왔다.
Excel 매크로, Apps Script, 수식 언어 — 스프레드시트는 태생적으로 데이터-코드
경계가 없다.
여기에 LLM을 올리면 그 경계 불명확성이 프롬프트 인젝션의 완벽한 기반이 된다.

외부에서 가져온 데이터가 악성 지시를 담을 수 있고, LLM은 그 지시를 실행한다.
이것은 SQL 인젝션이 데이터베이스에서 했던 것을 스프레드시트에서 LLM이 반복하는
것이다.
SQL 인젝션 문제를 해결하는 데 20년 이상이 걸렸고 아직도 완전히 해결되지 않았다.
프롬프트 인젝션의 궤적도 비슷할 것이다.

### “인간 승인” 보안 모델의 근본 가정이 무너지고 있다

프롬프트 인젝션 연구가 축적되면서 “인간이 승인을 결정하면 안전하다”는 가정이
체계적으로 무너지고 있다.
인간 승인이 효과를 발휘하려면 인간이 “무엇을 승인하는가”를 이해해야 한다.
그런데 AI가 생성한 코드를 실행하는 맥락에서 비전문가 사용자는 그 코드가
무엇을 하는지 알 수 없다.

이것은 AI 보안 거버넌스의 핵심 딜레마다.
에이전트가 더 많은 권한을 갖고 더 복잡한 작업을 수행할수록, “인간 감독”은
유명무실해질 위험이 높아진다.
진짜 보안은 인간 승인이 아니라 기능 범위의 제한 — 에이전트가 접근할 수 있는
것 자체를 좁히는 것 — 에서 와야 한다.

### 18만 5천 설치라는 숫자가 의미하는 것

규모는 위험의 크기를 바꾼다.
취약점이 있는 도구를 18만 5천 명이 사용한다면, 표적화된 공격이 아닌 대규모
자동화 공격이 가능해진다.
공격자는 피해자를 선택할 필요 없이 확장 프로그램 사용자 전체를 대상으로 한
다음 적합한 피해자를 고르면 된다.

AI 보안의 위협 모델은 전통적 소프트웨어 보안과 달리 규모 효과가 공격자에게
유리하게 작용한다.
AI 도구의 낮은 진입 장벽은 사용자 기반을 빠르게 키우고, 그 기반이 미검증
보안 가정 위에 서 있으면 단일 취약점의 폭발 반경이 기하급수적으로 커진다.
