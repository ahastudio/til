# Claude Code의 프롬프트 스테가노그래피

원문: <https://thereallo.dev/blog/claude-code-prompt-steganography>

HN 토론: <https://news.ycombinator.com/item?id=48734373> (1429점, 409개 댓글)

Lobste.rs 토론: <https://lobste.rs/s/qs2sxd/claude_code_is_steganographically>

## 요약

Thereallo가 2026년 6월 30일에 발표한 리버스 엔지니어링 보고서다.
저자는 프라이버시 점검 목적으로 Claude Code(v2.1.196) 바이너리를 분석하다가,
시스템 프롬프트의 날짜 문자열에 분류 마커가 숨겨져 있다는 것을 발견했다.

핵심 발견은 `Vla()` 함수다.
이 함수는 에이전트 컨텍스트를 구성할 때 삽입되는 날짜 문자열을 생성한다.
일반적으로는 `Today's date is 2026-06-30.`처럼 보이지만,
`ANTHROPIC_BASE_URL` 환경 변수가 설정돼 있을 때 두 가지를 변경한다.
첫째, 아포스트로피 문자를 `'`(일반)에서 `’`, `ʼ`, `ʹ` 중 하나로 바꾼다.
둘째, 시스템 타임존이 `Asia/Shanghai` 또는 `Asia/Urumqi`이면 날짜 구분자를 `-`에서 `/`로 바꾼다.

아포스트로피 인코딩은 4가지 상태를 구분한다.
`ANTHROPIC_BASE_URL`의 호스트가 알려진 도메인 목록에도 없고 AI 랩 키워드도 없으면 일반 아포스트로피(`'`)를,
알려진 도메인이면 오른쪽 단일 인용부호(`’`)를,
랩 키워드를 포함하면 변형 문자 아포스트로피(`ʼ`)를,
둘 다 해당하면 변형 문자 프라임(`ʹ`)을 삽입한다.
이 문자들은 모노스페이스 폰트에서 시각적으로 거의 구별되지 않는다.

도메인 목록과 키워드 목록은 XOR 키 91과 base64로 인코딩돼 있다.
복호화된 키워드 목록에는 `deepseek`, `moonshot`, `minimax`, `zhipu`, `bigmodel`, `baichuan` 등이 포함된다.
복호화된 도메인 목록은 훨씬 방대하며, 중국 기업 도메인, AI 회사 도메인,
프록시·리셀러·게이트웨이 도메인이 포함된다.
`anyrouter.top`, `claude-code-hub.app`, `openclaude.me`, `proxyai.com` 등이 그 예다.

`ANTHROPIC_BASE_URL`이 설정돼 있지 않거나 공식 `api.anthropic.com`을 가리키면
`Crt()` 함수가 조기 반환해 마킹이 수행되지 않는다.
저자는 이 기능이 API 리셀러, 무단 게이트웨이, 모델 증류(distillation) 공격 파이프라인을 탐지하려는
Anthropic의 의도에서 나왔을 것이라 추정한다.
그러나 개발자 도구가 시스템 프롬프트에 비가시적 유니코드 마커를 삽입하고
도메인 목록을 XOR로 숨기는 것은 신뢰를 요구하는 도구로서 이상한 선택이라고 비판한다.
우회 방법은 사소하다 — 호스트명 변경, 타임존 변경, 바이너리 패치면 충분하다.
그래서 실질적으로는 악의적 행위자보다 정당한 개발자를 더 많이 걸러낸다.

## 분석

### 이것은 스테가노그래피인가, 분류 신호인가

저자는 이 기술을 “프롬프트 스테가노그래피”라고 부른다.
스테가노그래피는 메시지의 존재 자체를 숨기는 기법이다.
전통적으로는 이미지에 데이터를 숨기거나 텍스트에 공백을 삽입하는 방식으로 구현된다.
여기서 사용된 기술은 가시적 문자열을 유지하면서 비가시적 유니코드 변형으로 정보를 전달한다는 점에서
스테가노그래피의 정의에 부합한다.

그러나 이것이 통신 은폐를 목적으로 하는 스테가노그래피와 다른 점이 있다.
수신자는 Anthropic의 백엔드이고, 송신자는 Claude Code 자신이다.
제3자에게 신호를 감추려는 것이 아니라,
사용자에게 보이는 프롬프트와 Anthropic이 처리하는 요청 사이에 정보를 삽입하는 구조다.
더 정확한 명칭은 “암묵적 컨텍스트 분류(implicit context classification)”에 가깝다.
그러나 이 구분이 우려를 줄이지는 않는다.

### 왜 명시적 필드 대신 숨겨진 마커를 택했는가

Anthropic이 API 요청에 별도의 헤더나 메타데이터 필드로 동일한 정보를 전달하지 않은 데는 이유가 있다.
명시적 필드는 우회가 쉽다.
게이트웨이가 `X-Classification: reseller` 헤더를 제거하는 것은 한 줄 코드다.
시스템 프롬프트 내부의 유니코드 변형을 제거하는 것은 프롬프트 전체를 파싱해야 하므로 상대적으로 어렵다.

도메인 목록을 XOR+base64로 숨긴 것도 같은 논리다.
목록이 평문으로 노출되면 리셀러들이 목록을 보고 자신의 도메인이 포함됐는지 확인하고 변경할 수 있다.
난독화는 분류 대상이 분류 로직을 손쉽게 역참조하지 못하게 막는다.
이것은 기술적으로 합리적인 선택이지만, 동시에 저자의 비판을 정당화한다.
탐지 로직을 숨겨야 한다는 사실 자체가 이 기능이 사용자에게 공개되기 어렵다는 것을 함의한다.

Lobste.rs에서 muvlon은 리셀러의 실제 수익 모델을 설명하며 탐지의 필요성에 구체적인 배경을 제공했다.[^muvlon]
고급 구독 티어는 대부분의 개인 사용자가 소모하지 못하는 넉넉한 토큰 한도를 제공한다.
리셀러는 이 여유분을 활용해 하나의 계정에 여러 고객을 묶는 구독 풀링(subscription pooling)으로 운영한다.
이것이 Anthropic 입장에서 계약 위반인지, 사용자 입장에서 새로운 형태의 공동구매인지는 법적으로 모호하다.
그러나 이 모델이 성립하려면 리셀러는 Anthropic API를 실제로 경유해야 하고,
그 경로에서 마커가 Anthropic 백엔드에 도달하는 것이 탐지 메커니즘의 핵심이다.

HN에서 edude03은 “스테가노그래피 ≠ 보안을 통한 모호함(security by obscurity)”이라는 유용한 구분을 제시했다.[^edude03]
RSA에서 개인 키를 비밀로 유지하는 것이 보안을 통한 모호함이 아닌 것처럼,
분류 신호를 비가시적으로 유지하는 것 자체가 설계의 일부라는 논리다.
이 구분은 저자의 “명시적으로 공개해야 한다”는 요구가 탐지 설계를 근본적으로 오해하고 있을 수 있음을 시사한다.

meowface는 구현의 조잡함을 지적했다.[^meowface]
같은 목적을 달성하면서 역공학을 통한 탐지 확률을 낮추는 더 교묘한 방법이 있다고 말한다.
이 분야는 “언더핸디드 코드(underhanded code)”로 알려져 있으며, Underhanded C 콘테스트에서 체계화됐다.
그는 일부 로직은 클라이언트에서 서버로 옮길 수 있었고, 클라이언트에 남아야 하는 부분도 더 설득력 있게 숨길 수 있었다고 본다.
“JS 번들이 어차피 철저히 분석될 것을 알고 귀찮지 않았던 것”일 수도 있다는 추정도 덧붙였다.

### 시스템 프롬프트 마커가 백엔드에서 무엇을 하는가

저자는 “Anthropic이 아마 백엔드에서 파싱할 것”이라고 언급하지만 더 이상 파고들지 않는다.
이 부분이 실제로 중요한 질문이다.
마커를 파싱해서 무엇을 하는가?

가능한 시나리오는 여럿이다.
요청 로깅에 분류 태그를 추가할 수 있다.
의심스러운 분류의 요청에 다른 모델 동작을 적용할 수 있다.
특정 분류의 API 사용을 차단하거나 속도 제한할 수 있다.
리셀러 탐지 데이터를 법적·계약적 조치의 증거로 수집할 수 있다.

마커가 단순 로깅에만 사용된다면 프라이버시 우려는 낮다.
그러나 마커가 모델 동작을 바꾼다면 — 예를 들어 중국 AI 랩 게이트웨이를 통과하는 요청에 다른 응답을 생성한다면 —
이것은 단순한 남용 탐지를 넘어선다.
저자는 이 가능성을 명시하지 않지만, 이것이 핵심 우려가 돼야 한다.

HN 토론에서 sebastiennight는 중요한 기술적 질문을 제기했다.[^sebastiennight]
`ANTHROPIC_BASE_URL`이 DeepSeek 같은 외부 제공자를 가리킨다면,
마킹된 시스템 프롬프트는 Anthropic이 아닌 그 제공자에게 전달된다.
그러면 Anthropic이 어떻게 이 마커를 수신할 수 있는가?
이에 pmxi는 핵심 케이스가 “Claude 리셀러”임을 명확히 했다.[^pmxi]
리셀러는 사용자 요청을 받아 Anthropic API로 전달하므로, 마킹된 프롬프트는 결국 Anthropic 백엔드에 도달한다.
모델 증류 공격의 경우도 마찬가지다 — 자신의 모델을 훈련시키기 위해 실제 Claude 응답이 필요하므로 Anthropic API를 경유한다.

### 저자가 암묵적으로 전제하는 것

저자는 “명시적 텔레메트리 필드와 문서화로 동일한 목적을 달성할 수 있다”고 주장한다.
이 전제는 탐지의 목적이 순수하게 내부 집계와 정책 집행에 있다고 가정한다.

그러나 명시적 공개에는 비용이 있다.
탐지 대상인 리셀러들이 탐지 로직을 알면 우회하기 쉬워진다.
“우리는 게이트웨이 사용을 탐지한다”고 공개하는 것과
“우리는 이 22개의 유니코드 아포스트로피 조합으로 분류한다”고 공개하는 것은 다르다.
전자는 정책 공개, 후자는 구현 공개다.
저자는 이 둘을 구분하지 않고 후자 수준의 투명성을 요구하는 것처럼 읽힌다.

## 비평

### “스테가노그래피”라는 프레임이 문제를 과대 포장한다

저자가 선택한 “스테가노그래피”라는 단어는 기술적으로 정확하지만 함의가 과도하다.
스테가노그래피는 통상 악의적 통신 은폐, 디지털 저작권 마킹, 감시 회피 기술과 연관된다.
이 맥락에 적용하면 Anthropic이 사용자를 능동적으로 기만하는 행위자처럼 보인다.

실제 구조는 더 단순하다.
Anthropic은 자사 API 클라이언트에 자사 API 백엔드로 보내는 요청에 자사가 정의한 분류 신호를 넣었다.
이 신호는 사용자 데이터가 아니라 사용자의 환경 설정(base URL, timezone)에서 파생된다.
“사용자를 속이는” 것이 아니라 “요청 환경을 기록하는” 것에 더 가깝다.
물론 이것이 바람직하지 않을 수 있지만, “스테가노그래피”라는 단어가 주는 인상보다는 덜 충격적이다.

HN에서 mrshadowgoose는 저자의 결론을 “과장됐다”고 평가했다.[^mrshadowgoose]
의도가 명백하다는 것이다 — 중국 AI 회사들의 모델 증류 탐지.
“일반 개발자를 처벌한다”는 주장이 어떤 식으로도 성립하는지 불분명하다는 반박이다.
tgtweak도 비슷하게, 리버스 엔지니어링으로 드러난 “나쁜 활동”의 전부가 중국 랩 모델 증류 탐지라면 오히려 안심이 된다고 평가했다.[^tgtweak]

Lobste.rs에서 bitshift는 비슷한 맥락에서 한 발 더 나아갔다.[^bitshift]
사용자는 이미 소스를 감사할 수 없는 클로즈드 소스 소프트웨어를 수락하고 있다는 것이다.
그러나 그는 한 가지 측면에서는 우려가 정당하다고 인정했다.
리셀러 마크업과 훈련 데이터 수익화가 결합되는 경제학 — 즉 리셀러가 단순히 API를 중개하는 것이 아니라 그 과정에서 사용자의 쿼리를 자신의 모델 훈련에 활용하는 시나리오 — 은 주목할 만한 문제라는 것이다.

### 우회 가능성을 지적하면서 탐지 실효성 논증은 생략한다

저자는 우회가 “사소하다(trivial)”고 말한다.
이것이 사실이라면 이 기능의 실효성에 대한 질문이 따라와야 한다.
그런데 저자는 그 방향으로 가지 않는다.
“우회가 쉬우므로 정당한 사용자만 걸린다”는 주장을 한다.

이 논리는 완전하지 않다.
대부분의 남용 탐지는 정교한 공격자보다 무지한 남용자를 더 많이 걸러낸다.
신용카드 사기 탐지도, 계정 도용 탐지도, API 남용 탐지도 정교한 공격자는 우회한다.
그렇다고 해서 해당 탐지가 무의미하지는 않다.
저자의 “우회 가능 → 정당한 사용자만 피해”라는 논리는 탐지 시스템에 대한 지나친 이상화다.

### 잠재적으로 심각한 질문을 본문에서 다루지 않는다

저자는 마커가 “Anthropic이 아마 백엔드에서 파싱할 것”이라고 언급하고 넘어간다.
그러나 이것이 글에서 가장 중요한 질문이다.
마커가 파싱돼서 모델 동작에 영향을 준다면, 이것은 단순한 프라이버시 문제가 아니다.
특정 API 게이트웨이를 통과하는 요청이 다른 모델 응답을 받는다는 것을 의미할 수 있다.

저자는 리버스 엔지니어링을 통해 클라이언트 코드를 분석했지만,
백엔드에서 이 마커가 실제로 어떻게 사용되는지는 확인하지 못했다.
“아마 파싱할 것”이라는 추정만으로 글을 끝내는 것은 독자에게 가장 중요한 질문을 열린 채로 남긴다.
백엔드 동작에 대한 추가 검증 없이 단정하기 어렵다는 점을 명시했어야 한다.

### 결론의 요구사항이 적대적 환경을 무시한다

저자는 “명시적 텔레메트리 필드, 문서화, 릴리스 노트에 공개”를 요구한다.
이 요구는 탐지 대상이 선의의 과실자(innocent mistake)라는 전제를 깔고 있다.

그러나 도메인 목록에는 `openclaude.me`, `proxyai.com`, `yunwu.ai`처럼
명시적으로 Claude를 리브랜딩해서 판매하는 서비스들이 포함돼 있다.
이들 중 일부는 사용자에게 Claude에 접근하는 것처럼 속이면서 다른 모델을 제공하거나,
Anthropic의 이용약관을 명시적으로 위반하며 운영한다.
이런 행위자들에게 탐지 로직을 완전히 공개하는 것이 정당한지는 논쟁의 여지가 있다.

## 인사이트

### 남용 탐지의 투명성 딜레마는 AI 도구에서 더 첨예하다

보안과 투명성의 긴장은 새로운 문제가 아니다.
은행의 사기 탐지 로직, 이메일 스팸 필터, 광고 사기 탐지는 모두 비공개다.
공개하면 회피하기 쉬워지기 때문이다.
이 원칙이 적용된 것이 이번 사례다.

그러나 AI 코딩 에이전트는 이 딜레마가 특히 날카롭다.
Claude Code는 파일 시스템 접근, 셸 실행, git 조작 권한을 요청한다.
사용자는 이 도구가 자신의 시스템에서 무엇을 하는지를 이해한다고 믿고 권한을 준다.
그 신뢰 관계에서 “우리는 당신의 API 베이스 URL을 분류해서 시스템 프롬프트에 숨깁니다”라는 사실이
공개되지 않으면, 사용자가 부여한 신뢰의 범위와 실제 동작 사이에 간극이 생긴다.
이 간극 자체가 신뢰 문제의 본질이다 — 기능의 악의성과 관계없이.

civet_java는 이 논리를 더 멀리 밀고 나갔다.[^civet_java]
“정직한 공개가 그들의 문제를 무력화한다면, 그건 나의 문제가 아니다.”
그는 이 발견이 “그 외에 내 기계에서 무엇을 수집하는가”라는 질문을 불가피하게 만든다고 지적했다.
PII 수집 가능성을 배제하기 어렵고, 이번 일이 유사한 비가시적 메커니즘이 더 있을 것이라는 추정을 뒷받침한다는 것이다.

### 이것은 브라우저 핑거프린팅과 구조가 동일하다

브라우저 핑거프린팅은 명시적 쿠키 없이 브라우저의 고유 특성 조합으로 사용자를 식별하는 기술이다.
Canvas 렌더링 결과, WebGL 파라미터, 폰트 목록, 타임존을 조합해 식별자를 만든다.
이것도 사용자에게 비가시적이며, 목적은 추적이다.

Claude Code의 마킹 메커니즘은 구조적으로 이것과 동일하다.
환경 특성(베이스 URL, 타임존)을 조합해 요청을 분류하고, 그 분류를 비가시적으로 전달한다.
차이는 브라우저 핑거프린팅이 제3자에게 정보를 전달하는 반면,
이것은 자사 백엔드에 전달한다는 것이다.
이 구분이 프라이버시 우려를 낮추는지 높이는지는 관점에 따라 다르다.

브라우저 핑거프린팅에 대한 업계의 반응은 시사적이다.
결국 브라우저 벤더들은 핑거프린팅을 줄이는 방향으로 움직였고,
일부는 적극적으로 차단하기 시작했다.
유사한 압력이 AI 도구 생태계에서도 발생할 수 있다.
특히 기업 환경에서 내부 게이트웨이를 통해 AI 도구를 라우팅하는 경우,
보안 팀은 이런 비가시적 분류 메커니즘에 민감하게 반응할 것이다.

### 리셀러 탐지와 모델 동작 변경은 분리돼야 하는 문제다

이 사례에서 가장 중요한 아직 답해지지 않은 질문은 백엔드 동작이다.
마커가 순수하게 로깅과 남용 탐지 집계에만 사용된다면,
이것은 프라이버시 정책 공개 문제로 볼 수 있다.
불편하지만 관리 가능한 수준이다.

그러나 만약 마커가 모델 응답에 영향을 준다면 — 예를 들어 특정 분류의 게이트웨이를 통한 요청에
다른 시스템 프롬프트를 삽입하거나, 모델의 특정 기능을 제한하거나, 다른 응답 품질을 제공한다면 —
이것은 완전히 다른 문제다.
사용자는 자신이 어떤 버전의 모델 경험을 받는지를 알 권리가 있다.
“동일한 모델에 동일한 요청을 보내도 게이트웨이에 따라 다른 답을 받을 수 있다”는 가능성은
AI 도구의 신뢰성 전반을 흔든다.

Anthropic이 할 수 있는 명확한 대응은 이 두 가지를 공식적으로 분리하는 것이다.
“우리는 남용 탐지를 위해 API 베이스 URL을 분석한다”는 정책 공개와,
“마커는 모델 동작에 영향을 주지 않는다”는 기술적 확약을 동시에 제공하는 것이다.

### 개발자 도구의 신뢰는 불투명한 이진법적 선택이 아니다

저자의 마지막 문장 — “신뢰는 지루한 부분에서 쌓인다” — 은 인상적인 결론이다.
그러나 이 문장이 가리키는 방향은 저자가 의도한 것보다 더 복잡하다.

Anthropic이 완전히 투명한 남용 탐지를 구현하면 탐지 효과가 줄어든다.
이것도 신뢰에 영향을 준다 — 남용을 방치하는 도구는 합법적 사용자에게도 피해를 준다.
반대로 비가시적 분류를 유지하면 투명성에 대한 신뢰가 낮아진다.
이 딜레마에 단순한 해답은 없다.

더 현실적인 경로는 중간 지점이다.
구현의 세부 사항이 아니라 정책을 공개하는 것이다.
“우리는 사용자의 API 베이스 URL을 기반으로 요청을 분류하며, 이 분류는 로깅과 정책 집행에 사용됩니다.
모델 응답에는 영향을 주지 않습니다.”
이 수준의 공개는 분류 로직을 완전히 노출하지 않으면서도
사용자가 자신의 요청이 어떻게 처리되는지를 이해할 수 있게 해준다.
이것이 저자가 말하는 “신뢰”를 구축하는 더 현실적인 방법이다.
