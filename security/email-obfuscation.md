# 이메일 주소 난독화 기법의 효과 분석

<https://spencermortensen.com/articles/email-obfuscation/>

## 요약

Spencer Mortensen이 이메일 주소를 스패머의 자동 수집(harvesting)으로부터 보호하는
다양한 난독화 기법의 실제 효과를 측정한 연구다. 일반 텍스트 보호 방식 15가지와
클릭 가능한 링크 보호 방식 10가지를 허니팟(honeypot) 시스템을 통해 테스트했다.

테스트 방법론은 자체 메일 서버에서 모든 스팸 필터링을 비활성화한 뒤, 각 난독화
기법이 적용된 이메일 주소를 웹에 노출시켜 스패머의 수집 성공률을 추적하는 것이다.
같은 스패머가 대량 발송하는 경우를 고려해 스패머 단위로 그룹화하여 집계했다.

CSS display:none, JavaScript 기반 방식(문자열 연결, 변환, AES 암호화, 사용자 상호작용),
HTML SVG 임베딩 등은 426명의 스패머 중 100%를 차단했다. HTML 엔티티 인코딩은
이론적으로 취약함에도 95%를 차단했고, HTML 주석 삽입은 99%를 차단했다.

클릭 가능한 링크 보호에서도 HTML 엔티티 인코딩과 다수의 JavaScript 방식이
399명의 스패머 중 100%를 차단하는 성과를 보였다. 기호 치환, 이미지, CSS content,
텍스트 방향 변경 등은 스팸을 차단하지만 사용성을 심각하게 해치는 것으로 나타났다.

## 분석

이 연구의 핵심 강점은 실제 스패머 데이터를 기반으로 한 실증적 접근에 있다.
이론적 취약성과 실제 공격 성공률 사이의 괴리를 명확히 보여준다. HTML 엔티티
인코딩이 대표적인데, 디코딩은 기술적으로 사소한 작업임에도 95%의 스패머가
이를 처리하지 못했다.

스패머 단위 그룹화는 통계적 정직성을 보여주는 좋은 방법론적 선택이다.
단일 스패머가 수천 통을 보낼 수 있으므로, 메시지 단위 집계는 결과를 왜곡할
수 있다. 이 접근은 "얼마나 많은 독립적 공격자가 기법을 뚫었는가"라는
실질적인 질문에 답한다.

사용성과 보안의 트레이드오프도 체계적으로 다룬다. 기호 치환(`user [at] domain
[dot] com`)은 스팸을 막지만 사용자가 주소를 복사해 쓸 수 없게 만든다.
JavaScript 기반 방식은 보안과 사용성을 모두 달성하지만, JavaScript가 비활성화된
환경에서는 작동하지 않는다는 암묵적 제약이 있다.

## 비평

연구의 가장 큰 강점은 장기간에 걸친 실증 데이터다. 대부분의 난독화 논의가
"이론적으로 이 방식이 더 안전하다"에 머무는 반면, 이 연구는 실제 수치를 제시한다.

그러나 몇 가지 한계가 있다. 첫째, 표본 편향의 가능성이다. 허니팟에 도달한
스패머 426명이 전체 스패머 생태계를 대표하는지 불분명하다. 정교한 타겟 공격을
수행하는 공격자는 허니팟을 회피할 수 있다. 둘째, 시간적 맥락이 부족하다.
"대다수의 수집기가 극도로 기초적"이라는 결론은 현재 시점에서 유효하지만,
LLM 기반 스크래핑이 보편화되면 상황이 급변할 수 있다.

셋째, JavaScript 의존 방식의 접근성 문제를 충분히 다루지 않는다.
스크린 리더 사용자, JavaScript를 차단하는 보안 의식 높은 사용자, 검색 엔진
크롤러 등에 대한 영향 분석이 빠져 있다. 넷째, CSS display:none 방식은
접근성 도구가 숨겨진 텍스트를 읽을 수 있어 의도치 않은 혼란을 야기할 수 있다는
점이 언급되지 않았다.

## 인사이트

### 공격자의 경제학이 방어자에게 유리한 드문 사례

보안 분야에서 "공격자가 유리하다"는 것은 거의 공리처럼 통용된다. 방어자는 모든
진입점을 막아야 하지만 공격자는 하나만 뚫으면 된다. 그러나 이메일 수집은 이
공식이 역전되는 희귀한 영역이다.

스패머의 수익 모델은 대량 발송에 기반한다. 개별 이메일 주소의 가치는 극히 낮으므로,
수집기에 정교한 JavaScript 엔진이나 렌더링 파이프라인을 통합하는 것은 경제적으로
정당화되기 어렵다. 단순한 정규식 매칭으로 수백만 개의 보호되지 않은 주소를
수집하는 것이 훨씬 효율적이다.

이것이 10년 이상 된 기법들이 여전히 100% 효과를 보이는 근본 원인이다.
기술적으로 뚫을 수 없어서가 아니라, 뚫을 경제적 동기가 없기 때문이다.
이는 보안 설계에서 "기술적 난이도"보다 "공격자의 경제적 인센티브"를 분석하는
것이 더 유용한 프레임워크일 수 있음을 시사한다.

### 충분히 좋은 보안(Good Enough Security)의 실증적 증거

보안 커뮤니티에는 "완벽하지 않으면 의미 없다"는 경향이 있다. HTML 엔티티
인코딩이 이론적으로 취약하다는 이유로 이를 기각하는 것이 전형적인 예다.
그러나 데이터는 다른 이야기를 한다. 95%의 차단율이면 대부분의 사용 사례에서
충분하다.

이는 보안 엔지니어링의 핵심 딜레마를 건드린다. 이론적 완벽함을 추구하면
구현 복잡도가 올라가고, 복잡도는 그 자체로 새로운 취약점의 원천이 된다.
AES 암호화로 이메일을 보호하는 것은 기술적으로 우아하지만, 구현 오류의
가능성도 함께 올라간다. 반면 HTML 엔티티 인코딩은 한 줄의 코드로 95%의
효과를 얻는다.

실용적 보안 설계에서 중요한 것은 "뚫을 수 있는가"가 아니라 "뚫릴 확률 ×
피해 규모"다. 이메일 주소 노출의 피해는 스팸 수신이라는 상대적으로 낮은
수준이므로, 95%의 차단율은 대부분의 웹사이트에서 합리적인 선택이다.

### 브라우저가 만든 의도치 않은 보안 계층

이 연구의 결과를 관통하는 패턴이 있다. 브라우저의 렌더링 파이프라인 자체가
강력한 보안 계층으로 기능한다는 것이다. CSS display:none, JavaScript 실행,
SVG 렌더링 등은 모두 완전한 브라우저 엔진을 요구한다.

스패머의 수집기는 대부분 HTTP 클라이언트 + 정규식 파서 수준에 머문다.
Headless 브라우저(Puppeteer, Playwright)를 동원하면 이 모든 기법을 무력화할
수 있지만, 수백만 페이지를 크롤링할 때 Headless 브라우저의 리소스 비용은
감당하기 어렵다.

이는 웹 플랫폼의 복잡성이 역설적으로 보안 자산이 된 사례다. 웹의 복잡한
렌더링 스택을 비판하는 목소리가 많지만, 바로 그 복잡성 덕분에 단순한 텍스트
파싱만으로는 페이지의 실제 콘텐츠를 추출할 수 없게 되었다. HTML, CSS,
JavaScript의 상호작용이 만드는 갭(gap)이 난독화의 기술적 기반이 되는 셈이다.

그러나 이 방어선은 영원하지 않을 수 있다. LLM 기반 에이전트가 브라우저를
조작하는 시대가 오면, 렌더링 비용이라는 장벽이 급격히 낮아질 수 있다.
이 연구의 결과는 현재의 스냅샷이지, 미래의 보장이 아니다.
