# axios npm 공급망 공격 분석 (2026-03-30)

> 원문:
> <https://www.stepsecurity.io/blog/axios-compromised-on-npm-malicious-versions-drop-remote-access-trojan>

## 요약

2026년 3월 30일, npm에서 가장 널리 사용되는 HTTP 클라이언트 라이브러리 중 하나인
axios의 메인테이너 계정이 탈취되어 악성 버전이 배포되었다. 공격자는
`plain-crypto-js`라는 악성 의존성을 주입하여 원격 접근 트로이목마(RAT)를
설치하는 드로퍼를 배포했다.

## 분석

### 공격의 정교함

이번 공격은 단순한 타이포스쿼팅(typosquatting)이 아니라 **정당한 메인테이너 계정
탈취**를 통한 공급망 공격이다. axios는 주간 다운로드 수가 수천만 건에 달하는
핵심 라이브러리이므로, 짧은 노출 시간에도 파급력이 크다. 타이포스쿼팅은 "이름이
비슷한 가짜 패키지"를 만들어 실수를 유도하는 방식이라 주의하면 피할 수 있지만,
정당한 패키지 자체가 오염되면 어떤 주의력으로도 막을 수 없다. 사용자는 평소와
동일하게 `npm install`을 실행했을 뿐이고, 그 행위 자체가 감염 경로가 되었다.

### 공격 전략의 특징

#### 시간대 선택

일요일 심야~월요일 새벽(UTC 기준)에 배포하여 보안 팀의 대응 시간을 최대한
벌었다. 대부분의 보안 팀은 평일 업무 시간에 모니터링 인력이 집중되므로, 주말
심야는 탐지까지의 시간을 극대화하는 최적의 창이다. 이는 우연이 아니라 의도적인
작전 계획의 증거다.

#### 이중 버전 공격

`1.x`와 `0.x` 두 메이저 라인에 동시 배포하여, 레거시 프로젝트까지 공격 범위를
넓혔다. 많은 프로젝트가 메이저 버전을 고정하고 마이너/패치만 자동
업데이트하므로, 한 메이저 라인만 공격하면 나머지는 놓친다. 두 라인을 동시에 노린
것은 npm 생태계의 버전 관리 패턴을 정확히 이해한 공격자의 전문성을 보여준다.

#### 의존성 주입 전략

악성 코드를 axios 패키지 자체에 직접 삽입하지 않고, `plain-crypto-js`라는 별도
패키지를 의존성으로 추가했다. 이는 코드 리뷰 시 "axios 소스 코드 변경"이 아닌
"의존성 추가"로 보이게 만들어 탐지 확률을 낮추는 전략이다. 또한 악성 로직을 외부
패키지에 격리함으로써, axios가 제거되더라도 `plain-crypto-js`는 다른 경로로
재활용될 수 있는 여지를 남긴다.

#### 자기소멸 메커니즘

실행 후 악성 코드를 삭제하고 정상 버전으로 위장하는 안티포렌식 기법은 탐지를
지연시킨다. `package.json`의 버전을 4.2.0으로 되돌리는 것은 `npm list`로
확인해도 정상처럼 보이게 만드는 영리한 전략이다. 사고 대응팀이 "현재 설치된
패키지 목록"을 점검해도 악성 버전의 흔적이 남지 않으므로, 감염 여부를 판단하려면
설치 시점의 네트워크 로그나 레지스트리 감사 로그까지 소급해야 한다. 이런 수준의
증거 수집은 대부분의 조직에서 현실적으로 불가능하다.

#### 크로스 플랫폼 페이로드

macOS, Windows, Linux 세 플랫폼 모두를 대상으로 각기 다른 형태의 페이로드를
준비했다. 바이너리(macOS), PowerShell 래퍼(Windows), Python 스크립트(Linux)로
분화한 것은 각 OS의 보안 모델과 탐지 도구 특성을 고려한 설계다. 이는 단독 해커가
아닌, 조직적 역량을 갖춘 공격 그룹임을 시사한다.

### 구조적 취약점

이 사건은 npm 생태계의 근본적인 약점을 드러낸다:

#### 단일 실패점

메인테이너 계정 하나가 탈취되면 수백만 프로젝트가 즉시 위험에 노출된다. 2FA를
강제하더라도 세션 하이재킹이나 토큰 탈취에는 무력할 수 있다. npm은 2022년부터
상위 패키지에 2FA를 의무화했지만, 이번 사건은 2FA만으로는 계정 보안을 보장할 수
없음을 실증했다. 세션 토큰이 탈취되면 2FA 이후의 인증 상태를 그대로 도용할 수
있기 때문이다.

#### `postinstall` 훅의 위험성

`npm install`만으로 임의 코드가 실행되는 구조는 본질적으로 위험하다. 대부분의
프로젝트가 `--ignore-scripts` 없이 설치하므로, 설치 시점에 공격이 자동으로
성공한다. `postinstall`은 네이티브 모듈 빌드 등 정당한 용도가 있어 완전히 제거할
수 없다는 점이 문제를 더 복잡하게 만든다. "편의"와 "보안" 사이의 트레이드오프가
편의 쪽으로 극단적으로 기울어진 설계다.

#### 버전 신뢰 모델의 한계

semver와 lockfile 기반의 의존성 관리는 "같은 이름 + 같은 레지스트리 = 신뢰"라는
가정에 기반한다. 이 가정은 메인테이너가 침해되면 무너진다. 레지스트리는 "누가
올렸는가"만 검증하지, "올린 코드가 안전한가"는 검증하지 않는다. 코드 서명(code
signing)이 보편화되지 않은 현재 생태계에서, 레지스트리는 사실상 무조건적 신뢰
위임 체계로 작동한다.

#### 전이적 의존성의 불투명성

axios에 `plain-crypto-js`가 추가된 것은 axios를 직접 의존하는 프로젝트의
`package.json`에는 드러나지 않는다. `node_modules` 깊숙이 설치되는 전이적
의존성은 대부분의 개발자가 인지하지 못하며, 코드 리뷰 대상에도 포함되지 않는다.
의존성 트리의 깊이가 곧 공격자의 은폐 깊이가 되는 셈이다.

## 비평

### npm의 대응: 빠르지만 충분하지 않다

악성 버전 배포 후 약 2~3시간 만에 제거한 것은 비교적 빠른 대응이다. 그러나
자동화된 CI/CD 파이프라인 시대에 2시간은 수만 건의 설치가 발생하기에 충분한
시간이다. npm이 악성 패키지를 "제거"하는 것은 사후 조치일 뿐, 이미 설치된
프로젝트에는 아무런 영향을 미치지 못한다. 제거 시점에 이미 감염된 CI/CD
파이프라인, 개발자 로컬 환경, 프로덕션 빌드는 여전히 오염된 상태로 남는다. npm에
필요한 것은 "빠른 제거"가 아니라 "배포 전 탐지" 이며, 이를 위한 자동화된 정적
분석, 행위 분석, 이상 탐지 파이프라인이 레지스트리 수준에서 작동해야 한다.

### 패키지 이름이라는 사회공학

`plain-crypto-js`라는 이름은 `crypto-js`의 변종처럼 보이도록 의도적으로
설계되었다. 코드 리뷰에서 의존성 목록을 볼 때, 이름이 익숙해 보이면 "아, 암호화
관련 라이브러리구나"하고 넘어가기 쉽다. 이는 기술적 공격인 동시에 사회공학적
공격이다. 의존성 추가를 코드 변경과 동일한 수준으로 리뷰하는 문화가 자리 잡지
않은 한, 이런 전략은 계속 유효할 것이다. "왜 이 의존성이 필요한가?"를 묻는 것이
"이 코드가 올바른가?"를 묻는 것만큼 중요하다.

### 메인테이너 계정 보안의 책임 소재

탈취된 `jasonsaayman` 계정의 이메일이 `ifstap@proton.me`로 변경되어 있었다는
것은 계정 복구 경로까지 차단당했음을 의미한다. 이는 단순한 비밀번호 유출이
아니라, 계정에 대한 완전한 통제권 탈취다. 문제는 이 메인테이너 한 명의 계정 보안
수준이 axios에 의존하는 수백만 프로젝트의 보안 수준을 결정한다는 것이다. npm은
메인테이너에게 2FA를 요구하지만, 세션 관리 정책, 이메일 변경 시 알림, 비정상
로그인 탐지 등 플랫폼 수준의 보호 장치가 충분했는지 의문이다.

### 사전 준비: `plain-crypto-js@4.2.0`의 의미

공격자는 악성 버전(4.2.1) 배포 18시간 전에 정상적인 4.2.0을 먼저 배포했다. 이는
레지스트리의 자동 스캐너나 이상 탐지를 우회하기 위한 사전 포석이다. 새로운
패키지가 처음 등록될 때 검사가 집중되므로, 먼저 깨끗한 버전으로 "정상
패키지"라는 이력을 만든 뒤 악성 업데이트를 밀어 넣는 전략이다. 이런 단계적
접근은 공격자가 npm의 탐지 메커니즘을 사전에 연구하고 우회 전략을 수립했음을
보여준다.

### C2 인프라의 단순함이 주는 교훈

C2 서버가 `sfrclak.com:8000`이라는 단일 도메인/포트로 운영된 것은 의외로
단순하다. CDN이나 도메인 프론팅(domain fronting) 같은 고급 은폐 기법을 사용하지
않았다는 것은, 공격자가 "탐지되기 전에 충분한 피해를 입히면 된다"는 판단을
했음을 시사한다. 실제로 공격 성공 후 C2 서버가 차단되더라도, RAT가 이미 설치된
시스템에서는 별도의 통신 채널로 전환할 수 있으므로 C2 차단만으로는 피해를 완전히
막을 수 없다.

## 인사이트

### 신뢰의 전이 문제

소프트웨어 공급망에서 신뢰는 전이된다. axios를 신뢰한다는 것은 axios의
메인테이너를 신뢰한다는 뜻이고, 그 메인테이너의 이메일 보안, 비밀번호 관리, 2FA
설정까지 암묵적으로 신뢰한다는 뜻이다. 의존성 트리가 깊어질수록 이 신뢰 사슬은
기하급수적으로 늘어나며, 가장 약한 고리 하나가 전체를 무너뜨린다. 이번 사건은
"코드를 신뢰하는 것"과 "사람을 신뢰하는 것"이 구분되지 않는 현재 패키지 생태계의
본질적 한계를 보여준다.

### npm은 특별하지 않다

이번 공격이 npm에서 발생했지만, 이는 npm 고유의 문제가 아니다. Maven Central,
PyPI, RubyGems, crates.io 등 모든 패키지 레지스트리가 동일한 구조적 취약점을
공유한다. 계정 하나가 탈취되면 수백만 프로젝트에 악성 코드가 전파되는 구조는
언어와 생태계를 가리지 않는다. "우리는 npm을 안 쓰니까 안전하다"는 착각이 가장
위험한 보안 허점이다.

### `postinstall`은 빙산의 일각

이번 공격은 `postinstall` 훅을 사용했지만, 설치 시점의 코드 실행은 공격 벡터 중
하나일 뿐이다. 빌드 플러그인, 코드 생성기, 애너테이션 프로세서 등 빌드
파이프라인의 모든 단계가 잠재적 실행 지점이다. Gradle 플러그인이 `buildscript`
블록에서 원격 코드를 가져오는 것도, Maven 플러그인이 빌드 시점에 네트워크 호출을
하는 것도 동일한 위험이다. "설치" 뿐 아니라 "빌드" 전체가 공격 표면이라는 인식이
필요하다.

### 탐지의 비대칭성

공격자는 한 번만 성공하면 되지만, 방어자는 매번 성공해야 한다. 이번 공격의
안티포렌식 기법—자기소멸, 버전 위장—은 "감염된 적이 없는 것처럼 보이게 만드는"
전략이다. 전통적인 보안 모니터링은 "비정상적인 상태"를 탐지하지만, 공격이 흔적을
지우면 모니터링 시점에는 이미 정상으로 보인다. 이는 사후 탐지 중심의 보안 모델이
공급망 공격에 근본적으로 불리함을 의미한다.

### 속도가 무기가 된 시대

CI/CD 자동화는 개발 속도를 높였지만, 동시에 공격의 전파 속도도 높였다.
`npm install`이 자동으로 실행되는 파이프라인에서는 악성 버전이 배포된 지 수 분
만에 프로덕션에 도달할 수 있다. "빠른 배포"를 추구하면서 "빠른 검증"은 따라가지
못하는 비대칭이 존재한다. 쿨다운 윈도우, 이그레스 허용 목록, 설치 스크립트 차단
같은 속도 제한 장치가 파이프라인에 내장되어야 한다.

### lockfile은 방패가 아니다

lockfile이 의존성을 고정한다고 믿지만, lockfile은 "최초 설치 시점의 스냅샷"일
뿐이다. `npm install`이 lockfile을 업데이트하는 순간, 또는 lockfile 없이 CI가
도는 순간, 고정은 풀린다. 더 근본적으로, lockfile은 "이 버전을 설치하라"고
지시하지만 "이 버전이 안전한지"는 검증하지 않는다. 체크섬 검증, 서명 확인,
SBOM(Software Bill of Materials) 기반의 지속적 모니터링이 lockfile 위에 쌓여야
한다.

### 오픈소스의 역설

axios처럼 수천만 프로젝트가 의존하는 라이브러리도 소수의 메인테이너가 무급으로
관리하는 경우가 많다. 거대한 영향력과 빈약한 보안 자원 사이의 간극이 공급망
공격의 온상이 된다. 메인테이너 한 명의 계정 보안이 수백만 사용자의 안전을
좌우하는 구조는 지속 가능하지 않다. OpenSSF, Tidelift 같은 이니셔티브가 이
간극을 메우려 하지만, 근본적으로는 "공짜로 쓰되 보안은 알아서"라는 생태계의
사회적 계약이 재협상되어야 한다.
