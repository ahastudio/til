# Sogen: 시스콜 수준에서 작동하는 Windows/Linux 유저스페이스 에뮬레이터

<https://sogen.dev/>

<https://github.com/momo5502/sogen>

HN 토론: <https://news.ycombinator.com/item?id=48537481> (81점, 21개 댓글)

## 소개

Sogen은 보안 연구, 악성코드 분석, DRM 연구를 위한 고성능 Windows & Linux 유저스페이스 에뮬레이터다.
C++로 작성된 이 프로젝트의 핵심 설계 결정은 Windows API를 직접 재구현하지 않고
**시스콜(syscall) 수준에서 에뮬레이션**한다는 것이다.
기존 Windows 시스템 DLL들이 에뮬레이션된 환경 안에서 실제로 실행되며,
에뮬레이터는 그 경계인 시스콜 인터페이스만 처리한다.

에뮬레이션 백엔드는 세 가지를 선택할 수 있다.
CPU 명령어 에뮬레이션 라이브러리인 Unicorn Engine,
퍼즈 테스팅에 특화된 icicle-emu,
그리고 Windows 10 이상에서 사용 가능한 하이퍼바이저 API인 Hyper-V(WHP).
GPL-2.0 라이선스로 공개되어 있으며, `pip install sogen`으로 설치하는 Python 바인딩도 제공된다.
웹 버전은 sogen.dev에서 브라우저의 샌드박스를 통해 더 안전한 악성코드 분석 환경을 제공한다.

주요 기능은 다음과 같다.
시스콜 수준 에뮬레이션, Windows 특유의 메모리 타입(예약/커밋) 지원,
PE 실행 파일 및 DLL의 완전한 로딩(메모리 매핑, 재배치, TLS 포함),
Windows 구조적 예외 처리(SEH)와 스택 언와인딩 구현,
라운드로빈(round-robin) 방식의 스레드 스케줄링,
전체 상태 직렬화 및 빠른 인메모리 스냅샷,
그리고 IDA Pro, GDB, LLDB, VS Code와 통합되는 GDB 직렬 프로토콜 구현.

Python 바인딩은 간결한 API를 제공한다.
`sogen.windows.create_application()`으로 에뮬레이션 컨텍스트를 생성하고,
모듈 로드 콜백을 통해 진입점이나 특정 함수 실행 시점에 훅을 걸 수 있다.
에뮬레이션 루트(emulation root)는 실제 Windows 시스템 DLL들을 포함하는 디렉터리로,
sogen.dev에서 미리 만들어진 루트를 다운로드하거나 직접 구성할 수 있다.

## 분석

### 시스콜 수준 에뮬레이션은 Wine과 근본적으로 다른 트레이드오프를 선택한다

Linux에서 Windows 프로그램을 실행하는 가장 잘 알려진 방법은 Wine이다.
Wine의 접근은 Windows API를 Linux에서 재구현하는 것 — `kernel32.dll`, `ntdll.dll` 등의
함수들을 Linux 시스템 호출로 대응시키는 방식이다.
이것은 실제 Windows DLL 없이도 Windows 프로그램을 실행할 수 있다는 장점이 있지만,
수만 개의 Windows API 함수를 정확하게 재구현해야 한다는 거대한 부담이 있다.

Sogen은 반대 방향을 선택한다.
Windows API를 재구현하는 대신, 실제 Windows 시스템 DLL들을 에뮬레이션 환경 안에서 실행시킨다.
에뮬레이터는 CPU 명령어와 시스콜 인터페이스만 처리하고,
`kernel32.dll`, `ntdll.dll` 같은 실제 Microsoft 코드가 그 위에서 돌아간다.
결과적으로 API 호환성 문제가 근본적으로 달라진다.
Wine은 “어떻게 이 API를 정확하게 재구현하는가”를 묻지만,
Sogen은 “어떻게 시스콜 인터페이스를 정확하게 처리하는가”를 묻는다.

시스콜 인터페이스는 Windows API보다 훨씬 작고 안정적이다.
Windows 10 64비트 기준 ntdll의 시스콜은 수백 개 수준이다.
Windows API 함수는 수만 개다.
Sogen의 접근은 구현해야 할 인터페이스의 크기를 극적으로 줄이는 동시에
실제 Microsoft 코드를 실행하므로 API 동작의 정확성은 보장된다.
이것이 “고성능”이라는 수식어의 기술적 근거다 — 에뮬레이터가 처리해야 할 경계가 좁기 때문이다.

### 악성코드 분석에서 에뮬레이터의 위치는 샌드박스와 가상 머신 사이다

악성코드를 분석하는 전통적인 환경은 두 가지다.
가상 머신(VM) 위에서 실제 실행하거나, Cuckoo 같은 샌드박스에서 동작을 관찰하거나.
둘 다 “실제 실행”을 기반으로 한다는 공통점이 있고,
그 때문에 고급 악성코드가 VM이나 샌드박스를 탐지하고 행동을 변경하는 기법이 발달했다.

에뮬레이터는 다른 범주다.
에뮬레이터에서는 실행 흐름의 모든 지점을 관찰하고 수정할 수 있다.
메모리 접근, 레지스터 상태, 함수 호출, 시스콜 — 이 모든 것이 에뮬레이터의 제어 하에 있다.
VM 탐지 기법 중 많은 것이 하드웨어 특성(CPUID, RDTSC 타이밍, 하이퍼바이저 흔적)에 의존하는데,
에뮬레이터는 이 값들을 임의로 설정할 수 있다.

Sogen의 스냅샷 기능은 에뮬레이터만이 제공할 수 있는 능력이다.
특정 시점의 전체 프로세스 상태를 저장하고 복원하는 것은
실제 OS 위의 프로세스에서는 매우 어렵다.
에뮬레이터에서는 메모리와 레지스터 상태가 에뮬레이터 데이터 구조에 불과하므로
직렬화가 자연스럽게 가능하다.
이것은 악성코드가 특정 조건에서만 활성화되는 행동(time bomb, logic bomb)을 분석할 때
유용하다 — 조건을 맞춘 직전 상태로 반복해서 되돌릴 수 있기 때문이다.

drdexebtjl은 이 도구의 위치를 더 명확하게 정의했다.[^drdexebtjl]
"디버깅·리버스 엔지니어링 도구이며, 유저스페이스를 에뮬레이션함으로써
커널 수준 디버거와 동일한 수준으로 대상 프로세스를 제어하고 내부를 관찰할 수 있다 — 유저스페이스에서."
이 표현은 에뮬레이터를 샌드박스나 가상 머신이 아니라
정밀 관찰 도구로 보는 프레임이다.
가시성(visibility)과 제어성(control)이 에뮬레이터의 핵심 가치라는 것을
가장 간결하게 포착한 정의다.

gavinray는 저자의 트위터 영상을 인용해 실제 성능을 증언했다.[^gavinray]
Call of Duty MW2/3가 Sogen 위에서 실행되며 멀티플레이어까지 작동하는 영상이 있다는 것이다.
이것은 단순한 콘솔 애플리케이션이 아니라 고성능 그래픽 처리와 네트워크 동시성이 필요한
게임이 에뮬레이션 환경에서 작동한다는 의미다.
악성코드 분석 도구라는 포지셔닝과 달리 범용 호환성 레이어로서의 가능성을 암시한다.

### GDB 프로토콜 구현은 기존 도구 생태계를 에뮬레이터와 연결하는 전략적 결정이다

Sogen이 GDB 직렬 프로토콜을 구현하는 것은 단순한 편의 기능이 아니다.
IDA Pro, GDB, LLDB, VS Code — 이것들은 이미 수십 년간 발전한 분석 도구 생태계다.
이 도구들을 처음부터 대체하는 대신 이 도구들이 “GDB 서버”로 인식하는 프로토콜을 구현함으로써
기존 워크플로와 매끄럽게 통합된다.

이 선택의 의미는 에뮬레이터가 투명한 레이어가 된다는 것이다.
분석가는 이미 익숙한 IDA Pro에서 브레이크포인트를 설정하고 스텝 실행을 하면서,
내부적으로는 에뮬레이션 환경에서 악성코드가 돌아가고 있다는 것이다.
GDB 프로토콜이 없다면 에뮬레이터 전용 UI나 스크립팅 인터페이스를 통해서만 접근할 수 있고,
이것은 채택 장벽이 된다.
“기존 도구에서 그냥 쓸 수 있다”는 것이 에뮬레이터 채택의 핵심 마찰을 제거한다.

역사적으로 이 전략은 QEMU가 성공한 방식과 유사하다.
QEMU가 다양한 아키텍처 바이너리를 기존 Linux 도구들과 함께 사용할 수 있게 만든 것처럼,
Sogen은 Windows 바이너리 분석을 기존 리버스 엔지니어링 도구 흐름 안에 놓는다.

## 비평

### 호스트 격리 경고가 에뮬레이터의 근본적 보안 모델을 드러낸다

README에 눈에 띄는 경고가 있다:
“호스트 격리가 완벽하지 않을 수 있으니 주의를 권장합니다.”
이것은 기술적으로 정직한 경고지만, 에뮬레이터를 악성코드 분석 도구로 사용하는 경우
이 한계의 의미는 심각하다.

악성코드 분석에서 가장 중요한 보안 속성은 분석 환경이 호스트를 오염시키지 않는 것이다.
가상 머신이 이 보장을 제공할 수 있는 이유는 VM이 하이퍼바이저 수준에서 호스트와 분리되기 때문이다.
에뮬레이터는 근본적으로 다르다 — 에뮬레이터 자체가 호스트 OS 위의 프로세스로 실행된다.
에뮬레이션된 코드가 에뮬레이터의 취약점을 통해 에스케이프할 수 있다면
호스트 시스템이 직접 위험에 처한다.

이것은 Sogen의 설계 결함이라기보다 에뮬레이터라는 범주의 본질적 한계다.
그러나 README가 “악성코드 분석에 완벽하다”라는 인상을 줄 수 있는 언어를 사용하면서
동시에 호스트 격리 경고를 단일 주의사항으로 처리하는 것은
위험성의 심각도를 충분히 전달하지 않는다.
고급 악성코드는 에뮬레이터 탈출을 시도할 수 있으며, 이것은 Unicorn Engine이나
Hyper-V 구현의 버그를 통해 발생할 수 있다.
“웹 버전을 사용하면 브라우저 샌드박스의 보호를 받는다”는 권고가 실질적인 완화인지도
브라우저 샌드박스와 악성코드 에뮬레이터의 상호작용을 깊이 분석하지 않으면 검증하기 어렵다.

저자 momo5502는 HN에서 Wine과의 보안 차이를 직접 설명했다.[^momo5502]
“Wine과 달리 프로세스를 완전히 샌드박싱한다.
따라서 게임이 취약하더라도 공격자가 탈출해 PC의 파일에 접근하는 의도된 경로가 없다.”
이것은 Sogen의 격리 목표가 VM 수준의 하드웨어 격리가 아니라
공격 표면(attack surface) 최소화라는 것을 명확히 한다.
Wine은 호스트 파일 시스템과 네트워크에 직접 접근하는 구조이므로
취약한 Windows 애플리케이션이 Wine을 통해 호스트를 공격할 수 있다.
Sogen은 그 경로를 차단하는 것이 목표다 — 에뮬레이터 자체의 버그를 통한 탈출은
별개의 위협 모델이다.

### “에뮬레이션 루트”의 법적·실용적 복잡성이 충분히 다루어지지 않는다

Sogen의 시스콜 수준 에뮬레이션 접근은 실제 Windows 시스템 DLL을 필요로 한다.
README는 “에뮬레이션 루트를 sogen.dev에서 다운로드하거나 직접 구성하라”고 안내한다.
그러나 이 Windows 시스템 DLL들은 Microsoft의 저작권 하에 있으며,
이것을 배포하거나 에뮬레이션 루트로 패키징하는 것의 라이선스 적법성은 명확하지 않다.

Wine이 수십 년간 이 문제를 회피한 방법이 바로 “Windows API를 처음부터 재구현”하는 것이었다.
Wine은 실제 Microsoft DLL을 필요로 하지 않는다(사용자가 직접 복사할 수는 있지만).
Sogen의 접근은 이 문제를 정반대 방향으로 해결한다 — 실제 DLL을 가져와 쓰는 것.
sogen.dev가 “ready-made root”를 제공한다는 것은 실제 Microsoft DLL들이 포함된
파일을 배포한다는 의미인데, 이것이 Microsoft의 EULA나 저작권법과 어떻게 부합하는지는
README에서 다루지 않는다.

학술 및 보안 연구 목적의 “공정 사용(fair use)” 예외가 적용될 수 있지만,
이것은 관할권마다 다르고 법적으로 확립된 기준이 아니다.
GPL-2.0으로 Sogen 자체를 오픈소스로 공개하는 것과
Microsoft의 DLL을 포함한 에뮬레이션 루트를 배포하는 것은 완전히 다른 법적 문제다.

### GUI 애플리케이션 지원 부재는 범용 도구로서의 가능성을 제한한다

p0w3n3d는 Sogen의 실용적 한계를 지적했다.[^p0w3n3d]
"대부분의 Windows 앱은 GUI를 가지고 있다.
어떻게 활용할 수 있는지 궁금하다 — 아마 오래된 C# 앱을 서비스하는 정도?"
이것은 Sogen의 현재 설계가 헤드리스(headless) 실행에 최적화되어 있다는 현실적 한계를 짚는다.

악성코드 분석에서는 샘플이 GUI 없이 실행되는 경우가 많으므로 이것이 덜 중요하지만,
보안 연구나 DRM 분석에서 대상 소프트웨어가 GUI를 요구하는 경우에는 문제가 된다.
실제로 DRM이 적용된 상업 소프트웨어의 대부분은 GUI 애플리케이션이다.
"DRM research"를 사용 사례로 명시하면서 GUI 지원이 어떻게 처리되는지를
README나 위키에서 다루지 않는 것은 정보 공백이다.

gavinray가 언급한 Call of Duty 실행 영상은 이 한계가 절대적이지 않음을 시사한다.
그러나 GUI 렌더링 지원의 범위와 조건이 어디까지인지는 공식 문서에서 명확하지 않다.

### DRM 연구 용도가 가져오는 법적 복잡성을 README가 의도적으로 생략한다

README는 Sogen의 사용 사례로 “security research, malware analysis, and DRM research”를
명시적으로 열거한다.
DRM(디지털 저작권 관리) 연구는 미국의 DMCA(디지털 밀레니엄 저작권법),
EU의 저작권 지침, 한국의 저작권법 등에서 복잡한 법적 지위를 가진다.
이 법들은 DRM 우회 기술의 개발과 배포를 광범위하게 금지하며,
연구 목적의 예외는 제한적이고 조건부다.

Sogen이 게임의 안티치트 시스템이나 소프트웨어 라이선스 보호를 우회하는 데 사용될 수 있다는 것은
README의 “DRM research” 문구가 암시하는 사용 사례와 연결된다.
저자가 GPL-2.0을 선택한 것은 도구의 공개성과 연구 목적을 강조하려는 의도이겠지만,
도구를 사용하는 방식에 따라 사용자가 법적 위험에 처할 수 있다.

이것은 README에서 단 한 문장도 다루지 않는 주제다.
보안 연구 도구가 법적 위험을 면책 고지 없이 “DRM research”를 합법적 용도로 나열하는 것은
사용자에게 오해를 줄 수 있다.
Metasploit 같은 침투 테스팅 도구가 “authorized testing only”를 명시적으로 강조하는 것과 대조적이다.

## 인사이트

### 시스콜 경계가 Windows 에뮬레이션의 최소 완전 구현 지점이다

Wine이 선택한 “API 재구현” 접근의 근본적 문제는 무한한 표면적(surface area)이다.
Windows는 수십 년간 수만 개의 API 함수를 추가하고 변경해 왔다.
이 모든 것을 재구현하는 것은 이론적으로 가능하지만 실용적으로 완성이 없는 프로젝트다.
Wine의 30년 역사가 여전히 “이 프로그램은 Wine에서 작동하지 않는다”는 보고를 생산하는 이유가 여기 있다.

시스콜 인터페이스는 다르다.
NT 커널의 시스콜 인터페이스는 Windows 버전에 따라 변하지만,
단일 버전 내에서는 완전하게 정의된 유한한 집합이다.
이 인터페이스를 완전히 구현하면 그 버전의 Windows 위에서 실행되는 모든 프로그램이
원칙적으로 작동한다.
Sogen의 접근은 “에뮬레이션 완성도”를 측정하는 기준 자체를 바꾼다.
“얼마나 많은 API를 구현했는가”가 아니라 “시스콜 인터페이스가 얼마나 정확한가”로.

이 이차적 함의는 에뮬레이터 업데이트 부담에 있다.
Windows API가 새 버전에서 변경되면 Wine은 그 변경을 추적해야 한다.
시스콜 인터페이스는 더 안정적이다 — 마이크로소프트는 하위 호환성을 위해
시스콜 번호를 함부로 바꾸지 않는다.
그러나 새로운 시스콜이 추가되거나 기존 시스콜의 의미가 변경되면
에뮬레이터도 업데이트해야 하고, 이 시점을 어떻게 추적할지는 여전히 도전이다.

dismanova는 HN에서 자연스러운 질문을 던졌다.[^dismanova]
"Android 네이티브 바이너리를 위한 유사한 도구가 있는가?"
이 질문은 시스콜 수준 에뮬레이션이라는 접근의 범용성을 드러낸다.
Linux 커널 기반의 Android 역시 시스콜 인터페이스를 가지며,
시스콜 수준에서 에뮬레이션하면 실제 Android 시스템 라이브러리를 그대로 쓸 수 있다.
Android 악성코드 분석이나 Android 앱 DRM 연구에도 동일한 접근이 적용될 수 있다.
Sogen이 Windows를 선택한 것은 시장 크기와 악성코드 생태계의 집중도 때문이겠지만,
시스콜 수준 에뮬레이션 패턴은 OS에 종속되지 않는 일반 원칙이다.

### 고급 악성코드 분석의 병목은 실행 환경이 아니라 분석자의 인지 능력이다

Sogen 같은 에뮬레이터가 풀려는 문제 — 악성코드를 안전하고 정밀하게 분석하는 것 — 는
기술적으로 올바른 방향이지만, 실제 악성코드 분석의 병목이 어디에 있는지를 재고할 필요가 있다.

고급 악성코드, 특히 국가 수준의 APT(Advanced Persistent Threat) 도구들은
이미 에뮬레이션 환경과 샌드박스를 탐지하는 기법을 내장하고 있다.
시간 측정(RDTSC 기반), 인간 행동 시뮬레이션 부재 감지,
특정 레지스트리 키나 파일 시스템 구조의 존재 여부 등.
에뮬레이터가 이 탐지 기법을 무력화할 수 있더라도,
더 정교한 악성코드는 새로운 탐지 기법을 개발한다.
이것은 본질적으로 군비 경쟁이다.

진짜 병목은 분석자가 에뮬레이터로 수집한 데이터를 해석하는 능력이다.
수백만 개의 시스콜 로그, 메모리 덤프, 실행 트레이스를 의미 있는 악성코드 행동으로
매핑하는 작업은 여전히 숙련된 분석자의 인지 작업이다.
Sogen의 훅 시스템과 스냅샷 기능이 이 과정을 보조하지만,
“어느 시스콜이 의심스러운가”, “이 메모리 패턴이 무엇을 의미하는가”는
에뮬레이터가 답할 수 없는 질문이다.
Sogen이 더 발전한다면, 에뮬레이터 자체보다 그 위에서 동작하는 분석 자동화 레이어가
실질적 가치를 결정하게 될 것이다.

### 하이퍼바이저 백엔드의 존재는 에뮬레이션과 가상화의 경계를 재정의한다

Sogen이 Hyper-V(WHP)를 에뮬레이션 백엔드 중 하나로 지원한다는 사실은
이 도구가 “에뮬레이터”라는 단일 범주에 속하지 않음을 보여준다.

전통적 에뮬레이터는 CPU 명령어를 소프트웨어로 해석한다.
하이퍼바이저 백엔드를 사용할 경우 CPU 명령어가 실제 하드웨어에서 네이티브로 실행된다.
이것은 “Type 2 가상화”에 가깝고, 그 경우 에뮬레이션 성능 패널티가 대폭 줄어든다.
반면 하이퍼바이저를 사용하지 않는 Unicorn이나 icicle-emu 백엔드는
진정한 에뮬레이션으로, 성능 비용이 있지만 더 완전한 제어가 가능하다.

이 설계 선택이 만드는 트레이드오프는 미묘하다.
Hyper-V 백엔드에서는 CPU 타이밍 측정이 실제 하드웨어와 유사해지므로
타이밍 기반 탐지 기법을 무력화하기 어려워진다.
반면 Unicorn 백엔드에서는 RDTSC 값을 임의로 제어할 수 있다.
“어떤 백엔드를 선택하는가”가 분석 목적에 따라 달라지고,
이것은 사용자가 에뮬레이션과 가상화의 기술적 차이를 이해해야 한다는 진입 장벽이 된다.

장기적으로 이 다중 백엔드 설계는 에뮬레이션 도구의 미래 방향을 암시한다.
CPU 에뮬레이션, 하이퍼바이저 가속, 그리고 그 위의 시스콜 처리를
단일 추상화 레이어로 통합하는 것이 가능해지면서,
“에뮬레이터”와 “가상 머신”의 경계가 점점 흐려지고 있다.
Sogen은 이 수렴의 초기 단계를 보여주는 구체적 사례다.

### 보안 연구 도구의 오픈소스 공개는 공격과 방어를 동시에 강화한다

Sogen이 GPL-2.0으로 공개된 것은 방어적 보안 커뮤니티에 이점을 주지만,
같은 이유로 공격자에게도 이점이 된다.
이것은 보안 연구 도구의 오픈소스 공개가 항상 내포하는 이중성이다.

방어적 측면에서 Sogen의 오픈소스 공개는 코드 검토를 통해 신뢰성을 높이고,
보안 연구자 커뮤니티가 기여를 통해 개선할 수 있게 하며,
악성코드 분석 능력을 더 많은 연구자에게 접근 가능하게 만든다.
이것은 방어자가 공격자보다 더 강력한 도구를 갖게 되는 방향이다.

그러나 같은 에뮬레이터를 공격자가 사용할 수 있다.
악성코드 개발자가 Sogen을 사용해 자신의 코드가 분석 환경에서 어떻게 보이는지 테스트하거나,
안티 에뮬레이션 기법을 개발하는 데 활용할 수 있다.
이것은 Metasploit, Cobalt Strike, Frida 같은 도구들이 공유하는 이중성이다.

역사적 데이터는 이 이중성에서 방어가 순이익을 얻는다는 것을 보여준다.
오픈소스 보안 도구가 공개될 때 공격자에게 단기적 이점이 생기지만,
장기적으로는 방어 커뮤니티의 집단 지식이 더 빠르게 성장한다.
Sogen이 malware analysis의 진입 장벽을 낮출수록,
악성코드를 분석하고 탐지할 수 있는 연구자의 풀이 넓어진다.
이 비대칭성이 오픈소스 보안 도구 공개의 윤리적 정당화 기반이다.
