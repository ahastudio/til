# Honda Civic 헤드유닛 역공학: EvilValet 취약점 3년 후

원문: <https://juniperspring.org/posts/honda-evil-valet/>

HN 토론: <https://news.ycombinator.com/item?id=48523080> (342점, 76개 댓글)

## 요약

Eric McDonald가 2026년 6월 13일에 발표한 2021 Honda Civic 헤드유닛 역공학 3년 경과 보고서다.
핵심 발견은 “EvilValet”이라 명명된 취약점으로,
Honda가 헤드유닛 업데이트 파일 서명에 공개적으로 알려진 AOSP 테스트 키를 사용한다는 것이다.

Honda는 USB를 통한 헤드유닛 업데이트를 지원한다.
업데이트 파일은 AOSP 복구 방식으로 적용되는 서명된 파일 형태다.
문제는 `res/keys`에 공개 AOSP 테스트 키가 남아 있고,
`verify_file` 서명 검증 로직이 표준 AOSP와 동일하다는 점이다.
결과적으로 공개 AOSP 테스트 키로 서명된 파일은 헤드유닛에 그대로 설치된다.
USB 드라이브를 적절히 포맷하고 공개 키로 서명만 할 수 있으면
루트 접근 없이도 헤드유닛에 임의 코드를 실행할 수 있다.

이것이 “이블 베일릿(Evil Valet)” 공격이다.
호텔 방 대신 차 안에 물리적 접근이 필요한 이블 메이드(evil maid) 공격의 자동차 버전이다.
차를 발레파킹 서비스에 맡기는 동안 USB를 꽂아 헤드유닛을 변조할 수 있다.

저자는 EU 공식 소프트웨어 업데이트 파일 `MRC_EU_SW_v12_4.zip`도 테스트 키로 서명되어 있음을 확인했다.
공개 포럼에서 다운로드한 이 파일은 수정된 적 없는 공식 배포본이다.
모든 업데이트가 테스트 키로 서명되었을 가능성이 높다고 판단하며 커뮤니티의 추가 검증을 요청한다.

프로젝트는 `ota-builder`(업데이트 파일 생성 도구)와 `apk-rebuilder`(역공학 자동화 도구)를 공개했다.
`apk-rebuilder`는 공식 업데이트 파일을 입력받아 리소스 해석, smali 코드 재구성,
APK 재패키징, 램디스크 추출 등을 자동화한다.
Honda 소스 코드 자체는 배포 불가이므로, 이 도구는 변환된 출력물을 생성하는 함수 역할을 한다.

## 분석

### AOSP 테스트 키 잔류는 구조적 실수다

AOSP 테스트 키는 개발 편의를 위해 존재하며, 프로덕션 배포 전 제거하거나
실제 서명 키로 교체하는 것이 기본 보안 요건이다.
이것이 2021년 출시 차량의 헤드유닛에 남아 있다는 것은 개발 관행의 문제다.
단순 실수로 보기 어렵다.
Mitsubishi의 AOSP 포크를 사용한다는 언급으로 볼 때,
공급망에서 헤드유닛 소프트웨어를 납품하는 벤더(Mitsubishi Electric으로 추정)와
완성차 제조사(Honda) 사이의 보안 검증 프로세스가 없거나 불충분했다는 뜻이다.

더 우려스러운 것은 EU 공식 업데이트 파일도 테스트 키로 서명되어 있다는 점이다.
이것은 초기 출하 시점의 실수가 아니라 유지보수 업데이트까지 이어지는 패턴이다.
배포 파이프라인 전체가 테스트 키를 사용하도록 구성되어 있을 가능성이 높다.

HN 댓글에서 BobbyTables2는 이 패턴의 전형적인 기업 내부 사례를 공유했다[^BobbyTables2].
펌웨어가 내부 서명 서비스로 서명되었느냐는 질문에 제품 관리자가 "그렇다"고 자랑스럽게 답했지만,
실제로는 서명 검증 로직이 아예 동작하지 않았다는 것이다.
"서명되었는가"라는 질문과 "서명이 실제로 검증되는가"라는 질문은 다르다.
Honda의 경우도 서명 프로세스는 존재하되 키 관리가 무의미했다는 점에서 구조적으로 같은 실수다.

Koffiepoeder는 더 극단적인 사례도 제보했다[^Koffiepoeder].
어떤 시스템에서는 서명 알고리즘 자체가 업데이트 패키지 안에 포함되어 있어,
업데이트 파일이 자신의 검증 방식까지 결정할 수 있었다는 것이다.
"알고리즘도 업데이트 가능해야 한다"는 요구사항이 어떻게 보안 설계를 완전히 무너뜨리는지를 보여주는 사례다.

### 헤드유닛은 자동차의 새로운 공격 표면이다

헤드유닛은 현대 차량에서 가장 복잡한 소프트웨어 컴포넌트다.
Android 기반 AOSP를 사용하며, CAN 버스를 통해 차량의 다른 ECU와 통신한다.
헤드유닛에 임의 코드 실행이 가능하다면,
이것은 단순히 미디어 재생 기기 침해가 아니라 차량 내부 네트워크 접근 가능성을 열 수 있다.

저자는 이 글에서 헤드유닛 이상의 차량 제어 침해 가능성을 주장하지 않는다.
그러나 헤드유닛과 다른 ECU 간의 통신 인터페이스가 어떻게 구성되어 있는가는
자연스러운 다음 질문이다.
`aidl-rebuilder` 도구가 가상 속도계 같은 커스텀 앱 가능성을 언급하는 것은
헤드유닛이 차량 센서 데이터에 접근할 수 있음을 시사한다.

hnav는 이 지점을 직접 짚었다[^hnav].
헤드유닛이 CAN 게이트웨이와 연결되어 있다면 텔레매틱스 시스템을 호출할 수 있는지,
또는 CarPlay나 Android Auto를 통한 새로운 공격 경로가 존재하는지를 물었다.
이 질문은 헤드유닛 침해가 단독 사건으로 끝나지 않을 수 있음을 시사하며,
저자가 미완성으로 남겨둔 조사 범위를 정확히 가리킨다.

한편 bigfatkitten은 정부 기관 수준에서 이미 이 위협을 공식 대응하기 시작했음을 언급했다[^bigfatkitten].
2026년 3월 호주 정부 정보보안 매뉴얼(ISM)은 정부 기기를 차량 인포테인먼트 시스템에 연결하거나
커넥티드 차량 근처에서 민감한 대화를 나누는 것을 제한하는 통제 항목을 신설했다.
마이크, 카메라, GNSS 수신기, 무선 통신 모듈을 내장한 현대 차량은
그 자체로 감시 플랫폼이 될 수 있다는 인식이 정책 수준에서 반영된 것이다.

### 역공학 도구화 전략은 커뮤니티 협업의 좋은 모델이다

저자가 Honda 소스 코드 자체를 배포하지 않고 변환 도구를 배포하는 선택은 법적 제약을 우회하면서
커뮤니티 기여를 가능하게 하는 영리한 접근이다.
`apk-rebuilder`는 공식 업데이트 파일(저자가 호스팅하지 않음)을 입력받아
역공학 친화적인 출력을 생성한다.
이 “함수” 모델은 저작권 있는 바이너리를 직접 배포하지 않으면서
역공학 결과물을 재현 가능하게 한다.

LLM 활용 접근도 주목할 만하다.
저자는 참조 문서 작성 대신 도구를 통해 헤드유닛 코드를 분석 가능한 형태로 변환하고,
사용자가 LLM에 이를 질의하도록 유도한다.
이것은 LLM을 문서 대체재로 사용하는 실용적이고 유지보수 부담이 낮은 접근이다.

그러나 이 접근에 대한 비판도 있었다.
hankbond는 “아키텍처가 잘 설계된 코드는 LLM이 쿼리할 수 있다”는 이유로
코드 문서를 생략하는 프로젝트가 늘고 있다고 지적했다[^hankbond].
xgulfie는 한 발 더 나아가, 사용자가 LLM으로 문서를 생성하려면 Claude 구독 등이 필요하다는 점을 들어
“그것은 형편없는 접근”이라고 비판했다[^xgulfie].
오픈소스 프로젝트의 접근성은 상용 AI 도구 구독 여부에 의존해서는 안 된다는 지적이다.

## 비평

### 취약점 공개 책임 논의가 없다

저자는 Honda에 이 취약점을 신고했는지 여부를 언급하지 않는다.
공개 AOSP 테스트 키 사용은 중대한 보안 결함이다.
`ota-builder` 도구를 공개해 헤드유닛 변조를 누구나 할 수 있게 만들면서
제조사에 대한 취약점 공개나 패치 요청 없이 진행했다면,
이것은 책임 있는 공개(responsible disclosure) 원칙에서 벗어난 것이다.

자동차 보안 취약점은 소프트웨어 취약점과 다른 위험 등급을 가진다.
헤드유닛 변조가 차량 안전에 영향을 줄 수 있다면
공개 타임라인과 방법에 대한 더 신중한 접근이 필요하다.
글이 이 과정을 다루지 않는 것은 중요한 공백이다.

### 이블 베일릿 공격의 실제 위협 수준이 모호하다

이블 베일릿은 물리적 접근이 필요한 공격이다.
발레파킹 직원이 공격자인 시나리오는 가능하지만,
이 공격이 얼마나 실용적인지는 공격 목적에 따라 크게 다르다.
저자가 언급한 “기자가 호텔에 차를 맡기는” 시나리오는 특정 고가치 표적을 겨냥한 공격이다.

그러나 일반 소비자 위협으로서의 이블 베일릿은 다르게 평가된다.
무작위 공격자가 특정 차량을 대상으로 발레파킹 직원에 의존하는 복잡한 공격 체인을
실행할 동기가 있는지는 불분명하다.
취약점의 기술적 심각성과 실제 악용 가능성 사이의 간극을
글이 충분히 다루지 않는다.

HN 커뮤니티도 이 지점에서 엇갈렸다.
hahamaster는 물리적 접근권이 있는 공격자가 헤드유닛을 변조하는 복잡한 방법 대신
차 어딘가에 감청 장치를 숨기는 편이 훨씬 간단하다고 지적했다[^hahamaster].
TheDong은 더 단도직입적으로 “GPS 추적 장치를 바닥 매트 아래 두면 된다.
더 많은 차종에 쓸 수 있고 설치도 빠르다”고 했다[^TheDong].
이에 willis936은 “그것은 비용이 들고 발각 위험이 있다.
이미 설치된 장치를 쓰는 편이 낫다. 아니면 데이터 브로커에서 사면 된다”고 반박했다[^willis936].
공격 경제학의 관점에서 어떤 방법이 실제로 선택되는지는 공격자의 자원과 목표에 따라 달라진다.

xandrius는 다른 각도로 위협 모델을 재구성했다[^xandrius].
“공격자가 기기에 물리적 접근과 시간을 얻으면 게임 오버”라는 것은 오래된 보안 원칙이다.
이 원칙에서 보면 Honda Civic의 취약점은 놀라운 것이 아니다.
문제는 이 취약점이 악용 가능성보다 소유자가 자신의 기기를 수정할 권리를 박탈당했다는 사실에 있다는 것이다.

veza는 위협 모델의 현실적 불일치를 다른 각도에서 꼬집었다[^veza].
Civic 오너가 발레파킹 서비스가 있는 고급 호텔에 차를 맡기는 시나리오 자체가
현실에서 얼마나 빈번한지를 물었다.
Type R 오너라면 모를까, 일반 Civic 오너가 표적이 될 확률에 대한 현실적 평가다.

### 기여자 모집과 리스크 전가

저자는 여러 미완성 도구에 대해 커뮤니티 기여를 요청한다.
`ota-builder`를 테스트하려는 사람은 “자신의 위험을 감수”하고 시도하도록 초대한다.
이것은 정직한 고지이지만, 동시에 헤드유닛을 소프트브릭할 수 있는 작업의 테스트를
커뮤니티에 아웃소싱하는 것이다.

역공학 커뮤니티에서 이런 접근은 일반적이지만,
자동차 헤드유닛은 제조사 서비스 없이는 복구가 어려울 수 있다는 점에서
스마트폰 역공학과 리스크 프로파일이 다르다.
기여자들이 이 차이를 충분히 인식하고 참여하는지는 불명확하다.

## 인사이트

### 자동차 소프트웨어는 모바일 소프트웨어 보안 기준에 한참 미치지 못한다

공개 AOSP 테스트 키를 프로덕션 차량에 사용하는 것은
2024년 기준 모바일 앱 개발에서는 상상하기 어려운 실수다.
Google Play Store 정책, Android 인증 프로그램, OEM 파트너십 프로그램은
이런 기본적인 보안 실수를 걸러내는 여러 겹의 필터를 갖추고 있다.

자동차 헤드유닛은 이 생태계 밖에 있다.
납품 벤더가 OEM에 소프트웨어를 납품하는 B2B 구조는
최종 사용자(운전자)가 보안 취약점을 보고하거나 패치를 요구하는 메커니즘이 없다.
차량 소프트웨어의 보안 책임 구조는 소비자 소프트웨어와 다른 거버넌스 모델이 필요하다.

### 헤드유닛 역공학은 수리권 논쟁의 새 전선이다

Honda가 헤드유닛 소프트웨어를 공개하지 않고
서드파티 수리 및 커스터마이징을 허용하지 않는 것은 수리권 논쟁의 자동차 버전이다.
`apk-rebuilder` 같은 역공학 도구는 이 폐쇄적 생태계에 대항하는 커뮤니티의 응답이다.

흥미로운 것은 저자가 발견한 보안 취약점이 역설적으로 커스터마이징 가능성을 열었다는 점이다.
Honda가 의도하지 않은 방식으로 소프트웨어를 변경할 수 있게 된 것이
보안 취약점 덕분이다.
수리권과 보안의 관계는 단순하지 않다.
더 개방적인 소프트웨어 접근이 오히려 보안 개선으로 이어질 수 있다.
커뮤니티가 코드를 검토하고 취약점을 발견하는 것이 폐쇄적 개발보다
취약점을 더 빠르게 찾아낼 수 있다.

Aerolfos는 이 딜레마를 정확히 표현했다[^Aerolfos].
제조사는 완전히 열린 기기(소유자가 자유롭게 수정 가능)이거나
완전히 닫힌 기기(실제로 안전한) 중 하나를 선택해야 한다.
현재 상태는 최악의 조합이다.
개인 데이터는 광범위하게 수집하면서도 보안은 취약한 채 방치되어 있다.
userbinator는 반대 관점을 제시했다[^userbinator].
"테스트 키 잔류는 시스템을 소유자에 대해 잠그려는 의도가 없었다는 신호"라고 해석하며,
이를 긍정적으로 볼 수도 있다고 했다.
t1234s는 이 가능성을 구체적인 질문으로 만들었다[^t1234s].
"이것을 이용해 헤드유닛에 LineageOS를 올릴 수 있을까?"
이 질문은 취약점이 보안 위협인 동시에 소유자 권한 회복의 도구가 될 수 있음을 보여준다.

### LLM을 역공학 도구로 사용하는 접근은 보안 연구 방법론을 바꾼다

저자가 “헤드유닛 코드를 분석 가능한 형태로 변환하고 LLM에 질의하도록”
설계한 방법론은 보안 역공학의 새 패러다임을 보여준다.
전통적인 역공학은 분석가가 어셈블리나 smali 코드를 직접 읽는 것을 요구했다.
이 방법론은 도구가 코드를 준비하고 LLM이 의미를 추출하는 분업 구조를 만든다.

이것은 역공학 접근 장벽을 낮춘다.
smali 코드를 읽을 수 없는 사람도 LLM에 변환된 코드를 입력해
“이 함수는 무엇을 하는가”를 물을 수 있다.
보안 연구의 민주화 측면에서 의미 있는 변화다.
동시에 취약점을 찾는 속도가 빨라진다는 것은
보안 취약점 발견과 공개 사이의 시간이 짧아진다는 의미이기도 하다.
