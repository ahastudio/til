# Scratch SVG Sanitization의 고통

원문: <https://muffin.ink/blog/scratch-svg-sanitization/>

HN 토론: <https://news.ycombinator.com/item?id=47922957> (253점, 100개 댓글)

## 요약

이 글은 2019년부터 2026년까지 Scratch 플랫폼이 SVG 보안 취약점과 싸워온 역사를
기록한다. 처음에는 `<script>` 태그를 제거하는 정규식 한 줄로 시작했지만, 이후 7년에
걸쳐 DOMPurify 도입, CSS 파서 추가, 훅(hook) 체인 등 점점 더 복잡한 레이어가 쌓였다.
그럼에도 새로운 우회 기법이 계속 등장했고, 저자는 이 “sanitization 더 쌓기” 전략이
근본적으로 잘못된 접근임을 결론으로 제시한다.

취약점의 유형은 XSS(Cross-Site Scripting)에서 IP 추적을 위한 HTTP 누수, CSS를 통한
전체 페이지 스타일 덮어쓰기, 피싱을 위한 UI 변조까지 다양하다. 각 취약점은 수정됐지만
그 수정 자체가 새로운 복잡성을 도입하고, 그 복잡성이 다시 다음 우회 경로를 만드는
악순환이 반복된다. CSS 파서(`css-tree`)와 실제 브라우저 CSS 파서 사이의 동작 불일치가
이 악순환의 핵심 원인으로 지목된다.

대안으로 TurboWarp가 구현한 iframe 샌드박스 방식이 소개된다. `sandbox=“allow-same-origin”`
속성과 엄격한 Content Security Policy(`default-src 'none'; style-src 'unsafe-inline'
data:; font-src data:; img-src data:`)를 결합하여, 브라우저 자체가 보안 경계를
강제하도록 설계한다. 이 방식에서는 새로운 CSS 기능이 추가되어도 CSP가 자동으로
외부 리소스 요청을 차단하므로, 코드 변경 없이 보안이 유지된다.

글 발행 이후 Claude AI가 `css-tree`가 올바르게 파싱하지 못하는 CSS 중첩(nesting)
완화 문법(relaxed syntax) 우회 기법을 추가로 발견했다고 저자가 후기로 덧붙인다.
이는 알려진 파서 불일치 문제가 여전히 미탐지 취약점을 내포하고 있음을 보여준다.

## 분석

### 취약점 연대기의 패턴

2019년부터 2026년까지 7년간의 취약점 목록을 보면 뚜렷한 패턴이 드러난다. 각 수정은
그 당시에 알려진 공격 벡터만 막는다. 그런데 SVG와 CSS 명세는 계속 진화하고 브라우저는
새로운 기능을 구현한다. 따라서 오늘 안전한 코드가 내일 새로운 기능 하나로 취약해진다.

| 연도  | 취약점              | 수정 방법                          |
| ----- | ------------------- | ---------------------------------- |
| 2019  | `<script>` 태그 XSS | 정규식 제거                        |
| 2020  | 대소문자 우회       | DOMPurify 도입                     |
| 2020  | 이벤트 핸들러 XSS   | DOMPurify 확장                     |
| 2022  | 이미지 href HTTP 누수 | DOMPurify 훅 추가                 |
| 2023  | CSS `@import` 누수  | JavaScript CSS 파서 통합           |
| 2024  | Paper.js 경유 XSS   | 별도 처리 로직 추가                |
| 2025  | CSS `url()` 우회    | 파서 확장 (이스케이프·변수 처리)   |
| 2026  | CSS 전환(transition) 기반 스타일 덮어쓰기 | 미해결 또는 부분 해결 |
| 2026  | `image-set()` 누수  | 미수정                             |

### 파서 불일치 문제의 구조적 성격

CSS `url()` 우회 사례는 파서 불일치가 얼마나 깊은 문제인지를 잘 보여준다.
`\75\72\6c` 같은 이스케이프 시퀀스는 `css-tree`가 `url()`로 인식하지 못하지만
브라우저는 정상적으로 처리한다. CSS 변수(`var(--name)`)는 파싱 시점에 값이 없어
정적 분석으로 탐지할 수 없다. 이런 불일치는 명세의 모호성, 각 파서 구현체의 허용
오차 차이, 그리고 파싱 컨텍스트 의존성에서 비롯된다. JavaScript CSS 파서 하나로
브라우저의 동작을 완벽히 모사하는 것은 원리적으로 불가능하다.

### iframe CSP 방식의 기술적 근거

TurboWarp의 iframe 샌드박스는 “sanitize less, isolate more” 전략이다. `sandbox`
속성은 스크립트 실행을 막고, CSP의 `default-src 'none'`은 외부 네트워크 요청 자체를
차단한다. SVG 안의 CSS가 아무리 복잡한 `url()`이나 `image-set()` 값을 가지더라도,
브라우저가 요청을 보내기 전에 CSP가 가로막는다. 이 방식은 “어떤 CSS 기능이 위험한지”를
파악하는 대신, “모든 외부 요청을 금지”하는 더 강한 불변식을 확립한다.

단, `allow-same-origin`을 `sandbox`에 포함한 이유는 `getBBox()` 같은 레이아웃 측정
API가 같은 오리진(origin) 맥락에서만 올바르게 동작하기 때문이다. 이 결정은 iframe이
부모 페이지와 같은 오리진을 공유한다는 의미이므로, 오프스크린 배치와 함께 신중하게
관리해야 하는 트레이드오프다.

CSP를 HTTP 헤더로만 설정해야 한다고 오해하기 쉽지만, 사실 HTML `<meta>` 태그로도
신뢰성 있게 설정할 수 있다. 페이지에서 직접 생성한 HTML처럼 HTTP 헤더 단계에 개입할
수 없는 시나리오에서 특히 유용하다. 직관적으로는 untrusted JavaScript가 DOM을 통해
이 meta 태그를 지우거나 수정해 CSP를 우회할 수 있을 것 같지만, 모든 최신 브라우저는
meta 태그가 로드된 시점부터 그 안의 CSP 규칙을 영구화(lock-in)하도록 되어
있다.[^simonw] 이 동작 보장이 “HTML 자체에 CSP 박아넣기”를 실용적인 격리 수단으로
만든다.

### SVG가 “이미지”가 아니라는 인식의 부재

같은 문제를 “왜 처음부터 이렇게 됐는가”의 관점에서 다시 보면, SVG는 애초에 이미지
포맷이 아니라 마크업 언어라는 사실이 충분히 인지되지 못한 결과로도 읽을 수 있다.[^etchalon]
플랫폼 설계자들이 SVG를 PNG/JPEG와 동급의 “안전한 이미지”로 다루기 시작한 순간부터
이번 7년 연대기의 출발점이 만들어진 셈이다. 실제로 이 글에 등장한 “XSS는 SVG 고유
이슈가 아니다 — 표준 HTML XSS 벡터들이 SVG 컨텍스트에서 동일하게 적용될 뿐이며,
오히려 발표 속성(presentational attribute)의 `cursor`가 url을 로드한다거나 DTD를
이용한 우회 같은 진짜 SVG-특이적인 까다로운 부분은 본문에서 다뤄지지도 않았다”는
지적이 그 인식의 부재를 잘 보여준다.[^bawolff]

## 비평

### 강점

글의 가장 큰 강점은 실제 취약점 연대기를 구체적인 기술 세부사항과 함께 공개한다는
점이다. 많은 보안 포스트모텀이 결론만 공유하지만, 이 글은 각 우회 기법의 작동 원리와
실패한 수정의 이유를 투명하게 기술한다. 이것이 독자에게 교훈이 되는 이유는 “무엇이
잘못됐는가”가 아니라 “왜 잘못됐는가”를 설명하기 때문이다.

CSP 기반 iframe 샌드박스라는 대안을 제시하고, 그것이 왜 더 견고한지 기술적 근거까지
설명한 것도 훌륭하다. 문제 제기로 끝나지 않고 실행 가능한 해결책을 함께 논의한다.

### 약점 및 한계

CSS 전환(transition) 기반 전체 페이지 스타일 덮어쓰기 공격에 대한 기술적 설명이
다소 부족하다. 이 취약점이 어떻게 실제로 피싱 또는 안전 기능 은폐로 이어지는지
구체적인 시나리오가 있었다면 독자가 위험도를 더 잘 가늠할 수 있었을 것이다.

또한 iframe 샌드박스 방식의 한계도 더 상세히 다뤄졌으면 좋겠다. `allow-same-origin`
과 오프스크린 배치를 함께 쓸 때 주의해야 할 에지 케이스, 측정 API(`getBBox`, `getComputedStyle`)
동작 방식 등 구현자가 실제로 마주칠 문제들에 대한 안내가 거의 없다.

글 마지막에 Claude AI가 우회 기법을 발견했다는 언급은 흥미롭지만, 해당 취약점의
구체적인 내용이나 수정 여부를 명확히 서술하지 않아 독자 입장에서 미완의 느낌이 있다.

### 정규식으로 (X)HTML 파싱: 10년 늦은 교훈

2019년 시점의 첫 수정이 정규식으로 `<script>`를 지우는 것이었다는 점은 그 자체로
약점이다. “정규식으로 (X)HTML을 파싱할 수 없다”는 명제는 2009년 StackOverflow의
유명한 답변으로 이미 광범위하게 알려져 있었다.[^kevinmgranger] 즉 이 글이 보여주는
초기 실수는 “SVG 보안이 어렵다”의 귀결이라기보다, 잘 알려진 일반론을 새 컨텍스트에
적용하지 못한 사고의 흔적에 가깝다. 이 점을 본문은 충분히 강조하지 않는다.

### 책임 있는 공개(responsible disclosure) 누락

별개의 비평 지점도 있다. 본문 발행 후 저자가 “현재 버전 Scratch에도 비슷한 임의
코드 실행이 가능하다”는 후속 글을 공개하면서, 정작 책임 있는 공개 절차에 대한 언급
없이 자신의 다른 프로젝트 홍보로 연결한 점은 보안 커뮤니티 관점에서 무책임에 가깝다는
강한 비판이 제기된다.[^nmilo] 본문이 “7년의 연대기”를 정리하는 가치는 분명하지만,
공개 윤리에 대한 자세는 본문이 보여주는 분석의 깊이와는 별개의 평가가 필요하다.

## 인사이트

### Sanitization 기반 보안의 근본 한계

SVG와 CSS 보안 문제는 단순히 구현 실수의 결과가 아니다. “신뢰할 수 없는 콘텐츠를
파싱하여 안전한 부분집합으로 정제한다”는 sanitization 전략이 갖는 근본적인 한계를
드러낸다. 이 전략이 올바르게 작동하려면 sanitizer의 파서가 실제 렌더러의 파서와
완벽히 동일하게 동작해야 한다. 그런데 명세의 모호성, 구현체별 허용 오차, 브라우저
업데이트가 이 동등성을 지속적으로 깨뜨린다.

이 문제는 HTML sanitization에서도 잘 알려져 있다. DOMPurify 자체도 여러 차례
브라우저별 파싱 차이를 이용한 우회 기법에 패치를 냈다. SVG는 HTML보다 훨씬 복잡한
명세를 가지며, CSS는 그 안에서 또 다른 완전한 언어로 동작한다. 중첩된 복잡성이
sanitization의 신뢰 가정을 더 빨리 무너뜨린다.

이 통찰의 함의는 단순히 “DOMPurify 써라”가 아니다. 사용자 제공 SVG를 직접 DOM에
삽입하는 모든 시스템은 “완벽한 sanitizer”를 만들 수 없다는 현실을 받아들이고, 격리
아키텍처로 전환해야 한다는 것이다. 보안 목표가 “안전한 SVG 파싱”에서 “SVG가 해를
끼칠 수 없는 환경 만들기”로 이동해야 한다.

이 결론을 가장 단호하게 채택한 사례가 Google Slides다. 사용자 요청에도 불구하고
거의 15년 묵은 SVG 지원 티켓이 여전히 열린 채 닫히지 않는 핵심 이유가 바로 이
sanitization 비용·위험 트레이드오프에 있다는 지적이 있다.[^spankalee] 거대 제품에서
“기능을 안 넣는 것”이 합리적인 보안 선택일 수 있음을 보여주는 사례다.

### 브라우저 보안 프리미티브를 레버리지 하는 설계

TurboWarp의 CSP + iframe 방식이 보여주는 더 깊은 원칙은 “브라우저의 보안 프리미티브
위에 구축하라”다. 브라우저는 수십 년간의 공격과 방어를 통해 정제된 보안 메커니즘을
내장하고 있다. CSP, sandbox iframe, SRI(Subresource Integrity), CORS — 이것들은
이미 전 세계 수십억 사이트를 보호하며 검증된 도구들이다. 그 위에 얇은 레이어를
얹는 것이 직접 구현한 복잡한 sanitizer보다 거의 항상 더 안전하다.

역사적 유비가 있다. 1990년대 암호화 라이브러리를 직접 구현하다 취약점이 발견되자,
업계는 점차 검증된 라이브러리(OpenSSL, libsodium 등)를 표준으로 채택했다. “직접
구현은 위험하다”는 교훈이 암호화에서 학습된 것처럼, SVG 보안에서도 “직접 sanitize는
위험하다”는 교훈이 쌓이고 있다. 브라우저 프리미티브 레버리지가 그 대안이다.

실무적 함의는 명확하다. 새로운 플랫폼이 사용자 SVG를 지원하려 할 때, 첫 번째 설계
결정이 “어떻게 sanitize할 것인가”여서는 안 된다. “어떻게 격리할 것인가”가 첫 번째
질문이어야 한다. CSP 헤더 설정, sandbox iframe 설계, 오프스크린 렌더링 파이프라인
구성이 sanitizer 라이브러리 선택보다 앞서야 한다.

### 소프트웨어 유지보수 부채와 보안의 교차점

Scratch의 7년 연대기는 보안 부채(security debt)가 어떻게 축적되는지를 보여주는
교과서적 사례다. 각 수정은 그 자체로는 합리적이다. 정규식으로 `<script>`를 제거하는
것은 당연해 보이고, DOMPurify를 추가하는 것도 올바른 결정이며, CSS 파서를 통합하는
것도 당시로서는 최선이었다. 그런데 이 합리적인 결정들이 쌓여 아무도 전체를 완전히
이해하지 못하는 복잡한 시스템이 된다.

이것은 보안만의 문제가 아니다. 기술 부채 일반의 패턴이다. 단기 해결책이 장기 복잡성을
만들고, 그 복잡성이 새로운 버그와 취약점의 온상이 된다. 보안 맥락에서 이 패턴이
특히 위험한 이유는 복잡성이 곧 공격 표면(attack surface)이기 때문이다. 코드 줄이
늘어날수록 파서 불일치가 숨을 공간이 넓어진다.

이 사례에서 얻을 수 있는 조직 수준의 교훈은 보안 아키텍처를 주기적으로 근본부터
재검토해야 한다는 것이다. “이 fix가 지금 작동하는가”를 묻는 것만큼, “이 방향이
앞으로도 유지 가능한가”를 묻는 것이 중요하다. Scratch가 7년 후에 도달한 결론을
2년째에 내릴 수 있었다면 누적된 복잡성의 상당 부분을 피할 수 있었을 것이다. 보안
코드 리뷰에서 “이 패치가 취약점을 막는가”와 함께 “이 전략이 5년 후에도 유지될 수
있는가”를 함께 물어야 하는 이유다.

LLM 시대에는 이 부채 축적 패턴이 더 빨리, 더 거칠게 나타날 가능성이 높다. LLM은
기존 코드 위에 “워크어라운드를 더 쌓는” 방식의 변경을 매우 잘 한다. 즉 sanitization
레이어를 한 단계 더 추가하는 것 같은 종류의 패치가 자동화되기 쉬운 형태다. 같은
근본 문제를 다른 모양으로 가린 코드가 빠르게 누적되면서, “복잡성이 곧 공격 표면”이
되는 속도는 더 빨라질 것이다.[^Hendrikto] 보안 아키텍처 리뷰의 주기를 더 짧게,
그리고 “리팩터링이 아니라 폐기”라는 옵션을 더 자주 테이블에 올려야 하는 이유다.

### 포맷 설계의 갈림길 — “HTML for vector art” vs “JPEG for vector art”

더 거시적인 인사이트는, SVG라는 포맷 자체가 “벡터 이미지를 위한 JPEG”이 아니라
“벡터 아트를 위한 HTML”에 가깝게 설계됐다는 점이다. CSS/JS/네트워크 같은 모든
복잡성이 SVG 안에서 그대로 살아 움직이므로, 이미지 한 장을 표시하기 위해 사실상
브라우저 전체를 끌어와야 한다.[^codedokode] 그 결과 사용자 제공 콘텐츠를 받는
플랫폼은 “이미지를 보여준다”는 단순한 행위에서 sanitization 전쟁을 치르게 된다.
원래 원하던 것이 “벡터 아트를 위한 JPEG”, 즉 sanitize 없이도 안전한 정적 포맷이었다는
회고도 같은 결을 짚는다.[^Pxtl]

이 통찰은 새 포맷 제안 같은 즉답을 강요하지는 않는다. 다만 사용자 콘텐츠 플랫폼이
새 미디어 타입을 받을 때 “이 포맷의 보안 모델이 정적 데이터인지, 마크업 + 동적
실행인지”를 첫 번째 질문으로 두어야 한다는 점은 분명하다. 답이 후자라면, sanitization
파이프라인의 첫 줄을 짜기 전에 격리 아키텍처를 먼저 설계해야 한다.
